单机到集群：为何需要分布式架构

当用户量、数据量或发布频率超过单机/单库舒适区，团队会把系统拆到多台机器上协作。常见驱动力：

  • 水平扩展 — 加机器分担读/写吞吐，而不是无限堆单机配置。
  • 故障隔离 — 单点故障只影响局部；配合副本与自动切换提高可用性。
  • 组织与交付 — 多团队并行演进不同业务能力（与 康威定律 相关，架构风格：单体、SOA 与微服务 展开）。
  • 地理与合规 — 多机房、多区域部署降低延迟或满足数据驻留要求。

何时值得认真考虑拆分？ 不必等到「千万 QPS」才行动，下列信号出现时应把分布式纳入架构讨论（满足其一即可启动评估）：

  • 单机或单库连接池、CPU、磁盘 IO 长期打满，垂直扩容性价比明显下降。
  • 发布耦合：改一个小模块需要全量回归、全量部署，交付周期被拖慢。
  • 可用性目标 要求多 AZ/多机房与明确 RTO/RPO，单机故障域无法接受。
  • 组织规模 扩大，多团队需并行演进不同业务线。
单体应用

开发简单、事务本地；扩展时往往整包复制，数据库易成瓶颈。

分布式系统

能力可独立扩展；需要处理网络调用、一致性与运维复杂度。

架构设计职责

划定边界、选通信方式、定义失败语义与可观测性，而不是「微服务越多越好」。

远程调用：反例 vs 推荐做法

不推荐
  • 无超时，线程一直阻塞
  • 把 Feign 当本地方法，异常吞掉
  • 超时后无幂等，立即重试 N 次
  • 用墙钟时间排序跨服务事件
推荐
  • 显式 connect/read 超时 + 有限重试
  • 业务幂等键 + 查询预留状态
  • 熔断 / 舱壁保护下游（详见 韧性工程：熔断、限流、重试与舱壁）
  • 版本号、逻辑时钟或中心化协调
先想 10 秒：没有分布式时，大促峰值你会先动哪一招？

常见：垂直扩容、读写分离、缓存。分布式是多机 + 明确失败语义，不是单机优化的换皮。

网络分区、时钟漂移与部分失败

ShopFlow · 大促下单（30 秒） 用户下单经网关到 order-service，同步调用 inventory-service 预留库存（超时 2s）。库存已扣减，响应却在第 8 秒丢包——客户端只看到超时。下面用序列图理解这种部分失败。

分布式系统的默认假设是：网络不可靠、节点会挂、时钟不完全同步。这三点直接决定你不能把远程调用当成本地方法调用。

  • 网络分区 — 节点之间暂时不可达，但各自仍在运行；此时「全局一致决策」可能无法同时满足。
  • 时钟漂移 — 即使部署 NTP，各节点墙钟仍可能偏差；不要用 System.currentTimeMillis() 在跨服务间排序事件。
  • 部分失败 — 请求可能「客户端以为失败、服务端已成功」或相反；必须设计超时、重试、幂等与可补偿流程。

调用方视角 × 服务端真实状态

调用方看到服务端可能真实状态典型应对
超时 / 连接失败 已成功（响应丢失） 幂等键 + 查询预留状态；避免盲目重试导致重复扣减
超时 / 连接失败 未执行 带幂等键的安全重试；记录业务请求号
收到成功响应 实际失败（少见） 对账任务、状态机补偿、告警
重复提交同一业务键 已处理过 幂等返回原结果

超时后重试：代码层反例与推荐

不推荐 · 盲目重试
java 
// 无幂等键、无上限 — 可能重复扣库存
for (int i = 0; i < 10; i++) {
  try {
    inventoryClient.reserve(orderId, skuId, qty);
    break;
  } catch (Exception e) {
    Thread.sleep(100);
  }
}
推荐 · 幂等 + 有限次
java 
// 同一 requestId：服务端已处理则直接返回原结果
ReserveResult r = inventoryClient.reserve(
  new ReserveCommand(requestId, orderId, skuId, qty));
if (r.isUnknown()) {
  r = inventoryClient.getReserveStatus(requestId);
}
// 仍不确定 → 标记订单「待对账」，勿直接成功

可扩展性、可用性与一致性三角

工程上常同时追求三者，但在故障与负载压力下必须排序优先级。例如：

目标典型手段代价/注意
可扩展性 无状态服务、分片、缓存、异步削峰 数据局部性、跨分片查询变难
可用性 多副本、健康检查、熔断、多活/主备 副本滞后时读到旧数据
一致性（详见 CAP、BASE 与一致性模型：强一致 / 最终一致） 强一致事务、共识协议、最终一致 + 对账 延迟上升、可用性在分区时受限

场景决策：大促秒杀 vs 财务对账

场景优先（1 最重要）可让步（3）一句话理由
大促秒杀 可扩展性 强一致（接受短暂不一致 + 对账） 峰值吞吐优先，库存可异步校准
财务对账报表 一致性 可扩展性（批处理窗口内完成即可） 金额差错成本高，宁可慢一点也要可对账

CAP、BASE 与一致性模型 将用 CAP 定理 与 BASE 把上表落到可讨论的取舍语言。

常见误区
  • 微服务越多越好 — 边界不清时，分布式复杂度会压过拆分收益。
  • 远程调用 = 本地方法 — 必须显式定义超时、失败语义与可观测性。
  • 超时后无限重试 — 可能打爆下游；需退避、熔断与幂等。
  • 同步 NTP 就够了 — 不能依赖墙钟做跨节点因果排序。

Java 技术栈在分布式中的常见角色

本课实践以 JVM 企业栈为主。常见分工如下：

  • 应用运行时 — Spring Boot、Actuator 健康检查。
  • 服务治理 — 注册发现、API 网关、配置中心。
  • 服务间通信 — OpenFeign（同步）、Spring Cloud Stream（异步）。
  • 数据与一致性 — 本地事务；跨服务用 Saga、Outbox、Seata。

通信模式：同步 vs 异步（标签页）

适用：需要立即知道成败的链路，如「下单当场预留库存」。

代价：延迟叠加、级联故障、部分失败（本章重点）。

ShopFlow：order → inventory 的 POST /reserve（详见 OpenFeign、负载均衡与服务间调用）。

适用：解耦、削峰、最终一致，如「支付完成通知订单」。

代价：复杂度在消费者、幂等、顺序与死信队列。

ShopFlow：PaymentCompleted 事件（详见 Kafka 事件驱动与 Redis 缓存）。

常见问题（FAQ）

有了 Kafka，是不是就不用 OpenFeign 了？
不是二选一。需要当场知道成败的链路（预留库存）仍常用同步 HTTP；可延迟、可重放的通知（支付完成）更适合事件。ShopFlow 两种都会用到。
微服务是不是拆得越细越好？
否。边界不清会带来分布式事务、排障与发布协调成本。架构风格：单体、SOA 与微服务 会讲「何时不应拆」。
超时设很长，是不是就不会部分失败了？
只会把「失败发现」推迟，并占用连接/线程。网络分区、进程崩溃仍会发生；仍需幂等、查询与对账。

本章结论

放在正文与复习之间，便于你先串起全章再做题。每条对应上文一个小节标题。

  • 为何走向分布式：用多机协作换扩展与故障隔离；常见动机是容量、交付节奏或可用性，而不是「为了微服务而微服务」。
  • 部分失败是常态：网络、节点、时钟都不可靠；调用超时后先幂等与查状态，避免无界重试把故障放大。
  • 三角要显式排序：扩展、可用、一致无法同时拉满；秒杀、对账、详情页等场景的优先级不同，CAP、BASE 与一致性模型 将命名这些取舍。
  • Java 栈分工：Boot 承载服务与可观测；Cloud 负责治理与同步/异步通信；跨服务写路径用 Saga、Outbox 等模式兜一致。
  • ShopFlow 主线：Order / Inventory / Payment 三个限界上下文；同步 POST /reserve 预留库存，PaymentCompleted 事件异步收尾支付。

CAP 定理与工程中的真实取舍

ShopFlow · 机房网络抖动（45 秒） 同城双活：订单库在 A 机房、库存缓存主在 B 机房。交换机故障导致 A↔B 分区 30 分钟。用户仍能下单吗？库存数字必须全局一致吗？这就是 CAP 在工程里的提问方式。

CAP 定理（Brewer）指出：在存在网络分区时，分布式系统无法同时满足：

  • C（Consistency） — 每次读都能读到最新写入，或返回错误（线性一致视角的简化理解）。
  • A（Availability） — 每个请求都能得到非错误响应（不要求读到最新）。
  • P（Partition tolerance） — 分区发生后系统仍能继续工作（节点间消息可能丢失/延迟）。

真实部署几乎必须接受 P，于是分区发生时要在 C 与 A 间做选择——不是「三选二」的日常开关，而是按业务路径分别设计。

  end
  normal --> partition

CP：宁可失败也不返回不确定的旧数据；AP：宁可旧数据也要保持入口可用。

对照：CP 与 AP 在 ShopFlow 中的例子

偏 CP 的路径
  • 支付扣款：余额不足必须立即拒绝，不能「先成功后对账」
  • 库存强一致扣减（少数场景）：超卖代价极高时用共识/锁
偏 AP 的路径
  • 商品详情页：可短暂展示旧库存，后台异步校准
  • 订单列表：最终一致 即可，配合状态机与对账

PACELC 延伸（了解即可）

无分区时仍要在延迟（L）与一致（C）间权衡：例如跨地域复制常降低一致级别以换更低延迟。设计评审时可问：「正常时要多一致？分区时要多可用？」

常见误区
  • 给整个系统贴一个 CP/AP 标签 — 应按接口/数据分别决策。
  • 认为选 AP 就不要一致性 — 最终一致 + 对账同样是一致性策略。
  • 忽略 P — 局域网也会分区，云多 AZ 更常见。
先想 10 秒：大促分区时，商品详情页更应该保 C 还是 A？

多数选 A（短暂旧库存 + 异步校准）；支付/防超卖路径才靠近 C。勿对全站贴单一 CAP 标签。

强一致、最终一致与因果一致

「一致性」在不同语境含义不同：数据库事务、副本同步、客户端会话。下表是架构讨论中最常用的划分。

模型含义（简化）典型实现/场景
强一致 / 线性一致 所有节点如同一份拷贝，读写全局有序 etcd/ZooKeeper、部分分布式锁；代价是延迟与可用性
最终一致 停止写入后，副本会在有限时间内收敛 Cassandra、CouchDB、Kafka 消费滞后；须幂等与对账
因果一致 有因果关系的操作被所有节点按因果序看到 会话粘滞 + 版本向量；评论回复链

BASE 与互联网业务常见策略

BASE 是对 AP 路径的实操总结（与 ACID 相对）：

  • Basically Available — 核心路径尽量有响应，非核心可降级。
  • Soft state — 中间状态允许存在（处理中、待对账）。
  • Eventually consistent — 通过复制、消息、定时任务达到一致。

ACID vs BASE（单服务 vs 跨服务）

维度ACID（单库事务）BASE（跨服务协作）
边界 一个数据库连接内 多服务、多库、消息
失败 回滚 补偿、重试、对账
ShopFlow 单服务内订单行 + 明细 下单 → 库存 → 支付链（详见 分布式数据：事务、幂等与 Saga）

代码：Transactional Outbox 预览

「写订单表 + 发 Kafka」无法靠本地事务一起提交。常见做法：Outbox 与业务写入同一事务，再由投递器异步发消息。

java 
@Transactional
public void createOrder(CreateOrder cmd) {
  orderRepository.save(cmd.toOrder());
  outboxRepository.save(OutboxEvent.orderCreated(cmd.orderId()));
  // 同一 DB 事务；消息由 poller 异步投递，消费者须幂等
}

互联网常见组合拳

  • 幂等键 — 重复请求返回同一业务结果。
  • 对账任务 — 日终比对订单、支付、库存流水。
  • 状态机 — 订单 CREATED → PAID，非法跳转拒绝。
  • 降级开关 — 分区时关闭非核心功能，保护核心下单路径。

分布式锁、租约与 fencing token 概念

跨节点互斥（定时任务抢主、库存扣减串行化）常用分布式锁。实现须考虑：锁持有者崩溃、时钟漂移、旧锁未释放就获得新锁。

  • 租约（Lease） — 锁带 TTL；持有者须周期性续租，否则自动释放。
  • Fencing token — 单调递增令牌；下游存储拒绝比当前令牌更旧的写，防止「僵尸持有者」覆写新主。

旧主恢复后若无 fencing 可能覆写新主
sequenceDiagram

  participant L as 锁服务
  participant DB as 存储
  N1->>L: 曾持有 lock#5
  Note over N1: GC 停顿后「以为」仍持有锁

存储侧记录「已见过的最大 token」，拒绝更小 token 的写。

代码对比：锁值是否单调

不推荐 · 仅 UUID 锁值
java 
// 下游无法区分「旧主迟到的写」

推荐 · 租约 + fencing token
java 
long token = lockService.acquire("inventory-shard-1", Duration.ofSeconds(30));

// 仓储层: if (token < lastAppliedToken) reject;

Java 生态常见组件：Redis Redisson、ZooKeeper Curator、数据库 SELECT … FOR UPDATE（有单点边界）。能不用锁就不用：优先幂等、分区键串行、CRDT 等业务手段。

读己之写、单调读等客户端语义

即使存储标记为「最终一致」，产品仍常要求会话级体验保证（由网关粘滞、读写路由或版本号实现）：

语义用户可感知的效果ShopFlow 例子
读己之写（Read-your-writes） 刚下单后刷新列表能看到自己的订单 写主库后读主或带 userId 路由到含最新会话的副本
单调读（Monotonic reads） 不会先看到新评论再看到旧评论 订单时间线只前进；用递增 version 过滤
单调写（Monotonic writes） 同一用户的写按顺序可见 同一 orderId 状态变更串行处理
写跟随（Writes-follow-reads） 改头像后评论里头像一起更新 用户资料变更广播或短期读主

否。须定义收敛时间、业务补偿与客户端语义；金融类路径仍可能要求强一致。

本章结论

本章回答「分区与复制时数据有多新」。下面几条是验收口径，可对照各节与 ShopFlow 接口回忆。

  • CAP 的工程问法：真实部署须接受分区容忍 P；分区发生时只能在一致性 C 与可用性 A 间取舍，且应按接口/数据集分别决策，勿给全系统贴单一 CP/AP 标签。
  • 一致模型落到接口：强一致、最终一致、因果一致含义不同；POST /reserve 防超卖宜强一致或串行化，商品详情等读多场景可接受短暂旧读。
  • BASE 是 AP 路径的实操：基本可用 + 软状态 + 最终一致；跨服务靠幂等、对账、状态机与 Outbox 等组合拳收敛。
  • 锁是最后手段：优先幂等与业务串行；必须用分布式锁时要租约续期与 fencing token，防止旧主迟到的写破坏新主。
  • 客户端语义不可省：除存储层一致级别外，还须定义读己之写等会话体验（粘滞、短期读主、版本号）。

分层单体与模块化单体的边界

ShopFlow · 创业期 8 人团队 订单、库存、支付逻辑都在一个 Spring Boot 工程里，按 controller / service / repository 分层。发布一次全量上线。此时是「坏单体」还是「合理起点」？

分层单体（Layered Monolith）：按技术层划分包结构，所有业务能力共享同一部署单元与数据库。优点是上手快、本地调试简单、事务在进程内完成；风险是边界模糊时演变成「大泥球」——改订单逻辑可能误触库存表。

典型分层（概念）

分层单体 · 请求自上而下
flowchart TB
  UI[Web / API Controller] --> APP[Application Service]
  APP --> DOM[Domain Model]
  DOM --> INF[Infrastructure / JPA]
  INF --> DB[(单一数据库)]

所有业务共享一个 DB 与一次发布；适合团队小、领域尚在高速试错阶段。

模块化单体（Modular Monolith）

仍在一个进程、一次部署，但按业务能力或限界上下文划分子模块（Maven/Gradle 多模块），模块间通过显式 API 通信，禁止随意跨模块访问对方表。

  • shopflow-order、shopflow-inventory、shopflow-payment 模块，共享 shopflow-app 启动器。
  • 模块内可继续分层；模块间仅暴露接口（Java interface / 应用服务门面）。
  • 为将来拆成独立服务做「排练」：边界清晰后再加网络与注册发现。
大泥球单体
  • 按技术层分包，订单 Service 直接注入库存 Repository
  • 无模块依赖规则，循环依赖普遍
  • 「先上线再说」，领域语言在代码里混杂
模块化单体
  • 按 Order / Inventory / Payment 划模块，跨模块只走公开 API
  • ArchUnit / 模块依赖图约束「inventory 不得依赖 order 内部实体」
  • 单库可多 schema 或表前缀，但所有权按模块划分

SOA、ESB 与微服务的演进脉络

2000 年代前后，面向服务架构（SOA） 强调将企业能力封装为可复用服务，常配合企业服务总线（ESB） 做路由、转换、编排与集中治理。

演进时间线（简化）

企业架构风格演进
timeline
  title 从集中到分布式的常见路径
  2000s : 分层单体 / 大泥球
  2005-2015 : SOA + ESB 集中编排
  2014+ : 微服务 + 轻量消息总线 / API 网关
  近年 : 模块化单体复兴 + 选择性微服务

不是「微服务取代一切」，而是按组织成熟度在谱系上移动。

SOA + ESB 的典型拓扑

ESB 居中编排

  I[库存系统] --> ESB
  P[支付系统] --> ESB
  ESB --> T[转换 / 路由 / 编排规则]

所有集成逻辑汇聚到总线：强大但易成瓶颈与「分布式大泥球」的控制中心。

微服务的关键转向

  • 智能端点， dumb pipe：业务逻辑在服务内；管道尽量简单（HTTP/gRPC、消息中间件），避免 ESB 里堆满业务规则。
  • 去中心化数据：每服务自有数据库（或 schema），不再默认「全企业一个库」。
  • 独立部署：团队拥有从设计到运行的全生命周期（DevOps）。
  • ESB 团队成为变更瓶颈，排期长
  • 编排逻辑藏在配置里，难以测试与版本化
  • 技术栈被总线产品绑定
  • 服务间契约（OpenAPI / 事件 schema）+ 消费者驱动测试
  • API 网关 做横切（路由、鉴权），业务编排尽量在域内
  • Spring Cloud Gateway + 注册发现，替代「胖 ESB」

康威定律与团队拓扑

康威定律（Conway's Law）：设计系统的组织，其产生的设计等价于组织内的沟通结构。换言之一一你的架构图最终会像你的组织架构图。

对微服务拆分的含义

  • 若订单与库存由同一小队维护，硬拆成两个远程服务只会增加沟通成本，而不会 magically 变快。
  • 若支付由合规独立团队负责，payment-service 作为独立部署单元更自然。
  • 平台团队提供注册发现、网关、CI 模板，让业务小队专注域逻辑（Team Topologies 中的 Platform / Enabling）。

服务边界与「能独立决策、独立发布」的团队边界尽量同构；跨团队集成走稳定契约。

反模式：组织未拆，服务先拆

两个团队仍共用一个 backlog，却维护六个微服务 — 变更仍需全员协调，却背上分布式排障成本。这叫分布式单体（领域建模、边界上下文与 API 契约 会进一步约束边界）。

何时不应拆微服务

下列信号出现越多，越应推迟物理拆分，优先模块化单体或合并服务。

  • 边界不清：Order 与 Inventory 实体互相引用，无法一句话说清谁拥有哪张表。
  • 无自动化：没有 CI/CD、容器化、集中日志与链路追踪，却要先上 8 个服务。
  • 事务密集：大量跨模块强一致 ACID 需求，拆后只能上重型分布式事务。
  • 团队过小：5 人维护 12 个服务，人均运维负担超过业务开发。
  • 为技术而拆：「微服务很潮」但流量与组织规模仍适合单体。

渐进路径

  1. 模块化单体 — 在进程内练边界与契约。
  2. 绞杀者（Strangler） — 新能力在新服务，旧模块逐步下线（演进模式、反模式与架构评审 再展开）。
  3. 选择性拆分 — 先拆变化最快或需独立扩缩的上下文（如 Inventory 读多写多）。
先想 10 秒：ShopFlow 只有 6 人，是否应立即拆成 order / inventory / payment 三个独立部署？

通常否：优先模块化单体 + 清晰门面；当发布阻塞或团队分组成熟后再拆。拆早会把 CAP、BASE 与一致性模型 中的 CAP、部分失败成本全部兑现且无人力消化。

本章结论

每条对应上一节 h3，用于架构评审前的 30 秒复述。

  • 单体谱系：分层单体按技术分包；模块化单体按业务能力/限界上下文分模块，单进程单部署，是拆微服务前的低成本排练。
  • 演进脉络：SOA+ESB 集中编排易成瓶颈；微服务强调智能端点、简单管道与去中心化数据，Spring Cloud 网关等承担横切而非「胖总线」。
  • 利弊权衡：微服务换独立发布与技术异构，付分布式事务、可观测与集成测试代价；边界未清时收益为负。
  • 康威定律：服务边界应与能独立交付的团队拓扑对齐；组织未拆却服务先拆，往往得到分布式单体。
  • 不宜硬拆：边界模糊、无自动化、事务密集、团队过小或为简历拆服务时，优先模块化单体或绞杀者渐进迁移。

DDD 战略设计：限界上下文与子域

ShopFlow · 术语冲突 订单团队把「商品」叫 Product（含促销价）；库存团队把「商品」叫 Sku（含仓库位）。若共用一个实体类，改促销逻辑会误改库存表字段——这就是缺少限界上下文的信号。

领域驱动设计（DDD）战略设计关注「大问题如何切」：识别子域（Subdomain）类型，并为每个核心能力划定限界上下文（Bounded Context）。

子域三分法

类型含义ShopFlow 示例
核心域 差异化竞争力，值得自研投入 订单编排、库存防超卖策略
支撑域 必需但非差异化，可适度定制 支付对接、物流通知
通用域 行业通用能力，优先买/用 SaaS 短信验证码、对象存储

限界上下文的关键属性

  • 统一语言（Ubiquitous Language）：上下文内术语一致；Order 里的 OrderLine 与 Inventory 里的 StockReservation 不必强行合并。
  • 模型边界：实体、聚合根、仓储仅在本上下文内完整；对外只暴露 DTO / 应用服务 / 事件。
  • 与部署单元的关系：理想上一上下文对应一个可独立演进的服务（或模块化单体中的一个模块），与 康威定律 对齐。

按业务能力拆分 vs 按技术层拆分

架构风格：单体、SOA 与微服务 已对比分层单体与模块化单体；本节从 DDD 视角强调：微服务边界应沿业务能力（限界上下文）切，而非沿 Controller / Service / Repository 等技术层切。

REST 资源建模、版本化与错误码规范

限界上下文对外通常以 RESTful HTTP API 暴露能力（practice 阶段亦用 OpenFeign 消费）。好的资源建模让调用方「读 URI 即懂语义」，并预留演进空间。

资源 URI 原则

  • 名词复数、表意资源：/orders、/skus/{skuId}/reservations，避免 /createOrder 动词路径。
  • 动作用语义子资源或 PATCH：取消订单 POST /orders/{id}/cancel 或 PATCH /orders/{id} + {"status":"CANCELLED"}。
  • 跨上下文 ID 引用：Order 只存 skuId，不在 URI 里嵌 inventory 内部仓库 ID，除非该 ID 已是稳定集成标识。
  • 幂等键：写操作（如 POST /reservations）要求 Header Idempotency-Key 或 body 内 requestId，应对 部分失败 重试。

契约优先：OpenAPI 与消费者驱动契约

契约优先（Contract First）：先写 OpenAPI 规范，再生成/实现 Controller 与客户端桩代码。这样 Order 与 Inventory 团队在编码前就对齐字段、错误码与幂等语义。

OpenAPI 片段（Inventory · 预留库存）

openapi: 3.1.0
info:
  title: ShopFlow Inventory API
  version: 1.0.0
paths:
  /api/v1/reservations:
    post:
      operationId: createReservation
      parameters:
        - name: Idempotency-Key
          in: header
          required: true
          schema: { type: string, format: uuid }
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [orderId, skuId, quantity]
              properties:
                orderId: { type: string, format: uuid }
                skuId: { type: string }
                quantity: { type: integer, minimum: 1 }
      responses:
        '201':
          description: 预留成功
        '409':
          description: 库存不足或冲突

契约优先工作流

  1. 领域专家 + 开发在 PR 中评审 openapi/inventory-api.yaml。
  2. CI 校验 breaking change（如删除字段、改类型）。
  3. 实现方对照契约写 Spring MVC / WebFlux；消费方用 OpenFeign + 生成 DTO 或手写映射。
  4. 契约测试（Pact 等）在 CI 验证「消费者期望 ⊆ 提供者实现」。

消费者驱动契约（CDC）

在 微服务 集成中，调用方（消费者） 定义「我需要哪些字段与状态码」，写入契约文件；提供方 在 CI 中验证能否满足。这样 inventory 重构内部表结构时，只要仍满足 order-service 的契约，集成测试即可通过。

本章结论

每条对应上一节 h3，用于评审前 30 秒复述。

  • 战略 DDD：用子域分类投入优先级；限界上下文划定统一语言与模型边界，跨上下文禁止共享实体与随意 JOIN。

本章先记住 3 件事
  • 同步要短：需要立即知道成败的步骤（预留库存）用请求/响应；链路过长须熔断、超时与幂等。

请求/响应、RPC 与 gRPC 适用场景

ShopFlow · 下单须当场知道库存 用户点击「提交订单」后，若 3 秒内看不到「库存不足」或「已预留」，体验不可接受。此时 order → inventory 的 POST /reservations 适合同步 HTTP。

请求/响应（Request/Response）是分布式中最直观的模式：调用方发请求、阻塞或异步等待响应。在 Java 微服务中常体现为 REST + OpenAPI 契约，practice 阶段用 OpenFeign 消费。

常见误区：把「能同步」当成「应该同步」——支付结果、积分通知等可接受延迟的步骤仍硬拉进同步链，会放大 部分失败 的级联影响。

消息队列、发布订阅与事件通知

当调用方不必立即知道下游处理结果，或需要削峰、解耦多订阅者时，使用异步消息。ShopFlow 中 PaymentCompleted、OrderCreated 等领域事件适合走 Kafka 等 MQ（详见 Kafka 事件驱动与 Redis 缓存）。

核心概念

  • 消息队列（Queue）：点对点，一条消息通常被一个消费者处理（任务分发）。
  • 发布/订阅（Pub/Sub）：一条事件可被多个订阅者各自处理（订单创建 → 库存、积分、通知各订一份）。
  • 领域事件：用过去式命名（OrderCreated、PaymentCompleted），载荷含最小集成字段 + 事件 ID。
  • 消费者组：同组内竞争消费保证扩展；不同组各自完整消费一遍。
ShopFlow · PaymentCompleted 发布订阅
flowchart LR
  Pay[payment-service] -->|publish| K[(Kafka topic)]
  K --> G1[order-service 消费者组]
  K --> G2[analytics 消费者组]
  G1 --> O[更新订单 PAID]
  G2 --> A[报表入库]

order 与 analytics 各用消费者组，互不影响进度；须幂等处理重复投递。

  • 须立即向用户反馈成败（预留库存、校验优惠券）
  • 链路短、调用方少、QPS 可控
  • 可接受 部分失败 语义已设计（幂等 + 查询）
  • 支付回调、发货通知、积分累计
  • 多下游订阅同一业务事实
  • 可接受秒级 最终一致

编排 vs 编舞（Choreography）

跨多个服务的业务流程（如下单 → 预留 → 支付 → 确认）如何协调？两种主流思路：编排（Orchestration） 由中央协调者按脚本推进；编舞（Choreography） 各服务监听事件、自行决定下一步，无中心大脑。

对比表

维度编排编舞
协调者 order-service 或独立 orchestrator 调用各步 各服务订阅事件，本地状态机驱动
可见性 流程集中在一点，易画序列图 流程分散，须链路追踪串联
耦合 协调者知悉所有步骤 API 服务只关心相关事件，添加订阅者较易
失败处理 常配合 Saga 补偿，由协调者触发 每服务自行补偿或发补偿事件

幂等键、去重与 Outbox 模式预览

异步与重试使「至少一次投递」成为常态。须从生产端与消费端双侧保证正确性。

消费端：幂等与去重

  • 幂等键：同步写操作 Header Idempotency-Key；异步用 eventId 或业务键。
  • 去重表：消费者处理前插入 processed_events(event_id)，重复则跳过。
  • 业务幂等：如「订单已 PAID 则忽略重复 PaymentCompleted」。

生产端：Transactional Outbox

问题：订单入库与发 Kafka 两个操作无法共用一个本地事务。 Outbox 将待发事件与业务写入同一数据库事务，后台投递器异步发到 MQ。

java 
@Transactional
void createOrder(CreateOrder cmd) {
  Order order = orderRepo.save(new Order(cmd));
  outboxRepo.save(new OutboxEvent(
    "OrderCreated",
    order.getId().toString(),
    toJson(new OrderCreatedPayload(order.getId(), ...))
  ));
  // 同一事务；投递器稍后 poll outbox → Kafka
}
先想 10 秒：消费者收到两次相同 eventId 的 PaymentCompleted，应如何处理？

第二次应识别为重复：查去重表或订单状态已为 PAID，直接 ack 不再改库。若仍重复扣减或重复发券，说明缺少消费端幂等。

Spring Cloud Gateway 路由与谓词过滤器

新增 shopflow-gateway 模块，依赖 spring-cloud-starter-gateway-server-webflux（Spring Cloud 2025 命名；WebFlux 网关为 ShopFlow 默认）。Gateway 从 Eureka 拉取实例，按路由（Route） = 谓词（Predicate）+ 过滤器（Filter）+ 目标 URI 转发流量。

最小路由：按路径前缀

yaml 
# gateway / application-local.yml（Spring Cloud 2025 属性前缀）
server:
  port: 8080
spring:
  application:
    name: api-gateway
  cloud:
    gateway:
      server:
        webflux:
          routes:
            - id: order-route
              uri: lb://order-service
              predicates:
                - Path=/api/orders/**
            - id: inventory-route
              uri: lb://inventory-service
              predicates:
                - Path=/api/inventory/**

lb:// 表示通过 Spring Cloud LoadBalancer 从 Eureka 选实例。若从旧版迁移，可用 spring-boot-properties-migrator 对照 spring.cloud.gateway.* → spring.cloud.gateway.server.webflux.*。

  • Path：路径匹配，最常用
  • Header、Method：按头或 HTTP 方法
  • After / Before：时间窗口（金丝雀可扩展）
  • AddRequestHeader：注入 traceId、租户 ID
  • StripPrefix：去掉路径前缀再转发
  • RequestRateLimiter：限流（advanced 章展开）

统一入口：鉴权头透传与路径重写

外部客户端只访问 https://api.shopflow.example（本地 :8080），Gateway 负责 TLS 终止（生产）、JWT 校验（安全与部署 章）与把用户身份传给下游。业务服务不应再对外暴露端口。

StripPrefix + 鉴权头透传

yaml 
spring:
  cloud:
    gateway:
      server:
        webflux:
          routes:
            - id: order-api
              uri: lb://order-service
              predicates:
                - Path=/api/orders/**
              filters:
                - StripPrefix=1
                - AddRequestHeader=X-ShopFlow-User-Id, #{headers['X-ShopFlow-User-Id']}
# 外部: GET /api/orders/123  →  order-service: GET /orders/123

本章仅演示头透传占位；完整 OAuth2 Resource Server 在 Gateway 统一验签后，将 sub 写入 X-ShopFlow-User-Id 即可。勿在 Gateway 写「下单编排」——那是 order-service 职责。

先想 10 秒：为何 StripPrefix=1 后，下游 Controller 仍映射 /orders/{id}？

对外 API 带 /api 命名空间便于版本与 BFF 区分；对内 微服务 保持领域路径简洁。StripPrefix 去掉第一段 api，避免每个 Controller 重复写前缀。
