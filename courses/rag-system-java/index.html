意图路由实现

CorpAssist · 用户意图识别用户输入「竞业限制补偿金标准」应路由到 Legal 知识库；「年假几天」路由到 HR 知识库；「VPN 连不上」路由到 IT 知识库。路由错误将导致检索不到相关文档。

意图路由是 RAG 系统中用户问题到知识库的映射层。CorpAssist 实现 规则 + LLM 双级路由：先试用关键词规则匹配（延迟 < 5ms），规则未命中时降级到 LLM 分类（延迟 ~200ms）。

// 双级意图路由器
@Component
public class IntentRouter {

    private final Map> keywordRules;

    public IntentRouter() {
        this.keywordRules = Map.of(
            "hr", List.of("年假", "考勤", "薪酬", "绩效", "PIP", "招聘", "社保", "公积金",
                          "加班", "请假", "工伤", "产假", "婚假"),
            "legal", List.of("竞业限制", "劳动合同", "第.*条", "合规", "保密协议",
                            "劳动仲裁", "赔偿金", "N\\+1", "经济补偿"),
            "it", List.of("VPN", "SSO", "LDAP", "密码重置", "网络故障", "邮箱",
                         "OA系统", "WiFi", "打印机", "软件安装")
        );
    }

    public RouteResult route(String query) {
        // 第一级: 关键词规则
        long start = System.nanoTime();
        for (var entry : keywordRules.entrySet()) {
            for (String keyword : entry.getValue()) {
                if (query.contains(keyword) || query.matches(keyword)) {
                    return new RouteResult(entry.getKey(), "rule", 0.95,
                        (System.nanoTime() - start) / 1_000_000);
                }
            }
        }

        // 第二级: LLM 分类（仅在规则未命中时）
        return llmClassify(query);
    }

    private RouteResult llmClassify(String query) {
        long start = System.nanoTime();
        var classification = callClassifierLLM(query);
        double latency = (System.nanoTime() - start) / 1_000_000;
        return new RouteResult(classification.kb(), "llm",
            classification.confidence(), latency);
    }

    private Classification callClassifierLLM(String query) {
        // 调用小模型 (gpt-4o-mini) 进行意图分类
        // Prompt: "将以下问题分类到 [hr, legal, it] 之一，输出 JSON:
        //          {\"kb\": \"hr\", \"confidence\": 0.92}"
        // 实现略...
        return new Classification("hr", 0.92);
    }

    public record RouteResult(String kb, String method,
                              double confidence, double latencyMs) {}
    public record Classification(String kb, double confidence) {}
}

@Service
public class MultiKbRagService {

    @Autowired private IntentRouter router;
    @Autowired private VectorStore vectorStore;

    public AskResponse ask(String question, String tenantId) {

        // 3. 组装回答

    }
}

CorpAssist 中不同角色（HR 专员、部门经理、普通员工）可见的文档范围不同。行级权限通过文档 metadata 中的 allowed_roles 字段实现，在检索时用 filterExpression 过滤。

// 行级权限过滤 — 写入时标记角色
@Service
public class AccessControlIngestionService {

    public Document applyAcl(Document doc, String role) {
        var metadata = doc.getMetadata();
        // 根据内容自动推断允许的角色
        String content = doc.getContent();
        Set allowedRoles = new HashSet<>(Set.of("employee")); // 全员可见

        if (content.contains("薪酬") || content.contains("绩效等级")) {
            allowedRoles.add("hr_specialist");
            allowedRoles.add("manager");
        }
        if (content.contains("高管") || content.contains("股权")) {
            allowedRoles.add("executive");  // 仅高管可见
        }
        if (content.contains("竞业") || content.contains("仲裁")) {
            allowedRoles.add("legal_team");
        }

        metadata.put("allowed_roles", String.join(",", allowedRoles));
        return doc;
    }
}

// 查询时过滤
@Service
public class AclSearchService {

    public List searchWithAcl(String query, String tenantId,
                                        String userRole, String kb) {
        var request = SearchRequest.builder()
            .query(query)
            .topK(6)
            .filterExpression(
                "tenant_id == '" + tenantId + "'"
                + " && kb == '" + kb + "'"
                + " && allowed_roles CONTAINS '" + userRole + "'"
            )
            .build();
        return vectorStore.similaritySearch(request);
    }
}

多租户索引隔离

多租户隔离有两种方案：文档级隔离（同集合，metadata tenant_id 过滤）和 集合级隔离（每租户独立集合）。CorpAssist 使用方案二的变体——每租户每知识库独立集合。

方案实现隔离级别运维复杂度查询延迟CorpAssist 选择
文档级（同集合）一个集合，tenant_id 字段软隔离（靠 filter）低随租户数线性增长—
集合级（每租户）corp-a-hr, corp-b-hr 等命名物理隔离中恒定√ 默认方案
数据库级（集群）每租户独立 Milvus 集群完全隔离高恒定仅合规要求极高时

自检：集合级隔离有什么缺点？什么场景下应该考虑文档级隔离？

集合级隔离缺点：1) 集合数随租户*知识库增长，运维成本高（如 50 租户*3KB=150 个集合）；2) 跨知识库查询需要并行查多个集合再合并结果。文档级隔离更适合租户数 > 100 的场景。缺点是 filter 遗漏会导致数据泄漏风险。CorpAssist 当前 5 个租户，集合级隔离更安全。

面试可能问

规则路由和 LLM 路由各自的优缺点？你如何组合？

规则路由优点：低延迟（< 5ms）、零成本、可解释、无 API 依赖。缺点：对未见变体敏感（用户说「年假折算」但规则只配了「年假」）、正则维护成本高。LLM 路由优点：语义理解、泛化好。缺点：延迟高（200ms）、成本（每次 $0.0002）、依赖外部 API。CorpAssist 组合策略：规则优先级高（命中则直接返回），全部规则未命中才降级到 LLM。命中率 85% 走规则，15% 走 LLM。平均路由延迟：2ms * 85% + 200ms * 15% ≈ 32ms。

allowed_roles 字段在 delete/update 操作时，角色映射关系如何维护？

角色映射关系应该由统一的 IAM 系统维护，RAG 服务仅在检索时使用当前用户的角色。文档的 allowed_roles 在入库时基于内容自动推断，定期由审计任务校验。若某个文档被修改，重新走 ingestion pipeline 更新 allowed_roles。不支持运行时动态修改——因为需要重新 embedding 整个文档。

多租户场景下，如何在应用层实现跨知识库的查询（如同时查 HR 和 Legal）？

实现方式：1) 将问题同时发往多个知识库的 VectorStore；2) 合并检索结果（RRF 融合）；3) 用 LLM 对合并结果重排。注意跨集合向量相似度分数不可比（不同集合的向量分布不同），建议使用 rank-based 的 RRF 融合而不是 score-based 的加权和。CorpAssist 的路由器可在配置中将部分查询标记为「multi_kb」，自动走并行检索路径。

场景 1：用户输入跨领域问题（如「竞业限制期间年假怎么算」），路由到单一 KB 都不合适。如何处理？

策略：1) 路由器检测到涉及多个 KB 的关键词（legal: 竞业限制 + hr: 年假），标记为 multi_kb；2) 并行查询 legal 和 hr 两个集合；3) 合并检索结果后让 LLM 综合两个知识库的 context 回答。在 IntentRouter 中增加 multiKbKeywords 配置，匹配时返回 RouteResult.multiKb=true。

场景 2：新入职一名 legal_team 角色的用户，发现他能看到所有 HR 文档？如何修复？

问题在 allowed_roles 的 filterExpression：CONTAINS 'legal_team' 条件匹配了 allowed_roles 中包含 legal_team 的文档。但 HR 文档的 allowed_roles 包含了 "employee","hr_specialist","manager" 但不包含 legal_team。排查步骤：1) 检查测试文档的 allowed_roles 值是否正确设置；2) 检查 filterExpression 语法——Milvus 的 CONTAINS 是数组包含，但不是所有版都支持。Milvus 2.3+ 用 json_contains 函数：json_contains(allowed_roles, 'legal_team')。

public class GraphRagIndexReader {

    public GraphIndex loadIndex(String basePath) throws IOException {
        var entities = readParquet(basePath + "/entities.parquet");
        var relationships = readParquet(basePath + "/relationships.parquet");
        var communities = readParquet(basePath + "/communities.parquet");

        return new GraphIndex(entities, relationships, communities);
    }

    private List> readParquet(String path) throws IOException {
        var records = new ArrayList>();
        try (var reader = ParquetReader.builder(new GroupReadSupport(),
                new Path(path)).build()) {
            Group group;
            while ((group = reader.read()) != null) {
                var record = new HashMap();
                for (int i = 0; i < group.getType().getFieldCount(); i++) {
                    var fieldName = group.getType().getFieldName(i);
                    var value = group.getValueToString(i, 0);
                    record.put(fieldName, value);
                }
                records.add(record);
            }
        }
        return records;
    }
}

            "community_level", 2,
            "response_type", "multiple pages",

    private GraphRagResult convertToResult(GraphRagResponse resp) {
        if (resp == null) return new GraphRagResult("", List.of());

        var sources = resp.contextData().stream()
            .map(d -> new SourceRef(
                d.docId(), d.text().substring(0, Math.min(100, d.text().length())), 0.0))
            .toList();

        return new GraphRagResult(resp.response(), sources);
    }

        Map metadata

结果合并与排序

Java BFF 同时维护两套检索路径：Naive RAG（向量检索）和 GraphRAG（图查询）。当路由器标记查询为多知识库关联问题时，并行执行两路检索并通过 RRF 合并。

java
// 统一检索服务 — 合并 Naive RAG + GraphRAG
@Service
public class UnifiedRetrievalService {

    @Autowired private VectorStore vectorStore;
    @Autowired private GraphRagHttpClient graphRagClient;
    @Autowired private IntentRouter intentRouter;

    public UnifiedResult retrieve(String query, String tenantId, String userRole) {
        var route = intentRouter.route(query);

        // 向量检索 (Naive RAG)
        var vecRequest = SearchRequest.builder()
            .query(query)
            .topK(6)
            .filterExpression("tenant_id == '" + tenantId
                + "' && allowed_roles CONTAINS '" + userRole + "'")
            .build();
        var vecDocs = vectorStore.similaritySearch(vecRequest);

        // 仅当跨文档查询时调用 GraphRAG
        GraphRagResult graphResult = null;
        if (route.confidence() > 0.7 || route.kb().equals("multi")) {
            graphResult = graphRagClient.localSearch(query, tenantId);
        }

        // 合并结果
        return mergeResults(vecDocs, graphResult, query);
    }

    private UnifiedResult mergeResults(
            List vecDocs, GraphRagResult graphResult, String query) {

        var allSources = new ArrayList();

        // 向量结果
        for (int i = 0; i < vecDocs.size(); i++) {
            var d = vecDocs.get(i);
            double rrfScore = 1.0 / (60 + i + 1);
            allSources.add(new SourceRef(
                (String) d.getMetadata().get("doc_id"),
                d.getContent().substring(0, Math.min(100, d.getContent().length())),
                rrfScore
            ));
        }

        // GraphRAG 结果
        if (graphResult != null && graphResult.sources() != null) {
            int offset = vecDocs.size();
            for (int i = 0; i < graphResult.sources().size(); i++) {
                var s = graphResult.sources().get(i);
                double rrfScore = 0.5 * (1.0 / (60 + i + 1));  // GraphRAG 权重减半
                allSources.add(new SourceRef(s.docId(), s.snippet(), rrfScore));
            }
        }

        // 按 RRF 分数排序，取 Top-6
        allSources.sort(Comparator.comparing(SourceRef::score).reversed());
        var topSources = allSources.stream()
            .limit(6)
            .peek(s -> s = new SourceRef(s.docId(), s.snippet(), Math.round(s.score() * 10000) / 10000.0))
            .toList();

        return new UnifiedResult(topSources, graphResult != null);
    }

    public record UnifiedResult(List sources, boolean graphRagUsed) {}
}

面试可能问

Java 侧为什么不直接运行 GraphRAG 索引而要依赖 Python 侧产出？

原因：1) 微软 GraphRAG 的实体抽取使用 LLM 逐 chunk 处理，Python 的 asyncio + LangChain 生态更适合此模式（Java 虚拟线程虽好但生态不成熟）；2) GraphRAG 社区检测依赖 Python 科学计算栈（NetworkX、Leidenalg），Java 无等效库；3) 索引是非实时离线任务，Python 脚本更灵活。Java 侧专注于在线查询（低延迟、高并发），索引交给 Python 批处理。

List 出至少 3 个 GraphRAG 查询不如直接向量检索的场景。

1) 事实性单文档问答（「我们公司年假最低几天」）——向量检索 45ms 完成，GraphRAG 需要 2s+ 走整个图查询流程，准确率差异 < 5%；2) 实时更新场景（「今天下午的会议取消了吗」）——GraphRAG 索引更新需重新运行整个管线（18 分钟），向量库可以实时 upsert；3) 低成本场景（免费 tier 用户）——GraphRAG 每次查询 $0.005 vs 向量 $0.002，索引成本更相差 44 倍。

如果 GraphRAG 服务不可用，你的降级策略是什么？

CorpAssist 的降级策略分三级：1) 熔断：检测到 GraphRAG API 连续 3 次超时（>5s），启动熔断器（Resilience4j CircuitBreaker），后续查询仅走 Naive RAG；2) 半熔断：5 分钟后尝试半开，发 1 条探测请求，成功后逐步恢复 10% 流量；3) 降级通知：在响应 metadata 中添加 {"degraded": true, "reason": "graphrag_unavailable"}，前端可显示「部分检索源不可用」提示。熔断期间 Naive RAG 召回率从 93% 降至 76%（跨文档场景），但核心功能可用。

场景 1：GraphRAG HTTP 调用返回空结果，但索引 Parquet 中有数据，排查步骤？

1) 确认 Python GraphRAG 服务是否已加载索引（检查服务启动日志）；2) 检查请求的 tenant_id 是否匹配索引数据路径；3) community_level 参数是否合理（level=0 粒度过细可能无社区，level=3 摘要过粗；推荐 level=2）；4) 确认 Python 侧 GraphRAG API 的 query 路由是否正确（local vs global，选错会返回空）。

场景 2：合并结果中 GraphRAG 的 source 排名远低于向量 source，即使 GraphRAG 内容更相关。如何优化？

当前 GraphRAG 的 RRF 权重为 0.5 倍（减半），这可能是原因。建议：1) 对跨文档查询场景，将 GraphRAG 权重从 0.5 提升至 0.8；2) 使用 BGE-Reranker 对合并结果重排而非 RRF 融合；3) 如果向量检索噪声大，降低 TopK 或在 RRF 中对向量侧加 penalty。调优方法类似 practice-03 的 ablation 实验。

@Configuration
public class SplitterConfig {

    @Bean
    public TokenTextSplitter tokenSplitter() {
        return TokenTextSplitter.builder()
            .withChunkSize(800)          // 每个 chunk 的 token 数上限
            .withChunkOverlap(200)       // 相邻 chunk 重叠 token 数
            .withTokensPerWord(1.3f)     // 中文占 token 比例修正
            .build();
    }

    // 按文档类型选择 splitter
    public DocumentTransformer selectSplitter(String docType) {
        return switch (docType) {
            case "policy" -> TokenTextSplitter.builder()
                .withChunkSize(600).withChunkOverlap(150).build();  // 政策条款宜小
            case "faq" -> TokenTextSplitter.builder()
                .withChunkSize(300).withChunkOverlap(50).build();   // FAQ 问答对独立

                .withChunkSize(1200).withChunkOverlap(300).build(); // 操作手册宜保持上下文
            default -> tokenSplitter();
        };
    }
}

完整的 RAG Pipeline 需要将 索引 和 查询 两个阶段串联。引用溯源的实现方案是：在 QuestionAnswerAdvisor 的回调中捕获检索到的 Document 列表，将其转为 SourceRef 附加到响应中。

java
// 自定义 Advisor — 带引用溯源的 QuestionAnswerAdvisor
@Component
public class TraceableQuestionAnswerAdvisor implements CallAroundAdvisor {

    private final VectorStore vectorStore;
    private final int topK;

    @Override
    public AdvisorResponse aroundCall(AdvisorChain chain, UserMessage userMsg,
                                      Map context) {
        // 1. 检索阶段
        String query = userMsg.getText();
        var searchRequest = SearchRequest.builder()
            .query(query)
            .topK(topK)
            .similarityThreshold(0.45)
            .filterExpression("tenant_id == '" + context.get("tenant_id") + "'")
            .build();

        List docs = vectorStore.similaritySearch(searchRequest);

        // 2. 将检索文档注入 user message 的 context
        String contextStr = docs.stream()
            .map(d -> "[doc_id=" + d.getMetadata().get("doc_id") + "] " + d.getContent())
            .collect(Collectors.joining("\n---\n"));
        String enhancedPrompt = """
            请根据以下文档内容回答问题。若文档中无答案请说不知道。

            参考文档：
            %s

            问题：%s
            """.formatted(contextStr, query);

        // 3. 继续链式调用（后续可能还有别的 advisor）
        AdvisorResponse response = chain.nextAroundCall(
            new UserMessage(enhancedPrompt), context);

        // 4. 将 sources 写入 context 供 controller 提取
        List sources = docs.stream()
            .map(d -> new SourceRef(
                (String) d.getMetadata().get("doc_id"),
                d.getContent().substring(0, Math.min(100, d.getContent().length())),
                (double) d.getMetadata().getOrDefault("distance", 0.0)
            ))
            .toList();
        context.put("rag_sources", sources);

        return response;
    }
}

自检：Spring AI 的 QuestionAnswerAdvisor 与 Python LCEL 的 RunnableParallel 有何异同？

相同：两者都实现了「检索 → 拼 context → 注入 Prompt → 调 LLM」的 RAG 核心流程。不同：QuestionAnswerAdvisor 是 ChatClient 的 advisor（类似拦截器/中间件），可组合多个 advisor；RunnableParallel 是链式管道的组成节点，通过 | 连接。Spring AI advisor 模式更接近 AOP，LCEL 管道模式更接近函数组合。

  • DocumentReader：Spring AI 支持 PDF/Word/JSON/TXT 四种 Reader，PDF 每页产出一个 Document，Word 按段落拆分

面试可能问

Spring AI 的 Advisor 链和 Python LangChain 的 Chain 模式在设计上有什么不同？

Spring AI Advisor 模式基于 AOP 思想——每个 Advisor 包装（wrap）在被调用方法周围，形成一个洋葱模型。便于横切关注点如日志、鉴权、引用提取。LangChain Chain 模式基于组合模式——通过 | 运算符将多个 Runnable 连接成有向图。两者模式不同但都是装饰器变体。Spring AI 更擅长在 Java 生态中与其他 Spring AOP 组件集成。

如果 PDF 中有表格，PagePdfDocumentReader 解析后表格会变成什么？如何处理？

PagePdfDocumentReader 基于 Apache PDFBox，默认将表格渲染为纯文本（单元格内容按坐标拼成行，丢失表格结构）。处理方案有两层：1) 在 PDF 解析前先用 OCR+表格识别工具（如 Camelot/Tabula）将表格转为 Markdown 再喂入 Reader；2) 在 withPageExtractedTextFormatter 中自定义行列拼接逻辑。CorpAssist 对含表格的 PDF 先使用 Python Pandas 转为 CSV 再入库。

如何测试 Spring AI Pipeline 的每个环节是否正常工作？

分段测试：1) DocumentReader — 单测验证 Document 数量和 metadata；2) Splitter — 验证 chunk token 数在 chunkSize 范围内；3) VectorStore — 写入后 similaritySearch 验证返回预期文档；4) Advisor — Mock VectorStore 验证 Prompt 拼装是否正确；5) Controller — MockMvc 验证 JSON 响应结构。Spring AI 提供 TestAdvisors 和 MockVectorStore 辅助测试。

场景 1：分块后的 Document 数量远少于预期（500 页 PDF 只产出 30 个 chunk），应排查什么？

可能原因：1) chunkSize 设置过大（如 4000），每个 Document（一页）显著小于一个 chunk 大小，splitter 无需拆分；2) PDF 的文本内容被渲染为图片，PDFBox 提取不出文字，reader 读取为空。验证方法：检查 reader.get().size() 的原始 Document 数，以及每个 Document 的内容长度。

// 多 Embedding 模型切换
@Service
public class MultiEmbeddingService {

    private final Map models;

    public MultiEmbeddingService(
            @Qualifier("openaiSmall") EmbeddingModel openaiSmall,
            @Qualifier("openaiLarge") EmbeddingModel openaiLarge,
            @Qualifier("bgeM3") EmbeddingModel bgeM3) {
        this.models = Map.of(
            "text-embedding-3-small", openaiSmall,
            "text-embedding-3-large", openaiLarge,
            "bge-m3", bgeM3
        );
    }

    // 按策略选择 embedding 模型
    public String selectModel(String kb, String query) {
        if (query.length() > 50) return "text-embedding-3-large";  // 长查询高精度
        if ("legal".equals(kb)) return "text-embedding-3-large";   // 法务需高精度
        return "text-embedding-3-small";   // 默认小模型
    }

    public List searchWithModel(String query, String kb,
                                          String tenantId, String modelName) {
        var embeddingModel = models.get(modelName);
        var vectorStore = createStore(embeddingModel, modelName);
        var request = SearchRequest.builder()
            .query(query)
            .topK(4)
            .filterExpression("tenant_id == '" + tenantId + "' && kb == '" + kb
                + "' && embed_model == '" + modelName + "'")
            .build();
        return vectorStore.similaritySearch(request);
    }

        var config = MilvusVectorStoreConfig.builder()

                "nlist", 1024,          // 聚类中心数，越大精度越高但建索引慢
                "nprobe", 16            // 查询时搜索的聚类数，越大召回越高但慢
            ))
            // gRPC 连接池
            .withGrpcClientConfig(GrpcClientConfig.builder()
                .withConnectTimeout(Duration.ofSeconds(10))
                .withKeepAliveTime(Duration.ofSeconds(30))
                .withKeepAliveTimeout(Duration.ofSeconds(10))
                .withIdleTimeout(Duration.ofSeconds(60))
                .withMaxRetries(3)
                .build())
            .build();
        return new MilvusVectorStore(embeddingModel, config);
    }
}

面试可能问

Milvus 的 IVF_FLAT 和 HNSW 索引有什么区别？你如何选型？

IVF_FLAT：聚类划分搜索空间，构建快（小时级/百万）、内存占用低，但精度受聚类数影响。HNSW：分层导航小世界图，构建慢（天级/百万）、内存高（是 IVF 的 3~5 倍），但查询快且精度高。CorpAssist 选择 IVF_FLAT（nlist=1024）的理由：数据集百亿级以下 IVF 精度可接受（recall@10=0.95）、构建速度快（500 篇文档索引创建 < 1 分钟）、资源消耗低（适合容器化部署）。

如何在不停机的情况下将 SimpleVectorStore 迁移到 Milvus？

方案：1) 在应用层实现双写模式——同时写入 Simple 和 Milvus（Feature Flag 控制）；2) 后台跑数据复制任务，将已有 Simple 数据批量迁入 Milvus；3) 灰度切换查询——先 10% 查询走 Milvus，验证准确率无差异后逐步放大流量；4) 切完后移除 SimpleVectorStore 的写路径和 Bean 声明。整体迁移窗口约 1~2 天。

多 Embedding 模型混合存储时，检索精度是否有损失？

如果查出的文档使用了不同模型，向量相似度的绝对分数不可比——text-embedding-3-small 的 0.85 分和 bge-m3 的 0.72 分不代表语义相似度差异。有两种策略：1) 分集合存储每个模型的向量（推荐），查询时指定对应集合；2) 同一集合用 embed_model 字段过滤（需确保 filter 生效，否则混合分数排序会失真）。CorpAssist 使用策略 1，集合命名 corpassist_{model_name}。

BM25 + 向量融合实现

CorpAssist · 法务条款检索法务查询「依据 2024 年劳动合同法第 18 条解答」需精确命中「第18条」关键词。向量检索可能匹配到「第17条」「第19条」的语义相似片段，需 BM25 精确词匹配纠正排名。

Spring AI 的 VectorStore 接口只定义了向量检索方法。要实现 BM25 混合，需要在应用中单独集成全文检索引擎。CorpAssist Java 侧选择 Lucene MemoryIndex（适用于集合量 < 10 万篇）或 Elasticsearch（适用于大规模），通过 RRF 算法融合两路结果。

// Lucene BM25 实现 — 内存全文索引
@Component
public class LuceneBm25Indexer {

    private final Analyzer analyzer = new IKAnalyzer();  // 中文分词
    private IndexWriter writer;
    private IndexReader reader;

    @PostConstruct
    public void init() throws IOException {
        var config = new IndexWriterConfig(analyzer);
        config.setOpenMode(IndexWriterConfig.OpenMode.CREATE_OR_APPEND);
        writer = new IndexWriter(new RAMDirectory(), config);
    }

    public void addDocument(String docId, String content, Map metadata) {
        try {
            var doc = new Document();
            doc.add(new StringField("doc_id", docId, Field.Store.YES));
            doc.add(new TextField("content", content, Field.Store.YES));
            metadata.forEach((k, v) ->
                doc.add(new StringField(k, v, Field.Store.YES)));
            writer.addDocument(doc);
            writer.commit();
        } catch (IOException e) {
            throw new RuntimeException("BM25 index error", e);
        }
    }

    public List search(String query, int topK, String tenantId) {
        try {
            reader = DirectoryReader.open(writer);
            var searcher = new IndexSearcher(reader);
            var parser = new QueryParser("content", analyzer);
            var boolQuery = new BooleanQuery.Builder()
                .add(parser.parse(query), BooleanClause.Occur.MUST)
                .add(new TermQuery(new Term("tenant_id", tenantId)),
                     BooleanClause.Occur.FILTER)
                .build();
            var hits = searcher.search(boolQuery, topK * 2);
            var results = new ArrayList();
            for (var hit : hits.scoreDocs) {
                var doc = searcher.doc(hit.doc);
                results.add(new Bm25Result(
                    doc.get("doc_id"), hit.score,
                    doc.get("content").substring(0, Math.min(100, doc.get("content").length()))
                ));
            }
            return results;
        } catch (IOException e) {
            throw new RuntimeException("BM25 search error", e);
        }
    }
}

public record Bm25Result(String docId, double score, String snippet) {}

// RRF 融合器
@Component
public class RrfFusion {

    private static final int RRF_CONSTANT = 60;

    public List fuse(
            List vectorResults,
            List bm25Results,
            double alpha,
            int finalK) {

        Map fused = new HashMap<>();
        Set allIds = new HashSet<>();

        // 向量结果
        for (int i = 0; i < vectorResults.size(); i++) {
            var r = vectorResults.get(i);
            allIds.add(r.docId());
            double rrf = alpha * (1.0 / (RRF_CONSTANT + i + 1));
            fused.merge(r.docId(), rrf, Double::sum);
        }

        // BM25 结果
        for (int i = 0; i < bm25Results.size(); i++) {
            var r = bm25Results.get(i);
            allIds.add(r.docId());
            double rrf = (1 - alpha) * (1.0 / (RRF_CONSTANT + i + 1));
            fused.merge(r.docId(), rrf, Double::sum);
        }

        return fused.entrySet().stream()
            .sorted(Map.Entry.comparingByValue().reversed())
            .limit(finalK)
            .map(e -> new FusedResult(e.getKey(), e.getValue()))
            .toList();
    }
}

public record FusedResult(String docId, double rrfScore) {}

BGE-Reranker 调用

RRF 融合后，Top-30 结果再经 BGE-Reranker（BAAI/bge-reranker-v2-m3）交叉编码排序。BGE-Reranker 基于 transformer 交叉编码架构，对 (query, doc) 对计算相关性分数，精度高于向量相似度。

// BGE-Reranker HTTP 客户端（服务端独立部署）
@Component
public class BgeRerankerClient {

    private final RestTemplate restTemplate;
    private final String endpoint;

    public BgeRerankerClient(
            @Value("${reranker.bge.endpoint}") String endpoint) {
        this.restTemplate = new RestTemplate();
        this.restTemplate.setMessageConverter(
            new StringHttpMessageConverter(StandardCharsets.UTF_8));
        this.endpoint = endpoint;
    }

    public List rerank(String query,
                                       List candidates) {
        var request = Map.of(
            "query", query,
            "documents", candidates.stream()
                .map(FusedResult::docId).toList(),
            "top_k", Math.min(candidates.size(), 10)
        );

        var response = restTemplate.postForObject(
            endpoint + "/rerank", request, RerankResponse.class);

        if (response == null || response.results() == null) {
            return candidates.stream()
                .map(c -> new RerankedResult(c.docId(), 0.0))
                .toList();
        }

        return response.results().stream()
            .map(r -> new RerankedResult(
                candidates.get(r.index()).docId(), r.relevanceScore()))
            .sorted(Comparator.comparing(RerankedResult::score).reversed())
            .toList();
    }

    // BGE 服务端 API 协议
    public record RerankRequest(String query, List documents, int topK) {}
    public record RerankResponse(List results) {}
    public record RerankItem(int index, double relevanceScore) {}
    public record RerankedResult(String docId, double score) {}
}

自检：为什么 CorpAssist 中 BM25 权重（1-alpha）高于向量权重（alpha）？

在 CorpAssist 法务/政策场景中，精确的条款匹配（「第18条」「竞业限制」）比语义匹配更关键。BM25 精确命中这些关键词的文档应该排在向量语义相似但未命中关键词的文档之前。所以 alpha=0.3（BM25 占 0.7 权重）是合理的。如果是语义搜索场景（如「描述一下公司文化」），则应调高 alpha 至 0.6~0.7。

// 完整管线延迟基准测试
@Component
public class FullPipelineBenchmark {

    @Autowired private VectorStore vectorStore;
    @Autowired private LuceneBm25Indexer bm25Indexer;
    @Autowired private RrfFusion rrfFusion;
    @Autowired private BgeRerankerClient reranker;

    public Map benchmark(List queries, int warmup, int runs) {
        // Warmup
        for (int i = 0; i < warmup; i++) {
            runPipeline(queries.get(i % queries.size()));
        }

        var vecTimes = new ArrayList();
        var bmTimes = new ArrayList();
        var totalTimes = new ArrayList();

        for (int i = 0; i < runs; i++) {
            long start = System.nanoTime();
            var result = runPipeline(queries.get(i % queries.size()));
            long total = (System.nanoTime() - start) / 1_000_000;
            totalTimes.add(total);
            vecTimes.add((long) result.get("vecMs"));
            bmTimes.add((long) result.get("bmMs"));
        }

        return Map.of(
            "vector_only_p50", percentile(vecTimes, 50),
            "vector_only_p95", percentile(vecTimes, 95),
            "bm25_p50", percentile(bmTimes, 50),
            "bm25_p95", percentile(bmTimes, 95),
            "total_p50", percentile(totalTimes, 50),
            "total_p95", percentile(totalTimes, 95)
        );
    }

    private Map runPipeline(String query) {
        long vecStart = System.nanoTime();
        var vecResults = vectorStore.similaritySearch(
            SearchRequest.builder().query(query).topK(30).build());
        long vecEnd = System.nanoTime();

        long bmStart = System.nanoTime();
        var bmResults = bm25Indexer.search(query, 30, "corp-a");
        long bmEnd = System.nanoTime();

        var fused = rrfFusion.fuse(
            vecResults.stream().map(d -> new VectorResult(
                (String) d.getMetadata().get("doc_id"), 0.0)).toList(),
            bmResults, 0.3, 15);

        var reranked = reranker.rerank(query, fused);

        return Map.of(
            "vecMs", (vecEnd - vecStart) / 1_000_000,
            "bmMs", (bmEnd - bmStart) / 1_000_000,
            "fusedCount", fused.size(),
            "finalCount", reranked.size()
        );
    }
}

面试可能问

RRF 的常数 k（通常 60）如果改大会有什么影响？

RRF 公式：score = sum(1/(k + rank))。k 越大，排名差异对分数的贡献越小（所有文档的 RRF score 趋近于相近值），融合结果接近等权平均。k 越小，第一名文档的 RRF 分数优势越大，Top-1 权重极高。k=60 是一个经验值：当排名从第 1 降到第 10 时，分数从 1/61≈0.0164 降到 1/70≈0.0143，降幅 13%，保持了排名差异但不过度极端。

为什么不在 Milvus 里用内置的 hybrid_search 而是自己实现 RRF？

Milvus 2.4+ 支持 hybrid_search 但需要同时配置向量索引和 scalar 倒排索引。限制：1) BM25 的全文检索能力有限（不支持中文分词自定义）；2) alpha 权重只能在集合级别配置，无法按查询动态调整；3) RRF 的常数固定不可调。自实现 RRF 更灵活：可按知识库调 alpha、可动态切换融合策略、可独立升级 BM25 组件。

BGE-Reranker 的交叉编码与向量相似度的双编码（bi-encoder）有什么本质区别？

双编码（bi-encoder）：query 和 doc 分别编码成独立向量，用余弦相似度计算相关分数。query 编码后固定，与所有 doc 向量做点积即可，计算量 O(N)。交叉编码（cross-encoder）：query 和 doc 拼接成一个序列送入 transformer 计算相关性分数。每个 (query, doc) 对需要一次完整前向推理，计算量 O(N*T^2)，精度更高但速度慢 10~100 倍。BGE-Reranker 属于交叉编码，所以需要对 Top-15 做 15 次推理，不能像 bi-encoder 那样批量点积。

场景 1：混合检索后 recall 反而低于纯向量检索，如何排查？

可能原因：1) BM25 索引中 tenant_id filter 与向量库 filter 不一致，检索到不同租户的数据；2) BM25 中文分词效果差（IKAnalyzer 词典未更新），专业名词被错误切分；3) alpha 设置不合理——如果向量召回已足够好（>85%），BM25 的低质量结果拉低了整体排名。验证方法：单独跑 BM25+RRF(alpha=0) 看 BM25 单路召回率。

场景 2：Rerank 阶段 P95 延迟达到 450ms，如何在不显著降低召回率的前提下优化？

策略：1) 减少 Rerank 候选数（Top15→Top10），延迟约降 33%，recall 降 < 2%；2) 对简单查询（长度 < 8 字）跳过 Rerank，仅对复杂查询执行；3) 使用 ASR（Adaptive Scaling）——当 rerank 排队超过 5 个时自动降级为不 rerank；4) 升级 GPU 实例或使用批量推理（batch_size=32 比串行推理快 8 倍）。

集成测试设计

CorpAssist · Pipeline 回归测试每次修改 TokenTextSplitter 参数或 Advisor 逻辑时，测试需要验证：1) 检索到的文档是否正确；2) Advisor 是否能正确拼装 context；3) 最终响应结构和 Python 侧一致。

// Java 侧评测输出 — 供 RAGAS 消费
@Component
public class EvalRunner {

    @Autowired private ChatClient ragChatClient;

    public void runEval(String evalPath, String outputPath) throws IOException {
        var mapper = new ObjectMapper();
        var evalSet = mapper.readValue(
            new File(evalPath), EvalItem[].class);

        var results = new ArrayList>();
        for (var item : evalSet) {
            var response = ragChatClient.prompt()
                .user(item.question())
                .call();
            var context = response.getContext();
            @SuppressWarnings("unchecked")
            var sources = (List) context.get("rag_sources");

            var record = Map.of(
                "question", item.question(),
                "answer", response.getResult().getOutput().getText(),
                "contexts", sources.stream().map(SourceRef::snippet).toList(),
                "ground_truth", item.groundTruth()
            );
            results.add(record);
        }

        mapper.writerWithDefaultPrettyPrinter()
            .writeValue(new File(outputPath), results);
        System.out.printf("Eval output written to %s (%d queries)%n",
            outputPath, results.size());
    }
}

面试可能问

Mock ChatModel 是否可能漏测 Prompt 质量问题？

Mock ChatModel 只验证管线是否按预期传参（context 是否拼入、filter 是否正确），不验证 Prompt 质量（措辞、约束）。Prompt 质量需要通过 RAGAS 评测（faithfulness/relevancy）来验证。测试策略分层：单元测试（Mock LLM）验证编排逻辑，集成测试（真实 LLM 但小样本）验证 Prompt 效果，全量评测（CI 闸门）验证回归。

Java 和 Python 的评测结果应该完全一致吗？

不期望完全一致，两栈的 SystemPrompt 略有不同（Java 的约束更严格以满足合规审计），LLM 调用参数也可能不同（temperature）。但检索阶段指标（context_precision/recall）应几乎一致（delta < 0.02），生成阶段指标（faithfulness）应在同一量级（delta < 0.03）。若偏差 > 0.05 需查两栈的差异。

CI 闸门的阈值应该怎么设定？

方法：先跑 10 次历史 PR 的评测，取各指标的均值和标准差。设置阈值为 mean - 1.5*std（安全阀）或 mean - 2*std（宽松阀）。CorpAssist 使用上述方法且动态更新：每季度重新计算阈值，确保评测集扩展后阈值仍然合理。初始值：faithfulness>=0.85，context_recall>=0.80，这是经 200+ PR 验证的合理底线。
