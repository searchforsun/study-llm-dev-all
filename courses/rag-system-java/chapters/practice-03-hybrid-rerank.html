本章要点速记

  • Spring AI 无内置 BM25，通过 Lucene MemoryIndex 或 Elasticsearch 实现
  • RRF 融合用 rank 而非 score，解决跨模型分数不可比
  • BGE-Reranker 需独立部署（HTTP/gRPC），不可内嵌 JVM
  • CorpAssist 最优 alpha=0.3（向量权重），与 Python 侧一致

BM25 + 向量融合实现

CorpAssist · 法务条款检索法务查询「依据 2024 年劳动合同法第 18 条解答」需精确命中「第18条」关键词。向量检索可能匹配到「第17条」「第19条」的语义相似片段，需 BM25 精确词匹配纠正排名。

Spring AI 的 VectorStore 接口只定义了向量检索方法。要实现 BM25 混合，需要在应用中单独集成全文检索引擎。CorpAssist Java 侧选择 Lucene MemoryIndex（适用于集合量 < 10 万篇）或 Elasticsearch（适用于大规模），通过 RRF 算法融合两路结果。

java
// Lucene BM25 实现 — 内存全文索引
@Component
public class LuceneBm25Indexer {

    private final Analyzer analyzer = new IKAnalyzer();  // 中文分词
    private IndexWriter writer;
    private IndexReader reader;

    @PostConstruct
    public void init() throws IOException {
        var config = new IndexWriterConfig(analyzer);
        config.setOpenMode(IndexWriterConfig.OpenMode.CREATE_OR_APPEND);
        writer = new IndexWriter(new RAMDirectory(), config);
    }

    public void addDocument(String docId, String content, Map metadata) {
        try {
            var doc = new Document();
            doc.add(new StringField("doc_id", docId, Field.Store.YES));
            doc.add(new TextField("content", content, Field.Store.YES));
            metadata.forEach((k, v) ->
                doc.add(new StringField(k, v, Field.Store.YES)));
            writer.addDocument(doc);
            writer.commit();
        } catch (IOException e) {
            throw new RuntimeException("BM25 index error", e);
        }
    }

    public List search(String query, int topK, String tenantId) {
        try {
            reader = DirectoryReader.open(writer);
            var searcher = new IndexSearcher(reader);
            var parser = new QueryParser("content", analyzer);
            var boolQuery = new BooleanQuery.Builder()
                .add(parser.parse(query), BooleanClause.Occur.MUST)
                .add(new TermQuery(new Term("tenant_id", tenantId)),
                     BooleanClause.Occur.FILTER)
                .build();
            var hits = searcher.search(boolQuery, topK * 2);
            var results = new ArrayList();
            for (var hit : hits.scoreDocs) {
                var doc = searcher.doc(hit.doc);
                results.add(new Bm25Result(
                    doc.get("doc_id"), hit.score,
                    doc.get("content").substring(0, Math.min(100, doc.get("content").length()))
                ));
            }
            return results;
        } catch (IOException e) {
            throw new RuntimeException("BM25 search error", e);
        }
    }
}

public record Bm25Result(String docId, double score, String snippet) {}

// RRF 融合器
@Component
public class RrfFusion {

    private static final int RRF_CONSTANT = 60;

    public List fuse(
            List vectorResults,
            List bm25Results,
            double alpha,
            int finalK) {

        Map fused = new HashMap<>();
        Set allIds = new HashSet<>();

        // 向量结果
        for (int i = 0; i < vectorResults.size(); i++) {
            var r = vectorResults.get(i);
            allIds.add(r.docId());
            double rrf = alpha * (1.0 / (RRF_CONSTANT + i + 1));
            fused.merge(r.docId(), rrf, Double::sum);
        }

        // BM25 结果
        for (int i = 0; i < bm25Results.size(); i++) {
            var r = bm25Results.get(i);
            allIds.add(r.docId());
            double rrf = (1 - alpha) * (1.0 / (RRF_CONSTANT + i + 1));
            fused.merge(r.docId(), rrf, Double::sum);
        }

        return fused.entrySet().stream()
            .sorted(Map.Entry.comparingByValue().reversed())
            .limit(finalK)
            .map(e -> new FusedResult(e.getKey(), e.getValue()))
            .toList();
    }
}

public record FusedResult(String docId, double rrfScore) {}

BGE-Reranker 调用

RRF 融合后，Top-30 结果再经 BGE-Reranker（BAAI/bge-reranker-v2-m3）交叉编码排序。BGE-Reranker 基于 transformer 交叉编码架构，对 (query, doc) 对计算相关性分数，精度高于向量相似度。

java
// BGE-Reranker HTTP 客户端（服务端独立部署）
@Component
public class BgeRerankerClient {

    private final RestTemplate restTemplate;
    private final String endpoint;

    public BgeRerankerClient(
            @Value("${reranker.bge.endpoint}") String endpoint) {
        this.restTemplate = new RestTemplate();
        this.restTemplate.setMessageConverter(
            new StringHttpMessageConverter(StandardCharsets.UTF_8));
        this.endpoint = endpoint;
    }

    public List rerank(String query,
                                       List candidates) {
        var request = Map.of(
            "query", query,
            "documents", candidates.stream()
                .map(FusedResult::docId).toList(),
            "top_k", Math.min(candidates.size(), 10)
        );

        var response = restTemplate.postForObject(
            endpoint + "/rerank", request, RerankResponse.class);

        if (response == null || response.results() == null) {
            return candidates.stream()
                .map(c -> new RerankedResult(c.docId(), 0.0))
                .toList();
        }

        return response.results().stream()
            .map(r -> new RerankedResult(
                candidates.get(r.index()).docId(), r.relevanceScore()))
            .sorted(Comparator.comparing(RerankedResult::score).reversed())
            .toList();
    }

    // BGE 服务端 API 协议
    public record RerankRequest(String query, List documents, int topK) {}
    public record RerankResponse(List results) {}
    public record RerankItem(int index, double relevanceScore) {}
    public record RerankedResult(String docId, double score) {}
}

权重调优与 Ablation

在 CorpAssist 法务测试集（200 条需精确条款匹配的查询）上对 alpha 和 rerank 策略进行 ablation 实验。

混合 RRF alpha=0.389.7%0.63868ms核心方案

Ablation 结论：1) 仅 BM25 或仅向量召回均不足（~70%），RRF 融合提升至 ~90%；2) Rerank 将 Top30 候选重排至高精度 Top5，recall@10 再提升 4.4 个百分点；3) Rerank 候选数从 30 降至 15，延迟从 320ms 降至 185ms（降 42%），而 recall 仅降 1.8%。CorpAssist 采用混合 RRF(alpha=0.3) + Rerank(Top15→5) 作为默认生产配置。

自检：为什么 CorpAssist 中 BM25 权重（1-alpha）高于向量权重（alpha）？

在 CorpAssist 法务/政策场景中，精确的条款匹配（「第18条」「竞业限制」）比语义匹配更关键。BM25 精确命中这些关键词的文档应该排在向量语义相似但未命中关键词的文档之前。所以 alpha=0.3（BM25 占 0.7 权重）是合理的。如果是语义搜索场景（如「描述一下公司文化」），则应调高 alpha 至 0.6~0.7。

在 CorpAssist 性能测试环境（4C8G JVM, Milvus 集群, BGE GPU 节点）上，对完整管线进行延迟基准测试：

java
// 完整管线延迟基准测试
@Component
public class FullPipelineBenchmark {

    @Autowired private VectorStore vectorStore;
    @Autowired private LuceneBm25Indexer bm25Indexer;
    @Autowired private RrfFusion rrfFusion;
    @Autowired private BgeRerankerClient reranker;

    public Map benchmark(List queries, int warmup, int runs) {
        // Warmup
        for (int i = 0; i < warmup; i++) {
            runPipeline(queries.get(i % queries.size()));
        }

        var vecTimes = new ArrayList();
        var bmTimes = new ArrayList();
        var totalTimes = new ArrayList();

        for (int i = 0; i < runs; i++) {
            long start = System.nanoTime();
            var result = runPipeline(queries.get(i % queries.size()));
            long total = (System.nanoTime() - start) / 1_000_000;
            totalTimes.add(total);
            vecTimes.add((long) result.get("vecMs"));
            bmTimes.add((long) result.get("bmMs"));
        }

        return Map.of(
            "vector_only_p50", percentile(vecTimes, 50),
            "vector_only_p95", percentile(vecTimes, 95),
            "bm25_p50", percentile(bmTimes, 50),
            "bm25_p95", percentile(bmTimes, 95),
            "total_p50", percentile(totalTimes, 50),
            "total_p95", percentile(totalTimes, 95)
        );
    }

    private Map runPipeline(String query) {
        long vecStart = System.nanoTime();
        var vecResults = vectorStore.similaritySearch(
            SearchRequest.builder().query(query).topK(30).build());
        long vecEnd = System.nanoTime();

        long bmStart = System.nanoTime();
        var bmResults = bm25Indexer.search(query, 30, "corp-a");
        long bmEnd = System.nanoTime();

        var fused = rrfFusion.fuse(
            vecResults.stream().map(d -> new VectorResult(
                (String) d.getMetadata().get("doc_id"), 0.0)).toList(),
            bmResults, 0.3, 15);

        var reranked = reranker.rerank(query, fused);

        return Map.of(
            "vecMs", (vecEnd - vecStart) / 1_000_000,
            "bmMs", (bmEnd - bmStart) / 1_000_000,
            "fusedCount", fused.size(),
            "finalCount", reranked.size()
        );
    }
}

瓶颈分析：BGE-Reranker 占检索阶段延迟的 60%，主要耗时在模型推理和 HTTP 传输。优化方向：1) 将候选数从 15 降至 10（延迟降 30%，recall 降 1.2%）; 2) 使用 gRPC 替代 HTTP 传输（延迟降 15%）；3) 对简单查询跳过 Rerank（基于 query 长度分类）。

本章结论

  • BM25 实现：Spring AI 无内置 BM25，通过 Lucene MemoryIndex（< 10 万文档）或 Elasticsearch（大规模）实现
  • RRF 融合：Rank-based 融合避免跨模型分数不可比问题，alpha=0.3 在 CorpAssist 法务场景最优
  • BGE-Reranker：交叉编码重排显著提升排序精度（recall@10 从 89.7% 升至 94.1%），但延迟从 68ms 增至 320ms
  • Ablation 结论：混合 RRF（alpha=0.3） + Rerank（Top15→5）是精度与速度的最优平衡
  • 瓶颈管理：Rerank 占检索延迟 60%，根据查询复杂度动态降级可进一步优化

面试可能问

RRF 的常数 k（通常 60）如果改大会有什么影响？

RRF 公式：score = sum(1/(k + rank))。k 越大，排名差异对分数的贡献越小（所有文档的 RRF score 趋近于相近值），融合结果接近等权平均。k 越小，第一名文档的 RRF 分数优势越大，Top-1 权重极高。k=60 是一个经验值：当排名从第 1 降到第 10 时，分数从 1/61≈0.0164 降到 1/70≈0.0143，降幅 13%，保持了排名差异但不过度极端。

为什么不在 Milvus 里用内置的 hybrid_search 而是自己实现 RRF？

Milvus 2.4+ 支持 hybrid_search 但需要同时配置向量索引和 scalar 倒排索引。限制：1) BM25 的全文检索能力有限（不支持中文分词自定义）；2) alpha 权重只能在集合级别配置，无法按查询动态调整；3) RRF 的常数固定不可调。自实现 RRF 更灵活：可按知识库调 alpha、可动态切换融合策略、可独立升级 BM25 组件。

BGE-Reranker 的交叉编码与向量相似度的双编码（bi-encoder）有什么本质区别？

双编码（bi-encoder）：query 和 doc 分别编码成独立向量，用余弦相似度计算相关分数。query 编码后固定，与所有 doc 向量做点积即可，计算量 O(N)。交叉编码（cross-encoder）：query 和 doc 拼接成一个序列送入 transformer 计算相关性分数。每个 (query, doc) 对需要一次完整前向推理，计算量 O(N*T^2)，精度更高但速度慢 10~100 倍。BGE-Reranker 属于交叉编码，所以需要对 Top-15 做 15 次推理，不能像 bi-encoder 那样批量点积。

场景 1：混合检索后 recall 反而低于纯向量检索，如何排查？

可能原因：1) BM25 索引中 tenant_id filter 与向量库 filter 不一致，检索到不同租户的数据；2) BM25 中文分词效果差（IKAnalyzer 词典未更新），专业名词被错误切分；3) alpha 设置不合理——如果向量召回已足够好（>85%），BM25 的低质量结果拉低了整体排名。验证方法：单独跑 BM25+RRF(alpha=0) 看 BM25 单路召回率。

场景 2：Rerank 阶段 P95 延迟达到 450ms，如何在不显著降低召回率的前提下优化？

策略：1) 减少 Rerank 候选数（Top15→Top10），延迟约降 33%，recall 降 < 2%；2) 对简单查询（长度 < 8 字）跳过 Rerank，仅对复杂查询执行；3) 使用 ASR（Adaptive Scaling）——当 rerank 排队超过 5 个时自动降级为不 rerank；4) 升级 GPU 实例或使用批量推理（batch_size=32 比串行推理快 8 倍）。
