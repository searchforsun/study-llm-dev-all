为什么学这章

GraphRAG 索引在 Python 侧完成（advanced-01），但 Java 侧 BFF 需要消费图索引结果对外提供服务。CorpAssist 的 Java 服务不能因为 Python 侧 GraphRAG 索引的存在而改变接口契约——必须通过同样的 /v1/rag/ask 端点处理基于图的查询。

学完后能做什么

在 Spring AI 中读取 GraphRAG 索引输出的 Parquet 文件建立本地查询索引，集成 GraphRAG HTTP API 实现 Local/Global Search，将图查询结果与向量检索结果合并排序，并能向面试官清晰解释何时应该上 GraphRAG。

本章要点速记

  • GraphRAG 索引由 Python 侧产出（.parquet），Java 侧通过本地缓存或 HTTP API 消费
  • Local Search 用于具体问题，Global Search 用于总结性问题
  • 图查询结果与向量检索结果通过 RRF 合并
  • GraphRAG 适用于跨文档推理场景（35% 的 CorpAssist 查询），Naive RAG 覆盖剩余 65%

GraphRAG 索引消费契约

CorpAssist · Java 消费 GraphRAGPython 侧运行完 GraphRAG 索引后产出 entities.parquet、relationships.parquet、communities.parquet。Java BFF 需要读取这些文件并提供查询 API。

GraphRAG 索引输出存储在共享文件系统或对象存储（S3/MinIO）中。Java 侧通过读取 Parquet 文件加载实体和社区数据，构建查询索引。Spring AI 不直接提供 GraphRAG 集成，需要自行实现 Parquet 读取和查询逻辑。

// 读取 GraphRAG Parquet 工件
@Component
public class GraphRagIndexReader {

    public GraphIndex loadIndex(String basePath) throws IOException {
        var entities = readParquet(basePath + "/entities.parquet");
        var relationships = readParquet(basePath + "/relationships.parquet");
        var communities = readParquet(basePath + "/communities.parquet");

        return new GraphIndex(entities, relationships, communities);
    }

    private List> readParquet(String path) throws IOException {
        var records = new ArrayList>();
        try (var reader = ParquetReader.builder(new GroupReadSupport(),
                new Path(path)).build()) {
            Group group;
            while ((group = reader.read()) != null) {
                var record = new HashMap();
                for (int i = 0; i < group.getType().getFieldCount(); i++) {
                    var fieldName = group.getType().getFieldName(i);
                    var value = group.getValueToString(i, 0);
                    record.put(fieldName, value);
                }
                records.add(record);
            }
        }
        return records;
    }
}

    Map attributes

    String id, String source, String target,
    String description, double weight

    String id, int level, String title,
    String summary, List entityIds

public class GraphIndex {
    private final List> entities;
    private final List> relationships;
    private final List> communities;

    public GraphIndex(List> entities,
                      List> relationships,
                      List> communities) {
        this.entities = entities;
        this.relationships = relationships;
        this.communities = communities;
    }

    // 根据实体名搜索关联社区
    public List searchCommunities(String entityName) {
        return communities.stream()
            .filter(c -> c.get("entityIds").toString().contains(entityName))
            .map(c -> c.get("summary").toString())
            .toList();
    }
}

GraphRAG 提供了 Local Search 和 Global Search 的 REST API。Java 侧通过 HTTP 客户端调用，注意请求体格式和对齐搜索参数。

java
// GraphRAG HTTP 客户端
@Service
public class GraphRagHttpClient {

    private final RestTemplate restTemplate;
    private final String graphRagEndpoint;

    public GraphRagHttpClient(
            @Value("${graphrag.endpoint:http://graphrag-python:8001}") String endpoint) {
        this.restTemplate = new RestTemplate();
        this.graphRagEndpoint = endpoint;
    }

    // Local Search — 面向具体问题
    public GraphRagResult localSearch(String query, String tenantId) {
        var request = Map.of(
            "query", query,
            "tenant_id", tenantId,
            "community_level", 2,
            "response_type", "multiple pages",
            "stream", false
        );
        var response = restTemplate.postForObject(
            graphRagEndpoint + "/v1/graphrag/local_search",
            request, GraphRagResponse.class);
        return convertToResult(response);
    }

    // Global Search — 面向全局总结
    public GraphRagResult globalSearch(String query, String tenantId) {
        var request = Map.of(
            "query", query,
            "tenant_id", tenantId,
            "community_level", 2,
            "response_type", "multiple pages",
            "stream", false
        );
        var response = restTemplate.postForObject(
            graphRagEndpoint + "/v1/graphrag/global_search",
            request, GraphRagResponse.class);
        return convertToResult(response);
    }

    private GraphRagResult convertToResult(GraphRagResponse resp) {
        if (resp == null) return new GraphRagResult("", List.of());

        var sources = resp.contextData().stream()
            .map(d -> new SourceRef(
                d.docId(), d.text().substring(0, Math.min(100, d.text().length())), 0.0))
            .toList();

        return new GraphRagResult(resp.response(), sources);
    }

    public record GraphRagResponse(
        String response,
        List contextData,
        Map metadata
    ) {}
    public record ContextData(String docId, String text, String sourceType) {}
    public record GraphRagResult(String answer, List sources) {}
}

结果合并与排序

Java BFF 同时维护两套检索路径：Naive RAG（向量检索）和 GraphRAG（图查询）。当路由器标记查询为多知识库关联问题时，并行执行两路检索并通过 RRF 合并。

java
// 统一检索服务 — 合并 Naive RAG + GraphRAG
@Service
public class UnifiedRetrievalService {

    @Autowired private VectorStore vectorStore;
    @Autowired private GraphRagHttpClient graphRagClient;
    @Autowired private IntentRouter intentRouter;

    public UnifiedResult retrieve(String query, String tenantId, String userRole) {
        var route = intentRouter.route(query);

        // 向量检索 (Naive RAG)
        var vecRequest = SearchRequest.builder()
            .query(query)
            .topK(6)
            .filterExpression("tenant_id == '" + tenantId
                + "' && allowed_roles CONTAINS '" + userRole + "'")
            .build();
        var vecDocs = vectorStore.similaritySearch(vecRequest);

        // 仅当跨文档查询时调用 GraphRAG
        GraphRagResult graphResult = null;
        if (route.confidence() > 0.7 || route.kb().equals("multi")) {
            graphResult = graphRagClient.localSearch(query, tenantId);
        }

        // 合并结果
        return mergeResults(vecDocs, graphResult, query);
    }

    private UnifiedResult mergeResults(
            List vecDocs, GraphRagResult graphResult, String query) {

        var allSources = new ArrayList();

        // 向量结果
        for (int i = 0; i < vecDocs.size(); i++) {
            var d = vecDocs.get(i);
            double rrfScore = 1.0 / (60 + i + 1);
            allSources.add(new SourceRef(
                (String) d.getMetadata().get("doc_id"),
                d.getContent().substring(0, Math.min(100, d.getContent().length())),
                rrfScore
            ));
        }

        // GraphRAG 结果
        if (graphResult != null && graphResult.sources() != null) {
            int offset = vecDocs.size();
            for (int i = 0; i < graphResult.sources().size(); i++) {
                var s = graphResult.sources().get(i);
                double rrfScore = 0.5 * (1.0 / (60 + i + 1));  // GraphRAG 权重减半
                allSources.add(new SourceRef(s.docId(), s.snippet(), rrfScore));
            }
        }

        // 按 RRF 分数排序，取 Top-6
        allSources.sort(Comparator.comparing(SourceRef::score).reversed());
        var topSources = allSources.stream()
            .limit(6)
            .peek(s -> s = new SourceRef(s.docId(), s.snippet(), Math.round(s.score() * 10000) / 10000.0))
            .toList();

        return new UnifiedResult(topSources, graphResult != null);
    }

    public record UnifiedResult(List sources, boolean graphRagUsed) {}
}

面试：何时上 GraphRAG

面试中高频问题：「为什么你们用了 Naive RAG 还要上 GraphRAG？」 和 「GraphRAG 的 ROI 如何衡量？」。以下是 CorpAssist 的决策框架：

不应该上 GraphRAG
  • 知识库 < 100 篇文档，文档间无交叉引用
  • 查询主要是事实性单文档问答（「年假几天」）
  • 团队无 LLM 编排经验，GraphRAG 索引调优困难
  • 索引成本（$12+/次）超过业务预算
应该上 GraphRAG
  • 跨文档推理查询占比 > 30%（日志分析确认）
  • 知识库 > 500 篇文档且文档间有密集引用
  • Naive RAG 的 context_recall < 0.75 且调参无效
  • 合规审计需要答案覆盖多源文档

索引阶段：500 篇文档 → $12.45 + 18 分钟（gpt-4o）。查询阶段：跨文档推理准确率从 52% 升至 85%（+33%），单文档问答从 87% 升至 91%（+4%）。索引成本增加 44 倍，查询成本增加 2.5 倍。ROI 体现在：用户满意度提升（CSAT +8%）、人工转接率从 22% 降至 15%（跨文档 query 减少转人工）。

自检：如果要向你的 tech lead 证明需要上 GraphRAG，你会提供哪 3 个数据点？

1) 用户日志分析：跨文档查询占比（示例：35% 的 query 隐含引用 2+ 文档）；2) 当前评测：Naive RAG 跨文档场景准确率仅 52%，用户投诉中 40% 来自跨文档问题未满意解决；3) 小规模验证：10 篇关联文档的 GraphRAG PoC 将跨文档准确率提升至 80%+。提供这三点足以让 tech lead 支持 PoC 资源投入。

本章结论

  • 索引消费：Java 侧通过读取 Parquet 工件或调用 HTTP API 消费 Python GraphRAG 索引结果
  • HTTP 集成：Local Search 用于具体问题，Global Search 用于全局总结，通过 RestTemplate/WebClient 调用
  • 结果合并：Naive RAG + GraphRAG 通过 RRF 融合，GraphRAG 权重重减半（0.5x）避免噪声主导
  • 决策框架：跨文档 query > 30% + Naive RAG recall < 0.75 + 知识库 > 500 篇 → 值得上 GraphRAG
  • ROI 数据：跨文档准确率 +33%，CSAT +8%，转人工率 -7%，索引成本 +$12.45/次

面试可能问

Java 侧为什么不直接运行 GraphRAG 索引而要依赖 Python 侧产出？

原因：1) 微软 GraphRAG 的实体抽取使用 LLM 逐 chunk 处理，Python 的 asyncio + LangChain 生态更适合此模式（Java 虚拟线程虽好但生态不成熟）；2) GraphRAG 社区检测依赖 Python 科学计算栈（NetworkX、Leidenalg），Java 无等效库；3) 索引是非实时离线任务，Python 脚本更灵活。Java 侧专注于在线查询（低延迟、高并发），索引交给 Python 批处理。

List 出至少 3 个 GraphRAG 查询不如直接向量检索的场景。

1) 事实性单文档问答（「我们公司年假最低几天」）——向量检索 45ms 完成，GraphRAG 需要 2s+ 走整个图查询流程，准确率差异 < 5%；2) 实时更新场景（「今天下午的会议取消了吗」）——GraphRAG 索引更新需重新运行整个管线（18 分钟），向量库可以实时 upsert；3) 低成本场景（免费 tier 用户）——GraphRAG 每次查询 $0.005 vs 向量 $0.002，索引成本更相差 44 倍。

如果 GraphRAG 服务不可用，你的降级策略是什么？

CorpAssist 的降级策略分三级：1) 熔断：检测到 GraphRAG API 连续 3 次超时（>5s），启动熔断器（Resilience4j CircuitBreaker），后续查询仅走 Naive RAG；2) 半熔断：5 分钟后尝试半开，发 1 条探测请求，成功后逐步恢复 10% 流量；3) 降级通知：在响应 metadata 中添加 {"degraded": true, "reason": "graphrag_unavailable"}，前端可显示「部分检索源不可用」提示。熔断期间 Naive RAG 召回率从 93% 降至 76%（跨文档场景），但核心功能可用。

场景 1：GraphRAG HTTP 调用返回空结果，但索引 Parquet 中有数据，排查步骤？

1) 确认 Python GraphRAG 服务是否已加载索引（检查服务启动日志）；2) 检查请求的 tenant_id 是否匹配索引数据路径；3) community_level 参数是否合理（level=0 粒度过细可能无社区，level=3 摘要过粗；推荐 level=2）；4) 确认 Python 侧 GraphRAG API 的 query 路由是否正确（local vs global，选错会返回空）。

场景 2：合并结果中 GraphRAG 的 source 排名远低于向量 source，即使 GraphRAG 内容更相关。如何优化？

当前 GraphRAG 的 RRF 权重为 0.5 倍（减半），这可能是原因。建议：1) 对跨文档查询场景，将 GraphRAG 权重从 0.5 提升至 0.8；2) 使用 BGE-Reranker 对合并结果重排而非 RRF 融合；3) 如果向量检索噪声大，降低 TopK 或在 RRF 中对向量侧加 penalty。调优方法类似 practice-03 的 ablation 实验。
