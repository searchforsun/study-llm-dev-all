学完后能做什么

配置 DocumentReader 解析 PDF/Word/TXT 文档，选型 TokenTextSplitter 参数并配合同一 VectorStore，串联 QuestionAnswerAdvisor 实现检索增强生成，输出与 Python LCEL 链一致的 answer+sources JSON 响应。

本章要点速记

  • Spring AI Pipeline = DocumentReader → DocumentTransformer(Splitter) → VectorStore → QuestionAnswerAdvisor
  • TokenTextSplitter 默认 chunkSize=800, chunkOverlap=200，需按文档类型调优
  • QuestionAnswerAdvisor 自动拼接 context 到 Prompt，等效 Python LCEL 的 RunnableParallel
  • 响应通过 AbstractMemoryAdvisor 实现引用溯源，sources 格式与 Python 侧对齐

DocumentReader 配置

CorpAssist · PDF 政策文档入库HR 部每月发布《员工手册》PDF 版，含表格和多级标题。DocumentReader 需正确解析章节结构和表格内容，乱码或丢表将导致检索结果残缺。

Spring AI 提供多种 DocumentReader 实现：PagePdfDocumentReader（PDF）、TextDocumentReader（TXT/CSV）、JsonDocumentReader（JSON）、WordDocumentReader（DOCX）。每个 Reader 将原始文件转为 Document 对象列表，含 content 和 metadata。

java
// DocumentReader 配置 — PDF 解析
@Configuration
public class DocumentReaderConfig {

    @Bean
    public PdfDocumentReader pdfReader() {
        var resource = new FileSystemResource("classpath:docs/hr_policy_2024.pdf");
        var config = PdfDocumentReaderConfig.builder()
            .withPageExtractedTextFormatter(page -> {
                // 保留章节标题格式，去掉页眉页脚
                return page.getText()
                    .replaceAll("CorpAssist Confidential", "")
                    .replaceAll("第 \\d+ 页", "");
            })
            .withPagesPerDocument(1)  // 每页作为一个 Document
            .build();
        return new PagePdfDocumentReader(resource, config);
    }

    @Bean
    public WordDocumentReader wordReader() {
        var resource = new FileSystemResource("classpath:docs/轮岗管理办法.docx");
        return new WordDocumentReader(resource);
    }

    @Bean
    public JsonDocumentReader jsonReader() {
        var resource = new FileSystemResource("classpath:docs/faq_hr.json");
        return new JsonDocumentReader(resource, "question", "answer");
    }
}
Reader 类型适用场景输出 Document 粒度注意事项
PagePdfDocumentReader政策 PDF、合同扫描每页一个 Document表格可能丢失行列结构，建议转 Markdown 再解析
WordDocumentReaderWord 格式的 SOP、制度按段落拆分嵌入图片丢失，仅提取文字
JsonDocumentReaderFAQ 库、结构化数据每个 JSON 对象一个 Document需指定 content 字段名
TextDocumentReader纯文本日志、存档整个文件文件过大会超出 token 限制，需配合 splitter

Splitter 选型与参数

DocumentReader 输出的 Document 粒度通常不适合直接存入 VectorStore（PDF 每页可能 500~2000 token，超出 embedding 模型上下文窗口或降低检索精度）。Spring AI 提供 DocumentTransformer 接口用于拆分。

java
// TokenTextSplitter 配置与参数调优
@Configuration
public class SplitterConfig {

    @Bean
    public TokenTextSplitter tokenSplitter() {
        return TokenTextSplitter.builder()
            .withChunkSize(800)          // 每个 chunk 的 token 数上限
            .withChunkOverlap(200)       // 相邻 chunk 重叠 token 数
            .withTokensPerWord(1.3f)     // 中文占 token 比例修正
            .build();
    }

    // 按文档类型选择 splitter
    public DocumentTransformer selectSplitter(String docType) {
        return switch (docType) {
            case "policy" -> TokenTextSplitter.builder()
                .withChunkSize(600).withChunkOverlap(150).build();  // 政策条款宜小
            case "faq" -> TokenTextSplitter.builder()
                .withChunkSize(300).withChunkOverlap(50).build();   // FAQ 问答对独立
            case "manual" -> TokenTextSplitter.builder()
                .withChunkSize(1200).withChunkOverlap(300).build(); // 操作手册宜保持上下文
            default -> tokenSplitter();
        };
    }
}
文档类型chunkSizechunkOverlap理由CorpAssist 效果
政策条款（Policy）600150条款短且独立，交叉引用少recall@4=78%
FAQ 问答30050问答对通常<200 token，需保持完整性recall@4=91%
操作手册（Manual）1200300操作步骤需连续上下文faithfulness=0.88
默认800200通用平衡综合最优
java
// 完整索引流程：Read → Split → Store
@Service
public class IngestionService {

    private final VectorStore vectorStore;
    private final TokenTextSplitter splitter;

    public void ingest(String filePath, String kb, String tenantId) {
        // 1. 读取
        var resource = new FileSystemResource(filePath);
        var reader = new PagePdfDocumentReader(resource);
        List docs = reader.get();

        // 2. 拆分
        List chunks = splitter.apply(docs);

        // 3. 注入元数据
        chunks.forEach(doc -> {
            doc.getMetadata().put("tenant_id", tenantId);
            doc.getMetadata().put("kb", kb);
            doc.getMetadata().put("doc_id", UUID.randomUUID().toString());
            doc.getMetadata().put("ingested_at", Instant.now().toString());
        });

        // 4. 写入 VectorStore
        vectorStore.add(chunks);
        System.out.printf("Ingested %d chunks from %s%n", chunks.size(), filePath);
    }
}

QuestionAnswerAdvisor 链路

Spring AI 的 QuestionAnswerAdvisor 等价于 Python LCEL 中的 RunnableParallel(context=retriever, question=passthrough) | prompt | llm。它接收用户问题，自动调用 VectorStore 检索相关文档，将 context 拼入 Prompt，最后用 LLM 生成回答。

// Spring AI RAG Pipeline 完整配置
@Configuration
public class RagPipelineConfig {

    @Bean
    public ChatClient ragChatClient(
            ChatModel chatModel,
            VectorStore vectorStore,
            TokenTextSplitter splitter) {

        var advisor = new QuestionAnswerAdvisor(
            vectorStore,
            SearchRequest.builder()
                .query("")
                .topK(4)
                .similarityThreshold(0.45)
                .build()
        );

        return ChatClient.builder(chatModel)
            .defaultSystem("""
                你是 CorpAssist AI 助手。严格根据提供的 context 回答问题。
                若 context 中找不到依据，请回答"无法从现有知识库中找到答案"。
                在回答末尾列出引用来源 doc_id。
                """)
            .defaultAdvisors(advisor)
            .build();
    }
}

// Controller — 与 Python LCEL 对齐的 API
@RestController
@RequestMapping("/v1/rag")
public class RagController {

    private final ChatClient chatClient;

    @PostMapping("/ask")
    public AskResponse ask(@RequestBody AskRequest request) {
        var response = chatClient.prompt()
            .user(request.question())
            .advisors(a -> a.param("tenant_id", request.tenantId()))
            .call()
            .content();

        // 从 advisor context 提取 sources
        List sources = extractSources(response);
        return new AskResponse(response, sources);
    }

    @PostMapping("/ask/stream")
    public Flux askStream(@RequestBody AskRequest request) {
        return chatClient.prompt()
            .user(request.question())
            .advisors(a -> a.param("tenant_id", request.tenantId()))
            .stream()
            .content();
    }
}

// DTO — 与 Python 侧一致的 JSON 契约
public record AskRequest(String question, String tenantId, String kb) {}
public record SourceRef(String docId, String snippet, double score) {}
public record AskResponse(String answer, List sources) {}

完整的 RAG Pipeline 需要将 索引 和 查询 两个阶段串联。引用溯源的实现方案是：在 QuestionAnswerAdvisor 的回调中捕获检索到的 Document 列表，将其转为 SourceRef 附加到响应中。

java
// 自定义 Advisor — 带引用溯源的 QuestionAnswerAdvisor
@Component
public class TraceableQuestionAnswerAdvisor implements CallAroundAdvisor {

    private final VectorStore vectorStore;
    private final int topK;

    @Override
    public AdvisorResponse aroundCall(AdvisorChain chain, UserMessage userMsg,
                                      Map context) {
        // 1. 检索阶段
        String query = userMsg.getText();
        var searchRequest = SearchRequest.builder()
            .query(query)
            .topK(topK)
            .similarityThreshold(0.45)
            .filterExpression("tenant_id == '" + context.get("tenant_id") + "'")
            .build();

        List docs = vectorStore.similaritySearch(searchRequest);

        // 2. 将检索文档注入 user message 的 context
        String contextStr = docs.stream()
            .map(d -> "[doc_id=" + d.getMetadata().get("doc_id") + "] " + d.getContent())
            .collect(Collectors.joining("\n---\n"));
        String enhancedPrompt = """
            请根据以下文档内容回答问题。若文档中无答案请说不知道。

            参考文档：
            %s

            问题：%s
            """.formatted(contextStr, query);

        // 3. 继续链式调用（后续可能还有别的 advisor）
        AdvisorResponse response = chain.nextAroundCall(
            new UserMessage(enhancedPrompt), context);

        // 4. 将 sources 写入 context 供 controller 提取
        List sources = docs.stream()
            .map(d -> new SourceRef(
                (String) d.getMetadata().get("doc_id"),
                d.getContent().substring(0, Math.min(100, d.getContent().length())),
                (double) d.getMetadata().getOrDefault("distance", 0.0)
            ))
            .toList();
        context.put("rag_sources", sources);

        return response;
    }
}

常见陷阱
  • 未在检索时加 tenant_id filter，跨租户数据泄漏
  • Splitter 参数未按文档类型调优，chunk 过大或过小
  • 未实现自定义 Advisor，sources 引用无法透传到 controller
  • 默认 SystemPrompt 未约束「仅根据 context」，模型可能虚构
推荐实践
  • 检索 filter 表达式硬编码 tenant_id
  • 文档类型路由到专属 splitter 配置
  • 实现 TraceableQuestionAnswerAdvisor 捕获 sources
  • system prompt 使用否定约束（不可、禁止）
自检：Spring AI 的 QuestionAnswerAdvisor 与 Python LCEL 的 RunnableParallel 有何异同？

相同：两者都实现了「检索 → 拼 context → 注入 Prompt → 调 LLM」的 RAG 核心流程。不同：QuestionAnswerAdvisor 是 ChatClient 的 advisor（类似拦截器/中间件），可组合多个 advisor；RunnableParallel 是链式管道的组成节点，通过 | 连接。Spring AI advisor 模式更接近 AOP，LCEL 管道模式更接近函数组合。

本章结论

  • DocumentReader：Spring AI 支持 PDF/Word/JSON/TXT 四种 Reader，PDF 每页产出一个 Document，Word 按段落拆分
  • Splitter 选型：TokenTextSplitter 默认 800/200，Policy 文档宜用 600/150，FAQ 宜用 300/50
  • QuestionAnswerAdvisor：自动完成检索+拼装增强，等价 Python LCEL 的 RunnableParallel+Prompt+LLM
  • 引用溯源：通过自定义 Advisor 捕获检索文档列表，转为 SourceRef 写入 context 供 API 响应
  • 契约对齐：Spring AI 端 AskResponse(answer, sources[]) 与 Python LCEL 端 JSON 字段名一致

面试可能问

Spring AI 的 Advisor 链和 Python LangChain 的 Chain 模式在设计上有什么不同？

Spring AI Advisor 模式基于 AOP 思想——每个 Advisor 包装（wrap）在被调用方法周围，形成一个洋葱模型。便于横切关注点如日志、鉴权、引用提取。LangChain Chain 模式基于组合模式——通过 | 运算符将多个 Runnable 连接成有向图。两者模式不同但都是装饰器变体。Spring AI 更擅长在 Java 生态中与其他 Spring AOP 组件集成。

如果 PDF 中有表格，PagePdfDocumentReader 解析后表格会变成什么？如何处理？

PagePdfDocumentReader 基于 Apache PDFBox，默认将表格渲染为纯文本（单元格内容按坐标拼成行，丢失表格结构）。处理方案有两层：1) 在 PDF 解析前先用 OCR+表格识别工具（如 Camelot/Tabula）将表格转为 Markdown 再喂入 Reader；2) 在 withPageExtractedTextFormatter 中自定义行列拼接逻辑。CorpAssist 对含表格的 PDF 先使用 Python Pandas 转为 CSV 再入库。

如何测试 Spring AI Pipeline 的每个环节是否正常工作？

分段测试：1) DocumentReader — 单测验证 Document 数量和 metadata；2) Splitter — 验证 chunk token 数在 chunkSize 范围内；3) VectorStore — 写入后 similaritySearch 验证返回预期文档；4) Advisor — Mock VectorStore 验证 Prompt 拼装是否正确；5) Controller — MockMvc 验证 JSON 响应结构。Spring AI 提供 TestAdvisors 和 MockVectorStore 辅助测试。

场景 1：分块后的 Document 数量远少于预期（500 页 PDF 只产出 30 个 chunk），应排查什么？

可能原因：1) chunkSize 设置过大（如 4000），每个 Document（一页）显著小于一个 chunk 大小，splitter 无需拆分；2) PDF 的文本内容被渲染为图片，PDFBox 提取不出文字，reader 读取为空。验证方法：检查 reader.get().size() 的原始 Document 数，以及每个 Document 的内容长度。

场景 2：Spring AI API 返回了 answer 但 sources 为空数组，引用提取环节出了什么问题？

可能性：1) 自定义 Advisor 未正确捕获 VectorStore 的检索结果——检查 aroundCall 中 vectorStore.similaritySearch 是否被调用；2) 检索结果未写入 context map——context.put("rag_sources", sources) 必须在 response 返回前执行；3) Controller 中未从 context 读取 rag_sources——确认 Advisor 和 Controller 使用同一个 context 实例。
