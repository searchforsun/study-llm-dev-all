为什么学这章

Java 侧 BFF 是 CorpAssist 的生产入口，每次修改都必须通过回归测试。但 Spring AI 的 Pipeline 含 LLM 调用（非确定性输出），传统 JUnit 断言失效。需要构建 Advisor 级别的集成测试：验证检索阶段而非生成阶段，与 Python RAGAS 共用评测集并对照结果。

学完后能做什么

编写 Spring AI Pipeline 的集成测试（Mock LLM 验证检索、Mock VectorStore 验证 Advisor 行为），运行与 Python RAGAS 同一评测集的对照测试，配置 CI 闸门确保 faithfulness/recall 不退化。

本章要点速记

  • Spring AI 集成测试需 Mock ChatModel（固定输出验证检索）和 Mock VectorStore（可控文档验证 Advisor）
  • 双栈对照：Java 侧的评测数据（answer+sources）输出到 JSON，由 Python RAGAS 脚本统一计算指标
  • CI 闸门：faithfulness >= 0.85, context_recall >= 0.80，与 Python 侧同一阈值
  • 测试覆盖：DocumentReader、Splitter、Advisor 链、Controller JSON 契约四个层次

集成测试设计

CorpAssist · Pipeline 回归测试每次修改 TokenTextSplitter 参数或 Advisor 逻辑时，测试需要验证：1) 检索到的文档是否正确；2) Advisor 是否能正确拼装 context；3) 最终响应结构和 Python 侧一致。

Spring AI Pipeline 集成测试分四层：Reader 测试 → Splitter 测试 → Advisor 测试 → Controller 测试。核心原则：LLM 调用必须 Mock（固定输出），检索结果必须可控（Mock VectorStore），只验证管线编排逻辑而非 AI 输出质量。

java
// 四层集成测试示例
@SpringBootTest
class RagPipelineIntegrationTest {

    // === 第一层：Reader 测试 ===
    @Test
    void testPdfReader_shouldExtractCorrectPageCount() {
        var resource = new ClassPathResource("docs/hr_policy_sample.pdf");
        var reader = new PagePdfDocumentReader(resource);
        List docs = reader.get();
        assertThat(docs).hasSizeGreaterThanOrEqualTo(4);  // 样本 PDF 至少 4 页
        assertThat(docs.get(0).getContent()).contains("年假");
    }

    // === 第二层：Splitter 测试 ===
    @Test
    void testTokenSplitter_chunkSizeShouldNotExceedLimit() {
        var splitter = TokenTextSplitter.builder()
            .withChunkSize(800).withChunkOverlap(200).build();
        var doc = new Document("年假 ".repeat(2000));
        List chunks = splitter.apply(List.of(doc));
        for (var chunk : chunks) {
            assertThat(chunk.getContent().length())
                .isLessThanOrEqualTo(600);  // 800 token ≈ 600 中文字
        }
    }

    // === 第三层：Advisor 测试（Mock VectorStore） ===
    @Test
    void testQuestionAnswerAdvisor_shouldAppendContextToPrompt() {
        var mockStore = mock(VectorStore.class);
        var mockDocs = List.of(
            new Document("年假最低5天，工龄满10年10天，满20年15天",
                Map.of("doc_id", "hr-001", "location", "第4章"))
        );
        when(mockStore.similaritySearch(any())).thenReturn(mockDocs);

        var advisor = new QuestionAnswerAdvisor(mockStore,
            SearchRequest.builder().query("").topK(4).build());

        var mockModel = mock(ChatModel.class);
        when(mockModel.call(any())).thenReturn(new ChatResponse(List.of(
            new AssistantMessage("根据文档，年假最低5天")
        )));

        var client = ChatClient.builder(mockModel)
            .defaultAdvisors(advisor)
            .build();

        String response = client.prompt().user("年假几天").call().content();
        assertThat(response).contains("年假最低5天");
        verify(mockStore).similaritySearch(argThat(req ->
            req.getTopK() == 4));  // 验证检索参数
    }

    // === 第四层：Controller 测试（JSON 契约） ===
    @Test
    void testAskEndpoint_shouldReturnExpectedJsonStructure() throws Exception {
        var request = """
            {"question":"年假几天","tenantId":"corp-a","kb":"hr"}
            """;
        mockMvc.perform(post("/v1/rag/ask")
                .contentType(MediaType.APPLICATION_JSON)
                .content(request))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.answer").isString())
            .andExpect(jsonPath("$.sources").isArray())
            .andExpect(jsonPath("$.sources[0].docId").isString());
    }
}

Advisor 断言

Advisor 是 Spring AI Pipeline 的核心编排组件，测试中需断言的关键行为包括：检索调用是否执行、context 是否拼入 Prompt、sources 是否正确提取。

java
// Advisor 行为断言
@SpringBootTest
@AutoConfigureMockMvc
class AdvisorBehaviorTest {

    @Test
    void testCustomAdvisor_shouldCaptureSourcesInContext() {
        var mockStore = mock(VectorStore.class);
        var testDocs = List.of(
            new Document("PIP流程：绩效改善计划周期30-90天",
                Map.of("doc_id", "hr-pip-001", "location", "绩效管理第3章")),
            new Document("PIP期间暂停年假累计",
                Map.of("doc_id", "hr-pip-002", "location", "绩效管理第5条"))
        );
        when(mockStore.similaritySearch(any())).thenReturn(testDocs);

        // 自定义 Advisor 将 sources 写入 context
        var advisor = new TraceableQuestionAnswerAdvisor(mockStore, 4);
        var mockModel = mock(ChatModel.class);
        when(mockModel.call(any())).thenAnswer(invocation -> {
            var request = invocation.getArgument(0, ChatRequest.class);
            var promptText = request.getMessages().get(1).getText();
            assertThat(promptText).contains("hr-pip-001");  // 验证 doc_id 在 prompt 中
            assertThat(promptText).contains("hr-pip-002");
            return new ChatResponse(List.of(
                new AssistantMessage("PIP周期为30-90天")
            ));
        });

        var context = new HashMap();
        context.put("tenant_id", "corp-a");

        var client = ChatClient.builder(mockModel)
            .defaultAdvisors(advisor)
            .build();

        client.prompt()
            .user("PIP流程多久")
            .advisors(a -> a.param("tenant_id", "corp-a"))
            .call()
            .context(context);

        @SuppressWarnings("unchecked")
        var sources = (List) context.get("rag_sources");
        assertThat(sources).hasSize(2);
        assertThat(sources.get(0).docId()).isEqualTo("hr-pip-001");
    }

    // 验证 filter 表达式是否传递
    @Test
    void testAdvisor_shouldPassFilterExpression() {
        var mockStore = mock(VectorStore.class);
        when(mockStore.similaritySearch(any())).thenReturn(List.of());

        var advisor = new TraceableQuestionAnswerAdvisor(mockStore, 4);

        var mockModel = mock(ChatModel.class);
        when(mockModel.call(any())).thenReturn(
            new ChatResponse(List.of(new AssistantMessage("ok"))));

        var client = ChatClient.builder(mockModel)
            .defaultAdvisors(advisor)
            .build();

        client.prompt()
            .user("年假几天")
            .advisors(a -> a.param("tenant_id", "corp-a"))
            .call();

        verify(mockStore).similaritySearch(argThat(req ->
            req.getFilterExpression() != null &&
            req.getFilterExpression().contains("corp-a")));
    }
}

与 Python RAGAS 结果对照

Java 侧 BFF 使用与 Python 侧相同的评测集（corpassist_eval_v2.jsonl），但 Java 侧不直接调用 RAGAS。Java 将评测结果（question, answer, contexts, ground_truth）输出为 JSON，由 Python RAGAS 脚本统一计算指标。

// Java 侧评测输出 — 供 RAGAS 消费
@Component
public class EvalRunner {

    @Autowired private ChatClient ragChatClient;

    public void runEval(String evalPath, String outputPath) throws IOException {
        var mapper = new ObjectMapper();
        var evalSet = mapper.readValue(
            new File(evalPath), EvalItem[].class);

        var results = new ArrayList>();
        for (var item : evalSet) {
            var response = ragChatClient.prompt()
                .user(item.question())
                .call();
            var context = response.getContext();
            @SuppressWarnings("unchecked")
            var sources = (List) context.get("rag_sources");

            var record = Map.of(
                "question", item.question(),
                "answer", response.getResult().getOutput().getText(),
                "contexts", sources.stream().map(SourceRef::snippet).toList(),
                "ground_truth", item.groundTruth()
            );
            results.add(record);
        }

        mapper.writerWithDefaultPrettyPrinter()
            .writeValue(new File(outputPath), results);
        System.out.printf("Eval output written to %s (%d queries)%n",
            outputPath, results.size());
    }
}

CorpAssist Java 侧的 CI 闸门与 Python 侧同一标准：faithfulness >= 0.85, context_recall >= 0.80。每次 PR 合并前触发。

yaml
# .github/workflows/rag-eval-java.yml
name: Java RAG Eval Gate
on:
  pull_request:
    paths:
      - 'rag-service-java/**'

jobs:
  eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '21'
          cache: 'maven'

      - name: Run Java eval output
        run: |
          cd rag-service-java
          mvn test -Dtest=EvalRunnerIntegrationTest
          cp target/eval_output.json ../eval/

      - name: Run Python RAGAS on Java output
        run: |
          cd eval
          pip install ragas datasets
          python compute_ragas.py \
            --input ../rag-service-java/target/eval_output.json \
            --threshold '{"faithfulness":0.85,"context_recall":0.80}' \
            --output ragas_java_report.json

      - name: Check gate
        run: |
          python -c "
          import json
          r = json.load(open('eval/ragas_java_report.json'))
          assert r['faithfulness'] >= 0.85, f'faithfulness {r[\"faithfulness\"]} < 0.85'
          assert r['context_recall'] >= 0.80, f'recall {r[\"context_recall\"]} < 0.80'
          print(f'Gate passed: faith={r[\"faithfulness\"]:.3f} recall={r[\"context_recall\"]:.3f}')
          "
自检：如果 Java 侧 faithfulness 0.82 低于闸门，而 Python 侧 0.89 通过，说明什么？

说明两栈的生成环节存在差异，最可能的原因是 Prompt 模板不同。检查 Java 侧 SystemPrompt 是否写了比 Python 侧更强的约束条件（或更弱）。对比两栈的 Prompt 模板变量名、system message 措辞、context 拼装格式。CorpAssist 规范：双栈必须使用同一份 Prompt 模板（通过共享配置文件管理）。

本章结论

  • 四层测试：Reader 测试文档解析 → Splitter 测试分块正确性 → Advisor 测试检索与 Prompt 拼装 → Controller 测试 JSON 契约
  • Mock 策略：Mock LLM（固定输出验证检索），Mock VectorStore（可控文档验证 Advisor），只测编排不测 AI
  • 双栈对照：Java 输出评测结果 JSON，Python RAGAS 统一计算指标，确保 delta < 0.02
  • CI 闸门：faithfulness >= 0.85 且 context_recall >= 0.80，与 Python 侧同一阈值
  • Prompt 对齐：双栈 faithfulness 偏差 < 0.01 需使用共享 Prompt 模板

面试可能问

Mock ChatModel 是否可能漏测 Prompt 质量问题？

Mock ChatModel 只验证管线是否按预期传参（context 是否拼入、filter 是否正确），不验证 Prompt 质量（措辞、约束）。Prompt 质量需要通过 RAGAS 评测（faithfulness/relevancy）来验证。测试策略分层：单元测试（Mock LLM）验证编排逻辑，集成测试（真实 LLM 但小样本）验证 Prompt 效果，全量评测（CI 闸门）验证回归。

Java 和 Python 的评测结果应该完全一致吗？

不期望完全一致，两栈的 SystemPrompt 略有不同（Java 的约束更严格以满足合规审计），LLM 调用参数也可能不同（temperature）。但检索阶段指标（context_precision/recall）应几乎一致（delta < 0.02），生成阶段指标（faithfulness）应在同一量级（delta < 0.03）。若偏差 > 0.05 需查两栈的差异。

CI 闸门的阈值应该怎么设定？

方法：先跑 10 次历史 PR 的评测，取各指标的均值和标准差。设置阈值为 mean - 1.5*std（安全阀）或 mean - 2*std（宽松阀）。CorpAssist 使用上述方法且动态更新：每季度重新计算阈值，确保评测集扩展后阈值仍然合理。初始值：faithfulness>=0.85，context_recall>=0.80，这是经 200+ PR 验证的合理底线。

场景 1：集成测试中 Mock VectorStore 返回空列表，但生产环境能检索到文档，说明什么？

这是预期的隔离测试行为。Mock 返回空列表是为了测试「无检索结果」场景下 Advisor 的行为（是否返回「无法找到答案」）。如果测试想验证检索集成，应该编写真正的集成测试（连接测试用的 Milvus/SimpleVectorStore），与 Mock 测试分开。CorpAssist 同时维护 Mock 测试（CI 快速）和真实集成测试（每日一次）。

场景 2：Java 侧评测结果 faithfulness=0.82 比上次 0.87 显著下降，但代码未改。怎么排查？

可能的非代码原因：1) OpenAI API 模型侧更新（gpt-4o-mini 悄悄升级了版本）；2) 评测数据集被修改（ground_truth 更新但未通知）；3) 向量库中索引数据被更新（旧文档下架、新文档 embedding 模型不同）。排查方法：回滚到上次通过的 commit 重新运行评测，若仍下降则说明外部环境变化；对比评测集版本号和向量库文档数。
