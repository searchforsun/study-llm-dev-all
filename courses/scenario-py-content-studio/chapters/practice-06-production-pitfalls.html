库存/政策API真查 — 防止过度承诺

LLM生成内容时天然倾向于「答应客户一切」，包括承诺有货、承诺折扣、承诺服务。必须在生成内容中插入真实的API查询结果。

API真查数据流
sequenceDiagram
    participant LLM as LLM生成
    participant Guard as API真查守卫
    participant API as 库存/政策API

    LLM->>Guard: 生成内容含承诺声明
    Guard->>API: 查询库存/价格/政策
    API-->>Guard: 返回真实数据
    Guard->>Guard: 替换生成内容中的承诺
    Guard-->>LLM: 返回修正后内容

API真查守卫在LLM生成后，用真实API数据替换所有承诺性声明。

Python：API真查守卫

python
# api_guard.py — API真查守卫
import re
from typing import Dict, Optional
import httpx

class APIGuard:
    def __init__(self, inventory_api: str, policy_api: str):
        self.inventory_api = inventory_api
        self.policy_api = policy_api

    def scan_promises(self, text: str) -> list:
        """扫描内容中的承诺性表述"""
        patterns = [
            (r"(?P.{2,20})库存充足|有货|现货", "inventory"),
            (r"价格.*?(?P\d+(\.\d+)?)元", "price"),
            (r"(?P\d+)天.*?退货|退款|保修", "policy"),
            (r"(?P\d+)折|打.*?折|优惠.*?\d+%", "discount"),
        ]
        promises = []
        for pat, ptype in patterns:
            for match in re.finditer(pat, text):
                promises.append({"type": ptype, "text": match.group(), "span": match.span()})
        return promises

    async def verify(self, text: str, sku: str) -> str:
        """真查并替换承诺"""
        # 查询真实库存
        async with httpx.AsyncClient() as client:
            inv_resp = await client.get(f"{self.inventory_api}/{sku}")
            inv_data = inv_resp.json()

        # 查询真实政策
        policy_resp = await client.get(f"{self.policy_api}/{sku}")
        policy_data = policy_resp.json()

        # 替换承诺
        text = re.sub(
            r"库存充足|有货|现货",
            f"当前库存{inv_data['stock']}件" if inv_data['stock'] > 0 else "需确认库存",
            text
        )
        return text

解码约束与对抗样本防御

对抗样本通过特殊编码（Unicode变体、Base64、反转文本）绕过内容安全检测。

# adversarial_defense.py — 对抗样本防御
import re, unicodedata, base64

class AdversarialDefense:
    def normalize(self, text: str) -> str:
        """Unicode归一化"""
        text = unicodedata.normalize("NFKC", text)
        # 去零宽字符
        text = re.sub(r"[​‌‍⁠﻿]", "", text)
        # 去多余空格
        text = re.sub(r"\s+", "", text)
        return text

    def check_base64(self, text: str) -> list:
        """检测Base64编码的敏感内容"""
        found = []
        b64_pattern = r"[A-Za-z0-9+/]{20,}={0,2}"
        for match in re.finditer(b64_pattern, text):
            try:
                decoded = base64.b64decode(match.group()).decode("utf-8", errors="ignore")
                if self._is_sensitive(decoded):
                    found.append({"type": "base64", "decoded": decoded[:50]})
            except Exception:
                pass
        return found

    def check_reversed(self, text: str) -> list:
        words = text.split()
        for w in words:
            if self._is_sensitive(w[::-1]):
                return [{"type": "reversed", "original": w[::-1]}]
        return []

    def _is_sensitive(self, text: str) -> bool:
        sensitive = ["暴力", "赌博", "色情", "毒品", "枪支"]
        return any(s in text for s in sensitive)

    def full_check(self, text: str) -> dict:
        normalized = self.normalize(text)
        return {
            "base64": self.check_base64(text),
            "reversed": self.check_reversed(text),
            "normalized_sensitive": self._is_sensitive(normalized),
        }

私域记忆分层与角色OOC

私域记忆分三层：System Prompt（固定角色）、用户画像（中期）、短期记忆（对话轮次）。分层目的是防止短期记忆污染角色定义。角色OOC（Out Of Character）检测确保LLM不超出预设角色行为边界。

Python：记忆分层与OOC检测

python
# memory_tier.py — 记忆分层与OOC检测
from typing import List, Dict

class MemoryTier:
    LAYER_0 = "system"     # 固定，不可覆盖
    LAYER_1 = "profile"    # 用户画像，持久化
    LAYER_2 = "session"    # 短期记忆，滑动窗口

class TieredMemory:
    def __init__(self):
        self.system_prompt = ""
        self.profile = {}
        self.session: List[Dict] = []

    def set_system(self, prompt: str):
        self.system_prompt = prompt

    def update_profile(self, key: str, value: str):
        self.profile[key] = value

    def add_session(self, turn: Dict):
        self.session.append(turn)
        if len(self.session) > 10:
            self.session.pop(0)

    def build_context(self) -> str:
        return f"{self.system_prompt}\n{self.profile}\n{self.session}"

class OOCDetector:
    def __init__(self, role_boundary: dict):
        self.role = role_boundary

    def check(self, response: str) -> dict:
        violations = []
        if self.role.get("forbidden_actions"):
            for action in self.role["forbidden_actions"]:
                if action in response:
                    violations.append({"type": "forbidden_action", "match": action})
        if self.role.get("allowed_topics"):
            if not any(t in response for t in self.role["allowed_topics"]):
                violations.append({"type": "off_topic", "detail": "超出允许话题范围"})
        return {"is_ooc": len(violations) > 0, "violations": violations}

不推荐
  • LLM直接承诺「有货」「包邮」不验证
  • 安全检测只用简单关键词匹配
  • System Prompt和短期记忆混在一起
  • 无角色边界定义和OOC检查
推荐
  • API真查替换所有承诺性声明
  • 对抗样本检测（Unicode/Base64/反转）
  • System > Profile > Session三层记忆
  • OOC检测器防止角色越界

常见问题（FAQ）

API真查会不会增加响应延迟？
会，约增加50-200ms。优化：1) 对高频查询商品用Redis缓存库存数据（TTL 30s）；2) 将API查询与LLM生成并行执行；3) 非承诺性内容跳过真查。
对抗样本检测的漏报率多高？
单纯依赖规则（正则/归一化）漏报率约15-20%。建议叠加LLM判別层：将疑似内容送给qwen-turbo做二次判別。规则层拦截80%，LLM层再拦截15%，综合漏报率降至5%以下。
记忆分层后用户画像更新太慢怎么办？
用户画像更新应该走异步事件（用户行为→MQ→画像更新），不阻塞推理路径。画像的TTL设为24小时，确保次日看到最新数据。紧急更新（如用户主动修改偏好）走同步路径。

本章结论

  • API真查：所有承诺性声明必须通过真实API验证，不能依赖LLM输出。
  • 对抗样本防御：Unicode归一化+Base64检测+反转检测+LLM二次判別。
  • 记忆分层：System > Profile > Session三层，上层保护下层。
  • OOC检测：角色边界定义+禁止行为+允许话题，越界时拦截或修正。
  • 生产坑是常态：上线前做红队测试，覆盖承诺/对抗/角色三类场景。
