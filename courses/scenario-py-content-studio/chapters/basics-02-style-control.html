数据流与核心路径

理解数据流是架构设计的核心。内容工坊有三种典型数据流：

路径一：单条实时生成（低延迟）

用户提交单条内容请求 → 接入层校验 → 模板引擎变量替换 → LLM生成 → 安全检测 → 返回结果。适用场景：在线客服回复、实时文案助手。

路径二：批量异步生成（高吞吐）

用户提交批量任务入MQ → Worker消费 → 模板渲染 → LLM填充 → 审核队列 → 人工抽检 → 批量发布。适用场景：商品描述、营销文案批量生产。

Python：数据流核心编排

python
# pipeline_orchestrator.py — 数据流编排
import asyncio
from dataclasses import dataclass
from typing import Optional

@dataclass
class ContentTask:
    task_id: str
    template_name: str
    variables: dict
    status: str = "pending"
    result: Optional[str] = None

class PipelineOrchestrator:
    def __init__(self, template_engine, llm_client, review_client):
        self.template_engine = template_engine
        self.llm = llm_client
        self.review = review_client

    async def process_single(self, task: ContentTask) -> ContentTask:
        # 1. 模板渲染
        prompt = self.template_engine.render(task.template_name, task.variables)
        # 2. LLM生成
        response = await self.llm.generate(prompt)
        # 3. 安全检测（快速通过则直接返回）
        review_result = await self.review.check(response)
        if review_result["passed"]:
            task.result = response
            task.status = "approved"
        else:
            task.status = "rejected"
        return task

    async def process_batch(self, tasks: list[ContentTask],
                            concurrency: int = 5) -> list[ContentTask]:
        sem = asyncio.Semaphore(concurrency)
        async def process(t: ContentTask):
            async with sem:
                return await self.process_single(t)
        return await asyncio.gather(*[process(t) for t in tasks])

CorpAssist · 选型实战 CorpAssist内容工坊采用FastAPI + Jinja2 + RabbitMQ + DashScope(qwen-plus) + Milvus + Redis的组合。接入层用FastAPI处理REST请求，模板生成层用Jinja2渲染+LLM填充，审核层用阿里云内容安全API，存储层用PostgreSQL+Redis。全套技术栈纯Python实现，与Spring Boot后端通过HTTP/REST联调。
不推荐

  • 直接拼接Prompt而不使用模板引擎
  • 同步生成1000条商品描述（超时）
  • 不接入内容安全API
推荐
  • FastAPI异步架构 + Pydantic校验
  • Jinja2模板引擎统一管理所有内容模板
  • RabbitMQ编排批量生成任务
  • 内容安全API + 驳回重写回路

常见问题（FAQ）

为什么选FastAPI而不是Flask？
FastAPI原生异步，调用LLM SDK（httpx异步）时不阻塞Worker线程。Flask的同步模型在LLM高延迟场景下会消耗大量线程资源。另外FastAPI内建Pydantic校验，对模板变量的入参校验天然适合。
RabbitMQ和Redis Stream怎么选？
内容工坊推荐RabbitMQ：1) 消息持久化和ACK机制更成熟；2) 死信队列方便处理失败任务；3) 管理台支持人工干预。Redis Stream适合对吞吐要求极高、可接受少量丢消息的场景。
模板引擎选Jinja2会不会性能不够？
Jinja2渲染性能在毫秒级，对于内容生成场景不是瓶颈。真正的瓶颈在LLM调用（秒级）和审核API。如果真有极端吞吐需求，可以预编译模板或用缓存渲染结果。

本章结论

架构和选型决定了内容工坊的天花板。

  • 四层分离：接入/模板生成/审核/存储，每层独立扩展。
  • 双数据流：单条实时走同步，批量异步走MQ，两套路径不混淆。
  • 选型决策树：每层按吞吐/延迟/预算权衡，FastAPI+Jinja2+RabbitMQ为推荐组合。
  • 审核闭环：内容安全检测是必选，驳回回写模板引擎形成闭环。
  • 面试框架：架构→数据流→选型→质量→扩展，五维度覆盖。

判断练习

  1. 团队使用Flask同步框架调用LLM，高并发下Worker线程池打满。最直接的修复方案是什么？

    参考答案

    方案一（推荐）：迁移到FastAPI异步架构。方案二（临时）：增加Flask线程池大小+引入外部任务队列。但根本解是异步化，避免一个LLM请求阻塞整个Worker。

  2. 审核驳回的内容回写模板引擎，发现模板变量已经变了。怎么处理这种情况？

    参考答案

    两种方案：1) 在批量任务中保存原始变量快照，驳回后恢复原始变量重新生成；2) 驳回时保存LLM生成的完整结果，人工修改后直接发布（不重新生成）。内容工坊推荐方案1，保证内容一致性。
