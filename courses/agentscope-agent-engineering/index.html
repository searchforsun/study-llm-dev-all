AOP 编程模型

AOP（Agent-Oriented Programming） 把 Agent 视为程序的基本单元：每个 Agent 拥有名称、系统提示、模型、工具集与记忆，通过消息传递协作。与面向对象的「类+方法」不同，AOP 强调异步消息与可观测事件流。

AgentScope AOP 组件关系
flowchart TB
    User[UserMsg] --> Agent[Agent 实例]
    Agent --> Model[DashScopeChatModel]
    Agent --> Toolkit[Toolkit / Tools]
    Agent --> Memory[Memory]
    Agent --> Events[Event Stream]
    Events --> UI[AG-UI / SSE]
    Agent --> Hub[Message Hub]
    Hub --> Agent2[其他 Agent]

AgentScope 2.0 以 Agent 为中心，模型、工具、记忆、事件均为可插拔组件。

LangGraph 思维硬套
  • 先画 StateGraph 再强行映射 AgentScope API
  • 把 Msg 当成普通 dict 塞进 state
  • 忽略 reply_stream 事件，只等最终字符串
AOP 原生写法
  • 先定义 Agent 角色与 Toolkit，再考虑 Pipeline / Hub
  • 用 UserMsg / AgentMsg 显式构造消息
  • 消费 EventType 流驱动 UI 与审计

AgentScope 2.0：最小 Agent 骨架

python 
import asyncio
import os
from agentscope.agent import Agent
from agentscope.credential import DashScopeCredential
from agentscope.message import UserMsg
from agentscope.model import DashScopeChatModel
from agentscope.tool import Toolkit

async def main() -> None:
    agent = Agent(
        name="CorpAssist-S3",
        system_prompt="你是 CorpAssist 办公助手，负责日历与工单查询。",
        model=DashScopeChatModel(
            credential=DashScopeCredential(api_key=os.environ["DASHSCOPE_API_KEY"]),
            model="qwen-plus",
        ),
        toolkit=Toolkit(tools=[]),
        max_iters=10,
    )
    reply = await agent.reply(UserMsg("user", "帮我查明天 10 点的会议"))
    print(reply.content)

asyncio.run(main())
先想 10 秒：AOP 与 OOP 最大的工程差异是什么？

OOP 以对象方法调用为主，调用栈同步可见；AOP 以 Agent 间异步 Msg 传递为主，行为由事件流驱动，调试需依赖 EventType 与 trace 而非单线程栈。

AgentScope 2.0 还需要自己画状态图吗？
单 Agent ReAct 不需要；多 Agent 串并行可用 Pipeline API 声明式编排，Message Hub 负责 Msg 路由。仅当嵌入 LangGraph 子流程作为 Tool 时，状态图仍在外部维护。
AOP 是否排斥确定性工作流？
不排斥。Pipeline 可固定步骤顺序；AOP 强调的是 Agent 作为组合单元，而非禁止预定义流程。CorpAssist 审批链可 Pipeline + HITL 实现。

内置 ReAct 循环

AgentScope 2.0 的 ReAct 循环 在 Agent 内部实现：模型生成 → 解析 tool_call → 执行 Toolkit → 将观察写回上下文 → 再次调用模型，直到输出最终文本或达到 max_iters。与 LangGraph 需手写 agent/tools 节点不同，此处循环对外不可见，通过事件暴露中间态。

AgentScope ReAct 循环
sequenceDiagram
    participant U as User
    participant A as Agent
    participant M as DashScopeChatModel
    participant T as Toolkit

    U->>A: UserMsg
    loop max_iters
        A->>M: 推理请求
        M-->>A: thinking / tool_call / text
        alt 有 tool_call
            A->>T: 执行工具
            T-->>A: 观察结果
        else 纯文本
            A-->>U: 最终回复
        end
    end

ReAct 循环在 Agent 内部闭合；外部通过 EventType 观察每一步。

Event System（text/thinking/tool_call）

Event System 将模型输出与工具执行拆成细粒度事件。常见类型包括 TEXT_BLOCK_DELTA（流式文本）、thinking 相关块、TOOL_CALL_START / TOOL_CALL_END 等。CorpAssist 前端可据此展示「正在查日历…」进度条。

EventType 类别含义UI 映射
REPLY_START / END整轮回复边界会话气泡起止
TEXT_BLOCK_*流式可见文本打字机效果
thinking 块模型推理过程（若模型支持）可折叠「思考过程」
TOOL_CALL_*工具名与参数工具调用卡片

消费 reply_stream 事件流

python 
import asyncio
from agentscope.event import EventType
from agentscope.message import UserMsg

async def stream_to_ui(agent, question: str) -> None:
    buffer = []
    async for evt in agent.reply_stream(UserMsg("user", question)):
        match evt.type:
            case EventType.TEXT_BLOCK_DELTA:
                buffer.append(evt.delta)
                print("".join(buffer), end="", flush=True)
            case EventType.TOOL_CALL_START:
                print(f"\n[工具] {evt.tool_name}({evt.args})")
            case EventType.REPLY_END:
                print("\n--- 完成 ---")

asyncio.run(stream_to_ui(agent, "明天有什么会？"))
反模式
  • 只用 reply() 阻塞等待，前端无进度
  • 忽略 TOOL_CALL 事件，用户不知 Agent 在做什么
  • thinking 块直接展示给终端用户（可能泄露推理链）
推荐
  • SSE / WebSocket 转发 EventType 到 AG-UI
  • 工具事件显示名称与耗时，隐藏敏感参数
  • thinking 默认折叠，仅 debug 模式开启

错误重试策略

重试策略 需区分可重试错误（5xx、超时）与不可重试（4xx 参数错误）。Agent 层应限制重试次数，避免 ReAct 循环中对同一坏参数无限重试。

工具失败处理流
flowchart TD
    Call[工具调用] --> OK{成功?}
    OK -->|是| Obs[观察写回上下文]
    OK -->|否| Type{错误类型}
    Type -->|超时/5xx| Retry{重试 < 3?}
    Retry -->|是| Call
    Retry -->|否| Degrade[结构化错误 + 降级提示]
    Type -->|4xx/鉴权| Fail[立即失败 + 人工指引]

结构化错误信息帮助模型修正参数，而非空异常。

带重试的工具包装

python 
import asyncio
from functools import wraps

def retry_tool(max_attempts: int = 3, backoff: float = 1.0):
    def decorator(fn):
        @wraps(fn)
        async def wrapper(*args, **kwargs):
            last_err = None
            for attempt in range(max_attempts):
                try:
                    return await fn(*args, **kwargs)
                except (TimeoutError, ConnectionError) as e:
                    last_err = e
                    await asyncio.sleep(backoff * (attempt + 1))
            return f"TOOL_ERROR: {last_err}（已重试 {max_attempts} 次，请换参数或人工处理）"
        return wrapper
    return decorator
CorpAssist · S3 工单创建 模型误传非法 priority 值时，工具返回 400——不应重试，应返回清晰错误让 Agent 改用 enum 值。网络抖动导致 MCP 超时时，最多重试 2 次后提示用户稍后再试。

自动裁剪 触发时优先删除最旧的 tool 观察，保留最近 user 指令与 system_prompt。生产环境可配合 摘要钩子 在裁剪前将最旧段落压缩为一条 system 摘要 Msg，避免完全丢失早期上下文。

摘要钩子与 keep_recent_turns 如何配合？
当 Token 超过 summarize_after_tokens 时触发钩子：最旧 N 轮 Msg 送入 qwen-turbo 生成一条「历史摘要」Msg，再从 Memory 删除原文。keep_recent_turns 保证最近对话完整保留，摘要 Msg 插入 system 段之后。

python 
import asyncio
import os
from agentscope.agent import ReActAgent
from agentscope.formatter import DashScopeChatFormatter
from agentscope.memory import InMemoryMemory
from agentscope.message import Msg
from agentscope.model import DashScopeChatModel
from agentscope.tool import Toolkit, ToolResponse

async def create_calendar_event(title: str, start_time: str, attendees: str = "") -> ToolResponse:
    """创建日历事件。title 为标题，start_time 为 ISO8601 时间，attendees 为逗号分隔邮箱。"""
    event_id = f"EVT-{hash(title + start_time) % 10000:04d}"
    return ToolResponse(content=f"已创建日历 {event_id}：{title} @ {start_time}")

async def send_email(to: str, subject: str, body: str) -> ToolResponse:
    """发送邮件。to 为收件人，subject 为主题，body 为正文。"""
    return ToolResponse(content=f"邮件已发送至 {to}，主题：{subject}")

async def create_ticket(category: str, description: str) -> ToolResponse:
    """创建 IT 工单。category 为类别，description 为问题描述。"""
    ticket_id = f"TK-{hash(description) % 100000:05d}"
    return ToolResponse(content=f"工单 {ticket_id} 已创建（{category}）")

async def transfer_to_human(reason: str) -> ToolResponse:
    """转接人工。Agent 无法完成、用户明确要求或预算耗尽时调用。"""
    return ToolResponse(content=f"已转人工，原因：{reason}。预计 5 分钟内接入。")

def build_office_agent() -> ReActAgent:
    toolkit = Toolkit()
    for fn in (create_calendar_event, send_email, create_ticket, transfer_to_human):
        toolkit.register_tool_function(fn)
    return ReActAgent(
        name="CorpAssistOffice",
        sys_prompt="你是 CorpAssist 办公助手。优先用工具完成请求；缺参数时追问；无法处理时调用 transfer_to_human。",
        model=DashScopeChatModel(
            model_name="qwen-max",
            api_key=os.environ["DASHSCOPE_API_KEY"],
        ),
        formatter=DashScopeChatFormatter(),
        memory=InMemoryMemory(),
        toolkit=toolkit,
        max_iters=8,
    )

async def main():
    agent = build_office_agent()
    msg = await agent(Msg("user", "帮我把周五产品评审加到日历并邮件通知 team@corp.com", "user"))
    print(msg.get_text_content())

asyncio.run(main())

多一跳 LLM 增加延迟与成本，且分类错误会级联。ReActAgent 在单循环内完成意图识别与工具调用，事件流可观测每一步，更适合生产排障。

工具 docstring 写多长合适？
一句话说明职责 + 每个参数一行说明 + 一条「不要用于…」反例。CorpAssist 四类工具 docstring 各 3～5 行即可；过长会挤占 tool schema Token。

日历/邮件/工单工具

三类工具应遵循 CorpAssist 统一契约：输入 schema 明确、返回结构化文本、失败时返回可读的 ToolResponse 而非抛异常。工单工具对接内部 ITSM，邮件工具需校验收件人域名。

本章先记住 3 件事

主管-工人模式

CorpAssist 项目协调场景：Supervisor 理解用户意图，调度 CalendarWorker、MailWorker、TicketWorker。Supervisor 不直接操作外部系统，而是通过 MsgHub 广播任务描述，专家 Agent 在共享上下文中各自 reply。

CorpAssist · 项目启动会 用户：「订明天 10 点 A301，发邮件通知项目组，并建 IT 工单申请投影设备。」Supervisor 在 Hub 内依次调度 CalendarWorker → MailWorker → TicketWorker，每个 Worker 只看到自己领域工具，最终 Supervisor 汇总三份汇报给用户。

角色分工与 Handoff

Handoff 将子 Agent 包装为工具，Orchestrator 自主决定调用哪个 Worker；Worker 仅收到任务描述，不共享完整对话。MsgHub 则共享完整上下文，适合需要交叉引用的协作。

MsgHub 不适用
  • 独立子任务可完全并行且无交叉引用
  • Worker 不应看到彼此中间结果
  • 需要动态创建/销毁 Worker
Handoff 更合适
  • Orchestrator 用 toolkit 注册 Worker 工具
  • 子任务隔离、结果汇总
  • 与 Agent-as-Tool 模式一致

MultiAgentFormatter 选错会怎样？
模型无法区分 Msg 来自哪个 Agent，可能把 CalendarWorker 的回复当成用户输入，或 Supervisor 误以为自己已执行工具。CorpAssist 多 Agent 场景必须使用 DashScopeMultiAgentFormatter。
MsgHub 内 Agent 调用顺序如何定？
MsgHub 不自动调度——编排代码显式 await agent() 顺序。Supervisor 先理解 → 各 Worker 执行 → Supervisor 汇总是常见三段式；顺序错误会导致 Worker 在缺少上下文时行动。
MsgHub 与 Pipeline sequential 如何组合？
Hub 内某一阶段可跑 sequential_pipeline：例如 Supervisor 广播后，三个 Worker 用 Pipeline 串行处理子 SOP。见 practice-03 组合模式。

  • MsgHub：async 上下文管理器，自动广播参与者 reply。

  • 显式传递：关闭 auto_broadcast 时用 broadcast 精确路由。
  • Handoff：任务隔离场景优于 MsgHub 全量共享。
  • Java 对照：Sub-agent 偏层级，MsgHub 偏对等广播。

import asyncio

async def fanout_with_timeout(agents, msg, timeout_sec=30):
    async def run_one(agent):
        return await asyncio.wait_for(agent(msg), timeout=timeout_sec)
    tasks = [run_one(a) for a in agents]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    return [r for r in results if not isinstance(r, Exception)]

生产做法

  • 只读 Fanout 优先并行
  • return_exceptions=True 部分降级

失败重试与降级

失败重试与降级 分三层：模型调用 model_config.max_retries、工具超时重试、Pipeline 步骤失败跳过或转人工。Sequential 中某步失败时，可将错误 Msg 注入下一步由「修复 Agent」处理，或直接 abort 并调用 transfer_to_human。

python 
async def resilient_sequential(agents, msg, fallback_agent):
    current = msg
    for agent in agents:
        try:
            current = await agent(current)
        except Exception as e:
            err = Msg("system", f"步骤 {agent.name} 失败：{e}", "system")
            current = await fallback_agent(err)
            break
    return current

循环检测

循环检测 防止 Agent 在 Pipeline 或 ReAct 中反复调用相同工具。策略：记录最近 N 次 (tool_name, args_hash)，相同组合连续 3 次则强制终止并降级。

python 
import json
from collections import deque

class LoopGuard:
    def __init__(self, window: int = 3):
        self.recent: deque = deque(maxlen=window)

    def check(self, tool_name: str, args: dict) -> bool:
        sig = (tool_name, json.dumps(args, sort_keys=True))
        self.recent.append(sig)
        if len(self.recent) == self.recent.maxlen and len(set(self.recent)) == 1:
            return True  # 检测到循环
        return False

# 在 middleware 或 post_acting hook 中调用 loop_guard.check(...)

Realtime Steering

Realtime Steering 允许用户在 Agent 执行过程中随时干预：调用 agent.interrupt() 取消当前任务，Agent 运行 handle_interrupt() 做清理，之后可注入新指令继续。适用于用户发现 Agent 偏航、或紧急变更优先级。

CorpAssist · 紧急改向 用户发起「批量发邮件通知全员」，Agent 正在生成收件人列表时用户点击「停止」并改口「只发项目组」——interrupt() 终止当前 Acting，清理半成品状态，注入新 UserMsg 重新 reply。

为什么要学本章

CorpAssist 知识库问答不能仅靠静态 RAG 管线——Agent 需自主决定何时检索、检索什么、是否多轮。Agentic RAG 把 RAG 系统（Python）rag-system-py 的检索链封装为 Agent 工具，并配合 引用溯源 与 幻觉抑制 策略落地生产。

本章先记住 3 件事
  • 检索是工具不是前置步骤：Agent 按需调用，避免每次都全库检索。
  • 引用必须绑 chunk metadata：doc_id、page、score 写入回答。
  • 低置信度拒答：score 低于阈值时拒绝编造，建议转人工或换问法。

from agentscope.agent import ReActAgent
from agentscope.tool import Toolkit, ToolResponse
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

# 复用 rag-system-py 课 demo 构建的 vectorstore
vectorstore = Chroma(persist_directory="./data/chroma", embedding_function=OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

async def search_knowledge_base(query: str) -> ToolResponse:
    """检索 CorpAssist 企业知识库。输入自然语言问题，返回带 doc_id/page/score 的片段。"""
    docs = retriever.invoke(query)
    if not docs:
        return ToolResponse(content="未找到相关文档，请换关键词或转人工。")
    blocks = []
    for i, doc in enumerate(docs, 1):
        meta = doc.metadata
        score = meta.get("score", "N/A")
        blocks.append(
            f"[{i}] doc={meta.get('source','?')} page={meta.get('page','?')} score={score}\n{doc.page_content[:400]}"
        )
    return ToolResponse(content="\n---\n".join(blocks))

toolkit = Toolkit()
toolkit.register_tool_function(search_knowledge_base)

kb_agent = ReActAgent(

    formatter=DashScopeChatFormatter(),

Agentic RAG 比固定 RAG 链更费 Token，何时仍值得？

当问题类型混合（部分需检索、部分需工具操作）、或需多轮改写 query 时，Agent 按需检索更省平均 Token。纯 FAQ 高 QPS 场景可保留固定链 + 小模型。

Agentic RAG 如何与 rag-system-py 评估指标对齐？
复用 rag-system-py 的 faithfulness / answer_relevance 评测集：Agentic 路径在 golden 中增加 must_call_tools: ["search_knowledge_base"] 断言。引用准确率单独测 citation 与 chunk 对应关系。
多轮检索何时触发？
首次检索 score 偏低或结果不覆盖子问题时，Agent 在 ReAct 第二轮改写 query 再调工具。max_iters=6 限制最多约 2～3 次检索；更多轮应拆分为明确子问题或转人工。
Chroma 与 DashScope Embedding 如何配合？
索引阶段 embedding 模型须与查询阶段一致（见 rag-system-py）。CorpAssist 生产可用 DashScope text-embedding-v3；Agent 工具内只做 retriever.invoke，不在线换 embedding 模型。
CorpAssist · 制度问答拒答 用户问「火星移民政策」——检索 score 均 <0.65，工具返回拒答文案，Agent 不编造数字并建议联系 HR 或换关键词。审计记录 tool 空结果与拒答原因。
静态 RAG 链
  • 所有问题都检索，闲聊也 embedding
  • 无法在同一 Agent 内调工单工具
  • query 改写需额外 pipeline 节点
Agentic RAG
  • 按需检索，混合意图统一入口
  • 检索 + 办公工具同一 ReActAgent
  • 多轮 query 改写原生支持

# session_store.py — Redis 会话隔离示意
from dataclasses import dataclass
import json
import redis

@dataclass
class SessionKey:
    tenant_id: str
    session_id: str

    @property
    def redis_key(self) -> str:
        return f"corpassist:agent:{self.tenant_id}:{self.session_id}:msgs"

class SessionStore:
    def __init__(self, r: redis.Redis, ttl_seconds: int = 86400):
        self.r = r
        self.ttl = ttl_seconds

    def load_messages(self, key: SessionKey) -> list[dict]:
        raw = self.r.get(key.redis_key)
        return json.loads(raw) if raw else []

    def append_message(self, key: SessionKey, msg: dict) -> None:
        msgs = self.load_messages(key)
        msgs.append(msg)
        self.r.setex(key.redis_key, self.ttl, json.dumps(msgs, ensure_ascii=False))

    def purge(self, key: SessionKey) -> None:
        self.r.delete(key.redis_key)

    history = session_store.load_messages(SessionKey(x_tenant_id, req.session_id))
    user_msg = Msg(name="user", content=req.message, role="user")

    async def event_generator():
        step = 0
        async for event in agent.run_stream([*history, user_msg]):
            step += 1
            payload = {
                "event_type": event.type,
                "session_id": req.session_id,
                "step": step,
                "trace_id": x_trace_id,
                "payload": event.to_dict(),
            }
            yield f"data: {json.dumps(payload, ensure_ascii=False)}\n\n"
        yield "event: done\ndata: {}\n\n"

    return StreamingResponse(event_generator(), media_type="text/event-stream")

常见问题（FAQ）

Agent Service 和 llm-gateway 应该合并还是分开？
分开。 Gateway 负责模型路由、Token 计量与密钥；Agent Service 负责 ReAct 循环、工具编排与会话。合并会导致 Gateway 承担有状态逻辑，难以水平扩展。
SSE 连接断开时 Agent 还在跑怎么办？
客户端断开应触发 cancellation token，AgentScope 侧停止后续 tool_call。若业务要求任务继续（如后台订会议室），改用异步 Job 模式：POST 返回 task_id，轮询或 Webhook 通知结果。
多实例部署时 session 粘滞吗？
不需要。 会话状态存 Redis，任意 Pod 可处理同一 session_id。仅需保证同一 session 并发请求串行化（分布式锁或队列）。

本章结论

  • 双接口：同步 /v1/chat 供评测；SSE /v1/chat/stream 供 UI。
  • Event 映射：SSE payload 对齐 AgentScope text/thinking/tool_call。
  • 多租户：tenant 级工具白名单 + Redis session 外置。
  • FastAPI：统一 Agent 内核，StreamingResponse 转发事件流。
  • Spring 契约：WebClient + 错误码映射 + trace_id 透传。

常见问题（FAQ）

CorpAssist 应该统一一个框架还是多框架并存？
工具层统一，编排层可并存。 查单、检索等工具只写一份；S2 可继续 LangGraph，S3 用 AgentScope Message Hub。BFF 通过统一 Agent Service 契约屏蔽差异。
Dify 导出的 workflow 能直接跑在 AgentScope 吗？
不能一键导入。 Dify DSL 与 AgentScope 抽象不同。可复用 Prompt 文本与 HTTP 工具 URL，编排逻辑需重写。建议 Dify 仅作 Prompt 实验室。
CrewAI 适合 CorpAssist 生产吗？
不推荐作 S3 生产主框架。 CrewAI 擅长角色扮演原型，但 HITL、步数预算、双栈对照与评测门禁需大量自研。可用于 2 周 PoC，生产迁移到 AgentScope 或 LangGraph。

# eval_capstone.py — 离线 golden set 评测
import json
import pytest

GOLDEN = json.load(open("golden_set.json"))  # 10 条用例

def judge_success(expected: dict, actual: dict) -> bool:
    if expected.get("must_call_tools"):
        called = set(actual["tools_called"])
        if not set(expected["must_call_tools"]).issubset(called):
            return False
    if "must_contain" in expected:
        return all(k in actual["content"] for k in expected["must_contain"])
    return True

@pytest.mark.parametrize("case", GOLDEN)
def test_capstone_case(case):
    result = run_capstone(case["input"])
    assert judge_success(case["expected"], result)
    assert result["steps"] <= 5

def test_aggregate_metrics():
    results = [run_capstone(c["input"]) for c in GOLDEN]
    success_rate = sum(1 for r, c in zip(results, GOLDEN)
                       if judge_success(c["expected"], r)) / len(GOLDEN)
    avg_steps = sum(r["steps"] for r in results) / len(results)
    assert success_rate >= 0.9, f"success_rate={success_rate}"
    assert avg_steps <= 4.0, f"avg_steps={avg_steps}"

# loop_detector.py — AgentScope 死循环检测 Hook（AOP）
from agentscope.pipelines import functional_user_agent

def detect_tool_loop(tool_trace: list[tuple[str, dict]], window: int = 3) -> bool:
    """最近 window 次工具调用完全相同则判定死循环"""
    if len(tool_trace) < window:
        return False
    recent = tool_trace[-window:]
    return len(set(recent)) == 1

class LoopGuardHook:
    def post_tool_call(self, agent, tool_name: str, args: dict, result: str):
        trace = agent.get_tool_trace()
        if detect_tool_loop(trace):
            raise RuntimeError(
                f"LOOP_DETECTED: {tool_name} called {len(trace)} times with same args"
            )

from agentscope.message import Msg

def trim_messages(msgs: list[Msg], max_rounds: int = 10) -> list[Msg]:
    """保留 system + 最近 max_rounds 轮 user/assistant"""
    system = [m for m in msgs if m.role == "system"]
    dialogue = [m for m in msgs if m.role != "system"]
    keep = dialogue[-(max_rounds * 2):]
    return system + keep

def compress_tool_results(msgs: list[Msg], max_chars: int = 500) -> list[Msg]:
    out = []
    for m in msgs:
        if m.role == "tool" and len(m.content) > max_chars:
            out.append(Msg(m.name, m.content[:max_chars] + "…[truncated]", m.role))
        else:
            out.append(m)
    return out

# sse_heartbeat.py — 防止代理 idle 断连
import asyncio

async def stream_with_heartbeat(agent, messages, ping_interval=15):
    queue = asyncio.Queue()
    agent_task = asyncio.create_task(_run_agent(agent, messages, queue))

    try:
        while True:
            try:
                event = await asyncio.wait_for(queue.get(), timeout=ping_interval)
                if event is None:
                    break
                yield event
            except asyncio.TimeoutError:
                yield ": ping\n\n"  # SSE comment 保活
    finally:
        if not agent_task.done():
            agent_task.cancel()

Trace：每次 tool_result 仅返回 {"title": "年假制度"} 无正文。

修复：工具返回 summary 字段 + sys_prompt 增加「单次搜索足够则回答」。

案例 2：假执行 — 「已订 A301」但未调用 book_meeting

现象：用户到会议室发现未预订，Trace 无 tool_call。

根因：qwen 某版本 function call 偶发失败，模型在 text 中幻觉执行结果。

修复：后置校验——回复含「预订成功」时必须存在 book_meeting tool_call + confirmation_id。

案例 3：SSE 断连 + 重复订会议室

现象：前端超时后用户重试，产生两个预订。

修复：BFF 幂等键 Idempotency-Key + Agent cancel on disconnect + book 工具幂等。

常见问题（FAQ）

AgentScope Trace 和 LangSmith 能否同时使用？
可以。AgentScope 2.0 支持 OpenTelemetry 导出；LangSmith 可作为统一 Trace 后端。面试中强调：排障习惯应统一到一种 Trace UI，避免双系统查不到关联。
max_iters 设多少合适？
CorpAssist 短链路 5～8；S3 复杂任务 10～12。超过 12 应考虑 Harness 任务分解，而非无限提高 max_iters。
面试如何体现 AgentScope 深度而非泛泛 Agent 经验？
举具体 API：Msg 结构、ReActAgent max_iters、Event type 枚举、Toolkit 注册、AOP Hook 做 LoopGuard。对比 LangGraph checkpoint 说明迁移踩坑。

本章结论

  • 死循环：按 Trace 模式分类——同工具/AB 切换/无工具/偏航。
  • 假执行：Trace 无 tool_call 却声称完成 → Prompt + 审计 + golden。
  • 记忆溢出：Msg 裁剪 + sys_prompt 前置 + tool_result 压缩。
  • SSE 断连：heartbeat + cancel + 幂等 + 工具 timeout。
  • 面试：STAR 案例 + AgentScope 特有 API 细节。
