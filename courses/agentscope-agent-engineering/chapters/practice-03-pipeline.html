为什么要学本章

MsgHub 适合讨论式协作；CorpAssist 许多 SOP（入职流程、审批链）需要确定性顺序。Sequential Pipeline 与 Fanout Pipeline 提供可复用的串行/并行编排，并内置并发工具执行与失败降级能力。

本章先记住 3 件事
  • 串行 = 上一步输出是下一步输入：等价于连续 await agent(msg)。
  • 并行 = fanout + enable_gather：asyncio.gather 并发 I/O。
  • Pipeline 可与 MsgHub 组合：Hub 内跑 sequential 处理子流程。

串行/并行步骤

CorpAssist 员工入职 SOP：HR 填表 → IT 开账号 → 行政发设备。三步 Agent 顺序固定，用 sequential_pipeline。若需法务、安全、合规三方并行审阅同一份政策，用 fanout_pipeline。

CorpAssist · 新员工入职 HR Agent 收集员工信息并生成 onboarding 包 → IT Agent 依据包开 AD 账号 → Admin Agent 分配笔记本。任一步失败应注入错误 Msg 给 fallback Agent，而非静默跳过。

Sequential 传递 Msg；Fanout 深拷贝输入给各 Agent，返回 Msg 列表。

python 
import asyncio
from agentscope.message import Msg
from agentscope.pipeline import sequential_pipeline, fanout_pipeline, SequentialPipeline

async def onboarding_flow(hr, it, admin, employee_name: str):
    initial = Msg("user", f"为新员工 {employee_name} 办理入职", "user")
    # 串行：HR → IT → Admin
    result = await sequential_pipeline(agents=[hr, it, admin], msg=initial)
    return result.get_text_content()

async def policy_review(legal, security, compliance, policy_text: str):
    msg = Msg("user", f"审阅政策：{policy_text}", "user")
    # 并行：三路同时审阅
    reviews = await fanout_pipeline(
        agents=[legal, security, compliance],
        msg=msg,
        enable_gather=True,
    )
    return [r.get_text_content() for r in reviews]

# 可复用 Pipeline 对象
onboarding_pipe = SequentialPipeline(agents=[hr_agent, it_agent, admin_agent])
final_msg = await onboarding_pipe(msg=initial_msg)
先想 10 秒：入职 SOP 用 MsgHub 讨论式协作合适吗？

不合适。SOP 步骤顺序固定、无交叉引用需求，Sequential Pipeline 更简洁可测。MsgHub 适合步骤间需协商的场景；确定性流程用 Pipeline 降低 trace 复杂度。

并发工具执行

AgentScope 2.0 ReActAgent 在单轮 Acting 阶段可批量并发执行多个 tool_call（模型一次返回多个调用）。Pipeline 层的 enable_gather=True 则在多 Agent 维度并发。CorpAssist 查「三系统状态」场景：Fanout 三个只读 Agent 各查一系统，总延迟接近最慢单路而非三者之和。

并发层级机制典型场景
单 Agent 多工具ReAct acting batch同时查日历+通讯录
多 Agent Fanoutasyncio.gather多专家并行审阅
外部 I/O工具内 async/awaitHTTP 并行请求
python 
import asyncio

async def fanout_with_timeout(agents, msg, timeout_sec=30):
    async def run_one(agent):
        return await asyncio.wait_for(agent(msg), timeout=timeout_sec)
    tasks = [run_one(a) for a in agents]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    return [r for r in results if not isinstance(r, Exception)]
并发陷阱
  • Fanout 不设超时，一路阻塞全体
  • 并行写同一资源无锁
  • gather 异常未捕获导致整体失败
生产做法
  • wait_for 单 Agent 超时
  • 只读 Fanout 优先并行
  • return_exceptions=True 部分降级

失败重试与降级

失败重试与降级 分三层：模型调用 model_config.max_retries、工具超时重试、Pipeline 步骤失败跳过或转人工。Sequential 中某步失败时，可将错误 Msg 注入下一步由「修复 Agent」处理，或直接 abort 并调用 transfer_to_human。

python 
async def resilient_sequential(agents, msg, fallback_agent):
    current = msg
    for agent in agents:
        try:
            current = await agent(current)
        except Exception as e:
            err = Msg("system", f"步骤 {agent.name} 失败：{e}", "system")
            current = await fallback_agent(err)
            break
    return current

# model_config 层重试（ReActAgent 构造参数）
from agentscope.model import ModelConfig
model_cfg = ModelConfig(max_retries=2, fallback_model="qwen-turbo")
不推荐
  • Pipeline 内无限重试同一步
  • 失败静默跳过不留 audit
  • 并行 Fanout 不设单 Agent 超时
推荐
  • 指数退避 + 最多 2 次重试
  • 失败 Msg 写入审计日志
  • Fanout 用 asyncio.wait_for 包单层超时
Pipeline 步骤失败应重试还是 skip？
CorpAssist SOP 关键步骤（如 IT 开账号）失败应 abort 并转人工；非关键步骤（如发送欢迎邮件）可 skip 并记录 audit。策略写在 Pipeline 编排层，而非 Agent sys_prompt。

循环检测

循环检测 防止 Agent 在 Pipeline 或 ReAct 中反复调用相同工具。策略：记录最近 N 次 (tool_name, args_hash)，相同组合连续 3 次则强制终止并降级。

python 
import json
from collections import deque

class LoopGuard:
    def __init__(self, window: int = 3):
        self.recent: deque = deque(maxlen=window)

    def check(self, tool_name: str, args: dict) -> bool:
        sig = (tool_name, json.dumps(args, sort_keys=True))
        self.recent.append(sig)
        if len(self.recent) == self.recent.maxlen and len(set(self.recent)) == 1:
            return True  # 检测到循环
        return False

# 在 middleware 或 post_acting hook 中调用 loop_guard.check(...)
Sequential Pipeline 会把上一步的 tool 原始结果传给下一步吗？

传的是上一步 Agent 的最终 Msg（含文本与 metadata），不是未加工的 tool JSON。若下一步需要结构化数据，上一步 Agent 的 sys_prompt 应要求输出固定格式。

flowchart TB

讨论式协作用 Hub；Hub 内子流程用 Pipeline 固化 SOP。

LoopGuard 与 max_iters 如何互补？
max_iters 限制总轮次；LoopGuard 检测相同 tool+args 重复——模型「卡住」反复查同一日历。两者同时使用：LoopGuard 提前终止无进展循环，max_iters 兜底 worst-case。
enable_gather=False 何时使用？
当 Fanout 各路结果需严格按 agents 列表顺序处理、或需避免并发压垮下游 API 时，设 False 串行 Fanout。牺牲延迟换确定性。

CorpAssist 入职 Pipeline 完整编排

python 
async def corpassist_onboarding(employee: str):
    hr = make_agent("HR", "收集信息并生成 onboarding 包")
    it = make_agent("IT", "依据 onboarding 包开 AD 账号")
    admin = make_agent("Admin", "分配设备与工位")
    fallback = make_agent("Fallback", "处理失败并转人工")

    msg = Msg("user", f"为新员工 {employee} 办理入职", "user")
    try:
        return await sequential_pipeline(agents=[hr, it, admin], msg=msg)
    except Exception as e:
        err = Msg("system", f"Pipeline 失败：{e}", "system")
        return await fallback(err)
CorpAssist · 政策并行审阅 法务、安全、合规三 Agent 用 fanout_pipeline 同时审阅新版远程办公政策，Supervisor 汇总三路「通过/修改/否决」意见——总耗时 ≈ max(单路) 而非三路之和。
Pipeline 与 StateGraph 如何选型？
CorpAssist 确定性 SOP（入职、审批链顺序固定）优先 AgentScope Pipeline；需要可视化分支图、复杂 conditional 或已有 LangGraph checkpoint 资产时用 StateGraph。两者可共存：Pipeline 步骤内嵌 LangGraph 子图作 Tool。

本章结论

  • Sequential：SOP 顺序流程，Msg 逐步传递。
  • Fanout：并行审阅/采集，enable_gather=True 提速。
  • 重试降级：模型、工具、Pipeline 三层，失败须可审计。
  • 循环检测：相同 tool+args 连续 N 次即终止。
  • 组合：MsgHub 内嵌 Pipeline 处理复杂子流程。
