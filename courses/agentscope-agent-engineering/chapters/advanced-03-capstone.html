短链路限定 3 工具、单 Agent；多 Agent Message Hub 作加分项非必需。

python 
# capstone_agent.py — CorpAssist S3 短链路 Capstone
import agentscope
from agentscope.agents import ReActAgent
from agentscope.message import Msg

agentscope.init(model_configs=[{
    "config_name": "qwen",
    "model_type": "dashscope_chat",
    "model_name": "qwen-max",
}])

CAPSTONE_SYS_PROMPT = """你是 CorpAssist 办公助手。任务：理解用户意图，按顺序调用工具。
规则：
1. 先 search_policy 查制度（若用户问及制度）
2. 再 book_meeting_room 订会议室
3. 最后 send_email 通知参会人
4. 每步确认工具返回后再继续；失败则说明原因并停止
5. 最多 5 步，完成后用三句话总结"""

def run_capstone(user_input: str) -> dict:
    agent = ReActAgent(
        name="s3_capstone",
        model_config_name="qwen",
        sys_prompt=CAPSTONE_SYS_PROMPT,
        tools=[search_policy, book_meeting_room, send_email],
        max_iters=5,
    )
    msg = Msg(name="user", content=user_input, role="user")
    response = agent(msg)
    return {
        "content": response.content,
        "steps": agent.get_step_count(),  # 示意 API
        "tools_called": agent.get_tool_trace(),
    }

先想 10 秒：Capstone 为何限定 3 工具、≤5 步，而不是直接做 Message Hub 多 Agent？

短链路 Capstone 目标是可评测的工程基线：golden set 能稳定判定 pass/fail。多 Agent 引入 Msg 路由变量，badcase 难归因。先证明单 Agent ReAct + 工具链 + 指标门禁，再扩展 Message Hub 是合理递进。

  • DELIVERY.md 含已知限制

成功率/步数指标

Capstone 验收不靠「感觉能跑」，而靠可量化指标——核心看 task success rate 与 平均步数 是否同时过线。与 评测、实验与用户反馈闭环llm-evaluation-quality 衔接。

指标定义Capstone 门禁采集方式
Task Success Rategolden set 中最终答案满足判定规则的比例≥ 90%LLM-as-Judge + 规则

python 
# eval_capstone.py — 离线 golden set 评测
import json
import pytest

GOLDEN = json.load(open("golden_set.json"))  # 10 条用例

def judge_success(expected: dict, actual: dict) -> bool:
    if expected.get("must_call_tools"):
        called = set(actual["tools_called"])
        if not set(expected["must_call_tools"]).issubset(called):
            return False
    if "must_contain" in expected:
        return all(k in actual["content"] for k in expected["must_contain"])
    return True

@pytest.mark.parametrize("case", GOLDEN)
def test_capstone_case(case):
    result = run_capstone(case["input"])
    assert judge_success(case["expected"], result)
    assert result["steps"] <= 5

def test_aggregate_metrics():
    results = [run_capstone(c["input"]) for c in GOLDEN]
    success_rate = sum(1 for r, c in zip(results, GOLDEN)
                       if judge_success(c["expected"], r)) / len(GOLDEN)
    avg_steps = sum(r["steps"] for r in results) / len(results)
    assert success_rate >= 0.9, f"success_rate={success_rate}"
    assert avg_steps <= 4.0, f"avg_steps={avg_steps}"
CorpAssist · Capstone 答辩 评审问：「成功率 91%、avg_steps 3.8，但有一条 golden 因 send_email 参数缺失失败——你如何修？」应检查 sys_prompt 是否要求缺参追问、工具 docstring 是否标注必填字段，并在 golden 增加「缺参会人」边界用例验证修复。
golden set 判定用 LLM-as-Judge 还是规则？
Capstone 推荐规则优先：must_call_tools + must_contain 关键字。LLM Judge 作补充，避免 Judge 本身波动导致 CI flaky。10 条用例中至少 8 条应纯规则判定。
Mock 工具返回格式应如何设计？
返回与真实 API 同结构的 JSON 字符串（含 id、status 字段），便于后续替换为 HTTP 工具而不改 Agent 逻辑。在 DELIVERY.md 注明 Mock 字段与生产 API 映射表。

长任务升级 → Harness 课

当任务超出短链路能力——步数 >10、跨会话恢复、多文件批处理、子 Agent 并行——应升级到 AgentScope Harness 长任务工程agentscope-harness-engineering。Harness 提供任务分解、检查点与长时监控，避免在 ReAct 上无限堆叠 max_iters。

常见问题（FAQ）

Capstone 必须接真实日历/邮件 API 吗？
不必须。 Mock 工具 + golden set 可完成本课验收。交付文档须注明 Mock 与生产的差异及接入计划。
成功率 85% 能否通过？
不能。 门禁 ≥90%。85% 说明 Prompt、工具 description 或 golden 判定需迭代；先修 badcase 再答辩。
短链路 Capstone 与毕业项目 M3 Agent 有什么关系？
本章是本课结业 Capstone（AgentScope 工程能力证明）；毕业项目路线图llm-application-fundamentals 的 M3 是全栈五场景里程碑。本章产出可作为 M3 的 AgentScope 技术栈证明。
