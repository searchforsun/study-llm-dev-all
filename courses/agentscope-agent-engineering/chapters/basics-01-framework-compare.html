本章先记住 3 件事

AOP 编程模型

AOP（Agent-Oriented Programming） 把 Agent 视为程序的基本单元：每个 Agent 拥有名称、系统提示、模型、工具集与记忆，通过消息传递协作。与面向对象的「类+方法」不同，AOP 强调异步消息与可观测事件流。

AgentScope AOP 组件关系
flowchart TB
    User[UserMsg] --> Agent[Agent 实例]
    Agent --> Model[DashScopeChatModel]
    Agent --> Toolkit[Toolkit / Tools]
    Agent --> Memory[Memory]
    Agent --> Events[Event Stream]
    Events --> UI[AG-UI / SSE]
    Agent --> Hub[Message Hub]
    Hub --> Agent2[其他 Agent]

AgentScope 2.0 以 Agent 为中心，模型、工具、记忆、事件均为可插拔组件。

LangGraph 思维硬套
  • 先画 StateGraph 再强行映射 AgentScope API
  • 把 Msg 当成普通 dict 塞进 state
  • 忽略 reply_stream 事件，只等最终字符串
AOP 原生写法
  • 先定义 Agent 角色与 Toolkit，再考虑 Pipeline / Hub
  • 用 UserMsg / AgentMsg 显式构造消息
  • 消费 EventType 流驱动 UI 与审计

AgentScope 2.0：最小 Agent 骨架

python 
import asyncio
import os
from agentscope.agent import Agent
from agentscope.credential import DashScopeCredential
from agentscope.message import UserMsg
from agentscope.model import DashScopeChatModel
from agentscope.tool import Toolkit

async def main() -> None:
    agent = Agent(
        name="CorpAssist-S3",
        system_prompt="你是 CorpAssist 办公助手，负责日历与工单查询。",
        model=DashScopeChatModel(
            credential=DashScopeCredential(api_key=os.environ["DASHSCOPE_API_KEY"]),
            model="qwen-plus",
        ),
        toolkit=Toolkit(tools=[]),
        max_iters=10,
    )
    reply = await agent.reply(UserMsg("user", "帮我查明天 10 点的会议"))
    print(reply.content)

asyncio.run(main())
先想 10 秒：AOP 与 OOP 最大的工程差异是什么？

OOP 以对象方法调用为主，调用栈同步可见；AOP 以 Agent 间异步 Msg 传递为主，行为由事件流驱动，调试需依赖 EventType 与 trace 而非单线程栈。

AgentScope 2.0 还需要自己画状态图吗？
单 Agent ReAct 不需要；多 Agent 串并行可用 Pipeline API 声明式编排，Message Hub 负责 Msg 路由。仅当嵌入 LangGraph 子流程作为 Tool 时，状态图仍在外部维护。
AOP 是否排斥确定性工作流？
不排斥。Pipeline 可固定步骤顺序；AOP 强调的是 Agent 作为组合单元，而非禁止预定义流程。CorpAssist 审批链可 Pipeline + HITL 实现。

Msg/Agent 抽象

Msg 是 AgentScope 的统一消息类型：UserMsg(name, content) 表示用户输入，Agent 回复封装为带角色与元数据的结构化对象。多 Agent 协作时，Message Hub 在 Agent 之间路由 Msg，而非共享可变 state dict。

与 LangGraph StateGraph 选型

前置课中的 StateGraph 以状态 + 节点 + 边描述流程，适合步骤固定、需 checkpoint 恢复的单 Agent 或有限分支。AgentScope 2.0 的 Pipeline / Message Hub 覆盖串并行工作流，且内置 ReAct、MCP、事件流，更适合多 Agent + 工具生态 + DashScope 原生的 CorpAssist 栈。

框架选型决策流
flowchart TD
    Q1[需要精细 checkpoint 与 HITL interrupt？] -->|强需求| LG[LangGraph StateGraph]
    Q1 -->|一般| Q2[多 Agent 显式 Msg 路由？]
    Q2 -->|是| AS[AgentScope Message Hub]
    Q2 -->|否| Q3[需 AG-UI 事件流 + MCP 原生？]
    Q3 -->|是| AS
    Q3 -->|否| LG
    LG --> Migrate[已有 LangGraph 资产可渐进迁移]
    AS --> CorpAssist[CorpAssist S3 默认栈]

两框架可共存：LangGraph 做确定性子流程，AgentScope 做多 Agent 编排外壳。

CorpAssist 场景适配

CorpAssist 北星能力 NS1（可靠问答）与 NS3（成本可控）在 S3 办公自动化中同时出现：简单日历查询适合单 Agent + 少步 ReAct；跨系统报告生成适合 Message Hub 多 Agent + Pipeline。本章为后续 单 Agent：CorpAssist 办公助手 与 Message Hub 多 Agent 协作 打基础。

  • 范式：单 Agent + DashScope + 2～3 个工具
  • 框架：AgentScope Agent 即可，不必上 StateGraph
  • 预算：max_iters=8，控制 NS3 成本
  • 范式：主管 Agent + 数据/写作 Worker
  • 框架：Message Hub + Pipeline 串并行
  • 质量：关键步骤 HITL，见 Human-in-the-loop 与 Realtime Steering
已有 LangGraph 的 CorpAssist 代码必须全部重写吗？
不必。可保留 LangGraph 子图处理确定性 ETL，外层用 AgentScope Agent 调用该子图作为 Tool。渐进迁移比一次性替换风险更低。
AgentScope 2.0 还保留 ReActAgent 类吗？
2.0 以 Agent 为主入口并内置 ReAct 循环；部分文档仍引用 ReActAgent 别名。下一章 ReActAgent 与事件流 详述事件消费方式。

本章结论

  • AOP 编程模型：以 Agent + Msg + 事件为一等公民，而非先画状态图。
  • Msg/Agent 抽象：UserMsg / AgentMsg 贯穿单 Agent 与 Message Hub 多角色协作。
  • StateGraph 选型：LangGraph 擅 checkpoint 与确定性图；AgentScope 擅多 Agent、MCP 与 DashScope 原生集成。
  • CorpAssist 适配：S3 轻量任务单 Agent，复杂报告用 Hub + Pipeline，可与既有 LangGraph 子流程共存。
