为什么要学本章

CorpAssist S3 复杂任务需要「主管 + 专家工人」分工。MsgHub 是 AgentScope 的消息广播中枢，让多个 ReActAgent 在同一上下文中互相 observe，无需手写消息转发。本章对比 Handoff 与 Java Sub-agent 模式，帮你选对协作范式。

学完你能
  • 用 主管-工人模式 分解 CorpAssist 任务
  • 在 MsgHub 内实现显式消息传递与动态增删参与者
  • 区分 MsgHub 广播协作与 Handoff 任务委派
  • 对照 AgentScope Java Sub-agent 的异同
本章先记住 3 件事
  • MsgHub = 自动广播：参与者 reply 时，其他人自动 observe。
  • Handoff = 层级委派：Orchestrator 创建 Worker，上下文隔离更强。
  • 显式广播：hub.broadcast 可分级投递消息。
  • 多 Agent 需 MultiAgentFormatter：否则角色名混淆导致偏航。

主管-工人模式

CorpAssist 项目协调场景：Supervisor 理解用户意图，调度 CalendarWorker、MailWorker、TicketWorker。Supervisor 不直接操作外部系统，而是通过 MsgHub 广播任务描述，专家 Agent 在共享上下文中各自 reply。

CorpAssist · 项目启动会 用户：「订明天 10 点 A301，发邮件通知项目组，并建 IT 工单申请投影设备。」Supervisor 在 Hub 内依次调度 CalendarWorker → MailWorker → TicketWorker，每个 Worker 只看到自己领域工具，最终 Supervisor 汇总三份汇报给用户。

Supervisor 在 Hub 内依次 invoke 专家；专家 reply 自动被 Supervisor observe。

python 
import asyncio
import os
from agentscope.agent import ReActAgent
from agentscope.formatter import DashScopeMultiAgentFormatter
from agentscope.message import Msg
from agentscope.model import DashScopeChatModel
from agentscope.pipeline import MsgHub
from agentscope.tool import Toolkit

def make_worker(name: str, role: str, tools: list) -> ReActAgent:
    tk = Toolkit()
    for t in tools:
        tk.register_tool_function(t)
    return ReActAgent(
        name=name,
        sys_prompt=f"你是 {name}，专职{role}。只处理自己领域，完成后简要汇报。",
        model=DashScopeChatModel(model_name="qwen-max", api_key=os.environ["DASHSCOPE_API_KEY"]),
        formatter=DashScopeMultiAgentFormatter(),
        toolkit=tk,
        max_iters=5,
    )

async def run_supervisor_flow(user_text: str):
    supervisor = ReActAgent(
        name="Supervisor",
        sys_prompt="协调 Calendar/Mail 专家完成用户请求，最后汇总。",
        model=DashScopeChatModel(model_name="qwen-max", api_key=os.environ["DASHSCOPE_API_KEY"]),
        formatter=DashScopeMultiAgentFormatter(),
        max_iters=6,
    )
    calendar = make_worker("CalendarWorker", "日历", [create_calendar_event])
    mail = make_worker("MailWorker", "邮件", [send_email])

    async with MsgHub(
        participants=[supervisor, calendar, mail],
        announcement=Msg("user", user_text, "user"),
    ) as hub:
        await supervisor(Msg("system", "请调度专家完成上述请求", "system"))
        await calendar()
        await mail()
        final = await supervisor()
    return final.get_text_content()

先想 10 秒：Supervisor 为何不应注册全部业务工具？

Supervisor 持有全工具集会导致它绕过专家直接执行，破坏分工与审计边界。正确做法是 Supervisor 只协调，专家各自 Toolkit；必要时 Supervisor 仅保留 transfer_to_human。

显式消息传递

显式消息传递 指在 enable_auto_broadcast=False 时，由编排代码通过 hub.broadcast() 精确控制谁能看到什么。适用于部分信息需要对某 Agent 保密（如 HR 薪资）的场景。

模式配置适用
自动广播enable_auto_broadcast=True开放式讨论、头脑风暴
手动广播False + hub.broadcast()分级信息、合规隔离
动态参与者hub.add() / hub.delete()按需拉入专家
python 
async with MsgHub(participants=[supervisor], enable_auto_broadcast=False) as hub:
    hub.add(calendar_worker)
    await hub.broadcast(Msg("user", "只需 Calendar 处理：订明天 10 点会议室", "user"))
    await calendar_worker()
    # MailWorker 未加入，不会看到消息
广播误用
  • 薪资信息 auto_broadcast 给所有 Worker
  • 忘记 hub.add 就 await Worker
  • 手动 broadcast 后仍开 auto_broadcast
合规隔离
  • HR 消息只 broadcast 给 HRWorker
  • enable_auto_broadcast=False 精确控制
  • 动态 hub.add 按需拉专家

动态增删参与者

python 
async with MsgHub(participants=[supervisor], enable_auto_broadcast=True) as hub:
    await supervisor(Msg("user", "先查日程", "user"))
    await calendar_worker()
    # 用户追加需求 — 动态拉入 MailWorker
    hub.add(mail_worker)
    await hub.broadcast(Msg("user", "再加发邮件通知", "user"))
    await mail_worker()
    hub.delete(calendar_worker)  # 日历已完成，移出 Hub
    final = await supervisor()
hub.add 后加入的 Agent 为什么看不到历史？
MsgHub 只向当前参与者广播之后的消息；新 Agent 无 retroactive observe。若新专家需要上下文，Supervisor 应 broadcast 一条摘要 Msg 而非依赖自动历史。

角色分工与 Handoff

Handoff 将子 Agent 包装为工具，Orchestrator 自主决定调用哪个 Worker；Worker 仅收到任务描述，不共享完整对话。MsgHub 则共享完整上下文，适合需要交叉引用的协作。

MsgHub 不适用
  • 独立子任务可完全并行且无交叉引用
  • Worker 不应看到彼此中间结果
  • 需要动态创建/销毁 Worker
Handoff 更合适
  • Orchestrator 用 toolkit 注册 Worker 工具
  • 子任务隔离、结果汇总
  • 与 Agent-as-Tool 模式一致

Handoff 实现示意

python 
from agentscope.tool import Toolkit

async def calendar_worker_tool(task: str) -> ToolResponse:
    """委派日历专家处理任务。输入任务描述，返回执行摘要。"""
    worker = make_worker("CalendarWorker", "日历", [create_calendar_event])
    msg = Msg("user", task, "user")
    result = await worker(msg)
    return ToolResponse(content=result.get_text_content())

orchestrator = ReActAgent(
    name="Orchestrator",
    sys_prompt="用 calendar_worker_tool / mail_worker_tool 委派子任务。",
    toolkit=Toolkit(tools=[calendar_worker_tool, mail_worker_tool]),
    max_iters=6,
)
# Handoff：子 Agent 上下文隔离，Orchestrator 只见工具返回

与 Java Sub-agent 对照

AgentScope Java 2.0 的 Sub-agent 通过 SAA Starter 以嵌套 Agent 或 Tool 形式暴露；Python MsgHub 更偏「会议室广播」，Java 侧常见「Router + SubAgentTool」层级。双栈联调时，Python MsgHub 产出的事件流可映射为 Java streamEvents 的多参与者 trace。

FormatterDashScopeMultiAgentFormatterMultiAgentContentFormatter
可观测reply_stream 事件streamEvents SSE
CorpAssist 跨栈场景：Python 编排 + Java 工具服务，MsgHub 放哪一侧？

编排与 MsgHub 通常在 Python 侧（本课主栈）；Java 侧暴露 REST/MCP 工具供 Worker 调用。避免双端各建 MsgHub 导致消息环。

Python MsgHub + Java 工具服务
flowchart LR
    Hub[Python MsgHub] --> CW[CalendarWorker]
    CW --> MCP[MCP / REST]
    MCP --> Java[Spring 日历服务]
    Hub --> MW[MailWorker]
    MW --> MCP2[邮件 MCP]

MsgHub 编排留在 Python；Java 提供工具实现，不重复建 Hub。

MultiAgentFormatter 选错会怎样？
模型无法区分 Msg 来自哪个 Agent，可能把 CalendarWorker 的回复当成用户输入，或 Supervisor 误以为自己已执行工具。CorpAssist 多 Agent 场景必须使用 DashScopeMultiAgentFormatter。
MsgHub 内 Agent 调用顺序如何定？
MsgHub 不自动调度——编排代码显式 await agent() 顺序。Supervisor 先理解 → 各 Worker 执行 → Supervisor 汇总是常见三段式；顺序错误会导致 Worker 在缺少上下文时行动。
MsgHub 与 Pipeline sequential 如何组合？
Hub 内某一阶段可跑 sequential_pipeline：例如 Supervisor 广播后，三个 Worker 用 Pipeline 串行处理子 SOP。见 practice-03 组合模式。

本章结论

  • 主管-工人模式：MsgHub 广播 + Supervisor 调度专家，MultiAgentFormatter 必备。
  • 显式消息传递：关闭 auto_broadcast 时用 broadcast 精确路由。
  • 角色分工与 Handoff：任务隔离场景用 Handoff 优于 Hub 全量共享。
  • 与 Java Sub-agent 对照：Sub-agent 偏层级，MsgHub 偏对等广播。
