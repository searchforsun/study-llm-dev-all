已完成记忆：短期与长期控制

约 35 分钟 · 阶段：AgentScope 核心抽象 · 能力：控制对话历史、长期记忆与上下文裁剪

为什么要学本章

CorpAssist S3 用户可能连续多轮布置任务——Agent 必须记住「刚才说的报告模板」又不能撑爆 Token 预算。短期记忆 管理当前会话 Msg 列表；Agentic 长期记忆 让 Agent 主动写入/检索跨会话事实。本章衔接 上下文与记忆工程context-memory-engineering 的系统化方法。

学完你能
  • 配置 AgentScope Memory 管理多轮 UserMsg / AgentMsg
  • 使用 Agentic 记忆 API 持久化用户偏好与任务上下文
  • 启用上下文窗口自动裁剪 策略
  • 将 AgentScope 记忆层与 context-memory-engineering 课对照
本章先记住 3 件事
  • 短期 = 会话内 Msg 链：ReAct 每轮都会读取完整可见历史。
  • 长期 = Agent 主动存取：不是简单 append，而是检索后再注入。
  • 裁剪先于溢出：Token 超阈值时 summarization 或 sliding window。

对话历史管理

对话历史 在 AgentScope 中由 Memory 组件持有。默认 InMemoryMemory 适合 Demo；生产可换 Redis / 数据库后端，与 LangGraph checkpoint 的 messages 字段类似但 API 不同。

多轮对话与 Memory
sequenceDiagram
    participant U as User
    participant A as Agent
    participant Mem as Memory

    U->>A: UserMsg 轮次1
    A->>Mem: 写入 Msg
    A-->>U: AgentMsg 回复1
    U->>A: UserMsg 轮次2
    A->>Mem: 读取历史 + 新 Msg
    A-->>U: AgentMsg 回复2

每轮 reply 前 Memory 提供完整可见上下文。

Memory 初始化与多轮对话

python 
from agentscope.agent import Agent
from agentscope.memory import InMemoryMemory
from agentscope.message import UserMsg

memory = InMemoryMemory()
agent = Agent(
    name="S3-Assistant",
    model=model,
    toolkit=toolkit,
    memory=memory,
)

async def multi_turn():
    await agent.reply(UserMsg("user", "我是华东区销售张三"))
    await agent.reply(UserMsg("user", "刚才说的区域，帮我查 Q1 数据"))  # 依赖短期记忆
先想 10 秒：工具返回的大段 JSON 应进 Memory 吗？

应截断或摘要后再写入。原始 JSON 快速占满上下文，导致后续轮次挤掉用户关键指令——这是 CorpAssist 记忆溢出的常见原因。

Redis 持久化 Memory（生产示意）

python 
# 概念：session_id 绑定 Memory 后端
SESSION_ID = "corp-user-zhangsan-20260610"

    redis_url="redis://localhost:6379/0",

  • 多 Worker 部署时，Memory 必须外置，否则请求打到不同 Pod 会丢上下文。
  • 短期记忆 与 LangGraph checkpoint 可并存：前者供 AgentScope reply，后者供子图恢复。
  • Token 预算 与 Memory max_tokens 应联动配置，防止长对话撑爆上下文。

Agentic 长期记忆

Agentic 长期记忆 允许 Agent 在对话中调用 memory 工具写入「用户偏好：报告用表格」或检索「上次会议纪要模板」。AgentScope 2.0 提供长期记忆存储抽象，可对接向量库或 KV。

类型生命周期CorpAssist 示例
短期 Memory单 session当前任务的多轮指令
长期 Agentic跨 session用户默认邮件签名、部门代码
外部 RAG知识库版本公司制度 PDF（见 practice-05）

长期记忆读写工具

python 
from agentscope.memory import LongTermMemory

ltm = LongTermMemory(store="redis://localhost:6379/1")

agent = Agent(
    name="S3-Assistant",
    model=model,
    toolkit=toolkit,
    memory=InMemoryMemory(),
    long_term_memory=ltm,
    enable_memory_tools=True,  # Agent 可主动 store/recall
)

# Agent 在 ReAct 中可调用：
# memory_store(key="report_format", value="表格+趋势图")
# memory_recall(query="用户报告格式偏好")
反模式
  • 把所有历史邮件全文写入长期记忆
  • 长期记忆无 TTL，隐私数据永久保留
  • recall 不限制条数，一次注入 50 条
推荐
  • 只存结构化偏好与任务状态
  • 敏感字段加密 + 保留策略
  • recall top_k=3 再注入 prompt

上下文窗口自动裁剪

自动裁剪 在 Token 接近模型上限时触发：保留 system + 最近 N 轮 + 可选摘要中间段。AgentScope 2.0 Memory 支持 max_tokens 与 summarization 钩子。

反模式
  • 无上限 append，直到 API 400
  • 裁剪时丢掉 system_prompt
  • 摘要用同一模型但不缓存，成本翻倍
推荐
  • max_tokens=0.8 × 模型窗口
  • 优先裁最旧 tool 观察，保留 user 目标句
  • 摘要用小模型（NS3 成本）

裁剪策略配置

python 
from agentscope.memory import InMemoryMemory

memory = InMemoryMemory(
    max_tokens=120_000,          # qwen-plus 上下文约 128k
    keep_recent_turns=6,         # 至少保留最近 6 轮
    summarize_after_tokens=100_000,
    summarizer_model="qwen-turbo",  # 廉价摘要模型
)
裁剪决策流
flowchart TD
    Check[Token 计数] --> Over{超过阈值?}
    Over -->|否| OK[正常 reply]
    Over -->|是| Sum[摘要最旧段落]
    Sum --> Drop[丢弃已摘要 Msg]
    Drop --> OK

摘要与 sliding window 可组合使用。

工具观察截断示例

python 
MAX_TOOL_OBS_CHARS = 2000

def truncate_tool_result(raw: str) -> str:
    if len(raw) <= MAX_TOOL_OBS_CHARS:
        return raw
    return raw[:MAX_TOOL_OBS_CHARS] + f"\n...[truncated {len(raw)-MAX_TOOL_OBS_CHARS} chars]"

自动裁剪 触发时优先删除最旧的 tool 观察，保留最近 user 指令与 system_prompt。生产环境可配合 摘要钩子 在裁剪前将最旧段落压缩为一条 system 摘要 Msg，避免完全丢失早期上下文。

摘要钩子与 keep_recent_turns 如何配合？
当 Token 超过 summarize_after_tokens 时触发钩子：最旧 N 轮 Msg 送入 qwen-turbo 生成一条「历史摘要」Msg，再从 Memory 删除原文。keep_recent_turns 保证最近对话完整保留，摘要 Msg 插入 system 段之后。

与 context-memory-engineering 衔接

系统课 记忆工程 讲解 Working Memory、Semantic Memory 分层与评估指标。AgentScope 的 Memory / LongTermMemory 是该理论在 Python 栈的落地：短期对应 Working，长期 Agentic 对应 Semantic 写入通道。

概念（系统课）AgentScope 2.0实践建议
Working MemoryInMemoryMemory 当前 session配 max_tokens 裁剪
Semantic MemoryLongTermMemory + 向量检索用户偏好与事实
Episodic会话 log + OTel审计不注入 prompt
评估recall@k、任务成功率见 context-memory 课
CorpAssist · 跨天任务续接 用户周一说「用表格模板」，周三回来问「按我的模板导出」——长期记忆 recall 命中 key=report_format；若仅短期 Memory，周三新 session 会丢失偏好。
Memory 与 RAG 知识库如何分工？
RAG 存公司制度等公共知识；Agentic 长期记忆存用户级偏好与任务状态。CorpAssist 制度问答走 RAG Agent（practice-05），个人模板走 LongTermMemory。

与 context-memory-engineering 对照清单

  • 完成系统课「Working Memory 评估」实验后，在本课用相同 session 脚本对比 AgentScope Memory 行为。
  • 记忆工程 中的 recall@k 指标可用于长期记忆质量验收。
  • Episodic 日志进 OTel，不进入 prompt，避免审计数据污染推理。
记忆混用
  • 把 RAG chunk 写入 LongTermMemory
  • 短期 Memory 无 session 隔离
  • 裁剪后 UI 仍展示「已遗忘」的旧事实
分层清晰
  • 公共知识 → RAG 工具按需检索
  • 用户偏好 → LongTermMemory store/recall
  • 当前任务 → 短期 Memory + 裁剪

摘要钩子配置示意

python 
async def summarize_old_turns(msgs: list) -> str:
    """摘要钩子：将最旧 Msg 压缩为一段文本。"""
    text = "\n".join(m.content for m in msgs if hasattr(m, "content"))
    summary_model = DashScopeChatModel(model="qwen-turbo")
    reply = await summary_model.chat(
        messages=[{"role": "user", "content": f"用 3 句话摘要：\n{text[:8000]}"}]
    )
    return reply.content

memory = InMemoryMemory(
    max_tokens=120_000,
    summarize_after_tokens=100_000,
    on_summarize=summarize_old_turns,
)

本章结论

  • 对话历史管理：Memory 持有多轮 Msg，生产需持久化后端。
  • Agentic 长期记忆：Agent 主动 store/recall，跨 session 保留偏好与事实。
  • 上下文裁剪：max_tokens + 摘要防止溢出，优先保留用户目标与最近轮次。
  • 记忆工程衔接：Working/Semantic 分层与系统课概念一一对应。
