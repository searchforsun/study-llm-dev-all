已完成单 Agent：CorpAssist 办公助手

约 45 分钟 · 阶段：CorpAssist 工作流实战 · 能力：用 AgentScope 构建单 Agent 办公助手

为什么要学本章

CorpAssist S3 自动化场景从「单 Agent + 多工具」起步。AgentScope 2.0 的 ReActAgent 内置推理-行动循环，配合 意图识别与工具路由，可在一个 Agent 内完成日历、邮件、工单等办公操作。本章是后续 MsgHub 与 Pipeline 的基础。

学完你能
  • 为 CorpAssist 办公助手注册日历/邮件/工单三类工具
  • 用结构化输出或工具描述实现意图路由
  • 配置 步数与 Token 预算 防止失控循环
  • 在工具失败或超预算时执行人工转接降级
本章先记住 3 件事
  • 单 Agent 先跑通：多 Agent 编排之前，务必验证工具链与预算控制。
  • 工具描述即路由：清晰的 docstring 比额外分类器更可靠。
  • 降级要显式：人工转接 应是注册工具，而非异常兜底。

意图识别与工具路由

CorpAssist 办公助手需区分「查日程」「发邮件」「建工单」三类意图。AgentScope 推荐两种方式：结构化输出路由（主 Agent 先分类再调用子流程）与工具自路由（ReActAgent 直接选工具）。对 S3 场景，工具自路由足够且可观测性更好。

CorpAssist · S3 办公自动化 员工通过 CorpAssist 说：「帮我把明天 10 点会议室 A 加到日历，并邮件通知项目组。」单 Agent 需依次调用 create_calendar_event 与 send_email，并在参数缺失时追问而非臆造。
单 Agent 工具路由
flowchart TD
    User[用户请求] --> ReAct[ReActAgent 推理]
    ReAct -->|查日程| Cal[create_calendar_event]
    ReAct -->|发邮件| Mail[send_email]
    ReAct -->|建工单| Ticket[create_ticket]
    ReAct -->|无法处理| Human[transfer_to_human]
    Cal --> ReAct
    Mail --> ReAct
    Ticket --> ReAct
    Human --> End([结束])

ReActAgent 在推理环中根据工具描述选择下一步；转人工是与其他工具同级的显式工具。

python 
import asyncio
import os
from agentscope.agent import ReActAgent
from agentscope.formatter import DashScopeChatFormatter
from agentscope.memory import InMemoryMemory
from agentscope.message import Msg
from agentscope.model import DashScopeChatModel
from agentscope.tool import Toolkit, ToolResponse

async def create_calendar_event(title: str, start_time: str, attendees: str = "") -> ToolResponse:
    """创建日历事件。title 为标题，start_time 为 ISO8601 时间，attendees 为逗号分隔邮箱。"""
    event_id = f"EVT-{hash(title + start_time) % 10000:04d}"
    return ToolResponse(content=f"已创建日历 {event_id}：{title} @ {start_time}")

async def send_email(to: str, subject: str, body: str) -> ToolResponse:
    """发送邮件。to 为收件人，subject 为主题，body 为正文。"""
    return ToolResponse(content=f"邮件已发送至 {to}，主题：{subject}")

async def create_ticket(category: str, description: str) -> ToolResponse:
    """创建 IT 工单。category 为类别，description 为问题描述。"""
    ticket_id = f"TK-{hash(description) % 100000:05d}"
    return ToolResponse(content=f"工单 {ticket_id} 已创建（{category}）")

async def transfer_to_human(reason: str) -> ToolResponse:
    """转接人工。Agent 无法完成、用户明确要求或预算耗尽时调用。"""
    return ToolResponse(content=f"已转人工，原因：{reason}。预计 5 分钟内接入。")

def build_office_agent() -> ReActAgent:
    toolkit = Toolkit()
    for fn in (create_calendar_event, send_email, create_ticket, transfer_to_human):
        toolkit.register_tool_function(fn)
    return ReActAgent(
        name="CorpAssistOffice",
        sys_prompt="你是 CorpAssist 办公助手。优先用工具完成请求；缺参数时追问；无法处理时调用 transfer_to_human。",
        model=DashScopeChatModel(
            model_name="qwen-max",
            api_key=os.environ["DASHSCOPE_API_KEY"],
        ),
        formatter=DashScopeChatFormatter(),
        memory=InMemoryMemory(),
        toolkit=toolkit,
        max_iters=8,
    )

async def main():
    agent = build_office_agent()
    msg = await agent(Msg("user", "帮我把周五产品评审加到日历并邮件通知 team@corp.com", "user"))
    print(msg.get_text_content())

asyncio.run(main())
先想 10 秒：为什么不单独写一个「意图分类 LLM」再调 Agent？

多一跳 LLM 增加延迟与成本，且分类错误会级联。ReActAgent 在单循环内完成意图识别与工具调用，事件流可观测每一步，更适合生产排障。

工具 docstring 写多长合适？
一句话说明职责 + 每个参数一行说明 + 一条「不要用于…」反例。CorpAssist 四类工具 docstring 各 3～5 行即可；过长会挤占 tool schema Token。

日历/邮件/工单工具

三类工具应遵循 CorpAssist 统一契约：输入 schema 明确、返回结构化文本、失败时返回可读的 ToolResponse 而非抛异常。工单工具对接内部 ITSM，邮件工具需校验收件人域名。

工具典型场景必填参数权限
create_calendar_event会议预约、提醒title, start_time员工日历写
send_email通知、摘要to, subject, body企业邮箱发
create_ticketIT 报障、权限申请category, description工单系统写
transfer_to_human复杂审批、情绪投诉reason无外部副作用
不推荐
  • 工具 docstring 为空或「处理请求」
  • 一个 mega-tool 包打天下
  • 失败抛异常中断 ReAct 循环
推荐
  • 每工具单一职责 + 清晰参数说明
  • 失败返回 ToolResponse 含错误码
  • 敏感操作预留 HITL（见 Human-in-the-loop 与 Realtime Steering）

工单工具 Mock 示例

async def create_ticket(category: str, description: str) -> ToolResponse:
    """创建 IT 工单。category 取 hardware/software/access；description 为问题描述。"""
    if category not in ("hardware", "software", "access"):
        return ToolResponse(content="错误：category 无效，请从 hardware/software/access 选择")
    ticket_id = f"TK-{hash(description) % 100000:05d}"
    return ToolResponse(content=json.dumps({"ticket_id": ticket_id, "status": "open"}))
CorpAssist · 权限申请 员工：「帮我申请 VPN 权限。」Agent 应调用 create_ticket(category="access", ...) 而非臆造审批结果；若描述含糊，追问具体系统名后再建单。

人工转接降级

人工转接降级 是显式工具而非 try/except 兜底。触发条件：用户明确要求、连续工具失败、步数/Token 预算耗尽、或模型主动判断无法完成。降级后 Agent 应停止继续调用业务工具，仅输出转接摘要。

python 
from agentscope.agent import ReActAgent

def wrap_with_budget_guard(agent: ReActAgent, max_iters: int = 8) -> ReActAgent:
    """在 sys_prompt 中注入预算与降级策略（生产可用 middleware 替代）。"""
    agent.sys_prompt += (
        f"\n\n安全规则：最多 {max_iters} 步推理。"
        "若即将超限或连续 2 次工具失败，必须调用 transfer_to_human 并说明原因。"
    )
    agent.max_iters = max_iters
    return agent

# 连续失败检测示例（middleware 伪代码）
async def post_acting_hook(agent, kwargs, output):
    recent = agent.memory.get_memory()[-6:]
    tool_errors = sum(1 for m in recent if "失败" in str(m.content))
    if tool_errors >= 2:
        agent.sys_prompt += "\n[系统] 已连续失败，请转人工。"
    return output
降级触发决策
flowchart TD
    Start[ReAct 循环] --> Fail{连续工具失败?}
    Fail -->|是| Human[transfer_to_human]
    Fail -->|否| Budget{步数将超限?}
    Budget -->|是| Human
    Budget -->|否| Continue[继续推理]

降级是显式工具调用，事件流可审计触发原因。

步数与 Token 预算

Token 预算 与 max_iters 联合约束 ReAct 循环。AgentScope 2.0 的 context_config 可自动压缩历史；react_config.max_iters 限制推理轮次。CorpAssist 办公场景建议 max_iters=8，单次对话 Token 上限 8k（含工具结果）。

参数推荐值作用
max_iters8–12ReAct 最大推理-行动轮次
context_config.max_tokens8192触发上下文压缩阈值
model_config.max_retries2模型调用失败重试
单次工具超时15s防止外部 API 拖死循环
python 
from agentscope.agent import ReActAgent
from agentscope.model import DashScopeChatModel

agent = ReActAgent(
    name="CorpAssistOffice",
    sys_prompt="...",
    model=DashScopeChatModel(model_name="qwen-max", api_key="..."),
    formatter=DashScopeChatFormatter(),
    memory=InMemoryMemory(),
    toolkit=toolkit,
    max_iters=8,  # 步数硬上限
)

# 观测 Token：通过 reply_stream 的 EventType.MODEL_CALL_END 读取 usage
async for evt in agent.reply_stream(Msg("user", "查下周会议并发邮件", "user")):
    if hasattr(evt, "usage"):
        print(f"tokens: {evt.usage}")
预算失控
  • max_iters=50「保证能跑完」
  • 不压缩 tool 结果导致 Token 膨胀
  • 超预算后仍继续调业务工具
三重约束
  • max_iters=8 + sys_prompt 约定
  • context_config.max_tokens 压缩
  • 超限前 transfer_to_human

事件流观测与排障

单 Agent 上线前须验证 reply_stream 事件序列：推理 → tool_call → tool_result → 最终回复。CorpAssist 办公场景常见 badcase 是模型跳过工具直接编造日历 ID——通过事件流可立即发现「无 tool_call 却有结论」。

python 
from agentscope.event import EventType

async def debug_office_agent(agent, user_text: str):
    tools_called = []
    async for evt in agent.reply_stream(Msg("user", user_text, "user")):
        if evt.type == EventType.TOOL_CALL:
            tools_called.append(evt.tool_name)
            print(f"[tool_call] {evt.tool_name} args={evt.tool_args}")
        elif evt.type == EventType.REPLY_END:
            print(f"[done] tools={tools_called} iters={evt.iter_count}")
    assert tools_called, "未调用任何工具却给出结论 — 检查 sys_prompt"
单 Agent 何时升级到 Message Hub？
当任务需多角色专业分工（日历专员 vs 邮件专员各自独立工具集）、或 Supervisor 需汇总多专家中间结果时，升级到 practice-02 MsgHub。单 Agent 工具数 ≤6 且步骤 ≤8 时不必过早拆分。
Mock 工具与生产 API 如何切换？
工具函数内部读环境变量 CORPASSIST_USE_MOCK=1 分支；Agent / Toolkit 注册不变。golden set 用 Mock 离线跑；staging 接真实 API 做集成测试。

常见问题（FAQ）

ReActAgent 与 LangGraph StateGraph 在本场景如何选型？
单 Agent + 固定工具集优先 ReActAgent；需要可视化图、中断恢复或多分支编排时再上 LangGraph。本课后续 Pipeline/MsgHub 覆盖 AgentScope 原生多 Agent 路径。
工具路由错误如何排查？
打开 reply_stream 观察 tool_call 事件，核对 docstring 与参数 schema；常见原因是工具描述过于相似或缺少反例说明。

本章结论

  • 意图识别与工具路由：清晰 docstring 引导模型选工具，优于额外分类 LLM。
  • 日历/邮件/工单工具：四类工具覆盖 CorpAssist 办公主路径。
  • 人工转接降级：transfer_to_human 显式工具，与预算耗尽联动。
  • 步数与 Token 预算：max_iters + 上下文压缩 + 工具超时三重约束。
