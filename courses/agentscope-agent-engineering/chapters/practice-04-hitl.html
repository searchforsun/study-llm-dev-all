Realtime Steering

Realtime Steering 允许用户在 Agent 执行过程中随时干预：调用 agent.interrupt() 取消当前任务，Agent 运行 handle_interrupt() 做清理，之后可注入新指令继续。适用于用户发现 Agent 偏航、或紧急变更优先级。

CorpAssist · 紧急改向 用户发起「批量发邮件通知全员」，Agent 正在生成收件人列表时用户点击「停止」并改口「只发项目组」——interrupt() 终止当前 Acting，清理半成品状态，注入新 UserMsg 重新 reply。
HITL 与 Steering 事件流
sequenceDiagram
    participant U as 用户
    participant A as ReActAgent
    participant T as 工具
    U->>A: 用户请求
    A->>A: 推理
    A-->>U: RequireUserConfirmEvent
    U->>A: 确认/改参/拒绝
    A->>T: 执行工具
    U->>A: interrupt() 可选
    A-->>U: 最终回复

敏感工具先 ASK；用户可在 Acting 前 interrupt 改向。

python 
import asyncio
from agentscope.agent import ReActAgent
from agentscope.event import EventType

async def run_with_steering(agent: ReActAgent, user_msg):
    task = asyncio.create_task(agent.reply(user_msg))

    async def watch_ui():
        await asyncio.sleep(2)
        if user_clicked_stop():
            agent.interrupt()
            print("[Steering] 用户中断，Agent 清理中…")

    watcher = asyncio.create_task(watch_ui())
    try:
        result = await task
        return result
    finally:
        watcher.cancel()

async for evt in agent.reply_stream(user_msg):
    if evt.type == EventType.REQUIRE_USER_CONFIRM:
        print(f"待审批工具：{evt.tool_name} args={evt.tool_args}")
干预方式API适用场景
中途改向interrupt()用户取消或变更意图
工具前审批Permission ASK敏感写操作
改参执行confirm.approve(modified_args)收件人/金额修正
拒绝继续confirm.deny()合规拦截

审批与改参

工具权限配置为 ASK 时，Agent 发出 RequireUserConfirmEvent 并暂停。前端展示参数表单；用户可选 ALLOW（原参执行）、MODIFY（改参后执行）或 DENY（拒绝并告知 Agent）。CorpAssist 发邮件给外部域名、批量删数据等须 ASK。

决裁行为CorpAssist 示例
ALLOW按原 args 执行工具确认发送会议纪要邮件
MODIFY替换 args 再执行修改收件人列表后发送
DENY返回错误结果给 LLM拒绝向个人邮箱发内部文档
python 
from agentscope.tool import Toolkit
from agentscope.permission import PermissionRule, PermissionDecision

toolkit = Toolkit()
toolkit.register_tool_function(
    send_email,
    permission=PermissionRule(decision=PermissionDecision.ASK),
)

# 用户确认后恢复 Agent（伪代码）
async def resume_after_confirm(agent, confirm_event, user_decision, new_args=None):
    if user_decision == "ALLOW":
        result_event = confirm_event.approve()
    elif user_decision == "MODIFY":
        result_event = confirm_event.approve(modified_args=new_args)
    else:
        result_event = confirm_event.deny(reason="用户拒绝")
    final = await agent.reply(result_event)
    return final
审批反模式
  • 所有工具都 ASK，体验极差
  • 前端超时自动 ALLOW
  • 改参后不更新 audit modified_args
分级 ASK
  • 只读工具 ALLOW 默认
  • 外发/删改/权限 ASK
  • 超时 DENY 而非自动执行
CorpAssist 哪些工具应默认 ASK？
① 向外部域名发邮件；② 批量操作（>10 收件人）；③ 权限/账号变更类工具。只读查询（查日程、查制度）默认 ALLOW。清单写入 PermissionRule 配置而非 scattered sys_prompt。

审计日志

审计日志 须记录：session_id、agent_name、tool_name、original_args、modified_args、decision、operator_id、timestamp。CorpAssist 合规要求保留 180 天，且与 OTel trace_id 关联便于排障。

python 
import json
from datetime import datetime, timezone

def audit_hitl(record: dict) -> None:
    entry = {
        "ts": datetime.now(timezone.utc).isoformat(),
        "session_id": record["session_id"],
        "tool": record["tool_name"],
        "original_args": record["original_args"],
        "modified_args": record.get("modified_args"),
        "decision": record["decision"],
        "operator": record["operator_id"],
        "trace_id": record.get("trace_id"),
    }
    # 生产：写入 append-only 存储 / SIEM
    print(json.dumps(entry, ensure_ascii=False))

# 在 confirm 回调中调用
audit_hitl({
    "session_id": "sess-001",
    "tool_name": "send_email",
    "original_args": {"to": "partner@external.com"},
    "modified_args": {"to": "partner@approved-domain.com"},
    "decision": "MODIFY",
    "operator_id": "user-zhang",
})
字段必填说明
session_id是关联用户会话
tool_name是待审批工具名
original_args是模型提议参数 JSON
modified_argsMODIFY 时用户改后参数
decision是ALLOW / MODIFY / DENY
trace_id推荐OTel 关联排障
CorpAssist · 合规审计 安全团队抽查：「上周是否有 Agent 向个人邮箱发内部文档？」审计员查 audit.jsonl 中 decision=DENY 与 MODIFY 记录，按 operator_id 与 tool_name 过滤——DENY 记录不可省略。

与 SAA 审批节点对照

Spring AI Alibaba Graph 的 interrupt_before / 人工节点与 AgentScope HITL 概念对齐：Graph 在节点前持久化 checkpoint；AgentScope 在 tool 权限 ASK 时 emit 事件暂停。双栈 CorpAssist 可约定：Java 侧跑长事务审批流，Python AgentScope 侧跑轻量 ASK，审计格式统一 JSON Schema。

AgentScope 侧避免
  • ASK 事件不落库
  • 改参后不写 modified_args
  • interrupt 后不 handle_interrupt
SAA 侧对齐
  • Graph interrupt 对应 ASK 暂停点
  • 统一 audit schema 跨栈
  • SSE 推送待审批到同一前端
外部执行工具（is_external_tool）与 ASK 有何区别？

ASK 是权限确认，工具仍在 Agent 进程内执行；external 工具逻辑完全在外部系统运行，Agent 等外部回传结果事件。CorpAssist 大额转账宜用 external + 后端审批。

AgentScope ASK 与 LangGraph interrupt_before 有何异同？
同：都在敏感步骤前暂停等人输入。异：LangGraph interrupt 绑 checkpoint 节点；AgentScope ASK 绑 tool Permission，粒度更细到单次 tool_call。CorpAssist 双栈可统一前端审批 UX，后端分别 resume。
interrupt 后 Agent 状态如何清理？
Agent 运行 handle_interrupt() 取消进行中的 model/tool 任务；Memory 保留已确认 Msg，未完成的 tool_call 标记 cancelled。用户新指令作为新 UserMsg reply，而非从中间 tool 继续。
CorpAssist 场景Graph interruptAgentScope ASK
长事务审批（采购流程）interrupt_before 节点external tool + 后端
单次发邮件确认过重Permission ASK ✓
用户中途改需求checkpoint 回滚interrupt() ✓

本章结论

  • Realtime Steering：interrupt() 中途改向，须配合 handle_interrupt。
  • 审批与改参：Permission ASK → 事件暂停 → ALLOW/MODIFY/DENY；敏感工具默认 ASK。
  • 审计日志：原参/改参/决裁/操作者全量记录，DENY 亦须落库。
  • 与 SAA 审批节点对照：Graph interrupt 与 AgentScope ASK 可双栈统一 UX 与 audit schema。
