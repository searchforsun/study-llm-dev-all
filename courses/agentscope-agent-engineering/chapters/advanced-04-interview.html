死循环与偏航

AgentScope ReActAgent 在 max_iters 内反复调用工具或输出无效推理，常见于工具返回信息不足、sys_prompt 缺少终止条件、或两个工具职责重叠。

AgentScope 死循环排查流程
flowchart TD
    Alert[max_iters 触发 / 用户超时] --> Trace[导出 AgentScope Trace]
    Trace --> Pattern{调用模式}
    Pattern -->|同工具同参数| P1[工具返回无效/模型未解析]
    Pattern -->|工具 AB 交替| P2[职责重叠或权限边界]
    Pattern -->|无 tool_call 纯文本| P3[sys_prompt 缺终止指令]
    Pattern -->|参数微调重试| P4[Toolkit schema 过严]
    P1 --> Fix1[充实 tool_result + description]
    P2 --> Fix2[合并工具或加前置校验]
    P3 --> Fix3[增加「信息足够则直接回答」]
    P4 --> Fix4[放宽校验或给示例参数]

从 Trace 中按 tool_call 模式分类，AgentScope Event 日志含 step 序号便于识别。

# loop_detector.py — AgentScope 死循环检测 Hook（AOP）
from agentscope.pipelines import functional_user_agent

def detect_tool_loop(tool_trace: list[tuple[str, dict]], window: int = 3) -> bool:
    """最近 window 次工具调用完全相同则判定死循环"""
    if len(tool_trace) < window:
        return False
    recent = tool_trace[-window:]
    return len(set(recent)) == 1

class LoopGuardHook:
    def post_tool_call(self, agent, tool_name: str, args: dict, result: str):
        trace = agent.get_tool_trace()
        if detect_tool_loop(trace):
            raise RuntimeError(
                f"LOOP_DETECTED: {tool_name} called {len(trace)} times with same args"
            )

工具假执行

工具假执行：模型在 text 或 thinking 中说「已为您预订 A301」，但 Trace 中无对应 tool_call 事件，或 tool_call 被 Mock 空实现。这在 Demo 切生产时极高频。

假执行征象
  • 回复含「确认号 #12345」但工具日志无 book_meeting 记录
  • tool_call 有，Toolkit 函数体是 pass 或固定 return "ok"
  • 模型在 thinking 中「模拟」工具结果而未触发 call
防护
  • 结构化输出必须引用 tool_result 中的 confirmation_id
  • 工具层写审计日志，BFF 校验 side effect
  • golden set 断言 must_call_tools

# 假执行排查 Checklist — AgentScope 专用
FAKE_EXECUTION_CHECKLIST = [
    "1. Trace 中是否有 tool_call 事件？若无 → 模型未走 function call",
    "2. tool_call 的 name 是否与用户感知操作一致？",
    "3. Toolkit 函数是否真正调用 HTTP/DB？（非 return 硬编码）",
    "4. sys_prompt 是否鼓励「假设已执行」？（禁止）",
    "5. Mock 工具是否在非 Mock 环境被误加载？",
    "6. DashScope 模型是否开启 tools 参数？",
]

def verify_side_effect(tool_name: str, trace: list) -> dict:
    calls = [e for e in trace if e.get("event_type") == "tool_call"]
    matching = [c for c in calls if c["payload"]["tool_name"] == tool_name]
    return {
        "claimed_in_text": True,  # 由 NLP 或规则检测
        "actually_called": len(matching) > 0,
        "call_count": len(matching),
        "verdict": "FAKE" if not matching else "OK",
    }
面试题：用户说 Agent「撒谎」已发邮件，你如何排查？

STAR 答法：S——生产用户投诉未收到邮件；T——确认是否真调用 send_email；A——查 AgentScope Trace 无 tool_call，根因为 sys_prompt 示例含「已发送」诱导幻觉，且 Mock 工具在生产未替换；R——改 Prompt 禁止未调用断言 + 工具审计 + golden 断言 must_call_tools，投诉率降 80%。

记忆溢出

AgentScope 默认将 Msg 历史 追加进每次 LLM 调用。长会话下历史占满上下文，sys_prompt 被挤到窗口尾部，Agent 开始「失忆」——不遵守工具白名单、重复询问已确认信息。

症状Trace 信号修复
忘记 System 规则prompt token 中 sys 占比 < 5%每轮重写 sys_prompt 到消息列表首部
重复问用户历史 > 20 轮未裁剪滑动窗口 10 轮 + 摘要 Msg
Token 费用暴涨单次调用 input > 8kAgentic 记忆：只保留 task_state JSON
Redis session 过大session 键 > 1MBTTL + 归档 + 压缩 tool_result

# memory_budget.py — Msg 历史裁剪
from agentscope.message import Msg

def trim_messages(msgs: list[Msg], max_rounds: int = 10) -> list[Msg]:
    """保留 system + 最近 max_rounds 轮 user/assistant"""
    system = [m for m in msgs if m.role == "system"]
    dialogue = [m for m in msgs if m.role != "system"]
    keep = dialogue[-(max_rounds * 2):]
    return system + keep

def compress_tool_results(msgs: list[Msg], max_chars: int = 500) -> list[Msg]:
    out = []
    for m in msgs:
        if m.role == "tool" and len(m.content) > max_chars:
            out.append(Msg(m.name, m.content[:max_chars] + "…[truncated]", m.role))
        else:
            out.append(m)
    return out

事件流断连

SSE 断连 表现为前端卡住、Spring BFF 报 ReadTimeout，但 Agent Service 仍在执行。根因包括：Nginx 缓冲 SSE、Pod 重启、客户端未处理 event: done、或 AgentScope Trace 阻塞主线程。

SSE 断连排查序列
sequenceDiagram
    participant UI as 前端
    participant BFF as Spring BFF
    participant AS as agent-service
    participant Agent as ReActAgent
    UI->>BFF: SSE 订阅
    BFF->>AS: /v1/chat/stream
    AS->>Agent: run_stream
    Note over AS,Agent: 工具 HTTP 30s 阻塞
    BFF--xUI: 60s ReadTimeout 断连
    Agent->>Agent: 继续执行无 cancellation
    Note over UI: 用户以为失败，实际已订会议室

断连后 Agent 未取消是「重复订会议室」类 bug 的常见根因。

python 
# sse_heartbeat.py — 防止代理 idle 断连
import asyncio

async def stream_with_heartbeat(agent, messages, ping_interval=15):
    queue = asyncio.Queue()
    agent_task = asyncio.create_task(_run_agent(agent, messages, queue))

    try:
        while True:
            try:
                event = await asyncio.wait_for(queue.get(), timeout=ping_interval)
                if event is None:
                    break
                yield event
            except asyncio.TimeoutError:
                yield ": ping\n\n"  # SSE comment 保活
    finally:
        if not agent_task.done():
            agent_task.cancel()

生产排障案例

案例 1：CorpAssist S3 死循环 — search_policy × 8

现象：用户问「年假几天」，Agent 8 次调用 search_policy 后 max_iters 终止。

Trace：每次 tool_result 仅返回 {"title": "年假制度"} 无正文。

修复：工具返回 summary 字段 + sys_prompt 增加「单次搜索足够则回答」。

案例 2：假执行 — 「已订 A301」但未调用 book_meeting

现象：用户到会议室发现未预订，Trace 无 tool_call。

根因：qwen 某版本 function call 偶发失败，模型在 text 中幻觉执行结果。

修复：后置校验——回复含「预订成功」时必须存在 book_meeting tool_call + confirmation_id。

案例 3：SSE 断连 + 重复订会议室

现象：前端超时后用户重试，产生两个预订。

修复：BFF 幂等键 Idempotency-Key + Agent cancel on disconnect + book 工具幂等。

常见问题（FAQ）

AgentScope Trace 和 LangSmith 能否同时使用？
可以。AgentScope 2.0 支持 OpenTelemetry 导出；LangSmith 可作为统一 Trace 后端。面试中强调：排障习惯应统一到一种 Trace UI，避免双系统查不到关联。
max_iters 设多少合适？
CorpAssist 短链路 5～8；S3 复杂任务 10～12。超过 12 应考虑 Harness 任务分解，而非无限提高 max_iters。
面试如何体现 AgentScope 深度而非泛泛 Agent 经验？
举具体 API：Msg 结构、ReActAgent max_iters、Event type 枚举、Toolkit 注册、AOP Hook 做 LoopGuard。对比 LangGraph checkpoint 说明迁移踩坑。

本章结论

  • 死循环与偏航：按 Trace 模式分类——同工具/AB 切换/无工具/偏航。
  • 工具假执行：Trace 无 tool_call 却声称完成 → Prompt + 审计 + golden。
  • 记忆溢出：Msg 裁剪 + system_prompt 前置 + tool_result 压缩。
  • 事件流断连：heartbeat + cancel + 幂等 + 工具 timeout。

判断练习

  1. max_iters 触发率从 5% 升到 40%，团队建议「把 max_iters 从 8 调到 20」——是否采纳？

    参考答案

    不采纳。 触发率飙升说明工具返回、Prompt 或模型行为退化。应查 Trace 找死循环/偏航根因；单纯提高上限会放大 Token 成本与假执行风险。

  2. 排障顺序：A) 查 Redis session 大小 B) 看 AgentScope Trace tool_call 模式 C) 猜模型版本——正确顺序？

    参考答案

    B → A → 其他。 先看 Trace 定位故障类型（死循环/假执行/记忆/SSE），再针对性查 session 或网络。避免无 Trace 猜测。
