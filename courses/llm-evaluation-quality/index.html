约 35 分钟 · 阶段：评测体系设计 · 能力：定义 RAG/Agent 核心指标并与 SLA 对齐

为什么要学本章

CorpAssist 企业知识库 RAGS1 上线前，业务会问「比人工客服准多少、够快吗」——若只有「感觉不错」，无法发版、无法 A/B、无法在面试中答辩。本章建立双栈共用指标口径，是 NS1 评测飞轮 的计量基础。

学完你能
  • 为 S1 制度问答列出 faithfulness、context recall、answer relevance 的可接受阈值
  • 区分「检索失败」与「生成胡编」在指标上的不同表现
  • 设计 Python 与 Spring 侧共用的 JSON 指标 schema，避免各报各的分
  • 将 P95 延迟与可用性 SLA 纳入发版门禁，而不只看离线准确率
本章先记住 4 件事
  • 离线 vs 在线：RAGAS 等指标用于离线回归；线上还要看 thumbs、转人工率、P95。
  • 忠实 ≠ 相关：faithfulness 看是否胡编；answer relevance 看是否答非所问。
  • 拒答可评测：「无法从制度找到依据」是合法输出，应计入 hit rate 与拒答率。
  • 双栈同 schema：dual stack metrics 用同一 JSON 字段名，便于 CI 对比 Python 索引链与 Spring 查询链。

RAG 指标定义

CorpAssist · S1 制度问答评审 法务上传《远程办公管理办法》后，产品问：「模型说审批 3 天，制度里写的是 5 个工作日——这算检索错还是生成错？」本章指标帮你定位链路环节，而不是笼统说「模型不行」。

RAG 评测通常从「检索是否找对」「生成是否忠实」「答案是否切题」三个维度展开。CorpAssist 采用 RAGAS 兼容命名，便于与后续 RAGAS 与自动评测 流水线对接。

核心 RAG 指标对照

指标衡量什么低分常见原因CorpAssist 阈值示意
context recall 检索片段是否覆盖 gold 所需信息 分块切坏、Embedding 模型不匹配、TopK 过小 ≥ 0.85（回归集）
faithfulness 回答是否仅依据检索上下文、无胡编 Prompt 未约束「仅依据上下文」、模型过度发挥 ≥ 0.90
answer relevance 回答是否解决用户问题 检索偏题、生成跑题、多轮上下文干扰 ≥ 0.88
引用覆盖率 制度问答是否带 chunk 引用 未强制 citation 格式、检索为空仍生成 S1 必达 100% 有引用或明确拒答

反例 vs 推荐：指标选型

不推荐
  • 只报 BLEU/ROUGE 与 gold 文本字面重合
  • 用「用户满意度」替代可回归指标
  • 检索与生成混在一个「准确率」里
  • 拒答样本从评测集删除
推荐
  • RAGAS 三维 + 引用率 + 拒答正确率
  • 难例库单独统计，不与简单题混平均
  • 检索失败与生成失败分开归因（见 坏例归因方法论）
  • 拒答作为一等公民写入 gold 标签

指标：准确、召回、忠实、延迟 · RAG 评测流
flowchart LR
  DS[评测集 JSONL] --> RUN[双栈执行 RAG]
  RUN --> RET[检索 trace]
  RUN --> GEN[生成 answer]
  RET --> MET[RAGAS / 自定义 metric]
  GEN --> MET
  MET --> GATE[阈值门禁]
  GATE -->|通过| REL[发布 / A/B]
  GATE -->|失败| FIX[坏例回流]

离线流水线：同一评测集分别跑 Python 与 Spring 链，对比 aggregate 字段差异应在 ±0.02 以内（同模型同索引版本）。

先想 10 秒：faithfulness 很高但 answer relevance 很低，最可能哪一环出问题？

检索到的片段「忠实」地被复述，但与用户问题无关——优先查检索/query 改写与多轮上下文污染，而非换更大模型。

Agent 成功率与 SLA

除 RAG 外，CorpAssist 智能客服S2 与 Agent 任务自动化S3 需要 agent success：任务是否在步数预算内完成、工具调用是否成功、最终状态是否符合业务规则。

  • 工具调用成功率：schema 合法且 HTTP 2xx；失败要区分「模型选错工具」与「下游 API 超时」。
  • 任务完成率：用户意图「改收货地址」是否在规则校验后真正写入订单系统。
  • 步数预算：超过 N 步强制转人工或拒答，避免 Agent 空转烧 Token。

Agent vs RAG 指标侧重

场景首要指标次要指标
S1 制度 RAG faithfulness、citation_rate context_recall、P95 延迟
S2 智能客服 任务完成率、转人工率 answer_relevance、工具成功率
S3 办公 Agent agent success、步数分布 faithfulness（计划是否可执行）
Agent 场景还要跑 RAGAS 吗？

若 Agent 内部含 RAG 子链（查制度再答），应对该子链单独跑 context recall / faithfulness；整体任务用 agent success + 端到端 rubrics。不要用一个平均分掩盖子链路失败。

SLA

质量指标必须与 latency sla 和可用性一起门禁。ToB 验收常见：P95 < 3s、可用性 99.9%、离线 faithfulness 不低于 baseline - 0.02。

延迟分位与用户体验

分位CorpAssist S1 目标超标时的工程动作
p95 latency ≤ 2500 ms（含检索+生成） Rerank 降级、缓存热问、小模型路由
P99 ≤ 4500 ms 排查慢检索、Embedding 冷启动、长上下文
可用性 99.9% / 月 模型 Fallback、索引只读副本、熔断

    faithfulness_delta_max: -0.02

双栈同口径

Python 侧常负责索引与 RAGAS 批跑；Spring 侧常负责 BFF 与集成测试。若 Python 报 0.92、Spring 报 0.78，多半是索引版本、Prompt 版本或检索参数不一致，而非「Java 比较笨」。

  • 维护评测集 JSONL 与 dataset_version
  • 批跑 RAGAS，产出 json artifact 上传 CI
  • 索引 pipeline 版本写入报告 metadata
  • 集成测试读取同一 JSONL 子集
  • Advisor 链路与生产一致，禁止 mock 检索结果做「假绿」
  • 报告字段名与 Python schema 逐 key 对齐
常见误区
  • Spring 测试用内存 H2 向量库、Python 用 Milvus — 分数不可比。
  • 只对比平均分，不看同一题的双栈 diff 列表。
  • 延迟只测 LLM，不含检索与 Rerank。

本章结论

指标是 NS1 飞轮的「刻度尺」。发版前必须同时看质量分与 P95，且 Python/Spring 共用 schema。

  • RAG 指标定义：context recall、faithfulness、answer relevance 分工明确；拒答与引用率单独统计。
  • Agent 成功率：S2/S3 看任务完成与工具成功，RAG 子链仍要拆出 RAGAS 指标。
  • SLA：P95/P99 与可用性写入 yaml 门禁，不与离线分混为一谈。
  • 双栈同口径：同一 dataset_version、同一索引与 Prompt pin，字段级 diff 排查 parity。
