DeepEval 简介

deepeval（Confident AI）是面向 LLM 应用的 Python 评测框架：把「输入 / 实际输出 / 检索上下文」组织为 Test Case，再对每条用例调用一个或多个 auto metric 打分，最后像单元测试一样 assert 通过与否。

CorpAssist S1 · 制度引用合规（30 秒） 产品要求：凡给出天数/金额，答案必须带 [pol-YYYY-NN] 引用。RAGAS 的 faithfulness 能抓「是否贴上下文」，但抓不到引用格式。DeepEval 允许你叠加一条 deterministic 规则 metric，与 LLM-as-judge 并存。
维度RAGAS（本课主链）DeepEval
核心抽象 Dataset + eval pipeline + RAG 指标 TestCase + Metric + assert
CI 体验 pytest 聚合 + JSON 报告（Python 服务评测流水线） deepeval test run / pytest 插件，失败即 non-zero exit
内置 RAG 指标 faithfulness、context recall 等一等公民 Answer Relevancy、Faithfulness 等 + llm evaluator
Spring 侧 Python 出 JSON，Spring 对齐 schema 通常 Python 专用；Spring 仍走集成测试（Spring AI 服务评测）

DeepEval 最小用例（概念）

python 
from deepeval import assert_test
from deepeval.test_case import LLMTestCase
from deepeval.metrics import FaithfulnessMetric

def test_corpassist_faithfulness():
    case = LLMTestCase(
        input="年假可休几天？",
        actual_output="依据 [pol-2024-03]，带薪年假 10 天。",
        retrieval_context=["pol-2024-03: 带薪年假 10 天…"],
    )
    metric = FaithfulnessMetric(threshold=0.85)
    assert_test(case, [metric])
不推荐
  • 因「听说 DeepEval 新」就替换已跑通的 RAGAS CI
  • DeepEval 与 RAGAS 各报一套字段名，看板无法合并
  • 把所有指标都交给 LLM judge，忽略可 deterministic 的规则
推荐
  • RAGAS 继续负责 RAG 三角；DeepEval 补 assert 风格与 Confident AI 平台（可选）
  • 输出映射到 metrics-schema.json 同一键名
  • 规则类 metric 零 API 成本、可重复
RAGAS 主链 + DeepEval 扩展（CorpAssist）
flowchart TB
  JSONL[eval JSONL] --> PY[Python CI]
  PY --> RAGAS[RAGAS 三角]
  PY --> DE[DeepEval 用例]
  RAGAS --> MAP[映射 schema]
  DE --> MAP
  MAP --> GATE[threshold gate]

两框架可并行；门禁只认 schema 字段，不认框架品牌。

先想 10 秒：DeepEval 能否替代 Spring 集成测试？

不能。DeepEval 是 Python 测试栈；Spring 侧仍需 spring integration test + advisor assert。双栈对齐靠同一 JSONL 与 schema，而非同一框架。

自定义 metric

CorpAssist 大量质量要求可写成规则：引用前缀、拒答模板、禁词、表格字段完整度。这类逻辑适合 custom metric：输入 TestCase，输出 0～1 或 pass/fail，并可挂到 ci gate。

Demo 思路：引用格式检测

仓库 demos/advanced-03-deepeval-lab/custom_metric_demo.py 用极简函数演示「答案含 [pol- 则满分」——真实项目可升级为正则 + 白名单 policy id。

python 
def cite_ok(answer: str) -> bool:
    """CorpAssist S1：制度答案须含 policy 引用标记。"""
    return "[pol-" in answer

def custom_cite_metric(actual_output: str) -> float:
    return 1.0 if cite_ok(actual_output) else 0.0

# DeepEval 中可 subclass BaseMetric，measure() 内调用 custom_cite_metric
# 并与 FaithfulnessMetric 同列 assert_test(case, [faith, cite])
metric 类型实现方式示例成本
规则 / 确定性 Python 函数、正则、JSON schema cite_ok、拒答关键词 低，适合 CI 全量跑
llm judge DeepEval / RAGAS 内置模板 faithfulness、relevance 按 token 计费，宜抽样+门禁子集
人工 rubric 导出 badcase → 标注（人工评测与仲裁） 合规语气、表格可读性 高，用于校准自动分

封装为可复用 metric plugin

  • 单一职责：一个 metric 只答一个问题（引用 / 拒答 / 长度），便于坏例归因（坏例归因方法论）。
  • 版本钉死：metric 逻辑变更视为「评测逻辑版本」，与 eval dataset 一并 version pin。
  • 映射 schema：自定义分写入 extensions.cite_format_ok 或合并进业务门禁，但核心 RAG 键名不变。
json 
{
  "stack": "python",
  "faithfulness": 0.91,
  "extensions": {
    "cite_format_score": 1.0,
    "metric_version": "cite-v1"
  },
  "gate": "pass"
}
custom metric 与 RAGAS 指标冲突时听谁的？

核心 RAG 阈值（faithfulness、context recall）优先；extensions 是附加 AND 条件。两者都 pass 才允许 merge，避免「格式对但内容错」。

判断：custom metric 得 1.0 但 faithfulness 0.4，能否发布？

不能。引用格式只是合规层；内容幻觉仍由 faithfulness 拦截。门禁应 AND 核心 RAG 阈值，而非用规则分替代三角。

选择建议

框架选型不是「谁更先进」，而是 eval abstraction 能否嵌入已有 NS1 飞轮：eval dataset → 双栈执行 → schema 报告 → regression gate → 看板（实战：CorpAssist 质量看板）。

团队现状建议理由
已稳定 RAGAS + pytest CI 保持 RAGAS；DeepEval 仅用于新增 assert 用例或 Confident AI 云报表 迁移成本高，vendor neutral 靠 schema 而非框架
Python 团队强 pytest、缺 RAG 指标经验 DeepEval 快速起步 + 逐步对齐 RAGAS 指标定义 学习曲线低，但需补 双栈同口径 文档
大量企业规则指标 custom metric 独立包；RAGAS/DeepEval 二选一跑 LLM 分 规则与 judge 解耦，tool selection 聚焦「谁跑 judge」
ToB 要第三方评测报告 固定 dataset 版本 + 双框架交叉验证 + 人工抽检（质量治理流程） 单框架分数不足以 tob acceptance

常见误区
  • 框架换血式迁移 — 未改 schema 就换工具，质量看板 历史趋势断裂。
  • 重复 judge — RAGAS 与 DeepEval 对同一用例各调一遍 LLM，CI 成本翻倍、结论不一致。
  • 忽视 Spring 栈 — Python 选型不能代替 eval parity 验证。
tool selection：以 schema 为中心的抽象层
flowchart LR
  ADPT_R[RAGAS adapter]
  ADPT_D[DeepEval adapter]
  ADPT_R --> SCHEMA[metrics-schema]
  ADPT_D --> SCHEMA
  SCHEMA --> DASH[质量看板]
  SCHEMA --> REG[回归门禁]

适配器模式保证 vendor neutral；换框架不换契约。

本章结论

每条对应上文一个小节，便于串起全章再做题。

  • DeepEval 简介：TestCase + assert 风格，补 RAGAS 主链；Spring 仍走集成测试，双栈靠 schema 对齐。
  • 自定义 metric：企业规则（如 [pol- 引用）用 deterministic 函数；LLM judge 负责 RAG 三角；二者 AND 门禁。
  • 选择建议：以 eval abstraction / schema 选型，避免换框架导致飞轮与看板断裂。

判断练习

  1. 同事提议「全面弃用 RAGAS，CI 只跑 DeepEval」。请指出最可能破坏 NS1 飞轮哪一环？

    参考答案

    双栈同口径与历史回归：Spring 侧与已有 baseline、质量看板 趋势若未做 schema 映射与 version diff 对齐，会造成「分数变好但无法对比」的伪提升。

  2. cite_format_score=1.0 且 faithfulness=0.55，能否对客户说「质量合格」？

    参考答案

    不能。引用格式只是形式合规；0.55 表示内容不贴证据，存在幻觉风险。应对外强调核心 RAG 阈值未达标。
