为什么要学本章

面试常问「你怎么证明 RAG 变好了」。要能讲清 before/after、A/B、诚实边界。量化叙事 + 可视化 + 坦诚局限 = 可信的工程叙事。

学完你能

  • 设计 before/after 图表含置信区间
  • 提供 A/B 实验证据链

本章先记住 4 件事
  • 先 baseline：没有基准线的「提升」不可信
  • 再 diff：delta 比绝对值更有说服力
  • A/B 证据：离线 + 在线双重验证
  • 诚实说局限：坦诚比完美更可信

量化叙事结构

quant narrative 回答面试官的核心问题：「你怎么知道变好了？」叙事结构：问题 → baseline → 改动 → 离线回归 → A/B → 业务指标。eval storytelling 要有 ns1 proof。

面试官：你怎么证明 Prompt v7 比 v6 更好？ 候选人答：「我们用固定的评测集 corpassist-s1-v3（200 条）在 Python 和 Spring 双栈上跑了 RAGAS。faithfulness 从 0.90 到 0.92 (+0.02)，同时线上 A/B 10% 流量观察 3 天，thumbs_up 从 85% 提升到 87% (p=0.03, n=3200)。P95 延迟从 2050ms 变为 2100ms（+50ms，在 SLA 范围内）。」这就是一个完整的量化叙事。
量化叙事框架
flowchart LR
  BASELINE[建设 baseline] --> CHANGE[实施改动]
  CHANGE --> OFF[离线回归]
  OFF --> AB[A/B 在线验证]
  AB --> METRIC[业务指标]
  METRIC --> STORY[量化故事]

  BASELINE --> |fixed dataset| PIN[version pin]
  OFF --> |RAGAS| SCORES[faithfulness +0.02]
  AB --> |guardrails| THUMBS[thumbs_up +2%, p=0.03]
  METRIC --> |business| TRANSFER[转人工 -1%]

量化叙事需要三个层级的证据：离线指标 → 在线 A/B → 业务指标。缺一不可。

Before/After 图表设计

chart story 的核心原则：

  • 双轴图：左轴 faithfulness，右轴 P95 延迟
  • 标注事件：在时间轴上标注「Prompt v7 上线」
  • 置信区间：A/B 指标展示 error bar
  • 数据集版本：标注 dataset_version 变化
  • 样本量：标注每个数据点的样本量
不推荐
  • 只有一句「提升 20%」没有上下文
  • 用不同尺度的 Y 轴误导
  • 不标样本量，让人怀疑统计意义
  • 只展示最好的一周
推荐
  • 曲线 + 置信区间 + 样本量标注
  • 前后一致的 Y 轴尺度
  • 标注数据集版本和变更事件
  • 展示完整趋势而非 cherry-pick

A/B 证据：说服力的关键

ab evidence 是面试中最有说服力的材料。离线指标可以「调参调出来」，但 A/B 是用户真实反馈。

证据类型示例说服力
离线 RAGAS 提升faithfulness 0.90 → 0.92中等
线上 A/B 提升thumbs_up +2%, p=0.03高
A/B + 离线一致两者都正向非常高
业务指标提升转人工率 -1%, 工单解决时长 -20%最高

面试常见陷阱

interview trap 是面试官用来考察你深度的问题。提前准备能让你脱颖而出。

markdown 
## Trap 1: 「我们感觉 RAG 效果不错」
❌ 回答：「是的，我们也觉得很好。」
✅ 回答：「我们有量化证据。固定评测集 corpassist-s1-v3，
    faithfulness 0.90→0.92，A/B 实验 3 天 thumbs_up +2% (p=0.03)。」

## Trap 2: 「离线 0.95 说明很好了」
❌ 只展示离线分数
✅ 补充：「离线 0.95 但我们也关注线上。A/B 实验中 thumbs_up 确实提升，
    同时 P95 延迟从 2050 只升到 2100ms，在 SLA 范围内。」

## Trap 3: 「只展示最好的案例」
❌ 只挑 3 个成功的 badcase
✅ 「200 条评测集中 3 个 badcase 修复了，但我们也发现 2 个新的 hard case，
    已入库跟踪。整体 faithfulness +0.02。」

## Trap 4: 「没有 A/B 就说提升」
❌ 只跑离线就说上线也会好
✅ 「离线提升 +0.02，计划下周 10% A/B 验证。如果显著则全量。」

## Trap 5: 「忽略局限性」
❌ 不说缺点
✅ 「Prompt v7 在拒答率上有轻微提升（+1%），但 thumbs_up 也提升。
    我们正在观察拒答率是否回到合理范围。」

metric honesty 是所有工程叙事的基础。不夸大、不 cherry-pick、不隐藏退化。面试官欣赏坦诚的候选人。

代码：Metric Honesty 检查器

python 
# metric_honesty.py — 指标诚实性检查
class MetricHonestyCheck:
    """防止指标被 cherry-pick"""

    @staticmethod
    def check_report(report: dict) -> list:
        """检查报告是否包含必要的诚实信息"""
        warnings = []
        if "sample_count" not in report:
            warnings.append("缺少样本量")
        if "dataset_version" not in report:
            warnings.append("缺少数据集版本")
        if "aggregate" in report:
            agg = report["aggregate"]
            if not any(v <= 0.95 for v in agg.values() if v is not None):
                warnings.append("所有指标 > 0.95，可能评测集过简单")
        if "per_sample" in report:
            scores = [s.get("faithfulness", 1) for s in report["per_sample"]]
            if max(scores) - min(scores) < 0.05:
                warnings.append("样本间差异过小，可能数据泄露")
        return warnings

回答框架：三句话应对任何提问

场景回答框架示例
「怎么证明效果？」baseline → 离线 → A/B → 业务「基于 corpassist-s1-v3 评测集，faithfulness 0.90→0.92，A/B 实验 thumbs_up +2% (p=0.03)」
「离线高线上差？」分布偏移 → 代理指标 → 补充 A/B「离线集与线上分布有偏移，我们在补充线上样本。同时已经在 10% 流量上跑 A/B」
「指标有下降吗？」坦诚下降 → 根因 → 修复计划「context_recall 下降了 0.01，排查发现 embedding 模型版本变更，已回滚」
「为什么信你的数字？」固定评测集 → CI 门禁 → artifact「dataset_version pin 在 git tag，每次 MR 自动跑 CI，报告含 git_sha 可复现」
先想 10 秒：面试官问「你的 RAG 离线 0.95 但线上用户说不好，你怎么回答」？

坦诚原因：① 离线集可能不覆盖线上分布 ② 离线测的是「能否回答正确」，用户还关心「是否好用」 ③ 需要看线上 A/B 指标如 thumbs_up 和转人工率 ④ 展示你已经在做线上指标监控。

Interview 模拟：1 分钟自述

准备一个 1 分钟的 NS1 飞轮叙事，包含：

  • 问题：S1 制度问答 faithfulness 在难例上 0.82，法务部门投诉
  • baseline：corpassist-s1-v3 评测集 200 条
  • 改动：优化检索 top_k 从 3 到 5 + Prompt v7 约束「未找到拒答」
  • 离线结果：faithfulness 0.82 → 0.91 (+0.09)
  • A/B 结果：10% 流量 3 天，thumbs_up +3% (p=0.02, n=3200)
  • 局限：P95 从 2000ms 升到 2150ms，仍在 SLA 内；继续观察拒答率
NS1 飞轮叙事流程
flowchart LR
  P[问题发现] --> B[Baseline 建立]
  B --> C[改动实施]
  C --> O[离线回归确认]
  O --> A[A/B 在线验证]
  A --> R[业务指标改善]
  R --> L[局限坦诚]
  L --> N[下一轮迭代]

完整的量化叙事包含问题 → 证据 → 局限 → 下一步，而非只展示成功。

代码：叙事模板生成器

python 
# interview_narrative.py — 面试叙事模板
class InterviewNarrative:
    """生成量化叙事模板"""

    @staticmethod
    def build(problem: str, baseline: dict, change: str,
              offline_result: dict, ab_result: dict,
              limitation: str) -> str:
        """构建完整的量化叙事"""
        lines = [
            f"## 问题：{problem}",
            "",
            "### Baseline",
            f"- 评测集：{baseline.get('dataset', 'corpassist-s1-v3')}（{baseline.get('n', 200)} 条）",
            f"- Faithfulness: {baseline['faithfulness']}",
            f"- Context Recall: {baseline['context_recall']}",
            "",
            f"### 改动：{change}",
            "",
            "### 离线结果",
            f"- Faithfulness: {offline_result['faithfulness_old']} → {offline_result['faithfulness_new']} (Δ{offline_result['faithfulness_delta']})",
            f"- Context Recall: {offline_result['recall_old']} → {offline_result['recall_new']} (Δ{offline_result['recall_delta']})",
            "",
            "### A/B 结果",
            f"- 流量：{ab_result.get('traffic', '10%')}",
            f"- 时长：{ab_result.get('duration', '3 天')}",
            f"- Thumbs Up: {ab_result['thumbs_control']}% → {ab_result['thumbs_treatment']}% (p={ab_result['p_value']})",
            "",
            "### 局限",
            f"- {limitation}",
            "",
            "### 下一步",
            "- 持续监控 x 指标",
            "- 下一轮迭代计划",
        ]
        return "\n".join(lines)

常见回答对比：差 vs 好

差的回答
  • 「我们换了 GPT-4，效果变好了」
  • 「用户反馈很正面」
  • 「评测大概提升了 20%」
  • 「没有发现什么副作用」
好的回答
  • 「基于 corpassist-s1-v3 评测集，faithfulness 从 0.90 到 0.92」
  • 「A/B 实验 3 天，thumbs_up +2% (p=0.03, n=3200)」
  • 「P95 延迟从 2050ms 变为 2100ms (+50ms, 在 SLA 范围内)」
  • 「同时我们发现 hard case 集上 faithfulness 还有提升空间」

本章结论

面试中证明效果提升需要三层证据：离线 RAGAS + 线上 A/B + 业务指标。再加上坦诚的局限分析，就是最可信的工程叙事。

  • 量化叙事：问题 → baseline → 改动 → 离线 → A/B → 业务指标。
  • 图表：双轴 + 置信区间 + 标注变更事件。
  • 陷阱：避开「感觉很好」「只展示最好案例」「无 A/B 说提升」等 5 个陷阱。
  • Metric Honesty：坦诚局限比完美更可信。

答辩 / 面试口述题

  2. 面试官问「离线很高线上很差」如何回答？
  3. 如何证明不是偷换了评测集才提升的分数？
  4. Prompt v7 提升 faithfulness 但增加拒答率，你如何决策？
更多面试题
  1. 你的 Prompt 只改了 temperature，faithfulness 从 0.88 升到 0.92，能说是你的改进吗？
  2. A/B 实验中 control 和 treatment 的用户量差异很大（900 vs 100），如何保证结论可靠？
  3. 客户说「离线 0.95 但我们还是不满意」，你如何应对？
面试中最容易犯的错误是什么？

三个最常见的错误：① 只说结果不说方法（「提升 20%」但不解释如何测量）② 只展示正面案例不展示局限 ③ 无法回答「离线高线上差」的问题。准备面试时，确保你能用一个完整叙事讲清一个改进案例。

如何应对压力面中的指标质疑？

面试官可能会说「这些指标都是你选的数据集算出来的」。应对方法：① 承认数据集有局限 ② 展示你用了 version pin 防止偷换 ③ 展示 A/B 实验的线上证据。最终要表达：我不仅关心指标数字，更关心指标的可信度。
