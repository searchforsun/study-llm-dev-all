为什么要学本章

CorpAssist S1 企业知识库 RAG 上线前，业务会问「比人工客服准多少、够快吗」——若只有「感觉不错」，无法发版、无法 A/B、无法在面试中答辩。本章建立双栈共用指标口径，是 NS1 评测飞轮 的计量基础。没有量化指标，就无法衡量改进，也无法在 ToB 验收中证明价值。

  • 区分「检索失败」与「生成胡编」在指标上的不同表现

本章先记住 4 件事
  • 离线 vs 在线：RAGAS 等指标用于离线回归；线上还要看 thumbs、转人工率、P95。
  • 忠实 ≠ 相关：faithfulness 看是否胡编；answer relevance 看是否答非所问。
  • 拒答可评测：「无法从制度找到依据」是合法输出，应计入 hit rate 与拒答率。
  • 双栈同 schema：dual stack metrics 用同一 JSON 字段名，便于 CI 对比 Python 索引链与 Spring 查询链。

RAG 指标定义

CorpAssist · S1 制度问答评审 法务上传《远程办公管理办法》后，产品问：「模型说审批 3 天，制度里写的是 5 个工作日——这算检索错还是生成错？」本章指标帮你定位链路环节，而不是笼统说「模型不行」。

RAG 评测通常从「检索是否找对」「生成是否忠实」「答案是否切题」三个维度展开。CorpAssist 采用 RAGAS 兼容命名，便于与后续 RAGAS 与自动评测 流水线对接。

核心 RAG 指标对照

指标衡量什么低分常见原因CorpAssist 阈值示意
context recall 检索片段是否覆盖 gold 所需信息 分块切坏、Embedding 模型不匹配、TopK 过小 ≥ 0.85（回归集）
faithfulness 回答是否仅依据检索上下文、无胡编 Prompt 未约束「仅依据上下文」、模型过度发挥 ≥ 0.90
answer relevance 回答是否解决用户问题 检索偏题、生成跑题、多轮上下文干扰 ≥ 0.88
引用覆盖率 制度问答是否带 chunk 引用 未强制 citation 格式、检索为空仍生成 S1 必达 100% 有引用或明确拒答

反例 vs 推荐：指标选型

不推荐
  • 只报 BLEU/ROUGE 与 gold 文本字面重合
  • 用「用户满意度」替代可回归指标
  • 检索与生成混在一个「准确率」里
  • 拒答样本从评测集删除
推荐
  • RAGAS 三维 + 引用率 + 拒答正确率
  • 难例库单独统计，不与简单题混平均
  • 检索失败与生成失败分开归因（见 坏例归因方法论）
  • 拒答作为一等公民写入 gold 标签

先想 10 秒：faithfulness 很高但 answer relevance 很低，最可能哪一环出问题？

检索到的片段「忠实」地被复述，但与用户问题无关——优先查检索/query 改写与多轮上下文污染，而非换更大模型。

Agent 成功率与 SLA

除 RAG 外，CorpAssist S2 智能客服 与 S3 Agent 任务自动化 需要 agent success：任务是否在步数预算内完成、工具调用是否成功、最终状态是否符合业务规则。

  • 工具调用成功率：schema 合法且 HTTP 2xx；失败要区分「模型选错工具」与「下游 API 超时」。
  • 任务完成率：用户意图「改收货地址」是否在规则校验后真正写入订单系统。
  • 步数预算：超过 N 步强制转人工或拒答，避免 Agent 空转烧 Token。CorpAssist 设定 S3 Agent 步数上限为 8 步。

Agent vs RAG 指标侧重

场景首要指标次要指标
S1 制度 RAG faithfulness、citation_rate context_recall、P95 延迟
S2 智能客服 任务完成率、转人工率 answer_relevance、工具成功率
S3 办公 Agent agent success、步数分布 faithfulness（计划是否可执行）

Agent 指标代码示例

python 
# agent_metrics.py — Agent 成功率计算
from dataclasses import dataclass, field
from typing import List, Optional

@dataclass
class AgentTrace:
    """Agent 单次执行追踪"""
    task_id: str
    steps: int
    max_steps: int = 8
    tool_calls: List[dict] = field(default_factory=list)
    final_status: str = ""  # success / failure / timeout
    error_type: Optional[str] = None  # tool_error / plan_error / timeout

def compute_agent_success(traces: List[AgentTrace]) -> dict:
    """计算 Agent 成功率指标组"""
    total = len(traces)
    if total == 0:
        return {}
    success = sum(1 for t in traces if t.final_status == "success")
    tool_errors = sum(1 for t in traces if t.error_type == "tool_error")
    timeout = sum(1 for t in traces if t.final_status == "timeout")
    avg_steps = sum(t.steps for t in traces) / total

    return {
        "agent_success_rate": success / total,
        "tool_error_rate": tool_errors / total,
        "timeout_rate": timeout / total,
        "avg_steps": avg_steps,
    }
Agent 场景还要跑 RAGAS 吗？

若 Agent 内部含 RAG 子链（查制度再答），应对该子链单独跑 context recall / faithfulness；整体任务用 agent success + 端到端 rubrics。不要用一个平均分掩盖子链路失败。

SLA：延迟与可用性门禁

质量指标必须与 latency sla 和可用性一起门禁。ToB 验收常见：P95 < 3s、可用性 99.9%、离线 faithfulness 不低于 baseline - 0.02。

延迟分位与用户体验

分位CorpAssist S1 目标超标时的工程动作
p95 latency ≤ 2500 ms（含检索+生成） Rerank 降级、缓存热问、小模型路由
P99 ≤ 4500 ms 排查慢检索、Embedding 冷启动、长上下文
可用性 99.9% / 月 模型 Fallback、索引只读副本、熔断

双栈同口径：Python 与 Spring 共用指标

Python 侧常负责索引与 RAGAS 批跑；Spring 侧常负责 BFF 与集成测试。若 Python 报 0.92、Spring 报 0.78，多半是索引版本、Prompt 版本或检索参数不一致，而非「Java 比较笨」。

  • 维护评测集 JSONL 与 dataset_version
  • 批跑 RAGAS，产出 json artifact 上传 CI
  • 索引 pipeline 版本写入报告 metadata
  • 提供 baseline 报告供 Spring 对照
  • 集成测试读取同一 JSONL 子集
  • Advisor 链路与生产一致，禁止 mock 检索结果做「假绿」
  • 报告字段名与 Python schema 逐 key 对齐
  • 产出 parity diff 报告

双栈指标对齐流程

Python ↔ Spring 指标对比
flowchart TB
  DS[(同一 JSONL)] --> PY[Python RAGAS]
  DS --> SP[Spring IT]
  PY --> REP1[python-report.json]
  SP --> REP2[spring-report.json]
  REP1 --> DIFF{parity diff}
  REP2 --> DIFF
  DIFF -->|delta ≤ 0.02| PASS[通过 ✓]
  DIFF -->|delta > 0.02| INVESTIGATE[排查索引/Prompt 差异]

字段级 diff 确保双栈指标可比。常见差异原因：Python 用 OpenAI Embedding，Spring 用 ONNX 模型。

常见误区
  • Spring 测试用内存 H2 向量库、Python 用 Milvus — 分数不可比。
  • 只对比平均分，不看同一题的双栈 diff 列表。
  • 延迟只测 LLM，不含检索与 Rerank。

本章结论

指标是 NS1 飞轮的「刻度尺」。发版前必须同时看质量分与 P95，且 Python/Spring 共用 schema。

  • RAG 指标定义：context recall、faithfulness、answer relevance 分工明确；拒答与引用率单独统计。
  • Agent 成功率：S2/S3 看任务完成与工具成功，RAG 子链仍要拆出 RAGAS 指标。
  • SLA：P95/P99 与可用性写入 yaml 门禁，不与离线分混为一谈。
  • 双栈同口径：同一 dataset_version、同一索引与 Prompt pin，字段级 diff 排查 parity。

动手练习

为 CorpAssist S1 填写指标 schema 阈值，并对照正文对照表解释一条「faithfulness 高、relevance 低」的坏例（约 25 分钟）。

操作步骤

  1. 打开正文「核心 RAG 指标对照」表，为 faithfulness、context_recall 填写你们 S1 的可接受下限（可沿用示意值或按业务调整）。
  2. 复制 JSON schema 示例，增加字段 refusal_correct_rate 并设阈值 0.90，说明拒答题在 JSONL 中如何标注 gold。
  3. 对照 Mermaid 流程图，写出离线流水线中「Python 跑」与「Spring 跑」各发生在哪两个 CI job。
  4. 可选：进入 demos/basics-01-metrics-lab/，编辑 metrics-schema.json 中 2 个阈值并 README 验收。

判断练习

  1. 产品要求「只要 faithfulness ≥ 0.95 就发版」，忽略 P95。S1 ToB 合同要求 3 秒内响应。这是否足够？

    参考答案

    不足。高质量但 8s 响应仍违约；且 faithfulness 单指标无法覆盖检索失败、拒答错误等。门禁应质量 + 延迟 + 相对 baseline 回归组合（见 yaml 示例）。

  2. Agent trace 显示 tool_error_rate 30%，但 avg_steps 正常。你应该从哪开始排查？

    参考答案

    先排查工具调用 schema 是否正确、下游 API 是否有超时。30% tool_error 不是步数问题，而是工具执行层。检查 tool_call 日志中的 HTTP 状态码与响应体。
