为什么要学本章

CorpAssist 企业知识库 RAGS1 已在 指标：准确、召回、忠实、延迟 定义了 faithfulness 等契约，在 评测集构建 准备了 JSONL。若仍靠人工逐条打分，MR 合并前无法做回归。ragas 把「检索→生成→指标」串成可重复的 eval pipeline，是 NS1 飞轮 中自动评测一环。

学完你能
  • 画出 S1 RAG 的 RAGAS 输入输出（question / contexts / answer / ground_truth）与指标映射
  • 用 Python 脚本或 pytest 对 corpassist eval JSONL 做 batch eval
  • 在 CI 配置 ci gate 与 threshold gate，低于阈值阻塞合并
  • 解读自动分与人工 rubrics 的偏差，避免「分数好看、客户不买账」
本章先记住 4 件事
  • Pipeline 四元组：question、contexts、answer、（可选）ground_truth → RAGAS auto metric。
  • Judge 成本：多数指标依赖 llm judge，CI 应固定模型与 prompt 版本。
  • 双栈同集：Python RAGAS 与 Spring 集成测试读同一 JSONL hash，输出映射到 dual stack metrics schema。
  • 分≠放行：自动分是门禁必要条件，不是充分条件；上线仍要人工抽检与 A/B。

pipeline

ragas 的 eval pipeline 把 CorpAssist S1「问→检→答」产物标准化为评测行，再批量调用指标函数。核心不是「调一个 API」，而是数据契约 + 执行顺序 + 结果落盘。

CorpAssist S1 · MR 前回归（45 秒） 工程师改了检索 top-k。CI 拉取 eval-sample.jsonl（与 Spring 侧同 hash），对每条样例：① 跑 S1 RAG 得到 answer 与 contexts ② 调用 RAGAS 算 faithfulness / context_recall ③ 汇总均值并与 metrics-schema.json 阈值比 ④ 任一项低于 threshold gate 则 block merge。
CorpAssist S1 · RAGAS eval pipeline
flowchart LR
  DS[(eval JSONL)] --> RUN[S1 RAG 推理]
  RUN --> ROW[行: q / ctx / ans / gold]
  ROW --> RAGAS[RAGAS metrics]
  RAGAS --> MAP[映射 metrics-schema]
  MAP --> GATE{threshold gate}
  GATE -->|pass| ART[json artifact]
  GATE -->|fail| BLOCK[CI block]

与 NS1 飞轮：自动评测输出进入「门禁发布」，坏例进入后续人工/A/B 环节。

阶段输入输出常见失败
加载评测集 JSONL：query、gold、tags 内存 Dataset / 迭代器 评测集 version pin 与 MR 不一致
推理 query answer + retrieved contexts 环境变量指向错误索引/模型
打分 四元组 faithfulness、context_recall 等 Judge 模型漂移导致分不可比
门禁 聚合分 + schema 阈值 pass/fail + 报告 只报均值不看 hard case 分位

反例 vs 推荐：Pipeline 设计

不推荐
  • Notebook 手点跑分，无版本、无 CI
  • 只存 answer，不存 contexts，无法算 faithfulness
  • 换评测集仍用旧阈值，误杀/误放
  • Python 与 Spring 各跑各的 CSV 字段名
推荐
  • 脚本化 batch eval + JSON 报告 artifact
  • 每行保留 contexts 快照便于归因
  • 阈值与 schema、评测集 tag 同 PR 审查
  • 输出映射 dual stack metrics
先想 10 秒：没有 ground_truth 还能跑 RAGAS 吗？

可以。faithfulness、answer relevance 等可不依赖 gold；context recall 通常需要 gold 或人工标注的参考证据。S1 评测集应尽可能带 gold。

Python 脚本

工程上常用两种方式：独立 runner 脚本（本地/CI 一条命令）或 pytest 用例（与单元测试同仓库）。CorpAssist Python 栈以前者做 nightly、后者做 MR 快路径。

最小 mock runner（与 Demo 对齐）

python 
# demos/basics-03-ragas-lab/mock_ragas.py — 示意，不调用真实 API
def faithfulness_score(answer: str, contexts: list[str]) -> float:
    """S1：答案含 doc_id 引用则高分"""
    cited = any(c.split("-")[0] in answer for c in contexts if "-" in c)
    has_ref = "pol-" in answer or "doc_id" in answer
    return 0.91 if (cited or has_ref) else 0.42

def run_row(query: str, answer: str, contexts: list[str]) -> dict:
    return {
        "query": query,
        "faithfulness": faithfulness_score(answer, contexts),
        "answer_relevance": 0.88 if len(answer) > 20 else 0.55,
        "context_recall": 0.85 if contexts else 0.0,
    }

if __name__ == "__main__":
    row = run_row("年假几天？", "依据 pol-001…", ["pol-001 text"])
    print(row)  # 期望 faithfulness >= 0.9

pytest 门禁片段（示意）

python 
import json
from pathlib import Path

THRESHOLDS = json.loads(Path("metrics-schema.json").read_text(encoding="utf-8"))

def test_s1_faithfulness_regression():
    from mock_ragas import run_row
    bad = run_row("报销上限？", "随便说说", ["fin-002"])
    assert bad["faithfulness"] >= THRESHOLDS["faithfulness_min"], bad

适用：全量 JSONL、生成 HTML/JSON 报告、上传 artifact。命令：python -m ragas_runner --dataset eval-v3.jsonl（命名见 ragas runner）。

适用：MR 快反馈、与现有 pytest eval 同 job。注意：抽样 20 条 hard tag + 全量 smoke，避免 CI 超时。

双栈提示

Spring 侧不直接跑 RAGAS Python 包时，应对同一 JSONL 做集成测试，断言后写入相同 schema 字段——详见 Spring AI 服务评测。

CI 集成

自动评测的价值在「每次变更可重复、可阻塞」。ci gate 把 RAGAS 报告挂到 PR：失败即不可合并，避免 NS1 飞轮在发布环节失守。

GitHub Actions 示意

yaml 
jobs:
  rag-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Pin eval dataset
        run: echo "EVAL_SHA=$(sha256sum eval/corpassist-s1.jsonl | cut -d' ' -f1)" >> $GITHUB_ENV
      - name: Run RAGAS batch
        run: python scripts/run_ragas.py --out reports/ragas.json
      - name: Threshold gate
        run: python scripts/check_thresholds.py reports/ragas.json metrics-schema.json
场景门禁策略说明
日常 MR hard 子集 + 核心指标均值 平衡成本与信号；< 10 min
Release candidate 全量 JSONL + SLA 分位 与 regression set 对齐
仅 Prompt 微调 faithfulness + answer_relevance 检索未动时可跳过 recall 全量
索引/embedding 变更 context_recall 必跑 生成再好也救不了漏检索

CI 集成 FAQ

Judge 调用太慢，CI 超时怎么办？
① 分层：smoke 子集每 MR，全量 nightly；② 缓存相同 query 的推理结果；③ 固定 judge 模型与 batch size；④ 极端情况用本章 mock 做结构测试，真实 judge 放 release 流水线。
阈值设多少「合理」？
从当前 baseline 减 0.02～0.05 作 MR 门禁，全量发布用 baseline 本身。阈值必须与 metrics-schema.json 和评测集版本同 PR 修改，避免偷换集合作弊。
Python CI 绿了，Spring 还需要跑吗？
需要。双栈客户可能走不同部署路径；Spring 集成测试验证 Advisor 链与 HTTP 契约，见 Spring AI 服务评测 的 eval parity。
先想 10 秒：MR 只改文档，还要跑全量 RAGAS 吗？

文档-only 可配置 skip 规则，但需团队显式约定。默认仍跑 smoke 子集，防止「顺手改了一行 Prompt 配置」被漏检。

结果解读

自动分是 llm judge 或启发式函数的输出，不是用户满意度。metric interpret 要求你看分布、坏例、与人工 rubrics 的对照，而不是只报一个均值。

现象可能含义建议动作
faithfulness 高、客户仍投诉 Judge 与业务 rubrics 不一致；引用格式「骗过」指标 抽 20 条做 人工评测与仲裁
recall 跌、faithfulness 升 检索变保守，生成「安全但漏答」 查索引与 top-k；勿单看忠实
双栈分差 > 0.05 推理配置或 schema 映射不一致 cross stack compare 对齐 trace
hard tag 均值远低于 easy 回归集未覆盖真实难例 坏例回流 badcase reflow

NS1 飞轮：自动评测之后是什么？

门禁发布 → 线上 A/B / 灰度 → 用户反馈 → 坏例回流评测集。只停在离线高分，飞轮不会转。

本章结论

每条对应上文一个小节，便于串起全章再做题。

  • pipeline：JSONL → 推理 → RAGAS 四元组 → 映射 schema → threshold gate；须 version pin 与 contexts 快照。
  • Python 脚本：runner 全量 + pytest MR 快路径；Demo mock_ragas.py 展示引用与分数关系。
  • CI 集成：ci gate 阻塞 merge；分层 smoke / nightly 控制 judge 成本。
  • 结果解读：看分布与 hard 子集；自动分与人工 rubrics 偏差要进 calibration，支撑 dual stack metrics 与 NS1 飞轮。
