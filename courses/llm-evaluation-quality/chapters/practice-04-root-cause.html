为什么要学本章

线上 thumbs signal 与离线红灯若只停留在「模型不行」，团队会反复调温度却修不好索引切片。CorpAssist NS1 评测飞轮要求每条坏例有 root cause 标签，才能进入 fix loop 并回流 regression set——本章把 error taxonomy、trace analysis 与 case library 串成可执行方法论。

学完你能
  • 用 context recall / faithfulness 组合判断 retrieval failure、generation failure 或 prompt regression
  • 在 LangSmith / OTel 链路上关联 request_id、检索 snapshot 与 prompt_version（双栈同 schema）
  • 按 badcase-template.md 维护案例库，并写清回流 hard_cases.jsonl 的 regression_id
  • 衔接 线上 A/B 与灰度 回滚样本与 用户反馈与迭代闭环
本章先记住 3 件事
  • 先看指标三角：recall↓ 优先检索；faith↓ 且 recall 正常优先生成；全指标漂移查 Prompt/模型版本。
  • trace 是证据：没有 retrieve span 与 top_chunks 快照，不要拍脑袋改 Prompt。
  • 案例库要可回流：每条 badcase 带 owner、fix 状态与 version pin，禁止 Excel 散落。

检索 vs 生成 vs Prompt

CorpAssist S1 · 制度问答 用户问「年假可否顺延至次年 Q1」。离线评测：context_recall=0.42、faithfulness=0.91——上下文未命中 HR 制度第 7 条，但模型在已有片段上回答保守，未胡编。应标 retrieval，修复 chunk 边界与 embedding，而非改 system Prompt。
CorpAssist S5 · 营销文案 生成活动邮件时 contexts 已含品牌禁用词表，输出仍出现竞品名。recall 正常、faithfulness 低 → generation；同时检查 S5 模板 Prompt 是否缺少「禁止提及竞品」约束。

归因不是主观标签，而是可观测信号 → 错误类型 → 修复入口。下表是 CorpAssist 默认 error taxonomy（与 RAGAS 指标对齐）。

观测信号优先归因典型修复勿误判为
context_recall ↓，faithfulness 正常或偏高 retrieval failure re-embed、调 topK、改 chunk、加 metadata 过滤 生成胡编（上下文根本没进来）
contexts 含 gold 依据，faithfulness ↓ generation failure 降温度、加强拒答、Citation 指令、工具校验 检索失败
多指标同向漂移，A/B 仅改 Prompt prompt regression prompt_version diff、回滚模板、加 few-shot 单条检索坏例
仅 answer relevance ↓，检索与忠实正常 指令跟随 / 格式 输出 schema、JSON mode、后处理 faithfulness 问题

指标三角 · Mermaid

坏例归因 · 指标三角决策
flowchart TD
  START[坏例进入 triage] --> R{context_recall 低?}
  R -->|是| RET[retrieval failure]
  R -->|否| F{faithfulness 低?}
  F -->|是| GEN[generation failure]
  F -->|否| P{prompt_version 变更?}
  P -->|是| PR[prompt regression]
  P -->|否| OTHER[工具/Agent 等扩展类]

先判 recall，再判 faith，最后对照发版与 prompt_version；Agent 场景叠加 agent success。

反例：只改 Prompt
  • recall 0.4 仍加「请严格依据上下文」
  • 分数不变，浪费一轮发版
  • 案例库无检索快照，无法复盘
推荐：证据驱动
  • 导出 top_chunks 与 gold doc id 对比
  • 标 retrieval + 开 KB 工单
  • 修复后把 id 写入 gate 子集
先想 10 秒：faithfulness 低、context_recall 高，最不该做的第一件事是什么？

不应先 re-embed 全库。应打开生成 span，看是否忽略 contexts、是否工具幻觉，再改 Prompt/解码参数。

工具链

trace analysis 把一次用户请求拆成可审计片段：检索耗时、rerank 得分、LLM token、prompt_hash。CorpAssist 要求 Python FastAPI 与 Spring BFF 同一 trace_id，并写入评测 artifact，便于 cross stack compare。

OTel span 结构（示意）

json
{
  "trace_id": "t-7f3a",
  "request_id": "req-20260530-8831",
  "scenario": "S1",
  "experiment_id": "ca-prompt-v3-202605",
  "prompt_version": "v12",
  "spans": [
    { "name": "retrieve", "duration_ms": 120, "attrs": { "top_k": 8, "index": "hr-handbook-v4" } },
    { "name": "rerank", "duration_ms": 40, "attrs": { "model": "bge-reranker" } },
    { "name": "generate", "duration_ms": 800, "attrs": { "model": "gpt-4o-mini", "tokens_out": 312 } }
  ],
  "retrieval_snapshot": {
    "chunk_ids": ["c-883", "c-102", "c-441"],
    "scores": [0.81, 0.77, 0.71]
  }
}

Python 侧：把 trace 挂到评测报告

python
# eval_runner/attach_trace.py — 失败 case 必带 trace_id
def enrich_failure(case_id: str, trace_id: str, report: dict) -> None:
    report.setdefault("failures_detail", []).append({
        "case_id": case_id,
        "trace_id": trace_id,
        "langsmith_url": f"https://smith.langchain.com/o/corp/public/{trace_id}",
        "prompt_version": otel_get_attr(trace_id, "prompt_version"),
    })

java

适合 Prompt 迭代与人工标注：按 dataset 回放，对比两次 run 的 retrieval_snapshot。CorpAssist MR 评论机器人可贴 LangSmith 链接。

适合 ToB 私有化：span 写入 ClickHouse，看板按 scenario（S1/S5）聚合 p95 与失败率。需自建「坏例详情页」关联 request_id。

排查步骤看什么结论动作
1retrieve span 是否为空或 lat 异常索引/路由问题
2top_chunks 是否含 gold doc检索 vs 切片
3generate 输入 token 是否截断 contexts生成长度/模板
4prompt_version 是否与 baseline 一致Prompt 回归
FAQ：Python trace 正常、Spring 同 query 失败，先查什么？

查双栈 index_version、top_k、rerank 是否启用，以及 BFF 是否走了缓存旧 contexts。用同一 request_id 在两边导出 retrieval_snapshot 做 diff，而不是先改模型。

本章结论

  • 检索 vs 生成 vs Prompt：用 recall / faithfulness 与 prompt_version 决策，S1 偏检索、S5 偏生成与模板约束。
  • 工具链：trace 关联 request_id、retrieval_snapshot、prompt_version；双栈 artifact 可 diff。
  • 案例库：Git 管理 badcase + regression_id，修复后回流 jsonl 并过 CI gate。

动手练习

在 demos/practice-04-root-cause-lab/ 用 badcase-template.md 完成 1 条 S1 与 1 条 S5 虚构坏例，并写明回流步骤。

操作步骤

  1. 阅读「检索 vs 生成 vs Prompt」决策表：为 S1 年假问题填「观测信号 → 归因 → 修复」三列。
  2. 对照「工具链」OTel JSON：写出 retrieval_snapshot.chunk_ids 与 gold doc 不一致时的排查顺序（4 步表）。
  3. 复制 badcase-template.md 为 badcase-s1.md，补全 regression_id、fix_owner、fix_status。
  4. 在草稿中写：该 id 如何进入 corpassist-regression-v3.jsonl（需 bump 哪个 version 字段）。
  5. 可选：打开 LangSmith 文档，列出 2 个你会在 MR 评论里贴的 trace 字段。

判断练习

  1. （场景）contexts 已含正确制度条款，答案仍编造不存在的「第 99 条罚款」。context_recall=0.9，faithfulness=0.4。应标哪类 root cause？首要修复方向？

    参考答案

    标 generation failure。优先检查生成 Prompt、温度与 citation 约束，而非 re-embed。若 prompt_version 刚变更，同时查 prompt regression diff。

  2. （找错）团队把坏例存在个人 Excel，MR 未更新 jsonl，faithfulness 周报却显示「已修复」。这违反本章哪条约定？

    参考答案

    违反案例库 Git 化与 regression_id 回流。修复未进入 regression set，CI gate 无法防止回归；NS1 飞轮在「案例库→评测集」环节断裂。
