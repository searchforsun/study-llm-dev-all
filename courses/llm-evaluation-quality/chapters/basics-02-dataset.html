为什么要学本章

没有固定评测集，每次改 Prompt 都像「凭感觉验收」。Golden Set 是 NS1 飞轮的燃料：坏例回流、发版回归都依赖同一套可 pin 的 JSONL。

学完你能
  • 设计 S1 JSONL 字段（question、contexts、gold_answer、metadata）
  • 用分层采样覆盖拒答/多跳/表格题
  • 建立难例库与 version pin 流程
  • 编写标注 rubric 减少标注员漂移
本章先记住 4 件事
  • 版本 pin：dataset_version 与 git tag 对齐，禁止偷换评测集
  • gold 多样性：拒答、引用、数值计算题都要占比例
  • 分层采样：按意图/文档类型分层，避免只测简单 FAQ
  • 难例库：线上 badcase 必须可回流且带 trace

采样

采样 决定评测集是否代表真实流量。CorpAssist S1 应从生产日志脱敏抽样，而非只用产品同事写的 20 条 demo 问句。

  • 随机抽样：适合建立 baseline，但易漏长尾
  • stratified sample：按 intent、doc_type、是否拒答分层
  • 主动学习：优先标注模型不确定或 disagreement 高的样本

标注规范

annotation rubric 要可操作：什么算 faithful、什么算应拒答、引用指向哪个 chunk_id。

不推荐
  • 「答案要好」无细则
  • 每标注员自定标准
  • gold 写完整段落复制粘贴
推荐
  • Likert + 通过/不通过二元门
  • 双盲标注 + 仲裁
  • gold 允许 paraphrase 但关键数字必须对

代码示例

markdown
## 忠实度
- Pass：每个事实可在 contexts 找到
- Fail：出现库外政策名称或编造天数

## 拒答
- Pass：明确说明无依据
- Fail：无检索仍给具体审批流程

难例库

CorpAssist上次发版后线上出现「扫描件 PDF 表格审批天数错误」，已回流 12 条。难例库标记 source=prod_badcase，发版前 faithfulness 在此子集不得下降。

hard case 单独成库，回归时必须全量跑，不可与普通集混平均稀释。

难例回流
flowchart LR
  PROD[线上 thumbs-down] --> TRIAGE[归因标注]
  TRIAGE --> HARD[hard_cases.jsonl]
  HARD --> CI[回归全量跑]
  CI --> MERGE[合并进下版 dataset vN+1]

CorpAssist 评测集

corpassist eval 最终产物：eval/corpassist-s1-vN.jsonl + manifest（条数、分层比例、version pin）。

  • Python 轨维护 ingest 脚本与 hash
  • Spring 轨集成测试通过 classpath 或 artifact 拉同一文件
  • CI 打印 dataset_version 到报告 header

能否用「更多简单题」把平均 faithfulness 刷高？

可以，但掩盖回归。应报告分层指标与难例子集，并在门禁中对 hard_case 单独设阈值。

动手练习

设计 S1 评测集分层比例并写 2 条 JSONL 样例（含 1 条拒答）。

操作步骤

  1. 按正文分层表填写你们 S1 的五层比例（可调整百分比）
  2. 写一条拒答 gold：requires_refusal=true
  3. 说明 version pin 在 CI 报告里如何打印
  4. 可选：demos/basics-02-dataset-lab/ 编辑 eval-sample.jsonl

判断练习

  1. 发版前只跑 10 条样例手工点检，不做回归集与 CI 门禁。在 CorpAssist 评测体系设计 场景下风险是什么？

    参考答案

    无法发现 Prompt/索引变更导致的回归；难例与拒答边界未覆盖；不符合 NS1 飞轮。应固定 dataset_version + 阈值门禁。
