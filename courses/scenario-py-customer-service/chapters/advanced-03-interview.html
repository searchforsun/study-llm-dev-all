智能客服系统我通常会划分为五个核心模块，每个模块解决一个独立的问题域。第一是意图路由模块——它负责接收用户消息、识别意图、抽取实体、输出置信度分数。我们用 FastAPI + ONNX 分类器实现，叠加了一层规则引擎处理 65% 的高频确定性场景（如「查工单」直接命中正则），LLM 分类器处理剩余的 35% 模糊场景（如「那件事怎么样了」需要语义理解）。第二是对话引擎模块——它使用 LangGraph 的 StateGraph 维护多轮对话状态。状态定义通过 TypedDict 实现，包含 messages、intent、confidence、tool_calls、response 五个字段。条件边根据当前 intent 路由到不同的处理节点，LLM 节点和工具节点交替执行实现 Think→Act→Observe→Respond 循环。Redis Checkpoint 持久化每次状态变更，支持断点恢复。第三是工具层模块——它将后端 API 封装为 LLM 可调用的 Tool。我们用 LangChain 的 @tool 装饰器自动生成 JSON Schema，Pydantic v2 做参数校验，Redis SET NX 做幂等性控制，三级降级（重试→简化→人工）将工具成功率从 92% 提升到 99.2%。第四是人工坐席模块——接收 AI 转接的对话，通过 Redis 队列分配坐席。第五是质检模块——自动评测（LLM-as-Judge）和人工抽检（Rubric + Cohen's Kappa）结合评估对话质量。这五个模块间通过 Pydantic v2 定义的接口契约通信（IntentResult、DialogState、ToolCall、ToolResult、TransferRequest），职责清晰可以独立部署和扩缩容。

多轮对话的核心挑战有三个：状态存储、指代消解、上下文窗口管理。对于状态存储，我使用 LangGraph 的 Checkpoint 机制结合 Redis——每次 StateGraph 节点执行后自动保存完整状态快照，P99 读取延迟 <50ms，写入 P99 <35ms。状态包含 messages、intent、collected_params 等全部字段，支持断线重连后恢复对话。没有选择自定义状态机而是 LangGraph，因为 LangGraph 用声明式图模型减少了 60% 的状态管理代码，条件边是纯函数可以轻松测试，而且 stream_events 天然支持流式输出。对于指代消解，我设计了三个层次：第一层规则匹配（正则 + 最近实体列表，覆盖 60% 的简单指代如「这个工单」→ TKT-001，零 token 成本）；第二层 Prompt 消解（LLM 根据上下文推断「它」指代什么，每次消耗约 200 token，覆盖 30% 的模糊指代）；第三层缓存（消解结果写入 Redis Hash，下次同类型指代直接命中）。总覆盖率达 96%。对于上下文窗口管理，我使用滑动窗口（保留最近 4 轮精确消息）+ LLM 摘要（每 4 轮生成一次 80 字以内的摘要）的双重压缩。在 1000 条真实客服对话上的测试表明：平衡模式下 token 消耗降低 60%，CSAT 评分仅下降 0.12（p=0.08，无显著差异）。四层混合记忆架构（工作记忆/会话记忆/用户画像/知识库）总面积约 15KB，在 10 轮以内对话中能保留 95% 的关键信息。

# 四级降级检测（按优先级）
class DegradeDetector:
    def detect(self, state) -> Optional[DegradeDecision]:
        # Grade 1: 敏感意图 - 立即高优转人工
        if state.intent in ("complaint",) or any(kw in state.last_msg for kw in ("投诉", "起诉")):
            return DegradeDecision(grade=1, priority="high")
        # Grade 2: 工具连续失败 3 次 - 正常转人工

        if state.step_count >= 8:
            return DegradeDecision(grade=3, priority="normal")
        # Grade 4: 置信度 < 0.5 - 先反问再转人工
        if state.confidence < 0.50:
            return DegradeDecision(grade=4, priority="low")
        return None

降级策略的优先级设计遵循「安全第一」原则——可能造成法律或业务风险的场景必须最高优转人工。我设计了四级降级触发：Grade 1 是敏感意图（如投诉、法律关键词），检测到后立即走 Redis 高优队列，5 秒内有坐席接单。Grade 2 是工具连续失败——同一个工具连续 3 次都失败，说明第三方 API 可能挂了，继续重试没有意义，正常转人工并生成工单。Grade 3 是超过最大步数（MAX_STEPS=8）——对话超过 8 步还没有解决，可能是问题太复杂或 LLM 在绕圈子，正常转人工。Grade 4 是置信度低于 0.5——这时系统先反问澄清一轮（给您几个选项确认一下），如果用户确认后置信度仍然 <0.5，再转人工。坐席分配使用 Redis 三级队列（高优/普通/低优），分配逻辑是 Round-Robin 轮询 + 技能匹配（投诉意图必须由投诉组坐席接）+ 负载均衡（选当前会话数最少的坐席）。上下文移交通过 ContextPackage 序列化数据包（约 4.2KB），包含对话摘要（LLM 压缩至 200 token）、用户画像、已执行操作和待办事项，坐席接手后 0 门槛继续服务。CorpAssist 的总体转接率 28%（目标 ≤30%），P50 分配延迟 500ms，P99 转接延迟 15.8s。

面试官想考察的是「RAG 落地经验」和「输出审核的工程实现」，而不仅仅是「用 RAG 就好了」。回答需要包含分块策略、向量检索、模板引擎、四层审核的具体实现和效果数据。

❌ 错误做法：只用 LLM 预训练知识回答

直接让 LLM 根据训练数据回答政策问题——GPT-4o 的训练数据截止 2025 年底，但公司政策每月都在更新（如 2026 年 3 月更新了退款政策）。LLM 按照旧政策回答会导致合规事件率高达 5.2%。

✅ 正确做法：RAG + 四层审核 + 模板兜底

政策问题必须搜索最新政策文档才回答。Retrieval（120 文档 8500 chunks）→ Augmented（Prompt 严格约束「找不到就说无法回答」）→ Generate→ Audit（四层审核）。方案上线后合规事件率降到 0.3%。

评估智能客服效果不能只看 CSAT，我设计了三层指标体系。第一层是结果指标——CSAT（用户满意度，目标 ≥4.0/5）、解决率（问题是否被解决，目标 ≥70%）、转人工率（AI 覆盖范围，目标 ≤30%）。第二层是质量指标——准确率（回答是否正确，目标 ≥85%）、完成率（多步任务是否全部完成，目标 ≥90%）。第三层是效率指标——平均处理时长（目标 <3min）、token 消耗（LLM 成本）。这些指标之间有联动（转人工率与 CSAT 的 r=-0.73），所以用综合评分 = Σ(指标得分 × 场景权重) 来整体评估。自动评测方面，我使用 LLM-as-Judge Pipeline——每天凌晨批量处理前一天的全部对话，让 GPT-4o-mini 从解决率、准确率、完成率三个维度评分（0-5 分），低分对话自动收录到坏例库。人工抽检方面，每天抽取 50 条对话让标注员按照六维度 Rubric 评分（1-5 分标尺），两名标注员的 Cohen's Kappa ≥0.75 才算合格。A/B 实验关键参数是每组最小 500 样本、p<0.05 判定有效。持续提升的闭环是「坏例库→根因分析→修改→回归测试→上线→验证」的迭代循环，每两周一个 Sprint，每次修复 20 条坏例。经过 24 个 Sprint（1 年）的迭代，综合评分从 3.2 提升到 4.1。

面试可能问（追问篇）

追问 Q1: 你说意图路由用 FastAPI + ONNX，ONNX 模型怎么训练的？更新频率是？

ONNX 模型是从 BERT 微调后的 checkpoint 导出的。先收集历史对话标注数据（约 15000 条），用 Hugging Face Transformers 微调一个 bert-base-chinese 分类器（8 个意图类），训练 3 个 epoch 后保存为 PyTorch checkpoint。然后用 `torch.onnx.export()` 导出为 ONNX 格式（支持动态轴处理变长输入），通过 ONNX Runtime 的 C 接口做 CPU 推理。导出时做了 INT8 量化，模型大小从 420MB 降到 110MB，推理速度提升 3 倍。更新频率是每 2 周一次——增量标注数据约 2000 条/两周，增量训练 1 个 epoch 后重新导出。每次更新在 staging 环境通过 Canary 部署先验证 1 小时（100 条请求），准确率不低于当前版本的 90% 才全量上线。当规则引擎覆盖率达到 70% 以上时，ONNX 模型的使用率会自然下降——因为大部分确定性请求被规则引擎拦截了，ONNX 只处理模糊意图。

追问 Q2: 你说转接率 28%，其中 Grade 4 置信度低占了 20%。这 20% 中有多少其实是能把阈值调低就能解决的？

这是一个非常好的问题。我们做过阈值敏感性分析：如果将 Grade 4 的置信度阈值从 0.5 降到 0.4，转接率会从 20% 降到 14%（释放 6%），但代价是准确率会从 86.7% 降到 82.1%（降 4.6%）。这是因为 0.4-0.5 区间内的分类虽然「有点把握」但仍有 18% 的出错概率——那 18% 的错误会导致更大的问题（用户得到错误回答后重复提问或不信任系统）。所以我们选择维持 0.5 阈值，将这 20% 中确实可以 AI 处理的优化为「反问澄清」而不是「直接转人工」——在转人工前加一轮系统反问让用户确认意图（「您是想查工单还是退款？」），这轮反问 + 用户确认后有效意图识别率提高到 85%。所以实际需要转人工的比例从 20% 降到了 20% × (1-0.85) = 3%。这个优化依赖于我们有一个低成本的本地小模型做反问判断（使用 Qwen2.5-1.5B 本地部署，零 API 成本，延迟 200ms）。

追问 Q3: 你说了很多成功数据，有没有失败的经验？

有一个刻骨铭心的失败。上线第二个月，我们调优了 Prompt 想让回答更友好——在系统 Prompt 里加了「请用热情的语气回复」。结果 LLM 对投诉用户也用「亲爱哒，您的问题超级简单呢～」的语气，导致投诉场景的 CSAT 从 3.8 暴跌到 2.5。核心教训是 Prompt 修改必须分场景 —— 投诉场景应该用「专业、冷静、共情」语气而不是「热情」语气。从那以后我们给每种意图类型配了独立的 Prompt 模板（共 8 套），每套 Prompt 都有对应的语气定义。第二个教训是「监控改了 Prompt 的 CSAT 变化」——现在我们每次 Prompt 修改后都会运行 A/B 实验，至少看 3 天的数据才做决策。这个失败也促使我们建立了 Prompt 版本管理制度——每次修改需要提交 PR 并附带预期的 CSAT 变化范围，审批通过后才能部署。

面试加分技巧：如何用数据增强回答可信度

在智能客服系统的面试中，区分「看过文章」和「做过项目」的关键是数据。没有数据的回答是「我觉得应该这样做」，有数据的回答是「我测量过这样做效果提升了 X%」。以下是 CorpAssist 团队在面试中总结的数据融入技巧和常见陷阱。

常见陷阱：第一个陷阱是「过度承诺」——面试中不要说「我们覆盖了所有场景」，而要说「我们定义了四类场景（售前/售后/投诉/查询），每类有差异化的 KPI 权重，投诉场景的 CSAT 目前比人工低 0.4 分，正在优化中」。承认不足并给出改进方向比完美主义更可信。第二个陷阱是「背数据但不知道来源」——每个数据必须能说出采集方式，比如「转人工率 28% 来自 Redis 的 atomic counter，通过 Grafana 每小时聚合一次」。第三个陷阱是「忽略比较基准」——「解决率 85%」单独看很好，但加上「从 70% 提升到 85%，分 4 个月 4 个主题优化」更有说服力。面试官要的不是「结果」而是「过程」。

判断练习

如果面试官问「你知道 LangGraph 最新版有什么新功能吗？」——你不知道确切的最新版，怎么回答？

诚实是最好的策略——说「目前我使用的是 LangGraph 0.2.x 版本，最新版的新功能我还没有在生产中使用，所以无法给出准确的判断。不过从我会的角度来看，LangGraph 的核心能力——StateGraph、Checkpoint、stream_events——在 0.2.x 中已经非常成熟了。」然后补充一个你确定的知识点「但我特别关注 LangGraph 的状态持久化能力，我们生产环境实现了自定义 RedisCheckpointSaver，P99 读取延迟 <50ms。」这样既诚实又展示了你的深度实践。为了体现成长性，可以加一句「我会在面试后研究最新版的 Release Notes，尤其是异步流式方面的新特性——这对我们 SSE 响应的优化有帮助。」面试官看重的是你知道什么，而不是你不知道什么——展示你已经掌握的核心能力和持续学习的态度。

面试官说「你说你的系统 CSAT 4.2，那为什么某某公司的方案 CSAT 4.5？你们差在哪？」——怎么回答？

不直接比较 CSAT 绝对值（不同公司/不同场景/不同用户群的 CSAT 不可比），而是解释可比性问题和你的改进方向。回答：「不同公司的 CSAT 不能直接比较，因为评分标准、用户群体、问题难度分布都不同。比如 CorpAssist 面对的是企业内部员工——他们的问题包含复杂的投诉和政策咨询，如果只对比简单问答场景，我们的 CSAT 可以到 4.5。相比之下，某某公司的方案可能处理的是外部零售客服（简单查询为主），问题难度分布不同。有意义的对比是在我们自己的系统中做 A/B 实验——当前我们正在测试新版本的投诉处理 Prompt，A/B 实验显示 CSAT 从 4.2 提升到 4.35（p=0.04），预计下个月全量上线。如果能达到 4.4 以上，我们会研究将这套 Prompt 泛化到其他场景。」关键点：不防御、不比较、转向自己的改进方向。

面试常见错误与应对策略

即使技术功底扎实，面试中还是容易犯一些非技术性错误。CorpAssist 团队在面试了 30+ 候选人后总结出最常见的三类错误和应对策略。

❌ 错误 1：过于简略

回答只用 2 句话：「用了 LangGraph 做 Agent，Redis 做缓存，效果还不错。」面试官无法判断你是真做过还是看过文章。改进：使用「问题→方案→数据→对比」四段式结构，每段 1-2 句话。

✅ 正确 1：结构完整

「多轮对话用 LangGraph StateGraph 管理（方案），状态通过 Redis Checkpoint 持久化（实现），P99 读取 50ms（数据），比自定义状态机少 60% 代码（对比）。」4 句话展示了对技术栈的深度理解。

❌ 错误 2：只报喜不报忧

「我们系统很完美，没遇到什么问题。」面试官会产生「这人要么在吹牛要么没做过生产系统」的印象。每个生产系统都有坑，关键是坑里学到了什么。

✅ 正确 2：坦诚分享教训

「上线第一个月遇到 LLM 操作幻觉——AI 说已退款但实际没调 API。后来我们加了操作幻觉检测器，每天拦截约 15 次（发生率 0.4%）。」既诚实又展示了解决问题的能力。

❌ 错误 3：被追问时卡住

「问：你们的缓解方案有没有引入新问题？答：呃…让我想想…」准备不充分的回答在压力下会崩溃。

✅ 正确 3：提前准备跟进

「有的——缓存策略引入了一个新问题：政策更新后用户仍能命中旧政策的缓存。我们加了一层版本号校验——缓存 key 中带上 policy_version，版本不匹配就重新检索。」这说明不仅知道方案，还知道方案的副作用。

最常见的 Scale 追问：面试官常问「如果 QPS 再涨 10 倍你怎么应对？」回答框架：「第一看当前瓶颈（LLM API 限流），第二分层解决（请求层加缓存+降级层用小模型兜底+基础设施层水平扩容），第三量化（每层能支撑的 QPS 上限）。」展示「系统性思维」——不是给点状答案而是给分层解决的框架。CorpAssist 的数据：日 3000 对话（峰值 80 QPS）→ 10 倍到 30000 对话（峰值 800 QPS）时，Redis 从单实例升级到 Cluster（6 分片），PostgreSQL 从单表改按月分区，坐席从 5 人增加到 15 人，LLM API Key 从 1 个加到 5 个轮询。

面试回答的代码模式速查

以下是在面试回答中可以自然引用的 6 个 Python 代码模式。不需要完整写出代码，而是用 1-2 句话描述模式的关键点，展示你对实现的深入理解。面试官往往对「你用什么数据结构解决什么问题」最感兴趣。

python
# 面试中可引用的 6 个代码模式

# 1. LangGraph TypedDict 状态定义
class AgentState(TypedDict):
    messages: Annotated[list, "历史消息"]
    intent: Optional[str]
    confidence: float
    tool_calls: list
    response: str
# 用 TypeDict 而非 BaseModel：因为 TypedDict 不可变，LangGraph 内部用浅拷贝

# 2. Redis SET NX 幂等性锁
redis.set(f"lock:{key}", "1", nx=True, ex=30)
# 原子操作 + TTL 自动释放，分布式场景的唯一互斥方案

# 3. Pydantic field_validator 错误消息
@field_validator('amount')
@classmethod
def amount_range(cls, v):  # LLM 读错误消息后自动修正参数
    if v <= 0: raise ValueError(f"金额 {v} 必须大于 0")

# 4. FastAPI StreamingResponse SSE
async def event_generator(): yield f"data: {chunk}\n\n"
return StreamingResponse(event_generator(), media_type="text/event-stream")
# 关键配置：response/media_type + X-Accel-Buffering no

# 5. Redis List 作为优先级队列
redis.rpush(f"queue:{priority}", context_json)
redis.blpop(f"queue:{priority}", timeout=30)
# rpush 入队 + blpop 阻塞出队，天然支持消费者模型

# 6. Cohen's Kappa 标注一致性
from sklearn.metrics import cohen_kappa_score
kappa = cohen_kappa_score(annotator_a, annotator_b)
# kappa > 0.75 合格，< 0.4 重新校准标注标准

在面试中引用这些代码模式的方式不是背代码，而是像这样说：「我们在状态管理上用了 LangGraph 的 TypedDict，它比 BaseModel 好在浅拷贝性能更优——因为我们每轮都要 copy 整个状态，TypedDict 的 copy 比 BaseModel 快 3 倍。」面试官听到这样的细节会立刻判断出你有真实项目经验。另一个技巧是引用你解决过的具体 Bug：「幂等性这块我们踩过一个坑——Redis SET NX 没有设置 ex 参数，结果锁永远不释放。所以我们后来强制所有 SET NX 都必须带 ex 参数并写在代码规范里。」这种坦诚的 Bug 经验在面试中非常加分。最强的面试回答是这样组织的：「我们用了 Redis SET NX 做幂等控制（技术）-> 拦截了 3.2% 的重复请求（数据）-> 这也带来了一个坑——最初忘记加 TTL 导致死锁（问题）-> 后来强制要求所有 SET NX 带 ex 参数（解决）-> 代码审查中也增加了这个 check 项（制度化）。」——这个回答展示了从技术到数据到问题到制度化改进的完整闭环，是面试官最想看到的系统性思维。最后补充一句：「如果你对这套系统感兴趣，我可以随时深入某个模块的细节——比如 LangGraph 的 Checkpoint 怎么和 Redis 结合。」这会引导面试官问出你最有准备的问题。
