python
"""模块间接口定义"""

from pydantic import BaseModel
from typing import Optional, List

class IntentResult(BaseModel):
    """意图路由模块输出"""
    intent: str  # query_ticket, reset_password, etc.
    confidence: float
    entities: dict  # {ticket_id: "TKT-001"}
    raw_text: str

class DialogState(BaseModel):
    """对话引擎状态"""
    session_id: str
    user_id: str
    current_intent: Optional[str] = None
    collected_params: dict = {}
    history: List[dict] = []
    turn_count: int = 0

class ToolCall(BaseModel):
    """工具调用请求"""
    tool_name: str
    params: dict
    timeout_ms: int = 5000

class ToolResult(BaseModel):
    """工具调用结果"""
    success: bool
    data: dict
    error: Optional[str] = None

class TransferRequest(BaseModel):
    """转人工请求"""
    session_id: str
    user_info: dict
    dialog_history: str  # 格式化后的对话历史
    reason: str  # 转接原因
    priority: str = "normal"  # high / normal / low

与微服务架构的对应关系

智能客服的五模块设计天然对应微服务架构的经典模式。

客服模块对应微服务模式说明
意图路由API Gateway + 内容路由统一入口，按意图分发给不同处理流
对话引擎有状态编排服务 + Saga维护对话状态，长事务管理
工具层后端服务 + 防腐层封装异构后端的 API，提供统一接口
人工坐席事件驱动 + WebSocket双向实时通信
质检模块监控 + 异步分析离线分析对话质量

理解这种对应关系的价值在于：微服务的成熟模式（限流/熔断/重试/幂等）可以直接套用到客服模块中。例如，工具层实现了重试和熔断——当第三方工单系统响应时间超过 3s，工具层自动熔断并返回「系统繁忙，请稍后重试」，避免级联故障。

面试可能问

Q1: 对话引擎为什么用 LangGraph 而不是单纯的 LCEL？

因为客服对话不是线性管线——它有分支和条件边。LangGraph 的 StateGraph 可以定义状态节点和条件边，根据意图不同路由到不同处理节点。比如用户说「查工单」和「重置密码」走不同的工具调用链路。LCEL 适合「检索→生成」的线性流程，无法表达这种分支逻辑。LangGraph 的有状态图模型天然匹配客服的多轮场景。

Q2: 工具层怎么保证调用第三方系统时的幂等性？

每个工具调用生成全局唯一的 request_id，第三方系统在收到重复 request_id 时跳过执行返回上次结果。如果第三方不支持幂等，工具层在本地维护「已执行记录」缓存（TTL=24h），同一 request_id 的重复请求直接返回缓存结果。对于查询类操作，天然幂等；对于修改类操作（如重置密码），幂等性至关重要。

判断练习

如果意图路由和对话引擎合并为一个服务，有什么好处和坏处？

好处：减少一次服务间调用（省 20-50ms），部署运维简化。坏处：内部耦合，无法独立扩缩容——意图路由是轻计算（分类器），对话引擎是重计算（LLM 生成），合并后资源浪费。高流量场景下，意图路由需要较多的 CPU 实例，对话引擎需要 GPU 实例，分开部署更能精准分配资源。

如果用户连续 3 次意图识别失败，系统应该怎么做？

策略是「降级：反问 → 转人工」。第 1 次失败：反问澄清「请问您是想查工单、重置密码还是咨询制度？」第 2 次再失败：给出示例提示「您可以试试说：查工单 TKT-001」。第 3 次还失败：自动转人工，附带对话历史让客服了解上下文。这个策略将意图识别失败场景的转人工率从 60% 降到 15%。
