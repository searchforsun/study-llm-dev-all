五大模块的职责边界与技术选型

模块间的完整数据流与延迟分解

"""模块间接口契约：Pydantic v2 模型定义"""
from pydantic import BaseModel, Field, field_validator
from typing import Optional, List, Dict, Any
from datetime import datetime

class IntentResult(BaseModel):
    """意图路由模块输出"""
    intent: str                                  # query_ticket, reset_password
    confidence: float = Field(ge=0.0, le=1.0)
    entities: Dict[str, str] = {}                # {ticket_id: "TKT-001"}
    raw_text: str
    processing_time_ms: int                      # ONNX 推理耗时

    @field_validator('confidence')
    @classmethod
    def confidence_valid(cls, v: float) -> float:
        if v < 0.3 and v > 0.0:
            raise ValueError(f'低置信度 {v:.2f} 需特殊标记')
        return v

class DialogState(BaseModel):
    """对话引擎持久化状态"""
    session_id: str
    user_id: str
    current_intent: Optional[str] = None
    collected_params: Dict[str, Any] = {}
    history: List[Dict[str, Any]] = []
    turn_count: int = 0
    created_at: datetime = Field(default_factory=datetime.now)
    updated_at: datetime = Field(default_factory=datetime.now)

class ToolCall(BaseModel):
    """工具调用请求体"""
    tool_name: str
    params: Dict[str, Any]
    idempotency_key: str                         # 幂等性 Token
    timeout_ms: int = Field(default=5000, le=10000)

class ToolResult(BaseModel):
    """工具调用结果"""
    success: bool
    data: Dict[str, Any] = {}
    error: Optional[str] = None
    execution_time_ms: int

class TransferRequest(BaseModel):
    """转人工请求"""
    session_id: str
    user_info: Dict[str, Any]                    # {user_id, name, role, department}
    dialog_history: str                          # 格式化的对话历史摘要
    current_intent: str
    executed_actions: List[str]                  # [query_ticket, check_refund]
    reason: str                                  # 转接原因
    priority: str = "normal"                     # high / normal / low

延迟预算分解：CorpAssist 对 End-to-End P99 延迟的目标是 <3s。实际拆解到每个模块的延迟预算为——意图路由 ≤200ms（ONNX 推理 50ms + 网络 150ms）、对话引擎状态加载 ≤50ms（Redis 读取 5ms + 序列化 45ms）、工具调用 ≤1.5s（HTTP 调用 1.2s + 校验 300ms）、LLM 生成 ≤1.0s（GPT-4o-mini 典型值 800ms）、网络传输 ≤250ms。当监控发现某个模块的 P99 超过预算的 1.5 倍时，自动触发告警并缩放到更大规格实例。

FastAPI + Redis + Celery 的技术选型理由

选择 FastAPI + Redis + Celery 作为 CorpAssist 智能客服的技术底座，是基于实时性、状态管理和异步任务三个维度的对比决策。以下是从 8 种候选方案中筛选后的最终选型及其核心理由。

场景候选方案选型决定原因（含对比数据）
API 框架FastAPI / Flask / DjangoFastAPIFastAPI 异步原生支持 SSE（对比 Flask P99 高 45%），Pydantic v2 集成减少 30% 验证代码量，OpenAPI 文档自动生成
状态缓存Redis / Memcached / 本地内存RedisRedis 支持 5 种数据结构（String/List/Hash/Set/SortedSet），适合会话状态的 Hash 存储 + 排队队列的 List + 幂等性 Token 的 String。Memcached 只支持 KV，不满足需求
异步任务Celery / Dramatiq / ArqCeleryCelery 生态成熟（Redis Broker + PostgreSQL Backend），任务重试/定时/优先级/Rate Limit 内置。对比 Dramatiq，Celery 的社区包数和文档量约为 5:1
SSE 流式输出FastAPI SSE / WebSocket / PollingFastAPI SSESSE 单向流比 WebSocket 节省 30% 连接开销，浏览器原生支持 EventSource API。对比 Polling，SSE 减少 90% 的无效请求
LLM 框架LangChain / LlamaIndex / 裸 APILangChain + LangGraphLangGraph 的状态图模型比裸 API 的自定义状态机减少 60% 的状态管理代码。LCEL 的流式支持是默认的，无需额外开发

"""FastAPI + Celery 的智能客服异步架构"""
from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
from celery import Celery
from redis import Redis
from pydantic import BaseModel
from typing import AsyncGenerator
import json, asyncio

app = FastAPI(title="CorpAssist 智能客服")
redis_client = Redis(host="redis.internal", port=6379, decode_responses=True)
celery_app = Celery("corp_assist", broker="redis://redis.internal:6379/1")

class DialogRequest(BaseModel):
    session_id: str
    user_id: str
    message: str

@celery_app.task(bind=True, max_retries=3, default_retry_delay=5)
def async_log_dialog(self, session_id: str, dialog_data: dict):
    """异步写入对话日志，不阻塞主流程"""
    try:
        redis_client.rpush(f"log:{session_id}", json.dumps(dialog_data))
    except Exception as exc:
        raise self.retry(exc=exc, countdown=30)

@app.post("/dialog/stream", response_class=StreamingResponse)
async def stream_dialog(req: DialogRequest) -> StreamingResponse:
    """SSE 流式对话"""
    async def event_generator() -> AsyncGenerator[str, None]:
        # 1. 从 Redis 加载会话状态
        state = redis_client.get(f"session:{req.session_id}")
        yield f"data: {json.dumps({'type': 'status', 'msg': '加载状态中'})}\n\n"

        # 2. 意图识别（调用 LLM 分类器）
        intent = await classify_intent(req.message)
        yield f"data: {json.dumps({'type': 'intent', 'intent': intent})}\n\n"

        # 3. 流式生成回复
        async for chunk in generate_response(req.session_id, req.message, intent):
            yield f"data: {json.dumps({'type': 'token', 'text': chunk})}\n\n"

        # 4. 异步写日志（不阻塞）
        async_log_dialog.delay(req.session_id, {"message": req.message, "intent": intent})

        yield "data: [DONE]\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no"
        }
    )

async def classify_intent(text: str) -> str:

CorpAssist 客服系统的完整架构拓扑

CorpAssist 的完整架构采用 Python 对话引擎 + Java 管理端双栈设计。Python 栈负责高实时性的对话处理，Java 栈负责管理面功能（坐席工作台、报表、配置）。两栈之间通过内部 HTTP API 和 Redis 事件通道进行数据交换。这种双栈架构的核心理念是「Java 做擅长的事（复杂事务、权限管理），Python 做擅长的事（AI/LLM、异步 I/O）」。

网络拓扑说明：Python 服务与 Java 服务之间通过内部 VPC 通信，不经过公网。Python 服务通过 Redis Pub/Sub 发送坐席分配事件，Java 坐席工作台订阅该通道实时接收。所有外部 API 调用（LLM、工单系统）通过工具层统一出口，既便于审计日志记录，也方便流量控制和熔断配置。Kong API Gateway 负责全系统的路由、限流和 JWT 鉴权卸载，将 Auth 压力从应用层剥离。

❌ 单栈架构缺陷

所有逻辑塞入一个 Spring Boot 应用：意图识别用 Java 调用 Python 脚本（序列化开销每次 200ms+），LLM 调用阻塞 Servlet 线程池（Tomcat 200 线程很快耗尽），每个对话的上下文存在内存 Map 中——重启即丢失。

✅ 双栈架构优势

Python 栈异步非阻塞处理 LLM/IO 密集型任务，Java 栈处理事务和管理面。对比单栈方案：P99 延迟从 4.2s 降到 2.1s，开发效率 Python 侧提升 40%，故障隔离（Python 栈崩溃不影响 Java 管理端）。

技术选型决策树：从需求到框架的映射

智能客服的技术选型不是「流行什么用什么」，而是根据实时性、状态性、计算量三个维度做决策。以下决策树可以帮助架构师在 8 种常见场景中快速做出选择。核心原则是「高实时 + 有状态选 in-memory + 异步复制，低实时 + 无状态选 Serverless + 托管服务」。

python
"""技术选型决策引擎"""
from dataclasses import dataclass
from enum import Enum

class RealTimeLevel(Enum):
    HIGH = "real_time"       # P99 < 500ms
    MEDIUM = "near_real"     # P99 < 3s
    LOW = "batch"            # 批量处理

class StateRequirement(Enum):
    STATELESS = "stateless"  # 无状态，可任意扩缩容
    SESSION = "session"      # 会话级状态
    PERSISTENT = "persist"   # 持久化状态

class ComputeIntensity(Enum):
    LIGHT = "light"          # CPU 密集型 < 100ms
    MEDIUM = "medium"        # GPU 推理 < 1s
    HEAVY = "heavy"          # LLM 生成 > 1s

@dataclass
class TechDecision:
    real_time: RealTimeLevel
    state: StateRequirement
    compute: ComputeIntensity

    def recommend(self) -> dict:
        recs = {
            "framework": "",
            "deploy": "",
            "cache": "",
            "async_task": ""
        }
        if self.real_time == RealTimeLevel.HIGH and self.state == StateRequirement.STATELESS:
            recs["framework"] = "FastAPI (异步)"
            recs["deploy"] = "K8s + HPA (CPU >70%)"
            recs["cache"] = "Redis String"
            recs["async_task"] = "不需要"
        elif self.state == StateRequirement.SESSION:
            recs["framework"] = "LangGraph StateGraph"
            recs["deploy"] = "K8s + StatefulSet"
            recs["cache"] = "Redis Hash (session:{id})"
            recs["async_task"] = "Celery (日志/质检)"
        else:
            recs["framework"] = "Celery + FastAPI"
            recs["deploy"] = "K8s + KEDA (队列长度)"
            recs["cache"] = "Redis List (任务队列)"
            recs["async_task"] = "Celery Worker × N"
        return recs

# 对话引擎场景
decision = TechDecision(
    real_time=RealTimeLevel.MEDIUM,
    state=StateRequirement.SESSION,
    compute=ComputeIntensity.HEAVY
)
print(decision.recommend())

CorpAssist 架构场景

某次生产事故：工具层调用工单系统 API 超时（响应时间从 800ms 升到 5s），导致 FastAPI 工作线程被占满，意图路由也受到影响——因为所有请求共享同一个线程池。根因是工具层没有独立的线程池隔离。修复方案：给工具层分配独立的 ThreadPoolExecutor（max_workers=10），并添加熔断器（错误率 >30% 时熔断 30s）。修复后，即使工具层故障，意图路由和对话引擎仍可以正常服务，并将需要工具调用的对话标记为「暂时无法处理」并缓存到 Redis。

模块间通信协议选择：gRPC vs HTTP vs Redis Pub/Sub

五个模块间的通信方式选择直接影响系统的延迟、可靠性和可维护性。CorpAssist 采用了混合通信策略——不同场景用不同协议：同步高实时走 gRPC（意图路由→对话引擎），异步非关键走 HTTP（对话引擎→质检），事件通知走 Redis Pub/Sub（坐席分配通知）。以下是每种协议的选择理由。

通信场景协议延迟理由替代方案
意图路由 → 对话引擎gRPC< 5ms高实时性（P99 目标 <10ms），protobuf 序列化比 JSON 快 4 倍，支持双向流HTTP/2 REST（延迟多 30ms）
对话引擎 → 工具层HTTP REST< 50ms工具层调用时间主要花在第三方 API（800ms+），通信协议差异可忽略gRPC 收益不大
转人工通知Redis Pub/Sub< 10ms发布-订阅模式天然适合事件通知，Python 和 Java 都可订阅同一通道RabbitMQ（太重）
对话日志写入Celery 异步消息~50ms 入队写入不阻塞主流程，批量写入比实时 HTTP 效率高 10 倍Kafka（当前规模不需要）
服务健康检查HTTP /health< 5msK8s liveness/readiness probe 标准方案，配合 readiness 做滚动更新gRPC health protocol

协议选择原则：核心对话链路的模块间通信追求最低延迟——所以意图路由到对话引擎用 gRPC（节省约 30ms 序列化时间）。非核心链路（日志/质检）用异步消息，保证主流程不受影响。事件通知用 Redis Pub/Sub 因为 Python 和 Java 端都在使用 Redis，不需要引入消息中间件。选择合适的通信协议可以将端到端 P99 延迟降低约 15%，而且故障隔离效果更好——gRPC 调用超时会快速失败（fail-fast），而 HTTP 长连接挂起可能导致资源泄露。

本章结论

  • 五大模块（意图路由/对话引擎/工具层/人工坐席/质检）的职责边界清晰，每个模块有独立的 P99 延迟目标——意图路由 <200ms、工具层 <1.0s、对话引擎 <1.5s
  • 数据流设计遵循「同步调用为主 + 异步事件为辅」，核心链路走 HTTP/gRPC 保证实时性，日志/质检走 Celery 异步处理
  • FastAPI + Redis + Celery 的技术组合在异步 SSE 流式响应、Hash 结构会话状态管理、异步任务重试三方面提供完整能力，对比 Flask + 内存方案性能提升 45%
  • 双栈架构（Python 对话 + Java 管理）的 P99 延迟比纯 Java 方案降低 50%（4.2s → 2.1s），故障隔离更彻底
  • 技术选型决策应按实时性/状态性/计算量三维度判断，避免「流行什么用什么」导致过度设计

面试可能问

Q1: 对话引擎为什么用 LangGraph 而不是单纯的 LCEL LangChain Expression Language？

客服对话不是线性管线——它有分支和条件边。一个用户说「查工单」需要走工具调用链路，说「重置密码」需要走身份验证链路，说「转人工」需要立即中断当前流程。LCEL 适合「A→B→C→D」的线性流程（如 RAG 的检索→生成），但无法表达这种分支逻辑。LangGraph 的 StateGraph 可以定义多个节点（Node）和条件边（Conditional Edge），根据当前意图状态路由到不同的下一节点。比如我们定义了一个路由函数 `route_by_intent(state)`，当 state.intent == "query_ticket" 时路由到 `ticket_tool_node`，当 intent == "transfer_human" 时路由到 `handoff_node`。LangGraph 还内置了 Checkpoint 机制——每个步骤的状态都会自动保存到 Redis，支持对话的中断恢复和调试回放。在 CorpAssist 的实践中，LangGraph 比自定义状态机减少了 60% 的状态管理代码量。

Q2: 工具层怎么保证调用第三方系统时的幂等性和可靠性？

工具层的可靠性设计分三层。第一层是幂等性——每个工具调用生成全局唯一的 `idempotency_key`（格式：`{session_id}:{tool_name}:{uuid4}`），Redis 中存储该 key 的执行结果（TTL=24h）。当收到重复 Key 的请求时，直接返回缓存结果。第三方系统如果支持 `Idempotency-Key` Header 则优先用它，否则工具层自己做去重。第二层是超时和熔断——每个工具调用设置 timeout_ms 参数（默认 5000ms，最大 10000ms），当连续 5 次调用都超时时，熔断器打开，30 秒内直接返回「服务暂不可用」而不实际调用。第三层是三级降级——第一级重试（最多 2 次，指数退避），第二级简化查询（如果精确查询失败，尝试模糊搜索），第三级人工确认（如果前两级都失败，将问题标记为「需人工处理」并转接）。CorpAssist 线上数据显示，三级降级使工具调用成功率从 92% 提升到 99.2%。

Q3: 双栈架构中 Python 和 Java 之间如何通信？为什么不用纯 Python 方案？

Python 和 Java 之间的通信有两种途径：同步走 HTTP REST API，异步走 Redis Pub/Sub。Python 对话引擎通过 HTTP POST 将转人工请求发送到 Java 坐席工作台，Java 通过 Redis Pub/Sub 通道 `agent:status` 发布坐席上下线状态，Python 订阅该通道进行坐席分配。不用纯 Python 方案的原因有三：第一，Java 在事务管理方面生态更成熟——Spring Boot 的声明式事务 @Transactional、JPA 的乐观锁、Seata 的分布式事务在 Python 中缺乏等价实现；第二，企业现有后端基础设施（ERP/OA/HR 系统）通常提供 Java SDK，直接复用比 Python 重写成本低 60%；第三，管理端需要复杂的权限模型（RBAC+ABAC），Java Spring Security 的方案比 Python Casbin 更成熟，社区包更多。双栈并非追求统一技术栈，而是让每个技术栈做自己最擅长的事。

判断练习

如果意图路由的 P99 延迟从 200ms 飙升到 800ms，应该先排查什么？

优先排查 ONNX 分类器的推理耗时——在日志中按百分位聚合分类器延迟，看是平均升高还是少数极端值。如果是平均升高（200ms→800ms），检查推理服务是否 CPU 资源不足（看 K8s CPU Throttling 指标），也可能是部署了新版本的分类模型导致推理变慢。如果是极端值（P99 升高但 P50 正常），大概率是网络抖动或 GC 停顿——检查 Redis 连接池是否被耗尽（等待获取连接的时间过长）。在 CorpAssist 的实践中，有一次是 Redis 连接数达到 max_connections=100，新连接排队等待导致意图路由 P99 飙升。修复后把连接池从 100 扩到 500，同时给意图路由增加了连接数监控告警。如果是模型推理变慢，可以考虑将 ONNX Runtime 的 intra_op_num_threads 从 4 调到 2（减少 CPU 争抢），或升级到更高 CPU 规格的 Pod。

质检模块为什么设计成离线分析而不是实时打分？

质检评分需要 LLM-as-Judge——每次评分调用消耗约 500-1000 token，如果实时打分，意味着一张对话的每一次交互都要触发 LLM 调用，成本太高（日均 3000 对话 × 5 轮 × 800 token = 1200 万 token/日，GPT-4o-mini 约 12 元/日）。离线方案：对话结束后，Celery Worker 在队列中等待处理，批量调用 LLM-as-Judge，日均 token 消耗约 200 万（降低 83%）。而且离线分析可以做更复杂的处理——比如跨对话的用户 NPS 趋势分析、质检 Rubric 的多维度评分、坏例库的自动聚类。离线分析的延迟是 10-30 分钟，对于质检场景完全可以接受——质检不看实时，看趋势和异常检测。CorpAssist 的实践是质检报表每天 8:00 自动生成并推送到企业微信群，坐席团队据此改进服务质量。
