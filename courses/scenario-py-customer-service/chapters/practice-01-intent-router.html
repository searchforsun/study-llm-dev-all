为什么学这章

意图识别是智能客服系统的第一个也是最重要的决策点——用户说「我收不到快递」到底是物流查询还是投诉？分类错误会导致整个对话链路的错误。如果做成简单的 if-else 关键词匹配，准确率可能不到 60%；如果直接用 LLM 裸分类，延迟太高且成本不可控。CorpAssist 采用「规则引擎 + LLM 分类器」双引擎路由方案——规则处理高频确定性场景（命中率 65%，零延迟），LLM 处理模糊/新意图（延迟 1.2s，但成本可控）。本章将带你实现一个生产级的意图路由服务。

学完后能做什么

用 Pydantic + JSON Schema 实现 LLM 的结构化输出确保分类结果可解析；设计规则引擎 + LLM 双引擎路由方案，规则命中率 65% 时零延迟；设置合理的置信度阈值（0.85/0.5）并解释统计学依据；用混淆矩阵 + F1 Score 评估意图识别准确率，建立持续标注闭环。

本章要点速记

1) LLM 结构化输出通过 Pydantic model + with_structured_output 实现，确保分类返回 JSON 格式可解析，规避 LLM 输出格式不一致问题；2) 规则引擎负责 65% 的高频确定性场景（关键词 + 正则，延迟 <5ms），LLM 处理剩余的 35%（平均延迟 1.2s）；3) 置信度阈值 0.85（高置信度直接路由）和 0.5（低置信度反问澄清或转人工）的设置基于 5000 条测试样本的 Precision-Recall 曲线分析；4) 评估体系包含混淆矩阵（计算每类意图的 Precision/Recall/F1）、分层抽样确保低频率意图被充分标注、持续标注闭环每月 1000 条新增标注。

基于 LLM 的结构化输出：Pydantic + JSON Schema

传统的 LLM 意图分类需要写复杂的 Prompt 让模型输出指定格式的结果，但 LLM 经常不按格式输出——用中文代替了「intent」字段名，多出了未定义的键，甚至只输出了文字描述。LangChain 的 `with_structured_output` 方法解决了这个问题：它通过 Pydantic 模型定义输出结构，内部使用 JSON Schema 约束或者 Tool Calling 机制，保证 LLM 的输出一定是符合模型定义的 JSON。

"""基于 Pydantic 的结构化意图分类"""
from pydantic import BaseModel, Field, field_validator
from typing import Optional, List, Dict
from enum import Enum
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

class IntentType(str, Enum):
    QUERY_TICKET = "query_ticket"          # 查询工单
    RESET_PASSWORD = "reset_password"      # 重置密码
    APPLY_LEAVE = "apply_leave"            # 申请请假
    CHECK_REFUND = "check_refund"          # 查询退款
    COMPLAINT = "complaint"                # 投诉
    TRANSFER_HUMAN = "transfer_human"     # 转人工
    GREETING = "greeting"                  # 问候
    UNKNOWN = "unknown"                    # 未知

class IntentResult(BaseModel):
    """LLM 意图分类的结构化输出"""
    intent: IntentType
    confidence: float = Field(ge=0.0, le=1.0, description="分类置信度")
    entities: Dict[str, str] = Field(
        default_factory=dict,
        description="抽取的实体, 如 {ticket_id: TKT-001}"
    )
    reasoning: str = Field("", description="分类依据简述")

    @field_validator('confidence')
    @classmethod
    def confidence_check(cls, v: float) -> float:
        if v < 0.3:
            raise ValueError(f'置信度 {v:.2f} 过低')
        return v

class IntentClassifier:
    """结构化意图分类器"""

    def __init__(self, model_name: str = "gpt-4o-mini"):
        self.llm = ChatOpenAI(
            model=model_name,
            temperature=0,  # 分类任务用 0 温度保证确定性
        ).with_structured_output(IntentResult, method="function_calling")

        self.prompt = ChatPromptTemplate.from_messages([
            ("system", """你是一个企业客服意图分类器。
请将用户意图分类到以下类别之一：
- query_ticket: 查询工单/任务/审批进度
- reset_password: 重置密码/找回密码
- apply_leave: 申请请假/年假/事假
- check_refund: 查询退款/报销进度
- complaint: 投诉/不满/抱怨
- transfer_human: 明确要求转人工
- greeting: 问候/打招呼
- unknown: 以上都不属于

输出要求：
1. 只返回 JSON，不要多余说明
2. entities 字段提取关键实体（工单号、金额等）
3. confidence 反映你对该分类的确定程度"""),
            ("human", "{text}")
        ])

    async def classify(self, text: str) -> Optional[IntentResult]:
        """分类入口（含重试）"""
        for attempt in range(2):
            try:
                result = await self.llm.ainvoke(
                    self.prompt.format_messages(text=text)
                )
                if result.confidence > 0:
                    return result
            except Exception as e:
                if attempt == 1:
                    return None
                continue
        return None

规则引擎 + LLM 双引擎路由

纯 LLM 分类的问题不在于准确率（gpt-4o-mini 的 8 分类准确率约 93%），而在于成本和延迟。每一次 LLM 分类调用消耗约 200-300 token，延迟 0.8-1.5s。对于高频的确定性场景（如用户说「查工单 TKT-001」其中包含明显的关键词），用规则引擎处理可以做到零 token 成本和 <5ms 延迟。CorpAssist 的双引擎路由方案将 65% 的流量分流到规则引擎，显著降低了系统总成本和延迟。

"""规则引擎 + LLM 双引擎路由"""
import re
from typing import Optional, Callable, Awaitable
from dataclasses import dataclass, field

@dataclass
class Rule:
    pattern: str
    intent: IntentType
    priority: int = 0
    extract_entities: Optional[Callable[[str], dict]] = None

class RuleEngine:
    """规则引擎"""

    def __init__(self):
        self.rules: list[Rule] = [
            Rule(r"查.*工单\s*[A-Z]{3}[-]?\d+", IntentType.QUERY_TICKET),
            Rule(r"工单.*进度|审批.*到哪", IntentType.QUERY_TICKET),
            Rule(r"重置密码|忘记密码|找回密码", IntentType.RESET_PASSWORD),
            Rule(r"申请.*假|年假|事假|请假", IntentType.APPLY_LEAVE),
            Rule(r"退款|退钱|报销", IntentType.CHECK_REFUND),
            Rule(r"投诉|不满|差评|投诉电话", IntentType.COMPLAINT),
            Rule(r"转人工|人工客服|找个人", IntentType.TRANSFER_HUMAN),
            Rule(r"你好|您好|嗨|早上好|下午好", IntentType.GREETING),
        ]

    def match(self, text: str) -> Optional[IntentResult]:
        """规则匹配"""
        for rule in self.rules:
            match = re.search(rule.pattern, text, re.IGNORECASE)
            if match:
                entities = {}
                if rule.extract_entities:
                    entities = rule.extract_entities(text)
                # 尝试提取工单号
                ticket_match = re.search(r"[A-Z]{3}[-]?\d+", text)
                if ticket_match:
                    entities["ticket_id"] = ticket_match.group()
                return IntentResult(
                    intent=rule.intent,
                    confidence=0.95,  # 规则命中视为高置信度
                    entities=entities,
                    reasoning=f"规则匹配: {rule.pattern}"
                )
        return None

class DualRouter:
    """双引擎路由器"""

    def __init__(self, rule_engine: RuleEngine,
                 llm_classifier: IntentClassifier):
        self.rule_engine = rule_engine
        self.llm = llm_classifier
        self.high_threshold = 0.85
        self.low_threshold = 0.50

    async def route(self, text: str) -> dict:
        """路由决策"""
        # 第一优先：规则引擎
        rule_result = self.rule_engine.match(text)
        if rule_result:
            return {
                "source": "rule",
                "intent": rule_result.intent,
                "confidence": rule_result.confidence,
                "entities": rule_result.entities,
                "action": "route_directly"
            }

        # 第二优先：LLM 分类器
        llm_result = await self.llm.classify(text)
        if llm_result:
            confidence = llm_result.confidence
            if confidence >= self.high_threshold:
                return {
                    "source": "llm",
                    "intent": llm_result.intent,
                    "confidence": confidence,
                    "entities": llm_result.entities,
                    "action": "route_directly"
                }
            elif confidence >= self.low_threshold:
                return {
                    "source": "llm",
                    "intent": llm_result.intent,
                    "confidence": confidence,
                    "entities": llm_result.entities,
                    "action": "ask_clarify"
                }
            else:
                return {
                    "source": "llm",
                    "intent": llm_result.intent,
                    "confidence": confidence,
                    "entities": llm_result.entities,
                    "action": "transfer_human"
                }

        # LLM 分类失败
        return {
            "source": "fallback",
            "intent": IntentType.UNKNOWN,
            "confidence": 0.0,
            "entities": {},
            "action": "transfer_human"
        }

# 流量分布统计
router = DualRouter(RuleEngine(), IntentClassifier())
# 线上数据: 规则 65%, LLM 高置信 25%, 低置信 7%, 转人工 3%

置信度阈值的设定依据

双引擎路由的关键参数是置信度阈值——多少分以上可以信任 LLM 的分类结果直接路由，多少分以下需要转人工？这个问题不能拍脑袋，必须基于数据的 Precision-Recall 分析。CorpAssist 在 5000 条标注样本上进行了阈值扫描实验，找到了 0.85 和 0.5 两个最优切分点。

python
"""置信度阈值分析"""
import numpy as np
from sklearn.metrics import precision_recall_curve, f1_score

def find_optimal_thresholds(y_true: np.ndarray, y_scores: np.ndarray):
    """
    基于 5000 条标注样本找到最优阈值
    y_true: 真实标签 (1 正确 / 0 错误)
    y_scores: LLM 返回的 confidence
    """
    precisions, recalls, thresholds = precision_recall_curve(y_true, y_scores)

    # 找到 F1 最大化的阈值
    f1_scores = 2 * (precisions * recalls) / (precisions + recalls + 1e-10)
    best_idx = np.argmax(f1_scores)
    best_threshold = thresholds[best_idx]

    # 找到 Precision > 0.95 的最大阈值（高置信度）
    high_conf_idx = np.where(precisions[:-1] >= 0.95)[0]
    high_threshold = thresholds[high_conf_idx[-1]] if len(high_conf_idx) > 0 else 0.85

    # 找到 Recall > 0.9 的最小阈值（低置信度兜底）
    low_conf_idx = np.where(recalls[:-1] >= 0.9)[0]
    low_threshold = thresholds[low_conf_idx[0]] if len(low_conf_idx) > 0 else 0.5

    return {
        "high_threshold": float(f"{high_threshold:.2f}"),
        "low_threshold": float(f"{low_threshold:.2f}"),
        "best_f1_threshold": float(f"{best_threshold:.2f}"),
        "best_f1": float(f"{f1_scores[best_idx]:.3f}")
    }

# CorpAssist 实际阈值分析结果
threshold_analysis = {

将意图路由器暴露为 FastAPI 微服务。`POST /intent` 接收用户消息返回意图分类结果，`POST /intent/route` 在分类后直接路由到目标处理节点。服务通过/health 端点进行存活检查，通过 Prometheus 指标暴露 QPS/延迟/分布。

评估与持续标注闭环

意图分类模型不是一次训练就完事的——线上数据分布会随时间变化（比如新政策导致「请假」类请求增多），标注偏差会累积导致模型退化。CorpAssist 建立了持续标注闭环：每天自动抽取 100 条低置信度分类结果 + 50 条随机抽样结果进行人工标注，标注数据进入迭代 pipeline。每个月的模型微调版本基于最新的 5000 条标注数据。

python
"""持续评估与标注闭环"""
from sklearn.metrics import confusion_matrix, classification_report
import pandas as pd
import numpy as np
from datetime import datetime

class IntentEvalPipeline:
    """意图分类评估管线"""

    def __init__(self):
        self.intents = [i.value for i in IntentType]

    def compute_metrics(self, y_true: list, y_pred: list,
                        y_conf: list) -> dict:
        """计算混淆矩阵 + 分类报告"""
        cm = confusion_matrix(y_true, y_pred, labels=self.intents)
        report = classification_report(
            y_true, y_pred, labels=self.intents,
            target_names=self.intents, output_dict=True
        )
        # 格式化混淆矩阵为可读结构
        cm_df = pd.DataFrame(
            cm,
            index=[f"实际_{i}" for i in self.intents],
            columns=[f"预测_{i}" for i in self.intents]
        )
        return {
            "confusion_matrix": cm_df.to_dict(),
            "classification_report": report,
            "overall_accuracy": report["accuracy"],
            "macro_f1": report["macro avg"]["f1-score"],
            "weighted_f1": report["weighted avg"]["f1-score"],
        }

    def sampling_strategy(self, all_results: list) -> list:
        """分层抽样策略"""
        samples = []
        # 策略 1: 全部低置信度结果
        low_conf = [r for r in all_results if r["confidence"] < 0.50]
        samples.extend(low_conf)        # 约 50-100 条/天

        # 策略 2: 随机抽样
        import random
        high_conf = [r for r in all_results if r["confidence"] >= 0.85]
        samples.extend(random.sample(high_conf, min(30, len(high_conf))))

        # 策略 3: 低频率意图过采样
        from collections import Counter
        intent_counts = Counter(r["intent"] for r in all_results)
        rare_intents = [i for i, c in intent_counts.items() if c < 20]
        for result in all_results:
            if result["intent"] in rare_intents:
                samples.append(result)

        return list({r["session_id"]: r for r in samples}.values())

    def generate_report(self, metrics: dict) -> str:
        """生成中文评估报告"""
        report = metrics["classification_report"]
        lines = [
            f"意图分类评估报告 - {datetime.now().strftime('%Y-%m-%d')}",
            f"总体准确率: {metrics['overall_accuracy']:.2%}",
            f"Macro F1: {metrics['macro_f1']:.3f}",
            f"Weighted F1: {metrics['weighted_f1']:.3f}",
            "",
            "各意图 F1 Score:",
        ]
        for intent in self.intents:
            if intent in report:
                lines.append(
                    f"  {intent}: Precision {report[intent]['precision']:.3f}, "
                    f"Recall {report[intent]['recall']:.3f}, "
                    f"F1 {report[intent]['f1-score']:.3f}"
                )
        return "\n".join(lines)

标注闭环效果：经过 3 个月的持续标注和迭代，CorpAssist 意图分类的总体准确率从 87% 提升到 93%，unknown 类别的 F1 从 0.35 提升到 0.45（虽然仍然最低但有所改善）。投诉类别的 Precision 从 0.78 提升到 0.82——主要改善来自标注了更多「投诉」与「转人工」的边界案例。每月新增约 4500 条标注数据，每次模型微调后运行回归测试（2000 条标准测试集），确保不出现退化。

本章结论

  • Pydantic + with_structured_output 实现 LLM 结构化输出，确保分类结果 100% 可解析 JSON 格式
  • 规则引擎 + LLM 双引擎路由：规则引擎处理 65% 高频场景（延迟 <5ms），LLM 处理 35% 模糊场景（平均 1.1s），加权平均延迟约 395ms
  • 置信度阈值 0.85（Precision 97%）和 0.5（Recall 94%）基于 5000 条样本的 Precision-Recall 曲线扫描确定
  • FastAPI 意图路由服务 3 副本部署，日均 3500 次请求，/health 端点用于 K8s 存活检查
  • 持续标注闭环每日 150 条抽样 + 低频过采样，3 个月的迭代使准确率从 87% 提升到 93%，投诉类别 Precision 从 0.78 提升到 0.82

面试可能问

Q1: 意图识别的准确率到多少算合格？投诉和查询退款经常混淆怎么解决？

意图识别的合格线取决于业务场景——高频标准化场景（密码重置/问候）要求 Precision >95%，因为这些场景一旦误分类会导致完全错误的处理链路。低频复杂场景（投诉/未知意图）可以接受 Precision 80-85%，因为后续的对话引擎有纠正机会。投诉和查询退款的混淆是常见问题——用户说「我退的钱怎么还没到」既可能是查询退款（check_refund）也可能是投诉退得慢（complaint）。CorpAssist 的解决方案不是在意图层强行区分，而是把这种边界案例交给双引擎的 Prompt 层处理——Prompt 中注明「当用户同时涉及查询和不满情绪时，优先分类为 check_refund，但在 entities 中标记 has_complaint_tone=true」。这样对话引擎可以根据这个标记决定是否需要安抚情绪。这个方案使 check_refund 和 complaint 的互混淆率从 12% 降到 5%。

Q2: LLM 意图分类的延迟太高怎么办？能不能不用 LLM 做分类？

LLM 分类的延迟（1-2s）确实远高于传统 ML 分类器（50ms）。但 LLM 的优势在于零样本泛化——新意图无需标注数据即可支持。CorpAssist 的方案是「LLM 作为分类兜底，规则引擎做第一道防线」。具体：规则引擎用关键词+正则覆盖 65% 的流量（延迟 5ms，准确率 98%），只将 35% 的余量交给 LLM。加权平均后，整体意图识别的 P50 延迟仅 395ms。如果完全不用 LLM 分类，就需要维护一个传统分类模型（如 BERT 微调）——这意味着每次新增意图类型都需要收集标注数据 + 重新训练 + 部署模型，迭代周期从 1 天变成 2 周。所以正确的策略不是「替代 LLM」，而是「让 LLM 只处理它最擅长的部分——边界案例和未知意图」。对于极端延迟敏感的场景，还可以在 LLM 前加一层 Embedding 相似度匹配（将用户输入与预定义的意图示例向量做余弦相似度计算，延迟仅 30ms）——如果相似度 >0.92 直接通过，不调 LLM。

Q3: 怎么持续改进意图分类的准确率？标注成本怎么控制？

持续改进的核心理念是「标注在刀刃上」——不应该随机抽样标注全部结果，那样 90% 的标注预算会浪费在高置信度的正确分类上。CorpAssist 用分层抽样策略：第一优先级标注所有低置信度（<0.5）的结果——这些是最可能错误的分类，也是最有改进价值的。第二优先级从高置信度结果中随机抽取 10%——这些看起来都正确，但可能存在系统性偏差（比如所有「查退款」都被正确分类，但都分类成了「query_ticket」而非「check_refund」）。第三优先级对低频率意图过采样——如果「投诉」类样本只有 20 条/天，从中抽 15 条标注，确保低频类别也有足够的标注数据。标注成本控制：使用「LLM 预标注 + 人工修正」模式——先让 GPT-4o 做一个预分类，标注员只需确认或修改，标注速度提升 3 倍，成本降低 60%。CorpAssist 每月标注约 4500 条，标注费用约 1500 元（2 名兼职标注员 + LLM 预标注 API 费用）。每个月的改进体现在 F1 Score 上——平均每月提升 0.5-1.5%。

判断练习

如果意图分类中 complaint 类的 Precision 只有 0.72（28% 的分错了），应该先优化什么？

优先分析 complaint 的混淆矩阵——看 28% 的分错到底分到了哪些类别。通常 complaint 最容易和 transfer_human、check_refund 混淆。如果是和 transfer_human 混淆（用户说「我要投诉转人工」），解决方案是更新 Prompt 的规则——在 Prompt 中明确写「当用户同时表达投诉和转人工需求时，优先分类为 transfer_human，并将 complaint_flag=true 写入 entities」。如果是和 check_refund 混淆（用户说「退款太慢了我要投诉退的钱」），解决方案同上——分类为 check_refund，但 entities 中添加 has_complaint_tone=true。数据增强方法：从现有标注数据中找到所有误分类的 complaint 样本，把它们作为 few-shot 示例加入 LLM 的 Prompt（在 system prompt 中加 3-5 个示例）。通常这种优化可以在 1-2 天内将 complaint 的 Precision 提升 5-10 个百分点。如果还没改善，说明 complaint 类的标注数据本身有问题——需要重新检视标注规范。

如果用户输入「我想休息一下」，规则引擎没匹配到，LLM 分类为 apply_leave（置信度 0.78）。这个结果对吗？路由应该怎么做？

这个案例很好地说明了模糊意图的处理。「我想休息一下」这句话有歧义——可能是请假（apply_leave），也可能是单纯抱怨累（complaint），或者只是随口一说。置信度 0.78 处于 0.5-0.85 的「中置信度」区间，路由策略应该是「反问澄清」而不是直接路由。反问方式：先假设是 apply_leave，但请用户确认——「您是想申请请假吗？如果是，请告诉我请假类型和天数。」这样做的好处是：如果用户确实是想请假，只需一轮确认即可继续流程；如果用户说「不是，我就是吐槽一下」，则重新分类为 complaint 或 unknown。不能直接路由为 apply_leave——因为一旦路由到请假流程，对话引擎会开始收集「请假类型」「开始日期」「天数」等参数，如果用户其实不是想请假，这个过程会让用户困惑并给出低 CSAT。CorpAssist 线上数据：反问澄清的准确率为 92%（大部分用户会确认或否认），每次反问消耗 1 轮额外对话和约 100 token，属于可接受的成本。
