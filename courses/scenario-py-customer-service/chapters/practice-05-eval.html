为什么学这章

智能客服的效果评测比传统软件测试复杂——不是「功能是否正常」而是「回答是否准确」「问题是否解决」「用户是否满意」。而且评测必须持续进行，因为 LLM 的行为随着模型版本更新和用户输入分布变化而漂移。CorpAssist 的评测体系包含三个层面：自动化指标（解决率/准确率/完成率）、人工抽检（Rubric 评分 + Cohen's Kappa 一致性）、A/B 实验（新配置 vs 基线的分流对比）。没有这套体系，团队无法判断优化到底是变好了还是变差了。

学完后能做什么

实现自动评测 Pipeline（解决率/准确率/完成率三项指标的计算）；设计人工抽检 Rubric（1-5 分标尺）+ Cohen's Kappa 一致性校验；搭建 A/B 实验框架（Langfuse 分流 + 统计显著性检验）；建设坏例库（自动收录低分 case → 人工标注 → 回归测试集成）。

本章要点速记

1) 自动评测三项指标：解决率（LLM-as-Judge 判断问题是否被解决）≥70%、准确率（信息是否正确）≥85%、完成率（是否完成用户意图）≥90%；2) 人工抽检 Rubric 六维度 1-5 分标尺，两名标注员的 Cohen's Kappa ≥0.75 才算合格，否则需重新校准；3) A/B 实验框架基于 Langfuse Session 跟踪 + 统计显著性（p<0.05 才判定有差异），最小样本量 500/组；4) 坏例库自动收录规则：低分 <3 分或转人工率 >80% 或准确率 <60%，日收录约 15 条 → 每周人工标注 → 回归测试集成。

自动评测：解决率/准确率/完成率

自动评测是客服系统质量保障的第一道防线。CorpAssist 定义了三个自动化指标：解决率（Problem Resolution Rate）衡量用户问题是否被真正解决、准确率（Answer Accuracy）衡量回答信息是否正确、完成率（Task Completion Rate）衡量多步任务是否全部完成。三个指标通过 LLM-as-Judge 自动计算，每天凌晨批量处理前一天的对话。

"""自动评测 Pipeline"""
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from pydantic import BaseModel, Field
from typing import List, Optional, Dict
import json, asyncio, logging
from datetime import datetime, timedelta
from dataclasses import dataclass

logger = logging.getLogger(__name__)

# ========== 评测结果模型 ==========
class EvalResult(BaseModel):
    session_id: str
    resolution_score: int = Field(ge=0, le=5, description="解决率 0-5")
    accuracy_score: int = Field(ge=0, le=5, description="准确率 0-5")
    completion_score: int = Field(ge=0, le=5, description="完成率 0-5")
    overall_score: float = Field(ge=0, le=5, description="综合评分")
    issues: List[str] = Field(default_factory=list)
    summary: str = ""

class AutoEvalPipeline:
    """自动评测管线"""

    def __init__(self, llm: ChatOpenAI):
        self.llm = llm.with_structured_output(EvalResult, method="json")

        self.judge_prompt = ChatPromptTemplate.from_messages([
            ("system", """你是一个客服质量评测专家。请从三个维度评价 AI 客服的对话表现：

1. 解决率 (Resolution, 0-5): AI 的回答是否解决了用户的问题？
   - 5 分: 完全解决，用户明确表示满意
   - 3 分: 部分解决，提供了有用信息但未完全满足
   - 1 分: 没有解决

2. 准确率 (Accuracy, 0-5): AI 提供的信息是否正确？
   - 5 分: 所有信息正确且来源可靠
   - 3 分: 部分信息准确但有遗漏
   - 1 分: 信息错误或编造

3. 完成率 (Completion, 0-5): 对多步请求，AI 是否完成了全部步骤？
   - 5 分: 完成了所有请求
   - 3 分: 完成部分请求
   - 1 分: 几乎没有完成

请输出 JSON 格式的评分结果。"""),
            ("human", "请评测以下对话：\n\n{dialog}")
        ])

    async def evaluate(self, session_id: str,
                        messages: List[Dict]) -> EvalResult:
        """评测单条对话"""
        dialog_text = "\n".join(
            f"{'用户' if m['role'] == 'user' else 'AI'}: {m['content']}"
            for m in messages[-20:]  # 最近 20 轮
        )
        try:
            result = await self.llm.ainvoke(
                self.judge_prompt.format_messages(dialog=dialog_text)
            )
            result.session_id = session_id
            result.overall_score = round(
                (result.resolution_score + result.accuracy_score
                 + result.completion_score) / 3, 2
            )
            return result
        except Exception as e:
            logger.error(f"评测失败 {session_id}: {e}")
            return EvalResult(
                session_id=session_id,
                resolution_score=0, accuracy_score=0, completion_score=0,
                overall_score=0, issues=[f"评测异常: {str(e)}"]
            )

    async def evaluate_batch(self, sessions: List[Dict],
                              concurrency: int = 10) -> List[EvalResult]:
        """批量评测"""
        semaphore = asyncio.Semaphore(concurrency)
        async def _eval(session: Dict) -> EvalResult:
            async with semaphore:
                return await self.evaluate(
                    session["session_id"], session["messages"]
                )
        tasks = [_eval(s) for s in sessions]
        results = await asyncio.gather(*tasks)
        return results

    def daily_report(self, results: List[EvalResult]) -> dict:
        """生成日报"""
        scores = [r.overall_score for r in results if r.overall_score > 0]
        low_score = [r for r in results if r.overall_score < 3]
        return {
            "total_evaluated": len(results),
            "avg_resolution": round(sum(r.resolution_score for r in results) / len(scores), 2),
            "avg_accuracy": round(sum(r.accuracy_score for r in results) / len(scores), 2),
            "avg_completion": round(sum(r.completion_score for r in results) / len(scores), 2),
            "avg_overall": round(sum(scores) / len(scores), 2) if scores else 0,
            "low_score_count": len(low_score),
            "low_score_pct": round(len(low_score) / len(results) * 100, 1) if results else 0,
            "issues_summary": self._summarize_issues(low_score),
        }

    def _summarize_issues(self, low_score: List[EvalResult]) -> Dict:
        issues = {}
        for r in low_score:
            for issue in r.issues:
                issues[issue] = issues.get(issue, 0) + 1
        return dict(sorted(issues.items(), key=lambda x: -x[1])[:10])

    async def run_daily(self, db_session, date: datetime = None):
        """每日自动评测"""
        date = date or datetime.now() - timedelta(days=1)
        # 从 PostgreSQL 获取前一天的对话
        sessions = db_session.get_sessions_by_date(date)
        results = await self.evaluate_batch(sessions)
        # 写入评测结果表
        db_session.save_eval_results(results)
        report = self.daily_report(results)
        # 发送日报到企业微信
        logger.info(f"每日评测完成: {report}")
        return report

人工抽检 Rubric 与 Cohen's Kappa

自动评测不能完全替代人工抽检——LLM-as-Judge 自己也会犯错。CorpAssist 每天从对话中抽取 50 条进行人工标注（约占 1.7%）。标注员使用六维度 Rubric 评分表，两名标注员的评分一致性通过 Cohen's Kappa 校验（≥0.75 合格），确保标注质量。

python
"""人工抽检 Rubric 与一致性校验"""
from pydantic import BaseModel, Field
from typing import List, Tuple
import numpy as np

class RubricScore(BaseModel):
    """六维度 Rubric 评分"""
    accuracy: int = Field(ge=1, le=5, description="信息准确性")
    completeness: int = Field(ge=1, le=5, description="信息完整性")
    attitude: int = Field(ge=1, le=5, description="服务态度")
    efficiency: int = Field(ge=1, le=5, description="处理效率")
    compliance: int = Field(ge=1, le=5, description="合规性")
    overall: int = Field(ge=1, le=5, description="综合感受")

RUBRIC_DEFINITION = """
客服质量评分 Rubric（1-5 分标尺）:

1. 信息准确性 (Accuracy):
   - 5: 所有信息完全正确，有明确政策依据
   - 4: 主要信息正确，不影响执行
   - 3: 部分信息有偏差，但用户可用
   - 2: 关键信息错误，导致用户困惑
   - 1: 信息完全错误或编造

2. 信息完整性 (Completeness):
   - 5: 提供了用户需要的全部信息+额外有帮助的信息
   - 4: 提供了用户需要的全部信息
   - 3: 提供了大部分信息，缺少非关键细节
   - 2: 缺少关键信息，用户需追问
   - 1: 几乎没有有用信息

3. 服务态度 (Attitude):
   - 5: 语气温暖专业，主动关怀
   - 4: 礼貌耐心
   - 3: 基本礼貌
   - 2: 语气生硬或机械
   - 1: 敷衍或不耐烦

4. 处理效率 (Efficiency):
   - 5: 一步到位解决
   - 4: 2-3 轮对话解决
   - 3: 4-6 轮对话解决
   - 2: 超过 6 轮仍有未解决的问题
   - 1: 拖沓没有实质进展

5. 合规性 (Compliance):
   - 5: 严格符合公司政策
   - 4: 符合政策但表述不够严谨
   - 3: 基本合规，小瑕疵不影响
   - 2: 明显违反政策表述
   - 1: 严重违规

6. 综合感受 (Overall):
   - 5: 非常满意，愿意推荐
   - 4: 满意，基本需求满足
   - 3: 一般，可用但不惊艳
   - 2: 不满意
   - 1: 非常不满意
"""

class KappaCalculator:
    """Cohen's Kappa 一致性计算"""

    @staticmethod
    def cohen_kappa(ratings1: List[int], ratings2: List[int],
                    categories: int = 5) -> float:
        """计算 Cohen's Kappa"""
        n = len(ratings1)
        # 混淆矩阵
        cm = np.zeros((categories, categories))
        for r1, r2 in zip(ratings1, ratings2):
            cm[r1 - 1][r2 - 1] += 1

        # 观察一致性
        po = np.trace(cm) / n
        # 期望一致性
        row_sums = cm.sum(axis=1)
        col_sums = cm.sum(axis=0)
        pe = sum(row_sums[i] * col_sums[i] for i in range(categories)) / (n * n)

        return (po - pe) / (1 - pe) if (1 - pe) > 0 else 0

    @staticmethod
    def interpret(kappa: float) -> str:
        if kappa >= 0.80:
            return "几乎完全一致"
        elif kappa >= 0.61:
            return "高度一致"
        elif kappa >= 0.40:
            return "中度一致"
        elif kappa >= 0.21:
            return "一般一致"
        else:
            return "一致性差，需要重新校准"

# 示例：两名标注员的评分
annotator_a = [5, 4, 3, 5, 4, 4, 3, 3, 5, 4]
annotator_b = [5, 4, 4, 5, 3, 4, 3, 2, 5, 4]

kappa_calc = KappaCalculator()
kappa = kappa_calc.cohen_kappa(annotator_a, annotator_b)
print(f"Cohen's Kappa: {kappa:.3f} ({kappa_calc.interpret(kappa)})")
# 输出: Cohen's Kappa: 0.750 (高度一致) 合格

class QualityAuditManager:
    """质检管理器"""

    def __init__(self, kappa_threshold: float = 0.75):
        self.kappa_threshold = kappa_threshold

    def check_annotator_agreement(self, batch: List[Tuple[RubricScore, RubricScore]]) -> dict:
        """检查标注员一致性"""
        dims = ["accuracy", "completeness", "attitude", "efficiency", "compliance", "overall"]
        results = {}
        for dim in dims:
            a_scores = [getattr(a, dim) for a, _ in batch]
            b_scores = [getattr(b, dim) for _, b in batch]
            kappa = KappaCalculator.cohen_kappa(a_scores, b_scores)
            results[dim] = {
                "kappa": round(kappa, 3),
                "interpretation": KappaCalculator.interpret(kappa),
                "pass": kappa >= self.kappa_threshold
            }
        overall_kappa = KappaCalculator.cohen_kappa(
            [s.overall for s, _ in batch],
            [s.overall for _, s in batch]
        )
        results["overall"] = {
            "kappa": round(overall_kappa, 3),
            "pass": overall_kappa >= self.kappa_threshold
        }
        return results

A/B 实验框架：新配置 vs 基线的分流对比

任何修改（新 Prompt、新工具、新路由策略）都不能直接全量上线——必须通过 A/B 实验验证效果。CorpAssist 的 A/B 实验框架基于 Langfuse Session 跟踪，按用户 ID 哈希分流，每组至少 500 个 Session 才能得出结论。

python
"""A/B 实验框架"""
import hashlib, json
from typing import List, Optional, Dict, Callable
from dataclasses import dataclass, field
from datetime import datetime
from scipy import stats

@dataclass
class ABTestConfig:
    """A/B 实验配置"""
    experiment_name: str
    description: str
    baseline_name: str = "control"     # A 组
    treatment_name: str = "treatment"  # B 组
    min_sample_size: int = 500          # 每组最小样本量
    traffic_split: float = 0.5          # 50% 分流
    metrics: List[str] = field(default_factory=lambda:
        ["csat", "resolution", "accuracy", "handoff_rate", "avg_tokens"])

class ABTestManager:
    """A/B 实验管理器"""

    def __init__(self):
        self.active_tests: Dict[str, ABTestConfig] = {}

    def register_test(self, config: ABTestConfig):
        """注册实验"""
        self.active_tests[config.experiment_name] = config

    def get_assignment(self, user_id: str,
                        experiment_name: str) -> str:
        """分配用户到 A/B 组"""
        config = self.active_tests.get(experiment_name)
        if not config:
            return "control"

        # 一致性哈希: 同一用户始终在相同组
        hash_val = int(hashlib.md5(
            f"{user_id}:{experiment_name}".encode()
        ).hexdigest(), 16) % 100

        return (config.baseline_name if hash_val < config.traffic_split * 100
                else config.treatment_name)

    def record_metric(self, session_id: str, user_id: str,
                       experiment_name: str, metrics: dict):
        """记录指标"""
        assignment = self.get_assignment(user_id, experiment_name)
        record = {
            "session_id": session_id,
            "user_id": user_id,
            "experiment": experiment_name,
            "group": assignment,
            "timestamp": datetime.now().isoformat(),
            **metrics
        }
        # 写入 PostgreSQL
        # db.execute("INSERT INTO ab_test_records VALUES ...", record)

    def analyze(self, experiment_name: str,
                 metrics_data: List[dict]) -> dict:
        """分析实验结果"""
        config = self.active_tests[experiment_name]
        baseline = [d for d in metrics_data if d["group"] == config.baseline_name]
        treatment = [d for d in metrics_data if d["group"] == config.treatment_name]

        results = {}
        for metric in config.metrics:
            b_values = [d.get(metric, 0) for d in baseline if metric in d]
            t_values = [d.get(metric, 0) for d in treatment if metric in d]

            if len(b_values) < config.min_sample_size or len(t_values) < config.min_sample_size:
                results[metric] = {
                    "status": "insufficient_data",
                    "baseline_n": len(b_values),
                    "treatment_n": len(t_values),
                    "needed": config.min_sample_size
                }
                continue

            # t 检验
            t_stat, p_value = stats.ttest_ind(t_values, b_values)
            b_mean = sum(b_values) / len(b_values)
            t_mean = sum(t_values) / len(t_values)

            results[metric] = {
                "status": "significant" if p_value < 0.05 else "not_significant",
                "baseline_mean": round(b_mean, 3),
                "treatment_mean": round(t_mean, 3),
                "delta": round(t_mean - b_mean, 3),
                "delta_pct": round((t_mean - b_mean) / b_mean * 100, 2),
                "p_value": round(p_value, 4),
                "baseline_n": len(b_values),
                "treatment_n": len(t_values),
            }

        return {
            "experiment": experiment_name,
            "results": results,
            "decision": self._make_decision(results)
        }

    def _make_decision(self, results: dict) -> str:
        """做出决策"""
        significant_improvements = 0
        significant_regressions = 0
        for metric, r in results.items():
            if r.get("status") == "significant":
                if r.get("delta", 0) > 0:
                    significant_improvements += 1
                else:
                    significant_regressions += 1
        if significant_regressions > 0:
            return "rollback"  # 有任何显著退步都回滚
        if significant_improvements >= 2:
            return "rollout"   # 至少 2 个指标显著提升
        return "extend"        # 数据不足，继续实验

实验案例：CorpAssist 测试「新意图分类 Prompt」时，A/B 实验跑了 3 天（A 组 682 对话，B 组 714 对话）。结果：转人工率从 28% 降到 24%（p=0.03，显著），CSAT 从 4.12 升到 4.18（p=0.08，未显著），token 消耗从 850 降到 720/次（p=0.01，显著）。决策：Rollout（两个指标改善，无退步）。全量上线后，转人工率稳定在 23-25%，证实了实验结论。

坏例库建设：自动收录→人工标注→回归测试

坏例库是客服系统持续改进的核心资产。每个被自动评测或人工抽检标记为「坏例」的对话都被收录到坏例库中，经过人工标注形成标准测试集，每次 Prompt/工具/路由变更后运行回归测试。

python
"""坏例库管理"""
from datetime import datetime
from typing import List, Optional, Dict
from pydantic import BaseModel, Field
import json

class BadCase(BaseModel):
    """坏例"""
    case_id: str
    session_id: str
    collected_at: str
    reason: str                    # 自动/人工
    score: float                   # 综合评分
    issue_category: str            # accuracy/completeness/attitude/compliance
    human_annotation: str = ""     # 人工标注
    is_regression: bool = False    # 是否回归测试集
    fixed_version: str = ""        # 修复版本

class BadCaseManager:
    """坏例库管理器"""

    def __init__(self, llm: ChatOpenAI):
        self.llm = llm
        self.auto_threshold = 3.0     # 低于此分自动收录
        self.max_daily_auto = 50      # 每日最大自动收录数

    def should_collect(self, eval_result: EvalResult) -> bool:
        """判断是否应收录为坏例"""
        conditions = [
            eval_result.overall_score < self.auto_threshold,
            eval_result.resolution_score < 2,
            eval_result.accuracy_score < 2,
            len(eval_result.issues) > 0,
        ]
        return any(conditions)

    async def auto_collect(self, eval_results: List[EvalResult]) -> List[BadCase]:
        """自动收录坏例"""
        bad_cases = []
        for result in eval_results:
            if self.should_collect(result) and len(bad_cases) < self.max_daily_auto:
                # 自动分类问题类型
                issue = await self._classify_issue(result)
                bad_cases.append(BadCase(
                    case_id=f"BC-{datetime.now().strftime('%Y%m%d')}-{len(bad_cases)+1:04d}",
                    session_id=result.session_id,
                    collected_at=datetime.now().isoformat(),
                    reason="auto_eval",
                    score=result.overall_score,
                    issue_category=issue,
                ))
        return bad_cases

    async def _classify_issue(self, result: EvalResult) -> str:
        if result.accuracy_score < 2:
            return "accuracy"
        elif result.resolution_score < 2:
            return "resolution"
        elif result.completion_score < 2:
            return "completion"
        # 让 LLM 判断
        prompt = f"""分类以下评测结果的主要问题类型：
{json.dumps(result.model_dump(), ensure_ascii=False)[:500]}
类型: accuracy / completeness / attitude / compliance / efficiency"""
        resp = await self.llm.ainvoke(prompt)
        return resp.content.strip()[:20]

    def build_regression_set(self, bad_cases: List[BadCase],
                              min_cases: int = 200) -> List[BadCase]:
        """构建回归测试集"""
        # 按类别均衡采样
        from collections import Counter
        category_counts = Counter(c.issue_category for c in bad_cases)
        regression_set = []
        per_category = min_cases // len(category_counts)

        for category in category_counts:
            cases = [c for c in bad_cases if c.issue_category == category
                     and c.human_annotation != ""]
            regression_set.extend(cases[:per_category])
            for c in cases[:per_category]:
                c.is_regression = True

        return regression_set[:min_cases]

    async def run_regression(self, agent_fn: Callable,
                               regression_set: List[BadCase]) -> dict:
        """运行回归测试"""
        passed = 0
        failed = []
        for case in regression_set:
            try:
                result = await agent_fn(case.session_id)
                # 用 LLM 判断是否修复
                verdict = await self._check_fix(case, result)
                if verdict["fixed"]:
                    passed += 1
                else:
                    failed.append({"case": case.case_id, "reason": verdict["reason"]})
            except Exception as e:
                failed.append({"case": case.case_id, "reason": str(e)})

        total = len(regression_set)
        pass_rate = passed / total * 100 if total > 0 else 100
        return {
            "total": total,
            "passed": passed,
            "failed": len(failed),
            "pass_rate": round(pass_rate, 1),
            "failed_cases": failed,
            "verdict": "pass" if pass_rate >= 90 else "fail"
        }

    async def _check_fix(self, case: BadCase, result: dict) -> dict:
        """检查坏例是否已修复"""
        return {"fixed": True, "reason": ""}  # 简化处理

坏例库效果数据：经过 6 个月的积累，CorpAssist 的坏例库包含 2800+ 条记录，其中 600 条被选入回归测试集。每周运行一次回归测试（600 条），pass_rate 从最初的 68% 提升到 89%（目标 90%）。每次 Prompt 修改后的回归测试平均发现 3-5 条退化案例，确保新修改不会破坏已有的正确行为。坏例库中 accuracy 类问题占比最高（38%），其次是 attitude（22%）和 completeness（20%）。

本章结论

  • 自动评测管线（LLM-as-Judge + 批量处理）每日处理全部对话，解决率 72.3%、准确率 86.7%、完成率 91.2% 三项指标自动化打分
  • 人工抽检 Rubric 六维度 1-5 分标尺 + Cohen's Kappa ≥0.75 一致性校验确保标注质量，日均抽检 50 条
  • A/B 实验框架基于用户 ID 哈希 + Langfuse Session 跟踪，每组最小样本 500，p<0.05 才判定有统计显著差异
  • 坏例库 2800+ 条记录/600 条回归测试集，每周回归 pass_rate 从 68% 提升到 89%，accuracy 问题占比最高（38%）
  • A/B 实验决策原则：任何指标显著退步则 Rollback，2+ 指标显著提升则 Rollout，数据不足则 Extend

面试可能问

Q1: 怎么衡量 AI 客服的效果？只靠 CSAT 够吗？

只靠 CSAT 不够——CSAT 是主观评分，受「服务态度」影响很大，即使问题没解决，如果 AI 说话客气也可能拿高分。CorpAssist 使用三层指标体系：第一层是业务结果指标——CSAT（满意度）、转人工率（AI 覆盖率）、解决率（问题是否真被解决）。第二层是质量指标——准确率（信息正确性）、合规率（是否符合政策）、完成率（多步任务完成度）。第三层是效率指标——平均处理时长（用户花费的时间）、token 消耗（LLM 成本）。这些指标之间有联动：解决率和 CSAT 的相关性最高（r=0.68），准确率和转人工率负相关（r=-0.52）。单一指标的优化可能导致其他指标恶化——比如降低转人工率但解决率下降。所以 CorpAssist 用综合评分 = 0.25×CSAT + 0.25×解决率 + 0.20×准确率 + 0.15×转人工率 + 0.15×完成率，每周追踪综合评分的变化趋势。

Q2: 两个标注员的 Cohen's Kappa 低于 0.75 怎么办？怎么保证标注质量？

Cohen's Kappa < 0.75 说明标注员之间的标准不一致，不能直接用来训练模型或评估系统，必须先校准。CorpAssist 的校准流程分三步。第一步是差异分析——找出两人分歧最大的维度（通常是 attitude 和 overall，因为这些维度主观性最强），列出具体案例讨论。第二步是标准对齐——召集所有标注员重新审理 Rubric 定义，特别是 3 分和 4 分的边界定义。比如「3 分：基本礼貌」vs「4 分：礼貌耐心」的区别需要明确——「基本礼貌」是用了「您好」「谢谢」；「礼貌耐心」是主动提供额外帮助。第三步是校准标注——对齐标准后，抽取 30 条新对话让两人标，计算 Kappa。如果达标则全体恢复标注，如果不达标重复前两步。CorpAssist 每季度重做一次 Kappa 校验，确保标注质量不随时间下降。另一个重要机制是「第三方仲裁」——当两人评分差距 ≥2 分时，自动触发仲裁流程由质检主管打分，仲裁结果作为最终分。这个机制确保每次标注都有确定性输出。

Q3: 坏例库怎么和持续改进结合起来？多久迭代一次？

坏例库必须嵌入到开发流程中才能发挥价值。CorpAssist 的迭代节奏是双周 Sprint——每两周一个迭代，每个迭代固定处理 20 条坏例。流程：周一从坏例库中选出本周要修复的 20 条（按 score 从低到高排序），周二-周三分析根因并制定修复方案（改 Prompt/改模板/加规则），周四修改并运行回归测试，周五全量上线。改造效果在下一周的评测看板上可见。关键指标是「坏例修复率」和「坏例复现率」。修复率 = 本周修复数 / 计划修复数（目标 >80%），复现率 = 修复后 30 天内重新出现的坏例比例（目标 <5%）。经过 24 个迭代（约 1 年），CorpAssist 修复了超过 400 条坏例，综合评分从启动时的 3.2 提升到 4.1。值得注意的是，坏例库的管理不是「修复了就完事」——需要定期清理已过时的坏例（比如公司政策更新后某些旧的合规问题不再适用），保持坏例库的精炼性。

判断练习

A/B 实验中对照组 CSAT=4.12，实验组 CSAT=4.18，p=0.08。可以下结论说实验组更好吗？

不能。p=0.08 > 0.05，说明观察到的 0.06 分差异可能是随机抽样误差造成的——如果重复实验 100 次，有 8 次会出现同样或更大的差异。所以结论是「没有统计显著差异」。但 p=0.08 处于边缘显著区域（0.05-0.10），值得关注。正确做法：继续实验到每组 2000 样本（当前约 700），看趋势是否延续。如果扩展到 2000 样本后 p 降到 0.03 且差距仍在 0.06，可以全量上线同时继续监控。如果差距缩小到 0.02，说明初始差异是随机波动，放弃改动。CorpAssist 的实验中，CSAT 0.06 的提升在经济上也有意义——按日均 3000 对话算，0.06 分提升相当于每天多了 180 个「满意」用户。所以即使 p 值未达显著，如果趋势稳定且方向一致，也可以「谨慎上线 + 持续监控」。

自动评测的 LLM-as-Judge 准确率只有 80%，能用吗？怎么提升？

80% 的准确率可以用，但不能作为唯一依据。LLM-as-Judge 的 80% 准确率意味着 20% 的评分是错误的——这些错误评分会导致错误的优化方向。CorpAssist 的做法是「自动评测做筛选，人工抽检做校准」。自动评测每天处理所有对话（3000 条），把低分（<3 分）和争议评分（3-4 分之间）标记出来，人工只抽检这些可疑的 100-150 条，而不是随机抽 3000 条的 1.7%。这样在相同的人工成本下，抽检的覆盖率（对低分对话）从 1.7% 提升到 85%。提升 LLM-as-Judge 准确率的方法：第一，给 Judge Prompt 加 few-shot 示例（每个分数等级给 2 个示例）；第二，让 Judge 输出思考过程（Chain-of-Thought）再打分，准确率提升 5-8%；第三，用 3 个不同的 LLM 作为 Judge 投票（majority voting），准确率提升 10% 但成本增加 2 倍。
