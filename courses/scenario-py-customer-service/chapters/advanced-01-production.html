为什么学这章

智能客服从开发环境到生产环境不是一个简单的「换 URL」过程。SSE 流式需要通过 Nginx/Traefik 正确代理（否则连接会被缓冲或断开），高并发下需要限流保护下游 LLM API 不被打爆，JWT 鉴权需要中间件统一处理，7x24 运维需要完善的 oncall 手册和故障分类。这些「非 AI」的工程基础设施决定了客服系统的可用性和 SLA。CorpAssist 从 MVP 到生产化经历了 3 个月的打磨，本章总结了所有关键配置和坑。

本章要点速记

Nginx/Traefik + FastAPI SSE 流式代理配置

SSE（Server-Sent Events）在开发环境直接访问 FastAPI 端口时工作正常，但到了生产环境（前面有 Nginx/Traefik 反向代理）就会出现各种问题——连接不推送、推送被缓冲后一次性发送、长连接被断开。CorpAssist 踩过这些坑后总结了以下关键配置。

nginx
# Nginx SSE Agent 代理配置
upstream agent_backend {
    least_conn;
    server 127.0.0.1:8002 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8003 max_fails=3 fail_timeout=30s;
    keepalive 256;  # 连接池大小
}

server {
    listen 443 ssl http2;
    server_name agent.corp-assist.com;

    # SSL 配置（略）
    ssl_certificate /etc/ssl/certs/corp-assist.crt;
    ssl_certificate_key /etc/ssl/private/corp-assist.key;

    # SSE 流式端点
    location /chat/stream {
        proxy_pass http://agent_backend;

        # SSE 关键配置
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;          # 禁止缓冲
        proxy_cache off;              # 禁止缓存
        proxy_read_timeout 300s;      # 5 分钟超时（SSE 长连接）
        proxy_send_timeout 60s;
        proxy_connect_timeout 10s;

        # 传递原始信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 禁止 Nginx 缓冲（重要）
        add_header X-Accel-Buffering no;
        add_header Cache-Control no-cache;
        add_header Content-Type text/event-stream;

        # 客户端连接断开时停止上游请求
        proxy_ignore_client_abort on;
    }

    # REST API
    location /api/ {
        proxy_pass http://agent_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_read_timeout 30s;
        proxy_buffering on;
    }

    # WebSocket 坐席端
    location /ws/agent/ {
        proxy_pass http://agent_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 3600s;
    }
}

SSE 代理的坑：`proxy_buffering off` 是最关键的配置——如果没有这行，Nginx 会缓冲 SSE 数据直到 4KB（默认 buffer size）满才发送，用户看到的是变长的字符块分批出现而不是逐字显示。另外，`proxy_http_version 1.1` 和 `proxy_set_header Connection ""` 启用 HTTP/1.1 keepalive，避免每次请求重新建立连接（节省 30-50ms 握手时间）。`keepalive 256` 的上游连接池确保 256 个并发 SSE 连接复用 TCP 连接。

令牌桶 + 滑动窗口混合限流

智能客服的流量模式独特——有突发（10:00 AM 全体员工同时上班咨询）有规律（午休和下班前高峰）。单一限流策略效果不佳：纯令牌桶无法限制单秒突发（用户 1 秒发 100 条消息），纯滑动窗口无法平滑长期速率（窗口到期后瞬间恢复满速）。CorpAssist 的混合限流策略同时使用两种算法。

python
"""令牌桶 + 滑动窗口混合限流 FastAPI 中间件"""
from fastapi import FastAPI, Request, HTTPException
from fastapi.responses import JSONResponse
import time, asyncio
from redis import Redis
from typing import Optional

class HybridRateLimiter:
    """混合限流器"""

    def __init__(self, redis: Redis):
        self.redis = redis
        # 令牌桶参数
        self.bucket_rate = 50       # 每秒补充 50 个令牌
        self.bucket_capacity = 100  # 桶容量
        # 滑动窗口参数
        self.window_seconds = 1
        self.window_limit = 100     # 1 秒内最多 100 次

    async def check(self, key: str, cost: int = 1) -> tuple[bool, dict]:
        """
        检查是否允许请求
        返回: (allowed, headers)
        """
        now = time.time()
        bucket_key = f"rate_limit:bucket:{key}"
        window_key = f"rate_limit:window:{key}:{int(now)}"

        # 1. 滑动窗口检查
        window_count = self.redis.get(window_key)
        if window_count and int(window_count) >= self.window_limit:
            return False, {"Retry-After": "1", "X-RateLimit-Reset": str(int(now) + 1)}

        # 2. 令牌桶检查
        bucket = self.redis.hgetall(bucket_key)
        if bucket:
            tokens = float(bucket.get(b"tokens", 0))
            last_time = float(bucket.get(b"last_time", now))
            # 补充令牌
            elapsed = now - last_time
            tokens = min(self.bucket_capacity, tokens + elapsed * self.bucket_rate)
            if tokens < cost:
                wait = (cost - tokens) / self.bucket_rate
                return False, {"Retry-After": str(round(wait, 2)), "X-RateLimit-Reset": str(int(now + wait))}
        else:
            tokens = self.bucket_capacity

        # 3. 消耗令牌 & 更新窗口
        pipe = self.redis.pipeline()
        pipe.hset(bucket_key, mapping={
            "tokens": max(0, tokens - cost),
            "last_time": now
        })
        pipe.expire(bucket_key, 60)
        pipe.incr(window_key)
        pipe.expire(window_key, 2)  # 窗口 Key 2 秒过期
        pipe.execute()

        return True, {}

# FastAPI 中间件
def setup_rate_limiter(app: FastAPI, redis: Redis):
    limiter = HybridRateLimiter(redis)

    @app.middleware("http")
    async def rate_limit_middleware(request: Request, call_next):
        # 跳过健康检查
        if request.url.path == "/health":
            return await call_next(request)

        # 按用户或 IP 限流
        user_id = request.headers.get("X-User-ID", "")
        client_ip = request.client.host if request.client else "unknown"
        key = user_id or client_ip

        allowed, headers = await limiter.check(key)
        if not allowed:
            return JSONResponse(
                status_code=429,
                content={
                    "error": "Too Many Requests",
                    "retry_after": headers.get("Retry-After", "1"),
                    "message": "请求过于频繁，请稍后再试"
                },
                headers=headers
            )

        response = await call_next(request)
        return response

# FastAPI 应用
app = FastAPI(title="CorpAssist 客服 Agent", version="1.0.0")
redis = Redis(host="redis.internal", port=6379, decode_responses=True)
setup_rate_limiter(app, redis)

限流效果数据：混合限流上线后，API 错误率（5xx）从 2.1% 降到 0.3%（主要是防止了 LLM API 被打爆导致的 502）。429 返回率约 0.8%（日均 240 次），用户通常 1-2 秒后自动重试成功。峰值 QPS 从 80 平滑限制到 50（令牌桶 rate），确保了 LLC API 调用稳定在配额内。P99 限流中间件延迟 <5ms。

JWT 鉴权 FastAPI 中间件

智能客服必须确保只有合法用户可以调用 API。JWT（JSON Web Token）是标准方案，但在分布式部署中需要处理 Token 黑名单（用户登出后 Token 失效）、登录踢出（同一账号新登录使旧 Token 失效）。

python
"""JWT 鉴权 FastAPI 中间件"""
from fastapi import FastAPI, Request, HTTPException, Depends
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from jose import jwt, JWTError
from redis import Redis
from pydantic import BaseModel
from typing import Optional, List
import time, os

# 配置
SECRET_KEY = os.getenv("JWT_SECRET_KEY", "your-256-bit-secret")
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE = 3600  # 1 小时

security = HTTPBearer()

class JWTManager:
    """JWT 管理器"""

    def __init__(self, redis: Redis, secret: str = SECRET_KEY):
        self.redis = redis
        self.secret = secret
        self.algorithm = ALGORITHM

    def create_token(self, user_id: str, role: str = "user",
                     extra_claims: dict = None) -> str:
        """创建 Token"""
        now = time.time()
        payload = {
            "sub": user_id,
            "role": role,
            "iat": now,
            "exp": now + ACCESS_TOKEN_EXPIRE,
            "jti": f"{user_id}:{int(now)}",  # Token ID
        }
        if extra_claims:
            payload.update(extra_claims)
        return jwt.encode(payload, self.secret, algorithm=self.algorithm)

    async def verify(self, token: str) -> dict:
        """验证 Token"""
        # 1. 检查黑名单
        jti = None
        try:
            unverified = jwt.get_unverified_claims(token)
            jti = unverified.get("jti")
        except:
            raise HTTPException(401, "无效 Token")

        blacklisted = self.redis.sismember("token:blacklist", jti)
        if blacklisted:
            raise HTTPException(401, "Token 已失效（已登出）")

        # 2. 验证签名
        try:
            payload = jwt.decode(token, self.secret, algorithms=[self.algorithm])
            return payload
        except JWTError as e:
            raise HTTPException(401, f"Token 验证失败: {str(e)}")

    async def logout(self, token: str):
        """登出（加入黑名单）"""
        try:
            payload = jwt.decode(token, self.secret, algorithms=[self.algorithm])
            jti = payload.get("jti")
            exp = payload.get("exp", time.time() + 3600)
            ttl = int(exp - time.time())
            if ttl > 0 and jti:
                self.redis.sadd("token:blacklist", jti)
                self.redis.expire("token:blacklist", max(ttl, 3600))
        except:
            pass

    async def kick_out_previous(self, user_id: str):
        """踢出旧登录（同一用户新登录时使用）"""
        # 给该用户的旧 Token 打上踢出标记
        self.redis.sadd(f"kicked:{user_id}", str(int(time.time())))
        self.redis.expire(f"kicked:{user_id}", ACCESS_TOKEN_EXPIRE)

# FastAPI 依赖注入
async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security),
    jwt_manager: JWTManager = Depends(lambda: app.state.jwt_manager),
) -> dict:
    """获取当前用户"""
    return await jwt_manager.verify(credentials.credentials)

def setup_jwt_auth(app: FastAPI, redis: Redis):
    jwt_manager = JWTManager(redis)
    app.state.jwt_manager = jwt_manager

    # 白名单路径
    whitelist = ["/health", "/docs", "/openapi.json"]

    @app.middleware("http")
    async def jwt_middleware(request: Request, call_next):
        if request.url.path in whitelist:
            return await call_next(request)

        auth = request.headers.get("Authorization", "")
        if not auth.startswith("Bearer "):
            return JSONResponse(401, {"error": "Missing Authorization header"})

        try:
            token = auth.replace("Bearer ", "")
            payload = await jwt_manager.verify(token)
            request.state.user = payload
        except HTTPException as e:
            return JSONResponse(e.status_code, {"error": e.detail})

        return await call_next(request)

    return jwt_manager

7x24 oncall 手册

智能客服系统一旦上线，故障会影响所有员工的工作体验。CorpAssist 的 oncall 手册定义了告警等级、升级路径、Runbook 和复盘模板。

oncall 实战数据：CorpAssist 上线 6 个月触发了 3 次 P0 告警（2 次因 LLM API 限流导致服务不可用，1 次因 Redis 内存溢出崩溃）、12 次 P1 告警、45 次 P2 告警。平均 P0 恢复时间 12 分钟（目标 <15 分钟）。每次 P0/P1 故障后 24 小时内完成复盘，action items 的关闭率 95%。最有效的改进是：LLM API 限流后自动降级到小模型（而非直接报错），这消除了 2 个 P0 类故障。

本章结论

  • Nginx SSE 代理关键配置（proxy_buffering off / proxy_read_timeout 300s / keepalive 256）保证流式输出正确逐字推送
  • 令牌桶（50rps/桶 100）+ 滑动窗口（1s/100 次）混合限流使 API 5xx 错误率从 2.1% 降到 0.3%，P99 中间件延迟 <5ms
  • JWT 鉴权（python-jose 签名验证 + Redis 黑名单 + 登录踢出）验证延迟 <2ms，白名单路径跳过鉴权
  • oncall 手册定义 P0-P3 四级告警（响应时间 5min-24h），每次 P0/P1 故障后 24h 内完成复盘，action items 关闭率 95%
  • 最关键的运维改进：LLM API 限流后自动降级到小模型而不是直接报错——消除了 2/3 的 P0 故障

面试可能问

Q1: 生产环境的 SSE 和开发环境有什么不同？Nginx 代理 SSE 需要注意什么？

开发环境前端直接访问 FastAPI 端口，SSE 工作正常。生产环境前面有 Nginx 反向代理，SSE 会踩到几个坑。第一是缓冲——Nginx 默认 buffering on，会缓冲响应直到 4KB 满才发给客户端，导致 SSE 按块输出而不是逐字推送。解决方法：`proxy_buffering off; proxy_cache off;`。第二是超时——SSE 是长连接，Nginx 默认 proxy_read_timeout=60s，超过后断开连接。客服对话可能持续 2-5 分钟，需要设为 300s 或更高。第三是 HTTP 版本——SSE 需要 HTTP/1.1 keepalive（HTTP/1.0 不支持），需设置 `proxy_http_version 1.1; proxy_set_header Connection "";`。第四是连接池——大量 SSE 连接会耗尽 Nginx worker 连接，需要设置 `keepalive 256` 减少 TCP 握手。Traefik 的配置更简单——只需在 middleware 中设置 `X-Accel-Buffering: no` 响应头。另外，`proxy_ignore_client_abort on;` 也很重要——当用户关闭浏览器时，Nginx 默认会断开上游连接，导致 FastAPI 抛出 ConnectionResetError。开启此选项后 Nginx 不中断上游，由 FastAPI 自己检测并清理。

Q2: 限流策略怎么设计？只限单用户还是全局限流？

智能客服需要两层限流：用户级限流和全局限流。用户级限流用滑动窗口（1 秒内最多 5 次请求）防止单个用户刷 API。全局限流用令牌桶（50rps + 桶容量 100）保护后端 LLM API 不被打爆。为什么两层？单用户刷屏可能不是恶意而是用户连点了发送按钮（前端没做好防抖），限到 5rps 用户能正常使用。全局限流应对所有用户同时暴增的极端情况（如全员公告后 200 人同时咨询）。两层限流的 Key 设计：用户级 `rate_limit:user:{user_id}`，全局 `rate_limit:global:agent`。限流后返回 429 状态码 + Retry-After Header，客户端自动重试。CorpAssist 的限流效果：用户级限流拦截约 2% 的请求（主要是连点），全局限流几乎不触发（因为有用户级限流在它前面）。最重要的限流是 LLM API 级别的限流——用令牌桶确保每分钟不超过 OpenAI 配额（如 GPT-4o-mini 限 500 RPM），峰值时自动排队等待而不是直接拒绝。

Q3: oncall 手册怎么写才能让新人也能处理故障？

oncall 手册的核心受众是「被电话吵醒的新人」——他可能不清楚系统全貌，压力下会犯错。所以手册必须具体到命令级别。每条告警的 Runbook 包含两部分：诊断步骤和修复方案。诊断步骤用「可执行的命令」描述——不是「检查服务状态」而是「kubectl get pods -n agent | grep -v Running」看有多少非 Running Pod。修复方案必须提供至少两条路径——A 方案是首选（快速修复），B 方案是备份（A 方案失效时用），C 方案是终极方案（回滚或重启）。corpAssist 的 P0 Runbook 示例：A 方案「kubectl rollout restart deployment/agent」重启服务（耗时 30s），B 方案「kubectl rollout undo deployment/agent --to-revision=3」回滚到前 3 个版本（耗时 15s），C 方案「docker-compose -f docker-compose.fallback.yml up -d」启动独立备用集群（耗时 2min）。关键要求：每条命令必须经过验证（在 staging 环境实际执行过），不能从网上直接拷贝。另外，oncall 手册要定期演练——CorpAssist 每月做一次 P0 故障模拟演练，新人必须独立完成一个完整 Runbook。

判断练习

如果用户反馈「发送消息后没有回复，等了 30 秒都没收到」，可能是什么问题？

SSE 连接没有收到数据，可能的原因依次排查。第一，SSE 连接是否建立——检查浏览器 Network 面板中 /chat/stream 的 Status 是否为 200，Content-Type 是否为 text/event-stream。如果 Status 是 502/504，Nginx 或上游有问题。第二，Nginx 缓冲配置——检查 proxy_buffering 是否确实已关闭（通过响应头 X-Accel-Buffering 确认）。如果开启，SSE 数据被缓冲直到 4KB 或连接关闭才发送。第三，proxy_read_timeout——如果设为默认的 60s，60 秒后 Nginx 断开连接，用户端 onerror 触发。第四，FastAPI 端——检查 ASGI 服务器的 keep_alive_timeout 和 graceful_shutdown 设置。Uvicorn 默认 keep_alive_timeout=5s，太短了，应该设为 75s。第五，浏览器连接限制——一个域名最多 6 个 SSE 连接，如果用户开了多个标签页并行对话，超过 6 个后新的 SSE 连接排队等待。这些排查步骤按「从外到内」的顺序（用户端→Nginx→FastAPI→浏览器限制），能快速定位 80% 的 SSE 问题。

如果凌晨 3 点触发 P0 告警——所有 LLM 调用都返回 429（配额耗尽），值班人员应该怎么做？

P0 告警意味着所有人打不开客服系统。值班人员第一步（5 分钟内）执行 Runbook A 方案：检查 OpenAI API 状态面板确认是否全局故障（OpenAI 服务本身挂了还是我们配额超了）。如果是配额超了（常见原因是其他服务共享了同一个 API Key 用量超标），执行 B 方案：切换到备用 API Key（在环境变量中预留了第二个 Key）。同时立即打开「小模型兜底」开关——将所有 LLM 调用降级到本地部署的 Qwen2.5-1.5B 模型（虽然回答质量下降但系统可用）。第二步（15 分钟内）通知团队 Lead 和 Prompt 工程师评估影响。第三步（24 小时内）复盘：为什么配额会用超？是正常的业务增长还是异常流量？修复措施：给智能客服分配独立的 API Key 并设置硬配额（500 RPM），不与其他服务共享。
