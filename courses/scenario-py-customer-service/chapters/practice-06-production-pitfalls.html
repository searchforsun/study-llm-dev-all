为什么学这章

智能客服上线后最危险的问题不是「不好用」而是「看起来好用但偷偷犯错」。CorpAssist 在生产运行中踩过三个大类坑：LLM 幻觉导致对话状态错误（DST 污染）、LLM「声称」已调 API 但实际上没调（不回写幻觉）、高并发下 LLM 调用导致缓存穿透和成本失控。这三个问题在 demo 环境中永远不会暴露，只在生产环境（日均 3000+ 对话、峰值 QPS 80+）中才会显现。理解这些坑和对应的防护措施，是生产级客服系统必备的能力。

学完后能做什么

实现对话状态追踪（DST）的规则 + 模型双校验，防止 LLM 幻觉导致状态错误；检测和防止 LLM 声称已调 API 但实际未调的回写幻觉；设计截图分级路由（小模型 OCR vs 视觉大模型）平衡准确率和成本；实现三层缓存（语义缓存 GPTCache + Redis + 小模型路由）将高并发下的 LLM 调用降低 60%。

本章要点速记

1) DST 双校验：规则层校验状态变更的合法性（如「已退款」状态的订单不应再触发退款操作），模型层每 2 轮校验一次 DST 一致性——不一致时回滚到上一轮有效状态；2) 意图→API 不回写幻觉的检测通过「工具调用日志交叉验证」——对比 LLM 声称的操作与实际工具调用记录，不一致时拦截并通知；3) 截图分级路由：简单文字截图（<50 字）用 PaddleOCR（延迟 200ms，成本 0）→ 复杂表格截图用 GPT-4o Vision（延迟 2s，成本 $0.01/次）；4) 三层缓存——GPTCache 语义缓存（相似度 >0.92 命中，命中率 30%）+ Redis KV 缓存（精确匹配，命中率 15%）+ 小模型路由（确定性查询用小模型，命中率 20%），总 LLM 调用减少 60%。

DST 的规则 + 模型双校验

对话状态追踪（Dialogue State Tracking, DST）的核心任务是维护当前对话状态——用户意图、已收集的参数、已执行的操作。LLM 的幻觉可能污染 DST：它可能在状态中加入了不存在的参数（如虚构的「优惠券」字段），或者错误地标记操作已完成（如订单还在处理就说「已退款」）。CorpAssist 的 DST 双校验机制在每一轮状态更新时同时执行规则检查和模型检查。

"""DST 规则 + 模型双校验"""
from typing import Dict, Any, Optional, Tuple, List
from pydantic import BaseModel, Field
from langchain_openai import ChatOpenAI
import json, logging

logger = logging.getLogger(__name__)

class DSTState(BaseModel):
    """对话状态"""
    intent: str = ""
    collected_params: Dict[str, Any] = {}
    executed_actions: List[str] = []
    order_status: Optional[str] = None   # pending/processing/completed/cancelled
    refund_status: Optional[str] = None  # none/submitted/approved/rejected
    error_count: int = 0

class DSTRuleValidator:
    """DST 规则校验器"""

    def __init__(self):
        # 状态转换白名单
        self.allowed_order_transitions = {
            "pending": ["processing", "cancelled"],
            "processing": ["completed", "cancelled"],
            "completed": [],
            "cancelled": [],
        }
        self.allowed_refund_transitions = {
            "none": ["submitted"],
            "submitted": ["approved", "rejected"],
            "approved": [],
            "rejected": [],
        }

    def validate(self, current: DSTState, update: Dict[str, Any]) -> Tuple[bool, str]:
        """校验状态更新的合法性"""
        # 规则 1: 状态转换合法性
        if "order_status" in update:
            new_status = update["order_status"]
            allowed = self.allowed_order_transitions.get(current.order_status or "pending", [])
            if new_status not in allowed and current.order_status:
                return False, f"订单状态不允许 {current.order_status} → {new_status}"

        if "refund_status" in update:
            new_status = update["refund_status"]
            allowed = self.allowed_refund_transitions.get(current.refund_status or "none", [])
            if new_status not in allowed and current.refund_status:
                return False, f"退款状态不允许 {current.refund_status} → {new_status}"

        # 规则 2: 退款金额不超过订单金额
        if "collected_params" in update:
            params = update["collected_params"]
            if "refund_amount" in params and "order_amount" in params:
                if params["refund_amount"] > params["order_amount"]:
                    return False, f"退款金额 {params['refund_amount']} 超过订单金额 {params['order_amount']}"

        # 规则 3: 已取消的订单不能操作
        if current.order_status == "cancelled":
            if update.get("intent") in ["submit_refund", "update_address"]:
                return False, "已取消订单不可操作"

        return True, ""

class DSTModelValidator:
    """DST 模型校验器"""

    def __init__(self, llm: ChatOpenAI):
        self.llm = llm

    async def check_consistency(self, dialog: List[Dict],
                                  current_state: DSTState,
                                  proposed_update: Dict) -> Tuple[bool, str]:
        """检查 DST 更新与对话历史的一致性"""
        prompt = f"""检查对话状态更新是否与对话历史一致。

对话历史：
{chr(10).join(f"{'用户' if m['role']=='user' else 'AI'}: {m['content']}" for m in dialog[-6:])}

当前状态：
{json.dumps(current_state.model_dump(), ensure_ascii=False, indent=2)}

状态更新：
{json.dumps(proposed_update, ensure_ascii=False, indent=2)}

请判断状态更新是否与对话历史一致。
注意：如果用户没有明确说某件事，状态不应包含该信息。

返回 JSON: {{"consistent": bool, "reason": "str"}}"""
        try:
            result = await self.llm.ainvoke(prompt)
            data = json.loads(result.content.strip())
            return data["consistent"], data.get("reason", "")
        except Exception as e:
            logger.warning(f"DST 模型校验失败: {e}")
            return True, "校验异常，通过"  # 模型校验失败时默认通过

class DSTManager:
    """DST 管理器"""

    def __init__(self, llm: ChatOpenAI):
        self.rule_validator = DSTRuleValidator()
        self.model_validator = DSTModelValidator(llm)
        self.check_interval = 2  # 每 2 轮模型校验一次
        self.state_history: Dict[str, List[DSTState]] = {}  # session_id → 状态历史

    async def update(self, session_id: str, dialog: List[Dict],
                      current_state: DSTState,
                      proposed_update: Dict[str, Any]) -> DSTState:
        """更新 DST（双校验）"""
        # 1. 规则校验（每次必做）
        ok, reason = self.rule_validator.validate(current_state, proposed_update)
        if not ok:
            logger.warning(f"DST 规则校验拒绝: {reason}")
            return current_state  # 拒绝更新

        # 2. 应用更新
        new_state = current_state.model_copy(deep=True)
        for key, value in proposed_update.items():
            if hasattr(new_state, key):
                setattr(new_state, key, value)

        # 3. 模型校验（每 check_interval 轮一次）
        if len(dialog) % self.check_interval == 0:
            consistent, reason = await self.model_validator.check_consistency(
                dialog, current_state, proposed_update
            )
            if not consistent:
                logger.warning(f"DST 模型校验不一致: {reason}")
                # 重试一次
                consistent2, reason2 = await self.model_validator.check_consistency(
                    dialog, current_state, proposed_update
                )
                if not consistent2:
                    return current_state  # 两次都不一致，回滚

        # 4. 保存历史
        if session_id not in self.state_history:
            self.state_history[session_id] = []
        self.state_history[session_id].append(new_state)
        # 限制历史长度
        self.state_history[session_id] = self.state_history[session_id][-20:]

        return new_state

DST 防护效果：上线双校验后，CorpAssist 的 DST 错误率从 3.2% 降到 0.4%。规则校验拦截了约 70% 的非法状态更新（主要是状态转换错误和参数超限），模型校验拦截了约 30% 的「看起来合法但实际不一致」的更新（主要是 LLM 幻觉添加了对话中未提及的信息）。最重要的一次拦截是：LLM 在更新 DST 时添加了「用户同意支付 500 元加急费」——但实际上用户并未同意，如果将这个状态写入后续系统可能会发起错误扣款。模型校验发现「支付同意」在对话历史中不存在，拦截了这次更新。

意图→API 不回写幻觉的检测与防护

LLM 生成回复时可能「声称」已经执行了某项操作，但实际上没有调用对应的工具 API。比如 LLM 回复「已为您重置密码，新密码已发送到您的邮箱」——但 LLM 并没有调用 reset_password 工具，只是「以为」自己做了。这种「不回写幻觉」（hallucination of action execution）比普通幻觉更危险，因为用户会按 LLM 说的去做而实际上操作没被执行。

python
"""意图→API 不回写幻觉检测"""
from typing import List, Dict, Tuple, Optional
from langchain_openai import ChatOpenAI
import json, re, logging

logger = logging.getLogger(__name__)

class ActionHallucinationDetector:
    """操作幻觉检测器"""

    def __init__(self, llm: ChatOpenAI):
        self.llm = llm
        # 操作关键词映射
        self.action_keywords = {
            "query_order": ["查", "查询", "看", "找", "订单", "工单"],
            "submit_refund": ["退", "退款", "退钱", "提交退款"],
            "reset_password": ["重置密码", "新密码", "密码已重置"],
            "update_address": ["地址", "修改地址", "改地址"],
        }

    def detect(self, response: str,
                actual_tool_calls: List[Dict]) -> List[str]:
        """检测是否声称了但没执行的操作"""
        halluciations = []

        for action, keywords in self.action_keywords.items():
            # 检查回复中是否提及了该操作
            mentioned = any(kw in response for kw in keywords)
            if not mentioned:
                continue

            # 检查实际是否调用了对应的工具
            actually_called = any(
                tc.get("tool") == action or tc.get("name") == action
                for tc in actual_tool_calls
            )

            if mentioned and not actually_called:
                halluciations.append(action)

        return halluciations

    async def deep_detect(self, response: str, dialog: List[Dict],
                            tool_calls: List[Dict]) -> Tuple[bool, List[str]]:
        """深度检测（含 LLM 判断）"""
        # 第一关：关键词检测
        simple_hallucinations = self.detect(response, tool_calls)
        if simple_hallucinations:
            return True, simple_hallucinations

        # 第二关：LLM 语义检测
        prompt = f"""判断以下 AI 回复是否声称执行了某项操作但实际未执行。

AI 回复：{response}

已执行的工具调用：
{json.dumps(tool_calls, ensure_ascii=False)}

返回 JSON: {{"has_hallucination": bool, "claimed_actions": ["list of unexecuted actions"], "reason": "str"}}"""
        try:
            result = await self.llm.ainvoke(prompt)
            data = json.loads(result.content.strip())
            if data.get("has_hallucination"):
                return True, data.get("claimed_actions", [])
        except Exception as e:
            logger.warning(f"深度检测异常: {e}")

        return False, []

class ActionVerifier:
    """操作验证器"""

    def __init__(self, detector: ActionHallucinationDetector):
        self.detector = detector

    async def verify_before_send(self, response: str,
                                   dialog: List[Dict],
                                   tool_calls: List[Dict]) -> Tuple[str, bool]:
        """发送前验证"""
        has_hallucination, claimed = await self.detector.deep_detect(
            response, dialog, tool_calls
        )
        if has_hallucination:
            logger.warning(f"检测到操作幻觉: {claimed}")
            # 修复回复：移除未执行的操作声明
            fixed = self._fix_response(response, claimed)
            return fixed, True  # 需要拦截

        return response, False

    def _fix_response(self, response: str,
                       claimed_actions: List[str]) -> str:
        """修复回复"""
        fixes = {
            "reset_password": "密码重置需要您验证身份后才能进行，请问您方便验证吗？",
            "submit_refund": "退款申请已记录，正在为您提交处理...",
        }
        for action in claimed_actions:
            if action in fixes:
                response += f"\n\n{fixes[action]}"
        return response

线上数据：检测器每天从约 4200 条回复中检测到约 15-20 次操作幻觉（发生率约 0.4%）。最常见的幻觉操作是「reset_password」（LLM 声称已重置但未调工具）、「submit_refund」（LLM 说已退款但实际没提交）。大多数幻觉发生在 LLM 生成回复时「过度承诺」——它想表现得 helpful 于是假装做了某事。检测器拦截后，回复被修复为「密码重置需要您验证身份后才能进行」，避免用户按照错误操作行动。

截图分级路由与小模型 OCR

用户经常上传截图来咨询问题（如「这个报错是什么意思」）。如果所有截图都用 GPT-4o Vision 处理，成本太高——一张图约 $0.01（GPT-4o），日均 500 张图 = $5/天 ≈ $150/月。CorpAssist 的截图分级路由方案：简单文字截图用 PaddleOCR 提取文字（免费，延迟 200ms），复杂表格/图表才用视觉大模型。

python
"""截图分级路由"""
from typing import Optional, Tuple
from dataclasses import dataclass
import base64, json

@dataclass
class ImageAnalysisResult:
    source: str         # paddleocr / gpt4o_vision
    text: str
    confidence: float
    processing_time_ms: int
    cost: float         # $ per request

class ImageRouter:
    """截图路由"""

    def __init__(self):
        self.text_threshold = 50      # 字数阈值：<50 字用 OCR
        self.conf_threshold = 0.85    # OCR 置信度阈值

    async def route(self, image_base64: str) -> ImageAnalysisResult:
        """路由到合适处理器"""
        # 先尝试 PaddleOCR（简单文字截图）
        import time
        start = time.time()

        ocr_result = await self._ocr(image_base64)
        if ocr_result:
            text_len = len(ocr_result["text"])
            conf = ocr_result["confidence"]
            elapsed = (time.time() - start) * 1000

            if text_len <= self.text_threshold and conf >= self.conf_threshold:
                return ImageAnalysisResult(
                    source="paddleocr",
                    text=ocr_result["text"],
                    confidence=conf,
                    processing_time_ms=int(elapsed),
                    cost=0.0
                )

        # 复杂截图用 GPT-4o Vision
        elapsed = (time.time() - start) * 1000
        vision_result = await self._gpt4o_vision(image_base64)
        return ImageAnalysisResult(
            source="gpt4o_vision",
            text=vision_result["text"],
            confidence=vision_result["confidence"],
            processing_time_ms=int(elapsed),
            cost=0.01
        )

    async def _ocr(self, image_b64: str) -> Optional[dict]:
        """PaddleOCR 文字提取"""
        # 模拟 PaddleOCR 调用
        import random
        texts = ["系统错误代码: 500", "订单状态: 已发货", ""]
        return {
            "text": random.choice(texts),
            "confidence": random.uniform(0.7, 0.95)
        }

    async def _gpt4o_vision(self, image_b64: str) -> dict:
        """GPT-4o Vision 分析"""
        # 模拟 GPT-4o 调用
        return {
            "text": "这是一个包含表格和数据的复杂截图...",
            "confidence": 0.95
        }

高 QPS 的三层缓存策略

客服系统在高峰期（10:00-11:00 AM）QPS 可达 80+，如果每个请求都调用 LLM API，不仅成本高（日均 ~$120），还可能导致 OpenAI API 限流。CorpAssist 的三层缓存策略将高峰期 LLM 调用量降低 60%，保证了 P99 延迟 <2s。

python
"""三层缓存策略"""
from typing import Optional, Dict, Any, Callable, Awaitable
from redis import Redis
import hashlib, json, time, asyncio
import numpy as np
from sentence_transformers import SentenceTransformer

class GPTCache:
    """语义缓存（GPTCache 简化版）"""

    def __init__(self, redis: Redis, model: SentenceTransformer,
                 similarity_threshold: float = 0.92):
        self.redis = redis
        self.model = model
        self.threshold = similarity_threshold

    async def get(self, query: str) -> Optional[str]:
        """语义匹配查找"""
        query_embed = self.model.encode(query, normalize_embeddings=True)
        # 从 Redis 中查找所有缓存的 key
        cursor = 0
        while True:
            cursor, keys = self.redis.scan(cursor, match="gptcache:*", count=100)
            for key in keys:
                cached = self.redis.get(key)
                if not cached:
                    continue
                cached_data = json.loads(cached)
                # 计算余弦相似度
                cached_embed = np.array(cached_data["embedding"])
                similarity = np.dot(query_embed, cached_embed)
                if similarity >= self.threshold:
                    return cached_data["response"]
            if cursor == 0:
                break
        return None

    async def set(self, query: str, response: str, ttl: int = 3600):
        """写入缓存"""
        query_embed = self.model.encode(query, normalize_embeddings=True)
        key = f"gptcache:{hashlib.md5(query.encode()).hexdigest()}"
        self.redis.setex(
            key, ttl,
            json.dumps({
                "query": query,
                "embedding": query_embed.tolist(),
                "response": response,
                "timestamp": time.time()
            })
        )

class ThreeLayerCache:
    """三层缓存"""

    def __init__(self, redis: Redis, gptcache: GPTCache):
        self.redis = redis
        self.gptcache = gptcache
        self.stats = {"layer1_hits": 0, "layer2_hits": 0,
                       "layer3_hits": 0, "misses": 0}

    async def get(self, query: str, intent: str = "") -> Optional[str]:
        """三层缓存查询"""
        # 第一层：Redis KV 精确匹配
        key = f"cache:kv:{hashlib.md5(query.encode()).hexdigest()}"
        cached = self.redis.get(key)
        if cached:
            self.stats["layer1_hits"] += 1
            return cached

        # 第二层：GPTCache 语义匹配
        semantic_result = await self.gptcache.get(query)
        if semantic_result:
            self.stats["layer2_hits"] += 1
            return semantic_result

        # 第三层：小模型路由
        if intent in ("greeting", "query_ticket_simple"):
            small_model_result = await self._small_model_call(query)
            if small_model_result:
                self.stats["layer3_hits"] += 1
                return small_model_result

        self.stats["misses"] += 1
        return None

    async def set(self, query: str, response: str, intent: str = ""):
        """写入所有缓存层"""
        # 第一层：精确匹配
        key = f"cache:kv:{hashlib.md5(query.encode()).hexdigest()}"
        self.redis.setex(key, 3600, response)

        # 第二层：语义缓存
        if intent not in ("greeting",):
            await self.gptcache.set(query, response)

    async def _small_model_call(self, query: str) -> Optional[str]:
        """小模型路由"""
        # 模拟本地小模型调用
        if "你好" in query:
            return "您好！我是 CorpAssist 智能客服，请问有什么可以帮您的？"
        if "查工单" in query:
            return "请提供您的工单号，我来帮您查询。"
        return None

    def get_stats(self) -> dict:
        total = sum(self.stats.values())
        return {
            **self.stats,
            "hit_rate": round((total - self.stats["misses"]) / total * 100, 1) if total else 0,
        }

缓存效果数据：日均 30000 次请求中，三层缓存命中 19500 次（命中率 65%），每日 LLM 调用仅 10500 次。GPTCache 语义缓存（相似度阈值 0.92）命中率最高（30%），适合「不同用户问相同问题但措辞略有差异」的场景。Redis 精确缓存（15%）适合完全相同的重复查询。小模型路由（20%）适合问候和简单查询。三层缓存实施后，LLM API 月费用从 $3000 降到 $1200（降 60%），高峰期 QPS 从容应对，未再触发 OpenAI 限流。

本章结论

  • DST 双校验（规则每次 + 模型每 2 轮）将 DST 错误率从 3.2% 降到 0.4%，拦截了 LLM 虚构「用户同意支付加急费」等危险状态更新
  • 操作幻觉检测器日均发现 15-20 次 LLM「声称已操作但实际未调 API」的幻觉（发生率 0.4%），拦截后修复回复避免用户误操作
  • 截图分级路由（PaddleOCR + GPT-4o Vision）将月视觉成本从 $150 降到 $30（80% 的简单截图由免费 OCR 处理）
  • 三层缓存（Redis KV + GPTCache + 小模型路由）总命中率 65%，LLM 调用减少 60%，月 API 费用从 $3000 降到 $1200
  • 所有这些坑的共同特征：在 demo 环境中不会暴露，只在日均 3000+ 对话/峰值 QPS 80+ 的生产环境中显现

面试可能问

Q1: 怎么防止 LLM 在对话状态中编造信息？DST 的校验机制怎么设计？

LLM 在更新对话状态时可能编造用户未提供的信息——比如用户说了「查一下订单」，LLM 在 DST 中加了一个「用户要求退款」的错误字段。CorpAssist 的双校验机制的思路是「规则限制范围 + 模型验证内容」。规则校验定义了一套状态转换白名单和参数合法性规则。比如「订单状态 cancelled → 不能调用 submit_refund」「refund_amount ≤ order_amount」「intent 必须是预定义的 8 个类别之一」。规则校验在每次 DST 更新时执行，延迟约 1ms，拦截约 70% 的非法更新。模型校验每 2 轮执行一次——让 LLM 检查当前 DST 是否与对话历史一致。Prompt 设计成「对话历史：... 当前 DST: ... 状态更新: ... 是否一致？」如果模型判定不一致，再让同一个模型重校验一次。两次都不一致则回滚到上一轮有效状态。这个机制比较昂贵（每 2 轮一次 LLM 调用），所以只适用于重要状态变更。

Q2: LLM 回复说「已为您退款」但实际没调退款 API，这种幻觉怎么检测和防止？

这是客服系统中最危险的一类幻觉。检测方法分两道关。第一关是在 LLM 生成回复后、发送给用户前，用一个简单的关键词检测器扫描回复——如果回复中包含「退款」「密码重置」「地址修改」等操作关键词，检查本次对话的工具调用记录中是否有对应的 Tool Call。如果没有，标记为操作幻觉。这个检测器零 LLM 成本，延迟 <5ms。第二关是 LLM 深度语义检测——有些 LLM 回复比较隐晦（如「您的退款已在处理中，请耐心等待」），关键词检测可能漏过。第二关将回复发送给一个低成本的 Judge LLM 判断「AI 是否声称执行了某操作但实际没执行」。检测到幻觉后的处理：将回复中的操作声明替换为「您请求的操作需要进一步验证，请问您方便提供更多信息吗？」。同时，记录幻觉到监控系统，如果某个意图的幻觉发生率 >1%，自动告警触发 Prompt 修改。CorpAssist 的数据显示，操作幻觉发生率约 0.4%，主要集中在 reset_password 和 submit_refund 两个操作。

Q3: 高并发下 LLM 调用导致成本失控怎么办？缓存策略怎么设计？

高并发下 LLM 成本控不住是客服系统最常见的生产事故。CorpAssist 的三层缓存策略将日均 30000 次请求中的 65% 用缓存代替了 LLM 调用。第一层是 Redis KV 精确缓存——Hash key 为 query 的 MD5，TTL=1h。适合用户重复问同一个问题（如「退款政策是什么」）。命中率 15%，延迟 <5ms。第二层是 GPTCache 语义缓存——将 query Embedding 后存到 Redis，新请求的 Embedding 与缓存库中计算余弦相似度，>0.92 时命中。适合「不同用户问类似问题但措辞不同」的场景（如「怎么退款」「退款流程是」）。命中率 30%，延迟 50ms（含 Embedding 计算）。第三层是小模型路由——对于问候和简单查询（如「你好」「查工单 TKT-001」），用本地部署的小模型（如 Qwen2.5-1.5B）处理，零 LLM API 成本。命中率 20%，延迟 200ms。三层缓存加起来命中率 65%，使月 API 费用从 $3000 降到 $1200。

判断练习

如果缓存命中率从 65% 突然降到 40%，可能是什么原因？怎么排查？

缓存命中率骤降说明缓存的内容突然「不适用」了。排查维度：第一时间维度——降速是几天内缓慢下降还是断崖式下跌？如果是缓慢下降，可能是用户咨询的主题在自然变化（如新政策发布导致用户问的新问题缓存中没有）；如果是断崖式下跌，很可能是缓存服务故障或缓存 Key 过期策略被误改。第二意图维度——哪个意图的缓存命中率下降最多？如果 greeting 类的命中率降了但 query_ticket 没降，说明缓存本身没问题而是用户行为变了。第三相似度阈值——检查 GPTCache 的 similarity_threshold 是否被误调。如果从 0.92 升到 0.95，很多原本命中的请求会因为分数不够而不命中。最可能的根因：新政策上线导致用户问的问题与缓存中的历史问题语义差异大（相似度 <0.92）。解决方案：新政策发布后主动预缓存常见问答（运营团队准备的 FAQ）。CorpAssist 的缓存命中率通常在政策发布后下降 10-15%，然后 2-3 天随着新问题的缓存而恢复。

截图分级路由中，PaddleOCR 识别错误的文字（如将「500」识别成「S00」）导致 GPT-4o 也不一定能纠正，怎么保证准确性？

OCR 错误是截图处理中的常见问题。CorpAssist 的设计是「OCR 置信度作为第一道门」。PaddleOCR 对每个识别结果输出置信度分数，只有分数 >0.85 且文字长度 <50 字的简单截图才走 OCR 路线——不满足任意条件都升级到 GPT-4o Vision。即使如此，OCR 仍然可能有 8% 的错误率（主要是数字混淆，如 0/O、1/l、S/5）。CorpAssist 的补偿方案：第一，对 OCR 识别的文字做「常识性校验」——如检查提取的数字是否在合理范围内（报错代码通常是 3-4 位数字），如果不合理则标注 low confidence 升级到 Vision。第二，在发送给用户前让 LLM 做一次「信息合理性检查」——「这个截图分析结果看起来合理吗？如果发现奇怪的字符或数字，标记出来」。第三，如果用户反馈截图分析结果有误，将该截图加入「OCR 失败库」并由人工标注正确文字，下次相同截图先用人工标注结果。这套方案使截图分析的总体准确率保持在 96% 以上。
