为什么要学本章

S2 客服需要「刚才说的工号是多少」这类多轮上下文。Chat memory 在 Advisor 链中自动读写消息历史。单机开发可用内存实现；生产必须 redis memory + multi tenant 隔离，并与 Python 实验服务约定 python session sync。

学完你能
  • 选择 MessageChatMemory 与 message window 策略
  • 用 Redis 持久化 conversation id 维度历史
  • 设计 session id 键：tenant:user:conversation
  • 处理双栈会话冲突与 memory prune
本章先记住 3 件事
  • conversationId 由客户端传：BFF 生成 UUID，Spring 不凭空造 session。
  • 窗口优于全量：保留最近 N 条 + 可选 summary，控制 token 与成本。
  • 键空间含 tenantId：避免 Redis 键碰撞与越权读历史。

Memory 类型：Advisor 与窗口

CorpAssist · S2 多轮追问 用户：「查 E1024 年假」→ Bot 答 3 天 → 用户：「那差旅呢？」Bot 需记得工号。Memory Advisor 在 call 前加载历史，call 后写回。

java 
@Bean
ChatMemory chatMemory() {
  return MessageWindowChatMemory.builder()
      .maxMessages(20)
      .build();
}

@Bean
ChatClient sessionChatClient(ChatModel model, ChatMemory chatMemory) {
  var memoryAdvisor = MessageChatMemoryAdvisor.builder(chatMemory).build();
  return ChatClient.builder(model)
      .defaultAdvisors(memoryAdvisor)
      .build();
}

// 每次请求传入 conversationId
sessionChatClient.prompt()
    .advisors(a -> a.param(ChatMemory.CONVERSATION_ID, conversationId))
    .user("那差旅呢？")
    .call()
    .content();

Redis 存储：持久化与 TTL

多实例 Spring 服务必须共享 Memory。Spring AI 提供 Redis ChatMemory 实现（或自定义 ChatMemoryRepository），键格式见 Demo session-key-template.txt。

yaml 
spring:
  data:
    redis:
      host: localhost
      port: 6379
corpassist:
  chat:
    memory:
      key-prefix: corpassist:chat
      ttl: 24h
java 
@Component
public class RedisChatMemoryRepository implements ChatMemoryRepository {
  private final StringRedisTemplate redis;
  private final Duration ttl = Duration.ofHours(24);

  public String key(String tenantId, String userId, String conversationId) {
    return "corpassist:chat:" + tenantId + ":" + userId + ":" + conversationId;
  }

  public void saveMessages(String key, List<Message> messages) {
    redis.opsForValue().set(key, serialize(messages), ttl);
  }
}
仅 conversationId 作键
  • 不同租户 id 碰撞
  • 用户 A 读到用户 B 历史
  • 合规事故
tenant:user:conversation
  • 与 Demo 模板一致
  • TTL 24h 自动 prune
  • 审计按 tenant 分库

多租户 session：鉴权与 conversationId

BFF 从 JWT 解析 tenantId、userId，客户端只传 conversationId（或新建）。Spring Controller 禁止信任客户端传的 tenantId。

java 
@PostMapping("/v1/chat")
public ChatReply chat(@RequestBody ChatRequest req,
    @AuthenticationPrincipal CorpUser user) {
  String convId = Optional.ofNullable(req.conversationId())
      .orElse(UUID.randomUUID().toString());
  String compositeKey = memoryKey(user.tenantId(), user.userId(), convId);
  String answer = chatService.reply(compositeKey, req.message());
  return new ChatReply(convId, answer);
}
CorpAssist · S2 门户 同一员工在 Web 与移动端各开对话，应使用不同 conversationId；同一 conversationId 在多 Tab 共享历史。
FAQ：conversationId 过期后用户继续聊怎么办？

Redis TTL 过期后 Memory 为空，模型无历史。API 可返回新 conversationId，或提供「继续上次主题」按钮由 BFF 显式传旧 id（未过期时）。

Python 会话同步：双栈仲裁

Python 实验 Bot 可能与 Spring 共用同一 Redis 前缀，或各自存储。必须文档化权威源：生产以 Spring 为准；Python 只读或写 shadow key。

会话冲突仲裁
flowchart TB
  BFF[BFF / 门户]
  Spring[Spring ChatMemory Redis]
  Py[Python 实验 Bot]
  BFF -->|写| Spring
  Py -.->|eval only shadow key| Redis[(Redis)]
  Spring --> Redis
  • 键前缀分离：生产 corpassist:chat:；Python eval corpassist:py-eval:。
  • 同步协议：若必须共享，JSON 序列化格式版本号写入 value 头。
  • 冲突：同时写同一 key 时，Spring 侧 wins；Python 脚本禁止写生产前缀。
  • prune：定时任务按 tenant 统计 key 数量，超限触发 memory prune 或缩短 TTL。

暴露 HTTP 时见 暴露 REST 与 SSE；与 Function Calling / Tools 同会话内可连续 tool_call。

FAQ：水平扩展时 Memory 会丢吗？

使用 Redis 等共享 ChatMemoryRepository 时不会；InMemory 仅适合单实例 local。K8s 多 Pod 必须 Redis，且 conversationId 由 BFF 稳定传递。

适用：@SpringBootTest、本地单实例。重启丢失；conversationId 仅进程内有效。勿用于 K8s 多副本。

适用：预发/生产。键 corpassist:chat:{tenant}:{user}:{conv}，TTL 24h。序列化 JSON 含 schemaVersion 便于 Python 只读解析。

memory prune 策略（运维）

  • TTL 到期：Redis 自动删除；用户新开 conversationId。
  • 窗口裁剪：MessageWindow 只保留最近 20 条，旧消息不落盘或异步归档。
  • 租户配额：每 tenant 活跃 conversation 超阈值时 LRU 淘汰或拒绝新 session。
  • 合规删除：员工离职 API 按 userId 批量 DEL 前缀键。
CorpAssist · S2 多 Tab 同一 conversationId 在 Web 多 Tab 共享 Redis 历史；移动端新开会话应生成新 conversationId，避免与桌面长对话混淆。
FAQ：Memory Advisor 与 RAG 谁先？

通常 Memory 在 RAG 之前，保证检索 query 含完整指代；若先 RAG 再拼历史，「它」类指代可能检索到错误部门制度。

自检：conversationId 由谁生成？

推荐 BFF 在创建会话时生成 UUID 并下发门户；禁止客户端随意复用他人 conversationId，网关须校验 user/tenant 与 Redis 键一致。

eval 影子会话使用独立 Redis 前缀，避免压测数据污染生产 conversation 键空间。

延伸：conversationId 由谁生成？

推荐 BFF 在创建会话时生成 UUID 并下发门户；禁止客户端随意复用他人 conversationId，网关须校验 user/tenant 与键一致。

MessageWindow 条数与 token 预算联动：窗口过大时优先 prune 旧 tool 结果而非截断最新用户问题。

Redis 序列化升级 schemaVersion 时，保留向后兼容读路径至少一个发布周期。

本章结论

  • Memory 类型：MessageWindow + MessageChatMemoryAdvisor；conversationId per 请求。
  • Redis：corpassist:chat:{tenant}:{user}:{conv}，TTL 24h。
  • 多租户：tenant/user 来自鉴权，不信任客户端。
  • Python 同步：生产键权威在 Spring；eval 用 shadow 前缀或只读。
