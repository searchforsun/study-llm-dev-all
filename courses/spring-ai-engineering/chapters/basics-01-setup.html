为什么要学本章

CorpAssist S2 智能客服 的 Java 轨从 Spring Boot 3.3 + Spring AI 1.0 起步。若不清楚 Starter 拉入了哪些 Bean、application yml 如何与 Python 轨共用网关，后面 chat client、RAG、NS2 工具与护栏 都会变成「复制粘贴配置」。本章把依赖 → 配置 → Bean 生命周期 → 模块边界一次讲透。

学完你能
  • 在父 POM 引入 dependency bom 与 OpenAI 兼容 Starter，并说明各自动配置类职责
  • 用环境变量注入 spring.ai.openai.*，与 Python 工程轨python-engineering-for-llm 共用 openai compatible 网关
  • 画出 corpassist spring 多模块分工，并指出 Demo 中 3 处需新增的依赖/配置
本章先记住 3 件事
  • 抽象优先：业务依赖 ChatClient/VectorStore，而非厂商 HTTP SDK。
  • 配置外部化：模型名、密钥、base-url 来自环境，不写死在源码。
  • 模块边界：corpassist-ai-chat 管对话，corpassist-ai-rag 管检索增强，网关只做鉴权与路由。

Starter 依赖

Spring AI 1.0.x 通过 spring ai starter 把模型客户端、auto configuration 与 Actuator 指标挂钩。CorpAssist 父工程先 import BOM，子模块只声明「能力」依赖。

依赖注册的核心 BeanCorpAssist 用途
spring-ai-starter-model-openai OpenAiChatModel、OpenAiApi S2 客服对话、结构化抽取（兼容网关）
spring-ai-starter-model-ollama（可选） OllamaChatModel 本地联调、CI 替身（见 测试与本地替身）
spring-boot-starter-actuator 健康检查、指标端点 与 Micrometer 章衔接，暴露 LLM 延迟
spring-ai-bom

锁定 Spring AI 与 Spring Boot 3.3 对齐版本，避免子模块各自声明版本号。

Starter

按 classpath 条件注册 ChatModel；无 API Key 时启动可失败或降级（由你配置决定）。

业务模块

只依赖 Starter，不直接依赖 OkHttp/厂商 SDK，便于换模型与 Mock 测试。

不推荐
  • 业务模块直接依赖厂商 Java SDK
  • 子模块各自写 Spring AI 版本号

  • 密钥仅来自环境或密钥管理服务
先想 10 秒：为何 CorpAssist 不直接在 Controller 里 new OpenAiApi？

因为需要统一配置、测试替身（Mock ChatModel）与 Advisor 链注入；这些都依赖 Spring 容器管理的 chat model bean。

application.yml

Spring AI 约定前缀 spring.ai.*。CorpAssist 与 Python 轨共用同一套 CORPASSIST_LLM_BASE_URL，实现双栈同网关、同模型名策略。

yaml
spring:
  application:
    name: corpassist-spring-demo
  ai:
    openai:
      api-key: ${OPENAI_API_KEY}
      base-url: ${CORPASSIST_LLM_BASE_URL:https://api.openai.com}
      chat:
        options:
          model: ${CORPASSIST_CHAT_MODEL:gpt-4o-mini}
          temperature: 0.2
配置项含义与 Python 轨对齐
base-urlOpenAI 兼容网关根地址同 OPENAI_BASE_URL 环境变量
chat.options.model默认对话模型评测脚本中的 model 字段一致
temperature随机性；客服 FAQ 宜偏低Python temperature=0.2 对照
CorpAssist · S2 智能客服 · 配置切换 大促前把 CORPASSIST_CHAT_MODEL 从「高质量」切到「低成本」模型，无需改 Java 代码；NS2 工具调用仍走同一网关，仅路由策略在 advanced 章展开。
常见误区
  • 只配 api-key 不配 base-url — 内网网关无法生效，联调时 Python 能通、Java 不能。
  • 把 model 写死在 @Value 常量 — 违背配置外部化，A/B 与降级困难。
  • dev/prod 共用同一 Key — 审计与配额无法分环境。

Bean 生命周期

引入 Starter 后，auto configuration 在应用启动期注册 ChatModel；业务层通过 ChatClient.Builder 构建门面 Bean。理解 bean lifecycle 才能正确写 @Configuration 与测试替身。

启动期：从 Starter 到可注入的 ChatClient
sequenceDiagram
  participant Boot as Spring Boot
  participant AC as OpenAiAutoConfiguration
  participant CM as ChatModel Bean
  participant CFG as AiChatConfiguration
  participant SVC as CustomerChatService
  Boot->>AC: 读取 spring.ai.openai.*
  AC->>CM: 创建 OpenAiChatModel
  CFG->>CM: ChatClient.builder(chatModel)
  CFG->>SVC: 注入 ChatClient

业务 Service 只依赖 ChatClient；换模型或 Mock 时替换 ChatModel Bean 即可。

显式注册 ChatClient（推荐写法）

java
@Configuration
public class AiChatConfiguration {

  @Bean
  ChatClient customerChatClient(ChatModel chatModel) {
    return ChatClient.builder(chatModel)
        .defaultSystem("你是 CorpAssist S2 智能客服，回答须基于制度库。")
        .build();
  }
}
阶段发生什么你可介入的方式
条件装配classpath 有 OpenAI Starter 且配置完整@ConditionalOnProperty 自定义开关
Bean 创建ChatModel 单例就绪@Primary / @Qualifier 多模型（见下一章）
门面构建ChatClient 绑定默认 System、Advisorbuilder().defaultAdvisors(...)
请求时prompt → model → 响应Advisor 链、指标（advanced）
自检：单元测试时如何不连真实 LLM？

用 @MockBean ChatModel 或 Testcontainers + WireMock 替身；advanced 章有完整 CI 范式。本章只需记住：测试替换的是 ChatModel，不是绕过 Spring 直接 HTTP。

CorpAssist-spring 模块

maven module 按变更频率与职责拆分，避免「一个 Boot 应用装下 RAG + 客服 + 批处理」。corpassist spring 典型布局如下。

    RAG[corpassist-ai-rag]
    COM[corpassist-common]
  end
  Portal["企业门户 / BFF"]

S2 客服入口走 corpassist-ai-chat；制度库 RAG 可走内置 VectorStore 或委托 Python 索引服务。

模块职责本阶段关联
corpassist-gateway鉴权、限流、request_id透传会话 ID 给 Memory 章
corpassist-ai-chatChatClient、Prompt、Advisor本章 Starter 落点；NS2 工具在 practice 章
corpassist-ai-ragVectorStore、检索增强practice 阶段展开
corpassist-commonDTO、异常、OpenAPI 模型与 Python 契约同构 JSON
不推荐
  • 单体模块内 Controller 直接拼 HTTP 调模型
  • chat 与 rag 共用一个巨型 @Service
  • 无 common 模块，DTO 在模块间复制粘贴
推荐
  • chat 模块只关心对话与护栏；索引在 rag 或 Python
  • 配置集中在 application.yml + 环境变量
  • 双栈契约：同一 LeaveRequest JSON 形状（见 structured 章 Demo）

与 Python 工程轨python-engineering-for-llm 分工：Python 擅长大批量索引、实验性算法；Spring 轨负责企业治理下的在线 API（鉴权、审计、Advisor、OpenAPI）。

能否只加一个 spring-ai-starter 就跑通 S2 客服？
可以完成本章 Demo 级联调；生产还需网关、Memory、RAG、NS2 护栏与指标，分布在后续章节。
CorpAssist 为何强调 OpenAI 兼容而不是绑死一家云？
便于内网网关、私有化模型与双栈共用 endpoint；换供应商时改配置而非改业务代码（advanced 可移植性章展开）。
延伸：本地联调检查清单
  1. mvn -q test 在 corpassist-ai-chat 通过（Mock ChatModel）。
  2. spring.ai.openai.base-url 与 Python 轨 OPENAI_BASE_URL 指向同一网关。
  3. API Key 仅来自环境变量；Actuator /health 可达。
  4. 门户请求带 X-Request-Id，便于 advanced 章 trace 关联。
FAQ：子模块何时拆 ai-rag？
当 VectorStore、ingest 与 Chat 配置膨胀到同一模块难以单测时，将 RAG 迁入 corpassist-ai-rag；gateway 仍只做鉴权与路由。

Starter 版本以 spring-ai-bom 为准，勿在子模块手写与 BOM 冲突的 Spring AI 版本号。

本章结论

每条对应上文一个小节，便于串起全章再做题。

  • Starter 依赖：父 POM import BOM，子模块声明 OpenAI（或 Ollama）Starter，业务不直接依赖厂商 SDK。
  • application.yml：spring.ai.openai.* 从环境注入，与 Python 轨共用 base-url 与 model 策略。
  • Bean 生命周期：AutoConfiguration 注册 ChatModel，@Configuration 构建 ChatClient 供 Service 注入。
  • CorpAssist-spring 模块：gateway / ai-chat / ai-rag / common 分责，S2 客服与 NS2 能力有清晰落点。

判断练习

  1. 同事在 Controller 里 new OpenAiApi(apiKey) 调模型，并说「这样更直观」。请指出违反本章哪条原则及推荐改法。

    参考答案

    违反抽象优先与 Bean 生命周期管理。应使用 Starter 注册的 ChatModel + ChatClient Bean，便于配置外部化、Advisor 注入与 @MockBean 测试。

  2. 联调时 Python 脚本连网关成功，Java 服务仍访问 api.openai.com。最可能漏配哪一项？

    参考答案

    spring.ai.openai.base-url（或环境变量 CORPASSIST_LLM_BASE_URL）未注入；仅配置 api-key 不够。
