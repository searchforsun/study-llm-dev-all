本章先记住 3 件事
  • 换配置不换代码：starter + yml；业务只依赖 ChatClient。
  • OpenAI 兼容网关：openai compatible base-url 可接多数国产私有化部署。
  • 换模型必跑评测：引用率、must_contain、P95 三线对比，见 demo model-migration.md。

换模型步骤

CorpAssist · 网关迁移 集团要求 Q2 前从公有云 API 切到内网「OpenAI 兼容」网关（通义 qwen-plus）。运维更新 Secret；开发确认 Spring AI starter 仍指向 /v1/chat/completions；算法同学用同一 golden 集对比 qwen 与 gpt-4o-mini 的引用率。

五步检查单（见 demo model-migration.md）

  1. 更新 base-url 指向私有化网关（含路径前缀是否含 /v1）
  2. 确认 OpenAI 兼容路径 /v1/chat/completions 与 embedding 路径
  3. 调整 model 名 与 options（temperature、max_tokens）
  4. 跑评测集 对比分数、引用率、must_contain
  5. 灰度 10% 流量，看 P95 与 error rate，再全量
  6. 准备回滚：保留旧 model 环境变量与镜像 tag 24h，metrics 劣化时一键切回

自检：换 Chat 模型为何要关心 embedding？

Chat 与 embedding 独立；但若同时换 embedding 模型，向量空间变，旧 PGVector 索引失效，需 re-embed（Python 批处理或 Spring 调度任务）。

配置外部化

Spring Boot 3.3 推荐：12-factor 配置进环境变量；密钥进 K8s Secret；非敏感路由策略进 ConfigMap。禁止在 Git 提交 api-key。

application.yml + 环境变量

yaml
spring:
  ai:
    openai:
      base-url: ${CORPASSIST_LLM_BASE_URL:https://api.openai.com}
      api-key: ${CORPASSIST_LLM_API_KEY}
      chat:
        options:
          model: ${CORPASSIST_CHAT_MODEL:qwen-plus}
          temperature: ${CORPASSIST_CHAT_TEMP:0.2}
      embedding:
        options:
          model: ${CORPASSIST_EMBED_MODEL:text-embedding-v3}

corpassist:
  retrieval:
    mode: ${CORPASSIST_RETRIEVAL_MODE:local}
  python-search:
    base-url: ${CORPASSIST_PYTHON_SEARCH_BASE_URL:http://localhost:8000}

K8s Secret / ConfigMap 片段

对比 Python 抽象

双栈可移植性来自同层抽象 + 同契约，而非「语言相同」。

能力Spring AI 1.0.xPython 轨（典型）

Embedding EmbeddingModel OpenAIEmbeddings

配置 application.yml + env Pydantic Settings + env

Python 批量索引脚本与 Spring 在线 API 共用 embedding model 名与维度配置；否则检索「能跑但召回差」难以排查。OpenAPI 契约（RAG API）是跨语言边界，比抽象类名更重要。

python
# Python 轨 — settings.py（对照 Spring application.yml）
class Settings(BaseSettings):
    openai_base_url: str = Field(alias="OPENAI_BASE_URL")
    openai_api_key: str = Field(alias="OPENAI_API_KEY")
    chat_model: str = "qwen-plus"
    embed_model: str = "text-embedding-v3"

# 与 spring.ai.openai.* 环境变量同名便于双栈运维
仍会产生锁定的场景
  • Prompt 针对某一模型调优（few-shot 仅适用于 GPT 风格）
  • Structured Output JSON schema 厂商支持度不同
  • Tool calling 字段格式差异（需 Adapter）
  • 向量维度变更未 re-embed

面试题

以下结合 CorpAssist 场景，用于 TOP5 defense 与 interview rag 模块。

Q1：如何降低 LLM 供应商锁定？

要点
  • 业务依赖 ChatClient / 领域 Service，不依赖厂商 SDK 类型
  • OpenAI 兼容网关 + base-url 外部化
  • Prompt / 检索 / 评测与模型解耦；golden 集回归
  • 双栈 OpenAPI 契约，避免 BFF 绑死某一后端

Q2：换模型后如何证明「没有变劣」？

答：三线证据——(1) golden JSONL 引用率与 must_contain ≥ 阈值；(2) Grafana P95 / error rate 不劣于基线；(3) 人工 badcase 抽检 20 条。灰度期对比 model tag 的 metrics。

Q3：Spring AI 与直接调 REST 有何取舍？

答：Spring AI 提供 Advisor、Observation、VectorStore 统一抽象；直接 REST 少依赖但需自研护栏与指标。CorpAssist 选 Spring AI 因 NS2/NS3 需要 Advisor 链与 Micrometer 集成。

Q4：embedding 换模型要注意什么？

答：维度可能变化 → 重建 vector 表或 re-embed 全量文档；检索评测 recall@k 必须重跑；Chat 模型可独立更换。

自检：「我们用 Spring AI 所以不会 vendor lock-in」成立吗？

不成立。 抽象降低切换成本，但 Prompt 调优、Tool schema、索引与评测仍绑定行为；可移植性需要流程（配置 + 契约 + 评测）而不只是框架。
