为什么要学本章

chat client 是 Spring AI 推荐的业务门面；底层 chat model 才对接 HTTP。S2 客服要在门户里流式吐字、按意图路由多模型，并和 Python 实验脚本保持同一套 openai compatible 语义——本章把 call chain 画清楚。

学完你能
  • 口述 Controller → ChatClient → ChatModel → 网关的完整 call chain
  • 用 @Qualifier 注册快/慢两路 multi model Bean，并说明 NS2 工具调用前的路由点
  • 返回 Flux<String> 对接 SSE，并与 Python stream=True 对照（python parity）
本章先记住 3 件事
  • 门面：业务只调 ChatClient，不直接调 ChatModel API。
  • 同步 vs 流式：.call() 阻塞拿全文；.stream() 走 Reactor Flux。
  • 双栈：同一 prompt 在 Python 脚本与 Spring Service 应得到可比的模型行为（温度、model 名一致）。

调用链

一次 S2 客服提问从 HTTP 进入，到模型返回，经过多层抽象。理解每层职责，排障时才知道日志该打在哪。

CorpAssist S2：同步调用链
sequenceDiagram
  participant U as 员工门户
  participant C as ChatController
  participant CC as ChatClient
  participant CM as ChatModel
  participant GW as LLM 网关
  U->>C: POST /api/chat
  C->>CC: prompt().user(q).call()
  CC->>CM: 组装 Prompt / Options
  CM->>GW: OpenAI 兼容 HTTP
  GW-->>CM: ChatResponse
  CM-->>CC: 文本 + 元数据
  CC-->>C: content()
  C-->>U: JSON 回复

Advisor、Memory、RAG 会在 ChatClient 层挂接（后续章节）；本章先掌握主干。

层次类型职责
ControllerREST鉴权、DTO 校验、request_id
ChatClient门面Fluent API：system/user、options、advisors
ChatModel端口厂商无关的 call(Prompt) / stream(Prompt)
OpenAiApi适配器HTTP 序列化、重试（由 Spring AI 封装）

同步问答：Service 示例

java
@Service
public class CustomerChatService {

  private final ChatClient chatClient;

  public CustomerChatService(ChatClient customerChatClient) {
    this.chatClient = customerChatClient;
  }

  public String ask(String question, String sessionId) {
    return chatClient.prompt()
        .user(u -> u.text(question).param("sessionId", sessionId))
        .call()
        .content();
  }
}
CorpAssist · S2 · 制度 FAQ 员工问「年假还剩几天」——当前章用同步 call() 返回完整句子；下一章用 PromptTemplate 注入制度片段；practice 章再接 RAG。
不推荐
  • Controller 内联一长串 prompt 字符串
  • 直接注入 OpenAiApi 发 HTTP
  • 忽略 ChatResponse 中的 token 用量
推荐
  • Service 封装 ChatClient，Controller 只传 DTO
  • 需要用量时 .call().chatResponse()
  • 日志记录 model 名、latency（advanced 指标章）
先想 10 秒：ChatClient 与 ChatModel 能否只保留一个？

不能省略 ChatModel——它是可替换的「端口」；ChatClient 是易用的「门面」，负责组装 Prompt、Advisor 与默认选项。

多模型 Bean

生产上常用快模型做意图分类与简单 FAQ，强模型做复杂工单总结。Spring 容器里注册多个 ChatModel，用 @Qualifier 注入不同 ChatClient。

java
@Configuration
public class MultiModelConfiguration {

  @Bean
  @Qualifier("fast")
  ChatClient fastChatClient(
      @Qualifier("openAiChatModel") ChatModel chatModel,
      @Value("${corpassist.ai.fast-model:gpt-4o-mini}") String model) {
    return ChatClient.builder(chatModel)
        .defaultOptions(ChatOptions.builder().model(model).temperature(0.0).build())
        .build();
  }

  @Bean
  @Qualifier("quality")
  ChatClient qualityChatClient(
      @Qualifier("openAiChatModel") ChatModel chatModel,
      @Value("${corpassist.ai.quality-model:gpt-4o}") String model) {
    return ChatClient.builder(chatModel)
        .defaultOptions(ChatOptions.builder().model(model).temperature(0.2).build())
        .build();
  }
}
场景推荐 Bean理由
寒暄、分类、是否转人工fastChatClient低延迟、低成本
多轮工单摘要、合规审核草稿qualityChatClient更高推理质量
NS2 工具调用后的结果润色quality（practice 章 Tools）工具返回 JSON，需强模型组织自然语言

advanced 章的 multi model 路由会加入熔断与成本策略；本章只需会注册两路 Bean 并在 Service 中按意图选择。

@Primary

默认注入用；仅应有一个「主」ChatClient，避免歧义。

@Qualifier

显式命名 fast/quality；测试时可替换为 Mock。

配置外部化

model 名来自 corpassist.ai.*，大促切换不改代码。

流式 Flux

门户 SSE 需要逐 token 推送。flux stream 将模型流式响应映射为 Reactor Flux<ChatResponse> 或 Flux<String>。

java
public Flux<String> askStream(String question) {
  return chatClient.prompt()
      .user(question)
      .stream()
      .content();
}

// Controller 侧（practice REST 章细化）
// return ResponseEntity.ok()
//     .contentType(MediaType.TEXT_EVENT_STREAM)
//     .body(askStream(q));
API返回适用
.call()完整字符串后台任务、结构化抽取
.stream().content()Flux<String>SSE、WebFlux 前端
.stream().chatResponse()含 metadata 的块需要每块 token 统计
流式注意点
  • 网关或反向代理需关闭缓冲，否则 SSE 「攒包」看起来像卡住。
  • 客户端断开时要 cancel() 订阅，避免浪费 token（Reactor 会向下游传播取消）。
  • NS2 工具调用链通常先同步拿到 tool 结果再流式输出最终答案（practice 章）。
自检：Flux 与 WebMvc 的 SseEmitter 选哪个？

WebFlux 项目优先 Flux 端到端；传统 Servlet 可用 SseEmitter 包装 Flux 订阅。practice-05 章给出 Controller 模板。

与 Python 客户端对照

python parity 指：同一业务 prompt、同一 model/temperature，在 Python 脚本与 Spring Service 应得到可比对的输出，便于评测与排障。

能力Spring AI 1.0.xPython（OpenAI SDK）
同步补全 chatClient.prompt().user(q).call().content() client.chat.completions.create(...)
流式 .stream().content() → Flux stream=True 迭代 chunk
系统提示 .defaultSystem(...) 或 .system(...) messages=[{"role":"system",...}]
配置 spring.ai.openai.* OPENAI_API_KEY、base_url
Spring · 流式
java
chatClient.prompt().user("制度里年假几天？").stream().content();
Python · 流式
python
for chunk in client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "制度里年假几天？"}],
    stream=True,
):
    print(chunk.choices[0].delta.content or "", end="")

实验与批量评测多在 Python 工程轨python-engineering-for-llm 完成；Spring 轨提供带鉴权与 Advisor 的在线 API。双栈联调时对齐：model 名、temperature、base-url、prompt 文本。

Flux 一定要 WebFlux 吗？
不一定。可在 Servlet 栈订阅 Flux 写入 SseEmitter；但全链路异步时 WebFlux 更自然。
多模型 Bean 与 advanced 路由章区别？
本章是静态 @Qualifier 选择；advanced 加入熔断、成本与动态 fallback 链。

本章结论

  • 调用链：Controller → ChatClient → ChatModel → 网关；排障按层打日志。
  • 多模型 Bean：fast/quality 两路 ChatClient，配置外部化 model 名。
  • 流式 Flux：.stream().content() 对接 SSE，注意代理缓冲与取消订阅。
  • 与 Python 客户端对照：同步/流式/配置一一对应，双栈共用网关做 parity 验证。
