本章先记住 3 件事
  • 路由是业务策略：「带截图消息 → vision 模型」写在 gateway policy，不在 Controller 硬编码。
  • Fallback 有顺序：fallback chain = 主模型 → 备模型 → 缓存 → 固定文案。
  • NS3 看单位成本：cost routing 按 intent + token 预算选 tier，不是永远用最强大模型。

路由策略

CorpAssist · NS3 路由与成本 S2 客服 80% 是 FAQ（年假几天、VPN 怎么连）；15% 需多步推理（对比两版制度差异）；5% 带截图。NS3 模块按 intent 与附件类型选模型，避免全部走 reasoning 档。

路由输入通常包含：tenantId、intent、hasAttachment、estimatedTokens、slaTier。输出为 ModelSelection(modelName, tier, reason)。

ModelRouter 核心实现

java
@Service
public class ModelRouter {

  private final Map<String, ChatModel> models;
  private final RoutingProperties props;

  public RouteDecision route(RouteContext ctx) {
    if (ctx.hasImageAttachment()) {
      return RouteDecision.of("qwen-vl-plus", Tier.VISION, "attachment");
    }
    if (ctx.intent() == Intent.TICKET_ANALYSIS || ctx.estimatedTokens() > 4000) {
      return RouteDecision.of(props.getReasoningModel(), Tier.REASONING, "complex");
    }
    if (ctx.intent() == Intent.FAQ) {
      return RouteDecision.of(props.getFastModel(), Tier.FAST, "faq");
    }
    return RouteDecision.of(props.getDefaultModel(), Tier.STANDARD, "default");
  }

  public ChatModel resolveChatModel(RouteDecision decision) {
    return models.get(decision.modelName());
  }
}

熔断

模型网关 429/5xx 连续失败时，circuit breaker 打开，避免线程池被打满并快速走 fallback chain。

Resilience4j 包装 ChatModel

java
@Component
public class ResilientChatModel implements ChatModel {

  private final ChatModel delegate;
  private final CircuitBreaker breaker;
  private final ChatModel fallbackModel;

  @Override
  public ChatResponse call(Prompt prompt) {
    Supplier<ChatResponse> supplier = CircuitBreaker
        .decorateSupplier(breaker, () -> delegate.call(prompt));
    try {
      return supplier.get();
    } catch (CallNotPermittedException ex) {
      log.warn("breaker open, using fallback model");
      return fallbackModel.call(prompt);
    }
  }
}
yaml
resilience4j.circuitbreaker:
  instances:
    openaiPrimary:
      slidingWindowSize: 20
      failureRateThreshold: 50
      waitDurationInOpenState: 30s
      permittedNumberOfCallsInHalfOpenState: 5

NS3 成本视角：熔断打开后 fallback 到更便宜的 fast 模型通常优于无限重试 reasoning 档；需在 gateway policy 文档中写清降质话术。

自检：熔断打开时是否还要计入 Token 预算？

要。fallback 调用仍产生 Token；指标需 tag model=fallback 与 breaker=open，便于 NS3 成本报表区分。

缓存

FAQ 类问句重复率高，response cache 可在 Router 判定为 Intent.FAQ 且无会话上下文时，用「归一化 question + tenantId」作 key，TTL 15–60 分钟。

语义缓存 Advisor（示意）

java
@Component
@Order(20)
public class ResponseCacheAdvisor implements CallAroundAdvisor {

  private final CacheManager cacheManager;

  @Override
  public AdvisedResponse aroundCall(AdvisedRequest request, CallAroundAdvisorChain chain) {
    if (!isCacheable(request)) {
      return chain.nextAroundCall(request);
    }
    String key = cacheKey(request.tenantId(), normalize(request.userText()));
    Cache cache = cacheManager.getCache("llm-responses");
    String cached = cache.get(key, String.class);
    if (cached != null) {
      return AdvisedResponse.from(cached)
          .mutate().adviseContext(Map.of("cacheHit", true)).build();
    }
    AdvisedResponse resp = chain.nextAroundCall(request);
    cache.put(key, resp.response().getResult().getOutput().getText());
    return resp;
  }

  private boolean isCacheable(AdvisedRequest req) {
    return req.adviseContext().getOrDefault("intent", "") .equals("FAQ")
        && !req.adviseContext().containsKey("conversationId");
  }
}

成本路由

cost routing 在 Router 层增加「预算守卫」：单请求 prompt 预估 + 历史 tenant 用量超阈值时，强制降级 fast tier 或拒答。

CostAwareRouter 扩展

java
public RouteDecision route(RouteContext ctx) {
  RouteDecision base = super.route(ctx);
  if (budgetService.overDailyCap(ctx.tenantId())) {
    return RouteDecision.of(props.getFastModel(), Tier.FAST, "budget_cap");
  }
  if (base.tier() == Tier.REASONING && ctx.estimatedTokens() < 800) {
    return RouteDecision.of(props.getFastModel(), Tier.FAST, "cost_save");
  }
  return base;
}

ChatService 整合路由 + 熔断 + 缓存

java
public String chat(ChatRequest req) {
  RouteDecision route = modelRouter.route(toContext(req));
  ChatClient client = chatClientFactory.forModel(route.modelName());
  return client.prompt()
      .advisors(a -> a.param("intent", req.intent()).param("tenantId", req.tenantId()))
      .user(req.message())
      .call()
      .content();
}
先想 10 秒：成本路由与护栏 Advisor 谁先谁后？

护栏（输入 Sanitizer）始终最先；路由选模型在 ChatClient 构建或 call 前；成本降级是路由策略的一种，不应绕过审核。

本章结论

  • 路由策略：ModelRouter 按 intent/附件选 tier；规则外部化到 yaml。
  • 熔断：Resilience4j 包装 ChatModel；打开时走 fallback 模型并打指标。
  • 缓存：FAQ 无会话可缓存；vision/Tool 副作用禁止缓存。
  • 成本路由：tenant 预算 + 小 prompt 避免滥用 reasoning 档。

判断练习

  1. 所有请求都路由到最强 reasoning 模型，是否有利于 NS3 成本？

    参考答案

    否。 FAQ 应走 fast + 缓存；reasoning 仅用于复杂 intent，否则 Token 账单与延迟都会恶化。

  2. 缓存命中后是否还需要走 OutputModerationAdvisor？

    参考答案

    需要。 缓存内容入库前已审核；若词库更新，应失效旧缓存。返回路径仍建议轻量复核或信任带 version 的缓存条目。
