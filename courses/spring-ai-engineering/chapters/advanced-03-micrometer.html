为什么要学本章

没有延迟、错误率、token usage 指标，无法证明 S2 客服 SLA，也无法验证 NS3 成本路由 是否生效。Spring Boot 3 + micrometer 可导出 Prometheus；配合 otel 与 Python 检索服务共 traceId，双栈联调才能「一条请求查到底」。

学完你能
  • 为 ChatClient 调用注册 llm metrics：请求数、Token、latency histogram
  • 用 Observation / Tracer 把 model 名、tenantId 写入 distributed tracing
  • 配置 Micrometer Tracing → OTLP，与 Jaeger/Tempo 及 Python 侧对齐
  • 搭建 grafana dashboard 看板：P95 延迟、Token 日趋势、熔断次数
本章先记住 3 件事
  • 指标带维度：至少 tag model、tenant、intent、cache_hit。
  • Trace 贯穿 BFF：mdc trace 把 traceId 写入日志与响应头 X-Request-Id。
  • LLM 不是黑盒：Advisor 链每一环可嵌套 span，定位慢在 RAG 还是模型。

指标埋点

CorpAssist 推荐三类 llm metrics（见 demo metrics-list.md）：

  • corpassist.chat.requests — Counter，tag: model, status
  • corpassist.chat.tokens — Counter，tag: model, type=prompt|completion
  • corpassist.chat.latency — Timer，发布 histogram 供 P95

LlmMetricsAdvisor

java
@Component
public class LlmMetricsAdvisor implements CallAroundAdvisor {

  private final MeterRegistry registry;

  @Override
  public AdvisedResponse aroundCall(AdvisedRequest request, CallAroundAdvisorChain chain) {
    String model = (String) request.adviseContext().getOrDefault("model", "unknown");
    Timer.Sample sample = Timer.start(registry);
    try {
      AdvisedResponse resp = chain.nextAroundCall(request);
      registry.counter("corpassist.chat.requests", "model", model, "status", "ok").increment();
      recordTokens(resp, model);
      return resp;
    } catch (Exception ex) {
      registry.counter("corpassist.chat.requests", "model", model, "status", "error").increment();
      throw ex;
    } finally {
      sample.stop(registry.timer("corpassist.chat.latency", "model", model));
    }
  }

  private void recordTokens(AdvisedResponse resp, String model) {
    var usage = resp.response().getMetadata().getUsage();
    if (usage != null) {
      registry.counter("corpassist.chat.tokens", "model", model, "type", "prompt")
          .increment(usage.getPromptTokens());
      registry.counter("corpassist.chat.tokens", "model", model, "type", "completion")
          .increment(usage.getCompletionTokens());
    }
  }
}

Tracing

单次问答可能经过：Gateway → Spring Controller → RAG 检索（HTTP 调 Python）→ ChatModel → Tool。用 Micrometer Observation 为每段创建 span，并在调用 Python 时注入 W3C traceparent。

Observation 包装 ChatService

java
@Service
public class ChatService {

  private final ObservationRegistry observationRegistry;
  private final ChatClient chatClient;

  public String chat(String tenantId, String message) {
    return Observation.createNotStarted("corpassist.chat", observationRegistry)
        .lowCardinalityKeyValue("tenant", tenantId)
        .observe(() -> chatClient.prompt().user(message).call().content());
  }
}

mdc trace：在 Filter 中把 traceId 放入 MDC，日志 pattern 含 %X{traceId}。

java
@Component
public class TraceIdFilter extends OncePerRequestFilter {
  @Override
  protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
      FilterChain chain) throws ServletException, IOException {
    String traceId = Optional.ofNullable(req.getHeader("traceparent"))
        .orElse(UUID.randomUUID().toString());
    MDC.put("traceId", traceId);
    res.setHeader("X-Request-Id", traceId);
    try { chain.doFilter(req, res); } finally { MDC.clear(); }
  }
}

与 OTel 打通

Spring Boot 3.2+ 可通过 micrometer-tracing-bridge-otel + OTLP exporter 把 span 送到 Tempo/Jaeger，与 Python OpenTelemetry SDK 共用 Collector。

依赖与配置

xml
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
  <groupId>io.opentelemetry</groupId>
  <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
yaml
management:
  tracing:
    sampling:
      probability: 1.0
  otlp:
    tracing:
      endpoint: http://otel-collector:4318/v1/traces
spring:
  application:
    name: corpassist-spring

Python 检索服务在 FastAPI 中间件解析同一 traceparent，span 名 python.search 与 Spring corpassist.rag.retrieve 同 trace，联调时可在 Jaeger 一眼看到慢在 embedding 还是 LLM。

自检：生产 sampling probability 是否应永远 1.0？

高流量环境通常 0.1–0.3；CorpAssist 预发可 1.0 便于评测。指标（Counter/Timer）不受采样影响，应全量保留。

Dashboard

grafana dashboard 建议四行：流量与错误率、latency histogram P50/P95、Token 日累计、NS3 路由 tier 占比。

PromQL 示例

bash
# P95 延迟（按 model）
histogram_quantile(0.95, sum(rate(corpassist_chat_latency_bucket[5m])) by (le, model))

# 每分钟 prompt Token
sum(rate(corpassist_chat_tokens_total{type="prompt"}[1m])) by (model)
面板告警阈值（示意）动作
P95 latency> 3s 持续 5m检查 reasoning 占比 / 网关
error rate> 5%看熔断是否打开
Token 日增> 预算 120%收紧 NS3 cost routing
cache hit< 20%审查 FAQ 归一化规则
与 Python 轨对齐
  • 共用 Prometheus remote write 或同一 Grafana 文件夹
  • Trace 用同一 OTel Collector；日志用 traceId 关联
  • 评测跑批时 tag source=eval，避免污染生产 SLA 面板

本章结论

  • 指标埋点：requests / tokens / latency 三件套，Advisor 或 Service 层统一记录。
  • Tracing：Observation + traceparent 贯穿 Python 检索。
  • OTel：OTLP exporter；采样与指标分离配置。
  • Dashboard：P95 + Token + tier 占比支撑 NS3 与 SLA 复盘。
