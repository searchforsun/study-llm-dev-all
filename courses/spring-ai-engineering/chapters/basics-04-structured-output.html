为什么要学本章

S2 客服里「帮我填请假单」不能只返回散文——下游 OA 系统要可解析的 JSON。structured output 把模型输出约束为 record dto，并在 validation error 时走 retry parse 或降级，这是 NS2 办公 Agent 与 Python 轨表单契约的汇合点。

学完你能
  • 开启 json mode 并用 .entity(LeaveRequest.class) 完成 bean mapping
  • 用 schema hint 与 Bean Validation 处理字段缺失
  • 将 LeaveRequest.json 与 CorpAssist form extraction 业务字段对齐，并与 Python Pydantic 模型对照
本章先记住 3 件事
  • 契约优先：先定 JSON 字段与校验规则，再写 prompt 让模型填满。
  • 失败可恢复：解析失败 → 收紧 prompt 重试 → 仍失败则转人工。
  • 双栈同形：Java Record 与 Python Pydantic 字段名一致，便于评测。

JSON mode

厂商支持 response_format: json_object 时，Spring AI 通过 ChatOptions 或 .call().entity(Class) 启用 json mode，减少模型输出 Markdown 包裹或多余解释。

java
LeaveRequest draft = chatClient.prompt()
    .system("""
        你是 CorpAssist 办公助手。仅输出 JSON，字段：employeeId,startDate,endDate,reason。
        日期格式 yyyy-MM-dd。无法推断的字段填 null。
        """)
    .user("我想请 6月1日到2日年假，工号 E1024")
    .call()
    .entity(LeaveRequest.class);
方式优点注意
.entity(Record.class)自动反序列化 + 类型安全需模型遵守 JSON；配合 system 约束
手动 ObjectMapper完全控制错误处理样板代码多
schema hint 在 prompt 中贴 JSON Schema对小模型友好token 变长；与 Bean 注解保持一致
不推荐
  • 用正则从散文里抠 JSON
  • 不校验日期顺序（end < start）
  • 把解析失败当 500 直接抛给用户
推荐
  • Record + Jakarta Validation
  • 失败时 retry parse 一次并收紧 system
  • 审计日志存原始模型输出（脱敏后）
先想 10 秒：流式场景还能用 .entity() 吗？

流式更适合先收全文再解析，或不用流式做表单抽取；门户展示「正在填写」可用进度事件，最终字段仍走同步 entity。

Bean 映射

bean mapping 把 JSON 键映射到 Java record dto。CorpAssist 请假单与 Demo 中 LeaveRequest.json 对齐。

json
{
  "employeeId": "E1024",
  "startDate": "2026-06-01",
  "endDate": "2026-06-02",
  "reason": "年假"
}
java
public record LeaveRequest(
    @NotBlank String employeeId,
    @NotNull LocalDate startDate,
    @NotNull LocalDate endDate,
    String reason) {

  @AssertTrue
  boolean isDateRangeValid() {
    return endDate == null || startDate == null || !endDate.isBefore(startDate);
  }
}
JSON 字段业务含义校验
employeeId工号，对接 HR 主数据@NotBlank
startDate / endDate请假区间LocalDate + 区间合法
reason事由（年假/病假等）可选枚举（扩展）
结构化输出：从自然语言到 OA

校验失败处理

模型 JSON 可能缺字段、日期格式错误或逻辑冲突。validation error 应有分层策略，避免直接把堆栈抛给员工。

失败类型用户可见系统动作
JSON 语法错误「请补充日期与工号」retry parse + 更短 system
Bean Validation「结束日期不能早于开始日期」返回 422 + 字段级错误
业务规则（余额不足）「年假余额不足」调 HR 服务，非 LLM 职责
连续失败 ≥2「已转人工客服」降级 + 工单
java
public LeaveRequest extractLeaveForm(String utterance) {
  try {
    LeaveRequest req = chatClient.prompt()
        .user(utterance)
        .call()
        .entity(LeaveRequest.class);
    validate(req);
    return req;
  } catch (Exception first) {
    LeaveRequest retry = chatClient.prompt()
        .system("上次输出无效。仅输出合法 JSON，不要 markdown。")
        .user(utterance)
        .call()
        .entity(LeaveRequest.class);
    validate(retry);
    return retry;
  }
}

private void validate(LeaveRequest req) {
  var violations = validator.validate(req);
  if (!violations.isEmpty()) {
    throw new FormValidationException(violations);
  }
}
CorpAssist · S2 → 办公 Agent 员工口语「下周请两天假」——需推断具体日期；若模型填 null，应追问而非提交空表单到 OA。NS2 工具可在校验通过后调用「提交草稿」工具（practice 章）。
重试

限 1～2 次，附带错误摘要给 model，避免无限循环烧 token。

降级

转人工或只展示预填草稿让员工改。

审计

存 request_id、原始 JSON、校验结果，供合规追溯。

能否用 function calling 代替 JSON mode？
可以，且对支持 tool 的模型更稳；本章 entity 更易与 Record 校验结合。practice 章对比 @Tool 与 entity 选型。
敏感字段如何防泄露？
输出 Advisor 打码、日志脱敏；DTO 只含业务必要字段（advanced 护栏章）。
常见误区
  • 只信模型不校验 — 幻觉日期会直接进 OA。
  • 字段名中英混用 — 双栈解析失败，评测无法对齐。
  • 无人工确认 — 高合规场景必须展示草稿再提交。

评测与观测挂钩

表单抽取应记录：model 名、解析是否一次成功、校验失败字段列表、是否触发 retry。与 Python 轨共用 golden utterance → JSON 用例，在 CI 用 @MockBean 固定模型输出为 LeaveRequest.json，只测校验与 OA 映射逻辑。

指标名建议：llm.form.extract.success、llm.form.extract.retry，便于与 S2 对话指标在同一 Grafana 看板对比（advanced Micrometer 章展开）。

  • 一次成功：直接返回 DTO 给门户预览。
  • retry 后成功：日志打 warn，仍允许用户确认。
  • 两次失败：触发转人工，不调用 OA 工具。

本章结论

  • JSON mode：用 entity/API 约束输出形状，system 明确「仅 JSON」。
  • Bean 映射：Record + Validation 对接 OA；与 Demo LeaveRequest.json 同形。
  • 校验失败处理：有限重试、422 字段错误、多次失败转人工。
  • CorpAssist 表单抽取：意图 → 抽取 → 校验 → 确认 → NS2 工具提交，全链路可审计。
