为什么要学本章

上一章 已为 S2 客服链路埋点；若 CI 仍调用真实模型网关，MR 会因网络抖动、配额、随机性而红绿不定。CorpAssist 要求：业务逻辑、Advisor 顺序、RAG 引用格式必须在无 LLM 密钥的环境跑通。mock chat model 与 testcontainers 是 Java 轨与 Python 轨 pytest + httpx mock 的对等实践——Spring 侧用 mockbean 替换 ChatModel Bean，向量层用 PGVector 容器，Prompt 渲染用 snapshot test 锁结构。

学完你能
  • 在 @SpringBootTest 中用 mockbean 注入固定 ChatResponse，断言 RAG API 含 sources[] 与 Advisor 行为
  • 用 Testcontainers 启动 PostgreSQL + pgvector，验证 VectorStore 相似度检索与 ACL filter
  • 对 Prompt 模板与 OpenAPI 响应做 contract snapshot，MR 时人工审核 diff
  • 设计 ci pipeline：单元 / 集成 / 契约三层，并配置 flaky guard 禁止真实 LLM 调用
本章先记住 3 件事
  • 测行为不测文采：断言结构（引用 docId、降级码、HTTP 状态），不断言模型「措辞是否优美」。
  • 分层替身：ChatModel → MockBean；VectorStore → Testcontainers；Python 检索 → wiremock llm / WireMock。
  • CI 零密钥：spring.ai.openai.api-key 在 CI 留空时测试仍绿，靠 Mock 与 Testcontainers 覆盖。

@MockBean ChatModel

CorpAssist · NS1 制度问答 员工问「年假如何申请？」——Controller 应返回带 pol-001 引用的 JSON。测试只需验证：RAG Advisor 是否把检索片段拼进 Prompt、Mock 模型是否被调用一次、响应 DTO 字段完整；无需真实 GPT 生成段落。

Spring Boot 3.3 + Spring AI 1.0.x 中，自动配置会注册 ChatModel Bean。在测试类上用 @MockBean ChatModel chatModel 可替换容器内 Bean，配合 Mockito 返回固定 ChatResponse。

MockChatModelTest（见 demo）

java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureMockMvc
class MockChatModelTest {

  @Autowired MockMvc mockMvc;

  @MockBean ChatModel chatModel;

  @Test
  void ragQueryReturnsCitation() throws Exception {
    when(chatModel.call(any(Prompt.class))).thenReturn(
        ChatResponse.builder()
            .generations(List.of(new Generation(
                new AssistantMessage("依据 [pol-001]，年假至少 5 天，需提前在 OA 提交。"))))
            .metadata(Map.of("model", "mock"))
            .build());

    mockMvc.perform(post("/v1/rag/query")
            .contentType(MediaType.APPLICATION_JSON)
            .content("""
                {"query":"年假如何申请？","topK":5}
                """))
        .andExpect(status().isOk())
        .andExpect(jsonPath("$.answer").value(containsString("pol-001")))
        .andExpect(jsonPath("$.sources[0].docId").exists());

    verify(chatModel, times(1)).call(any(Prompt.class));
  }
}
@Mock 局部 new ChatService
  • 绕开 Spring 容器，Advisor 链未装配
  • 测不到 @Order 与 Filter 交互
  • 与生产 Bean 图不一致
@MockBean + @SpringBootTest
  • 真实 ChatClient、Advisor 顺序生效
  • 仅替换最外层 ChatModel
  • 可叠加 MockMvc 测 REST 契约

流式 SSE 测试要点

S2 客服 SSE 路径应对 StreamingChatModel 或 chatModel.stream() 使用 Flux<ChatResponse> 的 stub。断言：首 event 含 event: message、结束 event 含 [DONE]；勿在 CI 等待真实 token 流。

先想 10 秒：为何 Mock ChatModel 而不是 Mock ChatClient？

ChatClient 是门面，内部仍走 Advisor 链与 Observation。Mock ChatClient 会跳过 RAG/护栏测试；Mock ChatModel 保留链路上游逻辑，只切断外部 LLM。

Testcontainers

VectorStore 集成测需要真实 SQL 与 pgvector 运算符；H2 不支持 vector 类型。Testcontainers 在 CI 拉取 pgvector/pgvector:pg16 镜像，@DynamicPropertySource 注入 JDBC URL。

PGVector 容器 + Spring AI VectorStore

java
@Testcontainers
@SpringBootTest
class VectorStoreIntegrationTest {

  @Container
  static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>(
      "pgvector/pgvector:pg16")
      .withDatabaseName("corpassist")
      .withUsername("test")
      .withPassword("test");

  @DynamicPropertySource
  static void datasourceProps(DynamicPropertyRegistry registry) {
    registry.add("spring.datasource.url", postgres::getJdbcUrl);
    registry.add("spring.datasource.username", postgres::getUsername);
    registry.add("spring.datasource.password", postgres::getPassword);
  }

  @Autowired VectorStore vectorStore;
  @MockBean ChatModel chatModel; // embedding 仍 Mock，见下节

  @Test
  void similarSearchRespectsDeptFilter() {
    vectorStore.add(List.of(
        new Document("pol-001 年假制度", Map.of("dept", "HR")),
        new Document("it-002 VPN 手册", Map.of("dept", "IT"))));

    var results = vectorStore.similaritySearch(
        SearchRequest.query("年假").withTopK(3)
            .withFilterExpression("dept == 'HR'"));

    assertThat(results).hasSize(1);
    assertThat(results.get(0).getMetadata()).containsEntry("dept", "HR");
  }
}

Embedding 策略：集成测可 (1) Mock EmbeddingModel 返回固定 1536 维向量，或 (2) 使用 SimpleVectorStore 内存实现测 filter 逻辑。CorpAssist 推荐 (1)+Testcontainers 组合，避免 CI 调 embedding API。

快照测试

snapshot test 将「Prompt 渲染结果」「OpenAPI JSON」「RAG 响应样例」写入 src/test/resources/__snapshots__/。下次运行 byte-level 或 JSON 结构化对比；变更需 MR 里人工说明「为何 Prompt 结构变了」。

Prompt 模板快照（AssertJ）

java
@Test
void ragSystemPromptMatchesSnapshot() {
  String rendered = promptRenderer.render("rag-system.st",
      Map.of("context", "[pol-001] 年假...", "query", "如何申请？"));

  // 首次运行生成 snapshot；后续 compare
  assertThat(rendered)
      .isEqualToNormalizingNewlines(
          Files.readString(Path.of("src/test/resources/snapshots/rag-system.txt")));
}

OpenAPI 契约快照

与 Python 轨共用 contracts/rag-query.openapi.yaml。Spring 测试导出 /v3/api-docs，与 baseline JSON diff；禁止静默删除 sources 或 requestId 字段。Python 侧 FastAPI 同样对比，实现 contract snapshot 双栈一致。

快照含动态 UUID / 时间戳
  • 每次 CI 都失败
  • Review 噪音大
规范化后再快照
  • 替换 requestId 为 <REQ_ID>
  • 只快照 schema 与 Prompt 结构
自检：快照失败时能否直接 --updateAll？

仅本地开发可更新；CI 必须 fail 并让人审 diff。否则 Prompt 注入或字段删除会静默上线。

CI

ci pipeline 建议三 job：unit（无 Docker）、integration（Testcontainers，需 Docker-in-Docker）、contract（OpenAPI + WireMock）。环境变量 CORPASSIST_LLM_API_KEY 在 CI 不设置，由 flaky guard 断言无 outbound 调用模型域名。

GitHub Actions 片段

yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: "21"
          distribution: temurin
      - name: Unit + Integration
        run: ./mvnw -B verify -Dspring.profiles.active=test
        env:
          TESTCONTAINERS_RYUK_DISABLED: false
      - name: Fail if real LLM called
        run: ./scripts/assert-no-llm-calls.sh target/test-traffic.log

WireMock 模拟 Python 检索

java
@RegisterExtension
static WireMockExtension pythonSearch = WireMockExtension.newInstance()
    .options(wireMockConfig().dynamicPort())
    .build();

@DynamicPropertySource
static void pythonUrl(DynamicPropertyRegistry r) {
  r.add("corpassist.python-search.base-url", pythonSearch::baseUrl);
}

@BeforeEach
void stubPythonSearch() {
  pythonSearch.stubFor(post(urlEqualTo("/v1/search"))
      .willReturn(okJson("""
          {"chunks":[{"docId":"pol-001","text":"年假..."}]}
          """)));
}

与 Python 轨对齐：Java CI 验 Spring 契约；Python CI 验 FastAPI 契约； nightly 可选双栈联调冒烟（真实密钥仅在密钥库）。

本章结论

每条对应上一节 h3。

  • @MockBean ChatModel：保留 Advisor 链，只 stub 模型输出；MockMvc 断言 REST 与引用。
  • Testcontainers：PGVector 验 VectorStore 与 metadata filter；Embedding 仍 Mock。
  • 快照测试：Prompt / OpenAPI / 响应结构；规范化动态字段后再对比。
  • CI：零密钥、WireMock Python、flaky guard；单元与集成分 job。

判断练习

  1. QA 要求「CI 必须调用真实 GPT 才能合并」，是否应采纳？

    参考答案

    不应。 真实 LLM 导致 flaky、成本与密钥泄露风险。应用 Mock + 快照 + 预发/nightly 冒烟；评测集回归放在受控环境。

  2. 只测 ChatService 单元测试、不测 MockMvc，能否保证 REST 契约正确？

    参考答案

    不够。 序列化、状态码、响应头（X-Request-Id）在 Controller 层；需 MockMvc 或 WebTestClient 集成测。
