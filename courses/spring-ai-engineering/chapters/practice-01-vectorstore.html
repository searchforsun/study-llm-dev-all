接口模型：Document、Embedding 与 VectorStore

CorpAssist · S2 制度问答 HR 上传《员工手册》片段后，客服 Bot 需根据员工所属部门检索相关制度。Spring AI 用 Document 承载文本 + metadata，EmbeddingModel 转向量，VectorStore 负责写入与 similarity search。

Spring AI 1.0 的 vector store 是统一 SPI：业务只依赖接口，具体实现由 Starter 或 @Bean 注入。核心方法：

  • add(List<Document>)：批量写入；Document 含 id、text、metadata Map。
  • similaritySearch(SearchRequest)：按 query 向量 + topK + filter 返回最相似 Document。
  • delete(filter)：按 metadata 或 id 清理过期索引（制度版本升级时用）。

检索时 query 与入库文档走同一 EmbeddingModel，模型不一致会导致相似度失真。

CorpAssist 最小配置

java 
@Configuration
public class VectorStoreConfig {

  @Bean
  VectorStore vectorStore(EmbeddingModel embeddingModel) {
    return SimpleVectorStore.builder(embeddingModel).build();
  }

  @Bean
  EmbeddingModel embeddingModel(OpenAiEmbeddingModel openAi) {
    return openAi; // 与 Python 侧共用 text-embedding-3-small
  }
}

每个 Document 的 document id 建议稳定可追踪，例如 pol-001，便于 RAG 引用溯源（见 RAG Pipeline 实现）。

FAQ：VectorStore 与 ChatModel 能否用不同厂商？

可以，但生产环境建议统一网关与计费账户；Embedding 与 Chat 模型维度、tokenizer 不同不影响存储，只要入库与检索使用同一 EmbeddingModel。

Simple vs 外部向量库

本地开发用内存型 SimpleVectorStore；联调与生产切换到 PgVector、Redis、Milvus 等，仅改依赖与配置，业务 Service 不变。

元数据过滤：部门隔离与 topK

S2 场景中，FIN 员工问报销不应检索到 HR 专用制度。Spring AI 的 SearchRequest 支持 FilterExpression，与 chunk meta 字段对齐。

java 
@Service
public class PolicySearchService {
  private final VectorStore vectorStore;

  public List<Document> searchForDept(String question, String dept, int topK) {
    SearchRequest request = SearchRequest.builder()
        .query(question)
        .topK(topK)
        .filterExpression("dept == '" + dept + "'")
        .build();
    return vectorStore.similaritySearch(request);
  }

  public void ingestPolicy(Document doc) {
    vectorStore.add(List.of(doc));
  }
}

Demo 中 sample-documents.json 两条制度分别带 dept: HR 与 dept: FIN。验收：对「年假」提问且 filter dept == 'HR' 应命中 pol-001，filter FIN 时不应返回 HR 片段。

检索后内存过滤
  • topK=10 再 Java stream filter
  • 相似条目被其他部门占满 topK
  • latency 与 token 浪费
存储层 filter
  • filter 下推至 VectorStore
  • topK 在目标集合内有效
  • 审计日志记录 filter 表达式

Python 索引同源：双栈契约

CorpAssist 双栈分工：Python 轨批量解析 PDF、切块、调用 Embedding API 写入向量库；Java 轨在运行时检索并注入 RAG Advisor。两边必须共享索引契约，避免「Python 建了索引、Java 查不到」。

Python 建索引 · Java 消费检索

  Java->>VS: similaritySearch(filter=dept)

  • embedding 模型名：写入配置中心，Python/Java 读同一 key。
  • chunk 参数：size 500、overlap 50（与 RAG Pipeline 实现 Splitter 一致）。
  • id 规则：{docId}-{chunkIndex}，删除旧版时 Python 脚本按 docVersion 批量 delete。
  • 验收：Python 写入后，Java 集成测试对固定 query 断言 top1 id 与 Python eval 一致。

FAQ：similaritySearch 的 topK 设多少？

S2 制度问答默认 topK=4～6；过多 chunk 会撑爆 prompt token。与 RAG Advisor 的 SearchRequest 保持一致，并在 Micrometer 记录「检索条数 vs 实际注入条数」（advanced 章）。

判断练习

  1. 同事在 similaritySearch 后 Java 里 .filter(d -> "HR".equals(d.getMetadata().get("dept")))。应如何改？风险是什么？

    参考答案

    改为 SearchRequest 的 filterExpression 下推。风险：topK 被其他部门占满导致 HR 条目漏检，且无法利用向量库索引。

  2. Python 用 text-embedding-ada-002 建库，Java 用 text-embedding-3-small 检索。现象与根因？

    参考答案

    检索结果随机或相似度极低。根因：入库与查询 EmbeddingModel 不一致，违反 indexContract 的 embeddingModel 字段。
