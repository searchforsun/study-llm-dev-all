为什么要学本章

代码索引是所有代码助手场景的基础设施。没有索引就没有 RAG，没有增量更新就没有实时性。本章从零构建一个 Git 增量驱动的代码索引流水线：从仓库推送触发、AST 解析切片、Embedding 向量化，到存入 Milvus 向量库并施加权限过滤。index pipeline 是 CorpAssist 的「数据发动机」。

本章先记住 3 件事
  • 增量索引 > 全量重建：Git push 触发增量更新，秒级完成，而非全量重建的数十分钟。
  • Embedding 是检索质量的瓶颈：代码专用 Embedding 模型比通用模型准确率高 20%+。
  • 权限必须内置：索引层就要做权限过滤，而不是在 API 层事后补。

CorpAssist · 增量索引流程 开发者推送了一个 commit，修改了 OrderService.java 中的 createOrder 方法并新增了 PaymentService.java。CorpAssist 的 Git webhook 收到 push 事件后：① 对比 diff 识别变更文件；② 只解析这 2 个文件的新增/修改部分；③ 向量化新的代码块；④ 更新 Milvus 中对应 chunk（覆盖旧版本）；⑤ 刷新索引缓存。整个过程约 2-5 秒，不影响正常服务。

Git 增量索引：只处理变更

git incremental 策略的核心是：只处理 Git diff 中涉及的文件，不重新解析未变更的代码。这比全量索引快 10-50 倍。incremental index 的难点在于「旧 chunk 的失效检测」——修改后的函数必须覆盖旧索引。

Python：增量索引触发器

python 
# CorpAssist Git 增量索引器
import subprocess
import json
from pathlib import Path

class IncrementalIndexer:
    def __init__(self, repo_path: str, indexer):
        self.repo_path = Path(repo_path)
        self.indexer = indexer

    def process_push(self, old_rev: str, new_rev: str) -> dict:
        """处理 Git push 事件，增量更新索引"""
        changed_files = self._get_changed_files(old_rev, new_rev)
        stats = {"files_changed": 0, "chunks_added": 0,
                 "chunks_removed": 0, "errors": []}

        for file_path in changed_files:
            full_path = self.repo_path / file_path
            if not full_path.exists():
                # 文件被删除，移除索引
                removed = self.indexer.remove_file(file_path)
                stats["chunks_removed"] += removed
                continue

            try:
                # 解析新版本，覆盖旧索引
                new_chunks = self.indexer.index_file(str(full_path))
                self.indexer.overwrite_chunks(file_path, new_chunks)
                stats["chunks_added"] += len(new_chunks)
                stats["files_changed"] += 1
            except Exception as e:
                stats["errors"].append(f"{file_path}: {e}")

        return stats

    def _get_changed_files(self, old_rev: str,
                           new_rev: str) -> list:
        """获取两次提交间的变更文件列表"""
        result = subprocess.run(
            ["git", "diff", "--name-only",
             old_rev, new_rev],
            capture_output=True, text=True,
            cwd=self.repo_path
        )
        files = result.stdout.strip().split('\n')
        # 过滤出代码文件
        code_exts = {'.py', '.java', '.go', '.js', '.ts'}
        return [f for f in files
                if Path(f).suffix in code_exts]

    def full_reindex(self, branch: str = "main"):
        """全量重建（用于初始或修复）"""
        files = Path(self.repo_path).rglob("*")
        code_files = [f for f in files
                      if f.suffix in {'.py', '.java', '.go'}]
        for f in code_files:
            rel_path = str(f.relative_to(self.repo_path))
            chunks = self.indexer.index_file(str(f))
            self.indexer.overwrite_chunks(rel_path, chunks)

    @property
    def branch_scope(self) -> list:
        """返回配置的索引分支列表"""
        return ["main", "develop"]
不推荐
  • 每次 push 触发全量重建，耗时 30 分钟+
  • 所有分支都索引，浪费存储和计算
  • 删除文件后不清理索引，产生脏数据
  • 不处理并发 push 的冲突
推荐
  • 增量索引只处理 diff 文件
  • 只索引 main/develop 等核心分支
  • 文件删除时同步清除对应 chunk
  • 使用分布式锁处理并发 push
检查理解：增量索引和全量重建如何配合使用？

增量索引处理日常变更（秒级），全量重建用于初始构建或数据修复（定时每周一次凌晨执行）。增量索引失败时自动回退到该文件的上一版本索引，不影响整体服务。全量重建前先验证增量日志是否有累计偏差。

Code Embedding：代码专用的向量化

code embedding 是代码检索质量的核心。通用文本 Embedding（如 text-embedding-ada-002）在代码上的表现不如专用模型。CorpAssist 使用专门的代码 Embedding 模型，支持跨语言语义对齐。

Python：向量化批处理

python 
# CorpAssist 批处理向量化
import numpy as np
from sentence_transformers import SentenceTransformer

class CodeEmbeddingService:
    def __init__(self, model_name: str = "code-embed-v1"):
        self.model = SentenceTransformer(model_name)

    def embed_chunks(self, chunks: list) -> np.ndarray:
        """批处理向量化"""
        texts = [c["text"] for c in chunks]
        # 添加元数据特征增强
        augmented = []
        for chunk in chunks:
            meta = chunk["metadata"]
            augmented.append(
                f"{meta['language']} {meta['type']} "
                f"{meta['name']}\n{chunk['text']}"
            )
        embeddings = self.model.encode(
            augmented,
            batch_size=32,
            show_progress_bar=True
        )
        return embeddings

    def embed_query(self, query: str) -> np.ndarray:
        """查询向量化"""
        # 添加代码查询前缀
        enhanced = f"code query: {query}"
        return self.model.encode([enhanced])[0]
常见误区：Embedding 模型不更新

代码 Embedding 模型需要定期用公司代码库微调。私有代码中的特定缩写、框架名、业务术语可能不在通用模型的训练集中。建议每季度用最新的代码库数据微调一次 Embedding 模型。

Milvus 集合设计与权限控制

milvus collection 是 Milvus 中的表结构设计。CorpAssist 采用「每个租户一个 collection，每个 chunk 带权限标签」的混合策略。

Java：Milvus 集合管理与权限查询

java 
// CorpAssist Milvus 集合管理器
public class MilvusCollectionManager {
    private final MilvusServiceClient client;

    public MilvusCollectionManager(MilvusServiceClient client) {
        this.client = client;
    }

    public void createCodeChunksCollection(String tenantId) {
        String collectionName = "code_chunks_" + tenantId;
        var schema = CollectionSchema.newBuilder()
            .addField("chunk_id", DataType.VarChar)
            .addField("embedding", DataType.FloatVector, 768)
            .addField("repo_id", DataType.VarChar)
            .addField("file_path", DataType.VarChar)
            .addField("start_line", DataType.Int64)
            .addField("language", DataType.VarChar)
            .addField("visibility", DataType.VarChar)
            .build();

        var params = IndexParam.newBuilder()
            .withFieldName("embedding")
            .withIndexType(IndexType.IVF_FLAT)
            .build();

        client.createCollection(CreateCollectionParam.newBuilder()
            .withCollectionName(collectionName)
            .withSchema(schema)
            .build());
        client.createIndex(CreateIndexParam.newBuilder()
            .withCollectionName(collectionName)
            .withIndexParam(params)
            .build());
    }

    public SearchResult searchWithACL(String tenantId,
                                       float[] queryVector,
                                       String userRole,
                                       int topK) {
        String collectionName = "code_chunks_" + tenantId;
        // 使用布尔表达式做权限过滤
        String filter = userRole.equals("admin")
            ? "" : "visibility in ['public', 'internal']";

        return client.search(SearchParam.newBuilder()
            .withCollectionName(collectionName)
            .withVectors(Collections.singletonList(queryVector))
            .withVectorFieldName("embedding")
            .withTopK(topK)
            .withExpr(filter)
            .build());
    }
}

repo acl 权限模型分为三层：租户隔离（collection 级别）、仓库可见性（repo_id 过滤）、文件权限（visibility 字段）。branch scope 决定哪些分支的代码被索引——通常只索引 main 和 release 分支，避免 feature 分支的临时代码污染索引。

常见问题（FAQ）

增量索引中，如果两个 push 几乎同时到达，怎么处理冲突？
用乐观锁 + 文件级锁机制。每个文件有一个版本号（基于 commit SHA），索引写入时检查版本号。如果写入时的版本号低于当前索引的版本号，说明该文件已被更新的 commit 处理，跳过本次写入。同时用消息队列缓冲 push 事件，消费端单线程逐个处理。
Embedding 模型的维度（768）是怎么确定的？可以改吗？
768 是 code-embed-v1 模型的默认输出维度。Milvus 支持不同维度的向量，但同一 collection 中的向量维度必须一致。如果需要切换模型（如从 768 维切到 1024 维），需要重建 collection。建议上线前确定模型，避免中途切换。
索引失败会影响线上服务吗？如何保证高可用？
索引和检索是分离的，索引失败不影响检索服务。检索读的是已经索引好的数据。如果增量索引失败，CorpAssist 会：① 记录失败日志到死信队列；② 该文件的索引回退到上一个成功版本；③ 告警通知运维。索引服务本身无状态，可以多副本部署。
不推荐
  • 所有租户共享一个 collection，无隔离
  • 权限过滤在 API 层做，索引层不感知
  • 所有分支都索引，包括 feature 分支
  • 索引失败后无降级策略
推荐
  • 每个租户独立 collection
  • 索引层通过字段过滤实现权限
  • 只索引 main/release 等稳定分支
  • 索引失败回退上一版本 + 死信队列

本章结论

  • 增量索引核心：git incremental 只处理 diff 文件，秒级完成。
  • 代码专用 Embedding：code embedding 比通用模型 Recall 高 20%+。
  • Milvus 多租户设计：每个租户独立 collection，权限在索引层过滤。
  • 分支范围控制：branch scope 只索引稳定分支，避免脏数据。
  • 可观测性：索引流水线的每个阶段都需要监控速率、延迟和错误率。

判断练习

  1. 全量重建时线上检索服务还在运行，怎么保证不丢数据？

    参考答案

    采用双 collection 策略：重建时写入一个新的 collection（code_chunks_v2），完成后原子切换别名。检索服务始终读别名指向的 collection。切换后旧 collection 保留一段时间用于回退。

  2. 开发者发现搜索结果是旧版本的代码（刚 push 但还没搜到）。怎么解决？

    参考答案

    延迟容忍度取决于场景。对于补全，秒级延迟可以接受（开发者刚提交的代码不会立即用到）。对于问答，可以在响应中标注索引时间戳，并提示开发者「索引延迟约 1-2 分钟」。极端实时性需求的场景可以用 LSP 直接读取磁盘文件。
