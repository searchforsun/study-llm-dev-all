为什么要学本章

LLM 的上下文窗口有限但代码仓库通常包含数万文件。不加管理地「整库塞入 prompt」会导致 token budget code 爆炸，LLM 丢失关键上下文。本章教你在代码场景中如何分配窗口预算、选择相关文件、构建符号表，以及利用 LSP 获取实时上下文——这是代码助手准确性的基石。

本章先记住 3 件事
  • 窗口是稀缺资源：上下文窗口不是无限缓存，需要显式管理每块内容的 Token 预算。
  • 文件选择决定准确性：给 LLM 100 个无关文件，不如给 5 个高度相关的文件效果好。
  • 符号表是结构化的理解：用类名、方法签名、变量类型帮助 LLM 理解代码结构，远胜于原始文本。

上下文预算：为代码场景分配 Token 窗口

window budget 是上下文工程的第一步。代码场景的上下文包含五部分：系统指令、光标上下文（当前文件）、相关代码片段（RAG 检索）、对话历史、工具输出。每部分都需要明确的 Token 预算上限。

层级内容补全场景占比问答场景占比
System Prompt 角色定义、输出格式、风格约束 10% (~1K) 10% (~3K)
Cursor Context 光标附近的代码（前缀+后缀） 40% (~4K) 5% (~1.5K)
Retrieved Chunks RAG 检索的相关代码片段 25% (~2.5K) 50% (~15K)
Conversation History 多轮对话上下文 5% (~0.5K) 20% (~6K)
Tool Outputs LSP 查询结果、编译输出 10% (~1K) 10% (~3K)
Reserve 模型输出 + 缓冲 10% (~1K) 5% (~1.5K)

Python：上下文预算验证器

python 
# CorpAssist 上下文预算验证器
import tiktoken

class ContextBudget:
    SCENARIOS = {
        "completion": {
            "max_tokens": 10000,
            "alloc": {
                "system": 0.10, "cursor": 0.40, "retrieval": 0.25,
                "history": 0.05, "tool": 0.10, "reserve": 0.10,
            }
        },
        "qa": {
            "max_tokens": 32000,
            "alloc": {
                "system": 0.10, "cursor": 0.05, "retrieval": 0.50,
                "history": 0.20, "tool": 0.10, "reserve": 0.05,
            }
        }
    }

    def __init__(self, scenario: str = "completion"):
        self.cfg = self.SCENARIOS[scenario]
        self.encoder = tiktoken.get_encoding("cl100k_base")
        self.usage = {k: 0 for k in self.cfg["alloc"]}

    def _count_tokens(self, text: str) -> int:
        return len(self.encoder.encode(text))

    def add(self, layer: str, content: str) -> bool:
        tokens = self._count_tokens(content)
        max_layer = int(self.cfg["max_tokens"] * self.cfg["alloc"][layer])
        if self.usage[layer] + tokens > max_layer:
            return False
        self.usage[layer] += tokens
        return True

    def validate(self) -> dict:
        total = sum(self.usage.values())
        limits = {k: int(self.cfg["max_tokens"] * v)
                  for k, v in self.cfg["alloc"].items()}
        return {
            "total": total,
            "max": self.cfg["max_tokens"],
            "usage_pct": round(total / self.cfg["max_tokens"] * 100, 1),
            "by_layer": {k: {"used": self.usage[k], "limit": limits[k]}
                         for k in self.cfg["alloc"]}
        }

budget = ContextBudget("qa")
budget.add("system", "你是一个专业的代码助手...")
budget.add("retrieval", "def calculate_tax...")
print(budget.validate())
不推荐
  • 无预算规划，所有内容堆到一个 prompt
  • 把整个仓库文件全部塞入上下文
  • 对话历史无限制累积到 50+ 轮
推荐
  • 显式分配每层的 budget split
  • 检索的代码片段按相关性裁剪
  • 历史对话使用滑动窗口限制轮次
常见误区
  • 以为「模型窗口是 128K 就不用管预算」——窗口越大越需要管理，否则 LLM 注意力会被稀释。
  • Retrieval 和 Cursor Context 相互挤压——补全场景应优先保障光标上下文，问答场景应优先保障检索片段。

文件选择策略：找到对的文件

file selection 解决「从仓库数万文件中选出最相关的几个文件」。在代码补全场景中，光标位置附近的相关文件决定了补全质量；在问答场景中，相关文件是 RAG 检索的基石。

CorpAssist · 文件选择流程 开发者在一个 Spring Boot 项目中编写 OrderService，光标在 createOrder 方法内。CorpAssist 需要选择哪些文件作为上下文？流程：① 解析当前文件的 imports，找到 ProductRepository、InventoryClient；② 从符号表中查找这些类的定义文件；③ 根据 Git 提交历史找最近修改的相关文件。最终选中 5 个文件注入上下文。

三级文件选择策略

级别策略方法典型数量
L1 精确匹配 当前文件 + import/依赖文件 AST 解析 import 语句 + 符号解析 2-5 个
L2 语义匹配 向量检索最相似的代码块 Embedding + Milvus ANN 搜索 3-8 个
L3 频率匹配 近期 Git 变更 / 高频修改文件 Git log 分析 + TF-IDF 2-3 个

Java：文件选择器实现

java 
public class FileSelector {
    private final SymbolTable symbolTable;
    private final VectorStore vectorStore;
    private final GitHistoryAnalyzer gitAnalyzer;

    public FileSelector(SymbolTable symbolTable,
                        VectorStore vectorStore,
                        GitHistoryAnalyzer gitAnalyzer) {
        this.symbolTable = symbolTable;
        this.vectorStore = vectorStore;
        this.gitAnalyzer = gitAnalyzer;
    }

    public List selectFiles(CursorContext cursor, int maxFiles) {
        Set selected = new LinkedHashSet<>();
        selected.add(cursor.currentFile());
        selected.addAll(resolveImports(cursor));

        if (selected.size() < maxFiles) {
            List semantic = vectorStore.search(
                cursor.getSurroundingCode(), maxFiles - selected.size());
            selected.addAll(semantic);
        }
        if (selected.size() < maxFiles) {
            List recent = gitAnalyzer.recentlyModified(
                cursor.currentFile(), maxFiles - selected.size());
            selected.addAll(recent);
        }
        return new ArrayList<>(selected).stream()
            .limit(maxFiles).collect(Collectors.toList());
    }

    private List resolveImports(CursorContext cursor) {
        return symbolTable.getDefinitions(cursor.getImportSymbols());
    }
}
检查理解：为什么 L1 精确匹配比 L2 语义匹配优先级更高？

当前文件调用的类和方法直接影响补全的语法正确性。先确保「编译不报错」再优化「语义相关性」。如果缺少依赖类的上下文，LLM 可能生成未导入的类型名称。

符号表：结构化代码理解

symbol table 是代码的「注册表」。它将源代码中的类、方法、变量、接口等符号提取为结构化索引，供文件选择和上下文组装使用。符号表 ast context 比纯文本检索更精确——它知道 UserService 定义在哪个文件、有哪些方法签名、继承关系如何。

符号表从 AST 提取结构化信息，支持多种查询模式。

Python：Tree-sitter 符号提取

python 
from tree_sitter import Language, Parser

class SymbolTableBuilder:
    QUERIES = {
        "java": """
            (class_declaration
              name: (identifier) @class.name) @class
            (method_declaration
              name: (identifier) @method.name) @method
        """,
        "python": """
            (class_definition
              name: (identifier) @class.name) @class
            (function_definition
              name: (identifier) @func.name) @function
        """,
        "go": """
            (method_declaration
              name: (field_identifier) @method.name) @method
            (function_declaration
              name: (identifier) @func.name) @function
        """
    }

    def __init__(self):
        self.parser = Parser()
        self.symbols = {}

    def extract_from_file(self, filepath: str) -> list:
        with open(filepath, 'rb') as f:
            tree = self.parser.parse(f.read())
        lang = filepath.split('.')[-1]
        query_str = self.QUERIES.get(lang, "")
        if not query_str:
            return []
        query = Language(lang).query(query_str)
        captures = query.captures(tree.root_node)
        symbols = []
        for node, tag in captures:
            symbols.append({
                "name": node.text.decode(),
                "kind": tag,
                "file": filepath,
                "line": node.start_point[0] + 1,
            })
            self.symbols[node.text.decode()] = symbols[-1]
        return symbols
常见误区
  • 把符号表当全文索引——符号表存的是「符号名→位置」映射，不是代码内容。检索内容仍需向量或全文索引。
  • 忽略跨语言符号——微服务项目中 Java 调 Go 服务，符号表需要支持跨语言引用解析。

LSP 概念：实时代码上下文

lsp concept（Language Server Protocol）是 IDE 与语言服务之间的标准化协议。在代码助手场景中，LSP 提供光标位置处的精确上下文：当前符号定义、类型推断、引用跳转、诊断信息。这些信息比纯文本解析更准确、更实时。

LSP 在代码助手中的关键能力

LSP 能力协议方法代码助手用途
跳转到定义textDocument/definition解析类/方法定义，注入文件上下文
悬停信息textDocument/hover获取变量类型、方法签名文档
查找引用textDocument/references识别哪些地方使用了当前符号
诊断textDocument/diagnostic检测编译错误，过滤无效补全建议
完成项textDocument/completion作为 AI 补全的候选参考

relevant files 的精确识别是 LSP 与符号表协同的结果：LSP 提供光标处的精确符号引用，符号表将符号名映射到文件路径，最终选出最相关的几个文件注入上下文。

常见问题（FAQ）

LSP 查询会引入额外延迟吗？IDE 响应变慢怎么办？
LSP 查询通常 < 50ms，但如果 LSP Server 本身很慢（如大型 TypeScript 项目），可以缓存结果并用异步查询。策略：首次查询 LSP 后缓存符号位置，后续直接读缓存。另外控制 LSP 查询频次——不是每次键入都查，只在光标停留 > 200ms 时触发。
符号表多久更新一次？CI 流程中怎么同步？
推荐 Git post-commit hook 触发增量更新：只重新解析变更文件的符号。全量重建每天一次（凌晨低峰期）。CI 流程中，符号表作为构建产物的一部分发布，确保模型服务读取的符号表和代码版本一致。
没有 LSP Server 的语言（如某些 DSL）怎么办？
回退到 Tree-sitter 纯语法解析。优先级：LSP > Tree-sitter > 正则。没有 LSP 时可以用 Tree-sitter 提取 AST 结构作为伪符号表，或者对于简单 DSL，直接传原始文本。
不推荐
  • 忽略 LSP，只用字符串匹配找定义
  • 每次补全都全量重建符号表
  • 零缓存策略，LSP 查询无限制调用
  • 不处理跨文件符号引用
推荐
  • 优先使用 LSP 获取精确上下文
  • 符号表增量更新而非全量重建
  • LSP 结果缓存 + 异步刷新
  • 符号表支持跨文件/跨语言引用

本章结论

  • 预算分配是起点：context budget 显式管理每层 Token，防止窗口无序占用。
  • 文件选择三策略：L1 import 解析 + L2 向量检索 + L3 Git 变更，覆盖精确到模糊的需求。
  • 符号表结构化理解：symbol table 提供类/方法级别的精确定位。
  • LSP 实时上下文：lsp concept 提供光标处的精确符号、类型和诊断信息。
  • 上下文选得对，补全才准确：相关文件的选择比模型大小更容易被忽视，但影响更大。

判断练习

  1. 团队把全部 128K 上下文都用来塞相关代码，没有保留 Reserve。你可能遇到什么问题？

    参考答案

    模型输出被截断——因为输出 Token 也占用上下文窗口。实际可用输出减少后，补全建议可能不完整（中途截断）。Reserve 的 10-15% 必须留给输出和临时缓冲。

  2. LSP 返回的 definition 与 Git 中最新代码不一致，哪个可信？

    参考答案

    LSP 读的是磁盘文件，Git 是已提交版本。正在编辑的未保存内容，LSP 更准。对已提交的版本，Git 更权威。最佳实践：LSP 用于实时编辑上下文，符号表基于 Git main 分支全量构建。
