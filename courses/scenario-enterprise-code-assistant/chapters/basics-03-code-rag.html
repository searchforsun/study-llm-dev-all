为什么要学本章

通用 RAG 将代码当作普通文本切片，丢失了语法结构和引用关系。代码 RAG 必须基于 AST（抽象语法树）切片，保留函数完整性、类继承关系和跨文件引用。不理解 code rag arch，构建的代码搜索就像「给图书馆按字数切书」——结构全丢，检索效果差。

本章先记住 3 件事
  • AST 切片 > 文本切片：按函数/类/方法切块保留语义完整性，检索精度提升 30%+。
  • 索引需多模态：向量索引 + 符号索引 + 引用关系图协同工作。
  • 引用跳转是核心 UX：检索结果中的代码块必须可追溯到源文件位置。

从通用 RAG 到代码 RAG：架构差异

通用 RAG 对文本按固定长度（如 256/512 tokens）切片，这种方式对代码是破坏性的——它会把一个完整的函数定义切成两半，丢失语法结构。代码 RAG（s4 rag）的核心区别在于：以 AST 节点为切分单元，保留函数、类、方法定义的完整性。

代码 RAG 架构包含切片、索引、检索三个阶段，与通用 RAG 的核心差异在于以 AST 为切分单元。

先想 10 秒：如果你把整个 Java 文件按 200 token 切块，会发生什么问题？

一个 Java 方法通常 10-50 行（约 100-500 tokens）。按 200 token 切块会导致：① 方法定义被截断，补全时语法错误；② 无法找到「这个方法属于哪个类」；③ 检索到的方法名不完整。这些都会导致补全建议质量下降。

AST 切片策略：保留代码结构

ast chunk 是指以 AST 节点为边界进行代码切片。最小单元是函数/方法定义，最大单元是文件。切片策略需要在「保留完整语义」和「控制块大小」之间平衡。

CorpAssist · 多语言切片策略 CorpAssist 团队为 Java/Python/Go 项目设计统一的切片策略：① 每个函数/方法作为一个 chunk，保留完整签名和 docstring；② 类定义作为一个整体 chunk（包含其所有方法）；③ 超过 200 行的大函数按逻辑块进一步切分（如循环体、条件分支）。每 chunk 控制在 50-150 行之间。

Python：AST 切片器实现

python 
# CorpAssist AST 切片器
from tree_sitter import Language, Parser

class CodeChunker:
    """基于 AST 的代码切片器"""

    NODE_TYPES = {
        "java": ["class_declaration", "method_declaration",
                 "interface_declaration", "enum_declaration"],
        "python": ["class_definition", "function_definition"],
        "go": ["method_declaration", "function_declaration",
               "type_declaration"],
    }

    def __init__(self):
        self.parser = Parser()

    def chunk_file(self, filepath: str) -> list:
        lang = filepath.split('.')[-1]
        node_types = self.NODE_TYPES.get(lang, [])
        with open(filepath, 'rb') as f:
            source = f.read()
            tree = self.parser.parse(source)

        chunks = []
        self._extract_chunks(tree.root_node, source,
                             filepath, node_types, chunks)

        # 如果没有提取到命名节点，整个文件作为一个 chunk
        if not chunks:
            chunks.append({
                "text": source.decode("utf-8"),
                "metadata": {
                    "path": filepath,
                    "start_line": 1,
                    "end_line": len(source.decode().splitlines()),
                    "type": "file",
                    "name": filepath.split("/")[-1]
                }
            })
        return chunks

    def _extract_chunks(self, node, source, path,
                        node_types, chunks):
        if node.type in node_types:
            name_node = node.child_by_field_name("name")
            start = node.start_point[0] + 1
            end = node.end_point[0] + 1
            text = source[node.start_byte:node.end_byte].decode()
            chunks.append({
                "text": text,
                "metadata": {
                    "path": path,
                    "start_line": start,
                    "end_line": end,
                    "type": node.type,
                    "name": (name_node.text.decode()
                             if name_node else "unknown"),
                    "language": path.split('.')[-1]
                }
            })
        for child in node.children:
            self._extract_chunks(child, source, path,
                                 node_types, chunks)

Java：Chunk 合并策略

java 
public class ChunkMerger {
    private static final int MAX_CHUNK_LINES = 150;
    private static final int MIN_CHUNK_LINES = 10;

    public List mergeChunks(List chunks) {
        List merged = new ArrayList<>();
        CodeChunk current = null;

        for (CodeChunk chunk : chunks) {
            if (current == null) {
                current = chunk;
                continue;
            }
            // 如果当前 chunk 太小且下一个 chunk 在同一文件，合并
            if (chunk.metadata().file().equals(
                    current.metadata().file())
                && current.lineCount() < MIN_CHUNK_LINES) {
                current = current.merge(chunk);
            } else {
                merged.add(current);
                current = chunk;
            }
        }
        if (current != null) {
            merged.add(current);
        }
        return merged;
    }

    public record CodeChunk(
        String text,
        ChunkMetadata metadata
    ) {
        public int lineCount() {
            return text.split("\n").length;
        }
        public CodeChunk merge(CodeChunk other) {
            return new CodeChunk(
                this.text + "\n" + other.text,
                new ChunkMetadata(
                    metadata.file(),
                    metadata.startLine(),
                    other.metadata.endLine(),
                    metadata.type(),
                    metadata.name() + "+" + other.metadata.name(),
                    metadata.language()
                )
            );
        }
    }
    public record ChunkMetadata(
        String file, int startLine, int endLine,
        String type, String name, String language
    ) {}
}
不推荐
  • 按固定 token 数切片，函数被截断
  • 所有语言使用同一切片策略
  • 切片后丢弃 AST 元数据
  • 忽略类-方法层级关系
推荐
  • 以函数/方法为最小切片单元
  • 按语言特点定制切片节点类型
  • 保留 path/line/lang 等完整元数据
  • 维护类-方法的父子关系链

仓库索引与多语言支持

repo index 将切片后的代码块构建为可检索的索引。它包含三层：向量索引（语义搜索）、符号索引（精确查找）、引用图（跨文件关系）。repo embedding 使用专门的代码 Embedding 模型（如 code-embed-v1 或 starencoder），比通用 Embedding 更理解代码语义。

multi lang 支持是企业代码助手的刚需——一个微服务项目可能同时包含 Java、Go、Python 和 JavaScript。统一索引的关键是：

  • 每种语言使用对应的 Tree-sitter 语法解析器
  • 切片元数据统一为 path/start_line/end_line/type/name/language 六个字段
  • Embedding 模型需要跨语言对齐（同一语义在不同语言中向量相近）
  • 引用图支持跨语言符号解析（Java 类调用 Go 服务的方法）

多语言索引配置：YAML

yaml 
# CorpAssist 多语言索引配置
index_config:
  version: "2.0"

  languages:
    java:
      parser: tree-sitter-java
      chunk_nodes: [class_declaration, method_declaration]
      embed_model: code-embed-v1
    python:
      parser: tree-sitter-python
      chunk_nodes: [class_definition, function_definition]
      embed_model: code-embed-v1
    go:
      parser: tree-sitter-go
      chunk_nodes: [method_declaration, function_declaration]
      embed_model: code-embed-v1

  vector_store:
    type: milvus
    collection: code_chunks
    dimension: 768
    metric: IP

  symbol_index:
    type: postgres
    schema: [id, name, kind, file_path, line, language]

  reference_graph:
    enabled: true
    cross_language: true
常见误区：多语言 Embedding 不对齐

如果 Java 用 code-embed-v1，Python 用另一个模型，检索效果会变差。同一个语义在不同语言中应该映射到相近向量空间。推荐用统一的 Code Embedding 模型（如 CodeBERT 或 StarCoder Embedding）覆盖所有目标语言。

引用跳转：从检索到精准定位

xref jump（交叉引用跳转）是代码 RAG 区别于通用 RAG 的关键 UX 能力。检索结果中的每个代码块都附带源文件路径和行号范围，用户可以直接跳转到 IDE 中对应的位置。

常见问题（FAQ）

函数级别的 chunk 大小该怎么控制？太大会不会影响检索效果？
函数 chunk 推荐 50-150 行。超过 200 行的函数可以按逻辑块切分：if/for 分支提取为独立 chunk，但保留 parent 字段指向父函数。检索时若找到子块，同时返回父函数上下文以提高相关性。
代码 RAG 的准确率比通用 RAG 高多少？有数据吗？
CorpAssist 内部测试表明：在仓库问答场景中，AST 切片的 Top-5 相关代码命中率约 78%，相同数据下通用文本切片仅约 52%。提升主要在两个方面：函数边界完整（减少半个函数的噪声）、元数据丰富（可按符号名精确过滤）。
大仓库（1000+ 文件）的全量索引需要多久？如何加速？
取决于文件数量和语言。1000 个 Java 文件约 5-10 分钟完成解析+向量化。加速方案：① 多进程并行解析；② 只索引 main/production 分支，跳过 test 目录；③ 使用 GPU 批量 Embedding；④ Git push 触发增量更新，避免全量重建。
不推荐
  • 检索结果只返回代码文本，无位置信息
  • 所有语言共享一个 Tree-sitter 解析器
  • 索引全量重建，不支持增量更新
  • 不维护跨文件引用关系
推荐
  • 每个代码块带 path/line 元数据
  • 每种语言使用对应 Tree-sitter 语法
  • Git push 触发增量索引
  • 构建符号引用图支持跨文件跳转

本章结论

  • 代码 RAG ≠ 通用 RAG：以 AST 为切分单元是代码 RAG（code rag arch）的核心特征。
  • AST 切片保留语义：function chunk 按函数/类/方法为边界切分，提升检索精度 30%+。
  • 多模态索引：向量索引 + 符号索引 + 引用图协同，覆盖语义搜索和精确查找。
  • 引用跳转提升 UX：每个代码块附带精确文件位置，支持 IDE 一键跳转。
  • 多语言统一架构：统一元数据格式 + 跨语言 Embedding + 每种语言专用解析器。

判断练习

  1. 你的代码 RAG 系统在检索「如何实现用户认证」时返回了 5 个不相关的 Java 工具类。最可能的原因是什么？

    参考答案

    最可能的原因：Embedding 模型是通用文本模型而非代码专用模型。通用模型不理解「认证」在代码中的具体表现形式。改用 code-embedding 模型，或检查索引中是否只包含了工具类文件（这类文件在仓库中占比过大也会导致检索偏差）。

  2. 你发现某个大函数（300 行）被作为一个 chunk 索引，检索时命中率很低。怎么办？

    参考答案

    对大函数做二次切分：按内部逻辑块（if/for/try 等）分割为子 chunk，每个子 chunk 保留 parent 字段指向父函数。检索时优先匹配子 chunk，同时携带父函数上下文。另外考虑代码重构——300 行函数本身就应该是多个小函数的组合。
