坑一：AST 检索导致的 Token 爆炸

token explosion 是最常见的生产事故。典型场景是：开发者在问答中提问「这个项目使用的所有技术栈」，索引服务返回了 50+ 个代码片段，加起来超过 20K tokens，直接撑爆上下文窗口。

CorpAssist · Token 爆炸现场 开发者提问「如何实现用户认证」，AST 检索返回了 AuthController.java（150 行）、AuthService.java（300 行）、JwtUtil.java（80 行）、UserRepository.java（200 行）、SecurityConfig.java（120 行）、TokenFilter.java（180 行）等 20 个代码块，总计约 15K tokens。LLM 的注意力被稀释，回答中引用了错误的方法名。

ast not whole repo 是核心原则：绝不要把整个仓库的代码塞入 prompt。解决方案：

问题根因解决方案
检索返回太多 chunk Top-K 设置过大（如 K=20） Top-K 限制为 3-5，按相关性裁剪
单个 chunk 太大 函数/类定义过长（> 200 行） 拆分子 chunk，只返回相关部分
无关文件被检索 Embedding 模型在代码上区分度不够 使用代码专用 Embedding + 关键词过滤
检索与对话历史叠加 多轮问答累积 chunks 每轮独立检索，不累积历史 chunks

Python：Token 爆炸检测器

python 
# CorpAssist Token 爆炸检测
class TokenExplosionDetector:
    def __init__(self, max_context: int = 32000):
        self.max_context = max_context

    def analyze_request(self, prompt_parts: dict) -> dict:
        """分析 prompt 各部分的 token 占比"""
        total = 0
        details = {}
        for layer, content in prompt_parts.items():
            tokens = self._count_tokens(content)
            details[layer] = {
                "tokens": tokens,
                "pct": 0  # 待计算
            }
            total += tokens

        for layer in details:
            details[layer]["pct"] = round(
                details[layer]["tokens"] / total * 100, 1
            )

        # 检测爆炸风险
        risks = []
        for layer, info in details.items():
            if info["pct"] > 40:
                risks.append(f"{layer} 占比 {info['pct']}% > 40%，建议裁剪")

        if total > self.max_context * 0.9:
            risks.append(f"总 tokens {total} 超过上限 90%，即将爆窗")

        return {
            "total_tokens": total,
            "max_tokens": self.max_context,
            "usage_pct": round(total / self.max_context * 100, 1),
            "details": details,
            "risks": risks,
            "is_overflow": total > self.max_context
        }

    def _count_tokens(self, text: str) -> int:
        return len(text) // 4  # 估算：约 4 字符/token
不推荐
  • Top-K=20，一次性注入过多代码块
  • 所有检索结果不裁剪直接拼接
  • 历史对话累积 chunks（一轮加 5 个，5 轮后 25 个）
  • 问答和补全场景使用相同的检索策略
推荐
  • Top-K=3-5，按相关性裁剪
  • 每轮检索独立，不累积
  • 问答场景 Top-K=5，补全场景 Top-K=2
  • 运行时检测 token 占比，触发裁剪

坑二：FIM 与私库微调的边界混淆

fim ft boundary 是指 FIM 补全和私有微调的边界选择。常见错误是团队直接使用私有代码微调通用模型，期望解决补全问题。但微调成本高、周期长、可能导致通用能力退化。正确路径是：先用 FIM + RAG 解决 90% 的问题，微调只用于特定场景的最后一公里。

方案适用场景成本效果
FIM + RAG 日常代码补全，90% 场景 低（无训练成本） 采纳率 30-40%
PEFT LoRA 微调 特定框架/DSL 的高频模式 中（数小时训练） 采纳率+10-15%
全量微调 代码风格非常独特的企业 高（数天训练） 采纳率+15-20%
常见误区：微调可以替代 FIM 格式学习

微调不会自动教会模型 FIM 格式。如果预训练模型不支持 FIM（如某些 Base 模型），微调后仍然不行。FIM 能力是预训练阶段决定的。选择模型时，优先选原生支持 FIM 的模型（CodeLLaMA、StarCoder、Qwen-Coder），而不是微调一个不支持 FIM 的模型。

检查理解：什么时候应该考虑微调而不是 FIM+RAG？

当出现以下情况时考虑微调：① 企业内部有大量的重复性代码模式（如统一的状态机框架、特定的 ORM 封装）；② FIM+RAG 的采纳率稳定在 30% 以下且其他优化手段无效；③ 有充足的高质量训练数据（至少 5000+ 条企业代码）。否则，优先优化 RAG 检索质量和上下文组装策略。

坑三：缺乏 LSP 验证闭环

lsp verify loop 是生成代码后的质量验证闭环。CorpAssist 的实践：每次补全生成后，将完整代码（prefix + middle + suffix）提交给 LSP 验证语法和类型错误。如果有编译错误，回退到上一版本或标记为「需要验证」。

LSP 验证闭环流程
sequenceDiagram
    participant Plugin as IDE 插件
    participant LSP as LSP Server
    participant AI as CorpAssist

    Plugin->>AI: 请求补全
    AI-->>Plugin: 返回 Middle 代码
    Plugin->>Plugin: 组装完整代码
    Plugin->>LSP: textDocument/diagnostic
    LSP-->>Plugin: 诊断结果
    alt 无错误
        Plugin->>Plugin: 展示补全建议
    else 有编译错误
        Plugin->>Plugin: 隐藏建议/标记警告
        Plugin->>AI: 错误反馈（用于优化）
    end

生成代码必须经过 LSP 验证才能展示给开发者。

verify compile 是验证的最终手段——对于 Java 项目，可以快速编译补全后的临时文件。如果编译失败，标记该建议为「未验证」。验证闭环的好处：补全建议的「编译通过率」从 60% 提升到 90%+。

不推荐
  • 补全后不做任何语法验证
  • 编译错误的建议仍展示给开发者
  • 不收集验证反馈用于优化
  • 忽略 LSP 对生成代码的诊断结果
推荐
  • 补全后自动 LSP 诊断验证
  • 编译错误的建议隐藏或标记
  • 错误类型和频率反馈给模型优化
  • LSP 验证在 50ms 内完成

坑四：工具调用与三步规划偏航

plan drift code 是指 AI 在多次工具调用后偏离原始目标。典型场景：AI 先检索代码 → 发现需要修改多个文件 → 读取每个文件 → 修改过程中忘记了最初的目的。tool loop code 是 AI 陷入无限工具调用的陷阱——不断检索查看更多代码但从不输出最终结果。

防止规划偏航的策略

常见问题（FAQ）

LSP 验证发现语法错误后，该直接丢弃建议还是提供给开发者？
建议标记为「可能有语法错误」并展示给开发者。原因：① LSP 可能误报（如未 import 类，但开发者知道需要 import）；② 即使有语法错误，代码逻辑可能是正确的，开发者能更快修正。但对于严重错误（如括号不匹配、非法语法），应该直接隐藏并提示开发者。
微调后模型的通用补全能力下降了，怎么办？
这是典型的「灾难性遗忘」。解决方案：① 使用 PEFT/LoRA 而非全量微调，冻结 Base 模型的大部分权重；② 微调数据中混入 20-30% 的通用代码数据；③ 微调后做对比测试——在通用 benchmark（如 HumanEval）上的得分下降不应超过 5%；④ 一个更激进的做法：保留 Base 模型，用 LoRA adapter 做路由，特定场景调用 adapter，通用场景调用 base。
Agent 调用工具时陷入死循环，怎么检测和恢复？
死循环的特征是工具调用模式重复（如一直检索同一个文件但从不输出）。检测：① 工具调用签名去重——如果最近 5 次调用中有 3 次签名相同 → 判为循环；② 输出 token 计数——如果连续 5 次工具调用后输出为 0 → 判为循环。恢复：① 强制输出当前已收集的信息；② 中断本次调用，向用户返回中间结果和错误提示。
不推荐
  • 无限制的工具调用次数
  • 不验证生成代码的语法正确性
  • 直接用私库全量微调通用模型
  • 陷入工具循环时不做中断
推荐
  • 最大工具调用 10 次 + 超时 60s
  • LSP 验证闭环保障语法正确性
  • FIM+RAG 优先，微调最后
  • 检测工具循环，强制中断并返回

本章结论

  • Token 爆炸可预防：限制 Top-K、每轮独立检索、运行时检测占比。
  • FIM 优先于微调：fim ft boundary 原则——用 FIM+RAG 解决 90% 问题。
  • 验证闭环不可少：lsp verify loop 将编译通过率提升到 90%+。
  • 防止工具偏航：限制调用次数，设置超时，检测循环模式。
  • 一切坑都是设计问题：在架构设计阶段就考虑这些边界条件，比事后修坑节省 10 倍时间。

判断练习

  1. 你的代码助手在问答场景中频繁返回「在代码库中未找到」——即使代码确实存在。排查步骤是什么？

    参考答案

    ① 检查索引是否是最新的（增量索引是否正常工作）；② 检查检索的 Top-K 命中率——在测试集上运行检索，看是否能正确召回相关代码；③ 检查 Embedding 模型是否匹配——如果用通用模型可能无法正确编码代码语义；④ 检查 ACL 权限——用户当前角色是否有权访问这些代码；⑤ 检查 prompt 组装——检索到的代码是否被正确注入到 prompt 中。

  2. FIM 补全建议 50% 以上无法通过编译验证。最可能的原因是什么？

    参考答案

    最可能的原因是上下文不足：模型没有获得必要 import 语句或依赖类型定义。检查：① prefix/suffix 是否包含了当前方法的所有必要上下文；② 相关文件是否被正确检索并注入；③ FIM 格式是否正确——格式错误会导致模型生成随机内容。其次检查模型选择——7B 以下的模型在复杂逻辑上的语法正确率较低。
