为什么要学本章

代码补全是代码助手使用频率最高的场景——开发者每天触发数十到数百次。补全质量直接决定开发者对 AI 助手的信任度。FIM（Fill-in-the-Middle）是代码补全的核心技术，正确的格式拼装比模型选择更重要——格式错误时大模型也会输出垃圾。本章深入 FIM 的三种格式、上下文拼装策略和企业风格约束的实现。

本章先记住 3 件事
  • FIM 格式决定生死：prefix/suffix/middle 的顺序和分隔符必须严格匹配模型训练时的格式，否则补全质量暴跌。
  • 上下文要有优先级：当前文件 > import 文件 > 语义相似文件 > Git 记录，层级裁剪。
  • 风格约束在 prompt 中：在 system prompt 中描述代码风格规范，而非依赖模型训练数据。

FIM 格式：Prefix/Suffix/Middle

fim format 是代码补全模型的输入格式。它将光标位置之前的代码作为 Prefix，之后的代码作为 Suffix，模型生成中间部分（Middle）。prefix suffix 的正确拼装是补全质量的基础。

FIM 将光标前后的代码分开输入，模型只生成中间缺失部分。

三种 FIM 格式

格式模板适用模型
CodeLLaMA 格式 <FILL_HERE>{suffix}<FILL_HERE>{prefix} CodeLLaMA 系列
StarCoder 格式 <|fim_prefix|>{prefix}<|fim_suffix|>{suffix}<|fim_middle|> StarCoder / StarCoder2
Qwen Coder 格式 <|fim_prefix|>{prefix}<|fim_suffix|>{suffix}<|fim_middle|> Qwen2.5-Coder 系列

Python：FIM Prompt 构建器

python 
# CorpAssist FIM Prompt 构建器
class FIMBuilder:
    """FIM 格式构建器，支持多种模型格式"""

    TEMPLATES = {
        "starcoder": {
            "prefix": "<|fim_prefix|>",
            "suffix": "<|fim_suffix|>",
            "middle": "<|fim_middle|>",
        },
        "codel lama": {
            "prefix": "",
            "middle": "",
            "suffix_first": True,
        },
        "qwen-coder": {
            "prefix": "<|fim_prefix|>",
            "suffix": "<|fim_suffix|>",
            "middle": "<|fim_middle|>",
        },
    }

    def __init__(self, model_format: str = "qwen-coder"):
        self.template = self.TEMPLATES[model_format]
        self.model_format = model_format

    def build_prompt(self, prefix: str, suffix: str) -> str:
        fmt = self.model_format
        if fmt in ("starcoder", "qwen-coder"):
            return (f"{self.template['prefix']}{prefix}"
                    f"{self.template['suffix']}{suffix}"
                    f"{self.template['middle']}")
        elif fmt == "codel lama":
            return (f"{self.template['middle']}{suffix}"
                    f"{self.template['middle']}{prefix}")
        else:
            raise ValueError(f"Unsupported format: {fmt}")

    def extract_completion(self, full_output: str) -> str:
        """从模型输出中提取中间补全部分"""
        middle_token = self.template.get("middle", "")
        if middle_token and middle_token in full_output:
            return full_output.split(middle_token)[-1].strip()
        return full_output.strip()

    @staticmethod
    def build_prefix_suffix(source_code: str,
                            cursor_line: int,
                            cursor_col: int) -> tuple:
        """从源代码和光标位置提取 prefix 和 suffix"""
        lines = source_code.split('\n')
        prefix_lines = lines[:cursor_line - 1]
        current_line = lines[cursor_line - 1]
        prefix_lines.append(current_line[:cursor_col])
        prefix = '\n'.join(prefix_lines)

        suffix_lines = [current_line[cursor_col:]]
        suffix_lines.extend(lines[cursor_line:])
        suffix = '\n'.join(suffix_lines)

        return prefix, suffix
不推荐
  • FIM 格式与模型不匹配（如用 StarCoder 格式调 CodeLLaMA）
  • 光标前后的上下文不分，直接用「前缀」格式
  • suffix 太长（超过 2048 tokens），影响补全质量
  • 不做格式校验，模型输出包含多余的分隔符
推荐
  • 严格匹配模型的 FIM 训练格式
  • prefix + suffix 总长度控制在 4K tokens 以内
  • 限制 suffix 在 512 tokens 以内（suffix 太长会混淆模型）
  • 后处理裁剪模型输出的分隔符和特殊 token
先想 10 秒：为什么 suffix 不能太长？

大部分 FIM 模型的训练数据中，suffix 占比较小（约 10-20%）。如果 suffix 太长，超出了模型训练时的分布范围，模型可能无法正确理解上下文关系。实践中限制 suffix ≤ 512 tokens，如果光标后代码太长，只取后续的简要结构（如函数签名+大括号）。

上下文拼装：优先级与裁剪

context assemble 是将 FIM 部分与相关代码上下文合并为最终 prompt 的过程。拼装的顺序和优先级直接影响补全质量。

Java：上下文拼装器

java 
// CorpAssist 上下文拼装器
public class ContextAssembler {
    private static final int MAX_CONTEXT_TOKENS = 4096;
    private final TokenCounter tokenCounter;

    public ContextAssembler(TokenCounter tokenCounter) {
        this.tokenCounter = tokenCounter;
    }

    public CompletionRequest assemble(ContextBundle bundle) {
        var parts = new ArrayList();

        // L1: System prompt（风格约束）
        parts.add(new ContextPart("system", bundle.systemPrompt(), 100));

        // L2: 当前文件上下文（光标前 50 行 + 后 10 行）
        parts.add(new ContextPart("current_file",
            bundle.currentFileContext(), 500));

        // L3: import 的依赖文件签名
        if (bundle.hasImportContext()) {
            parts.add(new ContextPart("imports",
                bundle.importContext(), 300));
        }

        // L4: 相关文件片段（语义检索）
        if (bundle.hasRelatedChunks()) {
            parts.add(new ContextPart("related",
                bundle.relatedChunks(), 1000));
        }

        // 裁剪到预算内
        var assembled = new StringBuilder();
        int totalTokens = 0;
        for (var part : parts) {
            int tokens = tokenCounter.count(part.content());
            if (totalTokens + tokens > MAX_CONTEXT_TOKENS) {
                String truncated = tokenCounter.truncate(
                    part.content(), MAX_CONTEXT_TOKENS - totalTokens);
                assembled.append(truncated).append("\n");
                break;
            }
            assembled.append(part.content()).append("\n");
            totalTokens += tokens;
        }

        // 追加 FIM 部分
        String fimPrompt = FIMBuilder.build(
            bundle.prefix(), bundle.suffix());

        return new CompletionRequest(
            assembled.toString(),
            fimPrompt,
            bundle.cursorPosition()
        );
    }

    public record ContextPart(
        String layer, String content, int maxTokens
    ) {}
    public record CompletionRequest(
        String context, String fimPrompt, CursorPosition cursor
    ) {}
}
常见误区：上下文越多越好

上下文不是越多越好。过多的上下文会稀释注意力（Lost in the Middle 问题），并且增加延迟。补全场景建议上下文总量控制在 4K tokens 以内。优先保障当前文件的上下文（光标附近），再按依赖链添加相关文件。

风格约束：企业代码规范

style constraint 和 corp style 确保 AI 生成的代码符合企业的编码规范。风格约束通过两个渠道实现：System Prompt 中描述风格要求 + 示例代码隐式约束。

企业风格 Prompt 示例

yaml 
# CorpAssist 补全风格约束配置
completion_style:
  # Java 风格约束
  java:
    naming:
      - "类名: UpperCamelCase，如 UserService"
      - "方法名: lowerCamelCase，如 getUserById"
      - "常量: UPPER_SNAKE_CASE，如 MAX_RETRY_COUNT"
    import_order:
      - "1: java.*"
      - "2: javax.*"
      - "3: org.springframework.*"
      - "4: com.yourcompany.*"
      - "5: 其他第三方"
    pattern:
      - "使用 @Autowired 注入（而非构造器注入）"
      - "Controller 层统一 @RestController"
      - "Service 层必须有接口 + 实现类"
      - "所有 public 方法必须有 JavaDoc"
      - "禁止使用 System.out.println，使用 Logger"

  # Python 风格约束
  python:
    naming:
      - "类名: PascalCase"
      - "函数/变量: snake_case"
    pattern:
      - "优先使用类型注解"
      - "使用 pathlib 替代 os.path"
      - "禁止使用 import *"
      - "所有函数必须包含 docstring"
      - "使用 f-string 而非 % 或 .format()"

常见问题（FAQ）

System Prompt 中的风格约束和模型训练风格冲突时怎么办？
System Prompt 的约束优先级高于训练数据。如果模型生成的代码不符合企业风格，说明约束不够强或上下文不够充分。建议：① 在 System Prompt 中明确说明「这是 XXX 公司的代码规范，必须严格遵守」；② 在上下文中包含同一项目已有的代码示例作为隐式约束；③ 后处理用 linter 验证并修正风格问题。
不同项目有不同的风格规范，怎么管理？
CorpAssist 支持基于仓库的配置：每个仓库在索引时记录其 .editorconfig 或 checkstyle 配置路径，补全时自动加载对应的风格约束。后端存储为 key-value（repo_id → style_config）。如果项目没有显式配置，可以通过分析仓库已有代码自动推断风格。
补全建议太长时怎么控制？
通过 max_tokens 参数控制。单行补全（如变量名、方法调用）设置 max_tokens=50；语句补全设置 max_tokens=100；代码块补全设置 max_tokens=300。同时可以在 prompt 中说明「生成最简补全即可」。另外要检测「是否生成了不必要的内容」，后处理时裁剪掉空行和注释。
不推荐
  • 风格约束全部放在 system prompt 中，冗长且不精确
  • 所有项目使用同一套风格规范
  • 风格约束只说不展示示例
  • 补全后不做 style lint 验证
推荐
  • System + 示例代码双重约束
  • 按项目/仓库配置独立的风格规范
  • 在 prompt 中包含同项目代码示例
  • 补全后用 linter 验证并修复

本章结论

  • FIM 格式精确匹配：每种模型有特定的 FIM 模板，格式错误会导致补全质量暴跌。
  • 上下文分层装：context assemble 按 System → Current → Import → Related 的顺序拼装，总量 ≤ 4K tokens。
  • 风格约束双重保证：System Prompt 描述 + 示例代码隐式约束，后处理 linter 验证。
  • 控制输出长度：根据场景设 max_tokens，单行补全 vs 代码块补全区分处理。
  • 内联补全体验：inline complete 需要 < 200ms 首 token，建议用小模型 + vLLM 加速。

判断练习

  1. 补全建议总是生成不完整的函数（只生成到一半就停了）。最可能的原因是什么？

    参考答案

    最常见的原因是 max_tokens 太小或 FIM 格式错误。检查：① max_tokens 是否足够容纳完整的补全；② FIM 的 middle 分隔符是否正确，模型可能没有识别到需要生成完整函数；③ suffix 是否包含了函数的结束标记（如 }），导致模型认为已经结束。

  2. 开发者反馈补全建议的代码风格和项目不一致，但 System Prompt 已经写了风格规范。怎么办？

    参考答案

    ① 确认 System Prompt 确实在上下文中且未被截断（检查 budget 分配）；② 在 prompt 中添加项目中的具体代码示例（隐式约束通常比显式规则更有效）；③ 后处理阶段运行 linter（如 checkstyle/eslint）自动修正风格问题；④ 微调模型时加入项目代码作为训练数据。
