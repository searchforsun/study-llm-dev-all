补全、问答与 Review：代码助手的三大场景

企业代码助手通常覆盖三个核心场景，每个场景对延迟、准确率、上下文深度的要求截然不同。

场景典型交互延迟要求核心指标落地难度
代码补全 IDE 内联建议，Tab 采纳 < 500ms adoption rate ≥ 30% 中
仓库问答 对话框提问，引用代码块 < 3s 准确率 ≥ 80% 高
Code Review PR 提交后自动审查 < 30s 漏洞检出率 ≥ 70% 高

completion qa review 这三个场景并非独立运行。CorpAssist 团队的实际做法是：先用补全场景建立开发者信任（采纳率 > 30%），再推进仓库问答解决「代码理解」痛点，最后用 Review 场景保障代码质量门禁。

不推荐
  • 三个场景同时上线，团队运维压力过大
  • 补全采纳率未达标就推进 Review
  • 用同一套模型服务所有场景，延迟不能满足差异需求
推荐
  • 按补全 → 问答 → Review 顺序迭代
  • 每个场景设独立 SLO 和评估数据集
  • 补全用小模型（< 7B），问答和 Review 用大模型（≥ 70B）

采纳率分析：Python 采样脚本

python 
# CorpAssist 采纳率分析器
import json
from datetime import datetime, timedelta

def analyze_adoption(logs_path: str, days: int = 30) -> dict:
    """分析代码补充采纳率"""
    cutoff = datetime.now() - timedelta(days=days)
    total_suggestions = 0
    accepted = 0
    lang_stats = {}

    with open(logs_path, 'r') as f:
        for line in f:
            entry = json.loads(line)
            ts = datetime.fromisoformat(entry['timestamp'])
            if ts < cutoff:
                continue
            total_suggestions += 1
            if entry.get('accepted'):
                accepted += 1
            lang = entry.get('language', 'unknown')
            lang_stats.setdefault(lang, {'total': 0, 'accepted': 0})
            lang_stats[lang]['total'] += 1
            if entry.get('accepted'):
                lang_stats[lang]['accepted'] += 1

    overall = accepted / total_suggestions if total_suggestions else 0
    per_lang = {
        lang: stats['accepted'] / stats['total']
        for lang, stats in lang_stats.items()
        if stats['total'] > 0
    }
    return {
        "total_suggestions": total_suggestions,
        "overall_adoption": round(overall, 4),
        "per_language": {k: round(v, 4) for k, v in per_lang.items()}
    }

# 使用示例
stats = analyze_adoption("completion_logs.jsonl")
print(f"总体采纳率: {stats['overall_adoption']:.1%}")
for lang, rate in stats['per_language'].items():
    print(f"  {lang}: {rate:.1%}")

Java：ROI 计算器

java 
// CorpAssist ROI 计算器
public class CodeAssistantROI {
    public record ROIMetrics(
        int developerCount,
        double dailyTimeSavedMinutes,
        double monthlyLicenseCost,
        double avgDeveloperHourlyRate
    ) {
        public double monthlyTimeSavedHours() {
            return developerCount * dailyTimeSavedMinutes * 22 / 60.0;
        }

        public double monthlyMonetaryValue() {
            return monthlyTimeSavedHours() * avgDeveloperHourlyRate;
        }

        public double monthlyROI() {
            double benefit = monthlyMonetaryValue();
            return (benefit - monthlyLicenseCost) / monthlyLicenseCost;
        }

        public String report() {
            return String.format("""
                CorpAssist ROI Report
                ========================
                开发者数量: %d
                人均日节省: %.1f 分钟
                月节省时: %.1f 小时
                月节省金额: ¥%.2f
                月许可证成本: ¥%.2f
                月度 ROI: %.2f
                """, developerCount, dailyTimeSavedMinutes,
                monthlyTimeSavedHours(), monthlyMonetaryValue(),
                monthlyLicenseCost, monthlyROI());
        }
    }

    public static void main(String[] args) {
        var metrics = new ROIMetrics(500, 18, 80000, 150.0);
        System.out.println(metrics.report());
    }
}
检查理解：采纳率 30% 意味着什么？

参照 code assistant roi 行业基准，30% 是补全场景的「及格线」。低于 30% 说明开发者不太信任建议，可能问题在模型质量或上下文不够；超过 40% 是优秀水平，可以开始 ROI 正向循环。

场景选择矩阵与落地路径

不同企业面临的问题不同。CorpAssist 使用 s4 scenario 选择矩阵，从两个维度评估场景优先级：开发者痛点强度 和 落地技术难度。

场景痛点强度技术难度推荐优先级启动条件
代码补全 高（每天数百次触发） 中 P0 首选 部署 vLLM + FIM 模型即可
仓库问答 高（代码理解频次高） 高 P1 次选 需要先建代码索引
Code Review 中（PR 提交时触发） 高 P2 后续 需整合 Git 流程和规则引擎
不推荐
  • 未经调研就选 Review 为第一场景
  • 指标笼统不清，只说「提升效率」
  • 所有场景共享一个评估数据集
推荐
  • 先调研开发者痛点，用数据说话
  • 每个场景设独立 SLO 指标
  • 补全用 HumanEval 类基准，问答用仓库问答对，Review 用漏洞注入测试

常见问题（FAQ）

补全和问答哪个场景先落地更好？
一般推荐先做补全。补全门槛低（部署 FIM 模型即可）、开发者感知强（每天触发几十次）。采纳率做到 30% 以上后，团队积累了模型优化和延迟调优经验，再推进问答场景会更顺畅。但如果团队主要痛点是「代码理解」，也可以先上问答。
Adoption Rate 和 Time Saved 怎么精确衡量？
采纳率从插件端采集：每次补全建议打日志，记录是否被 Tab 采纳。时间节省用 A/B 对比：对照组无助手，实验组有助手，统计同一类任务（如写单元测试）的平均完成时间。注意区分「感知节省」和「实际节省」，建议两者都追踪。
小团队（< 50 人）值得落地代码助手吗？
值得，但路径不同。小团队可以直接用商业方案（如 GitHub Copilot）快速体验，积累使用数据后再评估自建 RAG 问答的必要性。关键是先让开发者用起来，而不是先建基础设施。

本章结论

  • 三大场景各有侧重：补全追求低延迟高采纳，问答追求准确率与引用，Review 追求漏洞检出与覆盖率。
  • 指标要量化：adoption rate、time saved、code assistant roi 是向管理层汇报的关键数据。
  • 模块化架构：Index / Completion / Q&A / Review 四模块分工明确，可独立迭代。
  • 场景有顺序：从补全起步，积累模型优化经验后依次推进问答和 Review。
  • 持续度量：采纳率 < 30% 时先优化质量，> 40% 时关注延迟和覆盖率。

判断练习

  1. 团队选择「先做 Code Review」为第一场景，你认为合理吗？说明理由。

    参考答案

    通常不合理。Review 场景技术难度高（需静态分析 + AI 结合）、触发频次低（仅 PR 时）、开发者感知弱。建议先做补全建立信任，再做 Review。但若团队主要痛点是代码质量门禁且已有 SAST 基础，可以例外。

  2. 补全采纳率 35%，但开发者反馈「建议质量不高」。你怎么看这个矛盾？

    参考答案

    采纳率 35% 在及格线以上，说明多数建议是可用的。开发者「感觉质量不高」可能因为：① 简单补全（如括号/变量名）拉高了采纳率，复杂逻辑补全质量差；② 延迟偏高 200ms+ 影响体验。建议分层统计采纳率，按代码行数 / 复杂度分桶分析。
