代码助手项目验收时，技术实力≠交付能力。你需要一套完整的交付物来证明：技术方案是好的、代码是可用的、安全是合规的。本章教你整理 portfolio s4 交付物体系：索引设计文档、API Demo、安全扫描报告、架构文档、交接文档。

本章先记住 3 件事

索引设计文档与 Demo 脚本

index design doc 是索引方案的完整描述。它需要让一个新工程师能独立搭建完整的索引管线。capstone s4 的 Demo 脚本需要覆盖三个核心场景：补全、问答、Review。

索引设计文档模板

markdown 
# CorpAssist 代码索引设计文档 v2.0

## 1. 架构总览
- 触发层：Git Webhook → RabbitMQ → 索引消费
- 解析层：Tree-sitter 多语言解析
- 向量化层：code-embed-v1 (768d) + GPU 批处理
- 存储层：Milvus + PostgreSQL + Redis

## 2. 切片策略
- 最小单元：函数/方法定义
- 最大单元：类定义（含子方法）
- 超大函数（>200行）：按逻辑块二次切分
- 元数据：{path, start_line, end_line, type, name, language, repo_id, commit_sha}

## 3. 索引 Schema (Milvus)
- 集合：code_chunks_{tenant_id}
- 向量字段：embedding (FLOAT_VECTOR, 768d)
- 标量字段：chunk_id, repo_id, file_path, language, start_line, commit_sha
- 索引类型：IVF_FLAT, nlist=1024

## 4. 增量更新
- 触发：post-receive hook
- 处理：只索引 diff 中的代码文件
- 版本控制：每个 chunk 记录 commit_sha，支持回滚

## 5. 权限模型
- 租户隔离：独立 collection
- 仓库可见性：repo_id 字段过滤
- 文件级别：visibility 字段 (public/internal/restricted)

安全报告与架构文档

security report 必须包含三个部分：SAST 静态扫描结果、依赖库漏洞扫描、红队测试报告。所有 Critical/High 漏洞必须有关闭状态（已修复 / 已接受风险 / 已规避）。architecture doc 描述系统的整体架构、关键设计决策和技术选型理由。

常见问题（FAQ）

索引设计文档需要多详细？
详细到「一个新后端工程师可以照着文档独立搭建索引管线」。包含：① 环境依赖（Python 版本、Milvus 版本、GPU 驱动）；② 配置文件（YAML 或 JSON 示例）；③ 启动命令和验证步骤；④ 常见问题和排障指南。索引 schema 的每个字段都要有注释说明用途。
Demo 需要准备几个环境？
至少两个环境：① 开发环境（用于日常开发和调试）；② 验收环境（与生产环境配置一致，用于正式演示）。验收环境需要预装：VS Code + CorpAssist 插件、服务端（索引+API+LLM）、示例项目代码。Demo 前检查网络状况、API 连通性、模型可用性。
安全报告中的「已接受风险」是什么意思？什么时候用？
「已接受风险」是指经过评估后认为该风险可接受，不需要修复。适用条件：① 风险发生概率极低；② 影响范围很小；③ 修复成本远大于风险损失。需要安全团队负责人在报告上签字确认。例如：某个低风险的告警只在特定配置下出现，且不影响核心功能。
不推荐
  • 只有代码没有文档，无法交接
  • Demo 环境与生产大不相同，演示时出问题
  • 安全报告只列出问题没有修复方案
  • 架构文档过时（代码改了文档没改）
推荐
  • 文档先写再开发（AD/IDD 先行）
  • Demo 环境与生产同配置
  • 每个安全漏洞有具体的修复方案
  • 文档纳入 CI 检查（修改代码需更新文档）

本章结论

  • 交付物体系化：portfolio s4 包含文档、Demo、代码三类，分别面向不同受众。
  • 索引文档要可复制：新工程师依据 index design doc 能独立搭建。
  • 安全报告要可追踪：每个漏洞有状态和修复方案，Critical 必须清零。
  • Demo 讲故事：展示一个完整的开发者工作流（编码→问答→审查）。
  • 文档即代码：文档与代码同步更新，纳入 CI 检查。

判断练习

  1. 技术委员会质疑「你们的架构设计文档和代码对不上」。你怎么解决？

    参考答案

    根本原因是文档没有随代码更新。解决方案：① 将架构文档纳入 CI 检查——修改核心组件时需要同步更新文档；② 使用「文档即代码」工具（如 PlantUML、Mermaid、OpenAPI），文档从代码注释生成；③ 每次架构变更评审时，同时 review 文档更新。

  2. Demo 演示时，补全建议太慢（> 1s），观众体验很差。你怎么办？

    参考答案

    事前预防：① Demo 前预热模型（发送一个 dummy 请求让模型加载到 GPU）；② 降低演示环境的 QPS（避免资源争抢）；③ 准备录屏作为 Plan B。事后应对：① 幽默化解（「这是我们的延迟优化前版本，你看到的就是我们要解决的问题」）；② 切换到录屏。现场优化很难，重点是预防。
