为什么要学本章

「这段鉴权逻辑怎么实现」——开发者每天问几十次这样的问题。仓库问答（Repo QA）是代码助手使用频率最高的场景之一，也是技术难度最高的。本章教你构建检索增强的代码问答 API，从向量检索到 LLM 生成，从代码引用到流式输出，完整覆盖端到端管线。

本章先记住 3 件事
  • 检索是 QA 的基础：没有精确的代码检索，LLM 再强也答不准。Top-3 命中率决定回答质量上限。
  • 引用必须精确：每个答案中的代码块都必须附带 file:line 引用，让开发者可以跳转验证。
  • 流式输出提升体验：SSE 流式返回，首 token 延迟 < 500ms，开发者可以边看边滚动。

检索+生成管线的设计与实现

retrieve generate 是仓库问答的核心模式。管线分为四步：① 查询 Embedding 化 → ② 向量检索 Top-K → ③ 上下文组装 + 引用展开 → ④ LLM 生成回答 + 引用标注。

仓库问答 API 流程
sequenceDiagram

    User->>API: "如何实现用户认证？"

    API->>Index: 向量检索 topK=5
    Index-->>API: 返回代码块+引用链
    API->>API: 上下文组装+引用展开
    API->>LLM: prompt + 代码上下文
    LLM-->>API: SS 流式回答
    API-->>User: 回答 + 代码引用块

    Note over API,User: 每个代码块附带 file:line 引用

完整管线从用户提问到流式返回，每步的时间预算需严格控制。

Python：仓库问答核心实现

python 
# CorpAssist 仓库问答服务
from typing import AsyncGenerator
import numpy as np

class RepoQAService:
    def __init__(self, vector_store, llm_client, embed_model):
        self.vector_store = vector_store
        self.llm = llm_client
        self.embedder = embed_model

    async def answer(self, query: str, repo_id: str,
                     user_context: dict = None) -> dict:
        # 1. 查询向量化
        query_vec = self.embedder.embed_query(query)

        # 2. 向量检索（带 ACL 过滤）
        results = self.vector_store.search(
            query_vec, top_k=5,
            filter_expr=f"repo_id == '{repo_id}'"
        )

        # 3. 构建上下文
        context_chunks = []
        citations = []
        for r in results:
            context_chunks.append(r["text"])
            citations.append({
                "file": r["file_path"],
                "start_line": r["start_line"],
                "end_line": r["end_line"],
                "snippet": r["text"][:200]
            })

        # 4. 组装 prompt
        prompt = self._build_prompt(query, context_chunks)

        # 5. 生成回答
        answer = await self.llm.complete(prompt)

        return {
            "answer": answer,
            "citations": citations,
            "metadata": {
                "chunks_retrieved": len(results),
                "model": self.llm.model_name
            }
        }

    async def stream_answer(self, query: str, repo_id: str
                            ) -> AsyncGenerator[str, None]:
        """流式版本"""
        query_vec = self.embedder.embed_query(query)
        results = self.vector_store.search(
            query_vec, top_k=5,
            filter_expr=f"repo_id == '{repo_id}'"
        )
        context_chunks = [r["text"] for r in results]
        prompt = self._build_prompt(query, context_chunks)
        async for token in self.llm.stream_complete(prompt):
            yield token

    def _build_prompt(self, query: str,
                      chunks: list) -> str:
        context = "\n\n---\n\n".join([
            f"文件 {i+1}:\n{c}"
            for i, c in enumerate(chunks)
        ])
        return f"""基于以下代码上下文回答问题。
如果代码中不包含相关信息，请说明"代码库中未找到相关信息"。

代码上下文：
{context}

问题：{query}

回答（引用对应代码）："""
不推荐
  • 检索结果直接拼接，不排序不裁剪
  • 引用格式不统一，无法跳转
  • 非流式返回，首屏延迟 5s+
  • 不处理检索结果为空的情况
推荐
  • 检索结果按相关性排序并裁剪到预算内
  • 统一引用格式（file:line）支持 IDE 跳转
  • SSE 流式返回，首 token < 500ms
  • 空结果时给出友好提示而非编造
检查理解：为什么检索质量比模型大小更重要？

在仓库问答场景中，如果检索不到相关代码，即使 GPT-4 也无法凭空回答「这个项目的认证流程怎么实现的」。检索的 Top-3 命中率决定了回答质量的上限。CorpAssist 内部测试显示：检索命中率从 50% 提升到 80%，回答准确率从 55% 提升到 85%。

代码引用块：精确到行号

code citation 是代码引用格式的标准。每个被引用的代码块必须包含：文件路径、起始行号、结束行号、代码片段。引用格式要支持 IDE 插件一键跳转。chunk cite 和 sse code 配合，在流式输出中也实时标注引用。

Java：引用构建器

java 
public class CitationBuilder {
    public record Citation(
        String file,
        int startLine,
        int endLine,
        String snippet,
        String symbol,
        String type
    ) {
        public String toMarkdown() {
            return String.format(
                "```%s:%d-%d\n%s\n```",
                file, startLine, endLine, snippet
            );
        }

        public String toIDEUri() {
            return String.format(
                "vscode://file/%s:%d:%d",
                file, startLine, endLine
            );
        }
    }

    public List buildCitations(List chunks) {
        return chunks.stream()
            .map(chunk -> new Citation(
                chunk.path(), chunk.startLine(), chunk.endLine(),
                truncate(chunk.text(), 200),
                chunk.symbolName(),
                chunk.type()
            ))
            .collect(Collectors.toList());
    }

    private String truncate(String text, int maxLen) {
        return text.length() <= maxLen
            ? text
            : text.substring(0, maxLen) + "...";
    }

    public String formatWithCitations(String answer,
                                       List citations) {
        StringBuilder sb = new StringBuilder(answer);
        sb.append("\n\n**引用代码：**\n");
        for (int i = 0; i < citations.size(); i++) {
            Citation c = citations.get(i);
            sb.append(String.format(
                "%d. [`%s:%d`](%s)\n",
                i + 1, c.file(), c.startLine(), c.toIDEUri()
            ));
        }
        return sb.toString();
    }
}
常见误区：引用太多反而不好

5 个以上引用块会让回答变得冗长。最佳实践是：LLM 在回答中引用 Top-3 最相关的代码块，其余作为「参考文件」列在回答末尾。每个引用块控制在 50 行以内，过长的代码块可以截断为关键部分 + 省略号。

流式响应与 SS E 推送

stream qa code 使用 Server-Sent Events (SSE) 实现流式输出。SSE 比 WebSocket 更轻量（单工、HTTP 原生支持），适合 LLM 这种「服务端单向推送」的场景。流式输出的关键在 Token 级别的引用标注——当 LLM 生成到引用标记时，实时嵌入引用信息。

Java：SSE 流式控制器

java 
// CorpAssist SSE 流式控制器
@RestController
public class QAController {
    private final RepoQAService qaService;

    @PostMapping("/v1/qa/stream")
    public SseEmitter streamAnswer(@RequestBody QARequest request) {
        SseEmitter emitter = new SseEmitter(30_000L); // 30s 超时

        executorService.execute(() -> {
            try {
                // 发送引用元数据（流式头部）
                emitter.send(SseEmitter.event()
                    .name("metadata")
                    .data(request.getCitations()));

                // 流式发送回答
                for (String token : qaService.streamAnswer(request)) {
                    emitter.send(SseEmitter.event()
                        .name("token")
                        .data(token));
                }
                emitter.send(SseEmitter.event()
                    .name("done")
                    .data(""));
                emitter.complete();
            } catch (Exception e) {
                emitter.completeWithError(e);
            }
        });
        return emitter;
    }

    @PostMapping("/v1/qa")
    public CompletableFuture answer(
            @RequestBody QARequest request) {
        return qaService.answerAsync(request);
    }
}

常见问题（FAQ）

检索结果为空时应该返回什么？
不要编造答案。返回「代码库中未找到相关信息，你可能需要在提问中使用更准确的函数名或类名」。同时建议用户关联的检索关键词。如果检索为空但 LLM 仍然回答，可能产生幻觉（hallucination），这是最危险的情况之一。
QA 的准确率怎么评估？需要人工标注吗？
建议使用人工标注的问答对评估集（至少 200 对）。评估维度：① 答案准确性（是否基于真实代码）；② 引用精确性（引用位置是否正确）；③ 完整性（是否覆盖了问题所有方面）。也可以用 LLM-as-Judge 做自动化评估，但需要交叉验证。
双栈架构（Dual Stack）是怎么工作的？
dual stack qa 指 Python 负责检索服务（向量搜索 + 排序），Java 负责 API 网关和业务编排。Python 栈利用其丰富的 ML/AI 库（sentence-transformers、numpy），Java 栈利用其成熟的框架（Spring Cloud、事务管理、权限控制）。两层通过 gRPC 或 HTTP 通信。
不推荐
  • 非流式，用户等待 5-10s 才看到回答
  • 引用格式混乱，不支持 IDE 跳转
  • 检索为空时 LLM 编造答案
  • Python 和 Java 栈职责不清，耦合严重
推荐
  • SSE 流式，首 token < 500ms
  • 统一引用格式，支持 IDE 一键跳转
  • 检索为空时明确告知用户
  • 双栈分离：Python 检索 + Java 编排

本章结论

  • 检索精度决定上限：retrieve generate 管线中，检索 Top-3 命中率是回答准确率的前提。
  • 引用必须精确：每个代码引用块附带 file:startLine:endLine，支持 IDE 跳转验证。
  • 流式输出降低等待感：stream qa code 首 token < 500ms，用户感知延迟大幅降低。
  • 双栈各司其职：dual stack qa 让 Python 做 ML 检索，Java 做业务编排。
  • 防幻觉是底线：检索为空时绝不编造，明确告知用户信息缺失。

判断练习

  1. 用户提问「如何部署这个服务」，检索命中的是 Dockerfile 和 deployment.yaml。但 LLM 回答了「使用 Spring Boot 的 main 方法启动」。问题出在哪里？

    参考答案

    LLM 没有使用检索到的代码上下文，而是依赖自己的训练数据回答了通用方案。这是「上下文忽略」问题。解决方案：① 强化 System Prompt 中的约束，明确要求「只基于给定代码回答」；② 在 prompt 中明确标注「相关代码包含 Dockerfile 部署配置」；③ 检索不到相关内容时返回「信息缺失」而非编造。

  2. SSE 流式连接经常超时中断，怎么处理？

    参考答案

    ① 增加 SSE 超时时间（从 15s 到 60s）；② 实现心跳机制（每 5s 发送 keepalive 事件）；③ 客户端实现自动重连（SSE 原生支持 EventSource 重连）；④ 服务端记录断点，支持重连后从断点续传。
