本章先记住 3 件事
  • 不使用公开数据集做唯一评估：HumanEval 不能代表企业代码场景，需要构建企业内部的 benchmark。
  • 采纳率是「商业价值」指标：30% + 采纳率意味着正向 ROI，低于 20% 要排查根本原因。
  • 延迟不是越低越好：在准确率和延迟之间找平衡——200ms 的补全如果全是错的，不如 500ms 的正确建议。

Python：评估执行器

python 
# CorpAssist 评估执行器
from difflib import SequenceMatcher

class CodeEvalExecutor:
    def __init__(self, eval_dataset: list):
        self.dataset = eval_dataset  # 每项: {prefix, suffix, expected, language}

    def evaluate_completion(self, model_client) -> dict:
        results = []
        for item in self.dataset:
            prompt = build_fim_prompt(
                item["prefix"], item["suffix"],
                item["language"]
            )
            generated = model_client.complete(prompt)

            # Edit-Similarity 评估
            similarity = SequenceMatcher(
                None, item["expected"], generated
            ).ratio()

            # 精确匹配
            exact_match = (generated.strip()
                           == item["expected"].strip())

            # Token 级别准确率
            expected_tokens = set(item["expected"].split())
            gen_tokens = set(generated.split())
            precision = len(expected_tokens & gen_tokens) / max(
                len(gen_tokens), 1)
            recall = len(expected_tokens & gen_tokens) / max(
                len(expected_tokens), 1)
            f1 = (2 * precision * recall / (precision + recall)
                  if precision + recall > 0 else 0)

            results.append({
                "id": item.get("id"),
                "edit_similarity": similarity,
                "exact_match": exact_match,
                "f1_score": f1,
                "generated": generated,
                "expected": item["expected"]
            })

        # 汇总
        total = len(results)
        avg_similarity = sum(r["edit_similarity"]
                             for r in results) / total
        exact_match_rate = sum(1 for r in results
                               if r["exact_match"]) / total
        avg_f1 = sum(r["f1_score"] for r in results) / total

        return {
            "total_cases": total,
            "avg_edit_similarity": round(avg_similarity, 4),
            "exact_match_rate": round(exact_match_rate, 4),
            "avg_f1": round(avg_f1, 4),
            "details": results
        }
检查理解：为什么不用 HumanEval 直接评估企业代码助手？

HumanEval 是「从 docstring 生成独立函数」，而企业代码助手的核心场景是「在已有代码上下文中的补全和问答」。HumanEval 的任务类型不匹配——它测试的是「零样本生成能力」，而非「上下文中的代码补全」。企业评估集应该来自真实的 commit 和 PR 变更。

采纳率分析：多维度的有效性指标

acceptance rate 是代码补全场景的核心商业指标。但单一的「总体采纳率」无法定位问题。CorpAssist 按四个维度拆解：语言维度、代码长度维度、函数类型维度、时间维度。

Java：多维度采纳率统计器

java 
// CorpAssist 多维度采纳率分析
public class AcceptanceAnalyzer {
    private final CompletionLogRepository logRepo;

    public AcceptanceAnalyzer(CompletionLogRepository repo) {
        this.logRepo = repo;
    }

    public AcceptanceReport analyze(LocalDate start, LocalDate end) {
        var logs = logRepo.findByDateRange(start, end);
        var report = new AcceptanceReport();

        // 总体采纳率
        long total = logs.size();
        long accepted = logs.stream()
            .filter(CompletionLog::isAccepted).count();
        report.setOverallRate((double) accepted / total);

        // 按语言维度
        Map> byLang = logs.stream()
            .collect(Collectors.groupingBy(CompletionLog::getLanguage));
        for (var entry : byLang.entrySet()) {
            long langTotal = entry.getValue().size();
            long langAccepted = entry.getValue().stream()
                .filter(CompletionLog::isAccepted).count();
            report.addLanguageRate(entry.getKey(),
                (double) langAccepted / langTotal);
        }

        // 按补全长维度（短 < 10 tokens, 中 10-50, 长 > 50）
        report.setShortRate(calcRateByLength(logs, 0, 10));
        report.setMediumRate(calcRateByLength(logs, 10, 50));
        report.setLongRate(calcRateByLength(logs, 50, Integer.MAX_VALUE));

        // 按函数类型
        report.setNewFuncRate(calcRateByType(logs, "new_function"));
        report.setEditRate(calcRateByType(logs, "edit_existing"));
        report.setCommentRate(calcRateByType(logs, "comment"));

        // 按时间维度（周趋势）
        report.setWeeklyRates(calcWeeklyRates(logs));

        return report;
    }

    public record AcceptanceReport(
        double overallRate,
        Map languageRates,
        double shortRate, double mediumRate, double longRate,
        double newFuncRate, double editRate, double commentRate,
        Map weeklyRates
    ) { /* ... 略 getters/setters ... */ }
}

常见问题（FAQ）

采纳率从 30% 提升到 40% 后一直停滞，怎么突破？
30-40% 采纳率是多数代码助手的「中等水平」。要突破 40%，需要：① 深入分析未采纳的 60%——是质量差还是延迟高？② 针对长代码块的补全质量做专项优化（长补全的采纳率通常低于短补全）；③ 引入企业专用词汇和框架知识；④ 考虑领域微调。突破口在长尾场景——那些「偶尔触发但一旦触发就很有价值」的补全。
离线评估准确率 85%，线上采纳率只有 25%，为什么偏差这么大？
典型「评估-生产偏差」。原因：① 离线测试集是精心标注的，线上场景的多样性远超测试集；② 离线测试不涉及延迟——线上如果延迟高，即使结果正确开发者也会跳过；③ UI/UX 因素——补全的展示方式（位置、颜色、触发时机）影响采纳率。解决方案：让评估集更接近线上分布（从生产日志采样），并增加 UX 因素的分析。
如何向管理层汇报代码助手的 ROI？
用「时间节省 × 开发者数量 × 人力成本 = ROI」的公式。关键数据：① 日节省时间（来自采纳率 × 每次补全节省时间）；② 月度节省金额（节省小时数 × 平均时薪）；③ 成本（API/GPU/人力）。举例：500 人团队，日节省 18 分钟 = 月节省 3300 小时 ≈ 时薪 150 元 = 月节省 50 万元，减去成本月 8 万元 = ROI 525%。
不推荐
  • 离线评估和线上指标割裂
  • 只看采纳率不看延迟和覆盖率
  • 没有建立基准线（before 数据）
  • ROI 计算忽略隐性成本
推荐
  • 评估集从生产日志采样，缩小偏差
  • 多维度指标体系：准确率+采纳率+延迟
  • 上线前记录基线数据
  • ROI 计算包含 GPU/人力/运维全成本

本章结论

  • 建企业评估集：humaneval analog 从真实 commit 构建，而非依赖公开 benchmark。
  • 多维度分析：按语言/长度/函数类型/时间四维拆解采纳率。
  • 持续对比：优化前后的延迟 P50/P95/P99 都要追踪。
  • 评估生产一致性：缩小离线评估与线上指标的偏差。
  • ROI 汇报框架：时间节省 × 人力成本 - 总成本。

判断练习

  1. 采纳率从 30% 掉到 20%，你最先看什么数据？

    参考答案

    先看「分语言采纳率」——可能某语言的采纳率暴跌（如新接入的 Go 语言），拉低了总体。其次看「延迟趋势」——如果同期延迟上升（如 GPU 负载高），采纳率下降是正常的。最后看「代码变更率」——如果团队最近接手了大量新项目（不熟悉代码）可能采纳率暂时下降。

  2. ROI 计算中月节省 50 万，但管理层质疑数据真实性。你怎么回应？

    参考答案

    ① 区分「感知节省」和「实测节省」——建议做 A/B 测试：对照组无助手，实验组有助手，统计完成标准任务（如 CRUD 接口）的时间差异；② 采纳率数据可从日志直接验证，不做估算；③ 提供行业对比数据（GitHub Copilot 的研究报告显示节省约 20-30% 时间）；④ 建议先做小规模试点（50 人），收集 1 个月的数据再推算全团队。

# CorpAssist 代码索引设计文档 v2.0

## 1. 架构总览
- 触发层：Git Webhook → RabbitMQ → 索引消费
- 解析层：Tree-sitter 多语言解析
- 向量化层：code-embed-v1 (768d) + GPU 批处理
- 存储层：Milvus + PostgreSQL + Redis

## 2. 切片策略
- 最小单元：函数/方法定义
- 最大单元：类定义（含子方法）
- 超大函数（>200行）：按逻辑块二次切分
- 元数据：{path, start_line, end_line, type, name, language, repo_id, commit_sha}

## 3. 索引 Schema (Milvus)
- 集合：code_chunks_{tenant_id}
- 向量字段：embedding (FLOAT_VECTOR, 768d)
- 标量字段：chunk_id, repo_id, file_path, language, start_line, commit_sha
- 索引类型：IVF_FLAT, nlist=1024

## 4. 增量更新
- 触发：post-receive hook
- 处理：只索引 diff 中的代码文件
- 版本控制：每个 chunk 记录 commit_sha，支持回滚

- 租户隔离：独立 collection
- 仓库可见性：repo_id 字段过滤

常见问题（FAQ）

索引设计文档需要多详细？
详细到「一个新后端工程师可以照着文档独立搭建索引管线」。包含：① 环境依赖（Python 版本、Milvus 版本、GPU 驱动）；② 配置文件（YAML 或 JSON 示例）；③ 启动命令和验证步骤；④ 常见问题和排障指南。索引 schema 的每个字段都要有注释说明用途。
Demo 需要准备几个环境？

安全报告中的「已接受风险」是什么意思？什么时候用？
「已接受风险」是指经过评估后认为该风险可接受，不需要修复。适用条件：① 风险发生概率极低；② 影响范围很小；③ 修复成本远大于风险损失。需要安全团队负责人在报告上签字确认。例如：某个低风险的告警只在特定配置下出现，且不影响核心功能。

  • 只有代码没有文档，无法交接
  • Demo 环境与生产大不相同，演示时出问题
  • 安全报告只列出问题没有修复方案

第三问：准确率 — 如何衡量和提升代码补全的准确率

accuracy question 考察候选人对代码助手质量评估的理解。面试官想看的是：如何定义「准确」、如何度量、如何持续提升。

回答框架

定义层：准确率≠采纳率。准确率是「补全代码是否符合预期」（离线指标），采纳率是「用户是否接受」（线上指标）。两类指标都要追踪。code assistant qa 评估框架覆盖三层：单元测试级（Edit-Similarity）、功能级（是否正确实现功能）、安全级（是否引入漏洞）。

度量层：① 离线评估——构建企业 HumanEval 类比评估集（500+ 人工标注的补全测试），计算 Edit-Similarity 和 Exact Match；② 线上评估——多维采纳率（按语言/长度/类型/时间）；③ 回归门禁——每次模型更新必须通过评估集。

提升层：从三方面持续优化——检索质量（Embedding 模型 + chunk 策略）、上下文质量（文件选择 + 预算管理）、模型质量（FIM 格式 + LoRA 微调）。

第四问：安全 — 如何保障代码助手的数据安全

security code qa 在企业面试中越来越重要。面试官想知道：你是否理解企业安全合规的底线。

答题要点

  • 数据边界：插件只发送必要上下文（prefix/suffix + 文件 hash），不全量上传。文件 hash 用于服务端判断是否需要从索引读取。
  • 传输安全：全链路 HTTPS + mTLS 双向认证 + JWT 短期 token（15 分钟刷新）。
  • 隐私模式：插件提供「隐私模式」开关，开启后所有请求走本地小模型，数据不离开本机。
  • 数据留存：服务端不持久化用户的代码片段，响应后 5 分钟 TTL 自动清除。
  • 审计日志：记录请求元数据（时间、用户、操作类型等），不记录代码全文。
面试加分项

提及「做过安全红队测试」「发现过提示注入漏洞」「有 OWASP Top 10 的检测经验」会显著加分。如果还处理过「内部员工通过代码助手窃取代码」的失控场景，说明你有真实的安全攻防经验。

常见问题（FAQ）

面试中的「代码助手五连问」每个问题的回答时间应该多长？

# CorpAssist 采纳率分析器
import json
from datetime import datetime, timedelta

def analyze_adoption(logs_path: str, days: int = 30) -> dict:
    """分析代码补充采纳率"""
    cutoff = datetime.now() - timedelta(days=days)
    total_suggestions = 0
    accepted = 0
    lang_stats = {}

    with open(logs_path, 'r') as f:
        for line in f:
            entry = json.loads(line)
            ts = datetime.fromisoformat(entry['timestamp'])
            if ts < cutoff:
                continue
            total_suggestions += 1
            if entry.get('accepted'):
                accepted += 1
            lang = entry.get('language', 'unknown')
            lang_stats.setdefault(lang, {'total': 0, 'accepted': 0})
            lang_stats[lang]['total'] += 1
            if entry.get('accepted'):
                lang_stats[lang]['accepted'] += 1

    overall = accepted / total_suggestions if total_suggestions else 0
    per_lang = {
        lang: stats['accepted'] / stats['total']
        for lang, stats in lang_stats.items()
        if stats['total'] > 0
    }
    return {
        "total_suggestions": total_suggestions,
        "overall_adoption": round(overall, 4),
        "per_language": {k: round(v, 4) for k, v in per_lang.items()}
    }

# 使用示例
stats = analyze_adoption("completion_logs.jsonl")
print(f"总体采纳率: {stats['overall_adoption']:.1%}")
for lang, rate in stats['per_language'].items():
    print(f"  {lang}: {rate:.1%}")

// CorpAssist ROI 计算器
public class CodeAssistantROI {
    public record ROIMetrics(
        int developerCount,
        double dailyTimeSavedMinutes,
        double monthlyLicenseCost,
        double avgDeveloperHourlyRate
    ) {
        public double monthlyTimeSavedHours() {
            return developerCount * dailyTimeSavedMinutes * 22 / 60.0;
        }

        public double monthlyMonetaryValue() {
            return monthlyTimeSavedHours() * avgDeveloperHourlyRate;
        }

        public double monthlyROI() {
            double benefit = monthlyMonetaryValue();
            return (benefit - monthlyLicenseCost) / monthlyLicenseCost;
        }

        public String report() {
            return String.format("""
                CorpAssist ROI Report
                ========================
                开发者数量: %d
                人均日节省: %.1f 分钟
                月节省时: %.1f 小时
                月节省金额: ¥%.2f
                月许可证成本: ¥%.2f
                月度 ROI: %.2f
                """, developerCount, dailyTimeSavedMinutes,
                monthlyTimeSavedHours(), monthlyMonetaryValue(),
                monthlyLicenseCost, monthlyROI());
        }
    }

    public static void main(String[] args) {
        var metrics = new ROIMetrics(500, 18, 80000, 150.0);
        System.out.println(metrics.report());
    }

常见问题（FAQ）

补全和问答哪个场景先落地更好？
一般推荐先做补全。补全门槛低（部署 FIM 模型即可）、开发者感知强（每天触发几十次）。采纳率做到 30% 以上后，团队积累了模型优化和延迟调优经验，再推进问答场景会更顺畅。但如果团队主要痛点是「代码理解」，也可以先上问答。
Adoption Rate 和 Time Saved 怎么精确衡量？
采纳率从插件端采集：每次补全建议打日志，记录是否被 Tab 采纳。时间节省用 A/B 对比：对照组无助手，实验组有助手，统计同一类任务（如写单元测试）的平均完成时间。注意区分「感知节省」和「实际节省」，建议两者都追踪。
小团队（< 50 人）值得落地代码助手吗？
值得，但路径不同。小团队可以直接用商业方案（如 GitHub Copilot）快速体验，积累使用数据后再评估自建 RAG 问答的必要性。关键是先让开发者用起来，而不是先建基础设施。

  • 三大场景各有侧重：补全追求低延迟高采纳，问答追求准确率与引用，Review 追求漏洞检出与覆盖率。
  • 指标要量化：adoption rate、time saved、code assistant roi 是向管理层汇报的关键数据。
  • 模块化架构：Index / Completion / Q&A / Review 四模块分工明确，可独立迭代。
  • 场景有顺序：从补全起步，积累模型优化经验后依次推进问答和 Review。
  • 持续度量：采纳率 < 30% 时先优化质量，> 40% 时关注延迟和覆盖率。

# CorpAssist 上下文预算验证器
import tiktoken

class ContextBudget:
    SCENARIOS = {
        "completion": {
            "max_tokens": 10000,
            "alloc": {
                "system": 0.10, "cursor": 0.40, "retrieval": 0.25,
                "history": 0.05, "tool": 0.10, "reserve": 0.10,
            }
        },
        "qa": {
            "max_tokens": 32000,
            "alloc": {
                "system": 0.10, "cursor": 0.05, "retrieval": 0.50,
                "history": 0.20, "tool": 0.10, "reserve": 0.05,
            }
        }
    }

    def __init__(self, scenario: str = "completion"):
        self.cfg = self.SCENARIOS[scenario]
        self.encoder = tiktoken.get_encoding("cl100k_base")
        self.usage = {k: 0 for k in self.cfg["alloc"]}

    def _count_tokens(self, text: str) -> int:
        return len(self.encoder.encode(text))

    def add(self, layer: str, content: str) -> bool:
        tokens = self._count_tokens(content)
        max_layer = int(self.cfg["max_tokens"] * self.cfg["alloc"][layer])
        if self.usage[layer] + tokens > max_layer:
            return False
        self.usage[layer] += tokens
        return True

    def validate(self) -> dict:
        total = sum(self.usage.values())
        limits = {k: int(self.cfg["max_tokens"] * v)
                  for k, v in self.cfg["alloc"].items()}
        return {
            "total": total,
            "max": self.cfg["max_tokens"],
            "usage_pct": round(total / self.cfg["max_tokens"] * 100, 1),
            "by_layer": {k: {"used": self.usage[k], "limit": limits[k]}
                         for k in self.cfg["alloc"]}
        }

budget = ContextBudget("qa")

Java：文件选择器实现

public class FileSelector {
    private final SymbolTable symbolTable;
    private final VectorStore vectorStore;
    private final GitHistoryAnalyzer gitAnalyzer;

    public FileSelector(SymbolTable symbolTable,
                        VectorStore vectorStore,
                        GitHistoryAnalyzer gitAnalyzer) {
        this.symbolTable = symbolTable;
        this.vectorStore = vectorStore;
        this.gitAnalyzer = gitAnalyzer;
    }

    public List selectFiles(CursorContext cursor, int maxFiles) {
        Set selected = new LinkedHashSet<>();
        selected.add(cursor.currentFile());
        selected.addAll(resolveImports(cursor));

        if (selected.size() < maxFiles) {
            List semantic = vectorStore.search(
                cursor.getSurroundingCode(), maxFiles - selected.size());
            selected.addAll(semantic);
        }
        if (selected.size() < maxFiles) {
            List recent = gitAnalyzer.recentlyModified(
                cursor.currentFile(), maxFiles - selected.size());
            selected.addAll(recent);
        }
        return new ArrayList<>(selected).stream()
            .limit(maxFiles).collect(Collectors.toList());
    }

    private List resolveImports(CursorContext cursor) {
        return symbolTable.getDefinitions(cursor.getImportSymbols());
    }
}

Python：Tree-sitter 符号提取

python 
from tree_sitter import Language, Parser

class SymbolTableBuilder:
    QUERIES = {
        "java": """
            (class_declaration
              name: (identifier) @class.name) @class
            (method_declaration
              name: (identifier) @method.name) @method
        """,
        "python": """
            (class_definition
              name: (identifier) @class.name) @class
            (function_definition
              name: (identifier) @func.name) @function
        """,
        "go": """
            (method_declaration
              name: (field_identifier) @method.name) @method
            (function_declaration
              name: (identifier) @func.name) @function
        """
    }

    def __init__(self):
        self.parser = Parser()
        self.symbols = {}

    def extract_from_file(self, filepath: str) -> list:
        with open(filepath, 'rb') as f:
            tree = self.parser.parse(f.read())
        lang = filepath.split('.')[-1]
        query_str = self.QUERIES.get(lang, "")
        if not query_str:
            return []
        query = Language(lang).query(query_str)
        captures = query.captures(tree.root_node)
        symbols = []
        for node, tag in captures:
            symbols.append({
                "name": node.text.decode(),
                "kind": tag,
                "file": filepath,
                "line": node.start_point[0] + 1,
            })
            self.symbols[node.text.decode()] = symbols[-1]
        return symbols

常见问题（FAQ）

LSP 查询会引入额外延迟吗？IDE 响应变慢怎么办？
LSP 查询通常 < 50ms，但如果 LSP Server 本身很慢（如大型 TypeScript 项目），可以缓存结果并用异步查询。策略：首次查询 LSP 后缓存符号位置，后续直接读缓存。另外控制 LSP 查询频次——不是每次键入都查，只在光标停留 > 200ms 时触发。
符号表多久更新一次？CI 流程中怎么同步？
推荐 Git post-commit hook 触发增量更新：只重新解析变更文件的符号。全量重建每天一次（凌晨低峰期）。CI 流程中，符号表作为构建产物的一部分发布，确保模型服务读取的符号表和代码版本一致。
没有 LSP Server 的语言（如某些 DSL）怎么办？
回退到 Tree-sitter 纯语法解析。优先级：LSP > Tree-sitter > 正则。没有 LSP 时可以用 Tree-sitter 提取 AST 结构作为伪符号表，或者对于简单 DSL，直接传原始文本。
不推荐
  • 忽略 LSP，只用字符串匹配找定义
  • 每次补全都全量重建符号表
  • 零缓存策略，LSP 查询无限制调用
  • 不处理跨文件符号引用
推荐
  • 优先使用 LSP 获取精确上下文
  • 符号表增量更新而非全量重建
  • LSP 结果缓存 + 异步刷新
  • 符号表支持跨文件/跨语言引用

ast chunk 是指以 AST 节点为边界进行代码切片。最小单元是函数/方法定义，最大单元是文件。切片策略需要在「保留完整语义」和「控制块大小」之间平衡。

    }

    def __init__(self):
        self.parser = Parser()

    def chunk_file(self, filepath: str) -> list:
        lang = filepath.split('.')[-1]
        node_types = self.NODE_TYPES.get(lang, [])
        with open(filepath, 'rb') as f:
            source = f.read()
            tree = self.parser.parse(source)

        chunks = []
        self._extract_chunks(tree.root_node, source,
                             filepath, node_types, chunks)

        # 如果没有提取到命名节点，整个文件作为一个 chunk
        if not chunks:
            chunks.append({
                "text": source.decode("utf-8"),
                "metadata": {
                    "path": filepath,
                    "start_line": 1,
                    "end_line": len(source.decode().splitlines()),
                    "type": "file",
                    "name": filepath.split("/")[-1]
                }
            })
        return chunks

    def _extract_chunks(self, node, source, path,
                        node_types, chunks):
        if node.type in node_types:
            name_node = node.child_by_field_name("name")
            start = node.start_point[0] + 1
            end = node.end_point[0] + 1
            text = source[node.start_byte:node.end_byte].decode()
            chunks.append({
                "text": text,
                "metadata": {
                    "path": path,
                    "start_line": start,
                    "end_line": end,
                    "type": node.type,
                    "name": (name_node.text.decode()
                             if name_node else "unknown"),
                    "language": path.split('.')[-1]
                }
            })
        for child in node.children:
            self._extract_chunks(child, source, path,
                                 node_types, chunks)

java 
public class ChunkMerger {
    private static final int MAX_CHUNK_LINES = 150;
    private static final int MIN_CHUNK_LINES = 10;

    public List mergeChunks(List chunks) {
        List merged = new ArrayList<>();
        CodeChunk current = null;

        for (CodeChunk chunk : chunks) {
            if (current == null) {
                current = chunk;
                continue;
            }
            // 如果当前 chunk 太小且下一个 chunk 在同一文件，合并
            if (chunk.metadata().file().equals(
                    current.metadata().file())
                && current.lineCount() < MIN_CHUNK_LINES) {
                current = current.merge(chunk);
            } else {
                merged.add(current);
                current = chunk;
            }
        }
        if (current != null) {
            merged.add(current);
        }
        return merged;
    }

    public record CodeChunk(
        String text,
        ChunkMetadata metadata
    ) {
        public int lineCount() {
            return text.split("\n").length;
        }
        public CodeChunk merge(CodeChunk other) {
            return new CodeChunk(
                this.text + "\n" + other.text,
                new ChunkMetadata(
                    metadata.file(),
                    metadata.startLine(),
                    other.metadata.endLine(),
                    metadata.type(),
                    metadata.name() + "+" + other.metadata.name(),
                    metadata.language()
                )
            );
        }
    }
    public record ChunkMetadata(
        String file, int startLine, int endLine,
        String type, String name, String language
    ) {}
}

仓库索引与多语言支持

repo index 将切片后的代码块构建为可检索的索引。它包含三层：向量索引（语义搜索）、符号索引（精确查找）、引用图（跨文件关系）。repo embedding 使用专门的代码 Embedding 模型（如 code-embed-v1 或 starencoder），比通用 Embedding 更理解代码语义。

multi lang 支持是企业代码助手的刚需——一个微服务项目可能同时包含 Java、Go、Python 和 JavaScript。统一索引的关键是：

  • 每种语言使用对应的 Tree-sitter 语法解析器
  • 切片元数据统一为 path/start_line/end_line/type/name/language 六个字段
  • Embedding 模型需要跨语言对齐（同一语义在不同语言中向量相近）
  • 引用图支持跨语言符号解析（Java 类调用 Go 服务的方法）

常见误区：多语言 Embedding 不对齐

如果 Java 用 code-embed-v1，Python 用另一个模型，检索效果会变差。同一个语义在不同语言中应该映射到相近向量空间。推荐用统一的 Code Embedding 模型（如 CodeBERT 或 StarCoder Embedding）覆盖所有目标语言。

常见问题（FAQ）

函数级别的 chunk 大小该怎么控制？太大会不会影响检索效果？
函数 chunk 推荐 50-150 行。超过 200 行的函数可以按逻辑块切分：if/for 分支提取为独立 chunk，但保留 parent 字段指向父函数。检索时若找到子块，同时返回父函数上下文以提高相关性。
代码 RAG 的准确率比通用 RAG 高多少？有数据吗？
CorpAssist 内部测试表明：在仓库问答场景中，AST 切片的 Top-5 相关代码命中率约 78%，相同数据下通用文本切片仅约 52%。提升主要在两个方面：函数边界完整（减少半个函数的噪声）、元数据丰富（可按符号名精确过滤）。
大仓库（1000+ 文件）的全量索引需要多久？如何加速？
取决于文件数量和语言。1000 个 Java 文件约 5-10 分钟完成解析+向量化。加速方案：① 多进程并行解析；② 只索引 main/production 分支，跳过 test 目录；③ 使用 GPU 批量 Embedding；④ Git push 触发增量更新，避免全量重建。
不推荐
  • 检索结果只返回代码文本，无位置信息
  • 所有语言共享一个 Tree-sitter 解析器
  • 索引全量重建，不支持增量更新
  • 不维护跨文件引用关系
推荐
  • 每个代码块带 path/line 元数据
  • 每种语言使用对应 Tree-sitter 语法
  • Git push 触发增量索引
  • 构建符号引用图支持跨文件跳转

本章结论

  • 代码 RAG ≠ 通用 RAG：以 AST 为切分单元是代码 RAG（code rag arch）的核心特征。
  • AST 切片保留语义：function chunk 按函数/类/方法为边界切分，提升检索精度 30%+。
  • 多模态索引：向量索引 + 符号索引 + 引用图协同，覆盖语义搜索和精确查找。
  • 引用跳转提升 UX：每个代码块附带精确文件位置，支持 IDE 一键跳转。
  • 多语言统一架构：统一元数据格式 + 跨语言 Embedding + 每种语言专用解析器。

# CorpAssist 静态规则引擎
import re
from abc import ABC, abstractmethod

class SecurityRule(ABC):
    @abstractmethod
    def check(self, code: str) -> list:
        pass

class SqlInjectionRule(SecurityRule):
    PATTERNS = [
        (r'SELECT\s+.*\s+FROM\s+.*\s+WHERE\s+.*\+\s*[\'"]', "SQL 拼接注入"),
        (r'execute\s*\(\s*["\']\s*SELECT', "直接执行未参数化 SQL"),
        (r'\${\s*\w+\s*}\s*\+\s*[\'"]', "MyBatis $ 符号拼接"),
    ]

    def check(self, code: str) -> list:
        findings = []
        for pattern, desc in self.PATTERNS:
            for i, line in enumerate(code.split('\n'), 1):
                if re.search(pattern, line, re.IGNORECASE):
                    findings.append({
                        "rule": "SQL_INJECTION",
                        "severity": "critical",
                        "line": i,
                        "message": f"检测到{desc}，建议使用参数化查询",
                        "suggestion": (
                            "PreparedStatement / parameterized query "
                            "替代字符串拼接"
                        )
                    })
        return findings

class HardcodedSecretRule(SecurityRule):
    PATTERNS = [
        (r'(password|secret|token|api_key)\s*=\s*[\'\"][^\'\"]+[\'\"]',
         "硬编码密钥/密码"),
        (r'AKIA[0-9A-Z]{16}', "AWS Access Key"),
        (r'sk-[a-zA-Z0-9]{32,}', "OpenAI API Key"),
    ]

    def check(self, code: str) -> list:
        findings = []
        for pattern, desc in self.PATTERNS:
            for i, line in enumerate(code.split('\n'), 1):
                if re.search(pattern, line):
                    findings.append({
                        "rule": "HARDCODED_SECRET",
                        "severity": "critical",
                        "line": i,
                        "message": f"检测到{desc}",
                        "suggestion": "使用环境变量或密钥管理服务"
                    })
        return findings

class RuleEngine:
    def __init__(self):
        self.rules = [SqlInjectionRule(), HardcodedSecretRule()]

    def analyze(self, code: str) -> list:
        all_findings = []
        for rule in self.rules:
            all_findings.extend(rule.check(code))
        return sorted(all_findings, key=lambda x: x["line"])

  • 规则过于严格，大量误报导致开发者忽略
  • 规则集与语言无关，Java 规则检测 Python 代码
  • 检测到问题但不提供修复建议

检查理解：为什么 AI 安全审查不能完全替代静态规则？

静态规则检测到了误报怎么办？开发者可以忽略吗？

Git 增量索引：只处理变更

git incremental 策略的核心是：只处理 Git diff 中涉及的文件，不重新解析未变更的代码。这比全量索引快 10-50 倍。incremental index 的难点在于「旧 chunk 的失效检测」——修改后的函数必须覆盖旧索引。

Python：增量索引触发器

python 
# CorpAssist Git 增量索引器
import subprocess
import json
from pathlib import Path

class IncrementalIndexer:
    def __init__(self, repo_path: str, indexer):
        self.repo_path = Path(repo_path)
        self.indexer = indexer

    def process_push(self, old_rev: str, new_rev: str) -> dict:
        """处理 Git push 事件，增量更新索引"""
        changed_files = self._get_changed_files(old_rev, new_rev)
        stats = {"files_changed": 0, "chunks_added": 0,
                 "chunks_removed": 0, "errors": []}

        for file_path in changed_files:
            full_path = self.repo_path / file_path
            if not full_path.exists():
                # 文件被删除，移除索引
                removed = self.indexer.remove_file(file_path)
                stats["chunks_removed"] += removed
                continue

            try:
                # 解析新版本，覆盖旧索引
                new_chunks = self.indexer.index_file(str(full_path))
                self.indexer.overwrite_chunks(file_path, new_chunks)
                stats["chunks_added"] += len(new_chunks)
                stats["files_changed"] += 1
            except Exception as e:
                stats["errors"].append(f"{file_path}: {e}")

        return stats

    def _get_changed_files(self, old_rev: str,
                           new_rev: str) -> list:
        """获取两次提交间的变更文件列表"""
        result = subprocess.run(
            ["git", "diff", "--name-only",
             old_rev, new_rev],
            capture_output=True, text=True,
            cwd=self.repo_path
        )
        files = result.stdout.strip().split('\n')
        # 过滤出代码文件
        code_exts = {'.py', '.java', '.go', '.js', '.ts'}
        return [f for f in files
                if Path(f).suffix in code_exts]

    def full_reindex(self, branch: str = "main"):
        """全量重建（用于初始或修复）"""
        files = Path(self.repo_path).rglob("*")
        code_files = [f for f in files
                      if f.suffix in {'.py', '.java', '.go'}]
        for f in code_files:
            rel_path = str(f.relative_to(self.repo_path))
            chunks = self.indexer.index_file(str(f))
            self.indexer.overwrite_chunks(rel_path, chunks)

    @property
    def branch_scope(self) -> list:
        """返回配置的索引分支列表"""
        return ["main", "develop"]
不推荐
  • 每次 push 触发全量重建，耗时 30 分钟+
  • 所有分支都索引，浪费存储和计算
  • 删除文件后不清理索引，产生脏数据
  • 不处理并发 push 的冲突
推荐
  • 增量索引只处理 diff 文件
  • 只索引 main/develop 等核心分支
  • 文件删除时同步清除对应 chunk
  • 使用分布式锁处理并发 push
检查理解：增量索引和全量重建如何配合使用？

增量索引处理日常变更（秒级），全量重建用于初始构建或数据修复（定时每周一次凌晨执行）。增量索引失败时自动回退到该文件的上一版本索引，不影响整体服务。全量重建前先验证增量日志是否有累计偏差。

# CorpAssist 批处理向量化
import numpy as np
from sentence_transformers import SentenceTransformer

class CodeEmbeddingService:
    def __init__(self, model_name: str = "code-embed-v1"):
        self.model = SentenceTransformer(model_name)

    def embed_chunks(self, chunks: list) -> np.ndarray:
        """批处理向量化"""
        texts = [c["text"] for c in chunks]
        # 添加元数据特征增强
        augmented = []
        for chunk in chunks:
            meta = chunk["metadata"]
            augmented.append(
                f"{meta['language']} {meta['type']} "
                f"{meta['name']}\n{chunk['text']}"
            )
        embeddings = self.model.encode(
            augmented,
            batch_size=32,
            show_progress_bar=True
        )
        return embeddings

    def embed_query(self, query: str) -> np.ndarray:
        """查询向量化"""
        # 添加代码查询前缀
        enhanced = f"code query: {query}"
        return self.model.encode([enhanced])[0]
常见误区：Embedding 模型不更新

代码 Embedding 模型需要定期用公司代码库微调。私有代码中的特定缩写、框架名、业务术语可能不在通用模型的训练集中。建议每季度用最新的代码库数据微调一次 Embedding 模型。

Milvus 集合设计与权限控制

milvus collection 是 Milvus 中的表结构设计。CorpAssist 采用「每个租户一个 collection，每个 chunk 带权限标签」的混合策略。

Java：Milvus 集合管理与权限查询

java 
// CorpAssist Milvus 集合管理器
public class MilvusCollectionManager {
    private final MilvusServiceClient client;

    public MilvusCollectionManager(MilvusServiceClient client) {
        this.client = client;
    }

    public void createCodeChunksCollection(String tenantId) {
        String collectionName = "code_chunks_" + tenantId;
        var schema = CollectionSchema.newBuilder()
            .addField("chunk_id", DataType.VarChar)
            .addField("embedding", DataType.FloatVector, 768)
            .addField("repo_id", DataType.VarChar)
            .addField("file_path", DataType.VarChar)
            .addField("start_line", DataType.Int64)
            .addField("language", DataType.VarChar)
            .addField("visibility", DataType.VarChar)
            .build();

        var params = IndexParam.newBuilder()
            .withFieldName("embedding")
            .withIndexType(IndexType.IVF_FLAT)
            .build();

        client.createCollection(CreateCollectionParam.newBuilder()
            .withCollectionName(collectionName)
            .withSchema(schema)
            .build());
        client.createIndex(CreateIndexParam.newBuilder()
            .withCollectionName(collectionName)
            .withIndexParam(params)
            .build());
    }

    public SearchResult searchWithACL(String tenantId,
                                       float[] queryVector,
                                       String userRole,
                                       int topK) {
        String collectionName = "code_chunks_" + tenantId;
        // 使用布尔表达式做权限过滤
        String filter = userRole.equals("admin")
            ? "" : "visibility in ['public', 'internal']";

        return client.search(SearchParam.newBuilder()
            .withCollectionName(collectionName)
            .withVectors(Collections.singletonList(queryVector))
            .withVectorFieldName("embedding")
            .withTopK(topK)
            .withExpr(filter)
            .build());
    }
}

repo acl 权限模型分为三层：租户隔离（collection 级别）、仓库可见性（repo_id 过滤）、文件权限（visibility 字段）。branch scope 决定哪些分支的代码被索引——通常只索引 main 和 release 分支，避免 feature 分支的临时代码污染索引。

常见问题（FAQ）

增量索引中，如果两个 push 几乎同时到达，怎么处理冲突？
用乐观锁 + 文件级锁机制。每个文件有一个版本号（基于 commit SHA），索引写入时检查版本号。如果写入时的版本号低于当前索引的版本号，说明该文件已被更新的 commit 处理，跳过本次写入。同时用消息队列缓冲 push 事件，消费端单线程逐个处理。
Embedding 模型的维度（768）是怎么确定的？可以改吗？
768 是 code-embed-v1 模型的默认输出维度。Milvus 支持不同维度的向量，但同一 collection 中的向量维度必须一致。如果需要切换模型（如从 768 维切到 1024 维），需要重建 collection。建议上线前确定模型，避免中途切换。
索引失败会影响线上服务吗？如何保证高可用？
索引和检索是分离的，索引失败不影响检索服务。检索读的是已经索引好的数据。如果增量索引失败，CorpAssist 会：① 记录失败日志到死信队列；② 该文件的索引回退到上一个成功版本；③ 告警通知运维。索引服务本身无状态，可以多副本部署。
不推荐
  • 所有租户共享一个 collection，无隔离
  • 权限过滤在 API 层做，索引层不感知
  • 所有分支都索引，包括 feature 分支
  • 索引失败后无降级策略
推荐
  • 每个租户独立 collection
  • 索引层通过字段过滤实现权限
  • 只索引 main/release 等稳定分支
  • 索引失败回退上一版本 + 死信队列

本章结论

  • 增量索引核心：git incremental 只处理 diff 文件，秒级完成。
  • 代码专用 Embedding：code embedding 比通用模型 Recall 高 20%+。
  • Milvus 多租户设计：每个租户独立 collection，权限在索引层过滤。
  • 分支范围控制：branch scope 只索引稳定分支，避免脏数据。
  • 可观测性：索引流水线的每个阶段都需要监控速率、延迟和错误率。

python 
# CorpAssist 仓库问答服务
from typing import AsyncGenerator
import numpy as np

class RepoQAService:
    def __init__(self, vector_store, llm_client, embed_model):
        self.vector_store = vector_store
        self.llm = llm_client
        self.embedder = embed_model

    async def answer(self, query: str, repo_id: str,
                     user_context: dict = None) -> dict:
        # 1. 查询向量化
        query_vec = self.embedder.embed_query(query)

        # 2. 向量检索（带 ACL 过滤）
        results = self.vector_store.search(
            query_vec, top_k=5,
            filter_expr=f"repo_id == '{repo_id}'"
        )

        # 3. 构建上下文
        context_chunks = []
        citations = []
        for r in results:
            context_chunks.append(r["text"])
            citations.append({
                "file": r["file_path"],
                "start_line": r["start_line"],
                "end_line": r["end_line"],
                "snippet": r["text"][:200]
            })

        # 4. 组装 prompt
        prompt = self._build_prompt(query, context_chunks)

        # 5. 生成回答
        answer = await self.llm.complete(prompt)

        return {
            "answer": answer,
            "citations": citations,
            "metadata": {
                "chunks_retrieved": len(results),
                "model": self.llm.model_name
            }
        }

    async def stream_answer(self, query: str, repo_id: str
                            ) -> AsyncGenerator[str, None]:
        """流式版本"""
        query_vec = self.embedder.embed_query(query)
        results = self.vector_store.search(
            query_vec, top_k=5,
            filter_expr=f"repo_id == '{repo_id}'"
        )
        context_chunks = [r["text"] for r in results]
        prompt = self._build_prompt(query, context_chunks)
        async for token in self.llm.stream_complete(prompt):
            yield token

    def _build_prompt(self, query: str,
                      chunks: list) -> str:
        context = "\n\n---\n\n".join([
            f"文件 {i+1}:\n{c}"
            for i, c in enumerate(chunks)
        ])
        return f"""基于以下代码上下文回答问题。
如果代码中不包含相关信息，请说明"代码库中未找到相关信息"。

代码上下文：
{context}

问题：{query}

回答（引用对应代码）："""

代码引用块：精确到行号

code citation 是代码引用格式的标准。每个被引用的代码块必须包含：文件路径、起始行号、结束行号、代码片段。引用格式要支持 IDE 插件一键跳转。chunk cite 和 sse code 配合，在流式输出中也实时标注引用。

Java：引用构建器

java 
public class CitationBuilder {
    public record Citation(
        String file,
        int startLine,
        int endLine,
        String snippet,
        String symbol,
        String type
    ) {
        public String toMarkdown() {
            return String.format(
                "```%s:%d-%d\n%s\n```",
                file, startLine, endLine, snippet
            );
        }

        public String toIDEUri() {
            return String.format(
                "vscode://file/%s:%d:%d",
                file, startLine, endLine
            );
        }
    }

    public List buildCitations(List chunks) {
        return chunks.stream()
            .map(chunk -> new Citation(
                chunk.path(), chunk.startLine(), chunk.endLine(),
                truncate(chunk.text(), 200),
                chunk.symbolName(),
                chunk.type()
            ))
            .collect(Collectors.toList());
    }

    private String truncate(String text, int maxLen) {
        return text.length() <= maxLen
            ? text
            : text.substring(0, maxLen) + "...";
    }

    public String formatWithCitations(String answer,
                                       List citations) {
        StringBuilder sb = new StringBuilder(answer);
        sb.append("\n\n**引用代码：**\n");
        for (int i = 0; i < citations.size(); i++) {
            Citation c = citations.get(i);
            sb.append(String.format(
                "%d. [`%s:%d`](%s)\n",
                i + 1, c.file(), c.startLine(), c.toIDEUri()
            ));
        }
        return sb.toString();
    }
}
常见误区：引用太多反而不好

5 个以上引用块会让回答变得冗长。最佳实践是：LLM 在回答中引用 Top-3 最相关的代码块，其余作为「参考文件」列在回答末尾。每个引用块控制在 50 行以内，过长的代码块可以截断为关键部分 + 省略号。

流式响应与 SS E 推送

stream qa code 使用 Server-Sent Events (SSE) 实现流式输出。SSE 比 WebSocket 更轻量（单工、HTTP 原生支持），适合 LLM 这种「服务端单向推送」的场景。流式输出的关键在 Token 级别的引用标注——当 LLM 生成到引用标记时，实时嵌入引用信息。

Java：SSE 流式控制器

java 
// CorpAssist SSE 流式控制器
@RestController
public class QAController {
    private final RepoQAService qaService;

    @PostMapping("/v1/qa/stream")
    public SseEmitter streamAnswer(@RequestBody QARequest request) {
        SseEmitter emitter = new SseEmitter(30_000L); // 30s 超时

        executorService.execute(() -> {
            try {
                // 发送引用元数据（流式头部）
                emitter.send(SseEmitter.event()
                    .name("metadata")
                    .data(request.getCitations()));

                // 流式发送回答
                for (String token : qaService.streamAnswer(request)) {
                    emitter.send(SseEmitter.event()
                        .name("token")
                        .data(token));
                }
                emitter.send(SseEmitter.event()
                    .name("done")
                    .data(""));
                emitter.complete();
            } catch (Exception e) {
                emitter.completeWithError(e);
            }
        });
        return emitter;
    }

    @PostMapping("/v1/qa")
    public CompletableFuture answer(
            @RequestBody QARequest request) {
        return qaService.answerAsync(request);
    }
}

常见问题（FAQ）

检索结果为空时应该返回什么？
不要编造答案。返回「代码库中未找到相关信息，你可能需要在提问中使用更准确的函数名或类名」。同时建议用户关联的检索关键词。如果检索为空但 LLM 仍然回答，可能产生幻觉（hallucination），这是最危险的情况之一。
QA 的准确率怎么评估？需要人工标注吗？
建议使用人工标注的问答对评估集（至少 200 对）。评估维度：① 答案准确性（是否基于真实代码）；② 引用精确性（引用位置是否正确）；③ 完整性（是否覆盖了问题所有方面）。也可以用 LLM-as-Judge 做自动化评估，但需要交叉验证。
双栈架构（Dual Stack）是怎么工作的？
dual stack qa 指 Python 负责检索服务（向量搜索 + 排序），Java 负责 API 网关和业务编排。Python 栈利用其丰富的 ML/AI 库（sentence-transformers、numpy），Java 栈利用其成熟的框架（Spring Cloud、事务管理、权限控制）。两层通过 gRPC 或 HTTP 通信。
不推荐
  • 非流式，用户等待 5-10s 才看到回答
  • 引用格式混乱，不支持 IDE 跳转
  • 检索为空时 LLM 编造答案
  • Python 和 Java 栈职责不清，耦合严重
推荐
  • SSE 流式，首 token < 500ms
  • 统一引用格式，支持 IDE 一键跳转
  • 检索为空时明确告知用户
  • 双栈分离：Python 检索 + Java 编排

本章结论

  • 检索精度决定上限：retrieve generate 管线中，检索 Top-3 命中率是回答准确率的前提。
  • 引用必须精确：每个代码引用块附带 file:startLine:endLine，支持 IDE 跳转验证。
  • 流式输出降低等待感：stream qa code 首 token < 500ms，用户感知延迟大幅降低。
  • 双栈各司其职：dual stack qa 让 Python 做 ML 检索，Java 做业务编排。
  • 防幻觉是底线：检索为空时绝不编造，明确告知用户信息缺失。

class FIMBuilder:
    """FIM 格式构建器，支持多种模型格式"""

    TEMPLATES = {
        "starcoder": {
            "prefix": "<|fim_prefix|>",
            "suffix": "<|fim_suffix|>",
            "middle": "<|fim_middle|>",
        },
        "codel lama": {
            "prefix": "",
            "middle": "",
            "suffix_first": True,
        },
        "qwen-coder": {
            "prefix": "<|fim_prefix|>",
            "suffix": "<|fim_suffix|>",
            "middle": "<|fim_middle|>",
        },
    }

    def __init__(self, model_format: str = "qwen-coder"):
        self.template = self.TEMPLATES[model_format]
        self.model_format = model_format

    def build_prompt(self, prefix: str, suffix: str) -> str:
        fmt = self.model_format
        if fmt in ("starcoder", "qwen-coder"):
            return (f"{self.template['prefix']}{prefix}"
                    f"{self.template['suffix']}{suffix}"
                    f"{self.template['middle']}")
        elif fmt == "codel lama":
            return (f"{self.template['middle']}{suffix}"
                    f"{self.template['middle']}{prefix}")
        else:
            raise ValueError(f"Unsupported format: {fmt}")

    def extract_completion(self, full_output: str) -> str:
        """从模型输出中提取中间补全部分"""
        middle_token = self.template.get("middle", "")
        if middle_token and middle_token in full_output:
            return full_output.split(middle_token)[-1].strip()
        return full_output.strip()

    @staticmethod
    def build_prefix_suffix(source_code: str,
                            cursor_line: int,
                            cursor_col: int) -> tuple:
        """从源代码和光标位置提取 prefix 和 suffix"""
        lines = source_code.split('\n')
        prefix_lines = lines[:cursor_line - 1]
        current_line = lines[cursor_line - 1]
        prefix_lines.append(current_line[:cursor_col])
        prefix = '\n'.join(prefix_lines)

        suffix_lines = [current_line[cursor_col:]]
        suffix_lines.extend(lines[cursor_line:])
        suffix = '\n'.join(suffix_lines)

        return prefix, suffix
不推荐
  • FIM 格式与模型不匹配（如用 StarCoder 格式调 CodeLLaMA）
  • 光标前后的上下文不分，直接用「前缀」格式
  • suffix 太长（超过 2048 tokens），影响补全质量
  • 不做格式校验，模型输出包含多余的分隔符
推荐
  • 严格匹配模型的 FIM 训练格式
  • prefix + suffix 总长度控制在 4K tokens 以内
  • 限制 suffix 在 512 tokens 以内（suffix 太长会混淆模型）
  • 后处理裁剪模型输出的分隔符和特殊 token
先想 10 秒：为什么 suffix 不能太长？

大部分 FIM 模型的训练数据中，suffix 占比较小（约 10-20%）。如果 suffix 太长，超出了模型训练时的分布范围，模型可能无法正确理解上下文关系。实践中限制 suffix ≤ 512 tokens，如果光标后代码太长，只取后续的简要结构（如函数签名+大括号）。

上下文拼装：优先级与裁剪

context assemble 是将 FIM 部分与相关代码上下文合并为最终 prompt 的过程。拼装的顺序和优先级直接影响补全质量。

Java：上下文拼装器

java 
// CorpAssist 上下文拼装器
public class ContextAssembler {
    private static final int MAX_CONTEXT_TOKENS = 4096;
    private final TokenCounter tokenCounter;

    public ContextAssembler(TokenCounter tokenCounter) {
        this.tokenCounter = tokenCounter;
    }

    public CompletionRequest assemble(ContextBundle bundle) {
        var parts = new ArrayList();

        // L1: System prompt（风格约束）
        parts.add(new ContextPart("system", bundle.systemPrompt(), 100));

        // L2: 当前文件上下文（光标前 50 行 + 后 10 行）
        parts.add(new ContextPart("current_file",
            bundle.currentFileContext(), 500));

        // L3: import 的依赖文件签名
        if (bundle.hasImportContext()) {
            parts.add(new ContextPart("imports",
                bundle.importContext(), 300));
        }

        // L4: 相关文件片段（语义检索）
        if (bundle.hasRelatedChunks()) {
            parts.add(new ContextPart("related",
                bundle.relatedChunks(), 1000));
        }

        // 裁剪到预算内
        var assembled = new StringBuilder();
        int totalTokens = 0;
        for (var part : parts) {
            int tokens = tokenCounter.count(part.content());
            if (totalTokens + tokens > MAX_CONTEXT_TOKENS) {
                String truncated = tokenCounter.truncate(
                    part.content(), MAX_CONTEXT_TOKENS - totalTokens);
                assembled.append(truncated).append("\n");
                break;
            }
            assembled.append(part.content()).append("\n");
            totalTokens += tokens;
        }

        // 追加 FIM 部分
        String fimPrompt = FIMBuilder.build(
            bundle.prefix(), bundle.suffix());

        return new CompletionRequest(
            assembled.toString(),
            fimPrompt,
            bundle.cursorPosition()
        );
    }

    public record ContextPart(
        String layer, String content, int maxTokens
    ) {}
    public record CompletionRequest(
        String context, String fimPrompt, CursorPosition cursor
    ) {}
}
常见误区：上下文越多越好

上下文不是越多越好。过多的上下文会稀释注意力（Lost in the Middle 问题），并且增加延迟。补全场景建议上下文总量控制在 4K tokens 以内。优先保障当前文件的上下文（光标附近），再按依赖链添加相关文件。

常见问题（FAQ）

System Prompt 中的风格约束和模型训练风格冲突时怎么办？
System Prompt 的约束优先级高于训练数据。如果模型生成的代码不符合企业风格，说明约束不够强或上下文不够充分。建议：① 在 System Prompt 中明确说明「这是 XXX 公司的代码规范，必须严格遵守」；② 在上下文中包含同一项目已有的代码示例作为隐式约束；③ 后处理用 linter 验证并修正风格问题。
不同项目有不同的风格规范，怎么管理？
CorpAssist 支持基于仓库的配置：每个仓库在索引时记录其 .editorconfig 或 checkstyle 配置路径，补全时自动加载对应的风格约束。后端存储为 key-value（repo_id → style_config）。如果项目没有显式配置，可以通过分析仓库已有代码自动推断风格。
补全建议太长时怎么控制？
通过 max_tokens 参数控制。单行补全（如变量名、方法调用）设置 max_tokens=50；语句补全设置 max_tokens=100；代码块补全设置 max_tokens=300。同时可以在 prompt 中说明「生成最简补全即可」。另外要检测「是否生成了不必要的内容」，后处理时裁剪掉空行和注释。
不推荐
  • 风格约束全部放在 system prompt 中，冗长且不精确
  • 所有项目使用同一套风格规范
  • 风格约束只说不展示示例
  • 补全后不做 style lint 验证
推荐
  • System + 示例代码双重约束
  • 按项目/仓库配置独立的风格规范
  • 在 prompt 中包含同项目代码示例
  • 补全后用 linter 验证并修复

本章结论

  • FIM 格式精确匹配：每种模型有特定的 FIM 模板，格式错误会导致补全质量暴跌。
  • 上下文分层装：context assemble 按 System → Current → Import → Related 的顺序拼装，总量 ≤ 4K tokens。
  • 风格约束双重保证：System Prompt 描述 + 示例代码隐式约束，后处理 linter 验证。
  • 控制输出长度：根据场景设 max_tokens，单行补全 vs 代码块补全区分处理。
  • 内联补全体验：inline complete 需要 < 200ms 首 token，建议用小模型 + vLLM 加速。

多级缓存策略：从内存到语义

latency cache 是延迟优化的第一道防线。缓存分为三级：L1 精确匹配缓存、L2 语义缓存、L3 上下文预加载。semantic cache code 通过 Embedding 相似度匹配缓存命中，可以处理「语义相同、表述不同」的请求。

Python：语义缓存实现

python 
# CorpAssist 语义缓存
import hashlib
import numpy as np
from redis import Redis

class SemanticCache:
    def __init__(self, redis_client: Redis, embed_model):
        self.redis = redis_client
        self.embedder = embed_model
        self.similarity_threshold = 0.95

    def get(self, prompt: str) -> str | None:
        # L1: 精确哈希匹配
        hash_key = self._hash(prompt)
        cached = self.redis.get(f"exact:{hash_key}")
        if cached:
            return cached.decode()

        # L2: 语义匹配
        query_vec = self.embedder.embed(prompt)
        sem_keys = self.redis.smembers("semantic:keys")
        for key in sem_keys:
            vec_bytes = self.redis.get(f"vec:{key}")
            if vec_bytes is None:
                continue
            stored_vec = np.frombuffer(vec_bytes, dtype=np.float32)
            sim = np.dot(query_vec, stored_vec) / (
                np.linalg.norm(query_vec) * np.linalg.norm(stored_vec)
            )
            if sim > self.similarity_threshold:
                result = self.redis.get(f"result:{key}")
                if result:
                    return result.decode()
        return None

    def set(self, prompt: str, result: str, ttl: int = 3600):
        hash_key = self._hash(prompt)
        self.redis.setex(f"exact:{hash_key}", ttl, result)

        # 语义索引
        vec = self.embedder.embed(prompt)
        vec_bytes = vec.astype(np.float32).tobytes()
        self.redis.setex(f"vec:{hash_key}", ttl, vec_bytes)
        self.redis.setex(f"result:{hash_key}", ttl, result)
        self.redis.sadd("semantic:keys", hash_key)

    def _hash(self, prompt: str) -> str:
        return hashlib.sha256(prompt.encode()).hexdigest()[:16]
先想 10 秒：缓存的 TTL（过期时间）设多少合适？

取决于场景。补全场景的建议是：简单补全（变量名、方法调用）TTL=1h，因为同一项目中间一代码位置触发的补全高度相似。问答场景 TTL=5min，因为代码可能被更新，回答需要实时性。注意缓存需要在代码更新时失效——可以在增量索引时清除相关文件的缓存。

小模型路由：用最合适的模型

small model route 是根据请求复杂度智能选择模型大小。CorpAssist 的路由规则：预计补全长 < 10 tokens（变量名、属性访问）→ 1-3B 模型，10-50 tokens（简单语句）→ 7B 模型，> 50 tokens（代码块）→ 14B+ 模型。

Java：智能模型路由

java 
// CorpAssist 智能模型路由器
public class ModelRouter {
    private final ModelClient smallModel;   // 1-3B
    private final ModelClient mediumModel;  // 7B
    private final ModelClient largeModel;   // 14B+

    public ModelRouter(ModelClient small, ModelClient medium,
                       ModelClient large) {
        this.smallModel = small;
        this.mediumModel = medium;
        this.largeModel = large;
    }

    public ModelClient route(CompletionRequest request) {
        Complexity complexity = estimateComplexity(request);
        return switch (complexity) {
            case SIMPLE -> smallModel;
            case MODERATE -> mediumModel;
            case COMPLEX -> largeModel;
        };
    }

    private Complexity estimateComplexity(CompletionRequest req) {
        // 特征 1: 上下文中的代码模式
        String prefix = req.prefix();
        boolean isSimpleCompletion = prefix.endsWith(".")
            || prefix.endsWith("->")
            || prefix.matches(".*\\w+\\s*=\\s*$");

        if (isSimpleCompletion) {
            return Complexity.SIMPLE;
        }

        // 特征 2: 需要补全的代码块大小
        int contextLength = prefix.length() + req.suffix().length();
        boolean needsBlockCompletion = contextLength > 500
            || prefix.contains("try") || prefix.contains("for");

        if (needsBlockCompletion) {
            return Complexity.COMPLEX;
        }

        // 特征 3: 是否包含多个文件上下文
        if (req.relatedFiles() > 3) {
            return Complexity.COMPLEX;
        }

        return Complexity.MODERATE;
    }

    public enum Complexity { SIMPLE, MODERATE, COMPLEX }
}
不推荐
  • 所有请求用同一个模型（大模型延迟高，小模型质量差）
  • 路由规则过于简单（只按文件类型分）
  • 路由判断本身耗时超过 50ms
  • 不监控模型级延迟，无法调优路由规则
推荐
  • 按补全复杂度动态路由到合适模型
  • 结合上下文长度、代码模式、文件数量多维判断
  • 路由判断 < 5ms（基于规则而非模型调用）
  • 每个模型的 P99 延迟独立监控

常见误区：盲目增加 TP 数

常见问题（FAQ）

缓存命中率太低（< 10%）怎么办？
首先确认缓存粒度是否合适。补全场景由于光标位置的不同，精确命中率天然较低。建议：① 启用语义缓存，可以将命中率提升到 25-35%；② 预热常见文件（如核心工具类、配置类）的 KV Cache；③ 热点代码段（如 CRUD 模板）独立缓存。如果缓存仍然命中率低，重点优化模型推理而非缓存。
P99 比 P50 高 5 倍以上，可能是什么原因？
典型的「长尾延迟」问题。常见原因：① GPU 显存不足导致 swap；② 大请求阻塞小请求（vLLM 中可以用_chunked_prefill 缓解）；③ 缓存 miss 导致突发的模型推理；④ 网络抖动。建议用延迟分位数热力图定位异常时段，检查对应时段的资源使用情况。
vLLM 和前向推理对比，实际提升多少？
CorpAssist 测试数据：7B 模型单卡 A100，HuggingFace Transformers 吞吐 ~50 tokens/s，vLLM 启用 continuous batching 后 ~800 tokens/s（16x 提升）。延迟方面：TTFT 从 ~300ms 降到 ~80ms。关键参数：max_num_seqs=256, enable_prefix_caching=true。
不推荐
  • 只用平均延迟（P50）衡量性能，忽视长尾
  • 不做延迟预算，各环节不可观测
  • 没有缓存兜底，所有请求都走模型推理
  • 不分场景统一模型（补全用大模型也用小模型）
推荐
  • 同时追踪 P50/P95/P99 三个分位
  • 每环节有明确的延迟预算和监控
  • 三级缓存 + 语义缓存提升命中率
  • 小模型路由处理 80% 简单补全请求

本章结论

  • 三级缓存第一道防线：L1 精确 + L2 语义 + L3 预加载，总命中率约 35%。
  • 小模型路由降延迟：small model route 根据补全复杂度选择模型，80% 请求走 < 7B 模型。
  • vLLM 核心优化：前缀缓存、FP8 KV Cache、分块预填充三项关键配置。
  • P99 量化目标：补全 < 500ms，问答 < 5s，每环节有延迟预算。
  • 可观测性驱动调优：obs latency 体系覆盖全环节，定位长尾延迟根因。

检查理解：文件 hash 的作用是什么？

文件 hash 是服务端判断「是否需要从索引中读取该文件」的依据。如果 hash 命中服务端缓存，说明服务端已有该文件的最新索引，可以减少数据传输。如果 hash 不匹配，服务端只请求该文件的代码而非整个项目。hash 用 SHA256 足够，碰撞概率忽略不计。

本地/云端拆分：敏感代码不离开本机

local cloud split 根据代码敏感级别决定处理位置：通用代码 → 云端 API（更快、更强模型），敏感代码 → 本地推理（更安全）、高度机密 → 不上模型（只做符号检索）。

Java：插件端分级路由

java 
// CorpAssist IDE 插件路由控制器
public class IdeRoutingController {
    private final CloudApiClient cloudClient;
    private final LocalInferClient localClient;
    private final SensitivityAnalyzer sensitivityAnalyzer;

    public IdeRoutingController(CloudApiClient cloud,
                                LocalInferClient local,
                                SensitivityAnalyzer analyzer) {
        this.cloudClient = cloud;
        this.localClient = local;
        this.sensitivityAnalyzer = analyzer;
    }

    public CompletionResponse getCompletion(CompletionContext ctx) {
        SensitivityLevel level = sensitivityAnalyzer
            .analyze(ctx.currentFile());

        return switch (level) {
            case PUBLIC -> cloudClient.complete(ctx);      // 云 API
            case INTERNAL -> localClient.complete(ctx);     // 本地推理
            case RESTRICTED -> localClient.symbolOnly(ctx); // 仅符号补全
        };
    }

    public enum SensitivityLevel {
        PUBLIC,     // 开源项目、公共库 → 云 API
        INTERNAL,   // 内部业务代码 → 本地 vLLM
        RESTRICTED  // 支付/密码/密钥 → 仅符号补全
    }
}

// 敏感度分析器
class SensitivityAnalyzer {
    private final List restrictedPatterns = List.of(
        Pattern.compile(".*(payment|password|secret|auth).*"),
        Pattern.compile(".*(crypto|security|key).*")
    );

    public SensitivityLevel analyze(String filePath) {
        String lower = filePath.toLowerCase();
        boolean isRestricted = restrictedPatterns.stream()
            .anyMatch(p -> p.matcher(lower).matches());
        if (isRestricted) {
            return SensitivityLevel.RESTRICTED;
        }
        boolean isInternal = lower.startsWith("src/main/java/com/company");
        return isInternal
            ? SensitivityLevel.INTERNAL
            : SensitivityLevel.PUBLIC;
    }
}

on prem index 是企业私有化部署的索引方案。对于不能将代码传出机房的客户，CorpAssist 提供全本地部署：本地 vLLM + 本地 Milvus + 本地符号索引，数据完全不离开内网。

常见误区：私有部署就是全部本地

大多数企业不需要「全部本地」。正确方案是混合部署：通用代码（开源库、框架调用）走云端 API 获得最佳质量，敏感业务代码走本地小模型保证安全。折中方案是云端模型处理时数据不落盘（in-memory only），响应后即清除。

常见问题（FAQ）

开发者想要禁用数据收集功能，应该在插件中提供开关吗？
必须提供。插件设置中应有「隐私模式」开关：开启后，所有请求走本地处理（小模型 + 本地索引），不调用云端 API。代价是补全质量和问答能力下降，但数据完全不离开本机。这个开关是企业合规的必要条件。
VS Code 和 JetBrains 插件开发有什么不同？
核心 API 类似但实现细节差异大。VS Code 用 TypeScript/JavaScript 开发，提供丰富的 extension API（InlineCompletionItem、TextDocumentContentChangeEvent）。JetBrains 用 Kotlin/Java 开发，通过 PsiManager 和 Editor 接口交互。推荐先开发 VS Code 版本（开发效率高），再移植到 JetBrains。
插件更新频繁会影响开发者体验吗？怎么做好版本管理？
插件更新策略：① 后端 API 保持向后兼容（至少保留一个 major 版本）；② 插件采用多通道发布（stable/beta/nightly）；③ 重大变更通过「功能开关」逐步开放（先 5% 用户 → 50% → 100%）；④ 插件版本与服务端版本兼容性自动检测，不匹配时提示升级。
不推荐
  • 无隐私模式开关，所有数据强制上传
  • 插件无本地降级能力
  • 数据留存策略不透明

  • 离线降级到 LSP 本地补全
  • 数据 TTL 5 分钟后清除，可审计
  • 同时支持 VS Code 和 JetBrains

坑一：AST 检索导致的 Token 爆炸

token explosion 是最常见的生产事故。典型场景是：开发者在问答中提问「这个项目使用的所有技术栈」，索引服务返回了 50+ 个代码片段，加起来超过 20K tokens，直接撑爆上下文窗口。

CorpAssist · Token 爆炸现场 开发者提问「如何实现用户认证」，AST 检索返回了 AuthController.java（150 行）、AuthService.java（300 行）、JwtUtil.java（80 行）、UserRepository.java（200 行）、SecurityConfig.java（120 行）、TokenFilter.java（180 行）等 20 个代码块，总计约 15K tokens。LLM 的注意力被稀释，回答中引用了错误的方法名。

ast not whole repo 是核心原则：绝不要把整个仓库的代码塞入 prompt。解决方案：

问题根因解决方案
检索返回太多 chunk Top-K 设置过大（如 K=20） Top-K 限制为 3-5，按相关性裁剪
单个 chunk 太大 函数/类定义过长（> 200 行） 拆分子 chunk，只返回相关部分
无关文件被检索 Embedding 模型在代码上区分度不够 使用代码专用 Embedding + 关键词过滤
检索与对话历史叠加 多轮问答累积 chunks 每轮独立检索，不累积历史 chunks

Python：Token 爆炸检测器

python 
# CorpAssist Token 爆炸检测
class TokenExplosionDetector:
    def __init__(self, max_context: int = 32000):
        self.max_context = max_context

    def analyze_request(self, prompt_parts: dict) -> dict:
        """分析 prompt 各部分的 token 占比"""
        total = 0
        details = {}
        for layer, content in prompt_parts.items():
            tokens = self._count_tokens(content)
            details[layer] = {
                "tokens": tokens,
                "pct": 0  # 待计算
            }
            total += tokens

        for layer in details:
            details[layer]["pct"] = round(
                details[layer]["tokens"] / total * 100, 1
            )

        # 检测爆炸风险
        risks = []
        for layer, info in details.items():
            if info["pct"] > 40:
                risks.append(f"{layer} 占比 {info['pct']}% > 40%，建议裁剪")

        if total > self.max_context * 0.9:
            risks.append(f"总 tokens {total} 超过上限 90%，即将爆窗")

        return {
            "total_tokens": total,
            "max_tokens": self.max_context,
            "usage_pct": round(total / self.max_context * 100, 1),
            "details": details,
            "risks": risks,
            "is_overflow": total > self.max_context
        }

    def _count_tokens(self, text: str) -> int:
        return len(text) // 4  # 估算：约 4 字符/token
不推荐
  • Top-K=20，一次性注入过多代码块
  • 所有检索结果不裁剪直接拼接
  • 历史对话累积 chunks（一轮加 5 个，5 轮后 25 个）
  • 问答和补全场景使用相同的检索策略
推荐
  • Top-K=3-5，按相关性裁剪
  • 每轮检索独立，不累积
  • 问答场景 Top-K=5，补全场景 Top-K=2
  • 运行时检测 token 占比，触发裁剪

坑二：FIM 与私库微调的边界混淆

fim ft boundary 是指 FIM 补全和私有微调的边界选择。常见错误是团队直接使用私有代码微调通用模型，期望解决补全问题。但微调成本高、周期长、可能导致通用能力退化。正确路径是：先用 FIM + RAG 解决 90% 的问题，微调只用于特定场景的最后一公里。

方案适用场景成本效果
FIM + RAG 日常代码补全，90% 场景 低（无训练成本） 采纳率 30-40%
PEFT LoRA 微调 特定框架/DSL 的高频模式 中（数小时训练） 采纳率+10-15%
全量微调 代码风格非常独特的企业 高（数天训练） 采纳率+15-20%
常见误区：微调可以替代 FIM 格式学习

微调不会自动教会模型 FIM 格式。如果预训练模型不支持 FIM（如某些 Base 模型），微调后仍然不行。FIM 能力是预训练阶段决定的。选择模型时，优先选原生支持 FIM 的模型（CodeLLaMA、StarCoder、Qwen-Coder），而不是微调一个不支持 FIM 的模型。

检查理解：什么时候应该考虑微调而不是 FIM+RAG？

当出现以下情况时考虑微调：① 企业内部有大量的重复性代码模式（如统一的状态机框架、特定的 ORM 封装）；② FIM+RAG 的采纳率稳定在 30% 以下且其他优化手段无效；③ 有充足的高质量训练数据（至少 5000+ 条企业代码）。否则，优先优化 RAG 检索质量和上下文组装策略。

坑三：缺乏 LSP 验证闭环

lsp verify loop 是生成代码后的质量验证闭环。CorpAssist 的实践：每次补全生成后，将完整代码（prefix + middle + suffix）提交给 LSP 验证语法和类型错误。如果有编译错误，回退到上一版本或标记为「需要验证」。

LSP 验证闭环流程
sequenceDiagram
    participant Plugin as IDE 插件
    participant LSP as LSP Server
    participant AI as CorpAssist

    Plugin->>AI: 请求补全
    AI-->>Plugin: 返回 Middle 代码
    Plugin->>Plugin: 组装完整代码
    Plugin->>LSP: textDocument/diagnostic
    LSP-->>Plugin: 诊断结果
    alt 无错误
        Plugin->>Plugin: 展示补全建议
    else 有编译错误
        Plugin->>Plugin: 隐藏建议/标记警告
        Plugin->>AI: 错误反馈（用于优化）
    end

生成代码必须经过 LSP 验证才能展示给开发者。

verify compile 是验证的最终手段——对于 Java 项目，可以快速编译补全后的临时文件。如果编译失败，标记该建议为「未验证」。验证闭环的好处：补全建议的「编译通过率」从 60% 提升到 90%+。

不推荐
  • 补全后不做任何语法验证
  • 编译错误的建议仍展示给开发者
  • 不收集验证反馈用于优化
  • 忽略 LSP 对生成代码的诊断结果
推荐
  • 补全后自动 LSP 诊断验证
  • 编译错误的建议隐藏或标记
  • 错误类型和频率反馈给模型优化
  • LSP 验证在 50ms 内完成

坑四：工具调用与三步规划偏航

plan drift code 是指 AI 在多次工具调用后偏离原始目标。典型场景：AI 先检索代码 → 发现需要修改多个文件 → 读取每个文件 → 修改过程中忘记了最初的目的。tool loop code 是 AI 陷入无限工具调用的陷阱——不断检索查看更多代码但从不输出最终结果。

防止规划偏航的策略

常见问题（FAQ）

LSP 验证发现语法错误后，该直接丢弃建议还是提供给开发者？
建议标记为「可能有语法错误」并展示给开发者。原因：① LSP 可能误报（如未 import 类，但开发者知道需要 import）；② 即使有语法错误，代码逻辑可能是正确的，开发者能更快修正。但对于严重错误（如括号不匹配、非法语法），应该直接隐藏并提示开发者。
微调后模型的通用补全能力下降了，怎么办？
这是典型的「灾难性遗忘」。解决方案：① 使用 PEFT/LoRA 而非全量微调，冻结 Base 模型的大部分权重；② 微调数据中混入 20-30% 的通用代码数据；③ 微调后做对比测试——在通用 benchmark（如 HumanEval）上的得分下降不应超过 5%；④ 一个更激进的做法：保留 Base 模型，用 LoRA adapter 做路由，特定场景调用 adapter，通用场景调用 base。
Agent 调用工具时陷入死循环，怎么检测和恢复？
死循环的特征是工具调用模式重复（如一直检索同一个文件但从不输出）。检测：① 工具调用签名去重——如果最近 5 次调用中有 3 次签名相同 → 判为循环；② 输出 token 计数——如果连续 5 次工具调用后输出为 0 → 判为循环。恢复：① 强制输出当前已收集的信息；② 中断本次调用，向用户返回中间结果和错误提示。
不推荐
  • 无限制的工具调用次数
  • 不验证生成代码的语法正确性
  • 直接用私库全量微调通用模型
  • 陷入工具循环时不做中断
推荐
  • 最大工具调用 10 次 + 超时 60s
  • LSP 验证闭环保障语法正确性
  • FIM+RAG 优先，微调最后
  • 检测工具循环，强制中断并返回

本章结论

  • Token 爆炸可预防：限制 Top-K、每轮独立检索、运行时检测占比。
  • FIM 优先于微调：fim ft boundary 原则——用 FIM+RAG 解决 90% 问题。
  • 验证闭环不可少：lsp verify loop 将编译通过率提升到 90%+。
  • 防止工具偏航：限制调用次数，设置超时，检测循环模式。
  • 一切坑都是设计问题：在架构设计阶段就考虑这些边界条件，比事后修坑节省 10 倍时间。
