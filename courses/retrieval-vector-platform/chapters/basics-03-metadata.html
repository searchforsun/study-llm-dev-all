为什么要学本章

检索层 ACL 是数据安全底线——没有 metadata filter，HR 部门的机密制度文档可能被研发人员搜到。filter expression 必须在向量搜索前应用，而非在应用层做后过滤。

学完你能
  • 用 Milvus/ES filter 表达式实现部门级 ACL
  • 设计 multi tenant 隔离方案（collection-per-tenant vs shared+filter）
  • 配置 quota 限制单个租户的 QPS 和存储量
  • 避免「后过滤」导致的检索精度损失
本章先记住 3 件事
  • Filter 在向量搜索前生效：Milvus 是先 filter 再向量检索，不是搜完再过滤。这决定了 ACL 不会漏数据。
  • 标量索引加速 filter：对 tenant_id 等高频过滤字段建 inverted index，否则全量扫描拖慢搜索。
  • 共享 collection + filter 隔离优于 per-tenant collection：管理简单，一条 filter 就够了。

Filter 表达式：语法与执行模式

metadata filter 使用类似 SQL 的 filter expression 语法。Milvus 支持关系运算符（==, !=, >, <, >=, <=）、逻辑运算符（and, or, not）和范围运算符（in, like）。

CorpAssist · 制度文档 ACL 员工查询「年假政策」时，系统必须在向量搜索中就限定：tenant_id == 用户.tenant AND department == 用户.dept AND classification <= 用户.clearance。如果 filter 写错，制度文档就漏出去了。

Milvus Filter 表达式示例

Python：带 Filter 的向量搜索

python 
from pymilvus import Collection
import numpy as np

collection = Collection("corpassist_kb")
collection.load()

query_vec = np.random.rand(768).tolist()

# 带 ACL filter 的搜索
expr = 'tenant_id == 1 and dept in ["HR", "Finance"] and classification <= 2'
results = collection.search(
    data=[query_vec],
    anns_field="embedding",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=5,
    expr=expr,  # filter 在向量搜索前执行
    output_fields=["id", "chunk_text", "dept", "classification"],
)

for hits in results:
    for hit in hits:
        print(f"score={hit.score:.4f} dept={hit.entity.get('dept')} "
              f"class={hit.entity.get('classification')}")
collection.release()

Pre-filter 先缩小搜索范围再 ANN，保证结果数量。Post-filter 先搜全量再过滤，可能返回不足 K 条。

Java：Spring 中的 Filter 构建

java 
import io.milvus.v2.service.vector.request.SearchReq;
import io.milvus.v2.service.vector.request.data.FloatVec;

@Service
public class AclSearchService {
    private final MilvusClientV2 client;

    public SearchResp searchWithAcl(float[] queryVec, long tenantId,
            List depts, int maxClassification) {
        String expr = String.format(
            "tenant_id == %d and dept in %s and classification <= %d",
            tenantId, depts.toString(), maxClassification
        );
        return client.search(SearchReq.builder()
            .collectionName("corpassist_kb")
            .data(List.of(new FloatVec(queryVec)))
            .annsField("embedding")
            .topK(5)
            .expr(expr)
            .searchParams(Map.of("ef", 64, "metric_type", "COSINE"))
            .outputFields(List.of("id", "chunk_text", "dept"))
            .build());
    }
}
不推荐
  • 后过滤：搜完 top-100 再过滤，filter 后可能只剩 2 条
  • filter 字段不建标量索引，每次全量扫描
  • filter 表达式拼错语法，M ilvus 报错
推荐
  • Pre-filter：filter 在 ANN 前执行，保证结果数量
  • 高频过滤字段建 inverted index
  • 用参数化表达式构建 filter，避免拼接错误

多租户隔离方案设计

tenant isolation 是检索平台安全架构的核心。CorpAssist 采用 共享 Collection + Filter 模式，所有租户数据存储在同一 collection 中，通过 tenant_id 字段做 ACL。

标量索引加速 Filter

Milvus 支持为标量字段创建 inverted index。对 tenant_id 等高频过滤字段建索引后，filter 耗时从几十 ms 降至 1-2ms。

python 
from pymilvus import Collection, IndexType

collection = Collection("corpassist_kb")

# 为高频过滤字段建标量索引
collection.create_index("tenant_id", {
    "index_type": "INVERTED",
    "params": {}
})
collection.create_index("dept", {
    "index_type": "INVERTED",
    "params": {}
})
print("Scalar indexes created for tenant_id and dept")
先想 10 秒：如果 100 个租户，其中 1 个大租户占了 80% 数据，filter 会影响搜索延迟吗？

会。大租户 filter 后候选集占 80%，ANN 搜索范围大，延迟可能翻倍。这种情况下应考虑用 Partition-per-tenant 将大租户数据单独分区。

配额管理与资源限制

quota 机制防止单个租户消费过量资源。Milvus 支持在 Collection 和 Partition 级别配置配额。

CorpAssist 配额策略

配额项默认值超限处理
QPS per tenant100429 Too Many Requests
存储容量 per tenant10 GB拒绝写入
Concurrent queries20排队等待
Top-K 上限100截断至 100

Python：应用层配额检查

python 
import time
from collections import defaultdict

class TenantQuotaManager:
    def __init__(self, max_qps: int = 100):
        self.max_qps = max_qps
        self._counters = defaultdict(lambda: [0, 0.0])  # {tenant: [count, window_start]}

    def check_and_consume(self, tenant_id: int) -> bool:
        now = time.time()
        count, window_start = self._counters[tenant_id]
        if now - window_start > 1.0:  # new window
            self._counters[tenant_id] = [1, now]
            return True
        if count >= self.max_qps:
            return False
        self._counters[tenant_id][0] += 1
        return True

quota = TenantQuotaManager(max_qps=100)
if quota.check_and_consume(tenant_id=1):
    # execute search
    pass
else:
    raise Exception("429 Too Many Requests")

Java：Spring 配额拦截器

java 
import com.google.common.util.concurrent.RateLimiter;
import java.util.concurrent.ConcurrentHashMap;

@Component
public class QuotaInterceptor {
    private final ConcurrentHashMap limiters = new ConcurrentHashMap<>();

    public void checkQuota(long tenantId) {
        RateLimiter limiter = limiters.computeIfAbsent(tenantId,
            id -> RateLimiter.create(100.0)); // 100 QPS
        if (!limiter.tryAcquire()) {
            throw new RuntimeException("429 Too Many Requests - tenant " + tenantId);
        }
    }
}

常见问题（FAQ）

Pre-filter 会不会漏掉相关结果？
理论上会——如果 filter 后候选集太小，ANN 在这部分候选集中找 top-k，可能漏掉 filter 范围外的更相似结果。但这是业务需求决定的：ACL 过滤后的结果才是「应该看到的」。不是漏，是安全约束。
标量索引和向量索引有什么区别？
标量索引（inverted index）用于加速 filter 条件过滤，类似 Elasticsearch 的倒排索引。向量索引（HNSW/IVF）用于加速 ANN 搜索。两者在 Milvus 中独立管理，互不干扰。
Quota 超限时应该报错还是降级？
优先降级。CorpAssist 的做法是：返回缓存的热门结果，同时返回 429 的响应头。让调用方知道被限流但仍能获得部分服务。

本章结论

  • Pre-filter 先行：Filter 在向量搜索前执行，保证结果数量和 ACL 安全。
  • 标量索引加速：高频过滤字段建 inverted index，避免全量扫描。
  • 共享 + Filter 隔离：一条 filter 实现多租户隔离，管理复杂度最低。
  • Quota 保护：QPS、存储、并发三重配额防止单个租户影响全局。
  • ACL 一致性：Python 写入时标注 tenant/dept，Java 查询时自动注入 filter。
