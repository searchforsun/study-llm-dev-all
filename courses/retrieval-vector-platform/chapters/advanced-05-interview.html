典型根因分析与修复

embedding mismatch 是最常见的「低级错误」——查询用的 embedding 模型和索引构建时用的不一致。chunk issue 指 chunk 切分切断了关键短语或句子。

案例 1：Embedding 模型不一致

python 
# 问题：索引用 text-embedding-3-small，查询用 text-embedding-ada-002
# 两者向量空间不一致，cosine 相似度混乱

# 验证方法：打印两端使用的模型
def check_embedding_model():
    index_model = "text-embedding-3-small"   # 从索引 Job 配置读取
    query_model = "text-embedding-ada-002"   # 从 API 调用读取
    print(f"Index model: {index_model}")
    print(f"Query model: {query_model}")
    assert index_model == query_model, f"Model mismatch: {index_model} vs {query_model}"

# 修复：统一为 text-embedding-3-small
# 注意：模型切换后需要重建索引

案例 2：Filter 过严导致空结果

python 
# 问题：Filter 条件误用了 AND 而非 OR，导致无结果
def diagnose_filter(query, filter_expr):
    """排查 filter 是否过严"""
    # Step 1: 不带 filter 搜索
    results_no_filter = search(query, expr=None)
    print(f"Without filter: {len(results_no_filter)} results")

    # Step 2: 带 filter 搜索
    results_with_filter = search(query, expr=filter_expr)
    print(f"With filter '{filter_expr}': {len(results_with_filter)} results")

    # Step 3: 如果带 filter 后显著减少，说明 filter 过严
    if len(results_with_filter) < len(results_no_filter) * 0.3:
        print("[WARN] Filter may be too strict!")
        print("  Suggestions: check AND/OR logic, verify tenant_id")

# 使用
diagnose_filter("年假政策", 'department == "HR" AND classification == 2')
# Output:
# Without filter: 15 results
# With filter: 2 results
# [WARN] Filter may be too strict!
CorpAssist · 真实排障故事 某天突然发现「年假政策」搜索不到任何结果。排查链路：① 确认 embedding 模型一致 → ② query 单条验证发现数据存在 → ③ 打印 chunk 发现「年假」被切成了「年」和「假」两个 chunk（chunk_size=256 导致）→ ④ 改为 chunk_size=512 → 恢复。面试时讲清楚这个「chunk 切断语义」的故事就是加分项。

Java：排查工具类

java 
import io.milvus.v2.service.vector.request.SearchReq;

@Service
public class RecallDiagnosisService {
    private final MilvusClientV2 client;

    // 诊断：比较带/不带 filter 的结果数
    public void diagnoseFilter(String query, float[] queryVec, String expr) {
        // 带 filter
        var withFilter = client.search(SearchReq.builder()
            .collectionName("corpassist_kb")
            .data(List.of(new FloatVec(queryVec)))
            .annsField("embedding")
            .topK(10)
            .expr(expr)
            .build());
        // 不带 filter
        var withoutFilter = client.search(SearchReq.builder()
            .collectionName("corpassist_kb")
            .data(List.of(new FloatVec(queryVec)))
            .annsField("embedding")
            .topK(10)
            .build());
        System.out.printf("Without filter: %d hits, With filter: %d hits%n",
            withoutFilter.getSearchResults().get(0).size(),
            withFilter.getSearchResults().get(0).size());
    }
}

模拟面试 Q&A

「RAG 召回低，请说排查思路」
建议回答：「我会从最外层开始排查。第一，确认 query 和索引用的 embedding 模型一致；第二，确认文档已写入且未被误删；第三，检查 chunk 切分是否切断关键语义；第四，排除 filter 过严；第五，检查 ef 参数是否太低；第六，确认索引版本没有过期。每一步都有对应验证方法，比如不带 filter 对比搜索。通常前三个点就能覆盖 60% 的问题。」
「如果 embedding 一致但召回还是低，下一步查什么？」
建议回答：「查 chunk 切分。打印几条召回结果和未召回结果的 chunk 内容，人工审查。常见问题：chunk_size 太小切断了关键句、overlap 不够导致上下文断裂。改进后重新索引再验证。」
「业务方说昨天还好好的，今天突然不行了，怎么办？」
建议回答：「优先看变更历史：昨天到今天有没有部署新版本、改 filter 配置、切换 embedding 模型、或重建索引。如果有，先回滚。同时查监控看 p95 延迟和空结果率是否突变。大概率是配置变更或索引版本问题。」

本章结论

  • 排查框架八步：从 embedding 到 Top-K 逐层排查。
  • Embedding 不一致排第一：最常见低级错误。
  • Chunk 切断语义排第二：chunk_size 和 overlap 直接影响。
  • Filter 过严排第三：不带 filter 对比验证。
  • 面试答法要结构化：STAR 框架 + 检查清单。
