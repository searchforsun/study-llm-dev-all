Spring 搜索指标埋点

通过 Micrometer 在 Spring 服务中埋点，按 tenant 和 collection 打标签。

Java：Micrometer 搜索指标

java 
import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Timer;
import io.micrometer.core.instrument.Counter;
import java.util.concurrent.TimeUnit;

@Component
public class SearchMetrics {
    private final MeterRegistry registry;

    public SearchMetrics(MeterRegistry registry) {
        this.registry = registry;
    }

    public void recordSearch(long tenantId, String collection, long latencyMs, boolean success, int resultCount) {
        // 延迟直方图
        Timer.Sample sample = Timer.start(registry);
        Timer timer = Timer.builder("retrieval.search.latency")
            .tag("tenant", String.valueOf(tenantId))
            .tag("collection", collection)
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
        sample.stop(timer);

        // 搜索计数
        Counter.builder("retrieval.search.total")
            .tag("tenant", String.valueOf(tenantId))
            .tag("collection", collection)
            .register(registry).increment();

        // 空结果计数
        if (resultCount == 0) {
            Counter.builder("retrieval.search.empty")
                .tag("tenant", String.valueOf(tenantId))
                .register(registry).increment();
        }

        // 错误计数
        if (!success) {
            Counter.builder("retrieval.search.error")
                .tag("tenant", String.valueOf(tenantId))
                .register(registry).increment();
        }
    }
}

Python：FastAPI 搜索指标导出

python 
from prometheus_client import Histogram, Counter, generate_latest
from fastapi import FastAPI, Request
from fastapi.responses import Response
import time

app = FastAPI()

SEARCH_LATENCY = Histogram(
    "retrieval_search_latency_seconds",
    "Search latency in seconds",
    ["tenant", "collection"],
    buckets=(0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.0)
)

SEARCH_TOTAL = Counter(
    "retrieval_search_total",
    "Total search count",
    ["tenant", "collection", "status"]  # status: success/empty/error
)

@app.post("/search")
async def search(request: Request):
    data = await request.json()
    tenant = str(data.get("tenant_id", "unknown"))
    start = time.time()

    try:
        results = do_search(data)
        elapsed = time.time() - start
        SEARCH_LATENCY.labels(tenant=tenant, collection="corpassist_kb").observe(elapsed)
        status = "empty" if len(results) == 0 else "success"
        SEARCH_TOTAL.labels(tenant=tenant, collection="corpassist_kb", status=status).inc()
        return {"results": results}
    except Exception:
        elapsed = time.time() - start
        SEARCH_LATENCY.labels(tenant=tenant, collection="corpassist_kb").observe(elapsed)
        SEARCH_TOTAL.labels(tenant=tenant, collection="corpassist_kb", status="error").inc()
        raise

@app.get("/metrics")
async def metrics():
    return Response(content=generate_latest(), media_type="text/plain")

告警规则与异常检测

retrieval alert 在指标超 SLO 时触发。CorpAssist 使用 Prometheus AlertManager 配置告警规则。

Prometheus 告警规则

yaml 
groups:
  - name: retrieval_alerts
    rules:
      - alert: HighSearchLatency
        expr: histogram_quantile(0.95, rate(retrieval_search_latency_seconds_bucket[5m])) > 0.2
        for: 5m
        labels: { severity: warning }
        annotations:
          summary: "搜索 P95 延迟超过 200ms"

      - alert: HighEmptyResultRate
        expr: rate(retrieval_search_total{status="empty"}[5m]) / rate(retrieval_search_total[5m]) > 0.05
        for: 10m
        labels: { severity: critical }
        annotations:
          summary: "空结果率超过 5%"

      - alert: HighErrorRate
        expr: rate(retrieval_search_total{status="error"}[5m]) / rate(retrieval_search_total[5m]) > 0.001
        for: 5m
        labels: { severity: critical }
        annotations:
          summary: "搜索错误率超过 0.1%"
不推荐
  • 只监控 avg latency（掩盖长尾）
  • 不按 tenant 拆分指标
  • 无空结果率监控
  • 告警阈值拍脑袋不调优
推荐
  • 监控 p50/p95/p99 全部三位延迟
  • 按 tenant、collection 打标签
  • 空结果率 + 错误率 + 延迟三合一
  • 根据历史基线动态调优阈值

常见问题（FAQ）

Grafana 看板应该展示哪些核心图表？
至少四个面板：1) 实时搜索延迟 P50/P95/P99 折线图；2) 空结果率趋势；3) QPS 按 tenant 堆叠图；4) 错误率。可选：索引数据量增长（文档数/向量数）。
空结果率高一定是索引问题吗？
不一定是。空结果率高可能原因：filter 过严、embedding 模型不匹配、chunk 切分导致语义丢失、或确实没有相关文档。需要通过日志分析分布。
P95 飙升但 P50 正常，说明什么？
说明有少量「慢查询」在拖后腿。可能原因：某些租户数据量大导致 filter 后候选集过大、某些查询词向量在所有簇中都接近边界、或者节点 GC 抖动。需要按 tenant 拆分排查。

本章结论

  • 三大黄金指标：延迟（P95）、空结果率、错误率。
  • 按 tenant 拆指标：隔离问题域，快速定位问题租户。
  • Micrometer 埋点：Spring 应用内埋点 + Prometheus 采集。
  • SLO 驱动告警：超阈值后自动通知响应。
  • Grafana 看板：四面板覆盖检索核心健康状态。
