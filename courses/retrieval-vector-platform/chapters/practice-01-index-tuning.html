为什么要学本章

默认 HNSW 参数（M=16, efConstruction=200）在大规模制度库上 recall 可能不足。增大 ef 和 M 能提升召回但增加延迟——latency tradeoff 是索引工程的核心决策。grid search 是找到最优组合的科学方法。

学完你能
  • 设计 param tuning 的 grid search 流程
  • 用固定的 query set 做 index benchmark
  • 输出 recall@k 与 p95 延迟的 tradeoff 表格
  • 根据 cost per qps 选择最优参数
本章先记住 3 件事
  • 没有最优参数，只有最优 tradeoff：recall 提高 2% 可能让延迟翻倍，场景决定目标。
  • Benchmark 用固定 query set：100 条代表性查询，跑 3 次取均值。
  • ef 是搜索参数，M 是构建参数：ef 可动态调，M 一旦建索引就固定。

HNSW 参数详解：M、efConstruction、ef_search

ef construction 控制构建时的搜索宽度，M 控制每个节点最大连接数，ef_search 是查询时的动态宽度参数。

参数含义典型范围对 recall 影响对延迟影响
M节点最大邻居数8-64大构建变慢，搜索略增
efConstruction构建时搜索宽度100-500中构建时间线性增长
ef_search查询时动态宽度16-512大查询延迟线性增长

Python：Benchmark 脚本

python 
from pymilvus import Collection
import numpy as np
import time

def benchmark_params(collection_name, query_vectors, gt_ids, ef_values, top_k=10):
    collection = Collection(collection_name)
    collection.load()
    results = []

    for ef in ef_values:
        latencies = []
        recall_scores = []

        for q_vec, gt in zip(query_vectors, gt_ids):
            start = time.perf_counter()
            hits = collection.search(
                data=[q_vec], anns_field="embedding",
                param={"metric_type": "COSINE", "params": {"ef": ef}},
                limit=top_k
            )
            elapsed = time.perf_counter() - start
            latencies.append(elapsed * 1000)  # ms

            ann_ids = [h.id for h in hits[0]]
            hits_count = sum(1 for aid in ann_ids if aid in set(gt[:top_k]))
            recall_scores.append(hits_count / top_k)

        results.append({
            "ef": ef,
            "avg_recall": np.mean(recall_scores),
            "p50_ms": np.percentile(latencies, 50),
            "p95_ms": np.percentile(latencies, 95),
            "p99_ms": np.percentile(latencies, 99),
        })
        print(f"ef={ef}: recall@10={results[-1]['avg_recall']:.3f}, "
              f"p95={results[-1]['p95_ms']:.1f}ms")

    collection.release()
    return results

# 使用
query_vecs = [np.random.rand(768).tolist() for _ in range(100)]
gt_ids = [list(range(20)) for _ in range(100)]  # ground truth
results = benchmark_params("corpassist_kb", query_vecs, gt_ids, [16, 32, 64, 128, 256])

Java：Benchmark 调用器

java 
import io.milvus.v2.service.vector.request.SearchReq;
import java.util.*;

@Service
public class BenchmarkRunner {
    private final MilvusClientV2 client;

    public Map benchmarkEf(List queries, int[] efValues) {
        Map result = new LinkedHashMap<>();
        for (int ef : efValues) {
            List latenciesMs = new ArrayList<>();
            for (float[] q : queries) {
                long start = System.nanoTime();
                client.search(SearchReq.builder()
                    .collectionName("corpassist_kb")
                    .data(List.of(new io.milvus.v2.service.vector.request.data.FloatVec(q)))
                    .annsField("embedding")
                    .topK(10)
                    .searchParams(Map.of("ef", ef, "metric_type", "COSINE"))
                    .build());
                long elapsed = (System.nanoTime() - start) / 1_000_000;
                latenciesMs.add(elapsed);
            }
            Collections.sort(latenciesMs);
            long p95 = latenciesMs.get((int)(latenciesMs.size() * 0.95));
            result.put("ef_" + ef, Map.of("p95_ms", p95, "samples", latenciesMs.size()));
        }
        return result;
    }
}

Benchmark 方法论与评估指标

recall at k 是最常用的评估指标。但仅看 recall 不够——还需结合 precision、延迟和索引构建成本。

评估矩阵

指标计算公式目标值（CorpAssist）
Recall@10ANN top-10 中 ground-truth 占比> 0.95
P95 Latency95% 查询耗时< 100ms
Index Build Time建索引耗时< 30min（500万向量）
Memory Usage索引内存占用< 16GB
不推荐
  • 只测一次，取单次延迟
  • 用训练集 query 当评估集
  • 只看 avg latency 不看 p95
  • 不同参数组合在不同数据集上比
推荐
  • 跑 3 次取均值，预热后开始记录
  • 独立的评估 query set
  • 报告 p50/p95/p99 三个分位
  • 固定数据集，只变一个参数

成本与权衡决策

cost per qps 是工程决策的最终指标：在满足 recall SLO 的前提下，最小化单位 QPS 成本。

场景recall 目标延迟 SLO推荐参数
FAQ 精确问答> 0.98< 50msef=128, M=16
文档检索> 0.95< 100msef=64, M=16
批量处理> 0.90< 500msef=32, M=8
海量检索> 0.97< 200msef=256, M=32

常见问题（FAQ）

efConstruction 设越大越好吗？
不是。efConstruction 越大索引质量越高，但构建时间和内存占用也越大。超过 300 后 recall 提升趋于收敛。推荐 100-200，大规模可提升到 300。
M=32 比 M=16 好在哪？代价是什么？
M 越大，图连接越密，recall 越高，但索引内存占用约翻倍（M=16 时每节点存 16 个邻居指针）。搜索延迟略有增加。通常 M=16 是平衡点。
如何攒 ground-truth 数据集？
暴力搜索法：用 brute-force（flat index）在全部数据上计算真 top-k。小数据集（<10万条）可做，大数据集用代表性样本。CorpAssist 从各 tenant 随机抽 100 条 query。

本章结论

  • 参数是 tradeoff：ef 和 M 提升 recall 但增加延迟和内存。
  • Grid Search 找最优组合：固定 query set，遍历参数，输出对比表。
  • 报告多分位延迟：avg 瞒人，p95 + p99 暴露长尾。
  • 场景决定目标：FAQ > 0.98 recall，文档检索 > 0.95。
  • Benchmark 需标准化：固定数据集、预热、多次取均值。
