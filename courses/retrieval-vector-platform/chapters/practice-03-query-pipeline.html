检索管线 DAG 设计

pipeline DAG 将检索流程定义为有向无环图。CorpAssist 的标准管线包含四阶段：Query Preprocess → Multi-Recall → Fusion → Rerank。

检索管线 DAG 完整流程
flowchart TB
    Q["原始查询"] --> REWRITE["Query Rewrite\nHyDE / MultiQuery"]
    REWRITE --> PAR["并行分支"]
    PAR --> VEC["向量检索 Milvus\nHNSW top-50"]
    PAR --> BM["关键词检索 ES\nBM25 top-50"]
    PAR --> RULE["规则匹配\n知识库标签"]
    VEC --> FUSE["RRF Fusion"]
    BM --> FUSE
    RULE --> FUSE
    FUSE --> RERANK["Cross-encoder Rerank\ntop-50 → top-5"]
    RERANK --> OUT["最终结果"]

四阶段 DAG：Rewrite → Parallel Recall → RRF Fusion → Rerank。

Query Rewrite：提升召回覆盖率

query rewrite 在检索前改写用户查询，提升与文档的语义匹配。CorpAssist 用两种策略：HyDE（生成假设文档）和 MultiQuery（生成多个查询变体）。

Python：MultiQuery 改写

python 
from openai import OpenAI

client = OpenAI()

def multi_query_rewrite(original_query: str, n: int = 3) -> list[str]:
    """生成 n 个查询变体"""
    prompt = f"""你是一个查询改写专家。给定用户原始查询，生成 {n} 个不同的搜索查询版本，
覆盖同义替换、细节补充和不同表述方式。

原始查询：{original_query}

要求：
1. 每个查询保持原始意图
2. 覆盖不同的表达方式
3. 保持简洁

返回每行一个查询："""
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3,
    )
    queries = resp.choices[0].message.content.strip().split("\n")
    return [q.strip() for q in queries if q.strip()]

# CorpAssist 场景
original = "年假怎么算"
variants = multi_query_rewrite(original, n=3)
print(f"Original: {original}")
for v in variants:
    print(f"  -> {v}")
# -> "年假天数计算规则"
# -> "年假政策是什么"
# -> "如何计算年假"

Java：Spring Query Rewrite 调用

java 
import org.springframework.web.client.RestTemplate;

@Service
public class QueryRewriteService {
    private final RestTemplate rest;
    private final String rewriteUrl = "http://llm-service:8000/rewrite";

    public List rewrite(String query, int n) {
        Map req = Map.of("query", query, "n_queries", n);
        var response = rest.postForObject(rewriteUrl, req, List.class);
        return (List) response;
    }
}
先想 10 秒：MultiQuery 改写后是否每次都要搜多次？延迟怎么控制？

是的。改写出 3 个变体就搜 3 次，然后合并结果。CorpAssist 的做法是：改写后并行搜索，总延迟和最慢的一次持平。3 路并行 ≈ 单路延迟 + 改写延迟（~100ms）。

Branch-Merge 模式与管线实现

branch merge 是多路召回的核心模式：多个 recall path 并行执行，结果在 merge 节点融合。

召回路径引擎Top-K场景
向量语义路Milvus50语义相似文档
关键词路ES BM2550精确匹配文档
标签规则路知识库 tag 索引10热门置顶文档

Python：管线引擎实现

python 
from concurrent.futures import ThreadPoolExecutor, as_completed
from typing import List, Dict, Callable

class RetrievalPipeline:
    def __init__(self):
        self.stages = []

    def add_stage(self, name: str, fn: Callable, parallel: bool = False):
        self.stages.append({"name": name, "fn": fn, "parallel": parallel})

    def execute(self, query: str) -> Dict:
        results = {}
        for stage in self.stages:
            if stage["parallel"]:
                # 并行执行
                with ThreadPoolExecutor() as exe:
                    futures = {exe.submit(s["fn"], query): s["name"]
                               for s in stage["parallel_branches"]}
                    for f in as_completed(futures):
                        results[futures[f]] = f.result()
            else:
                results[stage["name"]] = stage["fn"](query)
        return results

# 使用
pipeline = RetrievalPipeline()
pipeline.add_stage("rewrite", lambda q: multi_query_rewrite(q, n=3))
pipeline.add_stage("vector_search", lambda q: milvus_search(q, top_k=50))
pipeline.add_stage("bm25_search", lambda q: es_search(q, top_k=50))

常见问题（FAQ）

多路召回中各路的 Top-K 怎么设？
各路 Top-K 相加应远大于最终需要的数量。CorpAssist 通常设各路 50，融合后共 100-150 条，Rerank 后取 5 条。这样保证召回率而精排筛选。
什么时候需要加规则召回路？
当业务有强规则匹配需求时。例如「置顶政策」必须出现在结果中，「最新政策」按时间优先。规则路用标签或元数据过滤实现。
管线执行失败应该整条重试还是部分重试？
部分重试。某一路召回超时不影响其他路——降级为该路返回空集，继续融合。CorpAssist 给每路设独立超时（默认 200ms）。

本章结论

  • DAG 管线编排：Rewrite → Multi-Recall → Fusion → Rerank 四阶段。
  • Query Rewrite 提升召回：MultiQuery 和 HyDE 改写后召回 +15%。
  • 多路并行召回：向量+BM25+规则三路并行，覆盖更多维度。
  • RRF 融合：不依赖分数的融合方法，天然去重。
  • 降级是设计的一部分：每路独立超时，一路失败不影响整体。
