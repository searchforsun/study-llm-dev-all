Qdrant 实战

Qdrant 的 API 设计简洁，RESTful + gRPC 双接口。Payload（元数据）索引是其特色功能。

Python：Qdrant CRUD

python 
from qdrant_client import QdrantClient, models

client = QdrantClient(host="localhost", port=6333)

# 创建 Collection（类似 Milvus 的 schema）
client.create_collection(
    collection_name="corpassist_kb",
    vectors_config=models.VectorParams(
        size=768,
        distance=models.Distance.COSINE,
    ),
)

# 写入向量 + payload（元数据）
client.upsert(
    collection_name="corpassist_kb",
    points=[
        models.PointStruct(
            id=i,
            vector=np.random.rand(768).tolist(),
            payload={"chunk_text": f"chunk_{i}", "source": "wiki", "tenant_id": 1},
        )
        for i in range(100)
    ],
)

# 搜索 + filter
results = client.search(
    collection_name="corpassist_kb",
    query_vector=np.random.rand(768).tolist(),
    query_filter=models.Filter(
        must=[
            models.FieldCondition(
                key="tenant_id",
                match=models.MatchValue(value=1),
            ),
        ]
    ),
    limit=5,
)
for r in results:
    print(f"score={r.score:.4f} text={r.payload['chunk_text'][:50]}")

Java：Qdrant Spring 集成

java 
import io.qdrant.client.QdrantClient;
import io.qdrant.client.QdrantGrpcClient;
import io.qdrant.client.grpc.Points;

@Service
public class QdrantSearchService {
    private final QdrantClient client;

    public QdrantSearchService() {
        this.client = new QdrantClient(
            QdrantGrpcClient.newBuilder("localhost", 6334, false).build());
    }

    public List search(float[] queryVector, long tenantId) {
        var result = client.searchAsync(Points.SearchPoints.newBuilder()
            .setCollectionName("corpassist_kb")
            .addAllVector(floatArrayToList(queryVector))
            .setLimit(5)
            .setFilter(Points.Filter.newBuilder()
                .addMust(Points.Condition.newBuilder()
                    .setField("tenant_id")
                    .setMatch(Points.Match.newBuilder().setInteger(tenantId))
                    .build())
                .build())
            .build()).join();
        return result;
    }
}

常见问题（FAQ）

什么时候应该从 Milvus 迁移到 Qdrant？
当团队规模小、运维能力有限、数据量千万级以内时。Qdrant 单二进制部署免运维。但如果已有 Milvus Cluster 基础设施，通常不迁移。
数据迁移时向量需要重新计算吗？
不需要。向量本身和具体数据库无关。导出向量 + payload（JSON格式），目标库重新创建 collection 后批量写入。注意度量类型（cosine/IP/L2）和维度必须一致。
Spring AI 的 VectorStore 可以屏蔽厂商差异吗？
Spring AI VectorStore 提供了 Milvus、Qdrant、Chroma 等多种实现，API 一致。但高级功能（filter 语法、索引参数）仍需厂商 SDK。推荐：通用 CRUD 用 Spring AI，高级能力用厂商 SDK。

本章结论

  • 选型三要素：数据规模、运维能力、Spring 集成需求。
  • Milvus 是大规模标准：分布式架构，功能丰富。
  • Qdrant 是轻量替代：单二进制部署，Rust 高性能。
  • Chroma 只用于开发：嵌入式本地测试。
  • 抽象层防锁定：Repository 接口隔离开关。

分片策略：水平扩展的核心

sharding 将数据水平切分到多个物理分片。Milvus 支持自动分片（基于 partition key）和手动分片。

按 tenant_id 哈希tenant_id均匀分布，天然隔离大租户热分片
按 dept 范围dept同一部门数据局部性部门间数据不均

Python：按 tenant 分片写入

python 
from pymilvus import Collection, FieldSchema, CollectionSchema, DataType

# 分片 schema：用 tenant_id 做 partition_key
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=False),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="tenant_id", dtype=DataType.INT64, is_partition_key=True),
    FieldSchema(name="chunk_text", dtype=DataType.VARCHAR, max_length=65535),
]
schema = CollectionSchema(fields)
collection = Collection("corpassist_kb", schema, num_shards=4)  # 4 分片

# 写入时自动按 tenant_id 路由到对应分片
collection.insert([
    [1, 2, 3],
    [vec1, vec2, vec3],
    [1, 1, 2],  # tenant_id: 1, 1, 2
    ["text1", "text2", "text3"],
])

import io.milvus.v2.service.vector.request.SearchReq;

@Service
public class TenantAwareSearchService {
    private final MilvusClientV2 client;

    public SearchResp searchByTenant(float[] queryVector, long tenantId) {
        return client.search(SearchReq.builder()
            .collectionName("corpassist_kb")
            .data(List.of(new FloatVec(queryVector)))
            .annsField("embedding")
            .topK(5)

            .build());
    }
}

副本配置与读扩展

replica 是 Milvus 查询节点的数据副本。多副本可横向扩展查询吞吐。

配置读副本

python 
from pymilvus import Collection, ResourceGroup

# 创建资源组
rg = ResourceGroup("search_rg", config={"requests": {"node": {"replicas": 2}}})
# 将 collection 的指定副本部署到此资源组
collection = Collection("corpassist_kb")
collection.load(replica_number=2, resource_groups=["search_rg"])
print(f"Collection loaded with 2 replicas in search_rg")
先想 10 秒：2 副本能让 QPS 翻倍吗？

理论上接近 2 倍，但受限于最慢的分片副本。如果某个分片的热点数据集中，副本增加可能达不到线性扩展。实际应用中 2 副本通常能提升 1.5-1.8x QPS。

冷热数据分离

hot cold 架构将热数据（近期高频访问）放在高性能 SSD，冷数据（归档历史）放在 HDD 或对象存储。

层级存储介质数据范围查询频率延迟要求
热数据（Hot）NVMe SSD最近 90 天高< 50ms
温数据（Warm）SSD90-365 天中< 200ms
冷数据（Cold）HDD / 对象存储> 1 年低< 2s

容量估算（500 万向量）

  • 每向量 768 维 float32 = 3KB
  • 500 万向量 ≈ 15GB 向量数据
  • HNSW 索引 ≈ 原始数据大小 × 1.5 ≈ 22GB
  • 元数据（chunk_text, source 等）≈ 10GB
  • 总计 ≈ 50GB（4 分片 × 12.5GB）
不推荐
  • 所有数据存在同一类存储
  • 不估算容量直接上线
  • 分片数过大（如 64 片）
  • 副本数过多（如 10 副本）
推荐
  • 冷热分层降成本
  • 按增长速率估算容量
  • 分片数建议 2-8
  • 副本 2-3 份足够

常见问题（FAQ）

分片数设多少合适？
建议分片数 = 预估节点数 × 2。500 万向量 4 节点时设 8 分片。分片太少每个分片数据量大，分片太多跨分片查询开销大。Milvus 建议每分片 100-500 万向量。
副本数增加会影响写入吗？
写入只需写入主分片，副本异步拉取——写延迟不受影响。但副本越多，数据同步延迟稍大。副本主要用于读扩展和高可用。
冷数据怎么归档？
Milvus 没有内置冷热分层，需要应用层实现：创建 hot/cold 两个 collection，按时间迁移数据。查询时先搜 hot，不足时补搜 cold。或直接用对象存储做归档。

本章结论

  • 分片水平扩展：按 tenant 哈希分片，查询路由到单分片。
  • 副本读扩展：2-3 副本提升查询吞吐。
  • 冷热分层降成本：热 SSD，冷 HDD/归档。
  • 容量估算先行：500 万向量约 50GB，4 分片 × 12.5GB。
  • 分片数不宜过多：2-8 片足够，太多跨分片开销。

实验矩阵与假设验证

experiment matrix 系统性地组合变量。先用 hypothesis test 提出假设，再设计实验验证。

Python：实验矩阵运行器

python 
import itertools
import json

class RetrievalExperiment:
    def __init__(self, query_set, ground_truth):
        self.query_set = query_set
        self.ground_truth = ground_truth
        self.results = []

    def run(self, params: dict):
        """运行单组参数"""
        # 模拟检索并计算指标
        # 真实实现调用 Milvus/ES
        recall, precision, latency = self._execute(params)
        self.results.append({**params, "recall@10": recall, "p95_ms": latency})
        return self.results[-1]

    def grid_search(self, param_grid: dict):
        """遍历参数组合"""
        keys = param_grid.keys()
        for values in itertools.product(*param_grid.values()):
            params = dict(zip(keys, values))
            self.run(params)

    def report(self) -> str:
        """生成实验报告"""
        lines = ["## 检索质量实验报告", "", "| " + " | ".join(self.results[0].keys()) + " |"]
        lines.append("|" + "---|" * len(self.results[0].keys()))
        for r in self.results:
            row = " | ".join(f"{v:.4f}" if isinstance(v, float) else str(v) for v in r.values())
            lines.append(f"| {row} |")
        return "\n".join(lines)

# 使用
exp = RetrievalExperiment(query_set=range(100), ground_truth={})
exp.grid_search({"ef": [32, 64, 128], "rewrite": [True, False]})
print(exp.report())

常见问题（FAQ）

消融实验和 A/B 测试有什么区别？
消融实验是离线评估：用固定数据集评估检索质量。A/B 测试是在线评估：真实用户流量分桶对比。消融实验快速验证假设，A/B 测试验证实际效果。两者互补。
评估集应该多大？
至少 100 条代表性查询。太少统计波动大，太多人工标注成本高。CorpAssist 从各租户按比例抽样 200-500 条。
实验发现 ef=256 提升很小但延迟翻倍，怎么决策？
看 SLO。如果 135ms 仍在 SLO 内且提升 2% recall 有价值，就选 ef=256。否则选 ef=128。关键是用业务场景决策。

本章结论

  • 消融实验科学归因：一次只变一个变量。
  • 假设先行再验证：先写清楚猜测再设计实验。
  • 实验矩阵系统化：Grid Search 覆盖参数组合。
  • 报告要标准化：背景、矩阵、结果、结论四部分。
  • 结论要有推荐：数字 + 业务解读 = 可执行。

import time

class DegradeLevel(Enum):
    NORMAL = 0
    VECTOR_DEGRADED = 1
    KEYWORD_ONLY = 2
    CACHE_ONLY = 3

class DegradeRouter:
    def __init__(self, milvus_threshold_ms=500):
        self.level = DegradeLevel.NORMAL
        self.milvus_threshold = milvus_threshold_ms
        self.last_check = 0

    def check_health(self):
        """定期检查向量库健康状态"""
        # 模拟检查：调用 Milvus 健康端点
        # 实际中通过 Prometheus 指标判定
        milvus_ok = self._check_milvus()

        if not milvus_ok:
            self.level = DegradeLevel.KEYWORD_ONLY
            return
        # 检查延迟
        latency = self._measure_latency()
        if latency > self.milvus_threshold:
            self.level = DegradeLevel.VECTOR_DEGRADED
        else:
            self.level = DegradeLevel.NORMAL

        self.check_health()

            return self._hybrid_search(query, query_vec)
        elif self.level == DegradeLevel.VECTOR_DEGRADED:
            return self._bm25_search(query)
        elif self.level == DegradeLevel.KEYWORD_ONLY:
            return self._bm25_search(query)
        else:
            return self._cache_search(query)

    def _check_milvus(self):
        # 模拟 health check
        return True

    def _measure_latency(self):
        # 模拟延迟测量
        return 100

Java：Spring 熔断+降级

java 
import io.github.resilience4j.circuitbreaker.CircuitBreaker;
import io.github.resilience4j.circuitbreaker.CircuitBreakerConfig;
import io.github.resilience4j.circuitbreaker.CircuitBreakerRegistry;
import java.time.Duration;

@Service
public class DegradeSearchService {
    private final MilvusClientV2 milvusClient;
    private final ElasticsearchClient esClient;
    private final CircuitBreaker milvusBreaker;

    public DegradeSearchService() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            .failureRateThreshold(50)   // 50% 失败率触发熔断
            .waitDurationInOpenState(Duration.ofSeconds(30))
            .slidingWindowSize(10)
            .build();
        this.milvusBreaker = CircuitBreakerRegistry.of(config)
            .circuitBreaker("milvus");
    }

    public List search(String query, float[] queryVec) {
        // 使用熔断器调用 Milvus
        try {
            return milvusBreaker.executeSupplier(() -> {
                var resp = milvusClient.search(SearchReq.builder()
                    .collectionName("corpassist_kb")
                    .data(List.of(new FloatVec(queryVec)))
                    .annsField("embedding")
                    .topK(5)
                    .build());
                return extractIds(resp);
            });
        } catch (Exception e) {
            // 熔断或异常时降级到 BM25
            return keywordFallback(query);
        }
    }

    private List keywordFallback(String query) {
        // 调用 ES BM25
        var resp = esClient.search(s -> s
            .index("corpassist_index")
            .query(q -> q.match(t -> t.field("chunk_text").query(query)))
            .size(5),
            Map.class
        );
        return extractEsIds(resp);
    }
}

Resilience4j 默认 30 秒后转入 HALF_OPEN 状态，允许一个探测请求。成功则关闭熔断器，失败则继续保持 OPEN。CorpAssist 设置 60 秒等待时长。

# 问题：Filter 条件误用了 AND 而非 OR，导致无结果
def diagnose_filter(query, filter_expr):
    """排查 filter 是否过严"""
    # Step 1: 不带 filter 搜索
    results_no_filter = search(query, expr=None)
    print(f"Without filter: {len(results_no_filter)} results")

    # Step 2: 带 filter 搜索
    results_with_filter = search(query, expr=filter_expr)
    print(f"With filter '{filter_expr}': {len(results_with_filter)} results")

    # Step 3: 如果带 filter 后显著减少，说明 filter 过严
    if len(results_with_filter) < len(results_no_filter) * 0.3:
        print("[WARN] Filter may be too strict!")
        print("  Suggestions: check AND/OR logic, verify tenant_id")

向量检索基础：Embedding 与距离度量

向量检索的核心思想是将非结构化数据（文本、图片、音视频）映射到 embedding space 中的一个点，然后通过计算向量间的 distance metric 找到最「相似」的邻居。

CorpAssist · 知识库语义搜索 员工输入「年假剩余天数查询」，系统需要从 500 万条文档中找到语义最相关的 5 条。向量检索将每条文档转为 768 维向量，搜索时计算员工 query 与所有文档向量的相似度——但绝不能逐个计算，必须用 ANN 索引加速。

import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """余弦相似度 [-1, 1]，1 表示完全同向"""
    dot = np.dot(a, b)
    norm_a = np.linalg.norm(a)
    norm_b = np.linalg.norm(b)
    return float(dot / (norm_a * norm_b + 1e-10))

def inner_product(a: np.ndarray, b: np.ndarray) -> float:
    """内积，值越大越相似"""
    return float(np.dot(a, b))

def l2_distance(a: np.ndarray, b: np.ndarray) -> float:
    """欧氏距离，值越小越相似"""
    return float(np.linalg.norm(a - b))

public class DistanceMetric {

    public static double cosineSimilarity(float[] a, float[] b) {
        double dot = 0.0, normA = 0.0, normB = 0.0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            normA += a[i] * a[i];
            normB += b[i] * b[i];
        }
        return dot / (Math.sqrt(normA) * Math.sqrt(normB) + 1e-10);
    }

    public static double l2Distance(float[] a, float[] b) {
        double sum = 0.0;
        for (int i = 0; i < a.length; i++) {
            double diff = a[i] - b[i];
            sum += diff * diff;
        }
        return Math.sqrt(sum);
    }

    // CorpAssist 向量字段的索引参数
    public static final String MILVUS_METRIC_TYPE = "COSINE"; // 与 Python 索引保持对齐
    public static final int HNSW_M = 16;
    public static final int HNSW_EF_CONSTRUCTION = 200;
}

HNSW 核心参数直觉

先想 10 秒：HNSW 的「层级」和二叉树的分层有何本质区别？

HNSW 的层级是概率分配的——顶层节点只是「碰巧」被选中的少数节点，它们不代表任何语义分类。二叉树的分层有明确的比较逻辑。所以 HNSW 搜索时从顶层随机进入，而非二分查找。

召回率与精度：评估检索质量

recall precision 是向量检索最核心的评估指标。recall@k 衡量 top-k 结果中包含了多少「真正的」最近邻（相对 brute-force 全集）。

评估方法论

标准做法：先用暴力搜索算出一组 ground-truth（精确 top-100），再用 ANN 索引跑同样的查询，计算交集比例。

python 
import numpy as np
from typing import List

def recall_at_k(ann_ids: List[int], gt_ids: List[int], k: int) -> float:
    """召回率@k：ANN 结果中有多少在 ground-truth 中"""
    gt_set = set(gt_ids[:k])
    hits = sum(1 for idx in ann_ids[:k] if idx in gt_set)
    return hits / k

def precision_at_k(ann_ids: List[int], relevant_ids: List[int], k: int) -> float:
    """精确率@k：ANN 结果中有多少是相关的"""
    relevant_set = set(relevant_ids)
    hits = sum(1 for idx in ann_ids[:k] if idx in relevant_set)
    return hits / k

# CorpAssist 检索评估
gt_top10 = [101, 203, 45, 678, 890, 12, 456, 789, 34, 567]   # 暴力搜索结果
ann_top10 = [101, 45, 203, 678, 12, 789, 890, 999, 34, 111]  # HNSW 结果

for k in [1, 3, 5, 10]:
    print(f"recall@{k}={recall_at_k(ann_top10, gt_top10, k):.2f}")

Java：Recall 评估器

java 
import java.util.*;
import java.util.stream.*;

public class RecallEvaluator {

    public static double recallAtK(List annIds, List gtIds, int k) {
        Set gtSet = gtIds.stream().limit(k).collect(Collectors.toSet());
        long hits = annIds.stream().limit(k).filter(gtSet::contains).count();
        return (double) hits / k;
    }

    public static void main(String[] args) {
        List annIds = List.of(101L, 45L, 203L, 678L, 12L);
        List gtIds = List.of(101L, 203L, 45L, 678L, 890L);
        System.out.printf("recall@5 = %.2f%n", recallAtK(annIds, gtIds, 5));
    }
}
常见误区
  • 只看 recall 不看 precision：recall 高但返回了大量不相关结果，LLM 会「迷失在中间」。
  • 用固定 k 评估全部场景：FAQ 检索可能只看 top-1，文档检索需要 top-10。评估 k 应与业务对齐。
  • 忽略 latency 约束：在 200ms SLO 下，ef_search=512 不可接受。

Top-K 选择与业务影响

top k 的选择直接影响检索质量和 LLM 生成质量。k 值太小会遗漏相关信息，k 值太大会稀释上下文质量并增加 token 消耗。

业务场景推荐 Top-K理由
FAQ 精确匹配3-5答案通常集中在一两条文档中，k 小降低噪音
知识库问答5-10需要多片段信息综合，k 大覆盖全面
代码搜索3-8代码片段短而精，过多反而干扰
多文档摘要10-20需要跨文档信息汇总

常见问题（FAQ）

为什么 HNSW 的 recall 做不到 100%？
HNSW 在多层图中做近似搜索，顶层粗筛可能遗漏「正确但连接稀疏」的节点。召回率通常 95-99%，可通过增大 ef_search 来逼近 100%，但延迟会非线性增长。生产环境 95%+ 通常可接受。
Cosine 和 IP 在向量已归一化后等价吗？
是的。当向量被 L2 归一化后，cosine(a,b) = dot(a,b)。但仍建议在 schema 中明确声明 COSINE，避免混淆。CorpAssist 使用 text-embedding-3-small 默认返回归一化向量。
向量维度对检索延迟有多大影响？
维度直接影响内存带宽和计算时间。768 维向量的 HNSW 搜索延迟比 384 维高约 2x。Milvus 支持通过 quantization（量化）压缩维度，但会降低召回。建议 embedding 维度选 384-768 之间。

本章结论

向量检索的三大基石：距离度量、ANN 索引、评估指标。后续所有章节都建立在这些概念之上。

  • 距离度量决定语义尺度：Cosine / IP / L2 三种度量各有适用场景，CorpAssist 文本检索首选 COSINE。
  • ANN 索引用精度换速度：HNSW 是最主流的 ANN 方案，ef 和 M 参数控制 tradeoff。
  • Recall@k 是核心评估指标：用 brute-force 集合做 ground-truth，量化 tradeoff 效果。
  • 多维度的质量评估：不能只看 recall，还要结合 precision、latency 和业务场景。
  • Top-K 直接影响业务效果：k 值设计需平衡召回完整性与上下文洁净度。

常见问题（FAQ）

insert 后搜不到新数据？
刚插入的数据在 delta buffer 中只对 query() 可见。调用 flush() 并等待索引构建完成（几秒到几十秒）后才可检索。
load() 和 release() 是什么？
load() 将数据加载到查询节点内存，搜索前必须调用。release() 释放内存。CorpAssist 生产环境保持常驻 loaded。
Python 和 Java SDK 怎么分工？
Python SDK（PyMilvus）功能完整，负责写入和 schema 管理。Java SDK（MilvusClientV2）只做只读搜索。CorpAssist 原则：Python 管写和 schema，Java 只管读。

Python：应用层配额检查

python 
import time
from collections import defaultdict

class TenantQuotaManager:
    def __init__(self, max_qps: int = 100):
        self.max_qps = max_qps
        self._counters = defaultdict(lambda: [0, 0.0])  # {tenant: [count, window_start]}

    def check_and_consume(self, tenant_id: int) -> bool:
        now = time.time()
        count, window_start = self._counters[tenant_id]
        if now - window_start > 1.0:  # new window
            self._counters[tenant_id] = [1, now]
            return True
        if count >= self.max_qps:
            return False
        self._counters[tenant_id][0] += 1
        return True

quota = TenantQuotaManager(max_qps=100)
if quota.check_and_consume(tenant_id=1):
    # execute search
    pass
else:
    raise Exception("429 Too Many Requests")

import com.google.common.util.concurrent.RateLimiter;
import java.util.concurrent.ConcurrentHashMap;

@Component
public class QuotaInterceptor {
    private final ConcurrentHashMap limiters = new ConcurrentHashMap<>();

    public void checkQuota(long tenantId) {
        RateLimiter limiter = limiters.computeIfAbsent(tenantId,
            id -> RateLimiter.create(100.0)); // 100 QPS
        if (!limiter.tryAcquire()) {
            throw new RuntimeException("429 Too Many Requests - tenant " + tenantId);
        }
    }
}

常见问题（FAQ）

Pre-filter 会不会漏掉相关结果？
理论上会——如果 filter 后候选集太小，ANN 在这部分候选集中找 top-k，可能漏掉 filter 范围外的更相似结果。但这是业务需求决定的：ACL 过滤后的结果才是「应该看到的」。不是漏，是安全约束。
标量索引和向量索引有什么区别？
标量索引（inverted index）用于加速 filter 条件过滤，类似 Elasticsearch 的倒排索引。向量索引（HNSW/IVF）用于加速 ANN 搜索。两者在 Milvus 中独立管理，互不干扰。
Quota 超限时应该报错还是降级？
优先降级。CorpAssist 的做法是：返回缓存的热门结果，同时返回 429 的响应头。让调用方知道被限流但仍能获得部分服务。

本章结论

  • Pre-filter 先行：Filter 在向量搜索前执行，保证结果数量和 ACL 安全。
  • 标量索引加速：高频过滤字段建 inverted index，避免全量扫描。
  • 共享 + Filter 隔离：一条 filter 实现多租户隔离，管理复杂度最低。
  • Quota 保护：QPS、存储、并发三重配额防止单个租户影响全局。
  • ACL 一致性：Python 写入时标注 tenant/dept，Java 查询时自动注入 filter。

python 
from elasticsearch import Elasticsearch
import numpy as np

es = Elasticsearch("http://localhost:9200")

def hybrid_search(query: str, query_vec: list, top_k: int = 5):
    """BM25 + kNN hybrid search with RRF"""
    body = {
        "query": {
            "bool": {
                "must": [
                    {"match": {"chunk_text": query}}
                ],
                "filter": [
                    {"term": {"tenant_id": 1}}
                ]
            }
        },
        "knn": {
            "field": "embedding",
            "query_vector": query_vec,
            "k": top_k,
            "num_candidates": 100,
            "filter": {"term": {"tenant_id": 1}}
        },
        "rank": {
            "rrf": {
                "window_size": 50,
                "rank_constant": 60
            }
        },
        "size": top_k,
        "_source": ["id", "chunk_text", "source", "dept"]
    }
    resp = es.search(index="corpassist_index", body=body)
    return resp["hits"]["hits"]

# 使用

RRF 融合算法详解

RRF 是 Reciprocal Rank Fusion 的缩写。每条文档的融合得分 = Σ(1 / (rank_i + k))，其中 rank_i 是该文档在第 i 路检索中的排序位置，k 是 rank_constant（默认 60）。

参数含义默认值调整效果
rank_constant (k)平滑常数60k 越小，排名靠前的文档权重越大
window_size参与融合的最大窗口50超出窗口的结果不计入融合

Python：手动 RRF 实现（理解原理）

python 
def rrf_fusion(lists: list, k: int = 60, top_n: int = 5):
    scores = {}
    for rank_list in lists:
        for pos, doc_id in enumerate(rank_list):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (pos + k)
    # 按融合得分降序
    ranked = sorted(scores.items(), key=lambda x: -x[1])
    return [doc_id for doc_id, _ in ranked[:top_n]]

# CorpAssist 示例
bm25_rank = ["doc_A", "doc_B", "doc_C", "doc_D", "doc_E"]
knn_rank  = ["doc_F", "doc_A", "doc_B", "doc_G", "doc_H"]

fused = rrf_fusion([bm25_rank, knn_rank], k=60, top_n=3)
print(f"Fused top-3: {fused}")  # doc_A, doc_B, doc_C/F depending on scores

常见问题（FAQ）

RRF 和加权融合哪个好？
RRF 不需要分数归一化，对不同量纲的检索结果天然友好。加权融合要求两路分数在同一量纲（如都归一化到 0-1），否则权重无意义。推荐优先用 RRF。
num_candidates 设多大合适？
通常设 k * 20。k=10 时 num_candidates=200 合适。越大搜索精度越高但延迟越大。CorpAssist 生产环境设 100-500。
ES 向量搜索比 Milvus 慢吗？
ES 8.x 的 kNN 用 HNSW 实现，小规模（<100万）与 Milvus 延迟接近。大规模场景 Milvus 优化更好。CorpAssist 用 ES 做混合检索，Milvus 做主向量库。

本章结论

  • BM25 + kNN 互补：关键词精确匹配 + 语义相似搜索。
  • RRF 是推荐融合方法：基于排序位置，无需分数归一化。
  • Mapping 声明 dense_vector：dim 和 similarity 建索引前确认。
  • Filter 在双路中都需应用：BM25 和 kNN 各自带 ACL filter。
  • 权重靠评估集调优：不同查询类型的最佳混合比不同。

Python：Benchmark 脚本

python 
from pymilvus import Collection
import numpy as np
import time

def benchmark_params(collection_name, query_vectors, gt_ids, ef_values, top_k=10):
    collection = Collection(collection_name)
    collection.load()
    results = []

    for ef in ef_values:
        latencies = []
        recall_scores = []

        for q_vec, gt in zip(query_vectors, gt_ids):
            start = time.perf_counter()
            hits = collection.search(
                data=[q_vec], anns_field="embedding",
                param={"metric_type": "COSINE", "params": {"ef": ef}},
                limit=top_k
            )
            elapsed = time.perf_counter() - start
            latencies.append(elapsed * 1000)  # ms

            ann_ids = [h.id for h in hits[0]]
            hits_count = sum(1 for aid in ann_ids if aid in set(gt[:top_k]))
            recall_scores.append(hits_count / top_k)

        results.append({
            "ef": ef,
            "avg_recall": np.mean(recall_scores),
            "p50_ms": np.percentile(latencies, 50),
            "p95_ms": np.percentile(latencies, 95),
            "p99_ms": np.percentile(latencies, 99),
        })
        print(f"ef={ef}: recall@10={results[-1]['avg_recall']:.3f}, "
              f"p95={results[-1]['p95_ms']:.1f}ms")

    collection.release()
    return results

# 使用
query_vecs = [np.random.rand(768).tolist() for _ in range(100)]
gt_ids = [list(range(20)) for _ in range(100)]  # ground truth
results = benchmark_params("corpassist_kb", query_vecs, gt_ids, [16, 32, 64, 128, 256])

常见问题（FAQ）

efConstruction 设越大越好吗？
不是。efConstruction 越大索引质量越高，但构建时间和内存占用也越大。超过 300 后 recall 提升趋于收敛。推荐 100-200，大规模可提升到 300。
M=32 比 M=16 好在哪？代价是什么？
M 越大，图连接越密，recall 越高，但索引内存占用约翻倍（M=16 时每节点存 16 个邻居指针）。搜索延迟略有增加。通常 M=16 是平衡点。
如何攒 ground-truth 数据集？
暴力搜索法：用 brute-force（flat index）在全部数据上计算真 top-k。小数据集（<10万条）可做，大数据集用代表性样本。CorpAssist 从各 tenant 随机抽 100 条 query。

  • 参数是 tradeoff：ef 和 M 提升 recall 但增加延迟和内存。
  • Grid Search 找最优组合：固定 query set，遍历参数，输出对比表。
  • 报告多分位延迟：avg 瞒人，p95 + p99 暴露长尾。
  • 场景决定目标：FAQ > 0.98 recall，文档检索 > 0.95。

import matplotlib.pyplot as plt
import numpy as np

def precision_curve(queries, candidate_sets, reranker, max_n=100, step=10):
    results = []
    for n in range(step, max_n + 1, step):
        precisions = []
        for query, candidates in zip(queries, candidate_sets):
            pairs = [(query, c["text"]) for c in candidates[:n]]
            scores = reranker.predict(pairs)
            ranked = sorted(zip(candidates[:n], scores), key=lambda x: -x[1])
            # top-5 精排中相关比例
            relevant = sum(1 for _, s in ranked[:5] if s > 0.5)
            precisions.append(relevant / 5)
        results.append({"rerank_n": n, "avg_precision@5": np.mean(precisions)})
    return results

常见问题（FAQ）

Rerank 和 ANN 第二次搜索有什么区别？
本质不同。ANN 是 bi-encoder：query 和 doc 各自独立编码为向量，用距离计算相似度。Cross-encoder 把 query+doc 拼接后过 transformer，能捕捉 query 和 doc 之间的深层交互。Cross-encoder 精度更高但无法用 ANN 索引。
Rerank 模型怎么部署？
推荐用 Triton Inference Server 或 FastAPI 封装。CorpAssist 将 bge-reranker-v2-m3 部署为独立 HTTP 服务，GPU 推理。batch size 可调整——batch=16 时 50 条只需 1-2 次推理。
Rerank TOP-5 和 ANN TOP-5 可能不同吗？
非常常见。ANN 的 bi-encoder 打分可能误判，cross-encoder 能纠正。CorpAssist 实践中约 30% 的最终 top-5 会被 rerank 重排序。

本章结论

  • 两阶段是标配：ANN 召回 + Cross-encoder 精排，精度和延迟兼顾。
  • Rerank 精度高于 ANN：cross-encoder 捕捉深层交互，约 30% 结果被重新排序。
  • 延迟与候选数线性相关：用精度曲线选最佳 rerank_n。
  • 部署为独立服务：Rerank 需要 GPU，独立部署不影响 ANN 召回。
  • Python 推理，Spring 调用：Rerank 模型在 Python 侧推理，Spring 侧 HTTP 调用。

        self.stages.append({"name": name, "fn": fn, "parallel": parallel})

    def execute(self, query: str) -> Dict:
        results = {}
        for stage in self.stages:
            if stage["parallel"]:
                # 并行执行
                with ThreadPoolExecutor() as exe:
                    futures = {exe.submit(s["fn"], query): s["name"]
                               for s in stage["parallel_branches"]}
                    for f in as_completed(futures):
                        results[futures[f]] = f.result()
            else:
                results[stage["name"]] = stage["fn"](query)
        return results

# 使用
pipeline = RetrievalPipeline()
pipeline.add_stage("rewrite", lambda q: multi_query_rewrite(q, n=3))

常见问题（FAQ）

多路召回中各路的 Top-K 怎么设？
各路 Top-K 相加应远大于最终需要的数量。CorpAssist 通常设各路 50，融合后共 100-150 条，Rerank 后取 5 条。这样保证召回率而精排筛选。
什么时候需要加规则召回路？
当业务有强规则匹配需求时。例如「置顶政策」必须出现在结果中，「最新政策」按时间优先。规则路用标签或元数据过滤实现。
管线执行失败应该整条重试还是部分重试？
部分重试。某一路召回超时不影响其他路——降级为该路返回空集，继续融合。CorpAssist 给每路设独立超时（默认 200ms）。

本章结论

  • DAG 管线编排：Rewrite → Multi-Recall → Fusion → Rerank 四阶段。
  • Query Rewrite 提升召回：MultiQuery 和 HyDE 改写后召回 +15%。
  • 多路并行召回：向量+BM25+规则三路并行，覆盖更多维度。
  • RRF 融合：不依赖分数的融合方法，天然去重。
  • 降级是设计的一部分：每路独立超时，一路失败不影响整体。

java 
import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Timer;
import io.micrometer.core.instrument.Counter;
import java.util.concurrent.TimeUnit;

@Component
public class SearchMetrics {
    private final MeterRegistry registry;

    public SearchMetrics(MeterRegistry registry) {
        this.registry = registry;
    }

    public void recordSearch(long tenantId, String collection, long latencyMs, boolean success, int resultCount) {
        // 延迟直方图
        Timer.Sample sample = Timer.start(registry);
        Timer timer = Timer.builder("retrieval.search.latency")
            .tag("tenant", String.valueOf(tenantId))
            .tag("collection", collection)
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
        sample.stop(timer);

        // 搜索计数
        Counter.builder("retrieval.search.total")
            .tag("tenant", String.valueOf(tenantId))
            .tag("collection", collection)
            .register(registry).increment();

        // 空结果计数
        if (resultCount == 0) {
            Counter.builder("retrieval.search.empty")
                .tag("tenant", String.valueOf(tenantId))
                .register(registry).increment();
        }

        // 错误计数
        if (!success) {
            Counter.builder("retrieval.search.error")
                .tag("tenant", String.valueOf(tenantId))
                .register(registry).increment();
        }
    }
}

Python：全量索引 Job

python 
from pymilvus import Collection, utility
from langchain.text_splitter import RecursiveCharacterTextSplitter
from openai import OpenAI
import psycopg2
import numpy as np

client = OpenAI()
emb_model = "text-embedding-3-small"

def full_build():
    # 1. 从 PostgreSQL 拉取全量文档
    conn = psycopg2.connect("host=localhost dbname=corpassist user=...")
    cur = conn.cursor()
    cur.execute("SELECT id, content, source, tenant_id FROM documents WHERE is_deleted=false")
    rows = cur.fetchall()
    print(f"Fetched {len(rows)} documents from DB")

    # 2. Chunk
    splitter = RecursiveCharacterTextSplitter(chunk_size=1024, chunk_overlap=200)
    chunks = []
    for row in rows:
        doc_chunks = splitter.split_text(row[1])
        for i, chunk in enumerate(doc_chunks):
            chunks.append({
                "doc_id": row[0],
                "chunk_text": chunk,
                "source": row[2],
                "tenant_id": row[3],
                "chunk_index": i,
            })
    print(f"Split into {len(chunks)} chunks")

    # 3. Embed + Write（batch）
    collection = Collection("corpassist_kb")
    batch_size = 1000
    for start in range(0, len(chunks), batch_size):
        batch = chunks[start:start + batch_size]
        texts = [c["chunk_text"] for c in batch]
        resp = client.embeddings.create(input=texts, model=emb_model, dimensions=768)
        embeddings = [d.embedding for d in resp.data]

        entities = [
            [hash(c["doc_id"]) for c in batch],      # id
            embeddings,                                # embedding
            texts,                                     # chunk_text
            [c["source"] for c in batch],              # source
            [c["tenant_id"] for c in batch],           # tenant_id
        ]
        collection.insert(entities)

    collection.flush()
    print(f"Full build complete: {len(chunks)} chunks indexed")
    conn.close()

if __name__ == "__main__":
    full_build()

from pymilvus import Collection
from openai import OpenAI
import json

client = OpenAI()

class IncrementalIndexer:
    def __init__(self):
        self.collection = Collection("corpassist_kb")

    def handle_insert(self, doc_id: str, content: str, source: str, tenant_id: int):
        """处理文档新增"""
        from langchain.text_splitter import RecursiveCharacterTextSplitter
        splitter = RecursiveCharacterTextSplitter(chunk_size=1024, chunk_overlap=200)
        chunks = splitter.split_text(content)

        texts = chunks
        resp = client.embeddings.create(input=texts, model="text-embedding-3-small", dimensions=768)
        embeddings = [d.embedding for d in resp.data]

        self.collection.insert([
            [hash(f"{doc_id}_{i}") for i in range(len(chunks))],
            embeddings,
            texts,
            [source] * len(chunks),
            [tenant_id] * len(chunks),
        ])
        self.collection.flush()
        print(f"Inserted {len(chunks)} chunks for doc {doc_id}")

    def handle_delete(self, doc_id: str):
        """处理文档删除"""
        expr = f"id >= {hash(doc_id + '_0')} and id <= {hash(doc_id + '_999')}"
        self.collection.delete(expr)
        print(f"Deleted chunks for doc {doc_id}")

    def handle_update(self, doc_id: str, content: str, source: str, tenant_id: int):
        """处理文档更新 = delete + insert"""
        self.handle_delete(doc_id)
        self.handle_insert(doc_id, content, source, tenant_id)

索引 Runbook 与运维

常见问题（FAQ）

全量构建期间增量事件怎么处理？
全量构建时，增量事件暂存到缓冲区。全量完成后按时间戳回放增量。通常全量在凌晨执行，增量少不影响。
索引版本回滚怎么操作？
切换 alias 回旧 collection。假设旧版本是 corpassist_kb_v1，执行 utility.alter_alias("corpassist_kb_v1", "corpassist_kb")。整个过程是别名切换，秒级完成。
Spring 服务何时确认索引更新完成？
Spring 服务不做更新确认——它只消费数据。索引更新由 Python 索引工程负责，Spring 侧通过查询验证数据可见性。CorpAssist 在索引 Job 完成后发送通知消息。
