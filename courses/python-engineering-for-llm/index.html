pyproject 布局

PEP 621 规定 pyproject.toml 为项目元数据与构建配置的入口。CorpAssist-python 教学骨架（见 demo）典型结构如下：

推荐目录树（src 布局）

corpassist-python/
├── pyproject.toml          # 项目声明 + ruff/mypy 配置
├── uv.lock                 # 锁定传递依赖（提交 Git）
├── .python-version         # 本地 Python 版本提示（可选）
├── README.md
├── src/
│   └── corpassist/
│       ├── __init__.py
│       ├── core/           # 配置、异常、日志
│       ├── llm/            # 网关客户端
│       └── rag/            # 检索与索引（后续章展开）
├── tests/

src 布局 把可安装包放在 src/corpassist/，避免「仓库根目录与 import 路径冲突」——心智上接近 Java 的 src/main/java。在 IDE 里打开仓库根目录时，不会误把 corpassist.py 同名文件当作包入口。

名称、版本、运行时依赖 — 类比 <dependencies>。

ruff、mypy、pytest 等工具配置 — 类比 <pluginManagement>。

[build-system]

打包后端（如 hatchling）— 类比 maven-compiler-plugin。

先想 10 秒：为什么依赖不要写进 requirements.txt 就完事？

裸 requirements.txt 往往只锁顶层包，缺少传递依赖与工具链版本；pyproject + uv.lock 才是可审计、可复现的「工程声明 + 锁定」组合。

与 Maven 模块对照

CorpAssist 双栈（30 秒） Spring Boot 侧：corpassist-gateway、corpassist-bff 等 Maven 子模块 负责鉴权、聚合与 OpenAPI 契约；Python 侧独立仓库提供 RAG 检索与 LLM 编排 API。两边通过 HTTP + 统一 request_id 协作，而非同一父 POM。

Python 生态更常见 单仓单服务：一个 deployable 镜像对应一个 pyproject，用包内子目录划分职责，而不是 Maven 式的多 <module> 聚合构建。

反例 vs 推荐

不推荐
  • 在 Spring 仓库里塞 scripts/rag.py 无包结构
  • Python 侧再拆 5 个独立 pip 包，本地联调成本爆炸
  • 包名与 Spring 模块名完全无关，联调文档对不上
推荐
  • 独立 corpassist-python 仓库 + src 布局
  • corpassist.{core,llm,rag} 与模块表一一可述
  • 网关 URL、API Key 走环境变量，不写进 POM/pyproject 默认值

依赖锁定

依赖锁定 解决「开发机 pip 解析出 httpx 0.27.2、CI 解析出 0.28.0」类漂移。CorpAssist 要求：uv.lock 提交 Git，流水线执行 uv sync --frozen（或等价）拒绝擅自升级。

锁定文件里有什么

  • 每个包精确版本与哈希（含传递依赖）
  • 与 pyproject.toml 中范围声明（如 httpx>=0.27）的关系：改范围后需 uv lock 再提交
  • 类比 Maven：dependencyManagement + 可复现的 mvnw 锁定插件版本

Git 应提交 vs 忽略

路径进 Git？原因
uv.lock是保证 CI/生产与本地一致
pyproject.toml是声明意图与工具配置
.venv/否机器本地产物，体积大且不可移植
__pycache__/否字节码缓存
.env（含密钥）否12-factor：密钥只经环境注入

升级依赖的安全节奏

# 1. 本地改 pyproject 或直接 uv add package==x.y.z
uv lock
uv sync
uv run pytest

# 2. 提交 pyproject.toml + uv.lock 同一 PR
# 3. CI 必须 green 才合并

LLM 相关库（httpx、pydantic）小版本也可能改变序列化行为；锁文件 PR 应带回归说明，与 Java 侧 BOM 升级同理。

poetry.lock 与 uv.lock 能混用吗？
CorpAssist 教学统一选 uv；若历史仓库用 Poetry，迁移时一次性切到 uv 并删除旧 lock，避免双 lock 源打架。
CI 里每次 uv sync 会不会很慢？
配合缓存层（wheel 缓存 / uv cache）通常可接受；比「CI 每次 pip 解析出不同树再 mysteriously fail」便宜得多。

typing 常用模式

CorpAssist Python 侧高频类型模式如下；与 demo type_patterns_demo.py 一一对应。

TypedDict：Chat 消息

from typing import Literal, TypedDict

class ChatMessage(TypedDict):
    role: Literal["system", "user", "assistant"]
    content: str

def build_messages(system: str, history: list[ChatMessage]) -> list[ChatMessage]:
    return [{"role": "system", "content": system}, *history]

TypedDict 适合 JSON 形状稳定、字段有限的 DTO；完整校验仍交给 常用库与 HTTP 客户端 中 Pydantic v2 的 BaseModel（HTTP 边界）。

Protocol：Embedding 客户端

from typing import Protocol

class EmbeddingClient(Protocol):
    def embed(self, texts: list[str]) -> list[list[float]]: ...

class MockEmbeddingClient:
    def embed(self, texts: list[str]) -> list[list[float]]:
        return [[0.1, 0.2, 0.3] for _ in texts]

def index_chunks(client: EmbeddingClient, texts: list[str]) -> list[list[float]]:
    return client.embed(texts)

Protocol 描述「必须有 embed 方法」；鸭子类型 在运行时成立，mypy 在静态侧验证。Mock 实现无需声明 implements。

泛型与可选

from collections.abc import Sequence

def truncate_history(messages: list[ChatMessage], max_turns: int) -> list[ChatMessage]:
    if max_turns <= 0:
        return messages[:1]
    system = [m for m in messages if m["role"] == "system"]
    rest = [m for m in messages if m["role"] != "system"]
    return system + rest[-(max_turns * 2):]

def first_item(items: Sequence[str]) -> str:
    return items[0]

Python 3.12+ 使用内置 泛型 list[str]，不再依赖 typing.List。只读参数优先 Sequence，类似 Java 只读视图。

ruff/mypy

CorpAssist 在 pyproject.toml 集中配置两类门禁：ruff 负责 lint + format，mypy 负责静态类型。

pyproject 中的工具段

[tool.ruff]
line-length = 100
target-version = "py312"

[tool.ruff.lint]
select = ["E", "F", "I", "UP"]  # pycodestyle, pyflakes, isort, pyupgrade

[tool.mypy]
python_version = "3.12"
strict = true
warn_return_any = true

本地与 CI 命令

命令作用Java 近似
uv run ruff check .Lint 全仓库Checkstyle
uv run ruff format .统一格式Spotless / google-java-format
uv run mypy src类型检查源码javac + 注解处理器

strict 模式常见报错与修复

mypy 会拦
  • 缺返回类型 / 参数类型
  • role = "admin" 赋给 Literal 联合
  • 隐式 Any 传播
  • 未处理的 Optional
修复习惯
  • 公共 API 全标注；私有函数至少标注参数
  • 用 Literal / Enum 限制枚举字段
  • 第三方缺 stub 时用 types-* 或 targeted # type: ignore
  • if x is None: return 早返回收窄类型

CI 门禁建议

uv run ruff check .
uv run ruff format --check .
uv run mypy src
uv run pytest

建议在 PR 模板写清：ruff + mypy + pytest 三者皆绿，与 Java 模块「编译 + 单测」同级。AI 生成代码不得跳过 mypy。

ruff 与 mypy 分工（标签页）

拦什么：未使用 import、格式不一致、可自动 upgrade 的旧语法（UP 规则）。

类比：Checkstyle + isort + 部分 formatter。

CorpAssist：uv run ruff check . 与 ruff format --check . 进 CI。

拦什么：类型不匹配、缺注解、Literal 赋值越界、隐式 Any。

类比：javac 类型检查（非运行时）。

CorpAssist：strict = true 覆盖 src/corpassist，DTO 重构前最后一道闸。

TypedDict 与 @dataclass 怎么选？
与 JSON 互转多、字段即 dict 键 → TypedDict；需要方法、不可变对象、非 dict 序列化 → dataclass。LLM messages 通常 TypedDict + Pydantic 边界。

Java 有而 Python 需注意的差异

  • 无默认编译期：类型错可能跑到生产；mypy 是团队契约，不是语言内置。

Java record vs Python TypedDict

Java record
java 
public record ChatMessage(String role, String content) {}
List<ChatMessage> history;
Python TypedDict
python 
class ChatMessage(TypedDict):
    role: Literal["system", "user", "assistant"]
    content: str
history: list[ChatMessage]

record 在运行时自带校验与不可变语义；TypedDict 主要服务静态检查，运行时仍是普通 dict — HTTP 边界仍应交给 Pydantic。

可读性原则

可读性 不是「代码短」，而是下一个 Java 同事扫一眼能改对。CorpAssist Python 约定：

  • 命名：模块/函数/变量 snake_case；类 PascalCase；常量 UPPER_SNAKE — 与 PEP 8 一致，JSON 字段经 Pydantic alias 与 Java camelCase 对齐。
  • 早返回：减少嵌套 if，与 Java Guard Clause 相同。
  • 小函数：truncate_history、build_messages 单一职责；单函数建议 < 40 行。
  • 显式优于隐式：拒绝魔法全局；配置走 Settings（结构化配置）。
  • 类型即文档：公共函数必须有 类型注解 与一句 docstring（非 trivial 时）。

反例 vs 推荐（RAG 片段处理）

不推荐
  • def go(d, m, f): ... 无类型无 docstring
  • 200 行「上帝函数」含 HTTP + 切块 + 向量化
  • except: 裸捕获吞掉栈
推荐
  • def truncate_history(messages: list[ChatMessage], max_turns: int) -> list[ChatMessage]
  • 分层：llm/client.py 只谈 HTTP，rag/chunking.py 只谈文本
  • 异常转 AppError（见 异常、日志与配置）

Optional 与类型收窄

def pick_model(explicit: str | None, settings_default: str) -> str:
    if explicit is None:
        return settings_default
    return explicit  # mypy 知 explicit 为 str

Java Optional<T> 在 Python 侧常用 T | None；配合早返回，mypy 可自动收窄，减少「忘判空」类 bug。

异步预告

异步客户端 httpx.AsyncClient 将在 asyncio 与并发 与 FastAPI 路由配合；本章先用同步 Client 理解请求/响应模型。

不推荐
  • 每次请求 httpx.post(url) 无 Client 复用
  • 无 timeout，LLM 挂死占满 worker
  • 裸 requests 且无 mock 策略
推荐
  • 应用 lifespan 创建/关闭单例 Client
  • 区分 connect/read 超时
  • 测试用 MockTransport
先想 10 秒：为何 LLM 读超时要比普通 REST 长？

Token 生成是流式/长耗时计算；过短 read timeout 会在模型仍正常生成时误杀。应对：合理 read 超时 + 可观测 latency + 重试策略 仅用于幂等/可恢复错误。

Client 生命周期（FastAPI 预告）

CorpAssist 生产环境应在应用 lifespan 中创建单例 httpx.Client，避免每个请求新建 TCP 连接。类比 Spring WebClient Bean 单例：

from contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http = httpx.Client(
        base_url=settings.llm_base_url,
        timeout=httpx.Timeout(connect=5, read=60, write=10, pool=5),
        limits=httpx.Limits(max_connections=100, max_keepalive_connections=20),
    )
    yield
    app.state.http.close()

app = FastAPI(lifespan=lifespan)

关闭时务必 client.close()，否则容器滚动时可能留下半开连接。测试环境可注入 MockTransport 替换真实 Client。

嵌套模型与 field_validator

CorpAssist RAG 请求常含嵌套结构（如 messages 列表）。Pydantic v2 用 field_validator 做跨字段校验：

from pydantic import BaseModel, field_validator

class RagQueryRequest(BaseModel):
    query: str
    top_k: int = 5
    filters: dict[str, str] | None = None

    @field_validator("query")
    @classmethod
    def query_not_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("query must not be empty")
        return v.strip()

    @field_validator("top_k")
    @classmethod
    def top_k_in_range(cls, v: int) -> int:
        if not 1 <= v <= 50:
            raise ValueError("top_k must be between 1 and 50")
        return v

校验失败抛 ValidationError，在 FastAPI 路由层自动映射 422；业务层可再包装为 ValidationAppError（见下一章）。

完整调用函数

def chat_completion(
    client: httpx.Client,
    base_url: str,
    api_key: str,
    req: ChatRequest,
) -> ChatResponse:
    url = f"{base_url.rstrip('/')}/chat/completions"
    headers = {"Authorization": f"Bearer {api_key}"}
    resp = client.post(url, json=req.model_dump(), headers=headers, timeout=30.0)
    resp.raise_for_status()
    return ChatResponse.model_validate(resp.json())

class AppError(Exception):
    code: str = "INTERNAL_ERROR"
    http_status: int = 500

    def __init__(self, message: str, *, details: dict | None = None) -> None:
        super().__init__(message)
        self.details = details or {}

class LlmUpstreamError(AppError):
    code = "LLM_UPSTREAM"
    http_status = 502

class ConfigError(AppError):
    code = "CONFIG_INVALID"

边界转换函数

在 LLM 调用边界统一捕获 httpx 与 Pydantic 异常，避免业务代码重复 try/except：

import httpx
from pydantic import ValidationError

def wrap_llm_call(fn, *args, **kwargs):
    try:
        return fn(*args, **kwargs)
    except httpx.TimeoutException as exc:
        raise LlmUpstreamError("gateway timeout", details={"cause": str(exc)}) from exc
    except httpx.HTTPStatusError as exc:
        if exc.response.status_code == 429:
            raise LlmUpstreamError("rate limited", details={"status": 429}) from exc
        raise LlmUpstreamError("upstream error", details={"status": exc.response.status_code}) from exc
    except ValidationError as exc:
        raise ValidationAppError("invalid response schema", details={"errors": exc.errors()}) from exc

包装器在 corpassist.core.errors 模块集中维护；FastAPI 路由只抛 AppError 子类，handler 统一序列化。

中间件绑定 request_id（FastAPI 预告）

每个 HTTP 请求进入时从 header 读取或生成 request_id，写入 structlog contextvars，请求结束清理：

import uuid
from starlette.middleware.base import BaseHTTPMiddleware

class RequestIdMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        rid = request.headers.get("X-Request-Id") or str(uuid.uuid4())
        request.state.request_id = rid
        structlog.contextvars.clear_contextvars()
        structlog.contextvars.bind_contextvars(request_id=rid, service="corpassist-python")
        response = await call_next(request)
        response.headers["X-Request-Id"] = rid
        return response

与 Spring OncePerRequestFilter 注入 MDC 对称；BFF 必须转发同一 X-Request-Id，否则双栈联查断裂。

12-factor 配置

12-factor 中与本章最相关的两条：III Config 与 XI Logs。CorpAssist Python 与 Spring Boot 部署在同一 K8s 集群时，应遵守相同约束。

Config：环境注入

  • 构建时 vs 运行时：Docker 镜像 build 阶段不含 staging/prod 密钥；运行时由 Secret 挂载 env。
  • 配置与环境分离：CorpAssistSettings 只读 env，不把 .env 打进镜像。
  • 启动失败要快：缺 CORPASSIST_LLM_API_KEY 时 ConfigError 阻止进程起来，而非首请求才炸。

  • 应用只写 stdout/stderr，不写本地 rotating file（容器内路径难采集）。
  • JSON 一行一条，便于 ELK json_lines 解析。
  • 人类可读说明可写 stderr（demo 做法），生产 JSON 仍走 stdout。

async/await

CorpAssist · Chat 代理 用户提问后，Python 服务需并行检索向量库并调用 LLM。单线程阻塞模型下每个请求占一个线程；asyncio 在 I/O 等待时切换协程，单机可承载更多并发连接。

FastAPI 的 async def 路由跑在 asyncio 事件循环上，类比 Spring WebFlux 非阻塞栈（非 Servlet 阻塞模型）。CorpAssist 约定：对外 HTTP 用 async 路由，对内 LLM 用 httpx.AsyncClient。

async 路由 + AsyncClient

python 
import httpx
from fastapi import Depends, FastAPI

app = FastAPI()

async def get_http_client() -> httpx.AsyncClient:
    # 生产环境在 lifespan 中创建并挂载 app.state
    async with httpx.AsyncClient(timeout=30.0) as client:
        yield client

@app.get("/v1/rag/query")
async def rag_query(
    q: str,
    client: httpx.AsyncClient = Depends(get_http_client),
) -> dict:
    # await 让出控制权，等待网关响应
    resp = await client.get("https://httpbin.org/get", params={"q": q})
    resp.raise_for_status()
    return {"status": "ok", "upstream": resp.status_code}

并发限制

模型网关与向量库都有 QPS/并发上限。无节制 asyncio.gather 可能触发 429 或拖垮本进程内存。Semaphore 限制同时 in-flight 的请求数。

Semaphore 包装 LLM 调用

python 
import asyncio

# 进程级单例，数值来自 CORPASSIST_LLM_MAX_CONCURRENT
_LLM_SEM = asyncio.Semaphore(10)

async def call_llm(client: httpx.AsyncClient, payload: dict) -> dict:
    async with _LLM_SEM:
        resp = await client.post("/v1/chat/completions", json=payload)
        resp.raise_for_status()
        return resp.json()
无限制 gather
  • 100 个用户同时触发 100 路 LLM
  • 网关返回 429，客户端重试雪崩
  • 内存中堆积未完成的 response body
Semaphore + 队列
  • 最多 10 路并发，其余 await 槽位
  • 配合 重试策略 处理 429
  • 指标可暴露「排队长度」
先想 10 秒：Semaphore(10) 与线程池 max=10 有何不同？

Semaphore 限制的是协程同时 await 的数量，线程开销极小；线程池限制 OS 线程数，适合阻塞 IO。LLM HTTP 调用用 Semaphore + AsyncClient 即可。

流式响应

Chat 场景需逐 token 推送。HTTP 使用 text/event-stream（SSE）：每行 data: ...\n\n。FastAPI 用 StreamingResponse 包装异步生成器。

SSE 流式端点

python 
import asyncio
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()
SEM = asyncio.Semaphore(5)

async def token_generator():
    for chunk in ["你好", "，", "CorpAssist"]:
        yield f"data: {chunk}\n\n"
        await asyncio.sleep(0.05)

@app.get("/v1/chat/stream")
async def chat_stream():
    async with SEM:
        return StreamingResponse(
            token_generator(),
            media_type="text/event-stream",
        )

Spring BFF 可用 WebClient 订阅 SSE 并转发给前端；注意设置 Cache-Control: no-cache 与连接超时（见 中间件、鉴权与 CORS）。

  • 单向 server → client，HTTP 友好
  • CorpAssist Chat 默认选型
  • 网关与 Nginx 需关闭缓冲
  • 双向，适合协作编辑
  • LLM Chat 通常不需要 client 持续上行
  • 运维与鉴权复杂度更高

阻塞陷阱

async 路由中最常见的生产事故：在 async def 内调用同步阻塞 API（requests.get、time.sleep、CPU 密集 numpy、同步数据库驱动），导致整个事件循环停滞。

反例与修复

python
# 反例：阻塞整个 loop
@app.get("/bad")
async def bad():
    time.sleep(5)  # 或 requests.get(...)
    return {"ok": True}
python
import asyncio

@app.get("/good")
async def good():
    await asyncio.sleep(5)  # 异步等待

# 必须阻塞时：
@app.get("/legacy")
async def legacy():
    loop = asyncio.get_running_loop()
    return await loop.run_in_executor(None, sync_heavy_fn)
  • 优先：换 async 库（httpx 而非 requests，asyncpg 而非 psycopg2 同步）。
  • 次选：run_in_executor 把阻塞代码丢进线程池。
  • 兜底：路由改为 def（非 async），FastAPI 自动放线程池——但高并发下线程数仍有限。
FAQ：向量检索库只有同步 SDK 怎么办？

用 run_in_executor 包装，或在本章之后的 性能与资源 中评估批处理与连接池。切忌在 hot path 上长时间阻塞 loop。

lifespan 中挂载 AsyncClient（推荐）

python 
from contextlib import asynccontextmanager
import httpx
from fastapi import FastAPI, Request

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http = httpx.AsyncClient(timeout=30.0)
    yield
    await app.state.http.aclose()

app = FastAPI(lifespan=lifespan)

def get_http(request: Request) -> httpx.AsyncClient:
    return request.app.state.http

与 FastAPI 路由与依赖注入 中的 lifespan 衔接：进程级单例客户端，避免每请求创建 TCP 连接。

JWT 对接 Spring

用户身份由 Spring 鉴权后，BFF 将 JWT 转发给 Python（或换发内部 token）。Python 侧验证签名、issuer、exp，提取 sub / tenant_id 供审计。

解析 JWT claims（HS256 示例）

python 
import jwt
from fastapi import Header, HTTPException

async def get_current_user(authorization: str | None = Header(default=None)) -> dict:
    if not authorization or not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Missing bearer token")
    token = authorization.removeprefix("Bearer ").strip()
    try:
        payload = jwt.decode(
            token,
            settings.jwt_secret,
            algorithms=["HS256"],
            issuer="corpassist-spring",
        )
    except jwt.PyJWTError as exc:
        raise HTTPException(status_code=401, detail="Invalid token") from exc
    return {"user_id": payload["sub"], "tenant": payload.get("tenant_id")}

生产可用 RS256 + JWK 集；issuer / audience 必须与 Spring Security 配置一致。双栈联调清单见 双栈协作规范。

import structlog

class RequestIdMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        rid = request.headers.get("X-Request-Id") or str(uuid.uuid4())
        structlog.contextvars.clear_contextvars()

        return response

超时

LLM 与 RAG 链路耗时长，但 HTTP 连接不能无限挂起。timeout 分层：网关全局、Python middleware、httpx 客户端 per-request。

中间件超时（示意）

python 
import asyncio
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.responses import JSONResponse

class TimeoutMiddleware(BaseHTTPMiddleware):
    def __init__(self, app, seconds: float = 60.0):
        super().__init__(app)
        self.seconds = seconds

    async def dispatch(self, request, call_next):
        try:
            return await asyncio.wait_for(call_next(request), timeout=self.seconds)
        except asyncio.TimeoutError:
            return JSONResponse(
                status_code=504,
                content={"code": "GATEWAY_TIMEOUT", "message": "Request timed out"},
            )
  • 流式 SSE 端点应排除或放宽全局 timeout，改用 idle timeout。
  • httpx timeout=httpx.Timeout(30.0, connect=5.0) 控制上游 LLM。
  • 与 Spring Cloud Gateway response-timeout 协调，避免一层先断一层仍等。

CorpAssist-python 通过 httpx 调用 LLM 网关与向量库 HTTP 接口。每次请求 async with httpx.AsyncClient() 都会新建连接池，高 QPS 下 TLS 握手与端口耗尽是常见瓶颈——类比 Spring 侧未复用 WebClient Bean。

Connection pool 在进程内共享；多 Worker 时每个 worker 各有一份池，需与网关 max_connections 配额对齐。

lifespan 中创建与注入

python 
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator
import httpx
from fastapi import Depends, FastAPI, Request

@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    limits = httpx.Limits(max_connections=100, max_keepalive_connections=20)
    timeout = httpx.Timeout(30.0, connect=5.0)
    app.state.http = httpx.AsyncClient(limits=limits, timeout=timeout)
    yield
    await app.state.http.aclose()

app = FastAPI(lifespan=lifespan)

async def get_http(request: Request) -> httpx.AsyncClient:
    return request.app.state.http
参数建议起点说明
max_connections 50–200 与 Pod 副本数 × worker 数 × 并发乘积对照网关限流
max_keepalive_connections 10–20 空闲连接过多占内存，过少则频繁握手
Timeout connect 5s / read 30s 与 中间件超时 及 Spring RestTemplate 一致

批处理 Embedding

单条调用 Embedding API 简单，但索引 10 万 chunk 时 RTT 与序列化开销主导耗时。Batch embedding 将多条文本合并一次请求（受模型 max_batch_size 与 token 上限约束），是提升 throughput 的首要手段。

切块迭代器模式

python 
from itertools import batched  # Py 3.12+

async def embed_all(
    client: httpx.AsyncClient,
    texts: list[str],
    *,
    batch_size: int = 32,
) -> list[list[float]]:
    vectors: list[list[float]] = []
    for batch in batched(texts, batch_size):
        resp = await client.post(
            "/v1/embeddings",
            json={"input": list(batch), "model": "text-embedding-3-small"},
        )
        resp.raise_for_status()
        data = resp.json()["data"]
        vectors.extend(item["embedding"] for item in data)
    return vectors
  • 内存峰值低，适合小 Pod（512Mi–1Gi）
  • 429 限流时单批失败影响面小
  • 适合在线补索引与混部 API 服务
  • 离线全量重建吞吐高
  • 需监控单批响应体大小与超时
  • 建议独立 Worker 进程，避免挤占 API semaphore 配额

429 退避重试（Embedding 可重试）

python 
async def embed_batch_with_retry(client, batch: list[str], max_attempts: int = 3):
    for attempt in range(max_attempts):
        resp = await client.post("/v1/embeddings", json={"input": batch})
        if resp.status_code not in (429, 503):
            resp.raise_for_status()
            return resp.json()
        await asyncio.sleep(min(2 ** attempt, 30))
    raise LlmRateLimitError("embedding batch exhausted retries")

批处理与 重试策略 配合：对 429/503 做指数退避；非幂等场景（带随机噪声的 Chat）勿盲目重试整批，Embedding 通常可安全重试单批。

CorpAssist · 知识库 nightly rebuild 每晚 02:00 从对象存储拉取增量文档，重建向量索引。API 服务 3 副本常驻；索引 Worker 1–2 副本按需扩容。需要队列解耦「提交」与「执行」。

Python 生态常见两类 Worker 框架。选型时对照团队已有中间件与 async 栈深度——CorpAssist 默认 API 全 async，Worker 侧 IO 密集，ARQ 往往更贴手；若运维已标准化 Celery + RabbitMQ，沿用 Celery 亦可。

框架Broker 常见选型CorpAssist 适用场景
Celery Redis / RabbitMQ 团队已有 RabbitMQ、需复杂路由与 Flower 监控

Celery 任务定义（sync 包装 async 时慎用）

from celery import Celery

Celery 任务默认同步；若内部跑 asyncio 事件循环，需单独进程并避免与 uvicorn 共进程。CorpAssist 新 Worker 优先 ARQ，减少「sync 包 async」的嵌套复杂度。

  • Worker 与 API 共进程抢 CPU

  • Redis/RabbitMQ 持久化队列
  • 任务函数纯 async IO + 批处理（见 性能章）
先想 10 秒：BackgroundTasks 适合 CorpAssist 的哪些场景？

仅适合请求返回后几秒内可完成的副作用，如写审计日志、发 Webhook 通知。超过 30 秒或需重试的任务必须进 task queue。

集成模式优点CorpAssist 注意点
Spring 直发 RabbitMQ，Python 消费 与 Java 定时任务同 broker 队列名、DLQ、消息 TTL 与运维文档一致
Spring HTTP 202 → Python 入 Redis 联调简单，契约即 OpenAPI 需 API Key；高吞吐时 HTTP 成瓶颈
Outbox 模式（Java 写 DB 再发 MQ） 事务与消息一致 Python 只消费，不参与 Java 事务
FAQ：Kafka 还是 RabbitMQ？

CorpAssist 教程默认 RabbitMQ + Redis（ARQ）；若平台已标准化 Kafka，Celery 4.x+ 或独立 consumer 均可，但需团队统一运维。双栈联调优先「消息体 Schema 一致」而非 broker 品牌。

长任务 RAG 索引

RAG indexing 典型阶段：拉文档 → 分块 → Embedding 批处理 → upsert 向量 → 更新元数据索引状态。长任务要支持进度查询、可取消与失败重试。

状态机与存储

状态含义对外 API
queued已入队未执行GET /jobs/{id} → progress 0
runningWorker 执行中progress 1–99
succeeded完成progress 100 + stats
failed可重试错误error_code + request_id
cancelled用户或 BFF 取消Worker 协作式检查 cancel flag
python 
async def index_documents(ctx, job_id: str, doc_ids: list[str], doc_set_version: str) -> dict:
    redis = ctx["redis"]
    key = f"index:idem:{doc_set_version}"
    if await redis.setnx(key, job_id):
        await redis.expire(key, 86400)
    else:
        existing = await redis.get(key)
        return {"job_id": existing.decode(), "status": "duplicate"}

    total = len(doc_ids)
    for i, doc_id in enumerate(doc_ids, start=1):
        if await redis.get(f"job:{job_id}:cancel"):
            return {"job_id": job_id, "status": "cancelled"}
        await index_one(ctx, doc_id)
        await redis.hset(f"job:{job_id}", mapping={"progress": int(i / total * 100)})
    return {"job_id": job_id, "status": "succeeded"}

答辩题：为何不让 Python Worker 直接读 Spring 的数据库？

跨栈共享 DB 耦合 schema 迁移节奏，且 Worker 需 DB 凭证。推荐 BFF 发消息或 HTTP 提交 doc_ids；元数据状态回写经 API 或 MQ 事件，保持 shared contract 在 HTTP/MQ 层。

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix="CORPASSIST_", env_file=".env", extra="ignore")

    env: Literal["local", "staging", "prod"] = "local"
    llm_base_url: HttpUrl

    @model_validator(mode="after")

DTO 映射

DTO mapping 解决三类差异：命名（snake/camel）、可选字段默认值、枚举与日期格式。原则：OpenAPI schema 为真源，避免手写两套模型漂移。

python 
class SearchRequest(BaseModel):
    model_config = ConfigDict(extra="forbid")
    query: str = Field(min_length=1, max_length=2000)
    top_k: int = Field(default=5, ge=1, le=50)
    # JSON 输出 snake_case（默认）
java 
@JsonNaming(PropertyNamingStrategies.SnakeCaseStrategy.class)
public record SearchRequest(
    @NotBlank String query,
    @Min(1) @Max(50) int topK  // Java 字段 camel，JSON snake
) {}

可选字段与 null 语义

双栈常见漂移点：filters: null vs 省略字段 vs {}。CorpAssist 约定：

  • OpenAPI 中 optional 字段用 nullable: true 或 Python Field(default=None) 显式声明。
  • JSON 输出省略 None 值（Pydantic model_dump(exclude_none=True)），Java 侧用 Optional 接收。
  • 禁止 Python 返回 null 字符串或空对象混用 — contract test 固定 3 组样例 JSON 校验。
