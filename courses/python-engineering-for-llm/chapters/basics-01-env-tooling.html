本章先记住 4 件事

pyproject 布局

PEP 621 规定 pyproject.toml 为项目元数据与构建配置的入口。CorpAssist-python 教学骨架（见 demo）典型结构如下：

推荐目录树（src 布局）

corpassist-python/
├── pyproject.toml          # 项目声明 + ruff/mypy 配置
├── uv.lock                 # 锁定传递依赖（提交 Git）
├── .python-version         # 本地 Python 版本提示（可选）
├── README.md
├── src/
│   └── corpassist/
│       ├── __init__.py
│       ├── core/           # 配置、异常、日志
│       ├── llm/            # 网关客户端
│       └── rag/            # 检索与索引（后续章展开）
├── tests/
└── demos/                  # 课程实验（不入 wheel）

src 布局 把可安装包放在 src/corpassist/，避免「仓库根目录与 import 路径冲突」——心智上接近 Java 的 src/main/java。在 IDE 里打开仓库根目录时，不会误把 corpassist.py 同名文件当作包入口。

pyproject.toml 核心字段

[project]
name = "corpassist-python"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = [
  "httpx>=0.27",
  "pydantic>=2.7",
  "pydantic-settings>=2.3",
  "structlog>=24.1",
]

[project.optional-dependencies]
dev = ["ruff>=0.4", "mypy>=1.10", "pytest>=8.2"]

[tool.ruff]
line-length = 100
target-version = "py312"

[tool.mypy]
python_version = "3.12"
strict = true

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
[project]

名称、版本、运行时依赖 — 类比 <dependencies>。

[tool.*]

ruff、mypy、pytest 等工具配置 — 类比 <pluginManagement>。

[build-system]

打包后端（如 hatchling）— 类比 maven-compiler-plugin。

常用 uv 命令

命令作用Maven 近似
uv init --package初始化带包结构的项目archetype:generate
uv add httpx添加依赖并更新 lock改 POM + 解析
uv sync按 lock 安装到 虚拟环境mvn dependency:resolve
uv run pytest在隔离环境中执行命令./mvnw test
先想 10 秒：为什么依赖不要写进 requirements.txt 就完事？

裸 requirements.txt 往往只锁顶层包，缺少传递依赖与工具链版本；pyproject + uv.lock 才是可审计、可复现的「工程声明 + 锁定」组合。

与 Maven 模块对照

CorpAssist 双栈（30 秒） Spring Boot 侧：corpassist-gateway、corpassist-bff 等 Maven 子模块 负责鉴权、聚合与 OpenAPI 契约；Python 侧独立仓库提供 RAG 检索与 LLM 编排 API。两边通过 HTTP + 统一 request_id 协作，而非同一父 POM。

Python 生态更常见 单仓单服务：一个 deployable 镜像对应一个 pyproject，用包内子目录划分职责，而不是 Maven 式的多 <module> 聚合构建。

Spring 模块 ↔ Python 包职责映射

Spring（Java）Python 包/目录职责
corpassist-gateway （仍由 Java 承担） 路由、限流、对外 OpenAPI
corpassist-bff （仍由 Java 承担） 聚合前端契约、JWT 校验
corpassist-rag-api（若独立 Spring 服务） corpassist.rag 文档索引、检索 API
Spring AI / 模型网关客户端 corpassist.llm httpx 调 OpenAI 兼容网关
公共 DTO、异常、配置 corpassist.core Settings、AppError、structlog

Python 侧不复制 Maven 多模块结构，但职责边界应与 Spring 模块表一致，便于联调与 ownership。

反例 vs 推荐

不推荐
  • 在 Spring 仓库里塞 scripts/rag.py 无包结构
  • Python 侧再拆 5 个独立 pip 包，本地联调成本爆炸
  • 包名与 Spring 模块名完全无关，联调文档对不上
推荐
  • 独立 corpassist-python 仓库 + src 布局
  • corpassist.{core,llm,rag} 与模块表一一可述
  • 网关 URL、API Key 走环境变量，不写进 POM/pyproject 默认值

依赖锁定

依赖锁定 解决「开发机 pip 解析出 httpx 0.27.2、CI 解析出 0.28.0」类漂移。CorpAssist 要求：uv.lock 提交 Git，流水线执行 uv sync --frozen（或等价）拒绝擅自升级。

锁定文件里有什么

  • 每个包精确版本与哈希（含传递依赖）
  • 与 pyproject.toml 中范围声明（如 httpx>=0.27）的关系：改范围后需 uv lock 再提交
  • 类比 Maven：dependencyManagement + 可复现的 mvnw 锁定插件版本

Git 应提交 vs 忽略

路径进 Git？原因
uv.lock是保证 CI/生产与本地一致
pyproject.toml是声明意图与工具配置
.venv/否机器本地产物，体积大且不可移植
__pycache__/否字节码缓存
.env（含密钥）否12-factor：密钥只经环境注入

升级依赖的安全节奏

# 1. 本地改 pyproject 或直接 uv add package==x.y.z
uv lock
uv sync
uv run pytest

# 2. 提交 pyproject.toml + uv.lock 同一 PR
# 3. CI 必须 green 才合并

LLM 相关库（httpx、pydantic）小版本也可能改变序列化行为；锁文件 PR 应带回归说明，与 Java 侧 BOM 升级同理。

poetry.lock 与 uv.lock 能混用吗？
CorpAssist 教学统一选 uv；若历史仓库用 Poetry，迁移时一次性切到 uv 并删除旧 lock，避免双 lock 源打架。
CI 里每次 uv sync 会不会很慢？
配合缓存层（wheel 缓存 / uv cache）通常可接受；比「CI 每次 pip 解析出不同树再 mysteriously fail」便宜得多。

先想 10 秒：CorpAssist-python 要不要放进 Spring 父 POM 的 <modules>？

不要。双栈是独立仓库 + HTTP 契约；Python 用 pyproject + uv.lock 自治，职责对齐靠包名与 OpenAPI，而非 Maven reactor 聚合。

本章结论

放在正文与复习之间，便于你先串起全章再做题。每条对应上文一个小节标题。

  • pyproject 布局：pyproject.toml + src 布局是单一真相源；工具配置与依赖声明同文件维护，避免散落 requirements。
  • 与 Maven 模块对照：Python 单仓单服务，用 corpassist.* 子包对齐 Spring 模块职责，而非机械复制多模块 POM。
  • 依赖锁定：uv.lock 进 Git、.venv 不进；升级走 lock + CI 回归，与 Java BOM 同级严肃。
  • CorpAssist-python 骨架：第一版交付可 sync、可 pytest、可映射 core/llm/rag 三包的目录与元数据。

答辩 / 面试口述题

  2. uv.lock 与 pyproject.toml 里 httpx>=0.27 各扮演什么角色？
  3. 若 CI 未提交 lock 就 uv sync，可能出现什么联调问题？
uv 与 poetry 二选一还是都要？
CorpAssist 教学默认 uv（速度快、lock 与 sync 一体）。若团队已标准化 poetry，可保留 poetry.lock，但仍须提交 lock 且 CI 冻结安装；勿混用两套工具链解析同一仓库。
能否把 demos/ 打进生产 wheel？
课程 demos/ 仅本地实验；生产包通过 [tool.hatch.build.targets.wheel] 只打包 src/corpassist。类比 Maven 不把 src/test 打进可执行 jar。
Python 版本与 Spring JDK 如何对齐？
无强制同版本；对齐的是容器基镜像与 CI 矩阵：Java 21 + Python 3.12 各自钉在父 POM / requires-python，避免「CI 3.11、本地 3.12」类型特性不一致。

操作步骤

  1. 对照正文「Spring 模块 ↔ Python 包职责映射」表，为你司实际仓库写 3 行：哪条仍由 Java 承担、哪条归 corpassist.rag / corpassist.llm。

  3. 打开 demo 中 pyproject.toml，找出 requires-python 与 [tool.mypy] python_version 两处版本钉住，类比 Maven 父 POM 的 java.version 写一句对照。
  4. （可选）在 demos/basics-01-env-tooling-lab/ 运行 python scaffold_check.py，核对输出的 Maven 映射与上述纸面答案。

判断练习

  1. 找错：新同事提交 PR 包含 ① uv.lock ② .venv/lib/... ③ pyproject.toml ④ 删除 requires-python 以便「兼容 3.9」。哪些应拒绝合并？

    参考答案

    应拒绝：② 和 ④。.venv 是本地产物不应进 Git；CorpAssist 基线 Python 3.12，删除 requires-python 会破坏工具链与类型特性。①③ 正常应出现在 PR 中。

  2. 权衡：有人建议「依赖只在 pyproject 写精确版本，不要 uv.lock」。在 CorpAssist CI 场景下是否可行？

    参考答案

    不可行。精确版本只锁顶层，传递依赖仍可能漂移；uv.lock 冻结整棵树才是与 Maven BOM + lock 插件同级的可复现构建。pyproject 写范围、lock 写精确版本是推荐分工。
