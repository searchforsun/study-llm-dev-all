profiling

CorpAssist · 索引变慢投诉 运营反馈「昨晚知识库重建从 20 分钟变成 2 小时」。在没有 profiling 数据前，团队猜测是 Embedding 模型变慢——实际是 Python 侧 JSON 解析与重复建连吃掉了 70% CPU。本章先教你怎么用数据说话。

Profiling 分两类：开发/压测用标准库 cProfile；生产用 py-spy 采样，无需改代码、可 attach 到运行中 uvicorn 进程（类似 Java 的 async-profiler）。

开发环境：cProfile + pstats

python 
import cProfile
import pstats
from corpassist.rag.indexer import embed_chunks  # 伪路径

def profile_index_run(chunks: list[str]) -> None:
    pr = cProfile.Profile()
    pr.enable()
    embed_chunks(chunks, batch_size=32)
    pr.disable()
    stats = pstats.Stats(pr).sort_stats("cumulative")
    stats.print_stats(15)  # 看 cumulative 最高的 15 个函数

阅读输出时关注：cumulative（含子调用总耗时）与 ncalls（调用次数）。若某工具函数 ncalls == len(chunks) 且 cumulative 高，优先考虑批处理或缓存。

structlog 耗时与 request_id 关联

python 
import time
import structlog

log = structlog.get_logger()

async def embed_with_timing(chunks: list[str], request_id: str) -> None:
    t0 = time.perf_counter()
    await embed_chunks(chunks, batch_size=32)
    log.info(
        "embed_batch_done",
        request_id=request_id,
        duration_ms=int((time.perf_counter() - t0) * 1000),
        chunk_count=len(chunks),
    )

与 Spring 网关 access log 的 duration_ms 对照，可快速判断瓶颈在 Python 还是下游 LLM。P99 突增时优先 py-spy 采样，而非盲目加副本。

先想 10 秒：async 路由里调用 time.sleep(1) 会怎样？profile 上如何体现？

会阻塞整个 event loop，所有并发请求卡顿；cProfile 可能显示 sleep 占比高，但更应在代码审查阶段禁止同步阻塞，改用 asyncio.sleep 或 run_in_executor。

答辩题：Java async-profiler 与 py-spy 在 CorpAssist 联调中如何配合？

用户投诉慢请求时，用 request_id 定位时间窗口；Java 侧查 BFF 与网关 span，Python 侧 py-spy attach uvicorn worker。两侧日志对齐后再决定优化 BFF 聚合还是 Python Embedding 批大小。

连接池

CorpAssist-python 通过 httpx 调用 LLM 网关与向量库 HTTP 接口。每次请求 async with httpx.AsyncClient() 都会新建连接池，高 QPS 下 TLS 握手与端口耗尽是常见瓶颈——类比 Spring 侧未复用 WebClient Bean。

httpx 客户端生命周期（FastAPI lifespan）
sequenceDiagram
  participant Uvicorn
  participant App as FastAPI lifespan
  participant C as httpx.AsyncClient
  participant GW as LLM Gateway
  Uvicorn->>App: startup
  App->>C: 创建单例 pool
  loop 每个请求
    App->>C: Depends 注入
    C->>GW: 复用连接
  end
  Uvicorn->>App: shutdown
  App->>C: await aclose()

Connection pool 在进程内共享；多 Worker 时每个 worker 各有一份池，需与网关 max_connections 配额对齐。

lifespan 中创建与注入

python 
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator
import httpx
from fastapi import Depends, FastAPI, Request

@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    limits = httpx.Limits(max_connections=100, max_keepalive_connections=20)
    timeout = httpx.Timeout(30.0, connect=5.0)
    app.state.http = httpx.AsyncClient(limits=limits, timeout=timeout)
    yield
    await app.state.http.aclose()

app = FastAPI(lifespan=lifespan)

async def get_http(request: Request) -> httpx.AsyncClient:
    return request.app.state.http
参数建议起点说明
max_connections 50–200 与 Pod 副本数 × worker 数 × 并发乘积对照网关限流
max_keepalive_connections 10–20 空闲连接过多占内存，过少则频繁握手
Timeout connect 5s / read 30s 与 中间件、鉴权与 CORS 及 Spring RestTemplate 一致

多 Worker 连接数估算

uvicorn --workers 4 时，每 worker 各持一份 connection pool。若 max_connections=100，理论峰值 400 条到 LLM 网关——需与平台配额和 semaphore 并发上限一并规划。

部署参数示例值到网关最大连接
2 副本 × 2 workers × max_connections2×2×100400
加 semaphore 限 20/路由—实际并发受 semaphore 约束
FAQ：向量库 SDK 自带连接池，还要 httpx 池吗？

SDK 管的是到向量库的连接；到 LLM 网关仍走 httpx。两套池分别配置上限，避免某一侧慢请求占满全部连接。

批处理 Embedding

单条调用 Embedding API 简单，但索引 10 万 chunk 时 RTT 与序列化开销主导耗时。Batch embedding 将多条文本合并一次请求（受模型 max_batch_size 与 token 上限约束），是提升 throughput 的首要手段。

切块迭代器模式

python 
from itertools import batched  # Py 3.12+

async def embed_all(
    client: httpx.AsyncClient,
    texts: list[str],
    *,
    batch_size: int = 32,
) -> list[list[float]]:
    vectors: list[list[float]] = []
    for batch in batched(texts, batch_size):
        resp = await client.post(
            "/v1/embeddings",
            json={"input": list(batch), "model": "text-embedding-3-small"},
        )
        resp.raise_for_status()
        data = resp.json()["data"]
        vectors.extend(item["embedding"] for item in data)
    return vectors
  • 内存峰值低，适合小 Pod（512Mi–1Gi）
  • 429 限流时单批失败影响面小
  • 适合在线补索引与混部 API 服务
  • 离线全量重建吞吐高
  • 需监控单批响应体大小与超时
  • 建议独立 Worker 进程，避免挤占 API semaphore 配额

429 退避重试（Embedding 可重试）

python 
async def embed_batch_with_retry(client, batch: list[str], max_attempts: int = 3):
    for attempt in range(max_attempts):
        resp = await client.post("/v1/embeddings", json={"input": batch})
        if resp.status_code not in (429, 503):
            resp.raise_for_status()
            return resp.json()
        await asyncio.sleep(min(2 ** attempt, 30))
    raise LlmRateLimitError("embedding batch exhausted retries")

批处理与 重试策略 配合：对 429/503 做指数退避；非幂等场景（带随机噪声的 Chat）勿盲目重试整批，Embedding 通常可安全重试单批。

内存注意

Python 进程常因「同时持有过多大对象」被 OOMKilled：全文列表 + 向量矩阵 + 中间 JSON 副本。Java 有堆上限与 GC 日志；Python 需主动控制 memory footprint 并了解 GC 行为。

CorpAssist 索引内存 checklist

  • 流式读文档：不要 read() 整个 PDF 目录进 list；按文件迭代。
  • 及时释放：每批 embedding 写入向量库后 del batch_vectors，避免累积。
  • 避免巨型中间结构：优先生成器；必要时用 __slots__ 或 dataclass 减小对象头开销。
  • 副本与 Worker：K8s limits.memory 要按「单 worker 峰值 × 1.2」留余量，不是按空载估算。
