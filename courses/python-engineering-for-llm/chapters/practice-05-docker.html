Dockerfile 最佳实践

CorpAssist · 交付镜像 DevOps 要求 Python 服务镜像 <200MB、以非 root 运行、启动 10s 内 health 为 healthy，以便与现有 Spring 微服务同一 K8s 集群部署。

基础镜像选 python:3.12-slim；依赖安装用 uv sync 或 pip + lock。遵循：少层、多缓存、无密钥、非 root 用户。

单阶段精简 Dockerfile（Demo 级）

dockerfile 
FROM python:3.12-slim
WORKDIR /app
RUN pip install --no-cache-dir fastapi uvicorn pydantic
COPY demos/practice-01-fastapi-intro-lab/main.py ./main.py
USER nobody
EXPOSE 8000
HEALTHCHECK CMD curl -f http://localhost:8000/health || exit 1
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
实践原因反例
--no-cache-dir减小 镜像层缓存 wheel 进镜像
USER nobody降低容器逃逸风险root 跑 uvicorn
HEALTHCHECK编排器等待就绪仅依赖端口映射
.dockerignore排除 .venv、__pycache__整仓 COPY . .
镜像内写死配置
  • ENV LLM_API_KEY=sk-xxx 进镜像
  • 换环境需重 build
运行时注入
  • compose / K8s Secret 挂载
  • 同一镜像 dev/staging/prod

  • 无 gcc/uv 于 runtime，攻击面更小

多阶段构建

构建阶段安装编译依赖与 uv，运行阶段只复制 venv 与源码，显著缩小最终镜像。生产 CorpAssist 推荐此模式。

builder + runtime 两阶段

dockerfile 
# --- builder ---
FROM python:3.12-slim AS builder
WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN pip install uv && uv sync --frozen --no-dev

# --- runtime ---
FROM python:3.12-slim AS runtime
WORKDIR /app
COPY --from=builder /app/.venv /app/.venv
COPY src ./src
ENV PATH="/app/.venv/bin:$PATH" \
    CORPASSIST_ENV=production
RUN useradd -m appuser
USER appuser
EXPOSE 8000
CMD ["uvicorn", "corpassist.main:app", "--host", "0.0.0.0", "--port", "8000"]

构建缓存：先 COPY lock 文件再 uv sync，源码变更时不重装依赖。与 Maven「先 pom 后源码」分层同理。

多阶段构建产物流向
flowchart LR
  src["pyproject + uv.lock"]
  builder["builder: uv sync"]
  venv[".venv"]
  runtime["runtime 镜像"]
  src --> builder --> venv
  venv -->|COPY --from=builder| runtime
  code["src/"] --> runtime

runtime 不含 gcc、uv 源码；仅解释器 + venv + 应用包。

先想 10 秒：为何 runtime 阶段不再装 gcc？

编译已在 builder 完成；runtime 只需解释器与 wheel。去掉 build-essential 可减数百 MB，并缩小攻击面。

compose 联调

本地双栈：compose 起 Python 服务（+ 可选 Redis/Postgres），Spring Boot 在 IDE 或 compose 另一 service 中运行，通过服务名 http://corpassist-python:8000 访问。

docker-compose.yml

yaml 
services:
  corpassist-python:
    build:
      context: ../..
      dockerfile: demos/practice-05-docker-lab/Dockerfile
    ports:
      - "8000:8000"
    environment:
      CORPASSIST_ENV: local
      CORPASSIST_INTERNAL_API_KEY: dev-key-change-me
      CORPASSIST_LOG_LEVEL: debug
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 15s
compose 本地双栈
flowchart LR
  dev["开发者"]
  compose["docker compose"]
  py["corpassist-python:8000"]
  spring["Spring Boot IDE :8080"]
  dev --> compose --> py
  spring -->|X-API-Key| py

depends_on: condition: service_healthy 可避免 BFF 启动时连接拒绝。

FAQ：slim 镜像缺 curl，HEALTHCHECK 失败怎么办？

安装 curl 或改用 Python 一行脚本：python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"。Demo Dockerfile 若未装 curl，compose healthcheck 需与镜像能力一致。

compose 网络与服务发现

同一 compose 项目内，服务名即 DNS 主机名。Spring 在宿主机 IDE 运行时用 localhost:8000；Spring 在compose 另一 service 内时用 http://corpassist-python:8000。勿混用，否则出现 Connection refused。

yaml 
# Spring 也在 compose 时（示意）
services:
  corpassist-bff:
    image: corpassist-bff:local
    depends_on:
      corpassist-python:
        condition: service_healthy
    environment:
      CORPASSIST_PYTHON_BASE_URL: http://corpassist-python:8000

章末 Java 消费

Spring BFF 作为 Java 消费方，通过 WebClient 调 Python 服务。联调验收：health 200、带 API Key 的 RAG 请求、响应头 X-Request-Id 回传。

Spring WebClient 调用 Python

java 
// application-local.yml: corpassist.python.base-url=http://localhost:8000
@Service
public class PythonRagClient {
    private final WebClient client;

    public PythonRagClient(WebClient.Builder builder,
                           @Value("${corpassist.python.base-url}") String baseUrl) {
        this.client = builder.baseUrl(baseUrl).build();
    }

    public Mono<RagQueryResponse> query(RagQueryRequest req, String requestId) {
        return client.post()
            .uri("/v1/rag/query")
            .header("X-API-Key", "${corpassist.internal-api-key}")
            .header("X-Request-Id", requestId)
            .bodyValue(req)
            .retrieve()
            .bodyToMono(RagQueryResponse.class);
    }
}

application-local.yml（Spring 侧）

yaml 
corpassist:
  python:
    base-url: http://localhost:8000   # IDE 联调
  internal-api-key: ${CORPASSIST_INTERNAL_API_KEY:dev-key-change-me}
  • DTO 字段 snake_case 与 Python Pydantic 一致，或在 BFF 统一转换（双栈协作规范）。
  • 先 curl /health，再在 Spring 写 @SpringBootTest 集成测试。
  • practice 阶段完成「Compose Python + IDE Spring」；full compose 见 实战：CorpAssist Python 服务 API 交付章。

答辩题：为何 Java 联调前必须先等 compose health 为 healthy？

depends_on 只保证容器启动顺序，不保证 uvicorn 已监听。healthcheck 通过才表示 /health 可达，避免 BFF 集成测试因 Connection refused 误报失败。

操作步骤

判断练习

  1. 镜像 build 时 COPY .env . 把 LLM 密钥打进镜像。应如何改为符合 配置与环境分离？

    参考答案

    删除 Dockerfile 中 COPY .env；在 compose environment 或 K8s Secret 注入 CORPASSIST_LLM_API_KEY。重建镜像不应包含任何环境密钥，同一镜像可用于多环境。

  2. Spring 启动报 Connection refused，但 curl localhost:8000/health 正常。常见配置错误是什么？

    参考答案

    Spring 在容器内仍用 localhost:8000（指向 Java 容器自身）。应改为 compose 服务名 http://corpassist-python:8000，或 Spring 在 host 网络时用 host.docker.internal。
