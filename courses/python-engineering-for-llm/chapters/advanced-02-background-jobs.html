为什么要学本章

全量 RAG 索引、大文件解析与批量 Embedding 动辄数十分钟，不能占用 FastAPI 请求线程。CorpAssist 双栈里，Spring BFF 负责鉴权与编排，Python 侧需要可靠的 task queue 执行 RAG indexing。本章对比 Celery 与 ARQ，说明与 Java MQ 的 桥接，以及 Spring 触发 模式。

学完你能
  • 为 CorpAssist 选型 Celery（成熟生态）或 ARQ（async 原生）
  • 设计 幂等 索引任务与 retry backoff
  • 描述 Spring 经 HTTP/MQ 投递任务到 Python Worker 的序列
  • 把任务状态（pending/running/done）通过 API 或 Webhook 回传 BFF
本章先记住 4 件事
  • HTTP 只提交：POST 返回 job_id，不在路由里 await 全量索引。
  • 幂等键：同一 doc_set_version 重复投递不应双写向量。
  • 可见性：任务日志带 request_id，与 Spring 触发请求一致。
  • 失败可恢复：DLQ + 人工重放，类比 Java @Retryable。

Celery/ARQ 简介

CorpAssist · 知识库 nightly rebuild 每晚 02:00 从对象存储拉取增量文档，重建向量索引。API 服务 3 副本常驻；索引 Worker 1–2 副本按需扩容。需要队列解耦「提交」与「执行」。

Python 生态常见两类 Worker 框架。选型时对照团队已有中间件与 async 栈深度——CorpAssist 默认 API 全 async，Worker 侧 IO 密集，ARQ 往往更贴手；若运维已标准化 Celery + RabbitMQ，沿用 Celery 亦可。

框架Broker 常见选型CorpAssist 适用场景
Celery Redis / RabbitMQ 团队已有 RabbitMQ、需复杂路由与 Flower 监控
ARQ Redis 全 async 栈、任务以 await httpx 为主、部署要轻

ARQ 任务定义（async 友好）

python 
from arq.connections import RedisSettings

async def index_documents(ctx, job_id: str, doc_ids: list[str]) -> dict:
    # ctx["http"] 可在 worker startup 注入 AsyncClient
    indexed = 0
    for doc_id in doc_ids:
        await index_one(ctx, doc_id)
        indexed += 1
    return {"job_id": job_id, "indexed": indexed, "status": "done"}

class WorkerSettings:
    redis_settings = RedisSettings(host="redis", port=6379)
    functions = [index_documents]
    on_startup = startup
    on_shutdown = shutdown

Celery 任务定义（sync 包装 async 时慎用）

python 
from celery import Celery

app = Celery("corpassist", broker="amqp://guest@rabbitmq//")

@app.task(bind=True, max_retries=3, default_retry_delay=60)
def index_documents_celery(self, job_id: str, doc_ids: list[str]) -> dict:
    try:
        return run_async(index_documents_impl(job_id, doc_ids))
    except TransientError as exc:
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)

Celery 任务默认同步；若内部跑 asyncio 事件循环，需单独进程并避免与 uvicorn 共进程。CorpAssist 新 Worker 优先 ARQ，减少「sync 包 async」的嵌套复杂度。

反模式
  • FastAPI 路由 BackgroundTasks 跑 30 分钟索引
  • 无 broker 的 while True 线程（进程崩溃任务丢失）
  • Worker 与 API 共进程抢 CPU
推荐
  • 独立 Deployment：corpassist-worker
  • Redis/RabbitMQ 持久化队列
  • 任务函数纯 async IO + 批处理（见 性能与资源）
先想 10 秒：BackgroundTasks 适合 CorpAssist 的哪些场景？

仅适合请求返回后几秒内可完成的副作用，如写审计日志、发 Webhook 通知。超过 30 秒或需重试的任务必须进 task queue。

与 MQ 集成

企业内常已有 RabbitMQ / Kafka（Spring @RabbitListener）。Python 侧可通过 MQ bridge 消费同一 Topic/Queue，或 Spring 转发为 Redis 任务——关键是契约统一：消息体 JSON Schema 与 OpenAPI DTO 同源。

Spring → MQ → Python Worker
flowchart LR
  BFF["corpassist-bff\nSpring"]
  MQ["RabbitMQ\nindex.jobs"]
  PY["corpassist-worker\nARQ/Celery"]
  VDB["向量库"]
  BFF -->|"发布 IndexJobMessage"| MQ
  MQ --> PY
  PY --> VDB
  PY -->|"状态回调 HTTP"| BFF

BFF 不写 Python 实现细节，只发 job_id、doc_ids、request_id。

消息体契约（Pydantic）

python 
from pydantic import BaseModel, Field

class IndexJobMessage(BaseModel):
    job_id: str
    doc_ids: list[str] = Field(max_length=500)
    request_id: str
    doc_set_version: str  # 幂等键组成部分

    model_config = {"extra": "forbid"}

Java 侧用同名 DTO（Jackson @JsonProperty snake_case）序列化；CI 可做 contract test 比对样例 JSON。

RabbitMQ 消费桥接（示意）

python 
async def on_index_message(body: bytes, arq_pool: ArqRedis) -> None:
    msg = IndexJobMessage.model_validate_json(body)
    await arq_pool.enqueue_job(
        "index_documents",
        msg.job_id,
        msg.doc_ids,
        msg.doc_set_version,
        _job_id=msg.job_id,
    )

桥接层职责单一：校验 JSON → 入 ARQ 队列。复杂索引逻辑仍在 Worker 函数内，便于单测与 structlog 打 request_id。

集成模式优点CorpAssist 注意点
Spring 直发 RabbitMQ，Python 消费 与 Java 定时任务同 broker 队列名、DLQ、消息 TTL 与运维文档一致
Spring HTTP 202 → Python 入 Redis 联调简单，契约即 OpenAPI 需 API Key；高吞吐时 HTTP 成瓶颈
Outbox 模式（Java 写 DB 再发 MQ） 事务与消息一致 Python 只消费，不参与 Java 事务
FAQ：Kafka 还是 RabbitMQ？

CorpAssist 教程默认 RabbitMQ + Redis（ARQ）；若平台已标准化 Kafka，Celery 4.x+ 或独立 consumer 均可，但需团队统一运维。双栈联调优先「消息体 Schema 一致」而非 broker 品牌。

长任务 RAG 索引

RAG indexing 典型阶段：拉文档 → 分块 → Embedding 批处理 → upsert 向量 → 更新元数据索引状态。长任务要支持进度查询、可取消与失败重试。

状态机与存储

状态含义对外 API
queued已入队未执行GET /jobs/{id} → progress 0
runningWorker 执行中progress 1–99
succeeded完成progress 100 + stats
failed可重试错误error_code + request_id
cancelled用户或 BFF 取消Worker 协作式检查 cancel flag
python 
async def index_documents(ctx, job_id: str, doc_ids: list[str], doc_set_version: str) -> dict:
    redis = ctx["redis"]
    key = f"index:idem:{doc_set_version}"
    if await redis.setnx(key, job_id):
        await redis.expire(key, 86400)
    else:
        existing = await redis.get(key)
        return {"job_id": existing.decode(), "status": "duplicate"}

    total = len(doc_ids)
    for i, doc_id in enumerate(doc_ids, start=1):
        if await redis.get(f"job:{job_id}:cancel"):
            return {"job_id": job_id, "status": "cancelled"}
        await index_one(ctx, doc_id)
        await redis.hset(f"job:{job_id}", mapping={"progress": int(i / total * 100)})
    return {"job_id": job_id, "status": "succeeded"}

重试与 DLQ

对 Embedding 429/503 使用 retry backoff；超过 max_retries 写入 DLQ 队列 index.jobs.dlq，运维控制台可重放。日志必须保留 job_id 与 request_id，便于与 Spring 触发记录对齐。

python 
# 反例：失败静默吞掉
try:
    await embed_batch(batch)
except Exception:
    pass  # 向量缺失，检索静默降级
python 
# 推荐：可观测失败 + 有限重试
for attempt in range(3):
    try:
        return await embed_batch(batch)
    except RateLimitError:
        await asyncio.sleep(2 ** attempt)
raise IndexFailedError(code="LLM_RATE_LIMIT", job_id=job_id)
先想 10 秒：为何用 doc_set_version 而不是 job_id 做幂等键？

同一批文档重复触发应合并为一次索引；job_id 每次新建无法去重。version 通常来自 Git commit、ETag 或业务批次号。

Spring 触发

Spring trigger 指 BFF 在管理端操作或定时任务（@Scheduled）中调用 Python「提交索引」API，或直接发 MQ。Python 不反向调用 Spring 启动任务，保持依赖方向清晰。

HTTP 提交 + 轮询（入门联调）

java 
// corpassist-bff — 伪代码
var body = Map.of(
    "doc_ids", docIds,
    "doc_set_version", version,
    "request_id", MDC.get("requestId")
);
var resp = webClient.post()
    .uri(pythonBaseUrl + "/api/v1/index/jobs")
    .bodyValue(body)
    .retrieve()
    .bodyToMono(IndexJobResponse.class)
    .block();
// 前端轮询 GET /api/v1/index/jobs/{job_id}

@Scheduled 定时触发

java 
@Scheduled(cron = "0 0 2 * * *")
public void nightlyIndexTrigger() {
    var version = docCatalogService.currentVersion();
    indexJobClient.submit(new IndexJobCreate(docIds, version, traceId()));
}

定时任务在 Java 侧调度；Python 只负责执行。若需「仅 Python 侧 cron」，用 ARQ cron 作业，但仍应通过 BFF 暴露管理 API 供人工重跑。

FastAPI 提交端点

python 
@router.post("/index/jobs", status_code=202)
async def create_index_job(
    body: IndexJobCreate,
    arq: ArqRedis = Depends(get_arq),
) -> IndexJobAccepted:
    job = await arq.enqueue_job(
        "index_documents",
        body.job_id,
        body.doc_ids,
        body.doc_set_version,
    )
    return IndexJobAccepted(job_id=body.job_id, arq_job_id=job.job_id)
鉴权

触发接口应校验 API Key 或 JWT（见 中间件、鉴权与 CORS），仅允许内网 BFF 调用；返回 202 Accepted 而非 200 OK 表示异步受理。

答辩题：为何不让 Python Worker 直接读 Spring 的数据库？

跨栈共享 DB 耦合 schema 迁移节奏，且 Worker 需 DB 凭证。推荐 BFF 发消息或 HTTP 提交 doc_ids；元数据状态回写经 API 或 MQ 事件，保持 shared contract 在 HTTP/MQ 层。

本章结论

  • Celery/ARQ：长任务独立 Worker；async 栈优先 ARQ + Redis。
  • 与 MQ 集成：消息体 Pydantic 与 Java DTO 同源；Spring 发布、Python 消费。
  • 长任务 RAG 索引：状态机 + 进度 + doc_set_version 幂等 + DLQ。
  • Spring 触发：BFF POST 202 或发 MQ；Python 回调状态，不反向编排 Spring。

  1. （找错）某同事在 FastAPI 路由里 await index_all_docs() 并设置 timeout=600s。违反本章哪 2 条原则？应如何改？

    违反「HTTP 只提交」与「Worker 与 API 分离」。应改为 POST 返回 job_id + 202，索引在 ARQ/Celery Worker 执行；API 路由 timeout 保持短（如 30s）。

  2. （排障）BFF 显示 job 一直 queued，Redis 无消费。请按优先级列出 3 个检查点（Worker 进程、队列名、消息 Schema）。

    参考答案

    ① Worker Deployment 是否运行且连同一 Redis/RabbitMQ；② 入队函数名与 Worker 注册名是否一致；③ 消息 JSON 校验失败是否被桥接层丢弃（查 structlog 与 DLQ）。
