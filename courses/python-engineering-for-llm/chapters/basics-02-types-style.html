本章先记住 4 件事
  • 注解是文档 + 门禁：IDE 补全靠它，mypy strict 靠它，OpenAPI 生成也间接靠 FastAPI 签名。
  • 半结构化用 TypedDict：OpenAI 风格 messages[] 比裸 list[dict] 更可维护。
  • 行为接口用 Protocol：不必继承，只要实现了方法就符合 — 静态侧类似 Java interface。
  • 风格与类型一起拦：ruff 管格式/导入，mypy 管类型，pre-commit 或 CI 统一跑。

typing 常用模式

CorpAssist Python 侧高频类型模式如下；与 demo type_patterns_demo.py 一一对应。

TypedDict：Chat 消息

python 
from typing import Literal, TypedDict

class ChatMessage(TypedDict):
    role: Literal["system", "user", "assistant"]
    content: str

def build_messages(system: str, history: list[ChatMessage]) -> list[ChatMessage]:
    return [{"role": "system", "content": system}, *history]

TypedDict 适合 JSON 形状稳定、字段有限的 DTO；完整校验仍交给 常用库与 HTTP 客户端 中 Pydantic v2 的 BaseModel（HTTP 边界）。

Protocol：Embedding 客户端

python 
from typing import Protocol

class EmbeddingClient(Protocol):
    def embed(self, texts: list[str]) -> list[list[float]]: ...

class MockEmbeddingClient:
    def embed(self, texts: list[str]) -> list[list[float]]:
        return [[0.1, 0.2, 0.3] for _ in texts]

def index_chunks(client: EmbeddingClient, texts: list[str]) -> list[list[float]]:
    return client.embed(texts)

Protocol 描述「必须有 embed 方法」；鸭子类型 在运行时成立，mypy 在静态侧验证。Mock 实现无需声明 implements。

泛型与可选

python 
from collections.abc import Sequence

def truncate_history(messages: list[ChatMessage], max_turns: int) -> list[ChatMessage]:
    if max_turns <= 0:
        return messages[:1]
    system = [m for m in messages if m["role"] == "system"]
    rest = [m for m in messages if m["role"] != "system"]
    return system + rest[-(max_turns * 2):]

def first_item(items: Sequence[str]) -> str:
    return items[0]

Python 3.12+ 使用内置 泛型 list[str]，不再依赖 typing.List。只读参数优先 Sequence，类似 Java 只读视图。

先想 10 秒：为何不全用 Pydantic，还要 TypedDict？

轻量纯函数、测试 fixture、与 Java 侧简单 JSON 互转时 TypedDict 开销更低；进入 HTTP/配置边界再 model_validate，分层更清晰。

ruff/mypy

CorpAssist 在 pyproject.toml 集中配置两类门禁：ruff 负责 lint + format，mypy 负责静态类型。

pyproject 中的工具段

toml 
[tool.ruff]
line-length = 100
target-version = "py312"

[tool.ruff.lint]
select = ["E", "F", "I", "UP"]  # pycodestyle, pyflakes, isort, pyupgrade

[tool.mypy]
python_version = "3.12"
strict = true
warn_return_any = true

本地与 CI 命令

命令作用Java 近似
uv run ruff check .Lint 全仓库Checkstyle
uv run ruff format .统一格式Spotless / google-java-format
uv run mypy src类型检查源码javac + 注解处理器

strict 模式常见报错与修复

mypy 会拦
  • 缺返回类型 / 参数类型
  • role = "admin" 赋给 Literal 联合
  • 隐式 Any 传播
  • 未处理的 Optional
修复习惯
  • 公共 API 全标注；私有函数至少标注参数
  • 用 Literal / Enum 限制枚举字段
  • 第三方缺 stub 时用 types-* 或 targeted # type: ignore
  • if x is None: return 早返回收窄类型

CI 门禁建议

bash 
uv run ruff check .
uv run ruff format --check .
uv run mypy src
uv run pytest

建议在 PR 模板写清：ruff + mypy + pytest 三者皆绿，与 Java 模块「编译 + 单测」同级。AI 生成代码不得跳过 mypy。

ruff 与 mypy 分工（标签页）

拦什么：未使用 import、格式不一致、可自动 upgrade 的旧语法（UP 规则）。

类比：Checkstyle + isort + 部分 formatter。

CorpAssist：uv run ruff check . 与 ruff format --check . 进 CI。

拦什么：类型不匹配、缺注解、Literal 赋值越界、隐式 Any。

类比：javac 类型检查（非运行时）。

CorpAssist：strict = true 覆盖 src/corpassist，DTO 重构前最后一道闸。

能否用 Pyright 替代 mypy？
可以，团队择一即可。CorpAssist 教学默认 mypy + pyproject 配置，与多数 CI 模板兼容；切 Pyright 时保持同一 strict 标准，勿双跑两套除非有明确收益。
TypedDict 与 @dataclass 怎么选？
与 JSON 互转多、字段即 dict 键 → TypedDict；需要方法、不可变对象、非 dict 序列化 → dataclass。LLM messages 通常 TypedDict + Pydantic 边界。

与 Java 类型系统对比

双栈 DTO 对齐（30 秒） Spring BFF 暴露 List<MessageDto>，Python RAG 服务内部用 list[ChatMessage] 组装 prompt。字段命名：Java camelCase JSON 经网关可配置 alias，Python 侧 Pydantic 用 Field(alias="messageId") 对齐（advanced 章展开）。

Java 有而 Python 需注意的差异

  • 无默认编译期：类型错可能跑到生产；mypy 是团队契约，不是语言内置。
  • 可变默认参数：禁用 def f(x=[]) 这类 footgun；Java 无此问题。
  • None 即 Null：函数应显式 | None，避免 Java 式 NPE 的 Python 版。

Java record vs Python TypedDict

Java record
java 
public record ChatMessage(String role, String content) {}
List<ChatMessage> history;
Python TypedDict
python 
class ChatMessage(TypedDict):
    role: Literal["system", "user", "assistant"]
    content: str
history: list[ChatMessage]

record 在运行时自带校验与不可变语义；TypedDict 主要服务静态检查，运行时仍是普通 dict — HTTP 边界仍应交给 Pydantic。

可读性原则

可读性 不是「代码短」，而是下一个 Java 同事扫一眼能改对。CorpAssist Python 约定：

  • 命名：模块/函数/变量 snake_case；类 PascalCase；常量 UPPER_SNAKE — 与 PEP 8 一致，JSON 字段经 Pydantic alias 与 Java camelCase 对齐。
  • 早返回：减少嵌套 if，与 Java Guard Clause 相同。
  • 小函数：truncate_history、build_messages 单一职责；单函数建议 < 40 行。
  • 显式优于隐式：拒绝魔法全局；配置走 Settings（常用库与 HTTP 客户端）。
  • 类型即文档：公共函数必须有 类型注解 与一句 docstring（非 trivial 时）。

反例 vs 推荐（RAG 片段处理）

不推荐
  • def go(d, m, f): ... 无类型无 docstring
  • 200 行「上帝函数」含 HTTP + 切块 + 向量化
  • except: 裸捕获吞掉栈
推荐
  • def truncate_history(messages: list[ChatMessage], max_turns: int) -> list[ChatMessage]
  • 分层：llm/client.py 只谈 HTTP，rag/chunking.py 只谈文本
  • 异常转 AppError（见 异常、日志与配置）

Optional 与类型收窄

python 
def pick_model(explicit: str | None, settings_default: str) -> str:
    if explicit is None:
        return settings_default
    return explicit  # mypy 知 explicit 为 str

Java Optional<T> 在 Python 侧常用 T | None；配合早返回，mypy 可自动收窄，减少「忘判空」类 bug。

判断练习

  1. 权衡：团队提议「CI 只跑 ruff，mypy 太慢先不上」。作为 CorpAssist 双栈项目，你是否支持？说明一条最强理由。

    参考答案

    不支持。LLM DTO 与网关 JSON 字段频繁变更，缺 mypy 时类型错只能运行时或联调暴露；Java 侧有编译期，Python 侧应对齐用 strict mypy 保契约。可优化 scope（先 src/corpassist）而非取消。

  2. 找错：PR 中 ① 删除所有返回类型注解以「加快开发」② 为第三方库全局 # type: ignore ③ 新增 Protocol 描述 EmbeddingClient ④ 在 HTTP 路由参数用裸 dict。哪些应拒绝？

    参考答案

    应拒绝：① ② ④。公共 API 必须标注；滥用 ignore 掩盖真实类型错误；HTTP 边界应 Pydantic/BaseModel 而非裸 dict。③ 是推荐做法。
