为什么要学本章

CorpAssist RAG 服务的核心外向依赖是模型网关 HTTP API 与可校验配置。Java 侧你可能用 RestTemplate/WebClient + @ConfigurationProperties；Python 侧事实标准是 httpx + Pydantic v2 + pydantic-settings。本章打通「读配置 → 发请求 → 解析 DTO」闭环。

  • 定义 BaseModel 请求/响应并在边界 model_validate

本章先记住 4 件事
  • HTTP 客户端选 httpx：API 与 requests 相近，原生支持 async 与 MockTransport。
  • 边界数据用 Pydantic：入参/outbound JSON 一律 BaseModel，内部可再转 TypedDict。
  • 配置是 typed Settings：禁止在代码里硬编码网关 URL；12-factor 环境注入。
  • LLM 调用要可测：MockTransport + fixture，CI 禁止直连真实模型。

httpx

httpx 是 CorpAssist Python 调 Spring 网关或直连 DashScope/OpenAI 兼容端点的默认客户端。对比 Java WebClient：同样需显式超时、headers 与连接复用。

同步客户端最小示例

import httpx

def fetch_health(base_url: str) -> dict[str, object]:
    with httpx.Client(base_url=base_url, timeout=5.0) as client:
        resp = client.get("/health")
        resp.raise_for_status()
        return resp.json()

生产级 Client 参数

参数建议说明
timeouthttpx.Timeout(connect=5, read=60, write=10, pool=5)Chat 读超时通常 > 普通 REST
base_urlSettings 注入避免 URL 拼接错误
headersAuthorization: Bearer …与 Spring AI 相同 Bearer 约定
limitshttpx.Limits(max_connections=100)类比连接池（advanced 章展开）

异步预告

异步客户端 httpx.AsyncClient 将在 asyncio 与并发 与 FastAPI 路由配合；本章先用同步 Client 理解请求/响应模型。

不推荐
  • 每次请求 httpx.post(url) 无 Client 复用
  • 无 timeout，LLM 挂死占满 worker
  • 裸 requests 且无 mock 策略
推荐
  • 应用 lifespan 创建/关闭单例 Client
  • 区分 connect/read 超时
  • 测试用 MockTransport
先想 10 秒：为何 LLM 读超时要比普通 REST 长？

Token 生成是流式/长耗时计算；过短 read timeout 会在模型仍正常生成时误杀。应对：合理 read 超时 + 可观测 latency + 重试策略 仅用于幂等/可恢复错误。

pydantic v2

Pydantic v2 负责 CorpAssist 所有 HTTP JSON 边界的解析与校验，对齐 Java Bean Validation + Jackson。

Chat 请求/响应模型

from pydantic import BaseModel, Field

class ChatRequest(BaseModel):
    model: str
    messages: list[dict[str, str]]
    temperature: float = 0.3
    max_tokens: int = Field(default=512, ge=1)

class ChatChoice(BaseModel):
    index: int
    message: dict[str, str]
    finish_reason: str | None = None

class ChatResponse(BaseModel):
    id: str
    choices: list[ChatChoice]
    usage: dict[str, int] | None = None

model_validate 与 model_dump

raw = resp.json()
parsed = ChatResponse.model_validate(raw)
outbound = ChatRequest(model="qwen-plus", messages=[...]).model_dump()

入站：model_validate 抛 ValidationError → 映射 400（practice 章 exception_handler）。出站：model_dump 给 httpx JSON body。

v2 常用配置

model_validate 解析网关 JSON，失败即 400。

出站

与 TypedDict

内部仍可用 TypedDict；HTTP 边界统一 BaseModel。

结构化配置

CorpAssist 网关 URL、模型名、日志级别不应写死在源码。使用 pydantic-settings 定义 Settings 模型，类比 Spring @ConfigurationProperties(prefix="corpassist")。

Settings 示例

from pydantic import Field, SecretStr
from pydantic_settings import BaseSettings, SettingsConfigDict

class CorpAssistSettings(BaseSettings):
    model_config = SettingsConfigDict(
        env_prefix="CORPASSIST_",
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )

    llm_base_url: str = Field(default="http://localhost:8080/v1")
    llm_api_key: SecretStr = Field(default=SecretStr("dev-key"))
    default_model: str = "qwen-plus"
    log_level: str = "INFO"

环境变量映射

环境变量字段说明
CORPASSIST_LLM_BASE_URLllm_base_urlSpring 网关或 DashScope 兼容 base
CORPASSIST_LLM_API_KEYllm_api_keySecretStr 日志脱敏
CORPASSIST_DEFAULT_MODELdefault_model默认 chat 模型
CORPASSIST_LOG_LEVELlog_level与 Logback 级别对齐

本地开发可复制 .env.example；真实密钥仅经 K8s Secret / CI 变量注入 — 见 12-factor 配置。

调用 LLM API

CorpAssist S1 制度问答（45 秒） 用户经 Java BFF 提问 → BFF 转发 Python RAG 服务 → Python 组装 prompt 后调用 OpenAI 兼容 网关 POST /v1/chat/completions。Python 需解析 choices[0].message.content 与 usage 供成本统计。

完整调用函数

def chat_completion(
    client: httpx.Client,
    base_url: str,
    api_key: str,
    req: ChatRequest,
) -> ChatResponse:
    url = f"{base_url.rstrip('/')}/chat/completions"
    headers = {"Authorization": f"Bearer {api_key}"}
    resp = client.post(url, json=req.model_dump(), headers=headers, timeout=30.0)
    resp.raise_for_status()
    return ChatResponse.model_validate(resp.json())

MockTransport 测试

def mock_transport(request: httpx.Request) -> httpx.Response:
    body = {
        "id": "mock-1",
        "choices": [{"index": 0, "message": {"role": "assistant", "content": "（mock）答复"}, "finish_reason": "stop"}],
        "usage": {"prompt_tokens": 156, "completion_tokens": 42, "total_tokens": 198},
    }
    return httpx.Response(200, json=body)

transport = httpx.MockTransport(mock_transport)
with httpx.Client(transport=transport, base_url="http://mock/v1") as client:
    result = chat_completion(client, "http://mock/v1", "demo-key", req)

错误与重试（概要）

HTTP含义CorpAssist 策略
401/403密钥/权限不重试；告警配置
429限流有限退避重试（重试策略）
502/503上游不可用短重试 + 映射 LlmUpstreamError
超时read timeout记录 latency；是否重试看幂等性

为何不用 requests？
httpx 支持 async、HTTP/2 与 MockTransport，与 FastAPI 异步栈一致；requests 无官方 async，测试 mock 也更繁琐。
SecretStr 日志会泄露吗？
Pydantic SecretStr 默认不在 repr 中暴露；仍禁止把 API Key 打进 info 日志，只记录「已配置」类元数据。

操作步骤

  1. 阅读「生产级 Client 参数」表中 timeout 一行，写出 connect 与 read 超时各自保护哪类失败。
  2. 对照「环境变量映射」表，写出 CORPASSIST_LLM_BASE_URL 在本地指向 Spring 网关时的示例值。
  3. 阅读「Chat 请求/响应模型」代码块，说明 Field(ge=1) 校验失败在 HTTP 层应对应什么状态码（概念上 400）。
  4. 打开 demo mock_llm_client.py，核对 mock_transport 返回的 usage.total_tokens 是否等于 prompt + completion；可选运行 python mock_llm_client.py 验证无真实网络。

判断练习

  1. 排障顺序：联调 Chat 502。同事计划 ① 加大 read timeout 到 300s ② 在代码写死生产 API Key ③ 用 curl 打网关 ④ 检查 CORPASSIST_LLM_BASE_URL。请排序最优排查步骤。

    参考答案

    推荐顺序：④ → ③ → ①。先确认 Settings 注入的 base URL 是否正确，再用 curl 验证网关本身；最后再调 timeout。② 写死密钥违反 12-factor，应拒绝。

  2. 找错：下列哪条违反「调用 LLM API」规范？① 每次请求新建 Client 无 timeout ② 用 MockTransport 做单元测试 ③ 在源码写死生产 API Key ④ 对 429 做有限退避重试

    参考答案

    违反：① 和 ③。应复用 Client 并配置 Timeout；密钥走 Settings/环境变量。②④ 是推荐做法。
