语义化版本

CorpAssist · 双栈发版窗口 Python 组准备发 0.2.0：新增 RAG search 可选字段 filters。Java BFF 需在同一个 release train 内升级 openapi-generator 客户端；若 Python 误 bump 到 1.0.0 却未删旧路径，前端会收到两套不兼容响应。

Semver 格式 MAJOR.MINOR.PATCH，与 Maven ${revision} 策略类似。CorpAssist-python 服务版本写在 pyproject.toml 与 Docker 标签，Java BFF 依赖的 OpenAPI info.version 应同步 bump。

变更类型版本位CorpAssist 示例
修复索引 bug、无 API 变化 PATCH 0.1.0 → 0.1.1
新增可选 API 字段、向后兼容 MINOR 0.1.1 → 0.2.0
删除/重命名 API 路径 MAJOR 0.2.0 → 1.0.0，需 Java 侧同步发版

pyproject 版本与 __version__ 同步

toml 
[project]
name = "corpassist-python"
version = "0.2.0"
description = "CorpAssist Python RAG 服务"

[tool.hatch.version]
path = "src/corpassist/__init__.py"

Version bump 应伴随 CHANGELOG 条目与 Git tag v0.2.0，CI 在 tag 推送时触发 release 流水线。

答辩题：预发布版本 0.2.0-rc.1 何时用？

给 Java 联调或 staging 验证用，不应进生产私服 default 通道。CorpAssist 约定：rc 仅 internal index；正式 tag 去掉后缀并同步 Docker :0.2.0 与 OpenAPI info.version。

先想 10 秒：仅修复 Worker 内存泄漏、HTTP JSON 完全不变，应 bump 哪一位？

PATCH（如 0.2.0 → 0.2.1）。BFF 无需改客户端；平台组只需滚动 Worker 镜像。

内部 PyPI

企业私服（DevPI、Artifactory PyPI、Nexus）托管内部 wheel。共享库如 corpassist-schemas（Pydantic 模型）可独立发包，供 Python 服务与代码生成工具使用——类比 Maven 私服上的 corpassist-api jar。

构建物从 CI 到运行时
flowchart LR
  git["Git tag v0.2.0"]
  ci["CI: uv build"]
  pypi["internal PyPI"]
  docker["Docker 镜像"]
  k8s["K8s 部署"]
  git --> ci --> pypi
  ci --> docker --> k8s
  k8s -->|"uv sync --index"| pypi

生产镜像可只 COPY wheel 或直接 uv sync --frozen 从 lock 安装；私服用于跨项目复用 schemas 包。

uv 配置私服 index

toml 
[[tool.uv.index]]
name = "corp"
url = "https://pypi.corp.example/simple/"
default = true

[tool.uv.sources]
corpassist-schemas = { index = "corp" }
风险做法
  • 从 Git URL 直接 pip install 无版本 pin
  • 私服凭证写进 pyproject 提交 Git
  • 本地 pip install -e 与生产 wheel 行为不一致
推荐
  • CI 用 UV_INDEX_CORP_PASSWORD 环境变量
  • lock 文件锁定私服包精确版本
  • 发布前 uv build 本地验证 wheel 可安装

本地 build 与 publish 命令

bash 
uv build                          # dist/corpassist_python-0.2.0-py3-none-any.whl
uv publish --index corp             # CI 中执行；本地需 UV_PUBLISH_TOKEN
pip install corpassist-schemas==0.2.0 --index-url https://pypi.corp.example/simple/

共享 schemas 包独立 semver：仅改 Pydantic 模型时 bump schemas MINOR，API 服务依赖 corpassist-schemas>=0.2,<0.3，与 Maven BOM 对齐。

FAQ：wheel 与 Docker 镜像都要发吗？

API/Worker 运行时通常以 Docker 为主；wheel 用于跨仓库复用 schemas 与 CLI 工具。两者 version tag 应同源 Git commit，避免「镜像 0.2.0 内嵌 schemas 0.1.9」漂移。

配置分环境

镜像构建一次，多环境运行靠环境变量——与 Spring application-{profile}.yml 同构。CorpAssist 使用 CORPASSIST_* 前缀与 Settings 模型 校验。

bash 
CORPASSIST_ENV=local
CORPASSIST_LLM_BASE_URL=http://mock-llm:8080
CORPASSIST_LOG_LEVEL=DEBUG
CORPASSIST_REDIS_URL=redis://redis:6379/0
bash 
CORPASSIST_ENV=prod
CORPASSIST_LLM_BASE_URL=https://llm-gateway.corp.example
CORPASSIST_LOG_LEVEL=INFO
# 密钥来自 K8s Secret，不进镜像

Settings 分环境校验

python 
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix="CORPASSIST_", env_file=".env", extra="ignore")

    env: Literal["local", "staging", "prod"] = "local"
    llm_base_url: HttpUrl
    log_level: str = "INFO"

    @model_validator(mode="after")
    def prod_requires_https(self) -> Self:
        if self.env == "prod" and str(self.llm_base_url).startswith("http://"):
            raise ValueError("prod 必须 https")
        return self
先想 10 秒：为何不把 .env.production 打进 Docker 镜像？

违反 配置与环境分离；同一镜像无法多环境复用，且易泄露密钥。应使用 K8s ConfigMap/Secret 或 CI 注入。

CI 钩子预览

CI pipeline 在 merge 与 tag 两阶段职责不同：PR 验证质量，tag 产出 制品。

PR 与 tag 流水线分叉
flowchart TB
  pr["PR push"]
  tag["tag v0.2.0"]
  verify["verify: lint test lock"]
  release["release: build publish docker"]
  pr --> verify
  tag --> verify --> release

release job 仅在 tag 且 verify 绿时运行；PR 不得 publish 到生产私服。

典型阶段（GitHub Actions / GitLab CI）

  1. lint — uv run ruff check + mypy
  2. test — uv run pytest（Mock LLM，见 测试策略）
  3. lock 校验 — uv lock --check 或 uv sync --frozen
  4. build — uv build 生成 wheel；Docker build push
  5. release — tag v* 时上传私服 + 写 CHANGELOG
yaml 
# .github/workflows/ci.yml 片段
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install uv && uv sync --frozen
      - run: uv run ruff check && uv run mypy src
      - run: uv run pytest -q
  release:
    if: startsWith(github.ref, 'refs/tags/v')
    needs: verify
    steps:
      - run: uv build && uv publish --index corp
CI 失败常见原因
  • 改了 pyproject 未提交 uv.lock
  • semver 与 __version__ 不一致
  • 集成测试误连真实 LLM（缺 Mock）
  • tag release 未等 verify job 绿
答辩题：Python 服务为何仍要发 wheel 而不只发 Docker 镜像？

corpassist-schemas 等库供多服务复用；wheel + 私服让 Worker/API 共享同一 DTO 版本。镜像负责运行时，wheel 负责可复用 制品 与 pyproject 元数据。

本章结论

  • 语义化版本：PATCH/MINOR/MAJOR 与 OpenAPI、Java 发版联动。
  • 内部 PyPI：wheel + lock + 私服 index；凭证仅 CI 环境变量。
  • 配置分环境：CORPASSIST_* + Settings 校验；镜像无环境专属文件。
  • CI 钩子：PR 跑 lint/test/lock；tag 触发 build 与 publish。
