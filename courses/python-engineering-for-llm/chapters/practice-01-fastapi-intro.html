与 Spring MVC 对照

CorpAssist · Spring BFF 调 Python Java 网关/BFF 通过 HTTP JSON 调用 Python 侧 RAG 服务。团队多数成员熟悉 Spring MVC；FastAPI 路由与注解一一对应，降低双栈协作沟通成本。

先想 10 秒：为何 BFF 只依赖 OpenAPI 路径而不 import Python 包？

双栈进程隔离；契约是 HTTP JSON + OpenAPI，不是共享 JVM classpath。Python 内部重构包名不影响 BFF，只要路径与 schema 不变。

单文件堆路由
  • 所有 @app.post 写在 main.py
  • 无 tags，OpenAPI 难以分组
  • 与 Maven 多模块边界不对齐
APIRouter 分模块
  • api/rag.py、api/health.py 各一 router
  • tags=["rag"] 便于 /docs 分组
  • 对应 corpassist.rag 包职责

Pydantic 模型

BaseModel 承担 Java DTO + Bean Validation 双重角色：字段类型、长度、范围在入站时校验，出站时序列化为 JSON。CorpAssist RAG 查询契约如下。

RagQuery 请求/响应模型

python 
from pydantic import BaseModel, Field, ConfigDict

class RagQueryRequest(BaseModel):
    model_config = ConfigDict(str_strip_whitespace=True)
    query: str = Field(min_length=1, max_length=2000, description="用户问题")
    top_k: int = Field(default=5, ge=1, le=20)

class RagQueryResponse(BaseModel):
    answer: str
    sources: list[str]
    request_id: str | None = None

与 Spring 侧 JSON 对齐时注意 snake_case：Python 字段 top_k 在 OpenAPI 中保持一致；若 BFF 用 camelCase，在网关层转换而非 Python 内混用两种风格。

FAQ：校验失败返回什么？

FastAPI 默认 HTTP 422 + detail 数组，字段路径与错误类型清晰。CorpAssist 可在全局 exception_handler 中映射为与 Java 统一的错误码（见 异常、日志与配置）。

与 OpenAPI 对齐

Field(description=...) 会出现在 /docs 与导出的 schema 中，便于 Java 团队对照生成 DTO。修改 Pydantic 模型后应跑契约快照测试（见 测试策略）。

依赖注入

Depends 类似 Spring 的构造器/方法注入：FastAPI 在调用路由前解析依赖图。常用模式：Settings 单例、按请求注入客户端、鉴权占位。

Settings 与工厂函数

python 
from functools import lru_cache
from fastapi import Depends
from corpassist.config import CorpAssistSettings

@lru_cache
def get_settings() -> CorpAssistSettings:
    return CorpAssistSettings()

def get_llm_client(settings: CorpAssistSettings = Depends(get_settings)):
    # practice-02 改为 httpx.AsyncClient + lifespan
    return {"base_url": str(settings.llm_base_url)}

路由中使用 Depends

python 
from fastapi import APIRouter, Depends

router = APIRouter(prefix="/v1/rag", tags=["rag"])

@router.post("/query", response_model=RagQueryResponse)
async def rag_query(
    body: RagQueryRequest,
    settings: CorpAssistSettings = Depends(get_settings),
) -> RagQueryResponse:
    return RagQueryResponse(
        answer=f"[demo] 收到: {body.query[:80]}",
        sources=["demo-doc-1"],
    )

lifespan 上下文在应用启动时创建 httpx.AsyncClient 连接池，关闭时释放；比在每个 handler 内 Depends new 客户端更省资源，详见 asyncio 与并发。

先想 10 秒：为何不在 handler 里直接 CorpAssistSettings()？

每次请求重复读环境并校验浪费 CPU；@lru_cache 或 lifespan 保证进程内单例。测试时可通过 app.dependency_overrides[get_settings] 注入 mock，类似 Spring @MockBean。

健康检查

Spring Cloud Gateway 与 K8s 探针依赖稳定的 /health。CorpAssist 约定：返回 200、JSON 含 status 与 service，不调用模型网关（避免探活产生费用）。

健康检查实现

python 
@app.get("/health")
async def health(settings: CorpAssistSettings = Depends(get_settings)) -> dict[str, str]:
    # 只验证配置可读，不 ping LLM
    _ = settings.service_name
    return {"status": "ok", "service": settings.service_name}
  • liveness：进程存活，默认 /health 即可。
  • readiness：后续可加向量库连通性；本章保持轻量。
  • 与 Java /actuator/health 一同出现在监控面板，字段命名保持一致。
  • 路径 /health，JSON status + service
  • 不调 LLM；Compose/K8s 探针默认 GET
  • 与 Demo main.py 返回值一致
  • 路径 /actuator/health，{"status":"UP"}
  • Gateway 可同时配置两后端探活
  • 字段名不同，监控面板需分别映射

FAQ：OpenAPI 文档要对外暴露吗？

内网 dev/staging 可开 /docs；生产通常由网关限制或关闭，改导出静态 OpenAPI 给 Java 团队（见 实战：CorpAssist Python 服务 API）。

答辩题：为何 CorpAssist 把 /health 与 /v1/rag/query 分在不同 router？

探活路由不挂载 API Key 与 JWT，避免 K8s 探针因鉴权失败误杀 Pod；业务路由走完整中间件链（见 中间件、鉴权与 CORS）。

动手练习

对照正文「Spring MVC ↔ FastAPI 映射」表与 Demo main.py，本地启动服务并完成 /health 与 POST /v1/rag/query 的命令级验收。

操作步骤

  1. 打开正文「Spring MVC ↔ FastAPI 映射」表，在笔记中为 @RequestBody、@Autowired 各写一行 FastAPI 等价写法。
  2. 阅读「RagQuery 请求/响应模型」代码块，给 query 字段增加 max_length=500，用 /docs 发送超长字符串观察 422 响应。
  3. 在 Demo 的 rag_query handler 签名中加入 Depends(get_settings) 占位（可先返回固定 dict），理解依赖解析顺序。
  4. 进入 demos/practice-01-fastapi-intro-lab/，uvicorn main:app --reload --port 8000，curl -s localhost:8000/health 与 curl -s -X POST localhost:8000/v1/rag/query -H "Content-Type: application/json" -d "{\"query\":\"hello\"}"。

判断练习

  1. 同事把全部路由写在 main.py 且 handler 参数用 body: dict。违反本章哪两条约定？应如何改？

    参考答案

    违反「APIRouter 分模块」与「Pydantic 模型校验」。应拆 api/rag.py router，请求体改为 RagQueryRequest，并设置 response_model=RagQueryResponse。

  2. 探活 handler 内调用 await llm_client.chat(...) 验证网关可达。为何不符合 CorpAssist 健康检查约定？

    参考答案

    探活高频触发会产生 LLM 费用与延迟抖动；网关故障会导致 Pod 被误杀。健康检查应只验证进程与配置可读，外部依赖检查放到 readiness 或独立诊断端点。
