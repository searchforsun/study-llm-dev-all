办公场景自动化四象限

企业办公场景可按「决策复杂度」和「风险等级」分为四类：

风险 \ 复杂度低（查/读）高（写/改）
低风险全自动：查考勤余额、查会议日程自动执行 + 日志：修改个人签名、更新日报
高风险自动生成 → 人工确认：开离职证明、生成报销单人工审批：对外付款、发布公告、批量改权限

CorpAssist 的 automatable task 通常落在前两象限。第三象限需配套 human-in-the-loop（见 多工具协作与编排）。第四象限必须走工单审批系统。

不推荐
  • 把审批功能直接写在 Agent Prompt 里
  • 所有操作全自动，不区分风险等级
  • 无 audit trail，不知道 Agent 执行了什么
推荐
  • 审批规则写在 Graph 节点中，而非系统提示词
  • 按风险等级划定 automation scope
  • 全量操作日志写入审计系统（见 可靠性工程）
CorpAssist · 发票处理场景 员工拍照上传发票 → Agent 识别内容（全自动）→ 匹配报销政策（全自动）→ 生成报销单（全自动）→ 金额超过 5000 元触发审批节点 → 财务审核（人工）→ 推送审批结果。这个流程横跨三个象限：识别匹配全自动、生成草案全自动、审批人工。
办公场景自动化分级流程
flowchart TB
      Input[用户输入任务] --> Classifier{风险等级}
      Classifier -->|低风险| Auto[全自动执行]
      Classifier -->|高风险| Draft[生成草案]
      Draft --> Approval{人工审批}
      Approval -->|通过| Execute[执行操作]
      Approval -->|驳回| Reject[返回修改建议]
      Auto --> Log[写入审计日志]
      Execute --> Log
      Reject --> Log

自动化等级决定了 Agent 流程中是否需要 human collab 节点。

人机协同与审批 scope

approval scope 定义了哪些操作必须停下等人。在 CorpAssist 实践中，审批 scope 通常配置在 YAML 中而非写死在代码里：

          operator: "not_in"
          value: ["company.com"]
    - action: "payment"
      risk: "high"
      conditions:
        - field: "amount"
          operator: ">"
          value: 5000
    - action: "leave_apply"
      risk: "medium"
      conditions:
        - field: "days"
          operator: ">"
          value: 3

Python 侧的审批配置读取同样遵循分层设计：

python
# approval_config.py
from typing import Dict, List
import yaml

class ApprovalRule:
    def __init__(self, action: str, risk: str, conditions: List[Dict]):
        self.action = action
        self.risk = risk
        self.conditions = conditions

    def evaluate(self, context: Dict) -> bool:
        """返回 True 表示需要审批"""
        for cond in self.conditions:
            field_value = context.get(cond["field"])
            op = cond["operator"]
            target = cond["value"]
            if op == ">" and field_value > target:
                return True
            elif op == "not_in" and field_value not in target:
                return True
        return False

def load_approval_rules(path: str = "corp-assist-approval.yaml"):
    with open(path) as f:
        config = yaml.safe_load(f)
    return [ApprovalRule(**r) for r in config["approval"]["rules"]]

    }
}

from langgraph.graph import StateGraph, END
from typing import TypedDict, Optional

class InvoiceState(TypedDict):
    image_path: str
    ocr_result: Optional[dict]
    category: Optional[str]
    amount: float
    draft_id: Optional[str]
    approval_status: str

def ocr_node(state: InvoiceState) -> dict:
    # 调用 OCR MCP 工具提取发票信息
    result = mcp_tools.ocr.invoke({"image": state["image_path"]})
    return {"ocr_result": result, "amount": result.get("amount", 0)}

def categorize_node(state: InvoiceState) -> dict:
    # 自动匹配费用科目
    cat = erp_service.match_category(state["ocr_result"])
    return {"category": cat}

def create_draft(state: InvoiceState) -> dict:
    draft_id = finance_service.create_reimbursement_draft(
        amount=state["amount"], category=state["category"]
    )
    return {"draft_id": draft_id}

def approval_router(state: InvoiceState) -> str:
    if state["amount"] > 5000:
        return "approval"
    return "finish"

graph = StateGraph(InvoiceState)
graph.add_node("ocr", ocr_node)
graph.add_node("categorize", categorize_node)
graph.add_node("create_draft", create_draft)
graph.add_node("approval", human_approval_node)
graph.set_entry_point("ocr")
graph.add_edge("ocr", "categorize")
graph.add_edge("categorize", "create_draft")
graph.add_conditional_edges("create_draft", approval_router)
graph.add_edge("approval", END)
graph.add_edge("finish", END)

场景二：邮件自动生成

Agent 根据用户口述要点生成邮件草稿，对外发送需审批：

// EmailAgentService.java
@Service
public class EmailAgentService {

    private final DashScopeChatModel chatModel;
    private final EmailMcpTool emailTool;
    private final ApprovalService approvalService;

    public String draftEmail(String userInput) {
        // Agent 生成邮件草稿
        String prompt = "根据以下要点生成正式邮件：\n" + userInput;
        ChatResponse response = chatModel.call(
            new Prompt(List.of(new UserMessage(prompt))));
        String draft = response.getResult().getOutput().getContent();

        // 判断是否需要审批
        if (emailTool.isExternal(draft)) {
            approvalService.createTask(draft, "email_approval");
            return "草稿已生成，等待审批后发送";
        }
        return emailTool.send(draft);
    }
}

常见问题（FAQ）

全自动和半自动的划分标准是什么？
核心看两个维度：操作是否可逆（删文件不可逆 vs 发草稿可撤销）和 金额/权限影响。不可逆 + 高影响 = 必须人工。
审批节点放在 Agent 代码里还是配置中心？
推荐配置中心（Apollo/Nacos）。审批规则经常随业务变化，硬编码意味着每次改规则都要走发布流程。
低风险操作是否完全不需要审计？
需要。所有 Agent 操作都应记录 audit trail，低风险操作至少记录「谁 + 何时 + 做了什么」。这是合规要求。

ReAct 思考-行动-观察

think act observe 是 Agent 的最小工作单元。每一步，模型先思考当前状态和下一步行动，调用工具得到观察结果，然后决定是否继续或结束。

ReAct 循环：思考 → 行动 → 观察
flowchart LR
      Start[用户输入] --> Think{思考}
      Think -->|规划下一步| Act[行动\n调用 MCP 工具]
      Act --> Observe[观察\n工具返回结果]
      Observe -->|未完成| Think
      Observe -->|已完成| End[输出结果]
      Note1[步数预算 max_steps=10] -.-> Think
      Note2[超时终止] -.-> Act

ReAct Loop 的核心在于 Observe 之后的反馈决策。

CorpAssist · 请假审批 Agent 用户说「请下周一到周三的年假」。Agent 思考：需要查余额、算天数、判断审批策略。行动1：调用日历工具查假期余额 = 15 天。观察1：余额充足。行动2：调用日历工具查下周是否有会议 = 无冲突。观察2：可请假。思考后输出请假草稿供审批。总共 4 步，在预算之内。

Python：LangGraph 实现 ReAct

python
# react_agent.py
from langgraph.graph import StateGraph, END
from typing import TypedDict, Optional

class AgentState(TypedDict):
    input: str
    thoughts: list[str]
    actions: list[str]
    observations: list[str]
    step: int
    max_steps: int
    output: Optional[str]

def think_node(state: AgentState) -> dict:
    # LLM 推理：根据当前观察决定下一步
    prompt = f"当前任务：{state['input']}\n"
    prompt += f"历史观察：{state['observations'][-1] if state['observations'] else '无'}\n"
    prompt += f"已执行步骤：{state['step']}/{state['max_steps']}"
    thought = llm.invoke(prompt)  # 调用大模型
    return {"thoughts": state["thoughts"] + [thought], "step": state["step"] + 1}

def act_node(state: AgentState) -> dict:
    # 从 thought 中解析工具调用
    tool_call = parse_tool_call(state["thoughts"][-1])
    result = mcp_tools.call(tool_call["tool"], tool_call["args"])
    return {"actions": state["actions"] + [str(tool_call)],
            "observations": state["observations"] + [result]}

def should_continue(state: AgentState) -> str:
    if state["step"] >= state["max_steps"]:
        return "end"  # 超步数预算，终止
    if "已完成" in state["observations"][-1]:
        return "end"
    return "continue"

graph = StateGraph(AgentState)
graph.add_node("think", think_node)
graph.add_node("act", act_node)
graph.set_entry_point("think")
graph.add_edge("think", "act")
graph.add_conditional_edges("act", should_continue, {
    "continue": "think",
    "end": END
})

Java：Spring AI Alibaba 实现 ReAct

java
// ReactAgentService.java
@Service
public class ReactAgentService {

    private final DashScopeChatModel chatModel;
    private final List<ToolCallback> tools;

    public ReactAgentService(DashScopeChatModel chatModel,
                             List<ToolCallback> tools) {
        this.chatModel = chatModel;
        this.tools = tools;
    }

    public String execute(String userInput, int maxSteps) {
        var messages = new ArrayList<AbstractMessage>();
        messages.add(new UserMessage(userInput));
        int step = 0;

        while (step < maxSteps) {
            step++;
            var prompt = new Prompt(messages);
            var response = chatModel.call(prompt);
            var result = response.getResult().getOutput();

            if (result.hasToolCalls()) {
                messages.add(result);
                for (var toolCall : result.getToolCalls()) {
                    var toolResult = executeTool(toolCall);
                    messages.add(toolResult);
                }
            } else {
                return result.getContent();
            }
        }
        return "已达最大步数限制 " + maxSteps + "，任务未完成";
    }

    private ToolResponseMessage executeTool(ToolCall toolCall) {
        for (ToolCallback tool : tools) {
            if (tool.getName().equals(toolCall.getName())) {
                String result = tool.call(toolCall.getArguments());
                return new ToolResponseMessage(toolCall.getId(), result);
            }
        }
        return new ToolResponseMessage(toolCall.getId(), "工具不存在");
    }
}

ReAct vs 纯对话 LLM

vs chat only 的差异不仅是「能不能调工具」，而是执行模型完全不同：

维度纯对话 LLMReAct Agent
执行方式一次推理 + 一次输出多步推理 + 工具调用 + 观察反馈
状态管理无（每次独立）有状态图（StateGraph）
工具调用无（或模拟调用）真实 MCP/API 调用
错误恢复只能重说观察结果后自动调整
可观测性只有输入输出每步可审计、可回放
适用场景问答、摘要、翻译多步骤任务、办公自动化
不推荐 — 纯对话当 Agent 用
  • 在系统提示里写「你是 Agent，可以调用工具」但无实际 tool call 机制
  • 期望 LLM 一次输出就完成所有步骤
  • 无步数预算，对话失控
推荐 — 真正的 Agent 架构
  • StateGraph 显式定义每步状态转换
  • 每步调用工具并观察结果再决策
  • max_steps 硬限制 + 超时降级
先想 10 秒：为什么纯对话 LLM 不适合做办公自动化？

纯对话 LLM 每次调用是独立的，不维护中间状态，也不具备工具调用反馈循环。办公任务（如「查张三的考勤，算出勤率，生成报告」）需要至少 3 步推理 + 2 次工具调用 + 结果组合，纯对话无法可靠完成。

状态机与预算控制

agent state machine 是 Agent 可靠运行的核心。LangGraph 的 StateGraph 本质上就是一个有限状态机（FSM），每一步的状态转换都显式定义。

步数预算设计

step budget 是防止 Agent 陷入无限循环的关键控制手段。办公场景建议值：

场景max_steps单步超时总超时
简单查询（查余额）3-510s30s
多步流程（报销）8-1220s120s
复杂分析（报表生成）15-2030s300s

当 Agent 步数超预算时，应触发降级而非直接报错：

python
# budget_controller.py
from dataclasses import dataclass
from enum import Enum
import time

class BudgetAction(Enum):
    CONTINUE = "continue"
    DEGRADE = "degrade"
    TERMINATE = "terminate"

@dataclass
class BudgetConfig:
    max_steps: int = 10
    max_duration_sec: int = 120
    degrade_after_steps: int = 8  # 接近限制时降级

class BudgetController:
    def __init__(self, config: BudgetConfig):
        self.config = config
        self.start_time = time.time()

    def check(self, step: int) -> BudgetAction:
        if step >= self.config.max_steps:
            return BudgetAction.TERMINATE
        elapsed = time.time() - self.start_time
        if elapsed >= self.config.max_duration_sec:
            return BudgetAction.TERMINATE
        if step >= self.config.degrade_after_steps:
            return BudgetAction.DEGRADE  # 触发降级：只执行关键步骤
        return BudgetAction.CONTINUE

    private final int maxSteps = 10;

    @Around("@annotation(AgentStep)")

        stepCounter.set(0);
        return null;
    }

ReAct 循环的常见陷阱

observation step 是最容易被忽视的环节。常见问题：

  • 规划超过 3 步无观察循环 — Agent 一开始就把所有步骤列完，不观察中间结果就执行下一步。解决方案：每步必须观察才能继续。
  • 工具假执行 — Agent 声称调用了工具但实际没调，直接在输出里伪造结果。解决方案：工具调用结果必须来自真实 API 返回。
  • 无步数预算 — Agent 可能陷入无限循环（如反复查询同一接口）。解决方案：硬限制 max_steps。
不推荐 — 无观察循环
  • Agent 一次规划 5 步不观察
  • 工具调用结果不反馈给下一轮
  • 无步数/时间预算
推荐 — 每步观察
  • 每步 Think → Act → Observe 完整循环
  • Observe 结果影响下一步的 Think
  • max_steps=15 硬限制 + 超时降级

本章结论

  • ReAct = 思考 → 行动 → 观察，是最小 Agent 工作单元
  • Agent 不是对话 LLM + 工具列表，而是有状态的执行循环
  • StateGraph 显式定义状态转换，优于隐式 Prompt 控制
  • 步数预算和时间预算是 Agent 可靠性的第一道防线
  • 每步必须观察真实工具结果，避免假执行

常见问题（FAQ）

ReAct 的「思考」步骤会不会浪费 Token？
会，但这是必要的开销。思考步骤的 Token 消耗通常占总消耗的 20-30%，但换来的是：避免错误执行导致更大的浪费（如错误发送邮件后的回滚成本）。可以优化思考 Prompt 长度来控制。
什么时候应该中断 ReAct 循环？
三种情况：步数超限、时间超时、Agent 判断任务完成。建议再加一种：连续 2 次观察结果相同（陷入循环）也应主动中断。
LangGraph 和 ReactAgent 在 ReAct 实现上有什么区别？
LangGraph 提供 StateGraph + Checkpoint 的完整状态管理，适合复杂多分支。Spring AI Alibaba ReactAgent 封装了标准 ReAct Loop，适合单轮多步调用。复杂流程推荐 LangGraph，简单工具链用 ReactAgent 更轻量。

Plan-and-Execute 架构

plan execute 是 Agent 任务规划的经典模式。与 ReAct 的「边走边想」不同，Plan-and-Execute 先由 Planner 生成完整计划，再由 Executor 逐条执行。适合步骤明确的办公场景。

# plan_execute_agent.py
from langgraph.graph import StateGraph, END
from typing import TypedDict, List, Optional

class SubTask(TypedDict):
    id: str
    description: str
    status: str
    result: Optional[str]
    retry_count: int
    max_retries: int
    depends_on: List[str]

class PlanState(TypedDict):
    input: str
    plan: List[SubTask]
    current_step: int
    errors: List[str]

def planner_node(state: PlanState) -> dict:
    prompt = f"将以下任务分解为子任务：\n{state['input']}\n"
    prompt += "每个子任务含 id、description、depends_on。"
    response = llm.invoke(prompt)
    tasks = parse_subtasks(response)
    return {"plan": tasks, "current_step": 0}

def executor_node(state: PlanState) -> dict:
    task = state["plan"][state["current_step"]]
    # 检查依赖是否完成
    for dep_id in task["depends_on"]:
        dep = next(t for t in state["plan"] if t["id"] == dep_id)
        if dep["status"] != "done":
            return {"errors": state["errors"] + [f"依赖 {dep_id} 未完成"]}
    try:
        result = execute_tool(task["description"])
        task["status"] = "done"
        task["result"] = result
    except Exception as e:
        if task["retry_count"] < task["max_retries"]:
            task["status"] = "pending"
            task["retry_count"] += 1
        else:
            task["status"] = "failed"
        state["errors"].append(str(e))
    return {"plan": state["plan"]}

def router(state: PlanState) -> str:
    task = state["plan"][state["current_step"]]
    if task["status"] == "done":
        state["current_step"] += 1
        if state["current_step"] >= len(state["plan"]):
            return "finish"
        return "continue"
    elif task["status"] == "failed":
        return "fail"
    return "retry"

builder = StateGraph(PlanState)
builder.add_node("planner", planner_node)
builder.add_node("executor", executor_node)
builder.set_entry_point("planner")
builder.add_edge("planner", "executor")
builder.add_conditional_edges("executor", router, {
    "continue": "executor",
    "retry": "executor",
    "finish": END,
    "fail": END
})

    private final ToolExecutor toolExecutor;

        String prompt = "将以下任务分解为子任务列表，每个子任务用 JSON：" +
            "{\"id\":\"1\",\"description\":\"...\",\"depends_on\":[]}\n任务：" + userInput;
        ChatResponse r = chatModel.call(new Prompt(List.of(new UserMessage(prompt))));
        return parseTaskList(r.getResult().getOutput().getContent());
    }

    public ExecutionResult execute(List<SubTask> tasks) {
        var results = new ArrayList<TaskResult>();
        for (SubTask task : tasks) {
            try {
                task.setStatus(SubTask.Status.RUNNING);
                String result = toolExecutor.execute(task.getDescription());
                task.setStatus(SubTask.Status.DONE);
                task.setResult(result);
                results.add(new TaskResult(task.getId(), result, true));
            } catch (Exception e) {
                if (task.getRetryCount() < task.getMaxRetries()) {
                    task.setRetryCount(task.getRetryCount() + 1);
                    task.setStatus(SubTask.Status.PENDING);
                    results.add(execute(List.of(task)).getResults().get(0));
                } else {
                    task.setStatus(SubTask.Status.FAILED);
                    results.add(new TaskResult(task.getId(), e.getMessage(), false));
                }
            }
        }
        return new ExecutionResult(results);
    }
}

规划中的边界情况处理

任务规划在实际执行中常遇到边界情况，需要针对性处理：

  • 空计划：用户输入模糊（如「帮我看看」），Planner 无法生成有意义的计划 → 反馈用户需要更具体的描述
  • 循环依赖：步骤 A 依赖 B，B 又依赖 A → Planner 生成时检测循环引用，如果出现则合并步骤 A 和 B
  • 并行爆炸：20 个无依赖的子任务全部并行执行 → 限制并行度（max_concurrent=10），其余排队
  • 长计划截断：Planner 生成了 30 步的计划 → 按优先级分阶段，第一阶段只执行前 10 步，第一阶段完成后生成第二阶段
  • 计划与执行不一致：执行结果和 Planner 预期不符 → 停止当前计划，让 Planner 重新规划
python
def detect_cycle(tasks: List[SubTask]) -> bool:
    """检测计划中的循环依赖"""
    graph = {t["id"]: t["depends_on"] for t in tasks}
    visited = set()

    def dfs(node_id: str, path: set) -> bool:
        if node_id in path:
            return True  # 发现循环
        if node_id in visited:
            return False
        visited.add(node_id)
        path.add(node_id)
        for dep in graph.get(node_id, []):
            if dfs(dep, path):
                return True
        path.remove(node_id)
        return False

    return any(dfs(t["id"], set()) for t in tasks)

def merge_cyclic_tasks(tasks: List[SubTask]) -> List[SubTask]:
    """检测到循环依赖时合并相关步骤"""
    if not detect_cycle(tasks):
        return tasks
    # 简化实现：将循环中的步骤合并为一个
    merged = SubTask(id="merged", description="合并后的步骤",
                     depends_on=[], status="pending")
    return [merged] + [t for t in tasks
            if t["id"] not in [m["id"] for m in [merged]]]

# recovery_handler.py
import time
from functools import wraps

def with_retry(max_retries=3, delay=1.0, fallback=None):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exception = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    last_exception = e
                    if attempt < max_retries - 1:
                        time.sleep(delay * (2 ** attempt))
            return fallback(*args, **kwargs) if fallback else None
        return wrapper
    return decorator

@with_retry(max_retries=2, fallback=lambda x: {"default": True})
def query_attendance(dept: str) -> dict:
    return api.call(f"/attendance/{dept}")

from collections import deque
from typing import List, Dict
import json

class MemoryManager:
    def __init__(self, window_size: int = 20):
        self.short_term = deque(maxlen=window_size)  # 滑动窗口
        self.long_term: Dict[str, str] = {}  # 用户偏好
        self.summary: str = ""

    def add_short(self, item: dict):
        self.short_term.append(item)

    def summarize_and_archive(self):
        """将短期记忆摘要后归档到长期"""
        if len(self.short_term) >= self.short_term.maxlen:
            recent = list(self.short_term)[-5:]  # 保留最近 5 条
            old = list(self.short_term)[:-5]
            self.summary = self._generate_summary(old)
            self.short_term = deque(recent, maxlen=20)

    def get_context(self) -> str:
        """组装最终上下文：摘要+短期+用户偏好"""
        parts = []
        if self.summary:

工具结果缓存策略

短期内重复调用同一工具会浪费 Token 和 API 配额。工具结果缓存（Tool Result Caching）有效减少重复调用：

python
# tool_cache.py
import hashlib
import json
import time
from typing import Dict, Any, Optional

class ToolResultCache:
    def __init__(self, ttl_sec: int = 60):
        self.cache: Dict[str, tuple] = {}
        self.ttl = ttl_sec

    def _make_key(self, tool_name: str, args: dict) -> str:
        raw = f"{tool_name}:{json.dumps(args, sort_keys=True)}"
        return hashlib.md5(raw.encode()).hexdigest()

    def get(self, tool_name: str, args: dict) -> Optional[str]:
        key = self._make_key(tool_name, args)
        if key in self.cache:
            result, timestamp = self.cache[key]
            if time.time() - timestamp < self.ttl:
                return result
            del self.cache[key]
        return None

    def set(self, tool_name: str, args: dict, result: str):
        key = self._make_key(tool_name, args)
        self.cache[key] = (result, time.time())

    def invalidate(self, tool_name: str):
        """工具数据变化时清除相关缓存"""
        keys_to_delete = [k for k in self.cache if k.startswith(tool_name)]
        for k in keys_to_delete:
            del self.cache[k]

# 使用示例
tool_cache = ToolResultCache(ttl_sec=120)

def cached_tool_call(tool_name: str, args: dict) -> str:
    cached = tool_cache.get(tool_name, args)
    if cached:
        return cached  # 命中缓存
    result = mcp_tools.call(tool_name, args)
    if not result.get("error"):
        tool_cache.set(tool_name, args, result)
    return result

常见问题（FAQ）

短期记忆和长期记忆的界限在哪里？
短期：当前会话上下文（通常 20 轮对话或 4096 tokens）。长期：跨会话的用户偏好、知识库、历史摘要。超过短期窗口的自动摘要归档到长期。
MCP Server 和传统 API Gateway 有什么区别？
MCP 专门为 LLM Agent 设计：工具描述（Function Calling Schema）是 MCP Server 的一部分，Agent 可以动态发现工具。传统 API Gateway 面向人类开发者，没有工具发现机制。
工具契约变了怎么办？
工具契约版本化。Agent 启动时从 MCP Server 拉取最新契约。契约不兼容时应有迁移期，期间新旧版本同时支持。

StateGraph 定义办公工作流

langgraph workflow 通过 StateGraph 显式定义状态和节点转换。每个节点是一个处理函数，边定义了状态流向。Agent 的状态在每一步都可审计和回放。

      CheckBalance --> Draft[Draft Leave\n生成请假草稿]
      Draft --> HumanGate{人工审批\ninterrupt_before}

      Escalate --> HumanGate

CorpAssist · 请假流程 员工说「请下周二到周四的年假」。Agent 先查余额（15 天），判断可用后生成请假草稿，然后在「提交审批」节点前 interrupt，等待直属上级审批。上级在审批页面点击「通过」，Agent 继续执行提交。整个过程由 LangGraph Checkpoint 持久化。

# leave_agent_graph.py
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict, Optional

class LeaveState(TypedDict):
    user_input: str
    balance: Optional[float]
    draft: Optional[dict]
    approval: Optional[str]  # approved/rejected/escalated
    result: Optional[str]

def planner_node(state: LeaveState) -> dict:
    prompt = f"解析请假请求：{state['user_input']}\n"
    prompt += "提取：开始日期、结束日期、请假类型"
    parsed = llm.invoke(prompt)
    return {"draft": parsed}

def check_balance_node(state: LeaveState) -> dict:
    balance = leave_api.get_balance(user_id)
    return {"balance": balance}

def draft_node(state: LeaveState) -> dict:
    draft = leave_api.create_draft(
        start=state["draft"]["start"],
        end=state["draft"]["end"],
        leave_type=state["draft"]["type"]
    )
    return {"draft": draft}

def submit_node(state: LeaveState) -> dict:
    result = leave_api.submit(state["draft"]["id"])
    return {"result": result, "approval": "submitted"}

def should_approve(state: LeaveState) -> str:
    return state.get("approval", "pending")

# Graph 构建
builder = StateGraph(LeaveState)
builder.add_node("planner", planner_node)
builder.add_node("check_balance", check_balance_node)
builder.add_node("draft", draft_node)
builder.add_node("submit", submit_node)

builder.set_entry_point("planner")
builder.add_edge("planner", "check_balance")
builder.add_edge("check_balance", "draft")
builder.add_conditional_edges("draft", should_approve, {
    "approved": "submit",
    "rejected": END,
    "escalated": END,
    "pending": "submit"  # 中断后恢复继续执行
})
builder.add_edge("submit", END)

# 编译时配置 checkpointer 和 interrupt
checkpointer = MemorySaver()
graph = builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["submit"]  # 在 submit 前中断等待审批
)

Checkpoint 与状态持久化

graph checkpoint 是 LangGraph 的核心能力之一。每次节点执行后，Checkpointer 自动保存完整状态。Agent 重启或中断后可以从上次 checkpoint 恢复，无需重新执行已完成步骤。

python

print(result)  # 执行到 interrupt_before，等待审批

# 审批通过后，恢复执行
result = graph.invoke({
    "approval": "approved",
    "draft": result["draft"]
}, config)
print(result)  # 继续执行 submit 节点

Checkpoint 支持多种存储后端：

存储适合场景特性
MemorySaver开发测试进程内存，重启丢失
SqliteSaver单机部署持久化到文件
PostgresSaver生产环境分布式、高可用
RedisSaver高吞吐场景低延迟、支持 TTL

interrupt node 实现了 human-in-the-loop。当执行到 interrupt_before 指定的节点时，Graph 暂停并等待外部输入。审批系统通过调用 graph.invoke 传入审批结果，Graph 从断点继续。

python
# approval_callback.py
class ApprovalHandler:
    def __init__(self, graph, thread_id: str):
        self.graph = graph
        self.thread_id = thread_id
        self.config = {"configurable": {"thread_id": thread_id}}

    def check_pending(self) -> list:
        """查询所有等待审批的 Agent 任务"""
        state = self.graph.get_state(self.config)
        if state.next and "submit" in state.next:
            return [{
                "thread_id": self.thread_id,
                "pending_node": state.next[0],
                "context": state.values.get("draft", {})
            }]
        return []

    def approve(self) -> dict:
        """批准：传入 approved 状态继续执行"""
        return self.graph.invoke({"approval": "approved"}, self.config)

    def reject(self, reason: str) -> dict:
        """驳回：传入 rejected 状态终止"""
        return self.graph.invoke({"approval": "rejected", "reason": reason}, self.config)

    def escalate(self) -> dict:
        """升级到上级审批"""
        return self.graph.invoke({"approval": "escalated"}, self.config)

条件边与状态路由实践

LangGraph 的条件边（Conditional Edge）是 Agent 智能决策的关键。相比硬编码的 if-else，条件边让 LLM 根据当前状态动态决定下一步走向。

python
# conditional_router.py
def approval_router(state: LeaveState) -> str:
    """根据审批状态动态路由"""
    status = state.get("approval_status", "pending")

    if status == "approved":
        return "submit"
    elif status == "rejected":
        return "notify_rejection"
    elif status == "escalated":
        return "escalate_approval"
    else:
        # pending → 继续等待（保持 interrupt）
        return "wait"

def timeout_router(state: LeaveState) -> str:
    """审批超时处理"""
    elapsed = time.time() - state["approval_created_at"]
    if elapsed > 1800:  # 30 分钟超时
        return "escalate"
    return "wait"

# Graph 中配置条件边
builder.add_conditional_edges(
    "approval_wait",
    approval_router,
    {
        "submit": "submit_node",
        "notify_rejection": "rejection_node",
        "escalate": "escalate_node",
        "wait": "approval_wait"
    }
)

这种路由方式比传统工作流引擎更灵活：路由规则由 LLM 动态生成，而不是预定义的 XML/DSL。但也要注意：条件边不能太多（建议不超过 5 条），否则 LLM 的决策准确率会下降。

    public String route(LeaveState state) {
        ApprovalStatus status = state.getApprovalStatus();
        return switch (status) {
            case APPROVED -> "submit";
            case REJECTED -> "notify";
            case ESCALATED -> "escalate";

条件边的设计原则：每个条件边的出边不超过 5 条。分支太多时，拆分为多层条件边（先判断大类，再判断子类）。

def stream_agent_execution(user_input: str):
    """流式输出 Agent 执行过程"""
    config = {"configurable": {"thread_id": "session-01"}}

常见问题（FAQ）

interrupt_before 和 interrupt_after 有什么区别？
interrupt_before 在节点执行前中断（适合审批场景），interrupt_after 在节点执行后中断（适合审计场景）。审批一般用 interrupt_before，防止未经授权的操作已执行。
Checkpoint 存储会不会太大？
每个 checkpoint 只存储状态变化（delta）而非全量，大小通常为几 KB。长时间运行的 Agent 建议定期归档旧 checkpoint 或用 TTL 自动清理。
审批超时不处理怎么办？
可以在 Graph 中加一个超时节点：定时检查审批状态，超时后走 escalate 分支升级到上级审批人。LangGraph 的 @workflow.timeout 可以配置整体运行超时。

@Aspect
@Component
public class ApprovalAspect {

    private final ApprovalService approvalService;

    @Around("@annotation(RequireApproval)")
    public Object checkApproval(ProceedingJoinPoint pjp) throws Throwable {
        MethodSignature sig = (MethodSignature) pjp.getSignature();
        RequireApproval ann = sig.getMethod()
            .getAnnotation(RequireApproval.class);

        // 检查是否需要审批
        if (approvalService.needsApproval(ann.value(), pjp.getArgs())) {
            String taskId = approvalService.createTask(
                ann.value(), pjp.getArgs());

            ApprovalResult result = approvalService.waitForApproval(taskId);
            if (!result.isApproved()) {
                throw new ApprovalRejectedException(result.getReason());
            }
        }
        return pjp.proceed();
    }
}

    SAA 流式处理与 SSE 集成

    Spring AI Alibaba 支持流式（Streaming）方式返回 Agent 执行结果。这在办公场景中非常实用：用户可以实时看到 Agent 的思考过程和工具调用结果，而不是等待最终答案。

    java
    // StreamingAgentController.java
@RestController
@RequestMapping("/api/agent/stream")
public class StreamingAgentController {

    private final ReactAgent agent;

    @GetMapping(produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> streamExecute(
            @RequestParam String input) {
        var messages = List.of(new UserMessage(input));
        var prompt = new Prompt(messages);

        return agent.stream(prompt)
            .map(response -> {
                var output = response.getResult().getOutput();
                String content = output.getContent();

                // 判断是否需要工具调用
                if (output.hasToolCalls()) {
                    return ServerSentEvent.<String>builder()
                        .event("tool_call")
                        .data(output.getToolCalls().toString())
                        .build();
                }
                return ServerSentEvent.<String>builder()
                    .event("message")
                    .data(content)
                    .build();
            })
            .doOnError(e -> log.error("Stream error", e));
    }
}

    SAA 的 ReactAgent 和 LangGraph 的 ReactAgent 有什么区别？

    推荐通过消息队列（RocketMQ/Kafka）或 MCP 桥接。Python 的复杂 Agent 处理结果通过事件发送给 Java 侧的审批服务，Java 的审批结果通过回调通知 Python。

    Python：并行工具调用

    python
    # parallel_tool_executor.py
import asyncio
from typing import List, Dict, Any, Callable

class ParallelToolExecutor:
    def __init__(self, timeout_sec: int = 30):
        self.timeout = timeout_sec

    async def execute_tool(self, tool_call: Callable,
                           args: dict) -> Dict[str, Any]:
        try:
            result = await asyncio.wait_for(
                tool_call(**args), timeout=self.timeout)
            return {"success": True, "result": result}
        except asyncio.TimeoutError:
            return {"success": False, "error": "timeout"}
        except Exception as e:
            return {"success": False, "error": str(e)}

    async def execute_parallel(self, calls: List[tuple]) -> List[Dict]:
        """并行执行多个工具调用"""
        tasks = [self.execute_tool(fn, args) for fn, args in calls]
        return await asyncio.gather(*tasks)

    def execute_serial(self, calls: List[tuple]) -> List[Dict]:
        """串行执行，前一步结果可影响后一步"""
        results = []
        for fn, args in calls:
            # 可以将前一步结果注入 args
            result = asyncio.run(self.execute_tool(fn, args))
            results.append(result)
            if not result["success"]:
                break  # 串行调用中断
        return results

# 使用示例：会议安排
async def arrange_meeting():
    executor = ParallelToolExecutor()

    # 串行：先查主会议室
    room_result = await executor.execute_tool(
        calendar.query, {"room": "A201", "time": "2025-06-01T15:00"})

    // ParallelToolService.java
@Service
public class ParallelToolService {

    private final ExecutorService executor =
        Executors.newFixedThreadPool(10);
    private final Duration timeout = Duration.ofSeconds(30);

    public List<ToolResult> executeParallel(List<ToolCall> calls) {
        var futures = calls.stream()
            .map(call -> CompletableFuture
                .supplyAsync(() -> executeSingle(call), executor)
                .orTimeout(timeout.toMillis(), TimeUnit.MILLISECONDS)
                .exceptionally(ex -> ToolResult.failure(call.id(), ex.getMessage())))
            .toList();

        return futures.stream()
            .map(CompletableFuture::join)
            .toList();
    }

    private ToolResult executeSingle(ToolCall call) {
        try {
            String result = toolRegistry.call(call.name(), call.args());
            return ToolResult.success(call.id(), result);
        } catch (Exception e) {
            return ToolResult.failure(call.id(), e.getMessage());
        }
    }

    // 串行执行，失败即终止
    public List<ToolResult> executeSerial(List<ToolCall> calls) {
        var results = new ArrayList<ToolResult>();
        for (ToolCall call : calls) {
            ToolResult result = executeSingle(call);
            results.add(result);
            if (!result.isSuccess()) break;
        }
        return results;
    }
}

    工具错误处理与重试

    tool retry 处理工具调用的瞬时失败。办公工具的常见失败模式：

    失败类型概率处理策略
    网络超时（5xx）高指数退避重试 3 次
    限流（429）中等待后重试，增加间隔
    参数错误（4xx）低记录日志，返回 Agent 修正
    数据不存在（404）低返回空结果，继续执行
    python
    # resilient_tool.py
import time
from functools import wraps

def resilient_tool(max_retries=3, base_delay=1.0,
                   retryable_exceptions=(TimeoutError, ConnectionError)):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_error = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except retryable_exceptions as e:
                    last_error = e
                    if attempt < max_retries - 1:
                        delay = base_delay * (2 ** attempt)
                        time.sleep(delay)
                except Exception as e:
                    # 非可重试异常，立即返回错误
                    return {"error": str(e), "fatal": True}
            return {"error": str(last_error), "retries_exhausted": True}
        return wrapper
    return decorator

@resilient_tool(max_retries=3)
def call_email_api(to: str, subject: str, body: str) -> dict:
    return email_client.send(to, subject, body)

    partial failure 是并行工具调用的典型问题：部分工具成功、部分失败。Agent 应处理这种「部分成功」状态，例如：会议创建成功了但通知邮件失败了，应标记「部分完成」并记录异常。

    工具超时与隔离设计

    每个工具的失败不应该影响其他工具的正常执行。工具隔离（Tool Isolation）通过以下机制实现：

    • 线程隔离：每个工具在独立的线程池中执行，一个工具阻塞不阻塞其他工具
    • 超时隔离：每个工具有独立的超时配置，超时不影响其他正在执行的工具
    • 熔断隔离：每个工具有独立的熔断状态，一个工具熔断不影响其他工具

    办公工具编排最佳实践

    office tools 的编排原则：

    • 读优于写：查询类工具优先并行执行，修改类工具串行执行
    • 先检查再操作：创建/发送前先查询权限和状态
    • 幂等设计：每个写操作带幂等键，重复调用不产生副作用
    • 部分成功处理：并行调用中部分失败不影响已成功的部分
    • 隔离设计：每个工具有独立的线程池和超时配置
    不推荐 — 全串行或全并行
    • 所有工具串行调用，延迟叠加
    • 无依赖工具也串行，浪费等待时间
    • 一个工具失败导致整个流程中止
    • 所有工具共享线程池，一个阻塞全阻塞
    推荐 — 按依赖编排+隔离
    • 无依赖的查询并行执行（查多人日历）
    • 有依赖的步骤串行执行（查完再创建）
    • 部分失败仅影响相关分支
    • 每个工具独立线程池+超时

    本章结论

    • 多工具编排分并行和串行，按依赖关系决定
    • 并行调用用 asyncio（Python）或 CompletableFuture（Java）
    • 工具失败用指数退避重试，不可重试异常记录后返回
    • 部分失败是并行调用的常态，需设计兼容策略
    • 读操作并行、写操作串行+幂等键

    常见问题（FAQ）

    并行调用的并发数设多少合适？
    取决于 API 限流和系统资源。办公场景建议硬件限制为 10。如果调用企业微信/钉钉 API，注意它们的 QPS 限制（通常 5-20/s），并发数不要超过限流阈值。
    并行调用中一个失败，其它要不要回滚？
    取决于业务场景。查询类失败无需回滚。写操作类（如批量创建工单）建议用 Saga 模式：先记录已成功部分，失败部分补偿或人工处理。
    工具调用顺序是谁决定的？Agent 还是代码？
    推荐 Agent 决定「调什么」，代码决定「怎么调」。Agent 的 Planner 决定工具调用顺序和参数，工具的执行引擎决定并行/串行和执行策略。

    task: str
    params: Dict
    status: str
    result: str

    user_input: str
    plan: List[AgentTask]
    results: Dict[str, str]
    conflicts: List[str]
    final_output: str

def supervisor_node(state: MultiAgentState) -> dict:
    """Supervisor：分析输入，拆解任务，分派给 Worker"""
    prompt = f"分析采购请求，拆解为并行子任务：{state['user_input']}"
    plan = llm.invoke(prompt)
    tasks = []
    for item in parse_plan(plan):
        tasks.append(AgentTask(
            worker_id=item["worker"],
            task=item["description"],
            params=item["params"],
            status="pending",
            result=""
        ))
    return {"plan": tasks}

def worker_node(state: MultiAgentState, worker_id: str) -> dict:
    """Worker：执行指定任务"""
    task = next(t for t in state["plan"] if t["worker_id"] == worker_id)
    try:
        result = execute_worker_task(worker_id, task)
        return {"results": {worker_id: result},
                "plan": [update_status(task, "done")]}
    except Exception as e:
        return {"results": {worker_id: f"ERROR: {str(e)}"},
                "plan": [update_status(task, "failed")]}

def aggregator_node(state: MultiAgentState) -> dict:
    """聚合所有 Worker 结果，检查冲突"""
    conflicts = []
    # 检查预算 vs 政策冲突
    budget = state["results"].get("budget_worker", "")
    compliance = state["results"].get("compliance_worker", "")
    if "超预算" in budget and "合规" in compliance:
        conflicts.append("预算不足但合规允许，需人工审核")
    return {"conflicts": conflicts}

    if state["conflicts"]:
        return "human_review"
    return "output"

    Agent 注册与发现机制

    在微服务架构中，多 Agent 需要服务注册与发现机制。每个 Worker Agent 启动时向注册中心注册自己的能力和地址，Supervisor 通过查询注册中心找到合适的 Worker。

    python
    # agent_registry.py
from dataclasses import dataclass, field
from typing import Dict, List
import redis

@dataclass
class AgentCapability:
    agent_id: str
    agent_type: str  # supervisor / worker
    capabilities: List[str]  # ["budget_query", "inventory_query"]
    endpoint: str  # gRPC / HTTP 地址
    status: str = "active"  # active / busy / offline

class AgentRegistry:
    def __init__(self, redis_client):
        self.redis = redis_client

    def register(self, capability: AgentCapability):
        key = f"agent:{capability.agent_id}"
        self.redis.hset(key, mapping={
            "type": capability.agent_type,
            "capabilities": ",".join(capability.capabilities),
            "endpoint": capability.endpoint,
            "status": capability.status
        })
        self.redis.expire(key, 30)  # 30 秒心跳

    def discover(self, required_capability: str) -> List[AgentCapability]:
        """发现具有特定能力的 Agent"""
        agents = []
        for key in self.redis.scan_iter("agent:*"):
            data = self.redis.hgetall(key)
            if required_capability in data.get("capabilities", ""):
                agents.append(AgentCapability(
                    agent_id=key.split(":")[1],
                    agent_type=data["type"],
                    capabilities=data["capabilities"].split(","),
                    endpoint=data["endpoint"],
                    status=data["status"]
                ))
        return agents

    Supervisor Agent 自身会不会成为瓶颈？

    Agent 间通信延迟怎么处理？

    python
    # agent_monitor.py
import time
from dataclasses import dataclass, field
from typing import List, Dict
from datetime import datetime

@dataclass
class StepMetrics:
    step_id: int
    node_name: str
    tool_calls: List[Dict]
    input_tokens: int
    output_tokens: int
    duration_ms: float
    success: bool
    error: str = ""

class AgentMonitor:
    def __init__(self):
        self.steps: List[StepMetrics] = []

    def start_step(self, node_name: str) -> dict:
        return {
            "step_id": len(self.steps) + 1,
            "node_name": node_name,
            "start_time": time.time()
        }

    def end_step(self, step: dict, success: bool,
                 tool_calls=None, tokens=None, error=""):
        duration = (time.time() - step["start_time"]) * 1000
        metrics = StepMetrics(
            step_id=step["step_id"],
            node_name=step["node_name"],
            tool_calls=tool_calls or [],
            input_tokens=tokens.get("input", 0) if tokens else 0,
            output_tokens=tokens.get("output", 0) if tokens else 0,
            duration_ms=round(duration, 2),
            success=success,
            error=error
        )
        self.steps.append(metrics)
        return metrics

    def summary(self) -> dict:
        total_tokens = sum(s.input_tokens + s.output_tokens for s in self.steps)
        total_duration = sum(s.duration_ms for s in self.steps)
        success_rate = sum(1 for s in self.steps if s.success) / len(self.steps)

        return {
            "total_steps": len(self.steps),
            "total_tokens": total_tokens,
            "total_duration_ms": round(total_duration, 2),
            "success_rate": round(success_rate * 100, 1),
            "step_details": [
                {"step": s.step_id, "node": s.node_name,
                 "tokens": s.input_tokens + s.output_tokens,
                 "duration_ms": s.duration_ms, "success": s.success}
                for s in self.steps
            ]
        }

# LangGraph 集成
monitor = AgentMonitor()

def monitored_node(node_func):
    def wrapper(state):
        step = monitor.start_step(node_func.__name__)
        try:
            result = node_func(state)
            monitor.end_step(step, True)
            return result
        except Exception as e:
            monitor.end_step(step, False, error=str(e))
            raise
    return wrapper

    坑 1：无观察循环

    observe loop 是最常见的生产事故。Agent 规划了 3 步以上任务但不观察中间结果，直接按初始计划执行到底。一旦第一步结果和预期不符，后续步骤全部错误。

    有观察 vs 无观察循环
    flowchart LR
      subgraph Bad[无观察循环 - 错误]
        Plan[初始计划] --> Act1[行动 1]
        Act1 --> Act2[行动 2]
        Act2 --> Act3[行动 3]
        Act3 --> Output[错误输出]
      end
      subgraph Good[每步观察 - 正确]
        Plan2[初始计划] --> Think1[思考]
        Think1 --> Act1G[行动 1]
        Act1G --> Observe1[观察 1]
        Observe1 -->|调整| Think2[再思考]
        Think2 --> Act2G[行动 2]
        Act2G --> Observe2[观察 2]
        Observe2 -->|完成| OutputG[正确输出]
      end

    step budget pitfall：无观察循环的 Agent 可能在第一步失败后继续执行。

    解决方案：在 LangGraph 中强制每步有条件边，Observe 结果必须影响下一轮的 Think。

    python
    # observe_guard.py
def observe_guard(state) -> str:
    """强制观察检查：如果没有最新观察结果，不允许继续"""
    if not state.get("latest_observation"):
        return "observe_missing_error"
    # 检查观察结果是否有实质内容
    obs = state["latest_observation"]
    if "ERROR" in obs or "失败" in obs:
        return "observation_failed"
    if state["step"] >= state["max_steps"]:
        return "step_limit_exceeded"
    return "continue"

    坑 2：步数与返回长度预算

    return length cap 控制工具返回结果的长度。如果工具返回 10 万字符的文档，Agent 的上下文会瞬间爆满。step budget pitfall 指无步数上限导致的无限循环。

    python
    # budget_guard.py
from typing import Dict, Any

MAX_TOOL_RETURN_CHARS = 2000
MAX_STEPS = 15

def truncate_tool_return(result: str) -> str:
    """截断工具返回结果"""
    if len(result) > MAX_TOOL_RETURN_CHARS:
        return result[:MAX_TOOL_RETURN_CHARS] + \
               f"\n... (已截断，共 {len(result)} 字符)"
    return result

def check_step_budget(state: Dict[str, Any]) -> str:
    """检查步数预算"""
    if state["step"] >= MAX_STEPS:
        return "budget_exceeded"
    # 检查是否连续重复步骤
    if len(state["actions"]) >= 3:
        last_three = state["actions"][-3:]
        if all(a == last_three[0] for a in last_three):
            return "repeated_action"
    return "ok"

class BudgetEnforcedTool:
    """带预算限制的工具包装器"""
    def __init__(self, tool_fn, name: str):
        self.tool_fn = tool_fn
        self.name = name
        self.call_count = 0

    def call(self, **kwargs) -> str:
        self.call_count += 1
        if self.call_count > 5:  # 单工具调用上限
            return "TOOL_CALL_LIMIT_EXCEEDED"
        result = self.tool_fn(**kwargs)
        return truncate_tool_return(result)

    坑 3：工具假执行检测

    fake tool detect 指 Agent 声称调用了工具但实际没有调用，直接在输出中伪造了工具返回结果。这是 LLM 的 hallucination 在工具层面的表现。tool verify 通过以下方式检测：

    • 响应签名校验：工具返回结果必须携带签名或时间戳
    • 调用链追踪：每次工具调用在日志中有唯一 trace_id
    • 结果可验证：工具返回的数据可以通过另一个只读工具交叉验证
    python
    # fake_tool_detector.py
import hashlib
import json
import time

class FakeToolDetector:
    def __init__(self):
        self.call_log = []

    def verify_call(self, tool_name: str, args: dict, result: str) -> bool:
        """验证工具是否真的被调用了"""
        # 1. 检查结果格式是否符合工具契约
        if not self._validate_schema(tool_name, result):
            return False

        # 2. 检查结果中的时间戳是否接近当前时间
        try:
            data = json.loads(result)
            ts = data.get("timestamp", 0)
            if abs(time.time() - ts) > 5:  # 5 秒偏差
                return False  # 可能是伪造的结果
        except:
            pass

        # 3. 验证签名（如果工具有签名机制）
        if not self._verify_signature(tool_name, result):
            return False

        self.call_log.append({
            "tool": tool_name,
            "args": args,
            "time": time.time(),
            "verified": True
        })
        return True

    def _validate_schema(self, tool_name: str, result: str) -> bool:
        contract = tool_registry.get_contract(tool_name)
        # 检查返回结果是否包含契约中的必需字段
        for field in contract.required_return_fields:
            if field not in result:
                return False
        return True

    工具假执行是 LLM 的问题还是我们代码的问题？
    两者都有。LLM 可能「编造」工具结果，但代码层可以通过强制验证（签名、时间戳、追踪链）来检测和阻断。修复重点在代码层。
    截断工具返回结果会不会丢失重要信息？
    会。所以截断策略是返回摘要 + 截断正文。先让 LLM 看摘要决定是否需要完整内容，需要时再查详情。
    记忆分层的容量参数怎么定？
    根据模型窗口大小和场景需要。以 32K 窗口为例：system prompt 占 4K、工具返回占 8K、短期记忆占 10K、检索结果占 10K。按这个比例分配。

    circuit agent 状态机：Closed → Open → Half-Open → Closed。

    Python：熔断实现

    python
    # circuit_breaker.py
import time
from enum import Enum
from typing import Callable, Any

class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(self, threshold: int = 3,
                 reset_timeout: int = 60):
        self.state = CircuitState.CLOSED
        self.threshold = threshold
        self.reset_timeout = reset_timeout
        self.fail_count = 0
        self.last_fail_time = 0

    def call(self, func: Callable, *args, **kwargs) -> Any:
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_fail_time > self.reset_timeout:
                self.state = CircuitState.HALF_OPEN
            else:
                raise CircuitBreakerOpenError("熔断已打开")

        try:
            result = func(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.fail_count = 0
            return result
        except Exception as e:
            self.fail_count += 1
            self.last_fail_time = time.time()
            if self.fail_count >= self.threshold:
                self.state = CircuitState.OPEN
            raise e

# 使用示例
email_breaker = CircuitBreaker(threshold=3, reset_timeout=60)

def send_email_with_breaker(to, subject, body):
    try:
        return email_breaker.call(email_api.send, to, subject, body)
    except CircuitBreakerOpenError:
        return {"error": "邮件服务熔断", "fallback": "已存入延迟队列"}

    python
    # timeout_handling.py
import asyncio
from typing import Any, Callable

class TimeoutHandler:
    def __init__(self, default_timeout: int = 30):
        self.default_timeout = default_timeout

    async def call_with_timeout(self, func: Callable,
                                 timeout: int = None,
                                 fallback: Callable = None,
                                 *args, **kwargs) -> Any:
        t = timeout or self.default_timeout
        try:
            return await asyncio.wait_for(func(*args, **kwargs), timeout=t)
        except asyncio.TimeoutError:
            print(f"工具调用超时 ({t}s)，执行降级")
            if fallback:
                return fallback(*args, **kwargs)
            return {"error": "timeout", "message": f"超过 {t} 秒未响应"}

    人工兜底（Human Fallback）

    human fallback 是可靠性的最后一道防线。当所有自动策略都失败时，必须有路径转到人工处理。

    python
    # human_fallback.py
class HumanFallback:
    def __init__(self, approval_queue):
        self.queue = approval_queue

    def fallback_to_human(self, task: dict, reason: str):
        """将失败的任务转人工处理"""
        fallback_task = {
            "original_task": task,
            "fallback_reason": reason,
            "status": "pending_human",
            "created_at": time.time(),
            "priority": task.get("priority", "normal")
        }
        self.queue.push(fallback_task)
        return {
            "status": "FALLBACK_TO_HUMAN",
            "message": f"已转人工处理，原因：{reason}",
            "task_id": fallback_task["id"]
        }

    def query_human_result(self, task_id: str) -> dict:
        """查询人工处理结果"""
        return self.queue.get_result(task_id)

    常见问题（FAQ）

    熔断打开后，Agent 怎么知道恢复了？
    Half-Open 状态会定期发送探测请求。如果探测成功则关闭熔断，失败则继续 Open。探测频率可配，建议 60s 一次。
    降级返回「稍后处理」，用户怎么知道结果？
    降级策略应该包含通知机制：异步任务完成后通过 WebSocket/邮件/钉钉通知用户结果。Agent 应同步返回「你的请求已提交，处理完成后会通知你」。
    审计日志会不会太大？
    全量审计确实会产生大量数据。建议：全量写 Elasticsearch（保留 90 天），抽样写成本更低的存储（如 S3 冷存）。监控系统只关注错误日志和性能指标。

    STAR 案例：量化你的 Agent 项目

    star case 是简历和答辩的黄金模板。STAR 代表 Situaion（场景）、Task（任务）、Action（行动）、Result（结果）。关键在于 Result 必须有量化指标。

    markdown
    ## STAR: CorpAssist 请假 Agent

### S (Situation 场景)
- 5000 名员工，每月 2000+ 次请假流程
- 全人工操作：查余额→填单→审批→同步考勤
- 痛点：员工单次 15 分钟，审批周期平均 8 小时
- 越权风险：纸质审批单难追溯

### T (Task 任务)
- 设计办公 Agent 覆盖请假全流程自动化
- 支持智能填单、人工审批、自动同步、审计追溯
- 零越权事故，符合企业内部合规要求

### A (Action 行动)

#### 架构设计
- 核心：LangGraph StateGraph 定义 6 节点请假流程
- 审批：interrupt_before submit 节点，超时自动升级
- 工具：MCP 协议接入考勤 API、日历 API、邮件系统
- 双栈：Python LangGraph 编排，Java SAA 企业治理入口
- 可靠性：工具熔断 + 3 次重试 + 延迟队列 + 审计日志

#### 关键技术实现
- Planner Node：LLM 解析自然语言请假请求，提取日期/类型
- Executor Node：查询余额、生成草案、提交系统
- Approval Node：interrupt 等待上级审批，30 分钟超时升级
- Audit Node：全量 tool I/O 记录到 Elasticsearch

#### 代码示意 (Python 伪代码)

- 员工操作时间：15 分钟 → 2 分钟（降幅 87%）

- 零越权事故，审计日志 100% 覆盖
- 员工满意度 NPS 从 62 提升到 91

    delivery story 是你在答辩中的 3 分钟叙事能力。练习框架：

    1. 30 秒 — 问题和背景：5000 员工，2000 次/月，15 分钟/次，8 小时审批
    2. 30 秒 — 架构决策：选 LangGraph 因为 Checkpoint+interrupt，MCP 解耦工具
    3. 60 秒 — 核心实现：状态图展示 6 节点流程，interrupt 审批，熔断降级
    4. 30 秒 — 量化效果：-87% 时间，-81% 审批周期，零事故

    常见问题（FAQ）

    项目没有量化数据怎么办？

    第一问：Agent vs 纯对话 LLM

    agent vs chat 是最基础也最容易暴露短板的问题。面试官想听的不是概念解释（「Agent 能调用工具」），而是你对架构差异的深度理解。

    Agent vs 纯对话架构对比
    flowchart LR
      subgraph Chat[纯对话 LLM]
        Q[用户提问] --> LLM1[LLM 一次推理]
        LLM1 --> A[直接文本回答]
        noteChat[无状态/无工具/无循环]
      end
      subgraph Agent[ReAct Agent]
        Q2[用户任务] --> Think[LLM 多步推理]
        Think --> Act[工具调用]
        Act --> Observe[观察工具返回]
        Observe -->|未完成| Think
        Observe -->|已完成| Result[最终输出]
        noteAgent[有状态+有工具+循环+可恢复]
      end

    interview agent five 核心对比：纯对话一次推理 vs Agent 循环推理+工具+状态。

    满分回答框架（2 分钟）：

    1. 执行模式不同：纯对话是「一次推理一次输出」；Agent 是「Think → Act → Observe」循环，可执行多步才能完成复杂任务
    2. 状态管理不同：纯对话无状态（每次调用独立）；Agent 用 StateGraph + Checkpoint 维护中间状态，中断后可恢复
    3. 工具能力不同：纯对话只能「说」不能「做」；Agent 通过 MCP 调用真实工具，获取观察结果后再决策
    4. 可靠性不同：纯对话出错无法恢复；Agent 有重试、熔断、降级、人工兜底
    5. 可观测性不同：纯对话无中间过程；Agent 每步可审计、可回放、可监控 Token 和延迟

    一句话总结：Agent = 有状态 + 调工具 + 循环推理 + 可恢复的执行引擎，不是「会对话 + 会调用工具」的简单组合。这也是为什么 Agent 适合办公自动化而纯对话 LLM 不适合的原因。

### 1. 技术选型（30 秒）

### 2. 核心组件（60 秒）
| 组件 | 职责 | 技术实现 |
|------|------|---------|
| Planner Node | 用户输入 → 子任务列表 | LLM + 结构化输出 |
| Executor Node | 按依赖执行子任务 | LangGraph 节点 |
| Approval Node | 高风险操作前中断 | interrupt_before |

→ 结果聚合器收集结果
→ 审批节点（interrupt_before）
→ 最终输出 → 审计日志

    第三问：任务规划

    plan question 考察 Agent 如何从模糊的用户需求中生成可执行的计划。面试题例如：「用户说'帮我安排下周的部门会议'，Agent 怎么规划？」。

    回答要点（2 分钟）：

    • Planner 职责：将自然语言输入分解为结构化的子任务列表。每个任务包含 id、description、depends_on、max_retries。以「安排会议」为例：① 查参会人空闲时间 ② 查会议室 ③ 创建会议邀请 ④ 发送通知
    • 检查点机制：每个子任务执行后验证结果。如果第一步「查参会人」发现周三下午全员没空，Agent 不应继续执行后续步骤，而应返回建议时间
    • 动态调整 vs 静态计划：不推荐一次规划全部然后盲执行。推荐每执行完一步后重新评估剩余计划（RePlan），参考 Plan-and-Execute 章节
    • 失败处理：某步骤失败后，Planner 应重新规划绕过失败步骤的剩余路径

    代码框架：

    python
    def planner_node(user_input: str) -> List[SubTask]:
    """面试中展示的 Planner 伪代码"""

            status="pending"
        ))
    return tasks

    追问「如果 Planner 生成 10 步计划但第 7 步失败了怎么办？」—— 回答框架：第 7 步可重试 → 指数退避重试 3 次 → 还失败 → 标记第 7 步为 failed → 调用 Planner 重新规划第 8-10 步（auto-replan）→ 返回部分成功结果 + 失败告警 → 第 7 步转人工处理。关键点：不要让失败步骤阻塞整体流程。

    java
    @Service
public class InterviewPlannerService {

    public List<SubTask> createPlan(String userInput) {
        String prompt = "你是一个办公任务规划师。将以下任务分解为" +
            "可独立执行的子步骤，每个步骤用 JSON 格式返回：\n" + userInput;
        ChatResponse r = chatModel.call(
            new Prompt(List.of(new UserMessage(prompt))));
        return parseTaskList(r.getResult().getOutput().getContent());
    }

    public SubTaskResult executeWithReplan(List<SubTask> plan) {
        for (int i = 0; i < plan.size(); i++) {
            try {
                retryTemplate.execute(ctx -> executeTask(plan.get(i)));
                plan.get(i).setStatus("done");
            } catch (Exception e) {
                // 自动重新规划剩余部分
                List<SubTask> remaining = createPlan(
                    "原计划失败于第" + (i+1) + "步，请重新规划后续");
                return executeWithReplan(remaining);
            }
        }
        return SubTaskResult.success(plan);
    }
}

    第四问：记忆系统

    memory question 考察 Agent 的上下文管理。面试官想知道的是「32K 窗口够用吗？不够怎么办？」。

    层级技术实现容量限制淘汰策略
    L1 短期Redis List 滑动窗口20 轮 / ~4K tokensFIFO，旧消息自动淘汰
    L2 近期摘要LLM 生成的会话摘要50 条 / ~8K tokensLRU + 7 天时间衰减
    L3 长期知识Vector DB（Milvus）无硬限制语义相似度检索 + 衰减

    回答框架（2 分钟）：Agent 记忆分三层。短期存当前会话上下文（20 轮滑动窗口），满了之后 LLM 自动摘要归档到 L2。L2 存近期摘要（50 条、8K tokens），按最近使用优先（LRU）淘汰。L3 存跨会话的用户偏好和历史关键信息，通过向量检索在需要时召回。这样即使会话超过 100 轮，32K 窗口仍然够用。

    追问：「长期记忆很大时向量检索不准怎么办？」—— 答：分库策略。用户偏好向量库（小，高精度）、历史对话摘要库（中，中精度）、企业知识库（大，召回率优先）。还可以用 hybrid search（向量 + 关键词）提高召回准确率。

    第五问：工具失败处理

    tool fail question 是最能区分「做过」和「没做过」Agent 的问题。面试官想知道你在生产环境碰到过什么故障、怎么修的。

    满分回答框架（3 分钟，按优先级）：

    1. 熔断保护（Circuit Breaker）：连续 3 次工具调用失败后打开熔断，后续请求直接走降级而不必继续调用已挂掉的下游。状态机：CLOSED → OPEN → HALF_OPEN → CLOSED。参考 可靠性工程 章节
    2. 指数退避重试：1s → 2s → 4s，最多 3 次。只重试 5xx 类异常，4xx 不重试（参数错了重试也没用）
    3. 降级策略（Degradation）：查询类工具失败返回缓存数据 + 标记「非实时」；写操作类失败存入延迟队列异步重试；关键路径失败触发人工兜底
    4. 人工兜底（Human Fallback）：所有自动策略都失败后，自动生成人工工单进入审批队列。返回用户「您的请求已收到，处理完成后会通知您」
    5. 审计记录（Audit）：全量 tool I/O 记录（入参、出参、耗时、状态），写入 Elasticsearch 用于事后复盘和排查

    面试追问：「工具返回了错误结果但没抛异常，怎么发现？」—— 答：工具契约定义返回格式（tool contract schema），Agent 执行 schema validation。结果不符合契约的走降级流程。例如：邮件工具返回了 「{status: 'sent'}」 但没有 message_id 字段，说明可能未真正发送。

    追问2：「支付工具超时了但实际支付成功了，怎么防止重复支付？」—— 答：幂等键 design pattern。每个写操作工具调用时传入幂等键（idempotency key），服务端已处理则直接返回原结果。Agent 超时后先查幂等键状态再决定是否重试。

    Java 侧幂等设计

    java
    // IdempotentToolWrapper.java
public class IdempotentToolWrapper implements ToolCallback {
    private final ToolCallback delegate;
    private final StringRedisTemplate redis;

    @Override
    public String call(String toolInput) {
        String idempotencyKey = extractKey(toolInput);
        // 检查是否已处理
        String cached = redis.opsForValue().get("idem:" + idempotencyKey);
        if (cached != null) return cached;

        String result = delegate.call(toolInput);
        // 缓存结果，TTL 24h
        redis.opsForValue().set("idem:" + idempotencyKey, result,
            Duration.ofHours(24));
        return result;
    }
}

    问「你最大的失败项目」怎么答？
    答：早期 Agent 没有步数预算和熔断，上线第一天 Agent 陷入死循环不断调考勤 API，把考勤系统打崩了。修复方案：加 max_steps=15 硬限制 + 连续重复步骤检测 + 熔断机制。这个事故让我深刻理解了「Agent 的可靠性」和「Agent 的步数预算」的区别，从此成为我在 Agent 项目中的核心关注点。
    被问到不会的技术栈怎么办？
    坦诚说没用过但有过类似经验。例如：「我没用过 CrewAI，但我深入用过 LangGraph 的多 Agent 模式。两者的原理都是 Supervisor-Worker 架构，我相信给我 2 天可以上手 CrewAI。你可以了解一下我对多 Agent 架构的理解……」然后切换到熟悉的领域。
    五连问的答题时间怎么控制？
    每问控制在 2-3 分钟。先给结论（10s）→ 展开框架（60s）→ 结合项目（60s）→ 预留追问空间（30s）。不要一上来就讲细节，面试官可能打断。如果你发现面试官没有追问，说明你的回答足够完整。
    面试官全程不笑很严肃怎么办？
    正常。面试官需要评估你的能力而不是聊天。他严肃不代表你的回答不好。保持你的节奏，回答完每一个问题后主动问「这个问题您想深入哪个方面？」来引导对话。
