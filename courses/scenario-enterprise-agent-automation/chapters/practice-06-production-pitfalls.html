坑 1：无观察循环

observe loop 是最常见的生产事故。Agent 规划了 3 步以上任务但不观察中间结果，直接按初始计划执行到底。一旦第一步结果和预期不符，后续步骤全部错误。

有观察 vs 无观察循环
flowchart LR
      subgraph Bad[无观察循环 - 错误]
        Plan[初始计划] --> Act1[行动 1]
        Act1 --> Act2[行动 2]
        Act2 --> Act3[行动 3]
        Act3 --> Output[错误输出]
      end
      subgraph Good[每步观察 - 正确]
        Plan2[初始计划] --> Think1[思考]
        Think1 --> Act1G[行动 1]
        Act1G --> Observe1[观察 1]
        Observe1 -->|调整| Think2[再思考]
        Think2 --> Act2G[行动 2]
        Act2G --> Observe2[观察 2]
        Observe2 -->|完成| OutputG[正确输出]
      end

step budget pitfall：无观察循环的 Agent 可能在第一步失败后继续执行。

解决方案：在 LangGraph 中强制每步有条件边，Observe 结果必须影响下一轮的 Think。

python
# observe_guard.py
def observe_guard(state) -> str:
    """强制观察检查：如果没有最新观察结果，不允许继续"""
    if not state.get("latest_observation"):
        return "observe_missing_error"
    # 检查观察结果是否有实质内容
    obs = state["latest_observation"]
    if "ERROR" in obs or "失败" in obs:
        return "observation_failed"
    if state["step"] >= state["max_steps"]:
        return "step_limit_exceeded"
    return "continue"

在 Spring AI Alibaba 中，通过自定义 ToolCallback 包装器强制校验工具返回结果：

java
// ObservedToolWrapper.java
public class ObservedToolWrapper implements ToolCallback {
    private final ToolCallback delegate;

    @Override
    public String call(String toolInput) {
        String result = delegate.call(toolInput);
        // 校验工具实际调用了（防假执行）
        if (result == null || result.isEmpty()) {
            throw new FakeToolCallException(
                "工具 " + getName() + " 返回空结果，疑似假执行");
        }
        return result;
    }
}

坑 2：步数与返回长度预算

return length cap 控制工具返回结果的长度。如果工具返回 10 万字符的文档，Agent 的上下文会瞬间爆满。step budget pitfall 指无步数上限导致的无限循环。

python
# budget_guard.py
from typing import Dict, Any

MAX_TOOL_RETURN_CHARS = 2000
MAX_STEPS = 15

def truncate_tool_return(result: str) -> str:
    """截断工具返回结果"""
    if len(result) > MAX_TOOL_RETURN_CHARS:
        return result[:MAX_TOOL_RETURN_CHARS] + \
               f"\n... (已截断，共 {len(result)} 字符)"
    return result

def check_step_budget(state: Dict[str, Any]) -> str:
    """检查步数预算"""
    if state["step"] >= MAX_STEPS:
        return "budget_exceeded"
    # 检查是否连续重复步骤
    if len(state["actions"]) >= 3:
        last_three = state["actions"][-3:]
        if all(a == last_three[0] for a in last_three):
            return "repeated_action"
    return "ok"

class BudgetEnforcedTool:
    """带预算限制的工具包装器"""
    def __init__(self, tool_fn, name: str):
        self.tool_fn = tool_fn
        self.name = name
        self.call_count = 0

    def call(self, **kwargs) -> str:
        self.call_count += 1
        if self.call_count > 5:  # 单工具调用上限
            return "TOOL_CALL_LIMIT_EXCEEDED"
        result = self.tool_fn(**kwargs)
        return truncate_tool_return(result)

Java 侧用 Spring AOP 实现：

java
@Aspect
@Component
public class BudgetAspect {
    @Around("@annotation(ToolCall)")
    public Object enforceToolBudget(ProceedingJoinPoint pjp) throws Throwable {
        String result = (String) pjp.proceed();
        if (result.length() > 2000) {
            result = result.substring(0, 2000)
                + "\n... (truncated, total " + result.length() + " chars)";
        }
        return result;
    }
}

坑 3：工具假执行检测

fake tool detect 指 Agent 声称调用了工具但实际没有调用，直接在输出中伪造了工具返回结果。这是 LLM 的 hallucination 在工具层面的表现。tool verify 通过以下方式检测：

  • 响应签名校验：工具返回结果必须携带签名或时间戳
  • 调用链追踪：每次工具调用在日志中有唯一 trace_id
  • 结果可验证：工具返回的数据可以通过另一个只读工具交叉验证
python
# fake_tool_detector.py
import hashlib
import json
import time

class FakeToolDetector:
    def __init__(self):
        self.call_log = []

    def verify_call(self, tool_name: str, args: dict, result: str) -> bool:
        """验证工具是否真的被调用了"""
        # 1. 检查结果格式是否符合工具契约
        if not self._validate_schema(tool_name, result):
            return False

        # 2. 检查结果中的时间戳是否接近当前时间
        try:
            data = json.loads(result)
            ts = data.get("timestamp", 0)
            if abs(time.time() - ts) > 5:  # 5 秒偏差
                return False  # 可能是伪造的结果
        except:
            pass

        # 3. 验证签名（如果工具有签名机制）
        if not self._verify_signature(tool_name, result):
            return False

        self.call_log.append({
            "tool": tool_name,
            "args": args,
            "time": time.time(),
            "verified": True
        })
        return True

    def _validate_schema(self, tool_name: str, result: str) -> bool:
        contract = tool_registry.get_contract(tool_name)
        # 检查返回结果是否包含契约中的必需字段
        for field in contract.required_return_fields:
            if field not in result:
                return False
        return True

坑 4：记忆分层与溢出

memory tiering 解决记忆无限堆积问题。memory prune 策略包括滑动窗口、摘要归档和时间衰减。

层级容量淘汰策略存储
L1: 当前会话20 轮 / 4K tokensFIFO 滑动窗口Redis List
L2: 近期摘要50 条 / 8K tokensLRU + 时间衰减Redis String
L3: 长期记忆无限制LLM 摘要 + 向量化Vector DB
记忆坏实践
  • 所有历史消息直接塞入上下文
  • 无摘要、无裁剪
  • 爆窗后 Agent 行为异常
记忆好实践
  • 分层存储：短/中/长期
  • 短期溢出后自动摘要归档
  • 长期记忆向量化后 RAG 检索

为了避免这些坑反复出现，建议将其自动化检测集成到 CI/CD 流水线中。每次 Agent 代码变更时自动运行以下检查：

本章结论

  • 无观察循环：强制每步有条件边，Observe 影响下一轮 Think
  • 步数预算：max_steps 硬限制 + 连续重复步骤检测
  • 返回长度预算：工具结果截断 + 签名校验防止假执行
  • 工具假执行：响应签名、调用链追踪、交叉验证
  • 记忆溢出：分层存储 + 滑动窗口 + 摘要归档

常见问题（FAQ）

工具假执行是 LLM 的问题还是我们代码的问题？
两者都有。LLM 可能「编造」工具结果，但代码层可以通过强制验证（签名、时间戳、追踪链）来检测和阻断。修复重点在代码层。
截断工具返回结果会不会丢失重要信息？
会。所以截断策略是返回摘要 + 截断正文。先让 LLM 看摘要决定是否需要完整内容，需要时再查详情。
记忆分层的容量参数怎么定？
根据模型窗口大小和场景需要。以 32K 窗口为例：system prompt 占 4K、工具返回占 8K、短期记忆占 10K、检索结果占 10K。按这个比例分配。
