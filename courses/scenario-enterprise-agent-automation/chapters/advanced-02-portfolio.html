状态图叙事：一张图讲清架构

state diagram 是 Agent 项目答辩的核心素材。不要用文字描述架构，直接画图。一张好的状态图应包含：状态流转、分支条件、工具调用、审批节点。面试官看一张图的时间不超过 30 秒，所以必须让关键信息一目了然。

CorpAssist 请假流程状态图
stateDiagram-v2
      [*] --> 解析意图: 用户输入「请假」
      解析意图 --> 查余额: 提取日期/类型/天数
      查余额 --> 余额不足: 余额 < 天数
      查余额 --> 生成草稿: 余额充足

workflow narrative 是讲故事的技巧。答辩时不要平铺直叙说「我们实现了请假流程」，而是：先给结论（我们实现了请假自动化，审批周期降低 81%），再画状态图展示架构，最后点关键设计决策（为什么在 submit 前 interrupt、为什么用 Checkpoint 而不是数据库事务）。这种「结论+图+决策」的叙事结构在面试中非常有效。

状态图的绘制要点：

  • 状态命名统一：用动宾结构（查余额、生成草稿、等待审批），不用名词结构（余额查询、草稿生成）
  • 分支条件清晰：每个条件转换标注触发条件（余额充足/不足、通过/驳回）
  • 标注关键设计：用 note 标注面试官可能关注的设计点（interrupt、checkpoint、熔断）
  • 控制节点数量：5-8 个状态最合适，太多让人记不住
差评叙事
  • 「我们开发了一个请假 Agent，用了 LangGraph，还接了 MCP 工具」
  • 「我负责 Agent 的后端开发」
  • 没有架构图，纯文字描述
好评叙事
  • 「我设计了请假 Agent 的完整架构（展示状态图），核心是 interrupt_before 确保审批后才提交」
  • 「通过 MCP 协议接入了 5 个企业系统，Agent 可动态发现工具无需硬编码」
  • 「审批周期从 8 小时降到 1.5 小时，零越权事故」
CorpAssist · 答辩现场 你面试一家公司的 Agent 架构师岗位。面试官问「介绍一下你做过的 Agent 项目」。你打开状态图，30 秒画完图，然后说：「这是我们上线的请假 Agent，核心是 interrupt_before 审批节点。员工操作时间从 15 分钟降到 2 分钟，审批周期从 8 小时降到 1.5 小时。架构上我选择了 LangGraph 的 StateGraph + Checkpoint，因为……」面试官立刻对你产生兴趣。

mcp story 是展示 Agent「连接企业系统能力」的最佳方式。面试官想看到的不只是「你会调 API」，而是「你理解企业系统集成的复杂性并设计了合理的解耦方案」。

STAR 案例：量化你的 Agent 项目

star case 是简历和答辩的黄金模板。STAR 代表 Situaion（场景）、Task（任务）、Action（行动）、Result（结果）。关键在于 Result 必须有量化指标。

markdown
## STAR: CorpAssist 请假 Agent

### S (Situation 场景)
- 5000 名员工，每月 2000+ 次请假流程
- 全人工操作：查余额→填单→审批→同步考勤
- 痛点：员工单次 15 分钟，审批周期平均 8 小时
- 越权风险：纸质审批单难追溯

### T (Task 任务)
- 设计办公 Agent 覆盖请假全流程自动化
- 支持智能填单、人工审批、自动同步、审计追溯
- 零越权事故，符合企业内部合规要求

### A (Action 行动)

#### 架构设计
- 核心：LangGraph StateGraph 定义 6 节点请假流程
- 审批：interrupt_before submit 节点，超时自动升级
- 工具：MCP 协议接入考勤 API、日历 API、邮件系统
- 双栈：Python LangGraph 编排，Java SAA 企业治理入口
- 可靠性：工具熔断 + 3 次重试 + 延迟队列 + 审计日志

#### 关键技术实现
- Planner Node：LLM 解析自然语言请假请求，提取日期/类型
- Executor Node：查询余额、生成草案、提交系统
- Approval Node：interrupt 等待上级审批，30 分钟超时升级
- Audit Node：全量 tool I/O 记录到 Elasticsearch

#### 代码示意 (Python 伪代码)
builder = StateGraph(LeaveState)
builder.add_node("check_balance", check_leave_balance)
builder.add_node("draft", create_leave_draft)
builder.add_node("approval", human_approval_gate)
builder.add_node("submit", submit_to_system)
builder.set_entry_point("check_balance")
builder.add_conditional_edges("draft", router)
graph = builder.compile(checkpointer=cp,
                        interrupt_before=["approval"])

### R (Result 结果)
- 员工操作时间：15 分钟 → 2 分钟（降幅 87%）
- 审批周期：8 小时 → 1.5 小时（降幅 81%）
- 日处理量：2000 次 → 5000 次（扩容后弹性伸缩）
- 零越权事故，审计日志 100% 覆盖
- 员工满意度 NPS 从 62 提升到 91

写 STAR 时的常见错误：「我参与了请假 Agent 的开发」→ 面试官不知道你做了什么。「我独立设计并实现了审批节点的 interrupt 机制，解决了审批安全和超时升级的问题」→ 面试官知道你的具体贡献。

delivery story 是你在答辩中的 3 分钟叙事能力。练习框架：

  1. 30 秒 — 问题和背景：5000 员工，2000 次/月，15 分钟/次，8 小时审批
  2. 30 秒 — 架构决策：选 LangGraph 因为 Checkpoint+interrupt，MCP 解耦工具
  3. 60 秒 — 核心实现：状态图展示 6 节点流程，interrupt 审批，熔断降级
  4. 30 秒 — 量化效果：-87% 时间，-81% 审批周期，零事故
  5. 30 秒 — 反思与改进：如果重做，我会在 Planner 中加入用户确认环节

记住：面试官一天面 5-8 个人，只有讲得清晰、有数据、有深度的项目才能被记住。

简历包装实战：3 个版本对比

portfolio s3 的简历包装不是夸大其词，而是把技术术语转化为业务语言。对比三个版本：

版本简历描述面试官反应
初级参与了请假 Agent 项目的开发哦（无感）
中级用 LangGraph 实现了请假 Agent，接入了 MCP 工具有点意思，继续说说
高级独立架构并落地了 CorpAssist 请假 Agent：LangGraph StateGraph 6 节点流程 + interrupt_before 审批 + MCP 集成 5 系统 + 熔断可靠性。审批周期 8h→1.5h，零事故。这个候选人值得深入聊

关键原则：每个技术名词后面紧跟一个「所以呢？」的答案。用了 LangGraph 所以呢？所以有 Checkpoint 状态恢复。接入了 MCP 所以呢？所以工具可以动态发现、契约版本化管理。

常见问题（FAQ）

项目没有量化数据怎么办？
尽量找对比数据：上线前后业务报表、A/B 测试结果、用户调研 NPS。实在没有就定性描述（如「减少了员工手动填单的出错率 90%」「审批流程从 5 步减少到 3 步」）。关键是要证明 Agent 带来了具体可感知的变化。
面试官问「为什么不用 XX 框架」怎么答？
展示你的选型分析过程：对比表（LangGraph vs CrewAI vs AutoGen vs 传统工作流引擎），从状态管理、审批能力、社区活跃度、企业适配四个维度评分。选型过程本身比选了哪个重要。参考 双栈对照表 的分析方法。
项目只有一个人做，怎么包装？
强调你「独立设计并实现了公司首个办公 Agent 系统，从架构到落地全流程把控」。一个人做意味着你的职责范围更广——架构设计、核心代码、部署运维、演示汇报全是你。这比「团队里做了一小部分」更有说服力。
MCP 和传统 API 的区别面试官听不懂怎么办？
用比喻：传统 API 像给你一本电话簿（你要自己查号码打），MCP 像 Siri（你说「帮我查张三的日历」，手机自动完成）。MCP 的核心是 tool discovery + function calling 的标准化。

生产案例：从 0 到 1 落地 CorpAssist Agent

以下是一个真实的 Agent 落地过程复盘，帮助你理解从架构到交付的完整闭环。某企业 2025 年启动 CorpAssist 办公 Agent 项目，由一位架构师（你）从 0 到 1 完成：

第一阶段：边界评估（第 1-2 周）

  • 梳理 30 个办公操作，按四象限分类出 12 个可自动化、8 个需半自动、10 个保持人工
  • 确定第一个切入场景：请假流程（高频、低风险、好量化）
  • 产出：task boundary 矩阵 + 审批规则 YAML

第二阶段：技术选型与 POC（第 3-4 周）

  • 对比 LangGraph vs CrewAI vs AutoGen，选型表格见 双栈对照
  • 最终选择 LangGraph（Python 编排）+ Spring AI Alibaba（Java 治理）
  • 用 MCP 协议接入 3 个企业系统（考勤、日历、邮件）
  • POC 验证：5 个请假场景全部通过，端到端延迟 < 15s

第三阶段：生产开发（第 5-8 周）

  • 实现 StateGraph 6 节点请假流程（参考 LangGraph 工作流）
  • 配置 interrupt_before 审批节点 + 30 分钟超时自动升级
  • 集成熔断 + 重试 + 审计日志（参考 可靠性工程）
  • Java 侧实现审批页面和审计查询 API

第四阶段：上线与持续优化（第 9-12 周）

  • 灰度 100 人 → 全量 5000 人，期间修复 3 个生产坑（见 生产坑章节）
  • 最终效果：员工操作时间 -87%，审批周期 -81%，零事故
  • 扩展场景：报销流程、工单创建、报告生成

这个案例就是你的 STAR 素材。面试时对照这个时间线讲，比空谈理论有力得多。
