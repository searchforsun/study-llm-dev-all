重试与熔断策略

tool retry 是可靠性的第一道防线。circuit agent 采用熔断模式，防止无效重试打爆下游。

Agent 熔断机制
flowchart TB
      Call[工具调用] --> State{熔断状态}
      State -->|CLOSED\n正常| Execute[执行调用]
      Execute -->|成功| Return[返回结果]
      Execute -->|失败 3 次| Open[OPEN\n熔断打开]
      Open -->|直接拒绝| Fallback[降级策略]
      Open -->|超时后| HalfOpen[HALF_OPEN\n半开探测]
      HalfOpen -->|探测成功| Close[CLOSE\n恢复]
      HalfOpen -->|探测失败| Open

      subgraph Config[熔断配置]
        Threshold[阈值: 3 次失败]
        Timeout[超时: 30s]
        Reset[重置: 60s]
      end

circuit agent 状态机：Closed → Open → Half-Open → Closed。

Python：熔断实现

python
# circuit_breaker.py
import time
from enum import Enum
from typing import Callable, Any

class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(self, threshold: int = 3,
                 reset_timeout: int = 60):
        self.state = CircuitState.CLOSED
        self.threshold = threshold
        self.reset_timeout = reset_timeout
        self.fail_count = 0
        self.last_fail_time = 0

    def call(self, func: Callable, *args, **kwargs) -> Any:
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_fail_time > self.reset_timeout:
                self.state = CircuitState.HALF_OPEN
            else:
                raise CircuitBreakerOpenError("熔断已打开")

        try:
            result = func(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.fail_count = 0
            return result
        except Exception as e:
            self.fail_count += 1
            self.last_fail_time = time.time()
            if self.fail_count >= self.threshold:
                self.state = CircuitState.OPEN
            raise e

# 使用示例
email_breaker = CircuitBreaker(threshold=3, reset_timeout=60)

def send_email_with_breaker(to, subject, body):
    try:
        return email_breaker.call(email_api.send, to, subject, body)
    except CircuitBreakerOpenError:
        return {"error": "邮件服务熔断", "fallback": "已存入延迟队列"}

Java：Resilience4j 熔断

java
// CircuitBreakerConfig.java
@Configuration
public class CircuitBreakerConfig {

    @Bean
    public CircuitBreaker emailCircuitBreaker() {
        return CircuitBreaker.of("emailService",
            CircuitBreakerConfig.custom()
                .failureRateThreshold(50)
                .waitDurationInOpenState(Duration.ofSeconds(60))
                .slidingWindowSize(10)
                .build());
    }

    @Bean
    public CircuitBreaker attendanceCircuitBreaker() {
        return CircuitBreaker.of("attendanceService",
            CircuitBreakerConfig.custom()
                .failureRateThreshold(30)
                .waitDurationInOpenState(Duration.ofSeconds(30))
                .slidingWindowSize(5)
                .build());
    }
}

// 使用示例
@Service
public class ResilientEmailService {

    private final CircuitBreaker circuitBreaker;
    private final EmailClient emailClient;

    public String sendWithCircuitBreaker(EmailRequest req) {
        return circuitBreaker.executeSupplier(
            () -> emailClient.send(req));
    }
}

超时与降级

timeout tool 是防止 Agent 线程耗尽的关键。每个工具调用必须有独立超时。degrade agent 策略定义了降级时的后备方案。

工具超时降级策略
邮件发送30s存入延迟队列，返回「稍后发送」
考勤查询10s返回上次缓存数据，标记「非实时」
日历创建15s生成 ICS 文件让用户手动导入
工单创建20s记录工单草稿到本地，异步同步
python
# timeout_handling.py
import asyncio
from typing import Any, Callable

class TimeoutHandler:
    def __init__(self, default_timeout: int = 30):
        self.default_timeout = default_timeout

    async def call_with_timeout(self, func: Callable,
                                 timeout: int = None,
                                 fallback: Callable = None,
                                 *args, **kwargs) -> Any:
        t = timeout or self.default_timeout
        try:
            return await asyncio.wait_for(func(*args, **kwargs), timeout=t)
        except asyncio.TimeoutError:
            print(f"工具调用超时 ({t}s)，执行降级")
            if fallback:
                return fallback(*args, **kwargs)
            return {"error": "timeout", "message": f"超过 {t} 秒未响应"}

人工兜底（Human Fallback）

human fallback 是可靠性的最后一道防线。当所有自动策略都失败时，必须有路径转到人工处理。

python
# human_fallback.py
class HumanFallback:
    def __init__(self, approval_queue):
        self.queue = approval_queue

    def fallback_to_human(self, task: dict, reason: str):
        """将失败的任务转人工处理"""
        fallback_task = {
            "original_task": task,
            "fallback_reason": reason,
            "status": "pending_human",
            "created_at": time.time(),
            "priority": task.get("priority", "normal")
        }
        self.queue.push(fallback_task)
        return {
            "status": "FALLBACK_TO_HUMAN",
            "message": f"已转人工处理，原因：{reason}",
            "task_id": fallback_task["id"]
        }

    def query_human_result(self, task_id: str) -> dict:
        """查询人工处理结果"""
        return self.queue.get_result(task_id)

故障演练与混沌测试

可靠性不是配置完就完事的，需要通过故障演练来验证。混沌测试（Chaos Engineering）可以有效发现 Agent 可靠性的薄弱环节。

演练场景注入故障预期行为验证方法
考勤 API 超时工具调用延迟 > 30s超时 → 熔断 → 降级返回缓存检查审计日志有 timeout 记录
邮件服务 500连续 5 次返回 5003 次重试 → 熔断 → 延迟队列确认邮件在 60s 后重新发送
Redis 宕机停止 Redis 服务记忆降级到本地缓存Agent 仍可执行，但标记 non-realtime
审批人全离线审批接口一直 pending30 分钟超时 → 自动升级确认升级后的审批人收到通知

Python 侧注入故障的混沌测试框架：

python
# chaos_test.py
import asyncio
from unittest.mock import patch

class ChaosInjector:
    def __init__(self, tool_fn, failure_rate=0.3, delay_range=(0.1, 5.0)):
        self.tool_fn = tool_fn
        self.failure_rate = failure_rate
        self.delay_range = delay_range

    async def call(self, *args, **kwargs):
        import random, time
        delay = random.uniform(*self.delay_range)
        await asyncio.sleep(delay)

        if random.random() < self.failure_rate:
            raise TimeoutError(f"混沌注入：工具超时 {delay:.1f}s")

        return await self.tool_fn(*args, **kwargs)

# 在测试中使用
chaos_email = ChaosInjector(email_api.send, failure_rate=0.3)
result = await chaos_email.call("test@corp.com", "subject", "body")
assert result.get("fallback"), "熔断未触发"

本章结论

  • 熔断机制保护下游：Closed → Open → Half-Open 状态机
  • 每个工具设置独立超时，超时后执行降级策略
  • 降级兜底：缓存 → 延迟队列 → 人工兜底
  • 所有工具调用记录审计日志（入参、出参、耗时、成功/失败）
  • 可靠性配置集中管理推荐 YAML 或配置中心

常见问题（FAQ）

熔断打开后，Agent 怎么知道恢复了？
Half-Open 状态会定期发送探测请求。如果探测成功则关闭熔断，失败则继续 Open。探测频率可配，建议 60s 一次。
降级返回「稍后处理」，用户怎么知道结果？
降级策略应该包含通知机制：异步任务完成后通过 WebSocket/邮件/钉钉通知用户结果。Agent 应同步返回「你的请求已提交，处理完成后会通知你」。
审计日志会不会太大？
全量审计确实会产生大量数据。建议：全量写 Elasticsearch（保留 90 天），抽样写成本更低的存储（如 S3 冷存）。监控系统只关注错误日志和性能指标。
