Plan-and-Execute 架构

plan execute 是 Agent 任务规划的经典模式。与 ReAct 的「边走边想」不同，Plan-and-Execute 先由 Planner 生成完整计划，再由 Executor 逐条执行。适合步骤明确的办公场景。

Plan-and-Execute 架构图
flowchart TB
      Input[用户任务] --> Planner[Planner Node\n生成执行计划]
      Planner --> Plan[任务计划列表]
      Plan --> Executor[Executor Node]
      Executor --> Step1[子任务 1\n查考勤数据]
      Executor --> Step2[子任务 2\n算出勤率]
      Executor --> Step3[子任务 3\n生成 PDF 报告]
      Executor --> Step4[子任务 4\n邮件发送]
      Step1 --> Check{检查点}
      Step2 --> Check
      Step3 --> Check
      Check -->|通过| Next[下一步]
      Check -->|失败| Retry[重试/降级]
      Next --> Step2 --> Step3 --> Step4
      Step4 --> Output[最终输出]

planner node 生成计划，Executor 按序执行，Checkpoint 中间验证。

CorpAssist · 月度报告生成 用户：「生成研发部上月考勤报告，算出勤率和迟到率，生成 PDF，发邮件给总监」。Planner 分解：① 查询考勤原始数据 ② 计算统计指标 ③ 生成 PDF 报告 ④ 查询总监邮箱 ⑤ 发送邮件。每步独立执行并验证检查点。

Python：Plan-and-Execute 实现

python
# plan_execute_agent.py
from langgraph.graph import StateGraph, END
from typing import TypedDict, List, Optional

class SubTask(TypedDict):
    id: str
    description: str
    status: str
    result: Optional[str]
    retry_count: int
    max_retries: int
    depends_on: List[str]

class PlanState(TypedDict):
    input: str
    plan: List[SubTask]
    current_step: int
    errors: List[str]

def planner_node(state: PlanState) -> dict:
    prompt = f"将以下任务分解为子任务：\n{state['input']}\n"
    prompt += "每个子任务含 id、description、depends_on。"
    response = llm.invoke(prompt)
    tasks = parse_subtasks(response)
    return {"plan": tasks, "current_step": 0}

def executor_node(state: PlanState) -> dict:
    task = state["plan"][state["current_step"]]
    # 检查依赖是否完成
    for dep_id in task["depends_on"]:
        dep = next(t for t in state["plan"] if t["id"] == dep_id)
        if dep["status"] != "done":
            return {"errors": state["errors"] + [f"依赖 {dep_id} 未完成"]}
    try:
        result = execute_tool(task["description"])
        task["status"] = "done"
        task["result"] = result
    except Exception as e:
        if task["retry_count"] < task["max_retries"]:
            task["status"] = "pending"
            task["retry_count"] += 1
        else:
            task["status"] = "failed"
        state["errors"].append(str(e))
    return {"plan": state["plan"]}

def router(state: PlanState) -> str:
    task = state["plan"][state["current_step"]]
    if task["status"] == "done":
        state["current_step"] += 1
        if state["current_step"] >= len(state["plan"]):
            return "finish"
        return "continue"
    elif task["status"] == "failed":
        return "fail"
    return "retry"

builder = StateGraph(PlanState)
builder.add_node("planner", planner_node)
builder.add_node("executor", executor_node)
builder.set_entry_point("planner")
builder.add_edge("planner", "executor")
builder.add_conditional_edges("executor", router, {
    "continue": "executor",
    "retry": "executor",
    "finish": END,
    "fail": END
})

Java：Spring AI Alibaba 计划执行

java
// PlanExecuteService.java
@Service
public class PlanExecuteService {
    private final DashScopeChatModel chatModel;
    private final ToolExecutor toolExecutor;

    public List<SubTask> plan(String userInput) {
        String prompt = "将以下任务分解为子任务列表，每个子任务用 JSON：" +
            "{\"id\":\"1\",\"description\":\"...\",\"depends_on\":[]}\n任务：" + userInput;
        ChatResponse r = chatModel.call(new Prompt(List.of(new UserMessage(prompt))));
        return parseTaskList(r.getResult().getOutput().getContent());
    }

    public ExecutionResult execute(List<SubTask> tasks) {
        var results = new ArrayList<TaskResult>();
        for (SubTask task : tasks) {
            try {
                task.setStatus(SubTask.Status.RUNNING);
                String result = toolExecutor.execute(task.getDescription());
                task.setStatus(SubTask.Status.DONE);
                task.setResult(result);
                results.add(new TaskResult(task.getId(), result, true));
            } catch (Exception e) {
                if (task.getRetryCount() < task.getMaxRetries()) {
                    task.setRetryCount(task.getRetryCount() + 1);
                    task.setStatus(SubTask.Status.PENDING);
                    results.add(execute(List.of(task)).getResults().get(0));
                } else {
                    task.setStatus(SubTask.Status.FAILED);
                    results.add(new TaskResult(task.getId(), e.getMessage(), false));
                }
            }
        }
        return new ExecutionResult(results);
    }
}

规划中的边界情况处理

任务规划在实际执行中常遇到边界情况，需要针对性处理：

  • 空计划：用户输入模糊（如「帮我看看」），Planner 无法生成有意义的计划 → 反馈用户需要更具体的描述
  • 循环依赖：步骤 A 依赖 B，B 又依赖 A → Planner 生成时检测循环引用，如果出现则合并步骤 A 和 B
  • 并行爆炸：20 个无依赖的子任务全部并行执行 → 限制并行度（max_concurrent=10），其余排队
  • 长计划截断：Planner 生成了 30 步的计划 → 按优先级分阶段，第一阶段只执行前 10 步，第一阶段完成后生成第二阶段
  • 计划与执行不一致：执行结果和 Planner 预期不符 → 停止当前计划，让 Planner 重新规划
python
def detect_cycle(tasks: List[SubTask]) -> bool:
    """检测计划中的循环依赖"""
    graph = {t["id"]: t["depends_on"] for t in tasks}
    visited = set()

    def dfs(node_id: str, path: set) -> bool:
        if node_id in path:
            return True  # 发现循环
        if node_id in visited:
            return False
        visited.add(node_id)
        path.add(node_id)
        for dep in graph.get(node_id, []):
            if dfs(dep, path):
                return True
        path.remove(node_id)
        return False

    return any(dfs(t["id"], set()) for t in tasks)

def merge_cyclic_tasks(tasks: List[SubTask]) -> List[SubTask]:
    """检测到循环依赖时合并相关步骤"""
    if not detect_cycle(tasks):
        return tasks
    # 简化实现：将循环中的步骤合并为一个
    merged = SubTask(id="merged", description="合并后的步骤",
                     depends_on=[], status="pending")
    return [merged] + [t for t in tasks
            if t["id"] not in [m["id"] for m in [merged]]]

子任务划分与依赖管理

subtask split 遵循四项原则：单一职责（每个子任务只做一件事）、可验证（有明确完成标准）、可独立重试（失败不影响其他）、依赖显式化。以报告生成为例，task decompose 将任务分拆为依赖明确的多步：

json
{
  "plan": [
    {"id":"1","description":"查询研发部上月考勤原始数据","depends_on":[],"max_retries":3},
    {"id":"2","description":"计算出勤率=出勤天数/工作日","depends_on":["1"],"max_retries":2},
    {"id":"3","description":"计算迟到率=迟到次数/总天数","depends_on":["1"],"max_retries":2},
    {"id":"4","description":"生成 PDF 报告含图表","depends_on":["2","3"],"max_retries":1},
    {"id":"5","description":"查询总监邮箱并发送报告","depends_on":["4"],"max_retries":2}
  ]
}

步骤 2 和 3 无依赖关系可并行执行，步骤 4 必须等 2 和 3 完成，步骤 5 等 4 完成。LangGraph 的 fan-out 模式（见 多 Agent 协作）可实现并行子任务。

不推荐 — 一步到位
  • Prompt: 「生成一份上月部门报告」期望一次产出
  • 无中间验证，出错全盘重来
  • 半年后无法追溯数据来源
推荐 — 分解执行
  • Planner 分解为 5 个子任务
  • 每步独立执行 + 检查点验证
  • 单步失败仅重试该步
  • 每步持久化结果可审计

检查点与失败恢复

checkpoint 是 Plan-and-Execute 的安全机制。每步执行后验证结果，LangGraph 的 Checkpoint 自动保存中间状态（详见 LangGraph 工作流实现）。failure recovery 策略如下：

失败类型策略实现
工具超时指数退避重试 3 次 → 跳过Spring Retry
数据格式异常重新解析 → 默认值替代Schema Validation
前置任务失败跳过当前 → 完成其他 → 告警Conditional Edge
LLM 输出异常重新生成 → 截断 → 备选方案Fallback Model
python
# recovery_handler.py
import time
from functools import wraps

def with_retry(max_retries=3, delay=1.0, fallback=None):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exception = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    last_exception = e
                    if attempt < max_retries - 1:
                        time.sleep(delay * (2 ** attempt))
            return fallback(*args, **kwargs) if fallback else None
        return wrapper
    return decorator

@with_retry(max_retries=2, fallback=lambda x: {"default": True})
def query_attendance(dept: str) -> dict:
    return api.call(f"/attendance/{dept}")

本章结论

  • Plan-and-Execute = Planner 分解 + Executor 逐行执行
  • 子任务划分遵循单一职责、可验证、可独立重试、依赖显式化
  • 检查点是失败恢复的基础，LangGraph Checkpoint 自动保存中间状态
  • 失败策略：重试 → 跳过 → 降级 → 告警
  • 依赖关系决定执行顺序和并行度

常见问题（FAQ）

Planner 生成的计划错了怎么办？
两种策略：执行前让用户确认计划（human-in-the-loop）；执行中某步失败回馈 Planner 重新规划。推荐后者用于高频场景。
子任务可以并行执行吗？
可以。无依赖的子任务可并行（如同时查考勤和会议数据）。LangGraph 支持 fan-out 模式。注意并行度不超过 API 限流阈值。
Plan-and-Execute 和 ReAct 怎么选？
步骤明确用 PnE（报表生成），探索性强用 ReAct（客服问答）。也可混合：Planner 定方向，Executor 内部用 ReAct 微调。
