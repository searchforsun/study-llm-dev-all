代码补全从触发到采纳的全链路，TTFT 和采纳率是两个核心指标。

采纳率追踪实现

python 
# metrics_tracker.py — 采纳率和性能指标追踪
from pydantic import BaseModel
from datetime import datetime
from typing import Optional
import time

class CompletionEvent(BaseModel):
    """代码补全事件"""
    event_id: str
    user_id: str
    project: str
    file_path: str
    language: str
    cursor_line: int
    context_chars: int           # 上下文长度
    completion_length: int       # 补全的代码长度
    ttft_ms: int                 # 首 Token 延迟
    total_latency_ms: int        # 总延迟
    accepted: Optional[bool]     # 是否被采纳
    timestamp: str = ""

class MetricsAggregator:
    """指标聚合器"""

    def __init__(self):
        self.events: list[CompletionEvent] = []

    def record_event(self, event: CompletionEvent):
        self.events.append(event)

    def acceptance_rate(self) -> float:
        """计算采纳率"""
        total = len([e for e in self.events if e.accepted is not None])
        accepted = sum(1 for e in self.events if e.accepted)
        return accepted / total if total > 0 else 0.0

    def p99_latency(self) -> int:
        """计算 P99 延迟"""
        latencies = sorted(e.total_latency_ms for e in self.events)
        if not latencies:
            return 0
        idx = int(len(latencies) * 0.99)
        return latencies[idx]

    def avg_ttft(self) -> float:
        """平均首 Token 延迟"""
        values = [e.ttft_ms for e in self.events]
        return sum(values) / len(values) if values else 0.0

    def report(self) -> dict:
        return {
            "total_completions": len(self.events),
            "acceptance_rate": round(self.acceptance_rate() * 100, 1),
            "avg_ttft_ms": round(self.avg_ttft(), 1),
            "p99_latency_ms": self.p99_latency(),
            "avg_completion_length": int(sum(e.completion_length for e in self.events) / len(self.events)) if self.events else 0,
        }

# 使用示例
if __name__ == "__main__":
    agg = MetricsAggregator()
    agg.record_event(CompletionEvent(
        event_id="1", user_id="u1", project="payment-service",
        file_path="PaymentService.java", language="java",
        cursor_line=42, context_chars=800, completion_length=120,
        ttft_ms=180, total_latency_ms=1200, accepted=True,
    ))
    agg.record_event(CompletionEvent(
        event_id="2", user_id="u1", project="payment-service",
        file_path="OrderController.java", language="java",
        cursor_line=15, context_chars=600, completion_length=0,
        ttft_ms=350, total_latency_ms=2500, accepted=False,
    ))
    print(agg.report())

延迟对开发体验的影响

代码补全的延迟直接影响开发者的心流。研究表明：当补全延迟超过 300ms 时，开发者会感觉「卡顿」；超过 1s 时，大部分开发者会关闭自动补全功能。

TTFT 范围用户体验预估采纳率优化重点
< 100ms即时反馈，无感知35-45%保持
100-200ms流畅，偶尔微卡30-35%优化模型推理
200-500ms明显等待20-30%缓存+小模型
> 500ms打断思路< 15%降级方案
CorpAssist 实践

代码补全的 TTFT 优化是核心工程挑战。我们采用三级加速：L1 缓存命中（< 30ms）、L2 小模型推理（< 100ms）、L3 大模型推理（< 300ms）。超过 300ms 的请求返回「补全不可用」，避免打断开发者。代码问答则放宽到 5s，因为开发者可以等待更完整的答案。

常见问题（FAQ）

代码补全采纳率 30% 算好吗？
算及格。GitHub Copilot 公开数据显示采纳率约 25-35%。但不同类型代码差异很大：样板代码（getter/setter）采纳率可达 60%+，算法逻辑代码可能 < 20%。建议按文件类型和代码复杂度分层统计采纳率。
代码问答的准确率怎么评估？
使用人工标注的问答对（至少 200 条）。每条包含：问题、预期答案、预期引用的代码行。评估指标：答案正确率 + 引用准确率（引用的代码行是否正确）。CorpAssist 用 gpt-4o 作为评估模型，人工抽检 20% 确保评估质量。
代码审查的检出率和误报率怎么平衡？
检出率（召回问题的比例）和误报率（报告的问题中实际不是问题的比例）是 trade-off。推荐先追求低误报率（< 20%），再逐步提升检出率。因为高误报率会让开发者忽略所有审查意见。CorpAssist 的策略：高风险问题（安全漏洞）优先检出，风格问题可选择性报告。

本章结论

  • 三大场景：代码补全（低延迟）、代码问答（中延迟、高准确率）、代码审查（全量分析）。
  • 核心指标：采纳率、TTFT、P99 延迟、准确率。每个指标有明确的业务意义。
  • 延迟影响采纳率：TTFT > 200ms 时采纳率明显下降，200ms 是代码补全的生命线。
  • 安全是红线：代码助手绝不能生成有安全漏洞的代码，需要独立的审核流水线。
  • 基线驱动：上线前先建立指标基线，每次优化后对比。
