为什么选择FastAPI而不是Flask？

FastAPI原生async/await支持高并发SSE连接，Pydantic集成自动请求验证和OpenAPI文档。性能测试显示FastAPI比Flask快3倍以上，特别适合AI服务的流式响应场景。

LangGraph在什么场景下比LangChain更合适？

LangGraph适合需要多步状态管理和条件路由的复杂工作流(如Agent)。LangChain适合简单的链式调用(如RAG流水线)。在S4模块中，以LangGraph为主、LangChain为辅。

不推荐
  • 单体应用所有功能一体部署
  • 无状态管理每次请求独立
  • 无缓存每次重新计算
推荐
  • 微服务按业务领域拆分
  • Redis持久化会话状态
  • 多层缓存(本地+Redis+向量库)

        stream_response(prompt),

from typing import Protocol

class ToolProtocol(Protocol):
    async def execute(self, params: dict) -> dict: ...

class MCPTool:
    def __init__(self, name: str, schema: dict):
        self.name = name
        self.schema = schema
    async def execute(self, params: dict) -> dict:
        async with httpx.AsyncClient() as c:
            r = await c.post(f'/tools/{self.name}', json=params)
            return r.json()

判断练习

使用同步Flask替代异步FastAPI是否合理？

不合理。Flask同步阻塞无法处理AI流式响应和SSE连接。高并发场景下性能下降明显。FastAPI的async/await是更好的选择。

是否需要为每个微服务单独写Dockerfile？

是的。每个微服务独立容器化，通过Docker Compose编排。这有利于独立扩缩容和部署。建议使用多阶段构建减小镜像体积。
