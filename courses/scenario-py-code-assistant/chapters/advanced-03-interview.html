代码助手是 AI 工程化方向的高频面试题。面试官通常从五个维度考察：架构分层、代码切分策略、检索优化、RAG vs 微调、评估与安全。CorpAssist 的实践覆盖了所有维度，本章帮你系统化准备。

问题核心答案参考章节生产坑提示
1. 代码助手整体架构？ 四层：IDE 插件（上下文提取）→ API 网关（鉴权/限流）→ 推理层（FIM 补全 + RAG 问答）→ 质量网关（AST+安全）。FastAPI 异步服务，Python 双栈（推理 + 缓存 + 质检）。 架构设计 单一服务包揽所有功能 → 运维耦合
2. 代码怎么切分和索引？ 按函数/类级别 AST 切分，Python 用 ast.parse。双通道索引：符号表（PostgreSQL 存函数名/文件路径）+ 向量库（Milvus 存代码语义）。Git 增量索引检测 diff 更新，先删后插。 索引流水线 行级切分切断函数体 → 检索不完整
3. 检索不准怎么排查？ 四步法：1) 检查召回结果是否相关（不相关改 Embedding 或 chunk 策略）；2) 召回相关但 LLM 没用（改 Prompt 或加 Rerank）；3) 加符号索引兜底（函数名精确匹配）；4) 检查切分策略。60% 的检索问题由切分不合理导致。 混合检索 只用向量检索 → 漏掉精确匹配
4. RAG vs 微调？ 代码补全用微调（FIM 模型如 CodeLlama），因为需要低延迟和高生成速度。代码问答用 RAG，因为需要检索最新代码。组合方案：微调负责生成风格和规范，RAG 提供最新代码上下文。 本章综合 RAG 用于补全 → 延迟不达标
5. 怎么评估？ 四层评估：语法层（AST 通过率 ≥95%）、语义层（LLM 评判准确率 ≥80%）、采纳层（IDE 埋点 ≥40%）、安全层（Semgrep 违规率 ≤3%）。每个优化必须 A/B 对比基线数据。 准确率评估 只看采纳率 → 忽略安全问题

面试回答模板：检索不准

【原理 + 实现 + 坑】三段式回答模板

面试官：「代码助手检索结果不准，你怎么排查？」

1. 原理（30s）：
"检索不准可能出现在三个环节：切分策略、Embedding 模型、Rerank 排序。切分不合理（如行级切断函数）导致召回片段语义不完整是最常见的原因。"

2. 工程实现（60s）：
"我采用四步排查法：第一步检查召回结果——如果召回的代码跟问题不相关，说明 Embedding 或 chunk 策略有问题，会改 AST 函数级切分；第二步检查 LLM 是否使用了召回结果——如果召回了但 LLM 没用，说明 Prompt 需要优化；第三步加符号索引兜底——函数名精确匹配解决常见问题；第四步加 BGE-Reranker 重排序。"

3. 生产坑（30s）：
"CorpAssist 早期遇到过的问题是 chunk_size 设得太大（2000 字符），函数定义被截断导致检索不完整后改为函数级切分，检索准确率从 60% 升到 82%。另外要注意 embedding 模型维度——256 维比 512 维快但精度低，需要根据业务平衡。"

场景模拟：检索不准排查对话

面试官：「用户反馈代码助手检索到的代码跟问题没关系，你怎么排查？」

候选人：「我会按四步排查。第一步，检查召回结果是否相关——如果召回的 top-5 代码块跟问题语义不相关，说明问题出在 Embedding 或 chunk 策略上。我会先检查 chunk 策略，因为 60% 的检索问题由切分不合理导致。第二步，如果召回相关但 LLM 生成的答案不准确，说明 Prompt 没告诉 LLM 如何使用召回结果，这时优化 Prompt 模板。第三步，如果语义检索找不到，加符号索引兜底——函数名精确匹配。第四步，加 BGE-Reranker 对召回结果重排序。」

面试官：「chunk 策略具体怎么影响检索？」

候选人：「行级切分（如 50 行一刀切）很可能切断一个函数——函数头在第 40 行、主体在第 60 行，检索时两个 chunk 都不完整。函数级切分（AST 提取）保证每个 chunk 是一个完整的函数或类定义。CorpAssist 的实践：从行级切分改为 AST 函数级切分后，检索准确率从 60% 升到 82%。对于超长函数（200+ 行），我们在函数内按逻辑块再次切分（用空行或 return 语句分界）。」

场景模拟：RAG vs 微调决策

面试官：「代码助手的补全功能和问答功能，分别用什么技术方案？为什么？」

候选人：「补全用微调，问答用 RAG。补全要求延迟 <200ms，微调模型（如 CodeLlama 7B 做 FIM 微调）用 vLLM 部署，单次推理 50-100ms。问答需要检索最新代码库内容，必须用 RAG——代码库每天都有新提交，微调跟不上更新频率。组合方案效果最好：微调模型保证补全的生成质量，RAG 提供最新的代码上下文。」

面试官：「微调模型会不会遗忘旧知识（灾难性遗忘）？」

候选人：「会。CorpAssist 的做法是混合训练：每次微调时保留 20% 的通用代码补全数据（来自开源代码），防止模型只记住企业特定模式而遗忘通用编程能力。同时设置学习率 1e-5，低学习率减少遗忘。每季度做一次通用能力的回归评测。」

场景模拟：评估指标冲突

面试官：「采纳率从 40% 降到 35%，但用户满意度评分反而从 4.0 升到 4.2，你怎么分析？」

候选人：「采纳率下降说明用户更少接受我们的补全建议，但满意度上升说明用户对展示的补全质量更满意。可能的原因是我们提高了质量网关的阈值，拦截了更多低质量补全（采纳率分母变了）。我会看两组数据：一是质量网关的拦截率是否升高（从 8% 到 15%）；二是被采纳的补全中，修改后采纳的比例是否下降（说明展示的补全质量确实提高了）。如果确实是质量网关优化导致，这是好现象——用户看到更少但更准确的补全。」

先想 10 秒：面试官问「代码补全和代码问答为什么用不同的模型架构」？

代码补全需要低延迟（<200ms）和高吞吐，使用 FIM 微调的因果语言模型（如 CodeLlama），用 vLLM 部署。代码问答需要检索能力，使用 RAG 架构的 chat 模型（如 gpt-4o-mini），首 token 延迟 500ms 可以接受。两者对延迟和检索的要求不同，所以模型架构不同。

面试追问准备

常见追问集锦
  • 追问 1：「你的代码助手支持多少种语言？」→ 核心支持 Python/Java/JS/TS/Go/Rust，其他语言通过通用行级切分兜底。
  • 追问 2：「大仓库（100 万+ 文件）怎么索引？」→ 分层索引：热仓（活跃文件）全量入 Milvus，冷仓（历史文件）用符号索引 + 摘要向量。
  • 追问 3：「补全结果用户不采纳的原因可能有哪些？」→ 三大原因：补全不相关（上下文提取问题）、补全有语法错误（质量网关没拦住）、补全风格不一致（style 参数没传）。
  • 追问 4：「代码助手和 Copilot 的核心差异是什么？」→ Copilot 提供通用补全，CorpAssist 面向企业私有代码库，支持 RAG 问答、政策合规、多仓库权限隔离。
  • 追问 5：「如果让你用 6 个月从零搭建一个代码助手，怎么规划？」→ 第 1-2 月：索引流水线 + 补全 API；第 3-4 月：问答 API + 质量网关；第 5 月：IDE 插件 + 效果评测；第 6 月：生产化（限流/熔断/SLA）+ 交付文档。

面试回答禁忌

  • 别只说原理不说实现：「我们用了 RAG」太笼统，要说「用了 Milvus + text-embedding-3-small + BGE-Reranker」。面试官想听具体的技术选型和原因。
  • 别回避生产坑：主动提遇到的坑和解决过程，展示工程经验。如「上线第一周发现质量网关误拦截了合法代码，原因是安全扫描规则太粗——把 JSON 中的 eval 字段名也匹配了」。
  • 别忽视非功能需求：主动提延迟优化、安全合规、运维监控。面试官关注的不只是功能实现，还有生产化能力。
  • 别把架构画得太简单：如果画出只有两三个模块的架构图（IDE → 后端），说明没有深入思考。至少画到四层。
