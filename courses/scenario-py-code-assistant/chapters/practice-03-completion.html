为什么要学本章

代码补全是代码助手最核心的功能。FIM 补全需要精确构建上下文拼装策略（prefix/suffix 格式），并通过风格约束确保补全的代码符合项目规范。CorpAssist 的补全服务每天处理 2 万+ 次补全请求，P99 延迟 < 2s。

学完你能
  • 构建 FIM（Fill-in-the-Middle）格式的代码补全 Prompt
  • 实现上下文选择策略（光标附近 + 当前函数 + 相关 import）
  • 用 style 参数约束补全代码的风格规范
  • 实现后处理管道（缩进修复、代码块清理）
本章先记住 3 件事
  • FIM 格式：prefix + suffix 描述光标位置，模型补全 middle 内容。
  • 上下文窗口有限：优先给最相关的代码（光标附近 > 当前函数 > import）。
  • 风格约束在 prompt 中：用自然语言描述风格规则，比微调更灵活。

FIM 补全服务

CorpAssist · 代码补全场景 开发者在 VS Code 中编写一个 FastAPI 路由，在函数体内输入 return 触发补全。IDE 发送 prefix（return 前的代码）和 suffix（return 后到函数结束的代码）到补全服务。服务构建 FIM Prompt 后调用 LLM，返回补全的 return 语句。

FIM 补全：IDE 提供光标上下文的 prefix 和 suffix，服务构建标准 FIM Prompt。

Python FIM 补全服务

python
# completion_service.py — CorpAssist FIM 补全服务
from fastapi import FastAPI
from pydantic import BaseModel, Field
from openai import OpenAI
from typing import Optional
import re

app = FastAPI(title="CorpAssist 代码补全服务", version="1.0.0")

class CompletionRequest(BaseModel):
    prefix: str = Field(..., description="光标前的代码")
    suffix: str = Field(default="", description="光标后的代码")
    language: str = Field(default="python", description="语言标识")
    max_tokens: int = Field(default=256, ge=16, le=2048)
    temperature: float = Field(default=0.2, ge=0.0, le=1.0)
    style: Optional[str] = Field(default=None, description="风格约束: google / project / none")
    context: Optional[str] = Field(default=None, description="额外上下文(当前函数/导入)")

class CompletionResponse(BaseModel):
    completion: str
    finish_reason: str

STYLE_GUIDES = {
    "google": """
遵循 Google Python 风格指南：
- 类型注解：所有函数参数和返回值标注类型
- 缩进：4 空格
- 变量命名：小写+下划线，描述性名称
- 函数注释：reStructuredText 格式
""",
    "project": """
遵循项目已有的代码风格：
- 与上下文的缩进一致
- 变量名风格与已有代码相同
- 不使用未 import 的库
""",
}

class FIMCompletionService:
    """FIM 补全服务"""

    FIM_PREFIX = "<|fim_prefix|>"
    FIM_SUFFIX = "<|fim_suffix|>"
    FIM_MIDDLE = "<|fim_middle|>"

    def __init__(self):
        self.llm = OpenAI(base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")

    def build_prompt(self, req: CompletionRequest) -> str:
        style_guide = STYLE_GUIDES.get(req.style, "")
        context_block = f"\n额外上下文（当前文件）：\n{req.context}\n" if req.context else ""

        # 截断控制（防止 token 超限）
        max_prefix = 3000
        max_suffix = 1000
        prefix = req.prefix[-max_prefix:] if len(req.prefix) > max_prefix else req.prefix
        suffix = req.suffix[:max_suffix] if len(req.suffix) > max_suffix else req.suffix

        return f"""你是一个代码补全模型。请补全光标  处的代码。

文件类型：{req.language}
{style_guide}{context_block}

光标前的代码：
```{req.language}
{prefix}
```

光标后的代码：
```{req.language}
{suffix}
```

请只输出光标处要补全的代码，不要包含光标前后的内容。不要用代码块包裹输出："""

    def complete(self, req: CompletionRequest) -> CompletionResponse:
        prompt = self.build_prompt(req)
        resp = self.llm.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": prompt}],
            temperature=req.temperature,
            max_tokens=req.max_tokens,
        )
        raw = resp.choices[0].message.content
        completion = self.post_process(raw, req)
        return CompletionResponse(completion=completion, finish_reason="stop")

    def post_process(self, completion: str, req: CompletionRequest) -> str:
        code = completion.strip()
        # 清理代码块标记
        code = re.sub(r'^```[\w]*\n?', '', code)
        code = re.sub(r'\n```$', '', code)
        # 缩进修复：继承 prefix 末行的缩进级别
        lines = code.split('\n')
        if req.prefix and lines:
            prefix_last_line = req.prefix.rstrip().split('\n')[-1] if req.prefix else ""
            indent = len(prefix_last_line) - len(prefix_last_line.lstrip())
            if indent > 0:
                indented = []
                for i, line in enumerate(lines):
                    if line.strip():
                        indented.append(' ' * indent + line.lstrip())
                    else:
                        indented.append(line)
                code = '\n'.join(indented)
        return code[:req.max_tokens * 4]  # token 还原为字符上限

completer = FIMCompletionService()

@app.post("/v1/completions", response_model=CompletionResponse)
async def complete(req: CompletionRequest):
    return completer.complete(req)

@app.post("/v1/completions/stream")
async def complete_stream(req: CompletionRequest):
    from fastapi.responses import StreamingResponse
    import json

    prompt = completer.build_prompt(req)

    async def generate():
        resp = completer.llm.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": prompt}],
            temperature=req.temperature,
            max_tokens=req.max_tokens,
            stream=True,
        )
        full = ""
        for chunk in resp:
            if chunk.choices[0].delta.content:
                token = chunk.choices[0].delta.content
                full += token
                yield f"data: {json.dumps({'token': token})}\n\n"
        final = completer.post_process(full, req)
        yield f"data: {json.dumps({'done': True, 'completion': final})}\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")

if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8001)
不推荐
  • prefix/suffix 不加截断，token 超限被截断导致上下文不完整
  • 无缩进修复，补全的代码缩进错乱
  • 无风格约束，补全结果和项目风格不一致
  • 代码块标记未清理，用户看到 ```python 标记
推荐
  • prefix 截断 3000 字符，suffix 截断 1000 字符
  • 继承 prefix 末行缩进级别修复补全代码
  • style 参数注入 Google/Project 风格指南
  • 正则清理 ``` 和语言标记
先想 10 秒：FIM 和传统的「根据上文生成下文」有什么本质区别？

传统生成只有上文（prefix），模型不知道光标后面还有代码。FIM 同时给 prefix 和 suffix，模型知道「光标后面还有代码，我需要补全中间部分」——这更符合 IDE 中的补全场景（用户在代码中间插入）。FIM 格式的补全质量比纯 prefix 高出 20-30%（根据论文结果）。

本章结论

  • FIM 格式：prefix + suffix 描述光标位置，比纯 prefix 生成更准确。
  • 上下文截断：prefix 3000 字符 + suffix 1000 字符，优先保留光标附近内容。
  • 风格约束：style 参数注入 Google/Project 风格指南，prompt 级别约束。
  • 后处理管道：缩进修复 + 代码块标记清理 + 字符上限截断。
