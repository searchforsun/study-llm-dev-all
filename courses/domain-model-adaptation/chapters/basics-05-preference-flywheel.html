为什么要学本章

SFT 让模型学会「正确回答」，但不能控制模型「偏好哪种回答」。Preference flywheel 将线上用户反馈 → 标注 → 偏好训练 → 上线 → 再收集形成闭环，持续提升回答质量。DPO 和 RLAIF 使偏好对齐不再需要 RLHF 的复杂训练基础设施。

数据飞轮：从 SFT 到持续改善

Preference flywheel 是应用侧微调区别于一次性训练的核心概念。模型上线不是终点，而是数据收集的起点。经典的飞轮包含五个阶段：

CorpAssist · 话术飞轮（90 秒） CorpAssist v1 上线后，客服团队发现模型在处理「员工情绪激动」场景时话术不够温和。团队从线上日志中收集了 200 条此类 badcase，每条的模型回答与人工修订版组成偏好对，用 DPO 训练 v1.1，上线后「情绪安抚」场景的 CSAT 提升了 23%。

坏例收集的三种来源

坏例标注 是飞轮的起始点。CorpAssist 中三种主要来源：

来源触发条件CorpAssist 示例优先级
CSAT 评分低用户评价 < 3 分「回答太生硬」P0
人工修正客服手动修改模型回答话术不含标准开场白P0
安全告警内容安全检测触发模型建议「投诉到消协」P0
采样抽查随机 1% 对话抽样回答不完整或偏离主题P1

DPO 原理：直接偏好优化

DPO（Direct Preference Optimization） 由 Rafailov et al. (2023) 提出，核心思想是将偏好概率直接转化为训练监督信号，不再需要 RLHF 中的 Reward Model 和 PPO 两个额外组件。

对比传统 RLHF：

  • RLHF：SFT → 训练 Reward Model → PPO 优化策略（三阶段）。
  • DPO：SFT → 直接基于偏好对优化策略（两阶段，去掉 RM + PPO）。

DPO Loss 函数：L = -E[log σ(β * (r(x,y_w) - r(x,y_l)))]，其中 r(x,y) = log(π(y|x) / π_ref(y|x))。

在应用侧，你不需要理解 DPO 的数学推导，但需要掌握其应用条件：

  • 需要 偏好对数据：(prompt, chosen_response, rejected_response) 三元组。
  • chosen 和 rejected 必须来自同一 prompt。
  • chosen 必须明显优于 rejected——模糊的偏好对会噪声化训练信号。
python 
# dpo_data.py — DPO 偏好数据格式
import json

# DPO 训练数据格式：每条含 prompt, chosen, rejected
dpo_sample = {
    "prompt": "员工提问：我想请 3 天事假，怎么操作？",
    "chosen": "您好，我是 CorpAssist 小C。事假申请流程如下：1) 登录 OA 系统 2) 选择「请假申请」3) 事假类型、起止时间 4) 提交后自动流转至主管审批。请问您还需要其他帮助吗？",
    "rejected": "请 3 天事假需要在 OA 提交申请，找主管审批。"
}

# 批量转换成 LLaMA-Factory/DashScope 格式
def convert_to_dpo_format(preference_pairs: list, output_path: str):
    """转换偏好对为标准 DPO 训练格式"""
    records = []
    for pair in preference_pairs:
        records.append({
            "conversations": [
                {"from": "human", "value": pair["prompt"]},
            ],
            "chosen": {"from": "gpt", "value": pair["chosen"]},
            "rejected": {"from": "gpt", "value": pair["rejected"]},
        })
    with open(output_path, "w", encoding="utf-8") as f:
        for r in records:
            f.write(json.dumps(r, ensure_ascii=False) + "\n")
    print(f"生成 {len(records)} 条 DPO 训练数据: {output_path}")

if __name__ == "__main__":
    pairs = [
        {"prompt": "员工提问：请假流程？",
         "chosen": "您好，请假请登录 OA 系统。",
         "rejected": "要请假，去 OA。",
         },
    ]

RLAIF：LLM 替代人工标注

RLAIF（Reinforcement Learning from AI Feedback） 用 LLM 代替人工标注偏好对。核心流程：对于每个 prompt，让 LLM 生成两个回答，然后让另一个 LLM（Judge LLM）判断哪个更好。RLAIF 可将标注成本降低 10-100 倍。

python 
# rlaif_generate.py — RLAIF 偏好对自动生成
from openai import OpenAI
import json

client = OpenAI()
REFERENCE_LLM = "qwen-plus"    # 用于生成回答
JUDGE_LLM = "qwen-max"         # 更强的模型做 Judge

def generate_preference_pair(prompt: str) -> dict:
    """用 RLAIF 自动生成偏好对"""
    # Step 1: 用参考 LLM 生成两个回答
    resp1 = client.chat.completions.create(
        model=REFERENCE_LLM,
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7,
    )
    resp2 = client.chat.completions.create(
        model=REFERENCE_LLM,
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7,
    )
    answer_a = resp1.choices[0].message.content
    answer_b = resp2.choices[0].message.content

    # Step 2: 用 Judge LLM 判断哪个更好
    judge_resp = client.chat.completions.create(
        model=JUDGE_LLM,
        messages=[{"role": "user", "content": f"""对比两个客服回答：

用户问题：{prompt}
回答A：{answer_a}
回答B：{answer_b}

哪个更符合 CorpAssist 客服话术标准？回复 'A' 或 'B'。"""}],
        temperature=0,
    )
    winner = judge_resp.choices[0].message.content.strip()

    chosen = answer_a if "A" in winner else answer_b
    rejected = answer_b if "A" in winner else answer_a
    return {"prompt": prompt, "chosen": chosen, "rejected": rejected}

# 批量生成
prompts = [
    "我的电脑蓝屏了怎么办？",
    "年假有几天？",
    "食堂投诉电话是多少？",
]
for p in prompts:
    pair = generate_preference_pair(p)
    print(f"Prompt: {p}")
    print(f"Chosen: {pair['chosen'][:50]}...")
先想 10 秒：RLAIF 生成的数据一定比人工标注好吗？

不一定。RLAIF 的优势是成本低、速度快，但 Judge LLM 可能产生系统性偏好（如偏好更长回答、偏好某个话术模板）。建议 RLAIF 数据 + 人工抽检 10%，确保质量。

RLHF / DPO / RLAIF 的应用边界

RLHF boundary 是应用工程师最常混淆的概念。在企业应用场景中：

技术需要什么适合场景不适合场景
SFT标注 (instruction, output)行为对齐、话术固化复杂偏好判断
DPO偏好对 (prompt, chosen, rejected)话术偏好、安全对齐超过 3 个偏好等级的场景
RLAIFLLM + Judge LLM大规模偏好数据生成需要人工判断的价值对齐
RLHF (PPO)Reward Model + PPO 训练复杂奖励函数、多目标优化应用侧（成本高，算法团队负责）

实践建议：CorpAssist 项目从 SFT + DPO 开始，飞轮成熟后引入 RLAIF 做扩展。RLHF 不需要在应用侧实现，调用平台 API（如百炼）即可。

CorpAssist 话术对齐实战

话术对齐 是 CorpAssist 应用偏好飞轮的核心目标：让模型在「标准回答」和「非标准回答」之间学会选择标准回答。

python 
# tone_dpo.py — CorpAssist 话术 DPO 训练脚本
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model
from trl import DPOTrainer  # pip install trl
import json

model_name = "Qwen/Qwen2.5-7B-Instruct"

# 加载模型 + LoRA
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype="auto")
tokenizer = AutoTokenizer.from_pretrained(model_name)

lora_config = LoraConfig(
    r=16, lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.05,
)

# 加载 DPO 数据
def load_dpo_data(path: str):
    data = []
    with open(path) as f:
        for line in f:
            obj = json.loads(line)
            data.append({
                "prompt": obj["prompt"],
                "chosen": obj["chosen"],
                "rejected": obj["rejected"],
            })
    return data

train_data = load_dpo_data("data/corpassist_dpo_v1.jsonl")

# DPO 训练（核心参数）
dpo_trainer = DPOTrainer(
    model=model,
    ref_model=None,  # 自动使用冻结基座
    args=dict(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,
        learning_rate=5e-6,      # DPO lr 通常比 SFT 小
        num_train_epochs=3,
        max_length=2048,
        max_prompt_length=512,
        beta=0.1,                # DPO 温度参数
    ),
    train_dataset=train_data,
    tokenizer=tokenizer,
)
dpo_trainer.train()
dpo_trainer.save_model("./outputs/corpassist-dpo-v1")
常见误区
  • SFT 直接做偏好对齐：SFT 只能学「正确回答」，不能学「偏好什么回答」。
  • 偏好对质量差：chosen 和 rejected 差异不大时，DPO 训练效果趋近于零。
  • 飞轮无闭环：收集了 badcase 不训练，或训练了不上线，飞轮空转。
  • 忽略通用能力：DPO 训练可能导致模型在其他能力上退步，需要回归测试。

本章结论

  • 数据飞轮：线上 → 收集 → 标注 → 训练 → 上线，持续迭代模型。
  • DPO：直接偏好优化，不需要 Reward Model 和 PPO，适合应用侧。
  • RLAIF：LLM 替代人工标注偏好对，降低成本 10-100 倍。
  • 话术对齐：用 DPO 让模型学会选择标准、温和的客服话术。
