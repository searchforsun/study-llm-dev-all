为什么要学本章

工程岗最大误区：什么都微调。事实知识更新应走 RAG，只有行为对齐（语气、格式、流程固化）才需要 SFT。本章用 CorpAssist 客服话术场景建立决策树，避免「该 RAG 却微调」的资源浪费。

  • 为 CorpAssist 客服场景列出至少 3 个必须微调和 3 个不应微调的子任务
  • 区分 knowledge update 与 behavior align 两条适配路径
本章先记住 3 件事
  • RAG 第一：95% 的领域知识注入先用检索，微调只补 RAG 做不到的行为对齐。
  • 事实 vs 行为：知识更新→RAG；语气/格式/流程→SFT；两者可叠加。
  • 适配成本：一次 LoRA 微调约 2-8 小时 GPU 训练 + 数据标注 + 评测，远贵于 prompt 调优。

先想 10 秒：如果团队已经用了 RAG，为什么还需要微调？

RAG 控制事实正确性，但无法控制表达方式。例如 RAG 检索到「年假 5 天」后，基座模型可能回答「根据制度你有 5 天年假」——但 CorpAssist 要求话术是「您好，根据公司规定您可享受 5 天带薪年假，请问需要帮您申请吗？」。后者是行为固化，必须 SFT。

RAG 基线：先做最简单的

RAG first 是应用侧微调的第一原则。在考虑任何微调之前，先用 Prompt + 检索 建立一个可运行的基线。基线能暴露真实瓶颈：是知识不足（RAG 可解）、格式不对（Prompt 工程可解）还是行为不符（需要 SFT）。

不推荐
  • 直接上手微调，没有基线对比
  • 用微调解决知识更新问题
  • 微调后不对比 RAG 基线效果
  • 所有问题「微调一把梭」
推荐
  • 先用 RAG + Prompt 建立基线
  • 记录基线 win rate 后再微调
  • 微调后与基线做 A/B 对比
  • 每轮只改一个变量

RAG 基线评估 checklist

  • 召回率：检索是否覆盖正确答案？低于 60% 先优化 RAG 而非微调。
  • 格式匹配度：Prompt 能否控制输出格式？不能→考虑 Few-shot 或 SFT。
  • 语气一致性：模型是否总以「企业客服」口吻回答？不稳定→需要行为对齐。
  • 错误模式：是「不知道」还是「乱说」？不知道→补充知识库；乱说→SFT 或 RLHF。
python 
# baseline_rag.py — CorpAssist RAG 基线评估
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from typing import List, Dict

SYSTEM_TEMPLATE = """你是一个专业的企业客服助手 CorpAssist。
请基于以下【参考资料】回答用户问题。
如果资料不足，请说「我需要查证后回复」，不要编造。

参考资料：
{context}

用户：{question}
CorpAssist："""

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM_TEMPLATE),
    ("human", "{question}"),
])

def evaluate_baseline(test_cases: List[Dict], retriever) -> Dict:
    """评估 RAG 基线的准确率和格式匹配度"""
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.1)
    chain = prompt | llm
    results = {"correct": 0, "format_ok": 0, "hallucination": 0}
    for case in test_cases:
        docs = retriever.invoke(case["question"])
        context = "\n".join(d.page_content for d in docs)
        answer = chain.invoke({"context": context, "question": case["question"]})
        # 人工/自动评估
        is_correct = case["expected"] in answer.content
        if is_correct:
            results["correct"] += 1
    return results

何时必须微调：行为对齐场景

behavior align 是微调不可替代的场景。当要求的输出风格、话术流程、安全边界与基座模型默认行为不一致时，Prompt 工程无法稳定控制。典型行为对齐场景包括：

  • 固定话术模板：必须用「您好，我是 CorpAssist 小C」开头，结尾带「感谢咨询」。
  • 流程固化：投诉处理必须走「确认情绪→记录问题→给出方案→确认满意」四步。
  • 安全边界：客服不得回答「建议投诉」等对抗性内容，应转接人工。
  • 品牌语调：专业、温和、拒绝口语化表达。

CorpAssist 案例：客服适配选型

corpassist case 是企业客服领域的典型微调场景。我们用一个具体案例贯穿全课：某企业需要为员工提供 HR + IT + 行政 一站式问答服务。

通过决策树分析，我们将 CorpAssist 能力拆解为三层：

层级能力选型CorpAssist 案例
L1 知识 制度查询、文档检索 RAG 「年假几天？公积金比例？WiFi 密码？」
L2 话术 固定语气、开场/结束语 SFT 「您好，我是 CorpAssist 小C」
L3 流程 多步骤固化工单流程 LoRA 投诉处理 → 记录 → 转交 → 回访

每一层有独立的评测指标：L1 看 Recall@K，L2 看话术合规率，L3 看流程完整度。后续章节会逐一展开。

常见误区
  • 该 RAG 却微调：花 2 小时训练模型记住静态知识，不如花 2 分钟更新知识库。
  • 无基线对比：微调后不对比 RAG 基线，无法判断提升来自哪里。
  • 语料不匹配：用通用语料微调领域模型，导致模型「通用能力下降、领域能力未提升」。
  • 行为与知识混为一谈：用同一份数据同时做 RAG 和 SFT，评估时无法定位问题。

本章结论

决策先行，RAG 第一，微调兜底。

  • 决策树：事实知识走 RAG，行为对齐走 SFT，两者叠加用 Hybrid。
  • RAG 基线：微调前必须建立可对比的 RAG 基线，避免盲目投入。
  • 行为对齐：微调的核心价值是固化输出风格和流程，不是注入知识。
  • CorpAssist 分层：L1 知识 RAG、L2 话术 SFT、L3 流程 LoRA 三层独立演进。
