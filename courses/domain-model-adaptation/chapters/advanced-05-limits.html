已完成微调的天花板与边界

约 40 分钟 · 阶段：生产与治理 · 能力：识别微调能力的边界，知道「不做什么」

为什么要学本章

知道微调能做什么，更要清楚微调不能做什么。微调的天花板 来自 LoRA 的低秩约束、数据量的有限性、以及基座模型的能力上限。不了解边界，就会在错误的方向上投入资源。

学完你能
  • 识别微调的 3 大天花板：算法岗 vs 应用岗 差异
  • 理解 本课不覆盖 的内容（RLHF 深度、预训练、算法创新）
  • 知道什么场景不应微调
本章先记住 3 件事
  • LoRA 不是万能：低秩约束限制学习复杂度，某些任务需要全量微调。
  • 基座能力是天花板：微调不能给模型「注入」它原本不具备的推理能力。
  • 本课是应用视角：不教算法创新、不教 RLHF 实现、不教预训练。

微调的三层天花板

CorpAssist · 微调边界反思（90 秒） CorpAssist 团队讨论：能不能用微调让模型学会「解决复杂 IT 故障」？结论：不能。模型基座（Qwen2.5-7B）的推理能力有限，LoRA 只能在已有能力上调整行为，不能创造新能力。正确的方案是用 RAG 检索故障处理手册 + Chain-of-Thought Prompt。

容量天花板 LoRA 学不会复杂推理 低秩约束 (r=16) 限制了 adapter 容量 加大 r 或改用全量微调

LoRA 的低秩约束

LoRA 的低秩约束 意味着它只能学到「低秩」的权重变化。对于某些需要大幅改变权重分布的任务，LoRA 可能不够：

  • 适合 LoRA：话术微调、风格迁移、简单格式固化—这些变化在权重空间中维度较低。
  • 不适合 LoRA：全新知识注入、复杂推理流程、大规模行为转变—这些需要高维权重变化。
不应微调
  • 用微调让模型学会新语言
  • 用微调让 7B 模型做高中数学推理
  • 用数千条数据注入数万条知识
  • 用 LoRA 改变模型的安全价值观
应该微调
  • 客服话术对齐（行为固化）
  • 特定输出格式（JSON/步骤模板）
  • 小规模知识注入（< 1000 条）
  • 偏好对齐（DPO 调整回答偏好）

典型「不要微调」案例分析

通过 3 个真实案例加深对微调边界的理解：

案例提出的方案为什么不 Work正确的方案
用微调让模型学会多语言翻译 准备 5000 条中英文对照客服对话做 SFT LoRA 低秩约束 + 数据量不足以改变语言的底层表示 使用专门的翻译模型或调用翻译 API
用微调让 7B 模型通过 CPA 考试 用 10000 条注会题目和答案做 SFT 基座模型 7B 的知识容量和推理能力有限 换用大基座 + CoT Prompt + RAG 查条文
用微调解决所有用户投诉 把客服标准回答全部做成 SFT 数据 语料只能覆盖已知场景，新问题会 fallback SFT + RAG 知识库 + 人工转接兜底

每次在决定「是否微调」之前，先问自己三个问题：

  1. 这个问题基座模型能干吗？如果基座模型本身就不会，微调也教不会。
  2. 我的数据量够吗？低于 200 条或超过 50000 条都要警惕。
  3. 问题会变化吗？如果下周知识就更新，应该用 RAG 而不是微调。

本课不覆盖：算法岗 vs 应用岗

算法岗 vs 应用岗 的分工决定了本课的内容边界。本课定位是应用工程师的微调指南，不涉及以下算法岗核心工作：

领域算法岗（本课不覆盖）应用岗（本课覆盖）
预训练Pretraining from scratch, data curation基座模型选型与评估
RLHF 实现Reward Model 训练、PPO 实现DPO 使用、偏好数据准备
模型架构Attention 改进、MoE 设计LoRA/QLoRA 配置与调参
分布式训练DeepSpeed ZeRO-3、FSDP 底层LLaMA-Factory 配置多卡训练
量化算法量化感知训练（QAT）PTQ 量化（AWQ/GGUF）使用
数据合成Self-Instruct 算法、数据蒸馏研究LLM 辅助数据标注与格式转换

角色边界：应用工程师应该理解原理、知道怎么选参数、能排障，但不需要从零实现 PPO 或设计新的注意力机制。本课帮助你在面试中清晰说明自己的定位——你是「用微调解决问题的工程师」，而非「发明微调算法的研究员」。

何时不应微调：mustNotCover 复习

text 
【场景 1】需要注入 10000+ 条新知识
 → 方案：RAG 知识库，而非微调

【场景 2】需要模型做复杂数学推理
 → 方案：更换更强基座 + CoT Prompt，而非微调

【场景 3】模型已有安全偏见过高
 → 方案：安全对齐数据 + DPO，而非再用 SFT

【场景 4】基座模型本身能力不足
 → 方案：更换基座模型，而非微调来「补能力」

【场景 5】业务尚未稳定，需求每周变化
 → 方案：先用 Prompt 工程快速迭代，稳定后再微调

【场景 6】数据无法出企业内网
 → 方案：私有化部署训练，而非用百炼托管

本章结论

  • 三层天花板：容量天花板、数据天花板、基座天花板。
  • LoRA 边界：适合行为对齐，不适合大规模知识注入或复杂推理。
  • 应用岗定位：本课覆盖应用工程师需要的微调知识，不涉及算法创新。
  • 不微调清单：6 种场景不应使用微调。
