面试中「微调项目」是高频考点。面试官不仅关心你「会不会训练」，更关心你做决策的逻辑和排障的框架。简历故事 需要从技术细节中提炼出清晰的 项目叙事。

STAR 项目叙事：微调项目模板

STAR 方法 是面试回答的经典框架：Situation（背景）、Task（任务）、Action（行动）、Result（结果）。面试官通常期待你在 2-3 分钟内讲完一个完整故事。

面试 · CorpAssist 微调项目叙事（120 秒） 面试官：「请分享一个你做过的大模型微调项目。」你的回答可以用 CorpAssist 案例，按 STAR 框架组织。
STAR 环节面试话术示例埋下的考点
Situation 「我们公司有一个内部客服助手 CorpAssist，采用基座模型 + RAG 架构。用户反馈回答太 AI 味，缺少企业客服的温暖感。」 RAI first 原则
Task 「目标是通过微调让模型学会专业的客服话术，提升 CSAT 评分 5% 以上。」 指标量化
Action 「我经过决策树分析确定需要行为对齐，采集了 2000 条客服对话做 SFT 语料。选 LoRA (r=16) 参数量小、迭代快。用百炼平台训练+部署，配置了 A/B 灰度。」 选型逻辑+技术栈
Result 「上线后话术合规率从 72% 提升到 96%，CSAT 提升 8%，人工转接率下降 5%。2000 个员工月咨询量中约 30% 由模型自动解决。」 数字说话
STAR 模板中埋知识点
  • S 中埋：「先用 RAG 基线验证了知识不足不是瓶颈」展现 rag-first 原则。
  • T 中埋：「把 CSAT 从 4.0 提升到 4.3」展现指标驱动。
  • A 中埋：「对比了 LoRA r=16 和 r=32，选了 r=16 防过拟合」展现实验意识。
  • R 中埋：「安全违规率 0.08% 低于基线的 0.3%」展现安全意识。

高频追问 1：选型决策类

面试官追问的「为什么」系列，考察你决策的深度。

text 
Q: 为什么选 LoRA 而不是全量微调？
A: 3 个原因。1) 成本：全量微调 7B 需要 56GB 显存，LoRA 只需 16GB，训练时间 1/4。
   2) 数据量：我们只有 2000 条语料，全量微调容易过拟合。
   3) 迭代：LoRA adapter 可以快速切换，全量微调每次需重新训练基座。

Q: 为什么选 Qwen2.5-7B 而不是更大模型？
A: 1) 推理成本：14B 模型推理成本是 7B 的 2-3 倍，但性能提升在话术任务上不明显。
   2) 部署：7B 单卡 A100 可部署，14B 需要多卡。
   3) 数据量：2000 条语料不足以充分利用 14B 模型的容量。

Q: 为什么先 RAG 后微调？
A: RAG 基线帮助我们排除了「知识不足」这个因素。如果基座模型+RAG 的回答
   事实准确但话术不对，我们才确定需要微调解决行为对齐问题。

高频追问 2：排障过程类

面试官想了解你的系统化排障能力。

text 
Q: 微调后模型在通用能力上下降了，你怎么排查？
A: 排查四步走：
   1) 确认下降幅度：用 MMLU/C-Eval 子集对比基座和微调模型得分。
   2) 分析原因：通用能力下降通常是语料偏差——CorpAssist 语料只有客服话术。
   3) 缓解方案：混合 10-20% 通用语料做数据增强。
   4) 门禁预防：在回滚门禁中添加通用能力指标，下降 > 3% 自动拦截。

Q: 训练 loss 正常下降但 CSAT 没提升，为什么？
A: Loss 下降 ≠ 业务指标提升。3 种可能原因：
   1) Loss 衡量的是 token 预测准确度，不是业务指标。
   2) 训练集分布与线上分布不一致。
   3) 话术在评测集上提高了，但线上用户问的问题更复杂。
   排查方法：对比线下 win rate 和线上 CSAT，看是否存在过度拟合评测集。

Q: 如果微调后模型输出乱码或重复，你怎么排查？
A: 按数据→模型→推理三层排查：
   1) 数据层：检查 JSONL 格式是否正确，chat template 是否匹配。
   2) 模型层：检查 tokenizer 和模型是否匹配，pad_token 是否设置。
   3) 推理层：检查 temperature/top_p 是否合理，max_new_tokens 是否太小。

高频追问 3：效果衡量类

用指标说话 是面试中区分「做过」和「深入做过」的关键。

面试问题差回答好回答
微调效果怎么样？「效果还不错」「话术合规率 72%→96%，Win Rate 65%，CSAT 4.0→4.3，安全违规率下降了 3 倍」
为什么觉得效果变好了？「模型回答更像客服了」「我们做了 A/B 测试，5% 流量对比微调模型和基座模型。微调模型的 CSAT 4.3 vs 基座 4.0，p < 0.01 有统计显著性」
最大的挑战是什么？「训练不好调」「数据漂移——公司发布新政策后模型回答旧政策。我们建立了漂移检测 + 自动触发再训练的机制」

微调陷阱类反问

微调陷阱 是面试官用来判断你是否「真做过」的反问。

text 
陷阱 1: 「3000 条语料够不够微调？」
应对：不是看数量是看质量。3000 条高质量、无偏、覆盖全面的语料 > 30000 条
低质量数据。关键是类别平衡 + 多样性。

陷阱 2: 「LoRA 训练时 loss 一直不降怎么办？」
应对：检查数据格式（chat template 是否正确）、learning rate（是否太大/太小）、
adapter 参数量（r 是否太小）。先在小数据集上跑通再全量。

陷阱 3: 「微调后模型不会说人话了，怎么修？」
应对：这是灾难性遗忘。缓解方案：1) 混合通用语料 10-20% 2) 使用 LoRA 而非
全量微调 3) 在 DPO 阶段用偏好对纠正。

陷阱 4: 「同一个问题每次回答不一样？」
应对：设置 temperature=0（纯贪心解码）。如果还是不同，检查 dropout
是否在推理时未关闭（model.eval()）。

简历中的微调项目描述

简历故事 需要在 2-3 句话内展现技术深度和业务价值。

不推荐
  • 「使用 LoRA 微调了 Qwen 模型」
  • 「负责大模型微调训练」
  • 「用 2000 条数据做了 SFT」
推荐
  • 「主导 CorpAssist 客服助手 LoRA 微调，话术合规率 72%→96%，CSAT ↑ 8%，覆盖 2000+ 员工」
  • 「搭建微调全流程：语料清洗 → MinHash 去重 → SFT → 评测门禁 → A/B 灰度，3 周内从需求到上线」
  • 「设计数据飞轮：线上 badcase → DPO 训练 → 评测 → 上线，月迭代 2 次，持续改善 csat」

工程师角度 的微调项目应该强调：决策逻辑 + 工程落地 + 业务指标，而非算法创新。

面试常见场景模拟：你能答好吗？

以下是面试中可能出现的高频剧情，你能否在 30 秒内组织好回答？

text 
场景 1：面试官质疑「你们用 LoRA 而不是全量微调，是不是因为技术不行？」
应对技巧：把「不行」转化成「理性选择」。
回答框架：
- 承认全量微调的潜力：「全量微调理论上上限更高，但需要更多数据和算力。」
- 给出 LoRA 的选型理由：「2000 条语料下 LoRA 效果接近全量，且迭代快、易回滚。」
- 反问：「如果数据增长到 2 万条，我也会考虑全量微调。」

场景 2：面试官问「你的微调项目效果不好，怎么向老板汇报？」
应对技巧：数据驱动，诚实但积极。
回答框架：
- 先说数据：「CSAT 从 4.0 提升到 4.2，提升 5%，未达到 10% 的目标。」
- 分析原因：「我们排查后发现知识库更新不及时是主要原因。」
- 给出改进计划：「下一轮补充新制度数据，预期 CSAT 达到 4.4+。」

场景 3：面试官追问「为什么不用 RLHF？」
应对技巧：展示你了解但不越界。
回答框架：
- 认可 RLHF 的价值：「RLHF 在 OpenAI 等场景效果显著。」
- 给出不用的理由：「应用侧需要 SFT + DPO 就够用，RLHF 的 reward model 训练成本 > 收益。」
- 展示清晰边界：「本课定位是应用工程，算法团队如果需要可以独立实现 PPO。」

本章结论

  • STAR 叙事：背景→任务→行动→结果，2-3 分钟讲完。
  • 选型决策：准备好「为什么 LoRA」「为什么 7B」「为什么先 RAG」的回答。
  • 排障框架：数据→模型→推理三层排查。
  • 指标说话：用具体数字代替模糊描述。
