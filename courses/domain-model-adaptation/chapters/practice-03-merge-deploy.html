为什么要学本章

LoRA 训练产物是 adapter，但生产推理框架通常需要合并后的完整模型。合并 LoRA、GGUF 量化、AWQ 量化、vLLM 部署——每个环节做错都会导致推理性能下降或兼容性问题。

学完你能
  • 用 PEFT 合并 LoRA adapter 并保存为完整模型
  • 用 llama.cpp / AutoAWQ 做 模型量化
  • 用 vLLM 部署合并后的模型并启动推理服务
  • 从 Spring 侧调用 私有化部署的模型
本章先记住 3 件事
  • 合并 = 推理兼容：合并后的模型不需要 PEFT 依赖，任何框架都可加载。
  • 量化省显存：AWQ 4-bit 比 bf16 省 4 倍显存，精度损失 < 1%。
  • vLLM 是生产首选：PagedAttention + 连续批处理，吞吐比 HF 高 10 倍。

合并 LoRA Adapter

合并 LoRA 将 adapter 权重写回基座模型，生成一个完整的、不依赖 PEFT 的模型文件。合并后可用任何推理框架加载。

CorpAssist · Adapter 合并部署（60 秒） CorpAssist 的 LoRA adapter 训练完成（corpassist-tone-v1-lora）。团队决定用 vLLM 部署私有化推理。但 vLLM 不直接支持 PEFT adapter 加载，需要先将 adapter 合并回 Qwen2.5-7B 基座。
python 
# merge_lora.py — 合并 LoRA adapter 到基座模型
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel
import torch

# 配置
BASE_MODEL = "Qwen/Qwen2.5-7B-Instruct"
ADAPTER_PATH = "./outputs/corpassist-lora-v1/checkpoint-500"
OUTPUT_PATH = "./outputs/corpassist-lora-v1-merged"

# 加载基座
print("加载基座模型...")
model = AutoModelForCausalLM.from_pretrained(
    BASE_MODEL,
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# 加载 adapter
print("加载 adapter...")
model = PeftModel.from_pretrained(model, ADAPTER_PATH)

# 合并并卸载
print("合并 adapter...")
merged_model = model.merge_and_unload()

# 保存
print("保存合并后模型...")
merged_model.save_pretrained(OUTPUT_PATH, safe_serialization=True)

# 保存 tokenizer
tokenizer = AutoTokenizer.from_pretrained(BASE_MODEL)
tokenizer.save_pretrained(OUTPUT_PATH)

# 验证
from transformers import pipeline
pipe = pipeline("text-generation", model=merged_model, tokenizer=tokenizer)
result = pipe("你好，我是 CorpAssist", max_new_tokens=50, do_sample=False)
print(f"验证输出: {result[0]['generated_text']}")
print(f"合并完成！模型保存在: {OUTPUT_PATH}")

合并后模型 的大小与基座模型相同（7B ≈ 14GB bf16），但可以直接被推理框架加载。

不推荐
  • 用 PEFT 在推理时动态加载 adapter（依赖 PEFT 库）
  • 合并时用 fp32 而非 bf16（显存翻倍）
  • 合并后不验证直接部署（可能加载失败）
推荐
  • 先合并再部署，任何框架都可加载
  • 用 bf16 或 fp16 合并，节省显存
  • 合并后运行推理验证后再部署

GGUF 量化：llama.cpp 生态

GGUF 是 llama.cpp 推出的模型格式，专为 CPU 和混合设备推理优化。GGUF 量化可大幅减小模型体积，适合边缘部署。

bash 
# 用 llama.cpp 将合并模型转为 GGUF Q4_K_M
# 1. 安装 llama.cpp
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
make -j

# 2. 转为 GGUF 格式
python convert.py ../outputs/corpassist-lora-v1-merged/ \
    --outfile ../outputs/corpassist-q4.gguf \
    --outtype q4_k_m

# 3. 验证推理
./main -m ../outputs/corpassist-q4.gguf \
    -p "您好，我是 CorpAssist 小C" \
    -n 50

# 常见量化级别：q4_k_m（推荐）, q5_k_m（更高精度）, q8_0（几乎无损）
量化级别模型大小 (7B)显存精度损失推荐场景
bf1614 GB~16 GB无GPU 推理
Q8_07 GB~8 GB< 1%GPU 高精度
Q4_K_M4 GB~5 GB~2-3%推荐均衡选择
Q2_K2.5 GB~3 GB~5-10%边缘设备

AWQ 量化：GPU 推理优化

AWQ (Activation-aware Weight Quantization) 是专为 GPU 推理设计的量化方法。与 GGUF 不同，AWQ 保留 GPU 友好的张量布局，vLLM 原生支持。

python 
# quantize_awq.py — AWQ 量化
from awq import AutoAWQForCausalLM  # pip install autoawq
from transformers import AutoTokenizer

model_path = "./outputs/corpassist-lora-v1-merged"
quant_path = "./outputs/corpassist-lora-v1-awq"

# 加载模型
model = AutoAWQForCausalLM.from_pretrained(
    model_path, device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# AWQ 量化配置
quant_config = {
    "quant_method": "awq",
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,              # 4-bit 量化
    "version": "gemm",
}

# 量化
model.quantize(tokenizer, quant_config=quant_config)

# 保存
model.save_quantized(quant_path, safetensors=True)
tokenizer.save_pretrained(quant_path)

print(f"AWQ 量化完成: {quant_path}")
# 大小: ~4.5 GB (vs 14 GB bf16)

vLLM 部署推理服务

vLLM 是目前最流行的大模型推理引擎。其 PagedAttention 算法解决了 KV Cache 碎片化问题，配合连续批处理（continuous batching），吞吐量可达 HF pipeline 的 10-20 倍。

bash 
# 启动 vLLM 推理服务（合并后模型）
python -m vllm.entrypoints.openai.api_server \
    --model ./outputs/corpassist-lora-v1-merged \
    --dtype bfloat16 \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.9 \
    --port 8000

# 或加载 AWQ 量化模型
python -m vllm.entrypoints.openai.api_server \
    --model ./outputs/corpassist-lora-v1-awq \
    --quantization awq \
    --dtype auto \
    --max-model-len 4096 \
    --port 8000
python 
# vllm_client.py — 调用 vLLM 推理服务
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed",
)

def ask_corpassist(question: str) -> str:
    response = client.chat.completions.create(
        model="corpassist-lora-v1-merged",  # 对应 --model 参数
        messages=[{"role": "user", "content": question}],
        temperature=0.3,
        max_tokens=512,
    )
    return response.choices[0].message.content

print(ask_corpassist("OA 密码怎么重置？"))

Spring Boot 调用 vLLM 服务

推理桥接 指的是 Spring 应用通过 HTTP 调用私有化部署的 vLLM 服务。

yaml 
# application.yaml — Spring 配置 vLLM 接口
corpassist:
  vllm:
    base-url: http://localhost:8000/v1
    model: corpassist-lora-v1-merged
    temperature: 0.3
    max-tokens: 512
java 
// VllmClient.java — Spring 调用 vLLM API
@Service
public class VllmClient {
    private final RestTemplate restTemplate;
    @Value("${corpassist.vllm.base-url}")
    private String baseUrl;

    public String chat(String question) {
        // 构造 OpenAI 兼容请求
        Map request = Map.of(
            "model", "corpassist-lora-v1-merged",
            "messages", List.of(Map.of("role", "user", "content", question)),
            "temperature", 0.3,
            "max_tokens", 512
        );
        var response = restTemplate.postForEntity(
            baseUrl + "/chat/completions", request, Map.class
        );
        return extractContent(response.getBody());
    }
}

私有化部署 vs DashScope 托管

自行部署 vs 托管 的选择取决于成本、延迟和合规要求。

维度私有化部署 (vLLM)DashScope 托管
GPU 成本~0.5-1 美元/小时 (A100)按 token 付费，~0.004 元/千 token
延迟首 token ~50ms首 token ~100-200ms（网络开销）
数据主权数据不出内网数据经阿里云
运维需自行管理 GPU 集群零运维
弹性固定资源自动弹性扩缩
常见误区
  • 合并后不量化直接部署：14GB bf16 模型 vs 4GB Q4_K_M，显存差距 3.5 倍。
  • vLLM 不支持 PEFT：vLLM 不能动态加载 LoRA adapter（需先合并）。
  • 所有层都量化：embedding 层和 lm_head 量化后精度损失大，AWQ 默认保留 bf16。
  • 忽略 vLLM 配置：gpu_memory_utilization 太低浪费显存，太高容易 OOM。

本章结论

  • 合并 Adapter：merge_and_unload() 生成不依赖 PEFT 的完整模型。
  • GGUF / AWQ：两种主流量化方法，GGUF 适合边缘，AWQ 适合 GPU 推理。
  • vLLM 部署：PagedAttention + 连续批处理，吞吐量 10x+。
  • Spring 调用：OpenAI 兼容 API 格式，RestTemplate 即可对接。
