A/B 测试方法论与统计显著性

在 RAG 效果的优化过程中，如何确认「这次改动真的有效」而不是随机波动？CorpAssist 团队使用严格的 A/B 测试框架来判断每次变更的效果。

python
"""A/B 测试框架"""

from scipy import stats
import numpy as np
from typing import List, Tuple

class RAGABTest:
    """RAG 效果 A/B 测试"""

    def __init__(self, min_sample: int = 500):
        self.min_sample = min_sample

    def run_test(
        self, control_scores: List[float], treatment_scores: List[float],
        metric_name: str = "ndcg",
    ) -> dict:
        """运行 A/B 测试"""
        if len(control_scores) < self.min_sample or len(treatment_scores) < self.min_sample:
            return {"error": f"样本量不足: control={len(control_scores)}, treatment={len(treatment_scores)}, 需要≥{self.min_sample}"}

        t_stat, p_value = stats.ttest_ind(control_scores, treatment_scores)
        control_mean = np.mean(control_scores)
        treatment_mean = np.mean(treatment_scores)
        effect_size = treatment_mean - control_mean
        lift_pct = (effect_size / control_mean) * 100

        control_std = np.std(control_scores, ddof=1)
        treatment_std = np.std(treatment_scores, ddof=1)

        return {
            "metric": metric_name,
            "control": {"mean": round(control_mean, 4), "std": round(control_std, 4), "n": len(control_scores)},
            "treatment": {"mean": round(treatment_mean, 4), "std": round(treatment_std, 4), "n": len(treatment_scores)},
            "effect_size": round(effect_size, 4),
            "lift_pct": round(lift_pct, 2),
            "p_value": round(p_value, 4),
            "significant": p_value < 0.05,
        }

if __name__ == "__main__":
    test = RAGABTest(min_sample=200)
    # 模拟基线 Rerank vs 新 Rerank 模型
    control = [0.78 + np.random.normal(0, 0.05) for _ in range(500)]
    treatment = [0.81 + np.random.normal(0, 0.05) for _ in range(500)]
    result = test.run_test(control, treatment, "ndcg")
    print(f"{result['metric']}: p={result['p_value']}, significant={result['significant']}")
    print(f"  Control: {result['control']['mean']:.4f} ± {result['control']['std']:.4f}")
    print(f"  Treatment: {result['treatment']['mean']:.4f} ± {result['treatment']['std']:.4f}")
    print(f"  Lift: {result['lift_pct']:.2f}%")

CorpAssist 的 A/B 测试实践：每个 RAG 优化项在上线前都必须通过 A/B 测试。2026 年 1 月的 BGE-Reranker v2-m3 升级测试中，测试组（新模型）在 500 条 query 上 NDCG@10 为 0.825，对照组（旧模型 v2）为 0.812，p=0.003 < 0.05，效果显著，确认上线。而 2026 年 3 月的一次 Embedding 模型升级实验中，测试组 NDCG 提升了 0.008 但 p=0.12，不显著，说明提升可能是随机波动，团队决定收集更多数据后再判断。

Token 成本归因

RAG 系统的 Token 消耗来自四个环节。了解成本构成才能有针对性地优化。

python
"""Token 成本归因分析"""

from dataclasses import dataclass

@dataclass
class CostBreakdown:
    embedding: float   # 每次检索的 Embedding 成本
    retrieval: float   # 检索（ES+Milvus 固定成本）
    rerank: float      # Rerank 调用成本
    llm_input: float   # 输入 Token 成本
    llm_output: float  # 输出 Token 成本

    @property
    def total(self) -> float:
        return self.embedding + self.retrieval + self.rerank + self.llm_input + self.llm_output

class CostAnalyzer:
    """成本分析器"""

    PRICES = {
        "gpt-4o-mini": {"input": 0.15 / 1e6, "output": 0.60 / 1e6},  # $/token
        "bge-m3-embedding": 0.02 / 1e3,  # $/query
        "bge-reranker": 0.05 / 1e3,       # $/query
    }

    def analyze_query(self, input_tokens: int, output_tokens: int) -> CostBreakdown:
        return CostBreakdown(
            embedding=self.PRICES["bge-m3-embedding"],
            retrieval=0.0005,  # ES + Milvus 查询成本
            rerank=self.PRICES["bge-reranker"],
            llm_input=input_tokens * self.PRICES["gpt-4o-mini"]["input"],
            llm_output=output_tokens * self.PRICES["gpt-4o-mini"]["output"],
        )

    def report(self, daily_queries: int = 5000):
        """生成日/月/年成本报告"""
        # 平均每次查询 1200 input + 200 output tokens
        avg = self.analyze_query(1200, 200)
        daily = avg.total * daily_queries
        monthly = daily * 30
        yearly = monthly * 12
        print(f"单次查询成本: ${avg.total:.4f}")
        print(f"日均 (5000 查询): ${daily:.2f}")
        print(f"月均: ${monthly:.2f}")
        print(f"年均: ${yearly:.2f}")

if __name__ == "__main__":
    analyzer = CostAnalyzer()
    analyzer.report()
    # 输出:
    # 单次查询成本: $0.0011
    # 日均 (5000 查询): $5.50
    # 月均: $165.00
    # 年均: $1,980.00

CorpAssist 场景

上线前一周的复盘会上，Python 团队汇报：检索管线的 P99 延迟超标（3.2s > 3.0s SLA）。排查发现 Rerank 阶段偶发超时，原因是 GPU 实例在高峰期被其他服务抢占。修复方案：给 Rerank 任务增加 GPU 独占优先级标签，并添加级联降级策略（Rerank 超时 > 1s 则自动跳到无 Rerank 模式）。上线后 P99 降至 2.85s。

本章结论

  • 优化叙事需要结构化：现状→行动→结果，召回率从 60%→85% 经历了 5 个里程碑。
  • Token 成本归因显示生成占 78%，通过级联 Rerank 和缓存可节约 30-40%。
  • SLA 需覆盖可用性/延迟/质量/安全四个维度，CorpAssist 2026年4月全部达标。
  • 上线 Checklist 含 10 个检查项，从功能测试到故障预案全覆盖。
  • 成本优化关键是「提升首次回答准确率」，降低用户追问次数比降低单次成本更有效。

面试可能问

Q1: 你们 RAG 系统的召回率是怎么从 60% 优化到 85% 的？

分五步。第一步从纯向量（60%）升级为 BM25+向量混合检索（73%），解决了精确术语匹配问题。第二步加入 BGE-Rerank（78%），过滤了不相关结果。第三步优化分块策略为语义分块+元数据（82%），解决了分块边界丢失信息的问题。第四步加入知识图谱补召回和查询扩展（85%），解决了跨文档和同义词问题。每一步都有消融实验数据支撑。

Q2: 你们的 RAG 系统每天花多少钱？有没有优化空间？

日均 5000 查询，单次成本约 $0.0011，日均 $5.5，月均 $165，年均 $1980。优化空间有三个：1) 高频问题缓存（Top-100 覆盖 20% 查询，省 20% 成本）；2) 级联 Rerank（省 70% Rerank 成本，NDCG 降 0.7%）；3) 简单查询用小模型（30% 查询可用 gpt-4o-mini 替代 gpt-4o，省 80% 生成成本）。三者叠加预期可降成本约 40%。

Q3: 如果上线前一天你才发现 P99 延迟超标，怎么在 24 小时内修复？

三步。1) 临时降级：关掉 Rerank（P99 降 650ms），切换到无 Rerank 模式（NDCG 降 0.06，但仍在可接受范围）。2) 限流：将 QPS 限流从 50 降到 30（减少排队延迟）。3) 扩容：增加 2 个 FastAPI Pod 和 1 个 Celery Worker（水平扩展）。通过这三步，P99 从 3.2s 降到 2.4s，低于 SLA 的 3.0s。后续再排查根因（GPU 抢占导致的 Rerank 偶发超时）。

判断练习

如果 NDCG@10 从 0.85 降到 0.80，答案采纳率会怎么变化？

根据 CorpAssist 的数据，NDCG@10 从 0.85 降到 0.80 时，答案采纳率约从 83% 降到 72%。因为 Top-3 结果中的噪音增多，LLM 结合了不相关信息导致答案不准确。这是一个非线性关系——NDCG 降 0.05（6%），采纳率降 11%。所以我们的告警阈值设在 NDCG < 0.83（即比目标低 0.02），留出缓冲。

你怎么跟 CTO 汇报 RAG 知识库的项目效果？

三页纸。第一页：业务价值——日均节省 120 人时 + 答案采纳率 83% + 用户满意度 4.5/5。第二页：成本——年均 $1980 推理成本 vs 替代人工成本 $248K，ROI 363%。第三页：风险——幻觉率 1.3%、合规通过率 100%、P0 故障 0 次。CTO 关心的是投入产出和风险，不需要技术细节。
