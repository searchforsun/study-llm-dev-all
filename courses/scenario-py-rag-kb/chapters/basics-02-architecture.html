CorpAssist 场景

用户在 CorpAssist 门户提问「今年的春节放假安排是什么」。请求经 Nginx 转发到 FastAPI 接入层，JWT 鉴权通过后进入编排层。编排层先根据问题判断需要检索「节假日制度」，混合检索从 ES 和 Milvus 各取 10 条，加权融合后取 top 5 送入 BGE Rerank 重排序，最终取 top 3 填充到 Prompt 模板，LLM 生成带引用的回答，通过 SSE 流式推送到门户前端的对话框。

RAG vs 微调的决策树

在项目初期最常被问到的问题是：「为什么用 RAG 而不是微调？」答案不是一个非此即彼的二元选择，而是一个基于场景特征的决策树。

    Q2[输出格式固定？] -->|是, 如摘要/分类| FT[微调]

    RAG --> Q3[检索准确率要求？]

    Q4 -->|>5000条| FT_Go[可以微调]

数据支撑：在 CorpAssist 的实验中，对于「制度问答」场景，RAG 方案准确率 82.3%，微调方案准确率 76.1%（微调后 3 个月知识更新后掉到 68.4%）。但如果只看「提取工单编号」这种固定格式任务，微调准确率 94.7% vs RAG 的 88.2%，微调更优。

python
"""RAG vs 微调决策树实现"""

def decide_approach(
    update_freq_days: int,
    output_fixed: bool,
    accuracy_required: float,
    data_count: int,
) -> str:
    """Return recommended approach based on scenario characteristics."""
    if update_freq_days <= 30:
        return "RAG (知识频繁更新，不适合微调)"
    if output_fixed and data_count >= 5000:
        return "微调 (固定格式 + 充足数据)"
    if output_fixed and data_count < 5000:
        return "RAG + 少量样本提示 (数据不足5000，微调风险高)"
    # 开放式问答，走 RAG
    if accuracy_required > 0.95:
        return "RAG + Rerank + 人工复核兜底"
    return "标准 RAG"

if __name__ == "__main__":
    test_cases = [
        (7, False, 0.90, 0),     # 制度问答: RAG
        (180, True, 0.95, 8000), # 工单分类: 微调
        (90, True, 0.90, 200),   # 小样本分类: RAG
        (1, False, 0.98, 0),     # 高频更新的高精度: RAG+Rerank
    ]
    for case in test_cases:
        print(f"update={case[0]}d fixed={case[1]} acc={case[2]} data={case[3]} → {decide_approach(*case)}")

关键发现：微调模型的「知识保质期」是一个被低估的风险。在 CorpAssist 测试中，微调模型上线 3 个月后，由于制度更新（如 2026 年社保基数调整），已有 12.4% 的回答使用了过时知识。RAG 只需要更新索引文档即可，无需重新训练。

架构扩展性设计

RAG 知识库的架构需要预留扩展点以应对未来的业务增长和技术演进。以下是 CorpAssist 架构设计中的关键扩展性考量。

关键扩展性决策：

  • 文档量 2 万 → 100 万：Milvus 从单 collection 转为按部门分片（每个部门一个 collection），ES 引入冷热分层架构（近 3 个月文档在热节点，3 个月前的在冷节点，检索时指定时间范围过滤）。
  • QPS 50 → 500：当前单实例 FastAPI 扩展为 K8s HPA 自动扩缩容（CPU > 70% 触发扩容），Celery Worker 从 10 个扩展到 50 个。
  • LLM 切换：通过 LangChain 的 model provider 抽象层，可以在不修改编排逻辑的前提下切换 gpt-4o / Claude / 国产模型。

在面试中，主动聊扩展性设计会加分——它表明你不只考虑当前的「玩具系统」，而是在设计一个能成长的「产品级」架构。

部署架构与容灾设计

RAG 知识库的部署架构需要考虑高可用和容灾。CorpAssist 采用多可用区 (Multi-AZ) 部署。

    subgraph AZ1[可用区 A - 主]
        A1[FastAPI Pod × 3]
        A2[Celery Worker × 5]
        A3[Milvus 主副本]
        A4[ES 热节点]
    end
    subgraph AZ2[可用区 B - 备]
        B1[FastAPI Pod × 2]
        B2[Celery Worker × 3]

        B4[ES 温节点]
    end
    subgraph Global[全局]
        C[LB 负载均衡]
        D[PostgreSQL 主从]

        F[OSS 对象存储]
    end
    C --> A1
    C --> B1
    A3 <--> B3
    A4 <--> B4

容灾策略：

  • Milvus：主从副本部署在不同可用区，主副本故障时 30 秒内自动切换到从副本。
  • ES：跨可用区分片，至少 2 个副本，保证一个可用区完全故障时数据不丢失。
  • PostgreSQL：流复制主从，主库故障时 60 秒内自动提升从库。
  • Redis：Sentinel 监控，主节点故障时 10 秒内自动切换。
  • FastAPI：K8s HPA 自动扩缩容，Pod 故障时自动重启替换。

这套容灾设计保障了 P99 可用率 99.95%（年停机 ≤ 4.38 小时），已通过压力测试——单可用区完全故障时，服务降级到 60% 容量但在 2 分钟内自动恢复。面试中如果聊到部署架构，提到 Multi-AZ 和容灾设计会显著加分。

面试答题框架：架构题的 5 步回答法

面试中被问到「请设计一个 RAG 知识库系统」，不要直接画架构图——用五步法组织回答，让面试官觉得你思路清晰。

五步法框架：

  1. 定边界：「我先明确一下这个知识库覆盖哪些文档、多少用户、什么查询类型」——展现你的需求分析能力。
  2. 分层次：「我把它分为接入层、编排层和数据层」——展现你的架构抽象能力。
  3. 描述数据流：「用户请求从 API 进入，经过鉴权、检索、融合、生成、返回」——展现你的端到端视角。
  4. 讲选型理由：「为什么用 Milvus 而不是 ES 做向量存？因为...用数据说话」——展现你的技术判断力。
  5. 谈权衡：「这个设计的最大 trade-off 是索引延迟 vs 检索精度，我们的解法是...」——展现你的工程务实态度。

❌ 不好的回答

「用 LangChain 搭个 RAG，用向量数据库存 embedding，用 LLM 生成答案就好了。」没有边界、没有数据、没有选型理由、没有权衡。

✅ 好的回答

「这是一个面向 5000 名员工的制度文档问答系统，日活 2000。我设计了三层架构。接入层 FastAPI 做鉴权和限流。编排层 LangChain LCEL 串联检索和生成，其中检索用了 ES BM25 + Milvus 双路融合，权重 α=0.3。数据层 PostgreSQL 存元数据和审计日志。最大权衡是 Rerank 加了 650ms P99 延迟但让准确率从 78% 提到 89%。」三句话里包含 7 个具体数据和决策理由。

本章结论

  • 三层架构（接入/编排/数据）是 RAG 知识库的标准范式，总 P99 延迟约 2.8s。
  • RAG vs 微调的决策关键在于更新频率和输出格式，制度问答推荐 RAG。
  • 数据流涉及 7 个组件串行协作，检索阶段是主要的延迟瓶颈（Rerank 占 650ms P99）。
  • 面试架构题的五步法（边界→分层→数据流→选型→权衡）能系统化展示架构能力。
  • 微调模型存在「知识保质期」问题，3 个月后 12.4% 的回答使用了过时知识。

面试可能问

Q1: 如果知识库有 100 万份文档，你的架构需要做什么调整？

三点调整。1) 数据层：Milvus 需要按业务域分 collection，ES 需要按月份或部门建 index，避免单索引过大导致检索退化。2) 编排层：引入级联检索策略——先用粗粒度 Tag 过滤缩小范围，再在缩小后的集合上做语义检索。3) 接入层：引入异步模式，对复杂查询先返回「任务 ID」，用户轮询或 WebSocket 获取结果。

Q2: 为什么用了 RAG 还要 Rerank？不止一次检索够吗？

不够。首次检索（BM25/向量）的 top-K 结果里通常有 30-40% 是不相关的。Rerank 用交叉编码器（Cross-Encoder）对问题-文档对逐条打精确分，能把 top-5 准确率从 65% 提到 89%。P99 延迟增加 650ms，但效果提升显著。对 P99 要求低于 2s 的场景可以省略或使用小模型（如 BGE-Reranker-v2-mini）。

Q3: 如果你们的知识库既要服务中文又要服务英文，架构上有什么不同？

核心差异在 Embedding 模型。需要选择多语言模型如 BGE-M3 或 intfloat/multilingual-e5-large，确保中英文语义空间对齐。中文分词的 ES 分析器需要用 ik_smart 或 jieba，英文用 standard。Rerank 模型同样需要多语言版本。在我们的测试中，BGE-M3 的中英文混合检索 NDCG@10 是 0.812，而纯中文模型在英文查询上只有 0.423。

判断练习

技术团队提议把所有文档直接放到 LLM 的 128K 上下文窗口中，不做 RAG。你同意吗？

在大规模场景下不可行。128K 上下文看起来能塞下很多文档，但实际「大海捞针」测试中，当上下文超过 64K tokens，LLM 对中间部分的召回率低于 60%。且每次请求都把所有文档拼进去，Token 成本是 RAG 的 20-50 倍（例如 100 份文档每次消耗 128K tokens，RAG 仅消耗 3K tokens 的上下文）。RAG 的核心价值是「精确召回相关文档，减少无效输入」。

如果你的 P99 延迟要求小于 1.5s，你会怎么调整架构？

首先去掉 Rerank 阶段（省 650ms），只做 BM25 + 向量融合检索。其次使用 2-bit 量化的小 Embedding 模型（如 BGE-small-v1.5）减少向量检索时间。如果 LLM 首包时间超过 500ms，考虑用 vLLM 或 TensorRT-LLM 加速推理。最后，对常见问题（如「年假几天」）做缓存，命中率约 25%，可绕过完整链路。
