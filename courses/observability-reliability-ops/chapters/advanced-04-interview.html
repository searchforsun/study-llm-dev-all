典型面试题 1：SLO 设计

题面：你为一个 LLM 客服系统设计 SLO，选哪些指标？目标设多少？怎么监控？

text 
CORAL 回答示例：

C (Context): 以上海证券的 CorpAssist 客服为例，用户通过 Web/App 询问业务问题，
系统通过 RAG + LLM 生成回答。SLA 要求 99.5%。

O (Objective): 我们选了三个黄金信号：
1. 延迟 SLO: P95 < 5s（含 TTFT 单独监控 < 2s）
2. 错误率 SLO: < 1%（含 HTTP 5xx + LLM 异常 + 业务错误）
3. 质量分 SLO: > 85%（LLM-as-judge 自动评估）

R (Rationale): 传统 SRE 四个信号不够，因为 LLM 可能延迟和错误率正常
但回复质量差。TTFT 是用户感知延迟的关键。

A (Action): Micrometer 埋点 → Prometheus 存储 → Grafana SLO 看板。
Burn rate 告警：2x Warning、5x Critical、10x Page。

L (Learning): 最初没按 tenant 分 SLI，acme 租户慢但被其它租户拉平。
后来每个租户独立 SLI + 独立错误预算。

Java：SLO 面试代码题

题面：实现一个 SLO 计算器，输入每日成功/总请求数，输出当月 SLO 达标情况。

java 
public class SLOCalculator {
    private final double sloTarget;  // 0.99
    private final int windowDays;    // 30
    private final List dailySLIs = new ArrayList<>();

    public SLOCalculator(double sloTarget, int windowDays) {
        this.sloTarget = sloTarget;
        this.windowDays = windowDays;
    }

    // 添加一天的 SLI 数据
    public void addDailyData(long success, long total) {
        double sli = total > 0 ? (double) success / total : 1.0;
        dailySLIs.add(sli);
        // 只保留窗口期数据
        while (dailySLIs.size() > windowDays) {
            dailySLIs.remove(0);
        }
    }

    // 当前 SLO 达标情况
    public SLOStatus checkSLO() {
        if (dailySLIs.isEmpty()) {
            return new SLOStatus(true, 100.0, "No data");
        }
        double avgSLI = dailySLIs.stream()
            .mapToDouble(Double::doubleValue).average().orElse(1.0);
        double errorBudgetUsed =
            Math.max(0, (sloTarget - avgSLI) / (1 - sloTarget) * 100);
        boolean met = avgSLI >= sloTarget;
        return new SLOStatus(met, 100 - errorBudgetUsed,
            met ? "SLO met" : "SLO breached");
    }

    public record SLOStatus(boolean met, double budgetRemaining, String message) {}
}

// 使用示例
SLOCalculator calc = new SLOCalculator(0.99, 30);
calc.addDailyData(9950, 10000);  // 99.5%
calc.addDailyData(9900, 10000);  // 99.0%
SLOStatus status = calc.checkSLO();
// budgetRemaining: 50% (用了 50% 的错误预算)

典型面试题 2：成本治理

题面：LLM API 成本快速增长，你怎么治理？

text 
CORAL 回答示例：

C: CorpAssist 月 LLM 成本从 $2000 涨到 $8000，财务要求控制。

O: 目标是成本可归因、可预警、可控制。具体：1) 分租户计量，
2) 设置 daily cap，3) 成本异常自动降级。

R: 没有计量就没有治理。要先知道钱花在哪（哪个租户、哪个模型、
哪类问题），才能针对性地优化。

A:
1. 计量：Micrometer Counter 按 tenant+model 统计 token，LangSmith API 做 chargeback
2. 预算：每个租户 daily cap，超 80% 通知、100% 限流
3. 优化：引入语义缓存（命中率 40% 降成本 35%）、route cheap（简单问题用小模型）
4. 告警：三级告警 Info/Warning/Critical

L: 一开始没给每个租户独立 cap，某个租户的脚本循环调用导致
整体成本翻倍。隔离后该租户自行优化用量。

面试官问「你犯过最大的错是什么」怎么答？
选和岗位相关的、有复盘有改进的案例。例：我曾没给每个租户设独立 Token cap，导致一个租户的 bug 脚本把整个月的预算烧光了。后来我加了租户级 daily cap + 异常检测告警。
被问到「LLM 观测和传统观测有什么不同」？
三个核心差异：1) 质量分是新的黄金信号（传统没有），2) Token 成本是新的观测维度，3) TTFT 比总延迟更能衡量用户感知。另外 prompt/模型/索引需要各自版本管理。
为什么 CORAL 比 STAR 更适合 SLO 题？
STAR（Situation/Task/Action/Result）适合具体事例，CORAL 适合「设计类」问题——先建立思维坐标系（C/O），再展示理解深度（R），最后落地和迭代（A/L）。面试 SLI/SLO 设计题时 CORAL 更自然。

本章结论

  • CORAL 框架：Context → Objective → Rationale → Action → Learning，适合 SLO/成本设计类面试题。
  • STAR 框架：适合事故排障类题目，展示从发现问题到解决问题的完整过程。
  • 有故事有数字：面试中结合 CorpAssist 实际经验，说出具体数字和踩过的坑。
  • 四个核心话题：SLO/SLA 设计、成本治理、故障排障、灰度发布。
  • 区别传统观测：LLM 观测的独特之处——质量分、Token 成本、TTFT。
