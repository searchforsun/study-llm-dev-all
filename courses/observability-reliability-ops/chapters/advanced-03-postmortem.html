为什么要学本章

事故不是 bug 修复完就结束的——不复盘的事故还会以不同形式重演。Postmortem（复盘）通过结构化回顾找到系统原因而非个人原因，把每个事故变成系统改进的驱动力。本章用 CorpAssist 事故案例带你走一遍完整复盘流程。

学完你能
  • 编写结构化的 incident timeline
  • 用 5 whys 方法找到根因而非表面原因
  • 区分 root cause 和 trigger，写出有效的 action item
  • 使用 blameless 的复盘框架
本章先记住 2 件事
  • 不追责只追因：复盘的目标是找到「系统为何没检测/没拦住/没自动恢复」，不是「谁犯了错」。
  • Action item 必须有 owner：没有 owner 的改进项会在迭代中被遗忘，同样的事故会再次发生。

事故复盘模板

CorpAssist · 复盘会议（45 秒） 昨天 CorpAssist 发生了一次 P0 事故：LLM API key 过期导致全量请求 503，持续 22 分钟。根因监控没告警（API key 过期不是 HTTP 错误，业务层静默失败）。复盘后提出的 action item：增加 API key 有效期检查 + 自动切换备用 key + 添加「key 即将过期」告警。

Postmortem 模板

markdown 
# 故障复盘：CorpAssist LLM API Key 过期

## 基本信息
- **事故 ID**: INC-2026-05-29-001
- **严重等级**: P0
- **发现时间**: 2026-05-29 10:05 UTC
- **恢复时间**: 2026-05-29 10:27 UTC
- **时长**: 22 分钟
- **影响用户**: 所有在线用户 (约 1500 DAU)
- **负责人**: @sre-alice

## 时间线 (Timeline)
| 时间 (UTC) | 事件 |
|------------|------|
| 09:55 | API key 到达过期时间 |
| 10:00 | 第一个用户反馈「无响应」 |
| 10:05 | 产品经理在 #ops 报告问题 |
| 10:08 | Oncall 确认 LLM 调用全部 401 |
| 10:12 | 发现 API key 过期 |
| 10:15 | 切换备用 key |
| 10:20 | 确认恢复 |
| 10:27 | 确认所有请求正常 |

## 影响评估
- 约 300 次 LLM 调用失败
- 用户侧看到「服务繁忙」提示
- 无数据丢失

## 根因分析 (5 Whys)
1. API key 为什么过期了？ → 证书有效期 90 天到期
2. 为什么没有提前更换？ → 没有 key 过期检查
3. 为什么监控没告警？ → 监控只检查 HTTP 5xx，401 被 SDK 静默处理为超时
4. 为什么没有备用 key？ → 配置只有单 key 无 failover
5. 为什么 key 到期前没有预警？ → 无 key 生命周期管理

**根因**: API key 生命周期管理缺失 + 监控盲区

## Action Items
| # | 行动项 | Owner | 截止日期 | 状态 |
|---|--------|-------|----------|------|
| 1 | 增加 API key 有效期检查，提前 7 天告警 | @infra-bob | 2026-06-05 | [ ] |
| 2 | 配置备用 API key 自动切换 | @infra-bob | 2026-06-05 | [ ] |
| 3 | 增加「API key 过期」告警规则 | @sre-alice | 2026-06-03 | [ ] |
| 4 | 评审所有供应商凭据的生命周期管理 | @sec-carl | 2026-06-10 | [ ] |

## 经验教训
- 监控盲区：401 错误被 SDK 吞掉，业务层看到的只是超时
- 单点故障：任何没有备用方案的依赖都是风险
- 凭据老化：90 天以上的凭据应该定期轮换演练

5 Whys 分析法

5 Whys 不是严格限定 5 次「为什么」，而是层层深入直到找到系统原因而非人因。CorpAssist 场景的 5 Whys 示例：

5 Whys 分析树
flowchart TB
  A[LLM 调用全部失败] --> B[Why: API key 401]
  B --> C[Why: key 已过期]
  C --> D[Why: 无到期前检查]
  D --> E[Why: 凭据生命周期管理缺失]
  E --> F[Root Cause: 供应商凭据无自动化管理]
  C --> G[Why: 无告警通知到期]
  G --> H[Root Cause: 监控没有覆盖 key 有效期]
  B --> I[Why: 备用 key 无自动切换]
  I --> J[Root Cause: 无多 key failover 机制]

5 Whys 可能产生多个分支，每个分支都对应一条 action item。

Java：事故时间线记录

java 
import java.time.Instant;
import java.util.List;
import java.util.ArrayList;

// 事故时间线记录器
@Component
public class IncidentTimelineLogger {
    private final List events = new ArrayList<>();

    public record TimelineEvent(
        Instant timestamp,
        String event,
        String source  // "system" | "manual"
    ) {}

    public void record(String event, String source) {
        TimelineEvent e = new TimelineEvent(Instant.now(), event, source);
        events.add(e);
        log.info("INCIDENT TIMELINE: {} - {} [{}]", e.timestamp(), e.event(), e.source());
    }

    public String export() {
        StringBuilder sb = new StringBuilder("## 时间线\n");
        sb.append("| 时间 | 事件 | 来源 |\n");
        sb.append("|------|------|------|\n");
        for (var e : events) {
            sb.append(String.format("| %s | %s | %s |\n",
                e.timestamp(), e.event(), e.source()));
        }
        return sb.toString();
    }

    public void clear() {
        events.clear();
    }
}

// 使用示例
@Service
public class IncidentService {
    private final IncidentTimelineLogger timeline;

    public void handleIncident(String incidentId) {
        timeline.record("开始处理事故 " + incidentId, "manual");
        timeline.record("检查 LLM 状态", "system");
        timeline.record("发现 API key 过期", "manual");
        timeline.record("切换备用 key", "manual");
        timeline.record("恢复确认", "system");
    }
}

Python：复盘报告生成

python 
from dataclasses import dataclass, field
from typing import List, Optional
from datetime import datetime, timedelta


@dataclass
class ActionItem:
    """改进项"""
    number: int
    description: str
    owner: str
    deadline: datetime
    status: str = "open"


@dataclass
class PostmortemReport:
    """结构化的故障复盘报告。"""
    incident_id: str
    severity: str
    detected_at: datetime
    recovered_at: datetime
    impact: str
    affected_users: int
    timeline: List[dict]
    root_cause: str
    action_items: List[ActionItem] = field(default_factory=list)

    @property
    def duration_minutes(self) -> int:
        return int((self.recovered_at - self.detected_at).total_seconds() / 60)

    def add_action(self, desc: str, owner: str, days: int = 7):
        """添加改进项"""
        n = len(self.action_items) + 1
        deadline = datetime.utcnow() + timedelta(days=days)
        self.action_items.append(
            ActionItem(n, desc, owner, deadline))

    def render_to_markdown(self) -> str:
        """输出 Markdown 复盘报告"""
        lines = [
            f"# 故障复盘：{self.incident_id}",
            "",
            f"**严重等级**: {self.severity}",
            f"**时长**: {self.duration_minutes} 分钟",
            f"**影响用户**: {self.affected_users}",
            "",
            "## 时间线",
            "| 时间 | 事件 |",
            "|------|------|",
        ]
        for evt in self.timeline:
            lines.append(f"| {evt['time']} | {evt['event']} |")

        lines.extend([
            "",
            "## 根因",
            self.root_cause,
            "",
            "## Action Items",
            "| # | 行动项 | Owner | 截止日期 |",
            "|---|--------|-------|----------|",
        ])
        for item in self.action_items:
            lines.append(
                f"| {item.number} | {item.description} | "
                f"{item.owner} | {item.deadline.date()} |")
        return "\n".join(lines)
不推荐
  • 复盘只到「当事人下次注意」
  • 无时间线，凭记忆回顾
  • Action item 无 owner 无截止日
  • 把「换人」当作改进措施
推荐
  • Blameless 文化，找系统原因
  • 精确记录每步时间线
  • 每个 action item 有 owner 和 due date
  • 改进项可跟踪可验证
复盘频率应该多高？
每次 P0/P1 事故必须做正式复盘；P2 做轻量复盘（5 行总结 + action items）；P3 不强制。建议每月开一次「事故趋势分析」会，看哪些类型的事故在反复出现。
Action item 完不成怎么办？
下次复盘中 review 未完成的 action item。如果在截止日未完成，需要说明原因和新的预期完成时间。连续两次延期的 action item 需要重新评估优先级——也许它其实不重要。
Blameless 文化怎么落地？
几个原则：1) 复盘文档不写「谁做了什么」只写「系统状态是什么」，2) 分享事故经验奖励而非惩罚，3) 管理层不以事故数量考核个人，而是看「团队从事故中学到了什么」。详见 retro template 中的 blameless 指南。
常见误区
  • 复盘 = 找责任人 — 这样下次没人愿意报问题，隐瞒事故。
  • Action item 太多 — 每个事故提 20 个改进项，结果一个也完不成。建议不超过 5 个。
  • 只复盘 P0 — P1/P2 累积多了也会变成 P0。
  • 不复盘「险些发生」 — Near miss 是免费的学习机会，不需要等事故发生。

本章结论

  • Postmortem 结构化模板：基本信息 → 时间线 → 影响评估 → 5 Whys → Action Items → 经验教训。
  • 5 Whys 找根因：层层深入直到找到系统原因，而非停留在「人忘了」。
  • Blameless 文化：不追责只追因。找到了系统漏洞比找到谁犯错更有价值。
  • Action item 有 owner：每个改进项必须有负责人和截止日，否则等于没有。
  • 复盘频率：P0/P1 正式复盘，P2 轻量复盘，每月事故趋势分析。
