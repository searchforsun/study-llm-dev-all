排障流程：三步法

排障三步法
flowchart TB
  A[接收告警] --> B{Step 1: 确认}
  B --> B1[看 SLO 看板: 影响范围]
  B --> B2[看 Grafana: 全局 vs 单租户]
  B1 --> C{Step 2: 定位}
  C --> C1[Tempo Trace: 慢在哪个 span]
  C --> C2[Loki 日志: 错误详情]
  C1 --> D{Step 3: 止血}
  D --> D1[LLM 超时→切备用模型]
  D --> D2[RAG 慢→降级纯 LLM]
  D --> D3[服务挂→重启/扩容]
  D1 --> E[通知团队 + 写复盘]

三步排障法确保 oncall 不是随机试错，而是按系统流程操作。

Java：运维健康检查端点

java 
import org.springframework.boot.actuate.health.Health;
import org.springframework.boot.actuate.health.HealthIndicator;

// LLM 健康检查
@Component
public class LLMHealthIndicator implements HealthIndicator {
    private final List models = List.of(
        "claude-sonnet-4", "deepseek-v3", "gpt-4o");

    @Override
    public Health health() {
        var builder = Health.up();
        for (String model : models) {
            boolean reachable = checkModel(model);
            if (!reachable) {
                builder.withDetail(model, "unreachable");
            } else {
                builder.withDetail(model, "ok");
            }
        }
        return builder.build();
    }

    private boolean checkModel(String model) {
        // 发一个轻量请求到 LLM API
        try {
            return chatClient.prompt()
                .model(model)
                .user("ping")
                .call()
                .content() != null;
        } catch (Exception e) {
            return false;
        }
    }
}

// RAG 健康检查
@Component
public class RAGHealthIndicator implements HealthIndicator {
    @Override
    public Health health() {
        boolean dbOk = checkVectorDB();
        boolean indexOk = checkIndexExists();
        if (dbOk && indexOk) {
            return Health.up().build();
        }
        return Health.down()
            .withDetail("vector_db", dbOk ? "ok" : "fail")
            .withDetail("index", indexOk ? "ok" : "missing")
            .build();
    }
}

不推荐
  • 运维知识散落在各人笔记中
  • 只有架构图没有排障流程
  • 告警无 Runbook oncall 自己猜
  • 从不演练，真出事故时手忙脚乱
推荐
  • 运维手册集中维护，版本管理
  • 排障三步法：确认→定位→止血
  • 每个告警绑定 Runbook 链接
  • 每季度至少一次故障演练
运维手册应该谁来维护？
开发团队维护技术内容（架构、指标、Runbook），SRE 团队维护排障流程和演练计划。手册放在 Git 仓库中，每次架构变更同步更新。关键的检查点：每次上线后更新版本记录。
演练多久做一次？
推荐频率：关键场景（LLM 全挂、数据库故障）每季度一次；常规场景（单节点故障）每半年一次。演练后必须有复盘和改进项。CorpAssist 的 drill exercise 脚本支持场景化注入。
Oncall ready 怎么判断？
新人加入 oncall 轮值前的 checklist：1) 读过运维手册，2) 能登录 Grafana/Tempo/Loki/LangSmith，3) 知道每个告警的 Runbook 在哪，4) 完整走过一次排障三步法，5) 参加过至少一次演练。

本章结论

  • 运维手册是团队资产：8 个章节覆盖架构、SLO、告警、排障、版本、灾备、成本、演练。
  • 排障三步法：确认影响范围 → 定位根因 → 执行止血操作。可循环使用。
  • 演练验证手册：每季度至少一次故障演练，验证手册完整性和 oncall 熟练度。
  • Ops package 可交付：7 项产出物确保团队交接时没有知识盲区。
  • Oncall ready checklist：新人按 5 步清单完成 oncall 上岗准备。
