为什么要学本章

LLM 调用的主要成本是 token 消耗。大量用户提问是相似的（「密码怎么重置」「发票怎么开」）。语义缓存通过缓存语义相似的 LLM 回复，可以降低 30-60% 的 token 消耗和延迟。本章讲如何设计缓存策略、衡量 cache hit rate 和计算 savings metric。

  • 配置 cache ttl 和缓存失效策略
  • 在 Grafana 上监控 cache hit rate 并量化成本节省
  • 理解 route cheap 策略——简单问题用小模型、复杂问题用大模型
本章先记住 2 件事
  • 语义缓存 ≠ 精确匹配：两个问题用词不同但意图相同（「如何重置密码」vs「忘记密码怎么办」），精确 cache 命中不了，语义 cache 可以。
  • 缓存不是万能：需要设计合理的 TTL、缓存大小和路由策略，否则可能命中过期知识或消耗过多存储。

语义缓存策略设计

CorpAssist · 高频问题（30 秒） CorpAssist 上线后发现 top-10 问题占了 40% 的请求量，但每次还是完整调 LLM。用户问「工资单怎么查」和「怎么查我的工资单」本质是同一问题。引入语义缓存后，相似度 > 0.95 直接返回缓存结果，P95 延迟从 4s 降到 200ms，Token 成本降低 35%。

缓存架构

语义缓存请求流程
flowchart TB
  Q[用户提问] --> E[生成 embedding]
  E --> S[向量相似度搜索]
  S --> H{相似度 > 阈值?}
  H -->|是| CACHE[返回缓存回复]
  H -->|否| LLM[调 LLM]
  LLM --> R[生成回复]
  R --> STORE[存储到缓存]
  STORE --> RESP[返回用户]
  CACHE --> RESP

语义缓存流程：每次用户提问先算 embedding，在向量库中找相似度 > 0.95 的缓存结果。命中则直接返回，不命中才调 LLM 并写入缓存。

Python：语义缓存实现

python 
from typing import Optional, List, Tuple
import numpy as np
from openai import OpenAI
import redis
import json

class SemanticCache:
    """语义缓存 —— 基于 embedding 相似度匹配。"""

    def __init__(self, embedding_model: str = "text-embedding-3-small",
                 similarity_threshold: float = 0.95,
                 redis_url: str = "redis://localhost:6379",
                 max_cache_size: int = 10000):
        self.client = OpenAI()
        self.embedding_model = embedding_model
        self.threshold = similarity_threshold
        self.redis = redis.from_url(redis_url)
        self.max_size = max_cache_size

    def _get_embedding(self, text: str) -> List[float]:
        resp = self.client.embeddings.create(
            model=self.embedding_model,
            input=text
        )
        return resp.data[0].embedding

    def _similarity(self, a: List[float], b: List[float]) -> float:
        return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

    def get(self, question: str,
            tenant_id: str) -> Optional[Tuple[str, float]]:
        """查找缓存，返回 (回复, 相似度) 或 None"""
        embedding = self._get_embedding(question)
        key = f"cache:{tenant_id}"

        # 从 Redis 读取所有缓存条目（生产用向量数据库更高效）
        cached = self.redis.hgetall(key)
        best_sim = 0
        best_answer = None

        for q_bytes, answer_bytes in cached.items():
            cached_q = q_bytes.decode()
            cached_embedding = json.loads(cached_q)  # 简化：实际存向量
            sim = self._similarity(embedding, cached_embedding)
            if sim > best_sim:
                best_sim = sim
                best_answer = answer_bytes.decode()

        if best_sim >= self.threshold:
            return best_answer, best_sim
        return None

    def set(self, question: str, answer: str,
            tenant_id: str, ttl: int = 3600):
        """写入缓存"""
        embedding = self._get_embedding(question)
        key = f"cache:{tenant_id}"
        # 生产环境使用向量数据库（Milvus/Pinecone）
        self.redis.hset(key, json.dumps(embedding), answer)
        self.redis.expire(key, ttl)

    def hit_rate(self, tenant_id: str) -> float:
        """返回缓存命中率"""
        hits = int(self.redis.get(f"stats:{tenant_id}:hits") or 0)
        total = int(self.redis.get(f"stats:{tenant_id}:total") or 0)
        return hits / total if total > 0 else 0.0

Java：缓存命中率监控

java 
import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Counter;

@Service
public class CacheMetricsService {
    private final Counter hitCounter;
    private final Counter missCounter;

    public CacheMetricsService(MeterRegistry registry) {
        this.hitCounter = Counter.builder("cache.hits")
            .tag("type", "semantic")
            .register(registry);
        this.missCounter = Counter.builder("cache.misses")
            .tag("type", "semantic")
            .register(registry);
    }

    public void recordHit(String tenantId) {
        hitCounter.increment();
        recordEvent(tenantId, "hit");
    }

    public void recordMiss(String tenantId) {
        missCounter.increment();
        recordEvent(tenantId, "miss");
    }

    // 计算每小时命中率
    @Scheduled(fixedRate = 3600000)
    public void reportHitRate() {
        double rate = hitCounter.count() /
            (hitCounter.count() + missCounter.count()) * 100;
        log.info("Semantic cache hit rate: {}.1f%%", rate);
        if (rate < 20) {
            log.warn("Low cache hit rate: {}.1f%%. Consider tuning threshold", rate);
        }
    }
}

// 成本节省计算
@Component
public class CostSavingCalculator {
    private static final double COST_PER_1K_TOKENS = 0.003;

    public double calculateSavings(long tokensCached) {
        double saved = (tokensCached / 1000.0) * COST_PER_1K_TOKENS;
        log.info("Cache savings: ${}.2f from {} tokens", saved, tokensCached);
        return saved;
    }
}

缓存 TTL 与失效策略

类型TTL适用场景失效方式
知识库问答24 小时 - 7 天制度 FAQ、产品文档知识库更新时主动失效
个人化查询不缓存订单状态、个人信息N/A
实时数据1-5 分钟天气、股价TTL 到期自然失效
临时会话会话结束同一 session 重复提问Session 关闭时清除
不推荐
  • 所有问题统一 TTL（如 1 小时）
  • 无相似度阈值，精确匹配
  • 缓存命中后不看质量
  • 无缓存失效机制
推荐
  • 按问题类型分 TTL
  • embedding 余弦相似度 + 可调阈值
  • 定期采样检查缓存质量
  • 知识更新时主动失效缓存

Route Cheap：模型路由降本

Route cheap 策略：简单问题用便宜小模型，复杂问题用贵的大模型。CorpAssist 中：

  • 简单问题（查政策、查状态）→ deepseek-v3（$0.001/1K tokens）
  • 复杂问题（多步推理、生成报告）→ claude-sonnet-4（$0.003/1K tokens）
  • 关键任务（退款、法律咨询）→ gpt-4o（$0.005/1K tokens）
语义缓存的 embedding 成本怎么算？
每次缓存未命中时产生 1 次 embedding 调用（约 $0.0001）。如果命中率 40%，每 100 次请求：60 次 embedding + 40 次命中，embedding 成本约 $0.006。相比 LLM 调用的 $0.30-3.00，embedding 成本微不足道。
相似度阈值怎么调优？
从 0.95 开始，如果命中率 < 20% 逐步降低到 0.90。调低阈值会增加命中率但也增加返回不相关内容的风险。推荐用 A/B 测试：先对 5% 流量用新阈值，对比用户满意度和质量分。
Savings metric 怎么计算？
Savings = (未命中时 LLM 成本总和) - (缓存存储 + embedding 成本)。更直观的指标是「平均每请求成本」：部署缓存前 $0.05/请求 → 部署后 $0.03/请求，节省 40%。

本章结论

  • 语义缓存大幅降本：相似度匹配命中率 30-60%，可降低 30-60% 的 LLM token 成本和延迟。
  • 分 TTL 策略：知识库长 TTL、个人化不缓存、实时数据短 TTL。
  • Route cheap：简单用小模型、复杂用大模型，进一步降本。
  • 监控命中率：缓存命中率 < 20% 说明阈值过高或问题多样性太大，需要调优。
  • 知识更新主动失效：RAG 索引更新后必须清除相关缓存，否则返回过期信息。
