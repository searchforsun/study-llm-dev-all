已完成告警与 oncall

约 40 分钟 · 阶段：可靠性运维 · 能力：为 CorpAssist 设计分级告警、升级链与可执行 Runbook

为什么要学本章

Dashboard 与 SLO 定义了「正常」边界，但越界时谁被叫醒、先做什么仍须工程化。alert fatigue 常来自「每条指标抖动都 P1 电话」或告警无 Runbook。本章把 CorpAssist 双栈（Spring BFF + Python RAG）的 oncall 流程落到可执行规则。

学完你能
  • 编写带 severity level 与 burn-rate 的 alert rule
  • 设计 oncall escalation：L1 群消息 → L2 电话 → 经理
  • 为每条 P1/P2 告警绑定 runbook link，支撑 incident triage
本章先记住 3 件事
  • 告警 = 可行动信号：须链到 Runbook，而非「图表红了」。
  • SLO 驱动分级：slo alert 用错误预算 burn-rate，比固定阈值更抗抖动（衔接 Dashboard 与 SLO）。
  • 双栈统一 request_id：oncall 从告警跳进 Trace/日志时，日志、指标、追踪 的 correlation id 必须贯通。

规则

CorpAssist · 智能客服 oncall（S2） 周五 22:10，Grafana 触发「LLM P95 > 3s」。oncall 工程师打开告警，需 30 秒内判断：是模型侧慢、RAG 检索慢，还是下游 ERP 超时？若 alert rule 只写 latency > 3 而无 service/tenant 维度，会把「单租户压测」误判为全站事故。

LLM 应用告警除传统 5xx/延迟，还应覆盖：首 token 延迟、RAG 空召回率、Token 预算（见 Token 成本治理）、工具调用失败率。规则设计原则：

原则说明CorpAssist 示例
用户面向 告警 SLI 对齐 SLO 定义 P95 端到端含 BFF + Python + LLM
多维分组 service、tenant_id、model 避免「finance 租户」拖垮全局阈值
抑制抖动 for: 5m 或 burn-rate 窗口 短 spike 只记指标，不叫醒人
symptom vs cause 用户症状告警 + 组件因果告警分层 P2「P95 高」+ P3「向量库 QPS 饱和」

反例 vs 推荐

不推荐
  • CPU > 80% 即 P1 电话（与用户体验无关）
  • 同一故障 12 条重复告警（无 grouping/inhibit）
  • 阈值拍脑袋，与 SLO 文档不一致
  • 告警注解只有「请看 Grafana」
推荐
  • slo alert：5m burn-rate > 14× 预算消耗
  • 父告警抑制子告警（ingress 5xx 抑制 pod restart）
  • 每条规则带 runbook link 与 dashboard 深链
  • LLM 专属：TTFT、拒答率、工具超时率

Prometheus 告警规则示例

yaml 
groups:
  - name: corpassist_slo
    rules:
      - alert: CorpAssistChatP95High
        expr: |
          histogram_quantile(0.95,
            sum(rate(http_server_duration_bucket{service="corpassist-bff", route="/api/chat"}[5m])) by (le)
          ) > 3
        for: 5m
        labels:
          severity: P2
          team: llm-platform
        annotations:
          summary: "CorpAssist 客服 P95 > 3s"
          runbook_url: "https://wiki.internal/runbooks/llm-latency"
          dashboard: "/d/corpassist/chat-latency?var-tenant={{ $labels.tenant_id }}"
      - alert: TokenBudgetBurnFast
        expr: |
          sum(increase(llm_tokens_total[1h])) by (tenant_id)
            / on(tenant_id) token_budget_hourly > 0.9
        for: 10m
        labels:
          severity: P3
        annotations:
          runbook_url: "https://wiki.internal/runbooks/token-budget"
先想 10 秒：为什么 Token 预算告警设 P3 而非 P1？

超预算通常不立即伤害可用性，但预示成本失控或遭滥用；P3 群消息 + 限流 Runbook 足够，除非连带拖垮推理集群才升级。

升级

oncall escalation 回答：谁第一响应、多久未 ACK 升级、何时拉经理。severity level 与通知渠道须一一映射，避免 P1 只发 Slack。

级别定义（CorpAssist）通知ACK 时限
P1 全站客服不可用 / 大面积越权风险 pager duty 电话 + war room 5 分钟
P2 SLO 显著违约、单区域降级 IM @oncall + 工单 15 分钟
P3 预算/质量分预警、非用户面向 群消息，工作时间处理 下一工作日
CorpAssist oncall 升级链
flowchart TD
  A[告警触发] --> B{L1 oncall ACK?}
  B -->|5min 内| C[incident triage]
  B -->|超时| D[L2 备班电话]
  C --> E{需变更/回滚?}
  E -->|是| F[拉 SRE + 业务负责人]
  E -->|否| G[Runbook 闭环]
  D --> C

L1 负责 incident triage：先定范围（租户/版本/模型），再选 Runbook 分支。

L1 oncall

ACK、开 incident 频道、按 Runbook 前 3 步采集 trace_id。

L2 领域专家

RAG/模型/网关分工；决定是否触发灰度回滚（见 灰度与版本管理）。

经理 escalation

P1 超 30 分钟未缓解、或需对外公告时介入。

对抗 alert fatigue
  • 合并：同一 root cause 只保留一条父告警。
  • 静默窗口：计划发布期间抑制已知噪声（须登记变更单）。
  • 定期回顾：每月统计「无行动告警」比例，删或降级规则。

Runbook

runbook link 是告警的「下一步按钮」。好的 Runbook 在 3 分钟内让 L1 完成：确认影响面 → 定位组件 → 临时缓解。

Runbook 模板（LLM P95 高）

markdown 
# Runbook: CorpAssist 客服 P95 过高

## 0. 影响确认（2 min）
- [ ] Dashboard: chat-latency — 是否全租户 or 单 tenant_id?
- [ ] 最近变更: prompt / model / index 版本（version-registry）
- [ ] 关联 Trace: 过滤 service=corpassist-bff AND span=llm.invoke

## 1. 快速分流
| 现象 | 可能根因 | 动作 |
|------|----------|------|
| Python RAG span 慢 | 向量库 / 重排 | 查 retrieval P95 |
| llm.invoke 慢 | 模型 API 限流 | 切备用 model-version |
| BFF 5xx 升 | 下游 ERP | 熔断 + 降级回复 |

## 2. 缓解
- 触发 rollback-switch（feature-flag 回滚上一 prompt-version）
- 若单租户: 租户级限流

## 3. 升级条件
- 15 min 未恢复 → L2 + 考虑 P1
- 怀疑越权 → 立即拉安全 oncall

Runbook 中的 Trace/日志跳转依赖 LangSmith 与 Micrometer 统一埋点；排障细节见 日志分析。

Runbook 写进 Wiki 还是放在 Git？
Git 版本化 + Wiki 渲染均可；关键是告警 annotations.runbook_url 稳定可访问，且与代码变更同步 review。
PagerDuty 与 Grafana OnCall 选哪个？
工具无关；pager duty 在本课指轮值排班 + 电话升级能力，须与 severity 映射文档化。

本章结论

每条对应上文一个小节。

  • 规则：SLO 对齐、多维分组、burn-rate 抑抖；LLM 补 TTFT/Token/工具失败维度。
  • 升级：P1–P3 映射通知渠道与 ACK 时限；用合并与静默对抗 alert fatigue。
  • Runbook：告警必链可执行步骤；CorpAssist 先定租户/版本范围再分流 BFF/RAG/LLM。

动手练习

基于 CorpAssist 客服 S2 场景，完成告警规则表、升级矩阵与Runbook 骨架（约 25 分钟）。

操作步骤

  1. 对照 Dashboard 与 SLO，为「P95 < 3s、错误率 < 0.1%」各写一条 alert rule 草案（含 for、severity、runbook_url）。
  2. 填写 P1/P2/P3 升级矩阵：通知渠道、ACK 时限、超时后升级到谁；标注哪些走 pager duty 电话。
  3. 为「LLM P95 高」Runbook 写 §0 影响确认清单（至少 3 个 checkbox），含 tenant_id 与最近 prompt/model 版本查询。
  4. （可选）完成 demos/practice-01-alerting-lab/，与上述产出交叉核对。

判断练习

  1. oncall 收到 12 条「Pod CPU > 80%」同时告警，用户侧无感知。最应先做：① 全部 ACK ② 合并为一条父告警并查 ingress 5xx ③ 逐条打电话确认 ④ 删除 CPU 规则。

    参考答案

    ②：先找用户面向症状，配置 inhibit 避免 alert fatigue。④ 可后续评估，但不应在 incident 中仓促删除。

  2. 哪条违反 good alert 原则？① slo alert 用 5m burn-rate ② P1 告警无 runbook_url ③ Token 超预算 P3 群消息 ④ 告警按 tenant_id 分组

    参考答案

    ②：P1 无 Runbook 会导致 incident triage 无从下手。①③④ 均符合本章原则。
