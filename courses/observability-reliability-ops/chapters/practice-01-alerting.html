约 40 分钟 · 阶段：可靠性运维 · 能力：设计 CorpAssist 分级告警规则、升级策略与 Runbook

为什么要学本章

前几章搭建了观测和 SLO，但指标不会自动让人醒——需要告警把「异常信号」变成「人该行动了」。告警的常见陷阱：噪音太多导致 alert fatigue，或者告警来了没人知道怎么处理。本章用 CorpAssist 场景教你怎么设计有效的 alert rule、oncall escalation 和 runbook link。

学完你能
  • 设计 CorpAssist 的分级告警规则（严重等级对应不同的 MTTR 期望）
  • 配置 severity level 和升级策略，确保告警不会漏掉
  • 为每个告警关联 Runbook，让 oncall 能「按剧本操作」
  • 理解 slo alert 与基础设施告警的区别
本章先记住 2 件事
  • 告警是剧本索引不是通知：每个告警消息必须包含到 Runbook 的链接，否则 oncall 不知道下一步做什么。
  • SLO 告警优先于指标告警：当 SLO 错误预算燃烧时才真正需要 page；单指标抖动（如某台机器 CPU 高）应先自查再升级。

告警规则设计

CorpAssist · oncall 的一天（30 秒） 凌晨 3 点，你被 PagerDuty 叫醒：「CorpAssist P95 延迟 > 5s」。你打开 Runbook，第一步：检查 Grafana 看板确认影响范围。第二步：查 Tempo 定位慢在哪个服务。第三步：如果是 LLM 超时，切备用模型。整个过程应该不超过 15 分钟——前提是告警消息里直接有 Grafana 链接和 Runbook 地址。

Prometheus 告警规则

yaml 
# prometheus-alerts.yaml — CorpAssist 告警规则
groups:
  - name: corpassist-slo
    rules:
      # P0: SLO 错误预算燃烧过快
      - alert: SLOErrorBudgetBurnRate
        expr: |
          (1 - (sum(rate(http_server_requests_seconds_count{status="200"}[1h]))
                / sum(rate(http_server_requests_seconds_count[1h]))))
          * 100 > 5
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "CorpAssist 错误预算燃烧过快"
          runbook: "https://wiki/runbooks/slo-burn"

      # P1: LLM 延迟 P95 超阈值
      - alert: LLMHighLatency
        expr: |
          histogram_quantile(0.95,
            sum(rate(llm_client_latency_bucket[5m])) by (le)
          ) > 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "LLM P95 延迟 > 10s"
          runbook: "https://wiki/runbooks/llm-latency"

      # P2: 租户级错误率上升
      - alert: TenantErrorRate
        expr: |
          sum(rate(llm_client_errors_total[5m])) by (tenant_id)
          / sum(rate(llm_client_latency_count[5m])) by (tenant_id)
          > 0.05
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "租户 {{ $labels.tenant_id }} 错误率超过 5%"

Java：告警事件日志与严重等级

java 
public enum Severity {
    CRITICAL("P0", 15),   // 15 分钟响应
    HIGH("P1", 30),       // 30 分钟
    MEDIUM("P2", 120),    // 2 小时
    LOW("P3", 1440);      // 24 小时

    public final String label;
    public final int responseMinutes;

    Severity(String label, int responseMinutes) {
        this.label = label;
        this.responseMinutes = responseMinutes;
    }
}

// 告警事件记录器
@Component
public class AlertEventLogger {
    private static final Map RUNBOOKS = Map.of(
        Severity.CRITICAL, "https://wiki/runbooks/critical",
        Severity.HIGH, "https://wiki/runbooks/high",
        Severity.MEDIUM, "https://wiki/runbooks/medium"
    );

    public void logAlert(Severity severity, String title,
                         String summary, String tenantId) {
        log.warn("ALERT severity={} title={} summary={} tenant={} runbook={}",
            severity.label, title, summary, tenantId,
            RUNBOOKS.get(severity));
        // 发送到 PagerDuty / 钉钉 / Slack
        alertWebhook.send(Map.of(
            "severity", severity.label,
            "title", title,
            "summary", summary,
            "tenant", tenantId,
            "runbook", RUNBOOKS.get(severity)
        ));
    }
}

Python：告警消息处理

python 
import json
import requests
from typing import Optional

class AlertManager:
    """告警管理 —— 格式化并发送告警到多个通道。"""

    PAGERDUTY_URL = "https://events.pagerduty.com/v2/enqueue"
    SLACK_WEBHOOK = "https://hooks.slack.com/services/xxx"

    def send_alert(self, severity: str, title: str,
                   summary: str, runbook: str,
                   tenant: Optional[str] = None):
        payload = {
            "title": f"[{severity}] {title}",
            "summary": summary,
            "tenant": tenant,
            "runbook": runbook,
        }
        if severity in ("P0", "P1"):
            self._pagerduty(payload)
        else:
            self._slack(payload)

    def _pagerduty(self, payload: dict):
        body = {
            "routing_key": "your-pd-key",
            "event_action": "trigger",
            "payload": {
                "summary": payload["title"],
                "severity": "critical" if payload.get("severity") == "P0" else "error",
                "source": "corpassist-alert",
                "custom_details": payload
            }
        }
        requests.post(self.PAGERDUTY_URL, json=body)

    def _slack(self, payload: dict):
        body = {
            "text": f"*{payload['title']}*\n{payload['summary']}\n"
                    f"Runbook: {payload['runbook']}"
        }
        requests.post(self.SLACK_WEBHOOK, json=body)

告警升级策略

CorpAssist 告警升级流程
flowchart TB
  A[告警触发] --> B{严重等级}
  B -->|P0| C[立即电话 + 钉钉]
  B -->|P1| D[PagerDuty 通知]
  B -->|P2| E[Slack 频道]
  B -->|P3| F[Jira 工单]
  C --> G{5 分钟未确认?}
  G -->|是| H[升级到值班经理]
  D --> I{15 分钟未确认?}
  I -->|是| J[升级到团队 Lead]

每个严重等级有对应的通知渠道和未确认升级时间。P0 最短 5 分钟即升级。

Runbook 驱动排障

Runbook 是每个告警的「应急预案」。没有 Runbook 的告警只制造焦虑不解决问题。

LLM 高延迟 Runbook 模板

markdown 
# Runbook: LLM 高延迟
## Severity: P1
## 影响: CorpAssist 客服响应变慢

## Step 1: 确认影响范围
- 打开 Grafana: https://grafana.internal/d/llm-latency
- 检查受影响租户、模型、时间段
- 判断是全局还是单租户

## Step 2: 定位瓶颈
- Tempo Trace: 按 request_id 搜索最近错误请求
- 观察是 embedding / LLM / RAG 哪一段慢
- 检查 LLM provider 状态页

## Step 3: 执行止血
- 如果是 LLM provider 问题: 切备用模型
- 如果是 RAG 慢: 降级为纯 LLM 回答（无检索）
- 如果 BFF: 扩容 Pod / 重启异常实例

## Step 4: 通知
- 更新状态页 status.corpassist.com
- 在 #ops 频道发布事故通告
不推荐
  • 告警消息只有指标值没有 Runbook 链接
  • 所有告警发同一个通道（Slack #general）
  • P0/P1/P2 都用同样的响应时间要求
  • 告警来了才临时想怎么处理
推荐
  • 每条告警 annotations 携带 Runbook URL
  • P0 电话 → P1 PagerDuty → P2 Slack → P3 Jira
  • 不同级别不同响应时间和升级策略
  • 每个 Runbook 有 Step1/2/3/4 的具体操作
Alert fatigue 怎么避免？
核心原则：每一个告警都要对应一个人需要执行的动作。如果收到告警只是看看然后关闭，那就应该降低级别或改成日志。技巧：1) 使用 burn rate 告警而非固定阈值，2) 关联 Runbook 确保 actionable，3) 每月 review 告警列表删除噪音。
Incident triage 和告警有什么区别？
告警是信号（Alert），事故是确认的问题（Incident）。告警可能误报，也可能多个告警指向同一事故。Triage 阶段要把告警归并为事故，确认影响范围、严重等级和负责人。
oncall 轮值如何与告警配合？
推荐方案：PagerDuty 或 OpsGenie 管理轮值表。周一早 9 点交接。oncall 只需要处理 P0/P1，P2 由对应服务团队在工作时间处理。P3 走 Jira 工单队列。
常见误区
  • 指标抖动就告警 — 单次 P95 突增持续 1 分钟可能只是自然波动；加 for 子句沉降后再告警。
  • 告警无 Runbook — oncall 收到告警不知道做什么，浪费时间查资料。
  • 不区分严重等级 — LLM 超时和证书过期响应方式完全不同。
  • 告警太多导致关闭通知 — 宁可少告警但要保证每个都 actionable。

本章结论

  • 分级告警：P0/P1/P2/P3 四级，不同响应时间和通知通道。
  • Runbook 驱动：每个告警必须携带 Runbook 链接，oncall 按剧本操作。
  • 升级策略：P0 5 分钟未确认自动升级到值班经理，确保绝不漏报。
  • 告警疲劳治理：减少噪音，确保每个告警 actionable，定期 review。
  • Burn rate 优先：SLO 错误预算燃烧比单指标抖动更适合作为告警源。

操作步骤

  1. 为 CorpAssist 的 LLM 错误率写一条 Prometheus 告警规则（P1 级别）。

判断练习

  1. P95 延迟在 5 分钟内从 2s 涨到 15s 又回到 2s。要不要告警？

    参考答案

    取决于持续时间。5 分钟的抖动可能是 GC pause 或网络抖动。建议加 for: 10m 条件，持续超过 10 分钟再告警。但如果抖动频率高（每天多次），说明有慢性问题需要调查。
