为什么要学本章

前一章建好了 OTel 基线，但 CorpAssist 双栈存在两个「观测孤岛」：Python 侧用 LangSmith （擅长 LLM 语义 trace、token 计算），Java 侧用 Micrometer（擅长 JVM/HTTP/DB 指标）。oncall 排障时需要在两套 UI 之间来回切换，无法在一个地方看完「这个请求花了多少钱 + 慢在哪 + JVM GC 是否异常」。本章讲如何用 trace link 和 unified dashboard 打破孤岛。

本章先记住 2 件事
  • 双栈不意味着双面板：cross stack obs 的目标是在同一张 Grafana 看板上同时看到 Java 指标与 LangSmith trace。
  • 关联靠 metadata 非靠 UI：langsmith trace 与 micrometer metrics 通过 request_id + trace_id 双向绑定，而非人工粘贴 URL。

LangSmith Trace 与 Micrometer 的双向关联

CorpAssist · 排障现场（60 秒） Oncall 收到告警「acme 租户 P95 延迟从 2s 涨到 8s」。他在 Grafana 看到 Micrometer llm.client.latency 直方图异常，点击数据点携带的 trace_id 跳转到 LangSmith 查看具体那次 LLM 调用——发现是因为某次 RAG 检索返回了 50 个 chunk 导致 context 过长。如果没有双向关联，他需要先查日志找 request_id，再复制到 LangSmith 搜索，浪费时间。

聚合能力 按 project/run 聚合 按 tag（tenant/model）多维聚合

Java BFF：Micrometer LLM 指标埋点

java 
import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Timer;
import io.micrometer.core.instrument.Counter;
import io.micrometer.core.instrument.Tag;
import java.util.List;

@Service
public class LLMMetricsService {
    private final MeterRegistry registry;

    public LLMMetricsService(MeterRegistry registry) {
        this.registry = registry;
    }

    // 记录 LLM 调用延迟 — 分模型和租户
    public void recordLatency(String modelId, String tenantId,
                              String traceId, long durationMs) {
        Timer.Sample sample = Timer.start(registry);
        List tags = List.of(
            Tag.of("model_id", modelId),
            Tag.of("tenant_id", tenantId),
            Tag.of("trace_id", traceId)
        );
        Timer timer = Timer.builder("llm.client.latency")
            .description("LLM 调用延迟毫秒")
            .tags(tags)
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
        sample.stop(timer);
    }

    // 记录 Token 消耗 — 分模型和租户
    public void recordTokenUsage(String modelId, String tenantId,
                                 int promptTokens, int completionTokens) {
        Counter.builder("llm.tokens.prompt")
            .tag("model_id", modelId)
            .tag("tenant_id", tenantId)
            .register(registry)
            .increment(promptTokens);

        Counter.builder("llm.tokens.completion")
            .tag("model_id", modelId)
            .tag("tenant_id", tenantId)
            .register(registry)
            .increment(completionTokens);
    }

    // 记录错误数
    public void recordError(String modelId, String errorType) {
        Counter.builder("llm.client.errors")
            .tag("model_id", modelId)
            .tag("error_type", errorType)
            .register(registry)
            .increment();
    }
}

// 在 ChatClient 中使用
@Service
public class LLMService {
    private final LLMMetricsService metrics;
    private final ChatClient chatClient;

    public String chat(ChatRequest request) {
        long start = System.currentTimeMillis();
        String traceId = MDC.get("trace_id");
        try {
            String result = chatClient.prompt()
                .user(request.getQuestion())
                .call()
                .content();
            long duration = System.currentTimeMillis() - start;
            metrics.recordLatency(request.getModelId(),
                request.getTenantId(), traceId, duration);
            metrics.recordTokenUsage(request.getModelId(),
                request.getTenantId(), 150, 80);
            return result;
        } catch (Exception e) {
            metrics.recordError(request.getModelId(),
                e.getClass().getSimpleName());
            throw e;
        }
    }
}

Python：将 request_id 写入 LangSmith metadata

python 
from langsmith import traceable
from langchain_core.runnables import RunnableConfig

@traceable(
    project_name="corpassist-agent",
    metadata={"source": "agent-service"}
)
def invoke_agent(question: str, request_id: str,
                 tenant_id: str, model_id: str) -> str:
    """CorpAssist 客服 Agent 入口。

    request_id 和 tenant_id 注入 LangSmith metadata，
    用于从 Grafana 跳转到此 trace。
    """
    config = RunnableConfig(
        metadata={
            "request_id": request_id,
            "tenant_id": tenant_id,
            "model_id": model_id,
            "service": "agent-service",
        }
    )
    # 构建 RAG chain
    chain = (
        {"context": retriever, "question": lambda x: x}
        | prompt_template
        | llm
    )
    result = chain.invoke(question, config=config)
    return result.content

从指标到 trace 的跳转

Grafana 的数据链接（Data Link）功能允许在点击面板数据点时携带变量跳转到其他系统。CorpAssist 的跳转链路：

  • Metrics → Trace：面板图标携带 trace_id tag，点击跳转 Grafana Tempo 的 trace 视图
  • Trace → LangSmith：Tempo span 的 request_id attribute 作为链接跳转 https://smith.langchain.com/…
  • Trace → Logs：Tempo 与 Loki 集成，span 点一下显示关联日志行

Trace Link 实战

一次排障的跳转路径
flowchart LR
  A[Grafana 告警\nP95 > 5s] -->|点击面板| B[Tempo Trace View]
  B -->|查看 span detail| C[发现 RAG span 耗时 4.2s]
  B -->|request_id 跳转| D[LangSmith Trace]
  D -->|查看 token 详情| E[发现 context 过长]
  B -->|trace_id 关联| F[Loki 日志]
  F -->|查看 BFF 日志| G[发现 Python 返回 503]

一次排障在多个系统间的跳转闭环，所有跳转只需点击无需复制 ID。

不推荐
  • Grafana 面板上只有指标没有 Trace 跳转链接
  • LangSmith trace 不注入 request_id metadata
  • 指标 tag 不含 trace_id，无法从指标下钻
  • 所有面板堆在同一个大看板上，不分层级
推荐
  • 关键面板配置 Data Link 跳转到 Tempo 和 LangSmith
  • LangSmith metadata 包含 request_id + tenant_id
  • Micrometer 指标 tag 包含 trace_id（有限基数）
  • 看板按 L1/L2/L3 分层，从宏观到微观

Micrometer 指标命名规范

为了避免指标混乱，CorpAssist 定义以下 LLM 相关指标命名规则：

指标名类型Tag说明
llm.client.latencyTimermodel_id, tenant_id, trace_idLLM 调用延迟
llm.tokens.promptCountermodel_id, tenant_id输入 token 数
llm.tokens.completionCountermodel_id, tenant_id输出 token 数
llm.client.errorsCountermodel_id, error_typeLLM 调用错误
rag.retrieval.latencyTimertenant_idRAG 检索延迟
rag.chunks.returnedDistributionSummarytenant_id每次返回 chunk 数
常见误区
  • trace_id 作为 Micrometer tag — trace_id 基数极高（每个请求不同），直接作为 Prometheus tag 会导致时序膨胀。只对错误请求或慢请求附加 trace_id，正常请求不携带。
  • 只看 LangSmith 忽略 JVM — LLM 慢可能是因为 Java 侧 GC 或连接池瓶颈，这些在 LangSmith 里看不到。
  • 所有 span 导两份 — OTel span 已经包含调用信息，不需要再手动通过 Micrometer 记录同维度，避免重复。
为什么 Micrometer 指标要限制 tag 基数？

时序数据库（Prometheus）对每个唯一 tag 组合都会创建新时间序列。如果 trace_id 作为 tag 写入每个请求，每分钟数万新序列会迅速撑爆存储。技巧：只在 错误请求 和 慢请求（P99+） 附加 trace_id tag。

LangSmith 能否替代 OTel？
不能替代。LangSmith 专注于 LLM 调用层次的 trace（prompt、token、模型参数），不覆盖 HTTP 调用、DB 访问、JVM 系统指标。最佳实践是 LangSmith + OTel 并用，通过 request_id 关联。
Micrometer 和 Spring Actuator 有什么关系？
Spring Actuator 提供开箱即用的健康检查和预定义指标端点；Micrometer 是 Actuator 的指标门面，可以导出到 Prometheus、Datadog 等后端。Actuator 暴露 /actuator/prometheus 端点供 Prometheus 抓取。
使用 LangSmith 的 LLM span 能否导出到 Grafana？
LangSmith 支持 OTel 格式导出（LANGSMITH_EXPORT_OTEL=true），将 LLM span 输出到 OTel Collector。这样 LLM span 会在 Tempo 中出现，但元数据丰富度不如 LangSmith 原生界面。通常策略：Grafana 看全景，下钻细节跳转 LangSmith。

本章结论

  • 双栈关联：request_id 是连接 LangSmith trace 与 Micrometer 指标的桥梁，必须在两层同时注入。
  • 看板分层：统一看板按 L1 全局 / L2 租户 / L3 详情分层设计，既满足值班快速判断也支持深度排障。
  • 跳转闭环：Grafana → Tempo → LangSmith → Loki 的四步跳转链路让 oncall 无需复制任何 ID。
  • 指标命名规范：LLM 相关指标按 llm.* 前缀分 model 和 tenant tag，避免命名混乱。
  • 合理使用 trace_id tag：只对错误和慢请求附加 trace_id，防止时序数据库基数爆炸。

判断练习

  1. 你需要在 Grafana 上增加一个「点击请求行跳 LangSmith」的功能。以下哪种方式正确？A) 在面板设置中配置 Data Link 携带 trace_id B) 在 PromQL 中用 label_join 拼接 URL C) 在面板下方贴一个静态链接

    参考答案

    A。Data Link 是 Grafana 原生支持的数据点跳转机制，支持从查询结果中取变量。B 方式拼接 URL 不是标准做法；C 静态链接无法动态携带 request_id。

  2. 以下哪些 tag 适合作为 LLM 指标的常量 tag？① tenant_id ② trace_id ③ model_id ④ user_id ⑤ request_id

    参考答案

    ① 和 ③。tenant_id 和 model_id 基数有限（几十到几百），适合作为常量 tag。trace_id/user_id/request_id 基数无限，只能选择性附加到错误/慢请求。
