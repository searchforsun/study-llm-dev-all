为什么要学本章

指标告诉你「系统慢了」，Trace 告诉你「慢在哪个 span」，但为什么慢？——需要日志的详细上下文。LLM 系统的日志尤其复杂：prompt 片段、RAG chunk 来源、tool 调用参数。本章讲如何把结构化日志与 trace correlation 结合，用 log search 在秒级定位根因。

学完你能
  • 在 ELK / Loki 中按 request_id 搜索完整请求日志链
  • 从 Trace span 跳转到关联日志行，实现双向关联
  • 用 log query 分析「错误请求的共有模式」
  • 配置 span link 从 Tempo 跳转到 Loki
本章先记住 2 件事
  • 日志是 Trace 的注脚：Trace 告诉路径，日志告诉「每一站发生了什么」。两者组合才能闭环排障。
  • 结构化是搜索的前提：纯文本日志只能 grep，结构化 JSON 日志才能聚合分析。

    }

// 在代码中使用 MDC 填充关联字段
@Service
public class AgentService {
    public String process(ChatRequest request) {
        MDC.put("request_id", request.getRequestId());
        MDC.put("tenant_id", request.getTenantId());
        MDC.put("trace_id", Span.current().getSpanContext().getTraceId());
        try {
            log.info("Processing chat request, question_len={}",
                request.getQuestion().length());
            String result = callAgent(request);
            log.info("Chat completed, result_len={}", result.length());
            return result;
        } catch (Exception e) {
            log.error("Chat failed", e);
            throw e;
        } finally {
            MDC.clear();
        }
    }
}

Python：结构化日志配置

python 
import structlog
import logging
from opentelemetry import trace

# 配置结构化日志
structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.stdlib.filter_by_level,
        structlog.stdlib.add_logger_name,
        structlog.stdlib.add_log_level,
        structlog.stdlib.PositionalArgumentsFormatter(),
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
        structlog.processors.UnicodeDecoder(),
        structlog.processors.JSONRenderer()
    ],
    wrapper_class=structlog.stdlib.BoundLogger,
    context_class=dict,
    logger_factory=structlog.stdlib.LoggerFactory(),
    cache_logger_on_first_use=True,
)
log = structlog.get_logger()

# 中间件：自动注入 trace_id 和 request_id
@app.middleware("http")
async def logging_middleware(request: Request, call_next):
    request_id = request.headers.get("x-request-id", "unknown")
    span = trace.get_current_span()
    span_ctx = span.get_span_context()
    trace_id = format(span_ctx.trace_id, "032x") if span_ctx else "none"
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(
        request_id=request_id,
        trace_id=trace_id,
        service="corpassist-agent"
    )
    response = await call_next(request)
    return response

@app.post("/v1/agent/chat")
async def chat(request: Request, body: ChatRequest):
    structlog.contextvars.bind_contextvars(
        tenant_id=body.tenant_id,
        model_id=body.model_id
    )
    log.info("agent_start", question_len=len(body.question))
    result = await invoke_agent(body)
    log.info("agent_complete", result_len=len(result))
    return result

日志与 Trace 的关联架构

日志 ↔ Trace 双向关联
flowchart LR
  BFF[Spring AI BFF] -->|structlog| L1[Loki / ELK]
  PY[Python Agent] -->|structlog| L1
  BFF -->|OTel| T[Tempo]
  PY -->|OTel| T
  T -->|trace_id| L1
  L1 -->|request_id| T
  D[Grafana] -->|跳转| T
  D -->|跳转| L1

日志和 Trace 通过 trace_id 关联。在 Grafana 中，Tempo span 详情页配置 Loki 数据源，点击即可查看关联日志。

不推荐
  • 纯文本日志，无 trace_id 字段
  • 日志和 Trace 存在不同系统，无法交叉搜索
  • 日志格式各服务自行定义，无法统一搜索
  • 生产日志包含完整 prompt（含 PII）
推荐
  • JSON 结构化日志含 trace_id/request_id
  • Tempo + Loki 集成，点 span 可看日志
  • 所有服务统一日志 schema（timestamp/level/trace_id）
  • 日志只记录 token 数和 hash 值，不记录 PII

日志分析排障案例

Debug session 示例：acme 租户反馈客服不回答某些问题。

步骤操作日志查询
1找到问题请求的 request_id（用户提供）前端日志或 user feedback 中的 trace
2搜索该 request_id 的完整日志链{service=~".*"} |= req-001
3发现 RAG 返回空结果rag.chunks=0
4检查 RAG 查询日志，发现 query 被过滤acl filter blocked query
5根因：该租户 ACL 配置缺失对应知识库权限通知租户管理员修复 ACL
ELK 和 Loki 怎么选？
ELK（Elasticsearch + Logstash + Kibana） 功能全支持复杂聚合分析但资源消耗大；Loki 轻量、与 Grafana 原生集成、适合日志与 trace 关联场景。CorpAssist 推荐 Loki + Tempo + Grafana 三件套。
如何防止日志泄露 PII？
在日志输出层做过滤：1) 不记录完整 prompt 只记录 token 数和 hash，2) 使用 log redaction 库自动检测和屏蔽手机/身份证/邮箱，3) 日志系统中配置字段白名单，未在白名单的字段不存储。详见安全合规课程。
Request trail 和 debug session 的区别？
Request trail 是请求完整日志链的被动查询（事后搜索）；debug session 是主动录制（预先对特定租户或用户开启调试日志级别）。CorpAssist 用 request trail 做常规排障，用 debug session 做深度排查。
常见误区
  • 日志不打 trace_id — 脱离了 Trace 上下文的日志只能按时间猜链路。
  • 各服务日志格式不同 — BFF 用 JSON、Python 用 plain text、RAG 用 syslog，无法统一查询。
  • 日志级别不合理 — 所有日志都是 INFO，ERROR 日志里没有 stack trace。
  • 不配置 Tempo-Loki 集成 — 排障时需要在两个系统间手动复制 ID。

本章结论

  • 结构化日志是基础：JSON 格式 + 统一 schema（trace_id/request_id/tenant_id）是搜索和关联的前提。
  • Trace ↔ Log 双向关联：Tempo-Loki 集成使排障时从 span 一键跳到日志，从日志一键跳到 trace。
  • LogQL 聚合分析：通过日志聚合发现错误模式（如哪个 tool 最常超时、哪个租户错误最多）。
  • 避免日志 PII：日志中不记录 prompt 原文、用户个人信息。用 hash 或 token 数替代。
  • Request trail + debug session：事后搜索 + 主动录制配合使用，覆盖不同排障场景。
