为什么要学本章

CorpAssist 是 Spring AI BFF + Python Agent/RAG 双栈：用户一次客服提问会穿越网关、Java 编排、Python 检索与 LLM 调用。若只有「Java 有日志、Python 有 LangSmith」而没有统一的 correlation id，oncall 只能凭时间戳猜链路——成本也无法按租户归因。本章用 OpenTelemetry 把 three pillars 落到双栈工程约定，为后续 LangSmith/Micrometer 看板打地基。

  • 为 CorpAssist BFF 与 Python 服务列出 obs baseline 必含字段（含 request_id、trace_id、tenant_id）
  • 配置 W3C Trace Context 传播，使一次 S2 客服请求在 Java 与 Python 侧落在同一 trace

本章先记住 2 件事
  • 三支柱分工：日志讲「发生了什么」、指标讲「有多糟/多快」、追踪讲「慢在哪一段」——单靠其中一种无法闭环排障。
  • 双栈先统一 ID：dual stack instrument 的第一优先级是 request_id + W3C traceparent 全链路贯通，再谈各自 SDK 细节。

OTel 三支柱数据模型

OpenTelemetry（OTel）是厂商中立的观测标准：定义 trace/metric/log 的数据模型、API 与 Collector 管道，避免「Java 用 APM、Python 用 B 产品、日志又在 C 系统」的三套孤岛。CorpAssist 选型 OTel 作为传播与导出层，LangSmith 仍负责 LLM 语义 span，Micrometer 负责 Spring 侧指标——二者通过同一 trace 根 ID 关联。

三支柱：各回答什么问题？

支柱典型信号CorpAssist 示例排障场景
Logs 离散事件文本/JSON BFF 记录「转发 RAG 请求 tenant=acme」 查某次 403 是否 ACL 拦截
Metrics 聚合数值（counter/histogram） llm.tokens.total{tenant=acme} 某租户 Token 成本突增
Traces 请求级 span 树 + 耗时 gateway → bff → rag-service → embedding P95 从 2s 涨到 8s，定位慢在向量检索

三者在 OTel 里共享 Resource（service.name、deployment.environment）与 Context（trace/span id），这就是 metrics trace 关联的基础：指标卡片上点一下能跳进 trace，trace 里又能搜到同 ID 的日志。

反例 vs 推荐：观测选型

不推荐
  • Java 用 Zipkin、Python 用 Jaeger 不同 trace 格式
  • 只在 BFF 开 trace，Python 侧无 SDK
  • 日志纯文本，无 trace_id 字段
  • 100% 采样导致 Collector 与存储成本失控
  • 异常时只打印 stack trace 不记录 request_id
推荐
  • OTel + W3C Trace Context 作为唯一传播标准
  • 双栈均接入 OTel SDK 或兼容 Agent
  • structured log JSON，含关联字段
  • 按路由/租户分层采样（错误请求 100% 保留）
  • 异常日志包含 trace_id 供快速跳转
先想 10 秒：只有 Metrics 没有 Traces，能定位「RAG 慢还是 LLM 慢」吗？

很难。聚合 P95 只能告诉你「整体变慢」；没有 span 树无法区分 embedding、向量库、重排、LLM 哪一段占时。Metrics 适合告警，Traces 适合根因分析——three pillars 互补而非替代。

双栈埋点与关联字段

CorpAssist · 智能客服 S2（45 秒） 用户问「我的订单 ORD-8842 到哪了？」请求经网关进入 Spring AI BFF：鉴权、注入 request_id、调用 Python agent-service 查 ERP 工具。Python 侧又触发 RAG 检索制度 FAQ。oncall 收到告警「P95 > 5s」——若没有 correlation id 贯通，Java 日志与 Python 日志无法拼成一条故事线。

dual stack instrument 的核心不是「两边各写一套」，而是同一套关联键 + 同一套传播头。CorpAssist obs baseline 要求如下。

结构化日志必含字段

字段谁生成用途缺失后果
request_id 网关或 BFF（入口） 业务级关联键；写入 LangSmith metadata 无法从 Grafana 跳进 LangSmith trace

Java BFF：MDC + 出站传播

java 
import io.opentelemetry.api.baggage.Baggage;
import org.slf4j.MDC;
import jakarta.servlet.Filter;

// Filter：入口注入 request_id，写入 MDC 与 OTel Baggage
@Component
public class ObservabilityFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpReq = (HttpServletRequest) req;
        String requestId = Optional.ofNullable(
            httpReq.getHeader("X-Request-Id"))
            .orElse(UUID.randomUUID().toString());
        String tenantId = extractTenant(httpReq);
        MDC.put("request_id", requestId);
        MDC.put("tenant_id", tenantId);
        MDC.put("request_path", httpReq.getRequestURI());
        // 将 request_id 写入 Baggage 传播给下游
        Baggage.current().toBuilder()
            .put("request_id", requestId)
            .put("tenant_id", tenantId)
            .build().makeCurrent();
        try {
            chain.doFilter(req, res);
        } finally {
            MDC.clear();
        }
    }
}

// RestClient 调 Python：traceparent 由 OTel 自动注入
@Service
public class AgentClient {
    private final RestClient restClient;

    public AgentClient(RestClient.Builder builder) {
        this.restClient = builder.baseUrl("http://agent-service:8000").build();
    }

    public String chat(ChatRequest payload) {
        return restClient.post()
            .uri("/v1/agent/chat")
            .header("X-Request-Id", MDC.get("request_id"))
            .body(payload)
            .retrieve()
            .body(String.class);
    }
}

Python 服务：接收头 + 结构化日志

python 
import structlog
from opentelemetry import trace
from opentelemetry.propagate import extract

# 初始化结构化日志
structlog.configure(
    processors=[
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer()
    ]
)
log = structlog.get_logger()
tracer = trace.get_tracer(__name__)

@app.post("/v1/agent/chat")
async def chat(request: Request, body: ChatRequest):
    # 提取上游传递的 trace context
    ctx = extract(request.headers)
    request_id = request.headers.get("x-request-id", "unknown")
    tenant_id = body.tenant_id

    with tracer.start_as_current_span("agent.chat", context=ctx) as span:
        span.set_attribute("request_id", request_id)
        span.set_attribute("tenant_id", tenant_id)
        span.set_attribute("model_id", body.model_id)

        # 结构化日志
        log.info("agent_invoke",
            request_id=request_id,
            tenant_id=tenant_id,
            trace_id=format(span.get_span_context().trace_id, "032x"),
            model_id=body.model_id,
            question_len=len(body.question))

        # 调用 RAG 服务
        rag_result = await rag_service.search(body.question, tenant_id)
        span.set_attribute("rag.chunks", len(rag_result))

        # 调用 LLM
        llm_result = await llm_service.generate(
            question=body.question,
            context=rag_result,
            model_id=body.model_id
        )
        return llm_result

采样策略：按租户与错误率分层

生产环境 100% 采样会迅速耗尽存储与成本预算。推荐分层采样策略：

请求类型采样率理由
正常请求（小租户）1-5%足够反映整体分布
正常请求（大租户）10%保证关键租户有样本
错误/慢请求100%排障不能丢现场
新版本灰度流量100%新版本需完全观测
常见误区
  • 只传 request_id 不传 trace — LangSmith 与 OTel 各一条线，无法在同一 trace 视图看全链路。
  • 日志里打完整 prompt — 违反安全课 redaction 要求；obs baseline 只记录 hash/长度/模型，不记录 PII 原文。
  • 本地 dev 关 OTel — 导致「我机器上没问题」与生产传播差异；至少 dev 也走 Collector 或 console exporter。
  • 指标 tag 无限基数 — request_id 作为 tag 会导致时序数据库爆炸；指标用于聚合，详情靠 trace。
Baggage 与 MDC 有何不同？

MDC 通常只在本进程日志可见；OTel Baggage 会随 traceparent 传到下游，适合把 tenant_id、request_id 写入 Python span attribute，无需重复解析 JWT。

OTel Collector 是必须的吗？能否直接从 SDK 写 Grafana？
OTel Collector 不是强制但强烈推荐。Collector 提供采样、重试、缓冲、路由、转换等能力；直接 SDK→Grafana 增加服务端耦合，且无法灵活调整采样策略。CorpAssist 架构中 Collector 作为所有信号的统一出口。
LangSmith 与 OTel 的关系是什么？
LangSmith 专注于 LLM 语义 trace（prompt、completion、token 用量），底层支持 OTel 格式导出。CorpAssist 中：LangSmith 负责 LLM 级别的可观测，OTel 负责基础设施级的 trace/metric/log。二者通过 request_id 关联。
Structured log 的 schema 如何管理？
推荐用 JSON Schema 或 Protobuf 定义公共字段基线。CorpAssist 的 structured log 基线字段包括：timestamp、level、service、trace_id、request_id、tenant_id、message。各服务可扩展自有字段，但基线字段必须全部填充。

本章结论

  • OTel 三支柱：Logs 记录离散事件、Metrics 聚合数值趋势、Traces 提供请求级因果关系——三者缺一不可。LLM 系统因调用链路长、成本维度多，尤其需要三者结合。
  • 双栈统一关联：W3C traceparent + request_id 贯通 Java BFF 与 Python Agent 的所有日志和 span，是排障和归因的基础设施。
  • 结构化日志基线字段：request_id、trace_id、tenant_id、model_id 四项必须出现在每行日志中，否则无法关联。
  • 分层采样：正常流量 1-10%，错误流量 100%，在成本与可观测之间取得平衡。
  • OTel Collector：作为统一出口提供采样、路由、重试能力，避免 SDK 直连后端带来的架构耦合。

LangSmith Trace 与 Micrometer 的双向关联

CorpAssist · 排障现场（60 秒） Oncall 收到告警「acme 租户 P95 延迟从 2s 涨到 8s」。他在 Grafana 看到 Micrometer llm.client.latency 直方图异常，点击数据点携带的 trace_id 跳转到 LangSmith 查看具体那次 LLM 调用——发现是因为某次 RAG 检索返回了 50 个 chunk 导致 context 过长。如果没有双向关联，他需要先查日志找 request_id，再复制到 LangSmith 搜索，浪费时间。

import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Timer;

import io.micrometer.core.instrument.Tag;
import java.util.List;

@Service
public class LLMMetricsService {
    private final MeterRegistry registry;

    public LLMMetricsService(MeterRegistry registry) {
        this.registry = registry;
    }

    // 记录 LLM 调用延迟 — 分模型和租户
    public void recordLatency(String modelId, String tenantId,
                              String traceId, long durationMs) {
        Timer.Sample sample = Timer.start(registry);
        List tags = List.of(
            Tag.of("model_id", modelId),
            Tag.of("tenant_id", tenantId),
            Tag.of("trace_id", traceId)
        );
        Timer timer = Timer.builder("llm.client.latency")
            .description("LLM 调用延迟毫秒")
            .tags(tags)
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);
        sample.stop(timer);
    }

    // 记录 Token 消耗 — 分模型和租户
    public void recordTokenUsage(String modelId, String tenantId,
                                 int promptTokens, int completionTokens) {
        Counter.builder("llm.tokens.prompt")
            .tag("model_id", modelId)
            .tag("tenant_id", tenantId)
            .register(registry)
            .increment(promptTokens);

        Counter.builder("llm.tokens.completion")
            .tag("model_id", modelId)
            .tag("tenant_id", tenantId)
            .register(registry)
            .increment(completionTokens);
    }

    // 记录错误数
    public void recordError(String modelId, String errorType) {
        Counter.builder("llm.client.errors")
            .tag("model_id", modelId)
            .tag("error_type", errorType)
            .register(registry)
            .increment();
    }
}

Micrometer 指标命名规范

为了避免指标混乱，CorpAssist 定义以下 LLM 相关指标命名规则：

指标名类型Tag说明

常见误区
  • trace_id 作为 Micrometer tag — trace_id 基数极高（每个请求不同），直接作为 Prometheus tag 会导致时序膨胀。只对错误请求或慢请求附加 trace_id，正常请求不携带。
  • 只看 LangSmith 忽略 JVM — LLM 慢可能是因为 Java 侧 GC 或连接池瓶颈，这些在 LangSmith 里看不到。
  • 所有 span 导两份 — OTel span 已经包含调用信息，不需要再手动通过 Micrometer 记录同维度，避免重复。
为什么 Micrometer 指标要限制 tag 基数？

时序数据库（Prometheus）对每个唯一 tag 组合都会创建新时间序列。如果 trace_id 作为 tag 写入每个请求，每分钟数万新序列会迅速撑爆存储。技巧：只在 错误请求 和 慢请求（P99+） 附加 trace_id tag。

LangSmith 能否替代 OTel？
不能替代。LangSmith 专注于 LLM 调用层次的 trace（prompt、token、模型参数），不覆盖 HTTP 调用、DB 访问、JVM 系统指标。最佳实践是 LangSmith + OTel 并用，通过 request_id 关联。
Micrometer 和 Spring Actuator 有什么关系？
Spring Actuator 提供开箱即用的健康检查和预定义指标端点；Micrometer 是 Actuator 的指标门面，可以导出到 Prometheus、Datadog 等后端。Actuator 暴露 /actuator/prometheus 端点供 Prometheus 抓取。
使用 LangSmith 的 LLM span 能否导出到 Grafana？
LangSmith 支持 OTel 格式导出（LANGSMITH_EXPORT_OTEL=true），将 LLM span 输出到 OTel Collector。这样 LLM span 会在 Tempo 中出现，但元数据丰富度不如 LangSmith 原生界面。通常策略：Grafana 看全景，下钻细节跳转 LangSmith。

本章结论

  • 双栈关联：request_id 是连接 LangSmith trace 与 Micrometer 指标的桥梁，必须在两层同时注入。
  • 看板分层：统一看板按 L1 全局 / L2 租户 / L3 详情分层设计，既满足值班快速判断也支持深度排障。
  • 跳转闭环：Grafana → Tempo → LangSmith → Loki 的四步跳转链路让 oncall 无需复制任何 ID。
  • 指标命名规范：LLM 相关指标按 llm.* 前缀分 model 和 tenant tag，避免命名混乱。
  • 合理使用 trace_id tag：只对错误和慢请求附加 trace_id，防止时序数据库基数爆炸。

延迟与错误率 SLI

CorpAssist · SLO 设计会议（30 秒） CorpAssist 产品经理要求「客服响应快、不出错、回答质量好」。你需要把这三条需求变成可测量的 SLI 和带数字的 SLO：例如「95% 的请求在 5 秒内完成」「错误率低于 1%」「质量分高于 85%」。本章教你怎么定义、计算和可视化这些指标。

Java：SLO 指标收集与错误预算计算

java 
import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.instrument.Gauge;
import java.util.concurrent.atomic.AtomicReference;

@Service
public class SLOService {
    private final MeterRegistry registry;
    private final AtomicReference errorBudget = new AtomicReference<>(100.0);
    private final double SLO_TARGET = 0.99; // 99% 可用性

    public SLOService(MeterRegistry registry) {
        this.registry = registry;
        Gauge.builder("slo.error_budget_remaining", errorBudget,
                AtomicReference::get)
            .description("剩余错误预算百分比")
            .register(registry);
    }

    @Scheduled(fixedRate = 60000)
    public void calculateErrorBudget() {
        double sli = queryLast30DaysSLI();
        double consumed = 1.0 - sli;
        double allowed = 1.0 - SLO_TARGET;
        double remaining = Math.max(0, (allowed - consumed) / allowed * 100);
        errorBudget.set(remaining);
        if (remaining < 20) {
            log.warn("Error budget critically low: {}%", remaining);
        }
    }

    private double queryLast30DaysSLI() {
        // 生产环境从 Prometheus 查询
        // sum(rate(http_server_requests_seconds_count{status="200"}[30d]))
        // / sum(rate(http_server_requests_seconds_count[30d]))
        return 0.995;
    }
}

// 记录每次请求的 SLI 事件
@Component
public class SLIFilter implements Filter {
    private final MeterRegistry registry;

    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain) throws Exception {
        long start = System.nanoTime();
        boolean success = true;
        try {
            chain.doFilter(req, res);
        } catch (Exception e) {
            success = false;
            throw e;
        } finally {
            long duration = System.nanoTime() - start;
            Timer.builder("sli.request.duration")
                .tag("success", String.valueOf(success))
                .register(registry)
                .record(duration, TimeUnit.NANOSECONDS);
            Counter.builder("sli.request.total")
                .register(registry).increment();
            if (success) {
                Counter.builder("sli.request.success")
                    .register(registry).increment();
            }
        }
    }
}

from langsmith import Client

class QualityScorer:
    """CorpAssist 回复质量评分器，使用 LLM-as-judge 自动评分。"""
    def __init__(self):
        self.langsmith = Client()
        self.scorer_prompt = """
        评估客服回复质量（满分 100）：
        - 准确性（0-40）：是否基于企业知识库
        - 完整性（0-30）：是否充分回答
        - 安全性（0-30）：是否拒绝违规内容

        用户问题：{question}
        回复：{response}
        参考知识：{reference}
        """

    def score_run(self, run_id: str, question: str,
                  response: str, reference: str) -> int:
        score = self._llm_score(question, response, reference)
        self.langsmith.create_feedback(
            run_id=run_id, key="quality_score",
            score=score / 100.0
        )
        return score

    def batch_score_sample(self, sample_ratio: float = 0.05):
        """采样 5% 的生产流量进行质量评估"""
        for run in self.langsmith.list_runs(
            project_name="corpassist-agent", limit=1000):
            if hash(run.id) % 100 < sample_ratio * 100:
                self.score_run(
                    run_id=run.id,
                    question=run.inputs.get("question", ""),
                    response=run.outputs.get("response", ""),
                    reference=run.inputs.get("context", "")
                )

TTFT 作为独立 SLI

TTFT（Time To First Token） 是 LLM 应用的用户感知延迟关键指标。用户不在乎总耗时，而在乎「第一个字出来了没」。CorpAssist 的延迟 SLO 应分两个维度：

维度SLISLO测量点
TTFT请求到首个 token 到达P95 < 2sBFF 或前端 SDK
完整响应请求到 SSE 完成P95 < 10sBFF

错误预算与 Burn Rate

burn rate 表示 SLO 消耗的速度。1x burn rate = 在 SLO 窗口期内恰好用完预算；10x = 在 1/10 的窗口期烧完。这是决定告警严重等级的核心依据。

Burn Rate 告警规则设计

Burn Rate持续时间SeverityCorpAssist 场景
2x1 小时WarningLLM 延迟轻微抖动
5x30 分钟CriticalRAG 检索超时飙高
10x10 分钟PageLLM 服务全挂
错误预算消耗流程
flowchart LR
  A[SLO 目标: 99%] --> B[SLI: 99.5%]
  B --> C[错误预算: 100%]
  C --> D{流量正常?}
  D -->|是| E[消耗 0.5%/月]
  D -->|否| F[消耗 10%/小时]
  E --> G[预算充足]
  F --> H[预算不足 20%]
  H --> I[冻结高风险发布]

SLO 99% 意味着每月允许 0.01 * 总请求数的错误。Grafana 面板展示剩余百分比，低于 20% 触发告警。

不推荐
  • 用 100% 作为 SLO 目标
  • 只看错误率不看消耗速率
  • 所有请求统一 SLO 不区分场景
  • 只看端到端延迟不看 TTFT
推荐
  • 99% / 99.5% 作为合理的 LLM SLO
  • 用 burn rate 判断告警严重等级
  • 客服场景 SLO 严格，文档问答宽松
  • TTFT 和总响应延迟分别定义 SLO
SLO 违约了会怎样？
SLO 违约不直接等于罚款或事故。它是工程决策信号：错误预算花完后应当停止高风险发布（新模型、新 prompt），优先投入稳定性改进。SLA（服务等级协议）才是对客户有赔偿义务的合同条款。
质量分不是主观的吗？怎么量化？
质量分使用自动评估器（LLM-as-judge）打分，有明确 rubric。虽然不如人工准确，但可以持续监控趋势。关键指标是分数分布的变化——如果本周平均分从 88% 降到 82%，需要调查原因。
Dashboard SLO 面板应该包含哪些图？
推荐包含：1) 延迟 P50/P95/P99 曲线，2) 错误率 + 错误预算剩余（双轴），3) 质量分趋势（移动平均），4) burn rate 热力图。所有面板均按 tenant 可分。
常见误区
  • 100% SLO — 不可能达到且浪费资源。LLM 调用本身有失败率和延迟方差。
  • 只看全局 SLO — 大租户掩盖小租户的问题；建议分 tenant 维度定义 SLI。
  • 错误预算告警阈值设置太紧 — burn rate 2x 持续 10 分钟就 page，导致告警疲劳。
  • 质量分每次都用 LLM 评估 — 成本高；采样 5-10% 即可反映总体趋势。

本章结论

  • 三个黄金信号：延迟 + 错误率 + 质量分是 LLM 系统的核心 SLI。质量分是 LLM 独有的信号。
  • 错误预算驱动决策：SLO 不是目标而是决策框架——错误预算消耗告警决定何时停止发布转投稳定性。
  • TTFT 作为独立 SLI：用户感知延迟的关键指标是首个 token 到达时间，应与总响应时间分开定义 SLO。
  • Burn Rate 告警：不同燃烧速率对应不同严重等级和响应时间窗口，避免告警疲劳。
  • 分租户 SLI：全局 SLO 掩盖小租户问题，应按 tenant 维度分解 SLI 和错误预算。

Java：Token 计量与成本估算

java 
import io.micrometer.core.instrument.Counter;
import io.micrometer.core.instrument.MeterRegistry;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicLong;

@Service
public class TokenMeteringService {
    private final MeterRegistry registry;
    // 租户级日用量缓存
    private final Map tenantDailyTokens;

    // 模型单价（$/1K tokens）
    private static final Map MODEL_PRICES = Map.of(
        "claude-sonnet-4", 0.003,
        "gpt-4o", 0.005,
        "deepseek-v3", 0.001
    );

    public TokenMeteringService(MeterRegistry registry) {
        this.registry = registry;
        this.tenantDailyTokens = new ConcurrentHashMap<>();
    }

    public void recordTokenUsage(String tenantId, String modelId,
                                 int promptTokens, int completionTokens) {
        // Prometheus Counter — 长期趋势
        Counter.builder("llm.tokens.prompt")
            .tag("tenant", tenantId).tag("model", modelId)
            .register(registry).increment(promptTokens);
        Counter.builder("llm.tokens.completion")
            .tag("tenant", tenantId).tag("model", modelId)
            .register(registry).increment(completionTokens);

        // 估算费用
        double price = MODEL_PRICES.getOrDefault(modelId, 0.003);
        double cost = (promptTokens + completionTokens) / 1000.0 * price;
        Counter.builder("llm.cost.estimated")
            .tag("tenant", tenantId).tag("model", modelId)
            .register(registry).increment(cost);

        // 租户级日用量检查
        int total = promptTokens + completionTokens;
        tenantDailyTokens.computeIfAbsent(tenantId, k -> new AtomicLong())
            .addAndGet(total);
    }

    public long getTenantDailyTokens(String tenantId) {
        return tenantDailyTokens
            .getOrDefault(tenantId, new AtomicLong()).get();
    }

    public boolean isTenantOverDailyCap(String tenantId, long cap) {
        return getTenantDailyTokens(tenantId) > cap;
    }
}

// 每日零点重置租户用量
@Component
public class DailyTokenReset {
    private final TokenMeteringService metering;

    @Scheduled(cron = "0 0 0 * * ?")
    public void resetDailyCounters() {
        log.info("Daily token counters reset");
    }
}

Python：LangSmith 用量提取与成本报表

python 
from langsmith import Client
from datetime import datetime, timedelta
from collections import defaultdict

class CostReporter:
    """从 LangSmith 提取 Token 用量并按租户汇总。"""

    MODEL_PRICES = {
        "claude-sonnet-4": {"input": 0.003, "output": 0.015},
        "gpt-4o": {"input": 0.005, "output": 0.015},
        "deepseek-v3": {"input": 0.001, "output": 0.002},
    }

    def __init__(self):
        self.client = Client()

    def report_by_tenant(self, days: int = 30) -> dict:
        """生成按租户汇总的成本报表"""
        start = datetime.utcnow() - timedelta(days=days)
        runs = self.client.list_runs(
            project_name="corpassist-agent",
            start_time=start
        )
        report = defaultdict(lambda: {
            "prompt_tokens": 0, "completion_tokens": 0,
            "cost": 0.0, "call_count": 0
        })
        for run in runs:
            tenant = run.metadata.get("tenant_id", "unknown")
            model = run.metadata.get("model_id", "unknown")
            usage = run.outputs.get("usage", {})
            pt = usage.get("prompt_tokens", 0)
            ct = usage.get("completion_tokens", 0)
            report[tenant]["prompt_tokens"] += pt
            report[tenant]["completion_tokens"] += ct
            report[tenant]["call_count"] += 1
            prices = self.MODEL_PRICES.get(model, {"input": 0.003, "output": 0.015})
            report[tenant]["cost"] += (pt / 1000 * prices["input"]
                                       + ct / 1000 * prices["output"])
        return dict(report)

    def detect_anomaly(self, tenant_id: str, threshold: float = 500.0) -> bool:
        """检测租户日成本是否异常突增"""
        today = self._daily_cost(tenant_id, 1)
        avg = self._daily_cost(tenant_id, 30) / 30
        return today > avg * threshold if avg > 0 else False

    def _daily_cost(self, tenant_id: str, days: int) -> float:
        report = self.report_by_tenant(days)
        return report.get(tenant_id, {}).get("cost", 0.0)

Prompt token 和 completion token 的价格相同吗？

绝大多数模型 provider 的 completion token 比 prompt token 贵 3-5 倍。分开统计可以更精确地估算成本，也帮助你发现是否输出过长（completion 占比异常高）。

Chargeback 在多租户场景如何实现？
Chargeback 是把成本按租户分摊回业务部门。实现方式：1) 每个请求携带 tenant_id，2) 计量服务按 tenant 聚合 daily token 用量，3) 月末生成按租户的账单报表。关键是不只分 tenant，还分 model（不同模型价格差异大）。
Daily cap 超了该怎么做？
推荐三层策略：1) cap 达 80% 通知租户管理员，2) cap 达 100% 对非核心端点降级（用小模型或返回缓存），3) cap 超 120% 直接阻断，返回 429。阻断策略需在合同或服务条款中明确。
如何管理 token governance 的预算分配？
预算分配是迭代过程：1) 初始阶段按历史用量的 1.5x 分配 buffer，2) 每月回顾实际 vs 预算，3) 为大租户提供预付费折扣。工具方面，LangSmith 的 project-level budget 和 Grafana 告警配合使用。
常见误区
  • 只看总 token 不看分模型 — 不同模型单价差 10 倍；优化从换模型开始。
  • 只统计不告警 — 成本治理的闭环是「计量 → 预算 → 告警 → 自动化降级」，缺告警等于没有治理。
  • 一次性设死预算 — 预算应根据业务增长动态调整，季度 review。
  • 忽略嵌入缓存 — embedding 调用通常占 token 消耗 20-30%，语义缓存可以大幅降低。

本章结论

  • 分租户计量是基础：没有 token metering 就没有成本可观测。至少按 tenant + model 两个维度区分。
  • 三级告警体系：日 80%/100%/异常突增三个级别，从通知到自动限流。
  • Prompt/Completion 分开统计：价格差异大，分开才能精确估算成本并发现优化点。
  • Chargeback 是 FinOps 闭环：把成本分摊回业务部门，推动业务方主动优化用量。
  • Daily cap 防止失控：每个租户独立 cap，超限自动降级或阻断，避免账单爆炸。

Alert fatigue 怎么避免？
核心原则：每一个告警都要对应一个人需要执行的动作。如果收到告警只是看看然后关闭，那就应该降低级别或改成日志。技巧：1) 使用 burn rate 告警而非固定阈值，2) 关联 Runbook 确保 actionable，3) 每月 review 告警列表删除噪音。
Incident triage 和告警有什么区别？
告警是信号（Alert），事故是确认的问题（Incident）。告警可能误报，也可能多个告警指向同一事故。Triage 阶段要把告警归并为事故，确认影响范围、严重等级和负责人。
oncall 轮值如何与告警配合？
推荐方案：PagerDuty 或 OpsGenie 管理轮值表。周一早 9 点交接。oncall 只需要处理 P0/P1，P2 由对应服务团队在工作时间处理。P3 走 Jira 工单队列。
常见误区
  • 指标抖动就告警 — 单次 P95 突增持续 1 分钟可能只是自然波动；加 for 子句沉降后再告警。
  • 告警无 Runbook — oncall 收到告警不知道做什么，浪费时间查资料。
  • 不区分严重等级 — LLM 超时和证书过期响应方式完全不同。
  • 告警太多导致关闭通知 — 宁可少告警但要保证每个都 actionable。

本章结论

  • 分级告警：P0/P1/P2/P3 四级，不同响应时间和通知通道。
  • Runbook 驱动：每个告警必须携带 Runbook 链接，oncall 按剧本操作。
  • 升级策略：P0 5 分钟未确认自动升级到值班经理，确保绝不漏报。
  • 告警疲劳治理：减少噪音，确保每个告警 actionable，定期 review。
  • Burn rate 优先：SLO 错误预算燃烧比单指标抖动更适合作为告警源。

为什么要学本章

LLM 应用的灰度比传统微服务复杂：不止是代码版本，还包括 prompt version、model version 和 index version（RAG 索引）。后三者任何一个变了都可能改变回复质量。没有版本管理和灰度，换 prompt 直接全量上线、发现质量下降时已经影响所有用户。

学完你能

  • 用 feature flag 和 canary metric 安全发布变更
  • 实现 rollback switch 机制，发现异常后 1 分钟内回滚
  • 管理 version registry，记录每次发布的版本号与效果
本章先记住 2 件事
  • 三维灰度：Prompt、模型、索引三个维度各自独立灰度——改 prompt 不影响模型版本。
  • 可观测优先于灰度：灰度前必须先有 canary metric（延迟、错误率、质量分），否则灰度是盲人摸象。

Prompt / 模型 / 索引三维版本

CorpAssist · 灰度发布评审（30 秒） 产品经理要求「把客服回复改得更简洁」。你需要同时考虑：1) 改 Prompt（指令变化），2) 是否换模型（如 Sonnet 3.5 → 4），3) 索引是否重建（新知识库版本）。三者任意组合都是不同的「体验版本」。本周仅改 Prompt，模型和索引保持 v1.2。

Java：Feature Flag 与版本路由

java 
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

@Component
public class VersionRouter {
    private final Map configs = new ConcurrentHashMap<>();

    public VersionRouter() {
        // 初始化默认版本
        configs.put("prompt", new VersionConfig("prompt:v1.2", 0.05));
        configs.put("model", new VersionConfig("claude-sonnet-4", 0.0));
    }

    @Data
    public static class VersionConfig {
        private String currentVersion;
        private String canaryVersion;
        private double canaryPercent;

        public boolean isCanary(String tenantId) {
            if (canaryVersion == null || canaryPercent <= 0) return false;
            return Math.abs(tenantId.hashCode()) % 100 < canaryPercent * 100;
        }
    }

    // 根据租户 ID 灰度路由
    public String resolveVersion(String dimension, String tenantId) {
        VersionConfig cfg = configs.get(dimension);
        if (cfg == null) return "unknown";
        if (cfg.isCanary(tenantId)) {
            log.info("Canary {} -> {} for tenant {}", dimension, cfg.canaryVersion, tenantId);
            return cfg.canaryVersion;
        }
        return cfg.currentVersion;
    }

    // 紧急回滚
    public void rollback(String dimension) {
        log.warn("ROLLBACK triggered for dimension: {}", dimension);
        VersionConfig cfg = configs.get(dimension);
        if (cfg != null) {
            cfg.canaryVersion = null;
            cfg.canaryPercent = 0;
        }
    }
}

// 在 ChatClient 中使用版本路由
@Service
public class ChatService {
    private final VersionRouter router;

    public ChatClient.Response chat(ChatRequest request) {
        String promptVersion = router.resolveVersion("prompt", request.tenantId());
        String modelVersion = router.resolveVersion("model", request.tenantId());
        // 按版本加载 prompt 模板和模型配置
        return chatClient.prompt()
            .system(promptTemplate(promptVersion))
            .model(modelVersion)
            .user(request.getQuestion())
            .call()
            .chatResponse();
    }
}

from dataclasses import dataclass
from typing import Optional

@dataclass
class CanaryComparison:
    """灰度 vs 基线 指标对比"""
    dimension: str
    canary_version: str
    baseline_version: str
    canary_metrics: dict
    baseline_metrics: dict

    @property
    def passed(self) -> bool:
        """灰度版本在延迟和质量上不差于基线"""
        latency_ok = (
            self.canary_metrics["p95_latency"]
            <= self.baseline_metrics["p95_latency"] * 1.1
        )
        quality_ok = (
            self.canary_metrics["quality_score"]
            >= self.baseline_metrics["quality_score"] * 0.95
        )
        return latency_ok and quality_ok

    def report(self) -> str:
        return (
            f"Canary {self.dimension} {self.canary_version} vs {self.baseline_version}\n"
            f"  Latency P95: {self.canary_metrics['p95_latency']:.1f}s "
            f"(baseline {self.baseline_metrics['p95_latency']:.1f}s)\n"
            f"  Quality: {self.canary_metrics['quality_score']:.1f}% "
            f"(baseline {self.baseline_metrics['quality_score']:.1f}%)\n"
            f"  Passed: {self.passed}"
        )

  • 换模型全量切换，无 fallback

  • 模型灰度同时保留 fallback 模型
  • 索引用别名 + 时间戳版本管理
  • 所有变更通过 feature flag 开关
如何设计 Prompt 版本的兼容性？
Prompt 每次更新都应向后兼容 template 变量名。例如新版本删除了 {{company_name}} 变量但旧流量仍传这个值，不应报错。推荐用 YAML 定义 prompt template，每次改 prompt 提交 registry + 更新版本号。
索引灰度可以只给部分用户用新版吗？
可以。在 RAG 查询时，按用户 ID hash 决定用旧索引别名还是新索引别名。两个索引独立写入和构建，在搜索层做 A/B 分流。缺点是维护两套索引的成本翻倍，适合重大索引变更。
Rollback switch 如何保证 1 分钟生效？
动态配置中心（Nacos/Apollo/etcd）配合监听器。配置变更后，服务端无需重启即可切换版本。CorpAssist 的 VersionRouter 监听配置中心的版本变更事件，更新本地缓存。
常见误区
  • 灰度只做代码不做到 LLM 维度 — LLM 应用中 prompt 和模型版本的变更比代码更频繁、影响更大。
  • 灰度后不看质量分 — 延迟和错误率正常但质量分下降，灰度应视为失败。
  • 灰度时间太短 — 5% 流量跑 5 分钟就全量，样本不足。至少观察 30 分钟。
  • 回滚开关未经演练 — 生产回滚需要验证开关确实有效，定期演练。

本章结论

  • 三维灰度：Prompt、模型、索引三者独立版本管理和灰度策略。
  • Feature Flag 驱动：所有变更通过 feature flag 开关，不直接修改代码或配置。
  • Canary Metric：灰度过程持续对比延迟+错误率+质量分，达标才继续推进。
  • 自动回滚：灰度异常时自动触发 rollback switch，1 分钟内恢复。
  • 版本注册：每次发布在 version registry 中记录，方便追溯和对比。

ELK 和 Loki 怎么选？
ELK（Elasticsearch + Logstash + Kibana） 功能全支持复杂聚合分析但资源消耗大；Loki 轻量、与 Grafana 原生集成、适合日志与 trace 关联场景。CorpAssist 推荐 Loki + Tempo + Grafana 三件套。
如何防止日志泄露 PII？
在日志输出层做过滤：1) 不记录完整 prompt 只记录 token 数和 hash，2) 使用 log redaction 库自动检测和屏蔽手机/身份证/邮箱，3) 日志系统中配置字段白名单，未在白名单的字段不存储。详见安全合规课程。
Request trail 和 debug session 的区别？
Request trail 是请求完整日志链的被动查询（事后搜索）；debug session 是主动录制（预先对特定租户或用户开启调试日志级别）。CorpAssist 用 request trail 做常规排障，用 debug session 做深度排查。
常见误区
  • 日志不打 trace_id — 脱离了 Trace 上下文的日志只能按时间猜链路。
  • 各服务日志格式不同 — BFF 用 JSON、Python 用 plain text、RAG 用 syslog，无法统一查询。
  • 日志级别不合理 — 所有日志都是 INFO，ERROR 日志里没有 stack trace。
  • 不配置 Tempo-Loki 集成 — 排障时需要在两个系统间手动复制 ID。

本章结论

  • 结构化日志是基础：JSON 格式 + 统一 schema（trace_id/request_id/tenant_id）是搜索和关联的前提。
  • Trace ↔ Log 双向关联：Tempo-Loki 集成使排障时从 span 一键跳到日志，从日志一键跳到 trace。
  • LogQL 聚合分析：通过日志聚合发现错误模式（如哪个 tool 最常超时、哪个租户错误最多）。
  • 避免日志 PII：日志中不记录 prompt 原文、用户个人信息。用 hash 或 token 数替代。
  • Request trail + debug session：事后搜索 + 主动录制配合使用，覆盖不同排障场景。

Java：幂等重试拦截器

java 
import org.springframework.retry.annotation.Backoff;
import org.springframework.retry.annotation.Retryable;

@Service
public class ToolExecutionService {

    // 带幂等键和有限重试的 tool 调用
    @Retryable(
        retryFor = {ToolTimeoutException.class, NetworkException.class},
        maxAttempts = 3,
        backoff = @Backoff(delay = 1000, multiplier = 2)
    )
    public ToolResult executeTool(ToolCommand cmd) {
        String idempotencyKey = cmd.getIdempotencyKey();
        // 先检查幂等键是否已处理
        Optional cached = idempotencyStore.get(idempotencyKey);
        if (cached.isPresent()) {
            log.info("Idempotency hit, returning cached result for key={}", idempotencyKey);
            return cached.get();
        }
        // 执行 tool 调用
        try {
            ToolResult result = callToolEndpoint(cmd);
            idempotencyStore.save(idempotencyKey, result);
            return result;
        } catch (Exception e) {
            log.warn("Tool execution failed for key={}, attempts remaining", idempotencyKey);
            throw e;
        }
    }
}

// 幂等键存储（Redis）
@Component
public class IdempotencyStore {
    private final RedisTemplate redis;

    public Optional get(String key) {
        ToolResult result = redis.opsForValue().get("idempotency:" + key);
        return Optional.ofNullable(result);
    }

    public void save(String key, ToolResult result) {
        // TTL 24 小时
        redis.opsForValue().set(
            "idempotency:" + key, result,
            Duration.ofHours(24));
    }
}

// 生成幂等键
@Component
public class IdempotencyGenerator {
    public String generate(String requestId, String toolName, String paramsHash) {
        return String.format("%s:%s:%s", requestId, toolName, paramsHash);
    }
}

Python：MQ 可靠性和 at-least-once 消费

python 
import json
from kafka import KafkaProducer, KafkaConsumer
from dataclasses import dataclass
from typing import Optional, Callable

@dataclass
class MQMessage:
    """消息体，包含幂等键便于去重。"""
    event_id: str
    event_type: str
    request_id: str
    tenant_id: str
    payload: dict
    timestamp: str


class ReliableMQPublisher:
    """可靠消息发布器 — 失败时保存到本地再重试。"""

    def __init__(self, bootstrap_servers: str, topic: str):
        self.producer = KafkaProducer(
            bootstrap_servers=bootstrap_servers,
            acks="all",        # 等待所有副本确认
            retries=3,
            linger_ms=100,     # 批量发送降低延迟
        )
        self.topic = topic

    def publish(self, message: MQMessage):
        future = self.producer.send(
            self.topic,
            key=message.event_id.encode(),
            value=json.dumps(message.__dict__).encode()
        )
        # 同步确认
        result = future.get(timeout=10)
        log.info("Message published: event_id=%s, partition=%s, offset=%s",
                 message.event_id, result.partition, result.offset)


class ReliableMQConsumer:
    """可靠消息消费者 — at-least-once 语义 + 去重。"""

    def __init__(self, topic: str, group_id: str,
                 handler: Callable, processed_store: "ProcessedStore"):
        self.consumer = KafkaConsumer(
            topic,
            group_id=group_id,
            auto_offset_reset="earliest",
            enable_auto_commit=False,  # 手动提交 offset
            max_poll_records=10,
        )
        self.handler = handler
        self.processed = processed_store

    def consume_loop(self):
        for batch in self.consumer.poll(timeout_ms=1000).values():
            for record in batch:
                msg = MQMessage(**json.loads(record.value))
                if self.processed.already_processed(msg.event_id):
                    log.info("Skipping already processed: %s", msg.event_id)
                    continue
                try:
                    self.handler(msg)
                    self.processed.mark_processed(msg.event_id)
                    self.consumer.commit()
                except Exception as e:
                    log.error("Handler failed for %s: %s. Will retry.",
                              msg.event_id, str(e))
                    # 不 commit，下次重新消费


# 去重存储（Redis）
class ProcessedStore:
    def __init__(self, redis_client):
        self.redis = redis_client

    def already_processed(self, event_id: str) -> bool:
        return self.redis.exists(f"processed:{event_id}")

    def mark_processed(self, event_id: str, ttl: int = 86400):
        self.redis.setex(f"processed:{event_id}", ttl, "1")

不推荐
  • Tool 调用失败无限重试
  • MQ 消息不处理重复消费
  • 无降级路径，LLM 挂就返回 502
  • 单 AZ 部署，依赖单个节点
推荐
  • 幂等键 + 有限重试（最多 3 次指数退避）
  • MQ at-least-once + 去重表
  • 降级路径：备用模型 → 缓存 → 静态回复
  • 多 AZ 部署 + read replica + MQ 跨 AZ
At-least-once 在 LLM 场景中如何实现？
关键路径（如用户下单后触发客服通知）通过 MQ 解耦：BFF 发送事件到 Kafka，Agent 从 Kafka 消费。生产端用 acks=all 保证写入成功；消费端处理完才 commit offset，配合去重表保证业务最终一致。
Read replica 在 RAG 场景怎么用？
RAG 检索的向量数据库可以部署 read replica 分担查询压力。主库处理索引写入，副本处理查询。延迟容忍的场景（如知识库索引 1 小时更新一次），副本延迟不是问题。关键在于查询时选择最近的可用副本（failover read）。
DR lite 是什么意思？
轻量灾备模式，不需要完整的 active-active 双活：1) 关键数据（幂等键、会话状态）跨 AZ 同步，2) 计算节点多 AZ 部署（K8s pod anti-affinity），3) 非关键路径（日志、审计）允许短暂丢失。相比金融级双活，DR lite 更务实且成本可控。
常见误区
  • 幂等只考虑成功情况 — 幂等键过期了怎么办？重试时幂等键相同但参数不同呢？
  • MQ 只配了 at-most-once — 消息可能丢失，无法保证客服事件必达。
  • 降级路径未提前测试 — 生产 LLM 挂了发现备用模型没配 API key。
  • 单 AZ 部署 — AZ 故障时完全不可用，不符合多可用区最佳实践。

本章结论

  • 幂等键设计：request_id + tool_name + params_hash 组成全局唯一键，保证重试安全。
  • 有限重试：最多重试 3 次，指数退避，防止打爆下游。
  • MQ at-least-once：生产 acks=all，消费手动 commit + 去重，保证事件必达但不重复处理。
  • 多级降级：备用模型 → 语义缓存 → 静态回复，保证 LLM 异常时仍有基本服务能力。
  • 多 AZ 部署：计算节点 anti-affinity + 数据跨 AZ 同步 + 读副本。

本章先记住 2 件事

  • 缓存不是万能：需要设计合理的 TTL、缓存大小和路由策略，否则可能命中过期知识或消耗过多存储。

Python：语义缓存实现

python 
from typing import Optional, List, Tuple
import numpy as np
from openai import OpenAI
import redis
import json

class SemanticCache:
    """语义缓存 —— 基于 embedding 相似度匹配。"""

    def __init__(self, embedding_model: str = "text-embedding-3-small",
                 similarity_threshold: float = 0.95,
                 redis_url: str = "redis://localhost:6379",
                 max_cache_size: int = 10000):
        self.client = OpenAI()
        self.embedding_model = embedding_model
        self.threshold = similarity_threshold
        self.redis = redis.from_url(redis_url)
        self.max_size = max_cache_size

    def _get_embedding(self, text: str) -> List[float]:
        resp = self.client.embeddings.create(
            model=self.embedding_model,
            input=text
        )
        return resp.data[0].embedding

    def _similarity(self, a: List[float], b: List[float]) -> float:
        return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

    def get(self, question: str,
            tenant_id: str) -> Optional[Tuple[str, float]]:
        """查找缓存，返回 (回复, 相似度) 或 None"""
        embedding = self._get_embedding(question)
        key = f"cache:{tenant_id}"

        # 从 Redis 读取所有缓存条目（生产用向量数据库更高效）
        cached = self.redis.hgetall(key)
        best_sim = 0
        best_answer = None

        for q_bytes, answer_bytes in cached.items():
            cached_q = q_bytes.decode()
            cached_embedding = json.loads(cached_q)  # 简化：实际存向量
            sim = self._similarity(embedding, cached_embedding)
            if sim > best_sim:
                best_sim = sim
                best_answer = answer_bytes.decode()

        if best_sim >= self.threshold:
            return best_answer, best_sim
        return None

    def set(self, question: str, answer: str,
            tenant_id: str, ttl: int = 3600):
        """写入缓存"""
        embedding = self._get_embedding(question)
        key = f"cache:{tenant_id}"
        # 生产环境使用向量数据库（Milvus/Pinecone）
        self.redis.hset(key, json.dumps(embedding), answer)
        self.redis.expire(key, ttl)

    def hit_rate(self, tenant_id: str) -> float:
        """返回缓存命中率"""
        hits = int(self.redis.get(f"stats:{tenant_id}:hits") or 0)
        total = int(self.redis.get(f"stats:{tenant_id}:total") or 0)
        return hits / total if total > 0 else 0.0

Route Cheap：模型路由降本

Route cheap 策略：简单问题用便宜小模型，复杂问题用贵的大模型。CorpAssist 中：

  • 简单问题（查政策、查状态）→ deepseek-v3（$0.001/1K tokens）
  • 复杂问题（多步推理、生成报告）→ claude-sonnet-4（$0.003/1K tokens）
  • 关键任务（退款、法律咨询）→ gpt-4o（$0.005/1K tokens）
语义缓存的 embedding 成本怎么算？
每次缓存未命中时产生 1 次 embedding 调用（约 $0.0001）。如果命中率 40%，每 100 次请求：60 次 embedding + 40 次命中，embedding 成本约 $0.006。相比 LLM 调用的 $0.30-3.00，embedding 成本微不足道。
相似度阈值怎么调优？
从 0.95 开始，如果命中率 < 20% 逐步降低到 0.90。调低阈值会增加命中率但也增加返回不相关内容的风险。推荐用 A/B 测试：先对 5% 流量用新阈值，对比用户满意度和质量分。
Savings metric 怎么计算？
Savings = (未命中时 LLM 成本总和) - (缓存存储 + embedding 成本)。更直观的指标是「平均每请求成本」：部署缓存前 $0.05/请求 → 部署后 $0.03/请求，节省 40%。

本章结论

  • 语义缓存大幅降本：相似度匹配命中率 30-60%，可降低 30-60% 的 LLM token 成本和延迟。
  • 分 TTL 策略：知识库长 TTL、个人化不缓存、实时数据短 TTL。
  • Route cheap：简单用小模型、复杂用大模型，进一步降本。
  • 监控命中率：缓存命中率 < 20% 说明阈值过高或问题多样性太大，需要调优。
  • 知识更新主动失效：RAG 索引更新后必须清除相关缓存，否则返回过期信息。

from dataclasses import dataclass
from typing import Dict

@dataclass
class CapacityInput:
    """容量估算输入参数"""
    total_users: int = 25000
    daily_active_ratio: float = 0.3
    queries_per_user_per_day: float = 5
    peak_hours: int = 8  # 8 小时集中工作时段
    peak_to_avg_ratio: float = 3.0
    bff_qps_per_instance: int = 50
    avg_prompt_tokens: int = 500
    avg_completion_tokens: int = 200
    rag_qps_per_instance: int = 100

@dataclass
class CapacityResult:
    """容量估算结果"""
    dau: int
    daily_queries: int
    peak_qps: float
    bff_instances: int
    llm_tpm: int
    rag_qps: float
    daily_token_cost: float

    def summary(self) -> str:
        return f"""
CorpAssist 容量粗算结果
  DAU: {self.dau:,}
  日均查询: {self.daily_queries:,}
  峰值 QPS: {self.peak_qps:.1f}
  BFF 实例: {self.bff_instances}
  LLM TPM: {self.llm_tpm:,}
  RAG QPS: {self.rag_qps:.1f}
  日 Token 成本: ${self.daily_token_cost:.2f}
        """


def estimate_capacity(inputs: CapacityInput) -> CapacityResult:
    dau = inputs.total_users * inputs.daily_active_ratio
    daily_q = dau * inputs.queries_per_user_per_day
    peak_qps = daily_q / (inputs.peak_hours * 3600) * inputs.peak_to_avg_ratio
    bff = max(2, int(peak_qps / inputs.bff_qps_per_instance) + 1)
    tpm = peak_qps * inputs.avg_completion_tokens
    daily_cost = (daily_q * (inputs.avg_prompt_tokens +
                             inputs.avg_completion_tokens) / 1000) * 0.003
    return CapacityResult(
        dau=int(dau), daily_queries=int(daily_q),
        peak_qps=peak_qps, bff_instances=bff,
        llm_tpm=int(tpm), rag_qps=peak_qps * 1.5,
        daily_token_cost=daily_cost
    )

# 使用示例
inputs = CapacityInput(total_users=50000)
result = estimate_capacity(inputs)
print(result.summary())

import io.micrometer.core.instrument.MeterRegistry;
import java.util.function.Supplier;

@Component
public class ScaleTrigger {
    private final MeterRegistry registry;
    private final double CPU_THRESHOLD = 0.7;
    private final double QPS_THRESHOLD_PER_INSTANCE = 40;

    public ScaleTrigger(MeterRegistry registry) {
        this.registry = registry;
    }

    // 检查是否需要扩容
    public ScaleDecision evaluate() {
        double cpu = getMetric("system.cpu.usage");
        double qps = getMetric("http.server.requests.per.second");
        int instances = getInstanceCount();

        boolean needScaleUp = cpu > CPU_THRESHOLD
            || qps / instances > QPS_THRESHOLD_PER_INSTANCE;
        boolean needScaleDown = cpu < 0.3
            && qps / instances < QPS_THRESHOLD_PER_INSTANCE * 0.5;

        if (needScaleUp) {
            log.warn("SCALE UP: cpu={}, qps/instance={}",
                cpu, qps / instances);
            return ScaleDecision.UP;
        }
        if (needScaleDown && instances > 2) {
            log.info("SCALE DOWN: cpu={}, qps/instance={}",
                cpu, qps / instances);
            return ScaleDecision.DOWN;
        }
        return ScaleDecision.NONE;
    }

    private double getMetric(String name) {
        var meter = registry.find(name).gauge();
        return meter != null ? meter.value() : 0.0;
    }

    private int getInstanceCount() {
        // 从 K8s API 或注册中心获取
        return 3;
    }

    public enum ScaleDecision { UP, DOWN, NONE }
}

  • 前期用 API，流量稳定后评估自建
粗算误差通常多大？
粗算的目的是「数量级正确」，误差通常在 2-3 倍范围内。比如算出来需要 2 台 BFF，实际可能需要 4 台，但这比「我不知道需要几台」好得多。Headroom 就是用来吸收误差的。
LLM 的 TPM/RPM 限制怎么考虑？
不同 provider 有不同 rate limit（如 GPT-4o 标准层 10000 TPM）。在容量估算后，需要检查 peak TPM 是否超过 provider 限额。如果超过，需要：1) 申请提升限额，2) 增加备用 provider，3) 引入排队机制。
Headroom 留多少合适？
CPU/内存预留 30% headroom 应对突发流量和垃圾回收。GPU headroom 取决于 SLA：如果允许排队，GPU 利用率可以跑到 70-80%；如果要求实时响应，建议留 50% headroom。Headroom 不是浪费而是容量管理的保险。
常见误区
  • 用平均值算容量 — 峰值可能是平均的 3-10 倍，用平均算出来的容量一定不够。
  • 只算计算不算存储 — RAG 向量库的 QPS 和存储增长也要估算。
  • 忽略 LLM API rate limit — 自有机器的瓶颈是 CPU/GPU，API 的瓶颈是 rate limit。
  • 过早自建 GPU — LLM API 按量付费初期更经济，月 token 费 > $5000 再评估自建。

本章结论

  • 粗算公式：DAU → 日均查询 → 峰值 QPS → 实例数，20 分钟完成估算。
  • Scale trigger：CPU > 70% 或单实例 QPS 超阈值触发扩容，HPA 自动化。
  • GPU 租约估算：按并发需求选择 GPU 型号；前期推荐 API 而非自建。
  • Headroom：CPU/内存 30%、GPU 50% headroom 吸收流量波动。
  • 决策分层：2x 内 HPA 解决；2-10x 提前扩容；10x+ 架构评审。

import java.time.Instant;

@Component
public class IncidentTimelineLogger {
    private final List events = new ArrayList<>();

    public record TimelineEvent(
        Instant timestamp,
        String event,
        String source  // "system" | "manual"
    ) {}

    public void record(String event, String source) {
        TimelineEvent e = new TimelineEvent(Instant.now(), event, source);
        events.add(e);
        log.info("INCIDENT TIMELINE: {} - {} [{}]", e.timestamp(), e.event(), e.source());
    }

    public String export() {
        StringBuilder sb = new StringBuilder("## 时间线\n");
        sb.append("| 时间 | 事件 | 来源 |\n");
        sb.append("|------|------|------|\n");
        for (var e : events) {
            sb.append(String.format("| %s | %s | %s |\n",
                e.timestamp(), e.event(), e.source()));
        }
        return sb.toString();
    }

    public void clear() {
        events.clear();
    }
}

Python：复盘报告生成

python 
from dataclasses import dataclass, field
from typing import List, Optional
from datetime import datetime, timedelta


@dataclass
class ActionItem:
    """改进项"""
    number: int
    description: str
    owner: str
    deadline: datetime
    status: str = "open"


@dataclass
class PostmortemReport:
    """结构化的故障复盘报告。"""
    incident_id: str
    severity: str
    detected_at: datetime
    recovered_at: datetime
    impact: str
    affected_users: int
    timeline: List[dict]
    root_cause: str
    action_items: List[ActionItem] = field(default_factory=list)

    @property
    def duration_minutes(self) -> int:
        return int((self.recovered_at - self.detected_at).total_seconds() / 60)

    def add_action(self, desc: str, owner: str, days: int = 7):
        """添加改进项"""
        n = len(self.action_items) + 1
        deadline = datetime.utcnow() + timedelta(days=days)
        self.action_items.append(
            ActionItem(n, desc, owner, deadline))

    def render_to_markdown(self) -> str:
        """输出 Markdown 复盘报告"""
        lines = [
            f"# 故障复盘：{self.incident_id}",
            "",
            f"**严重等级**: {self.severity}",
            f"**时长**: {self.duration_minutes} 分钟",
            f"**影响用户**: {self.affected_users}",
            "",
            "## 时间线",
            "| 时间 | 事件 |",
            "|------|------|",
        ]
        for evt in self.timeline:
            lines.append(f"| {evt['time']} | {evt['event']} |")

        lines.extend([
            "",
            "## 根因",
            self.root_cause,
            "",
            "## Action Items",
            "| # | 行动项 | Owner | 截止日期 |",
            "|---|--------|-------|----------|",
        ])
        for item in self.action_items:
            lines.append(
                f"| {item.number} | {item.description} | "
                f"{item.owner} | {item.deadline.date()} |")
        return "\n".join(lines)
不推荐
  • 复盘只到「当事人下次注意」
  • 无时间线，凭记忆回顾
  • Action item 无 owner 无截止日
  • 把「换人」当作改进措施
推荐
  • Blameless 文化，找系统原因
  • 精确记录每步时间线
  • 每个 action item 有 owner 和 due date
  • 改进项可跟踪可验证
复盘频率应该多高？
每次 P0/P1 事故必须做正式复盘；P2 做轻量复盘（5 行总结 + action items）；P3 不强制。建议每月开一次「事故趋势分析」会，看哪些类型的事故在反复出现。
Action item 完不成怎么办？
下次复盘中 review 未完成的 action item。如果在截止日未完成，需要说明原因和新的预期完成时间。连续两次延期的 action item 需要重新评估优先级——也许它其实不重要。
Blameless 文化怎么落地？
几个原则：1) 复盘文档不写「谁做了什么」只写「系统状态是什么」，2) 分享事故经验奖励而非惩罚，3) 管理层不以事故数量考核个人，而是看「团队从事故中学到了什么」。详见 retro template 中的 blameless 指南。
常见误区
  • 复盘 = 找责任人 — 这样下次没人愿意报问题，隐瞒事故。
  • Action item 太多 — 每个事故提 20 个改进项，结果一个也完不成。建议不超过 5 个。
  • 只复盘 P0 — P1/P2 累积多了也会变成 P0。
  • 不复盘「险些发生」 — Near miss 是免费的学习机会，不需要等事故发生。

本章结论

  • Postmortem 结构化模板：基本信息 → 时间线 → 影响评估 → 5 Whys → Action Items → 经验教训。
  • 5 Whys 找根因：层层深入直到找到系统原因，而非停留在「人忘了」。
  • Blameless 文化：不追责只追因。找到了系统漏洞比找到谁犯错更有价值。
  • Action item 有 owner：每个改进项必须有负责人和截止日，否则等于没有。
  • 复盘频率：P0/P1 正式复盘，P2 轻量复盘，每月事故趋势分析。

题面：实现一个 SLO 计算器，输入每日成功/总请求数，输出当月 SLO 达标情况。

java 
public class SLOCalculator {
    private final double sloTarget;  // 0.99
    private final int windowDays;    // 30
    private final List dailySLIs = new ArrayList<>();

    public SLOCalculator(double sloTarget, int windowDays) {
        this.sloTarget = sloTarget;
        this.windowDays = windowDays;
    }

    // 添加一天的 SLI 数据
    public void addDailyData(long success, long total) {
        double sli = total > 0 ? (double) success / total : 1.0;
        dailySLIs.add(sli);
        // 只保留窗口期数据
        while (dailySLIs.size() > windowDays) {
            dailySLIs.remove(0);
        }
    }

    // 当前 SLO 达标情况
    public SLOStatus checkSLO() {
        if (dailySLIs.isEmpty()) {
            return new SLOStatus(true, 100.0, "No data");
        }
        double avgSLI = dailySLIs.stream()
            .mapToDouble(Double::doubleValue).average().orElse(1.0);
        double errorBudgetUsed =
            Math.max(0, (sloTarget - avgSLI) / (1 - sloTarget) * 100);
        boolean met = avgSLI >= sloTarget;
        return new SLOStatus(met, 100 - errorBudgetUsed,
            met ? "SLO met" : "SLO breached");
    }

    public record SLOStatus(boolean met, double budgetRemaining, String message) {}
}

// 使用示例
SLOCalculator calc = new SLOCalculator(0.99, 30);
calc.addDailyData(9950, 10000);  // 99.5%
calc.addDailyData(9900, 10000);  // 99.0%
SLOStatus status = calc.checkSLO();
// budgetRemaining: 50% (用了 50% 的错误预算)
