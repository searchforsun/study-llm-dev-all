路由

CorpAssist · 员工问「2026 病假扣薪规则」（45 秒） 用户属于华东区销售部。系统有 kb-hr-policy、kb-finance-payroll、kb-general-handbook 三个 Milvus collection。若三路全查，延迟 ×3 且易混入财务敏感 chunk；若只查 general，可能漏掉 HR 最新修订。需要 router kb 在 50ms 内决定查哪些库。

kb federation 的第一层是路由：把用户 query 映射到一个或多个子 KB。常见信号：

  • 显式标签 — 用户在前端选了「人事政策」；或 query 带 dept=hr。
  • Intent 分类 — 小模型 / 规则判断「薪酬 / 制度 / 产品 FAQ」。
  • 元数据索引 — 维护 KB Catalog（库名、owner、doc 类型、ACL 范围、embedding 版本）。
  • Fallback 广播 — 置信度低时并行查 Top-K 库，但须限流与 ACL 预过滤。

先想 10 秒：为何「检索后过滤 ACL」在联邦场景更危险？

广播多库会先拉取无权 chunk 到应用内存，日志与缓存都可能泄露；且浪费向量检索配额。应在路由 + 每库 filter 双层拦截（见 权限与多租户）。

合并

各子 KB 独立返回 top_k 后，需要 federate search 的合并层。难点在于分数不可比：A 库 cosine 0.82 与 B 库 0.79 不代表 A 更相关。

常见合并算法

方法做法适用
RRF（Reciprocal Rank Fusion） 按排名融合：1/(k+rank) 求和 默认首选；不依赖原始分数尺度
Min-Max 归一化 每库内归一化后再加权 库内分数分布稳定时
学习排序 LTR 用点击日志训练 有足够反馈数据后迭代

RRF 合并实现

python 
def rrf_merge(lists: list[list[Hit]], k: int = 60) -> list[Hit]:
    scores: dict[str, float] = {}
    best: dict[str, Hit] = {}
    for hits in lists:
        for rank, hit in enumerate(hits, start=1):
            cid = hit.chunk_id
            scores[cid] = scores.get(cid, 0.0) + 1.0 / (k + rank)
            best.setdefault(cid, hit)
    ordered = sorted(scores.keys(), key=lambda c: scores[c], reverse=True)
    return [best[c] for c in ordered]

去重

大企业常见「同一份制度 PDF」被 HR 与通用库各采一遍；或 cross kb 同步导致 chunk 重复。dedup federate 要在合并后、送 LLM 前完成。

去重维度

  • doc_id 级 — 同一 source_doc_id 只保留 RRF 分最高的一条。
  • chunk_hash 级 — 正文 MD5 / simhash 相同则合并 metadata（保留多库 provenance）。
  • 语义近邻 — 向量 cosine > 0.95 且标题相同，保留较新 index_version。

去重后 metadata 保留

python 
def dedup_by_doc_id(hits: list[Hit]) -> list[Hit]:
    seen: dict[str, Hit] = {}
    for h in hits:
        key = h.meta["source_doc_id"]
        if key not in seen or h.rrf_score > seen[key].rrf_score:
            seen[key] = h
    return list(seen.values())
不推荐
  • 只按 chunk 文本 == 精确匹配
  • 去重丢弃所有 provenance，无法回溯来源库
  • ingestion 不去重，全靠检索层兜底
推荐
  • 采集层 source_doc_id 全局唯一（见 分块与元数据规范）
  • dedup 后 provenance: [kb-hr, kb-general]
  • 运营台展示「重复入库」告警（见 知识质量运营）
常见误区
  • 联邦 = 物理合库 — 应保持库边界与 ACL 隔离，只在检索层联邦。
  • 合并后不再过滤 ACL — 每库检索必须带 filter；合并层不再「补过滤」。
  • 忽略 embedding 版本 — 不同库模型不一致时，RRF 仍可用，但勿跨版本做语义 dedup 阈值。

本章结论

  • 路由：KB Catalog + intent/rules 决定查哪些库；ACL 在路由与检索双层生效，避免广播泄露。
  • 合并：RRF 等 rank 融合解决分数不可比；保留 kb_id 与 index_version 便于观测与审计。
  • 去重：doc_id / chunk_hash / 语义近邻三级；采集层 source_doc_id 是治理根基。

本章先记住 3 件事
  • 留存是默认策略：没有 policy 的 KB 等于无限保留，合规风险最大。
  • 删除是级联作业：只删 Milvus 向量不够，LLM 上下文缓存也可能残留。
  • hold 优先于删除：诉讼保全期间，purge doc 必须被拦截并告警。

留存

CorpAssist · 客服录音转写入库（60 秒） 客服对话含客户手机号，进入 kb-support-transcript。法务规定：工单关闭后保留 3 年用于纠纷举证，3 年后自动销毁；涉诉案件进入 legal hold 则暂停 TTL。运营需要一张表回答「这类 doc 何时该删」。

compliance kb 的留存层要回答三件事：分类、期限、触发器。

  • 分类 — 按 doc_type（合同 / HR / 客服 / 公开制度）映射不同 policy。
  • 期限 — retention days 或「事件 + N 天」（如离职 + 90 天）。
  • 触发器 — 定时扫描 expires_at；或源系统 CDC 标记 deleted。

被遗忘权

用户或员工行使删除权时，系统必须在 SLA 内完成 right to forget。KB 场景比传统 DB 更难：向量不可逆推原文，但仍可能通过检索召回关联 chunk。

删除请求处理流程

  1. 身份核验 — 工单绑定 subject_id（员工工号 / 客户 ID）。
  2. 范围定位 — 按 subject_id、source_doc_id 查全库（含联邦子 KB）。
  3. hold 检查 — 任一 doc 在 legal hold 则部分拒绝并说明法律依据。
  4. 级联删除 — 执行 delete cascade。
  5. 审计落库 — audit delete 记录操作人、范围、条数、checksum（不存 PII 明文）。

级联删除实现要点

python 
def purge_cascade(doc_id: str, operator: str) -> PurgeReport:
    chunks = chunk_repo.list_by_doc(doc_id)
    chunk_ids = [c.id for c in chunks]
    object_store.delete(doc_id)
    chunk_repo.delete_by_doc(doc_id)
    milvus.delete(expr=f'source_doc_id == "{doc_id}"')
    embed_cache.invalidate(chunk_ids)
    audit.log(event="purge_doc", doc_id=doc_id, count=len(chunk_ids), operator=operator)
    return PurgeReport(deleted_chunks=len(chunk_ids))
不推荐
  • 只在 UI 隐藏文档，向量仍可被检索
  • 删除不记审计，无法应对监管问询
  • re-embed /job 队列仍引用已删 doc_id
推荐
  • 删除前暂停该 doc 相关增量 job（见 增量更新与重建）
  • 蓝绿索引双写时两版都删（见 索引版本与回滚）
  • 30 天内提供删除证明报告（chunk 数、库列表、时间戳）

合规 vs 业务冲突决策表

场景优先处理
员工要求删 PII，无 hold 删除权 72h 内 purge_cascade

常见误区
  • 向量删了 = 合规完成 — 对象存储、搜索日志、LLM 会话引用都要纳入范围。
  • 联邦只删一个库 — 同一 doc 可能在多库，需全局 doc_id 索引。
  • 留存与质量运营冲突 — stale 检测标记的 doc 若未到期，不能因「旧」而提前删。

本章结论

  • 留存：按 doc_type 配置 retention_days 与 anchor；scanner + legal_hold 构成自动与人工两道闸。
  • 被遗忘权：核验 → 定位 → hold 检查 → purge_cascade → audit_delete；联邦场景需跨库 doc_id 索引。

  • 分层是治理动作：tier storage 要与留存 policy、访问频率联动，不是一次性搬家。
  • 归档 ≠ 删除：冷库 doc 可脱离向量索引，但 metadata 与恢复路径要保留（合规见 合规保留与删除）。

分层

CorpAssist · 年度 FinOps 复盘（60 秒） Milvus 集群内存占用 78%，S3 向量快照 + 原文 420TB。过去 12 个月检索日志显示：80% query 命中近 90 天文档，2019 年制度 PDF 几乎零命中却占 35% 存储。CTO 要求在不降召回的前提下，把 index cost 降 30%。

tier storage 把 KB 资产按访问频率与业务关键度拆层：

层级数据范围索引形态检索 SLA
Hot 近 90 天 + 高频制度 Milvus 全量向量 + 内存友好索引（HNSW/IVF） < 200ms p95
Warm 90 天～2 年 向量在 Milvus 磁盘型 collection；原文 S3 Standard < 500ms p95
Cold 低命中历史 / 已 supersede 仅 S3 + metadata；无在线向量 按需 re-index，分钟级

NS3 成本分解（示意）

cost ns3 把账单拆成三块，便于和云厂商账单对照：

  • Node — Milvus query node / data node vCPU、RAM（HNSW 内存随向量数线性涨）。
  • Storage — 向量段文件、S3 原文、快照；compress archive 对原文 gzip/zstd。
  • Service — Embedding API 按 token 计费、跨 AZ 流量、Spring 检索网关实例。
python 
# NS3 粗算：向量存储 GB ≈ chunks * dim * 4 bytes * replica
def vector_storage_gb(chunks: int, dim: int = 1024, replica: int = 2) -> float:
    return chunks * dim * 4 * replica / (1024 ** 3)

# 例：5000 万 chunk → 约 390 GB（不含 HNSW 图额外开销 ~1.5x）
budget = {

归档

cold archive 是把文档从在线向量索引「退场」到低成本存储，同时保留可恢复性与合规留存。

归档触发条件

  • 检索命中次数 < 阈值 且 doc_age > N 天
  • 新版本 superseded_by 已发布 30 天
  • FinOps 预算告警触发人工归档批次

归档作业步骤

bash 
# 1. 标记 tier=cold，停止增量 embed
# 2. Milvus 删除向量（保留 collection schema）
# 3. S3 转 Glacier / Intelligent-Tiering + compress
# 4. metadata 保留 restore_uri、content_hash、retention_expires_at
不推荐
  • 归档时删掉 metadata，无法按需 re-index
  • 每次 query 冷数据都同步 re-embed（成本爆炸）
  • 无 embed cache，重建索引重复调用 API
推荐
  • 归档前导出 chunk 快照 + embedding 至 S3（可选，换模型时仍要 re-embed）
  • restore 走审批 + 异步 job，完成后 alias 切回（蓝绿）
  • embed cache key = hash(model_version, chunk_text)

痛点NS3 项治理手段

常见误区
  • 只砍 Storage 不砍 Node — 向量仍在 Milvus 时 Node 内存不会降。
  • 归档违反 legal hold — hold 中的 doc 禁止转 Cold 删除向量。
  • 无 storage budget 仪表盘 — 应每月 review chunks × dim × tier 分布。

本章结论

  • 分层：Hot/Warm/Cold 与检索 SLA 绑定；NS3 三分法对齐 FinOps 语言。
  • 归档：向量退场 + metadata 保留 + 按需 re-index；embed cache 与合规留存并行考虑。

案例 A：acl leak case

现象根因修复预防
销售员工问答出现「管理层薪酬带宽」 检索未带 ACL filter；chunk 有 acl:mgmt-only 但仍被召回 Spring VectorStore 查询加 acl_tags IN user.acl；purge 误入库 batch 入库 schema 必填 acl_tags；集成测试用无权限用户断言 zero hit
错误答法
  • 「让 LLM 不要泄露敏感信息」
  • 「输出后正则脱敏」
正确答法
  • 检索层 filter，chunk 根本不进 context
  • 审计日志记录 filter 表达式与 tenant

案例 B：re embed case

Embedding 从 v2 升到 v3，团队只改了 Python 脚本里的 model 名，未建新 collection、未 re-embed。现象：召回分数整体漂移，评测 faithfulness 从 0.88 跌到 0.71。

  • kb debug 第一步：查 index_version 与 embedding_model 是否一致（索引版本与回滚）。
  • 修复：build hr-policy-v3-emb-v3 → 评测 pass → alias 切换；保留 v2 24h 可回滚。
  • index migrate 话术：「Embedding 空间不可混用；变更等于新索引 + 蓝绿，不是原地改字段。」

回滚触发条件

信号阈值示例动作
Smoke recall@3 < 0.85（相对 blue 降 >10%） 自动 No-Go，保持 blue
空答率 spike 15 分钟内 +5pp alias 切回 blue + 告警
对账 FAIL chunk 数偏差 >0.5% 禁止切换；修复 green

bash

  --alias corpassist_kb_active \
  --target corpassist_kb_blue \
  --reason "smoke_fail_hr_v3.2"

# 2. 通知 Spring 刷新 + 复测 smoke

变更检测：CDC 与 content_hash

# corpassist/ingest/incremental_runner.py
def run_incremental(source: str, index_version: str, since: datetime) -> IngestReport:
    changed, deleted = cdc_sync.diff(source, since)
    for doc_id in deleted:
        vector_store.delete(filter=f'doc_id == "{doc_id}" and index_version == "{index_version}"')
    for doc in changed:
        chunks = split_with_schema(doc, schema=CORPASSIST_SCHEMA)
        vectors = embed_incremental(chunks)  # 仅变更块 re-embed
        upsert(vectors, metadata={"index_version": index_version, "acl_tags": doc.acl})
    return IngestReport(changed=len(changed), deleted=len(deleted))

推荐

def detect_conflicts(index_version: str) -> list[ConflictCase]:
    groups = group_by_doc_key(index_version)
    cases = []
    for key, chunks in groups.items():
        active = [c for c in chunks if c.status == "active"]
        if len(active) > 1 and max_pair_similarity(active) > 0.92:
            cases.append(ConflictCase(doc_key=key, versions=[c.version_tag for c in active]))
    return cases

覆盖率

CorpAssist · 季度复盘 Q1 共 12,400 次 KB 问答；运营抽样 400 条标注「是否由 KB 充分回答」。覆盖率 = 满意数 / 抽样数。若仅 68% 且「报销流程」类 recall gap 集中，则优先补 ingest 而非调 prompt。

coverage rate 衡量「用户问题被 KB 支撑的程度」。CorpAssist 采用分层抽样 + 意图标签，避免被高频简单 FAQ 拉高均值。

指标定义

coverage_rate satisfied_samples / total_samples ≥ 75%

python 
# corpassist/metrics/coverage.py
def daily_coverage(logs: list[RagLog]) -> CoverageReport:
    sampled = stratified_sample(logs, by="intent", n=200)
    satisfied = sum(1 for x in sampled if x.user_feedback == "helpful")
    gap = sum(1 for x in logs if x.max_score < 0.55 and x.hit_count == 0)
    return CoverageReport(
        rate=satisfied / len(sampled),
        recall_gap_rate=gap / len(logs),
    )

def classify_empty(log: RagLog) -> str:
    if log.hit_count_before_filter > 0 and log.hit_count_after_filter == 0:
        return "ACL_FILTER"
    if log.hit_count == 0 and log.max_score < 0.55:
        return "ZERO_HIT"
    return "LLM_REFUSAL"

空答

empty answer 在 CorpAssist 分三类，Remediation 不同：

  • Type A · zero hit：检索 score 全低于阈值，recall gap → 补 doc 或改分块
  • Type B · ACL filter：有 doc 但 filter 后 0 条 → 修 ACL 或用户权限
  • Type C · LLM 拒答：有 chunk 但模型判定不足以回答 → 改 prompt 或补摘要元数据
混为一谈
  • 空答率升高就加 chunk
  • 不记录 filter 后 hit_count
  • 无 Type 标签的聚合 KPI
分型治理
  • 日志带 empty_type 枚举
  • Type B 联动 权限章 审计
  • Type A 进 top query 补库 backlog

java 
@PostMapping("/quality/tickets/{id}/approve")
@PreAuthorize("hasRole('KB_OPS')")
public ResponseEntity<Void> approve(
    @PathVariable String id,
    @RequestBody ApproveRequest req,
    @AuthenticationPrincipal User ops) {
    workflowService.approve(id, req.action(), ops.email());
    ingestScheduler.enqueueFromTicket(id);
    return ResponseEntity.accepted().build();
}
