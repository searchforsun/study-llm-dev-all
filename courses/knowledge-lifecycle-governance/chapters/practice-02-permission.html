为什么要学本章

CorpAssist S1 KB 聚合 HR、财务、研发多部门文档。若在 LLM 生成答案后再做权限裁剪，模型已「看过」越权 chunk，存在 permission leak 风险。Basics 已在 分块与元数据规范 定义 acl tags；practice 要在检索层把 tenant 与 ACL 写进 filter，做到「搜不到 = 不存在」。

学完你能
  • 设计 kb acl 元数据模型（tenant + role + doc_type）
  • 编写 Milvus / Spring filter expr 实现 row filter
  • 为 multi tenant kb 配置 collection 或逻辑分区策略
  • 排查「能搜到但不应看到」的 retrieval isolation 故障
本章先记住 3 件事
  • 检索前过滤：filter 在 vector search 之前，非 prompt 事后删段落。
  • 元数据即权限：ingest 时必须写入 acl_tags；缺字段默认 deny。
  • 租户边界：tenant_id 是每个 filter 的 mandatory 条件。

ACL

CorpAssist · 财务政策隔离 员工张三（tenant=corp-a，roles=employee）问「年终奖计税规则」。KB 含 FIN-TAX-2026（acl=finance,hr-admin）与 EMP-HANDBOOK（acl=all-staff）。检索必须只返回后者相关 chunk。

acl metadata 在 ingest 阶段由源系统 IAM 映射写入向量库。CorpAssist schema 约定：

  • tenant_id：多租户硬隔离
  • acl_tags[]：角色/部门标签，检索时与用户 claims 求交
  • doc_type：policy / contract / faq，可叠加额外规则

ingest 写入 ACL 元数据

python 
# corpassist/ingest/acl_mapper.py
def map_acl(source_doc) -> dict:
    return {
        "tenant_id": source_doc.tenant,
        "acl_tags": iam.resolve_tags(source_doc.folder, source_doc.classification),
        "doc_type": source_doc.type,
    }
# upsert 时每 chunk 携带上述 metadata
事后过滤（禁止）
  • Top-K 检索后再用 Python 删 chunk
  • 把 ACL 写进 system prompt「不要泄露」
  • 缺 acl_tags 时默认 allow
检索层 filter（推荐）
  • Milvus expr 带 tenant + acl 交集
  • 缺 metadata → 不进入候选集
  • 审计日志记录 filter 与用户 claims

先想 10 秒：若 K=10 检索后再过滤，剩 2 条合法 chunk，对答案质量有何影响？

参考思路

有效召回不足，可能空答；且模型已接触 8 条越权上下文，存在泄露风险。必须在 search 前 filter。

行级过滤

row filter 将用户 JWT claims 翻译为 Milvus boolean expr。CorpAssist 规则：tenant 必等，acl 为数组包含（ANY）语义。

filter 表达式模板

yaml 
# corpassist/config/acl-filter.yaml
filter_template: |
  tenant_id == "{tenant_id}"
  and (
    array_contains_any(acl_tags, [{user_tags}])
    or array_contains(acl_tags, "all-staff")
  )
deny_if_missing_acl: true
max_candidates: 50   # filter 后再取 top_k

Spring VectorStore 检索

java 
public List<Document> search(String query, UserContext ctx) {
    String filter = aclFilterBuilder.build(
        ctx.tenantId(),
        ctx.roles(),
        ctx.departments()
    );
    return vectorStore.similaritySearch(
        SearchRequest.query(query).withTopK(8).withFilterExpression(filter)
    );
}
用户场景filter 要点期望结果
跨 tenant 攻击 tenant_id == user.tenant 返回 0 条
同 tenant 无 finance 角色 acl 不含 finance 标签 财务文档不可见
hr-admin acl 含 hr-admin 或 all-staff 可见 HR 政策

检索隔离

tenant kb 隔离有两种工程选型：物理隔离（每 tenant 独立 collection）与逻辑隔离（共享 collection + 强制 tenant filter）。CorpAssist S1 选逻辑隔离 + mandatory filter，advanced 章再讨论联邦多 KB。

检索隔离链路
sequenceDiagram
  participant U as 用户
  participant G as Spring Gateway
  participant R as RAG Service
  participant M as Milvus
  U->>G: 提问 + JWT
  G->>R: 转发 claims
  R->>R: 构建 filter expr
  R->>M: search(query, filter)
  M-->>R: 仅授权 chunks
  R-->>U: 生成答案

filter 在 M 侧执行；LLM never 看到越权 chunk。

  • 单 collection，运维简单
  • 必须代码层保证 filter 不可绕过
  • CorpAssist S1 默认
  • 每 tenant 独立 collection/alias
  • 误配 filter 风险低
  • 成本与运维复杂度更高
FAQ：BM25 混合检索如何加 ACL？

稀疏与稠密两路检索各自带相同 filter；RRF 合并前不得引入未过滤结果。CorpAssist 在 hybrid retriever 封装层统一注入 AclFilter 对象。

本章结论

  • ACL：ingest 写入 tenant + acl_tags；缺字段 deny；禁止 LLM 事后裁剪。
  • 行级过滤：claims → Milvus expr；tenant mandatory；array_contains_any 匹配角色。
  • 检索隔离：filter 在 vector search 前；审计 claims 与 expr；混合检索两路同 filter。

判断练习

  1. （找错）工程师在 RAG prompt 末尾加「若内容涉及财务请回答无权」。为何仍违反本章？

    参考答案

    模型已在 context 中看到财务 chunk，属于 permission leak；必须在 Milvus search 前 filter，使越权 chunk 不进入 context。

  2. （场景）混合检索只给稠密向量加 filter，BM25 路未加。攻击者可能得到什么？

    参考答案

    BM25 路可能召回越权文档进入 RRF 合并，最终答案引用敏感片段。两路检索必须注入相同 AclFilter。
