本章先记住 3 件事
  • 增量默认：按 doc_id + content_hash 识别变更；删除走 tombstone，勿留 orphan 向量。
  • 全量兜底：Embedding/分块/schema 变更必须新 index + alias；增量无法修复历史脏数据。
  • 双栈一致：Python Milvus 写入与 Spring VectorStore 读 alias 必须同一 index_version。

增量管道

CorpAssist · nightly 同步 每晚 02:00 从 Confluence 空间 HR-POLICY 拉取变更；法务文件夹平均 120 篇/天改动 3～8 篇。增量管道须在 30 分钟内完成 embed + upsert，并在 morning peak 前完成 Spring 侧 cache 刷新。

incremental pipe 由调度器、变更检测、分块/embed、向量 upsert 四段组成。与 basics 采集不同，practice 强调与当前 index_version 绑定的幂等写入。

CorpAssist 增量管道四步
flowchart LR
  CDC[变更检测] --> Split[分块+元数据]
  Split --> Embed[embed incremental]
  Embed --> Upsert[Milvus upsert]
  Upsert --> Notify[stack sync 事件]

删除文档走 delete by doc_id；更新走 upsert 同 chunk_id。

变更检测：CDC 与 content_hash

CorpAssist 使用 cdc sync 表记录上次同步水位。对比源站 updated_at 与本地 content_hash，输出 changed / deleted 两集合。

python 
# corpassist/ingest/incremental_runner.py
def run_incremental(source: str, index_version: str, since: datetime) -> IngestReport:
    changed, deleted = cdc_sync.diff(source, since)
    for doc_id in deleted:
        vector_store.delete(filter=f'doc_id == "{doc_id}" and index_version == "{index_version}"')
    for doc in changed:
        chunks = split_with_schema(doc, schema=CORPASSIST_SCHEMA)
        vectors = embed_incremental(chunks)  # 仅变更块 re-embed
        upsert(vectors, metadata={"index_version": index_version, "acl_tags": doc.acl})
    return IngestReport(changed=len(changed), deleted=len(deleted))
反模式
  • 全库扫描 re-embed，无 hash 短路
  • 删除源文档但不删 Milvus 向量
  • 增量写入未带 index_version
推荐
  • content_hash 不变则跳过 embed
  • delete filter 带 doc_id + version
  • upsert 携带 acl_tags 供检索层过滤

先想 10 秒：某篇政策标题未变、正文改了一个字，content_hash 会变吗？对应应触发 embed 还是跳过？

参考思路

hash 会变 → 进入 changed 集合 → 对该 doc 的 chunks 做 embed incremental；若只改元数据 ACL 则可能只 upsert metadata 不重算向量（视实现而定）。

全量重建

当 Embedding 模型从 bge-m3 升级到 text-embedding-3-large，或 分块与元数据规范 调整 overlap 从 128→256，增量无法修复历史向量，必须 full rebuild：在新 collection 全量 embed，对账通过后 alias 切换。

重建窗口与对账门禁

触发条件策略CorpAssist 示例
Embedding 模型变更 新 index + 全量 re-embed index_version=v20260601_emb_v2
分块/schema 变更 新 index + 全量重建 chunk_size 512→768
增量漂移超阈值 scheduled full rebuild 季度对账 diff > 0.5%
仅文档内容变更 增量即可 日常 Confluence 更新

蓝绿切换配置

yaml 
# corpassist/config/rebuild.yaml
rebuild:
  window_cron: "0 3 * * 0"          # 周日 03:00 低峰
  index_version: v20260530_full
  alias: corpassist_kb_prod
  blue_green: true
  keep_versions: 3
  reconcile:
    - source_doc_count
    - chunk_count
    - vector_count
  gate:
    max_diff_ratio: 0.005           # 对账 diff > 0.5% 禁止 alias switch
  • 耗时：分钟级（变更集小）
  • 风险：delete 遗漏、hash 碰撞（极低）
  • 适用：正文/元数据小改
  • 耗时：小时级（万级文档）
  • 风险：切换窗口内查询一致性
  • 适用：模型/schema 变更、对账修复

双栈触发

CorpAssist 采用 Python 索引管线写 Milvus，Spring AI VectorStore 负责在线检索。dual trigger 指：索引任务完成后发布事件，Spring 侧刷新 alias 绑定或本地 cache，避免读到 stale index。

事件契约（Python → Spring）

json 
{
  "event": "INDEX_PUBLISHED",
  "kb_id": "corpassist_s1_hr",
  "index_version": "v20260530",
  "alias": "corpassist_kb_prod",
  "mode": "incremental",
  "stats": { "upserted": 42, "deleted": 3 },
  "published_at": "2026-05-30T02:28:00Z"
}

Spring 侧监听刷新

java 
@EventListener
public void onIndexPublished(IndexPublishedEvent evt) {
    if (!evt.kbId().equals("corpassist_s1_hr")) return;
    vectorStoreRegistry.switchAlias(evt.alias(), evt.indexVersion());
    meterRegistry.counter("kb.index.sync", "mode", evt.mode()).increment();
}
排障提示

用户反馈「昨晚更新了政策仍搜到旧版」：先查 Python ingest 日志是否 SUCCESS → 事件是否投递 → Spring index_version 是否与 Milvus alias 一致；勿只重启 Spring 而不核对 alias。

本章结论

  • 增量管道：CDC + hash 识别变更；delete/upsert 带 doc_id 与 index_version；ACL 元数据随块写入。
  • 全量重建：模型/schema 变更必须新 index；对账 diff 超阈值禁止 alias switch。
  • 双栈触发：Python 发布 INDEX_PUBLISHED；Spring 切换 alias 并打点，保证读写同一版本。

操作步骤

判断练习

  1. （排障）运营称「源站已删文档，RAG 仍能引用」。请按优先级列出 3 个检查点（从 ingest 到检索）。

    参考答案

    1) CDC 是否把 doc 放进 deleted 集合；2) Milvus delete filter 是否执行成功（doc_id + index_version）；3) Spring alias 是否指向仍含旧向量的 index_version。常见根因：增量漏删或 Spring 未 sync alias。

  2. （找错）Embedding 刚从 v1 换 v2，运维选择「仅跑 incremental_runner」且 blue_green: false。违反哪条本章契约？

    参考答案

    违反「Embedding 变更必须 full rebuild + blue_green」。仅 incremental 会导致同一 alias 下混用两种 embedding 空间，检索质量不可控；应新建 index 全量 re-embed 再 alias switch。
