## KB 治理答辩模板（answer-template-kb）

1. **生命周期（Situation）**
   - 源：Confluence/OSS；采集 quality gate（C6）；血缘 source_uri
   - 发布：分块 schema（C7）→ embed → Milvus 蓝绿 alias

2. **版本与变更（Task）**
   - index_version + embedding_model 绑定
   - 模型变更 → 新 collection + 全量 re-embed + 评测门禁（NS1）
   - 回滚：alias 指回 v(N-1)（见 index-migrate）

3. **ACL（Action）**
   - acl_tags 入库；Spring SearchRequest filter 表达式
   - 禁止 LLM 输出后删句 — permission leak

4. **质量运营（Result 量化）**
   - 覆盖率、zero-hit、stale 队列、坏例进 Golden Set
   - 发版 eval_pass_rate ≥ 阈值

5. **成本（NS3）**
   - 热库 SSD / 冷库归档；增量 embed cache

案例 A：acl leak case

现象根因修复预防
销售员工问答出现「管理层薪酬带宽」 检索未带 ACL filter；chunk 有 acl:mgmt-only 但仍被召回 Spring VectorStore 查询加 acl_tags IN user.acl；purge 误入库 batch 入库 schema 必填 acl_tags；集成测试用无权限用户断言 zero hit
错误答法
  • 「让 LLM 不要泄露敏感信息」
  • 「输出后正则脱敏」
正确答法
  • 检索层 filter，chunk 根本不进 context
  • 审计日志记录 filter 表达式与 tenant

案例 B：re embed case

Embedding 从 v2 升到 v3，团队只改了 Python 脚本里的 model 名，未建新 collection、未 re-embed。现象：召回分数整体漂移，评测 faithfulness 从 0.88 跌到 0.71。

  • kb debug 第一步：查 index_version 与 embedding_model 是否一致（索引版本与回滚）。
  • 修复：build hr-policy-v3-emb-v3 → 评测 pass → alias 切换；保留 v2 24h 可回滚。
  • index migrate 话术：「Embedding 空间不可混用；变更等于新索引 + 蓝绿，不是原地改字段。」

案例 C：stale case

Wiki 制度已更新，用户仍得到 2024 版远程办公条款且 citation 指向旧 URL。

排查层检查项工具/指标
源Confluence 是否已 publishsource_version 对比
采集增量 job 是否跑成功ingest_job_id 日志
索引旧 chunk 是否 deletechunk_id 级 upsert/purge
运营stale 规则是否告警知识质量运营 队列
先想 10 秒：面试官问「RAG 不准怎么办」，你的第一句应说什么？

先区分检索不到 vs 检索到错的/旧的；查 zero-hit 与 stale 指标，而不是先换 GPT-4。

高频口述题（含 governance qa）
  1. （场景）10 万 PDF 如何增量更新？答：CDC/Webhook + chunk_id 幂等 upsert + 夜间 rebuild 窗口（增量更新）。
  2. （判断）ACL 在 Prompt 里写「不要泄露财务数据」够不够？答：不够，必须检索 filter（案例 A）。
  3. （权衡）全库广播 vs 意图路由？答：联邦路由降延迟与泄露面（多知识库联邦）。
  4. （合规）员工行使删除权后向量怎么办？答：按 source_uri 级联 purge + 审计（合规保留与删除）。
  5. （成本 NS3）历史制度很少查，如何降存储？答：冷热分层 + 归档不删血缘（存储与索引成本）。
