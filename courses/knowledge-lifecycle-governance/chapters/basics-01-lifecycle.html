为什么要学本章

企业 S1 知识库问答失败，往往不在「Prompt 写得不好」，而在文档从未正确入库、版本不可回滚、无人监控陈旧度。本章建立 kb lifecycle 全景，避免把 RAG 当成一次性脚本。

学完你能
  • 口述 CorpAssist 从 ingest publish 到 monitor retire 的闭环
  • 说明 Python 索引管线与 Spring VectorStore 在生命周期各阶段的职责边界
  • 为 采集调度与数据质量 的调度/校验/血缘打基础
本章先记住 3 件事
  • 四阶段：采集 → 发布 → 监控 → 退役；缺任一环节都会在生产暴露。
  • 治理先于算法：kb governance 决定召回上限，Rerank 救不了脏源数据。
  • 双栈同契约：索引版本、ACL、chunk schema 由治理层定义，Python/Spring 只是执行面。

采集-发布-监控-退役

CorpAssist · 人事制度库（30 秒） 员工问「2026 年远程办公政策」——若 Wiki 已更新但向量库仍是 2024 版 chunk，模型会自信地答错。生命周期要回答：谁触发入库、何时切换索引、如何发现陈旧文档。

每个 lifecycle stage 都有可验收产出，而不是「跑完脚本就算完」：

阶段核心动作可验收产出典型失败信号
采集 Ingest 从 Confluence/OSS/SharePoint 拉取、解析 PDF 原始对象 + 解析文本 + 血缘 ID 扫描件乱码、缺 OCR 链路
发布 Publish 分块、Embedding、写入 Milvus/ES，蓝绿切换 索引版本号 + alias 指向 Embedding 模型变更未 re-embed
监控 Monitor 覆盖率、空答率、热点 query、stale 检测 运营看板 + 告警阈值 用户反馈坏例无人进评测集（NS1）
退役 Retire 下线过期制度、归档冷存储、级联删 chunk 删除审计日志 + 检索不可命中 源文件删了向量仍在

反例 vs 推荐：把 KB 当一次性任务

不推荐
  • 上线前手工跑一遍 ingest，之后从不更新
  • 直接写 prod collection，无版本号
  • 用户投诉才人工查 Milvus
  • 源文档删除后不做向量 purge
推荐
  • 定时/事件触发增量管道（增量更新与重建）
  • 蓝绿 alias 切换 + 回滚预案
  • 覆盖率/空答/stale 进看板（NS1 飞轮）
  • 退役走 ops handoff checklist
先想 10 秒：如果只做了「采集+发布」，缺监控会怎样？

制度变更后 RAG 仍引用旧 chunk， citation 看起来合法但内容错误——比空答更危险，因为用户更信任「有出处」的答案。

角色分工

role division 不清会导致「研发只交付向量库、业务以为已经可用」：

业务方 / 知识 Owner

定义哪些文档进库、权限范围、废止日期；审核冲突与过期内容。

平台研发（Python 轨）

采集管道、解析、分块、Embedding 批处理、Milvus 写入与对账脚本。

应用研发（Spring 轨）

VectorStore 检索、ACL filter 注入、与 ChatClient/RAG 链路集成。

知识运营

看板、坏例队列、人工审核、驱动 re-ingest 与评测回归（NS1）。

双栈协作：同一份治理契约

负责：重计算 ingest、批量 re-embed、索引 build、与对象存储/CDN 交互。

输出：带 index_version、chunk_id、acl_tags 的向量记录（schema 见 分块与元数据规范）。

负责：在线检索、租户上下文注入 filter、引用溯源返回前端。

约束：不在 Spring 侧「事后过滤 ACL」——必须在 VectorStore 查询表达式中带权限（权限与多租户）。

阶段交接清单（节选）

# corpassist-kb-handoff.yaml — 发布前 ops-handoff 片段
publish_gate:
  index_version: "hr-policy-2026-03-v2"
  chunk_count_expected: 1842
  embedding_model: "text-embedding-v3"
  acl_schema_version: "1.2"
  eval_pass_rate_min: 0.85   # NS1 门禁
monitoring:
  owners: ["kb-ops@corp", "platform-rag@corp"]
  dashboards: ["kb-coverage", "zero-hit-top20"]
RAG 主课已经会 ingest 了，为什么还要单独学生命周期？
RAG 主课教链路怎么跑通；本课教怎么持续跑、谁负责、坏了怎么回滚。生产环境 80% 工时在治理与运营，不在第一次 PoC。
corpassist kb 和「一个 Milvus collection」是一回事吗？
不是。CorpAssist KB 是产品概念：含源系统、schema、ACL、版本策略与运营流程；Milvus 只是其中一个存储组件。

本章结论

  • 四阶段闭环：采集-发布-监控-退役缺一不可；缺监控会让陈旧制度静默害人。
  • 角色分工：Owner 供数与审核、Python 建管道、Spring 做带 ACL 的在线检索、运营驱动 NS1 飞轮。
