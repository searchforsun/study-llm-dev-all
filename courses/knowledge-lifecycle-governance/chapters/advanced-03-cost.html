为什么要学本章

KB 体量上来后，账单大头往往在向量存储、对象存储、embed API、检索 QPS。CorpAssist 用 Milvus + S3 双栈，若把所有历史 PDF 都保持热索引，index cost 会线性爆炸。本章用 cost ns3（Node-Storage-Service 三分法）帮你和 FinOps 对齐语言：哪些该热、哪些该冷、归档后如何按需召回。

  • 估算向量维数 × chunk 数 × 副本数的 storage budget
  • 制定 cold archive 与按需 re-index 流程
  • 用 embed cache 与 compress archive 降低重复 embed 成本
本章先记住 3 件事
  • NS3 分开看：Node（Milvus 计算/内存）、Storage（磁盘/S3）、Service（embed API、网关）各占一块预算。
  • 分层是治理动作：tier storage 要与留存 policy、访问频率联动，不是一次性搬家。
  • 归档 ≠ 删除：冷库 doc 可脱离向量索引，但 metadata 与恢复路径要保留（合规见 合规保留与删除）。

分层

CorpAssist · 年度 FinOps 复盘（60 秒） Milvus 集群内存占用 78%，S3 向量快照 + 原文 420TB。过去 12 个月检索日志显示：80% query 命中近 90 天文档，2019 年制度 PDF 几乎零命中却占 35% 存储。CTO 要求在不降召回的前提下，把 index cost 降 30%。

tier storage 把 KB 资产按访问频率与业务关键度拆层：

层级数据范围索引形态检索 SLA
Hot 近 90 天 + 高频制度 Milvus 全量向量 + 内存友好索引（HNSW/IVF） < 200ms p95
Warm 90 天～2 年 向量在 Milvus 磁盘型 collection；原文 S3 Standard < 500ms p95
Cold 低命中历史 / 已 supersede 仅 S3 + metadata；无在线向量 按需 re-index，分钟级

NS3 成本分解（示意）

cost ns3 把账单拆成三块，便于和云厂商账单对照：

  • Node — Milvus query node / data node vCPU、RAM（HNSW 内存随向量数线性涨）。
  • Storage — 向量段文件、S3 原文、快照；compress archive 对原文 gzip/zstd。
  • Service — Embedding API 按 token 计费、跨 AZ 流量、Spring 检索网关实例。
python 
# NS3 粗算：向量存储 GB ≈ chunks * dim * 4 bytes * replica
def vector_storage_gb(chunks: int, dim: int = 1024, replica: int = 2) -> float:
    return chunks * dim * 4 * replica / (1024 ** 3)

# 例：5000 万 chunk → 约 390 GB（不含 HNSW 图额外开销 ~1.5x）
budget = {
    "node_ram_gb": vector_storage_gb(50_000_000) * 1.5,
    "storage_s3_tb": 420,
    "service_embed_usd_month": 12_000,
}

Cold 层不在线检索；命中 metadata 触发异步 re-index 后再查 Hot（详见 增量更新与重建）。

先想 10 秒：把 Warm 全降到 Cold 会怎样影响 empty-answer？

历史制度类 query 会 zero-hit 上升；应结合 知识库可观测 的 top_query 与 recall_gap，只对低命中 doc 降温，而非一刀切。

归档

cold archive 是把文档从在线向量索引「退场」到低成本存储，同时保留可恢复性与合规留存。

归档触发条件

  • 检索命中次数 < 阈值 且 doc_age > N 天
  • 新版本 superseded_by 已发布 30 天
  • FinOps 预算告警触发人工归档批次

归档作业步骤

bash 
# 1. 标记 tier=cold，停止增量 embed
# 2. Milvus 删除向量（保留 collection schema）
# 3. S3 转 Glacier / Intelligent-Tiering + compress
# 4. metadata 保留 restore_uri、content_hash、retention_expires_at
不推荐
  • 归档时删掉 metadata，无法按需 re-index
  • 每次 query 冷数据都同步 re-embed（成本爆炸）
  • 无 embed cache，重建索引重复调用 API
推荐
  • 归档前导出 chunk 快照 + embedding 至 S3（可选，换模型时仍要 re-embed）
  • restore 走审批 + 异步 job，完成后 alias 切回（蓝绿）
  • embed cache key = hash(model_version, chunk_text)

成本优化决策表

痛点NS3 项治理手段
Milvus 内存打满 Node 降温 doc、降 replica、IVF 替代 HNSW（温层）
S3 账单高 Storage Intelligent-Tiering、compress archive
重建索引 API 贵 Service embed cache、增量 re embed 仅脏 chunk
常见误区
  • 只砍 Storage 不砍 Node — 向量仍在 Milvus 时 Node 内存不会降。
  • 归档违反 legal hold — hold 中的 doc 禁止转 Cold 删除向量。
  • 无 storage budget 仪表盘 — 应每月 review chunks × dim × tier 分布。

本章结论

  • 分层：Hot/Warm/Cold 与检索 SLA 绑定；NS3 三分法对齐 FinOps 语言。
  • 归档：向量退场 + metadata 保留 + 按需 re-index；embed cache 与合规留存并行考虑。
