为什么要学本章

索引「能搜到」不等于「该用」。CorpAssist 运营台每周收到「答案引用 2023 版报销制度」类工单——根因是 outdated doc 仍在索引中，或新旧两版 duplicate chunk 并存。Basics 采集校验拦不住「源站已过期但仍在 sync」；practice 用 quality ops 规则 + human review kb 队列持续治理。

学完你能
  • 配置 stale detect 规则（TTL、源站 etag、业务有效期）
  • 运行 conflict detect 发现同主题多版本 chunk
  • 设计 review queue 工作流与 SLA
  • 将质量事件关联到 增量更新 触发 re-ingest 或 tombstone
本章先记住 3 件事
  • 陈旧 ≠ 删除：先降权/打标，审核后再 tombstone 或归档。
  • 冲突要合并：同 doc_key 多 version 只保留 canonical 一条。
  • 人工不可省：自动规则产出 review queue，运营确认后才改索引。

陈旧检测

CorpAssist · 政策过期 「远程办公政策 v3」2026-04-01 生效，v2 应在 2026-03-31 后标记 stale。Confluence 标签未及时更新，CDC 仍增量 sync v2 正文。质量规则需在 valid_until 过期后自动入队。

stale detect 综合三类信号：

  • 元数据 TTL：valid_until < now()
  • 源站状态：HTTP 404 / archived flag
  • 使用反馈：用户点「答案过时」≥ N 次

质量规则配置

yaml 
# corpassist/quality/stale_rules.yaml
rules:
  - id: meta_expired
    when: "metadata.valid_until < now()"
    action: enqueue_review
    priority: high
  - id: source_gone
    when: "source.status in ['404','archived']"
    action: tombstone_candidate
  - id: feedback_stale
    when: "feedback.outdated_count >= 5"
    action: enqueue_review
    priority: medium
直接删索引
  • 误删仍有效的归档副本
  • 无审计记录
  • 用户突然大量空答
review queue
  • 运营确认后再 tombstone
  • 保留 decision 日志
  • 可触发增量 delete

先想 10 秒：valid_until 过期但源站仍显示 published，应先 tombstone 还是 enqueue_review？

参考思路

优先 enqueue_review：可能是元数据错误；确认后再 tombstone + 增量 delete。

冲突

conflict detect 识别「同一业务键多版本并存」。CorpAssist 用 doc_key（如 policy:remote-work）+ version_tag 检测冲突：若 index 中存在 2+ 个 active version_tag 且 embedding 相似度 > 0.92，标记 conflict。

冲突处理流程
flowchart TD
  Scan[ nightly quality scan ] --> Found{发现 conflict?}
  Found -->|是| Queue[review queue]
  Queue --> Decide{运营裁决}
  Decide -->|保留 v3| Deprecate[降权 v2 + 增量 delete]
  Decide -->|合并| Merge[canonical chunk 重写]
  Found -->|否| OK[记录 OK]

自动扫描不直接改索引，一律经人工审核。

冲突检测脚本

python 
def detect_conflicts(index_version: str) -> list[ConflictCase]:
    groups = group_by_doc_key(index_version)
    cases = []
    for key, chunks in groups.items():
        active = [c for c in chunks if c.status == "active"]
        if len(active) > 1 and max_pair_similarity(active) > 0.92:
            cases.append(ConflictCase(doc_key=key, versions=[c.version_tag for c in active]))
    return cases
冲突类型典型原因处理
版本并存 增量未 delete 旧版 tombstone 旧 version_tag
重复 ingest doc_id 变更但 doc_key 相同 合并 canonical doc_id
跨源重复 Confluence + SharePoint 各一份 运营指定主源

人工审核

kb quality 闭环依赖 review queue：每条工单含 doc_key、规则 id、优先级、建议动作。运营在 CorpAssist 后台执行 approve/reject，通过后触发 增量管道 或 tombstone API。

审核工单 JSON

json 
{
  "ticket_id": "QR-20260530-001",
  "rule_id": "meta_expired",
  "doc_key": "policy:remote-work",
  "suggested_action": "tombstone",
  "priority": "high",
  "assignee": "kb-ops",
  "sla_hours": 24
}

优先级SLA处理人典型规则

文档确认失效：增量 delete + 源站停止 sync。适用于过期政策、下架公告。

内容仍有效但 chunk/元数据错误：触发单 doc 增量 re-ingest，保留 doc_id。

本章结论

  • 陈旧检测：TTL + 源站状态 + 用户反馈；默认 enqueue_review 而非直接删。
  • 冲突：doc_key 分组 + 相似度；多 version 并存经运营裁决后 tombstone 或合并。
  • 人工审核：review queue 带 SLA；approve 后驱动增量 delete/re-ingest，留审计。

FAQ：质量规则与 incremental 如何衔接？
