为什么要学本章

分块与元数据规范 变更或 Embedding 模型升级时，直接在原 Milvus collection 上覆盖 upsert 风险极高：用户可能瞬间检索到空库或混合两个index version。CorpAssist 用blue green index + alias switch 实现零停机发布，并保留index rollback 与index reconcile 能力。

学完你能
  • 描述 Python 管线建 green collection、Spring 通过 alias 切换的完整步骤
  • 在 smoke 失败时执行rollback plan（alias 指回 blue）
  • 运行对账脚本，比较 staging chunk 数与 Milvus entity 数、version_tag 分布
本章先记住 3 件事
  • 物理双库、逻辑单 alias：线上只读 corpassist_kb_active，背后 blue/green 可切换。
  • 切换前必须对账：index reconcile 不通过禁止 alias switch。
  • 模型变更必 re-embed：re embed 产生新 collection，不是原地改向量。

蓝绿索引

CorpAssist · HR v3.2 发布窗口（50 秒） 当前线上 alias 指向 corpassist_kb_blue（v3.1）。Python 管线已将 v3.2 写入 corpassist_kb_green，共 412 chunks。运营 smoke 通过后，运维执行 alias 切换；Spring VectorStore 无需改 collectionName，只刷新配置缓存。
对象命名示例角色
Logical alias corpassist_kb_active Spring / 检索唯一入口
Blue collection corpassist_kb_blue 当前或回滚目标
Green collection corpassist_kb_green 新建/待切换
version tag 2026-05-28-hr-v3.2 metadata 对账维度

蓝绿切换序列（Mermaid）

alias switch：Python 写 green，Spring 读 alias
sequenceDiagram
  participant PY as Python 管线
  participant MV as Milvus
  participant OP as 运维/治理台
  participant SP as Spring VectorStore
  PY->>MV: 建 green + upsert v3.2
  PY->>OP: reconcile 报告 PASS
  OP->>MV: alias corpassist_kb_active → green
  OP->>SP: 刷新 alias 配置 / 健康检查
  SP->>MV: similaritySearch via alias
  Note over MV,SP: blue 保留 N 天供 rollback

切换的是 alias，不是 Spring 硬编码 collection 名（对照 知识库生命周期全景 双栈配置）。

Python：创建 green 并写入

python 
# CorpAssist 发布：写入 green（示意）
GREEN = "corpassist_kb_green"
ensure_collection(GREEN, schema=C7_SCHEMA, dim=1536)
for batch in load_staging(version_tag="2026-05-28-hr-v3.2"):
    upsert(GREEN, batch)
# 切换 alias 前禁止删除 blue

Spring：固定 alias

yaml 
spring:
  ai:
    vectorstore:
      milvus:
        collectionName: corpassist_kb_active  # logical alias only
        embeddingDimension: 1536
不推荐
  • 原地 delete+insert 同一 collection
  • 切换 alias 前不做 smoke
  • green 建完立刻删 blue
推荐
  • blue/green 物理隔离 + reconcile
  • 黄金问句 smoke + 指标对比
  • blue 保留rollback plan 窗口（如 7 天）
先想 10 秒：Embedding 模型从 text-embedding-3-small 换 large，能否只 re-embed 增量 doc？

维度或模型空间变时，需全量 re-embed 新 green；增量仅适用于同模型同维度下的内容变更（见 增量更新与重建）。

回滚

index rollback 在 CorpAssist 中定义为：alias 指回上一稳定 physical collection，并在治理台记录事件。不是「从备份恢复单个 chunk」。

回滚触发条件

信号阈值示例动作
Smoke recall@3 < 0.85（相对 blue 降 >10%） 自动 No-Go，保持 blue
空答率 spike 15 分钟内 +5pp alias 切回 blue + 告警
对账 FAIL chunk 数偏差 >0.5% 禁止切换；修复 green

回滚命令（示意）

bash 
# 1. alias 指回 blue（Milvus 2.x alias API 示意）
python -m corpassist.index.alias_switch \
  --alias corpassist_kb_active \
  --target corpassist_kb_blue \
  --reason "smoke_fail_hr_v3.2"

# 2. 通知 Spring 刷新 + 复测 smoke
curl -X POST http://rag-api/actuator/refresh
回滚后 green collection 要不要删？
先保留用于根因分析。确认 parser/embedding 问题修复后，可在下次发布窗口复用 green 名 drop+recreate，或轮换 blue/green 角色。
只回滚 Spring 配置不改 alias 行吗？
不行。双栈必须以 Milvus alias 为单一真相；只改 Spring 会导致 Python 对账与线上不一致。

对账

index reconcile 在 alias 切换前比较：staging manifest vs Milvus green 统计 vs 抽样向量质量。

对账项清单

对账项数据源 A数据源 B容差
chunk 总数 staging JSONL 行数 Milvus green num_entities 0
version_tag 分布 manifest 聚合 Milvus query group by 0 偏差
doc_id 集合 源登记册 metadata doc_id distinct 仅允许已标记 retire
随机向量非零 抽样 100 L2 norm > 0 100% pass

对账脚本输出（示意）

json 
{
  "green": "corpassist_kb_green",
  "version_tag": "2026-05-28-hr-v3.2",
  "staging_count": 412,
  "milvus_count": 412,
  "version_tag_match": true,
  "orphan_doc_ids": [],
  "status": "PASS"
}

对账 FAIL 时，常见根因：re embed 中断、metadata schema_version 不匹配、upsert 幂等键冲突。修复后重新跑 reconcile，再进入 alias 切换。

本章结论

  • 蓝绿索引：物理 blue/green + 逻辑 alias；Python 写 green，Spring 只读 corpassist_kb_active。
  • 回滚：index rollback = alias 指回 blue + 刷新 Spring；保留 rollback 窗口与事件审计。
  • 对账：index reconcile PASS 是切换前置条件；chunk 数与 version_tag 零容差。
