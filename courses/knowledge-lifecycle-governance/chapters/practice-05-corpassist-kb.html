先想 10 秒：为何后台禁止直连 Milvus，而要走 governance API？

参考思路

统一审计、RBAC、与增量/对账闸门集成；避免绕过 quality approve 或 ACL 校验。

流程

govern workflow 是 CorpAssist S1 标准作业：新文档入库、质量处置、索引发布、监控反馈形成闭环。

S1 治理主流程
flowchart TD
  A[业务提交源文档] --> B[Ingest 校验]
  B --> C{Quality gate}
  C -->|pass| D[Incremental embed]
  C -->|fail| Q[Review queue]
  Q -->|approve| D
  D --> E[对账 reconcile]
  E -->|ok| F[Alias switch + stack sync]
  F --> G[Metrics 监控]
  G -->|gap/stale| Q

approve flow 在 C 与 Q 节点；对账失败则 block F。

角色与 SLA

  • 业务供数：维护源站元数据 valid_until
  • kb-ops：24h 内处理 high priority review
  • kb-admin：full rebuild、alias 切换、ACL 模板变更
  • 研发：pipeline / API 维护，不日常改索引
java 
@PostMapping("/quality/tickets/{id}/approve")
@PreAuthorize("hasRole('KB_OPS')")
public ResponseEntity<Void> approve(
    @PathVariable String id,
    @RequestBody ApproveRequest req,
    @AuthenticationPrincipal User ops) {
    workflowService.approve(id, req.action(), ops.email());
    ingestScheduler.enqueueFromTicket(id);
    return ResponseEntity.accepted().build();
}

指标

kb metrics board 汇聚 practice-04 指标，并增加治理效率 KPI：review 积压、ingest 成功率、alias 切换次数。

S1 运营看板最小集

面板指标告警阈值
用户体验 coverage_rate、empty_answer_total coverage < 70%
检索健康 recall_gap_rate、ACL_FILTER 占比 gap > 10%
内容健康 stale tickets、conflict open high 积压 > 48h
索引健康 index_version、reconcile diff diff > 0.5%

查 incremental 日志 → review 队列 → 空答分型 Top5 → 必要时触发单 doc re-ingest。

full rebuild → reconcile → alias switch → stack sync 事件 → 24h 监控 coverage/empty 回归。

FAQ：运营台与 advanced 多 KB 联邦关系？

S1 运营台管理单一 logical KB（多 tenant）；多知识库联邦 在 advanced 扩展路由与合并，控制台增加 federate 模块。

本章结论

  • 后台：五大模块经 governance API；RBAC + 审计；禁止直连 Milvus。
  • 流程：ingest → quality gate → embed → reconcile → alias；review approve 驱动增量。
  • 指标：coverage/empty/hot + 治理 KPI；告警联动 ingest/ACL/quality。

判断练习

  1. （场景）kb-admin 为赶进度跳过 reconcile 直接 Milvus alias switch，随后 coverage 骤降。最可能缺失哪两个 practice 环节？

    参考答案

    1) 对账 diff 门禁未执行，索引与源不一致；2) stack sync 未确认，Spring 仍读旧 alias。应走运营台 publish 流程 + INDEX_PUBLISHED 验收。

  2. （找错）运营在 LLM 侧加「请勿回答财务问题」替代 ACL filter，是否满足 S1 治理？

    参考答案

    不满足。违反检索层 ACL 过滤原则，存在 permission leak；必须在 Milvus search filter 与运营台 ACL 审计模块修复。
