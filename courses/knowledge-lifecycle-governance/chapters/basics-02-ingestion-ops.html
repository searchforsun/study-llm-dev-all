为什么要学本章

脏 PDF、扫描件乱码、重复入库是 S1 召回上限的头号杀手。没有 ingestion ops，后面 Rerank 与 Prompt 都在垃圾上优化。

学完你能
  • 为 CorpAssist 选择 ingest schedule（cron / 事件 / CDC）
  • 定义 quality gate ingest 拦截规则
  • 用 data lineage 从 chunk 追溯到源文档
本章先记住 3 件事
  • 调度：全量窗口 + 增量 CDC，避免 peak 时段 rebuild。
  • 校验：解析成功率、空页率、字符集、最小长度——不过 gate 不入库。
  • 血缘：每个 chunk 必须能回答「来自哪份源、哪次 ingest job」。

调度

CorpAssist · 制度库 nightly ingestWiki 每天 02:00 批量导出；财务 PDF 在月末全量。需避免与大促检索高峰同时 re-embed。
模式适用注意
cron ingest可预测批量源（Wiki 导出）设 rebuild_window 与并发上限
事件/Webhook单文档审批通过即入库需幂等 job_id
CDC / 增量DB/对象存储变更流见 增量更新与重建
调度与队列
flowchart TB
  CRON[cron 02:00] --> Q[ingest 队列]
  WH[Webhook] --> Q
  Q --> WORKER[Python worker]
  WORKER --> GATE{quality gate}
  GATE -->|pass| PIPE[分块+embed]
  GATE -->|fail| DLQ[死信+告警]
先想 10 秒：为什么 ingest 不要和业务高峰重叠？

re-embed 占 GPU/CPU 与 Milvus 写入 IO，会拉高检索 P95（NS3）。

校验

data validate 在 Embedding 之前执行，失败文档进入人工队列而非静默跳过：

不推荐
  • PDF 解析失败仍写空 chunk
  • 扫描件无 OCR 直接入库
  • 重复 MD5 不 dedup
推荐
  • 空文本率 > 30% → reject
  • OCR 置信度阈值
  • 源 URL + content_hash 去重

质量门禁配置示例

yaml
quality_gate:
  min_chars: 80
  max_blank_page_ratio: 0.3
  require_ocr_for_scanned_pdf: true
  allowed_mime: [application/pdf, text/markdown]
  on_fail: route_to_review_queue  # c6-ingest 质量门禁

上述规则对应能力 c6 ingest：未过 gate 的文档不得进入 Embedding 队列。

血缘

source trace 让用户 citation 可点击回到源系统，也是合规删除的依据：

json
{
  "chunk_id": "hr-remote-2026-sec3-p2",
  "source_uri": "confluence://HR/remote-policy-v3",
  "source_version": "v3.2",
  "ingest_job_id": "job-20260315-0200",
  "ingested_at": "2026-03-15T02:14:00Z",
  "content_hash": "sha256:abc..."
}

Spring 侧返回引用时应带 source_uri；purge 时按 source_uri 级联删 Milvus 实体（合规保留与删除）。

本章结论

  • 调度：cron/事件/CDC 按源系统特性选型，避开检索高峰。
  • 校验：quality gate 在 embed 前拦截脏数据，失败进 review 而非空 chunk。
  • 血缘：chunk 必带 source_uri、job_id、content_hash，支撑溯源与删除。
