为什么要学本章

CorpAssist 代码助手（S4）既要低延迟补全，又要满足私有代码不出域；若一律走云 API，成本与合规都会失控；若一律私有化，又可能拖慢迭代。应用工程师必须先分清三种 Serving 模式 的边界，再谈 vLLM 与双栈接入。

学完你能
  • 用「合规 · 成本 · 延迟」三角为 S4 补全 vs 通用问答各写一条选型结论
  • 画出 CorpAssist 混合架构：默认云 API、敏感路径走 on-prem LLM
  • 向同事说明为何 S4 更适合 private deploy，而运营文案可暂留 cloud API
本章先记住 3 件事
  • 三种模式：API Serving（云）、私有化（vLLM 等）、混合 Serving（按路由分流）。
  • 选型先问场景：S4 补全看延迟与数据驻留；批量任务看吞吐与单价。
  • 应用层职责：选模式 + 接 OpenAI 兼容端点 + fallback；本章不写 CUDA 算子。

API

CorpAssist · 运营文案生成（30 秒） 市场部用 CorpAssist 生成活动 slogan，输入不含源码与身份证号，QPS 低、可接受 2～3 秒首 token。团队尚未采购 GPU，希望零运维上线——典型 cloud API 场景。

API Serving 指直接调用厂商托管模型（通义、OpenAI 等）：按 Token 计费，运维由云侧承担。优势是快迭代、免 GPU 机房；代价是数据出境/出域策略、单价随 QPS 线性上升，以及无法自定义底层调度（如 KV 块大小）。

适合

非敏感文本、PoC、峰值弹性、多模态尝鲜（视觉模型贵且更新快）。

慎选

私有仓库代码、制度全文、需固定 IP 白名单的内网系统。

CorpAssist 映射

S5 内容工坊默认路径；S4 仅作云侧兜底（私有化故障时）。

云 API 接入：客户端只需改 base_url

python 
from openai import OpenAI

# 云厂商 endpoint — 密钥走环境变量，勿写进仓库
client = OpenAI(
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key=os.environ["DASHSCOPE_API_KEY"],
)
resp = client.chat.completions.create(
    model="qwen-turbo",
    messages=[{"role": "user", "content": "写一句双 11 活动 slogan"}],
)

私有化

private deploy 是在企业 GPU 或专有云上自托管推理（本课主力 vLLM）：数据路径可控、可按 S4 infer 需求调 max-model-len 与量化。代价是GPU 资本支出、镜像升级、容量规划——详见 容量规划入门。

S4 为何倾向私有化

维度云 API私有化 vLLMS4 补全倾向
数据驻留 出域策略需逐条评估 流量不出 GPU 集群 私有化
延迟 p95 公网 + 厂商排队 内网 + 连续批处理 私有化
模型迭代 厂商发版即可用 需自行拉权重、回归评测 云作 fallback
长期成本 随 Token 线性涨 GPU 折旧 + 电费 高 QPS 私有化更优
私有化 Serving 逻辑拓扑
flowchart LR
  IDE[CorpAssist IDE 插件]
  GW[API Gateway]
  VLLM[vLLM OpenAI API :8000]
  GPU[(GPU 节点)]

应用只认 OpenAI 兼容 HTTP；vLLM 使用 负责 GPU 侧批处理与 KV 管理。

反例 vs 推荐：私有化边界

不推荐
  • 为「有 GPU」而私有化，却无 S4 延迟/合规硬指标
  • 私有化集群无 fallback，故障即全站不可用
  • 未做量化选型，7B 模型占满 24G 却无法并发
推荐
  • 用 serving choice 表对齐产品场景（见 Demo）
  • 私有化 + 云 API 双 endpoint，Spring AI 远程 ChatModel 配置 failover
  • 发版前跑代码补全 Golden Set（NS1 飞轮）

混合

混合 Serving 在同一 CorpAssist 产品内按路由规则分流：例如「含 repo:// 上下文的请求 → 私有化；纯自然语言问答 → 云 turbo；附件图片 → 云 VL」。这是 NS3「按场景选模型/部署形态」在基础设施层的落地。

混合 Serving 路由
flowchart TD
  Req[用户请求] --> Router{路由策略}
  Router -->|敏感/ S4 补全| OnPrem[私有化 vLLM]
  Router -->|通用/ 低成本| Cloud[云 API]
  Router -->|私有化不可用| Cloud
  OnPrem --> Resp[统一 OpenAI 响应格式]
  Cloud --> Resp

路由可在网关、Spring AI ChatModel 链或 Python 中间层实现；灰度与版本 可叠加在同一入口。

请求带私有仓库路径 / 密钥扫描命中 私有化 与 安全与合规工程 脱敏策略联动
FIM 补全、max_tokens 小、低 TTFT 私有化 INT4 小模型 见 量化选型

私有化 GPU 队列深度 > 阈值 云 fallback 需限流保护账单

FastAPI 中间件读 X-CorpAssist-Route，选择不同 base_url 的 OpenAI client；Python 指向推理集群 展开超时与连接池。

注册多个 OpenAiChatModel Bean + @Primary 路由 ChatClient；失败时切换备用 Model，见 Spring AI 远程 ChatModel。

常见误区
  • 混合 = 双写 — 同一次补全不应同时打云与私有化，除非 A/B 评测且采样率受控。
  • 忽略计费 — fallback 到云 API 若无 Token 上限，GPU 故障可能变成账单事故。
  • 路由只靠前端 — 敏感判定必须在服务端，客户端可被绕过。

本章结论

  • API：零 GPU 运维、适合非敏感与 PoC；CorpAssist 运营类场景默认路径，S4 仅作兜底。
  • 私有化：S4 补全与私有代码的主路径；OpenAI 兼容 API 屏蔽底层 vLLM，合规与延迟可控。
  • 混合：按敏感信号与负载路由；必须服务端 fallback + 成本护栏，避免「名义混合、实际全云」。
