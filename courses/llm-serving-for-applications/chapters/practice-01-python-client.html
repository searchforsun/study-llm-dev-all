为什么要学本章

CorpAssist 的后台服务（RAG、补全、Agent）常以 Python 实现。它们必须通过 OpenAI 客户端 指向私有化推理集群。跟云 API 不同的是：私有化环境下 base_url 是内网 vLLM、超时需重新估算、多实例还需 负载均衡。

学完你能
  • 配置 OpenAI SDK 指向 vLLM 集群并设置合理的超时与重试
  • 设计带连接池与退避策略的 Python 客户端
  • 排查「客户端报错但 vLLM 正常」类的典型问题
本章先记住 3 件事
  • base_url：私有化 vLLM 的 /v1 后缀不可缺少。
  • 超时：推理超时 必须大于最长生成时间，否则流式中断。
  • 连接池：httpx 连接池 限制并发；高 QPS 场景需调大 max_connections。

负载与超时

负载均衡 (load balance) 在 Python 客户端层有两种实现方式：

  1. 客户端侧 round-robin：在代码中维护多个 base_url，随机或轮询选择。
  2. 网关侧负载均衡：客户端只连一个网关 endpoint，网关转发到多个 vLLM 后端（推荐）。

CorpAssist 生产环境推荐方案 2（网关负载），但在开发和小规模部署中方案 1 更常见。

超时配置全景

超时类型参数S4 推荐值说明
连接超时 timeout.connect 5～10s vLLM 启动或重启时的 TCP 连接
读取超时 timeout.read 60～300s 流式场景必须大于最长生成时间
写入超时 timeout.write 10s 发送请求体，通常非瓶颈
池连接超时 httpx.Limits.max_connections ≥ 预期并发 × 1.5 避免池耗尽阻塞请求
CorpAssist · Python RAG 服务调推理 RAG 服务接收用户问题后检索相关代码片段，拼 prompt 调 vLLM 生成回答。要求：最大检索 8 个片段（约 3K tokens），生成 512 tokens，总耗时可能超过 30s。
不推荐
  • 不设任何超时（默认 5min read timeout 可能太长）
  • 超时后直接抛异常不重试
  • 每次请求新建 OpenAI client（无连接池）
  • 多个线程共用一个 client 但不设 limits
推荐
  • 设 read timeout=120s 兼顾短补全与长生成
  • 超时后指数退避重试（最多 2 次）
  • 全局单例 OpenAI client
  • 按并发预估配置 httpx limits

Python OpenAI 客户端实战

Python OpenAI 客户端 是连接 vLLM 的标准方式。推理客户端 的核心是正确配置 base_url 与 cluster endpoint。

python 
# corp_assist/infer_client.py — CorpAssist 推理客户端
import os
import time
import random
from typing import Optional

from openai import OpenAI, APIError, APITimeoutError
from openai.lib.streaming import ChatCompletionStream

# ---------- 全局单例 ----------
_INFER_CLIENT: Optional[OpenAI] = None

def get_infer_client() -> OpenAI:
    """获取全局推理客户端（单例模式，复用连接池）"""
    global _INFER_CLIENT
    if _INFER_CLIENT is None:
        # 支持多 endpoint 随机负载均衡
        base_urls = os.environ.get(
            "INFER_BASE_URLS",
            "http://vllm:8000/v1"
        ).split(",")
        selected_url = random.choice(base_urls)

        from httpx import Limits, Timeout
        _INFER_CLIENT = OpenAI(
            base_url=selected_url,
            api_key=os.environ.get("INFER_API_KEY", "EMPTY"),
            timeout=Timeout(
                connect=10.0,   # 连接超时
                read=120.0,     # 读超时（流式需更长）
                write=10.0,     # 写超时
            ),
            max_retries=2,
            default_headers={
                "X-CorpAssist-Service": "python-rag",
            }
        )
        # 调整 httpx 连接池
        _INFER_CLIENT._client._limits = Limits(
            max_connections=50,
            max_keepalive_connections=20,
        )
    return _INFER_CLIENT


# ---------- 带重试的推理调用 ----------
def chat_completion(
    messages: list,
    model: str = "corp-coder-7b",
    max_tokens: int = 512,
    temperature: float = 0.1,
    stream: bool = False,
    max_retries: int = 2,
):
    """
    统一推理入口：带指数退避重试。
    用于 CorpAssist 各 Python 后台服务。
    """
    client = get_infer_client()
    last_error = None

    for attempt in range(1 + max_retries):
        try:
            return client.chat.completions.create(
                model=model,
                messages=messages,
                max_tokens=max_tokens,
                temperature=temperature,
                stream=stream,
            )
        except APITimeoutError as e:
            last_error = e
            if attempt < max_retries:
                wait = 2 ** attempt + random.uniform(0, 1)
                print(f"推理超时，第 {attempt+1} 次重试，等待 {wait:.1f}s")
                time.sleep(wait)
        except APIError as e:
            if e.status_code in (429, 502, 503):
                last_error = e
                if attempt < max_retries:
                    wait = 2 ** attempt + random.uniform(0, 1)
                    print(f"API 错误 {e.status_code}，重试中...")
                    time.sleep(wait)
            else:
                raise  # 4xx 客户端错误直接抛出
    raise last_error or RuntimeError("推理失败")


# ---------- 流式补全（用于 IDE 插件后端） ----------
def stream_completion(messages: list, **kwargs):
    """流式生成，逐 token 返回"""
    response = chat_completion(messages=messages, stream=True, **kwargs)
    for chunk in response:
        if chunk.choices and chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content


# ---------- 使用示例 ----------
if __name__ == "__main__":
    # 非流式调用
    resp = chat_completion([
        {"role": "user", "content": "用 Python 写一个快速排序"}
    ])
    print("非流式结果:", resp.choices[0].message.content[:100])

    # 流式调用
    print("流式结果:", end="")
    for token in stream_completion([
        {"role": "user", "content": "Java 单例模式怎么写"}
    ]):
        print(token, end="", flush=True)
    print()

cluster endpoint 可以是网关或直连实例；Python 客户端通过 base_url 切换。

先想 10 秒：100 个并发线程共用一个 OpenAI client，连接池会怎样？

httpx 默认 max_connections=10，100 并发时 90 个线程等待连接，延迟飙升。必须按 httpx 连接池 估算调大。生产中常用 max_connections=并发数*1.5。

多集群 Fallback

推理重试 (retry infer) 应区分「可重试」与「不可重试」错误：

错误类型可重试?策略
连接超时 / DNS 解析失败 是 换 endpoint 重试
429 Rate Limit 是 退避重试或 fallback 云 API
502 / 503 网关错误 是 退避 + 换实例
400 model not found 否 修配置
401 认证失败 否 修 api_key

Fallback 到云 API 实现

python 
# fallback 策略：私有化失败时切换到 DashScope 云 API
import os
from openai import OpenAI

# 私有化 client
private_client = OpenAI(
    base_url="http://vllm:8000/v1",
    api_key="EMPTY",
    timeout=120,
)

# 云 API client (DashScope)
cloud_client = OpenAI(
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key=os.environ["DASHSCOPE_API_KEY"],
)


def infer_with_fallback(messages: list, model: str = "corp-coder-7b"):
    """
    优先私有化，失败后回退云 API。
    云侧使用不同 model 名（qwen-turbo）。
    """
    for attempt, (client, model_name, label) in enumerate([
        (private_client, model, "私有化"),
        (cloud_client, "qwen-turbo", "云 API"),
    ]):
        try:
            resp = client.chat.completions.create(
                model=model_name,
                messages=messages,
                max_tokens=512,
                timeout=60,
            )
            print(f"[{label}] 推理成功")
            return resp.choices[0].message.content
        except Exception as e:
            print(f"[{label}] 失败: {e}")
            if attempt == 1:
                raise  # 云 API 也失败则真的失败
    return None

常见问题（FAQ）

OpenAI client 的 base_url 为什么必须带 /v1？
因为 OpenAI SDK 拼路径时会在 base_url 后面追加 /chat/completions。如果 base_url 设为 http://vllm:8000，最终请求变成 http://vllm:8000/chat/completions（少了一级），vLLM 无法路由。正确写法：http://vllm:8000/v1。
连接池耗尽会怎样？
HTTPX 的连接池有上限。耗尽时新请求会等待空闲连接，导致延迟飙升。现象：请求数增加但 QPS 不变甚至下降。解决方法：调大 max_connections，或使用异步客户端 AsyncOpenAI。
客户端重试和 vLLM 的去重怎么配合？
vLLM 本身不提供幂等去重。如果业务需要幂等（如多次重试不产生重复扣费），应在请求中加入 X-Request-Id header 并在业务层去重。简单场景下最多 2 次重试 + 退避即可。
为什么我设置了 timeout 但请求还是卡住？
可能是流式场景下 read timeout 未被正确传递。OpenAI SDK 流式模式下 read timeout 作用于每个 chunk 之间，若 vLLM 生成长文本但客户端 read timeout 过短，可能在中途断开。建议调大 read timeout 到 120s+。

本章结论

  • 客户端配置：OpenAI 客户端 指向 vLLM 需正确 base_url、超时与连接池。
  • 负载均衡：负载均衡 推荐网关方案；客户端直连用于开发测试。
  • 超时：推理超时 按场景配置，流式需更大的 read timeout。
  • 重试：重试策略 区分可重试/不可重试错误，指数退避最多 2 次。
  • Fallback：私有化失败时切换到云 API，确保服务不中断。

判断练习

  1. 你的 Python 推理客户端报 Connection pool is full，但 vLLM 负载很低。问题出在哪？

    参考答案

    客户端 HTTPX 连接池 max_connections 太小，请求在客户端排队而非 vLLM。调大上限或使用异步 client。

  2. 流式补全在 30 秒后中断，但同样的非流式请求 60 秒能完成。最可能的原因？

    参考答案

    流式模式下的 read timeout 设置过短。非流式请求的统一超时可能恰好更长，或者流式场景中每个 SSE chunk 之间的间隔超过了 timeout。
