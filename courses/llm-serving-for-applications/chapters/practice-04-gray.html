为什么要学本章

CorpAssist S4 将 qwen-7b-awq 升级到 v2 时，不能一次性切流。infer gray 让 Python 补全服务与 Spring BFF 共用稳定的 model alias，由网关按 route weight 把少量流量打到新 vLLM 实例，指标劣化再 version switch 回滚。

学完你能
  • 设计 header / 比例两种 canary infer 路由
  • 列出 TTFT、错误率、补全采纳率三类回滚触发条件
  • 配置 Spring AI 端的 model alias 实现零代码版本切换
本章先记住 3 件事
  • 客户端不动 model 名：双栈只认 corp-coder-7b alias，版本差异在路由层。
  • 灰度在网关或 sidecar：勿在业务代码里 if random 切 URL（难观测、难回滚）。
  • 回滚要快：配置改 canary_percent: 0 优于重新发版应用。

模型版本路由策略

CorpAssist S4 · AWQ v2 上线 平台组发布 vllm-v2（新量化权重），研发侧仍调用 model: corp-coder-7b。网关 5% 流量走 v2，其余走 v1；内部评测用 X-Infer-Canary: 1 强制金丝雀。

Model version route 在推理入口完成：Spring 远程 ChatModel 与 Python OpenAI client 的 base_url 指向统一网关，由网关解析后端 vllm-v1:8000 / vllm-v2:8000。

灰度路由配置

yaml 
# demos/practice-04-gray-lab/gray-route.yaml
model_alias: corp-coder-7b
default_backend: vllm-v1:8000
canary_backend: vllm-v2:8000
canary_percent: 5          # 从 5% 开始

routing:
  strategies:
    - type: weight
      weight_canary: 5     # 5% 流量走 v2
      weight_default: 95   # 95% 走 v1
    - type: header
      header: X-Infer-Canary
      value: "1"          # 设置此 header 强制走金丝雀

rollback:
  triggers:
    - metric: p95_latency_ms
      threshold: 3000
      window: 5m
    - metric: error_rate
      threshold: 0.02
      window: 5m
    - metric: empty_completion_rate
      threshold: 0.05
      window: 5m
  action: set_canary_percent_to_0
策略适用CorpAssist 注意
route weight（比例） 生产默认 从 5% → 20% → 50% 阶梯；每步观察 24h
Header 强制金丝雀 内测 / 自动化对比 勿对普通用户默认带 header
租户 / 部门路由 大客单独版本 与 ab infer 区分：AB 要固定 cohort
反模式
  • Spring 配置两套 base-url，业务 if 切换
  • 灰度直接改 served-model-name，客户端跟着改
  • 无 trace：无法区分 v1/v2 延迟
  • 灰度期间不监控 S4 空补全率
推荐
  • 单 alias + 网关路由
  • 日志带 backend_version=v2 标签
  • 与 流式监控 的 TTFT 联动
  • 灰度期间双跑 A/B 对比

Spring AI 端灰度配置

Spring AI 通过 model 字段指向 alias，不感知后端版本：

java 
// Spring AI 灰度配置 — 客户端只看 model alias
import org.springframework.ai.openai.OpenAiChatModel;
import org.springframework.ai.openai.OpenAiChatOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class GrayReleaseConfig {

    /**
     * 客户端只认 model alias "corp-coder-7b"
     * 版本切换在网关层，此处不需要改任何代码
     */
    @Bean
    public OpenAiChatModel grayReadyChatModel() {
        return OpenAiChatModel.builder()
            .baseUrl("http://infer-gateway:8000/v1")  // 指向网关而非直连 vLLM
            .apiKey("EMPTY")
            .defaultOptions(OpenAiChatOptions.builder()
                .model("corp-coder-7b")   // alias
                .temperature(0.1)
                .maxTokens(512)
                .build())
            .build();
    }
}

// 灰度请求追踪：强制走金丝雀路径
@Service
public class CanaryService {

    private final ChatClient chatClient;

    public String chatWithCanary(String message, boolean forceCanary) {
        var spec = chatClient.prompt().user(message);
        if (forceCanary) {
            // 内部用户/自动化评测走金丝雀路径
            spec = spec
                .system(s -> s.param("X-Infer-Canary", "1"));
        }
        return spec.call().content();
    }
}

    model="corp-coder-7b",
    messages=[{"role": "user", "content": "写一个二分查找"}],
)

回滚与观测

Rollback model 是运维动作：当 canary infer 指标越过阈值，将 canary_percent 置 0 或切默认 backend，必要时配合 version switch 冻结发布。

灰度放量计划

阶段金丝雀比例观察窗口回滚条件
Phase 1 5% 24h P95 延迟 > 3s 或错误率 > 2%
Phase 2 20% 48h 空补全率 > 5% 或用户反馈负面
Phase 3 50% 24h 与 v1 的 A/B 对比劣化
全量 100% — 保留 v1 实例 72h 用于快速回滚

A/B 对比脚本

python 
# A/B 对比 — 固定 prompt 集，分别请求 v1 和 v2
import json, time
from openai import OpenAI

v1_client = OpenAI(base_url="http://vllm-v1:8000/v1", api_key="EMPTY")
v2_client = OpenAI(base_url="http://vllm-v2:8000/v1", api_key="EMPTY")

GOLDEN_PROMPTS = [
    "def fibonacci(n):",
    "public class Singleton {",
    "git merge 和 rebase 的区别",
    "SELECT * FROM users WHERE",
]

def ab_compare():
    results = {"v1": [], "v2": []}
    for prompt in GOLDEN_PROMPTS:
        for label, client in [("v1", v1_client), ("v2", v2_client)]:
            t0 = time.time()
            resp = client.chat.completions.create(
                model="corp-coder-7b",  # 两端都用 alias
                messages=[{"role": "user", "content": prompt}],
                max_tokens=128,
            )
            elapsed = time.time() - t0
            results[label].append({
                "prompt": prompt,
                "latency": elapsed,
                "output_length": len(resp.choices[0].message.content),
            })
    # 汇总对比
    v1_avg = sum(r["latency"] for r in results["v1"]) / len(results["v1"])
    v2_avg = sum(r["latency"] for r in results["v2"]) / len(results["v2"])
    print(f"v1 平均延迟: {v1_avg*1000:.0f}ms")
    print(f"v2 平均延迟: {v2_avg*1000:.0f}ms")
    print(f"差异: {(v2_avg-v1_avg)/v1_avg*100:+.1f}%")

if __name__ == "__main__":
    ab_compare()
FAQ：回滚 vLLM 版本要不要重启 Spring？

若客户端始终连网关且 alias 不变，不必重启应用；仅当 Spring 缓存了错误 endpoint 或本地 fallback 指向 v2 时才需刷新配置。

常见问题（FAQ）

灰度期间发现 v2 延迟高但错误率正常，应该回滚吗？
不一定立即全量回滚。先确认是否仅 v2 cohort 受影响。如果是量化导致的固有问题，降 canary_percent 至 0 并修复后重新灰度。如果延迟超过 SLA 则立即回滚。
没有网关层如何做灰度？
可以使用 Spring Cloud LoadBalancer 或客户端侧随机路由，但不推荐。更好的方案是使用 K8s Ingress 的金丝雀发布（nginx.ingress.kubernetes.io/canary-weight）或 Service Mesh 的流量分割。
灰度模型和当前模型的输出格式不一致怎么办？
这是灰度前必须验证的。在 Golden Set 上跑输出格式校验。如果新模型的输出格式有变化（如 JSON schema 不同），需要先升级消费者代码再灰度模型。
A/B 对比和灰度发布的区别？
A/B 是实验（固定 cohort 长时间对比），灰度为发布（逐步放量最终全量）。A/B 需要统计显著性，灰度主要看 SLA 回滚条件。CorpAssist 场景中，灰度前先用 A/B 做离线对比。

本章结论

  • 路由：model alias + 网关比例/header 灰度，双栈零改 model 字段。
  • 回滚：rollback model 延迟/错误率/业务指标触发，优先 canary_percent: 0。
  • 观测：请求必须带 version switch 标签，才能做 ab infer 与复盘。

判断练习

  1. 灰度第二天 P95 延迟升 40%，错误率正常。是否应立即 rollback？

    参考答案

    不一定立即全量回滚：先确认是否仅 v2 cohort、是否 TTFT 配置变化；若超 SLA 且影响用户则降 canary_percent 或回滚；保留 ab 样本对比 v1/v2。

  2. 团队在 Spring 里新增 vllmV2BaseUrl 字段按用户 ID 切换。这种 model version route 有何风险？

    参考答案

    逻辑散落、难统一回滚、观测标签不一致；应上移到网关并保留 model alias。
