为什么要学本章

7B 模型 FP16 权重约 14GB，24GB 单卡放不下权重 + KV cache + 并发槽位。INT4 量化 可将权重压至 5～6GB，但精度有代价。CorpAssist S4 场景下，代码补全对量化敏感度低于通用对话——你需要掌握 量化选型 方法，而非深入算子实现。

学完你能
  • 用决策树为 S4 与运营文案场景选择 AWQ / GPTQ / INT8
  • 估算量化后显存占用并判断 24GB 卡是否够用
  • 设计一条「量化评测流水线」：Golden Set → 推理 → 对比得分
本章先记住 3 件事
  • 量化减显存：memory save 主要来自权重精度降低，KV cache 仍需单独估算。
  • 精度与质量：quality loss 随任务而异；代码补全 INT4 通常可接受，创意写作更敏感。
  • 硬件匹配：hardware match 决定量化格式——AWQ 对 NVIDIA 各代架构支持最广。

INT4/8 精度与显存

精度-显存权衡 (precision trade) 是量化选型的核心。下表对比常见精度级别在 7B 模型上的大致表现（以实测为准）：

精度权重体积显存节省典型质量损失推荐场景
FP16 ~14 GB 基线 无 硬件充裕的基准评测
INT8 ~7 GB 50% 极微，通常 < 1% 高精度敏感任务
INT4 (AWQ/GPTQ) ~4–6 GB 60-70% 可接受，代码 < 3% CorpAssist S4 主力

AWQ 与 GPTQ 选型

AWQ 和 GPTQ 是当前最主流的 INT4 后训练量化方法。vLLM 对两者均原生支持，但选型有细微差别：

维度AWQGPTQ
量化原理 按激活值分布保护重要通道 基于 Hessian 矩阵的梯度补偿
vLLM 支持 原生 --quantization awq 原生 --quantization gptq
推理速度 略快（有 kernel 融合优化） 与 AWQ 接近
校准数据 无需，自动 需少量校准样本
CorpAssist 推荐 首选 备选（已有 GPTQ 权重时）

使用 AutoAWQ 量化模型

python 
# 量化脚本：将 HuggingFace 模型转为 AWQ INT4
# pip install autoawq
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "Qwen/Qwen2.5-Coder-7B-Instruct"
quant_path = "./models/Qwen2.5-Coder-7B-Instruct-AWQ"
quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,  # INT4
    "version": "GEMM"
}

model = AutoAWQForCausalLM.from_pretrained(
    model_path, device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained(model_path)

# 提供少量校准样本（约 128 条即可）
calib_data = [
    "def fibonacci(n):\n    \"\"\"计算斐波那契数列\"\"\"\n",
    "public class Singleton {\n    private static Singleton instance;\n",
    "SELECT * FROM users WHERE id = ?",
]  # 实际使用 128+ 条

model.quantize(tokenizer, quant_config=quant_config, calib_data=calib_data)
model.save_pretrained(quant_path, safetensors=True)
tokenizer.save_pretrained(quant_path)
print(f"AWQ 量化完成，权重保存至 {quant_path}")
# 输出：权重从 ~14GB 降至 ~5GB

硬件匹配

硬件与量化格式的匹配 (hardware match) 影响推理速度与兼容性：

GPU 架构推荐量化注意
NVIDIA A100 (Ampere) AWQ INT4 / INT8 INT8 tensor core 有加速；AWQ 权重 4bit 有 kernel
NVIDIA A10 / RTX 4090 (Ada) AWQ INT4 24GB 主打；AWQ + 7B 是最佳配合
NVIDIA V100 (Volta) INT8 或 FP16 INT4 kernel 支持有限，优先 INT8
昇腾 / 寒武纪 需厂商确认 NPU 量化格式与 CUDA 不同
python 
# 量化后质量评测脚本（Golden Set）
from openai import OpenAI
import json

client = OpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY")

# S4 代码补全 Golden Set（固定 10 条评测 prompt）
golden_set = [
    {"prompt": "def quick_sort(arr):", "expected_pattern": "return"},
    {"prompt": "public class Hello {", "expected_pattern": "System.out"},
    {"prompt": "SELECT * FROM users WHERE", "expected_pattern": "="},
]

def eval_quality():
    results = []
    for item in golden_set:
        resp = client.completions.create(
            model="corp-coder-7b",
            prompt=item["prompt"],
            max_tokens=64
        )
        completion = resp.choices[0].text
        # 简单质量检查：是否包含期望关键词
        passed = item["expected_pattern"] in completion
        results.append({
            "prompt": item["prompt"],
            "completion": completion[:80],
            "passed": passed
        })
    passed_rate = sum(r["passed"] for r in results) / len(results)
    print(f"Golden Set 通过率: {passed_rate:.0%}")
    return results

if __name__ == "__main__":
    eval_quality()

    @Primary

            .apiKey("EMPTY")
            .defaultOptions(OpenAiChatOptions.builder()

                .temperature(0.1)
                .build())
            .build();

    public ChatClient quantChatClient(OpenAiChatModel model) {
        return ChatClient.builder(model).build();
    }

CorpAssist S4 · 量化部署示例

团队采购 2 台 A10 24GB 跑 S4 代码补全。决策过程：

  1. 7B Full FP16 → 14GB，加上 KV cache 8K/8 并发 ≈ 6GB，剩余 4GB 紧张 → 必须量化。
  2. INT8 → 7GB 权重 + 6GB KV ≈ 13GB，剩余 11GB → 可选。
  3. AWQ INT4 → 5GB 权重 + 6GB KV ≈ 11GB，剩余 13GB → 最优性价比。
  4. Golden Set 评测：AWQ 通过率 97%（vs FP16 99%），接受。

最终采用 Qwen2.5-Coder-7B-Instruct-AWQ，单卡支撑 15 并发，TTFT < 500ms。

常见误区
  • 量化万能论— 量化后仍要留足 KV cache 余量，max-model-len 过大一样 OOM。
  • 只看权重体积— 显存大头可能是 KV cache 而非权重，尤其长上下文场景。
  • 量化不评测— 不同任务对精度敏感度天差地别，必跑 Golden Set。
  • 忽略 kernel 兼容— 老旧 GPU 可能没有 INT4 kernel，会退化为 CPU 计算。

常见问题（FAQ）

AWQ 和 GPTQ 可以混合使用吗？
同一 vLLM 实例只能加载一种量化格式。混合使用需要通过网关路由到不同后端实例。CorpAssist 场景建议统一为 AWQ，减少运维复杂度。
量化后的模型还能微调吗？
理论上 INT4 模型可以做 QLoRA，但 CorpAssist 实践中建议：先全精度微调 → 再量化部署，而非量化后微调。量化后微调的质量损失更难评估。
INT4 量化对 FIM（Fill-in-Middle）场景影响大吗？
实证表明影响较小。代码补全对数值精度不敏感，只要校准数据包含 FIM 格式的 prompt。建议在校准集中加入 <fim_prefix>/<fim_suffix> 标记。
没有 GPU 能不能跑量化？
CPU 推理可以用 llama.cpp 的 GGUF 量化格式（详见 TGI 与其他引擎），但吞吐远低于 GPU。CorpAssist 生产场景不建议 CPU 量化推理。

本章结论

  • INT4/8 量化：权重体积减半，显存释放给 KV cache 与并发槽位；memory save 是核心收益。
  • AWQ vs GPTQ：AWQ 校准简单、vLLM 原生、推理略快，是 S4 首选；awq gptq 对比需结合实际硬件。
  • 精度评估：quality loss 用 Golden Set 量化，不可凭感觉；代码补全 INT4 通常可接受。
  • 硬件匹配：hardware match 决定 kernel 可用性；旧架构优先 INT8 而非 INT4。
  • 评测流水线：固定 prompt → 多精度推理 → 对比得分，纳入 CI 防止回归。

判断练习

  1. 你的 AWQ 量化模型在通用对话上掉点 5%，但代码补全通过率 98%。能否上线 S4？

    参考答案

    可以。S4 场景是代码补全而非通用对话，代码指标通过即可。但需在 release note 注明量化适用范围，避免被误用于通用问答场景。
