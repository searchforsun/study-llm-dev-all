vLLM 启动

vLLM server 提供了 OpenAI 兼容的 HTTP API。启动方式有两种：直接使用 Python CLI 或通过 Docker 容器。

vLLM 请求处理路径
sequenceDiagram
          participant C as Client
          participant A as OpenAI API 层
          participant E as vLLM Engine
          participant G as GPU
          C->>A: POST /v1/chat/completions
          A->>E: 调度 batch
          E->>G: forward + PagedAttention
          G-->>E: logits
          E-->>A: stream tokens
          A-->>C: SSE chunks

应用工程师关注 API 层契约与超时；内核细节见 KV 缓存与性能。

CorpAssist · 启动 vLLM 推理服务 运维拿到 AWQ 量化权重，需要在一台 24GB GPU 上启动 vLLM 供双栈调用。要求：模型名用 corp-coder-7b，上下文 8K，启用 prefix caching 加速相同 system prompt。
bash 
# 方式一：直接 CLI 启动（推荐开发调试）
python -m vllm.entrypoints.openai.api_server \
  --model /models/Qwen2.5-Coder-7B-Instruct-AWQ \
  --quantization awq \
  --served-model-name corp-coder-7b \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.9 \
  --enable-prefix-caching \
  --host 0.0.0.0 \
  --port 8000

# 方式二：Docker 启动（推荐生产）
docker run --gpus all \
  -v /models:/models:ro \
  -p 8000:8000 \
  --shm-size 2gb \
  vllm/vllm-openai:latest \
  --model /models/Qwen2.5-Coder-7B-Instruct-AWQ \
  --quantization awq \
  --served-model-name corp-coder-7b \
  --max-model-len 8192

验证启动成功

bash 
# 健康检查：列出模型
curl -s http://localhost:8000/v1/models | jq .
# 期望输出：{"data":[{"id":"corp-coder-7b","object":"model",...}]}

# 确认 model id
curl -s http://localhost:8000/v1/models | jq .data[0].id
# "corp-coder-7b"  # 必须与 Spring/Python 的 model 字段一致
不推荐

  • 跳过健康检查直接对接客户端
  • 用 root 用户运行容器

先想 10 秒：为什么 Python CLI 启动时 --model 参数要用绝对路径？

vLLM 需要准确知道权重文件位置。相对路径可能因工作目录不同而解析失败。Docker 场景下权重挂载在固定路径如 /models/...。

核心参数详解

参数作用S4 提示
--model 权重路径或 HuggingFace 模型名 本地路径优先，避免启动时下载
--served-model-name API 中暴露的模型名 客户端 model 字段必须一致
--max-model-len 单序列最大 token（prompt+生成） FIM 上下文 8K 常见；过大吃 KV 显存
--gpu-memory-utilization 预占显存比例 0.85～0.92，留余量防 OOM
--tensor-parallel-size 多卡切分权重 14B+ 单卡放不下时启用
--enable-prefix-caching 相同 system prompt 复用 KV 多用户共享代码规范 prompt 时有效
--quantization 量化格式（awq/gptq） 与权重文件格式匹配

PagedAttention 是 vLLM 的核心技术：它将 KV cache 分页管理，按需分配显存，避免传统方法中的碎片和预分配浪费。对应用层的直接影响：同等显存下可支撑更多并发序列。

Tensor Parallel 将模型权重切分到多张 GPU。14B 模型单张 24GB 卡放不下时，设置 --tensor-parallel-size 2 跨两张卡推理。

bash 
# 14B 模型 + 张量并行 2 卡
python -m vllm.entrypoints.openai.api_server \
  --model /models/Qwen2.5-Coder-14B-Instruct-AWQ \
  --quantization awq \
  --tensor-parallel-size 2 \
  --served-model-name corp-coder-14b \
  --gpu-memory-utilization 0.9
先想 10 秒：把 max-model-len 从 8K 调到 32K，QPS 不变，显存会怎样？

KV cache 上限随序列长度线性增，并发槽位减少，可能吞吐下降或 OOM；需量化或减并发。

OpenAI 兼容 API

客户端只需切换 base_url 与 api_key（vLLM 常填 EMPTY）。支持的核心端点：

Chat Completions

非流式
bash 
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "corp-coder-7b",
    "messages": [{"role":"user","content":"解释 quicksort"}],
    "max_tokens": 256,
    "temperature": 0.1
  }'
流式
bash 
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "corp-coder-7b",
    "messages": [{"role":"user","content":"解释 quicksort"}],
    "max_tokens": 256,
    "stream": true
  }'

FIM Completions（S4 补全核心）

bash 
# FIM（Fill-in-the-Middle）补全 — S4 IDE 插件的核心 API
curl http://localhost:8000/v1/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "corp-coder-7b",
    "prompt": "<|fim_prefix|>def hello():\n    print(\"<|fim_suffix|>\")<|fim_middle|>",
    "max_tokens": 64,
    "temperature": 0.1
  }'

Python 调用 vLLM

python 
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="EMPTY",
)

# FIM 补全（S4 场景）
fim_resp = client.completions.create(
    model="corp-coder-7b",
    prompt="<|fim_prefix|>def add(a, b):<|fim_suffix|><|fim_middle|>",
    max_tokens=64,
    temperature=0.1,
)
print("FIM:", fim_resp.choices[0].text)

# Chat
chat_resp = client.chat.completions.create(
    model="corp-coder-7b",
    messages=[{"role": "user", "content": "Python 列表推导式示例"}],
)
print("Chat:", chat_resp.choices[0].message.content)

Spring AI 调用 vLLM

java 
// Spring AI 配置指向 vLLM
import org.springframework.ai.autoconfigure.openai.OpenAiAutoConfiguration;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.openai.OpenAiChatModel;
import org.springframework.ai.openai.OpenAiChatOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class VllmClientConfig {

    @Bean
    public OpenAiChatModel vllmChatModel() {
        return OpenAiChatModel.builder()
            .baseUrl("http://vllm:8000/v1")
            .apiKey("EMPTY")
            .defaultOptions(OpenAiChatOptions.builder()
                .model("corp-coder-7b")
                .temperature(0.1)
                .maxTokens(512)
                .build())
            .build();
    }

    @Bean
    public ChatClient vllmChatClient(OpenAiChatModel model) {
        return ChatClient.builder(model)
            .defaultSystem("你是 CorpAssist 代码助手，帮助开发者完成代码。")
            .build();
    }
}
常见误区
  • model 名不一致 — 客户端写 Qwen2.5-7B 但 served 名是 corp-coder-7b → 404。
  • 忽略 stream 超时 — 网关 read timeout 必须大于最长生成时间。
  • 遗忘 /v1 路径 — base_url 不加 /v1 导致请求路径错误。
  • FIM 格式错误 — 不同模型使用不同的 FIM token 格式（如 <|fim_prefix|> 与 <fim_prefix>）。

常见问题（FAQ）

vLLM 启动后 /v1/models 返回空列表，可能是什么原因？
可能是权重路径错误或加载失败。检查 vLLM 启动日志中是否有模型加载错误。运行 docker logs 或 CLI 控制台输出。常见原因：量化格式不匹配（指定 --quantization awq但权重是 GPTQ）。
Served model name 设置后可以修改吗？
需要重启 vLLM。运行时不能修改。如果有多个模型名需求，可以启动多个 vLLM 实例或使用 --served-model-name 指定一个稳定别名，客户端统一用这个别名。
vLLM 支持哪些量化格式？
vLLM 原生支持 AWQ、GPTQ、SqueezeLLM、FP8（H100 适用）。加载时需要 --quantization 参数指定格式。详见 量化选型。
vLLM 和 Ollama 用同样的模型，性能差多少？
vLLM 的 PagedAttention 和 continuous batching 在高并发场景吞吐远超 Ollama。CorpAssist 生产场景用 vLLM，开发者本机 Demo 可用 Ollama。详见 TGI 与其他引擎。

本章结论

  • 启动：vLLM start 通过 CLI 或 Docker；OpenAI API server 模块 + 明确 served-model-name。
  • 参数：max-model-len 与 gpu-memory-utilization 决定 S4 并发上限。
  • OpenAI 兼容：OpenAI compatible 确保 chat 与 FIM completions 共用同一 base_url。

动手练习

启动 vLLM 并分别验证 models 列表与非 stream / stream 补全各一次。

操作步骤

  1. 启动 vLLM，设置 --served-model-name corp-coder-7b。
  2. 用 curl 验证 /v1/models 返回正确的模型 ID。
  3. 分别发送非流式和流式 chat completions 请求。
  4. 用 Python SDK 调用 FIM completions 端点。
  5. 执行 demos/basics-03-vllm-lab/curl-test.sh 验证所有端点正常。

判断练习

  1. 找错：Spring 配置 model: Qwen2.5-Coder-7B，vLLM 启动参数 --served-model-name corp-coder-7b。现象会是什么？

    参考答案

    请求返回 model not found / 404。应统一为 corp-coder-7b 或使用 vLLM 返回的 id。

  2. vLLM 启动后显存占用比预期的 AWQ 7B（约 5GB）高很多，可能的原因？

    参考答案

    可能原因：① 未启用 --quantization awq 导致加载 FP16；② --gpu-memory-utilization 设置过高预占了余量；③ KV cache 预分配较多；④ 加载了错误的大模型版本。
