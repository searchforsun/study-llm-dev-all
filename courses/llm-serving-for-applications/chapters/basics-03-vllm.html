为什么要学本章

CorpAssist 双栈（Python / Spring AI）都假设OpenAI 兼容 HTTP。vLLM server 是私有化路径的默认引擎；不懂 --max-model-len 与 tensor parallel，S4 补全会要么 OOM 要么 latency 失控。

学完你能
  • 用 CLI 或 Docker vLLM start 拉起服务并设置 served model name
  • 解释 PagedAttention 对并发补全的意义（应用层直觉）
  • 用 curl / OpenAI SDK 调 API port 8000 完成 chat 与 stream
本章先记住 3 件事
  • 启动：python -m vllm.entrypoints.openai.api_server 或容器 entrypoint。
  • 参数：max-model-len、gpu-memory-utilization、量化格式与 TP 度。
  • 兼容：OpenAI compatible — /v1/chat/completions、/v1/completions（FIM）。

启动

bash 
python -m vllm.entrypoints.openai.api_server \
  --model /models/Qwen2.5-Coder-7B-Instruct-AWQ \
  --quantization awq \
  --served-model-name corp-coder-7b \
  --host 0.0.0.0 \
  --port 8000

健康检查：curl -s http://localhost:8000/v1/models | jq .data[0].id 应返回 corp-coder-7b。Spring AI 与 Python client 的 model 字段须与此一致。

vLLM 请求路径
sequenceDiagram
  participant C as Client
  participant A as OpenAI API 层
  participant E as vLLM Engine
  participant G as GPU
  C->>A: POST /v1/chat/completions
  A->>E: 调度 batch
  E->>G: forward + PagedAttention
  G-->>E: logits
  E-->>A: stream tokens
  A-->>C: SSE chunks

应用工程师关注 API 层契约与超时；内核细节见 KV 缓存与性能。

参数

参数作用S4 提示
--max-model-len / max model len 单序列最大 token（prompt+生成） FIM 上下文 8K 常见；过大吃 KV 显存
--gpu-memory-utilization 预占显存比例 0.85～0.92，留余量防 OOM
--tensor-parallel-size 多卡切分权重 14B+ 单卡放不下时启用
--enable-prefix-caching 相同 system prompt 复用 KV 多用户共享代码规范 prompt 时有效
先想 10 秒：把 max-model-len 从 8K 调到 32K，QPS 不变，显存会怎样？

KV cache 上限随序列长度线性增，并发槽位减少，可能吞吐下降或 OOM；需量化或减并发。

OpenAI 兼容

客户端只需切换 base_url 与 api_key（vLLM 常填 EMPTY）：

Chat Completions
bash 
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "corp-coder-7b",
    "messages": [{"role":"user","content":"解释 quicksort"}],
    "max_tokens": 256,
    "stream": true
  }'
FIM Completions（S4）
bash 
curl http://localhost:8000/v1/completions \
  -d '{
    "model": "corp-coder-7b",
    "prompt": "<fim_prefix>def foo():<fim_suffix><fim_middle>",
    "max_tokens": 64
  }'
常见误区
  • model 名不一致 — 客户端写 Qwen2.5-7B 但 served 名是 corp-coder-7b → 404。
  • 忽略 stream 超时 — 网关 read timeout 必须大于最长生成时间。

本章结论

  • 启动：OpenAI API server 模块 + 明确 served-model-name 与健康检查。
  • 参数：max-model-len 与 gpu-memory-utilization 决定 S4 并发上限。
  • OpenAI 兼容：chat 与 FIM completions 共用同一 base_url，双栈零改造接入。

操作步骤

  1. 修改正文启动命令中的 --served-model-name 为你本地模型别名，写出对应 curl /v1/models 期望 JSON 字段。
  2. 对照参数表：若 GPU 24GB、7B AWQ，写出你会设的 max-model-len 与理由。
  3. 用 stream=true 发一条 chat 请求，说明网关 read timeout 应如何设（相对 max_tokens）。
  4. （可选）执行 demos/basics-03-vllm-lab/curl-test.sh。

判断练习

  1. 找错：Spring 配置 model: Qwen2.5-Coder-7B，vLLM 启动参数 --served-model-name corp-coder-7b。现象会是什么？

    参考答案

    请求返回 model not found / 404。应统一为 corp-coder-7b 或使用 vLLM 返回的 id。
