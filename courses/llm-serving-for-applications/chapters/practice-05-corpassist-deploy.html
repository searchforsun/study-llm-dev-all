compose

CorpAssist S4 · 私有化交付包 客户机房一台 24GB GPU 服务器：运维只需 docker compose up -d，研发用统一 INFER_BASE_URL 做代码补全与对话。

Compose 骨架（与 basics-02 lab 对齐）

yaml 
# demos/practice-05-corpassist-deploy-lab/compose 思路（可对照 basics-02 compose-gpu.yaml）
services:
  vllm:
    image: vllm/vllm-openai:latest
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    volumes:
      - ./models:/models:ro
    command: >
      --model /models/Qwen2.5-Coder-7B-Instruct-AWQ
      --quantization awq
      --served-model-name corp-coder-7b
      --host 0.0.0.0 --port 8000
    ports:
      - "8000:8000"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

环境变量约定：INFER_BASE_URL=http://vllm:8000/v1 供 Python；Spring 使用 spring.ai.openai.base-url 同源。若启用 灰度网关，则 base_url 指向网关而非直连 vLLM。

验证

Compose verify 分三层：health check（进程存活）→ smoke test（API 契约）→ e2e infer（IDE → BFF → 推理）。

Smoke：chat completions

http 
### demos/practice-05-corpassist-deploy-lab/smoke.http
GET http://localhost:8000/v1/models

###
POST http://localhost:8000/v1/chat/completions
Content-Type: application/json

{
  "model": "corp-coder-7b",
  "messages": [{"role": "user", "content": "写一个 Java 单例"}],
  "max_tokens": 64,
  "stream": true
}
检查项命令 / 动作通过标准
vLLM healthy curl /health 或 compose healthcheck HTTP 200
双栈 client Python + Spring 各 1 次非 stream 200 且有 content
流式补全 stream=true SSE 有 data 行、TTFT < SLA
delivery-checklist（Lab）
  • vLLM healthy
  • 双栈 client 指向同一 corp-coder-7b
  • 流式补全通过（S4 路径）
先想 10 秒：health 200 但 chat 500，优先查什么？

模型路径/量化参数错误、GPU OOM、model 名与 served-model-name 不一致；health 只说明进程在听端口。

本章结论

  • compose：GPU + vLLM + healthcheck + 权重卷，一条命令交付。
  • 验证：models → chat stream → 双栈 e2e，对照 checklist。
  • capstone：与灰度、容量、K8s 章衔接，形成 S4 私有化交付叙事。

判断练习

  1. 客户要求「只交付镜像、不交付 compose」。你会丢失本章哪些 compose verify 能力？如何补救？

    参考答案

    丢失 GPU 绑定、卷挂载、healthcheck 的一键编排；需改成交付 Helm/K8s manifest 或安装脚本 + 同等 smoke 文档。

  2. 双栈 smoke 通过，但 IDE 补全仍超时。列出 e2e 路径上 3 个非 vLLM 的排查点。

    参考答案

    BFF 网关 timeout、IDE 插件 base_url、公司代理；以及流式缓冲未刷新（见 practice-03）。
