为什么要学本章

CorpAssist 课内默认 vLLM，但客户现场可能已有 HF TGI、边缘 Ollama 或 CPU llama.cpp。应用架构师要能讲清 TGI engine 与 vLLM 的 throughput compare，并给出 ecosystem fit 建议。

学完你能
  • 对比 vLLM / TGI / llama.cpp / Ollama 的部署形态与 API
  • 为 CorpAssist S4 GPU 集群与开发者本机 demo 各选一种引擎
  • 说明何时不应为了「换引擎」推翻 OpenAI 兼容层
本章先记住 3 件事
  • 生产 GPU 批服务：vLLM、TGI 是第一梯队。
  • 本机/边缘：Ollama、llama.cpp 更轻。
  • 双栈契约：优先 OpenAI compatible，减少 Spring/Python 分叉。

对比

引擎典型场景OpenAI APICorpAssist 备注
vLLM 多卡高并发私有化 是（chat + FIM） 课内默认；S4 FIM 路径成熟
TGI HF 生态、K8s 原生 兼容层 / 自有 schema 与 HF 模型卡、推理容器一体
llama.cpp CPU、嵌入式 可选 server 模式 不适合百并发 S4
Ollama 开发者本机 demo 部分兼容 lab 可写「边缘 demo 用 ollama」
选型误区
  • 只看 benchmark 峰值，忽略 FIM / stream 支持
  • 本机 Ollama 指标外推生产 QPS
  • 双栈为 TGI 单独写一套 SDK
推荐做法
  • 用 engine compare 表对齐 SLA 指标
  • 网关统一 OpenAI 入口，后端可换引擎
  • 固定 200 条 S4 prompt 做 throughput compare
引擎在栈中的位置
flowchart TB
  Clients[Python / Spring] --> API[OpenAI 兼容网关]
  API --> V[vLLM]
  API --> T[TGI]
  API --> O[Ollama 本机]
  V --> GPU[数据中心 GPU]
  O --> Laptop[开发者笔记本]

应用层只依赖 API 契约；引擎替换应局限在网关后端池。

- 边缘 demo / 本机：Ollama
- 客户已购 HF 推理套件：评估 TGI，网关保持 OpenAI 形状

选型

Infer engine select 决策树（应用架构师视角）：

  1. 是否要 S4 FIM + 高并发？→ 优先 vLLM（或验证 TGI 同等 API）。
  2. 是否 强依赖 HF 推理镜像与 K8s operator？→ hf tgi + ecosystem fit。
  3. 是否 仅本机 PoC？→ Ollama / llama.cpp，不进入生产清单。
先想 10 秒：客户说「我们标准化 TGI 了」，CorpAssist 还要改 Python/Spring 吗？

若 TGI 前仍有 OpenAI 兼容网关或 adapter，双栈可不改；否则需评估 chat/FIM 字段差异，避免两套客户端。

本章结论

  • 对比：vLLM/TGI 服务生产；Ollama/llama.cpp 面向边缘与本机。
  • 选型：以 OpenAI 契约 + S4 工作负载为轴，不唯吞吐榜单。
  • 落地：引擎变更放在网关后端，保护 deploy capstone 验收方式。
