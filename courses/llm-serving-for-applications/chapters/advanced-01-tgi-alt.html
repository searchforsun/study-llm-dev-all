为什么要学本章

CorpAssist 课内默认 vLLM，但客户现场可能已有 HF TGI、边缘 Ollama 或 CPU llama.cpp。应用架构师要能讲清 TGI engine 与 vLLM 的 throughput compare，并给出 ecosystem fit 建议。

学完你能
  • 对比 vLLM / TGI / llama.cpp / Ollama 的部署形态与 API
  • 为 CorpAssist S4 GPU 集群与开发者本机 demo 各选一种引擎
  • 说明何时不应为了「换引擎」推翻 OpenAI 兼容层
本章先记住 3 件事
  • 生产 GPU 批服务：vLLM、TGI 是第一梯队。
  • 本机/边缘：Ollama、llama.cpp 更轻。
  • 双栈契约：优先 OpenAI compatible，减少 Spring/Python 分叉。

  • 本机 Ollama 指标外推生产 QPS

  • 固定 200 条 S4 prompt 做 throughput compare
  • 优先验证 FIM 和流式的兼容性

Python 多引擎客户端对比

python 
# 多引擎吞吐对比脚本
import time
from openai import OpenAI

ENGINES = {
    "vLLM": "http://vllm:8000/v1",
    "TGI": "http://tgi:8080/v1",
    "Ollama": "http://localhost:11434/v1",
}

PROMPT = "def fibonacci(n):"
RESULTS = {}

for name, base_url in ENGINES.items():
    try:
        client = OpenAI(base_url=base_url, api_key="EMPTY")
        t0 = time.time()
        resp = client.completions.create(
            model="corp-coder-7b",
            prompt=PROMPT,
            max_tokens=64,
        )
        elapsed = time.time() - t0
        RESULTS[name] = {
            "latency": elapsed,
            "output": resp.choices[0].text[:50],
        }
        print(f"[{name}] {elapsed*1000:.0f}ms: {RESULTS[name]['output']}")
    except Exception as e:
        print(f"[{name}] 失败: {e}")

# 吞吐对比
for name, result in RESULTS.items():
    print(f"{name}: {result['latency']*1000:.0f}ms")

Spring AI 多引擎配置模式

java 
// Spring AI 配置 — 支持不同引擎后端
import org.springframework.ai.openai.OpenAiChatModel;
import org.springframework.ai.openai.OpenAiChatOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Primary;

@Configuration
public class MultiEngineConfig {

    @Bean
    @Primary
    public OpenAiChatModel defaultChatModel() {
        // 默认 vLLM
        return buildModel("http://vllm:8000/v1", "corp-coder-7b");
    }

    @Bean
    public OpenAiChatModel tgiChatModel() {
        // TGI 备选
        return buildModel("http://tgi:8080/v1", "corp-coder-7b");
    }

    @Bean
    public OpenAiChatModel ollamaChatModel() {
        // Ollama 本机开发
        return buildModel("http://localhost:11434/v1", "qwen2.5-coder:7b");
    }

    private OpenAiChatModel buildModel(String baseUrl, String model) {
        return OpenAiChatModel.builder()
            .baseUrl(baseUrl)
            .apiKey("EMPTY")
            .defaultOptions(OpenAiChatOptions.builder()
                .model(model)
                .temperature(0.1)
                .maxTokens(512)
                .build())
            .build();
    }
}

## CorpAssist 引擎选型结论
- 生产 GPU 集群：**vLLM** (默认首选)
- 边缘 demo / 本机：**Ollama**
- 客户已购 HF 推理套件：评估 **TGI**，网关保持 OpenAI 形状
常见误区
  • 吞吐决定一切 — API 兼容性、运维成熟度、社区支持更重要。
  • 引擎可以随意切换 — 换引擎意味着回归 FIM、流式、量化等全部路径。
  • Ollama 可以上生产 — Ollama 缺乏 continuous batching 和 PagedAttention，高并发明显弱于 vLLM。

常见问题（FAQ）

vLLM 和 TGI 的 FIM 兼容性一样吗？
不完全一样。vLLM 原生支持 FIM（/v1/completions）。TGI 也支持 FIM 但提示格式可能不同。如果客户使用 TGI，需要先在 Golden Set 上验证 FIM 输出格式是否符合预期。不兼容时可以通过适配层转换。
llama.cpp 的 GGUF 和 AWQ 哪个更好？
AWQ 用于 GPU 推理（vLLM 原生支持），GGUF 用于 CPU/边缘推理（llama.cpp）。两者不可直接比较。CorpAssist 生产用 AWQ，本机 PoC 用 GGUF。
为什么说 Ollama 不适合生产高并发？
Ollama 不实现 PagedAttention 等高级批处理机制，连续高并发时显存碎片化严重、吞吐下降明显。vLLM 在这些场景有 2-5x 吞吐优势。
可以混合使用多个引擎吗？比如 vLLM 做 FIM，TGI 做对话？
可以。在推理网关层按请求类型路由到不同后端。但运维复杂度倍增——需要维护两套引擎的镜像、监控和容量规划。建议单一引擎为主，特殊场景补充。

本章结论

  • 对比：vLLM/TGI 服务生产；Ollama/llama.cpp 面向边缘与本机。
  • 选型：以 OpenAI 契约 + S4 工作负载为轴，不唯吞吐榜单。
  • 落地：引擎变更放在网关后端，保护 deploy capstone 验收方式。

判断练习

  1. Benchmark 显示 TGI 峰值吞吐高 8%，是否应迁移生产？给出至少 2 个否决或暂缓理由。

    参考答案

    需验证 FIM、stream、量化路径与灰度运维成熟度；迁移成本包括双栈回归与灰度策略重做。

  2. 你在客户环境发现 TGI 不支持某种量化格式，怎么办？

    参考答案

    方案一：转换量化格式（AWQ→GPTQ），但需重新评测。方案二：在 TGI 前加 OpenAI 兼容适配层。方案三：部署 vLLM 作为补充引擎。
