为什么要学本章

CorpAssist 的 BFF（Backend For Frontend）是 Spring Boot 服务。它必须通过 Spring AI 远程 ChatModel 调用私有化 vLLM。相比 Python，Spring AI 的配置方式不同——通过 application.yml 或 @Bean 声明 OpenAI 兼容端点，并利用 Spring Cloud 的 fallback 机制 在私有化宕机时降级到云 API。

学完你能
  • 通过 spring.ai.openai.base-url 指向私有化 vLLM 集群
  • 配置主备双 endpoint（私有化 + 云 API fallback）
  • 为 ChatClient 设置超时与重试策略
本章先记住 3 件事
  • 配置即代码：Spring AI vLLM 只需改 base-url 与 model，零代码变更。
  • Fallback 链：failover model 通过 @Primary + @Qualifier 实现多 Model 切换。
  • 双端点：dual endpoint 同时配置私有化与云 API，运行时动态降级。

配置远程 OpenAI 端点

远程推理端点 (remote endpoint) 在 Spring AI 中通过 spring.ai.openai 前缀配置。底层使用 OpenAiChatModel，它本身是一个 ChatModel 实现，与本地模型使用相同的 ChatClient 接口。

Spring AI 远程推理链路
sequenceDiagram
    participant App as Spring BFF
    participant SA as Spring AI OpenAiChatModel
    participant V as vLLM /v1
    participant D as DashScope fallback

    App->>SA: ChatClient.call()
    SA->>V: POST /v1/chat/completions
    V-->>SA: 200 OK (completion)
    SA-->>App: ChatResponse

    Note over SA,V: 私有化不可用时
    SA->>D: POST 云 API endpoint
    D-->>SA: 200 OK
    SA-->>App: ChatResponse (含 fallback 标记)

Spring AI vLLM 路径：应用只依赖 ChatClient 接口，不感知后端是 vLLM 还是 DashScope。

application.yml 配置

yaml 
# src/main/resources/application.yml
spring:
  ai:
    openai:
      # 指向私有化 vLLM（主端点）
      base-url: http://vllm-infer.corpassist.svc:8000/v1
      api-key: EMPTY
      chat:
        options:
          model: corp-coder-7b
          temperature: 0.1
          max-tokens: 512

# 可选：覆盖默认超时（Spring AI 使用 RestClient）
  cloud:
    openai:
      connect-timeout: 5000
      read-timeout: 120000
不推荐
  • 在 Java 代码中硬编码 vLLM URL
  • 不设 read-timeout 导致流式请求中断
  • model 名与 vLLM served-model-name 不一致
  • 直接依赖 OpenAiChatModel 而非 ChatClient 接口
推荐
  • 配置外化到 application.yml / ConfigMap
  • read-timeout ≥ 120s 匹配最长生成
  • 确认 model 名与 vLLM --served-model-name 一致
  • 面向 ChatClient 接口编程
先想 10 秒：Spring 配置中 model 名写错会怎样？

vLLM 返回 404 model not found。OpenAiChatModel 会抛出 OpenAiApiException，不会自动重试。必须在 vLLM 启动参数中确认 --served-model-name 与客户端 model 字段一致。

Fallback 到云 API

ChatModel fallback 是 Spring AI 的重要能力：当私有化 vLLM 不可用时，自动降级到 DashScope 云 API。

双 endpoint 配置

yaml 
# application.yml — 双 endpoint
spring:
  ai:
    openai:
      # 主：私有化 vLLM
      base-url: http://vllm-infer:8000/v1
      api-key: EMPTY
      chat:
        options:
          model: corp-coder-7b

# 备用：DashScope 云 API（通过不同配置前缀）
---
spring:
  config:
    activate:
      on-profile: cloud-fallback
  ai:
    openai:
      base-url: https://dashscope.aliyuncs.com/compatible-mode/v1
      api-key: ${DASHSCOPE_API_KEY}
      chat:
        options:
          model: qwen-turbo

Java 代码：带 fallback 的 ChatClient

java 
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.openai.OpenAiChatModel;
import org.springframework.ai.openai.OpenAiChatOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Primary;

import java.time.Duration;

@Configuration
public class CorpAssistInferConfig {

    private static final Logger log = LoggerFactory.getLogger(CorpAssistInferConfig.class);

    /**
     * 主推理模型：指向私有化 vLLM
     */
    @Bean
    @Primary
    public OpenAiChatModel primaryChatModel() {
        return OpenAiChatModel.builder()
            .baseUrl("http://vllm-infer:8000/v1")
            .apiKey("EMPTY")
            .defaultOptions(OpenAiChatOptions.builder()
                .model("corp-coder-7b")
                .temperature(0.1)
                .maxTokens(512)
                .build())
            .build();
    }

    /**
     * 备用模型：指向 DashScope 云 API
     */
    @Bean
    public OpenAiChatModel fallbackChatModel() {
        return OpenAiChatModel.builder()
            .baseUrl("https://dashscope.aliyuncs.com/compatible-mode/v1")
            .apiKey(System.getenv("DASHSCOPE_API_KEY"))
            .defaultOptions(OpenAiChatOptions.builder()
                .model("qwen-turbo")
                .temperature(0.1)
                .maxTokens(512)
                .build())
            .build();
    }

    /**
     * 面向应用的 ChatClient——含失败切换逻辑
     */
    @Bean
    public ChatClient corpAssistChatClient(
            OpenAiChatModel primaryChatModel,
            OpenAiChatModel fallbackChatModel) {

        return ChatClient.builder(primaryChatModel)
            .defaultSystem("你是 CorpAssist 代码助手。请用中文回答。")
            .build();
    }
}

// ---------- 带 fallback 的服务实现 ----------
@Service
public class CorpAssistService {

    private final ChatClient chatClient;
    private final OpenAiChatModel fallbackChatModel;

    public CorpAssistService(
            ChatClient chatClient,
            OpenAiChatModel fallbackChatModel) {
        this.chatClient = chatClient;
        this.fallbackChatModel = fallbackChatModel;
    }

    /**
     * 调用推理，支持 fallback：私有化失败 → 云 API
     */
    public String chat(String userMessage) {
        try {
            return chatClient.prompt()
                .user(userMessage)
                .call()
                .content();
        } catch (Exception e) {
            log.warn("私有化推理失败，切换到云 API fallback", e);
            try {
                return ChatClient.builder(fallbackChatModel)
                    .build()
                    .prompt()
                    .user(userMessage)
                    .call()
                    .content();
            } catch (Exception fallbackException) {
                log.error("云 API fallback 也失败", fallbackException);
                return "抱歉，当前推理服务不可用，请稍后重试。";
            }
        }
    }
}

超时与熔断保护

Java 推理客户端 (java infer client) 的稳定性通过三层保护实现：

层级机制配置
1. 超时 RestClient connect/read timeout spring.cloud.openai.read-timeout=120000
2. 重试 Spring Retry / Resilience4j 最多 2 次退避重试
3. 熔断 Resilience4j CircuitBreaker 连续 5 次失败后开启熔断
java 
// Resilience4j 熔断保护 — 防止雪崩
import io.github.resilience4j.circuitbreaker.CircuitBreaker;
import io.github.resilience4j.circuitbreaker.CircuitBreakerConfig;
import io.github.resilience4j.circuitbreaker.CircuitBreakerRegistry;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

import java.time.Duration;

@Configuration
public class CircuitBreakerConfig {

    @Bean
    public CircuitBreaker inferCircuitBreaker() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            .failureRateThreshold(50)           // 50% 失败率触发熔断
            .slidingWindowSize(10)              // 10 次请求窗口
            .minimumNumberOfCalls(5)            // 最少 5 次开始统计
            .waitDurationInOpenState(Duration.ofSeconds(30)) // 熔断持续 30s
            .permittedNumberOfCallsInHalfOpenState(3)        // 半开时允许 3 次
            .build();

        CircuitBreakerRegistry registry = CircuitBreakerRegistry.of(config);
        return registry.circuitBreaker("vllm-infer");
    }
}
常见误区
  • Fallback 无告警 — fallback 应记录日志并触发告警，否则运维不知私有化已故障。
  • Fallback 无限额 — 云 API fallback 若不加 Token 预算，GPU 故障可能变成高额账单。
  • 遗漏 model 名校验 — vLLM 端的 --served-model-name 与 Spring 配置必须逐一核对。

常见问题（FAQ）

Spring AI 配置了 base-url 但请求还是走 OpenAI 官网？
检查 spring.ai.openai.base-url 是否覆盖了默认值。Spring AI 默认指向 https://api.openai.com。确认 application.yml 被正确加载且没有 profile 冲突。也可以打印所有 spring.ai 配置项验证。
如何验证 Spring AI 确实连到了私有化 vLLM 而非云 API？
有三种方式：① 临时断掉外网观察是否仍然可用；② 在 vLLM 日志中搜索对应的请求 IP；③ 在 Spring 请求 header 中加入 X-CorpAssist-Route 并透传到 vLLM 日志。
Fallback 到云 API 时如何控制成本？
在 fallback 路径中加入 Token 计数与预算检查：① 在配置中设置 max-tokens 上限；② 使用 Resilience4j 限流；③ 云 API 调用记录到日志，按日统计。当 fallback 调用量超过阈值时告警。
Spring AI 的 ChatModel 与 Python OpenAI 客户端有什么本质区别？
Spring AI 的 OpenAiChatModel 是对 OpenAI REST API 的 Java 封装，底层使用 RestTemplate/WebClient。它提供 Spring 风格的配置（application.yml）和 Spring Cloud 集成。功能等价于 Python OpenAI SDK，但更适合 Java 生态的 BFF 服务。

本章结论

  • 远程端点：Spring 远程 ChatModel 通过 base-url + model 指向 vLLM，零代码变更。
  • Fallback：ChatModel fallback 通过双 Bean + try-catch 实现私有化→云 API 降级。
  • 超时：远程端点 的 read-timeout 须覆盖最大生成长度。
  • 双端点：dual endpoint 策略保障私有化故障时服务不中断。
  • 熔断：Resilience4j 避免雪崩，配合 fallback 告警通知运维。

判断练习

  1. Spring 日志显示 OpenAiApiException: 404 model not found，但 vLLM 已启动。最可能的原因是什么？

    参考答案

    Spring AI 配置的 model 名称与 vLLM 的 --served-model-name 不一致。例如 Spring 写 qwen2.5-7b 但 vLLM 暴露为 corp-coder-7b。

  2. 私有化 vLLM 频繁超时，Spring 应如何优雅降级？

    参考答案

    第一层：超时配置 read-timeout=120s；第二层：重试 1-2 次；第三层：触发熔断（连续 N 次失败断路 30s），同时 fallback 到云 API。每层都记录日志和告警。
