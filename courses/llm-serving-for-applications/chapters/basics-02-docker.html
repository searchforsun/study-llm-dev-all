容器化镜像

container image 封装 CUDA、PyTorch 与 vLLM 版本对齐。生产推荐 vllm/vllm-openai（随 vLLM 发版），开发机可用同标签保证与 CI 一致。模型权重不要 bake 进镜像（体积与合规），用 volume 或 init 容器拉取。

CUDA base 镜像提供 GPU 驱动与 CUDA runtime；vLLM 层封装推理引擎；权重通过 volume 挂载，不打包进镜像。

CorpAssist · 容器化交付 需要向客户机房交付一套 vLLM 推理服务。客户无网络环境，只能交付 docker 镜像 + 权重文件。方案：vllm/vllm-openai:latest 镜像 + 外挂 AWQ 权重目录，一条 docker compose up 启动。

最小 Compose 服务定义

yaml 
services:
  vllm:
    image: vllm/vllm-openai:latest
    command: >
      --model /models/Qwen2.5-Coder-7B-Instruct-AWQ
      --served-model-name corp-coder-7b
      --max-model-len 8192
      --gpu-memory-utilization 0.9
    volumes:
      - ./models:/models:ro
    ports:
      - "8000:8000"
    shm_size: "2gb"   # vLLM 多进程通信需要足够共享内存
不推荐
  • 自构建镜像却不锁 vLLM/CUDA 版本
  • 把 14B 全精度权重打进镜像层
  • shm_size 默认 64MB 导致 worker 崩溃
  • 容器未设 memory limit，内存泄漏拖垮节点
推荐
  • 镜像 tag 与 vLLM 使用 文档版本一致
  • AWQ/GPTQ 权重挂载只读
  • 健康检查打 /v1/models
  • shm_size ≥ 2GB 保障 IPC

GPU 调度入门

Docker Desktop / Linux 需安装 NVIDIA Container Toolkit，使 推理容器 (infer container) 内 nvidia-smi 可见设备。Compose v2.20+ 写法：

yaml 
services:
  vllm:
    image: vllm/vllm-openai:latest
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    environment:
      NVIDIA_VISIBLE_DEVICES: "0"   # 指定 GPU 索引
宿主机 GPU 到容器
flowchart LR
          Host[NVIDIA Driver + Toolkit]
          RT[Container Runtime]
          C1[vLLM 容器 GPU0]
          C2[vLLM 容器 GPU1]
          Host --> RT
          RT --> C1
          RT --> C2

NVIDIA runtime 是容器访问 GPU 的桥梁。K8s 场景改用 nvidia.com/gpu 资源请求，见 K8s 部署清单。

GPU 调度排障流程

症状排查步骤常见原因
容器内 nvidia-smi 报错 ① 宿主机 nvidia-smi 是否正常 ② nvidia-container-toolkit 是否安装 宿主机驱动异常或未装 toolkit
vLLM 启动报 CUDA error ① 容器内 CUDA 版本 ② 镜像 CUDA 版本匹配 CUDA 版本不兼容
Compose 提示 unknown runtime nvidia-container-runtime 是否注册到 Docker Docker daemon 未配置 runtime
性能差于裸机 检查 shm_size 是否够大 缺省 64MB shm 限速
先想 10 秒：两实例共一张 24G 卡各加载 7B FP16，最可能发生什么？

显存双倍占用，第二个实例启动失败或运行时 OOM；应量化、合并实例用 continuous batching，或一卡一 Pod。

资源 Limit

除 GPU 外，vLLM 仍吃 CPU（tokenizer、调度）与主机内存。resource limit 防止异常请求拖垮节点：

资源典型配置说明
GPU 显存 由模型+KV 占满 用 --gpu-memory-utilization 0.9 留余量
shm_size ≥ 2GB tensor parallel 时更大
CPU / 内存 limit 例如 8 CPU / 32Gi Compose mem_limit 或 K8s limits
ulimit nofile 65536 高并发 SSE 连接时调高

完整 Compose 文件

yaml 
version: "3.8"
services:
  vllm:
    image: vllm/vllm-openai:latest
    command: >
      --model /models/Qwen2.5-Coder-7B-Instruct-AWQ
      --quantization awq
      --served-model-name corp-coder-7b
      --max-model-len 8192
      --gpu-memory-utilization 0.9
      --enable-prefix-caching
      --host 0.0.0.0 --port 8000
    volumes:
      - ./models:/models:ro
    ports:
      - "8000:8000"
    shm_size: "2gb"
    mem_limit: 32g
    cpus: 8
    ulimits:
      nofile:
        soft: 65536
        hard: 65536
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/v1/models"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 120s  # 权重加载可能需较长时间

Python 验证容器状态

python 
import subprocess, json, requests

def verify_container():
    """验证 vLLM 容器状态"""
    # 1. 检查容器是否运行
    result = subprocess.run(
        ["docker", "ps", "--filter", "name=vllm", "--format", "{{.Status}}"],
        capture_output=True, text=True
    )
    print(f"容器状态: {result.stdout.strip()}")

    # 2. 检查容器内 GPU 可用
    gpu_check = subprocess.run(
        ["docker", "exec", "corpassist-vllm-1", "nvidia-smi",
         "--query-gpu=name,memory.used", "--format=csv,noheader"],
        capture_output=True, text=True
    )
    print(f"GPU: {gpu_check.stdout.strip()}")

    # 3. 检查 API 是否就绪
    try:
        resp = requests.get("http://localhost:8000/v1/models", timeout=5)
        models = resp.json()
        print(f"模型: {[m['id'] for m in models['data']]}")
    except Exception as e:
        print(f"API 未就绪: {e}")

if __name__ == "__main__":
    verify_container()

常见问题（FAQ）

Windows Docker Desktop 怎么跑 GPU 容器？
Docker Desktop 对 Windows 的 GPU 支持有限。生产场景建议在 Linux 服务器上运行 vLLM。Windows 开发环境可用 WSL2 + Docker WSL2 backend，或使用远程 SSH 连接到 Linux GPU 服务器。
为什么容器内 nvidia-smi 正常但 vLLM 报 CUDA error？
vLLM 镜像的 CUDA 版本与宿主机驱动不兼容。检查 nvidia-smi 输出的 CUDA version，选择匹配的 vLLM 镜像 tag（如 vllm/vllm-openai:cuda-12.1）。
Compose 和 K8s 部署 vLLM 有什么区别？
Compose 适合单机 PoC 和小规模私有化。K8s 提供副本管理、自动扩缩容、灰度发布。CorpAssist 生产环境建议 K8s，开发/测试用 Compose。详见 K8s 部署清单。
容器重启后模型权重需要重新加载吗？
是的。vLLM 在启动时加载权重到 GPU 显存。如果 volume 挂载的权重文件在本地，重启容器会重新加载（耗时 1-5 分钟）。建议配置 start_period 避免健康检查误杀。

本章结论

  • 镜像：官方 vLLM OpenAI 镜像 + 外挂权重；锁版本、放大 shm_size。
  • GPU 调度：NVIDIA runtime + 明确 count/capabilities；一卡多实例需算清显存。
  • 资源 limit：CPU/内存/句柄 limit 与 GPU 利用率参数一起设，避免「能启动但高并发崩溃」。

判断练习

  1. 排障：容器内 nvidia-smi 报错「Driver not loaded」。最应先检查哪一层？

    参考答案

    先查宿主机驱动与 nvidia-container-toolkit 是否安装；再查 Compose 是否声明 capabilities: [gpu]。不是先改 vLLM 参数。

  2. vLLM 容器启动后 healthcheck 一直失败，但 5 分钟后 curl 正常。问题出在哪里？

    参考答案

    模型权重加载需要时间。healthcheck 的 start_period 设置太短或未设置。应该设 start_period=120s 匹配权重加载时间。
