为什么要学本章

私有化 Serving 若直接在宿主机装 CUDA、Python、vLLM，环境不可复现且难以灰度。容器把 Docker LLM 运行时与权重挂载标准化，是 CorpAssist S4 推理集群的最小交付单元。

学完你能
  • 解释官方 vllm/vllm-openai 镜像与 CUDA base 的关系
  • 编写带 compose GPU 的 Compose 文件并声明 resource limit
  • 排查「容器内看不到 GPU」类问题（runtime / driver / 设备映射）
本章先记住 3 件事
  • 镜像：优先官方 OpenAI 入口镜像，权重用 volume 挂载。
  • GPU：需要 NVIDIA runtime + 正确 GPU 调度（Compose deploy.resources 或 K8s，见 advanced）。
  • 资源：显存由模型+KV 决定，CPU/内存 limit 防止 OOM 拖垮节点。

镜像

container image 封装 CUDA、PyTorch 与 vLLM 版本对齐。生产推荐 vllm/vllm-openai（随 vLLM 发版），开发机可用同标签保证与 CI 一致。模型权重不要 bake 进镜像（体积与合规），用 volume 或对象存储 init 容器拉取。

最小 Compose 服务定义

yaml 
services:
  vllm:
    image: vllm/vllm-openai:latest
    command: >
      --model /models/Qwen2.5-Coder-7B-Instruct-AWQ
      --served-model-name corp-coder-7b
      --max-model-len 8192
    volumes:
      - ./models:/models:ro
    ports:
      - "8000:8000"
    shm_size: "2gb"   # vLLM 多进程通信需要足够共享内存
不推荐
  • 自构建镜像却不锁 vLLM/CUDA 版本
  • 把 14B 全精度权重打进镜像层
  • shm_size 默认 64MB 导致 worker 崩溃
推荐
  • 镜像 tag 与 vLLM 使用 文档版本一致
  • AWQ/GPTQ 权重挂载只读
  • 健康检查打 /v1/models

GPU 调度入门

CorpAssist · 单卡 PoC（45 秒） 一台 24GB RTX 4090 跑 7B AWQ 代码模型，Compose 本地验证；后续同一镜像上 K8s。GPU 调度要确保容器独占或共享策略明确，避免两个 vLLM 抢一张卡 OOM。

Docker Desktop / Linux 需安装 NVIDIA Container Toolkit，使 infer container 内 nvidia-smi 可见设备。Compose v2 写法：

yaml 
services:
  vllm:
    image: vllm/vllm-openai:latest
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    environment:
      NVIDIA_VISIBLE_DEVICES: "0"
宿主机 GPU 到容器
flowchart LR
  Host[NVIDIA Driver]
  RT[Container Runtime]
  C1[vLLM 容器 GPU0]
  Host --> RT
  RT --> C1

K8s 场景改用 nvidia.com/gpu 资源请求，见 K8s 部署清单。

先想 10 秒：两实例共一张 24G 卡各加载 7B FP16，最可能发生什么？

显存双倍占用，第二个实例启动失败或运行时 OOM；应量化、合并实例用 continuous batching，或一卡一 Pod。

资源

除 GPU 外，vLLM 仍吃 CPU（tokenizer、调度）与主机内存。resource limit 防止异常请求拖垮节点：

资源典型配置说明
GPU 显存 由模型+KV 占满 用 --gpu-memory-utilization 0.9 留余量，见 KV 缓存与性能
shm_size ≥ 2GB tensor parallel 时更大
CPU / 内存 limit 例如 8 CPU / 32Gi Compose mem_limit 或 K8s limits
ulimit 文件句柄 高并发 SSE 连接时调高

带 limit 的完整片段

yaml 
services:
  vllm:
    mem_limit: 32g
    cpus: 8
    ulimits:
      nofile:
        soft: 65536
        hard: 65536

本章结论

  • 镜像：官方 vLLM OpenAI 镜像 + 外挂权重；锁版本、放大 shm_size。
  • GPU 调度：NVIDIA runtime + 明确 count/capabilities；一卡多实例需算清显存。
  • 资源：CPU/内存/句柄 limit 与 GPU 利用率参数一起设，避免「能启动但高并发崩溃」。

操作步骤

  1. 合并正文两段 yaml：为 vLLM 服务同时加上 deploy.resources.reservations.devices 与 shm_size: 2gb。
  2. 写出启动后验证 GPU 可见的命令：docker compose exec vllm nvidia-smi 与期望输出要点（能看到 GPU 名称）。
  3. 对照资源表，为 7B AWQ 估算是否适合 24GB 卡（权重约 5～6GB + KV 余量）。

判断练习

  1. 排障：容器内 nvidia-smi 报错「Driver not loaded」。最应先检查哪一层？

    参考答案

    先查宿主机驱动与 nvidia-container-toolkit 是否安装；再查 Compose 是否声明 capabilities: [gpu]。不是先改 vLLM 参数。
