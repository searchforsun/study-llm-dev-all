采购 GPU 前需要 rough sizing：CorpAssist S4 峰值 s4 load 多少、单卡能扛多少并发、要不要第二张卡。本章不做精确仿真，只建立 qps estimate 与 vram estimate 的可沟通算法。

  • 按模型大小 + 量化 + 上下文估显存是否够

本章先记住 3 件事
  • QPS 粗算：先估并发用户与单次补全耗时，再算所需 tokens/s。
  • 显存：权重 + KV；max-model-len 与并发数乘在一起。
  • 留余量：灰度、峰值、OOM 缓冲 20～30%。

QPS 粗算

Concurrency math 基于 Little 定律的直觉应用：

并发请求数 N ≈ 到达率 λ × 平均响应时间 W

CorpAssist · S4 容量估算

假设：500 活跃开发者，平均每 30 秒触发一次代码补全，单次补全生成耗时 3 秒。

并发 ≈ 500 × (1/30) × 3 = 50 并发请求。

若单卡 vLLM 稳定支撑 20 并发（7B AWQ 实测），则需 50/20 ≈ 3 张 GPU。

考虑 25% 余量 → 建议 4 张 A10 24GB。

输入来源CorpAssist 提示

平均 tokens / 请求 网关日志 FIM 常 32～128 completion tokens
目标 P95 延迟 SLA 与 TTFT 分开看
单卡最大并发 压测数据 随 max-model-len 变化

容量计算脚本

python 
# capacity_calculator.py — CorpAssist 容量估算

def estimate_gpu_count(
    active_users: int = 500,
    avg_interval_sec: int = 30,
    avg_latency_sec: float = 3.0,
    concurrent_per_gpu: int = 20,
    buffer_pct: float = 0.25,
) -> dict:
    """
    估算 S4 补全场景所需 GPU 数量。
    """
    # 并发请求数 (Little 定律)
    arrival_rate = active_users / avg_interval_sec
    concurrent_requests = arrival_rate * avg_latency_sec

    # GPU 数量
    gpu_needed = concurrent_requests / concurrent_per_gpu
    gpu_with_buffer = gpu_needed * (1 + buffer_pct)

    # tokens/s 估算
    avg_tokens_per_request = 64  # FIM 平均补全长度
    total_tokens_per_sec = arrival_rate * avg_tokens_per_request

    return {
        "arrival_rate": arrival_rate,
        "concurrent_requests": round(concurrent_requests, 1),
        "gpu_needed_raw": round(gpu_needed, 1),
        "gpu_needed_with_buffer": round(gpu_with_buffer, 1),
        "gpu_suggested": int(gpu_with_buffer) + 1,
        "total_tokens_per_sec": round(total_tokens_per_sec, 0),
    }

# 示例
result = estimate_gpu_count()
for k, v in result.items():
    print(f"{k}: {v}")

# 输出:
# arrival_rate: 16.67
# concurrent_requests: 50.0
# gpu_needed_raw: 2.5
# gpu_needed_with_buffer: 3.12
# gpu_suggested: 4
先想 10 秒：QPS 翻倍但平均生成长度减半，GPU 压力怎样？

tokens/s 可能相近；若 KV 与批处理仍占满，压力不一定减半——要看 vLLM 批处理效率与 max-model-len。

capacity plan 连接业务、模型与基础设施评审。

组件7B AWQ14B AWQ说明
权重 ~5 GB ~10 GB 量化后，FP16 约 2 倍
KV cache / 序列 (8K) ~0.5 GB ~0.5 GB 与模型结构相关，非参数量
框架开销 ~1 GB ~2 GB CUDA context、临时 tensors
16 并发 KV 总和 ~8 GB ~8 GB 每序列独立 KV
合计 ~14 GB / 24 GB ~20 GB / 48 GB 24GB 卡可跑 7B 16 并发

显存估算 Python 函数

python 
def estimate_vram(
    model_size_b: int = 7,       # 模型参数量 (B)
    quant_bits: int = 4,         # 量化位数 (4/8/16)
    max_seq_len: int = 8192,     # 最大序列长度
    num_layers: int = 32,        # 层数
    num_heads: int = 32,         # 注意力头数
    head_dim: int = 128,         # 每头维度
    max_concurrency: int = 16,   # 最大并发数
):
    # 权重显存
    weight_gb = model_size_b * 2 * (16 / quant_bits) / 1024  # 单位是 FP16 的倍数

    # KV cache 显存 (每序列)
    kv_per_seq_gb = (2 * num_layers * num_heads * head_dim * max_seq_len * 2) / (1024**3)

    kv_total_gb = kv_per_seq_gb * max_concurrency

    overhead_gb = 1.0  # 框架开销

    total = weight_gb + kv_total_gb + overhead_gb
    return {
        "weight_gb": round(weight_gb, 1),
        "kv_per_seq_gb": round(kv_per_seq_gb, 2),
        "kv_total_gb": round(kv_total_gb, 1),
        "overhead_gb": overhead_gb,
        "total_gb": round(total, 1),
        "recommended_gpu": "24GB" if total < 20 else "48GB" if total < 40 else "80GB"
    }

# 7B AWQ 估算
vram = estimate_vram()
for k, v in vram.items():
    print(f"{k}: {v}")
# total_gb: ~14.2 → 推荐 24GB 卡
低估显存
  • 只算权重，忽略 KV
  • 把 max-model-len 拉到 32K 却按 8K 估并发
  • 灰度双版本两套路权重同卡
稳妥估法
  • 压测目标并发 + 观察 nvidia-smi 峰值
  • 量化降权重，再调 max-num-seqs
  • 写进 scale plan 的扩容触发条件

- 权重 5GB + KV 8GB + 开销 1GB = 14GB

- 4 × A10 24GB 按需: ~$X/小时
- 年化: ~$Y (含电费/折旧)

## 扩容触发
- P95 TTFT > 1s 持续 5min
- KV cache usage > 85%
- 队列深度 > 3

常见问题（FAQ）

容量估算和实际压测差多少？
粗算通常偏保守，偏差可能在 20-40%。因为 vLLM 的 continuous batching 效率随负载变化。建议用粗算确定下限，用压测确定上限。正文的 estimate_gpu_count 函数给出的是保守下限。
高峰期的突发流量怎么处理？
三种策略：① 超量采购（按峰值 + 25% 余量）② 用云 API fallback 吸收突发 ③ 排队 + 降级（非关键请求限流）。CorpAssist 推荐方案 ① + ② 组合。
7B 和 14B 模型的容量差异有多大？
显存约 2 倍，单卡并发约减半。14B AWQ 需要跨卡 TP 或更大的 GPU（48GB）。容量规划时需要按模型规格重新估算。
量化后还要按 FP16 做容量规划吗？
不需要。量化后的权重体积直接决定容量规划结果。但 kv cache 不受量化影响，它是 FP16/FP8 精度的 key/value 缓存。

本章结论

  • QPS：用并发与延迟估副本，不迷信峰值 QPS 宣传值。
  • 显存：权重 + KV + 余量；与量化、max-model-len 联动。
  • 交付：worksheet 一页纸即可支撑 s4 load 评审。
