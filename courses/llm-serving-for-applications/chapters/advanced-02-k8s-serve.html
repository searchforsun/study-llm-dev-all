Deployment

K8s serve 的核心是 Deployment 清单，需声明 GPU 资源请求与节点调度：

K8s 推理访问路径
flowchart LR
          IDE[IDE / 用户] --> Ing[Ingress infer.corp.internal]
          Ing --> Svc[Service vllm-infer]
          Svc --> Pod1[Pod vLLM]
          Svc --> Pod2[Pod vLLM]
          Pod1 --> GPU[GPU Node]
          Pod2 --> GPU
          GPU --> N1[NVIDIA A10]
          GPU --> N2[NVIDIA A10]

监控应打在 Ingress 与 Pod 两侧，区分网关排队与 GPU 算力瓶颈。

yaml 
# vllm-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-coder
  namespace: corpassist
spec:
  replicas: 2
  selector:
    matchLabels:
      app: vllm-coder
  template:
    metadata:
      labels:
        app: vllm-coder
    spec:
      nodeSelector:
        accelerator: nvidia-a10      # 调度到带 GPU 的节点
      containers:
        - name: vllm
          image: vllm/vllm-openai:latest
          command:
            - python
            - -m
            - vllm.entrypoints.openai.api_server
          args:
            - --model
            - /models/Qwen2.5-Coder-7B-Instruct-AWQ
            - --quantization
            - awq
            - --served-model-name
            - corp-coder-7b
            - --max-model-len
            - "8192"
            - --gpu-memory-utilization
            - "0.9"
          ports:
            - containerPort: 8000
          resources:
            limits:
              nvidia.com/gpu: 1        # GPU 资源申请
              memory: "32Gi"
              cpu: "8"
            requests:
              nvidia.com/gpu: 1
              memory: "24Gi"
              cpu: "4"
          volumeMounts:
            - name: models
              mountPath: /models
              readOnly: true
          env:
            - name: NVIDIA_VISIBLE_DEVICES
              value: "all"
          livenessProbe:
            httpGet:
              path: /v1/models
              port: 8000
            initialDelaySeconds: 120   # 冷启动时间长
            periodSeconds: 30
          startupProbe:
            httpGet:
              path: /v1/models
              port: 8000
            initialDelaySeconds: 60
            failureThreshold: 30       # 最多等 5 分钟
      volumes:
        - name: models
          hostPath:
            path: /data/models
            type: Directory

liveness infer 启动慢（加载权重），initialDelaySeconds 要大于冷启动时间，避免误杀 Pod。对于 7B AWQ 权重，建议 120s 以上。更稳妥的方式是使用 startupProbe。

不推荐
  • livenessProbe initialDelaySeconds=10，冷启动被误杀
  • 不设 nodeSelector，Pod 调度到无 GPU 的节点
  • requests == limits 不留弹性
  • 权重通过 initContainer 从外网拉取（离线环境失败）
推荐
  • startupProbe 宽容（30 次失败），livenessProbe 严格
  • nodeSelector 锁定 GPU 节点池
  • requests < limits 留调度弹性
  • 权重通过 hostPath / PV 预先部署
先想 10 秒：Pod 一直 CrashLoopBackOff 但手动进入容器 curl 正常，为什么？

liveness 探针配置不当：初始延迟太短或探针间隔太短，导致在模型加载完成前就判定失败。应使用 startupProbe 或加长 initialDelaySeconds。

Service

service infer 为双栈客户端提供稳定的集群内 DNS：

yaml 
# vllm-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: vllm-infer
  namespace: corpassist
  labels:
    app: vllm-coder
spec:
  selector:
    app: vllm-coder
  ports:
    - name: http
      port: 8000
      targetPort: 8000
  type: ClusterIP   # 集群内访问

双栈客户端配置：

yaml 
# Spring AI 指向 K8s Service
spring:
  ai:
    openai:
      base-url: http://vllm-infer.corpassist.svc.cluster.local:8000/v1
      api-key: EMPTY
      chat:
        options:
          model: corp-coder-7b

多版本灰度时可拆 vllm-v1 / vllm-v2 两个 Service，由 Ingress 加权（见 practice-04）。

Ingress

Ingress LLM 需放宽读超时以支撑 stream；CorpAssist S4 长生成常见 60～180s。

yaml 
# vllm-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: infer-gateway
  namespace: corpassist
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "180"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "180"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
    # 灰度发布（配合 practice-04）
    nginx.ingress.kubernetes.io/canary-by-header: "X-Infer-Canary"
    nginx.ingress.kubernetes.io/canary-by-header-value: "1"
    nginx.ingress.kubernetes.io/canary-weight: "5"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - infer.corp.internal
      secretName: infer-tls
  rules:
    - host: infer.corp.internal
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: vllm-infer
                port:
                  number: 8000

关键注解说明：

  • proxy-read-timeout: 180 — SSE 流式等待 180s 不超时
  • proxy-body-size: 10m — 长 prompt 请求体可能较大
  • canary-weight: 5 — 5% 流量导向金丝雀版本后端

监控

Infer monitor 最小集：

  • GPU：利用率、显存、温度（DCGM / nvidia-dcgm-exporter）
  • API：QPS、P95 TTFT、5xx 率（按 model 版本标签）
  • vLLM：队列长度、KV cache 使用率（/metrics 端点）
  • Pod：重启次数、OOM 事件

hpa infer 对 LLM 往往不灵敏：GPU 副本扩容慢、权重加载贵。更常见是固定副本 + 容量规划，或按队列深度定制 scaler。

yaml 
# Prometheus ServiceMonitor 采集 vLLM 指标
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: vllm-monitor
  namespace: corpassist
spec:
  selector:
    matchLabels:
      app: vllm-coder
  endpoints:
    - port: http
      path: /metrics
      interval: 15s
python 
# 从 vLLM metrics 读取关键推理指标
import requests

def get_infer_metrics(base_url="http://vllm:8000"):
    resp = requests.get(f"{base_url}/metrics")
    lines = resp.text.split("\n")
    metrics_of_interest = [
        "vllm:num_requests_running",
        "vllm:num_requests_waiting",
        "vllm:kv_cache_usage",
        "vllm:prompt_tokens_total",
        "vllm:generation_tokens_total",
    ]
    for line in lines:
        for metric in metrics_of_interest:
            if line.startswith(metric) and not line.startswith("#"):
                print(f"  {line}")
                break

if __name__ == "__main__":
    print("vLLM 推理指标:")
    get_infer_metrics()
FAQ：HPA 按 CPU 扩容 vLLM 有用吗？

通常无效：瓶颈在 GPU 与 KV 显存。CPU HPA 可能在 GPU 已饱和时仍扩容，浪费调度。建议固定副本 + 手动扩容，或基于 vLLM 排队长度定制 scaler。

常见问题（FAQ）

K8s 部署 vLLM 和 Compose 部署的核心区别是什么？
K8s 提供多副本、自动扩缩容、灰度发布、配置管理（ConfigMap）等高级能力。CorpAssist 生产用 K8s，开发/测试用 Compose。K8s 下 GPU 资源通过 nvidia.com/gpu 调度。
Ingress 超时设 180s 会不会太长？
对于流式场景，180s 是合理的。每次 SSE chunk 间隔通常在 50-200ms，180s 足够生成很长的回答。如果担心连接长期占用，可以在应用层设更短的 idle timeout。
没有 GPU Operator 怎么调度 GPU？
没有 GPU Operator 时，需要手动安装 NVIDIA device plugin。最简单的方式是使用 DaemonSet 部署 nvidia-device-plugin。GPU Operator 提供了更完整的生命周期管理。
如何在 K8s 中做 vLLM 的金丝雀发布？
使用 Nginx Ingress 的 canary 注解（如正文所示），或 Service Mesh 的流量分割（如 Istio VirtualService）。将新版本 vLLM 部署为独立 Deployment，通过 Ingress 权重分发流量。

本章结论

  • Service：集群内稳定入口，多版本可多 Service。
  • Ingress：长超时 + 可选金丝雀，对外统一 /v1。
  • 监控：GPU + TTFT + 错误率；慎用语义不明的 CPU HPA。

判断练习

  1. liveness 探活失败导致 Pod 反复重启，但手动 curl 正常。可能原因？

    参考答案

    initialDelay 过短、探针路径错误、启动中 GPU 初始化慢；应调 probe 或改用 startupProbe。

  2. 两个 vLLM Pod 部署在不同 GPU 节点上，但只有一个接收请求。可能原因？

    参考答案

    Service 的 selector 没有匹配到第二个 Pod（标签不一致），或第二个 Pod 未通过 readiness probe。检查 Pod 标签和探针状态。
