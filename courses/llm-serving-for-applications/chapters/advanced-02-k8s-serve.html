为什么要学本章

Compose 适合单机 capstone；CorpAssist 生产多副本 vLLM 要上 K8s。k8s serve 要求应用工程师读懂 service infer、ingress llm 与 infer monitor，并与 灰度 策略对齐。

学完你能
  • 草拟 vLLM Deployment + gpu node 调度
  • 配置 ClusterIP Service 与 Ingress 超时
  • 说明 liveness infer 与 hpa infer 在 LLM 上的局限
本章先记住 3 件事
  • Service：集群内稳定 DNS，Spring/Python 的 base_url 指向它。
  • Ingress：对外 TLS、超时、金丝雀权重（配合灰度）。
  • 监控：GPU、TTFT、队列深度比 CPU HPA 更重要。

Service

yaml 
apiVersion: v1
kind: Service
metadata:
  name: vllm-infer
  namespace: corpassist
spec:
  selector:
    app: vllm-coder
  ports:
    - port: 8000
      targetPort: 8000
  type: ClusterIP

双栈客户端：http://vllm-infer.corpassist.svc:8000/v1。多版本灰度时可拆 vllm-v1 / vllm-v2 两个 Service，由 Ingress 加权（见 practice-04）。

Deployment 片段（GPU）

yaml 
resources:
  limits:
    nvidia.com/gpu: 1
nodeSelector:
  accelerator: nvidia-a10
livenessProbe:
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 120
  periodSeconds: 30

liveness infer 启动慢（加载权重），initialDelaySeconds 要大于冷启动时间，避免误杀 Pod。

Ingress

Ingress LLM 需放宽读超时以支撑 stream； CorpAssist S4 长生成常见 60～180s。

yaml 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: infer-gateway
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "180"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "180"
spec:
  rules:
    - host: infer.corp.internal
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: vllm-infer
                port:
                  number: 8000
K8s 推理访问路径
flowchart LR
  IDE[IDE / 用户] --> Ing[Ingress infer.corp.internal]
  Ing --> Svc[Service vllm-infer]
  Svc --> Pod1[Pod vLLM]
  Svc --> Pod2[Pod vLLM]
  Pod1 --> GPU[GPU Node]

监控应打在 Ingress 与 Pod 两侧，区分网关排队与 GPU 算力瓶颈。

监控

Infer monitor 最小集：

  • GPU：利用率、显存、温度（DCGM / exporter）
  • API：QPS、P95 TTFT、5xx 率（按 model 版本标签）
  • vLLM：队列长度、KV cache 使用率（若 exporter 可用）

hpa infer 对 LLM 往往不灵敏：GPU 副本扩容慢、权重加载贵。更常见是固定副本 + 容量规划，或按队列深度定制 scaler。

FAQ：HPA 按 CPU 扩容 vLLM 有用吗？

通常无效：瓶颈在 GPU 与 KV 显存。CPU HPA 可能在 GPU 已饱和时仍扩容，浪费调度。

本章结论

  • Service：集群内稳定入口，多版本可多 Service。
  • Ingress：长超时 + 可选金丝雀，对外统一 /v1。
  • 监控：GPU + TTFT + 错误率；慎用语义不明的 CPU HPA。
