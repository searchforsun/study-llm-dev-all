Python 多引擎客户端对比

python 
# 多引擎吞吐对比脚本
import time
from openai import OpenAI

ENGINES = {
    "vLLM": "http://vllm:8000/v1",
    "TGI": "http://tgi:8080/v1",
    "Ollama": "http://localhost:11434/v1",
}

PROMPT = "def fibonacci(n):"
RESULTS = {}

for name, base_url in ENGINES.items():
    try:
        client = OpenAI(base_url=base_url, api_key="EMPTY")
        t0 = time.time()
        resp = client.completions.create(
            model="corp-coder-7b",
            prompt=PROMPT,
            max_tokens=64,
        )
        elapsed = time.time() - t0
        RESULTS[name] = {
            "latency": elapsed,
            "output": resp.choices[0].text[:50],
        }
        print(f"[{name}] {elapsed*1000:.0f}ms: {RESULTS[name]['output']}")
    except Exception as e:
        print(f"[{name}] 失败: {e}")

# 吞吐对比
for name, result in RESULTS.items():
    print(f"{name}: {result['latency']*1000:.0f}ms")

Spring AI 多引擎配置模式

java 
// Spring AI 配置 — 支持不同引擎后端
import org.springframework.ai.openai.OpenAiChatModel;
import org.springframework.ai.openai.OpenAiChatOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Primary;

@Configuration
public class MultiEngineConfig {

    @Bean
    @Primary
    public OpenAiChatModel defaultChatModel() {
        // 默认 vLLM
        return buildModel("http://vllm:8000/v1", "corp-coder-7b");
    }

    @Bean
    public OpenAiChatModel tgiChatModel() {
        // TGI 备选
        return buildModel("http://tgi:8080/v1", "corp-coder-7b");
    }

    @Bean
    public OpenAiChatModel ollamaChatModel() {
        // Ollama 本机开发
        return buildModel("http://localhost:11434/v1", "qwen2.5-coder:7b");
    }

    private OpenAiChatModel buildModel(String baseUrl, String model) {
        return OpenAiChatModel.builder()
            .baseUrl(baseUrl)
            .apiKey("EMPTY")
            .defaultOptions(OpenAiChatOptions.builder()
                .model(model)
                .temperature(0.1)
                .maxTokens(512)
                .build())
            .build();
    }
}

  • 吞吐决定一切 — API 兼容性、运维成熟度、社区支持更重要。
  • 引擎可以随意切换 — 换引擎意味着回归 FIM、流式、量化等全部路径。
  • Ollama 可以上生产 — Ollama 缺乏 continuous batching 和 PagedAttention，高并发明显弱于 vLLM。

常见问题（FAQ）

vLLM 和 TGI 的 FIM 兼容性一样吗？
不完全一样。vLLM 原生支持 FIM（/v1/completions）。TGI 也支持 FIM 但提示格式可能不同。如果客户使用 TGI，需要先在 Golden Set 上验证 FIM 输出格式是否符合预期。不兼容时可以通过适配层转换。
llama.cpp 的 GGUF 和 AWQ 哪个更好？
AWQ 用于 GPU 推理（vLLM 原生支持），GGUF 用于 CPU/边缘推理（llama.cpp）。两者不可直接比较。CorpAssist 生产用 AWQ，本机 PoC 用 GGUF。
为什么说 Ollama 不适合生产高并发？
Ollama 不实现 PagedAttention 等高级批处理机制，连续高并发时显存碎片化严重、吞吐下降明显。vLLM 在这些场景有 2-5x 吞吐优势。
可以混合使用多个引擎吗？比如 vLLM 做 FIM，TGI 做对话？
可以。在推理网关层按请求类型路由到不同后端。但运维复杂度倍增——需要维护两套引擎的镜像、监控和容量规划。建议单一引擎为主，特殊场景补充。

本章结论

  • 对比：vLLM/TGI 服务生产；Ollama/llama.cpp 面向边缘与本机。
  • 选型：以 OpenAI 契约 + S4 工作负载为轴，不唯吞吐榜单。
  • 落地：引擎变更放在网关后端，保护 deploy capstone 验收方式。

yaml 
# vllm-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-coder
  namespace: corpassist
spec:
  replicas: 2
  selector:
    matchLabels:
      app: vllm-coder
  template:
    metadata:
      labels:
        app: vllm-coder
    spec:
      nodeSelector:
        accelerator: nvidia-a10      # 调度到带 GPU 的节点
      containers:
        - name: vllm
          image: vllm/vllm-openai:latest
          command:
            - python
            - -m
            - vllm.entrypoints.openai.api_server
          args:
            - --model
            - /models/Qwen2.5-Coder-7B-Instruct-AWQ
            - --quantization
            - awq
            - --served-model-name
            - corp-coder-7b
            - --max-model-len
            - "8192"
            - --gpu-memory-utilization
            - "0.9"
          ports:
            - containerPort: 8000
          resources:
            limits:
              nvidia.com/gpu: 1        # GPU 资源申请
              memory: "32Gi"
              cpu: "8"
            requests:
              nvidia.com/gpu: 1
              memory: "24Gi"
              cpu: "4"
          volumeMounts:
            - name: models
              mountPath: /models
              readOnly: true
          env:
            - name: NVIDIA_VISIBLE_DEVICES
              value: "all"
          livenessProbe:
            httpGet:
              path: /v1/models
              port: 8000
            initialDelaySeconds: 120   # 冷启动时间长
            periodSeconds: 30
          startupProbe:
            httpGet:
              path: /v1/models
              port: 8000
            initialDelaySeconds: 60
            failureThreshold: 30       # 最多等 5 分钟
      volumes:
        - name: models
          hostPath:
            path: /data/models
            type: Directory

liveness infer 启动慢（加载权重），initialDelaySeconds 要大于冷启动时间，避免误杀 Pod。对于 7B AWQ 权重，建议 120s 以上。更稳妥的方式是使用 startupProbe。

不推荐
  • livenessProbe initialDelaySeconds=10，冷启动被误杀
  • 不设 nodeSelector，Pod 调度到无 GPU 的节点
  • requests == limits 不留弹性
  • 权重通过 initContainer 从外网拉取（离线环境失败）
推荐
  • startupProbe 宽容（30 次失败），livenessProbe 严格
  • nodeSelector 锁定 GPU 节点池
  • requests < limits 留调度弹性
  • 权重通过 hostPath / PV 预先部署
先想 10 秒：Pod 一直 CrashLoopBackOff 但手动进入容器 curl 正常，为什么？

liveness 探针配置不当：初始延迟太短或探针间隔太短，导致在模型加载完成前就判定失败。应使用 startupProbe 或加长 initialDelaySeconds。

Ingress

Ingress LLM 需放宽读超时以支撑 stream；CorpAssist S4 长生成常见 60～180s。

yaml 
# vllm-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: infer-gateway
  namespace: corpassist
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "180"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "180"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
    # 灰度发布（配合 practice-04）
    nginx.ingress.kubernetes.io/canary-by-header: "X-Infer-Canary"
    nginx.ingress.kubernetes.io/canary-by-header-value: "1"
    nginx.ingress.kubernetes.io/canary-weight: "5"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - infer.corp.internal
      secretName: infer-tls
  rules:
    - host: infer.corp.internal
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: vllm-infer
                port:
                  number: 8000

关键注解说明：

  • proxy-read-timeout: 180 — SSE 流式等待 180s 不超时
  • proxy-body-size: 10m — 长 prompt 请求体可能较大
  • canary-weight: 5 — 5% 流量导向金丝雀版本后端

监控

Infer monitor 最小集：

  • GPU：利用率、显存、温度（DCGM / nvidia-dcgm-exporter）
  • API：QPS、P95 TTFT、5xx 率（按 model 版本标签）
  • vLLM：队列长度、KV cache 使用率（/metrics 端点）
  • Pod：重启次数、OOM 事件

# 从 vLLM metrics 读取关键推理指标
import requests

def get_infer_metrics(base_url="http://vllm:8000"):
    resp = requests.get(f"{base_url}/metrics")
    lines = resp.text.split("\n")
    metrics_of_interest = [
        "vllm:num_requests_running",
        "vllm:num_requests_waiting",
        "vllm:kv_cache_usage",
        "vllm:prompt_tokens_total",
        "vllm:generation_tokens_total",
    ]
    for line in lines:
        for metric in metrics_of_interest:
            if line.startswith(metric) and not line.startswith("#"):
                print(f"  {line}")
                break

if __name__ == "__main__":
    print("vLLM 推理指标:")
    get_infer_metrics()
FAQ：HPA 按 CPU 扩容 vLLM 有用吗？

通常无效：瓶颈在 GPU 与 KV 显存。CPU HPA 可能在 GPU 已饱和时仍扩容，浪费调度。建议固定副本 + 手动扩容，或基于 vLLM 排队长度定制 scaler。

常见问题（FAQ）

K8s 部署 vLLM 和 Compose 部署的核心区别是什么？
K8s 提供多副本、自动扩缩容、灰度发布、配置管理（ConfigMap）等高级能力。CorpAssist 生产用 K8s，开发/测试用 Compose。K8s 下 GPU 资源通过 nvidia.com/gpu 调度。
Ingress 超时设 180s 会不会太长？
对于流式场景，180s 是合理的。每次 SSE chunk 间隔通常在 50-200ms，180s 足够生成很长的回答。如果担心连接长期占用，可以在应用层设更短的 idle timeout。
没有 GPU Operator 怎么调度 GPU？
没有 GPU Operator 时，需要手动安装 NVIDIA device plugin。最简单的方式是使用 DaemonSet 部署 nvidia-device-plugin。GPU Operator 提供了更完整的生命周期管理。
如何在 K8s 中做 vLLM 的金丝雀发布？

QPS 粗算

Concurrency math 基于 Little 定律的直觉应用：

并发请求数 N ≈ 到达率 λ × 平均响应时间 W

CorpAssist · S4 容量估算

假设：500 活跃开发者，平均每 30 秒触发一次代码补全，单次补全生成耗时 3 秒。

并发 ≈ 500 × (1/30) × 3 = 50 并发请求。

若单卡 vLLM 稳定支撑 20 并发（7B AWQ 实测），则需 50/20 ≈ 3 张 GPU。

考虑 25% 余量 → 建议 4 张 A10 24GB。

# capacity_calculator.py — CorpAssist 容量估算

def estimate_gpu_count(
    active_users: int = 500,
    avg_interval_sec: int = 30,
    avg_latency_sec: float = 3.0,
    concurrent_per_gpu: int = 20,
    buffer_pct: float = 0.25,
) -> dict:
    """
    估算 S4 补全场景所需 GPU 数量。
    """
    # 并发请求数 (Little 定律)
    arrival_rate = active_users / avg_interval_sec
    concurrent_requests = arrival_rate * avg_latency_sec

    # GPU 数量
    gpu_needed = concurrent_requests / concurrent_per_gpu
    gpu_with_buffer = gpu_needed * (1 + buffer_pct)

    # tokens/s 估算
    avg_tokens_per_request = 64  # FIM 平均补全长度
    total_tokens_per_sec = arrival_rate * avg_tokens_per_request

    return {
        "arrival_rate": arrival_rate,
        "concurrent_requests": round(concurrent_requests, 1),
        "gpu_needed_raw": round(gpu_needed, 1),
        "gpu_needed_with_buffer": round(gpu_with_buffer, 1),
        "gpu_suggested": int(gpu_with_buffer) + 1,
        "total_tokens_per_sec": round(total_tokens_per_sec, 0),
    }

# 示例
result = estimate_gpu_count()
for k, v in result.items():
    print(f"{k}: {v}")

# 输出:
# arrival_rate: 16.67
# concurrent_requests: 50.0
# gpu_needed_raw: 2.5
# gpu_needed_with_buffer: 3.12
# gpu_suggested: 4

显存估算 Python 函数

def estimate_vram(
    model_size_b: int = 7,       # 模型参数量 (B)
    quant_bits: int = 4,         # 量化位数 (4/8/16)
    max_seq_len: int = 8192,     # 最大序列长度
    num_layers: int = 32,        # 层数
    num_heads: int = 32,         # 注意力头数
    head_dim: int = 128,         # 每头维度
    max_concurrency: int = 16,   # 最大并发数
):
    # 权重显存
    weight_gb = model_size_b * 2 * (16 / quant_bits) / 1024  # 单位是 FP16 的倍数

    # KV cache 显存 (每序列)
    kv_per_seq_gb = (2 * num_layers * num_heads * head_dim * max_seq_len * 2) / (1024**3)

    kv_total_gb = kv_per_seq_gb * max_concurrency

    overhead_gb = 1.0  # 框架开销

    total = weight_gb + kv_total_gb + overhead_gb
    return {
        "weight_gb": round(weight_gb, 1),
        "kv_per_seq_gb": round(kv_per_seq_gb, 2),
        "kv_total_gb": round(kv_total_gb, 1),
        "overhead_gb": overhead_gb,
        "total_gb": round(total, 1),
        "recommended_gpu": "24GB" if total < 20 else "48GB" if total < 40 else "80GB"
    }

# 7B AWQ 估算
vram = estimate_vram()
for k, v in vram.items():
    print(f"{k}: {v}")
# total_gb: ~14.2 → 推荐 24GB 卡

稳妥估法

  • 量化降权重，再调 max-num-seqs
  • 写进 scale plan 的扩容触发条件

常见问题（FAQ）

容量估算和实际压测差多少？
粗算通常偏保守，偏差可能在 20-40%。因为 vLLM 的 continuous batching 效率随负载变化。建议用粗算确定下限，用压测确定上限。正文的 estimate_gpu_count 函数给出的是保守下限。
高峰期的突发流量怎么处理？
三种策略：① 超量采购（按峰值 + 25% 余量）② 用云 API fallback 吸收突发 ③ 排队 + 降级（非关键请求限流）。CorpAssist 推荐方案 ① + ② 组合。
7B 和 14B 模型的容量差异有多大？
显存约 2 倍，单卡并发约减半。14B AWQ 需要跨卡 TP 或更大的 GPU（48GB）。容量规划时需要按模型规格重新估算。
量化后还要按 FP16 做容量规划吗？
不需要。量化后的权重体积直接决定容量规划结果。但 kv cache 不受量化影响，它是 FP16/FP8 精度的 key/value 缓存。

API Serving：云上推理

CorpAssist · 运营文案生成（30 秒） 市场部用 CorpAssist 生成活动 slogan，输入不含源码与身份证号，QPS 低、可接受 2～3 秒首 token。团队尚未采购 GPU，希望零运维上线——典型 cloud API 场景。

API Serving 指直接调用厂商托管模型（通义、OpenAI 等）：按 Token 计费，运维由云侧承担。优势是快迭代、免 GPU 机房；代价是数据出境/出域策略、单价随 QPS 线性上升，以及无法自定义底层调度（如 KV 块大小）。

适合

非敏感文本、PoC、峰值弹性、多模态尝鲜（视觉模型贵且更新快）。

慎选

私有仓库代码、制度全文、需固定 IP 白名单的内网系统。

Spring Boot 混合路由配置

java 
// 混合路由 — 根据请求类型选择推理端点
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.openai.OpenAiChatModel;
import org.springframework.stereotype.Service;

@Service
public class HybridRouterService {

    private final OpenAiChatModel privateModel;  // 私有化 vLLM
    private final OpenAiChatModel cloudModel;     // 云 API

    public HybridRouterService(
            OpenAiChatModel privateModel,
            OpenAiChatModel cloudModel) {
        this.privateModel = privateModel;
        this.cloudModel = cloudModel;
    }

    /**
     * 根据请求中的 sensitivity 标记路由到不同模型
     */
    public String route(String userMessage, boolean isSensitive) {
        ChatModel target = isSensitive ? privateModel : cloudModel;
        String modelName = isSensitive ? "corp-coder-7b" : "qwen-turbo";

        return ChatClient.builder(target)
            .build()
            .prompt()
            .user(userMessage)
            .call()
            .content();
    }
}
常见误区
  • 混合 = 双写 — 同一次补全不应同时打云与私有化，除非 A/B 评测且采样率受控。
  • 忽略计费 — fallback 到云 API 若无 Token 上限，GPU 故障可能变成账单事故。
  • 路由只靠前端 — 敏感判定必须在服务端，客户端可被绕过。

常见问题（FAQ）

云 API 和私有化可以混用同一个 model 名吗？
不可以。云 API 的模型名是厂商定义的（如 qwen-turbo），私有化的模型名是 --served-model-name 指定的。混合路由时需在代码或配置中分别指定不同的 model 名。
S4 私有化需要什么 GPU 配置才能跑？
7B AWQ 量化模型推荐至少 24GB 显存（A10/RTX 4090）。14B 模型需要 48GB（A100）或使用张量并行跨两张卡。具体详见 量化选型 和 容量规划。
混合 Serving 中 fallback 到云 API 会暴增成本吗？
会。需要设置 fallback 配额：例如每分钟最多 1000 Token 走云 API，超过则返回错误。同时在云 API 端设 budget alert，避免 GPU 故障期间产生意外账单。
Python 和 Spring AI 在混合路由中各扮演什么角色？
Python 服务通常用于 RAG 管道、数据处理和 Agent 逻辑；Spring BFF 处理前端请求、会话管理和路由决策。两者都通过 OpenAI 兼容客户端指向私有化或云 API。详见 Python 客户端 和 Spring AI 远程。

本章结论

  • API Serving：零 GPU 运维、适合非敏感与 PoC；CorpAssist 运营类场景默认路径，S4 仅作兜底。
  • 私有化：S4 补全与私有代码的主路径；OpenAI 兼容 API 屏蔽底层 vLLM，合规与延迟可控。
  • 混合 Serving：按敏感信号与负载路由；必须服务端 fallback + 成本护栏，避免「名义混合、实际全云」。

GPU 调度入门

Docker Desktop / Linux 需安装 NVIDIA Container Toolkit，使 推理容器 (infer container) 内 nvidia-smi 可见设备。Compose v2.20+ 写法：

        reservations:

              count: 1

      NVIDIA_VISIBLE_DEVICES: "0"   # 指定 GPU 索引

Docker Desktop 对 Windows 的 GPU 支持有限。生产场景建议在 Linux 服务器上运行 vLLM。Windows 开发环境可用 WSL2 + Docker WSL2 backend，或使用远程 SSH 连接到 Linux GPU 服务器。
为什么容器内 nvidia-smi 正常但 vLLM 报 CUDA error？
vLLM 镜像的 CUDA 版本与宿主机驱动不兼容。检查 nvidia-smi 输出的 CUDA version，选择匹配的 vLLM 镜像 tag（如 vllm/vllm-openai:cuda-12.1）。
Compose 和 K8s 部署 vLLM 有什么区别？
Compose 适合单机 PoC 和小规模私有化。K8s 提供副本管理、自动扩缩容、灰度发布。CorpAssist 生产环境建议 K8s，开发/测试用 Compose。详见 K8s 部署清单。
容器重启后模型权重需要重新加载吗？
是的。vLLM 在启动时加载权重到 GPU 显存。如果 volume 挂载的权重文件在本地，重启容器会重新加载（耗时 1-5 分钟）。建议配置 start_period 避免健康检查误杀。

本章结论

  • 镜像：官方 vLLM OpenAI 镜像 + 外挂权重；锁版本、放大 shm_size。
  • GPU 调度：NVIDIA runtime + 明确 count/capabilities；一卡多实例需算清显存。
  • 资源 limit：CPU/内存/句柄 limit 与 GPU 利用率参数一起设，避免「能启动但高并发崩溃」。

核心参数详解

参数作用S4 提示
--model 权重路径或 HuggingFace 模型名 本地路径优先，避免启动时下载
--served-model-name API 中暴露的模型名 客户端 model 字段必须一致
--max-model-len 单序列最大 token（prompt+生成） FIM 上下文 8K 常见；过大吃 KV 显存
--gpu-memory-utilization 预占显存比例 0.85～0.92，留余量防 OOM
--tensor-parallel-size 多卡切分权重 14B+ 单卡放不下时启用
--enable-prefix-caching 相同 system prompt 复用 KV 多用户共享代码规范 prompt 时有效
--quantization 量化格式（awq/gptq） 与权重文件格式匹配

PagedAttention 是 vLLM 的核心技术：它将 KV cache 分页管理，按需分配显存，避免传统方法中的碎片和预分配浪费。对应用层的直接影响：同等显存下可支撑更多并发序列。

常见问题（FAQ）

vLLM 启动后 /v1/models 返回空列表，可能是什么原因？
可能是权重路径错误或加载失败。检查 vLLM 启动日志中是否有模型加载错误。运行 docker logs 或 CLI 控制台输出。常见原因：量化格式不匹配（指定 --quantization awq但权重是 GPTQ）。
Served model name 设置后可以修改吗？
需要重启 vLLM。运行时不能修改。如果有多个模型名需求，可以启动多个 vLLM 实例或使用 --served-model-name 指定一个稳定别名，客户端统一用这个别名。
vLLM 支持哪些量化格式？
vLLM 原生支持 AWQ、GPTQ、SqueezeLLM、FP8（H100 适用）。加载时需要 --quantization 参数指定格式。详见 量化选型。
vLLM 和 Ollama 用同样的模型，性能差多少？
vLLM 的 PagedAttention 和 continuous batching 在高并发场景吞吐远超 Ollama。CorpAssist 生产场景用 vLLM，开发者本机 Demo 可用 Ollama。详见 TGI 与其他引擎。

为什么要学本章

7B 模型 FP16 权重约 14GB，24GB 单卡放不下权重 + KV cache + 并发槽位。INT4 量化 可将权重压至 5～6GB，但精度有代价。CorpAssist S4 场景下，代码补全对量化敏感度低于通用对话——你需要掌握 量化选型 方法，而非深入算子实现。

  • 设计一条「量化评测流水线」：Golden Set → 推理 → 对比得分
本章先记住 3 件事
  • 量化减显存：memory save 主要来自权重精度降低，KV cache 仍需单独估算。
  • 精度与质量：quality loss 随任务而异；代码补全 INT4 通常可接受，创意写作更敏感。
  • 硬件匹配：hardware match 决定量化格式——AWQ 对 NVIDIA 各代架构支持最广。

精度-显存权衡 (precision trade) 是量化选型的核心。下表对比常见精度级别在 7B 模型上的大致表现（以实测为准）：

FP16 ~14 GB 基线 无 硬件充裕的基准评测
INT8 ~7 GB 50% 极微，通常 < 1% 高精度敏感任务
INT4 (AWQ/GPTQ) ~4–6 GB 60-70% 可接受，代码 < 3% CorpAssist S4 主力

AWQ 与 GPTQ 选型

AWQ 和 GPTQ 是当前最主流的 INT4 后训练量化方法。vLLM 对两者均原生支持，但选型有细微差别：

维度AWQGPTQ
量化原理 按激活值分布保护重要通道 基于 Hessian 矩阵的梯度补偿
vLLM 支持 原生 --quantization awq 原生 --quantization gptq
推理速度 略快（有 kernel 融合优化） 与 AWQ 接近
校准数据 无需，自动 需少量校准样本
CorpAssist 推荐 首选 备选（已有 GPTQ 权重时）

CorpAssist S4 · 量化部署示例

团队采购 2 台 A10 24GB 跑 S4 代码补全。决策过程：

  1. 7B Full FP16 → 14GB，加上 KV cache 8K/8 并发 ≈ 6GB，剩余 4GB 紧张 → 必须量化。
  2. INT8 → 7GB 权重 + 6GB KV ≈ 13GB，剩余 11GB → 可选。
  3. AWQ INT4 → 5GB 权重 + 6GB KV ≈ 11GB，剩余 13GB → 最优性价比。
  4. Golden Set 评测：AWQ 通过率 97%（vs FP16 99%），接受。

最终采用 Qwen2.5-Coder-7B-Instruct-AWQ，单卡支撑 15 并发，TTFT < 500ms。

常见误区
  • 量化万能论— 量化后仍要留足 KV cache 余量，max-model-len 过大一样 OOM。
  • 只看权重体积— 显存大头可能是 KV cache 而非权重，尤其长上下文场景。
  • 量化不评测— 不同任务对精度敏感度天差地别，必跑 Golden Set。
  • 忽略 kernel 兼容— 老旧 GPU 可能没有 INT4 kernel，会退化为 CPU 计算。

常见问题（FAQ）

AWQ 和 GPTQ 可以混合使用吗？
同一 vLLM 实例只能加载一种量化格式。混合使用需要通过网关路由到不同后端实例。CorpAssist 场景建议统一为 AWQ，减少运维复杂度。
量化后的模型还能微调吗？
理论上 INT4 模型可以做 QLoRA，但 CorpAssist 实践中建议：先全精度微调 → 再量化部署，而非量化后微调。量化后微调的质量损失更难评估。
INT4 量化对 FIM（Fill-in-Middle）场景影响大吗？
实证表明影响较小。代码补全对数值精度不敏感，只要校准数据包含 FIM 格式的 prompt。建议在校准集中加入 <fim_prefix>/<fim_suffix> 标记。
没有 GPU 能不能跑量化？
CPU 推理可以用 llama.cpp 的 GGUF 量化格式（详见 TGI 与其他引擎），但吞吐远低于 GPU。CorpAssist 生产场景不建议 CPU 量化推理。

本章结论

  • INT4/8 量化：权重体积减半，显存释放给 KV cache 与并发槽位；memory save 是核心收益。
  • AWQ vs GPTQ：AWQ 校准简单、vLLM 原生、推理略快，是 S4 首选；awq gptq 对比需结合实际硬件。
  • 精度评估：quality loss 用 Golden Set 量化，不可凭感觉；代码补全 INT4 通常可接受。
  • 硬件匹配：hardware match 决定 kernel 可用性；旧架构优先 INT8 而非 INT4。
  • 评测流水线：固定 prompt → 多精度推理 → 对比得分，纳入 CI 防止回归。

为什么要学本章

CorpAssist S4 的代码补全场景中，KV cache 占推理显存的大头（超过权重）。不懂 KV 与 vLLM KV 参数 的关系，调 max-model-len 和并发数就像碰运气。本章定位应用层，不写 CUDA kernel。

学完你能

  • 说明 prefix cache 如何降低首个请求的延迟
本章先记住 3 件事
  • KV cache 不是权重：它是推理过程中生成的 Key/Value 中间结果，随序列线性增长。
  • 参数调优杠杆：max-model-len 设得越大，每序列 KV 越贵；max-num-seqs 控制并发。
  • 不做算子开发：应用层不写 kernel，只调框架参数。

KV Cache 直觉

KV cache 是 Transformer 推理中每层 attention 模块缓存的 Key 矩阵和 Value 矩阵。自回归生成的每一步，模型只需计算最新 token 的 query，并读取之前所有 token 的 K/V——避免重复计算。

KV Cache 增长过程
sequenceDiagram
    participant GPU as GPU 显存
    participant Attn as Attention 层
    participant Token as 生成循环

    Token->>Attn: Step1: 生成 token1
    Attn->>GPU: 存储 K1/V1
    Token->>Attn: Step2: 生成 token2
    Attn->>GPU: 追加 K2/V2 (cache 变长)
    Token->>Attn: Step3: 生成 token3
    Attn->>GPU: 追加 K3/V3
    Note over GPU: 序列越长，KV cache 越大

每生成一个 token，KV cache 长度增加 1；最终大小 = 序列长度 × 层数 × 注意力头数 × (K+V) × 精度字节。

上下文显存 (context memory) 就是 KV cache 占用的空间。一个粗略公式：

KV cache ≈ 2 × 层数 × 头数 × 序列长度 × 精度字节

CorpAssist · KV cache 显存估算

7B 模型：32 层、32 头、head_dim=128、FP16(2bytes)。单序列 8K tokens 的 KV cache：

2 × 32 × 32 × 128 × 8192 × 2 ≈ 4 GB（实际 vLLM 有 PagedAttention 节省碎片，此处为理论估算）

若 8 并发 → 32 GB 仅 KV 就吃掉。这就是为什么要量化、要限制 max-model-len。

KV cache 开销场景

长文档摘要（32K+ tokens）、多轮对话（历史积累）——KV 可能大于权重。

KV cache 节省场景

S4 FIM 补全（64-256 completion tokens）、短问答——KV 开销小。

应用层调优范围

max-model-len、max-num-seqs、enable-prefix-caching。

先想 10 秒：序列长度翻倍，KV cache 增长多少？

线性增长——2倍序列 = 2倍 KV cache。但实际 vLLM 使用 PagedAttention，不会一次性分配全部显存，而是按 page 分配，碎片更少。延迟与显存权衡 的核心就是控制序列长度。

vLLM 相关参数

vLLM KV 参数 直接决定 KV cache 的分配策略与上限：

参数作用S4 推荐值影响
--max-model-len 单序列最大 token 数（prompt + completion） 8K～16K 越大 KV 上限越高；FIM 短补全用 8K 就够
--max-num-seqs 最大并发序列数 16～32 越高吞吐越大，但 KV 显存线性增
--gpu-memory-utilization vLLM 预占显存比例 0.85～0.92 留一部分给 kernel 与临时 tensor
--enable-prefix-caching 复用相同前缀的 KV 开启 相同 system prompt 或代码上下文减少预填充
不推荐
  • 无脑设 max-model-len=32K 导致 OOM
  • max-num-seqs 设 256 不调显存
  • 从不开 prefix caching
  • 把 vLLM 的 KV metrics 当黑盒不看
推荐
  • max-model-len 匹配实际场景：S4 8K、长文档 32K
  • max-num-seqs 从 8 逐步上调观察显存
  • 相同 system prompt 场景开启 prefix caching
  • 用 vLLM 的 /metrics 端点观察 KV 使用率

常见误区
  • KV cache 是权重的「缓存」 — 不对，它是在推理中动态生成的中间结果，不是权重的副本。
  • 长上下文 = 好质量 — S4 代码补全场景 max-model-len=8K 通常够用，32K 徒增 KV 开销。
  • Prefix caching 总有效 — 仅当请求前缀高度重复（如固定 system prompt）时才有用；随机 prompt 无收益。
  • 只看 GPU 利用率 — KV cache 使用率是更直接的瓶颈指标，应加入 Grafana 面板。

常见问题（FAQ）

Prefix caching 到底怎么工作？
prefix cache 对请求的初始 tokens 计算哈希。当新请求的前缀与缓存中的前缀匹配时，直接复用已计算的 KV，跳过预填充阶段。CorpAssist 场景中，相同 system prompt（如「你是代码助手」）被所有用户共享时收益显著。
为什么说 TTFT 与 KV cache 相关？
TTFT（time to first token） = 预填充时间。预填充阶段计算整个 prompt 的 KV cache，prompt 越长 TTFT 越高。优化 TTFT 的关键之一就是控制 prompt 长度以降低预填充开销。
vLLM 的 KV cache 是每个请求独占还是共享？
PagedAttention 按 page 管理，请求之间不共享 KV，但同一请求的不同 beam 可以共享部分 KV。Prefix caching 开启了跨请求共享前缀的 KV。
KV cache 用 INT8 能省多少显存？
vLLM 支持 KV cache 量化（--kv-cache-dtype fp8 等）。从 FP16 到 FP8 省一半，质量影响极小。但需要 GPU kernel 支持，A100/H100 原生支持 FP8。

本章结论

  • KV 直觉：KV cache 是中间结果而非权重，占显存大头；序列越长开销越大。
  • vLLM 参数：vLLM KV 参数（max-model-len、max-num-seqs）是应用层最直接的杠杆。
  • 延迟权衡：latency vs vram 跷跷板的平衡点由场景决定，S4 短补全偏压低延迟。
  • prefix caching：prefix cache 在固定 system prompt 场景能显著降低 TTFT。
  • 应用层边界：不做 kernel 开发，通过参数与架构隔离各场景的 KV 预算。

Python OpenAI 客户端实战

Python OpenAI 客户端 是连接 vLLM 的标准方式。推理客户端 的核心是正确配置 base_url 与 cluster endpoint。

python 
# corp_assist/infer_client.py — CorpAssist 推理客户端
import os
import time
import random
from typing import Optional

from openai import OpenAI, APIError, APITimeoutError
from openai.lib.streaming import ChatCompletionStream

# ---------- 全局单例 ----------
_INFER_CLIENT: Optional[OpenAI] = None

def get_infer_client() -> OpenAI:
    """获取全局推理客户端（单例模式，复用连接池）"""
    global _INFER_CLIENT
    if _INFER_CLIENT is None:
        # 支持多 endpoint 随机负载均衡
        base_urls = os.environ.get(
            "INFER_BASE_URLS",
            "http://vllm:8000/v1"
        ).split(",")
        selected_url = random.choice(base_urls)

        from httpx import Limits, Timeout
        _INFER_CLIENT = OpenAI(
            base_url=selected_url,
            api_key=os.environ.get("INFER_API_KEY", "EMPTY"),
            timeout=Timeout(
                connect=10.0,   # 连接超时
                read=120.0,     # 读超时（流式需更长）
                write=10.0,     # 写超时
            ),
            max_retries=2,
            default_headers={
                "X-CorpAssist-Service": "python-rag",
            }
        )
        # 调整 httpx 连接池
        _INFER_CLIENT._client._limits = Limits(
            max_connections=50,
            max_keepalive_connections=20,
        )
    return _INFER_CLIENT


# ---------- 带重试的推理调用 ----------
def chat_completion(
    messages: list,
    model: str = "corp-coder-7b",
    max_tokens: int = 512,
    temperature: float = 0.1,
    stream: bool = False,
    max_retries: int = 2,
):
    """
    统一推理入口：带指数退避重试。
    用于 CorpAssist 各 Python 后台服务。
    """
    client = get_infer_client()
    last_error = None

    for attempt in range(1 + max_retries):
        try:
            return client.chat.completions.create(
                model=model,
                messages=messages,
                max_tokens=max_tokens,
                temperature=temperature,
                stream=stream,
            )
        except APITimeoutError as e:
            last_error = e
            if attempt < max_retries:
                wait = 2 ** attempt + random.uniform(0, 1)
                print(f"推理超时，第 {attempt+1} 次重试，等待 {wait:.1f}s")
                time.sleep(wait)
        except APIError as e:
            if e.status_code in (429, 502, 503):
                last_error = e
                if attempt < max_retries:
                    wait = 2 ** attempt + random.uniform(0, 1)
                    print(f"API 错误 {e.status_code}，重试中...")
                    time.sleep(wait)
            else:
                raise  # 4xx 客户端错误直接抛出
    raise last_error or RuntimeError("推理失败")


# ---------- 流式补全（用于 IDE 插件后端） ----------
def stream_completion(messages: list, **kwargs):
    """流式生成，逐 token 返回"""
    response = chat_completion(messages=messages, stream=True, **kwargs)
    for chunk in response:
        if chunk.choices and chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content


# ---------- 使用示例 ----------
if __name__ == "__main__":
    # 非流式调用
    resp = chat_completion([
        {"role": "user", "content": "用 Python 写一个快速排序"}
    ])
    print("非流式结果:", resp.choices[0].message.content[:100])

    # 流式调用
    print("流式结果:", end="")
    for token in stream_completion([
        {"role": "user", "content": "Java 单例模式怎么写"}
    ]):
        print(token, end="", flush=True)
    print()

# fallback 策略：私有化失败时切换到 DashScope 云 API
import os
from openai import OpenAI

# 私有化 client
private_client = OpenAI(
    base_url="http://vllm:8000/v1",
    api_key="EMPTY",
    timeout=120,
)

    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key=os.environ["DASHSCOPE_API_KEY"],
)


def infer_with_fallback(messages: list, model: str = "corp-coder-7b"):
    """
    优先私有化，失败后回退云 API。
    云侧使用不同 model 名（qwen-turbo）。
    """
    for attempt, (client, model_name, label) in enumerate([
        (private_client, model, "私有化"),
        (cloud_client, "qwen-turbo", "云 API"),
    ]):
        try:
            resp = client.chat.completions.create(
                model=model_name,
                messages=messages,
                max_tokens=512,
                timeout=60,
            )
            print(f"[{label}] 推理成功")
            return resp.choices[0].message.content
        except Exception as e:
            print(f"[{label}] 失败: {e}")
            if attempt == 1:
                raise  # 云 API 也失败则真的失败
    return None

常见问题（FAQ）

OpenAI client 的 base_url 为什么必须带 /v1？
因为 OpenAI SDK 拼路径时会在 base_url 后面追加 /chat/completions。如果 base_url 设为 http://vllm:8000，最终请求变成 http://vllm:8000/chat/completions（少了一级），vLLM 无法路由。正确写法：http://vllm:8000/v1。
连接池耗尽会怎样？
HTTPX 的连接池有上限。耗尽时新请求会等待空闲连接，导致延迟飙升。现象：请求数增加但 QPS 不变甚至下降。解决方法：调大 max_connections，或使用异步客户端 AsyncOpenAI。
客户端重试和 vLLM 的去重怎么配合？
vLLM 本身不提供幂等去重。如果业务需要幂等（如多次重试不产生重复扣费），应在请求中加入 X-Request-Id header 并在业务层去重。简单场景下最多 2 次重试 + 退避即可。
为什么我设置了 timeout 但请求还是卡住？
可能是流式场景下 read timeout 未被正确传递。OpenAI SDK 流式模式下 read timeout 作用于每个 chunk 之间，若 vLLM 生成长文本但客户端 read timeout 过短，可能在中途断开。建议调大 read timeout 到 120s+。

本章结论

  • 客户端配置：OpenAI 客户端 指向 vLLM 需正确 base_url、超时与连接池。
  • 负载均衡：负载均衡 推荐网关方案；客户端直连用于开发测试。
  • 超时：推理超时 按场景配置，流式需更大的 read timeout。
  • 重试：重试策略 区分可重试/不可重试错误，指数退避最多 2 次。
  • Fallback：私有化失败时切换到云 API，确保服务不中断。

  • 确认 model 名与 vLLM --served-model-name 一致
  • 面向 ChatClient 接口编程

@Service
public class CorpAssistService {

    private final ChatClient chatClient;
    private final OpenAiChatModel fallbackChatModel;

    public CorpAssistService(
            ChatClient chatClient,
            OpenAiChatModel fallbackChatModel) {
        this.chatClient = chatClient;
        this.fallbackChatModel = fallbackChatModel;
    }

    /**
     * 调用推理，支持 fallback：私有化失败 → 云 API
     */
    public String chat(String userMessage) {
        try {
            return chatClient.prompt()
                .user(userMessage)
                .call()
                .content();
        } catch (Exception e) {
            log.warn("私有化推理失败，切换到云 API fallback", e);
            try {
                return ChatClient.builder(fallbackChatModel)
                    .build()
                    .prompt()
                    .user(userMessage)
                    .call()
                    .content();
            } catch (Exception fallbackException) {
                log.error("云 API fallback 也失败", fallbackException);
                return "抱歉，当前推理服务不可用，请稍后重试。";
            }
        }
    }
}

超时与熔断保护

Java 推理客户端 (java infer client) 的稳定性通过三层保护实现：

层级机制配置
1. 超时 RestClient connect/read timeout spring.cloud.openai.read-timeout=120000
2. 重试 Spring Retry / Resilience4j 最多 2 次退避重试

java 
// Resilience4j 熔断保护 — 防止雪崩
import io.github.resilience4j.circuitbreaker.CircuitBreaker;
import io.github.resilience4j.circuitbreaker.CircuitBreakerConfig;
import io.github.resilience4j.circuitbreaker.CircuitBreakerRegistry;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

import java.time.Duration;

@Configuration
public class CircuitBreakerConfig {

    @Bean
    public CircuitBreaker inferCircuitBreaker() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            .failureRateThreshold(50)           // 50% 失败率触发熔断
            .slidingWindowSize(10)              // 10 次请求窗口
            .minimumNumberOfCalls(5)            // 最少 5 次开始统计
            .waitDurationInOpenState(Duration.ofSeconds(30)) // 熔断持续 30s
            .permittedNumberOfCallsInHalfOpenState(3)        // 半开时允许 3 次
            .build();

        CircuitBreakerRegistry registry = CircuitBreakerRegistry.of(config);
        return registry.circuitBreaker("vllm-infer");
    }
}

为什么要学本章

CorpAssist S4 的 IDE 补全需要低首 token 延迟 (TTFT)，用户等不了全量生成。而运营文案批量生成则需要高吞吐。两种场景分别对应 流式吞吐 与 批处理 策略，vLLM 的 continuous batching 同时服务于它们。

本章先记住 3 件事
  • 流式不是魔法：首 token 仍需等预填充，流式只改善后续 token 的感知延迟。
  • Continuous batching：vLLM 引擎内部将多个请求合并为一个 batch，提升 GPU 利用率。
  • 并发限流：并发控制 (concurrency) 通过 max-num-seqs 防止 OOM。

流式推理原理

流式吞吐 (stream throughput) 指服务端逐 token 返回 SSE（Server-Sent Events）数据流。客户端在首 token 到达后即可开始展示，不必等待全部生成完毕。

流式 vs 非流式时序
sequenceDiagram
    participant C as 客户端
    participant V as vLLM
    participant G as GPU

    Note over C,G: 非流式（等待全部生成）
    C->>V: POST /v1/chat (stream=false)
    V->>G: 预填充 + 自回归生成
    G-->>V: ...
    V-->>C: 200 OK (完整输出)

    Note over C,G: 流式（逐 token 返回）
    C->>V: POST /v1/chat (stream=true)
    V->>G: 预填充
    G-->>V: 首 token
    V-->>C: SSE data: token1
    G-->>V: token2
    V-->>C: SSE data: token2
    Note over C,V: TTFT = 首 token 到达时间
    Note over C,V: TPOT = 后续每 token 时间

TTFT (Time To First Token) 对用户体验至关重要。流式将 TPOT 与 TTFT 解耦，用户感知延迟大幅降低。

CorpAssist · IDE 内嵌 FIM 补全 开发者每次按键触发 FIM 补全，期望 200ms 内看到首 token。非流式模式下生成 64 tokens 需约 2s → 用户等到全部生成完才能看到。流式模式下 200ms 首 token + 后续逐 token 渲染 → 体验大幅提升。

Python 流式补全

python 
# Python 流式调用 — 用于 IDE 补全后端
from openai import OpenAI
import time

client = OpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY")

def stream_fim_completion(prompt: str, max_tokens: int = 128):
    """流式 FIM 补全，逐 token 产出"""
    start = time.time()
    first_token = True

    response = client.completions.create(
        model="corp-coder-7b",
        prompt=prompt,
        max_tokens=max_tokens,
        temperature=0.1,
        stream=True,  # 启用流式
    )

    for chunk in response:
        if chunk.choices and chunk.choices[0].text:
            token = chunk.choices[0].text
            if first_token:
                ttft = time.time() - start
                print(f"[TTFT={ttft*1000:.0f}ms]")
                first_token = False
            yield token

    elapsed = time.time() - start
    print(f"\n[总耗时={elapsed*1000:.0f}ms]")


# 使用示例
prompt = "def calculate_sum(arr):\n    \n    return result"
print("流式补全输出:", end="", flush=True)
for token in stream_fim_completion(prompt):
    print(token, end="", flush=True)
print()

批处理策略

批处理推理 (batch infer) 有两种形态：

  1. vLLM Continuous Batching：引擎内部自动将等待中的序列合并到当前 batch，动态调度。
  2. 应用层显式批处理：客户端攒一批请求一次性发送，减少 GPU 切换开销。
策略适用场景CorpAssist 示例
流式 SSE 交互式、低延迟 IDE FIM 补全、对话
Continuous Batching 高并发混合负载 多用户同时补全（vLLM 自动处理）
离线 Batch API 非实时、高吞吐 夜间代码审查、批量分析
应用层批处理 短请求高 QPS 批量评分配对（20 条 prompt 一次性提交）

vLLM Batch API 调用

python 
# 离线批处理 — 适合夜间代码审查等批量任务
# vLLM 提供批量端点 /v1/completions（非流式）

import json
from openai import OpenAI
from concurrent.futures import ThreadPoolExecutor, as_completed

client = OpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY")

def batch_code_review(code_snippets: list[str]) -> list[str]:
    """批量代码审查：并发提交多个补全请求"""
    results = [None] * len(code_snippets)

    def single_review(idx: int, code: str) -> tuple[int, str]:
        resp = client.chat.completions.create(
            model="corp-coder-7b",
            messages=[
                {"role": "system", "content": "请审查以下代码的安全性。"},
                {"role": "user", "content": f"```\n{code}\n```"}
            ],
            max_tokens=256,
            temperature=0.1,
        )
        return idx, resp.choices[0].message.content

    # 使用线程池并发提交（vLLM 内部 continuous batching 会合并）
    with ThreadPoolExecutor(max_workers=8) as pool:
        futures = {
            pool.submit(single_review, i, code): i
            for i, code in enumerate(code_snippets)
        }
        for future in as_completed(futures):
            idx, content = future.result()
            results[idx] = content

    return results

# 示例
snippets = [
    "def unsafe_function(user_input):\n    exec(user_input)",
    "public class SecureClass {\n    private String secret;",
]
reviews = batch_code_review(snippets)
for i, review in enumerate(reviews):
    print(f"[Snippet {i+1}] {review[:100]}...")

并发数调优

并发控制 (concurrency) 是吞吐和显存之间的杠杆：

python 
# 并发压测：从低到高逐步上调，观察 TTFT 和显存

def benchmark_concurrency(
    max_concurrency: int = 32,
    prompt: str = "def hello():\n",
    tokens: int = 128
):
    """逐步增加并发，记录每个并发级别下的 TTFT p95"""
    import time
    import numpy as np

    for concurrency in [1, 2, 4, 8, 16, max_concurrency]:
        ttfts = []

        def single_request():
            t0 = time.time()
            response = client.completions.create(
                model="corp-coder-7b",
                prompt=prompt,
                max_tokens=tokens,
                stream=True,
            )
            for chunk in response:
                if chunk.choices and chunk.choices[0].text:
                    ttft = (time.time() - t0) * 1000
                    ttfts.append(ttft)
                    break  # 只需首 token
            return ttfts[-1]

        with ThreadPoolExecutor(max_workers=concurrency) as pool:
            list(pool.map(lambda _: single_request(), range(concurrency)))

        p95 = np.percentile(ttfts, 95) if ttfts else 0
        print(f"并发={concurrency:2d}, TTFT p95={p95:.0f}ms")

if __name__ == "__main__":
    benchmark_concurrency()
先想 10 秒：4 个并发请求各生成 64 tokens，vLLM 内部怎么调度？

vLLM 的 continuous batching 会将 4 个请求合并为一个 batch。每步解码 4 个 token（各序列 1 个），直到某个序列生成完毕退出 batch。总吞吐 ≈ 4 × 64 tokens / 总时间，GPU 利用率更高。

常见误区
  • 流式一定比非流式快 — TTFT 一致，总耗时可能更长（token 间有网络传输开销）。
  • 并发越高越好 — 超过 vLLM max-num-seqs 的请求会排队，TTFT 雪崩。
  • Continuous batching = 无限并发 — 显存是物理上限，max-num-seqs 控制实际并发。

常见问题（FAQ）

流式场景下怎么算 Token 消耗？
流式 token 和非流式 token 一样按实际生成数量计费。监控上两者无区别。但流式场景需要更精细的 TTFT / TPOT 指标来评估用户体验。
vLLM 的 continuous batching 和客户端并发有什么区别？
客户端并发是「同时发 N 个 HTTP 请求」。vLLM 收到后通过 continuous batching 将它们合并到 GPU batch 中，每次前向推理同时处理 N 个序列。客户端并发 32 个请求 ≠ vLLM 内部 batch size 32（受 max-num-seqs 限制）。
Batch API 怎么在 Spring AI 中实现？
Spring AI 的 ChatClient 不支持原生批量端点。可以实现业务层批处理：在 Service 中 for 循环或使用 @Async 并发提交多个请求，由 vLLM 的 continuous batching 自动合并。
TTFT 过高如何优化？
TTFT ≈ 预填充时间。降低 TTFT 的方法：① 降低 prompt 长度（减少预填充量）；② 启用 prefix caching（复用 KV）；③ 降低 max-num-seqs（减少排队）；④ 升级 GPU（内存带宽更大）。

本章结论

  • 流式原理：流式吞吐 通过 SSE 逐 token 返回，显著改善首 token 感知延迟。
  • TTFT：TTFT 是交互式场景的核心 SLA，受预填充和排队影响。
  • 批处理：批处理 用于离线高吞吐任务；vLLM continuous batching 自动优化。
  • 并发控制：并发 通过 max-num-seqs 限制，超过即排队导致 TTFT 劣化。
  • 场景匹配：S4 补全用流式，运营批量用非流式批处理。

A/B 对比脚本

python 
# A/B 对比 — 固定 prompt 集，分别请求 v1 和 v2
import json, time
from openai import OpenAI

v1_client = OpenAI(base_url="http://vllm-v1:8000/v1", api_key="EMPTY")
v2_client = OpenAI(base_url="http://vllm-v2:8000/v1", api_key="EMPTY")

GOLDEN_PROMPTS = [
    "def fibonacci(n):",
    "public class Singleton {",
    "git merge 和 rebase 的区别",
    "SELECT * FROM users WHERE",
]

def ab_compare():
    results = {"v1": [], "v2": []}
    for prompt in GOLDEN_PROMPTS:
        for label, client in [("v1", v1_client), ("v2", v2_client)]:
            t0 = time.time()
            resp = client.chat.completions.create(
                model="corp-coder-7b",  # 两端都用 alias
                messages=[{"role": "user", "content": prompt}],
                max_tokens=128,
            )
            elapsed = time.time() - t0
            results[label].append({
                "prompt": prompt,
                "latency": elapsed,
                "output_length": len(resp.choices[0].message.content),
            })
    # 汇总对比
    v1_avg = sum(r["latency"] for r in results["v1"]) / len(results["v1"])
    v2_avg = sum(r["latency"] for r in results["v2"]) / len(results["v2"])
    print(f"v1 平均延迟: {v1_avg*1000:.0f}ms")
    print(f"v2 平均延迟: {v2_avg*1000:.0f}ms")
    print(f"差异: {(v2_avg-v1_avg)/v1_avg*100:+.1f}%")

if __name__ == "__main__":
    ab_compare()
FAQ：回滚 vLLM 版本要不要重启 Spring？

若客户端始终连网关且 alias 不变，不必重启应用；仅当 Spring 缓存了错误 endpoint 或本地 fallback 指向 v2 时才需刷新配置。

常见问题（FAQ）

灰度期间发现 v2 延迟高但错误率正常，应该回滚吗？
不一定立即全量回滚。先确认是否仅 v2 cohort 受影响。如果是量化导致的固有问题，降 canary_percent 至 0 并修复后重新灰度。如果延迟超过 SLA 则立即回滚。
没有网关层如何做灰度？
可以使用 Spring Cloud LoadBalancer 或客户端侧随机路由，但不推荐。更好的方案是使用 K8s Ingress 的金丝雀发布（nginx.ingress.kubernetes.io/canary-weight）或 Service Mesh 的流量分割。
灰度模型和当前模型的输出格式不一致怎么办？
这是灰度前必须验证的。在 Golden Set 上跑输出格式校验。如果新模型的输出格式有变化（如 JSON schema 不同），需要先升级消费者代码再灰度模型。
A/B 对比和灰度发布的区别？
A/B 是实验（固定 cohort 长时间对比），灰度为发布（逐步放量最终全量）。A/B 需要统计显著性，灰度主要看 SLA 回滚条件。CorpAssist 场景中，灰度前先用 A/B 做离线对比。

  • 观测：请求必须带 version switch 标签，才能做 ab infer 与复盘。

        print(f"[FAIL] 健康检查失败: {e}")
        return False

def check_chat():
    """Step 2: Chat completions"""
    try:
        r = requests.post(f"{BASE}/v1/chat/completions", json={
            "model": "corp-coder-7b",
            "messages": [{"role": "user", "content": "1+1="}],
            "max_tokens": 16,
        }, timeout=30)
        assert r.status_code == 200
        content = r.json()["choices"][0]["message"]["content"]
        assert len(content) > 0
        print(f"[PASS] Chat 完成，输出: {content.strip()}")
        return True
    except Exception as e:
        print(f"[FAIL] Chat 失败: {e}")
        return False

def check_stream():
    """Step 3: 流式补全"""
    try:
        r = requests.post(f"{BASE}/v1/chat/completions", json={
            "model": "corp-coder-7b",
            "messages": [{"role": "user", "content": "写一个 word"}],
            "max_tokens": 32,
            "stream": True,
        }, timeout=60, stream=True)
        assert r.status_code == 200
        tokens = []
        for line in r.iter_lines():
            if line:
                decoded = line.decode("utf-8")
                if decoded.startswith("data: ") and decoded != "data: [DONE]":
                    tokens.append(".")
        print(f"[PASS] 流式通过，收到 {len(tokens)} 个 token")
        return True
    except Exception as e:
        print(f"[FAIL] 流式失败: {e}")
        return False

if __name__ == "__main__":
    results = [
        ("Health", check_health()),
        ("Chat", check_chat()),
        ("Stream", check_stream()),
    ]
    for name, passed in results:
        print(f"{'PASS' if passed else 'FAIL'}: {name}")
    sys.exit(0 if all(p for _, p in results) else 1)

常见问题（FAQ）

交付包中的权重太大无法传输怎么办？
AWQ 7B 约 5-6GB，建议使用对象存储（OSS/S3）分发，或使用 Git LFS。如果客户无网络，可以预装到移动硬盘交付。注意：交付后要校验 checksum 防止损坏。
客户环境没有 curl，怎么做 healthcheck？
可以用 Python 的 requests 做 healthcheck，或者使用 wget。Compose healthcheck 也支持 CMD-SHELL 方式：test: ["CMD-SHELL", "python -c \"import requests; requests.get('http://localhost:8000/v1/models')\""]。
双栈 smoke 通过但 IDE 补全仍超时，可能的排查点？
排查链路：IDE 插件 base_url → BFF 网关超时 → 公司代理 → 流式缓冲刷新。建议从 IDE 插件抓包开始，逐层排查到 vLLM 日志。常见问题是网关 read timeout 设得太短，匹配流式场景见 流式章节。
如何监控推理服务的健康状况？
vLLM 提供 /metrics 端点（Prometheus 格式），包含请求数、TTFT、KV cache 使用率等指标。配合 Grafana 仪表板可以全面监控。生产环境建议接入 PagerDuty/钉钉告警。
