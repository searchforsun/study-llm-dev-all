为什么学这章

仓库问答是代码助手最核心的用户交互场景——开发者用自然语言提问，助手 从代码库中检索相关代码并生成带引用的回答。本章从零构建完整的问答 REST API 端点，实现检索→Rerank→Prompt 拼装→ChatClient 调用→SSE 流式输出的完整链路，同时涵盖代码引用格式化输出、请求级缓存（Caffeine） 等生产级特性。掌握这些知识后，你可以独立搭建一个面向代码库的智能 问答服务，这是代码助手后端工程化的关键里程碑。

学完后能做什么

实现 @RestController 问答端点并编排检索→Rerank→生成流程；设计 带文件路径+行号+高亮的代码引用 Markdown 输出格式；实现 SSE 流式 响应的 Flux<ServerSentEvent> 端到端管线；配置 Caffeine 请求级缓存提升重复查询的响应速度；对问答 API 做延迟剖析和性能优化。

本章要点速记

1) 问答 API 端点 POST /api/chat/stream，请求体包含 query + repoId + userId，响应为 SSE 流。2) 检索→Rerank→生成三阶段编排：向量检 索取 Top-20（~320ms）、BGE-Reranker 重排序取 Top-5（~62ms）、 ChatClient 流式生成（首 Token ~800ms）。3) 代码引用格式化为 Markdown blockquote + 文件路径（行号）+ language 高亮代码块。4) Caffeine 缓存 Similarity 阈值 0.92 的查询，TTL 5 分钟，缓存命中率约 18-25% 且对延迟敏感度无显著负面影响。

@RestController 问答端点：检索→Rerank→Prompt→ChatClient→SSE

仓库问答 API 的核心编排流程包含五个步骤，每一步的输出是下一步的输 入，形成一条完整的处理链。第一步是检索（Retrieval）：接收用户查询 字符串，用 voyage-code-2 模型将其向量化（约 80ms），在 Milvus 的 code_snippets 集合中执行 ANN 搜索，取 Top-20 结果。检索时注入权 限过滤 FilterExpression。第二步是重排序（Rerank）：将 Top-20 结果 送入 BGE-Reranker-V2 交叉编码器，模型对每个（查询，代码片段）对计 算关联性得分，按得分降序取 Top-5。重排序是检索质量的关键——纯向量 检索的 Top-1 准确率约 61%，经 Rerank 后 Top-1 准确率提升至 83%。 第三步是 Prompt 拼装：将 System Prompt（角色定义、输出格式约束）、 检索上下文（Top-5 代码片段，每个包含文件路径+行号+代码内容+语言）、 用户问题、对话历史组装为完整的 Prompt。第四步是 ChatClient 调用： 使用 Spring AI 的 ChatClient 的 stream() 方法发起流式调用，返回 Flux<ChatResponse>。第五步是 SSE 流式输出：将 ChatResponse 转换为 ServerSentEvent，以 text/event-stream 格式推送至客户端。 整个链路端到端延迟：检索 320ms + Rerank 62ms + Prompt 拼装 5ms + 首 Token 生成 800ms ≈ 1187ms 首 Token 延迟，后续 Token 约 50ms/个。

package com.corpassist.api;

import com.corpassist.api.dto.ChatRequest;
import com.corpassist.api.dto.ChatResponse;
import com.corpassist.api.service.CodeQAService;
import org.springframework.http.MediaType;
import org.springframework.web.bind.annotation.*;
import reactor.core.publisher.Flux;

/**
 * 仓库问答 REST API 端点。
 * 支持同步（call）和流式（stream）两种调用模式。
 */
@RestController
@RequestMapping("/api/chat")
public class ChatController {

    private final CodeQAService codeQAService;

    public ChatController(CodeQAService codeQAService) {
        this.codeQAService = codeQAService;
    }

    /**
     * 流式问答端点。返回 SSE 流，客户端通过 EventSource 消费。
     * 请求体：
     * { "query": "UserService 的 createUser 方法在哪里？",
     *   "repoId": "repo-123",
     *   "userId": "user-456" }
     */
    @PostMapping(value = "/stream",
        produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> streamChat(
            @RequestBody ChatRequest request) {
        return codeQAService.streamAnswer(
            request.query(),
            request.repoId(),
            request.userId()
        ).map(token -> ServerSentEvent
            .<String>builder()
            .data(token)
            .event("token")
            .build()
        ).concatWith(Flux.just(
            ServerSentEvent.<String>builder()
                .event("done")
                .data("[DONE]")
                .build()
        ));
    }

    /**
     * 同步问答端点。直接返回完整回答字符串。
     * 适用于非交互式场景（如 CI 集成）。
     */
    @PostMapping("/call")
    public ChatResponse callChat(
            @RequestBody ChatRequest request) {
        String answer = codeQAService.getAnswer(
            request.query(),
            request.repoId(),
            request.userId()
        );
        return new ChatResponse(answer);
    }
}

package com.corpassist.api.service;

import com.corpassist.indexing.security.PermissionAwareMilvusRetriever;
import com.corpassist.rerank.BGEReranker;
import com.corpassist.context.DynamicPromptAdvisor;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.cache.annotation.Cacheable;
import org.springframework.stereotype.Service;
import reactor.core.publisher.Flux;
import java.util.List;

/**
 * 代码问答核心服务。
 * 编排检索→Rerank→Prompt→ChatClient→SSE 五步流程。
 */
@Service
public class CodeQAService {

    private static final Logger log = LoggerFactory
        .getLogger(CodeQAService.class);
    private static final int VECTOR_TOP_K = 20;
    private static final int RERANK_TOP_K = 5;

    private final PermissionAwareMilvusRetriever retriever;
    private final BGEReranker reranker;
    private final ChatClient chatClient;

    public CodeQAService(
            PermissionAwareMilvusRetriever retriever,
            BGEReranker reranker,
            ChatClient.Builder chatClientBuilder) {
        this.retriever = retriever;
        this.reranker = reranker;
        this.chatClient = chatClientBuilder
            .defaultSystem("""
                你是一个资深代码助手。你的任务是根据检索到的代码上下文
                回答开发者的问题。要求：
                1. 每个回答必须引用具体的文件路径和行号
                2. 代码示例请使用 ```language 格式并指明编程语言
                3. 如果检索到的上下文不足以回答问题，请明确指出
                4. 回答用中文，代码和专有名词保持英文
                """)
            .build();
    }

    /**
     * 流式回答：五步编排。
     */
    public Flux<String> streamAnswer(
            String query, String repoId, String userId) {
        long start = System.currentTimeMillis();
        // 第一步：向量检索 Top-20
        var searchDocs = retriever.search(
            "code_snippets",
            embedQuery(query), VECTOR_TOP_K, userId);
        log.debug("Retrieval: {} docs in {}ms",
            searchDocs.docs().size(),
            System.currentTimeMillis() - start);
        // 第二步：Rerank 取 Top-5
        var reranked = reranker.rerank(
            query, searchDocs.docs(), RERANK_TOP_K);
        log.debug("Rerank: {} docs", reranked.size());
        // 第三步：拼装 Prompt
        var prompt = buildPrompt(query, reranked);
        // 第四步：ChatClient 流式调用
        return chatClient.prompt(prompt)
            .stream()
            .content();
    }

    @Cacheable(value = "qaCache",
        key = "#query + ':' + #repoId",
        unless = "#result == null || #result.isEmpty()")
    public String getAnswer(
            String query, String repoId, String userId) {
        return streamAnswer(query, repoId, userId)
            .collectList()
            .block()
            .stream()
            .collect(java.util.stream.Collectors.joining());
    }

    private String buildPrompt(
            String query, List<ScoredDoc> docs) {
        StringBuilder sb = new StringBuilder();
        sb.append("## 检索到的代码上下文\n\n");
        for (int i = 0; i < docs.size(); i++) {
            ScoredDoc doc = docs.get(i);
            sb.append(String.format(
                "### %d. %s (行 %d-%d)\n",
                i + 1, doc.filePath(),
                doc.startLine(), doc.endLine()));
            sb.append("```")
              .append(detectLanguage(doc.filePath()))
              .append("\n")
              .append(doc.content())
              .append("\n```\n\n");
        }
        sb.append("## 用户问题\n").append(query);
        return sb.toString();
    }

    private float[] embedQuery(String query) {
        // 调用 EmbeddingModel 生成查询向量
        return new float[768]; // 桩实现
    }

    private String detectLanguage(String filePath) {
        if (filePath.endsWith(".java")) return "java";
        if (filePath.endsWith(".py")) return "python";
        if (filePath.endsWith(".ts")) return "typescript";
        return "";
    }
}

代码引用的格式化输出：文件路径 + 行号 + 高亮 Markdown

代码问答的回答必须包含可追溯的代码引用，否则开发者无法验证回答的准 确性。CorpAssist 定义了一套标准化的代码引用格式，确保在任何客户端 （IDE 插件、Web 界面、命令行）中保持一致。引用格式的核心原则是"每 个代码片段都包含三个要素：位置（在哪里）、内容（是什么）、上下文 （为什么）"。位置要素使用 Markdown blockquote + 文件路径 + 行号范 围，格式为 > 📁 src/main/java/com/example/UserService.java (第 42-58 行)。内容要素使用标准 Markdown 代码块，语言标签根据文件 扩展名自动识别。上下文要素在代码块前用一段自然语言描述该代码片段的 作用和引用的理由。

java 
package com.corpassist.api.format;

import org.springframework.stereotype.Component;
import java.util.List;
import java.util.stream.Collectors;

/**
 * 代码引用格式化器。
 * 将检索到的代码片段生成为包含文件路径、行号、
 * 语言高亮的标准 Markdown 引用块。
 */
@Component
public class CodeReferenceFormatter {

    /**
     * 格式化为 Markdown 引用字符串。
     * 输出示例：
     * > 📁 UserService.java (第 42-58 行)
     * ```java
     * public User createUser(String name) { ... }
     * ```
     */
    public String format(List<ScoredDoc> docs) {
        return docs.stream()
            .map(this::formatSingleDoc)
            .collect(Collectors.joining("\n\n"));
    }

    private String formatSingleDoc(ScoredDoc doc) {
        StringBuilder sb = new StringBuilder();
        // 位置信息
        String filePath = shortenPath(doc.filePath());
        sb.append("> 📁 ")
          .append(filePath)
          .append(" (第 ")
          .append(doc.startLine())
          .append("-")
          .append(doc.endLine())
          .append(" 行)\n");
        // 代码内容（语言高亮）
        String lang = detectLanguage(doc.filePath());
        sb.append("```").append(lang).append("\n")
          .append(doc.content())
          .append("\n```");
        return sb.toString();
    }

    /**
     * 格式化引用结果的整体回答。
     * 包含摘要、引用、补充说明三个部分。
     */
    public String formatFullAnswer(
            String summary,
            List<ScoredDoc> references,
            String extraNotes) {
        StringBuilder sb = new StringBuilder();
        sb.append("## 回答\n\n").append(summary).append("\n\n");
        sb.append("## 代码引用\n\n")
          .append(format(references)).append("\n\n");
        if (extraNotes != null && !extraNotes.isBlank()) {
            sb.append("## 补充说明\n\n").append(extraNotes);
        }
        return sb.toString();
    }

    private String shortenPath(String fullPath) {
        // 只保留 src/ 之后的路径片段
        int srcIndex = fullPath.indexOf("src/");
        return srcIndex >= 0
            ? fullPath.substring(srcIndex)
            : fullPath;
    }

    private String detectLanguage(String filePath) {
        if (filePath.endsWith(".java")) return "java";
        if (filePath.endsWith(".py")) return "python";
        if (filePath.endsWith(".ts") || filePath.endsWith(".tsx"))
            return "typescript";
        if (filePath.endsWith(".go")) return "go";
        if (filePath.endsWith(".sql")) return "sql";
        if (filePath.endsWith(".xml")) return "xml";
        if (filePath.endsWith(".yml") || filePath.endsWith(".yaml"))
            return "yaml";
        return "";
    }

    public record ScoredDoc(
        String filePath, String content,
        long startLine, long endLine, float score) {}
}

引用质量评估：开发者信任度调查

CorpAssist 对引用格式上线前后做了 A/B 测试（N=100 开发者）。A 组：纯文本回答，无引用格式；B 组：带文件路径+行号+高亮的完整引用 格式。结果：B 组的回答信任度评分 4.3/5.0（A 组 3.1/5.0），回答 可追溯性评分 4.5/5.0（A 组 2.8/5.0），开发者愿意继续使用的比例 78%（A 组 62%）。但 B 组的回答平均长度增加了 42%（从 280 字增至 398 字），首 Token 延迟增加了约 200ms（因为 Prompt 中需要包含更 多上下文）。权衡后确定：完整引用格式是必须的——信任度提升带来的 用户留存价值远超过多出的 200ms 延迟和额外的 Token 成本。

SSE 流式响应的端到端实现：Flux<ServerSentEvent>

SSE（Server-Sent Events）是一种基于 HTTP 的轻量级服务器推送协议， 相比 WebSocket，SSE 的优势在于：基于标准 HTTP 协议（不需升级握手）， 浏览器原生支持 EventSource API，自动重连机制，以及文本格式易于调试。 Spring Boot 通过 Spring WebFlux 的 Flux<ServerSentEvent> 提供了一流的 SSE 支持。CorpAssist 的 SSE 实现分为三个层次：Controller 层配置 produces = MediaType.TEXT_EVENT_STREAM_VALUE，Service 层返 回 Flux<String> 流，工具层处理错误流和完成信号。SSE 协议要求 响应 Content-Type 为 text/event-stream，每个事件以 "event: {type} \ndata: {payload}\n\n" 格式发送。CorpAssist 定义了三类事件： token（正常输出 Token）、error（错误信息）、done（完成信号）。

java 
package com.corpassist.api.stream;

import com.corpassist.api.dto.ChatRequest;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.http.MediaType;
import org.springframework.web.bind.annotation.*;
import reactor.core.publisher.Flux;
import reactor.core.publisher.Sinks;
import java.time.Duration;

/**
 * SSE 流式问答端点的高级实现。
 * 支持背压控制、超时处理和指标收集。
 */
@RestController
@RequestMapping("/api/chat")
public class SseChatController {

    private static final Logger log = LoggerFactory
        .getLogger(SseChatController.class);
    private static final long TIMEOUT_SECONDS = 60;

    private final CodeQAService codeQAService;

    public SseChatController(CodeQAService codeQAService) {
        this.codeQAService = codeQAService;
    }

    @PostMapping(value = "/sse/stream",
        produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> sseStream(
            @RequestBody ChatRequest request) {
        // 使用 Sink 实现背压和错误传播
        Sinks.Many<ServerSentEvent<String>> sink =
            Sinks.many().unicast().onBackpressureBuffer();
        // Token 流
        codeQAService.streamAnswer(
            request.query(),
            request.repoId(),
            request.userId()
        ).subscribe(
            token -> {
                ServerSentEvent<String> event =
                    ServerSentEvent.<String>builder()
                        .event("token")
                        .data(token)
                        .id(String.valueOf(
                            System.currentTimeMillis()))
                        .build();
                sink.tryEmitNext(event);
            },
            error -> {
                log.error("Stream error: {}", error.getMessage());
                sink.tryEmitNext(
                    ServerSentEvent.<String>builder()
                        .event("error")
                        .data("生成过程中发生错误: "
                            + error.getMessage())
                        .build());
                sink.tryEmitComplete();
            },
            () -> {
                sink.tryEmitNext(
                    ServerSentEvent.<String>builder()
                        .event("done")
                        .data("[DONE]")
                        .build());
                sink.tryEmitComplete();
            }
        );
        return sink.asFlux()
            .timeout(Duration.ofSeconds(TIMEOUT_SECONDS))
            .doOnError(e -> log.error(
                "SSE stream timeout for query: {}", e.getMessage()));
    }
}
WebSocket 和 SSE 有什么区别？为什么代码问答选择 SSE 而非 WebSocket？

SSE 是单向（服务器→客户端）的纯文本协议，基于标准 HTTP 连接， 浏览器通过 EventSource API 原生支持，自动重连，传输内容是 UTF-8 文本。WebSocket 是双向二进制协议，需要 HTTP Upgrade 握手 建立长连接，客户端通过 WebSocket API 使用，需要手动实现重连逻辑。 代码问答选择 SSE 的原因：1) 问答场景是天然单向流——用户发送一个 请求，服务器流式返回回答，不需要客户端实时向服务器推送额外数据。 2) SSE 的自动重连机制在处理不稳定的移动网络环境时比 WebSocket 更可靠——连接断开后浏览器自动重试 3 次，间隔 3 秒。3) SSE 与 现有的 HTTP 负载均衡器和反向代理兼容更好，不需要为 WebSocket 升级配置。唯一需要 WebSocket 的场景是双向实时通信（如协作者同 步编辑），代码助手不需要。如果客户端需要在回答过程中中断生成， 可以通过 SSE 流中发送一个专用的 cancel 事件来实现，或者使用 AbortController 中断 HTTP 请求。

请求级缓存（Caffeine）提升重复查询响应速度

观察 CorpAssist 的生产日志发现，约 18-25% 的查询是重复的——同一个 开发者或同一团队的多个成员在短时间内问相似的问题。例如"这个项目的 数据源配置在哪里"在团队新成员入职时可能被问 3-5 次。对这类重复查 询，完全重新执行检索+Rerank+LLM 生成的全流程是对计算资源的浪费。 CorpAssist 使用 Caffeine 缓存实现请求级缓存，以 (query_hash, repoId) 为缓存键，缓存完整回答内容。缓存相似度阈值设为 0.92—— 如果新查询与已缓存查询的 Embedding 余弦相似度 ≥0.92，直接返回缓存 结果而不走全流程。阈值选择经过实验：阈值 0.95 缓存命中率仅 8%， 阈值 0.85 会有 3.2% 的误命中（缓存了不相关的结果）。0.92 是命中率 （22%）和准确率（99.1%）的最佳平衡点。

java 
package com.corpassist.api.cache;

import com.github.benmanes.caffeine.cache.Cache;
import com.github.benmanes.caffeine.cache.Caffeine;
import jakarta.annotation.PostConstruct;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Component;
import java.util.concurrent.TimeUnit;

/**
 * 代码问答的请求级缓存。
 * 基于 Caffeine，缓存键为 query_hash + repoId，
 * 使用 Token 相似度阈值决定是否命中。
 */
@Component
public class QACacheManager {

    private static final Logger log = LoggerFactory
        .getLogger(QACacheManager.class);
    private static final double SIMILARITY_THRESHOLD = 0.92;
    private static final int MAX_CACHE_SIZE = 10_000;
    private static final int TTL_MINUTES = 5;

    private Cache<String, CachedAnswer> cache;
    private Cache<String, float[]> queryVectorCache;

    @PostConstruct
    public void init() {
        this.cache = Caffeine.newBuilder()
            .maximumSize(MAX_CACHE_SIZE)
            .expireAfterWrite(TTL_MINUTES, TimeUnit.MINUTES)
            .recordStats()
            .build();
        this.queryVectorCache = Caffeine.newBuilder()
            .maximumSize(5_000)
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .build();
    }

    /**
     * 尝试从缓存获取回答。
     * 计算查询向量与所有缓存查询向量的余弦相似度。
     * 如果最高相似度 ≥ 阈值则返回缓存结果。
     */
    public CachedResult tryGetCached(
            String query, String repoId,
            float[] queryVector) {
        String cacheKeyPrefix = repoId + ":";
        // 遍历所有缓存项，计算相似度
        for (var entry : cache.asMap().entrySet()) {
            String key = entry.getKey();
            if (!key.startsWith(cacheKeyPrefix)) continue;
            CachedAnswer cached = entry.getValue();
            float[] cachedVector = queryVectorCache
                .getIfPresent(key);
            if (cachedVector == null) continue;
            double similarity = cosineSimilarity(
                queryVector, cachedVector);
            if (similarity >= SIMILARITY_THRESHOLD) {
                log.debug("Cache hit: key={}, similarity={}",
                    key, similarity);
                return new CachedResult(
                    cached.answer(), cached.references(),
                    similarity, true);
            }
        }
        return new CachedResult(null, null, 0, false);
    }

    public void putCache(String query, String repoId,
                          float[] queryVector,
                          String answer, String references) {
        String key = repoId + ":" + hashQuery(query);
        cache.put(key, new CachedAnswer(answer, references));
        queryVectorCache.put(key, queryVector);
    }

    public CacheStats getStats() {
        var stats = cache.stats();
        return new CacheStats(
            stats.hitCount(), stats.missCount(),
            stats.hitRate(), stats.evictionCount());
    }

    private double cosineSimilarity(float[] a, float[] b) {
        double dot = 0, normA = 0, normB = 0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            normA += a[i] * a[i];
            normB += b[i] * b[i];
        }
        return dot / (Math.sqrt(normA) * Math.sqrt(normB));
    }

    private String hashQuery(String query) {
        return Integer.toHexString(query.hashCode());
    }

    public record CachedAnswer(
        String answer, String references) {}
    public record CachedResult(
        String answer, String references,
        double similarity, boolean hit) {}
    public record CacheStats(
        long hitCount, long missCount,
        double hitRate, long evictionCount) {}
}

面试可能问

如何评估代码问答 API 的回答质量？你的评估指标有哪些？

需要从三个维度建立评估体系。第一维度是检索质量（Retrieval Quality）：使用 Recall@K、MRR、NDCG@K 等排序指标，在人工标注 的测试集（CodingQA-500）上评估。目标值：Recall@10 ≥85%、MRR ≥0.80。第二维度是生成质量（Generation Quality）：使用 BLEU-4 （与标准答案的 n-gram 匹配）、ROUGE-L（最长公共子序列）、 人工评分（1-5 分，覆盖准确性、完整性、可读性）。其中人工评分最 重要，因为 BLEU/ROUGE 无法衡量代码引用的正确性。CorpAssist 的 内部目标：人工评分 ≥4.0/5.0。第三维度是用户体验（User Experience）：首 Token 延迟（目标 1.5s 以内）、完整回答延迟 （目标 5s 以内）、用户满意度（NPS 评分 ≥4.0）、回答采纳率（用 户是否参考了回答中的建议）。每周自动运行评测 Pipeline：从生产 日志采样 100 个问答对，人工标注质量，自动计算所有指标。如果任 何指标连续 2 周低于阈值，触发告警并回滚最近的变更。这种"自动化 评测+人工抽检"的混合模式确保在快速迭代的同时保持回答质量的稳定。

场景：开发者反馈问答 API 的返回速度时快时慢，有时候 1 秒 有时候 8 秒，你怎么排查根因？

速度波动通常是多因素叠加的结果，需要系统性地分层排查。第一层排查 网络：检查 API 与 Milvus、LLM 服务之间的网络延迟是否稳定——使用 ping 和 curl -w %{time_total} 分别测量。如果网络延迟波动大，可 能是共享网络带宽被其他服务占用（推荐将 CorpAssist 服务部署在独立 的 VPC 或 Kubernetes QoS Guaranteed 的 Pod 中）。第二层排查 Milvus：检查 Milvus 的查询队列深度和 CPU 使用率——如果并发查询突 发，IVF_FLAT 的 nprobe 参数过大会导致 CPU 瓶颈。解决方案：设置 Milvus 的 rate_limiter 参数限制最大 QPS，并对 nprobe 做自适应调 整（低负载时使用高精度 128、高负载时降级到 64）。第三层排查 LLM 服务：如果使用公共 API（如 DashScope），可能存在多租户争用—— qwen-plus 的响应时间 P99 在高峰期可能从 800ms 飙升至 3s。解决方 案：在 CorpAssist 中增加 LLM 调用的熔断器（Resilience4j CircuitBreaker），当连续 5 次调用超时时熔断 10 秒并降级到备选 模型（DeepSeek-Coder 本地部署版）。CorpAssist 的监控仪表盘显示， 经过这些优化后，P50 响应时间稳定在 1.2s，P99 从 8s 降至 2.5s。
