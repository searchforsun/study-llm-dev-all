Q1: 代码助手解决的核心问题是什么？

考点：理解代码助手的核心价值是"减少上下文切换"和"加速编码"，不仅仅是"生成代码"。

好回答

代码助手解决的核心问题是"上下文切换成本"。开发者平均每天经历 13 次上下文切换——从编辑器切到浏览器搜索 API、切到文档查找用法、切到命令行运行测试。每次切换平均损失 23 分钟回到心流状态。代码助手将这个过程从"切走搜索"变为"光标位置直接获取"——补全加速编码、问答替代搜索、Review 替代人工审查。2026 年 GitHub 的研究表明，使用代码助手的开发者进入心流状态的频率提升 40%。

差回答

代码助手就是用 AI 自动生成代码。可以省时间。

Q2: 代码上下文怎么选择和拼装？

考点：理解 Token 预算分配和文件选择策略。

java
// 面试回答话术
//
// "上下文选择是代码助手最核心的技术挑战。
// 我实现了三层选择策略：
//
// 1. LSP 符号引用——获取光标位置符号的定义和引用文件，
// 优先包含直接相关的文件（准确率 82%）。
//
// 2. MRU 回退——如果 LSP 信息不够，从最近打开的文件
// 中选择（准确率 65%）。
//
// 3. Token 预算管理——8K 窗口分配：System Prompt 20%、
// 当前文件 30%、相关文件 30%、符号表 5%、历史 5%。
// 超出预算时智能截断：保留开头和结尾，中间用
// '...截断...' 标记。
//
// 实验结论：3-5 个相关文件 + 符号表是最优组合。
// 更多文件引入噪音，准确率反而下降。"

Q4: 代码安全检测怎么做的？

考点：静态规则 + AI 审查的双重检测体系。

java
// 面试回答话术
//
// "四道检测防线：
// 1. 编译器语法检查——无法通过编译的直接标红
// 2. SpotBugs/Checkstyle 静态规则——检测空指针、
// 资源未关闭、命名规范等，准确率 95%+
// 3. AI 安全审查——Few-shot Prompt 检测 SQL 注入、
// XSS、硬编码密码、权限缺失等，准确率 70-80%
// 4. 人工复核——复杂逻辑由 PR Reviewer 审查
//
// 安全检测流水线集成在 CI 中，每次 PR 自动触发。
// 30 秒内完成检测，平均发现 2.3 个问题。
// 误报率：静态规则 5%，AI 审查 20%。
// 检测后生成安全报告，标注严重级别和修改建议。"

Q5: 延迟怎么优化的？

考点：从缓存、路由、部署三个角度的系统性延迟优化。
