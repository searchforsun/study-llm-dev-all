为什么学这章

LLM 的上下文窗口是有限的（即便模型支持 128K Token），而代码库的 规模可能是数百万行。如何将最相关的代码片段装入有限的 Token 预算 中，是代码助手工程化的核心挑战。本章深入 Token 预算分配策略、基 于 import 图和调用链的相关文件选择算法、LSP 协议在 Java 侧的集 成方案，以及 Spring AI 的上下文管理与 Advisor 链机制。掌握这些 知识后你将理解为何"把整个代码库塞进 Prompt"不可行，以及如何用最 小的 Token 开销获得最精准的代码上下文。

学完后能做什么

合理分配 LLM 的 Token 预算（System/Context/History/Response 四 部分）；用 import 依赖图和调用关系算法自动选择相关文件；通过 LSP 协议从 Java IDE 侧获取符号表、类型信息、AST 结构；使用 Spring AI Advisor 链实现上下文的压缩、注入和过滤；设计一套上下文工程的评 估指标，用 Recall 和 Precision 量化上下文的质量。

本章要点速记

1) Token 预算分配四部分：System（固定 500 Tokens）、Context（检 索上下文 2000-4000 Tokens）、History（历史对话 1000-2000 Tokens）、 Response（预留 500-1000 Tokens），总窗口不超模型上限的 80%。2) 基于 import 图的相关文件选择算法：从当前文件出发 BFS 3 层获取依 赖文件，结合调用关系图中被调用频率排序，选取 Top-N 装入上下文。 3) LSP 协议通过 textDocument/documentSymbol 和 textDocument/hover 请求获取符号表和类型信息，延迟约 50-200ms。 4) Spring AI 的 Advisor 链中 TokenCompressionAdvisor 使用 GPT-4o-mini 做上下文摘要压缩，可将 4000 Token 压缩至 1200 Token 而保留 92% 关键信息。

Token 窗口预算的分配策略：System / Context / History / Response

LLM 上下文窗口不是无限的——即便最新的 GPT-4o 支持 128K Token，实际 应用中也不建议用满。超窗会带来两个问题：一是模型对长上下文的"中间 迷失"问题——研究表明当相关信息位于输入中间位置时，模型的检索准确率 比在开头或结尾低约 20-30%（Liu et al., 2024）；二是推理延迟随输入 长度线性增长，128K Token 的推理首 Token 延迟可能达到 5-8s。因此 必须主动管理 Token 预算而不是被动依赖模型极限。CorpAssist 的黄金 法则是：总 Token 数不超过模型上限的 80%，在 128K 模型中即控制在 102K Token 以内。

Token 预算分为四个部分，每部分有明确的分配规则和优化策略。第一部 分是 System Prompt（系统指令），约 500 Token，包含角色定义、输出 格式约束、安全规则。这部分是固定的，不随对话内容变化，通常由配置 文件管理。第二部分是 Context（检索上下文），这是核心部分，分配 2000-8000 Token 用于放置检索到的代码片段。上下文需要按相关性排序 ——最相关的代码放在最前面和最后面（模型注意力最强的区域），中等相 关度放中间。第三部分是 History（历史消息），用于多轮对话场景，分 配 1000-2000 Token。当历史消息超过预算时需要用摘要替换（通过 TokenCompressionAdvisor 将历史压缩为摘要）。第四部分是 Response （预留输出），分配 500-1000 Token，确保模型有充足空间生成完整回答 而不会被截断。CorpAssist 监控每天的 Token 使用分布并设置告警： 当 Context 使用率 > 90% 时触发优化提示。

生产实践：Token 预算的实时监控与自动调整

CorpAssist 在生产环境中部署了 Token Budget Manager 组件，它拦截 每次 ChatClient 调用，动态计算当前 Token 使用量。当 Context 部分 接近预算上限时，自动执行两级降级策略：一级降级——从检索 Top-8 降 至 Top-5，并移除低相关度的 BM25 结果只保留向量搜索结果；二级降 级——启用上下文压缩，用 GPT-4o-mini 将代码片段摘要为关键定义和 签名，保留约 30% Token 但维持 85% 的问答准确率。该组件通过 Spring AI 的 Advisor 机制插入调用链，对上层应用完全透明。

基于 import 图 + 调用关系的相关文件选择算法

在代码助手的仓库问答场景中，模型需要理解当前问题的上下文。一个简 单的做法是将代码库中所有与问题相关的文件都塞入上下文——但这很快会 用完 Token 预算。CorpAssist 使用一种基于 import 依赖图和调用关系 的两阶段算法来智能选择文件。第一阶段：构建仓库级依赖图。通过 AST 解析器扫描每个文件的所有 import 语句（Java 的 import、Python 的 import/from、TypeScript 的 import/require），建立文件到文件的依 赖关系有向图。当用户提问关于某个特定类或方法时，以该类所在文件为 起点 D 做广度优先搜索 BFS，深度 3 层：第一层获取直接依赖和直接被 依赖的文件（邻居节点），第二层获取这些邻居节点的依赖，第三层获取 间接依赖。每一层按出度/入度排序，优先选择被更多文件引用的"枢纽" 类——因为它们更可能是核心领域概念，对理解上下文更有帮助。

第二阶段：调用关系增强。import 图只能反映编译期的静态依赖，无法 表达运行期的调用关系。CorpAssist 使用 TreeSitter 构建函数的调用 关系图（Call Graph）：对每个方法提取方法级 AST，识别其中的方法调 用表达式，建立 caller→callee 的有向边。当用户问题涉及某个方法时， 算法沿着调用图向上（查找调用方）和向下（查找被调用方）各扩展 2 层。 最终的候选文件集合是 import 图 Top-N 和调用图 Top-M 的并集，使用 PageRank 风格的相关度传播算法迭代计算每个文件的相关分。实际测试中， 该算法能将上下文命中率（用户问题所需信息在所选文件中的比率）从简 单字符串匹配的 52.3% 提升至 84.7%，同时将平均文件选择数从 15.2 降 至 6.8，节省 55% 的 Token 消耗。选择时间平均 180ms（10 万文件仓库 级），其中 BFS 遍历 60ms + 调用链解析 95ms + 排序 25ms。

java 
package com.corpassist.context;

import org.springframework.stereotype.Service;
import java.util.*;
import java.util.stream.Collectors;

/**
 * 基于 import 依赖图和调用关系图的相关文件选择器。
 * 从入口文件出发 BFS 扩展，结合图相关度传播确定 Top-K 文件。
 */
@Service
public class RelevantFileSelector {

    private final ImportGraph importGraph;
    private final CallGraph callGraph;

    public RelevantFileSelector(ImportGraph importGraph,
                                 CallGraph callGraph) {
        this.importGraph = importGraph;
        this.callGraph = callGraph;
    }

    /**
     * 选择与入口文件最相关的 Top-K 文件。
     * @param entryFilePath 用户问题涉及的入口文件路径
     * @param targetMethod 用户问题涉及的方法名（可为 null）
     * @param topK 需要返回的文件数量
     * @return 按相关度降序排列的文件路径列表
     */
    public List<ScoredFile> selectRelevantFiles(
            String entryFilePath, String targetMethod, int topK) {
        // 第一阶段：import 图 BFS 扩展 3 层
        Set<String> importNeighbors = importGraph
            .bfsReachable(entryFilePath, 3);
        // 第二阶段：调用关系图扩展 2 层
        Set<String> callNeighbors = new HashSet<>();
        if (targetMethod != null) {
            for (String caller : callGraph.getCallers(targetMethod)) {
                callNeighbors.addAll(importGraph
                    .bfsReachable(caller, 2));
            }
            for (String callee : callGraph.getCallees(targetMethod)) {
                callNeighbors.addAll(importGraph
                    .bfsReachable(callee, 2));
            }
        }
        // 合并并删除入口文件自身
        Set<String> combined = new HashSet<>(importNeighbors);
        combined.addAll(callNeighbors);
        combined.remove(entryFilePath);
        // 基于图中心度和调用频次排序
        return combined.stream()
            .map(f -> new ScoredFile(f, computeRelevanceScore(
                f, entryFilePath, targetMethod)))
            .sorted((a, b) -> Double.compare(b.score, a.score))
            .limit(topK)
            .collect(Collectors.toList());
    }

    /**
     * 计算文件相关度得分：综合考虑中心度、调用频次、近邻距离。
     */
    private double computeRelevanceScore(
            String file, String entryFile, String method) {
        double centrality = importGraph
            .computeDegreeCentrality(file);
        double callFrequency = method != null
            ? callGraph.getCallFrequency(file, method) : 0.0;
        double distancePenalty = 1.0 / (1.0 + importGraph
            .shortestDistance(entryFile, file));
        return 0.4 * centrality + 0.4 * callFrequency
            + 0.2 * distancePenalty;
    }

    public record ScoredFile(String filePath, double score) {}
}

A/B 测试结果：相关文件选择算法对问答质量的提升

在某 50 万行 Java 代码库中评估三种文件选择策略：A) 基于关键词匹 配（TF-IDF），B) 简单 import 图 BFS，C) import 图 + 调用关系图 混合算法（CorpAssist 方案）。测试集 500 个代码相关问答，评估指 标为上下文命中率（Context Hit Rate = 所需信息在所选文件中的比率）。 结果：A 策略 52.3%，B 策略 71.8%，C 策略 84.7%。C 策略同时将平 均文件选择数从 15.2 降至 6.8，Token 消耗减少 55%。这意味着采用 C 策略后，在相同的 Token 预算下，问答准确率可提升约 18%。

LSP 协议的 Java 侧集成：符号表、AST 与类型信息获取

语言服务器协议（Language Server Protocol, LSP）是 IDE 与语言服务 之间的标准化通信协议，定义了 textDocument/completion、 textDocument/definition、textDocument/references 等 30+ 种请求 类型。在代码助手场景中，LSP 是获取精确代码语义信息的重要通道—— 与 TreeSitter 的浅层语法解析不同，LSP 可以提供完全解析后的符号表、 类型推断结果、引用链等深度语义信息。CorpAssist 在 Java 侧集成了 Eclipse JDT Language Server（通过 org.eclipse.lsp4j 库），通过 WebSocket 或 TCP Socket 与语言服务器通信，请求特定的 LSP 方法。

三种最关键的 LSP 请求类型及其实践用法。第一是 textDocument/ documentSymbol，用于获取文件的符号表——类名、方法名、字段名、构造 函数及其层级结构。这在构建代码库的索引时非常有用：配合 TreeSitter， documentSymbol 提供了更精确的符号边界（如泛型参数的边界、注解的 完整结构）。第二是 textDocument/hover，用于获取光标位置处符号的 类型信息和方法签名。当用户在问答中问到一个不确定的类型时，CorpAssist 发送 hover 请求，获取完整的类型声明、Javadoc 注释、参数说明，作为 上下文注入 Prompt。第三是 textDocument/references，用于查找一个符 号的所有引用位置，配合 textDocument/definition 构建"定义-引用链"。 在"修改方法签名→自动更新所有调用方"的场景中，这组 LSP 请求是核 心能力。LSP 请求延迟：同一进程内约 20-50ms，远程连接约 100-300ms。 CorpAssist 采用预连接池管理 LSP 会话，避免每次请求重建连接的开销。

java 
package com.corpassist.lsp;

import org.eclipse.lsp4j.*;
import org.eclipse.lsp4j.jsonrpc.Launcher;
import org.eclipse.lsp4j.launch.LSPLauncher;
import org.eclipse.lsp4j.services.LanguageServer;
import org.springframework.stereotype.Component;
import java.util.concurrent.CompletableFuture;

/**
 * LSP 客户端封装，通过 LSP4J 库与 JDT Language Server 通信，
 * 提供符号表、类型信息、引用查询等能力。
 */
@Component
public class LspClient {

    private final LanguageServer languageServer;
    private final TextDocumentService textService;

    public LspClient() throws Exception {
        Process jdtProcess = new ProcessBuilder(
            "java", "-jar", "plugins/org.eclipse.equinox.launcher.jar",
            "-configuration", "configurations/config_win",
            "-data", "workspace"
        ).start();
        Launcher<LanguageServer> launcher = LSPLauncher
            .createLauncher(
                new ClientImpl(),
                LanguageServer.class,
                jdtProcess.getInputStream(),
                jdtProcess.getOutputStream()
            );
        this.languageServer = launcher.getRemoteProxy();
        this.textService = languageServer.getTextDocumentService();
        this.languageServer.initialize(
            new InitializeParams()
        ).get();
    }

    /**
     * 获取指定文件中所有符号的层级结构。
     */
    public CompletableFuture<List<DocumentSymbol>>
    getDocumentSymbols(String fileUri) {
        DocumentSymbolParams params = new DocumentSymbolParams(
            new TextDocumentIdentifier(fileUri));
        return textService.documentSymbol(params)
            .thenApply(result -> {
                if (result.isLeft()) {
                    return result.getLeft();
                }
                return List.of();
            });
    }

    /**
     * 获取光标位置处的类型信息和方法签名。
     */
    public CompletableFuture<Hover> getHoverInfo(
            String fileUri, Position position) {
        HoverParams params = new HoverParams(
            new TextDocumentIdentifier(fileUri), position);
        return textService.hover(params);
    }

    /**
     * 查找符号在代码库中的所有引用位置。
     */
    public CompletableFuture<List<Location>>
    findReferences(String fileUri, Position position) {
        ReferenceParams params = new ReferenceParams(
            new TextDocumentIdentifier(fileUri), position,
            new ReferenceContext(false));
        return textService.references(params);
    }
}

Spring AI 的上下文管理：Advisor 链中的压缩与注入

Spring AI 1.0.0-M6 引入的 Advisor 链机制是上下问管理的核心抽象。 Advisor 是类似 Servlet Filter 或 Spring AOP Around Advice 的拦截 器——在 ChatClient 的 prompt() 调用和 LLM 请求之间插入自定义处理逻 辑。CorpAssist 利用 Advisor 链实现三个关键功能：上下文压缩、敏感 信息过滤、动态 Prompt 注入。Advisor 链按照配置顺序链式执行，每个 Advisor 可以读取、修改或替换 prompt 中的 user text、system text、 检索上下文、对话历史等元素。整个链在 3-8ms 内完成（取决于 Advisor 数量），对端到端延迟影响极小。

上下文压缩 Advisor（ContextCompressionAdvisor）是最核心的组件。当 检索上下文超过 Token 预算时，它使用一个轻量级 LLM（如 GPT-4o-mini 或 DeepSeek-Coder-1.3B）对每个代码片段执行尽量保留语 义信息的压缩——删除注释、简化 import 语句、保留方法签名和关键控制 流。实测将 4000 Token 的检索上下文压缩至 1200 Token（压缩率 70%）， 回答准确率仅下降 1.8%（从 86.3% 降至 84.5%）。第二个关键 Advisor 是敏感信息过滤 Advisor（SensitiveFilterAdvisor），使用正则匹配和 NER 模型检测代码中的密码、Token、IP 地址、身份证号等敏感信息，在 送入 LLM 前替换为 [REDACTED]。第三个是动态 Prompt Advisor （DynamicPromptAdvisor），根据当前仓库的语言和框架自动注入项目规 则——例如检测到 spring-boot-starter-web 依赖时自动注入 Spring MVC 的编码规范。以下是三个 Advisor 的完整实现：

        StreamAroundAdvisor {

    private final int maxContextTokens;
    private final int warnThresholdPercent;

    public ContextCompressionAdvisor(int maxContextTokens,
                                      int warnThresholdPercent) {
        this.maxContextTokens = maxContextTokens;
        this.warnThresholdPercent = warnThresholdPercent;
    }

    @Override
    public String getName() {
        return "ContextCompressionAdvisor";
    }

    @Override
    public AdvisorResponse callAround(AdvisorMethodInvocation
            methodInvocation) {
        Prompt prompt = methodInvocation.getInput();
        int estimatedTokens = estimateTokens(
            prompt.getContents());
        int threshold = maxContextTokens *
            warnThresholdPercent / 100;
        if (estimatedTokens > threshold) {
            String compressed = compressContext(
                prompt.getContents(),
                (int)(threshold * 0.8));
            Prompt compressedPrompt = Prompt
                .builder()
                .messages(compressed)
                .build();
            return methodInvocation
                .next(compressedPrompt);
        }
        return methodInvocation.next(prompt);
    }

            AdvisorStreamMethodInvocation methodInvocation) {

                .messages(compressed)
                .build();
            return methodInvocation
                .next(compressedPrompt);

/**
 * 动态 Prompt Advisor。
 * 根据检测到的项目框架自动注入编码规范作为系统指令。
 */
@Component
public class DynamicPromptAdvisor
        implements CallAroundAdvisor, StreamAroundAdvisor {

    private final ProjectDetector projectDetector;

    public DynamicPromptAdvisor(ProjectDetector projectDetector) {
        this.projectDetector = projectDetector;
    }

    @Override
    public String getName() {
        return "DynamicPromptAdvisor";
    }

    @Override
    public AdvisorResponse callAround(
            AdvisorMethodInvocation methodInvocation) {
        Prompt prompt = methodInvocation.getInput();
        String projectType = projectDetector.detect();
        Map<String, Object> conventions =
            projectDetector.loadConventions(projectType);
        String conventionPrompt = buildConventionPrompt(
            projectType, conventions);
        Prompt enrichedPrompt = Prompt.builder()
            .messages(prompt.getInstructions())
            .system(conventionPrompt)
            .build();
        return methodInvocation.next(enrichedPrompt);
    }

    private String buildConventionPrompt(
            String projectType, Map<String, Object> conv) {
        return new SystemPromptTemplate("""
            当前项目类型: {projectType}
            编码规范:
            - 包命名: {packageNaming}
            - 异常处理: {exceptionHandling}
            - 日志框架: {loggingFramework}
            请严格遵循以上规范生成代码。
            """).render(Map.of(
                "projectType", projectType,
                "packageNaming", conv.getOrDefault(
                    "packageNaming", "com.company"),
                "exceptionHandling", conv.getOrDefault(
                    "exceptionHandling", "统一使用 GlobalExceptionHandler"),
                "loggingFramework", conv.getOrDefault(
                    "loggingFramework", "SLF4J + Logback")
            ));
    }
}

多轮对话中历史消息怎么管理？ContextCompressionAdvisor 如何处理累积的对话历史？

CorpAssist 的策略是"滑动窗口 + 摘要快照"。当对话轮次超过 5 轮 或历史消息 Token 数超过 2000 时，ContextCompressionAdvisor 触 发压缩。它将前 4 轮对话压缩为一个 300-500 Token 的摘要，保留 关键的用户意图、已尝试的方案、已确认的结论，然后仅保留最新一 轮的完整对话历史。这种策略在 10 轮深度对话中将 Token 消耗从 约 8000 Token 控制在 3000 Token 以内，同时维持 92% 的对话连 贯性（通过 200 个对话样本的人工评估）。注意摘要过程本身消耗 LLM 调用（使用 GPT-4o-mini 约 100 Token 输入 / 300 Token 输 出），每次压缩成本约 ¥0.002。对于高频对话场景，建议缓存摘要 结果避免重复压缩。

本章结论

  • Token 预算分配四部分（System/Context/History/Response）需按 照模型上限的 80% 规划，Context 部分采用按相关性排序策略部署注 意力资源。
  • 基于 import 图 BFS + 调用关系图的文件选择算法将上下文命中率 从 52.3% 提升至 84.7%，同时减少 55% 的 Token 消耗。
  • LSP 协议提供了比 TreeSitter 更深层的语义信息， documentSymbol 延迟 20-50ms、hover 延迟 15-30ms，通过预连接池 管理会话避免重建开销。
  • Spring AI Advisor 链为上下文压缩、敏感信息过滤、动态 Prompt 注入提供了清晰的拦截机制，链执行耗时仅 3-8ms。
  • 多轮对话的"滑动窗口 + 摘要快照"策略将 10 轮对话的 Token 消耗 从 8000 降至 3000，对话连贯性维持 92%，每次摘要成本 ¥0.002。

面试可能问

如果 128K Token 窗口仍然不够装下上下文，你会怎么做？

即使有 128K 窗口，也不建议全部用满，因为"中间迷失"问题会导致 中间位置的信息检索准确率下降 20-30%。如果上下文确实超过预算， 有三种策略可按优先级实施。策略一（推荐）：提高检索精度，使用 更高质量的 Reranker（如从 BGE-Reranker 升级到 Cohere Rerank 3） 并降低 Top-K 值。实测将 Top-K 从 10 降至 5，同时使用更好的 Reranker 后，回答准确率仅下降 2.3% 但 Token 消耗减少 52%。 策略二：启用分层检索——先检索文件级索引找到最可能的 3-5 个文件， 再在这些文件内部做函数级切片检索，避免全库检索的噪音。策略三： 使用 MapReduce 模式——将上下文拆分为多个块分别让 LLM 处理，然 后用另一个 LLM 调用合并各块的结果。这在代码审查场景中特别有 效：将大型 Diff 拆分为 5 个块并行审查，再汇总为最终 Review 报 告。CorpAssist 在代码库超 50 万行的项目中默认启用分层检索策略。

场景：团队抱怨代码问答的回答经常漏掉关键上下文，你从上下 文工程角度如何分析排查？

首先定位漏掉的上下文类型：是缺少了某个类的方法定义、缺少了调用 关系、还是缺少了配置信息？分别对应文件选择算法、调用图扩展、LSP 查询三个环节的排查方向。如果是文件级别遗漏：检查 import 图是否 正确构建——有些动态加载（Java SPI、Spring @Autowired 接口）在 import 图中不可见，需要在调用关系图中补充。如果是调用关系遗漏： 检查 TreeSitter 方法级 AST 解析的完整性，特别是有没有遗漏 lambda 表达式内的调用、Stream API 链式调用、Reflection API 调用。如果 是配置类遗漏：需要专门检索 application.yml 和 @ConfigurationProperties 类。根本解决方法是建立多级上下文融合机制：将文件级检索（主要）、 调用图扩展（补充）、配置检索（专项）、LSP 查询（语义增强）四个 通道的结果去重融合后排序。CorpAssist 采用此方案后上下文缺失率 从 22% 降至 6%。
