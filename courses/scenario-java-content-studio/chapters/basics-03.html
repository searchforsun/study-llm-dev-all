为什么学这章

AI 写 200 字的短文案已经足够好,但写 5000 字以上的长文（白皮书/行业报告/深度博客）仍然面临逻辑跳跃、内容重复和结构松散三大顽疾。CorpAssist 在处理金融机构的季度投资研报（平均 8000 字）时,发现未经结构规划的 AI 长文逻辑一致性评分仅 2.8/5,而采用了三阶段流水线后提升至 4.3/5。本章的三阶段流水线和多维质量评估方法是写好长文的关键。

学完后能做什么

实现大纲生成→分章节写作→拼接润色的三阶段长文流水线；使用 NLI 模型（自然语言推理）评估段落间的逻辑连贯性并生成衔接评分；实现基于 SimHash + 编辑距离的跨段落去重策略,将内容重复率控制在 3% 以下；使用四维质量评估指标（结构完整度/论据密度/可读性/原创度）量化长文质量；将长文生成流水线集成到 Spring Batch 的 Multi-step Job 中,支持 5000-15000 字的长文批量生成。

本章要点速记

1) 三阶段流水线:OutlineGenerator 生成大纲（含章节标题+子要点+关键词）→ ChapterWriter 逐章写作（每章独立调用 LLM,传入前章摘要保持上下文）→ ArticleMerger 拼接润色（消除过渡生硬和重复内容）；2) 逻辑连贯性检测基于 NLI 模型：使用 sentence-transformers 的 cross-encoder NLI 模型,计算相邻段落的 entailment（蕴含）和 contradiction（矛盾）概率,衔接评分 = P(entailment) / (P(entailment) + P(contradiction)),评分低于 0.6 时触发重写；3) SimHash 去重在 64 位指纹空间中检测高相似段落（海明距离 <3）,配合编辑距离做二次校验,误判率 <0.5%；4) 四维质量评估:结构完整度（章节覆盖率）、论据密度（事实+数据引用数/千字）、可读性（Flesch 阅读指数 60-70）、原创度（与已有内容的余弦相似度 <0.3）；5) Spring Batch 的多步骤 Job 天然适合三阶段流水线——每个阶段是一个独立的 Step,Step 之间通过 ExecutionContext 传递中间结果,支持断点续传和失败恢复。

大纲生成→分章节写作→拼接润色的三阶段流水线

长文生成面临的核心矛盾是:LLM 的上下文窗口有限（即使 GPT-4o 的 128K 窗口,处理 8000 字长文时中间部分的内容也会出现"迷失"）,且单次长输出容易产生逻辑漂移。CorpAssist 的三阶段流水线将这个矛盾分解为三个可管理的子问题。第一阶段是大纲生成（Outline Generation）:输入主题和 3-5 个关键词,让 LLM 生成一个包含 5-8 个章节的结构化大纲。每个章节包含标题、3-5 个子要点、目标字数（500-2000 字）以及 2-3 个引用的数据来源关键词。大纲以 JSON 格式输出（包含 outlineId、章节列表、总字数估计）,确保下游可以程序化解析。大纲生成使用的是低温度（temperature=0.3）、高 topP（0.95）的配置,优先保证结构的合理性和完整性,不过度追求创意。大纲生成后由人工或自动规则校验（章节数 ≥3、总字数 ≥3000、无重复标题）,校验通过进入第二阶段。

第二阶段是分章节写作（Chapter-by-Chapter Writing）:对大纲中的每个章节独立调用 LLM 生成完整内容。这里的关键设计是"上下文传递机制"——生成第 N 章时,将第 N-1 章的摘要（由 LLM 生成的 100 字概述）+ 第 N 章的大纲作为输入,确保章节间的逻辑过渡。上下文传递的格式为 "上章摘要: {previousSummary} → 本章大纲: {currentOutline} → 请基于以上背景开始写作"。这种机制避免了将整篇文章塞入一个 LLM 调用的窗口溢出问题,同时保持了跨章节的逻辑连贯性。分章节写作还支持并行加速——5-8 个章节可以在 RocketMQ 的异步任务中并行生成（每个章节一条消息）,将 8000 字长文的生成时间从串行的 8-12 分钟缩短到 3-5 分钟。并行生成时需要注意频率控制:对同一 LLM 后端的并发请求数控制在 5-10 个,避免触发 API 限流。每个章节生成完成后自动执行质量初筛（字数 >目标字的 50%、不含代码/HTML 标签、非空内容）,初筛不通过自动重试最多 2 次。

第三阶段是拼接润色（Merge & Polish）:将所有章节按照大纲顺序拼接,然后对整个长文做三次润色——第一次润色消除章节衔接处的过渡生硬（添加过渡句和承上启下段落）、第二次润色统一术语和风格（确保第一章和第五章对同一术语的使用一致）、第三次润色添加摘要和目录。拼接润色阶段需要引用 Spring Batch 的 Job 链机制——三个润色任务定义为三个连续的 Step,每个 Step 对整篇文章做一次遍历和修改,MergeProcessor 中通过正则和 LLM 调用实现。润色阶段的温度参数较低（0.2-0.4）,确保修改幅度可控,不会改变原文的事实和观点。CorpAssist 在金融研报场景的实测数据:8000 字长文的三阶段流水线总耗时约 5-8 分钟（大纲 30 秒 + 并行写作 3-5 分钟 + 拼接润色 1-2 分钟）,质量评分 4.3/5（直接生成的 8000 字长文在 3.0/5 左右）,篇章衔接评分从 0.55 提升至 0.82,重复内容率从 8% 降至 2%。

逻辑连贯性检测:基于 NLI 模型的段落衔接评分

长文写作最容易被忽略的问题是逻辑连贯性——单独看每个段落都写得不错,但段落之间缺乏逻辑过渡,读者会感到"跳来跳去"的割裂感。传统的语法检查工具（如 LanguageTool）只能处理句子级的问题,无法理解段落间的语义逻辑关系。CorpAssist 采用基于自然语言推理（NLI,Natural Language Inference）的段落衔接评分方法。NLI 模型的任务是判断两段文本之间的关系:蕴含（entailment,段落 B 可以从段落 A 逻辑推导出来）、矛盾（contradiction,段落 B 与段落 A 冲突）、中立（neutral,段落 B 与段落 A 无关或信息独立）。具体实现:将相邻的两个段落拼接为 "<paragraph_i> <sep> <paragraph_i+1>",输入到 cross-encoder NLI 模型（如 MoritzLaurer/mDeBERTa-v3-base-xnli 或中文场景的 Chinese-NLI）中,输出三个概率值 P(entailment)、P(contradiction)、P(neutral)。衔接评分公式为:衔接得分 = P(entailment) / (P(entailment) + P(contradiction)),取值 0-1,越接近 1 表示逻辑衔接越好。

CorpAssist 通过对 200 篇专业长文的人工标注确定了评分阈值:P(entailment) / (entailment+contradiction) > 0.8 为优秀衔接（人力标注的专家级长文平均分 0.85）,0.6-0.8 为可接受（需要轻微修改过渡句）,低于 0.6 为不合格衔接（需要重写段落边界部分）。以一篇"新能源汽车行业分析报告"为例:第 3 段讨论"锂电池技术路线",第 4 段突然跳到"充电桩基础设施建设",NLI 模型给出的 P(entailment)=0.32,P(contradiction)=0.28,P(neutral)=0.40,衔接得分=0.53,触发重写。重写建议是在第 3 段末尾添加过渡句"在电池技术路线确定后,配套的充电基础设施成为下一个关键议题——",然后将第 4 段开头调整为"充电桩基础设施的建设进度直接影响电池技术路线的商业化落地……",修改后衔接得分提升至 0.79。

NLI 检测的工程化实现需要注意三点:第一是性能——cross-encoder NLI 模型的推理延迟约 200-500ms/对,对于一篇 8000 字（约 20 个段落）的长文,需要检测 19 对相邻段落,总耗时约 4-10 秒。如果放在拼接润色阶段的 Step 中同步执行,会增加用户等待时间。建议在分章节写作阶段对每个章节内部的段落做 NLI 检测（更早发现问题）,在拼接润色阶段只检测跨章节边界的段落对（通常 5-8 对,2-4 秒）,并对分数低于 0.6 的段落触发二次润色。第二是模型选择——中文场景推荐使用 Chinese-NLI 或 mDeBERTa-v3 多语言模型,英文场景使用 Facebook/BART-large-MNLI。第三是缓存策略——相同或高度相似的段落对（SimHash 相似度 >0.9）可以直接使用缓存结果,减少重复计算。CorpAssist 的 NLI 缓存命中率约 35%,有效降低了检测延迟。

java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.stereotype.Service;
import java.util.*;
import java.util.stream.Collectors;
import java.util.stream.IntStream;

public record ParagraphPair(String left, String right, int leftIdx, int rightIdx) {}

public record CoherenceScore(String paragraphId, double entailmentProb,
    double contradictionProb, double coherenceScore, String suggestion) {}

@Service
public class CoherenceDetector {

    private final ChatClient chatClient;
    private static final double PASS_THRESHOLD = 0.6;
    private static final double EXCELLENT_THRESHOLD = 0.8;

    public CoherenceDetector(ChatClient.Builder builder) {
        this.chatClient = builder.build();
    }

    public List checkArticle(List paragraphs) {
        if (paragraphs.size() < 2) return List.of();

        List pairs = IntStream.range(0, paragraphs.size() - 1)
            .mapToObj(i -> new ParagraphPair(
                paragraphs.get(i), paragraphs.get(i + 1), i, i + 1))
            .collect(Collectors.toList());

        return pairs.parallelStream()  // 并行检测提速 3-5 倍
            .map(this::evaluatePair)
            .collect(Collectors.toList());
    }

    private CoherenceScore evaluatePair(ParagraphPair pair) {
        String prompt = """
            请分析以下两段文本之间的逻辑关系。
            判断第二段是否可以从第一段逻辑推导出来(蕴含)、
            是否与第一段矛盾、或者两者无关(中立)。

            第一段: %s
            第二段: %s

            请输出JSON格式:
            {"entailment": 0.0-1.0, "contradiction": 0.0-1.0,
             "neutral": 0.0-1.0, "suggestion": "改进建议"}
            """.formatted(pair.left(), pair.right());

        String response = chatClient.prompt()
            .user(u -> u.text(prompt))
            .call()
            .content();

        CoherenceResult result = parseResult(response);
        double coherenceScore = result.entailment /
            (result.entailment + result.contradiction + 0.0001);

        return new CoherenceScore(
            "p%d-p%d".formatted(pair.leftIdx(), pair.rightIdx()),
            result.entailment, result.contradiction,
            coherenceScore, result.suggestion);
    }

    public boolean needsRewrite(CoherenceScore score) {
        return score.coherenceScore() < PASS_THRESHOLD;
    }

    public List filterNeedsRewrite(List scores) {
        return scores.stream()
            .filter(this::needsRewrite)
            .collect(Collectors.toList());
    }

    private CoherenceResult parseResult(String json) {
        // 简单 JSON 解析 (生产环境使用 Jackson)
        double e = extractDouble(json, "entailment");
        double c = extractDouble(json, "contradiction");
        double n = extractDouble(json, "neutral");
        String s = extractString(json, "suggestion");
        return new CoherenceResult(e, c, n, s);
    }

    // 工具方法
    private double extractDouble(String json, String key) { return 0.5; }
    private String extractString(String json, String key) { return ""; }

    record CoherenceResult(double entailment, double contradiction,
                           double neutral, String suggestion) {}
}

基于 SimHash + 编辑距离的跨段落去重策略

长文写作的另一个常见问题是内容重复——同一个观点、数据或案例在文章的不同位置被反复提及。LLM 生成的长文中尤其容易出现这种情况,因为模型在生成后文时"忘记"了前文已经写过的内容。传统的去重方法（如 Jaccard 相似度）在大规模文本上计算效率低下,且对同义改写（paraphrase）的检测效果不佳。CorpAssist 采用 SimHash + 编辑距离的双层去重策略:第一层使用 SimHash 做快速粗筛,第二层对疑似重复的段落对使用编辑距离（Levenshtein Distance）做精确确认。SimHash 是 Google 提出的局部敏感哈希算法,可以将任意长度的文本映射为固定长度的哈希值（CorpAssist 使用 64 位）,并且语义相似的文本会得到海明距离相近的哈希值。SimHash 的计算过程:输入文本分词后,对每个词计算 64 位哈希,加权累加（词频作为权重）,最后对每个位取符号（正为 1 负为 0）得到最终的 64 位哈希值。

去重流程分四步。第一步将长文的每个段落（按自然段或 h2/h3 分割的子段落）计算 64 位 SimHash 值。第二步将所有段落的 SimHash 两两比较海明距离（异或运算后统计 1 的个数）,距离小于阈值 D（默认设为 3,调优范围 2-5）的段落对标记为"候选重复"。64 位海明距离的计算使用位运算优化——一条长文 20 个段落,生成 190 个候选对（C(20,2)）,SimHash 比较耗时约 0.5ms,总耗时 <100ms。第三步对候选重复的段落对计算编辑距离（Levenshtein）,编辑距离比上较长段落的字符数得到归一化重复率。当归一化重复率 >0.3 时判定为"确认重复"。第四步根据重复程度给出处理建议:重复率 0.3-0.5 标记为"建议合并",重复率 >0.5 标记为"强制删除",重复率 >0.8 标记为"完全重复段落可直接移除"。CorpAssist 在 500 篇测试长文上的评估:SIMHash 粗筛的召回率 97.2%（漏掉少量同义改写程度极高的重复对）,编辑距离二次确认后将精确率从 82% 提升至 99.5%,误判率低于 0.5%。

工程化实现的关键优化:SimHash 计算可以批量并行化——使用 Java 的 parallelStream 或虚拟线程,10 个段落的哈希计算从 100ms 降到 15ms。海明距离的比较使用位运算 + 预计算表（预置 65536 个 16 位整数的汉明重量）将每次比较从 64 次位运算优化为 4 次表查询。编辑距离的实现使用滚动数组（只保留两行,空间复杂度 O(min(m,n))）,避免 O(m×n) 的完整矩阵内存消耗。去重逻辑封装在 MergeProcessor 中,作为拼接润色阶段的一个子步骤:首先扫描整篇文章的重复段落,生成去重建议列表,然后通过人工审核或自动规则（重复率 >0.5 自动删除,0.3-0.5 标记待人工确认）执行去重操作。最终的重复率指标必须低于 3% 才能通过质量终检。

java
import java.util.*;
import java.util.stream.IntStream;

public class SimHashDedup {

    private static final int HASH_BITS = 64;
    private static final int SIMHASH_THRESHOLD = 3;     // 海明距离阈值
    private static final double EDIT_DIST_THRESHOLD = 0.3; // 编辑距离阈值

    private static final int[] BIT_COUNTS = new int[65536];

    static {
        for (int i = 0; i < 65536; i++) {
            BIT_COUNTS[i] = Integer.bitCount(i);
        }
    }

    public record DedupResult(int sourceIdx, int targetIdx,
        int hammingDist, double normalizedEditDist, String action) {}

    public List findDuplicates(List paragraphs) {
        long[] simhashes = paragraphs.parallelStream()
            .mapToLong(this::computeSimHash)
            .toArray();

        List results = new ArrayList<>();

        // 两两比较 SimHash 海明距离
        for (int i = 0; i < paragraphs.size(); i++) {
            for (int j = i + 1; j < paragraphs.size(); j++) {
                int hamming = hammingDistance(simhashes[i], simhashes[j]);
                if (hamming < SIMHASH_THRESHOLD) {
                    // 候选重复: 计算编辑距离确认
                    String longer = paragraphs.get(i).length() >=
                        paragraphs.get(j).length() ? paragraphs.get(i) : paragraphs.get(j);
                    String shorter = longer == paragraphs.get(i) ?
                        paragraphs.get(j) : paragraphs.get(i);
                    int editDist = levenshteinDistance(longer, shorter);

                    double normalized = (double) editDist / longer.length();
                    String action = normalized > 0.5 ? "FORCE_DELETE" :
                        normalized > 0.3 ? "SUGGEST_MERGE" : "KEEP";

                    results.add(new DedupResult(i, j, hamming, normalized, action));
                }
            }
        }
        return results;
    }

    private long computeSimHash(String text) {
        // 1. 分词 (实际使用分词器,这里简化)
        String[] words = text.split("[\\s,，。；;：:！!？?]+");
        int[] bits = new int[HASH_BITS];

        for (String word : words) {
            long wordHash = fnv1a64(word);
            int weight = word.length();  // 词频作为权重,简化
            for (int b = 0; b < HASH_BITS; b++) {
                if ((wordHash & (1L << b)) != 0) {
                    bits[b] += weight;
                } else {
                    bits[b] -= weight;
                }
            }
        }

        long simhash = 0;
        for (int b = 0; b < HASH_BITS; b++) {
            if (bits[b] > 0) simhash |= (1L << b);
        }
        return simhash;
    }

    private int hammingDistance(long a, long b) {
        long xor = a ^ b;
        // 使用预计算表加速: 每 16 位查一次表
        return BIT_COUNTS[(int)(xor & 0xFFFF)]
             + BIT_COUNTS[(int)((xor >>> 16) & 0xFFFF)]
             + BIT_COUNTS[(int)((xor >>> 32) & 0xFFFF)]
             + BIT_COUNTS[(int)((xor >>> 48) & 0xFFFF)];
    }

    private int levenshteinDistance(String a, String b) {
        int m = a.length(), n = b.length();
        if (m < n) return levenshteinDistance(b, a);
        int[] prev = new int[n + 1];
        int[] curr = new int[n + 1];

        for (int j = 0; j <= n; j++) prev[j] = j;

        for (int i = 1; i <= m; i++) {
            curr[0] = i;
            for (int j = 1; j <= n; j++) {
                int cost = a.charAt(i - 1) == b.charAt(j - 1) ? 0 : 1;
                curr[j] = Math.min(Math.min(
                    curr[j - 1] + 1,       // insert
                    prev[j] + 1),           // delete
                    prev[j - 1] + cost);    // replace
            }
            int[] tmp = prev; prev = curr; curr = tmp; // 滚动数组
        }
        return prev[n];
    }

    private long fnv1a64(String word) {
        long hash = 0xCBF29CE484222325L;
        for (char c : word.toCharArray()) {
            hash ^= c;
            hash *= 0x100000001B3L;
        }
        return hash;
    }
}
SimHash 和编辑距离的结合使用有什么优缺点？

SimHash 的核心优势是计算效率极高——一篇 8000 字长文（20 段落）的 SimHash 计算 + 海明距离比较总耗时小于 100ms,而纯编辑距离做全文两两比较需要 200-500ms。SimHash 的弱项是对短文本（<50 字）的哈希质量不稳定,以及同义改写（paraphrase）场景下的漏检率约 3%。编辑距离作为二次确认弥补了 SimHash 的精度问题——将精确率从 82% 提升到 99.5%。缺点是编辑距离的计算复杂度为 O(m×n),对于超长段落（1000+ 字）可能需要 10-50ms。优化策略是:SimHash 海明距离 <3 的段落对才进入编辑距离计算,这样 190 个候选对中通常只有 5-15 对需要编辑距离计算,将总耗时控制在 150ms 以内。另外,编辑距离仅适用于同语言文本（中日韩等字符文本）,对于多语言混合文本,建议使用基于 token 的编辑距离（如对 LLM 的 tokenization 结果计算距离）。

长文质量评估的四维指标

长文质量不能仅靠字数或语法来评判——一篇 8000 字的语法正确但观点空洞的文章仍然是垃圾内容。CorpAssist 建立了四维质量评估体系,从结构完整度、论据密度、可读性和原创度四个维度量化长文质量。第一维是结构完整度（Structural Completeness）,衡量长文是否覆盖了大纲中约定的所有章节和要点。计算方法:结构完整度 = (已覆盖章节数 / 总章节数) × 0.6 + (已覆盖子要点数 / 总子要点数) × 0.4。如果大纲约定 6 个章节共 22 个子要点,实际生成了 6 个章节但只覆盖了 18 个子要点,则结构完整度 = 6/6×0.6 + 18/22×0.4 = 0.6 + 0.327 = 0.927。低于 0.8 的触发"章节缺失"告警。第二维是论据密度（Evidence Density）,衡量文章的信息价值——论据密度 = (事实陈述数 + 数据引用数 + 案例引用数) / 千字。CorpAssist 通过 LLM-as-Judge 识别和计数文中的事实陈述（如"2025 年全球新能源汽车销量突破 2000 万辆"）和数据引用（如"同比增长 35%"）,并除以千字单位。行业基准:深度分析类文章 >8 个/千字为高质量,4-8 为中等,<4 为信息稀疏需要补充论据。

第三维是可读性（Readability）,衡量文章的阅读难度和流畅度。中文场景下,CorpAssist 使用改良的 Flesch 阅读指数——取句均长度（字/句,推荐 20-35）、字均笔画数（推荐 <8）和被动语态频率（每百句 <5）三个子指标加权计算。其中句均长度权重最高（50%）,因为长句是中文可读性的最大杀手——句长超过 60 字时读者的理解率下降 40%。可读性评分范围 0-100,60-70 为"标准"（适合大众阅读）,70-80 为"较易"（适合科普内容）,50-60 为"较难"（适合专业读者）。第四维是原创度（Originality）,衡量内容与已有知识库的相似程度。原创度 = 1 - max(与知识库中每篇内容的余弦相似度)。使用 text2vec 模型将全文编码为 768 维向量,与 CorpAssist 的内容知识库（存储已有文章、竞品文章和行业报告）中的向量做余弦相似度比较。原创度低于 0.7 时认为存在过度模仿或抄袭风险,标记为"需人工审核"。四个维度的综合质量评分 = 0.25 × 结构完整度 + 0.25 × 论据密度归一化值 + 0.25 × 可读性/100 + 0.25 × 原创度。综合评分低于 0.7 的长文不允许上线,需触发重写流程。

  • 三阶段流水线（大纲 30s + 并行写作 3-5min + 拼接润色 1-2min）将 8000 字长文质量评分从 3.0/5 提升至 4.3/5,段落衔接评分从 0.55 提升至 0.82

  • Spring Batch 多步骤 Job 天然适配三阶段流水线,每个阶段独立 Step 并通过 ExecutionContext 传递大纲/章节/润色结果

面试常见问答

长文（5000 字+）生成如何保证逻辑连贯性？具体方案和数据是什么？

保证长文逻辑连贯性的核心方案是三阶段流水线 + NLI 检测。三阶段流水线将长文生成拆解为大纲生成、分章节写作和拼接润色三个可管理的子问题。大纲阶段使用低温度(0.3)生成包含 5-8 章节的 JSON 结构化大纲,每个章节含标题、子要点和数据源。分章节写作阶段的关键是上下文传递机制——生成第 N 章时传入第 N-1 章的摘要（100 字概述）和本章大纲,确保章节间的逻辑过渡。5-8 个章节通过 RocketMQ 异步并行生成（5 并发）,将总耗时从 8-12 分钟压缩到 3-5 分钟。拼接润色阶段做三次迭代:消除过渡生硬、统一术语风格、生成摘要目录。逻辑连贯性的量化检测使用 NLI 模型:将相邻段落输入 cross-encoder NLI 模型,计算衔接得分 = P(entailment) / (P(entailment) + P(contradiction)),低于 0.6 触发段落边界的重写。CorpAssist 在金融研报场景的实测数据显示:采用三阶段流水线后段落衔接评分从 0.55 提升至 0.82,直接生成的成功率从 68% 提升至 99.5%,质量评分从 3.0/5 提升至 4.3/5。

如何检测和消除长文中的内容重复？SimHash 的工作原理是什么？

内容重复检测采用 SimHash + 编辑距离的双层策略。SimHash 是 Google 提出的局部敏感哈希算法:输入文本分词后对每个词计算 64 位 FNV 哈希,按词频加权累加,最后取符号生成 64 位指纹。语义相似的文本会得到海明距离相近的指纹——这是 SimHash 区别于传统哈希（像 md5,输入微小变化输出完全不同）的核心特性。检测流程:每个段落计算 64 位 SimHash → 两两比较海明距离（异或位运算,通过预计算 65536 个 16 位整数的汉明重量表加速）→ 距离小于阈值 3 的标记为候选重复 → 对候选对计算 Levenshtein 编辑距离 → 归一化重复率超过 0.3 确认为重复。实测数据:10 个段落的 SimHash 计算 + 45 次海明距离比较总耗时 <50ms,编辑距离二次确认将精确率从 82% 提升到 99.5%,误判率低于 0.5%。去重处理建议:重复率 0.3-0.5 建议合并,>0.5 强制删除,>0.8 直接移除。处理后的内容重复率必须低于 3% 才能通过质量终检。

如何量化评估长文的质量？四个维度的具体计算方法和行业基准是什么？

长文质量评估采用四维指标体系。第一维结构完整度 = 已覆盖章节数/总章节数×0.6 + 已覆盖子要点数/总子要点数×0.4,高质量基准 >0.95,达标阈值 0.80。第二维论据密度 = (事实陈述数 + 数据引用数 + 案例引用数) / 千字,通过 LLM-as-Judge 自动提取计数,深度分析类文章 >8 个/千字为高质量,4-8 为中等,<4 为信息稀疏。第三维可读性使用改良 Flesch 指数——权重最高的是句均长度（20-35 字为佳,句长超过 60 字时理解率下降 40%）,结合字均笔画（推荐 <8）和被动语态频率（每百句 <5）,评分 60-70 为适合大众阅读的标准水平。第四维原创度 = 1 - max(与知识库文章的余弦相似度),使用 text2vec 768 维向量做全文检索,<0.7 标记为需人工审核。综合评分 = 四个维度加权平均（各 25%）,低于 0.7 的文章不允许上线。这个体系需要定期校准——每季度收集 100 篇人工评分和自动评分的对比数据,计算 Pearson 相关系数并调整权重。

场景判断 1: 长文生成失败

一篇 8000 字的行业分析报告在分章节写作阶段第 5 章（共 7 章）生成失败（LLM API 超时）。请判断应该如何处理?
正确方案: 利用 Spring Batch 的 ExecutionContext 存储已成功生成的前 4 章内容,对第 5 章进行单独重试（最多 3 次,每次间隔 5 秒）,成功后续继续第 5→第 6→第 7 章的"上章摘要传递"链,第 6 章和第 7 章的上下文基于第 4 章摘要+第 5 章重试结果。

场景判断 2: 重复率过高

一篇 5000 字的科普文章经过三阶段流水线生成后,SimHash+编辑距离检测发现重复率 7.2%（阈值 3%）,其中 3 处为"数据引用完全重复"（同一组数据在第 2、4、6 段重复出现）,2 处为"观点同义表述"。请判断去重执行策略。
正确方案: 对 3 处完全重复的数据引用,保留首次出现的位置并在后续出现处替换为"如前文所述"+具体章节引用（如"参见第 2 章市场分析"）。对 2 处同义表述,根据上下文保留信息更完整、措辞更准确的一个版本,合并另一个版本。处理后重复率应降至 1% 以下。
