为什么学这章

企业内容最常被诟病的问题不是"写得不准确",而是"不像我们品牌的风格"。CorpAssist 在服务零售客户时发现,同一个产品描述由 5 个不同编辑撰写,风格偏差可达 18-25%,这对品牌一致性是致命打击。本章的品牌语调五维参数化模型和模板版本管理体系,可以量化和固化品牌风格,将风格偏差控制在 5% 以内。你将掌握从 Prompt 工程到 Spring 模板引擎的完整风格控制链路。

学完后能做什么

使用五维语调参数模型（正式度/情感度/专业度/亲和力/创意度）对任意品牌风格进行量化描述；构建和管理企业级 Few-shot 示例库,并实现基于向量相似度的动态示例选择；设计基于 Git 的 Prompt 模板版本管理和 A/B 测试框架；实现 CorpAssist 营销模板库的完整 Schema,支持多语言、多品类和多渠道适配；将模板注入 Spring 内容流水线,实现模板驱动的批量内容生成。

本章要点速记

1) 品牌语调五维参数模型将抽象的品牌风格量化为[0,10]区间的数值,每个维度对应具体的 Prompt 控制策略和 LLM 参数调节方式；2) Few-shot 示例库存储 50-200 条人工精标示例,通过文本 Embedding 余弦相似度动态选择 Top-3 示例注入 Prompt；3) Prompt 模板采用 Git 版本管理（SemVer 语义化版本号）,配合 Spring 配置中心实现灰度发布和 A/B 测试；4) CorpAssist 模板库 Schema 采用三层嵌套结构:品牌基调层 → 品类模板层 → 渠道适配层,支持 50+ 品牌和 200+ 品类的灵活配置；5) SpEL 条件渲染支持变量级别的内容定制,如根据 product.price 自动选择"高端/中端/性价比"三种描述风格。

品牌语调五维参数化模型

品牌语调（Brand Tone）是内容质量的第一道关卡,也是 AI 生成内容最容易失分的地方。传统的 Prompt "请用专业且生动的语气撰写产品描述"过于模糊,LLM 无法精确理解"专业"和"生动"分别意味着什么。CorpAssist 的品牌语调五维参数化模型将抽象的风格描述拆解为五个可量化的维度:正式度（Formality,0-10,0=口语化如"亲,这个超好用哦！",10=公文风格如"经测试,该产品性能指标达到行业领先水平"）、情感度（Emotion,0-10,0=中性客观不带任何情感色彩,10=高情感比如"我们深信这款产品将彻底改变您的生活方式"）、专业度（Expertise,0-10,0=大众通俗语言,10=专业术语密集如"该算法的时间复杂度为 O(log n),空间复杂度为 O(1)"）、亲和力（Warmth,0-10,0=冷冰冰的机构语气如"根据第 8 条第 3 款",10=朋友般的语气如"我们理解这可能会让您感到困惑,请随时联系我们"）、创意度（Creativity,0-10,0=完全模板化标准化表达,10=高度创意如运用隐喻拟人等修辞手法）。每个维度分为三个区间:0-3（保守风格,适合正式合同/法律声明/监管文件）、4-7（中性风格,适合产品说明/博客/内部通知）、8-10（激进风格,适合广告创意/社媒互动/品牌故事）。

五维参数映射到 LLM 调用策略上,每个维度影响不同的参数和 Prompt 指令。正式度通过调整温度参数 temperature（0.2-0.5 对应正式,0.7-1.0 对应口语化）和 Top-p（0.8-0.95）来控制语言的确切程度。情感度通过 system prompt 中的情绪标记词来实现,高情感度时注入如"请使用积极、温暖的语调,多使用感叹句和排比句"的指令。专业度通过控制术语密度——低专业度时在 Prompt 中加入"请用初中生能理解的语言解释",高专业度时加入"请使用行业标准术语,首次出现时给出括号注释"。亲和力通过人称代词和语气词的使用频率控制——高亲和力时要求使用"我们""您"等第二人称并加入语气词。创意度通过 few-shot 示例的选择来控制——低创意度时提供标准化模板示例,高创意度时提供修辞丰富的示例。实践经验表明,五维参数全部暴露给用户层的 UI 编辑成本太高,建议在 CorpAssist 的后台管理中预设 3-5 个品牌语调模板（如"高端专业型"对应 8/3/9/5/4,"亲和友新型"对应 4/8/3/9/7）,内容编辑只需要选择预设模板,无需单独调整每个维度。

五维模型不仅用于 Prompt 控制,还与内容质量的评估指标直接挂钩。内容审核阶段,LLM-as-Judge 会分别评估生成内容在五个维度上的得分,与目标分数的偏差超过 1.5 分的维度会被标记为"语调偏差"并触发驳回-重写。CorpAssist 的实际运营数据显示,引入五维语调模型后,内容的品牌一致性从 72%（编辑凭直觉判断）提升至 93%,驳回率从 18% 降低至 7%,风格偏差从 18-25% 缩小到 4-8%。五个维度的偏差分布中,创意度的偏差最大（平均 1.8 分）,因为 LLM 往往倾向于过度创意；正式度偏差最小（平均 0.6 分）,因为温度参数直接控制了语言的确切程度。

CorpAssist 实战:美妆品牌语调定义

某国际美妆品牌进入中国市场,需要在微信/小红书/天猫三个渠道保持一致的"轻奢专业"品牌语调。CorpAssist 为其定义的五维参数:正式度 6（中文优雅但不古板）、情感度 7（温暖积极但不过度热情）、专业度 8（强调成分科学背书）、亲和力 6（专业但不冷冰冰）、创意度 5（有美感但不过度修饰）。三个渠道的语调略有调整:微信 6/8/6/7/6、小红书 4/9/5/9/8（更口语化和情感化）、天猫 7/5/9/5/4（更专业和正式）。模板库共配置 120 个模板（40 个品牌故事 + 50 个产品描述 + 30 个促销文案）,每个模板自动适配三个渠道的语调参数。上线后品牌一致性从 65% 提升至 90%,小红书渠道的互动率提升 42%。

Few-shot 示例库的构建与动态选择策略

Few-shot 示例是 Prompt 工程中控制输出质量最有效的手段之一——在 Prompt 中提供 3-5 个人工精写示例,LLM 可以模仿示例的格式、风格和内容结构。但简单地随机选择示例无法适配不同的内容类型和品牌语调,甚至可能产生反效果（比如为"促销活动"内容提供了"客户感谢信"的示例）。CorpAssist 的 Few-shot 示例库采用三层构建策略:第一层是示例池层（Example Pool）,存储 50-200 条经人工审核的高质量示例,每条示例包含"内容文本 + 元数据标签 + Embedding 向量"三部分。元数据标签包括:内容类型（product_desc/marketing/email/faq/post）、品牌语调五维向量、目标受众（B2B/B2C/Internal）、字数范围、关键词列表。Embedding 向量通过 text2vec-large-chinese 模型生成 768 维向量,用于语义相似度检索。第二层是选择策略层（Selection Strategy）,根据输入内容的需求标签,从示例池中筛选出最匹配的 Top-K 条示例。筛选逻辑分两步:先通过标签过滤（确保内容类型匹配,如"产品描述"只匹配同类示例）,然后通过向量余弦相似度排序选择 Top-3。

第三层是动态适配层（Dynamic Adaptation）,在运行时动态调整示例的选择权重。CorpAssist 的 Selection Engine 实现了一个加权融合公式:总得分 = 0.4 × 类型匹配度 + 0.3 × 品牌语调余弦相似度 + 0.2 × 受众匹配 + 0.1 × 字数接近度。类型匹配度是硬性过滤条件（不匹配直接排除）,后三个维度是排序依据。例如输入"生成一款售价 ¥2,999 的扫地机器人的天猫产品描述,品牌语调为 7/6/8/5/4,B2C 受众,要求 200-300 字"。引擎会过滤出所有 type=product_desc 的示例,然后计算每个示例的品牌语调向量与目标语调向量的余弦相似度,最终选择得分最高的 3 条注入 Prompt。如果得分最高的示例得分低于 0.5,引擎会触发一个"示例不足"告警,通知管理员手动补充适配示例。线上运营数据显示,动态选择策略相比随机选择,生成内容的用户满意度从 3.1/5 提升至 4.3/5,一次通过率从 45% 提升至 78%。

示例池的维护需要建立闭环:内容审核通过后,如果编辑对 AI 初稿的修改量超过 30%,系统会自动将该篇内容（编辑修改版）标记为"候选示例",提交到人工示例审核队列。运营人员每周审核一次候选示例池,将高质量示例纳入正式示例池（并打上标签）,将低质量示例淘汰。这种"生成→修改→沉淀"的飞轮机制使得示例池每两周增长 5-10 条,质量持续提升。CorpAssist 的经验表明,示例池在达到 80 条之后,新增示例对质量提升的边际效应递减,因此建议将示例池上限控制在 200 条,并定期淘汰 60 天未匹配的冷门示例。

java
import org.springframework.ai.embedding.EmbeddingClient;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.stereotype.Service;
import java.util.*;
import java.util.stream.Collectors;

public record ExampleItem(String id, String contentType, String content,
    double[] toneVector, String audience, int wordCount, float[] embedding) {}

public record SelectionInput(String contentType, double[] targetTone,
    String audience, int targetWordCount, double minSimilarity) {}

@Service
public class FewShotSelector {

    private final EmbeddingClient embeddingClient;
    private final RedisTemplate redisTemplate;
    private static final int TOP_K = 3;
    private static final int MAX_POOL_SIZE = 200;
    private static final double[] TYPE_WEIGHTS = {0.4, 0.3, 0.2, 0.1};

    public FewShotSelector(EmbeddingClient embeddingClient,
                           RedisTemplate redisTemplate) {
        this.embeddingClient = embeddingClient;
        this.redisTemplate = redisTemplate;
    }

    public List select(SelectionInput input) {
        // 1. 从 Redis 加载所有示例（缓存 5 分钟）
        List pool = loadExamplePool();

        // 2. 类型硬过滤
        List filtered = pool.stream()
            .filter(e -> e.contentType().equals(input.contentType()))
            .collect(Collectors.toList());

        if (filtered.isEmpty()) {
            throw new IllegalStateException("示例池空: type=" + input.contentType());
        }

        // 3. 加权评分排序
        List scored = filtered.stream()
            .map(e -> new ScoredExample(e, computeScore(e, input)))
            .sorted((a, b) -> Double.compare(b.score(), a.score()))
            .collect(Collectors.toList());

        // 4. 阈值检查
        if (scored.isEmpty() || scored.get(0).score() < input.minSimilarity()) {
            // 触发告警通知管理员补充示例
            notifyExampleShortage(input.contentType(), scored.isEmpty()
                ? 0.0 : scored.get(0).score());
        }

        return scored.stream()
            .limit(TOP_K)
            .map(ScoredExample::example)
            .collect(Collectors.toList());
    }

    private double computeScore(ExampleItem example, SelectionInput input) {
        double toneSim = cosineSimilarity(example.toneVector(), input.targetTone());
        double audienceMatch = example.audience().equals(input.audience()) ? 1.0 : 0.3;
        double wordRatio = Math.min(example.wordCount(), input.targetWordCount())
            / (double) Math.max(example.wordCount(), input.targetWordCount());

        return TYPE_WEIGHTS[0] * 1.0   // 类型过滤已做,满分
             + TYPE_WEIGHTS[1] * toneSim
             + TYPE_WEIGHTS[2] * audienceMatch
             + TYPE_WEIGHTS[3] * wordRatio;
    }

    private double cosineSimilarity(double[] a, double[] b) {
        double dot = 0.0, normA = 0.0, normB = 0.0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            normA += a[i] * a[i];
            normB += b[i] * b[i];
        }
        return dot / (Math.sqrt(normA) * Math.sqrt(normB));
    }

    private void notifyExampleShortage(String contentType, double bestScore) {
        // 发送告警通知: 邮件/钉钉/企业微信
        System.out.printf("[ALERT] 内容类型 %s 最优示例得分 %.3f 低于阈值 0.5,请补充示例%n",
            contentType, bestScore);
    }

    private List loadExamplePool() {
        // 从 Redis 缓存加载,缓存 key="fewshot:pool",过期时间 300 秒
        return Optional.ofNullable(
            redisTemplate.opsForList().range("fewshot:pool", 0, MAX_POOL_SIZE))
            .orElse(List.of());
    }

    record ScoredExample(ExampleItem example, double score) {}
}
示例库需要多少条示例才能达到可接受的效果？

基于 CorpAssist 在 37 个品牌上的实践经验,最低有效规模是每个内容类型至少 10 条示例（覆盖 3 种不同的语调分布）,推荐规模是每个类型 20-30 条。当示例池总量达到 80 条时,生成质量进入稳定期（用户满意度 >4.0/5）,新增示例的边际收益显著递减。需要注意的是,示例的质量远重要于数量——1 条精心挑选和标注的示例价值超过 5 条随机选取的示例。因此建议优先确保每条示例都经过人工精写和审核,而不是追求数量。示例的淘汰策略也很关键:建议每季度执行一次淘汰,移除 60 天内未被匹配到的冷门示例,为新示例腾出空间。内容类型变化时（如新增了"视频脚本"类型）,需要立即补充至少 5 条该类型的初始示例,否则动态选择引擎会触发告警。

Prompt 模板的 Git 版本管理与 A/B 测试框架

Prompt 模板是内容系统的核心资产,也是持续的迭代对象——一个新的品牌语调可能需要调整 10-15 个模板的 Prompt 措辞,一个 LLM 模型升级可能需要修改 20+ 模板的温度参数。CorpAssist 将 Prompt 模板视为代码一样进行版本管理,采用 Git 仓库 + SemVer 语义化版本号的方案。每个模板是一个独立的 YAML 文件,包含元数据（templateId、version、author、created、lastModified）、System Prompt（角色定义 + 行为约束 + 品牌语调指令）、User Prompt（SpEL 模板,包含变量占位符如 {{productName}} 和条件渲染如 <#if price > 5000>高端产品</#if>）、LLM 参数（temperature/topP/maxTokens/stopSequences）以及示例引用列表（ExampleRef — 引用的 Few-shot 示例 ID 列表）。模板的 YAML 文件存储在 Git 仓库的 templates/ 目录下,按类型划分子目录（templates/product_desc/,templates/email/,templates/social/）,模板文件命名规范为 {type}-{channel}-{version}.yaml,如 product_desc-tmall-v2.3.1.yaml。

模板的 A/B 测试框架基于 Spring 的配置中心（Spring Cloud Config）实现。当需要测试一个新的 Prompt 变体时,运维人员在 Spring Cloud Config 的 Git 仓库中新增一个配置项:template.XX.abTest.enabled=true,并指定分流比例（如 A 版本 50%、B 版本 50%。分流基于用户 ID 的哈希值,确保同一用户始终看到同一版本。实验的流量入口在 Spring Cloud Gateway 层通过自定义过滤器实现——Gateway 从请求头中提取 userId,计算 hash(userId) % 100,如果值在 0-49 则路由到 A 版本的模板服务,50-99 则路由到 B 版本。实验结果（生成质量评分/用户点击率/转化率）通过埋点数据采集后,在 Grafana 面板上实时展示两个版本的对比,实验周期通常为 1-2 周。当实验组显著优于对照组（p < 0.05,t-test）时,将 B 版本合并为正式版本。CorpAssist 的经验:一次模板迭代平均提升质量评分 0.15-0.3 分（5 分制）,A/B 测试将迭代风险从"全量上线可能出问题"降低到"最多影响 50% 流量且可快速回滚"。

CorpAssist 营销模板库的完整 Schema 设计

CorpAssist 营销模板库是整个内容工坊的"内容图纸",决定了最终产出的格式、变量和风格。模板库采用三层嵌套 Schema 设计:第一层是品牌基调层（Brand Tone Profile）,定义品牌的全局语调参数、Logo/品牌色、目标客群画像、禁止词列表（如竞品品牌名、行业敏感词）和 SEO 关键词集合。每个品牌对应一个 BTP 文件（JSON/YAML）,存储在 PostgreSQL 的 brand_tone 表中,通过 Redis 缓存加速读取。第二层是品类模板层（Category Template）,针对品牌下的每个产品品类（如手机/笔记本/耳机）定义标准化的内容结构,包括段落顺序（标题→卖点→参数→评价→购买引导）、必含字段（如手机品类的"芯片/摄像头/电池"三项必须包含）、变量定义（变量名、类型、校验规则、默认值）。品类模板层的核心数据结构是 TemplateEntity,包含 templateId、brandId、category、channels（List）、variables（Map<String, VariableSchema>）、llmParams 等字段。第三层是渠道适配层（Channel Adapter）,针对同一品类在不同渠道（天猫/京东/微信/小红书）的差异化要求,定义渠道特有的格式约束（如天猫要求尺寸 750×400px,小红书要求 1:1 方形图）、字符限制、SEO 规则和小程序链接。渠道适配层通过 AOP 拦截器在生成前注入渠道特有的格式化指令。

模板的变量体系是 Schema 设计的核心复杂性所在。每个变量需要定义:name（变量名,如 productName）、type（数据类型: STRING/NUMBER/BOOLEAN/IMAGE/LIST）、required（是否必填,默认 true）、maxLength（最大字符数,用于控制生成内容长度）、defaultValue（默认值,当变量未提供时使用）、validation（校验规则:正则表达式或 SpEL 条件）、fallbackExpression（兜底表达式,当 LLM 调用失败时用于生成默认内容）。变量的 SpEL 条件渲染可以实现高级定制:例如 <#if product.price > 5000>享受尊享服务<#else>标准服务</#if>。CorpAssist 的模板变量引擎支持四类 SpEL 表达式:条件渲染（<#if>）、循环迭代（<#list>）、算术运算（{{price * 0.8}}）、字符串拼接（{{brandName + " " + productName}}）。变量引擎的性能指标:单个模板的 SpEL 解析耗时 <2ms,包含 20 个变量的模板填充耗时 <5ms。

模板库的管理界面提供 Web UI,内容运营人员可以通过拖拽方式配置模板的结构和变量,无需编程。模板发布前自动执行 3 项校验:变量完整性检查（所有 <template> 中引用的变量是否在 variables 定义中）、SpEL 语法检查（使用 SpelExpressionParser 解析验证）、消费端兼容性检查（当前模板定义的变量是否覆盖了所有下游系统的字段需求）。上线后的模板通过 JaCoCo 覆盖率工具统计覆盖率——每条模板在生成流水线中被使用的频次分布。CorpAssist 的数据显示:前 20% 的模板覆盖了 80% 的内容生成量,因此模板库管理应优先优化高频模板的性能和灵活性。

import jakarta.persistence.*;
import java.util.List;
import java.util.Map;

@Entity
@Table(name = "content_templates", indexes = {
    @Index(name = "idx_brand_category", columnList = "brandId,category"),
    @Index(name = "idx_template_version", columnList = "templateId,version")
})
public class ContentTemplate {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private String id;

    @Column(nullable = false, length = 64)
    private String templateId;    // 如 "product_desc-tmall"

    @Column(nullable = false, length = 16)
    private String version;       // SemVer: "2.3.1"

    @Column(nullable = false, length = 32)
    private String brandId;       // 品牌 ID

    @Column(nullable = false, length = 32)
    private String category;      // 品类: "phone"/"laptop"/"headphone"

    @ElementCollection
    @CollectionTable(name = "template_channels",
        joinColumns = @JoinColumn(name = "template_id"))
    @Column(name = "channel")
    private List channels; // ["tmall", "jd", "wechat"]

    @Column(columnDefinition = "TEXT", nullable = false)
    private String systemPrompt;   // System Prompt (YAML 格式)

    @Column(columnDefinition = "TEXT", nullable = false)
    private String userPrompt;     // User Prompt (含 SpEL 模板)

    @Column(columnDefinition = "TEXT")
    private String llmParams;      // JSON: {"temperature":0.7,"topP":0.9}

    @Column(columnDefinition = "JSONB")
    private String variables;      // JSON: 变量 Schema 定义

    @Column(columnDefinition = "JSONB")
    private String toneProfile;    // JSON: 五维语调参数

    @Column(nullable = false)
    private boolean active;        // 是否激活

    @Column(nullable = false)
    private Instant createdAt;

    @Column(nullable = false)
    private Instant updatedAt;

    @PrePersist
    void onCreate() {
        createdAt = updatedAt = Instant.now();
    }

    @PreUpdate
    void onUpdate() {
        updatedAt = Instant.now();
    }
}

@Entity
@Table(name = "template_variables")
public class TemplateVariable {

    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private String id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "template_id", nullable = false)
    private ContentTemplate template;

    @Column(nullable = false, length = 64)
    private String name;            // 变量名: "productName"

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private VariableType type;      // STRING/NUMBER/BOOLEAN/IMAGE/LIST

    @Column(nullable = false)
    private boolean required;       // 是否必填

    @Column
    private Integer maxLength;      // 最大字符数

    @Column(columnDefinition = "TEXT")
    private String defaultValue;    // 默认值

    @Column(columnDefinition = "TEXT")
    private String validationRule;  // 校验正则: "^[A-Z].*"

    @Column(columnDefinition = "TEXT")
    private String fallbackExpression; // 兜底 SpEL: "T(java.lang.Math).random()"

    @Column(name = "\"order\"")
    private int order;              // 排序序号

    public enum VariableType {
        STRING, NUMBER, BOOLEAN, IMAGE, LIST
    }

面试常见问答

如何系统性地让 AI 生成的内容符合品牌风格？具体技术方案是什么？

系统性解决方案包含四个技术层次。第一是品牌语调的量化定义——使用五维参数模型（正式度/情感度/专业度/亲和力/创意度,每维 0-10）将抽象风格转化为数值向量,这是所有后续控制的基础。第二是 Prompt 层面的精确控制——System Prompt 中注入品牌角色的详细定义（如"你是一位资深电商文案专家,擅长天猫详情页撰写"）,User Prompt 中通过 SpEL 变量传递语调参数（如 formality=7、emotion=6）,同时可以在 temperature/topP 等 LLM 参数中建立与语调维度的映射公式（如 temperature = 0.1 + formality/10 × 0.5）。第三是 Few-shot 示例的引导——从示例库中根据内容类型和语调向量动态选择最匹配的 3 条示例注入 Prompt,让 LLM 直接模仿示例的格式和风格。第四是事后的语调校验——LLM-as-Judge 在生成后评估五个维度的实际得分,偏差超过 1.5 分的维度触发驳回-重写流程。这四层体系结合,可以将品牌一致性从 70% 左右提升到 90% 以上。

Prompt 模板如何进行版本管理和在线实验？

Prompt 模板的版本管理遵循与代码管理相同的实践。每个模板以 YAML 格式存储在 Git 仓库的 templates/ 目录下,按类型划分子目录。模板文件包含五个区块:元数据（templateId/version/author）、System Prompt（角色+语调指令）、User Prompt（SpEL 模板）、LLM 参数（temperature/topP/maxTokens）和示例引用列表。版本号采用 SemVer 语义化规范——主版本表示不兼容的模板结构变更,次版本表示功能增强（如新增变量）,修订号表示 Bug 修复或 Prompt 措辞微调。每次模板变更自动创建 Git commit,支持版本回溯。A/B 测试基于 Spring Cloud Config 实现:在配置中心新增 template.XX.abTest.enabled=true 和分流比例,Spring Cloud Gateway 根据 userId 的哈希值将流量路由到不同的模板版本。实验的评估指标包括内容质量评分（LLM-as-Judge 自动打分）、用户点击率和转化率,通过 Grafana 实时监控两个版本的差异,使用 t 检验判断显著性（p < 0.05 为通过标准）。

Few-shot 示例库的设计原则和动态选择策略是什么？

Few-shot 示例库的设计遵循"质量优先于数量、标签驱动选择、持续迭代沉淀"三大原则。示例库的每条示例包含内容文本、元数据标签（内容类型、品牌语调五维向量、目标受众、字数）和 Embedding 向量（通过 text2vec-large-chinese 生成 768 维向量）三部分。动态选择策略分两步:第一步是硬性类型过滤——只选择与输入内容类型一致的示例,这一步确保示例格式匹配。第二步是加权评分排序——四个维度的权重分别为类型匹配度 40%（已过滤）、品牌语调相似度 30%（余弦相似度计算,确保风格匹配）、受众匹配度 20%（B2B/B2C/Internal）、字数接近度 10%（避免示例与目标内容长度差异过大导致结构偏差）。选择 Top-3 得分最高的示例注入 Prompt。当最优示例得分低于 0.5 时触发告警,通知运营人员补充示例。建议每个内容类型至少积累 20-30 条示例,示例池总量达到 80 条后质量进入稳定期。示例库需要闭环迭代——人工修改量超过 30% 的内容自动进入候选池,每周审核入库。

场景判断 1: 新品牌接入

一个奢侈手表品牌需要接入 CorpAssist,品牌团队要求内容保持"尊贵、典雅、专业"的风格。该品牌有 3 个产品系列（经典/运动/限量）和 2 个销售渠道（官网+京东）。内容类型包括产品描述、品牌故事和客户感谢信。请判断接入方案的技术要点。
正确方案: 首先定义品牌的五维语调参数（如正式度 9/情感度 3/专业度 8/亲和力 2/创意度 6）,然后按 3 个系列 × 2 个渠道 = 6 个品类模板,每个品类模板关联对应的语调子参数,最后构建 20-30 条该品牌的专属 Few-shot 示例并入库。
理由: 奢侈品牌对品牌一致性要求极高,语调偏差容忍度 <0.5 分（普通品牌 1.5 分）。需要每个系列独立配置语调参数（经典系列更传统,运动系列更活泼）,渠道差异也需要适配（官网更正式,京东详情页更注重参数对比）。

场景判断 2: 模板迭代冲突

内容运营团队同时提交了两个模板修改请求:编辑 A 修改了 product_desc-tmall 模板的 System Prompt（从"资深电商文案专家"改为"资深文案"）,编辑 B 在同一个模板中添加了新的 SpEL 变量 `{{warrantyInfo}}`。两个修改使用同一个 Git 分支,编译检查通过但上线后出现生成质量问题。请判断根本原因和解决方案。
正确判断: 两个修改产生了冲突——编辑 A 的 Prompt 修改依赖了编辑 B 尚未提交的新变量,导致运行时变量缺失。
理由: 模板的 YAML 文件中,System Prompt 中的 `{{warrantyInfo}}` 引用了需要 variables 区块定义的新变量。如果 Prompt 先上线而变量定义未同步上线,LLM 生成内容时变量解析失败。解决方案是:将所有模板变更纳入同一个 Git commit 并在上线前执行变量完整性检查（所有 Prompt 中的变量名是否在 variables 列表中定义）。建议使用预检脚本:解析 YAML 提取所有 `{{var}}` 模式,与 variables map 的 key 集合比对,不匹配的变量直接阻止上线。
