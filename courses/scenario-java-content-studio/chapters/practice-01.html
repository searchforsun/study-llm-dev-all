为什么学这章

企业内容生成不是单篇写作——而是每周 200+ 篇的批量生产。Spring Batch 是 Java 生态最成熟的批处理框架,与 Spring AI 结合可以实现"读取模板变量→调用 LLM 生成→写入结果库"的全自动批处理流水线。本章涉及完整的代码实现,从 Job 配置到 MQ 调度到 Redis 进度追踪,是内容工坊的工程核心。

学完后能做什么

配置 Spring Batch Job 实现 Reader→Processor→Writer 的内容生成流水线（Chunk 大小 10,支持断点续传）；使用 RocketMQ/Kafka 驱动异步批量任务调度,100 篇内容拆分为 10 个子任务并行处理；基于 Redis 实现实时进度追踪（已生成/总任务数/失败数/进度百分比,支持 WebSocket 推送）；配置失败重试（LLM 超时重试 3 次、参数错误跳过、数据库异常重试）和 Spring Batch Admin 监控面板；在生产环境中对 Spring Batch Job 做性能调优（内存/并发/事务）。

本章要点速记

1) Spring Batch Job = Step(Chunk-based Reader→Processor→Writer),Processor 中注入 ChatClient 调用 LLM 生成,Chunk 大小=10 为推荐配置（兼顾吞吐量和事务粒度）；2) RocketMQ 驱动异步调度:Producer 将 100 条生成任务拆分为 10 个 message（每个含 10 条）,Consumer 消费后启动子 Job 并行生成,通过 messageId 做幂等性去重；3) Redis 进度追踪:hash 存储 batchId→{total,completed,failed,percentage} 四个字段,每完成一个 Chunk 更新一次,前端通过 WebSocket 订阅进度变更事件（每秒推送）；4) 失败重试三层策略:Step 级别 retryLimit=3（仅对 TimeoutException 重试）、Skip 级别 skipLimit=5（跳过 IllegalArgumentException）、Job 级别 JobExecutionDecider 判断是否允许继续；5) Spring Batch Admin:通过 /actuator/batch 端点监控 30+ 指标（Job 执行时间/Step 成功率/Chunk 处理速度）,集成 Micrometer 上报到 Prometheus。

Spring Batch Job 设计：Reader→Processor→Writer 的内容生成流水线

Spring Batch 的内容生成 Job 结构遵循标准的"三步走"模式。第一步是 Job 配置定义——使用 @Configuration 标记配置类,通过 JobBuilderFactory 或直接 new JobBuilder 创建 Job 实例。Job 包含多个 Step,每个 Step 完成一个独立的处理阶段。对于内容生成场景,典型的 Job 结构是:ValidateStep（校验输入数据的合法性和模板完整性）→ GenerateStep（调用 LLM 生成内容）→ PostProcessStep（格式清理和质量初筛）。ValidateStep 和 PostProcessStep 是 Tasklet 模式（单一操作）,GenerateStep 是 Chunk 模式（批量操作）。Chunk 模式的核心思想是"读一批、处理一批、写一批"——Reader 从数据库或文件读取待生成条目（ContentInput）,Processor 完成业务逻辑（调用 ChatClient 生成内容）,Writer 批量写入结果库。Chunk 大小决定了每次事务的处理条数,推荐值为 10（经验值,需要根据 LLM 调用延迟和数据库写入吞吐量调优）。

内容生成的 Processor 是流水线的核心——它接收 ContentInput（包含 templateId、variables、brandProfile）,依次执行:SpEL 模板填充（将 variables 注入模板的 {{}} 占位符）→ 模板和示例组装（将 System Prompt + User Prompt + Few-shot 示例拼接为完整 Prompt）→ ChatClient 调用（配置 temperature/topP/maxTokens,设置超时 30 秒）→ 结果解析（从 LLM 响应中提取正文文本、去除多余的格式标记）→ 质量初筛检查（字数 >50 字、不含被禁止的关键词、与模板要求的格式一致）。Processor 返回 ContentOutput 对象,Writer 批量接收后写入 PostgreSQL 的 content_output 表和对应的 progress 更新。如果 Processor 中抛出异常,Spring Batch 根据配置的 retry/skip 策略决定是重试还是跳过。

Job 的启动时机有两个场景:实时触发（用户通过 API 提交批量生成请求 → Controller 调用 JobLauncher.run() 同步或异步启动 Job）和定时触发（通过 @Scheduled 或 Quartz 在每天凌晨低峰期启动定时批量生成任务,如每日凌晨 3 点生成当天的 EDM 邮件）。Job 的并行度通过 Split 和 Flow 控制——如果 100 篇内容覆盖多个品牌,可以将品牌作为分区维度,每个品牌启动一个独立的 Step 并行执行,通过 Spring Batch 的 Partition 功能实现。Partition 将 Step 拆分为多个 Worker Step,每个 Worker 处理一个品牌的数据分区,Manager Step 负责协调和汇总结果。Partition 的网格大小=品牌数,通常在 3-10 个之间。CorpAssist 的批量测试数据:单 Step Chunk=10,无分区时 100 篇耗时约 3 分钟;3 分区并行时 100 篇耗时约 70 秒（减少 61%）;10 分区时 100 篇耗时约 40 秒（再减少 43%）。但分区过多会导致数据库连接池争用和 Chunk 提交冲突,建议分区数 ≤10。

java
import org.springframework.batch.core.*;
import org.springframework.batch.core.configuration.annotation.EnableBatchProcessing;
import org.springframework.batch.core.job.builder.JobBuilder;
import org.springframework.batch.core.repository.JobRepository;
import org.springframework.batch.core.step.builder.StepBuilder;
import org.springframework.batch.item.*;
import org.springframework.batch.item.database.JdbcCursorItemReader;
import org.springframework.batch.item.database.builder.JdbcBatchItemWriterBuilder;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.transaction.PlatformTransactionManager;
import javax.sql.DataSource;
import java.util.List;

@Configuration
@EnableBatchProcessing
public class ContentBatchJobConfig {

    private static final int CHUNK_SIZE = 10;
    private static final int RETRY_LIMIT = 3;
    private static final int SKIP_LIMIT = 5;

    @Bean
    public Job contentGenerationJob(JobRepository jobRepository,
                                    Step validateStep,
                                    Step generateStep,
                                    Step postProcessStep) {
        return new JobBuilder("contentGenerationJob", jobRepository)
            .start(validateStep)                // Step 1: 校验
            .next(generateStep)                 // Step 2: 生成
            .next(postProcessStep)              // Step 3: 后处理
            .listener(new JobCompletionNotificationListener())
            .build();
    }

    @Bean
    public Step validateStep(JobRepository jobRepository,
                             PlatformTransactionManager transactionManager) {
        return new StepBuilder("validateStep", jobRepository)
            .tasklet(new ContentValidationTasklet(), transactionManager)
            .build();
    }

    @Bean
    public Step generateStep(JobRepository jobRepository,
                             PlatformTransactionManager transactionManager,
                             ItemReader reader,
                             ItemProcessor processor,
                             ItemWriter writer) {
        return new StepBuilder("generateStep", jobRepository)
            .chunk(CHUNK_SIZE, transactionManager)
            .reader(reader)
            .processor(processor)
            .writer(writer)
            .faultTolerant()
            .retryLimit(RETRY_LIMIT)
            .retry(org.springframework.dao.TransientDataAccessException.class)
            .retry(java.net.SocketTimeoutException.class)
            .noRetry(IllegalArgumentException.class)
            .skipLimit(SKIP_LIMIT)
            .skip(IllegalArgumentException.class)
            .listener(new ChunkProgressListener())
            .build();
    }

    @Bean
    public Step postProcessStep(JobRepository jobRepository,
                                PlatformTransactionManager transactionManager) {
        return new StepBuilder("postProcessStep", jobRepository)
            .tasklet(new QualityCheckTasklet(), transactionManager)
            .build();
    }

    @Bean
    public ItemReader contentReader(DataSource dataSource) {
        return new JdbcCursorItemReaderBuilder()
            .name("contentReader")
            .dataSource(dataSource)
            .sql("SELECT ci.id, ci.template_id, ci.variables_json, " +
                 "ci.brand_tone, ci.channel " +
                 "FROM content_input ci WHERE ci.status = 'PENDING' " +
                 "ORDER BY ci.priority DESC, ci.created_at ASC " +
                 "LIMIT 100")
            .rowMapper((rs, rowNum) -> new ContentInput(
                rs.getString("id"),
                rs.getString("template_id"),
                rs.getString("variables_json"),
                rs.getString("brand_tone"),
                rs.getString("channel")))
            .build();
    }

    @Bean
    public ItemProcessor contentProcessor(
            ChatClient chatClient, TemplateEngine templateEngine) {
        return input -> {
            // 1. SpEL 模板填充
            String filledPrompt = templateEngine.fill(
                input.templateId(), input.variablesJson());

            // 2. 调用 LLM 生成
            String result = chatClient.prompt()
                .user(u -> u.text(filledPrompt))
                .call()
                .content();

            // 3. 质量初筛
            if (result == null || result.length() < 50) {
                throw new IllegalArgumentException(
                    "生成内容过短: " + (result != null ? result.length() : 0) + " 字符");
            }

            return new ContentOutput(input.id(), result, "GENERATED");
        };
    }

    @Bean
    public ItemWriter contentWriter(DataSource dataSource) {
        return new JdbcBatchItemWriterBuilder()
            .dataSource(dataSource)
            .sql("INSERT INTO content_output (id, content, status, generated_at) " +
                 "VALUES (:id, :content, :status, NOW()) " +
                 "ON CONFLICT (id) DO UPDATE SET status = 'REGENERATED'")
            .beanMapped()
            .build();
    }

    // Listener: Job 完成通知
    public static class JobCompletionNotificationListener
            implements JobExecutionListener {
        @Override
        public void afterJob(JobExecution jobExecution) {
            long duration = jobExecution.getEndTime().getTime()
                - jobExecution.getStartTime().getTime();
            System.out.printf("[BATCH] Job %s 完成, 状态: %s, 耗时: %dms%n",
                jobExecution.getJobInstance().getJobName(),
                jobExecution.getStatus(), duration);
        }
    }

    // Listener: Chunk 进度
    public static class ChunkProgressListener
            implements ChunkListener {
        @Override
        public void afterChunk(ChunkContext context) {
            int count = context.getCompleteCount();
            System.out.printf("[CHUNK] 已完成 %d 条处理%n", count);
        }
    }

    record ContentInput(String id, String templateId,
                        String variablesJson, String brandTone,
                        String channel) {}
    record ContentOutput(String id, String content, String status) {}
}

MQ（RabbitMQ/Kafka）驱动的异步批量任务调度

当批量生成请求的规模达到 500+/批时,同步的 Spring Batch 执行会占用长时间的事务资源,导致数据库连接池耗尽。MQ 驱动的异步调度方案将任务拆分为可管理的小批次,通过消息队列做缓冲和削峰。架构设计:Producer 端接收批量请求后,将 500 条内容生成任务按每 10 条拆分,创建 50 条 MQ 消息并发送到指定队列。每条消息包含 batchId、taskIndex、inputs（List）三个字段。Consumer 端监听队列,消费消息后启动一个"迷你 Job"——直接调用 Service 层的 generateBatch(inputs) 方法,不启动完整的 Spring Batch Job 以避免 JobRepository 的写压力。每条消息的消费结果（成功/失败/部分成功）通过 RocketMQ 的回调机制通知 Producer,Producer 汇总所有 50 个子任务的执行结果,更新整体的批处理状态。

MQ 的选择:RocketMQ 相比 Kafka 更适合内容生成场景——原因是 RocketMQ 支持延迟消息（失败重试时延迟 10 秒/30 秒/5 分钟三种级别）、事务消息（确保消息发送和数据库更新的原子性）、消息重试（默认 16 次,间隔递增）和消息轨迹（追踪每条消息从生产到消费的完整生命周期）。CorpAssist 使用 RocketMQ 5.1 版本,配置:队列数=8（分区数,与 Consumer 并发数匹配）、批量消息大小=10、消息体最大 4MB（每条消息 10 条生成任务约 50KB）、消息保留 72 小时（允许长时间的业务排障）。如果团队已有 Kafka 基础设施,也可以使用 Spring Cloud Stream 的 Kafka Binder——Kafka 的优势是吞吐量更高（百万 msg/s 级别）和生态更成熟,但缺少 RocketMQ 的事务消息和延迟消息特性,需要自行实现消息去重（通过 messageId + Redis Set）。

java
import org.apache.rocketmq.spring.core.RocketMQTemplate;
import org.springframework.messaging.Message;
import org.springframework.messaging.support.MessageBuilder;
import org.springframework.stereotype.Component;
import java.util.*;

@Component
public class ContentTaskDispatcher {

    private final RocketMQTemplate rocketMQTemplate;
    private static final String TOPIC = "content-generate-topic";
    private static final int BATCH_SIZE = 10;  // 每批 10 条

    public ContentTaskDispatcher(RocketMQTemplate rocketMQTemplate) {
        this.rocketMQTemplate = rocketMQTemplate;
    }

    public String dispatch(List inputs) {
        String batchId = UUID.randomUUID().toString();
        int totalTasks = inputs.size();

        // 1. 初始化 Redis 进度
        initProgress(batchId, totalTasks);

        // 2. 按 BATCH_SIZE 拆分为多个子任务
        List> batches = partition(inputs, BATCH_SIZE);

        // 3. 发送消息到 RocketMQ
        for (int i = 0; i < batches.size(); i++) {
            ContentTask task = new ContentTask(batchId, i, batches.get(i));

            Message message = MessageBuilder
                .withPayload(task)
                .setHeader("KEYS", batchId + ":" + i)      // 消息唯一键,用于幂等
                .setHeader("DELAY", 0)                       // 延迟等级:0=立即
                .build();

            rocketMQTemplate.syncSend(TOPIC + ":tag-generate",
                message, 30000);  // 30秒超时
        }

        System.out.printf("[DISPATCH] batchId=%s, 总任务=%d, 子消息=%d%n",
            batchId, totalTasks, batches.size());

        return batchId;  // 返回 batchId 供客户端查询进度
    }

    @RocketMQMessageListener(topic = TOPIC, selectorExpression = "tag-generate",
                             consumerGroup = "content-generate-group")
    public static class GenerateTaskConsumer
            implements RocketMQListener {

        private final ContentGeneratorService generatorService;

        public GenerateTaskConsumer(ContentGeneratorService service) {
            this.generatorService = service;
        }

        @Override
        public void onMessage(ContentTask task) {
            try {
                // 幂等检查: Redis 中是否已处理过该 task
                if (isAlreadyProcessed(task.batchId(), task.taskIndex())) {
                    return;
                }

                // 执行批量生成
                List results = generatorService
                    .generateBatch(task.inputs());

                // 更新进度
                updateProgress(task.batchId(),
                    results.size(), 0);

                // 标记已处理
                markProcessed(task.batchId(), task.taskIndex());

            } catch (Exception e) {
                System.err.printf("[CONSUME] 失败 batch=%s task=%d: %s%n",
                    task.batchId(), task.taskIndex(), e.getMessage());

                // 更新失败计数
                updateProgress(task.batchId(),
                    0, task.inputs().size());

                // 抛出异常触发 RocketMQ 重试 (最多 16 次)
                throw new RuntimeException(e);
            }
        }

        private boolean isAlreadyProcessed(String batchId, int idx) {
            // 用 Redis SETNX 检查
            return false; // 简化
        }

        private void markProcessed(String batchId, int idx) {
            // 标记处理完成
        }
    }

    public record ContentTask(String batchId, int taskIndex,
                              List inputs) {}

    private void initProgress(String batchId, int total) {
        // Redis: HSET batch:{batchId} total {total} completed 0 failed 0
    }

    private void updateProgress(String batchId, int success, int fail) {
        // Redis: HINCRBY batch:{batchId} completed {success}
        // Redis: HINCRBY batch:{batchId} failed {fail}
    }

    private  List> partition(List list, int size) {
        List> result = new ArrayList<>();
        for (int i = 0; i < list.size(); i += size) {
            result.add(list.subList(i, Math.min(i + size, list.size())));
        }
        return result;
    }
}

Redis 进度追踪与失败重试机制

进度追踪是批量生成场景的用户体验关键——用户提交 100 篇内容生成请求后,如果不能实时看到进度,会产生"系统卡死"的焦虑。CorpAssist 使用 Redis Hash 结构存储每个批次的进度信息:key = "batch:{batchId}",field = {total, completed, failed, percentage, status}。每次 Chunk 完成或 MQ 消费回调时,通过 HINCRBY 原子操作更新 completed 或 failed 计数,并重新计算 percentage = (completed + failed) / total × 100。前端通过 WebSocket（STOMP over SockJS）订阅 /topic/batch/{batchId} 主题,后端在进度更新时通过 SimpMessagingTemplate.convertAndSend 推送进度事件,推送频率控制在每秒最多 1 次（通过 HINCRBY 后的检查,超过 500ms 的间隔才推送,避免高频推送压垮前端渲染）。进度数据结构:{"batchId":"xxx","total":100,"completed":45,"failed":2,"percentage":47,"status":"RUNNING","estimatedRemainingSeconds":120}。

失败重试机制需要区分三类异常。第一类是临时性异常（Transient）:LLM API 超时（SocketTimeoutException）、网络闪断、数据库连接池暂时耗尽——处理策略是 retryLimit=3,每次重试间隔指数退避（2s/4s/8s）。第二类是业务性异常（Business）:参数校验失败（IllegalArgumentException）、模板变量缺失——处理策略是 skip（跳过并记录,skipLimit=5）,因为重试也一定会失败。第三类是系统性异常（System）:LLM 返回空内容、内容质量初筛不通过——处理策略是 markAsFailed（使用模板兜底内容填充,并在结果中标记"FALLBACK"状态）。重试的去重由消息的幂等键保证——每次消息携带 KEYS=batchId:taskIndex,Consumer 在接收到消息后先检查 Redis 中的 processed:set 是否已经存在该键,如果存在直接 ACK 不处理。幂等键在消费成功后写入 Redis（TTL=7 天）,确保重试场景下的精确一次消费语义。CorpAssist 的生产数据:重试率为 3-5%（即 10000 篇中 300-500 篇需要重试）,重试成功率约 85%,最终失败率 <0.5%。

java
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.messaging.simp.SimpMessagingTemplate;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;
import java.util.Map;
import java.util.concurrent.TimeUnit;

@Component
public class BatchProgressTracker {

    private final RedisTemplate redis;
    private final SimpMessagingTemplate messagingTemplate;

    private static final String BATCH_KEY_PREFIX = "batch:";
    private static final String PROCESSED_KEY_PREFIX = "processed:";
    private static final long PROGRESS_TTL_SECONDS = 86400; // 1天
    private static final long PUSH_INTERVAL_MS = 500;

    private long lastPushTime = 0;

    public BatchProgressTracker(RedisTemplate redis,
                                SimpMessagingTemplate messagingTemplate) {
        this.redis = redis;
        this.messagingTemplate = messagingTemplate;
    }

    public void initBatch(String batchId, int total) {
        Map progress = Map.of(
            "total", String.valueOf(total),
            "completed", "0",
            "failed", "0",
            "percentage", "0",
            "status", "RUNNING"
        );
        redis.opsForHash().putAll(BATCH_KEY_PREFIX + batchId, progress);
        redis.expire(BATCH_KEY_PREFIX + batchId, PROGRESS_TTL_SECONDS,
            TimeUnit.SECONDS);
    }

    public synchronized void recordSuccess(String batchId, int count) {
        redis.opsForHash().increment(BATCH_KEY_PREFIX + batchId,
            "completed", count);
        tryPushProgress(batchId);
    }

    public synchronized void recordFailure(String batchId, int count) {
        redis.opsForHash().increment(BATCH_KEY_PREFIX + batchId,
            "failed", count);
        tryPushProgress(batchId);
    }

    private void tryPushProgress(String batchId) {
        long now = System.currentTimeMillis();
        if (now - lastPushTime < PUSH_INTERVAL_MS) {
            return; // 限频: 500ms内不重复推送
        }
        lastPushTime = now;

        Map data = redis.opsForHash()
            .entries(BATCH_KEY_PREFIX + batchId);

        if (!data.isEmpty()) {
            int total = Integer.parseInt((String) data.get("total"));
            int completed = Integer.parseInt((String) data.get("completed"));
            int failed = Integer.parseInt((String) data.get("failed"));
            int percentage = (int) (((double) (completed + failed) / total) * 100);

            String status = percentage >= 100 ? "COMPLETED" : "RUNNING";
            redis.opsForHash().put(BATCH_KEY_PREFIX + batchId,
                "percentage", String.valueOf(percentage));
            redis.opsForHash().put(BATCH_KEY_PREFIX + batchId,
                "status", status);

            // 估算剩余时间 (基于最近5个Chunk的平均速度)
            long remaining = estimateRemaining(batchId, total, completed, failed);

            // WebSocket 推送
            messagingTemplate.convertAndSend(
                "/topic/batch/" + batchId,
                Map.of("batchId", batchId, "total", total,
                    "completed", completed, "failed", failed,
                    "percentage", percentage, "status", status,
                    "estimatedRemainingSeconds", remaining));
        }
    }

    public BatchProgress getProgress(String batchId) {
        Map data = redis.opsForHash()
            .entries(BATCH_KEY_PREFIX + batchId);
        if (data.isEmpty()) return null;
        return new BatchProgress(
            batchId,
            Integer.parseInt((String) data.get("total")),
            Integer.parseInt((String) data.get("completed")),
            Integer.parseInt((String) data.get("failed")),
            Integer.parseInt((String) data.get("percentage")),
            (String) data.get("status")
        );
    }

    public boolean markProcessed(String batchId, int taskIndex) {
        String key = PROCESSED_KEY_PREFIX + batchId + ":" + taskIndex;
        Boolean success = redis.opsForValue()
            .setIfAbsent(key, "1", 7, TimeUnit.DAYS);
        return Boolean.TRUE.equals(success);
    }

    public boolean isProcessed(String batchId, int taskIndex) {
        return Boolean.TRUE.equals(
            redis.hasKey(PROCESSED_KEY_PREFIX + batchId + ":" + taskIndex));
    }

    private long estimateRemaining(String batchId, int total,
                                   int completed, int failed) {
        long elapsed = getElapsedSeconds(batchId);
        int done = completed + failed;
        if (done == 0 || elapsed == 0) return -1;
        long perItem = elapsed / done;
        return perItem * (total - done);
    }

    private long getElapsedSeconds(String batchId) {
        Long ttl = redis.getExpire(BATCH_KEY_PREFIX + batchId);
        if (ttl == null) return 0;
        return PROGRESS_TTL_SECONDS - ttl;
    }

    public record BatchProgress(String batchId, int total, int completed,
                                int failed, int percentage, String status) {}

    @Scheduled(fixedRate = 30000) // 每30秒清理过期批次
    public void cleanupExpiredBatches() {
        // 扫描 TTL 已过期的 batch key 并清理关联的 processed key
    }
}

本章结论

  • Spring Batch Chunk 模式（chunk=10）实现 Reader→Processor→Writer 内容生成流水线,100 篇内容无分区耗时约 3 分钟,10 分区并行缩短至 40 秒
  • RocketMQ 异步调度:100 条拆分为 10 子消息（每消息 10 条）,RocketMQ 事务消息和延迟重试保障消息可靠性,消费幂等由 Redis SETNX 确保
  • Redis Hash 进度追踪（field:total/completed/failed/percentage/status）,WebSocket 限频推送（≤1次/500ms）,P99 进度查询 <5ms
  • 三层失败重试策略:临时异常重试 3 次（指数退避 2s/4s/8s）→ 业务异常跳过 5 次 → 系统性异常使用模板兜底,最终失败率 <0.5%
  • Spring Batch Admin 通过 Micrometer 暴露 30+ 指标（Job 耗时/Step 成功率/Chunk TPS）,集成 Prometheus + Grafana 监控面板

面试常见问答

Spring Batch 内容生成 Job 如何设计？Chunk 大小如何确定？

Spring Batch Job 采用三步结构——ValidateStep（Tasklet 模式校验输入合法性）、GenerateStep（Chunk 模式批量生成）、PostProcessStep（Tasklet 模式汇总和质检）。GenerateStep 的 Chunk 大小推荐值为 10,这是综合考虑 LLM 调用延迟（平均 4-8 秒/次）、数据库事务开销和内存占用后的经验值。Chunk 过小（如 2-3）导致事务提交次数过多,数据库连接浪费；Chunk 过大（如 50+）导致 LLM 调用失败时回滚范围过大,且内存中同时保留 50 条生成结果约 5-10MB。调优方法:从 Chunk=5 开始逐步增加到 30,观察 throughput（篇/秒）和 P99 延迟的拐点。在 CorpAssist 实测中 Chunk=10 时 throughput 最高（约 0.55 篇/秒/线程）,Chunk 增加到 20 时 throughput 下降 10% 同时内存增加 2 倍。Processor 中调用 ChatClient 时需要注意设置 readTimeout=30s,connectTimeout=10s,避免 LLM API 慢响应阻塞整个 Step。

MQ 驱动的内容生成异步调度和 Spring Batch 同步执行如何配合？

两者配合的核心策略是"同步 Job 管控 + 异步 MQ 执行"。当用户提交 500 篇内容的批量生成请求时,API 层启动一个轻量级的 Spring Batch Job（只有一个 Step,Tasklet 模式）,这个 Step 的工作就是调用 MQ Producer 将 500 条任务拆分为 50 条消息发到 RocketMQ 后立即返回。真正的 LLM 生成工作由 MQ Consumer 消费消息后异步执行——Consumer 直接调用 Service 层的 generateBatch() 方法处理每条消息中的 10 条内容。这样做的好处是:MQS 的异步处理避免长事务占用数据库连接池,同时利用 RocketMQ 的延迟重试和死信队列做失败处理。Consumer 的结果通过 RocketMQ 的回调机制回写到 Redis 进度,前端通过 WebSocket 获取实时进度。配合的关键是消息的幂等设计——每条消息携带 batchId+taskIndex 作为幂等键 Consumer 在处理前用 Redis SETNX 检查避免重复消费。当 RocketMQ 重试 16 次仍然失败时消息进入死信队列由运维人员人工处理或通过另一个自动补偿 Job 定期重试死信队列中的任务。

进度追踪系统的设计和限频推送策略？

进度追踪系统基于 Redis Hash 存储批次进度,key="batch:{batchId}",field 包括 total/completed/failed/percentage/status。每次 Chunk 完成或 MQ 消费回调时调用 recordSuccess()/recordFailure() 方法使用 HINCRBY 原子操作更新计数。查询接口 getProgress(batchId) 通过 HGETALL 读取全部字段返回给客户端,P99 查询耗时 <5ms。前端通过 STOMP over WebSocket 订阅 /topic/batch/{batchId} 主题接收进度推送。推送频率需要通过限频策略控制——设置最小推送间隔 500ms,在 recordSuccess/recordFailure 方法中用 synchronized + 时间戳控制：如果距离上次推送不足 500ms 则跳过推送仅更新 Redis,超过 500ms 才推送。推算剩余时间的算法:维护一个最近的 Chunk 完成时间戳队列（存最近 5 个 Chunk 的完成时刻）,计算平均处理速度（篇/秒）,剩余时间 = 剩余篇数 / 平均速度。当批次完成（percentage=100）时推送一条 status=COMPLETED 的最终消息,前端收到后关闭 WebSocket 连接并触发后续操作（如展示生成结果）。

场景判断 1: Chunk 大小调优

100 篇内容的批量生成 Job,设置 Chunk=50,每篇 LLM 调用平均 6 秒,数据库写入平均 200ms。观察到此配置下 Job 耗时 35 分钟,而且出现了两次数据库连接超时。请判断问题原因和解决方案。
正确判断: Chunk=50 过大,导致每个 Chunk 的事务耗时约 310 秒（50×6s+50×0.2s）,远超过默认的事务超时（通常 30-60 秒）,触发事务回滚和重试。
解决方案: 将 Chunk 降低到 10（事务耗时约 62 秒,需要将事务超时调到 120 秒）,或者 Chunk=5（事务耗时 31 秒,无需调整超时）。同时启用 Partition 分区（按品类分区=5）并行处理。

场景判断 2: MQ 重复消费

某次 RocketMQ Broker 异常重启后,Consumer 收到大量重发消息。运维发现部分内容被重复生成了 2-3 次,导致了资源浪费（API 费用翻倍）。请分析原因和改进方案。
正确分析: 原因是没有实现消费幂等——Consumer 在处理消息前没有检查该消息是否已被处理过。改进方案:在 ContentTaskConsumer.onMessage() 的最前面加入幂等性检查:用 Redis SETNX "processed:{batchId}:{taskIndex}" 1 EX 604800（7天过期）。如果 SETNX 返回 false 说明已处理过直接 ACK 返回。同时确保数据库写入是幂等的（使用 ON CONFLICT DO UPDATE 或 INSERT IGNORE）。这样即使 RocketMQ 重发同一条消息,最多导致一次重复的 SETNX 写入但不会导致内容重复生成。
