REST Controller + SSE 流式输出的端到端实现

内容生成 API 的核心用户体验是"不要让我等"——如果用户提交生成请求后需要等待 5-10 秒才能看到结果,用户会焦虑甚至关闭页面。CorpAssist 采用 SSE（Server-Sent Events）流式输出方案,让 LLM 的 token 逐字推送到前端,用户在生成开始后的 1-2 秒就能看到第一行内容,后续内容逐步展示,显著降低感知延迟。实现方案:Controller 接口 POST /api/v1/generate/stream,接收生成请求（templateId + variables + toneParams）,返回 SseEmitter。SseEmitter 是 Spring 提供的 SSE 抽象,设置超时时间 60 秒（超过自动关闭连接）。Controller 中启用 @Async 注解,将 LLM 调用放入独立线程池执行（TaskExecutor 配置 corePoolSize=10,maxPoolSize=50,queueCapacity=200）,防止 SSE 长连接阻塞 Tomcat 的请求处理线程。LLM 调用使用 ChatClient 的 stream() 方法（而不是 call()）,返回 Flux<String>,每个 token 通过 SseEmitter.send() 推送到前端,推送完成（complete）或异常（completeWithError）时结束 SSE 流。

前端的接收代码使用原生 EventSource API（或 fetch + ReadableStream）,也可以使用 Spring 的 WebClient 做后端到后端的 SSE 对接。生产环境的优化点包括:①缓冲区控制——LLM 的单个 token 推送太快（每秒 20-50 个 token）会导致前端渲染卡顿,建议在 SseEmitter.send() 之前增加一个小型缓冲区（50ms 或 5 个 token 积压一次发送）,降低推送频率但保证流畅度；②心跳机制——SSE 连接可能因为网络中间件（Nginx/网关）的空闲超时而断开,需要每隔 15 秒发送一条空注释（": heartbeat\n\n"）保持连接存活；③断线重连——前端检测到 SSE 连接断开后,自动在 1 秒后重新连接,并在请求头中携带 lastTokenIndex（已接收到的最后一个 token 的索引）,服务端支持从断点处继续推送（需要 ChatClient 支持设置 startTokenIndex）。④前端的渲染优化——对于长内容（2000+ token）,前端不应该等全部 token 到达后才渲染,而应该使用增量渲染,每收到 5-10 个 token 就更新一次 DOM。

java
import org.springframework.web.bind.annotation.*;
import org.springframework.web.servlet.mvc.method.annotation.SseEmitter;
import org.springframework.scheduling.annotation.Async;
import org.springframework.ai.chat.client.ChatClient;
import reactor.core.publisher.Flux;
import java.util.Map;
import java.util.concurrent.TimeUnit;

@RestController
@RequestMapping("/api/v1/generate")
public class ContentGenerateController {

    private final ChatClient chatClient;
    private final TemplateEngine templateEngine;
    private final ContentBatchService batchService;

    public ContentGenerateController(ChatClient.Builder builder,
                                     TemplateEngine templateEngine,
                                     ContentBatchService batchService) {
        this.chatClient = builder.build();
        this.templateEngine = templateEngine;
        this.batchService = batchService;
    }

    // 单篇同步生成 (非流式)
    @PostMapping("/sync")
    public GenerateResponse generateSync(@RequestBody GenerateRequest request) {
        String prompt = templateEngine.fill(
            request.templateId(), request.variables());
        String content = chatClient.prompt()
            .user(u -> u.text(prompt))
            .call()
            .content();
        return new GenerateResponse(request.templateId(), content, "SUCCESS");
    }

    // 单篇流式生成 (SSE)
    @PostMapping("/stream")
    @Async("contentTaskExecutor")
    public SseEmitter generateStream(@RequestBody GenerateRequest request) {
        SseEmitter emitter = new SseEmitter(60_000L);  // 60s 超时

        // 心跳保活 (每15秒)
        new Thread(() -> {
            try {
                while (true) {
                    TimeUnit.SECONDS.sleep(15);
                    emitter.send(SseEmitter.event()
                        .comment("heartbeat")
                        .name("heartbeat")
                        .data(""));
                }
            } catch (Exception ignored) {}
        }).start();

        try {
            String prompt = templateEngine.fill(
                request.templateId(), request.variables());

            Flux tokenStream = chatClient.prompt()
                .user(u -> u.text(prompt))
                .stream()
                .content();

            StringBuilder fullContent = new StringBuilder();
            int tokenCount = 0;

            tokenStream.subscribe(
                token -> {
                    try {
                        fullContent.append(token);
                        tokenCount++;

                        // 每 5 个 tokens 推送一次,降低推送频率
                        if (tokenCount % 5 == 0) {
                            emitter.send(SseEmitter.event()
                                .name("token")
                                .data(Map.of(
                                    "partial", fullContent.toString(),
                                    "tokenCount", tokenCount,
                                    "finished", false)));
                        }
                    } catch (Exception e) {
                        emitter.completeWithError(e);
                    }
                },
                error -> {
                    try {
                        emitter.send(SseEmitter.event()
                            .name("error")
                            .data(Map.of("error", error.getMessage())));
                        emitter.complete();
                    } catch (Exception ignored) {}
                },
                () -> {
                    try {
                        // 流完成,推送完整内容
                        emitter.send(SseEmitter.event()
                            .name("complete")
                            .data(Map.of(
                                "content", fullContent.toString(),
                                "tokenCount", tokenCount,
                                "finished", true)));
                        emitter.complete();
                    } catch (Exception ignored) {}
                }
            );
        } catch (Exception e) {
            try {
                emitter.send(SseEmitter.event()
                    .name("error")
                    .data(Map.of("error", e.getMessage())));
                emitter.complete();
            } catch (Exception ignored) {}
        }

        return emitter;
    }

    // 批量生成 (异步,返回 batchId)
    @PostMapping("/batch")
    public BatchResponse generateBatch(@RequestBody BatchRequest request) {
        String batchId = batchService.submitBatch(request.inputs());
        return new BatchResponse(batchId, "SUBMITTED",
            "批次已提交,通过 WebSocket /topic/batch/" + batchId + " 查看进度");
    }

    // 查询批次进度
    @GetMapping("/batch/{batchId}/progress")
    public BatchProgress getBatchProgress(@PathVariable String batchId) {
        return batchService.getProgress(batchId);
    }

    // DTO 记录
    record GenerateRequest(String templateId, Map variables,
        double[] toneParams) {}
    record GenerateResponse(String templateId, String content, String status) {}
    record BatchRequest(String batchName, java.util.List inputs) {}
    record BatchResponse(String batchId, String status, String message) {}
    record BatchProgress(String batchId, int total, int completed,
        int failed, int percentage, String status) {}
    record ContentInput(String id, String templateId,
        Map variables, String channel) {}
}

批量生成的异步回调与 WebSocket 进度推送

批量生成的场景（如 100 篇产品描述）不适合使用 SSE——单篇生成的流式体验是好的,但 100 篇内容串行流式输出需要 8-10 分钟,前端保持长连接过于吃资源。CorpAssist 的批量生成 API 采用"异步提交 + WebSocket 进度推送 + 回调通知"的架构。用户通过 POST /api/v1/generate/batch 提交批量化请求,API 返回 batchId 和 HTTP 202 Accepted 状态码。后端异步执行生成（通过 RocketMQ + Spring Batch）,进度通过 WebSocket 实时推送给前端。WebSocket 基于 STOMP over SockJS 协议——前端使用 @stomp/stompjs 库连接 ws://host:port/ws-endpoint,订阅 /topic/batch/{batchId} 主题。后端在进度更新时通过 SimpMessagingTemplate.convertAndSend() 推送进度消息。批量完成时,推送一条 status=COMPLETED 的最终消息,前端收到后关闭订阅并触发结果展示。

回调通知机制:当批量生成完全部内容后,系统通过 Webhook 调用用户注册的回调 URL（在提交批量请求时通过 callbackUrl 参数指定）。Webhook 的 Payload 包含 batchId、totalCount、successCount、failCount、summaryReportUrl（包含所有生成内容的概览报告下载链接）、以及失败条目的错误信息列表。Webhook 的重试策略:如果回调返回非 2xx 状态码,系统在 5 分钟/15 分钟/30 分钟后各重试一次,最多 3 次。如果 3 次都失败,记录到 webhook_failure_log 表由运维人员手动处理。Webhook 的签名安全:Payload 通过 HMAC-SHA256 签名（密钥在注册回调时协商）,接收方验证签名确保消息未被篡改。

java
import org.springframework.messaging.simp.SimpMessagingTemplate;
import org.springframework.web.bind.annotation.*;
import org.springframework.web.client.RestTemplate;
import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;
import java.util.*;

@RestController
@RequestMapping("/api/v1/generate/batch")
public class BatchWithCallbackController {

    private final SimpMessagingTemplate messagingTemplate;
    private final BatchProgressTracker progressTracker;
    private final RestTemplate restTemplate;
    private static final String HMAC_KEY = "webhook-hmac-key-2026";

    public BatchWithCallbackController(
            SimpMessagingTemplate messagingTemplate,
            BatchProgressTracker progressTracker) {
        this.messagingTemplate = messagingTemplate;
        this.progressTracker = progressTracker;
        this.restTemplate = new RestTemplate();
    }

    @PostMapping
    public BatchResponse submitBatch(@RequestBody BatchWithCallbackRequest req) {
        String batchId = UUID.randomUUID().toString();

        // 保存回调信息
        progressTracker.saveCallback(batchId, req.callbackUrl());

        // 异步启动生成
        progressTracker.initBatch(batchId, req.inputs().size());
        asyncGenerate(batchId, req.inputs());

        return new BatchResponse(batchId, "ACCEPTED",
            "批量任务已提交,通过 WebSocket 追踪进度");
    }

    private void asyncGenerate(String batchId, List inputs) {
        // 异步: RocketMQ 或 CompletableFuture
        CompletableFuture.runAsync(() -> {
            int success = 0, fail = 0;
            List> errors = new ArrayList<>();

            for (ContentInput input : inputs) {
                try {
                    String content = generateOne(input);
                    progressTracker.recordSuccess(batchId, 1);
                    success++;

                    // WebSocket 推送进度
                    pushProgress(batchId);

                } catch (Exception e) {
                    progressTracker.recordFailure(batchId, 1);
                    fail++;
                    errors.add(Map.of(
                        "id", input.id(), "error", e.getMessage()));
                }
            }

            // 完成时推送最终状态并触发回调
            pushFinalStatus(batchId, success, fail, errors);
            triggerCallback(batchId, success, fail, errors);
        });
    }

    private void pushProgress(String batchId) {
        var progress = progressTracker.getProgress(batchId);
        if (progress != null) {
            messagingTemplate.convertAndSend(
                "/topic/batch/" + batchId, progress);
        }
    }

    private void pushFinalStatus(String batchId, int success,
                                  int fail, List> errors) {
        messagingTemplate.convertAndSend(
            "/topic/batch/" + batchId,
            Map.of("batchId", batchId, "status", "COMPLETED",
                "total", success + fail, "success", success,
                "failure", fail, "errors", errors));
    }

    private void triggerCallback(String batchId, int success,
                                  int fail, List> errors) {
        String callbackUrl = progressTracker.getCallback(batchId);
        if (callbackUrl == null || callbackUrl.isBlank()) return;

        // 构建回调 Payload
        Map payload = new HashMap<>();
        payload.put("batchId", batchId);
        payload.put("status", "COMPLETED");
        payload.put("total", success + fail);
        payload.put("successCount", success);
        payload.put("failCount", fail);
        payload.put("timestamp", System.currentTimeMillis());

        // HMAC-SHA256 签名
        String signature = signPayload(payload, HMAC_KEY);

        // 发送回调 (重试 3 次)
        for (int retry = 0; retry < 3; retry++) {
            try {
                var entity = org.springframework.http.HttpEntity
                    .>builder()
                    .body(payload)
                    .header("X-Signature", signature)
                    .header("Content-Type", "application/json")
                    .build();
                var response = restTemplate.postForEntity(
                    callbackUrl, entity, String.class);
                if (response.getStatusCode().is2xxSuccessful()) break;
            } catch (Exception e) {
                System.err.printf("[CALLBACK] retry=%d failed: %s%n",
                    retry, e.getMessage());
                try { Thread.sleep(300_000L * (retry + 1)); } catch (Exception ignored) {}
            }
        }
    }

    private String signPayload(Map payload, String key) {
        try {
            String json = new com.fasterxml.jackson.databind
                .ObjectMapper().writeValueAsString(payload);
            Mac mac = Mac.getInstance("HmacSHA256");
            mac.init(new SecretKeySpec(key.getBytes(), "HmacSHA256"));
            return Base64.getEncoder().encodeToString(mac.doFinal(json.getBytes()));
        } catch (Exception e) {
            return "";
        }
    }

    private String generateOne(ContentInput input) { return "生成内容"; }

    record BatchWithCallbackRequest(List inputs,
        String callbackUrl) {}
}

面试常见问答

Spring 中 SSE 流式输出如何实现？生产环境的优化和注意事项？

SSE 流式输出基于 Spring 的 SseEmitter 类实现。Controller 方法返回 SseEmitter 类型,设置超时时间（通常 60 秒）,并用 @Async 注解将 LLM 调用放入独立线程池（Tomcat 的请求线程不阻塞）。核心流程:创建 SseEmitter → 调用 ChatClient.stream() 获取 Flux<String> token 流 → 在 subscribe 的 onNext 回调中通过 emitter.send() 将 token 推送出去 → 流完成后调用 emitter.complete()。生产环境的优化:①缓冲区控制——单个 token 推送频率太高（每秒 20-50 次）会导致前端渲染卡顿,建议每 5 个 token 或每 50ms 做一次批量推送；②心跳机制——Nginx/网关可能因为 60 秒无数据传输而中断 SSE 连接,需要每隔 15 秒发送一条注释行保持连接存活；③断线重连——前端需要监听 onerror 事件,在断线后 1-3 秒自动重连,并在请求头携带 lastTokenIndex 让服务端支持断点续推；④线程池配置——建议 corePoolSize=10,maxPoolSize=50,queueCapacity=200,拒绝策略为 CallerRunsPolicy（调用者线程执行,降级为同步生成）。

与 Python 生成服务的 OpenAPI 契约如何定义才能避免跨团队协作冲突？

跨团队（Java 编排层 + Python 生成层）的 API 契约设计遵循 API First 开发模式:先定义契约再开发实现。具体做法:①两个团队共同维护一个 OpenAPI 3.0 规范的 YAML 文件（存储在 Git 仓库的 api/ 目录下）,所有 DTO 的字段名/POST 内容/响应结构/错误码都在这个文件中定义；②每个 DTO 字段使用 @Schema 注解标注描述（如 @Schema(description="品牌语调五维向量",example="[6,5,8,4,5]")）,字段类型明确（string/number/array）,不允许使用 Object 或 Any 类型；③错误码遵循 IEEE 标准——4xx 表示客户端错误（400 参数不合法/401 未授权/403 无授权/404 资源不存在/429 频率超限）,5xx 表示服务端错误（500 服务器内部错误/502 网关超时/503 服务暂不可用）；④版本协商通过 Accept-Version 请求头:客户端发送 Accept-Version: 2.1,服务端返回对应版本的响应,不支持的版本返回 406 Not Acceptable；⑤契约定义后双方使用契约生成工具:Java 侧使用 OpenAPI Generator 生成 Spring Controller 接口,Python 侧生成 FastAPI 路由定义,确保两端的 API 实现与契约完全一致。冲突发生时,不修改运行时的 API 实现,而是先修改 YAML 契约文件,双团队 review 后再同步修改两端代码。

API 验收清单包含哪些维度？如何执行性能和安全验收？

API 验收清单覆盖四个维度共 27 项检查点。功能维度（10 项,P0-P1 优先级）:所有端点返回正确的状态码,分页参数支持 page/size/sort,SSE 在断线重连后恢复,WebSocket 订阅正确,文件上传下载正常。性能维度（6 项,P0 优先级）:读 API 使用 wrk -c 50 -t 10 -d 60s 压测,通过标准为 500 QPS 且 P99 <500ms；写 API 使用 50 并发请求,SSE 流式接口在 1-2 秒内返回第一个 token,WebSocket 每秒至少推送一次进度。安全维度（6 项,P0-P1）:使用 curl 测试无 Token 请求返回 401,测试 XSS 注入返回 400,用 50 次/分钟的请求测试频率限制返回 429。可靠性维度（5 项,P1-P2）:模拟 LLM 100% 超时测试熔断在 10 秒内触发,kill -15 测试进程是否在 60 秒内完成优雅关闭,模拟断网测试重试机制在 3 次内恢复。建议验收工具使用 Postman Collection + Newman 命令行做自动化功能验收,Gatling 做性能验收,OWASP ZAP 做安全扫描。每次上线前必须执行完整的 27 项验收流程,任何 P0 项不通过则阻止上线。

场景判断 1: SSE 连接中断

用户在使用流式生成时,前 3 秒正常收到 token,第 4 秒突然连接中断,前端 onerror 触发。用户需要重新提交生成请求。请分析可能原因和改进方案。
正确分析: 可能原因:①Nginx 的 proxy_read_timeout 配置为 5 秒,超时后断开连接；②SseEmitter 的 60 秒超时被错误地设置为了 5 秒（检查代码中 SseEmitter 的参数）；③LLM 调用在第 3 秒时抛出了异常（如 API 限流错误）,emitter.completeWithError() 被调用。改进方案:①检查 Nginx 的 proxy_read_timeout 设置为 120 秒以上；②前端 onerror 处理中自动重新连接,并带上 lastToken 字段,服务端支持断点恢复；③添加心跳机制（每 15 秒发送注释行）,确保 Nginx 不会因为"无数据传输"而超时断开。

场景判断 2: 契约冲突

Java 编排层和 Python 生成层的 API 联调时发现:Java 端发送的 GenerateRequest 中 toneParams 字段的 JSON 结构为 {"formality":6,"emotion":5,"expertise":8,"warmth":4,"creativity":5},但 Python 端期望的是 [6,5,8,4,5] 数组形式。450 错误。请判断问题原因和修正方案。
正确判断: 违反 API First 原则——OpenAPI 契约中 toneParams 定义不明确,Java 和 Python 团队各自按照自己的理解实现。修正方案:①立即在 OpenAPI YAML 契约中明确 toneParams 的格式（如 type:array,items:type:number,minItems:5,maxItems:5,example:[6,5,8,4,5]）；②使用 OpenAPI Generator 重新生成两端的代码,确保实现与契约完全一致；③增加契约自动化测试——每次 CI/CD 构建时自动运行契约兼容性检查（使用 spring-cloud-contract 或 pact 框架）,验证两端 API 的请求/响应与契约匹配。
