多级敏感词过滤架构（前缀树+正则+LLM 判断三级递进）

敏感词过滤是内容安全的第一道防线,但单一方法无法同时满足"高召回"和"低误判"的要求——纯粹用词表匹配会漏掉变体（如"fáng 贷"、"fangdai"等变体）,语义判断又可能把正常内容误判为违规。CorpAssist 的三级递进架构将这矛盾分解:第一级用 AC 自动机做 O(n) 精确匹配,保证高召回;第二级用正则模式匹配变体;第三级用 LLM 做语义确认,减少误判。第一级使用 Aho-Corasick 自动机（Trie 树 + 失败指针）同时匹配 10 万+ 敏感词。AC 自动机的构建:建立 Trie 树（每个节点包含子节点数组和失败指针）,使用 BFS 计算每个节点的失败指针（当在当前节点匹配失败时跳转的节点）。匹配过程:从根节点开始,对输入文本的每个字符,沿着子节点或失败指针移动,如果当前节点是"单词结束"标记,则记录匹配。AC 自动机的初始化（构建 10 万词库的 Trie 树）约 50ms,单篇 500 字内容的匹配耗时 <1ms,内存占用约 50MB。

第二级使用正则模式库匹配 AC 自动机无法覆盖的变体。正则模式覆盖四类变体:分隔变体（如"f-a-n-g-d-a-i"用特殊字符隔开,正则:\bf\s*a\s*n\s*g\s*d\s*a\s*i\b）、谐音变体（如"房袋"、"FangDai",正则:房[贷袋代]|[Ff]ang[Dd]ai）、组合变体（如"防范+贷款"组合敏感,两个独立词组合后触发风控,正则:(?=.*防范)(?=.*贷款)）。正则模式库目前维护 500+ 条规则,按行业分类存储（finance/medical/politics/violence 等 12 个类别）。正则匹配使用 Java 的 Pattern 类,编译后复用。500 条正则匹配一篇 500 字内容的耗时为 2-8ms（取决于正剛复杂度）。第三级使用 LLM 语义判断:前两级匹配到的疑似内容可能不是真正的敏感表达（如"他确实很敏感"中的"敏感"不是违规内容）。LLM 对前两级标记的每条命中做二次确认:"内容'{hitWord}'在上下文'{context}'中是否构成违规？"如果 LLM 判断不违规,该命中标记为"误判"并降低风险评分。LLM 判断将误判率从 23% 降低到 8%（减少 65% 的误判）,但增加了 50ms-200ms 的延迟,因此只对 riskScore 在 0.5-0.8 之间的"灰名单"内容启用。

java
import org.springframework.stereotype.Component;
import java.util.*;
import java.util.regex.Pattern;

@Component
public class MultiLevelSensitiveFilter {

    private final AcAutomaton firstLevel;       // AC 自动机
    private final List secondLevel;  // 正则模式
    private final ThirdLevelJudge thirdLevel;    // LLM 确认

    private static final double LLM_CONFIRM_THRESHOLD_LOW = 0.5;
    private static final double LLM_CONFIRM_THRESHOLD_HIGH = 0.8;

    public MultiLevelSensitiveFilter(ThirdLevelJudge thirdLevel) {
        this.firstLevel = new AcAutomaton(loadWordLibrary());
        this.secondLevel = loadRegexRules();
        this.thirdLevel = thirdLevel;
    }

    public FilterResult filter(String content) {
        List hits = new ArrayList<>();
        double maxRisk = 0.0;

        // 第一级: AC 自动机精确匹配
        List level1Hits = firstLevel.match(content);
        hits.addAll(level1Hits);

        // 第二级: 正则模式匹配
        for (RegexRule rule : secondLevel) {
            var matcher = rule.pattern().matcher(content);
            while (matcher.find()) {
                Hit hit = new Hit("REGEX", rule.category(),
                    matcher.group(), matcher.start(), 0.85);
                hits.add(hit);
            }
        }

        // 第三级: LLM 语义确认 (只对 0.5-0.8 的风险内容)
        List confirmedHits = new ArrayList<>();
        for (Hit hit : hits) {
            if (hit.riskScore() > LLM_CONFIRM_THRESHOLD_HIGH) {
                confirmedHits.add(hit); // 高风险直接确认
                maxRisk = Math.max(maxRisk, hit.riskScore());
            } else if (hit.riskScore() > LLM_CONFIRM_THRESHOLD_LOW) {
                // 中风险: LLM 确认
                boolean isViolation = thirdLevel.confirm(
                    hit.word(), extractContext(content, hit.position()));
                if (isViolation) {
                    confirmedHits.add(hit);
                    maxRisk = Math.max(maxRisk, hit.riskScore());
                }
            }
        }

        return new FilterResult(!confirmedHits.isEmpty(),
            confirmedHits, maxRisk);
    }

    private String extractContext(String content, int pos) {
        int start = Math.max(0, pos - 20);
        int end = Math.min(content.length(), pos + 20);
        return content.substring(start, end);
    }

    public record Hit(String level, String category, String word,
                      int position, double riskScore) {}
    public record FilterResult(boolean blocked, List hits,
                               double maxRiskScore) {}
    public record RegexRule(String category, Pattern pattern, double riskScore) {}

    // AC 自动机简化实现
    static class AcAutomaton {
        private final AcNode root = new AcNode();

        public AcAutomaton(Set words) {
            for (String w : words) insert(w);
            buildFailureLinks();
        }

        public List match(String text) {
            List hits = new ArrayList<>();
            AcNode node = root;
            for (int i = 0; i < text.length(); i++) {
                char c = text.charAt(i);
                while (node != root && !node.children.containsKey(c)) {
                    node = node.fail;
                }
                node = node.children.getOrDefault(c, root);
                AcNode temp = node;
                while (temp != root) {
                    if (temp.isEnd) {
                        hits.add(new Hit("AC_AUTO", temp.category,
                            temp.word, i - temp.word.length() + 1, 0.95));
                    }
                    temp = temp.fail;
                }
            }
            return hits;
        }

        private void insert(String word) {
            AcNode node = root;
            for (char c : word.toCharArray()) {
                node = node.children.computeIfAbsent(c, k -> new AcNode());
            }
            node.isEnd = true;
            node.word = word;
        }

        private void buildFailureLinks() {
            Queue queue = new LinkedList<>();
            for (AcNode child : root.children.values()) {
                child.fail = root;
                queue.add(child);
            }
            while (!queue.isEmpty()) {
                AcNode node = queue.poll();
                for (Map.Entry entry : node.children.entrySet()) {
                    char c = entry.getKey();
                    AcNode child = entry.getValue();
                    AcNode fail = node.fail;
                    while (fail != root && !fail.children.containsKey(c)) {
                        fail = fail.fail;
                    }
                    child.fail = fail.children.getOrDefault(c, root);
                    queue.add(child);
                }
            }
        }

        static class AcNode {
            Map children = new HashMap<>();
            AcNode fail;
            boolean isEnd;
            String word;
            String category = "敏感词";
        }
    }

    private Set loadWordLibrary() {
        return Set.of("保本", "稳赚", "包治", "根治", "第一"); // 实际从数据库加载
    }

    private List loadRegexRules() {
        return List.of(
            new RegexRule("finance", Pattern.compile(
                "年化\\s*[收利].*\\d+\\s*[%％]", Pattern.CASE_INSENSITIVE), 0.85),
            new RegexRule("medical", Pattern.compile(
                "包[治愈好].*[病症]", Pattern.CASE_INSENSITIVE), 0.85));
    }
}

@Component
class ThirdLevelJudge {
    // LLM 二次确认
    public boolean confirm(String word, String context) {
        // 调用 LLM 判断该词在上下文中是否构成违规
        return true; // 简化
    }
}

行业合规规则引擎与热加载

不同行业对内容合规的要求差异巨大——金融行业禁止"保本""稳赚"等承诺性表述,医疗行业禁止"包治百病""根治"等绝对化表述,教育行业禁止"包过""保证提分"等效果承诺,广告行业禁止"第一""最好"等绝对化用语,B2B 合规禁止未经授权的客户案例引用。CorpAssist 的合规规则引擎基于 EasyRules 框架,每条规则以 YAML 文件存储,支持热加载（无需重启应用）。规则文件的目录结构:rules/{industry}/{rule-name}.yaml,如 rules/finance/profit-promise.yaml。每条规则定义为:name（规则名称）、description（规则描述）、industry（所属行业）、severity（CRITICAL/MAJOR/MINOR）、condition（SpEL 条件表达式,如 "content.contains('保本') || content.contains('稳赚')"）、action（BLOCK 直接拦截/WARN 告警+人审/ALLOW 允许但记录）,priority（1-10,越小越优先）。

规则引擎在 Spring 启动时扫描 rules/ 目录下的所有 YAML 文件,加载为 EasyRules 的 Rule 对象。通过 Spring Cloud Config 的 @RefreshScope 和 Actuator 的 /actuator/refresh 端点,可以在不重启应用的情况下热加载规则。更新流程:运维人员修改 YAML 文件 → 推送到 Git 仓库 → 触发 webhook 通知 Config Server → 调用 /actuator/refresh 刷新 Bean 中管理规则列表 → 规则引擎使用新规则。当前 65 条行业合规规则覆盖 5 个行业,Rules 引擎的执行性能:65 条规则单次匹配耗时 <3ms。规则的执行顺序按 priority 排序,priority=1 的先执行——因为一些"一票否决"规则（如命中"保本"直接 BLOCK）需要优先执行,后面的规则可以不跑（短路）。

全链路审计留痕与违规应急处理

全链路审计是合规系统的"黑匣子"——记录每一次内容操作,确保在任何时候都可以回溯"谁在什么时候生产了/审核了什么内容/发布到了哪里/结果如何"。审计日志的数据结构:audit_id（UUID,主键）、content_id（关联内容表）、content_version（版本号,每次修改递增）、action_type（GENERATE/MODIFY/REVIEW/AUTO_CHECK/PUBLISH/REJECT/REWRITE）、operator_id（操作用户 ID,系统操作标记为 SYSTEM:模块名）、operator_ip（操作来源 IP）、content_snapshot（操作时刻的内容快照,保存完整正文）、review_result（PASS/REJECT/INTERCEPT/ESCALATE,仅审核操作有）、risk_level（LOW/MEDIUM/HIGH）、compliance_tags（触发的合规规则 ID 列表）、created_at（操作时间,精确到微秒）。每条审计日志约 2-5KB（含内容快照）,日均产生约 10 万条,MongoDB 按月分片存储,保留 7 年（自动过期删除7年前数据）,支持审计日志的可视化查询和导出。审计日志的写入使用异步批量方式（每 100 条或每 5 秒批量插入一次）避免影响主流程性能。

违规应急处理流程定义了四个环节和 1 小时的 SLA。当系统检测到 HIGH 风险违规内容已发布（触发场景:合规过滤漏检或人审误通过）,自动触发应急流程:①自动化撤回（5 分钟内）——调用 CDN API 清理缓存（阿里云 CDN Refresh）、调用发布平台 API 下线内容（CMS/EDM/社媒）、记录撤回操作到审计日志；②通知（15 分钟内）——通过钉钉/企业微信机器人群发通知到品牌负责人、法务、技术负责人,通知内容包含内容 ID、违规类型、风险描述和影响范围；③复盘（1 小时内）——CorpAssist 自动生成违规复盘报告,包含根因分析（哪个环节漏检了？是第一级敏感词没覆盖还是第二级正则规则没匹配还是第三级 LLM 误判？）、影响范围（内容展示了多少次/被哪些用户看到）、改进措施（需要在哪个环节补充规则或模型调优）；④规则更新（1 小时内）——将违规内容中的新敏感词/新模式补充到词库,创建或修改合规规则 YAML 文件并热加载,将违规内容加入坏例库的"合规违规"类别。CorpAssist 的生产数据:2025年共触发 3 次应急流程,平均完成时间 47 分钟,均达到 1 小时 SLA。

如何确保审计日志的完整性和不可篡改性？

审计日志的完整性通过三个机制保证。第一是"写前日志"(Write-Ahead Log)——所有内容操作在业务执行前先写入审计日志表（使用 MongoDB 事务+writeConcern=majority）,如果业务执行失败,审计日志仍然保存"尝试操作"的记录,确保操作可追溯。第二是哈希链(Hash Chain)——每条审计日志的 id 字段不是随机 UUID,而是 hash(上一条日志的 hash + 本日志的 content),形成一条哈希链。如果某条日志被篡改（修改 content 字段）,它和后续所有日志的 hash 链接都会断裂。系统每天凌晨运行一个校验任务,遍历当天的审计日志哈希链,检查完整性。第三是 WORM 存储(Write Once Read Many)——审计日志表设置为 append-only（MongoDB 中通过 writeConcern+validator 约束不允许 update 和 delete 操作只有 insert）,数据库账号权限分离,应用层只有 INSERT 和 SELECT 权限。配合阿里云 OSS 的归档存储:每月的审计日志生成加密归档副本存入 OSS 的 Archive 存储类型（只读,无法修改）,保留 7 年。

全链路审计哈希链实现细节

审计日志的完整性通过三条防线保障。第一是WAL(Write-Ahead Log)——所有操作在业务执行前先写入审计日志,使用MongoDB事务+writeConcern=majority,即使业务失败日志也保存了"尝试操作"的记录。第二是哈希链(Hash Chain)——每条日志id = SHA256(上一条hash + 本日志内容 + 时间戳),形成一条不可篡改的链。系统每天凌晨校验整条链的完整性。第三是WORM存储(Write Once Read Many)——表设置为append-only,数据库账号只有INSERT和SELECT权限。每月生成加密归档副本存入OSS Archive,保留7年。365天校验全部通过零篡改。

面试常见问答

如何系统性地防止有害、违规内容的生成？技术架构和三级过滤如何配合？

系统性的内容安全防护采用"预防+检测+应急"三维体系。预防层面:在 Prompt 的 System Prompt 中加入硬性约束（"禁止生成任何涉及金融/医疗/政治敏感的内容"）,并在模板的变量校验中加入合规正则检查。检测层面为核心:三级递进敏感词过滤——第一级 AC 自动机 O(n) 精确匹配 10 万词库（50ms 初始化,<1ms/篇匹配）,保证高召回；第二级 500+ 正则模式匹配变体/谐音/组合词（2-8ms/篇）；第三级 LLM 语义确认对中风险命中做二次判断,减少误判 65%。检测之后还有行业合规规则引擎——65 条 EasyRules YAML 规则覆盖金融/医疗/教育/广告/B2B 五个行业,每条规则含 SpEL 条件（如 "content.contains('保本')"）和动作（BLOCK/WARN/ALLOW）,支持热加载。所有检测结果送入 Flowable 审核工作流,高风险自动驳回,低风险进入人审。应急层面:1 小时内完成撤回→通知→复盘→规则更新的闭环。2025 年全系统违规发布率 0.008%,零罚款。

行业合规规则引擎的设计原则和热加载如何实现？

规则引擎基于 EasyRules 框架,遵循三条设计原则:规则即代码（Rules as Code）、行业隔离（Industry Isolation）、分级响应（Severity Grading）。每条规则以独立的 YAML 文件存储（rules/{industry}/{rule-name}.yaml）,包含 name/description/industry/severity/condition(action)/priority 七个字段。condition 使用 SpEL 表达式（如 "content.contains('保本') || content.contains('稳赚')"）,action 有三级:BLOCK（直接拦截）+WARN（告警进入人审）+ALLOW（允许但记录）。热加载基于 Spring Cloud Config:@RefreshScope 标注 RuleEngine Bean,调用 /actuator/refresh 端点时重新加载 rules/ 目录下的所有 YAML 文件,编译为新 Rule 对象替换旧规则列表。YAML 变更推送到 Git 仓库后,通过 webhook 通知 Config Server 触发 refresh。整个过程不需要重启应用,规则文件在 5 秒内生效。规则的执行逻辑按 priority 排序（1-10）,priority=1 的规则优先执行且短路——如果 BLOCK 动作触发直接返回不执行后续规则。规则的编写需要团队共同维护规则库,每次新增规则需要通过测试用例验证（测试用例覆盖正例和反例,规则上线通过率需 100%）。

违规内容的应急处理流程如何设计？1 小时 SLA 的四个环节？

违规应急处理覆盖四个环节,总 SLA 1 小时。环节一撤回（5 分钟内）:系统检测到已发布的 HIGH 风险违规后,自动调用阿里云 CDN Refresh API 清理全网点播缓存,调用发布平台 API 下线内容（CMS/EDM/社媒渠道各需独立调用）,将内容状态置为 RECALLED 并记录撤回操作到审计日志。环节二通知（15 分钟内）:通过钉钉/企业微信机器人群发通知模板,内容包含本次违规的 contentId/违规类型/风险描述/影响范围/当前处理状态,@ 品牌负责人、法务负责人和技术负责人。环节三复盘报告（1 小时内）:系统自动收集违规链路的完整日志（生成→机审→人审→发布的日志时间线和结果）,定位漏检环节,生成结构化复盘报告。常见漏检原因:敏感词库未覆盖新词/人审误通过（人评分 4.5 但实际违规）/LLM 合规约束被忽视。环节四规则更新（1 小时内）:根据根因补充敏感词库、创建新规则 YAML 文件并热加载、将违规内容加入坏例库的 compliance 类别。四个环节完成的标准是:违规内容已不可访问、所有相关人员已通知、复盘报告生成、新规则已上线。每次应急后 1 周内进行"改进效果验证":运行 100 条类似的敏感测试内容,确保新规则能正确拦截。2025 年 CorpAssist 触发 3 次应急,均实现 1 小时 SLA,平均完成时间 47 分钟。

场景判断 1: 规则误判

某医疗品牌的内容"这款护肤品不含激素,适合敏感肌"被合规规则引擎拦截（条件:content.contains('激素') + industry=medical → BLOCK）。但内容本身合规（"不含激素"是合规表述）。请判断问题和改进方案。
正确判断: 规则条件过于简单——只检查了"含激素"的精确匹配,没有考虑否定词"不含"的前缀。改进方案:在规则的 SpEL 条件中增加否定词排除:condition="content.contains('含激素') || content.contains('添加激素')"（去掉"不含激素"的精确匹配）,或者使用更精细的正则:"(?<!不)(?<!无)含激素"。同时规则引擎应该支持"例外列表"（exception list）功能——如果命中词前面有"不/无/没有"等否定词,降低风险级别。

场景判断 2: 审计溯源

法务部门需要追溯某条违规邮件内容的完整生成和审核记录,但因为审计日志只记录了"内容快照"未记录"审核员的判定依据",无法判断是审核员误放行还是系统漏检。请分析问题并改进。
正确判断: 审计日志的 review_result 字段只记录了 PASS/REJECT,缺乏判定依据。改进方案:在审计日志中增加 judgement_basis 字段,记录审核员通过/驳回的具体原因（从审核界面获取审核员选择的原因标签,如"品牌语调偏差/事实错误/敏感词漏网"）和 confidence_level。同时增加审核员的操作上下文日志:审核员在审核页面上停留的时间、是否展开了内容详情查看、是否查看了机审结果详情——这些行为数据可以帮助判断审核员是否认真执行了审核。此外,A/B 对比审计日志的完整性和可搜索性:将 MongoDB 的审计日志同步一份到阿里云日志服务（SLS）,支持按内容关键词/用户/时间范围的全文本搜索,法务查询时间从 2 天缩至 10 分钟。
