Q1: 如何让AI生成的营销内容符合品牌风格？技术和流程上怎么做？

这是一个系统工程问题,不是靠"在 Prompt 里写一句'请用品牌风格'"就能解决的。我从四个技术层面来保证品牌一致性。第一是品牌语调的量化定义——使用五维参数模型（正式度/情感度/专业度/亲和力/创意度,每维 0-10）,将抽象的风格描述转化为数值向量。比如某个美妆品牌的语调向量是 [6,7,8,6,5],这意味着中等正式、高情感、高专业度（强调成分科学背书）、中等亲和力、中等创意度。每个维度和 LLM 的 temperature/topP 等参数有映射关系——正式度高则 temperature 低（0.2-0.4）,情感度高则 topP 高（0.9-0.95）。五维参数通过 SpEL 变量注入到 Prompt 模板中,实现"品牌风格即代码"的工程化管控。

第二是 Prompt 层面的精确控制——System Prompt 中注入品牌角色的详细定义（"你是一位资深美妆文案专家,擅长为 XX 品牌撰写产品描述"）,User Prompt 中通过 SpEL 变量传递语调参数（formality=6, emotion=7, expertise=8, warmth=6, creativity=5）,模板支持条件渲染（<#if product.price > 5000>高端产品</#if>）。模板使用 Git 做版本管理（SemVer 语义化版本号:主版本.次版本.修订号）,模板变更自动创建 Git commit,支持版本回溯。通过 Spring Cloud Config 实现模板的 A/B 测试——新模板上线前先灰度 5% 流量,运行 1-2 周后评估 p<0.05 才能全量上线。第三是 Few-shot 示例的动态选择——从 200+ 条精标示例库中,根据内容类型和品牌语调向量动态选择最匹配的 Top-3 示例注入 Prompt。选择算法:得分 = 类型匹配度 40% + 语调余弦相似度 30% + 受众匹配 20% + 字数接近度 10%。

第四是事后校验——生成内容后,ComplianceAdvisor 中的语调校验模块评估五个维度的实际得分,与目标语调向量做余弦相似度计算,阈值 0.85——低于 0.85 标记为"语调偏差"并触发驳回重写。四个层面结合,品牌一致性从 72% 提升到 93%,语调偏差从平均 1.8 分/维度缩小到 0.6 分/维度。面试官可能会追问:示例库怎么维护？回答要点是"生成→修改→沉淀"的飞轮机制——内容通过审核后,如果编辑修改量超过 30%,自动进入候选示例池,运营每周审核入库。同时每个季度淘汰 60 天未匹配的冷门示例。

Q2: 长文（5000字+）如何保证逻辑连贯不重复？

长文生成的核心矛盾是 LLM 在处理 5000 字以上的内容时会出现"中间迷失"——开头和结尾写得很好,中间部分逻辑松散,以及跨段落的重复（同一数据在第 2 段和第 5 段反复出现）。我的解决方案分三段:三阶段流水线架构、NLI 逻辑连贯性检测、SimHash 去重机制。三阶段流水线将长文生成拆解为"大纲 30 秒（temperature=0.3 低温度保证结构合理性,输出 5-8 章 JSON 结构化大纲）→ 分章节写作 3-5 分钟（每章独立调用 LLM,通过 RocketMQ 实现 5-8 并发,关键设计是'上下文传递'——生成第 N 章时传入第 N-1 章的 100 字摘要+本章大纲）→ 拼接润色 1-2 分钟（三次润色:消除过渡生硬→统一术语风格→生成摘要目录,通过 Spring Batch 3-Step Job 组织,temperature=0.2-0.4 低温度控制修改幅度）。

NLI 检测使用 cross-encoder 模型计算相邻段落的 P(entailment) 和 P(contradiction),衔接评分 = P(entailment) / (P(entailment)+P(contradiction)) <0.6 触发重写,单篇检测耗时 4-10 秒。SimHash 去重对每段落计算 64 位局部敏感哈希,海明距离阈值 3,候选对再用编辑距离确认（归一化重复率 >0.3 判定重复）。实测:段落衔接评分从 0.55 提升至 0.82,内容重复率从 8% 降至 2.1%,5000 字以上长文直接生成成功率从 68% 提升到 99.5%,质量评分从 3.0/5 提升到 4.3/5。

Q3: 用什么指标和流程评估AI生成内容的质量？

质量评估的核心原则是"可量化、可复现、可改进"。评估体系包含自动评估、人工校准和坏例回测三个层次。自动评估使用 LLM-as-Judge（temperature=0.1 保证稳定输出）对生成内容进行四维评分:流畅度（语言通顺度,权重 20%）、相关性（内容紧扣主题,25%）、品牌一致性（语调贴合品牌,30%)、合规性（安全规则和事实准确,25%）。品牌一致性权重最高(0.30),因为 AI 生成内容最容易被诟病的就是"写的不像我们品牌"。Judge 的 Prompt 包含角色定义+评分标准+正反例+JSON 输出格式约束。每次评估 500-800 tokens,成本约 ¥0.005-0.008。综合评分 = 0.20×流畅度 + 0.25×相关性 + 0.30×品牌一致性 + 0.25×合规性。

人工校准通过双周一次的人机一致性评估实现:抽取 50 条内容,3 名审核员使用 Rubric 标尺独立打分,与 Judge 评分对比计算 Cohen's Kappa 系数。当前 Kappa=0.73（高度一致级）。Kappa < 0.6 时触发修正流程:分析不一致方向→补充评分标准正反例→调整权重→重新校准,直到 Kappa 恢复到 >0.6。坏例回测是质量改进的闭环:每次 Prompt 模板或 LLM 模型变更后,自动化运行 200+ 条坏例（当前库 1850+ 条）回归测试。通过标准:通过率 >95% 且无退化（综合评分下降 >0.3 的用例 <5%）。此外,监控质量趋势的 Grafana Dashboard,14 天四维评分趋势线和通过率/驳回率变化,告警规则:综合评分 7 日均线下降 >0.3 分触发钉钉告警。

Q4: 如何系统性地防止有害、违规内容的生成？

内容安全采用"预防+检测+应急"三位一体体系。预防层面在 Prompt System Prompt 中加入硬性约束（"禁止生成金融承诺/医疗疗效/政治敏感内容"）,模板变量校验环节用正则拦截明显违规输入（如"年化\d+%"）。检测层面是核心:三级递进敏感词过滤。第一级 AC 自动机 O(n) 匹配 10 万词库,50ms 初始化,<1ms/篇匹配,高召回不遗漏已知词。第二级 500+ 条正则规则覆盖变体/谐音/组合词。第三级 LLM 语义确认——对前两级中风险命中（riskScore 0.5-0.8）做二次判断,将误判率从 23% 降低到 8%。检测之后还有行业合规规则引擎（EasyRules,65 条 YAML 规则覆盖金融/医疗/教育/广告/B2B 五个行业,通过 Spring Cloud Config 热加载,每条规则含 SpEL 条件和 BLOCK/WARN/ALLOW 动作）。

应急层面:高风险内容一旦发布——5 分钟自动 CDN Refresh+API 下架,15 分钟企业微信通知品牌/法务/技术负责人,1 小时内自动生成复盘报告并更新规则。全链路审计记录每个环节（谁/什么时间/生成什么/审核结果/发布到哪里）,审计日志通过哈希链保证完整性,每天校验,数据保留 7 年（WORM 存储+OSS 归档）。2025 年数据:违规发布率 0.008%,3 次应急均 1 小时内完成（平均 47 分钟）,零合规罚款。

Q5: 如何为不同用户群体实现个性化内容生成？

个性化方案分为三步:用户画像→特征匹配→语调自适应。第一步——用户画像数据使用 Redis 复合结构存储:Hash 存基础属性（<3ms 读取）、Set 存标签集合（支持集合运算便于分群）、Sorted Set 存行为序列（90 天窗口,最近的优先）。画像通过 Pipeline 批量写入（每次事件写入 3-5 个 key,耗时 2-5ms）。第二步——将画像编码为 15 维特征向量（年龄/会员等级/城市/活跃度/品类偏好/价格敏感度/设备类型）,与模板的"目标用户特征向量"做加权余弦相似度。品类偏好权重最高(0.30),年龄和会员等级各 0.15,选择 Top-1 模板。第三步——在选中模板基础上根据用户特定属性微调 Prompt 语调参数:青年用户（<25 岁）情感度+2+创意度+2,高端会员（gold+）专业度+1.5,老用户亲和力+1。所有个性化效果基于 A/B 实验验证——hash(userId)%100 分流,Spring Cloud Gateway 层注入分组,实验 1-2 周,最小样本量 1 万/组。三个核心指标:CTR 从 2.8% 到 6.5%（+132%）、CVR 从 1.2% 到 2.8%（+133%）、NPS 从 42 到 58（+38%）。增量收入公式 = [(CTR_B×CVR_B×AOV_B) - (CTR_A×CVR_A×AOV_A)] × N,一次大促活动个性化增量收入约 26 万元。冷启动问题——行为标签 <5 个的新用户使用统一模板+偏好收集机制,通过首次交互点击快速建立初始特征画像。

"先说结论再说细节"——每道题先一句话概括核心方案再展开。每条数据要有基线对比,不要说"一致性提升"要说"从72%提升到93%"。技术选型要给出理由——"用Spring Batch因为Chunk模式匹配批处理需求"。适当使用术语(SimHash,Cohen Kappa,AC自动机)但要说明是什么。更重要的是不要背稿要有交流感。每道题回答2-3分钟,追问时坦诚说"这个细节我理解的大致原理是…"即可。面试前将五道题要点写在卡片上,对着镜子练习5-10遍。重点记忆7组数据:一致性72%→93%/衔接0.55→0.82/质量3.0→4.3/Kappa=0.73/违规率0.008%/CTR+132%/CVR+133%。

模拟面试追问

追问Q1:品牌语调五维参数是怎么确定权重的？不同行业需要调整吗？

五维参数初始权重是通过对 20 个品牌团队的问卷调研确定的——让品牌团队对 50 篇内容的五个维度打分,通过主成分分析(PCA)确定每个维度的方差贡献率,方差贡献率高的维度权重更高。不同行业权重差异显著:奢侈品行业正式度权重最高(0.30),美妆行业情感度和创意度合计 0.50,B2B 科技公司专业度权重最高(0.35)。每个品牌接入时先按行业预设一套初始权重,然后通过 2 周磨合期的实际数据（至少 100 条人工标注）做回归分析微调权重,确保权重调整后人机评分一致性提升。

追问Q3:LLM Judge 和人工评估不一致时怎么处理？

不一致是常态——当前 Kappa=0.73,还有 27% 的不一致空间。双周校准发现 Kappa<0.6 时触发五步修正:①分析不一致方向（机器系统性给高分还是某个维度差异特别大）;②检查该维度的评分标准是否模糊——最常见问题是机器只比对了五维向量余弦距离,人工还考虑了"语感"和"品牌情感"等难以量化的因素;③在 Judge Prompt 中补充更具体的正反例;④重新校准测试到 Kappa 恢复到 >0.6;⑤更新 Rubric 标尺,将隐性的判断标准显性化。这个流程每季度至少执行一次,模型或模板变更后立即校准。

追问Q2:SimHash 阈值为什么设 3？太严格会不会误伤？

阈值通过实验确定——从生产环境随机抽取 2000 对段落,人工标注"是否重复",测试不同阈值下的精确率和召回率。Threshold=3 时精确率 82%（82% 候选对确实是重复的）,加上编辑距离确认后精确率提升到 99.5%,召回率 97%。设为 2 时精确率更高(95%)但召回率下降到 85%;设为 4 时召回率 99% 但精确率仅 70%,CPU 开销增加 50%。Threshold=3 是平衡点,而且 SimHash 候选对还要经过编辑距离确认（归一化重复率>0.3才算确认重复）,所以即使误报一些非重复对也会被过滤掉。实践中误判率 <0.5%。

场景判断 1: 面试官问准确率和召回率

回答 Q4 时说"AC 自动机精确率 82%,召回率 97%",面试官追问计算方式和意义。你的回答？
正确回答: 精确率=系统判定违规且人工确认违规数/系统判定违规总数=正确拦截/总拦截,召回率=系统判定违规且人工确认违规数/人工标注违规总数=正确拦截/应当拦截。AC 自动机精确率 82% 意味着 100 条拦截中有 18 条误判,召回率 97% 意味着 100 条违规中 97 条被正确拦截。加入 LLM 第三级确认后精确率提升到 92%（误判减少 55%）,召回率略微下降到 96%。

场景判断 2: 面试官问选型理由

介绍架构时提到使用 ChatClient 调用 LLM,面试官追问具体模型和选型理由。你的回答？
正确回答: 生成模型用 DeepSeek-V3（中文质量够用,¥0.0105/1K tokens,性价比最优）,Judge 模型用 DeepSeek-V3 低温度版本(0.1),多模态生成用通义万象（¥0.02/张）和 Stable Diffusion+LoRA（可微调保持品牌一致性）。选型三考量:质量 4.3/5 够用稳定,成本比 GPT-4o 便宜 3-4 倍,控制力可通过 temperature 和 Prompt 精确调节。备选方案:DeepSeek 不稳定时自动降级到 Qwen-Plus。
