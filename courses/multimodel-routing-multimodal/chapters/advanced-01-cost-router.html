为什么要学本章

CorpAssist NS3 · S5 内容创作 在营销旺季会同时跑文案生成、附件理解与批量改写；若所有请求一律走 qwen-plus 或 VL 大模型，账单会在一周内失控。本章把 cost aware route 落成可配置策略：small model first 做意图与草稿，semantic cache 命中热点问句，token budget 与 spend cap 做硬闸。Spring Router 与 Python 路由服务共用同一份 cost-policy.yaml，避免双栈各算各的账。

本章先记住 3 件事
  • 先便宜后贵：分类、摘要、模板填充走 turbo；仅「需创意/需视觉」才升档。
  • 缓存是路由的一环：命中缓存 = 路由决策，不是事后优化。
  • 预算可拒绝：超 spend cap 应降级或排队，而不是静默超支。

小模型前置

small model first 不是「永远用小模型」，而是在 cost aware route 管道最前段用低成本模型完成可验证的轻任务：意图识别、槽位抽取、草稿大纲、是否需要附件/VL 的二分类。CorpAssist S5 典型链路：用户上传品牌手册 PDF → qwen-turbo 判断「纯文本问答 / 需读图 / 需长文改写」→ 仅第三档才路由到 qwen-plus 或 DashScope VL。

阶段推荐模型档S5 示例升档条件
意图 / 分类turbo / mini「这条是 slogan 还是 FAQ？」置信度 < 0.85 或合规敏感
草稿 / 结构化turboJSON 槽位、邮件模板骨架用户点「精修」或品牌调性检测失败
终稿 / 多模态plus / VL海报文案 + 配图理解路由标签 needs_vision=true
cheap classifier

Spring RouteAdvisor 或 Python /v1/route/classify；输出 model_tier 与 estimated_tokens。

token budget

请求前扣减预算；VL 按图片块数乘系数，防止单请求吃光日配额。

spend cap

租户 / 活动级日上限；触顶后强制 degrade_tier=static 或排队。

成本策略配置（双栈共用）

yaml
cost_router:
  classify_with: qwen-turbo
  generate_with: qwen-plus
  vision_with: qwen-vl-max
  small_model_first: true
  token_budget:
    per_request_max: 8192
    vision_multiplier: 4.0
  spend_cap:
    daily_usd: 500
    tenant_override: ${TENANT_SPEND_CAP}
  escalate_rules:
    - when: confidence < 0.85
      then: generate_with
    - when: needs_vision == true
      then: vision_with

不推荐
  • 所有 S5 文案一律 plus，无分类
  • 分类与生成用同一 Prompt 重复调两次大模型
  • 预算是事后报表，不能拦截请求
推荐
  • cheap classifier 输出结构化路由标签
  • 升档可审计：记录 escalation_reason
  • spend cap 触顶有用户可见降级文案
CorpAssist · S5 营销战役 · 成本闸 大促前运营把 daily_usd 从 800 调到 500，并把「海报生成」默认从 plus 改为 turbo 草稿 + 人工审核升档；cost dashboard 按 campaign_id 看 spend，无需改 Java/Python 业务代码。
先想 10 秒：为何分类要用独立小模型而不是规则引擎？

S5 意图边界模糊（「改写 slogan」vs「解释附件表格」）；小模型成本低且可随 policy version 迭代。纯关键词规则在 NS3 多语言场景维护成本更高。

缓存

在 cost aware route 中，semantic cache 与模型选择同级：请求进入 Router 后先查 cache tier，命中则跳过 LLM 调用。CorpAssist 附件 FAQ、品牌话术库、活动期重复问句是三类高命中场景。

cache tier键策略适用风险
L0 精确prompt hash + model_id模板填充、固定 FAQ低；需注意 PII 不入缓存
L1 语义embedding 相似度 ≥ 0.92口语化同义问中；需 TTL 与版本戳
L2 会话session_id + 末轮摘要多轮改写草稿高；仅短 TTL（分钟级）

Spring Router：缓存 Advisor 示意

java
@Component
public class SemanticCacheAdvisor implements CallAdvisor {

  @Override
  public ChatClientResponse adviseCall(
      ChatClientRequest request, CallAdvisorChain chain) {
    Optional<CachedReply> hit = cache.lookup(
        request.prompt(), request.context("route_version"));
    if (hit.isPresent()) {
      return hit.get().toResponse(/* cache_hit=true */);
    }
    ChatClientResponse response = chain.nextCall(request);
    cache.store(request.prompt(), response, ttlOf(request));
    return response;
  }
}

Python 路由服务：语义命中

python
async def route_with_cache(body: RouteRequest) -> RouteDecision:
    if cached := await semantic_cache.get(body.text, min_score=0.92):
        return RouteDecision(model_id="cache", cached=True, cost_usd=0.0)
    tier = await cheap_classify(body.text)
    return RouteDecision(model_id=tier.model, cached=False)
常见误区
  • 缓存不含 route_version — 策略升级后仍返回旧模型答案，治理章会展开回滚。
  • VL 结果盲目缓存 — 图片内容变更时语义键失效，应对附件 content_hash 建复合键。
  • 只统计 LLM 成本 — embedding 与 Redis 也有 spend，应进 cost dashboard。

自检：S5 附件问答何时不应走语义缓存？

当附件 content_hash 变化、或问题含「最新/今天」等时效词、或用户显式要求「重新生成」时，应 bypass 缓存并打 cache_skip_reason 审计字段。

本章结论

成本路由 = 分类前置 + 缓存前置 + 预算硬闸，双栈共用策略文件。

  • 小模型前置：cheap classifier 决定 tier；升档可审计，VL 仅在有视觉需求时触发。
  • 缓存：cache tier 分层；语义命中需 version 与 content_hash；命中计入零成本路由。
  • 治理指标：token budget、spend cap、cost dashboard 三件套闭环。
  • 双栈：Spring Advisor 链与 Python /v1/route 读同一 cost-policy.yaml（见 Demo）。

判断练习

  1. 运营要求「所有回复更聪明」而把默认模型从 turbo 改成 plus，未改 spend cap。请指出违反本章哪条原则及推荐改法。

    参考答案

    违反 small model first 与预算闭环。应仅对低置信度或用户显式「精修」升档，并同步调高 spend cap 或缩小活动范围。

  2. 语义缓存命中率 40%，但用户投诉答案过时。最可能缺哪两个字段？

    参考答案

    缓存键缺 route_version 或附件 content_hash；且 TTL 过长。应对策略升级与附件变更失效缓存。
