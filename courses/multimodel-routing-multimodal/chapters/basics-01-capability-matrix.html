为什么要学本章

CorpAssist S5 内容创作 / 附件问答 同时处理 FAQ、长文档摘要、PDF 图文与品牌文案。若团队凭感觉选「最强模型」，NS3 成本会失控；若一律用 qwen-turbo，视觉 QA 与复杂创作质量又不够。本章用 capability matrix 把商用/开源 · 价格 · 上下文三维对齐，为后续 路由策略 与 Spring/Python 双栈提供可审计的 model catalog。

本章先记住 3 件事
  • 矩阵先于路由：没有能力矩阵的 route rule 是拍脑袋；NS3 成本路由需要可量化的 tier 字段。
  • 场景驱动选型：FAQ 重延迟与成本；附件 QA 重 VL 与上下文；创作重质量 — 不同列权重不同。
  • 双栈共用目录：Java Spring Router 与 Python 批处理读同一份 matrix/catalog，避免两栈各维护一份 Excel。

商用/开源

CorpAssist S5 在线链路以 DashScope 等 commercial model 为主：SLA、计费透明、VL/OCR 能力开箱即用。开源权重（open source llm）更适合内网离线评测、敏感数据沙箱或成本极限场景，但运维 GPU、版本升级与多模态插件需自建。

类型典型代表S5 场景路由含义
商用 API qwen family（turbo/plus/max/vl-plus） 在线附件 QA、品牌文案、DashScope VL 写入 model catalog 的 primary 候选
商用 API（对照） GPT-4o / GPT-4o-mini 跨境对照评测、采购议价基准 gpt compare 列，非默认生产路径
开源权重 Qwen2.5、Llama 等（自托管） 离线回归、脱敏文档试跑 Fallback 末级或 dev 替身，需单独 tier
商用模型

按 Token/QPS 计费；能力矩阵记录单价、上下文上限、是否支持 VL/JSON mode。

开源 LLM

许可与合规需法务确认；矩阵中标注「仅内网」「无 VL」等硬约束。

能力矩阵

不是「模型排行榜」，而是业务场景 × 模型能力 × 成本的可路由表格。

先想 10 秒：S5 品牌海报 OCR 问答，为何不能只用 qwen-turbo？

turbo 无可靠 VL 能力；应查矩阵中「multimodal=vision」行，路由到 qwen-vl-plus 或同级 VL 模型，否则 OCR 质量与引用图表会失败。

价格

NS3 要求「成本 vs 质量」可量化。矩阵为每个模型标注 price tier（相对档位即可），并记录输入/输出单价或「每 1M token 参考价」，供 Router 与 FinOps 仪表盘聚合。

yaml
# models/catalog.yaml — 与 routing.yaml 的 cost_tier 对齐
models:
  - id: qwen-turbo
    provider: dashscope
    price_tier: low
    input_per_1m: 0.3    # 示意价，以官网为准
    output_per_1m: 0.6
    modalities: [text]
  - id: qwen-vl-plus
    provider: dashscope
    price_tier: medium-high
    modalities: [text, vision]
  - id: gpt-4o-mini
    provider: openai_compat
    price_tier: medium
    gpt_compare: true      # 仅评测/降级，非默认 primary
常见误区
  • 只比单价不比调用量 — FAQ 量大时 turbo 省下的钱远大于偶尔升档。
  • VL 与文本共用 tier — 图片 token 计价方式不同，需单独列。
  • 矩阵不写更新日期 — 云厂商调价后路由成本预测会失真。

上下文

context window 决定单次请求能塞多少附件正文。S5 场景常见：100 页 PDF 经 OCR 后 80K token、营销长文 32K、FAQ 仅 2K。矩阵必须标注「硬上限」与「推荐上限（留出生成空间）」。

S5 场景估算 context矩阵推荐模型超出窗口时
单页 FAQ< 4Kqwen-turbo—
合同附件全文 QA32K~128Kqwen-plus / 长上下文版Map-Reduce 分块 + RAG
扫描 PDF + 图表16K + 图像 tokenqwen-vl-plus按页分片 vision_qa
品牌手册多轮创作历史 20K + 新稿 8Kqwen-plus摘要压缩 Memory

上下文边界：标签页对比

适用：token 估算 < 推荐上限的 70%，如 40K 合同走 128K 模型。

风险：接近上限时延迟与成本陡增；失败重试浪费双倍 token。

适用：超长附件、多 PDF 合并；先 OCR/分块入 VectorStore（见 practice 章 VL RAG）。

路由：矩阵标注 requires_rag_if_over=32k，Router 在超阈值时改 intent。

Java：按 token 估算选择 catalog 条目

java
@Service
public class ModelCatalogSelector {

  private final ModelCatalog catalog;

  public ModelEntry selectForAttachment(AttachmentMeta meta) {
    int estTokens = meta.estimatedTextTokens() + meta.visionTokenOverhead();
    if (meta.hasVision()) {
      return catalog.find("qwen-vl-plus")
          .filter(m -> estTokens <= m.recommendedContext())
          .orElseThrow(() -> new ContextTooLargeException(estTokens));
    }
    if (estTokens <= 8_000) {
      return catalog.get("qwen-turbo");
    }
    if (estTokens <= 128_000) {
      return catalog.get("qwen-plus");
    }
    throw new ContextTooLargeException("needs_rag_chunking", estTokens);
  }
}

Python：批处理评测读同一 catalog

python
# scripts/load_catalog.py — 与 Java 共用 models/catalog.yaml
from pathlib import Path
import yaml

def load_catalog(path: Path = Path("models/catalog.yaml")) -> dict:
    data = yaml.safe_load(path.read_text(encoding="utf-8"))
    return {m["id"]: m for m in data["models"]}

def pick_model(intent: str, est_tokens: int, has_vision: bool) -> str:
    catalog = load_catalog()
    if has_vision:
        return "qwen-vl-plus"
    if est_tokens <= 8_000:
        return "qwen-turbo"
    if est_tokens <= 128_000:
        return "qwen-plus"
    raise ValueError(f"context {est_tokens} requires RAG chunking")
自检：矩阵里「context_window=128K」是否意味着可以塞 128K 附件？

否。需预留 system prompt、工具输出与用户问题 token；矩阵应写 recommended_context（如 90K），Router 在超推荐值时走分块或升档策略。

能力矩阵与 routing.yaml 是什么关系？
catalog 描述「模型能做什么、多少钱、多长上下文」；routing.yaml 描述「什么 intent 用哪个模型」。Router 选 model 前应先校验其在 catalog 中存在且未下线。
NS3 成本路由在哪一章展开？
成本感知路由 会在矩阵 tier 之上增加小模型前置、缓存与 spend cap；本章只需把 tier 与 context 字段写对。

本章结论

每条对应上文一个小节，便于串起全章再做题。

  • 商用/开源：S5 在线以 DashScope 商用为主；开源入矩阵需标注合规与模态限制；VL 场景必须单独成行。
  • 价格：price tier 与 NS3 成本路由挂钩；gpt compare 用于对照与 multi-vendor Fallback，非默认 primary。
  • 上下文：context window 硬上限 + 推荐上限；超长附件走分块/RAG 而非硬塞 max 模型。
