为什么要学本章

CorpAssist NS3 · S5 的路由策略每周都会改：大促升档模型、合规要求禁用跨厂商、缓存 TTL 调整。若 routing.yaml 只躺在某台机器上，就无法回答「上周五 14:00 谁把 FAQ 从 turbo 改成 plus？」本章建立 route governance：route gitops 发布、change approval 门禁，以及 who called what 级审计日志。

学完你能
  • 用 Git 管理 policy version，并实现 config rollback 与双栈同步
  • 设计 audit log 必含字段，满足 NS3 合规与成本对账
  • 说明 compliance route 如何拦截跨境模型调用
本章先记住 3 件事
  • 策略即代码：routing / cost / degrade YAML 入 Git，CI 校验后再发布。
  • 每次调用可追溯：request_id + route_version + model_id + user_id 四件套。
  • 回滚要快：config rollback 应分钟级，且失效语义缓存。

策略版本

policy version 是路由决策的时间戳真源。Spring 与 Python 启动时加载 version 字段，请求上下文携带 route_version；变更走 PR → 审批 → 流水线热更新，而非 SSH 改文件。

制品版本字段发布方式回滚
routing.yamlversion: 2026.05.3Git tag + ConfigMaprevert tag，双栈 reload
cost-policy.yamlcost_router.version同上回滚 + 清 L1 缓存
degrade.yamldegrade.version同上回滚 + 重置 circuit
route gitops

策略仓库独立或 mono-repo config/route/；Argo CD / 自研 sync job 推送到 K8s。

change approval

升档到 plus/VL 或启用 multi-vendor 需二线 + 合规签字；CI 检查 approver 标签。

config rollback

一键回滚到上一 tag；Router 广播 policy_version 变更事件，缓存层 bulk invalidate。

不推荐
  • 生产环境 kubectl edit ConfigMap 无记录
  • Java 与 Python 各维护不同 version 字符串
  • 回滚策略但不失效 semantic cache
推荐
  • PR 附变更说明与 A/B 计划
  • 发布前后对比 cost dashboard 与错误率
  • rollback runbook 含缓存与 circuit 重置步骤
CorpAssist · NS3 合规 · 禁用 L3 跨厂商 合规要求关闭 openai-fallback；PR 修改 degrade.yaml 删除 L3，compliance_route.block_cross_border: true；发布后 audit 中 vendor=dashscope 占比应 100%。
先想 10 秒：policy version 为什么要进缓存键？

成本章语义缓存若不含 route_version，回滚后仍返回新策略下的旧答案。version 是治理与缓存的契约边界。

日志

每条 LLM 路由决策写入结构化 audit log，支撑 who called what 查询、账单分摊与事故复盘。NS3 S5 要求：租户、活动、模型、是否降级、是否缓存命中、估算 cost 均可筛。

Spring Audit Advisor 示意

java
@Component
public class RouteAuditAdvisor implements CallAdvisor {

  @Override
  public ChatClientResponse adviseCall(
      ChatClientRequest request, CallAdvisorChain chain) {
    long start = System.currentTimeMillis();
    ChatClientResponse response = chain.nextCall(request);
    auditLogger.info(RouteAuditRecord.builder()
        .requestId(request.context("request_id"))
        .routeVersion(policyLoader.currentVersion())
        .modelId(response.metadata("model"))
        .costUsd(response.metadata("cost_usd"))
        .latencyMs(System.currentTimeMillis() - start)
        .build());
    return response;
  }
}

常见误区
  • 只记 model 不记 version — 无法归因策略变更导致的质量波动。
  • 日志含完整 prompt — PII 与附件内容应哈希或脱敏，合规路由需 redact。
  • Python 批处理不写 audit — 双栈 spend 对不齐，cost dashboard 失真。
自检：compliance route 应在哪一层拦截跨境调用？

在 Router 选 model 之前：读取 compliance_route.allowed_vendors，非法则直接 L4 或拒绝，并写 compliance_blocked=true。不应依赖 LLM 厂商侧事后审计。

本章结论

治理 = Git 版本策略 + 审批发布 + 结构化 audit，支撑 NS3 合规与成本闭环。

  • 策略版本：route gitops + policy version；双栈 config sync。
  • 变更：change approval 门禁；回滚失效缓存与 circuit。
  • 审计：audit log 八字段起；who called what 可查询。
  • 合规：compliance route 在 Router 层拦截非法 vendor。

判断练习

  1. 运维直接在生产改 ConfigMap 把 FAQ 改回 turbo，未走 Git。请列出至少 2 个风险。

    参考答案

    无 change approval 记录；双栈 version 漂移；audit 中 route_version 与 Git 不一致；回滚无 tag；语义缓存可能仍按旧 version 命中。

  2. 合规要求禁止 prompt 原文入库。audit 应如何记录用户问题？

    参考答案

    存 prompt_hash、长度、intent 槽位；附件存 content_hash；必要时可逆加密单独权限域。compliance route 配置 redact 规则。
