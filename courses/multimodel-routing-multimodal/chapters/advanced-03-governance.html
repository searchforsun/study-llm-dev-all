为什么要学本章

路由策略变更频繁且直接影响成本和质量。route governance通过版本管理、审计日志和审批流程确保策略变更可追溯、可回滚。无治理的路由策略会导致成本失控、模型调用黑盒、故障定位困难。

  • 实现路由审计日志
  • 设计策略变更审批流程
  • 实现路由策略的一键回滚
本章先记住3件事
  • 策略即代码：路由规则用YAML管理，走GitOps流水线。
  • 每次调用都审计：谁→什么意图→用了哪个模型→花了多少钱。
  • 变更必须审批：路由策略变更需要PR+CI评测+人工审批。

策略版本管理与GitOps

路由策略用YAML文件管理，存储在Git仓库中。Policy version每次变更走GitOps流水线。

GitOps策略管理流水线
flowchart LR
    Dev[开发修改routing-rules.yaml] --> PR[提交PR]
    PR --> CI[CI运行Golden Set评测]
    CI --> Review[人工审批]
    Review --> Merge[合并到main]
    Merge --> Deploy[自动部署]
    Deploy --> Audit[记录变更日志]

    Merge -.-> Rollback[一键回滚到上一版本]

策略变更五步：修改→PR→CI→审批→部署。

Python：策略加载器

python
# policy_loader.py — 策略加载与回滚
import yaml, os
from typing import Optional

class PolicyManager:
    def __init__(self, policy_dir: str = "./policies"):
        self.policy_dir = policy_dir
        self.current_version: Optional[str] = None
        self.history = []

    def load(self, version: str) -> dict:
        path = os.path.join(self.policy_dir, f"routing-rules.{version}.yaml")
        with open(path) as f:
            policy = yaml.safe_load(f)
        self.current_version = version
        self.history.append({"version": version, "action": "load"})
        return policy

    def rollback(self) -> dict:
        """回滚到上一版本"""
        if len(self.history) < 2:
            raise ValueError("没有可回滚的版本")
        prev_version = self.history[-2]["version"]
        return self.load(prev_version)

    def validate(self, policy: dict) -> bool:
        """校验策略合法性"""
        if "version" not in policy:
            return False
        if "routing" not in policy:
            return False
        if "rules" not in policy["routing"]:
            return False
        for rule in policy["routing"]["rules"]:
            if "intent" not in rule or "model" not in rule:
                return False
        return True

审计日志与合规审批

Audit log记录每次模型调用的完整信息。change approval确保策略变更有人审批。

审计事件记录字段用途
模型调用user_id, intent, model, cost, latency成本归因、性能分析
策略变更operator, from_version, to_version, reason变更追溯
降级事件原模型, 降级后模型, 原因(429/timeout/error)容灾分析
预算超限预算值, 实际值, 超限百分比成本告警

Python：路由审计日志

python
# route_audit.py — 路由审计
import json, time, uuid

class RouteAuditLogger:
    def __init__(self, log_path: str = "route_audit.log"):
        self.log_path = log_path

    def log_call(self, user_id: str, intent: str, model: str,
                 cost: float, latency_ms: int, degraded: bool = False):
        entry = {
            "type": "call",
            "id": uuid.uuid4().hex[:12],
            "timestamp": time.time(),
            "user_id": user_id,
            "intent": intent,
            "model": model,
            "cost": cost,
            "latency_ms": latency_ms,
            "degraded": degraded,
        }
        self._write(entry)

    def log_change(self, operator: str, from_v: str, to_v: str, reason: str):
        entry = {
            "type": "change",
            "timestamp": time.time(),
            "operator": operator,
            "from_version": from_v,
            "to_version": to_v,
            "reason": reason,
        }
        self._write(entry)

    def query(self, model: str = None, user_id: str = None,
              limit: int = 100) -> list:
        results = []
        with open(self.log_path) as f:
            for line in f:
                entry = json.loads(line)
                if model and entry.get("model") != model:
                    continue
                if user_id and entry.get("user_id") != user_id:
                    continue
                results.append(entry)
        return results[-limit:]

    def _write(self, entry: dict):
        with open(self.log_path, "a") as f:
            f.write(json.dumps(entry) + "\n")
不推荐
  • 路由规则硬编码在代码里
  • 变更走「改完直接部署」
  • 无调用审计，成本说不清楚
  • 无法回滚有问题的策略
推荐
  • YAML配置+GitOps版本管理
  • PR+CI+审批三阶变更流程
  • 完整审计日志+成本归因
  • 支持一键回滚

常见问题（FAQ）

审计日志占用存储怎么办？
按时间分区存储，保留最近90天可在线查询，更早的数据归档到廉价存储。每条审计记录约200字节，日均5万次调用约10MB/天，90天约900MB，完全可控。
策略回滚有风险吗？
有。回滚可能引入旧bug。推荐：1) 回滚前做CI评测；2) 灰度发布新版策略（10%→50%→100%）；3) 回滚后监控指标（失败率/延迟/成本）确认正常。
多人同时修改策略怎么处理？
Git天然处理冲突。两个PR同时修改routing-rules.yaml，先合并的生效，后合并的需要解决冲突。冲突通常发生在同一intent的不同规则上。推荐分工：不同团队负责不同intent的规则。

本章结论

  • 策略即代码：YAML管理路由规则，GitOps流水线确保变更可追溯。
  • 变更加审批：PR→CI→人工审批三阶流程，防止随意变更。
  • 审计全覆盖：每次调用+每次变更+每次降级都记录日志。
  • 一键回滚：策略出问题时，一条命令回滚到上一版本。
  • 成本归因：审计日志按用户/意图/模型归因成本。
