指标

routing eval 的核心是把「路由决策好不好」拆成可量化、可对比的 router kpi。CorpAssist S5 至少跟踪质量、成本、可靠性三类。

KPI 类别代表指标S5 附件问答示例采集点
quality metric grounded 率、人工 rubric 均分 回答是否引用 OCR 片段 离线标注集 + 抽样人工
cost metric 每请求 Token、每成功问答成本 vision_qa 是否滥用 VL Gateway + Router 审计日志
可靠性 Fallback 触发率、P95 延迟 429 后降级成功率 Micrometer / 日志
offline benchmark

固定 200+ 条标注样本，每次改 routing.yaml 全量重跑，作为合并前门禁。

route regression

对比基线策略：质量下降 >2% 或成本上升 >15% 则阻断发布。

router kpi 看板

按 intent 分桶展示质量/成本，支撑 NS3 成本路由决策。

离线评测脚本（Python 轨）

python
import json
from pathlib import Path
from route_service import RouteService

def run_offline_benchmark(
    service: RouteService,
    dataset: Path,
    baseline: dict | None = None,
) -> dict:
    rows = [json.loads(line) for line in dataset.read_text().splitlines()]
    total_cost = 0.0
    grounded_hits = 0

    for row in rows:
        result = service.complete(row["intent"], row["messages"])
        total_cost += result.get("cost_usd", 0)
        if row.get("expected_citation") in result.get("content", ""):
            grounded_hits += 1

    metrics = {
        "samples": len(rows),
        "grounded_rate": grounded_hits / len(rows),
        "avg_cost_usd": total_cost / len(rows),
        "fallback_rate": service.stats.fallback_rate,
    }
    if baseline:
        metrics["regression"] = {
            "grounded_delta": metrics["grounded_rate"] - baseline["grounded_rate"],
            "cost_delta_pct": (metrics["avg_cost_usd"] - baseline["avg_cost_usd"])
                              / baseline["avg_cost_usd"],
        }
    return metrics

router-kpi 配置示例

yaml
router_kpi:
  version: "2026.05"
  gates:
    offline:
      min_samples: 200
      max_grounded_regression: -0.02   # 质量不得降 2%
      max_cost_increase_pct: 0.15      # 成本不得涨 15%
    online:
      p95_latency_ms: 3000
      fallback_rate_max: 0.08
  intents:
    vision_qa:
      quality: grounded_rate
      cost: avg_tokens_per_request
    faq:
      quality: exact_match_rate
      cost: avg_cost_usd

先想 10 秒：Fallback 触发率升高一定是坏事吗？

不一定。若主模型频繁 429，Fallback 保证可用性；需结合质量是否下降与成本是否飙升看。单独 KPI 可能误判，应纳入 router-kpi 看板联动分析。

A/B

离线通过后，用 ab test route 在小流量验证真实用户行为。实验单位通常是 user_id 或 session_id，保证同一用户始终落在 A 或 B 组。

要素推荐做法CorpAssist S5 注意点
分流键hash(user_id) % 100与 Gateway 鉴权后的 stable id 一致
周期至少 7 天（含周末流量）覆盖附件上传高峰
成功标准win rate + 成本不劣化B 组质量 ≥ A 且成本 ≤ A×0.9
回滚feature flag 切回 control保留 audit 字段对比

Spring 侧实验分流（示意）

java
@Component
public class RouteExperimentAssigner {

  public RouteVariant assign(String userId, ExperimentConfig exp) {
    int bucket = Math.floorMod(userId.hashCode(), 100);
    if (bucket < exp.trafficPercentB()) {
      return RouteVariant.TREATMENT;  // 新 routing.yaml 版本
    }
    return RouteVariant.CONTROL;
  }
}

@Service
public class EvaluatingRouter {
  public ChatResponse route(RouteRequest req) {
    RouteVariant variant = assigner.assign(req.userId(), experimentConfig);
    RoutePolicy policy = policyLoader.load(variant);
    ChatResponse resp = innerRouter.route(req, policy);
    audit.log(Map.of(
        "experiment", experimentConfig.id(),
        "variant", variant.name(),
        "model", resp.getMetadata().get("model")
    ));
    return resp;
  }
}

实验结束：若 treatment win rate ≥55% 且成本达标，promote v2 为全量 unified policy。

win-rate 计算（评测报告片段）

python
def compute_win_rate(paired_scores: list[tuple[float, float]]) -> float:
    """paired_scores: (control_quality, treatment_quality) per session"""
    wins = sum(1 for a, b in paired_scores if b > a)
    ties = sum(1 for a, b in paired_scores if b == a)
    return (wins + 0.5 * ties) / len(paired_scores)

# 示例：200 对会话，treatment 胜 118、平 12、负 70
# win_rate = (118 + 6) / 200 = 0.62
不推荐
  • A/B 只跑 1 天就大范围全量
  • 对照组与实验组用不同 DashScope 账号，成本不可比
  • 未设 win-rate 阈值，凭主观「感觉更快」上线
推荐

  • 审计日志带 experiment_id 与 variant
  • 离线 route regression 通过后再开 A/B

自检：离线 benchmark 已通过，是否可以直接 100% 切新策略？

不建议。离线集无法覆盖全部线上长尾（新附件格式、峰值 429）。应先用 ab test route 小流量验证，再用 win-rate 与 cost-metric 决策全量。

评测数据集 200 条从哪来？
脱敏历史 S5 问答 + 人工构造 vision_qa 边界样例（扫描 PDF、纯图、无 OCR 文本）。每季度增补，防止过拟合旧路由。
与 advanced NS3 成本路由章如何衔接？
本章建立 KPI 与 A/B 方法论；advanced 章在 KPI 达标前提下做 small-model-first、semantic cache 等策略，仍依赖同一 offline benchmark 门禁。

动手练习

约 25 分钟 · 勾选 eval-checklist，完成一次离线 benchmark 报告骨架。

操作步骤

  1. 打开 demos/basics-04-eval-routing-lab/eval-checklist.md，逐项勾选。
  2. 准备或引用 200 条离线样本 JSONL（含 intent、messages、expected_citation）。
  3. 用上一章 Python RouteService 跑 benchmark，记录 grounded_rate 与 avg_cost。
  4. 模拟 route regression：将某 intent 改错模型，确认 CI 门禁失败。
  5. 设计一张 7 天 A/B 实验表（分流比例、win-rate 阈值、回滚条件），完成右侧测验。

判断练习

  1. 离线 benchmark 显示成本降 30%、grounded_rate 降 0.5%，门禁阈值是质量不得降 2%。能否合并 PR？

    参考答案

    可以合并进仓库，但不应直接全量上线。0.5% 在门禁内，可进入 A/B；若线上 vision_qa 长尾更敏感，仍可能在 A/B 中暴露问题，需盯 router-kpi 看板。

  2. A/B 第 3 天 treatment 组 win-rate 已达 60%，能否提前全量？

    参考答案

    不建议。周期未覆盖完整周流量（尤其 S5 附件高峰），且样本不足。应跑满约定 7 天或达到最小样本量后再 promote。
