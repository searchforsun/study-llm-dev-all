为什么要学本章

CorpAssist NS3 · S5 内容创作 在营销高峰或 DashScope 区域维护时，单一 qwen-vl-max 端点会 503/429。若路由层没有 model degrade 链，用户只会看到空白页。本章把健康检查 → 同厂商备用 → 跨厂商 → 静态话术落成可配置策略，Spring Router 与 Python route-service 共用 degrade 配置，并与上一章 cost aware route 的 spend cap 联动。

学完你能
  • 设计四级 unavailable fallback 链，并说明 VL 与纯文本降级差异
  • 配置 429 handling、region failover 与 circuit route 熔断
  • 为 S5 附件问答编写用户可见的 degrade text，并完成 Demo 验收
本章先记住 3 件事
  • 降级是路由决策：失败切换下一跳，不是 Controller 里 catch 后随便 return。
  • 429 要退避：重试带 jitter，避免雪崩；仍失败则走链下一级。
  • 静态话术也是产品：degrade text 需可配置、可审计、可 A/B。

模型不可用

当主模型返回 5xx、超时或连续失败，model degrade 按预定义顺序尝试备用模型，最后落到静态响应。CorpAssist S5 典型链：DashScope 主区 VL → 同区 plus 文本摘要 → OpenAI 兼容备用 → 品牌方预设 FAQ 话术。

级别目标S5 示例质量预期
L1 主模型qwen-vl-max海报 + 附件联合理解最高
L2 同厂商备用qwen-vl-plus / qwen-plusVL 不可用，文本描述附件中高
L3 跨厂商openai-fallback gpt-4o-miniDashScope 全区域故障中（需合规审批）
L4 静态degrade_text 模板「服务繁忙，已为您保留草稿」保底可用
health check

后台探活主端点；连续 N 次失败则 circuit route 打开，请求直接跳过 L1。

429 handling

指数退避 2 次；仍 429 则视为不可用，进入 L2，并打 rate_limited=true 审计。

multi vendor

L3 需密钥隔离与数据出境策略；NS3 合规路由在 governance 章有字段清单。

降级链配置（双栈共用）

yaml
degrade:
  version: "2026.05.2"
  chain:
    vision_qa: [qwen-vl-max, qwen-vl-plus, qwen-plus, static]
    content_draft: [qwen-plus, qwen-turbo, openai-fallback, static]
  on_429:
    max_retries: 2
    backoff_ms: [200, 800]
    then: next_in_chain
  circuit_route:
    failure_threshold: 5
    open_duration_sec: 60
  static_fallback:
    vision_qa: "附件分析暂不可用，请稍后重试或联系运营。"
    content_draft: "文案生成排队中，您的草稿已保存。"

Spring：Fallback 链执行

java
public ChatResponse executeWithDegrade(RouteRequest req) {
  List<String> chain = degradePolicy.chainFor(req.intent());
  for (int i = 0; i < chain.size(); i++) {
    String hop = chain.get(i);
    if ("static".equals(hop)) {
      return staticDegrade.reply(req, degradePolicy.textFor(req.intent()));
    }
    try {
      return invokeModel(hop, req);
    } catch (RateLimitException ex) {
      rateLimitHandler.backoffOrSkip(ex, i, chain);
    } catch (ModelUnavailableException ex) {
      audit.logDegrade(req, hop, ex.getReason());
    }
  }
  throw new RouteExhaustedException(req.intent());
}

每一跳失败都写 audit；L4 仍返回 200 + 用户可见文案，避免裸 500。

不推荐
  • catch Exception 后 silent retry 同一模型 10 次
  • VL 失败直接 500，无静态话术
  • 跨厂商 L3 无合规字段，审计缺失
推荐
  • chain 配置化，与 路由策略 intent 对齐
  • 429 handling 有上限与 jitter
  • 响应头 X-Route-Degraded: L2 供前端展示提示
CorpAssist · S5 大促 · VL 限流 DashScope 返回 429；Router 退避 2 次后切 qwen-plus 仅根据附件文件名与 OCR 摘要作答，响应带「当前为简化模式」；cost 章的 spend cap 仍生效。
先想 10 秒：为何 L4 静态话术仍返回 200？

S5 用户已上传素材并等待生成；HTTP 500 会导致前端丢草稿。L4 返回 200 + degraded=true 保留会话态，运营可在 governance 审计里统计降级率。

区域

region failover 在端点级不可用时切换地域或读副本 API。DashScope 多 region 部署时，Router 维护 region 权重与健康度；Python 批处理与 Spring 在线 API 读同一份 region 表，避免「Java 走华东、Python 仍打华北坏节点」。

策略触发行为注意
主动 failoverregion 探活失败权重归零，流量切 backup region需同步 model 名映射
被动 failover单次 503 + region 头重试指定 backup endpoint仅一次，防延迟放大
sticky region多轮会话同 session 尽量同 region缓存与附件 URL 一致
global static全 region 不可用直接进入 L4 degrade text告警 P1

Python route-service：区域感知路由

python
async def route_with_region(body: RouteRequest) -> RouteDecision:
    primary = region_pool.pick(body.tenant_id, sticky=body.session_id)
    if circuit.is_open(primary.endpoint):
        backup = region_pool.failover(primary.region)
        return RouteDecision(
            endpoint=backup.endpoint,
            model_id=body.model,
            region_failover=True,
            degrade_level="region",
        )
    return RouteDecision(endpoint=primary.endpoint, model_id=body.model)

常见误区
  • 区域切换不更新 model 映射 — 备用 region 可能无 vl-max，应降级到 vl-plus 并记录。
  • 忽略附件 URL 地域 — OSS 链接与推理 region 不一致会导致超时，需 BFF 预签名同区。
  • failover 无上限 — 跨 region 重试应 ≤1 次，否则 P99 延迟爆炸。
自检：S5 全区域故障时 cost 章的 spend cap 还有意义吗？

有。L4 静态响应零 LLM 成本；若错误重试 L3 跨厂商，spend cap 仍可阻止账单失控。降级与成本策略应同版本发布。

本章结论

可用性 = 四级 degrade 链 + region failover + 429 退避，双栈共用 degrade.yaml。

  • 模型不可用：L1→L4 链；health check 与 circuit route 跳过坏端点。
  • 429：429 handling 有限重试后进链下一跳。
  • 区域：region failover 与 sticky session；全挂走 L4 degrade text。
  • 多厂商：multi vendor L3 需合规与 audit 字段（见 governance 章）。

判断练习

  1. VL 主模型 429 时，工程师在 Controller 里 Thread.sleep(5000) 后重试同一模型 20 次。请指出违反本章哪条原则及推荐改法。

    参考答案

    违反 429 handling 上限与 degrade 链原则。应在 Router 层有限退避（如 2 次）后切 L2，并打审计；sleep 不应散落在 Controller。

  2. 华东 region 探活失败，但 failover 后附件仍从华东 OSS 拉取导致超时。根因与改法？

    参考答案

    region failover 只切了推理 endpoint，未切附件存储区。BFF 应在 failover 时重新签发同区预签名 URL，或 sticky 会话绑定 region 三元组（推理+存储+缓存）。
