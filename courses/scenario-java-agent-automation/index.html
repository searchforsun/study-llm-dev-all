Resilience4j Circuit Breaker

熔断器是防止级联故障的关键机制。当 Agent 调用的下游服务（LLM、工具 API）出现高失败率时，熔断器快速失败而不继续消耗资源。

stateDiagram-v2
    [*] --> CLOSED: 初始化
    CLOSED --> OPEN: 失败率 > 50%
最近 20 次调用中
    OPEN --> HALF_OPEN: 等待 30 秒后
    HALF_OPEN --> CLOSED: 3 次探测全部成功
    HALF_OPEN --> OPEN: 任一次探测失败
    OPEN --> CLOSED: 手动重置（运维操作）
java
// Resilience4j 熔断器配置
@Configuration
public class CircuitBreakerConfig {

    @Bean
    public CircuitBreakerRegistry agentCircuitBreakerRegistry() {
        // LLM 调用熔断器
        CircuitBreakerConfig llmConfig = CircuitBreakerConfig.custom()
            .failureRateThreshold(40)            // 40% 失败率触发
            .slowCallRateThreshold(50)           // 50% 慢调用触发
            .slowCallDurationThreshold(Duration.ofSeconds(10))  // >10s 算慢
            .slidingWindowSize(20)               // 统计窗口 20 次
            .minimumNumberOfCalls(5)             // 最少 5 次才计算
            .waitDurationInOpenState(Duration.ofSeconds(30))
            .permittedNumberOfCallsInHalfOpenState(3)
            .build();

        // 工具调用熔断器（更严格）
        CircuitBreakerConfig toolConfig = CircuitBreakerConfig.custom()
            .failureRateThreshold(30)
            .slidingWindowSize(10)
            .waitDurationInOpenState(Duration.ofSeconds(15))
            .permittedNumberOfCallsInHalfOpenState(2)
            .build();

        return CircuitBreakerRegistry.of(llmConfig, toolConfig);
    }
}

// 熔断器装饰器
@Component
public class CircuitBreakerDecorator {

    private final CircuitBreaker llmBreaker;
    private final CircuitBreaker toolBreaker;

    public CircuitBreakerDecorator(CircuitBreakerRegistry registry) {
        this.llmBreaker = registry.circuitBreaker("llm-breaker");
        this.toolBreaker = registry.circuitBreaker("tool-breaker");
    }

    public String callLlm(Supplier<String> llmCall) {
        return llmBreaker.executeSupplier(llmCall);
    }

    public String callTool(Supplier<String> toolCall) {
        return toolBreaker.executeSupplier(toolCall);
    }

    // 熔断事件监听
    @EventListener
    public void onStateChange(CircuitBreaker.StateTransition event) {
        log.warn("CircuitBreaker state change: {} -> {}",
            event.getOldState(), event.getNewState());
        // 通知运维
        if (event.getNewState() == CircuitBreaker.State.OPEN) {
            alertService.sendAlert("熔断器打开: " + event.getCircuitBreakerName());
        }
    }
}

模型 Fallback 链

当主模型不可用时（熔断、超时、限流），Agent 需要降级到备用模型或本地规则引擎。

java
// 模型 Fallback 链
@Component
public class ModelFallbackChain {

    private final List<ModelProvider> providers;

    public ModelFallbackChain() {
        this.providers = List.of(
            new DashScopeProvider("qwen-max", 100),     // 主模型，权重 100
            new DashScopeProvider("qwen-plus", 80),      // 备用模型 1
            new DashScopeProvider("qwen-turbo", 60),     // 备用模型 2
            new LocalRuleEngine(),                        // 本地规则引擎
            new HumanFallback()                            // 人工兜底
        );
    }

    public String executeWithFallback(ModelRequest request) {
        for (ModelProvider provider : providers) {
            if (!provider.isAvailable()) {
                log.info("Provider {} is not available, skipping", provider.getName());
                continue;
            }
            try {
                long start = System.currentTimeMillis();
                String result = provider.call(request);
                long duration = System.currentTimeMillis() - start;
                log.info("Provider {} succeeded in {}ms", provider.getName(), duration);
                return result;
            } catch (Exception e) {
                log.warn("Provider {} failed: {}", provider.getName(), e.getMessage());
                // 继续下一个 Provider
            }
        }
        // 所有 Provider 都失败
        throw new AllProvidersFailedException("所有模型和兜底方案都失败");
    }
}

// 本地规则引擎（无模型 Fallback）
@Component
public class LocalRuleEngine implements ModelProvider {

    @Override
    public String getName() { return "local-rule-engine"; }

    @Override
    public boolean isAvailable() { return true; }  // 始终可用

    @Override
    public String call(ModelRequest request) {
        // 基于模板和规则生成响应（不需要 LLM）
        if (request.getIntent().contains("query_ticket")) {
            return "{\"status\": \"模拟工单数据（规则引擎模式）\"}";
        }
        return "系统当前使用降级模式，部分功能不可用。请稍后重试。";
    }
}

    }
}

人工兜底触发与审计追溯

当所有自动方案都失败时，需要平滑地切换到人工处理。每次切换都需要完整记录，以便事后分析。

java
// 人工兜底管理
@Component
public class HumanFallbackManager {

    private final TicketService ticketService;
    private final AuditLogger auditLogger;

    public EscalationResult escalateToHuman(EscalationRequest request) {
        // 1. 记录兜底原因
        String reason = String.format(
            "【人工兜底】会话: %s, 用户: %s, 原因: %s, 失败链路: %s",
            request.getSessionId(), request.getUserId(),
            request.getReason(), String.join(" -> ", request.getFailedProviders())
        );
        auditLogger.log(reason);

        // 2. 创建人工工单
        Ticket ticket = ticketService.create(
            request.getSessionId(),
            "Agent 自动兜底: " + request.getReason(),
            "admin-team",
            request.getPriority()
        );

        // 3. 通知用户
        notifyUser(request.getUserId(),
            "您的请求已转交人工处理，工单编号: " + ticket.getId());

        return new EscalationResult(ticket.getId(), "ESCALATED");
    }

    // 审计查询
    public List<EscalationRecord> queryEscalations(String timeRange) {
        return auditLogger.queryByEventType("ESCALATION", timeRange);
    }

    // 可靠性统计
    public ReliabilityReport generateReport() {
        long total = auditLogger.countByEventType("AGENT_REQUEST");
        long escalated = auditLogger.countByEventType("ESCALATION");
        double autoRate = (double) (total - escalated) / total * 100;
        return new ReliabilityReport(total, escalated, autoRate);
    }
}

CorpAssist 场景

某日下午 2:30，DashScope API 出现区域性故障，Agent 调用 qwen-max 连续超时。熔断器在 20 次调用中检测到 12 次超时（60%）后触发 OPEN。Agent 切换到 qwen-plus 备用模型，但备用模型也受到同一区域故障影响（50% 超时）。Agent 再切换到 qwen-turbo（不同区域），成功处理了 80% 的请求。剩余的 20% 请求因为需要高质量模型（如生成复杂周报），降级到本地规则引擎生成模板型响应 + 标注"使用降级模式"。如果本地规则引擎也无法处理（如用户要求解释合同条款），Agent 创建人工工单并通知用户。整个过程在 30 秒内完成决策，用户感知到的只是"部分功能暂时降级"。事后审计报告显示：该次故障持续 23 分钟，92% 的请求被自动降级处理，仅 8% 转人工。相比无 Fallback 链（100% 不可用），可用性从 0% 提升到 92%。

面试可能问

熔断器 OPEN 后，积压的请求如何处理？

三种策略：1) 快速失败——直接返回"服务暂时不可用"，让客户端自行重试。简单、资源消耗低，但用户体验差。2) 排队缓冲——将请求放入缓冲队列，等待熔断器恢复后重试。适合非实时场景（如周报生成），但需要队列有容量限制和 TTL。3) 降级响应——返回缓存结果或模板响应，如"系统当前负载较高，您的请求已加入队列"。CorpAssist 使用混合策略：实时对话用快速失败 + 友好提示，非实时任务排队缓冲。熔断器 OPEN 时，排队队列的 TTL 设为 5 分钟，超过 5 分钟的请求自动转人工工单。

Fallback 链中如何避免"降级风暴"（所有请求都触发降级）？

降级风暴是指主模型故障后所有请求涌向备用模型，导致备用模型也被压垮。避免策略：1) 限流降级——每个 Provider 有独立的限流配置（如 qwen-plus 最多接受 60% 的请求），超过部分跳过此 Provider 继续下一个；2) 负载检测——每个 Provider 在调用前检查当前负载（最近 1 分钟的调用量和失败率），如果负载超过阈值则跳过；3) 降级比例控制——控制降级到低质量模型的比例，如最多 20% 的请求使用本地规则引擎；4) 熔断器共享——主模型和备用模型共用同一个熔断器吗？不是，每个 Provider 有独立的熔断器，避免一个熔断导致全部降级。

人工兜底的 SLA 目标是多少？如何保证？

CorpAssist 人工兜底 SLA：P1（紧急）30 分钟内响应，P2（重要）2 小时内响应，P3（一般）8 小时内响应。保证措施：1) 值班制度——工作日 9:00-21:00 安排值班人员响应 P1 告警；2) 多级升级——P1 工单 15 分钟未响应升级到技术主管，30 分钟未响应升级到部门总监；3) 离线通知——P1 工单通过电话 + 短信 + 钉钉三通道通知；4) 自动关闭——P3 工单 24 小时未处理自动关闭并通知用户"请重新提交"。数据统计：2026 年 Q1 平均响应时间 P1=12 分钟，P2=45 分钟，P3=3.5 小时。

场景：熔断器在 CLOSED 和 OPEN 之间频繁抖动，怎么办？

这是"熔断抖动"问题。原因通常是统计窗口太小导致偶然的失败波动触发熔断。解决方案：1) 增大 slidingWindowSize——从 20 增加到 50，降低单次失败的影响；2) 增大 minimumNumberOfCalls——从 5 增加到 10，防止样本过少时误判；3) 增加失败率阈值——从 40% 调整到 50%；4) 启用慢调用检测——设置 slowCallDurationThreshold=10s，如果工具调用超过 10 秒算慢调用，慢调用比例高于 50% 也触发熔断，而不是仅看失败率。CorpAssist 推荐参数：windowSize=50, minCalls=10, failureRate=50, waitDuration=60s。

场景：规则引擎生成的模板响应质量太低，如何避免用户感知到严重降级？

优化规则引擎响应质量：1) 上下文模板——规则引擎根据请求关键词选择最相关的模板（如检测到"工单"使用工单模板），而不是统一的"降级响应"；2) 缓存结果——规则引擎优先返回缓存中同类型请求的历史成功响应，如 "根据上次的报告模板生成"；3) 降级标记——所有规则引擎响应附带 banner "当前使用智能降级模式，数据可能不是最新的"，让用户知道而非猜疑；4) 自动恢复——规则引擎每 30 秒测试一次主模型可用性，可用后自动切换回主模型并刷新页面。CorpAssist 实现中，用户几乎不会注意到降级——只有打开调试面板才能看到 "Provider: local-rule-engine" 的标记。

面试可能问

你这个 Agent 系统和 Dify/Copilot 这类现成平台比有什么优势？

区分"自研 vs 平台"的关键面试题。回答框架：自研的优势是深度定制和可控——1) 我们可以深度集成企业现有的 Java 技术栈（Spring Boot、Nacos、Kafka），而不是被平台限制；2) 审批流程和业务系统紧密耦合，现成平台的审批模块无法满足企业复杂的审批链（链式审批、会签、超时逐级升级）；3) 数据安全和合规——自研系统数据不出企业内网，通过了等保三级认证；4) 成本——每月 500 人企业的 API 调用成本约 8000 元，低于企业版 SaaS 的 3 万元/月。劣势是维护成本高。我的选择策略：核心场景自研（可深度定制），边缘场景接入平台 API（成本低）。

你的 Agent 项目部署在什么环境？CPU/GPU 配置？

部署架构：2 台 8C16G ECS 服务器作为 Agent 服务节点 + 1 台 4C8G 作为审批网关 + 1 台 4C8G 作为规则引擎。LLM 调用全部走阿里云 DashScope API（无需 GPU），本地不需要部署模型。数据库：RDS MySQL（审计数据）+ Redis 集群（检查点和缓存）+ Milvus（向量记忆）。所有服务在 Kubernetes 中部署，HPA 自动扩缩（根据 QPS 和 CPU 使用率）。整体月成本约 1.2 万元（含 API 调用费），相比雇佣 3 个 人工处理（月薪约 5 万元/人）节省了 13.8 万元/月。

如果让你重新做这个项目，会有什么不同的选择？

三个改进点：1) 早期就用 Graph 而非 ReAct——我们在 Phase 1 使用 ReAct Agent，但发现复杂的审批流程需要显式状态管理，Phase 2 才迁移到 Graph，增加了 2 周迁移成本。如果重来，对于包含审批节点的流程直接用 Graph。2) 更早引入熔断器——前 2 个月没有熔断保护，有一次 DashScope 故障导致 Agent 全部阻塞，平均响应时间飙升到 30 秒。后来加了 Resilience4j 熔断器，故障时自动降级到规则引擎。3) 前期就定义验收清单——前 3 周没有明确的验收标准，导致"开发完成"后和业务方对需求花了 1 周。重来的话先和业务方确定 8 项验收清单再开始编码。这些教训说明：先做对架构选型，再加好容错机制，最后用验收清单兜底。

Q1: Agent vs 普通 LLM 应用的区别

考点：理解 Agent 的核心价值——工具调用和自主决策，而非对话。

普通 LLM 应用

一问一答。用户输入→LLM→输出。LLM 只知道自己的训练数据，不知道外部信息。如"查张三的工单状态"——LLM 会编造一个答案（幻觉），因为它无法访问工单系统。

Agent 应用

多步推理+工具调用。用户输入→LLM 思考→调用工具→观察结果→继续推理→输出。如"查张三的工单状态"——Agent 调用 queryTicket("张三")→获得真实数据→输出。

// 用户登录时自动加载最近 5 条相关偏好。命中率约 70%。

// 后端服务的变化。"

面试可能问

你的 Agent 系统 QPS 能到多少？怎么测试的？

基准测试结果：单节点 8C16G 部署，QPS = 50（不考虑 LLM 延迟，因为 LLM 调用走 API 不占用本地 CPU）。如果考虑 LLM 延迟（平均 2 秒），实际吞吐 = 50/2 = 25 QPS。测试方法：使用 JMeter 模拟 100 个并发用户，每个用户执行"查工单"场景（1 次工具调用）。观察指标：平均响应时间、P99 响应时间、错误率、CPU/内存使用率。极限测试结果：200 并发时 P99 从 3.2s 升至 8.7s，错误率开始上升。扩容到 4 节点后恢复。瓶颈分析：主要瓶颈在 Redis 检查点读写（约 5ms/次）和 MCP 工具调用（约 50ms/次）。优化后将检查点改为批量写入（每 3 步写一次而非每步），MCP 调用增加连接池。

Agent 的 System Prompt 你是怎么设计的？

三层 Prompt 设计：1) 基础 Prompt——固定不变的角色定义和行为边界，如"你是 CorpAssist 企业办公助手，有以下工具可用..."。代码中硬编码，通过 @Value 或配置中心管理。2) 场景 Prompt——根据当前场景动态注入，如"当前是周报生成场景，需要先收集用户工时数据..."。由 Planner 在规划阶段生成。3) 工具 Prompt——工具调用时动态生成，包含工具的描述和参数说明。由 @Tool 注解的 description 自动生成。设计原则：基础 Prompt 占 60%（稳定），场景 Prompt 占 25%（动态），工具 Prompt 占 15%（自动生成）。总长度控制在 2000 Tokens 以内。使用 Prompt Template 模式管理，key 存储在 Nacos 配置中心，value 存储在 Git 仓库中版本管理——每次 Prompt 修改都有记录，可以回滚。

你的 Agent 系统如何做单元测试？

四层测试策略：1) 节点测试——每个 @Tool 方法和每个 Graph 节点函数独立单元测试。Mock 外部依赖，测试正常路径和异常路径。2) Agent 测试——Mock LLM 返回（固定返回特定的 tool_calls 和 final answer），测试 Agent 是否按预期调用工具和处理结果。3) 集成测试——使用 Testcontainers 启动 Redis + Kafka + PostgreSQL，测试完整工作流。LLM 用 WireMock 模拟。4) E2E 测试——连接真实的 LLM API，在测试环境（独立的 API Key）中执行端到端测试。难度最大的是 Agent 测试，因为 LLM 返回的不确定性。我们的办法：用录制回放工具（如 VCR）录制 LLM 的真实响应，在 CI 中回放。回归测试覆盖 200+ 个测试用例，CI 流水线每次提交运行测试，平均耗时 8 分钟。

场景：面试官问"你提到了 MCP，能说说它和 OpenAPI/Swagger 的区别吗？"

MCP vs OpenAPI：1) 设计目标不同——OpenAPI 是给"人"看的 API 文档（面向开发者），MCP 是给"AI"看的工具描述（面向 LLM）；2) 参数描述——OpenAPI 用 required/type/format 描述参数（适合代码生成），MCP 用 description/examples 描述参数（适合 LLM 理解用途）；3) 调用方式——OpenAPI 需要开发者写 HTTP 调用代码，MCP 的每一次调用是"LLM 自主决定"的（System Prompt 中给 LLM 描述，LLM 决定何时调用）；4) 动态性——MCP 支持工具自动发现（Agent 启动时 listTools），OpenAPI 需要开发者在代码中手动注册。如果你用过 Swagger Codegen，它生成的是开发者调用的代码；MCP 生成的是 LLM 理解的工具描述。两者的结合方式：内部我们先用 OpenAPI 定义接口，然后通过一个适配器自动生成 MCP 的工具描述。

ROI 量化模型

Agent 自动化项目的 ROI 需要可量化、可追溯。CorpAssist 使用以下模型进行 ROI 评估：

java
// CorpAssist ROI 计算器
@Component
public class RoiCalculator {

    private static final double AGENT_COST_PER_TASK = 0.02; // USD
    private static final double EMPLOYEE_HOURLY_RATE = 25.0; // USD

    /**
     * 计算单个任务的年化 ROI
     * @param taskName 任务名称
     * @param dailyFrequency 日均频次
     * @param manualMinutes 人工单次耗时（分钟）
     * @param workDays 年工作天数
     * @param confidence 自动化置信度（0-1）
     */
    public RoiResult calculate(String taskName, int dailyFrequency,
                               double manualMinutes, int workDays,
                               double confidence) {
        double manualHours = dailyFrequency * (manualMinutes / 60.0) * workDays;
        double manualCost = manualHours * EMPLOYEE_HOURLY_RATE;

        double agentTasks = dailyFrequency * workDays * confidence;
        double agentCost = agentTasks * AGENT_COST_PER_TASK;

        double savings = manualCost - agentCost;
        double roi = (savings - 0) / (agentCost + 1) * 100;

        return new RoiResult(taskName, manualCost, agentCost, savings, roi);
    }

    public record RoiResult(String taskName, double manualCost,
                            double agentCost, double savings, double roi) {}
}

如何判断一个办公任务是否适合 Agent 自动化？

三步判断法：1) 任务是否可以结构化描述——输入输出是否明确（如"查张三的工单"输入是姓名、输出是工单状态）；2) 任务是否需要实时人类判断——如果后续步骤取决于人类的主观判断（如"这个合同条款是否合理"），则不适合全自动；3) 任务失败影响是否可控——如果错误会导致经济损失或合规风险（如"发送 10 万元付款"），则必须有审批环节。CorpAssist 实践中，约 60% 的办公任务可部分自动化，30% 可全自动，10% 不适合自动化。

Agent 自动化的 ROI 如何向 CFO 汇报？

使用三层叙事框架：1) 成本节省——以 500 人企业为例，年化节省 24.3 万美元（见 ROI 计算器）；2) 效率提升——信息获取从 3 分钟到 15 秒，员工每天节省约 40 分钟；3) 非量化收益——员工满意度提升（少做重复工作）、响应速度提升（7×24 小时）、可追溯（所有操作有审计日志）。CFO 最关心的是成本节省，第一页放 ROI 数据表，第二页放对比案例，第三页放实施计划。

全自动、半自动、纯人工的划分标准是什么？

基于两个维度的矩阵：1) 错误影响——无影响(全自动)、低(需确认)、中(需审批)、高(纯人工)；2) 可回退性——可轻松回退(全自动)、可回退但有成本(需确认)、不可回退但可审计(需审批)、不可回退且影响大(纯人工)。CorpAssist 的配置文件中为每个任务标记 degreeOfAutomation 字段，从 0(纯人工)到 5(全自动)。系统根据这个字段决定是否需要调用审批网关。

ReAct 循环的 Spring AI 实现

Spring AI 的 ReAct 循环通过 ChatClient + ToolCallback 的组合实现。核心流程是：发送消息给 LLM → 检查响应中是否包含 tool_calls → 根据 tool_calls 执行对应工具 → 将工具结果作为新消息追加 → 再次发送给 LLM → 重复直到 LLM 返回不含 tool_calls 的最终回答。

// Spring AI ReAct 循环核心实现
@Service
public class ReactAgentService {

    private final ChatClient chatClient;
    private final List<ToolCallback> toolCallbacks;

    public ReactAgentService(ChatClient.Builder builder,
                             List<ToolCallback> toolCallbacks) {
        this.toolCallbacks = toolCallbacks;
        this.chatClient = builder
            .defaultSystem("""
                你是企业办公助手 CorpAssist。你有以下能力：
                - 查询员工信息、工单状态、日程安排
                - 发送邮件、创建工单、生成周报
                请按 ReAct 格式思考：先分析需要哪些信息，再逐步调用工具。
                如果任务完成或遇到无法解决的问题，给出友好提示。
                """)
            .build();
    }

    public String execute(String userMessage) {
        // 第一步：发送用户消息
        ChatResponse response = chatClient.prompt()
            .user(userMessage)
            .tools(toolCallbacks)  // 注册可用工具
            .call()
            .chatResponse();

        // 第二步：检查是否有工具调用
        int iteration = 0;
        while (hasToolCalls(response) && iteration < 10) {
            iteration++;
            // 工具调用已在 ChatClient 内部自动执行
            // 工具结果自动追加到消息历史
            response = chatClient.prompt()
                .user(userMessage)  // 携带完整历史
                .tools(toolCallbacks)
                .call()
                .chatResponse();
        }

        // 第三步：返回最终结果
        return response.getResult().getOutput().getText();
    }

    private boolean hasToolCalls(ChatResponse response) {
        return response.getResult().getOutput().getToolCalls() != null
            && !response.getResult().getOutput().getToolCalls().isEmpty();
    }
}

Agent 执行过程中可能因为各种原因中断（Token 超限、工具失败、用户暂停）。CorpAssist 使用状态机管理 Agent 的生命周期，支持从断点恢复。

// Agent 状态机实现
public enum AgentState {
    IDLE, RUNNING, WAITING_TOOL, WAITING_APPROVAL, COMPLETED, FAILED
}

@Component
public class AgentStateMachine {

    private final Map<String, AgentSession> sessions = new ConcurrentHashMap<>();

    public AgentSession createSession(String sessionId, String userId, String task) {
        AgentSession session = new AgentSession(sessionId, userId, task);
        session.setState(AgentState.IDLE);
        sessions.put(sessionId, session);
        return session;
    }

    public AgentSession resumeSession(String sessionId) {
        AgentSession session = sessions.get(sessionId);
        if (session == null) {
            throw new IllegalArgumentException("Session not found: " + sessionId);
        }
        if (session.getState() == AgentState.FAILED) {
            session.setState(AgentState.IDLE);
            session.incrementRetryCount();
        }
        return session;
    }

    public void checkpoint(String sessionId, List<Message> history) {
        AgentSession session = sessions.get(sessionId);
        if (session != null) {
            session.setHistory(history);
            session.setLastCheckpoint(Instant.now());
        }
    }

    public record AgentSession(String sessionId, String userId, String task) {
        private AgentState state;
        private List<Message> history;
        private Instant lastCheckpoint;
        private int retryCount;
        // getters and setters omitted for brevity
    }
}

步数与 Token 预算

Agent 的无限循环和 Token 溢出是生产环境最常见的两类故障。Spring AI 通过 maxIterations 和 maxTokenUsage 提供基础防护，CorpAssist 在此基础上增加了更精细的预算控制。

预算不足

maxIterations=5, maxTokenUsage=2048：复杂任务（如"生成周报并发送给团队"）在第 3 步就因 Token 不足中断，只收集了部分数据，周报内容不完整。

预算合理

maxIterations=15, maxTokenUsage=8192：95% 的办公任务可以在预算内完成。剩余 5% 的超预算任务触发降级策略——返回已收集的中间结果和进度摘要。

思考 10 秒：Agent 在循环中反复调用同一个工具（如 getUserInfo）怎么办？

这是"工具循环"问题。三种应对：1) 在 System Prompt 中加入"避免重复调用已获取信息的工具"；2) 在工具层增加缓存——相同输入返回缓存结果而非重新调用；3) 在循环层检测重复模式——连续 3 次调用相同工具且参数相同，自动终止循环并返回"检测到重复调用，请检查您的请求"。CorpAssist 使用方案 2+3 的组合：工具层缓存避免重复执行，循环层检测防止浪费 Token。

面试可能问

Spring AI 的 ChatClient 是如何处理工具调用的？

ChatClient 内部使用 AbstractToolCallSupport 机制：1) 发送 prompt 时注册 ToolCallback 列表；2) LLM 响应可能包含 tool_calls 数组（JSON 格式，含 name 和 arguments）；3) ChatClient 自动匹配对应的 ToolCallback 执行方法；4) 将工具执行结果封装为 ToolResponseMessage 追加到消息列表；5) 再次请求 LLM 获取下一轮响应。整个过程对开发者透明——只需提供 ToolCallback，循环由 ChatClient 自动管理。如果开发者需要手动控制循环，可以关闭自动执行，手动解析 tool_calls。

Agent 执行到一半用户想暂停，如何实现？

通过状态机实现：1) Agent 在每次循环开始前检查 session 状态；2) 用户点击"暂停"时，状态从 RUNNING 改为 PAUSED；3) Agent 在当前工具调用完成后（不是中断工具执行）检查状态变化，发现 PAUSED 则保存当前 history 到持久化存储（Redis）并退出循环；4) 用户点击"继续"时，状态改为 RUNNING，Agent 从 Redis 加载 history，从上次暂停处继续执行。关键设计：不要在工具执行中间中断（可能导致脏数据），而是在循环边界检查状态。

maxTokenUsage 到达后应该怎样处理？

三种策略分层使用：1) 友好降级——返回当前已收集的信息和进度，"由于 Token 预算限制，我已完成以下步骤：1) 查到了张三的信息；2) 查到了工单列表。但还没有来得及发送邮件。请重新请求发送邮件功能。"；2) 压缩重试——自动压缩历史消息（summarize 模式），释放 Token 空间后继续执行；3) 优先队列——在设置 maxTokenUsage 时预留 20% 的 Buffer（如目标 8192，实际设置 10240），当接近阈值时优先执行剩余步骤中 Token 消耗最小的。CorpAssist 默认使用方案 1，对于重要任务使用方案 2。

为什么学这章

复杂办公任务（如"生成季度总结报告并发送给各部门负责人"）需要拆分成多个子任务逐步执行。Plan-and-Execute 模式将任务规划与执行分离，Spring AI 通过 ChatClient 的链式调用支持这种模式。了解任务分解和编排，是构建可靠 Agent 系统的关键。

学完后能做什么

用 Spring AI 实现 Plan-and-Execute 模式；设计子任务拆分与编排策略；使用 Redis 持久化检查点支持断点恢复；实现失败补偿和重试逻辑。

本章要点速记

1) Plan-and-Execute 分两步：Planner 生成执行计划，Executor 按计划逐步执行；2) 子任务拆分粒度以"一个工具调用"为单位，避免子任务内部再需要多步推理；3) 检查点持久化使用 Redis Hash 存储每个子任务的执行状态和结果；4) 失败恢复策略：子任务级重试（最多 3 次）+ 任务级跳过 + 整体补偿回滚；5) CorpAssist 的 Planner 使用 System Prompt 约束输出 JSON 格式的计划数组。

Plan-and-Execute 在 Java 侧

// Plan-and-Execute 实现
@Component
public class PlanAndExecuteAgent {

    private final ChatClient planner;
    private final ChatClient executor;

    public PlanAndExecuteAgent(ChatClient.Builder builder) {
        // Planner: 生成执行计划
        this.planner = builder
            .defaultSystem("""
                你是任务规划器。分析用户请求，输出 JSON 格式的执行计划。
                计划格式：[{"step": 1, "action": "getUserInfo", "params": {"name": "张三"}}]
                约束：
                - 每个 step 只能调用一个工具
                - 步骤不超过 10 个
                - 步骤间有数据依赖时标注 dependsOn
                """)
            .build();

        // Executor: 按计划执行
        this.executor = builder
            .defaultSystem("你是任务执行器，严格按照计划执行每一步。")
            .build();
    }

    public String execute(String userRequest, List<ToolCallback> tools) {
        // Phase 1: 规划
        String planJson = planner.prompt()
            .user(userRequest)
            .call()
            .content();

        List<PlanStep> plan = parsePlan(planJson);

        // Phase 2: 执行
        StringBuilder result = new StringBuilder();
        Map<String, Object> context = new HashMap<>();

        for (PlanStep step : plan) {
            try {
                String stepResult = executor.prompt()
                    .user("执行步骤 " + step.step + ": " + step.action
                        + "，参数: " + step.params
                        + "，当前上下文: " + context)
                    .tools(tools)
                    .call()
                    .content();
                context.put("step_" + step.step, stepResult);
                result.append("步骤").append(step.step).append("完成\n");
            } catch (Exception e) {
                result.append("步骤").append(step.step).append("失败: ")
                    .append(e.getMessage()).append("\n");
                // 检查是否需要补偿
                if (step.critical) {
                    throw new RuntimeException("关键步骤失败，任务终止");
                }
            }
        }
        return result.toString();
    }

    record PlanStep(int step, String action,
                    Map<String, Object> params,
                    List<Integer> dependsOn,
                    boolean critical) {}
}

// 带有依赖关系的任务编排器
@Component
public class TaskOrchestrator {

    public void executeDag(List<PlanStep> steps) {
        // 构建 DAG：key=stepId, value=依赖的 stepId 列表
        Map<Integer, List<Integer>> dag = new HashMap<>();
        for (PlanStep step : steps) {
            dag.put(step.step, step.dependsOn);
        }

        // 拓扑排序：先执行无依赖的步骤
        Set<Integer> completed = new HashSet<>();
        while (completed.size() < steps.size()) {
            for (PlanStep step : steps) {
                if (completed.contains(step.step)) continue;
                // 检查依赖是否全部完成
                boolean depsMet = step.dependsOn.stream()
                    .allMatch(completed::contains);
                if (depsMet) {
                    // 执行当前步骤
                    executeStep(step);
                    completed.add(step.step);
                }
            }
        }
    }

    private void executeStep(PlanStep step) {
        // 注入依赖步骤的结果作为上下文
        String prompt = "执行: " + step.action;
        if (!step.dependsOn.isEmpty()) {
            prompt += ", 依赖数据: step_" +
                String.join(", step_",
                    step.dependsOn.stream().map(String::valueOf).toList());
        }
        // 调用 Executor 执行
    }
}

检查点持久化（Redis）

检查点持久化是实现断点恢复的基础。CorpAssist 使用 Redis Hash 存储每个 session 的执行状态，包括已完成的步骤、中间结果、当前步骤索引。

java
// Redis 检查点持久化
@Component
public class CheckpointStore {

    private final RedisTemplate<String, Object> redisTemplate;
    private static final String KEY_PREFIX = "agent:checkpoint:";

    public CheckpointStore(RedisTemplate<String, Object> redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    public void saveCheckpoint(String sessionId, int stepIndex,
                                String stepResult, List<Message> history) {
        String key = KEY_PREFIX + sessionId;
        Map<String, Object> checkpoint = new HashMap<>();
        checkpoint.put("stepIndex", stepIndex);
        checkpoint.put("stepResult", stepResult);
        checkpoint.put("history", serializeHistory(history));
        checkpoint.put("updatedAt", Instant.now().toString());
        checkpoint.put("status", "IN_PROGRESS");

        redisTemplate.opsForHash().putAll(key, checkpoint);
        redisTemplate.expire(key, 24, TimeUnit.HOURS); // 24小时过期
    }

    public Checkpoint loadCheckpoint(String sessionId) {
        String key = KEY_PREFIX + sessionId;
        Map<Object, Object> entries = redisTemplate.opsForHash().entries(key);
        if (entries.isEmpty()) {
            return null;
        }
        return new Checkpoint(
            (int) entries.get("stepIndex"),
            (String) entries.get("stepResult"),
            deserializeHistory((String) entries.get("history")),
            (String) entries.get("status")
        );
    }

    public void markCompleted(String sessionId) {
        String key = KEY_PREFIX + sessionId;
        redisTemplate.opsForHash().put(key, "status", "COMPLETED");
    }

    record Checkpoint(int stepIndex, String stepResult,
                      List<Message> history, String status) {}
}

失败恢复与补偿

Agent 执行过程中可能遇到的失败类型：工具调用超时、工具返回异常数据、LLM 返回格式错误。对于不同类型的失败，CorpAssist 采用不同的恢复策略。

CorpAssist 场景

Agent 在执行"批量创建工单"过程中，第 5 个工单创建失败（工单系统返回 500）。由于该步骤 non-critical，Agent 跳过该工单并在最终结果中说明"已成功创建 4/5 个工单，第 5 个工单（ID: PRJ-005）创建失败，已记录异常，请手动处理"。如果该步骤是"创建付款审批"（critical=true）且失败，Agent 需要执行补偿操作——撤销已创建的关联工单，防止出现"已发付款但审批未创建"的不一致状态。

本章结论

  • Plan-and-Execute 将 Agent 分为 Planner 和 Executor 两个阶段，Planner 输出 JSON 格式的执行计划
  • 子任务拆分以"一个工具调用"为单位，步骤数控制在 10 步以内，减少 Token 消耗
  • Redis Hash 存储检查点包含 stepIndex、stepResult、history、status，24 小时过期
  • 失败恢复使用分层策略：工具超时重试 2 次→非关键步骤跳过→关键步骤整体回滚并补偿
  • CorpAssist 的 Planner 通过严格 System Prompt 约束输出格式，Executor 通过上下文注入管理步骤间数据依赖

面试可能问

Plan-and-Execute 和 ReAct 的区别和适用场景？

ReAct 是"边想边做"：LLM 在每次循环中决定下一步做什么，适合不确定步骤顺序的任务（如"查一个信息后决定下一步"）。Plan-and-Execute 是"先想好再做"：Planner 先生成完整计划，Executor 按计划执行，适合步骤明确的流程化任务（如"周报生成→审批→发送"三步固定流程）。CorpAssist 实践中：Plan-and-Execute 用于 80% 的办公自动化任务（流程固定），ReAct 用于 20% 需要灵活决策的任务（如客服对话）。性能上，Plan-and-Execute 的 Token 消耗比 ReAct 少约 30%，因为 Planner 只需要一次 LLM 调用，后续 Executor 不需要重复思考。

检查点存储应该包含哪些信息？序列化怎么做？

检查点包含：1) 元数据——sessionId、userId、任务描述、创建时间；2) 执行状态——当前步骤索引、已完成步骤列表、失败步骤列表；3) 中间结果——每个已完成步驟的输出（用于后续步骤的上下文）；4) 对话历史——最近 N 条 Message（用于 LLM 理解上下文）；5) 环境状态——已加载的工具列表、当前配置参数。序列化策略：对话历史使用 Spring AI 的 MessageSerializer（JSON 格式），中间结果使用 Jackson ObjectMapper。存储量控制：单个检查点不超过 100KB，历史消息压缩到最近 20 条。

Planner 输出了错误的计划（如步骤顺序错误），如何检测和纠正？

三种检测手段：1) 静态校验——Planner 输出 JSON 后，校验每个步骤的 dependsOn 是否有效（引用的步骤 ID 是否存在）、步骤数是否在范围内；2) 执行时验证——Executor 执行前检查依赖步骤是否已完成，如果发现循环依赖（step1 依赖 step2，step2 也依赖 step1），自动修正为顺序执行；3) LLM 自检——将计划打印给 LLM 让它自我检查，"请审核以下执行计划是否有逻辑错误"。如果发现错误，CorpAssist 的策略是：将计划返回给 Planner 并要求重新生成（最多 2 次），如果仍错误则使用默认的固定流程模板。

场景：Agent 执行到第 3 步时系统崩溃重启，如何从检查点恢复？

1) Agent 启动时从 Redis 加载 session 的检查点；2) 检查 status 是否为 IN_PROGRESS（如果是 COMPLETED 或 FAILED 则不需要恢复）；3) 从检查点获取 history、已完成步骤结果、当前步骤索引；4) 恢复 ChatClient 的历史消息（使用 Spring AI 的 ChatHistory.withMessages）；5) 从断点步骤重新执行（不是重试已完成的步骤）。关键设计思路：不要重试已完成的步骤（可能产生副作用，如重复发邮件），而是直接从断点处继续。CorpAssist 使用幂等性设计——每个步骤的执行结果包含幂等键，系统恢复时检查幂等键避免重复执行。

场景：Planner 输出了 15 个步骤，但 maxIterations 只允许 10 步，怎么办？

解决方案：1) Planner 的 System Prompt 中加入"最多输出 8 步"的约束；2) 如果 Planner 仍然输出了超过限制的计划，触发计划压缩——将连续的 2-3 个步骤合并为一个"批量步骤"（如将 getUserInfo、getDepartmentInfo、getProjectList 合并为 gatherContextInfo 步骤）；3) 如果无法压缩，则分阶段执行——先执行前 8 步，完成后重新调用 Planner 生成剩余步骤的计划。CorpAssist 使用方案 3，因为方案 2 的合并步骤内部逻辑可能过于复杂。

// Spring AI 短期记忆配置
@Configuration
public class MemoryConfig {

    @Bean
    public ChatMemory shortTermMemory() {
        // Summarize 模式：每 5 轮对话压缩一次历史
        return ChatMemory.builder()
            .strategy(ChatMemory.Strategy.SUMMARIZE)
            .maxMessages(50)           // 保留最近 50 条消息
            .summarizeThreshold(10)    // 超过 10 条触发摘要
            .summarizeTokenLimit(1024) // 摘要最大 Token 数
            .build();
    }

    @Bean
    public ChatClient memoryChatClient(ChatClient.Builder builder,
                                        ChatMemory chatMemory) {
        return builder
            .defaultSystem("你是 CorpAssist 企业办公助手。")
            .chatMemory(chatMemory)
            .build();
    }
}

CorpAssist 场景

员工小张在 CorpAssist 中连续请求："查我的工单"→"看下还有没有未完成的"→"帮我给项目经理发个进度消息"。没有短期记忆的话，Agent 每次都需要重新问"你是谁""哪个项目"。有了 Summarize 模式的 ChatMemory，Agent 自动将前三轮对话压缩为"用户是小张，正在查看工单状态并想给项目经理发进度消息"，后续对话直接使用摘要作为上下文。每次摘要约 200 Tokens，相比完整对话历史节省 60% Token。

向量存储长期记忆

@Component
public class LongTermMemory {

    private final VectorStore vectorStore;

    public LongTermMemory(VectorStore vectorStore) {
        this.vectorStore = vectorStore;
    }

    public void rememberUserPreference(String userId, String key, String value) {
        Document doc = new Document(
            "preference:" + userId + ":" + key,
            Map.of("userId", userId, "key", key, "value", value, "type", "preference")
        );
        vectorStore.add(List.of(doc));
    }

    public List<Document> recallUserPreferences(String userId, String query) {
        // 构建搜索请求
        SearchRequest request = SearchRequest.builder()
            .query("用户 " + userId + " 的偏好: " + query)
            .topK(5)
            .similarityThreshold(0.7)
            .filterExpression("userId == '" + userId + "' && type == 'preference'")
            .build();
        return vectorStore.similaritySearch(request);
    }

    public void rememberTaskHistory(String userId, String taskType,
                                     String taskResult) {
        Document doc = new Document(
            "task:" + userId + ":" + taskType + ":" + Instant.now(),
            Map.of("userId", userId, "taskType", taskType,
                   "result", taskResult, "timestamp", Instant.now().toString())
        );
        vectorStore.add(List.of(doc));
    }
}

Q: MCP 工具和 @Tool 注解的区别是什么？

@Tool 注解是 Spring AI 原生的工具定义方式，方法直接写在 Java 代码中，适合内部工具。MCP 工具是通过标准协议调用的外部服务，适合跨团队、跨语言的服务集成。CorpAssist 实践中：核心工具（如 sendEmail、queryTicket）用 @Tool 注解直接实现，扩展工具（如 hrService、financeService）通过 MCP 注册。MCP 的优势在于：1) 工具服务器可以独立部署和扩展；2) 支持多种语言实现；3) 协议标准化，工具定义自动生成。

面试可能问

Summarize 模式的摘要丢失了重要信息怎么办？

三种应对策略：1) 摘要保留关键字段——在 Summarize Prompt 中指定"保留以下信息：用户身份、任务目标、已完成步骤、决策原因"，确保这些关键信息不会被压缩掉；2) 分级压缩——短期记忆（最近 10 轮）保留完整对话，中期记忆（10-30 轮）使用摘要，长期记忆（30 轮以上）仅保留关键决策点；3) 用户手动保留——用户可以标记某条消息为"重要"，标记的消息不会被压缩。CorpAssist 使用方案 2，根据经验 80% 的上下文来自最近 10 轮对话，摘要丢失重要信息的概率约 2%。

MCP 工具的延迟比 @Tool 高多少？如何优化？

MCP 工具比 @Tool 多一次网络调用（HTTP/SSE），典型的延迟差异：@Tool 调用约 5-20ms（同进程方法调用），MCP 工具约 50-200ms（网络往返 + 序列化）。优化策略：1) 连接池——MCP Client 使用 HTTP 连接池（keep-alive）减少连接建立时间；2) 本地缓存——对于频繁调用的工具（如 getUserInfo），使用本地缓存减少重复请求；3) 批量处理——MCP 协议支持批量调用，将多个工具请求合并为一次网络往返；4) 预连接——Agent 初始化时建立 MCP 连接，避免首次调用时的手续延迟。优化后 MCP 工具延迟可降至 20-50ms。

长期记忆的向量检索效果不好（检索不到相关记忆）怎么办？

四步优化：1) 元数据过滤——搜索时使用 filterExpression 限定用户范围（如 userId=='u123'），避免检索到其他用户的记忆；2) Embedding 模型优化——使用 domain-specific 的 Embedding 模型（如 bge-large-zh）替代通用模型，提升办公场景的语义匹配效果；3) 混合检索——结合关键词匹配（BM25）和向量检索，先精确匹配用户 ID+偏好键，再语义搜索；4) 记忆刷新——Agent 每次生成回答后，主动将关键的交互结果写入长期记忆，确保"刚刚使用过的偏好"一定在最新数据中。

// Graph 工作流配置（条件路由）
@Configuration
public class WorkflowConfig {

    @Bean
    public Graph<WorkflowState> reportWorkflow() {
        return Graph.builder()
            .stateClass(WorkflowState.class)
            // 注册节点
            .node("getUserInfo", "getUserInfo()")  // @Bean 方法名
            .node("queryTeamTickets", "queryTeamTickets()")
            .node("queryMyTickets", "queryMyTickets()")
            .node("generateReport", "generateReport()")
            .node("submitApproval", "submitApproval()")
            .node("sendEmail", "sendEmail()")
            .node("notifyUser", "notifyUser()")
            // 定义边
            .edge("START", "getUserInfo")
            // 条件边：根据 isManager 路由
            .condition("getUserInfo", state ->
                state.isManager() ? "queryTeamTickets" : "queryMyTickets")
            .edge("queryTeamTickets", "generateReport")
            .edge("queryMyTickets", "generateReport")
            // 条件边：根据 needApproval 路由
            .condition("generateReport", state ->
                state.isNeedApproval() ? "submitApproval" : "sendEmail")
            .edge("submitApproval", "sendEmail")
            .edge("notifyUser", "END")
            .build();
    }
}

状态序列化与恢复

工作流的持久化和恢复依赖于 State 对象的序列化。Spring AI Alibaba Graph 使用 Jackson 序列化 State，支持存储到 Redis 或数据库。

// 状态序列化与持久化
@Component
public class GraphStatePersistence {

    private final RedisTemplate<String, String> redisTemplate;
    private final ObjectMapper objectMapper;
    private static final String PREFIX = "graph:state:";

    public void saveState(String workflowId, WorkflowState state) {
        try {
            String json = objectMapper.writeValueAsString(state);
            redisTemplate.opsForValue().set(
                PREFIX + workflowId, json,
                Duration.ofHours(24)  // TTL 24 小时
            );
        } catch (JsonProcessingException e) {
            throw new RuntimeException("State 序列化失败", e);
        }
    }

    public WorkflowState loadState(String workflowId) {
        String json = redisTemplate.opsForValue().get(PREFIX + workflowId);
        if (json == null) return null;
        try {
            return objectMapper.readValue(json, WorkflowState.class);
        } catch (JsonProcessingException e) {
            throw new RuntimeException("State 反序列化失败", e);
        }
    }

    public void deleteState(String workflowId) {
        redisTemplate.delete(PREFIX + workflowId);
    }
}

public class GraphExecutor {

    public void executeWithRecovery(String workflowId, String userId) {
        // 尝试从上次中断处恢复
        WorkflowState state = persistence.loadState(workflowId);
        if (state == null) {
            state = WorkflowState.builder()
                .sessionId(workflowId).userId(userId).build();
        }
        // 继续执行未完成的节点
        graph.execute(state, workflowId);
    }
}

无持久化恢复

工作流执行到一半系统崩溃，所有进度丢失。用户需要重新发起请求，Agent 重新执行所有节点（包括已完成的），浪费 Token 和时间。

有持久化恢复

工作流每完成一个节点就持久化 State。系统崩溃恢复后，从 Redis 加载 State，直接从中断的节点继续执行。State 包含已完成节点列表和中间结果，不重复执行已完成的节点。

  • Spring AI Alibaba Graph 用 @Bean 函数定义节点，接收和返回 State 对象
  • 条件边通过 router-function 根据 State 字段值路由到不同分支，支持复杂的多条件逻辑
  • Admin Studio 提供实时可视化调试，显示节点耗时和执行路径
  • State 使用 Jackson 序列化，持久化到 Redis 支持断点恢复

面试可能问

Graph 和 ReAct 在状态管理上的核心区别是什么？

ReAct 的状态管理是隐式的——状态存在于对话历史（Message 列表）中，Agent 通过 ChatMemory 获得上下文。Graph 的状态管理是显式的——开发者定义 WorkflowState 类，明确所有状态字段，节点函数显式读写这些字段。核心区别：1) 可预测性——Graph 的 State 结构固定，执行路径确定；2) 可持久化——Graph 的 State 可序列化为 JSON 存储；3) 可测试——Graph 的每个节点可以独立单元测试；4) 调试难度——Graph 的 State 变化可追踪，ReAct 的中间推理过程不透明。CorpAssist 实践中：需要审计追溯的流程（如财务审批）用 Graph，快速问答用 ReAct。

Graph 工作流的 State 太大导致序列化慢怎么办？

优化策略：1) 状态裁剪——State 只保存必需的字段，大型数据（如完整的工单列表）只保存 ID 和摘要；2) 增量序列化——State 保存"增量"而非"全量"，每次只保存相比上一次 checkpoint 的变化部分；3) 分片存储——将 State 分为元数据（userInfo, status）和业务数据（tickets, report）两部分，元数据用 Redis Hash，业务数据用单独的 Key；4) 异步序列化——序列化操作异步执行，不影响主流程。CorpAssist 使用方案 1+3，典型 State 大小控制在 10KB 以内，序列化时间 <5ms。

Graph 工作流出现循环（loop）如何防止死循环？

三种防护：1) maxLoops 配置——每条 LOOP 类型的边配置最大循环次数（如 maxLoops=5）；2) State 中维护 loopCount——每次经过循环节点时 loopCount+1，到达阈值后自动退出循环；3) 循环检测——Graph 引擎内置检测器，如果同一节点连续执行超过 3 次且 State 无有效变化，判定为死循环并终止。CorpAssist 使用方案 1+2，在"审批驳回重新提交"循环中，最多允许 3 次驳回重提，超过后转人工处理。

并行/串行工具执行

任务规划时需要考虑工具间的依赖关系。无依赖的工具可以并行执行（提升效率），有依赖的工具必须串行执行（保证数据正确）。

java
// 并行/串行工具执行器
@Component
public class ToolExecutionOrchestrator {

    private final ExecutorService parallelExecutor =
        Executors.newVirtualThreadPerTaskExecutor();  // Java 21 虚拟线程

    /**
     * 执行无依赖的工具集（并行）
     */
    public Map<String, Object> executeParallel(List<ToolTask> tasks) {
        Map<String, Object> results = new ConcurrentHashMap<>();
        List<CompletableFuture<Void>> futures = tasks.stream()
            .map(task -> CompletableFuture.runAsync(() -> {
                results.put(task.name(), task.execute());
            }, parallelExecutor))
            .toList();
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
            .join();  // 等待所有并行任务完成
        return results;
    }

    /**
     * 执行有依赖关系的工具集（串行）
     */
    public Map<String, Object> executeSequential(List<ToolTask> tasks) {
        Map<String, Object> results = new LinkedHashMap<>();
        Map<String, Object> context = new HashMap<>();

        for (ToolTask task : tasks) {
            // 注入依赖的结果作为上下文
            task.setContext(context);
            Object result = task.execute();
            results.put(task.name(), result);
            // 将当前步骤结果加入上下文供后续步骤使用
            context.put(task.name(), result);
        }
        return results;
    }

    // CorpAssist 会议安排编排示例
    public Map<String, Object> arrangeMeeting(String organizer,
                                                String date, String topic) {
        // Step 1: 并行查询参与人的空闲时间
        List<ToolTask> parallelTasks = List.of(
            new ToolTask("getUserInfo", () -> userService.getUser(organizer)),
            new ToolTask("getTeamMembers", () -> orgService.getTeamMembers(organizer))
        );
        Map<String, Object> parallelResults = executeParallel(parallelTasks);

        // Step 2: 串行：查空闲→创建会议→发通知
        List<ToolTask> sequentialTasks = List.of(
            new ToolTask("queryFreeSlots", () -> calendarService
                .queryFreeSlots(organizer, date, date)),
            new ToolTask("createEvent", () -> calendarService
                .createEvent(topic, /* 从上一步获取 */ "")),
            new ToolTask("sendEmail", () -> emailService
                .send(/* 从上一步获取参与人 */ organizer, topic))
        );
        return executeSequential(sequentialTasks);
    }
}

工具失败的重试与熔断

工具调用可能因为网络问题、服务故障、数据异常而失败。CorpAssist 使用分层容错策略，结合 Spring Resilience4j 实现重试和熔断。

java
// 工具调用容错配置
@Configuration
public class ToolResilienceConfig {

    @Bean
    public RetryRegistry toolRetryRegistry() {
        RetryConfig config = RetryConfig.custom()
            .maxAttempts(3)
            .waitDuration(Duration.ofSeconds(2))
            .retryExceptions(IOException.class, TimeoutException.class)
            .ignoreExceptions(IllegalArgumentException.class)  // 参数错误不重试
            .build();
        return RetryRegistry.of(config);
    }

    @Bean
    public CircuitBreakerRegistry toolCircuitBreakerRegistry() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            .failureRateThreshold(50)           // 50% 失败率触发熔断
            .slidingWindowSize(10)              // 统计最近 10 次调用
            .waitDurationInOpenState(Duration.ofSeconds(30))  // 熔断持续 30 秒
            .permittedNumberOfCallsInHalfOpenState(3)         // 半开状态尝试 3 次
            .build();
        return CircuitBreakerRegistry.of(config);
    }
}

// 带容错的工具调用
@Component
public class ResilientToolExecutor {

    private final Retry retry;
    private final CircuitBreaker circuitBreaker;

    public ResilientToolExecutor(RetryRegistry retryRegistry,
                                  CircuitBreakerRegistry cbRegistry) {
        this.retry = retryRegistry.retry("toolRetry");
        this.circuitBreaker = cbRegistry.circuitBreaker("toolCircuitBreaker");
    }

    public Object executeWithResilience(String toolName,
                                        Supplier<Object> toolCall) {
        return circuitBreaker.executeSupplier(
            () -> retry.executeSupplier(toolCall)
        );
    }
}

上下文压缩 Adapter

工具返回的数据可能非常大（如查询整个团队的日程返回数百条事件），直接传给 LLM 会浪费 Token。上下文压缩 Adapter 在工具返回数据和 LLM 之间加一层截断/摘要。

java
// 上下文压缩适配器
@Component
public class ContextCompressionAdapter implements ToolCallback {

    private final ToolCallback delegate;
    private static final int MAX_TOOL_RESPONSE_LENGTH = 500;

    public ContextCompressionAdapter(ToolCallback delegate) {
        this.delegate = delegate;
    }

    @Override
    public String getName() {
        return delegate.getName();
    }

    @Override
    public String getDescription() {
        return delegate.getDescription();
    }

    @Override
    public JsonSchema getInputSchema() {
        return delegate.getInputSchema();
    }

    @Override
    public String call(String functionInput) {
        String rawResult = delegate.call(functionInput);

        // 如果结果太长，截断并添加摘要
        if (rawResult.length() > MAX_TOOL_RESPONSE_LENGTH) {
            return truncateAndSummarize(rawResult);
        }
        return rawResult;
    }

    private String truncateAndSummarize(String raw) {
        // 策略 1: 截断
        if (raw.length() > MAX_TOOL_RESPONSE_LENGTH * 2) {
            return raw.substring(0, MAX_TOOL_RESPONSE_LENGTH)
                + "\n...(截断，共 " + raw.length() + " 字符)";
        }
        // 策略 2: 提取关键字段
        if (raw.startsWith("[") || raw.startsWith("{")) {
            return extractKeyFields(raw);
        }
        return raw;
    }

    // 注册时包装所有工具
    @Bean
    public List<ToolCallback> compressedTools(List<ToolCallback> rawTools) {
        return rawTools.stream()
            .map(ContextCompressionAdapter::new)
            .collect(Collectors.toList());
    }
}

并行工具调用中一个工具失败了，应该全部回滚还是只回滚失败的工具？

取决于工具间是否有业务依赖。如果有依赖关系（如"创建会议"依赖"查空闲时间"的结果），则全部回滚——因为后续步骤基于错误数据执行。如果无依赖关系（如同时"查日历"和"查工单"），则只失败单个工具，其他工具的结果继续使用。CorpAssist 的实现：每个 ToolTask 有一个 rollback() 方法，如果并行组中有任一工具失败，调用所有已完成工具的 rollback()。对于不可回滚的操作（如"发邮件"已发送成功），记录审计日志并标记"部分成功"。实践中，CorpAssist 约 80% 的并行工具调用是无依赖的，只需要部分回滚。

上下文压缩 Adapter 截断了重要信息导致 LLM 决策错误怎么办？

解决方案：1) 分层压缩——重要数据字段（如工单 ID、状态、截止日期）完整保留，次要数据（如备注、操作日志）截断；2) 可扩展截断——当 LLM 发现需要更多信息时，可以调用 getToolFullResult(toolName) 获取完整数据；3) 压缩摘要中包含数据量提示——如"共 200 条日程，已显示前 5 条"，LLM 知道有更多数据可用；4) 关键字段白名单——使用 @ToolParam(important=true) 标记不会截断的关键字段。CorpAssist 使用方案 1+3：先根据字段重要性选择性保留，再在摘要末尾标注数据总量。

@Tool 的 returnDirect=true 应该什么时候用？

returnDirect=true 表示工具的结果直接返回给用户，不经过 LLM 的二次处理。适用场景：1) 纯数据查询类——"查天气"返回 JSON，直接展示即可，不需要 LLM 重新描述；2) 操作确认类——"发送邮件"返回{success:true, messageId:"xxx"}，直接显示确认信息；3) 文件下载类——返回文件 URL，用户直接点击下载。不适用场景：1) 需要 LLM 解释的数据——如"查工单"返回原始 JSON，需要 LLM 解释为自然语言；2) 多步任务的中间步骤——因为后续步骤需要 LLM 理解工具返回的内容。CorpAssist 中约 30% 的工具使用 returnDirect=true。

// 多 Agent 协作框架
@Component
public class MultiAgentOrchestrator {

    @Autowired
    private List<SpecializedAgent> agents;

    /**
     * Sequential 模式
     */
    public String executeSequential(String input, List<String> agentIds) {
        String context = input;
        for (String id : agentIds) {
            SpecializedAgent agent = findAgent(id);
            context = agent.process(context);
        }
        return context;
    }

    /**
     * Parallel 模式
     */
    public String executeParallel(String input, List<String> agentIds) {
        Map<String, CompletableFuture<String>> futures = new HashMap<>();
        for (String id : agentIds) {
            SpecializedAgent agent = findAgent(id);
            futures.put(id, CompletableFuture.supplyAsync(
                () -> agent.process(input)));
        }
        // 等待所有 Agent 完成
        StringBuilder result = new StringBuilder();
        futures.forEach((id, future) -> {
            try {
                result.append("【").append(id).append("】")
                    .append(future.get()).append("\n");
            } catch (Exception e) {
                result.append("【").append(id).append("】失败: ")
                    .append(e.getMessage()).append("\n");
            }
        });
        return result.toString();
    }

    /**
     * Routing 模式
     */
    public String executeRouting(String input) {
        // 路由 Agent 根据输入决定交给哪个 Agent
        String routingDecision = routingAgent.analyze(input);
        SpecializedAgent target = findAgent(routingDecision);
        return target.process(input);
    }

    private SpecializedAgent findAgent(String id) {
        return agents.stream()
            .filter(a -> a.getAgentId().equals(id))
            .findFirst()
            .orElseThrow(() -> new IllegalArgumentException("Agent not found: " + id));
    }
}

// Nacos 事件驱动的 Agent 消息总线
@Component
public class AgentEventBus {

    private final NacosEventPublisher eventPublisher;
    private final Map<String, List<AgentEventHandler>> handlers = new ConcurrentHashMap<>();

    public AgentEventBus(NacosEventPublisher eventPublisher) {
        this.eventPublisher = eventPublisher;
    }

    public void publish(AgentEvent event) {
        // 发布事件到 Nacos
        eventPublisher.publishEvent(
            "agent:event:" + event.getTargetAgent(),
            event
        );
    }

    @EventListener
    public void handleEvent(AgentEvent event) {
        String target = event.getTargetAgent();
        List<AgentEventHandler> agentHandlers = handlers.get(target);
        if (agentHandlers != null) {
            agentHandlers.forEach(h -> h.onEvent(event));
        }
    }

    public void subscribe(String agentId, AgentEventHandler handler) {
        handlers.computeIfAbsent(agentId, k -> new CopyOnWriteArrayList<>())
            .add(handler);
    }
}

// Handoff 管理器
@Component
public class AgentHandoffManager {

    private final AgentEventBus eventBus;
    private final RedisTemplate<String, Object> redisTemplate;

    public void handoff(HandoffContext context) {
        // 1. 持久化上下文到 Redis
        String key = "handoff:" + context.getSessionId();
        redisTemplate.opsForValue().set(key, context, Duration.ofHours(1));

        // 2. 发送 Handoff 事件
        AgentEvent event = new AgentEvent();
        event.setEventId(UUID.randomUUID().toString());
        event.setSourceAgent(context.getSourceAgent());
        event.setTargetAgent(context.getTargetAgent());
        event.setEventType("HANDOFF");
        event.setPayload(context.getSessionId());
        eventBus.publish(event);

        // 3. 源 Agent 清理（释放资源）
        cleanupSourceAgent(context);
    }

    public HandoffContext acceptHandoff(String sessionId) {
        String key = "handoff:" + sessionId;
        HandoffContext context = (HandoffContext) redisTemplate
            .opsForValue().get(key);
        if (context == null) {
            throw new HandoffException("Handoff context not found: " + sessionId);
        }
        return context;
    }
}

冲突检测

多 Agent 并行执行时可能出现冲突：两个 Agent 同时修改同一条数据、一个 Agent 的操作被另一个 Agent 覆盖。CorpAssist 使用乐观锁和向量对比两种方式检测冲突。

java
// 冲突检测器
@Component
public class ConflictDetector {

    @Value("${app.conflict.similarity-threshold:0.7}")
    private double similarityThreshold;

    /**
     * 检测两个 Agent 输出的决策是否存在冲突
     */
    public ConflictResult detectConflict(String sessionId,
                                          AgentOutput outputA,
                                          AgentOutput outputB) {
        // 1. 检查数据修改冲突
        if (outputA.getTargetResource() != null
            && outputA.getTargetResource().equals(outputB.getTargetResource())
            && outputA.getVersion() != outputB.getVersion()) {
            return ConflictResult.conflict("DATA_VERSION",
                "数据版本不一致: A=" + outputA.getVersion()
                    + ", B=" + outputB.getVersion());
        }

        // 2. 检查时间冲突
        if (outputA.getTimeRange() != null && outputB.getTimeRange() != null) {
            if (isTimeOverlapping(outputA.getTimeRange(), outputB.getTimeRange())) {
                return ConflictResult.conflict("TIME_OVERLAP",
                    "时间范围重叠: " + outputA.getTimeRange()
                        + " vs " + outputB.getTimeRange());
            }
        }

        // 3. 检查决策矛盾（使用向量相似度）
        double similarity = cosineSimilarity(
            outputA.getDecisionEmbedding(),
            outputB.getDecisionEmbedding()
        );
        if (similarity < similarityThreshold) {
            return ConflictResult.conflict("DECISION_CONFLICT",
                "决策差异大: 相似度=" + similarity);
        }

        return ConflictResult.ok();
    }

    private double cosineSimilarity(float[] a, float[] b) {
        double dot = 0, normA = 0, normB = 0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            normA += a[i] * a[i];
            normB += b[i] * b[i];
        }
        return dot / (Math.sqrt(normA) * Math.sqrt(normB));
    }
}

面试可能问

Routing Agent 如何判断输入应该交给哪个子 Agent？

Routing Agent 本身是一个 LLM 调用，System Prompt 中定义路由规则。示例 Prompt："分析用户输入，判断应该交给哪个 Agent 处理。可选 Agent：1) schedule-agent——处理日程查询和会议安排；2) ticket-agent——处理工单创建和查询；3) email-agent——处理邮件发送和查询。输出格式：{"agent":"schedule-agent","confidence":0.95,"reason":"用户请求查日程"}。" Routing Agent 返回的 confidence 如果低于阈值（如 0.7），则触发"澄清模式"反问用户。CorpAssist 还使用 fallback 路由：如果 Routing Agent 的置信度低，将请求同时发送给所有候选 Agent，取第一个返回结果的。

Handoff 过程中如何保证数据不丢失？

使用两阶段提交（2PC）风格的交接协议：1) Prepare 阶段——源 Agent 将 HandoffContext 持久化到 Redis，状态标记为 HANDOFF_PREPARED；2) Commit 阶段——源 Agent 发送 HANDOFF 事件到目标 Agent；3) Ack 阶段——目标 Agent 收到事件后从 Redis 加载 Context，处理完成后发送 HANDOFF_ACK 事件；4) 清理——源 Agent 收到 ACK 后删除 Redis 中的 Context（或保留 24 小时用于审计）。如果目标 Agent 处理失败，源 Agent 在超时后（如 30 秒）重试发送 HANDOFF 事件。如果 Redis 宕机，使用数据库作为后备存储。

三个 Parallel Agent 的执行结果矛盾，如何自动解决？

三阶段解决：1) 多数决——如果三个 Agent 中有两个输出一致（相似度 > 0.8），采用多数结果；2) 权重决——每个 Agent 有历史准确率权重（如 schedule-agent 准确率 95%，高于 ticket-agent 的 88%），加权投票；3) 仲裁决——如果前两阶段无法解决，调用一个专门的仲裁 Agent 分析三个矛盾结果并做出最终决定。如果仲裁 Agent 也无法解决（置信度 < 0.6），转人工。CorpAssist 统计数据：约 85% 的矛盾可通过多数决解决，10% 通过加权投票，5% 需要仲裁 Agent，不到 1% 需要转人工。

不是所有 Agent 决策都可以全自动执行。审批节点（Human-in-the-Loop）是 Agent 在企业落地的关键能力——Agent 执行到关键步骤时暂停，等待人工审批通过后继续。理解审批中断、超时降级、审计追溯是构建可信 Agent 系统的基础。

// 审批节点实现
@Component
public class ApprovalGateway {

    private final CheckpointStore checkpointStore;
    private final KafkaTemplate<String, Object> kafkaTemplate;

    public void requestApproval(String sessionId, ApprovalRequest request) {
        // 1. 保存检查点
        checkpointStore.saveCheckpoint(sessionId, ApprovalState.WAITING);

        // 2. 创建审批任务并发送到审批系统
        ApprovalTask task = ApprovalTask.builder()
            .taskId(UUID.randomUUID().toString())
            .sessionId(sessionId)
            .userId(request.getUserId())
            .title(request.getTitle())
            .description(request.getDescription())
            .amount(request.getAmount())
            .approvalType(request.getType())  // SINGLE / ALL / ANY
            .approvers(request.getApprovers())
            .status(ApprovalStatus.PENDING)
            .createdAt(Instant.now())
            .timeoutMinutes(request.getTimeoutMinutes())
            .build();

        kafkaTemplate.send("approval-tasks", task.getTaskId(), task);
    }

    /**
     * 审批结果回调
     */
    @KafkaListener(topics = "approval-results")
    public void handleApprovalResult(ApprovalResult result) {
        // 1. 验证审批结果
        if (!result.isApproved()) {
            // 审批拒绝：Agent 执行拒绝流程
            checkpointStore.updateState(result.getSessionId(),
                ApprovalState.REJECTED);
            notifyUser(result.getSessionId(), "审批被拒绝: " + result.getComment());
            return;
        }

        // 2. 审批通过：恢复 Agent 执行
        checkpointStore.updateState(result.getSessionId(),
            ApprovalState.APPROVED);
        resumeAgent(result.getSessionId(), result);
    }

    private void resumeAgent(String sessionId, ApprovalResult result) {
        // 加载检查点，从断点继续执行
        Checkpoint cp = checkpointStore.loadCheckpoint(sessionId);
        // 注入审批结果到上下文
        cp.getContext().put("approvalResult", result);
        // 继续执行
        agentEngine.resume(cp);
    }
}

审批超时自动降级

如果审批人在规定时间内未响应，Agent 需要自动降级——不能无限等待。CorpAssist 提供三种降级策略。

java
// 审批超时调度器
@Component
public class ApprovalTimeoutScheduler {

    @Scheduled(fixedRate = 30000)  // 每 30 秒检查一次
    public void checkTimeouts() {
        List<ApprovalTask> pendingTasks = approvalRepository
            .findByStatusAndCreatedAtBefore(
                ApprovalStatus.PENDING,
                Instant.now().minus(Duration.ofMinutes(30))
            );

        for (ApprovalTask task : pendingTasks) {
            handleTimeout(task);
        }
    }

    private void handleTimeout(ApprovalTask task) {
        switch (task.getTimeoutPolicy()) {
            case AUTO_APPROVE:
                // 自动通过
                processResult(new ApprovalResult(task.getTaskId(), true,
                    "系统自动通过（超时）"));
                break;
            case AUTO_REJECT:
                // 自动拒绝
                processResult(new ApprovalResult(task.getTaskId(), false,
                    "系统自动拒绝（超时）"));
                notifyUser(task.getSessionId(), "审批超时已自动拒绝");
                break;
            case ESCALATE:
                // 升级给上级
                String managerId = getManagerId(task.getOriginalApprover());
                task.setApprover(managerId);
                task.setCreatedAt(Instant.now());
                task.setEscalationCount(task.getEscalationCount() + 1);
                if (task.getEscalationCount() <= 2) {
                    approvalRepository.save(task);
                    notifyApprover(managerId, task);
                } else {
                    // 升级超过 2 次，自动拒绝
                    processResult(new ApprovalResult(task.getTaskId(), false,
                        "升级超过 2 次无人处理，自动拒绝"));
                }
                break;
        }
    }
}

            audits.stream().filter(a -> "TIMEOUT".equals(a.getStatus())).count(),

Resume 恢复

审批完成后，Agent 需要从保存的检查点恢复执行。恢复时需注意：不重复执行已完成的步骤、正确处理审批结果对后续步骤的影响。

java
// Agent 恢复执行器
@Component
public class AgentResumeHandler {

    private final CheckpointStore checkpointStore;
    private final ChatClient chatClient;

    public AgentSession resume(String sessionId, ApprovalResult result) {
        // 1. 加载检查点
        Checkpoint checkpoint = checkpointStore.loadCheckpoint(sessionId);
        if (checkpoint == null) {
            throw new ResumeException("检查点不存在: " + sessionId);
        }

        // 2. 重建 ChatClient 历史
        List<Message> history = checkpoint.getHistory();
        ChatClient resumeClient = chatClient.mutate()
            .chatMemory(memory -> memory.withMessages(history))
            .build();

        // 3. 注入审批结果作为系统消息
        String approvalContext = String.format(
            "【审批结果】步骤 %s 的审批结果: %s。审批意见: %s。继续执行后续步骤。",
            checkpoint.getPendingStep(),
            result.isApproved() ? "已通过" : "已拒绝",
            result.getComment()
        );

        // 4. 从暂停的步骤继续执行
        String response = resumeClient.prompt()
            .user(approvalContext)
            .tools(checkpoint.getRegisteredTools())
            .call()
            .content();

        // 5. 返回恢复后的会话
        return new AgentSession(sessionId, resumeClient, response);
    }

    // 恢复状态接口（供审批系统回调）
    @PostMapping("/api/agent/resume")
    public ResponseEntity<String> resumeEndpoint(
            @RequestBody ResumeRequest request) {
        try {
            AgentSession session = resume(
                request.getSessionId(),
                request.getApprovalResult()
            );
            return ResponseEntity.ok("Agent 已从审批恢复，继续执行");
        } catch (Exception e) {
            return ResponseEntity.status(500)
                .body("恢复失败: " + e.getMessage());
        }
    }
}

审批恢复时重新执行所有步骤

审批通过后 Agent 从头开始执行，导致重复查用户信息、重复创建工单（可能需要先删除已创建的工单）。Token 浪费 3 倍，时间浪费 5 倍。

审批恢复时从断点继续

审批通过后 Agent 从暂停的步骤继续执行，已完成的步骤不重复。Token 消耗仅增加 10%（用于注入审批结果上下文），时间无额外增加。

面试可能问

审批节点如何与业务系统（如 OA 系统）集成？

两种集成方式：1) API 回调集成——OA 系统审批完成后调用 CorpAssist 的 /api/agent/resume 接口，传入 sessionId 和审批结果。这种方式简单直接，但需要 OA 系统支持回调。2) 消息队列集成——CorpAssist 将审批任务发送到 Kafka/RocketMQ，OA 系统消费后处理审批，将结果发送到结果队列。推荐方式 2，因为解耦、支持异步、可追溯。CorpAssist 具体实现：审批任务发送到 approval-requests 队列，OA 系统消费并展示给审批人；审批人操作后 OA 发送结果到 approval-results 队列，Agent 监听并恢复执行。

审批环节的 State 快照包含哪些信息？占多少存储？

State 快照包含：1) 会话元数据——sessionId, userId, 任务描述；2) Agent 状态——已完成步骤列表、当前步骤、已收集数据；3) 对话历史——最近 20 条消息（用于 LLM 上下文重建）；4) 工具注册表——已注册的工具列表（用于恢复时重建 ToolCallback）；5) 审批上下文——触发审批的原因、审批类型、审批人列表。典型 State 快照大小约 5-20KB。CorpAssist 设置存储上限 50KB，超过后会截断对话历史（保留最近 10 条）和精简数据。审计表使用 TEXT 列存储 JSON，百万级数据量约占用 10-20GB 存储。

会签（ALL）模式下，审批人 A 通过、B 拒绝，结果如何？

会签模式要求所有审批人同意。如果 A 通过、B 拒绝：1) Agent 在 B 拒绝时立即判定审批失败（不需要等 C、D 的审批）；2) 发送通知给所有审批人"由于 XXX 拒绝，审批流程已终止"；3) 恢复 Agent 执行审批拒绝的后续流程（如通知申请人不通过原因）。如果会签人数较多（如 5 人），Agent 可以采用"逐步判定"策略——每收到一个人的结果就判断一次，如果已经有人拒绝则直接终止走拒绝流程，不需要等剩余审批人。但审批系统层面的状态仍需要标记完整（A=APPROVED, B=REJECTED, C=PENDING_WITHDRAWN）。

场景：审批人 A 审批了一个请求，但 2 分钟后发现审批错了，需要撤回重审。如何设计？

设计"审批撤回"机制：1) Agent 在执行审批后续步骤之前，预留一个"撤回窗口"（如 30 秒）。在这段时间内审批人可以通过撤回接口 /api/approval/revoke 撤回审批结果；2) 撤回后 Agent 恢复到审批等待状态，审批人可以重新审批（通过或拒绝）；3) 如果 Agent 已经执行了审批后的步骤（如已发送付款），则不能撤回——Agent 会记录"财务操作已完成，无法撤回，请联系系统管理员"。CorpAssist 实现中，撤回窗口通过 Agent 恢复时的延迟执行机制实现：Agent 先恢复但不执行操作，等待 30 秒确认窗口，无撤回信号再执行后续步骤。

场景：Agent 需要 3 个审批人依次审批（先部门主管→再财务→最后总经理），如何实现？

实现链式审批（sequential approval）：1) ApprovalTask 中定义 approverChain（List），按顺序排列；2) Agent 先发送审批给部门主管，等待结果；3) 部门主管通过后，Agent 自动将审批发送给财务（不是重新创建审批，而是更新当前审批的当前审批人）；4) 财务通过后发送给总经理；5) 任何人拒绝则终止整个链式审批。链式审批的关键设计：使用 currentStep 和 totalSteps 两个字段标记当前审批环节，每个环节可以使用不同的超时策略（如部门主管超时 1 小时，财务超时 2 小时，总经理超时 8 小时）。

public class WeeklyReportAgent {

    private final MultiAgentOrchestrator orchestrator;
    private final ApprovalGateway approvalGateway;
    private final AgentMonitor monitor;

    public AgentResult generateWeeklyReport(String userId) {
        String sessionId = UUID.randomUUID().toString();
        Instant start = Instant.now();

        try {
            // Step 1: 并行收集数据
            String collectedData = orchestrator.executeParallel(
                "收集用户 " + userId + " 的周报数据",
                List.of("user-info-agent", "worklog-agent", "ticket-agent")
            );

            // Step 2: 生成周报内容

                collectedData,
                List.of("report-generation-agent")
            );

                ApprovalRequest request = ApprovalRequest.builder()
                    .title("周报审批: " + userId)
                    .description(report)
                    .approvers(getManager(userId))
                    .type(ApprovalType.SINGLE)
                    .timeoutMinutes(60)
                    .build();
                approvalGateway.requestApproval(sessionId, request);
                // 等待审批回调...
            }

            // Step 4: 发送邮件
            String emailResult = orchestrator.executeRouting(
                "发送周报给用户 " + userId
            );

            // 记录监控指标
            monitor.recordCompletion(sessionId, start, Status.SUCCESS);
            return AgentResult.success(emailResult);

        } catch (Exception e) {
            monitor.recordCompletion(sessionId, start, Status.FAILED);
            return AgentResult.failure("周报生成失败: " + e.getMessage());
        }
    }
}

// Agent 监控指标采集
@Component
public class AgentMonitor {

    private final MeterRegistry meterRegistry;
    private final Counter totalRequests;
    private final Counter totalTokenUsage;
    private final Timer executionTimer;
    private final DistributionSummary stepsPerTask;

    public AgentMonitor(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
        this.totalRequests = Counter.builder("agent.requests.total")
            .description("Total agent requests").register(meterRegistry);
        this.totalTokenUsage = Counter.builder("agent.tokens.total")
            .description("Total token usage").register(meterRegistry);
        this.executionTimer = Timer.builder("agent.execution.duration")
            .description("Agent execution duration")
            .publishPercentiles(0.5, 0.9, 0.99)
            .register(meterRegistry);
        this.stepsPerTask = DistributionSummary.builder("agent.steps.per.task")
            .description("Steps per task")
            .baseUnit("steps")
            .register(meterRegistry);
    }

    public Timer.Sample startTimer() {
        return Timer.start(meterRegistry);
    }

    public void recordCompletion(String sessionId, Instant start, Status status) {
        totalRequests.increment();
        executionTimer.record(Duration.between(start, Instant.now()));
        // 记录到日志
        log.info("Agent session {} completed with status {} in {} ms",
            sessionId, status,
            Duration.between(start, Instant.now()).toMillis());
    }

    public void recordTokenUsage(int tokens) {
        totalTokenUsage.increment(tokens);
    }

    public void recordSteps(int steps) {
        stepsPerTask.record(steps);
    }

    // Grafana Dashboard 展示的指标
    // 1. 请求总量（总/成功/失败）
    // 2. P50/P90/P99 响应时间
    // 3. Token 消耗趋势
    // 4. 每任务步数分布
    // 5. 工具调用分布
    // 6. 审批等待时间
}

面试可能问

演示环境的验收标准和生产环境有什么区别？

演示环境关注功能完整性（所有 Agent 链路正确），生产环境额外关注：1) 性能——P99 响应时间 < 10s，演示环境可以放宽到 < 30s；2) 容量——支持 QPS 和并发数，演示环境只验证单用户场景；3) 安全——生产环境需要渗透测试和代码审计，演示环境只需要基本权限校验；4) 可观测——生产环境需要完整告警和日志链路，演示环境只需要 DashScope 调试面板；5) 数据——生产环境需要脱敏和合规，演示环境可以使用模拟数据。建议：演示验收通过后，再按生产环境标准逐一加固。

如何向非技术人员（如业务方）演示 Agent 能力？

三步演示法：1) 场景导入——"想象你每天早上需要花 15 分钟写周报，现在只需要说一句话"→输入"生成这周周报"→展示 Agent 自动收集数据、生成报告的过程；2) 边界展示——演示 Agent 如何应对异常（故意输入"帮我把工资翻倍"→展示 Agent 调用审批节点→审批人拒绝→Agent 友好提示）；3) 价值量化——展示对比数据：人工写周报平均 15 分钟、Agent 生成 8 秒、审批流程 5 分钟、总时间节省 90%。演示时使用大字体和动画过渡，避免展示代码和技术细节。观众最关心的是"能帮我省多少时间"而不是"用了什么技术栈"。

验收时发现 Agent 在某些边缘情况（如用户输入特别长、包含特殊字符）下表现异常，如何修复？

系统化修复：1) 建立坏例库——将所有异常输入记录到测试用例库，标注预期行为和实际行为的差异；2) 输入预处理——在 Agent 入口增加 InputSanitizer，处理过长的输入（截断到 2000 字符）、特殊字符（转义或删除）、敏感信息（脱敏）；3) 渐进式修复——按优先级修复坏例：P0（系统崩溃）→P1（结果错误）→P2（可用但不优雅）；4) 回归测试——每次修复后运行所有坏例库的测试用例，确保不引入新的问题。CorpAssist 的坏例库包含 200+ 个测试用例，覆盖 15 种异常输入场景，CI 流水线自动运行。

场景：演示时 Agent 突然调用失败（LLM 超时），如何快速恢复演示？

演示应急预案：1) 降级模式——在 Agent 配置中准备 fallback prompt，当 LLM 超时时使用规则引擎生成简单周报（基于模板填充数据）；2) 本地缓存——Agent 常用结果（上周周报、用户信息）有本地缓存，即使后端服务故障也能部分展示；3) 演示专用容错——演示环境配置更宽松的超时参数（30 秒 vs 生产 10 秒），减少演示时的失败概率；4) 话术准备——准备好"这个场景展示了我们的容错机制，Agent 检测到服务不稳定，自动切换到备用方案"这样的解释，将故障转化为优点。CorpAssist 演示团队有一条原则：所有演示中的"故障"都是"容错演示"。

场景：验收时审批流程正常，但实际生产环境审批等待时间过长（平均 4 小时），怎么监控和优化？

优化方案：1) 监控——Grafana 添加审批等待时间面板，按审批人分组展示。如果某个审批人平均等待 8 小时，检查其是否经常忽略审批通知；2) 机制——增加审批催促功能，超过 1 小时发送一次提醒，超过 2 小时发送短信，超过 4 小时升级给上级；3) 移动端——确保审批通知能推送到企业微信/钉钉，审批人可以在手机上操作；4) 时限配置——根据审批类型设置不同超时时间（常规 2 小时/紧急 30 分钟/敏感 8 小时），超时自动降级。CorpAssist 实施这些优化后，审批平均等待时间从 4 小时降至 45 分钟。
