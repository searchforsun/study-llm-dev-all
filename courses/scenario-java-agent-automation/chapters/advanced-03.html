Q1: Agent vs 普通 LLM 应用的区别

考点：理解 Agent 的核心价值——工具调用和自主决策，而非对话。

普通 LLM 应用

一问一答。用户输入→LLM→输出。LLM 只知道自己的训练数据，不知道外部信息。如"查张三的工单状态"——LLM 会编造一个答案（幻觉），因为它无法访问工单系统。

Agent 应用

多步推理+工具调用。用户输入→LLM 思考→调用工具→观察结果→继续推理→输出。如"查张三的工单状态"——Agent 调用 queryTicket("张三")→获得真实数据→输出。

java
// 面试回答话术
//
// "我认为 Agent 和普通 LLM 应用的核心区别在于 '自主性'。
//
// 普通 LLM 应用是'被动问答'——用户给什么 Prompt，LLM 回答什么。
// 它没有能力去获取外部信息或执行操作。
//
// Agent 应用是'主动执行'——给定一个目标，Agent 自主决定
// 需要调用哪些工具、按什么顺序调用、如何处理工具返回的数据。
// 用 Spring AI 的实现来说：普通应用用 ChatClient 就够了，
// Agent 需要 ChatClient + ToolCallback + ReactAgent 三重组合。
//
// 在 CorpAssist 中，我们的经验是：简单问答（如'什么是 RAG'）
// 用普通 LLM 应用；需要操作外部系统的任务（如'查工单'）
// 用 Agent。前者适合知识问答，后者适合任务执行。
// 我们内部有一个决策规则：如果任务涉及调用外部 API 或
// 多步推理（超过 2 步），用 Agent；否则用普通 LLM。"
//
// Java 实现对应：
// 普通 LLM: chatClient.prompt().user(msg).call().content()
// Agent: reactAgent.execute(msg)
//

// 面试回答话术 - 架构设计
//
// "我设计的多层 Agent 架构包含 4 个层次：
//
// 1. 入口层——API Gateway 统一接收请求，做限流和鉴权。
// 用 Spring Cloud Gateway 实现，结合 Sentinel 做流量控制。
//
// 2. Agent 层——分三层：Routing Agent 分析请求类型；
// Specialized Agent 处理具体领域（日程/工单/邮件）；
// Executor 执行具体步骤。每层之间通过 Nacos 事件异步通信。
//
// 3. 工具层——所有工具通过 MCP 网关注册，网关负责
// 权限校验、调用审计、熔断保护。新增一个工具只要
// 实现 MCP 接口，不需要改 Agent 代码。
//
// 4. 基础设施层——Redis 存检查点和缓存，Kafka 做审批
// 异步消息，PostgreSQL 存审计日志，Nacos 做配置和
// 服务发现。
//
// 这个架构的设计原则是：解耦、可观测、可容错。
// 任何一层出问题都不会影响其他层。比如工具层全部宕机，
// Agent 层还能使用规则引擎降级响应。"

Q3: 任务规划

考点：Agent 如何将复杂任务拆解为可执行的步骤，以及如何应对规划中的不确定性。

java
// 面试回答话术 - 任务规划
//
// "任务规划是 Agent 能否完成复杂任务的关键。
//
// 我使用 Plan-and-Execute 模式——分为 Planner 和 Executor 两个阶段。
//
// Planner 的 System Prompt 中明确要求输出 JSON 格式的计划：
// [{"step":1,"action":"getUserInfo","params":...}]
// 这种结构化输出让我们可以静态校验计划的有效性。
//
// 实践中遇到的最大坑是'规划偏航'——Planner 输出的计划
// 在执行过程中发现某个步骤的结果和预期不一致（如查不到用户），
// 导致后续步骤无效。我们的解决方案是'动态重规划'：
// Executor 每一步执行后检查结果，如果结果和预期偏差超过阈值，
// 重新调用 Planner 调整未执行的步骤。
//
// 举个例子：Agent 计划 ['查用户'->'查工单'->'发提醒']，
// 但查用户时发现用户已离职。此时不会继续执行查工单，
// 而是调用 Planner 重新规划，新的计划可能是：
// ['查用户'->'返回离职状态提示'][步骤 2-3 被跳过]。
//
// 这个机制在 CorpAssist 中覆盖了约 15% 的异常场景。
// 核心实现是基于步骤的 dependsOn 依赖检查和动态
// planUpdate 回调。"

Q4: 记忆管理

考点：Agent 如何管理短期对话记忆和长期用户偏好记忆，以及如何控制 Token 消耗。

java
// 面试回答话术 - 记忆管理
//
// "我设计的是四层记忆体系：
//
// 1. 短期记忆——Spring AI ChatMemory 的 Summarize 模式。
// 每 5 轮对话触发一次摘要压缩，将最近 10 条消息压缩为
// 200 Tokens 的摘要，节省约 60% Token。关键信息（用户身份、
// 任务目标、已完成的步骤）通过 Prompt 约束确保不被压缩掉。
//
// 2. 长期记忆——Milvus 向量数据库存储用户偏好和历史决策。
// 用户登录时自动加载最近 5 条相关偏好。命中率约 70%。
//
// 3. 任务记忆——每次 Agent 执行完成后将关键参数和结果
// 存入向量库。下次用户的"按上次方式处理"可以直接检索。
//
// 4. 审计记忆——PostgreSQL 存储操作日志，用于审计追溯。
// 记录所有工具调用、审批操作、降级事件。
//
// 记忆管理最深的教训是 Token 控制。早期我们用的是
// InMemory 策略（全部保留），10 轮对话后 Token 超过 5000。
// 切换到 Summarize 后，同样 10 轮对话只需要 800 Token。
// 我们还做了'上下文压缩适配器'，把工具返回的大数据
// 截断到 500 字符，进一步减少 Token 消耗。
//
// 最终效果：Token 消耗降低 60%，但关键信息保留率 > 95%。"

Q5: 工具失败处理

考点：Agent 调用外部工具时遇到各种失败情况，如何保证系统的可靠性和用户体验。

java
// 面试回答话术 - 工具失败处理
//
// "工具失败是 Agent 生产环境最常见的故障。
// 我设计了三层容错：
//
// 1. 重试层——使用 Resilience4j Retry 配置指数退避重试。
// 网络超时和 5xx 错误重试 3 次（间隔 1s/2s/4s），
// 参数错误（4xx）不重试。重试次数放在配置中心（Nacos），
// 可以动态调整。
//
// 2. 熔断层——每个工具独立的 Circuit Breaker。
// 20 次调用中 40% 失败则熔断 30 秒。熔断期间直接返回
// 降级结果"XX 服务暂时不可用"，不消耗实际 API 调用。
// 熔断恢复后自动切回正常模式。
//
// 3. 任务层容错——根据步骤的 critical 属性决定：
// 非关键步骤失败（如查天气），跳过并继续执行；
// 关键步骤失败（如创建付款单），触发回滚和补偿操作。
//
// 实践中印象最深的一次：工单系统升级导致 30 分钟内
// queryTicket 全部超时。熔断器在 40 秒内触发 OPEN，
// Agent 自动降级返回"工单查询暂时不可用，已记录您的请求"，
// 同时创建后台重试任务。10 分钟后工单系统恢复，熔断器
// HALF_OPEN 探测成功，自动恢复正常。整个过程没有引发
// 任何 P0 告警，用户感知到的只是几秒钟的延迟。
//
// 这个案例说明：容错设计不是防止故障，而是故障发生时
// 系统依然可用的能力。我们的目标是让用户感觉不到
// 后端服务的变化。"

本章结论

  • Q1 Agent vs LLM：核心在"自主性"——Agent 能自主决策调用工具完成任务，普通 LLM 只能被动问答
  • Q2 架构：四层架构（入口→Agent→工具→基础设施），每层独立可容错，通过 Nacos 事件解耦
  • Q3 规划：Plan-and-Execute + 动态重规划解决"规划偏航"，覆盖 15% 的异常场景
  • Q4 记忆：四层记忆体系，Summarize 压缩节省 60% Token，长期向量记忆命中率 70%
  • Q5 工具失败：三层容错（重试/熔断/任务级）+ 真实案例展示工程落地能力

面试可能问

你的 Agent 系统 QPS 能到多少？怎么测试的？

基准测试结果：单节点 8C16G 部署，QPS = 50（不考虑 LLM 延迟，因为 LLM 调用走 API 不占用本地 CPU）。如果考虑 LLM 延迟（平均 2 秒），实际吞吐 = 50/2 = 25 QPS。测试方法：使用 JMeter 模拟 100 个并发用户，每个用户执行"查工单"场景（1 次工具调用）。观察指标：平均响应时间、P99 响应时间、错误率、CPU/内存使用率。极限测试结果：200 并发时 P99 从 3.2s 升至 8.7s，错误率开始上升。扩容到 4 节点后恢复。瓶颈分析：主要瓶颈在 Redis 检查点读写（约 5ms/次）和 MCP 工具调用（约 50ms/次）。优化后将检查点改为批量写入（每 3 步写一次而非每步），MCP 调用增加连接池。

Agent 的 System Prompt 你是怎么设计的？

三层 Prompt 设计：1) 基础 Prompt——固定不变的角色定义和行为边界，如"你是 CorpAssist 企业办公助手，有以下工具可用..."。代码中硬编码，通过 @Value 或配置中心管理。2) 场景 Prompt——根据当前场景动态注入，如"当前是周报生成场景，需要先收集用户工时数据..."。由 Planner 在规划阶段生成。3) 工具 Prompt——工具调用时动态生成，包含工具的描述和参数说明。由 @Tool 注解的 description 自动生成。设计原则：基础 Prompt 占 60%（稳定），场景 Prompt 占 25%（动态），工具 Prompt 占 15%（自动生成）。总长度控制在 2000 Tokens 以内。使用 Prompt Template 模式管理，key 存储在 Nacos 配置中心，value 存储在 Git 仓库中版本管理——每次 Prompt 修改都有记录，可以回滚。

你的 Agent 系统如何做单元测试？

四层测试策略：1) 节点测试——每个 @Tool 方法和每个 Graph 节点函数独立单元测试。Mock 外部依赖，测试正常路径和异常路径。2) Agent 测试——Mock LLM 返回（固定返回特定的 tool_calls 和 final answer），测试 Agent 是否按预期调用工具和处理结果。3) 集成测试——使用 Testcontainers 启动 Redis + Kafka + PostgreSQL，测试完整工作流。LLM 用 WireMock 模拟。4) E2E 测试——连接真实的 LLM API，在测试环境（独立的 API Key）中执行端到端测试。难度最大的是 Agent 测试，因为 LLM 返回的不确定性。我们的办法：用录制回放工具（如 VCR）录制 LLM 的真实响应，在 CI 中回放。回归测试覆盖 200+ 个测试用例，CI 流水线每次提交运行测试，平均耗时 8 分钟。

场景：面试官说"你这些数据（24 万美元节省）太夸张了，我怀疑真实性"

从容应对：1) 先认同——"确实，如果不看计算过程，24 万美元听起来很大。" 2) 分步解释——"按 500 人企业计算：人均时薪 100 元，每天每个场景节省的时间不一样。周报场景从 15 分钟到 8 秒，节省 14.9 分钟；工单场景从 3 分钟到 15 秒，节省 2.75 分钟。我把 10 个场景的节省时间加起来得到每天总节省，乘以 250 个工作日的 800 元日均薪。" 3) 提供验证方式——"如果你有兴趣，我可以把 Excel 计算表发给你，或者我们约一个时间我用真实系统演示给你看。最有力的是我们上线前的 AO 测试数据——300 名员工测试 2 周，平均每人每天节省 37 分钟。" 坦诚的关键：数据来源于 AO 测试而非估算，有具体的时间记录支撑。

场景：面试官问"你提到了 MCP，能说说它和 OpenAPI/Swagger 的区别吗？"

MCP vs OpenAPI：1) 设计目标不同——OpenAPI 是给"人"看的 API 文档（面向开发者），MCP 是给"AI"看的工具描述（面向 LLM）；2) 参数描述——OpenAPI 用 required/type/format 描述参数（适合代码生成），MCP 用 description/examples 描述参数（适合 LLM 理解用途）；3) 调用方式——OpenAPI 需要开发者写 HTTP 调用代码，MCP 的每一次调用是"LLM 自主决定"的（System Prompt 中给 LLM 描述，LLM 决定何时调用）；4) 动态性——MCP 支持工具自动发现（Agent 启动时 listTools），OpenAPI 需要开发者在代码中手动注册。如果你用过 Swagger Codegen，它生成的是开发者调用的代码；MCP 生成的是 LLM 理解的工具描述。两者的结合方式：内部我们先用 OpenAPI 定义接口，然后通过一个适配器自动生成 MCP 的工具描述。
