为什么学这章

当 Agent 流程复杂到需要条件分支、并行执行、显式状态管理时，ReAct 的简单循环就不够用了。Spring AI Alibaba 的 Graph 模块提供了类似 LangGraph 的有向图工作流引擎，支持节点（@Bean 定义）、边（条件/无条件）和状态持久化。

学完后能做什么

用 Spring AI Alibaba Graph 的 @Bean 节点定义工作流；实现条件边根据状态路由到不同分支；使用 Admin Studio 可视化调试；实现状态序列化和反序列化支持断点恢复。

本章要点速记

1) Graph 工作流三个核心概念：State（共享状态）、Node（处理单元）、Edge（连接关系）；2) 节点用 @Bean 标注方法，接收 State 返回 State；3) 条件边通过 router-function 根据 State 中的字段值决定下一个节点；4) Admin Studio 实时显示 Graph 执行状态、节点耗时、State 变化；5) State 序列化使用 Jackson，支持 JSON 持久化到 Redis。

Graph 节点定义（@Bean）

// Graph 状态定义
@Data
@Builder
public class WorkflowState {
    private String sessionId;
    private String userId;
    private String userName;
    private boolean manager;
    private List<Ticket> tickets;
    private String reportContent;
    private boolean needApproval;
    private String approvalResult;
    private String emailResult;
    private List<String> errors;
}

// Graph 节点定义
@Configuration
public class WorkflowNodes {

    @Bean
    public Function<WorkflowState, WorkflowState> getUserInfo() {
        return state -> {
            // 查询用户信息（模拟）
            UserInfo user = userService.getUser(state.getUserId());
            state.setUserName(user.getName());
            state.setManager("MANAGER".equals(user.getRole()));
            return state;
        };
    }

    @Bean
    public Function<WorkflowState, WorkflowState> queryTeamTickets() {
        return state -> {
            List<Ticket> tickets = ticketService
                .queryByTeam(state.getUserId());
            state.setTickets(tickets);
            state.setNeedApproval(tickets.size() > 5);
            return state;
        };
    }

    @Bean
    public Function<WorkflowState, WorkflowState> queryMyTickets() {
        return state -> {
            List<Ticket> tickets = ticketService
                .queryByUser(state.getUserId());
            state.setTickets(tickets);
            state.setNeedApproval(false);
            return state;
        };
    }
}

条件边：根据状态路由

条件边是 Graph 工作流的核心灵活性——不同状态数据可以路由到不同的处理节点。Spring AI Alibaba Graph 通过 router-function 实现条件路由。

java
// Graph 工作流配置（条件路由）
@Configuration
public class WorkflowConfig {

    @Bean
    public Graph<WorkflowState> reportWorkflow() {
        return Graph.builder()
            .stateClass(WorkflowState.class)
            // 注册节点
            .node("getUserInfo", "getUserInfo()")  // @Bean 方法名
            .node("queryTeamTickets", "queryTeamTickets()")
            .node("queryMyTickets", "queryMyTickets()")
            .node("generateReport", "generateReport()")
            .node("submitApproval", "submitApproval()")
            .node("sendEmail", "sendEmail()")
            .node("notifyUser", "notifyUser()")
            // 定义边
            .edge("START", "getUserInfo")
            // 条件边：根据 isManager 路由
            .condition("getUserInfo", state ->
                state.isManager() ? "queryTeamTickets" : "queryMyTickets")
            .edge("queryTeamTickets", "generateReport")
            .edge("queryMyTickets", "generateReport")
            // 条件边：根据 needApproval 路由
            .condition("generateReport", state ->
                state.isNeedApproval() ? "submitApproval" : "sendEmail")
            .edge("submitApproval", "sendEmail")
            .edge("notifyUser", "END")
            .build();
    }
}

// 多条件分支查询示例
// condition() 方法支持复杂逻辑表达式
.condition("reviewNode", state -> {
    if (state.getErrors().size() > 3) return "rejectNode";
    if (state.getErrors().size() > 0) return "fixNode";
    return "approveNode";
})

CorpAssist 场景

运维人员在 Admin Studio 中看到：一个"生成部门周报"的工作流有 7 个节点，其中 queryTeamTickets 耗时最高（1.2 秒），因为工单系统返回慢。通过可视化界面，运维发现条件路由 isManager=true 的路径走了 80% 的执行，说明经理用户是主要使用者。基于这个数据，运维优化了 queryTeamTickets 节点的缓存策略，将平均耗时从 1.2 秒降至 200 毫秒。

状态序列化与恢复

工作流的持久化和恢复依赖于 State 对象的序列化。Spring AI Alibaba Graph 使用 Jackson 序列化 State，支持存储到 Redis 或数据库。

java
// 状态序列化与持久化
@Component
public class GraphStatePersistence {

    private final RedisTemplate<String, String> redisTemplate;
    private final ObjectMapper objectMapper;
    private static final String PREFIX = "graph:state:";

    public void saveState(String workflowId, WorkflowState state) {
        try {
            String json = objectMapper.writeValueAsString(state);
            redisTemplate.opsForValue().set(
                PREFIX + workflowId, json,
                Duration.ofHours(24)  // TTL 24 小时
            );
        } catch (JsonProcessingException e) {
            throw new RuntimeException("State 序列化失败", e);
        }
    }

    public WorkflowState loadState(String workflowId) {
        String json = redisTemplate.opsForValue().get(PREFIX + workflowId);
        if (json == null) return null;
        try {
            return objectMapper.readValue(json, WorkflowState.class);
        } catch (JsonProcessingException e) {
            throw new RuntimeException("State 反序列化失败", e);
        }
    }

    public void deleteState(String workflowId) {
        redisTemplate.delete(PREFIX + workflowId);
    }
}

// Graph 执行器（支持恢复）
@Service
public class GraphExecutor {

    public void executeWithRecovery(String workflowId, String userId) {
        // 尝试从上次中断处恢复
        WorkflowState state = persistence.loadState(workflowId);
        if (state == null) {
            state = WorkflowState.builder()
                .sessionId(workflowId).userId(userId).build();
        }
        // 继续执行未完成的节点
        graph.execute(state, workflowId);
    }
}

无持久化恢复

工作流执行到一半系统崩溃，所有进度丢失。用户需要重新发起请求，Agent 重新执行所有节点（包括已完成的），浪费 Token 和时间。

有持久化恢复

工作流每完成一个节点就持久化 State。系统崩溃恢复后，从 Redis 加载 State，直接从中断的节点继续执行。State 包含已完成节点列表和中间结果，不重复执行已完成的节点。

面试可能问

Graph 和 ReAct 在状态管理上的核心区别是什么？

ReAct 的状态管理是隐式的——状态存在于对话历史（Message 列表）中，Agent 通过 ChatMemory 获得上下文。Graph 的状态管理是显式的——开发者定义 WorkflowState 类，明确所有状态字段，节点函数显式读写这些字段。核心区别：1) 可预测性——Graph 的 State 结构固定，执行路径确定；2) 可持久化——Graph 的 State 可序列化为 JSON 存储；3) 可测试——Graph 的每个节点可以独立单元测试；4) 调试难度——Graph 的 State 变化可追踪，ReAct 的中间推理过程不透明。CorpAssist 实践中：需要审计追溯的流程（如财务审批）用 Graph，快速问答用 ReAct。

Graph 工作流的 State 太大导致序列化慢怎么办？

优化策略：1) 状态裁剪——State 只保存必需的字段，大型数据（如完整的工单列表）只保存 ID 和摘要；2) 增量序列化——State 保存"增量"而非"全量"，每次只保存相比上一次 checkpoint 的变化部分；3) 分片存储——将 State 分为元数据（userInfo, status）和业务数据（tickets, report）两部分，元数据用 Redis Hash，业务数据用单独的 Key；4) 异步序列化——序列化操作异步执行，不影响主流程。CorpAssist 使用方案 1+3，典型 State 大小控制在 10KB 以内，序列化时间 <5ms。

Graph 工作流出现循环（loop）如何防止死循环？

三种防护：1) maxLoops 配置——每条 LOOP 类型的边配置最大循环次数（如 maxLoops=5）；2) State 中维护 loopCount——每次经过循环节点时 loopCount+1，到达阈值后自动退出循环；3) 循环检测——Graph 引擎内置检测器，如果同一节点连续执行超过 3 次且 State 无有效变化，判定为死循环并终止。CorpAssist 使用方案 1+2，在"审批驳回重新提交"循环中，最多允许 3 次驳回重提，超过后转人工处理。

场景：queryTeamTickets 节点耗时 3 秒，导致整体工作流超时，如何优化？

优化方案：1) 加入 Redis 缓存——工单数据缓存 1 分钟，大多数用户看到的是 1 分钟内创建的数据；2) 异步执行——将 queryTeamTickets 改为异步节点（返回 CompletableFuture），与 generateReport 的初始化步骤并行执行；3) 超时降级——如果 queryTeamTickets 超过 2 秒，改为查询缓存数据（即使稍旧）+ 标记"数据可能不是最新"；4) 预加载——Graph 在 getUserInfo 节点完成后立即预加载工单数据。CorpAssist 使用方案 1+3，缓存命中时 queryTeamTickets 耗时从 3 秒降至 50 毫秒。

场景：submitApproval 节点的审批流程需要 2 小时，Graph 应该等吗？

不应该同步等待。解决方案：1) 在 submitApproval 节点完成后，Graph 将状态持久化并标记为 WAITING_APPROVAL；2) Graph 工作流暂停执行，释放资源；3) 审批系统通过回调接口（如 POST /workflow/{id}/resume?result=approve）通知 Graph 引擎审批完成；4) Graph 引擎从持久化存储恢复 State，继续执行后续节点。这个过程通过 Webhook 或 MQ 消息实现异步解耦。CorpAssist 使用 RocketMQ 消息队列监听审批结果，典型的审批等待时间从 5 分钟到 24 小时不等。
