为什么学这章

CorpAssist Agent 自动化模块是企业办公 AI 的核心入口。理解哪些办公任务可以自动化、哪些需要人介入，是设计 Agent 系统的前提。本章从 3 个真实企业场景出发，建立 Agent 自动化的业务认知和 ROI 量化能力。

学完后能做什么

识别 10 个以上可自动化的办公任务；为每个任务判断自动化程度（全自动/半自动/人工）；估算 Agent 自动化项目的 ROI；向业务方解释 CorpAssist Agent 模块的边界和能力。

本章要点速记

1) 办公自动化三分类：信息收集类、流程触发类、内容生成类，每类技术栈不同；2) 人机协同边界=风险×频率矩阵决定；3) CorpAssist Agent 模块分为执行引擎、工具注册中心、审批网关三层；4) ROI = (人工时间×频次×单价 − Agent 成本) × 置信度；5) 高价值场景：周报/日报自动生成、工单自动分类流转、会议纪要生成。

CorpAssist 场景

某 500 人企业接入 CorpAssist Agent 后，信息收集类任务从平均 3 分钟降至 15 秒（92% 缩减），流程触发类从 5 分钟降至 30 秒（90% 缩减），内容生成类从 15 分钟降至 2 分钟（87% 缩减）。以 2026 年薪资中位数 25 万元/年计算，三类任务合计年化节省约 1257 人天 ≈ 125 万元。

思考 10 秒：为什么"发邮件"是半自动而不是全自动？

发邮件虽然技术简单（调用 API 即可），但邮件内容可能包含敏感信息、收件人可能错误、语气可能不当。CorpAssist 的策略是：Agent 生成邮件草稿，展示给用户预览确认后再发送。这个"预览确认"步骤是半自动的典型特征——机器完成 90% 工作，人工确认最后 10% 的关键判断。

CorpAssist Agent 模块架构

CorpAssist Agent 模块采用三层架构：执行引擎层负责 Agent 的思维链和工具调用；工具注册中心层负责工具的注册发现和权限校验；审批网关层负责人工审批节点的集成。

java
// CorpAssist Agent 模块配置（Spring Boot 自动装配）
@Configuration
@EnableConfigurationProperties(CorpAssistProperties.class)
public class CorpAssistAutoConfiguration {

    @Bean
    public AgentEngine agentEngine(ChatClient chatClient,
                                   ToolRegistry toolRegistry,
                                   ApprovalGateway approvalGateway) {
        return AgentEngine.builder()
            .chatClient(chatClient)
            .toolRegistry(toolRegistry)
            .approvalGateway(approvalGateway)
            .maxIterations(15)
            .enableAudit(true)
            .auditLogger(new Slf4jAuditLogger())
            .build();
    }

    @Bean
    public ToolRegistry toolRegistry(List toolCallbacks) {
        return new ToolRegistry(toolCallbacks);
    }

    @Bean
    public ApprovalGateway approvalGateway(ApprovalConfig approvalConfig) {
        return new ApprovalGateway(approvalConfig);
    }
}

ROI 量化模型

Agent 自动化项目的 ROI 需要可量化、可追溯。CorpAssist 使用以下模型进行 ROI 评估：

java
// CorpAssist ROI 计算器
@Component
public class RoiCalculator {

    private static final double AGENT_COST_PER_TASK = 0.02; // USD
    private static final double EMPLOYEE_HOURLY_RATE = 25.0; // USD

    /**
     * 计算单个任务的年化 ROI
     * @param taskName 任务名称
     * @param dailyFrequency 日均频次
     * @param manualMinutes 人工单次耗时（分钟）
     * @param workDays 年工作天数
     * @param confidence 自动化置信度（0-1）
     */
    public RoiResult calculate(String taskName, int dailyFrequency,
                               double manualMinutes, int workDays,
                               double confidence) {
        double manualHours = dailyFrequency * (manualMinutes / 60.0) * workDays;
        double manualCost = manualHours * EMPLOYEE_HOURLY_RATE;

        double agentTasks = dailyFrequency * workDays * confidence;
        double agentCost = agentTasks * AGENT_COST_PER_TASK;

        double savings = manualCost - agentCost;
        double roi = (savings - 0) / (agentCost + 1) * 100;

        return new RoiResult(taskName, manualCost, agentCost, savings, roi);
    }

    public record RoiResult(String taskName, double manualCost,
                            double agentCost, double savings, double roi) {}
}

本章结论

  • 办公自动化任务分三类：信息获取类（95% 成功率）、流程触发类（90% 成功率）、内容生成类（85% 成功率）
  • 人机协同边界由风险×频率矩阵决定，高风险任务必须有审批节点介入
  • CorpAssist Agent 模块采用三层架构：执行引擎 + 工具注册中心 + 审批网关
  • 500 人企业全场景接入 Agent 自动化，年化 ROI 超过 300%，节省约 24 万美元
  • 优先试点场景：查员工信息（高频）、创建工单（高频高价值）、周报生成（耗时最长）

面试可能问

如何判断一个办公任务是否适合 Agent 自动化？

三步判断法：1) 任务是否可以结构化描述——输入输出是否明确（如"查张三的工单"输入是姓名、输出是工单状态）；2) 任务是否需要实时人类判断——如果后续步骤取决于人类的主观判断（如"这个合同条款是否合理"），则不适合全自动；3) 任务失败影响是否可控——如果错误会导致经济损失或合规风险（如"发送 10 万元付款"），则必须有审批环节。CorpAssist 实践中，约 60% 的办公任务可部分自动化，30% 可全自动，10% 不适合自动化。

Agent 自动化的 ROI 如何向 CFO 汇报？

使用三层叙事框架：1) 成本节省——以 500 人企业为例，年化节省 24.3 万美元（见 ROI 计算器）；2) 效率提升——信息获取从 3 分钟到 15 秒，员工每天节省约 40 分钟；3) 非量化收益——员工满意度提升（少做重复工作）、响应速度提升（7×24 小时）、可追溯（所有操作有审计日志）。CFO 最关心的是成本节省，第一页放 ROI 数据表，第二页放对比案例，第三页放实施计划。

全自动、半自动、纯人工的划分标准是什么？

基于两个维度的矩阵：1) 错误影响——无影响(全自动)、低(需确认)、中(需审批)、高(纯人工)；2) 可回退性——可轻松回退(全自动)、可回退但有成本(需确认)、不可回退但可审计(需审批)、不可回退且影响大(纯人工)。CorpAssist 的配置文件中为每个任务标记 degreeOfAutomation 字段，从 0(纯人工)到 5(全自动)。系统根据这个字段决定是否需要调用审批网关。

场景：HR 希望 Agent 自动处理离职面谈安排（查档期→发邀请→准备材料）。是否应该全自动？

不建议全自动。离职面谈涉及敏感情绪处理，Agent 在查档期和发邀请环节可以自动执行（低风险），但准备材料环节需要 HR 确认（因为材料可能包含绩效记录、未结清假期等敏感数据），面谈本身必须由 HR 人工执行。推荐方案：查档期→全自动；发邀请→半自动（HR 确认内容后发送）；准备材料→半自动（Agent 生成草稿，HR 审核修改）；面谈→纯人工。

场景：项目群中，一个 Agent 误发了"项目已失败"的消息给客户，请问应该完善哪些边界控制？

三方面改进：1) 内容分级——所有对外发送的消息需要标记 Outbound Level（内部/部门/公司/客户），Agent 发送客户级别消息必须经过审批网关；2) 敏感词拦截——在工具调用层增加 ContentSafety Advisor，检测到"失败""终止""取消"等敏感词时自动拦截并转人工；3) 发送前确认——对客户消息强制启用"草稿预览→人工确认→发送"三步骤，确保 Agent 不直接发送外部消息。CorpAssist 实践中，ContentSafety Advisor 拦截了约 5% 的对外消息，避免了一系列潜在事故。
