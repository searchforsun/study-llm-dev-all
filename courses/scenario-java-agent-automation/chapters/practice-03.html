为什么学这章

单个 Agent 的能力有限，复杂的办公任务需要多个专业 Agent 协作完成。Spring AI Alibaba 提供了三种多 Agent 协作模式：Sequential（流水线）、Parallel（并行处理）、Routing（按需路由）。理解这些模式是构建企业级 Agent 系统的核心能力。

学完后能做什么

实现 Sequential/Parallel/Routing 三种多 Agent 协作模式；使用 Nacos 事件作为 Agent 间的消息总线；实现 Agent Handoff 交接；设计冲突检测和解决方案。

本章要点速记

1) Sequential Agent：上一个 Agent 的输出是下一个 Agent 的输入，适合流水线处理；2) Parallel Agent：多个 Agent 独立处理不同维度，最后汇总；3) Routing Agent：根据任务类型路由到不同的专业 Agent；4) Handoff 交接通过共享的 Context 对象实现，包含状态和会话历史；5) 冲突检测使用向量相似度对比和规则引擎两种方式。

// 多 Agent 协作框架
@Component
public class MultiAgentOrchestrator {

    @Autowired
    private List<SpecializedAgent> agents;

    /**
     * Sequential 模式
     */
    public String executeSequential(String input, List<String> agentIds) {
        String context = input;
        for (String id : agentIds) {
            SpecializedAgent agent = findAgent(id);
            context = agent.process(context);
        }
        return context;
    }

    /**
     * Parallel 模式
     */
    public String executeParallel(String input, List<String> agentIds) {
        Map<String, CompletableFuture<String>> futures = new HashMap<>();
        for (String id : agentIds) {
            SpecializedAgent agent = findAgent(id);
            futures.put(id, CompletableFuture.supplyAsync(
                () -> agent.process(input)));
        }
        // 等待所有 Agent 完成
        StringBuilder result = new StringBuilder();
        futures.forEach((id, future) -> {
            try {
                result.append("【").append(id).append("】")
                    .append(future.get()).append("\n");
            } catch (Exception e) {
                result.append("【").append(id).append("】失败: ")
                    .append(e.getMessage()).append("\n");
            }
        });
        return result.toString();
    }

    /**
     * Routing 模式
     */
    public String executeRouting(String input) {
        // 路由 Agent 根据输入决定交给哪个 Agent
        String routingDecision = routingAgent.analyze(input);
        SpecializedAgent target = findAgent(routingDecision);
        return target.process(input);
    }

    private SpecializedAgent findAgent(String id) {
        return agents.stream()
            .filter(a -> a.getAgentId().equals(id))
            .findFirst()
            .orElseThrow(() -> new IllegalArgumentException("Agent not found: " + id));
    }
}

消息总线（Nacos 事件）

在多 Agent 系统中，Agent 间需要异步通信。CorpAssist 使用 Nacos 事件机制实现 Agent 间的消息发布和订阅。

java
// Nacos 事件驱动的 Agent 消息总线
@Component
public class AgentEventBus {

    private final NacosEventPublisher eventPublisher;
    private final Map<String, List<AgentEventHandler>> handlers = new ConcurrentHashMap<>();

    public AgentEventBus(NacosEventPublisher eventPublisher) {
        this.eventPublisher = eventPublisher;
    }

    public void publish(AgentEvent event) {
        // 发布事件到 Nacos
        eventPublisher.publishEvent(
            "agent:event:" + event.getTargetAgent(),
            event
        );
    }

    @EventListener
    public void handleEvent(AgentEvent event) {
        String target = event.getTargetAgent();
        List<AgentEventHandler> agentHandlers = handlers.get(target);
        if (agentHandlers != null) {
            agentHandlers.forEach(h -> h.onEvent(event));
        }
    }

    public void subscribe(String agentId, AgentEventHandler handler) {
        handlers.computeIfAbsent(agentId, k -> new CopyOnWriteArrayList<>())
            .add(handler);
    }
}

// Agent 事件定义
@Data
public class AgentEvent {
    private String eventId;
    private String sourceAgent;
    private String targetAgent;
    private String eventType;  // TASK_ASSIGN, TASK_COMPLETE, TASK_FAILED, HANDOFF
    private String payload;    // JSON 数据
    private Instant timestamp;
}

CorpAssist 场景

用户请求"帮我安排一个明天下午 2 点的项目评审会"。路由 Agent 分析后交给 schedule-agent 处理，schedule-agent 查空闲时间后通过 Nacos 事件发送 TASK_ASSIGN 给 meeting-agent 创建会议，meeting-agent 完成后发送 TASK_COMPLETE 给 email-agent 发送邀请通知，email-agent 发送完成后发送 TASK_COMPLETE 给汇总 Agent 生成最终结果。整个过程通过事件驱动，每个 Agent 异步处理，总耗时约 3 秒。

Handoff 交接实现

Agent Handoff 是指将一个任务从当前 Agent 转移给另一个 Agent 的过程。交接需要传递上下文（Context）和会话状态。

java
// Agent Handoff 上下文
@Data
@Builder
public class HandoffContext {
    private String sessionId;
    private String userId;
    private String taskId;
    private String sourceAgent;
    private String targetAgent;
    private Map<String, Object> collectedData;  // 已收集的数据
    private List<Message> conversationHistory;  // 对话历史
    private String handoffReason;               // 交接原因
    private Instant handoffAt;
}

// Handoff 管理器
@Component
public class AgentHandoffManager {

    private final AgentEventBus eventBus;
    private final RedisTemplate<String, Object> redisTemplate;

    public void handoff(HandoffContext context) {
        // 1. 持久化上下文到 Redis
        String key = "handoff:" + context.getSessionId();
        redisTemplate.opsForValue().set(key, context, Duration.ofHours(1));

        // 2. 发送 Handoff 事件
        AgentEvent event = new AgentEvent();
        event.setEventId(UUID.randomUUID().toString());
        event.setSourceAgent(context.getSourceAgent());
        event.setTargetAgent(context.getTargetAgent());
        event.setEventType("HANDOFF");
        event.setPayload(context.getSessionId());
        eventBus.publish(event);

        // 3. 源 Agent 清理（释放资源）
        cleanupSourceAgent(context);
    }

    public HandoffContext acceptHandoff(String sessionId) {
        String key = "handoff:" + sessionId;
        HandoffContext context = (HandoffContext) redisTemplate
            .opsForValue().get(key);
        if (context == null) {
            throw new HandoffException("Handoff context not found: " + sessionId);
        }
        return context;
    }
}

冲突检测

多 Agent 并行执行时可能出现冲突：两个 Agent 同时修改同一条数据、一个 Agent 的操作被另一个 Agent 覆盖。CorpAssist 使用乐观锁和向量对比两种方式检测冲突。

java
// 冲突检测器
@Component
public class ConflictDetector {

    @Value("${app.conflict.similarity-threshold:0.7}")
    private double similarityThreshold;

    /**
     * 检测两个 Agent 输出的决策是否存在冲突
     */
    public ConflictResult detectConflict(String sessionId,
                                          AgentOutput outputA,
                                          AgentOutput outputB) {
        // 1. 检查数据修改冲突
        if (outputA.getTargetResource() != null
            && outputA.getTargetResource().equals(outputB.getTargetResource())
            && outputA.getVersion() != outputB.getVersion()) {
            return ConflictResult.conflict("DATA_VERSION",
                "数据版本不一致: A=" + outputA.getVersion()
                    + ", B=" + outputB.getVersion());
        }

        // 2. 检查时间冲突
        if (outputA.getTimeRange() != null && outputB.getTimeRange() != null) {
            if (isTimeOverlapping(outputA.getTimeRange(), outputB.getTimeRange())) {
                return ConflictResult.conflict("TIME_OVERLAP",
                    "时间范围重叠: " + outputA.getTimeRange()
                        + " vs " + outputB.getTimeRange());
            }
        }

        // 3. 检查决策矛盾（使用向量相似度）
        double similarity = cosineSimilarity(
            outputA.getDecisionEmbedding(),
            outputB.getDecisionEmbedding()
        );
        if (similarity < similarityThreshold) {
            return ConflictResult.conflict("DECISION_CONFLICT",
                "决策差异大: 相似度=" + similarity);
        }

        return ConflictResult.ok();
    }

    private double cosineSimilarity(float[] a, float[] b) {
        double dot = 0, normA = 0, normB = 0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            normA += a[i] * a[i];
            normB += b[i] * b[i];
        }
        return dot / (Math.sqrt(normA) * Math.sqrt(normB));
    }
}

本章结论

  • 三种多 Agent 协作模式：Sequential（流水线）、Parallel（并行汇总）、Routing（按需路由），覆盖不同业务需求
  • Nacos 事件作为 Agent 间消息总线，支持 TASK_ASSIGN/TASK_COMPLETE/HANDOFF 等事件类型
  • Agent Handoff 通过 Redis 持久化上下文实现，交接后源 Agent 自动清理资源
  • 冲突检测涵盖数据版本、时间范围、决策一致性三类，使用余弦相似度对比决策冲突
  • CorpAssist 使用 Routing + Sequential 混合模式：先路由到专业 Agent，再在 Agent 内串行执行多步

面试可能问

Routing Agent 如何判断输入应该交给哪个子 Agent？

Routing Agent 本身是一个 LLM 调用，System Prompt 中定义路由规则。示例 Prompt："分析用户输入，判断应该交给哪个 Agent 处理。可选 Agent：1) schedule-agent——处理日程查询和会议安排；2) ticket-agent——处理工单创建和查询；3) email-agent——处理邮件发送和查询。输出格式：{"agent":"schedule-agent","confidence":0.95,"reason":"用户请求查日程"}。" Routing Agent 返回的 confidence 如果低于阈值（如 0.7），则触发"澄清模式"反问用户。CorpAssist 还使用 fallback 路由：如果 Routing Agent 的置信度低，将请求同时发送给所有候选 Agent，取第一个返回结果的。

Handoff 过程中如何保证数据不丢失？

使用两阶段提交（2PC）风格的交接协议：1) Prepare 阶段——源 Agent 将 HandoffContext 持久化到 Redis，状态标记为 HANDOFF_PREPARED；2) Commit 阶段——源 Agent 发送 HANDOFF 事件到目标 Agent；3) Ack 阶段——目标 Agent 收到事件后从 Redis 加载 Context，处理完成后发送 HANDOFF_ACK 事件；4) 清理——源 Agent 收到 ACK 后删除 Redis 中的 Context（或保留 24 小时用于审计）。如果目标 Agent 处理失败，源 Agent 在超时后（如 30 秒）重试发送 HANDOFF 事件。如果 Redis 宕机，使用数据库作为后备存储。

三个 Parallel Agent 的执行结果矛盾，如何自动解决？

三阶段解决：1) 多数决——如果三个 Agent 中有两个输出一致（相似度 > 0.8），采用多数结果；2) 权重决——每个 Agent 有历史准确率权重（如 schedule-agent 准确率 95%，高于 ticket-agent 的 88%），加权投票；3) 仲裁决——如果前两阶段无法解决，调用一个专门的仲裁 Agent 分析三个矛盾结果并做出最终决定。如果仲裁 Agent 也无法解决（置信度 < 0.6），转人工。CorpAssist 统计数据：约 85% 的矛盾可通过多数决解决，10% 通过加权投票，5% 需要仲裁 Agent，不到 1% 需要转人工。

场景：schedule-agent 和 email-agent 都需要获取用户信息，应该各查一次还是共享结果？

应该共享结果。设计思路：1) 在 HandoffContext 的 collectedData 中存放已获取的公共数据（如用户信息、部门信息）；2) Agent 在处理前先检查 collectedData 中是否已有需要的数据，如果有则直接使用，不需重新查询；3) 对于大型数据（如工单列表），使用 Redis 缓存并传入 cacheKey 而非完整数据。CorpAssist 的实现中，collectedData 包含一个 dataRegistry（Map），key 是数据名称，value 是 Redis key。Agent 按需从 Redis 加载数据，避免 HandoffContext 过大。

场景：Parallel 模式中一个 Agent 超时了（超过 10s），应该等还是跳过？

取决于任务类型：1) 如果该 Agent 的结果是"补充信息"（如查天气信息），可以跳过并在最终结果中标注"部分数据获取超时"；2) 如果该 Agent 的结果是"关键信息"（如查用户 ID，后续步骤依赖此 ID），则等待超时后整体失败。CorpAssist 的实现：每个 AgentTask 有 timeout 和 critical 两个属性。如果 critical=true 的超时，整个任务失败；如果 critical=false 的超时，使用默认值继续。默认的超时阈值是：重要信息 5 秒，非重要信息 2 秒。
