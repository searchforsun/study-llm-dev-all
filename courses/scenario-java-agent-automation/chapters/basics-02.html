为什么学这章

ReAct（Reasoning+Acting）是 Agent 最基础的设计模式。Spring AI 通过 ChatClient 和 ToolCallback 实现了完整的 ReAct 循环。理解这个循环如何在 Spring AI 的 Java 生态中落地，是开发 CorpAssist Agent 系统的核心技能。

学完后能做什么

用 Spring AI 实现 ReAct Agent 并配置循环参数；对比 ReactAgent 和纯 ChatClient 的差异；设计状态机管理多步 Agent 流程；配置步数和 Token 预算防止无限循环。

本章要点速记

1) ReAct 循环四步：Thought→Action→Observation→Repeat/FinalAnswer；2) Spring AI 通过 ChatClient 发送消息，ToolCallback 处理工具调用，循环控制由 AbstractToolCallSupport 实现；3) ReactAgent vs ChatClient：前者自动循环，后者一问一答；4) 状态机管理 Agent 状态（IDLE/RUNNING/WAITING_APPROVAL/COMPLETED/FAILED）；5) maxIterations（默认10）+ maxTokenUsage（默认4096）是防止无限循环的关键预算。

ReAct 循环的 Spring AI 实现

Spring AI 的 ReAct 循环通过 ChatClient + ToolCallback 的组合实现。核心流程是：发送消息给 LLM → 检查响应中是否包含 tool_calls → 根据 tool_calls 执行对应工具 → 将工具结果作为新消息追加 → 再次发送给 LLM → 重复直到 LLM 返回不含 tool_calls 的最终回答。

    participant CC as ChatClient
    participant LLM as LLM
    participant TC as ToolCallback
    participant T as 工具实现

    CC->>LLM: Prompt + System + History

[{name: "getUserInfo", args: {name: "张三"}}]

    TC->>T: getUserInfo("张三")
    T-->>TC: {userId: "u123"}

    TC->>T: queryTicket("u123")

    CC->>LLM: 消息 + ToolResponse
    LLM-->>CC: FinalAnswer: "张三有1个处理中的工单..."
    CC-->>U: 最终回答
java
// Spring AI ReAct 循环核心实现
@Service
public class ReactAgentService {

    private final ChatClient chatClient;
    private final List<ToolCallback> toolCallbacks;

    public ReactAgentService(ChatClient.Builder builder,
                             List<ToolCallback> toolCallbacks) {
        this.toolCallbacks = toolCallbacks;
        this.chatClient = builder
            .defaultSystem("""
                你是企业办公助手 CorpAssist。你有以下能力：
                - 查询员工信息、工单状态、日程安排
                - 发送邮件、创建工单、生成周报
                请按 ReAct 格式思考：先分析需要哪些信息，再逐步调用工具。
                如果任务完成或遇到无法解决的问题，给出友好提示。
                """)
            .build();
    }

    public String execute(String userMessage) {
        // 第一步：发送用户消息
        ChatResponse response = chatClient.prompt()
            .user(userMessage)
            .tools(toolCallbacks)  // 注册可用工具
            .call()
            .chatResponse();

        // 第二步：检查是否有工具调用
        int iteration = 0;
        while (hasToolCalls(response) && iteration < 10) {
            iteration++;
            // 工具调用已在 ChatClient 内部自动执行
            // 工具结果自动追加到消息历史
            response = chatClient.prompt()
                .user(userMessage)  // 携带完整历史
                .tools(toolCallbacks)
                .call()
                .chatResponse();
        }

        // 第三步：返回最终结果
        return response.getResult().getOutput().getText();
    }

    private boolean hasToolCalls(ChatResponse response) {
        return response.getResult().getOutput().getToolCalls() != null
            && !response.getResult().getOutput().getToolCalls().isEmpty();
    }
}

CorpAssist 场景

用户输入"帮我查张三的工单状态，如果还有未处理的工单就给他发邮件提醒"。Agent 需要三步：1) getUserInfo("张三")→获取 userId；2) queryTicket(userId)→查出未处理工单；3) sendEmail(userId, "你还有未处理的工单")→发送提醒。整个流程在 3 次 ReAct 循环内完成，约 5 秒。如果某个步骤失败（如工单系统超时），Agent 根据 System Prompt 规则进行重试或降级。

ReactAgent vs 纯 ChatClient

Spring AI 提供了两种使用方式：ChatClient 的一问一答模式和 ReactAgent 的自动循环模式。选择哪种取决于任务复杂度。

维度ChatClient（纯）ReactAgent
循环控制手动实现 while 循环内置 ReAct 循环引擎
工具调用需手动解析 tool_calls 并执行自动匹配并执行 @Tool 方法
状态管理无，需自行维护 Conversation内置 ChatHistory 管理
停止条件需自行实现 maxIterations内置 maxIterations/maxTokenUsage
错误处理需自行实现 try-catch内置重试和优雅降级
适用场景简单问答，无需多步推理需要 2+ 步工具调用的复杂任务
代码量少（~50 行）中（~150 行含配置）
调试难度低中（需理解循环机制）

// 方案 A：纯 ChatClient（适合简单问答）
@RestController
public class SimpleChatController {

    private final ChatClient chatClient;

    public SimpleChatController(ChatClient.Builder builder) {
        this.chatClient = builder.build();
    }

    @PostMapping("/chat")
    public String chat(@RequestBody String message) {
        return chatClient.prompt()
            .user(message)
            .call()
            .content();
    }
}

// 方案 B：ReactAgent（适合多步工具调用）
@Configuration
public class AgentConfig {

    @Bean
    public ReactAgent reactAgent(ChatClient.Builder builder,
                                  List<ToolCallback> tools) {
        return ReactAgent.builder()
            .chatClient(builder.build())
            .tools(tools)
            .maxIterations(10)
            .maxTokenUsage(4096)
            .build();
    }
}

@RestController
public class AgentController {

    private final ReactAgent reactAgent;

    public AgentController(ReactAgent reactAgent) {
        this.reactAgent = reactAgent;
    }

    @PostMapping("/agent")
    public String agent(@RequestBody String message) {
        return reactAgent.execute(message);
    }
}

状态机与断点续传

Agent 执行过程中可能因为各种原因中断（Token 超限、工具失败、用户暂停）。CorpAssist 使用状态机管理 Agent 的生命周期，支持从断点恢复。

stateDiagram-v2
    [*] --> IDLE: 用户发起请求
    IDLE --> RUNNING: 开始执行
    RUNNING --> WAITING_TOOL: 调用工具
    WAITING_TOOL --> RUNNING: 工具返回结果
    RUNNING --> WAITING_APPROVAL: 需要人工审批
    WAITING_APPROVAL --> RUNNING: 审批通过
    WAITING_APPROVAL --> FAILED: 审批拒绝
    RUNNING --> COMPLETED: 任务完成
    RUNNING --> FAILED: 错误/超限
    FAILED --> IDLE: 重试
    COMPLETED --> [*]
java
// Agent 状态机实现
public enum AgentState {
    IDLE, RUNNING, WAITING_TOOL, WAITING_APPROVAL, COMPLETED, FAILED
}

@Component
public class AgentStateMachine {

    private final Map<String, AgentSession> sessions = new ConcurrentHashMap<>();

    public AgentSession createSession(String sessionId, String userId, String task) {
        AgentSession session = new AgentSession(sessionId, userId, task);
        session.setState(AgentState.IDLE);
        sessions.put(sessionId, session);
        return session;
    }

    public AgentSession resumeSession(String sessionId) {
        AgentSession session = sessions.get(sessionId);
        if (session == null) {
            throw new IllegalArgumentException("Session not found: " + sessionId);
        }
        if (session.getState() == AgentState.FAILED) {
            session.setState(AgentState.IDLE);
            session.incrementRetryCount();
        }
        return session;
    }

    public void checkpoint(String sessionId, List<Message> history) {
        AgentSession session = sessions.get(sessionId);
        if (session != null) {
            session.setHistory(history);
            session.setLastCheckpoint(Instant.now());
        }
    }

    public record AgentSession(String sessionId, String userId, String task) {
        private AgentState state;
        private List<Message> history;
        private Instant lastCheckpoint;
        private int retryCount;
        // getters and setters omitted for brevity
    }
}

步数与 Token 预算

Agent 的无限循环和 Token 溢出是生产环境最常见的两类故障。Spring AI 通过 maxIterations 和 maxTokenUsage 提供基础防护，CorpAssist 在此基础上增加了更精细的预算控制。

预算不足

maxIterations=5, maxTokenUsage=2048：复杂任务（如"生成周报并发送给团队"）在第 3 步就因 Token 不足中断，只收集了部分数据，周报内容不完整。

预算合理

maxIterations=15, maxTokenUsage=8192：95% 的办公任务可以在预算内完成。剩余 5% 的超预算任务触发降级策略——返回已收集的中间结果和进度摘要。

思考 10 秒：Agent 在循环中反复调用同一个工具（如 getUserInfo）怎么办？

这是"工具循环"问题。三种应对：1) 在 System Prompt 中加入"避免重复调用已获取信息的工具"；2) 在工具层增加缓存——相同输入返回缓存结果而非重新调用；3) 在循环层检测重复模式——连续 3 次调用相同工具且参数相同，自动终止循环并返回"检测到重复调用，请检查您的请求"。CorpAssist 使用方案 2+3 的组合：工具层缓存避免重复执行，循环层检测防止浪费 Token。

本章结论

  • Spring AI 的 ReAct 循环通过 ChatClient + ToolCallback 实现，自动解析 tool_calls 并执行
  • ReactAgent 是 ChatClient 的上层封装，增加了循环控制、状态管理和错误处理
  • Agent 状态机（IDLE→RUNNING→WAITING_TOOL→...→COMPLETED/FAILED）支持断点续传和恢复
  • 步数预算 maxIterations=15 + Token 预算 maxTokenUsage=8192 覆盖 CorpAssist 95% 的场景
  • 工具循环检测和 Token 压缩是防止 Agent 失效的两个关键机制

面试可能问

Spring AI 的 ChatClient 是如何处理工具调用的？

ChatClient 内部使用 AbstractToolCallSupport 机制：1) 发送 prompt 时注册 ToolCallback 列表；2) LLM 响应可能包含 tool_calls 数组（JSON 格式，含 name 和 arguments）；3) ChatClient 自动匹配对应的 ToolCallback 执行方法；4) 将工具执行结果封装为 ToolResponseMessage 追加到消息列表；5) 再次请求 LLM 获取下一轮响应。整个过程对开发者透明——只需提供 ToolCallback，循环由 ChatClient 自动管理。如果开发者需要手动控制循环，可以关闭自动执行，手动解析 tool_calls。

Agent 执行到一半用户想暂停，如何实现？

通过状态机实现：1) Agent 在每次循环开始前检查 session 状态；2) 用户点击"暂停"时，状态从 RUNNING 改为 PAUSED；3) Agent 在当前工具调用完成后（不是中断工具执行）检查状态变化，发现 PAUSED 则保存当前 history 到持久化存储（Redis）并退出循环；4) 用户点击"继续"时，状态改为 RUNNING，Agent 从 Redis 加载 history，从上次暂停处继续执行。关键设计：不要在工具执行中间中断（可能导致脏数据），而是在循环边界检查状态。

maxTokenUsage 到达后应该怎样处理？

三种策略分层使用：1) 友好降级——返回当前已收集的信息和进度，"由于 Token 预算限制，我已完成以下步骤：1) 查到了张三的信息；2) 查到了工单列表。但还没有来得及发送邮件。请重新请求发送邮件功能。"；2) 压缩重试——自动压缩历史消息（summarize 模式），释放 Token 空间后继续执行；3) 优先队列——在设置 maxTokenUsage 时预留 20% 的 Buffer（如目标 8192，实际设置 10240），当接近阈值时优先执行剩余步骤中 Token 消耗最小的。CorpAssist 默认使用方案 1，对于重要任务使用方案 2。

场景：Agent 在第 4 步调用了 sendEmail，但邮件系统返回 503。Agent 接下来应该怎么做？

根据 CorpAssist 的容错策略：1) 自动重试 2 次（间隔 5 秒），因为 503 通常是临时故障；2) 如果重试仍然失败，检查是否有备用邮件通道（如备用 SMTP 服务器），有则切换重试；3) 所有重试都失败后，将失败信息记录到审计日志并返回友好提示："邮件发送暂时失败，我已记录下来，稍后系统会再次尝试发送。您的邮件草稿已保存。"；4) 如果任务的 degreeOfAutomation 为全自动，则标记任务为 FAILED 并通知管理员。

场景：Agent 连续 4 次循环都在调用 queryTicket（参数相同），如何检测并阻止？

实现工具调用去重检测器：1) 维护一个 Map<String, ToolCallKey> 记录每个 session 最近 10 次工具调用；2) ToolCallKey 包含 (toolName, argsJson, timestamp)；3) 每次新增调用前检查是否有相同 (toolName, argsJson) 的记录在最近 3 次内出现；4) 如果检测到重复模式≥3次相同，触发 StopRule——在当前循环结束时返回"检测到工具重复调用，可能陷入循环。当前已收集的信息：[摘要]"。CorpAssist 在 AgentEngine 中集成了这个检测器，线上数据显示约 2% 的 Agent 执行会触发该规则。
