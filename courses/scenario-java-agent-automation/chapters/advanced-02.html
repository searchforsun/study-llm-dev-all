CorpAssist 场景

面试中，你可以指着这张状态图说：这个 Agent 工作流包含 12 个状态和 13 条状态转换边。核心设计决策有：1) 审批等待状态有超时机制，30 分钟未响应自动升级；2) 并行收集状态使用 CompletableFuture 同时查询 3 个数据源，比串行快 3 倍；3) 路由状态负责分发到其他模块，解耦 Agent 间的依赖关系。面试官通过这张图可以在 10 秒内理解你的系统全貌。

MCP 集成亮点

MCP（Model Context Protocol）是 Agent 工具集成的关键亮点，面试中需要能清晰解释其架构优势。

java
// 面试用：MCP 集成核心代码（精简版）
// 展示你理解 MCP 的架构设计

// MCP 服务器端：将 Spring Bean 暴露为 MCP 工具
@McpServer(endpoint = "/mcp/calendar")
public class CalendarMcpServer {

    @McpTool
    public McpResponse queryFreeSlots(@McpParam("userId") String userId,
                                       @McpParam("date") String date) {
        List<TimeSlot> slots = calendarService.queryFreeSlots(userId, date);
        return McpResponse.success(slots);
    }
}

// MCP 客户端：动态发现和调用工具
@McpClient(serverUrl = "http://calendar-service:8081/mcp")
public class CorpMcpClient {

    private final McpSession session;

    public List<McpToolDefinition> discoverTools() {
        // MCP 标准：工具列表和 Schema 自动发现
        return session.listTools();
    }

    public McpResponse callTool(String toolName, JsonObject params) {
        // MCP 标准：工具调用统一接口
        return session.callTool(toolName, params);
    }
}

// 面试话术：
// "我使用 MCP 协议将 12 个不同的微服务（日历、邮件、工单、HR）
// 统一注册为 Agent 工具。MCP 的核心优势是：
// 1) 标准化——所有工具通过同一协议暴露和调用
// 2) 自动发现——Agent 启动时自动发现可用工具和参数 Schema
// 3) 跨语言——Java 服务、Python 服务都可以注册
// 4) 权限管控——在 MCP 网关层统一做权限校验和审计"

Q: 面试作品需要多大规模才够？

中级开发（3-5 年）：1 个端到端 Demo，3-5 个 Agent 节点，涵盖 ReAct 循环和 @Tool 定义。高级开发（5-8 年）：完整的 Plan-and-Execute 架构，6+ 个 Agent，审批节点，MCP 集成，熔断降级。架构师（8 年+）：多租户隔离，审计合规，高可用部署，性能调优。规模不重要，关键是每个选择有数据支撑。1 个"周报生成"场景做深做透（有架构图、有性能数据、有容错机制）比 5 个浅尝辄止的场景好得多。

面试可能问

你这个 Agent 系统和 Dify/Copilot 这类现成平台比有什么优势？

区分"自研 vs 平台"的关键面试题。回答框架：自研的优势是深度定制和可控——1) 我们可以深度集成企业现有的 Java 技术栈（Spring Boot、Nacos、Kafka），而不是被平台限制；2) 审批流程和业务系统紧密耦合，现成平台的审批模块无法满足企业复杂的审批链（链式审批、会签、超时逐级升级）；3) 数据安全和合规——自研系统数据不出企业内网，通过了等保三级认证；4) 成本——每月 500 人企业的 API 调用成本约 8000 元，低于企业版 SaaS 的 3 万元/月。劣势是维护成本高。我的选择策略：核心场景自研（可深度定制），边缘场景接入平台 API（成本低）。

你的 Agent 项目部署在什么环境？CPU/GPU 配置？

部署架构：2 台 8C16G ECS 服务器作为 Agent 服务节点 + 1 台 4C8G 作为审批网关 + 1 台 4C8G 作为规则引擎。LLM 调用全部走阿里云 DashScope API（无需 GPU），本地不需要部署模型。数据库：RDS MySQL（审计数据）+ Redis 集群（检查点和缓存）+ Milvus（向量记忆）。所有服务在 Kubernetes 中部署，HPA 自动扩缩（根据 QPS 和 CPU 使用率）。整体月成本约 1.2 万元（含 API 调用费），相比雇佣 3 个 人工处理（月薪约 5 万元/人）节省了 13.8 万元/月。

如果让你重新做这个项目，会有什么不同的选择？

三个改进点：1) 早期就用 Graph 而非 ReAct——我们在 Phase 1 使用 ReAct Agent，但发现复杂的审批流程需要显式状态管理，Phase 2 才迁移到 Graph，增加了 2 周迁移成本。如果重来，对于包含审批节点的流程直接用 Graph。2) 更早引入熔断器——前 2 个月没有熔断保护，有一次 DashScope 故障导致 Agent 全部阻塞，平均响应时间飙升到 30 秒。后来加了 Resilience4j 熔断器，故障时自动降级到规则引擎。3) 前期就定义验收清单——前 3 周没有明确的验收标准，导致"开发完成"后和业务方对需求花了 1 周。重来的话先和业务方确定 8 项验收清单再开始编码。这些教训说明：先做对架构选型，再加好容错机制，最后用验收清单兜底。

场景：面试官质疑"这个数据（24 万美元年化节省）怎么算出来的？"

透明回答：分三步计算。1) 基准数据——我们选取了 10 个最常用的办公场景，通过 OA 系统日志统计了每个场景的日均调用量和人工处理时间。比如周报：500 人中 300 人写周报，每人平均 15 分钟，日均 75 小时。2) Agent 处理时间——Agent 处理周报平均 8 秒，加上审批平均 45 分钟（审批人操作时间 2 分钟+等待时间）。关键点：Agent 节省的是"请求者"的时间，审批人的时间没有节省（因为审批仍然需要人）。3) 财务核验——年化节省 = 节省的人天 × 日均薪。按员工日均薪 800 元计算，周报场景年节省 = 75 小时/天 × 250 天 × 800 元/8h ≈ 187.5 万元。所有场景合计约 200 万元。减去 API 成本（约 10 万/年）和服务器成本（约 15 万/年），净节省约 175 万元（约 24 万美元）。

场景：面试官问"你遇到的最大技术挑战是什么？"

STAR 回答：S——Agent 在审批环节经常死锁：审批人等待 Agent 恢复，Agent 等待审批结果。T——解决这个循环等待问题。A——我引入了"审批网关"模式：Agent 执行到审批节点时保存检查点到 Redis，然后释放线程。审批结果通过 Kafka 异步通知，Agent 监听结果后从检查点恢复。同时配置了超时降级：如果 30 分钟内无响应自动升级给上级。R——死锁问题完全解决，审批平均等待时间 45 分钟（含审批人响应时间），P99 响应时间从原来的"无穷大"（死锁）降至 3.2 秒。这个案例展示了我对异步解耦和状态管理的理解。
