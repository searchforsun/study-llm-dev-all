// 周报 Agent 完整实现
@Component
public class WeeklyReportAgent {

    private final MultiAgentOrchestrator orchestrator;
    private final ApprovalGateway approvalGateway;
    private final AgentMonitor monitor;

    public AgentResult generateWeeklyReport(String userId) {
        String sessionId = UUID.randomUUID().toString();
        Instant start = Instant.now();

        try {
            // Step 1: 并行收集数据
            String collectedData = orchestrator.executeParallel(
                "收集用户 " + userId + " 的周报数据",
                List.of("user-info-agent", "worklog-agent", "ticket-agent")
            );

            // Step 2: 生成周报内容
            String report = orchestrator.executeSequential(
                collectedData,
                List.of("report-generation-agent")
            );

            // Step 3: 提交审批（如果需要）
            if (needApproval(userId)) {
                ApprovalRequest request = ApprovalRequest.builder()
                    .title("周报审批: " + userId)
                    .description(report)
                    .approvers(getManager(userId))
                    .type(ApprovalType.SINGLE)
                    .timeoutMinutes(60)
                    .build();
                approvalGateway.requestApproval(sessionId, request);
                // 等待审批回调...
            }

            // Step 4: 发送邮件
            String emailResult = orchestrator.executeRouting(
                "发送周报给用户 " + userId
            );

            // 记录监控指标
            monitor.recordCompletion(sessionId, start, Status.SUCCESS);
            return AgentResult.success(emailResult);

        } catch (Exception e) {
            monitor.recordCompletion(sessionId, start, Status.FAILED);
            return AgentResult.failure("周报生成失败: " + e.getMessage());
        }
    }
}

CorpAssist 场景

开发者在 DashScope 调试面板中查看一次周报生成的完整过程：1) 路由 Agent 的 Thought："用户请求周报生成，路由到 data-collection-agent"；2) 收集 Agent 依次调用了 getUserInfo（150ms）、queryWorklogs（800ms）、queryTickets（1.2s）；3) 生成 Agent 向 LLM 发送了包含 3200 Tokens 的提示词，返回了 800 Tokens 的报告；4) 审批节点发送了审批请求。开发者根据面板数据发现 queryTickets 是性能瓶颈，优化后从 1.2 秒降至 200 毫秒。

步数/Token Dashboard

监控是 Agent 系统运维的核心。CorpAssist 使用 Micrometer 采集指标 + Prometheus 存储 + Grafana 可视化。

java
// Agent 监控指标采集
@Component
public class AgentMonitor {

    private final MeterRegistry meterRegistry;
    private final Counter totalRequests;
    private final Counter totalTokenUsage;
    private final Timer executionTimer;
    private final DistributionSummary stepsPerTask;

    public AgentMonitor(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
        this.totalRequests = Counter.builder("agent.requests.total")
            .description("Total agent requests").register(meterRegistry);
        this.totalTokenUsage = Counter.builder("agent.tokens.total")
            .description("Total token usage").register(meterRegistry);
        this.executionTimer = Timer.builder("agent.execution.duration")
            .description("Agent execution duration")
            .publishPercentiles(0.5, 0.9, 0.99)
            .register(meterRegistry);
        this.stepsPerTask = DistributionSummary.builder("agent.steps.per.task")
            .description("Steps per task")
            .baseUnit("steps")
            .register(meterRegistry);
    }

    public Timer.Sample startTimer() {
        return Timer.start(meterRegistry);
    }

    public void recordCompletion(String sessionId, Instant start, Status status) {
        totalRequests.increment();
        executionTimer.record(Duration.between(start, Instant.now()));
        // 记录到日志
        log.info("Agent session {} completed with status {} in {} ms",
            sessionId, status,
            Duration.between(start, Instant.now()).toMillis());
    }

    public void recordTokenUsage(int tokens) {
        totalTokenUsage.increment(tokens);
    }

    public void recordSteps(int steps) {
        stepsPerTask.record(steps);
    }

    // Grafana Dashboard 展示的指标
    // 1. 请求总量（总/成功/失败）
    // 2. P50/P90/P99 响应时间
    // 3. Token 消耗趋势
    // 4. 每任务步数分布
    // 5. 工具调用分布
    // 6. 审批等待时间
}

思考 10 秒：验收中发现 Agent 在某些用户输入下会陷入死循环，如何排查？

排查方法：1) 在 DashScope 调试面板中查看 Agent 的 Thought 序列，确认是哪个环节开始循环；2) 查看工具调用日志，检查是否出现了重复工具调用模式；3) 检查 System Prompt 中是否有模糊的指令导致 Agent 无法决策；4) 查看 maxIterations 配置是否足够。CorpAssist 实践中 90% 的死循环由 System Prompt 中模糊的"如果...否则..."逻辑引起，修复方法是将模糊指令改为具体的判断规则。

面试可能问

演示环境的验收标准和生产环境有什么区别？

演示环境关注功能完整性（所有 Agent 链路正确），生产环境额外关注：1) 性能——P99 响应时间 < 10s，演示环境可以放宽到 < 30s；2) 容量——支持 QPS 和并发数，演示环境只验证单用户场景；3) 安全——生产环境需要渗透测试和代码审计，演示环境只需要基本权限校验；4) 可观测——生产环境需要完整告警和日志链路，演示环境只需要 DashScope 调试面板；5) 数据——生产环境需要脱敏和合规，演示环境可以使用模拟数据。建议：演示验收通过后，再按生产环境标准逐一加固。

如何向非技术人员（如业务方）演示 Agent 能力？

三步演示法：1) 场景导入——"想象你每天早上需要花 15 分钟写周报，现在只需要说一句话"→输入"生成这周周报"→展示 Agent 自动收集数据、生成报告的过程；2) 边界展示——演示 Agent 如何应对异常（故意输入"帮我把工资翻倍"→展示 Agent 调用审批节点→审批人拒绝→Agent 友好提示）；3) 价值量化——展示对比数据：人工写周报平均 15 分钟、Agent 生成 8 秒、审批流程 5 分钟、总时间节省 90%。演示时使用大字体和动画过渡，避免展示代码和技术细节。观众最关心的是"能帮我省多少时间"而不是"用了什么技术栈"。

验收时发现 Agent 在某些边缘情况（如用户输入特别长、包含特殊字符）下表现异常，如何修复？

系统化修复：1) 建立坏例库——将所有异常输入记录到测试用例库，标注预期行为和实际行为的差异；2) 输入预处理——在 Agent 入口增加 InputSanitizer，处理过长的输入（截断到 2000 字符）、特殊字符（转义或删除）、敏感信息（脱敏）；3) 渐进式修复——按优先级修复坏例：P0（系统崩溃）→P1（结果错误）→P2（可用但不优雅）；4) 回归测试——每次修复后运行所有坏例库的测试用例，确保不引入新的问题。CorpAssist 的坏例库包含 200+ 个测试用例，覆盖 15 种异常输入场景，CI 流水线自动运行。

场景：演示时 Agent 突然调用失败（LLM 超时），如何快速恢复演示？

演示应急预案：1) 降级模式——在 Agent 配置中准备 fallback prompt，当 LLM 超时时使用规则引擎生成简单周报（基于模板填充数据）；2) 本地缓存——Agent 常用结果（上周周报、用户信息）有本地缓存，即使后端服务故障也能部分展示；3) 演示专用容错——演示环境配置更宽松的超时参数（30 秒 vs 生产 10 秒），减少演示时的失败概率；4) 话术准备——准备好"这个场景展示了我们的容错机制，Agent 检测到服务不稳定，自动切换到备用方案"这样的解释，将故障转化为优点。CorpAssist 演示团队有一条原则：所有演示中的"故障"都是"容错演示"。

场景：验收时审批流程正常，但实际生产环境审批等待时间过长（平均 4 小时），怎么监控和优化？

优化方案：1) 监控——Grafana 添加审批等待时间面板，按审批人分组展示。如果某个审批人平均等待 8 小时，检查其是否经常忽略审批通知；2) 机制——增加审批催促功能，超过 1 小时发送一次提醒，超过 2 小时发送短信，超过 4 小时升级给上级；3) 移动端——确保审批通知能推送到企业微信/钉钉，审批人可以在手机上操作；4) 时限配置——根据审批类型设置不同超时时间（常规 2 小时/紧急 30 分钟/敏感 8 小时），超时自动降级。CorpAssist 实施这些优化后，审批平均等待时间从 4 小时降至 45 分钟。
