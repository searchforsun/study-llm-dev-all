为什么学这章

真实的办公任务通常需要多个工具协作完成（如查日程→约会议室→发通知→创建工单）。本章深入 Spring AI 的 @Tool 注解体系，学习如何定义、编排、容错多个工具，以及如何在并行和串行执行模式间切换。

学完后能做什么

用 @Tool 注解定义日历、邮件、工单等办公工具；实现并行和串行工具执行模式；配置工具失败的重试与熔断策略；实现上下文压缩适配器减少 Token 消耗。

本章要点速记

1) @Tool 注解参数：name（工具名）、description（描述）、returnDirect（是否直接返回）；2) 并行执行用 CompletableFuture 同时调用多个独立工具，串行执行按依赖关系顺序调用；3) 重试策略：工具调用失败后最多重试 2 次（指数退避），仍然失败则熔断；4) 上下文压缩 Adapter 将工具返回的大数据截断为摘要，防止 Token 爆炸；5) CorpAssist 典型场景：查日程（并行）+ 会议室（并行）+ 创建会议（串行依赖前面结果）。

          description = "查询指定日期范围内员工的空闲时段，返回空闲时间段列表")
    public List<TimeSlot> queryFreeSlots(
            @ToolParam(description = "员工ID") String userId,
            @ToolParam(description = "开始日期，格式 yyyy-MM-dd") String startDate,
            @ToolParam(description = "结束日期，格式 yyyy-MM-dd") String endDate) {
        // 调用日历服务 API
        return calendarService.getFreeSlots(userId, startDate, endDate);
    }

    @Tool(name = "createEvent",
          description = "创建日历事件/会议，需要主题、时间、参与人")
    public Event createEvent(
            @ToolParam(description = "会议主题") String title,
            @ToolParam(description = "开始时间 ISO 格式") String startTime,
            @ToolParam(description = "结束时间 ISO 格式") String endTime,

            List.of(attendees.split(",")), roomId);

并行/串行工具执行

任务规划时需要考虑工具间的依赖关系。无依赖的工具可以并行执行（提升效率），有依赖的工具必须串行执行（保证数据正确）。

java
// 并行/串行工具执行器
@Component
public class ToolExecutionOrchestrator {

    private final ExecutorService parallelExecutor =
        Executors.newVirtualThreadPerTaskExecutor();  // Java 21 虚拟线程

    /**
     * 执行无依赖的工具集（并行）
     */
    public Map<String, Object> executeParallel(List<ToolTask> tasks) {
        Map<String, Object> results = new ConcurrentHashMap<>();
        List<CompletableFuture<Void>> futures = tasks.stream()
            .map(task -> CompletableFuture.runAsync(() -> {
                results.put(task.name(), task.execute());
            }, parallelExecutor))
            .toList();
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
            .join();  // 等待所有并行任务完成
        return results;
    }

    /**
     * 执行有依赖关系的工具集（串行）
     */
    public Map<String, Object> executeSequential(List<ToolTask> tasks) {
        Map<String, Object> results = new LinkedHashMap<>();
        Map<String, Object> context = new HashMap<>();

        for (ToolTask task : tasks) {
            // 注入依赖的结果作为上下文
            task.setContext(context);
            Object result = task.execute();
            results.put(task.name(), result);
            // 将当前步骤结果加入上下文供后续步骤使用
            context.put(task.name(), result);
        }
        return results;
    }

    // CorpAssist 会议安排编排示例
    public Map<String, Object> arrangeMeeting(String organizer,
                                                String date, String topic) {
        // Step 1: 并行查询参与人的空闲时间
        List<ToolTask> parallelTasks = List.of(
            new ToolTask("getUserInfo", () -> userService.getUser(organizer)),
            new ToolTask("getTeamMembers", () -> orgService.getTeamMembers(organizer))
        );
        Map<String, Object> parallelResults = executeParallel(parallelTasks);

        // Step 2: 串行：查空闲→创建会议→发通知
        List<ToolTask> sequentialTasks = List.of(
            new ToolTask("queryFreeSlots", () -> calendarService
                .queryFreeSlots(organizer, date, date)),
            new ToolTask("createEvent", () -> calendarService
                .createEvent(topic, /* 从上一步获取 */ "")),
            new ToolTask("sendEmail", () -> emailService
                .send(/* 从上一步获取参与人 */ organizer, topic))
        );
        return executeSequential(sequentialTasks);
    }
}

工具失败的重试与熔断

工具调用可能因为网络问题、服务故障、数据异常而失败。CorpAssist 使用分层容错策略，结合 Spring Resilience4j 实现重试和熔断。

java
// 工具调用容错配置
@Configuration
public class ToolResilienceConfig {

    @Bean
    public RetryRegistry toolRetryRegistry() {
        RetryConfig config = RetryConfig.custom()
            .maxAttempts(3)
            .waitDuration(Duration.ofSeconds(2))
            .retryExceptions(IOException.class, TimeoutException.class)
            .ignoreExceptions(IllegalArgumentException.class)  // 参数错误不重试
            .build();
        return RetryRegistry.of(config);
    }

    @Bean
    public CircuitBreakerRegistry toolCircuitBreakerRegistry() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            .failureRateThreshold(50)           // 50% 失败率触发熔断
            .slidingWindowSize(10)              // 统计最近 10 次调用
            .waitDurationInOpenState(Duration.ofSeconds(30))  // 熔断持续 30 秒
            .permittedNumberOfCallsInHalfOpenState(3)         // 半开状态尝试 3 次
            .build();
        return CircuitBreakerRegistry.of(config);
    }
}

// 带容错的工具调用
@Component
public class ResilientToolExecutor {

    private final Retry retry;
    private final CircuitBreaker circuitBreaker;

    public ResilientToolExecutor(RetryRegistry retryRegistry,
                                  CircuitBreakerRegistry cbRegistry) {
        this.retry = retryRegistry.retry("toolRetry");
        this.circuitBreaker = cbRegistry.circuitBreaker("toolCircuitBreaker");
    }

    public Object executeWithResilience(String toolName,
                                        Supplier<Object> toolCall) {
        return circuitBreaker.executeSupplier(
            () -> retry.executeSupplier(toolCall)
        );
    }
}

CorpAssist 场景

Agent 调用 sendEmail 工具时邮件服务返回 503（临时不可用）。熔断器检测到最近 10 次调用中有 6 次失败（60% > 50% 阈值），触发熔断。后续 30 秒内所有 sendEmail 调用直接返回降级结果（"邮件服务暂时不可用，已保存草稿"），不消耗实际 API 调用。30 秒后熔断器进入半开状态，允许 3 次探测调用，如果成功则恢复正常。这套机制保障了邮件服务故障时 Agent 仍然可以处理其他任务，不被阻塞。

上下文压缩 Adapter

工具返回的数据可能非常大（如查询整个团队的日程返回数百条事件），直接传给 LLM 会浪费 Token。上下文压缩 Adapter 在工具返回数据和 LLM 之间加一层截断/摘要。

java
// 上下文压缩适配器
@Component
public class ContextCompressionAdapter implements ToolCallback {

    private final ToolCallback delegate;
    private static final int MAX_TOOL_RESPONSE_LENGTH = 500;

    public ContextCompressionAdapter(ToolCallback delegate) {
        this.delegate = delegate;
    }

    @Override
    public String getName() {
        return delegate.getName();
    }

    @Override
    public String getDescription() {
        return delegate.getDescription();
    }

    @Override
    public JsonSchema getInputSchema() {
        return delegate.getInputSchema();
    }

    @Override
    public String call(String functionInput) {
        String rawResult = delegate.call(functionInput);

        // 如果结果太长，截断并添加摘要
        if (rawResult.length() > MAX_TOOL_RESPONSE_LENGTH) {
            return truncateAndSummarize(rawResult);
        }
        return rawResult;
    }

    private String truncateAndSummarize(String raw) {
        // 策略 1: 截断
        if (raw.length() > MAX_TOOL_RESPONSE_LENGTH * 2) {
            return raw.substring(0, MAX_TOOL_RESPONSE_LENGTH)
                + "\n...(截断，共 " + raw.length() + " 字符)";
        }
        // 策略 2: 提取关键字段
        if (raw.startsWith("[") || raw.startsWith("{")) {
            return extractKeyFields(raw);
        }
        return raw;
    }

    // 注册时包装所有工具
    @Bean
    public List<ToolCallback> compressedTools(List<ToolCallback> rawTools) {
        return rawTools.stream()
            .map(ContextCompressionAdapter::new)
            .collect(Collectors.toList());
    }
}

无压缩

queryTeamSchedule("u123") 返回 200 条日程，JSON 约 15KB → LLM 处理这 15KB Token，Agent 每轮循环 Token 消耗大，maxTokenUsage 容易耗尽。

有压缩

queryTeamSchedule("u123") 返回 200 条日程 → 压缩为摘要："今天有 15 个会议，最早 09:00，最晚 17:30。关键会议：项目评审(14:00-15:00, 8人)"，约 200 字符 → LLM 只需要处理这个摘要，Token 消耗减少 98%。

本章结论

  • @Tool 注解定义日历/邮件/工单等办公工具，清晰的 description 是 LLM 正确调用的关键
  • 并行执行（CompletableFuture + 虚拟线程）提升无依赖工具的调用效率，混合编排覆盖大多数场景
  • Resilience4j 重试（3 次指数退避）+ 熔断（50% 失败率阈值）保障工具调用的可靠性
  • 上下文压缩 Adapter 将工具返回大数据截断为摘要，减少 90%+ 的 Token 消耗
  • CorpAssist 使用混合编排模式：并行查询用户和团队信息 + 串行创建会议通知

面试可能问

并行工具调用中一个工具失败了，应该全部回滚还是只回滚失败的工具？

取决于工具间是否有业务依赖。如果有依赖关系（如"创建会议"依赖"查空闲时间"的结果），则全部回滚——因为后续步骤基于错误数据执行。如果无依赖关系（如同时"查日历"和"查工单"），则只失败单个工具，其他工具的结果继续使用。CorpAssist 的实现：每个 ToolTask 有一个 rollback() 方法，如果并行组中有任一工具失败，调用所有已完成工具的 rollback()。对于不可回滚的操作（如"发邮件"已发送成功），记录审计日志并标记"部分成功"。实践中，CorpAssist 约 80% 的并行工具调用是无依赖的，只需要部分回滚。

上下文压缩 Adapter 截断了重要信息导致 LLM 决策错误怎么办？

解决方案：1) 分层压缩——重要数据字段（如工单 ID、状态、截止日期）完整保留，次要数据（如备注、操作日志）截断；2) 可扩展截断——当 LLM 发现需要更多信息时，可以调用 getToolFullResult(toolName) 获取完整数据；3) 压缩摘要中包含数据量提示——如"共 200 条日程，已显示前 5 条"，LLM 知道有更多数据可用；4) 关键字段白名单——使用 @ToolParam(important=true) 标记不会截断的关键字段。CorpAssist 使用方案 1+3：先根据字段重要性选择性保留，再在摘要末尾标注数据总量。

@Tool 的 returnDirect=true 应该什么时候用？

returnDirect=true 表示工具的结果直接返回给用户，不经过 LLM 的二次处理。适用场景：1) 纯数据查询类——"查天气"返回 JSON，直接展示即可，不需要 LLM 重新描述；2) 操作确认类——"发送邮件"返回{success:true, messageId:"xxx"}，直接显示确认信息；3) 文件下载类——返回文件 URL，用户直接点击下载。不适用场景：1) 需要 LLM 解释的数据——如"查工单"返回原始 JSON，需要 LLM 解释为自然语言；2) 多步任务的中间步骤——因为后续步骤需要 LLM 理解工具返回的内容。CorpAssist 中约 30% 的工具使用 returnDirect=true。

场景：同时查 10 个人的空闲时间，串行查要 10 秒，并行查要 2 秒。但工单系统限流单用户 5 QPS。如何平衡？

折中方案：1) 分片并行——将 10 人分为 2 组每组 5 人，组内并行，组间串行；2) 信号量控制——使用 Semaphore(5) 限制并发数不超过工单系统 QPS 限制；3) 自适应限流——调用前检查工单系统的当前负载，动态调整并行度。CorpAssist 使用方案 2：初始化一个全局 Semaphore，每个工具调用前 acquire()，用完后 release()。当系统负载高时调用方自然等待，不会触发限流。这样既利用了并行提速（从 10 秒降到 4 秒），又不会压垮下游系统。

场景：sendEmail 工具调用频繁触发熔断，但运维确认邮件服务正常，熔断器配置可能有问题？

排查方向：1) 检查熔断器统计窗口——slidingWindowSize=10 太小，10 次中仅 1 次超时就触发 50% 阈值（1/2 也可以到 50%），建议改为 20；2) 检查异常分类——retryExceptions 包含了所有 Exception，但某些业务异常（如"收件人邮箱不存在"）不应该计入熔断统计；3) 检查 waitDurationInOpenState——30 秒太短，如果邮件服务确实有问题，30 秒后会反复触发半开探测还是失败。CorpAssist 推荐配置：slidingWindowSize=20，failureRateThreshold=50，waitDurationInOpenState=60s。同时使用 ignoreExceptions 排除参数错误的业务异常。
