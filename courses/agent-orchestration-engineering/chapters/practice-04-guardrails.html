为什么要学本章

Agent 拥有调用工具和执行操作的能力，这意味着 guardrails 不是可选项，而是生产系统的必备组件。一个没有护栏的 Agent 可能被 prompt injection 利用、误调用高危工具、输出敏感信息、或生成不合规内容。

学完你能
  • 设计 Agent 的输入验证和注入防护机制
  • 实现工具白名单和权限控制
  • 配置输出合规检查和内容安全阀
  • 使用 Nvidia NeMo Guardrails 或自定义规则
本章先记住 3 件事
  • 输入过滤：用户输入可能包含 prompt injection，需要在进入 Agent 前清洗。
  • 工具权限：高危操作（删除、转账）需要白名单+人工确认双重保护。
  • 输出检查：Agent 的回复不应包含敏感信息、不合规内容或幻觉。

安全护栏分层架构

Agent 的安全护栏应该覆盖从输入到输出的全链路。每一层解决不同维度的安全问题。

输入验证与注入防护

Prompt injection 是最常见的 Agent 安全威胁。攻击者通过在用户输入中嵌入指令，试图覆盖或绕过 Agent 的 System Prompt。

Python：输入过滤器

python 
import re
from typing import Optional

class InputSanitizer:
    """Agent 输入过滤器"""

    INJECTION_PATTERNS = [
        r"忽略(之前|以上|所有).*指令",
        r"忘记.*(角色|规则|约束)",
        r"你(现在|接下来).*(是|扮演)",
        r"system\s*:",
        r"<\|im_start\|>",
        r"ignore.*(above|instructions|previous)",
    ]

    SENSITIVE_PATTERNS = [
        r"\b\d{18}\b",           # 身份证号
        r"\b1[3-9]\d{9}\b",      # 手机号
        r"\b\d{16,19}\b",        # 银行卡号
    ]

    @classmethod
    def sanitize(cls, text: str) -> tuple[bool, str, Optional[str]]:
        """清洗输入，返回(是否通过, 清洗后文本, 拒绝原因)"""
        # 1. 检测注入
        for pattern in cls.INJECTION_PATTERNS:
            if re.search(pattern, text):
                return False, text, "检测到可能的注入攻击"

        # 2. 检测敏感信息
        for pattern in cls.SENSITIVE_PATTERNS:
            if re.search(pattern, text):
                return False, text, "输入包含敏感信息，已拒绝"

        # 3. 长度检查
        if len(text) > 4000:
            text = text[:4000]

        return True, text, None

# 使用
is_safe, cleaned, reason = InputSanitizer.sanitize(user_input)
if not is_safe:
    return f"输入被拒绝：{reason}"

Java：Spring 拦截器实现

java 
@Component
public class AgentInputInterceptor {

    private static final List<Pattern> INJECTION_PATTERNS = List.of(
        Pattern.compile("忽略(之前|以上|所有).*指令"),
        Pattern.compile("忘记.*(角色|规则|约束)"),
        Pattern.compile("你(现在|接下来).*(是|扮演)")
    );

    public ValidationResult validate(String input) {
        // 注入检测
        for (Pattern p : INJECTION_PATTERNS) {
            if (p.matcher(input).find()) {
                return ValidationResult.rejected("检测到注入攻击");
            }
        }
        // 长度限制
        if (input.length() > 4000) {
            input = input.substring(0, 4000);
        }
        return ValidationResult.passed(input);
    }

    public record ValidationResult(boolean passed, String sanitized, String reason) {
        public static ValidationResult passed(String text) {
            return new ValidationResult(true, text, null);
        }
        public static ValidationResult rejected(String reason) {
            return new ValidationResult(false, null, reason);
        }
    }
}

工具白名单与权限控制

不是所有 Agent 都应该能调用所有工具。基于角色的权限控制确保每个 Agent 只能在其授权范围内操作。

不同角色的 Agent 有不同的工具权限矩阵。

python 
class ToolPermissionManager:
    """工具权限管理器"""

    def __init__(self):
        self.permissions = {
            "客服": {
                "query_order": "read",
                "create_ticket": "write",
            },
            "售后": {
                "query_order": "read",
                "refund_order": "write",
                "create_ticket": "write",
            },
            "管理员": {
                "query_order": "read",
                "refund_order": "write",
                "delete_order": "admin",
                "create_ticket": "write",
            }
        }

    def check_permission(self, role: str, tool: str, action: str) -> bool:
        """检查角色是否有权限"""
        if role not in self.permissions:
            return False
        if tool not in self.permissions[role]:
            return False
        required_level = self.permissions[role][tool]
        level_map = {"read": 1, "write": 2, "admin": 3}
        action_map = {"read": 1, "write": 2, "delete": 3}
        return level_map.get(required_level, 0) >= action_map.get(action, 0)
常见误区
  • 只依赖 LLM 判断安全性：LLM 本身可能被注入，安全规则必须在 LLM 外部实现。
  • 宽松的正则规则：注入检测模式太窄，攻击者可以轻松绕过。
  • 不做输出脱敏：工具返回了完整的用户数据，未脱敏直接输出。

输出合规检查

Agent 的输出需要经过合规检查再返回给用户。包括：敏感信息脱敏、内容安全审核、格式合规检查。

python 
class OutputGuardrail:
    """输出护栏：脱敏 + 内容安全 + 合规"""

    @staticmethod
    def desensitize(text: str) -> str:
        """脱敏处理"""
        text = re.sub(r'\b(\d{3})\d{4}(\d{4})\b', r'\1****\2', text)  # 手机号
        text = re.sub(r'\b(\d{6})\d{8}(\d{4})\b', r'\1********\2', text)  # 身份证
        return text

    @staticmethod
    def check_content_safety(text: str) -> tuple[bool, str]:
        """内容安全审核"""
        unsafe_patterns = [
            "银行账号.*\\d+", "密码.*\\S+",
            "内部资料", "机密"
        ]
        for pattern in unsafe_patterns:
            if re.search(pattern, text):
                return False, "输出包含受限信息"
        return True, text

    @classmethod
    def check(cls, output: str) -> str:
        """完整输出检查"""
        output = cls.desensitize(output)
        ok, result = cls.check_content_safety(output)
        if not ok:
            return "抱歉，无法提供此信息（已触发内容安全规则）"
        return result

常见问题（FAQ）

Guardrails 会影响 Agent 的性能吗？
每层护栏都有计算开销。输入过滤和参数校验通常在微秒级，影响可以忽略。内容安全审核如果用 LLM 做（如用另一个模型审核输出），延迟会显著增加。建议：规则类的检查用代码实现，语义类的检查用轻量模型或采样策略。
如何防止 LLM 被注入后调用高危工具？
三管齐下：① 在 Agent 层建立工具白名单，LLM 只能调用白名单内的工具；② 高危工具（转账、删除）需要额外的人工确认；③ 在工具层做参数校验，拒绝不合理参数。见 人机协作 中的审批流程。
Prompt Injection 检测的准确率有多高？
基于正则或关键词的检测（如本章示例）准确率 70-80%，会有误报和漏报。建议使用专门的安全模型（如 Nvidia NeMo Guardrails、Lakera Guard），准确率可达到 95%+。但安全模型本身也需要持续更新对抗新的攻击手法。

本章结论

  • 六层护栏：从输入过滤到输出检查，覆盖全链路。
  • 注入防护：输入必须经过清洗才能进入 Agent 流程。
  • 权限控制：基于角色的工具白名单，高危操作需人工确认。
  • 输出合规：脱敏、内容安全、格式合规三重检查。
  • 外部工具：NeMo Guardrails 和自定义规则结合使用效果最佳。
