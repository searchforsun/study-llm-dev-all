为什么要学本章

工具是 Agent 能力的外延。工具设计的质量直接决定 Agent 的可靠性——tool design 影响模型是否能正确理解和使用工具。一个 description 写得好的工具，模型调用准确率可以提高 40% 以上。本章从粒度、描述、错误处理和测试四个维度深入工具设计最佳实践。

学完你能
  • 判断工具的正确粒度和边界
  • 编写高准确率的工具 description
  • 实现工具的错误恢复和重试机制
  • 编写工具单元测试和集成测试
本章先记住 3 件事
  • 单一职责：一个工具只做一件事，做得好。
  • 描述即文档：description 是模型理解工具的接口文档，需要精确、完整。
  • 错误即反馈：工具错误应以结构化方式返回，让 Agent 能根据错误调整行为。

工具粒度与单一职责

工具的大小（粒度）直接影响模型的使用效果。粒度太粗的工具不够灵活，粒度太细的工具增加模型的选择难度。tool design 的核心原则是 单一职责。

工具粒度决策树

粒度例子问题
过粗process_order(order_id, action) action 含查询/退款/修改模型不知道 action 参数填什么
适中query_order(order_id) refund_order(order_id)每个工具职责单一，描述清晰
过细get_order_status(order_id) get_order_logistics(order_id)模型需要调用多个工具才能完成简单查询
CorpAssist · 工单工具设计反例 设计了一个 handle_ticket(action, ticket_id, data) 工具，action 可以是 "create"、"query"、"update"、"delete"、"assign"、"escalate"。模型频繁传错 action 参数——因为它不知道什么时候该用哪个 action。应该拆分为 6 个独立工具。

描述编写技巧

工具的 description 是模型理解工具的接口文档。好的 description 包含：工具的目的、何时调用、参数的含义和格式、返回什么。

不推荐
  • "查询订单" — 太短，缺触发条件和参数
  • "订单操作工具，输入参数 orderId" — 需要时调用不明确
  • "查数据库" — 从实现角度描述，非用户视角
推荐
  • "当用户询问订单状态或物流信息时调用。输入订单号，返回当前状态、物流单号和预计送达时间。如果订单不存在返回明确提示。"
  • 包含：目的+触发条件+参数说明+返回说明+边界情况

description 模板

text 
工具描述模板：
[当用户需要/XXX场景时]调用此工具。
输入参数：[参数名]是[含义]，格式为[格式示例]。
返回结果：[返回内容说明]，如果[异常情况]则返回[异常信息]。

示例：
当用户查询订单状态或物流信息时调用。
输入 orderId 是订单号，格式如 T2024-001。
返回订单状态、物流单号和预计送达时间。
如果订单不存在则返回"未找到订单: {orderId}"。
如果物流信息未更新则注明"物流信息待更新"。
先想 10 秒：如果工具 description 写得太详细，模型会被绕晕吗？

不会。模型对工具 description 的解析能力很强，详细的内容反而提高准确率。但要避免无关信息——只描述"做什么"和"怎么用"，不要描述"为什么要做"或"内部实现"。通常 50-200 字内最合适。

错误恢复与重试

工具执行可能因为各种原因失败。设计良好的错误处理让 Agent 能从失败中恢复，而不是直接崩溃。错误处理分三个层次：

错误层次处理策略工具实现
参数校验在工具入口校验，返回具体错误Pydantic validator / @ToolParam
业务异常捕获已知异常，返回业务错误try-except 包裹
系统异常捕获兜底，返回降级信息Resilience4j / @safe_tool

Python：错误恢复工具

python 
from tenacity import retry, stop_after_attempt, wait_exponential
import logging

logger = logging.getLogger(__name__)

class ToolError(Exception):
    """工具业务错误"""
    pass

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=1, max=10))
def call_external_api(url: str, params: dict) -> dict:
    """带重试的外部 API 调用"""
    response = requests.post(url, json=params, timeout=5)
    response.raise_for_status()
    return response.json()

@tool
def query_order_with_retry(order_id: str) -> str:
    """带重试和错误恢复的订单查询"""
    try:
        result = call_external_api(
            "http://order-service/api/orders",
            {"order_id": order_id}
        )
        return f"订单 {order_id}: {result['status']}"
    except ValueError as e:
        # 参数错误，不需要重试
        return f"参数错误：{e}"
    except TimeoutError:
        # 超时，tenacity 会自动重试
        return "查询超时，请稍后重试"
    except Exception as e:
        logger.exception(f"订单查询异常: {order_id}")
        return f"系统繁忙，已记录工单 {create_incident(order_id)}"

Java：Resilience4j 包装

java 
@Component
public class ResilientToolWrapper {
    private final Retry retryPolicy;
    private final CircuitBreaker circuitBreaker;

    public ResilientToolWrapper() {
        this.retryPolicy = Retry.of("tool-retry",
            RetryConfig.custom()
                .maxAttempts(3)
                .waitDuration(Duration.ofMillis(500))
                .retryExceptions(TimeoutException.class, IOException.class)
                .build());
        this.circuitBreaker = CircuitBreaker.of("tool-cb",
            CircuitBreakerConfig.custom()
                .failureRateThreshold(50)
                .slidingWindowSize(10)
                .build());
    }

    public <T> T executeWithResilience(Supplier<T> toolCall, String toolName) {
        return Try.ofSupplier(
            CircuitBreaker.decorateSupplier(circuitBreaker,
                Retry.decorateSupplier(retryPolicy, toolCall))
        ).recover(throwable -> {
            log.error("Tool {} failed: {}", toolName, throwable.getMessage());
            return (T) ("工具执行失败，已记录日志。请稍后重试。");
        }).get();
    }
}

测试与 Mock

工具需要有完整的测试覆盖，包括正常路径、异常路径和边界情况。使用 Mock 对象隔离外部依赖，确保测试可重复、可预测。

Python：工具测试

python 
import pytest
from unittest.mock import Mock, patch

@patch("tool_module.order_service")
def test_query_order_success(mock_service):
    mock_service.query.return_value = Order(
        id="T001", status="已发货",
        logistics="SF123", eta="2026-06-01"
    )
    result = query_order("T001")
    assert "已发货" in result
    assert "SF123" in result

@patch("tool_module.order_service")
def test_query_order_not_found(mock_service):
    mock_service.query.return_value = None
    result = query_order("T999")
    assert "未找到" in result

def test_query_order_invalid_id():
    result = query_order("invalid")
    assert "格式错误" in result

@patch("tool_module.order_service")
def test_query_order_timeout(mock_service):
    mock_service.query.side_effect = TimeoutError("timeout")
    result = query_order("T001")
    assert "超时" in result or "重试" in result
常见误区
  • 不测试工具：认为工具只是对 API 的简单包装，忽略参数校验和异常处理路径。
  • 忘记 Mock 外部依赖：测试会真实调用外部 API，导致测试不可重复、依赖网络。
  • 忽略返回值格式：工具返回格式变化后，Agent 端的解析逻辑未同步更新。

常见问题（FAQ）

工具描述中的关键词会帮助模型选择工具吗？
会。模型对 description 做语义理解，关键词确实影响匹配准确率。建议在 description 中包含场景相关的关键词。例如退款工具的 description 包含"退货、退款、取消订单、不满意"等触发词。但不要过度堆砌关键词导致描述不自然。
工具数量上限是多少？
大多数模型能同时接受 30-50 个工具定义。但模型在大量工具中选择的准确率会下降。建议单 Agent 的工具数控制在 10-20 个。超过 20 个时应考虑拆分为多 Agent 系统，每个 Agent 只带自己领域的工具。
工具返回值最长可以多长？
建议在 2000 字符以内。返回值会被注入到 LLM 的上下文窗口中，过长会挤占其他内容的空间。对于大数据量的工具（如查全部订单列表），应该：① 加分页参数；② 只返回汇总信息；③ 截断详情。参考 上下文预算分配。

本章结论

  • 单一职责：每个工具只做一件事，粒度适中。
  • 高质量描述：包含目的、触发条件、参数说明、返回说明、边界情况。
  • 错误分层处理：参数校验、业务异常、系统异常三个层次逐级兜底。
  • 重试策略：使用 tenacity 或 Resilience4j 实现指数退避重试。
  • 全面测试：正常路径、异常路径、边界情况全覆盖。
