三层记忆架构

Agent 的记忆系统分为三层：短期记忆（对话上下文）、长期记忆（外部持久化）、执行状态（Checkpoint）。每层有不同的存储介质、访问频率和生命周期。

短期记忆：对话历史管理

短期记忆是 LLM 上下文窗口中的对话历史部分。如果无限累积，历史会挤占 System Prompt 和工具输出的空间。常见的控制策略包括滑动窗口和滚动摘要。

滑动窗口 vs 滚动摘要
flowchart LR
    subgraph Sliding[滑动窗口]
        direction LR
        H1[第 1 轮] -->|丢弃| H2[第 2-11 轮]
    end
    subgraph Summary[滚动摘要]
        direction LR
        S1[1-5 轮摘要] --> S2[+6-10 轮]
        S2 --> S3[更新摘要]
    end

滑动窗口简单高效但丢失信息；滚动摘要保留关键信息但增加一轮 LLM 调用。

Python：滑动窗口实现

python 
from collections import deque
from typing import List

class SlidingWindowMemory:
    """滑动窗口短期记忆"""

    def __init__(self, max_turns: int = 10, max_tokens: int = 6000):
        self.history = deque(maxlen=max_turns)
        self.max_tokens = max_tokens

    def add_turn(self, user_msg: str, assistant_msg: str):
        self.history.append({
            "user": user_msg,
            "assistant": assistant_msg,
            "timestamp": time.time()
        })

    def get_context(self) -> List[dict]:
        """获取当前可用的对话历史"""
        # 当历史超出最大轮次时，从最早开始丢弃
        context = []
        token_count = 0
        for turn in reversed(self.history):
            turn_tokens = count_tokens(turn["user"]) + count_tokens(turn["assistant"])
            if token_count + turn_tokens > self.max_tokens:
                break
            context.insert(0, turn)
            token_count += turn_tokens
        return context

    def summarize(self, llm) -> str:
        """对旧的历史生成摘要"""
        if len(self.history) < 3:
            return ""
        text = "\n".join(
            f"用户：{t['user']}\n助手：{t['assistant']}"
            for t in list(self.history)[:-2]
        )
        summary = llm.invoke(f"请总结以下对话的关键信息：\n{text}")
        return summary.content

@Configuration
public class MemoryConfig {

    @Bean
    public ChatMemory corpAssistMemory() {
        // 基于消息数量的滑动窗口
        return MessageChatMemory.builder()
            .maxMessages(20)        // 最多保留 20 条消息
            .build();
    }

    @Bean
    public ChatMemory summaryMemory(ChatModel chatModel) {
        // 基于 token 数的摘要记忆
        return TokenWindowChatMemory.builder()
            .maxTokens(6000)
            .chatModel(chatModel)
            .build();
    }
}

// 使用示例
@Service
public class ChatService {
    private final ChatMemory memory;
    private final ChatModel chatModel;

    public ChatService(@Qualifier("corpAssistMemory") ChatMemory memory,
                       ChatModel chatModel) {
        this.memory = memory;
        this.chatModel = chatModel;
    }

    public String chat(String sessionId, String message) {
        // 记录用户消息
        memory.add(sessionId, new UserMessage(message));
        // 获取历史
        List<Message> history = memory.get(sessionId);
        // 构建 prompt
        Prompt prompt = new Prompt(history);
        ChatResponse response = chatModel.call(prompt);
        // 记录助手回复
        memory.add(sessionId, response.getResult().getOutput());
        return response.getResult().getOutput().getContent();
    }
}

长期记忆：外部持久化存储

长期记忆 存储跨会话的关键信息：用户偏好、历史事实、学习到的模式。常用的存储方案包括 Vector DB（语义检索）和 Redis（键值存储）。

长期记忆写入与检索流程
sequenceDiagram
    participant Agent as Agent
    participant Store as 记忆存储
    participant Vector as Vector DB
    participant Redis as Redis

    Agent->>Store: 写入记忆\n用户偏好：夜间模式
    Store->>Vector: 向量化存储
    Store->>Redis: 键值缓存

    Note over Agent,Redis: 新会话开始时
    Agent->>Store: 检索相关记忆
    Store->>Vector: 语义相似度查询
    Store->>Redis: 快速键值查询
    Vector-->>Store: Top-3 相关记忆
    Redis-->>Store: 用户基本信息
    Store-->>Agent: 注入 System Prompt

长期记忆的写入和检索流程：Vector DB 做语义检索，Redis 做快速存取。

Python：长期记忆存储

python 
import redis
import json
from langchain.embeddings import OpenAIEmbeddings

class LongTermMemory:
    """长期记忆：Vector DB + Redis 双层存储"""

    def __init__(self, vector_store, redis_client):
        self.vector_store = vector_store  # Chroma/Pinecone/Milvus
        self.redis = redis_client or redis.Redis(host="localhost", port=6379)

    def remember(self, user_id: str, key: str, value: str, metadata: dict = None):
        """存储一条记忆"""
        # Redis 快速存储
        memory_key = f"memory:{user_id}:{key}"
        self.redis.set(memory_key, json.dumps({
            "value": value,
            "metadata": metadata or {},
            "timestamp": time.time()
        }))
        self.redis.expire(memory_key, 86400 * 30)  # 30 天 TTL

        # Vector DB 语义存储
        doc_text = f"{key}: {value}"
        self.vector_store.add_texts(
            texts=[doc_text],
            metadatas=[{"user_id": user_id, "key": key, **(metadata or {})}]
        )

    def recall(self, user_id: str, query: str, top_k: int = 3) -> list:
        """检索相关记忆"""
        # 语义检索
        docs = self.vector_store.similarity_search(
            query, k=top_k,
            filter={"user_id": user_id}
        )
        return [doc.page_content for doc in docs]

    def forget(self, user_id: str, key: str):
        """删除记忆"""
        self.redis.delete(f"memory:{user_id}:{key}")

Checkpoint 与状态持久化

checkpoint 是 LangGraph 的核心机制，它在 Agent 的每一步执行后保存完整的状态快照。这使得 Agent 可以在异常中断后恢复，也支撑了人工审批（Human-in-the-Loop）等高级模式。

场景Checkpoint 作用恢复方式
系统崩溃从最后 checkpoint 恢复自动恢复
人工审批中断后保存状态等待确认手动恢复
调试回放逐步骤回放执行轨迹调试模式
审计追踪完整记录 Agent 决策过程只读查询
python 
from langgraph.checkpoint import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver
import sqlite3

# 内存 Checkpoint（开发环境）
memory_checkpointer = MemorySaver()

# SQLite Checkpoint（生产环境）
conn = sqlite3.connect("agent_state.db", check_same_thread=False)
sqlite_checkpointer = SqliteSaver(conn)

# 编译图时注入 Checkpointer
graph = graph_builder.compile(checkpointer=sqlite_checkpointer)

# 带 thread_id 的执行（支持跨会话恢复）
config = {"configurable": {"thread_id": "user_session_123"}}
result = graph.invoke({"messages": [user_msg]}, config)

# 从历史 checkpoint 恢复
state = graph.get_state(config)
print(f"当前步骤：{state.values.get('step', 0)}")

# 恢复到指定 checkpoint
graph.update_state(config, {"messages": corrected_msg})

Java：状态持久化

java 
@Service
public class AgentStateService {
    private final JdbcTemplate jdbcTemplate;

    public AgentStateService(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    public void saveCheckpoint(String threadId, int step, String stateJson) {
        jdbcTemplate.update(
            "INSERT INTO agent_checkpoints (thread_id, step, state_json, created_at) "
            + "VALUES (?, ?, ?, NOW()) "
            + "ON DUPLICATE KEY UPDATE state_json = ?, step = ?",
            threadId, step, stateJson, stateJson, step);
    }

    public String getCheckpoint(String threadId) {
        return jdbcTemplate.queryForObject(
            "SELECT state_json FROM agent_checkpoints "
            + "WHERE thread_id = ? ORDER BY step DESC LIMIT 1",
            String.class, threadId);
    }
}
不推荐
  • 无状态 Agent，每次对话从零开始
  • 历史消息无限累积，最终爆窗
  • 不使用 Checkpointer，崩溃后无法恢复
  • 所有会话共享同一个上下文
推荐
  • 三层记忆架构：短期 + 长期 + Checkpoint
  • 滑动窗口或摘要控制历史长度
  • 使用 Checkpointer 实现状态持久化
  • 每个 session/thread 独立隔离

上下文预算分配

即使有滑动窗口和摘要，上下文窗口仍然是有限资源。显式的 上下文预算分配 确保 System Prompt、检索结果、对话历史和工具输出各占合理比例。

典型的 32K 窗口分配方案：System Prompt 15%（~4.8K）、Retrieval 40%（~12.8K）、History 20%（~6.4K）、Tool Output 15%（~4.8K）、Reserve 10%（~3.2K）。

先想 10 秒：如果历史对话占了 80% 的上下文窗口，会发生什么？

System Prompt 和检索结果会被挤出可用窗口。LLM 失去角色约束和行为规范，回答变得不可控。这是最常见的 Agent「失忆」原因——不是模型忘了，而是 System Prompt 被历史挤出了窗口。

常见问题（FAQ）

短期记忆和长期记忆怎么配合使用？
典型的配合模式：每次读取用户消息时，先从长期记忆（Vector DB）检索相关用户偏好和历史事实，注入到 System Prompt 中。当前轮对话的结果写入短期记忆（滑动窗口）。会话结束时（或达到窗口上限时），将旧的历史摘要后写入长期记忆。
Checkpoint 存储会不会太慢？
Checkpoint 是异步写入的，不会阻塞 Agent 执行。LangGraph 的 MemorySaver 和 SqliteSaver 都支持批量写入。对于高并发场景，建议使用 RedisSaver 或 PostgresSaver，写入延迟在毫秒级。
长期记忆可以跨用户共享吗？
可以，但需要严格权限控制。企业知识库（如产品手册、政策文档）可以跨用户检索。个人偏好（如用户喜欢简短回答）应隔离。建议在 Vector DB 中使用 metadata 字段标注权限范围，检索时按权限过滤。

本章结论

  • 三层记忆：短期（会话内）、长期（跨会话）、Checkpoint（执行状态），各司其职。
  • 滑动窗口：控制对话历史长度，防止上下文溢出。
  • 长期记忆：Vector DB 做语义检索，Redis 做快速键值存取。
  • Checkpoint：LangGraph 核心机制，支撑中断恢复和人工审批。
  • 预算分配：显式规划每层内存用量，防止资源争抢。
