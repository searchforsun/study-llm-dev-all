为什么要学本章

Agent 运行在不确定的环境中——LLM 可能输出异常、工具可能超时、用户可能输入恶意内容。reliability 通过循环检测、熔断、重试和降级策略确保 Agent 在故障面前仍能稳定运行。

  • 死循环必须检测：重复工具调用、相同参数反复执行是常见故障模式。

  • 降级比崩溃好：Agent 无法完成任务时，降级返回部分结果或转人工。

循环检测与熔断

死循环是 Agent 最常见的故障模式。检测死循环需要监控工具调用序列：

python 
from collections import defaultdict
import hashlib

class LoopDetector:
    """Agent 死循环检测器"""

    def __init__(self, max_repeated_calls=3, window_size=6):
        self.call_history: list = []
        self.max_repeated = max_repeated_calls
        self.window_size = window_size

    def record_call(self, tool_name: str, args: dict, result: str):
        """记录一次工具调用"""
        call_hash = self._hash_call(tool_name, args)
        self.call_history.append({
            "tool": tool_name,
            "args_hash": call_hash,
            "result_hash": hashlib.md5(result.encode()).hexdigest()
        })

    def detect_loop(self) -> tuple[bool, str]:
        """检测死循环"""
        if len(self.call_history) < self.window_size:
            return False, ""

        recent = self.call_history[-self.window_size:]

        # 检测 1: 相同工具相同参数连续调用
        for tool in set(c["tool"] for c in recent):
            calls = [c for c in recent if c["tool"] == tool]
            if len(calls) >= self.max_repeated:
                args_hashes = [c["args_hash"] for c in calls[-self.max_repeated:]]
                if len(set(args_hashes)) == 1:
                    return True, f"死循环检测: 工具 {tool} 相同参数连续调用 {self.max_repeated} 次"

        # 检测 2: 来回切换两个工具
        if len(recent) >= 4:
            pairs = [(recent[i]["tool"], recent[i+1]["tool"])
                     for i in range(0, len(recent)-1)]
            if len(set(pairs)) == 1 and len(pairs) >= 3:
                return True, f"死循环检测: 工具间来回切换 {pairs[0]}"

        return False, ""

    def _hash_call(self, tool_name: str, args: dict) -> str:
        return hashlib.md5(
            f"{tool_name}:{json.dumps(args, sort_keys=True)}".encode()
        ).hexdigest()

# 集成到 Agent
detector = LoopDetector()

def tool_node_with_detection(state: AgentState) -> AgentState:
    for tool_call in state["messages"][-1].tool_calls:
        result = execute_tool(tool_call)
        detector.record_call(
            tool_call["name"],
            tool_call["args"],
            result
        )
        is_loop, reason = detector.detect_loop()
        if is_loop:
            state["loop_detected"] = True
            state["messages"].append(AIMessage(content=f"已检测到循环，转人工处理：{reason}"))
            break
        state["messages"].append({"role": "tool", "content": result})
    return state

Java 熔断器实现

java 
@Component
public class AgentCircuitBreaker {
    private final Map<String, AtomicInteger> failureCounts = new ConcurrentHashMap<>();
    private final Map<String, Long> circuitOpenUntil = new ConcurrentHashMap<>();
    private static final int THRESHOLD = 3;
    private static final long TIMEOUT_MS = 30_000;

    public boolean isAvailable(String toolName) {
        Long until = circuitOpenUntil.get(toolName);
        if (until != null) {
            if (System.currentTimeMillis() < until) {
                return false;
            }
            circuitOpenUntil.remove(toolName);
            failureCounts.put(toolName, new AtomicInteger(0));
        }
        return true;
    }

    public void recordFailure(String toolName) {
        AtomicInteger count = failureCounts.computeIfAbsent(
            toolName, k -> new AtomicInteger(0));
        if (count.incrementAndGet() >= THRESHOLD) {
            circuitOpenUntil.put(toolName,
                System.currentTimeMillis() + TIMEOUT_MS);
            log.warn("Circuit opened for tool: {}", toolName);
        }
    }

    public void recordSuccess(String toolName) {
        failureCounts.computeIfAbsent(toolName, k -> new AtomicInteger(0)).set(0);
    }
}

重试策略与退避

Agent 工具调用失败时，重试是常见的恢复策略。但重试需要有策略——不加控制的立即重试可能加剧故障。

python 
import time
import random
from functools import wraps

def retry_with_backoff(
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 30.0,
    jitter: bool = True
):
    """指数退避重试装饰器"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exception = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except (TimeoutError, ConnectionError) as e:
                    last_exception = e
                    delay = min(base_delay * (2 ** attempt), max_delay)
                    if jitter:
                        delay += random.uniform(0, delay * 0.1)
                    if attempt < max_retries - 1:
                        time.sleep(delay)
            raise last_exception
        return wrapper
    return decorator

# 使用
@retry_with_backoff(max_retries=3, base_delay=1.0)
def call_external_api(url: str) -> dict:
    response = requests.get(url, timeout=5)
    response.raise_for_status()
    return response.json()

降级与 Fallback

当重试和熔断都无效时，Agent 需要降级。降级策略确保 Agent 在故障下仍能提供一定程度的服务。

降级级别行为适用场景
L1 简化跳过非关键工具调用，只返回已收集信息部分工具异常
L2 缓存返回缓存的历史结果查询类工具超时
L3 转人工将对话转交人工客服核心工具全部故障
python 
class DegradationManager:
    """降级管理器"""

    def __init__(self):
        self.failure_count = defaultdict(int)
        self.degradation_level = 0  # 0=正常, 1=简化, 2=缓存, 3=转人工

    def on_tool_failure(self, tool_name: str):
        self.failure_count[tool_name] += 1
        total_failures = sum(self.failure_count.values())
        if total_failures >= 5:
            self.degradation_level = 3  # 转人工
        elif total_failures >= 3:
            self.degradation_level = 2  # 使用缓存
        elif total_failures >= 1:
            self.degradation_level = 1  # 简化

    def should_skip_tool(self, tool_name: str) -> bool:
        return self.degradation_level >= 1 and tool_name in self.failure_count

    def get_cache_result(self, tool_name: str, args: dict) -> str:

常见问题（FAQ）

熔断器打开后，什么时候关闭？
熔断器打开后进入半开状态（通常 30-60 秒后），允许少量请求通过测试下游是否恢复。如果请求成功则关闭熔断器，如果仍然失败则继续保持打开状态。LangGraph 内置的熔断器使用 Resilience4j 的实现。
降级和 Fallback 有什么区别？
降级（Degrade）是主动降低服务质量，在故障发生前或发生时减少功能。Fallback 是执行备选方案，当主方案失败时切换到备选。例如：降级可能是跳过非必须的工具调用，Fallback 可能是使用缓存数据替代实时查询。两者常常配合使用。
重试次数太多会有什么问题？
过多重试会：① 增加 Agent 执行延迟，用户等待时间变长；② 放大对下游系统的压力，可能导致系统崩溃；③ 浪费 Token 成本，每次重试都产生 LLM 调用。建议重试不超过 3 次，配合指数退避和熔断器使用。

本章结论

  • 死循环检测：基于工具调用序列的哈希和模式匹配，检测重复调用。
  • 熔断器：连续失败 N 次后暂时禁用工具，避免级联故障。
  • 重试策略：指数退避 + 随机抖动，避免重试风暴。
  • 降级策略：三级降级（简化/缓存/转人工），确保故障下仍能服务。
  • 容量规划：基于历史数据规划 Agent 并发和限流。
