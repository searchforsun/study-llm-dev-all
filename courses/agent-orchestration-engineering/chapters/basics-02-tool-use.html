为什么要学本章

工具是 Agent 与外部世界交互的唯一通道。Function Calling 的可靠性直接决定 Agent 的成败——工具定义不清、参数错误、结果不可解析是 Agent 项目最常见的三大故障源。本章从原理到实践，覆盖工具定义、Schema 设计、参数校验、错误处理和安全的完整链路。

学完你能
  • 理解 Function Calling 的底层原理：模型如何声明工具调用需求
  • 用 Python Pydantic 和 Java Record 定义高质量工具 Schema
  • 实现参数校验、错误处理和超时控制的健壮工具层
  • 避免工具设计中的常见陷阱（description 缺失、参数歧义等）
本章先记住 3 件事
  • 模型声明工具：LLM 在响应中返回 tool_calls 结构，不是 LLM 直接执行函数。
  • Schema 即契约：工具定义（参数名、类型、描述）是模型理解工具的接口说明书。
  • 描述决定调用率：工具描述越清晰，模型正确调用的概率越高。

Function Calling 底层原理

Function Calling 并非模型「调用函数」，而是模型在生成文本时声明「我需要调用某个函数」。Agent 框架拦截这个声明，执行对应函数，将结果以消息形式返回给模型继续推理。

Function Calling 完整流程
sequenceDiagram
    participant Dev as 开发者
    participant Model as LLM
    participant Agent as Agent Runtime
    participant API as 企业 API

    Dev->>Model: 注册工具定义（OpenAPI Schema）
    Note over Model: 模型内化工具描述
    Agent->>Model: 用户问题 + 可用工具列表
    Model-->>Agent: 响应 + tool_calls[query_order("T001")]
    Agent->>Agent: 解析 tool_calls
    Agent->>API: 执行 query_order("T001")
    API-->>Agent: 返回订单数据
    Agent->>Model: 追加 tool_result 消息
    Model-->>Agent: 生成最终回复

Function Calling 的完整数据流：工具定义、声明、执行、结果回传。

模型在训练阶段学习了大量 API 调用示例，因此能够理解工具的 JSON Schema 定义。当 tool schema 足够清晰时，模型可以在推理时自动判断是否需要调用工具、传入什么参数。

先想 10 秒：如果没有 Function Calling 能力，Agent 如何实现工具调用？

退化为 prompt 解析模式——让 LLM 输出固定格式字符串（如 TOOL_CALL: query_order|T001），Agent 再用正则或字符串解析提取。这种方式可靠性差，模型容易输出格式错乱。Function Calling 是原生支持的结构化输出，可靠性远超文本解析。

工具定义与 Schema 设计

工具 Schema 是模型理解工具的接口契约。设计良好的 Schema 需要包含：清晰的名称、准确的参数描述、合理的参数类型和必要的校验规则。

Python：Pydantic 工具定义

python 
from pydantic import BaseModel, Field
from typing import Optional
from langchain_core.tools import tool

class QueryOrderInput(BaseModel):
    """查询订单的输入参数"""
    order_id: str = Field(description="订单号，格式如 T2024-001")
    customer_phone: Optional[str] = Field(
        default=None,
        description="客户手机号后四位，用于身份验证"
    )

@tool(args_schema=QueryOrderInput)
def query_order(order_id: str, customer_phone: Optional[str] = None) -> str:
    """查询订单当前状态和物流信息。

    该工具从企业 ERP 系统获取订单数据，
    返回状态、物流单号和预计送达时间。
    敏感信息脱敏后返回。
    """
    result = order_service.query(order_id)
    return f"订单 {order_id}: {result.status}, 物流: {result.logistics}"

# 查看自动生成的 JSON Schema
print(query_order.args_schema.schema())

Java：Spring AI Alibaba 工具定义

java 
@Component
public class CorpAssistTools {

    @Tool(name = "query_order",
          description = "查询订单状态和物流信息。" +
                        "从企业 ERP 获取数据，返回状态、物流单号和预计送达。")
    public String queryOrder(
            @ToolParam(description = "订单号，格式如 T2024-001")
            String orderId,

            @ToolParam(description = "客户手机号后四位，用于身份验证（可选）",
                       required = false)
            String customerPhone) {

        if (orderId == null || orderId.isBlank()) {
            return "错误：订单号不能为空";
        }
        Order order = orderService.findById(orderId);
        if (order == null) {
            return "未找到订单: " + orderId;
        }
        return "订单 %s 状态：%s，物流单号：%s，预计送达：%s"
            .formatted(order.id(), order.status(),
                       order.logisticsNo(), order.estimatedDelivery());
    }

    @Tool(name = "create_ticket",
          description = "创建客服工单。需要用户 ID、问题类型和问题描述。")
    public String createTicket(
            @ToolParam(description = "用户 ID") String userId,
            @ToolParam(description = "问题类型：退款/退货/咨询/投诉")
            String issueType,
            @ToolParam(description = "问题详细描述，不少于 10 个字")
            String description) {

        Ticket ticket = ticketService.create(userId, issueType, description);
        return "工单已创建，编号：%s，预计处理时间：%s"
            .formatted(ticket.id(), ticket.estimatedResponse());
    }
}

      },

      }
    },
    "required": ["order_id"]
  }
}
不推荐
  • 工具方法无 description，模型不知道何时调用
  • 参数名含糊（id、data）
  • 返回原始 JSON 或 null 不加工
  • 一个工具方法做多件事（查询+创建+删除）
推荐
  • 每个 @Tool 写清晰中文 description
  • 参数名自解释（orderId、issueType）
  • 返回格式化的可读文本
  • 每个工具只做一件事（单一职责）

参数校验与错误处理

工具调用的可靠性取决于三个环节：参数校验、执行保护和结果格式化。模型可能传入不符合预期的参数，工具本身也可能抛出异常，这些都需要 Agent 层妥善处理。

异常类型典型场景处理方法
参数缺失模型未传必填参数在工具开头校验，返回友好错误
参数格式错误订单号格式不对正则校验，提示正确格式
业务异常订单不存在、库存不足返回业务错误描述，让模型决策
系统异常数据库连接超时捕获异常，返回降级信息
超时外部 API 超过 5 秒未响应设超时时间，返回超时错误

Python：健壮的工具实现

python 
import re
from functools import wraps
from typing import Any

def safe_tool(func):
    """工具安全包装器"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        try:
            result = func(*args, **kwargs)
            # 限制返回长度
            if isinstance(result, str) and len(result) > 2000:
                result = result[:2000] + "...(截断)"
            return result
        except ValueError as e:
            return f"参数错误：{e}"
        except TimeoutError:
            return "工具调用超时，请稍后重试"
        except Exception as e:
            return f"系统错误，已记录日志：{type(e).__name__}"
    return wrapper

@tool
@safe_tool
def query_order(order_id: str) -> str:
    """查询订单状态"""
    if not re.match(r'^T\d{4}-\d{3,}$', order_id):
        raise ValueError(f"订单号格式错误：{order_id}（正确格式：T2024-001）")
    order = order_service.query(order_id)
    if order is None:
        return f"未找到订单：{order_id}"
    return format_order(order)

Java：Resilience4j 包装

java 
@Component
public class ResilientTools {
    private static final Logger log = LoggerFactory.getLogger(ResilientTools.class);

    private final OrderService orderService;
    private final Retry retryPolicy;

    public ResilientTools(OrderService orderService) {
        this.orderService = orderService;
        this.retryPolicy = Retry.of("tool-retry",
            RetryConfig.custom()
                .maxAttempts(3)
                .waitDuration(Duration.ofSeconds(1))
                .retryExceptions(TimeoutException.class, IOException.class)
                .build());
    }

    @Tool(name = "query_order_resilient",
          description = "带重试的订单查询")
    public String queryOrder(String orderId) {
        return Try.ofSupplier(() ->
                Retry.decorateSupplier(retryPolicy, () -> {
                    Order o = orderService.findById(orderId);
                    if (o == null) return "未找到订单: " + orderId;
                    return formatOrder(o);
                })
            ).recover(e -> "查询异常：" + e.getMessage())
            .get();
    }
}
常见误区
  • 工具抛出异常 = Agent 崩溃：Agent 框架应捕获异常，以消息形式返回给模型，让模型决定是否重试或降级。
  • 返回过长不截断：工具返回 1 万行日志会撑爆上下文。建议截断至 2000 字符内。
  • 错误信息含敏感数据：异常信息可能包含堆栈和数据库细节，返回给模型前需脱敏。

安全调用策略

工具是 Agent 的风险入口。不良设计的工具可能被模型（或被用户 prompt injection 诱导的模型）滥用，造成数据泄露或业务损失。

工具安全调用分层
flowchart TB
    Input[用户输入] -->|1. 输入过滤| Sanitize[XSS/SQL 注入过滤]
    Sanitize -->|2. 权限检查| Auth[工具调用权限校验]
    Auth -->|3. 限流| RateLimit[QPS/并发限制]
    RateLimit -->|4. 执行| Execute[工具执行]
    Execute -->|5. 输出过滤| Output[脱敏/合规审查]
    Output -->|6. 审计| Audit[全链路日志]

六层安全防护：从输入到输出覆盖全链路。

安全层措施实现方式
输入过滤参数正则校验、长度限制Pydantic validator / @ToolParam 约束
工具白名单仅允许注册的工具被调用allowed_tools 配置列表
操作权限读写分离、高危操作确认require_confirmation 标志
限流熔断单工具 QPS 上限Resilience4j RateLimiter
输出脱敏手机号/身份证脱敏脱敏注解或正则替换
审计日志全量工具调用记录AOP 切面 + 数据库

常见问题（FAQ）

模型不调用工具怎么办？
首先检查工具 description 是否清晰。常见原因：工具描述模糊、参数描述不全、模型不了解何时该用此工具。解决方案：① 在 System Prompt 中举例说明何时使用该工具；② 增加 strict=True 强制模型必须使用工具；③ 切换到 Function Calling 能力更强的模型（如 Qwen、GPT-4、Claude）。
模型传错了参数怎么办？
模型不会「故意」传错参数，通常是因为：① 参数描述不够清晰；② 参数名含义模糊；③ 用户提供的信息不全。建议在工具开头做参数校验，返回友好错误让模型重新尝试。也可以让工具参数设计更宽松——例如支持模糊匹配的订单号。
工具可以调用另一个工具吗？
工具本身不应直接调用其他工具——这是编排层的职责。如果你发现一个工具需要调用另一个工具，说明这两个功能应该由 Agent 的 ReAct 循环串联，而不是在工具内部耦合。见 工具设计最佳实践 中的粒度讨论。

本章结论

  • Function Calling 是声明式：模型声明需要调用工具，Agent 运行时负责执行和结果回传。
  • Schema 即接口契约：参数的名称、类型和描述决定了模型能否正确调用工具。
  • 错误处理兜底：工具必须处理参数校验异常、业务异常和系统异常，返回友好错误让模型继续。
  • 安全分六层：从输入过滤到审计日志，每一层都不可或缺。
  • 双栈一致：Python Pydantic 和 Java @ToolParam 在 Schema 设计理念上完全一致。
