Chain-of-Thought 推理

Chain-of-Thought（CoT）通过在 prompt 中注入「一步一步思考」的指令，引导模型在给出最终答案前展示推理过程。在 Agent 中，CoT 不仅是推理策略，更是一种内置的 System Prompt 设计模式。

CoT 推理与直接回答对比
sequenceDiagram
    participant U as 用户
    participant LLM as LLM

    Note over U,LLM: 直接回答模式
    U->>LLM: 我的订单 T001 现在到哪了？
    LLM-->>U: 已发货，预计明天到达。
    Note over U,LLM: 缺少推理过程，可能出错

    Note over U,LLM: CoT 推理模式
    U->>LLM: 我的订单 T001 现在到哪了？
    Note over LLM: 推理：用户想知道物流状态\n我需要查订单 API\n传入参数 order_id=T001
    LLM-->>U: 我已查询订单 T001，\n当前状态：运输中，\n物流单号：SF123456，\n预计明天 18:00 前送达。

CoT 让 Agent 显式展示推理链，既提高准确率又便于调试。

CoT 在工具调用中的融合

在 Agent 设计中，CoT 不是独立步骤——它融入 ReAct 循环的"思考"阶段。System Prompt 中应明确指示模型先推理再行动：

text 
你是 CorpAssist S2 客服 Agent。
调用工具前，请先思考：
1. 用户的核心需求是什么？
2. 需要调用哪些工具？参数是什么？
3. 工具返回后如何解读结果？
4. 最终回答应该包含哪些信息？

请用"思考："开头展示你的推理过程，
然后用"行动："指示要调用的工具。
不推荐
  • System Prompt 无 CoT 引导
  • 要求模型直接输出答案，跳过推理
  • 不区分思考和行动阶段
推荐
  • 显式要求「请先思考后回答」
  • 提供思考框架模板
  • 工具调用前后都有推理步骤
先想 10 秒：如果用户说「直接给我答案，别废话推理过程」，还能用 CoT 吗？

可以。推理过程对模型是必要的内部步骤，但最终回答可以只展示结论。在 System Prompt 中说明「推理过程仅供内部使用，最终回答简洁直接」。Agent 框架可以在最终输出前过滤掉推理轨迹。

任务分解与子目标

task decomposition 是将复杂目标递归拆解为可执行子任务的能力。它位于规划层的核心——没有分解，Agent 就无法处理超出单步推理的复杂请求。

任务分解示例：生成季度销售报告
flowchart TD
    Root[生成 Q2 销售报告] --> Sub1[1. 检索各区域销售数据]
    Root --> Sub2[2. 数据分析与趋势计算]
    Root --> Sub3[3. 生成报告文本]
    Root --> Sub4[4. 格式化输出]

    Sub1 --> Sub1a[华东区数据]
    Sub1 --> Sub1b[华南区数据]
    Sub1 --> Sub1c[华北区数据]

    Sub2 --> Sub2a[计算环比增长率]
    Sub2 --> Sub2b[识别 Top 3 产品]
    Sub2 --> Sub2c[生成趋势结论]

    Sub3 --> Sub3a[写执行摘要]
    Sub3 --> Sub3b[写区域分析]
    Sub3 --> Sub3c[写建议部分]

任务分解将「生成报告」拆为 4 个主步骤和 9 个叶子子任务，每个都可独立执行。

Python：任务分解器

python 
from pydantic import BaseModel
from typing import List

class SubTask(BaseModel):
    """子任务定义"""
    id: str
    description: str
    depends_on: List[str] = []  # 前置依赖的任务 ID
    tool: str = ""              # 需要用到的工具
    status: str = "pending"     # pending/running/done/failed

class TaskDecomposer:
    """将复杂任务拆解为可执行的子任务列表"""

    def __init__(self, llm):
        self.llm = llm

    def decompose(self, task: str, context: dict = None) -> List[SubTask]:
        prompt = f"""将以下复杂任务拆解为可执行的子任务列表。
每个子任务应该：
- 独立可执行
- 有明确的输入输出
- 标明前置依赖

任务：{task}
上下文：{context or '无'}

输出格式：
1. 子任务描述 | 依赖 | 工具
2. 子任务描述 | 依赖 | 工具
..."""
        response = self.llm.invoke(prompt)
        return self._parse_subtasks(response.content)

    def _parse_subtasks(self, text: str) -> List[SubTask]:
        subtasks = []
        for i, line in enumerate(text.strip().split("\n")):
            parts = [p.strip() for p in line.split("|")]
            if len(parts) >= 1:
                subtasks.append(SubTask(
                    id=f"step_{i+1}",
                    description=parts[0],
                    depends_on=parts[1].split(",") if len(parts) > 1 and parts[1] != "无" else [],
                    tool=parts[2] if len(parts) > 2 else ""
                ))
        return subtasks

Java：任务分解实现

java 
public record SubTask(
    String id,
    String description,
    List<String> dependsOn,
    String tool,
    String status
) {
    public SubTask(String id, String description) {
        this(id, description, List.of(), "", "pending");
    }
}

@Service
public class TaskDecomposer {
    private final ChatModel chatModel;

    public TaskDecomposer(ChatModel chatModel) {
        this.chatModel = chatModel;
    }

    public List<SubTask> decompose(String task) {
        String prompt = """
            将以下任务拆解为可执行的子任务列表。
            每个子任务一行，格式：描述 | 依赖步骤ID（逗号分隔） | 所需工具
            任务：%s
            """.formatted(task);

        String response = chatModel.call(prompt);
        return parseSubtasks(response);
    }

    private List<SubTask> parseSubtasks(String text) {
        List<SubTask> tasks = new ArrayList<>();
        int i = 1;
        for (String line : text.split("\n")) {
            line = line.trim();
            if (line.isEmpty()) continue;
            String[] parts = line.split("\\|");
            String desc = parts[0].trim();
            SubTask task = new SubTask("step_" + i, desc);
            tasks.add(task);
            i++;
        }
        return tasks;
    }
}

动态重规划

Agent 执行过程中，外部环境可能变化——API 返回空数据、某个步骤失败、用户中途改变需求。静态的初始计划无法应对这些变化，Agent 需要具备 动态重规划 能力。

动态重规划流程
flowchart LR
    Start[初始计划] --> Execute[执行步骤]
    Execute --> Check[检查结果]
    Check -->|正常| Next[下一步骤]
    Check -->|异常| Replan[重规划]
    Next -->|全部完成| Done[完成]
    Next --> Execute
    Replan -->|调整计划| Execute
    Replan -->|无法完成| Escalate[升级到人工]

动态重规划在每一步执行后检查结果，异常时触发计划调整或人工升级。

触发条件重规划策略示例场景
工具返回空调整参数重试或换工具查订单无结果→尝试按手机号查询
步骤失败标记失败、跳过或重试数据源 A 超时→切到数据源 B
用户插入新需求暂停当前计划，合并新需求用户说「顺便查一下退款进度」

Plan-Execute 引擎设计

将规划与执行分离的 Plan-Execute 引擎是 Agent 架构中最常见的实现。它包含三个核心组件：Planner（规划器）、Executor（执行器）、Monitor（监控器）。

python 
class PlanExecuteEngine:
    """规划-执行引擎"""
    def __init__(self, planner_llm, executor_llm, tools: dict):
        self.planner = planner_llm
        self.executor = executor_llm
        self.tools = tools
        self.max_retries = 2
        self.max_steps = 20

    def run(self, task: str) -> dict:
        context = {"task": task, "results": {}, "step": 0}
        # 阶段 1：规划
        plan = self.planner.plan(task)
        context["plan"] = plan

        # 阶段 2：循环执行
        while context["step"] < self.max_steps:
            current = self._get_next_step(plan, context)
            if current is None:
                break

            success, result = self._execute_step(current, context)
            if not success:
                plan = self.planner.replan(task, context, current)
                continue

            context["results"][current.id] = result
            context["step"] += 1

        return context

    def _execute_step(self, step, context):
        for attempt in range(self.max_retries):
            try:
                if step.tool:
                    result = self.tools[step.tool](**context.get("params", {}))
                else:
                    result = self.executor.invoke(step.description)
                return True, result
            except Exception as e:
                if attempt == self.max_retries - 1:
                    return False, str(e)
        return False, "max retries exceeded"

Java Plan-Execute 引擎

java 
@Service
public class PlanExecuteEngine {
    private final Planner planner;
    private final ToolExecutor toolExecutor;
    private final ChatModel executorModel;
    private static final int MAX_STEPS = 20;
    private static final int MAX_RETRIES = 2;

    public ExecutionContext execute(String task) {
        ExecutionContext ctx = new ExecutionContext(task);

        // Phase 1: Plan
        List<Step> plan = planner.createPlan(task);
        ctx.setPlan(plan);

        // Phase 2: Execute with dynamic replanning
        while (ctx.getCurrentStepIndex() < plan.size()
               && ctx.getTotalSteps() < MAX_STEPS) {

            Step step = plan.get(ctx.getCurrentStepIndex());
            ExecutionResult result = executeWithRetry(step, ctx);

            if (!result.isSuccess()) {
                // Trigger replanning
                List<Step> newPlan = planner.replan(ctx, step);
                ctx.setPlan(newPlan);
                continue;
            }

            ctx.addResult(step.id(), result.getOutput());
            ctx.incrementStep();
        }
        return ctx;
    }

    private ExecutionResult executeWithRetry(Step step, ExecutionContext ctx) {
        for (int i = 0; i < MAX_RETRIES; i++) {
            try {
                String output = step.tool() != null
                    ? toolExecutor.execute(step.tool(), step.params())
                    : executorModel.call(step.description());
                return new ExecutionResult(true, output);
            } catch (Exception e) {
                log.warn("Step {} failed, attempt {}/{}",
                    step.id(), i + 1, MAX_RETRIES);
            }
        }
        return new ExecutionResult(false, "Max retries exceeded");
    }
}

常见问题（FAQ）

CoT 一定会提高准确率吗？
研究表明 CoT 对需要多步推理的任务（数学、逻辑、规划）提升显著，但对常识性问题反而可能增加 token 消耗无实质改善。建议根据任务类型决定是否启用 CoT——简单问答用直接模式，复杂推理用 CoT。
任务分解到什么粒度合适？
每个子任务应该在 1-2 次工具调用或 LLM 调用内完成。如果一个子任务本身需要再分解 3 层，说明粒度太粗。通常一个复杂任务的叶子节点数控制在 5-15 个之间。太多则管理成本高，太少则子任务本身过于复杂。
动态重规划会不会无限循环？
会。必须设置最大重规划次数（通常 3-5 次）。超过上限后触发人工升级或返回当前已收集的所有信息。在 Plan-Execute 引擎中，重规划计数器和步骤计数器应该独立追踪，任何一个达到上限都终止执行。

本章结论

  • CoT 推理：显式推理链提高准确率，融入 System Prompt 作为 Agent 的思考模板。
  • 任务分解：将复杂目标拆为独立子任务是规划的核心，粒度控制在 5-15 个叶子节点。
  • 动态重规划：执行结果异常时触发计划调整，设置最大重规划次数防止无限循环。
  • Plan-Execute：Planner + Executor + Monitor 三层架构，Python 和 Java 实现模式一致。
  • 实际应用：CorpAssist S3 办公自动化场景中，Plan-Execute 引擎处理报告生成、数据分析等复杂任务。
