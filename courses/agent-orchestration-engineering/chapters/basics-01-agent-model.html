为什么要学本章

CorpAssist 的 S2 智能客服与 S3 办公自动化都不是「一问一答」的 Chat，而是多步决策 + 工具调用。若把 Agent 当成「更聪明的 Prompt」，后面 LangGraph 的节点、边与步数预算都会变成背 API。本章用 ReAct 循环 建立概率理解 → 确定性执行的心智模型（NS2）。

本章先记住 3 件事
  • Agent ≠ Chat：多轮内部循环，每轮可能调工具；必须设 步数预算。
  • ReAct 骨架：Thought → Action → Observation → 再 Thought；Observation 来自真实工具，不是模型编造。
  • 双栈对照：LangGraph StateGraph 显式建模循环；SAA Graph 提供 Java 侧同类能力，FastAPI 暴露 Python Agent 供 Spring 调用。

感知-规划-行动

CorpAssist S2 · 查订单（45 秒） 用户：「我昨天下的单 #8821 到哪了？」Agent 需读会话状态、识别查单意图、调用订单 API、把结构化结果转成自然语言——中间任何一步都不能让模型「假装查过了」。

Agent 范式 把 LLM 放在闭环里：读状态 → 规划 → 调工具 → 读观察 → 再规划。CorpAssist 里状态包括用户消息、DST 槽位、上一轮工具结果。

ReAct 四步与职责边界

步骤谁负责CorpAssist 示例生产约束
感知 应用 + LLM 解析意图「查物流」、读取 session 中的 order_id 敏感字段脱敏后再进 prompt
规划 LLM（受 prompt/路由约束） 选择 get_order_status(order_id) 工具必须在白名单内；禁止自由 SQL
行动 工具运行时（确定性） HTTP 调订单服务，返回 JSON 超时、鉴权、幂等由代码保证
观察 工具返回 → 写入 state {"status":"shipped","eta":"..."} 截断过长 observation，避免淹没上下文

循环示意（Python 伪代码）

python 
MAX_STEPS = 8
state = {"messages": [], "steps": 0}

while not done(state) and state["steps"] < MAX_STEPS:
    thought = llm.plan(state)          # 规划：选工具或直接回复
    if thought.action is None:
        return thought.final_answer      # 结束
    observation = tools.run(            # 行动 + 观察（必须真实执行）
        thought.action.name,
        thought.action.args,
        timeout=5.0,
    )
    state = append_observation(state, observation)
    state["steps"] += 1

raise StepBudgetExceeded()             # 超限 → 降级转人工

Observation 必须来自工具；若模型跳过工具直接回答物流状态，即为幻觉改单/查单类事故。

反例 vs 推荐：谁执行「改地址」

不推荐
  • Prompt 写「你可以帮用户改地址」但不接 API
  • 模型输出「已为您修改」却无 tool use
  • 无步数上限，Planning 死循环
推荐
  • 改地址走 update_address 工具 + 业务规则校验
  • 高风险操作 interrupt 人工审批（见 人工审批与中断）
  • reasoning trace 入库便于排障
先想 10 秒：Observation 能否由 LLM 自己「编造」以加快回复？

不能。一旦允许编造，NS2 确定性桥梁失效：用户看到成功，后台无真实 API 调用。生产环境 Observation 只能来自工具/MCP/业务 API 的返回值（可截断，不可虚构）。

与 SAA 对照

CorpAssist 双栈：Python 轨用 LangGraph 实现 Agent 能力面；Java 轨用 Spring AI Alibaba 的 ReactAgent/Graph 做治理与对接企业中间件。二者语义对齐，不是逐行 API 相同。

S3 办公自动化：多步仍是一类 ReAct

S3 场景（审批汇总、跨系统填表）往往 >3 步。架构上仍是一系列 Thought-Action-Observation，但必须加执行-观察-反思检查点，避免长规划偏航——这在 失败、重试与降级 展开。

LangGraph 和「自己写 while 循环调 OpenAI」差在哪？
自写循环很快失控：无 checkpoint、无可视化 trace、难加 interrupt。LangGraph 把状态、边、持久化一等公民化，便于生产排障与评测回归。
是否两套 Agent 各写一套工具？
工具契约（OpenAPI/MCP）应共享；Python 与 Java 各实现适配层，避免业务 API 被调用两次且规则不一致。

本章结论

  • 感知-规划-行动：LLM 负责理解与规划，业务动作必须经工具/API 执行，Observation 不可编造。
  • 与 SAA 对照：LangGraph 与 SAA Graph 职责对齐；CorpAssist 用双栈分工而非重复造轮子。
