面试：Agent 架构设计

以下是 Agent 架构面试中的高频问题及回答框架：

Q1: 你如何为业务场景选择合适的 Agent 范式？

回答框架：从三个维度评估——任务复杂度（步骤数量和依赖关系）、实时性要求（用户能等多久）、质量要求（输出需要审核吗）。简单场景用 ReAct，多步骤用 Plan-Execute，高精度叠加 Reflexion。最好举 CorpAssist 的例子：S2 客服用 ReAct（实时交互），S3 办公自动化用 Plan-Execute（复杂报告任务）。

Q2: Agent 工具调用失败怎么办？

回答框架：分层处理——参数校验：在工具入口校验并返回具体错误；业务异常：捕获已知异常返回友好提示；系统异常：使用重试和熔断器，最终兜底转人工。关键点：异常信息要让模型能理解并决定下一步，不要抛到外层。

Q3: 多 Agent 系统中如何保证数据一致性？

回答框架：核心策略——使用共享存储（Redis/数据库）持久化关键状态；每个操作保持幂等性；使用 Saga 模式管理跨 Agent 事务；设置补偿操作回滚失败步骤。参考分布式系统设计模式。

Q4: 如何评测 Agent 的输出质量？

回答框架：离线+在线双轨。离线：构建标注数据集，使用 LLM-as-Judge 评分。在线：CSAT、转人工率、首次解决率。CI 集成：每次变更后运行回归测试，设置质量门禁。参考 Agent 评测方法论。

死循环诊断与排查

死循环是 Agent 最常见的生产故障。排查方法论：

死循环排查流程图
flowchart TD
    Symptom[Agent 执行超时/步数超限] --> CheckTrace[检查 LangSmith Trace]
    CheckTrace -->|工具调用重复| Pattern1[模式 1: 相同工具相同参数]
    CheckTrace -->|工具间来回跳| Pattern2[模式 2: AB 工具切换]
    CheckTrace -->|LLM 输出循环| Pattern3[模式 3: 无工具纯文本循环]
    Pattern1 --> Cause1{原因}
    Cause1 -->|工具返回无效信息| Fix1[修复工具返回]
    Cause1 -->|模型不理解工具| Fix2[优化 description]
    Pattern2 --> Cause2{原因}
    Cause2 -->|两个工具覆盖同一场景| Fix3[合并或拆分工具]
    Cause2 -->|边界条件未处理| Fix4[处理边界情况]
    Pattern3 --> Fix5[检查 System Prompt 是否缺少终止指令]

死循环排查从 Trace 开始，根据调用模式定位根因。

故障模式特征常见原因解决方案
相同工具重复query_order(T001) x 5工具返回结果太简略，模型无法推进充实工具返回信息
AB 工具切换query->refund->query->refund两个工具互相调用，模型陷入选择困难合并工具或检查权限
无工具循环LLM 自言自语 10 轮System Prompt 缺少终止指令增加"如果你有答案就直接回答"
参数反复修正模型反复修改参数重试参数校验太严格，模型不断微调放宽参数校验或提供更清晰提示

def diagnose_tool_failure(tool_name: str, args: dict):
    diagnosis = []
    if not args:
        diagnosis.append("参数为空：模型未传入参数")
    if "order_id" in args and not args["order_id"].startswith("T"):
        diagnosis.append(f"参数格式异常：{args['order_id']}")
    if circuit_breaker.is_open(tool_name):
        diagnosis.append(f"熔断器已打开：{tool_name} 暂时禁用")
    if not health_check("order-service"):
        diagnosis.append("下游服务异常：order-service 不可用")
    if rate_limiter.is_exceeded(tool_name):
        diagnosis.append(f"限流触发：{tool_name} 调用频率超限")
    return diagnosis if diagnosis else ["未检测到明显异常，需人工介入"]

生产排障案例

案例 1：客服 Agent 死循环导致 1000+ 次无效调用

现象：用户问「我的订单到哪了」，Agent 在 query_order 和 transfer_to_human 之间切换 15 次。

排查：Trace 显示每次 query_order 返回正常，但模型认为「用户语气不满需要退款」，调用 refund_order 失败又回退。

根因：System Prompt 缺少「先确认用户意图再操作」的约束。

修复：System Prompt 增加约束 + 工具 description 增加触发条件 + 连续失败后转人工。

案例 2：上下文溢出导致 Agent 失忆

现象：长对话中 Agent 突然不遵守 System Prompt，开始编造回答。

排查：Trace 发现上下文窗口中历史对话占了 90%，System Prompt 被挤到窗口后半部分。

根因：未设置上下文预算，历史对话无限累积。

修复：滑动窗口限制 10 轮 + 旧轮次生成摘要 + 每次 LLM 调用前重写 System Prompt。

常见问题（FAQ）

面试中被问到「你遇到过最复杂的 Agent 问题是什么」，应该怎么答？
推荐用 STAR 原则：Situation（背景）、Task（任务）、Action（行动）、Result（结果）。展示排查思路和技术深度。例如：客服 Agent 突然 50% 转人工->排查 Trace 发现模型升级后工具调用行为变化->回滚模型版本并添加回归测试->转人工率从 50% 降到 15%。
Agent 排障应该先看什么？
第一优先：LangSmith Trace。Trace 告诉你 Agent 每一步做了什么。第二优先：工具调用日志（入参和出参）。第三优先：LLM 调用日志（prompt 和 response）。比日志高效 10 倍。
模型升级后 Agent 表现突然变差，可能的原因是什么？
模型在 Function Calling 能力上发生变化；输出格式变化（如思考过程）；对 System Prompt 遵循度变化。解决方案：每次模型升级前运行回归测试，对比升级前后的关键指标。
