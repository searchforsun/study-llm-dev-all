为什么要学本章

面试官常问：「线上 Agent 转人工暴增你怎么查？」「tool 一直失败为什么模型还重试？」本章把 troubleshoot 变成可复述的 step debug 流程与 agent incident 案例库，覆盖 CorpAssist S2/S3 真实 production pitfall。

学完你能
  • 用「现象 → trace → 根因 → 修复 → 防再发」五步法答排障题
  • 诊断 debug loop 与 loop detection 失效
  • 分析 tool failure case 与 circuit break 缺失
  • 用 STAR 讲述 2 个 interview agent 案例
本章先记住 3 件事
  • 先 trace 后猜：LangSmith 定位 span，不在本地盲目改 prompt。
  • 区分症状与根因：转人工高可能是 loop、tool 超时或路由误判。
  • 修复要加门禁：坏例入库 dataset，eval gate 阻塞同类回归。

死循环

Case A · S2 改址死循环 用户要改地址，Agent 每轮 call query_order → 「请确认」→ 用户说「确认」→ again query_order，12 步后转人工，CSAT 1 分。

排障五步法

Agent incident 排障流程
flowchart TD
  P[现象/KPI 异常] --> T[LangSmith trace 抽样]
  T --> S[定位异常 span]
  S --> R[根因假设]
  R --> V[本地 trace replay]
  V --> F[修复 + 数据集回归]
trace 信号可能根因修复方向
同一 tool+args 重复 ≥3 loop detection 未启用或指纹过宽 收紧指纹；超限 degrade
LLM 不调用 write tool prompt 缺确认→行动映射 few-shot + 工具描述
steps 未递增 conditional edge 未接 increment 修图；见 步数控制
tool OK 但 LLM 重复问 observation 未写入 state 检查 message reducer

Case A 根因与修复（面试 STAR）

markdown 
Situation: S2 改址会话 avg steps 14, transfer +10%
Task: 24h 内止损并防复发
Action:
  - trace 发现 query_order 同 order_id 重复 5 次
  - 根因: 缺 change_address tool，模型用 query 代替
  - 加 tool + loop fingerprint (tool, order_id)
  - 坏例写入 dataset s2-loop-014
Result: avg steps 7.2, transfer 回基线; eval gate 新增序列断言
弱回答
  • 「加大 max_tokens」
  • 「换更强模型」
  • 未提 trace 与 dataset
强回答
  • 先 KPI 与时间线
  • trace 定位重复 span
  • 工具/图/fix + eval 防回归
面试快问：loop detection 用「连续相同 tool 名」够吗？

不够：须含关键 args 指纹（如 order_id）；否则合法重复查单误判。还要区分「用户改口」导致的新 query。

工具失败

tool failure 分超时、4xx/5xx、schema 校验失败、权限拒绝。模型若未收到结构化 error，会无限重试——须 retry policy + circuit break。

Case B · MCP 退款服务抖动

S2 促销午高峰 refund_propose 503 占 40%；Agent 仍每轮重试，p95 延迟 45s，用户放弃。
python 
@tool
def refund_propose(order_id: str) -> str:
    try:
        return mcp_client.call("refund.propose", order_id, timeout=3.0)
    except MCPTimeout:
        return "ERROR: refund service timeout; ask user to retry later or handoff"
    except MCPUnavailable:
        raise CircuitOpen("refund_mcp")  # 后续 call 直接降级

# conditional: circuit open -> handoff_human
失败类型给模型的 observation图行为
超时 ERROR + 可否重试 最多 1 次 retry
403 allowlist permission denied 停止 tool，转人工
503 持续 circuit open 跳过 tool，fallback 话术
schema 错 validation detail 让 LLM 改 args，≤2 次

面试强调：tool error 也是 observation，必须短、结构化、含下一步指引；不要把 Java stack trace 塞给模型。对照 失败与降级 与 权限拒绝 分支。

案例

整合 Case A/B 与 demos/advanced-06-interview-lab/incident-template.md，形成可背诵的 incident 报告。

incident 模板（填写示例）

markdown 
## 现象
S3 报销 Agent 48h 内 approval_submit 失败率 22% (基线 2%)

## trace
- span: approval_submit 401 missing scope
- 起始: role=end_user 误路由到 finance 图

## 根因
FastAPI 未校验 scene=s3 时 role 与 graph 绑定；Spring header 传错 X-Role

## 修复
- gateway 强制 role-scope 表
- eval 增 must_not_call approval_submit for end_user
- runbook 增 401 playbook

## 防再发
eval gate + 周度 trace 审计 sample 200

面试题速查

题目答题要点（30 秒）
Agent 和 ChatBot 排障有何不同 看 tool span 序列、步数、checkpoint；不只 final text
如何证明已修复 trace replay + dataset pass + KPI 24h
模型 vs 工程问题 同输入 replay：变=模型/prompt；不变=工具/权限/图
最高优先级 incident 越权 tool 成功 > loop 成本 > 文案差
生产陷阱 · 面试加分
  • 提到 root cause 层级：配置 < 代码 < 流程（缺 eval gate）
  • 提到跨团队：Spring 网关 vs Python Agent 各守一段
  • 提到 reasoning trace 隐私：排障脱敏后再给 LLM 分析

Case C · 综合排障题（自测）

「S2 CSAT 跌 0.3，transfer 不变，cost +50%」——你的第一步？（答案：LangSmith 按 scene=S2 过滤，对比 model_route 分布与 avg steps；若 turbo 占比降、plus 升，查路由 classifier 是否因促销话术 OOD。）

90 秒口述模板（可背诵）

markdown 
1. 现象+时间线+影响面（KPI）
2. 我先用 LangSmith 抽 20 条坏 trace，定位异常 span
3. 根因是…（tool/图/权限/路由，一点）
4. 修复：代码+配置+dataset 回归样例
5. 验证：replay pass + KPI 24h 恢复 + runbook 补充

无论 Case A/B/C，都按此五段说；面试官打断时优先保留第 2、3 步的证据与根因。

本章结论

  • 死循环：trace 看重复 span；tool+args 指纹；补 tool/修图；坏例入库。
  • 工具失败：结构化 error、有限 retry、circuit break、handoff。
  • 案例：incident 五段式 + STAR；安全 > 成本 > 文案。

动手练习

完成 2 份 incident 报告与 1 道模拟面试口述（约 40 分钟）。

操作步骤

  1. 用 demos/advanced-06-interview-lab/incident-template.md 完整填写 Case A（改址 loop）。
  2. 再填 Case B（MCP 503），在「修复」中写出 circuit break 阈值（如 5 分钟 50% 失败）。
  3. 从决策表选一题，录音 60 秒 STAR 回答（自检是否含 trace 与 eval）。
  4. 列出 Case C 的第二步、第三步排障动作（各一句）。
  5. 可选：在 LangSmith 找一条真实 trace 练 span 定位（脱敏）。

判断练习

  1. （面试）面试官：「用户说 Agent 变笨，你怎么查？」给出优先级排序：①看 prompt 版本 ②看 trace tool 序列 ③换 GPT-4 ④查路由 classifier 数据漂移。

    参考答案

    ② → ④ → ① → ③。先证据（trace）再配置/数据，换模型是最后手段且须 A/B。若涉及安全越权则跳过排序直接 P0。

  2. （场景）tool 返回 500 原文 50KB HTML，模型仍重试 10 次。缺哪两层防护？

    参考答案

    缺 tool 层 truncate/结构化 error 与 step/retry 上限；可能还缺 circuit break。见 tool return 规范。
