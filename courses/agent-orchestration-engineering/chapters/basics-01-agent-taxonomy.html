为什么要学本章

Agent 不是只有一种实现方式。ReAct、Plan-Execute、Reflexion 三种范式解决了不同的问题——ReAct 适合实时交互，Plan-Execute 适合复杂任务，Reflexion 负责自我纠错。选择错误的范式会导致 Agent 成本飙升或任务失败。本章建立 Agent 分类体系的全局视图。

学完你能

  • 为 CorpAssist S2 客服和 S3 办公自动化场景选择正确的范式
  • 用 Python LangGraph 和 Java Spring AI Alibaba 实现每种范式的骨架
  • 识别「范式滥用」的常见错误并给出修正方案
本章先记住 3 件事
  • ReAct = 边想边做：每一步推理后立即行动，适合客服、工具调用等实时场景。
  • Plan-Execute = 先想后做：先制定完整计划再逐步执行，适合多步骤复杂任务。
  • Reflexion = 做完检查：Agent 对自己的输出进行反思和修正，是质量兜底机制。

Agent 范式全景图

LLM 驱动的 Agent 遵循 感知 → 规划 → 行动 的通用框架，但不同范式在这三阶段的分配方式截然不同。agent taxonomy 帮助开发者根据任务特性选择最合适的架构。

三种 Agent 范式对比
flowchart TB
    subgraph ReAct[ReAct: 边想边做]
        direction LR
        R1[思考] --> R2[行动] --> R3[观察] --> R1
    end
    subgraph PlanExecute[Plan-Execute: 先想后做]
        direction LR

    end

        direction LR
        F1[执行] --> F2[反思] --> F3[修正] --> F1
    end

    ReAct -->|质量要求高| Reflexion

三种范式的适用边界：ReAct 最灵活，Plan-Execute 最结构化，Reflexion 是质量增强层。

范式核心特征适合场景成本延迟
ReAct推理与行动交替客服、工具调用、实时问答中低
Plan-Execute先规划再执行报告生成、数据流水线、审批流程高高
Reflexion自我反思纠错代码生成、关键决策、合规输出中+中
CorpAssist · S2 客服 vs S3 办公自动化 S2 客服场景：用户问「我的订单怎么还没到？」——Agent 需要查订单 API、查物流、回复用户。任务链短、实时性要求高，ReAct 最合适。
S3 办公自动化：用户要求「帮我写一份季度销售报告，包含各区域数据对比和趋势分析」——需要检索数据、分析、生成图表、排版输出。任务步骤多且依赖关系复杂，Plan-Execute 更适合。
先想 10 秒：如果让 Plan-Execute 处理客服问答，会出什么问题？

延迟过高。Plan-Execute 需要在第一步先制定完整计划（额外 LLM 调用），用户等不到实时响应。客服场景 2 秒内必须出结果，Plan-Execute 的首 token 延迟可能已超过 5 秒。

ReAct 范式详解

ReAct（Reasoning + Acting）由 Shinn 等人于 2023 年提出，核心思想是交替执行「思考 → 行动 → 观察」循环。模型每一步先推理当前状态，然后决定调用工具或输出回答，最后观察工具返回结果并继续推理。

ReAct 循环时序
sequenceDiagram
    participant LLM as LLM
    participant Agent as Agent Runtime
    participant Tool as 外部工具

    Agent->>LLM: System + 用户问题
    LLM-->>Agent: 思考：我需要查订单 API
    Agent->>Tool: 调用 query_order("T001")
    Tool-->>Agent: 返回：已发货，物流单号 SF123
    Agent->>LLM: 追加观察结果
    LLM-->>Agent: 思考：告知用户物流信息
    Agent-->>User: 最终回答

ReAct 中推理和工具调用交错进行，每次工具结果都作为下一轮推理的上下文。

Python：LangGraph ReAct 实现

python 
from langgraph.graph import StateGraph, END
from typing import TypedDict, Literal

class AgentState(TypedDict):
    messages: list
    next: Literal["tools", "end"]

def call_model(state: AgentState) -> AgentState:
    # LLM 推理：决定是调用工具还是直接回答
    response = llm.invoke(state["messages"])
    state["messages"].append(response)
    if response.tool_calls:
        state["next"] = "tools"
    else:
        state["next"] = "end"
    return state

def call_tool(state: AgentState) -> AgentState:
    tool_call = state["messages"][-1].tool_calls[0]
    result = tool_executor.invoke(tool_call)
    state["messages"].append({"role": "tool", "content": result})
    state["next"] = "continue"
    return state

# 构建状态图
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tool)
graph.add_conditional_edges(
    "agent",
    lambda s: s["next"],
    {"tools": "tools", "end": END}
)
graph.add_edge("tools", "agent")
graph.set_entry_point("agent")
react_agent = graph.compile()

Java：Spring AI Alibaba 实现

java 
@Configuration
public class ReactAgentConfig {

    @Bean
    public ReactAgent corpassistAgent(
            ChatModel chatModel,
            List<ToolCallback> toolCallbacks) {

        return ReactAgent.builder()
            .chatModel(chatModel)
            .tools(toolCallbacks)
            .systemPrompt("你是 CorpAssist S2 客服助手。" +
                "用 ReAct 模式：思考用户需求→调用工具→分析结果→回复。")
            .maxSteps(10)
            .build();
    }
}

// 工具定义
@Component
public class OrderTools {
    @Tool(name = "query_order",
          description = "查询订单状态，输入订单号返回物流信息")
    public String queryOrder(String orderId) {
        Order order = orderService.findById(orderId);
        return "订单 %s 状态：%s，物流：%s"
            .formatted(order.id(), order.status(), order.logisticsNo());
    }
}
不推荐
  • 不设 maxSteps，Agent 陷入死循环
  • 工具返回原始 JSON 不加工，模型解析困难
  • 无观察结果截断，上下文快速膨胀
推荐
  • maxSteps=10~15 + 超时兜底
  • 工具返回格式化文本，降低推理负担
  • 观察结果截断至 1000 字符内

Plan-Execute 范式详解

Plan-Execute 将 Agent 的工作分为两个阶段：规划阶段制定完整的步骤列表，执行阶段依次完成任务。这种分离让复杂任务的执行路径可预测、可审核。

Plan-Execute 工作流
flowchart LR
    Start[用户请求] --> Planner[规划器]
    Planner -->|生成步骤列表| Executor[执行引擎]
    Executor --> Step1[步骤 1]
    Step1 --> Step2[步骤 2]
    Step2 --> Step3[步骤 3]
    Step3 -->|全部完成| Verify[验证]
    Verify -->|通过| Done[最终输出]
    Verify -->|失败| Replan[重规划]
    Replan --> Planner

Plan-Execute 通过规划与执行分离，让 Agent 对复杂任务有全局视角。

Python：Plan-Execute 骨架

python 
class PlanExecutor:
    """Plan-Execute 模式实现"""
    def __init__(self, planner_llm, executor_llm, tools: dict):
        self.planner = planner_llm
        self.executor = executor_llm
        self.tools = tools

    def plan(self, task: str) -> list[str]:
        prompt = f"为以下任务制定执行步骤：\n{task}\n步骤列表："
        response = self.planner.invoke(prompt)
        return [s.strip() for s in response.content.split("\n") if s.strip()]

    def execute_step(self, step: str, context: dict) -> dict:
        prompt = f"执行步骤：{step}\n当前上下文：{context}"
        response = self.executor.invoke(prompt)
        # 解析是否需要调用工具
        if "TOOL_CALL:" in response.content:
            tool_name, args = parse_tool_call(response.content)
            result = self.tools[tool_name](**args)
            context[tool_name] = result
        context["last_output"] = response.content
        return context

    def run(self, task: str) -> str:
        steps = self.plan(task)
        context = {"task": task, "steps": steps}
        for step in steps:
            context = self.execute_step(step, context)
        return context.get("last_output", "")

Java：Spring AI Alibaba Plan-Execute 实现

java 
@Service
public class PlanExecuteService {
    private final ChatModel plannerModel;
    private final ChatModel executorModel;
    private final List<ToolCallback> tools;

    public PlanExecuteService(
            @Qualifier("plannerChatModel") ChatModel planner,
            @Qualifier("executorChatModel") ChatModel executor,
            List<ToolCallback> tools) {
        this.plannerModel = planner;
        this.executorModel = executor;
        this.tools = tools;
    }

    public String execute(String task) {
        // 阶段 1：规划
        List<String> steps = plan(task);
        // 阶段 2：依次执行
        Map<String, Object> context = new HashMap<>();
        context.put("task", task);
        for (String step : steps) {
            String result = executeStep(step, context);
            context.put("step_" + steps.indexOf(step), result);
        }
        return (String) context.get("final_result");
    }

    private List<String> plan(String task) { /* 调用 planner */ }
    private String executeStep(String step, Map<String, Object> ctx) { /* 调用 executor */ }
}

Reflexion 范式详解

Reflexion 让 Agent 对自己的输出进行评价和修正。它不替代其他范式，而是作为质量增强层叠加在 ReAct 或 Plan-Execute 之上。Reflexion 特别适合代码生成、金融合规、医疗建议等对准确性要求极高的场景。

Reflexion 自反思循环
sequenceDiagram
    participant Actor as Actor Agent
    participant Critic as Critic Agent
    participant Output as 输出

    Actor->>Output: 生成初始回答
    Critic->>Output: 检查：是否符合规范？
    alt 发现问题
        Critic->>Actor: 反馈：需要修正{原因}
        Actor->>Output: 生成修正版本
        Critic->>Output: 再次检查
    else 通过检查
        Critic->>Output: 确认通过
    end

Reflexion 通过 Critic Agent 对 Actor Agent 的输出进行评价和迭代修正。

不推荐
  • 每次输出都 Reflexion，成本翻 3 倍
  • 没有最大迭代次数，可能无限修正
  • Critic 使用和 Actor 相同的模型，缺乏多样性
推荐
  • 仅在高风险操作时启用 Reflexion
  • 设置 max_reflection=2，防止无限循环
  • Critic 使用不同模型或更严格的 System Prompt

范式选择决策树

如何为实际项目选择范式？以下决策树可以帮助快速判断：

Agent 范式选择流程
flowchart TD
    Q1[任务需要多步工具调用？] -->|否| Simple[单轮 LLM 即可]
    Q1 -->|是| Q2[步骤间有复杂依赖？]
    Q2 -->|否| React[ReAct]
    Q2 -->|是| Q3[结果需要质量审核？]
    Q3 -->|否| PlanExec[Plan-Execute]
    Q3 -->|是| PlanRef[Plan-Execute + Reflexion]
    React --> Q4[高准确性要求？]
    Q4 -->|是| ReactRef[ReAct + Reflexion]
    Q4 -->|否| ReactOnly[纯 ReAct]

范式选择取决于任务复杂度、步骤依赖关系和质量要求三个维度。

常见问题（FAQ）

ReAct 和 Plan-Execute 可以混合使用吗？
可以。这是常见的工程实践——外层用 Plan-Execute 做任务分解，内层每个子任务用 ReAct 执行。例如 CorpAssist S3 的「生成销售报告」任务：Plan-Execute 先规划步骤（查数据→分析→生成图表），每步用 ReAct 子 Agent 调用具体工具。
Reflexion 需要额外调用一次 LLM，成本翻倍值得吗？
取决于场景。对面向客户的回答，Reflexion 可以把幻觉率从 15% 降到 3% 以下。建议只对「高风险」输出启用——例如涉及金额、法律条款、医疗建议的回复。低风险场景（如天气查询）不需要 Reflexion。
LangGraph 支持 Plan-Execute 吗？
LangGraph 的 StateGraph 天然支持 Plan-Execute。你可以定义 planner_node 和 executor_node，通过条件边实现规划→执行→重规划的循环。Spring AI Alibaba 的 Graph 也支持类似模式。具体实现见 ReAct 循环实现 和 多 Agent 协作模式。

本章结论

  • 三种范式：ReAct（边想边做）、Plan-Execute（先想后做）、Reflexion（做完检查），各有适用场景。
  • 选择依据：任务复杂度决定是否需 Plan-Execute，质量要求决定是否需 Reflexion。
  • 双栈实现：Python LangGraph 和 Java Spring AI Alibaba 都支持三种范式，但 Java 更侧重开箱即用的封装。
  • 成本权衡：范式越复杂成本越高，不要为简单任务选择 heavyweight 范式。
  • 混合使用：三种范式不是互斥的，可以叠加组合实现最佳效果。
