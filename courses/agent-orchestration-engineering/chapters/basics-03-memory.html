为什么要学本章

S2 多轮客服与 S3 长流程 Agent 都需要「接着上次聊」。LangGraph 用 checkpoint 持久化图状态；无限堆消息会导致检索干扰与成本爆炸（S3 生产坑）。

学完你能
  • 解释 thread_id 与 state persist 在 FastAPI 请求间的传递
  • 选型 MemorySaver vs Redis checkpoint
  • 区分 short memory 与 long memory（store）的使用场景
本章先记住 3 件事
  • Checkpoint = 图快照：含 messages、tool 结果、步数；可 resume/interrupt。
  • Store = 跨会话知识：用户偏好、工单摘要；需 TTL 与 ACL。
  • 别无限 append：压缩/摘要旧轮次，见《上下文与混合记忆工程》。

checkpoint

每次 super-step 后 LangGraph 将 agent state 写入 checkpoint。同一 thread_id 的请求共享一条时间线。

编译时挂载 checkpointer

python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.redis import RedisSaver

# 开发
memory = MemorySaver()
app = graph.compile(checkpointer=memory)

# 生产
redis = RedisSaver.from_conn_string("redis://localhost:6379/0")
app = graph.compile(checkpointer=redis)

config = {"configurable": {"thread_id": "cs-session-9f3a"}}
app.invoke({"messages": [("user", "查单 8821")]}, config)

store

LangGraph store 存跨 thread 或长期键值：用户 VIP 等级、历史工单摘要。与 checkpoint 分工：checkpoint 管「这次对话跑到哪」，store 管「这个用户是谁」。

存储键典型内容TTL
Checkpointthread_idmessages, tool traces会话结束 + 保留期
Storeuser_id + namespace偏好、摘要向量按合规策略
先想 10 秒：能否把全部历史消息只放 checkpoint 不摘要？

短会话可以；S3 长流程会 token 爆炸并引入旧 tool 噪声。应对旧轮做摘要或检索式记忆。

Redis

生产多实例 FastAPI 必须用共享 checkpointer（Redis/Postgres），否则 sticky session 失效时状态丢失。

配置要点（示意）

yaml
agent:
  checkpoint:
    backend: redis
    url: ${REDIS_URL}
    ttl_hours: 72
  thread_id_header: X-CorpAssist-Thread-Id
不推荐
  • 进程内 MemorySaver 上生产多副本
  • thread_id 可猜测、无鉴权绑定 user
推荐
  • Redis + TTL + 加密敏感字段
  • thread 与 JWT user_id 绑定校验

本章结论

  • checkpoint：图状态快照，thread_id 串联多轮；支持 interrupt/resume。
  • store：跨会话长期记忆，需 ACL 与 TTL，勿与 checkpoint 混为一谈。
  • Redis：生产共享 checkpointer，配合网关 session header。
