为什么要学本章

Agent 不是只有一种实现方式。ReAct、Plan-Execute、Reflexion 三种范式解决了不同的问题——ReAct 适合实时交互，Plan-Execute 适合复杂任务，Reflexion 负责自我纠错。选择错误的范式会导致 Agent 成本飙升或任务失败。本章建立 Agent 分类体系的全局视图。

学完你能

  • 为 CorpAssist S2 客服和 S3 办公自动化场景选择正确的范式
  • 用 Python LangGraph 和 Java Spring AI Alibaba 实现每种范式的骨架
  • 识别「范式滥用」的常见错误并给出修正方案
本章先记住 3 件事
  • ReAct = 边想边做：每一步推理后立即行动，适合客服、工具调用等实时场景。
  • Plan-Execute = 先想后做：先制定完整计划再逐步执行，适合多步骤复杂任务。
  • Reflexion = 做完检查：Agent 对自己的输出进行反思和修正，是质量兜底机制。

范式核心特征适合场景成本延迟

ReAct 范式详解

ReAct（Reasoning + Acting）由 Shinn 等人于 2023 年提出，核心思想是交替执行「思考 → 行动 → 观察」循环。模型每一步先推理当前状态，然后决定调用工具或输出回答，最后观察工具返回结果并继续推理。

ReAct 循环时序
sequenceDiagram
    participant LLM as LLM
    participant Agent as Agent Runtime
    participant Tool as 外部工具

    Agent->>LLM: System + 用户问题
    LLM-->>Agent: 思考：我需要查订单 API
    Agent->>Tool: 调用 query_order("T001")
    Tool-->>Agent: 返回：已发货，物流单号 SF123
    Agent->>LLM: 追加观察结果
    LLM-->>Agent: 思考：告知用户物流信息
    Agent-->>User: 最终回答

ReAct 中推理和工具调用交错进行，每次工具结果都作为下一轮推理的上下文。

Python：LangGraph ReAct 实现

python 
from langgraph.graph import StateGraph, END
from typing import TypedDict, Literal

class AgentState(TypedDict):
    messages: list
    next: Literal["tools", "end"]

def call_model(state: AgentState) -> AgentState:
    # LLM 推理：决定是调用工具还是直接回答
    response = llm.invoke(state["messages"])
    state["messages"].append(response)
    if response.tool_calls:
        state["next"] = "tools"
    else:
        state["next"] = "end"
    return state

def call_tool(state: AgentState) -> AgentState:
    tool_call = state["messages"][-1].tool_calls[0]
    result = tool_executor.invoke(tool_call)
    state["messages"].append({"role": "tool", "content": result})
    state["next"] = "continue"
    return state

# 构建状态图
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tool)
graph.add_conditional_edges(
    "agent",
    lambda s: s["next"],
    {"tools": "tools", "end": END}
)
graph.add_edge("tools", "agent")
graph.set_entry_point("agent")
react_agent = graph.compile()

Plan-Execute 范式详解

Plan-Execute 将 Agent 的工作分为两个阶段：规划阶段制定完整的步骤列表，执行阶段依次完成任务。这种分离让复杂任务的执行路径可预测、可审核。

Plan-Execute 工作流
flowchart LR
    Start[用户请求] --> Planner[规划器]
    Planner -->|生成步骤列表| Executor[执行引擎]
    Executor --> Step1[步骤 1]
    Step1 --> Step2[步骤 2]
    Step2 --> Step3[步骤 3]
    Step3 -->|全部完成| Verify[验证]
    Verify -->|通过| Done[最终输出]
    Verify -->|失败| Replan[重规划]
    Replan --> Planner

Plan-Execute 通过规划与执行分离，让 Agent 对复杂任务有全局视角。

Python：Plan-Execute 骨架

python 
class PlanExecutor:
    """Plan-Execute 模式实现"""
    def __init__(self, planner_llm, executor_llm, tools: dict):
        self.planner = planner_llm
        self.executor = executor_llm
        self.tools = tools

    def plan(self, task: str) -> list[str]:
        prompt = f"为以下任务制定执行步骤：\n{task}\n步骤列表："
        response = self.planner.invoke(prompt)
        return [s.strip() for s in response.content.split("\n") if s.strip()]

    def execute_step(self, step: str, context: dict) -> dict:
        prompt = f"执行步骤：{step}\n当前上下文：{context}"
        response = self.executor.invoke(prompt)
        # 解析是否需要调用工具
        if "TOOL_CALL:" in response.content:
            tool_name, args = parse_tool_call(response.content)
            result = self.tools[tool_name](**args)
            context[tool_name] = result
        context["last_output"] = response.content
        return context

    def run(self, task: str) -> str:
        steps = self.plan(task)
        context = {"task": task, "steps": steps}
        for step in steps:
            context = self.execute_step(step, context)
        return context.get("last_output", "")

public class PlanExecuteService {
    private final ChatModel plannerModel;
    private final ChatModel executorModel;
    private final List<ToolCallback> tools;

    public PlanExecuteService(
            @Qualifier("plannerChatModel") ChatModel planner,
            @Qualifier("executorChatModel") ChatModel executor,
            List<ToolCallback> tools) {
        this.plannerModel = planner;
        this.executorModel = executor;
        this.tools = tools;
    }

    public String execute(String task) {
        // 阶段 1：规划
        List<String> steps = plan(task);
        // 阶段 2：依次执行
        Map<String, Object> context = new HashMap<>();
        context.put("task", task);
        for (String step : steps) {
            String result = executeStep(step, context);
            context.put("step_" + steps.indexOf(step), result);
        }
        return (String) context.get("final_result");
    }

Reflexion 范式详解

Reflexion 让 Agent 对自己的输出进行评价和修正。它不替代其他范式，而是作为质量增强层叠加在 ReAct 或 Plan-Execute 之上。Reflexion 特别适合代码生成、金融合规、医疗建议等对准确性要求极高的场景。

Reflexion 自反思循环
sequenceDiagram
    participant Actor as Actor Agent
    participant Critic as Critic Agent
    participant Output as 输出

    Actor->>Output: 生成初始回答
    Critic->>Output: 检查：是否符合规范？
    alt 发现问题
        Critic->>Actor: 反馈：需要修正{原因}
        Actor->>Output: 生成修正版本
        Critic->>Output: 再次检查
    else 通过检查
        Critic->>Output: 确认通过
    end

Reflexion 通过 Critic Agent 对 Actor Agent 的输出进行评价和迭代修正。

不推荐
  • 每次输出都 Reflexion，成本翻 3 倍
  • 没有最大迭代次数，可能无限修正
  • Critic 使用和 Actor 相同的模型，缺乏多样性
推荐
  • 仅在高风险操作时启用 Reflexion
  • 设置 max_reflection=2，防止无限循环
  • Critic 使用不同模型或更严格的 System Prompt

如何为实际项目选择范式？以下决策树可以帮助快速判断：

常见问题（FAQ）

ReAct 和 Plan-Execute 可以混合使用吗？
可以。这是常见的工程实践——外层用 Plan-Execute 做任务分解，内层每个子任务用 ReAct 执行。例如 CorpAssist S3 的「生成销售报告」任务：Plan-Execute 先规划步骤（查数据→分析→生成图表），每步用 ReAct 子 Agent 调用具体工具。
Reflexion 需要额外调用一次 LLM，成本翻倍值得吗？
取决于场景。对面向客户的回答，Reflexion 可以把幻觉率从 15% 降到 3% 以下。建议只对「高风险」输出启用——例如涉及金额、法律条款、医疗建议的回复。低风险场景（如天气查询）不需要 Reflexion。
LangGraph 支持 Plan-Execute 吗？
LangGraph 的 StateGraph 天然支持 Plan-Execute。你可以定义 planner_node 和 executor_node，通过条件边实现规划→执行→重规划的循环。Spring AI Alibaba 的 Graph 也支持类似模式。具体实现见 ReAct 循环实现 和 多 Agent 协作模式。

本章结论

  • 三种范式：ReAct（边想边做）、Plan-Execute（先想后做）、Reflexion（做完检查），各有适用场景。
  • 选择依据：任务复杂度决定是否需 Plan-Execute，质量要求决定是否需 Reflexion。
  • 双栈实现：Python LangGraph 和 Java Spring AI Alibaba 都支持三种范式，但 Java 更侧重开箱即用的封装。
  • 成本权衡：范式越复杂成本越高，不要为简单任务选择 heavyweight 范式。
  • 混合使用：三种范式不是互斥的，可以叠加组合实现最佳效果。

为什么要学本章

工具是 Agent 与外部世界交互的唯一通道。Function Calling 的可靠性直接决定 Agent 的成败——工具定义不清、参数错误、结果不可解析是 Agent 项目最常见的三大故障源。本章从原理到实践，覆盖工具定义、Schema 设计、参数校验、错误处理和安全的完整链路。

本章先记住 3 件事
  • 模型声明工具：LLM 在响应中返回 tool_calls 结构，不是 LLM 直接执行函数。
  • Schema 即契约：工具定义（参数名、类型、描述）是模型理解工具的接口说明书。
  • 描述决定调用率：工具描述越清晰，模型正确调用的概率越高。

Function Calling 底层原理

Function Calling 并非模型「调用函数」，而是模型在生成文本时声明「我需要调用某个函数」。Agent 框架拦截这个声明，执行对应函数，将结果以消息形式返回给模型继续推理。

Function Calling 完整流程
sequenceDiagram
    participant Dev as 开发者
    participant Model as LLM
    participant Agent as Agent Runtime
    participant API as 企业 API

    Dev->>Model: 注册工具定义（OpenAPI Schema）
    Note over Model: 模型内化工具描述
    Agent->>Model: 用户问题 + 可用工具列表
    Model-->>Agent: 响应 + tool_calls[query_order("T001")]
    Agent->>Agent: 解析 tool_calls
    Agent->>API: 执行 query_order("T001")
    API-->>Agent: 返回订单数据
    Agent->>Model: 追加 tool_result 消息
    Model-->>Agent: 生成最终回复

Function Calling 的完整数据流：工具定义、声明、执行、结果回传。

模型在训练阶段学习了大量 API 调用示例，因此能够理解工具的 JSON Schema 定义。当 tool schema 足够清晰时，模型可以在推理时自动判断是否需要调用工具、传入什么参数。

先想 10 秒：如果没有 Function Calling 能力，Agent 如何实现工具调用？

退化为 prompt 解析模式——让 LLM 输出固定格式字符串（如 TOOL_CALL: query_order|T001），Agent 再用正则或字符串解析提取。这种方式可靠性差，模型容易输出格式错乱。Function Calling 是原生支持的结构化输出，可靠性远超文本解析。

工具定义与 Schema 设计

工具 Schema 是模型理解工具的接口契约。设计良好的 Schema 需要包含：清晰的名称、准确的参数描述、合理的参数类型和必要的校验规则。

@Component
public class ResilientTools {
    private static final Logger log = LoggerFactory.getLogger(ResilientTools.class);

    private final OrderService orderService;
    private final Retry retryPolicy;

    public ResilientTools(OrderService orderService) {
        this.orderService = orderService;
        this.retryPolicy = Retry.of("tool-retry",
            RetryConfig.custom()
                .maxAttempts(3)
                .waitDuration(Duration.ofSeconds(1))
                .retryExceptions(TimeoutException.class, IOException.class)
                .build());
    }

    @Tool(name = "query_order_resilient",
          description = "带重试的订单查询")
    public String queryOrder(String orderId) {
        return Try.ofSupplier(() ->
                Retry.decorateSupplier(retryPolicy, () -> {
                    Order o = orderService.findById(orderId);
                    if (o == null) return "未找到订单: " + orderId;
                    return formatOrder(o);
                })
            ).recover(e -> "查询异常：" + e.getMessage())
            .get();
    }
}
常见误区
  • 工具抛出异常 = Agent 崩溃：Agent 框架应捕获异常，以消息形式返回给模型，让模型决定是否重试或降级。
  • 返回过长不截断：工具返回 1 万行日志会撑爆上下文。建议截断至 2000 字符内。
  • 错误信息含敏感数据：异常信息可能包含堆栈和数据库细节，返回给模型前需脱敏。

安全调用策略

工具是 Agent 的风险入口。不良设计的工具可能被模型（或被用户 prompt injection 诱导的模型）滥用，造成数据泄露或业务损失。

工具安全调用分层
flowchart TB
    Input[用户输入] -->|1. 输入过滤| Sanitize[XSS/SQL 注入过滤]
    Sanitize -->|2. 权限检查| Auth[工具调用权限校验]
    Auth -->|3. 限流| RateLimit[QPS/并发限制]
    RateLimit -->|4. 执行| Execute[工具执行]
    Execute -->|5. 输出过滤| Output[脱敏/合规审查]
    Output -->|6. 审计| Audit[全链路日志]

六层安全防护：从输入到输出覆盖全链路。

常见问题（FAQ）

模型不调用工具怎么办？
首先检查工具 description 是否清晰。常见原因：工具描述模糊、参数描述不全、模型不了解何时该用此工具。解决方案：① 在 System Prompt 中举例说明何时使用该工具；② 增加 strict=True 强制模型必须使用工具；③ 切换到 Function Calling 能力更强的模型（如 Qwen、GPT-4、Claude）。
模型传错了参数怎么办？
模型不会「故意」传错参数，通常是因为：① 参数描述不够清晰；② 参数名含义模糊；③ 用户提供的信息不全。建议在工具开头做参数校验，返回友好错误让模型重新尝试。也可以让工具参数设计更宽松——例如支持模糊匹配的订单号。
工具可以调用另一个工具吗？
工具本身不应直接调用其他工具——这是编排层的职责。如果你发现一个工具需要调用另一个工具，说明这两个功能应该由 Agent 的 ReAct 循环串联，而不是在工具内部耦合。见 工具设计最佳实践 中的粒度讨论。

本章结论

  • Function Calling 是声明式：模型声明需要调用工具，Agent 运行时负责执行和结果回传。
  • Schema 即接口契约：参数的名称、类型和描述决定了模型能否正确调用工具。
  • 错误处理兜底：工具必须处理参数校验异常、业务异常和系统异常，返回友好错误让模型继续。
  • 安全分六层：从输入过滤到审计日志，每一层都不可或缺。
  • 双栈一致：Python Pydantic 和 Java @ToolParam 在 Schema 设计理念上完全一致。

Chain-of-Thought 推理

Chain-of-Thought（CoT）通过在 prompt 中注入「一步一步思考」的指令，引导模型在给出最终答案前展示推理过程。在 Agent 中，CoT 不仅是推理策略，更是一种内置的 System Prompt 设计模式。

CoT 推理与直接回答对比
sequenceDiagram
    participant U as 用户
    participant LLM as LLM

    Note over U,LLM: 直接回答模式
    U->>LLM: 我的订单 T001 现在到哪了？
    LLM-->>U: 已发货，预计明天到达。
    Note over U,LLM: 缺少推理过程，可能出错

    Note over U,LLM: CoT 推理模式
    U->>LLM: 我的订单 T001 现在到哪了？
    Note over LLM: 推理：用户想知道物流状态\n我需要查订单 API\n传入参数 order_id=T001
    LLM-->>U: 我已查询订单 T001，\n当前状态：运输中，\n物流单号：SF123456，\n预计明天 18:00 前送达。

CoT 让 Agent 显式展示推理链，既提高准确率又便于调试。

CoT 在工具调用中的融合

在 Agent 设计中，CoT 不是独立步骤——它融入 ReAct 循环的"思考"阶段。System Prompt 中应明确指示模型先推理再行动：

text 
你是 CorpAssist S2 客服 Agent。
调用工具前，请先思考：
1. 用户的核心需求是什么？
2. 需要调用哪些工具？参数是什么？
3. 工具返回后如何解读结果？
4. 最终回答应该包含哪些信息？

请用"思考："开头展示你的推理过程，
然后用"行动："指示要调用的工具。
不推荐
  • System Prompt 无 CoT 引导
  • 要求模型直接输出答案，跳过推理
  • 不区分思考和行动阶段
推荐
  • 显式要求「请先思考后回答」
  • 提供思考框架模板
  • 工具调用前后都有推理步骤
先想 10 秒：如果用户说「直接给我答案，别废话推理过程」，还能用 CoT 吗？

可以。推理过程对模型是必要的内部步骤，但最终回答可以只展示结论。在 System Prompt 中说明「推理过程仅供内部使用，最终回答简洁直接」。Agent 框架可以在最终输出前过滤掉推理轨迹。

        self.llm = llm

    def decompose(self, task: str, context: dict = None) -> List[SubTask]:
        prompt = f"""将以下复杂任务拆解为可执行的子任务列表。

            if len(parts) >= 1:
                subtasks.append(SubTask(
                    id=f"step_{i+1}",
                    description=parts[0],
                    depends_on=parts[1].split(",") if len(parts) > 1 and parts[1] != "无" else [],
                    tool=parts[2] if len(parts) > 2 else ""
                ))
        return subtasks

动态重规划

Agent 执行过程中，外部环境可能变化——API 返回空数据、某个步骤失败、用户中途改变需求。静态的初始计划无法应对这些变化，Agent 需要具备 动态重规划 能力。

动态重规划流程
flowchart LR
    Start[初始计划] --> Execute[执行步骤]
    Execute --> Check[检查结果]
    Check -->|正常| Next[下一步骤]
    Check -->|异常| Replan[重规划]
    Next -->|全部完成| Done[完成]
    Next --> Execute
    Replan -->|调整计划| Execute
    Replan -->|无法完成| Escalate[升级到人工]

动态重规划在每一步执行后检查结果，异常时触发计划调整或人工升级。

步骤失败标记失败、跳过或重试数据源 A 超时→切到数据源 B
用户插入新需求暂停当前计划，合并新需求用户说「顺便查一下退款进度」

Plan-Execute 引擎设计

将规划与执行分离的 Plan-Execute 引擎是 Agent 架构中最常见的实现。它包含三个核心组件：Planner（规划器）、Executor（执行器）、Monitor（监控器）。

python 
class PlanExecuteEngine:
    """规划-执行引擎"""
    def __init__(self, planner_llm, executor_llm, tools: dict):
        self.planner = planner_llm
        self.executor = executor_llm
        self.tools = tools
        self.max_retries = 2
        self.max_steps = 20

    def run(self, task: str) -> dict:
        context = {"task": task, "results": {}, "step": 0}
        # 阶段 1：规划
        plan = self.planner.plan(task)
        context["plan"] = plan

        # 阶段 2：循环执行
        while context["step"] < self.max_steps:
            current = self._get_next_step(plan, context)
            if current is None:
                break

            success, result = self._execute_step(current, context)
            if not success:
                plan = self.planner.replan(task, context, current)
                continue

            context["results"][current.id] = result
            context["step"] += 1

        return context

    def _execute_step(self, step, context):
        for attempt in range(self.max_retries):
            try:
                if step.tool:
                    result = self.tools[step.tool](**context.get("params", {}))
                else:
                    result = self.executor.invoke(step.description)
                return True, result
            except Exception as e:
                if attempt == self.max_retries - 1:
                    return False, str(e)
        return False, "max retries exceeded"

java 
@Service
public class PlanExecuteEngine {
    private final Planner planner;
    private final ToolExecutor toolExecutor;
    private final ChatModel executorModel;
    private static final int MAX_STEPS = 20;
    private static final int MAX_RETRIES = 2;

    public ExecutionContext execute(String task) {
        ExecutionContext ctx = new ExecutionContext(task);

        // Phase 1: Plan
        List<Step> plan = planner.createPlan(task);
        ctx.setPlan(plan);

        // Phase 2: Execute with dynamic replanning
        while (ctx.getCurrentStepIndex() < plan.size()
               && ctx.getTotalSteps() < MAX_STEPS) {

            Step step = plan.get(ctx.getCurrentStepIndex());
            ExecutionResult result = executeWithRetry(step, ctx);

            if (!result.isSuccess()) {
                // Trigger replanning
                List<Step> newPlan = planner.replan(ctx, step);
                ctx.setPlan(newPlan);
                continue;
            }

            ctx.addResult(step.id(), result.getOutput());
            ctx.incrementStep();
        }
        return ctx;
    }

    private ExecutionResult executeWithRetry(Step step, ExecutionContext ctx) {
        for (int i = 0; i < MAX_RETRIES; i++) {
            try {
                String output = step.tool() != null
                    ? toolExecutor.execute(step.tool(), step.params())
                    : executorModel.call(step.description());
                return new ExecutionResult(true, output);
            } catch (Exception e) {
                log.warn("Step {} failed, attempt {}/{}",
                    step.id(), i + 1, MAX_RETRIES);
            }
        }
        return new ExecutionResult(false, "Max retries exceeded");
    }
}

常见问题（FAQ）

CoT 一定会提高准确率吗？
研究表明 CoT 对需要多步推理的任务（数学、逻辑、规划）提升显著，但对常识性问题反而可能增加 token 消耗无实质改善。建议根据任务类型决定是否启用 CoT——简单问答用直接模式，复杂推理用 CoT。
任务分解到什么粒度合适？
每个子任务应该在 1-2 次工具调用或 LLM 调用内完成。如果一个子任务本身需要再分解 3 层，说明粒度太粗。通常一个复杂任务的叶子节点数控制在 5-15 个之间。太多则管理成本高，太少则子任务本身过于复杂。
动态重规划会不会无限循环？
会。必须设置最大重规划次数（通常 3-5 次）。超过上限后触发人工升级或返回当前已收集的所有信息。在 Plan-Execute 引擎中，重规划计数器和步骤计数器应该独立追踪，任何一个达到上限都终止执行。

本章结论

  • CoT 推理：显式推理链提高准确率，融入 System Prompt 作为 Agent 的思考模板。
  • 任务分解：将复杂目标拆为独立子任务是规划的核心，粒度控制在 5-15 个叶子节点。
  • 动态重规划：执行结果异常时触发计划调整，设置最大重规划次数防止无限循环。
  • Plan-Execute：Planner + Executor + Monitor 三层架构，Python 和 Java 实现模式一致。
  • 实际应用：CorpAssist S3 办公自动化场景中，Plan-Execute 引擎处理报告生成、数据分析等复杂任务。

from collections import deque
from typing import List

class SlidingWindowMemory:
    """滑动窗口短期记忆"""

    def __init__(self, max_turns: int = 10, max_tokens: int = 6000):
        self.history = deque(maxlen=max_turns)
        self.max_tokens = max_tokens

    def add_turn(self, user_msg: str, assistant_msg: str):
        self.history.append({
            "user": user_msg,
            "assistant": assistant_msg,
            "timestamp": time.time()
        })

    def get_context(self) -> List[dict]:
        """获取当前可用的对话历史"""
        # 当历史超出最大轮次时，从最早开始丢弃
        context = []
        token_count = 0
        for turn in reversed(self.history):
            turn_tokens = count_tokens(turn["user"]) + count_tokens(turn["assistant"])
            if token_count + turn_tokens > self.max_tokens:
                break
            context.insert(0, turn)
            token_count += turn_tokens
        return context

    def summarize(self, llm) -> str:
        """对旧的历史生成摘要"""
        if len(self.history) < 3:
            return ""
        text = "\n".join(
            f"用户：{t['user']}\n助手：{t['assistant']}"
            for t in list(self.history)[:-2]
        )
        summary = llm.invoke(f"请总结以下对话的关键信息：\n{text}")
        return summary.content

import redis
import json
from langchain.embeddings import OpenAIEmbeddings

class LongTermMemory:
    """长期记忆：Vector DB + Redis 双层存储"""

    def __init__(self, vector_store, redis_client):
        self.vector_store = vector_store  # Chroma/Pinecone/Milvus
        self.redis = redis_client or redis.Redis(host="localhost", port=6379)

    def remember(self, user_id: str, key: str, value: str, metadata: dict = None):
        """存储一条记忆"""
        # Redis 快速存储
        memory_key = f"memory:{user_id}:{key}"
        self.redis.set(memory_key, json.dumps({
            "value": value,
            "metadata": metadata or {},
            "timestamp": time.time()
        }))
        self.redis.expire(memory_key, 86400 * 30)  # 30 天 TTL

        # Vector DB 语义存储
        doc_text = f"{key}: {value}"
        self.vector_store.add_texts(
            texts=[doc_text],
            metadatas=[{"user_id": user_id, "key": key, **(metadata or {})}]
        )

    def recall(self, user_id: str, query: str, top_k: int = 3) -> list:
        """检索相关记忆"""
        # 语义检索
        docs = self.vector_store.similarity_search(
            query, k=top_k,
            filter={"user_id": user_id}
        )
        return [doc.page_content for doc in docs]

    def forget(self, user_id: str, key: str):
        """删除记忆"""
        self.redis.delete(f"memory:{user_id}:{key}")

    public void saveCheckpoint(String threadId, int step, String stateJson) {
        jdbcTemplate.update(
            "INSERT INTO agent_checkpoints (thread_id, step, state_json, created_at) "
            + "VALUES (?, ?, ?, NOW()) "
            + "ON DUPLICATE KEY UPDATE state_json = ?, step = ?",
            threadId, step, stateJson, stateJson, step);
    }

    public String getCheckpoint(String threadId) {
        return jdbcTemplate.queryForObject(
            "SELECT state_json FROM agent_checkpoints "
            + "WHERE thread_id = ? ORDER BY step DESC LIMIT 1",
            String.class, threadId);
    }
}

上下文预算分配

即使有滑动窗口和摘要，上下文窗口仍然是有限资源。显式的 上下文预算分配 确保 System Prompt、检索结果、对话历史和工具输出各占合理比例。

典型的 32K 窗口分配方案：System Prompt 15%（~4.8K）、Retrieval 40%（~12.8K）、History 20%（~6.4K）、Tool Output 15%（~4.8K）、Reserve 10%（~3.2K）。

先想 10 秒：如果历史对话占了 80% 的上下文窗口，会发生什么？

System Prompt 和检索结果会被挤出可用窗口。LLM 失去角色约束和行为规范，回答变得不可控。这是最常见的 Agent「失忆」原因——不是模型忘了，而是 System Prompt 被历史挤出了窗口。

常见问题（FAQ）

短期记忆和长期记忆怎么配合使用？
典型的配合模式：每次读取用户消息时，先从长期记忆（Vector DB）检索相关用户偏好和历史事实，注入到 System Prompt 中。当前轮对话的结果写入短期记忆（滑动窗口）。会话结束时（或达到窗口上限时），将旧的历史摘要后写入长期记忆。
Checkpoint 存储会不会太慢？
Checkpoint 是异步写入的，不会阻塞 Agent 执行。LangGraph 的 MemorySaver 和 SqliteSaver 都支持批量写入。对于高并发场景，建议使用 RedisSaver 或 PostgresSaver，写入延迟在毫秒级。
长期记忆可以跨用户共享吗？
可以，但需要严格权限控制。企业知识库（如产品手册、政策文档）可以跨用户检索。个人偏好（如用户喜欢简短回答）应隔离。建议在 Vector DB 中使用 metadata 字段标注权限范围，检索时按权限过滤。

LangGraph 中的 ReAct 实现

LangGraph 的 StateGraph 是实现 ReAct 循环的容器。它定义了三个要素：State（在节点间传递的数据结构）、Node（处理逻辑的单元）、Edge（控制节点间的流转路径）。

LangGraph ReAct 状态图
flowchart LR
    START([开始]) --> AgentNode[Agent 节点\n思考+推理]
    AgentNode -->|有工具调用| ToolNode[工具节点\n执行工具]
    AgentNode -->|无工具调用| END([结束])
    ToolNode --> AgentNode

ReAct 循环在 Agent 节点和 Tool 节点之间交替，直到 Agent 决定直接回答。

Python：StateGraph ReAct 完整实现

python 
from typing import TypedDict, Literal, Sequence
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolExecutor
from langchain_core.messages import BaseMessage, HumanMessage, AIMessage
import json

class AgentState(TypedDict):
    messages: Sequence[BaseMessage]
    step: int
    max_steps: int

def agent_node(state: AgentState, llm, tools) -> AgentState:
    messages = state["messages"]
    response = llm.invoke(messages, tools=tools)
    state["messages"].append(response)
    state["step"] = state.get("step", 0) + 1
    return state

def tool_node(state: AgentState, tool_executor: ToolExecutor) -> AgentState:
    last_message = state["messages"][-1]
    for tool_call in last_message.tool_calls:
        try:
            result = tool_executor.invoke(tool_call)
            state["messages"].append({
                "role": "tool", "content": str(result),
                "tool_call_id": tool_call["id"]
            })
        except Exception as e:
            state["messages"].append({
                "role": "tool", "content": f"工具调用失败：{str(e)}",
                "tool_call_id": tool_call["id"]
            })
    return state

def should_continue(state: AgentState) -> Literal["tools", "end"]:
    if state.get("step", 0) >= state.get("max_steps", 15):
        return "end"
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    return "end"

def build_react_agent(llm, tools):
    tool_executor = ToolExecutor(tools)
    graph = StateGraph(AgentState)
    graph.add_node("agent", lambda s: agent_node(s, llm, tools))
    graph.add_node("tools", lambda s: tool_node(s, tool_executor))
    graph.set_entry_point("agent")
    graph.add_conditional_edges(
        "agent", should_continue,
        {"tools": "tools", "end": END}
    )
    graph.add_edge("tools", "agent")
    return graph.compile()

react_agent = build_react_agent(llm, [query_order_tool, get_user_tool])
result = react_agent.invoke({
    "messages": [HumanMessage(content="我的订单 T001 到哪了？")],

def should_continue_advanced(state: AgentState) -> str:
    """增强的终止条件检测"""
    if state.get("step", 0) >= state.get("max_steps", 15):
        return "end"
    messages = state["messages"]
    # 重复工具调用检测
    tool_msgs = [m for m in messages
                 if m.get("role") == "assistant" and m.get("tool_calls")]
    if len(tool_msgs) >= 3:
        recent_calls = []
        for m in tool_msgs[-3:]:
            calls = [(tc["name"], json.dumps(tc["args"], sort_keys=True))
                     for tc in m["tool_calls"]]
            recent_calls.append(calls)
        if recent_calls[0] == recent_calls[1] == recent_calls[2]:
            return "end"  # 连续 3 次相同调用，终止
    # 连续失败检测
    tool_results = [m for m in messages if m.get("role") == "tool"]
    recent_fails = [m for m in tool_results[-3:]
                    if "失败" in m.get("content", "")]
    if len(recent_fails) >= 3:
        return "end"
    # 正常判断
    last = messages[-1]
    if hasattr(last, "tool_calls") and last.tool_calls:
        return "tools"
    return "end"

from langgraph.graph import StateGraph, END
from typing import TypedDict

class ChainState(TypedDict):
    input_text: str
    analysis: str
    draft: str
    final: str

def analyze_node(state: ChainState) -> ChainState:
    """Agent 1：分析用户请求"""
    response = llm.invoke(f"分析用户请求：{state['input_text']}")
    state["analysis"] = response.content
    return state

def draft_node(state: ChainState) -> ChainState:
    """Agent 2：基于分析生成草稿"""
    response = llm.invoke(f"基于分析生成回复草稿：{state['analysis']}")
    state["draft"] = response.content
    return state

def review_node(state: ChainState) -> ChainState:
    """Agent 3：审核并定稿"""
    response = llm.invoke(f"审核以下回复并优化：{state['draft']}")
    state["final"] = response.content
    return state

# 构建链式图
chain = StateGraph(ChainState)
chain.add_node("analyze", analyze_node)
chain.add_node("draft", draft_node)
chain.add_node("review", review_node)
chain.set_entry_point("analyze")
chain.add_edge("analyze", "draft")
chain.add_edge("draft", "review")
chain.add_edge("review", END)

不推荐
  • Agent 间通过全局变量共享状态
  • 没有超时控制，一个 Agent 卡住整个链
  • 链过长（超过 5 个 Agent）

Routing 动态路由

Routing 模式使用一个路由器 Agent 分析输入，然后派发到对应的专业 Agent。这是多技能组客服系统的核心架构。

Java：Routing 实现

java 
@Service
public class RoutingAgentService {
    private final ChatModel routerModel;
    private final Map<String, Agent> specializedAgents;

    public RoutingAgentService(ChatModel routerModel,
                               List<Agent> agents) {
        this.routerModel = routerModel;
        this.specializedAgents = agents.stream()
            .collect(Collectors.toMap(Agent::getSkill, Function.identity()));
    }

    public String handleRequest(String userMessage) {
        // 路由决策
        String intent = detectIntent(userMessage);
        // 派发
        Agent targetAgent = specializedAgents.get(intent);
        if (targetAgent == null) {
            return "无法识别您的需求，已转接人工";
        }
        return targetAgent.execute(userMessage);
    }

    private String detectIntent(String message) {
        String prompt = """
            用户消息：%s
            可选意图：客服咨询、售后服务、技术支持
            请只输出意图名称。
            """.formatted(message);
        return routerModel.call(prompt).trim();
    }
}

Swarm 与 Supervisor-Worker

Supervisor-Worker 模式中，一个 Supervisor Agent 负责任务分解和调度，多个 Worker Agent 并行执行子任务，结果汇聚后返回。这是最灵活的多 Agent 模式。

python 
class SwarmOrchestrator:
    """Supervisor-Worker 编排器"""

    def __init__(self, supervisor_llm, workers: dict):
        self.supervisor = supervisor_llm
        self.workers = workers  # {"数据分析": agent1, "报告生成": agent2}

    def run(self, task: str) -> str:
        # 1. Supervisor 分解任务
        plan = self.supervisor.invoke(
            f"将任务分解为子任务并指派给 Worker：\n{task}\n"
            f"可用 Worker：{list(self.workers.keys())}"
        )
        subtasks = self._parse_plan(plan.content)

        # 2. Worker 并行执行
        results = {}
        for name, subtask in subtasks.items():
            worker = self.workers.get(name)
            if worker:
                results[name] = worker.execute(subtask)

        # 3. 结果汇聚
        summary = self.supervisor.invoke(
            f"汇聚以下结果生成最终回复：\n{results}"
        )
        return summary.content

Handoff 机制

Handoff 是 Agent 之间交接对话上下文和控制权的机制。当当前 Agent 无法处理用户请求时，通过 Handoff 将对话转交给更合适的 Agent。

常见问题（FAQ）

Routing 模式和 Sequential 模式可以混合使用吗？
完全可以。实际系统中最常见的是混合架构：外层 Routing 做意图分发，内层 Sequential 做流程处理。例如 CorpAssist 中，路由 Agent 先判断用户意图是「售后」，然后进入退款的 Sequential 流程（验证身份->查订单->处理退款->通知）。
Handoff 和 Routing 的区别是什么？
Routing 是上游 Agent 主动派发任务给下游，适合有明确路由规则的场景。Handoff 是当前 Agent 主动将控制权移交给另一个 Agent，适合 Agent 在执行过程中发现自己无法胜任时转交。Handoff 通常携带完整的对话上下文。
多 Agent 如何共享记忆？
每个 Agent 有独立的短期记忆，但共享一个长期记忆存储。建议使用 Vector DB 或 Redis 作为共享记忆层，所有 Agent 读写同一命名空间。注意权限隔离——客服 Agent 不应读到财务 Agent 的记忆。见 记忆与上下文管理。

本章结论

  • Sequential：线性流水线，适合有明确先后顺序的处理流程。
  • Routing：意图路由分发，适合多技能组专业分工。
  • Swarm：Supervisor-Worker 架构，适合并行处理汇聚结果。
  • Handoff：Agent 间上下文交接，支撑灵活的动态转交。
  • 混合使用：实际系统通常混合多种模式达到最佳效果。

描述编写技巧

工具的 description 是模型理解工具的接口文档。好的 description 包含：工具的目的、何时调用、参数的含义和格式、返回什么。

不推荐
  • "查询订单" — 太短，缺触发条件和参数
  • "订单操作工具，输入参数 orderId" — 需要时调用不明确
  • "查数据库" — 从实现角度描述，非用户视角
推荐
  • "当用户询问订单状态或物流信息时调用。输入订单号，返回当前状态、物流单号和预计送达时间。如果订单不存在返回明确提示。"
  • 包含：目的+触发条件+参数说明+返回说明+边界情况

先想 10 秒：如果工具 description 写得太详细，模型会被绕晕吗？

不会。模型对工具 description 的解析能力很强，详细的内容反而提高准确率。但要避免无关信息——只描述"做什么"和"怎么用"，不要描述"为什么要做"或"内部实现"。通常 50-200 字内最合适。

错误恢复与重试

工具执行可能因为各种原因失败。设计良好的错误处理让 Agent 能从失败中恢复，而不是直接崩溃。错误处理分三个层次：

错误层次处理策略工具实现
参数校验在工具入口校验，返回具体错误Pydantic validator / @ToolParam
业务异常捕获已知异常，返回业务错误try-except 包裹
系统异常捕获兜底，返回降级信息Resilience4j / @safe_tool

Java：Resilience4j 包装

java 
@Component
public class ResilientToolWrapper {
    private final Retry retryPolicy;
    private final CircuitBreaker circuitBreaker;

    public ResilientToolWrapper() {
        this.retryPolicy = Retry.of("tool-retry",
            RetryConfig.custom()
                .maxAttempts(3)
                .waitDuration(Duration.ofMillis(500))
                .retryExceptions(TimeoutException.class, IOException.class)
                .build());
        this.circuitBreaker = CircuitBreaker.of("tool-cb",
            CircuitBreakerConfig.custom()
                .failureRateThreshold(50)
                .slidingWindowSize(10)
                .build());
    }

    public <T> T executeWithResilience(Supplier<T> toolCall, String toolName) {
        return Try.ofSupplier(
            CircuitBreaker.decorateSupplier(circuitBreaker,
                Retry.decorateSupplier(retryPolicy, toolCall))
        ).recover(throwable -> {
            log.error("Tool {} failed: {}", toolName, throwable.getMessage());
            return (T) ("工具执行失败，已记录日志。请稍后重试。");
        }).get();
    }
}

常见问题（FAQ）

工具描述中的关键词会帮助模型选择工具吗？
会。模型对 description 做语义理解，关键词确实影响匹配准确率。建议在 description 中包含场景相关的关键词。例如退款工具的 description 包含"退货、退款、取消订单、不满意"等触发词。但不要过度堆砌关键词导致描述不自然。
工具数量上限是多少？
大多数模型能同时接受 30-50 个工具定义。但模型在大量工具中选择的准确率会下降。建议单 Agent 的工具数控制在 10-20 个。超过 20 个时应考虑拆分为多 Agent 系统，每个 Agent 只带自己领域的工具。
工具返回值最长可以多长？
建议在 2000 字符以内。返回值会被注入到 LLM 的上下文窗口中，过长会挤占其他内容的空间。对于大数据量的工具（如查全部订单列表），应该：① 加分页参数；② 只返回汇总信息；③ 截断详情。参考 上下文预算分配。

安全护栏分层架构

Agent 的安全护栏应该覆盖从输入到输出的全链路。每一层解决不同维度的安全问题。

    Input[用户输入] --> L1[L1: 输入过滤\nXSS/SQL注入/Prompt注入]
    L1 --> L2[L2: 意图验证\n是否在允许范围内]
    L2 --> L3[L3: 工具权限\n白名单+操作级别]
    L3 --> L4[L4: 参数校验\n格式/范围/业务规则]
    L4 --> L5[L5: 执行沙箱\n限流/熔断/审计]
    L5 --> L6[L6: 输出检查\n脱敏/合规/内容安全]
    L6 --> Output[最终输出]

输入验证与注入防护

Prompt injection 是最常见的 Agent 安全威胁。攻击者通过在用户输入中嵌入指令，试图覆盖或绕过 Agent 的 System Prompt。

import re
from typing import Optional

class InputSanitizer:
    """Agent 输入过滤器"""

    INJECTION_PATTERNS = [
        r"忽略(之前|以上|所有).*指令",
        r"忘记.*(角色|规则|约束)",
        r"你(现在|接下来).*(是|扮演)",
        r"system\s*:",
        r"<\|im_start\|>",
        r"ignore.*(above|instructions|previous)",
    ]

    SENSITIVE_PATTERNS = [
        r"\b\d{18}\b",           # 身份证号
        r"\b1[3-9]\d{9}\b",      # 手机号
        r"\b\d{16,19}\b",        # 银行卡号
    ]

    @classmethod
    def sanitize(cls, text: str) -> tuple[bool, str, Optional[str]]:
        """清洗输入，返回(是否通过, 清洗后文本, 拒绝原因)"""
        # 1. 检测注入
        for pattern in cls.INJECTION_PATTERNS:
            if re.search(pattern, text):
                return False, text, "检测到可能的注入攻击"

        # 2. 检测敏感信息
        for pattern in cls.SENSITIVE_PATTERNS:
            if re.search(pattern, text):
                return False, text, "输入包含敏感信息，已拒绝"

        # 3. 长度检查
        if len(text) > 4000:
            text = text[:4000]

        return True, text, None

# 使用
is_safe, cleaned, reason = InputSanitizer.sanitize(user_input)
if not is_safe:
    return f"输入被拒绝：{reason}"

class ToolPermissionManager:
    """工具权限管理器"""

    def __init__(self):
        self.permissions = {
            "客服": {
                "query_order": "read",
                "create_ticket": "write",
            },
            "售后": {
                "query_order": "read",
                "refund_order": "write",
                "create_ticket": "write",
            },
            "管理员": {
                "query_order": "read",
                "refund_order": "write",
                "delete_order": "admin",
                "create_ticket": "write",
            }
        }

    def check_permission(self, role: str, tool: str, action: str) -> bool:
        """检查角色是否有权限"""
        if role not in self.permissions:
            return False
        if tool not in self.permissions[role]:
            return False
        required_level = self.permissions[role][tool]
        level_map = {"read": 1, "write": 2, "admin": 3}
        action_map = {"read": 1, "write": 2, "delete": 3}
        return level_map.get(required_level, 0) >= action_map.get(action, 0)
常见误区
  • 只依赖 LLM 判断安全性：LLM 本身可能被注入，安全规则必须在 LLM 外部实现。
  • 宽松的正则规则：注入检测模式太窄，攻击者可以轻松绕过。
  • 不做输出脱敏：工具返回了完整的用户数据，未脱敏直接输出。

class OutputGuardrail:
    """输出护栏：脱敏 + 内容安全 + 合规"""

    @staticmethod
    def desensitize(text: str) -> str:
        """脱敏处理"""
        text = re.sub(r'\b(\d{3})\d{4}(\d{4})\b', r'\1****\2', text)  # 手机号
        text = re.sub(r'\b(\d{6})\d{8}(\d{4})\b', r'\1********\2', text)  # 身份证
        return text

    @staticmethod
    def check_content_safety(text: str) -> tuple[bool, str]:
        """内容安全审核"""
        unsafe_patterns = [
            "银行账号.*\\d+", "密码.*\\S+",
            "内部资料", "机密"
        ]
        for pattern in unsafe_patterns:
            if re.search(pattern, text):
                return False, "输出包含受限信息"
        return True, text

    @classmethod
    def check(cls, output: str) -> str:
        """完整输出检查"""
        output = cls.desensitize(output)
        ok, result = cls.check_content_safety(output)
        if not ok:
            return "抱歉，无法提供此信息（已触发内容安全规则）"
        return result

常见问题（FAQ）

Guardrails 会影响 Agent 的性能吗？
每层护栏都有计算开销。输入过滤和参数校验通常在微秒级，影响可以忽略。内容安全审核如果用 LLM 做（如用另一个模型审核输出），延迟会显著增加。建议：规则类的检查用代码实现，语义类的检查用轻量模型或采样策略。
如何防止 LLM 被注入后调用高危工具？
三管齐下：① 在 Agent 层建立工具白名单，LLM 只能调用白名单内的工具；② 高危工具（转账、删除）需要额外的人工确认；③ 在工具层做参数校验，拒绝不合理参数。见 人机协作 中的审批流程。
Prompt Injection 检测的准确率有多高？
基于正则或关键词的检测（如本章示例）准确率 70-80%，会有误报和漏报。建议使用专门的安全模型（如 Nvidia NeMo Guardrails、Lakera Guard），准确率可达到 95%+。但安全模型本身也需要持续更新对抗新的攻击手法。

本章结论

  • 六层护栏：从输入过滤到输出检查，覆盖全链路。
  • 注入防护：输入必须经过清洗才能进入 Agent 流程。
  • 权限控制：基于角色的工具白名单，高危操作需人工确认。
  • 输出合规：脱敏、内容安全、格式合规三重检查。
  • 外部工具：NeMo Guardrails 和自定义规则结合使用效果最佳。

class AgentEvaluator:
    """Agent 离线评测引擎"""

    def __init__(self, agent, test_cases: list):
        self.agent = agent
        self.test_cases = test_cases

    def evaluate(self) -> dict:
        results = {"passed": 0, "failed": 0, "details": []}
        for case in self.test_cases:
            result = self._evaluate_single(case)
            results["details"].append(result)
            if result["passed"]:
                results["passed"] += 1
            else:
                results["failed"] += 1
        results["pass_rate"] = results["passed"] / len(self.test_cases)
        return results

    def _evaluate_single(self, case: dict) -> dict:
        response = self.agent.invoke({"messages": [HumanMessage(content=case["input"])]})

        checks = {
            "tool_correct": self._check_tools(response, case.get("expected_tools", [])),
            "response_quality": self._check_response(response, case.get("expected_response_contains", [])),
        }
        passed = all(checks.values())
        return {"id": case["id"], "passed": passed, "checks": checks}

    def _check_tools(self, response, expected_tools):
        actual_tools = [m.name for m in response["messages"] if hasattr(m, "name")]
        return all(t in actual_tools for t in expected_tools)

    def _check_response(self, response, expected_contains):
        last_msg = response["messages"][-1].content
        return all(kw in last_msg for kw in expected_contains)

常见问题（FAQ）

评测数据集的标注工作量太大了，有什么省力的方法？
可以先用 LLM 生成初始数据集（合成数据），然后人工抽样审核修正。也可以从生产日志中采样用户问题和 Agent 回答，由人工标注是否合格。建议从 50 个用例开始，覆盖核心场景，然后逐步扩展到 200-500 个。
Agent 的回答是自然语言，如何自动化判断是否正确？
常用的方法有：① 关键词/正则匹配（简单但不够准确）；② LLM-as-Judge（用另一个 LLM 打分，准确率 85%+）；③ 语义相似度（用 embedding 比较回答和标准答案的相似度）。建议组合使用——先用规则做预检，再用 LLM 判分。
在线指标多久看一次比较合适？
CSAT 和转人工率建议每日看，因为样本量太小时指标波动大。延迟和 Token 消耗建议实时监控，设置告警阈值。每周汇总趋势分析，发现 Agent 表现是否在退化。每次模型升级或 Prompt 变更后需要密切观察 24-48 小时。

循环检测与熔断

死循环是 Agent 最常见的故障模式。检测死循环需要监控工具调用序列：

python 
from collections import defaultdict
import hashlib

class LoopDetector:
    """Agent 死循环检测器"""

    def __init__(self, max_repeated_calls=3, window_size=6):
        self.call_history: list = []
        self.max_repeated = max_repeated_calls
        self.window_size = window_size

    def record_call(self, tool_name: str, args: dict, result: str):
        """记录一次工具调用"""
        call_hash = self._hash_call(tool_name, args)
        self.call_history.append({
            "tool": tool_name,
            "args_hash": call_hash,
            "result_hash": hashlib.md5(result.encode()).hexdigest()
        })

    def detect_loop(self) -> tuple[bool, str]:
        """检测死循环"""
        if len(self.call_history) < self.window_size:
            return False, ""

        recent = self.call_history[-self.window_size:]

        # 检测 1: 相同工具相同参数连续调用
        for tool in set(c["tool"] for c in recent):
            calls = [c for c in recent if c["tool"] == tool]
            if len(calls) >= self.max_repeated:
                args_hashes = [c["args_hash"] for c in calls[-self.max_repeated:]]
                if len(set(args_hashes)) == 1:
                    return True, f"死循环检测: 工具 {tool} 相同参数连续调用 {self.max_repeated} 次"

        # 检测 2: 来回切换两个工具
        if len(recent) >= 4:
            pairs = [(recent[i]["tool"], recent[i+1]["tool"])
                     for i in range(0, len(recent)-1)]
            if len(set(pairs)) == 1 and len(pairs) >= 3:
                return True, f"死循环检测: 工具间来回切换 {pairs[0]}"

        return False, ""

    def _hash_call(self, tool_name: str, args: dict) -> str:
        return hashlib.md5(
            f"{tool_name}:{json.dumps(args, sort_keys=True)}".encode()
        ).hexdigest()

# 集成到 Agent
detector = LoopDetector()

def tool_node_with_detection(state: AgentState) -> AgentState:
    for tool_call in state["messages"][-1].tool_calls:
        result = execute_tool(tool_call)
        detector.record_call(
            tool_call["name"],
            tool_call["args"],
            result
        )
        is_loop, reason = detector.detect_loop()
        if is_loop:
            state["loop_detected"] = True
            state["messages"].append(AIMessage(content=f"已检测到循环，转人工处理：{reason}"))
            break
        state["messages"].append({"role": "tool", "content": result})
    return state

@Component
public class AgentCircuitBreaker {
    private final Map<String, AtomicInteger> failureCounts = new ConcurrentHashMap<>();
    private final Map<String, Long> circuitOpenUntil = new ConcurrentHashMap<>();
    private static final int THRESHOLD = 3;
    private static final long TIMEOUT_MS = 30_000;

    public boolean isAvailable(String toolName) {
        Long until = circuitOpenUntil.get(toolName);
        if (until != null) {
            if (System.currentTimeMillis() < until) {
                return false;
            }
            circuitOpenUntil.remove(toolName);
            failureCounts.put(toolName, new AtomicInteger(0));
        }
        return true;
    }

    public void recordFailure(String toolName) {
        AtomicInteger count = failureCounts.computeIfAbsent(
            toolName, k -> new AtomicInteger(0));
        if (count.incrementAndGet() >= THRESHOLD) {
            circuitOpenUntil.put(toolName,
                System.currentTimeMillis() + TIMEOUT_MS);
            log.warn("Circuit opened for tool: {}", toolName);
        }
    }

    public void recordSuccess(String toolName) {
        failureCounts.computeIfAbsent(toolName, k -> new AtomicInteger(0)).set(0);
    }
}

重试策略与退避

Agent 工具调用失败时，重试是常见的恢复策略。但重试需要有策略——不加控制的立即重试可能加剧故障。

python 
import time
import random
from functools import wraps

def retry_with_backoff(
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 30.0,
    jitter: bool = True
):
    """指数退避重试装饰器"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exception = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except (TimeoutError, ConnectionError) as e:
                    last_exception = e
                    delay = min(base_delay * (2 ** attempt), max_delay)
                    if jitter:
                        delay += random.uniform(0, delay * 0.1)
                    if attempt < max_retries - 1:
                        time.sleep(delay)
            raise last_exception
        return wrapper
    return decorator

# 使用
@retry_with_backoff(max_retries=3, base_delay=1.0)
def call_external_api(url: str) -> dict:
    response = requests.get(url, timeout=5)
    response.raise_for_status()
    return response.json()

class DegradationManager:
    """降级管理器"""

    def __init__(self):
        self.failure_count = defaultdict(int)
        self.degradation_level = 0  # 0=正常, 1=简化, 2=缓存, 3=转人工

        self.failure_count[tool_name] += 1
        total_failures = sum(self.failure_count.values())

常见问题（FAQ）

熔断器打开后，什么时候关闭？
熔断器打开后进入半开状态（通常 30-60 秒后），允许少量请求通过测试下游是否恢复。如果请求成功则关闭熔断器，如果仍然失败则继续保持打开状态。LangGraph 内置的熔断器使用 Resilience4j 的实现。
降级和 Fallback 有什么区别？
降级（Degrade）是主动降低服务质量，在故障发生前或发生时减少功能。Fallback 是执行备选方案，当主方案失败时切换到备选。例如：降级可能是跳过非必须的工具调用，Fallback 可能是使用缓存数据替代实时查询。两者常常配合使用。
重试次数太多会有什么问题？
过多重试会：① 增加 Agent 执行延迟，用户等待时间变长；② 放大对下游系统的压力，可能导致系统崩溃；③ 浪费 Token 成本，每次重试都产生 LLM 调用。建议重试不超过 3 次，配合指数退避和熔断器使用。

本章结论

  • 死循环检测：基于工具调用序列的哈希和模式匹配，检测重复调用。
  • 熔断器：连续失败 N 次后暂时禁用工具，避免级联故障。
  • 重试策略：指数退避 + 随机抖动，避免重试风暴。
  • 降级策略：三级降级（简化/缓存/转人工），确保故障下仍能服务。
  • 容量规划：基于历史数据规划 Agent 并发和限流。

from langgraph.graph import StateGraph, END
from langgraph.checkpoint import MemorySaver

# 定义需要审批的工具
APPROVAL_REQUIRED_TOOLS = ["refund_order", "delete_order", "transfer_money"]

def tool_node_with_approval(state: AgentState) -> AgentState:
    """需要审批的工具节点"""
    last_message = state["messages"][-1]
    for tool_call in last_message.tool_calls:
        tool_name = tool_call["name"]
        if tool_name in APPROVAL_REQUIRED_TOOLS:
            # 保存审批请求信息到 state
            state["pending_approval"] = {
                "tool": tool_name,
                "args": tool_call["args"],
                "status": "pending"
            }
            # 触发 interrupt_before 会在此处暂停
            # 等待人工审批
        result = execute_tool(tool_call)
        state["messages"].append({"role": "tool", "content": str(result)})
    return state

# 构建图，设置中断
graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node_with_approval)
graph.set_entry_point("agent")
graph.add_conditional_edges(
    "agent",
    should_continue,
    {"tools": "tools", "end": END}
)
graph.add_edge("tools", "agent")

# 编译时指定中断位置
app = graph.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["tools"]  # 在 tools 节点前中断
)

# 执行
config = {"configurable": {"thread_id": "session_001"}}
# 第一次 invoke：Agent 思考后会中断在 tools 前
result = app.invoke({"messages": [user_msg]}, config)

# 查看中断状态
pending = result.get("pending_approval")
if pending:
    print(f"等待审批：{pending['tool']} - {pending['args']}")
    # 人工审批通过后
    app.update_state(config, {"pending_approval": {"status": "approved"}})
    # 恢复执行
    result = app.invoke(None, config)

常见问题（FAQ）

中断后 Agent 的状态会丢失吗？
不会。LangGraph 的 Checkpoint 机制在中断时自动保存完整状态。恢复时从 Checkpoint 加载状态，从断点继续执行。即使进程重启，只要使用持久化 Checkpoint（如 SqliteSaver、RedisSaver），状态也不会丢失。
审批超时后应该怎么处理？
超时策略取决于业务场景：① 查询类操作可以自动超时通过（返回已存在的数据）；② 资金类操作应该超时拒绝并通知用户；③ 关键操作可以升级到更高审批层级。建议在审批请求中明确标注超时策略。
转人工时如何保证上下文完整传递？
转人工时需要将完整对话历史、Agent 推理轨迹、已调用的工具结果打包传递给人工客服。可以序列化为 JSON 格式，在客服系统中展示 Agent 的决策链。建议包含 Agent 的思考过程，帮助客服快速理解背景。

MCP 协议原理

MCP（Model Context Protocol）定义了 Agent 与外部工具之间的标准接口。它采用 Client-Server 架构：MCP Server 提供工具能力，MCP Client（Agent）发现和调用这些工具。

常见问题（FAQ）

MCP 和 Function Calling 是什么关系？
Function Calling 是 LLM 的能力——模型声明需要调用某个函数。MCP 是工具的标准——定义工具如何被发现和调用。两者是互补的：MCP 提供工具注册和发现，Function Calling 提供模型层面的调用声明。Agent 框架（LangGraph、Spring AI）将两者结合起来。
A2A 和 MCP 的区别是什么？
MCP 是 Agent-工具协议，解决 Agent 如何调用外部工具。A2A 是 Agent-Agent 协议，解决 Agent 之间如何通信协作。MCP 适合工具集成（查询 API、操作数据库），A2A 适合 Agent 间的任务委派和协同（客服 Agent 请求财务 Agent 协助）。
MCP Server 如何保证安全？
MCP Server 需要实现鉴权和权限控制。标准做法：① 使用 OAuth2 或 API Key 认证；② 每个工具定义调用权限级别；③ 工具调用记录审计日志；④ 敏感操作需要二次确认。MCP 协议本身不强制安全措施，需要在 Server 层实现。

  • MCP + A2A：两者互补，MCP 管工具，A2A 管 Agent。
  • 双栈实现：Python MCP SDK 和 Java Spring AI MCP 概念一致。
  • 安全考量：MCP Server 需要实现鉴权、权限和审计。

class ModelRouter:
    """模型路由器：根据任务复杂度选择模型"""

    def __init__(self):
        self.routes = {
            "simple": {"model": "qwen-turbo",    "cost_per_1k": 0.0015},
            "medium": {"model": "qwen-plus",     "cost_per_1k": 0.008},
            "complex": {"model": "qwen-max",     "cost_per_1k": 0.03},
        }

    def classify(self, task: str) -> str:
        """判断任务复杂度"""
        prompt = f"""判断以下任务复杂度，只返回 simple/medium/complex：
        简单：问候、查询订单状态、查询时间
        中等：创建工单、修改设置、多步骤查询
        复杂：数据分析、多步推理、生成报告
        任务：{task}"""
        level = classifier_llm.invoke(prompt).content.strip()
        return level if level in self.routes else "medium"

    def route(self, task: str) -> str:
        level = self.classify(task)
        return self.routes[level]["model"]

    def estimate_cost(self, task: str, tokens: int) -> dict:
        level = self.classify(task)
        model_conf = self.routes[level]
        cost = (tokens / 1000) * model_conf["cost_per_1k"]
        return {"model": model_conf["model"], "cost": round(cost, 4)}

步数预算优化

每步 ReAct 循环消耗一次 LLM 调用。步数预算优化是降低成本的直接手段。

class StepBudgetManager:
    """步数预算管理器"""

    def __init__(self):
        self.metrics = {
            "total_executions": 0,
            "total_steps": 0,
            "cost_by_tool": defaultdict(float)
        }

    def get_budget(self, task_type: str) -> int:
        """根据任务类型返回建议的 max_steps"""
        budgets = {
            "query": 3,       # 简单查询，1-2 步
            "ticket": 5,      # 工单处理，3-5 步
            "refund": 8,      # 退款流程，5-8 步
            "report": 15,     # 报告生成，10-15 步
        }
        return budgets.get(task_type, 10)

    def record_execution(self, steps: int, model: str, tokens: int):
        self.metrics["total_executions"] += 1
        self.metrics["total_steps"] += steps
        cost = self._calculate_cost(model, tokens)
        logger.info(f"执行完成: {steps}步, {model}, {tokens}tokens, ¥{cost}")

    def get_avg_cost(self) -> float:
        if self.metrics["total_executions"] == 0:
            return 0
        return self._calculate_cost("avg", 0) / self.metrics["total_executions"]

    def _calculate_cost(self, model: str, tokens: int) -> float:
        rates = {"qwen-turbo": 0.0015, "qwen-plus": 0.008, "qwen-max": 0.03}
        rate = rates.get(model, 0.03)
        return (tokens / 1000) * rate

缓存与复用

缓存减少重复的 LLM 调用。Agent 中可缓存的内容包括：工具调用结果、LLM 响应、推理链。

from functools import lru_cache
import hashlib
import redis

class AgentCache:
    """Agent 缓存"""

    def __init__(self, redis_client=None):
        self.cache = redis_client or redis.Redis(host="localhost", port=6379)
        self.ttl = 300  # 5 分钟

    def _make_key(self, prefix: str, *args, **kwargs) -> str:
        raw = f"{prefix}:{args}:{json.dumps(kwargs, sort_keys=True)}"
        return f"agent:{hashlib.md5(raw.encode()).hexdigest()}"

    def get_tool_result(self, tool_name: str, **kwargs) -> str | None:
        key = self._make_key("tool", tool_name, kwargs)
        result = self.cache.get(key)
        return result.decode() if result else None

    def set_tool_result(self, tool_name: str, result: str, **kwargs):
        key = self._make_key("tool", tool_name, kwargs)
        self.cache.setex(key, self.ttl, result)

    def get_llm_response(self, prompt: str) -> str | None:
        key = self._make_key("llm", prompt)
        result = self.cache.get(key)
        return result.decode() if result else None

    def set_llm_response(self, prompt: str, response: str):
        key = self._make_key("llm", prompt)
        self.cache.setex(key, self.ttl, response)

  • 成本只看模型单价：步数才是真正的成本乘数，一个 5 步的 Agent 成本是单步的 5 倍。

常见问题（FAQ）

模型路由增加了一次分类 LLM 调用，这是否增加了成本？
分类器本身可以使用最便宜的小模型（如 GPT-4o-mini），单次分类调用成本约 ¥0.001。如果分类器将请求路由到小模型，单次可节省 ¥0.05-0.08。路由的准确率通常在 90%+，净节省显著。建议用规则或关键词分类器代替 LLM 分类以进一步降低成本。
缓存会导致 Agent 回复过时信息吗？
取决于缓存 TTL 设置。订单状态等易变数据的缓存 TTL 应缩短（2-5 分钟），知识库等静态数据的 TTL 可以延长（1 小时+）。关键是判断数据的时效性要求——实时性要求高的数据（如物流状态）不缓存或短缓存。
除了模型选择和缓存，还有什么降本手段？
① 上下文压缩：减少历史对话的 Token 消耗（见 记忆与上下文管理）；② Prompt 精简：System Prompt 精简 20% 可以节省大量 Token；③ 小模型优先：先用小模型尝试，如果小模型置信度不足再升级到大模型；④ 批量处理：将多个工具调用合并为一次 LLM 调用。

本章结论

  • 成本模型：LLM 调用是主要成本，步数是最大的成本乘数。
  • 模型路由：根据任务复杂度选择不同成本的模型，简单任务用小模型。
  • 步数预算：根据任务类型设置不同的 max_steps 上限。
  • 缓存策略：缓存工具结果和 LLM 响应，减少重复计算。
  • 持续监控：记录每次执行的 Token 和步数，设置日环比告警。

面试：Agent 架构设计

以下是 Agent 架构面试中的高频问题及回答框架：

Q1: 你如何为业务场景选择合适的 Agent 范式？

回答框架：从三个维度评估——任务复杂度（步骤数量和依赖关系）、实时性要求（用户能等多久）、质量要求（输出需要审核吗）。简单场景用 ReAct，多步骤用 Plan-Execute，高精度叠加 Reflexion。最好举 CorpAssist 的例子：S2 客服用 ReAct（实时交互），S3 办公自动化用 Plan-Execute（复杂报告任务）。

Q2: Agent 工具调用失败怎么办？

回答框架：分层处理——参数校验：在工具入口校验并返回具体错误；业务异常：捕获已知异常返回友好提示；系统异常：使用重试和熔断器，最终兜底转人工。关键点：异常信息要让模型能理解并决定下一步，不要抛到外层。

Q3: 多 Agent 系统中如何保证数据一致性？

回答框架：核心策略——使用共享存储（Redis/数据库）持久化关键状态；每个操作保持幂等性；使用 Saga 模式管理跨 Agent 事务；设置补偿操作回滚失败步骤。参考分布式系统设计模式。

Q4: 如何评测 Agent 的输出质量？

回答框架：离线+在线双轨。离线：构建标注数据集，使用 LLM-as-Judge 评分。在线：CSAT、转人工率、首次解决率。CI 集成：每次变更后运行回归测试，设置质量门禁。参考 Agent 评测方法论。

生产排障案例

案例 1：客服 Agent 死循环导致 1000+ 次无效调用

现象：用户问「我的订单到哪了」，Agent 在 query_order 和 transfer_to_human 之间切换 15 次。

排查：Trace 显示每次 query_order 返回正常，但模型认为「用户语气不满需要退款」，调用 refund_order 失败又回退。

根因：System Prompt 缺少「先确认用户意图再操作」的约束。

修复：System Prompt 增加约束 + 工具 description 增加触发条件 + 连续失败后转人工。

案例 2：上下文溢出导致 Agent 失忆

现象：长对话中 Agent 突然不遵守 System Prompt，开始编造回答。

排查：Trace 发现上下文窗口中历史对话占了 90%，System Prompt 被挤到窗口后半部分。

根因：未设置上下文预算，历史对话无限累积。

修复：滑动窗口限制 10 轮 + 旧轮次生成摘要 + 每次 LLM 调用前重写 System Prompt。

Agent 排障应该先看什么？
第一优先：LangSmith Trace。Trace 告诉你 Agent 每一步做了什么。第二优先：工具调用日志（入参和出参）。第三优先：LLM 调用日志（prompt 和 response）。比日志高效 10 倍。
模型升级后 Agent 表现突然变差，可能的原因是什么？
模型在 Function Calling 能力上发生变化；输出格式变化（如思考过程）；对 System Prompt 遵循度变化。解决方案：每次模型升级前运行回归测试，对比升级前后的关键指标。

通往 AGI 的 Agent 路径

Agent 被认为是通往通用人工智能（AGI）的关键路径。从当前的 ReAct Agent 到 AGI，需要解决以下核心挑战：

  1. 长期记忆：当前 Agent 的记忆以会话为主，远远达不到人类的长期记忆能力。
  2. 持续学习：Agent 执行后应能从经验中学习，改善未来表现，而不会遗忘已学内容。
  3. 元认知：Agent 需要知道自己的能力边界——哪些能做、哪些不能做、什么时候需要求助。
  4. 价值对齐：Agent 的行为需要符合人类价值观和伦理规范。
  5. 多模态理解：真正的智能体应能像人类一样融合视觉、听觉、语言等多种感知。

常见问题（FAQ）

2026-2027 年哪个 Agent 方向最值得投入？
多模态 Agent 和 Agent 治理是两个最值得投入的方向。多模态扩展 Agent 的能力边界（从文本到图像/语音），治理体系确保 Agent 安全可靠地生产运行。同时，关注 MCP/A2A 协议生态，支持标准协议的 Agent 将更有生命力。
低代码 Agent 平台会取代专业开发吗？
不会完全取代。低代码平台适合简单场景（查询、通知），但复杂 Agent（多步推理、复杂工具编排、自定义安全策略）仍然需要专业开发。趋势是「低代码 + 专业开发」双轨并行，平台降低门槛，专业开发保障深度。
Agent 会在哪些行业最先大规模落地？
客服（已规模化）、运维（自动化排障）、软件开发（代码 Agent）、营销（个性化内容生成）是当前落地最快的领域。其次是金融（风控分析）、医疗（病历摘要）、教育（个性化教学）。制约因素主要是安全性和可靠性。

本章结论

  • 多模态：视觉、语音、代码多模态是下一波主要技术方向。
  • 平台化：低代码平台降低门槛，但复杂 Agent 仍需专业开发。
  • 治理框架：身份权限、审计合规、数据治理、风险控制四维度。
  • 协议标准：MCP 和 A2A 协议将推动 Agent 生态互联互通。
  • AGI 路径：Agent 从 L1（单步 ReAct）向 L4（通用智能体）逐步演进。
