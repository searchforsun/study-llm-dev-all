事件流断连

CorpAssist BFF 通过 TEXT_EVENT_STREAM 消费 AgentScope Java streamEvents()。断连表现为客户端收不到后续 tool_call 事件，或 Gateway 返回 504。与 Python FastAPI SSE 类似，但 Java 侧常见 Reactor 背压与 Spring Cloud Gateway 超时。

SSE 事件流断连排查流程
flowchart TD
    Alert[客户端 SSE 中断 / 504] --> Trace[查 trace_id 对应 Event 日志]
    Trace --> Q1{Agent 侧是否仍在 emit?}
    Q1 -->|是| GW[检查 Gateway / Nginx 超时]
    Q1 -->|否| Q2{是否 CancellationException?}
    Q2 -->|是| Client[客户端断开或未读 Flux]
    Q2 -->|否| Q3{Reactor 线程阻塞?}
    Q3 -->|是| Block[工具同步 IO 阻塞 event 线程]
    GW --> Fix1[proxy_read_timeout / response-timeout]
    Client --> Fix2[前端 EventSource reconnect + Last-Event-ID]
    Block --> Fix3[工具改 boundedElastic / 虚拟线程]

先确认 Agent 是否仍在发事件，再查网关超时与 Reactor 阻塞。

工具权限拒绝

工具权限拒绝 发生在 practice-03 的 Permissions 三态：deny 永久拒绝、approve 待人工、allow 放行。生产故障常表现为 Agent 反复尝试同一工具或卡在 HITL 等待。

权限故障征象
  • Trace 连续 permission_denied 同工具
  • 用户已审批但 Agent 仍 pending
  • bookMeeting 需审批，模型改调 sendEmail 绕过
防护
  • deny 返回结构化 reason 供模型换策略
  • 审批回调写 audit 表 + 唤醒外部执行循环
  • 敏感工具强制 approve，禁止 allow 默认
java 
// PermissionMiddleware — deny 时返回可解析错误，避免 silent loop
@Component
public class CorpAssistPermissionGate implements ToolPermissionEvaluator {

    @Override
    public PermissionDecision evaluate(ToolCallContext ctx) {
        if ("bookMeeting".equals(ctx.toolName()) && !ctx.userRoles().contains("CALENDAR_WRITE")) {
            return PermissionDecision.deny(
                "PERMISSION_DENIED: 用户无 CALENDAR_WRITE，请引导用户申请权限或改查可用会议室");
        }
        if (isSensitive(ctx.toolName())) {
            return PermissionDecision.approve("需人工确认订会议室操作");
        }
        return PermissionDecision.allow();
    }
}

// 外部执行循环：审批完成后 resume
@PostMapping("/agent/resume/{approvalId}")
public Mono<Msg> resume(@PathVariable String approvalId, @RequestBody ApprovalResult result) {
    return approvalStore.complete(approvalId, result)
        .flatMap(pending -> agent.resume(pending.ctx(), result.toPermissionOutcome()));
}

模型绕过审批换工具调用怎么办？
在 PermissionGate 对「等效副作用」工具统一审批（如 bookMeeting 与 createCalendarEvent）；golden set 增加 must_not_call 断言；systemPrompt 禁止绕过审批。

Session 丢失

Session 丢失 表现为第二轮对话 Agent 「失忆」、重复询问已提供信息，或 Sub-agent 拿不到主管上下文。CorpAssist 默认 Redis SessionStore，故障多在 TTL、key 前缀与序列化版本。

征象排查命令 / 日志根因修复
每轮像新用户Redis GET session:{id} 为空BFF 未透传 sessionId固定 X-Session-Id header
30 分钟后失忆TTL 约 1800Session TTL 过短延长至 24h + sliding refresh
升级后全员失忆反序列化异常Msg 类结构变更版本化 key session:v2:
Sub-agent 无上下文spawn 事件无 parent_sessionRuntimeContext 未继承spawn 时 copy sessionId + tenantId
java 
@Configuration
public class SessionStoreConfig {

    @Bean
    public SessionStore redisSessionStore(RedisTemplate<String, byte[]> redis) {
        return RedisSessionStore.builder()
            .redis(redis)
            .keyPrefix("corp-assist:session:v2:")
            .ttl(Duration.ofHours(24))
            .slidingExpiration(true)
            .serializer(new MsgHistorySerializer(/* schema version */ 2))
            .build();
    }
}

// 诊断：记录 session miss
@Component
public class SessionDiagnosticFilter implements WebFilter {
    public Mono<Void> filter(ServerWebExchange ex, WebFilterChain chain) {
        String sid = ex.getRequest().getHeaders().getFirst("X-Session-Id");
        if (sid != null && !sessionStore.exists(sid)) {
            log.warn("SESSION_MISS trace_id={} sessionId={}", traceId(ex), sid);
        }
        return chain.filter(ex);
    }
}

Sub-agent 死锁

Sub-agent 死锁 常见于 practice-02：主管 Agent 同步等待 Sub-agent，Sub-agent 又通过 agent_send 回调主管，或两个 Sub-agent 互相等待。Trace 表现为长时间无新 Event、线程 pool 耗尽。

Sub-agent 死锁模式
flowchart TD
    M[主管 Agent sync spawn Worker-A] --> W[Worker-A 等待主管回复]
    W -->|agent_send 回主管| M
    M -->|blocked on Future| W
    style M fill:#f96
    style W fill:#f96

同步 spawn + 双向 agent_send 是最典型死锁环。

模式Trace 特征修复
同步环spawn 与 send 交叉，无 tool_resultSub-agent 改后台模式；主管用事件回调
线程耗尽大量 BLOCKED on ReActAgent.call限制并发 spawn；虚拟线程 + 超时
权限阻塞Sub-agent tool 永久 approve子 Agent 独立 Permission 策略
maxIterations 互等双 Agent 均触顶主管设汇总步数预算，禁止递归 spawn

// 反模式：同步 spawn 并 block
// Msg result = subAgent.call(task, ctx).block();  // 若 subAgent 回调主管 → 死锁

// 推荐：后台 spawn + 事件驱动汇总
agentSpawnService.spawnBackground("Worker-Research", task, ctx)
    .subscribe(
        ev -> eventBus.publish(ev),
        err -> log.error("sub-agent failed", err)
    );

// 主管在 SubAgentCompletedEvent 时继续 ReAct，而非 block 等待
@EventListener
public void onSubAgentDone(SubAgentCompletedEvent ev) {
    orchestrator.continueWithResult(ev.parentCtx(), ev.resultMsg());
}
STAR 案例模板（面试）

S：CorpAssist 报告生成主管-双 Worker 上线后 P95 从 8s 升至 120s。
T：定位 AgentScope Java Sub-agent 阻塞。
A：导出 trace_id，发现主管 sync block + Worker agent_send 回环；改为 background spawn + SubAgentCompletedEvent。

本章结论

  • 事件流断连：先确认 Agent emit 完整，再查 Gateway 超时与 Reactor 阻塞。
  • 工具权限拒绝：deny 须结构化 reason；approve 须外部 resume 循环。
  • Session 丢失：稳定 sessionId、合理 TTL、Msg 序列化版本化。
  • Sub-agent 死锁：禁止 sync spawn + 双向 send；用后台模式与事件汇总。

判断练习

  1. （排障顺序）生产 SSE 中断：A 调大 Gateway timeout → B 查 Agent Trace 是否仍 emit → C 改前端 EventSource。最优顺序是？

    参考答案

    B → A → C。 先确认 Agent 侧是否完整 emit（定位问题在传输还是 Agent）；若 Agent 完整再调 Gateway；最后优化客户端重连。未确认 Agent 就调 timeout 可能掩盖 Reactor 阻塞。

  2. 主管 Agent 使用 subAgent.call(task, ctx).block()，Sub-agent 内 agent_send 回主管要澄清需求——会发生什么？

    参考答案

    典型死锁。 主管线程 block 等 Sub-agent，Sub-agent 等主管回复。应改为 background spawn，主管在 SubAgentCompletedEvent 或异步 Msg 通道继续。
