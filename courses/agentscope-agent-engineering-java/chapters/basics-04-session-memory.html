本章先记住 3 件事
  • Agent 实例无会话态：同一 ReActAgent 可并发服务多 Session，状态由 Store 按 (userId, sessionId) 加载。
  • 短期 = AgentState.context：每轮 call() 读写完整可见 Msg 链。
  • Compaction 先于硬溢出：推理前自动压缩；长任务可升级 Harness 完整 compaction 栈。

Session 生命周期

Session 生命周期 由「一次 call() 的 RuntimeContext」与「跨调用的 AgentStateStore」共同定义：调用开始加载状态，调用结束写回；同 Session 并发请求会被框架串行化。

Session 与 AgentStateStore
sequenceDiagram
    participant BFF as Spring BFF
    participant A as ReActAgent
    participant Store as AgentStateStore

    BFF->>A: call(msgs, RuntimeContext)
    A->>Store: load(userId, sessionId)
    Store-->>A: AgentState
    A->>A: ReAct 推理 + 工具
    A->>Store: save(AgentState)
    A-->>BFF: Msg 回复

每次 call 结束写回 AgentState；下一请求同 sessionId 自动恢复。

多 Session 并发（单 Agent 实例）

java 
import io.agentscope.core.ReActAgent;
import io.agentscope.core.agent.RuntimeContext;
import io.agentscope.core.message.UserMessage;
import io.agentscope.core.state.JsonFileAgentStateStore;
import java.nio.file.Paths;
import java.util.List;

ReActAgent agent = ReActAgent.builder()
        .name("CorpAssistS3")
        .sysPrompt("你是 CorpAssist 办公助手。")
        .model("dashscope:qwen-plus")
        .stateStore(new JsonFileAgentStateStore(
                Paths.get(System.getProperty("user.home"), ".agentscope/sessions")))
        .build();

RuntimeContext alice = RuntimeContext.builder()
        .userId("alice").sessionId("sess-001").build();
RuntimeContext bob = RuntimeContext.builder()
        .userId("bob").sessionId("sess-002").build();

agent.call(List.of(new UserMessage("我是华东区销售张三")), alice).block();
agent.call(List.of(new UserMessage("查我下周会议")), bob).block();
先想 10 秒：为何不每个 HTTP 请求 new 一个 ReActAgent？

Agent 实例无会话态，创建成本高且浪费连接池。Spring 中单例 Bean + 每请求传入不同 RuntimeContext 是推荐模式；状态由 Store 隔离，与 ChatClient 单例 + sessionId Advisor 类似。

Store 实现适用CorpAssist 场景
InMemoryAgentStateStore单元测试CI 无 Redis
JsonFileAgentStateStore本地 / 单节点开发 Demo
RedisAgentStateStoreK8s 多副本S3 生产推荐
MysqlAgentStateStore需 SQL 审计合规留存

短期记忆管理

AgentScope Java 2.0 已弃用 1.x 的 Memory 接口；短期记忆 即 AgentState 中的对话上下文（getContext()）。ReAct 每轮推理前读取完整可见历史，工具结果也以 Msg 形式 append。

CorpAssist · 多轮追问 用户：「帮我查 TKT-001。」Agent 回复处理人张三。用户：「给他发邮件提醒。」——第二轮必须能从 context 读到张三，而非重新查工单。

多轮对话与 stateStore

java 
RuntimeContext ctx = RuntimeContext.builder()
        .userId("tenant-hr")
        .sessionId("u001-sess-abc")
        .put("request_id", "req-20260610-001")
        .build();

agent.call(List.of(new UserMessage("偏好表格报告格式")), ctx).block();
Msg reply = agent.call(
        List.of(new UserMessage("刚才说的格式，导出 Q1 销售")), ctx).block();
System.out.println(reply.getTextContent());

工具观察截断（应用层）

java 
private static final int MAX_TOOL_CHARS = 2000;

static String truncateToolResult(String raw) {
    if (raw.length() <= MAX_TOOL_CHARS) return raw;
    return raw.substring(0, MAX_TOOL_CHARS)
            + "\n...[truncated " + (raw.length() - MAX_TOOL_CHARS) + " chars]";
}

基础 compaction 概念

Compaction 在 Token 接近模型窗口时触发：ReActAgent 主循环含「Compress context if needed」步骤；升级到 HarnessAgent 后可配置结构化摘要、大 tool 结果 offload 与 overflow 重试。

Compaction 决策流（概念）
flowchart TD
    Reason[进入推理] --> Check{上下文超阈值?}
    Check -->|否| LLM[调用 ChatModel]
    Check -->|是| Compact[摘要/裁剪旧 Msg]
    Compact --> LLM
    LLM --> Tool[工具执行]
    Tool --> Reason

Compaction 在内存中 mutate AgentState；call 结束时 Store 持久化的是压缩后状态。

Harness CompactionConfig（长会话预览）

java 
// HarnessAgent 完整 compaction — 详见 agentscope-harness-engineering 课
import io.agentscope.harness.agent.memory.compaction.CompactionConfig;

CompactionConfig compaction = CompactionConfig.builder()
        .triggerMessages(30)   // 消息数超 30 触发摘要
        .keepMessages(10)      // 保留最近 10 条原文
        .build();
层级组件CorpAssist 建议
ReAct 内置推理前自动压缩办公助手默认够用
Harness 摘要CompactionMiddleware长线程技术支持
Tool 结果ToolResultEviction大 JSON 查询结果
OverflowrecoverFromOverflow极端长对话兜底
Compaction 与 Session 写回的顺序？
先内存 compaction，再 end-of-call 写 Store。因此 Redis 中持久化的已是压缩后 AgentState，重启 Pod 不会恢复已丢弃的原文——原始 log 在 Harness workspace 的 *.log.jsonl 中可查。

java 
// agent-orchestration-java 风格 — ChatClient + RedisChatMemory
return builder
    .defaultAdvisors(new MessageChatMemoryAdvisor(
        chatMemory, sessionId, 10))

CorpAssist · 双栈并存 BFF 层 ChatClient 管轻量 FAQ；S3 办公自动化走 AgentScope ReActAgent + RedisAgentStateStore。勿混用同一 Redis key 前缀，避免 ChatMemory Hash 与 AgentState JSON 冲突。
同 (userId, sessionId) 并发两个 HTTP 请求会怎样？
框架会串行化同一 Session 的 call，第二个请求等待第一个完成后再加载最新 AgentState。不同 sessionId 的请求可并行。CorpAssist BFF 应避免对同一用户会话发起重叠的长任务 call。
AgentStateStore 与 Harness workspace 目录冲突吗？
不冲突。默认 JsonFileAgentStateStore 写入 ~/.agentscope/state/，workspace 内容在 .agentscope/workspace/。State 是恢复 workspace 的前提，故与 workspace 数据分离存储。

本章结论

  • Session 生命周期：RuntimeContext 选槽位，AgentStateStore 跨 call 持久化。
  • 短期记忆：AgentState.context 持有 Msg 链；生产必须外置 Store。
  • Compaction：ReAct 内置压缩 + Harness 可选完整栈。
  • ChatMemory 对照：Spring Advisor vs AgentScope Store，按场景分层。
