为什么要学本章

CorpAssist S3 项目协调场景需要「主管 + 领域专家」分工：Supervisor 理解意图，Calendar/Mail/Ticket 专家各自处理子任务。AgentScope Java 2.0 通过 Markdown 声明子 Agent 与 agent_spawn / agent_send 实现层级委派，避免主 Agent 上下文膨胀。本章对照 Python 轨 MsgHub 广播模式，帮你选对 Java 侧协作范式。

本章先记住 3 件事
  • 声明即规格：subagents/*.md 文件名即 agent_id，正文为 sys_prompt。
  • agent_key 是句柄：agent_spawn 返回完整 key，后续 agent_send 必须用它。
  • timeout_seconds=0：后台模式，结果经 system-reminder 推回父 Agent。

Markdown 声明子 Agent

AgentScope Java HarnessAgent 从 workspace/subagents/ 非递归扫描 *.md 文件加载子 Agent 规格。YAML front matter 定义 description、tools；Markdown 正文即子 Agent 的 system prompt。文件名（不含 .md）即运行时 agent_id——不要在 front matter 再写 name 覆盖。

CorpAssist · 项目启动会 用户：「订明天 10 点 A301，发邮件通知项目组，并建 IT 工单。」Supervisor HarnessAgent 依次 agent_spawn calendar-worker、mail-worker、ticket-worker，各专家在隔离 session 中执行，Supervisor 汇总 tool 返回摘要给用户。

java 
// 程序化声明（远程 Sub-agent 须用 API，Markdown 不支持 url/headers）
HarnessAgent supervisor = HarnessAgent.builder()
    .name("CorpAssistSupervisor")
    .model("dashscope:qwen-max")
    .workspace(Paths.get("./workspace"))
    .subagent(SubagentDeclaration.builder()
        .name("calendar-worker")
        .description("日历专家")
        .workspaceMode(WorkspaceMode.ISOLATED)
        .model("dashscope:qwen-plus")
        .maxIters(6)
        .tools(List.of("create_calendar_event", "query_calendar"))
        .build())
    .build();
先想 10 秒：为何 Sub-agent 用 ISOLATED workspace 而非共享父 workspace？

隔离防止子 Agent 误改父 Agent 的 AGENTS.md、plans/ 或 MEMORY.md；CorpAssist 专家只应写自己的执行摘要，不应污染主管的长期记忆。

agent_spawn / agent_send

父 Agent 在 ReAct Acting 阶段调用内置工具 agent_spawn 创建子 Agent 实例；返回的 agent_key 是后续通信的唯一句柄（不是 agent_id 或 session_id）。agent_send 向已 spawn 的子 Agent 发送跟进消息。

工具必填参数返回
agent_spawnagent_id；可选 task、label、timeout_secondsagent_key + 首任务结果（同步时）
agent_sendagent_key 或 label；message子 Agent 回复摘要
agent_list无当前活跃子 Agent 列表
java 
// 父 Agent 推理中会调用（示意 tool call 参数）
// agent_spawn agent_id="calendar-worker" task="订明天10点A301" timeout_seconds=30
// → 返回 agent_key: "agent:calendar-worker:abc-123" + 执行摘要

// agent_send agent_key="agent:calendar-worker:abc-123" message="改为11点"
// → 子 Agent 在同一隔离 session 中继续处理

// 消费 streamEvents 时，子 Agent 事件带 EventSource 元数据
supervisor.streamEvents(userMsg, RuntimeContext.builder()
        .sessionId("sess-corp-001")
        .userId("zhang")
        .build())
    .doOnNext(evt -> {
        if (evt.getSource() != null) {
            // 来自子 Agent：agentId、depth、path
        }
    })
    .subscribe();
spawn/send 反模式
  • 用 agent_id 代替 agent_key 调用 agent_send
  • 子 Agent 注册与父 Agent 相同的写权限工具
  • 无限嵌套 spawn 导致 depth 失控
推荐
  • 保存完整 agent_key 到 label 映射
  • 专家 Toolkit 最小化（领域工具 only）
  • max depth 与 max_iters 双重预算
agent_spawn 与 TaskToolsBuilder 的 Task 工具有何区别？
HarnessAgent 内置 agent_spawn 面向 workspace Markdown 声明 + 隔离 session；TaskToolsBuilder 面向 ReActAgent 编排，将 Markdown spec 编译为 Task 工具。CorpAssist S3 长任务推荐 HarnessAgent + SubagentsHook；轻量 ReAct 编排可用 Task 模式。

同步与后台模式

timeout_seconds 控制 同步与后台 Sub-agent 行为：默认 30 秒同步阻塞等待子 Agent 完成；设为 0 则立即返回 task_id，子 Agent 在后台运行，完成后通过 system-reminder 通知父 Agent，无需轮询。

同步 vs 后台 Sub-agent
sequenceDiagram
    participant P as Supervisor
    participant S as Sub-agent
    Note over P,S: 同步 timeout_seconds=30
    P->>S: agent_spawn + task
    S-->>P: 阻塞等待结果
    P->>P: 继续推理
    Note over P,S: 后台 timeout_seconds=0
    P->>S: agent_spawn task_id=T1
    S-->>P: 立即返回 task_id
    P->>P: 继续其他工作
    S-->>P: system-reminder 完成

后台模式适合耗时检索/代码审查；同步模式适合必须顺序依赖的子步骤。

call() 与 stream() 对 Sub-agent 事件有何差异？

stream() 模式下同步子 Agent 的中间事件（推理、工具调用）实时注入父 Flux，带 EventSource；call() 阻塞路径仅返回最终 ToolResult，无中间事件转发。

与 Message Hub 对照

Python AgentScope 的 MsgHub 让多个 ReActAgent 在共享上下文中互相 observe——适合专家需交叉引用结论的开放式协作。Java Sub-agent 偏层级委派 + 上下文隔离，类似 Handoff / Agent-as-Tool，trace 更短、审计边界更清晰。

能力Python MsgHubJava Sub-agent
上下文参与者共享 observe子 session 隔离
编排显式 await 顺序 + broadcastagent_spawn / agent_send 工具调用
并行Pipeline 组合timeout_seconds=0 后台任务
可观测reply_stream 事件streamEvents + EventSource
声明方式代码构造 ReActAgentworkspace/subagents/*.md
Sub-agent 不适用
  • 三专家需实时讨论、互相引用中间结论
  • 需要全员看到同一条用户消息的细粒度演变
Sub-agent 更合适
  • 子任务独立、结果汇总即可
  • 信息分级：专家不应见 HR 薪资上下文
  • CorpAssist Java 主栈 + Python 工具 MCP 分工

双栈 CorpAssist：Java 侧 Sub-agent 编排办公流；Python 侧若需 MsgHub 开放式讨论，可通过 A2A 将 Java 专家暴露为远程 Sub-agent（SubagentDeclaration.url）。避免双端各建 MsgHub 导致消息环。

CorpAssist 跨栈：MsgHub 放 Python 还是 Java？
本课 Java 轨以 Sub-agent 为主；Python 平行课用 MsgHub。生产推荐：编排层选一侧（通常 Java BFF），另一侧暴露 MCP/REST 工具。MsgHub 适合 Python 原型与 research agent；Java Sub-agent 适合 Spring Boot 生产部署与 SAA Starter 集成。
Remote Sub-agent（HTTP url）何时使用？
当专家 Agent 已独立部署为微服务（如 IT 工单服务、HR 政策 Agent）时，用 programmatic SubagentDeclaration.url + Authorization header 声明；Markdown 声明暂不支持 url/headers。

本章结论

  • Markdown 声明：subagents/*.md 文件名 = agent_id，正文 = sys_prompt。
  • spawn/send：agent_key 是运行时句柄，非 agent_id。
  • 同步/后台：timeout_seconds=0 后台 + system-reminder 回调。
  • stream 可观测：EventSource 标识子 Agent 事件来源。
  • 对照 MsgHub：隔离委派 vs 共享广播，按场景选型。

判断练习

  1. 找错：团队用 agent_send agent_id="calendar-worker" 跟进任务，子 Agent 无响应。最可能原因是？

    参考答案
