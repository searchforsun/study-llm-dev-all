AOP vs Workflow 哲学

AOP（Agent-Oriented Programming） 把 Agent 视为程序基本单元：每个 ReActAgent 拥有名称、系统提示、ChatModel、工具集与 Session，通过 Msg 传递协作。与 SAA Graph 的「状态 + 节点 + 边」不同，AOP 强调异步事件流与可插拔 Middleware，调试依赖 streamEvents() 而非图遍历日志。

AgentScope Java AOP 组件关系
flowchart TB
    User[User Msg] --> Agent[ReActAgent]
    Agent --> Model[DashScopeChatModel]
    Agent --> Tools[@Tool / MCP]
    Agent --> Session[Session + StateStore]
    Agent --> Events[streamEvents]
    Events --> UI[CorpAssist SSE]
    Agent --> Sub[Sub-agent Markdown]

AgentScope Java 2.0 以 ReActAgent 为中心，模型、工具、会话、事件均为可插拔组件。

Graph 思维硬套
  • 先画 CompiledGraph 再强行映射 ReActAgent API
  • 把 Msg 当成普通 Map 塞进 Graph State
  • 忽略 streamEvents，只等 call().block() 最终字符串
AOP 原生写法
  • 先定义 Agent 角色与 @Tool，再考虑 Sub-agent 或嵌入 Graph Tool
  • 用 Msg.builder() 显式构造用户输入
  • 消费 Typed Event 驱动 UI 与审计

AgentScope Java 2.0：最小 ReActAgent 骨架

java 
import io.agentscope.core.ReActAgent;
import io.agentscope.core.message.Msg;
import io.agentscope.core.message.MsgRole;
import io.agentscope.core.model.DashScopeChatModel;
import io.agentscope.core.runtime.RuntimeContext;

var agent = ReActAgent.builder()
    .name("CorpAssist-S3")
    .systemPrompt("你是 CorpAssist 办公助手，负责日历与工单查询。")
    .model(DashScopeChatModel.builder()
        .modelName("qwen-plus")
        .apiKey(System.getenv("DASHSCOPE_API_KEY"))
        .build())
    .maxIterations(10)
    .build();

Msg userMsg = Msg.builder()
    .role(MsgRole.USER)
    .textContent("帮我查明天 10 点的会议")
    .build();

var ctx = RuntimeContext.builder()
    .sessionId("s3-demo-001")
    .userId("zhangsan")
    .build();

Msg reply = agent.call(userMsg, ctx).block();
System.out.println(reply.getTextContent());
先想 10 秒：AOP 与 Graph Workflow 最大的工程差异是什么？

Graph 以显式节点与 State 传递为主，流程结构在编译期可见；AOP 以 Agent 内部 ReAct 循环为主，中间态通过事件流暴露，多 Agent 协作靠 Sub-agent 声明而非画边。

AgentScope Java 还需要自己画 Graph 吗？
单 Agent ReAct 不需要；多 Agent 可用 Markdown 声明 Sub-agent。仅当嵌入既有 SAA Graph 审批子流程时，Graph 仍在外部维护并由 AgentScope Tool 调用。
AOP 是否排斥确定性工作流？
不排斥。Middleware 可注入固定步骤；AOP 强调 Agent 作为组合单元，而非禁止预定义流程。CorpAssist 审批链可 Graph + HITL 节点实现，外层由 ReActAgent 路由。

ReActAgent vs Graph 节点

前置课中的 SAA Graph 以State + Node + Edge描述流程：每个节点是独立函数，边决定流转。AgentScope Java 的 ReActAgent 将 agent/tools 循环封装在 Agent 内部，对外只暴露 call() 与 streamEvents()，工具通过 @Tool 或 MCP 注册，无需手写条件边回环。

选型决策树

前置课中的 SAA Graph 适合步骤固定、需 checkpoint 恢复、HITL interrupt 的审批流。AgentScope Java 的 ReActAgent + Sub-agent 覆盖串并行协作，且内置 MCP、事件流与 Permission 三态，更适合多 Agent + 工具生态 + DashScope 原生的 CorpAssist 栈。

框架选型决策流
flowchart TD
    Q1[需要精细 Graph checkpoint 与 HITL interrupt？] -->|强需求| Graph[SAA Graph 节点]
    Q1 -->|一般| Q2[多 Agent Sub-agent 显式协作？]
    Q2 -->|是| AS[AgentScope Java Sub-agent]
    Q2 -->|否| Q3[需 streamEvents + MCP 原生？]
    Q3 -->|是| AS
    Q3 -->|否| Graph
    Graph --> Migrate[已有 Graph 资产可渐进迁移]
    AS --> CorpAssist[CorpAssist S3 默认栈]

CorpAssist 场景适配

CorpAssist 北星能力 NS1（可靠问答）与 NS3（成本可控）在 S3 办公自动化中同时出现：简单日历查询适合单 ReActAgent + 少步工具；跨系统报告生成适合 Sub-agent + 事件流。本章为后续 单 Agent：CorpAssist 办公助手 与 Sub-agent 多 Agent 协作 打基础。

  • 范式：单 ReActAgent + DashScope + 2～3 个 @Tool
  • 框架：AgentScope Java 即可，不必上 Graph
  • 预算：maxIterations(8)，控制 NS3 成本
  • 范式：主管 Agent + 数据/写作 Sub-agent
  • 框架：Sub-agent Markdown + streamEvents
  • 质量：关键步骤 HITL，见 Events + Permissions 与 HITL
已有 SAA Graph 的 CorpAssist 代码必须全部重写吗？
不必。可保留 Graph 子图处理确定性 ETL，外层用 ReActAgent 调用该子图作为 @Tool。渐进迁移比一次性替换风险更低。
AgentScope Java 与 SAA ReactAgent 能混用吗？
可以。通过 Spring AI Alibaba 工程实践spring-ai-alibaba-engineering 的 spring-ai-alibaba-starter-agentscope 将 ReActAgent 包装为 Graph 节点，见 spring-ai-alibaba-starter-agentscope 集成。

本章结论

  • AOP vs Workflow：AgentScope Java 以 ReActAgent + Msg + 事件为一等公民；SAA Graph 以 State + 节点 + 边描述确定性流程。
  • ReActAgent vs Graph 节点：内置 ReAct 循环对外暴露 call/streamEvents；Graph 需手写 agent/tools 节点与条件边。
  • 选型决策树：Graph 擅 checkpoint 与 HITL interrupt；AgentScope Java 擅多 Agent、MCP 与 streamEvents。
  • CorpAssist 适配：S3 轻量任务单 ReActAgent，复杂报告用 Sub-agent，可与既有 Graph 子流程共存。

Python 侧对照：平行课 AgentScope 2.0 与 LangGraph 对照agentscope-agent-engineering 以 Python Agent + UserMsg + Message Hub 对照 LangGraph；Java 课以 ReActAgent.builder() + Msg + Sub-agent 对照 SAA Graph。双栈联调时保持 Msg content 与事件 trace_id 一致即可，不必统一框架入口类名。
