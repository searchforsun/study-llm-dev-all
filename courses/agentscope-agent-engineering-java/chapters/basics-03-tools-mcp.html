注解驱动工具注册

@Tool 注解 将 Java 方法暴露为 Agent 可调用工具。框架自动提取方法签名与 @ToolParam 描述，生成 OpenAI 兼容的 Function Calling Schema。与 SAA Starter 的 @Tool 类似，但 AgentScope Java 的工具直接注册到 ReActAgent.builder().tools()，无需 Spring ChatClient 中转。

工具注册与调用链
flowchart TB
    Agent[ReActAgent] --> Registry[Tool Registry]
    Registry --> Local[@Tool 本地方法]
    Registry --> MCP[MCP 远程工具]
    Allow[tools.json allowlist] --> Registry
    Perm[Permission Engine] --> Local
    Perm --> MCP

allowlist 在 Workspace 层过滤可见工具；Permission 在执行前判定三态。

注册 CorpAssist 工单工具

java 
import io.agentscope.core.tool.annotation.Tool;
import io.agentscope.core.tool.annotation.ToolParam;

public class TicketTools {

    @Tool(name = "createTicket",
          description = "在 CorpAssist 创建工单。title: 标题；priority: low/normal/high")
    public String createTicket(
            @ToolParam(description = "工单标题") String title,
            @ToolParam(description = "优先级，默认 normal") String priority) {
        String ticketId = String.format("T-%04d", Math.abs(title.hashCode() % 10000));
        return "工单 " + ticketId + " 已创建，优先级 " + priority;
    }

    @Tool(name = "queryTicket", description = "根据工单编号查询状态")
    public String queryTicket(
            @ToolParam(description = "工单编号，格式 T-XXXX") String ticketId) {
        return ticketId + " 状态：处理中";
    }
}

// 注册到 Agent
ReActAgent agent = ReActAgent.builder()
    .tools(new TicketTools())
    .build();
先想 10 秒：@ToolParam description 为什么必须写业务说明？

DashScope 等模型靠 schema + 描述选择工具与填参；缺失描述会导致错误调用或幻觉参数，是 CorpAssist 工单类 bug 的常见根因。

工具描述模糊
  • @Tool(description = "查询信息") — LLM 误调用
  • 参数无 @ToolParam 说明 — 模型填错 priority 枚举
工具描述清晰
  • description 含场景与示例，20～50 字
  • @ToolParam 标明格式与枚举值
@Tool 要素说明CorpAssist 推荐
nameLLM 调用标识动词+名词，如 queryTicket
description模型判断何时调用含场景，如「客服跟进工单」
@ToolParam参数 schema 描述标明格式、枚举、是否必填

并发/串行调度

AgentScope Java 2.0 支持通过工具属性声明调度策略：batch（批量合并）、sequential（严格串行）、concurrent（并行执行）。CorpAssist 场景中，只读查询工具可并发以缩短延迟；有副作用的写操作必须串行并配合 Permission approve。

调度属性示例

java 
import io.agentscope.core.tool.annotation.ToolExecutionMode;

public class CalendarTools {

    @Tool(name = "queryCalendar", description = "查询指定日期会议安排（只读）")
    @ToolExecutionMode(executionMode = ExecutionMode.CONCURRENT)
    public String queryCalendar(@ToolParam(description = "ISO 日期") String date) {
        return date + " 10:00 产品评审会";
    }

    @Tool(name = "sendEmail", description = "发送邮件（有副作用，需审批）")
    @ToolExecutionMode(executionMode = ExecutionMode.SEQUENTIAL)
    public String sendEmail(
            @ToolParam(description = "收件人邮箱") String to,
            @ToolParam(description = "邮件主题") String subject) {
        return "邮件已发送至 " + to;
    }
}
调度模式适用CorpAssist 示例
CONCURRENT只读、无依赖并行查日历 + 查工单
SEQUENTIAL有副作用、有顺序依赖先查工单再发邮件
BATCH同类调用可合并批量查多个 ticketId
并发工具会导致 ReAct 循环乱序吗？
不会。框架在 Acting 阶段等待所有并发工具完成后，再将合并的 ToolResultBlock 写回 Session，模型下一轮看到的是完整观察结果。

MCP Server 接入

MCP（Model Context Protocol） 让外部服务以标准方式暴露 tools/resources。AgentScope Java 内置 MCP Client，可将远程 Server 工具挂载到 ReActAgent，避免为每个 SaaS 写适配层。

接入 MCP Server

java 
import io.agentscope.core.mcp.McpClient;
import io.agentscope.core.mcp.McpToolAdapter;

McpClient kbMcp = McpClient.builder()
    .serverUrl(System.getenv("KB_MCP_URL"))
    .headers(Map.of("Authorization", "Bearer " + System.getenv("CORPASSIST_MCP_TOKEN")))
    .timeout(Duration.ofSeconds(45))
    .build();

List<Object> remoteTools = kbMcp.listTools().block()
    .stream()
    .map(McpToolAdapter::fromMcpTool)
    .toList();

ReActAgent agent = ReActAgent.builder()
    .tools(merge(localTools, remoteTools))
    .build();
集成方式适用CorpAssist 示例
@Tool 本地内部微服务已有 Java SDK工单 API
MCP第三方或跨语言服务Confluence 知识库 MCP
Graph Tool确定性子流程审批 Graph 封装为 Tool

MCP 配置文件（mcp-servers.yaml 示例）

yaml 
servers:
  corpassist-kb:
    url: http://localhost:8765/mcp
    headers:
      Authorization: "Bearer ${CORPASSIST_MCP_TOKEN}"
    timeout: 45s
  corpassist-calendar:
    url: http://calendar-svc:8080/mcp
    timeout: 30s
  • MCP list_tools 返回的工具名可能与本地 @Tool 重名，注册前加前缀如 kb_。
  • CorpAssist 知识库 MCP 应只暴露 read/search，不暴露 write（权限最小化）。
  • 前置课 MCP 概念见 Agent 编排（Java）agent-orchestration-java。

tools.json allowlist

AgentScope Java Workspace 下的 tools.json 是工具白名单真源：声明 Agent 可见的工具名、MCP Server 来源与 Permission 默认策略。生产环境禁止「注册即可见」——必须先入 allowlist 再绑定 Permission 规则。

allowlist 与 Permission 联动
flowchart LR
    Reg[工具注册表] --> Allow[tools.json]
    Allow --> Perm[Permission Rules]
    Perm --> Allow2[allow]
    Perm --> Approve[approve]
    Perm --> Deny[deny]
    Approve --> HITL[ApprovalRequiredEvent]

未列入 allowlist 的工具对模型不可见，列入后由 Permission 决定执行三态。

workspace/tools.json 示例

json 
{
  "allowlist": [
    {
      "name": "queryCalendar",
      "source": "local",
      "permission": "allow"
    },
    {
      "name": "queryTicket",
      "source": "local",
      "permission": "allow"
    },
    {
      "name": "kb_search",
      "source": "mcp:corpassist-kb",
      "permission": "allow"
    },
    {
      "name": "sendEmail",
      "source": "local",
      "permission": "approve"
    },
    {
      "name": "deleteTicket",
      "source": "local",
      "permission": "deny"
    }
  ]
}
反模式
  • 注册所有 @Tool 不设 allowlist
  • MCP Token 写进 Git 仓库
  • 高危工具 permission 设为 allow
推荐
  • tools.json 版本化管理，CI 校验白名单
  • 环境变量 + Secret 注入 MCP headers
  • 写操作默认 approve，删除类 deny

加载 allowlist 与 Agent 绑定

java 
Path workspace = Paths.get(".agentscope/workspace");
ToolAllowlist allowlist = ToolAllowlist.load(workspace.resolve("tools.json"));

ReActAgent agent = ReActAgent.builder()
    .workspace(workspace)
    .tools(allowlist.filterRegistered(allTools))
    .permissionRules(allowlist.toPermissionRules())
    .build();
CorpAssist · S3 工具治理 运维在 tools.json 新增 kb_search 后 staging 验证，再推广到 prod。若模型尝试调用未列入 allowlist 的 deleteTicket，Permission 直接 deny 并返回结构化错误，审计日志记录拒绝事件。

本章结论

  • 注解驱动注册：@Tool + @ToolParam 生成 Schema，description 决定模型选型质量。
  • 并发/串行调度：只读 concurrent，写操作 sequential + approve。
  • MCP Server：标准协议接入外部系统，headers 从环境变量注入。
  • tools.json allowlist：Workspace 白名单 + Permission 三态统一工具治理。

Python 侧对照：平行课 Tool / Skill 与 MCPagentscope-agent-engineering 用 @tool + Toolkit + MCPClient；Java 课用 @Tool 注解 + McpClient + tools.json。双栈联调时保持工具 name 与 allowlist 条目一致，MCP Server URL 可不同但 tool schema 应对齐。
