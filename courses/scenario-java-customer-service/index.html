Spring Cloud Gateway + Resilience4j 的三层限流架构

智能客服系统的流量模式与普通 Web 应用有很大不同：消息的产生是突发性的（用户可能同时涌来）、每条消息的处理时间不确定（LLM 推理 1-3s 加上工具调用）、流式连接会长时间保持。因此需要精细化的限流策略来保护后端服务。CorpAssist 的限流架构分为三层。第一层是 IP 级别限流，基于 Spring Cloud Gateway 的 RequestRateLimiter 过滤器 + Redis 实现，限制单个 IP 每秒最多 10 次请求，防止恶意刷接口。第二层是用户级别限流，在 Gateway 层解析 JWT Token 中的 userId 后，对每个用户每分钟最多 30 次对话请求进行限流（主动发消息而非接收 SSE）。第三层是后端连接级别限流，通过 Resilience4j Bulkhead 限制同时进行中的 LLM API 调用不超过 20 个，超过的请求在 Gateway 层直接返回 429 状态码和「系统繁忙，请稍后再试」的兜底话术，而不是在后端排队等待导致所有请求都变慢。

限流策略在实践中需要持续调优。上线初期我们将 IP 级别限流设为 5 req/s，但发现有些企业用户的出口 IP 是共享的（整栋办公楼共享一个公网 IP），5 req/s 的限制会误伤正常用户。解决方案是引入「动态限流」：基于历史流量模式自动调整阈值。在低峰期（0:00-8:00）限流阈值设为 5 req/s，高峰期（10:00-12:00, 14:00-17:00）动态提升到 15 req/s，并结合 Token Bucket 算法允许短时间突发（burstCapacity = 2x replenishRate）。另外重要的策略是「限流失效降级」——当限流触发时，不是简单的返回 429，而是告知用户「当前咨询量较大，您的消息已入队，预计等待 1 分钟后为您回复」，同时将消息放入 Redis 队列由 Agent 异步处理。此策略将限流后的用户放弃率从 45% 降到 12%。

// 动态限流 Key Resolver 和限流降级处理
@Component
public class DynamicRateLimiterConfig {

    @Bean
    public KeyResolver ipKeyResolver() {
        return exchange -> {
            String ip = exchange.getRequest().getHeaders()
                    .getFirst("X-Forwarded-For");
            if (ip == null) {
                ip = exchange.getRequest().getRemoteAddress().getAddress().getHostAddress();
            }
            return Mono.just(ip);
        };
    }

    @Bean
    public KeyResolver userKeyResolver() {
        return exchange -> {
            // 从 JWT Token 中提取 userId
            String token = exchange.getRequest().getHeaders()
                    .getFirst("Authorization");
            if (token != null && token.startsWith("Bearer ")) {
                String jwtToken = token.substring(7);
                // 解析 JWT 获取 userId（简化版，实际用 JwtDecoder）
                String userId = extractUserIdFromToken(jwtToken);
                return Mono.justOrEmpty(userId);
            }
            return Mono.just("anonymous");
        };
    }

    // 限流响应降级处理
    @Bean
    public Map rateLimitFallback() {
        return Map.of(
            "status", 429,
            "code", "RATE_LIMITED",

            "retryAfter", 60
        );

SSE 流式连接的超时与重连配置

SSE 长连接的可靠性是智能客服用户体验的关键。如果连接频繁断开，用户会看到「正在输入中...」然后突然停止——这会让用户感到困惑，不知道是 Agent 在思考还是连接断了。CorpAssist 对 SSE 连接做了三层超时/重连保护。客户端层：前端 EventSource 配置了自动重连（reconnectionTime=3000ms），当连接断开时 3 秒后自动重连，并发送新的会话请求带上原来的 sessionId，后端通过 AgentStateManager 恢复执行状态。反向代理层（Nginx）：配置 proxy_read_timeout=120s 和 proxy_buffering=off，确保 Nginx 不会因为长时间无数据而切断 SSE 流。服务端层：Spring WebFlux 的 SSE 超时设置在 ServerSentEventHttpMessageWriter 中——连接空闲超过 30 秒发送心跳事件，总连接时间超过 60 秒强制断开（避免用户忘记关闭页面导致资源泄漏）。

还有一个容易被忽视的问题：SSE 连接在浏览器打开新标签页或页面刷新时会触发「连接抢占」——新连接建立后旧连接没有及时关闭，导致同一个 sessionId 有两个活跃的 SSE 连接，Agent 的输出流被发送到两个地方（其中一个可能已经不再展示）。解决方案是在 Redis 中维护每个 sessionId 的连接计数器，当新连接建立时，检查是否有旧连接存在——如果有，通过 event 类型为「replaced」的消息通知旧连接关闭，然后旧连接执行 doFinally 清理资源。这个机制上线后，SSE 相关的用户投诉从月均 15 起降到了 1 起以下。

// SSE 连接管理和重连配置
@Configuration
public class SseConnectionConfig {

    @Bean
    public SseConnectionManager sseConnectionManager(StringRedisTemplate redisTemplate) {
        return new SseConnectionManager(redisTemplate);
    }
}

// SSE 连接管理器 — 处理连接抢占和重连
public class SseConnectionManager {

    private final StringRedisTemplate redisTemplate;
    private static final String CONNECTION_KEY = "sse:conn:";
    private static final Duration CONNECTION_TTL = Duration.ofSeconds(90);

    // 连接唯一标识
    public record ConnectionInfo(
        String sessionId,
        String connectionId,
        Instant createdAt
    ) {}

    public SseConnectionManager(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    // 注册新连接，返回是否需要关闭旧连接
    public boolean registerConnection(String sessionId, String connectionId) {
        String key = CONNECTION_KEY + sessionId;
        String previousConnectionId = redisTemplate.opsForValue().get(key);
        redisTemplate.opsForValue().set(key, connectionId, CONNECTION_TTL);
        return previousConnectionId != null && !previousConnectionId.equals(connectionId);
    }

    // 关闭连接时清理
    public void unregisterConnection(String sessionId) {
        redisTemplate.delete(CONNECTION_KEY + sessionId);
    }

    // SSE 心跳 Flux — 每 15 秒发送一次
    public Flux> heartbeat(String sessionId) {
        return Flux.interval(Duration.ofSeconds(15))
                .map(i -> ServerSentEvent.builder("ping")
                        .event("heartbeat")
                        .build())
                .timeout(Duration.ofSeconds(70))
                .onErrorResume(e -> Flux.just(
                        ServerSentEvent.builder("timeout")
                                .event("connection-closed")
                                .build()));
    }
}

// 在 Controller 中使用连接管理
@PostMapping("/api/cs/chat/stream")
public Flux> streamChat(
        @RequestBody ChatRequest request,
        @RequestHeader("X-Connection-Id") String connectionId) {

    String sessionId = request.sessionId();

    // 检查是否有旧连接需要关闭
    boolean hasPrevious = connectionManager.registerConnection(sessionId, connectionId);
    if (hasPrevious) {
        // 发送通知关闭旧连接（通过另一个 Channel）
        redisTemplate.convertAndSend("sse:close:" + sessionId, connectionId);
    }

    return Flux.merge(
            agentService.processMessage(sessionId, request.userId(), request.message()),
            connectionManager.heartbeat(sessionId)
    ).doFinally(signal -> connectionManager.unregisterConnection(sessionId));
}

JWT + Spring Security OAuth2 鉴权流程

企业级智能客服需要统一身份认证（SSO）支持：用户登录企业 OA 系统后，带有 JWT Token 访问客服页面，无需重新登录。CorpAssist 使用 Spring Security OAuth2 Resource Server 配置 JWT 鉴权。认证流程：用户在企业 IDP（如 Keycloak 或 Azure AD）登录，获取 JWT Token，Token 包含 userId、displayName、roles、sessionId 等 Claims。用户在客服前端发起 WebSocket 或 SSE 连接时，在请求头中携带 Authorization: Bearer 。Gateway 层通过 Spring Security 的 JwtAuthenticationToken 过滤器解析 Token，提取 Claims 并设置到 SecurityContext。下游服务通过 @AuthenticationPrincipal JwtAuthenticationToken 注解获取用户信息，无需再次解析 Token。Gateway 对 /api/cs/** 路径进行鉴权——Token 过期或无效的请求返回 401 状态码。

Token 的生命周期和刷新策略直接影响用户体验。JWT Token 的有效期一般为 1 小时（配合 Refresh Token 7 天有效）。客服 SSE 长连接可能持续 10-30 分钟，在 Token 过期前需要自动刷新。CorpAssist 的前端 SDK 在 Token 过期前 5 分钟自动使用 Refresh Token 刷新，并将新 Token 通过 SSE 的 auth 事件发送给后端更新连接认证。如果 Refresh Token 也过期了，连接断开返回 401，前端引导用户重新登录。Token 刷新流程的端到端耗时约 200ms，对用户体验几乎无影响。此外，服务端 Token 缓存可以减少每次请求的 JWT 解析开销——在 Gateway 层使用 Redis 缓存 Token Claims 30 分钟（小于 Token 有效期 1 小时），避免每次请求都调用解密算法。

// Spring Security OAuth2 资源服务器配置
@Configuration
@EnableWebFluxSecurity
public class SecurityConfig {

    @Bean
    public SecurityWebFilterChain securityWebFilterChain(ServerHttpSecurity http) {
        return http
                .authorizeExchange(exchanges -> exchanges
                        // 客服 API 需要认证
                        .pathMatchers("/api/cs/**").authenticated()
                        // 健康检查和内部接口免认证
                        .pathMatchers("/api/cs/health", "/api/cs/internal/**").permitAll()
                        // 其他所有请求
                        .anyExchange().authenticated()
                )
                .oauth2ResourceServer(oauth2 -> oauth2
                        .jwt(jwt -> jwt
                                .jwtDecoder(jwtDecoder())
                                .jwtAuthenticationConverter(jwtAuthConverter())
                        )
                )
                .csrf(ServerHttpSecurity.CsrfSpec::disable)
                .build();
    }

    @Bean
    public ReactiveJwtDecoder jwtDecoder() {
        // 使用 JWKS 端点自动获取公钥
        return ReactiveJwtDecoders
                .fromIssuerLocation("https://idp.company.com/auth/realms/corpassist");
    }

    @Bean
    public Converter> jwtAuthConverter() {
        return new ReactiveJwtAuthenticationConverterAdapter(
                new JwtGrantedAuthoritiesConverter());
    }
}

// JWT Token 中提取用户信息的拦截器
@Component
public class UserContextFilter implements WebFilter {

    @Override
    public Mono filter(ServerWebExchange exchange, WebFilterChain chain) {
        return exchange.getPrincipal()
                .cast(JwtAuthenticationToken.class)
                .map(auth -> {
                    Jwt jwt = auth.getToken();
                    String userId = jwt.getClaimAsString("sub");
                    String displayName = jwt.getClaimAsString("name");
                    String sessionId = jwt.getClaimAsString("session_id");

                    // 设置请求头透传到下游服务
                    ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
                            .header("X-User-Id", userId)
                            .header("X-User-Name", displayName)
                            .header("X-Session-Id", sessionId)
                            .build();
                    return exchange.mutate().request(mutatedRequest).build();
                })
                .defaultIfEmpty(exchange)
                .flatMap(chain::filter);
    }
}

7×24 运维值班手册与告警规则

智能客服系统上线后需要 7×24 小时运维保障。CorpAssist 制定了值班手册（Runbook），涵盖 5 类关键告警和对应的恢复 SOP。第一类告警是延迟告警：P99 端到端延迟 > 8s（Critical）需要在 5 分钟内响应。第一步检查 LLM API 的响应时间趋势（如果 API 本身变慢需要联系模型提供商），第二步检查 Gateway 连接池是否满（如果并发连接数接近上限需要扩容），第三步检查 Redis 操作是否变慢（如果 Redis 的 P99 延迟 > 50ms 需要排查 Redis 集群）。第二类告警是解决率告警：解决率 < 60%（Warning）触发，通常需要检查意图分类器准确率是否下降或知识库是否有更新。第三类告警是错误率告警：LLM API 返回非 200 状态码的比例 > 10%（Critical），自动触发熔断并切换到 L3 降级模式。第四类告警是队列深度告警：转人工排队队列深度 > 100（Warning），说明坐席处理不过来，需要通知运营团队动员更多坐席上线。第五类告警是熔断告警：CircuitBreaker 进入 OPEN 状态（Critical），需要立即排查 LLM API 的连通性。

值班手册除了告警响应步骤外，还包含了日常巡检清单和版本发布 checklist。日常巡检（每 4 小时）：检查 Dashboard 上的 P99 延迟、解决率、转接率、LLM API 健康状态 4 个核心指标是否正常。版本发布 checklist：A/B 实验是否验证了 24 小时？回滚方案是否就绪？是否通知了运营团队？灰度发布比例是否在控制范围内（先 10% -> 24h 后 50% -> 48h 后 100%）？这些规范确保了每次生产变更都有据可查、有备无患。以下表格汇总了 CorpAssist 的关键告警规则。

本章结论

  • 三层限流保护架构：IP 级（10 req/s）+ 用户级（30 req/min）+ 连接池级（20 并发），配合动态限流（高峰期自动提升阈值）和限流失效降级（入队异步处理），用户放弃率从 45% 降到 12%。
  • SSE 连接管理三重超时：心跳 15s + 空闲超时 30s + 总超时 60s，Nginx proxy_read_timeout=120s，连接抢占通过 Redis 连接计数器处理，月均连接投诉从 15 起到 < 1 起。

面试可能问

Q1: 智能客服系统的三层限流怎么设计？为什么需要三层？

三层限流对应三种流量来源。第一层 IP 级别限流保护系统免遭恶意攻击和误刷——使用 RequestRateLimiter + Redis 按 IP 地址限流（10 req/s，burst=20），一台机器短时间内发送大量请求明显是非正常行为。第二层用户级别限流保护下游服务免遭单个用户过度使用——从 JWT Token 提取 userId 做限流（30 req/min，burst=50），防止某个用户因为前端 Bug 或异常行为产生大量请求。第三层连接池级别限流保护 LLM API 免遭过载——使用 Resilience4j Bulkhead 限制同时进行中的 LLM 调用不超过 20 个，这是最关键的一层。为什么需要三层？因为每一层的防护目标不同：IP 层防攻击、用户层防滥用、连接池层防雪崩。如果只有连接池层，LLM API 确实不会被打爆，但请求会在连接池排队等待，P99 延迟会飙升到 30s+，用户体验极差。有了前两层，大部分异常流量在 Gateway 层就被拒绝了（返回 429 或入队异步处理），不会进入连接池排队。三层配合实现了「上游快速失败、下游平稳运行」的目标。

Q2: SSE 长连接在生产环境有什么坑？怎么解决连接断开问题？

SSE 长连接有三大痛点。第一是反向代理超时——Nginx 默认的 proxy_read_timeout=60s 会导致 60 秒无数据时自动断开连接。解决方案是调大到 120s，并让服务端每 15 秒发送心跳保持连接活跃。第二是服务端进程重启——滚动更新或 Pod 重启时 SSE 连接全部断开。解决方案是在客户端配置自动重连（EventSource reconnectionTime=3000ms），并在重连时带上 sessionId，后端通过 AgentStateManager 恢复 Agent 的执行状态继续回复。第三是连接抢占——用户打开新标签页或刷新页面时，旧连接没有正常关闭，导致同一个 sessionId 有两个活跃连接。解决方案是用 Redis 的连接计数器：新连接建立时检查同一个 sessionId 是否有更旧的连接，如果有则发送「replaced」事件通知旧连接关闭。这第三种情况最难排查——你会看到用户收到重复的回复。我们花了 2 周才定位到根因，上线连接计数器后彻底解决。

Q3: 运维值班遇到 P99 延迟从 2s 飙升到 15s，排查步骤是什么？

排查步骤按分层隔离的方式进行。第一步确认影响范围——查看 Dashboard 的 P99 延迟趋势图，确认是整体上涨还是某个意图或用户的延迟高。第二步检查 Gateway 层——查看 Gateway 的 upstream_response_time 指标：如果转发延迟正常（< 10ms）但总延迟高，说明问题在下游服务。第三步检查 Conversation Service——检查 JVM GC 日志（GC 停顿是否频繁）、Redis 连接池利用率（是否达到上限）、线程池状态（是否有死锁或线程阻塞）。第四步检查 LLM API——查看 LLM API 的响应时间趋势图：如果是 API 本身变慢（从 1.5s 到 5s+），需要联系模型提供商排查。第五步检查工具调用——查看工具 API 的响应时间：如果 Tool Service 的 P99 从 500ms 飙升到 8s，问题可能在业务系统。通常在第三步或第四步就能找到根因。最常发生的情况是 LLM API 的某个实例出现故障导致局部延迟升高，或者是一次知识库更新导致的 Embedding 计算耗时增加。快速缓解措施：如果确认是 LLM API 问题，立即触发熔断切换到 L1 降级模式（规则引擎回复），保住基本的回复能力，等延迟恢复后再切回正常模式。

判断练习

场景题：你的同事说「限流阈值设得太低了，用户经常反馈被限流，把阈值调到 100 req/s 不就行了」。你同意吗？

不同意。直接提高阈值是治标不治本的短期做法。限流的目的不是「防止用户正常使用」，而是「在资源有限的情况下保证服务的公平性和稳定性」。如果只是把阈值从 10 调到 100，但单个 IP 确实有合法的高并发场景（如企业出口共享 IP），那需要区分对待而不是整体提高。正确做法是分析被限流的用户数据：有多少是真正的误杀（合法用户被限流）？有多少是正常限流（确实超过了合理使用范围）？如果是误杀，可以调整限流维度——从 IP 级别改为 IP + Cookie 组合键，或者只对非登录用户限流（已登录用户走用户级别限流）。如果是正常限流，说明需要扩容而非提高阈值——容量规划和限流阈值是匹配的，提高不扩容意味着后端会过载。CorpAssist 的实践表明，80% 的限流触发来自 5% 的 IP（主要是爬虫和测试工具），对这部分 IP 应该单独拉黑或限流到更低阈值，而不是为 5% 的异常提高整体阈值连累 95% 的正常用户。

场景题：生产环境 P99 延迟在平稳运行 2 个月后突然从 3s 升到 12s，排查发现 LLM API 的 P99 响应时间没有明显变化（还是 1.5s），Gateway 转发正常，Redis 正常。问题可能出在哪？

LLM API 正常但总体延迟飙升 4 倍，说明延迟增加在下游的 Java 服务内部，而不是外部依赖。需要排查以下几方面：第一是 Agent 的步骤数是否增加了——如果 System Prompt 或工具注册的变更导致 Agent 在推理时选择了更多的步骤（例如从平均 2 步变成 4 步），总延迟会翻倍。查看 Agent 执行的 step count 指标。第二是工具调用的次数是否增加了——如果 Agent 决策逻辑变化导致它更频繁地调用工具（从每次对话 1.5 次变成 3 次），每次工具调用 + LLM 反馈会额外增加 1-2s。第三是 ChatMemory 的长度是否变大了——如果会话历史没有被正确压缩（摘要压缩触发的阈值从 10 轮被改成了 20 轮），每次 Prompt 的 Token 数翻倍会导致 LLM 推理时间线性增加。第四是 JVM 配置是否有变化——检查 GC 日志和堆内存使用率：如果内存泄漏导致频繁 Full GC，GC 停顿时间会大幅增加 P99。最常发生的是第四种情况——JVM 参数被意外修改或者内存泄漏。建议先用 jstat 和 GC 日志分析工具查看 GC 停顿时间。

Token 成本分月归因分析（Prompt/对话/工具调用）

Token 成本是智能客服运营的主要变量成本。CorpAssist 的月 Token 费用约为 Y19,800（基于日均 3,000 对话、平均 800 Tokens/次、Y0.15/1K 单价），年化 Y237,600。理解 Token 消耗的构成是成本优化的前提。我们对 Token 消耗做了分月归因分析：System Prompt（系统角色定义 + 格式约束 + Few-Shot 示例）固定消耗约 300 Tokens/次，占比 38%；对话历史（从 ChatMemory 读取的上下文）每轮新增约 250 Tokens，占比 22%；工具调用 Schema（Function Calling 的 JSON 注册）每次工具调用约 150 Tokens，加上工具结果回注约 50 Tokens，占比 25%；回复生成（LLM 输出的 Token）平均 150 Tokens，占比 15%。值得注意的是，System Prompt 的 38% 是固定开销——无论对话多长、工具调用多少次，这部分都要消耗。因此优化重点应放在可变成本上。

基于归因分析，CorpAssist 实施了三个成本优化措施。第一是 System Prompt 精简——通过 A/B 实验逐步删除不必要的 Few-Shot 示例和最不常用的工具描述，将固定消耗从 300 Tokens 降到 220 Tokens（降幅 27%），月节省 Y2,376。第二是上下文摘要压缩——实现每 10 轮压缩机制后将对话历史的 Token 消耗减少 68%（从历史上下文平均 4K Tokens 降到 1.3K Tokens），月节省 Y3,326。第三是工具调用缓存——高频工具调用（如订单查询）添加 30 秒本地缓存，相同的查询参数在缓存期间直接返回结果不调用 LLM，减少约 8% 的工具调用 Token，月节省 Y594。三项措施叠加使 Token 成本从月均 Y19,800 降到 Y12,480（降幅 37%），年节省 Y87,840。而 CSAT 和解决率在优化前后无显著差异（A/B 实验验证通过）。

STAR 面试模板：Situation-Task-Action-Result

面试官问「请讲一个你做过最有成就感的项目」时，使用 STAR 模板回答可以让你的表达结构清晰、数据详实、逻辑严谨。STAR 代表 Situation（项目背景和上下文）、Task（需要解决的核心问题）、Action（你采取的具体措施和技术方案）、Result（最终的业务结果和量化数据）。CorpAssist 智能客服项目的完整 STAR 故事如下。Situation——某电商平台的客服系统解决率仅 70%，意味着 30% 的用户需要转人工。月均人工客服成本 Y330,000，CSAT 评分 3.7 低于行业基准 4.0。用户投诉中 40% 都与客服体验有关，管理层要求 6 个月内改善。Task——需要在 6 个月内将客服解决率从 70% 提升到 85%，同时控制转人工率不超过 25%、CSAT 不低于 4.0。不能简单靠增加人力——必须通过技术优化实现可规模化的提升。Action——分三个阶段：第一阶段（1-2 月）重构意图分类模块，引入 Spring AI Structured Output + 规则引擎双引擎架构；第二阶段（3-4 月）扩充政策知识库从 15 篇到 48 篇，引入二次确认机制；第三阶段（5-6 月）从 ChatClient 升级到 ReAct Agent，优化降级策略。每个阶段都经过 7 天 A/B 实验验证。Result——6 个月后解决率稳定在 85%（提升 15pp）、转人工率从 35% 降到 22%、CSAT 从 3.7 提升到 4.2、年节省人力成本 Y190 万。项目获得公司年度技术创新奖。

Q2: 在优化解决率的过程中遇到的最大困难是什么？你怎么解决的？

最大的困难是在第二阶段——知识库从 15 篇扩充到 48 篇后，解决率不仅没有提升，反而下降了 2 个百分点。当时分析发现根因是「知识噪声」——检索出来的文档片段太多了，LLM 在高相似度的多个文档之间无所适从，经常引用不相关的政策。比如用户问「退货运费谁出」，检索出来 3 个文档——运费政策、退货流程、保修政策——LLM 有时候会引用错误的政策。解决方案是在检索环节增加了三项优化：第一是 Hybrid Search（稠密向量 + BM25 关键词），不再是单纯的语义相似度；第二是相似度阈值（0.75 以下的不返回），避免低质量片段的干扰；第三是文档标题加权（标题权重 2x），让标题匹配的文档优先。同时，在 System Prompt 中增加了「只回答检索文档中明确提到的信息，不要跨文档推断」的指令。优化后解决率不仅回到 76%，还继续提升到 81%。这个教训让我意识到「更多的数据需要更好的检索策略」——单纯增加知识库数量而不优化检索策略，效果可能适得其反。后来我们把这个经验写成了团队的知识管理规范：每次知识库变更必须同步评估检索策略是否需要调整。

Q3: Token 成本怎么优化？优化了多少？对用户体验有影响吗？

Token 成本优化我们做了三件事，Total 节省 37%。第一件是最有效的——上下文摘要压缩。原来 20 轮对话的上下文约 4K Tokens，实现每 10 轮压缩机制后降到 1.28K（节省 68%），月节省 Y3,326。这个优化的效果最好，而且对用户体验没有任何负面影响——因为最近 2 轮完整对话保留了，指代消解和上下文连贯性不受影响。第二件是 System Prompt 精简——去掉了那些在实践中证明不必要的 Few-Shot 示例和工具描述，从 300 Tokens 降到 220（降幅 27%），月节省 Y2,376。这个需要谨慎做——每删掉一个 Few-Shot 示例都需要在坏例库回归测试中验证准确率没有下降。第三件是高频工具查询缓存——同样的订单号 30 秒内的重复查询直接返回缓存结果，减少约 8% 的工具调用 Token，月节省 Y594。但需要注意缓存时间不能太长——如果订单状态在 30 秒内更新了，用户可能看到过时的信息。所以我们只缓存查询类工具（queryOrder），不缓存写操作工具（applyRefund）。三项合计将月费从 Y19,800 降到 Y12,480，年节省 Y87,840。对用户体验的中立指标（CSAT、解决率）进行了 A/B 验证——优化前后无显著差异（p>0.05）。

判断练习

场景题：你的同事说「解决率 85% 已经很高了，不需要继续优化了」。你同意吗？

不同意。85% 的解决率意味着每 5 个用户中就有将近 1 个需要转人工。如果日均对话 3,000 次，每天有 450 个用户无法被 AI 解决。假设人工坐席每人每天处理 60 次对话，450 次意味着需要约 7.5 个坐席专门处理这些转接。如果能把解决率再提升 5 个百分点到 90%，就可以减少约 2 个坐席的人力，年节省约 Y48 万。更重要的是，分析 15% 未解决的对话类型——如果其中有一部分是可以通过技术手段减少的（如特定意图的识别率低、特定政策文档未被覆盖），那就应该持续优化。CorpAssist 的经验是，解决率的提升是有边界的——95% 可能是 LLM 当前能力的上限，但在达到这个上限之前不要轻易说「够了」。我们的做法是每季度评估一次优化目标，基于技术可行性、业务影响和投入产出比决定是否继续提升。当前 CorpAssist 的下一个目标是 88%。

Q1: 设计智能客服系统需要哪些核心模块？怎么划分？

面试官问：你来设计一个智能客服系统的模块划分，说明每个模块的职责和数据流。如果使用 Spring 技术栈，怎么选型？

智能客服系统按照「职责分离 + 异步解耦」的原则划分为六个微服务和五个核心模块。六个微服务分别是：Gateway（Spring Cloud Gateway，负责统一入口、鉴权限流和协议转换）、Intent Service（意图路由，负责规则引擎 + LLM 双引擎意图分类，无状态）、Conversation Service（对话引擎，管理 ChatClient + ChatMemory 的多轮对话，有状态）、Tool Service（工具层，注册 @Tool 业务工具，负责参数校验和幂等控制）、Agent Service（人工坐席模块，管理转人工队列 WebSocket 和坐席工作台）、Quality Service（离线质检模块，ComplianceAdvisor 异步审核和 CSAT 分析）。Gateway 作为统一入口，三层限流（IP 级别 10 req/s、用户级别 30 req/min、LLM 连接池 20 并发），P99 转发延迟 < 5ms。模块间数据流遵循责任链原则：消息 -> Intent Service 分类 -> Conversation Service（可调用 Tool Service）-> 质检通过后返回 -> 低置信度转 Agent Service。

关键技术选型：Spring AI 的 ChatMemory + Redis 管理会话状态、@Tool 注解注册业务工具（自动生成 JSON Schema）、ReactAgent 实现 Think-Act-Observe 循环（maxSteps=5）。模块间的同步通信使用 HTTP/2 + gRPC（内部调用 P99 < 50ms），异步通信使用 RocketMQ 事件（HandoffEvent、ToolResultEvent、EscalationEvent）。微服务部署在 K8s 集群，每个服务 2-3 个 Pod，有状态服务（Conversation、Agent）使用 StatefulSet 保证网络标识稳定，无状态服务（Intent、Tool）使用 Deployment。为什么 Conversation Service 要有状态？因为它的 ChatMemory 关联了 Redis 中的会话数据，虽然 Redis 是外部存储，但服务实例需要稳定的 Pod 名称来维护 WebSocket 连接和其他长连接状态。日均支撑 5 万对话，P99 端到端延迟 3.2s，可用性 99.95%。

Q2: 多轮对话如何管理上下文和指代消解？

面试官问：多轮对话怎么管理上下文？Token 超限怎么办？指代消解怎么处理？Session 之间如何共享信息？

多轮对话的上下文管理采用三层混合记忆架构。工作记忆（Working Memory）存储当前轮次的上下文——用户当前消息、正在进行的工具调用结果、临时变量，容量约 3K Tokens，存储在内存中随请求销毁。会话记忆（Session Memory）通过 Spring AI 的 ChatMemory + Redis 持久化，存储完整对话历史，TTL 30 分钟每次交互后重置。长期画像（Long-term Profile）存储用户级别的持久化信息——用户姓名、VIP 等级、历史交互记录，存储在 PostgreSQL 中永不过期，每次对话开始时通过 userId 加载注入 System Prompt。三层记忆在构建 Prompt 时按「长期画像 > 会话记忆摘要 > 工作记忆 > 当前消息」的优先级组合。当用户第 10 次找客服时，系统可以在对话开始就说「张先生您好，上次您的退货问题已经处理了，请问这次还有什么需要帮助的吗？」——这种个性化体验能显著提升 CSAT。

Token 超限的应对策略是 LLM 摘要压缩 + 滑动窗口。每 10 轮对话触发一次摘要压缩——将前 8 轮压缩为约 200 字的 LLM 摘要，保留最近 2 轮完整对话。20 轮上下文从约 6.4K Tokens 压缩到约 900 Tokens（节省 86%）。关键节点（订单号创建、退款金额确认）在压缩时被单独保留。指代消解使用三层策略：第一层在 System Prompt 中要求 LLM 注意代词指代；第二层确保 ChatMemory 提供至少最近 2 轮完整上下文；第三层在长对话中启用 CoreferenceResolver 预处理——将代词用 LLM 替换为具体名词短语。三层叠加后指代准确率从 67% 提升到 89%。线上调用指代消解的额外 Token 成本约 150 Tokens/次，但仅对 > 3 轮的对话启用，所以平均每次对话仅多消耗 30 Tokens 左右。

// 面试可写的精简版上下文管理代码
@Service
public class ConversationManager {
    private final ChatMemory chatMemory;      // Redis
    private final UserProfileRepo profileRepo; // PostgreSQL
    private final ChatClient summaryClient;

    public String buildContext(String sessionId, String userId, String message) {
        UserProfile profile = profileRepo.findByUserId(userId);
        List history = chatMemory.get(sessionId, 50);
        String summary = history.size() > 20
            ? compressSummary(history) : "";
        StringBuilder ctx = new StringBuilder();
        ctx.append("[画像] ").append(profile.name()).append("\n");
        ctx.append("[摘要] ").append(summary).append("\n");
        history.subList(Math.max(0, history.size()-4), history.size())
               .forEach(m -> ctx.append(m.getContent()).append("\n"));
        return ctx.toString();
    }

    private String compressSummary(List history) {
        String text = history.subList(0, history.size()-4).stream()
                .map(m -> m.getContent()).collect(Collectors.joining("\n"));
        return summaryClient.prompt()
                .user("摘要以下对话: " + text).call().content();
    }
}

Q3: 模型答不上来时有哪些降级策略？优先级如何排？

面试官问：LLM 不可用或者回答质量低时，你们的降级策略是什么？熔断和降级的区别？

降级策略按照系统受影响的程度分为四级。L0（正常运行）——LLM API 正常、置信度 >= 0.8、工具调用正常，使用完整 Agent + LLM + SSE 流式流程，占线上流量的 87%。L1（轻度降级）——LLM 延迟 > 5s 或单次工具调用超时时触发，系统自动切换到规则引擎 + 预设话术模式，30 秒后自动重试 LLM，占 8%。L2（转接降级）——置信度 < 0.3、Agent 超 maxSteps=5、用户要求转人工或 COMPLAINT 敏感场景时触发，收集完整 HandoffContext 发送到人工坐席队列，占 4.5%。L3（熔断降级）——LLM API 连续 3 次健康检查失败或错误率 > 50%，返回标准话术「系统维护中」，30 秒后尝试 Half-Open 恢复，占 0.5%。降级和熔断的区别：降级是「功能裁剪」（从 LLM 降到规则引擎），系统仍在运行但能力减弱；熔断是「完全断开」（不再调用 LLM API），系统进入维护模式。降级是软切换（延迟高就降级、恢复就升回来），熔断是有状态切换（Closed -> Open -> Half-Open -> Closed）。

四级降级的切换逻辑在 Gateway 层和 Conversation Service 层协同执行。Gateway 层监控 LLM API 的健康状态——如果连续 3 次健康检查失败，在 Gateway 层直接返回 L3 熔断话术，流量不进入下游服务。Conversation Service 层在每次 Agent 执行前检查降级级别，如果为 L1 则绕过 Agent 直接使用规则引擎回复。降级级别的切换是自动的，但在 Dashboard 上标记当前降级状态并发送告警。切换必须是平滑的——用户应该感受到的差异只是「回复变慢了」或「提示转人工了」，而不是「系统崩了」。CorpAssist 的降级切换耗时不超过 2s（通过健康检查的间隔配置），用户基本无感知。

Q4: 如何保证回答符合公司政策且不会产生合规风险？

面试官问：客服回答必须符合公司政策，否则可能产生法律风险。你们怎么保证机器人不乱承诺？政策文档更新了怎么办？

合规保障体系分三层构建。第一层是政策库 RAG——所有涉及政策的回答必须基于从向量库检索到的最新文档生成。政策文档以 Markdown 格式维护在 Git 仓库中，CI/CD 流水线自动解析分割为 500 字符的文本块（overlap=50），生成 1024 维 Embedding 写入 Milvus 向量库，每个文档块标注版本号和生效日期。检索时使用 FilterExpression 只查 status=active 的最新版本。Top-K=3、相似度阈值 0.75——低于该阈值的认为不相关直接转人工。第二层是 ComplianceAdvisor——在 LLM 输出返回到用户之前进行双层审核。规则层使用 Aho-Corasick 算法在 1ms 内匹配 30+ 禁止承诺词（「一定退款」「保证赔偿」等），匹配即拦截替换为合规话术。模型层使用独立的 Qwen-72B（Temperature=0.1）异步检测政策冲突。规则层拦截率 100%，模型层准确率 96%。第三层是敏感场景检测——COMPLAINT、LEGAL、CLAIM 三类意图强制转人工。三层叠加确保了系统的合规底线。过去 6 个月：合规事故从月均 2 起降到 0 起。

政策文档版本管理是合规体系的基础保障。GitLab MR + 法务审批 -> Merge 到 main -> CI/CD 触发向量化流水线 -> upsert 写入 Milvus（新版本 status=active，旧版本 status=archived）。从 MR 合并到系统生效约 5 分钟。紧急撤回可使用 ops 脚本删除指定文档的 Milvus 记录，5 秒内生效。每周围绕合规拦截数据召开合规回顾会议，更新禁止词列表。合规误拦率从初期的 15% 降到 4%（通过添加白名单 + 上下文敏感匹配优化）。为什么合规误拦率不能降到 0%？因为「安全优先宁可误拦」的原则意味着一定比例的假阳性是可接受的。4% 的误拦率在安全性和用户体验之间达到了合理的平衡。

Q5: 用什么指标评估智能客服效果？怎么持续提升？

面试官问：你怎么评估智能客服系统的好坏？线上监控哪些指标？怎么持续优化？

评估体系分为三层共 11 项关键指标。业务层指标直接反映系统的业务价值——解决率（>= 85%）、转人工率（<= 25%）、CSAT（>= 4.2）、首答时间（P99 <= 5s）。技术层指标反映系统的技术健康度——意图识别准确率（>= 93%）、工具调用成功率（>= 99%）、P99 端到端延迟（<= 3.5s）、Agent 平均步数（<= 3）。质量层指标反映系统的回答质量——引用覆盖率（>= 95%，回答中引用政策文档的比例）、合规拦截数、人工抽检合格率（>= 95%）。这三层指标有因果关联：技术层的意图准确率和工具成功率直接影响业务层的解决率和 CSAT。例如，意图准确率从 88% 提升到 94%（+6pp）直接带来了解决率从 70% 到 76%（+6pp）的等比例提升。

持续优化的闭环是：监控 -> 发现异常 -> 归因 -> 修复 -> 回归 -> 验证。每个变更先跑 7 天 A/B 实验（10% 流量），对比实验组和对照组的 5 个核心指标，p < 0.05 才全量。每周人工抽检 200 条对话计算准确率，发现新问题加入坏例库（目前 350 条）。坏例库在 CI/CD 中自动运行回归测试，通过率低于 95% 阻断发布。CorpAssist 上线 6 个月的优化历程：解决率 70%->85%（+15pp）、转人工率 35%->22%（-13pp）、CSAT 3.7->4.2（+0.5）、P99 延迟 5.8s->3.2s（-45%）。当前持续优化的重点是减少合规误拦截（目标从 4% 降到 2% 以下）和提升复杂多轮对话的 Agent 成功率。特别说明一下 P99 延迟降低 45% 的关键措施：主要是将 Agent 的 maxSteps 从 10 调整为 5，以及将 Gateway 连接池从 10 扩容到 20——两者加起来贡献了约 80% 的延迟改善。

面试可能问（进阶）

Q6: 你的系统和普通的 FAQ 机器人本质区别在哪？

本质区别在于「理解能力和自主决策能力」。FAQ 机器人基于固定规则或者关键词匹配——用户问「怎么退货」就匹配到「退货流程」的预设答案，不关心上下文、不问用户的细节、也记不住之前说了什么。而智能客服系统具有三个 FAQ 机器人不具备的能力。第一是意图理解和推理——系统能理解用户说「这个不要了」是在表达退货意图（83% 的准确率），甚至能区分「退货运费谁出」（政策咨询）和「我要退货」（操作请求）之间的细微差别。第二是多轮上下文感知——系统记住用户在第一轮提到的订单号，第二轮说「它什么时候到」时知道「它」指代的是哪个订单，不需要用户重复。第三是自主行动能力——Agent 可以自主决定调用订单系统查物流、调用售后系统创建退款工单，并组合多次操作的结果回复用户。FAQ 机器人只能「说」，智能客服系统不仅「说」还能「做」。CorpAssist 的数据显示，普通 FAQ 解决率通常在 40-50%，而智能客服系统可以达到 85%。

Q7: 如果用户用方言或者输入有错别字，你们的系统怎么处理？

智能客服系统天然具备对非标准输入的处理能力。第一层是 LLM 的语言理解优势——大语言模型在训练过程中接触过大量中文语料（包括方言表达和错别字），天然具有纠错和容错能力。例如，用户说「窝的订单到哪了」（别字），LLM 能理解「窝」应该是「我」。第二层是我们在系统层面做的补充处理——在用户消息进入 Agent 之前，先经过一个「消息标准化」步骤：敏感信息脱敏（手机号/身份证号替换为占位符）、URL 提取（将短链接转为完整链接）、明显的错别字纠正（使用一个轻量级的拼音-汉字映射表）。但要注意不要过度「标准化」——方言表达中有些是故意的（如「咋整」「啥时候到」），过度纠正会丢失语义和语气。第三层是在知识库检索中对非标准输入做 Query Expansion——用户的查询词「咋退货」被扩展为「退货」「退货流程」「退货申请」「怎么退货」等 5 个变体去向量库检索，提高命中率。线上数据表明，非标准输入（错别字/方言/口语化表达）约占总消息的 12%，经过这三层处理后，系统的理解和回答准确率与非标准输入之外的场景差距不到 3%。

Q8: 你在做这个系统的过程中犯过最大的错误是什么？学到了什么？

最大的错误是第一次上线前没有做压力测试。当时觉得开发环境自测都通过了，直接全量发布了。结果上线第一个小时，并发 150 用户就出了问题——LLM API 连接池只有 10 个连接，大量请求排队等待导致 P99 延迟从 2s 飙升到 30s。更糟糕的是，排队中的请求超时后不断重试，形成了「重试风暴」，LLM API 的错误率瞬间达到 60%，触发了熔断，整个系统进入了 L3 降级状态，所有用户都只能看到「系统维护中」。这个事故持续了 45 分钟才恢复，导致当天 CSAT 从 4.2 降到了 2.8，还有 3 个客户投诉。这次教训让我学到了三件事：第一，压力测试必须是上线前 Checklist 的硬性要求项，不能省略——后来我们做了 200 并发压力测试才让上线通过。第二，限流熔断的阈值和超时时间必须根据压力测试结果来设定，而不是拍脑袋设的——10 个连接显然太少，20 个连接是我们后来通过压力测试找出的最优值。第三，要有「优雅降级」而不是「全面崩溃」——在 Gateway 层做限流的目的是让大部分用户正常使用、小部分用户被限流（友好的 429），而不是让所有用户都看到系统崩溃。这个错误之后，我主导制定了 CorpAssist 上线 12 项 Checklist，把压力测试作为第 7 项硬性条件。

判断练习

场景题：面试官问「你的系统如果遇到 LLM API 完全不可用怎么办？」——你怎么回答能体现出系统的鲁棒性？

首先明确说明系统设计了 L3 熔断降级来应对这种情况。LLM API 连续 3 次健康检查失败或错误率 > 50% 时触发熔断，熔断后返回标准话术「系统维护中」，30 秒后自动切换到 Half-Open 尝试 1 次 LLM 调用，成功则关闭熔断器回到正常模式。但这是被动策略，我们还有主动策略：多模型备选——主模型 Qwen-72B 不可用时自动切换到备选模型（如 GPT-4o-mini 或本地部署的 Qwen-14B），切换过程对用户透明，延迟增加约 1-2s 但服务不中断。备选模型的成本是主模型的 3-5 倍，所以不能长期使用，但 15 分钟内可以保证服务不中断。如果备选模型也不可用（极端情况），引擎完全切换到规则引擎 + 预设话术模式，虽然能力大幅下降（解决率约 40%），但至少用户可以收到回复而不是空白。这种「主 -> 备 -> 兜底」的多层方案确保了系统在极端情况下的基本可用性。此外，我们还有离线缓存策略——将最高频的 200 个 FAQ 的 LLM 回复缓存到本地，在网络完全中断时这些缓存可以直接返回。

场景题：面试官问「你的效果评估体系中的引用覆盖率 96% 是怎么计算的？太低或者太高分别意味着什么？」

引用覆盖率的计算方法是：在所有涉及政策咨询的对话中（意图为 POLICY_QUERY 或对话中触发了政策 RAG 检索），人工检查 Agent 的回复是否明确引用了检索到的政策文档（如在回答中包含「根据《退货政策 V3.2》第 4.1 条」这样的引用）。回复中包含引用标记的计入「已覆盖」，不包含的计入「未覆盖」。覆盖率 = 已覆盖 / (已覆盖 + 未覆盖) x 100%。CorpAssist 的 96% 表明在政策相关对话中，Agent 在 96% 的情况下正确引用了政策来源。引用覆盖率太低（如低于 80%）说明 Agent 经常「自由发挥」——回复没有基于检索到的政策文档，这可能意味着 System Prompt 中的引用指令不够明确、或者 Agent 跳过检索直接凭记忆回复了。引用覆盖率太高（如 100%）也不一定好——有可能 Agent 在不需要引用的时候（如问候语「你好」）也试图引用文档，这属于过度引用。理想范围是 95-98%——政策相关的回答引用文档、非政策性的问候不需要引用。我们定期抽检未覆盖的 4%，分析是 Agent 忘了引用还是确实不需要引用，对前者调整 Prompt，对后者调低覆盖率分母。

客服核心指标：解决率/转人工率/CSAT/首答时间

智能客服的效果评估围绕四个核心 KPI 展开，每个指标都有明确的行业基准值和对应的技术优化路径。解决率（Resolution Rate）衡量用户在机器人内完成咨询而不需要转接人工坐席的比例，是客服场景最关键的北极星指标。CorpAssist 在某电商客户的 A/B 实验数据显示，解决率每提升 5 个百分点，人工客服成本降低约 12%，因此技术团队应将解决率作为首要优化目标。转人工率（Handoff Rate）与解决率呈负相关，行业基准为 20%-35%，金融行业因合规要求可达 40%-50%。CSAT（Customer Satisfaction Score）是用户对服务体验的主观评分（1-5 分），受回答准确性和对话连贯性的双重影响。首答时间（First Response Time）衡量用户发消息到机器人首次回复的延迟，直接影响用户对服务效率的第一印象。

在 CorpAssist 的实践中，这四项指标之间存在相互制约关系。例如，将转人工阈值从置信度 0.5 提升到 0.8 虽然可以降低转人工率（从 35% 降到 22%），但也会导致部分本应转人工的复杂问题滞留机器人侧，拉低 CSAT 评分。因此需要建立指标间的平衡，而不是孤立优化某一项。我们的经验法则是：首答时间优先保证（< = 3s P99），在此基础上逐步提升解决率，每隔两周通过人工抽检 200 条对话来校准 CSAT 评分与自动化指标之间的偏差。这四项指标的优先级排序在 CorpAssist 内部被称为 KPI 金字塔模型——首答时间是地基（直接影响对话放弃率），解决率和转人工率是主体（直接影响成本），CSAT 是塔尖（综合反映服务质量）。

CorpAssist 客服模块边界与功能范围

S2 客服模块覆盖三类核心对话场景：售前咨询（商品功能咨询、价格询问、库存查询）、售后工单（订单查询、退货申请、退款咨询、物流追踪、地址修改）、政策查询（保修政策、退换货运费规则、优惠券使用说明）。每类场景的技术方案不同：售前咨询以政策库 RAG 结合多轮追问为主，售后工单需要意图识别后调用 @Tool 业务工具（订单 API、售后 API、物流 API），政策查询则依赖 S1 知识库的检索增强生成和引用溯源机制。三类场景的流量占比在 CorpAssist 的实际部署中约为售前 25%、售后 55%、政策查询 20%。

S2 模块的能力边界严格遵循「自动化解决高频问题，人工处理复杂异常」的原则。在 In-Scope 范围内，系统支持多轮对话管理（ChatMemory + Redis 持久化）、意图识别与路由（Structured Output + 规则引擎双引擎）、工具调用（订单查询/退换货/改地址）、政策文档 RAG 检索、以及置信度不足时的自动转接。Out-of-Scope 的范围明确包括：涉及支付密码或银行账户的敏感操作（如修改银行卡号、密码重置）、需要多层审批的长流程（如企业级采购审批）、以及法律/合规要求必须由人类介入的对话——这些场景即使 LLM 给出了回答，系统也会强制转人工并记录日志。此外，S2 不处理代码辅助（S4 场景）和内容创作（S5 场景），保持模块职责的单一性。边界划分是 CorpAssist 架构设计中最关键的决策之一——在早期版本中，S2 尝试覆盖了太多能力（包括长流程审批），导致系统复杂度失控，花了两周重构才梳理清楚。从那以后我们严格坚守「每个场景只做一件事」的原则。

Spring 技术栈在智能客服中的优势

Spring AI 在企业级智能客服场景相比 Python 栈（如 LangChain + FastAPI）具有四个层面的差异化优势。第一，Spring AI 的 ChatMemory 接口提供统一的多轮会话管理抽象，支持 Redis/Jdbc/InMemory 三种后端实现，开发者只需配置 spring.ai.chat.memory.type=redis 即可获得生产级的会话持久化能力。第二，@Tool 注解将 Java 方法的参数和返回值自动映射为 LLM 的 Function Calling Schema，无需手动编写 JSON Schema 定义，@Tool(description = "查询订单状态") 即可注册一个工具。第三，Spring AI 的 Flux 响应式流与 Spring WebFlux 完全兼容，天然支持背压控制和服务端推送，相比 Python 的 FastAPI StreamingResponse 提供了更精细的流量控制。第四，企业集成方面，Spring Cloud Gateway、Spring Security、Resilience4j、Micrometer 等组件与 Spring AI 共享同一编程模型和配置体系，避免了 Python 栈中需要跨框架集成的碎片化问题。

在 CorpAssist 的实践中，Spring 技术栈带来的实际收益包括：开发周期从预估的 6 周缩短到 4 周（减少 33%），因为 LLM 集成、会话管理和工具注册都通过声明式配置完成；生产故障从每月平均 3 起减少到 1 起，因为 Spring Cloud Gateway 的限流和熔断机制在现网验证了 8 个月；运维人员可以从 Java 团队内部调配，无需额外招聘 Python 工程师。但需要注意的是，Spring AI 目前对 Python 生态的 LangChain 社区插件和第三方工具链集成度较低，如果业务需要大量使用社区工具（如 Wikipedia、PubMed 等），Python 栈可能是更好的选择。在选择技术栈时，除了功能对比外还需要考虑团队的技能分布——如果你的团队以 Java 工程师为主，选择 Spring AI 可以减少跨语言协作的沟通成本；如果是全栈 Python 团队，LangChain 可能更自然。

在 CorpAssist 某零售客户的实施数据基础上，我们构建了以下 ROI 量化模型。该客户日均对话 3,000 次，月均 90,000 次，高峰日可达 5,000 次。传统模式下需要 15 名人工坐席（两班倒），引入智能客服后减少到 6 名（负责复杂问题的处理和 AI 辅助审核）。智能客服解决了约 70% 的对话，剩余 30% 转人工。模型假设 LLM Token 均摊成本为 ¥0.15/1K Tokens（基于 Qwen-72B 企业级 API 报价），平均每次对话消耗 800 Tokens（含系统 Prompt 和工具调用开销）。基础设施包括 4 台 8 核 32G 服务器（¥12,000/月）和 Redis 集群（¥3,000/月），年化约 ¥180,000。计算得到智能客服年总成本 ¥1,830,000，较传统模式 ¥3,960,000 节省 54%，投资回收期为 4.5 个月。

Token 成本是 AI 客服的主要变量成本，需要精细化管理。我们的 Token 成本分拆模型显示：系统 Prompt（场景定义 + 格式约束）固定占用约 300 Tokens，用户消息平均 150 Tokens，工具调用逻辑（含 Function Calling Schema 和工具结果回注）占用约 200 Tokens，生成回复约 150 Tokens。日均 3,000 次对话的总 Token 消耗约为 2.4M Tokens/天（3,000 * 800），折合 ¥360/天。在 CorpAssist 的生产优化中，通过引入上下文摘要压缩（将 20 轮历史从 4K Tokens 压缩到 600 Tokens），Token 消耗降低了 37%，年节省约 ¥48,000。缓存高频问题（前 20% 的 FAQ）的 LLM 回复可以实现额外的 15% Token 节省。不同规模企业部署智能客服的 ROI 差异很大——我们绘制了不同日均对话量下的 ROI 曲线：日均 500 对话的回收期约 18 个月，日均 3,000 对话的回收期缩短到 4.5 个月，日均 10,000 对话的大型部署回收期仅 2.1 个月。

面试可能问

Q1: 智能客服的核心 KPI 有哪些？解决率提升到 70% 以上需要哪些技术支撑？

四个核心 KPI：解决率（Resolution Rate）、转人工率（Handoff Rate）、CSAT（满意度评分）、首答时间（First Response Time）。将解决率提升到 70% 以上需要三个关键技术支撑：第一是准确的意图识别，结构化输出方式的意图分类器准确率约 94%，误判率低于 5%；第二是全面的知识库覆盖，确保非个性化问题都能检索到对应的政策或话术，CorpAssist 的知识库覆盖率与解决率呈强正相关（r=0.87）；第三是优秀的对话管理能力，包括多轮追问、指代消解和上下文连贯性。实践中，仅靠意图识别可以达到约 55% 的解决率，加上知识库覆盖可以提升到 63%，再加上多轮对话管理可以稳定达到 72% 以上。Spring AI 在这三个层面提供了对应组件：Structured Output 分类器、VectorStore RAG、ChatMemory 会话管理。另外需要提醒的是，KPI 之间存在制约关系——不要孤立优化单一指标，需要搭建指标体系统一看板来监控综合效果。

Q2: 转人工率 30% 算高还是低？什么场景下需要接受更高的转人工率？

30% 在行业整体市场中属于平均水平（行业基准 20%-35%），但具体好坏取决于业务场景。金融行业的智能客服转人工率普遍在 40%-50% 之间，这是因为金融合规要求严格——涉及资金操作、账户安全、法律纠纷等问题即使 AI 能回答也必须转人工处理。电商场景的合理范围是 25%-30%，因为售后工单（退货退款等）的流程相对标准化，AI 处理成功率更高。CorpAssist 的经验是：不应当追求无限降低转人工率，而应该追求「正确的转人工」——将高价值、高风险、需要人工判断的复杂问题转人工（如纠纷调解、高额退款审核），低价值高频问题（如查物流、查订单状态）由机器人解决。可以通过设置意图级别的转人工阈值来实现精细化控制：例如，COMPLAINT 意图的转人工阈值设为 0.3（更易转出），而 ORDER_QUERY 设为 0.5（更倾向机器人处理）。

Q3: Spring AI 在企业级智能客服中相比 Python 栈的核心优势是什么？什么场景下 Python 栈更合适？

Spring AI 相比 Python LangChain 的四个核心优势：1) ChatMemory 接口提供统一的会话管理抽象，配置 spring.ai.chat.memory.type=redis 即可使用生产级 Redis 持久化，Python 需要额外集成和配置。2) @Tool 注解将 Java 方法自动暴露为 LLM 工具，包含完整的参数校验和异常处理，Python 需要手动定义 JSON Schema。3) Flux 原生响应式流式输出，与 Spring WebFlux 完美集成，Python 的 StreamingResponse 需要额外管理背压。4) Spring Cloud Gateway、Security、Resilience4j、Micrometer 等企业组件共享同一配置模型，降低了集成复杂度。在开发效率上，CorpAssist 团队使用 Spring AI 将开发周期从 6 周缩短到 4 周。但 Spring AI 对 Python 生态的 LangChain 社区插件和第三方工具链集成度较低。如果业务需要大量使用社区工具（如 Wikipedia、PubMed 检索器）或依赖 Python 的 NLP 模型生态（如 HuggingFace 的微调工具），Python 栈更合适。另外，团队技术栈取向也是重要考量——Java 团队维护 Spring 应用的成本远低于同时维护 Java 和 Python 两套系统。

判断练习

场景题：你的产品经理提出要将转人工率从 30% 降低到 15%，你会怎么做？

首先需要理解这个目标的合理性：15% 的转人工率意味着 85% 的对话由 AI 独立解决，这在当前的 LLM 能力水平下（意图分类上限约 94%，加上知识库覆盖率和对话管理损耗，实际解决率通常不超过 75%）是不现实的目标。我会和产品经理一起分析：转人工的 30% 对话中，有多少是「不应该转人工」的（即系统可以处理好但错误转出），有多少是「必须转人工」的（如 COMPLAINT 意图或用户明确要求转人工）。根据 CorpAssist 的实践数据，约 8% 的转人工是系统算法导致的误判，可以通过优化意图分类器（使用更细粒度的 Few-Shot 示例）和降低规则引擎误触率来减少。真正的极限目标应该是将「不应该转人工」的部分降到接近 0%，实际转人工率约 22%-25%，而非盲目追求 15%。我会提供数据支持这个结论，并建议将精力放在提升解决率和 CSAT 上而非单纯压低转人工率。

场景题：你收到告警发现首答时间 P99 从 2s 飙升到 8s，影响了 CSAT 评分。如何排查和修复？

这是一个典型的线上性能问题，排查需要分层进行。第一层检查 LLM API 的响应时间：如果 API 本身的 P99 延迟增加了（比如从 1.5s 到 6s），说明问题在模型服务端，需要联系模型提供商或自建模型服务团队排查。第二层检查 Spring Cloud Gateway 的限流情况：如果限流触发了排队，实际响应时间会显著增加，可以通过 Gateway 的 RequestRateLimiter 日志确认限流次数。第三层检查下游工具调用（如订单 API）的延迟：如果工具调用响应变慢（如 P99 从 500ms 到 3s），需要排查对应的内部服务。第四层检查 Redis 会话操作的延迟：频繁的读写操作在高并发下可能成为瓶颈。CorpAssist 的排查 SOP 是：先看 Gateway 监控大屏确认整体流量是否异常，再看 LLM API 的响应时间趋势图，最后看分布式链路追踪（Spring Cloud Sleuth + Zipkin）定位具体慢在哪个环节。修复措施可能包括：扩容模型推理实例、增加 Gateway 限流配额、对高频查询添加本地缓存（Caffeine Cache，过期时间 30s）、以及将非关键路径操作（如日志记录）改为异步处理。

五大核心模块：意图路由/对话引擎/工具层/人工坐席/质检

CorpAssist 客服系统划分为五个核心模块，每个模块有明确的职责边界和对外接口。意图路由模块（Intent Router）负责接收用户消息，通过双引擎（规则引擎 + LLM 结构化分类）识别用户意图并输出 IntentResult（包含 intent 枚举、confidence 置信度、slots 实体列表）。对话引擎模块（Conversation Engine）是系统的核心，基于 Spring AI 的 ChatClient 管理多轮对话，调用 ChatMemory 读写会话历史，并在需要时触发工具调用或知识库检索。工具层（Tool Layer）以 @Tool 注解注册的 Java 方法实现具体的业务操作，如订单查询、退款处理、地址修改——每个工具都有独立的参数校验和错误处理逻辑。人工坐席模块（Agent Workspace）管理转人工队列，通过 Redis 有序集合（ZSET）实现坐席分配和排队，在转接时将完整的上下文（对话历史 + 用户画像 + 当前意图）移交给坐席工作台。质检模块（Quality Check）运行 ComplianceAdvisor 对模型输出进行合规审核，并对已完结的对话进行离线抽检和评分。

这五个模块遵循「前一个模块处理不了的事情交给下一个模块」的责任链原则。具体来说：消息到达后，意图路由模块先做意图分类——如果置信度 >= 0.8，交给对话引擎；如果置信度在 0.5-0.8 之间，触发二次确认；如果置信度 < 0.5 或意图为 UNKNOWN，直接转人工。对话引擎在生成回复时，如果需要的工具调用失败（重试耗尽），也降级转人工。质检模块对所有输出进行后处理，如果检测到合规问题（如包含「一定退款」等禁止承诺词），拦截回复并转人工审核。这种责任链设计保证了每个模块只关注自己的职责，模块间的耦合度低，便于独立测试和部署。

Spring Cloud Gateway 作为统一入口的架构设计

Spring Cloud Gateway 在 CorpAssist 客服系统中扮演着统一流量入口的角色，负责认证鉴权、请求限流、意图路由和协议转换四个核心功能。所有用户请求首先到达 Gateway，它基于 JWT Token 进行身份认证（通过 Spring Security OAuth2 Resource Server 配置），然后根据请求路径将消息路由到不同的后端服务。客服模块的 API 路径统一以 /api/cs/ 为前缀，例如 /api/cs/chat 用于对话请求，/api/cs/handoff 用于转人工请求，/api/cs/history 用于历史查询。Gateway 还负责将 HTTP 长连接转换为 SSE（Server-Sent Events）流式响应，支持用户在对话过程中实时接收 LLM 的流式输出。在 CorpAssist 的现网部署中，Gateway 的 P99 转发延迟控制在 5ms 以内，每日处理约 50 万次请求。

限流策略是 Gateway 的一个关键设计点。CorpAssist 实现了三层限流：第一层是 IP 级别的 RequestRateLimiter，限制单个 IP 每秒最多 10 次请求；第二层是用户级别的令牌桶，每个用户 1 分钟最多 30 次对话请求；第三层是后端服务的连接数控制，限制同时进行中的 LLM 调用不超过 20 个，防止 LLM API 过载。在实践中最容易踩坑的是流式连接的限流——如果用户发一条消息后立刻关闭页面再打开重连，Gateway 会在短时间内积累大量未完成的 SSE 连接。我们的解决方案是在 Gateway 层添加连接池管理，每个用户 ID 最多同时保留 3 个 SSE 连接，超出后自动关闭最旧的连接。以下代码展示了 Gateway 的核心路由和限流配置。

模块间的数据流、接口契约与事件驱动通信

五个模块之间的数据流通过同步 API 和异步事件两种方式配合完成。同步 API 主要用于实时请求-响应场景：用户消息经 Gateway -> 意图路由模块（HTTP 调用）-> 对话引擎模块（HTTP 调用）-> 工具层（内部方法调用）-> 返回结果。同步链路的 P99 延迟要求控制在 5s 以内，因此每个环节的耗时都必须受控。异步事件用于非实时的模块间通信：当发生转人工操作时，对话引擎向「人工坐席模块」发送 HandoffEvent 消息（包含完整上下文）；当工具调用失败时，产生 ToolResultEvent 用于后续分析和告警；当质检模块检测到合规问题，发送 EscalationEvent 触发升级流程。异步通信基于 Spring Cloud Stream 实现，消息中间件可以选择 RabbitMQ（低延迟场景）或 RocketMQ（高吞吐场景）。

接口契约是模块间协作的基础。每个模块对外暴露的 API 都有明确的请求/响应结构和错误码规范。例如，意图路由模块的 /api/cs/intent/classify 接口接收 ClassifyRequest（包含 message、sessionId、userId），返回 IntentResponse（包含 intent、confidence、slots、rawText）。所有模块共享统一的 ApiResponse 包装类，错误码遵循 HTTP 状态码加业务码的格式（如 200 成功、400 参数校验失败、500 内部错误）。以下代码定义了核心的数据契约和事件结构。

        return ApiResponse.builder()
                .status(status).code(code).message(message)

    }
}

Q1: 设计一个智能客服系统的微服务划分方案，你会怎么拆分？为什么？」

我会按业务能力拆分为六个微服务：Gateway（统一入口，负责鉴权/限流/路由）、Intent Service（意图识别，无状态易扩缩容）、Conversation Service（对话管理，有状态需要 ChatMemory 持久化）、Tool Service（工具调用，依赖外部业务 API）、Agent Service（人工坐席管理，需要 WebSocket 长连接）、Quality Service（离线质检，异步处理）。这种拆分方案的核心原则是「按变更频率和性能需求差异化拆分」：Intent Service 算法迭代频繁需要独立部署和灰度发布，Conversation Service 是有状态服务不能随意扩缩容需要 StatefulSet 管理。在 CorpAssist 的实践中，这种六服务划分让每个团队的职责明确——算法团队只关注 Intent Service 和 Quality Service，后端团队关注 Conversation 和 Tool Service，运维团队关注 Gateway 和 Agent Service。每个服务可以独立发布、独立测试，互不阻塞。

Q2: 网关层在智能客服系统中的职责有哪些？如果流量突增 10 倍，网关应该怎么保护后端？

网关层在智能客服系统中有四个核心职责：路由转发（将 /api/cs/chat 映射到 Conversation Service）、认证鉴权（JWT Token 验证和用户身份提取）、流量控制（限流、熔断、重试）、协议转换（HTTP -> SSE 流式响应）。当流量突增 10 倍时，网关需要执行三重保护策略：第一层是请求级别限流，使用 RequestRateLimiter 限制单位时间内的总请求数和每个用户的并发数——例如将每秒总请求从 500 限制到 800（仍然允许一定增长但不完全放开）；第二层是后端连接数控制，使用 Resilience4j Bulkhead 限制与 LLM API 的同时连接数不超过 30 个，超过的请求直接返回 429 状态码和「系统繁忙，请稍后再试」的兜底话术；第三层是熔断，当 LLM API 的错误率超过 50% 时触发 CircuitBreaker 熔断 30 秒，在此期间所有对话请求直接使用规则引擎回复。这三层保护确保了后端服务的稳定性，即使网关层有 10 倍流量，后端 LLM API 和数据库也不会被打垮。

Q3: 模块间用同步调用还是异步消息？你怎么选择？

选择依据很简单：用户需要实时等待的结果用同步调用，不需要或不希望用户等待的操作用异步消息。在 CorpAssist 中，实时对话链路（用户消息 -> 意图识别 -> 对话生成 -> 回复）必须使用同步调用，因为用户期望在几秒内得到回复，这个链路的 P99 延迟要求 <= 5s。而转人工操作、合规拦截告警、质检评分报告等非实时操作使用异步消息。具体事件包括：HandoffEvent（转人工通知）、ToolResultEvent（工具调用结果记录）、EscalationEvent（合规升级通知）。异步消息的技术选型是 Spring Cloud Stream + RocketMQ，因为 RocketMQ 支持事务消息保证至少一次投递，且延迟低（P99 < 5ms）。混合通信的架构要求：同步链路必须设置超时（HTTP 连接超时 3s、读取超时 10s），异步链路必须设置死信队列处理消费失败的消息。曾经有个坑：转人工的 HandoffEvent 如果消费失败（坐席模块重启），因为没有死信队列，消息就丢失了——用户等着转人工却没有任何反馈。加上死信队列后，失败消息会被重新投递到坐席模块的备用队列，保证最终一致性。

判断练习

场景题：你的同事提议将 Intent Router 和 Conversation Engine 合并为一个服务，理由是「减少内部调用延迟」。你怎么看？

这个提议的部分理由是正确的——合并可以消除 HTTP 内部调用，减少约 20-50ms 的延迟。但从架构角度看，合并会带来三个更大问题：第一，意图识别的算法迭代速度比对话管理快得多（算法团队每两周更新一次分类器），合并在一个服务里会导致频繁的全量部署，影响对话引擎的稳定性。第二，两者对资源的需求不同：Intent Router 是计算密集型（LLM 推理占用 GPU），Conversation Engine 是 IO 密集型（大量 Redis 读写和外部 API 调用），混合部署会使资源调度复杂化。第三，可观测性受损——合并后无法独立监控意图识别的准确率和对话引擎的延迟。CorpAssist 的实践数据表明，不合并虽然增加 30ms 内部调用延迟，但两个服务各自独立发布 32 次和 18 次而互不影响，团队协作效率提升约 40%。对于延迟敏感场景，可以通过 gRPC 替代 HTTP 将内部调用延迟降到 2ms 以内。因此我建议保持拆分，优化通信协议而非合并服务。

场景题：Gateway 在压力测试中，当并发数超过 100 时 LLM 调用超时率上升到 30%。如何诊断和修复？

这个问题需要分层排查。首先确认超时是发生在 Gateway 到后端的连接上，还是后端到 LLM API 的连接上。在 Gateway 的访问日志中查看 upstream_response_time 指标，如果这个值正常（< 3s）但 total_time 很大（> 10s），说明是 Gateway 内部的连接池排队导致的。修复方案：调整 Gateway 的 HTTP 连接池参数——spring.cloud.gateway.httpclient.pool.maxConnections=500 和 maxPendingConnectionAcquire=100。如果 upstream_response_time 本身就很大，说明是 Conversation Service 的线程池满了或者 LLM API 限流了：检查 Conversation Service 的 ThreadPoolTaskExecutor 配置，核心线程数需要从默认的 8 调整为 20，队列容量从 100 调整为 500。更重要的改进是添加 Resilience4j Bulkhead 隔离：限制同时进行中的 LLM 调用不超过 20 个，多出的请求直接在 Gateway 层快速失败返回 429 状态码，而不是在后端排队消耗资源导致大面积超时。此外，为 LLM API 添加 Client-Side 的 ConnectionPool 和 Timeout 配置：connectTimeout=3s, readTimeout=15s, maxConnections=20。

Spring AI ChatMemory + Redis 持久化方案

Spring AI 的 ChatMemory 接口是构建多轮对话能力的基础设施。它抽象了会话历史的存储和读取操作，提供 add()（追加消息）、get()（按 sessionId 读取）、clear()（清理会话）三个核心方法。ChatMemory 有三种内置实现：InMemoryChatMemory（开发测试用，重启丢失）、JdbcChatMemory（基于 JDBC 的关系型存储）、RedisChatMemory（基于 Redis List 数据结构的生产级实现）。CorpAssist 在生产环境中使用 RedisChatMemory，因为它天然支持 TTL 过期、高并发读写（Redis 单实例 QPS 可达 10 万+）、以及数据持久化（AOF + RDB 混合持久化）。配置只需要一行：spring.ai.chat.memory.type=redis。会话的 TTL 设置为 30 分钟，每次用户发送新消息时重置过期时间，确保活跃会话不丢失而长时间不活跃的会话自动清理。在 Redis 中的存储结构是 chat_memory:{sessionId} 的 List，每个元素是一条 ChatMessage（包含 messageType、content、timestamp、metadata）。

生产环境中的实践教训值得注意。第一个问题是 ChatMemory 的大小控制：如果不加限制，极端情况下一个超长会话（50+ 轮）可能在 Redis 中存储数万 Tokens 的消息数据。我们的解决方案是在每次 add() 操作后检查 List 长度，如果超过 50 条消息则触发摘要压缩（将前 48 条摘要化，保留最新 2 条完整）。第二个问题是并发写入：当用户快速连续发送消息时，RedisChatMemory 的写入操作需要保证顺序。Spring AI 的实现已经使用 Redis 的 RPUSH 保证了追加顺序。第三个问题是序列化性能：默认的 JDK 序列化方式效率较低，我们切换到 Jackson JSON 序列化，消息大小减少了 60%（从平均 2.1KB 降到 0.8KB），Redis 内存占用显著降低。
