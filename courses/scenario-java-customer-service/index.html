JWT + Spring Security 鉴权

java
@Configuration
@EnableWebFluxSecurity
public class SecurityConfig {

    @Bean
    public SecurityWebFilterChain securityWebFilterChain(
            ServerHttpSecurity http) {
        return http
            .authorizeExchange(exchanges -> exchanges
                .pathMatchers("/api/v1/cs/chat/**").authenticated()
                .pathMatchers("/api/v1/cs/handoff/**").hasRole("AGENT")
                .pathMatchers("/fallback/**").permitAll()
                .anyExchange().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(Customizer.withDefaults())
            )
            .build();
    }

    @Bean
    public ReactiveJwtDecoder jwtDecoder() {
        return NimbusReactiveJwtDecoder
            .withJwkSetUri("https://auth.corp.example/.well-known/jwks.json")
            .build();
    }
}

SSE 全链路超时配置

yaml
# Gateway 层
spring:
  cloud:
    gateway:
      routes:
        - id: cs-chat-stream
          uri: lb://cs-chat-engine
          predicates:
            - Path=/api/v1/cs/chat/stream
          metadata:
            response-timeout: 60000  # 60 秒 SSE 超时

# 应用层
server:
  reactive:
    session:
      timeout: 30m
  netty:
    max-keep-alive-requests: 1000

# LLM 调用超时
spring:
  ai:
    openai:
      chat:
        options:
          timeout: 30s  # LLM 调用 30 秒超时

SSE 超时策略的三层配置：Gateway 层 60s -> 应用层 30s(LIM) -> LLM 层 30s。任何一层超时都会返回友好错误给用户，并记录超时事件到监控系统。

本章结论

  • Gateway + Resilience4j 限流(100 QPS) + 熔断(50% 失败率)
  • JWT + Spring Security OAuth2 鉴权，坐席角色额外权限
  • SSE 三层超时：Gateway 60s -> 应用 30s -> LLM 30s
  • 熔断降级到 FallbackController 返回友好提示

Q1: 模块划分

面试官问：设计一个智能客服系统的模块划分，说明每个模块的职责。

回答框架：

  1. 五大模块：意图路由(Intention Router) -> 对话引擎(Dialog Engine) -> 工具层(Tool Layer) -> 人工坐席(Agent Workspace) -> 质检(Quality Check)
  2. 数据流：消息经 Gateway 进入 -> 意图路由分类 -> 对话引擎处理(可调用工具/查知识库) -> 低置信度转人工坐席
  3. Spring 技术栈：Spring Cloud Gateway(路由/限流)、ChatClient(对话)、@Tool(工具)、ChatMemory(会话)、Advisor(质检)
  4. 关键数据：日均 3000 对话，意图识别准确率 94%，转人工率 22%

Q2: 多轮对话

面试官问：多轮对话怎么管理上下文？Token 超限怎么办？指代消解怎么处理？

回答：

  1. ChatMemory + Redis：Spring AI 的 ChatMemory 接口管理会话历史，Redis 存储。30 分钟 TTL，每次交互后重置
  2. 上下文摘要压缩：20 轮约 4K Tokens -> LLM 摘要压缩到 600 Tokens（保留最近 2 轮完整 + 摘要），节省 85%
  3. 指代消解：通过 System Prompt 约束 + 最近 N 轮上下文。Prompt 要求「当用户说「这个/它」时，结合最近两轮消息推断指代对象」
  4. Hybrid Memory：工作记忆(当前轮) + 会话记忆(N轮) + 长期画像(用户偏好/会员等级)

Q3: 降级策略

面试官问：LLM 不可用或者回答质量低时，你们的降级策略是什么？

回答使用四级降级模型：

级别条件行为
L0 正常LLM 可用，置信度 >= 0.8LLM 完整回答 + 工具调用
L1 降级LLM 延迟 > 5s 或 Tool 超时使用预设话术 + 规则引擎回复
L2 转接置信度 < 0.30 或用户要求转人工上下文移交到人工坐席队列
L3 熔断LLM API 不可用标准话术：「系统维护中，请稍后再试或致电 400-xxx」

Q4: 政策合规

面试官问：客服回答必须符合公司政策。你们怎么保证机器人不乱承诺？

三层合规体系：

  1. 政策库 RAG：所有回答基于检索到的最新政策文档，FilterExpression 确保只查 active 版本
  2. ComplianceAdvisor：在输出前检查回答是否包含禁止承诺词（「一定退款」「保证赔偿」），匹配即标记
  3. 敏感场景检测：投诉/索赔/法律等关键词自动触发转人工

实际数据：禁止承诺词的拦截率 100%（只要包含一定触发），政策冲突检测准确率约 96%。被拦截的回答自动送人工审核。

Q5: 效果评估

面试官问：你怎么评估智能客服系统的好坏？线上监控哪些指标？

三层指标体系：

业务层：解决率(>= 85%)、转人工率(<= 25%)、CSAT(>= 4.2)、首答时间(<= 5s)

技术层：意图识别准确率(>= 93%)、工具调用成功率(>= 99%)、P99 延迟(<= 3s)

质量层：引用覆盖率(>= 95%)、合规拦截数、人工抽检合格率(>= 95%)

闭环流程：指标 -> 发现异常 -> 归因 -> 修复 -> 回归 -> 验证指标恢复。

本章结论

  • Q1 模块：五大模块 + Spring Cloud Gateway 路由
  • Q2 多轮：ChatMemory + 摘要压缩 + 指代消解 + Hybrid Memory
  • Q3 降级：四级模型(L0-L3) 从正常到熔断
  • Q4 合规：政策 RAG + ComplianceAdvisor + 敏感场景检测
  • Q5 评估：三层指标体系 + 闭环迭代流程

Spring ChatMemory + Redis 持久化

Spring AI 的 ChatMemory 接口提供了统一的会话管理抽象。生产环境建议使用 Redis 实现，支持分布式共享。

public class ChatMemoryConfig {

@Service
public class ConversationService {

    private final ChatMemory chatMemory;
    private final ChatClient chatClient;

    // 会话配置
    private static final int MAX_HISTORY_TOKENS = 4096;
    private static final int MAX_TURNS = 20;

    public Flux chat(String sessionId, String userMessage) {
        // 1. 获取或创建会话
        List history = chatMemory.get(sessionId, MAX_TURNS);

        // 2. 添加用户消息
        chatMemory.add(sessionId, new UserMessage(userMessage));

        // 3. 检查 Token 上限，触发摘要压缩
        if (countTokens(history) > MAX_HISTORY_TOKENS) {
            history = compressHistory(sessionId, history);
        }

        // 4. 构建 Prompt（system + history + user）
        return chatClient.prompt()
            .system(buildSystemPrompt())
            .messages(history)
            .user(userMessage)
            .stream()
            .content()
            .doOnComplete(() -> {
                // 完成后将 assistant 消息加入历史
                chatMemory.add(sessionId,
                    new AssistantMessage(lastResponse));
            });
    }

    // Token 计数
    private int countTokens(List messages) {
        return messages.stream()
            .mapToInt(m -> m.getText().length() / 2) // 中文约 2 char/token
            .sum();
    }

    private String buildSystemPrompt() {
        return """
            你是一个智能客服助手。请根据对话历史理解上下文。
            - 如果用户说「这个」或「它」，指代最近提到的物品/话题
            - 如果用户说「刚才说的那个」，回溯到上上轮的讨论
            - 保持回答简洁、友好
            """;
    }
}

Redis 存储结构：session:{sessionId}:messages -> List。TTL 设置为 30 分钟无交互自动过期。每次对话打 ID 后重置 TTL。

上下文摘要压缩

当对话历史超过 Token 上限时，不是简单截断（会丢失信息），而是用 LLM 生成摘要替代。

java
@Service
public class ContextCompressionService {

    private final ChatClient summaryClient;

    private static final String COMPRESSION_PROMPT = """
        请将以下客服对话浓缩为简洁的摘要，保留关键信息：
        - 用户的主要诉求
        - 已经提供的帮助或操作
        - 待解决的问题
        - 用户的身份/订单号等关键信息

        对话历史：
        {history}

        摘要（200字以内）：
        """;

    public List compress(String sessionId,
                                   List history) {
        String historyText = history.stream()
            .map(m -> m.getMessageType() + ": " + m.getText())
            .collect(Collectors.joining("\n"));

        // 用 LLM 生成摘要
        String summary = summaryClient.prompt()
            .user(u -> u.text(COMPRESSION_PROMPT)
                .param("history", historyText))
            .call()
            .content();

        // 只保留最近 2 轮完整对话 + 之前的摘要
        List compressed = new ArrayList<>();
        compressed.add(new SystemMessage("对话摘要: " + summary));

        // 保留最后 2 轮完整消息
        int keepTurns = 2;
        for (int i = Math.max(0, history.size() - keepTurns * 2);
             i < history.size(); i++) {
            compressed.add(history.get(i));
        }

        return compressed;
    }
}

压缩效率：一个 20 轮的完整对话约 4,000 Tokens，压缩后约 600 Tokens（摘要 150 + 最近 2 轮 450）。Token 节省约 85%。CompressService 使用独立的轻量模型（qwen-turbo）以降低成本。

        if (profile.getMemberLevel() != null) {

            sb.append("，偏好语言 ").append(profile.getPreferredLang());
            if (profile.getRecentOrders() != null && !profile.getRecentOrders().isEmpty()) {

                    .append(profile.getRecentOrders().get(0).orderId());

                .append(" 轮对话。\n\n");

政策库 RAG：Spring AI 检索增强

客服场景的政策问答与 S1 知识库共享同一套检索基础设施，但需要额外的「版本锁」确保回答基于最新政策。

java
@Service
public class PolicyRagService {

    private final VectorStore vectorStore;
    private final ChatClient chatClient;

    public Answer answerWithPolicy(String query, String department) {
        // 1. 检索最新政策（按版本过滤）
        SearchRequest request = SearchRequest.builder()
            .query(query)
            .topK(3)
            .filterExpression(FilterExpression.builder()
                .eq("department", department)
                .eq("status", "active")     // 只查生效政策
                .eq("policy_type", "customer_service")
                .build())
            .build();

        List policies = vectorStore.search(request);

        // 2. 如果没有相关政策，明确告知
        if (policies.isEmpty()) {
            return new Answer("抱歉，当前知识库中没有找到相关的政策信息。请转接人工坐席获取帮助。",
                List.of(), false);
        }

        // 3. 回答并强制引用政策原文
        return chatClient.prompt()
            .system("你是一个客服政策助手。请基于提供的政策文档回答。" +
                "每次引述政策时必须标注政策编号和生效版本。")
            .user(query)
            .call()
            .entity(Answer.class);
    }
}

        String content = response.getResponse();
        List violations = new ArrayList<>();

        // 检查禁止承诺
        for (String word : FORBIDDEN_PROMISES) {

        }

        if (!violations.isEmpty()) {
            response.getAttributes().put("compliance_violations", violations);
            response.getAttributes().put("needs_review", true);
        }

        return response;
    }

@Service
public class DualEngineRouter {
    private final RuleEngine ruleEngine;
    private final IntentClassifier modelClassifier;
    private final MeterRegistry meterRegistry;

    public IntentResult route(String message) {
        // 1. 规则引擎优先
        IntentResult ruleResult = ruleEngine.match(message);
        if (ruleResult != null && ruleResult.confidence() >= 0.9) {
            meterRegistry.counter("intent.router", "engine", "rule").increment();
            return ruleResult;
        }

        // 2. 模型分类
        IntentResult modelResult = modelClassifier.classify(message);

        // 3. 模型与规则不一致时的仲裁
        if (ruleResult != null && modelResult.intent() != ruleResult.intent()) {
            // 记录不一致，用于后续分析
            meterRegistry.counter("intent.router", "engine", "conflict").increment();
            // 高置信度的一边获胜
            return modelResult.confidence() > 0.9 ? modelResult : ruleResult;
        }

        meterRegistry.counter("intent.router", "engine", "model").increment();
        return modelResult;
    }
}

java
@Service
public class IdempotentToolService {

    private final RedisTemplate redisTemplate;

    private static final long IDEMPOTENT_TTL_SECONDS = 3600; // 1 小时

    public  T executeWithIdempotency(
            String toolName,
            Map params,
            Supplier operation) {

        // 1. 生成幂等 Key
        String idempotencyKey = generateKey(toolName, params);

        // 2. Redis 分布式锁（防止并发执行两次）
        String lockKey = "lock:" + idempotencyKey;
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "executing",
                Duration.ofSeconds(30));

        if (Boolean.FALSE.equals(locked)) {
            // 另一个请求正在执行相同操作
            throw new IdempotentConflictException(
                "操作正在执行中，请稍后再试");
        }

        try {
            // 3. 检查是否已执行过
            String result = redisTemplate.opsForValue()
                .get("result:" + idempotencyKey);
            if (result != null) {
                return deserialize(result);
            }

            // 4. 执行工具
            T operationResult = operation.get();

            // 5. 缓存结果
            redisTemplate.opsForValue().set(
                "result:" + idempotencyKey,
                serialize(operationResult),
                Duration.ofSeconds(IDEMPOTENT_TTL_SECONDS));

            return operationResult;

        } finally {
            redisTemplate.delete(lockKey);
        }
    }

    private String generateKey(String toolName, Map params) {
        String paramStr = new TreeMap<>(params).entrySet().stream()
            .map(e -> e.getKey() + "=" + e.getValue())
            .collect(Collectors.joining("&"));
        return DigestUtils.md5Hex(toolName + ":" + paramStr);
    }
}

@Service
public class ResilientToolExecutor {

    private final RetryTemplate retryTemplate;
    private final MeterRegistry meterRegistry;

    public ResilientToolExecutor() {
        this.retryTemplate = RetryTemplate.builder()
            .maxAttempts(3)
            .exponentialBackoff(Duration.ofMillis(200), 2, Duration.ofSeconds(5))
            .retryOn(IOException.class, TimeoutException.class)
            .notRetryOn(IllegalArgumentException.class)  // 参数错误不重试
            .build();
    }

    public  T executeWithRetry(String toolName, Supplier operation) {
        try {
            return retryTemplate.execute(context -> {
                log.info("Tool {} attempt {}",
                    toolName, context.getRetryCount() + 1);
                return operation.get();
            });
        } catch (Exception e) {
            meterRegistry.counter("tool.failure",
                "tool", toolName,
                "error", e.getClass().getSimpleName())
                .increment();

            // 降级：返回错误消息供 LLM 使用
            throw new ToolExecutionException(
                "工具 " + toolName + " 执行失败：" + e.getMessage(),
                "请告诉用户：系统暂时无法完成这个操作，请稍后再试或联系人工客服。"
            );
        }
    }

            .set(AGENT_SESSION + agentId, pkg.sessionId(),

    }
