约 40 分钟 · 阶段：业务与架构 · 场景：S2 智能客服

为什么学这章

客服系统的核心能力是「记住用户说了什么、理解用户想做什么」——如果系统记不住上一轮对话的内容，用户每次都要重复描述问题，体验会急剧下降。CorpAssist 上线初期因为上下文管理不当，P50 对话轮次仅 2.3 轮（用户往往在第 3 轮就失去耐心转人工），上下文优化后提升到 5.1 轮，解决率提升了 14 个百分点。多轮会话管理是拉开智能客服和普通 FAQ 机器人的关键分水岭。

学完后能做什么

能用 Spring AI ChatMemory + Redis 实现生产级的对话历史持久化；能实现指代消解策略（用 LLM + 最近 N 轮上下文解决「这个」「那个」的指代问题）；能实现上下文摘要压缩将 20 轮历史从 4K Tokens 压缩到 600 Tokens；能设计三层混合记忆架构（工作记忆 + 会话记忆 + 长期画像），将用户偏好和画像注入对话上下文中。

本章要点速记

  • ChatMemory + Redis：Spring AI ChatMemory 接口 + Redis 持久化，TTL 30 分钟每次交互后重置，支持按 sessionId 读取和追加消息
  • 指代消解：System Prompt 约束 + 最近 N 轮上下文 + 专门的 CoreferenceResolution Prompt，准确率从 67% 提升到 89%
  • 摘要压缩：20 轮对话约 4K Tokens → LLM 摘要压缩到 600 Tokens（保留最近 2 轮完整 + 历史摘要），Token 节省 85%
  • 三层混合记忆：工作记忆（当前轮 + 工具结果）、会话记忆（N 轮历史 + 摘要）、长期画像（用户偏好/等级/历史交互），通过 Prompt 注入

Spring AI ChatMemory + Redis 持久化方案

Spring AI 的 ChatMemory 接口是构建多轮对话能力的基础设施。它抽象了会话历史的存储和读取操作，提供 add()（追加消息）、get()（按 sessionId 读取）、clear()（清理会话）三个核心方法。ChatMemory 有三种内置实现：InMemoryChatMemory（开发测试用，重启丢失）、JdbcChatMemory（基于 JDBC 的关系型存储）、RedisChatMemory（基于 Redis List 数据结构的生产级实现）。CorpAssist 在生产环境中使用 RedisChatMemory，因为它天然支持 TTL 过期、高并发读写（Redis 单实例 QPS 可达 10 万+）、以及数据持久化（AOF + RDB 混合持久化）。配置只需要一行：spring.ai.chat.memory.type=redis。会话的 TTL 设置为 30 分钟，每次用户发送新消息时重置过期时间，确保活跃会话不丢失而长时间不活跃的会话自动清理。在 Redis 中的存储结构是 chat_memory:{sessionId} 的 List，每个元素是一条 ChatMessage（包含 messageType、content、timestamp、metadata）。

生产环境中的实践教训值得注意。第一个问题是 ChatMemory 的大小控制：如果不加限制，极端情况下一个超长会话（50+ 轮）可能在 Redis 中存储数万 Tokens 的消息数据。我们的解决方案是在每次 add() 操作后检查 List 长度，如果超过 50 条消息则触发摘要压缩（将前 48 条摘要化，保留最新 2 条完整）。第二个问题是并发写入：当用户快速连续发送消息时，RedisChatMemory 的写入操作需要保证顺序。Spring AI 的实现已经使用 Redis 的 RPUSH 保证了追加顺序。第三个问题是序列化性能：默认的 JDK 序列化方式效率较低，我们切换到 Jackson JSON 序列化，消息大小减少了 60%（从平均 2.1KB 降到 0.8KB），Redis 内存占用显著降低。
