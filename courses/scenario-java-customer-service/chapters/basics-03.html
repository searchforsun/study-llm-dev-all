为什么学这章

智能客服与知识库的最大区别在于「多轮」——用户不会一句话问完，而是通过多轮对话逐步澄清需求。会话管理直接决定了用户体验：忘记上下文会让用户重复说，上下文太长会浪费 Token。

学完后能做什么

能使用 Spring AI ChatMemory + Redis 实现会话状态持久化；能配置上下文摘要压缩策略；能实现 Hybrid Memory（工作记忆 + 会话记忆 + 长期画像）；能处理指代消解的基本场景。

本章要点速记

  • ChatMemory 是 Spring AI 的会话管理接口，支持 Redis/Jdbc/InMemory 实现
  • 上下文摘要压缩：当 Token 超限时用 LLM 摘要替代完整历史
  • 指代消解通过 Prompt 模板 + 最近 N 轮上下文实现
  • Hybrid Memory 三层：工作记忆(当前轮) + 会话记忆(N轮) + 长期画像(用户偏好)

Spring ChatMemory + Redis 持久化

Spring AI 的 ChatMemory 接口提供了统一的会话管理抽象。生产环境建议使用 Redis 实现，支持分布式共享。

java
@Configuration
public class ChatMemoryConfig {

    @Bean
    public ChatMemory chatMemory(RedisConnectionFactory connectionFactory) {
        return new RedisChatMemory(connectionFactory);
    }
}

@Service
public class ConversationService {

    private final ChatMemory chatMemory;
    private final ChatClient chatClient;

    // 会话配置
    private static final int MAX_HISTORY_TOKENS = 4096;
    private static final int MAX_TURNS = 20;

    public Flux chat(String sessionId, String userMessage) {
        // 1. 获取或创建会话
        List history = chatMemory.get(sessionId, MAX_TURNS);

        // 2. 添加用户消息
        chatMemory.add(sessionId, new UserMessage(userMessage));

        // 3. 检查 Token 上限，触发摘要压缩
        if (countTokens(history) > MAX_HISTORY_TOKENS) {
            history = compressHistory(sessionId, history);
        }

        // 4. 构建 Prompt（system + history + user）
        return chatClient.prompt()
            .system(buildSystemPrompt())
            .messages(history)
            .user(userMessage)
            .stream()
            .content()
            .doOnComplete(() -> {
                // 完成后将 assistant 消息加入历史
                chatMemory.add(sessionId,
                    new AssistantMessage(lastResponse));
            });
    }

    // Token 计数
    private int countTokens(List messages) {
        return messages.stream()
            .mapToInt(m -> m.getText().length() / 2) // 中文约 2 char/token
            .sum();
    }

    private String buildSystemPrompt() {
        return """
            你是一个智能客服助手。请根据对话历史理解上下文。
            - 如果用户说「这个」或「它」，指代最近提到的物品/话题
            - 如果用户说「刚才说的那个」，回溯到上上轮的讨论
            - 保持回答简洁、友好
            """;
    }
}

Redis 存储结构：session:{sessionId}:messages -> List。TTL 设置为 30 分钟无交互自动过期。每次对话打 ID 后重置 TTL。

上下文摘要压缩

当对话历史超过 Token 上限时，不是简单截断（会丢失信息），而是用 LLM 生成摘要替代。

java
@Service
public class ContextCompressionService {

    private final ChatClient summaryClient;

    private static final String COMPRESSION_PROMPT = """
        请将以下客服对话浓缩为简洁的摘要，保留关键信息：
        - 用户的主要诉求
        - 已经提供的帮助或操作
        - 待解决的问题
        - 用户的身份/订单号等关键信息

        对话历史：
        {history}

        摘要（200字以内）：
        """;

    public List compress(String sessionId,
                                   List history) {
        String historyText = history.stream()
            .map(m -> m.getMessageType() + ": " + m.getText())
            .collect(Collectors.joining("\n"));

        // 用 LLM 生成摘要
        String summary = summaryClient.prompt()
            .user(u -> u.text(COMPRESSION_PROMPT)
                .param("history", historyText))
            .call()
            .content();

        // 只保留最近 2 轮完整对话 + 之前的摘要
        List compressed = new ArrayList<>();
        compressed.add(new SystemMessage("对话摘要: " + summary));

        // 保留最后 2 轮完整消息
        int keepTurns = 2;
        for (int i = Math.max(0, history.size() - keepTurns * 2);
             i < history.size(); i++) {
            compressed.add(history.get(i));
        }

        return compressed;
    }
}

压缩效率：一个 20 轮的完整对话约 4,000 Tokens，压缩后约 600 Tokens（摘要 150 + 最近 2 轮 450）。Token 节省约 85%。CompressService 使用独立的轻量模型（qwen-turbo）以降低成本。

java
@Service
public class HybridMemoryService {

    private final ChatMemory sessionMemory;  // Redis
    private final UserProfileRepository profileRepo;  // MongoDB

    public String buildSystemPrompt(String userId, String sessionId) {
        // 1. 长期画像
        UserProfile profile = profileRepo.findByUserId(userId)
            .orElse(new UserProfile(userId));

        // 2. 组装 Prompt 的不同记忆层
        StringBuilder sb = new StringBuilder();
        sb.append("你是一个智能客服助手。\n\n");

        // 长期画像
        if (profile.getMemberLevel() != null) {
            sb.append("用户信息：");
            sb.append("会员等级 ").append(profile.getMemberLevel());
            sb.append("，偏好语言 ").append(profile.getPreferredLang());
            if (profile.getRecentOrders() != null && !profile.getRecentOrders().isEmpty()) {
                sb.append("。最近订单：")
                    .append(profile.getRecentOrders().get(0).orderId());
            }
            sb.append("\n\n");
        }

        // 会话摘要（如果有）
        List history = sessionMemory.get(sessionId, MAX_TURNS);
        if (!history.isEmpty()) {
            sb.append("对话摘要：当前已完成 ")
                .append(history.size() / 2)
                .append(" 轮对话。\n\n");
        }

        sb.append("回答要求：基于历史理解上下文，如果用户有指代（这个/它/那个），根据最近对话推断。");

        return sb.toString();
    }
}

面试可能问

Q1: 多轮对话 Token 超限怎么处理？摘要压缩会丢失信息吗？

三种策略：① 摘要压缩——用 LLM 生成历史摘要保留关键信息，实验表明回答质量下降 < 5%；② 滑动窗口——只保留最近 N 轮完整对话，丢弃最早的部分；③ 混合——前 10 轮做摘要 + 最后 3 轮完整保留。CorpAssist 优先用混合策略。摘要会丢失一些细节（如用户最早提到的信息），但如果用户需要回溯，可以通过「你说的是刚才提到的那个？」来澄清。
