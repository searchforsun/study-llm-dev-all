Q1: 设计智能客服系统需要哪些核心模块？怎么划分？

面试官问：你来设计一个智能客服系统的模块划分，说明每个模块的职责和数据流。如果使用 Spring 技术栈，怎么选型？

智能客服系统按照「职责分离 + 异步解耦」的原则划分为六个微服务和五个核心模块。六个微服务分别是：Gateway（Spring Cloud Gateway，负责统一入口、鉴权限流和协议转换）、Intent Service（意图路由，负责规则引擎 + LLM 双引擎意图分类，无状态）、Conversation Service（对话引擎，管理 ChatClient + ChatMemory 的多轮对话，有状态）、Tool Service（工具层，注册 @Tool 业务工具，负责参数校验和幂等控制）、Agent Service（人工坐席模块，管理转人工队列 WebSocket 和坐席工作台）、Quality Service（离线质检模块，ComplianceAdvisor 异步审核和 CSAT 分析）。Gateway 作为统一入口，三层限流（IP 级别 10 req/s、用户级别 30 req/min、LLM 连接池 20 并发），P99 转发延迟 < 5ms。模块间数据流遵循责任链原则：消息 -> Intent Service 分类 -> Conversation Service（可调用 Tool Service）-> 质检通过后返回 -> 低置信度转 Agent Service。

关键技术选型：Spring AI 的 ChatMemory + Redis 管理会话状态、@Tool 注解注册业务工具（自动生成 JSON Schema）、ReactAgent 实现 Think-Act-Observe 循环（maxSteps=5）。模块间的同步通信使用 HTTP/2 + gRPC（内部调用 P99 < 50ms），异步通信使用 RocketMQ 事件（HandoffEvent、ToolResultEvent、EscalationEvent）。微服务部署在 K8s 集群，每个服务 2-3 个 Pod，有状态服务（Conversation、Agent）使用 StatefulSet 保证网络标识稳定，无状态服务（Intent、Tool）使用 Deployment。为什么 Conversation Service 要有状态？因为它的 ChatMemory 关联了 Redis 中的会话数据，虽然 Redis 是外部存储，但服务实例需要稳定的 Pod 名称来维护 WebSocket 连接和其他长连接状态。日均支撑 5 万对话，P99 端到端延迟 3.2s，可用性 99.95%。

Q2: 多轮对话如何管理上下文和指代消解？

面试官问：多轮对话怎么管理上下文？Token 超限怎么办？指代消解怎么处理？Session 之间如何共享信息？

多轮对话的上下文管理采用三层混合记忆架构。工作记忆（Working Memory）存储当前轮次的上下文——用户当前消息、正在进行的工具调用结果、临时变量，容量约 3K Tokens，存储在内存中随请求销毁。会话记忆（Session Memory）通过 Spring AI 的 ChatMemory + Redis 持久化，存储完整对话历史，TTL 30 分钟每次交互后重置。长期画像（Long-term Profile）存储用户级别的持久化信息——用户姓名、VIP 等级、历史交互记录，存储在 PostgreSQL 中永不过期，每次对话开始时通过 userId 加载注入 System Prompt。三层记忆在构建 Prompt 时按「长期画像 > 会话记忆摘要 > 工作记忆 > 当前消息」的优先级组合。当用户第 10 次找客服时，系统可以在对话开始就说「张先生您好，上次您的退货问题已经处理了，请问这次还有什么需要帮助的吗？」——这种个性化体验能显著提升 CSAT。

Token 超限的应对策略是 LLM 摘要压缩 + 滑动窗口。每 10 轮对话触发一次摘要压缩——将前 8 轮压缩为约 200 字的 LLM 摘要，保留最近 2 轮完整对话。20 轮上下文从约 6.4K Tokens 压缩到约 900 Tokens（节省 86%）。关键节点（订单号创建、退款金额确认）在压缩时被单独保留。指代消解使用三层策略：第一层在 System Prompt 中要求 LLM 注意代词指代；第二层确保 ChatMemory 提供至少最近 2 轮完整上下文；第三层在长对话中启用 CoreferenceResolver 预处理——将代词用 LLM 替换为具体名词短语。三层叠加后指代准确率从 67% 提升到 89%。线上调用指代消解的额外 Token 成本约 150 Tokens/次，但仅对 > 3 轮的对话启用，所以平均每次对话仅多消耗 30 Tokens 左右。

java 
// 面试可写的精简版上下文管理代码
@Service
public class ConversationManager {
    private final ChatMemory chatMemory;      // Redis
    private final UserProfileRepo profileRepo; // PostgreSQL
    private final ChatClient summaryClient;

    public String buildContext(String sessionId, String userId, String message) {
        UserProfile profile = profileRepo.findByUserId(userId);
        List history = chatMemory.get(sessionId, 50);
        String summary = history.size() > 20
            ? compressSummary(history) : "";
        StringBuilder ctx = new StringBuilder();
        ctx.append("[画像] ").append(profile.name()).append("\n");
        ctx.append("[摘要] ").append(summary).append("\n");
        history.subList(Math.max(0, history.size()-4), history.size())
               .forEach(m -> ctx.append(m.getContent()).append("\n"));
        return ctx.toString();
    }

    private String compressSummary(List history) {
        String text = history.subList(0, history.size()-4).stream()
                .map(m -> m.getContent()).collect(Collectors.joining("\n"));
        return summaryClient.prompt()
                .user("摘要以下对话: " + text).call().content();
    }
}

Q3: 模型答不上来时有哪些降级策略？优先级如何排？

面试官问：LLM 不可用或者回答质量低时，你们的降级策略是什么？熔断和降级的区别？

降级策略按照系统受影响的程度分为四级。L0（正常运行）——LLM API 正常、置信度 >= 0.8、工具调用正常，使用完整 Agent + LLM + SSE 流式流程，占线上流量的 87%。L1（轻度降级）——LLM 延迟 > 5s 或单次工具调用超时时触发，系统自动切换到规则引擎 + 预设话术模式，30 秒后自动重试 LLM，占 8%。L2（转接降级）——置信度 < 0.3、Agent 超 maxSteps=5、用户要求转人工或 COMPLAINT 敏感场景时触发，收集完整 HandoffContext 发送到人工坐席队列，占 4.5%。L3（熔断降级）——LLM API 连续 3 次健康检查失败或错误率 > 50%，返回标准话术「系统维护中」，30 秒后尝试 Half-Open 恢复，占 0.5%。降级和熔断的区别：降级是「功能裁剪」（从 LLM 降到规则引擎），系统仍在运行但能力减弱；熔断是「完全断开」（不再调用 LLM API），系统进入维护模式。降级是软切换（延迟高就降级、恢复就升回来），熔断是有状态切换（Closed -> Open -> Half-Open -> Closed）。

四级降级的切换逻辑在 Gateway 层和 Conversation Service 层协同执行。Gateway 层监控 LLM API 的健康状态——如果连续 3 次健康检查失败，在 Gateway 层直接返回 L3 熔断话术，流量不进入下游服务。Conversation Service 层在每次 Agent 执行前检查降级级别，如果为 L1 则绕过 Agent 直接使用规则引擎回复。降级级别的切换是自动的，但在 Dashboard 上标记当前降级状态并发送告警。切换必须是平滑的——用户应该感受到的差异只是「回复变慢了」或「提示转人工了」，而不是「系统崩了」。CorpAssist 的降级切换耗时不超过 2s（通过健康检查的间隔配置），用户基本无感知。

Q4: 如何保证回答符合公司政策且不会产生合规风险？

面试官问：客服回答必须符合公司政策，否则可能产生法律风险。你们怎么保证机器人不乱承诺？政策文档更新了怎么办？

合规保障体系分三层构建。第一层是政策库 RAG——所有涉及政策的回答必须基于从向量库检索到的最新文档生成。政策文档以 Markdown 格式维护在 Git 仓库中，CI/CD 流水线自动解析分割为 500 字符的文本块（overlap=50），生成 1024 维 Embedding 写入 Milvus 向量库，每个文档块标注版本号和生效日期。检索时使用 FilterExpression 只查 status=active 的最新版本。Top-K=3、相似度阈值 0.75——低于该阈值的认为不相关直接转人工。第二层是 ComplianceAdvisor——在 LLM 输出返回到用户之前进行双层审核。规则层使用 Aho-Corasick 算法在 1ms 内匹配 30+ 禁止承诺词（「一定退款」「保证赔偿」等），匹配即拦截替换为合规话术。模型层使用独立的 Qwen-72B（Temperature=0.1）异步检测政策冲突。规则层拦截率 100%，模型层准确率 96%。第三层是敏感场景检测——COMPLAINT、LEGAL、CLAIM 三类意图强制转人工。三层叠加确保了系统的合规底线。过去 6 个月：合规事故从月均 2 起降到 0 起。

政策文档版本管理是合规体系的基础保障。GitLab MR + 法务审批 -> Merge 到 main -> CI/CD 触发向量化流水线 -> upsert 写入 Milvus（新版本 status=active，旧版本 status=archived）。从 MR 合并到系统生效约 5 分钟。紧急撤回可使用 ops 脚本删除指定文档的 Milvus 记录，5 秒内生效。每周围绕合规拦截数据召开合规回顾会议，更新禁止词列表。合规误拦率从初期的 15% 降到 4%（通过添加白名单 + 上下文敏感匹配优化）。为什么合规误拦率不能降到 0%？因为「安全优先宁可误拦」的原则意味着一定比例的假阳性是可接受的。4% 的误拦率在安全性和用户体验之间达到了合理的平衡。

Q5: 用什么指标评估智能客服效果？怎么持续提升？

面试官问：你怎么评估智能客服系统的好坏？线上监控哪些指标？怎么持续优化？

评估体系分为三层共 11 项关键指标。业务层指标直接反映系统的业务价值——解决率（>= 85%）、转人工率（<= 25%）、CSAT（>= 4.2）、首答时间（P99 <= 5s）。技术层指标反映系统的技术健康度——意图识别准确率（>= 93%）、工具调用成功率（>= 99%）、P99 端到端延迟（<= 3.5s）、Agent 平均步数（<= 3）。质量层指标反映系统的回答质量——引用覆盖率（>= 95%，回答中引用政策文档的比例）、合规拦截数、人工抽检合格率（>= 95%）。这三层指标有因果关联：技术层的意图准确率和工具成功率直接影响业务层的解决率和 CSAT。例如，意图准确率从 88% 提升到 94%（+6pp）直接带来了解决率从 70% 到 76%（+6pp）的等比例提升。

持续优化的闭环是：监控 -> 发现异常 -> 归因 -> 修复 -> 回归 -> 验证。每个变更先跑 7 天 A/B 实验（10% 流量），对比实验组和对照组的 5 个核心指标，p < 0.05 才全量。每周人工抽检 200 条对话计算准确率，发现新问题加入坏例库（目前 350 条）。坏例库在 CI/CD 中自动运行回归测试，通过率低于 95% 阻断发布。CorpAssist 上线 6 个月的优化历程：解决率 70%->85%（+15pp）、转人工率 35%->22%（-13pp）、CSAT 3.7->4.2（+0.5）、P99 延迟 5.8s->3.2s（-45%）。当前持续优化的重点是减少合规误拦截（目标从 4% 降到 2% 以下）和提升复杂多轮对话的 Agent 成功率。特别说明一下 P99 延迟降低 45% 的关键措施：主要是将 Agent 的 maxSteps 从 10 调整为 5，以及将 Gateway 连接池从 10 扩容到 20——两者加起来贡献了约 80% 的延迟改善。

本章结论

  • Q1 模块划分：六服务（Gateway + Intent + Conversation + Tool + Agent + Quality）+ 五模块 + 事件驱动通信。Spring Cloud Gateway 三层限流，日均 5 万对话 P99 3.2s，可用性 99.95%。
  • Q2 多轮对话：三层混合记忆（工作/会话/长期）+ 摘要压缩（节省 86% Tokens）+ 指代消解准确率 89%。TTL 30 分钟 + 用户画像永不过期。核心是每 10 轮触发压缩，保留最近 2 轮完整。
  • Q3 降级策略：四级降级 L0(87%)->L1(8%)->L2(4.5%)->L3(0.5%)。降级是功能裁剪、熔断是完全断开。30 秒半开恢复状态机，切换对用户透明。
  • Q4 合规保障：三层体系（RAG + ComplianceAdvisor 双层审核 + 敏感场景升级）。规则拦截率 100%，模型准确率 96%，误拦率从 15% 降到 4%。政策文档从 GitLab MR 到系统生效约 5 分钟。
  • Q5 效果评估：三层指标（业务/技术/质量）共 11 项，A/B 实验 + 周抽检 200 条 + 坏例库 350 条回归。优化 6 个月：解决率 +15pp、转人工率 -13pp、CSAT +0.5、P99 延迟 -45%。

面试可能问（进阶）

Q6: 你的系统和普通的 FAQ 机器人本质区别在哪？

本质区别在于「理解能力和自主决策能力」。FAQ 机器人基于固定规则或者关键词匹配——用户问「怎么退货」就匹配到「退货流程」的预设答案，不关心上下文、不问用户的细节、也记不住之前说了什么。而智能客服系统具有三个 FAQ 机器人不具备的能力。第一是意图理解和推理——系统能理解用户说「这个不要了」是在表达退货意图（83% 的准确率），甚至能区分「退货运费谁出」（政策咨询）和「我要退货」（操作请求）之间的细微差别。第二是多轮上下文感知——系统记住用户在第一轮提到的订单号，第二轮说「它什么时候到」时知道「它」指代的是哪个订单，不需要用户重复。第三是自主行动能力——Agent 可以自主决定调用订单系统查物流、调用售后系统创建退款工单，并组合多次操作的结果回复用户。FAQ 机器人只能「说」，智能客服系统不仅「说」还能「做」。CorpAssist 的数据显示，普通 FAQ 解决率通常在 40-50%，而智能客服系统可以达到 85%。

Q7: 如果用户用方言或者输入有错别字，你们的系统怎么处理？

智能客服系统天然具备对非标准输入的处理能力。第一层是 LLM 的语言理解优势——大语言模型在训练过程中接触过大量中文语料（包括方言表达和错别字），天然具有纠错和容错能力。例如，用户说「窝的订单到哪了」（别字），LLM 能理解「窝」应该是「我」。第二层是我们在系统层面做的补充处理——在用户消息进入 Agent 之前，先经过一个「消息标准化」步骤：敏感信息脱敏（手机号/身份证号替换为占位符）、URL 提取（将短链接转为完整链接）、明显的错别字纠正（使用一个轻量级的拼音-汉字映射表）。但要注意不要过度「标准化」——方言表达中有些是故意的（如「咋整」「啥时候到」），过度纠正会丢失语义和语气。第三层是在知识库检索中对非标准输入做 Query Expansion——用户的查询词「咋退货」被扩展为「退货」「退货流程」「退货申请」「怎么退货」等 5 个变体去向量库检索，提高命中率。线上数据表明，非标准输入（错别字/方言/口语化表达）约占总消息的 12%，经过这三层处理后，系统的理解和回答准确率与非标准输入之外的场景差距不到 3%。

Q8: 你在做这个系统的过程中犯过最大的错误是什么？学到了什么？

最大的错误是第一次上线前没有做压力测试。当时觉得开发环境自测都通过了，直接全量发布了。结果上线第一个小时，并发 150 用户就出了问题——LLM API 连接池只有 10 个连接，大量请求排队等待导致 P99 延迟从 2s 飙升到 30s。更糟糕的是，排队中的请求超时后不断重试，形成了「重试风暴」，LLM API 的错误率瞬间达到 60%，触发了熔断，整个系统进入了 L3 降级状态，所有用户都只能看到「系统维护中」。这个事故持续了 45 分钟才恢复，导致当天 CSAT 从 4.2 降到了 2.8，还有 3 个客户投诉。这次教训让我学到了三件事：第一，压力测试必须是上线前 Checklist 的硬性要求项，不能省略——后来我们做了 200 并发压力测试才让上线通过。第二，限流熔断的阈值和超时时间必须根据压力测试结果来设定，而不是拍脑袋设的——10 个连接显然太少，20 个连接是我们后来通过压力测试找出的最优值。第三，要有「优雅降级」而不是「全面崩溃」——在 Gateway 层做限流的目的是让大部分用户正常使用、小部分用户被限流（友好的 429），而不是让所有用户都看到系统崩溃。这个错误之后，我主导制定了 CorpAssist 上线 12 项 Checklist，把压力测试作为第 7 项硬性条件。

判断练习

场景题：面试官问「你的系统如果遇到 LLM API 完全不可用怎么办？」——你怎么回答能体现出系统的鲁棒性？

首先明确说明系统设计了 L3 熔断降级来应对这种情况。LLM API 连续 3 次健康检查失败或错误率 > 50% 时触发熔断，熔断后返回标准话术「系统维护中」，30 秒后自动切换到 Half-Open 尝试 1 次 LLM 调用，成功则关闭熔断器回到正常模式。但这是被动策略，我们还有主动策略：多模型备选——主模型 Qwen-72B 不可用时自动切换到备选模型（如 GPT-4o-mini 或本地部署的 Qwen-14B），切换过程对用户透明，延迟增加约 1-2s 但服务不中断。备选模型的成本是主模型的 3-5 倍，所以不能长期使用，但 15 分钟内可以保证服务不中断。如果备选模型也不可用（极端情况），引擎完全切换到规则引擎 + 预设话术模式，虽然能力大幅下降（解决率约 40%），但至少用户可以收到回复而不是空白。这种「主 -> 备 -> 兜底」的多层方案确保了系统在极端情况下的基本可用性。此外，我们还有离线缓存策略——将最高频的 200 个 FAQ 的 LLM 回复缓存到本地，在网络完全中断时这些缓存可以直接返回。

场景题：面试官问「你的效果评估体系中的引用覆盖率 96% 是怎么计算的？太低或者太高分别意味着什么？」

引用覆盖率的计算方法是：在所有涉及政策咨询的对话中（意图为 POLICY_QUERY 或对话中触发了政策 RAG 检索），人工检查 Agent 的回复是否明确引用了检索到的政策文档（如在回答中包含「根据《退货政策 V3.2》第 4.1 条」这样的引用）。回复中包含引用标记的计入「已覆盖」，不包含的计入「未覆盖」。覆盖率 = 已覆盖 / (已覆盖 + 未覆盖) x 100%。CorpAssist 的 96% 表明在政策相关对话中，Agent 在 96% 的情况下正确引用了政策来源。引用覆盖率太低（如低于 80%）说明 Agent 经常「自由发挥」——回复没有基于检索到的政策文档，这可能意味着 System Prompt 中的引用指令不够明确、或者 Agent 跳过检索直接凭记忆回复了。引用覆盖率太高（如 100%）也不一定好——有可能 Agent 在不需要引用的时候（如问候语「你好」）也试图引用文档，这属于过度引用。理想范围是 95-98%——政策相关的回答引用文档、非政策性的问候不需要引用。我们定期抽检未覆盖的 4%，分析是 Agent 忘了引用还是确实不需要引用，对前者调整 Prompt，对后者调低覆盖率分母。
