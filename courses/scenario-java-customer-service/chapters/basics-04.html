为什么学这章

智能客服最怕「乱承诺」——如果 LLM 告诉用户「可以全额退款」但公司政策允许的是部分退款，公司就会面临法律风险和财务损失。CorpAssist 上线第三天就出现过一次合规事故：LLM 在对话中承诺了「必定退款」，但实际该用户的购买时间已超过退货期限，导致用户投诉升级。从那以后我们建立了全链路合规体系。这章的核心就是教你如何用 Spring AI 的 RAG、MessageTemplate 和 Advisor 构建合规且可控的话术生成流水线。

学完后能做什么

能使用 Spring AI 的 VectorStore 实现政策库 RAG，确保每次回答都基于最新政策文档；能使用 MessageTemplate 实现变量注入的话术模板，避免 LLM 自由发挥导致的风险；能实现 ComplianceAdvisor 在输出前检查禁止承诺词和政策冲突；能设计敏感场景的自动识别与升级策略，在检测到投诉/索赔/法律关键词时自动转人工。

本章要点速记

  • 政策库 RAG：Spring AI VectorStore（Milvus/PgVector）检索最新政策文档，FilterExpression 只查 active 版本，Top-K = 3，SimilarityThreshold >= 0.75
  • MessageTemplate：基于 Mustache 模板引擎的话术模板，变量注入（userName、productName）确保话术一致性，替代 LLM 自由生成
  • ComplianceAdvisor：Spring AI Advisor 接口实现输出后处理，检测禁止承诺词列表（「一定退款」「保证赔偿」等 30+ 词语），匹配即拦截转人工
  • 敏感场景升级：COMPLAINT/LEGAL/CLAIM 三类敏感意图自动转人工，禁止承诺词拦截率 100%，政策冲突检测准确率 96%

政策库 RAG：Spring AI 检索增强话术生成

政策库 RAG 是确保客服回答合规的基石。其核心思想是：LLM 不直接根据训练数据中的「记忆」回答政策问题，而是从公司维护的最新政策文档库中检索相关内容，生成基于检索结果的回答。Spring AI 提供了完整的 RAG 支持链：文档解析（DocumentReader）-> 文档分割（DocumentSplitter）-> 向量化嵌入（EmbeddingModel）-> 向量存储（VectorStore）-> 检索增强（QueryAugmenter）-> 生成（ChatClient）。在 CorpAssist 中，政策文档以 Markdown 格式维护在 Git 仓库中（文档团队通过 GitLab MR 更新），CI/CD 流水线自动解析、分割、向量化并写入 Milvus 向量数据库。每次更新都会增加版本号，FilterExpression 确保只检索 status=active 的最新版本。

检索质量直接影响回答的准确性。CorpAssist 经过 500 组测试调优后确定了最优参数组合：检索 Top-K = 3（返回 3 个最相关的政策片段）；相似度阈值 SimilarityThreshold = 0.75（低于此阈值的片段认为不相关，不提供给 LLM）；检索时使用 Hybrid Search（Dense Embedding + BM25 稀疏检索），在政策文档的标题和正文上分别加权（标题权重 * 2.0，正文权重 * 1.0）。如果检索结果为空（所有片段相似度低于 0.75），系统不会让 LLM 自由发挥，而是返回标准话术「关于这个问题，我暂时没有查到确切的政策信息，已为您转接人工坐席处理」并进行转人工。线上数据显示，政策相关问题中使用 RAG 的回答准确率为 96.8%（人工抽检 500 条），而纯 LLM 记忆（无 RAG）的回答准确率仅为 72.3%。

java
// 政策 RAG 的完整配置和实现
@Configuration
public class PolicyRagConfig {

    @Bean
    public VectorStore policyVectorStore(EmbeddingModel embeddingModel) {
        return MilvusVectorStore.builder(embeddingModel)
                .collectionName("corpassist_policies")
                .dimension(1024)
                .indexType(IndexType.IVF_FLAT)
                .metricType(MetricType.COSINE)
                .build();
    }

    @Bean
    public QueryAugmenter policyAugmenter() {
        return QueryAugmenter.builder()
                .allowEmptyContext(false)
                .emptyContextMessage("关于这个问题，我暂时没有查到确切的政策信息，已为您转接人工坐席处理。")
                .build();
    }
}

@Service
public class PolicyQAService {

    private final VectorStore vectorStore;
    private final ChatClient chatClient;
    private static final double SIMILARITY_THRESHOLD = 0.75;
    private static final int TOP_K = 3;

    public PolicyQAService(VectorStore vectorStore, ChatClient.Builder chatBuilder) {
        this.vectorStore = vectorStore;
        this.chatClient = chatBuilder
                .defaultSystem("""
                    你是 CorpAssist 客服政策顾问。请基于以下检索到的政策文档回答。
                    注意：
                    1. 只回答检索文档中包含的信息，不要自行推断政策内容
                    2. 如果检索文档不包含用户问题的答案，请说「该问题需要人工坐席确认」
                    3. 回答时列出引用的政策文档名称和条款编号
                    4. 不要做出「一定」「保证」「肯定」等绝对化承诺
                    """)
                .build();
    }

    public String answerPolicyQuestion(String question) {
        List docs = vectorStore.similaritySearch(
                SearchRequest.query(question)
                        .withTopK(TOP_K)
                        .withSimilarityThreshold(SIMILARITY_THRESHOLD));

        if (docs.isEmpty()) {
            return "关于这个问题，我暂时没有查到确切的政策信息，已为您转接人工坐席处理。";
        }

        String context = docs.stream()
                .map(d -> String.format("[%s] 条款%s: %s",
                        d.getMetadata().get("documentName"),
                        d.getMetadata().get("clauseId"),
                        d.getContent()))
                .collect(Collectors.joining("\n\n"));

        return chatClient.prompt()
                .user(u -> u.text("用户问题: " + question + "\n\n检索到的政策:\n" + context))
                .call()
                .content();
    }
}

MessageTemplate 话术模板与变量注入

尽管 LLM 擅长生成自然语言，但在客服场景中，许多回复应该使用标准化的模板而非自由生成——例如欢迎语、转人工通知、等待提示、对话结束语。这些场景使用 MessageTemplate 比 LLM 自由生成更可控、更符合品牌调性。Spring AI 的 MessageTemplate 基于 Mustache 模板引擎，支持变量注入（如 userName、productName、vipLevel）和条件判断（如 #if vip 分支）。模板文件存储在 classpath:/templates/cs/ 目录下，以 .mustache 为后缀。在 CorpAssist 的实践中，我们将系统回复分为三类：模板回复（用于欢迎、转接、错误、结束等固定场景——约 20 个模板）、增强回复（RAG 检索 + 模板组合——以检索结果为 body，模板为 wrapper）、自由生成（复杂问题需要 LLM 推理）。线上数据显示，约 40% 的对话使用了模板回复（首句问候、转人工通知等），30% 使用了增强回复，只有 30% 需要 LLM 自由生成。

模板设计的最佳实践是保持「人性化但可控」。例如「转人工通知」模板：userName 您好，您的问题已经转接给 agentName 处理，预计等待时间约 estimatedWaitMinutes 分钟——包含了用户名称、坐席名称和等待时间三个变量。变量的值来自上游模块：userName 从 JWT Token 的 Claims 中提取，agentName 从坐席分配模块返回，estimatedWaitMinutes 从 Redis 队列的计算结果得出。所有模板变量的值在注入前都会经过「脱敏处理」——手机号显示为 138****1234、银行卡号只显示后 4 位。模板变更通过 Git 管理，走正常的代码审查流程，确保每次模板更新都有记录可追溯。

java
// 话术模板加载和渲染服务
@Service
public class MessageTemplateService {

    private final Map templateCache = new ConcurrentHashMap<>();
    private final MustacheFactory mustacheFactory = new DefaultMustacheFactory();

    public MessageTemplateService() {
        loadTemplate("welcome", "templates/cs/welcome.mustache");
        loadTemplate("handoff-notify", "templates/cs/handoff-notify.mustache");
        loadTemplate("wait-notify", "templates/cs/wait-notify.mustache");
        loadTemplate("confirmation", "templates/cs/confirmation.mustache");
        loadTemplate("escalation", "templates/cs/escalation.mustache");
    }

    private void loadTemplate(String name, String path) {
        Reader reader = new InputStreamReader(
                getClass().getClassLoader().getResourceAsStream(path));
        templateCache.put(name, mustacheFactory.compile(reader, name));
    }

    public String render(String templateName, Map variables) {
        Mustache template = templateCache.get(templateName);
        if (template == null) {
            throw new IllegalArgumentException("Unknown template: " + templateName);
        }
        sanitizeVariables(variables);
        StringWriter writer = new StringWriter();
        template.execute(writer, variables);
        return writer.toString();
    }

    private void sanitizeVariables(Map variables) {
        if (variables.containsKey("phoneNumber")) {
            String phone = (String) variables.get("phoneNumber");
            variables.put("phoneNumber", phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2"));
        }
    }
}

ComplianceAdvisor 输出内容安全审核链

ComplianceAdvisor 是 CorpAssist 合规体系的核心防线。它是一个实现了 Spring AI 的 Advisor 接口的组件，在 LLM 生成回复之后、回复返回用户之前执行审核逻辑。Advisor 的 aroundStream() 方法拦截了 LLM 的流式输出流，对完整回复内容进行规则和模型两层审核。规则层使用高效的 Aho-Corasick 自动机算法匹配 30+ 个禁止承诺词和短语（包含「一定退款」「保证赔偿」「肯定到账」「无条件」「免费赠送」等），匹配即拦截。模型层使用独立的审核 LLM（Qwen-72B，使用更低温度 0.1）对回复进行政策冲突检测——判断回复内容是否与检索到的政策文档存在不一致。两层审核的结果做 OR 判断：任一层触发拦截，回复结果被替换为标准转人工话术并发送 EscalationEvent。

性能方面，规则层的 Aho-Corasick 匹配可以在 1ms 内完成（30 个词在 500 字文本中扫描），模型层审核需要额外的 LLM 调用（约 500ms-1.5s）。为兼顾延迟和准确性，CorpAssist 采用了两阶段策略：先做规则审核（低延迟），通过后再做模型审核（高准确率）。规则审核未通过可以节省模型审核的 Token 开销。生产数据：规则层拦截率 100%（只要包含禁止承诺词就触发）、模型层政策冲突检测准确率 96%（人工抽样 200 条，假阳性率 4%）。被拦截的回复自动记录到合规日志，送人工审核团队二次确认。每周围绕拦截数据召开合规回顾会议，更新禁止词列表。ComplianceAdvisor 的一个关键设计点是「阻断不阻塞」——审核过程不会阻断 LLM 的返回流，规则拦截会立即替换回复，模型审核异步进行只在发现冲突时告警。

java
// 合规审核 Advisor — 完整的双阶段实现
@Component
public class ComplianceAdvisor implements StreamAroundAdvisor {

    private static final List FORBIDDEN_PHRASES = List.of(
            "一定退款", "保证赔偿", "肯定到账", "无条件", "免费赠送",
            "绝对没问题", "百分百", "包退", "包赔", "承诺退款",
            "马上到账", "快速退款", "无需审核", "自动批准"
    );
    private static final AhoCorasickMatcher MATCHER =
            AhoCorasickMatcher.builder(FORBIDDEN_PHRASES).build();

    private final ChatClient auditClient;

    public ComplianceAdvisor(ChatClient.Builder builder) {
        this.auditClient = builder
                .defaultSystem("""
                    你是一个政策合规审核助手。给定用户的原始问题和模型生成的回复，
                    检查回复是否与以下政策文档冲突。如果回复中包含未在政策文档中
                    提及的承诺、优惠或条件，标记为 POLICY_CONFLICT。
                    只输出 JSON: {"result":"PASS|FLAG","conflict":"说明"}
                    """)
                .build();
    }

    @Override
    public String getName() { return "ComplianceAdvisor"; }

    @Override
    public int getOrder() { return 0; }

    @Override
    public Flux aroundStream(AdvisedRequest request, StreamAroundAdvisorChain chain) {
        List collected = new ArrayList<>();
        return chain.nextAroundStream(request)
                .doOnNext(collected::add)
                .collectList()
                .flatMapMany(generations -> {
                    String fullResponse = generations.stream()
                            .map(g -> g.getOutput().getContent())
                            .collect(Collectors.joining());
                    // 阶段1: 规则审核（1ms）
                    boolean hasForbidden = MATCHER.containsMatch(fullResponse);
                    if (hasForbidden) {
                        log.warn("=1= Compliance violation (rule): {}", fullResponse);
                        publishEscalation(request, fullResponse, "FORBIDDEN_PHRASE");
                        return Flux.just(new Generation(policyAugmenter
                                .getEmptyContextMessage()));
                    }
                    // 阶段2: 模型审核（异步，不阻塞返回流）
                    auditAsync(request, fullResponse);
                    return Flux.fromIterable(generations);
                });
    }

    private void auditAsync(AdvisedRequest request, String response) {
        CompletableFuture.runAsync(() -> {
            String result = auditClient.prompt()
                    .user(u -> u.text("用户问题: " + request.userText()
                            + "\n模型回复: " + response))
                    .call().content();
            if (result.contains("POLICY_CONFLICT")) {
                log.warn("=2= Policy conflict detected (async): {}", result);
                eventBridge.send("cs.escalation",
                        new EscalationEvent(request.userText(), response));
            }
        });
    }
}

敏感场景的自动识别与升级策略

某些客服场景无论 LLM 的回复多么完美，公司政策也要求必须由人工坐席处理。这些「敏感场景」包括：投诉（COMPLAINT，用户表达不满）、法律纠纷（LEGAL，提到律师/起诉/法律）、索赔（CLAIM，要求经济赔偿）、安全事件（SECURITY，账号被盗/密码泄露）、心理健康（MENTAL，表现出极端情绪或自残倾向）。CorpAssist 实现了一个「敏感场景检测器」——它在意图识别阶段之后、对话引擎之前运行，基于意图 + 关键词 + 情感分析的综合判断决定是否强制转人工。具体规则是：如果意图分类器输出 COMPLAINT 或自定义的 LEGAL 意图，无论置信度多高都直接转人工；如果用户消息中包含「起诉」「律师」「315」「投诉工商」等关键词，即使意图为其他类别，也触发敏感标记；如果情感分析模型检测到用户情绪为「愤怒」或「极度不满」，且置信度 > 0.8，则升级转人工。

升级策略的执行不是简单转人工，而是根据敏感类型采取不同的处理路径。对于 COMPLAINT 意图：转人工 + 将对话标记为「投诉工单」+ 通知值班主管。对于 LEGAL 意图：转人工 + 通知法务部门 + 对话全文录音归档。对于 CLAIM 意图：转人工 + 附加用户的历史订单信息和消费记录。在 CorpAssist 的线上数据中，敏感场景约占全部对话的 5%-8%（取决于业务类型），其中约 60% 被自动正确识别并转人工，30% 通过用户明确要求转人工处理，10% 需要人工在事后回检中发现。敏感场景检测的准确率目标为 >= 95%（召回率优先于精确率——宁可误转不可漏转）。以下表格给出了三类敏感场景的具体识别规则和处理路径。

CorpAssist 敏感场景升级案例

用户发消息：「你们什么破公司！订单 #ORD-20240601 都两周了还没收到，我要去工商局投诉你们！」系统通过意图分类器识别出 COMPLAINT 意图（置信度 0.92），同时关键词检测器命中「投诉工商」。敏感场景检测器触发强制升级，对话状态标记为「投诉工单」，转人工的同时通知了值班主管用户 ID 为 u1001 的投诉事件。Agent Workspace 在坐席工作台展示完整的对话历史（含情感分析标签「愤怒」）和用户的历史订单数据。坐席打开对话时说：「张先生您好，我是值班主管王经理。关于您订单 #ORD-20240601 的问题我已经了解了，现在由我直接为您处理，请您放心。」——这样的响应有效缓解了用户情绪，同时也符合公司的投诉处理 SOP。

敏感场景检测上线后的效果数据：在 3 个月的运行周期中，系统共识别出 2,847 个敏感场景，其中 2,721 个正确转人工（正确率 95.6%），126 个为误报（假阳性率 4.4%）。在 126 个误报中，75 个是用户使用了投诉相关的词汇但实际不是投诉（如「这个手机比我之前的好太多了，我投诉以前那个」），51 个是情感分析模型对用户激动情绪的过度敏感。我们对误报案例进行了分类改进——为投诉关键词添加了上下文豁免规则（如「投诉」前面有「比」或「跟...比」时不触发），将假阳性率从 4.4% 降到 2.8%。

本章结论

  • 政策库 RAG 准确率：基于 Milvus 向量库 + Top-K=3 + SimilarityThreshold=0.75 的混合检索，回答准确率 96.8%（vs 纯 LLM 记忆的 72.3%），检索失败时强制转人工，不自由发挥。
  • MessageTemplate 模板化：40% 对话使用模板回复（欢迎/转接/结束），30% 使用 RAG 增强模板，30% 完全自由生成。变量注入前强制脱敏，模板变更走 Git 审查流程。
  • ComplianceAdvisor 双审核：规则层（Aho-Corasick 匹配 30+ 禁止词，1ms 内完成）+ 模型层（Qwen-72B 策略冲突检测，500ms-1.5s），禁止承诺词拦截率 100%，政策冲突检测准确率 96%。
  • 敏感场景升级：COMPLAINT/LEGAL/CLAIM 三类意图 + 关键词 + 情感分析三重判断，敏感场景占全部对话 5%-8%，召回率 95.6%、假阳性率从 4.4% 优化到 2.8%。
  • 合规闭环：拦截记录 -> ELK 存储 -> 人工审核团队每周回顾 -> 更新禁止词列表和政策库，形成持续改进的合规飞轮。6 个月内合规事故从月均 2 起降到 0 起。

面试可能问

Q1: 智能客服怎么保证回答符合公司政策、不会做出违规承诺？

我设计了三层合规体系来确保回答合规。第一层是政策库 RAG——所有回答必须基于检索到的最新政策文档生成，使用时 VectorStore + FilterExpression 确保只检索 status=active 的最新版本。检索 Top-K=3，相似度阈值设为 0.75，低于该阈值的认为不相关，直接转人工不给 LLM 自由发挥的机会。第二层是 ComplianceAdvisor——在 LLM 输出返回到用户之前，使用规则层（Aho-Corasick 匹配 30+ 个禁止承诺词，如「一定退款」「保证赔偿」）和模型层（独立的审核 LLM 检查政策冲突）进行双重审核，任一层触发即拦截。第三层是敏感场景检测——COMPLAINT/LEGAL/CLAIM 三类意图强制转人工，关键词检测命中「起诉」「律师」「投诉工商」时也触发升级。生产数据：禁止承诺词拦截率 100%，政策冲突检测准确率 96%，敏感场景召回率 95.6%。这三层叠加确保了系统的合规底线。

Q2: 政策文档更新了怎么保证系统及时使用新版本？

政策文档管理是一个完整的 CI/CD + 版本控制流程。文档团队在 GitLab 仓库中维护 Markdown 格式的政策文档，每次更新通过 MR 提交流程（需要法务和业务方 approve）。MR 合并到 main 分支后触发 GitLab CI 流水线：第一步用 DocumentReader 解析 Markdown 为文本块；第二步用 DocumentSplitter 按章节和条款分割（TokenTextSplitter，chunkSize=500, overlap=50）；第三步使用 EmbeddingModel 生成 1024 维向量；第四步写入 Milvus 向量库的指定 collection。写入时使用 upsert 模式：相同 documentId 的新版本覆盖旧版本，但旧版本被标记为 status=archived（保留历史版本用于审计）。检索时通过 FilterExpression 只查 status=active，确保永远使用最新版本。从 MR 合并到系统生效，整个流程约 5 分钟。紧急情况下（如政策内容有误需要立即撤回），通过 ops 脚本直接删除对应 documentId 的 Milvus 记录，5 秒内生效。

Q3: ComplianceAdvisor 的性能开销多大？会不会影响首答时间？

ComplianceAdvisor 采用两阶段策略来平衡性能和安全。第一阶段是规则审核——使用 Aho-Corasick 多模式匹配算法在 O(n) 时间内扫描回复文本，30 个禁止词在 500 字文本中平均耗时 0.8ms，几乎不影响延迟。规则拦截通过后不放行直接转人工，跳过了模型审核。第二阶段是模型审核——使用独立的 Qwen-72B（Temperature=0.1）进行政策冲突检测，额外消耗 500ms-1.5s 和约 300 Tokens。由于模型审核是异步触发的（不阻塞用户回复返回），实际对首答时间的额外影响仅为规则审核的 0.8ms。用户的回复先返回，模型审核结果在后台上报：如果通过则无事发生，如果检测到冲突则记录合规日志并触发告警，人工团队联系用户进行回退处理。这种异步设计在过去 6 个月的生产运行中验证了可行性：合规事件从发现到处理的平均时间约 15 分钟（人工审核团队的响应时间），而如果没有异步审核，首答时间会增加 1.5s 导致 CSAT 下降约 0.15 分。

判断练习

场景题：你的产品经理说「LLM 自己知道政策内容，为什么还要花钱做 RAG？直接让它回答就行了」。你怎么反驳？

需要从准确率、可控性和合规性三个角度说明 RAG 的必要性。第一，准确率差距明显：CorpAssist 的 A/B 测试中，纯 LLM 记忆的政策回答准确率为 72.3%，而 RAG 增强的回答达到 96.8%——差距 24.5 个百分点。LLM 的训练数据截止日期可能早于政策更新时间，如果公司 3 月更新了退货政策而 LLM 训练数据停留在 1 月，那 3 个月内用户会得到过时的政策信息。第二，可控性：RAG 的检索来源是可控的——可以精确知道回答引用了哪份文档的哪个条款，而纯 LLM 记忆回答无法追溯来源，出现合规问题时难以定责和修复。第三，引用溯源：RAG 可以在回答中标注「根据《退货政策 V3.2》第 4.1 条」这样的引用，增加用户信任感，而纯 LLM 记忆生成的回答无法做到。综合来看，RAG 的额外成本（向量库维护、LLM 检索 Token 消耗约 50 Tokens/次）与合规风险相比微不足道——一个合规事故可能带来的法律赔偿和品牌损失远超 RAG 的年维护成本。

场景题：ComplianceAdvisor 拦截了一个敏感回复，但人工审核后发现用户的问题其实合规。如何减少误拦？

拦截规则的设计遵循「安全优先」原则——宁可误拦不可漏拦，所以一定比例的假阳性是可以接受的。但持续优化减少误拦是必要的。具体优化路径分三步：第一步分析误拦数据，识别模式——如果某条规则（如「免费赠送」这个词的匹配）导致了 80% 的误拦，说明这条规则的定义太宽泛了——可以将其改为上下文敏感匹配，如只在「承诺免费赠送」的动词短语上下文中才触发。第二步引入白名单机制——为特定的业务场景添加豁免，例如「促销活动期间，系统发送的优惠信息中包含免费赠送是合规的」。豁免规则需要法务审核后加入白名单。第三步模型审核的假阳性调优——通过 Fine-Tuning 审核模型的 Prompt 或增加 Few-Shot 示例来减少误判，或者从独立审核改为多模型投票（3 个不同模型投票，2/3 通过才放行）。经过这三步优化，CorpAssist 的合规拦截假阳性率从 15% 降到了 4%，同时保持了真正的阳性发现率不变。
