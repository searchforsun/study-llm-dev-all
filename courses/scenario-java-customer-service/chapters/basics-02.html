为什么学这章

智能客服系统涉及多模块协作——意图识别、对话管理、工具调用、人工坐席、质检——模块之间的接口设计和数据流决定了系统的可维护性和可扩展性。如果不清楚模块划分和职责边界，开发过程中会出现「这个逻辑该放哪里」的混乱，导致代码耦合严重、难以测试。CorpAssist 的首个版本正是因为模块职责不清，花费了 2 周重构才梳理清楚架构。

学完后能做什么

能画出 CorpAssist 客服系统的五大核心模块和它们之间的数据流；能基于 Spring Cloud Gateway 配置按意图路由的消息分发策略；能设计模块间的异步事件通信机制和接口契约；能独立搭建一个最小可用的客服系统微服务部署拓扑。

本章要点速记

  • 五大核心模块：意图路由（意图识别 + 规则引擎）→ 对话引擎（ChatClient + ChatMemory）→ 工具层（@Tool 业务工具）→ 人工坐席（上下文移交 + Redis 队列）→ 质检（ComplianceAdvisor + 人工抽检）
  • Spring Cloud Gateway：按意图路径路由（/api/cs/intent/query），集成 RequestRateLimiter 限流和 TokenRelay 鉴权透传
  • 事件驱动通信：模块间通过 Spring Cloud Stream (RabbitMQ/RocketMQ) 发送 HandoffEvent、ToolResultEvent、EscalationEvent 等异步消息
  • 微服务拓扑：Gateway + Intent Service + Conversation Service + Tool Service + Agent Service + Quality Service，服务间通过 gRPC 或 HTTP/2 通信，部署于 K8s 集群

五大核心模块：意图路由/对话引擎/工具层/人工坐席/质检

CorpAssist 客服系统划分为五个核心模块，每个模块有明确的职责边界和对外接口。意图路由模块（Intent Router）负责接收用户消息，通过双引擎（规则引擎 + LLM 结构化分类）识别用户意图并输出 IntentResult（包含 intent 枚举、confidence 置信度、slots 实体列表）。对话引擎模块（Conversation Engine）是系统的核心，基于 Spring AI 的 ChatClient 管理多轮对话，调用 ChatMemory 读写会话历史，并在需要时触发工具调用或知识库检索。工具层（Tool Layer）以 @Tool 注解注册的 Java 方法实现具体的业务操作，如订单查询、退款处理、地址修改——每个工具都有独立的参数校验和错误处理逻辑。人工坐席模块（Agent Workspace）管理转人工队列，通过 Redis 有序集合（ZSET）实现坐席分配和排队，在转接时将完整的上下文（对话历史 + 用户画像 + 当前意图）移交给坐席工作台。质检模块（Quality Check）运行 ComplianceAdvisor 对模型输出进行合规审核，并对已完结的对话进行离线抽检和评分。

这五个模块遵循「前一个模块处理不了的事情交给下一个模块」的责任链原则。具体来说：消息到达后，意图路由模块先做意图分类——如果置信度 >= 0.8，交给对话引擎；如果置信度在 0.5-0.8 之间，触发二次确认；如果置信度 < 0.5 或意图为 UNKNOWN，直接转人工。对话引擎在生成回复时，如果需要的工具调用失败（重试耗尽），也降级转人工。质检模块对所有输出进行后处理，如果检测到合规问题（如包含「一定退款」等禁止承诺词），拦截回复并转人工审核。这种责任链设计保证了每个模块只关注自己的职责，模块间的耦合度低，便于独立测试和部署。

Spring Cloud Gateway 作为统一入口的架构设计

Spring Cloud Gateway 在 CorpAssist 客服系统中扮演着统一流量入口的角色，负责认证鉴权、请求限流、意图路由和协议转换四个核心功能。所有用户请求首先到达 Gateway，它基于 JWT Token 进行身份认证（通过 Spring Security OAuth2 Resource Server 配置），然后根据请求路径将消息路由到不同的后端服务。客服模块的 API 路径统一以 /api/cs/ 为前缀，例如 /api/cs/chat 用于对话请求，/api/cs/handoff 用于转人工请求，/api/cs/history 用于历史查询。Gateway 还负责将 HTTP 长连接转换为 SSE（Server-Sent Events）流式响应，支持用户在对话过程中实时接收 LLM 的流式输出。在 CorpAssist 的现网部署中，Gateway 的 P99 转发延迟控制在 5ms 以内，每日处理约 50 万次请求。

限流策略是 Gateway 的一个关键设计点。CorpAssist 实现了三层限流：第一层是 IP 级别的 RequestRateLimiter，限制单个 IP 每秒最多 10 次请求；第二层是用户级别的令牌桶，每个用户 1 分钟最多 30 次对话请求；第三层是后端服务的连接数控制，限制同时进行中的 LLM 调用不超过 20 个，防止 LLM API 过载。在实践中最容易踩坑的是流式连接的限流——如果用户发一条消息后立刻关闭页面再打开重连，Gateway 会在短时间内积累大量未完成的 SSE 连接。我们的解决方案是在 Gateway 层添加连接池管理，每个用户 ID 最多同时保留 3 个 SSE 连接，超出后自动关闭最旧的连接。以下代码展示了 Gateway 的核心路由和限流配置。

yaml
# application-gateway.yml — Spring Cloud Gateway 路由和限流配置
spring:
  cloud:
    gateway:
      routes:
        # 客服对话路由 — SSE 流式响应
        - id: cs-chat-route
          uri: lb://corpassist-conversation
          predicates:
            - Path=/api/cs/chat
            - Header=X-Intent, ORDER_QUERY, RETURN_REQUEST, POLICY_QUERY
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10
                redis-rate-limiter.burstCapacity: 20
                key-resolver: "#{@userKeyResolver}"
            - StripPrefix=0
        # 转人工路由
        - id: cs-handoff-route
          uri: lb://corpassist-agent-workspace
          predicates:
            - Path=/api/cs/handoff
          filters:
            - name: Retry
              args:
                retries: 3
                statuses: BAD_GATEWAY, SERVICE_UNAVAILABLE
        # 静态资源路由
        - id: cs-static-route
          uri: lb://corpassist-frontend
          predicates:
            - Path=/cs/**
      default-filters:
        - name: JwtAuthenticationFilter
        - name: RequestRateLimiter
          args:
            redis-rate-limiter.replenishRate: 100
            redis-rate-limiter.burstCapacity: 200

resilience4j:
  circuitbreaker:
    instances:
      llm-api:

模块间的数据流、接口契约与事件驱动通信

五个模块之间的数据流通过同步 API 和异步事件两种方式配合完成。同步 API 主要用于实时请求-响应场景：用户消息经 Gateway -> 意图路由模块（HTTP 调用）-> 对话引擎模块（HTTP 调用）-> 工具层（内部方法调用）-> 返回结果。同步链路的 P99 延迟要求控制在 5s 以内，因此每个环节的耗时都必须受控。异步事件用于非实时的模块间通信：当发生转人工操作时，对话引擎向「人工坐席模块」发送 HandoffEvent 消息（包含完整上下文）；当工具调用失败时，产生 ToolResultEvent 用于后续分析和告警；当质检模块检测到合规问题，发送 EscalationEvent 触发升级流程。异步通信基于 Spring Cloud Stream 实现，消息中间件可以选择 RabbitMQ（低延迟场景）或 RocketMQ（高吞吐场景）。

接口契约是模块间协作的基础。每个模块对外暴露的 API 都有明确的请求/响应结构和错误码规范。例如，意图路由模块的 /api/cs/intent/classify 接口接收 ClassifyRequest（包含 message、sessionId、userId），返回 IntentResponse（包含 intent、confidence、slots、rawText）。所有模块共享统一的 ApiResponse 包装类，错误码遵循 HTTP 状态码加业务码的格式（如 200 成功、400 参数校验失败、500 内部错误）。以下代码定义了核心的数据契约和事件结构。

java
// 模块间共享的接口契约类

// 统一 API 响应包装
@Data
@Builder
public class ApiResponse {
    private int status;       // HTTP 状态码
    private String code;      // 业务码，如 "INTENT_LOW_CONFIDENCE"
    private String message;   // 人类可读消息
    private T data;           // 响应数据
    private long timestamp;   // 服务器时间戳

    public static  ApiResponse success(T data) {
        return ApiResponse.builder()
                .status(200).code("OK").data(data)
                .timestamp(System.currentTimeMillis()).build();
    }

    public static  ApiResponse error(int status, String code, String message) {
        return ApiResponse.builder()
                .status(status).code(code).message(message)
                .timestamp(System.currentTimeMillis()).build();
    }
}

// 意图路由模块的请求/响应
public record ClassifyRequest(
    @NotBlank String message,
    @NotBlank String sessionId,
    @NotBlank String userId,
    @Nullable List history
) {}

public record IntentResponse(
    Intent intent,
    double confidence,
    Map slots,
    String rawText
) {}

public record HandoffEvent(

    double confidence,
    List conversationHistory,
    UserProfile userProfile,
    String reason,      // 转接原因: LOW_CONFIDENCE / TOOL_FAILURE / COMPLIANCE_BLOCK / USER_REQUEST
    Instant timestamp

public record ToolResultEvent(
    String sessionId,
    String toolName,

    String requestPayload,

    @Nullable String errorMessage
) {}

CorpAssist 客服架构总图与微服务部署拓扑

CorpAssist 客服系统的生产部署采用微服务架构，将五大核心模块拆分为独立的微服务单元，每个服务可以独立扩缩容。服务划分遵循「按业务能力拆分」的原则：Intent Service 负责意图识别（无状态，可按流量扩缩容）、Conversation Service 负责对话管理（有状态，需要稳定的 Redis 连接）、Tool Service 负责业务工具调用（需要访问外部 API）、Agent Service 负责人工坐席管理（需要 WebSocket 长连接支持）、Quality Service 负责离线质检（异步处理）。服务之间的同步通信使用 HTTP/2（通过 Spring Cloud OpenFeign 声明式调用），异步通信使用 RocketMQ（Spring Cloud Stream 绑定）。每个服务都注册到 Spring Cloud Nacos 注册中心，实现服务发现和健康检查。部署在 Kubernetes 集群上，每个服务至少 2 个 Pod 保证高可用，Conversation Service 和 Agent Service 通过 StatefulSet 管理有状态实例。

数据库和中间件的部署策略是：Redis Cluster 用于会话存储（ChatMemory）和转人工队列，数据分片 16 个 Slot，TTL 统一设置为 30 分钟。PostgreSQL 主从复制用于持久化存储——已完结对话日志、用户画像数据、质检结果报表。Elasticsearch 用于知识库的全文检索和日志分析。Vector Database（Milvus）用于向量检索——政策库的话术嵌入和相似度匹配。在 CorpAssist 的现网部署中，这套拓扑支撑了日均 5 万次对话（峰值 8 万）的服务能力，P99 端到端延迟 3.2s，系统可用性 99.95%。以下表格总结了各服务的部署参数。

本章结论

  • 五大模块责任链：意图路由 → 对话引擎 → 工具层 → 人工坐席 → 质检，遵循「上一个处理不了交给下一个」的职责分离原则，模块间通过明确的接口契约解耦。
  • Gateway 三层限流：IP 级（10 req/s）、用户级（30 req/min）、后端连接数级（20 并发），P99 转发延迟 5ms，日处理 50 万次请求。
  • 同步 + 异步混合通信：实时对话链路使用 HTTP/2 同步调用（P99 < 5s），非实时操作（转人工、质检告警）通过 RocketMQ 异步事件驱动。
  • API 契约标准：统一 ApiResponse 包装类，/api/cs/ 路径前缀，错误码格式为 HTTP 状态码 + 业务码，所有模块共享相同的数据契约。
  • K8s 部署拓扑：6 个微服务 + 5 个数据中间件，日均支撑 5 万对话（峰值 8 万），P99 端到端延迟 3.2s，系统可用性 99.95%。

面试可能问

Q1: 设计一个智能客服系统的微服务划分方案，你会怎么拆分？为什么？」

我会按业务能力拆分为六个微服务：Gateway（统一入口，负责鉴权/限流/路由）、Intent Service（意图识别，无状态易扩缩容）、Conversation Service（对话管理，有状态需要 ChatMemory 持久化）、Tool Service（工具调用，依赖外部业务 API）、Agent Service（人工坐席管理，需要 WebSocket 长连接）、Quality Service（离线质检，异步处理）。这种拆分方案的核心原则是「按变更频率和性能需求差异化拆分」：Intent Service 算法迭代频繁需要独立部署和灰度发布，Conversation Service 是有状态服务不能随意扩缩容需要 StatefulSet 管理。在 CorpAssist 的实践中，这种六服务划分让每个团队的职责明确——算法团队只关注 Intent Service 和 Quality Service，后端团队关注 Conversation 和 Tool Service，运维团队关注 Gateway 和 Agent Service。每个服务可以独立发布、独立测试，互不阻塞。

Q2: 网关层在智能客服系统中的职责有哪些？如果流量突增 10 倍，网关应该怎么保护后端？

网关层在智能客服系统中有四个核心职责：路由转发（将 /api/cs/chat 映射到 Conversation Service）、认证鉴权（JWT Token 验证和用户身份提取）、流量控制（限流、熔断、重试）、协议转换（HTTP -> SSE 流式响应）。当流量突增 10 倍时，网关需要执行三重保护策略：第一层是请求级别限流，使用 RequestRateLimiter 限制单位时间内的总请求数和每个用户的并发数——例如将每秒总请求从 500 限制到 800（仍然允许一定增长但不完全放开）；第二层是后端连接数控制，使用 Resilience4j Bulkhead 限制与 LLM API 的同时连接数不超过 30 个，超过的请求直接返回 429 状态码和「系统繁忙，请稍后再试」的兜底话术；第三层是熔断，当 LLM API 的错误率超过 50% 时触发 CircuitBreaker 熔断 30 秒，在此期间所有对话请求直接使用规则引擎回复。这三层保护确保了后端服务的稳定性，即使网关层有 10 倍流量，后端 LLM API 和数据库也不会被打垮。

Q3: 模块间用同步调用还是异步消息？你怎么选择？

选择依据很简单：用户需要实时等待的结果用同步调用，不需要或不希望用户等待的操作用异步消息。在 CorpAssist 中，实时对话链路（用户消息 -> 意图识别 -> 对话生成 -> 回复）必须使用同步调用，因为用户期望在几秒内得到回复，这个链路的 P99 延迟要求 <= 5s。而转人工操作、合规拦截告警、质检评分报告等非实时操作使用异步消息。具体事件包括：HandoffEvent（转人工通知）、ToolResultEvent（工具调用结果记录）、EscalationEvent（合规升级通知）。异步消息的技术选型是 Spring Cloud Stream + RocketMQ，因为 RocketMQ 支持事务消息保证至少一次投递，且延迟低（P99 < 5ms）。混合通信的架构要求：同步链路必须设置超时（HTTP 连接超时 3s、读取超时 10s），异步链路必须设置死信队列处理消费失败的消息。曾经有个坑：转人工的 HandoffEvent 如果消费失败（坐席模块重启），因为没有死信队列，消息就丢失了——用户等着转人工却没有任何反馈。加上死信队列后，失败消息会被重新投递到坐席模块的备用队列，保证最终一致性。

判断练习

场景题：你的同事提议将 Intent Router 和 Conversation Engine 合并为一个服务，理由是「减少内部调用延迟」。你怎么看？

这个提议的部分理由是正确的——合并可以消除 HTTP 内部调用，减少约 20-50ms 的延迟。但从架构角度看，合并会带来三个更大问题：第一，意图识别的算法迭代速度比对话管理快得多（算法团队每两周更新一次分类器），合并在一个服务里会导致频繁的全量部署，影响对话引擎的稳定性。第二，两者对资源的需求不同：Intent Router 是计算密集型（LLM 推理占用 GPU），Conversation Engine 是 IO 密集型（大量 Redis 读写和外部 API 调用），混合部署会使资源调度复杂化。第三，可观测性受损——合并后无法独立监控意图识别的准确率和对话引擎的延迟。CorpAssist 的实践数据表明，不合并虽然增加 30ms 内部调用延迟，但两个服务各自独立发布 32 次和 18 次而互不影响，团队协作效率提升约 40%。对于延迟敏感场景，可以通过 gRPC 替代 HTTP 将内部调用延迟降到 2ms 以内。因此我建议保持拆分，优化通信协议而非合并服务。

场景题：Gateway 在压力测试中，当并发数超过 100 时 LLM 调用超时率上升到 30%。如何诊断和修复？

这个问题需要分层排查。首先确认超时是发生在 Gateway 到后端的连接上，还是后端到 LLM API 的连接上。在 Gateway 的访问日志中查看 upstream_response_time 指标，如果这个值正常（< 3s）但 total_time 很大（> 10s），说明是 Gateway 内部的连接池排队导致的。修复方案：调整 Gateway 的 HTTP 连接池参数——spring.cloud.gateway.httpclient.pool.maxConnections=500 和 maxPendingConnectionAcquire=100。如果 upstream_response_time 本身就很大，说明是 Conversation Service 的线程池满了或者 LLM API 限流了：检查 Conversation Service 的 ThreadPoolTaskExecutor 配置，核心线程数需要从默认的 8 调整为 20，队列容量从 100 调整为 500。更重要的改进是添加 Resilience4j Bulkhead 隔离：限制同时进行中的 LLM 调用不超过 20 个，多出的请求直接在 Gateway 层快速失败返回 429 状态码，而不是在后端排队消耗资源导致大面积超时。此外，为 LLM API 添加 Client-Side 的 ConnectionPool 和 Timeout 配置：connectTimeout=3s, readTimeout=15s, maxConnections=20。
