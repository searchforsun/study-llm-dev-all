约 45 分钟 · 阶段：Java 栈实战 · 场景：S2 智能客服

为什么学这章

简单的「接收消息 -> 回复消息」模式无法满足复杂客服场景。用户可能在同一轮对话中先说「查一下订单」又说「顺便退了这个」——涉及意图切换和工具链组合。Spring AI 的 ReAct Agent（ReactAgent）实现了 Think-Act-Observe 循环，让 LLM 可以自我推理、调用工具、观察结果，最终自主决定何时回复用户。CorpAssist 从简单的 ChatClient 升级到 ReAct Agent 后，一次性解决多步问题（如查物流 + 退其中一个商品）的成功率从 41% 提升到 78%。

学完后能做什么

能使用 Spring AI 的 ReactAgent 实现 Think-Act-Observe 循环，让 Agent 自主决定工具调用顺序和回复时机；能配置 ToolCallback 注册多个业务工具并设置 maxSteps 限制循环次数；能实现 SSE 流式响应的端到端 Controller，用户边收边看 Agent 的推理过程；能实现会话状态跨请求的保存和恢复，支持长时间多轮对话的连续性。

本章要点速记

  • ReAct 循环：Think（思考当前需要什么）-> Act（调用工具或检索知识）-> Observe（观察工具结果）-> 循环直到满足停止条件或达到 maxSteps=5
  • ToolCallback 注册：通过 ToolCallback.from(@Tool Bean) 批量注册工具到 ReactAgent，Agent 根据思考结果自主决定调用顺序
  • SSE 流式 Controller：Flux> 实现，用户实时看到 Agent 的思考过程和工具调用结果，连接超时 60s
  • 跨请求状态恢复：ChatMemory + Redis 保存会话上下文和 Agent 执行状态，即使服务重启用户也能在下次请求中继续对话

ReAct Agent 的 Think-Act-Observe 循环实现

ReAct（Reasoning + Acting）是让 LLM 具备自主决策能力的核心模式。与简单的「用户问 -> LLM 答」不同，ReAct Agent 在生成回复之前，会反复执行「思考（Think）-> 行动（Act）-> 观察（Observe）」的循环。在 Think 阶段，Agent 判断当前需要什么信息或操作来回答用户问题；在 Act 阶段，Agent 调用对应的工具函数（如查订单、查政策）或检索知识库；在 Observe 阶段，Agent 获取工具的执行结果并将结果纳入推理过程。然后 Agent 决定：是继续下一轮循环（还需要更多信息）还是停止循环给出最终回复。Spring AI 的 ReactAgent 实现了这个循环，开发者只需要提供工具定义和停止条件（maxSteps 最大循环次数、工具调用是否需要等决策规则）。

在 CorpAssist 的客服场景中，ReAct Agent 的典型运行流程：用户说「帮我查一下 #ORD-001 的物流，再退一下这个单」。Agent 第一次 Think 判断需要「查物流」和「退款」两个操作。它在 Act 阶段先调用 OrderTool.queryOrder("ORD-001")，Observe 到结果后第二次 Think 判断「物流已查到，还需要执行退款」。第二次 Act 调用 RefundTool.applyRefund("ORD-001", "用户要求退款")，Observe 到退款成功后第三次 Think 判断「所有操作已完成」，停止循环，将两次操作结果组织成自然语言回复。整个过程中 Agent 自主决定了工具调用顺序（先查后退）、调用次数（2 次）、和停止时机（全部完成）。对比非 Agent 模式需要在代码中硬编码这个流程，ReAct Agent 的灵活性和自适应能力明显更强。
