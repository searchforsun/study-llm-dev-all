同一模板对所有用户输出相同文案，CTR 和转化率会随着投放次数递减。personalize 通过用户画像动态调整变量与语调，让每条内容看起来像是「为这个人写的」。

本章先记住 3 件事
  • 画像决定变量：user profile 的标签直接影响模板变量的取值，如 tone、cta_style。
  • 不要为每个人写不同模板：用同一模板 + 不同变量注入实现千人千面。
  • A/B 验证才能闭环：个性化方案必须通过 A/B 实验验证 CTR/转化率是否真有提升。

动态 Prompt 与分群注入

dynamic prompt 是在模板渲染阶段根据用户画像动态调整 prompt 内容。不是为每个用户生成不同模板，而是同一模板的变量值根据画像变化。

Python：画像驱动的动态 Prompt

python 
# CorpAssist 个性化引擎
class PersonalizationEngine:
    def __init__(self, crm_client):
        self.crm = crm_client

    def get_user_vars(self, user_id: str, template_id: str) -> dict:
        """根据用户画像生成模板变量"""
        profile = self.crm.get_profile(user_id)
        base_vars = {
            "product": "春季精华",
            "discount": "8折"
        }

        # 分群个性化
        segment = profile.get("segment", "new")
        if segment == "vip":
            base_vars["greeting"] = "亲爱的VIP会员"
            base_vars["discount"] = "专属7折"
            base_vars["cta"] = "立即抢购"
        elif segment == "new":
            base_vars["greeting"] = "欢迎新朋友"
            base_vars["discount"] = "首单9折"
            base_vars["cta"] = "探索新品"
        elif segment == "sleeping":
            base_vars["greeting"] = "好久不见"
            base_vars["discount"] = "回归特惠8折"
            base_vars["cta"] = "重拾美丽"

        # 偏好品类调整
        fav_cat = profile.get("fav_category", "skincare")
        base_vars["product"] = self.match_product(fav_cat)

        return base_vars

    def render_personalized(self, user_id: str,
                            template_id: str) -> str:
        vars = self.get_user_vars(user_id, template_id)
        return engine.render(template_id, vars)

Java：Spring 个性化配置

java 
// Spring 个性化服务
@Service
public class PersonalizeService {

    public Map<String, Object> resolveVars(
            String userId, String templateId) {
        UserProfile profile = crmClient.getProfile(userId);
        Map<String, Object> vars = new HashMap<>();

        vars.put("greeting", switch (profile.segment()) {
            case "vip" -> "亲爱的VIP会员";
            case "new" -> "欢迎新朋友";
            case "sleeping" -> "好久不见";
            default -> "亲爱的用户";
        });

        vars.put("discount", profile.priceSensitivity() > 0.7
            ? "超值优惠" : "品质之选");

        return vars;
    }
}

cohort var 是分群变量的概念——不是每个用户独一无二，而是将用户归入 5-10 个分群，每个分群一套变量组合。这样既实现了个性化效果，又避免了每条文案都不同的缓存失效问题。

先想 10 秒：10 万用户，5 个分群 vs 10 万分群，哪个更实际？

5 个分群。10 万分群意味着每 1-2 个用户就只有一个，LLM 缓存几乎完全失效，且难以通过 A/B 实验验证效果。5-10 个分群是效果与可维护性的最佳平衡。

A/B 实验框架

ab test content 验证个性化方案是否真的有效。没有 A/B 实验的个性化是「自以为是的优化」。

不推荐
  • 新客老客同一套文案跑全量
  • 个性化上线后不看数据直接全量
  • A/B 实验只跑 1 小时就得出结论
  • 无统计显著性检验
推荐
  • A 组（通用）/ B 组（个性化）50:50 分流
  • 跑够 7 天或样本量 > 1000/组
  • 使用 χ² 检验确认显著性（p < 0.05）
  • CTR + 转化率 + CSAT 多维评估

Python：A/B 实验评估

python 
# A/B 实验评估
from scipy.stats import chi2_contingency
import numpy as np

class ABTestEvaluator:
    def evaluate(self, control: list, variant: list) -> dict:
        """control/variant 各为 [impressions, clicks] 列表"""
        c_imp, c_clicks = control
        v_imp, v_clicks = variant

        c_ctr = c_clicks / c_imp if c_imp else 0
        v_ctr = v_clicks / v_imp if v_imp else 0

        # χ² 检验
        table = np.array([[c_clicks, c_imp - c_clicks],
                          [v_clicks, v_imp - v_clicks]])
        _, p, _, _ = chi2_contingency(table)

        return {
            "control_ctr": round(c_ctr, 4),
            "variant_ctr": round(v_ctr, 4),
            "lift": round((v_ctr - c_ctr) / c_ctr * 100, 1),
            "p_value": round(p, 4),
            "significant": p < 0.05
        }

ab = ABTestEvaluator()
result = ab.evaluate([5000, 150], [5000, 210])
print(f"CTR 提升 {result['lift']}%")
print(f"统计显著: {result['significant']}")

常见问题（FAQ）

个性化文案的缓存策略怎么设计？
以「分群 x 模板」为缓存键。5 个分群 x 10 个模板 = 50 个缓存条目，缓存命中率高。不要以 user_id 为缓存键。
用户标签更新频率多高？
CRM 层标签（分群、品类偏好）按日更新；行为层标签（近期浏览、购物车）按小时更新。prompt 生成时使用实时行为标签。
A/B 实验发现个性化没有显著提升怎么办？
三个方向排查：① 画像标签是否准确（分群是否正确）；② 变量差异化是否足够（5 个分群的 prompt 差异太小）；③ 指标选择是否合理（也有可能个性化后满意度提升但 CTR 没变）。

本章结论

  • 画像标签：5-10 个分群覆盖 90% 的个性化场景。
  • 动态变量：同一模板 + 不同变量 = 千人千面，不是千人千模板。
  • Profile Inject：画像标签在模板渲染时注入，全链路 < 500ms。
  • A/B 验证：50:50 分流 + 7 天 + p < 0.05 验证效果。
  • 缓存策略：分群 x 模板为缓存键，而不是 user_id。
