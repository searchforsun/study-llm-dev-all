本章先记住 3 件事

品牌语调：从玄学到工程

brand tone 决定了「同样一句话，你的品牌怎么说」。它不是模型微调能解决的——同一模型在同一 prompt 下仍然会漂移。工程手段是：语调卡 + few-shot 示例 + 输出约束。

CorpAssist · 品牌语调落地（30 秒） 一个高端护肤品牌要求所有文案「专业、温和、有科技感」。运营团队制定了 6 条语调规则（如：用「焕活」不用「美白」、用「精华」不用「水」、避免感叹号）。将这些规则编码为 tone card 后注入 system prompt，加上 3 篇过往精品文案作为 few-shot，输出风格一致性从 60% 提升到 92%。

语调卡（Tone Card）结构

维度规则示例位置
用词白名单 用「焕活」「修护」「精华」；不用「美白」「祛斑」「特效」 System prompt
句式偏好 短句为主，每段不超过 3 句 System prompt
禁止表达 禁止「全网最低价」「立即抢购」等强促销口吻 Output guardrail
情感倾向 积极但不夸张，温和但有说服力 Few-shot 范例
人称视角 统一用「我们」第一人称，不用「小编」 System prompt
不推荐
  • 语调规则写在 PRD 里，不注入 prompt
  • 每次修改语调直接改 system prompt，无版本记录
  • 所有品牌共用一套语调规则
推荐
  • 语调卡片化，用 yaml 定义 + prompt 注入
  • 语调版本 + 模板版本双维度管理
  • 按品牌/场景分设语调配置文件

Python：语调卡加载与注入

python 
# CorpAssist 语调卡加载器
import yaml
from typing import List

class ToneCard:
    def __init__(self, brand_id: str, version: str):
        self.brand_id = brand_id
        self.version = version
        self.rules: List[str] = []
        self.few_shot_refs: List[str] = []

    @classmethod
    def from_yaml(cls, path: str) -> "ToneCard":
        with open(path) as f:
            data = yaml.safe_load(f)
        card = cls(data["brand_id"], data["version"])
        card.rules = data.get("rules", [])
        card.few_shot_refs = data.get("few_shot_refs", [])
        return card

    def to_system_block(self) -> str:
        """生成 system prompt 语调块"""
        lines = ["## 品牌语调规则"]
        lines.extend(f"- {r}" for r in self.rules)
        return "\n".join(lines)

# 加载语调卡
card = ToneCard.from_yaml("tones/skincare-premium.yaml")
print(card.to_system_block())

// Spring AI 语调配置
@Component
@ConfigurationProperties(prefix = "corpassist.tone")
public class ToneConfig {
    private String brandId;
    private String version;
    private List rules;
    private List fewShotRefs;

    public String buildSystemPrompt() {
        StringBuilder sb = new StringBuilder("## 品牌语调规则\n");
        rules.forEach(r -> sb.append("- ").append(r).append("\n"));
        return sb.toString();
    }

    // getters & setters
}
先想 10 秒：语调卡 10 条规则与 5 条规则，LLM 更遵守哪种？

5 条。超过 8 条后逐条遵守率下降。选择影响风格最关键的 5-8 条，其他的通过 few-shot 范例隐式传递。

Few-shot 风格注入

few shot style 是通过 2-5 篇精品范例让 LLM 隐式学习品牌风格。相比纯规则，few-shot 更擅长传递「只可意会」的风格要素——节奏、语气、篇章结构。

Few-shot 风格注入流程
flowchart LR
    A[品牌精品库] -->|挑选 3-5 篇| B[Few-shot 池]
    B --> C[模板组装器]
    D[Tone Card 规则] --> C
    C -->|System + Few-shot| E[LLM 调用]
    E --> F[生成内容]
    F -->|对比示例| G[风格一致性评分]

    style B fill:#4A90D9,color:#fff
    style C fill:#7B68EE,color:#fff

Few-shot 示例存储在模板注册中心，与模板版本绑定。更换示例即更换风格。

实践要点：

  • 示例新鲜度：定期（季度）更新 few-shot 池，避免过时文案影响当前风格。
  • 多样性覆盖：5 篇示例应覆盖「新品发布」「促销活动」「品牌故事」「用户互动」四种常见场景。
  • 长度控制：每篇示例控制在 200 字以内，避免挤占 token 预算。

参数化模板与版本管理

param template 通过 {{变量}} 占位符将内容模板与业务数据解耦。一条模板配合不同的变量值，可以产出 N 篇个性化文案。

模板 registry 版本管理

yaml 
template_registry:
  templates:
    - id: promo-product-v2
      version: "2.1.0"
      brand: skincare-premium
      tone_card: tones/skincare-premium.yaml
      variables:
        - name: product
          type: string
          description: 产品名称
        - name: discount
          type: string
          description: 折扣信息
        - name: cta
          type: string
          description: 行动号召
      few_shot_refs:
        - examples/promo-summer-2025.md
        - examples/new-launch-2025.md
      prompt_template: |
        为 {{product}} 写一篇促销文案。
        折扣力度：{{discount}}
        引导语：{{cta}}
        请遵循品牌语调规则。

template version 使用 semver（主版本.次版本.补丁）：

  • 主版本：模板结构变更（如增加/删除变量）。
  • 次版本：语调规则或 few-shot 更新，不影响变量接口。
  • 补丁：文案微调、bug 修复。

版本迁移策略

模板版本迁移流程
sequenceDiagram
    participant O as 运营
    participant R as Template Registry
    participant P as 生成服务
    O->>R: 创建 v2.1.0

版本升级走灰度流程：10% → 50% → 100%，每阶段评估风格一致性评分。

常见问题（FAQ）

语调卡和 few-shot 需要同时用吗？
建议同时用。语调卡给出显式规则（不可做什么），few-shot 给出隐式示范（应该怎么做）。两者互补。规则覆盖 60%、范例覆盖 30%、输出约束覆盖 10%。
模板版本多了怎么管理？
使用 template registry 统一存储，每个模板保留最近 3 个主版本。旧版本标记为 deprecated 但内容仍可查看，不可用于新生成。
参数化模板会不会导致内容太机械？
会，如果变量太少。推荐每条模板至少 3 个变量（产品、利益点、行动号召）。固定内容比例 ≤ 60%，变量内容 ≥ 40%。变量越多，输出越多样化。
风格一致性怎么量化评估？
用 auto score LLM-as-judge 按语调卡规则逐条打分（0-5），每两周跑一次全模板批次。详情见 内容质量评估。

本章结论

  • 品牌语调工程化：语调卡 + few-shot + 输出约束的三层风格控制体系。

  • 版本管理：semver 规范 + 灰度迁移 + 旧版保留回滚窗口。
  • 模板注册中心：统一存储语调卡、few-shot、变量定义与版本历史。
  • 风格评估闭环：自动评分驱动语调卡与 few-shot 持续更新。

操作步骤

  2. 设计一条含 3 个变量的促销模板，写一段渲染后的示例。
  3. 用本章 Python 代码加载语调卡并生成 system block。
  4. 为模板写一个版本号（假设这是第二次大改），说明 bump 的理由。

判断练习

  1. 运营反馈「AI 写的内容有时候太啰嗦」。你会改语调卡还是改 few-shot？

    参考答案

    优先改 few-shot 范例。啰嗦是风格问题，few-shot 最能传递节奏感。在语调卡中加一句「每段不超过 3 句」作为补充。

  2. 双十一大促模板需要临时加「限时抢购」的紧急话术，但模板版本刚刚升到 v3.0。怎么办？

    参考答案

    走补丁版本 v3.0.1，不改变量接口，只改 prompt 文本加一个条件变量 is_flash_sale。大促结束后恢复 v3.0.0。
