模板引擎：变量渲染

template engine 将模板字符串中的 {{variable}} 占位符替换为实际数据。Python 端常用 Jinja2，Java 端可用 Spring AI 的 PromptTemplate。

CorpAssist · 大促文案批量生成（30 秒） 运营上传 5000 条 SKU 数据（product, price, discount），选择模板 promo-product-v2。模板引擎 3 秒完成渲染，每条生成一个 prompt，推入 MQ 队列。10 个 worker 并发消费，调用 LLM 生成文案，结果写入数据库。用户通过 WebSocket 实时看到进度条从 0% 走到 100%。

Python：Jinja2 模板渲染引擎

python 
# CorpAssist 模板渲染引擎
from jinja2 import Template
from typing import List, Dict
import csv, json

class TemplateEngine:
    def __init__(self, template_dir: str = "templates/"):
        self.template_dir = template_dir
        self._cache = {}

    def render(self, template_id: str, variables: Dict) -> str:
        """渲染单条模板"""
        if template_id not in self._cache:
            with open(f"{self.template_dir}/{template_id}.j2") as f:
                self._cache[template_id] = Template(f.read())
        return self._cache[template_id].render(**variables)

    def batch_render(self, template_id: str,
                     rows: List[Dict]) -> List[Dict]:
        """批量渲染：每行一条 prompt"""
        return [
            {"prompt": self.render(template_id, row), "meta": row}
            for row in rows
        ]

    def from_csv(self, template_id: str, csv_path: str) -> List[Dict]:
        """从 CSV 批量导入变量"""
        with open(csv_path) as f:
            reader = csv.DictReader(f)
            return self.batch_render(template_id, list(reader))

# 使用
engine = TemplateEngine()
prompts = engine.from_csv("promo-product-v2", "skus_5000.csv")
print(f"已渲染 {len(prompts)} 条 prompt")

Java：Spring AI PromptTemplate

java 
// Spring AI 模板渲染
@Component
public class ContentTemplateEngine {

    private final Map<String, PromptTemplate> cache = new ConcurrentHashMap<>();

    public String render(String templateId, Map<String, Object> vars) {
        PromptTemplate pt = cache.computeIfAbsent(templateId,
            id -> new PromptTemplate(loadTemplate(id)));
        Message msg = pt.createMessage(vars);
        return msg.getContent();
    }

    public List<GenerateTask> batchRender(
            String templateId, List<Map<String, Object>> rows) {
        return rows.stream()
            .map(row -> new GenerateTask(
                UUID.randomUUID().toString(),
                render(templateId, row),
                row
            ))
            .toList();
    }

    private String loadTemplate(String id) {
        // 从文件/DB 加载模板内容
        return "为 {{product}} 写促销文案，折扣 {{discount}}";
    }
}

record GenerateTask(String jobId, String prompt,
                    Map<String, Object> meta) {}
先想 10 秒：5000 条变量用模板引擎渲染大概需要多久？

Jinja2 渲染 5000 条不到 3 秒。模板引擎不是瓶颈，LLM 调用才是。所以渲染与 LLM 调用必须解耦。

MQ 削峰与批量调度

mq content 是批量生成的可扩展架构核心。模板渲染后的 prompt 不直接调 LLM，而是推入消息队列，由 worker 池异步消费：

批量生成 MQ 架构
flowchart LR
    U[运营上传 CSV] --> E[Template Engine]
    E -->|每条一条 MQ 消息| Q[(RabbitMQ)]
    Q --> W1[Worker 1]
    Q --> W2[Worker 2]
    Q --> W3[Worker N]
    W1 --> LLM[通义千问 API]
    W2 --> LLM
    W3 --> LLM
    LLM --> DB[(MySQL)]

    P[Progress Tracker] -->|Redis 计数| WS[WebSocket → 前端]

MQ 解耦渲染与 LLM 调用，worker 水平扩展应对大促峰值。

job queue 的设计要点：

  • 消息不可丢失：MQ 队列设置为持久化（persistent），worker 消费后手动 ack。
  • 幂等消费：每条消息携带 job_id + row_id，worker 消费前检查是否已处理。
  • 死信队列：重试 3 次失败的消息转入死信队列，人工介入处理。
  • 动态扩缩：按队列积压数自动扩缩 worker 数量（KEDA 或自研）。
不推荐
  • 渲染完后直接 for 循环调 LLM
  • 无 MQ，生成与渲染耦合在一起
  • 失败后无重试，直接丢弃
  • 无进度反馈，用户只能干等
推荐
  • 渲染→MQ→Worker 三层解耦
  • RabbitMQ 持久化 + 手动 ack
  • 重试 3 次 + 死信队列兜底
  • Redis 计数 + WebSocket 推进度

python 
# CorpAssist MQ 批量调度
import pika, json, uuid
from concurrent.futures import ThreadPoolExecutor

class BatchProducer:
    def __init__(self, queue="content_gen"):
        self.conn = pika.BlockingConnection(...)
        self.channel = self.conn.channel()
        self.channel.queue_declare(queue, durable=True)

    def publish_tasks(self, tasks: list):
        """发布批量任务"""
        batch_id = str(uuid.uuid4())
        for task in tasks:
            msg = {"batch_id": batch_id,
                   "job_id": task["job_id"],
                   "prompt": task["prompt"],
                   "meta": task["meta"]}
            self.channel.basic_publish(
                exchange='',
                routing_key='content_gen',
                body=json.dumps(msg),
                properties=pika.BasicProperties(delivery_mode=2))
        print(f"已发布 {len(tasks)} 条任务, batch_id={batch_id}")
        return batch_id

Java：Spring AMQP 消费者

java 
// Spring RabbitMQ 消费者
@Component
public class ContentGenWorker {

    @RabbitListener(queues = "content_gen")
    public void handleTask(GenerateTask task) {
        try {
            // 幂等检查
            if (generationRepo.existsByJobId(task.jobId())) return;
            // 调用 LLM
            String content = llmClient.chat(task.prompt());
            // 写入数据库
            generationRepo.save(new GeneratedContent(
                task.jobId(), content, task.meta()));
            // 进度更新
            progressTracker.increment(task.batchId());
        } catch (Exception e) {
            // 重试 3 次后进入死信队列
            throw new AmqpRejectAndDontRequeueException(e);
        }
    }
}

进度追踪与失败处理

progress track 让用户知道「当前进度 + 剩余时间 + 失败条数」。实现方案：

  • Redis 原子计数：每条任务完成后 INCR batch_progress:{batch_id}
  • WebSocket 实时推送：Spring 通过 STOMP 将进度推送到前端
  • 失败详情可查：每一条失败记录保存到 failure_log 表，支持下载查看

进度 RDS 计数

python 
# Redis 进度追踪
import redis

r = redis.Redis()
BATCH_PREFIX = "batch:progress:"

def init_progress(batch_id: str, total: int):
    r.set(f"{BATCH_PREFIX}{batch_id}:total", total)
    r.set(f"{BATCH_PREFIX}{batch_id}:done", 0)
    r.set(f"{BATCH_PREFIX}{batch_id}:fail", 0)

def mark_done(batch_id: str, success: bool):
    key = f"{BATCH_PREFIX}{batch_id}:done" if success \
          else f"{BATCH_PREFIX}{batch_id}:fail"
    r.incr(key)

def get_progress(batch_id: str) -> dict:
    total = int(r.get(f"{BATCH_PREFIX}{batch_id}:total") or 0)
    done = int(r.get(f"{BATCH_PREFIX}{batch_id}:done") or 0)
    fail = int(r.get(f"{BATCH_PREFIX}{batch_id}:fail") or 0)
    return {
        "total": total, "done": done, "fail": fail,
        "percent": round((done + fail) / total * 100, 1) if total else 0
    }

常见问题（FAQ）

模板引擎渲染和 LLM 调用为什么要用 MQ 解耦？
渲染 1 万条只要几秒，但 LLM 调用 1 万条可能需要 30 分钟。耦合在一起意味着「渲染线程要等到 LLM 返回才能渲染下一条」，吞吐量受限于单线程。
批量生成时 LLM 的并发限制怎么处理？
在 Worker 侧做并发控制。Python 用 Semaphore + ThreadPoolExecutor 限制最大并发数（如 10 路并发）。Spring 用 @Async 配合线程池配置。同时利用 MQ prefetch 控制每个 worker 同时消费的消息数。
批量生成中途失败了一半，怎么恢复？
死信队列中的消息可以重新入队（requeue）。关键是每条任务要有幂等 ID，重新执行时不会生成重复内容。失败原因记录到日志，方便排查。

本章结论

  • 模板引擎：Jinja2 / Spring AI PromptTemplate，渲染性能 ≥ 1000 条/s。
  • MQ 削峰：RabbitMQ / Kafka 解耦渲染与 LLM 调用，Worker 可水平扩展。
  • 幂等消费：job_id 唯一标识，消费前检查避免重复生成。
  • 进度追踪：Redis 计数 + WebSocket 实时推送。
  • 失败处理：重试 3 次 → 死信队列 → 人工介入。
