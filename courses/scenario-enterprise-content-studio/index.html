有害内容拦截与 Moderation

harmful block 在敏感词匹配之外，还需 Moderation API 检测语义级违规（如暗示、隐喻、对抗性文本）：

不推荐
  • 只有敏感词匹配，不调 Moderation API
  • 违规内容直接删除无告警
  • 所有行业用同一套词库
推荐
  • AC 自动机 + Moderation API 双检测
  • 违规内容记录+告警+通知运营
  • 按行业加载专属词库

审计留痕

audit trail 记录每篇内容从生成到发布的全生命周期。合规审计时可以说「每条内容都有据可查」。

content log 的字段设计：

  • 生成阶段：模板 ID + 版本、渲染变量、LLM model + 参数、原始 prompt + 输出
  • 审核阶段：自动审核结果 + 命中规则、人工审核人 + 操作时间 + 结论
  • 重写阶段：驳回原因、重写轮次、修改前后 diff
  • 发布阶段：发布时间、渠道、操作人

审计日志存储

python 
# 审计日志写入
class AuditLogger:
    def __init__(self, db):
        self.db = db

    def log(self, content_id: str, stage: str,
            action: str, detail: dict, operator: str = "system"):
        """记录审计日志"""
        entry = {
            "content_id": content_id,
            "stage": stage,          # generate/audit/rewrite/publish
            "action": action,
            "detail": json.dumps(detail, ensure_ascii=False),
            "operator": operator,
            "timestamp": datetime.utcnow().isoformat()
        }
        self.db.insert("audit_log", entry)

    def get_trail(self, content_id: str) -> list:
        """获取完整审计链"""
        return self.db.query(
            "SELECT * FROM audit_log WHERE content_id=? ORDER BY timestamp",
            [content_id])

常见问题（FAQ）

行业规则更新后，已发布的历史内容需要重新审核吗？
取决于行业。广告法更新后，通常需要对在架内容做「合规重扫描」，标记违规内容下架。医药行业要求 24 小时内处理。
Moderation API 的误判率多高？
主流 Moderation API（如通义、OpenAI）的误判率在 1-3%。建议结合规则引擎降低误杀——只有规则 + API 都标记才拦截。
审计日志需要存多久？
行业法规通常要求 1-3 年。金融行业要求 5 年。推荐启用日志归档，90 天内的日志在线查询，更早的归档到对象存储。

本章结论

  • 敏感词库：通用 + 行业 + 品牌三层，AC 自动机 O(n) 匹配。
  • 行业规则引擎：医疗/金融/食品各专属规则，可配置。
  • 有害拦截：规则 + Moderation API 双检测，人工兜底。
  • 审计留痕：全链路日志，从生成到发布每步可追溯。
  • 合规下线：规则更新后，重新扫描在架内容。

ROI 叙事 = 投入成本（左）vs 产出价值（右），算回报率与回本周期。

Python：ROI 报告生成

python 
# CorpAssist 内容工坊 ROI 报告
class PortfolioReport:
    def __init__(self):
        self.before = {
            "fte": 5, "monthly_output": 200,
            "hours_per_piece": 4, "quality": 3.2
        }
        self.after = {
            "fte": 1.5, "monthly_output": 2000,
            "hours_per_piece": 0.42, "quality": 4.1,
            "llm_cost": 1200, "infra_cost": 800
        }

    def summary(self) -> dict:
        b, a = self.before, self.after
        fte_save = b["fte"] - a["fte"]
        monthly_salary_save = fte_save * 15000  # ¥15K/FTE

        return {
            "效率提升": f"{b['hours_per_piece']/a['hours_per_piece']:.1f}x",
            "月产出提升": f"{a['monthly_output']/b['monthly_output']:.0f}x",
            "FTE 节省": f"{fte_save} 人 (¥{monthly_salary_save:,.0f}/月)",
            "质量分提升": f"{a['quality']-b['quality']:+.1f}",
            "月总成本": {
                "传统": f"¥{b['fte'] * 15000:,.0f}",
                "AI 工坊": f"¥{a['fte']*15000 + a['llm_cost'] + a['infra_cost']:,.0f}"
            },
            "ROI": f"{monthly_salary_save/(a['llm_cost']+a['infra_cost']):.1f}x"
        }

    def narrative(self) -> str:
        s = self.summary()
        return f"""
## CorpAssist 内容工坊 ROI 报告

**效率**：单篇制作周期从 4h 缩短到 25min ({s['效率提升']})
**产能**：月产出从 200 篇提升到 2000 篇 ({s['月产出提升']})
**质量**：Rubric 均分从 3.2 提升到 4.1 ({s['质量分提升']})
**人力**：{s['FTE 节省']}
**ROI**：{s['ROI']}
"""

质量分趋势与坏例收敛

quality score 的展示不只平均分——还要看分布变化。核心是低分尾巴是否在缩短：

  • 平均分趋势：每周 Rubric 均分折线图（目标：稳步上升）。
  • 评分分布：每月的 1-5 分分布柱状图（目标：左偏→右偏）。
  • 低分率：≤3 分内容占比（目标：从 18% 降到 5% 以下）。
  • 坏例修复率：每周修复的坏例数量（目标：修复 > 新增）。

常见问题（FAQ）

业务方说「AI 产出量虽然高但质量不行」，怎么回应？
用数据说话：展示 Rubric 均分趋势 + 低分率变化 + 坏例修复率。如果可以，做一次盲测——让业务方盲评 AI 内容和人工内容，看能否区分。
LLM 调用费涨了怎么办？
区分成本的量(volume)和单价(unit cost)。产出量涨了总调用费自然涨，但单篇成本应该在下降。展示「单篇 LLM 成本」从 ¥0.5 降到 ¥0.3。
ROI 报告多久更新一次？
月度自动生成、季度正式汇报。月度用于内部迭代优化，季度纳入交付汇报材料。

第三问：质量评估

参考答案框架：

  3. 人工校准：每周抽检 30% 低分内容人工评分，校准自动分的偏差。

本章先记住 3 件事

  • 10× 不是模型能力：efficiency 10x 是模板引擎 + 批量作业 + 审核流水线三者的组合杠杆，不是单次生成速度的 10 倍。

营销/运营场景：ROI 从哪里来

企业内容生产通常覆盖三类场景，每类的生产频率、质量要求与 ROI 逻辑不同：

场景典型产出月均量ROI 核心指标

促销物料 海报文案、活动页、Banner 200-500 张 模板复用率、审核通过率、上线时效
私域触达 企微话术、短信、Push 1000+ 条 分群覆盖率、打开率、退订率
CorpAssist · 大促活动文案生产（30 秒） 大促前运营提了 200 条 SKU 级别的促销文案需求。传统方式：文案 4h/篇 × 2 人 × 5 天 = 80 人时，还不算审核改稿。CorpAssist 内容工坊用模板 + 批量生成 + 自动预审，将单篇周期降到 20min。核心 ROI 计算：production volume 从 50 篇/周涨到 500 篇/周。
不推荐：ROI 算不清
  • 只算「模型调用费」，忽略人工审核成本
  • 将「生成速度」等同「生产效率」
  • 不统计模板复用率，每篇都重新写 prompt
  • 无审核通过率，返工成本未被计入
推荐：ROI 分层统计
  • 全链路成本：模板制作 + LLM 调用 + 审核 + 发布
  • 质量折损：自动生成 → 人工修改率 < 20%
  • 模板复用率 ≥ 70%，每月新增模板 < 30%
  • 按场景（推文/物料/私域）分别统计

Python：ROI 粗算脚本

python 
# CorpAssist 内容工坊 ROI 粗算器
class ContentStudioROI:
    def __init__(self, monthly_volume: int, template_count: int):
        self.volume = monthly_volume
        self.templates = template_count

    def manual_cost(self, hours_per_piece=4, rate=80):
        """传统方式总成本"""
        return self.volume * hours_per_piece * rate

    def ai_cost(self, api_per_piece=0.3, review_hours=0.3, rate=80):
        """AI 工坊总成本（含人工审核）"""
        llm = self.volume * api_per_piece
        review = self.volume * review_hours * rate
        return llm + review

    def template_reuse_rate(self, used_templates: int) -> float:
        """模板复用率"""
        return used_templates / self.templates if self.templates else 0

    def efficiency_gain(self) -> str:
        human_hrs = self.volume * 4  # 传统每篇 4h
        ai_hrs = self.volume * 0.5   # AI 每篇含审核 0.5h
        return f"{human_hrs / ai_hrs:.1f}x"

roi = ContentStudioROI(monthly_volume=500, template_count=50)
print(f"效率提升: {roi.efficiency_gain()}")
print(f"月节省: ¥{roi.manual_cost() - roi.ai_cost():,.0f}")

public record ContentMetrics(
    int monthlyVolume,
    double templateReuseRate,    // ≥ 0.7
    double reviewPassRate,       // ≥ 0.85
    double avgHoursPerPiece,     // Before: 4.0, After: ≤ 0.5

            int volume, int templatesUsed, int templateTotal,
            int reviewed, int passed) {
        return new ContentMetrics(
            volume,
            (double) templatesUsed / templateTotal,
            (double) passed / reviewed,
            0.5,

        );

            monthlyVolume, templateReuseRate * 100,
            reviewPassRate * 100, fteSaved
        );

常见误区
  • ROI 只算 LLM 调用费——人工审核、模板维护、发布运营成本往往更高。
  • 上线后才补 ROI 数据——应在模板设计阶段就确定统计口径。
  • 所有场景用同一套模板——推文、海报、私话的文案风格和审核规则不同。
  • 忽略质量折损——生成 500 篇如果 200 篇需要重写，实际效率不到 3×。

常见问题（FAQ）

内容工坊的 ROI 应该项目启动前算还是上线后算？
启动前粗算、上线后精算。启动前用模板数 × 月需求量 × 单篇人工成本估出「不回本不上线」的门槛；上线后用实际数据（模板复用率、审核通过率、生产周期）验证。两个口径都要留。
模板复用率冲到 95% 是不是最好？
不一定。过度复用模板会导致内容同质化、CTR 下降。推荐 70-85% 为健康区间——既保证产能，又保留一定比例的人工创意内容来维持品牌新鲜度。
小团队（1-2 人运营）适合上内容工坊吗？
适合。从 2-3 个高频模板起步，月产出 50 篇即可验证 ROI。模板引擎 + 自动审核可以节省 80% 的重复劳动。Python FastAPI 单服务即可承载初期需求，无需全套微服务。

本章结论

内容工坊的 ROI 是「全链路效率」而非「单点生成速度」。

  • 三场景差异化：推文、物料、私域各有 ROI 口径与模板策略，不可混用。
  • 10× 效率是组合杠杆：模板引擎 × 批量生成 × 审核流水线三者协同，缺一不可。
  • CorpAssist S5 五模块：模板库、批量生成、审核流水线、个性化引擎、发布适配器。
  • ROI 口径对齐：项目启动前就定好统计口径，避免上线后扯皮。
  • 质量折损要计入：生成快 ≠ 产出高，审核通过率直接影响实际效率。

品牌语调：从玄学到工程

brand tone 决定了「同样一句话，你的品牌怎么说」。它不是模型微调能解决的——同一模型在同一 prompt 下仍然会漂移。工程手段是：语调卡 + few-shot 示例 + 输出约束。

CorpAssist · 品牌语调落地（30 秒） 一个高端护肤品牌要求所有文案「专业、温和、有科技感」。运营团队制定了 6 条语调规则（如：用「焕活」不用「美白」、用「精华」不用「水」、避免感叹号）。将这些规则编码为 tone card 后注入 system prompt，加上 3 篇过往精品文案作为 few-shot，输出风格一致性从 60% 提升到 92%。

Python：语调卡加载与注入

python 
# CorpAssist 语调卡加载器
import yaml
from typing import List

class ToneCard:
    def __init__(self, brand_id: str, version: str):
        self.brand_id = brand_id
        self.version = version
        self.rules: List[str] = []
        self.few_shot_refs: List[str] = []

    @classmethod
    def from_yaml(cls, path: str) -> "ToneCard":
        with open(path) as f:
            data = yaml.safe_load(f)
        card = cls(data["brand_id"], data["version"])
        card.rules = data.get("rules", [])
        card.few_shot_refs = data.get("few_shot_refs", [])
        return card

    def to_system_block(self) -> str:
        """生成 system prompt 语调块"""
        lines = ["## 品牌语调规则"]
        lines.extend(f"- {r}" for r in self.rules)
        return "\n".join(lines)

# 加载语调卡
card = ToneCard.from_yaml("tones/skincare-premium.yaml")
print(card.to_system_block())

Few-shot 风格注入

few shot style 是通过 2-5 篇精品范例让 LLM 隐式学习品牌风格。相比纯规则，few-shot 更擅长传递「只可意会」的风格要素——节奏、语气、篇章结构。

Few-shot 示例存储在模板注册中心，与模板版本绑定。更换示例即更换风格。

实践要点：

  • 示例新鲜度：定期（季度）更新 few-shot 池，避免过时文案影响当前风格。
  • 多样性覆盖：5 篇示例应覆盖「新品发布」「促销活动」「品牌故事」「用户互动」四种常见场景。
  • 长度控制：每篇示例控制在 200 字以内，避免挤占 token 预算。

常见问题（FAQ）

语调卡和 few-shot 需要同时用吗？
建议同时用。语调卡给出显式规则（不可做什么），few-shot 给出隐式示范（应该怎么做）。两者互补。规则覆盖 60%、范例覆盖 30%、输出约束覆盖 10%。
模板版本多了怎么管理？
使用 template registry 统一存储，每个模板保留最近 3 个主版本。旧版本标记为 deprecated 但内容仍可查看，不可用于新生成。
参数化模板会不会导致内容太机械？
会，如果变量太少。推荐每条模板至少 3 个变量（产品、利益点、行动号召）。固定内容比例 ≤ 60%，变量内容 ≥ 40%。变量越多，输出越多样化。
风格一致性怎么量化评估？
用 auto score LLM-as-judge 按语调卡规则逐条打分（0-5），每两周跑一次全模板批次。详情见 内容质量评估。

大纲生成：先定骨架再填血肉

outline gen 是长文生成的第一步。LLM 先生成 3-5 个 H2 一级标题，每个 H2 下再展开 2-3 个 H3 子节。大纲的每个节点就是一个 segment write 任务。

CorpAssist · 新品发布长文（30 秒） 需求：一篇 2000 字的品牌新品发布推文。传统做法让 LLM 一次生成——结果开头有力、中间重复、结尾突然。用分段法：先生成 4 个 H2（开篇悬念→技术亮点→用户证言→购买引导），每段独立写入 400-500 字，最后拼接去重。输出质量提升 40%。

Python：大纲生成与分段调度

python 
# CorpAssist 长文分段生成
import json
from concurrent.futures import ThreadPoolExecutor

class LongformWriter:
    def __init__(self, llm_client):
        self.client = llm_client

    def generate_outline(self, topic: str, sections: int = 4) -> list:
        """生成大纲"""
        resp = self.client.chat(f"为 '{topic}' 生成 {sections} 段大纲，返回 JSON")
        return json.loads(resp)  # [{"h2": "...", "h3": ["...", "..."]}]

    def write_segment(self, outline_item: dict, context: str) -> str:
        """写单段"""
        prompt = f"## {outline_item['h2']}\n"
        for h3 in outline_item['h3']:
            prompt += f"### {h3}\n"
        return self.client.chat(prompt + f"\n全文上下文：{context[:200]}")

    def write_article(self, topic: str) -> str:
        outline = self.generate_outline(topic)
        with ThreadPoolExecutor(max_workers=4) as ex:
            segments = list(ex.map(
                lambda item: self.write_segment(item, topic),
                outline
            ))
        return self.merge_with_transitions(segments)

    def merge_with_transitions(self, segments: list) -> str:
        transitions = []
        for i, seg in enumerate(segments):
            if i > 0:
                trans = self.client.chat(
                    f"写一句从'{segments[i-1][:50]}'到'{seg[:50]}'的过渡句")
                transitions.append(trans)
            transitions.append(seg)
        return "\n\n".join(transitions)

大纲→分段（并行）→过渡句→合并去重，总耗时 = 最长单段 + 过渡句生成。

section merge 的关键是过渡句：每两段之间由 LLM 生成一句自然的转承句，避免段落割裂感。

去重与连贯性

dedup longform 解决分段写作最常见的质量问题——不同段落在相似观点上反复啰嗦。使用 embedding 向量计算段落间语义相似度，超过阈值则合并或删除。

相似度区间判定处理方式
0 - 0.70 不同内容 保留两段
0.70 - 0.85 部分重叠 取并集，合并精简
0.85 - 1.0 高度重复 删除一条，保留信息更完整的

coherence 是长文质量的终极指标。除了过渡句，还可以通过以下手段提升连贯性：

  • 全局摘要：每段注入前文 100 字摘要作为 context。
  • 术语统一：建立术语表，每段生成时强制使用统一术语。
  • 回看机制：写下一段前先回顾上一段的最后一句，确保语义衔接。
不推荐
  • 一次生成 3000 字长文，不打断
  • 分段后不做去重，段落重复率高
  • 不使用过渡句，段落拼接生硬
  • 不控制每段长度，造成头重脚轻
推荐
  • 先大纲后分段，每段 ≤ 500 字
  • embedding 去重，相似度 > 0.85 做合并
  • LLM 生成过渡句
  • 每段注入前文摘要保持连贯

Python：去重与连贯性检查

python 
# 段落去重与连贯性检查
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

class DedupChecker:
    def __init__(self, embed_fn):
        self.embed = embed_fn  # embedding 函数

    def dedup_paragraphs(self, paragraphs: list, threshold=0.85) -> list:
        """删除或者合并高度重复段落"""
        if len(paragraphs) <= 1:
            return paragraphs
        embeddings = [self.embed(p) for p in paragraphs]
        keep = [True] * len(paragraphs)
        for i in range(len(paragraphs)):
            for j in range(i + 1, len(paragraphs)):
                sim = cosine_similarity(
                    [embeddings[i]], [embeddings[j]])[0][0]
                if sim > threshold:
                    keep[j] = False  # 删后面的
        return [p for p, k in zip(paragraphs, keep) if k]

    def coherence_score(self, article: str) -> float:
        """连贯性评分（0-1）"""
        sentences = article.split("。")
        if len(sentences) < 3:
            return 1.0
        embs = [self.embed(s) for s in sentences]
        scores = []
        for i in range(len(embs) - 1):
            scores.append(cosine_similarity(
                [embs[i]], [embs[i + 1]])[0][0])
        return float(np.mean(scores))

分段写作时，每段的最佳长度是多少？
推荐 300-500 字/段。太短（< 200 字）信息密度不够；太长（> 800 字）失去分段意义。2000 字文章建议分 4-5 段。
去重阈值 0.85 是固定的吗？
初始推荐值。不同领域的段落实则不同——技术文档重复率低可降到 0.80，营销文案重复率高可升到 0.88。建议用标注数据调优。
过渡句生成会不会引入新的事实错误？
有可能。过渡句需要基于前段和后段的事实，不能凭空编造。建议在过渡句 prompt 中注入前后段的 key points。
标题党式大纲能避免吗？
用 repeat detect 检测大纲中 H2 的语义距离，如果两个 H2 相似度 > 0.8 则要求 LLM 合并或重写。

本章结论

  • 大纲先行：Outline Gen 为长文定骨架，LLM 自动生成 H2/H3 多级结构。
  • 分段独立生成：Segment Write 每段 300-500 字，并行调用提高速度。
  • 过渡句连接：段间由 LLM 生成自然转承，避免段落割裂。
  • 去重保信息密度：Embedding 相似度 > 0.85 的段落合并或删除。
  • 连贯性可量化：Coherence Score 衡量句间语义平滑度。

Visual RAG：图文双模态检索

visual rag 是对品牌素材库进行语义检索的技术方案。与纯文本 RAG 不同，Visual RAG 需要将图片的视觉特征和文本标签同时编码为向量，实现图文双模态搜索。

不推荐
  • 只靠文件名关键词匹配图片（精确匹配，召回率低）
  • 从网上随意搜索图片（版权风险）
  • 每张图手动配标签（不可扩展）
  • 无图片审核，低质/违规图片混入
推荐
  • CLIP 双模态 embedding（文本+图片统一向量空间）
  • 品牌素材库统一管理，标注 license 元数据
  • 自动标签 + 人工校准
  • 图片审核前置（NSFW + 品牌一致）

# CorpAssist Visual RAG 素材检索
import numpy as np
from clip_client import ClipClient  # CLIP embedding

class VisualRAG:
    def __init__(self, vector_db_url: str):
        self.client = ClipClient(vector_db_url)

    def index_asset(self, image_path: str, tags: list[str],
                    license: str, expiry: str):
        """索引品牌素材"""
        embedding = self.client.encode_image(image_path)
        metadata = {"tags": tags, "license": license, "expiry": expiry}
        self.client.insert(embedding, metadata)

    def search(self, text_query: str, top_k: int = 3,
               license_filter: str = "commercial") -> list:
        """按文本搜索可用素材"""
        query_emb = self.client.encode_text(text_query)
        results = self.client.search(query_emb, top_k * 2)
        # 版权过滤
        valid = [r for r in results
                 if r.metadata.get("license") == license_filter
                 and r.metadata.get("expiry", "9999") > "2026-12-31"]
        return valid[:top_k]

    def auto_tag(self, image_path: str) -> list[str]:
        """自动生成图片标签"""
        return self.client.classify(image_path,
                                    ["产品图", "场景图", "人物图",
                                     "插画", "背景图"])

# CorpAssist 模板渲染引擎
from jinja2 import Template
from typing import List, Dict
import csv, json

class TemplateEngine:
    def __init__(self, template_dir: str = "templates/"):
        self.template_dir = template_dir
        self._cache = {}

    def render(self, template_id: str, variables: Dict) -> str:
        """渲染单条模板"""
        if template_id not in self._cache:
            with open(f"{self.template_dir}/{template_id}.j2") as f:
                self._cache[template_id] = Template(f.read())
        return self._cache[template_id].render(**variables)

    def batch_render(self, template_id: str,
                     rows: List[Dict]) -> List[Dict]:
        """批量渲染：每行一条 prompt"""
        return [
            {"prompt": self.render(template_id, row), "meta": row}
            for row in rows
        ]

    def from_csv(self, template_id: str, csv_path: str) -> List[Dict]:
        """从 CSV 批量导入变量"""
        with open(csv_path) as f:
            reader = csv.DictReader(f)
            return self.batch_render(template_id, list(reader))

# 使用

@Component
public class ContentTemplateEngine {

    private final Map<String, PromptTemplate> cache = new ConcurrentHashMap<>();

    public String render(String templateId, Map<String, Object> vars) {
        PromptTemplate pt = cache.computeIfAbsent(templateId,
            id -> new PromptTemplate(loadTemplate(id)));
        Message msg = pt.createMessage(vars);
        return msg.getContent();
    }

    public List<GenerateTask> batchRender(
            String templateId, List<Map<String, Object>> rows) {
        return rows.stream()
            .map(row -> new GenerateTask(
                UUID.randomUUID().toString(),
                render(templateId, row),
                row
            ))
            .toList();
    }

    private String loadTemplate(String id) {
        // 从文件/DB 加载模板内容
        return "为 {{product}} 写促销文案，折扣 {{discount}}";
    }
}

MQ 解耦渲染与 LLM 调用，worker 水平扩展应对大促峰值。

job queue 的设计要点：

  • 消息不可丢失：MQ 队列设置为持久化（persistent），worker 消费后手动 ack。
  • 幂等消费：每条消息携带 job_id + row_id，worker 消费前检查是否已处理。
  • 死信队列：重试 3 次失败的消息转入死信队列，人工介入处理。
  • 动态扩缩：按队列积压数自动扩缩 worker 数量（KEDA 或自研）。
不推荐
  • 渲染完后直接 for 循环调 LLM
  • 无 MQ，生成与渲染耦合在一起
  • 失败后无重试，直接丢弃
  • 无进度反馈，用户只能干等
推荐
  • 渲染→MQ→Worker 三层解耦
  • RabbitMQ 持久化 + 手动 ack
  • 重试 3 次 + 死信队列兜底
  • Redis 计数 + WebSocket 推进度

import redis

r = redis.Redis()
BATCH_PREFIX = "batch:progress:"

def init_progress(batch_id: str, total: int):
    r.set(f"{BATCH_PREFIX}{batch_id}:total", total)
    r.set(f"{BATCH_PREFIX}{batch_id}:done", 0)
    r.set(f"{BATCH_PREFIX}{batch_id}:fail", 0)

def mark_done(batch_id: str, success: bool):
    key = f"{BATCH_PREFIX}{batch_id}:done" if success \
          else f"{BATCH_PREFIX}{batch_id}:fail"
    r.incr(key)

def get_progress(batch_id: str) -> dict:
    total = int(r.get(f"{BATCH_PREFIX}{batch_id}:total") or 0)
    done = int(r.get(f"{BATCH_PREFIX}{batch_id}:done") or 0)
    fail = int(r.get(f"{BATCH_PREFIX}{batch_id}:fail") or 0)
    return {
        "total": total, "done": done, "fail": fail,
        "percent": round((done + fail) / total * 100, 1) if total else 0
    }

常见问题（FAQ）

模板引擎渲染和 LLM 调用为什么要用 MQ 解耦？
渲染 1 万条只要几秒，但 LLM 调用 1 万条可能需要 30 分钟。耦合在一起意味着「渲染线程要等到 LLM 返回才能渲染下一条」，吞吐量受限于单线程。
批量生成时 LLM 的并发限制怎么处理？
在 Worker 侧做并发控制。Python 用 Semaphore + ThreadPoolExecutor 限制最大并发数（如 10 路并发）。Spring 用 @Async 配合线程池配置。同时利用 MQ prefetch 控制每个 worker 同时消费的消息数。
批量生成中途失败了一半，怎么恢复？
死信队列中的消息可以重新入队（requeue）。关键是每条任务要有幂等 ID，重新执行时不会生成重复内容。失败原因记录到日志，方便排查。

本章结论

  • 模板引擎：Jinja2 / Spring AI PromptTemplate，渲染性能 ≥ 1000 条/s。
  • MQ 削峰：RabbitMQ / Kafka 解耦渲染与 LLM 调用，Worker 可水平扩展。
  • 幂等消费：job_id 唯一标识，消费前检查避免重复生成。
  • 进度追踪：Redis 计数 + WebSocket 实时推送。
  • 失败处理：重试 3 次 → 死信队列 → 人工介入。

class AuditResult(Enum):

public class AuditPipeline {

    private final List<AuditStep> steps = List.of(
        new SensitiveWordStep(),

                item.addAuditLog(step.name(), r);
                return r;  // 非 PASS 即终止流水线
            }
        }
        return AuditResult.PASS;
    }
}

人工复核与驳回重写

human review 是审核流程中唯一的人工环节。设计要点：

  • 待审队列：按优先级排序（疑似违规 > 品牌调性 > 常规抽检）。
  • 审核面板：显示原文 + AI 生成内容 + 自动审核标记，一次对比。
  • 驳回原因：结构化选项（用词不当、事实错误、语调不符）+ 自由输入。

reject rewrite 的闭环流程：驳回原因 → LLM 根据原因重写 → 重新走审核流水线。

不推荐
  • 驳回原因填写「不行，重写」，无具体指向
  • 重写后直接发布，不走审核
  • 无重写次数限制，陷入死循环
  • 人工复核无 SLA，积压越来越多
推荐
  • 驳回原因结构化：类别 + 具体描述
  • 重写后必须重新审核
  • 最多重写 3 次，超限转人工写
  • 审核队列 SLA：1 小时内响应

常见问题（FAQ）

自动审核的敏感词词库怎么维护？
建立三层词库：① 行业合规层（广告法、医疗法规）——法务维护，季度更新；② 品牌自定义层（竞品名、品牌违禁词）——运营维护；③ 实时热词层（突发舆情相关词）——AI 自动标注+人工确认。
人工复核队列积压怎么处理？
设置三级 SLA：队列深度 < 50 条正常流转；50-200 条触发告警，自动加急最早进入的 20 条；> 200 条触发熔断——暂停批量生成，优先处理积压。
驳回重写会不会越改越差？
会。限制重写次数（≤3 次），每次重写只修改驳回原因指出的问题，不改变整体结构。如果 3 次后仍未通过，说明模板本身有问题，需要人工调整模板而不是继续改内容。

A/B 实验框架

ab test content 验证个性化方案是否真的有效。没有 A/B 实验的个性化是「自以为是的优化」。

不推荐
  • 新客老客同一套文案跑全量
  • 个性化上线后不看数据直接全量
  • A/B 实验只跑 1 小时就得出结论
  • 无统计显著性检验
推荐
  • A 组（通用）/ B 组（个性化）50:50 分流
  • 跑够 7 天或样本量 > 1000/组
  • 使用 χ² 检验确认显著性（p < 0.05）
  • CTR + 转化率 + CSAT 多维评估

# A/B 实验评估
from scipy.stats import chi2_contingency
import numpy as np

class ABTestEvaluator:
    def evaluate(self, control: list, variant: list) -> dict:
        """control/variant 各为 [impressions, clicks] 列表"""
        c_imp, c_clicks = control
        v_imp, v_clicks = variant

        c_ctr = c_clicks / c_imp if c_imp else 0
        v_ctr = v_clicks / v_imp if v_imp else 0

        # χ² 检验
        table = np.array([[c_clicks, c_imp - c_clicks],
                          [v_clicks, v_imp - v_clicks]])
        _, p, _, _ = chi2_contingency(table)

        return {
            "control_ctr": round(c_ctr, 4),
            "variant_ctr": round(v_ctr, 4),
            "lift": round((v_ctr - c_ctr) / c_ctr * 100, 1),
            "p_value": round(p, 4),
            "significant": p < 0.05
        }

ab = ABTestEvaluator()
result = ab.evaluate([5000, 150], [5000, 210])
print(f"CTR 提升 {result['lift']}%")
print(f"统计显著: {result['significant']}")

常见问题（FAQ）

个性化文案的缓存策略怎么设计？
以「分群 x 模板」为缓存键。5 个分群 x 10 个模板 = 50 个缓存条目，缓存命中率高。不要以 user_id 为缓存键。
用户标签更新频率多高？
CRM 层标签（分群、品类偏好）按日更新；行为层标签（近期浏览、购物车）按小时更新。prompt 生成时使用实时行为标签。
A/B 实验发现个性化没有显著提升怎么办？
三个方向排查：① 画像标签是否准确（分群是否正确）；② 变量差异化是否足够（5 个分群的 prompt 差异太小）；③ 指标选择是否合理（也有可能个性化后满意度提升但 CTR 没变）。

本章结论

  • 画像标签：5-10 个分群覆盖 90% 的个性化场景。
  • 动态变量：同一模板 + 不同变量 = 千人千面，不是千人千模板。
  • Profile Inject：画像标签在模板渲染时注入，全链路 < 500ms。
  • A/B 验证：50:50 分流 + 7 天 + p < 0.05 验证效果。
  • 缓存策略：分群 x 模板为缓存键，而不是 user_id。

# CorpAssist 内容自动评分
class ContentScorer:
    RUBRIC = """
    请对以下文案进行 5 维评分（0-5，5 为最优）：

    1. 品牌一致性：是否符合品牌语调规则（专业、温和）？
    2. 事实准确性：是否有事实错误或夸大？
    3. 可读性：语言流畅吗？段落结构合理吗？
    4. 合规性：是否含敏感词或违规表达？
    5. 创意性：与历史内容相比是否有区分度？

    输出 JSON：{"brand": N, "accuracy": N, "readability": N,
                "compliance": N, "creativity": N, "total_avg": N}
    """

    def __init__(self, judge_client):
        self.judge = judge_client

    def score(self, content: str, tone_rules: list[str]) -> dict:
        prompt = f"{self.RUBRIC}\n\n品牌语调规则：\n"
        prompt += "\n".join(f"- {r}" for r in tone_rules)
        prompt += f"\n\n文案内容：\n{content}"
        result = self.judge.chat(prompt)
        return json.loads(result)

    def batch_score(self, contents: list, tone_rules: list) -> list:
        scores = []
        for c in contents:
            scores.append(self.score(c, tone_rules))
        return scores

    def weekly_report(self, contents: list) -> dict:
        scores = self.batch_score(contents)
        dims = ["brand", "accuracy", "readability", "compliance", "creativity"]
        return {
            dim: round(sum(s[dim] for s in scores) / len(scores), 2)
            for dim in dims
        } | {"total": round(
            sum(s["total_avg"] for s in scores) / len(scores), 2)}

常见问题（FAQ）

LLM-as-Judge 的评分一致性如何保证？
用同一评分 prompt 跑 3 次取均值。如果某条内容的 3 次评分标准差 > 0.5（5 分制），说明该内容争议大，标记为「需要人工评分」。
坏例库多大合适？
初始库 100-200 条即可覆盖主要问题类型。关键是分类准确（属于「品牌不一致」还是「事实错误」），而不是数量多。每两周评审一次，归档已修复的坏例。
自动评分和人工评分偏差很大怎么办？
偏差 > 1 分的样本需要分析：是评分 prompt 不准确，还是人类评分的标准不一致。用 3 个人同时评同一批内容，一致性（ICC）> 0.7 后才作为人工基线。

本章结论

  • 5 维评分：品牌一致、事实准确、可读性、合规、创意，各有权重。
  • LLM-as-Judge：独立模型 + 独立 prompt + 3 次取均值。
  • 人工 Rubric：校准自动评分偏差，抽检 20% 最低分 + 10% 随机。
  • 坏例库：分类归因、记录完整上下文、驱动模板迭代。
  • 评估飞轮：评分 → 分析 → 优化 → 回归验证。

    async def event_stream():
        async for chunk in llm_client.chat_stream(prompt):
            yield f"data: {json.dumps({'text': chunk})}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(event_stream(),
                             media_type="text/event-stream")

@app.post("/v1/batch")
async def create_batch(req: BatchRequest):
    """提交批量生成任务"""
    batch_id = producer.publish_batch(req.template_id, req.variables_csv)
    return {"batch_id": batch_id, "status": "submitted"}
先想 10 秒：为什么不直接用 Python 全栈，还需要 Spring BFF？

Python 全栈在鉴权、会话管理、企业合规（审计日志、租户隔离）方面生态不如 Java。Spring BFF 作为坚固的前端网关，Python 服务可以专注于 LLM 核心能力。

Spring BFF 编排

spring orchestrate 负责将前端的请求编排到 Python 服务，并通过 SSE 流式返回。

Java：Spring WebFlux 编排

// Spring BFF 编排 Python 生成服务
@RestController
@RequestMapping("/api/v1")
public class ContentBffController {

    private final WebClient pythonClient;

    @PostMapping("/generate")
    public Mono<GenerateResponse> generate(@RequestBody GenerateRequest req) {
        return pythonClient.post()
            .uri("/v1/generate")
            .bodyValue(req)
            .retrieve()
            .bodyToMono(GenerateResponse.class);
    }

    @GetMapping(value = "/generate/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> generateStream(
            @RequestParam String templateId,
            @RequestParam Map<String, String> vars) {

        return pythonClient.post()
            .uri("/v1/generate/stream")
            .bodyValue(new GenerateRequest(templateId, vars, true))
            .retrieve()
            .bodyToFlux(String.class)
            .map(data -> ServerSentEvent.<String>builder(data).build())
            .doOnError(e -> log.error("Stream error", e));
    }
}

                       Map<String, String> variables,
                       boolean stream) {}
record GenerateResponse(String jobId, String content,
                        Map<String, Double> score,
                        String auditResult) {}

bff compose 负责组合多个 Python 服务的结果。例如「生成 + 审核 + 评分」可以并行调 Python 的三个独立服务，Spring 等待所有结果后返回。

常见问题（FAQ）

Python 和 Spring 之间用什么协议通信？
HTTP/REST，通过 Spring WebClient（非阻塞）。批量场景通过 RabbitMQ 解耦，不直接 HTTP 调用。
双栈部署时要考虑什么？
Python 服务需要 GPU 实例（如 P100/V100 按需），Spring BFF 只需要普通 CPU 实例。网络延迟：双栈之间最好同区域部署，延迟 < 5ms。
API 升级如何做灰度？
Python 服务保留 2 个 API 版本（/v1/generate 和 /v2/generate），Spring BFF 通过配置切换版本。旧版本废弃前保留至少 1 个月。

本章结论

  • 双栈分工：Python 生成 + Spring 编排，各司其职。
  • Python FastAPI：/v1/generate 单条 + /v1/batch 批量 + SSE 流式。
  • Spring BFF：鉴权、编排、SSE 推送，WebClient 非阻塞调用。
  • SSE 流式：浏览器原生支持的流式输出协议。
  • 契约测试：确保双栈 API 的兼容性。

本章先记住 3 件事
  • 真查库存再承诺：促销文案说「限时 8 折」时必须调库存 API 确认商品有货，否则就是虚假宣传。
  • 解码约束 ≠ prompt 约束：prompt 说「不要输出价格」但 LLM 可能仍输出——解码层必须硬约束。
  • 记忆分三层：短期（对话上下文）≠ 长期（用户偏好）≠ 全局（产品知识），层间不互通。

坑 1：过度承诺——未接库存/政策 API

inventory api truth 要求所有涉及「库存、价格、时效」的文案在生成前调真实 API 验证。避免 LLM 编造不可用信息。

CorpAssist · 虚假促销灾难（30 秒） LLM 生成了一条「春季精华 5 折限量 100 份」的文案发布，但实际上该 SKU 库存仅 30 份且未设限时折扣。用户涌入后无法兑现，引发客诉。根因：生成时未调库存 API 验证折扣与库存量。修复：模板变量中强制注入 API 返回的 real_stock 和 real_price。
不推荐
  • prompt 中写死「限时 5 折」，不查 ERP
  • 变量值由运营手动填写，无 API 校验
  • 生成后再调 API，不一致才告警
推荐
  • 渲染前先调库存/政策 API，结果注入变量
  • API 不可用时降级为通用文案，不做承诺
  • 模板中标记 required_api 字段

Python：库存 API 注入

python 
# 库存 API 真查注入
def enrich_with_inventory(variables: dict) -> dict:
    """渲染前从 ERP 查真实库存"""
    sku = variables.get("sku")
    try:
        inv = erp_client.get_inventory(sku)
        variables["real_stock"] = inv.available
        variables["real_price"] = inv.price
        variables["can_promote"] = inv.available >= inv.min_promo_qty
    except Exception:
        # API 不可用，降级
        variables["can_promote"] = False
        variables["fallback"] = True
    return variables

坑 2：解码约束——对抗样本

decode constraint 指在模型解码层面强制约束输出，而非仅靠 prompt。例如禁止输出特定词汇、强制 JSON 格式。

adversarial sample 是用于测试审核流水线的对抗性输入。通过构造可能绕过审核的样本（如谐音词、拆字、隐写），检验自动审核的鲁棒性。

坑 3：记忆分层——防止上下文混层

memory tier 将记忆分为三层：短期记忆（当前对话/会话）、长期记忆（用户偏好画像）、全局知识（产品信息）。关键原则：层间不互通——短期记忆不能写入长期，全局知识不携带用户身份。

层级内容存活期访问控制
短期（Session） 本轮对话内容 会话结束清除 仅当前会话
长期（Profile） 用户偏好、历史选择 用户级别持久 需用户授权
全局（Knowledge） 产品详情、品牌规则 永久 开放

坑 4：S5-B 角色 OOC

ooc detect 针对 S5-B 选修场景（娱乐/品牌 IP 角色）。当 LLM 扮演固定角色时可能出现 Out-of-Character ——比如品牌 IP 突然说了不符合人设的话。用角色一致性检测器在每次输出后做校验。

常见问题（FAQ）

库存 API 查询超时怎么办？
降级策略：不生成含具体价格/库存的文案，改用「限时优惠」「数量有限」等模糊表述。在模板中用 conditional block 控制，API 不可用时自动切到通用版本。
对抗样本需要维护多大？
初始 100-200 条覆盖常见绕过手段（谐音、拆字、英文替换）。每月新增 20-50 条，来自实际运营中发现的漏报案例。
记忆分层后，用户要求「记住我的偏好」时怎么跨层写入？
通过显式 API：用户说「记住我喜欢清爽型产品」触发表层 Profile 写入接口。LLM 不能直接把 Session 内容写 Profile，必须经用户确认。

  • 记忆分层：Session / Profile / Knowledge 三层隔离。
  • 角色 OOC：角色一致性检测器，S5-B 选修场景。
