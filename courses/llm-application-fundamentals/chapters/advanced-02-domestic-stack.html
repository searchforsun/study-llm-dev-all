为什么要学本章

政企与金融客户常把「国产化 / 信创」写进 RFP：模型、算力、中间件须可审计、可替换。CorpAssist 若只熟悉直连 DashScope，会在信创采购答辩与私有化交付上失分。本章把 模型选型 与 数据分级 延伸到通义 / 昇腾 / 本地 Qwen组合，并说明 Spring AI Alibaba 与自建 OpenAI 兼容 网关各管哪一层。

本章先记住 4 件事
  • 通义：DashScope / 百炼商用产品线，政企可走专属实例与 VPC；仍可通过 compatible-mode 走 OpenAI 兼容 端点。
  • 昇腾：信创算力栈（Ascend NPU + CANN + MindIE / vLLM 适配），决定「本地 Qwen 能不能跑、跑多快」。
  • 本地模型：Qwen 等权重在内网推理；Embedding 与对话模型须同域，避免 L3 向量出网。
  • 集成二选一为主：Spring AI Alibaba 深绑阿里云能力；CorpAssist 默认仍推荐网关 + OpenAI 兼容 做多厂商路由，Alibaba 模块作加速选项。

通义/昇腾/本地模型

CorpAssist · 信创私有化投标 客户要求：核心制度库（L3）数据不出园区；对话模型须国产算力可验收；Embedding 与 RAG 索引同机房。架构师需在标书中写清：通义商用是否仅作非敏感模块、昇腾池跑哪档 Qwen、向量服务如何与对话服务同域部署。

信创选型不是「换一个 API Key」，而是模型 + 算力 + 部署形态的组合。CorpAssist 典型分层：

  • 通义（DashScope / 百炼）：L1/L2 模块或专线/VPC 内的 L3 子集；按 Token 计费，迭代快；合同覆盖「不用于训练」「境内驻留」。
  • 昇腾（Ascend）：信创机房 GPU/NPU 池；承载 Qwen 等权重的推理服务（MindIE、部分 vLLM 昇腾版）；需单独容量规划与驱动/CANN 版本矩阵。
  • 本地模型（Qwen 权重等）：完全内网；对话与 Embedding 均私有化；你方负责量化、升级与安全补丁。

L3 制度 RAG：检索与 Embedding 不出推理区；对话走内网 Qwen；L2 客服可经网关分流至通义专线。网关仍暴露 /v1/chat/completions，便于 Python 与 Spring 客户端复用。

反模式
  • L3 文档在公有云做 Embedding，仅对话走内网
  • 采购昇腾卡但未规划 CANN / 推理框架版本，上线前才发现模型不支持
  • 通义与本地 Qwen 混用，却无统一评测集，回答风格与引用格式不一致
推荐
  • 向量与对话同域；网关按 dataClass 路由（见 数据、隐私与合规入门）
  • 标书附算力验收指标：TTFT P95、并发、上下文长度
  • 主模型切换前跑 golden set，对比引用 ID 与拒答率
先想 10 秒：客户要求「全国产」，是否必须放弃通义 DashScope？

不一定。「国产」常指可审计的国产栈组合：算力（昇腾）、模型权重（Qwen）、中间件（国产 OS/DB）可验收；通义可作为非敏感模块或VPC 专线组件写入合同，但 L3/L4 默认仍走本地推理。答辩时区分「逻辑国产」与「物理不出园区」。

Spring AI Alibaba 生态位

CorpAssist Java 栈长期采用 Spring AI 抽象（ChatClient、RAG Advisor）。Spring AI Alibaba 在阿里生态中提供 DashScope、百炼、通义 Embedding 等一等公民 Starter，与 OpenAI 兼容网关 路线并行存在。

两条集成路径对比

维度Spring AI Alibaba（DashScope 原生）统一 llm-gateway（OpenAI 兼容）
适用场景 已采购百炼、希望直接用通义特有参数（搜索增强、工具链） 多厂商路由：通义 + DeepSeek + 内网 Qwen；Python/Java 共用 base_url
CorpAssist 默认 政企单云、阿里云全家桶交付时可作主路径 推荐默认：与 分层架构 中 llm-gateway 一致
切换成本 换非阿里模型需改依赖与配置 改网关 YAML 路由即可（见下节代码块）
信创答辩 强调「阿里云信创认证 + 专属实例」 强调「协议中立 + 内网 Qwen on 昇腾」可替换厂商
RAG / Embedding 通义 Embedding Starter 与向量存储集成 内网 Embedding 服务 + 任意向量库，不绑公有云

实践建议：业务代码只依赖 Spring AI 抽象；是否引入 Spring AI Alibaba 作为一个 Provider 实现，还是全部指向 spring.ai.openai.base-url=${LLM_GATEWAY}，由交付形态决定。CorpAssist 产品化版本倾向后者，避免每个客户 fork 一套阿里专用代码。

与 微调 的关系

Spring AI Alibaba 可对接百炼模型定制能力；若 CorpAssist 仅在本地昇腾上做 LoRA，则微调产物绑定内网推理镜像版本，与公有云定制模型不互通。选型时问清：定制模型能否导出到私有化集群，还是只能在 DashScope 调用。

配置示意：网关路由（OpenAI 兼容）

yaml 
# CorpAssist llm-gateway · 信创混合路由（OpenAI 兼容）
# Spring AI: spring.ai.openai.base-url=https://llm-gateway.corp.example/v1
routes:
  - scene: policy-rag
    data-class: L3
    provider: qwen-ascend-private
    model: qwen2.5-72b-instruct
    base-url: http://qwen-inference.llm.svc.cluster.local/v1
    hardware: ascend-910b
  - scene: customer-chat
    data-class: L2
    provider: dashscope-vpc
    model: qwen-plus
    base-url: https://dashscope-vpc.aliyuncs.com/compatible-mode/v1
  - scene: marketing-copy
    data-class: L1
    provider: dashscope
    model: qwen-turbo
    base-url: https://dashscope.aliyuncs.com/compatible-mode/v1
embedding:
  default:
    provider: qwen-embedding-private
    model: text-embedding-v3
    base-url: http://embedding-service.llm.svc.cluster.local/v1
    # L3 索引必须与对话同域；禁止 fallback 到公有 Embedding
    allow-public-fallback: false
fallback:
  on-rate-limit: retry-after-backoff
  on-timeout: route-to-backup-model
  # 信创场景：禁止 L3 自动降级到 L1 公有路由
  deny-data-class-downgrade: true

Python rag-service 与 Spring AI ChatClient 共用上述 base-url；仅网关持有各 Provider 的 API Key 或 mTLS 证书。这与 首次打通模型调用 中的双栈实验一致，只是把 provider 换成昇腾上的本地 Qwen。

采购评估表

CorpAssist · 信创采购评审 采购部收到三家方案：① 通义专属实例 + 公有 Embedding；② 全昇腾私有化 Qwen；③ 混合云 + 统一网关。信息安全、财务与架构需在同一张表上打分，避免「便宜但 L3 出网」或「全私有化但无 SLA」。

信创 RFP 除单价外，必须覆盖可验收项。下表可直接用于 CorpAssist 投标附件或内部选型会；证据列填官网链接、测试报告编号或「需合同附件」。

三家方案快速对照（口述用）

  • 方案 A · 通义为主：上线快；L3 须 VPC/专属实例 + 内网 Embedding，否则不合规。
  • 方案 B · 全昇腾私有化：L3/L4 友好；CapEx 高，需自建监控与模型升级流程。
  • 方案 C · 混合 + 网关（CorpAssist 推荐）：L1/L2 通义、L3 本地 Qwen；采购表须写清路由与禁止降级策略。

迁移风险

从「公有 DashScope 直连」迁到「昇腾 + 本地 Qwen + 网关」时，常见风险不在网络连通，而在行为不一致与资产沉没。

迁移阶段与风险矩阵

阶段风险CorpAssist 缓解

RAG 重建 换 Embedding 模型导致召回率下降 全量 re-embed + A/B；冻结 index_version
评测回归 本地 Qwen 引用格式、拒答策略与通义不一致 golden set 门禁；评测通过率 SLI

运维能力 昇腾 驱动/CANN 升级导致推理中断 蓝绿推理池；容量压测纳入采购验收

本章结论

每条对应上一节 h3，advanced 阶段支撑信创投标与私有化交付答辩。

  • 通义/昇腾/本地模型：L3 默认昇腾 + 本地 Qwen + 同域 Embedding；通义用于 L2 或 VPC 子集。
  • Spring AI Alibaba 生态位：阿里云深度集成可选；CorpAssist 产品化优先网关 + OpenAI 兼容 多厂商路由。
  • 采购评估表：驻留、算力认证、Embedding 同域、契约、SLA/退出、TCO、微调导出须可验收。
  • 迁移风险：契约、re-embed、评测、微调资产与昇腾运维；分阶段灰度 + golden set 门禁。

动手练习

基于 CorpAssist 信创投标场景，完成混合路由 YAML 与采购评估表，并口述迁移三阶段计划。

操作步骤

  1. 复制本章「网关路由」YAML：为 S4 代码辅助（L3～L4）增加一条仅指向内网 Qwen 的路由，并设置 allow-public-fallback: false。
  2. 在采购评估表中为「Embedding 同域」填证据列：写出你的 Embedding 服务 URL 与向量库集群名（可用占位符）。
  3. 在部署拓扑 Mermaid 上标出：若 rag-service 与 embedding-service 不同可用区，合规风险是什么。
  4. 写一段 200 字「迁移阶段 0～2」说明：何时跑 golden set、何时关闭 L3 公有路由。

判断练习

  1. 投标方案写「对话用昇腾 Qwen，Embedding 用 DashScope 公有 API 以节省成本」。L3 制度库是否可接受？

    参考答案

    不可接受（默认 L3 要求）。制度原文切块会在公有云做 Embedding，构成逻辑出网；应改为内网 Embedding 与索引同域。若客户仅 L2 数据，须下调 dataClass 并书面备案，不能混用 L3 库。

  2. 团队已在百炼上做 LoRA 微调，现要迁昇腾私有化。最可能踩坑的一点是什么？

    参考答案

    微调权重无法导出或格式不兼容昇腾 推理镜像。缓解：迁移前在合同/平台确认导出能力；否则在私有化环境重新微调或回退到 RAG + Prompt，并用 golden set 验收，避免假设「云端定制模型可直接下载」。
