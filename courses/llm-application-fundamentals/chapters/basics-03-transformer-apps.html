本章先记住 4 件事
  • Attention 是加权阅读：生成每个新 Token 时，模型会「回头看」上下文里哪些片段更相关。
  • 生成是逐字猜下一个：不是一次输出整段答案；API 的 流式输出 正是这个循环的外显。
  • 幻觉是统计补全的副作用：模型会流畅地「补全」看似合理但不存在的事实；制度问答必须加检索与引用。
  • 大模型不是万能函数：精确计算、强规则审批、毫秒级确定性逻辑，往往用传统代码更便宜可靠。

Attention 直觉（无公式推导）

CorpAssist · 制度问答 员工问：「驻外补贴能否与差旅报销叠加？」系统把《驻外管理办法》第 3 章与 FAQ 片段一并送入模型。模型在组织回答时，需要更关注与「叠加」相关的段落，而不是无关的入职流程。

主流大模型基于 Transformer 架构。对应用开发者而言，只需抓住一个直觉：Attention（注意力）= 在读完全文后，决定「此刻该重点参考哪几句」的加权机制。它让模型在处理长序列时，不必像老式 RNN 那样「读完就忘前面」，而是在 上下文窗口 内动态分配「关注度」。

类比：带高亮的阅读理解

  • 输入：系统提示 + 用户问题 +（可选）检索到的制度片段，全部被切成 Token 序列。
  • 生成每个新 Token 时：模型对上下文各位置赋予不同「权重」，相当于自动高亮最相关的句子。
  • 输出：按权重融合后的语义，预测「下一个最可能出现的 Token」——这就是后续「生成过程」的单元操作。
算法岗视角（本章不展开）
  • Query / Key / Value 矩阵乘法
  • 多头注意力、位置编码推导
  • 训练损失与反向传播
应用岗视角（本章重点）
  • 上下文越长，Attention 可「看见」的素材越多，但受窗口与费用约束（见 Token、上下文窗口与费用）
  • 把相关文档放进上下文，比指望模型「凭空记住」企业制度更可靠
  • 系统提示与 提示词工程 决定模型优先「关注」哪类指令
Attention 直觉 · 生成「叠加」相关回答时
flowchart LR
  Q[用户问题: 补贴能否叠加?] --> CTX[上下文 Token 序列]
  CTX --> A1[制度第3章 权重高]
  CTX --> A2[入职流程 权重低]
  CTX --> A3[FAQ 片段 权重中]
  A1 --> OUT[加权融合后预测下一 Token]
  A2 --> OUT
  A3 --> OUT

不必记公式：把 Attention 想成「对上下文做动态高亮，再据此续写下一个 Token」。

CorpAssist 工程含义

制度问答若只塞入整本 200 页 PDF 而不检索，Attention 仍会在窗口内分配权重，但无关段落会稀释有效信号，且 Token 费用暴涨。后续 RAG 课程会讲「先检索再生成」——本质是给 Attention 一份更干净的阅读材料。

生成过程与温度

大模型对话不是「一次性返回完整 JSON」，而是自回归（Autoregressive）：每产出 1 个 Token，就把它拼回上下文，再预测下一个，直到遇到停止条件（如 max_tokens、停止词或模型输出结束符）。

自回归生成循环

CorpAssist 单次回复 · 生成循环
flowchart TB
  START([收到 messages]) --> ENC[编码为 Token 序列]
  ENC --> LOOP{未达停止条件?}
  LOOP -->|是| PRED[模型预测下一 Token 概率分布]
  PRED --> SAMPLE[按 temperature / top_p 采样]
  SAMPLE --> APPEND[追加到输出与上下文]
  APPEND --> STREAM{流式开启?}
  STREAM -->|是| SSE[推送 SSE chunk 给前端]
  STREAM -->|否| BUF[累积在内存]
  SSE --> LOOP
  BUF --> LOOP
  LOOP -->|否| END([返回完整 assistant 消息])

你在网关看到的「打字机效果」，就是循环中每次 SAMPLE 后推送的一个 chunk；与 流式输出 章节将讨论的 SSE 一致。

temperature 与 top_p：控制「有多敢编」

预测下一 Token 时，模型会先给出候选词的概率分布，再采样出一个。两个最常用的 API 参数：

  • temperature：越高，低概率候选越有机会被选中 → 措辞更多样，但也更不稳定。
  • top_p（nucleus sampling）：只在累计概率达到 p 的最小候选集合里采样 → 砍掉长尾胡话，同时保留一定多样性。
CorpAssist 场景temperaturetop_p原因
内部客服 / 制度答复 0～0.3 0.8～1.0 措辞应稳定、可复核；避免同问不同答
营销海报文案头脑风暴 0.7～0.9 0.9～1.0 需要多种创意方向供人挑选
代码补全 / 结构化 JSON 提取 0～0.2 1.0 格式错误代价高；可配合 JSON mode

API 请求体示例：temperature 与 top_p

json 
{
  "model": "qwen-plus",
  "messages": [
    {
      "role": "system",
      "content": "你是 CorpAssist 企业助手。仅依据提供的制度片段回答；无依据时明确说「未找到相关规定」。"
    },
    {
      "role": "user",
      "content": "驻外补贴能否与差旅报销叠加？"
    }
  ],
  "temperature": 0.2,
  "top_p": 0.9,
  "max_tokens": 512,
  "stream": true
}

实践建议：一般先固定 temperature，再用 top_p 微调；两者同时拉太高时，制度问答容易出现「每次说法略不同」，不利于审计与客诉追溯。

同一问题多次调用，答案开头几乎一致；适合 CorpAssist 工单小结、合规答复模板填充。

同一问题多次调用，比喻与措辞差异明显；适合宣传语 A/B 候选，但不可直接作为唯一制度依据。

幻觉与局限

幻觉（Hallucination）指模型自信地输出看似合理、但与事实或上下文不符的内容。它不是偶发 bug，而是「按概率续写」机制的结构性风险：模型被训练成总是给出流畅答案，而不是「不知道就说不知道」。

CorpAssist · 反例 员工问某冷门补贴的审批链。模型未检索到条款，却编造了「需 VP 与 HRBP 双签」——格式正确、语气官方，但制度里根本没有这条流程。

常见幻觉类型（应用侧）

  • 事实性幻觉：捏造不存在的政策条款、法条、产品参数。
  • 引用幻觉：给出假的文档章节号、链接或「根据 XX 文件第 N 条」。
  • 推理跳跃：从部分正确前提推出错误结论，且不留不确定性表述。

为何单靠 Prompt 不够

在 system 里写「不要编造」能降低频率，但不能从机制上保证。CorpAssist 制度类问答应默认走 RAG（检索增强生成）：先用 Embedding 从向量库召回片段，再让模型仅基于召回内容组织语言，并返回引用来源供人工复核。

RAG 缓解幻觉 · CorpAssist 知识库问答
flowchart LR
  U[用户问题] --> RET[向量 + 关键词检索]
  RET --> CHUNKS[Top-K 制度片段]
  CHUNKS --> PROMPT[拼入 messages 上下文]
  PROMPT --> LLM[大模型生成]
  LLM --> ANS[回答 + 引用列表]
  ANS --> AUDIT[用户点击溯源核对]

RAG 不消除幻觉，但把「可核对的事实」前置到上下文中；无召回时应触发拒答话术而非硬答。

其他固有局限

局限对 CorpAssist 的影响工程应对方向
知识截止 / 无实时性 答不了「今天会议室是否空闲」 接工具 / API，而非只靠预训练权重
算术与精确逻辑 复杂费率加总可能算错 关键计算交给 Java 服务或 SQL
上下文长度 超长会议纪要无法整篇塞入 摘要、分块检索（见 Token 章）
延迟与成本 大窗口 + 高并发时 SLA 吃紧 缓存、小模型路由、异步任务
先想 10 秒：CorpAssist 能否只靠「更强的 system prompt」替代 RAG，解决制度编造问题？

不能作为唯一手段。Prompt 无法提供模型未见过的企业私有条款；RAG 解决的是知识从哪来，Prompt 解决的是怎么说、何时拒答。两者叠加才符合合规预期。

何时不用大模型

在 从大模型到应用工程 中已强调：LLM 是产业链中的一环。下列场景应优先传统工程，或仅让模型做「辅助」而非「裁决」。

场景决策表

信号更合适的做法CorpAssist 示例
输出必须 100% 可复现、可审计 规则引擎 / 工作流 年假天数计算：按工龄表由 Java 服务算，LLM 只解释结果
毫秒级延迟、高 QPS 缓存 + 确定性逻辑 登录验证码校验不走大模型
训练数据无法覆盖的私有知识 RAG 或结构化查询 制度库问答：检索 + 引用，而非裸聊
需要精确数值 / 账务 SQL、计算器工具、Agent 调确定性 API 报销总额：LLM 解析意图，金额由 ERP 接口返回
风格固定、话术极窄 模板 + 槽位填充 标准拒信：模板优先；微调 仅在模板无法覆盖时再评估
安全敏感、强合规审批 人工复核 + 最小权限 裁员补偿方案：模型可起草要点，最终由 HR 与法务签批
「那我们还要上大模型吗？」

要，但用在非结构化理解、草稿生成、多轮澄清、跨文档综合上。CorpAssist 的价值是把 LLM 放在编排层，与 Java 微服务、检索、审批流组合——而不是用聊天框替代 ERP。

本章结论

每条对应上一节 h3，用于评审前 30 秒复述。

  • Attention 直觉：不必推公式；把它当作对上下文做动态加权阅读，生成每个 Token 时决定「重点参考哪几句」。
  • 生成与温度：自回归循环逐 Token 采样；客服宜低温稳定，创意文案可高温；temperature 与 top_p 在 API JSON 中显式配置。
  • 幻觉与局限：流畅补全必然带来编造风险；制度类场景靠 RAG + 引用溯源，精确逻辑与实时数据应交给传统服务或工具。
  • 何时不用：强审计、毫秒确定性、纯算术、窄模板话术等场景，用大模型往往贵且不可靠；LLM 适合非结构化理解与草稿，不适合独自做裁决。
