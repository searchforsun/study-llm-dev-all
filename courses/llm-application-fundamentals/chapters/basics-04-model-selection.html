已完成模型选型：商用 API vs 开源

约 28 分钟 · 阶段：应用问题域与模型认知 · 能力：用能力/价格/合规三维为 CorpAssist 选模型与部署形态

为什么要学本章

Token、上下文窗口与费用 让你会算钱；Transformer 应用层理解 让你知道模型能做什么、不能做什么。本章把「选哪家、买 API 还是自建」落到企业采购与架构语言：通义、DeepSeek、Qwen 开源权重在 CorpAssist 里各扮演什么角色。

学完你能
  • 用能力 / 价格 / 合规三维矩阵比较商用 API 与私有化，而不是只看榜单分数
  • 为 CorpAssist 的 RAG 知识库、客服、代码辅助三条路径各写一句模型取向
  • 说明何时优先 RAG 而非 微调，以及 OpenAI 兼容协议 如何降低多模型切换成本
本章先记住 4 件事
  • 商用 API：上线快、按 Token 计费；数据出网与合同条款是采购第一关。
  • 开源权重：可私有化与信创；你要承担 GPU、推理框架与版本升级，不是「免费午餐」。
  • 通义 vs Qwen：通义是阿里云上的商用产品线（DashScope）；Qwen 是模型族，既可 API 也可自托管权重。
  • 工程默认：知识库问答先 RAG + 好 Prompt；微调是数据与评测就绪后的选项。

能力/价格/合规矩阵

CorpAssist · 采购评审会 信息安全部要求「核心制度文档不出公网」；财务部要月度 Token 上限；业务方要客服首字延迟 < 1s。选型表必须同时回答能不能用、贵不贵、能不能过审，而不是只贴 MMLU 分数。

应用侧选型先看三维矩阵，再落到具体厂商。能力维关注任务匹配（长上下文、工具调用、中文政策表述）；价格维把输入/输出单价与峰值 QPS 乘进 Token 成本；合规维看数据分级、驻留地域、审计与 SLA。

商用 API vs 开源权重（总览）

维度商用 API（通义 / DeepSeek 等）开源权重自托管（Qwen 等）
能力 厂商持续迭代；长上下文、多模态、工具调用由产品线打包 可选型号多；效果取决于量化、推理框架与是否配套 Embedding 服务
价格 按 Token + 阶梯价；无 GPU CapEx，适合流量波动 GPU/机房 OpEx + 运维；流量稳定且规模大时单位成本可能更低
合规 依赖合同、专属实例、VPC 专线；仍可能「逻辑出网」 数据可完全在内网；须自证安全补丁、日志与访问控制
上线速度 天级：申请 Key、走 OpenAI 兼容 网关即可联调 周～月级：选型卡、vLLM/Ollama、容量与监控

CorpAssist 模块 × 三维关注点

CorpAssist 模块能力侧重价格敏感点合规门槛
S1 制度 RAG 知识库 长 上下文、引用忠实、少 幻觉 检索片段 + 答案 Token 双份计费 制度 PDF 常 L3/L4，倾向专有云或私有化
S2 智能客服 低延迟流式、稳定话术、多轮记忆 高峰 QPS × 短回复仍积少成多 客户 PII 脱敏；可接受公有云脱敏后出网
S4 代码辅助 代码补全与解释；可选独立小模型 开发者并发低于客服峰值 源码仓库不出网 → 私有化或专线实例
常见误区
  • 只看公开榜单选模型 — 企业场景更看中文政策表述、长文档 RAG、工具调用与合同 SLA。
  • 把「开源」等同于零成本 — GPU、推理、监控与升级人力必须入账。
  • 一上来就 微调 — 多数知识问答应先把 RAG 与 提示词 做透。
先想 10 秒：CorpAssist 制度库若绝对不能出公网，矩阵里哪一列优先？

优先开源权重私有化或云厂商专属/VPC 实例；商用公有 API 仅在经过脱敏、摘要且法务书面批准的数据分级下使用。

通义/DeepSeek/Qwen 对比

三家在中文企业场景常被并列评估。表述上：通义千问是阿里云 DashScope 上的商用品牌；Qwen是阿里通义开源的模型族（权重可自托管）；DeepSeek以高性价比 API 与开源权重并行。下表按 CorpAssist 典型路径做应用侧对比（具体单价以当期官网为准）。

三家在 CorpAssist 场景的对照

对比项通义（DashScope）DeepSeek APIQwen（开源 / 自托管）
接入方式 DashScope API；OpenAI 兼容 端点，便于 Spring AI / Python 统一 base_url 官方 API，兼容 OpenAI SDK；适合作为「性价比副模型」 Hugging Face 权重 + vLLM/Ollama；需自建 向量 与评测
CorpAssist S1 知识库 长上下文商用型号成熟；与阿里云生态、国内合规采购路径顺 长文本与推理型号性价比高；适合非敏感摘要层或内部试点 制度库不出网时首选；可选 Qwen2.5 等 + 本地 Embedding
CorpAssist S2 客服 流式延迟与稳定性在公有云 SLA 下可预期 单价友好，适合高峰削峰或「简单意图」路由到便宜模型 需自证并发与流式 TTFT；运维成本高
CorpAssist S4 代码辅助 可选代码增强型号；与钉钉/阿里云 DevOps 集成方便 代码与数学推理口碑好；可作 Copilot 备选通道 内网代码库场景；注意显卡规格与上下文长度
切换成本 统一走 OpenAI 兼容网关 时，主要改 base_url、model 与限流策略，业务代码少改

通义 DashScope：采购合同、发票、专属实例与内容安全策略一站式；适合 CorpAssist 先上线 MVP、数据分级允许出网的模块。

Qwen 权重：同一模型族可在内网复现能力边界；适合信创机房、与现有 K8s GPU 池集成；你方负责版本升级与安全补丁。

推荐组合（多数企业）
  • 主通道：通义或合同内指定国产商用 API（制度/客服）
  • 副通道：DeepSeek 路由简单问答、草稿生成，控成本
  • 敏感通道：Qwen 私有化 + 内网 RAG，与公有云物理隔离
反模式
  • 每个业务线各申请一家 API，无统一网关与审计
  • 为「跟风口」同时微调三套模型，数据集与评测未统一

私有化边界

「私有化」在采购话术里可能指：专属公有云实例、VPC 专线访问商用 API、或完全离线自托管权重。CorpAssist 应按数据分级画边界，而不是一刀切全自建。

数据分级与部署形态

数据分级（示例）典型内容建议部署模型来源
L1 公开 已发布产品手册、官网 FAQ 公有云 API 通义 / DeepSeek 任选，走统一网关
L2 内部 内部 Wiki、会议纪要（已脱敏） 商用 API + 企业合约 / 专属实例 通义企业版或专线；日志脱敏
L3 敏感 人事制度、未公开财务口径 VPC 内推理或专有云 Qwen 私有化 + 内网 RAG；禁止明文出网
L4 机密 并购、客户合同原件 物理隔离区、离线或双域 仅本地模型；人工审批出域

何时不必私有化

  • 流量小、迭代快、数据已脱敏且法务书面同意使用公有 API。
  • 团队无 GPU 运维与模型升级能力 — 自托管反而扩大攻击面。
  • 问题可通过 RAG 引用溯源解决，而非靠 微调 背制度全文。
「专属实例」算私有化吗？
算逻辑隔离的一种：计算与存储在厂商侧专属于你，但权重与补丁仍由厂商维护。与完全离线自托管相比，运维轻、合规话术需写清数据驻留与审计接口。
微调是否必须在私有化里做？
训练数据若含 L3/L4，应在与推理同级的隔离域完成；多数 CorpAssist 场景先用 RAG，微调仅在有标注集、评测集与回滚方案时启动。

本章结论

选型 = 能力、价格、合规三维 + 场景决策矩阵，而非单一「最强模型」。

  • 能力/价格/合规矩阵：商用 API 快但受 Token 与合同约束；开源权重贵在人力和 GPU，换来数据驻留可控。
  • 通义/DeepSeek/Qwen 对比：通义偏主通道与政企采购；DeepSeek 偏性价比副通道；Qwen 权重偏 L3/L4 私有化与信创。
  • 私有化边界：按 L1～L4 分级选择公有云、专属实例或完全内网，勿为低敏感数据过度自建。
  • CorpAssist 选型决策：网关 + OpenAI 兼容多路由；知识库优先 RAG，微调在有数据与评测时再议。

判断练习

  1. （判断）CorpAssist 制度库已建 RAG 索引，但回答仍常编造不存在的条款。团队提议立即对通义做全量微调。是否应优先微调？应优先哪类手段？

    参考答案

    不应优先全量微调。 应先加强 RAG（切片质量、检索 Top-K、强制引用段落）、降低 temperature、在 Prompt 中要求「仅依据引用回答」并做引用溯源展示；评测集验证后再考虑小规模微调或换更强主模型。微调成本高且难回滚，无法替代坏检索。
