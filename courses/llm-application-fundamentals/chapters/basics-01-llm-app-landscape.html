大模型岗位 JD 常混用「算法 / 训练 / 应用 / Infra」——若不清楚应用工程在产业链哪一环，容易把精力投在不该优先掌握的领域。本章以 CorpAssist 企业智能助手 为统一案例，建立后续 RAG、Agent、API 调用的共同语言，并说明Python + Spring AI 双栈如何承接你已有的 Java 分布式经验。

应用侧在产业链的位置

从大模型到可上线产品，产业链可粗分为四层。作为应用开发者，你的主战场在「应用编排 + 数据 + 评测」，而非预训练或显卡集群运维。

应用侧消费模型 API，用 Embedding 与检索增强数据层，用评测闭环保证「能答、敢答、可审计」。

层级典型角色应用开发者是否主责
基座 / 训练 算法科学家、训练工程师 否 — 选型与估能力即可
推理 Infra MLOps、平台 SRE 部分 — 了解延迟、配额、私有化边界
应用工程 LLM 应用工程师、全栈 / 后端 是 — 编排、RAG、Agent、集成
业务交付 产品、领域专家、安全合规 协作 — 需求拆解、话术与审计

CorpAssist 不会要求你手写 Transformer；但会要求你设计：知识如何入库、Token 预算如何控、工具调用如何鉴权——这些在 Token、上下文窗口与费用 及后续实践章展开。

先想 10 秒：你们公司「大模型项目」更像买 API 接系统，还是自建训练集群？

多数企业智能助手属于前者：应用工程 + 私有化路由；训练集群通常由平台或厂商承担。

CorpAssist 业务全景

CorpAssist · 企业智能助手（30 秒） 某集团上线统一助手：员工问制度、客户问工单、运营要文案、研发要代码补全。平台组提供模型网关与审计，各业务线以模块方式接入，共享账号体系与知识分级策略。

本课用 CorpAssist 贯穿全系列。五大模块与常见技术映射如下：

模块业务场景典型能力本课后续章节
S1 企业知识库 制度 / 手册 / 标书问答 RAG、向量检索、引用溯源 practice 阶段架构与数据合规
S2 智能客服 工单摘要、多轮对话、转人工 提示词模板、会话状态、流式输出 basics API 与上下文控制
S3 办公 Agent 审批流、查 ERP、发通知 Agent、工具调用、规划循环 practice 架构中的编排层
S4 代码辅助 内部框架示例、SQL 草稿 代码补全、仓库上下文（慎出网） advanced 多模态与合规边界
S5 内容生成 营销话术、周报、邮件 结构化输出、风格约束 温度参数与评测（见 basics-03）

S1 常为 S2/S3 提供检索上下文；各模块经统一网关做模型路由、Token 计量与日志脱敏。

选型直觉
  • 「答得准、可引用」→ 优先 RAG，而非一上来 微调。
  • 「要替人点系统、跑流程」→ Agent + 工具白名单。
  • 「口径必须极稳」→ Prompt + 评测集 + 人工抽检，微调仅在有充足标注与合规审批时考虑。

双栈分工预览：Python 验证 + Spring AI 落地

本系列采用双轨：Python 适合 Notebook / 脚本快速试 Prompt 与 RAG 链路；Spring AI 适合接入现有注册中心、配置中心与企业安全规范。二者共享 OpenAI 兼容 的 base_url 与契约。

yaml 
# CorpAssist 本地开发：模型网关（Spring AI 与 Python 共用）
corpassist:
  llm:
    base-url: https://llm-gateway.corp.example/v1
    api-key: ${CORPASSIST_LLM_API_KEY}
    default-model: qwen-plus
  modules:
    s1-knowledge-base:
      embedding-model: text-embedding-v3
      index: corpassist-hr-policies
    s2-customer-service:
      max-history-turns: 8
    s3-agent:
      allowed-tools: [erp.lookup, notify.send]
    s4-code-assist:
      repo-scope: internal-spring-templates
    s5-content:
      temperature: 0.7
不推荐
  • 每个业务模块直连不同厂商 API，密钥散落
  • 把整库 PDF 每次全塞进 Prompt
  • 无评测集就上生产「感觉还行」
推荐
  • 统一网关 + 模块级路由与配额
  • RAG 只注入 Top-K 片段，控制 Token
  • 黄金问答集回归后再放量（见 practice 可观测章）

首次打通调用的动手实验见 practice 章；本章只需建立「编排层 ≈ 你熟悉的 BFF + 领域服务」心智。

简历可写：「在 CorpAssist 中负责 S1 知识库 RAG 链路，将制度问答准确率从 X% 提到 Y%，并通过网关统一 Token 成本」——比空泛「熟悉大模型」更有说服力。

本章结论

  • 产业链位置：应用工程站在模型 API 与业务之间，负责编排、数据、评测与合规，而非预训练。
  • CorpAssist 全景：S1 知识库、S2 客服、S3 Agent、S4 代码、S5 内容五模块，经统一网关共享审计与路由。
  • 经验迁移：网关≈模型路由，BFF≈编排层，搜索≈RAG；Python 快验、Spring AI 企业落地。
  • 岗位地图：JD 中的「大模型」多指应用工程；算法训练与 Infra 是协作方，不是本课主修。

  4. 阅读 yaml 代码块中的 corpassist.llm 与 modules，列出 Python 脚本与 Spring Boot 各应读取的 2 个配置键（双栈共用网关）。

判断练习

  1. 找错：同事提出 CorpAssist v1 方案 — ① 五个模块各申请一家厂商 API Key ② S1 用 RAG + Embedding ③ 每次客服对话附带 200 页 PDF 全文 ④ 统一模型网关做审计。哪条最违反本章推荐架构？

    参考答案

    答：① 与 ③。① 导致密钥与配额碎片化，违背统一网关；③ 会爆炸式消耗 Token 且降低命中率，应走 S1 检索 Top-K。② ④ 符合 RAG 与治理方向。

  2. 权衡：业务方希望「客服话术极度稳定」，应在 v1 优先加大 Prompt 模板与评测集，还是优先上 LoRA 微调？结合本章对 微调 边界的说明作答。

    参考答案

    优先 Prompt + 评测集 + 抽检。微调需要标注数据、版本管理与合规审批；v1 在话术可控、可快速迭代上，提示词与黄金集回归更划算。待数据与指标稳定后再评估微调。
