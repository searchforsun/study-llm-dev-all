为什么要学本章

在 API 契约与集成方式 中你已画出 BFF → LLM 网关 → 厂商的物理拓扑。本章把它上升为逻辑分层：哪些组件只做接入、谁负责 RAG 与 Agent 编排、数据与索引落在哪。没有分层，团队会在 BFF 里堆 Prompt、在业务服务里直连模型，最终无法治理 Token 成本与合规。

本章先记住 4 件事
  • 接入不智能：BFF + llm-gateway 管鉴权、限流、流式透传，不写 RAG 核心逻辑。
  • 编排在中间：dialog-orchestrator 组装 messages、调工具、控制步数预算。
  • 数据可演进：向量索引、文档版本、审计日志独立成数据层，与模型厂商解耦。
  • 双栈有边界：Python 擅检索/评测/Notebook 实验；Spring 擅租户、审计、与企业 IAM/配置中心集成。

一次 RAG 问答的调用序（概念）

  1. BFF 校验租户，附带 session_id 转发至 orchestrator。
  2. Orchestrator 调 rag-service：混合检索 → 得到 top-k 片段与 chunk_id。
  3. 拼装 messages（system + 引用块 + user），经 gateway POST /v1/chat/completions。
  4. 流式 chunk 经 BFF 写 SSE；末包 usage 写入审计仓，按部门归因 Token。
yaml 
# CorpAssist 分层职责配置片段（application-corpassist.yml）
corpassist:
  layers:
    access:
      bff: corpassist-bff          # 鉴权、聚合、SSE 透传
      gateway: llm-gateway         # OpenAI 兼容、路由、密钥、usage
    orchestration:
      dialog: dialog-orchestrator  # RAG 拼装、多轮、降级
      agent: agent-runtime         # 工具调用、步数预算 max_steps: 8
    data:
      rag_index: rag-index         # 向量集合 corpassist-hr-v3
      object_store: minio://docs
  rag:
    retrieval:
      top_k: 6
      rerank: true
    # 索引重建在数据层异步任务，不阻塞在线问答
    ingest:
      queue: corpassist-doc-ingest
      embedding_model: text-embedding-v3

同步 vs 异步

「同步」指用户或调用方阻塞等待一次 HTTP/SSE 往返内得到结果；「异步」指提交任务后立即返回 job_id，由消息队列或调度器在后台完成，结果通过轮询、Webhook 或站内信送达。

场景推荐模式CorpAssist 实现要点
Web 客服 / 制度问答 同步 + 流式 BFF 保持 SSE；编排层超时 60～120s；网关禁止缓冲
1000 页 PDF 全库重索引 异步 数据层 doc-ingest 队列；进度 API；失败可重试分片
夜间批量摘要、评测跑批 异步 非流式 chat/completions；Worker 消费；结果写对象存储
Agent 多步工具（查 ERP→发邮件） 同步长连接或异步任务 步数 > 3 或总 Token 高时改异步，前端轮询 /jobs/{id}
误用同步
  • HTTP 请求内同步遍历万篇文档做 Embedding
  • Agent 10+ 步工具调用堵在 BFF 线程池
  • 批处理走流式 SSE，占用网关长连接
推荐
  • 在线问答：同步编排 + 流式生成
  • 索引/评测：异步队列 + 可观测 job 状态
  • 超时后返回 job_id，而非无限挂起 HTTP
先想 10 秒：rag-service 检索 200ms，模型首 token 2s，整答 15s——应在哪一层设超时？

分层超时：BFF 对前端总超时略大于编排；编排对 RAG 单独 3～5s，对 gateway 读超时覆盖生成；网关对厂商再套一层。避免只用 BFF 一个 30s 掩盖 RAG 变慢。

双栈分工预览

CorpAssist 采用 Python AI 服务 + Spring 治理栈 并行：不是「二选一」，而是按层分工，经统一 OpenAI 兼容 网关与内部 REST 契约协作。

能力域Python AI 栈Spring 治理栈
典型服务 rag-service、评测脚本、Notebook 实验 corpassist-bff、llm-gateway、dialog-orchestrator
擅长 LangChain/LlamaIndex 原型、向量流水线、离线 Embedding 批处理 多租户、OAuth2、配置中心、与 hr/erp 现有 Feign 集成
模型调用 httpx / openai SDK → 同一 base_url Spring AI ChatClient → 同一 gateway
治理 容器化 + 资源限额；敏感逻辑仍走 Spring 审计入口 全链路 trace_id、操作审计、密钥不落 Python 配置仓库
java 
// Spring 编排层：调用 Python rag-service，再调统一网关
@Service
public class CorpAssistDialogService {
  private final RagClient ragClient;       // REST → rag-service (Python)
  private final ChatClient chatClient;   // Spring AI → llm-gateway

  public Flux<String> answerWithRag(String tenantId, String question) {
    List<RagChunk> chunks = ragClient.retrieve(tenantId, question, 6);
    String context = RagPromptBuilder.formatCitations(chunks);
    return chatClient.prompt()
        .system("仅依据引用回答；无依据则拒答。")
        .user(context + "\n\n问：" + question)
        .stream()
        .content();
  }
}

Python 侧 rag-service 暴露 POST /v1/retrieve，返回 chunk_id、文本与分数；不在 Python 进程内缓存厂商 API Key 给前端。后续 首次打通模型调用 将对照 httpx 与 Spring AI 配置项。

衔接 basics

双栈都依赖 BFF 与 OpenAI 兼容网关 的契约；分层架构决定哪一段用哪门语言，而不是两套互不相关的模型接入。

反模式列举

以下反模式在 Java 微服务团队接手 LLM 项目时高频出现。每条给出 CorpAssist 推荐替代。

#反模式后果推荐
1 智能堆在 BFF：BFF 内写 RAG、调 ERP 工具 无法独立扩缩容；前端协议变更牵动智能逻辑 下沉 dialog-orchestrator
2 业务服务直连模型：各服务自带 API Key 密钥泄露、Token 不可归因 仅 llm-gateway 持钥
3 无数据层版本：向量索引与 PDF 不同步 引用过期制度，合规风险 文档 version + 索引别名切换
4 先 微调 再建 RAG 知识更新慢、成本高 企业知识库优先 RAG；微调仅用于风格/格式
5 Agent 无步数与工具白名单 循环调用、费用爆炸 max_steps、工具审计、人工确认写操作
6 全链路同步：索引/Agent 长任务占 HTTP 线程 雪崩、网关连接耗尽 异步队列 + job 状态 API
  • BFF 解析 PDF 或跑 Embedding
  • 网关内嵌业务规则引擎替代 orchestrator
  • 对流式响应做全量缓冲再返回
  • 检索结果不记录 chunk_id，无法溯源
  • 把历史对话无限塞进 prompt 不用 上下文窗口 预算
  • 评测集与生产索引混用同一集合名

本章结论

四条对应四个 section，作为 practice 阶段架构共同语言。

  • 三层：接入（BFF+gateway）/ 编排（orchestrator+agent）/ 数据（rag-index+存储）；生成经 gateway，检索在数据与 rag-service。
  • 同步异步：用户可见用同步+流式；索引与批处理用异步；Agent 长链可转 job。
  • 双栈：Python 做 AI 流水线实验与检索；Spring 做 BFF、网关、编排与企业集成。
  • 反模式：忌 BFF 堆智能、业务直连模型、无索引版本、无 Agent 预算。

判断练习

  1. 产品要求在 corpassist-bff 内直接调用通义 API 并拼接检索结果，以「加快上线」。是否接受？

    参考答案

    不接受为长期方案。短期 PoC 可容忍，但违背接入/编排分层：密钥与 Token 治理失控，RAG 无法复用。应拆出 orchestrator + rag-service，BFF 只透传。

  2. Agent 办公自动化需连续调用 8 次工具，每次 3～5s，用户仍在浏览器等待。应如何调整架构？

    参考答案

    改为异步 job：提交返回 job_id，agent-runtime 在后台执行并推送进度；或限制 Agent max_steps 并合并工具调用。避免 BFF HTTP 线程阻塞 40s+。
