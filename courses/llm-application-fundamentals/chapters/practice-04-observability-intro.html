为什么要学本章

LLM 应用上线后，「能跑」不够——产品会问回答准不准，SRE 会问为什么 P95 突然 8 秒，财务会问Token 为何翻倍。可观测性把这三类问题放进同一套指标、日志与 Trace 语言里，避免应用团队与运维各说各话。

学完你能
  • 为 CorpAssist 列出三条核心 SLI：延迟 P95、Token 用量、评测通过率
  • 用 Trace / Span 描述一次 RAG 问答的跨服务耗时分布
  • 说明「离线评测 → 上线 → 监控回归」的 eval loop 预览，与 Prompt/检索改动的关系
  • 与 SRE 约定告警分级、仪表盘归属与 on-call 交接信息
本章先记住 4 件事
  • SLI 要可行动：不只「成功率」，还要区分首字延迟、端到端 P95、检索空结果率。
  • Trace 串起黑盒：一次用户提问可能穿越 BFF、编排、RAG、网关；没有 Span 只能猜。
  • 效果要评测闭环：线上指标反映体验，离线 golden set 反映「该不该发版」。
  • 与 SRE 分工：应用定义 SLO 与业务标签；平台提供采集、存储与告警路由。

核心 SLI

CorpAssist · 制度问答 企业知识库 RAGS1行业说法Enterprise RAG · Knowledge Base QA
检索增强生成回答制度/文档，带引用溯源；面试最高频系统设计题 知识库问答在促销期流量翻倍。领导要求：用户感知延迟不能劣化 30%；月度 Token 预算不能超支；抽检答案合格率不能低于 85%。本章用三条 SLI 把诉求翻译成可监控、可告警的数字。

SLI（Service Level Indicator）是「对用户或业务真正重要」的可测量指标。LLM 应用除传统 HTTP 可用性外，必须显式纳入生成质量与Token 成本，否则会出现「监控全绿、产品全红」。

从 SLI 到 SLO（与 SRE 对齐用）

SLO 是 SLI 的目标区间，例如「制度问答端到端 P95 < 4s（28 天滚动）」「评测通过率 ≥ 85%」。错误预算（error budget）耗尽时，应冻结 Prompt/索引大改，优先稳定性——详见下一节与 SRE 的协作接口。

反模式
  • 只监控 HTTP 200，不记录 usage 与 TTFT
  • 把「用户点赞」当唯一质量指标，无离线 golden set
  • P95 用全站混合流量，掩盖 RAG 检索变慢
推荐
  • 网关统一落库 usage，编排层打 trace_id
  • 发版门禁：评测通过率不低于基线 −2%
  • 按场景分 SLI 仪表盘（企业知识库 RAGS1行业说法Enterprise RAG · Knowledge Base QA
    检索增强生成回答制度/文档，带引用溯源；面试最高频系统设计题     知识库 vs 智能客服S2行业说法Conversational AI · Customer Service Bot · DST
    多轮对话、意图识别、工具查单与人工转接；会话状态需规则+模型双校验     客服）

Trace 概念

Trace 表示一次用户请求的全链路；其中的Span 是子步骤（如「向量检索」「调用 chat/completions」）。OpenTelemetry 等标准让 Java 编排服务与 Python RAG worker 共用 trace_id，在 Jaeger / Tempo 中拼成火焰图。

与网关 usage 对齐

API 契约与集成方式 中响应体的 usage 应在 llm-gateway 落库，并由编排 Span 引用同一 request_id。否则 Token SLI 与财务对账会出现「网关一套、业务日志一套」。

先想 10 秒：P95 从 3s 升到 7s，但 LLM Span 耗时不变，最该先看哪个 Span？

优先看 RAG 检索 Span（S2）或编排层串行步骤是否新增（如额外 rerank、未并行化的工具调用）。若检索正常，再查 BFF 流式缓冲、数据库或下游业务 API。

评测闭环预览

可观测性中的「效果」不能只靠线上点赞。Eval loop（评测闭环） 把 golden 问题集、自动/人工评判与发版门禁连起来，形成「改 Prompt / 改索引 → 跑评测 → 对比 SLI → 灰度上线 → 监控回归」的循环。

CorpAssist · Eval loop 预览
flowchart LR
  A[变更: Prompt / 分块 / Embedding] --> B[离线评测集]
  B --> C{通过率 ≥ SLO?}
  C -->|否| A
  C -->|是| D[灰度 5% 流量]
  D --> E[观察 P95 / Token / 抽检]
  E --> F{回归?}
  F -->|是| G[全量发布]
  F -->|否| A
  G --> H[持续采样 + 周度复评]

评测通过率是发版门禁；上线后仍用 P95 与 Token SLI 监控体验与成本。完整评测流水线在后续 RAG/Agent 课程展开，本章建立共同语言即可。

评测通过率 vs 线上 SLI

维度离线评测（golden set）线上 SLI / 采样
目的 发版前拦截明显退化 发现真实流量分布下的长尾问题
CorpAssist 示例 200 条制度问答，检查引用条文 ID 是否匹配 每周抽检 50 条 + 用户点踩率
与 Token 关系 固定集可预算单次评测成本 异常飙升可能意味检索片段过长或重试风暴

应用团队应维护版本标签（prompt_version、index_version），使线上日志与离线评测结果可对比。当评测通过率下降 2 个百分点时，即使 P95 未变，也应视为 incident 调查项。

与 SRE 协作接口

有 Java 微服务经验的读者熟悉与 SRE 的「错误预算 + 告警分级」。LLM 应用需在接口中多两类信息：效果退化（评测/抽检）与成本异常（Token）。

分工 RACI（简化）

事项应用团队SRE / 平台
定义 SLI / SLO 与业务标签 R/A C
Prometheus / OTel 采集、存储、告警路由 C R/A
golden set 与评测流水线 R/A I（提供 CI 算力可选）
on-call runbook：429 退避 vs RAG 索引损坏 R 业务章节 A 告警模板

告警分级示例（与 SRE 共识）

  • P1：评测通过率跌破 SLO 且与 24h 内发版相关；或核心场景可用性 < 99%。
  • P2：P95 超 SLO 15 分钟；单租户 Token 小时环比 > 200%（可能泄漏或滥用）。
  • P3：检索空结果率升高、非核心场景延迟波动。

判断练习

  1. 评测通过率 90% → 84%，但 HTTP 成功率 99.9%、P95 正常。是否应触发 P1？

    参考答案

    应视为 P1 或 P1 候选（若 SLO 为 85% 且刚发版）。效果 SLO 与可用性解耦；需比对 prompt_version / index_version，暂停灰度并回滚检索或 Prompt 变更，而非只看网关健康。

  2. 为降 P95，团队决定把所有历史对话全量塞进 prompt，不用 RAG。可能同时恶化哪条 SLI？

    参考答案

    Token 用量与延迟往往同时恶化（上下文变长，首字与生成更慢）；长上下文还可能增加幻觉，间接拉低评测通过率。应优化检索与截断，而非堆历史。
