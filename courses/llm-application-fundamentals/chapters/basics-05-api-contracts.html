为什么要学本章

CorpAssist 不会直接让业务微服务各自持有厂商 API Key。你需要理解统一的 HTTP 契约：请求体长什么样、流式如何推给前端、失败时如何重试。这与 Java 团队熟悉的 REST/OpenAPI 同源，只是资源从「订单」变成了「对话补全」。

  • 对比流式（SSE）与非流式在 CorpAssist 客服场景下的体验与实现差异
  • 根据 HTTP 状态与错误体设计重试、退避与熔断，避免重复扣 Token 费
  • 画出 BFF → LLM 网关 → 模型厂商的集成拓扑，并对接下一阶段应用分层
本章先记住 4 件事
  • 契约统一：POST /v1/chat/completions + JSON 消息数组是事实标准；改 base_url 即可换厂商。
  • 流式优先体验：客服/写作类前端用 stream: true + SSE；批处理、评测流水线可用非流式。
  • 错误要分层：4xx 多为参数/鉴权勿盲重试；429/5xx 配合指数退避与幂等请求 ID。
  • BFF 不堆智能：编排、RAG、工具调用放在后端服务；BFF 负责鉴权、聚合与流式透传。

OpenAI 兼容协议

CorpAssist · 多模型路由 采购了通义、DeepSeek 与内网私有化 Qwen，运维希望一套 Spring AI / Python 客户端，通过配置 spring.ai.openai.base-url 指向公司 LLM 网关，而不是每个业务服务写三套 SDK。

OpenAI 兼容（OpenAI-compatible）指请求/响应 JSON 形状、路径与 Header 约定与 OpenAI Chat Completions API 对齐。国内主流商用 API 与多数私有化推理框架（vLLM、TGI 等）均提供兼容端点，应用层只需替换 base_url、api-key 与 model 名称。

核心端点与消息结构

字段/路径含义CorpAssist 注意点
POST /v1/chat/completions 多轮对话补全 BFF 或 llm-gateway 对外暴露；业务服务不直连公网
messages[] role：system / user / assistant 制度问答的 system 提示放在网关或编排层统一注入
model 逻辑模型名 网关内映射到真实厂商型号（如 corpassist-fast → qwen-turbo）
max_tokens / temperature 生成长度与随机性 与 Token、上下文窗口与费用 中的 上下文窗口、计费直接相关

非流式调用（curl）

bash 
curl -s https://llm-gateway.corpassist.internal/v1/chat/completions \
  -H "Authorization: Bearer $CORPASSIST_LLM_KEY" \
  -H "Content-Type: application/json" \
  -H "X-Request-Id: $(uuidgen)" \
  -d '{
    "model": "corpassist-fast",
    "messages": [
      {"role": "system", "content": "你是 CorpAssist 企业助手，回答须简洁并标注信息来源类型。"},
      {"role": "user", "content": "年假最多可休几天？"}
    ],
    "temperature": 0.2,
    "max_tokens": 512
  }'

响应体中 choices[0].message.content 为完整回复；usage.prompt_tokens / completion_tokens 用于成本归因（按部门、应用 ID 打标签可在网关侧记录）。

反模式
  • 每个微服务内置不同厂商 SDK，密钥散落配置中心
  • 在业务库表硬编码 prompt 模板且无版本管理
  • 忽略 usage，无法做 Token 预算告警
推荐
  • 统一 OpenAI 兼容 网关 + 路由策略
  • Spring AI / httpx 仅配置 base_url 与超时
  • 请求 ID、租户 ID 贯穿日志与计费

流式与非流式

非流式一次返回完整 JSON；流式（streaming）在请求体设 "stream": true，服务端以 SSE（Server-Sent Events） 推送多个 data: 行，每块含增量 delta.content，最终以 data: [DONE] 结束。

选型对照

维度非流式流式 SSE
用户体验 等待数秒后出现整段文字 逐字/逐句出现，降低「卡住」感
CorpAssist 场景 夜间批处理摘要、评测脚本、Webhook 回调 Web 客服、制度问答、写作助手
实现 普通 HTTP 客户端即可 需保持长连接；BFF/网关勿缓冲整段 body
计费 响应末 usage 一次给出 常在最后一个 chunk 带 usage（依厂商略有差异）

流式请求与 SSE 片段

bash 
curl -N https://llm-gateway.corpassist.internal/v1/chat/completions \
  -H "Authorization: Bearer $CORPASSIST_LLM_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"corpassist-fast","stream":true,"messages":[{"role":"user","content":"用三句话介绍 RAG"}]}'

网关透传要点

Spring Cloud Gateway 或 Nginx 对流式路由须关闭响应缓冲（如 proxy_buffering off），并设置足够长的读超时。BFF 用 text/event-stream 原样写给浏览器，避免在内存中拼接完整回复再下发——否则失去 流式 意义。

python 
import httpx

async def ask_corpassist(user_text: str) -> str:
    async with httpx.AsyncClient(
        base_url="https://llm-gateway.corpassist.internal/v1",
        headers={"Authorization": f"Bearer {API_KEY}"},
        timeout=httpx.Timeout(60.0, connect=5.0),
    ) as client:
        r = await client.post(
            "/chat/completions",
            json={
                "model": "corpassist-fast",
                "messages": [{"role": "user", "content": user_text}],
                "stream": False,
            },
        )
        r.raise_for_status()
        return r.json()["choices"][0]["message"]["content"]

错误码与重试

LLM 调用跨公网或专有云，失败是常态。工程上须区分不可重试（修正参数）与可重试（瞬时过载），并控制 Token 重复消耗风险。

常见 HTTP 与处理策略

状态典型原因CorpAssist 策略
401 / 403 Key 无效、租户无权限 告警运维，不重试；检查网关到厂商的密钥轮换
400 消息超出 上下文窗口、非法 model 截断历史或降级模型；记录 X-Request-Id 给支持
429 限流、配额耗尽 指数退避 + 抖动；队列化非实时任务；切换备用模型路由
502 / 503 / 504 网关/厂商瞬时故障 有限次重试（如 2～3 次）；熔断后走降级文案

错误体通常含 error.message、error.type（如 rate_limit_exceeded）。幂等：对「已提交但响应丢失」的非流式请求，可用固定 X-Request-Id 在网关去重（若厂商支持）；流式中断则只能重新发起新对话轮次，并在产品层提示用户「生成已中断」。

  • 429 / 503，且未超过单用户 QPS 预算
  • 连接超时、读超时（确认请求体未损坏）
  • 网关到厂商链路抖动，备用路由可用
  • 400 参数错误、401 鉴权失败
  • 流式已输出部分 token 后中断（避免双倍计费与语义重复）
  • 业务侧已判定敏感内容拦截（policy violation）
先想 10 秒：非流式调用超时，客户端不知道厂商是否已生成完毕，该不该立刻原样重试？

不应无界重试。优先用同一 X-Request-Id 查询网关是否已有结果；若无查询能力，最多 1 次幂等重试并限制 max_tokens，同时在前端提示可能重复。更好的做法是在网关记录 in-flight 状态。

与微服务 BFF 关系

有 Java 微服务经验的读者可把 CorpAssist 看成：前端只认识 BFF，BFF 聚合身份、会话与业务 API；LLM 能力集中在网关与编排服务，而不是在每个 hr-service、finance-service 里各写一套 ChatClient。

BFF 负责鉴权、会话 ID、限流与流式透传；编排服务组合 RAG/工具；网关统一密钥、路由与 usage 落库。业务微服务仍用传统 REST，仅在需要时由编排层回调。

职责划分

  • BFF：用户身份、多租户、将前端 /api/chat/stream 转为对网关的 chat/completions；不写 Prompt 工程核心逻辑。
  • llm-gateway：OpenAI 兼容 入口、模型路由、密钥托管、全链路 usage 与审计。
  • dialog-orchestrator / rag-service：RAG 检索、引用拼接、工具调用；产出最终 messages 再调网关。
  • 业务微服务：提供权威业务数据（制度条文、审批状态），不直接持有大模型 API Key。

本章结论

每条对应上一节 h3，basics 阶段收官前复述 API 集成要点。

  • OpenAI 兼容协议：/v1/chat/completions + messages 是跨厂商统一契约；CorpAssist 通过网关换模型而非换 SDK。
  • 流式与非流式：面向用户用 SSE 流式；批处理与评测用非流式；网关与 BFF 须禁止缓冲流式响应。
  • 错误码与重试：4xx 修正参数，429/5xx 有限退避；用请求 ID 与 usage 控制重复 Token 成本。
  • BFF 与微服务：BFF 聚合与透传，智能编排下沉；业务服务不直连厂商，详见 LLM 应用分层架构。

动手练习

用 curl 与对照表摸清 CorpAssist 网关契约，并在 BFF–网关拓扑图上标注你负责的服务边界。

操作步骤

  1. 将「非流式 curl」中的 model 改为 stream: true，用 curl -N 观察至少 3 行 data: 与 [DONE]（可用公司网关或兼容沙箱）。
  2. 对照「常见 HTTP 与处理策略」表：为 429 与 400 各写一句 CorpAssist 运维/runbook 动作（是否重试、谁接手）。
  3. 在「BFF → LLM 网关 → 模型厂商」Mermaid 上，用不同颜色标出你当前 Java 项目属于 BFF、编排还是业务微服务（纸笔或导出图均可）。
  4. 阅读 Spring AI 与 httpx 代码块：列出两者共同依赖的环境变量或配置项（base_url、api-key、超时）。

判断练习

  1. finance-service 为省事先在进程内配置 DeepSeek API Key 并直接调用 api.deepseek.com，与「BFF → 网关」架构相比，最大风险是什么？

    参考答案

    密钥与审计失控：Key 散落多服务难以轮换；无法统一 usage 计费、限流与内容安全策略；合规上财务数据出网路径不可管。应改为经 llm-gateway 的 OpenAI 兼容 调用。

  2. 客服流式回答已输出一半，用户刷新页面。BFF 应对同一 chat/completions 请求自动重试以「续写」吗？

    参考答案

    不应原请求重试。流式中断后厂商可能已计费；重试会产生重复内容。应开启新请求（新 assistant 轮次或新会话），前端提示「已中断，请重新提问」；若需续写，由产品层显式携带已生成片段走编排逻辑，而非 HTTP 层盲重试。
