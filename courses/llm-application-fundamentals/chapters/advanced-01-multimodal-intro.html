OCR 在 RAG 中的位置

OCR（光学字符识别）在 CorpAssist 中不是「给用户看的功能」，而是RAG 入库链路的预处理环节：把非结构化像素变成可检索、可计费的文本。位置错误（例如只在问答时 OCR）会导致重复计算、无法增量更新索引、且难以做离线评测。

能力边界

CorpAssist · 附件问答预期管理 员工上传 60 页扫描合同问「违约金条款原文」。产品期望 10 秒内精确引用；若走 vision 全页读取，Token 与延迟爆炸，且模型可能漏读脚注。需要明确：什么必须 OCR 入库，什么可 vision 单页，什么应拒答并引导人工。

视觉语言模型（VLM）与 OCR+RAG 各有边界。面试与方案评审中，应能说明「为什么不把所有 PDF 都交给 GPT-4V / Qwen-VL 一次读完」。

多模态 API 请求示例（在线单页 vision 分支）

当用户临时上传单张 PNG 且问题为「提取图中金额」时，编排层可调用网关上的 vision 模型。注意：image_url 应使用短时 signed URL，勿把 base64 巨图长期写日志。

json 
{
  "model": "corpassist-vision-fast",
  "messages": [
    {
      "role": "system",
      "content": "你是 CorpAssist 附件助手。仅根据图片中的可见文字回答；看不清则明确拒答。不要编造金额或条款。"
    },
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "请读出报销单上的合计金额（数字即可）。"
        },
        {
          "type": "image_url",
          "image_url": {
            "url": "https://object-store.internal/corpassist/attachments/tmp-a8f3.png?X-Amz-Expires=300",
            "detail": "high"
          }
        }
      ]
    }
  ],
  "max_tokens": 256,
  "temperature": 0
}

响应中的 usage.prompt_tokens 对 vision 往往包含「图像 patch」折算，显著高于同等字数纯文本——预算估算见 Token、上下文窗口与费用。网关应打标签 modality=vision 便于 SLI 拆分。

拒答与降级
  • 页数 > 上限 → 提示「请先入库或选择单页」
  • OCR 置信度 < 阈值 → 不进入 RAG，进入人工审核
  • 含 PII 敏感类 → 走 数据、隐私与合规入门 中的脱敏流水线
先想 10 秒：一张 20 页扫描 PDF 应走 OCR 入库还是单次 vision？

若会反复问答同一文档 → OCR + 分块入向量库（摊薄多次提问的 Token）；若一次性字段抽取且页数 ≤2 → 可 vision 直读。超过页数上限必须拒答或引导用户选页，避免撑爆 上下文窗口。
