为什么要学本章

从大模型到应用工程 建立了 CorpAssist 全景；在写第一行 API 之前，必须理解按 Token 计费与上下文窗口 上限——它们直接决定多轮客服能否「记住」历史、长文档问答是否买得起。

学完你能
  • 用输入/输出单价粗算 CorpAssist 一次制度问答或客服回合的费用
  • 为多轮对话选择截断策略（保留 system、滑动窗口、摘要压缩）
  • 说明长 PDF 全量塞进提示词 vs RAG 检索片段的代价差异
本章先记住 4 件事
  • 计费单位：商用 API 通常按 Token（词元）计输入与输出，中英文切分方式不同，不能按「字数」心算。
  • 硬上限：上下文窗口 限制单次请求可携带的 Token 总量（系统提示 + 历史 + 用户本轮 + 预留输出）。
  • 截断是工程默认：超过窗口或预算时，必须主动丢弃/摘要旧消息，而不是假设模型「自动忘记」。
  • 长文档别硬塞：入库用 Embedding + 检索（RAG）通常比整份 PDF 进提示词更省、更稳。

Token 计费模型

CorpAssist · 财务与容量评审 产品要在 Q2 为「制度问答」开通 500 名活跃用户。运维问：若每人每天 20 轮、每轮平均 3k 输入 Token + 500 输出 Token，月费大概多少？本章先建立单价 × Token 数的心算框架。

对应用开发者而言，Token 不是语言学概念，而是计费与上下文配额的最小单位。同一句话在中英文、不同模型下的 Token 数可能差 30% 以上，预算应留余量。

  • 输入 Token（prompt）：发给模型的全部内容——系统提示、历史消息、检索到的制度片段、用户问题。
  • 输出 Token（completion）：模型生成部分；流式输出 仍按最终生成量计费。
  • 常见计价：输入/输出分开标价（如「￥x / 百万输入 Token」「￥y / 百万输出 Token」）；部分厂商对缓存命中输入有折扣（实现细节见厂商文档）。

代码：单次请求成本粗算（伪代码）

python 
# CorpAssist 制度问答 — 单次调用费用估算（数值仅为示意）
PRICE_IN_PER_M = 2.0    # 元 / 百万输入 token
PRICE_OUT_PER_M = 8.0   # 元 / 百万输出 token

prompt_tokens = 3200    # system + 检索片段 + 用户问题
completion_tokens = 450

cost_yuan = (
    prompt_tokens * PRICE_IN_PER_M / 1_000_000
    + completion_tokens * PRICE_OUT_PER_M / 1_000_000
)
# cost_yuan ≈ 0.01 量级；日活 500 × 20 轮需再乘调用量
常见误区
  • 用「汉字个数」代替 Token — 中文往往 1～2 字一个 Token，且含标点与 JSON 结构。
  • 只估输出、忽略输入 — 长文档场景输入常占 80% 以上费用。
  • 忘记工具/函数调用的额外 Token — Agent 场景下 schema 与中间结果也计入上下文。
先想 10 秒：同一回答，把 temperature 从 0.2 调到 0.9，Token 费用一定变多吗？

不一定。费用主要由实际上下文长度与生成 Token 数决定；temperature 影响采样随机性，可能改变措辞长度，但不是计费主因。应通过 max_tokens 与提示词约束控制输出上限。

上下文截断策略

上下文窗口 是单次请求可处理的 Token 上限（如 32k、128k、200k，随模型而变）。CorpAssist 多轮智能客服 若把全部历史原样拼接，几轮后就会触顶，必须在校验层实现截断策略。

不推荐 · 无界堆历史
  • 每轮把 30 轮完整对话发给模型
  • 触顶后 API 报错或厂商静默截断头部
  • 用户刚提供的订单号可能被挤掉
推荐 · 预算内组装
  • 固定保留 system + 本轮 RAG top-k
  • 历史按 Token 从新到旧填充至预算 70%
  • 关键槽位（工号、工单号）写入会话表再注入

长文档场景代价

CorpAssist · 200 页员工手册 PDF 业务方希望「上传即可问」。若将全文一次性放入提示词：单次输入可能 15 万 Token 以上，费用与延迟都不可接受，且仍受上下文窗口硬限制。正确路径是切块、向量化、检索 top-k 片段再生成。

长文档场景的核心矛盾：模型一次能「看见」的文本有限，而企业知识库体积无限。应用层应默认走索引管线，而非「超大提示词」。

入库时选择 Embedding 模型需兼顾维度、中文效果与单价；这与「对话模型」计费分开。CorpAssist 文档中心应监控：平均每问检索片段 Token 数、命中率，避免为追求召回把 top-k 开得过大。

窗口已经 128k，是否还需要 RAG？
需要。窗口变大不消除费用与延迟线性增长；多文档并发时仍会触顶。RAG 还改善可更新性（改制度只重索引片段）。
多模态 PDF（扫描件）Token 怎么算？
图文页常按 patch/页计费，往往高于纯文本。CorpAssist 附件问答应优先 OCR 再入向量库（见进阶多模态章）。

月费粗算模板

python 
# CorpAssist Q2 容量粗算 — 客服通道（示意）
users = 500
rounds_per_user_per_day = 20
days = 30
avg_in = 4200
avg_out = 350
price_in_m = 2.0
price_out_m = 8.0

total_in = users * rounds_per_user_per_day * days * avg_in
total_out = users * rounds_per_user_per_day * days * avg_out

monthly_yuan = (
    total_in * price_in_m / 1_000_000
    + total_out * price_out_m / 1_000_000
)
# 代入后得到月费数量级；评审时再加 30% 余量应对尖刺与重试

工程上还应预留：失败重试（同 prompt 可能计费多次）、评测流水线批量调用、以及未来 Agent 多步推理的链式放大。下一章从应用视角理解生成机制，便于解释「为何输出越长越慢、越贵」。

本章结论

本章建立 LLM 应用的「计量单位」。验收时能用 CorpAssist 场景各举一例费用与截断决策。

  • Token 计费模型：费用 ≈ 输入 Token × 输入单价 + 输出 Token × 输出单价；勿用字数代替 Token，输入在长文档场景常占大头。
  • 上下文截断策略：在 上下文窗口 与预算内组装 messages；system 与合规指令优先，历史从新到旧填充，超限丢最旧或摘要。
  • 长文档场景代价：整份 PDF 进 prompt 费用与延迟不可控；Embedding + RAG 片段是默认工程路径。
  • 成本估算练习：用日活 × 轮次 × 平均 Token × 单价粗算月费，并为尖刺上传与重试留余量。

判断练习

  1. CorpAssist 客服为「记住更多历史」，将 context_window 从 32k 换到 128k 模型且不做截断。费用与稳定性会如何？是否推荐？

    参考答案

    单次输入 Token 可随历史线性增长，费用与延迟显著上升；尖刺用户仍可能触顶或拖垮 P99。不推荐仅靠换大窗口；应保留滑动窗口/摘要 + 会话表存关键槽位，大窗口作为缓冲而非无界堆叠。

  2. 用户上传 200 页 PDF 并要求「每次问答都附带全文」。下列哪项最能同时降费用与触顶风险？A 换更大窗口 B RAG 检索 top-k 片段 C 提高 temperature

    参考答案

    选 B。RAG 将单次 prompt 控制在 top-k 片段 Token；A 只缓解触顶不解决费用线性增长；C 与上下文长度无关。
