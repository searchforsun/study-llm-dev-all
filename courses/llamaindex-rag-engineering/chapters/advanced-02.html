增量索引策略

增量索引指只对新增、修改、删除的文档重新索引，不触及其他文档。LlamaIndex 和 Milvus 都原生支持增量操作。

flowchart LR
    D[文档变更事件] -->|新增| INS[index.insert(new_doc)]
    D -->|修改| DEL[index.delete_ref_doc(doc_id)]
    DEL --> INS
    D -->|删除| DEL2[index.delete_ref_doc(doc_id)]
    INS -->|写入| VS[Milvus]
    DEL2 -->|删除向量| VS
    VS -->|检索| Q[查询引擎]
python
from llama_index.core import VectorStoreIndex, Document
from llama_index.core.ingestion import IngestionPipeline
from llama_index.vector_stores.milvus import MilvusVectorStore

vector_store = MilvusVectorStore(uri="http://milvus:19530", dim=1536)
index = VectorStoreIndex.from_vector_store(vector_store)

# 1. 增量添加文档
def add_document(text: str, metadata: dict) -> str:
    """增量添加单篇文档"""
    doc = Document(text=text, metadata=metadata)
    # 方法 A：直接插到已有索引
    index.insert(doc)
    # 方法 B：通过 IngestionPipeline（推荐，支持转换）
    # pipeline = IngestionPipeline(transformations=[...], vector_store=vector_store)
    # nodes = pipeline.run(documents=[doc])
    return doc.doc_id

# 2. 增量修改文档（先删除后插入）
def update_document(doc_id: str, new_text: str, new_metadata: dict):
    """增量修改：删除旧版本，插入新版本"""
    index.delete_ref_doc(doc_id, delete_from_docstore=True)
    new_doc = Document(text=new_text, metadata=new_metadata)
    index.insert(new_doc)
    print(f"文档 {doc_id} 已更新")

# 3. 批量增量更新
def batch_upsert(documents: list[Document]):
    """批量新增或更新"""
    for doc in documents:
        if doc.doc_id and doc.metadata.get("_is_update"):
            try:
                index.delete_ref_doc(doc.doc_id)
            except Exception:
                pass  # 新文档无旧记录
    index.insert_nodes([doc for doc in documents])  # 批量插入

CorpAssist 场景

HR 部门每周更新一次员工手册。增量更新流程：1) RocketMQ 收到文档变更消息；2) Worker 消费消息，下载新版本 PDF；3) 解析文档后与旧版本 diff，确定增删改的章节；4) 用 insert/delete_ref_doc 增量更新索引；5) 验证查询一致性后提交。全程 < 5 秒，检索服务零影响。

索引版本管理

索引版本管理解决"更新后需要回滚"的问题。LlamaIndex 不直接管理版本，但可以通过 Milvus 的集合别名机制实现。核心思路：每个版本是一个独立的 Milvus Collection，通过别名（alias）指向当前活跃版本。

python
from pymilvus import Collection, utility, connections

# Milvus 版本管理工具
class IndexVersionManager:
    ALIAS = "corpassist_active"

    def __init__(self, host="milvus", port=19530):
        connections.connect(host=host, port=port)

    def create_version(self, version_id: str, dim: int = 1536):
        """创建新版本的 Collection"""
        from pymilvus import CollectionSchema, FieldSchema, DataType

        schema = CollectionSchema([
            FieldSchema("id", DataType.INT64, is_primary=True, auto_id=True),
            FieldSchema("vector", DataType.FLOAT_VECTOR, dim=dim),
            FieldSchema("text", DataType.VARCHAR, max_length=65535),
            FieldSchema("metadata", DataType.JSON),
        ])
        collection = Collection(f"corpassist_v{version_id}", schema)
        collection.create_index("vector", {"index_type": "IVF_FLAT", "params": {"nlist": 1024}, "metric_type": "IP"})
        return collection

    def switch_active(self, version_id: str):
        """切换活跃版本（蓝绿切换）"""
        new_collection = f"corpassist_v{version_id}"
        # 移除当前别名
        if utility.has_collection(self.ALIAS):
            utility.drop_alias(self.ALIAS)
        # 别名指向新版本
        utility.create_alias(new_collection, self.ALIAS)
        print(f"活跃索引切换到: {new_collection}")

    def list_versions(self):
        """列出所有版本"""
        collections = utility.list_collections()
        return [c for c in collections if c.startswith("corpassist_v")]

    def rollback(self, version_id: str):
        """回滚到指定版本"""
        self.switch_active(version_id)
        print(f"已回滚到 v{version_id}")

# 使用示例
vman = IndexVersionManager()
vman.create_version("20260530")
vman.switch_active("20260530")
# 写入数据...
vman.create_version("20260601")
vman.switch_active("20260601")
# 发现有问题！
vman.rollback("20260530")

无版本管理

直接覆盖旧索引。新索引发问题后无法回滚，只能从备份恢复（耗时 2 小时+）。期间服务不可用。

Collection 别名切换

保留最近 7 个版本的索引。新版本出问题时，秒级切换别名回旧版本。查询服务无感，零停机。

蓝绿部署

蓝绿部署保持两套索引（蓝/绿）同时存在，任何时候只有一套提供服务。蓝色为新版，绿色为旧版，通过流量控制切换。

面试可能问

全量重建和增量更新分别应对什么场景？为什么两者需要共存？

全量重建：1) 嵌入模型升级（如 text-embedding-3-xxx → gte-Qwen2），新模型生成的向量与旧的不兼容，必须全部重新嵌入；2) 索引结构变更（如 chunk_size 从 512 改为 1024）；3) 数据迁移（开发→测试→生产）。增量更新：1) 日常文档新增和修改；2) 少量文档内容修正。两者共存原因：全量重建保证架构一致性，增量更新保证日常效率。CorpAssist 实践：每天全量重建一次，日常变更增量更新。

Milvus 集合别名切换时，正在进行的查询会受影响吗？

不会受影响。Milvus 通过引用计数管理 Collection 的生命周期：切换别名只是元数据操作（修改 Alias → Collection 映射），正在执行的查询持有底层 Collection 的引用，会继续完成。新查询通过别名访问新的 Collection。所以别名切换是"零停机"的。但如果切换时旧 Collection 有大量写入请求（如增量索引），需要等写入完成后才能释放旧 Collection 的引用。

如何设计 RAG 服务的自愈机制？当监控告警触发时自动做什么？

多层自愈：1) 检索延迟 P99 > 3s → 自动切换为快速检索模式（降级：关闭 Reranker、减小 top_k）；2) Milvus 连接失败 → 自动切换到 Qdrant 副本（双写双读）；3) 嵌入 API 错误 > 5% → 自动切换到本地嵌入模型（BGE-M3）；4) 新版本索引出问题 → 自动回滚到上一版本（保留最近 3 个版本异或，保留 7 个版本）。每层自愈都在 30 秒内完成，超过则告警人工介入。

判断练习

场景：全量重建时仍有查询流量，怎么保证两个操作不互相影响？

使用蓝绿部署模式。全量重建写入新的 Collection（蓝色），当前查询路由到旧的 Collection（绿色）。重建完成后通过切换别名将查询路由切换到新 Collection。关键点：1) 蓝色 Collection 要使用不同的名称（如 corpassist_v20260601）；2) 别名 corpassist_active 始终指向当前活跃的 Collection；3) 重建期间不要切换别名。这个过程对查询服务完全透明。

场景：增量删除后用户仍然能检索到已删除的文档，可能的原因？

可能原因：1) delete_ref_doc 的 delete_from_docstore=False（默认），只删除了索引结构中的引用，但向量库中的向量未被删除；2) Milvus 的删除操作默认是软删除，需要 cleanup 后才能真正生效；3) 请求的索引实例是旧的（连接到了旧 Collection）。排查：确认 delete_from_docstore=True；调用 collection.compact() 清理 Milvus 的已删除数据；检查当前查询连接的 Collection 是否已切换。
