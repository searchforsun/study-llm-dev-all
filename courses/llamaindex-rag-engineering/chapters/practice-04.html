为什么学这章

CorpAssist 服务多个部门（HR、IT、财务），每个部门文档独立且需要隔离。IT 员工搜索 IT 指南不应看到 HR 手册，财务人员只能查看财务文档。Naive RAG 将所有文档混在一起检索，无法满足多租户需求。本章解决多知识库的隔离、路由、鉴权和配置管理问题。

学完后能做什么

用 metadata filter 和 Milvus 分区实现多租户数据隔离；用 RouterQueryEngine 实现意图驱动的知识库路由；用自定义 NodePostprocessor 实现文档级权限过滤；用 YAML 配置管理多索引生命周期。

本章要点速记

1) 多租户隔离方案：Milvus 集合前缀（推荐）或 metadata filter；2) 意图路由用 RouterQueryEngine + LLMSingleSelector；3) 权限过滤在检索后、合成前通过 NodePostprocessor 实现；4) YAML 配置文件管理所有索引的地址、模型、过滤规则；5) 混合方案：全量索引+metadata过滤 vs 独立索引+路由，取决于数据量。

意图路由实现

意图路由根据查询内容判断属于哪个知识域，然后路由到对应的索引引擎。CorpAssist 使用 RouterQueryEngine + LLMSingleSelector 实现。

flowchart TB
    Q[用户查询] --> IR[意图识别]
    IR -->|"请假|年假|考勤"| HR[HR 索引]
    IR -->|"VPN|邮箱|权限"| IT[IT 索引]
    IR -->|"报销|发票|预算"| FIN[财务索引]
    IR -->|"产品功能|API"| PD[产品文档索引]
    subgraph 权限层
        HR -->|tenant_id=hr| F1[Metadata Filter]
        IT -->|tenant_id=it| F2
        FIN -->|tenant_id=finance| F3
    end
    F1 & F2 & F3 --> RS[结果合成]
    RS --> A[最终答案]
python
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.tools import QueryEngineTool, ToolMetadata
from llama_index.core.selectors import LLMSingleSelector

# 1. 为每个部门创建独立的查询引擎
def create_department_engine(department: str):
    """根据部门创建对应的检索引擎"""
    vector_store = MilvusVectorStore(
        uri="http://milvus:19530",
        collection_name=f"corpassist_{department}",
        dim=1536
    )
    index = VectorStoreIndex.from_vector_store(vector_store)
    return index.as_query_engine(
        similarity_top_k=5,
        filters=MetadataFilters(
            filters=[
                ExactMatchFilter(key="department", value=department)
            ]
        )
    )

# 2. 工具描述（选择器的决策依据）
tools = [
    QueryEngineTool(
        query_engine=create_department_engine("hr"),
        metadata=ToolMetadata(
            name="hr_knowledge_base",
            description="HR 人力资源知识库：请假制度、年假政策、考勤规则、薪酬福利、入职流程、离职手续等"
        )
    ),
    QueryEngineTool(
        query_engine=create_department_engine("it"),
        metadata=ToolMetadata(
            name="it_knowledge_base",
            description="IT 技术支持知识库：VPN 配置、邮箱设置、系统权限申请、软件安装、网络故障排查等"
        )
    ),
    QueryEngineTool(
        query_engine=create_department_engine("finance"),
        metadata=ToolMetadata(
            name="finance_knowledge_base",
            description="财务制度知识库：报销流程、发票要求、预算申请、费用标准、差旅政策等"
        )
    ),
]

# 3. 路由引擎
router = RouterQueryEngine(
    selector=LLMSingleSelector.from_defaults(),
    query_engine_tools=tools,
    verbose=True
)

# 4. 测试路由
for query in ["年假几天?", "VPN 怎么连?", "报销上限多少?"]:
    resp = router.query(query)
    print(f"Q: {query} -> A: {resp}")

CorpAssist 场景

员工在门户输入"怎么申请年假"，RouterQueryEngine 识别意图匹配到 hr_knowledge_base（score=0.92），查询 HR 部门的索引。返回结果时自动带上 tenant_id=hr 的 metadata filter，保证只检索 HR 文档。

权限过滤

权限过滤在检索完成后、答案合成前执行。用 NodePostprocessor 过滤掉用户无权查看的文档。LlamaIndex 支持自定义后处理器。

python
from llama_index.core.postprocessor import BaseNodePostprocessor
from llama_index.core.schema import NodeWithScore
from typing import Optional, List

class PermissionFilter(BaseNodePostprocessor):
    """根据用户角色过滤检索到的 Node"""
    def __init__(self, user_roles: List[str]):
        self.user_roles = user_roles

    def _postprocess_nodes(
        self, nodes: List[NodeWithScore],
        query_bundle: Optional = None
    ) -> List[NodeWithScore]:
        filtered = []
        for node in nodes:
            # Node metadata 中存储了 required_role
            required_role = node.metadata.get("required_role", "all")
            if required_role == "all" or required_role in self.user_roles:
                filtered.append(node)
        return filtered

# 给 Node 注入权限信息
def add_permission_metadata(nodes, role_map: dict):
    """role_map: {文档ID: [允许的角色列表]}"""
    for node in nodes:
        doc_id = node.metadata.get("doc_id", "")
        if doc_id in role_map:
            allowed_roles = role_map[doc_id]
            node.metadata["required_role"] = allowed_roles
        else:
            node.metadata["required_role"] = "all"
    return nodes

# 使用权限过滤
query_engine = index.as_query_engine(
    similarity_top_k=5,
    node_postprocessors=[
        SimilarityPostprocessor(similarity_cutoff=0.5),
        PermissionFilter(user_roles=["hr_manager", "admin"])
    ]
)

resp = query_engine.query("薪资调整政策是什么？")
# 如果用户没有 hr_manager 角色，包含薪资信息的 Node 会被过滤掉

检索前过滤（不安全）

只在检索时加 metadata filter。如果 filter 配置错误（漏写），用户可能看到无权访问的内容。攻击者可以通过 API 直接传参绕过。

检索后过滤（推荐）

检索时用 metadata filter 缩小范围，检索后用 NodePostprocessor 二次确认权限。双层保险。metadata filter 拦截 99% 无关文档，NodePostprocessor 拦截剩下的 1% 边界 case。

多租户索引隔离

多租户隔离有两个方案：独立索引（每个租户单独的 Milvus Collection）和共享索引（一个 Collection 内用 metadata 字段过滤）。选择取决于数据量和租户数量。

维度独立索引（每个租户一个 Collection）共享索引（一个 Collection + metadata 过滤）
数据隔离物理隔离，最强安全逻辑隔离，依赖 filter 正确性
运维成本高（每个租户独立建索引、独立备份）低（一个索引管理所有）
租户上限理论上无限（每个新租户建新 Collection）受限于 Collection 容量（Milvus 推荐 <500 万向量/Collection）
跨租户搜索需要搜索多个 Collection 合并结果天然支持（移除 filter 即可）
CorpAssist 推荐大型租户（>10 万文档）中小租户（<10 万文档）
python
# 方案 A：共享索引 + metadata 过滤（推荐 <10 租户）
class SharedTenantIndex:
    def __init__(self, milvus_uri: str):
        self.vector_store = MilvusVectorStore(
            uri=milvus_uri,
            collection_name="corpassist_shared",
            dim=1536,
            enable_dynamic_field=True  # 启用动态字段支持 tenant_id
        )
        self.index = VectorStoreIndex.from_vector_store(self.vector_store)

    def get_engine(self, tenant_id: str):
        return self.index.as_query_engine(
            similarity_top_k=5,
            filters=MetadataFilters(
                filters=[ExactMatchFilter(key="tenant_id", value=tenant_id)]
            )
        )

# 方案 B：独立索引（推荐大租户）
class IsolatedTenantIndex:
    def __init__(self, milvus_uri: str):
        self.milvus_uri = milvus_uri
        self.collections = {}  # 租户→Collection 映射

    def get_or_create_index(self, tenant_id: str):
        if tenant_id not in self.collections:
            vector_store = MilvusVectorStore(
                uri=self.milvus_uri,
                collection_name=f"corpassist_{tenant_id}",
                dim=1536
            )
            self.collections[tenant_id] = VectorStoreIndex.from_vector_store(vector_store)
        return self.collections[tenant_id]

配置管理

多索引系统的配置项多（数据库地址、租户映射、路由规则），需要集中管理。推荐 YAML 配置文件 + Python dataclass 加载模式。

yaml
# config/rag_config.yaml
milvus:
  uri: "http://milvus:19530"
  default_dim: 1536
  index_type: "IVF_FLAT"
  nlist: 1024

tenants:
  - id: "hr"
    collection: "corpassist_hr"
    embedding_dim: 1536
    reranker: false
    similarity_top_k: 5
  - id: "it"
    collection: "corpassist_it"
    embedding_dim: 1536
    reranker: true
    similarity_top_k: 3
  - id: "finance"
    collection: "corpassist_finance"
    embedding_dim: 768  # 小维度加速
    similarity_top_k: 5

routing:
  selector: "llm"  # llm / embedding
  fallback_tenant: "general"
  timeout_ms: 3000
python
from pydantic import BaseModel
import yaml
from typing import List, Optional

class TenantConfig(BaseModel):
    id: str
    collection: str
    embedding_dim: int = 1536
    reranker: bool = False
    similarity_top_k: int = 5

class RoutingConfig(BaseModel):
    selector: str = "llm"
    fallback_tenant: str = "general"
    timeout_ms: int = 3000

class RAGConfig(BaseModel):
    milvus_uri: str
    tenants: List[TenantConfig]
    routing: RoutingConfig

    @classmethod
    def from_yaml(cls, path: str):
        with open(path) as f:
            data = yaml.safe_load(f)
        return cls(**data)

# 加载配置
config = RAGConfig.from_yaml("config/rag_config.yaml")
print(f"已加载 {len(config.tenants)} 个租户配置")

# 动态创建引擎
engines = {}
for t in config.tenants:
    vector_store = MilvusVectorStore(
        uri=config.milvus_uri,
        collection_name=t.collection,
        dim=t.embedding_dim
    )
    index = VectorStoreIndex.from_vector_store(vector_store)
    engines[t.id] = index.as_query_engine(similarity_top_k=t.similarity_top_k)

本章结论

  • 意图路由用 RouterQueryEngine + LLMSingleSelector，工具描述中列举关键词可提升路由准确率到 95%+
  • 权限过滤采用双层策略：检索前 metadata filter + 检索后 NodePostprocessor 二次确认
  • 多租户隔离：中小租户用共享索引+metadata 过滤，大型租户用独立 Collection

  • CorpAssist 10 个以内租户推荐共享索引方案，运维成本低且支持跨租户搜索

面试可能问

共享索引和独立索引各自有什么优缺点？CorpAssist 怎么选？

共享索引优点：运维简单（一个 Collection、一套备份策略）、天然支持跨租户搜索。缺点：filter 配置错误可能导致数据泄露、单 Collection 容量有限（Milvus 推荐 <500 万向量）。独立索引优点：物理隔离最安全、容量理论上无限、独立备份和恢复。缺点：管理多个 Collection 运维成本高、跨租户搜索需要分别查询合并。CorpAssist 初始阶段（<10 租户，每个 <50 万文档）用共享索引；大租户入驻（>10 万文档）后切到独立索引。

RouterQueryEngine 路由错误时（如把 IT 问题路由到 HR），怎么排查和修复？

排查：1) 看 verbose 输出中 selector 对各工具的 score（分数差距小说明描述不够区分）；2) 检查 ToolMetadata.description 是否包含足够多的关键词。修复：1) 在描述中添加该域特有的术语和反例（"非HR问题如VPN配置"）；2) 改用 PydanticSingleSelector 添加结构化约束（如 "category = 'hr' | 'it' | 'finance'"）；3) 在系统 prompt 中添加路由规则说明。

NodePostprocessor 执行顺序重要吗？在 CorpAssist 中怎么安排？

顺序重要，因为后处理器按数组顺序执行，每个的输出是下一个的输入。CorpAssist 顺序：1) SimilarityPostprocessor（先过滤低分噪音）；2) PermissionFilter（再过滤无权内容）；3) SentenceTransformerRerank（最后对保留的内容重排序）。这样安排的理由：权限过滤在重排序之前，避免对无权内容做无意义的排序运算。每个后处理器都减少了后续处理的输入量，提升整体效率。

判断练习

场景：HR 部门的员工可以查看 HR 政策，但薪资信息只有 HR 经理才能看，怎么设计？

双层过滤：1) 检索层：所有 HR 文档 metadata 中标记 department=hr，HR 员工检索时 filter department=hr，只返回 HR 文档；2) Node 级别权限：包含薪资的 Node 额外标记 required_role=hr_manager，PermissionFilter 检查用户角色。这样 HR 员工搜"薪资"不会返回结果（因为 metadata filter 不够精确，但 NodePostprocessor 会拦截）；HR 经理搜"薪资"可以看到。

场景：用户查询"请帮我查一下 VPN 配置和年假政策"，RouterQueryEngine 怎么处理？

建议用 SubQuestionQueryEngine + RouterQueryEngine 组合。SubQuestionQueryEngine 先将问题拆为两个子问题："VPN 配置是什么"和"年假政策是什么"，然后分别路由到 IT 和 HR 索引，最后合并结果。如果只用 RouterQueryEngine，它会选择一个"最匹配"的索引，丢失另一个域的信息。所以检测到复合意图时，应切换到子查询模式。
