为什么学这章

CorpAssist 知识库当前使用 LangChain 做 RAG，但索引管理和检索灵活性不足。LlamaIndex 以 Document/Node/Index 为核心的数据模型，能直接映射企业文档结构（PDF 章节、代码段、数据库表），且内置 10+ 索引类型。团队需要决策：是继续 LangChain LCEL 路线，还是引入 LlamaIndex 作为检索层。

学完后能做什么

对比 Document/Node/Index vs LangChain Document/Splitter/VectorStore 的映射差异；解释 LlamaIndex 的"索引即抽象层"设计哲学；在 CorpAssist 场景中给出选型建议并列出 3 条迁移收益。

本章要点速记

1) LlamaIndex 用 Node 代替 LangChain 的 Document Chunk，自带 metadata 和关系；2) Index 是中心抽象，封装向量库+检索策略；3) LCEL 是声明式管线，LlamaIndex 是面向索引的封装；4) 路由查询是 LlamaIndex 独有特性；5) CorpAssist 适合用 LlamaIndex 做检索层 + LangChain 做对话层。

Document/Node/Index 数据模型

LlamaIndex 的数据模型由三个层次构成。Document 是原始输入的容器，对应一篇 PDF、一个 Markdown 文件或一段代码。Document 不要求纯文本——可以是 JSON、CSV 甚至二进制流，通过 metadata 字典挂载来源信息。Node 是 LlamaIndex 最小语义单元，每个 Node 包含文本块、元数据和关系列表（relationships），关系可以指向前驱/后继 Node 或引用，这对理解文档结构至关重要。Index 是对 Node 集合的变换与存储抽象，不限于向量索引——还有树索引、关键词表索引等。

flowchart TD
  D[Document: PDF/HTML/代码] -->|IngestionPipeline| N[Node 列表]
  N -->|嵌入 + 存储| VI[VectorStoreIndex]
  N -->|关键词提取| KI[KeywordTableIndex]
  N -->|文档树| SI[SummaryIndex]
  VI -->|query| QE[QueryEngine]
  QE -->|top-k| R[检索结果]
  subgraph LangChain 对照
    LDoc[Document] -->|TextSplitter| LChunk[Document chunks]
    LChunk -->|embedding| LVS[VectorStore]
  end

LangChain 的 Document 经 TextSplitter 切分后仍是 Document 列表，丢失了段落关系。LlamaIndex 的 Node 通过 NodeRelationship.SOURCE、NodeRelationship.NEXT、NodeRelationship.PREVIOUS 保留文档拓扑，在进行 SentenceWindowNodeParser 等高级检索时能回溯上下文窗口。

python
from llama_index.core.schema import Document, TextNode, NodeRelationship

# Document 是输入容器
doc = Document(
    text="# 第一章\nCorpAssist 是企业智能助手...",
    metadata={"source": "corpassist_manual.pdf", "page": 1, "version": "v2.1"}
)

# Node 是最小检索单元，保留关系
node = TextNode(
    text="CorpAssist 是企业智能助手",
    metadata={"source": "corpassist_manual.pdf", "page": 1},
    relationships={
        NodeRelationship.SOURCE: doc.as_related_node_info(),
        NodeRelationship.NEXT: TextNode(text="它支持以下功能...").as_related_node_info()
    }
)
print(f"Node ID: {node.node_id}, 关系数: {len(node.relationships)}")
# Node ID: a1b2c3..., 关系数: 2

与 LangChain LCEL 对照

LangChain LCEL 使用 | 操作符将组件串联成管道：prompt | model | output_parser。LlamaIndex 则围绕 Index 提供 as_query_engine()、as_chat_engine() 等方法。两者哲学不同：LCEL 是函数式管线，LlamaIndex 是面向索引的服务封装。

检索策略VectorStore.similarity_searchIndex.as_retriever(similarity_top_k=5)
多步推理LCEL chain 手动串联RouterQueryEngine 自动路由
流式支持.stream() 在链上.as_query_engine(streaming=True)

LangChain LCEL 方式

chain = prompt | model | output_parser 适合简单对话，但在 CorpAssist 多文档检索场景中，需要手写 RouterChain 或 LLMChain 组合，代码量多且调试困难。

LlamaIndex 方式

engine = index.as_query_engine() 一行即可。需要路由时切换 RouterQueryEngine.from_defaults(...)，检索与生成深度集成，调试时查看 retrieved Node 元数据即可。

python
# LangChain 方式
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
prompt = ChatPromptTemplate.from_template("基于{context}回答：{question}")
chain = {"context": retriever, "question": RunnablePassthrough()} | prompt | model | output_parser

# LlamaIndex 方式（更简洁）
from llama_index.core import VectorStoreIndex
index = VectorStoreIndex.from_documents(documents)
query_engine = index.as_query_engine(similarity_top_k=3, streaming=True)
response = query_engine.query("CorpAssist 支持哪些集成方式?")
print(response)

CorpAssist 场景

在 CorpAssist 中，用户上传的文档来自多个部门（HR 手册、IT 运维指南、产品文档）。用 LangChain 需要为每个文档集创建独立 VectorStore 并手写路由。LlamaIndex 的 Index 抽象允许一个 SummaryIndex 管理多个文档，通过 RouterQueryEngine 自动路由到对应子索引。

选型决策树

选型不是非此即彼。CorpAssist 架构中，对话管理用 LangChain 的 ChatModel 接口（Spring AI Python 侧兼容），而检索层用 LlamaIndex。以下是决策树：

flowchart TD
    Q[需要什么能力？] --> Q1{需要复杂文档索引?}
    Q1 -->|是,多文档关联| LI[选 LlamaIndex]
    Q1 -->|否,简单向量检索| LC[选 LangChain]

    C -->|客服| KB[选 LlamaIndex + 缓存]
    C -->|代码助手| LI
    C -->|简单 FAQ| LC

CorpAssist 场景适配

CorpAssist 采用双栈架构：Java Spring AI 处理对话编排，Python 服务处理检索和 RAG。LlamaIndex 在 Python 侧承担检索层职责。适配策略如下：

  • 文档摄入管线：Python 侧用 LlamaIndex IngestionPipeline 解析 PDF/HTML/代码，生成带关系的 Node，存入 Milvus 向量库。
  • 检索 API：FastAPI 封装 Index.as_retriever()，暴露 /api/v1/retrieve 接口，返回 JSON 给 Java 侧。
  • 双栈对齐：LlamaIndex 的 TextNode metadata 字段与 Java 侧 Document 类 JSON 对应，保证 session_id、tenant_id 透传。
  • 增量更新：Python 侧监听消息队列（RocketMQ），文档变更时调用 index.insert_nodes() 增量更新，无需重建全量。
python
# CorpAssist Python 侧检索服务
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from llama_index.core import VectorStoreIndex
from llama_index.vector_stores.milvus import MilvusVectorStore

app = FastAPI()
vector_store = MilvusVectorStore(uri="http://milvus:19530", dim=1024)
index = VectorStoreIndex.from_vector_store(vector_store)

class QueryRequest(BaseModel):
    question: str
    top_k: int = 3
    tenant_id: str  # 多租户

class QueryResponse(BaseModel):
    answer: str
    source_nodes: list[str]

@app.post("/api/v1/retrieve", response_model=QueryResponse)
async def retrieve(req: QueryRequest):
    retriever = index.as_retriever(
        similarity_top_k=req.top_k,
        filters={"tenant_id": req.tenant_id}  # 租户过滤
    )
    nodes = retriever.retrieve(req.question)
    return QueryResponse(
        answer=f"检索到 {len(nodes)} 条相关结果",
        source_nodes=[n.node_id for n in nodes]
    )

本章结论

  • LlamaIndex 的 Document/Node/Index 三层模型比 LangChain 的 Document-Chunk-VectorStore 更细粒度，Node 通过 relationships 保留文档拓扑，支持高级检索策略
  • LCEL 是函数式管线适合简单链，LlamaIndex Index + QueryEngine 是面向场景的封装，路由和子查询内置
  • CorpAssist 场景推荐双框架共存：LlamaIndex 做检索层，LangChain/LangGraph 做对话编排层
  • LlamaIndex 内置评估模块在迭代周期上比 LangChain + LangSmith 更快，且无需额外付费
  • 双栈架构中 LlamaIndex Python 侧负责检索 API，通过 JSON 协议与 Java 侧交互，metadata 字段对齐可保证多租户隔离

面试可能问

LlamaIndex 的 Node 和 LangChain 的 Document 有什么区别？为什么 LlamaIndex 要重新定义数据模型？

LangChain Document 只包含 page_content 和 metadata，切分后丢失了段落间关系。LlamaIndex 的 Node 多了 relationships 字段，可以指向 SOURCE（来源 Document）、NEXT/PREVIOUS（前后段落）、PARENT（父节点）。这在 SentenceWindow 检索中至关重要：检索到子节点时可以回溯到父节点获取完整上下文。LlamaIndex 重新定义模型是因为检索需要文档结构感知，而不仅是扁平文本。

CorpAssist 选择 LlamaIndex 还是 LangChain？请给出决策逻辑。

按文档复杂度决策：1) 单文档、百页内、简单 FAQ → LangChain RetrievalQA，代码少维护成本低；2) 多文档交叉检索、结构文档（PDF 带目录）→ LlamaIndex，Node 关系和 RouterQueryEngine 天然适合；3) 需要 Agent + 检索 → LangChain Agent 通过 QueryEngineTool 调用 LlamaIndex。CorpAssist 客服场景有多部门文档，推荐 LlamaIndex 检索层 + LangChain 对话层共存。

LlamaIndex 的 Index 是一个 VectorStore 吗？有哪些 Index 类型？

不是。Index 是抽象层，VectorStoreIndex 只是其中一种。还有 SummaryIndex（按文档顺序合成答案）、KeywordTableIndex（关键词匹配）、TreeIndex（树形摘要索引）、KnowledgeGraphIndex（知识图谱索引）等。每种 Index 实现了不同的检索策略，共享相同的 Node 数据模型。

判断练习

场景：有一个 500 页的 PDF 用户手册，需要实现"第几章的第几节提到什么功能"这种精确检索，选 LangChain 还是 LlamaIndex？

选 LlamaIndex。因为 500 页 PDF 包含章节层级，LangChain 的 TextSplitter 切分后丢失了"第几章第几节"信息。LlamaIndex 的 Node 通过 relationships 保留文档拓扑，配合 SentenceWindowNodeParser 可以回溯到父章节标题。具体实现：用 IngestionPipeline + HierarchicalNodeParser 生成父-子层级 Node，检索时返回子节点和父节点标题。

场景：团队已有 LangChain + FAISS 实现，想迁移到 LlamaIndex，最安全的切入口是什么？

最安全切入口是检索层替换：保留 LangChain 的 ChatModel 和 Prompt Template，只将 FAISS 检索替换为 LlamaIndex VectorStoreIndex。因为 LlamaIndex 的 index.as_retriever() 返回的 Node 可以从 metadata 提取文本，传入 LangChain 的 prompt。这样改动范围最小，验证通过后再逐步替换。迁移成本约 2-3 人天。
