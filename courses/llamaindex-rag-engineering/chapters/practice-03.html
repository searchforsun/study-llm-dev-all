本章要点速记

1) LlamaIndex 的 RetrieverEvaluator 评测检索（MRR、Recall）；2) RAGAS 的 Faithfulness 衡量答案是否基于检索内容（幻觉检测）；3) Ablation 实验每次只改一个变量（如 window_size），对比基线指标变化；4) 评测集应人工标注 100-200 条 QA 对作为黄金标准；5) 迭代顺序：先优化检索召回率，再优化答案质量。

LlamaIndex eval 模块

RetrieverEvaluator 评测检索器的召回能力。ResponseEvaluator 评测最终答案质量。两者都需要黄金标准数据集（人工标注的 query → relevant_docs 映射）。

与 RAGAS 结合

RAGAS 是一个专注于 RAG 评估的开源框架，提供更细粒度的指标。LlamaIndex 可以通过 llama_index.core.evaluation.RagasEvaluatorPack 或直接使用 RAGAS 库集成。

python
# pip install ragas llama-index-llms-openai

from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
    answer_correctness
)
from datasets import Dataset
import pandas as pd

# 准备评测数据
eval_data = {
    "question": ["年假几天?", "VPN 怎么配置?", ...],
    "answer": [response1, response2, ...],         # 系统输出
    "contexts": [[ctx1], [ctx2], ...],              # 检索到的文本
    "ground_truth": ["工作满一年可享5天年假", ...],  # 标准答案
}

dataset = Dataset.from_dict(eval_data)

# RAGAS 评测
result = evaluate(
    dataset,
    metrics=[
        faithfulness,          # 答案是否基于 context
        answer_relevancy,      # 答案与问题相关性
        context_precision,     # context 中相关信息比例
        context_recall,        # 标准答案信息是否被 context 覆盖
        answer_correctness     # 答案与标准答案的一致性
    ]
)

df = result.to_pandas()
print(df[["faithfulness", "context_recall", "answer_correctness"]].mean())
# faithfulness: 0.92 | context_recall: 0.78 | answer_correctness: 0.85

只用人工评测

每次改动需要 2 人天评测 200 条 QA，反馈周期长。迭代频率：每周 1-2 次。容易遗漏边缘 case。

RAGAS 自动评测

每次改动 3 分钟完成 200 条自动化评测，反馈即时。迭代频率：每天 5-10 次。自动发现 faithfulness 下降等退化问题。

思考10秒：RAGAS 的 faithfulness 和 context_recall 有什么区别？

Faithfulness 衡量"答案中的每句话是否都能在检索到的 context 中找到支持"——如果有回答"5 天年假"但 context 中只有"年假"没有"5 天"，则 faithfulness 低。Context_recall 衡量"标准答案的信息是否被检索到的 context 覆盖"——标准答案有"5 天"和"年假"两个信息点，如果 context 只包含"年假"没有"5 天"，则 context_recall 低。简单区别：faithfulness = 答案没说 context 没有的；context_recall = context 包含了答案需要的。

Ablation 实验设计

Ablation 实验（消融实验）每次只改动一个变量，测量其对指标的影响。这是 RAG 管线优化的核心方法。

import json
from dataclasses import dataclass, asdict

@dataclass
class AblationConfig:
    name: str
    window_size: int = 0
    reranker: bool = False
    hybrid: bool = False
    embedding_dim: int = 1536

def run_ablation(cfg: AblationConfig) -> dict:
    """运行一次 Ablation 实验，返回评测指标"""
    # 1. 构建配置对应的管线
    config = build_pipeline_config(cfg)

    # 2. 构建索引
    index = build_index(config)

    # 3. 评测
    retriever = index.as_retriever(similarity_top_k=5)
    eval_results = await retriever_eval.aevaluate_dataset(qa_dataset)

    return {
        "mrr": eval_results["mrr"].mean(),
        "recall@5": eval_results["recall_at_5"].mean(),
        "latency_ms": measure_latency(index)  # 平均查询延迟
    }

# 运行 Ablation 实验序列
experiments = [
    AblationConfig(name="baseline"),
    AblationConfig(name="+sentence_window", window_size=3),
    AblationConfig(name="+reranker", window_size=3, reranker=True),

评测报告模板

评测报告需要包含实验配置、指标数据和分析结论，便于团队决策和后续对比。

python
def generate_eval_report(results: list[dict], output_path: str):
    """生成 Markdown 格式的评测报告"""
    df = pd.DataFrame(results)

    report = f"""# RAG 管线评测报告
## 配置说明
- 数据集: {len(qa_dataset.queries)} 条黄金标准 QA
- 评测日期: 2026-05-30
- 基线版本: v2.1.0
- 嵌入模型: text-embedding-3-small (dim={df.iloc[0]['embedding_dim']})

## Ablation 结果

| 实验 | MRR | Recall@5 | 延迟(ms) | vs 基线 |
"""
    baseline_mrr = df.iloc[0]['mrr']
    for _, row in df.iterrows():
        change = f"+{row['mrr']-baseline_mrr:.3f}" if row['mrr'] >= baseline_mrr else f"{row['mrr']-baseline_mrr:.3f}"
        report += f"| {row['name']} | {row['mrr']:.3f} | {row['recall@5']:.3f} | {row['latency_ms']:.0f} | {change} |\n"

    report += f"""
## 结论
- 最优配置: SentenceWindow + Reranker + Hybrid
- 相比基线 MRR 提升: {(df.iloc[3]['mrr']-baseline_mrr)/baseline_mrr*100:.1f}%
- 推荐生产配置: SentenceWindow + Hybrid（延迟增量 <100ms，MRR 提升 >15%）
"""
    with open(output_path, "w", encoding="utf-8") as f:
        f.write(report)
    print(f"报告已生成: {output_path}")

面试可能问

如果 RAG 评测显示 MRR 高（0.85）但 Faithfulness 低（0.60），问题出在哪？怎么修？

MRR 高表示检索器找到了正确答案，但 Faithfulness 低表示最终答案存在幻觉（回答了检索内容之外的信息）。问题在合成阶段（ResponseSynthesizer）不在检索阶段。修复方法：1) 缩小 response_mode（从 tree_summarize 切换到 compact，减少 LLM 自由发挥空间）；2) 在 prompt 中强调"只基于检索内容回答，不知道就说不知道"；3) 增加 SimilarityPostprocessor 过滤低分 Node（>0.75），减少噪音输入导致的幻觉。

Ablation 实验中发现 +Reranker 使 MRR 提升了但 Recall 下降了，怎么理解？

Reranker 对检索结果重新排序，把最相关的提到前面（提升 MRR），但同时也过滤了部分可能相关的结果（降低 Recall）。这是排序质量 vs 召回覆盖率的固有权衡。对于 CorpAssist 客服场景，MRR 更重要（用户更关注前几个结果），所以接受 Recall 的小幅下降。但如果场景需要穷举所有相关信息（如合规审查），则需要降低 Reranker 的 top_n 值或不用 Reranker。

RAGAS 的 context_precision 和 MRR 有什么区别？

MRR 关注第一个正确答案的位置，不考虑其他结果的准确性。Context Precision 关注所有检索结果中相关信息（有用）占全部结果的比例，考虑精度。举例：检索到 5 条，只有第 1 条相关 → MRR=1.0（完美），context_precision=0.2（低）。两者互补：MRR 衡量是否能把正确答案排到前面，context_precision 衡量是否带了噪音。CorpAssist 要求 MRR>0.85 且 context_precision>0.6。

判断练习

场景：Ablation 结果显示 +Reranker 使延迟增加了 500ms 但 MRR 只提升了 0.02，怎么决策？

对于 CorpAssist 的在线场景，推荐不加 Reranker。500ms 延迟会显著影响用户体验（SLA 是 3 秒），而 MRR 收益 0.02 的实际体验差异很小。如果离线分析场景（如批量处理历史数据），可以加 Reranker 追求极致精度。替代方案：用使用更快的 Reranker 模型（如 BGE-reranker-v2-m3 比 cross-encoder 快 2 倍），或者只在复杂查询上启用 Reranker（通过 RouterQueryEngine 分流）。

场景：Automatic 评测（RAGAS）分数高，但人工评测发现很多答案不正确，哪里出了问题？

典型问题：RAGAS 的 LLM-as-judge 有偏差。1) 评测 LLM 偏好较长、格式漂亮的答案（长度偏差）；2) 评测 LLM 与回答 LLM 使用相同模型家族（模型偏差）；3) 黄金标准 QA 对覆盖不全（评测集偏差）。修复：1) 用不同模型做 judge（如 DeepSeek 评测 GPT 输出）；2) 人工抽检 10% 的评测结果校准 RAGAS 分数；3) 定期更新评测集，覆盖更多边缘 case。
