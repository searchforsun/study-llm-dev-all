本章要点速记

FunctionTool 定义

FunctionTool 是 LlamaIndex 将 Python 函数包装为 LLM 可调用工具的标准方式。它自动提取函数的签名、类型注解和 docstring，生成 OpenAI 兼容的 Function Calling Schema。

python
from llama_index.core.tools import FunctionTool
from typing import Optional
import json

# 1. 定义业务函数（CorpAssist 工单查询）
def query_ticket(ticket_id: str, user_id: Optional[str] = None) -> dict:
    """查询工单状态和详情。

    Args:
        ticket_id: 工单编号，格式如 TKT-2026-001
        user_id: 用户ID（可选），用于鉴权

    Returns:
        包含工单状态、创建时间、处理人的字典
    """
    # 模拟工单系统查询
    tickets_db = {
        "TKT-2026-001": {"status": "处理中", "created": "2026-05-20", "assignee": "张三"},
        "TKT-2026-002": {"status": "已完成", "created": "2026-05-19", "assignee": "李四"},
    }
    result = tickets_db.get(ticket_id, {"error": "工单不存在"})
    if user_id:
        result["user_id"] = user_id
    return result

# 2. 包装为 FunctionTool（自动生成 Schema）
ticket_tool = FunctionTool.from_defaults(
    fn=query_ticket,
    name="query_ticket",        # 工具名（LLM 用这个名字调用）
    description="根据工单编号查询工单状态、处理人和创建时间"
)

# 3. 查看自动生成的 Schema
print(json.dumps(ticket_tool.metadata.fn_schema_str, indent=2, ensure_ascii=False))
# {
#   "type": "object",
#   "properties": {
#     "ticket_id": {"type": "string", "description": "工单编号，格式如 TKT-2026-001"},
#     "user_id": {"type": "string", "description": "用户ID（可选），用于鉴权"}
#   },
#   "required": ["ticket_id"]

ReAct Agent with LlamaIndex

ReAct Agent 实现"思考-行动-观察"循环。每次循环中：Agent 调用 LLM 生成思考（Thought）和行动（Action），执行工具后获取观察（Observation），继续循环直到生成最终答案（Answer）。LlamaIndex 的 ReActAgent 通过 max_iterations 控制循环次数。

flowchart LR
    Q[用户查询] --> A[Agent]
    subgraph ReAct 循环
        A --> LLM[LLM 推理]
        LLM -->|思考: 需要查工单| ACT[行动: query_ticket]
        ACT -->|执行| OBS[观察: 工单状态]
        OBS -->|思考: 还需要什么?| ACT2[行动: query_user_info]
        ACT2 -->|执行| OBS2[观察: 用户信息]
        OBS2 -->|思考: 可以回答了| FINAL[最终回答]
    end
    A -->|max_iterations 耗尽| STOP[停止]

from llama_index.core.agent import ReActAgent
from llama_index.llms.openai import OpenAI
from llama_index.core.tools import FunctionTool, QueryEngineTool

# 1. 准备工具
def create_schedule(title: str, datetime: str, attendees: list[str]) -> dict:
    """创建日程"""
    return {"status": "ok", "event_id": "EVT-001", "title": title}

    """查询用户信息"""
    return {"name": "张三", "department": "IT", "email": "zhangsan@corp.com"}

    FunctionTool.from_defaults(fn=query_ticket),
    FunctionTool.from_defaults(fn=create_schedule),

)

工具说明模糊

"""查询信息""" — LLM 不确定该工具查什么，可能错误调用。"明天下午2点"可能被解析为 "2026-05-01 14:00" 或 "2026-06-01 14:00"。

工具说明清晰

"""根据用户ID查询姓名、部门、邮箱等信息，用于了解用户基本信息""" — LLM 明确知道该工具用途。日期时间参数格式在 docstring 中明确："格式为 YYYY-MM-DD HH:mm"。

MCP 工具接入

MCP（Model Context Protocol）是 Anthropic 推出的标准工具协议。LlamaIndex 通过 mcp-client 库支持连接 MCP 工具服务器。CorpAssist 的邮件、日历、工单系统各自独立部署 MCP Server，Agent 通过统一协议调用。

python

思考10秒：MCP Server 和 FunctionTool 在 Agent 中使用有什么区别？

FunctionTool 是将本地 Python 函数包装为工具，调用在同一进程内完成，延迟低（<10ms）。MCP Server 是独立的 HTTP 服务，通过网络调用，延迟高（50-200ms）但可以独立部署和维护。CorpAssist 策略：内部工具（计算、格式化）用 FunctionTool；外部系统（邮箱、日历、工单系统）用 MCP Server。

工具调用错误处理

Agent 的工具调用可能失败：网络超时、参数错误、权限不足。LlamaIndex 提供错误重试、max_iterations 限制和降级策略。

python
from llama_index.core.agent import ReActAgent
from llama_index.core.tools import FunctionTool
from functools import wraps

# 1. 带重试的工具包装
def with_retry(max_retries=3, timeout=5.0):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_error = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except TimeoutError as e:
                    last_error = f"超时(尝试{attempt+1}/{max_retries})"
                    time.sleep(1 * (attempt + 1))  # 退避
                except Exception as e:
                    last_error = str(e)
                    break  # 非超时错误不重试
            return {"error": last_error, "fallback": "工具调用失败，请稍后重试"}
        return wrapper
    return decorator

@with_retry(max_retries=3)
def query_external_api(api_name: str, params: dict) -> dict:
    """调用外部 API，带自动重试"""
    # 实际 API 调用...
    pass

# 2. Agent 级别错误处理
agent = ReActAgent.from_tools(
    tools=[query_ticket_tool, mail_tool, calendar_tool],
    llm=llm,
    max_iterations=10,              # 防止死循环
    max_function_calls=10,          # 单次对话最大工具调用次数
    system_prompt=(
        "当工具调用失败时：\n"
        "1. 如果是超时错误，告诉用户系统繁忙请稍后重试\n"
        "2. 如果是权限错误，提示用户联系管理员\n"
        "3. 如果是参数错误，检查参数格式后重试一次\n"
        "4. 重试 2 次仍然失败，给出人工处理建议"
    )
)
错误类型处理策略用户提示
网络超时 (>5s)退避重试 3 次（1s/2s/4s）"系统正在处理，请稍后查询"
参数校验失败LLM 重新生成参数后重试自动处理，不提示用户
权限不足 (403)不重试，直接提示"您没有该操作的权限，请联系管理员"

API 限流 (429)等待后重试（Retry-After）"请求频繁，请稍后重试"

本章结论

  • FunctionTool 自动提取函数签名生成 Schema，清晰的 docstring 是 LLM 正确调用的关键
  • ReAct Agent 的循环机制支持多步推理，max_iterations=10 覆盖 95% 的业务场景
  • MCP 协议让 Agent 接入外部系统（邮件、日历、工单），每个系统独立部署 MCP Server
  • 错误处理需要分层策略：工具级重试 + Agent 级降级 + 用户级友好提示
  • CorpAssist 中内部计算用 FunctionTool，外部系统用 MCP 工具，通过 ReActAgent 统一调度

面试可能问

LlamaIndex 的 ReActAgent 和 LangChain 的 Agent 有什么区别？

LlamaIndex ReActAgent 基于 ReAct 循环格式（Thought/Action/Observation），每一步都结构化输出。LangChain Agent 支持更多 Agent 类型（OpenAI Tools、XML、Structured Chat）。LlamaIndex 优势：1) 与 Index 和 QueryEngine 深度集成，直接传 QueryEngineTool 即可；2) 工具定义更简洁（FunctionTool 自动生成 Schema）；3) 支持 MCP 协议。LangChain 优势：1) Agent 生态更丰富；2) 记忆和回调系统更成熟。CorpAssist 选择 LlamaIndex Agent 因为其检索能力原生集成。

如果 Agent 陷入死循环（不断调用同一个工具），怎么处理？

多层防护：1) max_iterations=10，达到后强制停止并返回当前结果；2) max_function_calls=10，限制总工具调用次数；3) 检测重复工具调用模式——如果连续 N 次调用相同工具且参数相同，触发中断；4) 在每个工具的执行结果中添加"fix_hint"字段，引导 LLM 换方向。CorpAssist 还加了一个监控：单次对话超过 20 步自动上报告警。

MCP 协议的安全性如何？在 CorpAssist 中怎么鉴权？

MCP Server 通过 HTTP 暴露，安全性取决于传输层。CorpAssist 方案：1) 每个 MCP Server 部署在内部网络，不对外暴露；2) 使用 mTLS 双向认证；3) 每个请求携带 JWT Token（包含用户 ID、角色、租户 ID）；4) MCP Server 端校验 Token，根据角色限制可用工具。Agent 调用工具时自动将用户上下文透传到 MCP 请求头中。

判断练习

场景：Agent 调用了 query_user_info 工具，但返回了空结果，接下来 Agent 应该怎么做？

ReAct Agent 的观察阶段会收到 {"error": "用户不存在"}。下一步推理为：用户不存在 → 可能是工单的创建者被删除 → 不阻碍工单处理流程 → 跳过用户查询直接返回工单状态。或者：需要用户信息但查不到 → 提示用户提供正确的用户 ID。Agent 会自动根据工具返回结果调整计划，不需要硬编码处理分支。

场景：MCP Server 返回了非预期的数据格式（如字符串而不是 JSON），Agent 会怎么处理？

ReAct Agent 将工具返回的原始文本作为 Observation 放入下一轮推理。LLM 会尝试解析：如果返回的是纯字符串"success"，Agent 会将其解读为工具执行成功并继续下一步。如果返回的是错误信息"500 Internal Server Error"，Agent 会在下一轮思考中说"工具调用返回了服务器错误，请检查 MCP Server 状态"，然后可能重试或跳过。规范的做法：MCP 工具统一返回 JSON 格式的 dict。
