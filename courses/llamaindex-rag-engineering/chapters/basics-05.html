为什么学这章

CorpAssist 的检索质量直接取决于嵌入向量的质量和向量数据库的查询性能。选错嵌入模型（如用 text-embedding-ada-002 处理代码）会导致代码检索准确率下降 30%。错误配置向量库（如 Milvus 索引类型）会导致查询延迟从 50ms 飙升到 5s。本章让你掌握嵌入选型和向量库调优。

学完后能做什么

根据语言和领域选择嵌入模型；配置 Milvus 或 Qdrant 的索引参数和集合 schema；实现批处理嵌入优化吞吐；管理索引的持久化和增量更新。

本章要点速记

1) text-embedding-3-small 性价比最高（1536 维，每千条 $0.002）；2) Milvus 用 IVF_FLAT 索引适合百万级，Qdrant 用 HNSW 适合千万级；3) 批处理大小 100 条/批时吞吐最高；4) 索引持久化需定期备份 metadata；5) embedding 维度与检索速度负相关——维度减半速度约快 40%。

Embedding 模型选择

Embedding 模型将文本映射为稠密向量，向量间的余弦相似度反映语义相关度。选择嵌入模型的三个维度：语言覆盖、维度（影响存储和速度）、领域适应性。

    batch_size=32
)

# 测试嵌入质量

doc_vec = embed_model.get_text_embedding("请假制度说明")
similarity = sum(a*b for a,b in zip(query_vec, doc_vec))
print(f"语义相似度: {similarity:.4f}")  # 预期 >0.85

CorpAssist 场景

CorpAssist 服务 60% 中文文档 + 30% 英文文档 + 10% 中英混合。选择 text-embedding-3-small 维度缩减为 768：MTEB 得分从 62.3 降到 60.1（降幅 3.5%），但检索速度提升 40%，存储减少 50%。对于知识库场景，速度收益大于精度损失。

思考10秒：什么情况下不应该用 OpenAI 的嵌入模型？

1) 数据隐私敏感——HR 手册包含员工个人信息，不能发送到外部 API；2) 网络隔离——CorpAssist 部署在内网，无法访问 OpenAI API；3) 成本敏感——每天处理 100 万条文档索引，text-embedding-3-small 每天约 $2，大模型 $13/天；4) 需要特定领域调优——如法律文档专用嵌入。以上场景改用 BGE-M3 本地部署或私有化嵌入服务。

Milvus/Qdrant 集成

LlamaIndex 通过 VectorStore 抽象层支持 20+ 向量数据库。CorpAssist 主要用两种：Milvus 适合百万级向量（CorpAssist 主库），Qdrant 适合低延迟在线场景。

批处理优化

嵌入生成和向量写入是 RAG 管线中耗时最长的两个环节。批处理（batch processing）是优化的核心手段。LlamaIndex 在 Settings.embed_model 和 MilvusVectorStore 中分别提供了批处理参数。

python
import time
from llama_index.core.ingestion import IngestionPipeline
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.vector_stores.milvus import MilvusVectorStore

# 批处理参数调优
embed_model = OpenAIEmbedding(
    model="text-embedding-3-small",
    batch_size=100,           # 每批 100 条（OpenAI 限制 2048）
    embed_batch_size=100,     # 内部嵌入批大小
    max_retries=3
)

vector_store = MilvusVectorStore(
    uri="http://milvus:19530",
    batch_size=200,           # 每批写入 200 条向量
    enable_sparse=False       # 关闭稀疏向量提升写入速度
)

pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=512),
        embed_model
    ],
    vector_store=vector_store
)

# 批量处理 1000 个文档（2000 Node）
t0 = time.time()
nodes = pipeline.run(documents=documents[:1000])
t1 = time.time()
print(f"处理 {len(nodes)} Node 耗时 {t1-t0:.2f}s")
# 预期：1000 文档约 2000 Node，耗时 15-25s（含嵌入 API 调用）

# 优化建议：
# 1. batch_size=100 是吞吐最优值（<50 网络利用率低，>200 超时风险高）
# 2. 写入 Milvus 前先创建索引（index_config），避免边写边建索引
# 3. 使用 asyncio 并行化：文档读取和嵌入计算异步执行
参数默认值推荐值优化依据
embed_model.batch_size10100OpenAI API 并发限制，100 条/批吞吐最高
vector_store.batch_size100200Milvus 批量写入 200 条/批延迟最优
max_retries33嵌入 API 偶尔超时，3 次重试覆盖 99.9%
num_workers14文档读取和数据预处理可以并行
show_progressFalseTrue调试时开启，生产关闭（减少 tqdm 开销）

索引持久化

索引持久化涉及两个层面：向量数据的持久化（Milvus/Qdrant 自带）和 Index 对象的元数据持久化（LlamaIndex 提供 persist() 方法）。后者保存的是 Index 结构信息（文档 ID、节点关系、关键词表等），不保存向量本身。

python
from llama_index.core import VectorStoreIndex, StorageContext, load_index_from_storage
from llama_index.core.storage import StorageContext
from llama_index.vector_stores.milvus import MilvusVectorStore
import os

# --- 持久化 ---

# 1. 向量数据写入 Milvus（自动持久化，无需手动调用）
vector_store = MilvusVectorStore(uri="http://milvus:19530")
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex.from_documents(
    documents, storage_context=storage_context
)

# 2. Index 元数据持久化到本地磁盘
index.storage_context.persist(persist_dir="./storage/index_meta_v2")
print("Index 元数据已保存到 ./storage/index_meta_v2")
# 保存的文件：
# ./storage/index_meta_v2/docstore.json （文档与 Node 映射）
# ./storage/index_meta_v2/index_store.json（Index 结构）
# ./storage/index_meta_v2/vector_store.json（向量库连接信息）

# --- 恢复 ---
storage_context = StorageContext.from_defaults(
    persist_dir="./storage/index_meta_v2",
    vector_store=MilvusVectorStore(uri="http://milvus:19530")
)
index = load_index_from_storage(storage_context)

# --- 增量更新 ---
from llama_index.core import Document
new_doc = Document(text="新文档内容: CorpAssist v3.0 发布说明...")
index.insert(new_doc)
print(f"索引文档数: {len(index.ref_doc_info)}")

# --- 自动清理 ---
def rotate_storage(max_versions=5):
    """保留最近 5 个版本的索引元数据"""
    import shutil
    storage_dirs = sorted([d for d in os.listdir("./storage") if d.startswith("index_meta_v")])
    for old_dir in storage_dirs[:-max_versions]:
        shutil.rmtree(f"./storage/{old_dir}")
        print(f"清理过期索引: {old_dir}")

CorpAssist 场景

每天凌晨 2 点全量重建索引：读取所有文档 → IngestionPipeline → 写入 Milvus 新 Collection → 备份旧 Collection → 切换别名。保留最近 7 天的索引快照用于回滚。索引元数据（docstore.json）版本控制在 Git 中，方便追踪文档变更历史。

本章结论

  • Embedding 模型选型三因素：语言覆盖、维度（1536 通用）、领域适配（代码用 code-embedding 或 text-embedding-3-small 缩减维度的性能最优解）
  • Milvus 用 IVF_FLAT（nlist=1024）索引适合百万级向量，Qdrant 用 HNSW（m=16）适合低延迟在线查询
  • 批处理优化参数 embed_batch_size=100 + vector_store.batch_size=200 时吞吐最高，1000 文档可在 20 秒内完成嵌入和写入
  • 索引持久化分两层：向量数据由向量库管理，Index 元数据由 LlamaIndex persist() 管理，两者缺一不可
  • 增量更新用 index.insert() 而非重建全量，metadata 版本控制在 Git 中便于回滚

面试可能问

text-embedding-3-small 和 text-embedding-ada-002 有哪些区别？为什么要升级？

text-embedding-3-small 是 ada-002 的替代品。核心区别：1) 输出维度可缩减（默认 1536，可设为 512 甚至 256），ada-002 固定 1536 维；2) 支持 Matryoshka Representation Learning，缩减维度不会线性降低质量（1536→512 质量仅降 3%）；3) MTEB 得分 62.3 vs 61.0；4) 价格相同但性能更优。升级原因：维度缩减后存储减少 67%，检索速度提升 40%，适合 CorpAssist 百万级向量场景。

Milvus 的 IVF_FLAT 和 HNSW 索引在什么场景各有什么优势？

IVF_FLAT：建索引速度快（适合频繁全量重建的场景），精度高但查询速度线性增长。HNSW：建索引慢（需要构建多层图），但查询速度极快（log N 复杂度）。选型建议：CorpAssist 知识库每天全量重建一次用 IVF_FLAT（建库快）；在线查询层用 HNSW（查询快）。对于 100 万向量规模，IVF_FLAT 查询 P99 约 100ms，HNSW 约 20ms。

LlamaIndex 的 persist() 保存了什么？为什么向量数据还要单独持久化？

persist() 保存 Index 的元数据：docstore.json（Document→Node 映射和 Node 文本）、index_store.json（Index 结构和索引类型信息）、vector_store.json（向量库连接参数）。不保存向量数据本身，因为向量数据量巨大（100 万 × 1536 维 × 4 字节 ≈ 6GB），不适合写 JSON。向量数据由专门的向量数据库（Milvus/Qdrant）持久化。恢复时需要同时提供 persist_dir（元数据）和 vector_store 连接（向量数据）。

判断练习

场景：1000 万条知识库文档，查询 P99 延迟从 100ms 升到 800ms，怎么优化？

原因：Milvus IVF_FLAT 索引在数据量增大后，nprobe=16 探测 1024 个分区的查询效率下降。优化方案：1) 重建索引为 HNSW，m=16, ef_construction=200，查询延迟可降至 50ms 以下；2) 增加分区数 nlist 从 1024 到 4096；3) 增加副本数（readonly 副本），负载均衡；4) 使用 GPU 索引（GPU_IVF_FLAT）将查询卸载到 GPU。CorpAssist 场景推荐 GPU_IVF_FLAT + 2 个只读副本。

场景：HR 部门要求嵌入计算不能在外部 API 上完成，怎么选型？

选 BGE-M3 本地部署。BAAI/bge-m3 支持中文、英文、中英混合，1024 维，MTEB 得分 63.0（与 text-embedding-3-small 相近）。部署方式：1) CPU 推理：用 ONNX Runtime 优化，每秒处理 50 条文本；2) GPU 推理：单张 A10 每秒处理 500+ 条。CorpAssist 场景：4 核 CPU + 16GB 内存的实例约可承载每天 10 万文档的嵌入计算。成本对比：本地部署每月约 ¥2000（服务器费用），OpenAI API 每月约 ¥5000（100 万文档）。
