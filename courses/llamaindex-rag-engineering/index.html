Document/Node/Index 数据模型

LlamaIndex 的数据模型由三个层次构成。Document 是原始输入的容器，对应一篇 PDF、一个 Markdown 文件或一段代码。Document 不要求纯文本——可以是 JSON、CSV 甚至二进制流，通过 metadata 字典挂载来源信息。Node 是 LlamaIndex 最小语义单元，每个 Node 包含文本块、元数据和关系列表（relationships），关系可以指向前驱/后继 Node 或引用，这对理解文档结构至关重要。Index 是对 Node 集合的变换与存储抽象，不限于向量索引——还有树索引、关键词表索引等。

flowchart TD
  D[Document: PDF/HTML/代码] -->|IngestionPipeline| N[Node 列表]
  N -->|嵌入 + 存储| VI[VectorStoreIndex]
  N -->|关键词提取| KI[KeywordTableIndex]
  N -->|文档树| SI[SummaryIndex]
  VI -->|query| QE[QueryEngine]
  QE -->|top-k| R[检索结果]
  subgraph LangChain 对照
    LDoc[Document] -->|TextSplitter| LChunk[Document chunks]
    LChunk -->|embedding| LVS[VectorStore]
  end

LangChain 的 Document 经 TextSplitter 切分后仍是 Document 列表，丢失了段落关系。LlamaIndex 的 Node 通过 NodeRelationship.SOURCE、NodeRelationship.NEXT、NodeRelationship.PREVIOUS 保留文档拓扑，在进行 SentenceWindowNodeParser 等高级检索时能回溯上下文窗口。

python
from llama_index.core.schema import Document, TextNode, NodeRelationship

# Document 是输入容器
doc = Document(
    text="# 第一章\nCorpAssist 是企业智能助手...",
    metadata={"source": "corpassist_manual.pdf", "page": 1, "version": "v2.1"}
)

# Node 是最小检索单元，保留关系
node = TextNode(
    text="CorpAssist 是企业智能助手",
    metadata={"source": "corpassist_manual.pdf", "page": 1},
    relationships={
        NodeRelationship.SOURCE: doc.as_related_node_info(),
        NodeRelationship.NEXT: TextNode(text="它支持以下功能...").as_related_node_info()
    }
)
print(f"Node ID: {node.node_id}, 关系数: {len(node.relationships)}")
# Node ID: a1b2c3..., 关系数: 2

LangChain LCEL 方式

chain = prompt | model | output_parser 适合简单对话，但在 CorpAssist 多文档检索场景中，需要手写 RouterChain 或 LLMChain 组合，代码量多且调试困难。

LlamaIndex 方式

engine = index.as_query_engine() 一行即可。需要路由时切换 RouterQueryEngine.from_defaults(...)，检索与生成深度集成，调试时查看 retrieved Node 元数据即可。

python
# LangChain 方式
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
prompt = ChatPromptTemplate.from_template("基于{context}回答：{question}")
chain = {"context": retriever, "question": RunnablePassthrough()} | prompt | model | output_parser

# LlamaIndex 方式（更简洁）
from llama_index.core import VectorStoreIndex
index = VectorStoreIndex.from_documents(documents)
query_engine = index.as_query_engine(similarity_top_k=3, streaming=True)
response = query_engine.query("CorpAssist 支持哪些集成方式?")
print(response)

CorpAssist 场景适配

CorpAssist 采用双栈架构：Java Spring AI 处理对话编排，Python 服务处理检索和 RAG。LlamaIndex 在 Python 侧承担检索层职责。适配策略如下：

  • 文档摄入管线：Python 侧用 LlamaIndex IngestionPipeline 解析 PDF/HTML/代码，生成带关系的 Node，存入 Milvus 向量库。
  • 检索 API：FastAPI 封装 Index.as_retriever()，暴露 /api/v1/retrieve 接口，返回 JSON 给 Java 侧。
  • 双栈对齐：LlamaIndex 的 TextNode metadata 字段与 Java 侧 Document 类 JSON 对应，保证 session_id、tenant_id 透传。
  • 增量更新：Python 侧监听消息队列（RocketMQ），文档变更时调用 index.insert_nodes() 增量更新，无需重建全量。
python
# CorpAssist Python 侧检索服务
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from llama_index.core import VectorStoreIndex
from llama_index.vector_stores.milvus import MilvusVectorStore

app = FastAPI()
vector_store = MilvusVectorStore(uri="http://milvus:19530", dim=1024)
index = VectorStoreIndex.from_vector_store(vector_store)

class QueryRequest(BaseModel):
    question: str
    top_k: int = 3
    tenant_id: str  # 多租户

class QueryResponse(BaseModel):
    answer: str
    source_nodes: list[str]

@app.post("/api/v1/retrieve", response_model=QueryResponse)
async def retrieve(req: QueryRequest):
    retriever = index.as_retriever(
        similarity_top_k=req.top_k,
        filters={"tenant_id": req.tenant_id}  # 租户过滤
    )
    nodes = retriever.retrieve(req.question)
    return QueryResponse(
        answer=f"检索到 {len(nodes)} 条相关结果",
        source_nodes=[n.node_id for n in nodes]
    )

本章结论

  • LlamaIndex 的 Document/Node/Index 三层模型比 LangChain 的 Document-Chunk-VectorStore 更细粒度，Node 通过 relationships 保留文档拓扑，支持高级检索策略
  • LCEL 是函数式管线适合简单链，LlamaIndex Index + QueryEngine 是面向场景的封装，路由和子查询内置
  • CorpAssist 场景推荐双框架共存：LlamaIndex 做检索层，LangChain/LangGraph 做对话编排层
  • LlamaIndex 内置评估模块在迭代周期上比 LangChain + LangSmith 更快，且无需额外付费
  • 双栈架构中 LlamaIndex Python 侧负责检索 API，通过 JSON 协议与 Java 侧交互，metadata 字段对齐可保证多租户隔离

面试可能问

LlamaIndex 的 Node 和 LangChain 的 Document 有什么区别？为什么 LlamaIndex 要重新定义数据模型？

LangChain Document 只包含 page_content 和 metadata，切分后丢失了段落间关系。LlamaIndex 的 Node 多了 relationships 字段，可以指向 SOURCE（来源 Document）、NEXT/PREVIOUS（前后段落）、PARENT（父节点）。这在 SentenceWindow 检索中至关重要：检索到子节点时可以回溯到父节点获取完整上下文。LlamaIndex 重新定义模型是因为检索需要文档结构感知，而不仅是扁平文本。

CorpAssist 选择 LlamaIndex 还是 LangChain？请给出决策逻辑。

按文档复杂度决策：1) 单文档、百页内、简单 FAQ → LangChain RetrievalQA，代码少维护成本低；2) 多文档交叉检索、结构文档（PDF 带目录）→ LlamaIndex，Node 关系和 RouterQueryEngine 天然适合；3) 需要 Agent + 检索 → LangChain Agent 通过 QueryEngineTool 调用 LlamaIndex。CorpAssist 客服场景有多部门文档，推荐 LlamaIndex 检索层 + LangChain 对话层共存。

LlamaIndex 的 Index 是一个 VectorStore 吗？有哪些 Index 类型？

不是。Index 是抽象层，VectorStoreIndex 只是其中一种。还有 SummaryIndex（按文档顺序合成答案）、KeywordTableIndex（关键词匹配）、TreeIndex（树形摘要索引）、KnowledgeGraphIndex（知识图谱索引）等。每种 Index 实现了不同的检索策略，共享相同的 Node 数据模型。

场景：有一个 500 页的 PDF 用户手册，需要实现"第几章的第几节提到什么功能"这种精确检索，选 LangChain 还是 LlamaIndex？

选 LlamaIndex。因为 500 页 PDF 包含章节层级，LangChain 的 TextSplitter 切分后丢失了"第几章第几节"信息。LlamaIndex 的 Node 通过 relationships 保留文档拓扑，配合 SentenceWindowNodeParser 可以回溯到父章节标题。具体实现：用 IngestionPipeline + HierarchicalNodeParser 生成父-子层级 Node，检索时返回子节点和父节点标题。

场景：团队已有 LangChain + FAISS 实现，想迁移到 LlamaIndex，最安全的切入口是什么？

最安全切入口是检索层替换：保留 LangChain 的 ChatModel 和 Prompt Template，只将 FAISS 检索替换为 LlamaIndex VectorStoreIndex。因为 LlamaIndex 的 index.as_retriever() 返回的 Node 可以从 metadata 提取文本，传入 LangChain 的 prompt。这样改动范围最小，验证通过后再逐步替换。迁移成本约 2-3 人天。

PDF/HTML/代码解析

LlamaIndex 的 IngestionPipeline 是一个可配置的文档处理管线，通过 transformations 列表定义每个处理步骤。对于不同文档类型，需要选择合适的解析器：PDF 用 PyMuPDFReader 保留文本布局；HTML 用 HtmlReader 提取标题层级；代码文件用 SimpleDirectoryReader 配合 CodeSplitter 保留语言语义。

python
from llama_index.core import SimpleDirectoryReader
from llama_index.core.node_parser import CodeSplitter, SentenceSplitter
from llama_index.core.ingestion import IngestionPipeline
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.core.extractors import TitleExtractor, QuestionsAnsweredExtractor

# 代码文件解析
code_splitter = CodeSplitter(
    language="python",
    chunk_lines=50,            # 每块 50 行
    chunk_lines_overlap=10,    # 10 行重叠
    max_chars=1500             # 最大字符数
)

# PDF/文本解析
text_splitter = SentenceSplitter(
    chunk_size=512,
    chunk_overlap=128
)

pipeline = IngestionPipeline(
    transformations=[
        text_splitter,           # 或 code_splitter（按需切换）
        TitleExtractor(),        # 自动提取标题
        QuestionsAnsweredExtractor(questions=3),  # 提取可能的问题
        OpenAIEmbedding(model="text-embedding-3-small", dimensions=512)
    ]
)

# 读取目录中的文档
documents = SimpleDirectoryReader("./corp_docs").load_data()
nodes = pipeline.run(documents=documents)
print(f"生成了 {len(nodes)} 个 Node")
for n in nodes[:2]:
    print(f"  {n.node_id}: {n.metadata.get('title', '无标题')}")

Unstructured 集成

对于复杂 PDF（含表格、多栏布局、扫描件），LlamaIndex 基础解析器可能丢失布局信息。Unstructured 是一个开源文档解析框架，支持 PDF、图片、PPT、邮件等格式，能将文档还原为结构化元素（标题、段落、表格、列表）。LlamaIndex 通过 UnstructuredReader 集成。

flowchart LR
    PDF[复杂 PDF] --> UR[UnstructuredReader]
    HTML --> UR
    IMG[扫描件图片] --> OCR[OCR 预处理]
    OCR --> UR
    UR --> ELE[元素列表]
    ELE -->|Title| T1[# 报销流程]
    ELE -->|Table| T2[|部门|金额|]
    ELE -->|Narrative| T3[报销需要提供发票...]
    T1 & T2 & T3 --> ST[Splitter]
    ST --> N1[Node 1: 报销流程]
    ST --> N2[Node 2: 金额表格]
    ST --> N3[Node 3: 发票说明]
python
from llama_index.readers.file import UnstructuredReader
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter

# Unstructured 解析 PDF（保留表格和标题）
reader = UnstructuredReader(
    api_key=None,  # 用本地模式，无需 API Key
    strategy="hi_res"  # hi_res 适合复杂布局，auto 适合简单文档
)

# 解析 PDF 返回 Document 列表（每个元素一个 Document）
documents = reader.load_data(
    file_path="./corp_docs/hr_handbook.pdf",
    extra_info={"department": "hr", "version": "v3.2"}
)

# 后续管线
pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=256, chunk_overlap=32),
        OpenAIEmbedding()
    ]
)
nodes = pipeline.run(documents=documents)
print(f"Unstructured 生成了 {len(documents)} 个原始元素")
print(f"管线产出 {len(nodes)} 个 Node")

思考10秒：什么情况下不应该用 Unstructured？

1) 纯文本 Markdown 文档——Unstructured 增加了处理时间（hi_res 模式每页约 2-3 秒），直接用 MarkdownReader 更快；2) 只需要简单文本搜索——PyMuPDF 的纯文本输出足够，无需结构化元素；3) API 成本敏感——Unstructured Cloud API 按页计费，本地模式依赖 Tesseract OCR 需要额外安装。

元数据是 RAG 质量的关键，它决定了检索时能否按租户、日期、文档类型过滤。LlamaIndex 提供四种元数据提取方式：MetadataExtractor 基类、PropertyExtractor 内置提取器、自定义提取器、以及 IngestionPipeline 中的全局提取器。

class TenantExtractor(TitleExtractor):

async def process_document(file_path: str, tenant_id: str, doc_type: str):
    # 1. 文档读取
    if doc_type == "pdf" and "table" in file_path:
        reader = UnstructuredReader(strategy="hi_res")
    else:
        reader = PyMuPDFReader()
    docs = reader.load_data(file_path)

    # 2. 注入租户元数据
    for doc in docs:
        doc.metadata["tenant_id"] = tenant_id
        doc.metadata["doc_type"] = doc_type
        doc.metadata["upload_time"] = os.path.getmtime(file_path)

    # 3. 管线处理
    vector_store = MilvusVectorStore(
        uri="http://milvus:19530",
        collection_name=f"corpassist_{tenant_id}",  # 按租户隔离
        dim=1536
    )
    pipeline = IngestionPipeline(
        transformations=[
            SentenceSplitter(chunk_size=512, chunk_overlap=128),
            TitleExtractor(),
            OpenAIEmbedding(model="text-embedding-3-small", dimensions=1536)
        ],
        vector_store=vector_store
    )
    nodes = pipeline.run(documents=docs)
    return {"status": "ok", "node_count": len(nodes), "tenant_id": tenant_id}

面试可能问

IngestionPipeline 的 transformations 顺序重要吗？为什么？

重要。transformations 按顺序执行，每个 transform 的输出是下一个的输入。典型的顺序是：先分割（产生小文本块），再提取元数据（在分割后的文本上提取标题和关键词），最后嵌入（对最终文本块生成向量）。如果先嵌入再分割，每个大块的向量会丢失细粒度语义。正确的顺序可以保证每个 Node 的向量精准对应其文本内容。

CorpAssist 中一个 PDF 含有表格，检索时既要返回表格又要返回文字描述，怎么设计解析管线？

用 UnstructuredReader 的 hi_res 模式，它会将表格识别为独立的 Table 元素，文字描述为 Narrative 元素。IngestionPipeline 中用自定义 Splitter：对 Table 元素不分割（保持表格完整性），对 Narrative 元素用 SentenceSplitter 分割。在 metadata 中标记 element_type=table/narrative，检索时可以根据查询类型（"金额"可能查表格，"流程"可能查文字）过滤。

CodeSplitter 支持哪些语言？分割单元是什么？

CodeSplitter 支持 Python、Java、JavaScript、TypeScript、C++、Go、Rust 等主流语言。分割单元是：Python 按函数/类定义，Java 按方法签名，JavaScript 按函数声明。通过 tree-sitter 解析 AST 确定分割边界，保证分割后的代码块语法完整。参数包括 chunk_lines（默认 50）、chunk_lines_overlap（默认 10）、max_chars（默认 1500）。

VectorStoreIndex 与语义检索

VectorStoreIndex 是最常用的索引类型。它接收 Node 列表，调用嵌入模型生成向量，写入向量数据库，查询时用 ANN（近似最近邻）搜索。在 LlamaIndex 中，VectorStoreIndex 封装了嵌入、存储和检索三个环节，用户只需指定向量库和嵌入模型。

    QV -->|ANN 搜索| VS

    R -->|排序+组合| S[合成阶段]

SummaryIndex 与文档摘要

SummaryIndex 不依赖向量检索。它维护 Node 的原始顺序，查询时将检索到的 Node 按原序拼接后送入 LLM 生成答案。适合"总结整个文档"或"按时间顺序叙述"的场景。

用 VectorStoreIndex 做总结

向量检索只能返回语义相似的片段，但可能丢失时间顺序。员工问"本周团队完成了什么"，可能只返回"完成知识库建设"这条相似度高的，遗漏"周一修复Bug、周二上线新功能"等顺序信息。

用 SummaryIndex 做总结

按 Node 顺序拼接，LLM 能看到完整时间线。回答"周一修复了3个生产Bug，周二上线了新功能X，周三..."，信息完整且按序排列。

思考10秒：SummaryIndex 的 response_mode 各有什么优劣势？

1) compact：拼接 Node 文本后单次 LLM 调用，Token 省但可能超长，适合 20 个以内 Node。2) tree_summarize：递归两两合并摘要，适合大量 Node（100+），但速度慢。3) accumulate：逐 Node 回答后 LLM 再次合并，独立性强但 Token 消耗多 1.5x。CorpAssist 日报总结用 tree_summarize，10 个日报约 50 Node，递归 3 层完成。

KeywordTableIndex 与关键词匹配

KeywordTableIndex 从每个 Node 中提取关键词（使用 LLM 或简单 TF-IDF），构建关键词→Node 的映射表。查询时提取查询关键词，精确匹配 Node。适合代码搜索、法规条款查询等需要精确语义匹配而非向量相似度的场景。

索引选择决策表

实际项目中很少只用一种索引。RouterQueryEngine 可以根据查询意图自动路由到最合适的索引，同时利用多种索引的优势。

    Q[用户查询] --> R[RouterQueryEngine]
    R -->|选择器: LLM/Embedding| S{意图判断}

    S -->|全文总结| SM[SummaryIndex]
    S -->|精确匹配| KW[KeywordTableIndex]
    S -->|层级检索| TR[TreeIndex]
    V -->|top-k scores| CR1[组合结果]

    CR1 & CR2 & CR3 --> CO[合成最终答案]

from llama_index.core.tools import QueryEngineTool, ToolMetadata
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.selectors import LLMSingleSelector
from llama_index.core import VectorStoreIndex, SummaryIndex, KeywordTableIndex

# 构建三种索引
vec_index = VectorStoreIndex.from_documents(documents)
sum_index = SummaryIndex.from_documents(documents)
kw_index = KeywordTableIndex.from_documents(documents)

# 为每个索引创建工具
vector_tool = QueryEngineTool(
    query_engine=vec_index.as_query_engine(similarity_top_k=3),
    metadata=ToolMetadata(
        name="vector_search",
        description="用于语义搜索，查找相关概念和描述"
    )
)
summary_tool = QueryEngineTool(
    query_engine=sum_index.as_query_engine(response_mode="tree_summarize"),
    metadata=ToolMetadata(
        name="summary",
        description="用于总结整个文档或多个文档的内容"
    )
)
keyword_tool = QueryEngineTool(
    query_engine=kw_index.as_query_engine(),
    metadata=ToolMetadata(
        name="exact_match",
        description="用于精确匹配关键词、代码函数名、条款编号"
    )
)

# 路由查询引擎
router = RouterQueryEngine(
    selector=LLMSingleSelector.from_defaults(),
    query_engine_tools=[vector_tool, summary_tool, keyword_tool],
    verbose=True
)

response = router.query("第37条规定了什么")
# Router 选择 exact_match → KeywordTableIndex

本章结论

  • VectorStoreIndex 适合需要语义泛化的场景（FAQ、知识搜索），ANN 检索延迟 50-200ms
  • SummaryIndex 按 Node 顺序拼接，适合文档摘要和时间线总结，response_mode 决定 Token 策略
  • KeywordTableIndex 精确匹配关键词，适合代码搜索和法规条款，0 嵌入成本但依赖 LLM 提取关键词
  • RouterQueryEngine 通过查询意图路由到最合适的索引，召回率比单一索引高 25-35%
  • CorpAssist 推荐使用 VectorStoreIndex + KeywordTableIndex 组合，RouterQueryEngine 统一入口

面试可能问

VectorStoreIndex 和 KeywordTableIndex 在检索逻辑上的本质区别是什么？

VectorStoreIndex 将文本和查询都嵌入为向量，用余弦距离或点积衡量相似度，本质是语义泛化——"年假"和"休假"相似度高。KeywordTableIndex 从文本中提取关键词（用 LLM 或统计方法），构建倒排索引，查询时精确匹配关键词，不涉及向量计算。核心区别：语义泛化 vs 精确匹配。CorpAssist 中 FAQ 用 VectorStoreIndex（用户表述多变），代码搜索用 KeywordTableIndex（函数名必须精确）。

RouterQueryEngine 的选择器有哪几种？各有什么优劣？

LlamaIndex 提供 4 种选择器：1) LLMSingleSelector：用 LLM 选一个工具，准确率高（98%）但延迟高（1-2s）；2) LLMMultiSelector：可选多个工具，适合复杂查询；3) PydanticSingleSelector：基于 Pydantic schema 选择，结构约束强；4) EmbeddingSingleSelector：用嵌入模型选，速度快（50ms）但准确率低（85%）。CorpAssist 场景用 LLMSingleSelector，1-2s 延迟对知识库场景可接受。

在什么场景下 TreeIndex 比 SummaryIndex 更合适？

TreeIndex 递归构建摘要树：根节点是全局摘要，子节点是章节摘要，叶子节点是原始文本。适合：1) 超长文档（100+ 页），直接 summaryIndex 的 compact 模式会超 Token 限制；2) 分层检索需求，用户从"公司政策"→"HR 政策"→"年假政策"逐层深入。TreeIndex 的树结构天然支持这种导航。但构建时间比 SummaryIndex 多 3-5 倍，因为需要多轮 LLM 调用生成摘要。

场景：CorpAssist 需要实现"从 5000 个 Java 代码文件中找到 createUser 方法的所有调用处"，选什么索引和参数？

场景：用户查询"说下上周的五个重要事项"，但 VectorStoreIndex 只返回了 2 个 Node。怎么改进？

RouterQueryEngine 实现路由

RouterQueryEngine 的核心是选择器（Selector）。选择器读取每个子引擎的描述（ToolMetadata.description），判断当前查询最适合哪个引擎。选择器有 LLMSingleSelector（高准确率）、PydanticSingleSelector（结构约束）、EmbeddingSingleSelector（低延迟）三种。

响应合成模式与 Spring AI 对照

LlamaIndex 的 ResponseSynthesizer 负责将检索到的 Node 合成为最终答案。有 5 种模式，影响答案质量和 Token 消耗。同时与 Spring AI 的查询模式做对照。

模式工作机制Token 消耗适用场景
compactNode 文本拼接后一次调用低（1 次 LLM 调用）Node 数量少（<8），单次能放得下
refine逐 Node 提炼，先结合第一个，叠加第二个修正中（N 次调用）Node 量大但需要逐步细化答案
tree_summarize递归两两合并中（logN 次调用）大量 Node（20+），需要全面摘要
accumulate逐 Node 独立回答后合并高（N+1 次调用）需要每个 Node 单独处理后再聚合
compact_accumulate分组拼接后逐组回答中（分组数+1 次调用）Node 可分组，各组的答案独立

Spring AI QueryExecutor

Spring AI 的 SearchRequest 只能指定 vectorStore 和 topK，不支持路由、子查询或响应合成策略。复杂查询需要在 Java 侧手动编排多个 SearchRequest 调用。

LlamaIndex QueryEngine

提供 5 种合成模式 + 路由 + 子查询，通过 response_mode 参数一键切换。与 Spring AI 的桥接：LlamaIndex 的 QueryEngine 作为 Python 侧检索服务，通过 FastAPI 暴露 /api/v1/query 接口供 Spring AI 调用。

面试可能问

RouterQueryEngine 和 SubQuestionQueryEngine 的区别是什么？各自适用什么场景？

RouterQueryEngine 是"按意图分发"：查询进入后，选择器判定它适合哪个引擎，然后该引擎独立处理。适合查询内容明确指向某个知识领域。SubQuestionQueryEngine 是"分解后组合"：查询被拆为多个子问题，每个子问题分配到不同引擎，结果再合并。适合跨域查询。举例：Router 用于"年假政策"（单一意图路由到 HR），SubQuestion 用于"对比 HR 年假和 IT VPN"（两个实体需要分别检索再比较）。

compact 和 refine 两种响应合成模式在 Token 消耗和答案质量上的差异？

compact 将所有检索到的 Node 文本拼接后一次性送入 LLM，单次调用 context window 内处理。Token 消耗低（1 次 LLM 调用 + 拼接文本）。但若 Node 总量超过 context window（如 128K），会截断尾部。refine 逐个 Node 处理：先用第一个 Node 生成初始答案，然后用第二个 Node 修正，依此类推。Token 消耗高（N 次 LLM 调用），但不会截断。CorpAssist 场景：top_k=5 时用 compact（单次能放下）；top_k=20 且每个 Node 很长时用 refine。

Spring AI 的 QueryExecutor 与 LlamaIndex 相比缺少什么能力？如何双栈补齐？

Spring AI 的 QueryExecutor 只提供了简单的 vectorStore.similaritySearch + 直接拼接发送给 LLM。缺少：1) 多种响应合成模式（只能简单拼接）；2) 路由能力（只能对单一 VectorStore 操作）；3) 子查询拆解；4) Node 后处理器（无法过滤低分结果）。补齐方案：Python 侧用 LlamaIndex QueryEngine 封装所有能力，暴露 HTTP API，Java 侧用 RestTemplate 或 WebClient 调用。

判断练习

场景：RouterQueryEngine 总是把"VPN 配置"查询路由到 HR 引擎而不是 IT 引擎，可能的原因和修复方法？

原因：IT 引擎的 ToolMetadata.description 不够具体，没有包含"VPN""配置""网络"等关键词。LLM 选择了描述中"包含类似名词较多的" HR 引擎。修复方法：将 IT 引擎描述改为"IT 网络与系统配置：VPN 配置、邮箱设置、系统权限申请、软件安装等企业 IT 技术支持"。确保描述中包含目标查询的关键词。如果仍然路由错误，检查 LLMSingleSelector 使用的 LLM 是否够强（推荐 GPT-4 或 DeepSeek-V3）。

场景：查询返回答案截断了，原因是 10 个 Node 拼接后超出 128K context window，怎么解决？

方案一：切换 response_mode=refine，逐 Node 处理不拼接，不会超长但多轮调用。方案二：用 tree_summarize，递归合并摘要，每次只处理两个 Node 的摘要。方案三：减少 top_k 到 5，降低拼接长度。方案四：增大 chunk_size（从 512 到 1024），减少 Node 数量但保留更多单个 Node 内容。CorpAssist 推荐 tree_summarize + top_k=5，平衡质量和 Token 消耗。

面试可能问

text-embedding-3-small 和 text-embedding-ada-002 有哪些区别？为什么要升级？

text-embedding-3-small 是 ada-002 的替代品。核心区别：1) 输出维度可缩减（默认 1536，可设为 512 甚至 256），ada-002 固定 1536 维；2) 支持 Matryoshka Representation Learning，缩减维度不会线性降低质量（1536→512 质量仅降 3%）；3) MTEB 得分 62.3 vs 61.0；4) 价格相同但性能更优。升级原因：维度缩减后存储减少 67%，检索速度提升 40%，适合 CorpAssist 百万级向量场景。

Milvus 的 IVF_FLAT 和 HNSW 索引在什么场景各有什么优势？

IVF_FLAT：建索引速度快（适合频繁全量重建的场景），精度高但查询速度线性增长。HNSW：建索引慢（需要构建多层图），但查询速度极快（log N 复杂度）。选型建议：CorpAssist 知识库每天全量重建一次用 IVF_FLAT（建库快）；在线查询层用 HNSW（查询快）。对于 100 万向量规模，IVF_FLAT 查询 P99 约 100ms，HNSW 约 20ms。

LlamaIndex 的 persist() 保存了什么？为什么向量数据还要单独持久化？

persist() 保存 Index 的元数据：docstore.json（Document→Node 映射和 Node 文本）、index_store.json（Index 结构和索引类型信息）、vector_store.json（向量库连接参数）。不保存向量数据本身，因为向量数据量巨大（100 万 × 1536 维 × 4 字节 ≈ 6GB），不适合写 JSON。向量数据由专门的向量数据库（Milvus/Qdrant）持久化。恢复时需要同时提供 persist_dir（元数据）和 vector_store 连接（向量数据）。

判断练习

场景：1000 万条知识库文档，查询 P99 延迟从 100ms 升到 800ms，怎么优化？

原因：Milvus IVF_FLAT 索引在数据量增大后，nprobe=16 探测 1024 个分区的查询效率下降。优化方案：1) 重建索引为 HNSW，m=16, ef_construction=200，查询延迟可降至 50ms 以下；2) 增加分区数 nlist 从 1024 到 4096；3) 增加副本数（readonly 副本），负载均衡；4) 使用 GPU 索引（GPU_IVF_FLAT）将查询卸载到 GPU。CorpAssist 场景推荐 GPU_IVF_FLAT + 2 个只读副本。

场景：HR 部门要求嵌入计算不能在外部 API 上完成，怎么选型？

选 BGE-M3 本地部署。BAAI/bge-m3 支持中文、英文、中英混合，1024 维，MTEB 得分 63.0（与 text-embedding-3-small 相近）。部署方式：1) CPU 推理：用 ONNX Runtime 优化，每秒处理 50 条文本；2) GPU 推理：单张 A10 每秒处理 500+ 条。CorpAssist 场景：4 核 CPU + 16GB 内存的实例约可承载每天 10 万文档的嵌入计算。成本对比：本地部署每月约 ¥2000（服务器费用），OpenAI API 每月约 ¥5000（100 万文档）。

1) SentenceWindowNodeParser 检索到句子后自动附加前后 N 句，窗口大小控制上下文量；2) AutoMergingRetriever 按文档层级合并父节点，避免检索到不完整的子片段；3) 递归检索先粗后细，适合层级文档；4) 窗口越大召回率越高但延迟增加，窗口=5 是性价比拐点；5) 高级策略需要 IngestionPipeline 配合生成正确的 Node 关系。

SentenceWindowNodeParser

SentenceWindowNodeParser 将文档按句子分割，每个句子作为一个 Node，同时保留其前后句子的 ID。检索时返回匹配的句子 Node，但回答时自动扩展为包含上下文的"窗口"，让 LLM 看到更完整的语境。

AutoMergingRetriever

AutoMergingRetriever 解决文档结构问题。当检索到的多个子节点来自同一个父节点时，自动合并为父节点文本，避免 LLM 看到零散的碎片信息。它依赖 HierarchicalNodeParser 预先建立父子层级关系。

python
from llama_index.core.node_parser import HierarchicalNodeParser
from llama_index.core.retrievers import AutoMergingRetriever
from llama_index.core.indices.postprocessor import SentenceTransformerRerank
from llama_index.core import VectorStoreIndex, StorageContext
from llama_index.vector_stores.milvus import MilvusVectorStore

# 1. 层级解析：生成父子两层 Node
node_parser = HierarchicalNodeParser.from_defaults(
    chunk_sizes=[1024, 256],  # 父 1024 字符，子 256 字符
    chunk_overlap=20
)

# 2. 读取并解析
documents = SimpleDirectoryReader("./corp_docs").load_data()
nodes = node_parser.get_nodes_from_documents(documents)

# 3. 分别存储父节点和子节点
vector_store = MilvusVectorStore(uri="http://milvus:19530", dim=1536)
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex(nodes, storage_context=storage_context)

# 4. AutoMerging 检索器
base_retriever = index.as_retriever(similarity_top_k=6)
retriever = AutoMergingRetriever(
    vector_retriever=base_retriever,
    storage_context=storage_context,  # 用于查找父节点
    simple_ratio_thresh=0.5  # 50% 以上的子节点被检索到时合并
)

# 5. 查询引擎
query_engine = index.as_query_engine(
    retriever=retriever,
    node_postprocessors=[
        SentenceTransformerRerank(model="BAAI/bge-reranker-v2-m3", top_n=3)
    ]
)
resp = query_engine.query("报销流程是什么？")
# AutoMerging 自动合并了"报销条件""报销金额""报销材料"三个子节点为父节点"报销流程"
print(resp)

递归检索

递归检索（RecursiveRetriever）是一种"先粗后细"的策略：第一轮检索到概要信息，根据概要信息第二轮检索更具体的内容。适合章节-段落-句子层级的文档，如 CorpAssist 的 HR 手册。

面试可能问

SentenceWindowNodeParser 和 AutoMergingRetriever 有什么区别？

两者解决不同问题。SentenceWindow 解决"检索到正确句子但缺少上下文"的问题：检索时返回精准的短文本，但给 LLM 的上下文窗口是扩展后的。AutoMerging 解决"检索到零散片段"的问题：将属于同一父节点的多个子节点合并为完整段落。SentenceWindow 按时间/位置扩展上下文，AutoMerging 按文档结构合并内容。实际项目中两者可以组合使用。

AutoMergingRetriever 的 simple_ratio_thresh=0.5 是什么意思？调大或调小有什么影响？

当检索到的子节点中有 50% 以上同属一个父节点时，触发合并。调大到 0.8：更谨慎合并，减少误合并但可能漏合并（只合并高度确定的片段）。调小到 0.3：更积极合并，更多片段被合并但可能包含不相关内容。CorpAssist 测试：0.5 时答案完整率 92%，0.3 时 94% 但 Token 消耗增加 20%，0.8 时 88%。推荐 0.5 作为默认值。

在 CorpAssist 中，如何设计两阶段检索策略来平衡质量和延迟？

第一阶段（快速通道）：基础 VectorStoreIndex + SentenceWindow(window=1)，无 Reranker，总延迟 <1s。覆盖 80% 的简单查询（事实性问题）。第二阶段（深度通道）：全栈组合——HierarchicalNodeParser → AutoMergingRetriever → RecursiveRetriever → BGE Reranker，总延迟 <3s。覆盖 20% 的复杂查询（分析、比较、总结）。RouterQueryEngine 根据查询复杂度自动选择通道。监控两阶段的命中率分布，当复杂查询占比超过 30% 时优化快速通道的召回能力。

判断练习

场景：SentenceWindow 检索后答案仍然不正确，可能的原因和排查方法？

原因：1) window_size 太小，提供的上下文仍不够（调到 5 测试）；2) MetadataReplacementPostProcessor 未正确配置（检查 target_metadata_key="window"）；3) 解析时 window 文本没有正确保存（检查 Node.metadata 中是否有 'window' 字段）；4) 窗口文本超过了 LLM context window（窗口太大被截断）。排查方法：打印 resp.source_nodes 中的 metadata 字段，确认替换前后的文本内容。

场景：AutoMerging 从不触发合并，检查哪些配置环节？

检查顺序：1) HierarchicalNodeParser 是否正确生成了父子关系（检查 Node.relationships 中是否有 PARENT/CHILD 关系）；2) storage_context 是否正确传递给了 AutoMergingRetriever（需要从索引的 storage_context 读取父节点）；3) simple_ratio_thresh 是否太高（调到 0.3 测试）；4) 检索到的子节点是否真的同属一个父节点（打印子节点的 parent_id 验证）。常见错误是忘记传递 storage_context。

FunctionTool 定义

FunctionTool 是 LlamaIndex 将 Python 函数包装为 LLM 可调用工具的标准方式。它自动提取函数的签名、类型注解和 docstring，生成 OpenAI 兼容的 Function Calling Schema。

python
from llama_index.core.tools import FunctionTool

import json

# 1. 定义业务函数（CorpAssist 工单查询）
def query_ticket(ticket_id: str, user_id: Optional[str] = None) -> dict:
    """查询工单状态和详情。

    Args:
        ticket_id: 工单编号，格式如 TKT-2026-001
        user_id: 用户ID（可选），用于鉴权

    Returns:
        包含工单状态、创建时间、处理人的字典
    """
    # 模拟工单系统查询
    tickets_db = {
        "TKT-2026-001": {"status": "处理中", "created": "2026-05-20", "assignee": "张三"},
        "TKT-2026-002": {"status": "已完成", "created": "2026-05-19", "assignee": "李四"},
    }
    result = tickets_db.get(ticket_id, {"error": "工单不存在"})
    if user_id:
        result["user_id"] = user_id
    return result

ReAct Agent with LlamaIndex

ReAct Agent 实现"思考-行动-观察"循环。每次循环中：Agent 调用 LLM 生成思考（Thought）和行动（Action），执行工具后获取观察（Observation），继续循环直到生成最终答案（Answer）。LlamaIndex 的 ReActAgent 通过 max_iterations 控制循环次数。

flowchart LR
    Q[用户查询] --> A[Agent]
    subgraph ReAct 循环
        A --> LLM[LLM 推理]
        LLM -->|思考: 需要查工单| ACT[行动: query_ticket]
        ACT -->|执行| OBS[观察: 工单状态]
        OBS -->|思考: 还需要什么?| ACT2[行动: query_user_info]
        ACT2 -->|执行| OBS2[观察: 用户信息]
        OBS2 -->|思考: 可以回答了| FINAL[最终回答]
    end
    A -->|max_iterations 耗尽| STOP[停止]

思考10秒：MCP Server 和 FunctionTool 在 Agent 中使用有什么区别？

FunctionTool 是将本地 Python 函数包装为工具，调用在同一进程内完成，延迟低（<10ms）。MCP Server 是独立的 HTTP 服务，通过网络调用，延迟高（50-200ms）但可以独立部署和维护。CorpAssist 策略：内部工具（计算、格式化）用 FunctionTool；外部系统（邮箱、日历、工单系统）用 MCP Server。

def with_retry(max_retries=3, timeout=5.0):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_error = None
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except TimeoutError as e:
                    last_error = f"超时(尝试{attempt+1}/{max_retries})"
                    time.sleep(1 * (attempt + 1))  # 退避
                except Exception as e:
                    last_error = str(e)
                    break  # 非超时错误不重试
            return {"error": last_error, "fallback": "工具调用失败，请稍后重试"}
        return wrapper
    return decorator

面试可能问

LlamaIndex 的 ReActAgent 和 LangChain 的 Agent 有什么区别？

LlamaIndex ReActAgent 基于 ReAct 循环格式（Thought/Action/Observation），每一步都结构化输出。LangChain Agent 支持更多 Agent 类型（OpenAI Tools、XML、Structured Chat）。LlamaIndex 优势：1) 与 Index 和 QueryEngine 深度集成，直接传 QueryEngineTool 即可；2) 工具定义更简洁（FunctionTool 自动生成 Schema）；3) 支持 MCP 协议。LangChain 优势：1) Agent 生态更丰富；2) 记忆和回调系统更成熟。CorpAssist 选择 LlamaIndex Agent 因为其检索能力原生集成。

如果 Agent 陷入死循环（不断调用同一个工具），怎么处理？

多层防护：1) max_iterations=10，达到后强制停止并返回当前结果；2) max_function_calls=10，限制总工具调用次数；3) 检测重复工具调用模式——如果连续 N 次调用相同工具且参数相同，触发中断；4) 在每个工具的执行结果中添加"fix_hint"字段，引导 LLM 换方向。CorpAssist 还加了一个监控：单次对话超过 20 步自动上报告警。

MCP 协议的安全性如何？在 CorpAssist 中怎么鉴权？

MCP Server 通过 HTTP 暴露，安全性取决于传输层。CorpAssist 方案：1) 每个 MCP Server 部署在内部网络，不对外暴露；2) 使用 mTLS 双向认证；3) 每个请求携带 JWT Token（包含用户 ID、角色、租户 ID）；4) MCP Server 端校验 Token，根据角色限制可用工具。Agent 调用工具时自动将用户上下文透传到 MCP 请求头中。

判断练习

场景：Agent 调用了 query_user_info 工具，但返回了空结果，接下来 Agent 应该怎么做？

ReAct Agent 的观察阶段会收到 {"error": "用户不存在"}。下一步推理为：用户不存在 → 可能是工单的创建者被删除 → 不阻碍工单处理流程 → 跳过用户查询直接返回工单状态。或者：需要用户信息但查不到 → 提示用户提供正确的用户 ID。Agent 会自动根据工具返回结果调整计划，不需要硬编码处理分支。

场景：MCP Server 返回了非预期的数据格式（如字符串而不是 JSON），Agent 会怎么处理？

ReAct Agent 将工具返回的原始文本作为 Observation 放入下一轮推理。LLM 会尝试解析：如果返回的是纯字符串"success"，Agent 会将其解读为工具执行成功并继续下一步。如果返回的是错误信息"500 Internal Server Error"，Agent 会在下一轮思考中说"工具调用返回了服务器错误，请检查 MCP Server 状态"，然后可能重试或跳过。规范的做法：MCP 工具统一返回 JSON 格式的 dict。

思考10秒：RAGAS 的 faithfulness 和 context_recall 有什么区别？

Faithfulness 衡量"答案中的每句话是否都能在检索到的 context 中找到支持"——如果有回答"5 天年假"但 context 中只有"年假"没有"5 天"，则 faithfulness 低。Context_recall 衡量"标准答案的信息是否被检索到的 context 覆盖"——标准答案有"5 天"和"年假"两个信息点，如果 context 只包含"年假"没有"5 天"，则 context_recall 低。简单区别：faithfulness = 答案没说 context 没有的；context_recall = context 包含了答案需要的。

面试可能问

如果 RAG 评测显示 MRR 高（0.85）但 Faithfulness 低（0.60），问题出在哪？怎么修？

MRR 高表示检索器找到了正确答案，但 Faithfulness 低表示最终答案存在幻觉（回答了检索内容之外的信息）。问题在合成阶段（ResponseSynthesizer）不在检索阶段。修复方法：1) 缩小 response_mode（从 tree_summarize 切换到 compact，减少 LLM 自由发挥空间）；2) 在 prompt 中强调"只基于检索内容回答，不知道就说不知道"；3) 增加 SimilarityPostprocessor 过滤低分 Node（>0.75），减少噪音输入导致的幻觉。

Ablation 实验中发现 +Reranker 使 MRR 提升了但 Recall 下降了，怎么理解？

Reranker 对检索结果重新排序，把最相关的提到前面（提升 MRR），但同时也过滤了部分可能相关的结果（降低 Recall）。这是排序质量 vs 召回覆盖率的固有权衡。对于 CorpAssist 客服场景，MRR 更重要（用户更关注前几个结果），所以接受 Recall 的小幅下降。但如果场景需要穷举所有相关信息（如合规审查），则需要降低 Reranker 的 top_n 值或不用 Reranker。

RAGAS 的 context_precision 和 MRR 有什么区别？

MRR 关注第一个正确答案的位置，不考虑其他结果的准确性。Context Precision 关注所有检索结果中相关信息（有用）占全部结果的比例，考虑精度。举例：检索到 5 条，只有第 1 条相关 → MRR=1.0（完美），context_precision=0.2（低）。两者互补：MRR 衡量是否能把正确答案排到前面，context_precision 衡量是否带了噪音。CorpAssist 要求 MRR>0.85 且 context_precision>0.6。

判断练习

场景：Ablation 结果显示 +Reranker 使延迟增加了 500ms 但 MRR 只提升了 0.02，怎么决策？

对于 CorpAssist 的在线场景，推荐不加 Reranker。500ms 延迟会显著影响用户体验（SLA 是 3 秒），而 MRR 收益 0.02 的实际体验差异很小。如果离线分析场景（如批量处理历史数据），可以加 Reranker 追求极致精度。替代方案：用使用更快的 Reranker 模型（如 BGE-reranker-v2-m3 比 cross-encoder 快 2 倍），或者只在复杂查询上启用 Reranker（通过 RouterQueryEngine 分流）。

场景：Automatic 评测（RAGAS）分数高，但人工评测发现很多答案不正确，哪里出了问题？

典型问题：RAGAS 的 LLM-as-judge 有偏差。1) 评测 LLM 偏好较长、格式漂亮的答案（长度偏差）；2) 评测 LLM 与回答 LLM 使用相同模型家族（模型偏差）；3) 黄金标准 QA 对覆盖不全（评测集偏差）。修复：1) 用不同模型做 judge（如 DeepSeek 评测 GPT 输出）；2) 人工抽检 10% 的评测结果校准 RAGAS 分数；3) 定期更新评测集，覆盖更多边缘 case。

权限过滤

权限过滤在检索完成后、答案合成前执行。用 NodePostprocessor 过滤掉用户无权查看的文档。LlamaIndex 支持自定义后处理器。

python
from llama_index.core.postprocessor import BaseNodePostprocessor
from llama_index.core.schema import NodeWithScore
from typing import Optional, List

class PermissionFilter(BaseNodePostprocessor):
    """根据用户角色过滤检索到的 Node"""
    def __init__(self, user_roles: List[str]):
        self.user_roles = user_roles

    def _postprocess_nodes(
        self, nodes: List[NodeWithScore],
        query_bundle: Optional = None
    ) -> List[NodeWithScore]:
        filtered = []
        for node in nodes:
            # Node metadata 中存储了 required_role
            required_role = node.metadata.get("required_role", "all")
            if required_role == "all" or required_role in self.user_roles:
                filtered.append(node)
        return filtered

# 给 Node 注入权限信息
def add_permission_metadata(nodes, role_map: dict):
    """role_map: {文档ID: [允许的角色列表]}"""
    for node in nodes:
        doc_id = node.metadata.get("doc_id", "")
        if doc_id in role_map:
            allowed_roles = role_map[doc_id]
            node.metadata["required_role"] = allowed_roles
        else:
            node.metadata["required_role"] = "all"
    return nodes

# 使用权限过滤
query_engine = index.as_query_engine(
    similarity_top_k=5,
    node_postprocessors=[
        SimilarityPostprocessor(similarity_cutoff=0.5),
        PermissionFilter(user_roles=["hr_manager", "admin"])
    ]
)

resp = query_engine.query("薪资调整政策是什么？")
# 如果用户没有 hr_manager 角色，包含薪资信息的 Node 会被过滤掉

检索前过滤（不安全）

只在检索时加 metadata filter。如果 filter 配置错误（漏写），用户可能看到无权访问的内容。攻击者可以通过 API 直接传参绕过。

检索后过滤（推荐）

检索时用 metadata filter 缩小范围，检索后用 NodePostprocessor 二次确认权限。双层保险。metadata filter 拦截 99% 无关文档，NodePostprocessor 拦截剩下的 1% 边界 case。

多租户索引隔离

多租户隔离有两个方案：独立索引（每个租户单独的 Milvus Collection）和共享索引（一个 Collection 内用 metadata 字段过滤）。选择取决于数据量和租户数量。

维度独立索引（每个租户一个 Collection）共享索引（一个 Collection + metadata 过滤）
数据隔离物理隔离，最强安全逻辑隔离，依赖 filter 正确性
运维成本高（每个租户独立建索引、独立备份）低（一个索引管理所有）
租户上限理论上无限（每个新租户建新 Collection）受限于 Collection 容量（Milvus 推荐 <500 万向量/Collection）
跨租户搜索需要搜索多个 Collection 合并结果天然支持（移除 filter 即可）
CorpAssist 推荐大型租户（>10 万文档）中小租户（<10 万文档）

面试可能问

共享索引和独立索引各自有什么优缺点？CorpAssist 怎么选？

共享索引优点：运维简单（一个 Collection、一套备份策略）、天然支持跨租户搜索。缺点：filter 配置错误可能导致数据泄露、单 Collection 容量有限（Milvus 推荐 <500 万向量）。独立索引优点：物理隔离最安全、容量理论上无限、独立备份和恢复。缺点：管理多个 Collection 运维成本高、跨租户搜索需要分别查询合并。CorpAssist 初始阶段（<10 租户，每个 <50 万文档）用共享索引；大租户入驻（>10 万文档）后切到独立索引。

RouterQueryEngine 路由错误时（如把 IT 问题路由到 HR），怎么排查和修复？

排查：1) 看 verbose 输出中 selector 对各工具的 score（分数差距小说明描述不够区分）；2) 检查 ToolMetadata.description 是否包含足够多的关键词。修复：1) 在描述中添加该域特有的术语和反例（"非HR问题如VPN配置"）；2) 改用 PydanticSingleSelector 添加结构化约束（如 "category = 'hr' | 'it' | 'finance'"）；3) 在系统 prompt 中添加路由规则说明。

NodePostprocessor 执行顺序重要吗？在 CorpAssist 中怎么安排？

顺序重要，因为后处理器按数组顺序执行，每个的输出是下一个的输入。CorpAssist 顺序：1) SimilarityPostprocessor（先过滤低分噪音）；2) PermissionFilter（再过滤无权内容）；3) SentenceTransformerRerank（最后对保留的内容重排序）。这样安排的理由：权限过滤在重排序之前，避免对无权内容做无意义的排序运算。每个后处理器都减少了后续处理的输入量，提升整体效率。

@app.post("/api/v1/upload")
async def upload_document(file: UploadFile, tenant_id: str = "default", background: BackgroundTasks):
    """上传文档并异步建立索引"""
    file_path = f"{UPLOAD_DIR}/{uuid.uuid4()}_{file.filename}"
    content = await file.read()
    with open(file_path, "wb") as f:
        f.write(content)

    background.add_task(process_document, file_path, tenant_id)
    return {"status": "accepted", "file_id": file_path}

async def process_document(file_path: str, tenant_id: str):
    """后台处理文档：解析→分割→嵌入→写入 Milvus"""
    docs = SimpleDirectoryReader(input_files=[file_path]).load_data()
    for doc in docs:
        doc.metadata["tenant_id"] = tenant_id

    pipeline = IngestionPipeline(
        transformations=[
            SentenceSplitter(chunk_size=512, chunk_overlap=128),
            Settings.embed_model
        ],
        vector_store=vector_store
    )
    nodes = pipeline.run(documents=docs)
    print(f"文档处理完成: {len(nodes)} nodes from {file_path}")

# app/routes/query.py — 检索和问答路由
from fastapi import APIRouter, Depends
from llama_index.core.postprocessor import SimilarityPostprocessor

router = APIRouter(prefix="/api/v1", tags=["query"])

def get_tenant_engine(tenant_id: str):
    """根据租户获取检索引擎"""
    return index.as_query_engine(
        similarity_top_k=5,
        filters=MetadataFilters(
            filters=[ExactMatchFilter(key="tenant_id", value=tenant_id)]
        ),
        node_postprocessors=[SimilarityPostprocessor(similarity_cutoff=0.5)]
    )

@router.post("/query", response_model=QueryResponse)
async def query(req: QueryRequest):
    """同步问答"""
    engine = get_tenant_engine(req.tenant_id)
    resp = engine.query(req.question)

    return QueryResponse(
        answer=str(resp),
        sources=[
            {"node_id": n.node_id, "score": n.score, "text": n.text[:200]}
            for n in resp.source_nodes
        ],
        token_usage=resp.metadata.get("token_usage", 0)
    )

@router.post("/query/stream")
async def query_stream(req: QueryRequest):
    """流式问答（SSE）"""
    engine = get_tenant_engine(req.tenant_id)
    streaming_engine = index.as_query_engine(
        streaming=True,
        similarity_top_k=req.top_k,
        filters=MetadataFilters(
            filters=[ExactMatchFilter(key="tenant_id", value=req.tenant_id)]
        )
    )

    async def event_generator():
        response = streaming_engine.query(req.question)
        async for token in response.async_response_gen():
            yield f"data: {json.dumps({'token': token})}\n\n"
        # 发送结束标记
        yield f"data: {json.dumps({'done': True, 'sources': [n.node_id for n in response.source_nodes]})}\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
    )

@router.delete("/documents/{doc_id}")
async def delete_document(doc_id: str):
    """删除文档及对应索引"""
    index.delete_ref_doc(doc_id, delete_from_docstore=True)
    return {"status": "deleted", "doc_id": doc_id}

面试可能问

FastAPI 作为 RAG 服务的 Web 框架有哪些优势？对比 Flask 和 Django？

FastAPI 优势：1) 原生异步支持（async def），流式响应天然高效；2) Pydantic 模型自动生成 OpenAPI 文档，Java 侧可对接生成客户端代码；3) BackgroundTasks 适合异步文档处理。对比 Flask：同步框架，流式响应需要通过 Response 类手动实现。对比 Django：太重，RAG 服务不需要 ORM 和 Admin。CorpAssist 选择 FastAPI 是因为双栈架构中 Python 侧是轻量服务，不需要 Django 的重量级框架。

流式响应中如何传递 source_nodes 给客户端？

有两种常见策略：1) 在 SSE 流的最后发送一个特殊消息（如 {"done": true, "sources": [...]}），客户端解析最后一个消息获取来源。2) 在 SSE 流的第一个消息中发送 sources 元数据，之后逐 token 输出。第一种更常用，因为检索完成后才能确定 source_nodes。CorpAssist 使用策略 1：所有 token 输出完毕后发送 done 消息附带 sources。

文档上传后索引立即可用，但偶发"找不到刚上传内容"的问题，原因和解决方案？

原因：Async indexing 的时序问题——文档上传后立即查询，但后台任务还未完成索引写入（Index not yet ready）。解决方案：1) 上传接口返回 doc_id，查询时携带 doc_id 轮询索引状态；2) 实现索引写入确认机制——Milvus 写入后等待 flush() 完成再返回；3) 上传采用同步模式（第一次上传时阻塞等待索引完成）而不是后台任务；4) 使用 Milvus 的一致性级别（consistency_level="strong"），但会增加写入延迟。CorpAssist 采用方案 4，接受 3-5s 的写入延迟换取查询即时一致。

判断练习

场景：服务启动后第一次查询很慢（~10s），后续查询正常（~1s），怎么优化？

第一次查询慢是因为 Milvus 加载索引数据到内存（warm-up）。优化：1) 在 FastAPI lifespan 启动事件中预加载索引：vector_store.collection.load()；2) 配置 Milvus 的预加载参数 collection.load(replica_number=1)；3) 应用启动后发送一个预热查询空转。CorpAssist 方案：lifespan 中 load() 集合 + 每 5 分钟 health check 自动预热。

场景：上传了 100 个文档后，查询结果突然变差（Miss），可能的原因？

可能原因：1) IngestionPipeline 处理这些文档时 chunk_size 不合理（太大导致语义混杂，或太小导致切碎上下文）；2) 新文档的 metadata 缺少 tenant_id filter 字段，但查询时加了 filter 导致无法检索到；3) 文档内容与之前差异大（如纯英文文档混入中文索引）。排查：检查新文档的 Node.metadata 是否完整；对比新文档与旧文档的检索得分分布。修复：调整 chunk_size（从 512 调到 256），检查 metadata 注入逻辑。

CLIP（Contrastive Language-Image Pre-training）是 OpenAI 开源的图文多模态模型，包含 text encoder 和 image encoder，将文本和图片映射到同一向量空间。LlamaIndex 通过 MultiModalVectorStoreIndex 支持图文混合索引。

思考10秒：CLIP 模型什么时候会失效？

1) 模糊或低分辨率图片（<100x100）——CLIP 的 Vision Transformer 输入 224x224，小图 resize 后丢失细节；2) 纯文字截图——CLIP 不理解文字内容，需要 OCR 辅助；3) 专业领域术语（医学影像、工程图纸）——CLIP 训练数据以自然图片为主，领域特定图片效果好。CorpAssist 实践中，UI 截图（按钮、菜单、对话框）CLIP 表现好，但文档扫描件需要 OCR。

OCR + 向量检索

对于包含文字的图片（截图、扫描件），纯 CLIP 检索效果差。需要 OCR（Optical Character Recognition）先提取文字，再做文本嵌入和向量检索。LlamaIndex 通过 ImageDocument 的 text 字段存储 OCR 结果。

python
# pip install paddleocr 或 easyocr

from llama_index.core.schema import ImageDocument
from paddleocr import PaddleOCR
import cv2

# 1. OCR 提取图片文字
ocr = PaddleOCR(use_angle_cls=True, lang="ch", use_gpu=False)

def ocr_image(image_path: str) -> str:
    """OCR 提取图片文字"""
    result = ocr.ocr(image_path, cls=True)
    text_lines = []
    for line in result[0] if result[0] else []:
        text = line[1][0]  # (text, confidence)
        if line[1][1] > 0.8:  # 置信度 > 0.8
            text_lines.append(text)
    return "\n".join(text_lines)

# 2. 构造包含 OCR 文本的 ImageDocument
image_path = "corp_docs/screenshots/config_page.png"
ocr_text = ocr_image(image_path)
print(f"OCR 提取文字: {ocr_text[:200]}")

img_doc = ImageDocument(
    image_path=image_path,
    text=ocr_text,  # OCR 结果存为 text
    metadata={
        "source": "screenshot",
        "department": "it",
        "has_text": True
    }
)

# 3. 双路检索（图片+文字）
# 方式 A：OCR 文本作为 ImageDocument.text，索引时同时做 CLIP 和文本嵌入
# 方式 B：分别索引图片（CLIP 嵌入）和 OCR 文本（文本嵌入），结果合并

class OCRImageProcessor:
    def process_image(self, image_path: str) -> ImageDocument:
        ocr_text = ocr_image(image_path)
        return ImageDocument(
            image_path=image_path,
            text=ocr_text,
            metadata={"ocr_text": ocr_text[:500]}

面试可能问

CLIP 模型的向量空间中，文本和图片的嵌入分布是否一致？如何保证检索质量？

CLIP 通过 contrastive learning 训练，使配对的图文向量相似度高，非配对相似度低。但训练数据以自然图片为主，抽象图示（流程图、架构图）的图文对齐质量不如自然图片。保证检索质量的措施：1) 对截图类图片先用 OCR 提取文字存入 ImageDocument.text；2) 建立索引时对图片和 OCR 文本分别嵌入；3) 查询时使用 hybrid 检索（向量 + 关键词）。CorpAssist 测试显示：纯 CLIP 检索 UI 截图召回率 72%，OCR + CLIP 双路检索提升到 91%。

多模态索引的存储成本？相比纯文本索引增加了多少？

主要增加在向量存储：每张图片的 CLIP embedding 通常 512 维（float32 约 2KB），文本 embedding 1536 维（约 6KB）。100 万张图片的多模态索引需要约 2GB 向量存储（不包含原图）。此外还需要存储 OCR 文本（每张图平均 500 字符约 1KB）。总存储成本大约是纯文本索引的 3-5 倍。但 Milvus 等分布式向量库可以水平扩展，10 亿级向量成本在可控范围内。

CorpAssist 中用户上传的图片隐私如何保护？

多层保护：1) 图片仅存储在内部网络存储（MinIO），不对外暴露 URL；2) 每个图片的 metadata 中注入 user_id 和 tenant_id，检索时强制 filter 过滤；3) OCR 处理在本地 GPU 节点完成，图片不发送到外部 API；4) 图片保留策略：附件在 30 天后自动删除，只有 OCR 文本和向量保留；5) 用户可主动删除上传的附件，同时删除对应的向量和 OCR 文本。

判断练习

场景：用户上传了一张包含中英文混排的软件界面截图，OCR 只提取了中文部分（英文识别为乱码），怎么处理？

原因：PaddleOCR 默认中文模型对英文识别效果差。修复：1) 改用 PaddleOCR 的 multilingual 模型（lang='ch' 已包含英文，但置信度较低）；2) 降低置信度阈值从 0.8 到 0.6，接受更多 OCR 结果；3) 补充使用 EasyOCR 的英文模型做二次识别；4) 将 OCR 文本和原始截图分别嵌入（CLIP 对英文界面元素的视觉特征仍然有效）。最优方案：中英文分别用不同 OCR 引擎识别后合并文本。

场景：MultiModalVectorStoreIndex 检索时图片结果排序靠后（排在文本后面），怎么调整？

默认情况下 LlamaIndex 将文本和图片结果按统一 score 排序。如果图片结果偏后，可能是因为：1) CLIP embedding 维度（512）比文本 embedding 维度（1536）低，导致相似度计算分布不同；2) 图片的视觉特征与查询的文本描述存在模态差异。调整方法：1) 检索时分别查询图片和文本，然后加权合并排序（图片得分 ×1.2）；2) 对图片检索结果应用独立的 reranker；3) 在 query 中明确"找截图"语义（如 "截图中的配置界面" 而非 "配置界面"）。

增量索引策略

增量索引指只对新增、修改、删除的文档重新索引，不触及其他文档。LlamaIndex 和 Milvus 都原生支持增量操作。

    doc = Document(text=text, metadata=metadata)
    # 方法 A：直接插到已有索引

    # 方法 B：通过 IngestionPipeline（推荐，支持转换）

# 2. 增量修改文档（先删除后插入）
def update_document(doc_id: str, new_text: str, new_metadata: dict):
    """增量修改：删除旧版本，插入新版本"""
    index.delete_ref_doc(doc_id, delete_from_docstore=True)
    new_doc = Document(text=new_text, metadata=new_metadata)
    index.insert(new_doc)
    print(f"文档 {doc_id} 已更新")

# 3. 批量增量更新
def batch_upsert(documents: list[Document]):
    """批量新增或更新"""
    for doc in documents:
        if doc.doc_id and doc.metadata.get("_is_update"):
            try:
                index.delete_ref_doc(doc.doc_id)
            except Exception:
                pass  # 新文档无旧记录
    index.insert_nodes([doc for doc in documents])  # 批量插入

        """切换活跃版本（蓝绿切换）"""
        new_collection = f"corpassist_v{version_id}"
        # 移除当前别名
        if utility.has_collection(self.ALIAS):
            utility.drop_alias(self.ALIAS)
        # 别名指向新版本
        utility.create_alias(new_collection, self.ALIAS)
        print(f"活跃索引切换到: {new_collection}")

    def list_versions(self):
        """列出所有版本"""
        collections = utility.list_collections()
        return [c for c in collections if c.startswith("corpassist_v")]

    def rollback(self, version_id: str):
        """回滚到指定版本"""
        self.switch_active(version_id)
        print(f"已回滚到 v{version_id}")

面试可能问

全量重建和增量更新分别应对什么场景？为什么两者需要共存？

全量重建：1) 嵌入模型升级（如 text-embedding-3-xxx → gte-Qwen2），新模型生成的向量与旧的不兼容，必须全部重新嵌入；2) 索引结构变更（如 chunk_size 从 512 改为 1024）；3) 数据迁移（开发→测试→生产）。增量更新：1) 日常文档新增和修改；2) 少量文档内容修正。两者共存原因：全量重建保证架构一致性，增量更新保证日常效率。CorpAssist 实践：每天全量重建一次，日常变更增量更新。

Milvus 集合别名切换时，正在进行的查询会受影响吗？

不会受影响。Milvus 通过引用计数管理 Collection 的生命周期：切换别名只是元数据操作（修改 Alias → Collection 映射），正在执行的查询持有底层 Collection 的引用，会继续完成。新查询通过别名访问新的 Collection。所以别名切换是"零停机"的。但如果切换时旧 Collection 有大量写入请求（如增量索引），需要等写入完成后才能释放旧 Collection 的引用。

如何设计 RAG 服务的自愈机制？当监控告警触发时自动做什么？

多层自愈：1) 检索延迟 P99 > 3s → 自动切换为快速检索模式（降级：关闭 Reranker、减小 top_k）；2) Milvus 连接失败 → 自动切换到 Qdrant 副本（双写双读）；3) 嵌入 API 错误 > 5% → 自动切换到本地嵌入模型（BGE-M3）；4) 新版本索引出问题 → 自动回滚到上一版本（保留最近 3 个版本异或，保留 7 个版本）。每层自愈都在 30 秒内完成，超过则告警人工介入。

判断练习

场景：全量重建时仍有查询流量，怎么保证两个操作不互相影响？

使用蓝绿部署模式。全量重建写入新的 Collection（蓝色），当前查询路由到旧的 Collection（绿色）。重建完成后通过切换别名将查询路由切换到新 Collection。关键点：1) 蓝色 Collection 要使用不同的名称（如 corpassist_v20260601）；2) 别名 corpassist_active 始终指向当前活跃的 Collection；3) 重建期间不要切换别名。这个过程对查询服务完全透明。

场景：增量删除后用户仍然能检索到已删除的文档，可能的原因？

可能原因：1) delete_ref_doc 的 delete_from_docstore=False（默认），只删除了索引结构中的引用，但向量库中的向量未被删除；2) Milvus 的删除操作默认是软删除，需要 cleanup 后才能真正生效；3) 请求的索引实例是旧的（连接到了旧 Collection）。排查：确认 delete_from_docstore=True；调用 collection.compact() 清理 Milvus 的已删除数据；检查当前查询连接的 Collection 是否已切换。

缓存策略

缓存策略分两种：精确缓存（相同查询直接返回缓存答案）和语义缓存（语义相似的查询命中缓存）。LlamaIndex 不内置缓存，但可以通过装饰器或中间件实现。

import hashlib, json, time
import redis as redis_lib
from llama_index.embeddings.openai import OpenAIEmbedding

# Redis 连接
redis_client = redis_lib.Redis(host="redis", port=6379, decode_responses=True)
embed_model = OpenAIEmbedding(model="text-embedding-3-small", dimensions=512)

# ---- 1. 精确缓存 ----
def exact_cache_key(query: str, tenant_id: str) -> str:
    """精确缓存 Key: md5(query + tenant_id)"""
    raw = f"{query}:{tenant_id}"
    return f"cache:exact:{hashlib.md5(raw.encode()).hexdigest()}"

def get_exact_cache(query: str, tenant_id: str) -> str | None:
    key = exact_cache_key(query, tenant_id)
    return redis_client.get(key)

def set_exact_cache(query: str, tenant_id: str, answer: str, ttl=3600):
    key = exact_cache_key(query, tenant_id)
    redis_client.setex(key, ttl, answer)

# ---- 2. 语义缓存 ----
class SemanticCache:
    def __init__(self, similarity_threshold=0.95):
        self.threshold = similarity_threshold
        self.embed_model = OpenAIEmbedding(model="text-embedding-3-small", dimensions=512)

    def find_similar(self, query: str, tenant_id: str) -> str | None:
        """查找语义相似的缓存条目"""
        query_vec = self.embed_model.get_text_embedding(query)
        # 从 Redis 获取该租户的所有缓存向量
        cache_entries = redis_client.hgetall(f"cache:semantic:{tenant_id}")
        best_score, best_answer = 0, None
        for cached_query, cached_data in cache_entries.items():
            data = json.loads(cached_data)
            cached_vec = data["vector"]
            # 余弦相似度
            score = sum(a*b for a,b in zip(query_vec, cached_vec)) / (
                sum(a*a for a in query_vec)**0.5 * sum(b*b for b in cached_vec)**0.5
            )
            if score > self.threshold and score > best_score:
                best_score, best_answer = score, data["answer"]
        return best_answer

    def store(self, query: str, tenant_id: str, answer: str, ttl=7200):
        query_vec = self.embed_model.get_text_embedding(query)
        data = json.dumps({"vector": query_vec, "answer": answer, "timestamp": time.time()})
        redis_client.hset(f"cache:semantic:{tenant_id}", query, data)
        redis_client.expire(f"cache:semantic:{tenant_id}", ttl)

# ---- 3. 组合缓存中间件 ----
class CacheMiddleware:
    def __init__(self):
        self.exact_ttl = 3600       # 精确缓存 1 小时
        self.semantic_ttl = 7200    # 语义缓存 2 小时
        self.semantic_cache = SemanticCache(threshold=0.95)

    async def get_or_compute(self, query: str, tenant_id: str, compute_fn):
        # 1. 精确缓存
        cached = get_exact_cache(query, tenant_id)
        if cached:
            return json.loads(cached)

        # 2. 语义缓存
        semantic = self.semantic_cache.find_similar(query, tenant_id)
        if semantic:
            return json.loads(semantic)

        # 3. 计算新答案
        result = await compute_fn()

        # 4. 写入缓存
        result_json = json.dumps(result)
        set_exact_cache(query, tenant_id, result_json, self.exact_ttl)
        self.semantic_cache.store(query, tenant_id, result_json, self.semantic_ttl)
        return result

from llama_index.llms.openai import OpenAI
from llama_index.llms.deepseek import DeepSeek

class ModelRouter:
    """根据查询复杂度路由到不同 LLM"""
    def __init__(self):
        self.cheap_llm = DeepSeek(model="deepseek-chat", temperature=0.1)
        self.medium_llm = OpenAI(model="gpt-4.1-mini", temperature=0.1)
        self.expensive_llm = OpenAI(model="gpt-4.1", temperature=0.1)

    def classify_query(self, query: str) -> str:
        """用简单规则 + 小模型分类查询复杂度"""
        # 简单规则快速过滤
        simple_keywords = ["什么", "多少", "谁", "哪里", "何时", "是不是", "有没有"]
        complex_keywords = ["对比", "比较", "分析", "总结", "为什么", "如何解决", "区别"]

        if any(kw in query for kw in complex_keywords):
            return "complex"
        if any(kw in query for kw in simple_keywords):
            return "simple"
        # 不确定的用小模型分类
        response = self.cheap_llm.complete(
            f"判断以下查询的复杂度，只返回 simple/medium/complex: {query}"
        )
        return str(response).strip().lower()

    def get_llm(self, query: str):
        complexity = self.classify_query(query)
        if complexity == "simple":
            return self.cheap_llm
        elif complexity == "medium":
            return self.medium_llm
        else:
            return self.expensive_llm

预计算热点问答

预计算热点问答是通过分析历史查询日志，识别高频重复问题，提前计算答案并存入缓存。这样热点问题直接用预计算答案响应，零检索零生成。

python
# 1. 分析历史日志，找出热点查询
from collections import Counter
import json

def analyze_query_logs(log_path: str, top_n: int = 100):
    """分析查询日志，找出高频问题"""
    queries = []
    with open(log_path) as f:
        for line in f:
            record = json.loads(line)
            queries.append(record["query"])

    counter = Counter(queries)
    hotspots = counter.most_common(top_n)
    print(f"Top 5 热点查询:")
    for q, count in hotspots[:5]:
        print(f"  {count}次: {q}")
    return [q for q, _ in hotspots]

Token 成本归因

Token 成本归因将每次查询的 Token 消耗分解到检索、合成、路由各个环节，并归属到租户和部门。这是优化 ROI 的基础数据。

python
from dataclasses import dataclass
from datetime import datetime
import json

@dataclass
class CostRecord:
    query_id: str
    tenant_id: str
    query: str
    retrieval_tokens: int
    llm_model: str
    prompt_tokens: int
    completion_tokens: int
    latency_ms: int
    from_cache: bool
    created_at: str

class CostTracker:
    def __init__(self):
        self.records = []

    def record_query(self, **kwargs):
        record = CostRecord(
            query_id=kwargs['query_id'],
            tenant_id=kwargs['tenant_id'],
            query=kwargs['query'][:100],
            retrieval_tokens=kwargs.get('retrieval_tokens', 0),
            llm_model=kwargs.get('llm_model', 'none'),
            prompt_tokens=kwargs.get('prompt_tokens', 0),
            completion_tokens=kwargs.get('completion_tokens', 0),
            latency_ms=kwargs.get('latency_ms', 0),
            from_cache=kwargs.get('from_cache', False),
            created_at=datetime.utcnow().isoformat()
        )
        self.records.append(record)

    def calculate_cost(self, record: CostRecord) -> float:
        """计算单次查询的 Token 费用"""
        if record.from_cache:
            return 0.0
        rates = {
            "gpt-4.1": {"prompt": 15, "completion": 60},
            "gpt-4.1-mini": {"prompt": 1.5, "completion": 6},
            "deepseek-chat": {"prompt": 0.5, "completion": 2},
            "text-embedding-3-small": {"prompt": 0.002, "completion": 0},
        }
        model_rates = rates.get(record.llm_model, rates["gpt-4.1"])
        # 输出是百万 Token 价格，计算到分的费用
        cost = (record.prompt_tokens * model_rates["prompt"] +
                record.completion_tokens * model_rates["completion"]) / 1_000_000
        return round(cost, 6)

    def report_by_tenant(self) -> str:
        """按租户生成费用报告"""
        tenant_costs = {}
        for record in self.records:
            tenant = record.tenant_id
            if tenant not in tenant_costs:
                tenant_costs[tenant] = {"count": 0, "cost": 0, "cache_hit": 0}
            tenant_costs[tenant]["count"] += 1
            tenant_costs[tenant]["cost"] += self.calculate_cost(record)
            if record.from_cache:
                tenant_costs[tenant]["cache_hit"] += 1

        report = "租户费用报告:\n"
        for tenant, data in tenant_costs.items():
            cache_rate = data["cache_hit"] / data["count"] * 100
            report += f"  {tenant}: {data['count']}次, ${data['cost']:.2f}, 缓存率{cache_rate:.0f}%\n"
        return report

面试可能问

精确缓存和语义缓存的适用场景分别是什么？语义缓存的阈值怎么确定？

精确缓存：用户完全相同的查询（"年假几天"和"年假几天"）。适合系统自动生成的问题、常见 FAQ。语义缓存：表述不同但意图相同的查询（"年假几天"和"我有多少天年假"）。阈值通过实验确定：在验证集上测试不同阈值（0.90/0.95/0.99）的精度和召回率。阈值太高（0.99）命中少，太低（0.90）可能误命中不同意图的问题。CorpAssist 经验值 0.95 是平衡点。

小模型路由分类错误（把复杂查询路由到简单模型）导致回答质量差，怎么兜底？

双层兜底：1) 快速验证：简单模型生成答案后，用另一个模型快速评估答案质量（如检查答案长度、是否包含"不确定"等信号），质量低则重新路由到高级模型；2) 用户反馈：设置用户对答案的"有用/没用"评分，评分低的查询自动记录并重新路由。用户也可以手动触发"深入回答"按钮，强制使用高级模型重新生成。CorpAssist 策略：先 DeepSeek 快速回答，用户可一键切换到 GPT-4.1 深究。

预计算热点的答案会过期（HR 政策调整后旧答案错误），怎么处理？

答案过期是预计算的核心挑战。处理方案：1) TTL 管理：热点答案缓存 TTL 设为 24 小时，每天凌晨重新预计算；2) 文档变更触发：HR 文档更新时，清除关联的热点缓存（基于 metadata 中记录的 source 文档 ID）；3) 版本标记：预计算答案打上文档版本号，查询时如果发现文档版本高于缓存版本，重新生成；4) 渐进式更新：不一次全部重新计算，而是每小时扫描 10% 的热点题目，7 小时内完成全量刷新。

判断练习

场景：语义缓存命中了一个相似但不完全相同的问题，答案有些偏差，怎么在延迟和准确性之间平衡？

方案 1：提高阈值（从 0.95 到 0.98），减少误命中但降低命中率。方案 2：缓存答案时同时缓存该答案的 source_nodes，返回缓存答案时附带来源引用让用户自判。方案 3：两阶段策略——先用缓存快速返回答案（200ms），同时后台触发完整 LLM 调用，如果新答案与缓存答案不一致（通过 LLM 判断），推送更新页面。CorpAssist 采用方案 3，体验最佳。

场景：小模型路由中，查询"帮我写一篇关于公司年假政策的 500 字报告"被误判为"simple"，怎么优化？

误判原因是规则中没有识别"写报告"类复杂请求。优化：1) 在分类规则中加入"写|创建|生成...报告|文章|邮件"等动词+名词组合的复杂查询模式；2) 查询长度也是一个信号——超过 30 个字符的查询更可能是复杂查询；3) 使用更快速的轻量分类器（如朴素贝叶斯或 small LM 分类器）代替规则。修改后"写...报告"类查询被正确路由到 GPT-4.1，分类准确率从 82% 提升到 94%。

如果一个项目同时用了 LangChain 和 LlamaIndex，职责怎么划分？会不会增加复杂度？

职责划分：LangChain 负责 ChatModel 对话管理/Callbacks/Agent 生态；LlamaIndex 负责 Document Ingestion/Index/Retrieval/Routing。接口桥接：LangChain 通过 QueryEngineTool 调用 LlamaIndex 的检索能力。是否增加复杂度——会增加约 15% 的架构复杂度（两个框架的学习曲线、两套配置和监控），但收益大于成本：在 CorpAssist 中检索 MRR 提升 30%、路由能力从无到有、评估自动化。建议作为过渡架构，长期目标是用 LlamaIndex Agent 替代 LangChain Agent，最终统一为单框架。

从 LangChain 迁移到 LlamaIndex 时，历史已有的向量数据怎么处理？

三种方案：1) 全面重新嵌入（推荐）：用 LlamaIndex 的 IngestionPipeline 重新处理所有文档，写入新的 Milvus Collection。优点是向量空间一致，检索效果好；缺点是耗时（100 万文档约 2 小时）。2) 兼容模式：保持 LangChain 的向量库不变，LlamaIndex 只通过 MilvusVectorStore 连接不重新嵌入。缺点是 metadata 格式可能不一致。3) 混合模式：旧文档保留 LangChain 索引，新文档用 LlamaIndex 索引，路由引擎根据时间路由。CorpAssist 用方案 1，在周末窗口完成全量迁移，切换别名后旧索引保留 7 天回滚。

你做过的 RAG 系统中，最大的技术挑战是什么？怎么解决的？

最大挑战是多租户场景下的检索准确性和数据隔离。初期用共享索引+metadata filter，但 filter 漏配导致跨租户数据泄露。解决方案：1) 检索前强制 tenant_id filter（代码层校验，不传租户 ID 的请求拒绝）；2) 检索后加 PermissionFilter NodePostprocessor 二次确认；3) 每周自动检查隔离有效性（模拟跨租户查询，验证结果为空）。结果：零数据泄露事件，安全审计通过。

判断练习

面试中如果被追问"LlamaIndex 有什么缺点"，怎么回答才显得客观？

客观回答：1) 学习曲线——比 LangChain 陡，新团队成员需要 1-2 周上手 Node/Index/QueryEngine 概念；2) Agent 生态不如 LangChain——虽然基础 Agent 够用，但复杂 Agent 模式（Plan-and-Execute、Multi-agent Supervisor）需要自实现；3) 社区规模——LangChain GitHub 星星 10 万+，LlamaIndex 约 3 万，第三方工具集成和社区问题解答速度有差距；4) 文档质量——API 升级频繁（0.10→0.11 有 breaking change），文档偶有滞后。总体而言，检索场景 LlamaIndex 更强，对话/Agent 场景 LangChain 更成熟。

如果公司要求所有 RAG 项目标准化为一个框架，你推荐 LangChain 还是 LlamaIndex？

标准化推荐 LlamaIndex。推理：如果项目以 RAG/检索为核心（90% 以上的 LLM 项目都是 RAG 类），LlamaIndex 的数据模型、索引类型、路由引擎、评估工具都是更专业的。纯对话类项目（非 RAG）可以用 LangChain，但占比小。标准化的策略是：以 LlamaIndex 为主框架，对于极少数需要复杂 Agent 编排的项目由 LangChain 补充。用一个框架的好处是降低重复学习成本、统一监控和评估体系。CorpAssist 的经验验证了以 LlamaIndex 为核心的可行性。
