微软 GraphRAG 部署

CorpAssist · 跨文档推理员工问「如果我从工程部轮岗到产品部半年，年假和绩效怎么算？」——答案需要关联 HR 政策（年假折算）、部门轮岗条例、绩效评估办法三份文档，Naive RAG 通常只检到其中 1~2 份。

微软 GraphRAG 是学术的 GraphRAG 概念的一个工业化实现。其核心创新在于 自动化的实体-关系-社区抽取：将文档分块后，使用 LLM 抽取实体（Entity）、关系（Relation）、声明（Claim），然后通过层次化社区检测（Leiden 算法）组织社区摘要。

bash
# GraphRAG 部署与索引
cd demos/advanced-01-graphrag-hands-on-lab

# Step 1: 安装 graphrag
pip install graphrag

# Step 2: 初始化工作区
mkdir -p graphrag_input graphrag_output
# 将 CorpAssist HR 文档放入 graphrag_input/
cp ../../datasets/hr_policies/*.md graphrag_input/

# Step 3: 初始化配置
python -m graphrag.index --init --root .
# 生成 settings.yaml，需配置 LLM 和 Embedding 的 API key

# Step 4: 运行索引（耗时约 5~10 分钟/100 文档）
python -m graphrag.index --root . --resume graphrag_output

# 索引完成后输出结构:
# graphrag_output/artifacts/
#   - entities.parquet  (实体表)
#   - relationships.parquet  (关系表)
#   - communities.parquet  (社区表)
#   - text_units.parquet  (文本单元)

配置文件关键参数（settings.yaml）：

参数建议值影响

实体关系抽取流程

GraphRAG 索引的核心是 LLM 驱动的实体关系抽取。每个文本块被送入 LLM 并结构化输出 JSON 格式的实体与关系。抽取 prompt 示例及输出如下：

GraphRAG 索引管线
flowchart LR
  DOC[Documents] --> CHUNK[Chunk & Clean]
  CHUNK --> ENT[Entity Extraction
LLM per chunk]
  ENT --> MERGE[Merge Entities
resolve duplicates]
  MERGE --> LEIDEN[Leiden Community Detection]
  LEIDEN --> SUM[Community Summarization]
  SUM --> STORE[Knowledge Graph
+ Vector Index]
  STORE --> QLOCAL[Local Search]
  STORE --> QGLOBAL[Global Search]

Naive RAG 的局限
  • 每个 chunk 独立检索，无跨 chunk 关系建模
  • 「PIP 期间年假怎么算」此类跨主题查询需手动拼接
  • 每个 chunk 内容碎片化，LLM 难以综合推理
GraphRAG 的优势
  • 实体-关系图显式建模跨文档关联
  • 社区摘要在检索前已综合多个文档要点
  • Local Search 返回实体+关联关系+声明

对比 Naive RAG 索引：相同 500 篇文档，仅需 $0.28（chunk + embed + store）。GraphRAG 成本约 44 倍。查询阶段：GraphRAG Local Search 每次约 1850 输入 token + 350 输出 token（约 $0.005/次），Naive RAG 约 1200+200 token（$0.002/次）。

成本节省策略：若使用 gpt-4o-mini 替代 gpt-4o 做实体抽取，索引成本可从 $12.45 降至 $3.20，但实体准确率从 89% 降至 77%。CorpAssist 的折中方案为：首次使用 gpt-4o 全量建索引，增量更新使用 gpt-4o-mini 并做实体合并去重。若关闭 claim_extraction（声明抽取），索引成本再降 $1.80。查询阶段也可优化：对于简单单文档查询直接走 Naive RAG 避免 GraphRAG 的推理开销。

bash
# GraphRAG 查询（Local Search — 面向具体问题）
python -m graphrag.query \
  --root . \
  --method local \
  "我从工程部轮岗到产品部半年，年假怎么算？"

# 输出摘要示例:
# 根据公司政策，跨部门轮岗期间年假按原部门标准计算，
# 轮岗期超过3个月需在目标部门重新注册假期额度。
# 具体条款参考: 员工手册第四章第2条、轮岗管理办法第8条。

# GraphRAG 查询（Global Search — 面向全局总结）
python -m graphrag.query \
  --root . \
  --method global \
  "公司有哪些与休假相关的政策？"

与 Naive RAG 效果对比

在 CorpAssist 跨文档 QA 测试集（100 条需要 2+ 文档才能回答的问题）上对比：

指标Naive RAGGraphRAG LocalGraphRAG Global
准确率（人工评估）52.3%84.7%78.2%
faithfulness（RAGAS）0.870.910.88
答案完整性（1~5）2.44.13.8
P95 查询延迟1.3s2.1s3.5s
单次查询成本$0.002$0.005$0.008
索引成本（500 文档）$0.28$12.45

结论：对于单文档问答（占 CorpAssist 总查询的 65%），Naive RAG 已足够，准确率 87% vs GraphRAG 91% 差异不大。对于跨文档推理（占 35%），GraphRAG Local Search 准确率 84.7% 显著优于 Naive RAG 的 52.3%。建议 CorpAssist 采用 Hybrid 策略：先用 RAGAS 评测自动判断问题是否需要跨文档推理，再选择性路由到 GraphRAG。

Hybrid 路由的具体实现：在 RAGAS 评测管线中增加一个「cross_doc_score」分类器，基于问题中出现的实体数量（如问题中的命名实体 ≥ 3 个则大概率需要跨文档推理）。CorpAssist 日志分析显示：问题中含 1~2 个实体时仅 18% 需要跨文档，含 3~4 个实体时 62% 需要跨文档，含 5+ 实体时 89% 需要跨文档。基于该规则可实现一个零成本的路由器，无需额外 LLM 调用。

FAQ：什么时候不应该用 GraphRAG？

1) 知识库 < 100 篇文档或文档间无交叉引用关系 — 图结构信息量低；2) 查询主要是单文档事实问答（「今年的年假天数」）— Naive RAG 成本更低且延迟更短；3) 对索引成本敏感且更新频率高（每周更新）— 每次索引重新运行成本 $10+；4) 对 P95 延迟要求 < 1.5s — GraphRAG Local Search P95=2.1s 超出。

自检：GraphRAG 索引中最耗时的环节是哪个？如何优化？

最耗时的是实体抽取环节（占总时间的 ~67%），因为每个 chunk 需调用 LLM 提取实体+关系+声明。优化方向：1) 使用 gpt-4o-mini 替代 gpt-4o（成本降 6 倍，但实体准确率从 89% 降至 77%）；2) 增大 chunk_size 减少调用次数；3) 使用模型并行（多个 chunk 同时抽取）。

本章结论

  • GraphRAG 部署：微软 GraphRAG 索引管线分四步：chunk → entity extraction → community detection → summary，500 文档约 18 分钟完成
  • 实体抽取成本：索引阶段 Token 消耗约 858K（输入+输出），成本 $12.45（Naive RAG 仅 $0.28）
  • 查询效果：跨文档推理准确率 84.7% vs Naive RAG 52.3%，单文档查询差距不大
  • Hybrid 策略：CorpAssist 应选择性路由跨文档问题到 GraphRAG，其余走 Naive RAG
  • 适用场景：文档间关联密集、跨文档推理频繁的知识库才值得 GraphRAG 的索引成本

面试可能问

GraphRAG 的社区检测是如何实现的？为什么用了 Leiden 算法而非 Louvain？

Leiden 算法在 Louvain 基础上增加了 refine 阶段，保证社区连通性，避免 Louvain 可能产生的断连社区。GraphRAG 层次化社区检测：先构建实体图（节点=实体，边=关系强度），然后逐层应用 Leiden 算法形成多尺度社区，上层社区包含下层社区，形成金字塔结构。层次化使得 Local Search 和 Global Search 可在不同粒度检索。

List 出 3 个你可能在 CorpAssist 考虑使用 GraphRAG 的信号？

1) 用户频繁问「A 政策对 B 流程有什么影响」类问题（跨文档推理占查询 > 30%）；2) 知识库超过 500 篇且文档间有密集引用关系（如「详见第四章」）；3) Bad case 分析显示 Naive RAG 因 chunk 碎片化导致答案不完整（context_recall < 0.70）。

如何降低 GraphRAG 索引成本？

1) 实体抽取使用 gpt-4o-mini 替代 gpt-4o（成本降 6 倍，准确率从 89%→77%）；2) 增加 chunk_size 从 1200 到 2400 减少调用次数（但可能漏掉细粒度实体）；3) 关闭 claim_extraction（占 15% 成本）；4) 对更新文档使用增量索引（目前 graphrag 不支持，需自研 diff 策略）。

场景 1：知识库 200 篇文档，月更新 2 次，GraphRAG 索引成本月均 $50，ROI 怎么看？

先评估跨文档查询占比：如果 < 20%，建议维持 Naive RAG。如果 > 50%，$50/月的成本在 CorpAssist 场景中可接受（对比工程师人工处理跨文档查询的人力成本）。建议先对 1 个月的用户 query 做日志分析，统计需要跨文档推理的 query 比例再决策。

场景 2：索引运行中遇到 rate limit（429），如何配置重试策略？

在 settings.yaml 中设置 max_retries: 10 和 retry_delay: 30（秒）。若仍频繁触发，降低 parallelism 参数（默认 25→10）。GraphRAG 内部使用 tenacity 库进行指数退避重试，可在日志中观察 retry 次数。
