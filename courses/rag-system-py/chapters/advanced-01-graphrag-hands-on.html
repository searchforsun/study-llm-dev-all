为什么要学本章

传统 RAG 将文档切分成独立片段，丢失了实体间的关联关系。当用户问「A 部门和 B 部门的人员流动」「某项目的上下游协作」这类需要多跳推理的问题时，朴素 RAG 表现很差。GraphRAG 通过从文档中提取实体和关系构建知识图谱，用社区检测聚合信息，实现全局 + 局部两种检索模式。本章使用 Microsoft GraphRAG 开源框架，在 CorpAssist 场景中落地。

学完你能
  • 使用 GraphRAG 从文档中提取实体、关系和协变量
  • 理解社区检测（Leiden 算法）如何聚合信息生成社区摘要
  • 实现全局搜索（Global Search）和局部搜索（Local Search）两种查询模式
  • 评估 GraphRAG 的适用场景和额外成本，与朴素 RAG 对比
本章先记住 3 件事
  • GraphRAG ≠ 知识图谱 + 向量：它是先构建实体关系图，再做社区检测和摘要，最后基于社区级信息回答。
  • 全局搜索 vs 局部搜索：全局回答「整体趋势」，局部回答「具体实体」，两者用不同的 prompt 模板。
  • 成本是朴素 RAG 的 5-10 倍：实体提取、社区摘要都需要 LLM 调用。适用于高价值场景：合规分析、战略报告。

知识图谱构建：实体提取与关系识别

GraphRAG 的构建流程分三步：实体提取 → 关系识别 → 社区检测。前两步依赖 LLM 从文本中抽取结构化信息，第三步用图算法自动发现社区结构。

CorpAssist · 组织架构分析 CorpAssist 需要从公司组织架构文档、项目报告和邮件提取实体（人员、部门、项目）和关系（汇报、协作、转岗）。然后回答「技术部与产品部的协作项目有哪些」「近半年管理层变动情况」这类多跳问题。

初始化 GraphRAG 索引

python 
# graphrag_init.py — GraphRAG 索引初始化与配置
import os
import yaml
from pathlib import Path

# GraphRAG 使用 settings.yaml 配置文件
# 安装: pip install graphrag

GRAPH_CONFIG = {
    "root_dir": "./data/graphrag_index",
    "input_dir": "./data/docs",
    "output_dir": "./data/graphrag_output",
    "llm": {
        "model": "gpt-4o-mini",
        "max_tokens": 4000,
        "temperature": 0.0,
        "api_key": "${GRAPHRAG_API_KEY}",
        "model_supports_json": True,
    },
    "embeddings": {
        "model": "text-embedding-3-small",
        "dimensions": 512,
    },
    "chunks": {
        "size": 1200,
        "overlap": 100,
        "group_by_columns": ["id"],
    },
    "entity_extraction": {
        "prompt_template": "graphrag/prompts/entity_extraction.txt",
        "max_gleanings": 2,
    },
    "summarize_descriptions": {
        "max_length": 500,
        "prompt_template": "graphrag/prompts/summarize.txt",
    },
    "community_reports": {
        "max_length": 2000,
        "prompt_template": "graphrag/prompts/community_report.txt",
    },
}

def initialize_graphrag_index(root_dir: str = "./data/graphrag_index"):
    """初始化 GraphRAG 索引目录"""
    root = Path(root_dir)
    (root / "input").mkdir(parents=True, exist_ok=True)
    (root / "output").mkdir(parents=True, exist_ok=True)
    (root / "prompts").mkdir(parents=True, exist_ok=True)

    # 写入 settings.yaml
    config_path = root / "settings.yaml"
    with open(config_path, "w", encoding="utf-8") as f:
        yaml.dump(GRAPH_CONFIG, f, allow_unicode=True, default_flow_style=False)

    print(f"GraphRAG 索引目录已初始化: {root}")
    print(f"配置文件: {config_path}")

    # 创建默认 prompt 模板
    prompts = {
        "entity_extraction.txt": (
            "-Goal-\n从文本中提取实体和关系。\n"
            "-Entities-\nperson, organization, project, department\n\n"
            "输出 CSV 格式: entity,type,description"
        ),
        "summarize.txt": (
            "用一段话总结以下实体的描述信息。\n{entity_name}\n{description}"
        ),
        "community_report.txt": (
            "基于以下社区信息生成报告。\n{community_info}\n"
            "包含：社区概述、关键实体、关键关系、洞察"
        ),
    }
    prompt_dir = root / "prompts"
    for name, content in prompts.items():
        (prompt_dir / name).write_text(content, encoding="utf-8")

if __name__ == "__main__":
    initialize_graphrag_index()
    print("准备完成。运行索引: python -m graphrag.index --root ./data/graphrag_index")

不推荐
  • 手动维护实体关系表，每次更新需人工修改
  • 不使用社区摘要，直接检索实体邻域
  • 实体提取用关键词规则，漏报率高
  • LLM 提取时不设 max_gleanings，遗漏关系
推荐
  • GraphRAG 自动提取实体/关系/社区
  • 社区摘要提供全局视角
  • LLM 驱动提取 + 人工校验关键实体
  • 设置 max_gleanings=2，迭代完善提取
先想 10 秒：GraphRAG 的实体提取阶段为什么要设 max_gleanings？

LLM 一次提取可能遗漏部分实体。max_gleanings 让 LLM 检查已提取结果并补充遗漏。每轮 gleaning 增加一次 LLM 调用。设 2 表示「提取→检查→补充」三轮，在质量和成本间平衡。

社区检测与全局/局部搜索

GraphRAG 使用 Leiden 算法 对实体图进行社区检测。社区是高内聚的实体子图，使得同一个社区内的实体关联紧密。GraphRAG 为每个社区生成一份自然语言摘要，用于支持全局搜索。

搜索模式输入检索策略适用问题
Local Search 具体实体或关键词 提取实体邻域 → 向量相似扩展 → LLM 生成 「张三的汇报关系」「A 项目的负责人」
Global Search 抽象问题或趋势 匹配相关社区摘要 → 聚合多社区观点 → LLM 综合 「各部门协作效率」「今年组织架构变化」

from langchain_openai import ChatOpenAI

        llm_model: str = "gpt-4o-mini",

成本评估与场景选择

GraphRAG 的核心成本在索引构建阶段。每篇文档需要多次 LLM 调用（实体提取 + 关系识别 + gleaning + 社区摘要）。对于 50 篇文档约 200 页，索引构建成本约 $10-30，而朴素 RAG 的索引成本几乎为零。

维度朴素 RAGGraphRAG
索引成本~$0.01（仅 embedding）$10-30（LLM 提取 + 摘要）
查询成本$0.01-0.05/次$0.05-0.20/次
索引时间分钟级小时级（取决于文档量）
事实问答优优（两者差不多）
多跳推理差（片段无关联）优（实体关系图）
全局总结差（需额外 prompt）优（社区摘要内置）
更新灵活性增量更新容易需重新提取实体关系
CorpAssist 选型建议
  • 日常 Q&A（"今年年假几天？"）→ 朴素 RAG，成本低响应快
  • 组织分析（"技术部和产品部之间的人员流动"）→ GraphRAG Local Search
  • 战略报告（"公司各部门的核心能力分布"）→ GraphRAG Global Search
  • 合规审计（"所有涉及数据外泄的相关方"）→ GraphRAG + 图遍历

何时 GraphRAG 明显优于朴素 RAG

python 
# compare_scenarios.py — GraphRAG vs 朴素 RAG 场景对比
from langchain_openai import ChatOpenAI
from typing import Dict, List

class ComparisonDemo:
    """GraphRAG 与朴素 RAG 对比示例"""

    def __init__(self):
        self.llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

    def naive_rag_answer(self, question: str, chunks: List[str]) -> str:
        """模拟朴素 RAG 回答"""
        context = "\n\n".join(chunks[:3])
        prompt = f"基于上下文：\n{context}\n\n回答问题：{question}"
        return str(self.llm.invoke(prompt).content)

    def graphrag_local_answer(self, question: str, kg_info: str) -> str:
        """模拟 GraphRAG local search 回答"""
        prompt = f"基于知识图谱信息：\n{kg_info}\n\n回答问题：{question}"
        return str(self.llm.invoke(prompt).content)

    def demo(self):
        questions = [
            "张三和李四是什么关系？",
            "技术部有哪些项目是跟产品部合作的？",
            "去年以来公司核心管理层发生了哪些变动？",
        ]
        for q in questions:
            print(f"\n问题: {q}")
            # 朴素 RAG — 只能看到片段，缺乏全局关联
            mock_chunks = [
                "张三在2024年1月入职技术部，担任高级工程师。",
                "技术部与产品部的协作项目包括 Alpha 和 Beta。",
                "2024年3月，原CTO王五离职，由赵六接任。",
            ]
            naive = self.naive_rag_answer(q, mock_chunks)
            print(f"[朴素RAG] {naive[:100]}...")

            # GraphRAG — 全局关联
            mock_kg = (
                "实体: 张三, type: person, 部门: 技术部\n"
                "实体: 李四, type: person, 部门: 技术部, 角色: 技术总监\n"
                "关系: 张三 --汇报给--> 李四\n"
                "实体: 项目Alpha, 参与方: 技术部+产品部\n"
                "实体: 王五, 原CTO, 2024-03 离职\n"
                "实体: 赵六, 现任CTO, 2024-04 入职"
            )
            graph = self.graphrag_local_answer(q, mock_kg)
            print(f"[GraphRAG] {graph[:100]}...")

if __name__ == "__main__":
    demo = ComparisonDemo()
    demo.demo()

常见问题（FAQ）

GraphRAG 必须使用 Microsoft 的那个开源实现吗？
不是必须的。Microsoft GraphRAG 是目前最成熟的方案，但不是唯一选择。你也可以用 Neo4j + LangChain 手动构建图谱，或者使用 LlamaIndex 的 KnowledgeGraphIndex。Microsoft 方案的优点是社区检测和摘要生成开箱即用。
GraphRAG 索引构建太慢了，能增量更新吗？
GraphRAG 目前不支持真正的增量索引。新增文档需要重新运行完整索引管线。社区中有一些 workaround：为不同时间段的文档构建独立索引，查询时合并结果。长期来看，GraphRAG 的增量索引是个活跃开发方向。
我的文档量很小（10 篇），有必要用 GraphRAG 吗？
文档量小的情况下，朴素 RAG 加上一些手写的关系映射可能就够用了。但如果实体关联密集（如组织架构、项目依赖），GraphRAG 的社区摘要仍然能提供更好的全局视角。建议 50+ 篇文档时考虑 GraphRAG。
GraphRAG 的社区摘要是怎么保证质量的？
社区摘要的质量取决于 1) 实体提取的完整性；2) 社区检测的粒度（Leiden 算法的分辨率参数）；3) 摘要 prompt 的设计。可以通过调整 resolution 参数控制社区数量（resolution 越高社区越多越细），在精度和召回间平衡。

本章结论

  • 实体提取是基础：LLM 驱动的实体/关系/协变量提取质量直接决定图谱质量。
  • 社区检测自动聚合：Leiden 算法发现实体高内聚子图，支持全局视角的问答。
  • 全局搜索 vs 局部搜索：抽象问题用 Global（社区摘要聚合），具体问题用 Local（实体邻域）。
  • 成本较高但适合高价值场景：索引成本在 $10-30 级别，适用于多跳推理和全局总结场景。
  • 不是朴素 RAG 的替代品：两者互补——日常 Q&A 用朴素 RAG，分析推理用 GraphRAG。

动手练习

用 GraphRAG 构建组织架构知识图谱并进行查询（约 40 分钟）。

操作步骤

  1. 准备 5-10 篇组织架构相关文档（组织结构、岗位职责、项目报告）。
  2. 初始化 GraphRAG 索引目录并配置 settings.yaml。
  3. 运行 python -m graphrag.index 构建索引（注意：首次运行需大量 LLM 调用）。
  4. 使用 graphrag_query.py 分别测试 Global Search 和 Local Search。
  5. 对比同一问题在朴素 RAG 和 GraphRAG 下的答案差异。

判断练习

  1. 索引构建后，社区报告显示所有实体都在一个社区里。这是什么原因？

    参考答案

    Leiden 分辨率参数太低，导致图被过度合并为一个大社区。解决方法：在 settings.yaml 中降低 community_resolution 或对 community_reports 设置更高的 max_length。也可以检查实体提取的质量——如果实体太少或关系太稀疏，社区检测也会失效。

  2. Global Search 的答案过于泛泛，缺乏具体数据支撑。怎么改进？

    参考答案

    1) 改进社区摘要参数：增加 max_length（如 from 2000 to 3000），让摘要包含更多具体数字和实例；2) 在 Global Search prompt 中加入「请引用具体实体和关系」的指令；3) 考虑 Hybrid 模式：先用 Global 找相关社区，再对社区内关键实体做 Local Search。
