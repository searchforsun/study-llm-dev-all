LCEL 管线构建

CorpAssist · HR 政策问答员工提问「年假折算规则」，管线必须从 corpassist-hr-v2 collection 检索政策原文，填入 system prompt 约束模型，禁止跳过检索直接生成。

LangChain Expression Language（LCEL）以声明式管道组合 RAG。核心数据流：输入 dict → 检索并格式化 context → 注入 Prompt → LLM 推理 → 解析输出。每个 | 右侧可以是 Runnable 对象、函数或任意可调用对象，框架自动处理类型兼容。LCEL 的设计哲学是将 RAG 的每一步抽象为可组合的 Runnable 单元：RunnableParallel 并行构造输入、RunnablePassthrough 透传字段、RunnableLambda 嵌入自定义函数、StrOutputParser 解析 ChatMessage 输出为字符串。这一设计使得调试时可以 .get_graph() 导出管线拓扑，或使用 .with_listeners() 在每步执行前后插入回调。

LCEL 管线数据流
flowchart LR
  Q[{"question":str}] --> RC[Retrieve & Format]
  RC --> P[ChatPromptTemplate]
  Q --> P
  P --> L[LLM / ChatModel]
  L --> OP[StrOutputParser]
  OP --> R[{"answer":str,"sources":list}]
python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableParallel
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma(
    collection_name="corpassist-hr-v2",
    embedding_function=embeddings,
    persist_directory="./chroma_db",
)
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4, "filter": {"tenant_id": "corp-a", "kb": "hr"}},
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是CorpAssist政策助手。仅根据以下context回答，无依据则说不知道。\n{context}"),
    ("human", "{question}"),
])

def format_docs(docs):
    return "\n\n---\n\n".join(d.page_content for d in docs)

chain = (
    RunnableParallel({"context": retriever | format_docs, "question": RunnablePassthrough()})
    | prompt
    | llm
    | StrOutputParser()
)

Retriever 配置与调优

Retriever 是管线召回入口，其配置直接影响回答质量。CorpAssist 在 metadata 中存储 tenant_id、kb、doc_id、page_num，供上层过滤与引用溯源。

参数建议值调优依据CorpAssist 场景
k4 ~ 8k=4 召回率 72%；k=8 召回率 89% 但 context 增长 2x tokenHR 政策用 k=4；技术文档用 k=6
score_threshold0.45 ~ 0.55低于 0.45 的文档 68% 与问题无关（基于 500 条评测）配合 score 字段写入 trace 供拒答判断
filtertenant_id + kb强制过滤防止跨租户泄漏勿只依赖 prompt 约束
search_typesimilarity / mmrMMR 降低重复片段，lambda=0.7 时多样性+23%政策文档用 similarity，FAQ 用 mmr

    search_type="mmr",
    search_kwargs={
        "k": 6,
        "fetch_k": 20,
        "lambda_mult": 0.7,
        "filter": {"tenant_id": "corp-a"},
    },
)

FAQ：filter 在检索层 vs 在 prompt 层有什么区别？

检索层 filter 直接作用于向量数据库查询语句（WHERE 子句），确保跨租户数据物理隔离；prompt 层约束仅在推理时提示，模型仍可能从训练数据中越权输出。CorpAssist 安全审计要求检索层 filter 为必选项。

自检：k=12 比 k=4 好在哪里？坏在哪里？

好：召回率从 72% 提升至 94%，覆盖更多相关片段。坏：token 消耗增加 3 倍（约 2000→6000 token/次），P95 延迟从 0.8s 增至 1.9s，且噪声片段增多导致答案质量下降 11%（基于 RAGAS faithfulness 评测）。

自检：如果 sources 非空但 answer 胡编，最可能的原因是什么？

最常见的原因是 Prompt 的 system message 未明确约束「仅根据 context 回答」—— 模型将 context 视作参考而非限定条件。其次检查 RunnableParallel 的 context 键名是否与 Prompt 模板的 {context} 变量一致。第三，确认 format_docs 函数返回的字符串确实包含文档内容，而非空字符串或仅含 metadata。

  • sources 数组非空，至少 1 条 doc_id
  • answer 文本 length > 0
  • metadata 含 latency_ms、retrieval_k
  • HTTP 200 但 sources: [], answer: "无法找到相关信息"
  • 原因：filter 不匹配、collection 不存在、embedding 维度不一致
  • 检查：对比索引时的 collection_name 和 filter 值

API 暴露与流式输出

Python RAG 链通过 FastAPI 暴露，响应体与 Spring AI BFF 对齐。CorpAssist 协议规定 /v1/rag/ask 返回 answer 字段 + sources 数组，每个 source 含 doc_id 与 snippet。流式模式下先返回 answer 块，最后补 sources。

python
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List
from langchain_core.runnables import RunnableConfig

app = FastAPI(title="CorpAssist RAG API")

class AskRequest(BaseModel):
    question: str
    tenant_id: str = "corp-a"
    kb: str = "hr"
    stream: bool = False

class SourceRef(BaseModel):
    doc_id: str
    snippet: str
    score: float

class AskResponse(BaseModel):
    answer: str
    sources: List[SourceRef]

@app.post("/v1/rag/ask", response_model=AskResponse)
async def ask(req: AskRequest):
    filter_dict = {"tenant_id": req.tenant_id, "kb": req.kb}
    result = chain_with_sources.invoke(
        {"question": req.question, "filter": filter_dict}
    )
    return AskResponse(
        answer=result["answer"],
        sources=[
            SourceRef(doc_id=s["doc_id"], snippet=s["snippet"], score=s["score"])
            for s in result["sources"]
        ],
    )

# 流式端点
@app.post("/v1/rag/ask/stream")
async def ask_stream(req: AskRequest):
    async def generate():
        async for chunk in chain.astream(
            {"question": req.question, "filter": {"tenant_id": req.tenant_id}}
        ):
            if isinstance(chunk, str):
                yield f"data: {chunk}\n\n"

本章结论

  • LCEL 管线：RunnableParallel 并行构造 context+question，| 管道传递数据流，单链可 invoke/stream/batch
  • Retriever 配置：TopK(k=4~8) + metadata filter(tenant_id+kb) 在检索层实现数据隔离，score_threshold=0.45 过滤低质文档
  • 三步调试：Retriever-only → 半链(Prompt) → 全链，80% 检索问题可在第一步发现
  • API 契约：/v1/rag/ask 返回 answer+sources，与 Spring AI BFF 对齐 JSON 字段名
  • 流式输出：SSE 协议先推 answer 块后推 sources 元数据，P95 首 token 延迟控制在 1.2s 内

面试可能问

LCEL 中 RunnableParallel 与 RunnablePassthrough 各自的使用场景？

RunnableParallel 用于从同一输入构造多个分支，如同时产生 context 和透传 question；RunnablePassthrough 用于透传某个键的值不做变换。若检索结果需与原始 question 合并进 prompt，必须用 RunnableParallel。

如何排查 sources 为空但向量库有数据的情况？

1) filter 条件是否与索引数据的 metadata 精确匹配（如 tenant_id 大小写）；2) embedding 模型是否与索引时一致（模型版本/维度）；3) query 语言/分词是否与文档差异过大（可试混合检索缓解）。

LCEL 的 invoke 与 stream 底层如何实现？

invoke 走 RunnableSequence.__call__，逐节点同步执行；stream 利用生成器逐节点 yield 输出块，每个节点实现 transform 方法流式传递。框架自动处理类型适配与回调传播。

场景 1：员工查「竞业限制补偿金」，sources 返回空，但索引库含相关政策文档。列出优先级从高到低的 3 个排查项。

① filter 是否写了 kb:hr 但竞业文档存储在 kb:legal；② embedding 模型版本是否与索引时一致（text-embedding-3-small 维度 1536）；③ 查询中「竞业限制」为四字专业术语，分词后 token 分布是否与索引文档匹配，可试混合检索。

场景 2：启用流式输出后客户端只收到部分 answer，无 sources。可能的原因？

① 流式生成器未在 answer 完成后发送 sources 元数据事件；② 客户端的 SSE 解析器未处理 data: [DONE] 后追加的元数据行；③ 链的 sources 部分使用 .invoke 而非 .astream，导致阻塞在最后一个节点。
