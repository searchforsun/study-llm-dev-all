为什么要学本章

RAG（检索增强生成）是 LLM 落地最广泛的生产模式。LangChain 提供了最成熟的 Python RAG 生态：LCEL 声明式管线、多种 Retriever 策略、灵活的 QA Chain 组合。本章通过 CorpAssist 知识库场景，带你从零构建一条生产级 LangChain RAG 管线。

学完你能
  • 用 LCEL 声明式语法组合 document loader → splitter → embedding → retriever → QA chain
  • 为 CorpAssist 场景选择最合适的 Retriever 类型并配置参数
  • 实现 stuff / map_reduce 两种 QA Chain 并理解各自的 token 成本
  • 用 FastAPI + SSE 实现流式输出，首 token 延迟 < 300ms
本章先记住 3 件事
  • LCEL 是声明式组合：用 | 操作符串联组件，比回调式更可读、可追踪。
  • Retriever 决定召回质量：向量检索适合语义相似，MMR 适合多样性，阈值过滤确保精准。
  • 流式输出是生产必备：用户等待体验从 >5s 降至 <1s 首 token，必须原生支持 SSE。

LCEL 管线构建：从文档到检索

LCEL（LangChain Expression Language）是 LangChain 的声明式管线语言。核心思想是用 | 操作符将组件组合为计算图：input | component_a | component_b | output。每个组件实现 Runnable 接口，保证可组合、可追踪、可流式。

CorpAssist · 企业知识库 RAG CorpAssist 需要从企业内部文档（员工手册、产品说明、流程规范）构建知识库问答系统。原始文档为 PDF 格式，需经过加载→切分→向量化→检索→生成的全流程。

文档加载与切分

python 
# documents.py — CorpAssist 文档加载与切分
from langchain_community.document_loaders import PyPDFLoader, DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.schema import Document
from typing import List
import os

def load_documents(doc_dir: str = "./data/docs") -> List[Document]:
    """加载目录下所有 PDF 文档"""
    loader = DirectoryLoader(
        doc_dir,
        glob="**/*.pdf",
        loader_cls=PyPDFLoader,
        loader_kwargs={"extract_images": False},
        show_progress=True,
    )
    documents = loader.load()
    print(f"Loaded {len(documents)} document pages")
    return documents

def split_documents(documents: List[Document]) -> List[Document]:
    """递归字符切分，保留文档元数据"""
    text_splitter = RecursiveCharacterTextSplitter(
        chunk_size=1024,
        chunk_overlap=200,
        length_function=len,
        separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""],
    )
    chunks = text_splitter.split_documents(documents)
    print(f"Split into {len(chunks)} chunks (avg {sum(len(c.page_content) for c in chunks) // len(chunks)} chars)")
    return chunks

# 使用示例
if __name__ == "__main__":
    docs = load_documents("./data/docs")
    chunks = split_documents(docs)
    print(f"Chunk range: {min(len(c.page_content) for c in chunks)}-{max(len(c.page_content) for c in chunks)} chars")

向量存储与检索器构建

python 
# vector_store.py — 向量存储与检索器
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain.schema import Document
from typing import List, Optional

EMBEDDING_MODEL = "text-embedding-3-small"
COLLECTION_NAME = "corpassist_kb"

def create_vector_store(
    chunks: List[Document],
    persist_dir: str = "./data/chroma_db",
) -> Chroma:
    """创建并持久化向量存储"""
    embeddings = OpenAIEmbeddings(
        model=EMBEDDING_MODEL,
        dimensions=512,  # text-embedding-3-small 支持降维
    )
    vector_store = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory=persist_dir,
        collection_name=COLLECTION_NAME,
    )
    print(f"Vector store created with {vector_store._collection.count()} vectors")
    return vector_store

def get_retriever(
    vector_store: Chroma,
    k: int = 5,
    score_threshold: Optional[float] = None,
) -> Chroma:
    """获取检索器，支持相似度阈值过滤"""
    search_kwargs = {"k": k}
    if score_threshold is not None:
        search_kwargs["score_threshold"] = score_threshold
    return vector_store.as_retriever(
        search_type="similarity_score_threshold" if score_threshold else "similarity",
        search_kwargs=search_kwargs,
    )

# 验证检索质量
if __name__ == "__main__":
    store = Chroma(
        persist_directory="./data/chroma_db",
        embedding_function=OpenAIEmbeddings(model=EMBEDDING_MODEL),
        collection_name=COLLECTION_NAME,
    )
    retriever = get_retriever(store, k=3)
    results = retriever.invoke("员工休假政策有哪些？")
    for i, doc in enumerate(results):
        print(f"[{i}] score={doc.metadata.get('score', 'N/A'):.4f} | {doc.page_content[:80]}...")

完整的 LCEL 问答管线

python 
# pipeline.py — LCEL RAG 管线定义
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableParallel
from operator import itemgetter
from typing import List

# Prompt 模板
SYSTEM_TEMPLATE = """你是一个专业的企业知识库助手 CorpAssist。
请基于以下上下文回答用户问题。如果上下文中没有足够信息，请说「我需要查证后再回复」。

上下文：
{context}

用户问题：{question}
回答："""

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM_TEMPLATE),
    ("human", "{question}"),
])

def format_docs(docs: List) -> str:
    """将检索结果格式化为上下文文本"""
    return "\n\n---\n\n".join(
        f"[来源:{d.metadata.get('source', 'unknown')}] {d.page_content}"
        for i, d in enumerate(docs)
    )

def build_rag_chain():
    """构建完整 LCEL RAG 管线"""
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)
    vector_store = Chroma(
        persist_directory="./data/chroma_db",
        embedding_function=embeddings,
        collection_name="corpassist_kb",
    )
    retriever = vector_store.as_retriever(search_kwargs={"k": 5})

    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.1)

    # LCEL 管线：retrieve → format → prompt → llm → parse
    rag_chain = (
        RunnableParallel(
            {"context": retriever | format_docs, "question": RunnablePassthrough()}
        )
        | prompt
        | llm
        | StrOutputParser()
    )
    return rag_chain

if __name__ == "__main__":
    chain = build_rag_chain()
    answer = chain.invoke("CorpAssist 的休假政策是怎样的？")
    print(answer)
不推荐
  • 手动拼接 prompt 字符串，无结构化组件
  • chunk_size = 2000 无 overlap，段落被切断
  • 用 callback 模式嵌套回调，调试困难
  • 每次调用重新创建 embedding 连接
推荐
  • LCEL | 声明式组合，组件可独立测试
  • chunk_size=1024, overlap=200, 保留语义边界
  • RunnableParallel 实现检索与输入的并行计算
  • 单例模式复用 vector store 和 embedding 连接
先想 10 秒：LCEL 的 RunnableParallel 和 Python 的 ThreadPoolExecutor 有何本质区别？

RunnableParallel 是 LangChain 框架层的并发原语，它可以与 LCEL 的 batch/stream 语义集成，自动处理回调、错误传播和追踪。ThreadPoolExecutor 只能处理 Python 层面的并发，无法被 LangSmith 追踪。

Retriever 类型：选型与配置

Retriever 是 RAG 管线的核心——它直接决定召回质量的上限。LangChain 提供多种 Retriever 实现，适用不同场景。

检索类型适合场景核心参数延迟召回特点
Similarity 通用语义搜索 k, score_threshold 低 按向量距离取 top-k
MMR 去重、多样化结果 k, fetch_k, lambda_mult 中 兼顾相关性与多样性
Similarity Threshold 高精度场景 score_threshold, k 低 低于阈值的文档被过滤
Ensemble Retriever 混合检索 retrievers, weights 中高 融合多种检索结果
MultiQuery Retriever 查询改写增强 llm, n_queries 高 从多角度查询并合并

MMR 检索与 Ensemble 检索实战

python 
# retrievers.py — 多类型 Retriever 对比
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain.retrievers import MMRRetriever, EnsembleRetriever
from langchain.retrievers.multi_query import MultiQueryRetriever
from langchain_openai import ChatOpenAI

embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)
store = Chroma(persist_directory="./data/chroma_db", embedding_function=embeddings)

# 1. MMR 检索：多样性与相关性平衡
mmr_retriever = MMRRetriever(
    vectorstore=store,
    search_type="mmr",
    search_kwargs={"k": 5, "fetch_k": 20, "lambda_mult": 0.5},
)
# lambda_mult: 0=最大多样性, 1=最大相关性

# 2. 基于 BM25 的关键词检索器（需 pip install rank_bm25）
from langchain.retrievers import BM25Retriever
from langchain.schema import Document

all_docs = store.get()  # 获取所有文档
bm25_retriever = BM25Retriever.from_documents(
    [Document(page_content=t) for t in all_docs["documents"]],
    k=5,
)

# 3. Ensemble 融合检索
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, store.as_retriever(search_kwargs={"k": 5})],
    weights=[0.3, 0.7],  # BM25 权重 0.3, 语义权重 0.7
)

# 4. MultiQuery: LLM 自动生成多个查询变体
multiquery_retriever = MultiQueryRetriever.from_llm(
    retriever=store.as_retriever(search_kwargs={"k": 5}),
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
)

# 对比测试
question = "员工请假需要提前几天申请？"
for name, retriever in [
    ("Similarity", store.as_retriever(search_kwargs={"k": 3})),
    ("MMR", mmr_retriever),
    ("Ensemble", ensemble_retriever),
    ("MultiQuery", multiquery_retriever),
]:
    docs = retriever.invoke(question)
    print(f"[{name}] retrieved {len(docs)} docs")
    for d in docs:
        print(f"  -> {d.page_content[:60]}...")

CorpAssist 检索策略决策表

业务场景推荐 Retriever理由
员工手册问答 Similarity Threshold > 0.7 需要高精度，不回答未知信息
产品文档搜索 MMR (lambda_mult=0.5) 避免返回内容高度重复的片段
跨部门知识检索 Ensemble (0.4 BM25 + 0.6 语义) 术语匹配 + 语义理解的互补
模糊问题定位 MultiQuery (n_queries=3) 从不同角度改写后综合检索

QA Chain：Stuff 与 MapReduce 策略

QA Chain 决定如何将检索到的文档片段送入 LLM 生成答案。LangChain 支持多种组合策略，核心区别在于 token 消耗与上下文保留能力。

策略token 消耗适合场景流式支持
Stuff 低 (1次LLM调用) 文档总长 < 窗口，简单问答 是
MapReduce 高 (N+1次调用) 超长文档 > 窗口，需要全面覆盖 否
Refine 中 (N次迭代) 需要逐步精炼答案 否
python

CorpAssist 建议

对于绝大多数 CorpAssist 知识库场景，top-5 chunk 总长度约 5K tokens，Stuff 策略足够并支持流式输出。MapReduce 仅在单次检索结果超过 16K tokens 时考虑（如完整政策文档汇总）。

流式输出：SSE 实现与首 Token 延迟优化

生产级 RAG 系统必须支持流式输出。用户看到第一个 token 的时间（首 token 延迟）直接决定体验。LangChain LCEL 原生支持 .stream() 方法，结合 FastAPI 的 StreamingResponse 实现 SSE。

python 
# app.py — FastAPI + SSE 流式 RAG 服务
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough
from typing import AsyncGenerator, List
import json

app = FastAPI(title="CorpAssist RAG API")

class QueryRequest(BaseModel):
    question: str
    top_k: int = 5

class StreamChunk(BaseModel):
    token: str
    done: bool = False

def build_chain():
    """构建可流式的 LCEL 管线"""
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)
    vector_store = Chroma(
        persist_directory="./data/chroma_db",
        embedding_function=embeddings,
        collection_name="corpassist_kb",
    )
    retriever = vector_store.as_retriever(search_kwargs={"k": 5})

    prompt = ChatPromptTemplate.from_messages([
        ("system", "基于以下上下文回答问题。\n\n{context}\n\n问题：{question}"),
    ])
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.1, streaming=True)

    def format_docs(docs):
        return "\n\n".join(d.page_content for d in docs)

    chain = (
        RunnableParallel(
            {"context": retriever | format_docs, "question": RunnablePassthrough()}
        )
        | prompt
        | llm
        | StrOutputParser()
    )
    return chain

rag_chain = build_chain()

async def stream_answer(question: str) -> AsyncGenerator[str, None]:
    """SSE 流式生成"""
    async for chunk in rag_chain.astream(question):
        if chunk:
            data = json.dumps({"token": chunk, "done": False}, ensure_ascii=False)
            yield f"data: {data}\n\n"
    yield f"data: {json.dumps({'token': '', 'done': True})}\n\n"

@app.post("/ask")
async def ask(req: QueryRequest):
    """非流式问答"""
    answer = await rag_chain.ainvoke(req.question)
    return {"answer": answer}

@app.post("/ask/stream")
async def ask_stream(req: QueryRequest):
    """流式问答 SSE"""
    return StreamingResponse(
        stream_answer(req.question),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )

@app.get("/health")
async def health():
    return {"status": "ok", "chain_ready": rag_chain is not None}

    participant U as 用户
    participant API as FastAPI
    participant Ret as Retriever
    participant LLM as LLM
    participant S as Stream

    Note over U,S: 非流式 (TTFT ~5-8s)

    API->>Ret: 检索文档
    Ret-->>API: top-5 chunks
    API->>LLM: 完整 prompt
    LLM-->>API: 完整答案
    API-->>U: 200 OK (5-8s后)

    Note over U,S: 流式 (TTFT ~300ms)
    U->>API: POST /ask/stream
    API->>Ret: 检索文档
    Ret-->>API: top-5 chunks
    API->>LLM: 完整 prompt
    LLM-->>S: token 0 (300ms)
    S-->>U: data: token
    LLM-->>S: token 1
    S-->>U: data: token
    LLM-->>S: token N
    S-->>U: data: done

流式 SSE 将首 Token 感知时间从 >5s 降至 ~300ms，显著改善用户体验。

流式优化 Checklist

  • ChatOpenAI(streaming=True) — 开启模型流式
  • .astream() — 使用 LCEL 异步流式方法
  • StreamingResponse(media_type="text/event-stream") — FastAPI SSE 响应
  • X-Accel-Buffering: no — 禁用 Nginx 缓冲区
  • 检索阶段同步等待（不可流式），组装完成后 LLM 部分流式输出

本章结论

  • LCEL 声明式管线：用 | 和 RunnableParallel 组合组件，比回调式更可读、可追踪。
  • Retriever 选型决定上限：Similarity/MMR/Ensemble/MultiQuery 各有适用场景，按 CorpAssist 业务需求选择。
  • Stuff 策略优先：top-5 chunk 5K tokens 以内优先用 Stuff；文档过长才考虑 MapReduce。
  • 流式是生产标配：SSE + async LCEL 实现首 Token < 300ms，显著改善用户体验。
  • 文档切分是前置质量：chunk_size=1024, overlap=200, 保留语义边界直接影响检索效果。

动手练习

用 CorpAssist 知识库场景，完成 LCEL 管线构建的完整流程（约 30 分钟）。

操作步骤

  1. 下载 3-5 份企业内部 PDF 文档（员工手册、产品说明等），放入 data/docs/ 目录。
  2. 运行 documents.py 加载并切分文档，调整 chunk_size 观察切分效果。
  3. 运行 vector_store.py 创建向量索引，测试不同 Retriever 类型对同一问题的召回差异。
  4. 启动 app.py 的 FastAPI 服务，用 curl 测试 /ask/stream 流式端点。
  5. 比较 4 种 Retriever（Similarity/MMR/Ensemble/MultiQuery）的答案差异。

判断练习

  1. documents.py 中切分参数 chunk_size=1024, overlap=200。如果改 chunk_size=256，对检索质量有何影响？

    参考答案

    更精细但碎片化：chunk_size 减小会生成更多片段，可能丢失段落完整意思。语义检索时每个 chunk 上下文不足，召回命中率高但答案完整性下降。建议 CorpAssist 场景 chunk_size 在 512-1024 之间调优。

  2. 流式模式下，用户反馈「回答开始很快但中间停顿 3 秒」。可能的原因和解决思路？

    参考答案

    可能是 LLM 生成中遇到长句或代码块。解决方法：1) 确认 streaming=True 参数生效；2) 检查网络延迟（跨区域调用）；3) 考虑使用本地部署模型降低中间跳变；4) 在 prompt 中要求「简短分点输出」减少长段落。
