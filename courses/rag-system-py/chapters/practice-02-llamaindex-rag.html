同需求双实现对比

CorpAssist · LangChain vs LlamaIndex同一需求：HR 政策 RAG 问答。输入「年假折算规则」，两套框架各自输出 answer + sources。比较代码量、延迟、召回质量。

LlamaIndex 以 Index 为核心入口：VectorStoreIndex 封装了分块 → Embedding → 存储 → 检索的完整生命周期。查询时通过 as_query_engine() 获得配置好的检索+生成组件，省去手动搭建 LCEL 管道的步骤。

python
# LlamaIndex 等效实现 — HR 政策问答
from llama_index.core import VectorStoreIndex, Settings, Document
from llama_index.core.node_parser import SentenceSplitter
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.llms.openai import OpenAI

Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0)

# 从已有向量库加载 — 与 LangChain 同一数据源
from llama_index.vector_stores.chroma import ChromaVectorStore
import chromadb

db = chromadb.PersistentClient(path="./chroma_db")
chroma_collection = db.get_or_create_collection("corpassist-hr-v2")
vector_store = ChromaVectorStore(chroma_collection=chroma_collection)
index = VectorStoreIndex.from_vector_store(vector_store)

# 查询引擎 — 单行替代 LCEL 四段链
query_engine = index.as_query_engine(
    similarity_top_k=4,
    node_postprocessors=[],  # 可插入 rerank
    response_mode="compact",
)

response = query_engine.query("年假折算规则")
print(response.response)
for node in response.source_nodes:
    print(node.node_id, node.text[:80], node.score)

性能基准测试

在 CorpAssist HR 知识库（500 篇文档，3200 分块）上对两框架进行基准测试。测试环境：CPU Intel i7-12700, 32GB RAM, OpenAI text-embedding-3-small + gpt-4o-mini, 100 条随机问题。测试重复 3 轮取中位数，预先做 10 次 warmup 消除冷启动偏差。向量库使用 Chroma（SSD 持久化），LLM 调用统一使用 gpt-4o-mini（temperature=0）。

Benchmark 脚本的设计要点：1) warmup 阶段读取向量库到内存（Page cache），避免首次查询的冷延迟；2) 每轮测试前重新向量化 query 而非复用缓存结果，模拟真实用户场景；3) 设置超时 30s，超过则标记为失败并重试 1 次；4) 记录每个子步骤的延迟（embed、search、build_prompt、llm_infer、parse），便于定位瓶颈。所有测试结果写入 JSONL 格式，每行包含 query、两框架的完整延迟分解、answer 文本、召回 doc_id 列表。

结论：两框架在检索和生成阶段均调用相同的 OpenAI API，端到端延迟差异 < 2%，主要延迟瓶颈在 LLM 推理（~70%）。LlamaIndex 代码更精简（Index 封装了索引构建管线），但 LangChain 在 Prompt 自定义和调试工具（LangSmith）方面更成熟。

延迟分布详细分析：向量嵌入阶段两框架完全一致（均调用 OpenAI Embedding API，耗时 318~320ms），向量检索阶段 Chroma 的查询时间约 45~48ms，差异来自结果处理——LlamaIndex 的 Node 解析比 LangChain 的 Document 处理多约 3ms 的元数据提取。Prompt 构建阶段 LangChain 的 RunnableParallel 比 LlamaIndex 的 Synthesizer 少约 6ms。LLM 推理阶段两框架均调用同一 API 端点，耗时几乎相等（892ms vs 890ms，偏差在 API 响应抖动范围内）。

两框架在代码组织哲学上有根本差异：LangChain 强调「链」的显式组装，每一环节可观测、可替换；LlamaIndex 强调「索引」的智能封装，常用场景开箱即用。CorpAssist 项目中，Python 侧的索引任务使用 LlamaIndex 的 Index 抽象更高效，但查询链与 Spring 对齐的部分仍用 LangChain 的 LCEL 便于两侧对照调试。

LangChain 适用
  • 需精细控制 Prompt 模板
  • 链中插入自定义 callback/logging
  • 与 LangSmith 深度集成
  • 需要 handoff/human-in-loop 节点
LlamaIndex 适用
  • 快速原型验证 RAG 场景
  • 需要 RouterQueryEngine / SubQuestionQueryEngine
  • 处理复杂文档（PDF 表格、多级标题）
  • 倾向更少样板代码的团队
python
# LlamaIndex RouterQueryEngine — 内置意图路由
from llama_index.core.tools import QueryEngineTool, ToolMetadata
from llama_index.core.query_engine import RouterQueryEngine

# 为不同知识库创建专用查询引擎
hr_engine = hr_index.as_query_engine(similarity_top_k=4)
legal_engine = legal_index.as_query_engine(similarity_top_k=3)

router = RouterQueryEngine.from_defaults(
    llm=Settings.llm,
    query_engine_tools=[
        QueryEngineTool(
            query_engine=hr_engine,
            metadata=ToolMetadata(
                name="hr_policy",
                description="HR政策相关查询（年假、考勤、薪酬、福利）",
            ),
        ),
        QueryEngineTool(
            query_engine=legal_engine,
            metadata=ToolMetadata(
                name="legal_compliance",
                description="合规与法务相关查询（竞业限制、合同条款）",
            ),
        ),
    ],
    verbose=True,
)

# 自动路由到正确知识库
resp = router.query("竞业限制补偿金标准")
print(resp.metadata["selector_result"])  # 查看路由选择

FAQ：如果团队已深入 LangChain，迁移到 LlamaIndex 的成本多大？

迁移成本取决于依赖深度。若仅用基础 RAG，重构约 1~2 人天（替换索引构建 + 查询调用）。若使用了 LangGraph Agent、LangSmith 评测、自定义 Callback，迁移成本升至 1~2 周。建议新场景直接用 LlamaIndex 验证原型，存量 LangChain 场景逐步并行。

分步迁移路径建议：第 1 步（1 天），在新知识库上使用 LlamaIndex 做索引和查询原型验证；第 2 步（2 天），与 LangChain 管线并行运行，确认输出一致性（answer + sources 字段逐条对比）；第 3 步（1 天），将 RouterQueryEngine 替换手写路由逻辑；第 4 步（2 天），迁移自定义 Callback（LangSmith → LlamaIndex CallbackManager），配置 LlamaTrace。核心原则：查询链不要同时维护两套实现，应彻底替换。

作为补充对比，以下是两框架在功能点层面的逐一对比：对于流式输出，LangChain 通过 chain.stream() 返回逐 token 生成器，LlamaIndex 通过 query_engine.aquery(stream=True) 返回异步生成器。对于回调系统，LangChain 的 CallbackHandler 基于事件驱动（on_llm_start/on_retriever_end），LlamaIndex 通过 Event 机制发送每个步骤的通知。对于缓存策略，LangChain 提供 Runnable.cache() 装饰器自动缓存，LlamaIndex 需要手动实现 NodeParser 级别的缓存。对于并行处理，LangChain 的 batch() 方法可以并行处理多条输入，LlamaIndex 的 process_documents_async() 支持异步批量索引。对于错误处理，LangChain 的 with_fallbacks() 可以为 Runnable 配置降级方案，LlamaIndex 的 ErrorHandler 插件可配置重试逻辑。

自检：在什么场景下 LlamaIndex 比 LangChain 更适合 CorpAssist？

当需要快速接入多种知识库并自动路由时（如 HR + Legal + IT 三个知识库），LlamaIndex 的 RouterQueryEngine 可省去手写意图分类和工具绑定逻辑。此外处理 PDF 采购合同时，LlamaIndex 的 PDF 解析和 RecursiveRetriever 能自动拆解表格和层级标题。

本章结论

  • 双框架等效性：LangChain 和 LlamaIndex 在基础 RAG 场景下性能差异 < 2%，代码量 LlamaIndex 少 46%
  • RouterQueryEngine：LlamaIndex 内置路由省去手写意图分类，适合多知识库 CorpAssist 场景
  • 代码哲学差异：LangChain 显式链式组装适合调试与自定义，LlamaIndex Index 封装适合快速原型
  • 延迟瓶颈相同：两框架端到端 P95 约 1.9s，其中 LLM 推理占 ~70%，检索仅占 ~3%
  • 迁移建议：新场景用 LlamaIndex 原型，存量 LangChain 逐步并行，CorpAssist 双栈可共存

面试可能问

LlamaIndex 的 Index 和 LangChain 的 VectorStore 在设计理念上有什么本质不同？

Index 是「有状态的智能对象」—— 它记录了文档结构、节点关系、嵌入模型引用，查询时自动编排；VectorStore 是「无状态的存储抽象」—— 只负责向量存储和检索，查询编排由上层链完成。LlamaIndex 的 Index 更接近 ORM 思想，LangChain 的 VectorStore 更接近 DAO 思想。

为什么在 CorpAssist 中 LangChain 和 LlamaIndex 可以共存？

两者抽象层次不同：LangChain 用于查询链编排（与 Spring AI 对齐），LlamaIndex 用于索引构建和复杂文档解析。Python 侧内部调用无冲突，输出统一为 answer+sources 契约。共存策略让各框架发挥各自优势。

RouterQueryEngine 的底层实现原理是什么？

它本质上是 LLM 驱动的选择器：将每个 QueryEngineTool 的 name+description 拼入 system prompt，LLM 根据用户输入选择最匹配的工具。选择结果作为 metadata 返回，再调用对应 QueryEngine。类似 ReAct Agent 的单步路由变体。

场景 1：HR 政策查询返回的 answer 准确但缺少引用来源，应如何配置 QueryEngine？

设置 response_mode="compact" 并确保 similarity_top_k>=3。在 response 中通过 source_nodes 字段获取引用。若仍需更详细的引用格式，配置 CitationQueryEngine 或自定义 node_postprocessors。

场景 2：RouterQueryEngine 总是路由到同一个知识库不切换，如何调试？

启用 verbose=True 查看 selector 日志，检查 ToolMetadata 的 description 是否包含足够区分度。若 LLM 始终选同一工具，可尝试在 description 中加入反例描述（如「不要用于节假日查询」）缩小分类边界。
