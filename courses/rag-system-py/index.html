为什么要学本章

RAG（检索增强生成）是 LLM 落地最广泛的生产模式。LangChain 提供了最成熟的 Python RAG 生态：LCEL 声明式管线、多种 Retriever 策略、灵活的 QA Chain 组合。本章通过 CorpAssist 知识库场景，带你从零构建一条生产级 LangChain RAG 管线。

学完你能
  • 用 LCEL 声明式语法组合 document loader → splitter → embedding → retriever → QA chain
  • 为 CorpAssist 场景选择最合适的 Retriever 类型并配置参数
  • 实现 stuff / map_reduce 两种 QA Chain 并理解各自的 token 成本
  • 用 FastAPI + SSE 实现流式输出，首 token 延迟 < 300ms
本章先记住 3 件事
  • LCEL 是声明式组合：用 | 操作符串联组件，比回调式更可读、可追踪。
  • Retriever 决定召回质量：向量检索适合语义相似，MMR 适合多样性，阈值过滤确保精准。
  • 流式输出是生产必备：用户等待体验从 >5s 降至 <1s 首 token，必须原生支持 SSE。

LCEL 管线构建：从文档到检索

LCEL（LangChain Expression Language）是 LangChain 的声明式管线语言。核心思想是用 | 操作符将组件组合为计算图：input | component_a | component_b | output。每个组件实现 Runnable 接口，保证可组合、可追踪、可流式。

CorpAssist · 企业知识库 RAG CorpAssist 需要从企业内部文档（员工手册、产品说明、流程规范）构建知识库问答系统。原始文档为 PDF 格式，需经过加载→切分→向量化→检索→生成的全流程。

文档加载与切分

python 
# documents.py — CorpAssist 文档加载与切分
from langchain_community.document_loaders import PyPDFLoader, DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.schema import Document
from typing import List
import os

def load_documents(doc_dir: str = "./data/docs") -> List[Document]:
    """加载目录下所有 PDF 文档"""
    loader = DirectoryLoader(
        doc_dir,
        glob="**/*.pdf",
        loader_cls=PyPDFLoader,
        loader_kwargs={"extract_images": False},
        show_progress=True,
    )
    documents = loader.load()
    print(f"Loaded {len(documents)} document pages")
    return documents

def split_documents(documents: List[Document]) -> List[Document]:
    """递归字符切分，保留文档元数据"""
    text_splitter = RecursiveCharacterTextSplitter(
        chunk_size=1024,
        chunk_overlap=200,
        length_function=len,
        separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""],
    )
    chunks = text_splitter.split_documents(documents)
    print(f"Split into {len(chunks)} chunks (avg {sum(len(c.page_content) for c in chunks) // len(chunks)} chars)")
    return chunks

# 使用示例
if __name__ == "__main__":
    docs = load_documents("./data/docs")
    chunks = split_documents(docs)
    print(f"Chunk range: {min(len(c.page_content) for c in chunks)}-{max(len(c.page_content) for c in chunks)} chars")

向量存储与检索器构建

# vector_store.py — 向量存储与检索器
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain.schema import Document
from typing import List, Optional

EMBEDDING_MODEL = "text-embedding-3-small"
COLLECTION_NAME = "corpassist_kb"

def create_vector_store(
    chunks: List[Document],
    persist_dir: str = "./data/chroma_db",
) -> Chroma:
    """创建并持久化向量存储"""
    embeddings = OpenAIEmbeddings(
        model=EMBEDDING_MODEL,
        dimensions=512,  # text-embedding-3-small 支持降维
    )
    vector_store = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory=persist_dir,
        collection_name=COLLECTION_NAME,
    )
    print(f"Vector store created with {vector_store._collection.count()} vectors")
    return vector_store

def get_retriever(
    vector_store: Chroma,
    k: int = 5,
    score_threshold: Optional[float] = None,
) -> Chroma:
    """获取检索器，支持相似度阈值过滤"""
    search_kwargs = {"k": k}
    if score_threshold is not None:
        search_kwargs["score_threshold"] = score_threshold
    return vector_store.as_retriever(
        search_type="similarity_score_threshold" if score_threshold else "similarity",
        search_kwargs=search_kwargs,
    )

# 验证检索质量
if __name__ == "__main__":
    store = Chroma(
        persist_directory="./data/chroma_db",
        embedding_function=OpenAIEmbeddings(model=EMBEDDING_MODEL),
        collection_name=COLLECTION_NAME,
    )
    retriever = get_retriever(store, k=3)
    results = retriever.invoke("员工休假政策有哪些？")
    for i, doc in enumerate(results):
        print(f"[{i}] score={doc.metadata.get('score', 'N/A'):.4f} | {doc.page_content[:80]}...")

完整的 LCEL 问答管线

python 
# pipeline.py — LCEL RAG 管线定义
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableParallel
from operator import itemgetter
from typing import List

# Prompt 模板
SYSTEM_TEMPLATE = """你是一个专业的企业知识库助手 CorpAssist。
请基于以下上下文回答用户问题。如果上下文中没有足够信息，请说「我需要查证后再回复」。

上下文：
{context}

用户问题：{question}
回答："""

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM_TEMPLATE),
    ("human", "{question}"),
])

def format_docs(docs: List) -> str:
    """将检索结果格式化为上下文文本"""
    return "\n\n---\n\n".join(
        f"[来源:{d.metadata.get('source', 'unknown')}] {d.page_content}"
        for i, d in enumerate(docs)
    )

def build_rag_chain():
    """构建完整 LCEL RAG 管线"""
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)
    vector_store = Chroma(
        persist_directory="./data/chroma_db",
        embedding_function=embeddings,
        collection_name="corpassist_kb",
    )
    retriever = vector_store.as_retriever(search_kwargs={"k": 5})

    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.1)

    # LCEL 管线：retrieve → format → prompt → llm → parse
    rag_chain = (
        RunnableParallel(
            {"context": retriever | format_docs, "question": RunnablePassthrough()}
        )
        | prompt
        | llm
        | StrOutputParser()
    )
    return rag_chain

if __name__ == "__main__":
    chain = build_rag_chain()
    answer = chain.invoke("CorpAssist 的休假政策是怎样的？")
    print(answer)
不推荐
  • 手动拼接 prompt 字符串，无结构化组件
  • chunk_size = 2000 无 overlap，段落被切断
  • 用 callback 模式嵌套回调，调试困难
  • 每次调用重新创建 embedding 连接
推荐
  • LCEL | 声明式组合，组件可独立测试
  • chunk_size=1024, overlap=200, 保留语义边界
  • RunnableParallel 实现检索与输入的并行计算
  • 单例模式复用 vector store 和 embedding 连接
先想 10 秒：LCEL 的 RunnableParallel 和 Python 的 ThreadPoolExecutor 有何本质区别？

RunnableParallel 是 LangChain 框架层的并发原语，它可以与 LCEL 的 batch/stream 语义集成，自动处理回调、错误传播和追踪。ThreadPoolExecutor 只能处理 Python 层面的并发，无法被 LangSmith 追踪。

Retriever 类型：选型与配置

Retriever 是 RAG 管线的核心——它直接决定召回质量的上限。LangChain 提供多种 Retriever 实现，适用不同场景。

检索类型适合场景核心参数延迟召回特点
Similarity 通用语义搜索 k, score_threshold 低 按向量距离取 top-k
MMR 去重、多样化结果 k, fetch_k, lambda_mult 中 兼顾相关性与多样性
Similarity Threshold 高精度场景 score_threshold, k 低 低于阈值的文档被过滤
Ensemble Retriever 混合检索 retrievers, weights 中高 融合多种检索结果
MultiQuery Retriever 查询改写增强 llm, n_queries 高 从多角度查询并合并

MMR 检索与 Ensemble 检索实战

python 
# retrievers.py — 多类型 Retriever 对比
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain.retrievers import MMRRetriever, EnsembleRetriever
from langchain.retrievers.multi_query import MultiQueryRetriever
from langchain_openai import ChatOpenAI

embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)
store = Chroma(persist_directory="./data/chroma_db", embedding_function=embeddings)

# 1. MMR 检索：多样性与相关性平衡
mmr_retriever = MMRRetriever(
    vectorstore=store,
    search_type="mmr",
    search_kwargs={"k": 5, "fetch_k": 20, "lambda_mult": 0.5},
)
# lambda_mult: 0=最大多样性, 1=最大相关性

# 2. 基于 BM25 的关键词检索器（需 pip install rank_bm25）
from langchain.retrievers import BM25Retriever
from langchain.schema import Document

all_docs = store.get()  # 获取所有文档
bm25_retriever = BM25Retriever.from_documents(
    [Document(page_content=t) for t in all_docs["documents"]],
    k=5,
)

# 3. Ensemble 融合检索
ensemble_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, store.as_retriever(search_kwargs={"k": 5})],
    weights=[0.3, 0.7],  # BM25 权重 0.3, 语义权重 0.7
)

# 4. MultiQuery: LLM 自动生成多个查询变体
multiquery_retriever = MultiQueryRetriever.from_llm(
    retriever=store.as_retriever(search_kwargs={"k": 5}),
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
)

# 对比测试
question = "员工请假需要提前几天申请？"
for name, retriever in [
    ("Similarity", store.as_retriever(search_kwargs={"k": 3})),
    ("MMR", mmr_retriever),
    ("Ensemble", ensemble_retriever),
    ("MultiQuery", multiquery_retriever),
]:
    docs = retriever.invoke(question)
    print(f"[{name}] retrieved {len(docs)} docs")
    for d in docs:
        print(f"  -> {d.page_content[:60]}...")

QA Chain：Stuff 与 MapReduce 策略

QA Chain 决定如何将检索到的文档片段送入 LLM 生成答案。LangChain 支持多种组合策略，核心区别在于 token 消耗与上下文保留能力。

策略token 消耗适合场景流式支持
Stuff 低 (1次LLM调用) 文档总长 < 窗口，简单问答 是
MapReduce 高 (N+1次调用) 超长文档 > 窗口，需要全面覆盖 否
Refine 中 (N次迭代) 需要逐步精炼答案 否

CorpAssist 建议

对于绝大多数 CorpAssist 知识库场景，top-5 chunk 总长度约 5K tokens，Stuff 策略足够并支持流式输出。MapReduce 仅在单次检索结果超过 16K tokens 时考虑（如完整政策文档汇总）。

流式输出：SSE 实现与首 Token 延迟优化

生产级 RAG 系统必须支持流式输出。用户看到第一个 token 的时间（首 token 延迟）直接决定体验。LangChain LCEL 原生支持 .stream() 方法，结合 FastAPI 的 StreamingResponse 实现 SSE。

python 
# app.py — FastAPI + SSE 流式 RAG 服务
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough
from typing import AsyncGenerator, List
import json

app = FastAPI(title="CorpAssist RAG API")

class QueryRequest(BaseModel):
    question: str
    top_k: int = 5

class StreamChunk(BaseModel):
    token: str
    done: bool = False

def build_chain():
    """构建可流式的 LCEL 管线"""
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)
    vector_store = Chroma(
        persist_directory="./data/chroma_db",
        embedding_function=embeddings,
        collection_name="corpassist_kb",
    )
    retriever = vector_store.as_retriever(search_kwargs={"k": 5})

    prompt = ChatPromptTemplate.from_messages([
        ("system", "基于以下上下文回答问题。\n\n{context}\n\n问题：{question}"),
    ])
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.1, streaming=True)

    def format_docs(docs):
        return "\n\n".join(d.page_content for d in docs)

    chain = (
        RunnableParallel(
            {"context": retriever | format_docs, "question": RunnablePassthrough()}
        )
        | prompt
        | llm
        | StrOutputParser()
    )
    return chain

rag_chain = build_chain()

async def stream_answer(question: str) -> AsyncGenerator[str, None]:
    """SSE 流式生成"""
    async for chunk in rag_chain.astream(question):
        if chunk:
            data = json.dumps({"token": chunk, "done": False}, ensure_ascii=False)
            yield f"data: {data}\n\n"
    yield f"data: {json.dumps({'token': '', 'done': True})}\n\n"

@app.post("/ask")
async def ask(req: QueryRequest):
    """非流式问答"""
    answer = await rag_chain.ainvoke(req.question)
    return {"answer": answer}

@app.post("/ask/stream")
async def ask_stream(req: QueryRequest):
    """流式问答 SSE"""
    return StreamingResponse(
        stream_answer(req.question),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )

@app.get("/health")
async def health():
    return {"status": "ok", "chain_ready": rag_chain is not None}

流式优化 Checklist

  • ChatOpenAI(streaming=True) — 开启模型流式
  • .astream() — 使用 LCEL 异步流式方法
  • StreamingResponse(media_type="text/event-stream") — FastAPI SSE 响应
  • X-Accel-Buffering: no — 禁用 Nginx 缓冲区
  • 检索阶段同步等待（不可流式），组装完成后 LLM 部分流式输出

本章结论

  • LCEL 声明式管线：用 | 和 RunnableParallel 组合组件，比回调式更可读、可追踪。
  • Retriever 选型决定上限：Similarity/MMR/Ensemble/MultiQuery 各有适用场景，按 CorpAssist 业务需求选择。
  • Stuff 策略优先：top-5 chunk 5K tokens 以内优先用 Stuff；文档过长才考虑 MapReduce。
  • 流式是生产标配：SSE + async LCEL 实现首 Token < 300ms，显著改善用户体验。
  • 文档切分是前置质量：chunk_size=1024, overlap=200, 保留语义边界直接影响检索效果。

from llama_index.core import (
    SimpleDirectoryReader,
    VectorStoreIndex,
    Settings,
    Document,
)
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.extractors import TitleExtractor
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.core.schema import MetadataMode
import chromadb
from typing import List, Optional

# 全局配置
Settings.embed_model = OpenAIEmbedding(
    model="text-embedding-3-small",
    embed_batch_size=16,
    dimensions=512,
)

def build_ingestion_pipeline(
    chroma_collection_name: str = "corpassist_llama",
    persist_dir: str = "./data/chroma_llama",
) -> IngestionPipeline:
    """构建 IngestionPipeline：读取 → 切分 → 嵌入 → 存储"""
    # Chroma 客户端
    db = chromadb.PersistentClient(path=persist_dir)
    collection = db.get_or_create_collection(
        name=chroma_collection_name,
        metadata={"hnsw:space": "cosine"},
    )
    vector_store = ChromaVectorStore(chroma_collection=collection)

    pipeline = IngestionPipeline(
        transformations=[
            SentenceSplitter(
                chunk_size=1024,
                chunk_overlap=200,
                separator=" ",
                paragraph_separator="\n\n",
                secondary_chunking_regex="[^。！？]+[。！？]?",  # 中文分句
            ),
            TitleExtractor(),
            OpenAIEmbedding(
                model="text-embedding-3-small",
                dimensions=512,
            ),
        ],
        vector_store=vector_store,
    )
    return pipeline

def run_ingestion(doc_dir: str = "./data/docs") -> VectorStoreIndex:
    """执行文档摄取"""
    documents = SimpleDirectoryReader(
        input_dir=doc_dir,
        required_exts=[".pdf", ".md", ".html"],
        recursive=True,
    ).load_data()
    print(f"Loaded {len(documents)} documents")

    pipeline = build_ingestion_pipeline()
    nodes = pipeline.run(documents=documents)
    print(f"Processed into {len(nodes)} nodes")

    # 从 vector store 创建索引
    index = VectorStoreIndex.from_vector_store(
        pipeline.vector_store,
        embed_model=Settings.embed_model,
    )
    return index

if __name__ == "__main__":
    index = run_ingestion("./data/docs")
    print(f"Index ready with {len(index.ref_doc_info)} documents")

Query Engine：检索与生成的组合

LlamaIndex 的 Query Engine 是检索与生成的组合层。它将 retriever、response synthesizer 和 postprocessor 组合为一个可调用的接口。与 LangChain 不同，LlamaIndex 将「检索什么」和「如何生成」分离为独立配置的 RetrieverMode 和 ResponseMode。

组件LangChain 对应LlamaIndex 对应核心差异
检索器 BaseRetriever BaseRetriever / RetrieverMode LlamaIndex 检索器与 Index 强绑定
响应合成 StuffDocumentsChain ResponseSynthesizer + ResponseMode LlamaIndex 5 种 mode：compact/refine/tree/accumulate/compact_accumulate
后处理 cust om callback BaseNodePostprocessor 内置 SimilarityPostprocessor, KeywordPostprocessor

from llama_index.core import VectorStoreIndex, Settings
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.core.response_synthesizers import ResponseMode
from llama_index.core.postprocessor import (

    MetadataReplacementPostProcessor,
)
from llama_index.core.retrievers import VectorIndexRetriever
from llama_index.core.indices.query.query_transform.base import (
    HyDEQueryTransform,
)
from llama_index.core.query_engine import TransformQueryEngine
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding

Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0.1)
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small", dimensions=512)

def create_basic_query_engine(index: VectorStoreIndex) -> RetrieverQueryEngine:
    """基础 Query Engine：similarity 检索 + compact 合成"""
    retriever = VectorIndexRetriever(
        index=index,
        similarity_top_k=5,
    )
    return RetrieverQueryEngine(
        retriever=retriever,
        response_synthesizer=index.as_response_synthesizer(
            response_mode=ResponseMode.COMPACT,
            verbose=True,
        ),
        node_postprocessors=[
            SimilarityPostprocessor(similarity_cutoff=0.7),
        ],
    )

def create_advanced_query_engine(index: VectorStoreIndex) -> RetrieverQueryEngine:
    """进阶 Query Engine：HyDE + tree_summarize"""
    # 基础检索
    base_retriever = VectorIndexRetriever(
        index=index,
        similarity_top_k=10,
    )
    base_engine = RetrieverQueryEngine(
        retriever=base_retriever,
        response_synthesizer=index.as_response_synthesizer(
            response_mode=ResponseMode.TREE_SUMMARIZE,
        ),
        node_postprocessors=[
            SimilarityPostprocessor(similarity_cutoff=0.6),
        ],
    )
    # 使用 HyDE (Hypothetical Document Embeddings) 改进查询
    hyde = HyDEQueryTransform(llm=Settings.llm, include_original=True)
    engine = TransformQueryEngine(base_engine, hyde)
    return engine

def query_corpassist(index: VectorStoreIndex, question: str) -> str:
    """使用 advanced engine 查询"""
    engine = create_advanced_query_engine(index)
    response = engine.query(question)
    return str(response)

Router：多策略查询路由

当 CorpAssist 需要同时处理多种查询类型（如 FAQ 检索、IT 工单查询、政策文档摘要），Router 可以根据问题内容自动选择合适的 Index 或 Query Engine。这是 LlamaIndex 相比 LangChain 更成熟的特性。

python 
# router.py — LlamaIndex Router 实现多策略路由
from llama_index.core import VectorStoreIndex
from llama_index.core.tools import QueryEngineTool, ToolMetadata
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.selectors import LLMSingleSelector, PydanticSingleSelector
from llama_index.llms.openai import OpenAI

def build_router_engine(
    hr_index: VectorStoreIndex,
    it_index: VectorStoreIndex,
    policy_index: VectorStoreIndex,
) -> RouterQueryEngine:
    """构建多策略路由引擎"""
    llm = OpenAI(model="gpt-4o-mini", temperature=0)

    # 每个子索引对应一个工具
    hr_tool = QueryEngineTool(
        query_engine=hr_index.as_query_engine(
            similarity_top_k=3,
            response_mode="compact",
        ),
        metadata=ToolMetadata(
            name="hr_knowledge",
            description="员工手册、休假政策、薪酬福利等人力资源相关问题。",
        ),
    )
    it_tool = QueryEngineTool(
        query_engine=it_index.as_query_engine(
            similarity_top_k=5,
            response_mode="tree_summarize",
        ),
        metadata=ToolMetadata(
            name="it_support",
            description="IT 运维、工单系统、网络故障、设备申请等技术支持问题。",
        ),
    )
    policy_tool = QueryEngineTool(
        query_engine=policy_index.as_query_engine(
            similarity_top_k=3,
            response_mode="compact",
        ),
        metadata=ToolMetadata(
            name="company_policy",
            description="公司制度、合规要求、安全规范、数据保护等政策文档。",
        ),
    )

    router = RouterQueryEngine(
        selector=PydanticSingleSelector.from_defaults(llm=llm),
        query_engine_tools=[hr_tool, it_tool, policy_tool],
        verbose=True,
    )
    return router

本章结论

  • IngestionPipeline 是 ETL 核心：文档解析→切分→嵌入→索引的全流程自动化，支持缓存和增量更新。
  • Query Engine 封装检索与生成：5 种 ResponseMode 适应不同场景，HyDE 变换可提升检索质量。
  • Router 实现多策略路由：按问题类型自动路由到不同索引，LLM/Pydantic 选择器各有优劣。
  • LangChain vs LlamaIndex：索引管理 LlamaIndex 胜出，Agent/Tool 生态 LangChain 更丰富。
  • 两者可互补：生产环境常见组合 LlamaIndex 索引 + LangChain Agent 管线。

为什么要学本章

纯向量检索对术语精确匹配天生弱势——「工单编号 WK-2024-001」在语义空间里没有意义。纯 BM25 关键词检索又无法理解「报销流程」和「费用申请」是同义表达。混合检索通过 RRF（Reciprocal Rank Fusion） 融合两者分数，实现 1+1 > 2 的召回效果。本章从 CorpAssist 工单与知识库场景出发，实现生产级混合检索。

学完你能
  • 实现 BM25 + 向量检索的双通道混合检索管线
  • 使用 RRF 和加权融合两种策略合并排序结果
  • 集成 BGE-Reranker 提升 top-k 排序精度
  • 设计权重调优实验，在延迟与召回之间做出工程权衡
本章先记住 3 件事
  • 稀疏 + 密集 = 互补：BM25 做精确匹配，向量做语义扩展，两者融合覆盖更多查询。
  • RRF 是简单有效的融合：用排名倒数加权，不需要归一化分数尺度。
  • Rerank 是最后的精度保障：交叉编码器重新排序 top-N，精度最高但延迟也最高。

BM25 + 向量：双通道检索原理

混合检索的两条通道各有优劣。BM25 基于词频-逆文档频率，擅长精确关键词匹配；向量检索基于语义嵌入，擅长同义扩展和概念检索。两者融合的关键是如何对齐分数尺度并合并排序。

CorpAssist · 工单与知识库混合检索 员工可能用精确工单号「INC-2024-0513」查询，也可能用模糊描述「上周的网络故障」提问。混合检索需要同时处理两种查询风格。BM25 命中工单号，向量检索匹配语义相关文档，RRF 融合后给出最优排序。

# hybrid_retriever.py — BM25 + 向量混合检索器
from typing import List, Tuple, Optional, Dict
from dataclasses import dataclass
import numpy as np

from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain.retrievers import BM25Retriever, EnsembleRetriever
from langchain.schema import Document

@dataclass
class RetrievalResult:
    """检索结果带分数信息"""
    document: Document
    bm25_score: float = 0.0
    vector_score: float = 0.0
    hybrid_score: float = 0.0
    rerank_score: float = 0.0

class HybridRetriever:
    """BM25 + 向量双通道混合检索器"""

    def __init__(
        self,
        vector_store: Chroma,
        bm25_retriever: BM25Retriever,
        alpha: float = 0.5,
    ):
        self.vector_store = vector_store
        self.bm25_retriever = bm25_retriever
        self.alpha = alpha  # BM25 权重: alpha, 向量权重: 1-alpha

    def retrieve(
        self,
        query: str,
        k: int = 10,
        bm25_k: int = 20,
        vector_k: int = 20,
    ) -> List[RetrievalResult]:
        """双通道检索 → RRF 融合"""
        # 通道 1: BM25 检索
        bm25_docs = self.bm25_retriever.invoke(query)
        bm25_scores_map = {}
        for rank, doc in enumerate(bm25_docs):
            doc_id = doc.metadata.get("doc_id", doc.page_content[:50])
            bm25_scores_map[doc_id] = 1.0 / (rank + 1)

        # 通道 2: 向量检索
        vector_results = self.vector_store.similarity_search_with_score(
            query, k=vector_k
        )
        vector_scores_map = {}
        for doc, score in vector_results:
            doc_id = doc.metadata.get("doc_id", doc.page_content[:50])
            vector_scores_map[doc_id] = float(score)

        # RRF 融合
        all_doc_ids = set(bm25_scores_map.keys()) | set(vector_scores_map.keys())
        results = []

        for doc_id in all_doc_ids:
            bm25_score = bm25_scores_map.get(doc_id, 0.0)
            vector_score = vector_scores_map.get(doc_id, 0.0)
            # 加权融合
            hybrid_score = self.alpha * bm25_score + (1 - self.alpha) * vector_score
            # 找到对应的文档对象
            doc = self._find_doc(doc_id, bm25_docs, vector_results)
            if doc:
                results.append(RetrievalResult(
                    document=doc,
                    bm25_score=bm25_score,
                    vector_score=vector_score,
                    hybrid_score=hybrid_score,
                ))

        # 按混合分数排序
        results.sort(key=lambda r: r.hybrid_score, reverse=True)
        return results[:k]

    def _find_doc(
        self,
        doc_id: str,
        bm25_docs: List[Document],
        vector_results: List[Tuple[Document, float]],
    ) -> Optional[Document]:
        for doc in bm25_docs:
            if doc.metadata.get("doc_id", doc.page_content[:50]) == doc_id:
                return doc
        for doc, _ in vector_results:
            if doc.metadata.get("doc_id", doc.page_content[:50]) == doc_id:
                return doc
        return None

RRF 算法实现

python 
# rrf_fusion.py — RRF 融合算法与对比
from typing import List, Dict, Tuple
from collections import defaultdict
import math

def rrf_fusion(
    rankings: List[List[str]],
    k: int = 60,
    weights: List[float] = None,
) -> List[Tuple[str, float]]:
    """
    Reciprocal Rank Fusion 融合多个排序结果。

    Args:
        rankings: 多个排序列表，每个列表为文档 ID 序列
        k: RRF 常数，通常 60
        weights: 每个排序通道的权重，默认等权重

    Returns:
        (doc_id, score) 按分数降序排列
    """
    if weights is None:
        weights = [1.0] * len(rankings)
    assert len(rankings) == len(weights), "rankings 和 weights 长度必须相同"

    scores = defaultdict(float)
    for rank_list, weight in zip(rankings, weights):
        for rank, doc_id in enumerate(rank_list):
            # RRF 分数: weight / (k + rank)
            scores[doc_id] += weight / (k + rank + 1)

    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

def linear_fusion(
    rankings: List[List[str]],
    scores_per_rank: List[List[float]],
    weights: List[float] = None,
) -> List[Tuple[str, float]]:
    """线性归一化融合（需要原始分数）"""
    if weights is None:
        weights = [1.0] * len(rankings)

    fused = defaultdict(float)
    for rank_list, score_list, weight in zip(rankings, scores_per_rank, weights):
        if not score_list:
            continue
        min_s, max_s = min(score_list), max(score_list)
        if max_s - min_s < 1e-6:
            norm_scores = [0.5] * len(score_list)
        else:
            norm_scores = [(s - min_s) / (max_s - min_s) for s in score_list]
        for doc_id, ns in zip(rank_list, norm_scores):
            fused[doc_id] += weight * ns

    return sorted(fused.items(), key=lambda x: x[1], reverse=True)

不推荐
  • 直接拼接 BM25 和向量结果而不去重
  • 用原始 L2 距离和 BM25 分数直接相加
  • 固定 alpha=0.5 不做场景调整

  • RRF 融合用排名倒数加权，免去分数归一化
  • 线性融合前做 min-max 归一化对齐尺度
  • 按场景调优 alpha（精确查询 alpha 高，模糊查询 alpha 低）
  • 设置 score_threshold 过滤明显不相关结果

权重调优与延迟权衡

alpha 参数控制 BM25 和向量检索的权重比例。alpha=1 是纯关键词，alpha=0 是纯语义。最优 alpha 取决于查询分布和文档特性。同时，双通道检索意味着双倍延迟，需要工程权衡。

Alpha 扫描实验

# alpha_sweep.py — 权重扫描实验
from typing import List, Dict
import time
import json
from hybrid_retriever import HybridRetriever
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain.retrievers import BM25Retriever
from langchain.schema import Document

def evaluate_recall(
    retriever: HybridRetriever,
    test_queries: List[Dict[str, List[str]]],
    k: int = 10,
) -> Dict[str, float]:
    """评估检索召回率"""
    total_relevant = 0
    retrieved_relevant = 0

    for item in test_queries:
        query = item["query"]
        expected_docs = set(item["expected_doc_ids"])
        results = retriever.retrieve(query, k=k)
        retrieved_ids = {r.document.metadata.get("doc_id", "") for r in results}
        matches = retrieved_ids & expected_docs
        retrieved_relevant += len(matches)
        total_relevant += len(expected_docs)

    recall = retrieved_relevant / total_relevant if total_relevant > 0 else 0
    return {"recall@k": recall, "relevant": retrieved_relevant, "total": total_relevant}

def run_alpha_sweep(
    vector_store: Chroma,
    bm25_retriever: BM25Retriever,
    test_queries: List[Dict[str, List[str]]],
) -> Dict[float, Dict]:
    """扫描 alpha 值并记录 Recall@K 和延迟"""
    results = {}

    for alpha in [0.0, 0.2, 0.4, 0.5, 0.6, 0.8, 1.0]:
        retriever = HybridRetriever(vector_store, bm25_retriever, alpha=alpha)

        # 计时
        start = time.perf_counter()
        metrics = evaluate_recall(retriever, test_queries, k=10)
        elapsed = time.perf_counter() - start

        metrics["avg_latency_ms"] = (elapsed / len(test_queries)) * 1000
        results[alpha] = metrics
        print(f"alpha={alpha:.1f}: recall@10={metrics['recall@k']:.4f}, "
              f"latency={metrics['avg_latency_ms']:.1f}ms")

    return results

延迟优化策略

融合策略优点缺点适用场景
RRF (k=60) 无需分数归一化，鲁棒 忽略分数绝对值信息 多通道融合、分数不可比场景
加权线性 保留分数信息，可解释 需归一化到同一尺度 分数分布已知的可比场景
学习排序 (LTR) 可学出最优权重组合 需要标注数据 有历史点击/反馈数据
策略效果代价
降低 recall 的 fetch_k 减少 BM25 和向量检索规模 可能丢失相关文档
并行执行双通道 总延迟 = max(bm25, vec) 而非 sum 需 ThreadPoolExecutor
向量量化 (PQ) 向量检索提速 3-5x 召回下降 1-3%
异步检索 释放线程处理其他请求 框架支持复杂度
先想 10 秒：双通道检索中如果 BM25 返回了 100 个结果，向量返回 100 个，RRF 融合后你取 top-10。那么 BM25 的 k 值应该设为多少？

建议 BM25 的 k 设为最终 top-k 的 2-4 倍（即 20-40）。k 太小可能提前截断本应出现在最终结果中的文档；k 太大徒增 RRF 计算开销。原则：各通道的候选集应覆盖足够多的候选，再由融合排序决定 top-k。

Rerank 集成：BGE-Reranker 精排

Rerank 是混合检索的最后一道精度保障。BM25 和向量检索都使用双编码器（bi-encoder）做近似搜索，速度高但精度有限。Reranker 使用交叉编码器（cross-encoder）对 query 和候选文档逐对计算相关性分数，精度更高但延迟更高。

python 
# reranker.py — BGE-Reranker 集成
from typing import List, Tuple
from dataclasses import dataclass
import numpy as np
from sentence_transformers import CrossEncoder
from hybrid_retriever import HybridRetriever, RetrievalResult

class BGERReranker:
    """BGE Reranker 交叉编码器精排"""

    def __init__(
        self,
        model_name: str = "BAAI/bge-reranker-v2-m3",
        batch_size: int = 16,
        device: str = "cpu",
    ):
        self.model = CrossEncoder(
            model_name,
            max_length=512,
            device=device,
        )
        self.batch_size = batch_size

    def rerank(
        self,
        query: str,
        candidates: List[RetrievalResult],
        top_k: int = 5,
    ) -> List[RetrievalResult]:
        """对候选结果重新排序"""
        if not candidates:
            return []

        pairs = [(query, r.document.page_content) for r in candidates]
        scores = self.model.predict(pairs, batch_size=self.batch_size)

        for result, score in zip(candidates, scores):
            result.rerank_score = float(score)

        candidates.sort(key=lambda r: r.rerank_score, reverse=True)
        return candidates[:top_k]

class PipelineWithRerank:
    """完整管线：混合检索 → Rerank"""

    def __init__(
        self,
        hybrid_retriever: HybridRetriever,
        reranker: BGERReranker,
        hybrid_k: int = 20,   # 混合检索取 top-20
        rerank_k: int = 5,    # Rerank 后取 top-5
    ):
        self.hybrid_retriever = hybrid_retriever
        self.reranker = reranker
        self.hybrid_k = hybrid_k
        self.rerank_k = rerank_k

    def retrieve(self, query: str) -> List[RetrievalResult]:
        """完整检索管线"""
        # 阶段 1: 混合检索 (快速)
        candidates = self.hybrid_retriever.retrieve(query, k=self.hybrid_k)
        # 阶段 2: Rerank (慢速但精准)
        reranked = self.reranker.rerank(query, candidates, top_k=self.rerank_k)
        return reranked

如果延迟敏感（如工单实时查询），考虑跳过 Rerank 或降低候选集大小。

CorpAssist 场景：Elasticsearch + Milvus 双后端

生产环境中，BM25 由 Elasticsearch 提供，向量检索由 Milvus 提供。双后端架构的挑战在于：如何保持两端数据一致、如何统一查询接口、如何处理一端故障。

后端引擎擅长索引结构
稀疏检索Elasticsearch 8.x精确关键词、通配符、短语匹配倒排索引 (Inverted Index)
密集检索Milvus 2.x语义相似、同义扩展、模糊概念IVF_FLAT / HNSW
双后端注意点
  • 文档 ID 在两系统间保持一致，用全局唯一 doc_id 关联
  • 写操作双写：ES 和 Milvus 都要写入，使用事务性消息队列保证最终一致
  • 读操作优先读 Milvus（向量检索）降级到 ES（仅 BM25），反过来亦可
  • 监控两端延迟 P50/P99，设置熔断阈值

本章结论

  • BM25 + 向量互补：精确匹配与语义扩展结合，覆盖更多查询类型。
  • RRF 融合简单有效：无参数、免归一化、多通道排序可解释。
  • Alpha 需要场景调优：精确查询 alpha 偏高，语义查询 alpha 偏低。
  • Rerank 是精度兜底：交叉编码器重排 top-N，精度最高但延迟最高。
  • 双后端架构：Elasticsearch + Milvus 各自服务不同检索类型，通过 doc_id 关联。

from typing import List, Dict
from dataclasses import dataclass, field
from datasets import Dataset
import json
import random

@dataclass
class EvalSample:
    """评测样本结构"""
    question: str
    ground_truth: str
    contexts: List[str]  # 相关文档 ID 列表
    category: str = "general"  # hr/it/policy
    difficulty: str = "normal"  # easy/normal/hard

@dataclass
class EvalDataset:
    """评测数据集"""
    samples: List[EvalSample] = field(default_factory=list)

    def to_hf_dataset(self, docs_map: Dict[str, str]) -> Dataset:
        """转换为 HuggingFace Dataset 供 RAGAS 使用"""
        data = {
            "question": [],
            "ground_truth": [],
            "contexts": [],
            "category": [],
        }
        for sample in self.samples:
            data["question"].append(sample.question)
            data["ground_truth"].append(sample.ground_truth)
            data["contexts"].append([
                docs_map.get(doc_id, doc_id) for doc_id in sample.contexts
            ])
            data["category"].append(sample.category)
        return Dataset.from_dict(data)

    def save(self, path: str):
        """保存为 JSONL"""
        with open(path, "w", encoding="utf-8") as f:
            for sample in self.samples:
                f.write(json.dumps(sample.__dict__, ensure_ascii=False) + "\n")

    @classmethod
    def load(cls, path: str) -> "EvalDataset":
        """从 JSONL 加载"""
        samples = []
        with open(path, "r", encoding="utf-8") as f:
            for line in f:
                data = json.loads(line)
                samples.append(EvalSample(**data))
        return cls(samples=samples)

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_chroma import Chroma
from typing import List, Dict
import json
import re

SYNTH_TEMPLATE = """你是一个 RAG 评测数据生成专家。
基于以下文档内容，生成一个问答对（question 和 ground_truth）。
要求：
1. 问题必须忠实于文档内容
2. 答案必须是文档中的完整事实，不要添加外部知识
3. 生成的问题难度应为 {difficulty}
4. 问题应该能被 RAG 系统合理回答

文档内容：
{document}

输出格式（JSON）：
{{"question": "...", "ground_truth": "..."}}"""

def generate_synthetic_samples(
    docs: List[str],
    n_per_doc: int = 3,
    difficulty: str = "normal",
    model: str = "gpt-4o-mini",
) -> List[Dict]:
    """LLM 生成合成评测样本"""
    llm = ChatOpenAI(model=model, temperature=0.7)
    prompt = ChatPromptTemplate.from_template(SYNTH_TEMPLATE)
    chain = prompt | llm

    samples = []
    for doc_text in docs:
        for _ in range(n_per_doc):
            try:
                response = chain.invoke({
                    "document": doc_text[:2000],
                    "difficulty": difficulty,
                })
                content = response.content.strip()
                # 提取 JSON
                json_match = re.search(r'\{.*\}', content, re.DOTALL)
                if json_match:
                    sample = json.loads(json_match.group())
                    samples.append(sample)
            except Exception as e:
                print(f"Error generating sample: {e}")
                continue

    return samples

RAGAS 指标：Faithfulness, Relevancy, Precision, Recall

RAGAS 是 RAG 评测的标准框架，核心指标覆盖 RAG 系统的两个关键环节：检索质量和生成质量。

指标测量对象范围说明
Faithfulness 生成答案 ⭢ 上下文 [0, 1] 答案中的每个声明是否都有上下文支持
Answer Relevancy 生成答案 ⭢ 问题 [0, 1] 答案是否完整且直接回答了问题
Context Precision 检索上下文 ⭢ 问题 [0, 1] 检索到的上下文中相关片段的比例
Context Recall 检索上下文 ⭢ ground_truth [0, 1] 基准答案所需的信息是否都在上下文中

# ragas_eval.py — RAGAS 评测管线
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from ragas.llms import LangchainLLMWrapper
from ragas.embeddings import LangchainEmbeddingsWrapper
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from typing import List, Dict, Optional
import json

class RAGEvaluator:
    """RAGAS 评测器"""

    def __init__(
        self,
        llm_model: str = "gpt-4o-mini",
        embedding_model: str = "text-embedding-3-small",
    ):
        # 包装 RAGAS 需要的 LLM 和 Embedding
        self.evaluator_llm = LangchainLLMWrapper(
            ChatOpenAI(model=llm_model, temperature=0)
        )
        self.evaluator_emb = LangchainEmbeddingsWrapper(
            OpenAIEmbeddings(model=embedding_model)
        )
        # 配置指标
        self.metrics = [
            faithfulness,
            answer_relevancy,
            context_precision,
            context_recall,
        ]
        for metric in self.metrics:
            metric.__setattr__("llm", self.evaluator_llm)
            if hasattr(metric, "embeddings"):
                metric.__setattr__("embeddings", self.evaluator_emb)

    def evaluate(
        self,
        dataset: Dataset,
        column_map: Optional[Dict] = None,
    ) -> Dict[str, float]:
        """运行完整评测"""
        if column_map is None:
            column_map = {
                "question": "question",
                "ground_truth": "ground_truth",
                "contexts": "contexts",
            }
        result = evaluate(
            dataset=dataset,
            metrics=self.metrics,
            column_map=column_map,
        )
        return dict(result)

    def print_report(self, scores: Dict[str, float]):
        """打印评测报告"""
        print("=" * 50)
        print("RAG 评测报告")
        print("=" * 50)
        for metric, score in scores.items():
            status = "PASS" if score >= 0.7 else "FAIL"
            bar = "█" * int(score * 20) + "░" * (20 - int(score * 20))
            print(f"{metric:25s} {bar} {score:.4f} [{status}]")
        print(f"\n平均得分: {sum(scores.values())/len(scores):.4f}")

    scores = evaluator.evaluate(eval_dataset_with_answers)
    evaluator.print_report(scores)

    # 4. 检查是否通过
    all_pass = True
    for metric, min_score in MIN_SCORES.items():
        if metric in scores and scores[metric] < min_score:
            print(f"FAIL: {metric} = {scores[metric]:.4f} < {min_score}")
            all_pass = False

    # 5. 保存报告
    report = {
        "scores": scores,
        "passed": all_pass,
        "thresholds": MIN_SCORES,
    }
    with open("./data/eval_report.json", "w") as f:
        json.dump(report, f, indent=2, ensure_ascii=False)

    return all_pass

if __name__ == "__main__":
    passed = run_ci_eval()
    sys.exit(0 if passed else 1)

Ablation 实验：参数影响量化

Ablation（消融实验） 通过逐一改变参数观察指标变化，找出对 RAG 质量影响最大的配置项。这是从「凭感觉调参」到「数据驱动调参」的关键手段。

# ablation.py — 消融实验框架
from itertools import product
from typing import List, Dict, Any
import json
import time
from dataclasses import dataclass, field

@dataclass
class AblationExperiment:
    """消融实验定义"""
    name: str
    chunk_sizes: List[int] = field(default_factory=lambda: [256, 512, 1024, 2048])
    top_ks: List[int] = field(default_factory=lambda: [3, 5, 10])
    retrievers: List[str] = field(default_factory=lambda: ["similarity", "mmr", "ensemble"])
    use_reranker: List[bool] = field(default_factory=lambda: [False, True])
    alphas: List[float] = field(default_factory=lambda: [0.3, 0.5, 0.7])

    def get_combinations(self) -> List[Dict[str, Any]]:
        """获取所有参数组合"""
        params = {
            "chunk_size": self.chunk_sizes,
            "top_k": self.top_ks,
            "retriever": self.retrievers,
            "use_reranker": self.use_reranker,
            "alpha": self.alphas,
        }
        keys = list(params.keys())
        values = list(params.values())
        combinations = []
        for combo in product(*values):
            combinations.append(dict(zip(keys, combo)))
        return combinations

def run_ablation_suite(

    eval_dataset_path: str = "./data/eval_dataset.jsonl",
) -> List[Dict]:

    combinations = experiment.get_combinations()

    print(f"Running {total} ablation combinations...")

    from build_dataset import EvalDataset

    eval_dataset = EvalDataset.load(eval_dataset_path)

    for idx, params in enumerate(combinations):
        start = time.perf_counter()
        print(f"\n[{idx+1}/{total}] Params: {params}")

        # 这里应调用实际 RAG 管线获取结果
        # 模拟评测分数
        import random

            "context_recall": 0.7 + random.uniform(-0.1, 0.1),
        }
        elapsed = time.perf_counter() - start

        result = {
            "params": params,
            "scores": scores,
            "avg_score": sum(scores.values()) / len(scores),
            "latency_s": round(elapsed, 2),
        }
        results.append(result)

        print(f"  avg_score={result['avg_score']:.4f}, latency={result['latency_s']:.2f}s")

    # 按平均分排序
    results.sort(key=lambda r: r["avg_score"], reverse=True)
    return results

常见问题（FAQ）

RAGAS 评测需要调用 LLM，每次评测的成本有多高？
50 个样本的完整评测约需 5-10 美元（gpt-4o-mini）。如果做 ablation 实验（如 20 种参数组合），成本升至 100-200 美元。建议：1) 先用 10 个样本做快速验证；2) 小参数量扫描先用模拟数据；3) 关键对比用完整评测集。
Faithfulness 分数低通常是什么原因？
最常见原因是 LLM 在上下文中找不到答案时「发挥」了。解决方案：1) 优化 prompt 强制要求「如果上下文无信息，请说不知道」；2) 提高检索质量，确保上下文包含答案；3) 降低 temperature（如 0.01）。
Context Precision 和 Context Recall 的区别是什么？
Precision 衡量「检索到的内容有多少是相关的」— 越低说明噪声越多。Recall 衡量「所有相关内容有多少被检索到」— 越低说明遗漏越多。理想系统两者都高，但实际需要在它们之间做权衡。
Ablation 实验中参数组合爆炸怎么办？
全量组合可能很快变成数百种。策略：1) 先用单因素实验（只变一个参数，其他固定）找到敏感参数；2) 对敏感参数做多因素实验；3) 使用 Plackett-Burman 或正交实验设计减少组合数。

判断练习

  1. Ablation 实验显示 chunk_size=2048 时 Faithfulness 最高。你会选择这个参数吗？

    参考答案

    不一定。chunk_size=2048 虽然 Faithfulness 高（LLM 看到更多上下文），但可能增加检索噪声，降低 Precision，并因为 token 过多增加成本和延迟。需要综合看：Avg Score + 延迟 + token 成本。如果 2048 只比 1024 高 0.01，但 token 成本翻倍，选择 1024 更合理。

  2. 评测报告显示 Context Recall=1.0 但 Context Precision=0.3。这是好还是坏？

    参考答案

    坏。Recall=1.0 说明所有相关信息都被检索到了，但 Precision=0.3 意味着 70% 的检索结果是噪声。这会稀释 LLM 的注意力（Lost in the Middle 问题）。需要提高 score_threshold 或增加相关性过滤。

先想 10 秒：GraphRAG 的实体提取阶段为什么要设 max_gleanings？

LLM 一次提取可能遗漏部分实体。max_gleanings 让 LLM 检查已提取结果并补充遗漏。每轮 gleaning 增加一次 LLM 调用。设 2 表示「提取→检查→补充」三轮，在质量和成本间平衡。

社区检测与全局/局部搜索

GraphRAG 使用 Leiden 算法 对实体图进行社区检测。社区是高内聚的实体子图，使得同一个社区内的实体关联紧密。GraphRAG 为每个社区生成一份自然语言摘要，用于支持全局搜索。

GraphRAG 的核心成本在索引构建阶段。每篇文档需要多次 LLM 调用（实体提取 + 关系识别 + gleaning + 社区摘要）。对于 50 篇文档约 200 页，索引构建成本约 $10-30，而朴素 RAG 的索引成本几乎为零。

CorpAssist 选型建议

  • 战略报告（"公司各部门的核心能力分布"）→ GraphRAG Global Search
  • 合规审计（"所有涉及数据外泄的相关方"）→ GraphRAG + 图遍历

常见问题（FAQ）

GraphRAG 必须使用 Microsoft 的那个开源实现吗？
不是必须的。Microsoft GraphRAG 是目前最成熟的方案，但不是唯一选择。你也可以用 Neo4j + LangChain 手动构建图谱，或者使用 LlamaIndex 的 KnowledgeGraphIndex。Microsoft 方案的优点是社区检测和摘要生成开箱即用。
GraphRAG 索引构建太慢了，能增量更新吗？
GraphRAG 目前不支持真正的增量索引。新增文档需要重新运行完整索引管线。社区中有一些 workaround：为不同时间段的文档构建独立索引，查询时合并结果。长期来看，GraphRAG 的增量索引是个活跃开发方向。
我的文档量很小（10 篇），有必要用 GraphRAG 吗？
文档量小的情况下，朴素 RAG 加上一些手写的关系映射可能就够用了。但如果实体关联密集（如组织架构、项目依赖），GraphRAG 的社区摘要仍然能提供更好的全局视角。建议 50+ 篇文档时考虑 GraphRAG。
GraphRAG 的社区摘要是怎么保证质量的？
社区摘要的质量取决于 1) 实体提取的完整性；2) 社区检测的粒度（Leiden 算法的分辨率参数）；3) 摘要 prompt 的设计。可以通过调整 resolution 参数控制社区数量（resolution 越高社区越多越细），在精度和召回间平衡。

本章结论

  • 实体提取是基础：LLM 驱动的实体/关系/协变量提取质量直接决定图谱质量。
  • 社区检测自动聚合：Leiden 算法发现实体高内聚子图，支持全局视角的问答。
  • 全局搜索 vs 局部搜索：抽象问题用 Global（社区摘要聚合），具体问题用 Local（实体邻域）。
  • 成本较高但适合高价值场景：索引成本在 $10-30 级别，适用于多跳推理和全局总结场景。
  • 不是朴素 RAG 的替代品：两者互补——日常 Q&A 用朴素 RAG，分析推理用 GraphRAG。

为什么要学本章

RAG 系统最危险的失败模式不是「答错」，而是「看起来正确的编造」。Grounding（接地/归因）确保生成答案的每个关键声明都可追溯到原始文档。对于 CorpAssist 这样的企业级应用，合规审计要求每个回答都能提供出处。Citation（引用） 溯源是 RAG 从「好玩」到「可信」的必修课。

学完你能
  • 实现从文档 chunk → 答案段落的引用映射机制
  • 构建 groundedness 评分器，自动检测答案中的非归因声明
  • 设计多种 Citation 格式（Markdown 脚注、JSON 引用、行内标注）
  • 建立引用验证验收测试，防止幻觉引用通过 CI
本章先记住 3 件事
  • 每句话都要有来源：答案中的关键事实声明必须链回到具体的 document chunk。
  • Groundedness != Faithfulness：Faithfulness 是「不矛盾」，Groundedness 是「有出处」。
  • 引用也要防幻觉：LLM 可能编造引用，需要用验证器至少检查引用格式和 chunk 存在性。

# citation_pipeline.py — 引用溯源管线
from dataclasses import dataclass, field
from typing import List, Dict, Optional
import json
import re
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough

@dataclass
class CitationSource:
    """单个引用的源信息"""
    chunk_id: str
    document_name: str
    page_number: Optional[int] = None
    text_snippet: str = ""
    relevance_score: float = 0.0

@dataclass
class CitationResult:
    """带引用的答案"""
    answer: str
    citations: List[CitationSource] = field(default_factory=list)

CITATION_PROMPT = """你是 CorpAssist 企业知识库助手。

基于以下文档片段回答问题。在回答中，对每个关键事实用 [N] 标注来源编号。
N 对应文档片段列表中片段的序号（从 1 开始）。

文档片段：
{context}

要求：
1. 每个关键事实都要标注来源
2. 不要标注没有文档支撑的信息
3. 如果不确定，请说「需要查证」

问题：{question}
回答："""

class CitationRAGPipeline:
    """带引用溯源的 RAG 管线"""

    def __init__(
        self,
        persist_dir: str = "./data/chroma_db",
        embedding_model: str = "text-embedding-3-small",
        llm_model: str = "gpt-4o-mini",
    ):
        embeddings = OpenAIEmbeddings(model=embedding_model, dimensions=512)
        self.vector_store = Chroma(
            persist_directory=persist_dir,
            embedding_function=embeddings,
        )
        self.retriever = self.vector_store.as_retriever(
            search_type="similarity",
            search_kwargs={"k": 5},
        )
        self.llm = ChatOpenAI(model=llm_model, temperature=0.1)
        self.prompt = ChatPromptTemplate.from_template(CITATION_PROMPT)

    def build_citation_chain(self):
        """构建 LCEL 引用管线"""
        def format_with_index(docs):
            """给文档添加序号用于引用"""
            return "\n\n".join(
                f"[{i+1}] {d.page_content}"
                for i, d in enumerate(docs)
            )

        chain = (
            RunnableParallel(
                {
                    "context": self.retriever | format_with_index,
                    "question": RunnablePassthrough(),
                }
            )
            | self.prompt
            | self.llm
            | StrOutputParser()
        )
        return chain

    def ask_with_citations(self, question: str) -> CitationResult:
        """问答并返回带引用信息的结果"""
        # 1. 检索
        docs = self.retriever.invoke(question)

        # 2. 生成带引用的答案
        chain = self.build_citation_chain()
        answer = chain.invoke(question)

        # 3. 解析引用标记并关联源文档
        citations = []
        ref_pattern = r'\[(\d+)\]'
        for match in re.finditer(ref_pattern, answer):
            idx = int(match.group(1)) - 1
            if 0 <= idx < len(docs):
                doc = docs[idx]
                citations.append(CitationSource(
                    chunk_id=doc.metadata.get("chunk_id", f"chunk_{idx}"),
                    document_name=doc.metadata.get("source", "unknown"),
                    page_number=doc.metadata.get("page", None),
                    text_snippet=doc.page_content[:200],
                    relevance_score=getattr(doc, "score", 0.0),
                ))

        return CitationResult(answer=answer, citations=citations)

不推荐
  • 不标注来源，答案不可追溯
  • 引用只是装饰性的 [1][2]，实际无对应 chunk
  • 引用格式不统一，有时用脚注有时用括号
  • 不校验引用的真实性——LLM 自己编引用编号
推荐
  • 每个 [N] 精确对应检索到的 chunk
  • 引用附带 source/document/page 元数据
  • 统一的引用格式（如 [doc_id:chunk_index]）
  • 使用引用验证器检查每个引用是否有效

置信度标注与 Groundedness 评分

Groundedness（归因度） 衡量答案中的声明有多少可以追溯到源文档。与上一章讲到的 RAGAS Faithfulness 不同，Groundedness 更严格：Faithfulness 要求「不矛盾」，Groundedness 要求「有来源」。

概念定义测量方式
Faithfulness 答案不含有与上下文矛盾的声明 LLM 判断（声明 → 上下文校验）
Groundedness 答案关键声明可直接追溯到指定源 chunk LLM 判断（声明 → 具体 chunk 匹配）
Citation Accuracy 引用标注 [N] 确实指向正确的 chunk 精确匹配 + 人工校验

# groundedness.py — Groundedness 评分器
from typing import List, Dict, Tuple
from dataclasses import dataclass
import re
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

@dataclass
class GroundednessScore:
    """归因评分结果"""
    overall_score: float  # 0.0 - 1.0
    total_claims: int
    grounded_claims: int
    ungrounded_claims: List[str]
    citation_errors: List[str]

CLAIM_EXTRACTION_PROMPT = """提取以下答案中的所有事实声明。
每个声明应该是独立的、可验证的陈述。
以 JSON 数组格式输出：[{"claim": "..."}]

答案：{answer}
"""

GROUNDEDNESS_CHECK_PROMPT = """判断以下声明是否可以被源文档片段支持。

声明：{claim}

源文档片段：
{context}

如果声明完全被源文档支持 -> SUPPORTED
如果声明部分被支持或有细微差异 -> PARTIALLY_SUPPORTED
如果声明与源文档矛盾或源文档无相关信息 -> NOT_SUPPORTED

只输出一个词：SUPPORTED / PARTIALLY_SUPPORTED / NOT_SUPPORTED"""

class GroundednessChecker:
    """归因度检查器"""

    def __init__(self, model: str = "gpt-4o-mini"):
        self.llm = ChatOpenAI(model=model, temperature=0)
        self.extraction_prompt = ChatPromptTemplate.from_template(CLAIM_EXTRACTION_PROMPT)
        self.check_prompt = ChatPromptTemplate.from_template(GROUNDEDNESS_CHECK_PROMPT)

    def extract_claims(self, answer: str) -> List[str]:
        """从答案中提取独立声明"""
        response = self.llm.invoke(
            self.extraction_prompt.format(answer=answer)
        )
        try:
            import json
            claims_data = json.loads(response.content)
            return [item["claim"] for item in claims_data]
        except (json.JSONDecodeError, KeyError):
            # Fallback: 按句子分割
            return [s.strip() for s in re.split(r'[。！？]', answer) if len(s.strip()) > 10]

    def check_claim(self, claim: str, context: str) -> str:
        """检查单个声明的归因状态"""
        response = self.llm.invoke(
            self.check_prompt.format(claim=claim, context=context)
        )
        return response.content.strip()

    def score(
        self, answer: str, context_chunks: List[str]
    ) -> GroundednessScore:
        """完整归因评分"""
        context = "\n\n".join(context_chunks)
        claims = self.extract_claims(answer)
        print(f"Extracted {len(claims)} claims from answer")

        grounded = []
        ungrounded = []
        citation_errors = []

        for claim in claims:
            status = self.check_claim(claim, context)
            if status == "SUPPORTED":
                grounded.append(claim)
            elif status == "PARTIALLY_SUPPORTED":
                grounded.append(claim)  # 部分支持也算通过
            else:
                ungrounded.append(claim)

        # 检查引用标注
        citation_refs = re.findall(r'\[(\d+)\]', answer)
        for ref in citation_refs:
            idx = int(ref)
            if idx < 1 or idx > len(context_chunks):
                citation_errors.append(f"引用 [{idx}] 超出范围 (1-{len(context_chunks)})")

        total = len(claims)
        grounded_count = len(grounded)
        overall = grounded_count / total if total > 0 else 1.0

        return GroundednessScore(
            overall_score=overall,
            total_claims=total,
            grounded_claims=grounded_count,
            ungrounded_claims=ungrounded,
            citation_errors=citation_errors,
        )

if __name__ == "__main__":
    checker = GroundednessChecker()
    answer = "正式员工年假为15天[1]。需提前3个工作日申请[1]。病假需提供医院证明[2]。"
    contexts = [
        "正式员工每年享有15天带薪年假。年假需提前3个工作日申请。",
        "病假需提供医院开具的诊断证明。",
    ]
    result = checker.score(answer, contexts)
    print(f"Groundedness Score: {result.overall_score:.2f}")
    print(f"Claims: {result.grounded_claims}/{result.total_claims} grounded")
    if result.citation_errors:
        print(f"Citation Errors: {result.citation_errors}")
    if result.ungrounded_claims:
        print(f"Ungrounded Claims: {result.ungrounded_claims}")

# confidence_tagging.py — 置信度标注与展示
from dataclasses import dataclass
from typing import List, Optional, Literal

ConfidenceLevel = Literal["high", "medium", "low", "unverifiable"]

@dataclass
class TaggedSegment:
    """带置信度标注的答案段落"""
    text: str
    confidence: ConfidenceLevel
    sources: List[str]  # 来源 chunk ID
    reason: str = ""  # 标注理由

class ConfidenceTagger:
    """答案段落置信度标注器"""

    def __init__(self, groundedness_checker=None):
        self.checker = groundedness_checker

    def tag_answer_segments(
        self, answer: str, source_chunks: List[str]
    ) -> List[TaggedSegment]:
        """将答案分割为段落并标注置信度"""
        segments = []
        # 按引用标记分割
        import re
        parts = re.split(r'(\[\d+\])', answer)
        current_text = ""

        for part in parts:
            if re.match(r'\[\d+\]', part):
                if current_text.strip():
                    confidence = self._infer_confidence(
                        current_text, source_chunks
                    )
                    segments.append(TaggedSegment(
                        text=current_text.strip(),
                        confidence=confidence,
                        sources=[part],
                        reason=self._confidence_reason(confidence),
                    ))
                current_text = ""
            else:
                current_text += part

        if current_text.strip():
            confidence = self._infer_confidence(current_text, source_chunks)
            segments.append(TaggedSegment(
                text=current_text.strip(),
                confidence=confidence,
                sources=[],
                reason=self._confidence_reason(confidence),
            ))

        return segments

    def _infer_confidence(
        self, text: str, chunks: List[str]
    ) -> ConfidenceLevel:
        """推断段落置信度"""
        if not chunks:
            return "unverifiable"

        # 核心规则：有引用且有匹配文本的为 high
        for chunk in chunks:
            if any(word in chunk for word in text.split("，")[0].split("为")[:3]):
                return "high"

        # 默认 medium
        return "medium"

    def _confidence_reason(self, level: ConfidenceLevel) -> str:
        reasons = {
            "high": "有源文档直接支持",
            "medium": "有源文档部分支持",
            "low": "源文档支持不足",
            "unverifiable": "无法找到对应源文档",
        }
        return reasons.get(level, "")

    def format_confident_answer(
        self, segments: List[TaggedSegment]
    ) -> str:
        """格式化为带置信度标记的展示文本"""
        output = []
        for seg in segments:
            badge = {
                "high": "🟢",
                "medium": "🟡",
                "low": "🟠",
                "unverifiable": "🔴",
            }.get(seg.confidence, "⚪")
            output.append(f"{badge} {seg.text} [{seg.confidence}]")
            output.append(f"  来源: {', '.join(seg.sources) if seg.sources else '无'}")
        return "\n".join(output)

if __name__ == "__main__":
    tagger = ConfidenceTagger()
    segments = tagger.tag_answer_segments(
        "正式员工年假为15天[1]。需提前3个工作日申请[1]。",
        ["正式员工每年享有15天带薪年假。年假需提前3个工作日申请。"],
    )
    print(tagger.format_confident_answer(segments))

# citation_validator.py — 引用验证器与验收测试
from typing import List, Dict, Optional, Tuple
from dataclasses import dataclass
import re
import json

@dataclass
class ValidationError:
    """引用验证错误"""
    error_type: str  # "missing_ref", "invalid_ref", "hallucinated_ref", "orphan_ref"
    description: str
    location: str = ""  # 错误位置

class CitationValidator:
    """引用格式与内容验证器"""

    def __init__(self, available_chunks: List[str]):
        self.available_chunks = available_chunks

    def validate_format(self, answer: str) -> List[ValidationError]:
        """检查引用格式是否正确"""
        errors = []

        # 1. 检查引用标记格式 [N]
        ref_pattern = re.findall(r'\[(\d+)\]', answer)
        for ref in ref_pattern:
            idx = int(ref)
            if idx < 1 or idx > len(self.available_chunks):
                errors.append(ValidationError(
                    error_type="invalid_ref",
                    description=f"引用 [{idx}] 超出有效范围 (1-{len(self.available_chunks)})",
                    location=ref,
                ))

        # 2. 检查重复引用
        if len(ref_pattern) != len(set(ref_pattern)):
            # 允许合理重复
            pass

        return errors

    def validate_content(self, answer: str) -> List[ValidationError]:
        """检查引用内容是否合理"""
        errors = []

        refs = re.finditer(r'\[(\d+)\]', answer)
        for match in refs:
            idx = int(match.group(1)) - 1
            chunk = self.available_chunks[idx]
            # 获取引用前的文本
            start = max(0, match.start() - 60)
            preceding_text = answer[start:match.start()]

            # 简单检查：引用前的文本是否能在 chunk 中找到
            key_words = [w for w in preceding_text.split() if len(w) > 2][-5:]
            found = any(word in chunk for word in key_words)
            if not found:
                # 不直接报错（可能 chunk 就是支持段落）
                pass

        return errors

    def detect_hallucinated_refs(
        self, answer: str, context_chunks: List[str]
    ) -> List[ValidationError]:
        """检测幻觉引用 — 引用标注了但内容不在上下文中"""
        errors = []
        refs = re.finditer(r'\[(\d+)\]', answer)
        for match in refs:
            idx = int(match.group(1))
            if idx > len(context_chunks):
                errors.append(ValidationError(
                    error_type="hallucinated_ref",
                    description=f"引用 [{idx}] 指向的 chunk 不存在",
                    location=match.group(),
                ))
        return errors

    def full_validation(
        self, answer: str, context_chunks: List[str]
    ) -> Tuple[bool, List[ValidationError]]:
        """完整引用验证"""
        all_errors = []
        all_errors.extend(self.validate_format(answer))
        all_errors.extend(self.validate_content(answer))
        all_errors.extend(self.detect_hallucinated_refs(answer, context_chunks))
        return len(all_errors) == 0, all_errors

def citation_acceptance_test():
    """引用验收测试"""
    validator = CitationValidator(
        available_chunks=[
            "正式员工年假政策",
            "病假需提供医院证明",
        ]
    )

    test_cases = [
        ("年假为15天[1]，病假需证明[2]", True, "正常引用"),
        ("年假为15天[1]，病假需证明[5]", False, "引用 [5] 超出范围"),
        ("年假为15天，无引用标注", True, "无引用不报错（只是质量建议）"),
        ("年假为15天[1][999]", False, "引用了不存在的 999"),
    ]

    for answer, expected_pass, desc in test_cases:
        passed, errors = validator.full_validation(answer, validator.available_chunks)
        status = "PASS" if passed else "FAIL"
        print(f"[{status}] {desc}: {answer}")
        if errors:
            for e in errors:
                print(f"       {e.error_type}: {e.description}")

常见问题（FAQ）

Groundedness 评分和 RAGAS Faithfulness 有什么区别？
Faithfulness 检查「有没有矛盾」— 答案说 15 天年假，上下文也提到 15 天，即通过。Groundedness 检查「有无来源」— 答案说 15 天年假，需要标注这信息来自文档 chunk_001。更严格。实践中，建议先保 Faithfulness，再追求 Groundedness。
LLM 经常编造引用编号，怎么防御？
多层防御：1) 在 prompt 中明确要求「只使用提供的文档片段编号」；2) 后处理阶段用 CitationValidator 检查每个 [N] 是否在有效范围；3) 对无法验证的引用从答案中移除并记录告警；4) 设置 CI 门禁，禁止引用错误率 > 5% 的部署。
引用信息应该放在 API 返回的哪个字段？
建议独立于 answer 字段。示例：{"answer": "年假为15天[1]", "citations": [{"id": "1", "source": "hr_policy.pdf", "page": 3, "text": "..."}]}。这样前端可以灵活选择渲染方式（行内标注或脚注），同时后端确保引用数据可被验证。
Groundedness 评分每次都要调用 LLM，成本怎么控制？
不是每次调用都做全量评分。建议策略：1) 开发阶段全量评分；2) 生产环境抽样（如 10% 流量做评分监控）；3) CI 门禁时对评测集做全量评分；4) 出现用户反馈问题时对相关答案做补充评分。

本章结论

  • 引用溯源是可信 RAG 的基础：从文档 chunk → 答案段落建立可追溯的映射，满足合规要求。
  • Groundedness 比 Faithfulness 更严格：Faithfulness 要求不矛盾，Groundedness 要求有出处。
  • 引用验证防幻觉：LLM 会编造引用，必须用验证器检查引用编号和 chunk 存在性。
  • 置信度标注提升透明度：用不同颜色/标记显示每个段落的可验证程度。
  • CI 门禁保证质量：将引用验证和 Groundedness 评分纳入 CI，防止低质量回答上线。
