微软 GraphRAG 部署

Hybrid 路由的具体实现：在 RAGAS 评测管线中增加一个「cross_doc_score」分类器，基于问题中出现的实体数量（如问题中的命名实体 ≥ 3 个则大概率需要跨文档推理）。CorpAssist 日志分析显示：问题中含 1~2 个实体时仅 18% 需要跨文档，含 3~4 个实体时 62% 需要跨文档，含 5+ 实体时 89% 需要跨文档。基于该规则可实现一个零成本的路由器，无需额外 LLM 调用。

FAQ：什么时候不应该用 GraphRAG？

1) 知识库 < 100 篇文档或文档间无交叉引用关系 — 图结构信息量低；2) 查询主要是单文档事实问答（「今年的年假天数」）— Naive RAG 成本更低且延迟更短；3) 对索引成本敏感且更新频率高（每周更新）— 每次索引重新运行成本 $10+；4) 对 P95 延迟要求 < 1.5s — GraphRAG Local Search P95=2.1s 超出。

自检：GraphRAG 索引中最耗时的环节是哪个？如何优化？

最耗时的是实体抽取环节（占总时间的 ~67%），因为每个 chunk 需调用 LLM 提取实体+关系+声明。优化方向：1) 使用 gpt-4o-mini 替代 gpt-4o（成本降 6 倍，但实体准确率从 89% 降至 77%）；2) 增大 chunk_size 减少调用次数；3) 使用模型并行（多个 chunk 同时抽取）。

  • 适用场景：文档间关联密集、跨文档推理频繁的知识库才值得 GraphRAG 的索引成本

面试可能问

GraphRAG 的社区检测是如何实现的？为什么用了 Leiden 算法而非 Louvain？

Leiden 算法在 Louvain 基础上增加了 refine 阶段，保证社区连通性，避免 Louvain 可能产生的断连社区。GraphRAG 层次化社区检测：先构建实体图（节点=实体，边=关系强度），然后逐层应用 Leiden 算法形成多尺度社区，上层社区包含下层社区，形成金字塔结构。层次化使得 Local Search 和 Global Search 可在不同粒度检索。

List 出 3 个你可能在 CorpAssist 考虑使用 GraphRAG 的信号？

1) 用户频繁问「A 政策对 B 流程有什么影响」类问题（跨文档推理占查询 > 30%）；2) 知识库超过 500 篇且文档间有密集引用关系（如「详见第四章」）；3) Bad case 分析显示 Naive RAG 因 chunk 碎片化导致答案不完整（context_recall < 0.70）。

如何降低 GraphRAG 索引成本？

1) 实体抽取使用 gpt-4o-mini 替代 gpt-4o（成本降 6 倍，准确率从 89%→77%）；2) 增加 chunk_size 从 1200 到 2400 减少调用次数（但可能漏掉细粒度实体）；3) 关闭 claim_extraction（占 15% 成本）；4) 对更新文档使用增量索引（目前 graphrag 不支持，需自研 diff 策略）。

引用溯源实现方案

CorpAssist · 合规审计法务团队抽查回答「竞业限制期是多久」，结果需提供：这句话引自哪份文档的哪一条款、文档版本号、最后更新时间。

引用溯源的核心是 将检索阶段的文档元数据贯穿到最终输出。LCEL 链中需要让 sources 流经整个管线，而非仅在检索阶段消费。

python
# 引用溯源实现 — 带 sources 的完整链
from langchain_core.runnables import RunnableParallel, RunnableLambda
from langchain_core.pydantic_v1 import BaseModel, Field
from typing import List

class CitationSource(BaseModel):
    doc_id: str = Field(description="文档唯一标识")
    title: str = Field(description="文档标题")
    snippet: str = Field(description="引用原文片段（30~100字）")
    location: str = Field(description="条款位置, 如第3章第5条")
    page_num: int = Field(default=0)
    confidence: str = Field(default="high")
    doc_version: str = Field(default="v1.0")

class RAGOutput(BaseModel):
    answer: str
    citations: List[CitationSource]

def build_citation_context(docs):
    """构造带引用的 context 字符串"""
    context_parts = []
    for d in docs:
        meta = d.metadata
        snippet = d.page_content[:200]
        citation = f"[source: {meta['doc_id']} | {meta.get('title','')} | {meta.get('location','')}]"
        context_parts.append(f"{citation}\n{snippet}")
    return "\n\n".join(context_parts)

citation_prompt = ChatPromptTemplate.from_messages([
    ("system", "你需严格根据 context 回答。对每个事实标注引用编号如[1][2]。\n{context}"),
    ("human", "{question}"),
])

def parse_with_citations(llm_output, docs):
    """解析 LLM 输出，提取引用标注并映射回文档"""
    import re
    import json
    citations = []
    refs = set(re.findall(r'\[(\d+)\]', llm_output))
    for ref in sorted(refs, key=int):
        idx = int(ref) - 1
        if idx < len(docs):
            d = docs[idx]
            citations.append(CitationSource(
                doc_id=d.metadata["doc_id"],
                title=d.metadata.get("title", ""),
                snippet=d.page_content[:100],
                location=d.metadata.get("location", ""),
                confidence="high",
                doc_version=d.metadata.get("version", "v1.0"),
            ).dict())
    # 清理引用标记
    clean_answer = re.sub(r'\s*\[\d+\]', '', llm_output)
    return {"answer": clean_answer, "citations": citations}

chain_with_citations = (
    RunnableParallel(
        context=lambda x: build_citation_context(x["docs"]),
        question=lambda x: x["question"],
        docs=lambda x: x["docs"],
    )
    | citation_prompt
    | llm
    | StrOutputParser()
    | RunnableLambda(lambda out: parse_with_citations(out, context.get("docs", [])))
)

python
# 置信度自动标注
from rouge_score import rouge_scorer

scorer = rouge_scorer.RougeScorer(["rougeL"], use_stemmer=True)

def compute_confidence(answer_segment: str, context_docs: list) -> dict:
    max_score = 0.0
    best_doc = None
    for d in context_docs:
        scores = scorer.score(answer_segment, d.page_content)
        rouge_l = scores["rougeL"].fmeasure
        if rouge_l > max_score:
            max_score = rouge_l
            best_doc = d

    if max_score >= 0.80:
        return {"confidence": "high", "score": max_score, "source": best_doc.metadata.get("doc_id") if best_doc else None}
    elif max_score >= 0.50:
        return {"confidence": "medium", "score": max_score, "source": best_doc.metadata.get("doc_id") if best_doc else None}
    elif max_score > 0:
        return {"confidence": "low", "score": max_score, "source": best_doc.metadata.get("doc_id") if best_doc else None}
    else:
        return {"confidence": "none", "score": 0.0, "source": None}

answer = "年假最低天数为5天。工龄满10年则为10天。"
for sent in answer.split("。"):
    if sent.strip():
        result = compute_confidence(sent, retrieved_docs)
        print(f"[{result['confidence']}] {sent} → {result['source']} ({result['score']:.3f})")

CorpAssist 合规审计

合规审计要求 CorpAssist 的每个 RAG 回答满足四项可追溯性要求。实现方案为：每次查询生成一条审计记录 JSON-L 并持久化到审计日志。

python
# 审计日志记录器
import json, datetime, uuid

class AuditLogger:
    def __init__(self, log_path="audit/rag_audit.jsonl"):
        self.log_path = log_path

    def log_query(self, question: str, answer: str, citations: list,
                  confidence_scores: dict, latency_ms: float):
        record = {
            "audit_id": str(uuid.uuid4()),
            "timestamp": datetime.datetime.utcnow().isoformat() + "Z",
            "question": question,
            "answer": answer,
            "citations": citations,
            "confidence_summary": confidence_scores,
            "latency_ms": latency_ms,
            "model": "gpt-4o-mini",
            "retriever_config": {"k": 4, "search_type": "similarity"}
        }
        with open(self.log_path, "a") as f:
            f.write(json.dumps(record, ensure_ascii=False) + "\n")
        return record["audit_id"]

    def query_trail(self, audit_id: str) -> dict:
        """根据 audit_id 检索完整审计链"""
        with open(self.log_path) as f:
            for line in f:
                rec = json.loads(line)
                if rec["audit_id"] == audit_id:
                    return rec
        return None

# 使用
logger = AuditLogger()

    latency_ms=1250,
)
print(f"查询审计ID: {audit_id}")

面试可能问

为什么不能用 LLM 自行声明引用来源而非要系统侧强制标注？

LLM 有幻觉倾向，可能编造看似合理的引用（hallucinated citation）。在一项研究中，GPT-4 对自身训练数据中的文档编造 doc_id 的比例达 12%。必须由系统侧将检索阶段的元数据（doc_id、snippet、location）注入输出，不与 LLM 的自述混淆。系统侧标注是可审计的，LLM 自述不可审计。

置信度标注为什么会用 ROUGE-L 而非 BERTScore？

ROUGE-L 基于词重叠（LCS），计算快（O(n*m)）且结果可解释（具体哪些词重叠了）。BERTScore 基于语义嵌入，对 paraphrase 更鲁棒但计算成本高（需 GPU），且分数语义模糊（0.85 是什么意思？）。在合规场景中，可解释性优先级高于语义弹性，因此选择 ROUGE-L。若后续需要更好的 paraphrase 检测，可在 ROUGE-L 后叠加 BERTScore 做二次验证。

如果引用格式从 JSON 改成了新的审计标准，你的变更面有多大？

影响范围：1) CitationSource 数据模型定义（Pydantic class）；2) API 序列化器（FastAPI response_model）；3) 审计日志写入格式；4) 前端渲染逻辑。可通过将引用格式抽离为独立模块（src/citation/formatter.py）降低变更影响面，业务代码不直接引用 JSON 字段名而引用 CitationSource 对象属性。

场景 1：审计发现高置信度引用的文档版本与实际不符，问题出在哪个环节？

可能是：1) 索引环节未更新 doc_version metadata；2) 检索 filter 未排除旧版本文档（缺 is_latest=true 条件）；3) 文档更新后未触发重新索引。修复方法：在索引管线中强制写入 doc_version 和 updated_at，检索时加 filter，定期清理旧版本。

场景 2：RAG 输出中的 medium/low 置信度占比超过 40%，你认为根因是什么？

medium+low > 40% 意味着近半数答案内容不能直接从 context 逐字匹配。根因可能：1) chunk_size 过大导致 context 含噪声，模型需要筛选综合；2) k 值偏低导致单篇文档片段不完整，LLM 需跨多篇拼接；3) 问题本身属于综合性查询（如「总结一下公司的休假政策」）— 此时 medium 合理。建议分别统计 medium 和 low 占比，若 low > 10% 说明检索严重不充分。

LCEL 管线构建

CorpAssist · HR 政策问答员工提问「年假折算规则」，管线必须从 corpassist-hr-v2 collection 检索政策原文，填入 system prompt 约束模型，禁止跳过检索直接生成。

LangChain Expression Language（LCEL）以声明式管道组合 RAG。核心数据流：输入 dict → 检索并格式化 context → 注入 Prompt → LLM 推理 → 解析输出。每个 | 右侧可以是 Runnable 对象、函数或任意可调用对象，框架自动处理类型兼容。LCEL 的设计哲学是将 RAG 的每一步抽象为可组合的 Runnable 单元：RunnableParallel 并行构造输入、RunnablePassthrough 透传字段、RunnableLambda 嵌入自定义函数、StrOutputParser 解析 ChatMessage 输出为字符串。这一设计使得调试时可以 .get_graph() 导出管线拓扑，或使用 .with_listeners() 在每步执行前后插入回调。

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableParallel
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma(
    collection_name="corpassist-hr-v2",
    embedding_function=embeddings,
    persist_directory="./chroma_db",
)
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4, "filter": {"tenant_id": "corp-a", "kb": "hr"}},
)

    ("system", "你是CorpAssist政策助手。仅根据以下context回答，无依据则说不知道。\n{context}"),
    ("human", "{question}"),
])

def format_docs(docs):
    return "\n\n---\n\n".join(d.page_content for d in docs)

chain = (
    RunnableParallel({"context": retriever | format_docs, "question": RunnablePassthrough()})
    | prompt
    | llm
    | StrOutputParser()

FAQ：filter 在检索层 vs 在 prompt 层有什么区别？

检索层 filter 直接作用于向量数据库查询语句（WHERE 子句），确保跨租户数据物理隔离；prompt 层约束仅在推理时提示，模型仍可能从训练数据中越权输出。CorpAssist 安全审计要求检索层 filter 为必选项。

    async def generate():
        async for chunk in chain.astream(
            {"question": req.question, "filter": {"tenant_id": req.tenant_id}}
        ):
            if isinstance(chunk, str):
                yield f"data: {chunk}\n\n"
        yield "data: [DONE]\n\n"
    return StreamingResponse(generate(), media_type="text/event-stream")

面试可能问

LCEL 中 RunnableParallel 与 RunnablePassthrough 各自的使用场景？

RunnableParallel 用于从同一输入构造多个分支，如同时产生 context 和透传 question；RunnablePassthrough 用于透传某个键的值不做变换。若检索结果需与原始 question 合并进 prompt，必须用 RunnableParallel。

如何排查 sources 为空但向量库有数据的情况？

1) filter 条件是否与索引数据的 metadata 精确匹配（如 tenant_id 大小写）；2) embedding 模型是否与索引时一致（模型版本/维度）；3) query 语言/分词是否与文档差异过大（可试混合检索缓解）。

LCEL 的 invoke 与 stream 底层如何实现？

invoke 走 RunnableSequence.__call__，逐节点同步执行；stream 利用生成器逐节点 yield 输出块，每个节点实现 transform 方法流式传递。框架自动处理类型适配与回调传播。

面试可能问

LlamaIndex 的 Index 和 LangChain 的 VectorStore 在设计理念上有什么本质不同？

Index 是「有状态的智能对象」—— 它记录了文档结构、节点关系、嵌入模型引用，查询时自动编排；VectorStore 是「无状态的存储抽象」—— 只负责向量存储和检索，查询编排由上层链完成。LlamaIndex 的 Index 更接近 ORM 思想，LangChain 的 VectorStore 更接近 DAO 思想。

为什么在 CorpAssist 中 LangChain 和 LlamaIndex 可以共存？

两者抽象层次不同：LangChain 用于查询链编排（与 Spring AI 对齐），LlamaIndex 用于索引构建和复杂文档解析。Python 侧内部调用无冲突，输出统一为 answer+sources 契约。共存策略让各框架发挥各自优势。

RouterQueryEngine 的底层实现原理是什么？

它本质上是 LLM 驱动的选择器：将每个 QueryEngineTool 的 name+description 拼入 system prompt，LLM 根据用户输入选择最匹配的工具。选择结果作为 metadata 返回，再调用对应 QueryEngine。类似 ReAct Agent 的单步路由变体。

混合检索典型方案有两种：倒数排序融合（RRF）和加权和（Weighted Sum）。RRF 不依赖分数绝对值（适合跨模型融合），加权和更直观但要求两路分数尺度一致。CorpAssist 使用 RRF 作为默认策略。

python
# 混合检索核心实现 — RRF 融合
import math
from rank_bm25 import BM25Okapi
from langchain_community.vectorstores import Chroma

class HybridRetriever:
    def __init__(self, vectorstore, bm25_corpus, alpha=0.3, k=10):
        self.vectorstore = vectorstore
        self.bm25 = BM25Okapi([doc.split() for doc in bm25_corpus])
        self.bm25_docs = bm25_corpus
        self.alpha = alpha  # 向量权重，BM25权重 = 1-alpha
        self.k = k

    def retrieve(self, query: str):
        # 向量检索
        vector_docs = self.vectorstore.similarity_search_with_score(query, k=self.k*2)
        vec_scores = {d.metadata["doc_id"]: 1-s for d,s in vector_docs}

        # BM25 检索
        bm25_scores_raw = self.bm25.get_scores(query.split())
        bm25_top_indices = sorted(
            range(len(bm25_scores_raw)), key=lambda i: bm25_scores_raw[i], reverse=True
        )[:self.k*2]
        bm25_scores = {}
        for idx in bm25_top_indices:
            bm25_scores[f"doc_{idx}"] = bm25_scores_raw[idx]

        # RRF 融合
        all_ids = set(vec_scores.keys()) | set(bm25_scores.keys())
        combined = {}
        for doc_id in all_ids:
            rank_vec = self._get_rank(doc_id, vec_scores, order="desc")
            rank_bm25 = self._get_rank(doc_id, bm25_scores, order="desc")
            rrf_vec = 1 / (60 + rank_vec) if rank_vec else 0
            rrf_bm25 = 1 / (60 + rank_bm25) if rank_bm25 else 0
            combined[doc_id] = self.alpha * rrf_vec + (1 - self.alpha) * rrf_bm25

        sorted_docs = sorted(combined.items(), key=lambda x: x[1], reverse=True)[:self.k]
        return sorted_docs

    @staticmethod
    def _get_rank(doc_id, scores, order="desc"):
        if doc_id not in scores:
            return None
        sorted_ids = sorted(scores.keys(), key=lambda x: scores[x], reverse=(order=="desc"))
        return sorted_ids.index(doc_id) + 1

FAQ：为什么不在向量库层面做混合检索（如 Elasticsearch 的 hybrid query）？

Elasticsearch 8.12+ 支持 hybrid query（kNN + BM25 自动融合），若公司已有 ES 集群可考虑。CorpAssist 选择独立 BM25 实现的原因：1) 与向量库解耦，可随时切换存储后端；2) RRF alpha 可针对不同知识库独立调优；3) 避免 ES 许可成本。

RRF 融合的边界情况处理：当某一路检索结果为空时（如 BM25 对所有文档评分均为 0），RRF 仅依赖另一路的结果，score 归一化后仍能输出 Top-K。当某一路的 Top-K 中存在大量低分文档（score < 0.1）时，RRF 常数 k=60 会将这些低分文档的排名权重压制到极低（约 0.016），避免了低质量结果干扰。当两路检索的结果集完全不重叠时（这在跨语言查询中常见），RRF 自动对每路的前几名给予公平的 rank 权重，不需要分数归一化。

面试可能问

为什么 RRF 比加权和更适合跨模型融合？

RRF 基于排名而非分数绝对值，不受不同检索模型分数分布差异影响（如向量相似度 0~1 vs BM25 分数 0~20+）。加权和需要先归一化两路分数到同一尺度（min-max 或 z-score），引入额外偏差。RRF 的常数 k（通常 60）控制排名权重衰减速度。

混合检索后 answer faithfulness 为什么提升 12%？

纯向量检索在专有名词查询中可能遗漏关键文档，导致 LLM 无依据可引用，产生幻觉。混合检索提高了准确文档出现在 context 中的概率，LLM 获得足够依据，faithfulness（是否与 context 一致）自然从 0.81 升至 0.91。

如何为不同知识库选择不同的 alpha？

方法：对每个知识库构建独立评测集（至少 100 条查询+标注），各自运行网格搜索。CorpAssist 实测：IT 知识库 alpha=0.5（BM25 对代码术语贡献大），HR 知识库 alpha=0.3，Legal 知识库 alpha=0.4（条款编号 BM25 优势明显）。

评测管线使用 ragas 库，核心输入为 Hugging Face Dataset 格式的 QA 三元组（question, answer, contexts, ground_truth）。CorpAssist 维护一个版本化的评测集 corpassist_eval_v2.jsonl，每次评测前拉取最新版本。

python
# 自动化评测脚本
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy,
    context_precision, context_recall, answer_correctness
)
import json, os

def run_eval(chain, eval_path: str, threshold: dict):
    with open(eval_path) as f:
        records = [json.loads(line) for line in f]

    answers, contexts = [], []
    for rec in records:
        result = chain.invoke(rec["question"])
        answers.append(result["answer"])
        contexts.append([c.page_content for c in result["context_docs"]])

    ds = Dataset.from_dict({
        "question": [r["question"] for r in records],
        "answer": answers,
        "contexts": contexts,
        "ground_truth": [r.get("ground_truth", "") for r in records],
    })

    scores = evaluate(ds, metrics=[
        faithfulness, answer_relevancy,
        context_precision, context_recall, answer_correctness
    ])

    report = {k: float(v) for k, v in scores.items()}
    with open("eval_report.json", "w") as f:
        json.dump(report, f, indent=2)

    passed = all(report[k] >= threshold[k] for k in threshold)
    if not passed:
        failed = {k: report[k] for k in threshold if report[k] < threshold[k]}
        raise RuntimeError(f"Eval gate failed: {failed}")
    return report

if __name__ == "__main__":
    from chain_stub import chain
    gate = {"faithfulness": 0.85, "context_recall": 0.80}
    report = run_eval(chain, "eval/corpassist_eval_v2.jsonl", gate)
    print(f"Eval passed: {json.dumps(report, indent=2)}")

CI 集成与闸门

在 GitHub Actions 中集成 RAGAS 评测，每次 PR 触发时运行。评测集选择仅执行与 PR 改动相关的子集（通过 changed_files 推断影响的知识库）。若关键指标低于阈值，CI 直接 FAIL 并 Slack 通知。

# 坏例归因脚本
def classify_bad_case(question: str, answer: str, contexts: list, ground_truth: str):
    """自动归因 bad case"""
    # Step 1: 检查 context 是否包含 ground_truth
    gt_in_context = any(gt_phrase in ctx for gt_phrase in split_sentences(ground_truth) for ctx in contexts)

    if not gt_in_context:
        return {
            "category": "检索遗漏",
            "action": "检查 chunk 粒度、embedding 模型、filter 配置",
            "severity": "high"
        }

    # Step 2: 检查答案是否与 context 一致
    from ragas.metrics import faithfulness
    score = faithfulness.score((answer, contexts))
    if score < 0.7:
        return {
            "category": "生成偏离",
            "action": "检查 prompt 约束、温度参数、是否绕过了 context",
            "severity": "high"
        }

    # Step 3: 检查噪声比例
    from ragas.metrics import context_precision
    precision = context_precision.score((question, contexts, ground_truth))
    if precision < 0.6:
        return {
            "category": "检索噪声",
            "action": "降低 k、增加 score_threshold、加 reranker",
            "severity": "medium"
        }

    return {
        "category": "格式/体验",
        "action": "检查 prompt 模板、输出格式、语气",
        "severity": "low"
    }

坏例归因自动化：在 CI 闸门失败后，自动触发的归因脚本将每条失败用例分类并生成归因报告。报告按检索遗漏/生成偏离/检索噪声/格式体验四类统计，并给出 Top-3 建议操作。CorpAssist 的归因流程已被集成到工单系统，当归因脚本发现某类 bad case 比例超过 30% 时自动创建 Jira 可执行任务。

本章结论

  • RAGAS 六指标：覆盖检索质量（context_precision/recall）和生成质量（faithfulness/relevancy/correctness），门禁以 faithfulness + context_recall 为核心
  • 评测自动化：Dataset 格式输入 QA 三元组，evaluate() 自动计算，输出 JSON 报告
  • CI 闸门：Blocking（faithfulness<0.85 或 recall<0.80）→ CI FAIL；Warning/Info 仅通知
  • 三级归因：检索遗漏 → 查 chunk/embedding；生成偏离 → 查 prompt/LLM；检索噪声 → 降低 k/加 rerank
  • 评测集维护：版本化管理 QA 三元组，每次评测拉取最新版本，覆盖所有知识库

面试可能问

RAGAS 的 faithfulness 是如何计算的？有什么局限性？

详细说明步骤如下：第一步，将 answer 拆分为独立陈述（atomic claims）；第二步，对每个陈述，用 LLM 判断是否可从 context 中推断（true/false/unverifiable）；第三步，faithfulness = true_claims / (true_claims + false_claims)。局限性：LLM-as-judge 的评判本身可能有偏差；对长 answer 中 claims 的拆分会漏掉隐含断言；对专业领域术语的理解可能不够准确。

评测集应该怎么构建和维护？

构建：从用户日志采样真实问题（~60%），基于知识库关键内容由专家编写覆盖性题目（~30%），补充边界情况（~10%）。维护：每个版本标注 ground_truth；每次 embedding 模型升级、chunk 策略变更后重新评测；定期盲测（人工评分 vs RAGAS 评分偏差 < 0.1 时 confidence 足够）。

CI 中运行完整评测集耗时较长，有什么加速策略？

1) 基于 PR 改动范围选择子集（改 HR 知识库只跑 HR 评测集）；2) 缓存最近一次通过的评测结果，仅运行 diff 影响的相关指标；3) 先快速过滤（样本量 20 条快筛），通过后再全量验证；4) 使用全局评测结果缓存，同一 commit 不重复评测。

场景 1：用户反馈 HR 政策查询答非所问，但 RAGAS 评分全部通过。怎么排查？

可能性：1) 评测集 ground_truth 已过时（政策更新后未同步）；2) 问题属于评测集未覆盖的新场景；3) RAGAS LLM-as-judge 对专业领域的判断不够准确。建议将用户 bad case 加入评测集并手动复审 RAGAS 评分与人工评分的一致性。

场景 2：CI 评测经常因高延迟超时失败，但 recall 正常，如何优化？

1) 评测集过大（>500 条），每个 query 触发完整的 embed+retrieve+llm 管线。建议按知识库分流并行评测；2) 对 LLM 调用的评测环节（faithfulness、relevancy）使用更小模型（gpt-4o-mini 替代 gpt-4o）；3) 设置评测 timeout 为 30 分钟，超过则自动 fail 并重试。
