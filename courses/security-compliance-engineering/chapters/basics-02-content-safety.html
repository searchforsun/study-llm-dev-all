为什么要学本章

S5 内容工坊与 S2 客服都要处理违规、仇恨、色情、政治敏感等内容。content safety 不是「上线前调一次 Prompt」，而是入站拦截 + 出站复核的可运维链路；缺任一侧会出现「脏输入进模型」或「有害输出已送达用户」。

学完你能
  • 配置 input moderation 与 output moderation 的触发点与拒答策略
  • 在 Spring 注册 safeguard advisor，在 Python 侧实现对称 dual stack filter
  • 区分 toxic detect 与 policy violation 的处置差异
本章先记住 3 件事
  • 双向审核：入站防注入与违规输入；出站防模型幻觉式有害内容（含流式 chunk）。
  • 双栈一致：同一 policy_version 与阈值；避免 Python 松、Spring 严导致路由绕过。
  • 可审计拒答：拒答须带 reason_code，勿统一 500 或空回复（合规与运营都需要）。

输入输出审核

CorpAssist · 内容工坊（S5） 运营同学生成「双十一促销文案」。入站需拦截竞品诋毁与绝对化用语；出站需拦截未验证的「全网最低价」等 policy violation，并保留审核记录供品牌合规抽查。
Moderation 双向链路
flowchart TB
  U[用户输入] --> IM[input moderation]
  IM -->|block| R1[拒答+reason_code]
  IM -->|pass| LLM[LLM/RAG]
  LLM --> OM[output moderation]
  OM -->|block| R2[替换/拒答]
  OM -->|pass| OUT[响应用户]

流式场景下 outbound 需缓冲或逐 chunk 检测，避免前半段已输出违规内容。

入站 vs 出站职责

阶段检测目标典型动作CorpAssist 配置键
input moderation 注入、辱骂、违法请求 HTTP 400 + 统一拒答模板 moderation.inbound.enabled
output moderation 有害生成、PII、过度承诺 截断流、替换句子、整段拒答 moderation.outbound.enabled

Spring 入站 Advisor 示例

java
@Bean
ChatClient chatClient(ChatModel model, ContentSafetyAdvisor safetyAdvisor) {
  return ChatClient.builder(model)
      .defaultAdvisors(safetyAdvisor)  // 入站+出站 hook
      .build();
}

@Component
class ContentSafetyAdvisor implements CallAdvisor, StreamAdvisor {
  @Override
  public ChatClientRequest before(ChatClientRequest req, AdvisorChain chain) {
    ModerationResult in = moderationApi.checkInput(req.prompt());
    if (in.blocked()) throw new PolicyBlockedException(in.reasonCode());
    return req;
  }
}
先想 10 秒：只做 outbound、不做 inbound 会怎样？

恶意/超长/注入 payload 仍进入 LLM，浪费 Token 并可能触发间接注入；inbound 是成本与安全的双保险。

双栈 Advisor

dual stack filter 指：CorpAssist 网关 → Python AI 服务 → Spring 治理面三层使用同一策略包，而非各写各的 regex。

位置：ChatClient Advisor 链、SSE 网关 filter。

集成：DashScope/OpenAI moderation api 或自建分类器；Micrometer 打点 moderation.blocked。

位置：FastAPI middleware + LangChain Runnable 前后置。

集成：同 policy_version YAML；RAG 检索结果也须过 inbound 规则（防间接注入）。

Python 对称 middleware

python
# corpassist/middleware/content_safety.py
async def moderation_middleware(request: Request, call_next):
    body = await request.json()
    result = moderation_client.check(body["messages"], policy_version=POLICY_V)
    if result.blocked:
        return JSONResponse({"error": "policy_blocked", "reason": result.code}, status=400)
    response = await call_next(request)
    return await scan_stream_if_needed(response)
反模式
  • Spring 调 cloud API，Python 裸奔
  • 拒答无 reason_code，运营无法统计
  • 流式 SSE 不做 outbound 扫描
推荐
  • Git 管理 policies/content-v{n}.yaml
  • CI 跑 golden 拒答/放行用例
  • blocked 事件进 audit（见 practice-03）

S2 客服 vs S5 内容工坊：profile 差异

维度S2 对外客服S5 内容工坊
inbound 重点辱骂、违法咨询、注入竞品攻击、敏感政治
outbound 重点PII、过度承诺广告法、绝对化用语
fail-closedModeration API 超时必拒答同左，profile 阈值可略调
判断：Python middleware 已拦截违规，Spring 侧可以不挂 Advisor 吗？

不可以。在线 ChatClient 路径必须经过 Spring Advisor，保证每一跳 LLM 调用都被覆盖；Python 可能只服务部分 endpoint 或离线批处理。

本章结论

  • 输入输出审核：双向链路；inbound 省 Token 并防注入，outbound 防有害生成与过度承诺。
  • 双栈 Advisor：Spring Advisor + Python middleware 共享 policy_version；RAG chunk 同样要审。
