为什么要学本章

CorpAssist 面向 ToB 客户，必须拦截违法违规、仇恨、色情及企业自定义敏感词。仅靠模型自带安全对齐无法通过合规验收。本章落地 content safety 能力 C26：在请求前后部署可观测、可配置的审核链。

学完你能
  • 区分 input moderation 与 output moderation 职责
  • 用 Python NeMo Guardrails 与 Spring AI Advisor 搭建双栈对照
  • 设计误杀率与漏杀率的权衡及人工兜底
本章先记住 3 件事
  • 输入审核拦「用户想做什么坏事」；输出审核拦「模型说了什么不该说的」。
  • 审核应确定性优先：规则 + 小模型，大模型仅作复审。
  • 双栈 Advisor 链顺序固定：Input → LLM → Output。

输入输出审核

能力 C26 要求 CorpAssist 对每条消息做分类：安全 / 需拒答 / 转人工。输入侧重点识别越狱、仇恨、违法指令；输出侧防止模型生成未授权承诺、泄露机密或歧视性言论。

CorpAssist · 客服拒答审计 用户发送仇恨言论，输入 Advisor 返回 refusal_code=HATE 并写入 audit；若仅在前端弹窗提示而未落库，等保测评会判定「安全审计不完整」。

输入用户 prompt关键词 + 分类器拒答 + 审计
输出模型回复PII 扫描 + 政策模板替换为安全模板
流式token 流滑动窗口检测截断并收尾

输入与输出审核职责分离，避免「只审输入」导致模型仍生成违规内容。

Java：Spring AI 输入 / 输出 Advisor

java
ChatClient chat = ChatClient.builder(chatModel)
    .defaultAdvisors(
        new RequestAdvisor(inputPolicy),      // 输入：越狱/违法
        new SafeGuardAdvisor(outputPolicy),   // 输出：敏感词（非输入侧）
        new AuditAdvisor())
    .build();

SafeGuardAdvisor 侧重生成结果过滤；输入侧越狱与违法指令应使用自定义 RequestAdvisor 或网关规则，避免与 C26「双端审核」职责混淆。

moderation latency 需纳入 SLO：同步审核增加 50–200ms 时，应使用轻量模型或并行批处理，并对 VIP 租户提供可配置严格度。

C26 输入输出审核检查清单

  • 违规分类是否覆盖违法、仇恨、越狱、泄密四类？
  • 拒答模板是否带 refusal_code 供审计？
  • 流式输出是否在违规 token 出现时截断？
  • 误杀样本是否进入人工复核队列？
  • 策略版本是否与 Java/Python 服务同步发布？
先想 10 秒：为什么输出审核不能省略？

模型可能在「干净输入」下仍生成歧视、泄密或虚假承诺；C26 要求双端闭环。

Java：ResponseAdvisor 模板替换

java
public AdvisedResponse aroundCall(AdvisedRequest req, CallAroundAdvisorChain chain) {
    AdvisedResponse resp = chain.nextAroundCall(req);
    if (policyEngine.violates(resp)) {
        return AdvisedResponse.from(resp).withContent(SAFE_TEMPLATE).build();
    }
    return resp;
}
生产坑
  • 仅拦截首句、后半段泄密：流式必须滑动窗口检测。
  • 拒答无 refusal_code：等保审计无法分类统计。
  • staging 与 production policy 版本不一致：见 实战：CorpAssist 安全基线 SEC-03 验收。

双栈 Advisor 对齐检查清单

  • Python colang 规则 ID 是否与 Java policy YAML 一一映射？
  • 集成测试是否对比同一违规样本的双栈拒答文案？
  • Micrometer/LangSmith 是否记录 advisor_name 与 policy_version？
  • 回滚时是否能在 5 分钟内切回上一策略版本？
  • VIP 租户严格度是否可配置且默认安全？

发布前务必用同一批违规样本跑通 Python 与 Java 集成测试，避免双栈拒答文案不一致引发合规质疑。

流式场景下建议在首个违规 token 触发时立即截断，并返回统一的 CorpAssist 安全模板。

检查理解：Advisor 顺序反了会怎样？

若输出 Advisor 在输入之前注册，将无法拦截用户侧越狱载荷，且可能重复改写历史消息。

不推荐
  • 仅依赖模型内置安全
  • 流式输出不做中途截断
推荐
  • 输入输出双审 + 审计
  • 策略版本化与 A/B 评测
CorpAssist · 误杀与漏杀权衡 金融租户要求更严：违法咨询拒答率 99.9%，可接受 0.5% 误杀转人工；内部研发租户可放宽至 98% 以提升体验。策略表需按 tenant_tier 加载不同阈值。

拒答模板与 audit 字段

refusal_code用户可见文案audit 必填
HATE无法处理该请求policy_version, tenant_id
JAILBREAK请换一种方式提问matched_rule_id
SECRET_LEAK无法提供该信息trace_id, channel

与《审计与溯源》衔接：每次拒答写 event_type=moderation_block，便于等保抽检与误杀分析。

人工复核队列（误杀兜底）

当分类器置信度处于灰区（如 0.45–0.55）时，CorpAssist 将请求路由到人工复核队列，而非直接拒答或放行。队列 SLA 与客服值班表绑定，避免夜间无人处理导致业务中断。

  1. 灰区请求打标 review_required=true 并暂停自动回复。
  2. 复核员在脱敏界面查看原文与命中规则。
  3. 结论写回 review_outcome 用于后续模型校准。
租户 tier灰区策略SLA
金融默认拒答 + 人工15 分钟
标准人工或安全模板4 小时
内部研发可放宽至告警下一工作日
检查理解：灰区一律放行以提升体验是否可行？

金融与政务租户不可行；须拒答或人工复核，并保留 audit 证据链。

SafeGuardAdvisor 能否替代输入审核？
不能单独替代。它主要过滤模型输出；越狱载荷需在 RequestAdvisor/Guardrails 输入 rail 拦截。
流式回复中途违规如何处理？
在 token 窗口检测到违规类时立即截断，返回统一 refusal 模板并写 audit 事件。
第三方 moderation API 宕机怎么办？
降级为本地规则集 + 默认拒答模板，并告警；禁止静默放行。
双栈发布检查
  • policy 版本号写入 Git tag 与镜像 label。
  • 集成测试报告附在 MR 描述中。
  • 灰度租户先跑 24h 再全量。

本章结论

C26 把内容安全做成可观测、可回滚的工程能力，而非模型黑盒。

  • C26 需要输入/输出分离的 content safety 链路。
  • NeMo Guardrails 与 Spring AI Advisor 可同构映射，便于双栈一致。
  • 审核策略要可观测、可回滚，并度量误杀/漏杀。
