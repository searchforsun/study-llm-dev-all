为什么要学本章

CorpAssist 默认链路会触达 DashScope、OpenAI、Embedding SaaS、Moderation API 等第三方。法务与信息安全要问：third party risk 谁承担？数据是否 data cross border？没有 vendor assess 与 contract clause，项目无法过 compliance review。

学完你能

  • 起草 DPA 中与 subprocessor、cloud region 相关的关键条款
  • 完成 CorpAssist 场景 dpia（数据处理影响评估）要点清单

本章先记住 3 件事
  • 评估先于接入：无问卷 / 无 SOC2 或等保报告，不进生产。
  • 合同写清区域与留存：cloud region、日志保留、删除权、 breach 通知时限。
  • 子处理商要列明：模型商背后的 GPU 云、CDN、日志 SaaS 都是 subprocessor。

评估

CorpAssist · 接入新 Embedding 供应商（60 秒） 采购想换更便宜的 Embedding API。技术可行，但安全部要求：数据存储在哪国、是否用于模型训练、有无 SOC2 Type II、 incident 72h 内是否通知、能否 DPA 禁止 subprocessor 再分包。缺一项则 compliance review 退回。

third party risk 管理不是填一次表了事，而是与 CorpAssist 架构绑定的持续过程：每新增一条出站 HTTP，就要更新供应商台账与 dpia。

DPIA 核心问题（CorpAssist）

  • 处理哪些个人数据（员工号、客户 PII 经用户输入误入）？
  • 是否 data cross border？跨境时法律依据与安全评估编号？
  • 双栈脱敏是否在出境前完成（见 隐私与脱敏）？
  • 泄露影响与缓解：moderation、工具白名单、审计（见 威胁建模）。
先想 10 秒：Dev 环境调 OpenAI 测 prompt，为何也可能违规？

开发可能复制生产脱敏不全的样本；OpenAI 侧留存策略若未 enterprise 关闭，仍构成 third party risk。应 synthetic 数据 + 网络隔离 + vendor 台账标注 dev_only。

合同

技术控制再完美，没有 contract clause 就无法向监管证明「已尽合理注意义务」。CorpAssist 交付时通常需三层协议：主合同、DPA（数据处理协议）、安全附录。

合同约束「允许谁处理什么」；OPA 与 registry 在运行时强制执行，避免配置漂移。

DPA 条款摘录（Markdown 模板）

markdown 
## 数据处理协议（摘录）

1. **处理目的**：仅为甲方（CorpAssist 客户）提供 LLM 推理服务，不得用于模型训练。
2. **数据位置**：所有 Customer Data 存储与处理限于 [cn-shanghai / 指定 region]。
3. **子处理商**：附件 B 列明；新增须提前 30 日书面通知，甲方有权反对。
4. **跨境传输**：未经书面授权，不得将 Customer Data 传输至中国大陆以外。
5. **删除**：合同终止后 30 日内删除全部副本，并提供书面删除确认。
6. **审计**：每年提供 SOC2 Type II 或等效报告；甲方有权合理范围内审计。
不推荐
  • 只用供应商标准 ToS，无 DPA
  • 合同写「尽量境内」无 region 枚举
  • 子处理商「以厂商为准」不附清单
推荐
  • vendor-registry 与合同附件 B 子处理商一致
  • OPA 阻断未注册 vendor_id
  • 年度 compliance review 更新评估日期
常见误区
  • 公有云 = 天然合规 — 区域选错仍出境；需显式 region + 专线。
  • Embedding 不算 personal data — 可逆关联或含 PII 片段时仍受规制。
  • 合同一劳永逸 — 供应商改 subprocessor 未通知即违约，需监控变更日志。

本章结论

  • 评估：vendor 台账 + DPIA；认证、区域、训练用途、子处理商缺一不可。
  • 合同：DPA 写清驻留、删除、审计、跨境；与 OPA / registry 技术 enforcement 对齐。
  • 持续：新 endpoint = 新评估；sector 变更触发 compliance review。
