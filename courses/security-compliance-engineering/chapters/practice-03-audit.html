为什么要学本章

合规审计不仅是记录谁做了什么，还要能回答「这个回复是怎么生成的？用了哪些知识？哪个模型版本？」。LLM 应用的黑盒特性让溯源比传统系统更难。Audit trail 需要记录从用户输入到 LLM 回复的完整链条。

学完你能
  • 设计满足等保要求的 audit trail
  • 实现 citation chain 追踪回复的知识来源
  • 配置 immutable log 保证日志不可篡改
  • 理解 RAG audit 的溯源方法
本章先记住 2 件事
  • Who-What-When：每行审计日志至少包含谁操作、做了什么、什么时间。
  • 溯源链：不只是记「回复内容」，还要记用了哪些 chunk 和哪个模型版本。

留痕、引用链与等保

CorpAssist · 审计调查（35 秒） 合规官要求查清：某条回复「产品 X 有问题」是否来自知识库。如果只有回复日志，无法回答。如果有 citation chain，可以看到该回复引用了知识库文章 KB-123 的段落 5，而 KB-123 是市场部上周更新的。溯源链让每一步都可追溯。
先想 10 秒：审计除了 who/what/when 还必须记什么 LLM 特有字段？

引用 chunk ID、模型版本、策略版本——否则无法复现回复依据。

Java：审计日志服务

java 
import java.time.Instant;
import java.util.UUID;
import java.util.List;
import java.util.Map;

@Component
public class AuditService {
    private final AuditLogStore logStore;

    public AuditService(AuditLogStore store) {
        this.logStore = store;
    }

    public void recordChat(String userId, String tenantId,
                           ChatRequest request, ChatResponse response,
                           List citations,
                           List toolCalls) {
        AuditEntry entry = new AuditEntry(
            UUID.randomUUID().toString(),
            Instant.now(),
            new Who(userId, tenantId, "agent", request.getSessionId()),
            new What("chat", request.getModelId(),
                request.getPromptVersion(),
                request.getQuestion().length(),
                response.getContent().length(),
                response.getTokenUsage()),
            citations,
            toolCalls,
            new ModerationResult(request.isFlagged(), response.isFlagged())
        );
        logStore.append(entry);
    }

    // 溯源查询：输入 request_id 查完整审计链
    public List traceByRequestId(String requestId) {
        return logStore.queryByField("request_id", requestId);
    }

    public record AuditEntry(
        String auditId, Instant timestamp,
        Who who, What what,
        List citations,
        List toolCalls,
        ModerationResult moderation
    ) {}
    public record Who(String userId, String tenantId, String role, String sessionId) {}
    public record What(String action, String model, String promptVersion,
                       int inputLen, int outputLen, Map tokenUsage) {}
    public record Citation(String chunkId, String source, String docId, double score) {}
    public record ToolCall(String tool, Map params,
                           String result, long durationMs) {}
    public record ModerationResult(boolean inputFlagged, boolean outputFlagged) {}
}

// 不可篡改存储
@Component
public class AuditLogStore {
    private static final String AUDIT_TOPIC = "corpassist-audit";

    public void append(AuditService.AuditEntry entry) {
        // 写入 Kafka 保证持久化
        kafkaTemplate.send(AUDIT_TOPIC, entry.auditId(), entry);
        // 同时写 Object Storage 做不可变副本
        objectStore.put("audit/" + entry.auditId(), entry);
    }

    // 验证审计日志是否被篡改
    public boolean verifyIntegrity(String auditId) {
        AuditEntry kafka = readFromKafka(auditId);
        AuditEntry store = readFromObjectStore(auditId);
        return kafka.equals(store);
    }
}

# 溯源报告生成
class TraceabilityReport:
    """溯源报告 —— 展示一条回复的完整生成链路。"""

    @staticmethod
    def generate(record: AuditRecord) -> str:
        lines = [
            f"## 溯源报告: {record.request_id}",
            f"- 用户: {record.user_id}",
            f"- 租户: {record.tenant_id}",
            f"- 模型: {record.model_id}",
            f"- Prompt 版本: {record.prompt_version}",
            f"- Token 用量: {record.token_usage}",
            "",
            "### 引用来源",
        ]
        for c in record.citations:
            lines.append(
                f"- {c.document_title} ({c.document_id}) "
                f"chunk={c.chunk_id} score={c.relevance_score:.2f}"
            )
        if record.tool_calls:
            lines.extend(["", "### 工具调用"])
            for tc in record.tool_calls:
                lines.append(f"- {tc['tool']}: {tc.get('result', 'unknown')}")
        return "\n".join(lines)
审计与溯源架构
flowchart LR
  U[用户请求] --> B[BFF]
  B -->|审计日志| K[Kafka]
  K -->|持久化| S[(Object Store)]
  K -->|实时| L[审计 Portal]
  B -->|RAG 引用| C[(Citation Store)]
  L -->|查询| S
  L -->|溯源| C

审计日志通过 Kafka 异步写入 Object Store 保证不可篡改。审计 Portal 提供查询和溯源能力。

不推荐
  • 审计日志可被开发人员删除
  • 只记录回复不记录引用来源
  • 日志中有完整 PII
  • 无法按 request_id 查询审计链
推荐
  • 不可篡改存储（Kafka + Object Store）
  • 记录完整的 citation chain
  • 日志脱敏后写入审计
  • 按多维度可查询
等保对 LLM 审计有什么要求？
等保 2.0 要求日志留存不少于 6 个月、日志不可篡改、支持溯源。CorpAssist 的审计方案：1) Kafka 实时写入 + Object Store 不可变备份，2) 签名校验保障完整性，3) 按 request_id/user/tenant 多维度查询。
审计日志包含 prompt 原文吗？不会被泄露吗？
审计日志存储脱敏后的输入输出（PII 被替换），不存原始 prompt。但保留了 token 数、模型版本、引用 chunk 等元数据用于溯源。如果法规要求保留原始输入（如金融行业），则需加密存储并严格限制访问权限。
RAG audit 怎么追踪知识来源？
每条回复记录其引用的知识库 chunk ID。溯源时可以看到：1) 用了哪些 chunk，2) 每个 chunk 来自哪个文档，3) 文档的版本和更新时间，4) chunk 的相关性分数。这样就形成了完整的知识溯源链。

本章结论

  • 审计格式：Who-What-When + citations + tool_calls + moderation。
  • 不可篡改：Kafka 持久化 + Object Store 备份，校验完整性。
  • 引用链：每条回复记录引用的 chunk ID / 文档 ID / 相关性分数。
  • 脱敏存储：审计日志存脱敏后的输入输出，原文不外泄。
  • 多维度查询：按 request_id / user / tenant 可追溯完整审计链。
