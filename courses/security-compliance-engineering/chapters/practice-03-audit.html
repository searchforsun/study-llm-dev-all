金融/政务客户会问：「谁何时用了模型做了什么？」没有 audit trail 无法通过等保测评，也无法复盘 injection 事件。本章把 who what when 落到双栈可查询结构。

  • 全链路：moderation 拒答、tool call、RAG retrieve 都要记。

留痕

CorpAssist · 合规调查 监管要求证明：某员工 3 月 12 日 14:02 的问答未使用未授权制度库。须从 audit 还原 retrieve 的 ACL filter 与命中 chunk 列表。
事件必填字段示例

审计事件 schema

json
{
  "event_type": "rag.retrieve",
  "who": {"user_id": "u-1024", "role": "employee"},
  "when": "2026-03-12T14:02:11Z",
  "what": {"query_hash": "a3f…", "chunk_ids": ["c-88","c-91"], "index_version": "hr-2026-v2"},
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736"
}

引用链

rag audit 要求用户可见 citation 与后台 provenance 一致：

Citation 溯源链
flowchart LR
  Q[用户问题] --> R[retrieve]
  R --> C1[chunk_id]
  C1 --> S[源文档 version]
  S --> A[audit 记录]
  A --> UI[前端 citation 链接]

若 UI 只显示「根据《员工手册》」而无 chunk_id，事后无法证明是否 ACL 越权命中。

等保

等保 2.0 对 equal protection 要求：审计日志防篡改、定期备份、访问分权。CorpAssist 实践：

  • 审计库独立账号，应用服务仅 INSERT
  • 敏感字段已 redact（见 隐私与脱敏）
  • 保留期 ≥ 6 个月（按行业上调）
yaml
audit_store:
  mode: append_only
  retention_days: 365
  access_roles: [sec_ops, compliance_readonly]
  immutability: hash_chain  # 或 WORM 对象存储

本章结论

  • 留痕：chat/moderation/tool/rag 四类事件 + who-what-when + trace_id。
  • 引用链：citation 须可复核到 chunk_id 与 index_version。
  • 等保：append-only、分权、redact 与 retention 写进策略。
