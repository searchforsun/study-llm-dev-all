政策模板

policy template 将企业价值观、合规红线与话术风格编码为可复用模板，按场景（客服、内部知识、代码助手）实例化。

政策模板分层
flowchart TB
    Global[全局基线\n禁止违法/歧视/泄露]
    Industry[行业层\n金融适当性/政务表述]
    Product[产品层\nCorpAssist 品牌话术]
    Scene[场景层\n客服 vs 内部 KB]
    Global --> Industry --> Product --> Scene

四层政策模板叠加，场景层覆盖产品默认值。

政策模板 YAML

yaml 
# corpassist-policy-customer-service.yaml
policy:
  id: cs-v2.3
  scene: customer_service
  must:
    - 仅基于检索到的知识库片段回答，无片段则拒答
    - 不承诺超出权限的退款/赔偿
    - 使用敬语，不出现竞争对手贬损
  must_not:
    - 输出完整手机号/身份证/银行卡
    - 提供医疗/法律最终结论
  tone: professional_friendly
  escalation: "无法处理时将转人工，工单号 {ticket_id}"

Python：模板渲染注入 System

python 
from pathlib import Path
import yaml

def load_policy(scene: str) -> dict:
    path = Path(f"policies/corpassist-policy-{scene}.yaml")
    return yaml.safe_load(path.read_text(encoding="utf-8"))["policy"]

def build_system_prompt(scene: str, tenant: str) -> str:
    p = load_policy(scene)
    must = "\n".join(f"- {m}" for m in p["must"])
    must_not = "\n".join(f"- {m}" for m in p["must_not"])
    return f"""你是 CorpAssist 企业助手（租户 {tenant}）。
必须遵守（{p['id']}）：
{must}
禁止：
{must_not}
语气：{p['tone']}"""

Java：Spring AI System Prompt 注入

java 
@Service
public class PolicyTemplateService {
    public SystemMessage forScene(String scene, String tenant) {
        PolicyDocument doc = policyRepository.load(scene);
        String body = String.format("""
            你是 CorpAssist（租户 %s）。政策版本 %s。
            必须：%s
            禁止：%s
            """, tenant, doc.id(), doc.mustList(), doc.mustNotList());
        return new SystemMessage(body);
    }
}

拒答

refusal guard 在输入/输出两侧触发：超范围请求、缺少依据、政策命中时返回标准拒答，并记录 refusal_code 供审计与评测。

refusal_code触发条件用户话术
OUT_OF_SCOPE意图不在 Agent 服务范围该问题超出当前助手能力，建议联系人工客服
NO_EVIDENCERAG 无相关片段知识库中未找到依据，无法确认回答
POLICY_VIOLATION输出审核命中红线抱歉，该内容不符合公司政策
SAFETY_BLOCK内容安全 API 拦截您的请求无法处理，请修改后重试
python 
REFUSAL_COPY = {
    "OUT_OF_SCOPE": "该问题超出 CorpAssist 当前服务范围，已为您转接人工入口。",
    "NO_EVIDENCE": "未在知识库检索到可靠依据，无法给出确认性答复。",
    "POLICY_VIOLATION": "抱歉，该回复不符合企业内容政策（代码 POLICY_VIOLATION）。",
}

def refuse(code: str, audit_ctx: dict) -> str:
    audit_ctx["refusal_code"] = code
    audit_ctx["refusal_at"] = datetime.utcnow().isoformat()
    return REFUSAL_COPY[code]
不推荐
  • 拒答时返回空字符串或 500 错误
  • 不同渠道拒答话术不一致
  • 不记录 refusal_code，无法做 badcase 分析
推荐
  • 标准话术 + refusal_code + trace_id
  • 与内容安全检测 Advisor 链串联
  • 拒答样本进入红队回归集

非 RLHF

应用层对齐不依赖重新训练基座模型。non rlhf alignment 组合：政策模板 + 输入/输出 Advisor + 规则引擎 + 可选 NeMo Guardrails colang 流程。

Java · Spring AI Output Advisor
java
@Component
public class PolicyOutputAdvisor implements CallAroundAdvisor {
    @Override
    public AdvisedResponse aroundCall(AdvisedRequest req, CallAroundAdvisorChain chain) {
        AdvisedResponse resp = chain.nextAroundCall(req);
        String text = resp.response().getContent();
        if (contentSafetyClient.isBlocked(text)) {
            return AdvisedResponse.from(
                new ChatResponse(List.of(new Generation(
                    new AssistantMessage(refusalService.message("POLICY_VIOLATION"))))));
        }
        return resp;
    }
}
检查理解：政策模板与 RLHF 的分工边界？

RLHF/DPO 在模型厂商侧改善通用安全；企业落地用政策模板 + Advisor + 拒答码实现可审计、可版本化的价值观约束，无需接触基座权重。

政策变更如何灰度？
policy id semver 化；按 tenant 或流量百分比加载新版本；对比 refusal_rate 与 CSAT 再全量。
拒答与内容安全 API 重复吗？
不重复。内容安全处理违法违规；拒答处理超范围、无依据、政策话术。两者串联，refusal_code 可区分 SAFETY_BLOCK vs POLICY_VIOLATION。

拒答率监控指标

  • refusal_total{code} — 按 refusal_code 分布，异常抬升触发政策复盘。
  • policy_version_active — 当前生效 policy_id，与 audit 字段交叉校验。
  • guardrail_latency_ms — Advisor 链延迟，避免同步审核拖垮 P95。
CorpAssist · 无依据拒答 用户问「明年差旅标准会涨吗？」知识库无 2027 政策。Agent 应返回 NO_EVIDENCE 拒答，而非编造数字。评测集应包含此类「应拒答」样本。

Spring AI Advisor 链推荐顺序：输入清洗 → RAG → 模型 → PII 脱敏 → refusal guard 输出审核。与 隐私与脱敏 一致：先掩码再政策审核，避免拒答模板或审计日志接触明文 PII。Python LangChain Runnable 序列保持同序，便于双栈对照排障。

与内容安全 Advisor 串联

输入侧调用内容安全检测 API 拦截违法违规；输出侧 policy template 约束话术边界。两层失败分别映射 SAFETY_BLOCK 与 POLICY_VIOLATION，避免混为一谈。

  • 输入 block → 不进入 RAG/模型，直接 SAFETY_BLOCK 拒答。
  • 模型输出违规 → POLICY_VIOLATION，可记录模型版本供 badcase 分析。
  • 无检索依据 → NO_EVIDENCE，与政策违规区分统计。
拒答决策流
flowchart TD
    In[用户输入] --> InSafe{输入安全?}
    InSafe -->|否| R1[SAFETY_BLOCK]
    InSafe -->|是| RAG[RAG 检索]
    RAG --> Ev{有依据?}
    Ev -->|否| R2[NO_EVIDENCE]
    Ev -->|是| LLM[模型生成]
    LLM --> OutSafe{输出合规?}
    OutSafe -->|否| R3[POLICY_VIOLATION]
    OutSafe -->|是| OK[正常回复]

三类拒答码对应不同防护层，metrics 分桶统计。

NeMo Guardrails colang 适合对话流分支；与 Spring AI PolicyOutputAdvisor 组合时，保持 refusal_code 枚举一致便于双栈 metrics 对齐。

政策模板中 must_not 变更属于安全变更，需同步更新红队 alignment 用例与 SEC-03 验收项。

双栈拒答话术文案存于共享 refusal-messages.yaml，避免 Python/Java 对用户表述不一致。

内容安全 API 超时时的降级策略：默认 POLICY_VIOLATION 拒答，禁止 silent fail 放行。

评测集应包含「应拒答」与「应回答」成对样本，防止政策过严导致正常咨询被 NO_EVIDENCE 误杀。
