政策模板

CorpAssist · S5-A 营销 生成朋友圈文案时，政策要求：不得贬损竞品、不得未验证库存承诺、须使用品牌称谓「CorpAssist 伙伴」而非「亲」。
yaml
# policies/marketing-v2.yaml
policy_version: marketing-v2
brand_tone:
  address: "CorpAssist 伙伴"
  forbidden_phrases: ["全网最低", "100% cure", "竞品XXX垃圾"]
refusal_templates:
  illegal_request:
    message: "该请求涉及违规内容，我无法协助。如需帮助请联系人工客服。"
    reason_code: REFUSE_ILLEGAL

Spring/Python 启动时加载同一 YAML；Git tag 对应发布版本。

拒答

refusal policy 分三类：硬拒（违法）、软拒（超出业务范围）、转人工（低置信/high risk）。

类型HTTP用户可见审计字段
硬拒400/403refusal_templates.*reason_code, policy_version
软拒200 + 模板建议替代问题soft_refuse=true
转人工200 + handoff排队提示handoff_queue
反模式
  • 模型自由发挥拒答话术，每次不同
  • 拒答不写 audit
推荐
  • 模板 + 少量 LLM 润色（仍过 outbound 检）
  • 拒答率进 Dashboard

非 RLHF

应用侧 value align 手段：

  • Policy + 规则引擎：关键词、正则、库存 API 真查（S5 production pitfall）
  • 分类器：小模型/Moderation API 做 outbound scan
  • 偏好数据：坏例进 golden set → 发版前回归（NS1，非本章训练 RLHF）
java
@Component
class OutputGuardrailAdvisor implements CallAdvisor {
  public ChatClientResponse after(ChatClientResponse resp, AdvisorChain chain) {
    GuardResult g = guard.check(resp.content(), policyVersion);
    if (g.violated()) return resp.mutate(refusalFactory.from(g.reasonCode()));
    return resp;
  }
}
先想 10 秒：DPO/RLHF 与 output guardrail 分工？

训练提升模型默认行为；guardrail 是最后一道可审计闸，发版可独立变更 policy_version。

本章结论

  • 政策模板：YAML 版本化，双栈同载；含 brand_tone 与 refusal_templates。
  • 拒答：硬/软/转人工三类 + reason_code 审计。
  • 非 RLHF：规则+分类器+评测回归；guardrail 可独立发版。
