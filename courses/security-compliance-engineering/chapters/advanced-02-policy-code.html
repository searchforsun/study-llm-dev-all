为什么要学本章

ToB 交付最怕「安全靠人肉 checklist」：上线后改一句拒答策略就要发版。把 policy as code 与 Advisor/Guardrails 解耦，策略可评审、可回滚、可在 CI 阻断不合规配置。

学完你能
  • 编写 opa rego 策略并本地 opa test
  • 发布 guard bundle 供双栈加载
  • 在流水线启用 ci policy gate
  • 把 Spring/Python 规则映射到 corpassist policy 统一 ID
本章先记住 3 件事
  • 策略与业务代码分离：Rego/规则 YAML 独立仓库或目录，签名发布。
  • 默认拒绝：工具调用、外呼模型、跨境传输均 fail-closed。
  • 测试即验收：每个控制项至少 1 条 allow + 1 条 deny 用例。

OPA

opa rego 适合描述「在给定输入 JSON 下是否允许某操作」。CorpAssist 在准入阶段把请求上下文（租户、用户角色、数据分级、工具名、模型端点）序列化为 input，由 OPA 决策。

策略即代码在 CorpAssist 流水线中的位置
sequenceDiagram
    participant Client as 客户端
    participant GW as API 网关
    participant OPA as OPA / 策略引擎
    participant App as CorpAssist 服务
    participant LLM as 模型/工具

    Client->>GW: 请求 + JWT
    GW->>OPA: input 包（租户/分级/工具）
    alt deny
        OPA-->>GW: 拒绝 + 原因码
        GW-->>Client: 403 POLICY_DENY
    else allow
        OPA-->>GW: allow + 义务(obligation)
        GW->>App: 转发 + 义务头
        App->>LLM: Advisor 链二次校验
        LLM-->>App: 响应
        App-->>Client: 响应 + 审计 ID
    end

OPA 在网关或服务侧做粗粒度准入；Advisor 链做细粒度内容/PII 处理。

Rego：工具白名单与数据分级

rego 
# policies/corpassist/tool_authz.rego
package corpassist.tool

default allow = false

allow {
  input.tenant == data.tenants[input.tenant].name
  tool := input.tool_name
  tool in data.tenants[input.tenant].allowed_tools
  input.data_class != "STATE_SECRET"
}

deny[msg] {
  input.data_class == "STATE_SECRET"
  msg := "STATE_SECRET 禁止进入 LLM 流水线"
}

deny[msg] {
  input.tool_name == "transfer_funds"
  not input.approvers
  msg := "资金类工具需要双人审批 approvers"
}

rego test 在 CI 中执行：opa test -v policies/，与 admission webhook（K8s 部署侧）或 API 网关插件（请求侧）配合，形成双层门禁。

先想 10 秒：若 OPA 返回 allow 但 Advisor 仍拒答，这算策略冲突还是纵深防御？你会如何记录审计？

不推荐
  • 在 Java/Python 业务里散落 if-else 合规逻辑
  • 无测试的 Rego，上线后才发现 deny 过宽
  • 策略与模型 Prompt 混在同一文件
推荐
  • 策略包版本号写入审计与响应头 X-Policy-Bundle
  • 每租户 overlay 只改 data，不改核心 rego
  • OPA + Advisor 职责清晰，冲突以 stricter 为准

规则

并非所有控制都适合 Rego。内容安全、越狱模式、PII 正则、拒答话术适合 guard bundle（YAML/JSON 规则 + NeMo/Spring 加载器）。policy as code 在此指「可 diff、可签名、可回滚」的规则制品，而非单指 OPA。

双栈加载同一 bundle

java 
@Configuration
public class PolicyBundleConfig {
  @Bean PolicyBundleClient bundleClient() {
    return PolicyBundleClient.fromUrl("https://policy.corp.internal/bundles/latest");
  }
  @Bean Advisor policyAdvisor(PolicyBundleClient c) {
    return new BundleDrivenAdvisor(c.control("CTRL-PII-02"));
  }
}
python 
bundle = GuardBundle.load("https://policy.corp.internal/bundles/latest")
chain = (
    bundle.nemo_rail("CTRL-INJ-01")
    | bundle.langchain_mask("CTRL-PII-02")
    | bundle.opa_sidecar("CTRL-TOOL-03")
)

corpassist policy 要求每个 CTRL-* 在合规矩阵有行业列（金融/政务）、双栈实现列与测试用例 ID。ci policy gate 在 MR 阶段执行：

  1. opa test 全绿
  2. 规则包签名验证
  3. 与上一版本 diff 生成「影响面报告」（哪些租户 overlay 受影响）
  4. 红队用例子集回归（见 红队与演练）

CI 流水线策略门禁（GitLab 摘录）

yaml 
stages: [policy, test, publish]

policy:opa:
  script:
    - opa test -v policies/
    - opa eval --fail-defined data.corpassist.deny < input/fixtures/attack.json

policy:bundle:
  script:
    - node scripts/verify-bundle-signature.mjs bundle/manifest.yaml
    - node scripts/diff-bundle.mjs --base main --head $CI_COMMIT_SHA

policy:redteam-subset:
  script:
    - python run_redteam_subset.py --cases inj_tool,piilog,policy_bypass
  allow_failure: false

发布阶段将 bundle_id 写入容器环境变量 CORPASSIST_POLICY_BUNDLE，Advisor 启动时校验签名并热加载。回滚时切换上一签名版本，审计中保留两个版本并存窗口的可观测指标。

OPA 与 NeMo Guardrails 会不会重复？
会部分重叠，但层次不同：OPA 擅长结构化准入（工具、租户、分级）；Guardrails 擅长对话流与 Colang 叙事。重复时以更严格结果为准，并在审计中记录两层决策。
策略热更新如何做？
bundle 拉取 + 版本头；禁止无版本静默覆盖。金融/政务常要求变更工单号写入 X-Change-Ticket，并与 spring advisor policy 缓存 TTL 对齐（例如 60s 内新旧并存需可观测）。

发布说明模板应包含：bundle_id、变更工单、影响租户列表、回滚步骤与红队回归结果链接——便于 corpassist policy 变更审计。

策略评审会议应出席：应用安全、法务、SRE、双栈负责人；会议输出 signed manifest 与回归用例列表，作为投产门禁附件。

本章结论

  • OPA：结构化准入用 Rego + 测试；与网关/Admission 集成，默认 deny。
  • 规则：内容/PII/注入用 guard bundle 统一 ID，双栈同版本加载，CI 签名与 diff 门禁。

编写一条 Rego 工具策略与对应测试，并组装最小 guard bundle manifest（约 35 分钟）。

操作步骤

  1. 在 tool_authz.rego 上为 transfer_funds 增加「必须含 approvers」的 deny 规则。
  2. 补充 test_deny_transfer_without_approvers 并在本地运行 opa test -v policies/。
  3. 在 manifest.yaml 为 CTRL-TOOL-03 填写 bundle_id 与签名占位。
  4. 可选：完成 demos/advanced-02-policy-code-lab/ 验收。

判断练习

  1. （找错）下列 manifest 片段哪项违反策略即代码最佳实践？A) 含 bundle_id B) 无 signature 字段 C) controls 含 CTRL-ID D) 文件路径相对 bundle 根

    参考答案

    B。生产 bundle 必须可验证签名；无签名无法防篡改，CI 应 fail。
