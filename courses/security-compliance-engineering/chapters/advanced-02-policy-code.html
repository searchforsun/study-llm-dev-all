本章先记住 3 件事
  • 默认拒绝：Rego 中 default allow = false，显式 allow 才放行。
  • 策略与代码分离：业务服务只问 OPA「这次允许吗」，不在 Controller 里堆规则。
  • 版本进审计：每条 LLM 请求日志带 policy_version，监管问询可复现当时规则。

OPA

CorpAssist · Agent 调用 propose_refund（45 秒） 一线客服角色 agent 在处理客诉时，模型想调用退款提案工具。安全要求：仅 agent 且客诉工单状态为 open 且金额 < 500 元可调用；user 角色永远不可。规则每月由合规部修订，需 PR 评审而非热补丁 Java 代码。

Open Policy Agent（OPA） 是 CNCF 毕业项目，用声明式策略回答：给定 input，allow 还是 deny？ CorpAssist 在 Spring 网关与 Python Agent 执行器前各设 OPA sidecar 或嵌入式 SDK，统一策略源。

OPA 在 CorpAssist 中的挂载点

挂载点input 示例决策
HTTP 网关 user_id, role, sector, route 是否允许访问 /chat、/admin
Tool 执行前 role, tool_name, ticket, amount 是否 allow 工具调用
RAG 检索 role, kb_id, doc_classification 是否允许检索该 collection
出站 LLM sector, model_endpoint, pii_detected 是否允许调用外网模型

Spring 侧调用 OPA 示例

yaml 
# deploy/opa-sidecar.yaml（节选）
services:
  corpassist-gateway:
    opa:
      url: http://localhost:8181
      policy_path: /v1/data/corpassist/authz/allow
      bundle:
        resource: corpassist-policies
        version: "1.4.2"
    audit:
      include: [policy_version, decision_id, input_hash]
先想 10 秒：为何不用 Spring Security 表达式替代 OPA？

Security 表达式适合单一应用 RBAC；CorpAssist 双栈（Java + Python）、多挂载点、行业 sector 组合爆炸，且金融客户要独立审策略文件。OPA 策略可导出、可 diff、可 ci policy 测，更符合 policy as code。

规则

Rego 是 OPA 的策略语言。CorpAssist 工具授权包路径为 corpassist.authz；完整样例见 demos/advanced-02-policy-code-lab/policy.sample.rego，正文展开如下。

CorpAssist 工具授权 Rego

rego 
# demos/advanced-02-policy-code-lab/policy.sample.rego（扩展版）
package corpassist.authz

import future.keywords.if

default allow := false

# 基线：已认证用户可调只读 FAQ 工具
allow if {
    input.role == "user"
    input.tool == "faq_search"
}

# 客服可调订单查询
allow if {
    input.role == "agent"
    input.tool == "query_order"
}

# 退款提案：角色 + 工单 + 金额上限
allow if {
    input.role == "agent"
    input.tool == "propose_refund"
    input.ticket.status == "open"
    input.ticket.amount < 500
}

# 金融加强：高风险工具需审批 token
allow if {
    input.sector == "finance"
    input.tool == "propose_refund"
    input.role == "agent"
    input.approval_token != ""
    valid_approval_token(input.approval_token)
}

# 管理员显式放行（仍需审计）
allow if {
    input.role == "admin"
    input.tool in data.corpassist.admin_tools
}

valid_approval_token(tok) if {
    data.corpassist.approval_tokens[tok].expires_at > time.now_ns()
}

策略数据与 sector 规则（YAML）

yaml 
# data/corpassist/policy-data.yaml — 随 bundle 发布
corpassist:
  admin_tools: [force_refund, rebuild_index, export_audit]
  approval_tokens: {}  # 运行时由工单系统注入
  sector_rules:
    finance:
      block_tools: [export_all_users]
      require_policy_version_min: "1.4.0"
    gov:
      block_model_hosts: [api.openai.com, api.anthropic.com]
      allow_model_hosts: [dashscope-vpc.internal, vllm-gov.local]

CI 策略测试

rego 
# policy_test.rego
package corpassist.authz_test

test_user_cannot_propose_refund if {
    not allow with input as {
        "role": "user",
        "tool": "propose_refund",
        "ticket": {"status": "open", "amount": 100}
    }
}

test_agent_small_refund_allowed if {
    allow with input as {
        "role": "agent",
        "tool": "propose_refund",
        "ticket": {"status": "open", "amount": 200}
    }
}
不推荐
  • Java / Python 各维护一份工具白名单
  • 生产环境手工改 OPA 规则无 Git 记录
  • 无 default allow = false，漏写规则即全放行
推荐
  • 单 repo corpassist-policies；tag = policy version
  • PR 必过 opa test + 合规 Reviewer approve
  • 审计日志写 decision_id 可关联 OPA decision log
常见误区
  • OPA = 只做 RBAC — 还可表达 sector、PII 级别、模型 endpoint 等任意结构化 input。
  • 策略越细越好 — 过度细粒度 Rego 难测；静态数据放 YAML bundle，Rego 只做决策逻辑。
  • deny 不写原因 — 生产应返回 deny_reason（内部）便于客服排障，但不暴露给终端用户。

本章结论

  • OPA：网关、Tool、RAG、出站模型统一策略决策点；bundle 版本化发布。
  • Rego：default deny；角色 / 工具 / sector 组合规则；data YAML 存静态配置。
  • CI：opa test 回归；policy_version 进审计，满足金融可追溯。

判断练习

  1. （场景）开发在 Java 里加了 if (role.equals("admin")) allowAllTools()，未同步 Rego。会有什么问题？

    参考答案

    双栈策略漂移：Python Agent 仍按 Rego deny，Java 侧却放行，审计 policy_version 也对不上。应删除 Java 硬编码，统一走 OPA；admin 工具列表只维护在 Rego data。
