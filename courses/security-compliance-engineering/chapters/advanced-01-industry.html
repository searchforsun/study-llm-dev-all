已完成行业合规差异

约 40 分钟 · 阶段：合规交付 · 能力：按行业映射 CorpAssist 控制项与验收口径

为什么要学本章

同一套 CorpAssist 能力在金融与政务客户现场的合规基线并不相同：数据分级、留痕年限、模型调用边界、人工复核点都不一样。本章把「通用 OWASP/Advisor 控制」翻译成可落地的行业差异表，避免用互联网 SaaS 经验硬套 ToB 招投标与等保测评。

学完你能
  • 用 fin reg compliance 清单对照金融客户验收项
  • 按 gov data classification 配置政务数据分级与出域策略
  • 为 CorpAssist 选择 industry guardrail 模板而非一刀切拒答
  • 在方案书中写明 mlps level 与测评边界
本章先记住 3 件事
  • 行业差异在「数据与流程」：模型能力相近，但谁能看谁的数据、留多久、谁审批差异最大。
  • 双栈要对齐同一控制项 ID：Java Advisor 与 Python Guardrails 必须映射到同一张合规矩阵行。
  • 验收用证据链：测评不看 PPT，看配置导出、审计样本与红队报告。

金融

CorpAssist · 某城商行智能客服 客户要求：交易类问答不得直连公网大模型；客户号、卡号、余额类字段必须脱敏；投资建议类输出需免责声明 + 人工复核队列；所有 Prompt/Completion 审计保留 ≥5 年。你需要在现有 Advisor 链上叠加行业控制，而不是重写业务代码。

fin reg compliance 在 LLM 应用语境下，通常叠加：个人信息保护法、网络安全法、数据安全法，以及人民银行/金监总局关于生成式 AI 与金融科技的风险提示。工程上要把「条文」落成可检测的控制项。

金融场景：控制重心在接入身份、PII 与模型路由；公网模型默认关闭。

金融监管报送与 LLM 日志字段

部分机构要求关键操作留痕字段包含：tenant_id、user_role、model_version、policy_bundle、tool_name、decision（allow/deny）、trace_id。CorpAssist 审计管道应在写入 WORM 前校验字段完整性，缺失则 fail-closed 并告警。

  • 拒答类事件单独计数，纳入「客诉与合规」周报。
  • 外呼例外工单必须关联 trace_id 列表，方便监管抽查。
  • 双栈写入同一审计 schema，避免 Java 与 Python 字段名不一致导致测评失败。

政务

政务与央企场景强调 gov data classification（国家秘密 / 工作秘密 / 内部 / 公开）与 dual approval：敏感事项常要求双人审批 + 留痕不可篡改，且对外网模型默认「物理不可达」。

政务 CorpAssist 数据分级决策

数据样例建议分级LLM 处理策略
公开政策 FAQ、办事指南公开可经内网模型；输出需引用公文编号
办事人手机号、办件编号内部脱敏后入模；检索库与模型同安全域
未公开批示、会议纪要工作秘密禁止入公网 RAG；仅隔离区小模型 + 人工发布
标密文件片段国家秘密禁止进入 CorpAssist 流水线

sector policy 在政务侧常体现为「办事指南准确性」与「意识形态与价值观」双底线：industry guardrail 需内置拒答政治敏感、谣言溯源，并强制引用权威来源片段。

政务上线前 10 项工程检查（CorpAssist）

  1. 数据分级策略已在 OPA/租户 YAML 生效，block_levels 含国家秘密。
  2. 模型 endpoint 仅内网 DNS，公网 egress 防火墙 deny-all 已抓包验证。
  3. RAG 语料发布流程含「人工批准 + 公文编号」字段，未批准文档不可索引。
  4. 拒答与转人工队列值班表与 SLA 已录入运维系统。
  5. 审计日志写入 WORM，保留策略与等保测评表一致。
  6. 红队用例覆盖「诱导泄露内部批示」「伪造政策解读」类 Prompt。
  7. 终端 DLP 阻止粘贴到公网 ChatGPT 的 CorpAssist 替代入口已开通。
  8. 双人审批对接 OA：敏感工具调用写回 approver_id。
  9. 灾备演练：专网模型不可用时只读 FAQ 模式，不降级到公网。
  10. 投标资料包含本检查表签字页与抽样审计截图索引。

政务客户能否用「混合云 + 公有云大模型」？
多数项目要求推理与向量库均在政务外网/专网区；若政策允许「非涉密数据」外呼，也需单独评估、单独租户与 DLP，且默认关闭工具写操作。方案阶段应写清「不可达」而非「不建议」。
等保 2.0 与 LLM 控制项如何对应？
mlps level 三级常见映射：身份鉴别（网关 MFA）、访问控制（RBAC）、安全审计（全链路 trace + 不可变存储）、入侵防范（WAF + 注入检测）、数据完整性（配置签名）。LLM 特有项用扩展表挂在「安全计算环境」测评项下。

本章结论

  • 金融：重心在 PII 驻留、模型路由、长周期审计与交易类工具四眼审批；验收看配置与 WORM 样本。
  • 政务：重心在数据分级、专网模型、引用溯源与敏感事项人工队列；涉密数据必须物理隔离不入流水线。

判断练习

  1. （场景）客户要求模型推理不出境，但希望用海外开源 Embedding。最合规的处置顺序是？A) 禁用 Embedding B) 仅境内镜像 + 向量不出境 C) 继续用海外 API 但签 DPA D) 不做变更

    参考答案

    B。Embedding 仍属数据处理；金融/政务常要求向量化与检索同域。若政策允许非敏感元数据外呼，也需单独评估与 DLP，不能默认 C。
