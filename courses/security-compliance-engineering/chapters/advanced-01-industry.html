为什么要学本章

CorpAssist 同一套产品要卖给银行、券商、省厅、央企——基线安全相同，控制强度与部署形态不同。若用「通用 SaaS 一套配置」硬推，金融客户嫌审计链不够，政务客户嫌数据可能出境。本章从应用架构视角梳理 industry diff，让你能写行业差异表、选对部署与留痕策略。

学完你能
  • 对比金融与政务在 finance compliance / gov compliance 下的控制差异
  • 为 CorpAssist 设计 sector policy 配置层（审计留存、脱敏、部署）
  • 识别 regulated data 在各行业的边界与双栈落地要点
  • 向客户说明「基线 + 行业加强包」而非 fork 代码
本章先记住 3 件事
  • 行业差异在配置与运维，不在重写 LLM 核心；用 industry control 开关加强项。
  • 金融重可追溯：每笔 Agent 工具调用、每次 RAG 命中都要能 financial audit 回放。
  • 政务重边界：gov cloud / 专有云、数据不出域、模型调用链路可证明。

金融

CorpAssist · 城商行智能客服（60 秒） 员工通过 CorpAssist 查「理财产品风险等级」并触发 query_product 工具。监管要求：谁问了什么、模型引用了哪份制度、是否给出投资建议，全部可审计 5 年。若日志缺字段或 PII 明文，等保与银保监会现场检查会直接亮红灯。

金融行业的 finance compliance 对 LLM 应用的核心诉求是可证明、可回放、可问责，而非「模型更聪明」。

金融加强控制清单

控制域通用基线金融加强
审计留痕 请求 ID + 脱敏摘要 全链路 trace：prompt hash、RAG doc_id、tool 参数签名、策略版本
内容安全 入出站 moderation 禁止投资建议类输出；敏感词库含监管禁语

CorpAssist 金融行业配置示例

yaml 
sector: finance
controls:
  audit_retention_days: 1825
  require_trace_fields: [user_id, session_id, rag_doc_ids, policy_version, tool_calls]
  output_guard:
    block_categories: [investment_advice, insider_hint]
  tool_authz:
    high_risk: [transfer_funds, modify_limit]
    require_approval_token: true
pii:
  mask_before_log: true
  fields: [id_card, bank_account, phone]

双栈落地：Spring AI Advisor 在 Java 网关写结构化审计；Python 护栏服务做 PII 检测与出站过滤，两栈共享 policy_version 便于监管问询时对齐。

先想 10 秒：为何金融客户常要求「prompt 不落盘明文」？

员工问题可能含客户账号、投诉细节；明文 prompt 既是 PII 又是商业敏感信息。应存 hash + 脱敏摘要，原文仅在授权调查窗口短时解密（见 隐私与脱敏）。

政务

政务场景的 gov compliance 强调数据主权、网络边界、供应链可控。CorpAssist 常见部署在政务云或客户机房，模型调用不能默认走公网 OpenAI。

政务 vs 金融：差异焦点

维度金融政务
部署 私有云 / 金融云专区 gov cloud、信创环境、物理隔离
模型 可混合：内网小模型 + 合规外采 优先国产化；外采需安全评估与备案
数据出境 严格但可能有跨境业务例外流程 原则上不出境；RAG 文档不出域
审计 交易级细粒度 操作留痕 + 等保测评证据链

政务部署检查要点

  • 向量库与对象存储 均在政务云 VPC；Milvus / MinIO 无公网 endpoint。
  • LLM 调用 走内网 DashScope 专线或本地 vLLM；网关层阻断未备案域名。
  • regulated data — 公民身份信息、内部公文标注密级，RAG 分库 + ACL（见 审计与留痕）。
  • 运维 补丁与镜像扫描纳入等保；红队用例在专网 CI 回归（见 红队与回归）。
不推荐
  • 为金融、政务各 fork 一套代码仓库
  • 行业差异只写在售前 PPT，工程无配置项
  • 政务项目仍默认调用境外 API「先上线再说」
推荐
  • sector 配置 + Feature Flag 控制加强项
  • 行业差异表进 Git，与 industry control 版本一起审计
  • 售前 Demo 与交付配置一致，避免「Demo 用 SaaS、交付要专有云」
常见误区
  • 等保 = 买防火墙 — LLM 应用的注入、越权、日志脱敏同样进测评范围。
  • 政务 = 完全离线 — 多数仍需内网模型服务；关键是边界可证明，而非绝对断网。
  • 金融审计 = 多打日志 — 日志需结构化、防篡改、与业务单据关联，不是 stderr 堆砌。

本章结论

  • 金融：可追溯与长留存；高风险工具复核；输出禁投资建议类内容。
  • 政务：专有云 / 不出域；信创与备案；RAG 与模型链路内网闭环。
  • 工程：基线 + sector 加强包；差异配置化、版本化，不 fork 产品代码。

    参考答案

    不满足。rerank 会把 query 与 chunk 文本发往外部 API，属于数据出境/出域。应改为内网 rerank 模型或取消 rerank，并在 DPIA 中说明链路。
