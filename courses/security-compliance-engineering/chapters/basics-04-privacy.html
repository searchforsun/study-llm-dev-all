为什么要学本章

LLM 应用天然处理自然语言，身份证、手机号、薪资、病历极易进入 prompt、向量库与 trace 日志——对应 OWASP LLM06。阶段二B 要求 basics 阶段建立 pii detect 与 log redaction 基线，否则 S2/S5 场景课无法通过合规门禁。本章贯穿 Spring Advisor 预扫描与 Python NER 双栈。

本章先记住 3 件事
  • 进模型前先脱敏：能不进 LLM 的 PII 就不要进；必须进则 token 化或掩码。
  • 日志默认有毒：OpenTelemetry span、prompt debug 是泄露高发区。
  • 存储最小化：对话历史、embedding、审计各设 retention policy，过期自动 purge。

PII

CorpAssist · HR 助手（30 秒） 员工问：「帮我查一下张三的手机号和身份证后四位。」若系统把完整 gdpr pii 喂给 LLM 再回答，模型可能在回复中复述全量 PII，同时写入 trace——一次问答触发双重泄露。

CorpAssist PII 分级（示例，企业可扩展）：

级别字段示例进 LLM 策略检测手段
L0 公开 公司名、部门公开电话 允许 —
L1 内部 工号、办公分机 掩码后可用 正则 + 词典
L2 敏感 手机号、邮箱、地址 默认掩码；高敏场景拒答 正则 + NER
L3 高敏 身份证、银行卡、薪资、病历 禁止进 prompt；工具层授权查询 NER + 规则 + 人工复核

双栈 PII 检测链路

PII 从输入到存储的管控点
flowchart LR
  IN[用户/RAG 输入] --> PRE[Spring PII Advisor]
  PRE -->|mask| LLM[(LLM)]
  PRE -->|block L3| REJ[拒答]
  LLM --> OUT[出站 PII 扫描]
  OUT --> USER[用户可见]
  PRE -.-> PY[Python NER 复核]
  OUT -.-> LOG[log redaction]
  LOG --> STORE[(privacy storage)]

L3 字段应在进 LLM 前 block；出站扫描防模型「补全」泄露。

python
# Python — PII 检测与掩码（示意）
import re
PHONE = re.compile(r"1[3-9]\d{9}")

def desensitize(text: str, policy: PiiPolicy) -> str:
    entities = ner.detect(text)  # 双栈：Spring 正则 + Python NER
    for e in sorted(entities, reverse=True):
        if e.label in policy.block_labels:
            raise PiiBlockedError(e.label)
        text = text[:e.start] + policy.mask(e) + text[e.end:]
    return text
先想 10 秒：为什么 RAG 索引阶段也要做 PII 扫描？

向量库可能长期保留明文 chunk；检索命中即把 PII 拉回上下文。应在 ingest 阶段 desensitize 或打 acl_tags 限制可见范围，而非只在问答时临时处理。

日志

LLM 应用的 observability 栈（OpenTelemetry、Langfuse、自研 trace）常默认记录完整 prompt/completion，是 OWASP LLM06 最高频踩坑点。CorpAssist 要求：

日志类型风险redaction 规则
请求 trace 含用户原文 PII 记录 hash + 掩码片段；L3 仅记 event 不记内容
工具调用 log API 返回全量 JSON 字段级 allowlist；secret scrub API Key
评测/坏例集 复制生产对话 入库前 batch desensitize；隔离环境
错误栈 异常 message 带 prompt 全局 log appender 过滤
不推荐
  • debug 环境开全量 prompt 日志上生产
  • 仅人工约定「工程师别看 PII」
  • 日志无 retention，永久保存对话
推荐
  • 统一 log redaction SDK（Spring Filter + Python processor）
  • 结构化日志：pii_detected=true 但不写原文
  • 访问日志与内容分离，RBAC 查原文需审批
java
// Spring — 日志写入前 redaction（示意）
log.info("chat_complete trace={} pii_flags={} prompt_len={}",
    traceId,
    piiScanner.flags(redactedPrompt),  // 仅标签，非原文
    redactedPrompt.length());
判断：「为了排障，临时打开 24 小时全量 prompt 日志」——合规吗？

高风险。须走变更审批、限定访问 RBAC、到期自动关闭，且仍应经 redaction。生产默认禁止全量明文 prompt；可用 sampling + 加密 vault 存极少数样本。

存储

privacy storage 涵盖对话历史、向量 chunk、审计库、缓存。原则：最小必要、加密、可删除。

数据类推荐保留期加密删除触发
在线会话 cache 24–72 h 内存/TLS TTL 或用户注销
对话审计（脱敏后） 按等保/行业 1–3 年 静态加密 retention job
向量 chunk（含 PII） 随源文档生命周期 collection 级 源文档退役级联删
评测样本 项目期 + 90 d 隔离库 批次 purge
yaml
# corpassist-privacy-retention.yaml
retention:
  chat_session_hours: 48
  audit_log_days: 365
  eval_sample_days: 90
storage:
  pii_in_vector: deny_l3   # L3 不得入 Milvus 明文
  encryption_at_rest: required
  user_erase:
    cascade: [session, audit_pointer, feedback]
用户行使「删除权」时 LLM 侧要删什么？
会话、可关联的 audit 指针、用户反馈/评测条目；向量 chunk 若含其 PII 须 re-ingest 或 tombstone。已进模型 provider 日志的部分需合同/DPA 约定，CorpAssist 侧仍须停止继续存储。
脱敏后数据还能不能做 RAG？
可以，但语义可能损失（如手机号掩码后无法精确匹配）。高敏字段应通过授权工具查询而非 RAG 明文召回，与 工具权限 联动。

本章结论

  • PII：L0–L3 分级；进 LLM 前检测/掩码/拒答；双栈 NER + 正则。
  • 日志：默认 redaction；禁止生产全量明文 prompt；结构化记 flags 非原文。
  • 存储：最小保留、加密、级联删除；L3 不进向量明文。
