为什么要学本章

CorpAssist 对外 API 仍是 REST/SSE（经 Gateway），但 BFF 与 Python 之间高频 RAG 检索、批量 embedding 可能成为瓶颈。grpc integration 在内网提供更低延迟与强类型契约；本章讲何时值得上，避免「全链路 gRPC」过度工程。

本章先记住 3 件事
  • 对外 REST：浏览器、第三方集成、OpenAPI 文档 — 走 Gateway。
  • 对内 gRPC：BFF ↔ Python 低延迟、performance rpc、双向流可选。
  • LLM 流式：用户侧仍 SSE；内网可用 server streaming 传 token chunk（与 SSE 流式端到端 对照）。

proto 定义：契约先行

CorpAssist S2 · BFF 调 Python RAG（30 秒） 用户经 Gateway 访问 /v1/chat（REST）。BFF 需内网调用 Python 做 retrieve + generate。OpenAPI HTTP 够用，但当 retrieve 要传大 batch、或要 server streaming 回传 token 时，proto define 的强类型与 HTTP/2 多路复用更有优势。

protobuf 是 gRPC 的 IDL。CorpAssist 约定：proto 文件放 contracts/proto/corpassist/v1/，Java 与 Python 共用同一套生成代码，CI 校验 breaking change。

RagService.proto（节选）

protobuf 
syntax = "proto3";
package corpassist.v1;

option java_package = "com.corpassist.contracts.v1";
option java_multiple_files = true;

message RetrieveRequest {
  string tenant_id = 1;
  string query = 2;
  int32 top_k = 3;
  string collection_prefix = 4;
}

message RetrieveHit {
  string doc_id = 1;
  string snippet = 2;
  float score = 3;
}

message RetrieveResponse {
  repeated RetrieveHit hits = 1;
}

message ChatRequest {
  string tenant_id = 1;
  string session_id = 2;
  string user_message = 3;
  repeated RetrieveHit context = 4;
}

message ChatChunk {
  string delta = 1;
  bool finished = 2;
  int32 token_count = 3;
}

service RagService {
  rpc Retrieve(RetrieveRequest) returns (RetrieveResponse);
  rpc StreamChat(ChatRequest) returns (stream ChatChunk);
}

RetrieveUnary向量检索，低延迟 batch query
StreamChatServer streaming内网 token 流；BFF 再转 SSE 给客户端

flowchart TB
  Client[App / Web]

  Client -->|REST / SSE| GW

Gateway 终结 HTTP/SSE；internal rpc 不暴露公网。Python 同时可提供 REST 供调试。

先想 10 秒：为何 tenant_id 在 proto 里必填，而不只信 HTTP Header？

gRPC metadata 可传 x-tenant-id，但 proto 字段强制校验更利于 codegen 与跨语言一致；BFF 从 JWT 解析后写入 request，Python 双检 metadata + body。

流式 RPC 与 LLM 生成

StreamChat 使用 gRPC server streaming：Python 每生成一个 token 块就 yield ChatChunk，BFF 聚合后写 SSE data: 行。这比「HTTP 长 JSON 数组」更省缓冲，且便于 performance rpc 背压控制。

StreamChat：Python → BFF → SSE
sequenceDiagram
  participant C as Client
  participant GW as Gateway
  participant B as BFF
  participant P as Python gRPC
  C->>GW: GET /v1/stream/chat (SSE)
  GW->>B: 透传
  B->>P: StreamChat RPC
  loop token chunks
    P-->>B: ChatChunk delta
    B-->>C: SSE data
  end
  P-->>B: finished=true
  B-->>C: SSE done

用户侧协议仍是 SSE（basics-02）；gRPC 流仅在服务间。Gateway 无需理解 protobuf。

BFF 侧 gRPC 客户端（Java 示意）

java 
ManagedChannel channel = ManagedChannelBuilder
    .forAddress("corpassist-ai-py", 50051)
    .usePlaintext()  // 内网 mTLS 生产环境替换
    .build();
RagServiceGrpc.RagServiceBlockingStub stub =
    RagServiceGrpc.newBlockingStub(channel)
        .withDeadlineAfter(120, TimeUnit.SECONDS);

Iterator<ChatChunk> stream = stub.streamChat(ChatRequest.newBuilder()
    .setTenantId(tenantId)
    .setSessionId(sessionId)
    .setUserMessage(message)
    .addAllContext(hits)
    .build());

while (stream.hasNext()) {
  ChatChunk chunk = stream.next();
  sseEmitter.send(chunk.getDelta());
  if (chunk.getFinished()) break;
}

Python 侧 server streaming（示意）

python 
class RagServicer(rag_pb2_grpc.RagServiceServicer):
    async def StreamChat(self, request, context):
        async for delta in llm_stream(request.user_message, request.context):
            yield rag_pb2.ChatChunk(delta=delta, finished=False, token_count=1)
        yield rag_pb2.ChatChunk(delta="", finished=True, token_count=0)

反例 vs 推荐

不推荐
  • 公网暴露 gRPC 50051，无 TLS
  • proto 随意改字段编号，无兼容性检查
  • 用 gRPC 替代全部 MQ 索引任务（见 practice-04）
推荐
  • 内网 mTLS；Gateway 对外只 REST/SSE
  • buf breaking change 或 protobuf 兼容规则 CI
  • 长任务走 MQ；实时 RPC 走 gRPC

何时用 gRPC：when grpc 决策

rest vs grpc 不是信仰之争。CorpAssist 默认 BFF→Python 用 OpenAPI REST；当出现下列信号时再引入 gRPC：

信号倾向 gRPC保持 REST
内网 P99 延迟 > 预算，HTTP 连接数高 HTTP/2 多路复用、二进制 payload —
需要 server/bidi streaming 传 token StreamChat HTTP chunked 也可但 BFF 代码更繁
第三方 / 浏览器直接调用 — REST + OpenAPI 文档
团队无 proto 治理经验 — 先 REST，度量后再迁移
批量索引、削峰 — MQ 异步 更合适

路径：/v1/chat、/v1/rag/** 经 Gateway。

优势：curl 调试、Swagger、WAF 友好。

路径：BFF → corpassist-ai-py:50051（ClusterIP）。

优势：强类型、流式、更低序列化开销。

常见误区
  • gRPC 一定比 REST 快 — 小 payload 下差距可能不明显；先度量 P99。
  • Gateway 转 gRPC — Spring Cloud Gateway 对 gRPC 需 grpc-proxy 插件；CorpAssist 选 BFF 直连 Python。
  • 流式 = 只要 gRPC — 对用户 SSE 足够；gRPC 流是内网优化项。

关键术语速览

grpc integration · proto define · grpc stream · when grpc · internal rpc · protobuf · performance rpc · rest vs grpc

已有 OpenAPI，还要 gRPC 吗？
可选。REST 保留作调试与降级；gRPC 作内网热路径。BFF 可 feature flag 切换 stub。
与 MQ 索引 如何配合？
索引任务仍走 MQ；gRPC 适合同步 retrieve/chat。勿用 streaming RPC 传整库 PDF。

  • 选型：度量驱动；批量异步用 MQ，低延迟流式用 gRPC。

动手练习

在 demos/practice-05-grpc-lab/grpc-vs-rest.md 填写选型表，并用 grpcurl 验证 Python 服务（约 20 分钟）。

操作步骤

  1. 将正文 RagService.proto 复制到 Demo 目录，运行 buf generate 或项目脚本生成 Java/Python stub。
  2. 启动 Python gRPC 服务（端口 50051），用 grpcurl 调用 unary Retrieve：
    bash
    grpcurl -plaintext -d '{"tenant_id":"demo","query":"退款政策","top_k":3}' \
  localhost:50051 corpassist.v1.RagService/Retrieve
  3. 对比同一 query 的 REST 路径（若已启用）：
    bash
    curl -s "http://localhost:8000/v1/rag/retrieve?tenant_id=demo&q=退款政策&top_k=3" | jq .
    记录两者延迟与 payload 大小（粗略即可）。
  4. 在 BFF 配置中增加 gRPC channel 地址（yaml 示意）：
    yaml
    corpassist:
  ai:
    grpc:
      host: corpassist-ai-py
      port: 50051
      deadline-seconds: 120
    rest:
      base-url: http://corpassist-ai-py:8000
      enabled: true  # 降级开关
  5. 在 grpc-vs-rest.md 写一条采用 gRPC与一条坚持 REST的 CorpAssist 场景理由。

判断练习

  1. 方案：「把 Gateway 改成 gRPC 入口，App 全部换 stub」。请指出违反本章原则的 2 点。

    参考答案

    ① 移动端/浏览器 gRPC 支持差，破坏 OpenAPI 生态；② Gateway 层 gRPC 治理复杂，与 CorpAssist「对外 REST」架构不符。应 BFF 内网 gRPC。

  2. 找错：索引 500 份 PDF 应走？A) index_jobs MQ B) StreamChat gRPC C) 同步 REST 长连接

    参考答案

    A。批量离线任务用 MQ 削峰；B 适合 token 流；C 会占满连接。
