为什么要学本章

智能客服首字延迟（TTFT）直接影响 S2 体验。LLM 回复必须经 SSE stream 逐 token 推送；若 Gateway 缓冲整包、BFF 聚合 JSON、或断线无 reconnect sse，用户会看到「卡住再一次性弹出」——这是 CorpAssist 生产高频坑，也是 NS2/NS3 联调时最容易被忽略的链路问题。

学完你能
  • 在 Spring BFF 用 spring sse 转发 Python python generator 输出，并为每个 chunk 分配 event id
  • 配置 Gateway stream proxy 不缓冲 event stream，实现逐跳 chunk forward
  • 设计 last event id、心跳与 Redis 短时缓冲，支持移动端断线续传
本章先记住 3 件事
  • SSE 是单向 HTTP 长连接：格式为 data: 行 + 双换行；比 WebSocket 更简单，适合 LLM 文本流（前提：客户端只收不发）。
  • 每一跳都要「流式不缓冲」：Gateway、BFF、Nginx/Ingress 都要关闭 response buffering 并设置合理读超时（见 Spring Cloud Gateway 与 LLM）。
  • 断线重连要有 id：服务端为每个 chunk 分配单调递增 id，客户端带 Last-Event-ID 重连；会话上下文仍靠 会话与状态。

Spring SSE

CorpAssist S2 · 客服流式回复（30 秒） 用户问「订单 SO-1024 物流到哪了？」App 经 Gateway 到 BFF：GET /v1/stream/chat?sessionId=c-8f2a。BFF 从 Redis 加载会话摘要，调用 Python 流式接口，将 token 逐段写入 WebFlux Flux<ServerSentEvent>；客户端 EventSource 实时渲染，首 token 须在 500ms 内可见（SLO 示意）。

BFF 层职责：鉴权后的会话上下文、调用 Python、统一 traceId、对前端暴露 text/event-stream。Gateway 只做 stream proxy，不在此层解析 SSE 帧内容。

Gateway

response-timeout: 0、透传 Last-Event-ID、关闭缓冲。

Spring BFF

WebFlux SSE 出口；httpx/WebClient 流式读 Python。

FastAPI

async generator 产出 SSE 帧；内网 only。

SSE 帧格式（HTTP 层）

字段示例作用
id:id: 42断线续传锚点（last event id）
event:event: token前端按类型分发（token / error / done）
data:data: {"text":"您好"}payload；多行 data 用 \n 拼接
注释行: ping心跳，防止代理空闲断开

WebFlux SSE 控制器

java 
@GetMapping(value = "/v1/stream/chat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> streamChat(
    @RequestHeader("X-Tenant-Id") String tenantId,
    @RequestParam String sessionId,
    @RequestHeader(value = "Last-Event-ID", required = false) String lastEventId) {

  long resumeFrom = lastEventId == null ? 0 : Long.parseLong(lastEventId);
  return pythonStreamClient.streamTokens(tenantId, sessionId, resumeFrom)
      .map(chunk -> ServerSentEvent.<String>builder()
          .id(String.valueOf(chunk.eventId()))
          .event(chunk.type())
          .data(chunk.payloadJson())
          .build())
      .concatWith(Flux.just(ServerSentEvent.builder().event("done").data("{}").build()));
}

SSE 端到端序列
sequenceDiagram
  participant C as Client
  participant GW as Gateway
  participant BFF as Spring BFF
  participant Py as FastAPI
  C->>GW: GET /v1/stream/chat + JWT
  GW->>BFF: 透传 SSE + X-Tenant-Id
  BFF->>Py: httpx stream (内网)
  loop 每个 token
    Py-->>BFF: id/data chunk
    BFF-->>GW: SSE frame
    GW-->>C: SSE frame (无缓冲)
  end
  BFF-->>C: event: done

chunk forward 要求每一跳禁止聚合为单 JSON 响应；任一中间层缓冲都会导致「假流式」。

反例 vs 推荐

不推荐
  • BFF 用 RestTemplate 同步等待完整 body
  • Gateway 对 /v1/stream/** 设 30s 读超时
  • SseEmitter 线程阻塞等待 Python 整段返回
推荐
  • WebFlux Flux 或 reactive WebClient 边读边写
  • Gateway response-timeout: 0 + X-Accel-Buffering: no
  • 每个 chunk 带递增 id，便于续传与排障
先想 10 秒：为什么 BFF 不直接把 Python URL 暴露给浏览器？

鉴权、租户隔离、Trace、内容安全与模型 Key 保护都应在 BFF/Gateway；Python 仅内网可达。直连会绕过 gateway auth 与 租户配额。

Python 生成器

FastAPI 用 StreamingResponse 包装 async generator 产出 SSE 帧；与 OpenAI 兼容 API 的 stream=True 模式对接。CorpAssist 能力面只信任 Gateway 注入的 X-Tenant-Id，不对外暴露。

FastAPI 流式生成器

python 
async def token_stream(session_id: str, after: int = 0):
    event_id = after
    async for chunk in llm.astream(session_id):
        event_id += 1
        payload = json.dumps({"text": chunk}, ensure_ascii=False)
        yield f"id: {event_id}\nevent: token\ndata: {payload}\n\n"
        if event_id % 20 == 0:
            yield ": ping\n\n"  # 心跳，防代理 60s 空闲断开

@app.get("/internal/v1/generate/stream")
async def stream(
    session_id: str,
    after: int = 0,
    x_tenant_id: str = Header(..., alias="X-Tenant-Id"),
):
    return StreamingResponse(
        token_stream(session_id, after),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )

推荐 · async generator
python 
async def gen():
    async with httpx.AsyncClient() as client:
        async with client.stream("POST", url, json=body) as r:
            async for line in r.aiter_lines():
                if line.startswith("data:"):
                    yield parse_sse(line)

适用：LLM 单向 token 推送、HTTP 生态友好、自动重连（EventSource）。

CorpAssist S2：客服 App 默认 SSE；Gateway 配置成熟。

适用：双向高频交互、二进制帧、游戏/协作编辑。

本课：可选 gRPC stream（见 gRPC 可选集成）；S2 主路径仍 SSE。

常见误区
  • Python 是 generator 就够了 — BFF/Gateway/Nginx 任一环节缓冲仍会「批量弹出」。
  • 无 heartbeat — 中间代理 60s 空闲断开，用户以为模型卡住。
  • event id 随机 UUID — 无法单调续传；应用递增整数或雪花序。

断线重连

移动网络切换、笔记本休眠、K8s 滚动发布会导致 SSE 断开。客户端应使用 EventSource（自带重连）或 fetch 流式库，携带 Last-Event-ID 请求头；服务端从 Redis 补发遗漏帧或告知「请重新提问」。

策略做法注意
服务端缓冲Redis List sse:buf:{sessionId} 存最近 N 帧TTL 5–10min；上限防 OOM
重连参数Last-Event-ID: 42只补发 id > 42 的事件
会话一致性同一 sessionId 绑定生成状态见 会话与状态
生成已结束返回 event: replay_done避免重复计 Token（配合 幂等）
断线重连时序
sequenceDiagram
  participant C as Client
  participant BFF as BFF
  participant R as Redis
  C->>BFF: SSE id=1..40
  Note over C: 网络断开
  C->>BFF: GET stream Last-Event-ID: 40
  BFF->>R: LRANGE buf 41..
  alt 缓冲命中
    BFF-->>C: 补发 41..N
  else 生成已结束
    BFF-->>C: event: replay_done
  end

reconnect sse 不是重新 POST chat；续传用 GET + Last-Event-ID，写操作仍须 Idempotency-Key。

Flux<ServerSentEvent<String>> replayThenLive(long after, String sessionId) {
  Flux<ServerSentEvent<String>> replay = sseBufferRepo.replayAfter(sessionId, after);

场景推荐避免

常见问题（FAQ）

SSE 断线后为什么要带 Last-Event-ID 而不是重新 POST？

重新 POST 可能触发第二次 LLM 推理，双倍 Token 费用且回复可能不一致。续传只补发未送达帧；若生成已结束则返回 replay_done 或缓存全文。

Gateway 限流应放在建立 SSE 之前还是之后？

尽量在建立连接前限流（见 gateway rate limit）；已建立的长连接中途超限应发 event: error 并关闭。

本章结论

放在正文与复习之间，便于串起全章再做题。每条对应上文一个小节标题。

  • Spring SSE：BFF 用 WebFlux 暴露 text/event-stream，WebClient 流式读 Python；禁止 RestTemplate 整包等待。
  • Python 生成器：async generator + StreamingResponse；heartbeat 与递增 event id 是生产必备。
  • 断线重连：Last-Event-ID + Redis 短时缓冲；Gateway 关闭读超时与响应缓冲；续传≠重新 POST。

判断练习

  1. 排障：用户反馈「流式变批量」。已确认 Python 是 async generator，下一步最应检查什么？① BFF 是否 json 聚合 ② Gateway response-timeout ③ Nginx X-Accel-Buffering ④ 前端是否 EventSource

    参考答案

    优先 ① 和 ③。BFF 聚合或反向代理缓冲是最常见根因。② 过长会导致断连而非「批量弹出」；④ 错误通常表现为完全不更新。

  2. 找错：方案写「SSE 断线后客户端立即重新 POST /v1/chat，不带 Last-Event-ID」。违反哪条原则？

    参考答案

    违反断线续传与 NS3 成本控制：可能触发第二次 LLM 调用、双倍 Token。应 GET stream + Last-Event-ID 补发，或返回 replay_done。
