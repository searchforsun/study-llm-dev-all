为什么要学本章

CorpAssist S2 智能客服面试常问：「日活 50 万、峰值 QPS 2000，如何设计 LLM 网关？」若只会背 Spring Cloud Gateway 配置而说不出 qps estimate、SSE 并发连接与 cache layer 分层，很难通过系统设计轮。本章把 Gateway 路由鉴权限流、压测基线 等已学内容压缩成 30 秒～3 分钟口述框架。

  • 现场估算 Gateway 实例数、SSE 连接上限与 Redis 限流 key 规模（scale number）

本章先记住 3 件事
  • 先画边界再谈技术：公网只进 Gateway；BFF 编排；Python 能力面；存量 ERP 经 Feign（见 存量集成）。
  • LLM 网关 ≠ 普通 API 网关：长连接 SSE、Token 计费、模型 RPM 三重约束（SSE、配额）。
  • 数字要可推导：峰值 QPS、平均响应时间、连接占用时长 → 实例与连接池（llm gateway design 量化部分）。

架构

S2 系统设计 · 30 秒开场（CorpAssist） 「我们做的是 ToC 智能客服：用户经 App 访问 Spring Cloud Gateway，JWT 鉴权 + 租户限流后进入 BFF 做多轮对话与查订单工具；流式回复走 SSE 透传到 Python 生成器；RAG 走独立路由。Gateway 挡入口洪水，BFF 管 Token 预算与熔断，Python 专注推理。」

S2 架构题考察的是分层职责与失败模式，不是背诵某个 Filter 类名。推荐用「入口治理 → 编排确定性 → AI 能力 → 横切观测」四段式展开 architecture qa。

L1 · 入口

Gateway：路由、JWT、QPS/连接限流、Trace 注入、SSE 超时 0（basics-01）。

L2 · 编排

BFF：会话 Redis、工具 Feign、Resilience4j 熔断降级（practice-01、basics-03）。

L3 · 能力

FastAPI：RAG、流式生成；MQ 异步索引（practice-04）；多租户索引前缀（practice-03）。

L4 · 横切

缓存、配置 Vault、K8s HPA、CI 门禁（advanced-02、advanced-03、advanced-04）。

cache layer 设计要点

面试常追问「缓存放哪一层」。CorpAssist 典型分工：

  • Gateway 前（CDN）：静态 FAQ、营销页；不缓存个性化对话。
  • Redis（BFF）：多轮会话、租户配额计数、限流令牌桶（与 Gateway 共享或分层 key）。
  • 语义缓存（Python，可选）：相同问题 embedding 相似度 > 阈值则直接返回；须带租户隔离与 TTL，避免跨租户泄漏。
  • 不缓存：写操作工具链、含 PII 的订单详情、实时物流状态。

架构反例 vs 推荐

面试扣分点
  • 「所有流量直连 Python，Gateway 只是 nginx 反代」
  • 「高并发 = 无限扩容模型副本」忽略入口限流与 Token 成本
  • 「SSE 和普通 REST 同一超时、同一限流 key」
  • 「缓存所有 LLM 输出」导致个性化与合规问题
加分表述
  • 分层限流：Gateway QPS + BFF Token 预算 + 模型 RPM
  • SSE response-timeout: 0 + 建立连接前限流
  • 确定性工具走 BFF→Feign，生成走 BFF→Python 流式
  • 熔断 fallback 默认回复，保护 ERP 与模型（practice-01）

## S2 高并发 LLM 网关 — 口述框架

### 30 秒版
1. 入口：Gateway 鉴权 + 租户 QPS + SSE 长连接治理
2. 编排：BFF 会话/工具/熔断；Redis 会话与配额
3. 能力：Python 流式 + RAG；MQ 削峰索引
4. 弹性：HPA + 降级文案 + 小模型路由降本

### 3 分钟版（每点 40s）
- 需求澄清：DAU、峰值 QPS、SSE 占比、SLA
- 数据流：查订单 vs 纯聊天 vs RAG 三条路径
- 瓶颈：连接数 > QPS；模型 TPM；ERP 工具 QPS
- 扩展：Gateway 水平扩展无状态；Redis Cluster；BFF 舱壁
- 观测：Trace 分段延迟；429/503 告警；压测 baseline（practice-06）
先想 10 秒：面试官问「为什么不用单一 Python 网关？」你怎么答？

治理面（鉴权、限流、审计、与存量 Java Feign 集成）在 Spring 生态更成熟；Python 专注 AI 能力面。双栈分工见 basics-01 与 practice-06 全链路压测结论——必须对外入口压测，不能直打 Python。

数

qps estimate 与 scale number 是 S2 系统设计第二轮常考点。目标不是精确到个位数，而是数量级正确、假设可解释、与架构决策一致。

CorpAssist 估算假设（教学基准）

指标假设值用途
日活 DAU50 万容量规划起点
峰值系数×5（晚 8–10 点）平均 QPS → 峰值 QPS
人均会话请求8 次/天（含 SSE 流）总请求量
SSE 平均时长45 s并发连接估算
Gateway 单实例~800 同步 QPS 或 ~2k SSE 连接水平扩展除数

推导示例（可在白板上写）

text 
日均请求 ≈ 50万 × 8 = 400万/天
平均 QPS ≈ 400万 / 86400 ≈ 46
峰值 QPS ≈ 46 × 5 ≈ 230（同步 chat 类）

若 40% 为 SSE 流式：
  峰值 SSE 新建连接/秒 ≈ 230 × 0.4 ≈ 92/s
  稳态并发连接 ≈ 92 × 45s ≈ 4140 连接

Gateway 实例（按 SSE 连接上限 2000/实例）：
  ceil(4140 / 2000) ≈ 3 实例 + 1 冗余 = 4

Redis 限流 key（按租户）：
  活跃租户 200 × 2 key（chat + stream）≈ 400 key — 可单机 Redis

量化决策表

资源估算输入公式/经验面试一句话
Gateway Pod峰值 SSE 并发连接数 / 单 Pod 上限 + 冗余「先算长连接，再算 QPS」

同步 JSON 受 Gateway 线程与下游 BFF 处理时间约束。Little 定律：并发 ≈ QPS × 延迟。P95 800ms、目标 500 QPS → 并发约 400，需结合压测验证而非纯心算。

SSE 占用的是连接槽位而非短请求线程。面试应强调：stream 路由单独限流、response-timeout: 0、Gateway HttpClient 连接池 max 与 BFF 对齐（basics-02）。

估算错了会被挂吗？

通常不会。面试官看你是否先澄清假设、能否把数字与 Gateway/BFF/Redis/模型分层对应。数量级差 2× 可接受，但「只扩模型不限流」会扣分。

如何把 practice-06 压测写进答案？

「我们在预发用 k6 从 Gateway 入口压测，30 VU 下 P95 1.2s、failed <1%，以此反推单实例容量并留 30% 冗余。」这比纯理论更有说服力。

429 大量出现说明什么？

入口限流生效或 replenishRate 过低。应先查 Gateway 租户配额（practice-02），而非盲目升模型规格。

常见误区
  • 只背组件清单 — 缺少数据流与失败降级路径。
  • 忽略 SSE — 按 REST QPS 扩容会在长连接场景击穿 Gateway。
  • 不算 Token 成本 — NS3 成本与 high concurrency 强相关，需提 BFF 预算层。

本章结论

面试答题 = 本课程知识压缩。每条对应一个小节。

  • 架构：四段式（入口 Gateway → BFF 编排 → Python/MQ → 横切）；cache layer 分层；SSE 与 REST 分治。
  • 数：qps estimate 从 DAU 推导；SSE 并发 = 新建率 × 平均时长；scale number 需冗余与压测校验。
  • 串联课程：basics 01–04 打底，practice 01–06 弹性与压测，advanced 01–04 企业集成；下一章 生产清单 补 SLA 与 runbook。

判断练习

  1. 面试答案：「高并发只需 HPA 自动扩 Gateway 到 50 副本，不限流。」请指出两处问题。

    参考答案

    ① 无限扩容不控模型 TPM/成本，NS3 账单失控；② 不限流时 ERP 工具与下游会被打穿，且 SSE 连接数可能先于 CPU 成为瓶颈。应先入口限流 + 分层配额 + 压测定容量。

  2. 估算：峰值 2000 QPS 全为 60s SSE，单 Gateway 3000 连接上限，最少几实例？

    参考答案

    稳态并发 ≈ 2000 × 60 = 120,000 连接；120000/3000 = 40 实例；加冗余约 41–44。若只按 QPS 算会严重低估。
