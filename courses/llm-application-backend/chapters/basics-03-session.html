已完成会话与状态

约 35 分钟 · 阶段：LLM 网关与流式 · 能力：用 Redis 管理多轮对话状态、粘性与 TTL

为什么要学本章

S2 智能客服的多轮上下文不能全靠 LLM「猜」。dialog state（意图、槽位、订单号）须存 redis session，并与 conversation id 绑定——纯 LLM 管状态是 NS2 明确反对的生产坑；SSE 断线重连（SSE 流式端到端）也依赖稳定的 sessionId。

学完你能
  • 设计 session key 命名：tenant:user:conversation，避免跨租户泄漏
  • 在水平扩展下选择 session stickiness 或无状态 + 集中存储
  • 配置 ttl policy：idle 滑动续期与 session expire 绝对上限
本章先记住 3 件事
  • 会话状态 ≠ 模型上下文：Redis 存结构化 DST；Prompt 只注入必要摘要，raw history 走 thread store 分层。
  • 优先无状态服务 + Redis：比强粘性更易 HPA；粘性仅在有不可外置的本地 SSE 状态时考虑。
  • TTL 要分场景：客服 idle 30min 滑动；合规归档用 max TTL；结束会话须归档而非仅 DEL。

Redis

CorpAssist S2 · 多轮查订单（30 秒） 用户：「查订单」→ 助手：「请提供订单号」→ 用户：「SO-1024」。BFF 在 Redis 更新 intent=query_order、slots.orderId=SO-1024，再触发 NS2 工具 lookup_order。下一轮用户改问「改地址」时，intent 切换，槽位重置规则由 DST 状态机定义——不靠模型记忆。

CorpAssist 用 Redis Hash 存会话：结构化 dialog state，而非整段聊天记录。Key 示例：ca:sess:{tenantId}:{conversationId}；用户维度索引可选 ca:user:sess:{tenantId}:{userId} 存活跃 conversationId 列表。

会话 Hash 字段

字段类型说明
conversationIdstring对外 conversation id，SSE/REST 共用
intentstring当前意图：query_order / change_address
slotsJSON槽位：orderId、addressId 等
turnCountint轮次，用于异常检测与限流
lastActiveAtISO8601滑动 TTL 依据
summarystring注入 Prompt 的短摘要（非全量 history）

BFF 读写 Repository（Java 示意）

java 
public DialogState load(String tenantId, String conversationId) {
  String key = "ca:sess:" + tenantId + ":" + conversationId;
  Map<Object, Object> hash = redis.opsForHash().entries(key);
  if (hash.isEmpty()) return DialogState.newSession(tenantId, conversationId);
  return DialogState.fromRedis(hash);
}

public void save(DialogState state) {
  String key = "ca:sess:" + state.tenantId() + ":" + state.conversationId();
  redis.opsForHash().putAll(key, state.toMap());
  redis.expire(key, idleTimeout);  // 滑动 TTL，见「过期」节
}

flowchart LR

任意 BFF 实例均可读写同一 session key；Python 不直接持有会话主权，由 BFF 合并后注入。

反例 vs 推荐

不推荐
  • Redis List 存全量 raw messages，每轮全量注入 Prompt
  • session key 不含 tenantId，仅靠 conversationId
  • 会话只存 JVM 内存，扩缩容丢失
推荐
  • Hash 存 DST + summary；长 history 进 thread store / 向量检索
  • key 必含 tenantId，Gateway 注入头与 key 交叉校验
  • 无状态 BFF + Redis 集中会话
先想 10 秒：为什么 conversationId 不能由客户端随意指定新 UUID 而不校验归属？

否则用户 A 可猜测用户 B 的 conversationId 读取会话。创建会话须绑定 JWT 的 userId/tenantId；读取时校验 key 前缀与 Header 一致。

粘性

SSE 长连接有时被误认为需要 session stickiness（同一连接落在同一 BFF Pod）。CorpAssist 推荐：会话与 SSE 缓冲外置 Redis，Gateway 轮询即可，滚动发布时配合 Last-Event-ID 续传。

方案优点缺点CorpAssist S2
Gateway 一致性 Hash / Cookie 粘性 Pod 内本地 SSE 状态简单 扩缩容连接迁移；热点 Pod 不推荐作主方案
Redis 集中会话 + SSE 缓冲（推荐） 实例无状态，HPA 友好 Redis 延迟与容量规划 默认
Sticky + Redis 双写 过渡兼容遗留本地缓存 复杂度最高 仅迁移期
K8s 滚动发布 vs 无状态 BFF
sequenceDiagram
  participant C as Client
  participant GW as Gateway
  participant P1 as BFF Pod-1
  participant P2 as BFF Pod-2
  participant R as Redis
  C->>GW: SSE on Pod-1
  Note over P1: Pod-1 终止
  C->>GW: 重连 Last-Event-ID
  GW->>P2: 新 Pod
  P2->>R: 加载 session + replay buffer
  P2-->>C: 续传 SSE

强粘性在 Pod 终止时必断；外置 Redis + 续传才是 S2 可扩缩路径。

yaml

适用：多轮 DST、SSE 缓冲、HPA、跨 AZ。

代价：Redis 高可用、key 设计、TTL 策略。

适用：单实例 PoC、无 Redis 的本地开发。

代价：无法水平扩展；生产禁用。

先想 10 秒：K8s 滚动发布时纯粘性会话会有什么问题？

旧 Pod 终止导致 SSE 断开且本地状态丢失。必须依赖 Last-Event-ID + Redis 会话恢复；否则用户需重新开始对话，体验与 NS2 工具链中断。

过期

session expire 与 ttl policy 要在「用户体验」「内存成本」「合规留存」之间折中。S2 客服典型：idle 30min 滑动；max 24h 绝对上限；转人工或结案后归档冷存储再 DEL。

TTL 策略对照

策略行为CorpAssist 示例
滑动 idle TTL每次用户消息 EXPIRE key 180030min 无发言则会话过期
绝对 max TTL创建时记录 expiresAt，到期强制结束24h 后须重新建会话（合规）
主动关闭转人工 / 问题解决 → 归档 + DEL写入 OSS/DB 审计，Redis 删除
SSE 缓冲 TTL短于会话 TTL5–10min（见 SSE 章）

application 配置

yaml 
corpassist:
  session:
    redis-prefix: "ca:sess:"
    idle-timeout: 30m
    max-ttl: 24h
    refresh-on-each-request: true
    archive-on-close: true
    archive-bucket: "s3://corpassist-session-archive"

滑动续期 vs 仅创建时 TTL：代码对比

不推荐 · 仅 SET 一次 TTL
java 
redis.opsForHash().putAll(key, state);
redis.expire(key, Duration.ofMinutes(30)); // 只在创建时
推荐 · 每次交互续期
java 
redis.opsForHash().putAll(key, state);
redis.expire(key, idleTimeout);
if (state.exceedsMaxTtl()) {
  archiveAndClose(state);
}

常见问题（FAQ）

会话过期后用户继续用旧 conversationId 发消息怎么办？

返回 410 Gone 或友好提示「会话已过期」，引导新建 conversation；可选携带脱敏 summary 到新会话。写操作须新 Idempotency-Key。

Redis 会话与 Spring Session 是什么关系？

Spring Session 可复用 Redis 存 HTTP Session；CorpAssist 更常用自定义 DialogState Repository，字段贴合 NLU/工具槽位，不必强行塞进 Servlet Session。

常见误区
  • 无限 TTL — Redis 内存膨胀；合规也无法解释数据留存。
  • 过期只 DEL 不归档 — 客诉与审计无法回溯。
  • 把 LLM 输出当唯一状态 — 工具参数须结构化存储，模型幻觉不能作为订单号来源。

本章结论

放在正文与复习之间，便于串起全章再做题。每条对应上文一个小节标题。

  • Redis：Hash 存结构化 dialog state；key 含 tenantId + conversationId；summary 注入 Prompt，非全量 history。
  • 粘性：优先无状态 BFF + Redis；避免强依赖 Gateway 粘性阻碍 HPA 与滚动发布。
  • 过期：idle 滑动 TTL + max 绝对上限；结案归档后 DEL；与 SSE 缓冲 TTL 分层。

判断练习

  1. 找错：「把完整多轮聊天记录都塞进 Redis List，每次全量注入 Prompt；session key 仅用 conversationId」。指出两条违反原则的做法。

    参考答案

    ① 违反「会话状态 ≠ 模型上下文」：应 DST + summary/thread store，而非无限 raw history（成本与干扰）。② key 缺 tenantId 可能导致跨租户访问若 conversationId 可猜测或泄漏。

  2. 大促：① 会话 idle TTL 改为 7 天 ② 关闭归档直接 DEL ③ 启用 Gateway IP 粘性替代 Redis ④ 保持 Redis 集中 + 30min idle。哪项最危险？

    参考答案

    最危险：②，审计与客诉无法回溯。③ 阻碍 HPA 且滚动发布断连。① 浪费 Redis 内存且可能违反留存策略。
