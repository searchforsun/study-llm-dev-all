约 35 分钟 · 阶段：LLM 网关与流式 · 能力：用幂等键、MQ 去重与安全重试处理重复请求

为什么要学本章

移动端弱网、Gateway 429 重试、MQ at least once 投递都会产生 duplicate request。S2 客服下重复「改地址」或重复 LLM 调用会直接引发客诉与 NS3 Token 成本失控——gateway rate limit 触发的 429 若盲目重试 POST，更是典型事故源。

学完你能
  • 为写操作设计 idempotency key 与 token store 状态机
  • 在 MQ 消费端实现 mq dedup 与 retry safe 策略
  • 区分工具改单的强幂等与 LLM 生成的弱幂等/缓存，理解 exactly once 的工程边界
本章先记住 3 件事
  • 写操作必须幂等：工具改单、发券、索引任务提交要有 Idempotency-Key（NS2 确定性写路径）。
  • 读/生成可接受 at-least-once + 去重：相同 key 返回缓存或相同 stream，避免双倍计费。
  • exactly-once 端到端很难：组合幂等键 + 去重表 + 对账；SSE 续传用 Last-Event-ID 而非新 key（见 SSE 流式端到端）。

令牌

CorpAssist S2 · 改地址重复提交（30 秒） 用户在弱网下双击「确认改地址」。两次 POST 携带相同 Idempotency-Key: idem-7c3f。BFF 第一次进入 Processing 调 ERP；第二次命中 token store 返回相同 200 与 orderSnapshot——ERP 只改一次。若第二次无 key，可能产生两次改址与双倍审计日志。

客户端或 BFF 为每次写操作生成 UUID，放入 Header Idempotency-Key。服务端在 Redis/DB token store 记录「处理中 / 已完成 + 响应快照」。TTL 建议 24–72h，覆盖移动端重试窗口。

幂等表状态机

Idempotency-Key 生命周期
stateDiagram-v2
  [*] --> Received
  Received --> Processing: 首次请求 SETNX lock
  Received --> Replay: 重复 key 且 Completed
  Processing --> Completed: 业务成功 存快照
  Processing --> Failed: 可重试失败 释放 lock
  Replay --> [*]: 返回原 HTTP 响应
  Completed --> [*]

Processing 中重复请求可返回 409 Conflict 或 202 Accepted，避免并发双写。

BFF 幂等拦截器（Java）

java 
public ResponseEntity<?> handle(String tenantId, String idempotencyKey, Supplier<Result> action) {
  if (idempotencyKey == null || idempotencyKey.isBlank()) {
    return ResponseEntity.badRequest().body("Idempotency-Key required");
  }
  String storeKey = "idem:" + tenantId + ":" + idempotencyKey;
  Optional<CachedResponse> cached = store.findCompleted(storeKey);
  if (cached.isPresent()) return cached.get().toResponse();  // Replay

  if (!store.tryLock(storeKey, Duration.ofMinutes(5))) {
    return ResponseEntity.status(HttpStatus.CONFLICT).build();
  }
  try {
    Result r = action.get();
    store.saveCompleted(storeKey, r, Duration.ofHours(48));
    return ResponseEntity.ok(r);
  } catch (Exception e) {
    store.releaseLock(storeKey);
    throw e;
  }
}

反例 vs 推荐

不推荐
  • 用「订单号 + 时间戳秒」作 key，同一秒内双击仍重复
  • 仅 DB 唯一约束，无响应快照，重试得到 409 但客户端不知原结果
  • 幂等记录永不过期，Redis 内存泄漏
推荐
  • 客户端 UUID v4 或 BFF 代生成并回传
  • Completed 存 status + body 快照，Replay 原样返回
  • TTL 48h + 归档审计日志
先想 10 秒：lookup_order 只读查询需要 Idempotency-Key 吗？

只读 GET 天然幂等，通常不需要。但若 POST 包装「提交查询任务」或触发计费 LLM，则需 key 或改用 GET + 缓存。

MQ

异步索引、通知、会话归档任务经 RabbitMQ/Kafka 投递。中间件多为 at least once——消费端必须 mq dedup，实现 retry safe 消费。CorpAssist 知识库索引任务 documentId + version 作业务去重键。

MQ 消费去重流
flowchart TD
  Q[Queue]
  C[Consumer]
  R[(Redis SETNX)]
  DLQ[Dead Letter Queue]
  Q --> C
  C --> R
  R -->|新消息| Work[执行业务]
  R -->|已处理| Skip[ACK 跳过]
  Work -->|失败超次| DLQ

去重键 TTL 应覆盖「重投窗口 + 对账周期」；失败进 DLQ 而非无限 requeue。

Redis 消费去重（Python 示意）

python 
async def handle_index_job(msg: IndexJob):
    dedup_key = f"processed:{msg.tenant_id}:{msg.document_id}:{msg.version}"
    if not await redis.set(dedup_key, "1", nx=True, ex=86400 * 7):
        logger.info("duplicate skip %s", dedup_key)
        return  # ACK without work
    try:
        await index_pipeline.run(msg)
    except RetryableError:
        await redis.delete(dedup_key)
        raise  # requeue with backoff
    except PermanentError:
        await dlq.publish(msg)

Spring AMQP 消费（Java 示意）

java 
@RabbitListener(queues = "corpassist.index")
public void onIndex(IndexJobMessage msg) {
  String dedup = "mq:dedup:" + msg.tenantId() + ":" + msg.messageId();
  if (!dedupStore.markIfAbsent(dedup, Duration.ofDays(7))) return;
  try {
    indexService.process(msg);
  } catch (TransientException e) {
    dedupStore.remove(dedup);
    throw e;
  }
}
不推荐
  • 消费失败无限 requeue，无 DLQ
  • 同一 documentId 索引 N 次，向量库重复 chunk
  • 先执行业务再 SETNX，并发下仍双写
推荐
  • SETNX / 唯一约束在业务之前
  • 业务键 upsert + 版本号
  • DLQ + 人工/脚本 replay（见 消息队列与异步任务）

重复请求

除 MQ 外，HTTP 层重复同样常见：429 退避重试、移动端双击、负载均衡重试。须按接口类型设计 idempotent api 策略。

接口类型幂等要求实现要点
工具改单 / 改地址强幂等Idempotency-Key + ERP 业务键 + 响应快照
POST /v1/chat（触发 LLM）弱幂等 / 缓存相同 key 返回同一 conversationId + cached answer
SSE 断线重连续传非新请求Last-Event-ID GET，非新 Idempotency-Key
Gateway 429 重试仅安全方法POST 须带 key；无 key 禁止自动重试

相同 Idempotency-Key 的第二次 POST 应 Replay 或返回 409 + 查询接口，而非再次调模型。

429 重试决策（标签页）

可退避重试；SSE 用 Last-Event-ID，不重复计费（补帧或 replay_done）。

必须 Idempotency-Key；无 key 时客户端应提示用户而非 SDK 自动重试。429 时读 Retry-After。

常见误区
  • 「LLM 输出随机，无法幂等」 — 弱幂等指「相同 key 返回同一 cached 结果」，不是要求模型 deterministic。
  • 幂等只做 Gateway — 写路径在 BFF/ERP；Gateway 无法感知业务是否已执行。
  • 去重键永不过期 — 与 session/idempotency 一样需要 TTL。

常见问题（FAQ）

Idempotency-Key 由客户端还是 BFF 生成？

写操作推荐客户端生成 UUID（双击同 key）；BFF 可为无 SDK 的 Web 代生成并随响应回传。tenantId 来自 JWT，不应由客户端在 Body 指定。

能否追求全链路 exactly-once？

跨 HTTP + MQ + ERP 的端到端 exactly-once 成本极高。CorpAssist 实践：at-least-once + 幂等键 + 对账；财务类写操作另加强一致与 Saga（见 advanced 集成章）。

本章结论

放在正文与复习之间，便于串起全章再做题。每条对应上文一个小节标题。

  • 令牌：Idempotency-Key + token store 状态机（Processing/Completed/Replay）；POST 写操作必带 key。
  • MQ：at-least-once 投递 + 消费端 SETNX dedup + DLQ；去重先于业务执行。
  • 重复请求：工具写路径强幂等；LLM POST 用 cache；SSE 续传用 Last-Event-ID；429 禁止无 key 重试 POST。

判断练习

  1. 权衡：Gateway 对 429 自动重试 POST /v1/chat，未带 Idempotency-Key。主要风险是什么？

    参考答案

    可能产生两次 LLM 调用与双倍 Token（NS3）；若 POST 链接触发工具写操作则可能重复改单。应对：客户端或 BFF 必带 Idempotency-Key；429 只重试 GET/已 key 的 POST，并读 Retry-After。

  2. 找错：消费端「先写向量库，成功后再 SETNX dedup」。哪一步顺序错误？应如何改？

    参考答案

    SETNX 必须在业务之前。否则并发重复消息仍会双写索引。失败 transient 时删除 dedup key 再 requeue；永久失败进 DLQ。
