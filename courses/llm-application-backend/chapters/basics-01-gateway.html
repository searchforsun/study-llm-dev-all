为什么要学本章

CorpAssist 企业助手在 S2 智能客服 场景下面向 ToC 高 QPS：流式对话、订单查询工具、RAG 检索不能各自为政。若鉴权、限流散落在 BFF 与 FastAPI 里各写一套，会出现密钥泄漏、配额穿透与排障盲区。统一 Spring Cloud Gateway 作为 api entry，才能在对模型计费之前挡住刷量、越权与异常路由——这是治理面（Spring）与能力面（Python）分工的第一道关。

学完你能
  • 按路径把 /v1/chat、/v1/tools/order、/v1/rag、/v1/stream 路由到 BFF 与 Python，并解释 SSE 路由的超时差异
  • 在 filter chain 中接入 gateway auth，将 tenantId 注入内网 Header
  • 用 Redis 令牌桶实现按租户的 gateway rate limit，并说明与 BFF Token 预算、模型 RPM 的分层关系（NS3 成本）
本章先记住 3 件事
  • 网关是治理面入口：路由、鉴权、限流在 Gateway 完成；Python 只做 AI 能力，不重复鉴权逻辑（NS2 确定性桥接在 BFF/工具层）。
  • LLM 路由按场景分：同步 JSON 与 llm route 的 SSE 流式须不同 route predicate 与超时策略。
  • 限流要分层：Gateway 挡 QPS/连接；BFF 管 Token 预算；模型侧还有厂商 RPM/TPM（见 限流与配额）。

路由

CorpAssist S2 · 查订单 + 流式回复（30 秒） 用户问「订单 SO-1024 到哪了？」App 经 Gateway 到 BFF：POST /v1/chat 触发 NS2 确定性工具 lookup_order；模型解释物流时走 GET /v1/stream/chat SSE。RAG 知识库问答走 /v1/rag/** 到 Python。Gateway 只做转发与治理，不执行业务逻辑。

Spring Cloud Gateway 基于 Predicate + Filter 组合。CorpAssist 双栈拓扑中，Gateway 是面向公网的唯一 upstream bff 入口；BFF 再按契约调用 Python 能力面。

治理面 · Gateway

路由、JWT、QPS 限流、Trace 注入、SSE 透传配置。

编排面 · Spring BFF

多轮会话、工具编排、Spring AI、对内调用 Python。

能力面 · FastAPI

RAG、评测、流式生成器；仅内网 + 信任 Gateway 头。

路由设计：反例 vs 推荐

不推荐
  • 所有路径 /** 打到单一服务
  • SSE 与 JSON 共用 30s 读超时
  • 公网暴露 Python :8000 绕过 Gateway
推荐
  • chat / stream / rag 分 route id，可独立扩缩与限流
  • stream 路由 response-timeout: 0 + 关闭缓冲（见 SSE 流式端到端）
  • Python 仅 ClusterIP，Ingress 只指向 Gateway
先想 10 秒：为什么「查订单」不宜把 ERP 直接配在 Gateway 路由里？

工具编排、会话上下文、幂等与审计在 BFF；Gateway 应保持薄。ERP 调用属于 NS2 确定性桥接，由 BFF 的 lookup_order 统一封装（见 幂等与重试 对写操作的要求）。

鉴权

LLM 接口不能「内网裸奔」。Gateway 层做 gateway auth：校验 JWT、解析 tenantId / userId，经 filter chain 注入 Header 传递给下游。下游在 mTLS 内网中信任 Gateway 已验签，不再各自持 JWT 公钥副本。

Filter 执行顺序（概念）

顺序Filter作用
1TraceId / RequestId全链路关联 S2 客诉与账单
2JWT 验签 + 租户解析拒绝过期/错 aud；注入 X-Tenant-Id
3RequestRateLimiter按租户 QPS（见下节）
4RouteToRequestUrl转发到 BFF / Python

反例 vs 推荐

不推荐
  • Python 服务各自解析 JWT，密钥多份拷贝
  • 仅校验 Token 存在，不校验 aud / 租户范围
  • 把模型 API Key 写在 URL Query 里转发
  • 信任客户端 Body 里的 tenantId
推荐
  • Gateway GlobalFilter 统一验签 + 注入 X-Tenant-Id、X-User-Id
  • 内网 mTLS；Python 只信 Gateway 注入的租户头
  • 模型 Key 只在 BFF/Vault，不下发到 App

GlobalFilter 示意（Java）

java 
@Component
public class TenantAuthFilter implements GlobalFilter, Ordered {
  @Override
  public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String auth = exchange.getRequest().getHeaders().getFirst(HttpHeaders.AUTHORIZATION);
    if (auth == null) {
      exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
      return exchange.getResponse().setComplete();
    }
    Claims claims = jwtParser.parse(auth);
    ServerHttpRequest req = exchange.getRequest().mutate()
        .header("X-Tenant-Id", claims.get("tenantId", String.class))
        .header("X-User-Id", claims.getSubject())
        .header("X-Trace-Id", traceId(exchange))
        .build();
    return chain.filter(exchange.mutate().request(req).build());
  }
  @Override public int getOrder() { return -100; }
}

常见误区
  • Gateway 只转发不过滤 — 恶意用户可直接打 Python 绕过配额（若网络可达）。
  • 鉴权下放给 LLM — 模型无法可靠执行安全策略；必须在 Gateway/BFF 硬编码。
  • SSE 不做鉴权 — 长连接更需建立前验签，避免「半开流式」占满连接。

限流

Gateway 限流保护全站：按 IP、用户或租户限制 QPS。S2 ToC 大促下，未做 Gateway 限流就扩容模型是典型 NS3 成本坑——重复请求还会放大 Token 账单（配合 幂等与重试）。

Redis 令牌桶（RequestRateLimiter）

yaml 
filters:
  - name: RequestRateLimiter
    args:
      redis-rate-limiter.replenishRate: 10
      redis-rate-limiter.burstCapacity: 20
      redis-rate-limiter.requestedTokens: 1
      key-resolver: "#{@tenantKeyResolver}"

限流维度适用场景主要实施层CorpAssist 示例

超限返回 429 Too Many Requests + Retry-After。客户端退避重试必须带 Idempotency-Key（写操作）或只重试 GET。

尽量在建立 SSE 连接前限流；已建立连接中途超限应发送 event: error 并关闭，避免半开连接泄漏。

常见问题（FAQ）

Gateway 限流了，还要在 BFF 限流吗？

要。Gateway 挡「入口洪水」；BFF 对 ERP 工具、单用户对话频率做细粒度保护。两层职责不同，不是重复建设。

流式路由为何不能和 chat 共用同一限流 key？

SSE 长连接占用的不是「每秒请求数」而是「并发连接数」。可为 /v1/stream/** 单独配置连接数上限或更长窗口的配额。

Python 服务还需要 API Key 吗？

对内可用 mTLS + 网络策略；若仍有服务间 Key，应通过 Vault 轮换，且不暴露给移动端。公网鉴权只在 Gateway/JWT 完成。

本章结论

放在正文与复习之间，便于串起全章再做题。每条对应上文一个小节标题。

  • 路由：chat/stream/rag/admin 分上游与超时；查订单工具经 BFF 编排，Gateway 保持薄；SSE 须 response-timeout: 0。
  • 鉴权：Gateway 统一 JWT 验签并注入租户/用户/Trace 头；禁止信任 Body 租户；Python 仅内网信任 Gateway。
  • 限流：Gateway 挡入口 QPS 与连接；Token 预算与 ERP 工具限流在 BFF；429 重试须配合幂等键。

判断练习

  1. 找错：同事配置「Python RAG 自行校验 JWT，Gateway 只做 Path 转发；移动端可配置 Python 直连地址做评测」。指出两条违反本章原则的做法。

    参考答案

    ① 鉴权未在 Gateway 统一，密钥与策略分裂，且可能绕过配额；② 公网/客户端直连 Python，审计与 gateway rate limit 失效。应 Gateway 验签 + 内网 mTLS + 租户头传递。

  2. 大促预案：① 只把模型副本数 ×3 ② Gateway 租户 QPS 减半 ③ BFF 对 lookup_order 加熔断 ④ 取消 JWT 校验提速。请选出应优先与最危险的一项。

    参考答案

    应优先：③，保护 ERP 与确定性工具链。最危险：④，入口失控且无法归因租户。仅 ① 不限入口，NS3 成本仍可能失控。
