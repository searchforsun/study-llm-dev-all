为什么要学本章

Gateway 入口限流 只挡连接级 QPS。CorpAssist S2 下，单租户内部测试脚本、超长 RAG 或恶意刷接口仍可能穿透 Gateway 耗尽模型账单——缺少 tenant quota 与 token budget 是 NS3 成本失控与「租户配额未隔离」生产坑。

学完你能
  • 用 Redis sliding window 实现 usage meter，区分 QPS 与 Token 两维配额
  • 按 tier 配置 qps cap 与月度 Token 上限，落实 fair usage 策略
  • 在 quota exceeded 时返回可行动 429，并触发 rate limit alert
本章先记住 3 件事
  • QPS 与 Token 分开计：高频短问与长 RAG 消耗模型成本完全不同。
  • 配额实时扣减 + 异步对账：Redis 热路径；日批落库与厂商账单核对。
  • 超限响应要可行动：429 + 剩余额度 + 升级入口，而非裸拒绝。

租户配额

CorpAssist S2 · 三租户同一集群（40 秒） 租户 A（标准企业）大促客服；租户 B（免费试用）被误配同一 Gateway 路由；租户 C（内部压测）脚本未停。若无 per-tenant tenant quota，B 的 2 QPS cap 无法阻止 C 占满模型 RPM，A 的真实用户反而被厂商 429——须全链路按 tenantId 计量。

配额是商业契约 + 稳定性的双重边界。CorpAssist 推荐三维度：

  • 入口 QPS — Gateway RequestRateLimiter，按 tenantId 令牌桶（见 basics-01）。
  • 业务 QPS — BFF 对 /v1/chat、/v1/rag 二次限流，防单连接 SSE 多路复用穿透。
  • 并发 SSE — 长连接占 Tomcat 线程与网关连接表；须单独 cap。

Gateway 租户限流（YAML）

yaml 
spring:
  cloud:
    gateway:
      routes:
        - id: corpassist-bff-chat
          uri: lb://corpassist-bff
          predicates:
            - Path=/v1/chat/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 20    # 稳态 QPS（按 tier 动态覆盖）
                redis-rate-limiter.burstCapacity: 40
                key-resolver: "#{@tenantKeyResolver}"   # 从 JWT 解析 tenantId
                deny-empty-key: true

sliding window 计数（Redis Lua 示意）

bash 
# Key: ratelimit:{tenantId}:chat:window
# ZSET score=毫秒时间戳 member=requestId
# 每次请求：ZREMRANGEBYSCORE 删除窗口外 → ZCARD 计数 → 与 qps cap 比较
不推荐
  • 全集群共用一个全局 QPS，不区分 tenantId
  • 仅限制 Gateway，BFF 直连模型绕过计量
  • 固定窗口计数，边界时刻双倍流量（burst 漏洞）
推荐
  • sliding window 或令牌桶，按 tenant 独立 key
  • Gateway + BFF 双层；模型厂商 RPM 作为第三层硬顶
  • fair usage：大租户 burst 不得挤占小租户 steady 配额
先想 10 秒：为何「全局限 1000 QPS」不够？

1000 QPS 可能被单一压测租户占满，其他企业租户 SLA 归零。配额必须按 tenantId 隔离，与 多租户边界 一致。

Token 预算

Token budget 反映模型成本，与 QPS 正交：一次 RAG 问答可能消耗 8k tokens，但只计 1 QPS。CorpAssist 在每次 LLM 调用后异步累加 usage，并在请求前做预检（soft）与调用后扣减（hard）。

Python usage meter（FastAPI 回调）

python 
async def record_usage(tenant_id: str, usage: TokenUsage) -> None:
    key = f"quota:{tenant_id}:{month_key()}"  # 如 quota:acme-corp:202605
    pipe = redis.pipeline()
    pipe.hincrby(key, "tokens", usage.total)
    pipe.hincrby(key, "requests", 1)
    pipe.expire(key, 60 * 60 * 24 * 40)  # 跨月保留 40 天供对账
    await pipe.execute()

async def pre_check_budget(tenant_id: str, estimated: int, cap: int) -> bool:
    current = int(await redis.hget(f"quota:{tenant_id}:{month_key()}", "tokens") or 0)
    return current + estimated <= cap

BFF 预检 + 超限拦截（Java）

java 
public void assertTokenBudget(String tenantId, int estimatedTokens) {
  QuotaTier tier = quotaService.getTier(tenantId);
  long used = usageMeter.getMonthlyTokens(tenantId);
  if (used + estimatedTokens > tier.getMonthlyTokenCap()) {
    throw new QuotaExceededException(tenantId, used, tier.getMonthlyTokenCap());
  }
}

告警

限流与配额不是「配完即忘」。须通过 rate limit alert 在用户投诉之前发现异常消耗或配置错误。

分级阈值与动作

用量占配额级别告警渠道建议动作
≥ 80%预警邮件 + 租户管理员 Dashboard通知客户续费/升级；检查是否有异常脚本
≥ 100%硬限企业微信 on-call + 租户 webhook新请求 429；存量 SSE 允许收尾
≥ 120%异常P2 告警可能存在 meter 漏计或 bypass；紧急对账

Prometheus 规则片段

yaml 
groups:
  - name: corpassist-quota
    rules:
      - alert: TenantTokenBudgetWarning
        expr: corpassist_tenant_token_ratio > 0.8
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "租户 {{ $labels.tenant_id }} Token 已达 80%"
      - alert: TenantQpsSaturated
        expr: rate(corpassist_ratelimit_denied_total[5m]) > 10
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "租户 {{ $labels.tenant_id }} 频繁触发 QPS 限流"
常见误区
  • 只告警不阻断 — 100% 后仍放行会导致 NS3 账单失控；须 hard limit。
  • 429 无 Retry-After — 客户端无法退避，形成 retry 风暴加剧限流。
  • 内部压测账号无独立 tenantId — 与生产租户共用配额，误伤 SLA。
QPS 未超限但月账单爆掉，最先查哪？
查 token budget meter 是否覆盖全部模型调用路径（含 Python 直连、批处理 embed、重试重复计费）。低频超长 RAG 是典型漏计场景。
fair usage 与大客 burst 如何平衡？
大客可协商更高 qps cap 与 burstCapacity，但不得占用共享模型 RPM 的全局上限；必要时专属模型 endpoint（见 advanced 配置）。

本章结论

配额是 Gateway 限流的延伸；做题前对照下表。

  • 租户配额：QPS / 月 Token / SSE 并发分 tier；Gateway + BFF 双层 + sliding window 按 tenantId 隔离。
  • Token 预算：预检 + 调用后 meter 累加；Redis 热路径，日批对账厂商账单。
  • 告警：80/100/120 分级；quota exceeded 返回可行动 429 + upgradeUrl。

判断练习

  1. 排障：某租户 Gateway QPS 未超限，但月 DashScope 账单为预期的 12 倍。最可能漏了哪层计量？

    参考答案

    Token meter（或 embed 子计量）：低频但超长 RAG / 批处理 embed 不触发 QPS 限流。须检查 BFF 与 Python 所有模型调用是否都调用 record_usage，以及重试是否重复计费。

  2. 找错：429 响应仅 {"error":"too many requests"}，无 Retry-After 与 upgradeUrl。违反本章哪条？

    参考答案

    违反「超限响应要可行动」：客户端无法退避（缺 Retry-After）且租户无法升级（缺 upgradeUrl / used / limit）。S2 体验与商务转化都会受损。

  3. 权衡：把标准企业月 Token 从 5M 提到 50M 且不告警，短期客诉下降，长期风险是什么？

    参考答案

    NS3 成本不可控；异常脚本或 prompt 注入导致静默超支；失去 80% 预警窗口。应提 tier 价格或专属模型，而非取消 meter 与告警。
