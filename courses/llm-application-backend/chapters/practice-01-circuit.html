已完成熔断、舱壁与降级

约 42 分钟 · 阶段：弹性与多租户 · 能力：用 Resilience4j 保护 LLM 下游链路并实现可观测降级

为什么要学本章

CorpAssist S2 智能客服 高峰时，DashScope 限流、Python RAG OOM、ERP 查单慢都会拖垮 BFF 线程池。没有 circuit breaker 与 model fallback，一次下游雪崩会让所有租户连「转人工」入口都进不去——本课明确的生产坑是：熔断配了但没 fallback，客户端只看到 HTTP 500。

学完你能
  • 为 BFF→Python RAG、BFF→模型 API 分别配置 Resilience4j 熔断实例与 bulkhead 舱壁
  • 设计 default reply 与转人工 degrade policy，SSE 流式也能优雅收尾
  • 根据 S2 压测数据设置 failure threshold，并用 timeout isolate 防止慢调用占满线程
本章先记住 3 件事
  • 熔断保护调用方：快速失败，把线程还给网关与 SSE 连接，给下游恢复窗口。
  • 舱壁隔离资源池：LLM 长调用与 ERP 短查询用不同 bulkhead，避免「查单慢拖死生成」。
  • 降级要有用户可见文案：不是空白 500，而是「咨询较多，已记录工单，可转人工」+ traceId。

Resilience4j

CorpAssist S2 · 大促咨询洪峰（45 秒） 09:00 活动开始，/v1/chat QPS 从 200 飙到 1800。Python RAG 向量检索 P99 从 800ms 升到 6s；DashScope 返回 429。BFF 若对每个慢请求阻塞 120s，Tomcat 200 线程 30 秒即耗尽——必须在调用边界用 Resilience4j 熔断 + 舱壁 + 超时隔离。

Spring Cloud 2024+ 推荐 Resilience4j 替代 Hystrix。CorpAssist 双栈至少包两层下游：

  • pythonRag — BFF 调 FastAPI /v1/rag/query，失败率与慢调用影响检索链路。
  • llmProvider — BFF / Spring AI 调 DashScope 等模型 API，429/5xx 须独立熔断，不与 RAG 共用计数窗口。
  • erpLookup（可选第三实例）— NS2 查订单工具 Feign 调用，与 LLM bulkhead 隔离。
CircuitBreaker

统计滑动窗口内失败率；超 failure threshold 则 Open，直接拒绝调用。

Bulkhead

限制并发槽位；LLM 与 ERP 分池，防止一类调用饿死另一类。

TimeLimiter

与 timeout isolate 配合：超时算失败，计入熔断窗口。

CorpAssist 熔断 + 舱壁配置（YAML）

yaml 
resilience4j:
  circuitbreaker:
    instances:
      pythonRag:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 20          # S2：窗口不宜过大，否则 Open 太慢
        failureRateThreshold: 50       # 50% 失败率触发 Open
        slowCallRateThreshold: 60
        slowCallDurationThreshold: 3s  # RAG P99 目标 < 3s
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 5
        recordExceptions:
          - java.util.concurrent.TimeoutException
          - org.springframework.web.client.HttpServerErrorException
      llmProvider:
        slidingWindowSize: 30
        failureRateThreshold: 40       # 模型 429 更敏感，阈值略低
        waitDurationInOpenState: 20s
  bulkhead:
    instances:
      llmCall:
        maxConcurrentCalls: 32         # 对齐 BFF 线程池 × 副本数预算
        maxWaitDuration: 500ms         # 超出则 BulkheadFullException → fallback
      erpLookup:
        maxConcurrentCalls: 64
        maxWaitDuration: 200ms
  timelimiter:
    instances:
      pythonRag:
        timeoutDuration: 4s
      llmProvider:
        timeoutDuration: 90s           # 流式首 token 可长，但须监控 stall

熔断状态机

Closed → Open → HalfOpen
stateDiagram-v2
  [*] --> Closed
  Closed --> Open: 失败率超 failureRateThreshold
  Closed --> Open: 慢调用率超 slowCallRateThreshold
  Open --> HalfOpen: waitDurationInOpenState 结束
  HalfOpen --> Closed: 探测调用成功达阈值
  HalfOpen --> Open: 任一次探测失败
  note right of Open: 直接走 fallback\n不再打满下游

Open 期间所有调用立即失败并触发 default reply；HalfOpen 用少量探测验证下游是否恢复。

S2 高峰参数：反例 vs 推荐

不推荐
  • 全下游共用一个 default 熔断实例
  • failureRateThreshold: 90 + 窗口 200，Open 前已拖垮线程池
  • 无 TimeLimiter，仅依赖 Feign readTimeout 120s
  • 熔断 Open 仍抛未捕获异常到 SSE 客户端
推荐
  • RAG / LLM / ERP 分实例，指标独立暴露到 Actuator
  • 结合压测设 slowCallDurationThreshold，慢调用计入失败
  • bulkhead maxWaitDuration 短于用户可感知等待
  • Open 走 fallbackMethod，SSE 发送 terminal event 后关闭

BFF 注解式接入（Java）

java 
@CircuitBreaker(name = "pythonRag", fallbackMethod = "ragFallback")
@Bulkhead(name = "llmCall", type = Bulkhead.Type.SEMAPHORE)
@TimeLimiter(name = "pythonRag")
public CompletableFuture<RagResponse> queryRagAsync(RagRequest req) {
  return CompletableFuture.supplyAsync(() -> pythonRagClient.query(req));
}

private CompletableFuture<RagResponse> ragFallback(RagRequest req, Throwable t) {
  log.warn("rag degraded tenant={} trace={}", req.tenantId(), MDC.get("traceId"), t);
  return CompletableFuture.completedFuture(
    RagResponse.degraded(DefaultReplies.RAG_BUSY, /* handoffTicket */ true));
}
先想 10 秒：为什么 LLM 与 ERP 不能共用同一个 bulkhead？

大促时 LLM 长连接占满槽位，NS2 查订单工具会被饿死——用户问「订单到哪了」也超时。舱壁本质是故障域隔离，与 Gateway 路由分路径 同一思路。

默认回复

Default reply 是熔断 / 舱壁 / 超时后的用户契约，不是日志里的异常栈。CorpAssist S2 要求：

  • 可读中文 — 说明「当前咨询较多」，避免技术错误码。
  • 可行动 — 提供转人工、工单号或预计回电时间。
  • 可追踪 — 响应体含 traceId，客服系统可关联全链路。
  • SSE 友好 — 流式场景发送 event: error 或 event: done 后正常关闭，防客户端挂死。

degrade policy 分级表

级别触发条件用户可见行为运维动作
D1 · 功能降级 RAG 熔断 Open 无知识库引用，通用话术 + 转人工入口 告警 RAG 失败率；检查 Python 副本与向量库
D2 · 模型降级 主模型 429/5xx 切换备用小模型（见下节 L2） 监控 fallback 比例；联系厂商提额
D3 · 全链路 LLM + RAG 均 Open 静态 FAQ 列表 + 强制转人工 P1 on-call；考虑 Gateway 入口限流（见 限流与配额）

常见误区
  • fallback 里再调 LLM — 下游已 Open 时二次调用会放大故障；D1/D3 应用静态话术或缓存 FAQ。
  • 吞掉异常不记指标 — degraded=true 须计入 Prometheus，否则「静默降级」数周无人知。
  • 仅返回 503 无 body — App 无法展示转人工；违反 S2 体验 SLA。

模型 fallback

Model fallback 是 degrade policy 在模型层的具体实现：主模型不可用时，按预定路由降级，并缩短 prompt 以控制成本（NS3）。

三级 fallback 路由

级别触发行为质量 / 成本
L1 单次超时 / 5xx（Closed 状态） 幂等键下重试 1 次；仍失败进入 L2 质量不变；延迟 +1 RTT
L2 熔断 Open 或连续 429 路由 qwen-turbo 等小模型；截断 RAG topK 与 history 质量略降；Token 成本降 60%+
L3 全模型不可用 Redis 缓存 FAQ + 静态 default reply + 转人工 无生成；成本为零
模型 fallback 决策流
flowchart TD
  Req[Chat 请求]
  CB{circuitbreaker\nllmProvider}
  L1[L1: 重试 1 次]
  L2[L2: 小模型 + 短 prompt]
  L3[L3: FAQ + 转人工]
  Ok[正常响应]
  Req --> CB
  CB -->|Closed 且失败| L1
  L1 -->|成功| Ok
  L1 -->|仍失败| L2
  CB -->|Open| L2
  L2 -->|仍失败| L3
  CB -->|Closed 且成功| Ok

L2/L3 须打点 llm_fallback_total{level="L2"}；fallback 率 > 5% 持续 10min 应触发告警。

Spring AI 路由示意（Java）

java 
@CircuitBreaker(name = "llmProvider", fallbackMethod = "chatFallback")
public ChatResponse chat(ChatRequest req) {
  return primaryModel.call(truncateContext(req, PRIMARY_MAX_TOKENS));
}

ChatResponse chatFallback(ChatRequest req, Throwable t) {
  meterRegistry.counter("llm_fallback_total", "level", "L2").increment();
  return fallbackModel.call(truncateContext(req, FALLBACK_MAX_TOKENS));
}

timeout isolate 与慢调用

LLM 流式场景下，「首 token 90s、后续 stall 无数据」也应由 TimeLimiter 或 idle timeout 切断，并计入慢调用统计——否则 failure threshold 永远达不到，线程却被挂住。BFF 侧建议：首 token 超时 90s，chunk 间隔超时 15s（详见 SSE 流式端到端）。

熔断 Open 后用户反复点「重试」会怎样？
若 App 直连 BFF 重试，Open 期间每次仍快速 fallback，不会打满下游——这是熔断的价值。但若 Gateway 未限流，重试风暴仍会占满 BFF 连接；须配合 租户 QPS 配额 与客户端指数退避。
HalfOpen 探测失败立刻再 Open，会不会「永远恢复不了」？
会，若下游真未修复。此时应依赖告警与 runbook 修下游，而不是调大 threshold 掩盖问题。可临时手动切 L3 FAQ 保体验，但须记录变更。

本章结论

每条对应上文一个小节；做题前可先对照自检。

  • Resilience4j：RAG / LLM / ERP 分实例熔断 + LLM 专用 bulkhead；S2 窗口 20～30、失败率 40～50% 为起点，压测后微调。
  • 默认回复：fallbackMethod 返回可读降级 + traceId；SSE 发 terminal event，禁止裸 500。
  • 模型 fallback：L1 幂等重试 → L2 小模型短 prompt → L3 FAQ + 人工；须监控 fallback 率并告警。
  • timeout isolate：TimeLimiter 与慢调用阈值并用；超时计入熔断窗口，避免「假 Closed 真阻塞」。

判断练习

  1. 找错：熔断 Open 后直接返回 HTTP 500，响应 body 为空。违反本章哪条原则？应如何改？

    参考答案

    违反「降级要有用户可见文案」：应走 fallbackMethod 返回 default reply（含转人工与 traceId），SSE 场景发 terminal event。500 无 body 会导致 App 无法兜底。

  2. 排障：S2 高峰 RAG P99 6s，但 circuitbreaker 一直 Closed，BFF 线程仍耗尽。最可能漏配了哪两项？

    参考答案

    ① slowCallRateThreshold / slowCallDurationThreshold — 慢调用未计入失败；② bulkhead 或 TimeLimiter — 无并发/超时隔离，线程被 120s 读超时占满。仅调低 failureRateThreshold 不够。

  3. 权衡：HalfOpen 期间把 permittedNumberOfCallsInHalfOpenState 从 5 调到 50，能加快恢复验证，但有何风险？

    参考答案

    下游仍不稳定时，50 路探测同时涌入可能再次打满 Python/向量库，导致刚恢复又崩溃（flapping）。S2 建议保持小批量探测，结合告警人工确认后再放大流量。
