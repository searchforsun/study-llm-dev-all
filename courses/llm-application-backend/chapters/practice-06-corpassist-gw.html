先想 10 秒：压测时为什么要带真实 JWT 与 tenantId？

否则限流 key、租户隔离、配额统计均与生产不一致，P95 会失真。k6 应用 setup 阶段换取 token 或预置测试租户。

E2E 验收：curl 走全链

在联调环境按 e2e gw 顺序验证：鉴权失败 → 限流 → 正常 chat → SSE → 异步索引。

1. 鉴权（应 401）

bash
curl -s -o /dev/null -w "%{http_code}" \
  -X POST "http://localhost:8080/v1/chat/completions" \
  -H "Content-Type: application/json" \
  -d '{"message":"hello"}'
# 期望: 401

2. 同步对话（应 200）

bash
curl -s -X POST "http://localhost:8080/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sessionId":"capstone-1","message":"CorpAssist 退款流程？"}' | jq .

3. SSE 流式

bash
curl -N -H "Authorization: Bearer $TOKEN" \
  "http://localhost:8080/v1/stream/chat?sessionId=capstone-1&message=你好"

4. 异步索引 + job 查询

bash
JOB=$(curl -s -X POST "http://localhost:8080/v1/rag/documents" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"uri":"s3://demo/policy.pdf","idempotencyKey":"cap:v1"}' | jq -r .jobId)
curl -s "http://localhost:8080/v1/rag/jobs/$JOB" -H "Authorization: Bearer $TOKEN" | jq .

k6 脚本节选（chat_sse.js）

javascript 
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 30,
  duration: '2m',
  thresholds: {
    http_req_duration: ['p(95)<3000'],
    http_req_failed: ['rate<0.01'],
  },
};

const TOKEN = __ENV.CORPASSIST_TOKEN;
const BASE = __ENV.GW_URL || 'http://localhost:8080';

export default function () {
  const res = http.post(
    `${BASE}/v1/chat/completions`,
    JSON.stringify({ sessionId: `k6-${__VU}`, message: 'ping' }),
    { headers: { Authorization: `Bearer ${TOKEN}`, 'Content-Type': 'application/json' } }
  );
  check(res, { 'status 200': (r) => r.status === 200 });
  sleep(1);
}

读结果：该担心什么

异常信号
  • P95 陡升但 CPU 低 → 下游 Python/模型排队
  • 429 激增 → Gateway 限流过严或 key 错误
  • 503 + 熔断日志 → 需调 Resilience4j 阈值
  • SSE 大量断连 → timeout 或 LB 空闲断开
健康基线
  • 错误率 < 1%，无持续 503
  • 限流 429 仅在超过 replenishRate 时出现
  • Trace 中 Gateway→BFF→Python 分段延迟可解释
  • 压测后 MQ 无 DLQ 堆积
常见误区
  • 直压 Python 当全链路 — 绕过 Gateway 限流/鉴权，结论不可上线。
  • 不设 threshold — k6 跑完无 PASS/FAIL，无法回归对比。
  • 忽略 SSE — 只压 REST 会漏掉长连接与缓冲问题。

动手练习

勾选 demos/practice-06-corpassist-gw-lab/delivery-checklist.md，跑 k6 并写一页压测摘要（约 30 分钟）。

操作步骤

  1. 启动 CorpAssist 最小栈（Gateway + BFF + Python + Redis + RabbitMQ），确认 health 均为 UP。
  2. 按正文顺序执行 E2E curl：401 → chat 200 → SSE → 索引 job；记录每步 HTTP 码。
  3. 制造熔断：停 Python 30s，再请求 chat，确认 BFF 返回 fallback 而非 504 风暴。
  4. 运行 k6：k6 run --vus 30 --duration 2m scripts/chat_sse.js，保存 terminal summary 截图或文本。
  5. 填写 delivery-checklist 四项；在报告中写：峰值 QPS、P95、是否触发 429/503，以及一条优化建议。

判断练习

  1. 同事说：「压测 P95 很好，因为直打 Python 8000 端口，没走 Gateway。」该结论能否代表 S2 上线表现？

    参考答案

    不能。绕过 corpassist gateway 则未测鉴权、限流与路由开销；e2e gw 必须从对外入口压测。

  2. capstone 验收：429 大量出现。应先查？A) 模型 API Key B) Gateway replenishRate C) Milvus 磁盘 D) proto 版本

    参考答案

    B。429 是 Gateway/限流层信号；Key 问题多为 401/402；Milvus 影响检索延迟；proto 不匹配多为 500/连接失败。
