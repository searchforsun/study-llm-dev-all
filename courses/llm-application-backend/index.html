CorpAssist 企业集成全景：新 AI 层与 Java ERP

CorpAssist · 采购智能审批（45 秒） 员工在 CorpAssist 提交采购申请；BFF 需从 ERP 拉取供应商信用与历史订单，再调 Python RAG 生成建议摘要，最后把「通过/驳回」写回 ERP 工单。Gateway 已在 实战：CorpAssist 网关层 完成鉴权与限流——本章关注 BFF 之后如何碰存量系统。

典型 legacy integration 分层：

  • 入口不变 — 仍经 Gateway；BFF 是对外聚合层，不暴露 ERP 原始 URL。

Feign 调用存量服务：契约、超时与幂等

BFF 通过 OpenFeign 声明式 HTTP 访问 ERP 暴露的 REST（或经 ESB 转发的接口）。生产必备：

  • 显式超时 — ERP 常比 BFF 慢一个数量级；connect/read 分开设，并与 Resilience4j 熔断对齐。
  • 幂等键 — 写工单时带 X-Idempotency-Key，避免 Feign 重试造成重复审批。
  • 降级语义 — 读失败返回缓存快照 + DEGRADED 标记，而不是 500 拖垮整条 chat 链路。

领域事件：解耦写回与跨系统通知

同步 Feign 适合当场需要结果的读路径。写 ERP、通知 OA、触发索引任务等副作用，应发布 领域事件，由消费者异步执行——这与 消息队列与异步任务 一脉相承，但事件名应体现业务语义（如 PurchaseApproved）。

审批通过：事件驱动写回 ERP
sequenceDiagram
  participant BFF as corpassist-bff
  participant DB as 本地库
  participant OB as outbox
  participant K as Kafka
  participant ERP as Legacy ERP
  BFF->>DB: 更新申请状态 APPROVED
  BFF->>OB: 同事务写入 Outbox 行
  Note over BFF,OB: 本地事务提交
  OB->>K: relay 发布 PurchaseApproved
  K->>ERP: erp-connector 消费写工单

Outbox 模式 保证「DB 状态」与「待发消息」同事务一致，避免 commit 成功但 MQ 发送失败。

不推荐
  • 事务内直接 kafkaTemplate.send
  • Feign 写 ERP 成功后再改本地库（双写不一致）
  • 事件 payload 塞整行 ERP 宽表
推荐
  • 本地状态 + Outbox 同一 @Transactional
  • 消费者 erp-connector 幂等消费
  • 事件只带业务键与摘要字段

事务边界：LLM、ERP 与本地库如何划分

事务边界 的核心问题：哪些操作必须在同一 ACID 事务里，哪些必须拆到事务外。 CorpAssist 典型错误是把「调 Python LLM 30s」包进 @Transactional，导致连接池耗尽、锁持有过久。

操作是否进本地 DB 事务一致性策略
保存用户草稿、租户配额扣减 是 本地 ACID
写入 Outbox + 更新申请状态 是 Outbox 保证与消息一致
调用 Python RAG / LLM 否 超时 + 熔断；结果落库单独短事务
Feign 写 ERP 工单 否（异步） Saga 轻量：失败事件触发补偿/人工工单
补偿路径（Saga-lite，无需 Seata）
stateDiagram-v2
  [*] --> Pending
  Pending --> ApprovedLocal: BFF 本地通过
  ApprovedLocal --> ErpSynced: 消费者写 ERP 成功
  ApprovedLocal --> NeedsReconcile: ERP 失败 N 次
  NeedsReconcile --> ErpSynced: 人工/批任务补偿
  NeedsReconcile --> Revoked: 撤销本地批准
java 
@Transactional
public UUID startReview(StartReviewCmd cmd) {
  UUID reviewId = reviewRepo.savePending(cmd);
  outbox.enqueue(new ReviewStarted(reviewId, cmd.tenantId()));
  return reviewId;
}

public void enrichWithLlm(UUID reviewId) {
  LlmSuggestion s = pythonClient.suggest(reviewId);
  reviewRepo.saveSuggestion(reviewId, s);
}
常见误区
  • Feign 当本地方法 — ERP 超时要可预期，并有熔断与降级。
  • 先 LLM 后开事务 — 用户取消请求时本地无记录，难以审计。
  • 双写 ERP + 本地库 — 任一失败即不一致；写 ERP 走 Outbox 消费者。
  • 事件不带 tenantId — 多租户场景下消费者无法隔离。

system-bridge 模块职责与对接清单

System bridge 是 BFF 内独立包：对外只暴露 CorpAssist 语义 API，对内集中处理 ERP quirks（时区、枚举码表、分页上限）。

集成点方式失败语义
读订单/供应商 Feign GET + 缓存 60s 降级快照 / 明确错误码
写审批结果 Outbox → erp-connector 重试 + NeedsReconcile 状态
LLM 建议摘要 HTTP → Python 熔断返回模板文案
能否让 Python 直接 Feign ERP？
不推荐。ERP 契约与 Java 团队共管，BFF 作为唯一桥 便于鉴权、审计与租户隔离；Python 只处理 AI 域。
Outbox 和 MQ 章节的索引任务有何不同？
模式相同：可靠投递。ERP 回写偏强业务语义与幂等，需独立 consumer group 与 DLQ 告警。
Feign 契约如何进 CI？
见 CI/CD 与发布：WireMock/契约测试失败则阻断镜像推送。

本章结论

CorpAssist 企业集成的「稳」来自边界清晰，而不是更多同步调用。

  • 存量集成：BFF + system-bridge 对接 ERP；Feign 同步读，带超时熔断与幂等写。
  • 事件解耦：写 ERP 走 Outbox + 领域事件，避免事务内外部 IO。
  • 事务边界：LLM/ERP 长调用不进 @Transactional；失败用 Saga-lite 状态机补偿。
  • 与前置章节：网关 capstone 提供入口治理。
  • 与后续章节：Nacos/Vault 管理 ERP 端点；CI/CD 对 Feign 契约设门禁。

架构

S2 系统设计 · 30 秒开场（CorpAssist） 「我们做的是 ToC 智能客服：用户经 App 访问 Spring Cloud Gateway，JWT 鉴权 + 租户限流后进入 BFF 做多轮对话与查订单工具；流式回复走 SSE 透传到 Python 生成器；RAG 走独立路由。Gateway 挡入口洪水，BFF 管 Token 预算与熔断，Python 专注推理。」

估算错了会被挂吗？

通常不会。面试官看你是否先澄清假设、能否把数字与 Gateway/BFF/Redis/模型分层对应。数量级差 2× 可接受，但「只扩模型不限流」会扣分。

如何把 practice-06 压测写进答案？

「我们在预发用 k6 从 Gateway 入口压测，30 VU 下 P95 1.2s、failed <1%，以此反推单实例容量并留 30% 冗余。」这比纯理论更有说服力。

429 大量出现说明什么？

入口限流生效或 replenishRate 过低。应先查 Gateway 租户配额（practice-02），而非盲目升模型规格。

常见误区
  • 只背组件清单 — 缺少数据流与失败降级路径。
  • 忽略 SSE — 按 REST QPS 扩容会在长连接场景击穿 Gateway。
  • 不算 Token 成本 — NS3 成本与 high concurrency 强相关，需提 BFF 预算层。

路由

CorpAssist S2 · 查订单 + 流式回复（30 秒） 用户问「订单 SO-1024 到哪了？」App 经 Gateway 到 BFF：POST /v1/chat 触发 NS2 确定性工具 lookup_order；模型解释物流时走 GET /v1/stream/chat SSE。RAG 知识库问答走 /v1/rag/** 到 Python。Gateway 只做转发与治理，不执行业务逻辑。

Spring Cloud Gateway 基于 Predicate + Filter 组合。CorpAssist 双栈拓扑中，Gateway 是面向公网的唯一 upstream bff 入口；BFF 再按契约调用 Python 能力面。

推荐
  • chat / stream / rag 分 route id，可独立扩缩与限流
  • stream 路由 response-timeout: 0 + 关闭缓冲（见 SSE 流式端到端）
  • Python 仅 ClusterIP，Ingress 只指向 Gateway
先想 10 秒：为什么「查订单」不宜把 ERP 直接配在 Gateway 路由里？

工具编排、会话上下文、幂等与审计在 BFF；Gateway 应保持薄。ERP 调用属于 NS2 确定性桥接，由 BFF 的 lookup_order 统一封装（见 幂等与重试 对写操作的要求）。

鉴权

LLM 接口不能「内网裸奔」。Gateway 层做 gateway auth：校验 JWT、解析 tenantId / userId，经 filter chain 注入 Header 传递给下游。下游在 mTLS 内网中信任 Gateway 已验签，不再各自持 JWT 公钥副本。

GlobalFilter 示意（Java）

java 
@Component
public class TenantAuthFilter implements GlobalFilter, Ordered {
  @Override
  public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String auth = exchange.getRequest().getHeaders().getFirst(HttpHeaders.AUTHORIZATION);
    if (auth == null) {
      exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
      return exchange.getResponse().setComplete();
    }
    Claims claims = jwtParser.parse(auth);
    ServerHttpRequest req = exchange.getRequest().mutate()
        .header("X-Tenant-Id", claims.get("tenantId", String.class))
        .header("X-User-Id", claims.getSubject())
        .header("X-Trace-Id", traceId(exchange))
        .build();
    return chain.filter(exchange.mutate().request(req).build());
  }
  @Override public int getOrder() { return -100; }
}

常见问题（FAQ）

Gateway 限流了，还要在 BFF 限流吗？

要。Gateway 挡「入口洪水」；BFF 对 ERP 工具、单用户对话频率做细粒度保护。两层职责不同，不是重复建设。

流式路由为何不能和 chat 共用同一限流 key？

SSE 长连接占用的不是「每秒请求数」而是「并发连接数」。可为 /v1/stream/** 单独配置连接数上限或更长窗口的配额。

Python 服务还需要 API Key 吗？

对内可用 mTLS + 网络策略；若仍有服务间 Key，应通过 Vault 轮换，且不暴露给移动端。公网鉴权只在 Gateway/JWT 完成。

为什么要学本章

智能客服首字延迟（TTFT）直接影响 S2 体验。LLM 回复必须经 SSE stream 逐 token 推送；若 Gateway 缓冲整包、BFF 聚合 JSON、或断线无 reconnect sse，用户会看到「卡住再一次性弹出」——这是 CorpAssist 生产高频坑，也是 NS2/NS3 联调时最容易被忽略的链路问题。

学完你能
  • 在 Spring BFF 用 spring sse 转发 Python python generator 输出，并为每个 chunk 分配 event id
  • 配置 Gateway stream proxy 不缓冲 event stream，实现逐跳 chunk forward
  • 设计 last event id、心跳与 Redis 短时缓冲，支持移动端断线续传
本章先记住 3 件事
  • SSE 是单向 HTTP 长连接：格式为 data: 行 + 双换行；比 WebSocket 更简单，适合 LLM 文本流（前提：客户端只收不发）。
  • 每一跳都要「流式不缓冲」：Gateway、BFF、Nginx/Ingress 都要关闭 response buffering 并设置合理读超时（见 Spring Cloud Gateway 与 LLM）。
  • 断线重连要有 id：服务端为每个 chunk 分配单调递增 id，客户端带 Last-Event-ID 重连；会话上下文仍靠 会话与状态。

Spring SSE

CorpAssist S2 · 客服流式回复（30 秒） 用户问「订单 SO-1024 物流到哪了？」App 经 Gateway 到 BFF：GET /v1/stream/chat?sessionId=c-8f2a。BFF 从 Redis 加载会话摘要，调用 Python 流式接口，将 token 逐段写入 WebFlux Flux<ServerSentEvent>；客户端 EventSource 实时渲染，首 token 须在 500ms 内可见（SLO 示意）。

BFF 层职责：鉴权后的会话上下文、调用 Python、统一 traceId、对前端暴露 text/event-stream。Gateway 只做 stream proxy，不在此层解析 SSE 帧内容。

async def token_stream(session_id: str, after: int = 0):
    event_id = after
    async for chunk in llm.astream(session_id):
        event_id += 1
        payload = json.dumps({"text": chunk}, ensure_ascii=False)
        yield f"id: {event_id}\nevent: token\ndata: {payload}\n\n"
        if event_id % 20 == 0:
            yield ": ping\n\n"  # 心跳，防代理 60s 空闲断开

常见问题（FAQ）

重新 POST 可能触发第二次 LLM 推理，双倍 Token 费用且回复可能不一致。续传只补发未送达帧；若生成已结束则返回 replay_done 或缓存全文。

尽量在建立连接前限流（见 gateway rate limit）；已建立的长连接中途超限应发 event: error 并关闭。

Redis

CorpAssist S2 · 多轮查订单（30 秒） 用户：「查订单」→ 助手：「请提供订单号」→ 用户：「SO-1024」。BFF 在 Redis 更新 intent=query_order、slots.orderId=SO-1024，再触发 NS2 工具 lookup_order。下一轮用户改问「改地址」时，intent 切换，槽位重置规则由 DST 状态机定义——不靠模型记忆。

public DialogState load(String tenantId, String conversationId) {
  String key = "ca:sess:" + tenantId + ":" + conversationId;
  Map<Object, Object> hash = redis.opsForHash().entries(key);
  if (hash.isEmpty()) return DialogState.newSession(tenantId, conversationId);
  return DialogState.fromRedis(hash);
}

public void save(DialogState state) {
  String key = "ca:sess:" + state.tenantId() + ":" + state.conversationId();
  redis.opsForHash().putAll(key, state.toMap());
  redis.expire(key, idleTimeout);  // 滑动 TTL，见「过期」节
}

常见问题（FAQ）

会话过期后用户继续用旧 conversationId 发消息怎么办？

返回 410 Gone 或友好提示「会话已过期」，引导新建 conversation；可选携带脱敏 summary 到新会话。写操作须新 Idempotency-Key。

Redis 会话与 Spring Session 是什么关系？

Spring Session 可复用 Redis 存 HTTP Session；CorpAssist 更常用自定义 DialogState Repository，字段贴合 NLU/工具槽位，不必强行塞进 Servlet Session。

常见误区
  • 无限 TTL — Redis 内存膨胀；合规也无法解释数据留存。
  • 过期只 DEL 不归档 — 客诉与审计无法回溯。
  • 把 LLM 输出当唯一状态 — 工具参数须结构化存储，模型幻觉不能作为订单号来源。

为什么要学本章

移动端弱网、Gateway 429 重试、MQ at least once 投递都会产生 duplicate request。S2 客服下重复「改地址」或重复 LLM 调用会直接引发客诉与 NS3 Token 成本失控——gateway rate limit 触发的 429 若盲目重试 POST，更是典型事故源。

本章先记住 3 件事
  • 写操作必须幂等：工具改单、发券、索引任务提交要有 Idempotency-Key（NS2 确定性写路径）。
  • 读/生成可接受 at-least-once + 去重：相同 key 返回缓存或相同 stream，避免双倍计费。
  • exactly-once 端到端很难：组合幂等键 + 去重表 + 对账；SSE 续传用 Last-Event-ID 而非新 key（见 SSE 流式端到端）。

令牌

CorpAssist S2 · 改地址重复提交（30 秒） 用户在弱网下双击「确认改地址」。两次 POST 携带相同 Idempotency-Key: idem-7c3f。BFF 第一次进入 Processing 调 ERP；第二次命中 token store 返回相同 200 与 orderSnapshot——ERP 只改一次。若第二次无 key，可能产生两次改址与双倍审计日志。

客户端或 BFF 为每次写操作生成 UUID，放入 Header Idempotency-Key。服务端在 Redis/DB token store 记录「处理中 / 已完成 + 响应快照」。TTL 建议 24–72h，覆盖移动端重试窗口。

幂等表状态机

Idempotency-Key 生命周期
stateDiagram-v2
  [*] --> Received
  Received --> Processing: 首次请求 SETNX lock
  Received --> Replay: 重复 key 且 Completed
  Processing --> Completed: 业务成功 存快照
  Processing --> Failed: 可重试失败 释放 lock
  Replay --> [*]: 返回原 HTTP 响应
  Completed --> [*]

Processing 中重复请求可返回 409 Conflict 或 202 Accepted，避免并发双写。

BFF 幂等拦截器（Java）

java 
public ResponseEntity<?> handle(String tenantId, String idempotencyKey, Supplier<Result> action) {
  if (idempotencyKey == null || idempotencyKey.isBlank()) {
    return ResponseEntity.badRequest().body("Idempotency-Key required");
  }
  String storeKey = "idem:" + tenantId + ":" + idempotencyKey;
  Optional<CachedResponse> cached = store.findCompleted(storeKey);
  if (cached.isPresent()) return cached.get().toResponse();  // Replay

  if (!store.tryLock(storeKey, Duration.ofMinutes(5))) {
    return ResponseEntity.status(HttpStatus.CONFLICT).build();
  }
  try {
    Result r = action.get();
    store.saveCompleted(storeKey, r, Duration.ofHours(48));
    return ResponseEntity.ok(r);
  } catch (Exception e) {
    store.releaseLock(storeKey);
    throw e;
  }
}

    except RetryableError:
        await redis.delete(dedup_key)
        raise  # requeue with backoff
    except PermanentError:
        await dlq.publish(msg)

重复请求

429 重试决策（标签页）

可退避重试；SSE 用 Last-Event-ID，不重复计费（补帧或 replay_done）。

必须 Idempotency-Key；无 key 时客户端应提示用户而非 SDK 自动重试。429 时读 Retry-After。

常见误区
  • 「LLM 输出随机，无法幂等」 — 弱幂等指「相同 key 返回同一 cached 结果」，不是要求模型 deterministic。
  • 幂等只做 Gateway — 写路径在 BFF/ERP；Gateway 无法感知业务是否已执行。
  • 去重键永不过期 — 与 session/idempotency 一样需要 TTL。

常见问题（FAQ）

Idempotency-Key 由客户端还是 BFF 生成？

写操作推荐客户端生成 UUID（双击同 key）；BFF 可为无 SDK 的 Web 代生成并随响应回传。tenantId 来自 JWT，不应由客户端在 Body 指定。

能否追求全链路 exactly-once？

跨 HTTP + MQ + ERP 的端到端 exactly-once 成本极高。CorpAssist 实践：at-least-once + 幂等键 + 对账；财务类写操作另加强一致与 Saga（见 advanced 集成章）。

Resilience4j

CorpAssist S2 · 大促咨询洪峰（45 秒） 09:00 活动开始，/v1/chat QPS 从 200 飙到 1800。Python RAG 向量检索 P99 从 800ms 升到 6s；DashScope 返回 429。BFF 若对每个慢请求阻塞 120s，Tomcat 200 线程 30 秒即耗尽——必须在调用边界用 Resilience4j 熔断 + 舱壁 + 超时隔离。

Spring Cloud 2024+ 推荐 Resilience4j 替代 Hystrix。CorpAssist 双栈至少包两层下游：

  • pythonRag — BFF 调 FastAPI /v1/rag/query，失败率与慢调用影响检索链路。
  • llmProvider — BFF / Spring AI 调 DashScope 等模型 API，429/5xx 须独立熔断，不与 RAG 共用计数窗口。
  • erpLookup（可选第三实例）— NS2 查订单工具 Feign 调用，与 LLM bulkhead 隔离。
CircuitBreaker

统计滑动窗口内失败率；超 failure threshold 则 Open，直接拒绝调用。

Bulkhead

限制并发槽位；LLM 与 ERP 分池，防止一类调用饿死另一类。

TimeLimiter

与 timeout isolate 配合：超时算失败，计入熔断窗口。

CorpAssist 熔断 + 舱壁配置（YAML）

  circuitbreaker:
    instances:
      pythonRag:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 20          # S2：窗口不宜过大，否则 Open 太慢
        failureRateThreshold: 50       # 50% 失败率触发 Open

          - org.springframework.web.client.HttpServerErrorException

熔断状态机

Closed → Open → HalfOpen
stateDiagram-v2
  [*] --> Closed
  Closed --> Open: 失败率超 failureRateThreshold

S2 高峰参数：反例 vs 推荐

不推荐
  • 全下游共用一个 default 熔断实例
  • failureRateThreshold: 90 + 窗口 200，Open 前已拖垮线程池
  • 无 TimeLimiter，仅依赖 Feign readTimeout 120s
  • 熔断 Open 仍抛未捕获异常到 SSE 客户端
推荐
  • RAG / LLM / ERP 分实例，指标独立暴露到 Actuator
  • 结合压测设 slowCallDurationThreshold，慢调用计入失败
  • bulkhead maxWaitDuration 短于用户可感知等待
  • Open 走 fallbackMethod，SSE 发送 terminal event 后关闭

public CompletableFuture<RagResponse> queryRagAsync(RagRequest req) {

private CompletableFuture<RagResponse> ragFallback(RagRequest req, Throwable t) {
  log.warn("rag degraded tenant={} trace={}", req.tenantId(), MDC.get("traceId"), t);
  return CompletableFuture.completedFuture(
    RagResponse.degraded(DefaultReplies.RAG_BUSY, /* handoffTicket */ true));

先想 10 秒：为什么 LLM 与 ERP 不能共用同一个 bulkhead？

大促时 LLM 长连接占满槽位，NS2 查订单工具会被饿死——用户问「订单到哪了」也超时。舱壁本质是故障域隔离，与 Gateway 路由分路径 同一思路。

默认回复

Default reply 是熔断 / 舱壁 / 超时后的用户契约，不是日志里的异常栈。CorpAssist S2 要求：

  • 可读中文 — 说明「当前咨询较多」，避免技术错误码。
  • 可行动 — 提供转人工、工单号或预计回电时间。
  • 可追踪 — 响应体含 traceId，客服系统可关联全链路。
  • SSE 友好 — 流式场景发送 event: error 或 event: done 后正常关闭，防客户端挂死。

模型 fallback

Model fallback 是 degrade policy 在模型层的具体实现：主模型不可用时，按预定路由降级，并缩短 prompt 以控制成本（NS3）。

timeout isolate 与慢调用

LLM 流式场景下，「首 token 90s、后续 stall 无数据」也应由 TimeLimiter 或 idle timeout 切断，并计入慢调用统计——否则 failure threshold 永远达不到，线程却被挂住。BFF 侧建议：首 token 超时 90s，chunk 间隔超时 15s（详见 SSE 流式端到端）。

熔断 Open 后用户反复点「重试」会怎样？
若 App 直连 BFF 重试，Open 期间每次仍快速 fallback，不会打满下游——这是熔断的价值。但若 Gateway 未限流，重试风暴仍会占满 BFF 连接；须配合 租户 QPS 配额 与客户端指数退避。
HalfOpen 探测失败立刻再 Open，会不会「永远恢复不了」？
会，若下游真未修复。此时应依赖告警与 runbook 修下游，而不是调大 threshold 掩盖问题。可临时手动切 L3 FAQ 保体验，但须记录变更。

  1. 找错：熔断 Open 后直接返回 HTTP 500，响应 body 为空。违反本章哪条原则？应如何改？

  2. 排障：S2 高峰 RAG P99 6s，但 circuitbreaker 一直 Closed，BFF 线程仍耗尽。最可能漏配了哪两项？

    参考答案

    ① slowCallRateThreshold / slowCallDurationThreshold — 慢调用未计入失败；② bulkhead 或 TimeLimiter — 无并发/超时隔离，线程被 120s 读超时占满。仅调低 failureRateThreshold 不够。

  3. 权衡：HalfOpen 期间把 permittedNumberOfCallsInHalfOpenState 从 5 调到 50，能加快恢复验证，但有何风险？

    参考答案

    下游仍不稳定时，50 路探测同时涌入可能再次打满 Python/向量库，导致刚恢复又崩溃（flapping）。S2 建议保持小批量探测，结合告警人工确认后再放大流量。

租户配额

CorpAssist S2 · 三租户同一集群（40 秒） 租户 A（标准企业）大促客服；租户 B（免费试用）被误配同一 Gateway 路由；租户 C（内部压测）脚本未停。若无 per-tenant tenant quota，B 的 2 QPS cap 无法阻止 C 占满模型 RPM，A 的真实用户反而被厂商 429——须全链路按 tenantId 计量。

配额是商业契约 + 稳定性的双重边界。CorpAssist 推荐三维度：

  • 业务 QPS — BFF 对 /v1/chat、/v1/rag 二次限流，防单连接 SSE 多路复用穿透。
  • 并发 SSE — 长连接占 Tomcat 线程与网关连接表；须单独 cap。

async def record_usage(tenant_id: str, usage: TokenUsage) -> None:
    key = f"quota:{tenant_id}:{month_key()}"  # 如 quota:acme-corp:202605
    pipe = redis.pipeline()
    pipe.hincrby(key, "tokens", usage.total)
    pipe.hincrby(key, "requests", 1)
    pipe.expire(key, 60 * 60 * 24 * 40)  # 跨月保留 40 天供对账
    await pipe.execute()

async def pre_check_budget(tenant_id: str, estimated: int, cap: int) -> bool:
    current = int(await redis.hget(f"quota:{tenant_id}:{month_key()}", "tokens") or 0)
    return current + estimated <= cap

常见误区
  • 只告警不阻断 — 100% 后仍放行会导致 NS3 账单失控；须 hard limit。
  • 429 无 Retry-After — 客户端无法退避，形成 retry 风暴加剧限流。
  • 内部压测账号无独立 tenantId — 与生产租户共用配额，误伤 SLA。
QPS 未超限但月账单爆掉，最先查哪？
查 token budget meter 是否覆盖全部模型调用路径（含 Python 直连、批处理 embed、重试重复计费）。低频超长 RAG 是典型漏计场景。
fair usage 与大客 burst 如何平衡？
大客可协商更高 qps cap 与 burstCapacity，但不得占用共享模型 RPM 的全局上限；必要时专属模型 endpoint（见 advanced 配置）。

  • Token 预算：预检 + 调用后 meter 累加；Redis 热路径，日批对账厂商账单。
  • 告警：80/100/120 分级；quota exceeded 返回可行动 429 + upgradeUrl。

@Component
public class TenantContextFilter implements Filter {
  @Override
  public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
      throws IOException, ServletException {
    String tenantId = ((HttpServletRequest) req).getHeader("X-Tenant-Id");
    if (tenantId == null || tenantId.isBlank()) {
      throw new UnauthorizedException("missing tenant");

    TenantContext.set(tenantId);

    } finally {

    }
  }

class RagServicer(rag_pb2_grpc.RagServiceServicer):

不推荐

何时用 gRPC：when grpc 决策

rest vs grpc 不是信仰之争。CorpAssist 默认 BFF→Python 用 OpenAPI REST；当出现下列信号时再引入 gRPC：

信号倾向 gRPC保持 REST
内网 P99 延迟 > 预算，HTTP 连接数高 HTTP/2 多路复用、二进制 payload —
需要 server/bidi streaming 传 token StreamChat HTTP chunked 也可但 BFF 代码更繁
第三方 / 浏览器直接调用 — REST + OpenAPI 文档
团队无 proto 治理经验 — 先 REST，度量后再迁移
批量索引、削峰 — MQ 异步 更合适

读结果：该担心什么

异常信号
  • P95 陡升但 CPU 低 → 下游 Python/模型排队
  • 429 激增 → Gateway 限流过严或 key 错误
  • 503 + 熔断日志 → 需调 Resilience4j 阈值
  • SSE 大量断连 → timeout 或 LB 空闲断开
健康基线
  • 错误率 < 1%，无持续 503
  • 限流 429 仅在超过 replenishRate 时出现
  • Trace 中 Gateway→BFF→Python 分段延迟可解释
  • 压测后 MQ 无 DLQ 堆积
常见误区
  • 直压 Python 当全链路 — 绕过 Gateway 限流/鉴权，结论不可上线。
  • 不设 threshold — k6 跑完无 PASS/FAIL，无法回归对比。
  • 忽略 SSE — 只压 REST 会漏掉长连接与缓冲问题。
