为什么要学本章

Python 改了响应字段名，Spring 解析失败——这是双栈集成中最常见的故障模式。契约测试（contract test）在 CI 中自动检测这类破坏性变更，在合并 PR 之前就发现问题。本章使用 Pact 和 Postman 两种工具，覆盖消费者驱动契约测试和 Schema 验证。

契约测试概述

contract test 验证双栈之间的 API 契约是否保持一致。在 CorpAssist 双栈中，Spring BFF 是 Python RAG API 的消费者，Python RAG API 是提供者。

消费者（Spring）定义期望——提供者（Python）验证实现，Pact Broker 作为契约管理中心。

consumer driven 契约的核心思想：消费者定义它期望的接口，提供者验证自己的实现满足所有消费者的期望。

CorpAssist · 契约漂移场景 Python 团队将 RagQueryResponse 中的 sources 字段类型从 array 改为 object，但没有通知 Spring 团队。Spring BFF 的反序列化失败，用户看到 500 错误。契约测试在 CI 阶段就发现了这个 breaking change，阻止了合并。

Pact 消费者端：Spring 生成契约

pact 在 Spring 端（消费者）定义对 Python API 的调用期望，生成 Pact 文件。

Spring Pact 消费者测试

java
// Spring Pact 消费者测试
@ExtendWith(PactConsumerTestExt.class)
@PactTestFor(providerName = "python-ai", port = "8001")
public class RagQueryConsumerPactTest {

    @Pact(consumer = "spring-bff")
    public V4Pact createRagQueryPact(PactDslWithProvider builder) {
        return builder
            .given("RAG 查询正常")
            .uponReceiving("RAG 查询请求")
                .path("/v1/rag/query")
                .method("POST")
                .headers("X-User-Id", "user-001",
                         "X-Tenant-Id", "corp-001")
                .body(new PactDslJsonBody()
                    .stringType("query", "报销流程是什么")
                    .integerType("top_k", 5))
            .willRespondWith()
                .status(200)
                .headers("Content-Type", "application/json")
                .body(new PactDslJsonBody()
                    .stringType("answer", "报销流程如下：...")
                    .minArrayLike("sources", 1, new PactDslJsonBody()
                        .stringType("title", "报销制度.md")
                        .stringType("url", "https://wiki/doc/1")
                        .numberType("score", 0.95)))
            .toPact(V4Pact.class);
    }

    @Test
    @PactTestFor(pactMethod = "createRagQueryPact")
    public void testRagQuery_whenValidRequest_shouldReturnExpectedResponse(MockServer mockServer) {
        // 使用 Mock Server 验证消费者逻辑
        RagQueryClient client = new RagQueryClient(mockServer.getUrl());
        RagQueryResponse response = client.query(
            new RagQueryRequest("报销流程是什么", 5),
            "user-001", "corp-001");

        assertThat(response.answer()).isNotEmpty();
        assertThat(response.sources()).hasSizeGreaterThanOrEqualTo(1);
    }
}

Pact 提供者端：Python 验证契约

Python 端（提供者）从 Pact Broker 拉取消费者定义的契约，验证自己的 API 是否满足所有期望。

python
# Python Pact 提供者验证
from pact import Verifier
import pytest

def test_rag_query_pact():
    """验证 Python API 满足 Spring 消费者契约"""
    verifier = Verifier(
        provider="python-ai",
        provider_base_url="http://localhost:8001",
    )

    # 配置提供者状态
    def provider_state_handler(state: str):
        if state == "RAG 查询正常":
            # 设置测试数据
            pass

    success, logs = verifier.verify_with_broker(
        broker_url="https://pact-broker.corpassist.com",
        broker_token="xxx",
        publish_version="1.0.0",
        publish_verification_results=True,
        provider_states_setup=provider_state_handler,
    )

    assert success, "Pact 验证失败，API 不满足消费者契约"

Postman CI：Schema 验证

postman ci 使用 Newman 在 CI 中运行 Postman 集合，验证 API 响应结构是否符合 OpenAPI Schema。

Postman Collection 核心测试

javascript
// Postman Tests - RAG Query Schema 验证
pm.test("状态码为 200", () => {
    pm.response.to.have.status(200);
});

pm.test("响应包含 answer 字段", () => {
    const jsonData = pm.response.json();
    pm.expect(jsonData).to.have.property("answer");
    pm.expect(jsonData.answer).to.be.a("string");
});

pm.test("响应包含 sources 字段", () => {
    const jsonData = pm.response.json();
    pm.expect(jsonData).to.have.property("sources");
    pm.expect(jsonData.sources).to.be.an("array");
});

pm.test("sources 中每项都有 title 和 url", () => {
    const jsonData = pm.response.json();
    jsonData.sources.forEach(source => {
        pm.expect(source).to.have.property("title");
        pm.expect(source).to.have.property("url");
    });
});

pm.test("响应时间小于 5 秒", () => {
    pm.expect(pm.response.responseTime).to.be.below(5000);
});

CI 配置：Newman + GitHub Actions

yaml
# .github/workflows/contract-test.yml
name: Contract Test
on:
  pull_request:
    paths:
      - "openapi/**"
      - "python-services/**"
      - "java-services/spring-bff/**"

jobs:
  contract-test:
    runs-on: ubuntu-latest
    services:
      docker:
        image: docker:24
    steps:
      - uses: actions/checkout@v4

      # 1. 启动 Python AI 服务
      - name: Start Python AI
        run: |
          docker compose -f docker-compose.ci.yml up -d python-rag
          sleep 10

      # 2. OpenAPI Schema 验证
      - name: Validate OpenAPI Schema
        uses: ibm/openapi-validator-action@v2
        with:
          file: openapi/corpassist-ai.yaml

      # 3. Newman 集成测试
      - name: Run Newman Tests
        uses: actions/setup-node@v4
      - run: |
          npm install -g newman
          newman run postman/corpassist-ai.postman_collection.json \
            --env-var base_url=http://localhost:8001 \
            --reporters cli,junit \
            --reporter-junit-export results/newman-report.xml

      # 4. Pact 验证
      - name: Pact Provider Verification
        run: |
          cd python-services
          pip install -r requirements.txt
          python -m pytest tests/pact/ -v

      # 5. Breaking Change 检测
      - name: OpenAPI Diff
        uses: ./.github/actions/openapi-diff
        with:
          base: main
          head: ${{ github.head_ref }}

Breaking Change 检测

breaking detection 在 CI 中自动检测 OpenAPI 变更是否是破坏性的。

不推荐
  • 改了 API 不通知下游团队
  • 没有自动化契约测试
  • 线上才发现不兼容
  • 手动验证, 靠「口口相传」
推荐
  • CI 自动检测 breaking change
  • Pact 消费者驱动验证
  • Postman Schema 校验
  • CI Gate 阻断破坏性合并
变更类型是否 BreakingCI 处理
新增可选字段否通过
删除字段是阻断
字段类型变更（string→object）是阻断
新增 required 字段是阻断
修改端点路径是阻断
新增端点否通过
修改描述/注释否通过
常见误区
  • 契约只在联调时验证：应该在 CI 中每次 PR 都验证，而不是开发完成后才发现不兼容
  • 只测 200 成功路径：错误响应（400/401/429/5XX）也应该有契约验证
  • 忽略 Header 约束：X-Tenant-Id 等 Header 的缺失/错误也应该在契约测试中覆盖
  • 不测试 Python 的健壮性：Python 端收到不符合契约的请求时应该返回 400 而非 500

常见问题（FAQ）

Pact 和 openapi-diff 应该都用吗？还是二选一？
建议两者都用。openapi-diff 验证 OpenAPI 规范层面的 breaking change（静态分析），Pact 验证运行时的一致性（消费者实际调用的接口是否满足期望）。两者互补：openapi-diff 发现 Schema 问题，Pact 发现交互协议问题。
Pact 测试运行慢，可以只在新版本发布时跑吗？
不建议。Pact 测试应作为 CI 的一部分在每次 PR 中运行。如果觉得慢，可以：1) 只运行受影响的 Pact 文件（使用 pact-mock-service 的隔离模式）；2) 将 Pact 验证与单元测试并行；3) 使用 Pact 的 pending 标记（允许未验证的契约进入生产，但标记为待处理）。
Python 的 FastAPI 怎么自动验证请求/响应符合 OpenAPI Schema？
FastAPI 内置了 Pydantic 模型验证，但这只验证 Python 端的请求解析。对于完整的契约验证，建议：1) 使用 openapi-schema-validator 库在测试中验证响应；2) 使用 Pact 提供端测试从消费者角度验证；3) 在 CI 中用 Newman 端到端验证实际 HTTP 响应。
契约测试失败后应该阻断合并还是仅告警？
对于 breaking change（字段删除、类型变更），应该阻断合并（CI gate）。对于非破坏性变更（新增可选字段），可以通过但需要在 PR 中标注。CorpAssist 的策略是：breaking→阻断，非 breaking→通过但需要团队成员 review。

本章结论

  • 消费者驱动：Spring（消费者）定义契约，Python（提供者）验证实现
  • 双工具策略：Pact 做消费者驱动契约测试，Postman/Newman 做 Schema 验证
  • CI 自动化：每次 PR 自动运行契约测试 + breaking change 检测，失败则阻断
  • 多维度验证：状态码、响应体、Header、错误响应都纳入契约测试范围

判断练习

  1. 你发现 Pact 测试通过了但线上还是报错了，可能是什么原因？

    参考答案

    可能的根本原因：1) Pact 只覆盖了部分场景（如只测了 200 成功路径），未覆盖错误场景（400/500）；2) Pact Mock Server 与实际 Python 服务的行为有差异；3) Header 传递逻辑未在测试中覆盖；4) 数据库/向量库中的真实数据结构与测试数据不同。
