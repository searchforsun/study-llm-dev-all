面试时在白板上画这张图：三层（客户端/Spring/Python）+ 两栈（治理/能力）+ 中间件。

不推荐：假大空
  • 只画个云朵图标，标注「AI 服务」
  • Spring 和 Python 混在一起，不分层
  • 没有中间件和数据存储
  • 不标注端口和通信协议
推荐：有细节
  • 画出 6 个以上的服务节点
  • 明确标注 Spring/Python 的边界
  • 标出通信协议（HTTP/SSE/MQ）
  • 标注关键端口号

追问应答示例

面试官追问考察点应答框架
「为什么要拆双栈？」 split rationale 1) 技术生态差异（Python 的 AI 生态 vs Java 的治理生态）
2) 团队技能（Java 平台团队 + Python AI 团队）
3) 独立演进（AI 快速迭代，治理稳定不变）
「Python 也做 Gateway 不行吗？」 职责边界 1) Spring Gateway 的过滤器链/限流/熔断生态更成熟
2) 单一治理面避免路由规则分散
3) 企业级安全审计需求（Python 缺少 AOP 能力）
「Python 服务挂了怎么办？」 容错设计 1) BFF 层降级（返回缓存或友好提示）
2) 熔断器（Resilience4j）
3) 多副本部署 + 健康检查
「怎么保证接口一致？」 契约管理 1) OpenAPI 3.1 作为唯一真相源
2) 契约优先开发流程
3) CI 中 Pact 测试 + breaking change 检测
「灰度发布怎么做？」 发布策略 1) Feature Flag 按租户/百分比控制
2) Gateway 权重路由
3) 每阶段指标监控 + 自动回滚
「跨栈排障怎么搞？」 可观测性 1) trace_id 全链路透传
2) 统一 JSON 日志格式
3) 日志按 trace_id 关联查询

权衡答题框架

tradeoff qa 是面试中最能拉开差距的部分。回答时要展示你对取舍的理解，而不是追求「完美方案」。

Tradeoff 应答三段式

  1. 列出方案：至少有 2 个可选方案（如全 Python vs 全 Java vs 双栈）
  2. 对比优劣：每个方案的延迟、成本、灵活度、可维护性
  3. 给出选择：基于场景给出推荐并说明理由
权衡维度方案 A方案 BCorpAssist 选择
技术栈 全 Python（FastAPI + LangChain） 全 Java（Spring AI + Boot） 双栈：Java 治理 + Python AI
通信方式 gRPC（高性能） HTTP REST（简单通用） HTTP + SSE（开箱即用，网关友好）
索引策略 实时索引（索引即写入） 批量索引（定时重建） 批量 + 版本化切换
会话存储 LangGraph Checkpointer Redis 集中存储 BFF 用 Redis + LangGraph Memory
部署方式 Kubernetes Docker Compose 开发：Compose；生产：K8s

常见问题（FAQ）

面试官问「你做过最复杂的系统是什么」，怎么回答？
推荐用 CorpAssist 双栈作为案例。回答模板：「我设计并实现了 CorpAssist 双栈架构，这是一个企业级 LLM 助手平台，采用 Spring（治理面）+ Python（能力面）双技术栈。我负责架构设计、OpenAPI 契约定义、双栈联调排障……」。重点突出你面临的挑战（双栈通信、契约同步、灰度发布）和解决方案。
面试官说「为什么不直接用 LangChain 全家桶？」
回答思路：1) LangChain 解决的是 AI 编排问题，不解决企业治理问题（鉴权、审计、限流）；2) 企业现有 Java 基础设施（Spring Cloud、Gateway、配置中心）不能扔掉；3) LangChain 的 Python 生态与 Spring 治理各司其职，不是替代关系而是互补。核心观点：没有银弹，双栈是当前企业级 LLM 应用的最优解。
面试官问「如果让你一个人做这个系统，你怎么做？」
真实回答：1) 个人做全栈优先选 Python（FastAPI + LangChain），减少 Java 负担；2) 用单服务 + 内部模块替代微服务架构；3) 等系统验证成功后再考虑拆分为双栈。关键要展示你知道什么时候该简化，而不是一直坚持复杂架构。
面试官问「怎么做容量估算和压力测试？」
回答框架：1) 估算 QPS —— 从业务数据推导（如 1000 企业 × 100 人 × 10 次/天 = 1M 次/天 ≈ 12 QPS）；2) 瓶颈分析 —— LLM API 调用（每个请求 2-5s，每秒最多 20 个并发请求）；3) 扩容策略 —— Python 水平扩展 + 限流保护 LLM API；4) 压测工具 —— Locust / JMeter 模拟真实用户请求。

判断练习

  1. 面试官说「你的方案太复杂了，为什么不在 Spring 里直接用 RestTemplate 调用 OpenAI API？」如何回答？

    参考答案

    这是一个很好的反问点。回答：1) 如果只有简单问答，确实不需要双栈，Spring 直接调 OpenAI API 就够了；2) 当系统加入 RAG（需要向量检索+文档解析）、Agent（需要多步推理+工具）、多租户隔离后，单一 Spring 服务会变得臃肿且 AI 能力受限；3) 双栈的复杂度来自业务需求，我们选择用架构复杂度换取业务扩展性。核心：双栈不是炫技，是业务驱动的选择。
