统一鉴权架构：Spring Gateway 集中式认证

双栈鉴权的核心设计原则：认证在 Gateway，授权在服务。Spring Cloud Gateway 作为统一入口，负责所有用户的认证，Python 服务不再需要关心用户是谁、令牌是否有效——它只需要相信 Gateway 注入的 Header。

统一鉴权流程
sequenceDiagram
    participant U as 用户
    participant GW as Spring Gateway
    participant AUTH as Auth Service\n(Keycloak)
    participant BFF as Spring BFF
    participant PY as Python AI 服务

    U->>GW: POST /api/v1/rag/query\nAuthorization: Bearer <jwt>
    GW->>GW: 1. 提取 JWT
    GW->>AUTH: 2. 校验 JWT 签名\n(JWK 端点)
    AUTH-->>GW: 3. 用户身份 + claims
    GW->>GW: 4. 注入可信 Header
    GW->>BFF: 5. 转发请求\nX-User-Id, X-Tenant-Id\nX-User-Role, X-Request-Id
    BFF->>BFF: 6. 校验 Header 完整性
    BFF->>PY: 7. 调用 Python\n（携带可信 Header）
    PY->>PY: 8. 按 X-Tenant-Id 隔离检索\n仅需信任 Header

    Note over GW,PY: Python 不需要验 JWT，相信 Gateway 注入的 Header

统一鉴权流程：Gateway 负责 JWT 校验，Python 服务只接收可信 Header。

Spring Gateway JWT 校验过滤器

java
// Spring Cloud Gateway - JWT 校验与 Header 注入
@Component
public class JwtAuthFilter implements GlobalFilter, Ordered {

    private final ReactiveJwtDecoder jwtDecoder;

    public JwtAuthFilter(ReactiveJwtDecoder jwtDecoder) {
        this.jwtDecoder = jwtDecoder;
    }

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String authHeader = exchange.getRequest()
            .getHeaders().getFirst(HttpHeaders.AUTHORIZATION);

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        String token = authHeader.substring(7);
        return jwtDecoder.decode(token)
            .flatMap(jwt -> {
                // 注入可信 Header
                ServerHttpRequest modifiedRequest = exchange.getRequest().mutate()
                    .header("X-User-Id", jwt.getClaimAsString("sub"))
                    .header("X-Tenant-Id", jwt.getClaimAsString("tenant_id"))
                    .header("X-User-Role", jwt.getClaimAsString("role"))
                    .header("X-Request-Id", UUID.randomUUID().toString())
                    .build();
                ServerWebExchange modifiedExchange = exchange.mutate()
                    .request(modifiedRequest).build();
                return chain.filter(modifiedExchange);
            })
            .onErrorResume(e -> {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            });
    }

    @Override
    public int getOrder() {
        return -100; // 高优先级
    }
}

from fastapi import FastAPI, Depends, HTTPException, Header
from pydantic import BaseModel
from typing import Optional

app = FastAPI(title="CorpAssist Python AI")

    # 使用 ctx.request_id 记录日志
    return await perform_rag_query(request, ctx)

JWT 传递与 Header 规范

jwt forward 指 Gateway 校验 JWT 后，将用户信息通过自定义 Header 传递到下游服务。Header 设计规范：

Header类型来源说明
X-User-Id String JWT sub 用户的唯一标识（如 1001）
X-Tenant-Id String JWT tenant_id 企业租户标识（如 corp-001）
X-User-Role String JWT role 角色：admin/user/auditor
X-Request-Id UUID Gateway 生成 全链路追踪 ID
X-User-Name String JWT name 用户显示名（可选）
X-User-Dept String JWT dept 用户部门（用于 ACL 过滤）
安全提示

Gateway 必须在注入可信 Header 之前清除客户端传入的同名 Header，防止用户伪造。Spring Gateway 的 RemoveRequestHeader 过滤器可用于此目的。

Gateway 清除恶意 Header 配置

yaml
# Spring Cloud Gateway 配置
spring:
  cloud:
    gateway:
      routes:
        - id: corpassist-bff
          uri: http://spring-bff:8081
          predicates:
            - Path=/api/v1/corpassist/**
          filters:
            # 清除客户端传入的可信 Header（防伪造）
            - RemoveRequestHeader=X-User-Id
            - RemoveRequestHeader=X-Tenant-Id
            - RemoveRequestHeader=X-User-Role
            - RemoveRequestHeader=X-Request-Id
            # JWT 校验过滤器（自定义）
            - name: JwtAuthFilter

服务间 M2M 认证

m2m auth 用于 Spring BFF 调用 Python 服务时的服务间认证。虽然 Python 服务部署在内网，但仍然建议添加一层轻量认证：

  • 内部 JWT（internal token)：BFF 使用自己的客户端凭证向 Auth Service 获取内部令牌
  • mTLS：服务间通信使用双向 TLS 证书验证（服务网格中常见）
  • 内网 IP 白名单：Python 服务只允许 Spring BFF 的 Pod IP 调用

Spring BFF M2M 认证

java
// Spring BFF - 服务间 M2M 认证
@Service
public class PythonServiceClient {

    private final WebClient pythonClient;
    private String cachedInternalToken;
    private Instant tokenExpiry;

    public Mono<RagQueryResponse> queryRag(RagQueryRequest request,
                                             String userId, String tenantId) {
        return getInternalToken()
            .flatMap(token -> pythonClient.post()
                .uri("/v1/rag/query")
                .header("Authorization", "Bearer " + token)
                .header("X-User-Id", userId)
                .header("X-Tenant-Id", tenantId)
                .bodyValue(request)
                .retrieve()
                .bodyToMono(RagQueryResponse.class));
    }

    private Mono<String> getInternalToken() {
        if (cachedInternalToken != null
            && Instant.now().isBefore(tokenExpiry.minusSeconds(60))) {
            return Mono.just(cachedInternalToken);
        }
        // 使用客户端凭证模式获取内部令牌
        return authClient.getToken(
            "corpassist-bff", "client-secret",
            "corpassist-python-api")
            .map(response -> {
                cachedInternalToken = response.accessToken();
                tokenExpiry = Instant.now()
                    .plusSeconds(response.expiresIn());
                return cachedInternalToken;
            });
    }
}

Python 验证内部 JWT

python
# Python - 验证内部 M2M JWT
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
import jwt

security_scheme = HTTPBearer(auto_error=False)

# 内部 JWT 公钥（从 Auth Service 同步）
INTERNAL_JWKS = {
    "keys": [
        # ... 内部服务的 RSA 公钥
    ]
}

async def verify_internal_token(
    credentials: HTTPAuthorizationCredentials = Depends(security_scheme),
) -> str:
    """验证内部 M2M 令牌"""
    if credentials is None:
        # 如果没有令牌，仅依赖内网 Header（信任 Gateway）
        return "anonymous"

    try:
        payload = jwt.decode(
            credentials.credentials,
            INTERNAL_JWKS,
            algorithms=["RS256"],
            audience="corpassist-python-api",
        )
        return payload.get("sub", "unknown")
    except jwt.PyJWTError:
        # M2M 验证失败时，仍允许内网请求通过
        # 但需要记录告警
        logger.warning("Internal token verification failed")
        return "untrusted"

多租户隔离

tenant isolation 是 CorpAssist 企业场景中的核心需求。不同企业（租户）的数据必须严格隔离。

隔离层级实现方式隔离强度
向量库 每个租户独立的 collection/partition 强
检索过滤 检索时传入 tenant_id 作为 filter 表达式 中
会话存储 Redis key 前缀: session:{tenant_id}:{user_id} 强
审计日志 DB 表中 tenant_id 字段分区 中
LLM 调用 按租户配置不同的 model/temperature 弱
不推荐
  • 所有租户共享同一个向量 collection
  • Python 自己从 JWT 解析 tenant_id
  • 不使用 X-Tenant-Id Header
  • 租户隔离仅依赖代码逻辑，无基础设施保障
推荐
  • 每个租户独立 collection + partition
  • Python 从可信 Header 读取 tenant_id
  • BFF 校验 tenant_id 与用户是否匹配
  • 向量检索时强制加入 tenant_id 过滤

async def perform_rag_query(
    request: RagQueryRequest,
    ctx: UserContext,

    """按租户隔离执行 RAG 查询"""

    # 1. 按租户选择 vector store
    collection_name = f"corp_{ctx.tenant_id}"

    vector_store = Milvus(
        embedding_function=embeddings,
        collection_name=collection_name,
        connection_args={"host": "milvus", "port": "19530"},

    # 2. 检索时强制加入部门过滤
    retriever = vector_store.as_retriever(
        search_kwargs={
            "k": request.top_k,
            "expr": f'dept == "{ctx.dept}"' if ctx.dept else None,
        }

        llm=ChatOpenAI(model="gpt-4o", temperature=0),
        retriever=retriever,
    )

    result = await qa_chain.ainvoke({"query": request.query})
    return RagQueryResponse(answer=result["result"], sources=[])

零信任架构

zero trust 在双栈鉴权中的应用：即使 Python 服务运行在内网，也不自动信任任何请求。实施要点：

  • 请求必须包含有效令牌：内部 JWT 或 Gateway 签发的会话令牌
  • 每次都验签：不缓存认证结果（除了 M2M token 的过期时间缓存）
  • 最小权限：Python 服务只获得执行所需的最小 scope
  • 审计日志：所有跨服务调用记录请求方身份和操作

常见问题（FAQ）

Python 服务如何知道当前用户的角色和权限？
通过 Spring Gateway 注入的 X-User-Role 和 X-User-Dept Header。Python 不需要自己解析 JWT，直接从 Header 读取即可。这样做的好处是：认证逻辑只在一个地方（Gateway）实现，Python 服务更轻量。
如果 Gateway 挂了，Python 服务还能工作吗？
不能直接对外提供服务，这是设计上的。Gateway 是整个系统的唯一入口，Gateway 不可用意味着整个系统不可用。但 Python 服务可以处理 BFF 的内部请求。高可用方案：Gateway 多副本 + 健康检查 + 自动重启。
内部 JWT 和用户 JWT 有什么区别？
用户 JWT 由外部 OAuth2 提供商签发，包含用户身份信息。内部 JWT 由 Auth Service 用客户端凭证模式签发，用于服务间认证。内部 JWT 的受众（aud）限制为特定的 Python API，生命周期更短（通常 5-15 分钟）。
X-Tenant-Id 会在响应中返回给客户端吗？
不应该。X-Tenant-Id 是内部 Header，Spring BFF 在返回响应给客户端前应该删除它（RemoveResponseHeader 过滤器）。客户端不应该知道内部租户标识的具体值。

本章结论

  • 集中认证：Spring Gateway 统一校验 JWT，Python 不用重复实现鉴权
  • 可信 Header：Gateway 注入 X-User-Id/X-Tenant-Id 等 Header，下游服务信任它们
  • M2M 认证：服务间使用内部 JWT 或 mTLS 增强安全
  • 租户隔离：基于 X-Tenant-Id 在向量库/Redis/数据库层面进行数据隔离

判断练习

  1. 你发现 Python 服务收到的 X-User-Id 和 JWT 中的 sub 不一致，应该怎么排查？

    参考答案

    1) 检查 Gateway 的 JWT 校验过滤器是否正确解析 sub claim；2) 检查 RemoveRequestHeader 是否清除了客户端传入的 X-User-Id；3) 检查 Gateway 日志看 JWT 解析结果；4) 检查 BFF 是否有额外的 Header 重写逻辑。

  2. 安全审计发现 X-Tenant-Id 在日志中没有脱敏，如何处理？

    参考答案

    1) 在日志配置中添加 X-Tenant-Id 的脱敏规则（如使用 Logback 的 PatternMasking）；2) 确保响应中不返回 X-Tenant-Id；3) 审计日志中记录租户信息但去标识化。
