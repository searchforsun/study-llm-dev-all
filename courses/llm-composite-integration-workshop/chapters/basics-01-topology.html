双栈拓扑全景

CorpAssist 采用双栈集成架构（dual stack topology），将系统分为两个明确的技术栈：

  • Spring 治理面（spring governance)：基于 Spring Cloud 生态，负责网关、鉴权、会话管理、服务编排、审计日志等治理能力
  • Python 能力面（python capability)：基于 FastAPI + LangChain/LangGraph 生态，负责索引构建、RAG 问答、Agent 推理、模型调用等 AI 能力

CorpAssist · 双栈交互场景 企业员工通过企业微信向 CorpAssist 提问「我的上个月报销到哪了」。请求经过：企业微信 → Spring Gateway(验签+路由) → Spring BFF(组装上下文+鉴权) → Python RAG API(向量检索+LLM 生成) → 响应经原路返回。整个过程涉及双栈 5 个服务协作。
不推荐：职责模糊
  • Python 服务自己鉴权、自己限流、自己管会话
  • Spring 服务直接调用 LLM 模型
  • 两边都能改向量库的索引数据结构
  • 没有明确的 BFF 层，客户端直连 Python
推荐：职责清晰
  • Spring 集中治理：鉴权/限流/审计/编排
  • Python 专注 AI：RAG/Agent/索引/评估
  • 向量库只读/写入分离：Python 索引写入，Spring/Python 只读查询
  • BFF 层作为双桥接点，透传可信上下文

Spring 治理面

Spring governance 层是 CorpAssist 的治理中枢，承担所有非 AI 相关的企业级能力。其核心组件包括：

组件职责技术选型
API Gateway 统一入口、路由分发、限流、IP 白名单 Spring Cloud Gateway
BFF 双栈桥接、会话管理、请求组装、响应聚合 Spring Boot WebFlux
Auth Service OAuth2 令牌校验、JWT 签发、M2M 认证 Spring Security + Keycloak
编排引擎 RAG 问答编排、Agent 状态管理、转人工 Spring AI + 状态机
审计日志 全量请求/响应记录、敏感操作追踪 Spring AOP + MongoDB

Spring 治理面的核心价值

为什么选择 Spring 承担治理层？而非用 Python 实现同样的功能？

  • 成熟的网关生态：Spring Cloud Gateway 提供完善的过滤器链、限流（RequestRateLimiter）、熔断（Resilience4j）、灰度路由
  • 标准化安全框架：Spring Security + OAuth2 + OIDC 已大规模验证
  • 企业合规要求：审计日志、RBAC、数据脱敏在 Java 生态有成熟的 AOP 实现
  • 团队舒适区：企业内部通常已有 Java 平台团队，复用治理能力

Spring BFF 核心代码片段

java
// Spring BFF - CorpAssist 请求转发核心逻辑
@RestController
@RequestMapping("/api/v1/corpassist")
public class CorpAssistController {

    private final WebClient pythonRagClient;
    private final WebClient pythonAgentClient;

    @PostMapping("/rag/query")
    public Mono<RagQueryResponse> ragQuery(
            @RequestBody @Valid RagQueryRequest request,
            @RequestHeader("X-User-Id") String userId,
            @RequestHeader("X-Tenant-Id") String tenantId) {

        // 1. 注入用户上下文到 Python 请求
        return pythonRagClient.post()
            .uri("/v1/rag/query")
            .header("X-User-Id", userId)
            .header("X-Tenant-Id", tenantId)
            .header("X-Request-Id", UUID.randomUUID().toString())
            .bodyValue(request)
            .retrieve()
            .bodyToMono(RagQueryResponse.class)
            // 2. 超时 + 降级
            .timeout(Duration.ofSeconds(30))
            .onErrorResume(ex -> Mono.just(
                new RagQueryResponse("error", "服务暂时不可用，请稍后重试")));
    }

    @PostMapping(value = "/agent/run", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> agentRun(
            @RequestBody AgentRunRequest request,
            @RequestHeader("X-User-Id") String userId) {
        // 1. 从 Redis 获取会话历史
        // 2. 转发到 Python Agent API（SSE 流式）
        // 3. 流式返回给客户端
        return pythonAgentClient.post()
            .uri("/v1/agent/run")
            .header("X-Session-Id", request.sessionId())
            .bodyValue(request)
            .retrieve()
            .bodyToFlux(new ParameterizedTypeReference<ServerSentEvent<String>>() {});
    }
}

Python 能力面

python capability 层是 CorpAssist 的AI 引擎，承载所有与模型相关的核心能力。其服务划分如下：

服务职责技术栈
RAG Query API 接收查询→向量检索→LLM 生成→返回答案 FastAPI + LangChain + Chromadb
Agent Run API 多步推理、工具调用、SSE 流式输出 FastAPI + LangGraph + httpx
Index Pipeline 文档解析→分块→向量化→写入向量库 Celery + Unstructured + Sentence-Transformers
评估服务 RAG/Agent 效果评测、回归测试 FastAPI + RAGAS + DeepEval

Python RAG Query API 核心代码

python
# Python RAG Query API - CorpAssist 问答核心
from fastapi import FastAPI, HTTPException, Header
from pydantic import BaseModel
from langchain_community.vectorstores import Milvus
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.chains import RetrievalQA

app = FastAPI(title="CorpAssist RAG API", version="1.0.0")

class RagQueryRequest(BaseModel):
    query: str
    top_k: int = 5

class RagQueryResponse(BaseModel):
    answer: str
    sources: list[str]

@app.post("/v1/rag/query")
async def rag_query(
    request: RagQueryRequest,
    x_user_id: str = Header(...),
    x_tenant_id: str = Header(...),
    x_request_id: str = Header(None),
):
    # 1. 根据租户隔离向量库
    vector_store = Milvus(
        embedding_function=OpenAIEmbeddings(),
        collection_name=f"corp_{x_tenant_id}",
        connection_args={"host": "milvus", "port": "19530"},
    )
    # 2. 构建 RAG Chain
    qa_chain = RetrievalQA.from_chain_type(
        llm=ChatOpenAI(model="gpt-4o", temperature=0),
        retriever=vector_store.as_retriever(
            search_kwargs={"k": request.top_k}
        ),
        return_source_documents=True,
    )
    # 3. 执行问答
    result = await qa_chain.ainvoke({"query": request.query})
    return RagQueryResponse(
        answer=result["result"],
        sources=[doc.metadata["source"] for doc in result["source_documents"]],
    )

data flow 描述了请求在双栈之间的完整流转路径。理解数据流是排障和优化的基础。

CorpAssist RAG 问答数据流
sequenceDiagram
    participant U as 用户
    participant GW as Spring Gateway
    participant BFF as Spring BFF
    participant AUTH as Auth Service
    participant PY as Python RAG API
    participant VDB as Vector DB
    participant LLM as LLM Model

    U->>GW: POST /api/v1/corpassist/rag/query
    GW->>AUTH: 校验 JWT 令牌
    AUTH-->>GW: 用户身份 + 租户
    GW->>BFF: 转发（注入 X-User-Id, X-Tenant-Id）
    BFF->>BFF: 组装请求上下文
    BFF->>PY: POST /v1/rag/query
    Note over PY: 注入请求头中的用户信息
    PY->>VDB: 向量检索（按租户隔离）
    VDB-->>PY: 相关文档片段
    PY->>LLM: 查询 + 文档片段
    LLM-->>PY: 生成答案
    PY-->>BFF: 200 {answer, sources}
    BFF-->>BFF: 记录审计日志
    BFF-->>GW: 响应返回
    GW-->>U: 最终响应

一次 RAG 问答经历 7 个节点，双栈之间通过 OpenAPI 契约通信。

关键数据流节点

节点输入输出关键操作
1. Gateway HTTP 原始请求 可信 Header JWT 校验、限流、路由
2. BFF 可信 Header + Body 增强请求 会话注入、权限校验、上下文组装
3. Python API 查询 + 用户上下文 检索结果 + LLM 答案 向量检索、LLM 调用
4. 响应返回 Python 响应 客户端响应 脱敏、格式转换、审计日志
常见误区
  • 双栈职责边界模糊：Spring 也写 AI 逻辑，Python 也写鉴权，导致两边代码耦合
  • BFF 过薄：直接把 Python 的响应原封不动返回，失去了治理层的价值
  • Python 直连数据库：Python 应通过 Spring BFF 或 MQ 访问业务数据库，而不是直接连
  • 忽略网络开销：每次双栈调用有 1-5ms 延迟，长链路需考虑超时和熔断

BFF 层：双栈桥接

bff layer 是 Spring 治理面中直接面向 Python 能力面的桥接层。它承担以下职责：

  • 请求翻译：将 Spring 内部上下文转换为 Python API 可识别的 HTTP Header + Body
  • 响应聚合：调用多个 Python 服务后聚合结果（如 RAG + Agent 组合）
  • 降级/熔断：Python 服务不可用时，BFF 返回缓存答案或友好提示
  • 超时控制：对 Python 的调用设置合理的超时，避免线程阻塞
不推荐：BFF 过厚
  • BFF 包含业务逻辑（如计算、状态机）
  • BFF 直接调用 LLM SDK
  • BFF 访问向量数据库做检索
  • BFF 包含复杂的缓存失效逻辑
推荐：BFF 适中
  • BFF 做请求组装/响应聚合/上下文注入
  • BFF 处理网关透传的 Header 增强
  • BFF 做会话管理和历史记录
  • BFF 处理超时/熔断/重试策略

常见问题（FAQ）

为什么不在 Python 中也实现网关功能？
理论上可行，但实践中建议单一网关层。如果 Python 也做网关，会导致路由规则分散、鉴权策略不统一、审计日志需要聚合两边。Spring Cloud Gateway 的生态成熟度（限流、熔断、过滤器链）远超 Python 网关实现。Python 应专注于 AI 能力，治理能力交给 Spring。
双栈之间是否必须走 HTTP？
不一定。非实时场景（如索引构建、批量评估）可以通过 MQ（RabbitMQ/Kafka）进行异步通信。实时问答场景（RAG/Agent）通常走 HTTP/gRPC。选择依据：实时性 vs 解耦度的权衡。详见 同步、异步与 MQ。
Python 服务如何知道当前用户的租户信息？
Spring Gateway 在 JWT 校验后会注入可信 Header（如 X-Tenant-Id、X-User-Id、X-User-Role）。Python 服务从 Header 读取即可，不需要自己验 JWT。Python 应信任这些 Header，因为上游 Gateway 已经做了鉴权。详见 统一鉴权。
双栈部署时，先部署 Spring 还是 Python？
建议先部署中间件（Redis、Milvus、PostgreSQL），再部署 Python 能力面（索引构建需要先就绪），最后部署 Spring 治理面。Spring 治理面启动时会检查依赖服务的健康状态。Docker Compose 中可以配置 depends_on + healthcheck 保证启动顺序。详见 Docker Compose 本地开发栈。

本章结论

  • 双栈分工：Spring 治理面负责网关/鉴权/编排，Python 能力面负责 RAG/Agent/索引
  • BFF 桥接：BFF 层是双栈通信的关键节点，负责上下文注入和请求转发
  • 数据流：请求经过 Gateway→BFF→Python→LLM 四跳，每一跳都有明确的职责
  • 职责清晰：避免「Spring 写 AI 逻辑，Python 写鉴权」的边界模糊

判断练习

  1. 同事提出「让 Python 也实现一套网关，减轻 Spring 压力」，你认为合理吗？理由是什么？

    参考答案

    不合理。双栈核心原则是单一治理面。多网关导致路由规则分散、鉴权策略不统一、审计日志需聚合。Python 应专注 AI 能力，治理交给 Spring。

  2. BFF 调用 Python RAG API 超时，用户看到 502。请列出至少 3 种可能原因。

    参考答案

    1) Python 服务负载过高，请求排队；2) LLM 调用超时（模型响应慢）；3) 向量数据库连接池耗尽；4) Python 与 BFF 之间的网络延迟/丢包。
