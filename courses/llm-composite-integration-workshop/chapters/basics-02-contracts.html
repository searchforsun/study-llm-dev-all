契约优先：OpenAPI 3.1 作为唯一真相源

openapi contract 是双栈集成的契约基础。Spring 治理面和 Python 能力面之间所有的 API 通信都基于同一份 OpenAPI 规范文件。

schema first 流程如下：

  1. 定义契约：团队（平台架构师）编写 OpenAPI 3.1 YAML
  2. 代码生成：Spring 端用 openapi-generator-maven-plugin 生成 Java DTO/API 接口；Python 端用 openapi-generator-cli 生成 Pydantic 模型
  3. 双栈实现：Spring 实现治理接口，Python 实现 AI 接口
  4. 契约测试：双方基于同一契约进行集成测试（参见 契约测试与集成测试）
  5. 破坏检测：CI 中运行 openapi-diff 检测 breaking change

DTO 对齐

dto align 确保双栈之间的数据传输对象在各语言中保持一致。关键原则：

  • 类型严格对应：Java String ↔ Python str，Java Integer ↔ Python int，Java BigDecimal ↔ Python Decimal
  • 命名统一：camelCase 用于 JSON 字段名，Java 和 Python 都使用 @JsonProperty/serialization_alias 映射
  • 可选性一致：required 在 OpenAPI 和 Java/Python 中必须一致，否则导致反序列化失败
  • 日期时间格式：统一使用 ISO-8601（2026-05-31T10:30:00Z），避免 LocalDateTime/Naive datetime 不匹配

Java DTO 生成示例

java
// 由 openapi-generator 自动生成，禁止手动修改
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class RagQueryRequest {
    @NotBlank
    @Size(max = 4000)
    private String query;

    @Min(1) @Max(20)
    @Builder.Default
    private Integer topK = 5;

    private Map<String, Object> filters;

    @JsonIgnore
    public boolean isValid() {
        return query != null && !query.isBlank()
            && topK != null && topK >= 1 && topK <= 20;
    }
}

@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class RagQueryResponse {
    @NotBlank
    private String answer;

    @NotEmpty
    private List<SourceRef> sources;
}

Python DTO 生成示例

python
# 由 openapi-generator 自动生成，禁止手动修改
from pydantic import BaseModel, Field, field_validator
from typing import Optional
from decimal import Decimal

class RagQueryRequest(BaseModel):
    query: str = Field(..., max_length=4000)
    top_k: int = Field(default=5, ge=1, le=20)
    filters: Optional[dict] = None

    @field_validator('query')
    @classmethod
    def query_not_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError('query must not be blank')
        return v.strip()

class SourceRef(BaseModel):
    title: Optional[str] = None
    url: Optional[str] = None
    score: Optional[float] = None

class RagQueryResponse(BaseModel):
    answer: str
    sources: list[SourceRef]

统一错误码

error code 是双栈排障的关键。当 Python 返回错误时，Spring 需要能够理解并翻译给用户。CorpAssist 采用 RFC 9457 Problem Details 规范。

错误码HTTP Status含义触发条件
INVALID_QUERY 400 查询参数非法 query 为空、top_k 超过范围
UNAUTHORIZED 401 未授权 JWT 过期或签名无效
FORBIDDEN 403 租户无权限 用户无该知识库的访问权限
RATE_LIMITED 429 频率超限 超过 API 调用配额
VECTOR_INDEX_EMPTY 422 向量索引为空 租户未上传任何文档
LLM_TIMEOUT 504 LLM 调用超时 模型响应超过 30 秒
CONTEXT_OVERFLOW 413 上下文超长 请求超过 32K token 限制
INTERNAL_ERROR 500 内部错误 未预期的服务端异常

Python 统一错误处理

python
# Python 统一错误处理中间件
from fastapi import Request, status
from fastapi.responses import JSONResponse
from traceback import format_exception

class CorpAssistError(Exception):
    def __init__(self, error_code: str, status_code: int,
                 detail: str, trace_id: str = ""):
        self.error_code = error_code
        self.status_code = status_code
        self.detail = detail
        self.trace_id = trace_id

@app.exception_handler(CorpAssistError)
async def corpassist_error_handler(request: Request, exc: CorpAssistError):
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "type": "https://api.corpassist.com/errors/" + exc.error_code,
            "title": "请求处理失败",
            "status": exc.status_code,
            "detail": exc.detail,
            "instance": str(request.url),
            "error_code": exc.error_code,
            "trace_id": exc.trace_id or request.headers.get("X-Request-Id", ""),
        },
    )

Java 统一错误处理

java
// Spring 统一错误处理
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(CorpAssistException.class)
    public ResponseEntity<ProblemDetail> handleCorpAssistException(
            CorpAssistException ex, HttpServletRequest request) {
        ProblemDetail problem = ProblemDetail.forStatus(ex.getStatus());
        problem.setType(URI.create(
            "https://api.corpassist.com/errors/" + ex.getErrorCode()));
        problem.setTitle("请求处理失败");
        problem.setDetail(ex.getDetail());
        problem.setInstance(URI.create(request.getRequestURI()));
        problem.setProperty("error_code", ex.getErrorCode());
        problem.setProperty("trace_id",
            request.getHeader("X-Request-Id"));
        return ResponseEntity.status(ex.getStatus()).body(problem);
    }
}

API 版本管理

api version 策略决定了双栈如何在升级过程中保持兼容。CorpAssist 采用路径版本 + Content-Type 版本双模式：

  • 路径版本：/v1/rag/query, /v2/rag/query —— 大版本升级时使用
  • Header 版本：Accept: application/vnd.corpassist.v2+json —— 小版本向后兼容

弃用策略

yaml
# OpenAPI 弃用标注示例
paths:
  /v1/rag/query:
    post:
      deprecated: true
      description: "已弃用，请使用 /v2/rag/query"
      x-sunset: "2026-09-30"

breaking change 包括：删除字段、修改字段类型、增加 required 字段、修改端点路径。这些变更必须走大版本更新。

不推荐：无版本管理
  • 直接修改已有 API 的响应结构
  • 删除字段不做兼容
  • 无版本号或只用 git commit 做版本
  • 双栈各自维护一份不同的契约
推荐：版本化管理
  • 向后兼容的字段只做新增不做修改
  • 弃用字段标注 deprecated + sunset 日期
  • 每次变更更新 OpenAPI 文件的 version 字段
  • 双栈共享同一份契约文件（contract repo）
常见误区
  • 契约不同步：Java 改了 DTO 但未更新 OpenAPI，Python 还在用旧字段
  • 生成代码被手动修改：openapi-generator 生成的文件被手动编辑，下次生成时覆盖
  • 版本号随意递增：微小的向后兼容变更也升大版本，导致客户端版本爆炸
  • 忽略可选字段：服务端新增可选字段，旧客户端收到未知字段时可能崩溃

常见问题（FAQ）

契约文件应该放在哪个代码仓库？
建议独立契约仓库（contract repo）或 monorepo 中的独立模块。Spring 和 Python 子模块通过 Git Submodule 或 CI 拉取同一份契约。契约仓库的写入权限应仅限架构师/平台团队，普通开发者只能提 PR。参考 团队拓扑与交付协作。
openapi-generator 生成的代码可以手动修改吗？
强烈不建议。生成的文件应标记为「auto-generated」，任何手动修改都会被下次生成覆盖。如果需要自定义逻辑，使用模板扩展（mustache 模板）或在生成的 DTO 外包装一层。CorpAssist 实践中，所有的 DTO 校验逻辑写在 Service 层而非 DTO 层。
如何检测 breaking change？
使用 openapi-diff 或 IBM OpenAPI Validator。在 CI 中对比本次 PR 的 OpenAPI 变更与基线的 diff，如果检测到 breaking change 则阻断合并。详见 契约测试。
Python 和 Java 的日期时间类型怎么对齐？
所有时间字段统一使用 ISO-8601 字符串（如 2026-05-31T10:30:00Z），在 OpenAPI 中定义为 format: date-time。Java 用 OffsetDateTime，Python 用 datetime（带 timezone）。不要在 JSON 中传输 LocalDateTime 或 Naive datetime。

判断练习

  1. 同事在 OpenAPI 中将 answer 字段类型从 string 改为 object，这算什么类型的变更？

    参考答案

    Breaking Change。字段类型修改会破坏所有客户端代码的反序列化逻辑。应使用新字段名（如 answer_v2）或升级大版本。

  2. Python 返回了 error_code: CONTEXT_OVERFLOW，Spring BFF 应该怎么处理？

    参考答案

    BFF 应将该错误码映射为用户可读的消息（如「问题太长了，请精简后重试」），并记录 trace_id 到日志。同时可以建议用户拆分问题或缩短历史上下文。
