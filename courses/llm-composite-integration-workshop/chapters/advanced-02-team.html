代码库策略：mono vs multi repo

mono repo 和 multi repo 各有优劣，CorpAssist 采用混合策略。

维度Mono RepoMulti RepoCorpAssist 混合
代码共享容易需要通过子模块/publish共享契约在 mono 目录
独立发布需要分支管理天然独立Python/Spring 各自独立仓库
CI 速度全量 CI 慢只运行本仓库 CI多 repo 分散 CI 负载
权限控制粗粒度仓库级别精细控制多 repo 支持细粒度权限
跨仓库修改单原子提交需要多 PR 协调契约变更 → 各仓库 PR + 同步
新人上手一个仓库就够了需要了解多个仓库各团队只需关注自己的仓库

CorpAssist 多仓库架构：契约仓库作为子模块挂载到 Java 和 Python 仓库中。

Git Submodule 管理

bash
# 在 Java 仓库中添加契约子模块
cd java-services
git submodule add https://github.com/corpassist/contract-repo.git openapi
git submodule update --init --recursive

# 更新子模块到最新版本
git submodule update --remote openapi

# 在 Python 仓库中添加（同上）
cd python-services
git submodule add https://github.com/corpassist/contract-repo.git openapi

# CI 中拉取子模块
git clone --recurse-submodules https://github.com/corpassist/java-services.git

契约所有权与变更流程

contract owner 是双栈协作中最关键的角色。契约的所有者是平台架构组，但任何团队都可以提 PR 修改契约。

契约变更流程
sequenceDiagram
    participant AI as AI 工程组
    participant PLAT as 平台架构组
    participant CI as CI 流水线

    AI->>PLAT: 1. PR: 修改 OpenAPI 契约\n（新增字段/端点）
    PLAT->>PLAT: 2. Review 是否 Breaking
    PLAT->>CI: 3. 运行 openapi-diff

    CI-->>PLAT: 4. Breaking? → 阻断/标记
    PLAT-->>AI: 5. 反馈 Review 结果

    alt 非 Breaking 变更
        PLAT->>CI: 6. 合并到 main
        CI->>CI: 7. 自动通知双栈团队
        CI->>Java: 8. 自动生成 Java 代码
        CI->>Python: 9. 自动生成 Python 代码
    else Breaking 变更
        PLAT->>AI: 需要升级大版本
        AI->>PLAT: 确认版本计划
        PLAT->>CI: 创建 v3 分支
    end

契约变更流程：AI 组提 PR，架构组 review，CI 自动检测 breaking change 并生成代码。

发布列车策略

release train 是双栈团队的协同发布策略。Spring 和 Python 服务虽然独立开发，但需要协同发布以保证兼容性。

  • 双周发布列车：每隔两周周三，Spring 和 Python 同时发布
  • 契约冻结窗口：发布前 2 天，契约仓库禁止修改（只修 bug）
  • 版本对齐：每次发布使用统一的版本号（如 2026.05.31-rc1）
  • 热修复：紧急问题走 Hotfix 通道，单服务发布，不要求对齐

常见问题（FAQ）

AI 工程组需要修改 OpenAPI 契约，但平台架构组回复慢怎么办？
建议设置SLA 时间：非 Breaking 变更 24 小时内 review，Breaking 变更 48 小时内 review。如果架构组长期未响应，AI 组可以先在本地 fork 契约进行开发，同时标记 PR 为「等待 review」。关键指标是「契约 review 的平均时间」，团队应持续优化这个指标。
mono repo 和 multi repo 哪个更好？
没有绝对的好坏。CorpAssist 选择 multi repo 的原因是：双栈团队独立、发布节奏不同、权限需要隔离。如果团队规模小（< 10 人）、发布节奏一致，mono repo 更简单。关键在于契约仓库必须是独立的，无论 mono 还是 multi，契约都需要被双方共享和版本管理。
如何避免「Spring 改了 BFF 但 Python 不知道」？
三种机制：1) CI 中运行 契约测试，Python 修改 API 后 Spring 的 Pact 测试会失败；2) 定期同步会议（每周 15 分钟）同步接口变更；3) 契约仓库的每次合并自动通知双方团队的 IM 群组。
平台工程组和 AI 工程组如何划分 KPI？
平台组的 KPI 偏向基础设施：服务可用性（99.9%）、部署成功率、P95 延迟。AI 组的 KPI 偏向业务：RAG 准确率、Agent 完成率、用户满意度。共享 KPI：端到端可用性、接口错误率、发布频率。避免「平台组不管业务指标，AI 组不管系统指标」的单维度考核。

本章结论

  • 团队分工：平台组（Spring 治理 + 基础设施）+ AI 组（Python 能力面），双方权责清晰
  • 代码库策略：Multi Repo + 独立契约仓库，支持独立发布和细粒度权限
  • 契约所有权：架构组拥有契约，双方提 PR 修改，CI 自动检测 breaking change
  • 发布列车：双周协同发布 + 契约冻结窗口，保障版本兼容

判断练习

  1. AI 组提出「我们要把 Python RAG 改为 gRPC 协议，不再使用 HTTP」。应该同意吗？为什么？

    参考答案

    不应该立即同意。更换通信协议属于 Breaking Change，需要评估：1) Gateway/BFF 是否支持 gRPC 代理；2) 现有 OpenAPI 契约体系如何迁移；3) 客户端是否需要更新。建议走正常契约变更流程：先讨论影响范围，制定迁移计划，灰度切换。
    注意：这不是 AI 组单方可以决定的事。
