灰度发布策略全景

gray release 是双栈架构的核心发布策略。当 Python 能力面发布新版本时，按一定规则逐步放量，监控确认稳定后再全量切换。

Feature Flag 实现

feature flag 是控制双栈功能开关的核心工具。CorpAssist 使用配置中心（Nacos/Spring Cloud Config）管理所有 feature flag。

Spring Feature Flag 配置

yaml
# application-feature-flags.yml
corpassist:
  features:
    # Python RAG 新管线
    new-python-rag-pipeline:
      enabled: true
      traffic-percentage: 20  # 20% 流量走新管线
      tenant-whitelist: []    # 空=不按租户
      fallback-behavior: "use_spring_rag"  # 回退到 Spring 旧版

    # 新版 Agent
    new-agent-langgraph-v2:
      enabled: false
      traffic-percentage: 0
      tenant-whitelist: ["corp-vip-001"]
      fallback-behavior: "return_error"

    # LLM 模型切换
    llm-model-gpt5:
      enabled: false
      model-name: gpt-5-preview
      traffic-percentage: 5
      fallback-model: gpt-4o

Spring Feature Flag 服务

java
// Spring Feature Flag 服务
@Service
public class FeatureFlagService {

    private final Map<String, FeatureFlag> flags;

    public boolean isEnabled(String featureKey) {
        FeatureFlag flag = flags.get(featureKey);
        if (flag == null) return false;
        return flag.enabled();
    }

    public boolean isInTraffic(String featureKey, String tenantId) {
        FeatureFlag flag = flags.get(featureKey);
        if (flag == null || !flag.enabled()) return false;

        // 白名单优先
        if (flag.tenantWhitelist() != null
            && flag.tenantWhitelist().contains(tenantId)) {
            return true;
        }

        // 百分比判定
        if (flag.trafficPercentage() >= 100) return true;
        int hash = Math.abs(tenantId.hashCode()) % 100;
        return hash < flag.trafficPercentage();
    }

    @RefreshScope
    @ConfigurationProperties(prefix = "corpassist.features")
    public void setFlags(Map<String, FeatureFlag> flags) {
        this.flags = flags;
    }

    public record FeatureFlag(
        boolean enabled,
        int trafficPercentage,
        List<String> tenantWhitelist,
        String fallbackBehavior
    ) {}
}

BFF 灰度路由

java
// BFF 灰度路由
@Service
public class RagRoutingService {

    private final FeatureFlagService featureFlags;
    private final PythonRagClient pythonRagClient;
    private final SpringRagService springRagService;

    public Mono<RagQueryResponse> routeQuery(
            RagQueryRequest request, String tenantId) {

        // 判断是否走新 Python 管线
        if (featureFlags.isInTraffic(
            "new-python-rag-pipeline", tenantId)) {
            return pythonRagClient.query(request, tenantId)
                .onErrorResume(ex -> {
                    log.warn("Python RAG failed, falling back to Spring RAG",
                        ex);
                    return springRagService.query(request, tenantId);
                });
        }

        // 默认走 Spring 旧版
        return springRagService.query(request, tenantId);
    }
}

Gateway 灰度路由

canary route 在 Gateway 层通过 Spring Cloud Gateway 的权重路由实现。

yaml
# Spring Cloud Gateway 灰度路由配置
spring:
  cloud:
    gateway:
      routes:
        # RAG 新版本灰度（10% 流量）
        - id: corpassist-rag-v2
          uri: http://python-rag-v2:8001
          predicates:
            - Path=/api/v1/corpassist/rag/**
            - Weight=rag-group, v2-weight: ${RAG_V2_WEIGHT:10}
          filters:
            - StripPrefix=1

        # RAG 旧版本（90% 流量）
        - id: corpassist-rag-v1
          uri: http://python-rag:8001
          predicates:
            - Path=/api/v1/corpassist/rag/**
            - Weight=rag-group, v1-weight: 100
          filters:
            - StripPrefix=1

Python 侧 Feature Flag

python
# Python Feature Flag 客户端
from redis import Redis
import json
import hashlib

redis_client = Redis(host="redis", port=6379, db=0)

class FeatureFlagClient:
    """从配置中心读取 Feature Flag"""

    def __init__(self):
        self._cache = {}

    def is_enabled(self, feature_key: str) -> bool:
        flag = self._get_flag(feature_key)
        return flag.get("enabled", False) if flag else False

    def is_in_traffic(self, feature_key: str, tenant_id: str) -> bool:
        flag = self._get_flag(feature_key)
        if not flag or not flag.get("enabled", False):
            return False

        # 白名单
        whitelist = flag.get("tenant_whitelist", [])
        if tenant_id in whitelist:
            return True

        # 百分比
        percentage = flag.get("traffic_percentage", 0)
        hash_val = int(hashlib.md5(
            tenant_id.encode()).hexdigest(), 16) % 100
        return hash_val < percentage

    def _get_flag(self, key: str) -> dict:
        if key not in self._cache:
            data = redis_client.get(f"feature_flag:{key}")
            if data:
                self._cache[key] = json.loads(data)
        return self._cache.get(key, {})

    def refresh(self):
        """刷新缓存"""
        self._cache = {}

feature_flags = FeatureFlagClient()

回滚策略

stack rollback 是灰度发布中最重要的保障机制。CorpAssist 的 rollback plan 包含三个层次：

  1. Feature Flag 回滚：修改配置中心的 flag，秒级生效，从新 Python 路由切回 Spring
  2. Gateway 路由回滚：调整权重路由，将新版流量降为 0%
  3. 服务版本回滚：回退 Python 或 Spring 服务的 Docker 镜像到上一个版本
不推荐
  • 直接改代码来关闭功能
  • 灰度比例需要重启服务才能生效
  • 没有回滚计划，出问题才想怎么办
  • 回滚后不回填结果（幂等性丢失）
推荐
  • 配置中心动态管理 Feature Flag
  • 灰度比例支持运行时动态调整
  • 每次灰度前制定回滚计划
  • 新/旧版本都支持 Idempotency-Key

回滚自动化脚本

bash
# rollback.sh - CorpAssist 回滚脚本
#!/bin/bash

# 步骤1: 关闭 Feature Flag（配置中心 API）
echo "=== 关闭 Feature Flag ==="
curl -X PUT https://config.corpassist.com/config/feature-flags \
  -H "Content-Type: application/json" \
  -d '{"new-python-rag-pipeline": {"enabled": false}}'

# 步骤2: 调整 Gateway 路由权重
echo "=== 调整路由权重 ==="
curl -X PUT https://gateway-admin.corpassist.com/routes/rag \
  -H "Content-Type: application/json" \
  -d '{"v1_weight": 100, "v2_weight": 0}'

# 步骤3: 监控确认恢复
echo "=== 验证回滚 ==="
sleep 10
curl -f http://localhost:8080/actuator/health

常见问题（FAQ）

Feature Flag 和 Gateway 权重路由有什么区别？
Feature Flag 是业务层的控制，在 BFF/Service 代码中判断，粒度更细（可按用户、租户、甚至请求内容）。Gateway 权重路由是基础设施层的控制，只按百分比分配流量。CorpAssist 同时使用两者：Gateway 做服务级别的流量切分，Feature Flag 做功能级别的灰度控制。
灰度期间如何监控新版本的健康状态？
关键指标包括：1) 错误率（4xx/5xx）—— 超过基线 1% 触发回滚；2) P95 延迟 —— 超过基线 50% 触发告警；3) 业务流程成功率 —— 比旧版本低触发回滚；4) 用户反馈 —— 灰度租户的投诉率。建议对接 APM（如 SkyWalking、Grafana）实现自动化监控。
如果新旧版本的 OpenAPI 契约不一致怎么办？
灰度期间必须保持新旧版本的契约完全兼容。如果新版本引入了 breaking change，需要升级大版本（如 v2→v3），通过 Gateway 路由到不同的前缀路径（/v2/rag/query vs /v3/rag/query）。灰度只灰度业务逻辑，不灰度接口规范。契约变更应走 OpenAPI 版本管理 流程。
如何让特定租户始终使用新版本？
两种方式：1) 在 Feature Flag 中配置 tenant_whitelist，白名单中的租户始终走新版本；2) 在 Gateway 中通过自定义谓词（如 X-Tenant-Id Header）将特定租户路由到新版。推荐方式 1，业务层控制更灵活且不依赖基础设施。

本章结论

  • 多策略灰度：Feature Flag + Traffic Split + Canary Route 组合使用
  • 动态控制：通过配置中心动态调整灰度比例，无需重启
  • 三层回滚：Flag 回滚（秒级）→ 路由回滚（分钟级）→ 镜像回滚（分钟级）
  • 监控驱动：每阶段灰度前制定指标基线，异常时自动回滚
