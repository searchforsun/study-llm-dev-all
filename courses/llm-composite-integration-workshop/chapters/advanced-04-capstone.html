演示脚本

demo script 是端到端演示的详细脚本。

Step 1: 启动环境

bash
# 启动 CorpAssist 完整开发栈
docker compose --profile full up -d

# 查看启动状态
docker compose ps
docker compose logs -f python-rag

# 验证所有服务健康
curl -s http://localhost:8080/actuator/health | jq .
curl -s http://localhost:8001/health | jq .
curl -s http://localhost:8082/actuator/health | jq .

Step 2: 索引文档

bash
# 上传文档并触发索引
curl -X POST http://localhost:8081/api/v1/admin/index \
  -H "Content-Type: multipart/form-data" \
  -H "X-Tenant-Id: corp-001" \
  -H "X-User-Id: admin-001" \
  -F "files=@docs/报销制度.pdf" \
  -F "files=@docs/休假政策.md"

# 查询索引任务状态
curl http://localhost:8081/api/v1/index/jobs/job-id-xxx | jq .

Step 3: RAG 问答

bash
# RAG 问答（经 Gateway → BFF → Python）
curl -X POST http://localhost:8080/api/v1/corpassist/rag/query \
  -H "Authorization: Bearer $USER_JWT" \
  -H "Content-Type: application/json" \
  -d '{"query": "报销超过5000元需要什么审批？", "top_k": 5}' | jq .

# 预期输出：
# {
#   "answer": "报销超过5000元需要总经理审批……",
#   "sources": [
#     {"title": "报销制度.pdf", "score": 0.95, "url": "..."},
#     {"title": "财务管理制度.pdf", "score": 0.87, "url": "..."}
#   ]
# }

Step 4: Agent 推理

bash
# Agent 推理（SSE 流式）
curl -N -X POST http://localhost:8080/api/v1/corpassist/agent/run \
  -H "Authorization: Bearer $USER_JWT" \
  -H "Content-Type: application/json" \
  -d '{"query": "帮我查上个月的报销进度", "session_id": "demo-001"}'

# SSE 输出示例：
# data: {"token": "好的"}
# data: {"token": "，"}
# data: {"token": "我"}
# data: {"token": "正在"}
# data: {"token": "查询"}
# data: {"tool_call": "query_order", "params": "..."}
# data: {"token": "您的"}
# data: {"token": "报销"}
# data: {"token": "正在"} request id
# data: [DONE]

Step 5: 转人工

bash
# Agent 触发转人工
curl -N -X POST http://localhost:8080/api/v1/corpassist/agent/run \
  -H "Authorization: Bearer $USER_JWT" \
  -H "Content-Type: application/json" \
  -d '{"query": "我要投诉，转人工！", "session_id": "demo-001"}'

# 预期输出包含 handoff 信号
# data: {"type": "handoff", "reason": "user_requested",
#        "summary": "用户投诉报销处理慢，需要人工介入"}

Step 6: 灰度切换

bash
# 通过配置中心动态开启灰度
curl -X PUT http://localhost:8888/config/feature-flags \
  -H "Content-Type: application/json" \
  -d '{"new-python-rag-pipeline": {"enabled": true, "traffic_percentage": 50}}'

# 验证灰度生效（50% 请求走新管线）
for i in {1..100}; do
  curl -s -X POST http://localhost:8080/api/v1/corpassist/rag/query \
    -H "Authorization: Bearer $USER_JWT" \
    -H "Content-Type: application/json" \
    -d '{"query": "测试请求", "top_k": 3}' \
    -H "X-Tenant-Id: corp-$(($RANDOM % 100))" \
    | jq -r '.x_trace_id' >> trace_ids.log
done

Step 7: Trace 排障

bash
# 按 trace_id 关联查日志
TRACE_ID="tr-abc-001"

# 查 Spring 日志
docker compose logs spring-bff | grep $TRACE_ID

# 查 Python 日志
docker compose logs python-rag | grep $TRACE_ID

# 查 Gateway 日志
docker compose logs spring-gateway | grep $TRACE_ID

# 端到端 timing
echo "=== 各阶段耗时 ==="
grep $TRACE_ID docker-compose-logs/*.log \
  --no-filename | jq '{service: .service, elapsed_ms: .elapsed_ms}'

里程碑复盘

milestone review 是毕业演示后的复盘环节。

走过场
  • 只看指标数字，不看趋势
  • 只报喜不报忧
  • 没有改进计划
  • 没有记录技术债务
有效复盘
  • 对比基线 vs 演示值，分析差异原因
  • 列出联调中发现的问题清单
  • 每项问题都有改进 owner 和 deadline
  • 记录架构决策和 tradeoff 理由

常见问题（FAQ）

演示时某个服务启动失败了怎么办？
应急预案：1) 先重启失败服务（docker compose restart python-rag）；2) 如果仍然失败，检查依赖服务（如 Milvus）是否健康；3) 如果服务短期无法恢复，跳过该演示步骤，口头说明该功能已验证通过；4) 演示结束后修复并补录演示视频。建议演示前 30 分钟完成全量启动检查。
演示时网络不好，LLM 调用很慢怎么办？
可以先用Mock LLM演示：在 Python RAG API 中配置 mock_mode=true，返回预设的示例答案。演示开始时说明「这是用 mock 数据演示交互流程，真实场景下 LLM 调用需要 2-5 秒」。演示完交互流程后，可以切换到真实 LLM 展示一次端到端效果。
演示完 stakeholder 问「什么时候可以上线？」怎么回答？
回答框架：1) 已完成的功能（S1 RAG 问答）可以进入灰度发布流程，预计 2 周内小范围验证；2) Agent 功能（S2）还需要完成若干优化（如超时处理、转人工体验）后才能灰度；3) 给一个保守的时间线，留出缓冲；4) 明确需要 stakeholder 提供的支持（如灰度租户、预算审批等）。
stakeholder 问「这和 ChatGPT 有什么区别？」
关键差异：1) 数据安全 —— CorpAssist 部署在企业内网或私有云，数据不出域；2) 知识定制 —— 使用企业的知识库（制度文档、流程规范）而不是通用知识；3) 工具集成 —— 可以调用企业内部 API（查订单、查工单、查审批）；4) 审批流程 —— 支持转人工、操作审计等企业合规要求。总结：ChatGPT 是通用助手，CorpAssist 是企业专属的智能工作助手。

判断练习

  1. 演示到第 4 步（Agent 推理）时 SSE 断流了。你手上有三种选项：A) 继续排查并修复；B) 跳过演示并口头说明；C) 重启服务重试。你会选哪个？为什么？

    参考答案

    选 B。演示不是排障现场。先口头说明 Agent 功能已通过验收，展示之前的测试截图或日志。演示结束后再排查修复。如果第 4 步是关键演示项，也可以在说明后切换到备用的 mock agent 演示交互流程。原则：不要让观众看到你在排障。
