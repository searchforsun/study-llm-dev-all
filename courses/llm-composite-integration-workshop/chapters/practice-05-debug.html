排障基础设施：全链路 Trace

distributed trace 是双栈排障的基础。每一次用户请求生成唯一的 tracing_id，贯穿 Spring Gateway → BFF → Python → LLM 的完整链路。

全链路 Trace 透传
sequenceDiagram
    participant GW as Gateway
    participant BFF as BFF
    participant PY as Python AI
    participant LLM as LLM API

    Note over GW,LLM: trace_id: tr-abc-001\nspan_id 逐跳变化

    GW->>BFF: X-Request-Id: tr-abc-001\nX-Span-Id: span-1
    BFF->>BFF: 创建子 span (span-2)
    BFF->>PY: X-Request-Id: tr-abc-001\nX-Span-Id: span-2
    PY->>PY: 创建子 span (span-3)
    PY->>LLM: X-Request-Id: tr-abc-001
    LLM-->>PY: response
    PY-->>BFF: response
    BFF-->>GW: response

全链路 Trace：trace_id 跨服务不变，span_id 每跳变化，实现调用链追踪。

Spring Trace 过滤器

java
// Spring Trace 过滤器
@Component
public class TraceFilter implements WebFilter {

    private static final String TRACE_HEADER = "X-Request-Id";
    private static final String SPAN_HEADER = "X-Span-Id";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String traceId = exchange.getRequest().getHeaders()
            .getFirst(TRACE_HEADER);
        if (traceId == null || traceId.isBlank()) {
            traceId = UUID.randomUUID().toString();
        }

        String spanId = UUID.randomUUID().toString().substring(0, 8);

        // 注入 MDC（日志上下文）
        MDC.put("trace_id", traceId);
        MDC.put("span_id", spanId);

        // 追加到响应 Header
        exchange.getResponse().getHeaders()
            .add(TRACE_HEADER, traceId);

        return chain.filter(
            exchange.mutate()
                .request(exchange.getRequest().mutate()
                    .header(TRACE_HEADER, traceId)
                    .header(SPAN_HEADER, spanId)
                    .build())
                .build())
            .doFinally(signalType -> {
                MDC.clear();
            });
    }
}

Python Trace 中间件

python
# Python Trace 中间件
import structlog
from fastapi import Request
import uuid

logger = structlog.get_logger()

@app.middleware("http")
async def trace_middleware(request: Request, call_next):
    # 1. 获取透传的 trace_id
    trace_id = request.headers.get("X-Request-Id") or str(uuid.uuid4())
    span_id = str(uuid.uuid4())[:8]

    # 2. 注入日志上下文
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(
        trace_id=trace_id,
        span_id=span_id,
        service="python-rag",
        path=request.url.path,
    )

    # 3. 记录请求开始
    logger.info("request_start",
        method=request.method,
        query_params=dict(request.query_params))

    # 4. 执行请求
    start = time.time()
    response = await call_next(request)
    elapsed = time.time() - start

    # 5. 记录请求结束
    logger.info("request_end",
        status_code=response.status_code,
        elapsed_ms=round(elapsed * 1000, 2))

    # 6. 注入响应 Header
    response.headers["X-Request-Id"] = trace_id
    response.headers["X-Span-Id"] = span_id

    return response

常见问题排障手册

common issues 是双栈联调中最常见的问题及排查步骤。

问题现象排查步骤根因
401 租户错 Python 返回 401 "tenant mismatch" 1) 查 Gateway 日志中 X-Tenant-Id 值
2) 查 Python 日志中收到的 tenant_id		 Gateway 未正确注入或清除客户端传的 X-Tenant-Id
404 版本错 Python 返回 404 路径不存在 1) 核对 Spring 调用的 URL
2) 核对 Python 的路由定义		 Spring 调 /v1/rag/query 但 Python 定义在 /v2/rag/query
SSE 断流 Agent 输出到一半客户端断开 1) 检查 BFF 的 read-timeout
2) 检查 Python Agent 的超时配置		 BFF 的 read-timeout 太短或客户端连接不稳定
500 反序列化 Spring 调用 Python 时 JSON 解析失败 1) 用 curl 直接调 Python 看响应格式
2) 对比 OpenAPI 契约		 Python 返回的字段名/类型与 Spring DTO 不匹配
超时 Spring 调用 Python 经常超时 1) 检查 Python 服务的响应时间
2) 检查 LLM 调用的延迟		 LLM 响应慢 / Python Worker 线程耗尽
幂等重复 索引任务被重复执行 1) 检查 Idempotency-Key 是否正确传递
2) 检查 Redis 中的幂等缓存		 客户端未传 Idempotency-Key 或 Gateway 清除了该 Header
不推荐：无系统排障
  • 出问题先重启服务
  • 两边各自查日志，手动比对时间戳
  • 没有 trace_id，只能用用户 ID 和时间大致关联
  • 排障过程不文档化
推荐：系统排障流程
  • 第一步：按 trace_id 关联所有服务的日志
  • 第二步：用 curl 绕过 BFF 直连 Python 隔离问题
  • 第三步：检查 OpenAPI 契约是否一致
  • 第四步：将排障过程写入 Runbook

Runbook：系统化排障流程

runbook 是双栈排障的标准化操作手册。以下是 CorpAssist 的核心排障 Runbook。

排障决策树
flowchart TB
    A[用户报错] --> B{有 trace_id 吗？}
    B -->|有| C[按 trace_id 查日志]
    B -->|没有| D[从请求日志找 trace_id]

    C --> E{Spring 端报错？}
    E -->|是| F{Spring 还是 Python 的错？}
    E -->|否| G[Python 端排查]

    F -->|Spring| H[查 OpenAPI 契约\n检查 DTO 对齐]
    F -->|Python| I[curl 直连 Python\n检查响应格式]

    G --> J[查 LLM 调用日志]
    J --> K{LLM 超时？}
    K -->|是| L[检查模型状态\n或降级模型]
    K -->|否| M[查向量库状态]

    H --> N[修复后更新契约测试]
    I --> N
    L --> N
    M --> N
    N --> O[验证通过 → 关闭]

排障决策树：按 trace_id 关联日志，逐层定位问题根因。

Runbook 模板

markdown
# CorpAssist 排障 Runbook

## 问题：Python RAG API 返回 500

### 排查步骤
1. 从 BFF 日志中找到出错的 trace_id
2. 按 trace_id 在 Python 日志中定位对应请求
3. curl 直连 Python 复现问题：
   ```bash
   curl -X POST http://localhost:8001/v1/rag/query \
     -H "X-User-Id: user-001" \
     -H "X-Tenant-Id: corp-001" \
     -d '{"query": "测试", "top_k": 5}'
   ```
4. 检查 Python 日志中的异常栈
5. 修复后添加契约测试防止再次发生

### 常见根因
- Milvus 连接超时 → 检查 Milvus 健康状态
- OpenAI API 速率限制 → 检查 API Key 配额
- 租户向量 collection 不存在 → 确认是否已创建索引

### 验证
- [ ] curl 直连 Python 返回 200
- [ ] Spring BFF 调用成功
- [ ] 用户侧确认正常

常见问题（FAQ）

两边日志的 trace_id 对不上怎么办？
可能原因：1) Spring Gateway 没有生成 trace_id（检查 TraceFilter 是否配置正确）；2) Python 端的 X-Request-Id Header 被错误地赋值（检查中间件处理顺序）；3) BFF 到 Python 的 HTTP Client 没有透传 trace_id（检查 WebClient 配置）。排查方法：在 Gateway 打印收到的请求 Header，然后在 BFF 和 Python 逐跳打印收到的 X-Request-Id。
SSE 流式输出的 Agent 如何排障？
SSE 排障的核心是逐段检查：1) 先用 curl 直连 Python Agent，确认 SSE 流正常输出；2) 通过 BFF 代理调用，确认 BFF 正确转发 SSE 事件；3) 通过 Gateway 调用，确认 Gateway 没有截断 SSE 流；4) 最终从客户端验证。关键指标是每段 SSE 的 timing：如果 BFF 和 Python 之间的 SSE 有间隔但客户端没有收到，说明 BFF 缓存了 SSE。
如何排查「Python 返回正确但 Spring 解析失败」？
这是典型的DTO 不匹配。排查步骤：1) 用 curl 直连 Python 获取完整响应 JSON；2) 将 JSON 与 Spring 的 DTO 类字段逐一比对；3) 检查字段名大小写（JSON 是 camelCase 还是 snake_case）；4) 检查类型（Java Integer vs Python int，Java OffsetDateTime vs Python datetime）；5) 检查 OpenAPI 规范与两端的实现是否一致。参见 OpenAPI 与契约。
已经上线的系统出现偶发超时，如何定位？
偶发超时是最难排查的问题之一。建议：1) 在 BFF 添加 retry + 熔断，避免偶发超时影响用户体验；2) 在日志中记录每次调用的 P50/P95/P99 延迟；3) 对比 Python 服务本身的处理时间与网络传输时间（通过 timing Header 记录）；4) 使用 APM（如 SkyWalking、Datadog）分析慢调用的调用链。常见原因是 LLM API 偶发延迟、Milvus 向量检索排队、JVM GC 暂停。
