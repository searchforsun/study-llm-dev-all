双栈拓扑全景

CorpAssist 采用双栈集成架构（dual stack topology），将系统分为两个明确的技术栈：

  • Spring 治理面（spring governance)：基于 Spring Cloud 生态，负责网关、鉴权、会话管理、服务编排、审计日志等治理能力
  • Python 能力面（python capability)：基于 FastAPI + LangChain/LangGraph 生态，负责索引构建、RAG 问答、Agent 推理、模型调用等 AI 能力

CorpAssist · 双栈交互场景 企业员工通过企业微信向 CorpAssist 提问「我的上个月报销到哪了」。请求经过：企业微信 → Spring Gateway(验签+路由) → Spring BFF(组装上下文+鉴权) → Python RAG API(向量检索+LLM 生成) → 响应经原路返回。整个过程涉及双栈 5 个服务协作。
不推荐：职责模糊
  • Python 服务自己鉴权、自己限流、自己管会话
  • Spring 服务直接调用 LLM 模型
  • 两边都能改向量库的索引数据结构
  • 没有明确的 BFF 层，客户端直连 Python
推荐：职责清晰
  • Spring 集中治理：鉴权/限流/审计/编排
  • Python 专注 AI：RAG/Agent/索引/评估
  • 向量库只读/写入分离：Python 索引写入，Spring/Python 只读查询
  • BFF 层作为双桥接点，透传可信上下文

Spring 治理面

Spring governance 层是 CorpAssist 的治理中枢，承担所有非 AI 相关的企业级能力。其核心组件包括：

组件职责技术选型
API Gateway 统一入口、路由分发、限流、IP 白名单 Spring Cloud Gateway
BFF 双栈桥接、会话管理、请求组装、响应聚合 Spring Boot WebFlux
Auth Service OAuth2 令牌校验、JWT 签发、M2M 认证 Spring Security + Keycloak
编排引擎 RAG 问答编排、Agent 状态管理、转人工 Spring AI + 状态机
审计日志 全量请求/响应记录、敏感操作追踪 Spring AOP + MongoDB

Spring 治理面的核心价值

为什么选择 Spring 承担治理层？而非用 Python 实现同样的功能？

  • 成熟的网关生态：Spring Cloud Gateway 提供完善的过滤器链、限流（RequestRateLimiter）、熔断（Resilience4j）、灰度路由
  • 标准化安全框架：Spring Security + OAuth2 + OIDC 已大规模验证
  • 企业合规要求：审计日志、RBAC、数据脱敏在 Java 生态有成熟的 AOP 实现
  • 团队舒适区：企业内部通常已有 Java 平台团队，复用治理能力

// Spring BFF - CorpAssist 请求转发核心逻辑
@RestController
@RequestMapping("/api/v1/corpassist")
public class CorpAssistController {

    private final WebClient pythonRagClient;
    private final WebClient pythonAgentClient;

    @PostMapping("/rag/query")
    public Mono<RagQueryResponse> ragQuery(
            @RequestBody @Valid RagQueryRequest request,
            @RequestHeader("X-User-Id") String userId,
            @RequestHeader("X-Tenant-Id") String tenantId) {

        // 1. 注入用户上下文到 Python 请求
        return pythonRagClient.post()
            .uri("/v1/rag/query")
            .header("X-User-Id", userId)
            .header("X-Tenant-Id", tenantId)
            .header("X-Request-Id", UUID.randomUUID().toString())
            .bodyValue(request)
            .retrieve()
            .bodyToMono(RagQueryResponse.class)
            // 2. 超时 + 降级
            .timeout(Duration.ofSeconds(30))
            .onErrorResume(ex -> Mono.just(
                new RagQueryResponse("error", "服务暂时不可用，请稍后重试")));
    }

    @PostMapping(value = "/agent/run", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> agentRun(
            @RequestBody AgentRunRequest request,
            @RequestHeader("X-User-Id") String userId) {
        // 1. 从 Redis 获取会话历史
        // 2. 转发到 Python Agent API（SSE 流式）
        // 3. 流式返回给客户端
        return pythonAgentClient.post()
            .uri("/v1/agent/run")
            .header("X-Session-Id", request.sessionId())
            .bodyValue(request)
            .retrieve()
            .bodyToFlux(new ParameterizedTypeReference<ServerSentEvent<String>>() {});
    }
}

Python RAG Query API 核心代码

python
# Python RAG Query API - CorpAssist 问答核心
from fastapi import FastAPI, HTTPException, Header
from pydantic import BaseModel
from langchain_community.vectorstores import Milvus
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.chains import RetrievalQA

app = FastAPI(title="CorpAssist RAG API", version="1.0.0")

class RagQueryRequest(BaseModel):
    query: str
    top_k: int = 5

class RagQueryResponse(BaseModel):
    answer: str
    sources: list[str]

@app.post("/v1/rag/query")
async def rag_query(
    request: RagQueryRequest,
    x_user_id: str = Header(...),
    x_tenant_id: str = Header(...),
    x_request_id: str = Header(None),
):
    # 1. 根据租户隔离向量库
    vector_store = Milvus(
        embedding_function=OpenAIEmbeddings(),
        collection_name=f"corp_{x_tenant_id}",
        connection_args={"host": "milvus", "port": "19530"},
    )
    # 2. 构建 RAG Chain
    qa_chain = RetrievalQA.from_chain_type(
        llm=ChatOpenAI(model="gpt-4o", temperature=0),
        retriever=vector_store.as_retriever(
            search_kwargs={"k": request.top_k}
        ),
        return_source_documents=True,
    )
    # 3. 执行问答
    result = await qa_chain.ainvoke({"query": request.query})
    return RagQueryResponse(
        answer=result["result"],
        sources=[doc.metadata["source"] for doc in result["source_documents"]],
    )

关键数据流节点

节点输入输出关键操作
1. Gateway HTTP 原始请求 可信 Header JWT 校验、限流、路由
2. BFF 可信 Header + Body 增强请求 会话注入、权限校验、上下文组装
3. Python API 查询 + 用户上下文 检索结果 + LLM 答案 向量检索、LLM 调用
4. 响应返回 Python 响应 客户端响应 脱敏、格式转换、审计日志
常见误区
  • 双栈职责边界模糊：Spring 也写 AI 逻辑，Python 也写鉴权，导致两边代码耦合
  • BFF 过薄：直接把 Python 的响应原封不动返回，失去了治理层的价值
  • Python 直连数据库：Python 应通过 Spring BFF 或 MQ 访问业务数据库，而不是直接连
  • 忽略网络开销：每次双栈调用有 1-5ms 延迟，长链路需考虑超时和熔断

BFF 层：双栈桥接

bff layer 是 Spring 治理面中直接面向 Python 能力面的桥接层。它承担以下职责：

  • 请求翻译：将 Spring 内部上下文转换为 Python API 可识别的 HTTP Header + Body
  • 响应聚合：调用多个 Python 服务后聚合结果（如 RAG + Agent 组合）
  • 降级/熔断：Python 服务不可用时，BFF 返回缓存答案或友好提示
  • 超时控制：对 Python 的调用设置合理的超时，避免线程阻塞
不推荐：BFF 过厚
  • BFF 包含业务逻辑（如计算、状态机）
  • BFF 直接调用 LLM SDK
  • BFF 访问向量数据库做检索
  • BFF 包含复杂的缓存失效逻辑
推荐：BFF 适中
  • BFF 做请求组装/响应聚合/上下文注入
  • BFF 处理网关透传的 Header 增强
  • BFF 做会话管理和历史记录
  • BFF 处理超时/熔断/重试策略

常见问题（FAQ）

为什么不在 Python 中也实现网关功能？
理论上可行，但实践中建议单一网关层。如果 Python 也做网关，会导致路由规则分散、鉴权策略不统一、审计日志需要聚合两边。Spring Cloud Gateway 的生态成熟度（限流、熔断、过滤器链）远超 Python 网关实现。Python 应专注于 AI 能力，治理能力交给 Spring。
双栈之间是否必须走 HTTP？
不一定。非实时场景（如索引构建、批量评估）可以通过 MQ（RabbitMQ/Kafka）进行异步通信。实时问答场景（RAG/Agent）通常走 HTTP/gRPC。选择依据：实时性 vs 解耦度的权衡。详见 同步、异步与 MQ。
Python 服务如何知道当前用户的租户信息？
Spring Gateway 在 JWT 校验后会注入可信 Header（如 X-Tenant-Id、X-User-Id、X-User-Role）。Python 服务从 Header 读取即可，不需要自己验 JWT。Python 应信任这些 Header，因为上游 Gateway 已经做了鉴权。详见 统一鉴权。
双栈部署时，先部署 Spring 还是 Python？
建议先部署中间件（Redis、Milvus、PostgreSQL），再部署 Python 能力面（索引构建需要先就绪），最后部署 Spring 治理面。Spring 治理面启动时会检查依赖服务的健康状态。Docker Compose 中可以配置 depends_on + healthcheck 保证启动顺序。详见 Docker Compose 本地开发栈。

契约优先：OpenAPI 3.1 作为唯一真相源

openapi contract 是双栈集成的契约基础。Spring 治理面和 Python 能力面之间所有的 API 通信都基于同一份 OpenAPI 规范文件。

schema first 流程如下：

  1. 定义契约：团队（平台架构师）编写 OpenAPI 3.1 YAML
  2. 代码生成：Spring 端用 openapi-generator-maven-plugin 生成 Java DTO/API 接口；Python 端用 openapi-generator-cli 生成 Pydantic 模型
  3. 双栈实现：Spring 实现治理接口，Python 实现 AI 接口
  4. 契约测试：双方基于同一契约进行集成测试（参见 契约测试与集成测试）
  5. 破坏检测：CI 中运行 openapi-diff 检测 breaking change

DTO 对齐

dto align 确保双栈之间的数据传输对象在各语言中保持一致。关键原则：

  • 类型严格对应：Java String ↔ Python str，Java Integer ↔ Python int，Java BigDecimal ↔ Python Decimal
  • 命名统一：camelCase 用于 JSON 字段名，Java 和 Python 都使用 @JsonProperty/serialization_alias 映射
  • 可选性一致：required 在 OpenAPI 和 Java/Python 中必须一致，否则导致反序列化失败
  • 日期时间格式：统一使用 ISO-8601（2026-05-31T10:30:00Z），避免 LocalDateTime/Naive datetime 不匹配

API 版本管理

api version 策略决定了双栈如何在升级过程中保持兼容。CorpAssist 采用路径版本 + Content-Type 版本双模式：

  • 路径版本：/v1/rag/query, /v2/rag/query —— 大版本升级时使用
  • Header 版本：Accept: application/vnd.corpassist.v2+json —— 小版本向后兼容

breaking change 包括：删除字段、修改字段类型、增加 required 字段、修改端点路径。这些变更必须走大版本更新。

不推荐：无版本管理
  • 直接修改已有 API 的响应结构
  • 删除字段不做兼容
  • 无版本号或只用 git commit 做版本
  • 双栈各自维护一份不同的契约
推荐：版本化管理
  • 向后兼容的字段只做新增不做修改
  • 弃用字段标注 deprecated + sunset 日期
  • 每次变更更新 OpenAPI 文件的 version 字段
  • 双栈共享同一份契约文件（contract repo）
常见误区
  • 契约不同步：Java 改了 DTO 但未更新 OpenAPI，Python 还在用旧字段
  • 生成代码被手动修改：openapi-generator 生成的文件被手动编辑，下次生成时覆盖
  • 版本号随意递增：微小的向后兼容变更也升大版本，导致客户端版本爆炸
  • 忽略可选字段：服务端新增可选字段，旧客户端收到未知字段时可能崩溃

常见问题（FAQ）

契约文件应该放在哪个代码仓库？
建议独立契约仓库（contract repo）或 monorepo 中的独立模块。Spring 和 Python 子模块通过 Git Submodule 或 CI 拉取同一份契约。契约仓库的写入权限应仅限架构师/平台团队，普通开发者只能提 PR。参考 团队拓扑与交付协作。
openapi-generator 生成的代码可以手动修改吗？
强烈不建议。生成的文件应标记为「auto-generated」，任何手动修改都会被下次生成覆盖。如果需要自定义逻辑，使用模板扩展（mustache 模板）或在生成的 DTO 外包装一层。CorpAssist 实践中，所有的 DTO 校验逻辑写在 Service 层而非 DTO 层。
如何检测 breaking change？
使用 openapi-diff 或 IBM OpenAPI Validator。在 CI 中对比本次 PR 的 OpenAPI 变更与基线的 diff，如果检测到 breaking change 则阻断合并。详见 契约测试。
Python 和 Java 的日期时间类型怎么对齐？
所有时间字段统一使用 ISO-8601 字符串（如 2026-05-31T10:30:00Z），在 OpenAPI 中定义为 format: date-time。Java 用 OffsetDateTime，Python 用 datetime（带 timezone）。不要在 JSON 中传输 LocalDateTime 或 Naive datetime。

统一鉴权架构：Spring Gateway 集中式认证

双栈鉴权的核心设计原则：认证在 Gateway，授权在服务。Spring Cloud Gateway 作为统一入口，负责所有用户的认证，Python 服务不再需要关心用户是谁、令牌是否有效——它只需要相信 Gateway 注入的 Header。

Spring Gateway JWT 校验过滤器

java
// Spring Cloud Gateway - JWT 校验与 Header 注入
@Component
public class JwtAuthFilter implements GlobalFilter, Ordered {

    private final ReactiveJwtDecoder jwtDecoder;

    public JwtAuthFilter(ReactiveJwtDecoder jwtDecoder) {
        this.jwtDecoder = jwtDecoder;
    }

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String authHeader = exchange.getRequest()
            .getHeaders().getFirst(HttpHeaders.AUTHORIZATION);

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        String token = authHeader.substring(7);
        return jwtDecoder.decode(token)
            .flatMap(jwt -> {
                // 注入可信 Header
                ServerHttpRequest modifiedRequest = exchange.getRequest().mutate()
                    .header("X-User-Id", jwt.getClaimAsString("sub"))
                    .header("X-Tenant-Id", jwt.getClaimAsString("tenant_id"))
                    .header("X-User-Role", jwt.getClaimAsString("role"))
                    .header("X-Request-Id", UUID.randomUUID().toString())
                    .build();
                ServerWebExchange modifiedExchange = exchange.mutate()
                    .request(modifiedRequest).build();
                return chain.filter(modifiedExchange);
            })
            .onErrorResume(e -> {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            });
    }

    @Override
    public int getOrder() {
        return -100; // 高优先级
    }
}

服务间 M2M 认证

m2m auth 用于 Spring BFF 调用 Python 服务时的服务间认证。虽然 Python 服务部署在内网，但仍然建议添加一层轻量认证：

                .header("X-User-Id", userId)
                .header("X-Tenant-Id", tenantId)
                .bodyValue(request)
                .retrieve()
                .bodyToMono(RagQueryResponse.class));
    }

    private Mono<String> getInternalToken() {
        if (cachedInternalToken != null
            && Instant.now().isBefore(tokenExpiry.minusSeconds(60))) {
            return Mono.just(cachedInternalToken);
        }
        // 使用客户端凭证模式获取内部令牌
        return authClient.getToken(
            "corpassist-bff", "client-secret",
            "corpassist-python-api")
            .map(response -> {
                cachedInternalToken = response.accessToken();
                tokenExpiry = Instant.now()
                    .plusSeconds(response.expiresIn());
                return cachedInternalToken;
            });
    }
}

# 内部 JWT 公钥（从 Auth Service 同步）
INTERNAL_JWKS = {
    "keys": [
        # ... 内部服务的 RSA 公钥
    ]
}

async def verify_internal_token(
    credentials: HTTPAuthorizationCredentials = Depends(security_scheme),
) -> str:
    """验证内部 M2M 令牌"""
    if credentials is None:
        # 如果没有令牌，仅依赖内网 Header（信任 Gateway）
        return "anonymous"

    try:
        payload = jwt.decode(
            credentials.credentials,
            INTERNAL_JWKS,
            algorithms=["RS256"],
            audience="corpassist-python-api",
        )
        return payload.get("sub", "unknown")
    except jwt.PyJWTError:
        # M2M 验证失败时，仍允许内网请求通过
        # 但需要记录告警
        logger.warning("Internal token verification failed")
        return "untrusted"

tenant isolation 是 CorpAssist 企业场景中的核心需求。不同企业（租户）的数据必须严格隔离。

隔离层级实现方式隔离强度
向量库 每个租户独立的 collection/partition 强
检索过滤 检索时传入 tenant_id 作为 filter 表达式 中
会话存储 Redis key 前缀: session:{tenant_id}:{user_id} 强
审计日志 DB 表中 tenant_id 字段分区 中
LLM 调用 按租户配置不同的 model/temperature 弱
不推荐
  • 所有租户共享同一个向量 collection
  • Python 自己从 JWT 解析 tenant_id
  • 不使用 X-Tenant-Id Header
  • 租户隔离仅依赖代码逻辑，无基础设施保障
推荐
  • 每个租户独立 collection + partition
  • Python 从可信 Header 读取 tenant_id
  • BFF 校验 tenant_id 与用户是否匹配
  • 向量检索时强制加入 tenant_id 过滤

零信任架构

zero trust 在双栈鉴权中的应用：即使 Python 服务运行在内网，也不自动信任任何请求。实施要点：

  • 请求必须包含有效令牌：内部 JWT 或 Gateway 签发的会话令牌
  • 每次都验签：不缓存认证结果（除了 M2M token 的过期时间缓存）
  • 最小权限：Python 服务只获得执行所需的最小 scope
  • 审计日志：所有跨服务调用记录请求方身份和操作

常见问题（FAQ）

Python 服务如何知道当前用户的角色和权限？
通过 Spring Gateway 注入的 X-User-Role 和 X-User-Dept Header。Python 不需要自己解析 JWT，直接从 Header 读取即可。这样做的好处是：认证逻辑只在一个地方（Gateway）实现，Python 服务更轻量。
如果 Gateway 挂了，Python 服务还能工作吗？
不能直接对外提供服务，这是设计上的。Gateway 是整个系统的唯一入口，Gateway 不可用意味着整个系统不可用。但 Python 服务可以处理 BFF 的内部请求。高可用方案：Gateway 多副本 + 健康检查 + 自动重启。
内部 JWT 和用户 JWT 有什么区别？
用户 JWT 由外部 OAuth2 提供商签发，包含用户身份信息。内部 JWT 由 Auth Service 用客户端凭证模式签发，用于服务间认证。内部 JWT 的受众（aud）限制为特定的 Python API，生命周期更短（通常 5-15 分钟）。
X-Tenant-Id 会在响应中返回给客户端吗？
不应该。X-Tenant-Id 是内部 Header，Spring BFF 在返回响应给客户端前应该删除它（RemoveResponseHeader 过滤器）。客户端不应该知道内部租户标识的具体值。

本章结论

  • 集中认证：Spring Gateway 统一校验 JWT，Python 不用重复实现鉴权
  • 可信 Header：Gateway 注入 X-User-Id/X-Tenant-Id 等 Header，下游服务信任它们
  • M2M 认证：服务间使用内部 JWT 或 mTLS 增强安全
  • 租户隔离：基于 X-Tenant-Id 在向量库/Redis/数据库层面进行数据隔离

首次启动 Milvus 需要很长时间，怎么办？
Milvus 首次启动需要初始化 etcd 和 MinIO，通常需要 1-3 分钟。建议：1) 使用 start_period 给 Milvus 更长的启动时间；2) 不要同时启动所有服务，先启动基础设施再启动应用服务；3) 使用卷持久化数据，避免每次重新初始化。
Python 热重载在容器中不生效？
确保：1) Dockerfile 中使用 CMD ["uvicorn", "main:app", "--reload", "--host", "0.0.0.0"]；2) Volume 挂载了正确的源码路径；3) 文件修改后容器确实能感知到（Windows 上可能需要设置文件共享）。如果依然不生效，可以使用 docker compose restart python-rag 手动重启。
如何调试 Spring 服务中的远程调用？
在 docker-compose 中为 Java 服务添加 JVM 调试参数：JAVA_TOOL_OPTIONS: "-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005"。然后映射调试端口（如 5005:5005），使用 IDE 的 Remote JVM Debug 配置连接到 localhost:5005。
本地开发时需要模拟外部 OAuth2 提供者？
可以使用 testcontainers（Java 集成测试）或 Keycloak 开发容器。在 docker-compose 中添加 auth-service 时，可以配置 Keycloak 的 dev 模式，导入预定义的 realm 和用户。BFF 和 Gateway 的 dev profile 配置指向本地 Keycloak。

Python 索引管线

Python 索引管线完整实现

python
# Python 索引管线 - indexing_pipeline.py
from celery import Celery
from langchain_community.document_loaders import UnstructuredFileLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_milvus import Milvus
import hashlib
from pathlib import Path

celery_app = Celery("corpassist_indexer",
    broker="redis://redis:6379/0",
    backend="redis://redis:6379/1")

@celery_app.task(bind=True, max_retries=3, default_retry_delay=60)
def index_document(self, file_path: str, tenant_id: str,
                  knowledge_base_id: str) -> dict:
    """索引单个文档"""
    try:
        # 1. 文档解析
        loader = UnstructuredFileLoader(file_path)
        documents = loader.load()

        # 2. 文档分块
        text_splitter = RecursiveCharacterTextSplitter(
            chunk_size=1000,
            chunk_overlap=200,
            separators=["\n\n", "\n", "。", "！", "？", " ", ""],
        )
        chunks = text_splitter.split_documents(documents)

        # 3. 添加元数据
        for chunk in chunks:
            chunk.metadata["tenant_id"] = tenant_id
            chunk.metadata["kb_id"] = knowledge_base_id
            chunk.metadata["chunk_hash"] = hashlib.md5(
                chunk.page_content.encode()).hexdigest()
            chunk.metadata["indexed_at"] = datetime.utcnow().isoformat()

        # 4. 写入向量库
        vector_store = Milvus(
            embedding_function=OpenAIEmbeddings(),
            collection_name=f"kb_{knowledge_base_id}",
            connection_args={"host": "milvus", "port": "19530"},
        )
        vector_store.add_documents(chunks)

        return {
            "status": "success",
            "file": file_path,
            "chunks": len(chunks),
            "tenant_id": tenant_id,
        }

    except Exception as e:
        self.retry(exc=e)

def index_batch(file_paths: list[str], tenant_id: str, kb_id: str):
    """批量提交索引任务"""
    tasks = []
    for fp in file_paths:
        task = index_document.delay(fp, tenant_id, kb_id)
        tasks.append(task.id)
    return tasks

# 索引版本管理
def create_index_version(kb_id: str, tenant_id: str) -> str:
    """创建新的索引版本"""
    version = f"kb_{kb_id}_v{int(time.time())}"
    # 创建新 collection
    vector_store = Milvus(
        embedding_function=OpenAIEmbeddings(),
        collection_name=version,
        connection_args={"host": "milvus", "port": "19530"},
    )
    # 记录版本元数据
    redis_client.hset(f"index_version:{kb_id}", "current", version)
    return version

def switch_index_version(kb_id: str, version: str):
    """切换索引版本"""
    redis_client.hset(f"index_version:{kb_id}", "current", version)
    redis_client.publish("index_switch", json.dumps({
        "kb_id": kb_id,
        "version": version,
        "switched_at": datetime.utcnow().isoformat(),

Spring 问答编排

spring orchestration 层使用 Spring AI 框架编排 RAG 问答流程。它负责：接收用户查询→检索相关文档→组装 LLM 上下文→生成回答。

Spring RAG 问答服务

java
// Spring RAG 问答编排服务
@Service
public class RagQueryService {

    private final VectorStore vectorStore;
    private final ChatClient chatClient;

    public RagQueryService(VectorStore vectorStore, ChatClient.Builder chatBuilder) {
        this.vectorStore = vectorStore;
        this.chatClient = chatBuilder
            .defaultSystem("""
                你是一个专业的企业知识助手。请基于提供的上下文回答用户问题。
                如果上下文不足以回答问题，请说明「无法从已有知识库中找到答案」。
                引用格式：[来源: 文档标题, 页码]
                """)
            .build();
    }

    public RagQueryResponse query(RagQueryRequest request, String tenantId) {
        // 1. 向量检索（按租户隔离）
        List<Document> documents = vectorStore.similaritySearch(
            SearchRequest.query(request.query())
                .withTopK(request.topK())
                .withFilterExpression("tenant_id == '" + tenantId + "'"));

        // 2. 组装 Prompt
        String context = documents.stream()
            .map(doc -> doc.getContent() + "\n[来源: " + doc.getMetadata().get("source") + "]")
            .collect(Collectors.joining("\n---\n"));

        // 3. LLM 生成
        String answer = chatClient.prompt()
            .user(u -> u.text("问题: {query}\n\n上下文:\n{context}")
                .param("query", request.query())
                .param("context", context))
            .call()
            .content();

        // 4. 返回结果
        List<SourceRef> sources = documents.stream()
            .map(doc -> new SourceRef(
                (String) doc.getMetadata().get("title"),
                (String) doc.getMetadata().get("source"),
                (Double) doc.getMetadata().get("distance")))
            .toList();

        return new RagQueryResponse(answer, sources);
    }
}

  • Spring 直接调用 LLM 做索引嵌入
  • Python 既做索引也做问答，没有职责分离

常见问题（FAQ）

为什么不在 Spring 中也实现索引功能？
索引需要大量 Python 生态库支持：Unstructured（文档解析）、Sentence-Transformers（本地嵌入）、PyMuPDF（PDF 处理）。Java 在这些领域的生态远不如 Python。此外，索引是低频率的批处理任务，与高频低延迟的问答服务混在一起会影响稳定性。
索引完成后如何通知 Spring BFF？
两种方式：1) Python 索引完成后通过 Webhook 回调 BFF 的 /api/v1/callbacks/index-complete 端点；2) 通过 MQ 发送 IndexCompleted 事件，BFF 监听消费。推荐 MQ 方式，解耦更强。详见 同步、异步与 MQ。
向量库中的旧索引如何处理？
使用索引版本化管理。每次全量重建创建新 collection（如 kb_123_v2），BFF 查询时从 Redis 获取当前活跃版本号。旧版本保留一段时间（如 7 天）用于回滚，之后自动清理。增量索引直接在原 collection 中追加，不需要版本切换。
Spring AI 的 VectorStore 支持 Milvus 的租户过滤吗？
支持。Spring AI 的 MilvusVectorStore 支持 FilterExpression，可以在检索时添加 tenant_id == 'xxx' 的表达式。CorpAssist 的每篇文档在索引时都写入了 tenant_id 元数据字段，检索时通过 filterExpression 参数实现租户隔离。

// Spring BFF - Agent 会话管理与 SSE 代理
@RestController
@RequestMapping("/api/v1/corpassist")
public class AgentController {

    private final WebClient pythonAgentClient;
    private final RedisTemplate<String, String> redis;

    @PostMapping(value = "/agent/run", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<?>> agentRun(
            @RequestBody AgentRunRequest request,
            @RequestHeader("X-User-Id") String userId,
            @RequestHeader("X-Tenant-Id") String tenantId) {

        // 1. 获取或创建会话
        String sessionId = resolveSessionId(request, userId, tenantId);

        // 2. 记录会话开始
        recordSessionStart(sessionId, userId, tenantId, request);

        // 3. 调用 Python Agent（SSE 代理）
        return pythonAgentClient.post()
            .uri("/v1/agent/run")
            .header("X-Session-Id", sessionId)
            .header("X-User-Id", userId)
            .header("X-Tenant-Id", tenantId)
            .bodyValue(request)
            .retrieve()
            .bodyToFlux(new ParameterizedTypeReference<ServerSentEvent<String>>() {})
            .doOnNext(event -> recordAgentEvent(sessionId, event))
            .doOnComplete(() -> recordSessionEnd(sessionId));
    }

    private String resolveSessionId(AgentRunRequest req,
                                     String userId, String tenantId) {
        if (req.sessionId() != null && !req.sessionId().isBlank()) {
            return req.sessionId();
        }
        return "session:" + tenantId + ":" + userId + ":" + UUID.randomUUID();
    }

    private void recordSessionStart(String sessionId, String userId,
                                     String tenantId, AgentRunRequest req) {
        Map<String, String> session = Map.of(
            "user_id", userId,
            "tenant_id", tenantId,
            "query", req.query(),
            "started_at", Instant.now().toString());
        redis.opsForHash().putAll("agent:session:" + sessionId, session);
        redis.expire("agent:session:" + sessionId, Duration.ofHours(24));
    }

    private void recordSessionEnd(String sessionId) {
        redis.opsForHash().put(
            "agent:session:" + sessionId,
            "ended_at", Instant.now().toString());
    }
}

def decide_next_action(state: AgentState) -> str:
    """决定下一步动作"""
    messages = state["messages"]

    # 检查是否触发了转人工
    if len(messages) > 20:  # 超过 10 轮对话
        return "handoff"

    last_message = messages[-1].content if messages else ""

    # 用户要求转人工

        return "query_order"
    if "查询工单" in last_message or "工单进度" in last_message:
        return "query_ticket"

    return "end"

常见问题（FAQ）

Spring 为什么不做 Agent 推理而交给 Python？
Agent 推理的核心是 LangGraph 的图执行引擎、checkpointer 状态管理、工具调用编排。这些功能在 Python 生态（LangChain/LangGraph）中非常成熟，Java 端的 Spring AI 虽然支持 Agent，但在复杂工作流、图状态管理、流式输出等方面还不如 Python 完善。CorpAssist 将推理执行放在 Python，Spring 做治理，各取所长。
Agent 执行时间很长（30秒+），如何处理超时？
BFF 对 Python Agent 的调用设置 120 秒 的 read-timeout。同时，Agent 本身需要实现 streaming（SSE），让用户能实时看到推理过程。如果 Agent 超时，BFF 返回部分结果并提示用户「处理时间过长，请稍后查看结果」。如果 Agent 超过最大步数（如 20 步），LangGraph 自动终止并返回已收集的信息。
如何保证 Agent 工具调用的安全性和可审计性？
tool audit 通过 Spring AOP 或过滤器实现：BFF 记录每次 Agent 请求中调用的工具名称、参数、返回值和耗时。Python 的每个工具函数也需要在调用前后记录日志。Redis 中存储每个 session 的工具调用链，方便事后审计。
Agent 会话如何跨请求保持状态？
LangGraph 的 checkpointer 在每个步骤后自动保存状态到 MemorySaver（内存/REDIS 后端）。Spring BFF 在下一次请求时通过 X-Session-Id 头传递给 Python Agent，LangGraph 从 checkpointer 恢复图状态。会话元数据（用户信息、开始/结束时间）由 Spring 存储在 Redis 中。

  • 分工明确：Python LangGraph 做 Agent 推理循环，Spring 做会话治理和工具审计

  • 会话管理：Redis 存储会话元数据，LangGraph checkpointer 保存图状态

// Spring 异步任务 API
@RestController
@RequestMapping("/api/v1/index")
public class IndexController {

    private final JobRepository jobRepository;
    private final RabbitTemplate rabbitTemplate;

    @PostMapping("/batch")
    public ResponseEntity<JobResponse> submitBatch(
            @RequestBody @Valid BatchIndexRequest request,
            @RequestHeader("X-Tenant-Id") String tenantId) {

        // 1. 创建任务记录
        IndexJob job = IndexJob.builder()
            .id(UUID.randomUUID().toString())
            .tenantId(tenantId)
            .type("index_batch")
            .status(JobStatus.PENDING)
            .params(objectMapper.writeValueAsString(request))
            .createdAt(Instant.now())
            .build();
        jobRepository.save(job);

        // 2. 发送 MQ 消息
        rabbitTemplate.convertAndSend("corpassist.index",
            "job.created",
            new IndexJobEvent(job.getId(), tenantId, request));

        // 3. 返回 202 Accepted
        URI statusUrl = URI.create(
            "/api/v1/index/jobs/" + job.getId());
        return ResponseEntity.accepted()
            .location(statusUrl)
            .body(new JobResponse(job.getId(), "pending",
                statusUrl.toString()));
    }

    @GetMapping("/jobs/{jobId}")
    public ResponseEntity<IndexJob> getJobStatus(
            @PathVariable String jobId) {
        return jobRepository.findById(jobId)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }
}

@Component
public class IdempotencyFilter implements WebFilter {

    private final StringRedisTemplate redis;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String idempotencyKey = exchange.getRequest()
            .getHeaders().getFirst("Idempotency-Key");

        if (idempotencyKey == null) {
            return chain.filter(exchange);
        }

        String cacheKey = "idempotency:" + idempotencyKey;

        return redis.opsForValue().get(cacheKey)
            .flatMap(cached -> {
                // 已处理过，返回上次结果
                ServerHttpResponse response = exchange.getResponse();
                response.setStatusCode(HttpStatus.OK);
                return response.writeWith(
                    Mono.just(response.bufferFactory()
                        .wrap(cached.getBytes(StandardCharsets.UTF_8))));
            })
            .switchIfEmpty(
                chain.filter(exchange).then(
                    // 缓存响应结果
                    Mono.defer(() -> cacheResponse(exchange, cacheKey))
                )
            );
    }
}

Python Idempotency 装饰器

python
# Python 幂等装饰器
from functools import wraps
from redis import Redis
import json

redis_client = Redis(host="redis", port=6379, db=0)

def idempotent(ttl_seconds: int = 86400):
    """幂等装饰器：Idempotency-Key 去重"""
    def decorator(func):
        @wraps(func)
        async def wrapper(request: Request, *args, **kwargs):
            idempotency_key = request.headers.get("Idempotency-Key")
            if not idempotency_key:
                return await func(request, *args, **kwargs)

            cache_key = f"idempotency:{idempotency_key}"
            cached = redis_client.get(cache_key)
            if cached:
                return JSONResponse(
                    content=json.loads(cached),
                    status_code=200,
                )

            response = await func(request, *args, **kwargs)
            if response.status_code == 200:
                redis_client.setex(
                    cache_key,
                    ttl_seconds,
                    json.dumps(response.body, ensure_ascii=False),
                )
            return response
        return wrapper
    return decorator

常见问题（FAQ）

202 Accepted 返回后，如果 Worker 挂了，任务会丢失吗？
不会。如果 Worker 在处理过程中崩溃，MQ 消息会重新入队（ack_late 模式）并重新投递到其他 Worker。如果 MQ 本身不可用，消息会持久化到磁盘，恢复后重新消费。如果任务已经提交到 Worker 但结果未持久化，BFF 的任务状态会保持 processing，可以设置超时检测并自动重置为 pending。
Idempotency-Key 存储多久？
CorpAssist 默认存储 24 小时（TTL=86400s），这与大多数业务场景的重复请求时间窗口一致。幂等键存储太久会占用 Redis 空间，太短则可能在 TTL 过期后用户重试导致重复。建议根据业务特性调整 TTL：索引任务 7 天，查询类任务 1 小时。
Python Worker 如何回传结果给 Spring BFF？
三种方式：1) MQ 回调：Python Worker 完成后通过 MQ 发送事件，Spring 监听消费；2) Webhook 回调：Python Worker 直接调用 BFF 的 /api/v1/index/callback；3) DB 共享：Worker 写入共享数据库，BFF 查询。推荐 MQ 方式，松耦合且支持重试。
同步和异步可以混用吗？
可以。CorpAssist 中 RAG 问答使用同步 SSE，索引任务使用异步 MQ。一个请求可以拆分为同步部分（立即返回初步结果）和异步部分（后台继续处理）。例如文档上传：同步 202 返回 job_id，异步索引完成后通知。关键的决策因素是「用户是否在等待该结果」。

本章结论

  • 模式选择：< 30s 用同步/SSE，> 30s 用异步 202 + Job Status
  • MQ 解耦：RabbitMQ 作为 Spring-Python 异步通道，事件驱动业务编排
  • 幂等：Idempotency-Key 防重复提交，Redis 缓存已处理结果
  • 状态管理：Job 状态机（pending→processing→completed/failed）支持追踪

Breaking Change 检测

breaking detection 在 CI 中自动检测 OpenAPI 变更是否是破坏性的。

不推荐
  • 改了 API 不通知下游团队
  • 没有自动化契约测试
  • 线上才发现不兼容
  • 手动验证, 靠「口口相传」
推荐
  • CI 自动检测 breaking change
  • Pact 消费者驱动验证
  • Postman Schema 校验
  • CI Gate 阻断破坏性合并
变更类型是否 BreakingCI 处理
新增可选字段否通过
删除字段是阻断
字段类型变更（string→object）是阻断
新增 required 字段是阻断
修改端点路径是阻断
新增端点否通过
修改描述/注释否通过
常见误区
  • 契约只在联调时验证：应该在 CI 中每次 PR 都验证，而不是开发完成后才发现不兼容
  • 只测 200 成功路径：错误响应（400/401/429/5XX）也应该有契约验证
  • 忽略 Header 约束：X-Tenant-Id 等 Header 的缺失/错误也应该在契约测试中覆盖
  • 不测试 Python 的健壮性：Python 端收到不符合契约的请求时应该返回 400 而非 500

常见问题（FAQ）

Pact 和 openapi-diff 应该都用吗？还是二选一？
建议两者都用。openapi-diff 验证 OpenAPI 规范层面的 breaking change（静态分析），Pact 验证运行时的一致性（消费者实际调用的接口是否满足期望）。两者互补：openapi-diff 发现 Schema 问题，Pact 发现交互协议问题。
Pact 测试运行慢，可以只在新版本发布时跑吗？
不建议。Pact 测试应作为 CI 的一部分在每次 PR 中运行。如果觉得慢，可以：1) 只运行受影响的 Pact 文件（使用 pact-mock-service 的隔离模式）；2) 将 Pact 验证与单元测试并行；3) 使用 Pact 的 pending 标记（允许未验证的契约进入生产，但标记为待处理）。
Python 的 FastAPI 怎么自动验证请求/响应符合 OpenAPI Schema？
FastAPI 内置了 Pydantic 模型验证，但这只验证 Python 端的请求解析。对于完整的契约验证，建议：1) 使用 openapi-schema-validator 库在测试中验证响应；2) 使用 Pact 提供端测试从消费者角度验证；3) 在 CI 中用 Newman 端到端验证实际 HTTP 响应。
契约测试失败后应该阻断合并还是仅告警？
对于 breaking change（字段删除、类型变更），应该阻断合并（CI gate）。对于非破坏性变更（新增可选字段），可以通过但需要在 PR 中标注。CorpAssist 的策略是：breaking→阻断，非 breaking→通过但需要团队成员 review。

排障基础设施：全链路 Trace

distributed trace 是双栈排障的基础。每一次用户请求生成唯一的 tracing_id，贯穿 Spring Gateway → BFF → Python → LLM 的完整链路。

全链路 Trace：trace_id 跨服务不变，span_id 每跳变化，实现调用链追踪。

Spring Trace 过滤器

java
// Spring Trace 过滤器
@Component
public class TraceFilter implements WebFilter {

    private static final String TRACE_HEADER = "X-Request-Id";
    private static final String SPAN_HEADER = "X-Span-Id";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String traceId = exchange.getRequest().getHeaders()
            .getFirst(TRACE_HEADER);
        if (traceId == null || traceId.isBlank()) {
            traceId = UUID.randomUUID().toString();
        }

        String spanId = UUID.randomUUID().toString().substring(0, 8);

        // 注入 MDC（日志上下文）
        MDC.put("trace_id", traceId);
        MDC.put("span_id", spanId);

        // 追加到响应 Header
        exchange.getResponse().getHeaders()
            .add(TRACE_HEADER, traceId);

        return chain.filter(
            exchange.mutate()
                .request(exchange.getRequest().mutate()
                    .header(TRACE_HEADER, traceId)
                    .header(SPAN_HEADER, spanId)
                    .build())
                .build())
            .doFinally(signalType -> {
                MDC.clear();
            });
    }
}

Python Trace 中间件

python
# Python Trace 中间件
import structlog
from fastapi import Request
import uuid

logger = structlog.get_logger()

@app.middleware("http")
async def trace_middleware(request: Request, call_next):
    # 1. 获取透传的 trace_id
    trace_id = request.headers.get("X-Request-Id") or str(uuid.uuid4())
    span_id = str(uuid.uuid4())[:8]

    # 2. 注入日志上下文
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(
        trace_id=trace_id,
        span_id=span_id,
        service="python-rag",
        path=request.url.path,
    )

    # 3. 记录请求开始
    logger.info("request_start",
        method=request.method,
        query_params=dict(request.query_params))

    # 4. 执行请求
    start = time.time()
    response = await call_next(request)
    elapsed = time.time() - start

    # 5. 记录请求结束
    logger.info("request_end",
        status_code=response.status_code,
        elapsed_ms=round(elapsed * 1000, 2))

    # 6. 注入响应 Header
    response.headers["X-Request-Id"] = trace_id
    response.headers["X-Span-Id"] = span_id

    return response

常见问题（FAQ）

两边日志的 trace_id 对不上怎么办？
可能原因：1) Spring Gateway 没有生成 trace_id（检查 TraceFilter 是否配置正确）；2) Python 端的 X-Request-Id Header 被错误地赋值（检查中间件处理顺序）；3) BFF 到 Python 的 HTTP Client 没有透传 trace_id（检查 WebClient 配置）。排查方法：在 Gateway 打印收到的请求 Header，然后在 BFF 和 Python 逐跳打印收到的 X-Request-Id。
SSE 流式输出的 Agent 如何排障？
SSE 排障的核心是逐段检查：1) 先用 curl 直连 Python Agent，确认 SSE 流正常输出；2) 通过 BFF 代理调用，确认 BFF 正确转发 SSE 事件；3) 通过 Gateway 调用，确认 Gateway 没有截断 SSE 流；4) 最终从客户端验证。关键指标是每段 SSE 的 timing：如果 BFF 和 Python 之间的 SSE 有间隔但客户端没有收到，说明 BFF 缓存了 SSE。
如何排查「Python 返回正确但 Spring 解析失败」？
这是典型的DTO 不匹配。排查步骤：1) 用 curl 直连 Python 获取完整响应 JSON；2) 将 JSON 与 Spring 的 DTO 类字段逐一比对；3) 检查字段名大小写（JSON 是 camelCase 还是 snake_case）；4) 检查类型（Java Integer vs Python int，Java OffsetDateTime vs Python datetime）；5) 检查 OpenAPI 规范与两端的实现是否一致。参见 OpenAPI 与契约。
已经上线的系统出现偶发超时，如何定位？
偶发超时是最难排查的问题之一。建议：1) 在 BFF 添加 retry + 熔断，避免偶发超时影响用户体验；2) 在日志中记录每次调用的 P50/P95/P99 延迟；3) 对比 Python 服务本身的处理时间与网络传输时间（通过 timing Header 记录）；4) 使用 APM（如 SkyWalking、Datadog）分析慢调用的调用链。常见原因是 LLM API 偶发延迟、Milvus 向量检索排队、JVM GC 暂停。

// Spring Feature Flag 服务
@Service
public class FeatureFlagService {

    private final Map<String, FeatureFlag> flags;

    public boolean isEnabled(String featureKey) {
        FeatureFlag flag = flags.get(featureKey);
        if (flag == null) return false;
        return flag.enabled();
    }

    public boolean isInTraffic(String featureKey, String tenantId) {
        FeatureFlag flag = flags.get(featureKey);
        if (flag == null || !flag.enabled()) return false;

        // 白名单优先
        if (flag.tenantWhitelist() != null
            && flag.tenantWhitelist().contains(tenantId)) {
            return true;
        }

        // 百分比判定
        if (flag.trafficPercentage() >= 100) return true;
        int hash = Math.abs(tenantId.hashCode()) % 100;
        return hash < flag.trafficPercentage();
    }

    @RefreshScope
    @ConfigurationProperties(prefix = "corpassist.features")
    public void setFlags(Map<String, FeatureFlag> flags) {
        this.flags = flags;
    }

    public record FeatureFlag(
        boolean enabled,
        int trafficPercentage,
        List<String> tenantWhitelist,
        String fallbackBehavior
    ) {}
}

// BFF 灰度路由
@Service
public class RagRoutingService {

    private final FeatureFlagService featureFlags;
    private final PythonRagClient pythonRagClient;
    private final SpringRagService springRagService;

    public Mono<RagQueryResponse> routeQuery(
            RagQueryRequest request, String tenantId) {

        // 判断是否走新 Python 管线
        if (featureFlags.isInTraffic(
            "new-python-rag-pipeline", tenantId)) {
            return pythonRagClient.query(request, tenantId)
                .onErrorResume(ex -> {
                    log.warn("Python RAG failed, falling back to Spring RAG",
                        ex);
                    return springRagService.query(request, tenantId);
                });
        }

        // 默认走 Spring 旧版
        return springRagService.query(request, tenantId);
    }
}

# Python Feature Flag 客户端
from redis import Redis
import json
import hashlib

redis_client = Redis(host="redis", port=6379, db=0)

class FeatureFlagClient:
    """从配置中心读取 Feature Flag"""

    def __init__(self):
        self._cache = {}

    def is_enabled(self, feature_key: str) -> bool:
        flag = self._get_flag(feature_key)
        return flag.get("enabled", False) if flag else False

    def is_in_traffic(self, feature_key: str, tenant_id: str) -> bool:
        flag = self._get_flag(feature_key)
        if not flag or not flag.get("enabled", False):
            return False

        # 白名单
        whitelist = flag.get("tenant_whitelist", [])
        if tenant_id in whitelist:
            return True

        # 百分比
        percentage = flag.get("traffic_percentage", 0)
        hash_val = int(hashlib.md5(
            tenant_id.encode()).hexdigest(), 16) % 100
        return hash_val < percentage

    def _get_flag(self, key: str) -> dict:
        if key not in self._cache:
            data = redis_client.get(f"feature_flag:{key}")
            if data:
                self._cache[key] = json.loads(data)
        return self._cache.get(key, {})

    def refresh(self):
        """刷新缓存"""
        self._cache = {}

feature_flags = FeatureFlagClient()

回滚策略

stack rollback 是灰度发布中最重要的保障机制。CorpAssist 的 rollback plan 包含三个层次：

  1. Feature Flag 回滚：修改配置中心的 flag，秒级生效，从新 Python 路由切回 Spring
  2. Gateway 路由回滚：调整权重路由，将新版流量降为 0%
  3. 服务版本回滚：回退 Python 或 Spring 服务的 Docker 镜像到上一个版本
不推荐
  • 直接改代码来关闭功能
  • 灰度比例需要重启服务才能生效
  • 没有回滚计划，出问题才想怎么办
  • 回滚后不回填结果（幂等性丢失）
推荐
  • 配置中心动态管理 Feature Flag
  • 灰度比例支持运行时动态调整
  • 每次灰度前制定回滚计划
  • 新/旧版本都支持 Idempotency-Key

常见问题（FAQ）

Feature Flag 和 Gateway 权重路由有什么区别？
Feature Flag 是业务层的控制，在 BFF/Service 代码中判断，粒度更细（可按用户、租户、甚至请求内容）。Gateway 权重路由是基础设施层的控制，只按百分比分配流量。CorpAssist 同时使用两者：Gateway 做服务级别的流量切分，Feature Flag 做功能级别的灰度控制。
灰度期间如何监控新版本的健康状态？
关键指标包括：1) 错误率（4xx/5xx）—— 超过基线 1% 触发回滚；2) P95 延迟 —— 超过基线 50% 触发告警；3) 业务流程成功率 —— 比旧版本低触发回滚；4) 用户反馈 —— 灰度租户的投诉率。建议对接 APM（如 SkyWalking、Grafana）实现自动化监控。
如果新旧版本的 OpenAPI 契约不一致怎么办？
灰度期间必须保持新旧版本的契约完全兼容。如果新版本引入了 breaking change，需要升级大版本（如 v2→v3），通过 Gateway 路由到不同的前缀路径（/v2/rag/query vs /v3/rag/query）。灰度只灰度业务逻辑，不灰度接口规范。契约变更应走 OpenAPI 版本管理 流程。
如何让特定租户始终使用新版本？
两种方式：1) 在 Feature Flag 中配置 tenant_whitelist，白名单中的租户始终走新版本；2) 在 Gateway 中通过自定义谓词（如 X-Tenant-Id Header）将特定租户路由到新版。推荐方式 1，业务层控制更灵活且不依赖基础设施。

本章结论

  • 多策略灰度：Feature Flag + Traffic Split + Canary Route 组合使用
  • 动态控制：通过配置中心动态调整灰度比例，无需重启
  • 三层回滚：Flag 回滚（秒级）→ 路由回滚（分钟级）→ 镜像回滚（分钟级）
  • 监控驱动：每阶段灰度前制定指标基线，异常时自动回滚

发布列车策略

release train 是双栈团队的协同发布策略。Spring 和 Python 服务虽然独立开发，但需要协同发布以保证兼容性。

  • 双周发布列车：每隔两周周三，Spring 和 Python 同时发布
  • 契约冻结窗口：发布前 2 天，契约仓库禁止修改（只修 bug）
  • 版本对齐：每次发布使用统一的版本号（如 2026.05.31-rc1）
  • 热修复：紧急问题走 Hotfix 通道，单服务发布，不要求对齐

常见问题（FAQ）

AI 工程组需要修改 OpenAPI 契约，但平台架构组回复慢怎么办？
建议设置SLA 时间：非 Breaking 变更 24 小时内 review，Breaking 变更 48 小时内 review。如果架构组长期未响应，AI 组可以先在本地 fork 契约进行开发，同时标记 PR 为「等待 review」。关键指标是「契约 review 的平均时间」，团队应持续优化这个指标。
mono repo 和 multi repo 哪个更好？
没有绝对的好坏。CorpAssist 选择 multi repo 的原因是：双栈团队独立、发布节奏不同、权限需要隔离。如果团队规模小（< 10 人）、发布节奏一致，mono repo 更简单。关键在于契约仓库必须是独立的，无论 mono 还是 multi，契约都需要被双方共享和版本管理。
如何避免「Spring 改了 BFF 但 Python 不知道」？
三种机制：1) CI 中运行 契约测试，Python 修改 API 后 Spring 的 Pact 测试会失败；2) 定期同步会议（每周 15 分钟）同步接口变更；3) 契约仓库的每次合并自动通知双方团队的 IM 群组。
平台工程组和 AI 工程组如何划分 KPI？
平台组的 KPI 偏向基础设施：服务可用性（99.9%）、部署成功率、P95 延迟。AI 组的 KPI 偏向业务：RAG 准确率、Agent 完成率、用户满意度。共享 KPI：端到端可用性、接口错误率、发布频率。避免「平台组不管业务指标，AI 组不管系统指标」的单维度考核。

  • 团队分工：平台组（Spring 治理 + 基础设施）+ AI 组（Python 能力面），双方权责清晰

  • 契约所有权：架构组拥有契约，双方提 PR 修改，CI 自动检测 breaking change
  • 发布列车：双周协同发布 + 契约冻结窗口，保障版本兼容

常见问题（FAQ）

面试官问「你做过最复杂的系统是什么」，怎么回答？
推荐用 CorpAssist 双栈作为案例。回答模板：「我设计并实现了 CorpAssist 双栈架构，这是一个企业级 LLM 助手平台，采用 Spring（治理面）+ Python（能力面）双技术栈。我负责架构设计、OpenAPI 契约定义、双栈联调排障……」。重点突出你面临的挑战（双栈通信、契约同步、灰度发布）和解决方案。
面试官说「为什么不直接用 LangChain 全家桶？」
回答思路：1) LangChain 解决的是 AI 编排问题，不解决企业治理问题（鉴权、审计、限流）；2) 企业现有 Java 基础设施（Spring Cloud、Gateway、配置中心）不能扔掉；3) LangChain 的 Python 生态与 Spring 治理各司其职，不是替代关系而是互补。核心观点：没有银弹，双栈是当前企业级 LLM 应用的最优解。
面试官问「如果让你一个人做这个系统，你怎么做？」
真实回答：1) 个人做全栈优先选 Python（FastAPI + LangChain），减少 Java 负担；2) 用单服务 + 内部模块替代微服务架构；3) 等系统验证成功后再考虑拆分为双栈。关键要展示你知道什么时候该简化，而不是一直坚持复杂架构。
面试官问「怎么做容量估算和压力测试？」
回答框架：1) 估算 QPS —— 从业务数据推导（如 1000 企业 × 100 人 × 10 次/天 = 1M 次/天 ≈ 12 QPS）；2) 瓶颈分析 —— LLM API 调用（每个请求 2-5s，每秒最多 20 个并发请求）；3) 扩容策略 —— Python 水平扩展 + 限流保护 LLM API；4) 压测工具 —— Locust / JMeter 模拟真实用户请求。

常见问题（FAQ）

应急预案：1) 先重启失败服务（docker compose restart python-rag）；2) 如果仍然失败，检查依赖服务（如 Milvus）是否健康；3) 如果服务短期无法恢复，跳过该演示步骤，口头说明该功能已验证通过；4) 演示结束后修复并补录演示视频。建议演示前 30 分钟完成全量启动检查。
演示时网络不好，LLM 调用很慢怎么办？
可以先用Mock LLM演示：在 Python RAG API 中配置 mock_mode=true，返回预设的示例答案。演示开始时说明「这是用 mock 数据演示交互流程，真实场景下 LLM 调用需要 2-5 秒」。演示完交互流程后，可以切换到真实 LLM 展示一次端到端效果。
演示完 stakeholder 问「什么时候可以上线？」怎么回答？
回答框架：1) 已完成的功能（S1 RAG 问答）可以进入灰度发布流程，预计 2 周内小范围验证；2) Agent 功能（S2）还需要完成若干优化（如超时处理、转人工体验）后才能灰度；3) 给一个保守的时间线，留出缓冲；4) 明确需要 stakeholder 提供的支持（如灰度租户、预算审批等）。
stakeholder 问「这和 ChatGPT 有什么区别？」
关键差异：1) 数据安全 —— CorpAssist 部署在企业内网或私有云，数据不出域；2) 知识定制 —— 使用企业的知识库（制度文档、流程规范）而不是通用知识；3) 工具集成 —— 可以调用企业内部 API（查订单、查工单、查审批）；4) 审批流程 —— 支持转人工、操作审计等企业合规要求。总结：ChatGPT 是通用助手，CorpAssist 是企业专属的智能工作助手。

常见问题（FAQ）

自检中有些能力我还不太熟，怎么办？
这是正常现象。自检的目的是找到差距而不是追求完美。对于不熟的能力：1) 回顾对应的课程章节；2) 完成对应的 Demo 练习；3) 在真实项目中实践。建议将自检结果保存，3 个月后再做一次对比进步。
我主要用 Python，Java 部分不熟怎么办？
双栈课程的目标不是让你成为 Java/Python 的专家，而是理解两边的协作方式。如果你主要是 Python 开发者，至少需要理解：1) Spring 治理面的职责和边界；2) 双栈之间的通信方式（OpenAPI/JWT/Header）；3) Docker Compose 整体架构。具体到 Java 代码实现，能读懂即可。
毕业后还有什么课程可以继续学习？
推荐进阶路径：1) 上下文工程（context-memory-engineering）—— 深入学习 LLM 上下文窗口管理；2) RAG 评估（llm-evaluation-quality）—— 学习评估 RAG 系统的准确性；3) LLM 应用后端（llm-application-backend）—— 深入学习分布式治理和微服务；4) A2A/Nacos 集成（ai-agent-protocol）—— 学习 Agent 间通信协议。
自检表对找工作有什么用？
自检表可以直接转化为简历技能列表和面试回答框架。例如：JD 要求「熟悉 RAG 系统」→ 你在自检中可以找到「LangChain RAG 实现、向量数据库集成、RAG 索引问答分离」对应的知识和经验。在面试中展示自检表，能系统性地证明你的双栈能力。
