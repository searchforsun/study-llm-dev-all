为什么要学本章

单个 Agent 的能力有限。把复杂任务拆成多个角色（规划者、执行者、审核者）让每个子 Agent 专注自己的领域，可以大幅提升质量和可靠性。multi agent 编排是 SAA 中实现复杂业务自动化的关键模式。

学完你能

  • 通过 shared state 在 Agent 间传递上下文
  • 实现 handoff 模式在 Agent 间移交控制权
  • 为 CorpAssist 实现 Planner-Executor-Reviewer 三 Agent 流水线
本章先记住 3 件事
  • 拆分职责：每个 Agent 只做一件事，做精做好。
  • 共享状态：Agent 间通过 orchestration 上下文传递信息。
  • 编排模式：顺序执行、并行执行、路由分发——按需选择。

编排模式概览

orchestration 是 SAA 中编排 Agent 的核心抽象。提供三种基本模式：

三种编排模式
flowchart LR
    subgraph Sequential[顺序编排]
        A1[Agent A\n规划] --> A2[Agent B\n执行] --> A3[Agent C\n审核]
    end

    subgraph Parallel[并行编排]
        P0[分发] --> P1[Agent X\n数据分析]
        P0 --> P2[Agent Y\n文本书写]
        P0 --> P3[Agent Z\n格式校验]
        P1 --> P4[汇总]
        P2 --> P4
        P3 --> P4
    end

    subgraph Routing[路由编排]
        R0[路由器] -->|技术类| R1[技术支持 Agent]
        R0 -->|财务类| R2[财务 Agent]
        R0 -->|人事类| R3[HR Agent]
    end

三种编排模式适用于不同场景：顺序用于有依赖管线的任务，并行用于可独立执行的子任务，路由用于按类型分派。

模式场景CorpAssist 示例
Sequential 有依赖的管线任务 理解问题 → 查询知识库 → 生成回答
Parallel 可独立执行的子任务 同时查库存、查物流、查订单
Routing 按类型分发 工单分类 → 分配到对应支持组

Sequential：Pipeline 编排

sequential flow 将多个 Agent 串联成管线，前一个 Agent 的输出作为后一个 Agent 的输入。

java 
@Configuration
public class SequentialPipelineConfig {

    @Bean
    public SequentialOrchestrator corpassistPipeline(
            ReactAgent plannerAgent,
            ReactAgent executorAgent,
            ReactAgent reviewerAgent) {

        return SequentialOrchestrator.builder()
            .addStage("planner", plannerAgent)
            .addStage("executor", executorAgent)
            .addStage("reviewer", reviewerAgent)
            // 传递上下文字段映射
            .stateMappings(Map.of(
                "plan", "planner.output",
                "execution", "executor.output",
                "review", "reviewer.output"
            ))
            .build();
    }
}

// 使用
@RestController
public class PipelineController {

    private final SequentialOrchestrator pipeline;

    @PostMapping("/corpassist/pipeline")
    public Mono execute(@RequestBody TaskRequest task) {
        return Mono.fromFuture(pipeline.execute(task))
            .map(result -> new OrchestrationResult(
                result.getState("execution"),
                result.getState("review")
            ));
    }
}
CorpAssist · 三阶段任务处理 Planner Agent 接收员工需求并制定执行计划；Executor Agent 根据 Plan 调用具体工具（查询工单、发通知）；Reviewer Agent 检查 Executor 的输出是否符合合规要求。

Parallel：并行执行

parallel agent 模式将任务拆分为可并行执行的子任务，最后汇总结果。

java 
@Configuration
public class ParallelOrchestrationConfig {

    @Bean
    public ParallelOrchestrator employeeQueryOrchestrator(
            ReactAgent orderAgent,
            ReactAgent ticketAgent,
            ReactAgent attendanceAgent) {

        return ParallelOrchestrator.builder()
            .addAgent("order", orderAgent)
            .addAgent("ticket", ticketAgent)
            .addAgent("attendance", attendanceAgent)
            .aggregator(results -> {
                // 汇总三个 Agent 的结果
                String orderInfo = results.get("order");
                String ticketInfo = results.get("ticket");
                String attendanceInfo = results.get("attendance");
                return """
                    === 员工综合信息 ===
                    订单: %s
                    工单: %s
                    考勤: %s
                    """.formatted(orderInfo, ticketInfo, attendanceInfo);
            })
            .timeout(Duration.ofSeconds(15))
            .build();
    }
}

并行执行适用于：同时查询多个系统、同时处理多份文档、多维度数据分析。

注意：并行 Agent 各自有独立的上下文窗口。如果子任务需要共享大量上下文，考虑 Sequential 模式。并行模式的收益取决于子任务之间是否真的独立。

Routing：路由分发

routing agent 根据输入内容将任务分派给不同的下游 Agent。

java 
@Configuration
public class RoutingConfig {

    @Bean
    public RoutingOrchestrator corpassistRouter(
            ReactAgent techSupportAgent,
            ReactAgent financeAgent,
            ReactAgent hrAgent) {

        return RoutingOrchestrator.builder()
            .router(routerPrompt -> """
                根据用户输入判断应该路由到哪个 Agent：
                - 技术问题 → tech_support
                - 财务/报销问题 → finance
                - 人事/请假问题 → hr
                - 其他 → tech_support

                用户输入：
                %s
                """.formatted(routerPrompt))
            .route("tech_support", techSupportAgent)
            .route("finance", financeAgent)
            .route("hr", hrAgent)
            .build();
    }
}

// 使用
@PostMapping("/corpassist/router")
public Mono routeQuestion(@RequestBody String question) {
    return Mono.fromFuture(router.execute(question));
}
不推荐
  • 一个 Agent 做所有事，system prompt 非常长
  • 硬编码 if-else 路由不在 Agent 内
  • 没有 fallback Agent，未知类型无法处理
推荐
  • 每个 Agent 专注单一领域，prompt 简洁
  • 使用 Router Agent 做语义分发
  • 始终有默认 fallback Agent

Handoff：控制权移交

handoff 模式允许 Agent 在运行中将控制权交给另一个 Agent，适用于需要专长切换的场景。

java 
// Handoff 示意：前台 Agent 了解意图后移交
@Tool(name = "handover_to_finance", description = "将问题移交给财务专家处理")
public String handoverToFinance(String question) {
    return "[HANDOFF] 移交财务 Agent 处理: " + question;
}

@Tool(name = "handover_to_tech", description = "将问题移交给技术支持专家处理")
public String handoverToTech(String question) {
    return "[HANDOFF] 移交技术支持处理: " + question;
}

Handoff 的优势在于：用户面对一个统一的入口 Agent，复杂问题自动转交到专业 Agent，用户无需知道后台切换。

共享状态设计

shared state 是多 Agent 协作的核心。SAA 使用 OrchestrationState 管理跨 Agent 的数据流动。

java 
// 共享状态定义
public record CorpAssistState(
    String userId,
    String question,
    String plan,
    String executionResult,
    String reviewResult,
    Status status
) {
    public enum Status { PLANNING, EXECUTING, REVIEWING, COMPLETED, FAILED }
}

// Agent 间通过状态对象传递信息
@Bean
public SequentialOrchestrator statefulPipeline(
        ReactAgent planner, ReactAgent executor, ReactAgent reviewer) {

    return SequentialOrchestrator.builder()
        .initialState(CorpAssistState::new)
        .addStage("planner", planner, (state, output) -> state.withPlan(output).withStatus(EXECUTING))
        .addStage("executor", executor, (state, output) -> state.withExecutionResult(output).withStatus(REVIEWING))
        .addStage("reviewer", reviewer, (state, output) -> state.withReviewResult(output).withStatus(COMPLETED))
        .build();
}

常见问题（FAQ）

多 Agent 编排会增加多少延迟？
每个 Agent 调用一次模型。Sequential N 个 Agent 延迟 ≈ N × 单 Agent 延迟。Parallel 模式下延迟 ≈ 最慢子 Agent 延迟。建议：对延迟敏感的场景减少 Agent 数量或将非关键 Agent 异步化。
子 Agent 失败会影响整个管线吗？
默认会。建议为每个子 Agent 设单独的超时和 fallback。失败时可进入"降级模式"：如 Executor 执行失败，管线可跳过 Review 直接进入 Error 处理流程。
Routing Agent 的路由准确率如何？
取决于路由 prompt 的清晰度和模型的智能水平。使用 qwen-max 路由准确率在 90%+。建议在路由指令中给出明确的分类条件和示例。

本章结论

  • 三种编排模式：Sequential（管线）、Parallel（并行）、Routing（路由）。
  • Handoff：控制权移交，实现专长切换。
  • 共享状态：通过 OrchestrationState 在 Agent 间传递上下文。
  • 角色分工：Planner-Executor-Reviewer 是经典的分工模式。
  • 容错设计：每个 Agent 设超时和 fallback，防止级联失败。
