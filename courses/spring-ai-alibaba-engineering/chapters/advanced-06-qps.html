为什么要学本章

Agent 调用链很长：用户请求 → 路由 Agent → Graph 多步 → 多次 LLM 调用 → 多次工具调用。每一步的延迟和吞吐都会积累。生产环境需要从 connection pool、async agent、cache tier 三个维度做 QPS 优化。

学完你能
  • 配置 HTTP 连接池和线程池优化 Agent 吞吐
  • 实现 async agent 非阻塞执行
  • 设计 peak shaving 削峰策略
  • 从 non infra view 角度优化 QPS
本章先记住 3 件事
  • 连接池先于应用优化：LLM API 调用依赖 HTTP 连接池，不当配置是最大瓶颈。
  • 异步是吞吐关键：使用虚拟线程或 Reactor 实现非阻塞 Agent 执行。
  • 缓存分层降 QPS：语义缓存命中一次 = 省下一次 LLM 调用。

连接池优化

connection pool 是 Agent 调用 LLM API 的底层瓶颈。DashScope 的 HTTP 连接池配置直接影响并发上限。

HTTP 连接池配置

yaml 
spring:
  ai:
    dashscope:
      chat:
        options:
          model: qwen-max
  # HTTP 客户端连接池
  reactor:
    netty:
      pool:
        max-connections: 500
        acquire-timeout: 5000
        max-idle-time: 30000

# WebClient 配置
corpassist:
  http-client:
    pool:
      max-connections: 500
      max-pending-acquire: 1000
      pending-acquire-timeout: 5s
    connect-timeout: 3s
    read-timeout: 30s
    write-timeout: 10s

配置化 Connection Manager

java 
@Configuration
public class HttpClientConfig {

    @Bean
    public WebClient dashscopeWebClient(HttpClient httpClient) {
        return WebClient.builder()
            .clientConnector(new ReactorClientHttpConnector(httpClient))
            .baseUrl("https://dashscope.aliyun.com")
            .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
            .build();
    }

    @Bean
    public HttpClient dashscopeHttpClient() {
        return HttpClient.create()
            .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 3000)
            .responseTimeout(Duration.ofSeconds(30))
            .doOnConnected(conn ->
                conn.addHandlerLast(new ReadTimeoutHandler(30))
            )
            .metrics(true, () -> new MicrometerChannelMetricsRecorder("dashscope", "http"));
    }
}
连接池优化要点
  • max-connections 至少要大于预期并发 × Agent 步数（如 100 QPS × 5 步 = 500）
  • 必须设 acquire-timeout，否则线程在等待连接时永久阻塞
  • 启用 HTTP 指标监控连接池使用率，超过 70% 时扩容

虚拟线程与异步 Agent

async agent 使用虚拟线程（Java 21+）或 Reactor 实现非阻塞执行。thread pool 配置不当是 QPS 瓶颈的另一来源。

yaml 
# 启用虚拟线程（Java 21+）
spring:
  threads:
    virtual:
      enabled: true

# Agent 线程池配置
corpassist:
  agent:
    thread-pool:
      # 不使用虚拟线程时，自定义线程池
      core-size: 50
      max-size: 200
      queue-capacity: 500
      keep-alive: 60s
java 
// 异步 Agent 执行器
@Service
public class AsyncAgentExecutor {

    private final GraphExecutor graphExecutor;

    public AsyncAgentExecutor(GraphExecutor graphExecutor) {
        this.graphExecutor = graphExecutor;
    }

    /**
     * 异步执行 Graph，不阻塞请求线程
     */
    public CompletableFuture executeAsync(String graphId, Object state) {
        return CompletableFuture.supplyAsync(() -> {
            try {
                return graphExecutor.execute(graphId, state).get();
            } catch (Exception e) {
                throw new RuntimeException("Agent execution failed", e);
            }
        });
    }

    /**
     * 批量异步执行（并行工具调用场景）
     */
    public List> executeBatch(List tasks) {
        return tasks.stream()
            .map(task -> executeAsync(task.graphId(), task.state()))
            .toList();
    }
}

// Controller 中异步处理
@PostMapping("/corpassist/batch")
public Flux batchProcess(@RequestBody List requests) {
    List> futures = requests.stream()
        .map(req -> asyncExecutor.executeAsync(req.graphId(), req.toState()))
        .toList();

    return Flux.fromIterable(futures)
        .flatMap(Mono::fromFuture);
}

QPS 优化：非 Infra 视角

non infra view 强调从 Agent 设计层面优化 QPS，而不是只堆基础设施。

不推荐
  • 只堆机器不优化应用
  • Agent 步数不加控制
  • 每次请求都走完整 LLM 调用
  • 同步阻塞执行
推荐
  • 先优化应用再扩容
  • 控制 maxSteps 减少不必要的推理
  • 语义缓存 + 模型分级减负
  • 异步 + 虚拟线程释放资源

削峰策略

peak shaving 使用消息队列缓冲突发的 Agent 请求。

yaml 
corpassist:
  peak-shaving:
    enabled: true
    queue: rocketmq  # 或 rabbitmq / kafka
    topic: corpassist-agent-tasks
    consumer-group: agent-consumer
    max-concurrent: 20   # 最大并发消费数
    queue-size: 10000    # 队列最大长度
java 
// RocketMQ 消费者异步执行 Agent
@Component
@RocketMQMessageListener(
    topic = "corpassist-agent-tasks",
    consumerGroup = "agent-consumer"
)
public class AgentTaskConsumer implements RocketMQListener {

    private final GraphExecutor graphExecutor;

    @Override
    public void onMessage(AgentTask task) {
        // 异步执行 Graph，不阻塞消息线程
        CompletableFuture.runAsync(() -> {
            try {
                graphExecutor.execute(task.graphId(), task.state());
            } catch (Exception e) {
                log.error("Agent execution failed: {}", task.taskId(), e);
                // 发送到死信队列
            }
        });
    }
}

batch tool 将多次工具调用合并为一次批量调用，显著减少步数：

java 
@Tool(name = "batch_query", description = "批量查询员工信息，输入多个员工 ID 用逗号分隔")
public String batchQuery(String employeeIds) {
    List ids = Arrays.asList(employeeIds.split(","));
    // 一次批量查询替代多次独立查询
    List employees = employeeService.batchFind(ids);
    return employees.stream()
        .map(e -> "%s: %s, %s".formatted(e.empId(), e.name(), e.department()))
        .collect(Collectors.joining("\n"));
}

缓存分层设计

cache tier 将缓存分为三层：

优化维度做法QPS 提升
减少步数 合并多次工具调用为批量 API 2-3x
语义缓存 高频问题缓存，跳过 LLM 5-10x (命中时)
模型分级 简单问题用 qwen-turbo 2x
并行工具 独立的工具调用并行执行 2-5x
异步释放 请求线程不阻塞等 Agent 3-5x
层级缓存内容存储TTL命中率
L1: 本地缓存 高频 FAQ 答案 Caffeine 5 min ~20%
L2: 语义缓存 语义相似的问题 Redis 1 h ~30%
L3: 结果缓存 精确 match 的中间结果 Redis 10 min ~10%
java 
@Service
public class TieredCacheService {

    private final Cache localCache;     // Caffeine L1
    private final SemanticCacheService semanticCache;   // Redis L2
    private final RedisTemplate resultCache; // L3

    public TieredCacheService() {
        this.localCache = Caffeine.newBuilder()
            .maximumSize(1000)
            .expireAfterWrite(Duration.ofMinutes(5))
            .recordStats()
            .build();
    }

    public Optional get(String userMessage) {
        // L1: 本地缓存
        String l1Result = localCache.getIfPresent(userMessage);
        if (l1Result != null) return Optional.of(l1Result);

        // L2: 语义缓存
        Optional l2Result = semanticCache.getCached(userMessage);
        if (l2Result.isPresent()) {
            localCache.put(userMessage, l2Result.get()); // 提升到 L1
            return l2Result;
        }

        // L3: 精确结果缓存
        String l3Key = "result:" + DigestUtils.md5Hex(userMessage);
        String l3Result = resultCache.opsForValue().get(l3Key);
        if (l3Result != null) {
            localCache.put(userMessage, l3Result);
            return Optional.of(l3Result);
        }

        return Optional.empty();
    }
}

常见问题（FAQ）

连接池设多大合适？
公式：maxConnections = 目标 QPS × 每个 Agent 的平均 LLM 调用次数 × 平均响应时间（秒）。例如：50 QPS × 5 次 LLM 调用 × 2 秒 = 500。再加 20% 余量 = 600。
虚拟线程和平台线程在 Agent 场景的性能差异？
虚拟线程的优势在 IO 密集型场景。Agent 调用 LLM 是 IO 等待（占 90%+ 时间），虚拟线程能将线程数从 200 提升到数千而不增加内存开销。实测虚拟线程 + Reactor Netty 可提升 3-5x 吞吐。
削峰队列满了怎么办？
队列满时应立即拒绝（而非阻塞）并返回"当前请求量较高，请稍后重试"。设置队列监控告警，使用率 > 80% 时触发扩容流程。
QPS 上不去，Log 显示连接池都在等待，怎么快速排查？
检查 (1) max-connections 是否太小；(2) acquire-timeout 是否太短导致误判；(3) 是否有 TLS 握手瓶颈。启用 HTTP 客户端 metrics 查看连接池状态。生产建议：连接池是第一个查询点。

本章结论

  • 连接池优先：正确的 HTTP 连接池配置是 QPS 提升的第一步。
  • 异步释放：虚拟线程 + Reactor 非阻塞执行提升吞吐。
  • 非 Infra 优化：减少步数、缓存、模型分级比堆机器更有效。
  • 削峰保护：消息队列缓冲突发流量，防止 LLM 限流。
  • 三层缓存：L1 Caffeine + L2 语义 + L3 结果，逐层降 QPS。
