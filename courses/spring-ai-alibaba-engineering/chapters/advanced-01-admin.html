为什么要学本章

saa admin 是 SAA 的管理控制台，提供可视化的 Graph 编排、Prompt 管理、运行监控等功能。平台组和运维组通过 Admin 管理 Agent 的生命周期，无需直接修改代码。

学完你能
  • 使用 agent studio 可视化的编辑 Graph 工作流
  • 使用 workflow dsl 导出/导入工作流定义
  • 管理 Prompt 版本和发布
  • 理解 SAA Admin 与 Dify 的差异，制定迁移路径
本章先记住 3 件事
  • Admin 不是 IDE：Admin 用于运行时管理，开发仍然在 IDE 中进行。
  • DSL 即代码：工作流 DSL 应纳入 Git 版本管理。
  • 平台化是趋势：从 Dify 到 SAA Admin 是从低代码到可编程的演进。

Admin 平台的功能分层：Studio 负责编排，Prompt 管理负责提示词，监控负责运行时，发布管理负责 CI/CD。

CorpAssist · 平台团队工作模式 平台工程师在 Admin Studio 中编辑请假审批 Graph，保存后导出 DSL 提交 Git PR。运维审批后，GitOps 自动将新 Graph 部署到 staging 环境，A/B 测试 10% 流量后再全量发布生产。

Agent Studio 可视化编排

agent studio 提供 SaaS 化的拖拽编排体验。与代码定义等价：

json 
{
  "graph": {
    "id": "leave-approval-v2",
    "version": "2.1.0",
    "nodes": [
      {"id": "validate", "type": "function", "handler": "validateLeave"},
      {"id": "manager_approval", "type": "human_approval",
       "config": {"approvers": ["manager"], "timeout": "24h"}},
      {"id": "hr_check", "type": "agent", "model": "qwen-max",
       "prompt": "请检查员工的请假记录和配额"},
      {"id": "notify", "type": "mcp_tool", "server": "oa-mcp",
       "tool": "send_notification"},
      {"id": "reject", "type": "function", "handler": "sendRejection"}
    ],
    "edges": [
      {"from": "validate", "to": "manager_approval"},
      {"from": "manager_approval", "to": "hr_check",
       "condition": "approved"},
      {"from": "manager_approval", "to": "reject",
       "condition": "rejected"},
      {"from": "hr_check", "to": "notify",
       "condition": "approved"},
      {"from": "hr_check", "to": "reject",
       "condition": "rejected"},
      {"from": "reject", "to": "notify"}
    ]
  }
}

workflow dsl 是本课程用的 JSON 格式，Studio 中编辑的内容导出后就是这段 JSON，可直接提交到 Git 进行版本管理。

Prompt 管理与 A/B 测试

prompt mgmt 提供 Prompt 版本控制、灰度发布和 A/B 测试能力。

json 
{
  "prompt": {
    "id": "corpassist-system-v3",
    "versions": [
      {
        "version": "3.0.0",
        "content": "你是 CorpAssist 客服助手。请基于企业知识库回答。",
        "status": "production",
        "traffic": 100
      },
      {
        "version": "3.1.0-beta",
        "content": "你是 CorpAssist 客服助手。请用结构化格式回答。",
        "status": "candidate",
        "traffic": 5
      }
    ]
  }
}
java 
// 从 Admin 配置中心加载 Prompt
@Service
public class DynamicPromptService {

    private final ConfigCenter configCenter;

    public DynamicPromptService(ConfigCenter configCenter) {
        this.configCenter = configCenter;
    }

    public String getSystemPrompt(String agentId) {
        // 从 Admin 配置中心读取当前生效的 Prompt
        return configCenter.getConfig("prompt." + agentId + ".system", String.class);
    }
}

visual editor 还支持编辑节点的 System Prompt，双击节点即可修改，所见即所得。

发布管理与多环境

graph publish 支持 Graph 和 Prompt 的灰度发布。

环境Graph 版本Prompt 版本流量
Dev latest latest 内部开发
Staging v2.1.0-rc1 3.1.0-beta, 5% 内部测试 + 5% 生产流量
Production v2.0.0 3.0.0, 100% 全量
不推荐
  • 直接在 Admin Studio 中修改生产环境的 Graph
  • Prompt 改了不测试直接全量发布
  • Graph 导出后不提交 Git
推荐
  • Studio 修改后导出 DSL → Git PR → 审批 → 发布
  • Prompt 变更先灰度 5% 流量验证
  • DSL 文件纳入 Git 版本管理

从 Dify 迁移到 SAA Admin

dify migrate 是团队从 Dify 低代码平台迁移到 SAA 可编程平台的常见路径。

能力DifySAA Admin
工作流编排 拖拽 + YAML DSL 拖拽 + JSON DSL（可版本管理）
Prompt 管理 在线编辑，无版本 版本管理 + A/B 测试
模型接入 多模型路由 DashScope + 多模型（同能力）
工具集成 自定义工具 API MCP 标准协议
代码扩展 Python 函数 Java @Tool + Graph Node
Git 集成 不支持 DSL 导出 + Git 版本管理

迁移建议：从简单的审批流程开始，逐步将 Dify 应用替换为 SAA Graph，MCP 工具可重用现有 Dify 的自定义工具 API。

常见问题（FAQ）

Admin Studio 中编排的 Graph 怎么和代码协同？
推荐"DSL 优先"模式：在 Studio 中可视化编排 → 导出 DSL JSON → 提交到 Git → CI 自动部署。代码中的 @GraphNode 注解方法作为节点实现，DSL 只定义拓扑结构。
可以只写代码不用 Admin Studio 吗？
完全可以。Graph Builder API 和 Admin Studio 生成的是等价的 DSL。小团队可直接用代码；平台团队建议用 Studio 降低沟通成本。
Dify 的已有工作流能自动迁移到 SAA 吗？
没有自动迁移工具。需要手动将 Dify 工作流的节点映射到 SAA Graph 的节点和条件边。建议先从简单的线性流程开始迁移。
Admin 的权限管理怎么做？
SAA Admin 集成 Spring Security，支持 RBAC。可配置：开发者只能编辑 dev 环境，平台工程师可管理 staging，运维可发布 production。

本章结论

  • Admin 平台能力：Studio 编排、Prompt 管理、运行监控、发布管理。
  • DSL 即代码：工作流定义应纳入 Git 版本管理。
  • Prompt 工程：版本管理 + A/B 测试提升质量。
  • 灰度发布：支持按百分比灰度测试新版本。
  • Dify 迁移：从低代码到可编程，MCP 工具可复用。
