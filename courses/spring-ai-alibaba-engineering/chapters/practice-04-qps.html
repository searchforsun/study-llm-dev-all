为什么要学本章

面试常问「S3 Agent 如何扛 500 QPS」——答案不应是「加 Pod」这么简单。LLM 调用慢、串行 Tool 多、同步 Servlet 占线程，都会让应用层先触顶。http connection pool、async llm 与 peak shaving 是 Java 工程师能直接落地的手段；本章刻意非 Infra 视角：不谈 GPU 集群，谈 Spring 侧如何把有限 DashScope QPS 用满。

学完你能
  • 调优 DashScope HTTP Client 连接池（max connections、keep-alive、超时）
  • 将长任务（OCR、批量汇总）改为 @Async / 消息队列 + SSE 通知
  • 用队列削峰：高峰订会议室请求排队而非打满 DashScope
  • 设计 L1/L2 缓存分层，与 practice-02 语义缓存衔接
  • 用压测指标（有效 QPS、线程池饱和度）回答 S3 面试题
本章先记住 3 件事
  • 瓶颈常在串行：intent + clarify + reply 三次 LLM 可合并或缓存意图。
  • 线程 ≠ 连接：Tomcat 200 线程但连接池 50，仍会排队。
  • 异步改体验：长任务返回 taskId，比同步 30s 超时更好。

连接池：DashScope HTTP 出口

Spring AI Alibaba 底层通过 HTTP 调百炼 API。默认连接池过小会导致 QPS 上升时大量等待连接；过大则占满 DashScope 侧配额。应在应用配置显式声明 http connection pool，并与 Resilience4j 超时联调。

CorpAssist · S3 · 早高峰 500 并发短问 FAQ，P99 从 2s 飙到 12s。排查发现 OkHttp maxRequests 默认 64，连接等待占 8s。调池后 P99 回到 3s——无需先扩 Pod。
yaml
corpassist:
  dashscope:
    http:
      max-connections: 200
      max-connections-per-route: 100
      connect-timeout-ms: 2000
      read-timeout-ms: 8000
      keep-alive-duration-ms: 300000
参数调大效果S3 风险
max-connections减少等连接超过 DashScope 账号 QPS 仍 429
read-timeout允许慢模型占连接过久，需配合异步
keep-alive降 TLS 握手长连接与 LB 空闲断开需对齐

自检：连接池 200 仍 429，下一步做什么？

应用层：语义缓存 + 合并 LLM 调用 + 削峰队列；配额层：申请提额或多 Key 分流。非 Infra 视角下仍应先证明无串行浪费。

异步：长任务与 Servlet 线程释放

S3 中「批量 OCR 发票」「生成月度报销摘要」远超 8s。async llm 模式：API 立即返回 taskId，Worker 消费队列调 DashScope，完成通过 SSE / 企微推送。同步 Servlet 只负责受理与鉴权。

java
@PostMapping("/s3/expense/summary")
public TaskAccepted submitSummary(@RequestBody SummaryRequest req) {
  String taskId = taskQueue.enqueue(req);
  return new TaskAccepted(taskId, "处理中，完成后通知");
}

@Async("s3LlmExecutor")
public void processSummary(String taskId, SummaryRequest req) {
  String result = chatClient.prompt().user(buildPrompt(req)).call().content();
  taskStore.complete(taskId, result);
  notifier.send(req.employeeId(), result);
}
路径同步 / 异步理由
FAQ 短问同步 + 缓存<3s，体验直接
订会议室多轮同步 SSE 流式逐 token 反馈
发票 OCR 批次异步队列避免占满 Tomcat
审批摘要生成异步 + 回调可重试、可审计
不推荐
  • 30s 同步阻塞等 OCR+LLM
  • @Async 无界线程池
  • 队列无 DLQ，失败任务丢失
推荐
  • 有界队列 + 拒绝策略返回 503+retry-after
  • Worker 与 API 分模块扩缩
  • task 状态可查询 GET /tasks/{id}

削峰：队列、令牌桶与合并请求

peak shaving 把瞬时 800 QPS 摊平到 DashScope 允许的 200 QPS：入口网关令牌桶、应用内 BlockingQueue、或对相同 FAQ 做 request coalescing（合并相同 query 的一次 LLM 调用）。

CorpAssist · 制度 FAQ 风暴 HR 发全员邮件后 1 分钟内 3000 人问「年假新规」。L1/L2 缓存 + singleflight 将 LLM 调用压到 <50 次；未命中排队 120s 内消化，DashScope 无 429。

非 Infra 视角：工程师能答的 S3 QPS 题

面试不要求你调 K8s HPA，而要求说清有效 QPS = 成功完成业务请求的速率，而非 HTTP 入口计数。下列指标应在 Micrometer 仪表盘可见（见 practice-03）。

有效 QPS

完成 S3 业务闭环（含 Tool）/s；排除仅返回「排队中」。

LLM 调用比

每业务请求平均 LLM 次数；目标 FAQ <1.2，复杂 Agent <3。

连接等待 P99

等 HTTP 连接时长；>500ms 先调池再扩 Pod。

缓存命中率

L1+L2 命中 / 只读请求；制度类目标 >60%。

压测清单（应用层）
  • Mock DashScope，测连接池与线程池饱和点
  • 混合 70% FAQ + 20% 订会议室 + 10% 异步 OCR
  • 记录 429、排队丢弃率、P95 延迟
  • 对比「无缓存 vs L1/L2 vs 削峰队列」三条曲线
何时才需要 Infra 扩 Pod？
当连接池、缓存、异步化已优化，CPU 仍 >70% 且 LLM 调用比合理，或 Worker 队列持续积压。应用层先证无浪费，再要资源。
流式 SSE 是否更占连接？
单请求占用连接时间更长，但用户体验更好。S3 订会议室等多轮场景用 SSE；纯 FAQ 用非流式 + 缓存。连接池 sizing 需按并发流式连接数单独估算。

// singleflight：相同 FAQ key 合并一次 LLM
public CompletableFuture<String> askFaq(String normalizedQuery) {
  return inflight.computeIfAbsent(normalizedQuery, key ->
      CompletableFuture.supplyAsync(() -> llmCall(key), faqExecutor)
          .whenComplete((r, e) -> inflight.remove(key)));
}

通义栈视角：配额隔离与削峰

Python 批量任务（索引、实验）不应与 Java 在线 S3 API 争用同一 DashScope Key 配额。在线轨优先 cache + singleflight + 异步；离线轨优先 batch 窗口与队列——配额隔离是双栈共存前提。

不推荐
  • dev / 在线 / 批处理共用一把 Key，429 互相拖累
  • 未算「有效 QPS」就要求扩 Pod
  • 写路径走缓存，导致脏读
推荐
  • 分 Key + 分连接池：在线 vs 批处理
  • 用 LLM 调用比、连接等待、命中率证明瓶颈
  • 只读 FAQ / availability 可缓存；写 Tool 强制 bypass

本章结论

  • 连接池：显式 max-connections / timeout；区分线程池与连接池瓶颈。
  • 异步：OCR/摘要走 taskId + 队列；同步路径留给短 FAQ 与 SSE。
  • 削峰：网关限流 + L1/L2 缓存 + singleflight + 排队反馈。
  • 非 Infra 视角：用有效 QPS、LLM 调用比、连接等待、命中率叙事，再谈扩 Pod。

面试问答

S3 办公 Agent 如何从 100 QPS 提到 500 QPS（不谈加机器）？

答：① L1/L2 缓存制度 FAQ，降 LLM 调用比；② 调 HTTP 连接池与超时；③ 意图+回复合并减少轮次；④ 高峰 singleflight + 令牌桶削峰；⑤ 长任务异步化释放线程；⑥ turbo 降级保可用。用压测证明有效 QPS 与 429 率。

连接池已经 200，延迟仍高，你还查什么？

答：Trace 看是 LLM 慢还是 Tool 慢；LLM 调用比是否过高；线程是否在等连接（metrics）；DashScope 429 重试风暴；必要时异步化或缓存，而非继续加连接。
