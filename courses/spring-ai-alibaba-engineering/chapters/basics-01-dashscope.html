为什么要学本章

CorpAssist S3 办公 Agent 的 Java 轨以通义千问为默认模型。若不清楚 dashscope api key 如何分环境注入、qwen model 如何选型，后面 Agent 编排 · SAA 版agent-orchestration-java 的 ReactAgent 与 streaming flux 都会变成「本地能跑、上线报错」。本章把密钥 → 模型 → 流式 → 计费一次讲透。

学完你能
  • 在 bailian console 创建 Key，并用环境变量注入 Spring Boot，不写死在源码
  • 为 S3 Agent 选择 qwen-plus / qwen-turbo 等模型，并说明与 Spring AI 工程化spring-ai-engineering OpenAI 兼容模式的差异
  • 用 ChatClient 发起流式响应，并估算 token billing 对 Agent 多步调用的成本影响
本章先记住 3 件事
  • 密钥外部化：生产 Key 只来自环境或密钥服务，禁止提交 Git。
  • 模型按场景选：Agent 规划用 plus，简单 FAQ 用 turbo；多模态另选 VL 系列。
  • 流式优先：门户 SSE 体验依赖 Flux；同步 call 仅用于批处理或测试。
预览 · RAM 子账号：本章 API Key 仅覆盖 dev/staging/prod 分环境与注入方式；RAM 子账号最小权限、Key 轮换、配额审计与多租户隔离的完整治理见 practice-05 · 百炼安全与合规治理。

API Key

DashScope（百炼 model studio）通过 API Key 鉴权。Spring AI Alibaba 1.1.x 约定环境变量 AI_DASHSCOPE_API_KEY 或配置项 spring.ai.dashscope.api-key。CorpAssist S3 模块与 Python 实验服务应使用不同子账号 Key，便于配额与审计分账。

来源配置方式CorpAssist 建议
百炼控制台 创建 RAM 子账号 Key，绑定「模型调用」权限 dev / staging / prod 各一把 Key
环境变量 AI_DASHSCOPE_API_KEY K8s Secret / CI 注入，本地 .env 不入库
application.yml spring.ai.dashscope.api-key: ${AI_DASHSCOPE_API_KEY} 与 corpassist s3 Agent 服务共用前缀
yaml
# corpassist-agent-office/application.yml
spring:
  application:
    name: corpassist-s3-agent
  ai:
    dashscope:
      api-key: ${AI_DASHSCOPE_API_KEY}
      chat:
        options:
          model: ${CORPASSIST_AGENT_MODEL:qwen-plus}
          temperature: 0.3
不推荐
  • 把 Key 写在 Java 常量或 application-prod.yml 提交 Git
  • dev 与 prod 共用同一 Key，无法按环境限流
  • 在日志中打印完整 api-key
推荐
  • 环境变量 + Spring 占位符引用
  • RAM 子账号最小权限；定期轮换 Key
  • 日志只记录 Key 后 4 位与 request_id
CorpAssist · S3 办公 Agent · 密钥分账 S3 Agent 服务（Java）与批量评测脚本（Python）各用独立 Key：Agent 在线 QPS 高、Token 消耗大；评测 Key 可设更低配额，避免压测误伤生产额度。
先想 10 秒：为何不用 OpenAI 的 OPENAI_API_KEY 命名？

Spring AI Alibaba DashScope Starter 默认读取 AI_DASHSCOPE_API_KEY；与 Spring AI 工程化spring-ai-engineering 的 OpenAI 兼容轨并行时，两套 Key 互不覆盖，便于双栈网关切换。

模型列表

百炼提供通义千问系列：turbo（低延迟）、plus（Agent 推理）、max（复杂规划）、vl-plus（图文）。S3 办公 Agent 默认 qwen-plus；工具调用密集时可升 max，简单意图分类可降 turbo。

模型典型场景CorpAssist S3 用法
qwen-turbo低延迟、低成本 FAQ意图路由、槽位抽取
qwen-plus平衡质量与成本ReactAgent 主模型（默认）
qwen-max复杂多步规划跨系统审批链、长上下文摘要
qwen-vl-plus图文理解扫描件请假单、发票 OCR（extensions 章展开）
DashScope 原生

通过 spring.ai.dashscope.* 由 SAA Starter 注册 DashScopeChatModel。

OpenAI 兼容模式

compatible mode：https://dashscope.aliyuncs.com/compatible-mode/v1，可与 spring-ai-openai Starter 共用（见 basics-04 对照）。

Embedding

text-embedding-v3 用于制度库 RAG；与 Agent 对话模型分开配置。

java
@RestController
@RequestMapping("/api/agent")
public class OfficeAgentController {

  private final ChatClient agentChat;

  public OfficeAgentController(ChatClient.Builder builder) {
    this.agentChat = builder
        .defaultSystem("你是 CorpAssist S3 办公助手，工具调用须遵守审批策略。")
        .build();
  }

  @PostMapping("/chat")
  public String chat(@RequestBody ChatRequest req) {
    return agentChat.prompt()
        .user(req.message())
        .call()
        .content();
  }
}

流式

企业门户期望「打字机」体验：Spring AI 的 ChatClient 返回 Flux<ChatResponse>，Controller 用 text/event-stream 推送。S3 Agent 在 ReactAgent 多步循环中，最终回答宜流式；中间 Thought 可只写 Trace 不落 SSE。

S3 门户 SSE 流式路径
sequenceDiagram
  participant UI as 企业门户
  participant API as corpassist-s3-agent
  participant DS as DashScope
  UI->>API: POST /chat/stream
  API->>DS: stream chat (qwen-plus)
  loop 每个 chunk
    DS-->>API: delta token
    API-->>UI: SSE data
  end
  API-->>UI: event: done

流式只暴露用户可见内容；Agent 内部工具 Observation 走结构化日志（practice-03 观测章）。

java
@PostMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> stream(@RequestBody ChatRequest req) {
  return agentChat.prompt()
      .user(req.message())
      .stream()
      .content()
      .map(chunk -> ServerSentEvent.builder(chunk).build());
}
不推荐
  • Agent 每步工具结果也 SSE 给前端（泄露内部推理）
  • 同步 .call() 等全量 Token 再返回（门户超时）
推荐
  • 最终答案 Flux；中间步骤进 Trace
  • 网关设置 SSE 超时与心跳
CorpAssist · 请假 Agent · 流式答复 用户问「帮我查年假并提交请假」：Agent 内部 3 次工具调用同步完成；门户仅流式展示「您剩余 5 天年假，已为您起草申请…」最终文案。

计费

百炼按 输入 Token + 输出 Token 计费，Agent 多轮对话会放大输入侧（历史 + 工具 Observation）。token billing 需在架构层设 maxSteps、上下文压缩（见 context-memory 课）与模型降级策略。

成本驱动S3 Agent 表现治理手段
多步 ReAct每步重发完整历史maxSteps=12；超步降级 turbo
长制度引用RAG 片段进入 PromptTop-K 限制 + 摘要 Advisor
流式输出输出 Token 按 chunk 累计maxTokens 上限
评测压测误用 prod Key独立 Key + 配额告警
yaml
spring:
  ai:
    dashscope:
      chat:
        options:
          model: qwen-plus
          max-tokens: 2048
    tool:
      max-steps: 12   # 与 agent-orchestration-java 对齐
自检：一次 Agent 任务为何比单轮 Chat 贵 5～10 倍？

每步工具调用都会把 Thought、Action、Observation 追加进上下文再次计费；若无步数预算与模型路由，成本线性膨胀。

百炼控制台配额告警应绑哪个 Key？
生产 S3 Agent 专用 Key；dev/staging 独立阈值。告警触发时 practice-02 熔断与 turbo 降级联动，避免单 Agent 拖垮全站额度。
compatible-mode 与原生 DashScope 计费是否相同？
同一模型通常同价；差异在集成方式而非单价。CorpAssist 选型看框架能力（ReactAgent、Graph），不是看计费公式差异。

请求超时、重试与 429 治理

门户 SSE 路径上，DashScope 偶发 429（配额）或 5xx 会表现为「流式中断」。S3 Agent 应在 WebClient / SDK 层配置指数退避重试，且不对用户可见 chunk 重复推送；重试仅发生在服务端与百炼之间。

HTTP 现象常见原因CorpAssist 处理
429Key 配额或 QPS 超限practice-02 熔断 + 切换 turbo；告警绑 S3 专用 Key
504 / 读超时长上下文 + 流式慢网关 SSE 超时 ≥ 120s；中间发心跳 event
401Key 过期或环境变量未注入启动时 HealthIndicator 探测，拒绝冷启动带病上线
yaml
# 与 DashScope HTTP 客户端对齐（示意）
spring:
  ai:
    dashscope:
      chat:
        options:
          timeout: 90s
      client:
        max-retries: 2
        retry-backoff: 1s,2s
CorpAssist · 月末报销高峰 S3 请假/报销 Agent 在月末 QPS 翻倍：Micrometer 按 model 与 http_status 打点；429 率超阈值时自动将非关键意图路由到 qwen-turbo，主链路 plus 保留给含 Tool 的多步任务。
自检：流式重试会不会让用户看到重复段落？

会，若在前端直接重放已收 chunk。正确做法：服务端重试时重置 Flux 订阅，门户收到 event: error 后整段重新请求，或仅展示「正在重试」占位，不拼接半截答案。

不推荐
  • 无限重试 429，放大账单与延迟
  • 把 DashScope request_id 从响应头丢弃，无法对账
推荐
  • 结构化日志记录 request_id、model、input/output tokens
  • 重试上限 + 熔断后返回友好降级文案

本章结论

面试快问快答

DashScope 原生接入与 OpenAI 兼容模式各适合什么场景？

答：原生 spring.ai.dashscope.* 适合 SAA 全特性（Agent、Graph、扩展）；兼容模式适合已有 spring-ai-openai 配置、需与 Python OpenAI SDK 共用 endpoint 的双栈迁移。CorpAssist 新模块优先原生 Starter。

生产 Agent 如何控制百炼账单失控？

答：步数上限、maxTokens、模型路由（turbo 降级）、独立评测 Key、Micrometer 按 model 统计 Token；配合 practice 章熔断与缓存。

判断练习

  1. 同事把 AI_DASHSCOPE_API_KEY 写进 application-prod.yml 并提交 Git。请指出风险与改法。

    参考答案

    密钥泄露与无法轮换。应改用 K8s Secret / 环境变量注入，YAML 仅保留 ${AI_DASHSCOPE_API_KEY} 占位符；已泄露 Key 立即在百炼控制台作废重建。

  2. S3 Agent 把每步 Thought 也 SSE 给前端，产品称「更透明」。违反本章哪条原则？

    参考答案

    违反流式只暴露用户可见内容原则：泄露内部推理与工具细节，且有合规风险。Thought/Observation 应进 Trace 与审计日志，SSE 仅推送最终答案。
