为什么要学本章

S3 Agent 在月底报销高峰、大模型侧 rate limit 或网络抖动时，若仍同步阻塞调 qwen-plus，整链路透传 503 到员工端。circuit breaker、semantic cache 与 multi model fallback 是应用层容灾三板斧——在 Spring AI Alibaba 栈内完成，而非等 K8s 重启 Pod。

本章先记住 3 件事
  • 先保护再优化：熔断失败率阈值先于缓存调参。
  • 写操作不缓存：roomBooking、submitExpense 禁止语义缓存命中。
  • 降级可感知：切 turbo 或模板答复时，响应头/文案告知用户「简版模式」。

熔断：失败率、慢调用与半开

DashScope 超时或 429 会拖垮 Tomcat 线程池。对 ChatModel 外包一层 circuit breaker：失败率超阈则快速失败，走 fallback 或排队；半开期放少量探测请求恢复。

CorpAssist · S3 · 百炼限流 09:00–10:00 订会议室集中爆发，DashScope 返回 429。熔断打开后，Agent 改答「当前繁忙，已为您排队；可查看常订会议室规则」并异步重试，而非 30s 白屏。

java
@Service
public class ResilientS3ChatService {
  private final ChatModel primary;
  private final ChatModel fallback;
  private final CircuitBreaker breaker;

  public String chat(String userMessage) {
    Supplier<String> supplier = CircuitBreaker
        .decorateSupplier(breaker, () ->
            primary.call(new Prompt(userMessage)).getResult().getOutput().getText());
    return Try.ofSupplier(supplier)
        .recover(CallNotPermittedException.class, e -> invokeFallback(userMessage))
        .get();
  }
}

缓存：精确键与语义相似

S3 大量重复问法：「年假制度在哪」「报销截止日」。semantic cache 对 embedding 相近的问题返回已缓存答复，降低 token 与延迟。须严格排除带 Tool 写路径与含 PII 的会话。

策略键S3 适用风险
精确缓存hash(system+user+model)静态 FAQ低
语义缓存query embedding 近邻制度类只读问答相似问法误命中
Tool 结果缓存tool+args hashroomAvailability 只读查询TTL 须短（≤60s）
禁止缓存—booking/expense/含工号合规
不推荐
  • 全 Agent 对话一把梭缓存
  • 无 TTL，会议室空闲状态过期仍命中
  • 缓存 Key 不含 model 版本
推荐
  • Advisor 链前置 CacheAdvisor，可配置 bypass 写 Tool
  • Redis + 向量近邻（CorpAssist 共用缓存层）
  • Key 含 appVersion 与 qwen-plus 标识
java
// CacheAdvisor 示意：写 Tool 路径 bypass
public class S3CacheAdvisor implements CallAroundAdvisor {
  @Override
  public AdvisedResponse aroundCall(AdvisedRequest req, CallAroundAdvisorChain chain) {
    if (req.toolsScheduled() || containsWriteTool(req)) {
      return chain.nextAroundCall(req);
    }
    return cache.getOrCompute(cacheKey(req), () -> chain.nextAroundCall(req));
  }
}

多模型：主备路由与静态降级

multi model fallback 在 DashScope 内切换 qwen-plus → qwen-turbo；再失败则返回规则模板或「请稍后再试」。与 Admin DSL 的 model.primary/fallback 字段保持一致，避免 Studio 配 plus、代码走 turbo。

Resilience4j 与 Spring AI Advisor 组合

推荐顺序：RateLimiter（入口） → CacheAdvisor（只读） → CircuitBreaker 包裹 ChatModel → Tool 调用各自 breaker。避免在 Controller 手写 try/catch，统一在 Advisor 链或可观测装饰器完成。

java
@Bean
ChatClient s3ChatClient(ChatModel primary, ChatModel fallback, S3CacheAdvisor cache) {
  return ChatClient.builder(primary)
      .defaultAdvisors(cache, new DegradeHeaderAdvisor())
      .build();
}

@Bean
@Primary
ChatModel resilientPrimary(@Qualifier("dashscopePlus") ChatModel plus,
                           CircuitBreakerRegistry registry) {
  return new CircuitBreakerChatModelProxy(plus, registry.circuitBreaker("dashscopeChat"));
}

RateLimiter 与熔断应设在网关还是 Agent 服务？

网关做粗粒度租户/全局 QPS；Agent 服务做 DashScope 配额保护与语义缓存。两层互补，不要只在网关限流却仍在应用内打满 429。

通义栈视角：容灾与百炼配额

DashScope 429 与下游 OA 超时是 S3 Agent 最常见的外部故障。Java 轨用 Resilience4j + 语义缓存 + 多模型 Fallback；Python 轨用 tenacity + Redis。双栈须共享降级文案与缓存 Key 规范（含 model 版本）。

不推荐
  • 只在网关限流，应用内仍打满 DashScope 429
  • Java 已切 turbo、Python 仍调 plus，用户体验分裂
  • catch 后返回空字符串，诱发 Agent 假执行
推荐
  • 网关粗粒度 QPS + Agent 服务 DashScope 配额保护
  • plus → turbo → 规则/异步，双栈同一降级文案
  • 熔断 OPEN 告警 + 语义缓存读路径兜底
S3 路径Java（本课）Python 对照共享契约
订会议室 FAQResilience4j + CacheAdvisortenacity + Redis降级文案 X-CorpAssist-Degraded
报销 OCR异步队列 + 独立 breakerCelery workertaskId 状态机一致
审批摘要plus → turbo → 模板同模型链Admin DSL model 字段
CorpAssist · S3 · 双栈 429 联动 Java 轨熔断 OPEN 后，网关向 Python 实验轨不应自动导流——两轨 DashScope Key 可能不同。On-call 应分别查 dashscopeChat 与 Python 侧 quota 告警，再决定是否临时调高百炼 RPM 或扩大语义缓存 TTL。
自检：语义缓存 Key 要不要带 tenantId？

要。多法人共用 Redis 时，Key 前缀 {tenant}:{appVersion}:{normalizedQuery}，避免 A 事业部的制度 FAQ 被 B 事业部命中。写 Tool 路径仍强制 bypass。

NS3 成本复盘时，将 corpassist.model.degraded 与百炼账单按日对齐，可区分「真降本」与「质量下滑」。

订会议室写路径（roomBooking）永不进入语义缓存；仅 roomAvailability 只读查询可设 TTL ≤ 60s。

本章结论

  • 熔断：DashScope 与下游 OA 分开实例；8s 超时 + 50% 失败率起步。
  • 缓存：只读 FAQ/availability 可缓存；写 Tool 与 PII 强制 bypass。
  • 多模型：plus → turbo → 规则/异步；与 Admin DSL model 字段一致。

面试问答

DashScope 429 时你会怎么做，而不是无限重试？

答：Resilience4j 熔断 + 指数退避有限次；打开熔断后切 qwen-turbo 或 FAQ；写操作进异步队列。监控 429 率与降级比例，配合配额申请与缓存降 QPS。

语义缓存会不会导致会议室状态答错？

答：会，若对 roomAvailability 缓存 TTL 过长或误缓存含时间槽的上下文。只读 availability TTL ≤60s；确认预订成功后 invalidate 相关 Key；写路径永不缓存。
