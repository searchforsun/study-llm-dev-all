为什么要学本章

上一章你已能调通百炼 API。若不清楚 spring ai alibaba bom 与 dashscope starter 注册了哪些 Bean，就无法与 Spring AI 工程化spring-ai-engineering 的抽象层对齐，更谈不上在 profile switch 下切换 dev/staging/prod 模型。本章讲清依赖 → 配置项 → 多环境。

学完你能
  • 在父 POM import spring ai alibaba bom 与 extensions BOM，子模块只声明 Starter
  • 列出 spring.ai.dashscope.* 核心配置项，并映射到 environment variable
  • 用 Spring Profile 为 S3 Agent 配置 dev 用 turbo、prod 用 plus，且密钥不落地
本章先记住 3 件事
  • 双 BOM：主框架 BOM + extensions BOM 管 DashScope Starter 版本。
  • AutoConfiguration：classpath 有 Starter 且 api-key 存在才注册 ChatModel。
  • Profile 分环境：模型名、base-url、配额策略按 profile 切换，Key 始终来自环境。

spring-ai-alibaba-bom

Agent、Graph 等核心模块版本真源。

extensions-bom

DashScope Starter、文档解析等扩展版本（勿漏 import）。

spring-ai-bom

ChatClient、Advisor 等 Spring AI 核心与 SAA 对齐。

不推荐
  • 只 import 主 BOM，DashScope Starter 版本解析失败
  • 子模块手写 1.1.2.1 与父 POM 不一致
  • 业务模块直接依赖 dashscope-sdk-java
推荐
  • 父 POM 双 BOM + spring-ai-bom 三件套
  • 子模块零 version 声明 Starter
  • 业务只依赖 ChatModel / ChatClient 抽象
先想 10 秒：Gradle 报「找不到 spring-ai-alibaba-starter-dashscope 版本」最常见原因？

未 import spring-ai-alibaba-extensions-bom，或 Maven 仓库未配置阿里云镜像；见 GitHub issue #4204 讨论。

配置项

auto configuration 类读取 spring ai dashscope 前缀。与 spring-ai-engineering 的 spring.ai.openai.* 平行存在，勿混用前缀于同一 ChatModel Bean。

yaml
spring:
  ai:
    dashscope:
      api-key: ${AI_DASHSCOPE_API_KEY}
      chat:
        options:
          model: ${CORPASSIST_AGENT_MODEL:qwen-plus}
          temperature: 0.3
          top-p: 0.8
          max-tokens: 2048
      embedding:
        options:
          model: text-embedding-v3
    tool:
      max-steps: 12
      human-approval-tools:
        - send_email
        - submit_leave
配置项含义环境变量
spring.ai.dashscope.api-key百炼鉴权AI_DASHSCOPE_API_KEY
chat.options.model默认对话模型CORPASSIST_AGENT_MODEL
chat.options.max-tokens单次输出上限可选 CORPASSIST_MAX_TOKENS
tool.max-stepsAgent 最大步数与 YAML 固定或 ConfigMap
Starter 自动配置到 ChatClient

与 spring-ai-engineering 相同模式：业务依赖 chat model bean 而非 HTTP SDK。

java
@Configuration
public class AgentChatConfiguration {

  @Bean
  ChatClient officeChatClient(ChatModel chatModel) {
    return ChatClient.builder(chatModel)
        .defaultSystem("CorpAssist S3 办公 Agent：工具调用须走审批白名单。")
        .build();
  }
}
CorpAssist · human-approval-tools spring.ai.tool.human-approval-tools 列出需人工确认的 Tool 名（发邮件、提交请假）。Starter 不会自动拦截，需与 agent-orchestration-java 的审批节点配合；此处先完成配置占位。

多环境

CorpAssist 至少 dev / staging / prod 三套 Profile：profile switch 切换模型与日志级别，environment variable 注入各环境 Key。模型名可 profile 差异化，密钥永不写入 profile 文件。

yaml
# application-dev.yml
spring:
  ai:
    dashscope:
      chat:
        options:
          model: qwen-turbo
          temperature: 0.5

---
# application-prod.yml
spring:
  ai:
    dashscope:
      chat:
        options:
          model: qwen-plus
          temperature: 0.2
环境Profile模型策略Key 来源
本地联调devqwen-turbo 降本.env 本地
预发staging与 prod 同模型，低配额 KeyK8s Secret
生产prodqwen-plus + 降级路由密钥管理服务
不推荐
  • application-prod.yml 内嵌 api-key
  • dev 连 prod 模型名却用 prod Key
  • 无 spring.profiles.active 默认值导致 CI 行为不一致
推荐
  • Key 仅环境变量；profile 只改 model/temperature
  • staging 独立 Key + 配额告警
  • CI 使用 test profile + Mock ChatModel

自检：单元测试如何不连百炼？

@SpringBootTest + @MockBean ChatModel，或 @ActiveProfiles("test") 禁用 DashScope AutoConfiguration；与 spring-ai-engineering advanced 测试章一致。

条件装配与启动诊断

DashScope AutoConfiguration 通常在「classpath 有 Starter + api-key 非空」时注册 ChatModel。本地忘配 Key 时，应用仍能启动但 Agent Bean 注入失败——应在 ApplicationRunner 或 Actuator 中显式探测。

java
@Component
@ConditionalOnProperty("spring.ai.dashscope.api-key")
class DashScopeReadinessProbe implements HealthIndicator {
  private final ChatModel chatModel;
  @Override public Health health() {
    // 轻量 ping 或仅校验 Bean 存在；勿在每次 health 全量 call LLM
    return chatModel != null ? Health.up().build() : Health.down().build();
  }
}
CorpAssist · K8s 滚动发布 新 Pod 启动时若 Secret 挂载晚于容器 ready，Readiness 应失败避免流量打入；Secret 就绪后 DashScope Bean 可用，再标记 ready。与 spring-ai-engineering 的外部化配置章同一套路。
症状排查
无 ChatModel Bean检查 extensions-bom、api-key 占位符是否解析
双 ChatModel 冲突同时引入 openai + dashscope 时需 @Qualifier
Profile 未生效确认 spring.profiles.active 与 ConfigMap 一致

每条对应上文一个小节，便于串起全章再做题。

  • 依赖：双 BOM + DashScope Starter；Agent 框架按需追加，业务不绑 SDK。

面试快问快答

Spring AI 与 Spring AI Alibaba Starter 能否同时引入？

答：可以共存（例如 openai 兼容轨 + dashscope 原生轨），但需 @Qualifier 或 @Primary 区分 ChatModel；CorpAssist S3 新服务优先单一 DashScope Starter，避免 Bean 冲突。

extensions BOM 漏 import 会怎样？

答：DashScope Starter 版本无法解析，构建失败；Gradle/Maven 报 artifact 无 version。解决：在 dependencyManagement 增加 extensions-bom。

动手练习

约 25 分钟 · 对照 pom 片段完成父 POM 三处 import，并验证 dev/prod Profile 切换模型。

操作步骤

  1. 打开 demos/basics-02-starters-lab/README.md 与 pom.snippet.xml。
  2. 在父 POM 添加双 BOM 与 spring-ai-bom import。
  3. 在 corpassist-s3-agent 模块声明 DashScope Starter（无 version）。
  4. 创建 application-dev.yml / application-prod.yml，仅差异 model 字段。
  5. 分别以 --spring.profiles.active=dev|prod 启动，日志确认 model 值并完成测验。

判断练习

  1. 团队只 import spring-ai-alibaba-bom，Maven 报 DashScope Starter 无版本。根因与修复？

    参考答案

    DashScope Starter 版本由 extensions-bom 管理。修复：dependencyManagement 增加 spring-ai-alibaba-extensions-bom import。

  2. staging 与 prod 共用 application-prod.yml 和同一 API Key。有何风险？

    参考答案

    压测/验收流量计入生产配额，且无法独立审计。应 staging 独立 Profile + 独立 Key + 更低配额。
