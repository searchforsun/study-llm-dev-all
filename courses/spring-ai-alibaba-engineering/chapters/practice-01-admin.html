为什么要学本章

CorpAssist S3 办公 Agent 的「订会议室、提报销、走审批」流程在 PoC 阶段常先在 Dify 里拖拽编排；上线 Java 轨后，若仍靠改 YAML 发版才能调 Prompt，运营与研发会反复扯皮。spring ai alibaba admin 提供 admin studio 可视化与版本治理，让 S3 Agent 的 DSL 成为可审计、可回滚的配置资产，而不是散落在个人笔记本里的导出 JSON。

本章先记住 3 件事
  • 配置即资产：Prompt、模型路由、Tool 白名单进 Admin，变更留痕。
  • DSL 是契约：Studio 导出 DSL 与 Java 运行时解析器版本必须对齐。
  • 迁移不是复制粘贴：Dify 的变量、知识库、插件要逐项映射到 DashScope + Spring 能力。

Studio：可视化编排与发布

Admin Studio 面向「业务可试、研发可管」：运营在浏览器里调整 S3 Agent 欢迎语与分支，研发通过 RBAC 控制谁可以发布到生产。与纯代码编排相比，Studio 降低 Prompt 迭代成本；与 Dify 相比，Studio 与 Spring Boot 运行时同源，避免双轨漂移。

CorpAssist · S3 办公 Agent · 会议室预订 员工说「帮我订明天下午 3 点的小会议室」。Studio 里可见：意图识别 → 调用 roomAvailability Tool → 缺参数时追问 → 确认后 roomBooking。运营可改追问话术，无需发 Java 版。

先想 10 秒：为何 S3 不继续只用 Dify 生产？

企业 Java 轨需要统一鉴权、审计、与 HR/OA 同事务；Dify 适合 PoC，但通义栈 + Admin 与 Spring 治理（限流、Trace、多模块）一体，减少双轨运维与数据出境风险。

DSL：节点、变量与运行时

agent dsl 是 Studio 导出的结构化描述（通常为 YAML/JSON），描述节点图、边条件与 Tool 绑定。Java 侧由 Spring AI Alibaba 运行时解析，等价于你在代码里写的 Graph，但变更走 Admin 发布流程。

yaml
# CorpAssist S3 · 会议室预订片段（示意）
app:
  id: corpassist-s3-room
  version: "1.2.0"
  model:
    primary: qwen-plus
    fallback: qwen-turbo
nodes:
  - id: intent_router
    type: llm
    promptRef: s3-intent-v3
    outputs: [intent]
  - id: check_room
    type: tool
    tool: roomAvailability
    when: "${intent == 'BOOK_ROOM'}"
  - id: confirm_book
    type: tool
    tool: roomBooking
    requires: user_confirm
节点类型含义S3 示例
llm调用 DashScope 做分类/生成意图识别、摘要审批理由
tool调用注册工具submitExpense、roomBooking
branch条件路由缺 roomId 则进入追问子图
human人工确认闸报销超 5000 元走主管审批
不推荐
  • Studio 改 DSL 与 Git 仓库各一份，互不同步
  • 生产直接改草稿未发布版本
  • Tool 名与 Java @Tool 不一致导致运行时 404
推荐
  • 发布后将 DSL 导出归档到 Git（只读快照）
  • CI 校验 DSL schema 与 Tool 白名单
  • 变量命名与 OpenAPI DTO 字段对齐
java
// 运行时：从 Admin 拉取已发布 DSL（示意）
@Configuration
public class S3AgentRuntimeConfiguration {

  @Bean
  AgentApplication roomAgent(AdminClient adminClient) {
    AgentDsl dsl = adminClient.loadPublished("corpassist-s3-room");
    return AgentApplication.fromDsl(dsl);
  }
}

迁移 Dify：对照、缺口与验收

许多团队的 S3 PoC 在 Dify 完成。迁移到 Spring AI Alibaba 不是「导入按钮一点就好」，而要建立能力对照表：知识库、工作流变量、插件 HTTP、会话存储分别落在哪。

Dify 概念通义栈落点迁移注意
应用 / 工作流Admin App + DSL并行运行期：双写流量 5% 对比答案
知识库百炼知识库或 CorpAssist RAG 服务索引 ID、权限模型不同，需重导文档
工具 / 自定义 API@Tool 或 MCPAuth Header 改走企业网关
对话变量DSL 上下文 + Chat Memorysys.user_id 映射工号
模型供应商DashScope qwen-*重跑 eval 集，勿假设输出一致
Dify → Admin 迁移阶段
flowchart TB
  A[导出 Dify DSL/YAML]
  B[字段映射表评审]
  C[Admin 重建 + Tool 对齐]
  D[Shadow 5% 流量]
  E[Eval 门禁通过]
  F[切主 + 保 Dify 只读 30 天]
  A --> B --> C --> D --> E --> F
CorpAssist · 报销 Agent 迁移 Dify 里「发票 OCR + 费控提交」两步工作流：OCR 改接 spring-ai-alibaba-extensions 文档解析；提交改 submitExpense Tool。迁移验收：同一批 50 张测试发票，字段抽取 F1 ≥ PoC 基线。
常见坑
  • 忽略会话 ID — Dify 内置 session 与 Java Memory 存储不一致，用户感觉「失忆」。
  • 插件 URL 写死内网 — 应走 corpassist-gateway，便于换环境与审计。
  • 无 Eval 门禁 — 视觉一致但 Tool 参数错，上线后才暴露。
Admin 与 agent-orchestration-java 课有何区别？
本课偏治理与发布（Studio、DSL 版本、Dify 迁移）；编排课偏 ReactAgent、Graph、MCP 的代码级深度实现。S3 团队通常 Admin 管 Prompt/发布，编排课管复杂多 Agent 拓扑。
DSL 变更需要重启 Java 服务吗？
取决于运行时实现：推荐拉模式——服务定时或订阅 Admin 发布事件，热加载已发布版本；避免每次改 Prompt 都滚动重启。预发环境可强制 refresh=30s 便于验证。
谁有权发布 S3 生产 Prompt？
建议 RBAC：运营可改草稿，Tech Lead + 合规审批生产发布；审计日志记录操作者与 diff。与 Git PR 不同，Admin 发布是运行时真源，权限应更严。

通义栈视角：Admin 治理侧

CorpAssist S3 在通义栈上的 Prompt/DSL 真源应在 Admin，而非 Dify 导出 JSON 或散落 YAML。Java 轨 Studio 与运行时同源；Python 轨 PoC 的 Prompt 须回写 Admin 草稿，避免双轨漂移。

不推荐
  • Dify 与 Admin 各改各的 Prompt，无版本对照
  • Studio 草稿未发布即被网关流量命中
  • DSL 导出不进 Git，无法 CI diff
推荐
  • Admin published version 为生产真源；Python 只读消费
  • 草稿 / 预发 / 生产三态分离，回滚 ≤ 5 分钟
  • DSL 归档进仓库，Prompt 变更走 PR 审阅

自检：Python 轨改 Dify Prompt 后，Java S3 何时感知？

仅当变更回写 Admin 并 published 后，Java 运行时拉取才生效。PoC 阶段可并行 Shadow 对比，但生产真源始终是 Admin 版本号，不是 Dify 导出文件时间戳。

面试问答

如何把 Dify PoC 的办公 Agent 迁到 Java 通义栈而不丢 Tool 能力？

答：先建能力对照（工作流→DSL、插件→@Tool/MCP、知识库→百炼/RAG）；在 Admin 重建应用并对齐 Tool 名与白名单；staging 跑 Shadow 流量与 Eval 集；通过后再切主，保留 Dify 只读回滚窗口。强调鉴权走 gateway、会话 ID 与 Memory 一致。

Studio 改 Prompt 和 Git 改代码，如何避免出现双源真相？

答：Admin 为运营真源；每次发布导出 DSL 快照进 Git 只读归档；CI 做 schema 与 Tool 校验。紧急 hotfix 也走 Admin 发布而非 SSH 改服务器 YAML。

判断练习

  1. 运营在 Studio 草稿里改了 System Prompt，生产用户立刻看到新话术。最可能违反了哪条发布原则？

    参考答案

    草稿未发布即影响生产，说明运行时拉取了非 published 版本或环境配置错误。应仅加载 Admin 已发布版本，草稿仅用于 Studio 试聊。

  2. 迁移后报销 Agent 能聊天但从不调用 submitExpense。优先检查哪两处？

    参考答案

    ① DSL 中 tool 名与 Java/MCP 注册名不一致；② Tool description 过模糊，模型未学会何时调用。对照 Dify 原插件描述补全 @ToolParam 业务化说明。
