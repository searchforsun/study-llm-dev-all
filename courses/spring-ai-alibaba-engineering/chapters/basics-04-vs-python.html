选型

dual stack policy 不是「谁好用选谁」，而是按变更频率、治理要求、团队边界分工。参考 spring-ai-engineering 与 python-engineering-for-llm 的双栈原则。

落 Python 却带来风险
  • 在线高 QPS 门户 Agent 全放 Python FastAPI
  • 需 NS2 审批链却无 Spring 侧 Advisor
  • 双栈各写一套 Tool 实现，契约不同步
推荐分工
  • Java：S3 Agent API、ReactAgent、MCP、审计
  • Python：制度库全量 reindex、Badcase 挖掘
  • 共用：环境变量命名 + OpenAPI DTO
任务推荐栈理由
员工门户对话 + ToolJava SAA治理、Trace、与 HR/OA 集成
夜间 PDF 批量 EmbeddingPython吞吐、与数据湖近
新 Prompt A/B 实验Python Notebook → 固化 Java快迭代后生产化
跨语言 Agent 编排Java Graph + A2A见 agent-orchestration-java practice 章
选 Java

需要 Spring Security、事务、企业 SSO、Micrometer、Admin 可视化。

选 Python

一次性数据分析、大规模离线评测、算法同学主导的原型。

选双栈协作

Python 产索引版本号，Java 检索时带 version 过滤；契约 OpenAPI。

先想 10 秒：能否 Java 与 Python 各跑一个 ReactAgent 处理同一用户会话？

不推荐：会话状态与 Tool 副作用会分裂。应单写路径在线 Agent（Java），Python 只做离线仿真；Memory 共享需统一 session store 设计（context-memory 课）。

# CI 片段：校验 Java/Python 使用同一默认模型（示意）
JAVA_MODEL=$(grep CORPASSIST_AGENT_MODEL deploy/java.env | cut -d= -f2)
PY_MODEL=$(grep CORPASSIST_AGENT_MODEL deploy/python.env | cut -d= -f2)
test "$JAVA_MODEL" = "$PY_MODEL" || exit 1
Python dashscope SDK 与 OpenAI 兼容 SDK 如何选？
新脚本优先 OpenAI 兼容 + CORPASSIST_LLM_BASE_URL，与 Java 配置同构；需 DashScope 专有 API（如特定多模态参数）时用官方 dashscope 包，并在 capability-matrix 标注差异。
双栈都要接 MCP 吗？
生产 Tool 执行以 Java Agent 为唯一写路径；Python 评测可 Mock Tool。避免双栈各注册 send_email 导致审计分裂。

OpenAPI 契约与索引版本握手

双栈「配置同构」不足以保证行为一致：Python 夜间写完 PGVector 后，须通过 OpenAPI 或 ConfigMap 发布 index_version，Java S3 Agent 检索时带 filter，避免读到半写入向量。

yaml
# corpassist-rag-index-service OpenAPI 片段（示意）
paths:
  /v1/index/releases/latest:
    get:
      responses:
        "200":
          content:
            application/json:
              schema:
                properties:
                  index_version: { type: string }
                  embedding_model: { type: string }
                  chunk_policy: { type: string, example: "500/50" }
契约字段Java 消费方Python 生产方
index_versionSearchRequest filter索引 Job 完成后 bump
embedding_model校验与 CORPASSIST_EMBEDDING_MODEL 一致manifest 写入
chunk_policy文档化，防 Java 热更新 chunk 漂移rag-system-py 管道常量
索引发布 → Java Agent 读取
sequenceDiagram
  participant PY as Python index Job
  participant PG as PGVector
  participant API as index-release API
  participant JAVA as S3 Agent
  PY->>PG: batch embed + commit
  PY->>API: PUT index_version=2026.04.1
  JAVA->>API: GET latest
  JAVA->>PG: search filter version
CorpAssist · 评测与生产对齐 评测脚本从同一 index_version 读取向量；若 Python 用 turbo 建库、Java 用 plus 对话，仍可能「检索对、生成偏」。CI 除 model 外应校验 embedding_model 与 index_version 三元组。
先想 10 秒：为何不让 Java 直接调 Python 的 DashScope SDK？

在线 Agent 需要 Spring 治理链（Security、Advisor、ReactAgent、审计）；跨语言 RPC 只应用于索引/评测等离线能力，而非替代 Java 上的 ChatModel Bean。

不推荐
  • Java 写路径、Python 写路径各改 Jira，无统一 audit_id
  • 索引未发布 version Java 已滚动 50% Pod
推荐
  • 单写路径：生产 Tool 仅 Java；Python 只读或仿真
  • 蓝绿索引：新版本 PG collection 就绪后再切流量
bridge 课衔接

跨栈 HTTP/gRPC 与 A2A 见 llm-application-fundamentals bridge 与 agent-orchestration 双轨 practice。

rag-system-java

完整 ingest、混合检索、评测集在此课展开；本章只定双栈边界与 env 单表。

面试快问快答

面试官：CorpAssist 为什么 Java 和 Python 都要接 DashScope？

答：分工不同——Java 承担 S3 在线 Agent、审批、审计与企业集成；Python 承担大规模索引与离线评测。共用百炼能力与 env 命名，避免重复造轮子；不是两套重复 Agent。

原生 DashScope Starter 与 OpenAI 兼容 mode 如何选？

答：新 Java 模块用 SAA 原生以用全 Agent/Graph 特性；需与现有 Python OpenAI SDK 或 spring-ai-openai 配置对齐时用 compatible-mode + 同一 CORPASSIST_LLM_BASE_URL。

判断练习

  1. 评测 Python 用 qwen-turbo，生产 Java 用 qwen-plus，团队称「差不多」。有何问题？

    参考答案

    配置漂移导致评测无法代表生产；Agent 行为与成本差异大。应双栈同 model 或评测显式标注差异并在 CI 门禁。

  2. 为「快速上线」把 S3 门户 Agent 全 rewrite 成 Python FastAPI，放弃 Spring。违背 CorpAssist 哪条架构原则？

    参考答案

    违背双栈 policy：在线治理（SSO、审计、NS2 审批、Admin）是 Java 轨职责；Python 擅离线而非替代企业在线 Agent 层。
