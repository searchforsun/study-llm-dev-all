为什么要学本章

高风险操作（如发邮件、创建工单、扣款）不能让 Agent 独立执行——需要人工确认。同时，长 Agent 轨迹会撑爆上下文窗口，需要压缩和裁剪。human in loop 和上下文管理是 Agent 上生产的两道必修课。

学完你能
  • 在 Graph 中嵌入 approval gate 审批节点
  • 实现 interrupt resume 暂停/恢复 Agent 执行
  • 使用 context compress 压缩历史轨迹
  • 设置 context limit 和 trajectory prune 策略
本章先记住 3 件事
  • 审批是安全底线：写操作必须经人工确认，只读操作可自动执行。
  • 上下文是有限资源：Agent 每一步的思考+工具返回都会累积，必须压缩。
  • Interrupt 不等于终止：暂停等待人工确认后可以恢复继续执行。

Human-in-the-Loop 架构

human in loop 在 SAA 中通过 InterruptNode 实现。Agent 执行到该节点时暂停，等待外部输入后继续。

人在回路审批流程
sequenceDiagram
    participant User as 用户
    participant Agent as Agent
    participant HITL as 审批节点
    participant Human as 审批人

    User->>Agent: 提交报销申请
    Agent->>Agent: 自动验证表单
    Agent->>HITL: 到达审批节点
    HITL-->>Human: 发送审批通知

    alt 审批通过
        Human->>HITL: approve(reason)
        HITL->>Agent: 继续执行
        Agent->>User: 报销处理完成
    else 审批驳回
        Human->>HITL: reject(reason)
        HITL->>User: 返回驳回原因
    end

审批节点暂停 Agent 执行，等待审批人通过或驳回后再决定后续路径。

CorpAssist · 报销审批已读回执 Agent 生成报销单后，需直属上级审批。审批人在钉钉/企业微信收到通知，点击"通过"后 Agent 恢复执行，通知财务打款。

审批节点实现

approval gate 是 SAA Graph 中的一个特殊节点类型。

java 
@Configuration
public class HITLGraphConfig {

    @Bean
    public Graph expenseGraph() {
        return Graph.builder()
            .addNode("validate", this::validateExpense)
            .addNode("llm_extract", this::extractFromReceipt)

            // 审批节点：暂停等待人工
            .addInterruptNode("manager_approval",
                ApproveConfig.builder()
                    .title("报销审批")
                    .description("请审批以下报销单")
                    .approvers(Set.of("manager-001"))
                    .timeout(Duration.ofHours(24))
                    .build())

            .addNode("notify_finance", this::notifyFinance)
            .addNode("reject", this::notifyRejection)

            .addEdge("validate", "llm_extract")
            .addEdge("llm_extract", "manager_approval")
            .addConditionalEdge("manager_approval", state ->
                state.approved() ? "notify_finance" : "reject")
            .build();
    }
}

审批结果处理

java 
@RestController
@RequestMapping("/corpassist/approve")
public class ApprovalController {

    private final GraphExecutor graphExecutor;

    public ApprovalController(GraphExecutor graphExecutor) {
        this.graphExecutor = graphExecutor;
    }

    // 审批人点击"通过"
    @PostMapping("/approve/{executionId}")
    public Mono approve(
            @PathVariable String executionId,
            @RequestBody ApproveRequest request) {

        return Mono.fromRunnable(() ->
            graphExecutor.resume(executionId, GraphResume.approve(request.comment()))
        );
    }

    // 审批人点击"驳回"
    @PostMapping("/reject/{executionId}")
    public Mono reject(
            @PathVariable String executionId,
            @RequestBody RejectRequest request) {

        return Mono.fromRunnable(() ->
            graphExecutor.resume(executionId, GraphResume.reject(request.reason()))
        );
    }
}

上下文压缩策略

Agent 多轮执行后，上下文会变得非常大。context compress 提供三种压缩策略：

策略描述适用场景
滑动窗口 保留最近 N 轮，丢弃最早轮次 对话历史压缩
摘要压缩 对历史做摘要，用摘要代替原文 长 Agent 轨迹
工具结果截断 工具返回超过阈值时自动截断 大结果集工具

滑动窗口配置

java 
@Bean
public ReactAgent compressedAgent(ChatModel chatModel, List tools) {
    return ReactAgent.builder()
        .chatModel(chatModel)
        .tools(tools)
        .contextStrategy(ContextStrategy.builder()
            .windowSize(6)                    // 保留最近 6 轮
            .compressAfterSteps(5)            // 每 5 步触发压缩
            .maxToolOutputTokens(2000)        // 工具返回最多 2000 token
            .summaryStrategy(SummaryStrategy.ROLLING)  // 滚动摘要
            .build())
        .maxSteps(15)
        .build();
}

手动触发轨迹裁剪

trajectory prune 是对已完成步骤的清理：

java 
// 在特定节点手动触发裁剪
@GraphNode(name = "prune_context")
public GraphState pruneContext(GraphState state) {
    // 压缩历史：只保留最近的 3 轮对话
    List history = state.getHistory();
    List pruned = history.size() > 3
        ? history.subList(history.size() - 3, history.size())
        : history;
    return state.withHistory(pruned);
}

上下文限制与告警

context limit 是上下文窗口的硬性上限。SAA 支持 saa context 全局上下文配置。

配置项默认值说明
maxContextTokens 32000 上下文窗口最大 Token 数
reserveTokens 4000 保留给模型输出的 Token
systemTokens 4000 System Prompt 预留
toolOutputRatio 0.3 工具输出占总窗口的最大比例
yaml 
spring:
  ai:
    dashscope:
      chat:
        options:
          max-tokens: 4096  # 模型输出上限
corpassist:
  context:
    max-tokens: 32000
    reserve-output: 4000
    system-prompt-budget: 4000
    tool-output-budget: 9600  # 30%
    history-budget: 6400      # 20%
    retrieval-budget: 12000   # 37%
不推荐
  • 从不检查当前上下文 Token 数
  • Agent 跑到上下文爆了才知道
  • 所有历史全部保留
  • 工具返回原始字符串不截断
推荐
  • 每步检查 Token 消耗，预判爆窗
  • 配置滑动窗口自动压缩
  • 工具返回主动截断
  • 设置告警：上下文使用率 > 85% 时通知

CorpAssist 场景：带审批的报销流程

java 
@Service
public class ExpenseHITLService {

    private final Graph expenseGraph;

    // 启动报销流程
    public String startExpense(ExpenseRequest request) {
        ExpenseState initialState = new ExpenseState(
            request.employeeId(), request.amount(),
            request.receiptImage(), ExpenseStatus.SUBMITTED
        );
        String executionId = expenseGraph.execute(initialState);
        // 返回 executionId，前端用此 ID 轮询审批状态
        return executionId;
    }

    // 审批人操作
    public void handleApproval(String executionId, boolean approved, String comment) {
        GraphResume resume = approved
            ? GraphResume.approve(comment)
            : GraphResume.reject(comment);
        graphExecutor.resume(executionId, resume);
    }

    // 查询当前执行状态
    public ExpenseState getStatus(String executionId) {
        return graphExecutor.getState(executionId);
    }
}

常见问题（FAQ）

Interrupt 后 Agent 状态会丢失吗？
不会。SAA 的 GraphExecutor 会将当前 GraphState 持久化到 StatePersister 中（默认 Redis），恢复时从 Redis 加载完整状态，包括历史消息和变量。
审批超时后消息去哪了？
超时后触发配置的 timeout-action（默认 auto-reject），审批节点返回 false 分支。同时记录超时日志用于审计。建议配置钉钉/企微机器人发送超时告警。
上下文压缩会不会丢失重要信息？
滑动窗口会丢弃最早轮次，但如果关键信息在早期被后续轮次引用过，摘要压缩能保留语义。建议对关键信息（如审批结论、合同金额）做显式的字段提取存入 State，不仅依赖上下文。
哪些操作应该需要人工审批？
黄金法则：写操作要审批，读操作可自动。具体包括：发邮件、创建工单、修改数据、扣款、删除资源。仅查询信息、读取知识库等读操作不需要。

本章结论

  • 审批节点：通过 InterruptNode 暂停 Agent，等待人工确认后恢复。
  • 上下文压缩：滑动窗口 + 摘要 + 截断三种策略防止爆窗。
  • 预算管理：显式设置每层 Token 上限，预留输出空间。
  • 监控告警：上下文使用率超 85% 时主动告警。
  • 审批规则：写操作要审批，读操作可自动。
