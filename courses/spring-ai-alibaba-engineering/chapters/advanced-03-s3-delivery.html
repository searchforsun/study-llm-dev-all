S3 Agent 能否直接回答纯制度 FAQ 而不走 S1？

可以但浪费步数预算。Router 应把纯 FAQ 导向 S1 ChatClient；S3 仅在需 Tool 写操作时使用。intent 误路由是可测 Badcase。

飞轮第一轮应优先修哪类 Badcase？

优先「写错数据」类（假执行、错误制度引用）> 纯体验类（措辞）。按 risk × frequency 排序；采购/Jira 写路径样本优先入库 golden。

面试快问快答

你如何证明 S3 Agent 已可生产交付？

答：出示 delivery-checklist 签字项：golden ≥85%、四坑各有检测与修复、RAM/tenant/Trace 齐全、Runbook 就绪；非 Demo 录屏。

Badcase 飞轮和 traditional bugfix 有何不同？

答：LLM 非确定性；单个 fix 需进 golden 防回归，且常改 DSL/Prompt 而非仅 Java 代码。Trace+pitfall 标签驱动优先级。

动手练习

约 45 分钟 · 填写 delivery-checklist.md 并设计一轮 Badcase 飞轮。

操作步骤

  1. 打开 demos/advanced-03-s3-delivery-lab/delivery-checklist.md。
  2. 勾选/补全四层交付项，为每项写「证据链接」占位符。

  5. 从 Trace 样例虚构 3 条 Badcase，标注 pitfall 并写入 golden 计划。

判断练习

  1. golden 通过率 90%，但无 HITL 节点、无 Tool 截断。能否签字上线？

    参考答案

    不能。治理项缺失：采购等写操作无 HITL；过长 Tool 返回会导致偏航与成本 spike。checklist 治理层未满足。

  2. 把 S1 RAG 与 S3 合并成一个「超级 Agent」简化路由。风险是什么？

    参考答案

    写 Tool 暴露给纯 FAQ 场景，Prompt 注入面增大；步数与 Token 浪费；Badcase 难归因。应 intent 路由分 Agent。
