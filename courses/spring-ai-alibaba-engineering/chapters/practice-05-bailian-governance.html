为什么要学本章

CorpAssist 多事业部共用百炼能力时，主账号 dashscope api key 一旦泄露，影响面覆盖 S3 Agent、制度 RAG 与批处理索引。bailian console 与阿里云 RAM 提供子账号、权限边界与用量审计；应用层还须 token billing 打标实现多租户成本分摊。

学完你能
  • 为 dev/staging/prod 与批处理/在线创建 RAM 子账号与最小权限策略
  • 设计 Key 轮换流程且不断服（双 Key 重叠窗口）
  • 从百炼控制台与 Micrometer 导出配额/Token 审计报表
  • 在 Spring 侧注入 tenantId 标签，支撑 NS3 成本治理
本章先记住 3 件事
  • 最小权限：在线 Agent 子账号禁止调管理 API；批处理账号禁止 Chat 写操作。
  • 密钥不进 Git：Secret Manager / K8s Secret + environment variable 注入。
  • 租户可审计：每个 DashScope 请求带 tenantId、scene（S1/S2/S3）标签。

Key 轮换

Key 轮换目标：定期更换 + 泄露应急，且 Java/Python 双栈同时切换。流程：新 Key 写入 Secret → 滚动发布 Pod（重叠 24h 双 Key 有效）→ 禁用旧 Key → 审计确认零 401。

多租户

CorpAssist 多法人实体共用集群时，数据与配额须 tenant 隔离：VectorStore metadata filter、ChatMemory 分区、DashScope 调用 tag。百炼侧无原生「租户」概念，靠应用层 X-Tenant-Id 与 RAM 子账号组合实现。

逻辑多租户

单 RAM 账号 + 应用 tag；成本低，适合初期。

物理分账

大事业部独立 RAM + 独立 Key；合规隔离强。

混合

核心 tenant 独立 Key；长尾 tenant 共享 + 硬 quota。

与 spring ai alibaba admin 配合：Studio 调试须选 tenant 沙箱，禁止直连 prod 数据。

dev 工程师能否共用 staging Key 加快联调？
可以共用 staging 子账号，但禁止共用 prod Key。个人本地须 .env.local；离职时只吊销个人沙箱 AK，不影响 staging 轮换窗口。
百炼控制台配额告警如何对接 On-call？
控制台阈值告警 → 飞书/钉钉 Webhook；与 Micrometer 429 率告警双通道。On-call runbook 第一步：查是否单 tenant 刷量或 Agent 偏航循环。

面试快问快答

百炼 API Key 泄露你怎么应急响应？

答：立即 RAM 禁用泄露 AK → 启用预备 Key 滚动发布 → CloudTrail/百炼用量查异常调用 → 根因（Git/日志/截图）→ 轮换所有同权限 Key → 复盘补 CI secret scan。

多租户共用一套 DashScope Key 有何风险？

答：无法分账与独立吊销；一 tenant 被攻击刷 Token 影响全体配额。应用层 tag 是底线，大 tenant 应独立 RAM 子账号或 Key。

判断练习

  1. 为「方便排查」，把 prod DashScope Key 贴进 Admin Studio 截图发群。是否合规？

    参考答案

    严重违规：Key 泄露需紧急轮换。Studio 应连 staging 沙箱；prod 排障用 Trace/Prompt 回放，密钥仅 Secret Manager。

  2. 批处理索引与 S3 Agent 共用 prod-online Key，仅依靠「晚上跑批」控制成本。问题在哪？

    参考答案

    无法分账与独立限流；批处理 spike 可能触发 429 影响白天 Agent。应 prod-batch 子账号 + 独立 quota 告警。
