常见问题（FAQ）

Admin Studio 中编排的 Graph 怎么和代码协同？
推荐"DSL 优先"模式：在 Studio 中可视化编排 → 导出 DSL JSON → 提交到 Git → CI 自动部署。代码中的 @GraphNode 注解方法作为节点实现，DSL 只定义拓扑结构。
可以只写代码不用 Admin Studio 吗？
完全可以。Graph Builder API 和 Admin Studio 生成的是等价的 DSL。小团队可直接用代码；平台团队建议用 Studio 降低沟通成本。
Dify 的已有工作流能自动迁移到 SAA 吗？
没有自动迁移工具。需要手动将 Dify 工作流的节点映射到 SAA Graph 的节点和条件边。建议先从简单的线性流程开始迁移。
Admin 的权限管理怎么做？
SAA Admin 集成 Spring Security，支持 RBAC。可配置：开发者只能编辑 dev 环境，平台工程师可管理 staging，运维可发布 production。

为什么要学本章

LLM API 调用存在延迟高、限流、临时不可用等问题。没有韧性设计的 Agent 在 DashScope 限流时会直接失败。circuit breaker、semantic cache、fallback model 是生产级 Agent 的必备能力。

学完你能
  • 使用 resilience4j 为 ChatModel 添加熔断保护
  • 实现 rate limit 防止 API 配额耗尽
  • 配置 multi model 降级策略
  • 设计 degrade policy 实现优雅降级
本章先记住 3 件事
  • 熔断保护上游：LLM 调用失败达到阈值后快速失败，不继续消耗资源。
  • 缓存降成本：semantic cache 同时降低延迟和 Token 消耗。
  • 降级 < 失败：降级到更弱但可用的模型比直接报错好。

CorpAssist · 大促期间防护 公司全员大会后大量员工同时查询工资单，Agent QPS 飙升至平时的 50 倍。RateLimiter 拦截超配额请求，CircuitBreaker 在 DashScope 返回 429 后快速熔断，SemanticCache 命中常见问题（如"工资几号发"），降级模型处理非常见问题。

Resilience4j 熔断保护

resilience4j 与 Spring Boot 深度集成，通过注解包装 ChatModel 调用。

java 
@Service
public class ResilientChatService {

    private final ChatModel chatModel;

    public ResilientChatService(ChatModel chatModel) {
        this.chatModel = chatModel;
    }

    @CircuitBreaker(name = "dashscopeChat", fallbackMethod = "chatFallback")
    @Retry(name = "dashscopeRetry", fallbackMethod = "chatFallback")
    @TimeLimiter(name = "dashscopeTimeout")
    public CompletableFuture chat(String message) {
        Prompt prompt = new Prompt(new UserMessage(message));
        return CompletableFuture.completedFuture(
            chatModel.call(prompt).getResult().getOutput().getContent()
        );
    }

    // 降级方法：切换到备选模型
    public CompletableFuture chatFallback(String message, Throwable t) {
        log.warn("Chat fallback triggered: {}", t.getMessage());
        // 使用 qwen-turbo 作为降级
        Prompt fallbackPrompt = Prompt.builder()
            .withUserMessage(message)
            .withModel("qwen-turbo")
            .build();
        return CompletableFuture.completedFuture(
            chatModel.call(fallbackPrompt).getResult().getOutput().getContent()
        );
    }
}

限流与配额管理

rate limit 保护 DashScope API 配额不被耗尽。

java 
@Component
public class DashScopeRateLimiter {

    private final RateLimiter rateLimiter;

    public DashScopeRateLimiter() {
        // 每秒最多 50 次调用（根据 DashScope 配额调整）
        this.rateLimiter = RateLimiter.create(50.0);
    }

    /**
     * 限流包装：调用前获取令牌
     */
    public  T executeWithLimit(Supplier call) {
        if (rateLimiter.tryAcquire(2, TimeUnit.SECONDS)) {
            return call.get();
        } else {
            throw new RateLimitException("DashScope 调用限流，请稍后重试");
        }
    }
}

// 使用
@Service
public class QuotaAwareChatService {

    private final DashScopeRateLimiter rateLimiter;
    private final ChatModel chatModel;

    public String chat(String message) {
        return rateLimiter.executeWithLimit(() -> {
            Prompt prompt = new Prompt(new UserMessage(message));
            return chatModel.call(prompt).getResult().getOutput().getContent();
        });
    }
}

语义缓存

semantic cache 对用户输入语义检索，命中缓存直接返回，节省 Token 和延迟。

java 
@Component
public class SemanticCacheService {

    private final EmbeddingModel embeddingModel;
    private final RedisTemplate redis;

    // 缓存配置
    private static final double SIMILARITY_THRESHOLD = 0.92;
    private static final Duration CACHE_TTL = Duration.ofHours(1);

    public SemanticCacheService(EmbeddingModel embeddingModel, RedisTemplate redis) {
        this.embeddingModel = embeddingModel;
        this.redis = redis;
    }

    public Optional getCached(String userMessage) {
        // 计算查询向量
        float[] queryVec = embeddingModel.embed(userMessage);

        // 在 Redis 中搜索相似缓存
        Set cachedKeys = redis.keys("semcache:*");
        for (String key : cachedKeys) {
            String cachedJson = redis.opsForValue().get(key);
            CacheEntry entry = JsonUtils.parse(cachedJson, CacheEntry.class);

            if (cosineSimilarity(queryVec, entry.embedding()) >= SIMILARITY_THRESHOLD) {
                log.info("Semantic cache hit: {} -> {}", userMessage, key);
                return Optional.of(entry.response());
            }
        }
        return Optional.empty();
    }

    public void putCache(String userMessage, String response, float[] embedding) {
        String key = "semcache:" + DigestUtils.md5Hex(userMessage);
        CacheEntry entry = new CacheEntry(userMessage, response, embedding, Instant.now());
        redis.opsForValue().set(key, JsonUtils.toJson(entry), CACHE_TTL);
    }

    private double cosineSimilarity(float[] a, float[] b) {
        double dot = 0, normA = 0, normB = 0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            normA += a[i] * a[i];
            normB += b[i] * b[i];
        }
        return dot / (Math.sqrt(normA) * Math.sqrt(normB));
    }

    record CacheEntry(String query, String response, float[] embedding, Instant cachedAt) {}
}
不推荐
  • 没有熔断保护，LLM 调用失败直接抛异常
  • 无缓存，每次调用都走 LLM
  • 限流时直接拒绝请求
  • 降级策略缺失或只返回"系统繁忙"
推荐
  • Resilience4j 熔断 + 重试 + 超时三层保护
  • 语义缓存 + 分级模型降级
  • 限流时返回降级结果而非直接拒绝
  • 降级链：主模型→备用模型→缓存→友好提示

// 分级降级策略
public class DegradeStrategy {

    private final ChatModel primaryModel;   // qwen-max
    private final ChatModel fallbackModel;  // qwen-plus
    private final ChatModel emergencyModel; // qwen-turbo
    private final SemanticCacheService cache;

    public String chatWithDegrade(String message) {
        // Level 1: 主模型
        try {
            return callWithTimeout(primaryModel, message);
        } catch (Exception e) {
            log.warn("Primary model failed: {}", e.getMessage());
        }

        // Level 2: 语义缓存
        Optional cached = cache.getCached(message);
        if (cached.isPresent()) {
            return cached.get() + " [缓存结果]";
        }

        // Level 3: 备选模型
        try {
            return callWithTimeout(fallbackModel, message) + " [降级: qwen-plus]";
        } catch (Exception e) {
            log.warn("Fallback model also failed: {}", e.getMessage());
        }

        // Level 4: 紧急模型
        try {
            return callWithTimeout(emergencyModel, message) + " [紧急降级: qwen-turbo]";
        } catch (Exception e) {
            return "系统暂时无法处理您的请求，请稍后重试。";
        }
    }
}

常见问题（FAQ）

熔断恢复后怎么判断 LLM 是否可用？
Resilience4j 的 Half-Open 状态：熔断开启后等待 30 秒（配置的 wait-duration），然后放少量请求（permitted-number-of-calls-in-half-open-state）探测是否恢复。恢复则关闭熔断，失败则继续开启。
语义缓存会不会返回过时信息？
会。建议设置合理的 TTL（如 1 小时），对时效性要求高的场景（如股票价格）不启用缓存。可以按问题类型决定是否缓存："工资几号发"这类静态问题可缓存，"今天股价"不缓存。
限流时用户体验怎么设计？
不要直接返回"限流"。最佳实践：返回降级结果（缓存/弱模型）并提示"已使用简化模式，完整回答可能受影响"。用户感知到的只是回答质量下降而不是服务不可用。
多个 Agent 实例如何共享限流配额？
单实例 RateLimiter 无法跨实例协同。生产环境使用 Redis 分布式限流（如 Redis Rate Limiter）或 API Gateway 层的全局限流。SAA 与 Sentinel 集成提供分布式限流。

本章结论

  • 韧性层次：限流 → 熔断 → 重试 → 缓存 → LLM 调用。
  • Resilience4j 集成：注解形式为 ChatModel 添加熔断和重试保护。
  • 语义缓存：相似度检索，命中直接返回，降低延迟和 Token 成本。

@Component
public class AgentStepLogger {

    private static final Logger log = LoggerFactory.getLogger("AgentStep");

    /**
     * 记录 Agent 每一步的完整信息
     */
    public void logStep(StepRecord record) {
        // 通过 MDC 注入上下文
        MDC.put("request_id", record.requestId());
        MDC.put("step", String.valueOf(record.stepNumber()));
        MDC.put("agent_id", record.agentId());

        log.info("Agent Step: {} | Thought: {} | Tool: {} | ToolInput: {} | ToolOutput: {}",
            record.stepNumber(),
            truncate(record.thought(), 500),
            record.toolName(),
            truncate(record.toolInput(), 300),
            truncate(record.toolOutput(), 1000)
        );

        MDC.clear();
    }

    /**
     * 完整的 StepRecord 记录
     */
    public record StepRecord(
        String requestId,
        String agentId,
        int stepNumber,
        String thought,        // 模型思考内容
        String toolName,       // 调用的工具名（如有）
        String toolInput,      // 工具输入
        String toolOutput,     // 工具输出
        long durationMs,       // 本步耗时
        int tokensUsed,        // 本步 Token 消耗
        Instant timestamp
    ) {}

    private String truncate(String s, int max) {
        return s != null && s.length() > max
            ? s.substring(0, max) + "..."
            : s;
    }
}

Prompt 回放

prompt replay 是调试 Agent 最强大的功能：通过保存的 request_id 复现 Agent 的完整执行过程。

java 
@Service
public class PromptReplayService {

    private final ElasticsearchRestTemplate esTemplate;
    private final ChatModel chatModel;

    public PromptReplayService(ElasticsearchRestTemplate esTemplate, ChatModel chatModel) {
        this.esTemplate = esTemplate;
        this.chatModel = chatModel;
    }

    /**
     * 根据 request_id 回放 Agent 执行
     */
    public ReplayResult replay(String requestId) {
        // 1. 从 ES 获取完整轨迹
        List steps = esTemplate.search(
            Query.findByRequestId(requestId),
            StepRecord.class
        );

        // 2. 按步数还原每一步的 Prompt
        List reconstructedPrompts = steps.stream()
            .map(step -> """
                === Step %d ===
                Thought: %s
                Tool Call: %s(%s)
                Tool Result: %s
                """.formatted(
                    step.stepNumber(), step.thought(),
                    step.toolName(), step.toolInput(),
                    step.toolOutput()
                ))
            .toList();

        // 3. 用同一 Prompt 重新调用模型
        // （用于对比实际输出和预期输出）
        String fullPrompt = String.join("\n", reconstructedPrompts);
        Prompt prompt = new Prompt(new UserMessage(fullPrompt));
        String replayed = chatModel.call(prompt).getResult().getOutput().getContent();

        return new ReplayResult(requestId, reconstructedPrompts, replayed);
    }

    public record ReplayResult(
        String requestId,
        List steps,
        String replayedOutput
    ) {}
}

常见问题（FAQ）

结构化日志和普通日志有什么区别？
结构化日志以 JSON 格式输出，包含 request_id、step、agent_id 等字段，可直接导入 ES/Kibana 搜索。普通文本日志只能 grep，无法做结构化分析。
Agent Trace 和普通微服务 Trace 有什么不同？
普通 Trace 追踪"请求→服务 A→服务 B→响应"的路径。Agent Trace 额外追踪"Step 1 思考→Step 1 工具调用→Step 2 思考→..."的推理过程。Agent Trace 的 Span 数量 = 推理步数 × 2。
保存每条请求的完整轨迹会不会太占存储？
会。建议：生产环境按 1:10 采样（保存 10% 的轨迹），异常轨迹 100% 保存；7 天后归档到冷存储；只保存关键字段（thought/tool call），不保存原始图片输入。
回放时发现模型回答和当时不一样，正常吗？
正常。LLM 非确定性输出导致相同 Prompt 可能得到不同回答。回放重点是验证"逻辑路径是否正确"而非"每字必须一致"。如果回放结果与原始结果差异很大，说明 Prompt 设计不够稳定。

  • 结构化日志：每步记录 request_id、thought、tool call、duration。
  • Metrics 埋点：Token 消耗、步数、延迟、成功率四类核心指标。
  • Trace 链路：OpenTelemetry 追踪 Agent 跨步骤的推理路径。
  • Prompt 回放：基于 request_id 复现 Agent 完整执行过程。
  • Debug Studio：Admin 中的可视化调试界面。

@RestController
@RequestMapping("/corpassist")
public class CorpAssistEntrypoint {

    private final ChatModel routingModel;
    private final GraphExecutor graphExecutor;
    private final AgentMetrics metrics;

    // 路由：判断用户意图并分发到对应的 Graph
    @PostMapping("/execute")
    public Mono execute(@RequestBody CorpAssistRequest request) {
        // 1. 意图识别（使用 qwen-turbo 做路由降低成本）
        String intent = classifyIntent(request.message());

        // 2. 根据意图分派
        String graphId = switch (intent) {
            case "leave" -> "leave-approval-graph";
            case "ticket" -> "ticket-graph";
            case "query" -> "query-agent";
            default -> "fallback-graph";
        };

        // 3. 执行 Graph
        return Mono.fromFuture(graphExecutor.execute(graphId, request))
            .doOnSuccess(r -> metrics.recordIntent(intent, true))
            .doOnError(e -> metrics.recordIntent(intent, false));
    }

    private String classifyIntent(String message) {
        Prompt prompt = new Prompt(new UserMessage(
            "将以下请求分类为 leave / ticket / query 之一：\n" + message));
        return routingModel.call(prompt)
            .getResult().getOutput().getContent()
            .trim();
    }
}

@Component
public class ToolSecurityManager {

    private final List whitelist;
    private final List requireApproval;
    private final AuditLogger auditLogger;

    public ToolSecurityManager(CorpAssistProperties props, AuditLogger auditLogger) {
        this.whitelist = props.getSecurity().getToolWhitelist();
        this.requireApproval = props.getSecurity().getRequireHumanApproval();
        this.auditLogger = auditLogger;
    }

    /**
     * 检查工具是否允许调用
     */
    public void checkToolAccess(String toolName, String userId, Map args) {
        // 1. 白名单检查
        if (!whitelist.contains(toolName)) {
            auditLogger.log("TOOL_BLOCKED", userId, toolName, args, "不在白名单中");
            throw new SecurityException("工具 " + toolName + " 不在白名单中");
        }

        // 2. 如需审批但未附加 approvalToken，拒绝
        if (requireApproval.contains(toolName) && !args.containsKey("_approvalToken")) {
            auditLogger.log("TOOL_NEEDS_APPROVAL", userId, toolName, args, "缺少审批");
            throw new ApprovalRequiredException("工具 " + toolName + " 需要人工审批");
        }

        auditLogger.log("TOOL_ALLOWED", userId, toolName, args, "");
    }
}

@Service
public class AgentEvaluationService {

    private final AgentStepLogger stepLogger;
    private final MeterRegistry meterRegistry;

    /**
     * 评测 Agent 执行质量
     */
    public EvalResult evaluate(AgentResult result, EvalCriteria criteria) {
        // 1. 步数检查
        boolean stepsOk = result.steps() <= criteria.maxSteps();

        // 2. 工具调用检查
        boolean toolsOk = result.toolCalls().stream()
            .allMatch(tc -> tc.isValid());

        // 3. 是否提前终止
        boolean completed = result.status() == AgentStatus.COMPLETED;

        // 4. 人工介入率
        boolean humanInLoop = result.humanApprovals() > criteria.maxHumanInterventions();

        return new EvalResult(
            result.requestId(),
            completed && stepsOk && toolsOk,
            result.steps(),
            humanInLoop,
            result.tokensUsed()
        );
    }

    public record EvalCriteria(int maxSteps, int maxHumanInterventions) {
        public static EvalCriteria DEFAULT = new EvalCriteria(10, 2);
    }

    public record EvalResult(
        String requestId, boolean passed,
        int steps, boolean excessiveHumanLoop,
        int tokensUsed
    ) {}
}

代码对照：Graph 定义

Python LangGraph 方式

python 
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, END

class PositionState(TypedDict):
    candidate: str
    approved: bool
    comment: str

def validate(state: PositionState) -> PositionState:
    return {**state, "comment": "验证通过"}

def manager_approve(state: PositionState) -> PositionState:
    return {**state, "approved": True}

def hr_check(state: PositionState) -> PositionState:
    return {**state, "approved": True}

def router(state: PositionState) -> Literal["hr", "reject"]:
    return "hr" if state["approved"] else "reject"

# 构建 Graph
graph = StateGraph(PositionState)
graph.add_node("validate", validate)
graph.add_node("manager", manager_approve)
graph.add_node("hr", hr_check)
graph.add_conditional_edges("manager", router)
graph.set_entry_point("validate")
app = graph.compile()

Java SAA 方式

java 
public record PositionState(
    String candidate, boolean approved, String comment
) {}

@Configuration
public class PositionGraph {

    @Bean
    public Graph positionGraph(PositionNodes nodes) {
        return Graph.builder()
            .addNode("validate", nodes::validate)
            .addNode("manager", nodes::managerApprove)
            .addNode("hr", nodes::hrCheck)
            .addConditionalEdge("manager",
                state -> state.approved() ? "hr" : "reject")
            .addEdge("hr", "notify")
            .addEdge("reject", "notify")
            .entryNode("validate")
            .build();
    }
}

@Component
class PositionNodes {
    PositionState validate(PositionState s) {
        return new PositionState(s.candidate(), false, "验证通过");
    }
    PositionState managerApprove(PositionState s) {
        return new PositionState(s.candidate(), true, "上级同意");
    }
    PositionState hrCheck(PositionState s) {
        return new PositionState(s.candidate(), true, "HR 确认");
    }
}

// 异步 Agent 执行器
@Service
public class AsyncAgentExecutor {

    private final GraphExecutor graphExecutor;

    public AsyncAgentExecutor(GraphExecutor graphExecutor) {
        this.graphExecutor = graphExecutor;
    }

    /**
     * 异步执行 Graph，不阻塞请求线程
     */
    public CompletableFuture executeAsync(String graphId, Object state) {
        return CompletableFuture.supplyAsync(() -> {
            try {
                return graphExecutor.execute(graphId, state).get();
            } catch (Exception e) {
                throw new RuntimeException("Agent execution failed", e);
            }
        });
    }

    /**
     * 批量异步执行（并行工具调用场景）
     */
    public List> executeBatch(List tasks) {
        return tasks.stream()
            .map(task -> executeAsync(task.graphId(), task.state()))
            .toList();
    }
}

// Controller 中异步处理
@PostMapping("/corpassist/batch")
public Flux batchProcess(@RequestBody List requests) {
    List> futures = requests.stream()
        .map(req -> asyncExecutor.executeAsync(req.graphId(), req.toState()))
        .toList();

    return Flux.fromIterable(futures)
        .flatMap(Mono::fromFuture);
}

缓存分层设计

cache tier 将缓存分为三层：

优化维度做法QPS 提升
减少步数 合并多次工具调用为批量 API 2-3x
语义缓存 高频问题缓存，跳过 LLM 5-10x (命中时)
模型分级 简单问题用 qwen-turbo 2x
并行工具 独立的工具调用并行执行 2-5x
异步释放 请求线程不阻塞等 Agent 3-5x
层级缓存内容存储TTL命中率
L1: 本地缓存 高频 FAQ 答案 Caffeine 5 min ~20%
L2: 语义缓存 语义相似的问题 Redis 1 h ~30%
L3: 结果缓存 精确 match 的中间结果 Redis 10 min ~10%
java 
@Service
public class TieredCacheService {

    private final Cache localCache;     // Caffeine L1
    private final SemanticCacheService semanticCache;   // Redis L2
    private final RedisTemplate resultCache; // L3

    public TieredCacheService() {
        this.localCache = Caffeine.newBuilder()
            .maximumSize(1000)
            .expireAfterWrite(Duration.ofMinutes(5))
            .recordStats()
            .build();
    }

    public Optional get(String userMessage) {
        // L1: 本地缓存
        String l1Result = localCache.getIfPresent(userMessage);
        if (l1Result != null) return Optional.of(l1Result);

        // L2: 语义缓存
        Optional l2Result = semanticCache.getCached(userMessage);
        if (l2Result.isPresent()) {
            localCache.put(userMessage, l2Result.get()); // 提升到 L1
            return l2Result;
        }

        // L3: 精确结果缓存
        String l3Key = "result:" + DigestUtils.md5Hex(userMessage);
        String l3Result = resultCache.opsForValue().get(l3Key);
        if (l3Result != null) {
            localCache.put(userMessage, l3Result);
            return Optional.of(l3Result);
        }

        return Optional.empty();
    }
}

常见问题（FAQ）

连接池设多大合适？
公式：maxConnections = 目标 QPS × 每个 Agent 的平均 LLM 调用次数 × 平均响应时间（秒）。例如：50 QPS × 5 次 LLM 调用 × 2 秒 = 500。再加 20% 余量 = 600。
虚拟线程和平台线程在 Agent 场景的性能差异？
虚拟线程的优势在 IO 密集型场景。Agent 调用 LLM 是 IO 等待（占 90%+ 时间），虚拟线程能将线程数从 200 提升到数千而不增加内存开销。实测虚拟线程 + Reactor Netty 可提升 3-5x 吞吐。
削峰队列满了怎么办？
队列满时应立即拒绝（而非阻塞）并返回"当前请求量较高，请稍后重试"。设置队列监控告警，使用率 > 80% 时触发扩容流程。
QPS 上不去，Log 显示连接池都在等待，怎么快速排查？
检查 (1) max-connections 是否太小；(2) acquire-timeout 是否太短导致误判；(3) 是否有 TLS 握手瓶颈。启用 HTTP 客户端 metrics 查看连接池状态。生产建议：连接池是第一个查询点。

本章结论

  • 连接池优先：正确的 HTTP 连接池配置是 QPS 提升的第一步。
  • 异步释放：虚拟线程 + Reactor 非阻塞执行提升吞吐。
  • 非 Infra 优化：减少步数、缓存、模型分级比堆机器更有效。
  • 削峰保护：消息队列缓冲突发流量，防止 LLM 限流。
  • 三层缓存：L1 Caffeine + L2 语义 + L3 结果，逐层降 QPS。

Chat API 与流式调用

SAA 的 stream chat 基于 SSE（Server-Sent Events）实现流式输出。生产环境必须使用流式以降低用户感知延迟。

非流式调用

java 
@Service
public class CorpAssistChatService {

    private final ChatModel chatModel;

    public CorpAssistChatService(ChatModel chatModel) {
        this.chatModel = chatModel;
    }

    public String answer(String userMessage) {
        Prompt prompt = new Prompt(new UserMessage(userMessage));
        ChatResponse response = chatModel.call(prompt);
        return response.getResult().getOutput().getContent();
    }
}

流式调用（推荐）

java 
import reactor.core.publisher.Flux;

@Service
public class CorpAssistStreamService {

    private final StreamingChatModel streamingChatModel;

    public CorpAssistStreamService(StreamingChatModel streamingChatModel) {
        this.streamingChatModel = streamingChatModel;
    }

    public Flux streamAnswer(String userMessage) {
        Prompt prompt = new Prompt(new UserMessage(userMessage));
        return streamingChatModel.stream(prompt)
            .map(response -> response.getResult().getOutput().getContent())
            .filter(text -> text != null);
    }
}

Token 计费与成本控制

DashScope 按 Token 计费，不同类型模型价格差异可达 20 倍。了解 billing 模型是控制成本的基础。

成本项说明估算公式
Input Token 用户消息 + System Prompt + 历史 (System + History + User) * 单价
Output Token 模型生成文本 Output 长度 * 单价（通常 Output 更贵）
缓存命中 同前缀 prompt 缓存 缓存命中仅收 10% 费用

CorpAssist 成本预算样例

java 
// 日成本估算器 - CorpAssist 客服场景
public class CostEstimator {
    // 假设日均 10 万次对话，平均每轮 500 input + 200 output tokens
    static final long DAILY_CONVERSATIONS = 100_000;
    static final int AVG_INPUT_TOKENS = 500;
    static final int AVG_OUTPUT_TOKENS = 200;
    static final double PLUS_INPUT_PRICE = 1.2e-6; // 每 token
    static final double PLUS_OUTPUT_PRICE = 4.0e-6;

    public static double estimateDailyCost() {
        double inputCost = DAILY_CONVERSATIONS * AVG_INPUT_TOKENS * PLUS_INPUT_PRICE;
        double outputCost = DAILY_CONVERSATIONS * AVG_OUTPUT_TOKENS * PLUS_OUTPUT_PRICE;
        return inputCost + outputCost; // 约 ¥140/日
    }
}
成本优化策略
  • 语义缓存：对高频 FAQ 启用 semantic cache，减少重复调用（详见 容灾、限流与降级）
  • 模型分级：简单分类用 qwen-turbo，复杂推理用 qwen-max
  • 上下文压缩：对长历史对话做摘要，减少 Input Token（详见 人在回路与上下文工程）

常见问题（FAQ）

DashScope 与百炼是什么关系？
DashScope 是阿里云大模型服务平台，百炼（Model Studio）是 DashScope 的商业化品牌。两者 API 兼容，SAA 统一使用 dashscope 配置前缀。详细信息参见 百炼官方文档。
qwen-max 和 qwen-plus 的智力差距有多大？
在复杂推理（数学、代码生成）场景，qwen-max 比 qwen-plus 准确率高约 15-20%。但简单 QA 场景差异不明显。建议：Agent 核心推理用 max，分类提取用 plus，可动态切换。
API Key 被泄露了怎么办？
立即在 DashScope 控制台 删除该 Key 并创建新 Key。同时检查调用记录是否有异常消耗。Key 泄漏是生产事故，建议启用 RAM Role 降低风险。
流式 URL 地址是什么？
SAA 默认使用官方 endpoint：https://dashscope.aliyun.com/api/v1/services/aigc/text-generation/generation。可通过 spring.ai.dashscope.base-url 自定义。

Document Parser 文档解析

doc parser 是 RAG 管线的前置模块，将 PDF/Word/Excel 转换为 LLM 友好的 Markdown 格式。

解析 PDF 合同

java 
@Service
public class CorpAssistDocParser {

    private final DocumentParser documentParser;

    public CorpAssistDocParser(DocumentParser documentParser) {
        this.documentParser = documentParser;
    }

    /**
     * 解析 PDF 合同，返回 Markdown 结构化文本
     */
    public String parseContract(InputStream pdfStream, String fileName) {
        Document doc = new Document(pdfStream, fileName);
        List parsed = documentParser.parse(doc);

        return parsed.stream()
            .map(Document::getText)
            .collect(Collectors.joining("\n\n"));
    }

    /**
     * 解析后分块，准备注入 VectorStore
     */
    public List chunkAndEmbed(InputStream pdfStream, String fileName) {
        Document doc = new Document(pdfStream, fileName);
        List parsed = documentParser.parse(doc);

        TokenTextSplitter splitter = new TokenTextSplitter(500, 50, 5, 0);
        return splitter.apply(parsed);
    }
}

CorpAssist 场景：多模态审批 Agent

综合各扩展模块，构建一个报销审批 Agent：

java 
@Service
public class ExpenseApprovalAgent {

    private final CorpAssistDocParser docParser;
    private final ReceiptOcrService ocrService;
    private final CorpAssistVisionService visionService;
    private final ChatModel chatModel;

    /**
     * 审批流程：接收附件 → 解析 → 提取关键字段 → 生成审批单
     */
    public ApprovalResult processExpense(ExpenseRequest request) {
        // 1. OCR 识别发票文字
        OcrResult ocrResult = ocrService.recognizeReceipt(request.receiptImage());

        // 2. 如果含 PDF 附件，解析文档
        String docText = request.attachment() != null
            ? docParser.parseContract(new ByteArrayInputStream(request.attachment()), "报销单.pdf")
            : "";

        // 3. 调用 LLM 提取结构化信息
        String extractPrompt = """
            从以下 OCR 结果和文档中提取报销信息：
            总金额、日期、报销人、项目名、是否合规

            OCR结果：
            %s

            文档内容：
            %s
            """.formatted(ocrResult.getText(), docText);

        Prompt prompt = new Prompt(new UserMessage(extractPrompt));
        String extraction = chatModel.call(prompt).getResult().getOutput().getContent();

        // 4. 生成审批摘要，返回给前端等待人工确认
        return ApprovalResult.of(request.requestId(), extraction, Status.PENDING);
    }
}

Document Parser 支持哪些文件格式？

PNG、JPG、JPEG、WEBP、BMP。支持 URL 和 base64 两种图片传入方式。单图建议小于 20MB。
社区插件如何确保稳定性和安全性？

OCR 识别率能达到多少？
对于标准印刷体，识别率可达 99%+。手写体在 85-95% 之间。建议对关键字段做二次校验（金额正则、日期格式）。

常见问题（FAQ）

Python dashscope SDK 和 Java SAA 的流式行为一致吗？
基本一致。底层都是 SSE 协议，Python 通过 generator 逐 chunk 输出，Java 通过 Reactor Flux 逐 chunk 发射。首 Token 延迟和 Token 分割方式一致。
LangGraph 写的 Agent 能自动迁移到 Java 吗？
不能直接迁移。架构概念映射（见上表），但需要按 SAA 的 API 重写。推荐保留 Python 原型作为测试基准，Java 端逐功能重写并对照结果。
两边用同一个模型名，结果会完全一样吗？
相同模型名 + 相同 temperature + 相同 prompt，结果应该一致。但需注意两边 tokenizer 的细微差异可能导致长度计算不完全一致。
Team 全是 Java 背景，是否还需要 Python 原型阶段？
如果有成熟的 Prompt 工程师或已经验证的方案，可以直接用 SAA。但建议至少用 Python 做 Prompt 初步调试——Jupyter 的交互式调试效率远高于 Java 的编译-运行-观察循环。

为什么要学本章

react agent 是 SAA 中 Agent 能力的核心。它实现 Reasoning + Acting（ReAct）循环：模型思考→调用工具→观察结果→继续思考。CorpAssist S3 办公 Agent 的本质就是一个 ReactAgent——自主决策任务流程、调用企业工具、处理中间结果。本章从零实现一个 ReactAgent。

本章先记住 3 件事
  • ReAct = 思考 + 行动 + 观察：模型不是一次性回答，而是逐轮推理。
  • 工具即函数：@Tool 注解将任意 Java 方法暴露为 Agent 可调用的能力。
  • 步数预算是安全带：没有 maxSteps 的 Agent 可能陷入无限循环。

ReAct 循环原理

react loop 遵循 Observe → Think → Act 循环：

ReAct 一次迭代流程
sequenceDiagram
    participant LLM as ChatModel
    participant Agent as ReactAgent
    participant Tool as 工具

    Agent->>LLM: 发送 System + User 消息
    LLM-->>Agent: 思考结果（thought）\n可能含工具调用请求
    Agent->>Agent: 检查 stop conditions

    alt 需要调用工具
        Agent->>Tool: 执行 @Tool 方法
        Tool-->>Agent: 观察结果（observation）
        Agent->>LLM: 追加 thought+observation
        LLM-->>Agent: 继续思考
    else 直接回答
        Agent-->>User: 返回最终回答
    end

ReAct 循环使 Agent 能在多次工具调用中逐步推理，每次"观察"结果都作为下一轮思考的上下文。

observe think 模式的关键特征是：工具返回结果不会直接拼接到回答中，而是作为"观察"反馈给模型，由模型决定下一步行动。

CorpAssist · 工单状态查询 员工问"我的报销单 #T2024-0381 审批到哪了？"，ReactAgent 先思考需要查工单系统，调用工单查询工具，获取状态"待财务审核"，再思考需要补充什么信息，最终给出完整回答。

@Tool 工具注册

tool calling 通过 @Tool 注解将任意 Spring Bean 方法注册为 Agent 工具。

定义工具

java 
@Component
public class CorpAssistTools {

    private final TicketService ticketService;
    private final EmployeeService employeeService;

    public CorpAssistTools(TicketService ticketService, EmployeeService employeeService) {
        this.ticketService = ticketService;
        this.employeeService = employeeService;
    }

    @Tool(name = "query_ticket", description = "查询工单当前状态，输入工单号")
    public String queryTicket(String ticketId) {
        Ticket ticket = ticketService.findById(ticketId);
        if (ticket == null) {
            return "未找到工单: " + ticketId;
        }
        return "工单 %s 当前状态：%s，处理人：%s，创建时间：%s"
            .formatted(ticket.id(), ticket.status(), ticket.assignee(), ticket.createdAt());
    }

    @Tool(name = "get_employee_info", description = "查询员工信息，输入员工工号")
    public String getEmployeeInfo(String empId) {
        Employee emp = employeeService.findByEmpId(empId);
        if (emp == null) {
            return "未找到员工: " + empId;
        }
        return "员工 %s %s，部门：%s，职位：%s"
            .formatted(emp.empId(), emp.name(), emp.department(), emp.position());
    }
}

常见问题（FAQ）

Agent 调用工具时超时了怎么办？
@Tool 方法本身应设内部超时。如果工具调用超时，ReactAgent 会将超时异常作为 observation 返回给模型，模型可能选择重试或跳过。建议使用 Resilience4j 包装工具调用，详见 容灾、限流与降级。
maxSteps 耗尽后 Agent 会怎样？
返回已收集到的所有信息作为最终回答，日志记录"max steps reached"。不会抛出异常，但回答可能不完整。建议设置告警：当 Agent 频繁达到 maxSteps 时，说明任务复杂度过高或工具设计不合理。
多个工具可以并行调用吗？
当前 ReactAgent 默认串行调用。如需并行，可使用 多 Agent 编排 中的 Parallel Agent 模式。
工具返回太长会不会撑爆上下文？
会。对于返回大量数据的工具（如查全部员工列表），应在工具方法内做截断或分页。全局上下文管理参见 人在回路与上下文工程。

orchestration 是 SAA 中编排 Agent 的核心抽象。提供三种基本模式：

三种编排模式

Sequential：Pipeline 编排

sequential flow 将多个 Agent 串联成管线，前一个 Agent 的输出作为后一个 Agent 的输入。

java 
@Configuration
public class SequentialPipelineConfig {

    @Bean
    public SequentialOrchestrator corpassistPipeline(
            ReactAgent plannerAgent,
            ReactAgent executorAgent,
            ReactAgent reviewerAgent) {

        return SequentialOrchestrator.builder()
            .addStage("planner", plannerAgent)
            .addStage("executor", executorAgent)
            .addStage("reviewer", reviewerAgent)
            // 传递上下文字段映射
            .stateMappings(Map.of(
                "plan", "planner.output",
                "execution", "executor.output",
                "review", "reviewer.output"
            ))
            .build();
    }
}

// 使用
@RestController
public class PipelineController {

    private final SequentialOrchestrator pipeline;

    @PostMapping("/corpassist/pipeline")
    public Mono execute(@RequestBody TaskRequest task) {
        return Mono.fromFuture(pipeline.execute(task))
            .map(result -> new OrchestrationResult(
                result.getState("execution"),
                result.getState("review")
            ));
    }
}

Parallel：并行执行

parallel agent 模式将任务拆分为可并行执行的子任务，最后汇总结果。

java 
@Configuration
public class ParallelOrchestrationConfig {

    @Bean
    public ParallelOrchestrator employeeQueryOrchestrator(
            ReactAgent orderAgent,
            ReactAgent ticketAgent,
            ReactAgent attendanceAgent) {

        return ParallelOrchestrator.builder()
            .addAgent("order", orderAgent)
            .addAgent("ticket", ticketAgent)
            .addAgent("attendance", attendanceAgent)
            .aggregator(results -> {
                // 汇总三个 Agent 的结果
                String orderInfo = results.get("order");
                String ticketInfo = results.get("ticket");
                String attendanceInfo = results.get("attendance");
                return """
                    === 员工综合信息 ===
                    订单: %s
                    工单: %s
                    考勤: %s
                    """.formatted(orderInfo, ticketInfo, attendanceInfo);
            })
            .timeout(Duration.ofSeconds(15))
            .build();
    }
}

并行执行适用于：同时查询多个系统、同时处理多份文档、多维度数据分析。

注意：并行 Agent 各自有独立的上下文窗口。如果子任务需要共享大量上下文，考虑 Sequential 模式。并行模式的收益取决于子任务之间是否真的独立。

    String reviewResult,
    Status status
) {
    public enum Status { PLANNING, EXECUTING, REVIEWING, COMPLETED, FAILED }
}

// Agent 间通过状态对象传递信息
@Bean
public SequentialOrchestrator statefulPipeline(
        ReactAgent planner, ReactAgent executor, ReactAgent reviewer) {

    return SequentialOrchestrator.builder()
        .initialState(CorpAssistState::new)
        .addStage("planner", planner, (state, output) -> state.withPlan(output).withStatus(EXECUTING))
        .addStage("executor", executor, (state, output) -> state.withExecutionResult(output).withStatus(REVIEWING))
        .addStage("reviewer", reviewer, (state, output) -> state.withReviewResult(output).withStatus(COMPLETED))
        .build();
}

常见问题（FAQ）

多 Agent 编排会增加多少延迟？
每个 Agent 调用一次模型。Sequential N 个 Agent 延迟 ≈ N × 单 Agent 延迟。Parallel 模式下延迟 ≈ 最慢子 Agent 延迟。建议：对延迟敏感的场景减少 Agent 数量或将非关键 Agent 异步化。
子 Agent 失败会影响整个管线吗？
默认会。建议为每个子 Agent 设单独的超时和 fallback。失败时可进入"降级模式"：如 Executor 执行失败，管线可跳过 Review 直接进入 Error 处理流程。
Routing Agent 的路由准确率如何？
取决于路由 prompt 的清晰度和模型的智能水平。使用 qwen-max 路由准确率在 90%+。建议在路由指令中给出明确的分类条件和示例。

本章结论

  • 三种编排模式：Sequential（管线）、Parallel（并行）、Routing（路由）。
  • Handoff：控制权移交，实现专长切换。
  • 共享状态：通过 OrchestrationState 在 Agent 间传递上下文。
  • 角色分工：Planner-Executor-Reviewer 是经典的分工模式。
  • 容错设计：每个 Agent 设超时和 fallback，防止级联失败。

    public AgentBasedNodes(
            @Qualifier("queryAgent") ReactAgent queryAgent,
            @Qualifier("approvalAgent") ReactAgent approvalAgent) {
        this.queryAgent = queryAgent;
        this.approvalAgent = approvalAgent;

    @GraphNode(name = "query_employee_info")
    public LeaveState queryEmployee(LeaveState state) {
        // 用 ReactAgent 查询员工信息
        AgentResult result = queryAgent.execute(
            state.employeeId(),
            "查询员工 " + state.employeeId() + " 的请假配额信息"
        );
        return state.withQueryResult(result.output());
    }

    @GraphNode(name = "llm_approve")
    public LeaveState llmApprove(LeaveState state) {
        // 用 ReactAgent 生成审批决策
        AgentResult result = approvalAgent.execute(
            state.employeeId(),

            .withStatus(LeaveState.Status.MANAGER_APPROVED);
    }

常见问题（FAQ）

Graph 和 SequentialOrchestrator 有什么区别？
SequentialOrchestrator 是线性管线，只能顺序执行；Graph 支持条件分支、循环、并行子图。Graph 是更通用的状态机，SequentialOrchestrator 是它的特例。
Graph 状态可以持久化吗？
可以。通过 StatePersister 接口，将 GraphState 持久化到 Redis 或数据库。这对于长时间运行（如审批流程可能持续数天）的工作流至关重要。
一个 Graph 可以嵌套另一个 Graph 吗？
可以。子图（Subgraph）是一种节点类型，内部包含完整的 Graph。适用于复杂的子流程，比如"请假审批"内部嵌套"余额检查"子图。
Conditional Edge 的路由函数抛异常会怎样？
Graph 会进入"错误处理"模式。可在 Graph Builder 中配置 .errorHandler(ErrorStrategy.SKIP_NODE) 或 ErrorStrategy.TO_END 来定义失败行为。

本章结论

  • 状态机模型：Graph 用 Node + Edge + State 显式建模工作流。
  • 条件路由：Conditional Edge 实现通过/驳回等多分支逻辑。
  • 可视化：导出 Mermaid 或使用 SAA Admin Studio 可视化编辑。
  • 嵌套协作：Graph 节点内部可使用 ReactAgent，组合确定性和灵活性。
  • 错误处理：配置 ErrorStrategy 定义节点失败时的行为。

常见问题（FAQ）

Interrupt 后 Agent 状态会丢失吗？
不会。SAA 的 GraphExecutor 会将当前 GraphState 持久化到 StatePersister 中（默认 Redis），恢复时从 Redis 加载完整状态，包括历史消息和变量。
审批超时后消息去哪了？
超时后触发配置的 timeout-action（默认 auto-reject），审批节点返回 false 分支。同时记录超时日志用于审计。建议配置钉钉/企微机器人发送超时告警。
上下文压缩会不会丢失重要信息？
滑动窗口会丢弃最早轮次，但如果关键信息在早期被后续轮次引用过，摘要压缩能保留语义。建议对关键信息（如审批结论、合同金额）做显式的字段提取存入 State，不仅依赖上下文。
哪些操作应该需要人工审批？
黄金法则：写操作要审批，读操作可自动。具体包括：发邮件、创建工单、修改数据、扣款、删除资源。仅查询信息、读取知识库等读操作不需要。

本章结论

  • 审批节点：通过 InterruptNode 暂停 Agent，等待人工确认后恢复。
  • 上下文压缩：滑动窗口 + 摘要 + 截断三种策略防止爆窗。
  • 预算管理：显式设置每层 Token 上限，预留输出空间。
  • 监控告警：上下文使用率超 85% 时主动告警。
  • 审批规则：写操作要审批，读操作可自动。

@Component
public class MCPAuditor {

    private static final Logger log = LoggerFactory.getLogger(MCPAuditor.class);

    /**
     * 记录所有 MCP 工具调用
     */
    @EventListener
    public void onToolCall(ToolCallEvent event) {
        log.info("MCP Tool Call: server={}, tool={}, args={}, user={}",
            event.serverId(),
            event.toolName(),
            sanitize(event.args()),
            event.userId()
        );
        // 写入审计数据库
        auditRepository.save(new AuditRecord(
            event.executionId(),
            event.serverId(),
            event.toolName(),
            event.args(),
            event.result(),
            event.timestamp()
        ));
    }

    private String sanitize(Map args) {
        // 脱敏敏感字段（如密码、token）
        Map safe = new HashMap<>(args);
        safe.keySet().forEach(k -> {
            if (k.toLowerCase().contains("password")
                || k.toLowerCase().contains("secret")) {
                safe.put(k, "****");
            }
        });
        return safe.toString();
    }
}

常见问题（FAQ）

MCP 和 @Tool 的关系是什么？
@Tool 是 SAA 本地工具定义方式，MCP 是远程工具协议。两者最终都通过 ToolCallback 接口暴露给 Agent。最佳实践：本地简单逻辑用 @Tool，外部系统集成用 MCP。详见 ReactAgent @Tool。
MCP Server 如果挂了怎么办？
Agent 调用时返回错误作为 observation，模型会自行决定是否重试。建议为关键 MCP Server 配置多副本和健康检查。SAA 的 Resilience4j 配置可包装 MCP 调用，参见 容灾、限流与降级。
stdio 和 SSE 模式如何选择？
stdio 适合部署在 Agent 同机或同容器的工具（CLI、脚本）。SSE 适合跨网络、独立部署的微服务。SSE 是生产推荐模式，支持独立扩缩容和权限管控。
MCP 工具返回的数据量有上限吗？
建议 MCP Tool 返回控制在 10KB 以内。大量数据应该通过 Resource 能力暴露，Agent 按需读取。返回超大数据可能撑爆上下文窗口。

本章结论

  • MCP 协议：标准化工具接入协议，支持 Tools/Resources/Prompts 三种能力。
  • 传输模式：stdio 适合本地，SSE 适合远程分布式部署。
  • ToolRegistry：自动发现并聚合所有 MCP Server 的工具。
  • 安全机制：白名单 + 鉴权 + 审计日志三层防护。
  • 自定义 Server：通过 MCPServer Builder 快速创建 MCP 服务。

A2A 协议概念

a2a protocol 定义了两个 Agent 之间通信的标准化方式。不同于传统的 RPC（远程过程调用），A2A 传递的是"任务描述"，由接收方 Agent 自行决定如何执行。

维度传统 RPCA2A 协议
通信内容 方法名 + 参数 任务描述 + 上下文
执行方自由度 低（必须实现接口） 高（自行决定如何完成任务）
结果返回 同步返回值 同步或异步 + 回调
错误处理 异常抛出 任务状态 + 重试策略

@RestController
@RequestMapping("/a2a")
public class FinanceAgentController {

    private final ReactAgent financeAgent;

    public FinanceAgentController(ReactAgent financeAgent) {
        this.financeAgent = financeAgent;
    }

    // A2A 协议：接收任务并返回结果
    @PostMapping("/execute")
    public Mono executeTask(@RequestBody A2ATask task) {
        return Mono.fromFuture(financeAgent.execute(task.userId(), task.description()))
            .map(result -> new A2AResult(
                task.taskId(),
                result.output(),
                result.status(),
                result.tokensUsed()
            ));
    }

    // A2A 协议：查询 Agent 能力
    @GetMapping("/capabilities")
    public List getCapabilities() {
        return List.of("query_invoice", "query_budget", "approve_payment");
    }

    // A2A 协议：健康检查
    @GetMapping("/health")
    public Map health() {
        return Map.of("status", "UP", "agent", "finance-agent");

通过 A2A 调用远程 Agent

rpc agent 不是真正的 RPC，而是 A2A 客户端的远程调用方式。

java 
@Component
public class A2AClient {

    private final DiscoveryClient discoveryClient;
    private final WebClient webClient;

    public A2AClient(DiscoveryClient discoveryClient, WebClient.Builder webClient) {
        this.discoveryClient = discoveryClient;
        this.webClient = webClient.build();
    }

    /**
     * 通过 A2A 调用远程 Agent
     */
    public Mono callAgent(String serviceName, A2ATask task) {
        // 从 Nacos 发现服务实例
        List instances = discoveryClient.getInstances(serviceName);
        if (instances.isEmpty()) {
            return Mono.error(new AgentNotFoundException("未找到 Agent: " + serviceName));
        }

        // 轮询选择实例
        ServiceInstance instance = instances.get(ThreadLocalRandom.current().nextInt(instances.size()));
        String url = "http://%s:%d/a2a/execute".formatted(instance.getHost(), instance.getPort());

        return webClient.post()
            .uri(url)
            .bodyValue(task)
            .retrieve()
            .bodyToMono(A2AResult.class)
            .timeout(Duration.ofSeconds(30));
    }
}

// 使用：工单 Agent 调用财务 Agent
@Service
public class TicketOrchestrator {

    private final A2AClient a2aClient;

    public Mono processExpenseTicket(String userId, String ticketId) {
        // 工单 Agent 自己的处理
        // ...

        // A2A 调用远程财务 Agent
        A2ATask task = new A2ATask(
            "task-" + ticketId,
            userId,
            "查询预算余额，预算编码：BUDGET-2026-Q2"
        );

        return a2aClient.callAgent("finance-agent", task)
            .map(A2AResult::output);

}

microservice agent 模式将 Agent 视为微服务中的一等公民，享受所有 Spring Cloud 基础设施：

  • service mesh：通过 Istio 等实现流量管理和安全策略
  • agent registry：Nacos 中 Agent 注册的特殊标记（metadata）
  • 负载均衡：Spring Cloud LoadBalancer 轮询 Agent 实例
  • 熔断保护：Sentinel 或 Resilience4j 包装 A2A 调用

A2A vs 多 Agent 编排

维度多 Agent 编排（同 JVM）A2A（跨服务）
部署方式 同进程，共享内存 独立进程/容器，网络通信
状态共享 直接共享 State 对象 通过序列化传递上下文
延迟 毫秒级（方法调用） 百毫秒级（HTTP）
服务发现 不需要 Nacos / Kubernetes
适合场景 同团队、同服务内 跨团队、跨集群
不推荐
  • 同 JVM 内的 Agent 间也用 HTTP 调用
  • 跨服务的 Agent 间用 RPC 传递内部对象
  • A2A 调用不做超时和重试
推荐
  • 同 JVM 用编排，跨服务用 A2A
  • A2A 传递序列化的任务描述
  • A2A 调用配置超时 + 熔断 + 重试

java 
@Configuration
public class AgentMeshConfig {

    // 注册所有远程 Agent 的能力描述
    @Bean
    public AgentMesh agentMesh(DiscoveryClient discoveryClient, A2AClient a2aClient) {
        AgentMesh mesh = new AgentMesh();

        // 从 Nacos 发现所有标记为 agent 的服务
        discoveryClient.getServices().stream()
            .filter(svc -> svc.contains("-agent"))
            .forEach(svc -> {
                List instances = discoveryClient.getInstances(svc);
                if (!instances.isEmpty()) {
                    // 获取 Agent 能力列表
                    List capabilities = a2aClient.fetchCapabilities(svc);
                    mesh.register(new RemoteAgent(svc, capabilities));
                }
            });

        return mesh;
    }

    // 路由 Agent：根据任务类型路由到对应的远程 Agent
    @Bean
    public ReactAgent routingAgent(ChatModel chatModel, AgentMesh mesh) {
        return ReactAgent.builder()
            .chatModel(chatModel)
            .systemPrompt("""
                你是 CorpAssist 路由 Agent。
                根据任务类型将任务分派给对应的专业 Agent：

                %s
                """.formatted(mesh.describe()))
            .maxSteps(5)
            .build();
    }
}

常见问题（FAQ）

A2A 和 MCP 有什么关系？
MCP 是 Agent 调用工具的协议（Agent→Tool），A2A 是 Agent 之间通信的协议（Agent→Agent）。两者互补：Agent 用 MCP 调用外部工具，用 A2A 与其他 Agent 协作。CorpAssist 同时使用两者。
A2A 任务超时了怎么处理？
建议设 30 秒超时 + 3 次重试。超时后 taskId 不变但新 executionId，接收方 Agent 需支持幂等处理。使用 Nacos 健康检查可以快速剔除故障实例。
Agent 需要独立数据库吗？
A2A 模式下每个 Agent 服务应有自己的持久化存储（状态、历史）。通过 Nacos 配置中心管理数据源配置。Agent 间不共享数据库，只通过 A2A 消息传递数据。
如何保证 A2A 通信的安全性？
建议：内网部署 + Nacos 鉴权 + Spring Security + Token 认证。敏感 Agent 间启用 mTLS。审计日志记录所有 A2A 调用。

本章结论

  • A2A 协议：Agent 间通过任务描述通信，不同于 RPC。
  • Nacos 注册发现：Agent 作为标准 Spring Cloud 服务注册。
  • 分布式协作：跨集群 Agent 通过 A2A 互相调用。
  • 微服务融合：复用 Nacos/Sentinel/Gateway 等基础设施。
  • Agent Mesh：自动发现和注册远程 Agent 形成协作网络。
