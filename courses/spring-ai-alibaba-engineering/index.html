答题：S3 Agent 五连问框架

CorpAssist · S3 办公流程 Agent 员工在门户提交「帮我整理上周会议纪要并创建 Jira 子任务」：Agent 需检索 Confluence、调用 Jira API、经主管 HITL 审批后写入。Java 轨用 Spring AI Alibaba ReactAgent + Graph；Python 轨用 LangGraph 做 Prompt/工具实验，OpenAPI 对齐后 Java 侧上线。
Q1：Agent 与普通大模型应用的区别？

答题框架（总-分-总）

总：「普通应用是单轮或固定 RAG 链；Agent 是多步规划-执行-观察循环，能调用外部工具并维护状态。」

分：CorpAssist S3 中，Chat 只回答制度问题；Agent 会分解任务（检索纪要 → 抽取 action items → 调 Jira createIssue），每步有 步数预算 与 Trace。Java 用 SAA Graph 持久化 checkpoint；Python 用 LangGraph 同构状态机。

总：「区别在可控的多步编排与工具契约，不是模型更大。」

Q3：任务规划有哪些方法？CorpAssist 怎么选？
方法适用CorpAssist S3 示例
ReAct 步骤 ≤10、需实时反馈 单页纪要摘要 + 创建 1 个 Jira
Plan-Execute 步骤多、可批处理 周报汇总 5 个部门 Confluence
SAA Graph 固定 DAG 合规强、路径固定 采购审批：检索制度 → 填表 → HITL

话术：「我们不用一种范式打天下：<10 步 ReAct；跨系统批处理 Plan-Execute；强合规用 Graph 固定节点 + HITL。选型看步数、实时性、审计要求。」

Q4：记忆系统如何设计？过载与不准怎么解？

三层记忆：工作记忆（当前 Graph state）、会话记忆（Redis ChatMemory，最近 N 轮）、长期记忆（VectorStore 存用户偏好与历史任务摘要）。

过载：设置 记忆预算——超过 token 阈值则摘要压缩；长期记忆检索 Top-K + 时间衰减，禁止整段会话塞 prompt。

不准：记忆写入需结构化（任务 ID、结论、待办）；检索时用 metadata filter（userId、tenantId）；与 RAG 制度库分离索引，避免「上次聊天」污染「公司制度」召回。

Q5：工具调用失败怎么办？

分层：(1) 参数校验 → 返回模型可理解的错误 JSON；(2) 业务异常 → 重试 1 次 + 换表述；(3) 系统异常 → 熔断 + 降级文案 + 转 HITL。

关键：工具返回需 截断（如 Confluence 正文 max 2k chars）；失败计入步数预算，连续 2 次同工具失败强制结束并告警。

60 秒 STAR 模板

S：S3 纪要 Agent 步数飙到 15 轮仍未创建 Jira。T：24h 内恢复 SLA。A：Trace 发现 Confluence 工具返回全文 40k 字；加 truncation + 步数上限 8 + 失败转 HITL。R：P95 从 90s 降到 18s，任务完成率 94%。

坑：S3 productionPitfalls 与 defense

outline-specs 为 S3 定义四个 生产坑。面试时必须能说出现象 → 根因 → 检测 → 修复。

生产坑典型现象检测信号工程解法
规划 >3 步无执行-观察-反思 Agent「空想」10 轮不调工具 Trace 中 LLM 步无 tool_calls；步数↑ 完成率↓ 强制 ReAct 循环；Plan 后必须 Execute 节点；maxSteps=8
工具假执行 / 返回过长 模型 hallucinate 已创建 Jira；或上下文被 40k 字淹没 审计对账 Jira 无记录；prompt token 暴增 工具必须真实 HTTP + 返回结构化 JSON；truncate + 摘要
记忆无限堆积 长会话后答非所问、检索到三个月前闲聊 memory retrieve 延迟↑；无关 chunk 进入 prompt 滑动窗口 + 摘要；长期记忆 TTL；检索 filter
工具失败无降级 Jira 超时后 Agent 空转至步数上限 同工具 error 重复；用户无反馈 Resilience4j 熔断；2 次失败 → HITL；语义缓存 读路径降级
坑 · 反模式
  • 只调大 max_tokens，不限制步数
  • 工具返回原始 HTML/PDF 全文
  • ChatMemory 无限 append
  • catch Exception 后返回空字符串给模型
坑 · 正确做法
  • StepBudget + 每步必须 tool 或 final answer
  • Tool 层截断 + 引用 doc_id
  • 记忆分层 + 租户隔离
  • 结构化 error + 转 HITL / 人工队列

// CorpAssist S3 · 步数预算 + 工具失败降级（示意）
@Configuration
public class AgentGuardConfig {
    @Bean
    StepBudgetAdvisor stepBudgetAdvisor() {
        return StepBudgetAdvisor.builder()
            .maxSteps(8)
            .onExceeded(ctx -> ctx.transitionTo("hitl_queue"))
            .build();
    }

    @Bean
    ToolCallAdvisor toolResilienceAdvisor() {
        return ToolCallAdvisor.builder()
            .maxRetries(1)
            .truncateResultChars(2048)
            .onPersistentFailure((tool, err) -> ToolResponse.error(
                "TOOL_FAILED", tool + ": " + err.getMessage() + "；已转人工审批"))
            .build();
    }
}

判断练习

  1. 面试官问「我们全用 LangGraph 就行，为什么还要 Spring AI Alibaba？」——只回答「公司规定 Java」是否足够？

    参考答案

    不够。 应讲双栈分工：Python 实验与评测、Java 门户集成/Admin/HITL/IAM/QPS/审计；OpenAPI 契约保证可替换。体现架构取舍而非组织惯性。

  2. Agent 任务完成率 92% 但平均步数 14，是否可接受？

    参考答案

    通常不可接受。 高步数意味着成本与延迟飙升、偏航风险大。应设步数预算（如 8），优化 Tool 返回与 Plan-Execute，完成率略降但 P95 与单任务成本显著改善。

  3. 工具失败时返回空字符串，让模型「自己想办法」，是否合理？

    参考答案

    不合理。 易诱发假执行与 hallucination。应返回结构化 error、计入重试/熔断，连续失败转 HITL——对应 S3 坑「工具失败无降级」。

  • 索引与查询分离：全量 reindex 走 Python 批处理；S3 Agent 在线只读 VectorStore。

熔断：失败率、慢调用与半开

DashScope 超时或 429 会拖垮 Tomcat 线程池。对 ChatModel 外包一层 circuit breaker：失败率超阈则快速失败，走 fallback 或排队；半开期放少量探测请求恢复。

@Service
public class ResilientS3ChatService {
  private final ChatModel primary;
  private final ChatModel fallback;
  private final CircuitBreaker breaker;

  public String chat(String userMessage) {
    Supplier<String> supplier = CircuitBreaker
        .decorateSupplier(breaker, () ->
            primary.call(new Prompt(userMessage)).getResult().getOutput().getText());
    return Try.ofSupplier(supplier)
        .recover(CallNotPermittedException.class, e -> invokeFallback(userMessage))
        .get();
  }
}

// CacheAdvisor 示意：写 Tool 路径 bypass
public class S3CacheAdvisor implements CallAroundAdvisor {
  @Override
  public AdvisedResponse aroundCall(AdvisedRequest req, CallAroundAdvisorChain chain) {
    if (req.toolsScheduled() || containsWriteTool(req)) {
      return chain.nextAroundCall(req);
    }
    return cache.getOrCompute(cacheKey(req), () -> chain.nextAroundCall(req));
  }
}

RateLimiter 与熔断应设在网关还是 Agent 服务？

网关做粗粒度租户/全局 QPS；Agent 服务做 DashScope 配额保护与语义缓存。两层互补，不要只在网关限流却仍在应用内打满 429。

通义栈视角：容灾与百炼配额

DashScope 429 与下游 OA 超时是 S3 Agent 最常见的外部故障。Java 轨用 Resilience4j + 语义缓存 + 多模型 Fallback；Python 轨用 tenacity + Redis。双栈须共享降级文案与缓存 Key 规范（含 model 版本）。

不推荐
  • 只在网关限流，应用内仍打满 DashScope 429
  • Java 已切 turbo、Python 仍调 plus，用户体验分裂
  • catch 后返回空字符串，诱发 Agent 假执行
推荐
  • 网关粗粒度 QPS + Agent 服务 DashScope 配额保护
  • plus → turbo → 规则/异步，双栈同一降级文案
  • 熔断 OPEN 告警 + 语义缓存读路径兜底

本章结论

  • 熔断：DashScope 与下游 OA 分开实例；8s 超时 + 50% 失败率起步。
  • 缓存：只读 FAQ/availability 可缓存；写 Tool 与 PII 强制 bypass。
  • 多模型：plus → turbo → 规则/异步；与 Admin DSL model 字段一致。

连接池：DashScope HTTP 出口

Spring AI Alibaba 底层通过 HTTP 调百炼 API。默认连接池过小会导致 QPS 上升时大量等待连接；过大则占满 DashScope 侧配额。应在应用配置显式声明 http connection pool，并与 Resilience4j 超时联调。

CorpAssist · S3 · 早高峰 500 并发短问 FAQ，P99 从 2s 飙到 12s。排查发现 OkHttp maxRequests 默认 64，连接等待占 8s。调池后 P99 回到 3s——无需先扩 Pod。
