本章先记住 3 件事

多 Agent 协作模式

CorpAssist 场景：知识库团队要在两周内交付「合同条款 RAG 检索 + 引用溯源」接口。Tech Lead 用 Planner Agent 拆任务，Coder Agent 改 FastAPI 路由，Reviewer Agent 对照 OpenAPI 与 AI 生成文档与 Schema 做契约检查。

自定义 Skill/Command 开发

Skill 是带触发描述的操作手册；Command 是斜杠命令入口。二者应引用仓库内的 AGENTS.md、context/ 与 .cursor/rules，避免模型「发明」项目结构。

反例：巨型 Skill
  • 一个 Skill 覆盖「从需求到上线」全部步骤
  • 无输出模板，每次 diff 格式不同
  • 未声明禁止改动的目录（如 infra/）
推荐：单一职责 Skill
  • corpassist-pytest-gen：只生成 pytest，必须跑 uv run pytest
  • 固定输出：测试文件路径 + 覆盖场景列表
  • 显式引用 AI 生成测试用例 中的 Fixture 约定

## 输入
- Issue 链接或 diff 范围
- 目标模块：`app/rag/` 或 `app/retrieval/`

## 步骤
1. 阅读 `context/tech/services/corpassist-python.md`
2. 只修改列出的文件；禁止改 `deploy/`
3. 输出：代码 diff 摘要、pytest 命令、OpenAPI 路径是否变更

## 验收
- `uv run ruff check app/`
- `uv run pytest tests/rag/ -q`

Command 适合一键动作，例如 /gen-changelog 只读最近 merge commit 生成发布说明，与 规则与约定驱动开发 中的 Conventional Commits 对齐。团队共享的 Cursor Skill 应通过内部 PR 评审，版本号写入 manifest，避免每人维护冲突副本。

Workflow 编排最佳实践

工作流文件（如 .github/workflows/ai-dev.yml）描述何时调用 Agent；Workflow DAG 中每个 stage 的 artifact 应可机器校验。Skill 描述怎么做。二者分离可避免把 Prompt 硬编码进 YAML。

委派子任务时使用 Subagent delegation 规范：只读、限时、输出引用路径。合并前由 CI agent hook 把评审结论写入 Check Run，供安全审计关联。

状态机显式化

为每个阶段定义输入/输出 artifact：design.md → patch.diff → test-report.xml。

上下文预算

Coder Agent 只挂载相关目录；Planner 可读 Issue，不读 node_modules。

失败可重入

步骤 id 幂等；重跑 Tester 不应重复提交 MR。

与 CorpAssist CI/CD 集成

Agent 工作流在 CI 中应作为可选加速层，合并门禁仍以确定性工具为准。推荐在 PR 打开时运行「AI 辅助评审」，在 main 合并前强制 pytest + 静态检查。

yaml 
# .github/workflows/corpassist-pr.yml（节选）
on:
  pull_request:
    paths: ["app/**", "tests/**"]
jobs:
  deterministic:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: uv sync --frozen
      - run: uv run ruff check .
      - run: uv run pytest --cov=app --cov-fail-under=80
  ai_assist:
    needs: deterministic
    if: github.event.pull_request.draft == false
    runs-on: ubuntu-latest
    steps:
      - name: Invoke corpassist review agent
        env:
          CURSOR_API_KEY: ${{ secrets.CURSOR_API_KEY }}
        run: |
          python scripts/ci_agent_review.py \
            --skill corpassist-pr-review \
            --pr ${{ github.event.pull_request.number }}

CI agent hook 必须把模型输出写入 PR Comment 或 Check Run，并附带 request_id 便于与网关日志关联。敏感仓库应使用自托管 Runner + 私有模型端点（详见 企业级 AI 工具链安全与合规）。

Agent 生成的 patch 能否自动 merge？
不建议。CorpAssist 生产变更至少保留：① 人类 Approve；② 确定性测试通过；③ 对安全/契约类 diff 的 CODEOWNERS 审查。Agent 可自动开 Draft PR，不可自动合入 main。

Subagent 查依赖：何时用、何时不用

当 Coder 需要确认第三方库 API（如 langchain 某版本签名）时，委派只读 Subagent 读取 lock 文件与文档，避免把整库塞进主会话。CorpAssist 约定：Subagent 超时 120s、最多 2 次重试，输出必须为「引用路径 + 结论」而非长文复述。

Planner 输出 tasks.json 应包含哪些字段？
建议最小字段：id、title、files_allowlist、acceptance_commands、depends_on。禁止 Planner 直接写「实现细节代码」，避免 Coder 失去检索上下文的机会。

AI 代码接受率统计

CorpAssist 场景：Python 团队在 Cursor 中生成 RAG 检索逻辑，部分建议被 Tab 接受、部分在 Chat 中被复制。Tech Lead 需要周报：「本周 AI 建议接受率是否下降？是否因模型切换？」

AI acceptance rate 是治理层最常被问到的指标之一。接受率常见口径：

指标分子分母用途
Tab 接受率接受的 inline 建议行数展示的建议行数衡量补全质量
Chat 采纳率合并进分支的 AI 生成行（git blame 标记）AI 会话产生的候选行衡量对话生成可用性
PR AI 占比标记为 Co-authored-by: AI 的 commit 行PR 总新增行治理与审计

# scripts/metrics/ai_lines.py — 统计 PR 中 AI 辅助行（示意）
import subprocess
import re

AI_TRAILER = re.compile(r"^AI-Assisted:\s*(\d+)$", re.M)

def ai_lines_in_range(base: str, head: str) -> dict:
    log = subprocess.check_output(
        ["git", "log", f"{base}..{head}", "--format=%B%n---"],
        text=True,
    )
    total_ai = 0
    for block in log.split("---"):
        m = AI_TRAILER.search(block)
        if m:
            total_ai += int(m.group(1))
    diffstat = subprocess.check_output(
        ["git", "diff", "--shortstat", base, head], text=True
    )
    # 解析 insertions...
    return {"ai_lines": total_ai, "raw": diffstat.strip()}

生成代码缺陷密度追踪

Defect density 在此定义为：每千行变更在发布后 14 天内关联的 P1/P2 缺陷数。需给 commit / PR 打标签 ai-assisted vs human-only。若密度连续高于基线，应联动 Model-as-Judge 代码质量门禁 收紧 rubric，而非简单禁用 AI 工具。

缺陷归因数据流
flowchart LR
  PR[Pull Request\n标签 ai-assisted] --> Merge[合并 main]
  Merge --> Deploy[发布 CorpAssist]
  Deploy --> Sentry[Sentry / 工单]
  Sentry --> Join[按 commit SHA 关联]
  Join --> Report[缺陷密度报表\nper 1k LOC]

从 PR 标签到工单系统，闭环计算 AI vs 人工缺陷密度。

反例
  • 把所有线上 Bug 都算在「AI 不好」上，无对照组
  • 不区分模块（RAG vs 网关配置）
推荐
  • 同模块、同周期对比 AI-assisted 与 human-only PR
  • 结合 AI 生成代码的质量管控 中的静态检查通过率

度量看板与告警

看板建议四层：用量（会话/Token）、效率（PR 周期）、质量（缺陷密度、回滚率）、合规（策略违规次数）。

yaml 
# alerting/corpassist-ai-metrics.yaml（示意）
groups:
  - name: corpassist_ai_governance
    rules:
      - alert: AiDefectDensityHigh
        expr: defect_density_ai_14d > 1.5 * defect_density_human_14d
        for: 7d
        labels:
          team: corpassist-python
        annotations:
          summary: "AI 辅助变更缺陷密度连续 7 天高于人工基线 50%"
      - alert: AiAcceptanceRateDrop
        expr: ai_tab_accept_rate_7d < 0.25
        for: 3d
        annotations:
          summary: "Tab 接受率跌破 25%，检查模型或规则变更"

Alert threshold 应经两周基线校准；避免上线首日误报风暴。告警路由到 Tech Lead + 工具管理员，而非全员。

接受率下降一定是模型变差吗？
不一定。常见原因：① 同事修改了 .cursor/rules 导致建议偏离项目风格；② 本周任务类型变化（更多 infra yaml，补全命中率天然低）；③ 开发者更谨慎，拒绝低置信建议。应结合会话备注与规则变更日志排查。

Model-as-Judge 不等于「再跑一次 Coder」。架构原则：

  • 输入最小化：只传 diff、hunk 上下文、关联测试；不传全库。
  • 输出结构化：强制 JSON Schema，便于 Gate 解析。
  • 温度低：评审用 temperature=0，减少随机误报。

同会话会「自我辩护」已生成代码；独立 Judge 模拟他人在 CR，且便于换更强模型专做评审。

误报与漏报调优

误报高发
  • Rubric 含糊：「代码要好」
  • 未提供项目约定（FastAPI 依赖注入模式）
  • Judge 模型过旧，不理解 Pydantic v2
调优手段
  • 在 rubric 中嵌入 context/tech/ 摘录
  • 黄金样本回归 + 每周抽检人工标注
  • 对 stylistic 维度降权，安全/正确性加权
现象可能原因动作
误报：风格挑剔可维护性权重过高降权或改为 warning
漏报：安全洞diff 上下文不足扩大 hunk 上下文或附加 SARIF
不稳定：同 PR 两次分不同温度 > 0固定 temperature + seed
Judge 失败但人类认为可合并，如何处理？
使用 judge-override 标签 + CODEOWNERS Approve，并在合并说明记录理由。该 PR 进入下周黄金样本评审：若确为误报，更新 rubric 或添加 negative example。

markdown

- 1 分：返回无溯源字段或静默吞掉空检索

同一 PR 多次 push 会重复计费 Judge 吗？
会。优化：仅对 synchronize 后变更的 hunk 增量评审；或设置「同一 PR 24h 内最多 N 次完整 Judge」。节省成本同时要在注释说明「增量评审可能漏掉跨文件关联」。
Judge 模型比 Coder 弱会怎样？
漏报上升。CorpAssist 实践：Judge 可用同族更强模型或专用「评审微调」权重；Coder 可用较快模型。禁止为省成本让 Judge 弱于 Coder。

  • Judge 替代 pytest：行为正确性必须以测试为准。
  • rubric 抄百科：缺少 CorpAssist tenant/RAG 专项，漏报多。
  • 无 golden：换模型后质量漂移无人知。
  • override 无审计：门禁被架空，合规风险上升。

FAQ

能否让开发自选 Cursor 或 Windsurf，Rules 仍统一？
可以，但须规定Rules 真源在 Git（.cursor/rules + CLAUDE.md），IDE 差异只在于 Agent 壳层；POC 确认两者对同一 Ticket 接受率差距可接受。
Copilot 与 GitHub Enterprise 集成是否是决定性优势？
若集团已 All-in GitHub EE，Copilot 的审计与策略下发更顺；但若主力在 GitLab，应优先验证 Cursor/Windsurf 的 SSO 与日志导出，而非强行换代码托管。
私有化部署是否等于「代码绝不出网」？
不一定：本地模型仍可能有日志、遥测与插件更新通道；须读安全评估问卷中的出站连接清单。
常见误区
  • 把 Agent 当自动 merge — 生成代码必须过 Review 与 CI（见 AI 生成代码的质量管控）。
  • 只采购不治理 — 无 Rules、无覆盖率门禁，AI 代码比例越高风险越大。
  • 忽视终端 Agent — Claude Code 类工具需限制 rm -rf 等危险命令。
  • 忽略 GitLab 场景 — 非 GitHub 托管时 Copilot EE 集成优势减弱，须单独评估 SSO。

Context 工程 for AI Coding

Context 工程 决定 Agent「看见什么」。CorpAssist 仓库模块多，错误 Context 会导致 AI 引用过时 DTO 或错用 sync HTTP 阻塞 async 路由。

Context 三层模型

L1 · 持久 Rules

.cursor/rules、CLAUDE.md：编码规范、禁止项、测试命令。每次会话自动注入。

L2 · 任务 Context 包

本 Ticket 相关：schema、router、2–3 个测试、OpenAPI 片段。用 @file 或显式路径列表。

L3 · 动态检索

Agent 按设计要点自行 grep / 读文件；须在 Rules 里限制目录（如勿读 config/prod/）。

  • 写入阶段 Context：chunker.py + 样本 JSONL
  • 检索阶段 Context：retriever.py + 向量库 mock fixture
  • 响应阶段 Context：schemas/query.py + OpenAPI

本章结论

  • 闭环四步：需求验收 → 设计要点 → Context 包生成 → diff/pytest/人审；缺设计要点返工率高。
  • Context 工程：L1 Rules + L2 任务包 + L3 受控检索；优先 schema 与测试。
  • Rules 最佳实践：短、可验证、带 globs；CLAUDE.md 做索引而非百科全书。
  • CorpAssist 结合：Ticket 绑 Context 包与验证命令；与团队 Rules 和 Review 流程一体。

可接受的 AI 代码比例

AI 代码可接受比例 不是「越高越好」，而是与缺陷密度、Review 容量平衡。建议 CorpAssist 团队度量以下指标（advanced 阶段将建看板）。

接受率

AI 建议中被开发者保留的行数 / 总建议行数。过低说明 Prompt 差；过高且缺陷多说明 Review 虚设。

AI 行占比

合并代码中标记或估算的 AI 生成比例。核心路径（鉴权、计费）宜更低。

AI 缺陷密度

合并后 30 天内与 AI 变更相关的 bug 数 / KLOC。用于调 Prompt 与 Rules。

团队政策示意（非硬性标准）

  • 样板代码（DTO、测试脚手架）：AI 行占比可达 70–80%，须过静态检查。
  • RAG 核心检索：AI 辅助 <40%，算法改动需双人 Review。
  • 安全/密钥相关：禁止 AI 直接 merge，仅允许生成测试或文档草稿。

        ac.headers.update({"X-Api-Key": "test-key"})
        yield ac

AI 生成的 patch 路径总是报错 ImportError？

parametrize 与 fixture 组合顺序？
fixture 先注入 client/mock，parametrize 提供输入；不要在 parametrize 里构造 AsyncClient。AI 生成后检查每个用例是否独立，避免共享 mutable mock 状态。

docstring 自动化

FastAPI 路由的说明来自函数 docstring 与 response_model。对 CorpAssist rag/query 端点，AI 可根据 Schema 生成包含参数、返回值、错误码的 docstring，供 Swagger UI 与 Redoc 展示。

场景：新增 stream=true 查询参数后，需在同一个 PR 更新 docstring 与示例 JSON，否则 Java 同事在 Swagger 中看不到新字段。
python
from fastapi import APIRouter, Depends
from app.rag.schemas import RagQueryRequest, RagQueryResponse
from app.rag.service import RagPipeline, get_pipeline

router = APIRouter(prefix="/v1/rag", tags=["rag"])


@router.post("/query", response_model=RagQueryResponse)
async def rag_query(
    body: RagQueryRequest,
    pipeline: RagPipeline = Depends(get_pipeline),
) -> RagQueryResponse:
    """企业知识库 RAG 查询（CorpAssist S1）。

    Args:
        body: 查询体。query 为自然语言问题；top_k 默认 3，最大 20。

    Returns:
        answer: 模型综合检索片段后的回答。
        citations: 引用列表，含 doc_id 与 snippet，供前端溯源。

    Raises:
        HTTPException 401: 缺少或无效 X-Api-Key。
        HTTPException 422: query 为空或超长。
    """
    return await pipeline.arun(query=body.query, top_k=body.top_k)

from pydantic import BaseModel, Field


class RagQueryRequest(BaseModel):

文档 FAQ

Swagger 上 examples 不显示？

检查是否写在 model_config json_schema_extra；FastAPI 版本是否支持；是否被自定义 openapi() 覆盖。重新 export 并在 /docs 强制刷新。

AI 生成的 docstring 与参数名不一致？

要求 Prompt 附带函数签名 AST 或粘贴完整 def 行；生成后跑 ruff D417（参数文档缺失）修复。

openapi-diff 报 incompatible 但确认是安全变更？

走 ADR 更新 baseline 文件，并在 PR 描述写 Java 回归结果；禁止在未通知 BFF 的情况下合并。

@dataclass
class RagPipeline:

    llm: "ChatClient"

        chunks = [c for c in chunks if c.text and c.text.strip()]

S3 办公自动化需要「查知识库 → 填表单 → 人工审批」多步流。可用 AI 生成 LangGraph 状态图草图，再按团队 Rules 加 interrupt 与步数上限。对外仍通过 FastAPI 暴露同步/流式接口，与 Java BFF 契约一致。

# app/agent/graph.py 示意 — AI 生成后须人工加预算与白名单
from langgraph.graph import StateGraph, END

def build_corpassist_agent():
    g = StateGraph(dict)
    g.add_node("retrieve", retrieve_node)
    g.add_node("draft_form", draft_form_node)
    g.add_node("human_approve", human_approve_node)
    g.set_entry_point("retrieve")
    g.add_edge("retrieve", "draft_form")
    g.add_conditional_edges("draft_form", need_approval, {"yes": "human_approve", "no": END})
    g.add_edge("human_approve", END)
    return g.compile(interrupt_before=["human_approve"])

评测脚本自动编写

RAG 质量不能靠「感觉」。让 AI 从 PRD 抽取 30 条 golden Q/A，生成 evals/rag_golden.jsonl 与跑分脚本，输出 faithfulness、 citation 命中率。

python
# scripts/run_rag_eval.py
import json, asyncio
from app.rag.pipeline import RagPipeline

async def main():
    pipeline = RagPipeline.from_settings()
    rows = [json.loads(l) for l in open("evals/rag_golden.jsonl")]
    ok = 0
    for row in rows:
        out = await pipeline.arun(query=row["question"], top_k=3)
        hit = any(k in out["answer"] for k in row.get("must_contain", []))
        ok += int(hit)
    print(f"hit_rate={ok/len(rows):.2%}")

if __name__ == "__main__":
    asyncio.run(main())
