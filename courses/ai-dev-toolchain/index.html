面试可能问

为什么 Cursor 的上下文理解比 Copilot 强？底层技术差异是什么？

Cursor 基于 VS Code Fork，可以完全控制编辑器的索引系统。它维护全项目的索引（AST + 符号表），通过 @ 符号可以精确引用文件/函数/类/接口。Copilot 作为插件只能通过 VS Code 的 API 获取当前文件上下文和有限的项目符号。具体差异：1) Cursor 的索引包含 10 万个符号的实时 AST；2) Cursor 的 @ 引用支持跨文件；3) Cursor 的 Compose 模式可以同时修改 10+ 文件并保持一致性。

Claude Code 的终端模式适合什么场景？不适合什么场景？

适合：1) CI/CD 自动化——在 GitHub Actions 中运行 Claude Code 自动修复 lint 错误；2) 大规模重构——"将所有 FeignClient 替换为 RestClient"这样的全仓操作；3) 复杂调试——OOM 分析、线程死锁排查（终端可以运行 jstack/jmap）；4) 批量任务——批量生成测试、批量重命名。不适合：1) 日常编码补全（Tab 补全场景）；2) 需要 GUI 调试（断点/变量观察）；3) 前端开发（无法预览 UI 效果）。

企业部署 AI 编码工具最大的安全风险是什么？怎么缓解？

最大风险：源代码通过 AI 工具的对话历史上传到第三方服务器（即使禁用训练，对话中的代码仍存储在厂商服务器）。缓解措施：1) 选择 Enterprise/Business 版（有数据保护协议）；2) 禁用对话历史存储（Cursor Business 版支持）；3) 敏感代码脱敏——在工具中使用变量名替换（用 "api_key" 替代真实的 key）；4) 定期审计——检查对话历史中是否包含敏感信息；5) 结合私有部署方案——敏感模块只使用本地 LLM（Ollama），非敏感模块用云端工具。

为什么学这章

用 AI 编码工具不是把需求丢给 AI 就完事了。质量取决于 Prompt 工程——如何提供正确的 Context、设置规则、审查生成结果。CorpAssist 团队原先用 Copilot 直接对话，生成代码经常需要返工。建立结构化的 Prompt 驱动开发流程后，代码一次性通过率从 45% 提升到 82%。

学完后能做什么

建立需求→设计→生成→审查的闭环；用 .cursorrules/claude.md 配置项目级 Context；设计有效的 Coding Prompt 模板；评估生成代码的质量并迭代 Prompt。

本章要点速记

1) 四步闭环：需求→设计→生成→审查，缺一不可；2) Context 工程 = 项目上下文 + 规则 + 示例 + 约束；3) .cursorrules 是 Cursor 的全局规则文件，claude.md 是 Claude Code 的规则文件；4) 好的 Prompt 包含：角色/任务/约束/输出格式/示例；5) 审查阶段除了代码正确性还要检查 API 版本和安全。

Context 工程 for AI Coding

Context 工程是 AI 编码的核心技能。提供给 AI 的上下文质量直接决定生成代码的质量。好的 Context 包含四层：项目背景、规则约束、代码示例、任务说明。

Context 层次内容在 Prompt 中的位置示例
项目背景技术栈、架构风格、项目结构Prompt 开头"Spring Boot 3.4, Java 21, 使用 Controller-Service-Repository 模式"
规则约束代码风格、命名规范、异常处理项目级规则文件"所有 Service 层抛出 BusinessException 不是 RuntimeException"
代码示例同类功能的实现参考用 @ 引用现有文件"参考 UserController 的格式实现 TicketController"
任务说明具体要做的改动Prompt 末尾"在 TicketService 中添加 exportToCsv() 方法"

Context 不足

"写一个 CSV 导出 Controller"——AI 不知道项目结构、不知道已有的工具类、不知道异常处理规范。生成的代码可能需要重构 3 次。

Context 完整

"在 CorpAssist Spring Boot 项目中，参考 UserController 的模式，在 TicketController 中添加 POST /api/tickets/export 端点，使用已有 CsvUtil 类处理 CSV 格式，异常用 BusinessException 抛出"——AI 一次性生成可用代码。

.cursorrules/claude.md 最佳实践

.cursorrules 和 claude.md 是项目级的 AI 规则文件，放在项目根目录，AI 工具会自动读取。前者用于 Cursor，后者用于 Claude Code。

markdown
# .cursorrules / CLAUDE.md

## 项目概述
- 项目: CorpAssist - 企业办公助手
- 技术栈: Spring Boot 3.4, Java 21, Maven, Milvus, Redis
- 架构: Controller → Service → Repository + AI Agent

## 代码规范
- Java 21 使用 record 定义不可变数据类
- Controller 统一加上 @RestController @RequestMapping("/api/...")
- Service 接口 + Impl 分离
- 异常用 BusinessException(code, message)，全局 @ControllerAdvice 处理
- 日志用 Slf4j + logback，不要用 System.out
- 单元测试用 JUnit 5 + Mockito

## API 规范
- 请求体用 @Valid 校验
- 分页统一 PageRequest/PageResponse
- 响应统一 ApiResult 包装

## 数据库
- JPA + MySQL (业务数据)
- Redis (缓存/ChatMemory)
- Milvus (向量检索)

## 常见模式
- CSV 导出: CsvUtil + StreamingResponseBody
- 文件上传: MultipartFile + MinIO
- 异步处理: @Async + CompletableFuture

面试可能问

Context 工程中"代码示例"为什么要用 @ 引用现有文件而不是粘贴代码？

@ 引用是动态链接——AI 工具读取的是该文件的最新版本。如果文件后面被修改了，@ 引用自动获取新内容。粘贴代码是静态的——文件改了粘贴的内容还是旧的。而且 @ 引用让 AI 能够理解文件的结构（类名、方法签名、注解），不仅仅是文本内容。Cursor 的 @ 引用还支持函数级别的引用（@functionName），在大型项目中精准定位参考代码。

四步闭环中生成阶段的代码需要人工验证什么？

三层验证：1) 编译验证——项目能否编译通过（自动）；2) 功能验证——代码是否实现了需求中的功能点（人工 + AI 测试）；3) 规范验证——是否符合项目的架构规范和 API 规范（AI 规则检查）。CorpAssist 中，生成代码后自动运行 mvn compile 和单元测试，通过后人工 review 业务逻辑。

.cursorrules 和 CLAUDE.md 可以共用吗？有什么区别？

不能直接共用，格式和触发机制不同。.cursorrules 是 Cursor 原生格式（自然语言 + 规则），每个 Prompt 都会自动加载（不可选择跳过）。CLAUDE.md 是 Claude Code 的规则文件，在会话开始时加载一次。Cursor 不支持 CLAUDE.md，Claude Code 不支持 .cursorrules。建议：两个文件都维护，内容尽量保持一致。用脚本同步：cp .cursorrules CLAUDE.md 并做格式转换。

// ❌ AI 生成的缺陷代码
public void exportCsv(List tickets, Writer writer) {
    for (Ticket t : tickets) {
        // 漏洞：没有对内容做 CSV 转义
        writer.write(t.getId() + "," + t.getTitle() + "\n");
        // 如果 title = "=SUM(A1:A10)"，Excel 会执行公式！
    }
}

// ✅ 修复后的代码
public void exportCsv(List tickets, Writer writer) {
    for (Ticket t : tickets) {
        writer.write(escapeCsv(t.getId()) + "," + escapeCsv(t.getTitle()) + "\n");
    }
}

private String escapeCsv(String value) {
    if (value == null) return "";
    // CSV 注入防护：= + - @ 开头的字段加单引号前缀
    if (value.matches("^[=+\\-@].*")) {
        return "'" + value;
    }
    // 包含逗号或引号的字段用双引号包裹
    if (value.contains(",") || value.contains("\"")) {
        return "\"" + value.replace("\"", "\"\"") + "\"";
    }
    return value;
}

面试可能问

AI 生成代码的"幻觉 API"（幻影 API）问题怎么系统性地检测和防止？

系统性方案：1) 编译检查（最直接——不存在的类编译不过）；2) 依赖检查工具（如 `mvn dependency:analyze` 检查未使用的依赖和未声明的依赖）；3) IDE 插件（Cursor 有内置的编译错误检测，`Cmd+K` 时自动检查）；4) AI Review Prompt 中明确要求"检查所有 import 的类是否在当前项目的依赖中"。防止：在 .cursorrules 中列出项目依赖的核心库和版本，减少 AI 使用未知类库的概率。

团队 AI 代码比例从 20% 提升到 60% 后，Bug 率反而下降了，为什么？

可能原因：1) 原来人工编写的 CRUD 代码容易出错（拼写错误、Copy-Paste 遗漏），AI 生成避免了这类低级错误；2) SonarQube + AI Review 双保险拦截了大部分缺陷；3) AI 生成的代码更一致（遵循相同的模式和规范）。但需要注意：如果 AI 代码比例继续提升到 80% 以上，核心业务逻辑也开始由 AI 生成，Bug 率可能反弹（复杂逻辑 AI 理解偏差）。建议监控 Bug 率随 AI 比例的变化曲线，找到最佳平衡点。

AI Review 的 Prompt 需要结构化设计。CorpAssist 的 Review Prompt 包含 6 个固定检查项：1) 安全——检查 OWASP Top 10（尤其是注入和认证）；2) 空指针——所有 Optional、可能为 null 的返回值是否判空；3) 资源——文件/数据库连接/HTTP 响应是否关闭；4) 并发——实例变量是否线程安全；5) API 契约——是否与 OpenAPI 定义一致；6) 性能——是否有 N+1 查询、大对象循环等。每个检查项有具体的检查标准和示例。

场景：AI 生成的代码在 SonarQube 中全是绿灯，但上线后出现 NPE，怎么改进检查流程？

# CorpAssist 项目 Rules（精简版）

## 命名规范
- 类名: PascalCase, 名词 (TicketService, UserController)
- 方法名: camelCase, 动词 (findById, createTicket)
- 变量名: camelCase, 含义清晰 (ticketList, userEmail)
- 常量: UPPER_SNAKE_CASE (MAX_RETRY_COUNT)
- 数据库表: snake_case, 复数 (tickets, users)
- API 路径: kebab-case, 复数 (/api/tickets, /api/users)

## 包结构
com.corpassist.agent
  ├── controller     # REST 控制器
  ├── service        # 业务逻辑
  ├── repository     # 数据访问
  ├── model          # 数据模型
  │   ├── entity     # JPA 实体
  │   ├── dto        # 传输对象
  │   └── vo         # 视图对象
  ├── config         # 配置类
  └── common         # 公共工具

## 架构规范
- Controller 只做参数校验和路由，不写业务逻辑
- Service 层必须实现接口 + Impl
- Repository 层用 Spring Data JPA
- 跨服务调用通过 FeignClient（不是 RestTemplate）
- 工具类方法必须是 static

## 异常处理
- 业务异常用 BusinessException(code, message)
- 枚举类定义异常码：com.corpassist.common.exception.ErrorCode
- Controller 层不 try-catch，由 GlobalExceptionHandler 统一处理

## API 规范
- 请求体必须 @Valid
- 分页必须用 PageRequest/PageResponse
- 响应用 ApiResult 统一包装
- 新端点必须在 OpenAPI 文档中记录

没有 Rules

AI 生成代码风格随机——有的 Controller 用 @GetMapping，有的用 @RequestMapping(method=GET)。有的异常用 BusinessException，有的直接返回 500。Review 时一半时间在修风格问题。

有 Rules

AI 生成的代码自动遵循规范——所有 Controller 统一风格，异常处理一致，包结构正确。AI 代码的一次性通过率从 45% 提升到 82%，Review 只需关注业务逻辑。

面试可能问

项目级 Rules 文件的内容应该怎么维护？谁来更新？

Architecture Owner 维护，团队评审更新。Rules 文件放在项目根目录，与代码同仓库，纳入版本管理。更新流程：1) 团队发现 AI 生成代码的风格问题或架构偏离；2) 讨论后在 Rules 中增加对应规则；3) 提交 MR 更新 Rules 文件；4) 通知团队 Restart Cursor 使新 Rules 生效。Rules 的版本更新也记录了团队编码规范的演进历史。

OpenAPI Schema 和 AI 生成的 Controller 不一致时，怎么自动发现？

工具层面：1) 使用 `springdoc-openapi` 在运行时生成 OpenAPI JSON；2) 在 CI 中对比生成的 JSON 与定义在 api-contracts/ 中的 Schema JSON（用 OpenAPI Diff 工具）；3) 如果发现不兼容的差异（字段缺失、类型不一致），CI 失败。也可以在 AI Review Prompt 中要求"对比生成的 Controller 与 api-contracts/*.yaml，检查路径是否一致、请求/响应格式是否匹配"。

Rules 文件越来越长（超过 200 行），AI 会不会忽略后面的规则？

Cursor 的 Context Window 可以容纳较长的 Rules 文件。但如果文件太长，可以考虑分层：1) 核心规则（50 行内）——命名/架构/异常等必须遵守的规则，放在 .cursorrules 顶部；2) 参考规范（100+ 行）——完整的编码规范，放在 .cursorrules 后半部分或引用其他文件；3) 按模块拆分——不同模块（controller/service/repository）的规则分别写在对应的模块目录中（Cursor 支持子目录的 .cursorrules）。

判断练习

场景：AI 生成的 FlightBookingController 放在了 package booking.service 而不是 booking.controller，怎么防止？

在 .cursorrules 中明确包结构规则："Controller 类必须放在 *.controller 包中，Service 必须在 *.service 包中"。并在 CI 的 AI Review 步骤中检查包分配——用脚本扫描生成的 Java 文件，检测 Controller/Service 是否放在了正确的包路径下。如果发现 Controller 放在 service 包，CI 失败。

场景：Rules 规定所有 API 返回 ApiResult，但 AI 生成的新端点直接返回了 TicketDTO（不带 ApiResult 包装）。怎么自动修复？

两种方案：1) 用 OpenAPI Schema 约束——在 CI 中对比 SpringDoc 生成的 JSON 与定义的 API 契约，发现响应格式不一致时构建失败；2) 用 Aspect 自动包装——在 Controller 层加一个 @ControllerAdvice 自动将返回值包装为 ApiResult。这样即使 AI 忘记包装，运行时也会自动包装。推荐方案 1（契约约束）+ 方案 2（安全网）组合。

约 25 分钟 · 阶段：AI 自动化实践 · 能力：pytest 用例自动生成、边界条件覆盖、Mock 与 Fixture 生成、覆盖率报告解读

为什么学这章

CorpAssist 团队发现，即使代码覆盖率达标，测试质量仍然不足——测试覆盖了 happy path 但遗漏了边界条件（空值、超长字符串、并发等）。AI 可以分析代码路径并生成覆盖边界条件的测试用例，补充人工测试的盲区。

学完后能做什么

用 AI 生成 pytest 单元测试（含边界条件）；自动生成 Mock 和 Fixture 简化测试准备；解读覆盖率报告发现未覆盖路径；实现"AI 生成→人工补充→AI 完善"的测试迭代流程。

本章要点速记

1) AI 生成测试的优势：快速覆盖分支路径，适合参数化测试；2) AI 容易遗漏边界条件（空值/0/负数/超大值），需要特别强调；3) Mock 和 Fixture 生成最省时——减少 70% 的样板代码；4) 覆盖率报告关注"未覆盖分支"而不是"总覆盖率"。

pytest 用例自动生成

AI 生成测试的核心思路是：给 AI 看被测代码，让它分析所有执行路径，生成对应测试用例。Claude Code 的 `Test` 命令可以直接生成测试文件。

# 被测代码：ticket_service.py
from datetime import datetime
from typing import Optional

class TicketService:
    def create_ticket(self, title: str, description: str,
                      priority: str, creator_id: str) -> dict:
        """创建工单"""
        if not title or len(title.strip()) == 0:
            raise ValueError("标题不能为空")
        if len(title) > 200:
            raise ValueError("标题不能超过200字符")
        if priority not in ["low", "medium", "high", "urgent"]:
            raise ValueError("优先级无效")
        if not creator_id:
            raise ValueError("创建人不能为空")

        ticket = {
            "id": f"TKT-{datetime.now().strftime('%Y%m%d%H%M%S')}",
            "title": title.strip(),
            "description": description,
            "priority": priority,
            "creator_id": creator_id,
            "status": "pending",
            "created_at": datetime.now().isoformat()
        }
        return ticket

AI 默认生成

只覆盖 happy path（正常输入+正常输出），测试代码覆盖率虽然高但行覆盖率 100% 不一定代表分支覆盖率 100%。漏掉了空标题和超长标题的验证。

指定边界的 Prompt

"生成 TicketService.create_ticket 的测试，必须覆盖：空值、超长（201字符）、无效优先级、空创建人 ID。使用 parametrize 减少代码重复。"——覆盖了所有边界。

面试可能问

AI 生成测试的最大局限性是什么？怎么弥补？

最大局限：AI 不理解业务逻辑的"预期行为"——它只能根据代码推导输入输出，但无法判断这个行为是否正确。例如：一个含 Bug 的计算函数，AI 会生成验证 Bug 输出的测试。弥补方法：1) AI 生成测试骨架，人工确认断言条件（expected value）的准确性；2) 用 Property-Based Testing（jqwik/Hypothesis）补充——AI 生成生成策略而非具体值；3) 先写测试再写代码（TDD）——人工定义预期行为，AI 生成实现代码。

AI 生成的 Mock 和真实系统行为不一致怎么办？

这是 Mock 的固有问题——Mock 只能模拟"你认为系统会怎么响应"，但真实行为可能不同。缓解：1) 使用录播（VCR）库（如 vcrpy）录制真实系统的响应，重放给测试；2) AI 生成的 Mock 先经过人工 review，确认 side_effect 和 return_value 符合真实 API 行为；3) 集成测试（Integration Test）用真实的系统调用，不 Mock。AI 生成 Mock 适合用单元测试，集成测试应该人工编写。

覆盖率 100% 但线上仍有 Bug，怎么系统性地提升测试有效性？

覆盖率 100% 只代表每一行代码都被执行过，不代表所有执行路径和条件都被测试到。系统性措施：1) 用分支覆盖率（Branch Coverage）替代行覆盖率——确保每个 if/else 分支都测试；2) 用变异测试（Pitest）——修改代码逻辑，测试能不能发现；3) 用基于属性的测试——随机生成大量输入，测试不变性；4) AI 分析代码的所有路径（圈复杂度），确保每条路径都有测试。

判断练习

场景：AI 生成的测试覆盖了 create_ticket 的所有分支路径，但运行时有 2 个测试失败，怎么排查？

排查：1) 测试给的输入参数是否在业务允许范围内（如 priority="urgent" 是否真的被支持？检查代码逻辑）；2) 浮点精度问题——如果涉及计算，检查 fixture 中的参数是否精确匹配；3) 时间相关测试——AI 生成的测试可能用了固定时间但代码用了真实时间（now()），导致断言失败。修复策略：让 AI 重新检查代码的精确行为，调整测试的参数和断言条件。

# AI 生成的 Google 风格 docstring

def calculate_priority_score(
    ticket: dict,
    escalation_rules: list[dict] | None = None
) -> float:
    """计算工单的优先级分数，用于自动排序。

    基于工单的紧急程度、影响范围和升级规则计算综合优先级分数。
    分数范围 0.0（最低）到 10.0（最高）。

    Args:
        ticket: 工单数据，必须包含以下字段：
            - impact: str, 影响范围 ("single_user"|"department"|"company")
            - urgency: str, 紧急程度 ("low"|"medium"|"high"|"critical")
            - created_at: str, ISO 格式时间戳
        escalation_rules: 可选的升级规则列表，覆盖默认规则。
            每条规则包含 {"keyword": str, "score_boost": float}

    Returns:
        0.0 到 10.0 的浮点数分数

    Raises:
        ValueError: ticket 缺少必需的字段
        TypeError: impact 或 urgency 不在允许范围内

    Example:
        >>> ticket = {"impact": "department", "urgency": "high",
        ...           "created_at": "2026-05-30T10:00:00"}
        >>> calculate_priority_score(ticket)
        7.5
    """
    if not ticket.get("impact") or not ticket.get("urgency"):
        raise ValueError("ticket 必须包含 impact 和 urgency 字段")

    impact_scores = {"single_user": 2, "department": 5, "company": 9}
    urgency_scores = {"low": 1, "medium": 3, "high": 6, "critical": 9}

    base_score = impact_scores[ticket["impact"]] + urgency_scores[ticket["urgency"]]

    if escalation_rules:
        for rule in escalation_rules:
            if rule["keyword"] in str(ticket):
                base_score += rule["score_boost"]

    return min(base_score / 1.8, 10.0)

面试可能问

AI Review 和人工 Review 的分工怎么设计？AI 发现了所有问题但人工还需要做什么？

分工：AI Review 负责"可自动化的检查"——安全、空指针、异常处理、资源关闭、API 契约、代码风格、重复代码。人工 Review 负责"需要理解业务逻辑的检查"——功能正确性、设计合理性、是否符合产品需求、边缘 case 处理。AI Review 承担了约 60% 的检查工作量，人工从 45 分钟减到 15 分钟。人工 Review 的核心时间应该花在"这个功能是否是用户想要的"上。

AI 性能分析发现了一个热点方法，但给出的优化建议（如改用 Stream API 并行处理）反而慢了，怎么处理？

AI 的优化建议不一定适合所有场景——并行流在小数据集上有线程调度开销。处理方法：1) 质疑策略——AI 给出的优化建议需要经过基准测试验证（JMH）；2) 提供更完整的上下文——告知 AI 这个方法的数据规模（"每秒调用 1000 次，每次处理 10 条数据"），小数据量下并行不一定更快；3) 让 AI 生成 A/B 测试代码——"生成两种实现的 JMH 基准测试代码，用 @Benchmark 对比性能"。

团队 PR 中有大量 AI 生成的代码，Review 时 AI Review 再检查 AI 生成的代码，会不会有盲区？

可能的盲区：AI Review 检查 AI 生成的代码时，可能继续沿用生成时的"错误逻辑假设"。例如：生成代码时 AI 假设了一个不存在的 API（幻觉），Review 时 AI 可能认为这个幻觉 API 是合理的。克服方法：1) AI Review 的 Prompt 中强调"检查代码中是否使用了不存在的 API/类"；2) 编译检查先行——编译不过的代码 AI Review 不会进一步检查；3) 分层 Review——SonarQube 检查语法 → AI Review 检查逻辑 → 人工确认业务正确性，每层使用不同的"智能体"减少盲区。

判断练习

场景：AI Review 的 PR 评论太长（2000 字），工程师不愿意看，怎么优化？

优化 AI Review 的输出格式：1) 按严重等级排序——Critical 优先显示，Low 级别的折叠（details 标签）；2) 每条问题控制在 3 行以内（文件:行号 + 一句话问题 + 修复建议链接）；3) 结尾给出总结建议——"共发现 5 个问题（Critical 1, High 2, Medium 2）, 预计修复时间 20 分钟"；4) 允许工程师回复 /ai-skip "这条是误报" 来标记误报，减少噪音。

场景：CI 中的 AI Review 每次运行要 3 分钟，团队觉得太慢想关掉，怎么优化性能？

优化：1) 增量审查——只审查 PR 中变更的文件，不是全量项目（用 git diff 限制输入范围）；2) 文件过滤——忽略测试文件、DTO 文件、配置文件（这些 AI Review 价值低）；3) 缓存——如果同一段代码在多 PR 中重复出现（如来自模板），缓存审查结果；4) 异步——AI Review 不阻塞 PR 合并流程（非 required check），结果作为评论参考。优化后每次 AI Review 约 30-60 秒。

ROI 最高的是"测试编写"和"文档生成"。原因是这两项工作重复性最高、创造性最低，AI 可以快速完成 80% 的工作量。CorpAssist 经验：测试编写时间从 1.5 小时降到 0.5 小时（-67%），文档从 1 小时降到 0.3 小时（-70%）。这两项节省的时间占比 40%，但成本几乎为零（已有的 AI 工具额度）。Bug 定位和性能分析虽然时间节省也多，但需要高级模型（GPT-4.1/Claude），成本较高。

团队引入 AI 辅助开发后，新成员的上手时间是变短还是变长？

初期（第 1-2 周）可能变长——新成员需要学习：1) AI 工具的用法和 Prompt 工程；2) 如何判断 AI 生成代码的质量；3) 何时依赖 AI 何时自己写。1 个月后上手时间显著缩短——原来需要 2 个月才能独立开发，现在约 1 个月。原因：AI 帮助新成员理解代码库（"解释这个模块的架构"）、生成初稿代码供参考、快速写单元测试。CorpAssist 经验：新成员第 3 周开始独立产出，第 6 周达到老成员效率水平（传统方式分别为第 6 周和第 12 周）。

AI 辅助开发最大的风险是什么？怎么管理？

最大的风险是"代码质量的幻觉"——团队过度信任 AI 生成的代码，Review 放松导致质量下降。具体风险：1) AI 生成的代码可能有隐蔽的 Bug（边界条件遗漏、安全漏洞），在特定输入下才触发；2) 团队依赖 AI 后，成员的代码理解和调试能力下降；3) 项目架构一致性被打破——AI 不了解全局架构，生成的代码可能在局部正确但整体不一致。管理措施：1) 坚持 SonarQube + AI Review + 人工 Review 三层检查；2) 核心模块（安全/支付/权限）限制 AI 代码比例 < 10%；3) 每月 Code Review 一次"AI 生成代码的质量"；4) 定期轮换模块负责人，避免单个成员只使用 AI 而失去对代码的深度理解。

判断练习

场景：AI 辅助开发的代码在一次线上故障中被发现有问题（边界条件遗漏），团队出现了"AI 代码质量下降"的质疑，怎么应对？

应对策略：1) 承认问题但不否定 AI 辅助的方向——"AI 确实遗漏了边界条件，但这次故障的根本原因是 Review 流程没有检查到这个边界"；2) 改进流程——在 AI Review 的 Prompt 中增加该类边界条件的专项检查；3) 数据说话——展示月度数据：引入 AI 后整体 Bug 率下降 35%，单个事件不否定总体趋势；4) 建立"AI 代码 Bug 根因分析"机制——每次 AI 相关 Bug 都分析根因（是 AI 生成错误还是人工 Review 遗漏），针对性地改进 Prompt 或 Review 流程。

场景：团队 Leader 质疑 AI 辅助的工具成本（Cursor Pro $20/月/人 + Claude API 按用量），怎么证明 ROI？

ROI 计算：15 人团队，月工具成本 = Cursor 15×$20 = $300 + Claude API 约 $200 = $500/月（约 ¥3600）。月节省时间 = 180 人时，按 ¥80/人时（工程师成本）计算 = ¥14400/月。ROI = 14400/3600 = 4:1，即每投入 1 元工具成本，节省 4 元人力成本。而且节省的时间是释放到更有创造性的工作（架构设计、技术决策），不是简单的减员。对于 Management，重点强调"同样的团队规模能做更多的事"而不是"同样的事用更少的人"。
