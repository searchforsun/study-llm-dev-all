为什么要学本章

CorpAssist Python 服务与 Java BFF 通过 OpenAPI 契约协作。接口迭代若不同步文档，网关代码生成与联调会频繁返工。AI 可从 Pydantic 模型反推 Schema、补全 docstring，并用脚本做 schema-diff 门禁。

  • 批量生成符合 Google 风格的模块/路由 docstring

  • 用 AI 生成 README 与 API 文档章节

本章先记住 3 件事

  • 单一真源是 Pydantic 模型与路由装饰器，文档由代码生成而非手写 YAML
  • AI 补 docstring 时必须引用真实字段约束（min_length、examples）
  • OpenAPI diff 失败应阻断合并，与测试失败同级

docstring 自动化

FastAPI 路由的说明来自函数 docstring 与 response_model。对 CorpAssist rag/query 端点，AI 可根据 Schema 生成包含参数、返回值、错误码的 docstring，供 Swagger UI 与 Redoc 展示。

场景：新增 stream=true 查询参数后，需在同一个 PR 更新 docstring 与示例 JSON，否则 Java 同事在 Swagger 中看不到新字段。
python
from fastapi import APIRouter, Depends
from app.rag.schemas import RagQueryRequest, RagQueryResponse
from app.rag.service import RagPipeline, get_pipeline

router = APIRouter(prefix="/v1/rag", tags=["rag"])


@router.post("/query", response_model=RagQueryResponse)
async def rag_query(
    body: RagQueryRequest,
    pipeline: RagPipeline = Depends(get_pipeline),
) -> RagQueryResponse:
    """企业知识库 RAG 查询（CorpAssist S1）。

    Args:
        body: 查询体。query 为自然语言问题；top_k 默认 3，最大 20。

    Returns:
        answer: 模型综合检索片段后的回答。
        citations: 引用列表，含 doc_id 与 snippet，供前端溯源。

    Raises:
        HTTPException 401: 缺少或无效 X-Api-Key。
        HTTPException 422: query 为空或超长。
    """
    return await pipeline.arun(query=body.query, top_k=body.top_k)

OpenAPI Schema 生成

FastAPI 自动生成 OpenAPI 3.1。CorpAssist 在 CI 导出 openapi.json，与仓库中 contracts/corpassist-rag-v1.json 比对；AI 可解释 diff hunk 并建议改模型还是改契约。

契约同步流
flowchart LR
    PY[FastAPI + Pydantic] -->|app.openapi| OAS[openapi.json]
    OAS --> DIFF[schema-diff CI]
    JAVA[Java BFF codegen] --> DIFF
    DIFF -->|fail| PR[阻断 PR]
    DIFF -->|pass| MERGE[合并]
python
# scripts/export_openapi.py
import json
from app.main import app

if __name__ == "__main__":
    schema = app.openapi()
    with open("contracts/corpassist-rag-v1.json", "w", encoding="utf-8") as f:
        json.dump(schema, f, ensure_ascii=False, indent=2)
python
from pydantic import BaseModel, Field


class RagQueryRequest(BaseModel):
    query: str = Field(..., min_length=1, max_length=8000, description="用户问题")
    top_k: int = Field(3, ge=1, le=20, description="检索返回条数")

    model_config = {
        "json_schema_extra": {
            "examples": [{"query": "年假几天", "top_k": 3}]
        }
    }

Prompt 示例：「根据下列 diff，判断应修改 Pydantic 还是升级契约 major 版本，并给出 semver 建议。」Breaking change（删除字段、改类型）必须走 ADR 与 Java 团队同步。

diff 类型处理
新增可选字段minor 版本，Java 可忽略
required 变 optional通常兼容，仍需回归
改字段类型major，禁止静默合并

README 与 API 文档

仓库根 README.md 面向工程师：本地启动、环境变量、uv sync。对外 API 文档可由 AI 从 OpenAPI 生成 Markdown 章节（认证、限流、示例 curl），但示例请求体必须来自 json_schema_extra.examples。

markdown
## CorpAssist RAG API（Python）

### 认证
所有请求需头 `X-Api-Key`，与网关一致。

### 查询
```bash
curl -X POST "$BASE/v1/rag/query" \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: $KEY" \
  -d '{"query":"年假几天","top_k":3}'
```

### 错误码
| HTTP | error_code | 说明 |
|------|------------|------|
| 401 | AUTH_INVALID | Key 无效 |
| 422 | VALIDATION_ERROR | 体校验失败 |

让 AI 「仅更新 README 的 API 小节，保留安装章节不动」，可避免覆盖人工写的部署说明。使用 .cursor/rules 声明 README 章节标题不可删改。

文档一致性校验

在 GitHub Actions / GitLab CI 中串联：导出 OpenAPI → openapi-diff → ruff docstring → AI 可选「解释失败原因」步骤（不阻断，仅评论 PR）。

yaml
# .github/workflows/docs.yml 片段
- run: uv run python scripts/export_openapi.py
- run: npx openapi-diff contracts/corpassist-rag-v1.json contracts/corpassist-rag-v1.baseline.json --fail-on-incompatible
- run: uv run ruff check app/ --select D
- run: uv run python scripts/check_readme_examples.py  # 校验 README curl 与 examples 一致
python
# scripts/check_readme_examples.py — 简化示意
import json, re, pathlib

examples = json.loads(pathlib.Path("contracts/corpassist-rag-v1.json").read_text())
body = examples["paths"]["/v1/rag/query"]["post"]["requestBody"]
sample = body["content"]["application/json"]["schema"]["examples"][0]
readme = pathlib.Path("README.md").read_text()
assert sample["query"] in readme, "README 缺少与 OpenAPI 一致的 query 示例"
OpenAPI diff 通过但 Java 仍报错，可能原因？

Java 使用旧 baseline；枚举序列化命名策略不同（camelCase vs snake_case）；日期格式不一致。检查 model_config populate_by_name 与网关全局 ObjectMapper。

将 diff 输出 @ 给 AI 时，附上「兼容策略表」与上一版契约链接，可减少错误建议（如误删 required 字段）。CorpAssist 文档 Owner 应在 CODEOWNERS 中声明，AI 生成 docstring 仍须人审。

仅改 examples 是否需要 major 版本？

通常否，但若 examples 被 Java 测试快照引用，仍可能触发 CI 失败。跑 openapi-diff 与 Java 契约测试双确认。

双栈契约协作要点

CorpAssist Java BFF 通过 Feign/RestTemplate 调用 Python RAG 服务。文档变更流程：Python PR 更新 openapi.json → CI diff → 通知 Java 组刷新 codegen → 联合回归。AI 可生成「变更说明」Comment 模板，但 semver 决策人必须是维护者。

json
{
  "paths": {
    "/v1/rag/query": {
      "post": {
        "operationId": "ragQuery",
        "summary": "企业知识库 RAG 查询",
        "requestBody": { "required": true },
        "responses": {
          "200": { "description": "成功" },
          "401": { "description": "鉴权失败" },
          "422": { "description": "参数校验失败" }
        }
      }
    }
  }
}

使用 FastAPI openapi_tags 将 rag、health、agent 分组，AI 生成文档时按 tag 输出章节，避免单页过长。对废弃端点使用 deprecated=True 并在 description 写明替代路径与移除日期。

能否让 AI 直接改 Java DTO？

可以辅助，但真源仍是 openapi.json。应先合并 Python 契约，再触发 Java 生成；否则两边各自改字段名会再次漂移。

文档 FAQ

Swagger 上 examples 不显示？

检查是否写在 model_config json_schema_extra；FastAPI 版本是否支持；是否被自定义 openapi() 覆盖。重新 export 并在 /docs 强制刷新。

AI 生成的 docstring 与参数名不一致？

要求 Prompt 附带函数签名 AST 或粘贴完整 def 行；生成后跑 ruff D417（参数文档缺失）修复。

openapi-diff 报 incompatible 但确认是安全变更？

走 ADR 更新 baseline 文件，并在 PR 描述写 Java 回归结果；禁止在未通知 BFF 的情况下合并。

Prompt 轮次输入期望输出
1router + schemas 无 docstringGoogle docstring + Field description
2export 后的 openapi.json解释 diff 是否 breaking
3README API 小节与 examples 一致的 curl
Redoc 与 Swagger 描述不一致？
两者同源 openapi.json；若不一致说明 export 步骤未跑或浏览器缓存旧版。CI 应 artifact 上传 openapi.json 供 Java 组下载。
AI 把 Field description 写成英文而团队要求中文？
在 Rules 中声明「面向国内实施文档用中文 description」；生成后 ruff 无法检查语言，需人工 spot check 或脚本抽样。
能否用 AI 从 Java OpenAPI 反向生成 Pydantic？
可以辅助，但 CorpAssist 约定 Python 为真源。Java 变更应通过 diff 驱动 Python 更新，避免双向漂移。

本章结论

  • Pydantic + FastAPI 是 OpenAPI 真源；docstring 服务人类阅读与 Swagger 描述
  • AI 适合批量补全文档与解释 diff，不适合擅自改 breaking 契约
  • README 示例必须与 json_schema_extra 同步，用脚本门禁
  • ruff pydocstyle 与 openapi-diff 应并列在 PR 检查中

面试可能问

Python 与 Java 双栈如何共享 API 契约？

Python 导出 openapi.json 入仓；Java 用 openapi-generator 生成 DTO；CI 双向 diff。变更走 semver 与联合 Review。

AI 生成的 README 能否直接对外发布？

需人工核对密钥占位、环境 URL、合规声明；示例必须经过 check_readme_examples 脚本。

docstring 与 OpenAPI description 重复维护吗？

FastAPI 可同时展示两者；团队可约定 docstring 给工程师、Field description 给 Swagger。AI 生成时一次输出两套，避免人工复制粘贴。

判断练习

  1. 仅改 docstring 未改 Schema，是否需要跑 openapi-diff？

    参考答案

    通常 OpenAPI 描述字段会变，diff 可能有 cosmetic 变更；若 CI 仅检查 incompatible，描述变更可通过。团队可配置是否把 description 变更视为需 Review。

  2. AI 建议在契约中删除 deprecated 字段，是否采纳？

    参考答案

    需查 Java 与移动端是否仍发送该字段；删除为 major，走废弃周期与 telemetry 确认零流量。
