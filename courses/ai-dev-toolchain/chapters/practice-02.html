docstring 自动化

AI 生成 docstring给代码添加注释和文档。Cursor 和 Claude Code 都内置了生成文档的功能。好的 docstring 包含：功能描述、参数说明、返回值、异常、示例。

python
# AI 生成的 Google 风格 docstring

def calculate_priority_score(
    ticket: dict,
    escalation_rules: list[dict] | None = None
) -> float:
    """计算工单的优先级分数，用于自动排序。

    基于工单的紧急程度、影响范围和升级规则计算综合优先级分数。
    分数范围 0.0（最低）到 10.0（最高）。

    Args:
        ticket: 工单数据，必须包含以下字段：
            - impact: str, 影响范围 ("single_user"|"department"|"company")
            - urgency: str, 紧急程度 ("low"|"medium"|"high"|"critical")
            - created_at: str, ISO 格式时间戳
        escalation_rules: 可选的升级规则列表，覆盖默认规则。
            每条规则包含 {"keyword": str, "score_boost": float}

    Returns:
        0.0 到 10.0 的浮点数分数

    Raises:
        ValueError: ticket 缺少必需的字段
        TypeError: impact 或 urgency 不在允许范围内

    Example:
        >>> ticket = {"impact": "department", "urgency": "high",
        ...           "created_at": "2026-05-30T10:00:00"}
        >>> calculate_priority_score(ticket)
        7.5
    """
    if not ticket.get("impact") or not ticket.get("urgency"):
        raise ValueError("ticket 必须包含 impact 和 urgency 字段")

    impact_scores = {"single_user": 2, "department": 5, "company": 9}
    urgency_scores = {"low": 1, "medium": 3, "high": 6, "critical": 9}

    base_score = impact_scores[ticket["impact"]] + urgency_scores[ticket["urgency"]]

    if escalation_rules:
        for rule in escalation_rules:
            if rule["keyword"] in str(ticket):
                base_score += rule["score_boost"]

    return min(base_score / 1.8, 10.0)

OpenAPI Schema 生成

OpenAPI Schema 从源码注解自动生成（springdoc），AI 补充接口描述和示例。反过来，AI 也可以从 OpenAPI Schema 生成前端客户端代码。

java
// Springdoc 自动生成 OpenAPI + AI 补充描述
@RestController
@RequestMapping("/api/tickets")
@Tag(name = "工单管理", description = "工单的 CRUD 和导出功能")
public class TicketController {

    @Operation(summary = "查询工单列表", description = "支持按状态过滤和分页")
    @ApiResponses({
        @ApiResponse(responseCode = "200", description = "分页结果"),
        @ApiResponse(responseCode = "400", description = "参数错误"),
    })
    @GetMapping
    public ApiResult> listTickets(
            @Parameter(description = "页码，从0开始") @RequestParam int page,
            @Parameter(description = "每页大小") @RequestParam int size,
            @Parameter(description = "过滤状态") @RequestParam(required = false) String status) {
        return ApiResult.success(ticketService.list(page, size, status));
    }
}

面试可能问

AI 生成的 docstring 怎么保证与代码实际行为一致（docstring 描述的和代码做的不一样）？

AI 生成时是基于代码文本理解行为，如果代码有 Bug，生成的 docstring 也会"描述 Bug 行为"。多层保障：1) 人工 review docstring——看看描述和预期是否一致；2) 在审查阶段加一项"检查 docstring 是否准确描述代码行为"；3) 编译/检查工具——Javadoc 的 @param 与实际参数不匹配时 Checkstyle 可以检测；4) 测试驱动——先写测试（明确预期行为），再生成代码和 docstring，保证 docstring 与测试一致。

README 被 AI 更新后，团队觉得内容太多、不够简洁，怎么平衡完整性和可读性？

分层 README：1) Top 部分（<1000 字）——项目简介、快速开始、核心技术栈，适合新成员快速了解；2) 架构部分——AI 生成的 Mermaid 架构图和模块说明；3) 附录部分——API 端点列表、配置项说明、常见问题。AI 管理第 2 和第 3 部分（经常需要同步代码），第 1 部分人工维护（更关注可读性）。AI 更新时只更新第 2/3 部分，不碰第 1 部分。

OpenAPI Schema 自动生成了，但生成的描述太简略（只有字段名没有说明），怎么让 AI 补充完整的描述？

方案：1) 在代码的 @Schema 注解中添加 description——@Schema(description = "工单标题，1-200字符")——springdoc 会自动包含；2) 用 AI 批量补充：claude "为所有 DTO 类的字段添加 @Schema description，描述字段的业务含义和约束条件"；3) 运行 springdoc 后，用 AI 补充 API 文档的描述部分：claude "为 OpenAPI JSON 中 summary/description 为空的端点补充描述，参考 Controller 的 @Operation/@ApiResponses"。

判断练习

场景：AI 生成的 README 架构图使用的是图形符号，但项目要求文字描述，怎么让 AI 调整？

在 Prompt 中明确输出格式要求："生成纯文字描述的架构概览，使用缩进表示层级关系，不要 Mermaid 图"。示例格式：核心服务（Ticket Service, Agent Service）→ 数据层（PostgreSQL, Redis, Milvus）→ 外部集成（邮件 MCP, 日历 MCP）。AI 会遵循这个格式输出纯文字性的架构概览。

场景：一致性校验发现 OpenAPI 文档中有 10 个端点，但代码中只有 8 个（2 个端点上个迭代删掉了但文档没更新），怎么自动清理？

方案 1（推荐）：用 springdoc 动态生成 OpenAPI，不维护静态的 openapi.json——代码即文档，删除端点自动从文档消失。方案 2：如果必须维护静态文档（比如发布到外网），用 CI 对比动态生成的和静态的差异，如果不一致 CI 失败。方案 3：用 AI 定期从代码重新生成 OpenAPI 文档并替换旧的。CorpAssist 使用方案 1——运行时动态生成，减少文档维护工作。
