为什么要学本章

CorpAssist 的 FastAPI RAG 接口迭代快，手工补测往往滞后于功能合并。AI 能在理解路由契约与 Pydantic 模型后，一次性产出可运行的 pytest 骨架，再配合人工审查与 ruff 静态检查，把「补测」从数小时压缩到数十分钟。

学完你能

  • 用结构化 Prompt 让 AI 为指定模块生成 pytest 文件与目录布局
  • 设计边界表驱动 parametrize，覆盖空查询、超长上下文、鉴权失败等场景
  • 生成 Mock LLM / 向量库 Fixture，避免 CI 调用真实模型
  • 阅读 coverage 报告，区分「行覆盖」与「分支覆盖」缺口

本章先记住 3 件事

  • 先给 AI「被测函数签名 + 错误码约定 + 禁止真实 LLM」三类约束，再要完整测试文件
  • 边界用例必须来自产品/安全清单，不能只让模型「随便编几个」
  • 生成测试同样要过 ruff + pytest，并纳入与业务代码相同的 Review 门禁

pytest 用例自动生成

在 CorpAssist 仓库中，典型被测对象是 app/rag/router.py 下的 POST /v1/rag/query。向 Cursor / Claude Code 提供：路由源码、schemas/query.py、团队 Rules（禁止直连 LLM），可得到可提交的测试模块。

场景：迭代「引用溯源」字段后，需在 1 个 PR 内补齐回归测试。你把 diff、OpenAPI 片段与现有 conftest.py 一并 @ 给 AI，要求输出 tests/rag/test_query_citations.py。
text
你是 CorpAssist Python 服务的测试工程师。请为以下被测代码生成 pytest 文件。

约束：
1) 使用 httpx.AsyncClient + ASGITransport，不要 TestClient 同步阻塞
2) 所有 LLM 调用必须 mock，禁止环境变量中的真实 API Key
3) 断言包含：HTTP 状态码、response_model 字段、X-Request-Id 响应头
4) 文件顶部注明 # generated-by-ai，人工 review 后删除该行

被测路由：（粘贴 router 源码）
共享 fixture：（粘贴 conftest 片段）
python
# tests/rag/test_query.py — AI 生成骨架经人工微调
import pytest
from httpx import ASGITransport, AsyncClient

from app.main import app


@pytest.mark.asyncio
async def test_rag_query_returns_answer_and_citations(mock_rag_pipeline):
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as client:
        resp = await client.post(
            "/v1/rag/query",
            json={"query": "年假几天", "top_k": 3},
            headers={"X-Api-Key": "test-key"},
        )
    assert resp.status_code == 200
    body = resp.json()
    assert body["answer"]
    assert len(body["citations"]) <= 3
    assert resp.headers.get("X-Request-Id")

低效 Prompt

「帮我写点测试」——缺少契约、无 mock 约束，常生成调用真实 OpenAI 的用例，CI 费用爆炸。

可合并 Prompt

提供路由 + Schema + conftest + 错误码表 + 「禁止真实 LLM」——输出可直接 uv run pytest 的文件。

思考 10 秒：为什么要求 AsyncClient 而不是 Starlette TestClient？

CorpAssist 路由多为 async；在 async 测试函数里用同步 TestClient 容易掩盖事件循环阻塞问题，且与生产 httpx 调用栈不一致。团队 Rules 统一要求 ASGITransport + AsyncClient。

输入给 AI

  • router 源码 + Pydantic 模型
  • 错误码枚举表
  • 现有 conftest 片段

人工必改

  • 断言 citations 字段结构
  • 删除重复 happy path
  • 核对 patch 路径

CI 门禁

  • ruff check tests/
  • pytest -q --cov-fail-under
  • 禁止 secrets 扫描告警

生成后执行 uv run ruff format tests/ 与 uv run ruff check，避免 AI 使用已弃用 API（如旧版 TestClient(app) 同步写法）。若项目启用 pytest-asyncio，在 pyproject.toml 中声明 asyncio_mode = auto，并在 Prompt 中写明，减少 @pytest.mark.asyncio 遗漏。

边界条件覆盖

AI 擅长枚举参数组合，但业务边界须由你提供清单。对 RAG 查询接口，常见边界包括：空字符串、仅空白、超长 query（>8k 字符）、top_k=0、未授权、租户隔离字段缺失等。

输入期望说明
query=""422Pydantic min_length
top_k=101422Schema 上限
无 API Key401网关对齐错误体
mock 检索为空200 + 空 citations降级文案，非 500
python
import pytest

BOUNDARY_CASES = [
    pytest.param("", 422, id="empty-query"),
    pytest.param("   ", 422, id="whitespace-query"),
    pytest.param("x" * 9000, 422, id="overlong-query"),
]


@pytest.mark.parametrize("query,status", BOUNDARY_CASES)
@pytest.mark.asyncio
async def test_rag_query_validation(client, query, status):
    resp = await client.post("/v1/rag/query", json={"query": query, "top_k": 3})
    assert resp.status_code == status

把上表粘贴进 Prompt，要求 AI 「为每一行生成独立 test id，并补充 mock 检索为空的第 4 条」。生成后人工删除重复或与产品冲突的用例。

python
@pytest.mark.asyncio
async def test_rag_query_empty_retrieval_returns_safe_answer(client, mock_rag_pipeline):
    mock_rag_pipeline.return_value = {"answer": "未找到相关政策，请联系 HR。", "citations": []}
    resp = await client.post("/v1/rag/query", json={"query": "火星年假", "top_k": 3})
    assert resp.status_code == 200
    assert "未找到" in resp.json()["answer"]
    assert resp.json()["citations"] == []

对多租户 CorpAssist 部署，边界表还应包含 X-Tenant-Id 缺失、跨租户 doc_id 等条目；把这些行写进 Prompt 的「必须覆盖」清单，避免 AI 只测单租户 happy path。

Prompt 片段作用
「每个边界独立 test id」CI 失败时一眼定位场景
「422 断言 detail[0].loc」与 OpenAPI 错误体一致
「401 断言 error_code 字段」与 Java 网关 BFF 契约对齐

Mock 与 Fixture 生成

Fixture 是 AI 测试生成的核心交付物。CorpAssist 约定在 tests/conftest.py 提供：mock_rag_pipeline（替换向量检索 + LLM 合成）、fake_settings（内存配置）、client（带鉴权头的 AsyncClient）。

Fixture 依赖关系
flowchart TB
    FS[fake_settings] --> APP[app lifespan]
    MRP[mock_rag_pipeline] --> APP
    APP --> CL[client AsyncClient]
    CL --> TC[test_rag_query_*]
python
# tests/conftest.py 片段
import pytest
from unittest.mock import AsyncMock, patch

from httpx import ASGITransport, AsyncClient
from app.main import app


@pytest.fixture
def mock_rag_pipeline():
    with patch("app.rag.service.RagPipeline.arun", new_callable=AsyncMock) as m:
        m.return_value = {
            "answer": "工作满一年可申请 5 天年假。",
            "citations": [{"doc_id": "hr-001", "snippet": "年假 5 天"}],
        }
        yield m


@pytest.fixture
async def client(mock_rag_pipeline):
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as ac:
        ac.headers.update({"X-Api-Key": "test-key"})
        yield ac

向 AI 描述「patch 路径必须与被测 import 路径一致」（常见坑：app.service vs app.rag.service），并要求为每个 mock 写 assert_called_once 副作用断言。

python
@pytest.mark.asyncio
async def test_rag_query_passes_top_k_to_pipeline(client, mock_rag_pipeline):
    await client.post("/v1/rag/query", json={"query": "报销流程", "top_k": 5})
    mock_rag_pipeline.assert_awaited_once()
    _, kwargs = mock_rag_pipeline.call_args
    assert kwargs.get("top_k") == 5

使用 pytest-httpx 时，可 mock 出站 embedding 请求；但 CorpAssist 默认在 service 层 patch RagPipeline，减少与 HTTP 层耦合。若 AI 生成 respx 用例，需确认不会与全局 fixture 冲突。

场景：合并 AI 生成的 12 个用例后，CI 偶发失败。根因是 mock_rag_pipeline 未 autouse，顺序依赖导致状态泄漏。解决：session 级 fixture + 每个测试独立 return_value，或在 teardown 中 reset_mock()。

覆盖率从 72% 提到 88%，但缺陷仍上线，可能原因？

1) 只覆盖 happy path，未 assert 错误体结构；2) mock 过度，未测真实 Pydantic 校验；3) 集成测试缺失。应结合变异测试或契约测试抽查关键断言。

AI 生成的 patch 路径总是报错 ImportError？
常见原因：测试 patch 的是定义处（app.rag.service.RagPipeline），但被测路由 import 的是别名。在 Prompt 中粘贴 router 文件顶部的 import 行，要求「patch 被测模块内可见的名字」。用 pytest --import-mode=importlib 与 CI 对齐。
如何测试 streaming SSE 响应？
httpx stream=True + 逐 chunk 断言 event 类型；mock 异步生成器返回固定 token 序列。Prompt 中附上 SSE 格式样例，避免 AI 用同步 TestClient 读流。
parametrize 与 fixture 组合顺序？
fixture 先注入 client/mock，parametrize 提供输入；不要在 parametrize 里构造 AsyncClient。AI 生成后检查每个用例是否独立，避免共享 mutable mock 状态。
yaml
# .github/workflows/test.yml 片段
- run: uv run ruff check tests/ app/
- run: uv run pytest tests/ -q --cov=app --cov-fail-under=85
  env:
    CORPAssist_DISABLE_LLM: "1"

延伸：与 Java 侧契约测试

Python pytest 如何配合 BFF 契约测试？

Python 导出 openapi.json；Pact 或 Schemathesis 对网关消费方做契约校验。AI 生成测试时可要求「断言响应 JSON 与 openapi components 中 RagQueryResponse 一致」，减少双栈字段漂移。

AI 生成的测试很多，如何删冗余？

按 coverage 增量排序保留；合并断言相同的 parametrize 行；删除只测 FastAPI 框架行为的用例（如 404 路由不存在）。

如何测试流式 RAG 响应？

使用 httpx 流式读取 + mock 异步迭代器；Prompt 中明确 SSE 事件格式；不要用一个巨型 assert 比较全文。

反模式修复
测试里 sleep 等待mock 或 freezegun
依赖上传真实 PDFfixtures 小文件或内存 BytesIO
无 assert 的 smoke test删除或补断言

本章结论

  • AI 生成 pytest 的价值在「契约对齐 + mock 骨架」，不是替代测试设计
  • 边界表由人定义，AI 负责 parametrize 与断言样板
  • Fixture 与 patch 路径必须与生产 import 一致，否则测试「全绿但无意义」
  • 覆盖率用于发现盲区，安全与鉴权模块阈值高于业务平均值

面试可能问

AI 生成的测试要不要进仓库？如何防劣化？

可以进，但需标 generated-by-ai、经人工改断言、过 ruff/pytest、与手写测试同 Review。定期用覆盖率 diff 删除重复用例；禁止合并仅 assert True 的占位测试。

为什么不建议在 CI 里调用真实 LLM 做断言？

成本高、结果非确定、易泄露密钥。应 mock 固定 JSON；模型质量用离线评测流水线（见 实战：CorpAssist 开发效率提升）。

操作步骤

判断练习

  1. AI 生成的测试在本地通过，但 CI 失败，优先查什么？

    参考答案

    环境变量未 mock 导致请求外网；patch 路径在 CI 包结构下不一致；async 插件未安装；uv.lock 与本地依赖不一致。先看 CI 日志是 401/timeout 还是 ImportError。

  2. 覆盖率已 90%，是否可以不 Review AI 测试？

    参考答案

    不可以。高覆盖可能缺少断言业务字段（如 citations 结构）。必须 Review 每个 assert 是否验证契约。
