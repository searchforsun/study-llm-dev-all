为什么学这章

CorpAssist 团队发现，即使代码覆盖率达标，测试质量仍然不足——测试覆盖了 happy path 但遗漏了边界条件（空值、超长字符串、并发等）。AI 可以分析代码路径并生成覆盖边界条件的测试用例，补充人工测试的盲区。

学完后能做什么

用 AI 生成 pytest 单元测试（含边界条件）；自动生成 Mock 和 Fixture 简化测试准备；解读覆盖率报告发现未覆盖路径；实现"AI 生成→人工补充→AI 完善"的测试迭代流程。

本章要点速记

1) AI 生成测试的优势：快速覆盖分支路径，适合参数化测试；2) AI 容易遗漏边界条件（空值/0/负数/超大值），需要特别强调；3) Mock 和 Fixture 生成最省时——减少 70% 的样板代码；4) 覆盖率报告关注"未覆盖分支"而不是"总覆盖率"。

pytest 用例自动生成

AI 生成测试的核心思路是：给 AI 看被测代码，让它分析所有执行路径，生成对应测试用例。Claude Code 的 `Test` 命令可以直接生成测试文件。

python
# 被测代码：ticket_service.py
from datetime import datetime
from typing import Optional

class TicketService:
    def create_ticket(self, title: str, description: str,
                      priority: str, creator_id: str) -> dict:
        """创建工单"""
        if not title or len(title.strip()) == 0:
            raise ValueError("标题不能为空")
        if len(title) > 200:
            raise ValueError("标题不能超过200字符")
        if priority not in ["low", "medium", "high", "urgent"]:
            raise ValueError("优先级无效")
        if not creator_id:
            raise ValueError("创建人不能为空")

        ticket = {
            "id": f"TKT-{datetime.now().strftime('%Y%m%d%H%M%S')}",
            "title": title.strip(),
            "description": description,
            "priority": priority,
            "creator_id": creator_id,
            "status": "pending",
            "created_at": datetime.now().isoformat()
        }
        return ticket

# AI 通过 Claude Code 生成的测试：

$ claude "为 ticket_service.py 的 TicketService.create_ticket
方法生成完整的 pytest 测试，包含所有边界条件。
输出到 tests/test_ticket_service.py"
"""

# AI 生成的测试文件：tests/test_ticket_service.py
import pytest
from datetime import datetime
from ticket_service import TicketService

class TestTicketService:
    @pytest.fixture
    def service(self):
        return TicketService()

    # === Happy Path ===
    def test_create_ticket_success(self, service):
        ticket = service.create_ticket(
            title="系统故障",
            description="数据库连接超时",
            priority="high",
            creator_id="u001"
        )
        assert ticket["id"].startswith("TKT-")
        assert ticket["status"] == "pending"
        assert ticket["priority"] == "high"

    # === 边界条件 ===
    @pytest.mark.parametrize("title", ["", "   ", None])
    def test_create_ticket_empty_title(self, service, title):
        with pytest.raises(ValueError, match="标题不能为空"):
            service.create_ticket(title, "desc", "low", "u001")

    def test_create_ticket_title_too_long(self, service):
        with pytest.raises(ValueError, match="不能超过200字符"):
            service.create_ticket("A" * 201, "desc", "low", "u001")

    @pytest.mark.parametrize("priority", ["", "critical", "unknown", None])
    def test_create_ticket_invalid_priority(self, service, priority):
        with pytest.raises(ValueError, match="优先级无效"):
            service.create_ticket("test", "desc", priority, "u001")

    def test_create_ticket_empty_creator(self, service):
        with pytest.raises(ValueError, match="创建人不能为空"):
            service.create_ticket("test", "desc", "low", None)

    # === 输入清洗 ===
    def test_create_ticket_title_trimmed(self, service):
        ticket = service.create_ticket(
            "  空间前标题  ", "desc", "low", "u001")
        assert ticket["title"] == "空间前标题"

    # === 输出格式 ===
    def test_create_ticket_has_required_fields(self, service):
        ticket = service.create_ticket(
            "test", "desc", "low", "u001")
        assert all(k in ticket for k in
            ["id", "title", "description", "priority",
             "creator_id", "status", "created_at"])

CorpAssist 场景

开发"工单导出"功能后，用 Claude Code 自动生成测试：claude test -f src/main/java/ExportService.java。AI 分析代码发现 6 条执行路径，生成 12 个测试用例（6 个 Happy Path + 6 个边界条件）。覆盖率从 0 直接到 88%，人工补充 2 个边缘 case 后达到 95%。

边界条件覆盖

AI 生成的测试容易遗漏的边界条件列表——需要在 Prompt 中特别强调。

边界类型AI 容易遗漏Prompt 中强调
空/Null字符串空、列表空、对象 Null"测试所有可能为 None 的参数"
长度限制超长字符串、超长列表"测试最大长度和超过最大长度"
数值边界0、负数、最大值、最小值"测试 0、负数、边界值"
特殊字符Unicode、SQL 注入、HTML 标签"测试特殊字符和注入模式"
并发多线程同时调用、竞态条件"用 ThreadPoolExecutor 测试并发"
状态转换非法状态转换"测试所有状态的非法转换"

AI 默认生成

只覆盖 happy path（正常输入+正常输出），测试代码覆盖率虽然高但行覆盖率 100% 不一定代表分支覆盖率 100%。漏掉了空标题和超长标题的验证。

指定边界的 Prompt

"生成 TicketService.create_ticket 的测试，必须覆盖：空值、超长（201字符）、无效优先级、空创建人 ID。使用 parametrize 减少代码重复。"——覆盖了所有边界。

Mock 与 Fixture 生成

编写 Mock 和 Fixture 是测试中最耗时的工作（约 40% 的测试代码）。AI 可以自动生成 Mock 配置和 Fixture 工厂函数。

python
# AI 生成的 Mock 和 Fixture
import pytest
from unittest.mock import Mock, patch, MagicMock
from datetime import datetime

# AI 生成的 Fixture：工单工厂
@pytest.fixture
def ticket_factory():
    """生成测试工单数据的工厂"""
    def _create_ticket(
        id="TKT-001",
        title="测试工单",
        status="pending",
        priority="high",
        creator_id="u001",
        assignee_id=None
    ):
        ticket = {
            "id": id,
            "title": title,
            "status": status,
            "priority": priority,
            "creator_id": creator_id,
            "assignee_id": assignee_id,
            "created_at": datetime.now().isoformat()
        }
        return ticket
    return _create_ticket

# AI 生成的 Mock：邮件服务
@pytest.fixture
def mock_email_service():
    """Mock 邮件服务"""
    mock = MagicMock()
    mock.send_email.return_value = {"status": "sent", "message_id": "MSG-001"}
    mock.send_email.side_effect = [
        {"status": "sent", "message_id": "MSG-001"},
        {"status": "sent", "message_id": "MSG-002"},
        TimeoutError("邮件服务超时"),  # 第三次调用模拟超时
    ]
    return mock

# AI 生成的 Mock：数据库 Repository
@pytest.fixture
def mock_ticket_repo():
    """Mock TicketRepository"""
    mock = MagicMock()
    # findById 模拟
    def mock_find_by_id(ticket_id):
        if ticket_id == "TKT-001":
            return {"id": "TKT-001", "title": "测试工单", "status": "pending"}
        return None
    mock.find_by_id.side_effect = mock_find_by_id
    return mock

面试可能问

AI 生成测试的最大局限性是什么？怎么弥补？

最大局限：AI 不理解业务逻辑的"预期行为"——它只能根据代码推导输入输出，但无法判断这个行为是否正确。例如：一个含 Bug 的计算函数，AI 会生成验证 Bug 输出的测试。弥补方法：1) AI 生成测试骨架，人工确认断言条件（expected value）的准确性；2) 用 Property-Based Testing（jqwik/Hypothesis）补充——AI 生成生成策略而非具体值；3) 先写测试再写代码（TDD）——人工定义预期行为，AI 生成实现代码。

AI 生成的 Mock 和真实系统行为不一致怎么办？

这是 Mock 的固有问题——Mock 只能模拟"你认为系统会怎么响应"，但真实行为可能不同。缓解：1) 使用录播（VCR）库（如 vcrpy）录制真实系统的响应，重放给测试；2) AI 生成的 Mock 先经过人工 review，确认 side_effect 和 return_value 符合真实 API 行为；3) 集成测试（Integration Test）用真实的系统调用，不 Mock。AI 生成 Mock 适合用单元测试，集成测试应该人工编写。

覆盖率 100% 但线上仍有 Bug，怎么系统性地提升测试有效性？

覆盖率 100% 只代表每一行代码都被执行过，不代表所有执行路径和条件都被测试到。系统性措施：1) 用分支覆盖率（Branch Coverage）替代行覆盖率——确保每个 if/else 分支都测试；2) 用变异测试（Pitest）——修改代码逻辑，测试能不能发现；3) 用基于属性的测试——随机生成大量输入，测试不变性；4) AI 分析代码的所有路径（圈复杂度），确保每条路径都有测试。

判断练习

场景：AI 生成的测试覆盖了 create_ticket 的所有分支路径，但运行时有 2 个测试失败，怎么排查？

排查：1) 测试给的输入参数是否在业务允许范围内（如 priority="urgent" 是否真的被支持？检查代码逻辑）；2) 浮点精度问题——如果涉及计算，检查 fixture 中的参数是否精确匹配；3) 时间相关测试——AI 生成的测试可能用了固定时间但代码用了真实时间（now()），导致断言失败。修复策略：让 AI 重新检查代码的精确行为，调整测试的参数和断言条件。

场景：团队希望将 AI 生成的测试作为 PR 的必须环节，怎么设计流程？

流程：1) 开发者完成功能代码后，运行 claude test -f {changed_files} 生成测试；2) AI 生成的测试随代码一起提交到 PR；3) CI 中运行 pytest --cov --cov-fail-under=80，覆盖率 < 80% 禁止合并；4) AI Review 检查"测试是否覆盖了所有分支路径"；5) 人工确认测试的断言条件正确。流程自动化程度约 80%，人工仍需要确认"测试测对了业务逻辑"。
