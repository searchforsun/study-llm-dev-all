为什么要学本章

管理层会问：「AI 工具花了多少钱、提效多少、有没有引入更多缺陷？」若只有主观感受，无法持续采购与排期。本章在 自定义 AI Agent 工作流设计 落地后，用可观测指标回答这些问题，并与 AI 辅助 Code Review 与调试 的评审数据打通。

本章先记住 3 件事
  • 度量要先定义口径：接受率、采纳行数、会话数不可混谈。
  • 缺陷要归因到变更类型：区分 AI 辅助提交与纯人工提交。
  • 治理是赋能而非禁止：明确红线（密钥、客户数据）比一刀切禁 AI 更有效。

AI 代码接受率统计

CorpAssist 场景：Python 团队在 Cursor 中生成 RAG 检索逻辑，部分建议被 Tab 接受、部分在 Chat 中被复制。Tech Lead 需要周报：「本周 AI 建议接受率是否下降？是否因模型切换？」

AI acceptance rate 是治理层最常被问到的指标之一。接受率常见口径：

指标分子分母用途
Tab 接受率接受的 inline 建议行数展示的建议行数衡量补全质量
Chat 采纳率合并进分支的 AI 生成行（git blame 标记）AI 会话产生的候选行衡量对话生成可用性
PR AI 占比标记为 Co-authored-by: AI 的 commit 行PR 总新增行治理与审计

采集示例（Git + 标签）

python 
# scripts/metrics/ai_lines.py — 统计 PR 中 AI 辅助行（示意）
import subprocess
import re

AI_TRAILER = re.compile(r"^AI-Assisted:\s*(\d+)$", re.M)

def ai_lines_in_range(base: str, head: str) -> dict:
    log = subprocess.check_output(
        ["git", "log", f"{base}..{head}", "--format=%B%n---"],
        text=True,
    )
    total_ai = 0
    for block in log.split("---"):
        m = AI_TRAILER.search(block)
        if m:
            total_ai += int(m.group(1))
    diffstat = subprocess.check_output(
        ["git", "diff", "--shortstat", base, head], text=True
    )
    # 解析 insertions...
    return {"ai_lines": total_ai, "raw": diffstat.strip()}

若采购谈判看「编辑器价值」→ Tab；若合规看「合并进生产的 AI 代码比例」→ PR 占比。CorpAssist 建议两者都采，分开展示。

接受率下降是否应立刻限流 AI？
先排查：rules 是否变更、任务类型是否变化、是否新同学上手。若同期 defect density 升，再收紧 Judge 与 Review rubric，而非一刀切禁工具。

生成代码缺陷密度追踪

Defect density 在此定义为：每千行变更在发布后 14 天内关联的 P1/P2 缺陷数。需给 commit / PR 打标签 ai-assisted vs human-only。若密度连续高于基线，应联动 Model-as-Judge 代码质量门禁 收紧 rubric，而非简单禁用 AI 工具。

缺陷归因数据流
flowchart LR
  PR[Pull Request\n标签 ai-assisted] --> Merge[合并 main]
  Merge --> Deploy[发布 CorpAssist]
  Deploy --> Sentry[Sentry / 工单]
  Sentry --> Join[按 commit SHA 关联]
  Join --> Report[缺陷密度报表\nper 1k LOC]

从 PR 标签到工单系统，闭环计算 AI vs 人工缺陷密度。

反例
  • 把所有线上 Bug 都算在「AI 不好」上，无对照组
  • 不区分模块（RAG vs 网关配置）
推荐
  • 同模块、同周期对比 AI-assisted 与 human-only PR
  • 结合 AI 生成代码的质量管控 中的静态检查通过率

缺陷密度常见误区

误区后果纠正
14 天窗未统一与发布节奏错位以 prod 部署日为 T0
把所有 Bug 算给 AI错误停用工具human-only 对照组
不拆模块掩盖 tests/ 与 rag/ 差异组件标签
sql 
-- 示意：按 PR 标签计算 14 日缺陷密度（千行）
SELECT
  pr_label,
  COUNT(issue_id) * 1000.0 / NULLIF(SUM(lines_changed), 0) AS defect_per_kloc
FROM pr_metrics
JOIN issues ON issues.first_commit_sha = pr_metrics.merge_sha
WHERE merged_at >= CURRENT_DATE - INTERVAL '14 days'
GROUP BY pr_label;
缺陷密度略高于人工是否就要回滚 AI 政策？
看 P0/P1 占比与模块。若集中在 AI 生成的 tests/ 样板，加强 Judge；若在 rag/ 核心，降低 AI 行占比并强制双人 Review。与 Model-as-Judge blockers 联动分析。
工单系统无 commit 关联怎么办？
最低配：MR 描述填 merge SHA；Sentry release 与 SHA 对齐。过渡期可手工抽样 20 个 AI PR，勿用不完整数据做采购决策。

度量看板与告警

看板建议四层：用量（会话/Token）、效率（PR 周期）、质量（缺陷密度、回滚率）、合规（策略违规次数）。

yaml 
# alerting/corpassist-ai-metrics.yaml（示意）
groups:
  - name: corpassist_ai_governance
    rules:
      - alert: AiDefectDensityHigh
        expr: defect_density_ai_14d > 1.5 * defect_density_human_14d
        for: 7d
        labels:
          team: corpassist-python
        annotations:
          summary: "AI 辅助变更缺陷密度连续 7 天高于人工基线 50%"
      - alert: AiAcceptanceRateDrop
        expr: ai_tab_accept_rate_7d < 0.25
        for: 3d
        annotations:
          summary: "Tab 接受率跌破 25%，检查模型或规则变更"

Alert threshold 应经两周基线校准；避免上线首日误报风暴。告警路由到 Tech Lead + 工具管理员，而非全员。

接受率下降一定是模型变差吗？
不一定。常见原因：① 同事修改了 .cursor/rules 导致建议偏离项目风格；② 本周任务类型变化（更多 infra yaml，补全命中率天然低）；③ 开发者更谨慎，拒绝低置信建议。应结合会话备注与规则变更日志排查。

周报模板（可直接复用）

markdown 
## CorpAssist Python · AI 工具链周报（第 N 周）

### 用量
- 活跃开发者：12 / 15
- AI 会话数：340（WoW +8%）

### 效率
- 中位 PR 周期：1.8d（baseline 2.4d）
- AI-assisted PR 占比：62%

### 质量
- AI 缺陷密度（14d）：0.4 / kLOC
- 人工缺陷密度（14d）：0.35 / kLOC
- 结论：AI 略高于人工，集中在 tests/rag/ 模块 → 加强 Judge rubric

### 合规
- 策略违规：0
- 待办：更新 Skill registry 审批流程

将「质量」段的 Judge 失败原因 Top3 与 Model-as-Judge 代码质量门禁 的 blockers 字段对齐，可快速定位 rubric 缺口。

没有 Cursor Teams 订阅能否做度量？
可以。最低配：Git trailer 标记 AI 行 + 工单系统组件 + 脚本周报。缺失 IDE 侧 Tab 率时，在周报中标注「未采集」，避免与 PR 占比混谈。
治理政策谁维护？
建议三角：Tech Lead（技术条款）、安全（数据分级）、法务（合同与驻留）。每季度复盘，变更走 ADR 或 docs/adr/。

常见坑

  • 只采 IDE 指标：无法回答「合并质量」，必须加 PR 标签。
  • 告警无 owner：消息进大群无人处理，等于没有治理。
  • 政策不可见：研发找不到 docs/ai-tooling-policy.md，违规重复发生。
  • 与 Judge 脱节：blockers 未进周报，缺陷密度升因不明。

落地节奏建议

  1. 第 1 周：统一 PR 标签与 Git trailer，手工周报。
  2. 第 2～3 周：接通工单 Join，发布 usage dashboard 只读版。
  3. 第 4 周：启用 alert threshold，与 Judge blockers 联动。
  4. 持续：每季度复审 governance policy。

若 AI acceptance rate 与 defect density 同时恶化，优先检查是否更换了默认模型或删除了关键 rules，而非追加更多 AI 会话配额。

本章结论

  • 接受率：分 Tab / Chat / PR 口径采集，禁止混算。
  • 缺陷密度：AI-assisted 与 human-only 对照，按模块分层。
  • 治理规范：红线场景清单 + Prompt 真源仓库化。
  • 看板告警：用量/效率/质量/合规四层，阈值基于基线校准。

动手练习

设计 CorpAssist Python 团队的一周 AI 度量周报字段，并写一条 Prometheus 告警规则草案。

操作步骤

  1. 复制「接受率」表到你的笔记，为 Tab / Chat / PR 各写一句「本周解读」示例。
  2. 根据 Mermaid「缺陷归因数据流」，列出 Join 步骤需要的两个系统字段（如 commit SHA、工单组件）。
  3. 从治理决策表选 2 行，写成 docs/ai-tooling-policy.md 的正式条款（允许/禁止+要求）。
  4. 修改告警 YAML：将 1.5 * 改为你们认为合理的倍数并注释理由。
  5. 可选：在 demos/advanced-02-lab/ 用假数据填一张周报 Markdown。

判断练习

  1. （场景）工单系统将 Bug 组件标为「RAG」但未关联 commit，缺陷密度报表会怎样？如何修复？

    参考答案

    Join 失败导致 AI 密度被低估或为 0。修复：强制 Bug 模板含 merge commit SHA；CI 合并时写回 release 版本号。

  2. （场景）Tab 接受率从 40% 降到 28%，但 PR 周期缩短 15%，是否应停用 AI 工具？

    参考答案

    不应仅凭 Tab 率停用。若 PR 周期缩短且缺陷密度未升，可能开发者更多用 Chat 批量生成。应分拆指标，并抽查本周合并 PR 质量。

Demo：advanced-02-lab

验收：周报含 4 类指标；告警 YAML 通过 promtool check rules（若已安装 Prometheus 工具链）。将周报发给 Tech Lead 评审时，附 1 条「本周 action item」（例如：为 tests/rag 补充 Judge rubric）。

复盘提示：若 defect_density_ai 连续两周高于 human，请在周报「质量」段链接到失败 Judge 的 PR 列表，避免只报数字不报上下文。
