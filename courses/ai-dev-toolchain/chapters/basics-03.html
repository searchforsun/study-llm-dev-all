为什么要学本章

Prompt 驱动开发工作流 提高了生成效率，但 LLM 仍会产出看似正确、实则漏边界的代码：错用 blocking IO、臆造 env 变量、测试只覆盖 happy path。本章用 静态检查 + AI Code Review 与覆盖率监控，为 CorpAssist RAG 服务设质量底线。

本章先记住 3 件事
  • AI 代码 ≠ 已审查代码：merge 前必须过自动化 + 人审。
  • 静态检查抓结构：AI Review 抓语义与安全；二者互补。
  • 比例要度量：接受率、缺陷密度、覆盖率变化要可追踪。

生成代码的常见缺陷

在 CorpAssist 仓库中，AI 代码异味 常出现在 RAG 检索、向量客户端与 FastAPI 路由层。识别模式比「感觉不对」更可靠。

CorpAssist · 一次「几乎合并」的 MR AI 为 hybrid_search() 生成了 120 行实现，pytest 全绿，但 Review 发现：在 async 路由里用了 requests.get、硬编码了 top_k=10、测试未覆盖空索引分支 — 若上线将造成事件循环阻塞与静默错误。

AI 常生成 · 阻塞 async
python 
# 反例：async 路由内同步 HTTP
import requests

async def fetch_embeddings(text: str) -> list[float]:
    resp = requests.post(EMBED_URL, json={"input": text}, timeout=30)
    return resp.json()["data"][0]["embedding"]
CorpAssist 推荐
python 
import httpx

async def fetch_embeddings(client: httpx.AsyncClient, text: str) -> list[float]:
    resp = await client.post("/v1/embeddings", json={"input": text}, timeout=30.0)
    resp.raise_for_status()
    return resp.json()["data"][0]["embedding"]
先想 10 秒：pytest 全绿为什么仍不能 merge？

测试可能未覆盖缺陷路径（空索引、超时、422）；静态检查未跑或 AI 只测了 mock happy path。需要分层门禁。

静态检查 + AI Review 双保险

静态分析 在 CI 里快速拦截语法、风格与部分逻辑问题；AI Code Review 对 MR diff 做语义审查（安全、并发、契约）。CorpAssist 推荐先机器后模型再人。

MR 质量双保险流水线
flowchart TD
  PR[Pull Request] --> R[ruff + mypy]
  R --> T[pytest + cov]
  T --> A[AI Review Bot]
  A --> H[人类 Reviewer]
  H --> M[Merge]
  R -->|fail| X[阻断]
  T -->|fail| X

AI Review 不应替代 ruff/mypy；它补充「业务逻辑与安全隐患」视角。

AI Review Prompt 要点（CI 或 Bot）

markdown 
审查此 MR diff（CorpAssist Python RAG 服务）：
1. async 路由是否存在 blocking IO？
2. 是否新增未文档化的 env 或硬编码 secret？
3. API 字段是否与 docs/openapi/ 一致？
4. 测试是否覆盖失败路径？
输出：P0/P1/P2 问题列表，每条含文件:行号
单保险风险
  • 仅 ruff：语义错误漏网
  • 仅 AI Review：幻觉误报/漏报，且无 deterministic 基线
双保险收益
  • ruff/mypy 快速反馈（<1 min）
  • AI 聚焦安全与 RAG 边界条件
  • 人审只看 P0/P1 与架构

测试覆盖率监控

AI 生成测试常覆盖率高但断言弱。CorpAssist 应监控分支覆盖率与关键模块下限，尤其 corpassist/rag/ 与 corpassist/api/。可配合 静态分析 的 ASYNC rules 在 CI 中与覆盖率同 gate。

覆盖率策略示意

模块行覆盖下限分支覆盖说明
corpassist/rag/85%必看检索逻辑边界多
corpassist/api/80%必看HTTP 错误码路径
corpassist/schemas/70%可选Pydantic 校验器重点

CI 覆盖率门禁（bash）

bash 
#!/usr/bin/env bash
set -euo pipefail
uv run pytest tests/ --cov=corpassist --cov-branch --cov-report=term-missing
uv run coverage report --fail-under=80
# 可选：diff-cover 只检查本 MR 新增行覆盖率

对 AI 生成的测试文件，Reviewer 应抽查：是否 mock 了被测逻辑本身（导致假绿）。

pytest 覆盖率报告解读（节选）

text

src/corpassist/rag/retriever.py     142     18     44     12    84%
src/corpassist/api/routes/rag.py      56      4     12      2    91%

关注 BrPart（分支部分覆盖）与 retriever.py 的 Miss 行：AI 测试常漏「向量库空结果」「embedding 超时」分支。团队可为 测试覆盖率门禁 设模块级 fail-under。

BrPart 升高说明什么？

部分分支只跑了一半路径（如 if 只测了 true）；需补 else/except 用例，而非再堆 happy path。

可接受的 AI 代码比例

AI 代码可接受比例 不是「越高越好」，而是与缺陷密度、Review 容量平衡。建议 CorpAssist 团队度量以下指标（advanced 阶段将建看板）。

接受率

AI 建议中被开发者保留的行数 / 总建议行数。过低说明 Prompt 差；过高且缺陷多说明 Review 虚设。

AI 行占比

合并代码中标记或估算的 AI 生成比例。核心路径（鉴权、计费）宜更低。

AI 缺陷密度

合并后 30 天内与 AI 变更相关的 bug 数 / KLOC。用于调 Prompt 与 Rules。

团队政策示意（非硬性标准）

  • 样板代码（DTO、测试脚手架）：AI 行占比可达 70–80%，须过静态检查。
  • RAG 核心检索：AI 辅助 <40%，算法改动需双人 Review。
  • 安全/密钥相关：禁止 AI 直接 merge，仅允许生成测试或文档草稿。

规则与 Review 流程整合见 规则与约定驱动开发；实践阶段 AI 生成测试用例 将深化 pytest 自动化。

接受率 90% 是否一定健康？
不一定：若 Review 流于形式，高接受率伴高缺陷密度仍危险。须与合并后 bug 率、覆盖率变化联看。
能否禁止 RAG 模块使用 AI？
极端政策会损失效率；更可行是低占比 + 双人 Review + 更高覆盖率下限。

GitHub Actions 双保险（示意）

yaml 
# .github/workflows/quality.yml（节选）
jobs:
  static:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: uv sync --frozen
      - run: uv run ruff check src tests
      - run: uv run mypy src/corpassist
  test:
    needs: static
    steps:
      - run: uv run pytest tests/ --cov=corpassist --cov-fail-under=80

本章结论

  • 常见缺陷：async 阻塞、契约漂移、弱测试、密钥与吞异常 — 用模式表 Review。
  • 双保险：ruff/mypy/pytest 先行，AI Review 补语义与安全，人审收 P0/P1。
  • 覆盖率：分支覆盖 + 模块下限；警惕 mock 导致的假绿。
  • 可接受比例：分模块政策 + 接受率/缺陷密度度量，非一刀切禁止 AI。

动手练习

审查一段 AI 生成的 hybrid_search 片段，输出 P0/P1 清单并补 2 条 pytest（约 25 分钟）。

操作步骤

  1. 从 Demo 目录取「问题代码」片段，用本章缺陷表标至少 3 个问题。
  2. 写 AI Review Prompt，对 diff 输出 P0/P1（含行号）。
  3. 本地运行 uv run ruff check 与 uv run mypy src/corpassist（或 Demo 等价命令）。
  4. 补 2 条测试：空索引分支 + 非法参数 422。
  5. 在 MR 描述中填写 AI 行占比估算（本章政策对照表）。

判断练习

  1. MR 行覆盖 92%，但新增 async 路由使用 requests。应 merge 吗？

    参考答案

    不应：覆盖率高不能抵消 P0 并发缺陷；应改 httpx 并重跑 CI。
