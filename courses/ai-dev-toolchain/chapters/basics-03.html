已完成AI 生成代码的质量管控

约 25 分钟 · 阶段：AI 编码工具全景 · 能力：识别生成代码的常见缺陷、应用静态检查+AI Review 双保险、监控测试覆盖率、控制 AI 代码比例

为什么学这章

AI 生成代码不是完美的——可能包含 API 版本错误、安全漏洞（SQL 注入、CSV 注入）、错误处理缺失、并发问题。CorpAssist 团队曾因 AI 生成的代码未处理空指针（NPE）导致线上故障。建立质量管控体系后，AI 相关 Bug 减少 70%。

学完后能做什么

识别 AI 生成代码的 6 类常见缺陷；配置静态检查工具（SonarQube/Checkstyle）+ AI Review 双保险；监控单元测试覆盖率确保 AI 代码被测试覆盖；设定和追踪团队的可接受 AI 代码比例。

本章要点速记

1) 6 类常见缺陷：API 版本/安全漏洞/错误处理/并发/NPE/幻觉 API；2) 静态检查（SonarQube）拦截 60% 缺陷，AI Review（Claude Code）拦截额外 25%；3) 新生成代码的单元测试覆盖率目标 > 80%；4) AI 代码比例建议 30-50%（新功能），核心模块 < 20%。

java
// 典型缺陷示例：AI 生成的 CSV 导出代码（有漏洞）

// ❌ AI 生成的缺陷代码
public void exportCsv(List tickets, Writer writer) {
    for (Ticket t : tickets) {
        // 漏洞：没有对内容做 CSV 转义
        writer.write(t.getId() + "," + t.getTitle() + "\n");
        // 如果 title = "=SUM(A1:A10)"，Excel 会执行公式！
    }
}

// ✅ 修复后的代码
public void exportCsv(List tickets, Writer writer) {
    for (Ticket t : tickets) {
        writer.write(escapeCsv(t.getId()) + "," + escapeCsv(t.getTitle()) + "\n");
    }
}

private String escapeCsv(String value) {
    if (value == null) return "";
    // CSV 注入防护：= + - @ 开头的字段加单引号前缀
    if (value.matches("^[=+\\-@].*")) {
        return "'" + value;
    }
    // 包含逗号或引号的字段用双引号包裹
    if (value.contains(",") || value.contains("\"")) {
        return "\"" + value.replace("\"", "\"\"") + "\"";
    }
    return value;
}

静态检查 + AI Review 双保险

单靠静态检查（SonarQube）或人工审查都不够。双保险策略：静态检查拦截可自动发现的缺陷，AI Review 发现逻辑和语义问题，人工确认最终质量。

flowchart TB
    AI_GEN[AI 生成代码] --> COMPILE[编译检查]
    COMPILE -->|通过| STATIC[静态检查
SonarQube/Checkstyle]
    COMPILE -->|失败| AI_GEN
    STATIC -->|通过| AI_REV[AI Review
Claude Code]
    STATIC -->|有缺陷| AI_GEN
    AI_REV -->|通过| HUMAN[人工审查
PR Review]
    AI_REV -->|发现问题| AI_GEN
    HUMAN -->|通过| MERGE[合并]
    HUMAN -->|有问题| AI_GEN
yaml
# CI 配置：双保险检查
# .github/workflows/ai-code-quality.yml
name: AI Code Quality Check
on: [pull_request]

jobs:
  quality-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 第 1 道防线：编译检查
      - name: Compile Check
        run: mvn compile -q

      # 第 2 道防线：静态分析
      - name: Static Analysis
        run: |
          mvn sonar:sonar \
            -Dsonar.host.url=${{ secrets.SONAR_HOST }} \
            -Dsonar.login=${{ secrets.SONAR_TOKEN }}

      # 第 3 道防线：AI Review
      - name: AI Code Review
        run: |
          claude "审查以下 PR 的代码变更，检查：
          1. 是否有安全漏洞（CSV注入/SQL注入/XSS）
          2. 是否有空指针风险
          3. 是否正确处理了异常
          4. 是否遵循项目的架构规范
          输出 JSON 格式的审查结果。" \
            --input-diff ${{ github.event.pull_request.diff_url }}

      # 第 4 道防线：人工审查（PR 必须 review 后才能合并）
      - name: Require Review
        run: |
          gh pr merge --merge --auto

只有 SonarQube

SonarQube 能发现 60% 的缺陷（CSV 注入、NPE），但逻辑错误（AI 用了错误的算法、API 语义不匹配）检测不到。缺陷漏检率 40%。

SonarQube + AI Review

SonarQube 拦截语法和安全问题，AI Review 发现逻辑和语义问题。组合覆盖 85% 的缺陷。剩余 15% 需要人工审查（业务逻辑正确性）。

可接受的 AI 代码比例

团队需要设定"AI 代码比例"指标，并持续追踪。不是越高越好——关键模块和复杂逻辑需要人工编写。

模块类型AI 代码比例建议原因
CRUD Controller/Service60-80%模式固定，AI 生成准确率高
DTO/VO 定义80-90%数据结构，AI 生成免手写
单元测试50-70%AI 生成用例骨架，人工补充边界
业务逻辑（核心）10-20%领域规则复杂，AI 可能理解偏差
安全/加密模块0-10%错误代价高，建议人工编写
配置文件30-50%AI 生成骨架，人工调整参数
数据库迁移脚本20-40%AI 生成基础 SQL，人工确认索引
bash
# AI 代码比例统计脚本
# 使用 git blame + AI 标记 统计 AI 生成的代码行数

# 思路：AI 生成的代码在 git blame 中有特殊标记
# Cursor 生成的代码 commit message 含 "🤖 Cursor"
# Claude Code 生成的 commit message 含 "🤖 Claude Code"

echo "AI 代码比例统计（本月）"
echo "="

# 统计总代码行
TOTAL=$(git log --since="2026-05-01" --until="2026-05-31" \
  --format='' --numstat | awk '{sum+=$1} END {print sum}')
echo "本月新增代码: $TOTAL 行"

# 统计 AI 生成行（按 commit 标记）
AI=$(git log --since="2026-05-01" --until="2026-05-31" \
  --grep="🤖" --format='' --numstat | awk '{sum+=$1} END {print sum}')

RATIO=$(echo "scale=1; $AI * 100 / $TOTAL" | bc)
echo "AI 生成代码: $AI 行"
echo "AI 比例: $RATIO%"

# 按模块统计
echo "按模块统计:"
for MODULE in controller service repository config; do
  M_TOTAL=$(find src/main/java -path "*$MODULE*" -name "*.java" \
    -exec git log --since="2026-05-01" --format='' --numstat {} + \
    | awk '{sum+=$1} END {print sum}')
  M_AI=$(find src/main/java -path "*$MODULE*" -name "*.java" \
    -exec git log --since="2026-05-01" --grep="🤖" --format='' --numstat {} + \
    | awk '{sum+=$1} END {print sum}')
  if [ "$M_TOTAL" -gt 0 ]; then
    echo "  $MODULE: $(echo "scale=1; $M_AI*100/$M_TOTAL" | bc)%"
  fi
done

面试可能问

AI 生成代码的"幻觉 API"（幻影 API）问题怎么系统性地检测和防止？

系统性方案：1) 编译检查（最直接——不存在的类编译不过）；2) 依赖检查工具（如 `mvn dependency:analyze` 检查未使用的依赖和未声明的依赖）；3) IDE 插件（Cursor 有内置的编译错误检测，`Cmd+K` 时自动检查）；4) AI Review Prompt 中明确要求"检查所有 import 的类是否在当前项目的依赖中"。防止：在 .cursorrules 中列出项目依赖的核心库和版本，减少 AI 使用未知类库的概率。

团队 AI 代码比例从 20% 提升到 60% 后，Bug 率反而下降了，为什么？

可能原因：1) 原来人工编写的 CRUD 代码容易出错（拼写错误、Copy-Paste 遗漏），AI 生成避免了这类低级错误；2) SonarQube + AI Review 双保险拦截了大部分缺陷；3) AI 生成的代码更一致（遵循相同的模式和规范）。但需要注意：如果 AI 代码比例继续提升到 80% 以上，核心业务逻辑也开始由 AI 生成，Bug 率可能反弹（复杂逻辑 AI 理解偏差）。建议监控 Bug 率随 AI 比例的变化曲线，找到最佳平衡点。

AI 审查代码时，怎么确保审查本身的 Prompt 足够全面？

AI Review 的 Prompt 需要结构化设计。CorpAssist 的 Review Prompt 包含 6 个固定检查项：1) 安全——检查 OWASP Top 10（尤其是注入和认证）；2) 空指针——所有 Optional、可能为 null 的返回值是否判空；3) 资源——文件/数据库连接/HTTP 响应是否关闭；4) 并发——实例变量是否线程安全；5) API 契约——是否与 OpenAPI 定义一致；6) 性能——是否有 N+1 查询、大对象循环等。每个检查项有具体的检查标准和示例。

判断练习

场景：AI 生成的代码在 SonarQube 中全是绿灯，但上线后出现 NPE，怎么改进检查流程？

SonarQube 的规则不是万能的——有些 NPE 模式（如 Stream API 中的 null 元素、JSON 解析中的缺失字段）SonarQube 检测不到。改进：1) 在 AI Review 的 Prompt 中增加"检查所有可能返回 null 的方法调用"这个专项检查；2) 在单元测试中增加基于 Property-Based Testing 的随机数据测试（jqwik 库），覆盖更多边界值；3) 在代码中启用 Spring 的 Null-safety 注解（@NonNull/@Nullable），让编译器帮助检查。

场景：团队新成员用 AI 工具生成了大量代码，但 AI 代码比例突然从 30% 飙升到 75%，怎么管理？

AI 代码比例飙升不一定是坏事，但需要关注质量。措施：1) 检查飙升主要来自哪些模块——如果是 CRUD 层是正常的，如果是核心业务逻辑则需要关注；2) 设置阶段性目标——新成员第一个月 AI 比例可以高（快速产出），第二个月逐步降低（学习业务）；3) 增加 AI 生成代码的审查力度——双保险 + 人工 review 时间加倍；4) 建立 AI 代码质量 Dashboard，月 Review 一次。
