为什么要学本章

在 实战：CorpAssist 开发效率提升 中你已用 AI 加速单点任务；当团队并行开发 RAG 管线、评测脚本与 API 契约时，需要把「一次性的 Prompt」升级为可版本化、可审计的 workflow DAG。本章教你用 Cursor / Claude Code 的 Cursor Skill 与 Command 把研发流水线产品化。

学完你能

  • 编写团队可共享的 Skill 与 Command，约束上下文与输出格式
  • 用 DAG 编排「需求→实现→测试→门禁」并接入 GitHub Actions
  • 在 CI 中触发 Agent 工作流且保留人工审批点
本章先记住 3 件事
  • Agent 要分工：单 Agent 包办设计+编码+评审会上下文爆炸，应用 multi-agent orchestration 拆角色。
  • Skill 是可复用 SOP：把团队规范写进 Skill，比每次粘贴长 Prompt 更稳。
  • CI 只跑确定性步骤：Agent 生成物必须经 AI 生成代码的质量管控 与测试后再合并。

多 Agent 协作模式

CorpAssist 场景：知识库团队要在两周内交付「合同条款 RAG 检索 + 引用溯源」接口。Tech Lead 用 Planner Agent 拆任务，Coder Agent 改 FastAPI 路由，Reviewer Agent 对照 OpenAPI 与 AI 生成文档与 Schema 做契约检查。
CorpAssist 研发多 Agent 协作
flowchart LR
  PO[产品/架构输入] --> Planner[Planner Agent\n拆 Issue / 子任务]
  Planner --> Coder[Coder Agent\n改 Python / 测试]
  Coder --> Tester[Tester Agent\npytest / 覆盖率]
  Tester --> Reviewer[Reviewer Agent\n安全 / 契约]
  Reviewer -->|通过| MR[Pull Request]
  Reviewer -->|打回| Coder
  MR --> CI[GitHub Actions\nlint + test + judge]

Planner→Coder→Tester→Reviewer 闭环；MR 进入确定性 CI。

模式适用CorpAssist 示例风险
顺序流水线步骤依赖强先改 chunker 再改检索 API前步错误向后传播
主从委派探索型任务Lead Agent 派发子 Agent 查依赖源码subagent delegation 超时
并行+汇总多模块无冲突同时生成单元测试与 docstring合并冲突需人工
人在回路生产变更发布索引前人工确认样本审批 SLA 需约定
先想 10 秒：你的团队更适合哪种模式？

若两人改同一 FastAPI 路由，优先顺序流水线；若一人写测试、一人写实现且文件不重叠，可并行+汇总。

自定义 Skill/Command 开发

Skill 是带触发描述的操作手册；Command 是斜杠命令入口。二者应引用仓库内的 AGENTS.md、context/ 与 .cursor/rules，避免模型「发明」项目结构。

反例：巨型 Skill
  • 一个 Skill 覆盖「从需求到上线」全部步骤
  • 无输出模板，每次 diff 格式不同
  • 未声明禁止改动的目录（如 infra/）
推荐：单一职责 Skill
  • corpassist-pytest-gen：只生成 pytest，必须跑 uv run pytest
  • 固定输出：测试文件路径 + 覆盖场景列表
  • 显式引用 AI 生成测试用例 中的 Fixture 约定

Skill 骨架示例

markdown 
---
name: corpassist-rag-api-change
description: 修改 CorpAssist Python RAG 路由时触发。生成实现+测试+OpenAPI 片段。
---

# CorpAssist RAG API 变更

## 输入
- Issue 链接或 diff 范围
- 目标模块：`app/rag/` 或 `app/retrieval/`

## 步骤
1. 阅读 `context/tech/services/corpassist-python.md`
2. 只修改列出的文件；禁止改 `deploy/`
3. 输出：代码 diff 摘要、pytest 命令、OpenAPI 路径是否变更

## 验收
- `uv run ruff check app/`
- `uv run pytest tests/rag/ -q`

Command 适合一键动作，例如 /gen-changelog 只读最近 merge commit 生成发布说明，与 规则与约定驱动开发 中的 Conventional Commits 对齐。团队共享的 Cursor Skill 应通过内部 PR 评审，版本号写入 manifest，避免每人维护冲突副本。

Workflow 编排最佳实践

工作流文件（如 .github/workflows/ai-dev.yml）描述何时调用 Agent；Workflow DAG 中每个 stage 的 artifact 应可机器校验。Skill 描述怎么做。二者分离可避免把 Prompt 硬编码进 YAML。

委派子任务时使用 Subagent delegation 规范：只读、限时、输出引用路径。合并前由 CI agent hook 把评审结论写入 Check Run，供安全审计关联。

状态机显式化

为每个阶段定义输入/输出 artifact：design.md → patch.diff → test-report.xml。

上下文预算

Coder Agent 只挂载相关目录；Planner 可读 Issue，不读 node_modules。

失败可重入

步骤 id 幂等；重跑 Tester 不应重复提交 MR。

工作流 manifest 片段

yaml 
# corpassist-ai-workflow.yaml（示意）
version: "1"
stages:
  - id: plan
    agent: planner
    inputs: ["issue_body"]
    outputs: ["tasks.json"]
  - id: implement
    agent: coder
    inputs: ["tasks.json", "repo_snapshot"]
    outputs: ["branch.patch"]
  - id: verify
    agent: tester
    inputs: ["branch.patch"]
    outputs: ["pytest.xml", "coverage.json"]
  - id: review
    agent: reviewer
    inputs: ["branch.patch", "pytest.xml"]
    gates: ["no_secrets", "openapi_unchanged_or_approved"]

与 CorpAssist CI/CD 集成

Agent 工作流在 CI 中应作为可选加速层，合并门禁仍以确定性工具为准。推荐在 PR 打开时运行「AI 辅助评审」，在 main 合并前强制 pytest + 静态检查。

yaml 
# .github/workflows/corpassist-pr.yml（节选）
on:
  pull_request:
    paths: ["app/**", "tests/**"]
jobs:
  deterministic:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: uv sync --frozen
      - run: uv run ruff check .
      - run: uv run pytest --cov=app --cov-fail-under=80
  ai_assist:
    needs: deterministic
    if: github.event.pull_request.draft == false
    runs-on: ubuntu-latest
    steps:
      - name: Invoke corpassist review agent
        env:
          CURSOR_API_KEY: ${{ secrets.CURSOR_API_KEY }}
        run: |
          python scripts/ci_agent_review.py \
            --skill corpassist-pr-review \
            --pr ${{ github.event.pull_request.number }}

CI agent hook 必须把模型输出写入 PR Comment 或 Check Run，并附带 request_id 便于与网关日志关联。敏感仓库应使用自托管 Runner + 私有模型端点（详见 企业级 AI 工具链安全与合规）。

Agent 生成的 patch 能否自动 merge？
不建议。CorpAssist 生产变更至少保留：① 人类 Approve；② 确定性测试通过；③ 对安全/契约类 diff 的 CODEOWNERS 审查。Agent 可自动开 Draft PR，不可自动合入 main。

Subagent 查依赖：何时用、何时不用

当 Coder 需要确认第三方库 API（如 langchain 某版本签名）时，委派只读 Subagent 读取 lock 文件与文档，避免把整库塞进主会话。CorpAssist 约定：Subagent 超时 120s、最多 2 次重试，输出必须为「引用路径 + 结论」而非长文复述。

Planner 输出 tasks.json 应包含哪些字段？
建议最小字段：id、title、files_allowlist、acceptance_commands、depends_on。禁止 Planner 直接写「实现细节代码」，避免 Coder 失去检索上下文的机会。
与 Java 轨同事如何共用工作流？
Python 轨 Skill 聚焦 FastAPI/pytest；Java 轨使用 Spring AI 专用 Skill。共用部分：Issue 模板、PR 检查项、request_id 规范。跨栈契约变更须同时触发 OpenAPI diff 检查（见 practice-02）。

本章结论

  • 多 Agent 协作：Planner/Coder/Tester/Reviewer 分工，避免单会话上下文失控。
  • Skill/Command：单一职责 + 固定验收命令，引用仓库真源文档。
  • Workflow 编排：YAML 管触发与 artifact，Prompt 留在 Skill 内。
  • CI/CD 集成：确定性 job 先行，Agent 评审作为可选加速且可审计。

动手练习

为 CorpAssist Python 服务起草一份「RAG API 变更」Skill，并在纸面设计 PR 上的 deterministic + ai_assist 两个 job。

操作步骤

  1. 对照本章 Mermaid 图，在笔记中写出 Planner/Coder/Tester/Reviewer 各 1 条输入与输出 artifact。
  2. 按「Skill 骨架示例」填写 corpassist-rag-api-change 的禁止目录与验收命令（ruff + pytest）。
  3. 在 workflow manifest 中为 verify 阶段增加 coverage.json 输出，并说明谁消费该文件。
  4. 从「与 CorpAssist CI/CD 集成」YAML 节选复制骨架，补全 needs: deterministic 的含义一句话。
  5. 可选：在 demos/advanced-01-lab/ 创建 Skill 文件并让小范围同事试用。

判断练习

  1. PR 上 Agent 评审通过但 pytest 失败，是否应允许合并？说明理由。

    参考答案

    不允许。确定性测试是硬门禁；Agent 评审仅辅助发现风格/安全问题，不能替代 pytest。应先修复测试再请求复审。

  2. Coder 与 Tester 两个 Agent 同时修改 tests/rag/test_retrieve.py，属于哪种协作模式？应如何调整？

    参考答案

    错误使用了并行+汇总且文件冲突。应改为顺序流水线：Coder 提交实现与测试草案 → Tester 只补充边界用例，或合并为单 Agent 会话。
