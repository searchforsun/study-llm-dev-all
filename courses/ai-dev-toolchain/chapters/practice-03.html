为什么要学本章

CorpAssist 合码前已有 ruff 与 pytest，但逻辑漏洞、Prompt 注入与 RAG 性能热点仍需人眼。AI Review 适合作为第二层：在 diff 上下文中执行 OWASP LLM 清单与 Prompt 调试 模板，并与 CI AI Review 机器人评论 PR。

学完你能

  • 使用结构化 Bug 定位 Prompt（复现步骤、假设、验证）
  • 对 RAG/Agent 代码做安全 Review 清单扫描
  • 结合 py-spy / 日志做性能热点分析 Prompt
  • 在 GitHub Actions 中集成 AI Review（仅评论、不自动合码）

本章先记住 3 件事

  • AI Review 不能替代 ruff/pytest，只补充语义与威胁建模
  • 给 AI 的上下文 = diff + 相关文件 + 错误日志，而非整个仓库
  • CI 中 AI 输出应脱敏，禁止把密钥片段贴回 PR 评论

Bug 定位 Prompt 模式

生产告警「RAG 偶发 500」时，把 Sentry 堆栈、request_id 关联日志与最近 diff @ 给 Cursor，使用固定四段式 Prompt，避免 AI 泛泛而谈。

text
【现象】POST /v1/rag/query 约 3% 返回 500，多发生在 top_k>10
【复现】附最小 curl 与 test 日志
【已排除】pytest 全绿；向量库健康
【任务】
1) 列出 3 个最可能根因（按概率排序）
2) 每个根因给出「验证命令或断点位置」
3) 给出最小修复 diff，不要重构无关代码

（粘贴 stack trace + rag/service.py 相关 80 行）
场景：日志显示 NoneType has no attribute 'strip' 在 chunk.text。AI 指出检索结果含空 Node，建议在 merge 前过滤；你补充单测覆盖空 snippet。
调试闭环
flowchart TD
    A[告警/工单] --> B[收集 request_id 日志]
    B --> C[AI 根因假设]
    C --> D[人工验证 1 条]
    D -->|确认| E[最小修复 + 回归测试]
    D -->|否定| C
python
# 修复示例：过滤空 chunk，避免 .strip() 崩溃
def normalize_chunks(chunks: list) -> list:
    return [c for c in chunks if c and getattr(c, "text", None) and c.text.strip()]

对异步 Bug，Prompt 中要求 AI 「检查 await 缺失、在 async 路由中调用同步 httpx.Client」。这与 静态分析 规则互补。

AI 给出 5 个根因都不对，下一步？

缩小上下文：只给失败请求的参数组合；添加「上一次正常发布的 git tag」对比 diff；用二分法注释检索步骤。避免一次性 @ 整个 monorepo。

性能热点分析

P95 延迟超标时，导出 py-spy 火焰图或 OpenTelemetry trace，让 AI 解读「哪一层占用」并建议优化（批量化 embedding、缓存、减少串行 LLM 调用）。

py-spy record -o profile.svg --pid $(pgrep -f "uvicorn app.main")

llm 35%串行重试并发上限 + 超时
milvus 15%无过滤索引tenant 分区键
json 序列化 10%大 citations 体截断 snippet 长度

# 批量化 embedding 示意
async def embed_batch(texts: list[str]) -> list[list[float]]:
  return await embedding_client.aembed_documents(texts, batch_size=32)

要求 AI 「估算优化后 P95 降幅区间」时，需附上优化前 baseline 数字，否则拒绝空泛「会更快」。

json
{
  "trace_id": "7f3c",
  "spans": [
    {"name": "embed", "duration_ms": 420},
    {"name": "milvus_search", "duration_ms": 180},
    {"name": "llm_complete", "duration_ms": 2100}
  ]
}

把 trace JSON 附在 Prompt 末尾，要求 AI 只建议改「占比 >25% 且非必要串行」的 span。CorpAssist 常见 wins：embedding batch、检索与 rerank 并行上限、LLM 超时从 60s 降到 15s 并配合重试预算。

CI 中集成 AI Review

推荐模式：机器门禁 + AI 评论。ruff/pytest/coverage 失败阻断；AI 通过 GitHub App 在 PR 留言，标签 ai-review:needs-human。

yaml
# .github/workflows/ai-review.yml 概念片段
jobs:
  ai-review:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - run: git diff origin/${{ github.base_ref }}...HEAD > pr.diff
      - name: Run AI review script
        env:
          REVIEW_API_KEY: ${{ secrets.REVIEW_BOT_KEY }}
        run: uv run python scripts/ai_pr_review.py pr.diff
      # 脚本：调用内部 LLM，输出 markdown 评论，禁止自动 approve
python
# scripts/ai_pr_review.py 核心逻辑（示意）
def build_prompt(diff: str) -> str:
    return f"""你是 CorpAssist 代码审查员。仅根据 diff 评论。
规则：不建议预训练/换模型；优先 tenant 隔离与测试。
输出 Markdown，分 重要/轻微。diff:\n{diff[:120_000]}"""

企业内网需走私有模型 endpoint；diff 超限时按文件拆分多次 Review 再合并。Secrets 扫描必须在送模前完成 redaction。

AI Review 与人工 Review 评论冲突怎么办？
以机器可验证项（pytest、bandit）为准；AI 与人工分歧时维护者裁决并更新 Prompt 模板。记录「误报/漏报」进 advanced 阶段 Judge 调优 backlog。
能否在 IDE 里实时 Review 未提交的 diff？
可以作个人自检；合码仍以 PR diff 为准。IDE 实时 Review 易漏 CI 才出现的路径（如仅 Linux 触发的 import）。
Review 发现 tenant 未过滤但 pytest 全绿？
说明测试未覆盖多租户场景。应补集成测试或 parametrize tenant_id；AI Review 与测试门禁互补，不能互相替代。
py-spy 需要 root 吗？
Linux 上 attach 其他用户进程可能需要权限；容器内可对 pid 1 采样。Windows 开发机可用本地压测 + trace 代替。

本章结论

  • Bug 定位 Prompt 要含现象、复现、已排除与验证任务
  • 安全 Review 必须覆盖租户隔离与 Prompt 注入，不能只看 bandit
  • 性能分析需附 profiling/trace 数据，AI 只做解读与建议
  • CI 中 AI 只评论不阻断（或仅对 labeled 文件），机器检查仍阻断

面试可能问

AI Review 误报很多，团队还要用吗？

用作提醒而非否决；维护者标记「误报」反馈微调 Prompt；高危项（tenant 泄漏）仍靠自动化测试与集成测试兜底。

能否让 AI 在 CI 里自动修复？

不建议默认开启。可选手动 workflow_dispatch 生成 patch 分支，必须经人 Review 后合并。

AI Review 与 Model-as-Judge 有何区别？

Review 偏 diff 语义与威胁建模；Judge 偏评分卡与 CI 阻断。详见 Model-as-Judge 代码质量门禁。两者可并行，不可互相替代。

判断练习

  1. AI 建议「换更大模型」降低延迟，是否采纳？

    参考答案

    通常否。先查批量化、缓存与检索过滤；换模型涉及成本与合规，走变更流程，不属于 Code Review 范畴。

  2. CI AI Review 把 API Key 打在评论里，如何处理？

    参考答案

    立即旋转密钥；修复 redaction 脚本；删除 PR 评论历史；事故复盘写入团队 Rules。
