为什么学这章

Code Review 和调试是最耗时的开发环节——CorpAssist 团队平均每次 PR Review 耗时 45 分钟，Bug 定位平均 2 小时。AI Review 可以 3 分钟完成初步审查。本章学习如何设计 Bug 定位的 Prompt 模式，在 CI 中集成 AI Review，用 AI 分析性能瓶颈。

学完后能做什么

设计 Bug 定位的 Prompt 模式（给定错误日志让 AI 分析根因）；用 AI 扫描代码的安全漏洞；用 AI 分析 JFR 快照定位性能热点；在 GitHub Actions CI 中集成 AI Code Review。

本章要点速记

1) Bug 定位 Prompt：错误现象 + 代码上下文 + 请求数据 + 已知约束；2) AI 安全扫描可以检测 OWASP Top 10 的 80%；3) 性能分析用 JFR（Java Flight Recorder）采集，AI 分析热点方法；4) CI 中 AI Review 在 SonarQube 后执行，直接评论在 PR 上。

Bug 定位 Prompt 模式

Bug 定位需要结构化的 Prompt——描述现象、上下文、约束条件。好的 Prompt 让 AI 在 80% 的情况下给出正确的根因分析。

text
# Bug 定位 Prompt 模板

## 模板结构
"""
## 错误现象
{具体的错误信息，含堆栈、状态码、错误消息}

## 代码上下文
{相关的代码文件和关键方法}

## 请求数据
{触发 Bug 的请求参数或操作步骤}

## 已知约束
{项目已知的技术约束、版本限制}

## 问题
上述 Bug 的根因是什么？给出修复方案和涉及的文件。
"""

## 实例：CorpAssist CSV 导出乱码问题
"""
## 错误现象
导出的 CSV 文件在 Excel 中打开后中文显示为乱码。
文件编码检测为 UTF-8。

## 代码上下文
TicketExportService.java:
  public void exportCsv(Writer writer) {
    // 直接用 OutputStreamWriter 输出
    // 没有设置 charset
    writer.write("ID,标题,状态\n");
  }

## 请求数据
POST /api/tickets/export → Content-Type: text/csv

## 已知约束
- 服务器编码: UTF-8
- Excel 中文版默认用 GBK 打开 CSV
- Windows 系统

## 问题
CSV 文件在 Excel 中中文乱码的根因？
"""

# AI 分析结果：
# 根因：CSV 文件是 UTF-8 编码，但 Excel 中文版
# 默认用 GBK 读取 CSV。修复方案：在 CSV 文件头部
# 写入 BOM（﻿）标记 UTF-8 编码，或者用
# OutputStreamWriter(writer, "GBK") 输出 GBK 编码。
# 推荐方案：加 BOM 标记，兼容性更好。
flowchart TB
    ERR[错误日志] --> PRS[模式匹配]
    PRS -->|NPE| NPE_PATH[空指针分析链]
    PRS -->|Timeout| TO_PATH[超时分析链]
    PRS -->|OOM| OOM_PATH[内存分析链]
    PRS -->|SQL Error| SQL_PATH[SQL 分析链]
    NPE_PATH --> CAUSE1[沿调用链回溯
检查所有可能 null]
    TO_PATH --> CAUSE2[检查网络/线程池/数据库连接池]
    OOM_PATH --> CAUSE3[检查堆转储/大对象/泄露]
    SQL_PATH --> CAUSE4[检查 SQL 语法/索引/连接]
    CAUSE1 & CAUSE2 & CAUSE3 & CAUSE4 --> ROOT[根因]

安全漏洞扫描 Review

AI 安全 Review检查代码的 OWASP Top 10 漏洞。与 SonarQube 不同，AI 可以发现逻辑层面的安全问题（如权限验证缺失）。

text
# AI 安全 Review Prompt

"""
审查以下代码的安全漏洞，按 OWASP Top 10 2026 评分。

需特别检查：
1. A1: 访问控制——是否有正确的权限校验？
2. A3: 注入——SQL/CSV/命令注入
3. A5: 安全配置——是否正确配置？
4. A7: 认证——用户身份验证？
5. A9: 日志——是否记录了安全事件？

对每个漏洞：
- 严重等级：Critical/High/Medium/Low
- 影响描述
- 修复建议和代码示例
"""

# 示例：AI 发现的权限漏洞
# 漏洞：High - TicketController.updateTicket 缺少
# 权限校验——任何用户都可以修改任何工单
# 修复：添加 @PreAuthorize("#ticketId ==
#   authentication.userId || hasRole('ADMIN')")
漏洞类型SonarQube 检测AI Review 检测
SQL 注入✅ 规则检测✅ 逻辑级检测
CSV 注入❌ 不支持✅ 可检测
XSS✅ 基础检测✅ 上下文感知
权限缺失❌ 不支持✅ 可分析
敏感数据泄露❌ 不支持✅ 可检测
SSRF❌ 不支持✅ 可检测

性能热点分析

AI 性能分析通过 JFR 快照或日志文件定位性能瓶颈。AI 可以分析调用链中哪个方法耗时最长。

bash
# 1. 采集 JFR 快照
jcmd $(pgrep -f corp-agent) JFR.start duration=60s filename=perf.jfr

# 2. AI 分析 JFR
claude "分析 perf.jfr 的性能数据，找出：
1. CPU 热点方法（Top 5）
2. 锁竞争热点
3. 数据库查询慢的调用
4. 内存分配最热的类型
5. 优化建议

输出格式：每个热点含方法名/耗时占比/调用次数/优化建议"

# 3. AI 分析慢查询日志
# 从 MySQL slow_query_log 或 Redis slowlog 分析
claude "分析 MySQL 慢查询日志，找出：
1. 最慢的 Top 5 查询（执行时间和次数）
2. 缺少索引的表
3. 优化建议（加索引/改写 SQL/加缓存）"

手动分析 JFR

用 JDK Mission Control 打开 JFR，找到热点方法需要 30 分钟。需要经验判断哪些是正常的、哪些是异常的。

AI 分析 JFR

AI 自动解析 JFR 文本转储（jcmd JFR.dump），3 分钟给出 Top 5 热点方法和优化建议。附带调用链上下文。

CI 中集成 AI Review

在 GitHub Actions 中集成 Claude Code，每次 PR 自动运行 AI Review，结果直接评论在 PR 上。

yaml
# .github/workflows/ai-code-review.yml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 获取完整 git 历史用于 diff

      - name: AI Review
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            审查这个 PR 的代码变更。

            检查清单（必须逐项检查）：
            1. 安全：CSV注入/SQL注入/XSS/权限校验
            2. 空指针：所有可能为 null 的返回值
            3. 异常处理：是否 try-catch 了应该处理的异常
            4. 资源：文件/数据库连接是否正确关闭
            5. 并发：是否有非线程安全的操作
            6. API 契约：是否与 OpenAPI Schema 一致

            对每个问题：
            - 严重等级（critical/high/medium/low）
            - 文件和行号
            - 问题说明
            - 修复建议

            如果没有发现问题，回复"✅ AI Review 通过"。

      - name: Post Comments
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review_result.md', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🤖 AI Code Review\n\n${review}`
            });

本章结论

  • Bug 定位 Prompt 四要素：错误现象 + 代码上下文 + 请求数据 + 已知约束，80% 的 Bug 可定位
  • AI 安全 Review 比 SonarQube 多检测 4 类漏洞（CSV 注入/权限/敏感数据/SSRF）

  • CI 中集成 AI Review 自动评论 PR，6 项检查清单逐项扫描
  • CorpAssist PR 流程：SonarQube → AI Review → 人工 Review，总时间从 45 分钟降至 20 分钟

面试可能问

AI Review 和人工 Review 的分工怎么设计？AI 发现了所有问题但人工还需要做什么？

分工：AI Review 负责"可自动化的检查"——安全、空指针、异常处理、资源关闭、API 契约、代码风格、重复代码。人工 Review 负责"需要理解业务逻辑的检查"——功能正确性、设计合理性、是否符合产品需求、边缘 case 处理。AI Review 承担了约 60% 的检查工作量，人工从 45 分钟减到 15 分钟。人工 Review 的核心时间应该花在"这个功能是否是用户想要的"上。

AI 性能分析发现了一个热点方法，但给出的优化建议（如改用 Stream API 并行处理）反而慢了，怎么处理？

AI 的优化建议不一定适合所有场景——并行流在小数据集上有线程调度开销。处理方法：1) 质疑策略——AI 给出的优化建议需要经过基准测试验证（JMH）；2) 提供更完整的上下文——告知 AI 这个方法的数据规模（"每秒调用 1000 次，每次处理 10 条数据"），小数据量下并行不一定更快；3) 让 AI 生成 A/B 测试代码——"生成两种实现的 JMH 基准测试代码，用 @Benchmark 对比性能"。

团队 PR 中有大量 AI 生成的代码，Review 时 AI Review 再检查 AI 生成的代码，会不会有盲区？

可能的盲区：AI Review 检查 AI 生成的代码时，可能继续沿用生成时的"错误逻辑假设"。例如：生成代码时 AI 假设了一个不存在的 API（幻觉），Review 时 AI 可能认为这个幻觉 API 是合理的。克服方法：1) AI Review 的 Prompt 中强调"检查代码中是否使用了不存在的 API/类"；2) 编译检查先行——编译不过的代码 AI Review 不会进一步检查；3) 分层 Review——SonarQube 检查语法 → AI Review 检查逻辑 → 人工确认业务正确性，每层使用不同的"智能体"减少盲区。

判断练习

场景：AI Review 的 PR 评论太长（2000 字），工程师不愿意看，怎么优化？

优化 AI Review 的输出格式：1) 按严重等级排序——Critical 优先显示，Low 级别的折叠（details 标签）；2) 每条问题控制在 3 行以内（文件:行号 + 一句话问题 + 修复建议链接）；3) 结尾给出总结建议——"共发现 5 个问题（Critical 1, High 2, Medium 2）, 预计修复时间 20 分钟"；4) 允许工程师回复 /ai-skip "这条是误报" 来标记误报，减少噪音。

场景：CI 中的 AI Review 每次运行要 3 分钟，团队觉得太慢想关掉，怎么优化性能？

优化：1) 增量审查——只审查 PR 中变更的文件，不是全量项目（用 git diff 限制输入范围）；2) 文件过滤——忽略测试文件、DTO 文件、配置文件（这些 AI Review 价值低）；3) 缓存——如果同一段代码在多 PR 中重复出现（如来自模板），缓存审查结果；4) 异步——AI Review 不阻塞 PR 合并流程（非 required check），结果作为评论参考。优化后每次 AI Review 约 30-60 秒。
