项目级 Rules 编写

CorpAssist Python 仓库建议采用分层 Rules：全局 CLAUDE.md / Cursor Rules + 模块级 README + OpenAPI 真源。与 Prompt 驱动开发工作流 的 L1 Context 一致，但本章强调治理与变更流程。

CorpAssist · 新同学第一周 新人用 Cursor 生成 POST /v1/agents/run 路由，返回体用了 camelCase，错误码未带 request_id。若有清晰 项目 Rules 与 MR checklist，Issue 在 Review 前即被 ruff/模板拦截。

全局 Rules

测试命令、错误体格式、禁止项；globs 覆盖 src/**/*.py。

模块 Rules

RAG：向量维度、hybrid 权重范围；Agent：工具调用日志规范。

变更流程

改 Rules 与改 API 一样走 MR；版本号写入 CHANGELOG。

先想 10 秒：Rules 写 500 行好还是 50 行 + 链接好？

50 行可执行摘要 + 链接更好：Agent context 有限；细节放 docs/standards/，Rules 只保留必遵守项与命令。

代码风格自动对齐

代码风格自动对齐 减少 Review 噪声，让 AI 与人审聚焦逻辑。CorpAssist 统一 ruff format + ruff check，禁止「各用各的 Black 配置」。

pre-commit 配置示例

yaml 
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.8.4
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
  - repo: local
    hooks:
      - id: mypy
        name: mypy
        entry: uv run mypy src/corpassist
        language: system
        pass_filenames: false
Review 浪费在风格
  • MR 200 行里 80 行是 import 顺序与引号
  • AI 每次生成不同 formatting
  • Reviewer 手动留言「请跑 formatter」
约定 + 自动 fix
  • pre-commit 本地与 CI 同一配置
  • Rules 写「提交前须 ruff format」
  • AI Prompt 加「遵循 ruff，勿手排 import」

在 Rules 中引用可执行命令而非抽象形容词（「整洁」），Agent 才能自动跑验证。

ruff 与 AI 协作：Rules 片段

markdown 
## 格式
- 提交前：`uv run ruff format src tests && uv run ruff check src tests --fix`
- import 顺序遵循 ruff isort；勿手改
- AI 生成代码若与 ruff 冲突，以 ruff 为准自动 fix

API 契约一致性

CorpAssist 与 Java 网关/BFF 共享 OpenAPI。API 契约一致性 要求：改 Python 实现前先改 docs/openapi/，CI 跑 schema 与实现对比。

契约优先改动流
flowchart LR
  O[openapi.yaml diff] --> S[Pydantic schema]
  S --> R[FastAPI routes]
  R --> T[contract tests]
  T --> CR[Code Review]

AI 生成代码时，Context 包必须包含 OpenAPI 片段，避免字段名漂移。

契约测试示例（pytest + schemathesis 思路）

python 
# tests/contract/test_openapi_rag.py
import json
from pathlib import Path

import pytest
import yaml
from jsonschema import validate

@pytest.fixture(scope="module")
def rag_query_schema() -> dict:
    spec = yaml.safe_load(Path("docs/openapi/openapi.yaml").read_text())
    return spec["components"]["schemas"]["QueryResponse"]

def test_query_response_matches_openapi():
    sample = {
        "answer": "根据制度...",
        "citations": [{"doc_id": "d1", "chunk_id": "c1", "snippet": "..."}],
        "request_id": "req-abc",
    }
    validate(instance=sample, schema=rag_query_schema())

check_openapi.sh（CI 门禁）

bash 
#!/usr/bin/env bash
set -euo pipefail
# 禁止未更新 OpenAPI 就改 route 签名（示意：对比 MR 变更文件列表）
if git diff --name-only origin/main | grep -q 'src/corpassist/api/' \
   && ! git diff --name-only origin/main | grep -q 'docs/openapi/'; then
  echo "API 代码变更须同步 docs/openapi/"
  exit 1
fi

Bot 预审通过是否可跳过人工 Review？
不可。Bot 漏报语义与业务风险；rag/ 等 CODEOWNERS 路径须 Owner 合入。
Rules 变更谁批准？
平台组或架构组 Review；与改 CI 同级，因影响全员 Agent 行为。
CorpAssist · 跨栈契约联调 Java BFF 调用 Python /v1/rag/query。MR 仅改 Python 响应字段而未更新 OpenAPI，联调阶段 Jackson 反序列化失败。约定驱动要求：OpenAPI PR 与 Python MR 同迭代合并，或 Python 侧 CI 阻断。

本章结论

  • 项目 Rules：分层、短摘要、可执行命令；与 OpenAPI 同级维护。
  • 风格对齐：ruff + pre-commit 本地与 CI 一致；Review 不讨论格式。
  • 契约一致：openapi 先 diff；契约测试 + 脚本门禁防漂移。
  • Review 整合：Bot 预审 + 模板 checklist + CODEOWNERS；声明 AI 占比。

动手练习

为 CorpAssist 草稿仓库添加 Rules 片段、pre-commit 与 PR 模板（约 30 分钟）。

操作步骤

  1. 创建 .cursor/rules/corpassist-python.mdc，含测试命令、错误体、禁止项各 1 条。
  2. 添加 .pre-commit-config.yaml 启用 ruff + ruff-format。
  3. 编写 .github/pull_request_template.md，含 OpenAPI、pytest、AI 占比三项 checkbox。
  4. （可选）写 scripts/check_openapi.sh 并在本地 dry-run。
  5. 在团队频道分享 MR 模板，收集 1 条改进建议。

判断练习

  1. 作者只改了 Pydantic 字段名，未改 OpenAPI，CI 静态检查全过。应否 merge？

    参考答案

    否：违反契约一致；Java 客户端将解析失败。须先更新 openapi.yaml 与契约测试。
