实战场景：CorpAssist RAG 检索端点从零构建

实战目标：为 CorpAssist 构建一个 RAG 知识库检索端点 POST /api/v1/rag/query，接收用户查询和过滤条件，返回基于员工权限的检索结果和 AI 生成答案。技术栈为 Python 3.12 + FastAPI + SQLAlchemy 2.0 async + LangChain + ChromaDB。整个开发过程分为五个阶段，每个阶段都使用 AI 工具辅助但保持开发者的主导权。以下沿袭 chapters/practice-04-lab/ 目录中的实现。

AI 辅助 RAG 管线开发全流程

整个开发过程分为五个阶段。每个阶段都有明确的输入输出和审查点，确保前一个阶段的质量后才进入下一阶段。

阶段一：需求分析与 Spec 编写 (30 分钟)。开发者使用 Given-When-Then 模板编写技术规格文档。Spec 定义了端点的输入参数（query 必填、top_k 可选、filters 可选）、输出格式（answer、sources、latency_ms）和错误码范围（400/404/429/500）。Spec 还包含了测试案例——正常检索、空结果、超长 query、无效 filter 四种场景。Spec 文档是后续所有阶段的唯一真理来源。

阶段二：架构设计与组件选型 (30 分钟)。AI 根据 Spec 生成技术方案：Embedding 模型使用 text-embedding-3-small（维度 1536），Chunk 策略为 256 tokens 重叠 32 tokens，Retriever 配置为 TopK=5 加相似度阈值 0.7，Prompt 模板采用 System/User/History 三段式。开发者审查确认方案的合理性后进入下一阶段。

阶段三：代码生成 (2 小时)。AI 分段生成六个文件：config.py（配置管理）、embedding.py（Embedding 封装）、vector_store.py（ChromaDB 操作）、rag_pipeline.py（检索 + 生成管线）、rag.py（FastAPI 路由）和 test_rag.py（测试）。每个文件生成后立即编译验证，通过后再生成下一个文件。

阶段四：测试与覆盖率 (30 分钟)。AI 根据 Spec 中的测试案例生成 pytest 测试，覆盖正常路径和异常路径。运行覆盖率检查后 AI 自动补充缺失的分支测试。最终分支覆盖率达到 92%。

阶段五：文档与 Review (30 分钟)。AI 生成 Google 风格 docstring、在 OpenAPI Schema 中补充业务描述、检查 API 一致性并生成审查报告。开发者确认后 PR 合并。

Agent 工作流自动生成与多文件一致性

Agent 工作流是 AI 辅助开发的高级模式——AI 不是单次生成代码，而是自主规划步骤、分步执行、在步骤间验证一致性。Claude Code 的 Agentic 模式在生成 CorpAssist RAG 管线时会自动执行以下步骤：第一步，读取 Spec 文档和项目结构，输出实现计划（6 个文件、依赖关系、生成顺序）。第二步，逐个生成文件并在生成过程中自动交叉引用——生成 vector_store.py 时引用 embedding.py 中定义的 get_embeddings 函数签名、生成 rag_pipeline.py 时引用 vector_store.py 中定义的 VectorStore 类接口。第三步，生成测试文件前先读取路由和服务代码，确保 Mock 的接口签名与真实代码一致。

Agent 工作流的一个关键特性是类型跟踪：AI 在整个生成过程中维护一个虚拟的类型表，记录每个文件中定义的类、函数和变量的签名信息。生成后续文件时查询这个类型表确保引用的一致性。实验表明 Agent 模式的多文件一致性比单次 Prompt 高 40%（因为单次 Prompt 中 AI 会根据上下文窗口裁剪忽略部分文件内容）。

评测脚本自动编写与检索质量评估

AI 自动编写评测脚本（eval 套件）评估 RAG 管线的检索质量和生成质量。评测脚本包含三项核心指标：命中率（Hit Rate）——检索结果中是否包含预期文档，衡量检索是否找到了正确的信源；精确率（Precision@K）——前 K 个结果中相关文档的比例，衡量检索结果的精确度；响应时间（P50/P95/P99 Latency）——端到端延迟分布。AI 从 Spec 文档中读取评测测试集（10 组 query + expected_source 对），自动编写评测脚本并在 CI 中运行。

python 
# eval_rag.py — AI 生成的 RAG 评测脚本
import json, time
from app.services.rag_pipeline import RAGPipeline

TEST_CASES = [
    {"query": "公司年假政策是什么？",
     "expected_sources": ["policy-001", "policy-002"],
     "expected_relevant": True},
    {"query": "如何申请远程办公？",
     "expected_sources": ["policy-005"],
     "expected_relevant": True},
    {"query": "不存在的无关内容",
     "expected_sources": [],
     "expected_relevant": False},
]

async def evaluate_rag(pipeline: RAGPipeline):
    results = []
    for case in TEST_CASES:
        start = time.time()
        response = await pipeline.query(
            case["query"], top_k=5
        )
        latency = (time.time() - start) * 1000
        hit = any(
            s in response["sources"]
            for s in case["expected_sources"]
        )
        results.append({
            "query": case["query"],
            "hit": hit,
            "precision": len(
                set(response["sources"]) &
                set(case["expected_sources"])
            ) / max(len(response["sources"]), 1),
            "latency_ms": latency
        })
    return results

CorpAssist 场景

评测脚本在 CI 中每周运行一次，跟踪检索质量趋势。初始评测结果：命中率 80%（TopK=5 时 8/10 查询命中预期来源）、精确率 85%、P50 延迟 800ms、P99 延迟 3.2s。经过 AI 辅助优化（索引优化、缓存策略、异步处理）后，命中率提升到 92%、精确率 93%、P99 延迟降到 1.5s。评测数据自动生成可视化报告展示在团队的 Grafana 仪表盘上。

# AI 辅助开发效率 ROI 计算
def calculate_roi(team_size: int, monthly_salary: float):
    """计算 AI 辅助开发的年度 ROI"""
    # 假设：AI 辅助提升 35% 效率
    efficiency_gain = 0.35
    annual_salary = monthly_salary * 12 * team_size
    annual_savings = annual_salary * efficiency_gain

    # AI 工具成本
    # Cursor Business $40/人/月
    # Claude Code API $200/月 (按用量)
    tool_cost = (40 * team_size * 12) + (200 * 12)

    # 培训成本 (一次性)
    training_cost = team_size * 2000  # $2,000/人培训费

    # ROI 计算
    net_gain = annual_savings - tool_cost - training_cost

    return {

        "tool_cost": tool_cost,
        "training_cost": training_cost,
        "net_gain": net_gain,
        "roi": f"{roi:.1f}x"

常见问题（FAQ）

AI 辅助开发总时间是 4 小时，但 Spec 编写花了 30 分钟——为什么不直接让 AI 写代码？

因为 Spec 编写的时间投入在后续阶段获得数倍回报。实验数据表明，在 Spec 阶段投入 30 分钟（占总时间 12%），可以使代码生成阶段的返工率降低 60%。如果跳过 Spec 直接生成代码，AI 会遗漏关键需求——35% 的缺陷来自需求理解错误而不是代码实现错误。Spec 的投入产出比约为 1:12——30 分钟的 Spec 编写节省了后续 6 小时的调试和修复时间。

Agent 工作流生成 6 个文件时类型一致性如何保证？

AI 在 Agent 模式中维护一个虚拟类型表：开始生成前，AI 读取项目中已有的类型定义建立初始类型表。生成过程中，每个新文件中定义的公共类、函数和变量的签名被加入类型表。生成后续文件时，AI 查询类型表确保使用的类型名称和签名与实际定义一致。如果引用了一个不存在的类型，AI 会自动暂停并修正。这是 Agent 模式的核心优势——跨文件的一致性由工作流的"状态维护"机制保证，而非单次 Prompt 的"上下文猜测"。

AI 辅助开发效率提升 83% 的数据是否可复制？适用于所有项目类型吗？

83% 的数据来自 CorpAssist RAG 管线这个特定场景——典型的"中等复杂度、模板代码占比高"的项目类型。效率提升在不同项目类型中有差异：CRUD 模块效率提升最显著（可达 85-90%）、数据处理和 ETL 脚本次之（60-70%）、核心算法和性能优化模块提升最小（20-30%）——因为核心算法需要深入理解业务领域和算法原理。关键因素不是项目类型而是开发流程的"AI 就绪度"——有清晰的 Spec、好的 Rules 文件和稳定的技术栈时效率提升最显著。

最后提醒一个容易被忽视的要点：AI 辅助开发不是零成本——每次 AI 调用消耗 API Token（成本）和开发者的审查时间（隐性成本）。在 CorpAssist 项目中，AI 辅助开发的 API 成本加上审查时间约为纯手动开发成本的 30%（按人时折算）。但考虑到效率提升 83% 和缺陷率降低 60%，综合 ROI 仍然非常可观。理解并量化这些成本，可以帮助团队在 AI 使用和手动开发之间做出明智的分配决策。

面试可能问

AI 辅助开发的效率提升 83% 是否夸大？什么样的项目能复现这个数字？

这个数据来自 CorpAssist 团队在特定场景（RAG 管线 CRUD 模块）的对照实验——同一开发者用两种方式实现同一功能各执行三次取平均值。能复现这个数字的项目特征：技术栈稳定（Python + FastAPI + 常见框架）、需求清晰可用 Spec 描述、代码中包含大量模板代码（路由、模型、配置）、测试和文档是整体开发的一部分。不适用的情况：探索性开发（需求不明确）、核心算法研发、性能优化和调试——这些场景的 AI 辅助效率提升约 20-30%。

Agent 工作流相比单次 Prompt 的核心优势是什么？最关键的实现机制是哪个？

核心优势是状态管理——Agent 在一次工作流中维护跨步骤的状态（类型表、依赖图、已完成文件列表）。最关键的实现机制是虚拟类型表：Agent 开始工作前读取项目现有类型签名，生成过程中每个新文件中的公共符号被加入类型表，后续文件生成时查询类型表保证交叉引用一致性。相比单次 Prompt 的"一次猜测所有类型签名"（上下文窗口限制导致部分类型被忽略），Agent 的"逐步构建类型表"保证了更高的引用准确率（91% vs 65%）。

团队推广 AI 辅助开发最大的阻力通常是什么？如何克服？

最大阻力不是技术而是习惯——工程师习惯了"打开 IDE 直接写"的工作方式。Spec-driven 要求先写文档再做开发，这看起来是"额外工作"。克服方法：1) 从低风险 CRUD 模块开始试点让团队先看到效果不强制全量执行；2) 提供 Spec 模板降低编写成本（30 分钟内完成）；3) 用量化数据说服——Spec 每 1 分钟投入在代码生成阶段节省 12 分钟返工；4) 让早期使用者成为内部 Champion 带动其他人。

如何衡量 AI 辅助开发的投入产出——除了开发时间外还应该跟踪哪些指标？

推荐跟踪以下六大指标：1) 首次通过率——AI 生成的代码无需修改即可运行的比例，反映 Prompt 质量和 Spec 完整性；2) 缺陷密度——每千行 AI 代码中的缺陷数，反映代码质量；3) AI 代码比例——各模块 AI 生成代码占比，反映 AI 使用深度；4) 审查轮次——从 PR 创建到合并的平均轮次，反映协作效率；5) 开发到上线时间——从需求确认到上线的时间窗口，反映整体交付速度；6) 开发者满意度——月度 NPS 评分，反映工具的可接受度。前五个是定量指标，第六个是定性指标，组合使用全面衡量 AI 辅助开发的 ROI。

判断练习

场景：AI 生成的 RAG 管线上线后命中率只有 60%。如何系统性地诊断和优化？

三步诊断法：1) 检查 Chunk 策略——256 tokens 是否过小导致语义被截断？尝试增加到 512 tokens；2) 检查 Embedding 模型——text-embedding-3-small 的 1536 维是否足够区分公司政策文档的细粒度语义？尝试升级到 text-embedding-3-large（3072 维）；3) 检查相似度阈值——0.7 是否过高导致过滤了相关结果？尝试降到 0.65 并观察命中率和精确率的 trade-off。AI 可以辅助分析每个步骤的优化效果并生成对比报告。

场景：团队不同意先写 Spec 再开发的流程，认为"浪费时间"。如何用数据说服？

用对比实验数据说话：A/B 测试两个组——A 组使用 Spec-driven、B 组使用随意的 Prompt 方式，开发同一个 CRUD 功能。记录两组的总开发时间、迭代次数和缺陷数。实验数据通常显示 Spec-driven 组的总时间更短（因为返工少）。额外强调：Spec 不是"额外负担"而是"第一次就把事情做对"——30 分钟的 Spec 编写约等于 6 小时的调试时间。

场景：RAG 管线的 P99 延迟从 1.5s 飙升到 5s。AI 如何辅助排查？

提交延迟飙升前后的代码变更 diff + 当前 py-spy 性能采样数据给 AI。AI 通过差异分析法找出导致性能下降的具体代码提交。同时 AI 分析性能采样数据中的热点函数——常见原因：Embedding 模型调用变成串行、向量数据库批量查询退化、Chunk 策略变更导致返回结果过多。AI 给出每个热点的修复建议后，开发者选择最有效的方案实施。
