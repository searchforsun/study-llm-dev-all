为什么要学本章

在 AI 辅助编码工具对比 中选定工具后，若仍用「想到哪问到哪」的 Chat，生成代码往往与 OpenAPI 契约、测试习惯不一致。Prompt 驱动开发 把需求、设计、生成、审查串成可重复流水线，是 CorpAssist Python 团队降返工的关键。

学完你能
  • 画出 需求→设计→生成→审查 闭环，并标注每步输入/产出
  • 为 RAG 模块组织 Context 工程：该 @ 哪些文件、该写进 Rules 什么
  • 编写可维护的 .cursor/rules 与 CLAUDE.md 片段
本章先记住 3 件事
  • Prompt 是接口：稳定 Prompt 模板 = 稳定生成质量，而非灵感文案。
  • Context 有预算：只喂相关 schema、测试与 Rules，避免整仓 dump。
  • 审查不可省：生成后必须过 diff + pytest + 人审（见 AI 生成代码的质量管控）。

需求→设计→生成→审查闭环

CorpAssist 典型需求：「检索接口返回引用片段列表」。若直接让 AI「帮我改 retriever」，容易漏改 OpenAPI 与测试。闭环把每步产物结构化，便于 Agent 与人类接力。

CorpAssist · Ticket CA-1428（1 分钟） 产品 PRD 要求 QueryResponse 增加 citations: list[Citation]。后端负责人先输出设计要点（字段、错误码、兼容策略），再交给 AI 改 4 个文件，最后 Reviewer 对照 checklist 合并。
Prompt 驱动开发闭环
flowchart LR
  R[需求 / Ticket] --> D[设计要点 半页]
  D --> C[Context 包: schema+tests+rules]
  C --> G[Agent 生成 patch]
  G --> V[审查: diff+pytest+人审]
  V -->|返工| C
  V --> M[Merge]

设计要点是人与 AI 的契约；没有设计要点就生成，返工率显著上升。

先想 10 秒：为什么「设计要点半页」比完整设计文档更适合 AI 生成？

半页约束边界（字段、错误码、兼容）又留出实现空间；长文设计易超 Context 或互相矛盾。AI 负责填实现细节，人负责定契约。

Context 工程 for AI Coding

Context 工程 决定 Agent「看见什么」。CorpAssist 仓库模块多，错误 Context 会导致 AI 引用过时 DTO 或错用 sync HTTP 阻塞 async 路由。

Context 三层模型

L1 · 持久 Rules

.cursor/rules、CLAUDE.md：编码规范、禁止项、测试命令。每次会话自动注入。

L2 · 任务 Context 包

本 Ticket 相关：schema、router、2–3 个测试、OpenAPI 片段。用 @file 或显式路径列表。

L3 · 动态检索

Agent 按设计要点自行 grep / 读文件；须在 Rules 里限制目录（如勿读 config/prod/）。

CorpAssist Context 包示例（YAML 清单）

yaml 
# context-pack-ca1428.yaml — 随 Ticket 提交，供 Agent 读取
ticket: CA-1428
goal: QueryResponse 增加 citations 字段
must_read:
  - src/corpassist/schemas/query.py
  - src/corpassist/api/routes/rag.py
  - tests/api/test_rag_query.py
  - docs/openapi/paths/rag_query.yaml
constraints:
  - 保持 snake_case 与现有 ErrorResponse 格式
  - 新增字段 optional，默认 []
commands:
  verify: uv run pytest tests/api/test_rag_query.py -q

Context 预算有限时，优先契约与测试，而非把整个 corpassist/rag/ 历史提交记录塞入 Prompt。

flowchart TD

  P --> T[schema + tests]

.cursorrules/claude.md 最佳实践

项目级 Rules 是 Prompt 驱动开发 的「标准库」。CorpAssist Python 服务应把与 Java 网关对齐的约定写进 .cursor/rules 或根目录 CLAUDE.md。

Rules 应写什么 / 不应写什么

应写不应写
测试命令、目录结构、命名约定整份 PRD 粘贴（放 Ticket 链接）
禁止：密钥入库、直连生产 LLM 做测试频繁变的临时分支策略
OpenAPI 字段与 Java DTO 对齐说明与代码相矛盾的过时规范

推荐：.cursor/rules/corpassist-python.mdc 片段

markdown 
---
description: CorpAssist Python RAG/API 开发约定
globs: src/corpassist/**/*.py, tests/**/*.py
---
- 使用 FastAPI + Pydantic v2；路由 async，IO 用 httpx AsyncClient
- 错误体：`{"error": {"code": "...", "message": "...", "request_id": "..."}}`
- 改 API 必须同步 tests/ 与 docs/openapi/
- 验证：`uv run pytest tests/ -q --tb=short`；禁止跳过 mypy/ruff
- 勿读取 .env、config/prod-*；勿在代码中硬编码 API Key

请写好代码，要注意性能和安全，多用 AI，随便改架构...

与 CorpAssist 开发的结合

将闭环落到 CorpAssist 日常：每个 RAG/Agent Ticket 附带 Context 包；MR 描述粘贴设计要点；CI 跑 pytest。与 规则与约定驱动开发 衔接，Rules 成为唯一风格真源。

生成 Prompt 模板（可复制）

markdown 
## Ticket CA-1428
**验收**：QueryResponse 含 citations；空结果时 citations=[]

**设计要点**：
- 新增 Citation {doc_id, chunk_id, snippet}
- 不破坏现有 clients（字段 optional）

**请先**：列出将修改的文件路径
**再**：实现并运行 `uv run pytest tests/api/test_rag_query.py -q`
**禁止**：修改 config/prod；不要新增加依赖

Agent 完成后，审查者对照设计要点与 diff，而非「感觉对不对」。下一章将展开质量门禁。

CorpAssist RAG 开发：闭环实例时间线

时间动作产出
T+0产品确认 Ticket 验收 3 条CA-1428 描述
T+30m负责人写设计要点 + Context YAML半页设计 + must_read 列表
T+1hCursor Agent 按 Prompt 改 4 文件本地 diff
T+1.5huv run pytest + 人审 citations 字段MR 就绪
设计要点与 OpenAPI PR 谁先谁后？
契约字段变更应OpenAPI 先 diff（见 规则与约定驱动开发）；设计要点引用 OpenAPI 版本号，Agent 才不易臆造字段。
Claude Code 没有 @file 怎么办？
在 Prompt 中粘贴 Context YAML 的 must_read 列表，并要求 Agent 先 Read 这些路径再改；与 Cursor @file 等价。
常见误区
  • Context 越多越好 — 噪声淹没契约，模型反而忽略测试。
  • Rules 与代码脱节 — 改了 pytest 命令却不更新 CLAUDE.md。
  • 跳过设计要点 — 把 PRD 全文丢给 AI，字段命名每次不同。
  • 无验证命令 — Prompt 未要求跑 pytest，Agent 不会自检。

与 RAG 开发工作流 的衔接

CorpAssist RAG 改动常跨 ingestion、retriever、api 三处。Prompt 闭环中应在设计要点里标明数据流阶段（写入 / 检索 / 响应组装），避免 Agent 只改 API 却未更新索引 schema。

  • 写入阶段 Context：chunker.py + 样本 JSONL
  • 检索阶段 Context：retriever.py + 向量库 mock fixture
  • 响应阶段 Context：schemas/query.py + OpenAPI

本章结论

  • 闭环四步：需求验收 → 设计要点 → Context 包生成 → diff/pytest/人审；缺设计要点返工率高。
  • Context 工程：L1 Rules + L2 任务包 + L3 受控检索；优先 schema 与测试。
  • Rules 最佳实践：短、可验证、带 globs；CLAUDE.md 做索引而非百科全书。
  • CorpAssist 结合：Ticket 绑 Context 包与验证命令；与团队 Rules 和 Review 流程一体。

动手练习

为虚构 Ticket CA-1500（top_k 参数校验）编写设计要点、Context 包 YAML与Agent Prompt（约 30 分钟）。

操作步骤

  1. 写 3 条验收标准（含非法 top_k 返回 422）。
  2. 写半页设计要点：字段范围 1–50、错误码与 request_id 格式。
  3. 创建 context-pack-ca1500.yaml，列出 must_read 至少 4 个路径。
  4. 从本章模板改一版 Prompt，要求 Agent 先列文件清单再改。
  5. 与同伴互相 Review Context 包：是否漏 OpenAPI 或误含 prod 配置路径。

判断练习

  1. 同事把 800 行架构 PPT 全文贴进 Prompt，导致模型忽略测试文件。这违反 Context 工程的哪条原则？

    参考答案

    Context 有预算、契约优先：应喂 schema/测试与短设计要点，而非长文噪声。
