词向量与序列任务回顾

CorpAssist · 工单分类 员工提交「报销单被驳回，提示发票抬头不符」——系统需把这句话映射为向量，再预测应路由到「财务合规」还是「IT 报障」。这是典型的序列分类：整句输入，单个标签输出。

自然语言处理的核心难题之一是：词是离散符号，神经网络需要连续数值。词向量（Word Embedding） 把词典中每个词映射到 ℝd 空间中的一个点，语义相近的词（如「补贴」「津贴」）在该空间中距离更近。CorpAssist 的检索与分类链路都建立在「文本 → 向量」这一步之上。

常见序列任务类型

任务输入 → 输出CorpAssist 示例
语言建模 前文 → 下一词概率 对话续写、邮件草稿补全
序列分类 整句 → 类别 工单意图识别、敏感词检测
序列标注 每个词 → 标签 NER 抽取合同甲乙方、金额字段
序列到序列 源序列 → 目标序列 会议纪要摘要、多语言制度翻译

从 one-hot 到稠密嵌入

早期用 one-hot 表示词：维度 = 词表大小 V，几乎正交，无法表达语义相似性。Word2Vec、GloVe 等学习到的 稠密嵌入 把维度压到 256～1024，既省空间又保留「国王−男人+女人≈女王」这类类比关系。现代大模型不再单独预训练 Word2Vec，但嵌入层（Embedding Layer） 仍承担相同角色：查表得到每个 Token 的初始向量。

嵌入查表示例（NumPy 直觉）

python 
import numpy as np

vocab_size, d_model = 50000, 512
# 嵌入矩阵 E: (V, d)，第 i 行是词 i 的向量
E = np.random.randn(vocab_size, d_model) * 0.02

# CorpAssist 工单分词后的 token id 序列
token_ids = [1042, 8831, 220, 9910]  # 「报销」「驳回」「发票」「抬头」
# 查表得到序列矩阵 X: (seq_len, d_model)
X = E[token_ids]
print(X.shape)  # (4, 512)

得到 X 之后，序列建模 模块要在时间维上传播或混合信息——这正是 RNN 与后来 Transformer 的分野所在。

CorpAssist 嵌入层在双栈中的位置

  • Python 推理服务：Hugging Face AutoModel 首层 embed_tokens 即查表；RAG 侧 bge/E5 同理。
  • Java Spring AI：调用远程 Embedding API 或 ONNX 本地模型，输出 float[] 与 Python 维数须契约一致。
  • 网关计量：Token 数在嵌入之前由分词器确定，与 d_model 无关——估费时勿混淆。

嵌入只解决「每个词是什么向量」；序列建模 解决「这些向量如何随上下文组合成句义」——下一节 RNN 是第一种答案，Transformer 是现行答案。

无序列建模
  • 仅对词向量做平均池化
  • 丢失词序：「不报销」与「报销不」相同
  • 无法建模长距离修饰关系
有序列建模
  • 保留位置与上下文依赖
  • 可区分否定、条件从句等语序敏感语义
  • 支撑 CorpAssist 多轮对话与长文档理解

RNN/LSTM 的并行与长依赖瓶颈

循环神经网络（RNN） 按时间步 t=1…n 依次处理序列：每步读取当前词向量 xt 与上一步隐状态 ht−1，输出 ht。公式可简写为 ht = f(W·xt + U·ht−1 + b)。这种串行依赖在 2017 年前是序列建模主流，但有两处硬伤。

瓶颈一：训练难以并行

计算 h5 必须先算完 h4，GPU 数千核心无法同时展开整条序列——批内只能并行不同样本，单条序列内部仍是串行。CorpAssist 若用纯 RNN 训练内部分类器，长工单序列的训练吞吐会明显低于同等参数规模的 Transformer。

瓶颈二：长距离梯度与信息衰减

句子开头的主语要约束句尾动词的一致性，或制度条文第 1 条与第 47 条的「除外条款」关系，都需要跨越数十到数百 Token 的 长程依赖。标准 RNN 反向传播时梯度连乘，易出现梯度消失/爆炸；LSTM 用门控机制缓解，但路径仍须经过 O(n) 步隐状态传递，远距离信号会被稀释。

RNN 串行 vs Transformer 并行（概念对比）
flowchart TB
  subgraph RNN["RNN：时刻串行"]
    x1[x1] --> h1[h1]
    h1 --> h2[h2]
    x2[x2] --> h2
    h2 --> h3[h3]
    x3[x3] --> h3
  end
  subgraph TR["Transformer：层内并行"]
    X[X: 整段嵌入] --> SA[Self-Attention 一次看全序列]
    SA --> H[H: 所有位置同时更新]
  end

RNN 的 ht 链式传递限制并行；Self-Attention 在单层内让任意位置直接交互（详见 Self-Attention 矩阵视角）。

LSTM 门控直觉（不必背公式）

  • 遗忘门：决定从细胞状态丢弃多少旧信息。
  • 输入门：决定写入多少当前输入。
  • 输出门：决定暴露多少状态给下一步。

LSTM 在 2014–2016 年的机器翻译、语音识别中表现优异，但无法从根本上消除序列内串行。当 CorpAssist 需要处理 8K～128K Token 的制度汇编时，纯 LSTM 架构在训练与推理效率上均已退出主流 LLM 选型。

面试口述：RNN 两大瓶颈（30 秒版）

  1. 并行：ht 依赖 ht−1，单序列内无法像 Self-Attention 那样一层算完全部位置。
  2. 长依赖：信息须沿时间步传递 O(n) 次，梯度与信号均易衰减；LSTM 门控是补丁非根治。

向 Java 后端同事解释时可用类比：RNN 像单线程 for 循环处理数组；Transformer 像对数组做一次矩阵批量变换（在层内）。

先想 10 秒：为何「更大 LSTM、更多层」仍难追上 GPT 类模型的训练速度？

层数加深只改善表达能力，不消除时间维串行：每个时间步仍须等待上一步。Transformer 在序列长度维上可并行计算 Attention 矩阵（代价是 O(n²) 显存，见 复杂度章节），因而更适合现代 GPU/TPU 数据并行。

Encoder-Decoder 与 Seq2Seq

机器翻译推动了 序列到序列（Seq2Seq） 范式：用编码器把源语言句子压成上下文向量，再用解码器逐词生成目标语言。CorpAssist 的「中文制度 → 英文摘要」「工单详情 → 结构化 JSON」在抽象上都属于 Seq2Seq。

经典 Encoder-Decoder 数据流

CorpAssist 会议纪要摘要 · Seq2Seq 骨架
flowchart LR
  SRC[源序列: 会议逐字稿 Token] --> ENC[Encoder RNN/LSTM]
  ENC --> CTX[上下文向量 c]
  CTX --> DEC[Decoder RNN/LSTM]
  DEC --> TGT[目标序列: 摘要 Token 逐个生成]

早期 Seq2Seq 把整段源文「压」进单个向量 c，长会议时 c 成为信息瓶颈。

单向量瓶颈与注意力萌芽

当源序列很长（如 2 小时会议纪要 8000 字），单个 c 无法承载全部细节，摘要容易漏掉「预算审批结论」等关键句。2015 年 Bahdanau 等人引入 Attention：解码每一步不再只读 c，而是对编码器所有时间步输出做加权求和——相当于「生成摘要每一句时，回头看转录稿里最相关的段落」。这是通往 Transformer 的关键一步。

Seq2Seq 训练目标（应用岗口述版）

text 
给定源序列 x = (x1, x2, ..., xm)
目标序列 y = (y1, y2, ..., yk)
训练最大化 P(y | x) = ∏ P(yt | y<t, x)

CorpAssist 摘要场景：
  x = 会议转录 Token 序列
  y = 摘要 Token 序列
  每步预测「下一个摘要词」，条件于已生成摘要 + 源文编码

长源文信息挤进单向量 c，CorpAssist 长纪要摘要易漏关键决议；解码器对所有编码步「一视同仁」。

解码每步动态对齐源文不同位置，相当于可微分的「软检索」；为 Q/K/V 注意力 与全 Transformer 编解码铺平道路。

为何需要全局注意力

到 2017 年，序列建模的矛盾已经很清晰：需要长距离、任意位置间的灵活依赖，同时需要 GPU 友好的并行。RNN 家族给不了后者；带 Attention 的 Seq2Seq 改善了信息通路，但编码器仍是串行 RNN。Vaswani 等人在《Attention Is All You Need》中提出：干脆去掉 RNN，只用 Attention 堆叠——即 Transformer。

本章先记住 4 件事
  • Q/K/V 都是学出来的投影：同一嵌入 X 乘以不同权重矩阵 WQ, WK, WV。
  • 相似度 = Q 与 K 的点积：越大表示「更相关」，再经 softmax 变概率分布。
  • 输出 = 权重的 V 加权和：相当于按相关性从「内容库」里取料。
  • 除以 √dk 防梯度爆炸：维度高时点积方差大，softmax 易极端化。

查询-键-值直觉

CorpAssist · 制度问答 用户问「驻外补贴能否与差旅叠加？」模型在组织每个新 Token 的答案时，需要决定：上下文里哪几句制度更值得参考。Attention 把这一过程变成可微分的矩阵运算。

注意力（Attention） 的核心抽象来自键值存储（Key-Value Store）：

  • Query（查询）：当前位置「想问什么」——例如正在生成「叠加」一词时，对上下文的提问向量。
  • Key（键）：每个上下文位置贴的「索引标签」——用于与 Query 算相似度。
  • Value（值）：每个位置实际携带的「内容」——相似度高则多取一些。

图书馆类比

Attention ≈ 软检索
  • Q = 你手里的问题卡片
  • K = 书架上每本书的标题/标签
  • 相似度 = Q·KT（点积）
  • V = 书的内容片段；输出 = 按相似度加权拼读
硬检索（RAG 工程层）
  • Query = 用户问题 Embedding
  • Key = 向量库文档 Embedding
  • Top-K 硬选 K 条 chunk
  • 无梯度、在模型外部完成

CorpAssist 的 RAG 在模型外做硬 Top-K；Transformer 内部每层都在做软 Attention——对所有 Token 赋予连续权重，且端到端可训练。

三矩阵形状速记（面试白板）

  • 输入 X：(n, d_model)
  • Q、K：(n, d_k)；V：(n, d_v)
  • 权重 A：(n, n)，每行和 = 1
  • 输出 O：(n, d_v)，常令 d_v = d_model

若面试官追问「Q 和 K 能否相同」：矩阵可相同但语义角色不同——Q 问「我要什么」，K 答「我有什么标签」；权重由二者交互产生，不是简单复制 X。

下列三点是阅读后续公式的锚点：① Q/K/V 同源不同投影；② 相似度 = 点积；③ 输出 = Value 的凸组合。

从嵌入到 Q/K/V 的线性投影

python 
# X: (seq_len, d_model) — 嵌入 + 位置编码后的序列
Q = X @ W_q   # (seq_len, d_k)
K = X @ W_k   # (seq_len, d_k)
V = X @ W_v   # (seq_len, d_v)
# W_q, W_k, W_v 为可学习参数；d_k 常取 d_model / num_heads

同一 Token 向量 X 经不同矩阵「扮演」不同角色——这是 Q/K/V 与 RAG 里「问题向量 / 文档向量」共用同一 Embedding 模型但用途不同的对照点。

Scaled Dot-Product 公式

原论文给出的 缩放点积注意力 是 Transformer 的默认注意力核：

核心公式

Attention(Q, K, V) = softmax(QKT / √dk) V

矩阵形状（NumPy 视角）

text 
Attention(Q, K, V) = softmax( Q @ K.T / sqrt(d_k) ) @ V

形状：
  Q: (n, d_k)   — n 个查询，每个 d_k 维
  K: (n, d_k)   — n 个键
  V: (n, d_v)   — n 个值
  Q @ K^T: (n, n) — 每对位置的相似度
  softmax 沿最后一维（对每个 Query 看所有 Key）
  输出: (n, d_v)

分步拆解

  1. 打分：S = QKT，Sij 表示第 i 个 Query 对第 j 个 Key 的相关性。
  2. 缩放：S' = S / √dk。当 dk=64 时除以 8，防止点积过大使 softmax 饱和。
  3. 归一化：A = softmax(S')，每行和为 1，得到注意力权重矩阵。
  4. 聚合：输出 = AV，按权重对 Value 行向量加权求和。
Scaled Dot-Product 数据流
flowchart LR
  X[嵌入 X] --> WQ[W_q]
  X --> WK[W_k]
  X --> WV[W_v]
  WQ --> Q[Query]
  WK --> K[Key]
  WV --> V[Value]
  Q --> DOT["QKᵀ / √d_k"]
  K --> DOT
  DOT --> SM["softmax → A"]
  SM --> OUT["A·V → 输出"]
  V --> OUT

Self-Attention 时 Q、K、V 同源序列 X；Cross-Attention 时 Q 来自解码器、K/V 来自编码器（见 多头与 Mask）。

NumPy 最小实现

python 
import numpy as np

def scaled_dot_product_attention(Q, K, V):
    d_k = Q.shape[-1]
    scores = Q @ K.T / np.sqrt(d_k)          # (n, n)
    weights = softmax(scores, axis=-1)       # 每行和为 1
    return weights @ V, weights

def softmax(x, axis=-1):
    x = x - np.max(x, axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)
先想 10 秒：若去掉 sqrt(d_k) 缩放，训练初期可能出现什么现象？

点积方差随 dk 增大而增大，softmax 输出接近 one-hot，梯度几乎只流向个别位置，注意力过于尖锐、难以学习多样对齐。缩放把分数拉回合理量级，使梯度更稳定。

Encoder/Decoder 栈结构

把 Transformer 块 理解成「可堆叠的乐高单元」：同一 Block 重复 N 次，仅参数不共享（每层独立 WQ…）。CorpAssist 调用 32 层 7B 模型时，一次 forward 等于把长度为 n 的序列依次穿过 32 个 Attn+FFN 组合。

原始 Transformer 为机器翻译设计：N 层 Encoder 处理源序列，N 层 Decoder 自回归生成目标序列。现代 LLM 多简化为单一栈：

选型时先问「要生成还是要向量」：生成走 Decoder，向量走 Encoder；CorpAssist 两条链路常并存，不要混用同一 checkpoint 承担两种角色。

残差连接与 LayerNorm

深层网络训练就像「信号穿过很多层放大器」：残差 保证至少有一条不经过非线性的直通路径；LayerNorm 把每层输出拉回稳定数值区间，避免某一 Token 的激活爆炸拖垮整批训练。

深层网络训练难点是梯度消失与表示退化。残差连接（Residual） 让子层学习「增量」Δ而非全新映射：output = x + SubLayer(x)。Layer Normalization 对每个 Token 的特征维做归一化，稳定激活分布。

Post-LN vs Pre-LN

Post-LN（原论文）
  • x → SubLayer → LayerNorm → 残差
  • 深模型训练需小心 warmup
  • 早期 BERT 采用
Pre-LN（GPT/LLaMA 主流）
  • x → LayerNorm → SubLayer → 残差加回 x
  • 深层栈更易训练
  • CorpAssist 调用的开源权重多为 Pre-LN

LayerNorm 公式直觉

python 
# 对单个 token 的 d_model 维向量 x 做 LN
mean = x.mean()
var = x.var()
x_hat = (x - mean) / sqrt(var + eps)
out = gamma * x_hat + beta   # gamma, beta 可学习

与 BatchNorm（按 batch 维）不同，LayerNorm 对序列长度维不变，适合变长工单与对话——这也是 NLP Transformer 普遍选 LN 的原因。后续 LLaMA 对照 会提到 RMSNorm 变体。

BatchNorm 在 batch=1 的在线推理时不稳定；LayerNorm 只依赖单 Token 向量统计，batch 大小不影响归一化语义——CorpAssist 低并发与压测高并发行为一致。

残差路径的梯度直觉

反传时损失 ∂L/∂x 除穿过子层 ∂SubLayer/∂x 外，还有恒等项 +1 直达更浅层。深层 Transformer（N=80+）若无残差，底层嵌入层几乎收不到有效梯度，无法学到可用表示。

先想 10 秒：若没有残差，100 层 Transformer 训练会怎样？

纯堆叠非线性变换时梯度须穿过所有层连乘，极易消失；残差提供恒等捷径，使底层表示仍可直达顶层，深层栈才可训练。CorpAssist 72B 级模型若无残差，工程上无法收敛到可用 checkpoint。

位置编码必要性

词嵌入只编码「是什么词」，不编码「在第几位」。位置编码 补上坐标；否则 Self-Attention 对输入集合是置换等变的——打乱 Token 顺序，若嵌入向量集合相同，则注意力输出可完全一致。

Self-Attention 对输入集合是置换等变的：打乱 Token 顺序但保留各向量，注意力模式可重组为相同——模型原生不知先后。语言中语序承载否定、条件、修饰关系，必须显式注入位置信息。

绝对正弦位置编码（原论文）

python 
# PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
# PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
# 输入 = TokenEmbedding + PositionalEncoding

位置编码 与词嵌入相加后送入第一层。不同频率的正弦波让模型学习相对距离模式。现代 LLM 多用可学习绝对嵌入或 RoPE，但「必须注入位置」这一点不变。

无位置编码的反例

  • 「不报销」与「报销不」嵌入集合相同 → Self-Attention 输出可完全一致 → 语义灾难。
  • CorpAssist 工单「驳回后再提交」vs「提交后再驳回」流程相反，缺位置则模型无法区分。
  • 条件句「若 A 则 B」中 A、B 的先后顺序改变政策含义，位置信息不可或缺。
  • 多轮对话里 user/assistant 轮换顺序依赖位置下标，而非仅靠 role 字符串。

正弦 PE 的频率直觉（不必背公式）

低频维度编码「大致位置区间」，高频维度编码「精细偏移」——类似用不同刻度的尺子量距离。位置编码 与词嵌入相加后，Self-Attention 的 Q/K 点积会隐含相对距离信息（原论文性质）；现代 RoPE 把这一思想做得更干净。

FFN 子层作用

Attention 负责Token 间混合；前馈网络（FFN） 负责每个 Token 位置独立做非线性变换，常写作两层 MLP：

标准 FFN 公式

FFN(x) = max(0, xW1 + b1) W2 + b2

ReLU 版（原论文）；W1: (dmodel, dff)，W2: (dff, dmodel)，常 dff = 4×dmodel。LLaMA/Qwen 等改用 SwiGLU（见 practice 章），但「先扩维再压回」结构不变。

  • 逐 Token：FFN 不跨位置混合，每个 Token 向量独立过同一组 MLP 权重。
  • 参数量：FFN 约占 Transformer Block 参数 2/3，Attention 约占 1/3。
  • CorpAssist：MoE 变体把 FFN 换成多专家路由（见 advanced 章），Attention 子层结构不变。
  • 激活：ReLU 将负值截零；SwiGLU 用门控乘法替代，实践中更稳。

分工对照

子层跨 Token主要作用参数量占比
Self-Attention 是（n×n 混合） 上下文依赖、动态对齐 中等
FFN 否（逐位置） 特征提炼、存储事实模式 Often ~2/3

面试常问「Attention 既然能全局看，还要 FFN 干嘛？」——答：Attention 是路由与混合，FFN 是逐位置的可学习「小 MLP 记忆」；二者互补。CorpAssist 模型越大，FFN 扩维倍数往往越高，推理显存相当一部分在 FFN 激活上。

参数量粗算（单层 FFN）

text 
W1: d_model × d_ff  → 4096 × 16384 ≈ 67M
W2: d_ff × d_model  → 16384 × 4096 ≈ 67M
单层 FFN 权重约 134M；Attention 权重同量级

懂 FFN 占参比后，选型面试可解释：「7B 并非 70 亿个 Attention 参数，FFN 与嵌入/LM head 占相当份额。」

与 practice 章预告

开源 LLM 常把 ReLU FFN 换成 SwiGLU、LayerNorm 换成 RMSNorm——块拓扑不变（仍是 Attn→FFN+残差），变的是子层内部激活与归一化实现（见 架构对照）。

CorpAssist 升级模型版本时，优先核对 hidden_size、num_hidden_layers 是否变化——接口不变但算力画像可能完全不同。

子层顺序为何固定为 Attn → FFN

先 Self-Attention 让 Token 间交换信息，再 FFN 在各位置独立提炼特征——若对调，FFN 在「尚未混合上下文」的表示上运算，再 Attention 混合，表达路径不同且与原论文/预训练权重不兼容。开源权重按 Attn→FFN 训练，推理实现不可擅自调换。

CorpAssist 一层 forward 的 Java 视角（口述）

  • 网关组装 messages → Tokenizer → token_ids[]
  • 推理服务 embedding + 位置 → 重复 N 次 Block
  • 每层 Block：LN → Attn(+Mask) → 残差 → LN → FFN → 残差
  • 末层 LM head → logits → 采样下一 Token
  • 你不需要实现 Block，但需知延迟与 n、N、d_model 线性/平方关系
  • 监控项：prompt_tokens（n）、model_layers（N）、time_to_first_token
一层 Transformer Block 在推理时算几次？

每个生成步对整段已生成序列做一次 forward 穿过 N 层（配合 KV Cache 可复用历史 K/V，见 KV Cache 章）。不是「每层每 Token 单独调 API」——那是工程批处理细节。

  • 栈结构：Encoder / Decoder / Decoder-only 三种；CorpAssist 对话以 Decoder-only 为主。
  • 残差 + LN：Pre-LN 主流；残差提供梯度捷径，LN 稳定特征尺度。
  • 位置编码：打破 Attention 置换不变性；嵌入与位置相加后入栈。
  • FFN：逐 Token MLP，扩维再压回；参数量与表达能力的重要来源。

本章先记住 4 件事

单头注意力矩阵运算

Self-Attention 即 Q、K、V 均由同一输入 X 投影：X ∈ ℝn×dmodel。单头时 dk=dv=dmodel（多头时见 下一章）。

下列推导默认 batch=1；n 为序列 Token 数。面试白板建议先写形状再填数字，避免矩阵乘维度对错。

形状追踪表

步骤运算输出形状含义
输入 X (n, d_model) n 个 Token，各 d_model 维
投影 Q=XWQ, K=XWK, V=XWV 各 (n, d_k) 同序列三种角色
打分 S = QKT/√dk (n, n) 每对位置相似度
权重 A = softmax(S) (n, n) 行随机矩阵
输出 O = AV (n, d_v) 更新后的序列表示

Self-Attention(X) = softmax(QKT / √dk) V，其中 Q=XWQ，K=XWK，V=XWV

矩阵视角：一次「全连接阅读」

把 A 想成 n 个 Query 各读一遍 n 个 Key 的软查表；O 的每一行是 V 的 n 行按 A 对应行权重混合。对 CorpAssist 制度问答，n 包含 system + 检索 chunk + 用户问题 + 已生成回答——所有 Token 两两可互看（Decoder 生成时另有因果 Mask）。

Self-Attention 矩阵乘法链
flowchart LR
  X["X (n×d)"] --> Q["Q (n×dₖ)"]
  X --> K["K (n×dₖ)"]
  X --> V["V (n×dᵥ)"]
  Q --> S["S=QKᵀ (n×n)"]
  K --> S
  S --> A["softmax → A (n×n)"]
  A --> O["O=A·V (n×dᵥ)"]
  V --> O

L 层堆叠时，每层重复；总计算量大致 ∝ L·n²·d。

向量化实现（批维 B）

python 
# X: (B, n, d_model) — B 条 CorpAssist 请求 batch
Q = X @ W_q          # (B, n, d_k)
K = X @ W_k
V = X @ W_v
scores = Q @ K.transpose(0, 2, 1) / math.sqrt(d_k)  # (B, n, n)
attn = softmax(scores, dim=-1)
out = attn @ V       # (B, n, d_v)

复杂度 O(n²) 含义

计算 QKT 需对 n×n 个元素各做 dk 次乘加 → 时间 O(n²·d)。存储中间 S 或 A 需 O(n²) 显存（训练时尤为明显）。层数 L 线性乘上，但长度 n 平方放大是长上下文的首要敌人。

长度翻倍代价示意

序列长度 n注意力矩阵元素 n²相对 2K 基线
2,048~4.2M1×
8,192~67M16×
32,768~1.07G256×
128,000~16.4G~4000×

这也解释为何「上下文窗口从 8K 提到 128K」不仅是营销：同等 batch 下算力与 GPU 显存 需求可差两个数量级以上。优化方向包括 Flash Attention（IO 感知）、稀疏注意力、KV Cache（推理）、长上下文技术。

prefill vs decode 两阶段（与延迟挂钩）

  • prefill：首次吃进整段 prompt，须算完整 n×n Attention → TTFT 敏感，延迟 ∝ n²。
  • decode：每步只新增 1 Token，配合 KV Cache 可复用历史 K/V → 单步代价 ∝ n，但步数 = 生成长度。

CorpAssist 网关若只优化 decode 而放任 32K RAG 前缀，用户仍会在「首字出现前」长时间等待——控 prompt n 比调 temperature 更影响 TTFT。

与 RNN O(n) 的对比（口述）

  • RNN：时间步串行 O(n)，每步隐状态维固定，难并行但内存 O(n)。
  • Transformer：层内并行，但 n×n 矩阵 O(n²) 时间与内存。
  • 权衡：GPU 时代选中 Transformer，但应用层必须控 n。
先想 10 秒：层数 L 从 32 增到 64，与 n 从 4K 增到 8K，哪个对延迟打击更大？

通常 n 翻倍打击更大：Attention 项 ∝ L·n²，n 翻倍带来 4× Attention 代价，而 L 翻倍仅 2×。CorpAssist 网关限流应优先监控输入 Token 总量，而非只数 API 调用次数。

步骤 1：S = QKT / √2

QKT 第 (0,0) 元 = 1×1+0×0 = 1；(0,1)=0；(0,2)=1。完整 S ≈

text 
S_raw = [[1, 0, 1],
         [0, 1, 1],
         [1, 1, 2]]

S = S_raw / 1.414 ≈ [[0.71, 0, 0.71],
                      [0, 0.71, 0.71],
                      [0.71, 0.71, 1.41]]

步骤 2：对第 0 行 softmax

exp(0.71)≈2.03, exp(0)≈1, exp(0.71)≈2.03 → 和≈5.06。A0≈[0.40, 0.20, 0.40]（行和=1）。

text 
O_0 = 0.40*[1,0] + 0.20*[0,1] + 0.40*[2,2]
    ≈ [0.40+0+0.80, 0+0.20+0.80]
    ≈ [1.20, 1.00]

  • Cross-Attention：Q 来自解码器，K/V 来自编码器或外部 memory。

多头并行子空间

单头 Attention 只有一套 Q/K/V。多头注意力（MHA） 把 dmodel 切成 h 份，每份维度 dk = dmodel/h，各自独立算 Attention，最后拼接再乘 WO 投影回 dmodel。

多头不改变渐近 O(n²)——每个头仍算 n×n 权重，只是 d_k 变小、h 路并行；总 FLOPs 与单头同量级，主要增益在表达多样性。

公式骨架

headi = Attention(Qi, Ki, Vi)，  i = 1…h
MultiHead = Concat(head1, …, headh) WO

CorpAssist 7B 级常见：h=32，dmodel=4096，dk=dv=128（每头 128 维）。

  • 切分：dmodel 均分 h 份，每头独立投影 Q/K/V，互不共享参数。
  • 拼接：h 个头输出沿特征维 concat 后，经 WO 线性映射回 dmodel。
  • 总 FLOPs：与单头全维 Attention 同量级；收益在表达多样性而非降复杂度。

为何要多头？

  • 多子空间并行：不同头可学习关注不同模式（句法、共指、数字邻近等）。
  • 稳定训练：较多个小头比单一大头更易优化（类似集成）。
  • 算力折中：总参数量与单头全维相近，但表达更丰富。

单头 vs 多头（表达力直觉）

单头 Attention 只有一套 Q/K/V 投影；多头 相当于并行跑 h 个「专家读者」，有的头偏向句法邻近，有的偏向共指、数字对齐等——最后拼接由 WO 学会如何混合。CorpAssist 评测 badcase 时，个别头可呈现对「制度编号」「金额」列的集中权重（研究向，非生产依赖）。

  • 头数 h 增大 → 每头 dk 变小（dmodel 固定时），单头容量下降但多样性上升。
  • GQA/MQA（practice 章）进一步在 K/V 头间共享，降 KV Cache 显存。
  • 应用岗读 config：num_attention_heads 与 num_key_value_heads 可能不同。

Padding Mask

训练 CorpAssist 意图分类小模型时，batch 内短句用 <pad> 对齐是常规操作。Padding Mask 确保模型不会把 pad 当成真实语义去 attend——否则「报销」可能错误地关注到另一句的 padding 列，拉低验证集准确率。

Hugging Face attention_mask 中 1 表示真实 Token、0 表示 pad；传入模型前须与 input_ids 等长，Java 侧拼 batch 时由网关统一生成。

批处理多条 CorpAssist 请求时，序列长短不一，短句右侧用 <pad> 填充到统一长度。Padding Mask 在 softmax 前把pad 位置分数置为 −∞，使权重为 0，避免模型从 padding 读取噪声。

Mask 矩阵示意（batch=2）

text 
序列 A: [我, 要, 报销]           → 长度 3
序列 B: [制度, 问答, pad, pad]   → 长度 4（pad 为填充）

对序列 B，任何 Query 对 pad 列的 score 设为 -inf
有效 Token 只 attend 到前 2 个真实词

工程注意

  • 训练与推理必须一致：漏 Mask 会导致 pad 污染表示，批推理准确率抖动。
  • 自研 Java 网关若做 batch 拼 prompt，须同步传递 attention_mask 给推理服务（与 Hugging Face 约定一致）。
  • 单条在线请求常无 pad，Mask 全 1；但微调数据管道仍要处理。
  • 动态 padding（按 batch 内最长句）可省算力，但须每 batch 重算 mask 张量。
  • 导出 ONNX/TensorRT 时确认 mask 作为输入传入，勿写死全 1。
先想 10 秒：Padding Mask 加在 softmax 之前还是之后？

之前：对无效 (i,j) 位置将 score 设为 −∞，softmax 后权重为 0。若在 softmax 之后乘 0 虽也可行，但数值上常在 softmax 前 mask 以避免 exp 溢出并实现标准 fused kernel。

Causal Mask（自回归）

CorpAssist 流式对话里，用户看到 assistant 消息逐字出现，本质是模型每步只多生成 1 个 Token。因果 Mask 在训练阶段就强制模型习惯「只看过去」——与推理时可见上下文一致，避免 train/serve skew。

推理框架在 KV Cache 命中时仍须对新 Token 行应用 Causal Mask，使其不能 attend 到未生成的未来位置。

去掉 Causal Mask 的「捷径」训练在上线后几乎必然产生泄漏式胡话，不可用于生产微调。

SFT 数据构造时，labels 仍须与 Causal 可见范围一致，仅对 assistant 段计算 loss。

GPT 类 因果 Mask（Causal / Look-ahead Mask） 保证位置 i 只能看见 ≤i 的 Token，不能偷看未来。否则训练时标签泄漏，推理时无法逐 Token 生成。

Cross-Attention 场景

当解码器生成目标语言每个词时，Cross-Attention 让 Q 来自「我已生成部分」，K/V 来自「源语言编码」——经典机器翻译里生成英文 "subsidy" 时对齐中文「补贴」位置。CorpAssist 若用 T5 做制度中英对照，栈内仍有 Cross-Attn；Qwen 对话则把双语材料拼进同一上下文用 Self-Attn 完成。

Cross-Attention（交叉注意力） 中 Query 来自一方序列（常为 Decoder 当前状态），Key/Value 来自另一方（Encoder 输出或外部 memory）。这是 Seq2Seq 翻译与部分多模态架构的核心。

仅解码器栈

CorpAssist · 制度问答流式回复 员工提问后，dialog-orchestrator 将 RAG 引用与 user 消息拼成 token 序列，经 llm-gateway 调用 Qwen/GPT 类模型。模型内部是 N 层 Decoder Block 堆叠：每层 = 因果 Self-Attention + FFN + 残差与归一化。没有单独的 Encoder 读完整文档——文档内容已在 prompt 里以 token 形式「一次性读入」前缀。

Next-Token 训练目标

Next-Token 预测（也称因果语言建模 CLM）是 GPT 预训练的核心：对序列 x_1, …, x_T，模型在每个位置 t 预测 x_{t+1}，损失为交叉熵之和。因因果掩码，位置 t 的状态只依赖 x_1…x_t，与推理时「给定前缀生成下一个 token」完全同构。

训练目标（概念公式）

ℒ = −∑t=1T−1 log P(xt+1 | x1, …, xt; θ)

等价写法：L = -sum_t log P(x_{t+1} | x_1..x_t; theta) — 对每个位置 t 预测下一 token 的交叉熵之和。

对齐（SFT）、RLHF 等后训练阶段不改变这一自回归接口，只是改变条件分布 P(·|context) 偏向「有用、安全、遵循指令」。CorpAssist 的 system prompt「仅依据引用回答」是在推理时塑造条件，而非运行时换架构。

阶段数据形式与 CorpAssist 关系
预训练 CLM 海量文本，下一 token 标签 基座能力：语法、常识、代码模式
SFT instruction-response 对 贴近「助手」口吻与格式
偏好对齐 人类偏好排序 / RL 减少有害输出、提升遵循度
在线推理 messages → 单序列前缀 gateway 流式返回 incremental tokens

伪代码：训练一步 vs 推理一步

python 
# 训练：整段序列并行算所有位置的 next-token loss（Teacher Forcing）
def training_step(model, token_ids: list[int]) -> float:
    logits = model(token_ids)  # shape: (seq_len, vocab)
    loss = cross_entropy(logits[:-1], token_ids[1:])
    return loss

# 推理：自回归循环，每次只取最后一个位置的 logits
def generate(model, prompt_ids: list[int], max_new: int = 128) -> list[int]:
    ids = list(prompt_ids)
    for _ in range(max_new):
        logits = model(ids)          # 生产环境用 KV Cache，只算最后一步
        next_id = sample(logits[-1]) # 见下一节 temperature / top-p
        ids.append(next_id)
        if next_id == eos_id:
            break
    return ids
Teacher Forcing 与推理不一致会出问题吗？
训练时模型总看到「正确」历史 token；推理时历史由自身采样得到，存在 exposure bias。工程上靠足够规模预训练、采样策略与对齐缓解。应用侧通过低温度、约束解码、引用块降低跑偏，而非改架构。

采样与温度（算法视角）

Decoder 最后一步输出词表上的 logits z。贪心取 argmax；采样先得到概率 p = softmax(z / T)，其中 T 为 temperature。T→0 趋近贪心；T>1 分布更平、更随机。CorpAssist 客服场景常用 T=0.3~0.7；创意文案可略高。

参数作用CorpAssist 建议
temperature 缩放 logits，调节随机性 制度问答偏低；头脑风暴略高
top_p (nucleus) 从累积概率达 p 的最小词集采样 与 temperature 二选一或组合，防长尾胡言
max_tokens 限制自回归步数上界 编排层与网关双重限制，控费用
stop / EOS 提前终止 decode 环 JSON 模式可配 stop sequence

temperature 对分布的影响（示意）

python 
import numpy as np

def softmax(logits, temperature=1.0):
    z = logits / max(temperature, 1e-8)
    z = z - z.max()
    e = np.exp(z)
    return e / e.sum()

logits = np.array([2.0, 1.0, 0.5, 0.1])  # 四个候选 token
print("T=0.3:", softmax(logits, 0.3).round(3))  # 更尖锐 → 几乎确定性
print("T=1.0:", softmax(logits, 1.0).round(3))
print("T=1.5:", softmax(logits, 1.5).round(3))  # 更平坦 → 更易选到低分 token
应用反模式
  • 制度问答 temperature=1.2 求「灵活」→ 幻觉增加
  • 不设 max_tokens → 自回归环过长，p95 爆炸
  • 把 top_p 当「质量开关」而不看任务类型
推荐
  • RAG 问答：低 T + 引用约束 + 拒答模板
  • 网关按租户存默认采样 profile
  • 评测集固定 seed / T=0 便于回归

MLM 与双向上下文

CorpAssist · 制度段落入库 rag-index 将 HR 制度 PDF 分块为 512 token 段落。每段经 bge-large-zh（BERT 类 Encoder）编码为 1024 维向量。用户问「产假天数」时，query 也用同一 Encoder 编码，在向量库做 ANN 检索——这要求表示对整句语义双向理解，而非自回归续写。

BERT（Bidirectional Encoder Representations from Transformers）使用仅 Encoder 栈：多层双向 Self-Attention + FFN。预训练任务为 Masked Language Model（MLM）：随机 mask 15% token，模型根据左右两侧上下文预测被 mask 词。与 GPT 的 Next-Token 不同，MLM 需要双向可见性，故不使用因果掩码。

Embedding 输出用途

Encoder 最后一层输出 shape 约为 (seq_len, hidden_dim)。检索常用句级 pooling：

  • [CLS] token：BERT 原设计，取第 0 位向量
  • Mean pooling：对非 pad 位置求平均（bge、E5 常见）
  • Last token：部分新模型取最后有效 token

向量经 L2 归一化后，用内积或余弦相似度检索。CorpAssist rag-index 存储 chunk_id, embedding, metadata，query 向量与库内向量做 top-k ANN。

与 RAG 向量模型关系

现代 RAG 向量模型多继承 BERT Encoder 架构，经对比学习（In-batch negatives、难负例）微调，使语义相近的 query-doc 向量接近。与原始 BERT MLM 不同，但双向 Encoder 骨干相同。CorpAssist 典型链路：

CorpAssist RAG：Encoder 检索 + Decoder 生成
flowchart LR
  DOC[制度 PDF 分块] --> ENC1[Encoder Embedding]
  ENC1 --> IDX[(rag-index)]
  Q[用户问题] --> ENC2[同一 Encoder]
  ENC2 --> IDX
  IDX --> CITE[top-k 引用块]
  CITE --> PROMPT[编排层拼 messages]
  PROMPT --> DEC[Decoder-only Chat 生成]

两阶段、两模型：检索质量上限由 Encoder 决定；最终措辞由 Decoder 决定。

常见增强：Hybrid 检索（BM25 + 向量）、Reranker（Cross-encoder，仍多为 Encoder 变体）。这些都不替代最终 Chat 生成，但影响引用是否准确——引用错则 Decoder 再强也会幻觉。

RMSNorm 与 SwiGLU

原始 Transformer 用 LayerNorm；LLaMA 改用 RMSNorm（Root Mean Square Layer Normalization）：不中心化，只按均方根缩放，计算更省。FFN 从「两层 GELU」改为 SwiGLU：三路投影 W1, W3 与门控 σ(W1x) ⊙ (W3x) 再经 W2，参数量与 expressivity 提升。

归一化 LayerNorm（Pre-LN 或 Post-LN） RMSNorm（多为 Pre-Norm）
FFN Linear → GELU → Linear SwiGLU 门控 FFN
位置 绝对正弦 / 可学习 RoPE（见 advanced 位置编码章）
Linear bias 通常有 多无 bias（LLaMA 传统）

SwiGLU FFN 形状（概念）

python 
# 单层 Decoder Block 内 FFN（示意，非完整 forward）
def swiglu_ffn(x, w_gate, w_up, w_down):
    # x: (batch, seq, hidden)
    gate = silu(x @ w_gate)      # silu = x * sigmoid(x)
    up = x @ w_up
    return (gate * up) @ w_down  # 中间维常为 8/3 * hidden 或 4*hidden
面试误区
  • 「SwiGLU 是激活函数名字」—— 是 FFN 结构
  • 「RMSNorm 与 LayerNorm 完全一样」—— 少减均值步骤
  • 「Norm 位置不影响训练」—— Pre-Norm 更稳深栈
工程关联
  • FFN 占参数量大头 → 量化 FFN 收益大
  • Pre-Norm 便于堆深 → 层数↑影响延迟
  • 读 config 看 intermediate_size
先想 10 秒：CorpAssist 换 Qwen2.5-32B 比 7B，FFN 哪部分算力涨最多？

FFN 与 Attention 均随 hidden 与 intermediate 涨；32B 的 hidden_size、num_hidden_layers、intermediate_size 均更大，单 token FLOPs 显著高于 7B。网关应配路由：简单 FAQ → 7B，复杂推理 → 32B。

Grouped-Query Attention

标准 MHA：Q、K、V 头数相同。MQA：K/V 仅 1 组头；GQA（Grouped-Query Attention）：K/V 头数少于 Q，多 Q 头共享一组 K/V。推理时 KV Cache 大小 ∝ K/V 头数，GQA 在显存与速度上显著优于 MHA，LLaMA-2/3、Qwen2、DeepSeek 等广泛采用。

推理时 KV 复用

CorpAssist · 流式制度问答 用户问题 + 6 段 RAG 引用共 4k token prefill。模型输出 200 token 答案时，第 50 个输出 token 的注意力需访问全部 4k+49 个历史位置的 K/V。若无 KV Cache，每步重算整段 Self-Attention，ITL 会线性恶化——用户感觉「越生成越卡」。

Self-Attention 中，对位置 t 的 query q_t 需与所有 k_1…k_t 做点积。自回归 decode 时 t 逐步增大；历史 token 的 K/V 在先前步已算过，存入 Cache 后仅计算新 token 的 Q/K/V 并 append。

Decode 步 KV Cache 复用
sequenceDiagram
  participant S as 序列状态
  participant C as KV Cache
  participant A as Attention
  Note over S,C: Prefill 阶段
  S->>A: 整段 prefix tokens
  A->>C: 写入每层 K/V
  Note over S,C: Decode 第 t 步
  S->>A: 新 token x_t
  A->>C: 读取 K/V 1..t-1
  A->>C: append K_t V_t
  A->>S: 输出 hidden_t

Prefill 一次写入长前缀 Cache；Decode 每步只增量计算，Attention 读全 Cache。

无 Cache vs 有 Cache（伪代码）

python 
# 简化：单层单头 decode 一步
def decode_step_no_cache(model, all_token_ids):
    # 每步重算 1..t 全部 K/V — O(t) 重复计算
    hidden = model.forward_full_sequence(all_token_ids)
    return hidden[-1]

def decode_step_with_cache(model, new_token_id, kv_cache):
    # 仅对新 token 算 Q/K/V，与 cache 中历史 K/V 做 attention
    q, k, v = model.project_new_token(new_token_id)
    kv_cache.append(k, v)
    out = attention(q, kv_cache.all_k, kv_cache.all_v)
    return out

每轮 user/assistant 消息 append 到上下文，有效 seq 长度增长，Cache 按当前全长存储。编排层需截断历史或摘要，否则 p95 与 OOM 风险上升——这是应用层上下文管理，不是模型 API 自动解决。

Flash Attention 直觉

标准 Attention 实现需物化 n×n 注意力分数矩阵，读写 HBM 成为瓶颈。Flash Attention 通过分块（tiling）在 SRAM 内完成 softmax 与加权，避免完整矩阵落盘，数学等价于精确 Attention（在 fp 误差内），显著加速 prefill 与长序列。

Flash Attention：分块避免完整 n×n 落 HBM
flowchart LR
  Q[Q 分块] --> SRAM[GPU SRAM 小块]
  K[K 分块] --> SRAM
  V[V 分块] --> SRAM
  SRAM --> ACC[在线 softmax 累加]
  ACC --> OUT[输出 O 无需存全 S]

应用岗只需知：启用 FA2 通常降延迟、降显存峰值；具体 CUDA 实现属 mustNotCover 边界。

推理框架（vLLM、TensorRT-LLM、SGLang 等）默认集成 Flash Attention 或同类 kernel。CorpAssist 私有化部署检查清单：

  • GPU 架构是否支持（Ampere+ 常见）
  • PyTorch / CUDA / flash-attn 版本匹配
  • 长上下文 prefill 是否走 FA（否则 TTFT 尖刺）
Flash Attention 与 KV Cache 关系？
正交优化：KV Cache 减重复计算；Flash Attention 让每一步 attention 算得更快、更省显存峰值。二者同时使用是生产标配。

  • Softmax 按行：对每个 query 位置，在所有 key 上归一化。

实现 Scaled Dot-Product

给定序列长度 seq_len 与模型维 d_model，设 Q/K/V 形状均为 (seq_len, d_model)（单头简化）。注意力分数 S = QKT / √dk，权重 A = softmax(S)，输出 O = AV。

核心公式

Attention(Q, K, V) = softmax(QKT / √dk) V

单头简化时 Q, K: (seq_len, d_k)，V: (seq_len, d_v)；Q @ K.T 得 (seq_len, seq_len) 分数矩阵，softmax 沿最后一维（Key 维）归一化，使每行权重和为 1，再与 V 相乘得输出 (seq_len, d_v)。

  • 缩放：除以 √dk 防止点积过大、softmax 过尖（见 注意力机制：Q/K/V 与缩放点积）。
  • 因果 Mask：GPT 解码时遮未来 token，上三角权重 ≈ 0（demo 第二段输出）。

python 
import numpy as np

def softmax(x, axis=-1):
    shifted = x - np.max(x, axis=axis, keepdims=True)
    exp = np.exp(shifted)
    return exp / np.sum(exp, axis=axis, keepdims=True)

def scaled_dot_product_attention(q, k, v, mask=None):
    d_k = q.shape[-1]
    scores = q @ k.T / np.sqrt(d_k)
    if mask is not None:
        scores = np.where(mask, scores, -1e9)
    weights = softmax(scores, axis=-1)
    return weights @ v, weights

CorpAssist 类比：RAG 引用 token 在 GPT 前缀中，因果 mask 保证生成第 t 个答案 token 时不能「偷看」尚未生成的后续答案 token；但对已写入前缀的引用，只要位置 ≤ 当前，即可 attend。

张量shape（单头）含义
Q(seq, d_k)「我要查什么」
K(seq, d_k)「可被匹配的键」
V(seq, d_v)「取回的内容」
weights(seq, seq)对每个 query 行 softmax 和为 1
output(seq, d_v)加权后的表示
先想 10 秒：为何不除以 d_k 而除以 sqrt(d_k)？

点积方差随 d_k 增大而增大，softmax 会极端 one-hot，梯度变小。缩放使分数分布更稳定，与 缩放点积 一致。

因果 Mask 为下三角矩阵 M：M[i,j]=1 当 j≤i，否则 mask 掉。实现上用 np.tril 生成 bool mask，在 softmax 前把禁止位置设为 -1e9。

  • mask 用 0/1 直接乘 scores（softmax 后非零）
  • softmax 沿错轴（列而非行）
  • 忘记 scale，权重 one-hot 极端

  • 禁止位 large negative 再 softmax
  • axis=-1 对 key 维归一化
  • 验收 weights.sum(axis=1) ≈ 1

正弦位置编码

原始 Transformer（Vaswani et al., 2017）在嵌入层相加固定 正弦位置编码：偶数维用 sin，奇数维用 cos，不同维度对应不同波长。其设计动机是：对任意固定偏移 Δ，PE(pos+Δ) 可表示为 PE(pos) 的线性函数，从而隐含相对位置信息。

CorpAssist · 制度 PDF 切块后的顺序 同一政策文档切成 40 个 chunk 再拼接进 prompt 时，模型需知道「第 12 段在讲请假流程、第 13 段在讲审批链」——若缺少位置信号，Attention 可能把相隔很远的两条款误当作相邻条款比对。位置编码 就是给每个 token 打上「序列坐标」。

正弦公式（概念级）

PE(pos, 2i) = sin(pos / 100002i/d)
PE(pos, 2i+1) = cos(pos / 100002i/d)

pos 为位置索引；i 为维度对索引；d 为模型维度。最终输入：x = TokenEmbed + PE(pos)

可学习绝对 PE 查表相加 是 无法直接超出训练最大长度 GPT-2、部分 Encoder

  • CorpAssist 选模型时看有效上下文与位置方案，而非背公式
FAQ：BERT 也用位置编码，为何说 GPT 系更关心外推？

BERT 编码器常处理固定 512 token 分类/句对；GPT 类 Decoder 在推理时要自回归展开，KV Cache 随序列增长，上下文长度 直接决定能否一次读完整份制度 PDF。应用岗应关注 API 标注的 context window 与实测 p95，而非推导 PE 梯度。

相对位置与 ALiBi

相对位置 强调 token 对 (i, j) 的距离 Δ = i − j，而非各自绝对坐标。T5 在 Attention logits 上加可学习 bias(Δ)；ALiBi（Attention with Linear Biases, Press et al.）则用固定线性斜率惩罚远距离键，使模型在未见过更长序列的训练下，仍能在推理时保持较稳定的注意力衰减模式。

ALiBi 核心直觉

对第 m 个 query 与第 n 个 key，在缩放点积分数上添加 −m·|m−n|（m 为 head 专属斜率，越靠后的 head 斜率越大）。效果：距离越远，分数被扣得越多，形成inductive bias 偏向局部上下文。与 RoPE 相比，ALiBi 不修改 Q/K 向量本身，实现简单，曾用于 MPT、BLOOM 等；但 2024 后主流开源 LLM 仍以 RoPE 为主。

RoPE 旋转直觉

RoPE（Rotary Position Embedding）把位置信息通过复平面旋转注入 Query 与 Key：将相邻维度两两配对，按位置 pos 旋转角度 θ(pos)。神奇之处在于：旋转后的 q·k 内积只依赖 (pos_q − pos_k)，即相对位置，且与绝对起点无关。

二维旋转示意（单对维度）

python 
import numpy as np

def rotate2d(x_pair, pos, base=10000.0, dim_pair_idx=0):
    """概念演示：对 (x_{2i}, x_{2i+1}) 施加位置旋转"""
    freq = 1.0 / (base ** (2 * dim_pair_idx / 64))  # 示意 d=64
    angle = pos * freq
    c, s = np.cos(angle), np.sin(angle)
    x0, x1 = x_pair
    return np.array([x0 * c - x1 * s, x0 * s + x1 * c])

# 关键性质（口述面试）：
# rotate(q, m) · rotate(k, n) 的标量积 ∝ f(q, k, m-n)
# 相对距离 m-n 决定注意力，而非绝对 m、n

高维时对各维度对使用不同旋转频率（波长从短到长），低频维度捕获长距离相对关系，高频维度捕获局部邻接——与正弦 PE 的频率分解思想一脉相承，但作用在 Q/K 而非 embedding 加法。

与 Multi-Head 关系？ 每个 head 独立 Q/K，RoPE 逐 head 应用；GQA 共享 K 时旋转仍按位置一致 Megatron 并行切分
推理增量解码？ 新 token 位置 pos，仅对新 Q/K 算旋转角；历史 KV 已带位置信息可缓存 KV Cache 内核实现

Self-Attention 本身不知道顺序；RoPE 在 Q/K 上做位置相关旋转，使注意力分数只依赖相对距离。LLaMA/Qwen 默认方案，配合 KV Cache 做增量解码。

  • 正弦位置编码：在嵌入层注入绝对坐标，隐含相对性但长外推仍弱；可学习 PE 受最大长度限制。

专家路由直觉

标准 Transformer 每层 FFN 对所有 token 使用同一组权重。MoE（Mixture of Experts）把 FFN 拆成 N 个专家（各自一套 MLP），另设门控（Gating / Router）为每个 token 输出 N 维分数，取 top-k 专家加权组合输出。于是模型容量随 N 增长，但每步只算 k 个专家。

CorpAssist · 多领域问题分流 员工问「年假制度」与「报销 SAP 科目」——理想化想象：不同专家擅长 HR 政策 vs 财务流程。真实 MoE 的专家分工未必可解释，但路由确实让不同 token 走不同 FFN 路径，这是「稀疏激活」的工程直觉来源。

Attention 子层通常仍稠密；稀疏发生在 FFN。面试白板至少画出 Router → Top-k → 加权合并。

def moe_ffn(h, experts, router, k=2):
    """概念演示：top-k 路由 MoE FFN"""
    logits = router(h)                    # shape: [num_experts]
    weights, indices = topk_softmax(logits, k)
    out = 0
    for w, idx in zip(weights, indices):
        out += w * experts[idx](h)
    return out

Ring/Chunked Attention 概念

当单卡装不下长序列 KV 时，分块注意力把序列切成 chunk，每块在本地算 Attention；块间通过 Ring All-Reduce 等通信交换 KV，实现「逻辑全上下文、物理分布式」。这是推理/训练系统概念，应用岗知「存在此类多卡方案」即可，不需手写通信原语。

Chunked 直觉

text 
序列: [chunk0 | chunk1 | chunk2 | chunk3]
单卡: 先算 chunk1 局部 QKᵀ
跨块: 向相邻卡请求 chunk0 的 K/V → 拼接到注意力范围
Ring: 卡0→卡1→卡2→卡3→卡0 传递 KV 块，覆盖全历史

RAG 与长窗口取舍

手写 Attention 口述

高频白板题：写出 缩放点积注意力 并解释每步维度。应用岗答到矩阵形状与因果 Mask 即可，勿展开 CUDA tile。

四步口述模板

Attention(Q, K, V) = softmax(QKT / √dk) V

  1. 投影：Q, K, V 由输入 X 乘以 WQ, WK, WV 得到，shape (n, d)
  2. 打分：S = QKT / √dk，表达 token 两两相似度
  3. Mask（可选）：padding 置 −∞；因果 Mask 遮未来 token
  4. 聚合：权重 A = softmax(S)，每行和为 1；输出 = AV
Self-Attention 数据流
flowchart LR
  X[输入 X n×d] --> QKV[线性层得 Q,K,V]
  QKV --> SCORE["QKᵀ / √d_k"]
  SCORE --> MASK[Mask 可选]
  MASK --> SM[softmax]
  SM --> OUT["A·V → 输出"]

面试时先画此图，再写公式；比单写公式更有结构感。

追问要点CorpAssist 例子
为何除以 √d_k？ 维度大时点积方差大，softmax 趋 one-hot，梯度变小 长 chunk 多时仍靠缩放稳定训练（概念）
多头 做什么？ 多子空间并行关注不同关系，再拼接投影 一词多义：「苹果」公司 vs 水果
与检索相似度？ Q·K 类似 query-doc 打分，V 像文档内容加权 RAG 外挂检索是显式版「找相关段」
先想 10 秒：softmax 前行和为何必须 = 1？

保证输出是 V 的凸组合，可解释为「按关注度加权平均」；数值上避免尺度任意放大。若 mask 后全 -inf，需处理 NaN（工程实现细节，可口述「全 mask 行置零输出」）。

复杂度与优化题

Self-Attention 对序列长度 n 复杂度 O(n²)（算力与内存）。推理优化：KV Cache 避免重复算历史 K/V；Flash Attention 融合 kernel 减 HBM 读写；滑动窗口 / 稀疏 Attention 限制感受野（见 长上下文技术概览）。

优化解决复杂度/资源应用岗关注
朴素 Attention 基线 O(n²) 算力，O(n) KV 存 长 prompt 贵
KV Cache 解码步复用历史 K/V 每步 O(n) 增存 TTFT 与并发显存
Flash Attention IO 融合 同渐进复杂度，常数更小 p95 延迟改善
GQA/MQA 减 K/V head 数 降 KV 体积 见 架构对照
滑动窗口 限制注意力范围 O(n·w) 远程依赖需 RAG

幻觉/长文失效算法解释

幻觉 不只有「模型胡说」：从算法视角，训练目标（预测下一 token）并不保证事实正确；上下文过长时注意力分散、位置外推弱化，远程证据权重下降。应用侧用 RAG 约束证据、评测集监控引用率，而非指望改 Attention 公式一劳永逸。

现象算法层解释（概念）工程缓解
无检索胡编制度条款 生成分布偏向流畅而非可验证 RAG + 强制引用 + 拒答
长 prompt 漏中间段 「Lost in the Middle」注意力偏首尾 减 K、Rerank、分段摘要
超长外推掉点 RoPE 分布外，远程依赖弱 YaRN + 长文档 golden set
工具参数编造 Decoder 仍做 token 预测 JSON schema、函数调用 API
答辩误区
  • 「换更大模型就没有幻觉」——仅降低概率
  • 「Attention 可视化能完全解释错误」——相关非因果
  • 承诺「微调一定消除幻觉」——需数据与评测（不讲 RLHF 推导）
推荐结构
  • 承认生成目标与事实一致性不等价
  • 给 RAG + 护栏 + 评测三板斧
  • 用引用率、幻觉率指标闭环
