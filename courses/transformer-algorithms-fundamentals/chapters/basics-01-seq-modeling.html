本章先记住 4 件事
  • 序列任务的本质：输入输出都是有序符号串；模型须保留顺序信息与语义关联。
  • 词向量是起点：每个 Token 先变成 d 维向量，再在时间上叠代或并行变换。
  • RNN 串行、难并行：时刻 t 依赖 t−1 的隐状态，GPU 难以吃满；长文信息经多步传递易衰减。
  • Attention 是破局点：任意两位置可直接交互，为 Transformer 全并行铺路。

词向量与序列任务回顾

CorpAssist · 工单分类 员工提交「报销单被驳回，提示发票抬头不符」——系统需把这句话映射为向量，再预测应路由到「财务合规」还是「IT 报障」。这是典型的序列分类：整句输入，单个标签输出。

自然语言处理的核心难题之一是：词是离散符号，神经网络需要连续数值。词向量（Word Embedding） 把词典中每个词映射到 ℝd 空间中的一个点，语义相近的词（如「补贴」「津贴」）在该空间中距离更近。CorpAssist 的检索与分类链路都建立在「文本 → 向量」这一步之上。

常见序列任务类型

任务输入 → 输出CorpAssist 示例
语言建模 前文 → 下一词概率 对话续写、邮件草稿补全
序列分类 整句 → 类别 工单意图识别、敏感词检测
序列标注 每个词 → 标签 NER 抽取合同甲乙方、金额字段
序列到序列 源序列 → 目标序列 会议纪要摘要、多语言制度翻译

从 one-hot 到稠密嵌入

早期用 one-hot 表示词：维度 = 词表大小 V，几乎正交，无法表达语义相似性。Word2Vec、GloVe 等学习到的 稠密嵌入 把维度压到 256～1024，既省空间又保留「国王−男人+女人≈女王」这类类比关系。现代大模型不再单独预训练 Word2Vec，但嵌入层（Embedding Layer） 仍承担相同角色：查表得到每个 Token 的初始向量。

嵌入查表示例（NumPy 直觉）

python 
import numpy as np

vocab_size, d_model = 50000, 512
# 嵌入矩阵 E: (V, d)，第 i 行是词 i 的向量
E = np.random.randn(vocab_size, d_model) * 0.02

# CorpAssist 工单分词后的 token id 序列
token_ids = [1042, 8831, 220, 9910]  # 「报销」「驳回」「发票」「抬头」
# 查表得到序列矩阵 X: (seq_len, d_model)
X = E[token_ids]
print(X.shape)  # (4, 512)

得到 X 之后，序列建模 模块要在时间维上传播或混合信息——这正是 RNN 与后来 Transformer 的分野所在。

CorpAssist 嵌入层在双栈中的位置

  • Python 推理服务：Hugging Face AutoModel 首层 embed_tokens 即查表；RAG 侧 bge/E5 同理。
  • Java Spring AI：调用远程 Embedding API 或 ONNX 本地模型，输出 float[] 与 Python 维数须契约一致。
  • 网关计量：Token 数在嵌入之前由分词器确定，与 d_model 无关——估费时勿混淆。

嵌入只解决「每个词是什么向量」；序列建模 解决「这些向量如何随上下文组合成句义」——下一节 RNN 是第一种答案，Transformer 是现行答案。

无序列建模
  • 仅对词向量做平均池化
  • 丢失词序：「不报销」与「报销不」相同
  • 无法建模长距离修饰关系
有序列建模
  • 保留位置与上下文依赖
  • 可区分否定、条件从句等语序敏感语义
  • 支撑 CorpAssist 多轮对话与长文档理解

RNN/LSTM 的并行与长依赖瓶颈

循环神经网络（RNN） 按时间步 t=1…n 依次处理序列：每步读取当前词向量 xt 与上一步隐状态 ht−1，输出 ht。公式可简写为 ht = f(W·xt + U·ht−1 + b)。这种串行依赖在 2017 年前是序列建模主流，但有两处硬伤。

瓶颈一：训练难以并行

计算 h5 必须先算完 h4，GPU 数千核心无法同时展开整条序列——批内只能并行不同样本，单条序列内部仍是串行。CorpAssist 若用纯 RNN 训练内部分类器，长工单序列的训练吞吐会明显低于同等参数规模的 Transformer。

瓶颈二：长距离梯度与信息衰减

句子开头的主语要约束句尾动词的一致性，或制度条文第 1 条与第 47 条的「除外条款」关系，都需要跨越数十到数百 Token 的 长程依赖。标准 RNN 反向传播时梯度连乘，易出现梯度消失/爆炸；LSTM 用门控机制缓解，但路径仍须经过 O(n) 步隐状态传递，远距离信号会被稀释。

RNN 串行 vs Transformer 并行（概念对比）
flowchart TB
  subgraph RNN["RNN：时刻串行"]
    x1[x1] --> h1[h1]
    h1 --> h2[h2]
    x2[x2] --> h2
    h2 --> h3[h3]
    x3[x3] --> h3
  end
  subgraph TR["Transformer：层内并行"]
    X[X: 整段嵌入] --> SA[Self-Attention 一次看全序列]
    SA --> H[H: 所有位置同时更新]
  end

RNN 的 ht 链式传递限制并行；Self-Attention 在单层内让任意位置直接交互（详见 Self-Attention 矩阵视角）。

LSTM 门控直觉（不必背公式）

  • 遗忘门：决定从细胞状态丢弃多少旧信息。
  • 输入门：决定写入多少当前输入。
  • 输出门：决定暴露多少状态给下一步。

LSTM 在 2014–2016 年的机器翻译、语音识别中表现优异，但无法从根本上消除序列内串行。当 CorpAssist 需要处理 8K～128K Token 的制度汇编时，纯 LSTM 架构在训练与推理效率上均已退出主流 LLM 选型。

面试口述：RNN 两大瓶颈（30 秒版）

  1. 并行：ht 依赖 ht−1，单序列内无法像 Self-Attention 那样一层算完全部位置。
  2. 长依赖：信息须沿时间步传递 O(n) 次，梯度与信号均易衰减；LSTM 门控是补丁非根治。

向 Java 后端同事解释时可用类比：RNN 像单线程 for 循环处理数组；Transformer 像对数组做一次矩阵批量变换（在层内）。

先想 10 秒：为何「更大 LSTM、更多层」仍难追上 GPT 类模型的训练速度？

层数加深只改善表达能力，不消除时间维串行：每个时间步仍须等待上一步。Transformer 在序列长度维上可并行计算 Attention 矩阵（代价是 O(n²) 显存，见 复杂度章节），因而更适合现代 GPU/TPU 数据并行。

Encoder-Decoder 与 Seq2Seq

机器翻译推动了 序列到序列（Seq2Seq） 范式：用编码器把源语言句子压成上下文向量，再用解码器逐词生成目标语言。CorpAssist 的「中文制度 → 英文摘要」「工单详情 → 结构化 JSON」在抽象上都属于 Seq2Seq。

经典 Encoder-Decoder 数据流

CorpAssist 会议纪要摘要 · Seq2Seq 骨架
flowchart LR
  SRC[源序列: 会议逐字稿 Token] --> ENC[Encoder RNN/LSTM]
  ENC --> CTX[上下文向量 c]
  CTX --> DEC[Decoder RNN/LSTM]
  DEC --> TGT[目标序列: 摘要 Token 逐个生成]

早期 Seq2Seq 把整段源文「压」进单个向量 c，长会议时 c 成为信息瓶颈。

单向量瓶颈与注意力萌芽

当源序列很长（如 2 小时会议纪要 8000 字），单个 c 无法承载全部细节，摘要容易漏掉「预算审批结论」等关键句。2015 年 Bahdanau 等人引入 Attention：解码每一步不再只读 c，而是对编码器所有时间步输出做加权求和——相当于「生成摘要每一句时，回头看转录稿里最相关的段落」。这是通往 Transformer 的关键一步。

Seq2Seq 训练目标（应用岗口述版）

text 
给定源序列 x = (x1, x2, ..., xm)
目标序列 y = (y1, y2, ..., yk)
训练最大化 P(y | x) = ∏ P(yt | y<t, x)

CorpAssist 摘要场景：
  x = 会议转录 Token 序列
  y = 摘要 Token 序列
  每步预测「下一个摘要词」，条件于已生成摘要 + 源文编码

长源文信息挤进单向量 c，CorpAssist 长纪要摘要易漏关键决议；解码器对所有编码步「一视同仁」。

解码每步动态对齐源文不同位置，相当于可微分的「软检索」；为 Q/K/V 注意力 与全 Transformer 编解码铺平道路。

为何需要全局注意力

到 2017 年，序列建模的矛盾已经很清晰：需要长距离、任意位置间的灵活依赖，同时需要 GPU 友好的并行。RNN 家族给不了后者；带 Attention 的 Seq2Seq 改善了信息通路，但编码器仍是串行 RNN。Vaswani 等人在《Attention Is All You Need》中提出：干脆去掉 RNN，只用 Attention 堆叠——即 Transformer。

三条动机对照

痛点RNN/LSTM 表现全局 Self-Attention 思路
训练并行度 时间维串行，GPU 利用率低 同层内 n 个位置并行计算
长依赖路径 O(n) 步隐状态传递 任意两位置一步可达（矩阵一次乘法）
对齐可解释性 Attention 仅解码侧看编码 编码器内部亦可 Self-Attention 全文互看

本章结论

每条对应上一节 h3，用于评审前 30 秒复述。

  • 词向量与任务：嵌入把离散 Token 变为连续向量；序列分类/标注/Seq2Seq 是 CorpAssist 常见形态。
  • RNN/LSTM 瓶颈：时间维串行限制并行；长依赖经多步传递易衰减，LSTM 仅缓解不根治。
  • Seq2Seq：Encoder-Decoder 支撑翻译与摘要；单向量 c 是长文瓶颈，Attention 引入动态对齐。
  • 全局注意力动机：要并行 + 短路径长依赖；Transformer 用 Self-Attention 取代 recurrence。

判断练习

  1. （概念辨析）同事说：「CorpAssist 摘要慢是因为 LSTM 层数不够，加到 20 层就能追上 GPT 速度。」哪处最关键错误？应如何纠正？

    参考答案

    错误在于把「表达能力」与「并行度」混为一谈。LSTM 无论多少层，单条序列时间维仍串行；GPT 类 Transformer 在层内对全序列并行做 Self-Attention。纠正：瓶颈是架构的 recurrence，不是层数不够；应换 Transformer 路线或接受 RNN 训练吞吐上限。
