为什么要学本章

CorpAssist 的对话、摘要、Agent 回复几乎全部走 Decoder-only 大模型（GPT、Qwen、LLaMA 等）。你在应用层调用的 /v1/chat/completions 背后，是仅解码器栈 + 因果掩码 + 逐 token 自回归。不懂这一路线，就无法解释「为什么流式首 token 慢」「为什么上下文越长 p95 越高」——这些正是网关与编排层的 SLA 指标。

本章先记住 4 件事
  • 只看过去：Decoder 自注意力带因果掩码，位置 t 只能 attend 到 ≤ t 的 token。
  • 训练=猜下一个：给定前缀，最大化下一 token 的对数似然；无独立 Encoder。
  • 推理=循环解码：每步 append 一个 token，KV 可缓存（见 KV Cache 与 Flash Attention）。
  • Chat ≠ 原始 GPT：API 侧的 messages 模板、system 指令是对齐训练后的使用契约，不改变底层自回归本质。

仅解码器栈

CorpAssist · 制度问答流式回复 员工提问后，dialog-orchestrator 将 RAG 引用与 user 消息拼成 token 序列，经 llm-gateway 调用 Qwen/GPT 类模型。模型内部是 N 层 Decoder Block 堆叠：每层 = 因果 Self-Attention + FFN + 残差与归一化。没有单独的 Encoder 读完整文档——文档内容已在 prompt 里以 token 形式「一次性读入」前缀。

原始 Transformer 含 Encoder 与 Decoder 两栈；GPT 路线 保留仅 Decoder：去掉 Cross-Attention，只保留带 因果掩码 的自注意力。这与 Transformer 块：层归一化与 FFN 中的块结构一致，但掩码策略不同。

Decoder-only 栈（GPT / LLaMA / Qwen 对话骨干）
flowchart TB
  subgraph input [输入]
    TOK[Token Embedding + 位置编码]
  end
  subgraph stack [N x Decoder Block]
    MHA[Causal Multi-Head Self-Attention]
    FFN[FFN 通常 SwiGLU]
    LN[RMSNorm / LayerNorm]
    MHA --> FFN --> LN
  end
  subgraph head [输出头]
    LM[Linear 到词表 logits]
    SM[Softmax 得下一 token 分布]
  end
  TOK --> stack
  stack --> LM --> SM
  SM -->|append token| TOK

推理时自回归环：新 token 拼回序列，再进下一层（配合 KV Cache 避免重复算历史 K/V）。

组件Decoder-only 中的角色CorpAssist 工程触点
Token + 位置编码 将 messages 模板化后的文本变为向量序列 编排层控制 prompt 长度，影响首 token 延迟
因果 Self-Attention 每个位置只看左侧上下文 长 system + RAG 引用 → 注意力矩阵更大
FFN 逐 token 非线性变换，参数量占比高 模型越大，单 token 算力越高
LM Head 词表维度 logits，供采样 网关 logprobs、评测对齐
常见误解
  • 「Chat 模型有独立 Encoder 读 PDF」—— 文档在 prompt 里，仍走 Decoder
  • 「流式是一个 token 一个 HTTP 请求」—— 通常一次连接内多次 decode 步
  • 「因果掩码只在训练时用」—— 推理同样必须，否则泄露未来
正确心智
  • Encoder 路线见 BERT 与表示学习，与 GPT 分工不同
  • 流式 = 同一生成任务逐步返回 delta
  • 前缀越长，每步 attention 成本越高（未优化前 O(n²)）
先想 10 秒：RAG 检索到的 6 段制度文本，是在模型「Encoder」里编码的吗？

不是独立 Encoder 栈。编排层把片段拼进 prompt 字符串，Tokenizer 后作为 Decoder 的前缀 token 一次性参与因果注意力。检索质量取决于 embedding 模型（常为 Encoder），生成仍由 Decoder-only 完成。

Next-Token 训练目标

Next-Token 预测（也称因果语言建模 CLM）是 GPT 预训练的核心：对序列 x_1, …, x_T，模型在每个位置 t 预测 x_{t+1}，损失为交叉熵之和。因因果掩码，位置 t 的状态只依赖 x_1…x_t，与推理时「给定前缀生成下一个 token」完全同构。

训练目标（概念公式）

latex 
\mathcal{L} = -\sum_{t=1}^{T-1} \log P(x_{t+1} \mid x_1, \ldots, x_t; \theta)

对齐（SFT）、RLHF 等后训练阶段不改变这一自回归接口，只是改变条件分布 P(·|context) 偏向「有用、安全、遵循指令」。CorpAssist 的 system prompt「仅依据引用回答」是在推理时塑造条件，而非运行时换架构。

阶段数据形式与 CorpAssist 关系
预训练 CLM 海量文本，下一 token 标签 基座能力：语法、常识、代码模式
SFT instruction-response 对 贴近「助手」口吻与格式
偏好对齐 人类偏好排序 / RL 减少有害输出、提升遵循度
在线推理 messages → 单序列前缀 gateway 流式返回 incremental tokens

伪代码：训练一步 vs 推理一步

python 
# 训练：整段序列并行算所有位置的 next-token loss（Teacher Forcing）
def training_step(model, token_ids: list[int]) -> float:
    logits = model(token_ids)  # shape: (seq_len, vocab)
    loss = cross_entropy(logits[:-1], token_ids[1:])
    return loss

# 推理：自回归循环，每次只取最后一个位置的 logits
def generate(model, prompt_ids: list[int], max_new: int = 128) -> list[int]:
    ids = list(prompt_ids)
    for _ in range(max_new):
        logits = model(ids)          # 生产环境用 KV Cache，只算最后一步
        next_id = sample(logits[-1]) # 见下一节 temperature / top-p
        ids.append(next_id)
        if next_id == eos_id:
            break
    return ids
Teacher Forcing 与推理不一致会出问题吗？
训练时模型总看到「正确」历史 token；推理时历史由自身采样得到，存在 exposure bias。工程上靠足够规模预训练、采样策略与对齐缓解。应用侧通过低温度、约束解码、引用块降低跑偏，而非改架构。

采样与温度（算法视角）

Decoder 最后一步输出词表上的 logits z。贪心取 argmax；采样先得到概率 p = softmax(z / T)，其中 T 为 temperature。T→0 趋近贪心；T>1 分布更平、更随机。CorpAssist 客服场景常用 T=0.3~0.7；创意文案可略高。

参数作用CorpAssist 建议
temperature 缩放 logits，调节随机性 制度问答偏低；头脑风暴略高
top_p (nucleus) 从累积概率达 p 的最小词集采样 与 temperature 二选一或组合，防长尾胡言
max_tokens 限制自回归步数上界 编排层与网关双重限制，控费用
stop / EOS 提前终止 decode 环 JSON 模式可配 stop sequence

temperature 对分布的影响（示意）

python 
import numpy as np

def softmax(logits, temperature=1.0):
    z = logits / max(temperature, 1e-8)
    z = z - z.max()
    e = np.exp(z)
    return e / e.sum()

logits = np.array([2.0, 1.0, 0.5, 0.1])  # 四个候选 token
print("T=0.3:", softmax(logits, 0.3).round(3))  # 更尖锐 → 几乎确定性
print("T=1.0:", softmax(logits, 1.0).round(3))
print("T=1.5:", softmax(logits, 1.5).round(3))  # 更平坦 → 更易选到低分 token
应用反模式
  • 制度问答 temperature=1.2 求「灵活」→ 幻觉增加
  • 不设 max_tokens → 自回归环过长，p95 爆炸
  • 把 top_p 当「质量开关」而不看任务类型
推荐
  • RAG 问答：低 T + 引用约束 + 拒答模板
  • 网关按租户存默认采样 profile
  • 评测集固定 seed / T=0 便于回归

与 Chat API 生成关系

你在 CorpAssist 调用的 OpenAI 兼容 API，请求体是 messages 数组；服务端将其渲染为单一 token 前缀（含特殊 token、角色标记），再进入 Decoder-only 自回归。流式响应每个 delta.content 对应 decode 环的一次或多次 token 增量。

CorpAssist Chat 请求 → Decoder-only 推理
sequenceDiagram
  participant ORCH as dialog-orchestrator
  participant GW as llm-gateway
  participant TOK as Chat Template + Tokenizer
  participant DEC as Decoder-only 模型
  ORCH->>GW: POST /v1/chat/completions stream=true
  GW->>TOK: messages → token ids
  loop 自回归 decode
    DEC->>DEC: causal attention + sample
    DEC-->>GW: token delta
    GW-->>ORCH: SSE chunk
  end

首 token 延迟（TTFT）≈ 前缀 prefill + 首步 decode；后续 token 间隔（ITL）与 KV Cache、批大小相关。

API 请求片段（与自回归参数对应）

json 
{
  "model": "qwen-plus",
  "messages": [
    {"role": "system", "content": "仅依据引用回答；无依据则拒答。"},
    {"role": "user", "content": "产假天数是多少？\n\n[引用1] ..."}
  ],
  "stream": true,
  "temperature": 0.3,
  "max_tokens": 512
}

编排层职责：控制前缀长度（RAG top-k、历史轮次）、设定采样默认值、在 TTFT 超 SLA 时降级（换小模型、缩短引用）。算法认知帮助你在面试中说清：不是 API 「变慢了」，而是序列变长或 decode 步数变多。

completion 与 chat 在架构上有区别吗？
底层通常是同一 Decoder-only；差异在模板与对齐数据。Chat 模型经对话格式 SFT，更适合多轮 messages；纯 completion API 直接续写前缀。CorpAssist 应统一走 chat/completions 以便 system 注入合规指令。
为何 Agent 多步工具会拉高 p95？
每轮工具结果拼回 messages → 前缀变长 → prefill 与每步 attention 成本上升；且多轮多次完整生成。需步数预算、上下文压缩与 KV Cache Aware 路由（见后续章）。

本章结论

  • 仅解码器栈：GPT/Qwen/LLaMA 对话骨干 = N 层因果 Decoder Block，无独立 Encoder。
  • Next-Token：训练与推理同构；SFT/对齐只改条件分布，不改自回归接口。
  • 采样：temperature / top-p 调节 softmax 后分布；RAG 宜低温、限 max_tokens。
  • Chat API：messages → 模板化前缀 → 流式自回归；TTFT/ITL 是工程 SLA 核心。

判断练习

  1. （场景）产品要求「把 PDF 放进 GPT 的 Encoder，再让 Decoder 生成答案」，是否准确描述当前 Qwen-Chat 架构？

    参考答案

    不准确。Qwen-Chat 为 Decoder-only：PDF 文本需先解析为字符串，经 Tokenizer 进入前缀，无独立 Encoder 栈。向量检索应使用 Encoder 类 embedding 模型，再拼入 prompt。

  2. （排障）同一模型 TTFT 从 800ms 升到 2.5s，ITL 正常。最应优先排查什么？

    参考答案

    优先查前缀变长：RAG top-k 增大、历史对话未截断、system 模板膨胀。ITL 正常说明 decode 步成本尚可，瓶颈在 prefill 阶段对长序列的因果注意力（见 KV Cache 与 Flash Attention）。
