本章先记住 3 件事
  • 不是所有专家都跑：每 token 只激活 top-k 个 FFN 专家，算力近似 k/总专家数 × 稠密 FFN。
  • 总参数 ≠ 推理 FLOPs：显存可能仍要加载全部专家权重（除非 CPU offload / 专家并行）。
  • 负载均衡是工程题：训练用 auxiliary loss；推理关注 p95 与热点专家。

专家路由直觉

标准 Transformer 每层 FFN 对所有 token 使用同一组权重。MoE（Mixture of Experts）把 FFN 拆成 N 个专家（各自一套 MLP），另设门控（Gating / Router）为每个 token 输出 N 维分数，取 top-k 专家加权组合输出。于是模型容量随 N 增长，但每步只算 k 个专家。

CorpAssist · 多领域问题分流 员工问「年假制度」与「报销 SAP 科目」——理想化想象：不同专家擅长 HR 政策 vs 财务流程。真实 MoE 的专家分工未必可解释，但路由确实让不同 token 走不同 FFN 路径，这是「稀疏激活」的工程直觉来源。
MoE 层数据流（单 token 视角）
flowchart LR
  X[Token 隐状态 h] --> G[Router / Gating]
  G -->|scores| TOPK[Top-k 专家选择]
  TOPK --> E1[Expert 1 FFN]
  TOPK --> E2[Expert 2 FFN]
  TOPK --> E3[Expert k FFN]
  E1 --> W[加权求和]
  E2 --> W
  E3 --> W
  W --> OUT[h 输出]

Attention 子层通常仍稠密；稀疏发生在 FFN。面试白板至少画出 Router → Top-k → 加权合并。

路由伪代码（概念级）

python 
def moe_ffn(h, experts, router, k=2):
    """概念演示：top-k 路由 MoE FFN"""
    logits = router(h)                    # shape: [num_experts]
    weights, indices = topk_softmax(logits, k)
    out = 0
    for w, idx in zip(weights, indices):
        out += w * experts[idx](h)
    return out

# 训练时常加 load_balance_loss，避免少数专家垄断
# 推理时关注：k、专家数、是否共享专家（见下节）
概念含义面试一句话
专家数 N并行 FFN 套数容量旋钮，增大 N 不线性增大每 token 算力
Top-k每 token 激活专家数Mixtral 常用 k=2；k 越大越接近稠密
稀疏激活仅部分专家参与计算算力 ∝ k/N × 稠密 FFN
专家路由门控选专家可学习、与 token 内容相关
负载均衡专家使用率均匀训练 auxiliary loss；推理看热点与排队
先想 10 秒：若 k=N，MoE 与稠密 FFN 有何区别？

计算量接近，但参数布局仍是 N 套独立权重 + 路由器；除非合并，否则显存仍存 N 套。面试答：「k=N 失去稀疏意义，一般 k≪N。」

Mixtral/DeepSeek-MoE 对照

开源 MoE 并非同一模板。Mixtral 8×7B：8 个专家，每 token 选 2 个，无共享专家；有效 FFN 约为 2/8 稠密。DeepSeek 系列引入共享专家（始终激活）+ 路由专家（稀疏选择），在保持稀疏的同时稳定通用能力。

模型 专家结构 每 token 激活 面试强调点
Mixtral 8×7B 8 路由专家，无共享 2 专家 FFN 「8×7B」指总参数规模感，非 8 倍算力
DeepSeek-MoE 共享专家 + 路由专家 共享全激活 + 少量路由 共享保底通用；路由增容量
Qwen-MoE 多专家 + top-k 依配置 国内生态与量化工具链
稠密 LLaMA-70B 单 FFN 全 FFN 显存高、延迟可预测
误区
  • 「Mixtral 8×7B = 每次推理 56B 参数全激活」——实际约 2 专家路径 + Attention 稠密
  • 「专家=人工规则模块」——分工通常不可解释，勿过度拟人化
  • 「MoE 一定比稠密快」——显存带宽与路由开销可能抵消收益
推荐表述
  • 用「总参数 / 激活参数 / 部署显存」三列回答 HR
  • DeepSeek 共享专家是架构亮点，体现「稳定 + 稀疏」折中
  • 选型看 vLLM / 昇腾是否优化 MoE 路由 kernel（协作方能力）
FAQ：MoE Attention 还是稠密的吗？

主流实现中 Self-Attention 子层仍为稠密（所有 head 全算）；稀疏主要在 FFN。长上下文瓶颈仍在 Attention O(n²) 与 KV Cache——见 长上下文技术概览 与 KV Cache 与 Flash Attention。

激活参数 vs 总参数

招聘与模型卡片常写「总参数 236B」。推理成本与延迟更相关的是每 token 激活的参数量（参与矩阵乘法的权重规模）与需加载的权重体积（部署显存）。二者常被混淆，是 MoE 面试核心考点。

负载均衡：训练时若路由器总选同一专家，其余专家荒废；常用 auxiliary load balancing loss 鼓励均匀。推理时若热点专家成为瓶颈，p95 延迟抖动——可观测性应分专家记录（平台方职责，应用岗知概念即可）。

专家并行 vs 张量并行（口述边界）

专家并行：不同 GPU 存不同专家，路由决定激活哪块——MoE 训练/推理常见。张量并行：单层权重切分到多卡。二者均属 mustNotCover 的千卡/Megatron 范畴；应用岗只需在压测报告里看到「MoE 路由延迟」指标并追问平台方。

MoE 与 MoE 量化的关系

量化降低权重体积，但不改变 top-k 路由逻辑。边缘部署若只能加载部分专家，需专用框架支持——多数 CorpAssist 企业场景走云端 API，本地只需知「总参数量化后仍可能数十 GB」。

本章结论
