本章先记住 4 件事

RMSNorm 与 SwiGLU

原始 Transformer 用 LayerNorm；LLaMA 改用 RMSNorm（Root Mean Square Layer Normalization）：不中心化，只按均方根缩放，计算更省。FFN 从「两层 GELU」改为 SwiGLU：三路投影 W1, W3 与门控 σ(W1x) ⊙ (W3x) 再经 W2，参数量与 expressivity 提升。

组件原始 Transformer / BERTLLaMA / Qwen / DeepSeek 主流
归一化 LayerNorm（Pre-LN 或 Post-LN） RMSNorm（多为 Pre-Norm）
FFN Linear → GELU → Linear SwiGLU 门控 FFN
位置 绝对正弦 / 可学习 RoPE（见 advanced 位置编码章）
Linear bias 通常有 多无 bias（LLaMA 传统）

SwiGLU FFN 形状（概念）

python 
# 单层 Decoder Block 内 FFN（示意，非完整 forward）
def swiglu_ffn(x, w_gate, w_up, w_down):
    # x: (batch, seq, hidden)
    gate = silu(x @ w_gate)      # silu = x * sigmoid(x)
    up = x @ w_up
    return (gate * up) @ w_down  # 中间维常为 8/3 * hidden 或 4*hidden
面试误区
  • 「SwiGLU 是激活函数名字」—— 是 FFN 结构
  • 「RMSNorm 与 LayerNorm 完全一样」—— 少减均值步骤
  • 「Norm 位置不影响训练」—— Pre-Norm 更稳深栈
工程关联
  • FFN 占参数量大头 → 量化 FFN 收益大
  • Pre-Norm 便于堆深 → 层数↑影响延迟
  • 读 config 看 intermediate_size
先想 10 秒：CorpAssist 换 Qwen2.5-32B 比 7B，FFN 哪部分算力涨最多？

FFN 与 Attention 均随 hidden 与 intermediate 涨；32B 的 hidden_size、num_hidden_layers、intermediate_size 均更大，单 token FLOPs 显著高于 7B。网关应配路由：简单 FAQ → 7B，复杂推理 → 32B。

Grouped-Query Attention

标准 MHA：Q、K、V 头数相同。MQA：K/V 仅 1 组头；GQA（Grouped-Query Attention）：K/V 头数少于 Q，多 Q 头共享一组 K/V。推理时 KV Cache 大小 ∝ K/V 头数，GQA 在显存与速度上显著优于 MHA，LLaMA-2/3、Qwen2、DeepSeek 等广泛采用。

MHA vs GQA（K/V 头数）
flowchart TB
  subgraph mha [MHA num_kv_heads = num_heads]
    Q1[Q heads x8]
    K1[K heads x8]
    V1[V heads x8]
  end
  subgraph gqa [GQA num_kv_heads = 2]
    Q2[Q heads x8]
    K2[K heads x2 shared]
    V2[V heads x2 shared]
  end

GQA：8 个 Q 头映射到 2 组 K/V，KV Cache 约为 MHA 的 1/4（此例）。

Attention 类型KV Cache（相对）典型场景
MHA 基准 1x 早期 GPT-3、部分 BERT
GQA 约 num_kv_heads/num_heads LLaMA-2 70B、Qwen2、DeepSeek-V2
MQA 最小 部分推理优化模型；表达力可能降

config.json 中 GQA 字段

json 
{
  "model_type": "qwen2",
  "hidden_size": 3584,
  "num_attention_heads": 28,
  "num_key_value_heads": 4,
  "num_hidden_layers": 28,
  "intermediate_size": 18944,
  "max_position_embeddings": 32768,
  "rms_norm_eps": 1e-6,
  "hidden_act": "silu"
}

num_key_value_heads: 4 且 num_attention_heads: 28 → GQA，每组 KV 被 7 个 Q 头共享。CorpAssist 长对话占显存主因之一是 KV Cache；选 GQA 模型同等上下文更省显存（见 KV Cache 与 Flash Attention）。

开源权重结构速读

拿到 Hugging Face 模型目录，优先读 config.json + tokenizer_config.json。下列字段决定 CorpAssist 路由与容量规划：

字段含义工程影响
num_hidden_layers Decoder 层数 深度 ↑ → 延迟 ↑、能力 ↑
hidden_size 隐藏维 与参数量、KV 维数正相关
num_attention_heads Q 头数 与 head_dim = hidden/heads 相关
num_key_value_heads KV 头数（GQA） 推理显存关键
max_position_embeddings 训练上下文上限 编排层 prompt 预算
vocab_size 词表大小 中文模型常扩词表

三族对照（典型开源版本，概念级）

text 
| 系列      | 骨干特点                          | CorpAssist 常见用途        |
|-----------|-----------------------------------|----------------------------|
| LLaMA 3   | RMSNorm+SwiGLU+GQA+RoPE           | 私有化英文/通用基座          |
| Qwen2.5   | 同上，中文词表与长上下文优化        | 国内厂商 API、私有化中文助手 |
| DeepSeek  | Dense 或 MoE；V2/V3 强 GQA+MLA 等   | 高性价比 API、代码/推理场景  |

DeepSeek-V2/V3 等还引入 MLA（Multi-head Latent Attention） 等压缩 KV 的技术，面试可答「进一步减 KV Cache、提长上下文效率」；细节属 advanced，应用岗知道对 p95 与显存有利即可。

如何快速估算 7B vs 32B 在线并发？
粗算：参数量 × 精度字节（FP16 约 2B）+ KV Cache（与层数、hidden、seq、batch 相关）。同等 GPU，32B 并发远低于 7B。CorpAssist 网关按模型设 max_concurrent 与队列，避免 p95 抖动。

选型面试答题框架

CorpAssist 模型选型面试推荐 S-C-T-L-M 五段式（场景 → 约束 → 技术对照 → _latency → 决策）：

  1. Scenario：S1 制度 RAG 问答 vs S3 Agent 工具 vs 代码助手
  2. Constraints：p95 目标、数据出域、中文、上下文长度、预算
  3. Compare：7B/14B/32B 或 Qwen vs DeepSeek；GQA、上下文窗口
  4. Latency：TTFT/ITL、KV Cache、批大小、是否 Flash Attention
  5. Move：默认路由 + 升级路径 + 评测 golden set 验证

追加题：GQA 为什么能降显存？

推理 KV Cache 存储每层每 token 的 K/V 向量。GQA 减少 K/V 头数，Cache 体积按比例下降；Q 头仍多，表达力损失小于 MQA。结合 Flash Attention 与量化，可撑更长上下文或更大 batch。

CorpAssist 场景推荐模型档关键 config 字段
在线 FAQ Qwen2.5-7B-Instruct（GQA） max_position_embeddings ≥ 32k
多段交叉推理 Qwen2.5-32B 或 DeepSeek-V3 num_hidden_layers、intermediate_size
代码工具 Agent DeepSeek-Coder 系列 vocab 与工具 JSON 格式对齐

本章结论

  • RMSNorm + SwiGLU：开源 Decoder 标配，影响参数量与 FFN 算力。
  • GQA：减 KV 头，降 Cache 显存，LLaMA/Qwen/DeepSeek 广泛采用。
  • config 速读：layers、hidden、heads、kv_heads、max_position 决定容量与 SLA。
  • 选型框架：场景-约束-对照-延迟-决策，结合网关路由与 golden set。
