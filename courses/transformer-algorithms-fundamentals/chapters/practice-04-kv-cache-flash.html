为什么要学本章

CorpAssist 网关监控的 p95 延迟 与 GPU 显存告警，多半来自 Decoder-only 自回归推理：前缀 prefill 算整段注意力，decode 步 若不用 KV Cache 会重复计算历史 K/V；长上下文还会触发显存 OOM。Flash Attention 等 IO 优化则决定「同样 32k 上下文能否上线」。

本章先记住 4 件事
  • Prefill vs Decode：首包前算全前缀；之后每 token 主要算新 query 对历史 K/V。
  • KV Cache：缓存每层每 token 的 K/V，避免 decode 重算，换显存换时间。
  • 显存 ∝ 层×seq×hidden×batch；GQA 减 KV 头（见 架构对照）。
  • Flash Attention：减少 HBM 读写，加速 prefill/decode，非改变数学结果。

推理时 KV 复用

CorpAssist · 流式制度问答 用户问题 + 6 段 RAG 引用共 4k token prefill。模型输出 200 token 答案时，第 50 个输出 token 的注意力需访问全部 4k+49 个历史位置的 K/V。若无 KV Cache，每步重算整段 Self-Attention，ITL 会线性恶化——用户感觉「越生成越卡」。

Self-Attention 中，对位置 t 的 query q_t 需与所有 k_1…k_t 做点积。自回归 decode 时 t 逐步增大；历史 token 的 K/V 在先前步已算过，存入 Cache 后仅计算新 token 的 Q/K/V 并 append。

Decode 步 KV Cache 复用
sequenceDiagram
  participant S as 序列状态
  participant C as KV Cache
  participant A as Attention
  Note over S,C: Prefill 阶段
  S->>A: 整段 prefix tokens
  A->>C: 写入每层 K/V
  Note over S,C: Decode 第 t 步
  S->>A: 新 token x_t
  A->>C: 读取 K/V 1..t-1
  A->>C: append K_t V_t
  A->>S: 输出 hidden_t

Prefill 一次写入长前缀 Cache；Decode 每步只增量计算，Attention 读全 Cache。

无 Cache vs 有 Cache（伪代码）

python 
# 简化：单层单头 decode 一步
def decode_step_no_cache(model, all_token_ids):
    # 每步重算 1..t 全部 K/V — O(t) 重复计算
    hidden = model.forward_full_sequence(all_token_ids)
    return hidden[-1]

def decode_step_with_cache(model, new_token_id, kv_cache):
    # 仅对新 token 算 Q/K/V，与 cache 中历史 K/V 做 attention
    q, k, v = model.project_new_token(new_token_id)
    kv_cache.append(k, v)
    out = attention(q, kv_cache.all_k, kv_cache.all_v)
    return out
先想 10 秒：Chat 多轮对话为何 KV Cache 会「累积变长」？

每轮 user/assistant 消息 append 到上下文，有效 seq 长度增长，Cache 按当前全长存储。编排层需截断历史或摘要，否则 p95 与 OOM 风险上升——这是应用层上下文管理，不是模型 API 自动解决。

显存与延迟权衡

KV Cache 显存粗算（每层）：2 × batch × seq_len × num_kv_heads × head_dim × dtype_bytes。乘层数 L 得总量。GQA 减小 num_kv_heads；INT8/FP8 KV 量化可再减半（略损精度）。

因素对 KV Cache 显存对 p95 / TTFT / ITL
上下文 seq 变长 线性增 TTFT（prefill）↑；decode 每步 attention 也 ↑
batch / 并发 线性增 吞吐↑但单请求可能排队，p95 尾部变差
层数、hidden 线性增 7B→32B 单步算力↑
GQA KV 部分按头数比下降 同等显存可撑更长 ctx 或更大 batch
关闭 Cache（调试） 显存低 延迟爆炸，仅实验
CorpAssist 踩坑
  • Agent 十轮对话不截断 → Cache 10k+ token
  • RAG 塞 20 段引用 → prefill 秒级 TTFT
  • 高峰 batch 拉满 → p95 尾部超时
治理手段
  • 编排层 Token 预算与滑动窗口
  • 检索 top-k 与片段长度上限
  • 网关并发队列 + 小模型降级路由

Flash Attention 直觉

标准 Attention 实现需物化 n×n 注意力分数矩阵，读写 HBM 成为瓶颈。Flash Attention 通过分块（tiling）在 SRAM 内完成 softmax 与加权，避免完整矩阵落盘，数学等价于精确 Attention（在 fp 误差内），显著加速 prefill 与长序列。

Flash Attention：分块避免完整 n×n 落 HBM
flowchart LR
  Q[Q 分块] --> SRAM[GPU SRAM 小块]
  K[K 分块] --> SRAM
  V[V 分块] --> SRAM
  SRAM --> ACC[在线 softmax 累加]
  ACC --> OUT[输出 O 无需存全 S]

应用岗只需知：启用 FA2 通常降延迟、降显存峰值；具体 CUDA 实现属 mustNotCover 边界。

推理框架（vLLM、TensorRT-LLM、SGLang 等）默认集成 Flash Attention 或同类 kernel。CorpAssist 私有化部署检查清单：

  • GPU 架构是否支持（Ampere+ 常见）
  • PyTorch / CUDA / flash-attn 版本匹配
  • 长上下文 prefill 是否走 FA（否则 TTFT 尖刺）
Flash Attention 与 KV Cache 关系？
正交优化：KV Cache 减重复计算；Flash Attention 让每一步 attention 算得更快、更省显存峰值。二者同时使用是生产标配。
PagedAttention（vLLM）要背吗？
知道「KV Cache 分页管理，减碎片、提并发」即可。面试可答：提升 GPU 利用率、改善 p95 尾部排队。

对 CorpAssist p95 的影响

p95 延迟 是 SLA 常用分位：95% 请求在该时间内完成。LLM 场景拆为 TTFT（首 token）与 总时长（含生成）。优化映射表：

现象可能原因算法/工程手段
TTFT p95 高 长 RAG 前缀 prefill 减 top-k、Flash Attention、speculative 不适用 prefill 主因
ITL p95 高 decode 算力、batch 争抢 KV Cache、GQA、量化权重、限并发
显存 OOM ctx×batch 过大 截断上下文、KV INT8、换 GQA 模型、PagedAttention
尾部 p95 尖刺 队列排队 多副本、优先级队列、小请求优先

面试题：如何把一个 TTFT p95=4s 的 RAG 服务降到 2s？
  1. 度量 prompt token 分布，砍超长引用与历史
  2. 确认推理栈启用 Flash Attention / 合适 batch
  3. 检索侧精排减 chunk 数，或摘要压缩引用
  4. 简单 query 路由 7B，复杂再 32B
  5. 监控验证 p95，而非只看均值
  • 一次性处理整段 prefix token
  • 主导 TTFT；RAG 引用越多越慢
  • Flash Attention 收益最明显
  • 每步仅新 token 参与 Q/K/V 计算
  • 主导 ITL；依赖 KV Cache
  • 输出越长，总时长线性增

本章结论

  • KV 复用：decode 步缓存历史 K/V，避免重复 attention 计算。
  • 显存权衡：Cache 随 seq、层、batch、KV 头数增长；GQA/量化缓解。
  • Flash Attention：分块精确 attention，降 HBM IO，加速 prefill/decode。
  • p95：TTFT 看 prefill 与 prompt；ITL 看 decode 与并发；编排层可截断上下文。
