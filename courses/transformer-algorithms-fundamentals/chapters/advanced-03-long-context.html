滑动窗口注意力

全量 Self-Attention 对每个 token 关注全部历史，复杂度 O(n²)，KV Cache 显存也随 n 线性增长。Mistral 等模型采用滑动窗口：每个 token 只关注前 w 个位置（如 w=4096），更远历史被丢弃，计算与缓存有界。

CorpAssist · 超长会话 智能客服多轮对话超过 50 轮时，即使用 128K 模型，窗口截断 可能导致早期「已核实订单号」被遗忘。应用层需：摘要压缩、结构化会话状态、或 RAG 拉回复关键事实——不能假设模型「天然记得一切」。
全量 Attention vs 滑动窗口（w=4 示意）
flowchart TB
  subgraph full["全量 O(n²)"]
    T1[每个 token 看全部历史]
  end
  subgraph sw["滑动窗口 O(n·w)"]
    T2[每个 token 仅看前 w 个]
    T2 --> DROP[更早 token 不参与注意力]

窗口外 token 仍存在于序列，但不参与当前注意力计算；与「上下文长度」产品宣传需区分。

方案复杂度KV Cache远程依赖局限
全量 Attention O(n²) O(n) 任意距离 长序列贵
滑动窗口 O(n·w) O(w) 有界 仅窗口内 超长依赖需外部记忆
稀疏全局 token 混合 中等 部分远程 架构专用
误区
  • 「窗口 128K = 每 token 都看 128K」——可能含滑动/稀疏策略
  • 「窗口模型不需要 RAG」——窗口外知识仍需检索
面试正答
  • 问清架构：全量 / 滑动 / 混合
  • CorpAssist 用会话摘要 + RAG 补窗口外事实

Ring/Chunked Attention 概念

当单卡装不下长序列 KV 时，分块注意力把序列切成 chunk，每块在本地算 Attention；块间通过 Ring All-Reduce 等通信交换 KV，实现「逻辑全上下文、物理分布式」。这是推理/训练系统概念，应用岗知「存在此类多卡方案」即可，不需手写通信原语。

Chunked 直觉

text 
序列: [chunk0 | chunk1 | chunk2 | chunk3]
单卡: 先算 chunk1 局部 QK^T
跨块: 向相邻卡请求 chunk0 的 K/V → 拼接到注意力范围
Ring: 卡0→卡1→卡2→卡3→卡0 传递 KV 块，覆盖全历史

应用岗 takeaway: 私有化「超长文档一次推理」可能需多卡 + 此类框架
技术解决什么谁负责应用岗接口
Flash Attention 单卡算子级省显存 推理框架 看 TTFT 是否改善
Chunked Attention 序列切分跨卡 Infra / 平台 问清最大可推理长度与副本数
Ring Attention KV 块环状传递 训练/推理集群 不在应用 BFF 实现
KV Cache 分页 显存碎片与共享 vLLM 等 并发客服 QPS 能力
先想 10 秒：Flash Attention 与 Ring Attention 解决的是同一问题吗？

不同层面。Flash 是单卡 kernel 融合减少 HBM 读写；Ring 是多卡间 KV 传递以扩展可处理序列长度。CorpAssist 压测报告应分「单请求最长上下文」与「并发 QPS」两项。

YaRN/NTK 外推（应用视角）

模型训练在 8K 长度，产品需 32K～128K。NTK-aware 插值 调整 RoPE 频率基座，使旋转角在更长 pos 上不过快饱和；YaRN（Yet another RoPE extensioN）在低频/高频维度上分段插值，兼顾短程与长程。二者常在 推理配置 或 vLLM --rope-scaling 中启用，不改权重。

方法操作层典型效果风险
NTK scaling 调 RoPE base 零样本略延长可用长度 短题质量偶发波动
YaRN 分段频率插值 长文 QA 更稳 需匹配模型卡片参数
继续预训练扩展 训练（mustNotCover） 最强 非应用岗职责

RAG 与长窗口取舍

长上下文 与 RAG 是互补关系：前者扩大「一次能读多少」；后者控制「读什么、能否溯源」。CorpAssist 企业知识库（百万 chunk）在可预见未来仍依赖检索；长窗口适合「已精选的少文档全文」或「多轮会话未压缩历史」。

维度纯长窗口纯 RAG推荐组合
知识边界 易混入无关段落 Top-K 可控 检索 Top-K + 适度拼接
引用溯源 需额外段落标记 chunk_id 天然支持 RAG 主链路 + 引用字段
成本 ∝ 序列长度 ∝ K × chunk 长 动态 K + 摘要压缩
延迟 首 token 随长度升 检索 + 生成两阶段 并行检索与流式生成
面试金句 「不能替代索引」 「不能替代跨段推理」 「检索定证据，长窗补连贯」
CorpAssist 长文档问答推荐路径
flowchart LR
  Q[用户问题] --> RET[混合检索 Top-K]
  RET --> FILT[租户/权限过滤]
  FILT --> PACK[拼接 prompt ≤ 窗口预算]
  PACK --> LLM[长窗口 LLM 生成]
  LLM --> CITE[引用 chunk_id 回显]
  PACK -.->|可选| SUM[会话摘要进 prompt]

长窗口是打包预算，不是全库入口；权限过滤在检索层完成。

架构反例
  • 每晚把全库 embedding 删了改「128K 直读」
  • 不做 chunk 边界，指望模型在长 prompt 里自己找页码
  • 忽视 长上下文 外推未回归的长文档用例
架构正例
  • 检索决定证据集；prompt 预算内拼接；超长用 YaRN 配置
  • 客服会话：滑动窗口 + 结构化槽位 + 摘要
  • 评测：短题 + 长文档双 golden set

本章结论

  • 滑动窗口注意力：有界 w 降低算力与 KV；远程依赖需外部记忆或 RAG。
  • Ring/Chunked Attention：多卡分块交换 KV；平台能力，应用岗知边界即可。
  • YaRN/NTK 外推：推理侧 RoPE 缩放；须长文档实测，非万能。
  • RAG 与长窗口取舍：检索定证据，长窗补连贯；全库入 prompt 不可行。
