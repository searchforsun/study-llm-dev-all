本章先记住 4 件事
  • Self-Attention：Q、K、V 同源，均来自同一序列嵌入 X。
  • 瓶颈矩阵是 (n,n)：n 为序列长度；层数 L 线性放大，n 平方放大。
  • 显存 ∝ n²：训练存注意力分数；推理可用 Flash Attention 等 IO 优化。
  • 工程解法是控 n：RAG Top-K、摘要、滑动窗口——算法认知指导应用架构。

单头注意力矩阵运算

Self-Attention 即 Q、K、V 均由同一输入 X 投影：X ∈ ℝn×dmodel。单头时 dk=dv=dmodel（多头时见 下一章）。

下列推导默认 batch=1；n 为序列 Token 数。面试白板建议先写形状再填数字，避免矩阵乘维度对错。

形状追踪表

步骤运算输出形状含义
输入 X (n, d_model) n 个 Token，各 d_model 维
投影 Q=XWQ, K=XWK, V=XWV 各 (n, d_k) 同序列三种角色
打分 S = QKT/√dk (n, n) 每对位置相似度
权重 A = softmax(S) (n, n) 行随机矩阵
输出 O = AV (n, d_v) 更新后的序列表示

Self-Attention(X) = softmax(QKT / √dk) V，其中 Q=XWQ，K=XWK，V=XWV

矩阵视角：一次「全连接阅读」

把 A 想成 n 个 Query 各读一遍 n 个 Key 的软查表；O 的每一行是 V 的 n 行按 A 对应行权重混合。对 CorpAssist 制度问答，n 包含 system + 检索 chunk + 用户问题 + 已生成回答——所有 Token 两两可互看（Decoder 生成时另有因果 Mask）。

Self-Attention 矩阵乘法链
flowchart LR
  X["X (n×d)"] --> Q["Q (n×dₖ)"]
  X --> K["K (n×dₖ)"]
  X --> V["V (n×dᵥ)"]
  Q --> S["S=QKᵀ (n×n)"]
  K --> S
  S --> A["softmax → A (n×n)"]
  A --> O["O=A·V (n×dᵥ)"]
  V --> O

L 层堆叠时，每层重复；总计算量大致 ∝ L·n²·d。

向量化实现（批维 B）

python 
# X: (B, n, d_model) — B 条 CorpAssist 请求 batch
Q = X @ W_q          # (B, n, d_k)
K = X @ W_k
V = X @ W_v
scores = Q @ K.transpose(0, 2, 1) / math.sqrt(d_k)  # (B, n, n)
attn = softmax(scores, dim=-1)
out = attn @ V       # (B, n, d_v)

复杂度 O(n²) 含义

计算 QKT 需对 n×n 个元素各做 dk 次乘加 → 时间 O(n²·d)。存储中间 S 或 A 需 O(n²) 显存（训练时尤为明显）。层数 L 线性乘上，但长度 n 平方放大是长上下文的首要敌人。

长度翻倍代价示意

序列长度 n注意力矩阵元素 n²相对 2K 基线
2,048~4.2M1×
8,192~67M16×
32,768~1.07G256×
128,000~16.4G~4000×

这也解释为何「上下文窗口从 8K 提到 128K」不仅是营销：同等 batch 下算力与 GPU 显存 需求可差两个数量级以上。优化方向包括 Flash Attention（IO 感知）、稀疏注意力、KV Cache（推理）、长上下文技术。

prefill vs decode 两阶段（与延迟挂钩）

  • prefill：首次吃进整段 prompt，须算完整 n×n Attention → TTFT 敏感，延迟 ∝ n²。
  • decode：每步只新增 1 Token，配合 KV Cache 可复用历史 K/V → 单步代价 ∝ n，但步数 = 生成长度。

CorpAssist 网关若只优化 decode 而放任 32K RAG 前缀，用户仍会在「首字出现前」长时间等待——控 prompt n 比调 temperature 更影响 TTFT。

与 RNN O(n) 的对比（口述）

  • RNN：时间步串行 O(n)，每步隐状态维固定，难并行但内存 O(n)。
  • Transformer：层内并行，但 n×n 矩阵 O(n²) 时间与内存。
  • 权衡：GPU 时代选中 Transformer，但应用层必须控 n。
先想 10 秒：层数 L 从 32 增到 64，与 n 从 4K 增到 8K，哪个对延迟打击更大？

通常 n 翻倍打击更大：Attention 项 ∝ L·n²，n 翻倍带来 4× Attention 代价，而 L 翻倍仅 2×。CorpAssist 网关限流应优先监控输入 Token 总量，而非只数 API 调用次数。

与 CorpAssist 长文档代价

CorpAssist · 制度全文灌入 PM 要求「把 200 页《员工手册》PDF 整本塞进 prompt，让模型自己找」——单次 n≈120K Token，Attention 矩阵元素 ~14×10⁹ 量级，p95 延迟与账单同时爆炸。

工程对策（算法认知 → 应用决策）

增大 n（硬灌全文）
  • O(n²) 算力与显存暴涨
  • 无关段落稀释 Attention 有效信号
  • 商用 API 按 Token 计费线性上升
控制有效 n
  • RAG Top-K 只送 4～8 段 chunk
  • 超长纪要先摘要再问答
  • 网关 max_input_tokens 硬限 + 告警

懂 O(n²) 后，与 Java 后端同事沟通 SLA 时可以说清：不是「模型变慢了」，而是序列长度平方级放大；解法在架构（检索、摘要、限长）而非单纯升配 GPU。

限长策略清单（可直接写进网关配置说明）

  1. max_input_tokens 硬顶（如 8192），超限拒收或触发摘要。
  2. RAG top_k 与 max_chunk_tokens 双限。
  3. 多轮对话滑动窗口：保留 system + 最近 K 轮。
  4. 异步「长文深度阅读」任务，在线问答走短上下文。

以上策略的共同目标：压低有效 n，使 Self-Attention 的 n² 项可控——这比单纯购买更大窗口模型更符合 CorpAssist SLA。

手算迷你例题

面试白板常考 n=3、dk=2 的 Self-Attention。下列数字已四舍五入便于手算。

S = QKT / √dk，  A = softmax(S)，  O = AV

已知

text 
序列 3 个 Token（已含嵌入+位置），d_k = 2

Q = [[1, 0],
     [0, 1],
     [1, 1]]

K = [[1, 0],
     [0, 1],
     [1, 0]]

V = [[1, 0],
     [0, 1],
     [2, 2]]

sqrt(d_k) = sqrt(2) ≈ 1.414

步骤 1：S = QKT / √2

QKT 第 (0,0) 元 = 1×1+0×0 = 1；(0,1)=0；(0,2)=1。完整 S ≈

text 
S_raw = [[1, 0, 1],
         [0, 1, 1],
         [1, 1, 2]]

S = S_raw / 1.414 ≈ [[0.71, 0, 0.71],
                      [0, 0.71, 0.71],
                      [0.71, 0.71, 1.41]]

步骤 2：对第 0 行 softmax

exp(0.71)≈2.03, exp(0)≈1, exp(0.71)≈2.03 → 和≈5.06。A0≈[0.40, 0.20, 0.40]（行和=1）。

步骤 3：O 第 0 行 = A0 @ V

text 
O_0 = 0.40*[1,0] + 0.20*[0,1] + 0.40*[2,2]
    ≈ [0.40+0+0.80, 0+0.20+0.80]
    ≈ [1.20, 1.00]

完整手算应逐行完成；验收标准：每行 softmax 和为 1，O 形状 (3,2)。NumPy 实验章 用代码验证。

手算检查清单

  • QKT 维度是否为 (3,3)
  • 除以 √dk 后数值是否在 0～2 附近（非几十）
  • 每行 exp 和是否归一化为 1
  • O 行向量是否为 V 行的凸组合（权重非负且和 1）

与 O(n²) 对照：n=3 时矩阵仅 9 元；n=8192 时约 6700 万元素——量级直觉比背公式更重要。

写到 S、一行 softmax、O 一行即可；强调形状 (n,n) 与行和为 1。

用 n² 表估算 prompt 从 8K→32K 时 Attention 代价倍数；指导 CorpAssist 限长策略。

本章结论

每条对应上一节 h3，用于评审前 30 秒复述。

  • 矩阵运算：Self-Attention 核心 (n,n) 权重矩阵；形状追踪是手写基本功。
