本章先记住 4 件事
  • 多头 = h 个独立子空间 各做 Scaled Dot-Product，再拼接投影。
  • Padding Mask：batch 对齐时屏蔽 pad Token，防止 attend 到无意义位置。
  • Causal Mask：解码器不能偷看未来 Token，保证自回归合法。
  • Cross-Attention：Q 来自解码器，K/V 来自编码器或外部 memory。

多头并行子空间

单头 Attention 只有一套 Q/K/V。多头注意力（MHA） 把 dmodel 切成 h 份，每份维度 dk = dmodel/h，各自独立算 Attention，最后拼接再乘 WO 投影回 dmodel。

多头不改变渐近 O(n²)——每个头仍算 n×n 权重，只是 d_k 变小、h 路并行；总 FLOPs 与单头同量级，主要增益在表达多样性。

公式骨架

headi = Attention(Qi, Ki, Vi)，  i = 1…h
MultiHead = Concat(head1, …, headh) WO

CorpAssist 7B 级常见：h=32，dmodel=4096，dk=dv=128（每头 128 维）。

  • 切分：dmodel 均分 h 份，每头独立投影 Q/K/V，互不共享参数。
  • 拼接：h 个头输出沿特征维 concat 后，经 WO 线性映射回 dmodel。
  • 总 FLOPs：与单头全维 Attention 同量级；收益在表达多样性而非降复杂度。

为何要多头？

  • 多子空间并行：不同头可学习关注不同模式（句法、共指、数字邻近等）。
  • 稳定训练：较多个小头比单一大头更易优化（类似集成）。
  • 算力折中：总参数量与单头全维相近，但表达更丰富。
多头 Attention 数据流
flowchart TB
  X[X d_model] --> SPL[切分 h 头]
  SPL --> H1[Head 1 Attn]
  SPL --> H2[Head 2 Attn]
  SPL --> Hh[Head h Attn]
  H1 --> CAT[Concat]
  H2 --> CAT
  Hh --> CAT
  CAT --> WO["W_O 投影 → d_model"]

推理引擎对多头做融合 kernel；应用岗只需知 h 与 d_model 影响显存与速度。

单头 vs 多头（表达力直觉）

单头 Attention 只有一套 Q/K/V 投影；多头 相当于并行跑 h 个「专家读者」，有的头偏向句法邻近，有的偏向共指、数字对齐等——最后拼接由 WO 学会如何混合。CorpAssist 评测 badcase 时，个别头可呈现对「制度编号」「金额」列的集中权重（研究向，非生产依赖）。

  • 头数 h 增大 → 每头 dk 变小（dmodel 固定时），单头容量下降但多样性上升。
  • GQA/MQA（practice 章）进一步在 K/V 头间共享，降 KV Cache 显存。
  • 应用岗读 config：num_attention_heads 与 num_key_value_heads 可能不同。
参数典型值（7B）含义
h32注意力头数
d_model4096模型隐藏维
d_k128每头 Q/K 维
GQA见 practice 章多 Q 头共享 K/V 降显存

Padding Mask

训练 CorpAssist 意图分类小模型时，batch 内短句用 <pad> 对齐是常规操作。Padding Mask 确保模型不会把 pad 当成真实语义去 attend——否则「报销」可能错误地关注到另一句的 padding 列，拉低验证集准确率。

Hugging Face attention_mask 中 1 表示真实 Token、0 表示 pad；传入模型前须与 input_ids 等长，Java 侧拼 batch 时由网关统一生成。

批处理多条 CorpAssist 请求时，序列长短不一，短句右侧用 <pad> 填充到统一长度。Padding Mask 在 softmax 前把pad 位置分数置为 −∞，使权重为 0，避免模型从 padding 读取噪声。

Mask 矩阵示意（batch=2）

text 
序列 A: [我, 要, 报销]           → 长度 3
序列 B: [制度, 问答, pad, pad]   → 长度 4（pad 为填充）

对序列 B，任何 Query 对 pad 列的 score 设为 -inf
有效 Token 只 attend 到前 2 个真实词

工程注意

  • 训练与推理必须一致：漏 Mask 会导致 pad 污染表示，批推理准确率抖动。
  • 自研 Java 网关若做 batch 拼 prompt，须同步传递 attention_mask 给推理服务（与 Hugging Face 约定一致）。
  • 单条在线请求常无 pad，Mask 全 1；但微调数据管道仍要处理。
  • 动态 padding（按 batch 内最长句）可省算力，但须每 batch 重算 mask 张量。
  • 导出 ONNX/TensorRT 时确认 mask 作为输入传入，勿写死全 1。
先想 10 秒：Padding Mask 加在 softmax 之前还是之后？

之前：对无效 (i,j) 位置将 score 设为 −∞，softmax 后权重为 0。若在 softmax 之后乘 0 虽也可行，但数值上常在 softmax 前 mask 以避免 exp 溢出并实现标准 fused kernel。

Causal Mask（自回归）

CorpAssist 流式对话里，用户看到 assistant 消息逐字出现，本质是模型每步只多生成 1 个 Token。因果 Mask 在训练阶段就强制模型习惯「只看过去」——与推理时可见上下文一致，避免 train/serve skew。

推理框架在 KV Cache 命中时仍须对新 Token 行应用 Causal Mask，使其不能 attend 到未生成的未来位置。

去掉 Causal Mask 的「捷径」训练在上线后几乎必然产生泄漏式胡话，不可用于生产微调。

SFT 数据构造时，labels 仍须与 Causal 可见范围一致，仅对 assistant 段计算 loss。

GPT 类 因果 Mask（Causal / Look-ahead Mask） 保证位置 i 只能看见 ≤i 的 Token，不能偷看未来。否则训练时标签泄漏，推理时无法逐 Token 生成。

下三角可见矩阵

text 
n=4 时允许 attend 位置（1=可见，0=屏蔽上三角）

     t0 t1 t2 t3
t0 [ 1  0  0  0 ]
t1 [ 1  1  0  0 ]
t2 [ 1  1  1  0 ]
t3 [ 1  1  1  1 ]

CorpAssist 流式生成：每新 Token 只能依赖已输出前缀
自回归生成与 Causal Mask
flowchart LR
  T0[t0 我] --> T1[t1 要]
  T1 --> T2[t2 报销]
  T2 --> T3[t3 ?]
  T3 -.->|不可见| T4[未来 Token]

生成 t3 时 Mask 屏蔽 t4 及之后；与 Chat API 逐 chunk 输出一致。

实现提示：mask 加在 logits 上

python 
# scores: (n, n)；mask 为 0/1 或 bool
scores = scores.masked_fill(causal_mask == 0, float('-inf'))
weights = softmax(scores, dim=-1)  # 被 mask 位置权重为 0

与 BERT 双向对比

Decoder（GPT/Qwen）
  • Causal Mask，单向上下文
  • 适合生成、对话续写
  • CorpAssist 主对话链路
Encoder（BERT 向量）
  • 无 Causal Mask，双向
  • 适合分类、Embedding
  • CorpAssist RAG 召回编码

Cross-Attention 场景

当解码器生成目标语言每个词时，Cross-Attention 让 Q 来自「我已生成部分」，K/V 来自「源语言编码」——经典机器翻译里生成英文 "subsidy" 时对齐中文「补贴」位置。CorpAssist 若用 T5 做制度中英对照，栈内仍有 Cross-Attn；Qwen 对话则把双语材料拼进同一上下文用 Self-Attn 完成。

Cross-Attention（交叉注意力） 中 Query 来自一方序列（常为 Decoder 当前状态），Key/Value 来自另一方（Encoder 输出或外部 memory）。这是 Seq2Seq 翻译与部分多模态架构的核心。

典型场景

场景Q 来源K/V 来源CorpAssist 映射
经典翻译 Decoder 目标语言已生成部分 源语言 Encoder 输出 多语言制度翻译（T5 类）
Self-Attention 同序列 X 同序列 X GPT 对话上下文互看
RAG「软」对照 生成中的回答 Token 上下文中的检索 chunk 无独立 Cross-Attn 层时由 Self-Attn 在同一上下文内完成
多模态 文本 Token 图像 patch 编码 未来 CorpAssist 图文工单（扩展）

Decoder-only 为何常说「没有 Cross-Attention」

CorpAssist 主力 Qwen/GPT 把检索 chunk 与问题拼成一条上下文，用 Self-Attention 统一处理——工程上等价于「让模型自己找对齐」，无需独立 Encoder 栈。理解 Cross-Attention 仍有助于读 T5/BART 权重与面试「Seq2Seq 与 GPT 差异」。

本章结论

每条对应上一节 h3，用于评审前 30 秒复述。

  • 多头：h 个子空间并行 Attention，Concat 后 WO；丰富表达且易训练。
  • Padding Mask：批处理屏蔽 pad；score 置 −∞ 再 softmax。
  • Causal Mask：下三角可见；自回归生成与 GPT 对话必备。
  • Cross-Attention：Q 与 K/V 异源；Decoder-only 用拼接上下文 + Self-Attn 替代。
