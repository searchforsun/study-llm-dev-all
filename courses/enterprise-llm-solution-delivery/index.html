为什么要学本章

ToB 大模型项目若跳过 delivery discovery，常见结果是 PoC 演示漂亮、试点却卡在数据权限或验收指标对不上。本章用 CorpAssist 企业知识助手立项场景，把「谁说了算、做什么、不做什么、会踩什么坑」在写方案前钉死。

学完你能
  • 列出业务 / IT / 安全三方 stakeholder map 及各自成功指标
  • 输出带 Out of Scope 的 mvp scope 与 requirement doc 骨架
  • 维护 risk register Top5，并衔接 双栈方案设计文档
  • 完成 interview 纪要与签字确认流程
本章先记住 3 件事
  • 访谈不是聊天：要有议题、纪要模板与「决策人签字」的验收假设。
  • 边界写「不做」：scope boundary 比功能列表更能防范围蔓延。
  • 风险前置：数据合规、幻觉、集成复杂度要在 PoC 前登记，而非验收时才暴露。

访谈

CorpAssist · 制造业知识库立项（15 分钟） 业务希望「制度问答 + 引用溯源」；IT 要求接入现有 OA 与 SSO；安全要求文档不出域、审计留痕。你需要在一次 client interview 里同时听到三类诉求，而不是只问「要什么功能」。

企业级 discovery 通常分三轮：业务痛点 → 数据与系统集成 → 合规与运维。每轮产出写入同一份需求纪要，避免「口头答应、文档没有」。

Stakeholder Map：三类角色与关注点

角色典型部门关注点成功指标决策权重
业务方 HR、IT 服务台、法务 场景优先级、回答准确率、响应速度 节省人天、首次解决率、用户满意度 高（场景定义）
IT 方 信息化部、基础设施 SSO 集成、API 网关、现有向量库/ES、发布窗口 系统稳定、无安全事件、可运维 高（技术选型）
安全 / 合规 数据安全、法务 数据分级、不出域、内容安全、工具调用白名单 审计日志完整、0 泄露事件 一票否决

访谈前应准备一份 stakeholder map，标注每个人的角色、决策权重与需要 cover 的问题。不要在访谈中临时问「您负责哪块」。

不推荐
  • 只 demo 通义聊天，不记录数据源与权限
  • 成功指标只有「领导满意」
  • 未确认谁能在合同上签字验收
  • 访谈完不输出纪要，口头共识转头就忘
推荐
  • 每场景对应可测 KPI（引用率、坏例率、成本/千次）
  • 访谈当场确认 MVP 只覆盖 1 个主场景
  • 纪要 48 小时内回传客户确认并签字
  • 每次访谈产出直接入需求文档骨架

访谈的技巧：多问「为什么」。客户说「我们想要 AI 客服」——追问「现在客服的痛点是什么？每天多少工单？多少转人工？」把模糊需求转化为可量化指标，是 requirement doc 的第一步。

先想 10 秒：若客户坚持「先全量接入 10 年制度库」，你第一句话回应什么？

应回到 MVP 数据子集 + 权限试点：全量接入是二期，一期用可审计的小集合验证解析、ACL 与指标。原因：全量 10 年数据的解析成本与标注成本远超 MVP 预算，且无法验证核心链路。

边界

scope boundary 要在方案阶段写清「不做清单」，并与合同附件一致。CorpAssist 一期常见边界如下。

在 MVP 内明确不做（Out of Scope）理由替代/过渡方案
单部门制度 RAG + 引用 全集团 20+ 系统实时写操作 NS2 工具链与审批未就绪 二期 SOW 覆盖 Agent 改单
DashScope 公有云 API 客户机房内全量微调 工期与数据标注不在一期 私有化 PoC 可验证推理能力
评测集 50~100 条 Golden 「100% 不出错」口头承诺 需 SLA + 回归门禁，见验收章 合同中写引用率与拒答条款
Web 门户接入 移动端 App / 钉钉小程序 移动端适配一期不做 二期可复用同一 OpenAPI

边界冻结后才进入 双栈方案设计文档，避免架构随需求漂移。

写 Out of Scope 的黄金法则：每写一条「不做」，附上理由与替代方案。客户看到「为什么不做」和「以后怎么做」，更容易接受边界。

业务想要的

「什么都问、越快越好」——需收敛到 1 个场景 + 可测 KPI。

交付方能做的

双栈 PoC、试点、评测报告——需写清人天与依赖。

合同能签的

指标、验收方法、除外责任——与 PoC 演示脚本一致。

先想 10 秒：客户说「先不做 ACL，上线后再加」，你怎么回应？

拒绝。ACL 是安全基线，不是功能特性。没有 ACL 的系统不能上线试点的原因是：一旦出现跨部门文档泄露，项目可能被终止。建议把 ACL 列为 P0 技术要求，写入 risk register 高亮。

风险

大模型交付的典型风险不是「模型不够聪明」，而是数据、合规与集成。建议在 risk register 中至少登记以下类别。

风险触发信号缓解措施责任人优先级
脏 PDF / 扫描件解析差 召回率上限卡在 60% 解析链路 + 人工抽检；见 S1 生产坑 数据工程师 P1
ACL 未在检索层过滤 试点出现越权片段 检索前 filter；安全评审 架构师 P0
幻觉导致错误制度解读 无引用或错误引用 强制 citation；拒答策略 算法 / 产品 P1
PoC 指标与合同脱节 演示 90% 、验收 70% PoC 脚本 = 合同 UAT 子集 项目经理 P0
过度承诺无 SLA 口头「7×24 秒回」 写清 P95、可用性、除外 商务 + 架构 P0
厂商模型变更/下线 通义模型升级后效果变化 模型版本锁定 + 回归集 算法 P2

概率 \ 影响低中高极高
高接受+监控缓解缓解规避

低接受接受监控缓解
客户说「先不上评测，上线后再优化」怎么办？
可接受分阶段，但 MVP 必须保留最小 Golden Set 与发版回归，否则无法兑现 NS1 飞轮。应在风险登记中标注为「验收争议高风险」。建议 PoC 阶段就建立一个 50 条的最小 golden set。
discovery 和 PoC 之间要不要签合同？
建议至少 signed 的 discovery 纪要或 SOW 附件：范围、数据责任、知识产权与保密。PoC 合同指标应直接引用纪要中的 KPI 表述。如果客户不愿签，至少发邮件确认纪要内容。
访谈时业务方与 IT 方就架构选择产生分歧，怎么办？
先记录双方诉求，不要当场选边。在纪要中单独列出分歧点，标注「待决策」。建议办法：由 CIO/CTO 做最终决策，交付团队提供选项的 TCO 对比。分歧本身也是 risk register 中的一项风险。

本章结论

三条对应上文三节：访谈、边界、风险——做好这三步，项目已经成功了一半。

  • 访谈：三方 stakeholder、可量化 KPI、48h 纪要确认——避免只 demo 不落地。问题驱动而非方案驱动。
  • 边界：MVP + Out of Scope 书面化，与 PoC / 合同同一套指标语言。Out of Scope 附理由与替代方案。
  • 风险：数据、ACL、幻觉、指标脱节、过度承诺进 risk register，并指定责任人。P0 风险必须在 PoC 前缓解。

Architecture Decision Records (ADR) 模板

markdown
# ADR-001: 选择双栈架构（Spring BFF + Python RAG）

## 背景
CorpAssist 需要同时满足：企业鉴权/合规（Java 生态优势）
与 AI 模型编排（Python 生态优势）。

## 决策
采用双栈：Spring Boot BFF + Python FastAPI RAG，
通过 OpenAPI REST 通信。

## 备选方案
1. 全 Python 单体 — 鉴权与审计不满足企业要求

先想 10 秒：为什么 BFF 不能直接调模型 API 跳过 RAG 服务？

因为 ACL 过滤、检索引用生成、Prompt 组装都在 Python RAG 服务中。BFF 直接调模型会绕过 ACL 与引用链路，导致合规风险和不一致的回答格式。

客户要求「接口文档放二期」？
PoC 可简化为 3 路径 Markdown 附件，但字段名与验收指标不得口头漂移，否则 验收与 SLA 无法对齐。建议将核心 API 的字段定义写入合同附件。

需要。Python RAG 服务在内网也应做服务间鉴权——推荐 mTLS 或 service token（BFF 签发）。不鉴权的内网接口是常见安全隐患。

OpenAPI 变更需要双方架构师审批：① 在 Git 上发起 PR ② 标注 breaking/ non-breaking ③ Java + Python 架构师 approve ④ 更新契约测试 ⑤ 同步更新 solution doc。

GPU 估算方法论

gpu estimate 公式：所需显存 = 模型参数量 × 精度系数 × 并发度 × 安全系数。示例：

python
# GPU 估算示例 — 7B 模型，INT4 量化，8 并发
model_params_b = 7        # 7B 模型
precision_bytes = 0.5     # INT4 = 0.5 bytes/param
concurrency = 8
safety_factor = 1.3

vram_per_model = model_params_b * precision_bytes  # 3.5 GB
vram_total = vram_per_model * concurrency * safety_factor
print(f"估算显存: {vram_total:.1f} GB")
# 输出：估算显存: 36.4 GB → 推荐 1 × A100-40G 或 2 × L20

FAQ：供应商锁定风险评估方法？

从三个维度评估：模型迁移成本（Prompt 是否跨模型兼容/需重新评测）、平台迁移成本（工作流/知识库是否可导出）、数据迁移成本（向量库/日志是否标准格式）。每项按低/中/高评分，总分决定锁定风险等级。

多家厂商同时 PoC 的资源分配？

建议最多 3 家并行。资源分配：第 1 家 40%（基准）、第 2 家 35%（对比）、第 3 家 25%（参考）。避免超过 3 家——每家的适配、评测、沟通成本都很高。PoC 周期统一为 2 周。

厂商选型不是一次性决策。CorpAssist 运维期需要与模型供应商保持良好合作关系，包括：定期 Review SLA 达标情况、提前沟通模型升级计划、反馈生产问题。建议：每季度与厂商开一次运营回顾会，汇总 Token 消耗、延迟趋势、故障记录。

Token 成本估算

token cost = 月请求量 ×（输入+输出 Token/次）× 单价。CorpAssist 制度问答可按 2k 输入 + 500 输出估算。但注意：检索注入的 chunk 文本也计入输入 token。

python
# Token 成本计算器 — CorpAssist 场景
def estimate_token_cost(
    monthly_queries: int,
    input_tokens_per_query: int,
    output_tokens_per_query: int,
    price_per_1k_input: float,
    price_per_1k_output: float,
) -> dict:
    monthly_input_tokens = monthly_queries * input_tokens_per_query
    monthly_output_tokens = monthly_queries * output_tokens_per_query

    input_cost = monthly_input_tokens / 1000 * price_per_1k_input
    output_cost = monthly_output_tokens / 1000 * price_per_1k_output

    return {
        "monthly_queries": monthly_queries,
        "monthly_input_tokens": monthly_input_tokens,
        "monthly_output_tokens": monthly_output_tokens,
        "monthly_token_cost": input_cost + output_cost,
        "yearly_token_cost": (input_cost + output_cost) * 12,
    }

# S1 RAG 场景估算（qwen-plus）
s1 = estimate_token_cost(
    monthly_queries=50000,
    input_tokens_per_query=2500,   # 提问 + 检索注入
    output_tokens_per_query=500,
    price_per_1k_input=0.008,
    price_per_1k_output=0.008,
)
print(f"S1 月 Token 成本: ¥{s1['monthly_token_cost']:.2f}")
print(f"S1 年 Token 成本: ¥{s1['yearly_token_cost']:.2f}")

# S2 客服场景估算（qwen-turbo，更便宜）
s2 = estimate_token_cost(
    monthly_queries=30000,
    input_tokens_per_query=1500,   # 会话历史 + 意图识别
    output_tokens_per_query=300,
    price_per_1k_input=0.002,
    price_per_1k_output=0.002,
)
print(f"S2 月 Token 成本: ¥{s2['monthly_token_cost']:.2f}")
print(f"S2 年 Token 成本: ¥{s2['yearly_token_cost']:.2f}")

成本控制三板斧

交付过程中，Token 成本和 Infra 成本容易超支。三种主要的降本措施：

  1. 模型分层：S1 用 qwen-plus（高精度），S2 用 qwen-turbo（低成本），S3 Agent 限步数。按场景分配模型可降低 40-60% Token 成本。
  2. 缓存策略：高频问题（如「年假几天」）缓存答案，缓存命中率可达 20-30%，显著降低重复 Token 消耗。
  3. 用量监控：BFF 层记录每次请求的 Token 消耗，按场景/部门生成成本报表。设置月度预算告警，超额自动通知。

def monthly_cost_report(usage_log: list) -> dict:
    """从 BFF usage log 生成月度成本报表"""
    scene_costs = {}
    for record in usage_log:
        scene = record["scene_id"]
        tokens = record["total_tokens"]
        model = record["model"]
        price = MODEL_PRICES[model]
        cost = tokens / 1000 * price
        scene_costs[scene] = scene_costs.get(scene, 0) + cost
    return scene_costs

# PoC 指标计算 — 演示前运行
import json

def evaluate_poc(golden_set_path: str, responses_path: str) -> dict:
    with open(golden_set_path) as f:
        golden = json.load(f)
    with open(responses_path) as f:
        responses = json.load(f)

    correct = 0
    cite_count = 0
    for g, r in zip(golden, responses):
        if g["expected_answer"] in r["answer"]:
            correct += 1
        if r.get("citations") and len(r["citations"]) > 0:
            cite_count += 1

    total = len(golden)
    return {
        "accuracy": correct / total,
        "cite_rate": cite_count / total,
        "total_queries": total
    }

# 输出：{'accuracy': 0.84, 'cite_rate': 0.96, 'total_queries': 50}

灰度发布

CorpAssist · HR 部门试点（5 周） PoC 通过后，HR 部门 200 人分三批灰度：第 1 周 10%「超级用户」内测；第 2 周扩大到 20%；第 4 周全量开放。每阶段锁定 pilot kpi 门禁。

gray release 在企业 LLM 项目中尤其重要——模型输出不可预知，不能像传统软件一样「发版就走」。灰度既是技术策略，也是风险管理策略。

FAQ：试点两周后用户活跃率低于 20%，怎么办？

先诊断 root cause：是培训不足、回答质量差、还是需求场景不匹配？方案：增加 champion 一对一访谈，收集前 5 个未使用原因；检查 BFF 埋点是否准确；考虑追加一次现场培训 + 部门负责人推动。

试点期间模型厂商 SLA 不达标，由谁负责？

按 合同 SLA 走：线上故障先降级到缓存/转人工；事后走厂商 SLA 赔偿流程。交付团队应提前准备降级方案，而不是等厂商修复。

用户反馈「回答太长，只想要一句话总结」怎么办？

在 Prompt 中加入格式指令：「先给一句话总结（≤50 字），再展开详情。」更新后经 golden set 回归测试再发布。这个调整属于 quick win 范畴。

本章结论

试点是从「能做」到「好用」的关键桥梁，灰度、培训与支持三者缺一不可。

  • 灰度发布：Alpha → Beta → GA 三阶段，每阶段有时间、比例、门禁与回滚条件。
  • 用户培训：Champion User 先行 → 部门管理员 → 全员覆盖；培训后确保实战演练。
  • 现场支持：L1/L2/L3 三层 + 4h 响应 SLA + 多渠道反馈汇聚；支持质量用满意度 KPI 衡量。

sla metric 与 contract sla 的关系：SLA 是服务等级协议，包含可用性、延迟、吞吐等；SLA metric 是具体量化指标。CorpAssist 的 SLA 建议涵盖可用性（99.5%）、P95（2.0s）和安全（0 泄露事件）。

先想 10 秒：客户说「不要写除外条款，显得你们没信心」，你如何回应？

正面解释：「除外条款保护双方——写了模型厂商故障不在此限，客户也不会因阿里云宕机而拒付尾款。」这是行业标准做法，不是缺乏信心。

FAQ：客户说「再试用 3 个月再签收」，怎么办？

不可接受。应在合同阶段约定验收窗口期（通常 UAT 后 10~15 个工作日）。超期未签收视为默认通过——这一条款必须写进合同。如果客户确有新需求，可签订二期 SOW，不影响一期验收。

签收后客户发现一个 P1 Bug，怎么办？

按运维流程走：创建 ticket → 按 SLA 时限修复 → 补丁发布。签收不代表零缺陷，而是 KPI 达标且签署了交付物。P1 Bug 属于运维期支持范畴，不应影响签收有效性。

评测结果双方不一致（甲方人工评审与乙方脚本结果偏差），怎么处理？

按合同争议条款处理：不一致条目由第三方重新标注。双方应在签约时就约定「标注一致性检查」——若标注不一致率 > 10%，则 frozen set 需回炉再标。

本章结论

验收不是终点，而是交付质量的一次「对账」——全对才能签收。

  • 合同指标：KPI 条款必须包含指标+数据集+评测方法+阈值+除外责任，签约时完成 frozen set 签字。
  • 评测报告：完整 eval report 包含概述、指标、细项、bad case 分析、结论；脚本与版本锁定。
  • 签收流程：6 步法从 eval report 到签收单签署；合同中约定验收窗口期与签收默认条款。

Dify 开源的是平台框架，不是你的业务代码。自研核心能力（ACL、SLA、审计、门禁）不是「重复造轮子」，而是企业生产必须的定制工程。Dify 团队不管你的 SLA 赔偿。

锁定

vendor lock 在 LLM 交付中有三个层次：模型锁定、平台锁定、数据锁定。低代码平台加剧了后两者。

锁定类型风险描述CorpAssist 示例缓解措施
模型锁定 Prompt 针对某模型优化，换模型效果下降 qwen-plus 换 qwen-turbo 后引用格式不对 Prompt 设计时跨模型兼容；使用 model-agnostic 指令
平台锁定 Dify/Coze 自研 Node 无法在其他平台运行 Dify Code Node 写 Python 处理逻辑 核心逻辑独立部署为微服务，低代码只调 API
数据锁定 标注数据、评测集、日志存在平台上难以导出 Dify Log 不能直接导出为 eval_report 标注数据独立存储；通过 API 定期同步

escape hatch（逃生门）是应对 vendor lock 的关键工程模式：核心能力保留自研版本，低代码仅做上层编排。当低代码平台不可用时，逃生门确保系统核心功能可用。

不推荐
  • 所有能力全放在 Dify，无逃生计划
  • Prompt 只在 Dify 页面上编辑，无 Git 备份
  • 标注数据不导出，全在平台里

  • Prompt 模板 Git 管理，低代码平台引用 Git
  • 标注数据独立存储，通过 API 同步
FAQ：Coze 免费额度很大，为什么还要花钱自研？

Coze 免费额度的代价是数据与工作流锁定。一旦超过免费限额或业务量增长，迁移成本远超自研投入。此外，Coze SaaS 无法满足数据不出域（国企）需求。建议：SaaS 版本仅用于 PoC 验证，生产必须私有化或自研。

FAQ：客户坚持全用低代码，如何说服？

不要全盘否定。展示成本对比：如果客户需要 ACL、SSO、审计、SLA 99.5%、评测门禁——这些在 Dify 上二次开发的总成本可能高于自研。建议：先用 Dify 做 PoC 快速看到效果，随后逐步过渡到混合架构。

Dify 部署版本升级会影响生产吗？

会。Dify 社区版升级可能导致 API 变更或工作流不兼容。建议：① 测试环境先行升级 ② 关键 Workflow 导出备份 ③ 核心 Prompt 在 Git 管理 ④ 预留 1 周回归测试窗口。

低代码平台能支撑「发版门禁」（advanced-07）吗？

目前 Dify/Coze 不支持 golden set 回归门禁。因此评测必须在自研层实现：Dify 负责「出回答」，自研 eval 系统负责「测质量」。这是混合架构的关键理由之一。

本章结论

低代码不是敌手而是工具，关键是把控哪些能力可以外包给平台、哪些必须自研掌控。

  • 迁移：从 Dify 到自研分 4 步走；workflow 导出后有转换成本。
  • 锁定：模型/平台/数据三层锁定；escape hatch 确保低代码故障时核心可用。
  • 混合架构：低代码做运营层（Prompt/知识库/简单 Agent），自研做核心层（鉴权/检索/门禁）。

## Runbook: CorpAssist 模型 API 超时

### 症状
- BFF 日志出现 `timeout calling DashScope API`
- Grafana 看板 P95 延迟 > 3s
- 用户反馈「一直在转圈」

### 影响
S1 RAG + S2 客服均不可用（核心功能中断，P0）

### 排查步骤
1. 确认 DashScope 状态：
   `curl https://dashscope.aliyuncs.com/api/v1/services/ping`
2. 检查 BFF 到 DashScope 网络：
   `curl -w "%{http_code} %{time_total}" https://dashscope.aliyuncs.com`
3. 检查 BFF 线程池状态：
   `curl http://bff:8080/actuator/threaddump | grep "pool-.*-thread"`

### 恢复措施
1. 紧急降级到缓存 + 转人工（BFF 配置 `feature.llm.fallback=true`）
2. 若因 API Key 配额超限：联系阿里云提配额
3. 若因网络故障：检查 VPC/VPN 连接

### 验证
- 降级后：回答切换为「正在检索...」或转人工
- 超时恢复后：手动切回正常模式

### 事后
- 更新 risk register：是否需增加备用模型路由？
- 更新告警阈值：P95 > 2.5s 提前预警

kb ops 指知识库运维——文档更新、索引重建、过期文档清理。必须定义操作频率与流程：

  • 文档更新：客户业务方通过 Dify 后台或 API 更新 → 触发增量索引
  • 索引重建：每月一次全量重建（修复一致性）或在 Embedding 模型升级时
  • 过期文档：业务方标记 status=obsolete → 检索过滤 → 定期清理向量
  • 权限变更：部门合并/拆分时更新 ACL 矩阵 → 重新索引受影响文档
不推荐
  • 无值班表，出问题临时找人
  • Incident 处理完就结束了，无 RCA
  • 知识库从来不更新，半年后准确率下降
推荐
  • L1/L2/L3 值班表 + 节假日备班
  • P0/P1 必须生成 RCA 与预防措施
  • KB ops 有固定频率与责任人

FAQ：模型 API 升级后准确率下降，能在生产直接回滚吗？

可以，前提是你提前冻结了旧版本的 model_alias。在 BFF 配置中将 model_alias 指向旧版本即可立即切回。这也是为什么模型升级前必须保留旧版本别名。

移交后客户 IT 能力弱，能否一直让交付 team 运维？

可以约定运维服务期（通常 3~6 个月，可付费续期）。但建议在运维期内逐步转移：第一个月全权运维、第二个月协同、第三个月监督。到期后客户 IT 应已具备独立运维能力。可将此写入 SLA 附件。

索引重建需要多长时间的停机？

零停机——通过双索引切换实现：先在新索引流水线重建，重建完成后切查询流量到新索引，旧索引保留为回滚快照。这个过程通常 2~4 小时。

本章结论

移交不是把所有文件打包丢给客户就结束了——它是一套让接盘团队能独立运维的完整体系。

  • 移交文档：6 模块可操作文档，重点是运维场景而非功能说明。
  • 值班与运维：L1/L2/L3 三层值班 + incident flow（止血→RCA→预防）+ KB ops 定期维护。
  • 版本升级：管理模型/索引/Prompt 三类版本；升级必走 CR→测试→审批→灰度→监控五步；每步可回滚。

STAR 面试叙事

star method 把 3 分钟项目答辩结构化为四段。面试官通过 STAR 评估的是你的决策能力和工程思维，不是工具熟练度。

STAR 完整示例（CorpAssist ACL 晚补齐）

markdown
## STAR — 索引 ACL 风险与补救（3 分钟版）

**S（Situation — 30s）**
某省国企 CorpAssist 试点第 2 周，HR/IT 两部门同时上线 RAG；
合规要求「员工只能看本部门文档」，但索引管线尚未接入 ACL 矩阵。
1.2 万份文档已索引完成，撤回成本极高。

**T（Task — 20s）**
我负责 Python 检索与 BFF 联调，需在 **3 天内** 消除跨部门泄露风险，
且不能回滚已索引的文档库。

**A（Action — 90s，重点）**
1. 与合规组 frozen ACL 矩阵（部门→doc_tag 映射），写入 risk register。
2. 检索层增加 acl_scope 必填校验；BFF 从 SSO 注入 scope。
3. 离线脚本按 tag 重建向量分区；hotfix 灰度 10%→20%→100%。
4. 补 UAT 用例 20 条「跨部门负例」，纳入 golden set 回归集。
5. 复盘将「ACL sign-off」前置到 Discovery checklist。

**R（Result — 30s）**
- 48h 内 hotfix 上线，0 起 P1 泄露事件。
- UAT 准确率仍达 86%（未因 hotfix 引入回归）。
- 复盘输出 1 条流程改进：ACL sign-off 写入 discovery 门禁。
- 该案例成为团队「安全交付」最佳实践。
  • S：行业、场景、约束（合规/工期），30 秒内建立可信度。
  • T：你的责任边界 + 成功标准（数字或日期）。
  • 忌：S 段讲 5 分钟公司史；T 段模糊「帮忙做项目」。
  • A：2~4 个决策动作，含 trade-off（为何 hotfix 而非全量重索引）。
  • R：指标 + 组织结果（复盘、流程改进）。
  • 忌：A 只列工具名；R 只有「项目成功」。

简历排版黄金法则

  • 一页原则：3 年以上经验尽量控制在一页；5 年以上最多两页。CorpAssist 项目放第一页中部。
  • 动词开头：每行 bullet 以「主导/设计/实现/优化」开头，不用「参与/协助」。
  • 数字优先：每条 bullet 至少含 1 个数字（时间、指标、规模、团队大小）。
  • STAR 精选：简历中写 Result（数字），Situation 和 Task 留到面试口述。
简历常见问题
  • 堆砌 10 个框架名无项目上下文
  • 「参与」「协助」过多
  • 无数字、无量化结果
  • 简历与岗位 JD 关键词不匹配
简历优化
  • 3 个重点项目各 3-4 条 bullet
  • 动词精准 + 数字量化和结果
  • 关键词从目标 JD 提取
  • PDF 命名含姓名和岗位
FAQ：项目经历与目标岗位不匹配怎么办？
使用双轨 bullet 中的「技术轨」匹配技术岗、「交付轨」匹配解决方案岗。如果目标岗位是纯算法，需要突出 CorpAssist 中评测集设计、Golden set 管理、模型选型等内容。
简历造假/夸张的后果？
面试官会深挖细节：如果写了「独立完成 sign-off」，会被追问签字流程、客户决策人信息、合同条款。「双轨叙事」要求同一项目的两套说法数据一致。夸张会导致面试穿帮，严重者影响背调结果。

  • 复合型描述：同一案例写交付轨与技术轨 bullet，共享复盘事实源。每条含动词+场景+数字。

  • 简历排版：一页原则 + 动词开头 + 数字优先 + ATS 关键词匹配。

为什么要学本章

  • 用反模式清单做 discovery / 验收门禁
  • 输出团队 delivery trap 预警表
本章先记住 2 件事
  • 过度承诺：没有 SLA、没有 golden set、没有 ACL 的「保证准确率」都是 silent assumption 的外显。

过度承诺

CorpAssist · 招标前夜 销售希望标书写「问答准确率 95%、7×24 智能替代人工」；算法同事说 PoC 最好的一次是 88%；法务问 SLA 赔偿条款。你作为交付负责人，需要把承诺拉回可验收口径。

over promise 在 ToB LLM 项目中的典型表现：

  • 把 PoC 单次 demo 指标写进合同，无数据集 frozen 条款
  • 承诺「全场景 Agent 自主办理」却无工具真执行与步数预算
  • 口头答应「私有化 + 国产化 + 2 周上线」但未估 gpu estimate 与人天
  • 简历 / 答辩夸大个人角色（与 advanced-02 呼应）
  • 对客户说「什么都能答」却不定义 scope boundary
承诺类型危险信号可验收替代写法
准确率「保证 95%」无评测集定义UAT golden set ≥ N 条，eval_report 双方签字
可用性「永不宕机」contract sla 99.5%，明确排除厂商侧故障
范围「所有部门所有文档」mvp scope：2 部门 + 3 类文档 pilot

FAQ：客户坚持高指标怎么办？

提供付费 PoC 加测或试点对赌：加测扩大 golden set、延长 pilot，达标后再签主合同 KPI。勿在无数据前提下签「95%」。也可以设置阶梯指标：PoC 80% → 试点 83% → 验收 85%。

FAQ：什么是 gold plating（镀金）？举例说明。
Gold plating 指团队自发添加客户未要求的、对核心目标无贡献的功能。CorpAssist 示例：花 2 周做了一个「天气查询」功能（与制度问答无关），而核心的 ACL 债仍未还。Gold plating 的识别标准：这个功能写在 scope 里了吗？不写会影响验收吗？
scope creep 和 gold plating 的区别？
Scope creep 源自客户不断新增需求；gold plating 源自团队「想做更好」的自发行为。都是负面——都不走 CR 流程，都消耗未规划的人天。

如何向客户说「不」

识别出 over promise 后最难的部分不是技术评估，而是如何在不丢单的前提下管理客户期望。以下是几种常见场景的应对话术。

客户要求不能直接说专业回应
「准确率保证 95%」「做不到」「我们建议以 UAT golden set 为基准，设为 85%。如果 Pilot 期间数据证明能达到更高，可在二期调整。」
「2 周全量上线」「时间不够」「我们可以在 2 周内完成 PoC 演示。但生产上线涉及数据接入、安全审计、用户培训，建议 8 周分期交付。」
「所有部门同时开通」「不行」「建议分部门灰度，先选 2 个试点部门验证流程和数据质量。通常 4 周后可以扩展到全部门。」

说「不」的原则：给替代方案，不给空头支票。每次拒绝都附带一个可执行的替代建议。同时，在 risk register 中记录客户原始要求和你的替代建议，为可能的商务纠纷保留证据。

本章结论

  • 过度承诺：把 verbal 承诺拉回 scope + 三阶段指标 + risk register。silent assumption 必须书面化。
  • 技术债：PoC 捷径必须登记偿还；P0 安全/降级/可观测优先于 gold plating。
  • 交付陷阱：大部分 trap 可预防——discovery 数据检查、PoC 指标锁定、试点 CR 流程。

S1 企业 RAG 架构与指标

S1 · 企业知识库 RAG制度/合同/运维手册问答，回答必须带引用；内网 ACL 按部门隔离。面试常问：混合检索、评测集、索引更新策略。

s1 rag defense 核心：检索增强 + 引用溯源 + 权限在检索层生效。CorpAssist 双栈下，Spring BFF 传 acl_scope，Python 检索服务在向量/关键词两路均过滤。

S1 企业 RAG 分层架构
flowchart TB
          subgraph ingest [索引链路]
            OSS[文档 OSS] --> Parse[解析 OCR/表格]
            Parse --> Chunk[分块+元数据]
            Chunk --> Embed[Embedding]
            Embed --> VDB[(向量库)]
          end
          subgraph query [在线问答]
            Q[用户问题] --> Hyb[混合检索 BM25+向量]
            Hyb --> ACL[ACL 过滤]
            ACL --> Rerank[重排]
            Rerank --> Gen[LLM 生成+引用]
          end
          VDB --> Hyb
          Gen --> Cite[引用 doc_id 回传 BFF]

解析与索引在 Python；BFF 只做鉴权、限流与审计日志。索引版本变更必须触发 re-embed 计划。

本场景必答生产坑（答辩高光）

坑：脏 PDF / 扫描件
  • 无 OCR/表格还原 → 召回上限锁死在 60%
  • 客户质疑「模型不行」实为数据管道缺失
防御
  • discovery 阶段签「可解析文档占比」基线
  • 解析质量进 golden set 分层（扫描件子集）
  • PoC 演示前先跑 50 页样本解析报告

另两条高频坑：ACL 未在检索层过滤；纯向量忽略制度 Graph 关联。

S2 智能客服与降级

S2 · 企业智能客服HR/IT 服务台场景，意图分类 + 槽位收集 + 知识库查询。面试常问：降级策略、转人工条件、成本控制。

s2 cs defense 核心：DST（对话状态跟踪）状态机 + 降级兜底 + 转人工流畅。

S2 客服降级架构
flowchart TD
          User[用户提问] --> Intent[意图分类]
          Intent -->|高置信| DST[DST 槽位收集]
          Intent -->|低置信| Human[转人工]
          DST -->|槽位不足| Confirm[确认+追问]
          DST -->|槽位满| Query[知识库查询]
          Query -->|LLM API 可用| Answer[生成回答]
          Query -->|LLM 超时/故障| Fallback[缓存/默认回复/转人工]
          Answer -->|需要二次确认| Human

降级不是失败——它是系统设计的一部分。90% 请求走正常链路，10% 降级不中断服务。

本场景必答生产坑

坑：转人工链路断裂——Agent 承诺「已退款」但未绑定支付工具 API。防御：DST 强制槽位满才允许生成「已退款」文案；支付等敏感操作必须工具调用 + 结果校验。

NS2 确定性DST 状态机 + 转人工网关；不纯靠 LLM 概率
NS3 成本qwen-turbo 用于客服（低成本），缓存常见问

S3 Agent 工作流

S3 · 企业 Agent工单审批、数据查询、跨系统操作。面试常问：步数预算、工具路由、Token 成本控制。

s3 agent defense 核心：工具定义 + 步数预算 + 结果验证 + 人工确认闸门。

S3 Agent 工作流结构
flowchart TD
          Task[用户任务] --> Plan[工具规划]
          Plan --> Call[工具调用 API]
          Call --> Verify[结果校验]
          Verify -->|成功| Next{更多步骤?}
          Verify -->|失败| Retry[重试/降级]
          Next -->|Y| Plan
          Next -->|N| Summary[结果汇总]
          Summary --> Confirm{需人工确认?}
          Confirm -->|Y| Human[等待审批]
          Confirm -->|N| Done[完成]
          Retry -->|超步数| Human

Agent 必须有步数预算（默认 5 步）和超步降级，否则循环调用会消耗大量 Token。

本场景必答生产坑

坑：Agent 超步仍调用 ERP 写接口——Agent 第 4 步仍调用 ERP 后超时，导致脏数据。防御：步数预算硬限制（最大 5 步）+ 写操作前人工确认闸门。

坑：工具调用返回过长——ERP API 返回 10K+ JSON 撑爆上下文。防御：截断 + 结构化提取关键字段。

S4 代码库 RAG

S4 · 代码库问答内部代码库/技术文档检索，要求代码片段可编译验证。面试常问：代码分块策略、重排、跨文件引用。

s4 code defense 核心：代码结构感知分块 + 跨文件引用追踪 + 可编译验证。

代码 RAG 分块策略

策略适用块大小
基于函数/类分块方法级检索20~100 行
基于文件分块文件概览整个文件
基于 AST 分块精确引用追踪语法节点

本场景必答生产坑

坑：跨文件引用丢失——单文件分块导致「这里调用了 format_date()」但搜索不到该函数定义。防御：解析 import 关系建立跨文件索引。

S5 内容生成流水线

S5 · 内容生成周报总结、公告生成、报告摘要。面试常问：格式控制、风格一致、批量成本。

s5 content defense 核心：模板驱动 + 格式约束 + 人工审核门禁。

内容生成流水线结构

  • 输入：结构化数据 / 文档 / 要点
  • 模板：Pregenerated 模板 + Slot 填充
  • 约束：长度、风格、敏感词过滤
  • 审核：自动规则 + 人工抽样

本场景必答生产坑

坑：批量生成无差异化——50 份周报内容高度雷同。防御：注入每份文档的独特点（数据/指标/事件），设置多样性约束。

高频追问与应答模板

markdown
Q: 「S1 RAG 你们的检索策略是什么？」
A: 混合检索（BM25 + 向量）→ Rerank → ACL 过滤 → LLM 生成。
   选择混合检索的原因：制度类文档需要关键词精确匹配（如「年假 15 天」），
   纯向量会丢失精确匹配能力。Rerank 解决 Top-K 精度问题。

Q: 「如果模型 API 挂了怎么办？」
A: 三层降级：① 缓存常见问答（兜底 30% 流量）
   ② 转人工话术（兜底 50% 流量）
   ③ 熔断后走 fallback 入口（兜底 100%）
   以上都在 BFF 层配置，不依赖 LLM。

Q: 「你的飞轮怎么转的？」
A: 线上 bad case → 周 triage → 四维归因 → 修复 → golden 更新
   → 发版门禁 → 灰度。P0 坏例 24h 入库，P1 本周修复。
   目标：从发现到 golden 入库中位时间 < 5 个工作日。

  • 数字口径与简历一致（PoC/试点/合同分阶段）
FAQ：答辩时被问到不会的问题怎么办？
诚实说「这个方向我没有深入做过，但我的理解是...」比硬编答案好。也可以引导到你知道的领域：「这个具体指标我记不清了，但我可以讲讲我们的评测方法论。」面试官看重的是问题解决框架，不是背诵能力。

为什么要学本章

TOP5 总答辩 里你承诺了 NS1「飞轮」。本章把承诺落成流程：bad case triage → root cause 4 归因 → golden update → release gate → rollback drill。没有飞轮，生产坑只会重复发生。

学完你能
  • 从线上日志/质检/客户工单选取 3 条 bad case 并完成分级

本章先记住 3 件事
  • 坏例是资产：未入库的投诉等于未发生（无法回归）。
  • 归因到可改层：避免「模型不行」笼统结论。
  • 发版门禁先于全量：flywheel ns1 闭环的硬卡口。

归因结论须对应可测验证（eval 用例或集成测试），不可泛泛说「模型不行」。

飞轮闭环自检

FAQ：修复后准确率不变，但 bad case 确实修了，怎么量化？
在 golden set 中新增该 bad case 的回归用例，eval 时单独看 regression subset 的通过率。总准确率可能不变，但 regression subset 通过率应从 0% 升到 100%。
谁负责维护 golden set？频率是多少？
算法团队维护 frozen set（验收用），QA 团队维护 regression set（周更）。每次 triage 后 PM 跟踪 golden 更新状态。频率：P0 实时更新，P1 本周更新。

飞轮成熟度模型

级别特征交付课对应的阶段
L1 手工无固定 bad case 收集渠道，修复凭经验PoC 阶段

作为起步，至少达到 L2（文档化）级别：有卡片模板、有归因流程、有 golden 更新机制。验收时向客户展示飞轮运转情况，是证明「NS1 飞轮」承诺已落地的关键证据。

飞轮反模式
  • Triage 会议无记录，口头说完了事
  • 修复后无验证，不确定是否修好
  • Golden set 从不更新
  • 飞轮转速无人跟踪
飞轮最佳实践
  • 每次 triage 有纪要 + action items
  • 修复后回跑 regression set
  • Golden set 随每次修复增量更新
  • 飞轮转速按月度量并展示
FAQ：飞轮系统应该由谁维护？
建议由算法团队主导，PM 跟踪进度，QA 维护 golden set。每周 triage 会议需要产品/交付/算法/运维各 1 人参加（30 分钟）。飞轮的长期维护责任应在移交文档中写明。

本章结论

  • 选取 3 条：按 P0/P1/P2 从工单/质检/日志入库，卡片字段齐全。
  • 四维归因：数据/工具/Prompt/路由，结论可测可验证。
  • golden 更新：P0/P1 必增回归用例，与 eval 报告同 PR。
  • 门禁与回滚：发版前 eval_pass_rate + regression diff < 2%；季度 rollback 演练留 RTO 记录。
  • 飞轮转速：从发现到 golden 入库的中位天数，P0 < 3 天，P1 < 5 个工作日。
