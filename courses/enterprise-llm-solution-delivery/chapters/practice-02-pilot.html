为什么要学本章

PoC 演示再好也是「温室里的花」；pilot 是第一次接受真实用户、真实数据与真实合规的考验。没有结构化 rollout plan，常见结果是：灰度 10% 用户不活跃、超级用户没培训就上岗、问题 72h 得不到回应。本章用 CorpAssist 试点案例讲透灰度、培训与支持三项必修课。

学完你能
  • 制定 gray release 分批策略与回滚条件
  • 设计 user training 课程体系与 champion user 计划
  • 建立 onsite support 值班制度与升级路径
  • 定义 pilot kpi 与 feedback channel
本章先记住 3 件事
  • 灰度不是 All or Nothing：5% → 20% → 100% 三步走，每步都有门禁。
  • 培训不是发说明文档：先培养 champion user 再辐射全部门。
  • 支持响应 = 信任度：4h 响应 SLA 比 99% 准确率更能影响续约。

灰度发布

CorpAssist · HR 部门试点（5 周） PoC 通过后，HR 部门 200 人分三批灰度：第 1 周 10%「超级用户」内测；第 2 周扩大到 20%；第 4 周全量开放。每阶段锁定 pilot kpi 门禁。

gray release 在企业 LLM 项目中尤其重要——模型输出不可预知，不能像传统软件一样「发版就走」。灰度既是技术策略，也是风险管理策略。

三阶段灰度计划模板

阶段时间比例用户选择门禁条件回滚触发
Alpha 第 1 周 5-10% 主动报名的 champion user 基础功能可走通 P0 缺陷或接口不可用
Beta 第 2-3 周 20-30% 随机抽样 + champion 推荐 Alpha 反馈已闭环 ≥ 80% bad case 率 > 5%
GA 第 4-5 周 100% 全部门开放 Beta 阶段 KPI 全部达标 P95 > 3s 或准确率 < 75%
灰度发布流程与门禁
flowchart TD
          GA1[5% Alpha] --> Check1{Alpha KPI 达标?}
          Check1 -->|Y| GA2[20% Beta]
          Check1 -->|N| Fix1[修复 + 回归]
          Fix1 --> GA1
          GA2 --> Check2{Beta KPI 达标?}
          Check2 -->|Y| GA3[100% Full]
          Check2 -->|N| Fix2[修复 + 灰度不变]
          Fix2 --> GA2
          GA3 --> Monitor[持续监控 + 反馈通道]
          Monitor -->|门禁触发| Rollback[回滚至上一版本]
          Rollback --> RCA[根因分析与修复]

每阶段必须通过门禁才能进入下一阶段；门禁阈值写入 contract sla 或 pilot 协议。

灰度配置示例（Spring BFF）

yaml 
# corp-assist-gray-release.yaml
pilot:
  phases:
    - name: alpha
      percent: 10
      duration_days: 7
      user_selection: "whitelist"
      whitelist_file: "config/alpha-users.csv"
      metrics_gate:
        p95_latency_ms: { max: 3000 }
        golden_accuracy: { min: 0.75 }
    - name: beta
      percent: 25
      duration_days: 14
      gate:
        alpha_feedback_closed_rate: { min: 0.8 }
    - name: ga
      percent: 100
      duration_days: 0
      gate:
        beta_badcase_rate: { max: 0.05 }

auto_rollback:
  triggers:
    - bad_case_rate_1h: { gt: 0.08 }
    - p95_latency_ms: { gt: 3000 }
    - availability_1h: { lt: 0.99 }
不推荐
  • 一次性全量开放所有用户
  • 灰度与生产环境混布，无隔离
  • 无回滚预案，出问题各自打补丁
推荐
  • 三阶段灰度 + 门禁 + 回滚触发条件
  • 灰度用户通过 BFF 路由规则控制
  • 每阶段结束出 KPI 报告 + 复盘
先想 10 秒：Alpha 阶段准确率 82% 但延迟有 3.2s，是否应进入 Beta？

不应进入。延迟门禁（P95 < 3s）未达标；先排查：是否缓存未预热/模型路由未优化/推理实例不足。延迟问题不解决，Beta 阶段用户增多只会更严重。

用户培训

user training 决定用户是否真正使用系统。CorpAssist 试点培训分三层：

培训对象内容时间方式
Champion User 系统功能 + 常见问题 + 反馈渠道 Alpha 启动前 面对面工作坊 2h
部门管理员 知识维护、用户管理、质检 Beta 启动前 线上演示 + QA 1h
全体用户 如何使用、哪些不能做、如何反馈 GA 前 3 天 文档 + 视频 + 启动会

Champion User 计划

champion user 是试点成功的关键角色——他们是内部"代言人"，帮助收集反馈、推广使用、减少交付团队的一线支持压力。CorpAssist 试点要求每个部门至少 2 名 champion user。

  • 选择标准：主动报名、具备基本 IT 素养、有影响力
  • 激励措施：专属答疑群、优先体验新功能、证书/徽章
  • 职责：每日使用 + 每周反馈 + 收集同事问题
  • 退出机制：超过 2 周不活跃 → 通知部门负责人换人

## CorpAssist 试点培训计划

### 第 1 周：Champion User 培训
**时间**：周一 14:00-16:00，面对面
**议程**：

- 确认反馈渠道可用
- 签署 champion user 承诺书

### 第 3 周：部门管理员培训

**议程**：
1. 知识库维护操作（20min）
2. 用户管理（10min）
3. 质检与 bad case 提交流程（20min）
4. QA（10min）

### 第 4 周：全员启动会
**内容**：邮件 + 视频教程 + 海报 + 门户弹窗引导

每日使用 + 每周反馈 + 收集同事问题；是交付团队的「内部耳目」。

先想 10 秒：Champion 反馈「搜索结果不准」，你第一反应做什么？

不是改模型——先查 query log 确认是否在 golden 覆盖范围内；若 out-of-domain，更新 feedback channel 记录并排入 next sprint。

现场支持

onsite support 在试点前 2 周最为关键。CorpAssist 试点支持体系包含三个层级：

层级响应时间处理范围值班人员
L1 一线支持 4h 内回应 账号、权限、操作指导、环境问题 客户 IT 部门或项目驻场
L2 技术支持 8h 内回应 检索结果不准、模型异常、接口报错 交付团队算法/Java 工程师
L3 研发支持 24h 内回应 模型替换、Prompt 调优、架构变更 后端研发团队

值班制度模板

markdown 
## CorpAssist 试点值班制度

### 值班安排
- **工作日 09:00-18:00**：交付团队全员在线（响应 ≤ 4h）
- **非工作时间**：L1 值班工程师（onsite），L2/L3 电话待命
- **周末**：优先 champion 群自助 FAQ，紧急问题 L2 电话

### 升级路径
1. 用户 → L1 值班（问题记录 + 初步排查）
2. L1 无法解决 → @L2 值班群（附 trace_id + 复现步骤）
3. L2 判定架构/模型问题 → 创建 Jira ticket → L3 排期
4. P0 事故 → 立即升级到 PM + 架构师决策

### 值班日志模板
| 时间 | 用户 | 问题 | 严重级 | 处理人 | 状态 |
|------|------|------|--------|--------|------|
| 2026-05-31 14:22 | HR 张三 | 搜索「年假」返回空 | P2 | 李四 | 已关闭 |
不推荐
  • 没有值班表，谁有空谁回复
  • 支持全靠邮件往来，没有即时通讯群
  • 无升级路径，用户反复被转接
推荐
  • 三层支持体系 + 明确响应时间 SLA
  • 即时通讯群（飞书/钉钉）+ 工单系统双通道
  • 每周值班总结 → 更新 FAQ 与培训材料

试点 KPI 看板

pilot kpi 是判断试点是否成功的定量依据：

KPI计算方式目标（HR 试点）采集方式
日活率 日活跃用户 / 部门总人数 ≥ 40% 门户埋点
提问采纳率 用户采纳回答的比例（点赞 + 引用点击） ≥ 60% BFF 日志
转人工率 转人工对话数 / 总对话数 < 18% 会话日志
问题解决率 用户 3 轮内未再追问的比例 ≥ 65% 会话分析
支持满意度 用户对支持响应的评分 ≥ 4.0 / 5.0 在线问卷

feedback channel 建议设置三个入口：① 门户「反馈」按钮 ② 钉钉/飞书支持群 ③ 每周 champion 例会。三入口数据汇聚到同一看板，避免信息孤岛。

FAQ：试点两周后用户活跃率低于 20%，怎么办？

先诊断 root cause：是培训不足、回答质量差、还是需求场景不匹配？方案：增加 champion 一对一访谈，收集前 5 个未使用原因；检查 BFF 埋点是否准确；考虑追加一次现场培训 + 部门负责人推动。

试点期间模型厂商 SLA 不达标，由谁负责？

按 合同 SLA 走：线上故障先降级到缓存/转人工；事后走厂商 SLA 赔偿流程。交付团队应提前准备降级方案，而不是等厂商修复。

用户反馈「回答太长，只想要一句话总结」怎么办？

在 Prompt 中加入格式指令：「先给一句话总结（≤50 字），再展开详情。」更新后经 golden set 回归测试再发布。这个调整属于 quick win 范畴。

本章结论

试点是从「能做」到「好用」的关键桥梁，灰度、培训与支持三者缺一不可。

  • 灰度发布：Alpha → Beta → GA 三阶段，每阶段有时间、比例、门禁与回滚条件。
  • 用户培训：Champion User 先行 → 部门管理员 → 全员覆盖；培训后确保实战演练。
  • 现场支持：L1/L2/L3 三层 + 4h 响应 SLA + 多渠道反馈汇聚；支持质量用满意度 KPI 衡量。

判断练习

  1. （排障）试点第 3 天，10 个 champion 用户中 8 人反馈「搜索没结果」。你最优先排查？A) 向量库是否在线 B) ACL 是否太过严格 C) 用户输入问题太短

    参考答案

    A → B 顺序排查。先确定基础设施是否正常（向量库、Embedding 服务）；再排查 ACL 是否意外拦截了所有文档（测试用 admin 账号对比）。C 可能性最低。

  2. （策略）客户坚持「全员同时开通，不做灰度」。你应优先采取哪项措施降低风险？

    参考答案

    至少保留：① 回滚按钮（BFF 切流量开关 + 索引快照）② 前 3 天按比例观察指标 ③ 与客户约定「试运行 2 周后再确认验收」。同时更新 risk register 记录此项风险。
