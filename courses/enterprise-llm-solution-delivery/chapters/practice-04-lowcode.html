为什么要学本章

客户可能已有 Dify 或 Coze 体验版，甚至要求「就用 Dify 交付，别自研」。但 dify coze 的低门槛背后藏着 vendor lock、migrate path 不清晰、build vs buy 决策失误等风险。本章帮你系统评估何时用低代码、何时自研、如何共存。

学完你能
  • 画出 hybrid arch：低代码做运营配置 + 自研做核心推理
  • 评估 vendor lock 风险：数据、Prompt、工作流能否导出
  • 设计 migrate path：从 Dify PoC 平滑过渡到自研生产
  • 为 lowcode trade 做四维决策（场景、成本、工期、锁定）
本章先记住 3 件事
  • 低代码 ≠ 零成本迁移：Dify workflow 导出后不一定能在自研栈直接运行。
  • 混合架构最现实：Prompt 编排用低代码，核心链路（检索/鉴权/SLA）自研。
  • Escape hatch 必须预留：escape hatch 让核心能力在平台失效时能独立运行。

迁移

CorpAssist · 客户已有 Dify 实例 客户 IT 已经用 Dify 搭建了一个知识库 RAG 原型，要求交付团队「直接在此基础上生产化」。你需要评估 Dify 当前工作流能否支撑生产：ACL、SLA、企业 SSO、审计日志——这些低代码平台往往缺。

dify coze 类平台的优势明显：快速原型（1 周可搭出可用的 RAG）、非技术团队可自行维护 Prompt、内置多模型路由。但企业交付场景下，以下能力需要重点评估：

低代码平台企业级能力对照表

能力需求Dify 社区版Dify 企业版Coze自研
自定义 ACL/SSO △ 需二次开发 ✅ 部分支持 ❌ ✅ 完全控制
审计日志 △ 基础操作日志 ✅ 操作审计 ❌ ✅ 全链路
自定义 Prompt 管控 ✅ ✅ ✅ ✅ 版本控制更强
Workflow 导出 ✅ JSON DSL ✅ JSON DSL ❌ N/A
SLA 99.5%+ ❌ 自建保障 △ 依赖部署 ❌ ✅ 可控
私有化部署 ✅ Docker ✅ ❌ SaaS only ✅

migrate path 设计最关键：从低代码到自研通常分 4 步走。

Dify → 自研迁移路径
flowchart LR
          Phase1[Dify PoC: 1-2 周快速验证] --> Phase2[混合期: Dify 做编排 + 自研 BFF 做鉴权/审计]
          Phase2 --> Phase3[核心链路自研: 检索/评测/门禁]
          Phase3 --> Phase4[全量自研: Dify 仅做运营后台]
          Phase1 -->|关键决策| Check{ACL / SSO / SLA 要求?}
          Check -->|低| Phase2
          Check -->|高| Phase3

迁移路径分 4 个阶段，每阶段 2~4 周；核心决策点在第 1 阶段末。

迁移注意事项

不推荐
  • Dify 搭完就直接给客户说「交付完成」
  • 所有 Prompt 写在 Dify 页面，无版本控制
  • 无备份策略，Dify 数据库坏了全部丢失
推荐
  • 明确 Dify 定位「原型工具」，生产走自研链路
  • Prompt 导出到 Git 做版本管理
  • Dify 数据（Document/Workflow）定时备份
bash 
# Dify Workflow 导出与备份脚本（示例）
# 导出单个 Workflow 为 JSON
curl -X GET "https://dify.corp.com/api/workflows/export?workflow_id=w-xxx" \
  -H "Authorization: Bearer $API_KEY" > dify-workflow-backup-$(date +%Y%m%d).json

# 导出所有 Document（知识库）
curl -X GET "https://dify.corp.com/api/datasets" \
  -H "Authorization: Bearer $API_KEY" > dify-datasets-backup-$(date +%Y%m%d).json

# 提示：Dify 导出的 JSON DSL 不能直接在自研栈运行，需手动转换。

workflow export 的局限性：导出的 JSON 会将 Dify 内置 Node（如 LLM Node、Knowledge Node）序列化成私有 Schema，不兼容 LangChain/LlamaIndex。迁移时需要「手动转换」，而非「直接导入」。

先想 10 秒：客户说「Dify 免费开源，为什么还要自研？」怎么回答？

Dify 开源的是平台框架，不是你的业务代码。自研核心能力（ACL、SLA、审计、门禁）不是「重复造轮子」，而是企业生产必须的定制工程。Dify 团队不管你的 SLA 赔偿。

锁定

vendor lock 在 LLM 交付中有三个层次：模型锁定、平台锁定、数据锁定。低代码平台加剧了后两者。

锁定类型风险描述CorpAssist 示例缓解措施
模型锁定 Prompt 针对某模型优化，换模型效果下降 qwen-plus 换 qwen-turbo 后引用格式不对 Prompt 设计时跨模型兼容；使用 model-agnostic 指令
平台锁定 Dify/Coze 自研 Node 无法在其他平台运行 Dify Code Node 写 Python 处理逻辑 核心逻辑独立部署为微服务，低代码只调 API
数据锁定 标注数据、评测集、日志存在平台上难以导出 Dify Log 不能直接导出为 eval_report 标注数据独立存储；通过 API 定期同步

escape hatch（逃生门）是应对 vendor lock 的关键工程模式：核心能力保留自研版本，低代码仅做上层编排。当低代码平台不可用时，逃生门确保系统核心功能可用。

Escape Hatch 设计方案

yaml 
# escape-hatch-config.yaml — CorpAssist 逃生门配置
escape_hatch:
  # 当 Dify/Coze 不可用时的降级策略
  enabled: true

  components:
    - name: "rag_search"
      primary: "dify_knowledge"     # 主路径：走 Dify API
      fallback: "self_rag_api"      # 降级路径：走自研 RAG API
      switch_condition: "dify_health_check_failed_3x"

    - name: "prompt_template"
      primary: "dify_prompt_registry"
      fallback: "local_prompt_file" # 本地 Git 仓库的 Prompt 模板
      switch_condition: "dify_api_latency > 5s"

  health_check:
    - target: "dify_api"
      interval_seconds: 30
      timeout_seconds: 5
      retry_count: 3
Escape Hatch 切换流程
flowchart TD
          Req[用户请求] --> Router{低代码可用?}
          Router -->|Y| Dify[Dify/Coze 处理]
          Router -->|N| Self[自研栈处理]
          Dify --> Response[返回结果]
          Self --> Response
          Dify -.->|健康检查失败| Router
          Self -->|逃生门记录| Log[告警 + 记录逃生事件]

逃生门在架构层面保证低代码平台故障时不中断核心服务。

不推荐
  • 所有能力全放在 Dify，无逃生计划
  • Prompt 只在 Dify 页面上编辑，无 Git 备份
  • 标注数据不导出，全在平台里
推荐
  • 核心能力预留自研版本（escape hatch）
  • Prompt 模板 Git 管理，低代码平台引用 Git
  • 标注数据独立存储，通过 API 同步
FAQ：Coze 免费额度很大，为什么还要花钱自研？

Coze 免费额度的代价是数据与工作流锁定。一旦超过免费限额或业务量增长，迁移成本远超自研投入。此外，Coze SaaS 无法满足数据不出域（国企）需求。建议：SaaS 版本仅用于 PoC 验证，生产必须私有化或自研。

混合架构

hybrid arch 是当前最主流的 ToB 交付模式：低代码平台做「运营层」（Prompt 编排、知识库管理、简单 Agent），自研栈做「核心层」（鉴权、检索、评测、门禁）。

低代码负责「运营配置」，自研负责「企业治理」；两者通过 OpenAPI 接口串联。

Build vs Buy 决策矩阵

build vs buy 需要从四个维度评估：

维度Buy（Dify/Coze）Build（自研）决策权重
场景复杂度 标准 RAG / 简单客服 复杂 Agent / 多步编排 / 自定义路由 高
交付工期 1~2 周出原型 6~12 周 MVP 中
长期总成本 按月付费，规模后成本上升 一次性投入，边际成本低 高
锁定风险 中等 - 高 低 高

## CorpAssist 低代码 vs 自研决策（示例）

### 用低代码（Dify）

- 试点阶段：BFF + 自研 RAG 上线，Dify 退居运营层
- 验收阶段：核心链路全部自研，Dify 仅做知识库管理

不推荐

  • 全量自研，PoC 耗时 3 个月才见到第一个 demo

  • 统一 OpenAPI 网关，对外一致性体验
FAQ：客户坚持全用低代码，如何说服？

不要全盘否定。展示成本对比：如果客户需要 ACL、SSO、审计、SLA 99.5%、评测门禁——这些在 Dify 上二次开发的总成本可能高于自研。建议：先用 Dify 做 PoC 快速看到效果，随后逐步过渡到混合架构。

Dify 部署版本升级会影响生产吗？

会。Dify 社区版升级可能导致 API 变更或工作流不兼容。建议：① 测试环境先行升级 ② 关键 Workflow 导出备份 ③ 核心 Prompt 在 Git 管理 ④ 预留 1 周回归测试窗口。

低代码平台能支撑「发版门禁」（advanced-07）吗？

目前 Dify/Coze 不支持 golden set 回归门禁。因此评测必须在自研层实现：Dify 负责「出回答」，自研 eval 系统负责「测质量」。这是混合架构的关键理由之一。

本章结论

低代码不是敌手而是工具，关键是把控哪些能力可以外包给平台、哪些必须自研掌控。

  • 迁移：从 Dify 到自研分 4 步走；workflow 导出后有转换成本。
  • 锁定：模型/平台/数据三层锁定；escape hatch 确保低代码故障时核心可用。
  • 混合架构：低代码做运营层（Prompt/知识库/简单 Agent），自研做核心层（鉴权/检索/门禁）。
