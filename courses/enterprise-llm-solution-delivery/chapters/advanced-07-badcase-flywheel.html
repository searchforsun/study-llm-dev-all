约 50 分钟 · 阶段：收官 · 能力：坏例 triage、四维归因、golden set 更新、发版门禁与回滚演练

为什么要学本章

TOP5 总答辩 里你承诺了 NS1「飞轮」。本章把承诺落成流程：bad case triage → root cause 4 归因 → golden update → release gate → rollback drill。没有飞轮，生产坑只会重复发生。

学完你能
  • 从线上日志/质检/客户工单选取 3 条 bad case 并完成分级
  • 用数据/工具/Prompt/路由四维归因，写出可验证修复项
  • 更新 regression set 并配置发版门禁阈值
  • 完成一次回滚演练记录（RTO、责任人、触发条件）
本章先记住 3 件事
  • 坏例是资产：未入库的投诉等于未发生（无法回归）。
  • 归因到可改层：避免「模型不行」笼统结论。
  • 发版门禁先于全量：flywheel ns1 闭环的硬卡口。

选取 3 条线上 bad case

CorpAssist 试点第 4 周起，每周固定 bad case triage（30 分钟）：产品、交付、算法、运维各 1 人。来源优先级：

  1. P0：合规/权限/资金类（24h 内必须入库）
  2. P1：答案错误且无引用、客户书面投诉
  3. P2：体验类（慢、口吻、格式）

归因：数据/工具/Prompt/路由

root cause 4 强制把讨论从「模型笨」拉到可改工程层：

归因层典型信号修复方向BC 示例
数据 引用过期文档、脏 chunk、ACL 漏过滤 索引版本、废止标记、re-embed BC-0412：废止 PDF 未打标
工具 API 未调、假执行、返回超长 契约测试、超时、结果校验 BC-0418：支付 API 未绑定 DST
Prompt 口吻/格式错、越权承诺 模板版本、解码约束 （若 BC-0418 仅改 Prompt 无效）
路由 错模型、错检索路、步数未截断 意图路由表、预算策略 BC-0425：超步数仍写 ERP
四维归因决策（简化）
flowchart TD
  Start[Bad Case] --> Q1{引用/doc 错?}
  Q1 -->|是| Data[数据层]
  Q1 -->|否| Q2{工具/API 错?}
  Q2 -->|是| Tool[工具层]
  Q2 -->|否| Q3{承诺/格式错?}
  Q3 -->|是| Prompt[Prompt层]
  Q3 -->|否| Route[路由层]
  Data --> Fix[修复+验证]
  Tool --> Fix
  Prompt --> Fix
  Route --> Fix

归因结论须对应可测验证（eval 用例或集成测试）。

BC-0412 归因实录（数据层）

  • 假设：索引未同步废止元数据
  • 验证：查 document_version vs 制度废止表
  • 修复：废止 doc 打 status=obsolete；检索过滤；补 12 条 golden

BC-0418 归因实录（工具层，非 Prompt）

  • 假设：退款意图未绑定支付工具
  • 验证：会话 trace 无 payment.refund 调用
  • 修复：DST 强制槽位满才允许生成「已退款」文案

修复闭环与 golden set 更新

continuous improve 的硬交付是 golden update：每条 P0/P1 bad case 至少产生 1 条回归用例。

坏例数据飞轮（NS1）
flowchart LR
  Prod[线上流量] --> Detect[发现: 监控/质检/工单]
  Detect --> Triage[Triage 分级]
  Triage --> RCA[四维归因]
  RCA --> Fix[修复: 数据/工具/Prompt/路由]
  Fix --> Golden[golden set 更新]
  Golden --> Eval[离线 eval]
  Eval --> Gate{发版门禁}
  Gate -->|通过| Release[灰度发布]
  Gate -->|失败| Fix
  Release --> Prod
  Detect -.->|新 bad case| Triage

飞轮转速 = 从发现到 golden 入库的中位天数；目标 < 5 个工作日（P1）。

golden set 更新规范

json 
{
  "case_id": "BC-0412-regression",
  "scene": "S1",
  "query": "2024年差旅报销上限是多少？",
  "expected": {
    "must_cite_doc_status": "active",
    "answer_contains": "2025年修订版",
    "forbid_doc_ids": ["FIN-2020-obsolete.pdf"]
  },
  "tags": ["data", "obsolete-filter"],
  "added_in": "v1.12.0"
}

regression set 与 UAT frozen set 关系：frozen 集变更需双方签字；回归集可每周增量。修复 PR 必须挂「关联 BC-id」。

反模式
  • 只改 Prompt 不补 eval
  • 修复后无对照实验（不知是否引入回归）
  • 废止文档手工删向量库，无元数据规则
推荐
  • 修复 + golden + eval 报告同 PR
  • 对比 v1.11 vs v1.12 diff 准确率
  • 数据修复可脚本化重复执行

发版门禁与回滚演练

release gate 把飞轮产出变成发布条件；rollback drill 证明 gate 失败或线上事故时可恢复。

发版门禁配置示例

yaml 
# corp-assist-release-gate.yaml
gates:
  - name: rag_golden_accuracy
    metric: eval_pass_rate
    dataset: golden_frozen_v3
    threshold: 0.85
    block_on_fail: true
  - name: regression_no_regress
    metric: diff_vs_baseline
    max_drop: 0.02
  - name: p95_latency_smoke
    metric: p95_ms
    threshold: 2500
  - name: security_acl_smoke
    metric: acl_leak_count
    threshold: 0
gray_release:
  stages: [5%, 20%, 100%]
  auto_rollback_on:
    - bad_case_rate_1h > 0.08
    - p95_ms > 3000

回滚演练 Runbook（季度 1 次）

步骤动作记录项
1选定演练窗口，通知运维时间、版本 A→B
2模拟 gate 失败或手动触发 rollback触发原因
3切流量至上一镜像 + 索引快照RTO 实测
4验证 golden smoke 10 条通过/失败
5复盘：谁决策、哪条链路慢改进 ticket

# 演练摘录：回滚到上一 Helm revision

# 跑 smoke eval（CI job）

本章结论

  • 选取 3 条：按 P0/P1/P2 从工单/质检/日志入库，卡片字段齐全。
  • 四维归因：数据/工具/Prompt/路由，结论可测。
  • golden 更新：P0/P1 必增回归用例，与 eval 报告同 PR。
  • 门禁与回滚：发版前 eval_pass_rate；季度 rollback 演练留 RTO 记录。

动手练习

在 demos/advanced-07-badcase-flywheel-lab/badcase-template.md 填写 3 条 bad case（可基于本章示例改写），完成归因、golden 条目与 gate 阈值各 1 项。

操作步骤

  1. 复制 BC-0412/0418/0425 到模板，改为你项目的脱敏描述。
  2. 为每条标注 root-cause-4 一层，并写 1 句验证方法。
  3. 为 BC-0412 写 1 条 JSON 回归用例（可简化 expected）。
  4. 从 gate yaml 抄 2 个 threshold 到模板「发版门禁」节。
  5. 写 5 行 rollback 演练记录（假设 RTO 8 分钟）。

判断练习

  1. （场景）发版前准确率 87%，较上周 86% 仅升 1%，是否可跳过 golden 增量？

    参考答案

    不可。本周 triage 的 P1 坏例必须有条目进 regression；准确率升 1% 不能证明废止文档类问题已覆盖。
