为什么要学本章

TOP5 总答辩 里你承诺了 NS1「飞轮」。本章把承诺落成流程：bad case triage → root cause 4 归因 → golden update → release gate → rollback drill。没有飞轮，生产坑只会重复发生。

学完你能
  • 从线上日志/质检/客户工单选取 3 条 bad case 并完成分级
  • 用数据/工具/Prompt/路由四维归因，写出可验证修复项
  • 更新 regression set 并配置发版门禁阈值
  • 完成一次回滚演练记录（RTO、责任人、触发条件）
本章先记住 3 件事
  • 坏例是资产：未入库的投诉等于未发生（无法回归）。
  • 归因到可改层：避免「模型不行」笼统结论。
  • 发版门禁先于全量：flywheel ns1 闭环的硬卡口。

修复闭环与 golden set 更新

continuous improve 的硬交付是 golden update：每条 P0/P1 bad case 至少产生 1 条回归用例。

坏例数据飞轮（NS1）
flowchart LR
          Prod[线上流量] --> Detect[发现: 监控/质检/工单]
          Detect --> Triage[Triage 分级 P0/P1/P2]
          Triage --> RCA[四维归因]
          RCA --> Fix[修复: 数据/工具/Prompt/路由]
          Fix --> Golden[golden set 更新 + regression 用例]
          Golden --> Eval[离线 eval 回归]
          Eval --> Gate{发版门禁}
          Gate -->|通过| Release[灰度发布]
          Gate -->|失败| Fix
          Release --> Prod
          Detect -.->|新 bad case| Triage

飞轮转速 = 从发现到 golden 入库的中位天数；目标 < 5 个工作日（P1）。

Golden Set 更新规范

json
{
  "case_id": "BC-0412-regression",
  "scene": "S1",
  "query": "2024年差旅报销上限是多少？",
  "expected": {
    "must_cite_doc_status": "active",
    "answer_contains": "2025年修订版",
    "forbid_doc_ids": ["FIN-2020-obsolete.pdf"]
  },
  "tags": ["data", "obsolete-filter"],
  "added_in": "v1.12.0",
  "trace_id": "bc-0412-root"
}

regression set 与 UAT frozen set 关系：frozen 集变更需双方签字；回归集可每周增量。修复 PR 必须挂「关联 BC-id」，并在 PR 描述中附 eval diff。

反模式
  • 只改 Prompt 不补 eval
  • 修复后无对照实验（不知是否引入回归）
  • 废止文档手工删向量库，无元数据规则
  • 修复 PR 不关联 BC-id
推荐
  • 修复 + golden + eval 报告同 PR
  • 对比 v1.11 vs v1.12 diff 准确率
  • 数据修复可脚本化重复执行
  • 修复链路跟进到 golden 入库

发版门禁与回滚演练

release gate 把飞轮产出变成发布条件；rollback drill 证明 gate 失败或线上事故时可恢复。

发版门禁配置示例

yaml
# corp-assist-release-gate.yaml
gates:
  - name: rag_golden_accuracy
    metric: eval_pass_rate
    dataset: golden_frozen_v3
    threshold: 0.85
    block_on_fail: true
  - name: regression_no_regress
    metric: diff_vs_baseline
    max_drop: 0.02
  - name: p95_latency_smoke
    metric: p95_ms
    threshold: 2500
  - name: security_acl_smoke
    metric: acl_leak_count
    threshold: 0
  - name: bad_case_rate
    metric: last_24h_badcase_rate
    threshold: 0.05

gray_release:
  stages: [5%, 20%, 100%]
  interval_minutes: [30, 60]
  auto_rollback_on:
    - bad_case_rate_1h > 0.08
    - p95_ms > 3000
    - availability_1h < 0.99

rollback:
  runbook: "docs/ops/rollback-procedure.md"
  drill_frequency: "quarterly"
  last_drill_result:
    rto_actual: "8 min"
    rpo_actual: "2 min"
    passed: true

飞轮闭环自检

FAQ：修复后准确率不变，但 bad case 确实修了，怎么量化？
在 golden set 中新增该 bad case 的回归用例，eval 时单独看 regression subset 的通过率。总准确率可能不变，但 regression subset 通过率应从 0% 升到 100%。
谁负责维护 golden set？频率是多少？
算法团队维护 frozen set（验收用），QA 团队维护 regression set（周更）。每次 triage 后 PM 跟踪 golden 更新状态。频率：P0 实时更新，P1 本周更新。

飞轮转速度量与持续改进

continuous improve 的核心度量是飞轮转速（Flywheel Velocity）——从发现 bad case 到修复后 golden 入库的中位时间。这个指标直接反映团队的响应和改进效率。

飞轮转速度量表

阶段P0 目标P1 目标P2 目标
发现→Triage≤ 4h≤ 24h≤ 1 周
Triage→归因完成≤ 8h≤ 48h≤ 1 周
归因→修复上线≤ 24h≤ 72h下个迭代
修复→golden 入库≤ 48h≤ 1 周可选
飞轮总耗时≤ 3 天≤ 5 个工作日按迭代
飞轮转速仪表盘
flowchart LR
          BC[Bad Case 发现] --> T[周 Triage 会议]
          T -->|P0 24h| RCA[四维归因]
          T -->|P1 48h| RCA
          RCA --> Fix[修复 + PR]
          Fix --> Review[Code Review + Eval]
          Review -->|通过| Release[发版 + Golden 更新]
          Release --> Prod[生产验证]
          Prod -->|新 bad case| BC
          note[飞轮转速 = 发现到 golden入库的中位天数]

飞轮转速仪表盘应按月更新，目标 P0 < 3 天、P1 < 5 个工作日。

飞轮成熟度模型

级别特征交付课对应的阶段
L1 手工无固定 bad case 收集渠道，修复凭经验PoC 阶段
L2 文档化有卡片模板 + 归因流程，但无自动化试点阶段
L3 自动化自动日志采样 → 触发 triage → 门禁阻断验收后运维期
L4 预测性基于历史数据预测召回率趋势、主动预防持续运营 > 6 个月

作为起步，至少达到 L2（文档化）级别：有卡片模板、有归因流程、有 golden 更新机制。验收时向客户展示飞轮运转情况，是证明「NS1 飞轮」承诺已落地的关键证据。

飞轮反模式
  • Triage 会议无记录，口头说完了事
  • 修复后无验证，不确定是否修好
  • Golden set 从不更新
  • 飞轮转速无人跟踪
飞轮最佳实践
  • 每次 triage 有纪要 + action items
  • 修复后回跑 regression set
  • Golden set 随每次修复增量更新
  • 飞轮转速按月度量并展示
FAQ：飞轮系统应该由谁维护？
建议由算法团队主导，PM 跟踪进度，QA 维护 golden set。每周 triage 会议需要产品/交付/算法/运维各 1 人参加（30 分钟）。飞轮的长期维护责任应在移交文档中写明。

本章结论

  • 选取 3 条：按 P0/P1/P2 从工单/质检/日志入库，卡片字段齐全。
  • 四维归因：数据/工具/Prompt/路由，结论可测可验证。
  • golden 更新：P0/P1 必增回归用例，与 eval 报告同 PR。
  • 门禁与回滚：发版前 eval_pass_rate + regression diff < 2%；季度 rollback 演练留 RTO 记录。
  • 飞轮转速：从发现到 golden 入库的中位天数，P0 < 3 天，P1 < 5 个工作日。

操作步骤

判断练习

  1. （场景）发版前准确率 87%，较上周 86% 仅升 1%，是否可跳过 golden 增量？

    参考答案

    不可。本周 triage 的 P1 坏例必须有条目进 regression；准确率升 1% 不能证明废止文档类问题已覆盖。总准确率可能提高但特定 bad case 子集可能仍为 0%。
