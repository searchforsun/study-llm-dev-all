评审先讲 L1：用户→门户→BFF→RAG→模型/向量库。再展开 L2 具体部署。信任边界用虚线框标注。

L2 部署图关键元素

  • BFF：2 节点 + 负载均衡 + Actuator 健康检查；JVM heap 4G；端口 8080
  • Python RAG：2 节点 + Gunicorn + 内网 LB；端口 50051（gRPC）
  • 向量库：Milvus 2 副本 + 对象存储挂载
  • 监控：Grafana + Prometheus + 结构化日志（JSON 格式）
  • 网关：Kong/APISIX + 限流 + 熔断

Architecture Decision Records (ADR) 模板

markdown
# ADR-001: 选择双栈架构（Spring BFF + Python RAG）

## 背景
CorpAssist 需要同时满足：企业鉴权/合规（Java 生态优势）
与 AI 模型编排（Python 生态优势）。

## 决策
采用双栈：Spring Boot BFF + Python FastAPI RAG，
通过 OpenAPI REST 通信。

## 备选方案
1. 全 Python 单体 — 鉴权与审计不满足企业要求
2. 全 Java 单体 — AI 框架集成周期长

## 影响
- 正：各栈发挥专长，独立发布
- 负：需要维护两套 CI/CD，联调成本增加

## 状态
已批准
先想 10 秒：客户要求「Python 直连公网门户」，图上应改哪条边？

应改为仅 BFF 对门户暴露，Python 在内网；否则 Spring 鉴权与审计链断裂。如果客户坚持，更新 risk register 标注「鉴权链路绕过 BFF」为高风险。

分工

dual stack split 按状态与责任切分，不是「谁更懂 AI」。核心原则：谁拥有这份数据的生命周期，谁就负责维护。

能力域Python（算法）Spring BFF（Java）
文档解析与切块✓ 拥有：解析 Pipeline、Chunk 策略、向量化触发 Job、读状态
RAG 检索 / rerank✓ 拥有：混合检索 + 重排序 + 引用生成聚合响应、限流、熔断
SSO / 部门 ACL消费 BFF 下发的 filter scope✓ 签发 claims + ACL 矩阵
SSE / 会话管理流式生成文本片段✓ 连接管理 + 会话持久化
Prompt 版本✓ 单一事实源禁止第二套副本，只传 scene_id
审计日志输出结构化日志✓ 汇聚 + 持久化 + 合规查询
算法负责人（Python）

对引用率、评测集、索引质量负责；不单独背合同 SLA 口径。

Java 架构（Spring）

对鉴权、熔断、对外 P95、合规审计负责。

项目经理

RACI 与 solution doc 版本对齐 discovery 纪要；维护双方联调计划。

不推荐
  • Python 与 Spring 各维护一套 Prompt
  • BFF 直接调 DashScope 绕过 RAG 服务
  • Java 团队自己写一个 Embedding 客户端
  • 两边都能改检索参数，无统一配置
推荐
  • 契约规定 /v1/chat（REST）仅 BFF→Python
  • Prompt 变更走 PR + golden 回归
  • 检索参数以 Python 配置为准
  • 统一配置中心（Apollo / Nacos）

任务PythonSpringPM客户 IT

RAG 检索质量R/ACII

先想 10 秒：为什么 BFF 不能直接调模型 API 跳过 RAG 服务？

因为 ACL 过滤、检索引用生成、Prompt 组装都在 Python RAG 服务中。BFF 直接调模型会绕过 ACL 与引用链路，导致合规风险和不一致的回答格式。

接口

openapi contract 是双栈主 integration point。PoC 前建议冻结 /v1/chat、/v1/ingest/status、/v1/eval/run 三条核心路径。

三条核心路径说明

路径方法用途调用方生产者
/v1/chatPOST问答请求（带 ACL scope）BFFPython RAG
/v1/ingest/statusGET查询索引任务状态BFFPython Index Worker
/v1/eval/runPOST触发离线评测CI/CDPython Eval 服务
yaml
openapi: 3.0.3
info:
  title: CorpAssist RAG API
  version: "1.0.0"
  description: Python RAG 服务接口（仅内网 BFF 调用）
servers:
  - url: http://rag-service.internal:50051/api/v1
paths:
  /v1/chat:
    post:
      summary: CorpAssist 问答（BFF 调用，内网）
      parameters:
        - in: header
          name: X-Trace-Id
          required: true
          schema: { type: string }
        - in: header
          name: X-Acl-Scope
          required: true
          schema: { type: string }
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [query, top_k]
              properties:
                query:
                  type: string
                  maxLength: 2000
                  description: "用户输入（已脱敏）"
                top_k:
                  type: integer
                  minimum: 1
                  maximum: 20
                  default: 5
                scene_id:
                  type: string
                  enum: [s1_rag, s2_cs, s3_agent]
                  description: "场景标识，BFF 传 scene_id，Python 端加载对应 Prompt 版本"
                citations_required:
                  type: boolean
                  default: true
      responses:
        "200":
          description: 成功响应 + 引用列表
          content:
            application/json:
              schema:
                type: object
                properties:
                  answer: { type: string }
                  citations:
                    type: array
                    items:
                      type: object
                      properties:
                        doc_id: { type: string }
                        score: { type: number }
                        snippet: { type: string }
                  trace_id: { type: string }
        "422":
          description: 参数校验失败
        "504":
          description: 模型 API 超时

契约测试门禁

  • 版本：URL 前缀 /v1；破坏性变更升 v2。
  • 错误码：4xx 可重试策略；5xx 触发 BFF 降级话术。
  • 门禁：consumer/provider 契约测试进 CI；任意 breaking change 阻断发布。
  • 字段兼容性：新增字段用 optional + default；不移除已有字段。

不推荐
  • 无版本号，直接改生产接口
  • Java 和 Python 各自写文档，不一致
  • 无 trace_id 贯穿，排障靠猜
  • 错误码混用，Java 5xx 当 Python 4xx
推荐
  • OpenAPI 统一管理 + 版本号
  • trace_id 贯穿所有服务
  • 契约测试（contract testing）进 CI
  • 错误码与服务端 ProblemDetail 对齐
客户要求「接口文档放二期」？
PoC 可简化为 3 路径 Markdown 附件，但字段名与验收指标不得口头漂移，否则 验收与 SLA 无法对齐。建议将核心 API 的字段定义写入合同附件。
Python RAG 接口需要鉴权吗？
需要。Python RAG 服务在内网也应做服务间鉴权——推荐 mTLS 或 service token（BFF 签发）。不鉴权的内网接口是常见安全隐患。
变更 OpenAPI 审批流程是什么？
OpenAPI 变更需要双方架构师审批：① 在 Git 上发起 PR ② 标注 breaking/ non-breaking ③ Java + Python 架构师 approve ④ 更新契约测试 ⑤ 同步更新 solution doc。

本章结论

三条对应上文：架构图、分工、接口——three artifacts that define the delivery baseline。

  • 架构图：L1 业务视图 + L2 部署；标注 SSO、向量库、通义与审计链。
  • 分工：Python 管道与评测；Spring 编排、鉴权、对外 SLA。RACI 矩阵明确责任。
  • 接口：OpenAPI 冻结 3 条路径；契约测试先于大规模联调。
