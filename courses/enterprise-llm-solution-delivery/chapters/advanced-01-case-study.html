双栈

CorpAssist · 某省国企知识助手 客户要求：内网 Spring 门户 + 通义私有化 RAG；合规组只审 Java 侧日志；算法组维护 Python 索引与检索。你需要在方案评审里讲清「谁负责什么、出问题查哪条链路」。

corpassist full 在本课指 S1~S5 场景在统一平台上的落地切片——本章以 S1 企业 RAG + S2 智能客服 为主线，其余场景在 TOP5 答辩章展开。

双栈分工表（方案文档必备）

能力Spring 栈Python 栈契约
用户鉴权与会话 spring role：Gateway + BFF JWT 校验 BFF 转发的 service token Header X-Corp-User + mTLS 可选
知识检索 路由、限流、缓存热点 query python role：混合检索 + rerank openapi contract /api/v1/rag/search
文档索引 触发任务、展示进度 解析 PDF、分块、向量化 异步 job API + webhook 回调
客服转人工 DST 状态机、工单对接 意图分类（可选） 事件总线 / 同步 API

dual stack split 的原则：业务编排与合规在 Java，模型与数据管道在 Python。避免「两边都能改 Prompt」导致发版责任不清。

常见双栈反例
  • Python 直接对门户暴露公网 API
  • BFF 内嵌 200 行 Prompt，与 Python 路由重复
  • 索引 job 无 idempotency，重跑产生重复 chunk
CorpAssist 推荐
  • 单一 integration point：OpenAPI + 统一 trace_id
  • Prompt 版本在 Python；BFF 只传 scene_id
  • 索引以 document_version 去重

yaml

FAQ：PoC 阶段能否先单 Python 栈演示？

可以作 quick win，但方案文档须标注「试点前必须补 BFF + ACL」。否则 PoC 指标与生产架构脱节，属于后续章讲的 demo to prod gap。

指标

企业交付里，key metric 必须能回答：客户在什么节点签字、运维按什么告警、算法按什么迭代。CorpAssist 案例采用三阶段指标包。

指标目标示例采集方式
问答准确率（golden set） ≥ 80%（50 条标注集） 离线 eval 脚本
引用可追溯率 ≥ 95% 回答带 doc_id 响应 JSON 校验
P95 延迟 < 2.5s（内网） demo metric 看板
指标目标示例采集方式
日活 / 采纳率 试点部门 60% 周活 门户埋点
转人工率 < 18%（S2 客服） BFF 会话日志
Bad case 率 < 5%（抽样质检） pilot kpi 周报
Token 成本 / 千次 预算内（见 cost model） token cost 账单

合同验收 KPI（与 PoC 区分）

markdown 
## 验收 KPI 摘录（合同附件）
| 编号 | 指标 | 阈值 | 评测方法 |
|------|------|------|----------|
| K1 | 标准集准确率 | ≥ 85% | 双方见证 eval_report |
| K2 | 服务可用性 | ≥ 99.5% / 月 | SLA 监控 |
| K3 | P95 响应 | ≤ 2.0s | 压测 + 生产抽样 |
| K4 | 安全 | 无 P1 数据泄露 | 渗透 + 审计

**缓冲条款**：PoC 80% 准确率仅作参考；合同 K1 以 UAT 前 frozen golden set 为准。

指标设计要点：PoC 用 success criteria 争取立项；试点用 feedback channel 驱动迭代；验收用 contract sla 与 eval report 锁定签字依据。

三阶段指标对照总表

阶段目的典型指标常见误区
PoC 争取立项、Demo Day golden 50 条 ≥80%、引用率、P95 把单次 demo 数字写进合同
试点 运营反馈、坏例 triage 周活、转人工率、bad case 率、token 成本 无 pilot kpi 周报
验收 sign-off、SLA 生效 frozen set ≥85%、可用性 99.5% 评测脚本与 PoC 不一致
先想 10 秒：PoC 演示准确率 88%，合同要求 85%，能否直接写进合同「已达标」？

不能。PoC 集规模、数据版本、评测脚本可能与 UAT 不同；须单独跑 frozen golden set 并出具 eval_report，避免「PoC 指标与合同脱节」。

复盘

# CorpAssist 项目复盘 — XX 国企知识助手

## 1. 背景与范围

- 周期：Discovery 2w → PoC 3w → Pilot 8w → 验收 2w

| 维度 | 结果 | 证据 |

## 3. 做得好的
- 提前冻结 OpenAPI，Java/Python 并行开发
- 试点第 2 周建立 bad case 群，每周 triage

## 4. 待改进
- **决策**：索引 ACL 晚于上线 1 周才补齐
  - **影响**：试点前 3 天出现跨部门文档泄露风险（已 hotfix）

## 5. 下轮（飞轮 / 运维）

- 移交 runbook + oncall 轮值表

复盘不是「项目结束才做」；试点期每两周做 mini-review，避免问题堆积到验收。

复盘一句话公式

  • 决策：我们当时选了什么（双栈 / 模型 / 范围）
  • 证据：指标、日志、客户原话
  • 代价：时间、成本、风险
  • 行动：下轮必须改的 1~3 条（可进 Jira）
