为什么要学本章

CorpAssist 交付经历在 HR 眼里不是「调过 API」，而是能讲清业务结果 + 工程边界的证据。本章把 CorpAssist 完整案例 压缩为 dual track resume（交付轨 + 技术轨）与可背诵的 interview story。

学完你能
  • 写 1 条 compound profile 项目 bullet：场景 + 双栈 + 量化结果
  • 用 star method 展开 3 分钟项目答辩
  • 区分 tech highlight 与 impact metric 的篇幅分配
  • 准备 3 个 STAR 故事库应对不同面试追问方向
本章先记住 2 件事
  • 复合型描述：同一段经历同时服务「解决方案/项目经理」与「Java/Python 工程师」两类岗位，靠双轨 bullet切换 emphasis。
  • STAR 不是流水账：Action 写你的决策；Result 写可核对指标或客户签字节点。

复合型描述

同一段 CorpAssist，两种读者 解决方案岗 HR 关心：范围控制、PoC→验收、客户 satisfaction 度。技术岗面试官关心：双栈分工、RAG 检索策略、SLA 与降级。你需要两套 bullet，共享同一事实源（案例复盘）。

resume llm 类项目的通病：堆砌「LangChain / RAG / Agent」名词，缺少 ToB 语境。CorpAssist 的 project narrative 应锚定：行业、角色、交付阶段、可量化结果。

简历结构：项目区 vs 技能区

区块写什么CorpAssist 示例避免
项目经历双轨 bullet + 1 个数字「Spring BFF + Python RAG，UAT 86%」堆 10 个框架名词
技能关键词与 bullet 呼应的技术栈Spring AI、OpenAPI、Milvus、DashScope与项目无关的「精通 CUDA」
交付关键词PoC / 试点 / 验收阶段动词discovery、sign-off、SLA、eval_report只写「参与大模型项目」

对于非「主导」角色的同学（实习/开发岗），使用准确动词：「负责 Python 检索模块」「实现 BFF 会话路由」「设计 golden set 评测脚本，支撑 UAT 86%」。勿虚构 sign-off 角色。

复合型 bullet 公式

  • 动词 + 行业场景 + 双栈/机制 + 数字/签字节点
  • 交付轨数字偏：周期、周活、sign-off、客户部门数
  • 技术轨数字偏：P95、准确率、可用性、索引规模
FAQ：没有「主导」权限，怎么把项目写好看？

用准确的动词 + 量化结果。例如：「负责 Python RAG 检索模块开发，设计混合检索 + ACL 校验，支撑 UAT 正确率 86%」。重点突出你的决策与贡献，而不是虚构管理角色。

STAR 面试叙事

star method 把 3 分钟项目答辩结构化为四段。面试官通过 STAR 评估的是你的决策能力和工程思维，不是工具熟练度。

STAR 完整示例（CorpAssist ACL 晚补齐）

markdown
## STAR — 索引 ACL 风险与补救（3 分钟版）

**S（Situation — 30s）**
某省国企 CorpAssist 试点第 2 周，HR/IT 两部门同时上线 RAG；
合规要求「员工只能看本部门文档」，但索引管线尚未接入 ACL 矩阵。
1.2 万份文档已索引完成，撤回成本极高。

**T（Task — 20s）**
我负责 Python 检索与 BFF 联调，需在 **3 天内** 消除跨部门泄露风险，
且不能回滚已索引的文档库。

**A（Action — 90s，重点）**
1. 与合规组 frozen ACL 矩阵（部门→doc_tag 映射），写入 risk register。
2. 检索层增加 acl_scope 必填校验；BFF 从 SSO 注入 scope。
3. 离线脚本按 tag 重建向量分区；hotfix 灰度 10%→20%→100%。
4. 补 UAT 用例 20 条「跨部门负例」，纳入 golden set 回归集。
5. 复盘将「ACL sign-off」前置到 Discovery checklist。

**R（Result — 30s）**
- 48h 内 hotfix 上线，0 起 P1 泄露事件。
- UAT 准确率仍达 86%（未因 hotfix 引入回归）。
- 复盘输出 1 条流程改进：ACL sign-off 写入 discovery 门禁。
- 该案例成为团队「安全交付」最佳实践。
  • S：行业、场景、约束（合规/工期），30 秒内建立可信度。
  • T：你的责任边界 + 成功标准（数字或日期）。
  • 忌：S 段讲 5 分钟公司史；T 段模糊「帮忙做项目」。
  • A：2~4 个决策动作，含 trade-off（为何 hotfix 而非全量重索引）。
  • R：指标 + 组织结果（复盘、流程改进）。
  • 忌：A 只列工具名；R 只有「项目成功」。

90 秒口述提纲（面试卡）

markdown
1. 【15s】国企知识助手，Spring+Python 双栈，我负责 ___。
2. 【20s】难点：___（合规 / 延迟 / 范围控制）。
3. 【35s】做了 A1、A2、A3（各一句决策 + trade-off）。
4. 【20s】结果：指标 + 流程改进；可接 TOP5 某场景深挖。

先想 10 秒：STAR 的 Action 写「用了 LangChain」够不够？

不够。要写你的决策：为何选混合检索、为何 acl_scope 必填、灰度策略如何定。工具名放从句，决策放主句。「用 LangChain 实现混合检索」比「用了 LangChain」好很多。

简历排版与 ATS 优化

现代企业招聘使用 ATS（Applicant Tracking System）筛选简历。CorpAssist 项目的简历需要同时通过机器筛选和人工筛选两道关。

ATS 关键字段布局

ATS 字段CorpAssist 示例命中关键词
职位名称LLM 解决方案架构师大模型、RAG、ToB
技能标签Spring AI / Python / RAG / OpenAPI双栈技术栈
项目经验CorpAssist 国企知识助手合规、ACL、PoC、验收
量化结果UAT 86% / P95 1.8s / 15 周交付指标驱动

简历排版黄金法则

  • 一页原则：3 年以上经验尽量控制在一页；5 年以上最多两页。CorpAssist 项目放第一页中部。
  • 动词开头：每行 bullet 以「主导/设计/实现/优化」开头，不用「参与/协助」。
  • 数字优先：每条 bullet 至少含 1 个数字（时间、指标、规模、团队大小）。
  • STAR 精选：简历中写 Result（数字），Situation 和 Task 留到面试口述。
简历常见问题
  • 堆砌 10 个框架名无项目上下文
  • 「参与」「协助」过多
  • 无数字、无量化结果
  • 简历与岗位 JD 关键词不匹配
简历优化
  • 3 个重点项目各 3-4 条 bullet
  • 动词精准 + 数字量化和结果
  • 关键词从目标 JD 提取
  • PDF 命名含姓名和岗位
FAQ：项目经历与目标岗位不匹配怎么办？
使用双轨 bullet 中的「技术轨」匹配技术岗、「交付轨」匹配解决方案岗。如果目标岗位是纯算法，需要突出 CorpAssist 中评测集设计、Golden set 管理、模型选型等内容。
简历造假/夸张的后果？
面试官会深挖细节：如果写了「独立完成 sign-off」，会被追问签字流程、客户决策人信息、合同条款。「双轨叙事」要求同一项目的两套说法数据一致。夸张会导致面试穿帮，严重者影响背调结果。

本章结论

  • 复合型描述：同一案例写交付轨与技术轨 bullet，共享复盘事实源。每条含动词+场景+数字。
  • STAR：S/T 建约束，A 写决策（含 trade-off），R 写可核对指标与流程改进。准备 3 则故事库应对追问。
  • 简历排版：一页原则 + 动词开头 + 数字优先 + ATS 关键词匹配。

操作步骤

  1. 从「双轨 bullet 模板」改写出交付轨 1 条：含阶段、至少 1 个 impact metric。
  2. 写技术轨 1 条：含双栈分工 + 1 个 tech highlight（如 ACL、OpenAPI、混合检索）。
  3. 按 STAR 示例填满 S/T/A/R 四段；A 至少 3 条编号动作，含 trade-off 说明。
  4. 用「90 秒口述提纲」压缩 STAR，朗读计时 ≤ 100 秒。
  5. 写 2 个模拟追问的应答草稿。

判断练习

  1. （找错）下列简历 bullet 哪句最可能引发面试追问后穿帮？A) 「UAT 准确率 86%，见 eval_report_v3」B) 「个人独立完成客户 sign-off」C) 「Spring BFF + Python RAG 双栈交付」

    参考答案

    B。sign-off 通常由项目经理/客户方负责人完成；实习或开发岗写「独立完成 sign-off」易被深挖授权与流程。应改为「支撑 UAT 评测与签字材料，团队按期 sign-off」。

  2. （场景）面试官问：「你们 PoC 88%，简历写 86% UAT，是不是造假？」如何用 STAR 口径回应？

    参考答案

    不回避差异：PoC 50 条 demo 集 88% 用于立项；UAT 用 frozen 200 条 golden set 测得 86% 并写入合同 K1。强调评测集版本、脚本 hash、eval_report 签字可核对；Result 写流程改进（PoC 指标不再直接进合同）。
