为什么要学本章

CorpAssist 交付经历在 HR 眼里不是「调过 API」，而是能讲清业务结果 + 工程边界的证据。本章把 CorpAssist 完整案例 压缩为 dual track resume（交付轨 + 技术轨）与可背诵的 interview story。

学完你能
  • 写 1 条 compound profile 项目 bullet：场景 + 双栈 + 量化结果
  • 用 star method 展开 3 分钟项目答辩
  • 区分 tech highlight 与 impact metric 的篇幅
  • 完成 Demo STAR 草稿并对照 rubric 自检
本章先记住 2 件事
  • 复合型描述：同一段经历同时服务「解决方案/项目经理」与「Java/Python 工程师」两类岗位，靠双轨 bullet切换 emphasis。
  • STAR 不是流水账：Action 写你的决策；Result 写可核对指标或客户签字节点。

复合型描述

同一段 CorpAssist，两种读者 解决方案岗 HR 关心：范围控制、PoC→验收、客户满意度。技术岗面试官关心：双栈分工、RAG 检索策略、SLA 与降级。你需要两套 bullet，共享同一事实源（案例复盘）。

resume llm 类项目的通病：堆砌「LangChain / RAG / Agent」名词，缺少 ToB 语境。CorpAssist 的 project narrative 应锚定：行业、角色、交付阶段、可量化结果。

双轨 bullet 模板

markdown 
### 交付轨（解决方案 / 项目经理 emphasis）
- 主导某省国企 **CorpAssist 知识助手** ToB 交付（Discovery→验收 15 周）：
  协调业务/合规/算法三方，PoC 3 场景获立项；试点 2 部门 **周活 62%**；
  UAT **准确率 86%** 达成合同 K1，按期 sign-off。

### 技术轨（Java / Python emphasis）
- 设计 **Spring BFF + Python RAG** 双栈：OpenAPI 契约驱动并行开发；
  混合检索 + ACL 双重校验；索引 job 幂等；生产 **P95 1.8s**，可用性 **99.6%**。
要素交付轨侧重技术轨侧重共用事实源
开头动词 主导 / 交付 / 协调 设计 / 实现 / 优化 案例复盘「背景」
中间主体 阶段、干系人、范围 架构、栈、关键机制 双栈分工表
结尾结果 签字、满意度、周期 延迟、准确率、可用性 三阶段指标 / eval_report
弱描述
  • 「负责大模型项目开发」
  • 「使用 RAG 提升问答效果」
  • 「参与 PoC 演示」
复合型强描述
  • 行业 + 场景（国企知识助手）
  • 双栈 + 1 个技术决策（OpenAPI 并行）
  • 数字 + 阶段（UAT 86%，15 周交付）

写 bullet 时控制篇幅：每条 1~2 行；impact metric 优先于技术栈清单。技术栈放技能区或第二条 bullet 的从句。

双轨简历信息流向
flowchart TB
  Case[CorpAssist 案例复盘]
  Case --> D[交付轨 bullet]
  Case --> T[技术轨 bullet]
  D --> PM[解决方案 / PM 岗位]
  T --> ENG[Java / Python / 架构岗]
  Case --> STAR[STAR 故事库]
  STAR --> INT[面试 3min 叙事]

案例复盘是唯一事实源；改指标须同步更新简历与 STAR，避免面试口径不一致。

FAQ：没有「主导」权限，实习/开发岗怎么写？

用准确动词：「负责 Python 检索模块」「实现 BFF 会话路由」。Result 仍写团队达成指标，Action 写你个人贡献（如「设计 golden set 评测脚本，支撑 UAT 86%」）。勿虚构 sign-off 角色。

简历结构：项目区 vs 技能区

区块写什么CorpAssist 示例避免
项目经历 双轨 bullet + 1 个数字 「Spring BFF + Python RAG，UAT 86%」 堆 10 个框架名词
技能关键词 与 bullet 呼应的技术栈 Spring AI、OpenAPI、Milvus、DashScope 与项目无关的「精通 CUDA」
交付关键词 PoC / 试点 / 验收阶段动词 discovery、sign-off、SLA、eval_report 只写「参与大模型项目」

复合型 bullet 公式

  • 动词 + 行业场景 + 双栈/机制 + 数字/签字节点
  • 交付轨数字偏：周期、周活、sign-off、客户部门数
  • 技术轨数字偏：P95、准确率、可用性、索引规模

STAR

star method 把 3 分钟项目答辩结构化为四段。CorpAssist 示例如下（可裁剪为 90 秒版）。

STAR 完整示例（CorpAssist ACL 晚补齐）

markdown 
## STAR — 索引 ACL 风险与补救

**S（Situation）**
某省国企 CorpAssist 试点第 2 周，HR/IT 两部门同时上线 RAG；
合规要求「员工只能看本部门文档」，但索引管线尚未接入 ACL 矩阵。

**T（Task）**
我负责 Python 检索与 BFF 联调，需在 **3 天内** 消除跨部门泄露风险，
且不能回滚已索引的 1.2 万份文档。

**A（Action）**
1. 与合规组 frozen ACL 矩阵（部门→doc_tag 映射），写入 risk register。
2. 检索层增加 acl_scope 必填校验；BFF 从 SSO 注入 scope。
3. 离线脚本按 tag 重建向量分区；hotfix 灰度 10%→100%。
4. 补 UAT 用例 20 条「跨部门负例」，纳入 golden set。

**R（Result）**
- 48h 内 hotfix 上线，0 起 P1 泄露事件。
- UAT 准确率仍达 86%；复盘将「ACL sign-off」前置到 Discovery。
- 该条进入发版门禁 checklist（衔接 badcase 飞轮章）。
  • S：行业、场景、约束（合规/工期），30 秒内建立可信度。
  • T：你的责任边界 + 成功标准（数字或日期）。
  • 忌：S 段讲 5 分钟公司史；T 段模糊「帮忙做项目」。
  • A：2~4 个决策动作，含 trade-off（为何 hotfix 而非全量重索引）。
  • R：指标 + 组织结果（复盘、流程改进）。
  • 忌：A 只列工具；R 只有「项目成功」。

模拟追问（准备 1 个坑位故事）

  • 「如果没有 48 小时，你的 Plan B？」→ 只读模式 + 缩小试点部门。
  • 「准确率 86% 如何评测？」→ golden set 规模、frozen 版本、eval_report。
  • 「双栈里 Java 做了什么？」→ BFF 鉴权、会话、转人工，Python 纯检索。
先想 10 秒：STAR 的 Action 写「用了 LangChain」够不够？

不够。要写你的决策：为何选混合检索、为何 acl_scope 必填、灰度策略如何定。工具名放从句，决策放主句。

90 秒口述提纲（面试卡）

markdown 
1. 【15s】国企知识助手，Spring+Python 双栈，我负责 ___。
2. 【20s】难点：___（合规 / 延迟 / 指标）。
3. 【35s】做了 A1、A2、A3（各一句决策）。
4. 【20s】结果：指标 + 流程改进；可接 TOP5 某场景深挖。

Action 占近半时长：面试官主要评估你的决策与 trade-off，不是工具清单。

STAR 弱 Action
  • 「使用了 LangChain 和 Milvus」
  • 「配合团队完成开发」
  • 「优化了 Prompt」无前后指标
STAR 强 Action
  • 「frozen ACL 矩阵，检索层 acl_scope 必填」
  • 「48h 灰度 10%→100%，补 20 条负例进 golden set」
  • 「将 ACL sign-off 前置到 Discovery checklist」

本章结论

  • 复合型描述：同一案例写交付轨与技术轨 bullet，共享复盘事实源。
  • STAR：S/T 建约束，A 写决策，R 写可核对指标与流程改进。

动手练习

基于 advanced-01 案例，在 demos/advanced-02-resume-lab/star-draft.md 完成双轨 bullet 各 1 条与 1 则完整 STAR（可选用 ACL 或 PoC 指标故事）。

操作步骤

  1. 从「双轨 bullet 模板」改写出交付轨 1 条：含阶段、至少 1 个 impact metric。
  2. 写技术轨 1 条：含双栈分工 + 1 个 tech highlight（如 ACL、OpenAPI、混合检索）。
  3. 按 STAR 示例填满 S/T/A/R 四段；A 至少 3 条编号动作。
  4. 用「90 秒口述提纲」压缩 STAR，朗读计时 ≤ 100 秒。

判断练习

  1. （找错）下列简历 bullet 哪句最可能引发面试追问后穿帮？
    A) 「UAT 准确率 86%，见 eval_report_v3」
    B) 「个人独立完成客户 sign-off」
    C) 「Spring BFF + Python RAG 双栈交付」

    参考答案

    B。sign-off 通常由项目经理/客户方负责人完成；实习或开发岗写「独立完成 sign-off」易被深挖授权与流程。应改为「支撑 UAT 评测与签字材料，团队按期 sign-off」。

  2. （场景）面试官问：「你们 PoC 88%，简历写 86% UAT，是不是造假？」如何用 STAR 口径回应？

    参考答案

    不回避差异：PoC 50 条 demo 集 88% 用于立项；UAT 用 frozen 200 条 golden set 测得 86% 并写入合同 K1。强调评测集版本、脚本 hash、eval_report 签字可核对；Result 写流程改进（PoC 指标不再直接进合同）。
