学完你能

  • 主持或参与 peer review：发现 scheme defect 并给出可执行改进
  • 完成 capstone layer1→Layer-3 衔接自检

本章先记住 2 件事
  • 评审 rubric 不是走形式：每项须映射到证据（架构图、eval_report、risk register、runbook）。
  • 互评找 defect，不找面子：用 defense rubric 统一语言，避免「感觉不行」式反馈。

评审 rubric

CorpAssist Layer-3 答辩 学员提交完整方案包（双栈架构图、PoC/试点指标、反模式清单、移交草案）。评审组（同学 + 助教）按 rubric 打分，低于阈值项必须返工后再进入 TOP5 总答辩。

review rubric 将主观「方案好不好」拆成可观测检查项。CorpAssist Layer-3 采用四维 20 分制（每维 1~5 分）。

CorpAssist 方案评审 Rubric（Layer-3）

维度5 分（达标）3 分（部分）1 分（缺失）证据
架构与双栈 分工清晰、OpenAPI 契约、ACL/鉴权链完整、信任边界标注 有双栈图但缺接口或 ACL 单栈或 PoC 架构未更新 solution doc、arch diagram
指标与验收 PoC/试点/合同三套指标 + eval 方法 + frozen set 签字 仅 PoC 指标 口头承诺无 KPI eval_report 模板、contract sla 摘录
风险与反模式 risk register + tech_debt 偿还计划 + over promise 识别 仅列风险无 owner 未见 over promise 识别 anti-pattern-list、tech_debt.yaml
运维与飞轮 runbook、oncall、坏例飞轮草案、版本升级流程 有移交文档无 oncall 无降级与回滚 handover doc、release gate 说明

及格线建议：总分 ≥ 14/20 且无任一维度 ≤ 2。Architecture 与指标两维为一票否决项（ToB 交付核心）。

## 方案评审 — 三座大山速查（advanced-06 预检）
| 大山 | Layer-3 最低要求 | 常见 scheme defect |
|------|-------------------|-------------------|
| NS1 飞轮 | 坏例收集渠道 + golden set 更新流程草案 | 只有「后续优化」无 owner |
| NS2 确定性 | 降级/转人工/规则桥接写进架构 | 100% 押 LLM 概率输出，无 fallback |
| NS3 成本 | token 估算或模型分层策略 | 无 pilot 成本监控 |

**defense rubric 附加项**：每场景 1 个生产坑（TOP5 章详述）
低质量评审
  • 只评 PPT 美观度
  • 无证据链，「感觉架构合理」
  • 不记录 defect，口头说完即散
  • 评分没理由，只有数字
rubric 驱动评审
  • 每项对照证据文档（架构图、eval_report）
  • defect 编号 + 严重度 + 返工建议
  • score 写入 rubric-self-score.md
  • 评分附 1 句理由 + 证据链接
FAQ：Layer-3 与 Layer-2 TOP5 答辩区别？

Layer-3 评整包方案（文档、指标、运维、反模式）；Layer-2 评五场景深度（架构细节 + 生产坑 + 三座大山）。先过 Layer-3 确保方案无结构性 defect，再在 TOP5 练口述与追问。

互评

peer review 流程：提交 → 匿名或固定搭档互评 → 缺陷汇总 → 24h 内 feedback incorporate → 复评。角色分工建议：

角色职责产出
方案作者提交包、回应 defect、更新文档revision log
主评（Peer A）按 rubric 四维打分 + ≥3 条 defectpeer-review 表单
副评（Peer B）交叉验证证据是否存在证据链接 checklist
助教/教练仲裁分歧、确认是否放行 Layer-2solution grade 签字

## Peer Review 议程 — CorpAssist 方案

| 时间 | 环节 | 说明 |
|------|------|------|
| 0~5m | 作者陈述 | 仅讲架构+指标+最大风险，不念 PPT 全文 |
| 5~20m | 主评提问 | 对照 rubric 四维，每维至少 1 问 |
| 20~35m | 开放挑战 | 副评/观众：over promise、tech debt、ACL |
| 35~40m | defect 汇总 | 编号 D-01…，严重度 P0/P1/P2 |
| 40~45m | 作者复述 | 24h 内 incorporate 计划（3 条以内） |

互评人提交前自检

先想 10 秒：互评时「架构图不好看」算不算有效 defect？

不算有效 scheme defect，除非影响可读性导致无法验证 ACL 链路。应改为「架构图未展示 BFF→Python 鉴权头」或「信任边界未标注」等可验证项。

作者答辩常用反问（准备证据）

  • 「accuracy 指标如何复现？」→ eval_report + golden set 版本号 + 脚本 hash
  • 「PoC 债何时还？」→ tech_debt.yaml payoff_plan 日期
  • 「厂商 SLA 不达标怎么办？」→ 降级 + 缓存 + 合同免责条款 + runbook
  • 「试点 KPI 是什么？」→ 日活率/采纳率/转人工率（见 practice-02）

FAQ：Layer-3 评审通过后还能改方案吗？
通过评审意味着方案基线已冻结（baseline）。后续修改需要走变更控制：小修改（如格式）直接更新；大修改（架构/指标变更）需重新评审。建议在评审通过后做一次 git tag 标记基线版本。

本章结论

  • 评审 rubric：架构、指标、风险、运维四维 20 分制；证据驱动，Architecture/指标为否决项。
  • 互评：defect 编号 + severity + incorporate 闭环；通过后进入 TOP5 Layer-2 答辩。
  • 方案包：标准化目录结构；评审后基线冻结，大修改重新评审。

动手练习

填写 demos/advanced-04-capstone-lab/rubric-self-score.md 四维自评，并任选一位同学方案做 3 条 peer defect 草稿。

操作步骤

  1. 按 rubric 表为自有方案四维各打 1~5 分并写理由（各 1 句 + 证据链接）。
  2. 对照「三座大山速查」表，标出 NS1~NS3 最低要求是否满足。
  3. 写 3 条 scheme defect（含 id、dimension、severity、recommendation）。
  4. 列出 24h 内 feedback incorporate 的 3 项行动。

判断练习

  1. （场景）某组 Layer-3 总分 16/20，但「指标与验收」维度仅 2 分（合同 KPI 直接抄 PoC）。是否应放行 TOP5 答辩？

    参考答案

    不应放行。指标维度为一票否决项；2 分表示 over promise 风险未消除。须返工：改写 KPI 条款、补 UAT eval 方法，复评该维 ≥3 后再进入 Layer-2。
