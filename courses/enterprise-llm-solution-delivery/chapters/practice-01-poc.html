为什么要学本章

Discovery 冻结范围与双栈方案后，poc design 是客户第一次看到可用的产品。若 PoC 只是「在笔记本电脑上跑通一个 demo」，试点必然出现指标脱节、架构返工与信任危机。本章用 CorpAssist S1 知识库 RAG 演示场景讲透「有脚本、有指标、有反馈」的 PoC 方法论。

学完你能
  • 输出带场景、数据准备与通过条件的 poc script
  • 设计 demo metric 看板：准确率、延迟、引用追溯率
  • 建立 PoC 后 feedback loop，与试点指标对齐
  • 区分 quick win 与长期架构债，写入 risk register
本章先记住 3 件事
  • PoC 不是技术验证：它是客户买你解决方案能力的证据，不是看你会不会调 API。
  • 脚本先行：先写 poc scope 再搭建，否则演示当天会手忙脚乱。
  • 指标绑定合同：PoC 演示用的指标与 UAT 验收指标必须是同一套 metrics 的不同阈值。

演示脚本

CorpAssist · 华北制造 PoC 演示（25 分钟） IT 总监、HR 负责人、合规经理三人评审。演示需覆盖：制度 RAG 问答 + 引用溯源 + ACL 隔离效果；每个场景 3~5 分钟；预留 10 分钟开放式提问。你只有一次印象分。

poc script 是演示的「乐谱」——规定每段话术、每个点击、预期客户反应与应对。写好脚本，演示成功率提升 60%。

## CorpAssist PoC 演示脚本 — 2026-05-31

### 场景 1：制度 RAG（5 分钟）
**话术**：「下面展示 CorpAssist 如何回答员工制度问题。」
**操作**：
1. 门户输入「2024 年差旅报销标准」
2. 指出回答中引用标记 [1][2] → 点击展开源文档
3. 切换日期筛选「仅显示 2025 年修订版」
**预期客户反应**：「能不能搜到更多非制度类文档？」
**应对**：「一期聚焦制度库，FAQ 类在试点范围。」

### 场景 2：ACL 隔离（3 分钟）
**话术**：「下面证明跨部门权限在检索层生效。」
**操作**：
1. HR 账号搜「研发绩效方案」
2. 系统返回「未找到相关文档」
3. Admin 账号同搜 → 显示研发文档
**关键 message**：「权限在 BFF 签发、检索层过滤，双栈保障。」

### 场景 3：客服转人工（5 分钟）
**话术**：「复杂请求走 DST 状态机，不会僵死。」
**操作**：
1. 输入「帮我退款」→ 收集订单号、原因
2. 超 3 步未完成 → 自动转人工
3. 工单推送给客服系统
**预期客户反应**：「转人工的触发条件能否自定义？」
**应对**：「可以，映射到 DST 步数预算配置。」
不推荐
  • 无脚本，现场随意提问
  • 只演示理想 case，不准备异常场景
  • 数据准备不充分，检索结果空白
  • 演示当天仍在改代码
推荐
  • 脚本提前 3 天定稿，内部 smoke test
  • 准备 1 个异常场景（如无结果 → 拒答话术）
  • 演示环境与 QA 环境一致，数据提前索引
  • 演示前 24h 冻结代码与 Prompt 版本

指标

success criteria 决定 PoC 是否通过——不是领导是否微笑，而是指标是否达标。CorpAssist PoC 一般设 3 个必达指标、2 个参考指标。

指标计算方式目标展示方式

python 
# PoC 指标计算 — 演示前运行
import json

def evaluate_poc(golden_set_path: str, responses_path: str) -> dict:
    with open(golden_set_path) as f:
        golden = json.load(f)
    with open(responses_path) as f:
        responses = json.load(f)

    correct = 0
    cite_count = 0
    for g, r in zip(golden, responses):
        if g["expected_answer"] in r["answer"]:
            correct += 1
        if r.get("citations") and len(r["citations"]) > 0:
            cite_count += 1

    total = len(golden)
    return {
        "accuracy": correct / total,
        "cite_rate": cite_count / total,
        "total_queries": total
    }

# 输出：{'accuracy': 0.84, 'cite_rate': 0.96, 'total_queries': 50}

PoC 指标 → 合同 KPI 映射表

PoC 指标PoC 目标合同 KPI 目标差异说明
Golden set 准确率 ≥ 80%（50 条） ≥ 85%（200 条 frozen） 数据集规模与版本不同
P95 延迟 < 2.5s（演示环境） < 2.0s（生产环境） 生产环境有 BFF + ACL 链路
引用追溯率 ≥ 95% ≥ 95% 阈值可一致

demo metric 的关键是：演示时实时展示数据，而不是事后给 Excel。客户看到 Grafana 上的 P95 曲线比任何 PPT 都更有说服力。

项目经理

演示前准备：脚本 + 检查清单 + 数据准备 + 环境预热。

算法工程师

确保 golden set 离线 eval 通过 + 异常场景备选。

Java 架构师

BFF 链路稳定 + 演示环境 API 不超时 + ACL 校验通过。

客户问：「PoC 80% 准确率，为什么合同只写 80% 而不是 90%？」

解释：PoC golden set 50 条为常用场景精选集；合同 KPI 使用 frozen 200 条全面集，难度更高。建议 PoC 写「≥ 80%（50 条）」，合同写「≥ 85%（200 条）」以示提升空间。

反馈

feedback loop 在 PoC 完成后启动，48 小时内发出结构化问卷，7 天内召开复盘会。反馈不仅用于改进产品，更是记录客户 verbal 承诺（如「我们优先上线 HR 部门」）。

关于 quick win：PoC 期间发现的「改一行 Prompt 就能提升 5% 准确率」的优化，应在复盘前完成。这能大幅提升客户对交付团队的信心。

FAQ：PoC 客户很满意但预算被砍了，怎么办？

使用 PoC 成果作为案例素材转入 business case：整理指标提升数据、客户背书（需脱敏）。同步更新 risk register：「客户预算风险」状态改为激活，准备 1~2 页 ROI 估算（工作量与成本估算 参考）。

PoC 演示中客户提出新需求，该当场答应吗？

记录但不当场承诺。在 Q&A 环节说：「很好的场景，我们会纳入试点评估范围。」会后更新 scope boundary 与 manday estimate。当场答应的最大风险就是 over promise。

本章结论

PoC 是客户对你交付能力的第一次全面检验，不是技术爱好者的玩具时间。

  • 演示脚本：3 场景 × 5 分钟 + smoke test 门禁；数据与代码在演示前 24h 冻结。
  • 指标：PoC 指标 ≠ 合同 KPI；看板实时展示比 PPT 更有说服力。
  • 反馈：48h 问卷 + 7d 复盘 + 录入 risk register；PoC 后的 quick win 快速交付建立信任。

操作步骤

  1. 复制正文「演示脚本详细模板」到 poc-script.md，将场景改为你负责的项目（如智能问答、知识助手等）。

  4. （可选）准备 1 个异常场景的 Plan B 话术（如模型超时）。

判断练习

  1. （场景）演示前 2h 发现向量库响应变慢，P95 从 1.2s 升到 4.5s。你优先做？A) 临时切小模型 B) 通知客户延期 C) 按原计划演示、口头说「还在优化」

    参考答案

    A 最优。切小模型（如 qwen-turbo）牺牲一点准确率保住演示体验；同时准备离线截图展示正常延迟。C 会降低信任。B 是最后选项。

  2. （排障）演示中客户说「回答没有引用来源」。最可能原因？A) Prompt 没要求引用 B) 检索命中 0 条 C) 前端渲染 Bug

    参考答案

    A/B 都可能，优先查 A。检查 Prompt 中是否强制要求 citationsRequired=true；再看检索日志是否有命中。前端渲染 Bug 属于第三排查项。
