Incident Flow 故障响应流程

incident flow 定义从发现故障到恢复的全过程：

Incident 响应流程
flowchart TD
          Detect[故障发现: 告警/用户报修] --> Triage{严重级评估}
          Triage -->|P0: 核心功能不可用| P0[立即响应: 15min 内确认]
          Triage -->|P1: 功能部分受损| P1[2h 内确认]
          Triage -->|P2: 体验问题| P2[24h 内确认]
          P0 --> Fix[修复/回滚/降级]
          P1 --> Fix
          P2 --> Sprint[排入下个迭代]
          Fix --> Verify[验证通过]
          Verify --> RCARCA[5Why 根因分析]
          RCA --> Prevent[更新 runbook + 告警]

Incident flow 的关键是「先止血再找因」——P0/P1 先降级恢复，事后再 RCA。

Runbook 模板（典型故障）

markdown 
## Runbook: CorpAssist 模型 API 超时

### 症状
- BFF 日志出现 `timeout calling DashScope API`
- Grafana 看板 P95 延迟 > 3s
- 用户反馈「一直在转圈」

### 影响
S1 RAG + S2 客服均不可用（核心功能中断，P0）

### 排查步骤
1. 确认 DashScope 状态：
   `curl https://dashscope.aliyuncs.com/api/v1/services/ping`
2. 检查 BFF 到 DashScope 网络：
   `curl -w "%{http_code} %{time_total}" https://dashscope.aliyuncs.com`
3. 检查 BFF 线程池状态：
   `curl http://bff:8080/actuator/threaddump | grep "pool-.*-thread"`

### 恢复措施
1. 紧急降级到缓存 + 转人工（BFF 配置 `feature.llm.fallback=true`）
2. 若因 API Key 配额超限：联系阿里云提配额
3. 若因网络故障：检查 VPC/VPN 连接

### 验证
- 降级后：回答切换为「正在检索...」或转人工
- 超时恢复后：手动切回正常模式

### 事后
- 更新 risk register：是否需增加备用模型路由？
- 更新告警阈值：P95 > 2.5s 提前预警

kb ops 指知识库运维——文档更新、索引重建、过期文档清理。必须定义操作频率与流程：

  • 文档更新：客户业务方通过 Dify 后台或 API 更新 → 触发增量索引
  • 索引重建：每月一次全量重建（修复一致性）或在 Embedding 模型升级时
  • 过期文档：业务方标记 status=obsolete → 检索过滤 → 定期清理向量
  • 权限变更：部门合并/拆分时更新 ACL 矩阵 → 重新索引受影响文档
不推荐
  • 无值班表，出问题临时找人
  • Incident 处理完就结束了，无 RCA
  • 知识库从来不更新，半年后准确率下降
推荐
  • L1/L2/L3 值班表 + 节假日备班
  • P0/P1 必须生成 RCA 与预防措施
  • KB ops 有固定频率与责任人

版本升级流程

version policy 管理三类版本的变更：模型版本、索引版本、Prompt 版本。每一类都需要独立的版本号、变更流程与回滚方案。

版本类型版本号示例变更触发回滚方式通知范围
模型 qwen-plus-2026-05 → 0601 厂商新版本发布 切回旧模型别名 客户 PM + 运维
索引 idx-v42 → idx-v43 文档更新 / Embedding 升级 恢复上一版索引快照 客户 PM + 业务方
Prompt prompt-v1.12 → v1.13 优化指令 / 修复 bad case Git revert + 重新部署 交付团队内部

升级审批流程

升级变更流程
flowchart TD
          Reg[变更申请] --> CR[CR 评审: 影响评估 + 回滚方案]
          CR --> Test[测试环境验证]
          Test -->|通过| Approve[审批: PM + 客户 IT]
          Test -->|失败| Reg
          Approve --> Deploy[灰度发布: 5% → 20% → 100%]
          Deploy --> Monitor[监控 24h]
          Monitor -->|回滚触发| Rollback[回滚]
          Monitor -->|稳定| Close[变更关闭 + 文档更新]

每次升级走完 CR → 测试 → 审批 → 灰度 → 监控 五步；模型升级建议在测试环境先跑 golden set 回归。

upgrade process 的黄金法则：每次升级必须可回滚。模型升级前：先冻结旧版本别名；索引重建前：先快照；Prompt 升级前：先 git tag。

yaml 
# corp-assist-upgrade-policy.yaml
upgrade_policy:
  # 模型版本升级
  model:
    notification_lead_days: 7    # 提前 7 天通知客户
    pre_upgrade_steps:
      - "测试环境运行 golden set 回归"
      - "冻结旧模型别名作为回滚点"
    rollback: "切换 model_alias 到旧版本"

  # 索引版本升级
  index:
    notification_lead_days: 3
    pre_upgrade_steps:
      - "创建当前索引快照"
      - "灰度 10% 查询走新版索引"
    rollback: "恢复快照 + 切流量"

  # Prompt 版本升级
  prompt:
    pre_upgrade_steps:
      - "git tag current-version"
      - "灰度 5% 流量验证"
    rollback: "revert git commit + 重新部署"
客户 IT 运维

按 runbook 做日常运维与 L1 值班；负责知识库内容更新。

交付团队

运维期 L2/L3 支持；版本升级执行；季度巡检与报告。

业务方

FAQ：模型 API 升级后准确率下降，能在生产直接回滚吗？

可以，前提是你提前冻结了旧版本的 model_alias。在 BFF 配置中将 model_alias 指向旧版本即可立即切回。这也是为什么模型升级前必须保留旧版本别名。

移交后客户 IT 能力弱，能否一直让交付 team 运维？

可以约定运维服务期（通常 3~6 个月，可付费续期）。但建议在运维期内逐步转移：第一个月全权运维、第二个月协同、第三个月监督。到期后客户 IT 应已具备独立运维能力。可将此写入 SLA 附件。

索引重建需要多长时间的停机？

零停机——通过双索引切换实现：先在新索引流水线重建，重建完成后切查询流量到新索引，旧索引保留为回滚快照。这个过程通常 2~4 小时。

本章结论

移交不是把所有文件打包丢给客户就结束了——它是一套让接盘团队能独立运维的完整体系。

  • 移交文档：6 模块可操作文档，重点是运维场景而非功能说明。
  • 值班与运维：L1/L2/L3 三层值班 + incident flow（止血→RCA→预防）+ KB ops 定期维护。
  • 版本升级：管理模型/索引/Prompt 三类版本；升级必走 CR→测试→审批→灰度→监控五步；每步可回滚。

判断练习

  1. （排障）BFF 健康检查失败，但 Python RAG 正常。最可能的根因？A) 数据库连接池满 B) DashScope API Key 过期 C) 向量库宕机

    参考答案

    A。BFF 健康检查失败通常与 BFF 内部依赖（DB/Redis）有关。DashScope API Key 过期（B）会在 RAG 层而非 BFF 层表现。向量库宕机（C）影响 Python RAG 而非 BFF。

  2. （策略）客户要求「每周升级一次模型」。按 version policy，你应如何回应？

    参考答案

    建议改为「月度升级」或按厂商发布节奏。每次模型升级需要：测试环境 golden 回归（1~2 天）+ 灰度观察（2~3 天）。周更会导致测试不足或频繁灰度影响用户体验。
