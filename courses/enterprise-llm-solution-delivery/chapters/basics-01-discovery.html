已完成需求调研与范围

约 35 分钟 · 阶段：需求与方案 · 能力：主持客户访谈、划定 MVP 边界并建立风险登记册

为什么要学本章

ToB 大模型项目若跳过 delivery discovery，常见结果是 PoC 演示漂亮、试点却卡在数据权限或验收指标对不上。本章用 CorpAssist 企业知识助手立项场景，把「谁说了算、做什么、不做什么、会踩什么坑」在写方案前钉死。

学完你能
  • 列出业务 / IT / 安全三方 stakeholder map 及各自成功指标
  • 输出带 Out of Scope 的 mvp scope 与 requirement doc 骨架
  • 维护 risk register Top5，并衔接 双栈方案设计文档
本章先记住 3 件事
  • 访谈不是聊天：要有议题、纪要模板与「决策人签字」的验收假设。
  • 边界写「不做」：scope boundary 比功能列表更能防范围蔓延。
  • 风险前置：数据合规、幻觉、集成复杂度要在 PoC 前登记，而非验收时才暴露。

访谈

CorpAssist · 制造业知识库立项（15 分钟） 业务希望「制度问答 + 引用溯源」；IT 要求接入现有 OA 与 SSO；安全要求文档不出域、审计留痕。你需要在一次 client interview 里同时听到三类诉求，而不是只问「要什么功能」。

企业级 discovery 通常分三轮：业务痛点 → 数据与系统集成 → 合规与运维。每轮产出写入同一份需求纪要，避免「口头答应、文档没有」。

  • 业务方：场景优先级（客服 / 知识库 / Agent）、成功指标（节省人天、首次解决率）。
  • IT：身份源、API 网关、现有向量库或 ES、发布窗口。
  • 安全 / 合规：数据分级、出境、内容安全、工具调用白名单。

访谈纪要模板（节选）

markdown
## CorpAssist Discovery — {客户简称} — {日期}
### 1. 业务痛点与成功指标
- 主场景：□ S1 知识库 RAG  □ S2 客服  □ 其他
- 量化目标：引用率 ≥ __% ；P95 ≤ __s ；人工转接率下降 __%

### 2. 数据与权限
- 文档源：OA / 网盘 / DB ；更新频率：__
- ACL：部门 / 职级 / 项目 ；检索层是否过滤：□ 是 □ 否

### 3. 决策人与签字假设
- 业务负责人：__ ；IT 负责人：__ ；验收签字人：__
不推荐
  • 只 demo 通义聊天，不记录数据源与权限
  • 成功指标只有「领导满意」
  • 未确认谁能在合同上签字验收
推荐
  • 每场景对应可测 KPI（引用率、坏例率、成本/千次）
  • 访谈当场确认 MVP 只覆盖 1 个主场景
  • 纪要 48 小时内回传客户确认
先想 10 秒：若客户坚持「先全量接入 10 年制度库」，你第一句话回应什么？

应回到 MVP 数据子集 + 权限试点：全量接入是二期，一期用可审计的小集合验证解析、ACL 与指标。

边界

scope boundary 要在方案阶段写清「不做清单」，并与合同附件一致。CorpAssist 一期常见边界如下。

在 MVP 内明确不做（Out of Scope）理由
单部门制度 RAG + 引用 全集团 20+ 系统实时写操作 NS2 工具链与审批未就绪
DashScope 公有云 API 客户机房内全量微调 工期与数据标注不在一期
评测集 50～100 条 Golden 「100% 不出错」口头承诺 需 SLA + 回归门禁，见 验收与 SLA

MVP 范围与交付物对照

Discovery → MVP 边界冻结
flowchart TD
  Interview[客户访谈] --> Scope[MVP 范围 + 不做清单]
  Scope --> Sign[双方签字纪要]
  Sign --> Arch[双栈方案文档]
  Arch --> PoC[PoC 脚本与指标]

边界冻结后才进入 双栈方案设计文档，避免架构随需求漂移。

业务想要的

「什么都问、越快越好」——需收敛到 1 个场景 + 可测 KPI。

交付方能做的

双栈 PoC、试点、评测报告——需写清人天与依赖。

合同能签的

指标、验收方法、除外责任——与 PoC 演示脚本一致。

风险

大模型交付的典型风险不是「模型不够聪明」，而是数据、合规与集成。建议在 risk register 中至少登记以下类别。

风险触发信号缓解措施责任人
脏 PDF / 扫描件解析差 召回率上限卡在 60% 解析链路 + 人工抽检；见 S1 生产坑 数据工程师
ACL 未在检索层过滤 试点出现越权片段 检索前 filter；安全评审 架构师
幻觉导致错误制度解读 无引用或错误引用 强制 citation；拒答策略 算法 / 产品
PoC 指标与合同脱节 演示 90% 、验收 70% PoC 脚本 = 合同 UAT 子集 项目经理
过度承诺无 SLA 口头「7×24 秒回」 写清 P95、可用性、除外 商务 + 架构
客户说「先不上评测，上线后再优化」怎么办？
可接受分阶段，但 MVP 必须保留最小 Golden Set 与发版回归，否则无法兑现 NS1 飞轮。应在风险登记中标注为「验收争议高风险」。
discovery 和 PoC 之间要不要签合同？
建议至少 signed 的 discovery 纪要或 SOW 附件：范围、数据责任、知识产权与保密。PoC 合同指标应直接引用纪要中的 KPI 表述。

本章结论

三条对应上文三节：访谈、边界、风险。

  • 访谈：三方 stakeholder、可量化 KPI、48h 纪要确认——避免只 demo 不落地。
  • 边界：MVP + Out of Scope 书面化，与 PoC / 合同同一套指标语言。
  • 风险：数据、ACL、幻觉、指标脱节、过度承诺进 risk register，并指定责任人。

动手练习

为 CorpAssist 虚构客户「华北制造」填写 discovery 纪要、MVP 边界表与 risk register 各一页（约 25 分钟），产出物放在 demos/basics-01-discovery-lab/。

操作步骤

  1. 复制正文「访谈纪要模板」到 demos/basics-01-discovery-lab/discovery-notes.md，补全业务 KPI、数据源与签字人（对照 stakeholder map 表）。
  2. 在「边界」决策表中新增一行：写明「二期 Agent 改单」为何放在 Out of Scope（引用 NS2）。
  3. 从「风险」表选 3 条标为 High，并写 1 条缓解措施与责任人（可照 risk register 格式）。
  4. （可选）用 Mermaid 流程图向同事口述 Discovery → MVP 冻结路径（1 分钟）。

判断练习

  1. （判断）客户要求 PoC 演示「全集团 12 个系统实时改单」。按本章边界原则，你应优先推进哪一项？A) 接 12 个 API 做 Demo B) 书面拒绝并提议一期仅 RAG 问答 C) 先口头答应、合同再改

    参考答案

    B。改单属 NS2 工具链，超出 MVP；C 制造验收风险，A 拉长 PoC 且无法验收。应更新 Out of Scope 并给替代路径（知识库问答 + 引用）。

  2. （场景）排障顺序：① 换更大模型 ② 补 discovery 纪要 ③ 加风险登记 ④ 直接写架构图。最应先做？

    参考答案

    ② 然后 ③。无纪要则架构与模型选型都缺验收锚点；① 无法解决范围与数据问题。
