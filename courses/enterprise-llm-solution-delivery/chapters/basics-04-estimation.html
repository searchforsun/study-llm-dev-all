缓冲

客户接口延迟、脏 PDF 抽检、需求变更——建议 15% 风险储备。

不加人天

Out of Scope 的 Agent 改单、全集团推广、移动端 App。

FTE 计划

按角色：PM 0.5 + 架构 0.5 + 算法 1 + Java 0.5 + QA 0.5 = 3 人全期投入。

客户砍预算先砍测试人天，你如何回应？

保留最小 golden set 与发版回归，否则无法验收；可砍非核心 UI 或文档美化，不砍 eval。如果必须砍测试人天，更新 risk register 标注验收风险。

Token 成本估算

token cost = 月请求量 ×（输入+输出 Token/次）× 单价。CorpAssist 制度问答可按 2k 输入 + 500 输出估算。但注意：检索注入的 chunk 文本也计入输入 token。

python
# Token 成本计算器 — CorpAssist 场景
def estimate_token_cost(
    monthly_queries: int,
    input_tokens_per_query: int,
    output_tokens_per_query: int,
    price_per_1k_input: float,
    price_per_1k_output: float,
) -> dict:
    monthly_input_tokens = monthly_queries * input_tokens_per_query
    monthly_output_tokens = monthly_queries * output_tokens_per_query

    input_cost = monthly_input_tokens / 1000 * price_per_1k_input
    output_cost = monthly_output_tokens / 1000 * price_per_1k_output

    return {
        "monthly_queries": monthly_queries,
        "monthly_input_tokens": monthly_input_tokens,
        "monthly_output_tokens": monthly_output_tokens,
        "monthly_token_cost": input_cost + output_cost,
        "yearly_token_cost": (input_cost + output_cost) * 12,
    }

# S1 RAG 场景估算（qwen-plus）
s1 = estimate_token_cost(
    monthly_queries=50000,
    input_tokens_per_query=2500,   # 提问 + 检索注入
    output_tokens_per_query=500,
    price_per_1k_input=0.008,
    price_per_1k_output=0.008,
)
print(f"S1 月 Token 成本: ¥{s1['monthly_token_cost']:.2f}")
print(f"S1 年 Token 成本: ¥{s1['yearly_token_cost']:.2f}")

# S2 客服场景估算（qwen-turbo，更便宜）
s2 = estimate_token_cost(
    monthly_queries=30000,
    input_tokens_per_query=1500,   # 会话历史 + 意图识别
    output_tokens_per_query=300,
    price_per_1k_input=0.002,
    price_per_1k_output=0.002,
)
print(f"S2 月 Token 成本: ¥{s2['monthly_token_cost']:.2f}")
print(f"S2 年 Token 成本: ¥{s2['yearly_token_cost']:.2f}")

客户问「Token 成本能固定包月吗」？

可以设计混合模式：基础 QPS 以内包月，超出按量计费。也可提供「Token 预充值」方案。关键是让客户理解 Token 成本的弹性属性，避免「无上限」的恐惧。

# ROI 估算 — CorpAssist 知识库项目
annual_infra_cost = 4750 * 12      # 公有云资源 ¥57,000
annual_token_cost = 1500 * 12       # Token 成本 ¥18,000

project_total = 450000               # 交付总费用（一次性）

annual_opex = annual_infra_cost + annual_token_cost + annual_labor_cost

# 客户节省估算（减少客服人力）

infra cost 通常被低估。建议在 quote sheet 中列出所有假设（QPS、数据量、模型版本），并标注「超出假设范围的费用另行评估」。

客户认为 Token 成本太高，怎么解释？
拆分场景成本：S1 RAG（¥1,200/月）对应 5 万次咨询，约 ¥0.024/次，远低于人工成本（客服每次约 ¥5-10）。同时展示缓存策略和模型分层（S2 用 qwen-turbo）的降本效果。
Pilot 期间 Token 成本比预估高 30%，正常吗？
Pilot 初期 Token 成本通常偏高（20-40%），原因：① 用户探索性提问，上下文更长 ② 缓存尚未预热 ③ 初始 Prompt 版本未优化。建议：Pilot 前两周按 130% 预算准备，第三周起逐步回归预估线。如果 Pilot 第四周仍高于 120%，需要检查模型路由配置和缓存策略。
私有化对比公有云，几年回本算合理？
通常 2-3 年的总拥有成本（TCO）对比是分水岭。如果 3 年公有云 Token 费 + 云资源费超过私有化 CapEx 的 60%，私有化就值得考虑。但需额外计算运维人力成本（私有化每年约加 1 人）。建议做 3 年和 5 年两个 TCO 方案供客户选择。
Quote sheet 需要包含利润率吗？
内部版本需要，客户版本不需要。给客户的报价单应展示：人天费率 + Token 成本 + 硬件费用（如适用）+ 总价。内部版本需额外计入管理费、风险储备和期望利润率。

成本控制三板斧

交付过程中，Token 成本和 Infra 成本容易超支。三种主要的降本措施：

  1. 模型分层：S1 用 qwen-plus（高精度），S2 用 qwen-turbo（低成本），S3 Agent 限步数。按场景分配模型可降低 40-60% Token 成本。
  2. 缓存策略：高频问题（如「年假几天」）缓存答案，缓存命中率可达 20-30%，显著降低重复 Token 消耗。
  3. 用量监控：BFF 层记录每次请求的 Token 消耗，按场景/部门生成成本报表。设置月度预算告警，超额自动通知。
python
# 成本监控看板 — 按月汇总
def monthly_cost_report(usage_log: list) -> dict:
    """从 BFF usage log 生成月度成本报表"""
    scene_costs = {}
    for record in usage_log:
        scene = record["scene_id"]
        tokens = record["total_tokens"]
        model = record["model"]
        price = MODEL_PRICES[model]
        cost = tokens / 1000 * price
        scene_costs[scene] = scene_costs.get(scene, 0) + cost
    return scene_costs

  • 所有场景用同一模型
推荐

与 roi estimate 的关系：ROI = 客户节省成本 - 总投入。通常 2 年内收回投资是可接受的。示例：3 个客服岗位年省 ¥240,000，项目总投入 ¥450,000，投资回收期约 1.9 年。

FAQ：报价报了 100 人天，客户砍到 60，怎么应对？

不直接降价。重新梳理 WBS：砍掉非核心项（如 UI 美化、二期预留），保留核心链路（RAG、ACL、评测）。同时在 risk register 中标注「人天不足导致的风险」。底线：不能砍测试与安全相关人天。

Token 成本能不能包月？

可以设计混合模式：保底 QPS 以内包月，超出按量计费。另一种方案：客户预充值 Token 包（如 100 万 Token），用完后自动告警。包月的好处是预算可预期，但需要合理估算用量，避免过低导致频繁超支。

预算管理与成本对账

报价交付后，cost model 需要持续管理。CorpAssist 运维期的成本对账机制确保 Token 和云资源支出在预算范围内。

对账项频率数据来源阈值超支处理
Token 消耗周DashScope 账单 API月度预算 80% 预警检查缓存命中率/模型路由
云资源费用月云厂商账单月度预算 90% 预警检查实例规格是否合理
人天实际 vs 计划双周工时系统偏差 > 20%PM 评估是否需变更 CR

建议在 BFF 层记录每次请求的模型、Token 消耗、场景 ID，定时同步到内部成本看板。这不仅能对账，还能发现异常调用（如某个场景 Token 突然增长 200%）。

不推荐
  • 等月底账单来了才看成本
  • 从不分析 Token 消耗趋势
  • 人天超支无预警
推荐
  • 周级 Token 消耗监控
  • BFF 日志自动同步成本看板
  • 超支自动告警

本章结论

  • 人天：WBS 对齐 MVP；含风险缓冲（15%）与依赖标注。每行对应具体交付物。
  • Token：含检索注入上下文；分场景估算；缓存降低实际成本。BFF 记录 usage 对账。
  • 硬件：随部署形态（公有云 Opex / 私有化 CapEx）；并入 quote sheet 与 ROI 计算。
