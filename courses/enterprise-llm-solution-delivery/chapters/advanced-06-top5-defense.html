  • 完成 demos/advanced-06-top5-defense-lab/top5-cheat-sheet.md 与章节测验
本章先记住 4 件事
  • 答辩顺序：先业务价值与边界 → 架构分层 → 指标 → 生产坑 → 三座大山如何兜底。
  • 生产坑必须「可观测」：能说清怎么发现、怎么止血、怎么进飞轮（见 坏例复盘与飞轮设计）。
  • 双轨叙事共享事实源：交付轨讲 PoC/试点/合同；技术轨讲 RAG/Agent/路由，数字一致。
  • S1 必修：选修 ≥1 门场景课，但 TOP5 总答辩仍须能口述 S2~S5 的架构骨架。

S1 企业 RAG 架构与指标

S1 · 企业知识库 RAG制度/合同/运维手册问答，回答必须带引用；内网 ACL 按部门隔离。面试常问：混合检索、评测集、索引更新策略。

s1 rag defense 核心：检索增强 + 引用溯源 + 权限在检索层生效。CorpAssist 双栈下，Spring BFF 传 acl_scope，Python 检索服务在向量/关键词两路均过滤。

S1 企业 RAG 分层架构
flowchart TB
          subgraph ingest [索引链路]
            OSS[文档 OSS] --> Parse[解析 OCR/表格]
            Parse --> Chunk[分块+元数据]
            Chunk --> Embed[Embedding]
            Embed --> VDB[(向量库)]
          end
          subgraph query [在线问答]
            Q[用户问题] --> Hyb[混合检索 BM25+向量]
            Hyb --> ACL[ACL 过滤]
            ACL --> Rerank[重排]
            Rerank --> Gen[LLM 生成+引用]
          end
          VDB --> Hyb
          Gen --> Cite[引用 doc_id 回传 BFF]

解析与索引在 Python；BFF 只做鉴权、限流与审计日志。索引版本变更必须触发 re-embed 计划。

答辩必讲指标（NS1 + NS3）

指标目标示例北极星
golden set 准确率PoC ≥80% / UAT ≥85%NS1
引用可追溯率≥95% 带 doc_idNS1
P95 延迟内网 <2.5sNS3
索引新鲜度 SLAT+1 或按合同NS1

本场景必答生产坑（答辩高光）

坑：脏 PDF / 扫描件
  • 无 OCR/表格还原 → 召回上限锁死在 60%
  • 客户质疑「模型不行」实为数据管道缺失
防御
  • discovery 阶段签「可解析文档占比」基线
  • 解析质量进 golden set 分层（扫描件子集）
  • PoC 演示前先跑 50 页样本解析报告

另两条高频坑：ACL 未在检索层过滤；纯向量忽略制度 Graph 关联。

S2 智能客服与降级

S2 · 企业智能客服HR/IT 服务台场景，意图分类 + 槽位收集 + 知识库查询。面试常问：降级策略、转人工条件、成本控制。

s2 cs defense 核心：DST（对话状态跟踪）状态机 + 降级兜底 + 转人工流畅。

S2 客服降级架构
flowchart TD
          User[用户提问] --> Intent[意图分类]
          Intent -->|高置信| DST[DST 槽位收集]
          Intent -->|低置信| Human[转人工]
          DST -->|槽位不足| Confirm[确认+追问]
          DST -->|槽位满| Query[知识库查询]
          Query -->|LLM API 可用| Answer[生成回答]
          Query -->|LLM 超时/故障| Fallback[缓存/默认回复/转人工]
          Answer -->|需要二次确认| Human

降级不是失败——它是系统设计的一部分。90% 请求走正常链路，10% 降级不中断服务。

本场景必答生产坑

坑：转人工链路断裂——Agent 承诺「已退款」但未绑定支付工具 API。防御：DST 强制槽位满才允许生成「已退款」文案；支付等敏感操作必须工具调用 + 结果校验。

三座大山对照

大山本场景要点
NS1 飞轮客服对话采样 → bad case triage → golden 更新
NS2 确定性DST 状态机 + 转人工网关；不纯靠 LLM 概率
NS3 成本qwen-turbo 用于客服（低成本），缓存常见问

S3 Agent 工作流

S3 · 企业 Agent工单审批、数据查询、跨系统操作。面试常问：步数预算、工具路由、Token 成本控制。

s3 agent defense 核心：工具定义 + 步数预算 + 结果验证 + 人工确认闸门。

S3 Agent 工作流结构
flowchart TD
          Task[用户任务] --> Plan[工具规划]
          Plan --> Call[工具调用 API]
          Call --> Verify[结果校验]
          Verify -->|成功| Next{更多步骤?}
          Verify -->|失败| Retry[重试/降级]
          Next -->|Y| Plan
          Next -->|N| Summary[结果汇总]
          Summary --> Confirm{需人工确认?}
          Confirm -->|Y| Human[等待审批]
          Confirm -->|N| Done[完成]
          Retry -->|超步数| Human

Agent 必须有步数预算（默认 5 步）和超步降级，否则循环调用会消耗大量 Token。

本场景必答生产坑

坑：Agent 超步仍调用 ERP 写接口——Agent 第 4 步仍调用 ERP 后超时，导致脏数据。防御：步数预算硬限制（最大 5 步）+ 写操作前人工确认闸门。

坑：工具调用返回过长——ERP API 返回 10K+ JSON 撑爆上下文。防御：截断 + 结构化提取关键字段。

S4 代码库 RAG

S4 · 代码库问答内部代码库/技术文档检索，要求代码片段可编译验证。面试常问：代码分块策略、重排、跨文件引用。

s4 code defense 核心：代码结构感知分块 + 跨文件引用追踪 + 可编译验证。

代码 RAG 分块策略

策略适用块大小
基于函数/类分块方法级检索20~100 行
基于文件分块文件概览整个文件
基于 AST 分块精确引用追踪语法节点

本场景必答生产坑

坑：跨文件引用丢失——单文件分块导致「这里调用了 format_date()」但搜索不到该函数定义。防御：解析 import 关系建立跨文件索引。

S5 内容生成流水线

S5 · 内容生成周报总结、公告生成、报告摘要。面试常问：格式控制、风格一致、批量成本。

s5 content defense 核心：模板驱动 + 格式约束 + 人工审核门禁。

内容生成流水线结构

  • 输入：结构化数据 / 文档 / 要点
  • 模板：Pregenerated 模板 + Slot 填充
  • 约束：长度、风格、敏感词过滤
  • 审核：自动规则 + 人工抽样

本场景必答生产坑

坑：批量生成无差异化——50 份周报内容高度雷同。防御：注入每份文档的独特点（数据/指标/事件），设置多样性约束。

three mountains（NS1 飞轮/NS2 确定性/NS3 成本）是每个场景的通用评估框架：

答辩常见失误
  • 5 场景平均用力，无重点
  • 只讲架构不讲生产坑
  • 指标混用（PoC 数字当合同 KPI）
  • 三座大山未提及
高分答辩结构
  • S1 重点讲（必修），S2~S5 骨架 + 各自生产坑
  • 每场景「架构+指标+坑+三座大山」四段式
  • 数字口径一致（PoC/试点/合同分级）
  • 结尾 1 分钟用飞轮/确定性/成本收束

答辩速查卡片（最终版）

markdown
## TOP5 答辩速查卡 — 每人 30 秒/场景

### S1 RAG
- 架构：混合检索 + ACL 过滤 + 引用溯源
- 指标：准确率 ≥85% (UAT frozen) / 引用率 ≥95%
- 坑：脏 PDF 无 OCR → 解析报告 + golden 分层
- 三山：飞轮(golden 迭代) 确定(强制引用) 成本(plus)

### S2 客服
- 架构：DST 状态机 + LLM 意图 + 转人工网关
- 指标：转人工率 <18% / 解决率 ≥65%
- 坑：假执行退款 → DST 槽位满才出文案
- 三山：飞轮(采样 triage) 确定(状态机) 成本(turbo)

### S3 Agent
- 架构：工具规划 + 步数预算 + 结果校验
- 坑：超步写 ERP → 步数硬限 + 写前确认
- 三山：飞轮(工具日志) 确定(步数+确认) 成本(限步)

### S4 代码
- 架构：AST 分块 + 跨文件引用
- 坑：跨文件引用丢失 → import 关系索引

### S5 内容
- 架构：模板 + 约束 + 审核
- 坑：批量无差异 → 独特点注入

答辩实战技巧与时间分配

TOP5 总答辩通常为 30-45 分钟，包含：5 分钟开场陈述 + 20-30 分钟场景答辩 + 5-10 分钟 Q&A。以下是答辩实战技巧。

30 分钟答辩时间分配方案

时段内容技巧
0-5 min开场：CorpAssist 案例全景（行业/双栈/指标）一句话说清：做什么+用什么+结果如何
5-12 minS1 RAG（重点，含生产坑）架构 1 张图 + 2 个指标 + 1 个坑 + 三座大山
12-18 minS2 客服 + S3 Agent各 3 分钟，重点讲降级和步数预算
18-23 minS4 代码 + S5 内容各 2.5 分钟，含核心坑
23-25 min三座大山总结 + 飞轮闭环用 NS1/NS2/NS3 收束
25-30 minQ&A准备 3 个高频追问的应答

高频追问与应答模板

markdown
Q: 「S1 RAG 你们的检索策略是什么？」
A: 混合检索（BM25 + 向量）→ Rerank → ACL 过滤 → LLM 生成。
   选择混合检索的原因：制度类文档需要关键词精确匹配（如「年假 15 天」），
   纯向量会丢失精确匹配能力。Rerank 解决 Top-K 精度问题。

Q: 「如果模型 API 挂了怎么办？」
A: 三层降级：① 缓存常见问答（兜底 30% 流量）
   ② 转人工话术（兜底 50% 流量）
   ③ 熔断后走 fallback 入口（兜底 100%）
   以上都在 BFF 层配置，不依赖 LLM。

Q: 「你的飞轮怎么转的？」
A: 线上 bad case → 周 triage → 四维归因 → 修复 → golden 更新
   → 发版门禁 → 灰度。P0 坏例 24h 入库，P1 本周修复。
   目标：从发现到 golden 入库中位时间 < 5 个工作日。

答辩准备检查清单

  • 准备 S1 完整答辩卡片（架构图+指标+坑+三座大山）
  • S2~S5 各准备骨架版卡片（至少含坑）
  • 计时演练 3 遍（30 分钟 / 15 分钟 / 5 分钟三个版本）
  • 准备 3 个高频追问的应答（检索策略/降级/飞轮）
  • 数字口径与简历一致（PoC/试点/合同分阶段）
FAQ：答辩时被问到不会的问题怎么办？
诚实说「这个方向我没有深入做过，但我的理解是...」比硬编答案好。也可以引导到你知道的领域：「这个具体指标我记不清了，但我可以讲讲我们的评测方法论。」面试官看重的是问题解决框架，不是背诵能力。

本章结论

  • S1~S5：每场景掌握「架构+指标+坑+三座大山」四段式答辩。
  • 三座大山：NS1 飞轮、NS2 确定性、NS3 成本为通用评估框架。
  • 双轨叙事：交付轨串 S1/S2，技术轨串 S3/S4/S5，数字口径一致。
  • 实战技巧：30 分钟时间分配 + 高频追问准备 + 3 个版本演练。
