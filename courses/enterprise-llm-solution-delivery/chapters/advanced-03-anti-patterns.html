为什么要学本章

CorpAssist 类项目失败 rarely 因为「模型不够大」，更多死于承诺与能力错位与PoC 债未偿还。本章系统梳理 anti pattern，让你在 sign-off 前主动拆弹。

学完你能
  • 识别 over promise 在合同、PoC、简历三处的不同形态
  • 区分 tech debt 与 demo to prod gap 并排偿还计划
  • 用反模式清单做 discovery / 验收门禁
  • 输出团队 delivery trap 预警表
本章先记住 2 件事
  • 过度承诺：没有 SLA、没有 golden set、没有 ACL 的「保证准确率」都是 silent assumption 外显化。
  • 技术债：PoC 为赶 Demo 留下的捷径，若未写入 risk register 与偿还排期，会在试点爆发。

过度承诺

CorpAssist · 招标前夜 销售希望标书写「问答准确率 95%、7×24 智能替代人工」；算法同事说 PoC 最好的一次是 88%；法务问 SLA 赔偿条款。你作为交付负责人，需要把承诺拉回可验收口径。

over promise 在 ToB LLM 项目中的典型表现：

  • 把 PoC 单次 demo 指标写进合同，无数据集 frozen 条款
  • 承诺「全场景 Agent 自主办理」却无工具真执行与步数预算
  • 口头答应「私有化 + 国产化 + 2 周上线」但未估 gpu estimate 与人天
  • 简历 / 答辩夸大个人角色（与 advanced-02 呼应）
承诺类型危险信号可验收替代写法
准确率 「保证 95%」无评测集定义 UAT golden set ≥ N 条，eval_report 双方签字
可用性 「永不宕机」 contract sla 99.5%，明确排除厂商侧故障
范围 「所有部门所有文档」 mvp scope：2 部门 + 3 类文档 pilot
智能程度 「完全替代人工」 转人工率 < X%，复杂工单仍走 DST
过度承诺话术
  • 「大模型什么都能答」
  • 「PoC 效果 = 生产效果」
  • 「后续迭代不算变更」
交付负责人回应
  • 列 scope boundary 与 out-of-scope
  • PoC / 试点 / 合同三套指标（见 advanced-01）
  • 变更走 CR，影响 manday estimate

| ID | 风险 | 承诺来源 | 证据缺口 | 缓解 |
|----|------|----------|----------|------|
| R-07 | 准确率 95% | 招标文件草案 | PoC 仅 88%（50条） | 改 K1 为 85% + UAT frozen set |

FAQ：客户坚持高指标怎么办？

提供付费 PoC 加测或试点对赌：加测扩大 golden set、延长 pilot，达标后再签主合同 KPI。勿在无数据前提下签「95%」。

技术债

tech debt 在本课指：为赶 PoC / Demo 而采取的已知捷径，且未计划偿还。LLM 项目常见债项与 CorpAssist 对照：

债项PoC 捷径生产代价偿还动作
索引 全量同步、无 ACL 泄露、重建成本 分区索引 + ACL 矩阵 sign-off
检索 单路向量、无 rerank 准确率下滑 混合检索 + eval 门禁
Prompt 硬编码在 notebook 不可审计、版本混乱 Prompt registry + 发版记录
观测 仅 print 日志 无法定责 SLA trace_id + 结构化日志 + 告警
降级 无超时/无 fallback 厂商故障拖垮门户 缓存答复 + 转人工 + 熔断
低代码 Dify 搭完即 demo vendor lock migrate path 或 hybrid arch

scope creep 与 tech debt 常共生：每个「小改动」不走 CR，堆成 gold plating（过度打磨无关功能）而核心债未还。

  • ACL / 鉴权 / 审计日志
  • 超时、限流、降级（S2 客服必备）
  • 索引幂等与回滚脚本
  • golden set + eval 门禁
  • runbook + oncall
  • 坏例飞轮最小闭环（advanced-07）

动手练习

在 demos/advanced-03-anti-patterns-lab/anti-pattern-list.md 列出 CorpAssist 情境下至少 5 条反模式，每条含「后果 + 缓解」；至少 2 条来自本章表格。

操作步骤

  1. 从「过度承诺决策表」选 1 行，写一条 risk register 风格缓解措施。
  2. 从「技术债对照表」选 2 个债项，填 tech_debt.yaml 风格字段（id/title/payoff_plan）。
  3. 画 mentally「over promise 因果链」：标出你项目最可能在哪一环断裂。
  4. 勾选「试点上线前反模式扫描」清单，记录未勾选项的 owner。

判断练习

  1. （排障顺序）试点第 1 周 P1：跨部门文档被检索到。可能根因有：①索引无 ACL ②Prompt 泄露 ③日志未脱敏。请排排查优先级并说明。

    参考答案

    优先 ①：直接阻断泄露面——临时下线检索或强制 default-deny acl_scope，再跑 ACL 重建。②③ 并行：查 Prompt 是否注入他部 doc_id；日志脱敏属合规但通常非首因。事后写入 tech_debt 与发版门禁。
