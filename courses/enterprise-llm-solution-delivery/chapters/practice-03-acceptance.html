为什么要学本章

ToB 大模型项目最常踩的坑：PoC 指标与合同脱节。演示时 88% 准确率，UAT 时因数据集不同变成 78%，商务纠纷由此而生。本章教你把 contract sla 写清楚、eval report 做规范、sign off 流程走顺畅。

学完你能
  • 写出包含 kpi contract 条款、评测方法与除外责任的验收附件
  • 出具一份完整的 eval report，含 golden set 版本、指标、混淆矩阵
  • 按 uat checklist 逐项检查并推动 delivery sign
  • 定义 sla metric 与赔偿条款
本章先记住 3 件事
  • 验收是签合同前就该想好的事：不是试点结束后才定义通过标准。
  • Golden set 要 frozen：UAT 用的评测集必须在验收前双方签字锁定版本。
  • SLA 写「除外」比写「涵盖」更重要：厂商故障、计划内维护、数据源变更都需明确。

合同指标

CorpAssist · UAT 验收会 客户 IT 总监说：「PoC 时准确率 88%，现在你说 UAT 只有 82%？」你的回答将决定验收能否通过、尾款能否按时到账。验收的核心是——contract sla 条款写得够不够清楚。

kpi contract 不是简单写「准确率 ≥ 85%」，而需写明：用什么数据集、谁标注、谁来跑、争议如何仲裁。

验收 KPI 完整条款结构

要素要求CorpAssist 示例
指标定义 明确计算口径（宏观/微观、含不含拒答） 准确率 = 回答包含期望关键词的比例（宏观平均）

## 合同附件 A — 验收 KPI 与 SLA（节选）

### A.1 验收指标
| 编号 | 指标 | 阈值 | 数据集 | 评测方式 |
|------|------|------|--------|---------|
| K1 | 准确率（Acc） | ≥ 85% | golden_frozen_v3 (200条) | eval_runner v1.2 |
| K2 | 引用可追溯率 | ≥ 95% | 同 K1 | 自动校验 citation 字段 |
| K3 | P95 响应时间 | ≤ 2.0s | 生产环境 1000 次抽样 | BFF 日志指标 |
| K4 | 月度可用性 | ≥ 99.5% | 生产监控（排除计划内维护） | 可用性计算公式 |

### A.2 SLA 赔偿
- **可用性 < 99.5%**：按月度服务费的 5% 减免
- **准确率 < 85%**：乙方应在 15 个工作日内修复后复测
- **除外条款**：模型厂商 API (DashScope) 故障不纳入可用性计算

### A.3 争议处理
- 双方对 eval 结果不一致时，取第三方标注结果
- 标注不一致率 > 10% 时，冻结数据集并重新标注入库

KPI 定义必须在签约时完成，而非 UAT 前临时商定。

不推荐
  • 只写「准确率 ≥ 85%」，不附数据集
  • PoC 指标直接当合同 KPI 写入
  • 无除外责任条款
推荐
  • 指标 + 数据集 + 评测方法 + 阈值 + 争议仲裁完整定义
  • PoC / 试点 / 合同三套指标分开（advanced-01）
  • 明确厂商故障、数据源变更等除外场景

sla metric 与 contract sla 的关系：SLA 是服务等级协议，包含可用性、延迟、吞吐等；SLA metric 是具体量化指标。CorpAssist 的 SLA 建议涵盖可用性（99.5%）、P95（2.0s）和安全（0 泄露事件）。

先想 10 秒：客户说「不要写除外条款，显得你们没信心」，你如何回应？

正面解释：「除外条款保护双方——写了模型厂商故障不在此限，客户也不会因阿里云宕机而拒付尾款。」这是行业标准做法，不是缺乏信心。

评测报告

eval report 是验收的「裁判文书」。它用数据证明系统是否达到 acceptance 标准。CorpAssist 的 eval report 包含以下章节：

Eval Report 标准结构

章节内容格式
1. 评测概述 目的、时间、参与方、数据集版本 文本 + 表格
2. 数据集说明 golden set 规模、来源、标注规范、分布统计 表格 + 柱状图
3. 指标结果 K1/K2/K3 逐项结果、置信区间 表格 + 混淆矩阵
4. 场景细分 S1/S2/S3 各场景准确率、引用率 雷达图
5. Bad Case 分析 错误样本分类（数据/工具/Prompt/路由） 表格 + 饼图
6. 结论与建议 是否通过验收、遗留问题清单 文本 + 签字栏

签收流程

delivery sign 是项目交付的最后一个里程碑，也是尾款支付的前提条件。签收不是简单的「签个字」，而是一套完整流程。

签收流程 6 步法

步骤动作产出负责人
1 出具 eval_report Eval report PDF 算法
2 提交运维移交包 移交文档 + runbook Java + 运维
3 发起验收会 会议邀请含 agenda PM
4 现场演示 + 指标展示 验收会纪要 全员
5 客户签字确认 签收单 (sign-off sheet) PM + 客户
6 归档 + 启动运维 项目归档文件夹 PM

确认系统可用、指标达标、运维就绪后签字。

FAQ：客户说「再试用 3 个月再签收」，怎么办？

不可接受。应在合同阶段约定验收窗口期（通常 UAT 后 10~15 个工作日）。超期未签收视为默认通过——这一条款必须写进合同。如果客户确有新需求，可签订二期 SOW，不影响一期验收。

签收后客户发现一个 P1 Bug，怎么办？

按运维流程走：创建 ticket → 按 SLA 时限修复 → 补丁发布。签收不代表零缺陷，而是 KPI 达标且签署了交付物。P1 Bug 属于运维期支持范畴，不应影响签收有效性。

评测结果双方不一致（甲方人工评审与乙方脚本结果偏差），怎么处理？

按合同争议条款处理：不一致条目由第三方重新标注。双方应在签约时就约定「标注一致性检查」——若标注不一致率 > 10%，则 frozen set 需回炉再标。

本章结论

验收不是终点，而是交付质量的一次「对账」——全对才能签收。

  • 合同指标：KPI 条款必须包含指标+数据集+评测方法+阈值+除外责任，签约时完成 frozen set 签字。
  • 评测报告：完整 eval report 包含概述、指标、细项、bad case 分析、结论；脚本与版本锁定。
  • 签收流程：6 步法从 eval report 到签收单签署；合同中约定验收窗口期与签收默认条款。

操作步骤

判断练习

  1. （场景）UAT 准确率 84%，合同要求 85%。差 1 个百分点，你会怎么做？A) 跳过验收先签收 B) 重新标 20 条看能否达 85% C) 进入修复窗口，修复后复测

    参考答案

    C 是正确做法。按合同条款走：准确率不达标则进入修复窗口（通常 15 个工作日）。重新标注（B）相当于改数据集，违反 frozen set 原则。跳过验收（A）是高风险行为。

  2. （排障）Eval 脚本在验收环境跑了 86%，客户自己跑只有 80%。哪个最不可能是原因？A) 评测脚本版本不一致 B) golden set 文件不同 C) 客户显卡更好

    参考答案

    C。评测脚本文本匹配不依赖 GPU 型号；A 和 B 是常见原因。应在签约时约定评测环境规格并写入合同附件。
