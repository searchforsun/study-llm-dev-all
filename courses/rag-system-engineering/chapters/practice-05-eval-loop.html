本章先记住 4 件事
  • 基准先于调参：没有固定黄金集，所有「提升」都是主观感受。
  • 双栈同测：Python 索引管线与 Spring 查询 API 对同一套用例跑 parity test，避免栈间漂移。
  • 报告要带版本：索引 alias、Embedding 模型、chunk 策略必须写入报告元数据。
  • 发布是闸门不是按钮：regression rag 任一核心指标跌破阈值即阻断。

基准

CorpAssist S1 · 制度问答 HR 希望上线前能回答「年假申请流程」「报销上限」等 80 条真实问法；每条需标注期望引用条款与可接受拒答场景。

黄金集结构：每条用例含 question、expected_doc_ids、must_cite（是否必须带引用）、acl_context（模拟部门权限）。指标至少覆盖：

  • faithfulness：答案是否 grounded 于检索片段（见 幻觉治理）
  • context_recall：检索是否召回了期望文档
  • answer_relevance：生成是否切题（可选 LLM-as-judge）

Python 侧：LangChain / Ragas 跑批

python
# demos/practice-05-eval-loop-lab/run_eval.py（示意）
from ragas.metrics import faithfulness, context_recall

def run_benchmark(cases, retriever, chain):
    rows = []
    for c in cases:
        docs = retriever.invoke(c["question"], filters=c["acl_context"])
        answer = chain.invoke({"question": c["question"], "context": docs})
        rows.append({"case_id": c["id"], "answer": answer, "docs": docs})
    return evaluate(rows, metrics=[faithfulness, context_recall])

迭代

调参不是拍脑袋：每次只改一个变量（chunk_size、overlap、hybrid search 权重、rerank 开关），在相同黄金集上跑 iter loop，记录 metric delta。

报告

eval report 是发布决策的唯一真源：须包含套件 ID、索引版本、模型版本、各指标汇总与失败用例明细。Demo 输出见 demos/practice-05-eval-loop-lab/eval-result.json。

报告 JSON 结构

json
{
  "run_id": "20260530-143022",
  "index_alias": "corp-s1-hr@exp-a",
  "embedding_model": "text-embedding-3-small@v2",
  "metrics": {
    "faithfulness": 0.88,
    "context_recall": 0.84
  },
  "failed_cases": ["hr-017", "fin-003"],
  "eval_bind": "git:abc1234 + index:corp-s1-hr-v7"
}

eval bind 将报告绑定到代码 commit + 索引 snapshot，便于事后复现。Spring 侧可读该 JSON 做 CI 断言。

Spring 侧：读取报告做 CI 门禁

java
// 集成测试：报告指标不得低于 baseline
@Test
void evalReportMeetsGate() throws Exception {
  EvalReport report = mapper.readValue(
      Path.of("eval-result.json").toFile(), EvalReport.class);
  assertThat(report.getMetrics().getFaithfulness())
      .isGreaterThanOrEqualTo(0.85);
  assertThat(report.getFailedCases()).hasSizeLessThan(5);
}

发布

通过评测不代表立刻全量：release gate 定义硬阈值与灰度策略。CorpAssist S1 典型闸门：

  • faithfulness ≥ 0.85 且 context_recall ≥ 0.80
  • 失败用例数 < 5 或均为已知 accepted 风险
  • 双栈 parity：Spring API 与 Python 批跑 Top-1 doc 一致率 ≥ 95%
  • publish check：alias 切换前人工 sign-off（合规类问法）

发布流水线（双栈）

yaml
# release-gate.yaml
gates:
  faithfulness_min: 0.85
  context_recall_min: 0.80
  parity_min: 0.95
steps:
  - python_index_build
  - run_eval_suite
  - spring_integration_test
  - alias_switch: corp-s1-hr@prod  # 仅 gates 全绿
双栈发布与 alias 切换
sequenceDiagram
  participant PY as Python 索引
  participant EV as 评测跑批
  participant SP as Spring RAG API
  participant AL as index alias
  PY->>EV: 新索引 snapshot

未过闸门时禁止切换 index alias；回滚只需指回上一版 snapshot + 报告。

本章结论

  • 基准：固定黄金集 + faithfulness/context_recall；双栈共用用例。
  • 迭代：单变量实验 + metric delta；NS1 飞轮避免多改难归因。
  • 报告：版本化 eval-report + eval_bind 绑定 commit 与索引。
  • 发布：release gate 硬阈值 + alias 切换；未过 publish-check 不上线。

判断练习

  1. faithfulness 0.92 但 context_recall 0.55，能否通过 S1 发布闸门？理由？

    参考答案

    不能。答案可能「自洽但瞎编」——检索未召回期望条款。S1 闸门通常要求 context_recall ≥ 0.80；应优先调检索（chunk、混合权重）而非只调 Prompt。

  2. Python 批跑全绿，Spring API 抽样 parity 仅 88%。最可能的三类原因？

    参考答案

    ① Spring 侧 ACL filter 与 Python 不一致；② 两栈 TopK / rerank 参数不同；③ alias 指向不同索引 snapshot。应对齐 filter DTO 与检索参数后再比 parity。
