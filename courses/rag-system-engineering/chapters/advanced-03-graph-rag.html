为什么要学本章

GraphRAG 概念与边界 讨论了 GraphRAG 边界；一旦上图，图谱与向量索引、文档 ACL、下线流程必须同版本治理，否则会出现「向量已删、图边仍在」的越权或多跳幻觉。本章聚焦 graph governance、incremental graph、kg version，并与企业知识库治理课形成 governance bridge。

与上一章的关系

本章沿用 GraphRAG 概念与边界 相同的小节骨架（何时需要 / 成本 / 简化方案），但视角切换为治理就绪：聚焦 kg version、tombstone 级联删除与 policy sync，回答「上图后如何与向量索引、ACL、文档下线同步」——同名小节，治理透镜不同。

学完你能
  • 制定图与向量 kg version 对齐策略
  • 设计文档下线时的图节点级联删除与 graph refresh
  • 实现 incremental graph 更新而非全量重抽
  • 说明 graph acl 与 policy sync 如何与 S1 多库路由一致
本章先记住 4 件事
  • 图向量同版本：alias 切换必须同时覆盖向量库与图存储 snapshot。
  • 下线要删边：文档失效 → chunk 删 → 实体/边 tombstone。
  • 增量优先：仅对变更文档子图重算社区摘要。
  • 治理课衔接：元数据、审计、合规流程与向量 RAG 共用同一治理面。

何时需要（治理就绪）

进入治理级 GraphRAG 的前提，仍是 when to use 成立（多跳/global 问法），且已通过 RAG 评测闭环 证明简化方案不足。治理视角额外要求：

  • 图谱变更可审计（谁触发、影响哪些社区摘要）
  • 租户/部门 ACL 在图遍历中可 enforcement
  • 与文档生命周期（发布/修订/废止）绑定

FAQ：向量 RAG 已治理良好，上图谱还要重做吗？

元数据与 ACL 策略可复用，但图有额外生命周期：实体合并、边 tombstone、社区摘要版本。需扩展 checklist，不能假设「向量 governance 自动覆盖图」。

废止制度事故复盘模板

若用户仍从已废止制度获得答案，审计应能回答：① 向量 snapshot 版本；② 命中 chunk 的 doc_id 是否已 tombstone；③ 若走图检索，边是否仍连通；④ 社区摘要 hash 是否过期。四项任一异常即判定为 graph governance 缺陷，而非单纯「模型幻觉」。

  • 复盘 SLA：安全事件 4h 内定位到 kg-version / job_id
  • 临时措施：关闭 graph 路由 + 全库 semantic-cache invalidate
  • 根因分类：policy-sync 延迟 / tombstone 遗漏 / 社区摘要未重算

简化方案（治理简化）

治理语境下的 simple alternative 不是「不用图」，而是降低图治理复杂度：

  1. 子图 scope：仅「跨制度引用」类目建图，其余仍向量
  2. 共现边替代 LLM 关系：规则边 + 人工审核，减少抽取噪声
  3. 社区摘要缓存：版本化 object store，未变更社区只读缓存
  4. 双轨发布：向量先上线，图检索 beta 路由，共用 eval gate
无治理上图
  • 向量/图不同步
  • 废止文档仍多跳可达
  • 无法回答「用的哪版图」
graph-governance
  • kg-version 绑定 release
  • tombstone + graph-refresh
  • audit trail 到社区摘要 job

双轨发布降低治理风险

向量 alias 切生产后，图检索仅对beta 用户组或内部审计账号开放；两轨共用 eval gate，但 API 响应须标注 retrieval_source: vector|graph，避免用户误以为答案已含多跳推理。

简化层级治理复杂度适用阶段
子图 scope低试点 1～2 个类目
规则边 + 人工审核中合规团队可介入
社区摘要版本缓存中增量更新成熟后
全量 LLM 抽图高eval 证明 ROI 后

incremental-graph job（示意）

python
def on_document_revoked(doc_id: str, kg_version: str):
    chunk_ids = vector_index.delete_by_doc(doc_id)
    entity_ids = graph.unlink_chunks(chunk_ids)
    graph.mark_tombstone(entity_ids)
    communities = graph.affected_communities(entity_ids)
    regen_community_summaries(communities, kg_version=kg_version)
    audit.log("graph_refresh", doc_id=doc_id, version=kg_version)
先想 10 秒：为何 governance bridge 强调与向量课共用 audit trail？

合规审计需一条时间线：用户看到的答案来自哪版索引、哪条边、哪份文档。向量与图分裂审计时，无法解释 cross-policy 多跳答案来源。

Spring 侧 policy-sync 监听（示意）

java
@EventListener
void onDocumentRevoked(DocumentRevokedEvent e) {
  vectorIndexService.deleteByDoc(e.docId());
  graphIndexClient.tombstone(e.docId(), e.kgVersion());
  semanticCache.invalidateKb(e.kbId(), e.kgVersion());
  audit.log("policy-sync", e.traceId(), Map.of("doc", e.docId()));
}
FAQ：governance bridge 谁负责维护 kg-version？

发布负责人（与 release gate 绑定）：Python 索引 job 产出 version 元数据，Spring alias 切换与 Neo4j snapshot 标签同一 PR合并。禁止运维手工改图库而不 bump version。

治理里程碑（与知识库课对齐）

  1. M1：向量与图共用 doc-acl YAML，抽检 100 chunk 一致
  2. M2：文档废止事件 5 分钟内完成 tombstone + audit
  3. M3：incremental-graph job 可回滚至上一 snapshot
  4. M4：合规审计可导出「答案→边→文档→版本」完整链

未达 M2 不得对生产流量开放 graph 检索；M4 为与外部治理课联合验收项。

里程碑状态应在发布看板与 kg-version JSON 中可见，避免「口头治理、不可查询」。

M3 完成前禁止将 graph 检索默认路由从 beta 调至全量。

FAQ：增量更新失败 halfway 如何回滚？

job 开始前写 rollback_snapshot；失败时恢复 Neo4j snapshot + 回退 kg-version 指针，并 invalidate 受影响 kb 的 semantic-cache。审计记录 graph_refresh_status=rolled_back。

回滚演练应纳入季度演练，与向量 alias 回滚脚本一并执行，避免事故时首次操作。

演练记录写入 audit trail，供合规抽查。

未在演练季完成回滚验证的团队，不得申请扩大 graph 检索流量占比。

演练报告模板见 demos/advanced-03-graph-rag-lab/graph-governance-checklist.md 附录。

本章结论

  • 何时需要：GraphRAG 生效且须满足文档生命周期与 ACL 治理就绪。
  • 成本：incremental-graph + 局部社区重摘要；避免无脑全量重抽。
  • 简化方案：子图 scope、规则边、双轨发布降低治理复杂度。
  • 与知识库治理课衔接：kg-version、graph-acl、policy-sync 与向量共用治理面。

判断练习

  1. 向量 alias 已切 v12，graph_snapshot 仍 v11。用户问跨制度冲突，最可能出现什么风险？

    参考答案

    向量检索到新制度 chunk，但多跳沿 v11 旧边连到已废止节点，答案半新半旧，faithfulness 与合规风险极高。release gate 应要求 kg-version 三元组同时切换。

  2. 仅对变更文档跑 incremental-graph，未 regen 受影响 community summary。查询「全局主题分布」会怎样？

    参考答案

    global 检索仍读旧社区摘要，与局部向量结果矛盾。应在 incremental job 中计算 affected_communities 并触发 graph refresh 重摘要。
