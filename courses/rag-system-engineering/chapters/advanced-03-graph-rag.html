为什么要学本章

上一章 讨论了 GraphRAG 边界；一旦上图，图谱与向量索引、文档 ACL、下线流程必须同版本治理，否则会出现「向量已删、图边仍在」的越权或多跳幻觉。本章聚焦 graph governance、incremental graph、kg version，并与企业知识库治理课形成 governance bridge。

学完你能
  • 制定图与向量 kg version 对齐策略
  • 设计文档下线时的图节点级联删除与 graph refresh
  • 实现 incremental graph 更新而非全量重抽
  • 说明 graph acl 与 policy sync 如何与 S1 多库路由一致
本章先记住 4 件事
  • 图向量同版本：alias 切换必须同时覆盖向量库与图存储 snapshot。
  • 下线要删边：文档失效 → chunk 删 → 实体/边 tombstone。
  • 增量优先：仅对变更文档子图重算社区摘要。
  • 治理课衔接：元数据、审计、合规流程与向量 RAG 共用同一治理面。

何时需要

进入治理级 GraphRAG 的前提，仍是 when to use 成立（多跳/global 问法），且已通过 评测闭环 证明简化方案不足。治理视角额外要求：

  • 图谱变更可审计（谁触发、影响哪些社区摘要）
  • 租户/部门 ACL 在图遍历中可 enforcement
  • 与文档生命周期（发布/修订/废止）绑定
CorpAssist · 制度废止 HR-2023 差旅制度废止后，向量 chunk 已删，但 GraphRAG 仍沿旧边回答「出差标准」——典型治理缺失。

成本

治理不会消除 graph cost，但可通过增量与范围控制降低反复全量抽取：

策略触发成本治理收益
全量重抽major 模型变更最高一致性最简单
incremental-graph单文档更新中需 kg-version 追踪
社区局部重摘要子图变更 > 阈值中低避免全库社区重算
仅向量、图延迟新库试点低双轨期需标注答案来源

entity resolution（同一实体多写法合并）应在增量更新中重跑，否则边会分裂，查询期多跳断裂。

简化方案

治理语境下的 simple alternative 不是「不用图」，而是降低图治理复杂度：

  1. 子图 scope：仅「跨制度引用」类目建图，其余仍向量
  2. 共现边替代 LLM 关系：规则边 + 人工审核，减少抽取噪声
  3. 社区摘要缓存：版本化 object store，未变更社区只读缓存
  4. 双轨发布：向量先上线，图检索 beta 路由，共用 eval gate
无治理上图
  • 向量/图不同步
  • 废止文档仍多跳可达
  • 无法回答「用的哪版图」
graph-governance
  • kg-version 绑定 release
  • tombstone + graph-refresh
  • audit trail 到社区摘要 job

incremental-graph job（示意）

python
def on_document_revoked(doc_id: str, kg_version: str):
    chunk_ids = vector_index.delete_by_doc(doc_id)
    entity_ids = graph.unlink_chunks(chunk_ids)
    graph.mark_tombstone(entity_ids)
    communities = graph.affected_communities(entity_ids)
    regen_community_summaries(communities, kg_version=kg_version)
    audit.log("graph_refresh", doc_id=doc_id, version=kg_version)
先想 10 秒：为何 governance bridge 强调与向量课共用 audit trail？

合规审计需一条时间线：用户看到的答案来自哪版索引、哪条边、哪份文档。向量与图分裂审计时，无法解释 cross-policy 多跳答案来源。

本章结论

  • 何时需要：GraphRAG 生效且须满足文档生命周期与 ACL 治理就绪。
  • 成本：incremental-graph + 局部社区重摘要；避免无脑全量重抽。
  • 简化方案：子图 scope、规则边、双轨发布降低治理复杂度。
  • 与知识库治理课衔接：kg-version、graph-acl、policy-sync 与向量共用治理面。

判断练习

  1. 向量 alias 已切 v12，graph_snapshot 仍 v11。用户问跨制度冲突，最可能出现什么风险？

    参考答案

    向量检索到新制度 chunk，但多跳沿 v11 旧边连到已废止节点，答案半新半旧，faithfulness 与合规风险极高。release gate 应要求 kg-version 三元组同时切换。

  2. 仅对变更文档跑 incremental-graph，未 regen 受影响 community summary。查询「全局主题分布」会怎样？

    参考答案

    global 检索仍读旧社区摘要，与局部向量结果矛盾。应在 incremental job 中计算 affected_communities 并触发 graph refresh 重摘要。
