代码切片与符号索引

CorpAssist S4 · monorepo 研发问：「OrderService#createOrder 的幂等逻辑在哪？」需要定位 Java 方法级片段，而非 512 token 随意切块。

ast split 按语法树切分：每个 chunk 对应一个 function / class / interface，metadata 含 symbol、file_path、start_line、language。

Python：tree-sitter 切片示意

python
def chunk_java_file(path: str, source: str) -> list[dict]:
    tree = parser.parse(bytes(source, "utf8"))
    chunks = []
    for node in tree.root_node.children:
        if node.type in ("class_declaration", "method_declaration"):
            chunks.append({
                "text": source[node.start_byte:node.end_byte],
                "symbol": extract_symbol(node),
                "file_path": path,
                "start_line": node.start_point[0] + 1,
            })
    return chunks

符号索引（并行于向量库）

json
{
  "symbol": "com.corp.OrderService#createOrder",
  "file": "src/main/java/.../OrderService.java",
  "line": 42,
  "vector_id": "chunk-8f3a2b"
}
固定 token 切块
  • 方法被截断，缺 import 上下文
  • 「类名」问法 recall 低
  • 符号无法精确跳转
AST + 符号索引
  • 一 chunk 一符号，边界清晰
  • 精确 lookup + 向量语义兜底
  • IDE 可跳转行号

Spring：符号表查询接口（对照）

java
public interface SymbolIndex {

// SymbolHit: file, line, vectorId — 与 Python symbol_index.json 同 schema
FAQ：代码 chunk 要不要带 import 与 package？

方法级 chunk 应在 metadata 存 package、imports_hash，正文可不含全量 import 以省 token；生成时若缺上下文，检索层可追加「同文件 class 头」chunk。CorpAssist S4 禁止把整文件无差别塞进向量库。

仓库级检索

repo rag 检索必须在 metadata 上过滤：repo_id、branch、path_prefix。多 monorepo 时禁止跨 repo 混检。

混合检索：符号 lookup + 向量

python
def retrieve_code(query: str, ctx: RepoContext) -> list[Document]:
    if looks_like_symbol(query):  # e.g. FooBar#baz
        hit = symbol_index.lookup(query, repo=ctx.repo, branch=ctx.branch)
        if hit:
            return [load_chunk(hit.vector_id)]
    return vector_store.similarity_search(
        query,
        filter={"repo": ctx.repo, "branch": ctx.branch},
        k=8,
    )

先想 10 秒：为何 branch 必须进 filter？

同一文件在不同分支内容不同；不过滤会把已合并/已删除分支的 stale chunk 检入，导致错误答案。索引 job 应随 push webhook 增量更新。

monorepo path_prefix 过滤

python
filter = {
    "repo": "corp-backend",
    "branch": "main",
    "path_prefix": "services/order/",  # 用户当前打开的子工程
}
docs = vector_store.similarity_search(query, filter=filter, k=8)

IDE 插件应把「当前工作区根」传给 API，避免在 200+ 微服务仓库里全库检索；与文档 RAG 的 kb 路由类似，属于范围收窄而非权限替代。

补全/问答 Prompt

两种场景 Prompt 不可混用：

  • FIM 补全：<prefix>…<suffix>…，检索片段作 <middle> 参考，强调语法正确与短输出
  • 问答 RAG：System 要求引用 file:line，User 含问句 + 检索块

延迟与安全扫描

latency budget 按场景拆分：

场景P95 目标策略
IDE 内联补全< 800ms符号优先、小 TopK、缓存热文件
侧边栏问答< 3s可 rerank、可流式
CI 代码审查 Bot< 30s异步、批量索引

索引流水线入口必须跑 secret scan（gitleaks / trufflehog）；命中则阻断 chunk 写入。

扫描配置片段

yaml
ingestion:
  pre_hooks:
    - secret_scan:
        block_on_match: true
        patterns: [AWS_KEY, PRIVATE_KEY, BEARER_TOKEN]
  on_match: skip_file_and_alert

延迟预算分解（IDE 补全示例）

阶段预算 (ms)优化手段
符号 lookup≤ 50内存索引、前缀匹配
向量检索≤ 120小 TopK、path 过滤
LLM 首 token≤ 400小模型、短 max_tokens
网络 + 序列化≤ 230同 AZ、gzip
FAQ：代码库 RAG 能否跳过 secret scan「先上线」？

不可以。向量库一旦写入密钥，删除需按 doc_id 全量 purge 且难保证无缓存泄漏。CorpAssist S4 与 S1 一样要求 ingest pre_hook；CI 失败即阻断索引 job。

双栈对照

Python 负责 AST 切片、embedding、符号表构建；Spring 暴露 /api/code-rag/ask 与 IDE 插件调用。对照要点：

  • filter 表达式语义一致（repo、branch、ACL）
  • TopK、score threshold 与 Python 批测对齐
  • 引用格式统一为 path:line

本章结论

  • 代码切片与符号索引：AST 保边界；symbol index 支撑精确跳转。
  • 仓库级检索：repo+branch 硬过滤；符号问法走 lookup。
  • 补全/问答 Prompt：FIM 与 RAG 分离；答案必须 file:line 引用。
  • 延迟与安全扫描：分场景 latency budget；索引前 secret scan。
  • 双栈对照：Python 建索引，Spring 对外 API；filter 与 TopK 对齐。

判断练习

  1. 用户问「这个项目怎么跑起来？」应走 symbol index 还是语义向量？

    参考答案

    语义向量为主。问法不含明确符号名，README / build.gradle 等文档型 chunk 靠 embedding；可对 README.md 提权而非 symbol lookup。

  2. Spring filter 漏写 branch，Python 索引已按 branch 分 collection。会出现什么现象？

    参考答案

    可能检索到其他 branch 的 stale chunk，或空结果（若 collection 按 branch 隔离且 filter 不匹配）。双栈 parity 失败；应在 Spring SearchRequest 与 Python filter 对齐 branch 字段。
