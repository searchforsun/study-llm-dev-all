为什么要学本章

分块之后，文本必须变成向量才能进检索。corpassist s1 的制度库通常由 Python 离线批处理写入向量库，而 Spring BFF 负责在线查询——若模型、维度、版本不一致，召回会「看起来正常、实则全错」。本章把双栈索引契约讲清楚。

本章先记住 3 件事
  • 写与查分离：dual stack rag 里 Python 负责索引管线，Spring 负责查询 API，共用同一向量库 schema。
  • 维度必须对齐：vector dim 与模型绑定；换模型必须全量 re embed。
  • 版本可切换：用 index alias 做蓝绿切换，避免查询打到半完成索引。

Embedding 模型选择

模型决定「语义空间」——同一句话在不同模型下的向量不可比较。CorpAssist S1 人事制度库以中文条款为主，选型时至少看四维：

索引工程师（Python）

选型、批处理 embed、写入 collection；变更模型时触发 re-embed 与 alias 切换。

查询服务（Spring）

只读已索引向量；query embedding 须与索引侧同一 embedding model 配置。

评测负责人

在 S1 黄金集上对比 recall@k；模型升级不得跳过 release gate。

  • 语种与领域 — 通用 multilingual 模型 vs 中文优化模型；制度、合同类文本优先测 recall@5。
  • vector dim — 常见 768 / 1024 / 1536；维度越高存储与检索成本越高，收益需用评测集验证。
  • 费用与 QPS — 索引阶段可离线慢速批处理；查询阶段 query embedding 需控延迟。
  • 版本治理 — 模型升级写入 model version 元数据，便于审计与回滚。

反例 vs 推荐做法

不推荐
  • 索引用 A 模型、查询用 B 模型
  • 换模型后直接覆盖旧向量，无版本号
  • 只看 MTEB 榜单，不在 S1 制度集上实测
  • 维度与向量库 schema 不一致仍强行写入
推荐
  • 索引与查询共用同一 embedding model 配置
  • 变更模型 → 新 collection + index alias 切换
  • 在 CorpAssist 基准问答集上对比 recall / MRR
  • collection meta 固定 model、dim、version
场景倾向选择注意
S1 中文制度库 中文或多语 small 模型 条款编号、专有名词 recall
百万 chunk 规模 维度适中（768～1024） 存储与 ANN 索引内存
多租户隔离 同模型、分 collection ACL 在 metadata 过滤
先想 10 秒：为什么「索引与查询用同一模型」比「各用各的」更重要？

向量相似度只在同一语义空间内有意义；混模型等于用不同坐标系比距离，TopK 结果不可信。

批处理索引（Batch Index）

CorpAssist S1 · 夜间全量索引（45 秒） HR 上传 200 份制度 PDF，经 分块策略 产出约 12 万 chunk。Python 索引任务在 02:00 启动：分批调用 Embedding API，写入 Milvus/PGVector，并在 meta 表记录 job_id 与进度。

batch index 的核心不是「一次 embed 很多条」，而是可恢复、可观测、可限流的离线管线：

  • 批大小 — 按 API 上限（如 64～256 条/请求）与内存折中；失败批次可单独重试。
  • 幂等写入 — chunk_id 作主键，重复跑任务不 duplicated。
  • checkpoint — 记录最后成功 doc_id，断点续跑。
  • 与查询隔离 — 写入 staging collection，校验通过后 alias 切流。
python 
BATCH_SIZE = 128
for batch in chunked(chunks, BATCH_SIZE):
    vectors = embed_client.embed_documents([c.text for c in batch])
    store.upsert(
        ids=[c.chunk_id for c in batch],
        vectors=vectors,
        metadatas=[{"doc_id": c.doc_id, "model_version": MODEL_VER} for c in batch],
    )
    job_repo.save_checkpoint(batch[-1].chunk_id)

索引完成前查询仍走旧 alias；切换是原子操作，避免「半新半旧」索引。

向量维度与 Re-embed

vector dim 是模型输出长度，向量库建表时必须固定。常见陷阱：

  • 从 768 维模型升级到 1536 维 → 旧索引不能原地 patch，必须 re embed 全库。
  • 降维截断（MRL）需确认检索库与 API 均支持同一截断策略。
  • metadata 应存 embedded_at、model_version，便于增量补跑。

维度 × 存储粗算（教学示意）

768

100 万 chunk × 768 维 ≈ 3 GB 向量本体（不含索引结构与 metadata）。

常见误区
  • 只 re-embed 增量文档 — 换模型后旧 chunk 与新 chunk 不在同一空间，必须全量或分版本 collection。
  • 查询端不校验 dim — Spring 启动时应读 collection schema，维度不符直接 fail-fast。
  • 忽略 embedding 缓存 — 同一 chunk 文本未变时不应重复计费 embed。

Python 索引 + Spring 查询（双栈分工）

CorpAssist dual stack rag 的典型分工：

层栈职责
索引管线 Python / LangChain python index：接入、分块、embed、写入、alias 切换
查询 API Spring AI spring query：query embed、VectorStore 检索、Advisor 组装上下文
共用契约 JSON / OpenAPI collection 名、dim、model_version、metadata 字段（doc_id、dept_acl）

索引配置 vs Spring 查询（对照）

Python · index-config
json 
{
  "model": "text-embedding-3-small",
  "dim": 1536,
  "model_version": "2026-03-01",
  "collection": "corpassist-hr-v3",
  "alias": "corpassist-hr-active"
}
Spring · application.yaml
yaml 
spring:
  ai:
    vectorstore:
      milvus:
        collection-name: ${RAG_ALIAS:corpassist-hr-active}
    openai:
      embedding:
        options:
          model: text-embedding-3-small
          dimensions: 1536
Python 索引完成后，Spring 需要重启吗？
若使用 index alias 且 Spring 只读 alias 名，切换 alias 后通常无需重启；若改了 embedding 模型或 dim，必须同步配置并重启或热刷新。
能否 Spring 也跑批处理索引？
可以但不推荐作为 S1 主路径：Python 生态在 PDF 解析、LangChain 管线、离线任务调度上更成熟；Spring 专注在线 spring query 与 ACL。
CorpAssist S1 模型升级时，staging 校验应抽哪些样本？
从基准问答集抽 30～50 条，对比新旧索引的 recall@5 与 MRR；重点覆盖专有名词、条款编号与跨页制度。校验未过则不切 alias，旧索引继续服务。

先想 10 秒：增量文档入库时，要不要对旧 chunk 重算 embedding？

若模型与 dim 未变，仅对新 chunk embed 即可；若文本变更（re-parse/re-chunk），应对受影响 chunk_id 做 upsert，并 bump 文档级 indexed_at。

双栈对齐的验收标准很简单：Python index-config.json 与 Spring application.yaml 的 model、dim、alias 三项一致，且 staging 校验通过后才切换 alias。

本章结论

Embedding 是检索的「坐标系」；双栈共用同一模型与维度契约。

  • 模型选择：在 S1 基准集上实测，记录 model version，索引与查询禁止混模型。
  • 批处理：batch index 要 checkpoint、幂等、staging + alias 切换。
  • 维度：vector dim 变更触发全量 re embed，不能原地混用。
  • 双栈：python index 写、spring query 读，共用 collection 与 metadata schema。

操作步骤

  1. 打开 demos/basics-04-embedding-lab/index-config.json，将 model、dim、collection 改成你选型的一组值，并补充 model_version 字段。
  2. 对照正文「批处理索引」Mermaid，写出 staging → 校验 → alias 切换的三步 checklist（每步一句验收标准）。
  3. 为 Spring application.yaml 写一段与 index-config 维度与模型名一致 的 embedding 配置（可纸面或注释块）。
  4. 假设要从 768 维升级到 1536 维：列出 re-embed 期间查询侧应读 old alias 还是 new alias，以及切换窗口如何定。

判断练习

  1. 排障：召回率骤降。同事提出 ① 重启 Spring ② 索引换更强模型但未 re-embed ③ 检查 query 与 doc 是否同 model ④ 把 TopK 从 5 改 50。请排序应先排查项。

    参考答案

    应先：③ → ②。模型不一致或半库旧向量是最常见根因；换 TopK 不能修复坐标系错乱；重启无效。

  2. 找错：哪条违反双栈契约？① Python 写 corpassist-hr-v3，Spring 读 corpassist-hr-v2 ② 两侧 dim 都是 1536 ③ batch 任务写 metadata.dept_acl

    参考答案

    答：①。索引与查询必须指向同一 active collection/alias。②③ 是正确做法。
