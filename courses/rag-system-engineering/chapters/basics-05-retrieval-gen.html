为什么要学本章

向量检索只解决「召回」；corpassist s1 制度问答还要精排、标注出处、流式呈现。若 TopK 过小漏召回、过大撑爆 context；若无 rerank 则噪声进 prompt；若无 citation 则合规无法审计。本章把检索→生成链路的工程位点讲透。

学完你能

  • 在检索与生成之间插入 rerank，组装 context pack
  • 设计带 citation 的答案格式与 stream gen / sse rag 推送协议
本章先记住 3 件事
  • 两阶段 K：召回 TopK（宽）≠ 进 LLM 条数（窄），中间常靠 rerank 截断。
  • 引用是交付物：答案每个事实点应对应 chunk_id / doc_id，便于复核。
  • 流式先 citation 后 token：sse rag 可先推引用列表再推正文，降低首字延迟感知。

TopK 召回与阈值

top k 是向量检索返回的候选 chunk 数。CorpAssist S1 常见配置：

  • 召回 K=20～50 — 保证制度条款编号、同义词不被漏掉。
  • 进 prompt K=3～5 — 受 LLM context 与噪声约束。
  • score threshold — 低于阈值直接丢弃，为后续拒答铺垫（见 幻觉治理）。
  • mmr — 在 TopK 内做多样性，避免多条重复条款占满 context。
参数典型值调大影响调小影响
召回 TopK 20 召回↑、延迟↑、rerank 成本↑ 漏召回、专有名词 miss
score threshold 0.65～0.75 更多 chunk 进 rerank 更早触发拒答
MMR λ 0.5～0.7 更偏相关、少多样性 更分散、可能丢最佳条
先想 10 秒：为什么「召回 20、进 prompt 5」比「全程 TopK=5」更好？

向量召回是粗排；宽召回 + rerank 精排能在不撑爆 context 的前提下提高 relevant chunk 进入 prompt 的概率。

Rerank 在链路中的位置

CorpAssist S1 · 「年假天数」查询（30 秒） 员工问「入职满一年年假几天？」。向量 TopK=30 召回了「考勤总则」「福利手册」「年假细则」等 chunk，但细则条款 cosine 仅排第 8。rerank（cross-encoder 或 API）将「年假细则 §3.2」升至第 1，再进入 context pack。

Rerank 应放在向量召回之后、LLM 之前：

  1. Query → Query Embedding → VectorStore.similaritySearch(topK=30)
  2. 可选：MMR / 阈值过滤
  3. Rerank(query, candidates) → 取 top 5
  4. 组装 context pack（含 doc_id、section、score）
  5. LLM generate
检索 → Rerank → 生成
sequenceDiagram
  participant U as 用户
  participant SP as Spring RAG API
  participant VS as 向量库
  participant RR as Rerank 服务
  participant LLM as 模型网关
  U->>SP: 年假几天？
  SP->>VS: similaritySearch TopK=30
  VS-->>SP: 30 chunks
  SP->>RR: rerank(query, chunks)
  RR-->>SP: top 5
  SP->>LLM: prompt + context pack
  LLM-->>SP: 流式 token + citations
  SP-->>U: SSE 推送

Spring 侧可用 Advisor 钩子插入 rerank；Python 离线评测可复用同一 rerank 模块做 parity。

反例 vs 推荐

不推荐
  • 直接把 TopK=30 全塞进 prompt
  • 无 rerank，仅靠向量分数排序
  • rerank 与召回用不同 query 文本
推荐
  • 召回宽、rerank 窄、context 有上限 token
  • context pack 带 score 供拒答/审核
  • 双栈共用 rerank 配置与 cutoff
python 
RECALL_K, PROMPT_K = 30, 5
hits = vector_store.similarity_search(query, k=RECALL_K)
hits = [h for h in hits if h.score >= MIN_SCORE]
reranked = rerank_client.rerank(query, hits, top_n=PROMPT_K)
context_pack = build_context_pack(reranked)  # doc_id, section, text

引用（Citation）设计

S1 制度问答的合规要求是：用户看到的每条结论可追溯到库内文档。citation 常见形态：

  • 行内标记 — 「年假 10 天[1]」，脚注链到 doc 标题 + 章节。
  • 结构化 JSON — citations: [{chunk_id, doc_id, page, score}] 随 API 返回。
  • Prompt 约束 — 要求模型「仅依据 context，每句末尾标注 [n]」。
java 
public record RagResponse(
    String answer,
    List<Citation> citations,
    List<ContextChunk> sources
) {}

// Citation: chunkId, documentTitle, sectionPath, relevanceScore
常见误区
  • 只返回纯文本 — 前端无法跳转 PDF 原文，审计失败。
  • 引用与 context 不一致 — 模型编造 [3] 但 context 只有 2 条；需后验校验。
  • ACL 泄漏 — citation 暴露用户无权 doc 标题；检索层须先过滤。

流式输出（Stream Gen）

用户感知延迟取决于首 token 时间。stream gen 通过 sse rag 分事件推送：

SSE 事件类型（标签页）

时机：rerank 完成后、LLM 首 token 前。

内容：sources 列表，前端可先渲染「参考文档」侧栏。

收益：用户 200ms 内看到「有依据」，降低焦虑。

时机：LLM 流式生成期间。

内容：data: {"type":"token","text":"..."}

结束：type: done + 完整 citation 校验结果。

bash 
event: citations
data: {"sources":[{"doc_id":"hr-2024-annual","section":"§3.2"}]}

event: token
data: {"text":"根据制度，入职满一年享有 "}

event: done
data: {"faithfulness":"pass"}
流式输出下如何做 citation 校验？
可在 done 事件前跑轻量校验：抽取答案中的 [n] 是否 ⊆ context_pack；严重不符则改写或拒答（衔接 幻觉治理）。
Python FastAPI 与 Spring 流式协议要对齐吗？
双栈应对齐 SSE event 名与 JSON schema，便于 BFF 聚合与前端一套组件复用。

本章结论

  • TopK：召回宽、进 prompt 窄；配合 score threshold 与 mmr。
  • Rerank：放在向量检索与 LLM 之间，输出 context pack。
  • Citation：结构化返回 + Prompt 约束，满足 S1 合规审计。
  • 流式：sse rag 先 citations 后 token，优化体验。

判断练习

  1. 方案对比：A 召回 TopK=5 直接生成；B 召回 30 + rerank 5 + citation。S1 制度问答应选哪个？各写 1 条理由。

    参考答案

    选 B。宽召回降 miss；rerank 提 precision；citation 满足合规。A 实现简单但漏条款与无出处。

    参考答案

    错：① 和 ③。rerank 需先有召回结果；TopK 过大增加噪声与成本。② 是推荐做法。
