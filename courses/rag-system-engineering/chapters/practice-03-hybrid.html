关键词 + 向量：为什么需要混合

CorpAssist · 政策编号检索 用户问「制度里 7.2 条怎么说」——向量可能偏离；BM25 对「7.2」更敏感。反之用户问「带薪休假有多少天」——向量更稳。

Hybrid search 在检索阶段并行（或串行）执行稀疏与稠密检索，再融合排序。双栈实现：Python 可在 LangChain 自定义 Retriever；Spring 可在 VectorStore 外加 Elasticsearch/OpenSearch 客户端，或平台原生 hybrid API。

混合检索数据流
flowchart LR
  Q[query] --> B[BM25 倒排]
  Q --> V[向量 ANN]
  B --> F[融合 RRF/加权]
  V --> F
  F --> R[Rerank 可选]
  R --> C[context 进 RAG]

融合发生在 Advisor/Retriever 之前，与 practice-01/02 的 RAG 链衔接。

常见误区
  • 为 BM25 单独复制一份未同步的文档库
  • 只对向量结果做关键词 contains 过滤
  • 权重写死在 Java/Python 代码里
推荐做法
  • shared index：同一 doc_id 主键
  • 融合配置进 hybrid-config.json
  • hybrid tuning 用评测集扫权重

权重与融合：RRF 与加权分

融合策略二选一或组合：加权线性（需归一化分数）与 Reciprocal Rank Fusion（RRF）（只看排名，对尺度不敏感）。CorpAssist 入门先用加权，便于理解 keyword boost。

共用配置 hybrid-config.json

json 
{
  "collection": "corpassist-hr-v2",
  "bm25_weight": 0.45,
  "vector_weight": 0.55,
  "fusion": "weighted",
  "top_k_each": 20,
  "final_top_k": 4,
  "rrf_k": 60
}

Python 读取示例：

python 
import json
from pathlib import Path

cfg = json.loads(Path("demos/practice-03-hybrid-lab/hybrid-config.json").read_text())
# score = cfg["vector_weight"] * vec_score + cfg["bm25_weight"] * bm25_score
# 按 doc_id 合并后取 final_top_k

Spring 在 application.yml 引用同文件或镜像字段：

yaml 
corpassist:
  rag:
    hybrid:
      config-path: classpath:hybrid-config.json
      bm25-weight: 0.45
      vector-weight: 0.55
FAQ：bm25_weight + vector_weight 必须等于 1 吗？

加权融合时常归一化为 1；RRF 不使用权重和。调参时固定 final_top_k，只扫相对比例，并在 practice-05 评测集上看 recall@k 变化。

共用索引：doc_id 与元数据一致

共用索引 要求：倒排索引与向量库指向同一批 chunk，doc_id、tenant_id、kb 元数据一致。索引 job（Python）写入时同时 upsert BM25 与向量；禁止两路 ingest 不同步。

先想 10 秒：为何不让 LLM 同时「猜关键词」？

混合检索是确定性召回；让 LLM 猜关键词成本高且不可复现。查询改写放在 advanced-01，与本章检索融合正交。

本章结论

  • 动机：专有名词/编号偏 BM25，语义 paraphrase 偏向量；混合提升召回。
  • 权重：hybrid-config.json 双栈共享；加权或 RRF 融合。
  • 索引：一路 doc_id，双路 ingest 同步；检索 filter 与 practice-02 一致。
  • 衔接：融合结果进入既有 RAG Advisor/链，后续可 rerank。

动手练习

改 demos/practice-03-hybrid-lab/hybrid-config.json 权重，用 curl 对比「7.2 条」与「年假」两题的 sources 排序。

操作步骤

  1. 复制 hybrid-config.json 为 hybrid-config.baseline.json 备份。
  2. 设 bm25_weight: 0.7、vector_weight: 0.3，对问题「7.2 条」curl，记录 top1 doc_id。
  3. 改回 0.3 / 0.7，对「年假几天」再 curl，对比 top1 是否变化。
  4. 在双栈对照笔记写：何时应提高 bm25_weight（1 句话 + 1 个 CorpAssist 例子）。

判断练习

  1. （配置）要把专有名词召回拉高，应调大还是调小 bm25_weight？若同时 final_top_k: 2 过小会有何副作用？

    参考答案

    调大 bm25_weight。final_top_k 过小会丢掉融合后排在 3～4 位的相关片段，导致 RAG 上下文不足或拒答增多。

  2. （配置）Spring yaml 写 0.45/0.55，Python 仍用默认 0.5/0.5。parity 失败时优先改哪一处真源？

    参考答案

    以仓库内 hybrid-config.json 为真源，双栈启动时加载同路径；禁止两处手写不同默认值。
