关键词 + 向量：为什么需要混合

CorpAssist · 政策编号检索 用户问「制度里 7.2 条怎么说」——向量可能偏离；BM25 对「7.2」更敏感。反之用户问「带薪休假有多少天」——向量更稳。

Hybrid search 在检索阶段并行（或串行）执行稀疏与稠密检索，再融合排序。双栈实现：Python 可在 LangChain 自定义 Retriever；Spring 可在 VectorStore 外加 Elasticsearch/OpenSearch 客户端，或平台原生 hybrid API。

混合检索数据流
flowchart LR
  Q[query] --> B[BM25 倒排]
  Q --> V[向量 ANN]
  B --> F[融合 RRF/加权]
  V --> F
  F --> R[Rerank 可选]
  R --> C[context 进 RAG]

融合发生在 Advisor/Retriever 之前，与 practice-01/02 的 RAG 链衔接。

常见误区
  • 为 BM25 单独复制一份未同步的文档库
  • 只对向量结果做关键词 contains 过滤
  • 权重写死在 Java/Python 代码里
推荐做法
  • shared index：同一 doc_id 主键
  • 融合配置进 hybrid-config.json
  • hybrid tuning 用评测集扫权重
问法类型CorpAssist 示例更依赖调参方向

口语 paraphrase 「带薪假能休多久」 向量 提高 vector_weight

Python：EnsembleRetriever 骨架

python 
from langchain.retrievers import EnsembleRetriever

bm25_retriever = BM25Retriever.from_documents(docs)  # 与向量同 doc_id
vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 20})

hybrid = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[cfg["bm25_weight"], cfg["vector_weight"]],
)

FAQ：混合检索放在 Retriever 内还是 Advisor 外？

放在 Retriever 层（融合后再进 RAG 链）。若在生成后做关键词过滤，会丢掉向量路已召回的相关片段。Spring 侧可自定义 Retriever Bean 或在 Advisor 前注入融合服务。

权重与融合：RRF 与加权分

融合策略二选一或组合：加权线性（需归一化分数）与 Reciprocal Rank Fusion（RRF）（只看排名，对尺度不敏感）。CorpAssist 入门先用加权，便于理解 keyword boost。

FAQ：bm25_weight + vector_weight 必须等于 1 吗？

加权融合时常归一化为 1；RRF 不使用权重和。调参时固定 final_top_k，只扫相对比例，并在 practice-05 评测集上看 recall@k 变化。

RRF 融合公式（面试常问）

python 
def rrf_score(rank: int, k: int = 60) -> float:
    return 1.0 / (k + rank)

def fuse_rrf(bm25_ranks: dict[str, int], vec_ranks: dict[str, int]) -> list[str]:
    scores: dict[str, float] = {}
    for doc_id, rank in bm25_ranks.items():
        scores[doc_id] = scores.get(doc_id, 0) + rrf_score(rank)
    for doc_id, rank in vec_ranks.items():
        scores[doc_id] = scores.get(doc_id, 0) + rrf_score(rank)
    return sorted(scores, key=scores.get, reverse=True)

RRF 不依赖 BM25/向量原始分数尺度，适合两路引擎分数不可比时；CorpAssist 默认加权便于运营理解，RRF 作为 A/B 备选写在 hybrid-config.json 的 fusion 字段。

加权融合坑
  • BM25 分数 0～30、向量 cosine 0～1 直接相加
  • 未做 min-max 归一化
  • 向量路永远 dominate
稳健做法
  • 各路先取 top_k_each 再融合
  • 加权前归一化或改 RRF
  • fusion score 写入 trace 便于调参

共用索引：doc_id 与元数据一致

共用索引 要求：倒排索引与向量库指向同一批 chunk，doc_id、tenant_id、kb 元数据一致。索引 job（Python）写入时同时 upsert BM25 与向量；禁止两路 ingest 不同步。

先想 10 秒：为何不让 LLM 同时「猜关键词」？

混合检索是确定性召回；让 LLM 猜关键词成本高且不可复现。查询改写放在 advanced-01，与本章检索融合正交。

FAQ：共用索引是否必须用 Elasticsearch？

不必须。PGVector + 外挂 BM25（OpenSearch、Meilisearch、甚至内存 BM25）均可；关键是 doc_id 主键一致 与双栈读同一 hybrid-config.json。选型看运维能力与 ACL 过滤语法是否满足 多知识库与路由。

本章结论

  • 动机：专有名词/编号偏 BM25，语义 paraphrase 偏向量；混合提升召回。
  • 权重：hybrid-config.json 双栈共享；加权或 RRF 融合。
  • 索引：一路 doc_id，双路 ingest 同步；检索 filter 与 practice-02 一致。
  • 衔接：融合结果进入既有 RAG Advisor/链，后续可 rerank。

判断练习

  1. （配置）要把专有名词召回拉高，应调大还是调小 bm25_weight？若同时 final_top_k: 2 过小会有何副作用？

    参考答案

    调大 bm25_weight。final_top_k 过小会丢掉融合后排在 3～4 位的相关片段，导致 RAG 上下文不足或拒答增多。

  2. （配置）Spring yaml 写 0.45/0.55，Python 仍用默认 0.5/0.5。parity 失败时优先改哪一处真源？

    参考答案

    以仓库内 hybrid-config.json 为真源，双栈启动时加载同路径；禁止两处手写不同默认值。
