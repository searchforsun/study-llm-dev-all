VectorStore：连接共用索引

CorpAssist · 同一 PGVector collection Python 夜间 job 写入 corpassist-hr-v2；白天 Spring 只读查询。禁止 Spring 侧换 embedding 模型或 collection 名不自知。

Spring AI VectorStore 抽象 PGVector、Redis、Elasticsearch 等后端。CorpAssist 选用与 Python 相同的 JDBC URL 与表/ collection。

application.yml（向量库 + 模型网关）

yaml 
spring:
  ai:
    vectorstore:
      pgvector:
        index-type: HNSW
        distance-type: COSINE_DISTANCE
        dimensions: 1536
    openai:
      api-key: ${OPENAI_API_KEY}
      embedding:
        options:
          model: text-embedding-3-small
      chat:
        options:
          model: gpt-4o-mini

corpassist:
  rag:
    collection: corpassist-hr-v2
    default-top-k: 4

检查项Python 索引Spring 查询
collection 名 corpassist-hr-v2 写入 meta corpassist.rag.collection 一致
embedding 模型 text-embedding-3-small 同 model + dimensions: 1536
距离度量 cosine（写入时） COSINE_DISTANCE
只读约束 批处理 upsert 禁止 Controller 直接 add()
FAQ：Spring 启动报 dimension mismatch 怎么查？

对照 application.yml 的 dimensions 与 PGVector 表定义；再查 Python 索引 job 写入的 meta。常见根因：换 embedding 模型后只 re-embed 了 Python 侧、Spring yaml 未同步。

QuestionAnswerAdvisor：挂载 RAG 到 ChatClient

QuestionAnswerAdvisor 是 Spring AI 1.x 推荐的 Advisor RAG 方式：在 ChatClient 上注册后，自动完成「检索 → 填入 Prompt → 调用 ChatModel」。

Java 配置与 Controller 片段

java 
// demos/practice-02-spring-rag-lab/src/main/java/.../SpringRagDemo.java 思路
@Configuration
class RagConfig {
  @Bean
  ChatClient chatClient(ChatClient.Builder builder, VectorStore vectorStore) {
    var advisor = QuestionAnswerAdvisor.builder(vectorStore)
        .searchRequest(SearchRequest.builder()
            .topK(4)
            .filterExpression("tenant_id == 'corp-a' && kb == 'hr'")
            .build())
        .build();
    return builder.defaultAdvisors(advisor).build();
  }
}

@RestController
@RequestMapping("/v1/rag")
class RagController {
  private final ChatClient chatClient;

  @PostMapping("/ask")
  AskResponse ask(@RequestBody AskRequest req) {
    String answer = chatClient.prompt()
        .user(req.question())
        .call()
        .content();
    // 从 Advisor 上下文或 Document 列表映射 sources — 与 Python 同 DTO
    return new AskResponse(answer, mapSources(...));
  }
}
Java 侧反模式
  • Controller 内手写 JDBC 查向量
  • TopK=20 且无 rerank
  • 返回字段 citations 与 Python sources 不一致
Advisor 推荐
  • filter 与 Python search_kwargs.filter 同语义
  • spring retrieval 参数集中配置
  • 集成测试对比 golden answer 的 doc_id 集合

AskResponse DTO（与 Python Pydantic 对齐）

java 
public record SourceRef(String docId, String snippet) {}
public record AskRequest(String question, String tenantId) {}
public record AskResponse(String answer, List<SourceRef> sources) {}

// JSON 序列化：Spring 默认 camelCase；若 Python 用 snake_case，
// 在 @JsonProperty 或全局 ObjectMapper 统一，避免 parity 工具误判

dto align 是双栈联调第一关：字段名、空列表语义（sources: [] 表示拒答或检索失败）、snippet 截断长度建议双栈均为 200 字符。

对照 Python：管线映射表

面试与联调时常问「Spring 这段等价 LangChain 哪一段」。下表用于 双栈对照 文档（建议写入团队 Wiki）。

Python（practice-01）Spring AI（本章）契约
vectorstore.as_retriever(k=4) SearchRequest.topK(4) TopK 一致
filter: {tenant_id, kb} filterExpression 检索层 ACL
ChatPromptTemplate Advisor 内置 QA 模板（可自定义） 「仅根据 context」
AskResponse record AskResponse answer + sources[]
FAQ：能否 Spring 也做离线索引？

可以但不推荐双写。CorpAssist 约定 Python 批处理索引 + Spring 查询，减少两套 ingest 逻辑漂移。若必须 Java ingest，需与 Python 共用同一 pipeline 版本号。

Python LCEL 等价片段（对照阅读）

python 
# practice-01 最小链 — 与本章 Advisor 语义对应
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
# Advisor ≈ retriever + 默认 QA 模板 + ChatModel 一次调用
  • Python：search_kwargs 字典
  • Spring：SearchRequest.builder() + filterExpression 字符串
  • 契约：filter 字段名必须与索引 metadata 键一致（tenant_id vs tenantId 是常见坑）
  • Python：LangSmith / 自定义 traced_retrieve
  • Spring：Micrometer + MDC trace_id
  • 联调：同一 question 对比两侧 top_doc_id 日志

RagParityTest 思路（集成测试）

java 
@SpringBootTest(webEnvironment = RANDOM_PORT)
class RagParityTest {
  @Test
  void sameQuestionSameDocIds() {
    var body = Map.of("question", "年假几天", "tenantId", "corp-a");
    var resp = rest.postForObject("/v1/rag/ask", body, AskResponse.class);
    var docIds = resp.sources().stream().map(SourceRef::docId).sorted().toList();
    assertThat(docIds).contains("hr-policy-2024#sec-3.2"); // 与 Python 黄金集对齐
  }
}
先想 10 秒：parity 失败时先改 Prompt 还是先对齐 VectorStore？

先对齐 VectorStore（collection、embedding、filter、topK）。两侧 doc_id 一致后再比 answer 文本；否则是在比较不同检索结果上的生成差异。

动手练习

改 yaml 中 topK 与 filter，用 mvn 启动后 curl 对照 Python 同一问题的 sources。

操作步骤

  1. 在 application-local.yml 设置 corpassist.rag.collection 与 Python 一致。
  2. 将 default-top-k 从 4 改为 8，curl 对比 sources 条数变化。
  3. 故意写错 filterExpression 的 kb，确认 sources 为空（与 Python 实验对照）。
  4. 运行 mvn test -Dtest=RagParityTest（或 lab 中占位测试），记录 doc_id 差异。

判断练习

  1. （找错）同事在 Controller 里先 chatClient.call() 再手动 vectorStore.similaritySearch 拼 context。错在哪？应如何改？

    参考答案

    顺序反了：未检索就生成，必然胡编。应使用 QuestionAnswerAdvisor 或先检索再带 context 调用；检索 filter 与 TopK 进 SearchRequest，勿散落 Controller。

  2. （找错）parity 测试失败：Python 有 pol-102，Java 只有 pol-99。yaml 与代码中最常见的 2 处不一致？

    参考答案

    embedding 模型/维度或 collection 名不一致；或 filterExpression / topK 与 Python search_kwargs 不同。先对齐配置再比 LLM 输出。
