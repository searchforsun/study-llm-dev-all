为什么要学本章

CorpAssist 不只有 HR 政策：还有 IT 手册、财务制度等多个 知识库。若所有问题都打向默认 collection，会串库回答或越权引用。本章在 practice-03 混合检索之上，用 意图路由 选库，并在检索层施加 acl filter 与 租户隔离。

学完你能

  • 在 Python/Spring 检索前应用 部门范围 与 tenant filter
  • 说明 router chain 与单一 RAG 链的关系

本章先记住 3 件事
  • 先路由后检索：选错库比检索算法差更致命。
  • 权限在检索层：Prompt 禁止越权不能替代 metadata filter。
  • 配置真源：新增库只改 yaml，不改 Java/Python 分支地狱。

意图路由：选 HR 还是 IT

CorpAssist · 企业助手入口 员工问「VPN 连不上」应路由 it 库；问「产假天数」应路由 hr 库。路由可规则（关键词）或小模型分类器，结果写入 kb 字段供检索 filter 使用。

Intent router 输出 {intent, kb, confidence}。低置信度时走澄清或默认库（需产品定义）。kb routing 配置应与混合检索、RAG 链解耦，便于运营增库。

多库 RAG 路由
flowchart TB
  Q[用户问题] --> IR[Intent Router]
  IR -->|hr| R1[HR Retriever]
  IR -->|it| R2[IT Retriever]
  R1 --> ACL[ACL filter]
  R2 --> ACL
  ACL --> RAG[RAG 生成]

每个 Retriever 指向不同 collection，但共用同一套 ACL 规则引擎。

kb-routing.yaml（Demo 真源）

yaml 
# demos/practice-04-multi-kb-lab/kb-routing.yaml
knowledge_bases:
  hr:
    collection: corpassist-hr-v2
    intents: [leave, benefit, policy]
    keywords: [年假, 产假, 人事]
  it:
    collection: corpassist-it-v1
    intents: [vpn, account, device]
    keywords: [VPN, 密码, 工单]
  finance:
    collection: corpassist-fin-v1
    intents: [expense, invoice]
    keywords: [报销, 发票]

default_kb: hr
router:
  mode: keyword_first   # keyword_first | classifier
  min_confidence: 0.6

权限与租户：acl filter 在检索层

ACL filter 将用户 JWT 中的 tenant_id、dept_ids、roles 转为向量库/倒排 filter。与 幻觉治理 的拒答互补：检不到优于检到无权限文档。

Python：路由 + ACL 片段

python 
def build_filter(user, kb: str) -> dict:
    return {
        "tenant_id": user.tenant_id,
        "kb": kb,
        "dept": {"$in": user.dept_ids},
    }

def route_and_ask(question: str, user) -> dict:
    kb = route_kb(question)  # 读 kb-routing.yaml
    retriever = get_retriever(cfg.knowledge_bases[kb].collection)
    docs = retriever.invoke(question, filter=build_filter(user, kb))
    return rag_with_docs(question, docs)

Spring：filterExpression 与请求头

java 
// 从 SecurityContext 取 tenantId、deptIds
String kb = kbRoutingService.route(request.question());
String filter = String.format(
    "tenant_id == '%s' && kb == '%s' && dept in [%s]",
    tenantId, kb, deptCsv);

SearchRequest search = SearchRequest.builder()
    .topK(4)
    .filterExpression(filter)
    .build();

Multi-tenant 要求：索引 metadata 必含 tenant_id；禁止跨租户默认库。Tenant isolation 需在 ingest 与 query 双侧校验。

FAQ：路由错了怎么办？

产品层：低置信度澄清；技术层：记录 routed_kb 与 sources 供评测；运营层：在 yaml 补关键词。不要用「检索全库」兜底。

配置驱动：增库不改代码

新增「法务库」时，理想 diff 仅为 kb-routing.yaml + 新 collection 的 ingest job，而非改 5 个 Java 类。Spring 用 @ConfigurationProperties 绑定；Python 启动时 load yaml。

yaml 
corpassist:
  rag:
    routing-config: classpath:kb-routing.yaml
    enforce-acl: true
bash 
curl -s -X POST http://localhost:8080/v1/rag/ask \
  -H "Content-Type: application/json" \
  -H "X-Tenant-Id: corp-b" \
  -d '{"question":"VPN 无法连接"}' | jq .sources[].doc_id

本章结论

  • 路由：kb-routing.yaml 定义库、意图、关键词；先路由后检索。
  • 权限：acl filter 在 Retriever/SearchRequest，含 tenant + dept + kb。
  • 租户：metadata 与请求头一致，禁止跨租户 fallback 全库。
  • 配置：增库以 yaml 为真源，双栈共享路由文件。

操作步骤

  1. 在 kb-routing.yaml 为 it 库增加关键词 Outlook，保存后重启 Spring/Python 服务（或热加载若已实现）。
  2. curl 提问「Outlook 收不到邮件」，确认 sources 的 metadata 含 kb: it（或 it 库 doc_id 前缀）。
  3. 用 X-Tenant-Id: corp-b 提问 HR 问题，确认不出现 corp-a 专属 doc_id。
  4. 将 default_kb 改为 it，对模糊问题「请假」观察误路由，再改回 hr。

判断练习

  1. （排障）用户有 hr 角色，却检索到 finance 库文档。列出 yaml、filter、ingest metadata 各 1 个排查点。

    参考答案

    yaml：关键词/意图是否误匹配 finance；filter：是否缺少 kb 条件；ingest：chunk 是否错标 kb 或 tenant。

  2. （排障）路由正确但 sources 为空。ACL 与混合检索配置中优先检查哪 2 项？

    参考答案

    filter 中 dept/tenant 是否过严；collection 名是否与 kb-routing 中该库一致（非 default 库名）。
