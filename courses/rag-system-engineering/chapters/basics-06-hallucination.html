为什么要学本章

RAG 不能消除幻觉，只能约束与兜底。corpassist s1 制度场景的原则是：宁可 refusal，不可编造。检索分数低、生成与 context 不符、或涉及敏感政策时，需要拒答、进 human review，并用 faithfulness metric 持续监控。

学完你能
  • 设计分层 hallucination guard：检索阈值拒答、生成后 groundedness 校验
  • 用 confidence score 路由到自动答复 / 拒答 / 人工队列
  • 定义 S1 的 faithfulness metric 与 audit trail 字段
本章先记住 3 件事
  • 拒答是产品能力：abstain 模板要明确、友好，并引导用户改问法或找人工。
  • 置信度多源：检索 max score、rerank 分差、LLM 自评仅作参考，需组合而非单点。
  • 指标闭环：线上采样 + 人工标注 → 调 threshold → RAG 评测闭环 回归。

拒答（Refusal）策略

拒答触发条件（满足其一即可 abstain）：

  • 检索 top1 score < min_score（无可靠依据）
  • 用户问题超出知识库 scope（如问 S4 代码库内容却走 S1 路由）
  • 生成后 groundedness 校验失败（答案句无法在 context 中找到支撑）
  • 安全策略命中（越权 doc、注入片段）
yaml 
rag:
  guard:
    min_score: 0.72
    refusal_template: "未在制度库中找到可靠依据，请补充关键词或联系 HR 值班。"
    abstain_on_empty_context: true

反例 vs 推荐

不推荐
  • 无检索结果仍让 LLM「尽量回答」
  • 拒答文案为空或 500 错误
  • 拒答不记 audit trail
推荐
  • Prompt 明确：无依据则输出固定 refusal 句
  • 返回 status: refused + 建议动作
  • 记录 query、scores、候选 doc_ids
先想 10 秒：拒答率上升 10%，一定是坏事吗？

不一定。可能是 threshold 收紧或库内确实缺文档；需结合 faithfulness metric 与用户投诉率一起看。

置信度（Confidence Score）

CorpAssist S1 · 灰色地带（25 秒） 问「远程办公补贴标准？」检索 top1=0.74、top2=0.71，条款版本冲突。系统 confidence score 判为 medium → 生成简答 + 打标进人工复核，而非直接拒答或高置信发布。

工程上常用组合分数（示意，权重需评测校准）：

信号含义权重思路
retrieval_max_score 最佳 chunk 相似度 低则倾向拒答
rerank_margin top1 − top2 分差 小则 ambiguous
groundedness_check 答案句覆盖 context 比例 生成后校验
citation_valid_rate 引用编号合法比例 低则拒发或改写
0.72

阈值↑ → 拒答↑、幻觉↓、用户「答不上来」投诉可能↑。

人工审核（Human Review）

并非所有问答都要人工；human review 队列应聚焦：

  • confidence ∈ (review_low, review_high) 的灰色答案
  • 用户点击「不准确」反馈
  • 涉及薪酬、合规、解雇等高风险 intent
  • faithfulness 自动校验 borderline
幻觉治理路由
flowchart TD
  Q[用户问题] --> RET[检索 + Rerank]
  RET --> SC{max score >= min?}
  SC -->|否| REF[拒答模板]
  SC -->|是| GEN[LLM 生成]
  GEN --> GD{groundedness OK?}
  GD -->|否| REF
  GD -->|是| CF{confidence}
  CF -->|高| AUTO[自动发布 + citation]
  CF -->|中| HR[人工审核队列]
  CF -->|低| REF
  HR --> AUTO
  REF --> AUDIT[audit trail]

拒答与发布都写 audit；人工审核 SLA 需单独定义（如 4h 内）。

yaml 
rag:
  guard:
    min_score: 0.72
    review_band: [0.72, 0.85]
    human_review_queue: hr-policy-review
  audit:
    log: query, scores, chunk_ids, decision, reviewer_id

指标与监控

faithfulness metric 衡量「答案是否被 context 支撑」；groundedness 常与之同族。S1 建议盯：

  • faithfulness@sample — 每日抽检 N 条，人工或 LLM-as-judge
  • refusal_rate — 拒答占比，按 intent 分桶
  • review_backlog — 人工队列深度与超时率
  • citation_error_rate — 引用编号非法占比
指标异常可能原因动作
faithfulness 下降 chunk 质量差、prompt 过松 收紧 prompt、加 groundedness 校验
refusal_rate 飙升 min_score 过高、库缺文档 补库或略降阈值 + 评测验证
review 积压 review_band 过宽 收窄 band 或增派人手
LLM 自评 confidence 可信吗？
单独使用不可靠；仅作 hallucination guard 的一路信号，必须与检索分、groundedness 交叉验证。
拒答和幻觉哪个更伤业务？
S1 制度场景：错误答案 > 拒答。拒答可引导人工；幻觉可能引发合规风险。

CorpAssist S1 幻觉治理分层

Hallucination guard 应覆盖检索前、生成中、生成后三层，而非只在最后抽检：

层级拦截点S1 动作
检索前 路由 / scope 检测 非 S1 意图拒答或转路由
检索后 max score < min_score refusal 模板，不写 audit 外的生成
生成后 groundedness / citation 校验 拒发、改写或进 human review

audit trail 最小 JSON（示意）

json 
{
  "query_id": "q-8f2a",
  "query": "远程办公补贴标准？",
  "max_score": 0.74,
  "rerank_margin": 0.03,
  "decision": "human_review",
  "chunk_ids": ["c-hr-12", "c-hr-09"],
  "faithfulness": null,
  "reviewer_id": null
}
治理反例
  • 只监控生成 token，不记 retrieval score
  • 拒答不写 audit，无法分析「库缺还是阈值紧」
  • 人工审核结论不回写评测集
S1 闭环
  • 拒答/发布均写 audit trail
  • 周度看 faithfulness@sample + refusal_rate 分 intent
  • 阈值变更走评测门禁再上线
先想 10 秒：faithfulness 升、refusal_rate 也升，说明什么？

常见是阈值收紧生效：更少低质量答案流出，但更多问题被拒答。需看用户投诉是否从「答错」转向「答不上」；若后者可控，往往是正确 trade-off。

本章结论

  • 拒答：无依据 abstain，模板 + audit，是正常路径非异常。
  • 置信度：多信号组合，驱动自动 / 拒答 / human review 三路。
  • 审核：灰色地带进队列，保留 audit trail。
  • 指标：faithfulness metric 与 refusal_rate 联动调参。

操作步骤

  1. 打开 guard-config.yaml，将 min_score 从 0.7 改为 0.75 与 0.65 各跑一次（纸面模拟），分别预测 refusal_rate 与 faithfulness 变化方向。
  2. 为「远程办公补贴」灰色场景写一条 confidence 组合公式（至少 2 个信号）。
  3. 设计 audit 日志 JSON：含 query、max_score、decision（auto/refuse/review）、chunk_ids。
  4. 对照 Mermaid 路由图，标出与 检索、重排与生成 章节的 score threshold 衔接点。

判断练习

  1. 排障：faithfulness 尚可但用户投诉「总说不确定」。最可能配置问题是？① min_score 过高 ② 未返回 citation ③ review_band 过窄

    参考答案

    最可能：①。阈值过高导致频繁 refusal。③ 导致人工慢而非直接拒答；② 影响信任但不必然「不确定」文案。

  2. 找错：哪条违反 S1 原则？① 无 chunk 仍生成 ② 低置信进 human review ③ 拒答写 audit

    参考答案

    答：①。无依据必须 abstain。②③ 是推荐治理手段。
