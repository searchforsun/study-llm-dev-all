LCEL 管线：从问题到带引用的回答

CorpAssist S1 · HR 政策问答 员工问「年假几天」——管线必须先检索 corpassist-hr-v2 分块，再把片段填入 Prompt；禁止跳过 Retriever 直接问 LLM。

LangChain Expression Language（LCEL） 用声明式管道组合 RAG：输入 dict → 检索 → 模板 → 模型 → 解析。与 Embedding 与索引 中 Python 写库、Spring 查库的分工衔接：本章链读取同一 collection。

最小 RAG 链（检索 + Prompt + Mock LLM）

python 
# demos/practice-01-python-rag-lab/chain_stub.py 扩展思路
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

prompt = ChatPromptTemplate.from_messages([
    ("system", "仅根据 context 回答；无依据则说不知道。\n{context}"),
    ("human", "{question}"),
])

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

# retriever 来自 VectorStore.as_retriever(search_kwargs={"k": 4})
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
链式反模式
  • 在 Controller 里手写拼接 Prompt 字符串
  • 检索与生成拆成两个无关联 HTTP 调用
  • 返回纯文本、无 doc_id 引用
LCEL 推荐
  • 单条 rag chain 可 .invoke / .stream
  • chain compose 可插 rerank、filter 中间件
  • 统一包装为 {"answer","sources"}
CorpAssist Python RAG 链（本章范围）
flowchart LR
  Q[question] --> R[Retriever]
  R --> C[context 拼装]
  C --> P[ChatPrompt]
  P --> L[LLM / Mock]
  L --> A[answer + sources]

索引侧见 basics-02～04；本章只实现查询链 + API，不写离线 ingest。

LCEL 与 Spring Advisor 对照（双栈预习）

LCEL 组件Spring AI（下一章）CorpAssist 契约
RunnablePassthrough 传 question ChatClient.prompt().user() 用户问句原样进检索与生成
retriever | format_docs QuestionAnswerAdvisor 内置检索 context 拼装发生在 LLM 调用前
.stream() 流式 ChatClient.stream() SSE 场景见 basics-05 sse rag
FAQ：LCEL 链何时用 .with_config({"run_name": "corp-rag"})？

联调与生产 trace 需要区分「检索 span」与「生成 span」。CorpAssist 在 LangSmith / OpenTelemetry 中为链命名，便于对比 Python 与 Spring 同一问题的延迟分布；Mock LLM 阶段也应保留 run_name，避免上线后 trace 断层。

Retriever：TopK、过滤与元数据

LangChain Retriever 封装向量库查询，对上层链暴露 invoke(query) → Document[]。CorpAssist 在 metadata 中写入 tenant_id、dept、doc_id（与后续 acl filter 章对齐）。

Retriever 配置片段

python 
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={
        "k": 4,
        "filter": {"tenant_id": "corp-a", "kb": "hr"},
    },
)

# 调试：只看检索、不调用 LLM
docs = retriever.invoke("年假 几天")
for d in docs:
    print(d.metadata.get("doc_id"), d.page_content[:80])
参数建议CorpAssist 注意点
k 4～8 起步，配合 检索、重排与生成 过大拖慢 API、挤爆上下文
filter 检索层强制 tenant / kb 勿只靠 Prompt「请勿越权」
score 记录 top1 分数进 trace 供拒答阈值（basics-06）
FAQ：Retriever 与 VectorStore 谁负责 Embedding？

查询时 Retriever 内部用与索引相同的 embedding 模型做 query vector；模型版本变更必须 re-embed（见 basics-04）。链上不要手写两套 embed 函数。

包装 sources：与 Spring DTO 同形的后处理

python 
def rag_chain_with_sources(question: str, *, filter: dict) -> dict:
    docs = retriever.invoke(question, filter=filter)
    if not docs:
        return {"answer": "未在知识库中找到依据。", "sources": []}
    context = format_docs(docs)
    answer = (prompt | llm | StrOutputParser()).invoke(
        {"question": question, "context": context}
    )
    sources = [
        {"doc_id": d.metadata["doc_id"], "snippet": d.page_content[:200]}
        for d in docs
    ]
    return {"answer": answer, "sources": sources}

Spring 侧 mapSources(...) 应产出相同字段；parity test 比较 doc_id 集合而非 answer 文本（LLM 措辞允许差异）。

先想 10 秒：为何先 Mock LLM 再换真模型？

分离检索问题与生成问题：Mock 固定返回可预测文本，若 sources 已错，接真模型只会浪费 token 且难排查。

结构化 trace 日志（生产最小集）

python 
import logging
log = logging.getLogger("corpassist.rag")

def traced_retrieve(question: str, filter: dict):
    docs = retriever.invoke(question, filter=filter)
    log.info(
        "retrieve",
        extra={
            "question_len": len(question),
            "filter": filter,
            "hit_count": len(docs),
            "top_doc_id": docs[0].metadata.get("doc_id") if docs else None,
            "top_score": getattr(docs[0], "score", None) if docs else None,
        },
    )
    return docs

与 Spring MDC（tenant_id、trace_id）对齐后，可在同一日志平台对比双栈检索命中率；排障时优先看 hit_count=0 再查 filter 与 embedding。

API 暴露：FastAPI 与 Spring 契约对齐

对外 BFF 将由 Spring AI：RAG 实现 承担；Python 服务常用于索引作业、回归脚本、原型。无论哪条栈，python rag api 响应体应对齐：

python 
# FastAPI 片段 — 与 Spring DTO 同形
from fastapi import FastAPI
from pydantic import BaseModel

class AskRequest(BaseModel):
    question: str
    tenant_id: str = "corp-a"

class SourceRef(BaseModel):
    doc_id: str
    snippet: str

class AskResponse(BaseModel):
    answer: str
    sources: list[SourceRef]

@app.post("/v1/rag/ask", response_model=AskResponse)
def ask(req: AskRequest):
    out = rag_chain_with_sources(req.question, filter={"tenant_id": req.tenant_id})
    return AskResponse(answer=out["answer"], sources=out["sources"])

双栈 curl 对照脚本

bash 
# 同一问题、同一 tenant — 比较 doc_id 集合（顺序可不同）
Q='{"question":"年假几天","tenant_id":"corp-a"}'
py_ids=$(curl -s -X POST http://localhost:8000/v1/rag/ask -H "Content-Type: application/json" -d "$Q" \
  | jq -r '.sources[].doc_id' | sort)
sp_ids=$(curl -s -X POST http://localhost:8080/v1/rag/ask -H "Content-Type: application/json" -d "$Q" \
  | jq -r '.sources[].doc_id' | sort)
diff <(echo "$py_ids") <(echo "$sp_ids") || echo "parity OK"
FAQ：Python 原型 API 何时下线？

Spring BFF 稳定且 parity 连续通过后，Python 服务保留为索引作业 + 评测批跑入口，对外查询流量切到 8080。避免长期双写两套鉴权与限流逻辑。

判断练习

  1. （排障）curl 返回 sources: [] 但向量库有数据。列出按优先级排查的 3 个检查点（配置/代码/数据各至少一项）。

    参考答案

    ① filter/collection 是否与索引一致；② embedding 模型与维度是否匹配 basics-04；③ 问题语言/分词是否导致相似度极低（可试混合检索）。勿先换 LLM。

  2. （排障）链式 trace 显示 context 有内容，但 answer 仍胡编。Prompt 与调用顺序上最可能缺哪一环？

    参考答案

    system 未约束「仅根据 context」；或 context 未传入最终 Prompt 模板变量；或绕过了带 context 的链而直接调 LLM。
