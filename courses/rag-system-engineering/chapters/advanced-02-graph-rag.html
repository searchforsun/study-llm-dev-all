为什么要学本章

当用户问「A 部门与 B 制度有哪些交叉引用？」纯向量 chunk 难以串联多跳关系。graph rag 在 entity graph 上做社区摘要与多跳检索；但构建与推理成本显著高于 Naive RAG。工程上要先回答：when to use、graph cost 是否可接受、能否用 simple alternative 解决。

FAQ：GraphRAG 与知识图谱（KG）是一回事吗？

不完全是。GraphRAG 强调从非结构化文档自动抽图 + 社区摘要 + RAG 查询；传统 KG 常有人工建模本体。CorpAssist S1 可先「轻量共现边 + 规则 REFERENCES」，再按需升级到 LLM 抽取，避免一步到位全量 GraphRAG。

local vs global 检索模式

Local：从问句实体出发，沿边扩展 1～2 跳，适合「A 引用 B 哪些条款」。Global：先选相关社区摘要，再下钻 chunk，适合「全库合规主题分布」。CorpAssist 默认 FAQ 不走二者，由 intent 路由显式选择，避免每次查询遍历全图。

  • Local P95 目标：+300～500ms（在向量 baseline 上）
  • Global P95 目标：+800ms～1.2s，仅对明确 global 意图开放
  • 监控：按 route 分桶记录 recall，防止图路由「误伤」简单 FAQ

成本

graph cost 主要来自三处：

  1. 实体/关系抽取：每 chunk 一次 LLM 调用（或 NER 模型 + LLM 校验）
  2. 社区摘要：Leiden/Louvain 分社区后对每社区生成摘要
  3. 查询期：多跳遍历 + 更大 context pack

先想 10 秒：为何社区摘要成本常高于单次向量 embedding？

社区摘要需把多个 chunk/实体的文本拼成 prompt 让 LLM 写摘要，输出 token 亦计入；而单向量 embedding 仅编码单向量输入，无生成阶段。

简化方案

在 GraphRAG 之前，按顺序尝试 simple alternative：

阶梯式增强

  1. 混合检索（混合检索接入）：条款号 + 语义
  2. 查询改写（查询改写与 HyDE）：口语 → 制度术语
  3. Parent/Child chunk：检索子块、展开父节上下文
  4. 轻量共现表：离线统计「同文档共现实体对」，查询时 lookup（无 LLM 构图）
  5. 全量 GraphRAG：上述仍无法达到 context_recall 目标时

轻量共现表（无 LLM 构图）

python
# 离线：统计同文档 / 同章内共现实体对
from collections import Counter
pairs = Counter()
for doc in corpus:
    entities = extract_entities_rule_based(doc)  # 制度编号、部门名
    for a, b in combinations(entities, 2):
        pairs[(a, b)] += 1
# 查询：「A 与 B 关系」先 lookup pairs，命中则拉取相关 chunk，未命中再向量

Parent/Child chunk 作为中间阶梯

检索子 chunk（精确），生成时展开父节（完整上下文）。对「制度 A 引用 B」类问法，常比全量 GraphRAG 便宜一个数量级，且与现有 分块策略 兼容。

跳过阶梯直接上图
  • 未证明 hybrid+改写不足
  • 无成本预算与回滚
  • 全库实体抽取拖慢索引发布
阶梯式验证
  • 每阶用黄金集测 recall 增量
  • 仅当 Δrecall ≥ 5% 进下一阶
  • GraphRAG 仅作「跨制度」类目插件

Spring 查询路由（vector / graph）

java
RetrievalResult route(String question, UserContext user) {
  Intent intent = intentRouter.classify(question);
  if (intent == Intent.CROSS_POLICY) {
    return graphRetriever.searchLocal(question, user.aclFilter());
  }
  if (intent == Intent.GLOBAL_THEME) {
    return graphRetriever.searchGlobal(question, user.aclFilter());
  }
  return hybridRetriever.search(question, user.aclFilter());
}
FAQ：简化方案仍不达标时如何决策？

导出失败用例聚类：若 > 60% 为跨文档多跳，批准 GraphRAG 试点；若分散在 OCR 质量、ACL、chunk 切分，先修数据管线。上图前须通过 RAG 评测闭环 证明简化阶梯已用尽且 ROI 可接受。

试点验收标准（GraphRAG 子集）

  • 类目：仅「跨制度引用」intent，约占总流量 8～12%
  • 指标：该类目 context_recall 从 0.68 → ≥ 0.80，faithfulness 不下降
  • 成本：索引期 token 在预算表 ±15% 内，查询 P95 +500ms 可接受
  • 回滚：关闭 graph 路由开关即可，向量路径不受影响

试点结论写入 eval report，作为是否扩大建图范围的唯一依据。

FAQ：Neo4j 与「向量 + 元数据」能否替代 GraphRAG？

若关系主要来自结构化字段（制度编号互引），用 metadata + BM25 往往足够；GraphRAG 价值在从正文自动抽关系 + 社区摘要。先穷尽 metadata 方案再评估上图 ROI。

CorpAssist 制度库约 70% 引用关系可从 PDF 目录与交叉引用表规则解析，宜作为 GraphRAG 之前的低成本阶梯。

规则解析结果应版本化入库，便于与后续 LLM 抽图结果 diff 审计。

本章结论

  • 何时需要：多跳关系与 global 摘要；单点 FAQ 不必 GraphRAG。
  • 成本：索引期抽取+社区摘要为主；按类目选择性建图。
  • 简化方案：hybrid → 改写 → parent chunk → 共现表 → 最后上图。
