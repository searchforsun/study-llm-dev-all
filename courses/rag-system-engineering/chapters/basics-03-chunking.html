为什么要学本章

检索单元是 chunk，不是整本 PDF。块太大塞不进上下文、太小丢语义；fixed chunk 一刀切常把一条制度拦腰截断。CorpAssist S1 要在「实现简单」与「召回完整」之间用可评测的方式选型，而不是凭感觉调 chunk_size。

学完你能
  • 为制度/手册选择 fixed / semantic / structure 分块组合
  • 配置 overlap 与 token window，并写入 chunk metadata
  • 设计 chunk experiment：同一评测集对比 recall@k
本章先记住 4 件事
  • 三种策略：固定长度最易落地；结构分块跟标题/条款；语义分块按相似度断句，成本高。
  • 重叠：相邻块共享 10%–20% 文本，缓解「答案跨块边界」问题。
  • 元数据：chunk 级 heading、clause_id、page 比纯文本更利于过滤与引用。
  • 实验：用固定问答基准对比策略，记录 chunk_policy_version，别 oral tuning。

固定/语义/结构

分块策略决定「检索最小单元」的粒度与语义完整性。三类策略可组合使用（例如 Word 先按 Heading 结构，超长节再固定切）。

策略做法优点风险
固定分块 按字符/token 窗口滑动切分 实现简单、可预测 断句、断条款
结构分块 按 H1/H2、PDF outline、制度「第 X 条」 块语义完整、便于引用 依赖 文档接入与解析 保留结构
语义分块 句向量相似度突变处切分 适应叙述性文本 计算贵、阈值难调
固定分块

按 token 窗口滑动；实现快，适合 OCR 段、无可靠结构的扫描件。

结构分块

按 H1/条款编号切；S1 制度库首选，引用可落到「第 X 条」。

语义分块

按句向量突变切分；适合叙述性 Wiki，索引成本较高需 A/B。

CorpAssist S1 · 制度 PDF（40 秒） 《员工手册》「试用期」一节跨两页，固定 512 token 切块可能把「考核标准」与「补偿说明」拆开；heading aware 结构分块可保持整节，再在超长节内 secondary 固定切。

策略选型：CorpAssist 推荐路径

不推荐
  • 全库统一 256 token，不看文档类型
  • 不保留 heading，引用只能显示「某 chunk#4821」
  • 语义分块全开，索引时间 ×5 无基准对比
  • 表格行被切成单 cell 块，丢失表头
推荐
  • 制度/手册：结构优先 + 超长节 fixed secondary
  • Wiki HTML：按 heading + main section
  • 扫描 OCR 段落：fixed + overlap，待质量提升再语义
  • 表格：整表或「表头+行组」为一块
先想 10 秒：FAQ 短问答库还需要结构分块吗？

若每条 Q/A 本身 < 窗口，可一条一 chunk（结构即条目）；不必再 fixed 切碎。策略应随文档类型配置，而非全局常量。

重叠

Overlap 指相邻 chunk 共享一段文本，避免关键句落在块边界外。与 token window 配合：窗口 512、overlap 64 是常见起点（需用评测集验证）。

有无 overlap 的差异（示意）

固定分块 · 边界问题 vs overlap
flowchart LR
  subgraph noOverlap["无 overlap"]
    A1["块1: …考核标准为"]
    A2["块2: 绩效不达标…补偿…"]
  end
  subgraph withOverlap["有 overlap"]
    B1["块1: …考核标准为绩效不达标"]

用户问「不达标补偿」时，无 overlap 可能只命中块 2 的前半句；overlap 让两块都含关键衔接词。

参数建议（教学示意，非压测结论）

文档类型window（token）overlap备注
制度 PDF（结构分块后）384–51210%–15%节内 secondary 切
OCR 扫描件256–38415%–20%断句不稳定，略增 overlap
Wiki 短文整节0一节一 chunk 即可
注意
  • overlap 过大 → 索引体积膨胀、重复命中；应用 benchmark 看 recall 与索引大小 trade-off。
  • overlap 不能替代结构分块；边界在条款中间时，应优先调整切分逻辑。

元数据

每个 chunk 除 text 外应携带 chunk metadata，供检索过滤、引用展示与评测归因。

推荐 meta 字段

字段用途示例
source引用链接/文件名hr-handbook-2024.pdf
page页码引用12
heading章节标题试用期管理
clause_id制度条款编号第 3.2 条
chunk_index同文档内序号5 / 28
chunk_policy分块策略版本structure-v2+overlap-64

chunk payload 示例

json 
{
  "text": "试用期考核不合格，依法无需支付经济补偿…",
  "meta": {
    "source": "hr-handbook-2024.pdf",
    "page": 12,
    "heading": "试用期管理",
    "clause_id": "3.2",
    "chunk_index": 5,
    "chunk_policy": "structure-v2_overlap-64"
  }
}
元数据不足
text
引用：[chunk a3f9c2] 无法打开原文
可审计引用
text
引用：《员工手册 2024》§3.2 试用期管理，p.12

实验对比

Chunk experiment：在同一批文档与同一评测问答集上，对比不同分块策略的检索指标，避免「感觉变好了」。

实验设计最小集

  1. 固定变量：Embedding 模型、TopK、查询集（≥30 条 S1 真实问题）。
  2. 自变量：fixed-only vs structure+overlap vs +semantic。
  3. 指标：recall@5、MRR、引用可点击率；记录索引条数与入库耗时。
  4. 决策：选 Pareto 最优；写入 chunk_policy_version 并 re-index。

制度 PDF 里嵌套表格，分块时整块还是按行？
优先整表或「表头 + 行组」为一块，避免单 cell 丢失列语义。若表超 token 窗口，按行组切并重复表头行到每个子块 meta（table_id、row_range）。
chunk_policy_version 变更后，Spring 查询侧要做什么？
查询侧通常无需改代码，但需确认 alias 已切到新索引；评测集应重跑 recall@k 与引用点击率，作为发布门禁（衔接 RAG 评测闭环）。
S1 分块决策口诀
  • 制度类 → 结构优先，fixed 作 secondary；Wiki 短文 → 一节一 chunk。
  • overlap 用 benchmark 定，不凭感觉；改 policy 必记 chunk_policy_version。
  • 引用靠 meta（heading/clause/page），不靠 chunk 哈希。
  • 实验结论写入 chunk-compare.md，与 corpassist s1 基准集版本号一并归档。

分块策略是 RAG 性价比最高的调参杠杆之一：在 CorpAssist S1 中，structure+overlap 往往比换更大 Embedding 模型带来更稳的 recall 提升。

上线前务必在 Demo chunk-compare.md 留档实验编号，便于与后续 Embedding 与索引 变更做联合回归。

本章结论

每条对应上文一个小节。

  • 固定/语义/结构：S1 制度库优先结构分块；fixed 作 fallback；语义分块需实验 justification。
  • 重叠：10%–20% overlap 缓解边界丢失；与过大索引体积权衡。
  • 元数据：heading/clause/page/source 支撑 ACL 过滤与用户可见引用。
  • 实验对比：固定评测集 + recall@k 选 policy；版本化 chunk_policy 便于回归。

动手练习

针对 CorpAssist 制度 PDF，完成策略选择、overlap 参数与实验记录（约 25 分钟）。

操作步骤

  1. 阅读 demos/basics-03-chunking-lab/chunk-compare.md，为「制度 PDF」选定一种主策略 + 一种 fallback，各写一句理由。
  2. 假设 window=512 token，为 OCR 扫描件与 Word 制度分别给出 overlap 建议值及原因。
  3. 设计 chunk meta schema（至少 5 个字段），说明每个字段如何支撑前端引用展示。
  4. 列出 chunk A/B 实验的 3 个固定变量与 2 个对比指标（含 recall@k）。

判断练习

  1. recall@5 从 0.62 提到 0.78，索引条数 ×2.1。产品问「是否上线 structure+overlap？」你会补充哪一个最关键决策维度？

    参考答案

    除 recall 外，必须看成本与 SLA：存储、检索延迟、索引 job 时长是否在预算内；若 ×2 条数导致 P99 检索超时，需调 TopK/Rerank 或略减 overlap，而不是只看 recall 数字。

  2. 找错：哪条不能代替 overlap？① 结构分块保持整节 ② 增大 TopK ③ 在块边界增加 64 token 重叠

    参考答案

    答：②。增大 TopK 可能多捞到相邻块，但增加噪声与 token 成本，不解决「单块边界切在关键词中间」的根因；①③ 是正确手段。
