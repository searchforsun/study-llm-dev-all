本章先记住 3 件事
  • RAG 解决什么：用外部知识补全 LLM 上下文，降低幻觉并支持文档更新，而不是靠微调背制度。
  • Naive → Advanced：先跑通「检索+生成」，再按需加改写、重排、混合检索与拒答（详见 检索、重排与生成）。
  • 双栈分工：Python 负责离线 索引管线；Spring AI 负责在线 查询与引用，共用同一 collection 与 ACL 元数据。

Naive vs Advanced RAG

检索增强生成（RAG） 的核心动作：把用户问题变成检索请求，从知识库取回相关片段，拼进 Prompt 再交给 LLM 生成。工程上常分两层理解：

  • Naive RAG — 问题直接向量化 → TopK 相似块 → 一次生成。实现快、适合 PoC，但对口语化问题、表格 PDF、权限过滤往往不够。
  • Advanced RAG — 在检索前后加「预处理与后处理」：查询改写、混合检索、Rerank、引用约束、拒答与评测闭环。CorpAssist S1 上线目标通常落在此层。
CorpAssist S1 · 制度问答（45 秒） HR 同事问：「试用期考核不过，补偿怎么算？」知识库里有 2024 版《员工手册》PDF 与 2023 年旧版 Word。Naive 只取向量最近的一块，可能命中旧版或半句话；Advanced 会结合元数据过滤、重排与「必须带引用」的生成策略。

模式对比：何时停在 Naive，何时必须 Advanced

维度Naive RAGAdvanced RAG
典型链路 Embed 问题 → 向量 TopK → LLM 改写/路由 → 混合检索 → Rerank → 打包上下文 → 带引用生成
CorpAssist 阶段 Demo、内部试点 S1 生产、需 ACL 与评测门禁
主要风险 检索不到、答非所问、幻觉 复杂度高、延迟与成本上升（需度量）
Naive 常见踩坑
  • 脏 PDF 不清洗就入库，检索全是页眉页脚
  • 检索层不做 ACL，越权片段进入 Prompt
  • 纯向量忽略制度编号等关键词
  • 无引用，用户无法核对来源
Advanced 最小增量
  • 接入与分块质量先行（文档接入与解析、分块策略）
  • 检索带 dept / doc_version 过滤
  • TopK 扩大 + Rerank 收窄（检索、重排与生成）
  • 生成 Prompt 要求「仅依据上下文 + 列出引用」
先想 10 秒：CorpAssist 只有 2 周做 MVP，你会先 Naive 还是直接 Advanced？

合理路径：Naive 跑通双栈与评测集，同时把接入/分块/ACL 设计按 Advanced 预留；别在 Naive 里堆 Rerank+GraphRAG，否则排障困难。2 周后用基准集决定加哪一环。

组件图

把 RAG 拆成可替换组件，便于双栈对齐与面试白板。CorpAssist S1 常用划分如下：

Ingestion / Indexer

离线：Reader → 清洗 → 分块 → Embedding → 写入向量库（Python 为主）。

Retriever

在线：把问题（及改写结果）映射为检索请求，带 ACL 与 TopK 策略。

Generator

在线：组装 Prompt、调用 LLM、流式输出与引用格式化（Spring ChatClient + Advisor）。

常见误区
  • 把 Retriever 当 Generator — 检索只负责「找材料」，不能代替 Prompt 约束与引用格式。
  • 组件图画成单进程 — 生产上索引与查询往往不同服务、不同发布节奏。
  • 忽略 meta 契约 — Python 写入的 dept、source 字段，Spring 侧 filter 必须同名。

CorpAssist 管线

CorpAssist S1 企业知识库的主线：制度与手册可检索、可引用、可审计。Dual stack rag 在此的具体含义：

  • Python 索引栈 — 定时/事件触发：扫描对象存储 → Reader → 清洗 → 分块 → Embedding → upsert 向量库，写入 model_version 与 ACL 字段。
  • Spring 查询栈 — BFF 或 RAG API：鉴权 → 注入用户租户/部门 → Retriever 带 filter → Generator 流式输出 + SSE 引用。
  • 共用索引 — 两边读同一 collection；索引变更走 alias 切换（详见 Embedding 与索引）。

双栈分工：标签页对照

职责：批处理、重跑、re-embed；适合 LangChain LCEL 编排。

产出：向量 + {source, page, dept, doc_version} payload。

注意：Embedding 模型变更必须全量或增量 re-embed，并 bump model_version。

职责：低延迟问答、会话、流式 SSE；QuestionAnswerAdvisor 注入检索结果。

输入：用户 JWT 中的 tenant/dept，转成 VectorStore metadata filter。

注意：不在 Spring 里做重 OCR；脏数据应回退到索引任务修复。

索引 meta 契约（示意）

Python 写入 payload
json 
{
  "text": "试用期考核不合格，依法无需支付经济补偿…",
  "meta": {
    "source": "hr-handbook-2024.pdf",
    "page": 12,
    "dept": "HR",
    "doc_version": "2024.06",
    "acl": ["all-staff"]
  }
}
Spring 检索 filter（概念）
java 
// 伪代码：用户部门 + 文档版本策略
SearchRequest.builder()
  .query(userQuestion)
  .topK(8)
  .filterExpression("dept in ['HR','all'] && doc_version >= '2024'")
  .build();
为什么不把索引也写在 Spring 里，少维护一套 Python？
企业制度批量入库、OCR、大文件解析在 Python 生态工具更成熟；Spring 侧聚焦低延迟 API 与权限。双栈增加对齐成本，但换来离线吞吐与在线 SLA的分工清晰。practice 阶段会做 parity 对照。
Advanced RAG 是否等于一定要上 GraphRAG？
否。S1 多数场景「混合检索 + Rerank + 引用 + 拒答」已足够；GraphRAG 在 advanced 阶段单独讲边界与成本。

本章结论

每条对应上文一个小节，便于串起全章再做题。

  • Naive vs Advanced：Naive 适合 PoC；S1 生产需在检索前后加 ACL、重排、引用与评测，而不是一次性堆复杂模块。
  • 组件图：离线 Indexer（Reader→Chunk→Embed）与在线 Retriever+Generator 分离；向量库是双栈唯一共享锚点。
  • CorpAssist 管线：Python 写索引、Spring 答问题；meta 契约（source/page/dept/version）两端一致，才能安全引用。

判断练习

  1. 同事说：「我们把 OCR、分块、Embedding、问答全放进 Spring Boot，就不用 Python 了。」请判断主要问题是什么。

    参考答案

    问题在于混淆离线吞吐与在线 SLA。批量 OCR、大 PDF 解析、全量 re-embed 适合异步索引任务；全塞进 Spring 会导致发布耦合、内存与超时风险，且工具链不如 Python 成熟。双栈是为分工，不是为「多一门语言而多一门」。

  2. 找错：哪条违反组件边界？① Retriever 返回 TopK 文本 ② Generator 要求仅依据上下文并附引用 ③ Generator 内部再次 OCR PDF

    参考答案

    答：③。OCR 属于接入/索引阶段（Reader），不应在在线生成路径重复做；①② 是正确分工。
