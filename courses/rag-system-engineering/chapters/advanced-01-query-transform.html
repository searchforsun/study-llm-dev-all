为什么要学本章

用户口语「年假咋请」与制度原文「年假申请流程」embedding 空间距离远，混合检索接入 仍可能 miss。查询侧 query rewrite 在检索前扩展或改写问句，提升 recall boost；但每多一次 LLM 调用都有 transform cost，必须量化对比。

学完你能
  • 实现 multi query 与结果融合
  • 配置 HyDE（假设性文档嵌入）链路
  • 用 RAG 评测闭环 对比改写前后 context_recall
  • 在 CorpAssist 双栈中选择 Python 改写或 Spring 网关改写
本章先记住 3 件事
  • 改写为检索服务：生成答案不是 HyDE 的目标，假文档只为 embedding。
  • Multi-query 要融合：多路检索后 RRF 去重，避免 Prompt 爆炸。
  • 成本与收益同测：latency 与 token 成本写入 eval report。

Multi-query

CorpAssist S1 · 口语问法 Demo 样例：原问「年假咋请」→ 改写「年假申请流程 制度」。一条用户问句生成 3～5 条 sub query，分别检索再融合。

Multi-query 检索
flowchart LR
  Q[用户问句] --> R[LLM 改写 N 条]
  R --> V1[检索 1]
  R --> V2[检索 2]
  R --> V3[检索 3]
  V1 --> F[RRF 融合]
  V2 --> F
  V3 --> F
  F --> C[context pack]

query expansion 适合短问、口语、多义词；专有编号问法（「制度 HR-2024-03 第 2 条」）通常不必改写，以免漂移。

改写 Prompt 约束（CorpAssist S1）

markdown
你是 CorpAssist HR 制度检索助手。将用户问句改写为 3 条检索子问句。
规则：
1. 保留原意，不编造制度编号或金额
2. 输出 JSON 数组，如 ["年假申请流程", "带薪年休假制度", "休假审批步骤"]
3. 每条 8～20 字，偏制度术语而非口语

融合配置：RRF 与去重

参数推荐值说明
sub_query 数量3～5过多导致 transform cost 线性上升
每路 top_k8单路不宜过大，融合后再 rerank
融合算法RRF k=60与 混合检索接入 一致，尺度无关
去重键chunk_id同 chunk 多路命中只保留最高分

双栈职责划分

双栈各写一套改写
  • Python 与 Spring 改写 Prompt 不一致
  • 评测无法对比 parity
  • 生产 A/B 结论不可复现
统一改写服务
  • Python 提供 /v1/query/expand 或共享 Prompt 文件
  • Spring 网关调用同一 endpoint
  • parity test 黄金集校验输出一致
FAQ：Multi-query 会放大 ACL 风险吗？

会。每路子问句检索都必须带相同 acl filter；禁止为「提高召回」在某路子问句去掉 dept 过滤。融合前对每路 hits 做权限校验，与 RAG 安全 一致。

HyDE

hyde（Hypothetical Document Embeddings）：让 LLM 先写一段假设性答案/段落，用该段落的 embedding 去向量库检索，而非用原始短问句。

HyDE 链路

python
def hyde_retrieve(question: str) -> list[Document]:
    hypothetical = llm.invoke(HYDE_PROMPT.format(q=question))
    query_vector = embedder.embed_query(hypothetical)
    return vector_store.similarity_search_by_vector(query_vector, k=8)
HyDE 误用
  • 把假设文档直接当最终答案
  • 不做 ACL filter
  • 合规场景未标注「非引用」
HyDE 正用
  • 假文档仅用于 embedding
  • 检索后仍走 citation + faithfulness
  • 与 multi-query 二选一或分层
先想 10 秒：HyDE 在「条款编号精确查询」上通常有效还是有害？

通常有害或无益。编号问法应走 BM25 / keyword boost（见混合检索章）；HyDE 可能编造不存在的条款表述，embedding 反而偏离真实文档。

HyDE Prompt 与假文档边界

python
HYDE_PROMPT = """根据问题写一段 80～120 字的假设性制度说明段落。
要求：语气像正式制度条文，但不要编造具体编号、金额、日期。
问题：{q}
段落："""

# 假文档仅用于 embed_query；禁止直接进入生成 Prompt 作为「权威答案」
hypothetical = llm.invoke(HYDE_PROMPT.format(q=question))
assert "HR-" not in hypothetical or audit_flag("hyde-invented-id")

Spring AI：HyDE 检索顾问（示意）

java
// 查询链：hydeExpand → embed → similaritySearch → 标准 RAG 生成
String hypothetical = hydeClient.expand(userQuestion);
float[] vec = embeddingModel.embed(hypothetical);
List<Document> docs = vectorStore.similaritySearch(
    SearchRequest.builder().query("").embedding(vec).topK(8)
        .filterExpression(aclFilter.forUser(user)).build());
// 生成阶段仍用原始 userQuestion + docs，不用假文档作答
HyDE 与生成隔离
flowchart LR
  Q[用户问句] --> H[LLM 写假文档]
  H --> E[embed 假文档]
  E --> V[向量检索]
  V --> C[真实 chunk]
  Q --> G[生成 Prompt]
  C --> G
  G --> A[带 citation 答案]

假文档只参与检索向量；最终答案必须 grounded 于检索到的真实 chunk。

效果对比

是否启用改写，必须用同一黄金集对比（见 rag benchmark）。记录指标 + 成本：

策略context_recallP95 延迟额外 token建议
baseline（无改写）0.801.2s0默认
multi-query ×30.872.1s~400口语 FAQ 开启
HyDE0.851.8s~250抽象问法试用

决策：何时启用改写

  • context_recall 提升 ≥ 5% 且 P95 可接受 → 生产开启
  • 仅个别类目提升 → 按 intent router 路由到改写链
  • 改写后 faithfulness 下降 → 检查假文档是否污染生成 Prompt

rewrite prompt 应约束：不编造事实、输出 JSON 数组、保留用户原意。示例见 demos/advanced-01-query-transform-lab/rewrite-example.txt。

A/B 实验设计（iter-loop 对齐）

每次实验只改一个改写策略，其余参数（chunk、hybrid 权重、rerank）固定。记录到 eval report：

  1. baseline：无改写，记录 faithfulness / context_recall / P95
  2. variant-mq：仅开启 multi-query ×3
  3. variant-hyde：仅开启 HyDE
  4. variant-route：按 intent 路由（口语→mq，抽象→hyde）

路由配置（按 intent 启用）

yaml
# demos/advanced-01-query-transform-lab/query-transform-routing.yaml
transform:
  default: none
  routes:
    - intent: faq_colloquial   # 「年假咋请」
      strategy: multi_query
      max_sub_queries: 3
    - intent: definition       # 「什么是竞业限制」
      strategy: hyde
    - intent: exact_clause     # 「HR-2024-03 第 2 条」
      strategy: none           # 走 BM25，见混合检索

FAQ：改写后 citation 消失怎么办？

常见原因是 context pack 膨胀或假文档污染生成 Prompt。修复：融合后 strict top_k（如 4）；生成 system 声明「仅引用检索 chunk」；对 HyDE 路径禁止把 hypothetical 拼入 user message。回归测 citation 字段非空率。

本章结论

  • Multi-query：LLM 生成 sub-query + RRF 融合；适合口语短问。
  • HyDE：假设文档 embedding 检索；假文档不作最终答案。
  • 效果对比：recall/latency/token 三维度；路由级启用而非全开。
