缓存

CorpAssist S1 · 高峰 FAQ 周一上午 2000 次「年假怎么请」——语义相近但字面不同。精确匹配 cache miss；semantic cache 用 query embedding 在 Redis 中找 cosine > 0.92 的历史问答，命中则跳过检索与 LLM。

缓存键设计（Spring 查询侧）

java
// cache key = hash(kb_id + user_dept + query_embedding_bucket)
String cacheKey = CacheKeys.semantic(user.getDept(), kbId, queryVector);
Optional<CachedAnswer> hit = semanticCache.lookup(cacheKey, queryVector, 0.92);
if (hit.isPresent()) return hit.get().withSource("semantic-cache");

Python 侧：embedding 结果缓存

python
# 索引 job 与在线 query 共用 embed cache，降低重复 chunk 费用
@lru_cache(maxsize=50_000)
def embed_text(text_hash: str) -> list[float]:
    return embedder.embed_query(load_text(text_hash))

制度更新、ACL 变更、embedding 模型切换时必须 invalidate 对应 kb 分区；否则 cost optimize 会以旧答案为代价。见 demos/advanced-04-performance-lab/perf-tactics.md。

小模型路由

small model route 在进入向量库之前用轻量分类器或规则判断：若为高频 FAQ、无多跳、无合规歧义，则用小模型（或模板）直接答；否则走完整 RAG。CorpAssist S1 典型路由信号：

  • intent = faq_entry（入口类）且 confidence > 0.85
  • 无跨库/跨部门关键词（「财务+HR 冲突」→ 禁走小模型）
  • 用户未要求引用条款编号
S1 性能路由
flowchart TD
  Q[用户问句] --> SC{semantic cache}
  SC -->|命中| OUT[返回答案]
  SC -->|未命中| RT{small model route}
  RT -->|FAQ| SM[小模型/模板]
  RT -->|复杂| RAG[Embed+检索+大模型]
  SM --> OUT
  RAG --> OUT
  OUT --> W[写 cache + metrics]
路由误用
  • 所有问法默认小模型
  • 无 citation 的合规 FAQ
  • 路由决策不可观测
路由正用
  • 仅 hot intent 白名单
  • 失败/低置信回退 RAG
  • 路由标签进 audit/metrics
先想 10 秒：semantic cache 为何必须把 user_dept 放进 cache key？

不同部门可见文档集不同；若只按 query 相似度命中，可能返回其他部门可见、本用户不可见的缓存答案，构成隐性 acl leak。键必须绑定 ACL scope。

预计算

precompute embed 把索引期与查询期的 embedding 成本前移：热门 chunk、高频改写 query 在离线 job 中算好向量并 pin 到内存/SSD；在线只查表。与 hot query 分析联动——从 access log 取 Top-N 问法， nightly 跑检索 + 答案缓存预热。

手段P95 延迟Token 节省风险
semantic-cache↓ 60～80%↓ 100%（命中时）制度变更未失效
small-model-route↓ 40%↓ 70%（FAQ 子集）误路由无引用
precompute-embed↓ 15～25%↓ 10%（embed API）热点漂移需重算

hot query 预热 job（Python）

python
hot = access_log.top_queries(kb="corp-hr", days=7, n=500)
for q in hot:
    vec = embedder.embed_query(q.text)
    semantic_cache.warm(key=scope_key(q.user_dept), query_vec=vec, answer=rag_pipeline(q))

所有优化需进 eval report：对比开启前后 faithfulness、context_recall，防止「快了但答错」。

本章结论

  • 缓存：semantic-cache + ACL scope；制度/模型变更必 invalidate。
  • 小模型路由：FAQ 白名单 + 低置信回退完整 RAG。
  • 预计算：precompute-embed 与 hot-query 预热；收益用 latency/token 量化。
