缓存

CorpAssist S1 · 高峰 FAQ 周一上午 2000 次「年假怎么请」——语义相近但字面不同。精确匹配 cache miss；semantic cache 用 query embedding 在 Redis 中找 cosine > 0.92 的历史问答，命中则跳过检索与 LLM。

缓存键设计（Spring 查询侧）

java
// cache key = hash(kb_id + user_dept + query_embedding_bucket)
String cacheKey = CacheKeys.semantic(user.getDept(), kbId, queryVector);
Optional<CachedAnswer> hit = semanticCache.lookup(cacheKey, queryVector, 0.92);
if (hit.isPresent()) return hit.get().withSource("semantic-cache");

Python 侧：embedding 结果缓存

python
# 索引 job 与在线 query 共用 embed cache，降低重复 chunk 费用
@lru_cache(maxsize=50_000)
def embed_text(text_hash: str) -> list[float]:
    return embedder.embed_query(load_text(text_hash))

制度更新、ACL 变更、embedding 模型切换时必须 invalidate 对应 kb 分区；否则 cost optimize 会以旧答案为代价。见 demos/advanced-04-performance-lab/perf-tactics.md。

先想 10 秒：kb_version 发布时为何要同时 invalidate L1 与 L2？

L1 精确 hash 可能仍命中旧问句的旧答案；L2 semantic cache 按 embedding 相似度命中，制度条款变更后问法相近仍会返回过期内容。版本号变更应清空该 kb 下全部缓存层。

缓存分层架构

CorpAssist S1 三级缓存
flowchart TB
  Q[query] --> L1[L1 精确 hash cache]
  L1 -->|miss| L2[L2 semantic cache]
  L2 -->|miss| L3[L3 embed cache]
  L3 --> RAG[完整 RAG]
  RAG --> W[回写 L2/L1]

L1 适合完全相同问句；L2 适合语义相近；L3 降低重复 embed API 调用。

invalidate 触发清单

事件失效范围实现
kb_version 发布该 kb 全部 L1/L2cache invalidate by prefix
单文档修订含该 doc_id 的缓存项倒排索引 cache tag
ACL 策略变更受影响 dept 分区重建 cache key schema
embedding 模型切换全库 L2/L3与 re-embed 同步

yaml
corpassist:

FAQ：semantic cache 与 查询改写与 HyDE 如何配合？

改写增加首包延迟；可对改写后的 canonical query 做 L2 缓存（存 embedding + 答案），避免同一口语问法重复走 LLM 改写。注意 canonical 仍须含 ACL scope。

embed cache 与索引 job 协同

Python 批处理索引时，embed cache 以 text_hash 为键，避免同一 chunk 在重跑 job 时重复调用 Embedding API。Spring 在线查询只读已索引向量，不重复 embed 文档正文；用户问句 embedding 可走独立的小缓存分区，TTL 短于答案缓存。

  • 索引 job 结束导出 embed_cache 命中率，目标 ≥ 40%（重复 ingest 场景）
  • 模型版本变更时清空 embed_cache，与 re-embed 同批次执行
  • 禁止在线服务写 embed_cache 污染批处理键空间（分区隔离）

小模型路由

small model route 在进入向量库之前用轻量分类器或规则判断：若为高频 FAQ、无多跳、无合规歧义，则用小模型（或模板）直接答；否则走完整 RAG。CorpAssist S1 典型路由信号：

  • intent = faq_entry（入口类）且 confidence > 0.85
  • 无跨库/跨部门关键词（「财务+HR 冲突」→ 禁走小模型）
  • 用户未要求引用条款编号
S1 性能路由
flowchart TD
  Q[用户问句] --> SC{semantic cache}
  SC -->|命中| OUT[返回答案]
  SC -->|未命中| RT{small model route}
  RT -->|FAQ| SM[小模型/模板]
  RT -->|复杂| RAG[Embed+检索+大模型]
  SM --> OUT
  RAG --> OUT
  OUT --> W[写 cache + metrics]
路由误用
  • 所有问法默认小模型
  • 无 citation 的合规 FAQ
  • 路由决策不可观测
路由正用
  • 仅 hot intent 白名单
  • 失败/低置信回退 RAG
  • 路由标签进 audit/metrics
先想 10 秒：semantic cache 为何必须把 user_dept 放进 cache key？

不同部门可见文档集不同；若只按 query 相似度命中，可能返回其他部门可见、本用户不可见的缓存答案，构成隐性 RAG 安全 风险。键必须绑定 ACL scope。

路由分类器实现选项

方案延迟准确率CorpAssist 建议
规则 + 关键词< 5ms中入口 FAQ 白名单首选
小分类模型~50ms高意图边界模糊时
大模型判断~500ms高仅离线标注，不进热路径

小模型路由配置

yaml
route:
  small_model:
    model: gpt-4o-mini
    allowed_intents: [faq_entry, portal_link]
    min_confidence: 0.85
    fallback: rag-full
    require_citation: false   # 入口类可模板；制度条款类必须 rag-full

回退条件（必须实现）

  1. 分类 confidence < 阈值 → rag-full
  2. 用户消息含「引用」「条款」「第 X 条」→ 强制 rag-full
  3. 多库/跨部门关键词命中 → 强制 rag-full
  4. 小模型输出触发 refusal → 回退并记录 metric

预计算

precompute embed 把索引期与查询期的 embedding 成本前移：热门 chunk、高频改写 query 在离线 job 中算好向量并 pin 到内存/SSD；在线只查表。与 hot query 分析联动——从 access log 取 Top-N 问法， nightly 跑检索 + 答案缓存预热。

手段P95 延迟Token 节省风险
semantic-cache↓ 60～80%↓ 100%（命中时）制度变更未失效
small-model-route↓ 40%↓ 70%（FAQ 子集）误路由无引用
precompute-embed↓ 15～25%↓ 10%（embed API）热点漂移需重算
先想 10 秒：precompute-embed 与 semantic-cache 节省的成本类型有何不同？

precompute 主要省在线 embed API与 ANN 冷启动；semantic-cache 省检索 + 大模型生成。两者可叠加，但热点漂移时 precompute 需 nightly 重算，cache 需 TTL/版本失效。

hot query 预热 job（Python）

python
hot = access_log.top_queries(kb="corp-hr", days=7, n=500)
for q in hot:
    vec = embedder.embed_query(q.text)
    semantic_cache.warm(key=scope_key(q.user_dept), query_vec=vec, answer=rag_pipeline(q))

所有优化需进 RAG 评测闭环：对比开启前后 faithfulness、context_recall，防止「快了但答错」。

预计算流水线（Python 索引侧）

python
# nightly job：热门 chunk + 热门 query 预计算
hot_chunks = access_log.top_chunks(kb="corp-hr", days=30, n=2000)
for c in hot_chunks:
    if not embed_cache.has(c.id):
        embed_cache.put(c.id, embedder.embed(c.text))

hot_queries = access_log.top_queries(kb="corp-hr", days=7, n=500)
for q in hot_queries:
    precompute_retrieval_cache(q, acl_scope="ALL")  # 不含用户级答案

FAQ：预计算与 semantic cache 区别？

precompute embed 解决索引/向量计算重复；semantic cache 缓存端到端问答结果。前者不改变答案逻辑；后者命中则跳过检索与生成，失效策略更严格。

S1 高峰场景组合推荐

CorpAssist 周一上午 FAQ 高峰推荐顺序：semantic-cache → small-model-route（入口类）→ precompute-embed。改写链（advanced-01）在 cache miss 后启用，避免热路径叠加 LLM 调用。目标：P95 ≤ 2.0s，token 成本较纯 RAG 降 35～45%（见 perf-tactics.md 样例）。

流量占比策略预期 P95
~30% 重复 FAQL2 semantic cache< 300ms
~15% 入口导航small-model-route< 800ms
其余完整 RAG< 2.5s
FAQ：成本优化后 faithfulness 略降是否可接受？

不可接受若降幅 > 2% 或影响合规 FAQ。可接受若仅入口类 route 略降且已标注无 citation。一切以 eval 黄金集为准，不以线上「体感」放行。

建议为 cache hit / miss 各抽样 50 条做 faithfulness 对比，写入 weekly cost-optimize 报告。

报告与 publish check 绑定，未附报告不得开启下一项优化。

本章结论

  • 缓存：semantic-cache + ACL scope；制度/模型变更必 invalidate。
  • 小模型路由：FAQ 白名单 + 低置信回退完整 RAG。
  • 预计算：precompute-embed 与 hot-query 预热；收益用 latency/token 量化。
