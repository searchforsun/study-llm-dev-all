smoke 脚本（curl）

bash
curl -s -X POST http://localhost:8080/v1/rag/ask \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \

# 双栈 parity：同一问句打 Python FastAPI 与 Spring，diff sources 应为空
diff <(curl -s ...python:8000...) <(curl -s ...spring:8080...)

parity test 是联调第 6 步：索引版本一致前提下，双栈返回的 doc_id 集合差异率应 < 1%（允许 rerank 微小浮动）。

指标

capstone rag 上线不以「能跑」为准，而以可观测指标为准。S1 典型闸门（可与 RAG 评测闭环 对齐）：

指标阈值（示例）采集点
faithfulness≥ 0.85eval 黄金集
context_recall≥ 0.80eval 黄金集
P95 latency≤ 2.5sAPI gateway
acl_leak 用例0 失败安全测试集

灰度期对比 v11/v12：指标 regression 自动阻断 dual stack launch。性能优化（cache/route）单独 AB，避免与索引版本混淆。

    action: block_traffic
FAQ：eval 通过但生产 faithfulness 偏低，先查什么？

① alias 是否指向预期 kb_version；② 生产 filter 与 eval ACL 是否一致；③ 是否开启 cache 返回旧答案；④ embedding 模型版本是否与索引一致。用 integration test 对齐环境。

文档

交付物三件套：OpenAPI（契约）、runbook（运维）、delivery-checklist（验收）。双栈团队 handoff 靠文档对齐 Python 与 Spring 的 kb_version、filter 字段、SSE 事件格式。

runbook 目录（示意）

markdown
## CorpAssist S1 RAG Runbook
1. 索引失败：查 Python job log → 重跑 checkpoint
2. 召回下降：确认 alias / embedding 版本
3. 越权嫌疑：导出 audit-log trace_id 复核 filter
4. 回滚：alias 指上一 kb_version，invalidate semantic-cache

OpenAPI 应含 sources[]、kb_version、流式 SSE 事件类型；与 检索、重排与生成 及 practice 章 POST /v1/rag/ask 契约一致（SSE 事件可保留 citations 别名，JSON 响应用 sources）。

S1 面试答题框架

s1 interview 系统设计题推荐五段式（见 demos/advanced-06-capstone-lab/s1-interview-outline.md）：

  1. 接入与分块：PDF 清洗、制度友好分块、metadata
  2. 索引与 ACL：Python 批处理、doc-acl、kb_version
  3. 检索 + 重排：混合检索、rerank、security-filter
  4. 生成 + 引用：citation、拒答、faithfulness
  5. 评测与迭代：黄金集、release gate、cache/route 优化
面试减分答法
  • 只讲向量检索，无 ACL/评测
  • Python/Spring 职责混乱
  • 无指标与回滚方案
S1 加分答法
  • 双栈边界清晰 + 共用 VectorStore
  • 安全/性能/治理各 1 句
  • 主动提 runbook 与 audit
先想 10 秒：面试官问「为何不用微调替代 RAG？」如何结合 S1 答？

制度频繁修订、需 citation 与 ACL；RAG 可热更新索引且答案可追溯 doc/chunk。微调难保证引用正确与部门隔离，且 re-train 成本高。S1 选 RAG+引用+ACL，微调仅作可选 rerank/意图小模型。

高频追问与答法要点

markdown
Q: 双栈为何 Python 写 Spring 读？
A: 索引重、依赖多，Python 生态适合 ETL；Spring 承接企业鉴权与 API 契约。

Q: 如何防越权？
A: doc-acl metadata + 检索层 security-filter，不靠 Prompt 保密。

Q: 上线标准？
A: faithfulness/recall/latency/ACL 四闸门 + 灰度 + runbook。

Q: 失败怎么回滚？
A: alias 指旧 kb_version + invalidate cache + audit 追溯。

SLA 演示要点（sla demo）

面试现场可演示：① smoke curl 返回 citation；② 跨 dept 问法被拒；③ Grafana 看 P95 与 cache hit_rate。体现 dual stack launch 可观测，而非 PPT 架构。

FAQ：S1 面试要先讲 GraphRAG 吗？

除非题目明确要求多跳/global 摘要，否则先讲通 hybrid + ACL + eval 主链路，GraphRAG 作为「recall 触顶后的可选增强」。过度强调 GraphRAG 易暴露成本与治理短板。

  • 演示环境须与生产配置同构（alias、ACL YAML、hybrid 权重）

FAQ：灰度 48h 观察期要看哪些曲线？

faithfulness 小时均值、P95、cache hit_rate、acl_leak 探测、error_rate。任一较金丝雀前基线恶化超 5% 即暂停放量并比对 kb_version / filter 配置 diff。
