HyDE

hyde（Hypothetical Document Embeddings）：让 LLM 先写一段假设性答案/段落，用该段落的 embedding 去向量库检索，而非用原始短问句。

HyDE 链路

python
def hyde_retrieve(question: str) -> list[Document]:
    hypothetical = llm.invoke(HYDE_PROMPT.format(q=question))
    query_vector = embedder.embed_query(hypothetical)
    return vector_store.similarity_search_by_vector(query_vector, k=8)
HyDE 误用
  • 把假设文档直接当最终答案
  • 不做 ACL filter
  • 合规场景未标注「非引用」
HyDE 正用
  • 假文档仅用于 embedding
  • 检索后仍走 citation + faithfulness
  • 与 multi-query 二选一或分层
先想 10 秒：HyDE 在「条款编号精确查询」上通常有效还是有害？

通常有害或无益。编号问法应走 BM25 / keyword boost（见混合检索章）；HyDE 可能编造不存在的条款表述，embedding 反而偏离真实文档。

效果对比

是否启用改写，必须用同一黄金集对比（见 rag benchmark）。记录指标 + 成本：

决策：何时启用改写

  • context_recall 提升 ≥ 5% 且 P95 可接受 → 生产开启
  • 仅个别类目提升 → 按 intent router 路由到改写链
  • 改写后 faithfulness 下降 → 检查假文档是否污染生成 Prompt

rewrite prompt 应约束：不编造事实、输出 JSON 数组、保留用户原意。示例见 demos/advanced-01-query-transform-lab/rewrite-example.txt。

本章结论

  • Multi-query：LLM 生成 sub-query + RRF 融合；适合口语短问。
  • HyDE：假设文档 embedding 检索；假文档不作最终答案。
  • 效果对比：recall/latency/token 三维度；路由级启用而非全开。

成本

  3. 查询期：多跳遍历 + 更大 context pack

先想 10 秒：为何社区摘要成本常高于单次向量 embedding？

社区摘要需把多个 chunk/实体的文本拼成 prompt 让 LLM 写摘要，输出 token 亦计入；而单向量 embedding 仅编码单向量输入，无生成阶段。

为什么要学本章

上一章 讨论了 GraphRAG 边界；一旦上图，图谱与向量索引、文档 ACL、下线流程必须同版本治理，否则会出现「向量已删、图边仍在」的越权或多跳幻觉。本章聚焦 graph governance、incremental graph、kg version，并与企业知识库治理课形成 governance bridge。

学完你能
  • 制定图与向量 kg version 对齐策略
  • 设计文档下线时的图节点级联删除与 graph refresh
  • 实现 incremental graph 更新而非全量重抽
  • 说明 graph acl 与 policy sync 如何与 S1 多库路由一致
本章先记住 4 件事
  • 图向量同版本：alias 切换必须同时覆盖向量库与图存储 snapshot。
  • 下线要删边：文档失效 → chunk 删 → 实体/边 tombstone。
  • 增量优先：仅对变更文档子图重算社区摘要。
  • 治理课衔接：元数据、审计、合规流程与向量 RAG 共用同一治理面。

何时需要

进入治理级 GraphRAG 的前提，仍是 when to use 成立（多跳/global 问法），且已通过 评测闭环 证明简化方案不足。治理视角额外要求：

  • 图谱变更可审计（谁触发、影响哪些社区摘要）
  • 租户/部门 ACL 在图遍历中可 enforcement
  • 与文档生命周期（发布/修订/废止）绑定
CorpAssist · 制度废止 HR-2023 差旅制度废止后，向量 chunk 已删，但 GraphRAG 仍沿旧边回答「出差标准」——典型治理缺失。

治理不会消除 graph cost，但可通过增量与范围控制降低反复全量抽取：

策略触发成本治理收益

entity resolution（同一实体多写法合并）应在增量更新中重跑，否则边会分裂，查询期多跳断裂。

简化方案

治理语境下的 simple alternative 不是「不用图」，而是降低图治理复杂度：

  1. 子图 scope：仅「跨制度引用」类目建图，其余仍向量
  2. 共现边替代 LLM 关系：规则边 + 人工审核，减少抽取噪声
  3. 社区摘要缓存：版本化 object store，未变更社区只读缓存
  4. 双轨发布：向量先上线，图检索 beta 路由，共用 eval gate

  • 废止文档仍多跳可达
  • 无法回答「用的哪版图」

缓存

CorpAssist S1 · 高峰 FAQ 周一上午 2000 次「年假怎么请」——语义相近但字面不同。精确匹配 cache miss；semantic cache 用 query embedding 在 Redis 中找 cosine > 0.92 的历史问答，命中则跳过检索与 LLM。

缓存键设计（Spring 查询侧）

java
// cache key = hash(kb_id + user_dept + query_embedding_bucket)
String cacheKey = CacheKeys.semantic(user.getDept(), kbId, queryVector);
Optional<CachedAnswer> hit = semanticCache.lookup(cacheKey, queryVector, 0.92);
if (hit.isPresent()) return hit.get().withSource("semantic-cache");

小模型路由

small model route 在进入向量库之前用轻量分类器或规则判断：若为高频 FAQ、无多跳、无合规歧义，则用小模型（或模板）直接答；否则走完整 RAG。CorpAssist S1 典型路由信号：

  • intent = faq_entry（入口类）且 confidence > 0.85
  • 无跨库/跨部门关键词（「财务+HR 冲突」→ 禁走小模型）
  • 用户未要求引用条款编号

先想 10 秒：semantic cache 为何必须把 user_dept 放进 cache key？

不同部门可见文档集不同；若只按 query 相似度命中，可能返回其他部门可见、本用户不可见的缓存答案，构成隐性 acl leak。键必须绑定 ACL scope。

预计算

precompute embed 把索引期与查询期的 embedding 成本前移：热门 chunk、高频改写 query 在离线 job 中算好向量并 pin 到内存/SSD；在线只查表。与 hot query 分析联动——从 access log 取 Top-N 问法， nightly 跑检索 + 答案缓存预热。

手段P95 延迟Token 节省风险
semantic-cache↓ 60～80%↓ 100%（命中时）制度变更未失效
small-model-route↓ 40%↓ 70%（FAQ 子集）误路由无引用
precompute-embed↓ 15～25%↓ 10%（embed API）热点漂移需重算

越权检索

CorpAssist S1 · 权限事故 市场部员工问「公司差旅标准」，检索未带 dept filter，命中仅 HR 可见的《高管差旅补充规定》chunk，答案泄露职级差异条款——典型 acl leak。

doc acl 在索引期写入 chunk metadata（dept、classification、tenant）；查询期 Spring AI 构造 filter，禁止把 user 输入拼进 filter 字符串而不经参数化。

注入

retrieval injection：攻击者在可上传文档中嵌入「系统指令」，被检索进 context 后操纵生成。与直接 prompt 注入不同，它借检索链路进入 Prompt。

  • 入库：剥离隐藏字、异常 Unicode；标记 external/untrusted 来源
  • 检索后：对 chunk 做指令模式检测；高风险片段降权或剔除
  • 生成：system prompt 声明「文档内容非指令」；启用 faithfulness 校验
注入敞口
  • 用户上传 PDF 不经清洗直接索引
  • 网页抓取与制度库同一 vector collection
  • context 原文拼接无分隔符
防护分层
  • 来源标签 + 信任域隔离 collection
  • XML/JSON 边界包裹 chunk 引用
  • 与 幻觉治理 拒答联动
先想 10 秒：为何 RAG 注入在「检索层 ACL 正确」时仍可能发生？

ACL 只限制谁能看到哪些文档，不审查文档内容是否含恶意指令。用户有权看到的制度 PDF 仍可能被人为投毒；需内容级检测与 prompt 结构隔离。

审计

audit log 支撑合规复核与事故溯源：谁、何时、问了什么、命中哪些 chunk、返回答案与 citation。与 semantic cache 命中也要记录，避免「无检索痕迹」的静默回答。

  RET --> HITS[chunk_ids]
  HITS --> GEN[生成+citation]
  GEN --> LOG[audit-log]
  CACHE[cache hit] --> LOG
  LOG --> SIEM[合规留存 180d]

  • Naive → Advanced：先跑通「检索+生成」，再按需加改写、重排、混合检索与拒答（详见 检索、重排与生成）。
  • 双栈分工：Python 负责离线 索引管线；Spring AI 负责在线 查询与引用，共用同一 collection 与 ACL 元数据。

Naive vs Advanced RAG

检索增强生成（RAG） 的核心动作：把用户问题变成检索请求，从知识库取回相关片段，拼进 Prompt 再交给 LLM 生成。工程上常分两层理解：

  • Naive RAG — 问题直接向量化 → TopK 相似块 → 一次生成。实现快、适合 PoC，但对口语化问题、表格 PDF、权限过滤往往不够。
  • Advanced RAG — 在检索前后加「预处理与后处理」：查询改写、混合检索、Rerank、引用约束、拒答与评测闭环。CorpAssist S1 上线目标通常落在此层。
CorpAssist S1 · 制度问答（45 秒） HR 同事问：「试用期考核不过，补偿怎么算？」知识库里有 2024 版《员工手册》PDF 与 2023 年旧版 Word。Naive 只取向量最近的一块，可能命中旧版或半句话；Advanced 会结合元数据过滤、重排与「必须带引用」的生成策略。

Naive 常见踩坑
  • 脏 PDF 不清洗就入库，检索全是页眉页脚
  • 检索层不做 ACL，越权片段进入 Prompt
  • 纯向量忽略制度编号等关键词
  • 无引用，用户无法核对来源
Advanced 最小增量
  • 接入与分块质量先行（文档接入与解析、分块策略）
  • 检索带 dept / doc_version 过滤
  • TopK 扩大 + Rerank 收窄（检索、重排与生成）
  • 生成 Prompt 要求「仅依据上下文 + 列出引用」
先想 10 秒：CorpAssist 只有 2 周做 MVP，你会先 Naive 还是直接 Advanced？

合理路径：Naive 跑通双栈与评测集，同时把接入/分块/ACL 设计按 Advanced 预留；别在 Naive 里堆 Rerank+GraphRAG，否则排障困难。2 周后用基准集决定加哪一环。

CorpAssist 管线

CorpAssist S1 企业知识库的主线：制度与手册可检索、可引用、可审计。Dual stack rag 在此的具体含义：

  • Python 索引栈 — 定时/事件触发：扫描对象存储 → Reader → 清洗 → 分块 → Embedding → upsert 向量库，写入 model_version 与 ACL 字段。
  • Spring 查询栈 — BFF 或 RAG API：鉴权 → 注入用户租户/部门 → Retriever 带 filter → Generator 流式输出 + SSE 引用。

为什么不把索引也写在 Spring 里，少维护一套 Python？
企业制度批量入库、OCR、大文件解析在 Python 生态工具更成熟；Spring 侧聚焦低延迟 API 与权限。双栈增加对齐成本，但换来离线吞吐与在线 SLA的分工清晰。practice 阶段会做 parity 对照。
Advanced RAG 是否等于一定要上 GraphRAG？
否。S1 多数场景「混合检索 + Rerank + 引用 + 拒答」已足够；GraphRAG 在 advanced 阶段单独讲边界与成本。

本章结论

每条对应上文一个小节，便于串起全章再做题。

  • Naive vs Advanced：Naive 适合 PoC；S1 生产需在检索前后加 ACL、重排、引用与评测，而不是一次性堆复杂模块。
  • 组件图：离线 Indexer（Reader→Chunk→Embed）与在线 Retriever+Generator 分离；向量库是双栈唯一共享锚点。
  • CorpAssist 管线：Python 写索引、Spring 答问题；meta 契约（source/page/dept/version）两端一致，才能安全引用。

为什么要学本章

RAG 效果上限往往卡在「第一公里」：ingestion 把扫描件当纯文本、HTML 把导航栏一起入库，后面分块与检索再强也是垃圾进垃圾出。CorpAssist S1 的制度库混有 PDF、Word 与内网 Wiki HTML，本章建立格式分流 + 清洗 + meta 抽取的统一规范。

本章先记住 4 件事
  • 格式分流：数字 PDF 走文本层；扫描 PDF 走 OCR；Word 保留标题结构；HTML 先去模板噪声。
  • OCR 是成本开关：只在文本层为空或质量低于阈值时触发，并记录置信度供人工复核。
  • 清洗目标：去掉页眉页脚、重复水印、乱码字符，保留可检索正文与表格语义。
  • 双栈 Reader：解析在 Python 索引任务完成；Spring 只消费已入库文本与 meta，不在线解析原件。

格式路由：推荐 vs 反例

不推荐
  • 所有 PDF 统一 pdftotext，扫描件得到空串仍入库
  • HTML 用正则粗暴 strip，正文与导航粘连
  • Word 只取 plain text，丢弃标题层级
  • 不记录 source / page，引用无法落地
推荐
  • PDF 先探测文本层；不足则标记 needs_ocr=true
  • HTML 用结构化解析（如 readability / trafilatura 思路）
  • Word 导出段落 + style=Heading*
  • 每段文本带 metadata extract：source、page、dept

OCR 策略

扫描 PDF、拍照归档、部分盖章件没有可靠文本层，需要 ocr strategy。策略核心：何时 OCR、用什么引擎、如何验收质量、失败怎么办。

OCR 平均置信度 < 0.85 标记 review_required 进入人工或二次引擎
表格页、双栏排版 专用版面分析 plain OCR 易乱序

flowchart TD
  PDF[PDF 入库] --> DET{文本层可用?}

  DET -->|否| OCR[OCR 引擎]

常见误区
  • 全量 OCR — 成本与耗时暴涨；数字 PDF 完全没必要。
  • OCR 后不清洗 — 断字、竖线、页码会进入分块，检索命中页眉。
  • 在线 OCR — 用户提问时才 OCR，违背离线索引原则；应回退索引任务。

清洗

清洗发生在 Reader 之后、分块之前。目标：让下游 embedding 看见语义连续的正文，而不是版式垃圾。

典型清洗规则

  • 重复噪声 — 页眉「内部资料」、页脚页码、水印文字（正则 + 频次检测）。
  • 空白与断行 — 合并错误换行；保留段落空行供结构分块。
  • 乱码 — 剔除不可见字符与错误编码产生的替换符（U+FFFD）。
  • 表格 — 转为 Markdown 表格或「列:值」行，避免列错位（制度里常见报销标准表）。

双栈 Reader

Dual reader 指：解析只发生在 Python 索引栈；Spring 查询栈读取的是已解析文本与 meta，而不是重复实现一套 Reader。

实现：LangChain DocumentLoader / Unstructured；按 MIME 路由。

输出：Document 列表 + metadata；失败任务进 dead-letter 队列。

版本：记录 parser_version，便于回归对比。

实现：VectorStore 检索返回 text + meta；可选返回 source 链接给前端引用。

禁止：在 Advisor 里打开 MinIO 原 PDF 现解析（延迟与一致性风险）。

本章结论

每条对应上文一个小节。

  • 双栈 Reader：Python 解析入库，Spring 只检索；meta 契约是双栈对齐的接口。

Embedding 模型选择

模型决定「语义空间」——同一句话在不同模型下的向量不可比较。CorpAssist S1 人事制度库以中文条款为主，选型时至少看四维：

  • 语种与领域 — 通用 multilingual 模型 vs 中文优化模型；制度、合同类文本优先测 recall@5。
  • vector dim — 常见 768 / 1024 / 1536；维度越高存储与检索成本越高，收益需用评测集验证。
  • 费用与 QPS — 索引阶段可离线慢速批处理；查询阶段 query embedding 需控延迟。
  • 版本治理 — 模型升级写入 model version 元数据，便于审计与回滚。

反例 vs 推荐做法

不推荐

  • 换模型后直接覆盖旧向量，无版本号
  • 只看 MTEB 榜单，不在 S1 制度集上实测
  • 维度与向量库 schema 不一致仍强行写入
推荐
  • 索引与查询共用同一 embedding model 配置
  • 变更模型 → 新 collection + index alias 切换
  • 在 CorpAssist 基准问答集上对比 recall / MRR
  • collection meta 固定 model、dim、version

批处理索引（Batch Index）

CorpAssist S1 · 夜间全量索引（45 秒） HR 上传 200 份制度 PDF，经 分块策略 产出约 12 万 chunk。Python 索引任务在 02:00 启动：分批调用 Embedding API，写入 Milvus/PGVector，并在 meta 表记录 job_id 与进度。

batch index 的核心不是「一次 embed 很多条」，而是可恢复、可观测、可限流的离线管线：

  • 批大小 — 按 API 上限（如 64～256 条/请求）与内存折中；失败批次可单独重试。
  • 幂等写入 — chunk_id 作主键，重复跑任务不 duplicated。
  • checkpoint — 记录最后成功 doc_id，断点续跑。
  • 与查询隔离 — 写入 staging collection，校验通过后 alias 切流。
python 
BATCH_SIZE = 128
for batch in chunked(chunks, BATCH_SIZE):
    vectors = embed_client.embed_documents([c.text for c in batch])
    store.upsert(
        ids=[c.chunk_id for c in batch],
        vectors=vectors,
        metadatas=[{"doc_id": c.doc_id, "model_version": MODEL_VER} for c in batch],
    )
    job_repo.save_checkpoint(batch[-1].chunk_id)

向量维度与 Re-embed

vector dim 是模型输出长度，向量库建表时必须固定。常见陷阱：

  • 从 768 维模型升级到 1536 维 → 旧索引不能原地 patch，必须 re embed 全库。
  • 降维截断（MRL）需确认检索库与 API 均支持同一截断策略。
  • metadata 应存 embedded_at、model_version，便于增量补跑。

维度 × 存储粗算（教学示意）

768

100 万 chunk × 768 维 ≈ 3 GB 向量本体（不含索引结构与 metadata）。

常见误区
  • 只 re-embed 增量文档 — 换模型后旧 chunk 与新 chunk 不在同一空间，必须全量或分版本 collection。
  • 查询端不校验 dim — Spring 启动时应读 collection schema，维度不符直接 fail-fast。
  • 忽略 embedding 缓存 — 同一 chunk 文本未变时不应重复计费 embed。

本章结论

Embedding 是检索的「坐标系」；双栈共用同一模型与维度契约。

  • 模型选择：在 S1 基准集上实测，记录 model version，索引与查询禁止混模型。
  • 批处理：batch index 要 checkpoint、幂等、staging + alias 切换。
  • 维度：vector dim 变更触发全量 re embed，不能原地混用。
  • 双栈：python index 写、spring query 读，共用 collection 与 metadata schema。

TopK 召回与阈值

top k 是向量检索返回的候选 chunk 数。CorpAssist S1 常见配置：

  • 召回 K=20～50 — 保证制度条款编号、同义词不被漏掉。
  • 进 prompt K=3～5 — 受 LLM context 与噪声约束。
  • score threshold — 低于阈值直接丢弃，为后续拒答铺垫（见 幻觉治理）。
  • mmr — 在 TopK 内做多样性，避免多条重复条款占满 context。
参数典型值调大影响调小影响
召回 TopK 20 召回↑、延迟↑、rerank 成本↑ 漏召回、专有名词 miss
score threshold 0.65～0.75 更多 chunk 进 rerank 更早触发拒答
MMR λ 0.5～0.7 更偏相关、少多样性 更分散、可能丢最佳条
先想 10 秒：为什么「召回 20、进 prompt 5」比「全程 TopK=5」更好？

向量召回是粗排；宽召回 + rerank 精排能在不撑爆 context 的前提下提高 relevant chunk 进入 prompt 的概率。

Rerank 在链路中的位置

CorpAssist S1 · 「年假天数」查询（30 秒） 员工问「入职满一年年假几天？」。向量 TopK=30 召回了「考勤总则」「福利手册」「年假细则」等 chunk，但细则条款 cosine 仅排第 8。rerank（cross-encoder 或 API）将「年假细则 §3.2」升至第 1，再进入 context pack。

Rerank 应放在向量召回之后、LLM 之前：

  1. Query → Query Embedding → VectorStore.similaritySearch(topK=30)
  2. 可选：MMR / 阈值过滤
  3. Rerank(query, candidates) → 取 top 5
  4. 组装 context pack（含 doc_id、section、score）
  5. LLM generate
检索 → Rerank → 生成

反例 vs 推荐

不推荐
  • 直接把 TopK=30 全塞进 prompt
  • 无 rerank，仅靠向量分数排序
  • rerank 与召回用不同 query 文本
推荐

  • context pack 带 score 供拒答/审核
  • 双栈共用 rerank 配置与 cutoff

context_pack = build_context_pack(reranked)  # doc_id, section, text

引用（Citation）设计

S1 制度问答的合规要求是：用户看到的每条结论可追溯到库内文档。citation 常见形态：

  • 行内标记 — 「年假 10 天[1]」，脚注链到 doc 标题 + 章节。
  • 结构化 JSON — citations: [{chunk_id, doc_id, page, score}] 随 API 返回。
  • Prompt 约束 — 要求模型「仅依据 context，每句末尾标注 [n]」。

常见误区
  • 只返回纯文本 — 前端无法跳转 PDF 原文，审计失败。
  • 引用与 context 不一致 — 模型编造 [3] 但 context 只有 2 条；需后验校验。

拒答（Refusal）策略

拒答触发条件（满足其一即可 abstain）：

  • 检索 top1 score < min_score（无可靠依据）
  • 用户问题超出知识库 scope（如问 S4 代码库内容却走 S1 路由）
  • 生成后 groundedness 校验失败（答案句无法在 context 中找到支撑）
  • 安全策略命中（越权 doc、注入片段）

    abstain_on_empty_context: true

// demos/practice-02-spring-rag-lab/SpringRagDemo.java 思路
@Configuration
class RagConfig {
  @Bean
  ChatClient chatClient(ChatClient.Builder builder, VectorStore vectorStore) {
    var advisor = QuestionAnswerAdvisor.builder(vectorStore)
        .searchRequest(SearchRequest.builder()
            .topK(4)
            .filterExpression("tenant_id == 'corp-a' && kb == 'hr'")
            .build())
        .build();
    return builder.defaultAdvisors(advisor).build();
  }
}

@RestController
@RequestMapping("/v1/rag")
class RagController {
  private final ChatClient chatClient;

  @PostMapping("/ask")
  AskResponse ask(@RequestBody AskRequest req) {

        .user(req.question())
        .call()
        .content();
    // 从 Advisor 上下文或 Document 列表映射 sources — 与 Python 同 DTO
    return new AskResponse(answer, mapSources(...));
  }
}

Hybrid search 在检索阶段并行（或串行）执行稀疏与稠密检索，再融合排序。双栈实现：Python 可在 LangChain 自定义 Retriever；Spring 可在 VectorStore 外加 Elasticsearch/OpenSearch 客户端，或平台原生 hybrid API。

def chunk_java_file(path: str, source: str) -> list[dict]:
    tree = parser.parse(bytes(source, "utf8"))
    chunks = []
    for node in tree.root_node.children:
        if node.type in ("class_declaration", "method_declaration"):
            chunks.append({
                "text": source[node.start_byte:node.end_byte],
                "symbol": extract_symbol(node),
                "file_path": path,
                "start_line": node.start_point[0] + 1,
            })
    return chunks

判断练习

  1. 用户问「这个项目怎么跑起来？」应走 symbol index 还是语义向量？

    参考答案

    语义向量为主。问法不含明确符号名，README / build.gradle 等文档型 chunk 靠 embedding；可对 README.md 提权而非 symbol lookup。

  2. Spring filter 漏写 branch，Python 索引已按 branch 分 collection。会出现什么现象？

    参考答案

    可能检索到其他 branch 的 stale chunk，或空结果（若 collection 按 branch 隔离且 filter 不匹配）。双栈 parity 失败；应在 Spring SearchRequest 与 Python filter 对齐 branch 字段。
