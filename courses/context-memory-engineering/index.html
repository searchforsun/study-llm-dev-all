System / Tool / History：LLM 上下文窗口的四层结构

每个 LLM 请求的上下文窗口由四类内容组成，它们在窗口中所占比例和稳定性各不相同。理解这四层是 context anatomy 的核心。

层级内容稳定性典型占比优化方向
System Prompt 角色定义、行为约束、输出格式 固定，每请求写入 15-25% 精炼指令，去冗余
Tool Outputs API 返回、DB 查询、代码执行结果 动态，每次调用产生 10-20% 结果精简、截断长字段
Conversation History 用户/助手多轮对话 累积增长 20-30% 摘要、滑动窗口
Retrieved Chunks RAG 检索的文档片段 动态，每轮检索 30-45% 块大小、相关性过滤

上下文窗口的四大组成及其数据来源。预算分配决定了每个层级的最大 Tokens。

CorpAssist 的 System Prompt 遵循 system tool history 分层原则：固定角色描述始终占据第一部分，Tool 输出注入到具体位置，历史对话通过 sliding window 裁剪后追加。

常见误区
  • System Prompt 越长越好 — 超过 2K 字后逐字遵守率下降；精炼是关键。
  • 历史轮次全部保留 — 20 轮对话约 8K tokens，挤占检索空间。
  • Tool 输出未经截断 — 日志原文可能有数万行，压缩后再注入。
先想 10 秒：如果 32K 窗口中 history 占了 25K，会发生什么？

System Prompt 和检索结果会被挤出可用窗口。LLM 在 token 压力下会丢失角色约束，回答变得不可控。这是最常见的爆窗表现。

Token 预算分配：策略与比例设计

token budget 是对上下文窗口的显式资源规划。budget split 决定了每一层能使用多少 tokens。没有预算规划，上下文就会进入「谁抢到是谁的」的无序状态。

CorpAssist · 企业客服窗口预算 CorpAssist 使用 32K 模型处理企业客服。用户询问「我的订单发货了吗」，系统需同时注入：客服角色指令（System）、用户企业信息（Retrieval）、最近 5 轮对话（History）、订单 API 返回值（Tool）。设计一个合理的分配比例。

推荐预算配比

层级T okens百分比配置策略
System Prompt ~4,800 15% 固定 1500 字以内的角色与约束
Retrieval Chunks ~12,800 40% Top-5 chunk，每 chunk ~2560 tokens
History ~6,400 20% 最近 5-8 轮 + 摘要
Tool Outputs ~4,800 15% 结果精简，限制字段数
Reserve ~3,200 10% 模型 output + 缓冲

Python：Token 预算计算

python 
# Token 预算计算器 - CorpAssist 场景
MAX_TOKENS = 32000

budget = {
    "system": int(MAX_TOKENS * 0.15),    # 4800
    "retrieval": int(MAX_TOKENS * 0.40), # 12800
    "history": int(MAX_TOKENS * 0.20),   # 6400
    "tool_output": int(MAX_TOKENS * 0.15), # 4800
    "reserve": int(MAX_TOKENS * 0.10),   # 3200
}

def validate_budget(used: dict) -> bool:
    total = sum(used.values())
    if total > MAX_TOKENS:
        overflow = total - MAX_TOKENS
        print(f"Budget overflow: {overflow} tokens over limit")
        return False
    print(f"Total: {total}/{MAX_TOKENS} tokens used ({total/MAX_TOKENS*100:.1f}%)")
    return True

# 验证
print(budget)
validate_budget(budget)

Java：Spring AI Token 预算配置

java 
// Token 预算配置 - CorpAssist 客服场景
public class TokenBudget {
    public static final int MAX_TOKENS = 32000;

    public record BudgetAllocation(
        int system, int retrieval, int history,
        int toolOutput, int reserve
    ) {
        public BudgetAllocation {
            int total = system + retrieval + history + toolOutput + reserve;
            if (total > MAX_TOKENS) {
                throw new IllegalArgumentException(
                    "Budget overflow: " + (total - MAX_TOKENS) + " tokens over limit");
            }
        }

        public static BudgetAllocation defaultAllocation() {
            return new BudgetAllocation(
                (int)(MAX_TOKENS * 0.15),   // system: 4800
                (int)(MAX_TOKENS * 0.40),   // retrieval: 12800
                (int)(MAX_TOKENS * 0.20),   // history: 6400
                (int)(MAX_TOKENS * 0.15),   // tool: 4800
                (int)(MAX_TOKENS * 0.10)    // reserve: 3200
            );
        }

        public double usagePercent() {
            int used = system + retrieval + history + toolOutput;
            return (double) used / MAX_TOKENS * 100;
        }
    }
}
不推荐
  • 无预算规划，所有内容堆到一个 prompt
  • 历史对话无上限累积
  • 检索 chunk 不限制数量，一次性注入 20+ 片段
  • System prompt 被历史挤压到窗口尾部
推荐
  • 显式分配每层 token 上限，使用 alloc ratio
  • 历史对话使用滑动窗口裁剪
  • 检索 chunk 限制 Top-K 并做相关性过滤
  • System prompt 放在上下文最前段保障优先级
常见误区：预算不是越小越好

预算压缩过度会导致 LLM「失忆」——System 约束被裁剪、检索信息不足。合理的预算分配是保留核心结构，压缩冗余表达。每层都留出 10-15% 冗余空间。

context layer 的动态组装逻辑

每层在运行时动态计算实际 token 数，如果超出预算，触发对应的降级策略：

  • Retrieval 超预算 → 减小 chunk size 或降低 Top-K
  • History 超预算 → 触发滚动摘要，丢弃最早轮次
  • Tool 超预算 → 截断字段（如只返回前 100 字符）
检查理解：为什么 History Layer 要放在 Retrieval Layer 之后？

System → Retrieval → History → Tool 的顺序确保最重要的约束（System）和知识（Retrieval）在上下文前段，它们的注意力衰减最小。历史对话虽重要但时效性信息通常更依赖最近几轮。

预算验证与运行时监控

预算分配是静态设计，但实际 token 消耗是动态的。必须引入运行时监控，防止实际上下文超出预算。

监控指标告警阈值触发动作
实际总 tokens 占比 ≥ 90% 触发压缩/裁剪
History 层占比 ≥ 30% 触发滚动摘要
单次检索占比 ≥ 50% 减少 Top-K 或 chunk size
Reserve 剩余 ≤ 5% 限制 tool 调用次数

常见问题（FAQ）

预算分配比例是固定的吗？每种场景都适用 15/40/20/15/10？
不是。不同场景需要不同配比。客服场景侧重检索，所以 Retrieval 占比高。Agent 场景 Tool Output 可能占更多。关键是要显式分配而不是默认不分配。建议每个场景单独设计 budget split。
Reserve 的 10% 是留给模型输出的吗？
部分是的。模型输出 token 同样消耗总窗口。Reserve 同时也作为缓存用——当某层临时超出预算时，从 Reserve 借用。但借用的需要在下一次请求中归还。
如何监控真正的 token 消耗？
使用 tokenizer 库（如 tiktoken）在每个请求组装前预计算 token 数。不要事后从模型返回中获取——那时已经来不及了。Log 每次请求的 token 分布，用于调优预算。

本章结论

上下文工程的第一步是理解窗口的组成结构并制定显式预算。

  • 四层结构：System / Tool / History / Retrieval 共同组成上下文窗口，各自有稳定的占比范围。
  • 显式预算分配：使用 budget split 为每层分配 token 上限，防止上下文无序增长。
  • 模板化组装：corpassist template 将预算分配落地为可复用模板，保障每次请求都在预算内。
  • 运行时监控：预算分配是静态设计，需要运行时 token 监控配合，触发降级策略。
  • 场景差异：不同场景（客服、Agent、RAG）需要不同的 alloc ratio，核心原则是显式分配。

三种截断策略：FIFO / Semantic / Sliding Window

当上下文总 token 超过预算上限时，需要执行 truncation strategy 来裁剪内容。三种主流策略各有优劣。

三种策略在实现难度、保留质量和成本之间有不同的取舍。

策略原理优点缺点适用场景
FIFO 截断 删除最早的对话轮次，直到满足预算 实现简单，O(1) 时间复杂度 可能删除关键约束或用户画像信息 工具调用、日志记录
Semantic 截断 用 LLM/embedding 评分每轮语义价值，保留高分的 保留语义重要内容，质量最高 需要额外 LLM 调用，延迟和成本高 Agent 决策、关键对话
Sliding Window 保留最近 N 轮对话，超出时将最早轮次压缩为摘要 平衡简单与智能，常用折中方案 摘要可能失真，多层摘要后质量下降 客服对话、通用场景
CorpAssist · 客服对话截断决策 用户与企业客服进行了 15 轮对话，第 3 轮用户说了「我的会员等级是钻石，应享受优先处理」。第 12 轮用户提供了订单号。现在要给 Llama 3 8K 模型构建上下文，窗口只能容纳最近 8 轮。如果使用 FIFO，第 3 轮的会员信息会丢失，导致 LLM 不知道用户应享受优先服务。选择语义截断或滑动窗口可以保留这一关键约束。

fifo trim 实现示例

python 
# FIFO 截断 - 移除最早对话轮次
from collections import deque

class FifoTruncation:
    def __init__(self, max_tokens: int):
        self.max_tokens = max_tokens
        self.history = deque()

    def add_turn(self, user_msg: str, assistant_msg: str):
        turn_tokens = len(user_msg) + len(assistant_msg)

// FIFO 截断 - Java 实现
import java.util.*;

public class FifoTruncation {
    private final int maxTokens;
    private final Queue<ConversationTurn> history = new LinkedList<>();

    public FifoTruncation(int maxTokens) {
        this.maxTokens = maxTokens;
    }

    public void addTurn(String user, String assistant) {
        int tokens = estimateTokens(user) + estimateTokens(assistant);
        history.add(new ConversationTurn(user, assistant, tokens));
        int total = history.stream().mapToInt(t -> t.tokens).sum();
        while (total > maxTokens && !history.isEmpty()) {
            ConversationTurn removed = history.poll();
            total -= removed.tokens;
        }
    }

    private int estimateTokens(String text) { return text.length(); }

    record ConversationTurn(String user, String assistant, int tokens) {}
}
先想 10 秒：为什么 FIFO 在客服场景可能不适合？

客服开始几轮通常包含用户身份、问题背景等关键信息。FIFO 总是移除最早的，可能把「我的会员等级是钻石」这类关键约束扔掉。

摘要链：滚动摘要与质量保障

summary chain 是指：当对话轮次超过阈值时，将最早的部分轮次压缩为一段摘要，后续不断追加新的对话。摘要本身也会被再次摘要，形成「rolling summary」链。

滚动摘要链：每次超出预算就对最早内容做摘要，但多层压缩可能导致信息丢失。

摘要链的 Python 实现

python 
class RollingSummaryChain:
    def __init__(self, max_turns: int = 5):
        self.max_turns = max_turns
        self.summary = None
        self.recent_turns = []

    def add_turn(self, user: str, assistant: str):
        self.recent_turns.append({"user": user, "assistant": assistant})
        if len(self.recent_turns) > self.max_turns:
            self._compress()

    def _compress(self):
        old_turns = self.recent_turns[:-self.max_turns]
        text = "\n".join(f"用户: {t['user']}\n助手: {t['assistant']}" for t in old_turns)
        # 调用 LLM 生成摘要（生产环境用真实 LLM）
        summary = f"[摘要] {text[:100]}..."
        if self.summary:
            self.summary = f"{self.summary}\n旧摘要: {summary}"
        else:
            self.summary = summary
        self.recent_turns = self.recent_turns[-self.max_turns:]

    def get_context(self) -> str:
        parts = []
        if self.summary:
            parts.append(f"[对话摘要]\n{self.summary}")
        parts.append("[最近对话]")
        for t in self.recent_turns:
            parts.append(f"用户: {t['user']}")
            parts.append(f"助手: {t['assistant']}")
        return "\n".join(parts)

质量风险分析

quality risk 是截断策略中最容易被忽视的问题。

  • lose constraint — 用户在第 2 轮说「请用英文回答」，第 8 轮摘要后这条约束消失了，LLM 切回中文。
  • 摘要偏斜 — LLM 生成摘要时可能放大某些细节忽略其他，导致后续回复偏离用户原意。
  • 错误累积 — 对摘要做摘要时，原始错误被逐级放大。

python 
# Token 阈值触发压缩
class CompressionTrigger:
    def __init__(self, budget: int, threshold_ratio: float = 0.8):
        self.budget = budget
        self.threshold = int(budget * threshold_ratio)

    def should_compress(self, current_tokens: int) -> bool:
        return current_tokens > self.threshold

    def decide_target(self, current_tokens: int) -> int:
        """计算需要压缩到的目标 token 数"""
        target = int(self.budget * 0.5)  # 压缩到预算的 50%
        print(f"Need compress: {current_tokens} -> {target}")
        return target

LLM 摘要压缩：提炼式上下文压缩

llm summary 使用 LLM 将多轮对话压缩为精炼摘要。与简单截断不同，LLM 摘要能保留关键实体、意图和约束。

压缩用 Prompt 设计

python 
# LLM 摘要压缩实现
SUMMARY_PROMPT = """请压缩以下对话历史，保留关键信息：
- 用户身份和偏好（如会员等级、部门）
- 已解决的问题和待办事项
- 关键实体（订单号、用户 ID、日期）
- 用户明确表达的约束（语言、格式要求）

对话历史：
{conversation_text}

压缩后的摘要（100 字以内）："""

class LlmContextCompressor:
    def __init__(self, llm_client):
        self.llm = llm_client

    def compress(self, conversations: list, max_tokens: int) -> str:
        text = "\n".join(
            f"用户: {c['user']}\n助手: {c['assistant']}"
            for c in conversations
        )
        prompt = SUMMARY_PROMPT.format(conversation_text=text)
        summary = self.llm.complete(prompt)
        return summary

    def compress_with_entities(self, conversations: list) -> dict:
        """压缩并同时提取结构化实体"""
        prompt = """从对话中提取结构化信息并生成摘要。
摘要保持 100 字以内。
实体提取输出 JSON 格式。"""
        result = self.llm.complete(prompt)
        return {"summary": result, "entities": {}}

// 基于 LLM 的上下文压缩
import org.springframework.ai.chat.model.ChatModel;

public class ContextCompressor {
    private final ChatModel chatModel;

    public ContextCompressor(ChatModel chatModel) {
        this.chatModel = chatModel;
    }

    public String compress(List<ConversationTurn> turns) {
        String prompt = buildCompressionPrompt(turns);
        return chatModel.call(prompt);
    }

    private String buildCompressionPrompt(List<ConversationTurn> turns) {
        StringBuilder sb = new StringBuilder();
        sb.append("压缩以下客服对话，保留用户身份、偏好、已办事项和未解决问题：\n\n");
        for (ConversationTurn t : turns) {
            sb.append("用户: ").append(t.user()).append("\n");
            sb.append("助手: ").append(t.assistant()).append("\n");
        }
        sb.append("\n压缩摘要（150字内）：");
        return sb.toString();
    }
}

quality risk：压缩信息丢失

LLM 摘要压缩的质量受 summary prompt 设计影响。常见的质量风险：

  • 约束丢失：用户的语言偏好、格式要求可能在压缩中被忽略
  • 事实错误：LLM 可能在摘要中「脑补」不存在的信息
  • 时效性丢失：时间敏感的指令（如「今天下午 3 点前处理」）可能被泛化
不推荐
  • 不做 compression prompt 设计，直接用通用摘要
  • 对摘要再做摘要超过 2 层
  • 压缩后不验证关键实体是否保留
  • 每轮都触发压缩，过度压缩降低质量
推荐
  • 设计结构化 summary prompt，明确要求保留实体和约束
  • 限制压缩嵌套 ≤ 2 层
  • 压缩后提取实体清单与原文交叉验证
  • 在 token 阈值 + 固定轮次双条件下触发

注入面检测示例

python 
# 上下文注入检测
INJECTION_PATTERNS = [
    r"忽略(之前|上面|所有)(指令|命令|要求)",
    r"忘记(所有|之前)?(内容|上下文)",
    r"输出(system|系统)(prompt|提示|指令)",
    r"以(JSON|XML|Markdown)格式输出.*(全部|所有)",
]

class InjectionDetector:
    def detect(self, text: str) -> list[dict]:
        import re
        matches = []
        for pattern in INJECTION_PATTERNS:
            for match in re.finditer(pattern, text, re.IGNORECASE):
                matches.append({
                    "pattern": pattern,
                    "matched": match.group(),
                    "position": match.span()
                })
        return matches

    def sanitize(self, text: str) -> str:
        """清洗注入尝试（用标记替换，不直接删除）"""
        import re
        for pattern in INJECTION_PATTERNS:
            text = re.sub(pattern,
                "[检测到可疑指令，已屏蔽]",
                text, flags=re.IGNORECASE)
        return text

隔离策略实现

python 
# 上下文隔离实现
from enum import Enum

class TrustZone(Enum):
    TRUSTED = "trusted"
    SEMI_TRUSTED = "semi_trusted"
    UNTRUSTED = "untrusted"

class IsolatedContext:
    def __init__(self):
        self.zones = {
            TrustZone.TRUSTED: [],
            TrustZone.SEMI_TRUSTED: [],
            TrustZone.UNTRUSTED: [],
        }

    def add_content(self, zone: TrustZone, content: str):
        if zone == TrustZone.UNTRUSTED:
            content = self._sanitize_untrusted(content)
        self.zones[zone].append(content)

    def _sanitize_untrusted(self, content: str) -> str:
        # 用标记包裹不可信内容
        return f"[UNTRUSTED_CONTENT]\n{content}\n[/UNTRUSTED_CONTENT]"

    def assemble(self) -> str:
        parts = []
        for zone in [TrustZone.TRUSTED, TrustZone.SEMI_TRUSTED, TrustZone.UNTRUSTED]:
            for content in self.zones[zone]:
                parts.append(content)
        return "\n\n".join(parts)
java 
// 上下文隔离 - Java 实现
public class ContextIsolation {
    public enum TrustZone { TRUSTED, SEMI_TRUSTED, UNTRUSTED }

    private final Map<TrustZone, List<String>> zones = new HashMap<>();

    public ContextIsolation() {
        for (TrustZone z : TrustZone.values()) {
            zones.put(z, new ArrayList<>());
        }
    }

    public void addContent(TrustZone zone, String content) {
        if (zone == TrustZone.UNTRUSTED) {
            content = "[UNTRUSTED]\n" + content + "\n[/UNTRUSTED]";
        }
        zones.get(zone).add(content);
    }

    public String assemble() {
        StringBuilder sb = new StringBuilder();
        for (TrustZone zone : List.of(TrustZone.TRUSTED, TrustZone.SEMI_TRUSTED, TrustZone.UNTRUSTED)) {
            for (String content : zones.get(zone)) {
                sb.append(content).append("\n\n");
            }
        }
        return sb.toString();
    }
}

常见误区：标记不是万能的

用 [UNTRUSTED] 标记包裹不可信内容可以减少注入风险，但不能完全防止。LLM 可能仍然「跨区域」读取内容。需要结合 system prompt 中明确的区域边界声明。

在 System Prompt 中声明区域边界：

Python：LangChain 会话记忆实现

python 
# LangChain 会话记忆 - CorpAssist 客服场景
from langchain.memory import ConversationBufferWindowMemory
from langchain_community.chat_message_histories import RedisChatMessageHistory

class CorpAssistSessionMemory:
    def __init__(self, tenant: str, user_id: str, redis_url: str):
        self.session_key = f"corpassist:session:{tenant}:{user_id}"
        self.history = RedisChatMessageHistory(
            url=redis_url,
            session_id=self.session_key,
            ttl=86400  # 24 小时
        )

        self.memory = ConversationBufferWindowMemory(
            chat_memory=self.history,
            k=8,  # 保留最近 8 轮
            return_messages=True
        )

    def add_turn(self, user_msg: str, ai_msg: str):
        self.memory.chat_memory.add_user_message(user_msg)
        self.memory.chat_memory.add_ai_message(ai_msg)

    def load_context(self) -> list:
        return self.memory.load_memory_variables({})["history"]

Redis 共享会话存储

redis session 作为双栈共享存储，提供高性能的会话读写和自然的 TTL 支持。

python 
# Redis 会话存储操作
import redis
import json
from datetime import datetime

class RedisSessionStore:
    def __init__(self, redis_url: str):
        self.client = redis.from_url(redis_url)
        self.key_prefix = "corpassist:session:"
        self.ttl = 86400  # 24h

    def _key(self, tenant: str, user_id: str) -> str:
        return f"{self.key_prefix}{tenant}:{user_id}"

    def save_turn(self, tenant: str, user_id: str,
                  user_msg: str, ai_msg: str):
        key = self._key(tenant, user_id)
        turn = {
            "user": user_msg,
            "assistant": ai_msg,
            "timestamp": datetime.now().isoformat()
        }
        self.client.rpush(key, json.dumps(turn))
        self.client.expire(key, self.ttl)
        # 限制长度
        if self.client.llen(key) > 8:
            self.client.lpop(key)

    def load_history(self, tenant: str, user_id: str) -> list:
        key = self._key(tenant, user_id)
        turns = self.client.lrange(key, 0, -1)
        return [json.loads(t) for t in turns]
检查理解：为什么双栈 session key 必须一致？

如果 Python 用 `py:session:{user}`、Java 用 `java:session:{user}`，当用户从 Python 端发起对话后切换到 Java 端，系统会创建新的空会话，用户需要重新描述上下文。统一 key 模式保证用户任一端都能继续之前的对话。

常见问题（FAQ）

Redis 会话的 TTL 如何设置？
推荐 24h。每次写入时重新设置 TTL，保证活跃会话不被自动清理。非活跃 24h 后自动过期。
会话超长怎么办？
限制最大轮次（如 8 轮），超出时压缩最早轮次或触发摘要。可以在写入 Redis 时做长度检查。

        vectorStore.add(List.of(

                .embedding(embedding)
                .metadata(Map.of(
                    "userId", fact.userId(),
                    "factType", fact.factType(),
                    "expiresAt", fact.getExpiresAt()
                ))
                .build()
        ));

        // 结构化存储
        jdbcTemplate.update("""
            INSERT INTO memory_facts (fact_id, user_id, fact_type, content, entities, expires_at)
            VALUES (?, ?, ?, ?, ?, ?)
            ON CONFLICT (fact_id) DO UPDATE SET content = EXCLUDED.content
            """, fact.factId(), fact.userId(), fact.factType(),
            fact.content(), String.join(",", fact.entities()),
            fact.getExpiresAt());
    }

        return vectorStore.similaritySearch(
            SearchRequest.query(query)
                .withTopK(topK)
                .withFilter("userId == '" + userId + "'")
        ).stream()
         .map(this::toMemoryFact)
         .toList();

# 读合并 - 三层信息组合
class HybridMemoryReader:
    def __init__(self, session_store, long_term_store):
        self.session = session_store
        self.long_term = long_term_store

    def read_context(self, user_id: str, query: str) -> dict:
        """读取三层记忆并合并为上下文"""

        # 1. 会话层：最近对话历史
        history = self.session.load_history(user_id)

        # 2. 长期层：相关事实
        facts = self.long_term.recall(user_id, query, top_k=3)

        # 3. 去重合并（按事实内容 hash 去重）
        seen = set()
        combined = []
        for item in history + facts:
            h = hash(item.get("content", str(item)))
            if h not in seen:
                seen.add(h)
                combined.append(item)

        return {
            "history": history,
            "long_term_facts": facts,
            "combined_count": len(combined)
        }

一致性策略

consistency 在三层之间可能面临数据不一致的问题。主要场景和解决策略：

不一致场景原因影响解决策略
会话中刚修正的事实长期记忆尚未同步LLM 使用旧事实会话层覆盖长期层
长期记忆过期TTL 到期但未标记召回无效事实查询时过滤 expires_at
双栈同时写入Python 和 Java 并发数据覆盖最后写入者获胜 + 版本号
不推荐
  • 三层各自独立，无写入协调
  • 长期记忆从不验证是否与当前会话矛盾
  • 无版本号，随机覆盖
  • 读取时不做去重，信息重复注入上下文
推荐
  • 写穿透（write-through）保证三层同步
  • 会话层优先级高于长期层
  • 版本号 + 最后写入者获胜
  • 读取时去重合并

sync policy 决定了异步同步的频率和时机：

  • 即时同步：会话层写入后立即触发长期记忆提取（推荐）
  • 批量同步：累积 N 轮后批量提取事实（适合吞吐量高的场景）
  • 定时同步：固定时间窗口执行同步（适合非实时场景）
检查理解：会话层为什么应该覆盖长期层？

用户最新表达的事实优先级最高。如果用户在对话中说「我之前说错了，我其实不是经理」， 会话层记录了修正，但长期层可能还存着「经理」的旧事实。会话层覆盖长期层保证最新表述优先。

class ApprovalStatus(Enum):
    PENDING = "pending"
    APPROVED = "approved"
    REJECTED = "rejected"
    EDITED = "edited"

class MemoryApprovalQueue:
    def __init__(self):
        self.pending_reviews = []

    def submit_for_review(self, fact: MemoryFact):
        self.pending_reviews.append({
            "fact": fact,
            "status": ApprovalStatus.PENDING,
            "submitted_at": datetime.now(),
            "reviewed_by": None,
            "review_notes": ""
        })

    def approve(self, fact_id: str, reviewer: str):
        entry = self._find(fact_id)
        entry["status"] = ApprovalStatus.APPROVED
        entry["reviewed_by"] = reviewer
        self._write_to_long_term(entry["fact"])

    def reject(self, fact_id: str, reviewer: str, reason: str):
        entry = self._find(fact_id)
        entry["status"] = ApprovalStatus.REJECTED
        entry["reviewed_by"] = reviewer
        entry["review_notes"] = reason

    def edit_and_approve(self, fact_id: str, reviewer: str,
                         edited_content: str):
        entry = self._find(fact_id)
        old_fact = entry["fact"]
        entry["fact"].content = edited_content
        entry["status"] = ApprovalStatus.EDITED
        entry["reviewed_by"] = reviewer
        self._write_to_long_term(entry["fact"])
        self._audit_change(old_fact, entry["fact"], reviewer)

class MemoryEditService:
    def edit_fact(self, fact_id: str, new_content: str,
                  editor: str, reason: str) -> dict:
        old = self._get_fact(fact_id)

        # 创建新版本
        new_version = {
            "fact_id": fact_id,
            "version": old["version"] + 1,
            "content": new_content,
            "changed_by": editor,
            "changed_at": datetime.now(),
            "reason": reason
        }

        # 覆盖当前事实
        self._update_fact(fact_id, new_version)

        # 记录审计
        self._audit_entry({
            "event": "fact_edited",
            "fact_id": fact_id,
            "before": old["content"],
            "after": new_content,
            "editor": editor,
            "reason": reason
        })

        return {"success": True, "new_version": new_version["version"]}

    def rollback_fact(self, fact_id: str, target_version: int,
                      editor: str):
        """回滚到指定版本"""
        version = self._get_version(fact_id, target_version)
        self._update_fact(fact_id, version)
        self._audit_entry({
            "event": "fact_rollback",
            "fact_id": fact_id,
            "rolled_back_to": target_version,
            "editor": editor
        })

class AuditLogStore:
    def __init__(self, db):
        self.db = db
        self.db.execute("""
            CREATE TABLE IF NOT EXISTS memory_audit_log (
                id INTEGER PRIMARY KEY AUTOINCREMENT,
                event_type TEXT NOT NULL,
                fact_id TEXT NOT NULL,
                before_content TEXT,
                after_content TEXT,
                operator TEXT NOT NULL,
                reason TEXT,
                created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
            )
        """)

    def append(self, entry: dict):
        # append-only，禁止 DELETE 和 UPDATE
        self.db.execute("""
            INSERT INTO memory_audit_log
            (event_type, fact_id, before_content, after_content,
             operator, reason)
            VALUES (?, ?, ?, ?, ?, ?)
        """, (
            entry["event"], entry["fact_id"],
            entry.get("before"), entry.get("after"),
            entry["operator"], entry.get("reason")
        ))

    def query_by_fact(self, fact_id: str) -> list:
        """查询某个事实的所有变更历史"""
        return self.db.execute(
            "SELECT * FROM memory_audit_log WHERE fact_id = ? ORDER BY id",
            (fact_id,)
        ).fetchall()

检查理解：审计日志为什么必须是 append-only？

如果审计日志可以被修改，就失去了审计的意义。恶意操作者可以删除自己的操作痕迹。Append-only 保证一旦写入，永久存在。

# 记忆子系统评测
class MemoryEvalSuite:
    def __init__(self, memory_api_client):
        self.client = memory_api_client

    def eval_consistency(self, scenario: str, expected_steps: int) -> dict:
        """评测多轮一致性"""
        # 加载标准评测场景
        turns = self._load_scenario(scenario)
        inconsistencies = []

        for i, turn in enumerate(turns):
            response = self.client.chat(turn["user_input"])
            for required in turn["must_remember"]:
                if required not in response:
                    inconsistencies.append({
                        "turn": i,
                        "required": required,
                        "missing": True
                    })

        consistency = 1 - len(inconsistencies) / expected_steps
        return {
            "consistency": consistency,
            "inconsistencies": inconsistencies
        }

    def eval_recall_accuracy(self, test_facts: list) -> dict:
        """评测记忆召回准确率"""
        correct = 0
        for item in test_facts:
            recalled = self.client.recall_facts(
                item["user_id"], item["query"])
            if any(item["fact_id"] == f["fact_id"] for f in recalled):
                correct += 1

        return {
            "accuracy": correct / len(test_facts),
            "correct": correct,
            "total": len(test_facts)
        }

// SAA Graph 状态管理
import com.alibaba.cloud.ai.graph.StateGraph;
import com.alibaba.cloud.ai.graph.state.AgentState;
import com.alibaba.cloud.ai.graph.state.StateReducer;

// 定义对话状态
public class ConversationState extends AgentState {
    public ConversationState(Map<String, Object> initData) {
        super(initData);
    }

    public String getUserId() { return get("user_id").toString(); }
    public List<String> getMessages() { return get("messages"); }
    public Map<String, Object> getMemory() { return get("memory"); }
}

// 构建 Graph
StateGraph<ConversationState> graph = new StateGraph<>(
    ConversationState::new,
    StateReducer.builder()
        .addReducer("messages", StateReducer::concatAndTrim)  // 消息追加+截断
        .addReducer("memory", StateReducer::lastWriteWins)    // 记忆最后写入者获胜
        .addReducer("user_id", StateReducer::keepOriginal)    // 用户 ID 保留原始值
        .build()
);

检查理解：State Reducer 和记忆系统的区别？

State Reducer 管理 Graph 执行的当前状态（如当前轮次的消息列表），是短暂的。记忆系统管理用户的长期知识（如用户偏好、事实），是持久的。两者分离，各司其职。

常见问题（FAQ）

SAA Advisor 和 LangGraph 有什么异同？
SAA Advisor 是 Spring 生态的拦截器模式，适合 Java 单体应用。LangGraph 是 Python 生态的图执行引擎，适合复杂 Agent 编排。两者在概念上类似但生态不同。
Graph Checkpoint 和 LangGraph Checkpoint 如何对应？
两者概念一致：都是持久化对话状态以支持中断恢复。SAA 使用 CheckpointStore 接口，LangGraph 使用 Checkpointer 类。thread_id 和 session_id 应对齐。

共享黑板书 + 私有记忆分离：协作信息共享，执行细节隔离。

python 
# 共享黑板书实现
from typing import Any
import json

class SharedBlackboard:
    """Redis 实现的共享黑板书"""
    def __init__(self, redis_client, session_id: str):
        self.redis = redis_client
        self.key = f"corpassist:blackboard:{session_id}"

    def write(self, key: str, value: Any, agent: str):
        data = json.dumps({"value": value, "writer": agent, "scope": "shared"})
        self.redis.hset(self.key, key, data)
        self.redis.expire(self.key, 86400)

    def read(self, key: str) -> Any:
        data = self.redis.hget(self.key, key)
        if data:
            return json.loads(data)["value"]
        return None

    def get_all(self) -> dict:
        result = {}
        for k, v in self.redis.hgetall(self.key).items():
            parsed = json.loads(v)
            result[k.decode()] = parsed["value"]
        return result


class PrivateMemory:
    """Agent 私有记忆"""
    def __init__(self, redis_client, agent_id: str, session_id: str):
        self.key = f"corpassist:private:{agent_id}:{session_id}"
        self.redis = redis_client

    def write_log(self, step: int, log: str):
        self.redis.rpush(self.key, f"[Step {step}] {log}")
        self.redis.expire(self.key, 86400)

// 多 Agent 共享记忆 - Java 实现
@Service
public class AgentMemoryService {

    private final RedisTemplate<String, Object> redis;

    public AgentMemoryService(RedisTemplate<String, Object> redis) {
        this.redis = redis;
    }

    public void writeToBlackboard(String sessionId, String key,
                                   Object value, String agent) {
        String redisKey = "corpassist:blackboard:" + sessionId;
        Map<String, Object> data = Map.of(
            "value", value, "writer", agent, "scope", "shared"
        );
        redis.opsForHash().put(redisKey, key, data);
        redis.expire(redisKey, 24, TimeUnit.HOURS);
    }

    public Object readFromBlackboard(String sessionId, String key) {
        String redisKey = "corpassist:blackboard:" + sessionId;
        Map<Object, Object> data = redis.opsForHash().entries(redisKey);
        return data.get(key);
    }
}

常见问题（FAQ）

Checkpoint 存储在哪里？性能如何？

多个 Agent 同时写入黑板书会冲突吗？
黑板书使用 Redis Hash，每个字段独立操作。不同 Agent 写不同 key 不会冲突。如果写同一 key，最后写入者获胜。需要事务性写入时使用 Redis Watch 或 Lua 脚本。

# 会话内事实缓存
class SessionFactCache:
    def __init__(self, ttl_seconds: int = 300):
        self.cache = {}  # user_id -> {facts, expires_at}
        self.ttl = ttl_seconds

    def get(self, user_id: str, query: str) -> list | None:
        """检查缓存中是否有匹配的事实"""
        if user_id not in self.cache:
            return None
        entry = self.cache[user_id]
        if time.time() > entry["expires_at"]:
            del self.cache[user_id]
            return None
        # 对 cache 中的事实做简单相关性过滤
        relevant = [f for f in entry["facts"]
                   if any(kw in query for kw in f["keywords"])]
        return relevant if relevant else entry["facts"]

    def set(self, user_id: str, facts: list):
        self.cache[user_id] = {
            "facts": facts,
            "expires_at": time.time() + self.ttl
        }

    def invalidate(self, user_id: str):
        """用户事实变更时使缓存失效"""
        self.cache.pop(user_id, None)

Lazy Loading：按需加载记忆

lazy load mem 只有在确定需要时才加载长期记忆。mem router 判断是否值得召回。

python 
# 懒加载记忆
class LazyMemoryLoader:
    def __init__(self, cache, long_term):
        self.cache = cache
        self.long_term = long_term

    def load_if_needed(self, user_id: str, query: str,
                       force: bool = False) -> list:
        """按需加载长期记忆"""
        # 1. 先检查缓存
        cached = self.cache.get(user_id, query)
        if cached:
            return cached

        # 2. 判断是否需要召回
        if not force and not self._needs_long_term(query):
            return []

        # 3. 执行召回
        facts = self.long_term.recall(user_id, query)
        self.cache.set(user_id, facts)
        return facts

    def _needs_long_term(self, query: str) -> bool:
        """判断是否需要查询长期记忆"""
        # 简单问候语不需要
        simple_patterns = ["你好", "在吗", "hello", "hi"]
        for p in simple_patterns:
            if p in query.lower():
                return False
        # 包含操作意图的需要
        action_keywords = ["查询", "修改", "删除", "创建", "查"]
        return any(kw in query for kw in action_keywords)

常见误区：过度优化导致记忆失效

缓存 TTL 设太短（< 60s）起不到作用；设太长（> 30min）可能导致用户修改偏好后响应不及时。推荐的折中：5 分钟缓存 TTL + 编辑时主动失效。

检查理解：为什么懒加载可能降低准确性？

懒加载判断逻辑可能过于保守，过滤掉实际需要长期记忆的查询。例如用户说「按照之前的设置处理」，懒加载可能认为不包含关键词而跳过召回，导致 Agent 不知道「之前的设置」是什么。

常见问题（FAQ）

缓存和懒加载应该同时使用吗？
是的。懒加载决定「是否需要召回」，缓存决定「是否使用已有的召回结果」。两者配合能最大化减少不必要的向量搜索。
如何确定最优的 Top-K 值？
从 Top-K=5 开始。观察召回的准确率和延迟。如果 P95 延迟 > 200ms，降低 K 值。如果关键事实经常召回不到，增加 K 值。通常 K=3-5 在准确率和性能间取得平衡。

# 爆窗诊断脚本
class WindowOverflowDiagnoser:
    def __init__(self, budget: int = 32000):
        self.budget = budget

    def diagnose(self, token_distribution: dict) -> list:
        issues = []
        total = sum(token_distribution.values())

        # 1. 检查总 token
        if total > self.budget:
            issues.append(f"爆窗: {total} > {self.budget} ({ (total/self.budget-1)*100:.0f}% 超限)")

        # 2. 检查 History 层占比
        history_ratio = token_distribution.get("history", 0) / total
        if history_ratio > 0.35:
            issues.append(f"History 过载: {history_ratio:.0%} > 35%，建议滑动窗口")

        # 3. 检查 System 层占比
        system_ratio = token_distribution.get("system", 0) / total
        if system_ratio < 0.10:
            issues.append(f"System 被挤压: {system_ratio:.0%} < 10%，需保护 system prompt")

        # 4. 检查 Tool 层异常
        tool_tokens = token_distribution.get("tool_output", 0)
        if tool_tokens > 3000:
            issues.append(f"工具输出过大: {tool_tokens}tokens，建议截断")

        return issues

排障清单：Debug Checklist

debug checklist 是系统排查上下文爆窗的步骤清单。

  1. 查看 Token 分布 — 使用 diagnoser 工具查看各层占比。正常范围：System 10-25%、History 15-30%、Retrieval 30-45%。

  3. 检查摘要链 — summary bug：摘要是否丢掉了关键实体或约束？多层摘要是否失真？

    def test_session_continuity(self) -> dict:
        """RAG 集成测试：会话连续性"""
        # 模拟 5 轮连续对话
        session = []
        for i, msg in enumerate(["查订单", "取消", "退款"]):
            resp = requests.post(f"{self.api}/v1/session/messages", json={
                "tenant": "test", "user_id": "user1", "content": msg
            })
            session.append(resp.json())

        # 验证上下文包含之前的意图
        context = session[-1].get("context", "")
        has_history = "订单" in context or "取消" in context
        return {"test": "session_continuity", "passed": has_history}

    def test_agent_checkpoint(self) -> dict:
        """Agent 集成测试：Checkpoint 恢复"""
        # 模拟 Agent 中断恢复
        agent_id = f"agent:{uuid4()}"
        session_id = f"session:{uuid4()}"

        # 写入 Checkpoint
        requests.post(f"{self.api}/v1/agent/checkpoint", json={
            "agent_id": agent_id,
            "session_id": session_id,
            "state": {"step": 3, "context": "test"}
        })

        # 恢复
        resp = requests.get(
            f"{self.api}/v1/agent/checkpoint/{agent_id}/{session_id}")
        restored = resp.json()

        return {
            "test": "agent_checkpoint",
            "passed": restored.get("state", {}).get("step") == 3
        }

@SpringBootTest
class MemoryIntegrationTest {

    @Autowired
    private MemoryApiClient memoryClient;

    @Test
    void testMultiTurnConsistency() {
        // 3 轮对话后检查状态一致性
        memoryClient.sendMessage("tenant1", "user1", "查订单 20240501");
        memoryClient.sendMessage("tenant1", "user1", "现在能取消吗");

        var context = memoryClient.getSessionContext("tenant1", "user1");
        assertThat(context.getHistory())
            .anyMatch(msg -> msg.contains("20240501"));
    }

    @Test
    void testRecallLatency() {
        // P95 延迟 <= 200ms
        var results = new ArrayList<Long>();
        for (int i = 0; i < 100; i++) {
            long start = System.currentTimeMillis();
            memoryClient.recallFacts("user1", "test query", 5);
            results.add(System.currentTimeMillis() - start);
        }
        Collections.sort(results);
        long p95 = results.get((int)(results.size() * 0.95));
        assertThat(p95).isLessThanOrEqualTo(200);

检查理解：为什么 API 契约要先于实现定义？

三组团队（记忆/RAG/Agent）并行开发时，如果各自按自己的理解定义接口，集成时必然出现参数类型、数据格式的 mismatch。先定义 OpenAPI 契约，团队可以并行 mock 测试。
