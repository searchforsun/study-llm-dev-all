本章先记住 3 件事
  • 三种裁剪策略：FIFO 简单但可能丢关键信息，语义截断智能但贵，滑动窗口是折中。

策略对比：FIFO / Semantic / Sliding Window

当上下文总 token 超过预算上限时，需要执行 truncation strategy 来裁剪内容。三种主流策略各有优劣。

三种策略在实现难度、保留质量和成本之间有不同的取舍。

策略原理优点缺点适用场景
FIFO 截断 删除最早的对话轮次，直到满足预算 实现简单，O(1) 时间复杂度 可能删除关键约束或用户画像信息 工具调用、日志记录
Semantic 截断 用 LLM/embedding 评分每轮语义价值，保留高分的 保留语义重要内容，质量最高 需要额外 LLM 调用，延迟和成本高 Agent 决策、关键对话
Sliding Window 保留最近 N 轮对话，超出时将最早轮次压缩为摘要 平衡简单与智能，常用折中方案 摘要可能失真，多层摘要后质量下降 客服对话、通用场景
CorpAssist · 客服对话截断决策 用户与企业客服进行了 15 轮对话，第 3 轮用户说了「我的会员等级是钻石，应享受优先处理」。第 12 轮用户提供了订单号。现在要给 Llama 3 8K 模型构建上下文，窗口只能容纳最近 8 轮。如果使用 FIFO，第 3 轮的会员信息会丢失，导致 LLM 不知道用户应享受优先服务。选择语义截断或滑动窗口可以保留这一关键约束。

fifo trim 实现示例

python 
# FIFO 截断 - 移除最早对话轮次
from collections import deque

class FifoTruncation:
    def __init__(self, max_tokens: int):
        self.max_tokens = max_tokens
        self.history = deque()

    def add_turn(self, user_msg: str, assistant_msg: str):
        turn_tokens = len(user_msg) + len(assistant_msg)
        self.history.append((user_msg, assistant_msg, turn_tokens))
        total = sum(t[2] for t in self.history)
        while total > self.max_tokens and self.history:
            removed = self.history.popleft()
            total -= removed[2]
            print(f"FIFO 移除: {removed[0][:30]}...")

    def get_context(self) -> list:
        return list(self.history)
java 
// FIFO 截断 - Java 实现
import java.util.*;

public class FifoTruncation {
    private final int maxTokens;
    private final Queue<ConversationTurn> history = new LinkedList<>();

    public FifoTruncation(int maxTokens) {
        this.maxTokens = maxTokens;
    }

    public void addTurn(String user, String assistant) {
        int tokens = estimateTokens(user) + estimateTokens(assistant);
        history.add(new ConversationTurn(user, assistant, tokens));
        int total = history.stream().mapToInt(t -> t.tokens).sum();
        while (total > maxTokens && !history.isEmpty()) {
            ConversationTurn removed = history.poll();
            total -= removed.tokens;
        }
    }

    private int estimateTokens(String text) { return text.length(); }

    record ConversationTurn(String user, String assistant, int tokens) {}
}
先想 10 秒：为什么 FIFO 在客服场景可能不适合？

客服开始几轮通常包含用户身份、问题背景等关键信息。FIFO 总是移除最早的，可能把「我的会员等级是钻石」这类关键约束扔掉。

摘要链：滚动摘要与质量保障

summary chain 是指：当对话轮次超过阈值时，将最早的部分轮次压缩为一段摘要，后续不断追加新的对话。摘要本身也会被再次摘要，形成「rolling summary」链。

Rolling Summary 摘要链示意图
sequenceDiagram
    participant C as Client
    participant M as Memory System
    participant S as Summarizer LLM

    C->>M: Turn 1-5 (full text)
    Note over M: Token budget: 80% used
    M->>S: Summarize turns 1-3
    S-->>M: Summary A: 用户钻石会员，询问发货
    M->>M: Replace turns 1-3 with Summary A
    M-->>C: Context: [Summary A, Turns 4-5]

    C->>M: Turn 6-8 (full text)
    Note over M: Budget exceeded again
    M->>S: Summarize Summary A + Turns 4-5
    S-->>M: Summary B: 钻石会员，发货已提供单号
    M->>M: Replace with Summary B
    Note over M: Risk: 多层摘要后丢失细节

滚动摘要链：每次超出预算就对最早内容做摘要，但多层压缩可能导致信息丢失。

摘要链的 Python 实现

python 
class RollingSummaryChain:
    def __init__(self, max_turns: int = 5):
        self.max_turns = max_turns
        self.summary = None
        self.recent_turns = []

    def add_turn(self, user: str, assistant: str):
        self.recent_turns.append({"user": user, "assistant": assistant})
        if len(self.recent_turns) > self.max_turns:
            self._compress()

    def _compress(self):
        old_turns = self.recent_turns[:-self.max_turns]
        text = "\n".join(f"用户: {t['user']}\n助手: {t['assistant']}" for t in old_turns)
        # 调用 LLM 生成摘要（生产环境用真实 LLM）
        summary = f"[摘要] {text[:100]}..."
        if self.summary:
            self.summary = f"{self.summary}\n旧摘要: {summary}"
        else:
            self.summary = summary
        self.recent_turns = self.recent_turns[-self.max_turns:]

    def get_context(self) -> str:
        parts = []
        if self.summary:
            parts.append(f"[对话摘要]\n{self.summary}")
        parts.append("[最近对话]")
        for t in self.recent_turns:
            parts.append(f"用户: {t['user']}")
            parts.append(f"助手: {t['assistant']}")
        return "\n".join(parts)

质量风险

quality risk 是截断与摘要策略中最容易被忽视的问题：窗口看似「还在预算内」，但关键约束已在压缩链路中丢失。

  • lose constraint — 用户在第 2 轮说「请用英文回答」，第 8 轮摘要后这条约束消失了，LLM 切回中文。
  • 摘要偏斜 — LLM 生成摘要时可能放大某些细节忽略其他，导致后续回复偏离用户原意。
  • 错误累积 — 对摘要做摘要时，原始错误被逐级放大。
风险类型表现影响缓解措施
约束丢失格式/语言/角色约束在摘要后消失LLM 输出不符合预期将约束单独放在 System Prompt
摘要偏斜摘要放大次要信息，忽略关键事实后续回答不准确使用结构化的提取式摘要
错误累积多层摘要后事实逐步失真最终回答与原始对话不一致限制摘要层级 ≤ 2

trim policy：为 CorpAssist 选择裁剪策略

不推荐
  • 全部用 FIFO — 会员等级等关键约束被丢弃
  • 全部用语义截断 — 每轮对话都 LLM 评分，成本爆炸
  • 无策略 — 超出窗口直接报错
  • 只保留摘要 — 当前轮次的关键信息被摘要稀释
推荐
  • 滑动窗口保留最近 5-8 轮完整对话
  • 超出时对最早轮次做单层摘要
  • 关键约束（语言、身份、权限）复制到 System Prompt
  • 使用 trim policy 配置化策略选择

常见问题（FAQ）

滑动窗口的 N 值如何确定？
对于 32K 窗口，History 层占 20% ≈ 6.4K tokens。假设每轮约 800 tokens，可保留约 8 轮。建议从 N=5 开始根据实际使用率调整。
摘要链的 LLM 调用是否太贵？
摘要可以异步进行，在用户请求间隙执行压缩。也可使用更便宜的模型（如 Claude Haiku 或 GPT-4o-mini）专用于摘要。

本章结论

  • FIFO 最简但最危险：可能丢弃关键约束，不适合对上下文敏感的场景。
  • 语义截断质量最高：但成本与延迟较高，适合 Agent 等关键路径。
  • 滑动窗口是折中方案：保留最近 N 轮完整对话，平衡质量与成本。
  • 摘要链有积累风险：lose constraint 是最大风险，需要结构化保护关键约束。
  • 策略配置化：使用 trim policy 将策略选择从代码中解耦。

判断练习

  1. FIFO 截断后用户抱怨「助手忘了开场约束」，最可能原因是什么？

    参考答案

    优先检查被裁掉的最早轮次是否含 system 重申或格式约束；应把硬约束固定在 system 层而非仅依赖 history。

  2. 滑动窗口摘要后检索命中率下降，你会先调哪一层预算？

    参考答案

    先收紧 history 占比并提高 retrieval Top-K 质量，而不是继续加长摘要；摘要丢失实体名会直接伤害 RAG。
