为什么要学本章

记忆系统不是免费的。每次请求全量召回长期记忆会增加延迟和 token 消耗。缓存策略、懒加载、top-k 限制——这些优化手段决定了记忆系统能否在生产环境落地。本章量化 cost latency mem 的成本与收益。

本章先记住 3 件事
  • 长期记忆召回不是免费的：每次向量搜索 + 排序引入延迟和计算成本。
  • 缓存是最高性价比的优化：会话内缓存事实，避免重复 embed 和搜索。
  • 不必要不召回：简单查询不需要长期记忆，mem router 减少无效召回。

# 读写放大监控
class LatencyMonitor:
    def __init__(self):
        self.metrics = []

    def record_operation(self, op: str, duration_ms: float):
        self.metrics.append({"op": op, "duration_ms": duration_ms})

    def get_amplification(self) -> dict:
        total = sum(m["duration_ms"] for m in self.metrics)
        return {
            "total_ops": len(self.metrics),
            "total_time_ms": total,
            "avg_per_op": total / len(self.metrics),
            "ops_breakdown": {
                op: sum(m["duration_ms"] for m in self.metrics if m["op"] == op)
                for op in set(m["op"] for m in self.metrics)
            }
        }

缓存策略：减少重复召回

mem cache 在会话内缓存已召回的长期记忆。cache hit 避免重复的向量搜索和 LLM 调用。

会话内事实缓存

python 
# 会话内事实缓存
class SessionFactCache:
    def __init__(self, ttl_seconds: int = 300):
        self.cache = {}  # user_id -> {facts, expires_at}
        self.ttl = ttl_seconds

    def get(self, user_id: str, query: str) -> list | None:
        """检查缓存中是否有匹配的事实"""
        if user_id not in self.cache:
            return None
        entry = self.cache[user_id]
        if time.time() > entry["expires_at"]:
            del self.cache[user_id]
            return None
        # 对 cache 中的事实做简单相关性过滤
        relevant = [f for f in entry["facts"]
                   if any(kw in query for kw in f["keywords"])]
        return relevant if relevant else entry["facts"]

    def set(self, user_id: str, facts: list):
        self.cache[user_id] = {
            "facts": facts,
            "expires_at": time.time() + self.ttl
        }

    def invalidate(self, user_id: str):
        """用户事实变更时使缓存失效"""
        self.cache.pop(user_id, None)

Lazy Loading：按需加载记忆

lazy load mem 只有在确定需要时才加载长期记忆。mem router 判断是否值得召回。

python 
# 懒加载记忆
class LazyMemoryLoader:
    def __init__(self, cache, long_term):
        self.cache = cache
        self.long_term = long_term

    def load_if_needed(self, user_id: str, query: str,
                       force: bool = False) -> list:
        """按需加载长期记忆"""
        # 1. 先检查缓存
        cached = self.cache.get(user_id, query)
        if cached:
            return cached

        # 2. 判断是否需要召回
        if not force and not self._needs_long_term(query):
            return []

        # 3. 执行召回
        facts = self.long_term.recall(user_id, query)
        self.cache.set(user_id, facts)
        return facts

    def _needs_long_term(self, query: str) -> bool:
        """判断是否需要查询长期记忆"""
        # 简单问候语不需要
        simple_patterns = ["你好", "在吗", "hello", "hi"]
        for p in simple_patterns:
            if p in query.lower():
                return False
        # 包含操作意图的需要
        action_keywords = ["查询", "修改", "删除", "创建", "查"]
        return any(kw in query for kw in action_keywords)
不推荐
  • 每次请求全量召回长期记忆
  • 无缓存，同一用户每轮重复向量搜索
  • Top-K 设为 20+，召回大量无关事实
  • 事实提取在每个请求同步执行
推荐
  • Mem router 过滤不需要召回的场景
  • 会话内缓存已召回事实（TTL 5min）
  • Top-K 设为 3-5，控制召回量
  • 事实提取异步执行，不阻塞用户请求

成本权衡：NS3 三角

ns3 trade 代表记忆系统中三个互相制约的目标：

  • N — 准确性（Needle Accuracy）：召回的事实是否正确相关
  • S — 速度（Speed）：召回延迟
  • S — 节省（Savings）：token 和成本
策略准确性速度成本节省适用场景
全量召回（Top-K=20）高慢低关键决策、合规查询
优化召回（Top-K=5+缓存）中高快高日常客服对话
懒加载（按需召回）中极快极高简单问答

token cost mem 衡量的不仅是存储成本，还有每次召回时注入上下文的 token 消耗。

python 
# Token 成本分析
class TokenCostAnalyzer:
    def __init__(self, price_per_1k_tokens: float = 0.003):
        self.price = price_per_1k_tokens

    def analyze_recall_cost(self, user_id: str, daily_queries: int) -> dict:
        """分析每日记忆召回的 token 成本"""
        avg_recall_tokens = 800  # 平均每次召回占 800 tokens
        daily_recall_tokens = daily_queries * avg_recall_tokens
        monthly_recall_tokens = daily_recall_tokens * 30

        return {
            "daily_queries": daily_queries,
            "daily_recall_tokens": daily_recall_tokens,
            "daily_cost": daily_recall_tokens / 1000 * self.price,
            "monthly_cost": monthly_recall_tokens / 1000 * self.price,
            "optimization_potential": {
                "with_cache": monthly_recall_tokens * 0.4 / 1000 * self.price,
                "with_lazy_load": monthly_recall_tokens * 0.6 / 1000 * self.price
            }
        }
常见误区：过度优化导致记忆失效

缓存 TTL 设太短（< 60s）起不到作用；设太长（> 30min）可能导致用户修改偏好后响应不及时。推荐的折中：5 分钟缓存 TTL + 编辑时主动失效。

检查理解：为什么懒加载可能降低准确性？

懒加载判断逻辑可能过于保守，过滤掉实际需要长期记忆的查询。例如用户说「按照之前的设置处理」，懒加载可能认为不包含关键词而跳过召回，导致 Agent 不知道「之前的设置」是什么。

常见问题（FAQ）

缓存和懒加载应该同时使用吗？
是的。懒加载决定「是否需要召回」，缓存决定「是否使用已有的召回结果」。两者配合能最大化减少不必要的向量搜索。
如何确定最优的 Top-K 值？
从 Top-K=5 开始。观察召回的准确率和延迟。如果 P95 延迟 > 200ms，降低 K 值。如果关键事实经常召回不到，增加 K 值。通常 K=3-5 在准确率和性能间取得平衡。

本章结论

  • 读写放大：1 次用户请求可触发 5+ 次后端操作，需要监控和分析。
  • 会话缓存：5 分钟 TTL 会话内缓存避免重复向量搜索。
  • 懒加载：mem router 判断是否需要召回长期记忆，减少无效调用。
  • NS3 三角：准确性/速度/成本三者需根据场景取舍。
  • 异步提取：事实提取异步执行，不阻塞用户请求路径。
