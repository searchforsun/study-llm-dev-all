为什么要学本章

工作记忆只能维持 24h，真正的「了解用户」需要长期记忆——记住用户的偏好、习惯、历史事实。但长期记忆必须可审计、可过期、可删除。本章实现 long term 记忆的完整生命周期。

本章先记住 3 件事
  • 长期记忆 ≠ 全部记忆：只存储有意义的事实，不是完整对话。
  • 向量 + 结构化双存：向量用于语义召回，结构化用于精确查询。
  • 可删除是强制要求：GDPR 要求用户有权删除自己的记忆数据。

# 事实提取与向量存储
from pydantic import BaseModel
from datetime import datetime
import json

class MemoryFact(BaseModel):
    fact_id: str
    user_id: str
    fact_type: str  # "profile" | "preference" | "interaction"
    content: str
    entities: list[str]
    source_turn: str
    confidence: float
    created_at: datetime
    ttl_days: int = 90

class FactExtractor:
    def extract_facts(self, user_msg: str, ai_msg: str) -> list[MemoryFact]:
        """从对话中提取重要事实"""

用户: {user_msg}
助手: {ai_msg}

提取为 JSON 列表：
[{{"fact_type": "preference", "content": "...", "entities": ["..."]}}]"""

        return self._parse_facts(result)

class VectorFactStore:
    def __init__(self, milvus_uri: str):
        from pymilvus import connections, Collection
        connections.connect(uri=milvus_uri)
        self.collection = Collection("memory_facts")

    def upsert_fact(self, fact: MemoryFact, embedding: list[float]):
        self.collection.upsert([{
            "fact_id": fact.fact_id,
            "user_id": fact.user_id,
            "content": fact.content,
            "embedding": embedding,
            "expires_at": int(fact.created_at.timestamp()) + fact.ttl_days * 86400
        }])

    def recall(self, user_id: str, query: str, top_k: int = 5) -> list[MemoryFact]:
        # 召回该用户的长期记忆
        results = self.collection.search(
            data=[embed(query)],
            anns_field="embedding",
            param={"filter": f"user_id == '{user_id}'"},
            limit=top_k
        )
        return [MemoryFact(**r.entity) for r in results[0]]
java 
// 长期记忆 - Java 实现
public class LongTermMemory {
    private final VectorStore vectorStore;
    private final JdbcTemplate jdbcTemplate;

    public void saveFact(MemoryFact fact, float[] embedding) {
        // 向量存储
        vectorStore.add(List.of(
            Document.builder()
                .id(fact.factId())
                .embedding(embedding)
                .metadata(Map.of(
                    "userId", fact.userId(),
                    "factType", fact.factType(),
                    "expiresAt", fact.getExpiresAt()
                ))
                .build()
        ));

        // 结构化存储
        jdbcTemplate.update("""
            INSERT INTO memory_facts (fact_id, user_id, fact_type, content, entities, expires_at)
            VALUES (?, ?, ?, ?, ?, ?)
            ON CONFLICT (fact_id) DO UPDATE SET content = EXCLUDED.content
            """, fact.factId(), fact.userId(), fact.factType(),
            fact.content(), String.join(",", fact.entities()),
            fact.getExpiresAt());
    }

    public List<MemoryFact> recall(String userId, String query, int topK) {
        return vectorStore.similaritySearch(
            SearchRequest.query(query)
                .withTopK(topK)
                .withFilter("userId == '" + userId + "'")
        ).stream()
         .map(this::toMemoryFact)
         .toList();
    }
}

过期策略与 GDPR 删除

expiry policy 管理长期记忆的生命周期。gdpr delete 实现用户删除权。

策略说明实现方式
TTL 过期记忆存储时长有限（如 90 天）写入时设置 expires_at，查询时过滤
LRU 淘汰不常召回的优先过期记录上次召回时间，冷数据优先清理
用户删除用户主动删除自己的记忆级联删除向量 + 结构化 + 审计
租户删除企业删除离职员工记忆按 tenant + user_id 批量删除

# GDPR 删除实现
class MemoryDeletionService:
    def __init__(self, vector_store, db):
        self.vector = vector_store
        self.db = db

    def delete_user_memories(self, user_id: str) -> dict:
        """级联删除用户在长期记忆中的所有数据"""
        result = {"deleted_facts": 0, "deleted_profile": False}

        # 1. 删除向量存储中的事实
        self.vector.delete(f"user_id == '{user_id}'")

        # 2. 删除结构化存储中的事实
        cur = self.db.execute(
            "DELETE FROM memory_facts WHERE user_id = ?", (user_id,))
        result["deleted_facts"] = cur.rowcount

        # 3. 删除用户画像
        self.db.execute(
            "DELETE FROM user_profiles WHERE user_id = ?", (user_id,))
        result["deleted_profile"] = True

        # 4. 记录审计
        self.db.execute("""
            INSERT INTO deletion_audit
            (user_id, deleted_at, facts_count)
            VALUES (?, datetime('now'), ?)
        """, (user_id, result["deleted_facts"]))

        return result
不推荐
  • 长期记忆永不删除
  • 无 TTL 策略，事实永久存储
  • 只存向量不存结构化，无法精确删除
  • 无审计日志，合规审计无从下手
推荐
  • 默认 90 天 TTL，可配置
  • 向量 + 结构化双存
  • 用户可一键删除所有记忆
  • 所有操作记录审计日志
先想 10 秒：为什么需要向量+结构化双存？

向量存储支持语义搜索（「帮我找找关于报告偏好的记忆」），结构化存储支持精确查询（「张经理的邮箱是什么」）和批量删除。单一存储无法同时支持两种查询模式。

长期记忆的召回与刷新

长期记忆的 mem ttl 策略在召回时刷新 TTL，热门记忆自动延长生命。

python 
# 长期记忆召回（刷新 TTL）
class LongTermRecallService:
    def __init__(self, fact_store):
        self.store = fact_store

    def recall_with_refresh(self, user_id: str, query: str) -> list:
        facts = self.store.recall(user_id, query)
        for fact in facts:
            self.store.refresh_ttl(fact.fact_id)
        return facts

    def get_profile(self, user_id: str) -> dict:
        """获取用户画像（结构化）"""
        profile = self.store.get_profile(user_id)
        if profile:
            self.store.refresh_profile_ttl(user_id)
        return profile

常见问题（FAQ）

长期记忆中的错误如何修正？
使用 memory edit 机制。用户可以手动编辑错误事实，系统记录旧值和新值到审计日志。
不同用户的记忆会互相干扰吗？
不会。每个事实都带有 user_id，召回时使用 filter 精确限定用户。向量搜索的 filter 条件和结构化查询的 WHERE 子句共同确保数据隔离。

本章结论

  • 用户画像：结构化存储用户偏好、角色和属性，支持精确查询。
  • 事实库：从对话中提取结构化事实，向量+结构化双存，支持语义召回和精确查询。
  • 过期策略：默认 90 天 TTL，结合 LRU 淘汰冷数据。
  • GDPR 删除：级联删除向量 + 结构化 + 审计，支持用户和租户级删除。
  • 合规审计：所有记忆操作记录审计日志，满足合规要求。
