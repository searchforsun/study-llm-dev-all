爆窗案例：CorpAssist 20 轮

overflow case：CorpAssist 客服系统上线后，前 5 轮表现良好，20 轮后 P95 延迟从 800ms 飙升到 8s，且 LLM 开始不遵守输出格式。诊断显示每轮 token spike 来自重复注入的长期记忆。该案例可作为面试中「现象—数据—根因」的完整叙事素材。

python 
# 爆窗诊断脚本
class WindowOverflowDiagnoser:
    def __init__(self, budget: int = 32000):
        self.budget = budget

    def diagnose(self, token_distribution: dict) -> list:
        issues = []
        total = sum(token_distribution.values())

        # 1. 检查总 token
        if total > self.budget:
            issues.append(f"爆窗: {total} > {self.budget} ({ (total/self.budget-1)*100:.0f}% 超限)")

        # 2. 检查 History 层占比
        history_ratio = token_distribution.get("history", 0) / total
        if history_ratio > 0.35:
            issues.append(f"History 过载: {history_ratio:.0%} > 35%，建议滑动窗口")

        # 3. 检查 System 层占比
        system_ratio = token_distribution.get("system", 0) / total
        if system_ratio < 0.10:
            issues.append(f"System 被挤压: {system_ratio:.0%} < 10%，需保护 system prompt")

        # 4. 检查 Tool 层异常
        tool_tokens = token_distribution.get("tool_output", 0)
        if tool_tokens > 3000:
            issues.append(f"工具输出过大: {tool_tokens}tokens，建议截断")

        return issues

# 使用
diagnoser = WindowOverflowDiagnoser()
issues = diagnoser.diagnose({
    "system": 2100, "retrieval": 15000,
    "history": 14000, "tool_output": 500
})
for issue in issues:
    print(f"[ISSUE] {issue}")

排障清单：Debug Checklist

debug checklist 是系统排查上下文爆窗的步骤清单。

现场排障时建议把 checklist 打印成一页纸，按顺序勾选；跳过步骤最容易把「检索重复注入」误判为「模型退化」。

  1. 查看 Token 分布 — 使用 diagnoser 工具查看各层占比。正常范围：System 10-25%、History 15-30%、Retrieval 30-45%。
  2. 检查历史累积 — 是否配置了滑动窗口或最大轮次限制？如果没有，这是最常见的爆窗原因。
  3. 检查摘要链 — summary bug：摘要是否丢掉了关键实体或约束？多层摘要是否失真？
  4. 检查记忆重复注入 — 长期记忆是否在每轮都被重复召回并注入？应缓存并在上下文变化时才刷新。
  5. 检查工具输出 — 是否有 API 返回了异常大量的数据？应设置输出截断。
  6. 检查双栈冲突 — Python 和 Java 是否都注入了同一会话，导致历史翻倍？
  7. 检查缓存 — trim debug：缓存是否生效？还是每次都重新召回？

每一步排查都应记录观测值（占比、轮次、配置项），而不是只记结论。这样在跨团队复盘时，运维、后端、算法能基于同一组数据对齐，而不是各说各话。

先想 10 秒：为什么「检查双栈冲突」要放在靠后步骤？

前序步骤能解释大部分单栈爆窗。只有 token 分布正常、策略已配置仍异常时，才值得怀疑 Python/Java 双写或重复注入，避免一上来就怀疑架构而忽略滑动窗口未开。

不推荐
  • 凭感觉猜测爆窗原因
  • 全量重启试试
  • 直接加大模型窗口（从 32K 到 128K）
  • 不做根因分析，只改参数
推荐
  • 使用 debug checklist 逐项排查
  • 先看 Token 分布数据
  • 找到根因后针对性修复
  • 修复后验证并加监控

常见面试题及回答要点

「对话长了 LLM 变慢变蠢，怎么排查？」
按五步法：① 确认变慢（延迟从多少到多少）② 查看 token 分布（重点关注 History 占比）③ 检查截断策略（是否配置了滑动窗口）④ 检查记忆重复注入（是否每轮都召回长期记忆）⑤ 修复并验证。通常根因是历史无限堆积或检索重复注入。
「为什么加大模型窗口不是好方案？」
加大窗口只是延缓爆窗，成本线性增加：32K→128K 价格约 4x。且研究表明 LLM 在超长上下文中注意力衰减严重（Lost in the Middle）。更好的方案是优化上下文管理：截断、压缩、缓存。
「双栈环境下如何排查爆窗问题？」
双栈环境中，常见爆窗原因是 Python 和 Java 分别注入各自的会话历史，导致总历史翻倍。排查方法：检查 Redis 中 session key 的存储内容，确认是否只有一端在写入会话。
「如何向业务解释『不是模型变差』？」
用 token 分布图说明：System 被挤压后角色约束失效，表现为「格式乱了」而非模型能力退化。修复上下文管理后质量可恢复，无需盲目换更大模型。

排障完成后应留下三类证据：修复前 token 分布截图、配置 diff（滑动窗口/压缩阈值）、修复后同会话回放对比。面试口述时用这三类证据支撑「根因—修复—预防」闭环。

先想 10 秒：History 占比 44% 时，为什么优先查滑动窗口而不是换 128K 模型？

换大窗口只延缓爆窗且成本线性上升；44% History 说明管理策略缺失。滑动窗口或摘要链通常可在不改模型的情况下把 History 压回 30% 以内。

信号可能根因优先动作
P95 延迟突增总 token 接近上限跑 diagnoser，看各层占比
输出 JSON 破格式System 层 <10%保护 System 预算 + 压缩 History
回答重复旧事实长期记忆每轮重复注入记忆召回去重 + 缓存
双端行为不一致session key 或双写统一 key、单一写入方

本章结论

  • 真实案例：20 轮后 History 从 20% 涨到 44%，System 被挤压到 7%，LLM 失去角色约束。
  • 排障清单：7 步检查法从 token 分布到双栈冲突，系统化排查根因。
  • 面试模板：现象 → 采集 → 定位 → 修复 → 预防，五步法组织回答。
  • 监控告警：token 分布 > 90% 告警，history > 35% 告警，system < 10% 紧急告警。
  • Key Takeaway：爆窗不是模型问题，是上下文管理问题。

操作步骤

判断练习

  1. 诊断显示 History 层占比持续 >60%，面试时你第一步会做什么？

    参考答案

    用 diagnoser 分层统计 Token 占比，确认是截断未生效还是未压缩/摘要，再提滑动窗口或摘要链方案——而非直接换更大模型。

  2. 爆窗时提议「升级到 200K 窗口模型」为何通常是下策？

    参考答案

    成本与延迟线性恶化，且不解决检索噪声与注入面扩大。应先预算分配、截断/压缩与懒加载。
