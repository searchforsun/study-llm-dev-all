为什么要学本章

截断是「扔掉」内容，压缩是「提炼」内容。当对话历史较长时，压缩能够在保留关键信息的前提下大幅减少 token 占用。本章学习 context compress 的核心技术，并与 Spring AI Alibaba 的生产级压缩能力对照。

学完你能
  • 设置合理的 compress timing 和 token threshold
  • 用 LLM 实现上下文摘要压缩（llm summary）
  • 对比 Python 手工压缩与 Spring AI Alibaba 压缩 Advisor
  • 为 CorpAssist 设计压缩策略方案
本章先记住 3 件事
  • 压缩 ≠ 截断：压缩是保留核心信息的浓缩，截断是直接丢弃。
  • 压缩有成本：每次压缩需要 LLM 调用，需平衡频次与收益。
  • SAA 提供生产级压缩：Spring AI Alibaba 内置上下文压缩 Advisor，可直接集成。

压缩时机：何时触发上下文压缩

compress timing 决定「什么时候开始压缩」。核心指标是 token threshold——当上下文 token 数达到预算的某个百分比时，由 compress trigger 触发压缩。

压缩触发与执行流程
flowchart TB
    Start[新对话轮次到达] --> Count[计算当前总 tokens]
    Count --> Check{超过 threshold?}
    Check -->|否| Skip[直接组装上下文]
    Check -->|是| Compress[执行压缩]
    Compress --> Decide{压缩方式?}
    Decide -->|LLM 摘要| LLM[LLM 生成摘要]
    Decide -->|实体提取| Extract[提取关键实体/事实]
    Decide -->|SAA Advisor| SAA[Spring AI Alibaba CompressAdvisor]
    LLM --> Replace[替换旧内容为摘要]
    Extract --> Replace
    SAA --> Replace
    Replace --> Assemble[组装最终上下文]

压缩触发流程：当 token 数超过阈值时，根据配置的压缩方式执行不同策略。

压缩时机策略触发条件优点缺点
固定轮次每 N 轮执行一次可预测，便于调试可能提前压缩或压不够
Token 阈值token 使用量 > 预算 × threshold%精准，只在必要时触发每次请求需计算 token
混合触发每 N 轮 + token 阈值双条件既有规律又避免爆窗逻辑稍复杂
先想 10 秒：压缩与截断在生产上最容易混淆的误区是什么？

截断直接丢弃轮次；压缩应保留约束与实体。若压缩后约束丢失，优先检查摘要 prompt 是否白名单了格式与合规句。

CorpAssist · 压缩时机决策 CorpAssist 客服系统使用 32K 窗口，设定 Retrieval 层占 40%。当 History 层 token 使用超过其预算的 80% 时触发压缩。如果一轮工具调用返回了大量数据，总 token 可能骤升，压缩触发机制需要快速响应。
python 
# Token 阈值触发压缩
class CompressionTrigger:
    def __init__(self, budget: int, threshold_ratio: float = 0.8):
        self.budget = budget
        self.threshold = int(budget * threshold_ratio)

    def should_compress(self, current_tokens: int) -> bool:
        return current_tokens > self.threshold

    def decide_target(self, current_tokens: int) -> int:
        """计算需要压缩到的目标 token 数"""
        target = int(self.budget * 0.5)  # 压缩到预算的 50%
        print(f"Need compress: {current_tokens} -> {target}")
        return target

LLM 摘要压缩：提炼式上下文压缩

llm summary 使用 LLM 将多轮对话压缩为精炼摘要。与简单截断不同，LLM 摘要能保留关键实体、意图和约束。

压缩前后 token 占用示意
flowchart TB
    Before[压缩前 History\n10 轮 ≈ 12K tokens]
    Trigger{token ≥ 80%?}
    After[压缩后\n摘要块 ≈ 3K tokens]
    Before --> Trigger
    Trigger -->|是| LLM[LLM/Rules 摘要]
    LLM --> After
    Trigger -->|否| Before
    After --> Verify[实体清单交叉验证]

触发压缩后 token 显著下降，但须做实体保留校验（见 basics-03 lab）。

压缩用 Prompt 设计

python 
# LLM 摘要压缩实现
SUMMARY_PROMPT = """请压缩以下对话历史，保留关键信息：
- 用户身份和偏好（如会员等级、部门）
- 已解决的问题和待办事项
- 关键实体（订单号、用户 ID、日期）
- 用户明确表达的约束（语言、格式要求）

对话历史：
{conversation_text}

压缩后的摘要（100 字以内）："""

class LlmContextCompressor:
    def __init__(self, llm_client):
        self.llm = llm_client

    def compress(self, conversations: list, max_tokens: int) -> str:
        text = "\n".join(
            f"用户: {c['user']}\n助手: {c['assistant']}"
            for c in conversations
        )
        prompt = SUMMARY_PROMPT.format(conversation_text=text)
        summary = self.llm.complete(prompt)
        return summary

    def compress_with_entities(self, conversations: list) -> dict:
        """压缩并同时提取结构化实体"""
        prompt = """从对话中提取结构化信息并生成摘要。
摘要保持 100 字以内。
实体提取输出 JSON 格式。"""
        result = self.llm.complete(prompt)
        return {"summary": result, "entities": {}}

Java 实现：Spring AI 压缩

java 
// 基于 LLM 的上下文压缩
import org.springframework.ai.chat.model.ChatModel;

public class ContextCompressor {
    private final ChatModel chatModel;

    public ContextCompressor(ChatModel chatModel) {
        this.chatModel = chatModel;
    }

    public String compress(List<ConversationTurn> turns) {
        String prompt = buildCompressionPrompt(turns);
        return chatModel.call(prompt);
    }

    private String buildCompressionPrompt(List<ConversationTurn> turns) {
        StringBuilder sb = new StringBuilder();
        sb.append("压缩以下客服对话，保留用户身份、偏好、已办事项和未解决问题：\n\n");
        for (ConversationTurn t : turns) {
            sb.append("用户: ").append(t.user()).append("\n");
            sb.append("助手: ").append(t.assistant()).append("\n");
        }
        sb.append("\n压缩摘要（150字内）：");
        return sb.toString();
    }
}

quality risk：压缩信息丢失

LLM 摘要压缩的质量受 summary prompt 设计影响。常见的质量风险：

  • 约束丢失：用户的语言偏好、格式要求可能在压缩中被忽略
  • 事实错误：LLM 可能在摘要中「脑补」不存在的信息
  • 时效性丢失：时间敏感的指令（如「今天下午 3 点前处理」）可能被泛化
不推荐
  • 不做 compression prompt 设计，直接用通用摘要
  • 对摘要再做摘要超过 2 层
  • 压缩后不验证关键实体是否保留
  • 每轮都触发压缩，过度压缩降低质量
推荐
  • 设计结构化 summary prompt，明确要求保留实体和约束
  • 限制压缩嵌套 ≤ 2 层
  • 压缩后提取实体清单与原文交叉验证
  • 在 token 阈值 + 固定轮次双条件下触发

SAA ContextCompressAdvisor 使用示例

示意代码：下列类名与 builder 链为教学示意，集成时请以 Spring AI Alibaba 官方文档 为准。进阶对照见 Spring AI Alibaba 上下文工程。

java 
// Spring AI Alibaba 上下文压缩 Advisor
import com.alibaba.cloud.ai.advisor.ContextCompressAdvisor;

@Configuration
public class CorpAssistCompressConfig {

    @Bean
    public ContextCompressAdvisor compressAdvisor() {
        return ContextCompressAdvisor.builder()
            .withCompressStrategy("summary")     // 摘要模式
            .withTokenThreshold(0.8f)             // 80% 触发
            .withMaxTokens(32000)
            .withKeepSystemPrompt(true)           // 保留 system prompt
            .withAuditEnabled(true)               // 开启审计日志
            .build();
    }
}

compress compare 提供了 Python 和 Java 双栈对照：Python 侧适合快速原型和自定义策略，Java/SAA 侧适合生产环境集成。

常见误区：压缩不能替代好的 budget split

压缩是预算超支时的止损手段，不是预算设计的替代。合理的 budget split 设计能减少压缩频次，保持上下文质量。

常见问题（FAQ）

压缩和截断应该配合使用吗？
是的。推荐流程：先尝试压缩（保留信息），压缩后仍超预算则截断（丢弃最早）。两层策略形成保护。
SAA 的压缩 Advisor 支持多轮压缩吗？
支持。SAA 的 ContextCompressAdvisor 会在每轮对话前后检查 token 使用量，自动决定是否执行压缩。压缩历史也通过 audit 记录可查。

本章结论

  • 压缩时机：Token 阈值（推荐 80%）是最精准的触发方式，也可配合固定轮次。
  • LLM 摘要压缩：通过精心设计的 summary prompt，在保留关键信息的前提下压缩上下文。
  • SAA 对照：Python 手工压缩灵活可控，Spring AI Alibaba 内置 Advisor 方便集成。
  • 质量风险：约束丢失和事实错误是压缩的主要风险，需进行 post-compression 验证。
  • Compression Pipeline：压缩 → 验证 → 替换的三步流程保证可靠性。
