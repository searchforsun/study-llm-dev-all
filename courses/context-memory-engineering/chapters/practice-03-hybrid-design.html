读写路径：Write-through + Read-merge

read write path 定义了数据在三层之间的流动方式。

写路径：Write-through 策略

write through 指写入时同步写所有相关层次。写操作的流程：

  1. 当轮对话写入工作记忆（内存）
  2. 同时写入会话记忆（Redis），保留最近 N 轮
  3. 异步提取关键事实写入长期记忆（Milvus + DB）
python 
# 写穿透实现
class HybridMemoryWriter:
    def __init__(self, session_store, long_term_store):
        self.session = session_store    # Redis
        self.long_term = long_term_store  # Milvus + DB

    def write_turn(self, user_msg: str, ai_msg: str,
                   user_id: str, tenant: str):
        # 1. 写入会话记忆（同步）
        self.session.save_turn(tenant, user_id, user_msg, ai_msg)

        # 2. 提取事实并写入长期记忆（异步）
        self._async_extract_and_store(user_msg, ai_msg, user_id)

    def _async_extract_and_store(self, user_msg, ai_msg, user_id):
        """异步提取并存储长期记忆"""
        facts = fact_extractor.extract(user_msg, ai_msg)
        for fact in facts:
            fact.user_id = user_id
            self.long_term.upsert_fact(fact, embed(fact.content))
java 
// 写穿透 - Java 实现
@Service
public class HybridMemoryService {
    private final ChatMemory sessionMemory;
    private final LongTermMemory longTermMemory;
    private final FactExtractor factExtractor;

    @Async
    public CompletableFuture<Void> writeTurn(
            String tenant, String userId,
            String userMsg, String aiMsg) {
        // 同步写入会话
        sessionMemory.add(
            buildSessionKey(tenant, userId),
            List.of(
                new UserMessage(userMsg),
                new AiMessage(aiMsg)
            )
        );

        // 异步提取并写入长期事实
        return CompletableFuture.runAsync(() -> {
            var facts = factExtractor.extract(userMsg, aiMsg);
            facts.forEach(fact -> {
                fact.setUserId(userId);
                longTermMemory.saveFact(fact, embed(fact.getContent()));
            });
        });
    }
}

读路径：Read-merge 合并

读取时，mem router 决定从哪些层次获取信息并合并去重：

python 
# 读合并 - 三层信息组合
class HybridMemoryReader:
    def __init__(self, session_store, long_term_store):
        self.session = session_store
        self.long_term = long_term_store

    def read_context(self, user_id: str, query: str) -> dict:
        """读取三层记忆并合并为上下文"""

        # 1. 会话层：最近对话历史
        history = self.session.load_history(user_id)

        # 2. 长期层：相关事实
        facts = self.long_term.recall(user_id, query, top_k=3)

        # 3. 去重合并（按事实内容 hash 去重）
        seen = set()
        combined = []
        for item in history + facts:
            h = hash(item.get("content", str(item)))
            if h not in seen:
                seen.add(h)
                combined.append(item)

        return {
            "history": history,
            "long_term_facts": facts,
            "combined_count": len(combined)
        }

一致性策略

consistency 在三层之间可能面临数据不一致的问题。主要场景和解决策略：

不一致场景原因影响解决策略
会话中刚修正的事实长期记忆尚未同步LLM 使用旧事实会话层覆盖长期层
长期记忆过期TTL 到期但未标记召回无效事实查询时过滤 expires_at
双栈同时写入Python 和 Java 并发数据覆盖最后写入者获胜 + 版本号
不推荐
  • 三层各自独立，无写入协调
  • 长期记忆从不验证是否与当前会话矛盾
  • 无版本号，随机覆盖
  • 读取时不做去重，信息重复注入上下文
推荐
  • 写穿透（write-through）保证三层同步
  • 会话层优先级高于长期层
  • 版本号 + 最后写入者获胜
  • 读取时去重合并

sync policy 决定了异步同步的频率和时机：

  • 即时同步：会话层写入后立即触发长期记忆提取（推荐）
  • 批量同步：累积 N 轮后批量提取事实（适合吞吐量高的场景）
  • 定时同步：固定时间窗口执行同步（适合非实时场景）
检查理解：会话层为什么应该覆盖长期层？

用户最新表达的事实优先级最高。如果用户在对话中说「我之前说错了，我其实不是经理」， 会话层记录了修正，但长期层可能还存着「经理」的旧事实。会话层覆盖长期层保证最新表述优先。

常见问题（FAQ）

三层记忆每次请求都要读全部三层吗？
不是。mem router 根据请求类型决定。简单查询只需要工作记忆和会话；需要用户知识的查询才触发长期记忆召回。
异步写入长期记忆失败了怎么办？
使用重试机制（最多 3 次）。3 次后仍失败则记录失败日志，由定时补偿任务重试。不影响用户当前请求。

本章结论

  • 三层分工：working session long 分别负责当轮/会话/长期记忆。
  • 写穿透：写入时同步写会话层，异步写长期层，保证数据不丢失。
  • 读合并：读取时从多层获取信息，去重合并，避免冗余。
  • 一致性保障：会话层覆盖长期层 + 版本号，处理冲突。
  • Mem Router：智能路由决定哪些请求需要访问哪些记忆层。
