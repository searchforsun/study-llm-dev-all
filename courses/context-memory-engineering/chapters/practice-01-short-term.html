为什么要学本章

工作记忆是 LLM 应用的「短期记忆」——它确保多轮对话中用户不必重复上下文。双栈（Python + Java）实现中，会话 key 必须一致，否则用户换栈丢状态。本章实战 short term 记忆的双栈实现。

学完你能
  • 使用 LangChain 的 langchain memory 实现会话记忆
  • 使用 Spring AI 的 spring chatmemory 实现会话记忆
  • 设计统一 redis session key 模式，保障双栈一致性
  • 实现 Redis 作为共享会话存储
本章先记住 3 件事
  • 工作记忆生命周期短：通常 TTL 24h，按 session 隔离。
  • 双栈 session key 必须一致：Python 和 Java 使用相同 key 模式才能共享会话。
  • Redis 是自然选择：快速、支持 TTL、双栈客户端成熟。

双栈共享同一 Redis 会话存储，通过统一 key 模式实现会话连续性。

Python：LangChain 会话记忆实现

每轮对话通过 turn-store 写入 Redis，与窗口截断策略配合控制会话层大小。

python 
# LangChain 会话记忆 - CorpAssist 客服场景
from langchain.memory import ConversationBufferWindowMemory
from langchain_community.chat_message_histories import RedisChatMessageHistory

class CorpAssistSessionMemory:
    def __init__(self, tenant: str, user_id: str, redis_url: str):
        self.session_key = f"corpassist:session:{tenant}:{user_id}"
        self.history = RedisChatMessageHistory(
            url=redis_url,
            session_id=self.session_key,
            ttl=86400  # 24 小时
        )

        self.memory = ConversationBufferWindowMemory(
            chat_memory=self.history,
            k=8,  # 保留最近 8 轮
            return_messages=True
        )

    def add_turn(self, user_msg: str, ai_msg: str):
        self.memory.chat_memory.add_user_message(user_msg)
        self.memory.chat_memory.add_ai_message(ai_msg)

    def load_context(self) -> list:
        return self.memory.load_memory_variables({})["history"]
CorpAssist · 客服会话连续性 用户首次对话：「帮我查订单 20240501 的状态」。5 分钟后用户说：「现在能取消吗？」。如果没有工作记忆，LLM 不知道「它」指的是哪个订单。使用 session key `corpassist:session:acme_corp:user_9527` 存储 Redis，LLM 从历史中获知上下文。

Spring ChatMemory：Java 会话记忆

spring chatmemory 是 Spring AI 提供的会话记忆接口，支持多种后端实现。

Java：Spring AI ChatMemory 实现

示意代码：与《Spring AI 工程》practice-04-chat-memory 对齐，使用 MessageWindowChatMemory.builder() + ChatMemory.CONVERSATION_ID；Redis 持久化可自定义 ChatMemoryRepository。Lab 最小契约见 demos/practice-01-short-term-lab/。

生产环境将 ChatMemoryRepository 落到 Redis 时，序列化格式须与 Python LangChain 侧一致（本课 lab 使用 corpassist-session-v1 JSON）。键名统一为 corpassist:session:{tenant}:{user_id}，避免 BFF 与 Python 记忆服务各用一套 key 导致「用户明明聊过，换栈后像新会话」。

窗口策略上，Java 侧 maxMessages(16) 约等于 8 轮完整对话，与 Python ConversationBufferWindowMemory(k=8) 对齐；超出时由后续章节的截断/压缩 Advisor 接管，本章只保证会话状态可共享、可过期、可限长。

// Spring AI ChatMemory — CorpAssist 客服（示意，以官方文档为准）
@Configuration
public class CorpAssistChatMemoryConfig {

    @Bean
    public ChatMemory chatMemory() {
        return MessageWindowChatMemory.builder()
            .maxMessages(16)  // 约 8 轮 user+assistant
            .build();
    }

    @Bean
    public MessageChatMemoryAdvisor memoryAdvisor(ChatMemory chatMemory) {
        return MessageChatMemoryAdvisor.builder(chatMemory).build();
    }
}

@Service
public class ChatService {
    private final ChatClient chatClient;

    public String chat(String tenant, String userId, String message) {
        String conversationId = String.format(
            "corpassist:session:%s:%s", tenant, userId);
        return chatClient.prompt()
            .advisors(a -> a.param(ChatMemory.CONVERSATION_ID, conversationId))
            .user(message)
            .call()
            .content();
    }
}

Session Key 模式设计

dual session 要求双栈使用完全相同的 key 模式，才能共享会话状态：

yaml 
# 统一 session key 约定
session:
  key_pattern: "corpassist:session:{tenant}:{user_id}"
  redis_key_example: "corpassist:session:acme_corp:user_9527"
  ttl_hours: 24
  max_turns: 8
  store: redis
  python_adapter: "RedisChatMessageHistory"
  java_adapter: "RedisChatMemory"
不推荐
  • Python 和 Java 用不同的 session key 模式
  • 无 TTL 设置，会话永久存储
  • 窗口无限，不限制保留轮次
  • 同一用户跨服务使用不同 session ID
推荐
  • 统一 `corpassist:session:{tenant}:{user}` key 模式
  • 设置 24h TTL，自动过期清理
  • 限制保留 8 轮，超出时压缩或截断
  • 使用同一 session ID 贯穿所有服务

Redis 共享会话存储

redis session 作为双栈共享存储，提供高性能的会话读写和自然的 TTL 支持。

python 
# Redis 会话存储操作
import redis
import json
from datetime import datetime

class RedisSessionStore:
    def __init__(self, redis_url: str):
        self.client = redis.from_url(redis_url)
        self.key_prefix = "corpassist:session:"
        self.ttl = 86400  # 24h

    def _key(self, tenant: str, user_id: str) -> str:
        return f"{self.key_prefix}{tenant}:{user_id}"

    def save_turn(self, tenant: str, user_id: str,
                  user_msg: str, ai_msg: str):
        key = self._key(tenant, user_id)
        turn = {
            "user": user_msg,
            "assistant": ai_msg,
            "timestamp": datetime.now().isoformat()
        }
        self.client.rpush(key, json.dumps(turn))
        self.client.expire(key, self.ttl)
        # 限制长度
        if self.client.llen(key) > 8:
            self.client.lpop(key)

    def load_history(self, tenant: str, user_id: str) -> list:
        key = self._key(tenant, user_id)
        turns = self.client.lrange(key, 0, -1)
        return [json.loads(t) for t in turns]
检查理解：为什么双栈 session key 必须一致？

如果 Python 用 `py:session:{user}`、Java 用 `java:session:{user}`，当用户从 Python 端发起对话后切换到 Java 端，系统会创建新的空会话，用户需要重新描述上下文。统一 key 模式保证用户任一端都能继续之前的对话。

常见问题（FAQ）

Redis 会话的 TTL 如何设置？
推荐 24h。每次写入时重新设置 TTL，保证活跃会话不被自动清理。非活跃 24h 后自动过期。
会话超长怎么办？
限制最大轮次（如 8 轮），超出时压缩最早轮次或触发摘要。可以在写入 Redis 时做长度检查。
Java 与 Python 谁负责写 Redis？
推荐单一写入方：BFF（Java）聚合用户可见轮次后写入；Python 记忆服务只读或通过内部 API 追加，避免双写导致历史翻倍。

会话状态变更（清空、切换租户、登出）必须同步失效 Redis key 与进程内 ChatMemory 缓存，否则会出现「UI 已新会话、模型仍带旧历史」的隐性 bug。

压测时建议同时观测 Redis 内存占用与单 key 长度，避免「轮次限制失效」导致单用户 key 膨胀拖垮集群。

S2 客服场景：指代消解与人工转接

CorpAssist 智能客服（S2）的两类高频记忆场景：指代消解（「它」「那个订单」）与人工转接（bot → 人工坐席时携带会话摘要）。工作记忆质量直接决定这两类体验。

轮次用户系统需记住失败表现
1查订单 20240501 状态实体 order=20240501—
2它还能加急吗？「它」→ 20240501无 session 则答「请问哪个订单」
3我要投诉，转人工触发 handoff + 摘要人工看不到前两轮则重复询问
S2 · 人工转接与会话 handoff
sequenceDiagram
    participant U as 用户
    participant Bot as CorpAssist Bot
    participant Redis as Redis Session
    participant Agent as 人工坐席

    U->>Bot: 查订单 20240501 / 它还能加急吗

    U->>Bot: 转人工客服
    Bot->>Redis: 读取最近 8 轮 + 实体清单

转接时从 Redis 会话生成结构化 handoff 摘要，避免人工重复提问；session key 与 bot/坐席控制台共用。

Mini-case · 指代消解 用户第 2 轮只说「取消它」——若 load_history 返回空，模型无法绑定实体。修复：保证 Redis 至少保留含订单号的 user turn；评测时用 must_remember: ["20240501"] 断言（见 practice-05 eval_scenarios）。
Mini-case · 人工转接 触发转接时生成 handoff 块：entities（订单/会员等级）+ last_intent（加急/投诉）+ constraints（中文回复）。禁止把完整 20 轮原文塞给坐席——应走与压缩章相同的摘要策略，并保留可点击的 session 链接。
不推荐
  • 每轮只把当前 user 消息送 LLM，不带 history
  • 转人工时复制粘贴整段对话给坐席
  • bot 与坐席使用不同 session id
推荐
  • Redis 保留最近 N 轮 + 关键实体 pinned
  • handoff 摘要结构化（实体/意图/约束）
  • 统一 corpassist:session:{tenant}:{user}

本章结论

  • LangChain Memory：Python 侧使用 ConversationBufferWindowMemory + RedisChatMessageHistory 实现会话记忆。
  • Spring ChatMemory：Java 侧使用 RedisChatMemory + MessageChatMemoryAdvisor 实现。
  • 统一 Session Key：`corpassist:session:{tenant}:{user_id}` 保障双栈会话连续性。
  • Redis 共享存储：TTL 24h、最大 8 轮、高性能读写。
  • 工作记忆是基础：没有会话记忆，LLM 无法维持多轮对话的上下文连续性。

判断练习

  1. Python 与 Java 使用相同 Redis key，但消息 JSON 字段名不一致（如 content vs text），双栈互读会怎样？

    参考答案

    反序列化失败或读到空消息，表现为会话断层。须约定统一消息 schema 与单一写入方，或加兼容层。

  2. 两栈同时对 corpassist:session:acme_corp:user_9527 写入，最可能引发什么问题？

    参考答案

    双写竞态：后写覆盖先写、轮次乱序。生产应指定单一写入方，另一栈只读或通过 BFF 转发。
