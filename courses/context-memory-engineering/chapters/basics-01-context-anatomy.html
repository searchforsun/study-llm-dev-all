为什么要学本章

LLM 上下文窗口不是无限缓存——它是有限、昂贵且共享的资源。爆窗的根本原因往往不是模型不够强，而是预算从未被分配。先量化再裁剪，是所有上下文工程的起点。本章用 CorpAssist 客服场景带你理解 context anatomy 的每一层。

本章先记住 3 件事
  • 上下文分四层：System / Tool Output / Conversation History / Retrieved Chunks，每层占固定 budget split。
  • 预算要显式分配：不分配预算 = 让 LLM 自己抢窗口，结果通常是历史爆窗、system 被挤掉。
  • CorpAssist 模板是起点：固定 system + 动态检索 + 滑动历史 = 可复用的上下文工程模式。

四层上下文结构：System / Tool / History

每个 LLM 请求的上下文窗口由四类内容组成，它们在窗口中所占比例和稳定性各不相同。理解这四层是 context anatomy 的核心。

层级内容稳定性典型占比优化方向
System Prompt 角色定义、行为约束、输出格式 固定，每请求写入 15-25% 精炼指令，去冗余
Tool Outputs API 返回、DB 查询、代码执行结果 动态，每次调用产生 10-20% 结果精简、截断长字段
Conversation History 用户/助手多轮对话 累积增长 20-30% 摘要、滑动窗口
Retrieved Chunks RAG 检索的文档片段 动态，每轮检索 30-45% 块大小、相关性过滤

上下文窗口的四大组成及其数据来源。预算分配决定了每个层级的最大 Tokens。

CorpAssist 的 System Prompt 遵循 system tool history 分层原则：固定角色描述始终占据第一部分，Tool 输出注入到具体位置，历史对话通过 sliding window 裁剪后追加。

常见误区
  • System Prompt 越长越好 — 超过 2K 字后逐字遵守率下降；精炼是关键。
  • 历史轮次全部保留 — 20 轮对话约 8K tokens，挤占检索空间。
  • Tool 输出未经截断 — 日志原文可能有数万行，压缩后再注入。
先想 10 秒：如果 32K 窗口中 history 占了 25K，会发生什么？

System Prompt 和检索结果会被挤出可用窗口。LLM 在 token 压力下会丢失角色约束，回答变得不可控。这是最常见的爆窗表现。

Token 预算分配：策略与比例设计

token budget 是对上下文窗口的显式资源规划；window budget 则是模型硬上限下的总配额。budget split 决定了每一层能使用多少 tokens。没有预算规划，上下文就会进入「谁抢到是谁的」的无序状态。

CorpAssist · 企业客服窗口预算 CorpAssist 使用 32K 模型处理企业客服。用户询问「我的订单发货了吗」，系统需同时注入：客服角色指令（System）、用户企业信息（Retrieval）、最近 5 轮对话（History）、订单 API 返回值（Tool）。设计一个合理的分配比例。

推荐预算配比

层级T okens百分比配置策略
System Prompt ~4,800 15% 固定 1500 字以内的角色与约束
Retrieval Chunks ~12,800 40% Top-5 chunk，每 chunk ~2560 tokens
History ~6,400 20% 最近 5-8 轮 + 摘要
Tool Outputs ~4,800 15% 结果精简，限制字段数
Reserve ~3,200 10% 模型 output + 缓冲

Python：Token 预算计算

python 
# Token 预算计算器 - CorpAssist 场景
MAX_TOKENS = 32000

budget = {
    "system": int(MAX_TOKENS * 0.15),    # 4800
    "retrieval": int(MAX_TOKENS * 0.40), # 12800
    "history": int(MAX_TOKENS * 0.20),   # 6400
    "tool_output": int(MAX_TOKENS * 0.15), # 4800
    "reserve": int(MAX_TOKENS * 0.10),   # 3200
}

def validate_budget(used: dict) -> bool:
    total = sum(used.values())
    if total > MAX_TOKENS:
        overflow = total - MAX_TOKENS
        print(f"Budget overflow: {overflow} tokens over limit")
        return False
    print(f"Total: {total}/{MAX_TOKENS} tokens used ({total/MAX_TOKENS*100:.1f}%)")
    return True

# 验证
print(budget)
validate_budget(budget)

Java：Spring AI Token 预算配置

java 
// Token 预算配置 - CorpAssist 客服场景
public class TokenBudget {
    public static final int MAX_TOKENS = 32000;

    public record BudgetAllocation(
        int system, int retrieval, int history,
        int toolOutput, int reserve
    ) {
        public BudgetAllocation {
            int total = system + retrieval + history + toolOutput + reserve;
            if (total > MAX_TOKENS) {
                throw new IllegalArgumentException(
                    "Budget overflow: " + (total - MAX_TOKENS) + " tokens over limit");
            }
        }

        public static BudgetAllocation defaultAllocation() {
            return new BudgetAllocation(
                (int)(MAX_TOKENS * 0.15),   // system: 4800
                (int)(MAX_TOKENS * 0.40),   // retrieval: 12800
                (int)(MAX_TOKENS * 0.20),   // history: 6400
                (int)(MAX_TOKENS * 0.15),   // tool: 4800
                (int)(MAX_TOKENS * 0.10)    // reserve: 3200
            );
        }

        public double usagePercent() {
            int used = system + retrieval + history + toolOutput;
            return (double) used / MAX_TOKENS * 100;
        }
    }
}
不推荐
  • 无预算规划，所有内容堆到一个 prompt
  • 历史对话无上限累积
  • 检索 chunk 不限制数量，一次性注入 20+ 片段
  • System prompt 被历史挤压到窗口尾部
推荐
  • 显式分配每层 token 上限，使用 alloc ratio
  • 历史对话使用滑动窗口裁剪
  • 检索 chunk 限制 Top-K 并做相关性过滤
  • System prompt 放在上下文最前段保障优先级
常见误区：预算不是越小越好

预算压缩过度会导致 LLM「失忆」——System 约束被裁剪、检索信息不足。合理的预算分配是保留核心结构，压缩冗余表达。每层都留出 10-15% 冗余空间。

CorpAssist 模板：从预算到实现

corpassist template 是将预算分配落地为具体 prompt 结构的模板框架。它将上下文组织为固定部分（System）和动态部分（Retrieval + History + Tool），确保每次请求都在预算内。

模板按 System → Retrieval → History → Tool 顺序组装，保障优先级。

模板示例：CorpAssist 客服 System Prompt

yaml 
corpassist_template:
  # CorpAssist 客服上下文模板
  version: "1.0"
  max_tokens: 32000

  system_layer:
    role: |
      你是一个专业的企业客服助手 CorpAssist。
      你的知识截止于 2025 年底。
      你使用中文回答。
    constraints: |
      始终使用 {{company_name}} 的官方知识库回答。
      如果不确定，请回答「我需要查证后再回复」。
      不要编造订单号、工单号或用户信息。
    output_format: |
      回答结构：确认理解 → 查询结果 → 建议行动。
      如果涉及敏感操作，要求用户二次确认。

  retrieval_layer:
    strategy: "semantic_search"
    top_k: 5
    max_chunk_tokens: 2560
    source: "enterprise_knowledge_base"

  history_layer:
    max_turns: 5
    summary: "rolling_summary"

  tool_layer:
    query_tools:
      - order_query
      - ticket_query
      - user_info
    max_result_tokens: 1600

context layer 的动态组装逻辑

每层在运行时动态计算实际 token 数，如果超出预算，触发对应的降级策略：

  • Retrieval 超预算 → 减小 chunk size 或降低 Top-K
  • History 超预算 → 触发滚动摘要，丢弃最早轮次
  • Tool 超预算 → 截断字段（如只返回前 100 字符）
检查理解：为什么 History Layer 要放在 Retrieval Layer 之后？

System → Retrieval → History → Tool 的顺序确保最重要的约束（System）和知识（Retrieval）在上下文前段，它们的注意力衰减最小。历史对话虽重要但时效性信息通常更依赖最近几轮。

预算验证与监控

预算分配是静态设计，但实际 token 消耗是动态的。必须引入运行时监控，防止实际上下文超出预算。

监控指标告警阈值触发动作
实际总 tokens 占比 ≥ 90% 触发压缩/裁剪
History 层占比 ≥ 30% 触发滚动摘要
单次检索占比 ≥ 50% 减少 Top-K 或 chunk size
Reserve 剩余 ≤ 5% 限制 tool 调用次数
预算监控与自动 remediation
flowchart LR
    Metrics[运行时 token 分布] --> Check{超阈值?}
    Check -->|总 token ≥90%| Compress[触发压缩/裁剪]
    Check -->|History ≥30%| Summary[滚动摘要]
    Check -->|Retrieval ≥50%| TrimRAG[降低 Top-K]
    Check -->|正常| OK[继续服务]
    Compress --> OK
    Summary --> OK
    TrimRAG --> OK

常见问题（FAQ）

预算分配比例是固定的吗？每种场景都适用 15/40/20/15/10？
不是。不同场景需要不同配比。客服场景侧重检索，所以 Retrieval 占比高。Agent 场景 Tool Output 可能占更多。关键是要显式分配而不是默认不分配。建议每个场景单独设计 budget split。
Reserve 的 10% 是留给模型输出的吗？
部分是的。模型输出 token 同样消耗总窗口。Reserve 同时也作为缓存用——当某层临时超出预算时，从 Reserve 借用。但借用的需要在下一次请求中归还。
如何监控真正的 token 消耗？
使用 tokenizer 库（如 tiktoken）在每个请求组装前预计算 token 数。不要事后从模型返回中获取——那时已经来不及了。Log 每次请求的 token 分布，用于调优预算。

本章结论

上下文工程的第一步是理解窗口的组成结构并制定显式预算。

  • 四层结构：System / Tool / History / Retrieval 共同组成上下文窗口，各自有稳定的占比范围。
  • 显式预算分配：使用 budget split 为每层分配 token 上限，防止上下文无序增长。
  • 模板化组装：corpassist template 将预算分配落地为可复用模板，保障每次请求都在预算内。
  • 运行时监控：预算分配是静态设计，需要运行时 token 监控配合，触发降级策略。
  • 场景差异：不同场景（客服、Agent、RAG）需要不同的 alloc ratio，核心原则是显式分配。

判断练习

  1. 团队反馈「System Prompt 太长，LLM 经常不遵守格式约束」。你会优先调整什么？

    参考答案

    调小 System 占比 + 精简指令。System 超过 2K 后逐字遵守率下降，优先压缩到 1.5K 以内（约 15% 预算），把空间释放给 Retrieval。

  2. 32K 窗口，History 轮次太多导致爆窗。你会优先用哪三种手段？

    参考答案

    1) 限制最大轮次（如 5 轮）；2) 对旧轮次生成摘要；3) 在预算中限制 History 层最大 25%。
