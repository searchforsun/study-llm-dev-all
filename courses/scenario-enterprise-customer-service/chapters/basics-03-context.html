DST：对话状态跟踪

dst state 维护多轮对话中的结构化信息。客服场景的典型槽位：

槽位示例值类型规则校验
order_id O-2025-001 字符串 正则 ^O-\d{4}-\d{3}$
intent refund 枚举 在预定义列表中
refund_amount 299.00 数字 ≤ 订单总金额
reason 商品破损 字符串 非空
confirmed true 布尔 用户明确确认
CorpAssist · DST 幻觉改单（30 秒） 用户 A：「我要退 #O-2025-001 的订单」→ Bot 记录 order_id=O-2025-001, intent=refund。用户 A：「改地址到北京朝阳」→ LLM 把 intent 改成 change_address 但忘了清 refund_amount——执行改地址 API 时带入了之前记录的退款金额参数，API 报错。根因：LLM 提取槽位时未重置无关字段。修复：规则引擎检测到 intent 变更时，自动清理旧意图的相关槽位。
DST 双校验流程
flowchart TB
    U[用户消息] --> LLM[LLM 提取槽位]
    LLM --> R[规则校验]
    R -->|不通过| F[要求用户重新输入]
    R -->|通过| M[合并到 DST 状态]
    M -->|intent 变化| C[清理旧槽位]
    C --> S[(DST State Store)]
    M --> S

LLM 提取 → 规则校验 → 清理无关槽位 → 更新状态。

Python：DST 双校验

python 
# DST 双校验
class DSTManager:
    SLOTS = {
        "order_id": {"type": str, "pattern": r"^O-\d{4}-\d{3}$"},
        "intent": {"type": str, "enum": [
            "refund", "change_address", "order_query"]},
        "amount": {"type": float, "min": 0},
    }
    INTENT_SLOTS = {
        "refund": ["order_id", "amount", "reason"],
        "change_address": ["order_id", "new_address"],
        "order_query": ["order_id"],
    }

    def __init__(self):
        self.state = {}

    def update(self, new_slots: dict, new_intent: str = None):
        # LLM 提取的新槽位
        validated = self.validate(new_slots)
        if new_intent and new_intent != self.state.get("intent"):
            # intent 变化：清理旧槽位
            self.state = {"intent": new_intent}
        self.state.update(validated)

    def validate(self, slots: dict) -> dict:
        valid = {}
        for k, v in slots.items():
            if k in self.SLOTS:
                spec = self.SLOTS[k]
                if isinstance(v, spec["type"]):
                    if "pattern" in spec and not re.match(spec["pattern"], str(v)):
                        continue
                    if "enum" in spec and v not in spec["enum"]:
                        continue
                    valid[k] = v
        return valid

指代消解与摘要压缩

coreference 解决「这个」「那个」「它」等代词指向谁的问题。在客服系统里，指代消解通过结合 DST 状态完成——当前轮用户说「改它的地址」时，前一轮 DST 中的 order_id 就是「它」。

summary compress 当对话超过 10 轮时，对历史对话生成摘要，释放上下文窗口给最新的对话。

不推荐
  • 所有历史对话原文保留，直到爆窗
  • 只靠 LLM 记忆槽位，无持久化 DST
  • 指代消解靠 prompt 说「注意代词」
推荐
  • DST 持久化槽位 + 滑动窗口历史
  • 规则 + LLM 双校验 DST
  • 指代消解 = DST 状态查询 + LLM 确认

常见问题（FAQ）

DST 状态存在哪里？
Redis。以 session_id 为 key，每次对话更新后 TTL 重置（推荐 30 分钟）。Python Agent 和 Spring BFF 都通过 Redis 访问同一 DST 状态。
摘要压缩会丢失信息吗？
会，所以需要混合记忆——摘要 + 最近 3 轮原文 + DST 槽位三者并存。摘要丢失的信息可以通过「用户告诉我」的方式让用户补充。
用户说「刚才那个订单」怎么处理？
查 Redis 中该用户的最近 5 个 order_id，让 LLM 选择或直接询问用户。不要猜——猜错比问用户更糟。

本章结论

  • DST：结构化槽位 + 规则 + LLM 双校验，intent 变更时清理旧槽位。
  • 指代消解：通过 DST 状态 + 用户最近历史查询。
  • 摘要压缩：超 10 轮触发，摘要 + 最近 3 轮原文 + DST 并存。
  • 混合记忆：短期（Redis TTL）+ 长期（用户画像）+ 全局。
