Spring Gateway 限流与灰度

rate limit cs 按租户限流，避免单个大客户耗尽所有模型容量。CorpAssist 的限流策略：每租户 100 req/s，突发 150 req/s，超过的请求返回 429 并附带降级话术。

CorpAssist · 大促限流实战 双十一当天 CorpAssist 的 QPS 从日常 50 飙升至 800。网关层按租户限流：电商大客户限 500 req/s，普通客户限 100 req/s，超出部分返回「当前咨询量较大，请稍候再试」或转人工。Python Agent 的熔断器也在当天触发 2 次，自动降级到预设话术。

spring:
  cloud:
    gateway:
      routes:
        - id: chat-route
          uri: http://python-agent:8000
          predicates:
            - Path=/api/chat/**
          filters:
            - name: RequestRateLimiter
              args:
                key-resolver: "#{@tenantKeyResolver}"
                redis-rate-limiter:
                  replenishRate: 100   # 每秒令牌数
                  burstCapacity: 150   # 突发容量
                  requestedTokens: 1
            - name: CircuitBreaker
              args:
                name: pythonAgentCircuitBreaker
                fallbackUri: forward:/fallback/chat

    circuitbreaker:
      circuit-breakers:
        pythonAgentCircuitBreaker:
          sliding-window-size: 10
          failure-rate-threshold: 50  # 50% 请求失败触发熔断
          wait-duration-in-open-state: 30s
          permitted-number-of-calls-in-half-open-state: 3

Python 侧熔断降级

python
# circuit_breaker.py — Python 侧熔断器
import time
from enum import Enum
from typing import Callable, Any

class CircuitState(Enum):
    CLOSED = "closed"       # 正常
    OPEN = "open"           # 熔断
    HALF_OPEN = "half_open" # 半开试探

class CircuitBreaker:
    """熔断器：保护下游 AI 服务"""

    def __init__(self, failure_threshold: int = 5, recovery_timeout: float = 30.0):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time = 0.0

    def call(self, func: Callable, fallback: Callable, *args, **kwargs) -> Any:
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                print("[熔断器] 进入半开状态")
            else:
                print("[熔断器] 熔断中，执行降级")
                return fallback(*args, **kwargs)

        try:
            result = func(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                print("[熔断器] 半开试探成功，恢复关闭状态")
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            print(f"[熔断器] 失败 ({self.failure_count}/{self.failure_threshold}): {e}")
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
                print("[熔断器] 触发熔断 → OPEN")
            return fallback(*args, **kwargs)


# 降级函数
def fallback_chat(*args, **kwargs) -> str:
    return "当前服务繁忙，已为您转接人工客服，请稍候。"


cb = CircuitBreaker(failure_threshold=3)

def call_llm(message: str) -> str:
    if "fail" in message.lower():
        raise Exception("LLM API 超时")
    return f"Echo: {message}"

for msg in ["hello", "fail", "fail", "world", "fail", "fail", "fail", "test"]:
    result = cb.call(lambda: call_llm(msg), fallback_chat)
    print(f"[{cb.state.value}] {msg[:20]} → {result[:30]}")

Java 侧：SSE 连接管理

java
// SseConnectionManager.java — SSE 生命周期管理
@Component
public class SseConnectionManager {

    private final ConcurrentHashMap<String, SseConnection> connections = new ConcurrentHashMap<>();

    public SseEmitter createConnection(String sessionId) {
        SseEmitter emitter = new SseEmitter(120_000L); // 2 分钟超时
        SseConnection conn = new SseConnection(sessionId, emitter, Instant.now());
        connections.put(sessionId, conn);

        emitter.onCompletion(() -> {
            connections.remove(sessionId);
            log.info("SSE 连接完成: {}", sessionId);
        });
        emitter.onTimeout(() -> {
            connections.remove(sessionId);
            log.warn("SSE 连接超时: {}", sessionId);
        });
        emitter.onError(e -> {
            connections.remove(sessionId);
            log.error("SSE 连接错误: {}", sessionId, e);
        });
        return emitter;
    }

    public boolean send(String sessionId, String event, String data) {
        SseConnection conn = connections.get(sessionId);
        if (conn != null) {
            try {
                conn.emitter().send(SseEmitter.event().name(event).data(data));
                return true;
            } catch (IOException e) {
                connections.remove(sessionId);
                return false;
            }
        }
        return false;
    }

    public record SseConnection(String sessionId, SseEmitter emitter, Instant createdAt) {}
}

// TenantKeyResolver.java — 租户限流 key 解析
@Component
public class TenantKeyResolver implements KeyResolver {

    @Override
    public Mono<String> resolve(ServerWebExchange exchange) {
        String tenantId = exchange.getRequest().getHeaders()
            .getFirst("X-Tenant-Id");
        if (tenantId == null) {
            tenantId = exchange.getRequest().getRemoteAddress()?.getHostString() ?? "unknown";
        }
        return Mono.just(tenantId);
    }
}

// FallbackController.java — 降级处理
@RestController
@RequestMapping("/fallback")
public class FallbackController {

    @PostMapping("/chat")
    public Mono<ResponseEntity<FallbackResponse>> chatFallback(ServerWebExchange exchange) {
        String reason = "AI 服务繁忙";
        return Mono.just(ResponseEntity.ok(
            new FallbackResponse("当前服务繁忙，已为您转接人工客服。", "HANDOFF", reason)));
    }
}

先想 10 秒：熔断器的半开状态为什么只允许 3 次试探调用？

试探调用太多（如 10 次）可能在服务未恢复时重复触发熔断；试探太少（如 1 次）可能因偶然成功而错误恢复。3 次是实践中的平衡值：1 次成功不能说明恢复，3 次中成功 2 次则认为恢复，统计上更可靠。

不推荐
  • 无限流，大模型 API 被一个客户打爆
  • 无熔断，模型挂了导致全系统 5xx
  • SSE 连接永不超时，泄漏连接资源
  • 值班无 SLA，告警没人处理
推荐
  • 按租户限流 100 req/s + 突发 150
  • 50% 失败率熔断，30s 后半开试探 3 次
  • SSE 120s 超时 + 3 次重连 + 资源清理
  • P0/P1/P2 三级值班 SLA

7×24 值班与告警

oncall cs 定义三级值班 SLA。sla 724 要求客服系统全年可用性 ≥99.9%（月故障时间 ≤43 分钟）。

等级定义响应定位止损
P0系统不可用（全部用户无法使用）5min15min30min
P1功能严重受损（如转人工不可用）15min30min2h
P2非关键功能异常（如 CSAT 评分不记录）2h4h下一工作日

主要风险点：模型 API 限流/超时（占 60% 故障）、Redis 故障（20%）、ERP 接口不可用（15%）、其他（5%）。

常见问题（FAQ）

限流阈值设多少合适？
取决于模型 API 配额和用户量。CorpAssist 的实践：每租户 100 req/s 是安全线，对应 gpt-4o-mini 的 TPM 配额（~200K TPM）。如果使用更小的模型（qwen-turbo），限流可放宽到 500 req/s。建议从低开始逐步上调，观察 P99 延迟和 API 限流率。
熔断后用户看到什么？
熔断后网关返回降级响应。CorpAssist 的做法：非紧急场景（FAQ）→ 返回「系统繁忙，请稍后重试」；紧急场景（投诉/退款）→ 直接转人工，坐席侧标注「AI 服务熔断」方便排障。熔断恢复后自动切回正常模式。
SSE 断线重连怎么做？
前端用 EventSource API 原生支持自动重连。后端推荐：1) SseEmitter 设置 120s 超时；2) 发送心跳包每 30s 一次保活；3) 断线后前端重新建立 SSE 连接，携带 lastEventId，后端从断点续传。SSE 协议标准支持 Last-Event-ID 头。

本章结论

  • 限流：100 req/s/tenant，突发 150，Redis Token Bucket 实现。
  • 熔断：50% 失败率触发，30s 后半开试探 3 次。
  • SSE Ops：120s 超时，心跳 30s，Last-Event-ID 续传。
  • 值班：P0 5min/15min/30min 三级 SLA。
  • 可用性：99.9% SLA，月故障 ≤43 分钟。
