政策库 RAG

policy rag 将企业的退款政策、运费说明、退换货规则等构建为向量知识库。用户咨询时，检索相关政策片段注入到 LLM 上下文中。

政策 RAG 检索流程
flowchart LR
    U[用户提问] --> Q[向量化]
    Q --> VS[(政策向量库)]
    VS -->|Top-3 片段| P[Prompt 组装]
    P --> LLM[LLM 回答]
    LLM --> C[引用核对]
    C --> R[最终回答 + 政策引用]

政策 RAG 确保 Bot 回答基于企业政策而非模型幻觉。

refund policy 是客服最高频的 RAG 查询。政策文档的切片策略：

  • 按商品品类切片：护肤品类退换货政策、食品类退换货政策不同。
  • 按场景切片：质量问题 vs 七天无理由 vs 赠品问题。
  • 元数据标注：生效日期、适用渠道（线上/线下）、特殊说明。

Python：政策 RAG 检索

python 
# 政策 RAG 检索
class PolicyRAG:
    def __init__(self, vector_db):
        self.db = vector_db

    def query(self, question: str, top_k: int = 3) -> list:
        query_emb = self.db.embed(question)
        results = self.db.search(query_emb, top_k)
        return [
            {"content": r.content,
             "source": r.metadata.get("policy_id"),
             "effective_date": r.metadata.get("effective_date")}
            for r in results
        ]

    def format_context(self, policies: list) -> str:
        parts = ["## 相关企业政策\n"]
        for p in policies:
            parts.append(f"[{p['source']}] {p['content']}\n")
        return "".join(parts)

话术模板与输出审核

script template 定义客服场景的标准话术骨架。模板保证核心信息不遗漏，LLM 只填充与用户相关的变量。

output review 对 Bot 的回复做合规检查：确保不承诺政策外的服务、不出现情绪化措辞、不泄露敏感信息。

不推荐
  • LLM 自由发挥话术，每次风格不一致
  • 无输出审核，可能承诺政策外退款
  • 敏感场景不识别，强行 Bot 回答
推荐
  • 话术模板 + LLM 变量填充
  • 输出审核规则：政策外承诺拦截
  • 敏感场景自动转人工并告警

常见问题（FAQ）

政策更新后，进行中的会话怎么处理？
进行中的会话使用「开始时的政策版本」。新会话加载最新版本。政策版本号记录在响应日志中，方便审计。
敏感场景的判定标准是什么？
三路触发：① 意图分类为 complaint/legal；② 用户消息含「投诉」「律师」「起诉」等关键词；③ CSAT 连续 2 次 1 分。任一路触发即转人工。
输出审核发现违规怎么处理？
拦截该回答，返回「抱歉，我需要确认后回复您」，同时告警运营。违规类型和拦截记录写入审计日志，用于改进 prompt 和审核规则。

本章结论

  • 政策 RAG：政策向量化检索，回答带引用。
  • 话术模板：固定骨架 + 变量填充，保证一致性。
  • 输出审核：规则拦截政策外承诺。
  • 敏感场景：自动转人工并告警。
