本章先记住 3 件事
  • KPI 先于模型：首解率 ≥85%、转人工 ≤15%、CSAT ≥4.2 是 CorpAssist S2 默认门禁，不是上线后「再看数据」。
  • 解决 ≠ 答对：first contact 解决要求任务闭环（查单、退款受理、改址成功），纯闲聊不算。
  • 双栈分工：Python Agent 负责对话与工具编排；Spring BFF 负责鉴权、会话持久化与 ERP 契约（见 corpassist cs 模块配置）。

解决率

CorpAssist · 大促售后高峰（30 秒） 用户问「我的订单 #8821 到哪了？」Bot 调用订单 API 返回物流节点并确认签收时间——用户未转人工即结束会话。这算一次首解。若 Bot 只回复「请耐心等待」而未查单，即使 CSAT 尚可，也不算解决。

resolution rate（首解率 / 一次解决率）衡量无需人工介入即完成用户诉求的会话占比。CorpAssist S2 默认目标 ≥85%。

口径计入解决不计入
任务型（查单、退款、改址） 工具返回成功 + 用户确认或无追问 30s 内结束 仅给政策说明但未执行用户要求的动作
FAQ 型（运费、时效） 政策 RAG 命中 + 用户点击「解决了」 答非所问后用户沉默离开（需结合 CSAT 抽检）
转人工后 — 整通会话从 Bot 侧计为未解决

工程侧如何支撑解决率

  • 意图路由准确：FAQ 走快路径，任务型进 Agent+工具（意图识别与路由）。
  • 工具真执行：查单/退款必须调 ERP OpenAPI，禁止模型「假装已改单」（s2 scenario 生产坑）。
  • DST 双校验：订单号、退款金额等槽位由规则校验后再调 API（多轮上下文与指代）。
先想 10 秒：用户说「谢谢」后挂断，但 Bot 从未查过订单——算解决吗？

不算。解决率看诉求是否闭环，礼貌结束不能替代任务完成。

转人工率

handoff rate 是 Bot 将会话移交人工坐席的占比。CorpAssist 目标 ≤15%——不是越低越好到 0%，而是该转才转：低置信、敏感投诉、大额退款应主动转人工，避免强行 Bot 导致 CSAT 崩盘。

转人工触发路径
flowchart TD
  U[用户消息] --> R[意图路由]
  R -->|高置信 FAQ| F[政策 RAG 快答]
  R -->|任务型| A[Agent + 工具]
  A -->|工具失败 2 次| H[转人工]
  A -->|低置信 <0.6| H
  F -->|用户点转人工| H
  H --> Q[排队] --> Agent[人工坐席 + 上下文摘要]

转人工是降级能力而非失败；关键是 warm transfer：坐席能看到 Bot 已收集的槽位与工具调用记录。

不推荐
  • 为压转人工率，低置信仍强行 Bot 回答
  • 转人工后坐席从零问订单号
  • 无排队 SLA，用户直接流失
推荐
  • 置信度 + 敏感词 + 连续失败次数多路触发
  • 移交 payload 含 DST 槽位与最近 3 轮摘要
  • 高峰时段动态扩容坐席或回调

CSAT

CSAT（Customer Satisfaction）在本课采用会话结束后的五星评分均值，目标 ≥4.2。它与解决率、转人工率共同构成 toc support 质量三角：

指标反映什么典型误读
解决率 ↑ 自动化程度与工具成功率 「刷解决」：未真执行就结束会话
转人工率 Bot 能力边界与降级策略 压到 0% 导致复杂客诉 CSAT 暴跌
CSAT 用户主观体验（态度、速度、准确性） 样本偏差：只有满意用户才评分

工程上应对 CSAT 抽样偏差：人工抽检 + 坏例进评测集（客服效果评测）。NS1 飞轮要求低分样本必须可追溯到意图、工具 Trace 与 Prompt 版本。

S2 模块 KPI 配置示例

yaml 
corpassist:
  modules:
    s2-customer-service:
      kpi:
        resolution-rate-target: 0.85
        handoff-rate-max: 0.15
        csat-target: 4.2
      routing:
        faq-model: qwen-turbo      # 低成本 FAQ
        agent-model: qwen-plus     # 任务型多轮
      tools:
        allowed: [order.lookup, refund.apply, address.update]

customer service roi 粗算：若人工单次成本 ¥8、Bot 单次 ¥0.3，首解率从 60% 提到 85% 且转人工可控，即可向业务方交付可答辩的 ROI 叙事（业务数据叙事）。

双模型路由成本对比

python 
# CorpAssist S2 双模型路由成本计算
class S2CostModel:
    def __init__(self):
        self.faq_cost_per_call = 0.025   # qwen-turbo
        self.agent_cost_per_call = 0.08  # qwen-plus
        self.human_cost_per_call = 8.0

    def monthly_cost(self, total_calls=100000,
                     faq_ratio=0.6, agent_ratio=0.3,
                     handoff_ratio=0.1) -> dict:
        faq = total_calls * faq_ratio * self.faq_cost_per_call
        agent = total_calls * agent_ratio * self.agent_cost_per_call
        human = total_calls * handoff_ratio * self.human_cost_per_call
        total = faq + agent + human

        return {
            "FAQ 模型": f"¥{faq:,.0f}",
            "Agent 模型": f"¥{agent:,.0f}",
            "人工坐席": f"¥{human:,.0f}",
            "月总成本": f"¥{total:,.0f}",
            "单次均摊": f"¥{total/total_calls:.2f}",
            "相比全人工节省": f"{1 - total/(total_calls*self.human_cost_per_call):.0%}"
        }

cost = S2CostModel()
print(cost.monthly_cost())
# 输出示例：月总成本 ¥109,500，相比全人工节省 86%

Java：Spring 指标采集

java 
// Spring KPI 指标采集
@Component
public class KpiMeter {

    private final MeterRegistry registry;

    public KpiMeter(MeterRegistry registry) {
        this.registry = registry;
    }

    public void recordResolution(String intentType, boolean resolved) {
        registry.counter("s2.resolution",
            "intent", intentType,
            "result", resolved ? "solved" : "unsolved"
        ).increment();
    }

    public void recordHandoff(String reason) {
        registry.counter("s2.handoff",
            "reason", reason
        ).increment();
    }

    public void recordCsat(String sessionId, int score) {
        registry.counter("s2.csat",
            "session", sessionId
        ).increment(score);
    }
}

KPI 三角的工程权衡

三个指标之间存在内在制约：

KPI 制约三角
flowchart LR
    A[解决率 ↑] -->|需要更多工具能力| B[转人工率 ↓]
    B -->|低置信也强答| C[CSAT ↓]
    C -->|用户不满意度涨| D[解决率 ↓]
    D -->|客诉转人工| B

    style A fill:#4A90D9,color:#fff
    style B fill:#D2691E,color:#fff
    style C fill:#7B68EE,color:#fff

三大 KPI 相互制约——解决率上升依赖工具成功率，但强行不转人工会拉低 CSAT。

不推荐：单一指标优化
  • 只看解决率，不追踪 CSAT
  • 为降转人工率，低置信也强答
  • 月度汇报只挑对团队有利的指标
推荐：看组合仪表盘
  • 解决率 + 转人工率 + CSAT 同屏展示
  • 关键指标未达标触发告警而非人工决断
  • 周维度观测趋势，月维度做优化决策

常见问题（FAQ）

解决率 85% 和模型准确率 90% 是一回事吗？
完全不是。模型准确率是「回答是否正确」，解决率是「用户诉求是否闭环」。模型正确回答了但工具调用失败，不算解决。解决率 = 正确回答 × 工具成功率 × 用户确认。
CSAT 评分怎么防止恶意刷分？
三道防线：① 仅对已完成会话弹评分（未闭环不弹）；② 同一用户 24h 内只计一次评分；③ 设定评分阈值——同一 IP 连续 5 次 1 分触发人工复核。
大促期间转人工率飙升怎么办？
先区分原因：是意图路由识别率下降（模型问题）还是用户真实需求超出 Bot 能力（业务问题）。前者优化分类器，后者动态扩容坐席并暂时放宽转人工阈值。

本章结论

  • 解决率：任务闭环才算首解；工具真执行 + DST 双校验是底座。
  • 转人工率：≤15% 是目标区间，低置信与敏感场景应主动 warm transfer。
  • CSAT：与解决率、转人工共同约束；需抽检与坏例飞轮防样本偏差。
  • CorpAssist S2：意图→对话→工具→坐席四层，Python Agent + Spring BFF 双栈落地。

判断练习

  1. 找错：团队为冲 KPI 让 Bot 在查单失败时仍回复「您的包裹正在路上，请放心」并结束会话。这会影响哪些指标？

    参考答案

    解决率虚高、CSAT 长期下降。短期可能减少转人工，但未闭环；抽检与客诉会暴露，坏例应进评测集。

  2. 权衡：大促前业务要求「转人工率再降 5%」，你更倾向加坐席、优化意图路由，还是强行压低置信度阈值？

    参考答案

    优先意图路由 + FAQ 缓存 + 小模型分流，而非降低转人工阈值。阈值过低会把复杂单困在 Bot 里，CSAT 与客诉风险更大。
