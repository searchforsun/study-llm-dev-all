本章先记住 3 件事
  • 规则优先、模型兜底：关键字规则先匹配（命中率 ≥60%），匹配不到再走模型，兼顾精度与召回。

混合分类器架构

intent classifier 分两阶段：规则匹配（高频高精度）+ 模型分类（覆盖长尾）。规则层覆盖退款、改地址、投诉等高频意图，置信度 0.98+；模型层覆盖语义相近但无关键词匹配的长尾场景。

CorpAssist · 混合路由流程 用户在 CorpAssist 客服输入「我要投诉快递太慢还丢件了」——规则匹配命中「投诉」关键词（conf=0.99），直接转人工。用户说「我那个订单有点问题麻烦帮看看」——无关键词命中，走模型分类为「query_order」（conf=0.72），走 Agent 确认流程。

混合路由将高频意图用规则拦截（延迟 <5ms），长尾由模型兜底。三级置信度控制分流粒度。

Python 混合分类器实现

python
# hybrid_classifier.py — CorpAssist 混合意图分类器
import re
from typing import Tuple, Optional
from openai import OpenAI

class RuleMatcher:
    """关键词规则匹配器，支持正则与精确匹配"""

    def __init__(self):
        self.rules = {
            "refund": (r"退款|退钱|退货|不要了|取消订单", 0.98),
            "change_address": (r"改地址|改收货|换地址|重新发", 0.97),
            "complaint": (r"投诉|举报|差评|客服态度|投诉快递", 0.99),
            "tracking": (r"物流|快递|到哪了|发货没|物流信息", 0.95),
            "human": (r"人工|转人工|客服|在线客服", 0.99),
        }

    def match(self, text: str) -> Optional[Tuple[str, float]]:
        for intent, (pattern, confidence) in self.rules.items():
            if re.search(pattern, text):
                return intent, confidence
        return None


class ModelClassifier:
    """轻量模型分类器，使用 qwen-turbo 做意图分类"""

    def __init__(self):
        self.client = OpenAI(base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")
        self.intents = ["refund", "change_address", "complaint", "tracking",
                        "query_order", "policy", "human", "other"]

    def classify(self, text: str) -> Tuple[str, float]:
        resp = self.client.chat.completions.create(
            model="qwen-turbo",
            messages=[{"role": "system", "content": f"从以下类别中选择：{self.intents}。仅返回 JSON：{{\"intent\":\"\",\"confidence\":0.0}}"},
                      {"role": "user", "content": text}],
            response_format={"type": "json_object"},
            temperature=0,
        )
        import json
        result = json.loads(resp.choices[0].message.content)
        return result.get("intent", "other"), result.get("confidence", 0.0)


class HybridIntentRouter:
    """混合意图路由：规则优先，模型兜底"""

    def __init__(self):
        self.rule_matcher = RuleMatcher()
        self.model_classifier = ModelClassifier()

    def route(self, text: str) -> dict:
        rule_result = self.rule_matcher.match(text)
        if rule_result:
            intent, confidence = rule_result
            return {"intent": intent, "confidence": confidence, "source": "rule"}

        intent, confidence = self.model_classifier.classify(text)
        return {"intent": intent, "confidence": confidence, "source": "model"}

    def decide_action(self, route_result: dict) -> str:
        intent = route_result["intent"]
        conf = route_result["confidence"]

        if intent == "complaint":
            return "HANDOFF"
        if conf >= 0.9:
            return "FAST_PATH" if intent in ("tracking", "policy") else "TASK"
        if conf >= 0.6:
            return "AGENT_CONFIRM"
        return "HANDOFF"


router = HybridIntentRouter()
test_cases = ["我要退款订单O2025001", "帮我查物流", "这个破客服太差了", "我想问一下那个事情怎么样了"]
for case in test_cases:
    result = router.route(case)
    action = router.decide_action(result)
    print(f"[{case[:12]}...] intent={result['intent']} conf={result['confidence']:.2f} src={result['source']} → {action}")

Java Spring 侧规则路由

// IntentRouter.java — Spring 侧规则路由组件
@Component
public class IntentRouter {

    private final Map<String, RouteAction> ruleMap = new LinkedHashMap<>();
    private final RestTemplate modelClient;

    public IntentRouter(RestTemplate modelClient) {
        this.modelClient = modelClient;
        ruleMap.put("(退款|退钱|退货)", RouteAction.of("refund", 0.98, ActionType.TASK));
        ruleMap.put("(改地址|换地址)", RouteAction.of("change_address", 0.97, ActionType.TASK));
        ruleMap.put("(投诉|举报)", RouteAction.of("complaint", 0.99, ActionType.HANDOFF));
        ruleMap.put("(人工|转人工)", RouteAction.of("human", 0.99, ActionType.HANDOFF));
        ruleMap.put("(物流|快递)", RouteAction.of("tracking", 0.95, ActionType.FAST_PATH));
    }

    public RouteResult route(String message) {
        // 规则优先
        for (Map.Entry<String, RouteAction> entry : ruleMap.entrySet()) {
            if (message.matches(entry.getKey())) {
                RouteAction action = entry.getValue();
                return new RouteResult(action.intent(), action.confidence(), "rule", action.type());
            }
        }
        // 模型兜底 — 调用 Python 分类器
        RouteResult modelResult = modelClient.postForObject(
            "http://python-agent/classify", message, RouteResult.class);
        return modelResult;
    }
}

record RouteAction(String intent, double confidence, ActionType type) {
    static RouteAction of(String intent, double confidence, ActionType type) {
        return new RouteAction(intent, confidence, type);
    }
}

record RouteResult(String intent, double confidence, String source, String actionType) {}

enum ActionType { FAST_PATH, TASK, HANDOFF, AGENT_CONFIRM }
不推荐
  • 纯模型路由：全部用大模型分类，延迟 1-3s，成本高
  • 纯规则路由：无法覆盖长尾语义，兜底率低
  • 无置信度分级：全量走 Agent 确认或全量直接处理
推荐
  • 规则优先：高频意图 <5ms 完成，命中率 ≥60%
  • 模型兜底：小模型 qwen-turbo 覆盖长尾语义
  • 三级阈值：高置信直接处理，中置信确认，低置信转人工
先想 10 秒：规则匹配到「投诉」直接转人工，为什么不是先确认一下再转？

投诉场景用户情绪已不稳定，Agent 确认只会激化矛盾。规则直接转人工，让坐席第一时间安抚用户。同理「人工」「转人工」关键词也直接转。

FAST PATH：FAQ 小模型快答

faq fast path 是成本优化的核心策略。对于物流查询、政策咨询等高频 FAQ 意图，使用 qwen-turbo 小模型 + 政策 RAG 缓存直接回答，不经过大模型。CorpAssist 生产实践中，FAST PATH 处理了约 40% 的会话量，节省 70% 推理成本。

路由路径模型延迟成本/请求会话占比
FAST PATH（FAQ）qwen-turbo + 缓存<200ms¥0.001~40%
任务型（DST）gpt-4o-mini~1s¥0.005~35%
Agent 确认gpt-4o-mini + 工具~2s¥0.01~15%
转人工人工坐席~30s¥2.5~10%
python
# fast_path.py — FAQ 快速通道
class FastPathHandler:
    """FAST PATH：小模型 + 缓存处理 FAQ"""

    def __init__(self):
        self.cache = {}  # 生产用 Redis
        self.client = OpenAI(base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")

    def handle(self, question: str) -> str:
        cache_key = hashlib.md5(question.encode()).hexdigest()[:16]
        if cache_key in self.cache:
            return self.cache[cache_key]  # < 5ms

        resp = self.client.chat.completions.create(
            model="qwen-turbo",
            messages=[
                {"role": "system", "content": "你是 CorpAssist 客服助手。简短回答 FAQ 问题，不超过 2 句。"},
                {"role": "user", "content": question},
            ],
            temperature=0.1,
            max_tokens=100,
        )
        answer = resp.choices[0].message.content
        self.cache[cache_key] = answer
        return answer
FAST PATH 命中率优化

生产环境中，FAST PATH 的命中率取决于意图分类准确率和缓存设计。建议：1) 缓存 key 用 Embedding 语义相似度而非全文匹配，相似度 ≥0.95 时命中；2) 每天凌晨全量预热高频 FAQ；3) 监控 FAST PATH 的转人工率，如果超过 5% 说明小模型质量不足，需要升级模型。

Fallback 策略与指标监控

fallback intent 是所有分类尝试均无匹配时的兜底策略。CorpAssist 的 fallback 策略链：规则未命中 → 模型分类置信度低 → 兜底为「other」→ 转人工。不强行回答未知意图，避免幻觉。

router metrics 定义四个核心路由指标：

指标定义目标预警阈值
intent accuracy意图分类准确率（自动标注集）≥92%<88%
规则命中率规则匹配的请求占比≥60%<50%
模型兜底率模型分类的请求占比≤40%>50%
转人工准确率转人工的请求中确实是需要转的≥85%<80%

routing s2 是 CorpAssist 的 S2 路由标准，要求路由模块具备：规则可热更新（不重启服务）、模型分类可灰度（按租户比例切换）、指标可观测（Prometheus + Grafana）。

常见问题（FAQ）

规则匹配和模型分类的优先级为什么是规则优先？
规则匹配延迟 <5ms，模型分类 200-500ms。对于「退款」「投诉」等高确定性关键词，规则匹配 100% 准确，不需要模型参与。规则优先兼顾延迟和准确性。
FAST PATH 用 qwen-turbo 会不会回答质量不够？
FAST PATH 仅处理 FAQ 类消息，如物流查询、政策咨询。这些场景的答案确定性高、篇幅短，小模型完全胜任。复杂推理（如退款金额计算）走任务型路径，用大模型处理。
转人工准确率怎么提升？
三管齐下：1) 丰富规则库，把确定性高需要转人工的场景用规则拦截；2) 优化模型分类置信度校准，确保 <0.6 的才转人工；3) 标注转人工后的会话，持续优化分类器。
规则库怎么维护？
规则库建议存储在配置中心（Nacos / Apollo），支持热更新。每周从坏例库提取分类错误案例，判断是否新增或修正规则。规则不宜过多（<100 条），过多会降低可维护性。

本章结论

  • 混合分类：规则优先（≥60% 命中率）、模型兜底（覆盖长尾），三级置信度阈值控制分流粒度。
  • FAST PATH：小模型 + 缓存处理 FAQ 消息，节省 70% 推理成本，覆盖 40% 会话量。
  • Fallback 兜底：所有分类尝试均无匹配时转人工，不强行回答未知意图。
  • 指标监控：意图准确率 ≥92%、规则命中率 ≥60%、模型兜底率 ≤40%、转人工准确率 ≥85%。
  • S2 路由标准：规则可热更新、模型可灰度、指标可观测。

判断练习

  1. 生产环境中发现 FAST PATH 的转人工率达到 8%（目标 5%），你会先排查哪个环节？

    参考答案

    先排查意图分类准确率。FAST PATH 高转人工率通常不是因为小模型回答不好，而是非 FAQ 意图被错误地路由到了 FAST PATH。检查 intent accuracy 是否低于 92%，如果是则优化分类器。其次检查缓存命中率是否正常，低命中率说明缓存设计不合理。

  2. 某关键词规则误触率较高（如「退货」规则常匹配到「查询退货政策」），你会如何优化？

    参考答案

    方案一：加负向规则，如果匹配到「退货」但同时包含「政策」「规则」「流程」等词，降低置信度或走模型分类。方案二：将规则匹配从关键词改为意图分类模型，对于模糊匹配走模型而不是规则直接命中。方案三：规则命中后加一层模型验证（级联分类），规则召回 + 模型精排。
