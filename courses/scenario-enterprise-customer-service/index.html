解决率

CorpAssist · 大促售后高峰（30 秒） 用户问「我的订单 #8821 到哪了？」Bot 调用订单 API 返回物流节点并确认签收时间——用户未转人工即结束会话。这算一次首解。若 Bot 只回复「请耐心等待」而未查单，即使 CSAT 尚可，也不算解决。

resolution rate（首解率 / 一次解决率）衡量无需人工介入即完成用户诉求的会话占比。CorpAssist S2 默认目标 ≥85%。

口径计入解决不计入
任务型（查单、退款、改址） 工具返回成功 + 用户确认或无追问 30s 内结束 仅给政策说明但未执行用户要求的动作
FAQ 型（运费、时效） 政策 RAG 命中 + 用户点击「解决了」 答非所问后用户沉默离开（需结合 CSAT 抽检）
转人工后 — 整通会话从 Bot 侧计为未解决

工程侧如何支撑解决率

  • 意图路由准确：FAQ 走快路径，任务型进 Agent+工具（意图识别与路由）。
  • 工具真执行：查单/退款必须调 ERP OpenAPI，禁止模型「假装已改单」（s2 scenario 生产坑）。
  • DST 双校验：订单号、退款金额等槽位由规则校验后再调 API（多轮上下文与指代）。
先想 10 秒：用户说「谢谢」后挂断，但 Bot 从未查过订单——算解决吗？

不算。解决率看诉求是否闭环，礼貌结束不能替代任务完成。

转人工率

handoff rate 是 Bot 将会话移交人工坐席的占比。CorpAssist 目标 ≤15%——不是越低越好到 0%，而是该转才转：低置信、敏感投诉、大额退款应主动转人工，避免强行 Bot 导致 CSAT 崩盘。

转人工触发路径
flowchart TD
  U[用户消息] --> R[意图路由]
  R -->|高置信 FAQ| F[政策 RAG 快答]
  R -->|任务型| A[Agent + 工具]
  A -->|工具失败 2 次| H[转人工]
  A -->|低置信 <0.6| H
  F -->|用户点转人工| H
  H --> Q[排队] --> Agent[人工坐席 + 上下文摘要]

转人工是降级能力而非失败；关键是 warm transfer：坐席能看到 Bot 已收集的槽位与工具调用记录。

不推荐
  • 为压转人工率，低置信仍强行 Bot 回答
  • 转人工后坐席从零问订单号
  • 无排队 SLA，用户直接流失
推荐
  • 置信度 + 敏感词 + 连续失败次数多路触发
  • 移交 payload 含 DST 槽位与最近 3 轮摘要
  • 高峰时段动态扩容坐席或回调

CSAT

CSAT（Customer Satisfaction）在本课采用会话结束后的五星评分均值，目标 ≥4.2。它与解决率、转人工率共同构成 toc support 质量三角：

指标反映什么典型误读
解决率 ↑ 自动化程度与工具成功率 「刷解决」：未真执行就结束会话
转人工率 Bot 能力边界与降级策略 压到 0% 导致复杂客诉 CSAT 暴跌
CSAT 用户主观体验（态度、速度、准确性） 样本偏差：只有满意用户才评分

工程上应对 CSAT 抽样偏差：人工抽检 + 坏例进评测集（客服效果评测）。NS1 飞轮要求低分样本必须可追溯到意图、工具 Trace 与 Prompt 版本。

# CorpAssist S2 双模型路由成本计算
class S2CostModel:
    def __init__(self):
        self.faq_cost_per_call = 0.025   # qwen-turbo
        self.agent_cost_per_call = 0.08  # qwen-plus
        self.human_cost_per_call = 8.0

    def monthly_cost(self, total_calls=100000,
                     faq_ratio=0.6, agent_ratio=0.3,
                     handoff_ratio=0.1) -> dict:
        faq = total_calls * faq_ratio * self.faq_cost_per_call
        agent = total_calls * agent_ratio * self.agent_cost_per_call
        human = total_calls * handoff_ratio * self.human_cost_per_call
        total = faq + agent + human

        return {
            "FAQ 模型": f"¥{faq:,.0f}",
            "Agent 模型": f"¥{agent:,.0f}",
            "人工坐席": f"¥{human:,.0f}",
            "月总成本": f"¥{total:,.0f}",
            "单次均摊": f"¥{total/total_calls:.2f}",
            "相比全人工节省": f"{1 - total/(total_calls*self.human_cost_per_call):.0%}"
        }

cost = S2CostModel()
print(cost.monthly_cost())
# 输出示例：月总成本 ¥109,500，相比全人工节省 86%

常见问题（FAQ）

解决率 85% 和模型准确率 90% 是一回事吗？
完全不是。模型准确率是「回答是否正确」，解决率是「用户诉求是否闭环」。模型正确回答了但工具调用失败，不算解决。解决率 = 正确回答 × 工具成功率 × 用户确认。
CSAT 评分怎么防止恶意刷分？
三道防线：① 仅对已完成会话弹评分（未闭环不弹）；② 同一用户 24h 内只计一次评分；③ 设定评分阈值——同一 IP 连续 5 次 1 分触发人工复核。
大促期间转人工率飙升怎么办？
先区分原因：是意图路由识别率下降（模型问题）还是用户真实需求超出 Bot 能力（业务问题）。前者优化分类器，后者动态扩容坐席并暂时放宽转人工阈值。

本章结论

  • 解决率：任务闭环才算首解；工具真执行 + DST 双校验是底座。
  • 转人工率：≤15% 是目标区间，低置信与敏感场景应主动 warm transfer。
  • CSAT：与解决率、转人工共同约束；需抽检与坏例飞轮防样本偏差。
  • CorpAssist S2：意图→对话→工具→坐席四层，Python Agent + Spring BFF 双栈落地。

意图路由与对话引擎

intent router mod 是客服系统的入口。它接受用户消息，输出意图分类和置信度。FAQ 类直接走政策 RAG 快答，任务型进入对话引擎。

dialog engine 负责任务型对话的多轮管理。核心是 DST（Dialog State Tracking），维护当前会话的槽位信息。

Python：意图路由 + 对话引擎

python 
# 意图路由与对话引擎
class IntentRouter:
    INTENTS = {
        "order_query": "任务型",
        "refund": "任务型",
        "change_address": "任务型",
        "shipping_policy": "FAQ",
        "complaint": "转人工",
    }

    def classify(self, message: str) -> tuple[str, float]:
        """返回 (intent, confidence)"""
        # 规则 + 模型混合分类
        for keyword, intent in {"退款": "refund",
                                "改地址": "change_address",
                                "投诉": "complaint"}.items():
            if keyword in message:
                return intent, 0.95
        # 模型兜底
        return self.model_classify(message)

    def route(self, message: str) -> str:
        intent, conf = self.classify(message)
        if conf < 0.6:
            return "转人工"
        if self.INTENTS.get(intent) == "FAQ":
            return "FAQ"
        return "任务型"

@RestController
public class RouterController {

    private final IntentClassifier classifier;

    @PostMapping("/api/chat")
    public Mono<ChatResponse> chat(@RequestBody ChatRequest req) {
        IntentResult intent = classifier.classify(req.message());
        return switch (intent.type()) {
            case "FAQ" -> faqHandler.handle(req);
            case "任务型" -> agentHandler.handle(req, intent);
            case "转人工" -> handoffHandler.transfer(req);
            default -> Mono.just(new ChatResponse("抱歉，我无法处理"));
        };
    }
}

工具层、坐席与质检

human agent 模块管理人工坐席的排队、分配、上下文移交。

qa module 质检模块对已结束的会话进行抽检，评估 Bot 回答质量、意图准确性、转人工的必要性。

  • 意图路由在 Spring 里硬编码
  • 工具调用直接在 LLM prompt 中写 HTTP 请求
  • 质检和对话引擎用同一个数据库
推荐：模块独立
  • Python 分类器独立部署
  • 工具层通过 OpenAPI Schema 暴露，由 Spring 管理
  • 质检数据独立存储，不对在线对话产生影响

常见问题（FAQ）

五个模块一定要分开部署吗？
初期可以少模块合并。最小可行性部署：意图路由 + 对话引擎合并在 Python 服务中，工具层 + 坐席合并在 Spring 中。随着业务增长再拆分。
质检模块应该全覆盖还是抽检？
抽检。日活 10 万会话全量质检成本太高。推荐：转人工的会话 100% 质检，Bot 解决的会话抽检 5-10%。出错率高的意图提高抽检比例。
工具层应该由 Python 还是 Spring 实现？
Spring。工具层需要与企业 ERP/CRM 对接，Java 在企业集成生态上更成熟（Spring Integration、JDBC、JMS）。Python 可以通过 HTTP 调用 Spring 暴露的工具 API。

本章结论

  • 五模块架构：意图路由 → 对话引擎 → 工具层 → 人工坐席 → 质检。
  • 独立部署：每个模块可独立扩缩，故障隔离。
  • 双栈分工：Python 管意图+对话，Spring 管工具+坐席+质检。
  • 模块通信：通过 API 或 MQ 通信，不共享数据库。

# DST 双校验
class DSTManager:
    SLOTS = {
        "order_id": {"type": str, "pattern": r"^O-\d{4}-\d{3}$"},
        "intent": {"type": str, "enum": [
            "refund", "change_address", "order_query"]},
        "amount": {"type": float, "min": 0},
    }
    INTENT_SLOTS = {
        "refund": ["order_id", "amount", "reason"],
        "change_address": ["order_id", "new_address"],
        "order_query": ["order_id"],
    }

    def __init__(self):
        self.state = {}

    def update(self, new_slots: dict, new_intent: str = None):
        # LLM 提取的新槽位
        validated = self.validate(new_slots)
        if new_intent and new_intent != self.state.get("intent"):
            # intent 变化：清理旧槽位
            self.state = {"intent": new_intent}
        self.state.update(validated)

    def validate(self, slots: dict) -> dict:
        valid = {}
        for k, v in slots.items():
            if k in self.SLOTS:
                spec = self.SLOTS[k]
                if isinstance(v, spec["type"]):
                    if "pattern" in spec and not re.match(spec["pattern"], str(v)):
                        continue
                    if "enum" in spec and v not in spec["enum"]:
                        continue
                    valid[k] = v
        return valid

class PolicyRAG:
    def __init__(self, vector_db):
        self.db = vector_db

    def query(self, question: str, top_k: int = 3) -> list:
        query_emb = self.db.embed(question)
        results = self.db.search(query_emb, top_k)
        return [
            {"content": r.content,
             "source": r.metadata.get("policy_id"),
             "effective_date": r.metadata.get("effective_date")}
            for r in results
        ]

    def format_context(self, policies: list) -> str:
        parts = ["## 相关企业政策\n"]
        for p in policies:
            parts.append(f"[{p['source']}] {p['content']}\n")
        return "".join(parts)

        result = json.loads(resp.choices[0].message.content)
        return result.get("intent", "other"), result.get("confidence", 0.0)


class HybridIntentRouter:
    """混合意图路由：规则优先，模型兜底"""

    def __init__(self):
        self.rule_matcher = RuleMatcher()
        self.model_classifier = ModelClassifier()

    def route(self, text: str) -> dict:
        rule_result = self.rule_matcher.match(text)
        if rule_result:
            intent, confidence = rule_result
            return {"intent": intent, "confidence": confidence, "source": "rule"}

        intent, confidence = self.model_classifier.classify(text)
        return {"intent": intent, "confidence": confidence, "source": "model"}

    def decide_action(self, route_result: dict) -> str:
        intent = route_result["intent"]
        conf = route_result["confidence"]

        if intent == "complaint":
            return "HANDOFF"
        if conf >= 0.9:
            return "FAST_PATH" if intent in ("tracking", "policy") else "TASK"
        if conf >= 0.6:
            return "AGENT_CONFIRM"
        return "HANDOFF"

@Component
public class IntentRouter {

    private final Map<String, RouteAction> ruleMap = new LinkedHashMap<>();
    private final RestTemplate modelClient;

    public IntentRouter(RestTemplate modelClient) {
        this.modelClient = modelClient;
        ruleMap.put("(退款|退钱|退货)", RouteAction.of("refund", 0.98, ActionType.TASK));
        ruleMap.put("(改地址|换地址)", RouteAction.of("change_address", 0.97, ActionType.TASK));
        ruleMap.put("(投诉|举报)", RouteAction.of("complaint", 0.99, ActionType.HANDOFF));
        ruleMap.put("(人工|转人工)", RouteAction.of("human", 0.99, ActionType.HANDOFF));
        ruleMap.put("(物流|快递)", RouteAction.of("tracking", 0.95, ActionType.FAST_PATH));
    }

    public RouteResult route(String message) {
        // 规则优先
        for (Map.Entry<String, RouteAction> entry : ruleMap.entrySet()) {
            if (message.matches(entry.getKey())) {
                RouteAction action = entry.getValue();
                return new RouteResult(action.intent(), action.confidence(), "rule", action.type());
            }
        }
        // 模型兜底 — 调用 Python 分类器
        RouteResult modelResult = modelClient.postForObject(
            "http://python-agent/classify", message, RouteResult.class);
        return modelResult;
    }
}

# fast_path.py — FAQ 快速通道
class FastPathHandler:
    """FAST PATH：小模型 + 缓存处理 FAQ"""

    def __init__(self):
        self.cache = {}  # 生产用 Redis
        self.client = OpenAI(base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")

    def handle(self, question: str) -> str:
        cache_key = hashlib.md5(question.encode()).hexdigest()[:16]
        if cache_key in self.cache:
            return self.cache[cache_key]  # < 5ms

        resp = self.client.chat.completions.create(
            model="qwen-turbo",
            messages=[
                {"role": "system", "content": "你是 CorpAssist 客服助手。简短回答 FAQ 问题，不超过 2 句。"},
                {"role": "user", "content": question},
            ],
            temperature=0.1,
            max_tokens=100,
        )
        answer = resp.choices[0].message.content
        self.cache[cache_key] = answer
        return answer
FAST PATH 命中率优化

生产环境中，FAST PATH 的命中率取决于意图分类准确率和缓存设计。建议：1) 缓存 key 用 Embedding 语义相似度而非全文匹配，相似度 ≥0.95 时命中；2) 每天凌晨全量预热高频 FAQ；3) 监控 FAST PATH 的转人工率，如果超过 5% 说明小模型质量不足，需要升级模型。

Fallback 策略与指标监控

fallback intent 是所有分类尝试均无匹配时的兜底策略。CorpAssist 的 fallback 策略链：规则未命中 → 模型分类置信度低 → 兜底为「other」→ 转人工。不强行回答未知意图，避免幻觉。

常见问题（FAQ）

规则匹配和模型分类的优先级为什么是规则优先？
规则匹配延迟 <5ms，模型分类 200-500ms。对于「退款」「投诉」等高确定性关键词，规则匹配 100% 准确，不需要模型参与。规则优先兼顾延迟和准确性。
FAST PATH 用 qwen-turbo 会不会回答质量不够？
FAST PATH 仅处理 FAQ 类消息，如物流查询、政策咨询。这些场景的答案确定性高、篇幅短，小模型完全胜任。复杂推理（如退款金额计算）走任务型路径，用大模型处理。
转人工准确率怎么提升？
三管齐下：1) 丰富规则库，把确定性高需要转人工的场景用规则拦截；2) 优化模型分类置信度校准，确保 <0.6 的才转人工；3) 标注转人工后的会话，持续优化分类器。
规则库怎么维护？
规则库建议存储在配置中心（Nacos / Apollo），支持热更新。每周从坏例库提取分类错误案例，判断是否新增或修正规则。规则不宜过多（<100 条），过多会降低可维护性。

本章结论

  • 混合分类：规则优先（≥60% 命中率）、模型兜底（覆盖长尾），三级置信度阈值控制分流粒度。
  • FAST PATH：小模型 + 缓存处理 FAQ 消息，节省 70% 推理成本，覆盖 40% 会话量。
  • Fallback 兜底：所有分类尝试均无匹配时转人工，不强行回答未知意图。
  • 指标监控：意图准确率 ≥92%、规则命中率 ≥60%、模型兜底率 ≤40%、转人工准确率 ≥85%。
  • S2 路由标准：规则可热更新、模型可灰度、指标可观测。

class QueryOrderParams(BaseModel):

    order_id: str = Field(..., description="订单号，格式 O+8位数字", pattern=r"^O\d{8}$")
    customer_phone: Optional[str] = Field(None, description="客户手机号后4位用于验证")

    @field_validator("order_id")
    @classmethod

        if not v.startswith("O") or len(v) != 9:
            raise ValueError(f"订单号格式错误: {v}")

class RefundParams(BaseModel):

    order_id: str = Field(..., description="订单号", pattern=r"^O\d{8}$")

// ErpTools.java — Spring AI Tools 注册
@Component
public class ErpTools {

    private final ErpClient erpClient;
    private final IdempotencyService idempotencyService;

    public ErpTools(ErpClient erpClient, IdempotencyService idempotencyService) {
        this.erpClient = erpClient;
        this.idempotencyService = idempotencyService;
    }

    @Tool(description = "查询订单状态。读操作，无需幂等。")
    public String queryOrder(@P("订单号，格式 O+8位数字") String orderId) {
        Order order = erpClient.queryOrder(orderId);
        return String.format("订单 %s：状态 %s，金额 ¥%.2f，物流 %s",
            order.orderId(), order.status(), order.amount(), order.logistics());
    }

    @Tool(description = "提交退款申请。写操作，Idempotency-Key 防重复。")
    public String applyRefund(
            @P("订单号") String orderId,
            @P("退款金额，单位元") double amount,
            @P("退款原因") String reason,
            @P("幂等键，防重复") String idempotencyKey) {

        // 幂等检查
        if (idempotencyService.exists(idempotencyKey)) {
            return idempotencyService.getResult(idempotencyKey);
        }

        // 参数校验
        if (amount <= 0 || amount > 99999.99) {
            throw new IllegalArgumentException("退款金额超出范围");
        }

        // 调用 ERP
        RefundResult result = erpClient.refund(orderId, amount, reason);
        idempotencyService.save(idempotencyKey, result.toString());

        return String.format("退款成功：订单 %s，金额 ¥%.2f，预计 %d 个工作日到账",
            orderId, amount, result.expectedDays());
    }

    @Tool(description = "修改订单收货地址。写操作，Idempotency-Key 防重复。")
    public String updateAddress(
            @P("订单号") String orderId,
            @P("新地址") String newAddress,
            @P("联系电话") String contactPhone,
            @P("幂等键") String idempotencyKey) {

        if (idempotencyService.exists(idempotencyKey)) {
            return idempotencyService.getResult(idempotencyKey);
        }
        if (newAddress.length() < 5) {
            throw new IllegalArgumentException("地址至少 5 个字符");
        }
        AddressResult result = erpClient.updateAddress(orderId, newAddress, contactPhone);
        idempotencyService.save(idempotencyKey, result.toString());
        return "地址修改成功：" + newAddress;
    }
}

// IdempotencyService.java — 幂等服务
@Service
public class IdempotencyService {
    private final RedisTemplate<String, String> redis;

    public IdempotencyService(RedisTemplate<String, String> redis) {
        this.redis = redis;
    }

    public boolean exists(String key) {
        return Boolean.TRUE.equals(redis.hasKey("idempotent:" + key));
    }

    public String getResult(String key) {
        return redis.opsForValue().get("idempotent:" + key);
    }

    public void save(String key, String result) {
        redis.opsForValue().set("idempotent:" + key, result, Duration.ofDays(30));
    }
}

Spring BFF。Python Agent 可能因为网络问题重复发送请求，如果由 Python 生成幂等键，重复请求可能携带相同的幂等键，但首次请求的结果还未返回时第二次请求就到了——幂等检查无效。Spring 侧在入口处生成幂等键更可靠，或者由客户端生成但要求幂等键包含业务唯一标识（订单号+金额+日期）。

ERP Connector：适配层设计

ERP Connector 是企业 ERP 系统的适配层，负责将 Spring AI Tools 的调用翻译为 ERP 的 API 请求。核心职责：协议转换（HTTP/SOAP/DB）、认证鉴权（API Key/OAuth）、错误处理（ERP 返回的业务异常映射为标准化错误码）。

java
// ErpConnector.java — ERP 适配层
@Component
public class ErpConnector {

    private final RestTemplate restTemplate;

    public ErpConnector(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public Order queryOrder(String orderId) {
        try {
            ResponseEntity<Order> resp = restTemplate.getForEntity(
                "http://erp-system/api/orders/{id}", Order.class, orderId);
            return resp.getBody();
        } catch (HttpClientErrorException.NotFound e) {
            throw new BusinessException("ORDER_NOT_FOUND", "订单不存在: " + orderId);
        } catch (HttpServerErrorException e) {
            throw new BusinessException("ERP_UNAVAILABLE", "ERP 系统繁忙，请稍后重试");
        }
    }

    public RefundResult refund(String orderId, double amount, String reason) {
        // 构建 ERP 请求
        Map<String, Object> request = new HashMap<>();
        request.put("orderId", orderId);
        request.put("amount", amount);
        request.put("reason", reason);
        // 调用 ERP 退款接口
        ResponseEntity<RefundResult> resp = restTemplate.postForEntity(
            "http://erp-system/api/refunds", request, RefundResult.class);
        return resp.getBody();
    }
}
ERP Connector 生产实践

1) 所有 ERP 调用设置 5s 超时，超时后触发熔断（转人工而非重试）；2) ERP 返回的业务异常（如订单已退款、订单不存在）映射为标准化 BusinessException，LLM 不直接暴露错误细节；3) 审计日志记录每次工具调用的请求和响应，用于对账和排查。

常见问题（FAQ）

为什么工具调用结果要直接写回，不让 LLM 转述？
LLM 转述可能产生幻觉——ERP 返回「退款成功」但 LLM 可能说「退款被驳回」或编造不存在的详细信息。api not hallucinate 原则要求工具结果以模板格式化后直接展示给用户，LLM 仅负责选择模板和填充参数。
幂等键的过期时间怎么设？
建议 30 天。Refund 幂等键的覆盖周期是退款处理的全流程（包括退款到账），通常 3-7 个工作日。30 天可以覆盖绝大多数退款周期。过期后幂等键失效，但历史记录在审计日志中可查。
参数校验在哪一层做？Python 还是 Spring？
两层都做。Python 侧用 Pydantic 做快速校验，拦截明显不合法的参数（如金额负数），减少无效请求对 Spring 的冲击。Spring 侧做完整校验和业务规则校验（如退款金额不能超过订单金额），确保进入 ERP 的数据合规。
工具调用如果超时了，应该重试吗？
写操作不自动重试，因为不确定 ERP 端是否已处理。应该返回超时错误给 Agent，Agent 标记订单为「待确认」状态，后续通过对账任务确认 ERP 状态。读操作可以有限重试（2 次，间隔 1s）。

# agent_graph.py — CorpAssist 双栈 Agent 状态图
from typing import TypedDict, List, Optional, Literal
from langgraph.graph import StateGraph, END
from openai import OpenAI
import httpx
import json

class AgentState(TypedDict):
    """Agent 状态"""
    session_id: str
    message: str
    intent: Optional[str]
    confidence: float
    dst_slots: dict
    tool_result: Optional[str]
    response: Optional[str]

class CSToolExecutor:
    """工具执行器：通过 HTTP 调用 Spring BFF 的工具"""

    def __init__(self, spring_base: str = "http://spring-bff:8080"):
        self.spring_base = spring_base

    async def execute(self, tool_name: str, params: dict) -> str:
        async with httpx.AsyncClient(timeout=10) as client:
            resp = await client.post(
                f"{self.spring_base}/tools/{tool_name}",
                json=params,
            )
            resp.raise_for_status()
            return resp.json()["result"]

class CSLangGraphAgent:
    """客服 Agent — LangGraph 实现"""

    def __init__(self):
        self.tool_executor = CSToolExecutor()
        self.llm = OpenAI(base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")
        self.graph = self._build_graph()

    def _build_graph(self) -> StateGraph:
        builder = StateGraph(AgentState)

        builder.add_node("receive", self._receive)
        builder.add_node("update_dst", self._update_dst)
        builder.add_node("select_action", self._select_action)
        builder.add_node("execute_tool", self._execute_tool)
        builder.add_node("generate_reply", self._generate_reply)

        builder.set_entry_point("receive")
        builder.add_edge("receive", "update_dst")
        builder.add_edge("update_dst", "select_action")
        builder.add_conditional_edges("select_action",
            self._decide_next, {
                "execute_tool": "execute_tool",
                "generate_reply": "generate_reply",
                "handoff": "generate_reply",
            })
        builder.add_edge("execute_tool", "generate_reply")
        builder.add_edge("generate_reply", END)
        return builder.compile()

    async def _receive(self, state: AgentState) -> AgentState:
        """接收用户消息"""
        print(f"[receive] 会话 {state['session_id']}: {state['message'][:50]}")
        return state

    async def _update_dst(self, state: AgentState) -> AgentState:
        """LLM 提取槽位更新 DST"""
        resp = self.llm.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "system", "content": "提取用户消息中的客服槽位，返回 JSON：{intent, slots}"},
                      {"role": "user", "content": state["message"]}],
            response_format={"type": "json_object"},
        )
        result = json.loads(resp.choices[0].message.content)
        state["intent"] = result.get("intent")
        state["dst_slots"].update(result.get("slots", {}))
        return state

    async def _select_action(self, state: AgentState) -> AgentState:
        """选择下一步动作"""
        if state["intent"] in ("complaint", "human"):
            state["response"] = "handoff"
            return state
        if state["intent"] in ("refund", "change_address") and self._slots_ready(state):
            return state  # 走 execute_tool
        return state  # 走 generate_reply

    def _decide_next(self, state: AgentState) -> Literal["execute_tool", "generate_reply", "handoff"]:
        if state.get("response") == "handoff":
            return "handoff"
        if state["intent"] in ("query_order", "refund", "change_address") and self._slots_ready(state):
            return "execute_tool"
        return "generate_reply"

    def _slots_ready(self, state: AgentState) -> bool:
        required = {"query_order": ["order_id"], "refund": ["order_id", "amount"],
                    "change_address": ["order_id", "new_address"]}
        slots = required.get(state.get("intent", ""), [])
        return all(s in state["dst_slots"] for s in slots)

    async def _execute_tool(self, state: AgentState) -> AgentState:
        tool_map = {"query_order": "query", "refund": "refund", "change_address": "address"}
        tool = tool_map.get(state["intent"], "")
        if tool:
            state["tool_result"] = await self.tool_executor.execute(tool, state["dst_slots"])
        return state

    async def _generate_reply(self, state: AgentState) -> AgentState:
        if state["intent"] == "complaint":
            state["response"] = "已为您转接人工客服，请稍候。"
        elif state["tool_result"]:
            state["response"] = state["tool_result"]
        else:
            # 槽位不足，反问
            missing = self._missing_slots(state)
            state["response"] = f"请提供以下信息：{', '.join(missing)}"
        return state

    async def run(self, session_id: str, message: str) -> str:
        initial = AgentState(session_id=session_id, message=message, intent=None,
                             confidence=0.0, dst_slots={}, tool_result=None, response=None)
        result = await self.graph.ainvoke(initial)
        return result.get("response", "抱歉，暂时无法处理，已转人工。")

// ChatController.java — Spring BFF 会话端点
@RestController
@RequestMapping("/api/chat")
public class ChatController {

    private final PythonAgentClient agentClient;
    private final RedisTemplate<String, Object> redis;
    private final SseEmitterManager sseManager;

    public ChatController(PythonAgentClient agentClient, RedisTemplate<String, Object> redis, SseEmitterManager sseManager) {
        this.agentClient = agentClient;
        this.redis = redis;
        this.sseManager = sseManager;
    }

    @PostMapping("/send")
    public SseEmitter sendMessage(@RequestBody ChatRequest request) {
        String sessionId = request.sessionId();
        SseEmitter emitter = sseManager.createEmitter(sessionId);

        // 异步处理
        CompletableFuture.runAsync(() -> {
            try {
                // 1. 读取 DST 状态
                Map<String, Object> dst = redis.opsForHash().entries("dst:" + sessionId);

                // 2. 调用 Python Agent
                AgentResponse response = agentClient.runAgent(sessionId, request.message(), dst);

                // 3. SSE 流式推送
                for (String chunk : response.stream()) {
                    emitter.send(SseEmitter.event().name("token").data(chunk));
                }
                emitter.send(SseEmitter.event().name("done").data(""));

                // 4. 更新 DST
                redis.opsForHash().putAll("dst:" + sessionId, response.updatedDst());
            } catch (Exception e) {
                try { emitter.send(SseEmitter.event().name("error").data("系统繁忙，请稍后重试")); } catch (Exception ignored) {}
            } finally {
                emitter.complete();
            }
        });
        return emitter;
    }
}

// SseEmitterManager.java — SSE 连接管理
@Component
public class SseEmitterManager {
    private final Map<String, SseEmitter> emitters = new ConcurrentHashMap<>();

    public SseEmitter createEmitter(String sessionId) {
        SseEmitter emitter = new SseEmitter(120_000L); // 2 分钟超时
        emitters.put(sessionId, emitter);
        emitter.onCompletion(() -> emitters.remove(sessionId));
        emitter.onTimeout(() -> emitters.remove(sessionId));
        return emitter;
    }
}

先想 10 秒：LangGraph 状态图和传统的 if-else 客服流程有什么本质区别？

LangGraph 状态图是声明式有向图，节点之间通过条件边路由，状态显式传递。if-else 是过程式判断，逻辑硬编码在代码中。状态图的优势：节点可独立测试、条件边可动态扩展（如新增意图时只需加一条边）、状态可序列化（存 Redis 实现跨请求持久化）。

Session Bridge：Redis DST 状态共享

bff session 在 Spring 侧管理会话生命周期，python agent cs 在 Python 侧执行推理。两者通过 Redis 共享 DST 状态：DST 数据以 Hash 结构存储在 Redis 中，key 为 dst:{sessionId}，字段包括 intent、slots、history_summary、tool_trace 等。

Redis Key字段说明
dst:{sessionId}intent, slots当前意图和槽位
dst:{sessionId}history_summary最近 3 轮摘要
dst:{sessionId}tool_trace本轮工具调用记录
session:{sessionId}ssemitterSSE 连接标识
DST 状态共享实践

1) Redis 中的 DST 设置 30 分钟 TTL，与客服会话超时一致；2) Python Agent 每次推理前先读 DST，推理后写回；3) intent 变更时自动清理旧槽位（如从 change_address 切为 refund，清理 address 相关槽位）；4) 转人工时将 DST 序列化为 JSON 放入移交 payload。

常见问题（FAQ）

为什么不直接用 Python 实现全部功能，还要引入 Spring BFF？
企业级部署要求：1) 工具调用经过 Spring 层可以统一做鉴权、审计、限流；2) ERP 接口通常只有 Java SDK 或 SOAP 协议，Java 侧集成更成熟；3) 双栈架构允许 Python Agent 和 Spring 工具独立扩缩容（Agent 需要 GPU，工具需要网络吞吐）。
LangGraph 和 LangChain Agent 有什么区别？
LangGraph 是图结构的状态机，天然支持循环、条件分支、并行节点，适合客服这种需要多轮对话和工具调用的场景。LangChain Agent 是线性循环（AgentExecutor），每次循环重新选择工具，客服场景下状态管理不够精确。
SSE 流式断了怎么办？
SSE 有自动重连机制（EventSource 默认重试）。Spring BFF 侧的 SseEmitter 设置 2 分钟超时，断线后前端重新建立连接。Python Agent 的推理结果如果已经生成但连接断了，下次请求时复用之前的 DST 状态，不会丢失上下文。
Redis DST 和 Agent 中的状态如何保持同步？
Python Agent 每次推理前从 Redis 读取最新 DST，推理结束后写回。写操作用 Redis Hash 的 HSET 命令，支持字段级别更新。如果 Agent 推理过程中 DST 被其他请求修改（极小概率），下一次推理时会读取到最新状态。

# degrade_manager.py — CorpAssist 降级策略管理器
from typing import Optional, Dict, Any
import json

class DegradeManager:
    """降级管理器：评估触发条件，生成移交 Payload"""

    SENSITIVE_KEYWORDS = ["投诉", "举报", "法律", "律师", "媒体", "曝光", "315"]
    HUMAN_REQUEST_KEYWORDS = ["转人工", "真人", "人工客服", "在线客服", "找人工"]
    CONFIDENCE_THRESHOLD = 0.6
    MAX_TOOL_FAILURES = 2

    def __init__(self):
        self.session_failures: Dict[str, int] = {}  # session_id -> tool_failure_count

    def should_handoff(self, session_id: str, message: str, intent: str,
                       confidence: float, tool_failed: bool = False) -> tuple[bool, str]:
        """判断是否需要转人工，返回 (是否转, 原因)"""
        # 1. 用户主动要求
        for kw in self.HUMAN_REQUEST_KEYWORDS:
            if kw in message:
                return True, "user_request"

        # 2. 敏感内容
        for kw in self.SENSITIVE_KEYWORDS:
            if kw in message:
                return True, "sensitive_content"

        # 3. 低置信度
        if confidence < self.CONFIDENCE_THRESHOLD:
            return True, "low_confidence"

        # 4. 工具连续失败
        if tool_failed:
            self.session_failures[session_id] = self.session_failures.get(session_id, 0) + 1
            if self.session_failures[session_id] >= self.MAX_TOOL_FAILURES:
                del self.session_failures[session_id]
                return True, "tool_failure"
        else:
            self.session_failures[session_id] = 0

        return False, ""

    def build_payload(self, session_id: str, dst_state: dict, history: list,
                      tool_trace: list, reason: str) -> dict:
        """构建 Warm Transfer 移交 Payload"""
        return {
            "session_id": session_id,
            "handoff_reason": reason,
            "timestamp": __import__("datetime").datetime.now().isoformat(),
            "summary": self._generate_summary(history),
            "dst_state": dst_state,
            "history_summary": "\n".join(history[-3:]),
            "tool_trace": tool_trace[-3:],
        }

    def _generate_summary(self, history: list) -> str:
        if not history:
            return "新会话"
        return f"用户共说了 {len(history)} 轮，最近意图：{history[-1][:50] if history else '未知'}"

    def create_ticket(self, payload: dict) -> str:
        """创建工单，返回工单号"""
        ticket_id = f"TK{__import__('time').time_ns() % 1000000:06d}"
        print(f"[工单] {ticket_id}: {payload['handoff_reason']} - {payload.get('summary', '')}")
        return ticket_id

// HandoffController.java — 转人工控制器
@RestController
@RequestMapping("/api/handoff")
public class HandoffController {

    private final QueueService queueService;
    private final RedisTemplate<String, Object> redis;
    private final ObjectMapper mapper;

    public HandoffController(QueueService queueService, RedisTemplate<String, Object> redis, ObjectMapper mapper) {
        this.queueService = queueService;
        this.redis = redis;
        this.mapper = mapper;
    }

    @PostMapping("/request")
    public ResponseEntity<HandoffResponse> requestHandoff(@RequestBody HandoffRequest request) {
        // 1. 从 Redis 读取 DST 状态
        Map<Object, Object> dst = redis.opsForHash().entries("dst:" + request.sessionId());

        // 2. 构建移交 payload
        HandoffPayload payload = new HandoffPayload(
            request.sessionId(), request.reason(),
            dst, request.history(), request.toolTrace()
        );

        // 3. 创建工单
        String ticketId = queueService.createTicket(payload);

        // 4. 排队
        int position = queueService.enqueue(ticketId, payload);

        return ResponseEntity.ok(new HandoffResponse(ticketId, position, "排队中，预计30s内响应"));
    }

    @GetMapping("/ticket/{ticketId}")
    public ResponseEntity<TicketStatus> getTicketStatus(@PathVariable String ticketId) {
        TicketStatus status = queueService.getStatus(ticketId);
        return ResponseEntity.ok(status);
    }
}

// QueueService.java — 排队服务
@Service
public class QueueService {
    private final ConcurrentLinkedQueue<Ticket> queue = new ConcurrentLinkedQueue<>();
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);

    public String createTicket(HandoffPayload payload) {
        String ticketId = "TK" + System.currentTimeMillis() % 1000000;
        Ticket ticket = new Ticket(ticketId, payload, Instant.now());
        return ticketId;
    }

    public int enqueue(String ticketId, HandoffPayload payload) {
        queue.add(new Ticket(ticketId, payload, Instant.now()));
        // 30s 超时升级
        scheduler.schedule(() -> {
            if (getStatus(ticketId).status().equals("waiting")) {
                escalate(ticketId);
            }
        }, 30, TimeUnit.SECONDS);
        return queue.size();
    }

    public void escalate(String ticketId) {
        System.out.println("[升级] 工单 " + ticketId + " 排队超时，通知组长");
    }
}

常见问题（FAQ）

转人工后，用户可以在 Bot 和人工之间切换吗？
可以。CorpAssist 的设计是转人工后用户仍可召唤 Bot 协助查单，但所有写操作必须由坐席确认。Bot 的回答会同步展示在坐席侧，坐席可以引用或修正。
排队人数太多时，应该加坐席还是优化 Bot？
先优化 Bot 降低转人工率。加坐席是线性扩缩，成本高；提升 Bot 解决率是杠杆效应。CorpAssist 的做法是当转人工率超过 20% 时，分析转人工原因，针对性优化 Bot 分类器或工具成功率。
敏感内容转人工的敏感词库怎么维护？
敏感词库分为静态层和动态层。静态层包含法律法规要求的必转词（投诉、举报、315），由法务团队提供。动态层包含业务相关的敏感词（媒体、曝光、律师），每周运营团队更新。所有变更记录审计。
工具连续失败转人工后，坐席看到什么？
坐席侧展示完整的工具 Trace：意图、调用的工具、传入参数、ERP 返回的错误信息。坐席可直接在工单系统中修改参数重新提交，不必重新让用户描述问题。同时坐席侧显示用户最近 3 轮对话摘要。

# auto_eval.py — CorpAssist 自动评测脚本
import json
import csv
from datetime import datetime
from typing import List, Dict
from dataclasses import dataclass

@dataclass
class EvalSample:
    id: str
    message: str
    expected_intent: str
    expected_tool: str
    expected_success: bool
    expected_csat: int  # 1-5

@dataclass
class EvalResult:
    sample_id: str
    predicted_intent: str
    intent_correct: bool
    tool_called: str
    tool_success: bool
    latency_ms: float

class AutoEvalPipeline:
    """自动评测管线"""

    def __init__(self, classifier, tools: dict):
        self.classifier = classifier
        self.tools = tools
        self.samples: List[EvalSample] = []
        self.results: List[EvalResult] = []

    def load_samples(self, path: str = "./eval_data/eval_dataset.json"):
        with open(path) as f:
            data = json.load(f)
        self.samples = [EvalSample(**s) for s in data]
        print(f"加载 {len(self.samples)} 条评测样本")

    def run(self) -> dict:
        """运行全量评测"""
        for sample in self.samples:
            import time
            start = time.time()

            route = self.classifier.route(sample.message)
            intent = route["intent"]
            llm_latency = (time.time() - start) * 1000

            tool_called = ""
            tool_success = False
            if route.get("action") == "TASK":
                tool_called = intent
                tool_success = self.tools.get(intent, lambda: False)()

            self.results.append(EvalResult(
                sample_id=sample.id,
                predicted_intent=intent,
                intent_correct=intent == sample.expected_intent,
                tool_called=tool_called,
                tool_success=tool_success,
                latency_ms=llm_latency,
            ))

        return self.report()

    def report(self) -> dict:
        """生成评测报告"""
        n = len(self.results)
        if n == 0:
            return {}
        intent_acc = sum(1 for r in self.results if r.intent_correct) / n
        tools_attempted = [r for r in self.results if r.tool_called]
        tool_success_rate = sum(1 for r in tools_attempted if r.tool_success) / max(len(tools_attempted), 1)
        avg_latency = sum(r.latency_ms for r in self.results) / n
        return {
            "date": datetime.now().date().isoformat(),
            "samples": n,
            "intent_accuracy": round(intent_acc * 100, 1),
            "tool_success_rate": round(tool_success_rate * 100, 1),
            "avg_latency_ms": round(avg_latency, 1),
            "p95_latency_ms": round(sorted(r.latency_ms for r in self.results)[int(n * 0.95)], 1),
        }

    def export_badcases(self, path: str = "./eval_data/badcases.json"):
        """导出坏例"""
        badcases = []
        for r in self.results:
            sample = next(s for s in self.samples if s.id == r.sample_id)
            if not r.intent_correct:
                badcases.append({
                    "id": r.sample_id,
                    "message": sample.message,
                    "expected": sample.expected_intent,
                    "predicted": r.predicted_intent,
                    "category": "intent_error",
                })
        with open(path, "w") as f:
            json.dump(badcases, f, ensure_ascii=False, indent=2)
        print(f"导出 {len(badcases)} 条坏例到 {path}")

先想 10 秒：CSAT 4.2 但任务成功率只有 70%，你信哪个？

A/B 实验与坏例库

ab cs 定义 A/B 实验框架：按租户 50/50 分流，实验组用新路由策略，对照组用旧策略。观察期至少 7 天，统计显著性采用卡方检验。

badcase cs 是坏例库的核心。每条坏例记录：原始消息、预期的正确行为、实际行为、错误分类（intent_error / tool_error / reply_error）、归因（规则不完善/模型未覆盖/prompt 歧义）、修复建议、状态（待修复/已修复/已验证）。

常见问题（FAQ）

自动评测的标注集怎么维护？
CorpAssist 的标注集分为核心集（500 条，覆盖所有意图，每周更新）和增量集（每周从生产会话中新增 50-100 条，覆盖新增场景）。标注集存储在 JSON 文件中，版本化管理。
人工抽检的样本怎么选？
分层抽样：Bot 解决会话按 5% 随机抽，转人工会话 100% 抽检，低 CSAT 会话 100% 抽检。确保抽检样本既能反映整体质量，又能覆盖高风险会话。
A/B 实验最少需要多少样本量？
取决于指标提升幅度。CorpAssist 的做法：预期提升 2pp，需每组至少 5000 条会话（约 2-3 天流量），显著性水平 α=0.05，统计功效 80%。建议 A/B 实验至少运行 7 天覆盖不同日期的用户行为差异。
指标之间相互矛盾怎么办（如 CSAT 升了但解决率降了）？
优先相信解决率。CSAT 受短期情绪影响大，解决率反映实际问题是否解决。如果解决率下降，说明产品本身有问题，CSAT 的回升可能是偶然因素（如坐席态度改善）。应该深挖解决率下降的原因，而不是庆祝 CSAT 上升。

  • 五维指标：意图准确率 ≥92%、任务成功率 ≥85%、CSAT ≥4.2、转人工准确率 ≥80%、工具成功率 ≥95%。
  • 自动 + 人工双轨：每日 2000 条自动跑分，每周 50-100 条人工校准。

  • 解决率优先：指标矛盾时优先相信任务成功率，深挖根因。

# dst_validator.py — DST 双校验：LLM 提取 + 规则校验
import re
from typing import Dict, Any

class DSTValidator:
    """DST 双校验器：LLM 提取槽位后，用规则校验合法性"""

    INTENT_SCHEMA = {
        "refund": {"required": ["order_id", "amount"], "optional": ["reason"]},
        "change_address": {"required": ["order_id", "new_address"], "optional": ["contact_phone"]},
        "query_order": {"required": ["order_id"], "optional": []},
    }

    def __init__(self):
        self.current_intent = None
        self.slots = {}
        self.last_intent = None

    def update(self, new_intent: str, new_slots: Dict[str, Any]) -> Dict[str, Any]:
        """更新 DST 状态，含意图切换检测和槽位校验"""
        # 意图切换检测
        if self.current_intent and new_intent != self.current_intent:
            print(f"[DST] 意图切换: {self.current_intent} → {new_intent}")
            self._clear_slots_for_intent(new_intent)

        self.last_intent = self.current_intent
        self.current_intent = new_intent

        # 合并新槽位
        for k, v in new_slots.items():
            self.slots[k] = v

        # 校验合法性
        self._validate()

        return self.slots

    def _clear_slots_for_intent(self, new_intent: str):
        """切换意图时清理无关槽位"""
        required_for_new = self.INTENT_SCHEMA.get(new_intent, {}).get("required", [])
        optional_for_new = self.INTENT_SCHEMA.get(new_intent, {}).get("optional", [])
        allowed = set(required_for_new + optional_for_new)
        self.slots = {k: v for k, v in self.slots.items() if k in allowed}
        print(f"[DST] 清理槽位，保留: {allowed}")

    def _validate(self):
        """规则校验槽位值"""
        for k, v in self.slots.items():
            if k == "order_id" and not re.match(r"^O\d{8}$", str(v)):
                print(f"[DST] 警告: order_id 格式异常 {v}")
                del self.slots[k]
            if k == "amount":
                try:
                    amt = float(v)
                    if amt <= 0 or amt > 99999.99:
                        del self.slots[k]
                except (ValueError, TypeError):
                    del self.slots[k]
            if k == "contact_phone" and not re.match(r"^1\d{10}$", str(v)):
                del self.slots[k]

    def get_required_missing(self) -> list:
        """返回缺失的必填字段"""
        schema = self.INTENT_SCHEMA.get(self.current_intent, {})
        required = schema.get("required", [])
        return [f for f in required if f not in self.slots]

    def is_ready(self) -> bool:
        """所有必填字段是否已收集"""
        return len(self.get_required_missing()) == 0

    enum Complexity { SIMPLE, MEDIUM, COMPLEX }
    enum ModelTier { TURBO, PLUS, MAX }

        ModelTier.TURBO, "qwen-vl-turbo",

常见问题（FAQ）

DST 双校验和纯规则 DST 有什么区别？
纯规则 DST 要求每条规则精确匹配，覆盖率低。双校验是「LLM 提取 + 规则验证」：LLM 负责从自然语言中提取槽位（高召回），规则负责校验槽位格式和业务逻辑（高精度），两者互补。
截图分级会不会漏掉重要的视觉信息？
分级的目标是「大多数用便宜模型，少数复杂场景用贵模型」。分级分类器本身会用 qwen-vl-plus 做校准：如果 turbo 模型的输出置信度低于 0.7，自动升级到 plus 或 max 重新分析。确保不会因为省钱而降低质量。
语义缓存的相似度阈值设多少合适？
CorpAssist 的实践是 0.92。太高（如 0.98）命中率太低，太低（如 0.85）容易命中语义相近但答案不同的内容。建议在生产用 0.90-0.95 之间调试，同时缓存 key 包含意图标签以减少误命中。
高 QPS 场景下如果缓存全失效怎么办？
缓存雪崩是真实风险。防护措施：1) 缓存 key 的 TTL 加随机偏移（±10% 避免同时过期）；2) 本地缓存兜底（Caffeine，1000 条容量）；3) 如果 QPS 超过模型配额，开启弹性降级——转人工而非排队等待模型。

# circuit_breaker.py — Python 侧熔断器
import time
from enum import Enum
from typing import Callable, Any

class CircuitState(Enum):
    CLOSED = "closed"       # 正常
    OPEN = "open"           # 熔断
    HALF_OPEN = "half_open" # 半开试探

class CircuitBreaker:
    """熔断器：保护下游 AI 服务"""

    def __init__(self, failure_threshold: int = 5, recovery_timeout: float = 30.0):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time = 0.0

    def call(self, func: Callable, fallback: Callable, *args, **kwargs) -> Any:
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                print("[熔断器] 进入半开状态")
            else:
                print("[熔断器] 熔断中，执行降级")
                return fallback(*args, **kwargs)

        try:
            result = func(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                print("[熔断器] 半开试探成功，恢复关闭状态")
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            print(f"[熔断器] 失败 ({self.failure_count}/{self.failure_threshold}): {e}")
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
                print("[熔断器] 触发熔断 → OPEN")
            return fallback(*args, **kwargs)

// SseConnectionManager.java — SSE 生命周期管理
@Component
public class SseConnectionManager {

    private final ConcurrentHashMap<String, SseConnection> connections = new ConcurrentHashMap<>();

    public SseEmitter createConnection(String sessionId) {
        SseEmitter emitter = new SseEmitter(120_000L); // 2 分钟超时
        SseConnection conn = new SseConnection(sessionId, emitter, Instant.now());
        connections.put(sessionId, conn);

        emitter.onCompletion(() -> {
            connections.remove(sessionId);
            log.info("SSE 连接完成: {}", sessionId);
        });
        emitter.onTimeout(() -> {
            connections.remove(sessionId);
            log.warn("SSE 连接超时: {}", sessionId);
        });
        emitter.onError(e -> {
            connections.remove(sessionId);
            log.error("SSE 连接错误: {}", sessionId, e);
        });
        return emitter;
    }

    public boolean send(String sessionId, String event, String data) {
        SseConnection conn = connections.get(sessionId);
        if (conn != null) {
            try {
                conn.emitter().send(SseEmitter.event().name(event).data(data));
                return true;
            } catch (IOException e) {
                connections.remove(sessionId);
                return false;
            }
        }
        return false;
    }

    public record SseConnection(String sessionId, SseEmitter emitter, Instant createdAt) {}
}

常见问题（FAQ）

限流阈值设多少合适？
取决于模型 API 配额和用户量。CorpAssist 的实践：每租户 100 req/s 是安全线，对应 gpt-4o-mini 的 TPM 配额（~200K TPM）。如果使用更小的模型（qwen-turbo），限流可放宽到 500 req/s。建议从低开始逐步上调，观察 P99 延迟和 API 限流率。
熔断后用户看到什么？
熔断后网关返回降级响应。CorpAssist 的做法：非紧急场景（FAQ）→ 返回「系统繁忙，请稍后重试」；紧急场景（投诉/退款）→ 直接转人工，坐席侧标注「AI 服务熔断」方便排障。熔断恢复后自动切回正常模式。
SSE 断线重连怎么做？
前端用 EventSource API 原生支持自动重连。后端推荐：1) SseEmitter 设置 120s 超时；2) 发送心跳包每 30s 一次保活；3) 断线后前端重新建立 SSE 连接，携带 lastEventId，后端从断点续传。SSE 协议标准支持 Last-Event-ID 头。

# cost_analyzer.py — CorpAssist 成本效益分析
from dataclasses import dataclass

@dataclass
class CSConfig:
    monthly_salary_per_agent: float = 8000  # 坐席月薪
    daily_sessions: int = 15000
    bot_cost_per_session: float = 0.15       # Bot 单次推理成本
    agent_cost_per_session: float = 2.5      # 人工单次成本
    handoff_rate: float = 0.15               # 转人工率
    monthly_agent_count: int = 12
    bot_monthly_fixed: float = 15000         # Bot 月固定成本(服务器+API)

class CSROIAnalyzer:
    """客服 ROI 分析器"""

    def __init__(self, before: CSConfig, after: CSConfig):
        self.before = before
        self.after = after

    def monthly_cost_before(self) -> float:
        agents = 20
        return agents * self.before.monthly_salary_per_agent

    def monthly_cost_after(self) -> float:
        sessions = self.after.daily_sessions * 30
        bot_cost = sessions * self.after.bot_cost_per_session
        handoff_sessions = sessions * self.after.handoff_rate
        agent_cost = handoff_sessions * self.after.agent_cost_per_session
        return agent_cost + bot_cost + self.after.bot_monthly_fixed

    def monthly_savings(self) -> float:
        return self.monthly_cost_before() - self.monthly_cost_after()

    def annual_savings(self) -> float:
        return self.monthly_savings() * 12

    def report(self) -> str:
        return f"""
=== CorpAssist 客服 ROI 报告 ===
上线前月成本: ¥{self.monthly_cost_before():,.0f}
上线后月成本: ¥{self.monthly_cost_after():,.0f}
月净节省:     ¥{self.monthly_savings():,.0f}
年净节省:     ¥{self.annual_savings():,.0f}
ROI (首年):   {self.annual_savings() / self.monthly_cost_after() * 100:.0f}%
        """


# 计算
before = CSConfig(monthly_agent_count=20, daily_sessions=5000, handoff_rate=0.25,
                  agent_cost_per_session=8.0, bot_cost_per_session=0)
after = CSConfig(monthly_agent_count=12, daily_sessions=15000, handoff_rate=0.15)
analyzer = CSROIAnalyzer(before, after)
print(analyzer.report())

常见问题（FAQ）

转人工率降低到多少是合理的？
因行业而异。CorpAssist 的目标是 15%（即 100 个用户中 85 个由 Bot 解决）。但投诉、大额退款、法律相关场景必须转人工，所以转人工率不会低于 10%。如果转人工率低于 10%，说明 Bot 可能「强留」了本应转人工的用户，会导致 CSAT 下降。
ROI 报告中应该包含哪些成本项？
应包括：1) 直接成本——模型 API 调用费、服务器/GPU 成本；2) 人力成本——坐席团队、运维团队、算法团队分摊；3) 机会成本——Bot 处理更多会话带来的增量收入。注意区分固定成本和可变成本。
业务方最关心哪个指标？
首解率（Resolution Rate）。业务方最关心「多少用户不需要找人工就能解决问题」。首解率直接关联坐席成本、用户满意度、复购率。转人工率是次要指标——只要首解率高，转人工率自然低。
CSAT 从 3.9 到 4.3 的经济价值怎么量化？
CSAT 与用户留存率正相关。研究表明 CSAT 每提升 0.1，留存率提升约 1-2pp。按 CorpAssist 的 DAU 100 万计算，0.4 分的 CSAT 提升意味着 4-8 万日活用户的留存改善，折合年收入提升约 ¥200-400 万。

本章结论

  • 六维指标：首解率、转人工率、CSAT、会话量、坐席数、单次成本。
  • 叙事重点：技术指标翻译为业务语言，数字 + 故事结合。
  • 成本算总账：坐席节省 - Bot 投入 = 月净节省，年 ROI 通常 200-500%。
  • 复盘框架：效果回顾 + 技术复盘 + 过程复盘 + 后续规划。
  • CSAT 价值量化：CSAT 0.1 分 ≈ 1-2pp 留存率提升。

3. 生产坑（30s）：
   "早期做过 Cold Transfer，CSAT 从 4.3 降到 3.5——因为用户需要重复描述问题。后来改为 Warm Transfer 并加入排队 SLA，CSAT 回升到 4.3。注意转人工率不是越低越好，该转的必须转。"

面试场景模拟

s2 interview 要求候选人不仅能答出「做了什么」，还要能说明「为什么这么做」和「遇到的坑」。下面模拟一个典型的系统设计面试场景。

模拟场景：系统设计面试

面试官：「请设计一个智能客服系统，支持多轮对话和工具调用。你从整体架构开始讲。」

候选人：「我们的系统采用双栈架构。Python Agent 侧用 LangGraph 做状态机编排——每个客服会话对应一个 Agent 实例，状态图包含 receive → update_dst → select_tool → execute → reply 五个节点。Spring BFF 侧负责工具执行（Spring AI Tools 注解注册 ERP 接口）和 SSE 流式推送。双栈通过 Redis 共享 DST 状态，Python 每次推理前读 Redis，推理后写回。」

面试官：「为什么不用 Python 全部实现，要引入 Spring？」

候选人：「三个原因。第一，安全审计——所有工具调用必须在 Spring 侧记录完整审计日志，Python 侧做不到；第二，ERP 集成——企业 ERP 系统通常只提供 Java SDK 或 SOAP 接口，Java 集成成本更低；第三，独立扩缩容——Python Agent 是 GPU 密集型（推理），Spring BFF 是 IO 密集型（API 调用），独立扩容更节省成本。」

面试官：「如果 Redis 挂了怎么办？」

候选人：「Redis 我们部署了主从 + Sentinel 自动切换。极端情况 Redis 完全不可用时，Spring BFF 从本地缓存读取最近一次 DST 快照（每个会话缓存 30 分钟），Python Agent 从请求中携带的 DST 摘要恢复状态。但写操作会暂停，直到 Redis 恢复。Redis 恢复后自动同步最新状态。」

面试官：「用户从中途切换意图（从退款改为查物流），DST 怎么处理？」

候选人：「这是我们的生产踩坑经历。初期纯靠 LLM 管理 DST，用户切换意图时旧槽位不清，导致工具调用参数错误。后来加入 DST 双校验：LLM 提取新意图后，规则层检测意图是否变化——如果变化，清理旧意图的所有槽位，只保留跨意图通用的字段（如用户 ID）。同时自动反问用户确认意图变更，减少误判断。」

面试官追问集锦
  • 追问 1：「你的意图路由规则冲突了怎么处理？」→ 优先级排序（投诉 > 退款 > 改地址 > 查询），同优先级按最早匹配。
  • 追问 2：「DST 状态存 Redis 挂了怎么办？」→ Redis 主从 + Sentinel，宕机自动切换。极端情况读 MySQL 备份 DST。
  • 追问 3：「模型 API 限流了怎么优雅降级？」→ 限流返回 429 + 降级话术，熔断后自动转人工，恢复后重试。
  • 追问 4：「用户说方言或者有错别字影响意图识别怎么办？」→ 规则层用拼音匹配，模型层用数据增强（加噪声训练）。
  • 追问 5：「1000 万日活的客服系统需要多少资源？」→ 按 1000 万 × 10% 日咨询率 = 100 万日会话。100 万 × 10 轮平均 = 1000 万次推理。需要 4-8 个 GPU 节点运行 Agent，BFF 层 10-20 个 Spring 实例。

常见问题（FAQ）

面试官问「这个系统最大的挑战是什么」怎么回答？
推荐回答：部分失败的处理。客服系统涉及 Python Agent、Spring BFF、Redis、ERP 四个子系统，任何一个都可能出问题。最大的挑战不是实现功能，而是保证 7x24 的可靠性——模型 API 限流、网络超时、ERP 不可用都在生产中出现过。我们通过限流、熔断、幂等、Warm Transfer 四项措施保证系统在故障时仍然可用。
系统设计面试画图画到什么程度？
足够展示模块间关系和关键技术选型即可，不需要精确的端口号或配置参数。推荐画出：客户端 → 网关层 → 核心服务层 → 工具层 → 存储层 五层架构，标注每层的技术选型（Spring Gateway / LangGraph / Redis 等）和通信协议（HTTP / SSE / Redis）。
如果面试官说「这个设计太复杂了，为什么不用一个 Python 服务全部搞定」？
回答框架：1) 安全角度——ERP 调用需要在 Java 侧做完整的审计日志，Python 侧做不到；2) 性能角度——Python 的异步 IO 不如 Java Netty 高效，SSE 推送场景 Java 更有优势；3) 运维角度——Python 主要负责推理（GPU 密集型），Java 主要负责 IO（CPU 密集型），独立扩缩容可以节省成本。不是故意复杂，是企业级部署的必然选择。

本章结论

  • 五连问结构：模块架构 → 指代消解 → 转人工策略 → 政策合规 → 评估体系。
  • 回答模板：原理 (30s) → 工程实现 (60s) → 生产坑 (30s)，2 分钟结构化回答。
  • 系统设计图：五层架构（客户端/网关/核心/工具/存储），标注选型和协议。
  • 面试追问：准备好解释架构选型原因和应对故障的兜底方案。
  • 最终话术：复杂度不是设计出来的，是问题逼出来的——每个组件都有其存在的理由。
