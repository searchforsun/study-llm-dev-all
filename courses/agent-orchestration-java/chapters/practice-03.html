为什么学这章

CorpAssist 需要接入多个外部系统：邮件系统、日历系统、工单系统、审批系统。每个系统都有自己的 API。MCP（Model Context Protocol）提供了统一的标准接口——系统提供方实现 MCP Server，Agent 通过 MCP Client 调用。无需为每个系统写适配器。

学完后能做什么

用 Java 实现 MCP Server 暴露工具；用 MCP Client 发现和调用远程工具；配置 JWT 鉴权和 mTLS；实现超时控制和降级策略。

本章要点速记

1) MCP = Model Context Protocol，Anthropic 推出的标准工具协议；2) MCP Server 暴露工具列表，Client 发现并调用；3) 鉴权支持 JWT Token 和 mTLS；4) 超时控制分连接超时（5s）和读取超时（30s）；5) 降级策略：MCP 失败时切换到本地缓存或提示用户。

// 1. MCP Client 配置（Agent 端）
@Configuration
public class McpClientConfig {

    @Bean
    public McpClient mailMcpClient() {
        return McpClient.builder()
            .serverUrl("http://mail-mcp:8080")
            .transport(HttpClientTransport.builder()
                .connectTimeout(Duration.ofSeconds(5))
                .readTimeout(Duration.ofSeconds(30))
                .build())
            .authProvider(new JwtAuthProvider(
                "corp-agent", "secret-key"
            ))
            .build();
    }

    @Bean
    public McpClient calendarMcpClient() {
        return McpClient.builder()
            .serverUrl("https://calendar-mcp:8443")
            .transport(HttpClientTransport.builder()
                .sslContext(createMtlsContext())  // mTLS
                .build())
            .authProvider(new MtlsAuthProvider())
            .build();
    }
}

// 2. Agent 集成 MCP 工具
@Component
public class McpToolRegistry {

    private final List mcpClients;
    private final List toolCallbacks = new ArrayList<>();

    @PostConstruct
    public void discoverTools() {
        for (McpClient client : mcpClients) {
            // 发现所有可用工具
            List tools = client.listTools();
            for (McpTool tool : tools) {
                // 注册为 ToolCallback
                toolCallbacks.add(
                    new McpToolCallback(client, tool)
                );
            }
        }
        System.out.println("已发现 " + toolCallbacks.size() + " 个 MCP 工具");
    }

    @Bean
    public List mcpToolCallbacks() {
        return toolCallbacks;
    }
}

// 3. MCP Server 端（邮件系统方）
@SpringBootApplication
@EnableMcpServer
public class MailMcpServerApplication {

    @Bean
    public McpServer mailServer() {
        return McpServer.builder()
            .name("corp-mail-server")
            .version("1.0.0")
            .tools(List.of(
                new SendEmailTool(),
                new ReadInboxTool(),
                new SearchEmailTool()
            ))
            .build();
    }

    @Component
    public class SendEmailTool implements McpTool {
        @Override
        public String getName() { return "send_email"; }

        @Override
        public String getDescription() {
            return "发送邮件，支持 HTML 正文和附件";
        }

        @Override
        public JsonSchema getInputSchema() {
            return JsonSchema.builder()
                .addProperty("to", JsonSchema.string("收件人邮箱"))
                .addProperty("subject", JsonSchema.string("邮件主题"))
                .addProperty("body", JsonSchema.string("邮件正文"))
                .addRequired("to")
                .addRequired("subject")
                .build();
        }

        @Override
        public McpToolResult execute(Map args) {
            // 调用真实的邮件服务
            try {
                emailService.send(
                    (String) args.get("to"),
                    (String) args.get("subject"),
                    (String) args.get("body")
                );
                return McpToolResult.success("邮件已发送");
            } catch (Exception e) {
                return McpToolResult.error("发送失败: " + e.getMessage());
            }
        }
    }
}

工具注册与发现

MCP 规范定义了标准化的工具注册和发现流程。Server 端通过 McpServer.builder() 注册工具，Client 端通过 listTools() 发现可用工具。

MCP 方法方向说明
list_toolsClient → Server获取 Server 的所有工具列表
call_toolClient → Server调用指定工具并传入参数
list_resourcesClient → Server获取 Server 的资源列表（可选）
read_resourceClient → Server读取指定资源内容（可选）
ping双向健康检查
tool_resultServer → Client工具执行结果返回

鉴权与安全

MCP Server 需要鉴权防止未授权调用。支持 JWT Token 和 mTLS 双向认证。

java
// Server 端 JWT 鉴权过滤
@Component
public class McpAuthFilter implements WebFilter {

    private final JwtDecoder jwtDecoder;

    @Override
    public Mono filter(ServerWebExchange exchange, WebFilterChain chain) {
        String authHeader = exchange.getRequest()
            .getHeaders().getFirst("Authorization");

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        try {
            String token = authHeader.substring(7);
            Jwt jwt = jwtDecoder.decode(token);

            // 验证 Agent 身份
            String agentId = jwt.getClaimAsString("agent_id");
            String tenantId = jwt.getClaimAsString("tenant_id");

            // 注入到请求上下文
            McpContext.setCurrent(
                new McpContext(agentId, tenantId)
            );

            return chain.filter(exchange);
        } catch (JwtException e) {
            exchange.getResponse().setStatusCode(HttpStatus.FORBIDDEN);
            return exchange.getResponse().setComplete();
        }
    }

    // Client 端 JWT Provider
    @Component
    public class JwtAuthProvider implements AuthProvider {

        @Override
        public Map getAuthHeaders() {
            String token = Jwts.builder()
                .claim("agent_id", "corp-main-agent")
                .claim("tenant_id", TenantContext.getCurrent())
                .expiration(Date.from(
                    Instant.now().plus(5, ChronoUnit.MINUTES)))
                .signWith(SignatureAlgorithm.HS256, secretKey)
                .compact();
            return Map.of("Authorization", "Bearer " + token);
        }
    }
}

@Bean
public SslContext createMtlsContext() throws Exception {

        .keyManager(

            new File("client-key.pem")

        .trustManager(new File("ca-cert.pem"))

超时与降级

MCP 调用可能超时或失败。需要超时控制和降级策略保证 Agent 的可用性。

java
// 带超时和降级的 MCP Client 包装
@Component
public class ResilientMcpClient {

    private final McpClient delegate;
    private final Map localCache = new ConcurrentHashMap<>();

    public ResilientMcpClient(McpClient delegate) {
        this.delegate = delegate;
    }

    public McpToolResult callTool(String toolName, Map args) {
        try {
            // 超时控制
            McpToolResult result = delegate.callTool(toolName, args)
                .timeout(Duration.ofSeconds(10))
                .block();
            return result;
        } catch (TimeoutException e) {
            // 降级：使用缓存数据
            Object cached = localCache.get(toolName + ":" + args);
            if (cached != null) {
                return McpToolResult.success(cached,
                    "(缓存数据，可能不是最新)");
            }
            return McpToolResult.error("服务超时，请稍后重试");
        } catch (Exception e) {
            return McpToolResult.error("服务不可用: " + e.getMessage());
        }
    }

    // 健康检查
    public boolean isHealthy() {
        try {
            return delegate.ping().timeout(Duration.ofSeconds(2))
                .block() != null;
        } catch (Exception e) {
            return false;
        }
    }
}

// 熔断器
@Component
public class McpCircuitBreaker {

    private final Map failureCounts = new ConcurrentHashMap<>();
    private final Map circuitStates = new ConcurrentHashMap<>();
    private static final int THRESHOLD = 5;
    private static final long RESET_TIMEOUT_MS = 30000;

    public boolean isAllowed(String serverName) {
        if (circuitStates.getOrDefault(serverName, false)) {
            // 熔断已开启，检查是否到恢复时间
            return false;
        }
        return true;
    }

    public void recordFailure(String serverName) {
        int count = failureCounts.computeIfAbsent(
            serverName, k -> new AtomicInteger()).incrementAndGet();
        if (count >= THRESHOLD) {
            circuitStates.put(serverName, true);
            // 定时恢复
            scheduler.schedule(() -> {
                circuitStates.put(serverName, false);
                failureCounts.get(serverName).set(0);
            }, RESET_TIMEOUT_MS, TimeUnit.MILLISECONDS);
        }
    }

    public void recordSuccess(String serverName) {
        failureCounts.getOrDefault(serverName,
            new AtomicInteger()).set(0);
    }
}

CorpAssist 场景

邮件 MCP Server 部署在邮件系统侧，工单 MCP Server 部署在工单系统侧。Agent 启动时通过 list_tools 发现所有可用的工具。发送邮件时调用 mail-mcp 的 send_email 工具，查工单时调用 ticket-mcp 的 query_ticket 工具。如果某个 MCP Server 不可达（如工单系统维护中），Agent 的熔断器打开该 Server，调用降级逻辑返回"工单系统维护中"。

本章结论

  • MCP 协议通过 list_tools/call_tool 标准化工具调用，Server 和 Client 通过 HTTP/SSE 通信
  • 工具注册在 Server 端用 @EnableMcpServer 自动扫描，Client 端用 listTools() 发现
  • 鉴权支持 JWT Token（客户端身份验证）和 mTLS（双向证书认证）
  • 超时控制连接 5s + 读 30s，熔断器 5 次失败开启 30s，降级返回提示或缓存
  • CorpAssist 4 个 MCP Server（邮件/日历/工单/审批），Agent 通过 MCP Client 统一接入

面试可能问

MCP 协议和 OpenAPI（Swagger）有什么区别？为什么要用 MCP 而不是直接调 REST API？

MCP 和 OpenAPI 目标不同。OpenAPI 描述"静态的 HTTP API 接口"，适合手工或代码调用。MCP 描述"LLM 可调用的工具"，包含工具的描述、参数 Schema、调用逻辑，专门为 LLM 优化。区别：1) MCP 的 tool description 和参数 description 让 LLM 理解何时调用；2) MCP 的 list_tools 动态发现，OpenAPI 需要编译期生成；3) MCP 支持标准化鉴权握手。CorpAssist 中，底层系统接口用 OpenAPI 文档化，面向 Agent 的接口用 MCP。

MCP Server 如何保证工具执行结果的实时性？缓存怎么处理？

MCP 没有内置缓存机制。实时性由工具实现者保证。策略：1) 读操作（查询类）——MCP Server 内部可以做短时缓存（如 30 秒），减少重复查询；2) 写操作（创建/修改）——直接操作真实数据源，不缓存；3) 在工具返回结果中标记"cached_at"时间戳，让 Client 端知道数据的时效性。Agent 端根据查询类型判断：查询"我的工单"可以直接用缓存（30秒内），"发邮件"必须实时。

多个 Agent 共用一个 MCP Server 时，怎么防止相互影响？

隔离策略：1) Agent ID 标帜——每个 MCP 请求携带 X-Agent-Id header，Server 端按 Agent ID 做资源隔离（如 Redis key 前缀）；2) 租户隔离——Server 端从 JWT Token 提取 tenant_id，限制请求只能访问该租户的数据；3) 写锁——关键操作（创建工单）用分布式锁，防止两个 Agent 同时操作同一资源。MCP Server 可以拒绝未知 Agent ID 的请求。

判断练习

场景：MCP Server 返回了一个非常大的数据集（10000 条工单），Agent 的 Context Window 装不下，怎么解决？

三种方法组合：1) MCP Server 内部支持分页——工具参数加 page/size，Agent 先取第一页，再根据需要请求后续页；2) Server 端预聚合——返回统计摘要"共 10000 条，待处理 2000 条，已完成 8000 条"而不是返回所有明细；3) Agent 端的工具结果后处理——收到大结果后，先问用户"查到了 10000 条，需要看明细吗？"再决定是否全部加载。CorpAssist 优先用方案 2，必要是方案 1。

场景：MCP Client 发现两个 Server 都提供了 send_email 工具（重名），Agent 怎么选择？

Agent 端应该根据工具描述自动选择。两个 send_email 的描述应该不同，一个可能是"发送内部邮件，仅限公司内部通讯"，另一个是"发送客户邮件，包含合规检查和审批"。Agent 根据场景选择。如果描述不足以区分，在工具注册时加前缀（mail_internal_send、mail_customer_send）。MCP 规范不强制工具名全局唯一，但生产使用需要统一命名空间避免冲突。
