约 30 分钟 · 阶段：生产化与治理 · 能力：对比 CrewAI/AutoGen/Dify 的能力矩阵、对照 LangGraph 与 SAA Graph、选型决策、迁移评估

为什么学这章

多 Agent 框架在快速演进：Python 生态有 CrewAI、AutoGen、Dify、LangGraph，Java 生态有 Spring AI Alibaba 的 SAA Graph。选型时需要考虑团队技术栈、部署方式、功能覆盖度。本章提供完整的对比分析和迁移成本估算。

学完后能做什么

对比 CrewAI/AutoGen/Dify/SAA 四框架的能力矩阵；对照 LangGraph 和 SAA Graph 的异同；根据业务场景选择最合适的框架；估算从 Python 框架迁移到 Java 框架的成本。

本章要点速记

1) CrewAI 专注角色扮演，AutoGen 专注对话交互，Dify 提供低代码平台；2) LangGraph 是 Python 最成熟的多 Agent 编排框架；3) SAA Graph 与 LangGraph 概念兼容但功能覆盖约 80%；4) 选型决策因素：技术栈/部署方式/功能需求/团队经验；5) 迁移成本约 2-4 周，核心风险在 Graph 定义和状态管理。

CrewAI vs AutoGen vs Dify

三个 Python 多 Agent 框架各自有不同的设计哲学：CrewAI 角色扮演（Role-based）、AutoGen 对话交互（Conversation-based）、Dify 低代码平台（Workflow-based）。

CrewAI 在 Java 栈中的问题

CrewAI 的 Task 定义和 Process 调度在 Python 进程中运行，需要额外部署 Python 服务。与 Java 微服务的集成通过 HTTP 桥接（增加 50-100ms 延迟），故障排查需要在两个技术栈间切换。

Spring AI Alibaba 优势

与 Java 微服务同进程运行，自然集成 Nacos/Sentinel/Gateway 等中间件。同一个事务中完成 Agent 调用和业务逻辑，无跨进程开销。运维工具统一（Prometheus/Grafana/Skywalking）。

思考10秒：什么场景应该用 LangGraph（Python）而不是 SAA Graph（Java）？

1) 团队是 Python 技术栈，没有 Java 经验；2) 需要 LangSmith 的在线调试和 Prompt 管理功能；3) 需要 Checkpointer 的状态持久化和回放能力；4) 已经使用了 LangChain 生态（RAG/Agent/Tools）。Java 团队且需要与 Spring 微服务集成的场景选 SAA Graph。CorpAssist 选 SAA Graph，因为后端已经是 Java 微服务栈。

本章结论

  • CrewAI 角色+任务模式适合简单编排，AutoGen 对话交互适合复杂对话，Dify 适合非技术人员
  • LangGraph 是 Python 最成熟的多 Agent 框架，SAA Graph 与它概念兼容但功能覆盖约 80%
  • CorpAssist 选型 SAA Graph（8.1 分），Java 栈集成和生产化能力是核心优势
  • 从 Python 框架迁移到 SAA Graph 约 2-3 周，核心风险在 State 设计和路由逻辑
  • 如果团队是纯 Python 栈且需要 LangSmith 调试工具，LangGraph 是更好的选择

面试可能问

CrewAI 的 Crew 和 SAA Graph 在概念上有什么对应关系？

CrewAI 的 Crew 是一个"Agent 团队"，包含多个 Agent 和 Task 列表，Process 定义执行模式（顺序/层级）。在 SAA Graph 中，Crew 对应 CompiledGraph，Agent 映射到 Node，Task 映射到 Node 中的处理逻辑，Process 映射到 Edge/ConditionalEdge。Task 间的依赖关系在 CrewAI 中用上下文传递，在 SAA Graph 中用 State 传递。CrewAI 更抽象（只需定义 Agent 和 Task 即可运行），SAA Graph 更灵活（可以精细控制每一步的流转）。

AutoGen 的 GroupChat 和 SAA Graph 的多 Agent 编排有什么本质区别？

AutoGen 的 GroupChat 是基于"对话"的：多个 Agent 轮流发言，Manager LLM 决定谁发言。适合辩论、头脑风暴等需要多轮对话的场景。SAA Graph 是基于"工作流"的：预定义的节点和有向边决定执行路径。适合有明确步骤的业务流程。区别本质：AutoGen 是 Agent 间自由对话（不确定性强），SAA Graph 是预定义路径（确定性强）。CorpAssist 的审批流程更适合 Graph 的确定性，而创新讨论场景可能适合 GroupChat。

Dify 的低代码平台能否用于 CorpAssist 的 Agent？有什么局限性？

Dify 适合：1) 非技术人员创建简单的对话应用；2) 快速原型验证（几天内做出 MVP）；3) 不需要深度定制的工作流。局限性：1) 无法与 Java 微服务集成（Dify 是 Python 独立部署）；2) 自定义代码只能通过"代码节点"以 Python 脚本形式运行，无法调用 Java 类库；3) 高并发场景性能有限（单节点约 200 QPS）；4) 无法使用 Nacos/Sentinel/Spring Cloud 等中间件。CorpAssist 的 Java 后端用 Dify 不合适，但如果业务团队想自己搭建简单的 HR 问答机器人，Dify 可以单独部署使用。

判断练习

场景：项目中只用了 CrewAI 的 20% 功能（仅有顺序执行 Role + Task），迁移到 SAA Graph 的成本是增高还是降低？

成本降低。只用 20% 功能意味着迁移范围小，且 CrewAI 的顺序执行模式在 SAA Graph 中对应 addEdge（普通边），不需要条件边。迁移成本约 5-7 人天（比平均 11 天低）。而且迁移后能获得 Java 栈的 Nacos/Sentinel 等能力，并简化运维。但如果项目深度使用了 CrewAI 的 Process.hierarchical（层级处理），迁移成本可能增加。

场景：计划使用 Dify 的对话流 + 工作流构建简单 Agent，但未来需要深度定制。从 Dify 迁移到 SAA Graph 的难度？

从 Dify 迁移到 SAA Graph 难度较高，因为 Dify 的"低代码"隐藏了底层实现细节。迁移难点：1) Dify 的知识检索节点（RAG）需要重构为 LlamaIndex 或 Spring AI；2) Dify 的变量传递（Memory/Context）需要重新设计为 State；3) Dify 的代码节点用 Python 脚本编写，需要重写为 Java。建议：如果预见到未来需要深度定制，一开始就用 SAA Graph。
