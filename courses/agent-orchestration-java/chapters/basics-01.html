为什么学这章

CorpAssist 需要 Agent 来执行多步操作（查工单→查日程→发邮件），而不是简单问答。ReAct（Reasoning+Acting）循环是 Agent 最基础的设计模式——模型思考需要做什么、调用工具、观察结果、继续思考直到任务完成。Spring AI Alibaba 的 ReactAgent 实现了这个模式。

学完后能做什么

解释 ReAct 循环的四个阶段（思考/行动/观察/推理）；用 Spring AI Alibaba 定义 Agent 工具和 System Prompt；配置 maxIterations 和 Token 预算控制 Agent 行为；对比 ReAct 与 LangGraph 的工作流差异。

本章要点速记

1) ReAct 循环 = 思考(Thought)→行动(Action)→观察(Observation) 循环直到最终回答；2) 工具定义需包含 name/description/parameters 三要素；3) maxIterations 防止 Agent 无限循环（默认 10）；4) System Prompt 中固定 Agent 角色和行为边界；5) Spring AI Alibaba 的 ReactAgent 基于 Spring AI 的 AbstractToolCallSupport。

ReAct 思考-行动-观察

ReAct 循环是 Agent 的核心模式。每次循环：LLM 生成思考（Thought），决定调用哪个工具（Action），执行工具并获取结果（Observation），然后继续思考。循环持续到 LLM 认为任务完成并生成最终答案（Final Answer）。Spring AI Alibaba 的 ReactAgent 在 Spring AI 的 ChatClient 基础上增加了循环控制。

sequenceDiagram
    participant U as 用户
    participant AG as ReactAgent
    participant LLM as 大模型
    participant T as 工具
    U->>AG: "帮我查张三的工单状态"
    AG->>LLM: SystemPrompt + 用户请求
    LLM-->>AG: Thought: 需要查用户信息和工单
Action: getUserInfo("张三")
    AG->>T: getUserInfo("张三")
    T-->>AG: Observation: {userId: "u123", role: "IT"}
    AG->>LLM: 历史 + Observation
    LLM-->>AG: Thought: 知道了 userId
Action: queryTicket("u123")
    AG->>T: queryTicket("u123")
    T-->>AG: Observation: 有3个待处理工单
    AG->>LLM: 历史 + Observation
    LLM-->>AG: Final Answer: 张三有3个待处理工单...
    AG-->>U: 最终回答
java
// ReactAgent 基础配置
@Configuration
public class AgentConfig {

    @Bean
    public ChatClient chatClient(ChatClient.Builder builder) {
        return builder
            .defaultSystem("""
                你是一个企业办公助手。你有以下工具可用：
                1. getUserInfo(name) - 查询员工信息
                2. queryTicket(userId) - 查询工单状态

                请按照 ReAct 格式执行：先思考需要做什么，再调用工具。
                如果不需要工具，直接回答问题。
                """)
            .build();
    }

    @Bean
    public ReactAgent corpAgent(ChatClient chatClient,
                                 List tools) {
        return ReactAgent.builder()
            .chatClient(chatClient)
            .tools(tools)
            .maxIterations(10)        // 最大循环次数
            .maxTokenUsage(4096)      // 最大 Token 消耗
            .systemPrompt("你是一个企业办公助手。")
            .build();
    }
}

CorpAssist 场景

员工在门户输入"帮我查一下张三的工单状态，然后给张三发邮件提醒"，Agent 三步完成：1) getUserInfo 查张三的 userId；2) queryTicket 查工单；3) sendEmail 发提醒。每一步的 Observation 作为下一步的上下文，最终返回完整结果。

工具定义规范

Spring AI Alibaba 通过 @Tool 注解将 Java 方法暴露为 Agent 工具。框架自动提取方法签名和参数信息，生成 OpenAI 兼容的 Function Calling Schema。

java
import com.alibaba.spring.ai.tool.annotation.Tool;
import com.alibaba.spring.ai.tool.annotation.ToolParam;
import java.util.Map;

@Component
public class TicketTools {

    @Tool(name = "queryTicket", description = "根据工单编号或用户ID查询工单状态")
    public Map queryTicket(
            @ToolParam(description = "工单编号，格式 TKT-2026-XXX，可选") String ticketId,
            @ToolParam(description = "用户ID，可选") String userId) {

        // 模拟工单系统查询
        if (ticketId != null) {
            return Map.of(
                "ticketId", ticketId,
                "status", "处理中",
                "assignee", "张三",
                "createdAt", "2026-05-20"
            );
        }
        return Map.of("error", "请提供工单编号或用户ID");
    }

    @Tool(name = "getUserInfo", description = "根据姓名查询员工信息")
    public Map getUserInfo(
            @ToolParam(description = "员工姓名") String name) {
        return Map.of(
            "name", name,
            "userId", "u_001",
            "department", "IT",
            "email", name + "@corp.com"
        );
    }
}
@Tool 参数说明推荐值
name工具名，LLM 调用时使用的标识动词+名词，如 queryTicket
description工具描述，LLM 判断何时调用的依据包含场景和示例，20-50 字
returnDirect是否直接返回（跳过 Agent 二次处理）简单数据返回 true，否则 false

工具描述模糊

@Tool(description = "查询信息") — LLM 不确定该工具查什么，可能误调用。如查询日程时调用了工单查询工具。

工具描述清晰

@Tool(description = "根据工单编号或用户ID查询工单状态和详情，用于在客服场景跟进用户问题") — LLM 明确知道用途和场景。

停止条件与预算

Agent 需要预算控制防止无限循环和 Token 溢出。Spring AI Alibaba 提供了 maxIterations 和 maxTokenUsage 两个核心参数。

java
@Bean
public ReactAgent budgetAgent(ChatClient chatClient, List tools) {
    return ReactAgent.builder()
        .chatClient(chatClient)
        .tools(tools)
        // 停止条件配置
        .maxIterations(15)               // 最大 ReAct 循环次数
        .maxTokenUsage(8192)             // 单次对话最大 Token 数
        .maxToolCallAttempts(3)          // 单工具最大调用次数
        .toolRetryAttempts(2)            // 工具失败重试次数
        // 超时控制
        .toolTimeout(Duration.ofSeconds(30))
        // Token 预算耗尽时的行为
        .onTokenBudgetExhausted(session -> {
            // 生成摘要并返回当前进度
            return "Token 预算已用完，已处理的部分：" + session.getSummary();
        })
        .build();
}

// 监控 Agent 状态
@Component
public class AgentMonitor {
    private final MeterRegistry meterRegistry;

    public void recordIteration(String sessionId, int iterationCount) {
        meterRegistry.counter("agent.iterations", "session", sessionId)
            .increment(iterationCount);
    }

    public void recordTokenUsage(String sessionId, int tokens) {
        meterRegistry.gauge("agent.token_usage", tokens);
    }
}
参数默认值说明CorpAssist 推荐
maxIterations10ReAct 循环最大次数15（复杂任务 5-10 步）
maxTokenUsage4096单次对话最大 Token 消耗8192（含工具返回文本）
toolTimeout10s单个工具调用超时30s（外部 API 可能慢）
toolRetryAttempts0工具失败重试次数2（覆盖网络抖动）

与 LangGraph 概念对照

LangGraph 是 LangChain 的图形化 Agent 框架，用节点和边定义工作流。理解 ReAct 和 LangGraph 的对照关系，有助于在简单 Agent 和复杂工作流之间做选择。

概念Spring AI Alibaba ReactAgentLangGraph
执行模式循环（loop）直到条件满足有向图（DAG），节点可循环
状态管理隐式（对话历史）显式 State 对象，可持久化
条件分支LLM 通过 Thought 决定下一步条件边（conditional_edge）
记忆ChatHistory 自动管理State 中自定义记忆
并行执行不支持（顺序执行）支持（fan-out / fan-in）
调试可视化日志文本LangSmith / Mermaid 图
学习成本低（几小时）中（几天）
思考10秒：什么场景应该选 ReAct Agent 而不是 LangGraph？

ReAct Agent 适合：1) 单 Agent 顺序执行工具调用（CorpAssist 的办公助手场景）；2) 不需要复杂并行/条件分支；3) 团队对 Spring AI 熟悉。LangGraph 适合：1) 需要多个 Agent 协作（主管-工人模式）；2) 需要人在回路中断和恢复；3) 状态需要持久化和回放。简单原则：5 步以内的顺序工具调用用 ReAct，需要人工介入或多 Agent 用 LangGraph。

本章结论

  • ReAct 循环的四个阶段（思考/行动/观察/推理）驱动 Agent 多步推理，Spring AI Alibaba 通过 ReactAgent 实现
  • @Tool 注解定义工具，清晰的 description 是 LLM 正确调用工具的关键
  • 停止条件配置 maxIterations=15 + maxTokenUsage=8192 覆盖 CorpAssist 95% 的业务场景
  • ReAct Agent 适合单 Agent 顺序工具调用，LangGraph 适合多 Agent 协作和工作流编排
  • CorpAssist 办公助手（查工单/日程/邮件）是 ReAct Agent 的典型应用场景

面试可能问

ReAct Agent 陷入死循环时如何检测和恢复？

检测：1) 设置 maxIterations=10，到达后强制中断；2) 检测相同工具+参数的调用模式——连续 3 次调用同一个工具且参数相同，判定为死循环，中断并触发降级。恢复：1) 中断时返回当前已收集的信息（部分结果）；2) 在 System Prompt 中加入"如果在 5 步内没有进展，请直接总结已有的信息并提交"；3) 添加工具调用 dedup 逻辑——对相同输入返回缓存结果。

Spring AI Alibaba 的 ReactAgent 与 Spring AI 的 ChatClient 是什么关系？

ReactAgent 构建在 ChatClient 之上。ChatClient 提供与 LLM 的通信（组装 Prompt、处理 Response、管理 ChatHistory）。ReactAgent 在 ChatClient 的基础上增加了 ReAct 循环控制：每次 LLM 返回结果后，检查是否有 tool_calls，如果有则执行工具并将结果作为新的 Message 发送给 LLM，重复此过程直到 LLM 返回 Final Answer 或达到 maxIterations。相当于 ChatClient 是"一回合对话"，ReactAgent 是"多回合自动对话"。

工具返回的 Observation 太长导致 Token 超限怎么办？

三种策略：1) 截断——工具返回结果前截断到 1000 字符，超出部分用"{{截断}}..."标记；2) 摘要——工具返回结构数据时提取关键字段丢弃细节（如工单列表只返回数量和摘要不返回全文）；3) 压缩——Agent 在发送给 LLM 前自动压缩历史消息（Spring AI 的 ChatMemory 支持 summarize 模式）。CorpAssist 三种都用：工具层截断 + Agent 层摘要 + ChatMemory 压缩。

  4. 故意让工具返回错误，观察 Agent 的降级行为

判断练习

场景：Agent 在第三步循环中调用了 getUserInfo，但 Observation 显示用户不存在。接下来 Agent 应该怎么做？

Observation 返回 {"error": "用户不存在"} 后，LLM 的下一轮推理应该判断：用户不存在 → 可能是工单创建者已离职 → 跳过错信息 → 直接返回"该用户不存在，建议联系管理员"。或者：用户不存在但工单号码有效 → 跳过用户查询仅返回工单状态。Agent 不应该继续重试 getUserInfo 或死在这里，而应该调整计划继续执行或友好提示。

场景：需求从"查工单"变为"查工单→如果状态是待处理则给处理人发邮件→发完邮件创建一个跟进日程"，Agent 需要增加几步？

原始任务 1 步（queryTicket），扩展后需要：1) queryTicket 查状态；2) 如果是待处理则 getUserInfo 查处理人邮箱；3) sendEmail 发通知；4) createSchedule 创建跟进日程。共 4 步，其中步骤 2-4 有条件依赖。maxIterations 从 5 调到 10。System Prompt 需要补充"根据工单状态决定是否需要发邮件，发邮件后自动创建跟进日程"的规则。
