为什么学这章

ReAct Agent 适合简单的工具调用链，但 CorpAssist 中有更复杂的场景：审批流程需要等待人工确认、跨系统调用需要并行执行、条件分支需要根据结果选择不同路径。Spring AI Alibaba 的 Graph（基于 StateGraph）支持节点/边/条件路由，让复杂工作流可编排、可调试、可恢复。

学完后能做什么

用 StateGraph 定义节点和边；实现条件路由和状态管理；可视化 Graph 调试工作流；对比 Graph 与微服务 Saga 模式的异同。

本章要点速记

1) Graph = 节点(Node) + 边(Edge) + 条件边(ConditionalEdge)；2) State 在节点间传递，每个节点读取并更新 State；3) Spring AI Alibaba 的 StateGraph 兼容 LangGraph 的概念；4) 可视化调试用 mermaid 格式导出 Graph；5) 与 Saga 模式对照：Agent Graph 是编排型，Saga 是 choreography 型。

节点/边/条件路由

StateGraph 是有向图工作流：节点（Node）是工作单元，边（Edge）定义节点间的顺序，条件边（ConditionalEdge）根据 State 内容选择不同路径。

flowchart TD
    START([开始])
    START --> N1[查工单状态]
    N1 --> C1{状态判断}
    C1 -->|待处理| N2[查处理人信息]
    C1 -->|已完成| N4[生成完成报告]
    C1 -->|已关闭| N5[通知工单已关闭]
    N2 --> N3[发送催办邮件]
    N3 --> N6[创建跟进日程]
    N4 --> END
    N5 --> END
    N6 --> END
java
import com.alibaba.spring.ai.graph.*;
import com.alibaba.spring.ai.graph.state.*;
import java.util.Map;
import java.util.function.Function;

// 1. 定义 State（工作流的共享状态）
public record CorpState(
    String ticketId,
    String status,
    String assignee,
    String report
) implements AgentState {}

// 2. 定义节点
public class QueryTicketNode implements Node {
    @Override
    public CorpState apply(CorpState state) {
        // 模拟查工单
        String status = "待处理";  // 来自工单系统
        String assignee = "张三";
        return new CorpState(
            state.ticketId(), status, assignee, null
        );
    }
}

public class NotifyAssigneeNode implements Node {
    @Override
    public CorpState apply(CorpState state) {
        // 发送邮件通知
        String report = "已通知处理人: " + state.assignee();
        return new CorpState(
            state.ticketId(), state.status(),
            state.assignee(), report
        );
    }
}

// 3. 构建 Graph
public class TicketWorkflow {
    public static StateGraph buildGraph() {
        StateGraph graph = new StateGraph<>(CorpState::new);

        // 添加节点
        graph.addNode("query_ticket", new QueryTicketNode());
        graph.addNode("notify", new NotifyAssigneeNode());
        graph.addNode("report_completed", state ->
            new CorpState(state.ticketId(), state.status(),
                state.assignee(), "工单已完成")
        );
        graph.addNode("report_closed", state ->
            new CorpState(state.ticketId(), state.status(),
                state.assignee(), "工单已关闭")
        );

        // 添加入口和边
        graph.setEntryPoint("query_ticket");

        // 条件边：根据工单状态路由
        graph.addConditionalEdge(
            "query_ticket",
            (Function) state -> {
                return switch (state.status()) {
                    case "待处理" -> "notify";
                    case "已完成" -> "report_completed";
                    case "已关闭" -> "report_closed";
                    default -> "report_completed";
                };
            },
            Map.of(
                "notify", "notify",
                "report_completed", "report_completed",
                "report_closed", "report_closed"
            )
        );

        // 普通边：顺序执行
        graph.addEdge("notify", "report_completed");
        graph.setFinishPoint("report_completed");

        return graph;
    }
}

CorpAssist 场景

审批工作流中，State 包含 ticketId/status/approver/comment 字段。节点 checkPermission 检查用户权限后，ConditionalEdge 路由到 approve（自动通过）或 reject（转人工审批）。每个节点更新 State，最终 State 包含完整的审批链路日志。

状态定义与传递

AgentState 是 Graph 工作流的核心。它是一个不可变记录（record），每次节点处理返回一个新的 State 实例，保证状态变更可追溯、可回放。

java
// 复杂 State 定义（支持逐步累加）
public class WorkflowState {
    private final Map data = new ConcurrentHashMap<>();
    private final List logs = new ArrayList<>();
    private final List> artifacts = new ArrayList<>();

    public WorkflowState with(String key, Object value) {
        this.data.put(key, value);
        return this;
    }

    public  T get(String key, Class type) {
        return type.cast(data.get(key));
    }

    public WorkflowState log(String message) {
        this.logs.add(
            LocalDateTime.now() + " - " + message
        );
        return this;
    }

    public List getLogs() { return List.copyOf(logs); }
}

// 状态传递示例
public class ProcessTicketNode implements Node {
    @Override
    public WorkflowState apply(WorkflowState state) {
        String ticketId = state.get("ticketId", String.class);
        // 处理逻辑...
        return state
            .with("status", "processed")
            .with("processedAt", Instant.now().toString())
            .log("工单 " + ticketId + " 已处理");
    }
}

// 使用 Graph
public void executeWorkflow() {
    StateGraph graph = new StateGraph<>(WorkflowState::new);
    // ... 配置节点和边
    CompiledGraph app = graph.compile();

    WorkflowState result = app.run(
        new WorkflowState()
            .with("ticketId", "TKT-001")
            .with("userId", "u123")
    );
    System.out.println("日志:\n" + String.join("\n", result.getLogs()));
}

可视化调试

StateGraph 支持导出 Mermaid 格式用于可视化。调试时可以打印节点的输入输出 State 和耗时。

java
// 1. 导出 Mermaid 图
public void visualizeGraph(StateGraph graph) {
    String mermaid = graph.getGraph(Marker.SELECTED);
    System.out.println("=== Mermaid 流程图 ===");
    System.out.println(mermaid);
    // 输出示例：
    // flowchart TD
    //   __START__ --> query_ticket
    //   query_ticket -->|待处理| notify
    //   query_ticket -->|已完成| report
    //   notify --> report
    //   report --> __END__
}

// 2. 调试拦截器
public class DebugInterceptor implements NodeInterceptor {

    @Override
    public void beforeProcess(String nodeName, AgentState state) {
        System.out.printf("[DEBUG] 进入节点: %s | State: %s%n",
            nodeName, state);
    }

    @Override
    public void afterProcess(String nodeName, AgentState state,
                              long durationMs) {
        System.out.printf("[DEBUG] 离开节点: %s | 耗时: %dms%n",
            nodeName, durationMs);
    }

    @Override
    public void onError(String nodeName, Throwable error) {
        System.err.printf("[ERROR] 节点 %s 异常: %s%n",
            nodeName, error.getMessage());
    }
}

// 3. 启用调试
StateGraph graph = TicketWorkflow.buildGraph();
graph.addNodeInterceptor(new DebugInterceptor());
CompiledGraph app = graph.compile();

// 4. 运行并输出 trace
AgentState result = app.run(
    Map.of("ticketId", "TKT-001")
);
// 输出:
// [DEBUG] 进入节点: query_ticket | State: {ticketId=TKT-001}
// [DEBUG] 离开节点: query_ticket | 耗时: 152ms
// [DEBUG] 进入节点: notify | State: {ticketId=TKT-001, status=待处理}
// ...

与微服务 Saga 模式对照

Saga 是微服务中管理分布式事务的模式。Agent Graph 和 Saga 都解决跨步骤编排的问题，但设计哲学不同。

维度Agent StateGraph微服务 Saga
编排方式编排型（Orchestration）——一个中央编排器决定流程编排型 或 协作型（Choreography）
状态存储内存中 AgentState 对象数据库 Saga 日志表
补偿机制节点级别的错误处理 + 回退节点Compensation Transaction（回滚）
执行单元LLM 节点 + 工具节点微服务 API 调用
事务语义最终一致性（无强事务保证）Saga 保证最终一致性
适用场景AI Agent 工作流分布式事务/跨服务数据一致性
思考10秒：Agent Graph 和 Saga 可以结合使用吗？

可以。架构设计：Agent Graph 负责"决策"——判断工作流的路径、调用哪些工具、何时需要人工介入。Saga 负责"执行"——Agent Graph 中的工具节点调用微服务，微服务通过 Saga 保证数据一致性。例如 CorpAssist 的报销审批流程：Agent Graph 判断是否需要审批、推送到谁；Saga 保证审批通过后，财务系统和报销系统的数据一致性。

本章结论

  • StateGraph 由节点（工作单元）、边（顺序执行）、条件边（状态路由）三要素构成
  • AgentState 作为不可变记录在节点间传递，保证状态变更的完整性和可追溯性
  • Mermaid 可视化 + NodeInterceptor 调试器是定位工作流问题的核心手段
  • Agent Graph 是编排型工作流，与微服务 Saga 在不同抽象层次上互补
  • CorpAssist 审批/流转场景适合用 StateGraph 编排，工具执行部分可选 Saga 保证一致性

面试可能问

StateGraph 和 LangGraph 的主要区别是什么？

StateGraph（Spring AI Alibaba）是 LangGraph 的 Java 移植版本，概念上兼容：节点/边/条件边/State/EntryPoint/FinishPoint。主要区别：1) Java 版强类型——State 用 record 或 POJO 定义，编译期检查；2) 序列化——Java 版 State 需要可序列化（JSON/Java Serialization），支持跨节点传递；3) 集成——Java 版与 Spring AI 的 ChatClient/ToolCallback 无缝集成；4) 生态——LangGraph 的 checkpoint/回放功能更丰富。功能覆盖度 Java 版约 80%，核心流控能力一致。

Graph 工作流中节点执行失败怎么恢复？

三层恢复策略：1) 节点内部重试——通过 NodeInterceptor 的 onError 捕获异常，判断可重试的异常（网络超时）进行 3 次退避重试；2) 回退节点——在 Graph 中定义 error_handler 节点，ConditionalEdge 从失败节点路由到 error_handler，清理或回滚状态；3) 人工介入——复杂错误路由到 human_approval 节点，等待人工处理后继续。State 的不可变性保证了重试时可以从任意节点恢复。

多个条件边组合时出现死路（没有匹配的路径），Graph 会怎么处理？

StateGraph 的 compile 阶段会做静态分析，检查所有节点是否可达。如果有节点没有边连接，compile 会抛出 IllegalGraphException。运行时如果 ConditionalEdge 的返回值没有对应的边，框架会使用默认路由（如果设置了 defaultEdge）或抛出 NoEdgeMatchException。最佳实践：每个 ConditionalEdge 都设一个 defaultEdge 指向 error_handler 节点，处理未预期的状态值。

判断练习

场景：Graph 中有一个"审批"节点需要等待外部系统响应（异步回调），怎么设计？

方案：在审批节点中，将审批请求发送到外部系统后，State 标记为 pending_approval，ConditionalEdge 路由到 wait_node。wait_node 设置轮询或回调机制（CompletableFuture + 超时），每 5 秒检查审批状态。超时 60 秒后路由到 timeout_handler。审批完成后 ConditionalEdge 路由到 continue_processing。这个模式叫做"悬挂节点"——等待外部异步结果时 Graph 不阻塞，通过轮询/回调恢复执行。

场景：条件边的路由逻辑需要查询外部数据才能确定，怎么设计？

在 ConditionalEdge 的 Function 中调用外部服务是反模式——Lambda 不应有副作用。正确做法：增加一个"路由决策"节点，在节点中查询外部数据，将决策结果写入 State（如 routeTo = "path_a"）。ConditionalEdge 只从 State 中读取 routeTo 字段做简单判断。这样职责清晰：节点做"数据获取"，边做"路由决定"。
