为什么学这章

CorpAssist 的 Agent 不是单机运行——HR Agent、IT Agent、审批 Agent 分别部署在不同的微服务节点上，需要互相发现和通信。Nacos 作为注册中心管理 Agent 实例，A2A（Agent-to-Agent）协议定义 Agent 间的通信规范，网关统一入口。

学完后能做什么

设计分布式 Agent 架构；用 Nacos 注册和发现 Agent 实例；实现 Agent 间负载均衡；将 Agent 集成到微服务网关（Spring Cloud Gateway）。

本章要点速记

1) 每个 Agent 是一个微服务实例，注册到 Nacos；2) A2A 协议定义 Agent 间的通信格式（请求/响应）；3) 负载均衡策略：会话亲和性（同一用户固定到同一实例）；4) Spring Cloud Gateway 路由 /agent/* 请求到对应 Agent 服务。

分布式 Agent 将不同类型的 Agent 部署为独立的微服务，通过注册中心互相发现。每个 Agent 服务可以独立扩缩容。

@SpringBootApplication
@EnableDiscoveryClient
public class HrAgentApplication {

    public static void main(String[] args) {
        SpringApplication.run(HrAgentApplication.class, args);
    }

    @Bean
    public AgentMetadata agentMetadata() {
        return AgentMetadata.builder()
            .agentId("hr-agent")
            .agentName("HR 政策 Agent")
            .version("1.0.0")
            .capabilities(List.of(
                "query_hr_policy",
                "query_employee_info",
                "query_attendance"
            ))
            .maxConcurrentSessions(100)
            .build();
    }
}

服务发现与注册

Agent 通过 Nacos 注册自身信息，其他 Agent 或网关通过 Nacos 发现可用的 Agent 实例。

java
// Agent 服务发现客户端
@Component
public class AgentDiscoveryClient {

    private final DiscoveryClient discoveryClient;
    private final LoadBalancerClient loadBalancer;

    public AgentDiscoveryClient(DiscoveryClient discoveryClient,
                                 LoadBalancerClient loadBalancer) {
        this.discoveryClient = discoveryClient;
        this.loadBalancer = loadBalancer;
    }

    // 发现指定能力的 Agent
    public List findAgentsByCapability(String capability) {
        List agents = new ArrayList<>();

        // 从 Nacos 发现所有 agent 服务
        for (String serviceId : discoveryClient.getServices()) {
            if (!serviceId.endsWith("-agent")) continue;

            List instances =
                discoveryClient.getInstances(serviceId);

            for (ServiceInstance instance : instances) {
                String caps = instance.getMetadata()
                    .get("capabilities");
                if (caps != null && caps.contains(capability)) {
                    agents.add(new AgentInstance(
                        instance.getServiceId(),
                        instance.getHost(),
                        instance.getPort(),
                        instance.getMetadata()
                    ));
                }
            }
        }
        return agents;
    }

    // 选择最合适的 Agent 实例（负载均衡）
    public AgentInstance selectAgent(String capability) {
        List instances = new ArrayList<>();
        for (AgentInstance agent : findAgentsByCapability(capability)) {
            instances.add(agent.toServiceInstance());
        }
        if (instances.isEmpty()) {
            throw new AgentNotFoundException(
                "没有找到支持 " + capability + " 的 Agent");
        }
        ServiceInstance selected = loadBalancer.choose(
            instances.get(0).getServiceId());
        return AgentInstance.fromServiceInstance(selected);
    }
}

@Component
public class AgentHealthIndicator implements HealthIndicator {

    private final AgentMetrics metrics;

    @Override
    public Health health() {
        // 检查 LLM 连接
        boolean llmHealthy = checkLlmConnection();
        // 检查工具 API
        boolean toolsHealthy = checkToolsConnection();
        // 检查内存使用
        long memoryUsage = Runtime.getRuntime()
            .totalMemory() - Runtime.getRuntime().freeMemory();
        boolean memoryHealthy = memoryUsage < 1024 * 1024 * 1024; // <1GB

        if (llmHealthy && toolsHealthy && memoryHealthy) {
            return Health.up()
                .withDetail("llm", "connected")
                .withDetail("tools", "available")
                .withDetail("memory", memoryUsage)
                .build();
        }
        return Health.down()
            .withDetail("reason", llmHealthy ? "" : "LLM 连接失败")
            .withDetail("memory", memoryUsage)
            .build();
    }
}

负载均衡

Agent 实例间的负载均衡需要考虑会话亲和性——同一用户的多次请求尽量路由到同一实例（利用缓存的 ChatMemory）。

java
// 会话亲和性负载均衡
@Component
public class SessionAffinityLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private final Random random = new Random();
    private final Cache affinityCache =
        Caffeine.newBuilder()
            .expireAfterWrite(30, TimeUnit.MINUTES)
            .maximumSize(10000)
            .build();

    @Override
    public Mono> choose(Request request) {
        // 从请求中提取会话 ID
        String sessionId = extractSessionId(request);
        if (sessionId != null) {
            ServiceInstance cached = affinityCache.getIfPresent(sessionId);
            if (cached != null && isInstanceAlive(cached)) {
                return Mono.just(Response.of(cached));
            }
        }

        // 随机选择一个活跃实例
        List instances = getAliveInstances();
        if (instances.isEmpty()) {
            return Mono.just(Response.error(new EmptyInstanceException()));
        }
        ServiceInstance selected = instances.get(
            random.nextInt(instances.size()));

        // 缓存亲和性
        if (sessionId != null) {
            affinityCache.put(sessionId, selected);
        }
        return Mono.just(Response.of(selected));
    }

    private String extractSessionId(Request request) {
        // 从请求 header 或 cookie 提取 sessionId
        if (request.getHeaders() != null) {
            return request.getHeaders().getFirst("X-Session-Id");
        }
        return null;
    }
}
负载均衡策略适用场景效果
轮询（Round Robin）无状态 Agent请求均匀分布
最少连接（Least Connections）处理时间不均匀避免某个 Agent 负载过重
会话亲和性有状态 Agent（推荐）利用率略有倾斜但 ChatMemory 命中率高
加权响应时间异构实例快的实例收到更多请求

与微服务网关集成

Spring Cloud Gateway 作为统一入口，按照 Agent 类型路由到不同的 Agent 微服务。

// Spring Cloud Gateway 路由配置
@Configuration
public class AgentGatewayConfig {

    @Bean
    public RouteLocator agentRoutes(RouteLocatorBuilder builder) {
        return builder.routes()
            // HR Agent 路由
            .route("hr-agent", r -> r
                .path("/agent/hr/**")
                .filters(f -> f
                    .stripPrefix(2)
                    .addRequestHeader("X-Agent-Type", "hr")
                    .circuitBreaker(config -> config
                        .setName("hrAgentCB")
                        .setFallbackUri("forward:/fallback/agent-unavailable")))
                .uri("lb://hr-agent"))
            // IT Agent 路由
            .route("it-agent", r -> r
                .path("/agent/it/**")
                .filters(f -> f
                    .stripPrefix(2)
                    .addRequestHeader("X-Agent-Type", "it"))
                .uri("lb://it-agent"))
            // 审批 Agent 路由
            .route("approval-agent", r -> r
                .path("/agent/approval/**")
                .filters(f -> f.stripPrefix(2))
                .uri("lb://approval-agent"))
            .build();
    }

    // A2A 通信路由（Agent 之间）
    @Bean
    public RouteLocator a2aRoutes(RouteLocatorBuilder builder) {
        return builder.routes()
            .route("a2a-communication", r -> r
                .path("/a2a/**")
                .filters(f -> f
                    .stripPrefix(1)
                    .addRequestHeader("X-A2A-Protocol", "v1")
                    // A2A 鉴权
                    .filter(new A2AAuthGatewayFilter()))
                .uri("lb://a2a-router"))
            .build();
    }
}

// A2A 鉴权过虑器
@Component
public class A2AAuthGatewayFilter implements GatewayFilter {

    @Override
    public Mono filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String sourceAgent = exchange.getRequest()
            .getHeaders().getFirst("X-Source-Agent");
        String targetAgent = exchange.getRequest()
            .getHeaders().getFirst("X-Target-Agent");

        // 校验 source 和 target 都是注册的 Agent
        if (!isRegisteredAgent(sourceAgent) ||
            !isRegisteredAgent(targetAgent)) {
            exchange.getResponse().setStatusCode(
                HttpStatus.FORBIDDEN);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }
}

面试可能问

分布式 Agent 和单体 Agent 的优缺点是什么？什么场景需要分布式部署？

单体 Agent：所有能力在一个进程中。优点是开发简单、调试方便、没有网络延迟。缺点是扩容粒度粗（单一能力流量大时需要整体扩容）。分布式 Agent：每种能力独立部署。优点是隔离性好（HR Agent 故障不影响 IT Agent）、弹性伸缩精细（只扩容瓶颈 Agent）、技术栈可异构（HR Agent 用 Spring Boot，IT Agent 用 Python FastAPI）。但复杂度指数级增加（服务发现/网络/一致性）。CorpAssist 选择：初始阶段单体部署，3 个 Agent 服务后切到分布式。

Nacos 和 Eureka 作为 Agent 注册中心的区别？

Nacos 比 Eureka 更强大：1) 支持配置管理（可以动态更新 Agent 的参数而不用重启）；2) 支持权重路由（可以控制流量分配到不同版本的 Agent）；3) 支持临时/持久实例（持久实例即使心跳停止也保留记录，适合 Agent 维护场景）；4) 支持健康检查策略更多样（HTTP/TCP/MySQL）。CorpAssist 选择 Nacos 因为它也管理了其他微服务，统一注册中心。如果只用服务发现，Eureka 也够用。

A2A 通信协议怎么保证可靠性？掉消息了怎么办？

A2A 通信走 HTTP（同步）或消息队列（异步）。同步：超时重试（3次）+ 幂等性（请求 ID 去重）。异步：死信队列 + 补偿机制。对于 Agent 间的高价值通信（如工单状态变更），建议用消息队列：A Agent 发送消息到 RocketMQ → 主题过滤 → B Agent 消费。确保 at-least-once 语义。写操作（如"创建工单"）需要幂等——同一请求重复调用不产生两笔工单。消费者端用请求 ID 去重。

判断练习

场景：HR Agent 需要调用 IT Agent 的 queryTicket 工具，A2A 调用路径是什么？

完整路径：HR Agent → 内部 HTTP 请求 → Spring Cloud Gateway → 路由到 IT Agent → IT Agent 执行 queryTicket → 返回结果给 Gateway → Gateway 返回给 HR Agent。或者直接走 Nacos 服务发现：HR Agent 用 DiscoveryClient 找到 IT Agent 实例，直连（绕过网关但需要独立鉴权）。CorpAssist 走网关路径，便于统一鉴权和监控。

场景：Agent 实例滚动更新时，正在处理的请求会怎样？怎么保证不中断？

Nacos 在实例下线前有 graceful shutdown 机制：1) Agent 收到 SIGTERM 后，从 Nacos 反注册（不再接收新请求）；2) 处理完当前正在进行的请求（设置一个超时，如 30 秒）；3) 30 秒后如果还有请求未完成，中断并返回"Service updating, please retry"给客户端。网关端的 Spring Cloud Gateway 通过 discoveryClient 感知实例下线（约 5-10 秒）。加上重试机制：客户端收到 503 后等待 1 秒重试到新实例。
