行级权限的检索过滤 FilterExpression

企业知识库中，不同角色的用户只能看到不同范围的文档。权限过滤必须在检索层完成——不是在检索后丢弃无权结果，而是在检索时就通过 FilterExpression 限定。

java
@Service
public class SecureKnowledgeService {

    private final VectorStore vectorStore;

    public SearchResponse secureSearch(String query, UserPrincipal user) {
        // 1. 构建权限过滤条件
        FilterExpression.Builder filter = FilterExpression.builder();

        // 基础过滤: 安全等级
        if (user.hasRole("admin") || user.hasRole("vp")) {
            // 管理员和 VP 可以看到所有等级
            filter.in("security_level",
                List.of("public", "internal", "confidential"));
        } else if (user.hasRole("manager")) {
            // 经理可以看到 internal 及以上
            filter.in("security_level", List.of("public", "internal"));
        } else {
            // 普通员工只能看 public
            filter.eq("security_level", "public");
        }

        // 部门过滤: 员工只看自己部门的文档
        if (!user.hasRole("admin")) {
            filter.or(
                eq("department", user.getDepartment()),
                eq("department", "ALL")
            );
        }

        // 2. 执行带权限过滤的检索
        SearchRequest request = SearchRequest.builder()
            .query(query)
            .topK(5)
            .similarityThreshold(0.5)
            .filterExpression(filter.build())
            .build();

        return vectorStore.search(request);
    }
}

                                     HttpServletResponse response,
                                     FilterChain chain)

        String token = extractToken(request);

        chain.doFilter(request, response);

检索层过滤的安全原则：人在哪一层鉴权，就在哪一层过滤。如果在 Controller 层鉴权但不在检索层过滤，检索结果仍然包含了用户无权访问的文档——信息可能通过缓存、日志、错误信息等侧信道泄露。

审计日志：Spring AOP + MongoDB

知识库的每一次问答都必须可审计——谁、什么时间、问了什么、系统怎么回答的、引用了哪些文档。Spring AOP 提供了声明式的审计方案。

java
@Aspect
@Component
public class AuditLogAspect {

    private final AuditLogRepository auditLogRepository;

    @Around("@annotation(auditable)")
    public Object logAudit(ProceedingJoinPoint pjp,
                            Auditable auditable) throws Throwable {
        long startTime = System.currentTimeMillis();

        // 获取请求参数
        Object[] args = pjp.getArgs();
        String query = extractQuery(args);
        UserPrincipal user = getCurrentUser();

        // 执行原方法
        Object result = pjp.proceed();

        // 记录审计日志
        long duration = System.currentTimeMillis() - startTime;
        AuditLog log = AuditLog.builder()
            .timestamp(Instant.now())
            .userId(user.userId())
            .department(user.department())
            .query(query)
            .duration(duration)
            .build();

        if (result instanceof Answer answer) {
            log.setResponse(answer.content());
            log.setCitations(answer.citations());
            log.setConfidenceScore(computeScore(answer));
        }

        auditLogRepository.save(log);
        return result;
    }
}

审计日志的关键设计决策：

  • 存储选型 MongoDB——审计日志是写多读少、Schema 灵活的数据，MongoDB 比 MySQL 更适合
  • TTL 策略——审计日志保存 180 天（合规要求），到期自动删除（MongoDB TTL 索引）
  • 复合索引——userId + timestamp 用于单个用户的操作追溯
  • 数据脱敏——查询中包含个人身份信息时自动脱敏（如员工姓名 -> 工号）
  • 保留策略——180 天后归档到冷存储（S3/OSS），保留 7 年

多租户索引前缀隔离

SaaS 化部署时，多个租户共享同一个 Milvus/ES 集群。通过索引前缀实现数据隔离，比每个租户一个独立集群更经济。

java
@Component
public class TenantAwareVectorStore {

    private final MilvusServiceClient milvusClient;

    // 获取租户专属的 Collection 名称
    private String tenantCollection(String tenantId) {
        return "kb_" + tenantId + "_policies";
    }

    // 写入时按租户写入独立 Collection
    public void addDocuments(String tenantId,
                              List documents) {
        String collection = tenantCollection(tenantId);

        // 确保租户 Collection 存在
        ensureCollectionExists(collection);

        // 写入数据
        documents.forEach(doc ->
            doc.getMetadata().put("tenant_id", tenantId));
        vectorStore.add(documents);
    }

    // 检索时按租户检索
    public List search(String tenantId, String query) {
        String collection = tenantCollection(tenantId);

        SearchRequest request = SearchRequest.builder()
            .query(query)
            .topK(5)
            .filterExpression(
                FilterExpression.builder()
                    .eq("tenant_id", tenantId)
                    .build()
            )
            .build();

        return vectorStore.search(request);
    }

    @Bean
    @RequestScope
    public TenantContext tenantContext(HttpServletRequest request) {
        String tenantId = request.getHeader("X-Tenant-Id");
        return new TenantContext(tenantId);
    }
}

record TenantContext(String tenantId) {
    public boolean isValid() {
        return tenantId != null && !tenantId.isBlank();
    }
}

面试可能问

Q1: 多租户场景下，如果两个租户的文档都包含「产假天数」，怎么保证各自只搜到各自的文档？

两层保证：① 物理隔离——租户 A 的文档写入 collection "kb_tenant_a_policies"，租户 B 写入 "kb_tenant_b_policies"，写入时 metadata 中标记 tenant_id；② 检索过滤——检索时从请求 Header X-Tenant-Id 获取租户 ID，通过 FilterExpression.eq("tenant_id", tenantId) 过滤。即使误写了也不影响。

Q2: 审计日志量很大（日均 10 万条），存储成本高怎么办？

分层存储策略：① MongoDB 保留最近 30 天的热数据（快速查询）；② 30-180 天的温数据存储在 MongoDB 但缩采样（去掉 response 完整内容）；③ 180 天以上的冷数据压缩后存到 OSS/S3 对象存储。查询时先查 MongoDB，查不到再从 OSS 拉取。总存储成本降低约 70%。

Q1: 召回率从 60% 优化到 92% 的过程中，最大的挑战是什么？

Q2: 分块怎么选

面试官问：你的知识库分块参数怎么确定的？为什么这样选？

回答结构：

  1. 初始选择：chunk-size 512 + chunk-overlap 64，基于 Spring AI TokenTextSplitter 的 tiktoken 计算
  2. 实验验证：在 CorpAssist 的 846 页 HR 制度文档上做消融实验，对比 5 组参数（256/512/768/1024）。512 在 Recall@5(92.3%) 和回答完整性(4.5/5)上均为最优
  3. 特殊处理：表格和代码块保持完整性——表格问答准确率 +34%，技术文档问答 +28%
  4. 元数据标记：通过 MetadataEnricher 自动标记 content_type/security_level/policy_code，检索时用 FilterExpression 精确过滤
  5. 配置化：分块参数通过配置中心管理，不硬编码，支持运行时调优

加分项：提及「分块参数需要根据文档类型调整——操作手册类可以稍大（768），制度类 512 最优」。这展示了你的实际落地经验。

Q4: RAG vs 微调

面试官问：知识库为什么用 RAG 而不是微调？什么场景该用微调？

回答框架：

维度RAG微调
知识更新改文档后重新索引，分钟级生效需要标注->训练->评估->部署，以周计
幻觉可控强制引用原文片段，引用准确率 97.3%模型可能「记住」了但无法追溯来源
审计追溯每个回答附带 citations 列表，逐条可追溯无法追溯
适用场景知识频繁变化/内容多的制度问答格式固定/术语专有的场景（如合同模板生成）
成本每次问答消耗 Token（约 ¥0.18）一次性训练成本（数千元）+ 推理成本

结论：CorpAssist 的制度问答优先 RAG。微调仅在两种场景考虑：① 需要固定输出格式（如合同条款必须按固定模板输出）；② 需要理解深度行业术语（如法律文本解析）。实际落地中，80% 的场景用 RAG，20% 用 RAG + 微调增强。

Q5: 评估指标

面试官问：你怎么评估 RAG 系统的好坏？线上用什么指标监控？

评估分三层：

1. 检索层（自动化）

  • Recall@5：目标 >= 90%，黄金评测集周运行
  • NDCG@5：目标 >= 0.88，考虑排序位置的质量
  • MRR：目标 >= 0.85，第一个相关结果的排名

2. 生成层（自动化 + RAGAS）

  • Faithfulness（忠实度）：回答是否基于检索内容，RAGAS 评分目标 >= 0.85
  • 引用覆盖率：有引用支持的句子比例 >= 95%
  • 拒答准确率：该拒答时正确拒答 >= 98%

3. 体验层（人工）

  • CSAT：用户满意度评分，目标 >= 4.5/5.0
  • 人工抽检：每周随机抽检 100 条回答，标注正确/部分正确/错误
  • 坏例库：积累的 Bad Case 驱动持续优化

面试加分：提到「我们建立了从评估 -> 发现 Bad Case -> 归因 -> 修复 -> 回归的闭环流程，每周迭代一次。坏例库从最初的 200 条降到 30 条。」

CorpAssist 知识库模块的业务边界

CorpAssist S1 知识库模块有清晰的能力边界——超出边界意味着系统被滥用、幻觉率上升。

能力之内（In Scope）能力之外（Out of Scope）
制度/政策问答——HR 制度、财务报销、IT 规范等结构化文档实时数据查询——「我的年假余额」需对接 HR 系统（转 S3 Agent）
多轮追问与引用溯源——每次回答附带原文片段链接多轮深层对话——跨 10 轮以上的复杂对话由 S2 客服负责
权限过滤——按组织架构/角色过滤可见文档范围文档协作编辑——知识库只读消费，编辑走 OA 发布流程
版本管理——增量索引 + 蓝绿切换支持版本回滚非文本内容理解——图片/图表仅 OCR 后做文本索引

边界划分的工程意义：限定 RAG 的使用场景。超出边界的情况触发异常处理——要么返回「当前知识库无法回答」，要么通过意图路由转到其他模块。这样避免了 RAG 能力被滥用导致幻觉率上升。

不推荐
  • 知识库试图回答所有问题，包括实时数据查询
  • 不设权限过滤，全库检索后在后端丢弃无权文档
  • 文档更新靠全量重建索引，操作耗时数小时
推荐
  • 知识库明确定义检索范围，实时查询转交 Agent 模块
  • 在检索层用 FilterExpression 过滤权限
  • 增量更新 + 蓝绿索引切换，支持秒级版本回滚

面试可能问

Q1: 为什么知识库系统采用 RAG 而非微调？展开说说技术选型依据。

RAG 相比微调的三大核心优势：① 知识更新成本低——修改文档后重新索引即可生效，微调需要重新准备数据集、训练和部署模型，周期通常以周计；② 幻觉可控——回答基于检索片段生成，可通过 Prompt 约束强制引用原文，微调模型可能「记住」了训练数据但无法追溯来源；③ 审计可追溯——每个回答都能追溯到具体的文档和片段 ID。CorpAssist 实践中制度类问答优选 RAG，对专有名词理解可通过 LoRA 微调辅助增强。微调适合知识固定、需要深度理解的场景。

Q2: ROI 模型中的隐性收益怎么量化？给一个具体计算过程。

隐性收益的核心是员工等待时间缩短。以 5000 人企业日均 400 次问答为例：传统模式每次 8 分钟查找 + 回复 = 3,200 分钟/天；RAG 每次 15 秒首答 = 100 分钟/天；节省 3,100 分钟/天。按员工平均薪酬 ¥60/h 折算，日节省 ¥3,100，年化 ¥1,034,000。加上人力替代的显性节省（¥1,184,000/年），年综合收益约 ¥2,218,000。基础设施建设成本约 ¥300,000，投资回收期 = 300,000 / (2,218,000/12) ≈ 2.1 个月。

Q3: 双栈架构中 Java 和 Python 的职责边界如何划分？怎么确保两栈协作不混乱？

划分原则：Python 负责需要快速实验和 ML 生态的范围——检索参数消融实验、RAGAS 评测、Embedding 模型对比、Bad Case 分析等探索性工作。Java 负责需要企业治理的范围——REST API 服务、权限过滤、审计日志、配置中心集成、生产级高可用。两栈通过统一的 OpenAI 兼容模型网关 衔接，索引契约通过 REST API + JSON Schema 定义。POM 和 requirements.txt 各自独立维护，共用配置中心（Nacos/Apollo）下发参数。

判断练习

场景：项目初期 PM 提议「把所有文档的 PDF 直接塞进 Prompt，让大模型自己找答案。这样不用做分块和向量库，快速上线。」你怎么评估这个方案？

不推荐。原因：Token 成本爆炸——一个 200 页 PDF 约 60,000 Tokens × ¥0.15/1K = ¥9/次，日均 400 次 = ¥3,600/天；定位精度下降——大模型在长上下文中难以精准定位信息，实验中 200 页上下文的回答准确率仅 58%。正确做法：文档分块 → 向量化 → 检索 Top-K(5~10) → 只将高相关性片段送入 Prompt。Token 消耗降低 95% 以上，准确率提升约 30%。

场景：知识库上线后某 HR 制度的回答有 15% 的概率引用了已废止的 2019 年版而不是 2024 年版。怎么定位并解决？

根因：索引库中混入了旧版本文档，检索排序时旧版可能排在新版之前。解决方案：① 元数据中加入 document_version: "2024" 和 effective_date 字段；② 检索时通过 FilterExpression 只查询 version == "2024" 的文档；③ 采用蓝绿索引切换——新版文档入新索引（green），旧索引（blue）设置权重衰减并在验证完成后删除；④ 对已索引历史数据做增量重索引。

CorpAssist 知识库的数据流设计

数据流分为写入流（文档索引 Pipeline）和读取流（问答检索 Pipeline），两者通过索引存储解耦。

写入流（文档索引 Pipeline）

  1. OA 系统通过 Webhook 通知文档变更 → 发送 MQ 消息到 corpassist-doc-ingest 队列
  2. 索引服务消费消息，调用 TikaDocumentReader 解析文档
  3. TokenTextSplitter 分块（chunk-size: 512, overlap: 64）
  4. MetadataEnricher 提取元数据（作者/部门/版本/权限标签）
  5. 调用 Embedding API 向量化后并行写入 Milvus + ES
  6. 确认双写成功后 ACK 消息；失败进入 DLQ 重试

  5. Prompt 模板将引用片段拼入 System Message + User Message
  6. ChatClient 调用 LLM 生成回答 + 引用溯源

面试可能问

Q1: 三层架构中编排层和接入层的职责边界在哪里？

接入层负责非智能的透传职责：JWT 鉴权、SSE 流式转发、模型路由、API Key 托管、用量落库。编排层负责智能的组装职责：RAG 检索链调用、Prompt 模板版本管理、多轮对话状态维护、工具调用编排与降级。关键约束：编排层不持有厂商 API Key，不暴露公网；接入层不做 RAG 核心逻辑。

Q2: 为什么选 Milvus + ES 双路而不是单一向量库？

纯向量库的缺点：① 关键词精确匹配弱，特殊编号（HR-2024-001）无法精确命中；② 元数据过滤依赖向量库实现，不如 ES 成熟。双路的优势：向量召回语义相似文档 + BM25 精确匹配编号/专有名词 + RRF 融合排序。实际业务中 Recall@5 从 84.1% 提升到 92.3%，代价是额外 15-25ms 延迟。

Q3: 数据写入流被脏文档堵住了怎么办？

生产级方案：① MQ 消费端设置 max-retry: 3 和死信队列 DLQ；② 脏文档进入 DLQ 后通过管理后台人工审核和重试；③ DocumentReader 设置 skip-on-error: true 跳过异常文档；④ 监控仪表板跟踪索引队列积压和失败率；⑤ 定期巡检 DLQ 并实现半自动重放机制。

DocumentReader 与 Splitter 配置

Spring AI 的 DocumentReader 是文档解析入口，支持 Tika/PDFBox/JsonReader 等实现。Splitter 负责将解析后的文档切分成适合检索的片段。

java
@Service
public class DocumentIngestionService {

    private final ResourcePatternResolver resolver;
    private final VectorStore vectorStore;
    private static final Logger log = LoggerFactory.getLogger(DocumentIngestionService.class);

    public DocumentIngestionService(VectorStore vectorStore,
                                     ResourcePatternResolver resolver) {
        this.vectorStore = vectorStore;
        this.resolver = resolver;
    }

    public int ingestPdf(String path) {
        // 1. 读取文档
        Resource resource = resolver.getResource("file:" + path);
        DocumentReader reader = new TikaDocumentReader(resource);
        List documents = reader.read();

        // 2. 分块 - 使用 TokenTextSplitter
        TokenTextSplitter splitter = TokenTextSplitter.builder()
            .withChunkSize(512)
            .withChunkOverlap(64)
            .withSeparators(".", "\n", " ", "")
            .build();
        List chunks = splitter.apply(documents);

        // 3. 添加元数据
        Instant now = Instant.now();
        chunks.forEach(doc -> {
            doc.getMetadata().put("source", path);
            doc.getMetadata().put("indexed_at", now.toString());
            doc.getMetadata().put("department", extractDepartment(doc));
            doc.getMetadata().put("version", extractVersion(doc));
            doc.getMetadata().put("doc_id", UUID.randomUUID().toString());
        });

        // 4. 写入 VectorStore
        vectorStore.add(chunks);
        log.info("Ingested {} chunks from {}", chunks.size(), path);
        return chunks.size();
    }

    private String extractDepartment(Document doc) {
        String text = doc.getText();
        if (text.contains("人力资源") || text.contains("HR")) return "HR";
        if (text.contains("财务") || text.contains("FIN")) return "FIN";
        if (text.contains("信息") || text.contains("IT")) return "IT";
        return "GENERAL";
    }

    private String extractVersion(Document doc) {
        Pattern p = Pattern.compile("版本[号：:]*\\s*(\\d+\\.\\d+)");
        Matcher m = p.matcher(doc.getText());
        return m.find() ? m.group(1) : "unknown";
    }
}

TokenTextSplitter 的关键参数：chunk-size 控制每个分块的最大 Token 数，chunk-overlap 控制相邻块的重叠 Token 数。Spring AI 的 TokenTextSplitter 基于 tiktoken 计数，比字符分块更准确。对于中文内容，512 Tokens ≈ 300-350 个汉字，重叠 64 Tokens ≈ 40 个汉字。

表格与代码块的元数据标记

企业文档中大量包含表格（政策对照表）和代码片段（IT 规范示例）。分块时应通过元数据标记特殊内容类型，方便检索时做精细过滤。

java
@Component
public class MetadataEnrichmentProcessor {

    public Document process(Document document) {
        String text = document.getText();
        Map meta = document.getMetadata();

        // 标记表格片段 -- 检测 Markdown 表格或文本对齐行
        if (text.contains("|---") && text.lines().anyMatch(l -> l.contains("|"))) {
            meta.put("content_type", "table");
            meta.put("table_headers", extractTableHeaders(text));
        }

        // 标记代码片段 -- 检测代码块标记或编程语言关键字
        if (text.contains("```") || hasCodePatterns(text)) {
            meta.put("content_type", "code_block");
            meta.put("language", detectLanguage(text));
        }

        // 标记制度编号 -- 如 HR-2024-001, IT-SEC-03
        Pattern policyPattern =
            Pattern.compile("(HR|IT|FIN|OPS)-(\\d{4}-\\d{2})");
        Matcher matcher = policyPattern.matcher(text);
        if (matcher.find()) {
            meta.put("policy_code", matcher.group(0));
        }

        // 标记安全等级
        if (text.contains("保密") || text.contains("机密")) {
            meta.put("security_level", "confidential");
        } else if (text.contains("内部")) {
            meta.put("security_level", "internal");
        }

        return document;
    }

    private boolean hasCodePatterns(String text) {
        return text.contains("class ") || text.contains("public ") ||
               text.contains("@Bean") || text.contains("interface ") ||
               text.contains("function ") || text.contains("def ");
    }

    private String detectLanguage(String text) {
        if (text.contains("public class")) return "java";
        if (text.contains("def ") || text.contains("import ")) return "python";
        if (text.contains("application:")) return "yaml";
        return "unknown";
    }

    private List extractTableHeaders(String text) {
        return text.lines()
            .skip(1).limit(1)
            .flatMap(l -> Arrays.stream(l.split("\\|")))
            .map(String::trim)
            .filter(s -> !s.isEmpty() && !s.startsWith("-"))
            .collect(Collectors.toList());
    }
}

面试可能问

Q1: chunk-size 过大或过小分别有什么影响？最佳值怎么确定？

过小（128）：片段语义不完整，回答准确率下降约 12%；索引数量激增，检索延迟上升。过大（1024+）：单块包含多个主题，检索命中噪声增多，回答相关性下降约 8%；Token 消耗增加。最佳值的确定方法：准备黄金评测集，在 5-8 组参数组合上计算 Recall@K，选择 Recall 最高且回答完整性评分最好的参数。CorpAssist 在 8 个客户项目中验证 512 是最佳平衡点。

Q2: 怎么判断当前的分块策略是否合适？出了线上问题怎么排查？

三步诊断法：① 检索质量评估——对 100 个测试问题计算 Recall@K，如果 <85% 则分块可能过细或过粗；② 边界采样——随机查看 20 个 chunk 边界前后各 50Tokens 的内容，如果频繁出现截断的句子、未闭合的表格、不完整的代码，则 overlap 不足或需要语义分块；③ A/B 对比——在线上切 10% 流量用新分块参数，对比回答采纳率和用户反馈。

Q3: 表格和代码块为什么要特殊处理？不加处理会怎么样？

TokenTextSplitter 按分隔符和 Token 数机械切分，不加干预会把表格的行、代码的行切到不同 chunk 中。检索时单块语义不完整——用户问「表格中产假和陪产假的天数」时，检索只命中了表格的一部分行，回答只能基于不完整信息生成。解决方案：在 MetadataEnricher 中检测内容类型，对表格和代码块做完整性约束，或者使用 SemanticSplitter 按自然语义边界切分。

判断练习

场景：分块后用 FilterExpression 过滤无效片段，比不分块直接检索更好吗？

同意。分块将长文档切分为可定位的信息单元（chunk），FilterExpression 进一步精确限定范围（部门/版本/类型）。两者结合 = 粒度控制 + 精度控制。不分块直接检索长文档，检索系统会返回整篇文档的向量表示，其中的噪声信息会严重稀释回答质量。

场景：分块参数确定后文档做了修改，需要重建索引吗？生产环境怎么减少影响？

Spring AI VectorStore + Elasticsearch 混合检索

混合检索结合了向量语义检索和 BM25 关键词匹配的优势，是生产级 RAG 系统的标配。

java
@Service
public class HybridRetrievalService {

    private final MilvusVectorStore milvusStore;
    private final ElasticsearchVectorStore esStore;
    private final EmbeddingModel embeddingModel;

    public HybridRetrievalService(
            @Qualifier("milvusVectorStore") MilvusVectorStore milvusStore,
            @Qualifier("elasticsearchVectorStore") ElasticsearchVectorStore esStore,
            EmbeddingModel embeddingModel) {
        this.milvusStore = milvusStore;
        this.esStore = esStore;
        this.embeddingModel = embeddingModel;
    }

    public List hybridSearch(String query, int topK) {
        // 1. 向量检索 (Milvus ANN)
        SearchRequest vectorRequest = SearchRequest.builder()
            .query(query)
            .topK(topK * 2)
            .similarityThreshold(0.5)
            .build();
        List vectorResults = milvusStore.search(vectorRequest);

        // 2. 关键词检索 (ES BM25)
        SearchRequest keywordRequest = SearchRequest.builder()
            .query(query)
            .topK(topK * 2)
            .build();
        List keywordResults = esStore.search(keywordRequest);

        // 3. RRF 融合排序
        return ReciprocalRankFusion.merge(
            vectorResults, keywordResults, topK,
            FusionWeights.builder()
                .vectorWeight(0.7)
                .keywordWeight(0.3)
                .build()
        );
    }
}

RRF（Reciprocal Rank Fusion）的核心公式：score(d) = w_v / (k + rank_v(d)) + w_k / (k + rank_k(d))。其中 k 为融合常量（通常 60），rank_v 和 rank_k 是文档在向量路和关键词路的排名位置。这种方法不需要训练，实现简单且效果稳定。

java
public class ReciprocalRankFusion {
    private static final int K = 60;

    public static List merge(
            List vecResults,
            List keyResults,
            int topK,
            FusionWeights weights) {

        Map scoreMap = new LinkedHashMap<>();
        Map docMap = new LinkedHashMap<>();

        // 计算向量路的 RRF 分数
        for (int i = 0; i < vecResults.size(); i++) {
            Document doc = vecResults.get(i);
            String id = doc.getId();
            double score = weights.vectorWeight() / (K + i + 1);
            scoreMap.merge(id, score, Double::sum);
            docMap.putIfAbsent(id, doc);
        }

        // 计算关键词路的 RRF 分数
        for (int i = 0; i < keyResults.size(); i++) {
            Document doc = keyResults.get(i);
            String id = doc.getId();
            double score = weights.keywordWeight() / (K + i + 1);
            scoreMap.merge(id, score, Double::sum);
            docMap.putIfAbsent(id, doc);
        }

        // 按总分降序排序，取 topK
        return scoreMap.entrySet().stream()
            .sorted(Map.Entry.comparingByValue().reversed())
            .limit(topK)
            .map(e -> docMap.get(e.getKey()))
            .collect(Collectors.toList());
    }
}

@Value
@Builder
public class FusionWeights {
    double vectorWeight;   // 默认 0.7
    double keywordWeight;  // 默认 0.3
}

BGE-Reranker 的 HTTP 调用封装

Reranker 使用交叉编码器架构——query 和每个 document 拼接后输入模型做深度交互打分，比向量相似度（双编码器）精确得多。代价是推理速度较慢。

java
@Service
public class RerankService {

    private final RestClient restClient;

    public RerankService(
            @Value("${corpassist.rerank.url}") String url,
            @Value("${corpassist.rerank.timeout:5000}") int timeout) {
        this.restClient = RestClient.builder()
            .baseUrl(url)
            .defaultHeader("Content-Type", "application/json")
            .requestFactory(new JdkClientHttpRequestFactory(
                HttpClient.newBuilder()
                    .connectTimeout(Duration.ofSeconds(5))
                    .build()))
            .build();
    }

    public List rerank(String query, List candidates) {
        if (candidates.isEmpty()) return candidates;

        RerankRequest request = new RerankRequest(
            query,
            candidates.stream().map(Document::getText).toList()
        );

        try {
            RerankResponse response = restClient.post()
                .uri("/rerank")
                .body(request)
                .retrieve()
                .body(RerankResponse.class);

            // 按相关性分数降序排列，取 Top-5
            List indices = response.indices();
            List scores = response.scores();

            List reranked = new ArrayList<>();
            for (int i = 0; i < Math.min(5, indices.size()); i++) {
                Document doc = candidates.get(indices.get(i));
                doc.getMetadata().put("rerank_score", scores.get(i));
                reranked.add(doc);
            }
            return reranked;
        } catch (Exception e) {
            log.warn("Rerank failed, falling back to original order", e);
            return candidates.subList(0, Math.min(5, candidates.size()));
        }
    }
}

@JsonIgnoreProperties(ignoreUnknown = true)

@JsonIgnoreProperties(ignoreUnknown = true)

面试可能问

Q1: 向量检索和 BM25 各自擅长什么场景？为什么需要两者结合？

向量检索擅长语义相似度匹配——用户说「年假」而文档写「带薪年休假」，向量距离仍然小。BM25 擅长精确关键词匹配——「HR-2024-001」这类编号只有 BM25 能精确命中。企业知识库中既有自由表述（「产假多少天」）又有精确代码（「查阅 HR-2024-001 号制度」），两者缺失任何一个都会造成大量漏召回。双路结合后 Recall@5 从 84.1% 提升到 92.3%。

Q2: Reranker 的延迟开销怎么接受？怎么评估值不值？

Reranker 延迟增量约 200-400ms（P99），但能带来 Recall@5 +8.2%。CorpAssist 把 Reranker 作为第二阶段精排：第一阶段宽召回 50 条用轻量级的 RRF 融合（~20ms），第二阶段只对 50 条做交叉编码器重排（~250ms）。整体链路 P99 约 1.8s，对知识库问答场景是可接受的。如果延迟压力大，可以：① 减少 Rerank 候选数（50 -> 20）；② 用更轻量的 Reranker（如 BGE-Reranker-v2-minicpm）；③ 对高频查询加缓存跳过 Reranker。

Q3: 怎么评估当前的检索召回率是否达标？怎么建立评测体系？

建立黄金评测集的过程：① 从生产日志中采集 200 个真实用户查询；② 为每个查询标注其对应的正确答案文档 ID（人工标注，可众包）；③ 定义一个「相关」标准（精确匹配、语义匹配、部分匹配三级）；④ 对每个查询计算 Recall@K 和 NDCG@K。CorpAssist 标准：Recall@5 >= 90%，NDCG@5 >= 0.88。低于标准触发优化流程——调整分块参数、双路权重或 Reranker 模型。

场景：MVP 阶段团队想跳过 Reranker，只用双路 RRF 融合 + Prompt。你觉得可行吗？有什么风险？

场景：线上发现某高频查询（占 10% 流量）的召回率只有 60%，其他查询正常。怎么排查？

            .withChunkOverlap(props.getChunking().getChunkOverlap())
            .withSeparators(props.getChunking().getSeparators()
                .toArray(new String[0]))
            .build();
    }

    public DocumentTransformer metadataTransformer(AppVersion appVersion) {
        return documents -> documents.stream()
            .peek(doc -> doc.getMetadata().put("ingested_at",
                Instant.now().toString()))

                appVersion.getVersion()))
            .peek(doc -> doc.getMetadata().put("doc_id",
                UUID.randomUUID().toString()))

元数据提取与权限标签

企业文档的元数据不仅用于检索过滤，更用于行级安全控制。以下是 CorpAssist 的元数据模型和自动提取实现。

java
@Component
public class PermissionTagEnricher implements DocumentTransformer {

    @Override
    public List transform(List documents) {
        return documents.stream()
            .map(this::enrich)
            .collect(Collectors.toList());
    }

    private Document enrich(Document doc) {
        String text = doc.getText();
        Map meta = doc.getMetadata();

        // 1. 基于内容自动推断安全等级
        if (text.contains("保密") || text.contains("机密") || text.contains("薪酬")) {
            meta.put("security_level", "confidential");
            meta.put("allowed_roles", List.of("admin", "vp", "hr_manager"));
        } else if (text.contains("内部") || text.contains("仅限")) {
            meta.put("security_level", "internal");
            meta.put("allowed_roles", List.of("admin", "manager", "employee"));
        } else {
            meta.put("security_level", "public");
            meta.put("allowed_roles", List.of("*"));
        }

        // 2. 从文档路径推断所属部门
        String path = (String) meta.get("source_path");
        if (path != null) {
            for (String dept : List.of("HR", "IT", "FIN", "OPS", "LEGAL")) {
                if (path.contains("/" + dept + "/")) {
                    meta.put("department", dept);
                    break;
                }
            }
        }

        // 3. 提取文档标题（首行或 Markdown H1）
        String firstLine = text.lines().findFirst().orElse("");
        if (!firstLine.isEmpty() && firstLine.length() < 100) {
            meta.put("title", firstLine.replace("# ", "").trim());
        }

        // 4. 计算内容哈希用于去重
        meta.put("content_hash",
            DigestUtils.sha256Hex(text.substring(0, Math.min(1000, text.length()))));

        return doc;
    }
}

// 检索时注入用户权限条件
@Component
public class SecureSearchService {

    public SearchRequest buildSecureRequest(String query, UserContext user) {
        FilterExpression.Builder filter = FilterExpression.builder();

        // 角色过滤
        filter.in("allowed_roles", user.getRoles());

        // 部门过滤
        if (user.getDepartment() != null) {
            filter.eq("department", user.getDepartment());
        }

        return SearchRequest.builder()
            .query(query)
            .topK(10)
            .filterExpression(filter.build())
            .build();
    }
}

权限标签的核心价值：检索时通过 FilterExpression 自动注入用户角色和部门条件，确保「用户只能看到其有权访问的文档」。如果在检索层不做过滤而只靠后端丢弃结果，属于安全设计缺陷——信息可能通过检索日志、缓存等侧信道泄露。

生产实践中，自动推断只做初标，标记结果需人工确认后方可生效。安全团队定期抽查标记准确率。

脏文档处理策略

企业文档质量参差不齐——扫描件 OCR 错误、表格格式混乱、图片型 PDF 无法解析。脏文档是最容易被低估的线上问题源头。

脏文档类型现象处理策略
扫描件（图片PDF）Tika 解析结果为空或乱码先用 OCR 服务（PaddleOCR / Tesseract）处理后再进 Pipeline
加密/密码保护读取时抛出 IOException记录到 DLQ，通知管理员解密后重新上传
超大文档(>200页/50MB)解析耗时 >30s，OOM 风险按页分批处理，设置 max-pages: 100，max-file-size: 50MB
损坏文件POI/Tika 解析异常转 PDF 后重试，连续失败 3 次进入人工审核
非中文编码乱码文本自动检测编码（ICU4J），转换失败记录日志并跳过
java
@Service
public class DirtyDocumentHandler {
    private static final int MAX_PAGES = 100;
    private static final long MAX_FILE_SIZE_BYTES = 50 * 1024 * 1024L;
    private static final int MAX_RETRIES = 3;

    private final OcrService ocrService;
    private final DeadLetterSender dlqSender;
    private final RetryTemplate retryTemplate;

    public IngestionResult handleWithRetry(Resource resource) {
        return retryTemplate.execute(ctx -> {
            try {
                return handle(resource);
            } catch (Exception e) {
                log.warn("Attempt {} failed for {}",
                    ctx.getRetryCount(), resource.getFilename(), e);
                if (ctx.getRetryCount() >= MAX_RETRIES - 1) {
                    dlqSender.send(new DirtyDocumentEvent(
                        resource.getFilename(), e.toString()));
                    return IngestionResult.failed(resource.getFilename(),
                        "Max retries exceeded: " + e.getMessage());
                }
                throw e;
            }
        });
    }

    public IngestionResult handle(Resource resource) throws Exception {
        // 前置校验
        validateFileSize(resource);

        // OCR 预处理（图片类 PDF）
        if (isImageBasedPdf(resource)) {
            log.info("OCR preprocessing for: {}", resource.getFilename());
            resource = ocrService.preprocess(resource);
        }

        // 按页分批（超大文档）
        if (isLargeDocument(resource)) {
            return batchProcessByPage(resource);
        }

        return processNormally(resource);
    }

    private void validateFileSize(Resource resource) throws IOException {
        if (resource.contentLength() > MAX_FILE_SIZE_BYTES) {
            throw new IllegalArgumentException(
                "File exceeds max size: " + resource.getFilename());
        }
    }

    private boolean isImageBasedPdf(Resource resource) throws IOException {
        // 通过 PDF 元数据判断是否扫描件
        try (var reader = new com.itextpdf.kernel.pdf.PdfReader(
                resource.getInputStream())) {
            var doc = new com.itextpdf.kernel.pdf.PdfDocument(reader);
            // 如果所有页面都不含文本内容，判断为扫描件
            for (int i = 1; i <= Math.min(doc.getNumberOfPages(), 5); i++) {
                var page = doc.getPage(i);
                var text = com.itextpdf.kernel.pdf.canvas.parser.PdfTextExtractor
                    .getTextFromPage(page);
                if (!text.isBlank()) return false;
            }
            return true;
        }
    }
}

脏文档处理的基本原则：永远不让脏文档阻塞 Pipeline 主链路。CorpAssist 的策略是「跳过 -> 记录 -> 通知 -> 人工复核」，保证 99.9% 的正常文档及时更新。DLQ 中的脏文档通过独立的人工审核流程处理。

本章结论

  • 索引 Pipeline 四阶段：Read（TikaDocumentReader）-> Split（TokenTextSplitter）-> Transform（MetadataEnricher）-> Write（VectorStore），通过 MQ 异步触发
  • 元数据模型包含权限标签（security_level / allowed_roles / department），由自动推断 + 人工确认的结合方式生成，检索时通过 FilterExpression 实现行级安全
  • Java-Python 索引契约通过 OpenAPI 规范定义（POST /api/v2/index/batch），解耦两栈开发
  • 脏文档五大类型各有处理策略，核心原则是跳过 + DLQ + 人工复核，不阻塞主链路
  • 自动权限推断只做初标，最终结果需人工确认，安全团队定期抽查标记准确率

面试可能问

Q1: 索引 Pipeline 遇到大文件解析超时怎么办？

三种策略：① 按页拆分——在 DocumentReader 层设置 max-pages: 100，每次只处理 100 页，分多次 MQ 消息完成；② 异步拆分——PDF 解析后按页面拆成多条 MQ 消息，每条独立消费，并行索引；③ 预检与超时控制——上传时校验文件大小和页数，超限即拒绝并由异步服务处理。CorpAssist 优先采用策略 ②，在上传时不阻塞，由后台 MQ 并行消费。

Q2: 权限标签如果标记错了怎么办（机密文档标成公开）？

属于安全事故等级。多层防御：① 自动推断只做初标，标记结果需要人工确认后方可生效；② 每次索引生成审计事件，安全团队定期抽查（建议每周 5% 样本）；③ 敏感关键词黑名单机制——包含「薪酬」「股权」「审计」等自动标记 confidential；④ 安全设计原则——「降级不降权」，自动推断无法确定时默认为机密而非公开；⑤ 万一发生泄露，通过审计日志追溯责任人。

Q3: Java 和 Python 两套索引服务同时跑，怎么保证索引数据一致？

推荐单写多读模式：生产环境中数据写入由一方负责（Java 侧生产索引），Python 实验只从同样的索引库做只读查询（不会写入）。开发环境两栈各自使用独立的测试索引库，通过统一提交到 Git 的分块参数配置确保一致性。在生产切换前用 Python 的评测工具（RAGAS）对 Java 侧索引结果做一次全量验证。

QuestionAnswerAdvisor 链路配置

Spring AI 的 QuestionAnswerAdvisor 是 RAG 编排的核心组件。它在调用 LLM 之前自动将检索结果注入到 Prompt 的上下文中，并约束 LLM 基于检索内容回答。

java
@Service
public class RagQuestionService {

    private final ChatClient chatClient;
    private final VectorStore vectorStore;

    public RagQuestionService(ChatClient.Builder chatClientBuilder,
                               VectorStore vectorStore) {
        // 创建 Advisor 链
        var advisorChain = new DelegatingAdvisorChain(
            List.of(
                // 1. 检索增强 Advisor - 核心
                new QuestionAnswerAdvisor(vectorStore,
                    SearchRequest.builder()
                        .topK(5)
                        .similarityThreshold(0.5)
                        .build()),
                // 2. 引用溯源 Advisor - 确保回答带引用
                new CitationGuardAdvisor(),
                // 3. 安全检查 Advisor - 内容过滤
                new SafeOutputAdvisor()
            )
        );

        this.chatClient = chatClientBuilder
            .defaultAdvisors(advisorChain)
            .build();
        this.vectorStore = vectorStore;
    }

    public Answer answer(String question) {
        return chatClient.prompt()
            .user(question)
            .call()
            .entity(Answer.class);
    }

    public Flux streamAnswer(String question) {
        return chatClient.prompt()
            .user(question)
            .stream()
            .content();
    }
}

VectorStore 检索 -> Prompt 模板 -> ChatClient 调用

除了使用 QuestionAnswerAdvisor 的自动化方式，也可以手动控制检索和 Prompt 组装，适合需要精细控制 Prompt 内容的场景。

java
@Service
public class ManualRagService {

    private final VectorStore vectorStore;
    private final ChatClient chatClient;

    // 手动定义的 RAG Prompt 模板
    private static final String RAG_PROMPT_TEMPLATE = """
        你是一个企业知识库问答助手。请基于以下检索到的文档片段，准确回答用户的问题。
        如果检索到的片段不足以回答问题，请明确说明"当前知识库中没有足够的信息"。

        ## 检索到的文档片段
        {context}

        ## 回答要求
        1. 只基于提供的文档片段回答，不要编造信息
        2. 在回答中引用文档来源（格式：[来源: {source}, 第{page}页]）
        3. 如果多个片段有冲突，指出冲突并说明哪个是更新版本
        4. 用中文回答，保持专业和简洁

        ## 用户问题
        {question}
        """;

    public Answer answer(String question) {
        // 1. 检索
        List docs = vectorStore.search(
            SearchRequest.builder()
                .query(question)
                .topK(5)
                .build()
        );

        // 2. 构建上下文
        String context = buildContext(docs);

        // 3. 填充 Prompt 模板
        PromptTemplate template = new PromptTemplate(RAG_PROMPT_TEMPLATE);
        Message systemMessage = template.createMessage(
            Map.of("context", context, "question", question));

        // 4. 调用 LLM
        return chatClient.prompt()
            .messages(systemMessage)
            .user(question)
            .call()
            .entity(Answer.class);
    }

    private String buildContext(List docs) {
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < docs.size(); i++) {
            Document doc = docs.get(i);
            sb.append("--- 片段 ").append(i + 1).append(" ---\n");
            sb.append("来源: ").append(doc.getMetadata().get("source")).append("\n");
            sb.append("版本: ").append(doc.getMetadata().get("version")).append("\n");
            sb.append("内容: ").append(doc.getText()).append("\n\n");
        }
        return sb.toString();
    }
}

引用溯源：让回答可审计

企业知识库的每个回答必须能追溯到原文——这是合规审计的基本要求。引用溯源通过 Document 的 metadata 实现。

java
@Component
public class CitationGuardAdvisor implements Advisor {

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request) {
        // 在 System Prompt 末尾添加引用约束
        String systemText = request.systemText() + """

            重要约束 - 引用溯源：
            - 你必须在回答中引用每个关键信息的来源文档
            - 引用格式：[来源: {filename}, 版本 {version}, 第{page}页]
            - 如果信息不在提供的文档中，必须说明"当前知识库未覆盖此内容"
            - 不要编造引用来源
            """;

        AdvisedRequest modified = AdvisedRequest.builder()
            .from(request)
            .systemText(systemText)
            .build();

        // 调用下一 Advisor 或 LLM
        AdvisedResponse response = request.next().aroundCall(modified);

        // 将检索片段的引用信息附加到响应中
        List contexts = (List) modified.getAttributes()
            .get(QuestionAnswerAdvisor.RETRIEVED_DOCUMENTS);
        if (contexts != null) {
            List citations = contexts.stream()
                .map(doc -> new Citation(
                    (String) doc.getMetadata().get("source"),
                    (String) doc.getMetadata().get("version"),
                    (String) doc.getMetadata().get("page", "N/A"),
                    doc.getText().substring(0,
                        Math.min(200, doc.getText().length()))
                ))
                .collect(Collectors.toList());
            response.getAttributes().put("citations", citations);
        }

        return response;
    }
}

// 引用数据传输对象
record Citation(String source, String version, String page,
                String snippet) {}

record Answer(String content, List citations) {}

在前端展示时，Citation 列表渲染为可点击的引用链接。用户点击后可跳转到原文的对应页面（通过 source + page 定位）。审计日志中记录每次问答的 Answer.citations 字段，确保每句话可追溯。

SSE 流式输出的 Controller 实现

知识库问答必须支持流式输出（SSE）——用户不需要等待完整回答生成完毕，可以边生成边阅读，体验大幅提升。

java
@RestController
@RequestMapping("/api/v1/kb")
public class KbController {

    private final RagQuestionService ragService;
    private final AuditService auditService;

    public KbController(RagQuestionService ragService,
                         AuditService auditService) {
        this.ragService = ragService;
        this.auditService = auditService;
    }

    // 非流式接口 - 适合简单测试
    @PostMapping("/ask")
    public ResponseEntity ask(
            @RequestBody @Valid QuestionRequest request,
            @AuthenticationPrincipal UserPrincipal user) {
        Answer answer = ragService.answer(request.question());
        auditService.log(user, request.question(), answer);
        return ResponseEntity.ok(answer);
    }

    // 流式接口 - 生产环境使用
    @PostMapping(value = "/ask/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux> streamAsk(
            @RequestBody @Valid QuestionRequest request,
            @AuthenticationPrincipal UserPrincipal user) {

        return ragService.streamAnswer(request.question())
            .map(content -> ServerSentEvent.builder()
                .event("token")
                .data(content)
                .build())
            .concatWithValues(
                ServerSentEvent.builder()
                    .event("done")
                    .data("[DONE]")
                    .build()
            )
            .doOnComplete(() ->
                auditService.logAsync(user, request.question(), null));
    }
}

// 请求/响应 DTO
record QuestionRequest(@NotBlank String question,
                       @Nullable String sessionId) {}

record QuestionResponse(String question, String answer,
                        List citations) {}

流式接口的注意事项：

  • 超时配置：SSE 长连接需要配置合理的超时时间（Spring Boot 默认 30s，建议设为 60s）
  • 跨域处理：前端 SSE 客户端（EventSource API）不支持自定义 Header，需用 GET + query param 或 iframe fallback
  • 背压控制：LLM 生成速度远快于网络传输，Reactor 框架自动处理背压

面试可能问

Q1: QuestionAnswerAdvisor 的原理是什么？怎么保证它只注入相关片段而不是全部检索结果？

QuestionAnswerAdvisor 的工作原理：① 实现 Advisor 接口的 aroundCall 方法；② 在调用 LLM 前拦截请求，提取 query；③ 调用 VectorStore.search(query, topK, similarityThreshold) 检索；④ 将检索结果格式化为 <context>...</context> 追加到 System Prompt；⑤ 将检索片段通过 attributes 传递到响应侧。相关性通过 similarityThreshold（CorpAssist 推荐 0.5）和 topK（5）两个参数控制——低于阈值的片段不会被注入，避免噪声。

Q2: 引用溯源怎么确保 LLM 真的基于检索内容回答而不编造引用？

三层保证：① Prompt 约束——在 System Prompt 中明确要求「只基于提供的文档片段回答」「不编造引用来源」；② CitationGuardAdvisor——在响应后检查 answer.citations 中的 source/page 是否能对应到检索片段的 metadata，不匹配的标记为「未验证」；③ 离线评测——定期用黄金评测集验证引用准确率，低于 95% 触发告警。CorpAssist 实践中三层结合的引用准确率为 97.3%。

Q3: SSE 流式输出的超时怎么配置？流式连接断了怎么处理？

配置：① Spring Boot application.yml 中设置 spring.webflux.base-path 和 spring.codec.max-in-memory-size；② 网关层配置 spring.cloud.gateway.routes[0].filters[0].name=RequestRateLimiter 限制并发流式连接数。断连处理：① LLM 侧无需处理——Reactor Flux 自动在连接断开时取消上游；② 客户端需要实现断线重连——指数退避（1s, 2s, 4s, 8s）最多重试 5 次；③ 记录断连事件到监控——帮助判断是客户端网络问题还是服务端超时。

场景：有同事提议「不用 Advisor 链，直接在 Controller 里手动调 VectorStore 再调 LLM，更简单」。你同意吗？

为什么 0.7:0.3 比 0.5:0.5 更好？

企业知识库中 60% 的查询偏向语义匹配（「年假多少天」），40% 包含精确关键词（「HR-2024-001」）。给向量路稍高权重（0.7）可以覆盖语义为主的多数查询，同时给 BM25 保留足够的权重（0.3）来保证编号类查询的精确命中。实际数据的查询分布决定了这一最优比例。

java
@Service
public class AblationExperimentService {

    private final HybridSearchService hybridSearchService;
    private final List goldenQueries;

    public AblationReport runAblation() {
        List weightCombinations = List.of(
            new FusionWeights(1.0, 0.0),
            new FusionWeights(0.7, 0.3),
            new FusionWeights(0.5, 0.5),
            new FusionWeights(0.3, 0.7),
            new FusionWeights(0.0, 1.0)
        );

        AblationReport report = new AblationReport();

        for (FusionWeights weights : weightCombinations) {
            int hits = 0;
            double ndcg = 0;

            for (TestQuery tq : goldenQueries) {
                List results = hybridSearchService.hybridSearch(
                    tq.query(), 5, weights);

                boolean hasRelevant = results.stream()
                    .anyMatch(d -> tq.relevantDocIds()
                        .contains(d.getId()));
                if (hasRelevant) hits++;

                ndcg += computeNdcg(results, tq.relevantDocIds(), 5);
            }

            report.addResult(weights,
                new MetricResult(
                    (double) hits / goldenQueries.size(),
                    ndcg / goldenQueries.size()));
        }

        return report;
    }
}

java
@Service
public class LatencyBenchmarkService {

    private final RerankService rerankService;
    private final MeterRegistry meterRegistry;

    public void benchmarkRerankLatency() {
        List testQueries = loadTestQueries();
        List candidates = loadCandidates();

        for (int candidateCount : List.of(20, 50, 100)) {
            List subset = candidates.subList(0, candidateCount);
            List latencies = new ArrayList<>();

            for (String query : testQueries) {
                long start = System.nanoTime();
                rerankService.rerank(query, subset);
                long elapsed = TimeUnit.NANOSECONDS.toMillis(
                    System.nanoTime() - start);
                latencies.add(elapsed);
            }

            Collections.sort(latencies);
            long p50 = latencies.get((int) (latencies.size() * 0.50));
            long p95 = latencies.get((int) (latencies.size() * 0.95));
            long p99 = latencies.get((int) (latencies.size() * 0.99));

            // 记录到 Micrometer
            Timer.Sample sample = Timer.start(meterRegistry);
            sample.stop(Timer.builder("rerank.latency")
                .tag("candidates", String.valueOf(candidateCount))
                .register(meterRegistry));

            log.info("Candidates={}: P50={}ms, P95={}ms, P99={}ms",
                candidateCount, p50, p95, p99);
        }
    }
}

Rerank 延迟优化的三个方向：① 减少候选数（50 -> 20），牺牲 2.6% 召回率换取 220ms P99 优化；② 升级 GPU（T4 -> A10G），P99 延迟降低约 40%；③ 对高频查询做缓存——Top-100 热门查询的 Rerank 结果缓存 5 分钟，命中率约 30%。

Spring Boot 完整联调代码

完整的 RAG 检索 Controller，整合双路检索 + Rerank + 流式输出：

java
@RestController
@RequestMapping("/api/v1/rag")
public class RagController {

    private final HybridSearchService hybridSearch;
    private final RerankService rerankService;
    private final ChatClient chatClient;

    @PostMapping("/retrieve")
    public ResponseEntity retrieve(
            @RequestBody RagRequest request) {
        // 1. 双路检索
        List hybridResults = hybridSearch.hybridSearch(
            request.query(), 50);

        // 2. Rerank 精排
        List reranked = rerankService.rerank(
            request.query(), hybridResults);

        // 3. 构建响应（包含检索片段信息）
        List chunks = reranked.stream()
            .map(doc -> new RetrievedChunk(
                doc.getId(),
                doc.getText().substring(0,
                    Math.min(200, doc.getText().length())),
                (String) doc.getMetadata().get("source"),
                (double) doc.getMetadata().getOrDefault("rerank_score", 0.0)
            ))
            .collect(Collectors.toList());

        return ResponseEntity.ok(new RagResponse(chunks));
    }

    @PostMapping(value = "/ask/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux> askStream(@RequestBody RagRequest request) {
        return chatClient.prompt()
            .user(request.query())
            .advisors(new QuestionAnswerAdvisor(hybridSearch, 5))
            .stream()
            .content()
            .map(token -> ServerSentEvent.builder()
                .event("token").data(token).build())
            .concatWithValues(
                ServerSentEvent.builder()
                    .event("done").data("[DONE]").build());
    }
}

record RagRequest(String query, @Nullable String sessionId) {}
record RagResponse(List chunks) {}
record RetrievedChunk(String id, String snippet, String source,
                      double score) {}

/api/v1/rag/retrieve 用于调试和展示检索中间结果，/api/v1/rag/ask/stream 用于最终的问答流式接口。生产环境中可以暴露 retrieve 接口给调试工具而只向最终用户暴露 ask/stream 接口。

Advisor 链中的置信度检查

置信度检查的核心思想：如果检索到的片段与问题的相关性不够高，系统不应该强硬回答，而是应该拒答或降级。

java
@Component
public class ConfidenceCheckAdvisor implements Advisor {

    private static final double CONFIDENCE_THRESHOLD = 0.45;
    private static final double LOW_CONFIDENCE_THRESHOLD = 0.25;

    private final MeterRegistry meterRegistry;
    private final MessageQueueService mqService;

    public ConfidenceCheckAdvisor(MeterRegistry meterRegistry,
                                   MessageQueueService mqService) {
        this.meterRegistry = meterRegistry;
        this.mqService = mqService;
    }

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request) {
        // 获取检索片段的相关性分数
        List contexts = getRetrievedDocuments(request);
        double maxScore = contexts.stream()
            .mapToDouble(d -> (double) d.getMetadata()
                .getOrDefault("rerank_score",
                    d.getMetadata().getOrDefault("distance", 0.0)))
            .max()
            .orElse(0.0);

        String classification;
        if (maxScore >= CONFIDENCE_THRESHOLD) {
            // 高置信度 -> 正常回答
            classification = "high_confidence";
        } else if (maxScore >= LOW_CONFIDENCE_THRESHOLD) {
            // 中置信度 -> 回答但附带 disclaimer
            classification = "medium_confidence";
            request = appendSystemInstruction(request,
                "注意：以下回答基于相关性不够高的文档。请明确告诉用户" +
                "「此回答基于相关性有限的文档，建议进一步核实。」");
        } else {
            // 低置信度 -> 拒答
            classification = "rejected";
            return buildRejectedResponse(request,
                "当前知识库中没有找到足够相关的信息来回答这个问题。请尝试换一种问法，或联系 HR 部门。");
        }

        // 记录指标
        meterRegistry.counter("kb.confidence.classification",
            "level", classification).increment();
        meterRegistry.gauge("kb.retrieval.max_score",
            Collections.singletonList(Tag.of("query", hashQuery(request.userText()))),
            maxScore);

        // 中/低置信度发送到人工审核队列
        if (!"high_confidence".equals(classification)) {
            mqService.send("corpassist-review-queue",
                new ReviewEvent(request.userText(), maxScore, classification));
        }

        return request.next().aroundCall(request);
    }
}

置信度阈值需要根据业务场景调整。CorpAssist 在三个客户项目的实验表明：0.45 作为高置信度阈值（覆盖约 78% 的查询）可以达到 95.3% 的回答准确率。0.25 至 0.45 的中置信度区间（约 15% 的查询）准确率降至 82%，需要加 disclaimer。低于 0.25（约 7% 的查询）准确率仅 51%，必须拒答。

强制引用（Citation Guard Advisor）

强制引用比「建议引用」更严格——它要求 LLM 生成的每个具体主张都必须关联到检索片段。如果 LLM 无法关联，必须在回答中明确标注「未找到引用来源」。

java
@Component
public class CitationGuardAdvisor implements Advisor {

    private static final String CITATION_PROMPT = """
        重要 - 引用溯源要求:
        1. 对于回答中的每一个关键信息(数字、日期、条款、政策名称),
           必须在句末标注引用来源: [引用来源: {source_file}, 第{page_number}页]
        2. 如果某个信息无法在提供的文档片段中找到来源,
           必须标注 [未验证: 知识库中未找到直接来源]
        3. 在回答末尾列出本次使用的所有引用来源列表
        4. 不要编造引用来源
        """;

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request) {
        // 注入引用约束到 System Prompt
        AdvisedRequest modified = AdvisedRequest.builder()
            .from(request)
            .systemText(request.systemText() + "\n" + CITATION_PROMPT)
            .build();

        AdvisedResponse response = request.next().aroundCall(modified);

        // 提取检索片段信息附加到响应
        List docs = getRetrievedDocuments(modified);
        if (docs != null) {
            List citations = docs.stream()
                .map(d -> new Citation(
                    str(d.getMetadata(), "source"),
                    str(d.getMetadata(), "page"),
                    str(d.getMetadata(), "version")))
                .collect(Collectors.toList());
            response.getAttributes().put("_citations", citations);
        }

        return response;
    }

    private String str(Map meta, String key) {
        Object v = meta.get(key);
        return v != null ? v.toString() : "N/A";
    }
}

record Citation(String source, String page, String version) {}

// 后处理校验 - 验证引用是否真实
@Component
public class CitationValidator {

    public ValidationResult validate(AdvisedResponse response) {
        @SuppressWarnings("unchecked")
        List citations =
            (List) response.getAttributes().get("_citations");

        if (citations == null || citations.isEmpty()) {
            return ValidationResult.warning("No citations found in response");
        }

        // 检查每个 citation 的 source 是否在检索片段中存在
        Set validSources = citations.stream()
            .map(Citation::source)
            .collect(Collectors.toSet());

        // 记录引用覆盖率指标
        return ValidationResult.ok(citations.size(), validSources.size());
    }
}

幻觉监控（Micrometer 指标）

幻觉不能完全消除，但可以通过持续监控量化其趋势。CorpAssist 使用 Micrometer（Spring Boot Actuator 内置）构建幻觉监控看板。

java
@Component
public class HallucinationMetrics {

    private final MeterRegistry registry;

    // 计数器
    private final Counter totalQuestions;
    private final Counter rejectedQuestions;
    private final Counter lowConfidenceQuestions;
    private final Counter citationMissingResponses;

    // 分布
    private final DistributionSummary retrievalScoreSummary;

    public HallucinationMetrics(MeterRegistry registry) {
        this.registry = registry;
        this.totalQuestions = Counter.builder("kb.questions.total")
            .description("Total questions received").register(registry);
        this.rejectedQuestions = Counter.builder("kb.questions.rejected")
            .description("Questions rejected due to low confidence")
            .register(registry);
        this.lowConfidenceQuestions = Counter.builder("kb.questions.low_confidence")
            .description("Questions answered with disclaimer")
            .register(registry);
        this.citationMissingResponses = Counter.builder("kb.responses.citation_missing")
            .description("Responses without valid citations")
            .register(registry);
        this.retrievalScoreSummary = DistributionSummary.builder("kb.retrieval.max_score")
            .description("Distribution of max retrieval scores")
            .publishPercentiles(0.5, 0.75, 0.9, 0.95, 0.99)
            .register(registry);
    }

    public void recordQuestion(String classification) {
        totalQuestions.increment();
        switch (classification) {
            case "rejected" -> rejectedQuestions.increment();
            case "low_confidence" -> lowConfidenceQuestions.increment();
        }
    }

    public void recordCitationCheck(boolean hasCitation) {
        if (!hasCitation) citationMissingResponses.increment();
    }

    public void recordRetrievalScore(double score) {
        retrievalScoreSummary.record(score);
    }

    // 派生指标 - 在 Grafana 中配置告警
    // 1. 拒答率 = kb.questions.rejected / kb.questions.total
    //    告警条件: > 15% 持续 5 分钟
    // 2. 引用缺失率 = kb.responses.citation_missing / kb.questions.total
    //    告警条件: > 5% 持续 5 分钟
    // 3. 检索分数 P50 < 0.3 持续 10 分钟 -> Embedding 模型或索引问题
}

面试可能问

Q1: 置信度阈值怎么确定的？0.45 是怎么来的？

通过数据驱动的实验确定：从生产日志中收集 500 个查询-回答对，人工标注每个回答是否正确。然后绘制「阈值 vs 准确率」曲线——发现阈值在 0.45 时准确率达到 95.3%，再提高阈值只会增加拒答率而不显著提升准确率。不同业务的阈值不同，需要在自己的数据上重新实验。

Q2: 人工复核队列如果积压了处理不过来怎么办？

三级应对：① 自动兜底——审核积压超过 30 分钟的系统自动给用户标准回复（「您的问题已提交审核，结果将通过消息通知您」）；② 智能推荐——根据历史决策为审核员推荐操作（相似查询的决策命中率约 82%）；③ 动态阈值——审核积压时自动提高拒答阈值（从 0.45 提到 0.55），减少进入审核队列的查询量。

            report.contextPrecision());

        // 规则：如果 RAGAS 分数低于阈值，自动运行回归测试
        if (report.faithfulness() < 0.80 || report.contextPrecision() < 0.85) {
            RegressionTestResult result = regressionTestRunner.runAll();
            log.warn("RAGAS report triggered regression tests: {}",
                result.summary());

            // 如果回归测试也失败，创建告警
            if (result.hasFailures()) {
                alertService.sendAlert("RAG quality regression detected. " +
                    "RAGAS faithfulness: " + report.faithfulness() +
                    ", Regression failures: " + result.failureCount());
            }
        }

        return ResponseEntity.ok("Report received");
    }
}

CorpAssist 实践中，RAGAS 触发回归测试的阈值为：faithfulness < 0.80 或 contextPrecision < 0.85。回归测试失败时会自动创建 Jira 工单并通知值班工程师。这种 Java-Python 交叉验证机制确保了 RAG 系统持续达到质量标准。

索引蓝绿部署（别名切换）

当文档更新或分块参数变化时，需要重建索引。蓝绿部署使用别名实现零停机索引切换。

java
@Service
public class BlueGreenIndexService {
    private static final String ALIAS = "corpassist-policies-active";
    private static final String BLUE = "corpassist-policies-blue";
    private static final String GREEN = "corpassist-policies-green";

    private final MilvusServiceClient milvusClient;
    private final ElasticsearchClient esClient;

    // 切换到下一个索引
    public void switchIndex() {
        String current = identifyActiveIndex();
        String next = current.equals(BLUE) ? GREEN : BLUE;

        log.info("Switching active index: {} -> {}", current, next);

        // 1. 将别名从当前索引移除
        removeAlias(current);

        // 2. 将别名赋给新索引
        assignAlias(next);

        // 3. 验证新索引可用
        validateIndex(next);

        log.info("Active index switched to: {}", next);
    }

    // 初始创建索引
    public void initializeIndexes() {
        // 创建 blue 和 green 两个索引集合
        createCollection(BLUE);
        createCollection(GREEN);

        // 初始将别名指向 green（空索引可正常服务）
        assignAlias(GREEN);
    }

    // 重建索引(在新索引上)
    public void rebuildIndex(Collection documents) {
        String next = identifyInactiveIndex();
        log.info("Rebuilding index: {}", next);

        // 清空旧数据
        clearCollection(next);

        // 批量写入
        vectorStore.add(documents);

        // 验证数据完整性
        int docCount = countDocuments(next);
        log.info("Index {} rebuilt with {} documents", next, docCount);

        // 自动切换
        switchIndex();

        // 清理旧索引中的过期数据
        clearCollection(identifyInactiveIndex());
    }
}

蓝绿部署的关键优势：重建索引耗时（大型文档库可能需要数小时）完全不影响在线问答服务。旧索引在重建期间继续服务，新索引就绪后通过别名切换秒级生效。如果新索引有问题，别名可以立即切换回旧索引实现回滚。

面试可能问

Q1: Testcontainers 集成测试和单元测试的比例怎么分配？

测试金字塔：单元测试 70% + 集成测试 20% + E2E 测试 10%。单元测试测试单个组件（Splitter/MetadataEnricher），集成测试测试检索链路完整正确性（用真实 Milvus/ES），E2E 测试从 HTTP 请求到回答的端到端链路。Testcontainers 适合集成测试层。

Q2: 蓝绿部署切换时如果新索引有问题，怎么快速回滚？

别名切换的特性：切换别名到新索引后，旧索引的数据并没有删除。如果发现问题，只需要将别名再切回旧索引即可（秒级）。旧索引在切换后保留 72 小时作为回滚窗口，72 小时后自动清理。清理前发通知确认是否还有回滚需求。
