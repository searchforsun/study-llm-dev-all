Elasticsearch BM25 + Milvus 向量双路

双路检索需要两个 VectorStore 实例协同工作。Milvus 负责向量 ANN 检索，ES 负责 BM25 关键词检索，两者通过 RRF 融合。

java
@Configuration
public class DualRetrievalConfig {

    @Value("${spring.ai.vectorstore.milvus.host:localhost}")
    private String milvusHost;

    @Value("${spring.ai.vectorstore.milvus.port:19530}")
    private int milvusPort;

    @Bean
    public MilvusServiceClient milvusClient() {
        ConnectParam connectParam = ConnectParam.newBuilder()
            .withHost(milvusHost)
            .withPort(milvusPort)
            .build();
        return new MilvusServiceClient(connectParam);
    }

    @Bean
    @Primary
    public MilvusVectorStore milvusVectorStore(
            MilvusServiceClient milvusClient,
            EmbeddingModel embeddingModel) {
        return new MilvusVectorStore(milvusClient, embeddingModel,
            MilvusVectorStoreConfig.builder()
                .collectionName("corpassist_policies")
                .dimension(1024)
                .metricType(MetricType.COSINE)
                .indexType(IndexType.IVF_FLAT)
                .indexParam("nlist", 1024)
                .build());
    }

    @Bean
    public ElasticsearchVectorStore elasticsearchVectorStore(
            ElasticsearchClient esClient,
            EmbeddingModel embeddingModel) {
        return new ElasticsearchVectorStore(esClient, embeddingModel,
            ElasticsearchVectorStoreConfig.builder()
                .indexName("corpassist_policies_text")
                .build());
    }

    @Bean
    public HybridSearchService hybridSearchService(
            @Qualifier("milvusVectorStore") MilvusVectorStore milvusStore,
            @Qualifier("elasticsearchVectorStore") ElasticsearchVectorStore esStore,
            EmbeddingModel embeddingModel) {
        return new HybridSearchService(milvusStore, esStore, embeddingModel);
    }
}

双路检索的关键配置要点：两个 VectorStore 使用不同的索引名/collection名、但共用一个 Embedding 模型（保证向量维度一致）。写入时双写（同时写入 Milvus 和 ES），检索时各读各的。

权重调优与 Ablation 实验

RRF 融合权重（vectorWeight : keywordWeight）直接影响检索效果。通过消融实验找到最佳权重组合。

向量权重关键词权重Recall@5NDCG@5特征
1.00.084.1%0.81纯向量，语义匹配但编号检索弱
0.01.076.3%0.72纯 BM25，精确匹配强但语义差
0.50.589.8%0.85均等权重，中庸
0.70.392.3%0.88CorpAssist 推荐
0.80.291.5%0.87偏向量
0.60.490.2%0.86偏 BM25
为什么 0.7:0.3 比 0.5:0.5 更好？

企业知识库中 60% 的查询偏向语义匹配（「年假多少天」），40% 包含精确关键词（「HR-2024-001」）。给向量路稍高权重（0.7）可以覆盖语义为主的多数查询，同时给 BM25 保留足够的权重（0.3）来保证编号类查询的精确命中。实际数据的查询分布决定了这一最优比例。

java
@Service
public class AblationExperimentService {

    private final HybridSearchService hybridSearchService;
    private final List goldenQueries;

    public AblationReport runAblation() {
        List weightCombinations = List.of(
            new FusionWeights(1.0, 0.0),
            new FusionWeights(0.7, 0.3),
            new FusionWeights(0.5, 0.5),
            new FusionWeights(0.3, 0.7),
            new FusionWeights(0.0, 1.0)
        );

        AblationReport report = new AblationReport();

        for (FusionWeights weights : weightCombinations) {
            int hits = 0;
            double ndcg = 0;

            for (TestQuery tq : goldenQueries) {
                List results = hybridSearchService.hybridSearch(
                    tq.query(), 5, weights);

                boolean hasRelevant = results.stream()
                    .anyMatch(d -> tq.relevantDocIds()
                        .contains(d.getId()));
                if (hasRelevant) hits++;

                ndcg += computeNdcg(results, tq.relevantDocIds(), 5);
            }

            report.addResult(weights,
                new MetricResult(
                    (double) hits / goldenQueries.size(),
                    ndcg / goldenQueries.size()));
        }

        return report;
    }
}

黄金评测集的构建方法：从生产日志中采集 200 个真实用户查询，为每个查询标注其对应的正确答案文档 ID。标注工作可以众包给业务部门同事完成（每人标 20-30 条，5 分钟可完成）。

Rerank 延迟基准（P50/P95/P99）

Reranker 的延迟是企业关注的关键指标。以下是 CorpAssist 在不同配置下的实测数据（GPU: T4, 模型: BGE-Reranker-v2-m3, batch=8）：

候选数P50 延迟P95 延迟P99 延迟批次大小
2095ms140ms180ms4
50180ms320ms400ms8
100350ms590ms720ms16
200680ms1150ms1400ms32

CorpAssist 生产推荐 50 条候选（P99 400ms），在 Recall 提升和延迟之间取得平衡。如果对延迟敏感可以降为 20 条（P99 180ms），但 Recall@5 会从 92.3% 降到 89.7%。

java
@Service
public class LatencyBenchmarkService {

    private final RerankService rerankService;
    private final MeterRegistry meterRegistry;

    public void benchmarkRerankLatency() {
        List testQueries = loadTestQueries();
        List candidates = loadCandidates();

        for (int candidateCount : List.of(20, 50, 100)) {
            List subset = candidates.subList(0, candidateCount);
            List latencies = new ArrayList<>();

            for (String query : testQueries) {
                long start = System.nanoTime();
                rerankService.rerank(query, subset);
                long elapsed = TimeUnit.NANOSECONDS.toMillis(
                    System.nanoTime() - start);
                latencies.add(elapsed);
            }

            Collections.sort(latencies);
            long p50 = latencies.get((int) (latencies.size() * 0.50));
            long p95 = latencies.get((int) (latencies.size() * 0.95));
            long p99 = latencies.get((int) (latencies.size() * 0.99));

            // 记录到 Micrometer
            Timer.Sample sample = Timer.start(meterRegistry);
            sample.stop(Timer.builder("rerank.latency")
                .tag("candidates", String.valueOf(candidateCount))
                .register(meterRegistry));

            log.info("Candidates={}: P50={}ms, P95={}ms, P99={}ms",
                candidateCount, p50, p95, p99);
        }
    }
}

Rerank 延迟优化的三个方向：① 减少候选数（50 -> 20），牺牲 2.6% 召回率换取 220ms P99 优化；② 升级 GPU（T4 -> A10G），P99 延迟降低约 40%；③ 对高频查询做缓存——Top-100 热门查询的 Rerank 结果缓存 5 分钟，命中率约 30%。

Spring Boot 完整联调代码

完整的 RAG 检索 Controller，整合双路检索 + Rerank + 流式输出：

java
@RestController
@RequestMapping("/api/v1/rag")
public class RagController {

    private final HybridSearchService hybridSearch;
    private final RerankService rerankService;
    private final ChatClient chatClient;

    @PostMapping("/retrieve")
    public ResponseEntity retrieve(
            @RequestBody RagRequest request) {
        // 1. 双路检索
        List hybridResults = hybridSearch.hybridSearch(
            request.query(), 50);

        // 2. Rerank 精排
        List reranked = rerankService.rerank(
            request.query(), hybridResults);

        // 3. 构建响应（包含检索片段信息）
        List chunks = reranked.stream()
            .map(doc -> new RetrievedChunk(
                doc.getId(),
                doc.getText().substring(0,
                    Math.min(200, doc.getText().length())),
                (String) doc.getMetadata().get("source"),
                (double) doc.getMetadata().getOrDefault("rerank_score", 0.0)
            ))
            .collect(Collectors.toList());

        return ResponseEntity.ok(new RagResponse(chunks));
    }

    @PostMapping(value = "/ask/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux> askStream(@RequestBody RagRequest request) {
        return chatClient.prompt()
            .user(request.query())
            .advisors(new QuestionAnswerAdvisor(hybridSearch, 5))
            .stream()
            .content()
            .map(token -> ServerSentEvent.builder()
                .event("token").data(token).build())
            .concatWithValues(
                ServerSentEvent.builder()
                    .event("done").data("[DONE]").build());
    }
}

record RagRequest(String query, @Nullable String sessionId) {}
record RagResponse(List chunks) {}
record RetrievedChunk(String id, String snippet, String source,
                      double score) {}

/api/v1/rag/retrieve 用于调试和展示检索中间结果，/api/v1/rag/ask/stream 用于最终的问答流式接口。生产环境中可以暴露 retrieve 接口给调试工具而只向最终用户暴露 ask/stream 接口。

本章结论

  • 双路检索需配置两个 VectorStore Bean（Milvus + ES），共用 Embedding 模型

  • Spring Boot 联调 Controller 提供 /retrieve（调试）和 /ask/stream（生产）两个接口

面试可能问

Q1: 权重消融实验的具体步骤是怎样的？

步骤：① 准备黄金评测集（200 个查询，每个标注相关文档 ID）；② 选择 5-6 组权重组合（从纯向量到纯 BM25）；③ 对每组权重计算 Recall@K 和 NDCG@K；④ 分析结果选择最优组合。CorpAssist 结果：0.7:0.3 最优。企业知识库的查询分布决定——60% 语义型 + 40% 编号型，偏向量可覆盖多数查询。

Q2: Rerank 延迟高怎么办？什么时候该放弃 Rerank 降级？

优化顺序：① 减少候选数（50->20），P99 从 400ms 到 180ms；② 升级 GPU（T4->A10G），P99 再降 40%；③ 对热门查询做结果缓存（5 分钟 TTL，约 30% 命中率）。降级条件：当 Rerank 连续 3 次超时（>1s）时自动降级到 RRF 排序，降级期间记录指标；当恢复后自动切回。CorpAssist 政策：P99 > 1s 触发降级。

判断练习

场景：上线后发现双路检索比之前单路向量还慢（+200ms），但 Recall 只提升 3%。该怎么办？

评估决策：如果 Recall 已经达到业务标准（如 >=90%），那就没必要为了 3% 提升增加 200ms 延迟。建议：① 将双路作为可选模式（通过配置开关控制），默认单路向量；② 对 Recall 敏感的场景（制度编号查询）走双路；③ 持续监控 Recall——如果后续掉到 90% 以下再切回双路。
