为什么学这章

问答编排是 RAG 系统的「大脑」——它把检索结果、Prompt 模板、LLM 调用串联成完整的问答链路。Spring AI 的 ChatClient 和 QuestionAnswerAdvisor 提供了标准化的编排能力，但需要正确配置才能保证回答质量、引用溯源和流式输出。

学完后能做什么

能使用 Spring AI ChatClient 和 QuestionAnswerAdvisor 构建完整的问答服务；能实现引用溯源让回答可审计；能实现 SSE 流式输出的 Controller；能配置 Advisor 链实现检索增强和内容过滤。

本章要点速记

  • QuestionAnswerAdvisor 是检索增强的核心 Advisor，自动注入检索片段到 Prompt
  • ChatClient 提供流式（Stream）和非流式（Call）两种调用模式
  • 引用溯源通过 Document 的 metadata 中的 chunk_id/source/page 字段实现
  • Advisor 链支持按顺序组装多个 Advisor（检索 -> 内容审查 -> 格式控制）

QuestionAnswerAdvisor 链路配置

Spring AI 的 QuestionAnswerAdvisor 是 RAG 编排的核心组件。它在调用 LLM 之前自动将检索结果注入到 Prompt 的上下文中，并约束 LLM 基于检索内容回答。

java
@Service
public class RagQuestionService {

    private final ChatClient chatClient;
    private final VectorStore vectorStore;

    public RagQuestionService(ChatClient.Builder chatClientBuilder,
                               VectorStore vectorStore) {
        // 创建 Advisor 链
        var advisorChain = new DelegatingAdvisorChain(
            List.of(
                // 1. 检索增强 Advisor - 核心
                new QuestionAnswerAdvisor(vectorStore,
                    SearchRequest.builder()
                        .topK(5)
                        .similarityThreshold(0.5)
                        .build()),
                // 2. 引用溯源 Advisor - 确保回答带引用
                new CitationGuardAdvisor(),
                // 3. 安全检查 Advisor - 内容过滤
                new SafeOutputAdvisor()
            )
        );

        this.chatClient = chatClientBuilder
            .defaultAdvisors(advisorChain)
            .build();
        this.vectorStore = vectorStore;
    }

    public Answer answer(String question) {
        return chatClient.prompt()
            .user(question)
            .call()
            .entity(Answer.class);
    }

    public Flux streamAnswer(String question) {
        return chatClient.prompt()
            .user(question)
            .stream()
            .content();
    }
}

QuestionAnswerAdvisor 的工作流程：

  1. 拦截用户请求，提取 query
  2. 调用 VectorStore.search() 检索相关片段
  3. 将检索片段格式化为 <context>...</context> 注入到 System Prompt
  4. LLM 根据上下文 + 用户问题生成回答
  5. 在响应中携带检索片段的 metadata（source / page / chunk_id）

VectorStore 检索 -> Prompt 模板 -> ChatClient 调用

除了使用 QuestionAnswerAdvisor 的自动化方式，也可以手动控制检索和 Prompt 组装，适合需要精细控制 Prompt 内容的场景。

java
@Service
public class ManualRagService {

    private final VectorStore vectorStore;
    private final ChatClient chatClient;

    // 手动定义的 RAG Prompt 模板
    private static final String RAG_PROMPT_TEMPLATE = """
        你是一个企业知识库问答助手。请基于以下检索到的文档片段，准确回答用户的问题。
        如果检索到的片段不足以回答问题，请明确说明"当前知识库中没有足够的信息"。

        ## 检索到的文档片段
        {context}

        ## 回答要求
        1. 只基于提供的文档片段回答，不要编造信息
        2. 在回答中引用文档来源（格式：[来源: {source}, 第{page}页]）
        3. 如果多个片段有冲突，指出冲突并说明哪个是更新版本
        4. 用中文回答，保持专业和简洁

        ## 用户问题
        {question}
        """;

    public Answer answer(String question) {
        // 1. 检索
        List docs = vectorStore.search(
            SearchRequest.builder()
                .query(question)
                .topK(5)
                .build()
        );

        // 2. 构建上下文
        String context = buildContext(docs);

        // 3. 填充 Prompt 模板
        PromptTemplate template = new PromptTemplate(RAG_PROMPT_TEMPLATE);
        Message systemMessage = template.createMessage(
            Map.of("context", context, "question", question));

        // 4. 调用 LLM
        return chatClient.prompt()
            .messages(systemMessage)
            .user(question)
            .call()
            .entity(Answer.class);
    }

    private String buildContext(List docs) {
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < docs.size(); i++) {
            Document doc = docs.get(i);
            sb.append("--- 片段 ").append(i + 1).append(" ---\n");
            sb.append("来源: ").append(doc.getMetadata().get("source")).append("\n");
            sb.append("版本: ").append(doc.getMetadata().get("version")).append("\n");
            sb.append("内容: ").append(doc.getText()).append("\n\n");
        }
        return sb.toString();
    }
}

手动编排的适用场景：当 QuestionAnswerAdvisor 默认的 Prompt 格式无法满足业务需求时（例如需要自定义引用格式、需要多片段冲突检测、需要特定输出格式约束）。CorpAssist 实践中有 70% 的场景使用 QuestionAnswerAdvisor 自动化编排，30% 的场景需要手动控制。

引用溯源：让回答可审计

企业知识库的每个回答必须能追溯到原文——这是合规审计的基本要求。引用溯源通过 Document 的 metadata 实现。

java
@Component
public class CitationGuardAdvisor implements Advisor {

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request) {
        // 在 System Prompt 末尾添加引用约束
        String systemText = request.systemText() + """

            重要约束 - 引用溯源：
            - 你必须在回答中引用每个关键信息的来源文档
            - 引用格式：[来源: {filename}, 版本 {version}, 第{page}页]
            - 如果信息不在提供的文档中，必须说明"当前知识库未覆盖此内容"
            - 不要编造引用来源
            """;

        AdvisedRequest modified = AdvisedRequest.builder()
            .from(request)
            .systemText(systemText)
            .build();

        // 调用下一 Advisor 或 LLM
        AdvisedResponse response = request.next().aroundCall(modified);

        // 将检索片段的引用信息附加到响应中
        List contexts = (List) modified.getAttributes()
            .get(QuestionAnswerAdvisor.RETRIEVED_DOCUMENTS);
        if (contexts != null) {
            List citations = contexts.stream()
                .map(doc -> new Citation(
                    (String) doc.getMetadata().get("source"),
                    (String) doc.getMetadata().get("version"),
                    (String) doc.getMetadata().get("page", "N/A"),
                    doc.getText().substring(0,
                        Math.min(200, doc.getText().length()))
                ))
                .collect(Collectors.toList());
            response.getAttributes().put("citations", citations);
        }

        return response;
    }
}

// 引用数据传输对象
record Citation(String source, String version, String page,
                String snippet) {}

record Answer(String content, List citations) {}

在前端展示时，Citation 列表渲染为可点击的引用链接。用户点击后可跳转到原文的对应页面（通过 source + page 定位）。审计日志中记录每次问答的 Answer.citations 字段，确保每句话可追溯。

CorpAssist 实践中的引用溯源准确率：QA 验证表明，配合引用约束 Prompt 和 CitationGuardAdvisor，引用准确率可达 97.3%（即 97.3% 的回答中所有引用都可以在检索片段中找到对应源文本）。

SSE 流式输出的 Controller 实现

知识库问答必须支持流式输出（SSE）——用户不需要等待完整回答生成完毕，可以边生成边阅读，体验大幅提升。

java
@RestController
@RequestMapping("/api/v1/kb")
public class KbController {

    private final RagQuestionService ragService;
    private final AuditService auditService;

    public KbController(RagQuestionService ragService,
                         AuditService auditService) {
        this.ragService = ragService;
        this.auditService = auditService;
    }

    // 非流式接口 - 适合简单测试
    @PostMapping("/ask")
    public ResponseEntity ask(
            @RequestBody @Valid QuestionRequest request,
            @AuthenticationPrincipal UserPrincipal user) {
        Answer answer = ragService.answer(request.question());
        auditService.log(user, request.question(), answer);
        return ResponseEntity.ok(answer);
    }

    // 流式接口 - 生产环境使用
    @PostMapping(value = "/ask/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux> streamAsk(
            @RequestBody @Valid QuestionRequest request,
            @AuthenticationPrincipal UserPrincipal user) {

        return ragService.streamAnswer(request.question())
            .map(content -> ServerSentEvent.builder()
                .event("token")
                .data(content)
                .build())
            .concatWithValues(
                ServerSentEvent.builder()
                    .event("done")
                    .data("[DONE]")
                    .build()
            )
            .doOnComplete(() ->
                auditService.logAsync(user, request.question(), null));
    }
}

// 请求/响应 DTO
record QuestionRequest(@NotBlank String question,
                       @Nullable String sessionId) {}

record QuestionResponse(String question, String answer,
                        List citations) {}

流式接口的注意事项：

  • 超时配置：SSE 长连接需要配置合理的超时时间（Spring Boot 默认 30s，建议设为 60s）
  • 跨域处理：前端 SSE 客户端（EventSource API）不支持自定义 Header，需用 GET + query param 或 iframe fallback
  • 背压控制：LLM 生成速度远快于网络传输，Reactor 框架自动处理背压
  • 审计延迟：流式输出无法在响应中直接携带 citations，审计在 doOnComplete 中异步执行

流式接口的前端调用示例：fetch('/api/v1/kb/ask/stream', {method:'POST', body:JSON.stringify({question:'产假几天'})}) 后通过 ReadableStream 读取 SSE 事件。

面试可能问

Q1: QuestionAnswerAdvisor 的原理是什么？怎么保证它只注入相关片段而不是全部检索结果？

QuestionAnswerAdvisor 的工作原理：① 实现 Advisor 接口的 aroundCall 方法；② 在调用 LLM 前拦截请求，提取 query；③ 调用 VectorStore.search(query, topK, similarityThreshold) 检索；④ 将检索结果格式化为 <context>...</context> 追加到 System Prompt；⑤ 将检索片段通过 attributes 传递到响应侧。相关性通过 similarityThreshold（CorpAssist 推荐 0.5）和 topK（5）两个参数控制——低于阈值的片段不会被注入，避免噪声。

Q2: 引用溯源怎么确保 LLM 真的基于检索内容回答而不编造引用？

三层保证：① Prompt 约束——在 System Prompt 中明确要求「只基于提供的文档片段回答」「不编造引用来源」；② CitationGuardAdvisor——在响应后检查 answer.citations 中的 source/page 是否能对应到检索片段的 metadata，不匹配的标记为「未验证」；③ 离线评测——定期用黄金评测集验证引用准确率，低于 95% 触发告警。CorpAssist 实践中三层结合的引用准确率为 97.3%。

Q3: SSE 流式输出的超时怎么配置？流式连接断了怎么处理？

配置：① Spring Boot application.yml 中设置 spring.webflux.base-path 和 spring.codec.max-in-memory-size；② 网关层配置 spring.cloud.gateway.routes[0].filters[0].name=RequestRateLimiter 限制并发流式连接数。断连处理：① LLM 侧无需处理——Reactor Flux 自动在连接断开时取消上游；② 客户端需要实现断线重连——指数退避（1s, 2s, 4s, 8s）最多重试 5 次；③ 记录断连事件到监控——帮助判断是客户端网络问题还是服务端超时。

判断练习

场景：有同事提议「不用 Advisor 链，直接在 Controller 里手动调 VectorStore 再调 LLM，更简单」。你同意吗？

不推荐。Advisor 链的三个优势：① 职责分离——每个 Advisor 只处理一个横切关注点（检索/引用/安全），可以独立测试和替换；② 可组合性——可以灵活增删 Advisor（例如新加一个 PII 脱敏 Advisor），不需要改 Controller 代码；③ 统一拦截——ChatClient 的 Advisor 链在所有调用中生效，不会遗漏。手动在 Controller 里编排，上述能力都需要自己实现。

场景：线上发现某些回答没有引用来源（citation 为空），可能是什么原因？怎么修？

原因分析：① 检索结果为空（VectorStore 未命中任何文档）——检查索引是否正常、FilterExpression 是否过严；② QuestionAnswerAdvisor 的 similarityThreshold 过高（如 0.9）——所有片段都被过滤；③ CitationGuardAdvisor 代码逻辑有 Bug——attributes 中未正确传递检索片段。排查：先看检索日志确认是否有检索结果；再看 Advisor 链配置确认阈值是否合理；最后看 CitationGuardAdvisor 的 attributes 读取逻辑。修复：降阈值 / 修 Bug。
