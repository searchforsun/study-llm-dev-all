行级权限的检索过滤 FilterExpression

企业知识库中，不同角色的用户只能看到不同范围的文档。权限过滤必须在检索层完成——不是在检索后丢弃无权结果，而是在检索时就通过 FilterExpression 限定。

java
@Service
public class SecureKnowledgeService {

    private final VectorStore vectorStore;

    public SearchResponse secureSearch(String query, UserPrincipal user) {
        // 1. 构建权限过滤条件
        FilterExpression.Builder filter = FilterExpression.builder();

        // 基础过滤: 安全等级
        if (user.hasRole("admin") || user.hasRole("vp")) {
            // 管理员和 VP 可以看到所有等级
            filter.in("security_level",
                List.of("public", "internal", "confidential"));
        } else if (user.hasRole("manager")) {
            // 经理可以看到 internal 及以上
            filter.in("security_level", List.of("public", "internal"));
        } else {
            // 普通员工只能看 public
            filter.eq("security_level", "public");
        }

        // 部门过滤: 员工只看自己部门的文档
        if (!user.hasRole("admin")) {
            filter.or(
                eq("department", user.getDepartment()),
                eq("department", "ALL")
            );
        }

        // 2. 执行带权限过滤的检索
        SearchRequest request = SearchRequest.builder()
            .query(query)
            .topK(5)
            .similarityThreshold(0.5)
            .filterExpression(filter.build())
            .build();

        return vectorStore.search(request);
    }
}

    String department,

// JWT 鉴权过滤器
@Component
public class JwtAuthFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain)
            throws ServletException, IOException {
        String token = extractToken(request);
        if (token != null) {
            UserPrincipal user = parseToken(token);
            SecurityContextHolder.getContext()
                .setAuthentication(new JwtAuthentication(user));
        }
        chain.doFilter(request, response);
    }
}

检索层过滤的安全原则：人在哪一层鉴权，就在哪一层过滤。如果在 Controller 层鉴权但不在检索层过滤，检索结果仍然包含了用户无权访问的文档——信息可能通过缓存、日志、错误信息等侧信道泄露。

审计日志：Spring AOP + MongoDB

知识库的每一次问答都必须可审计——谁、什么时间、问了什么、系统怎么回答的、引用了哪些文档。Spring AOP 提供了声明式的审计方案。

java
@Aspect
@Component
public class AuditLogAspect {

    private final AuditLogRepository auditLogRepository;

    @Around("@annotation(auditable)")
    public Object logAudit(ProceedingJoinPoint pjp,
                            Auditable auditable) throws Throwable {
        long startTime = System.currentTimeMillis();

        // 获取请求参数
        Object[] args = pjp.getArgs();
        String query = extractQuery(args);
        UserPrincipal user = getCurrentUser();

        // 执行原方法
        Object result = pjp.proceed();

        // 记录审计日志
        long duration = System.currentTimeMillis() - startTime;
        AuditLog log = AuditLog.builder()
            .timestamp(Instant.now())
            .userId(user.userId())
            .department(user.department())
            .query(query)
            .duration(duration)
            .build();

        if (result instanceof Answer answer) {
            log.setResponse(answer.content());
            log.setCitations(answer.citations());
            log.setConfidenceScore(computeScore(answer));
        }

        auditLogRepository.save(log);
        return result;
    }
}

审计日志的关键设计决策：

  • 存储选型 MongoDB——审计日志是写多读少、Schema 灵活的数据，MongoDB 比 MySQL 更适合
  • TTL 策略——审计日志保存 180 天（合规要求），到期自动删除（MongoDB TTL 索引）
  • 复合索引——userId + timestamp 用于单个用户的操作追溯
  • 数据脱敏——查询中包含个人身份信息时自动脱敏（如员工姓名 -> 工号）
  • 保留策略——180 天后归档到冷存储（S3/OSS），保留 7 年

多租户索引前缀隔离

SaaS 化部署时，多个租户共享同一个 Milvus/ES 集群。通过索引前缀实现数据隔离，比每个租户一个独立集群更经济。

java
@Component
public class TenantAwareVectorStore {

    private final MilvusServiceClient milvusClient;

    // 获取租户专属的 Collection 名称
    private String tenantCollection(String tenantId) {
        return "kb_" + tenantId + "_policies";
    }

    // 写入时按租户写入独立 Collection
    public void addDocuments(String tenantId,
                              List documents) {
        String collection = tenantCollection(tenantId);

        // 确保租户 Collection 存在
        ensureCollectionExists(collection);

        // 写入数据
        documents.forEach(doc ->
            doc.getMetadata().put("tenant_id", tenantId));
        vectorStore.add(documents);
    }

    // 检索时按租户检索
    public List search(String tenantId, String query) {
        String collection = tenantCollection(tenantId);

        SearchRequest request = SearchRequest.builder()
            .query(query)
            .topK(5)
            .filterExpression(
                FilterExpression.builder()
                    .eq("tenant_id", tenantId)
                    .build()
            )
            .build();

        return vectorStore.search(request);
    }

    @Bean
    @RequestScope
    public TenantContext tenantContext(HttpServletRequest request) {
        String tenantId = request.getHeader("X-Tenant-Id");
        return new TenantContext(tenantId);
    }
}

record TenantContext(String tenantId) {
    public boolean isValid() {
        return tenantId != null && !tenantId.isBlank();
    }
}

层次实现对优点缺点
索引前缀Milvus collection 前缀 + metadata 过滤资源利用率高一个租户的查询可能影响集群性能
集群级每个租户独立 Milvus 集群完全隔离成本高，资源浪费

面试可能问

Q1: 多租户场景下，如果两个租户的文档都包含「产假天数」，怎么保证各自只搜到各自的文档？

两层保证：① 物理隔离——租户 A 的文档写入 collection "kb_tenant_a_policies"，租户 B 写入 "kb_tenant_b_policies"，写入时 metadata 中标记 tenant_id；② 检索过滤——检索时从请求 Header X-Tenant-Id 获取租户 ID，通过 FilterExpression.eq("tenant_id", tenantId) 过滤。即使误写了也不影响。

Q2: 审计日志量很大（日均 10 万条），存储成本高怎么办？

分层存储策略：① MongoDB 保留最近 30 天的热数据（快速查询）；② 30-180 天的温数据存储在 MongoDB 但缩采样（去掉 response 完整内容）；③ 180 天以上的冷数据压缩后存到 OSS/S3 对象存储。查询时先查 MongoDB，查不到再从 OSS 拉取。总存储成本降低约 70%。
