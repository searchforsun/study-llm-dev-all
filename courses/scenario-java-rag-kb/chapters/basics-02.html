三层架构：Gateway/编排层/数据层

CorpAssist S1 知识库采用经典的三层架构，与微服务架构中的 BFF/Service/DAO 一一映射。

层级CorpAssist 组件核心职责不应做
接入层corpassist-bff + llm-gatewayJWT 鉴权、SSE 透传、模型路由、API Key 托管、用量落库文档分块策略、向量重建、复杂 Agent 规划
编排层dialog-orchestratorRAG 检索链调用、Prompt 模板版本管理、多轮对话状态维护、降级处理持有厂商 API Key、直接暴露给公网
数据层rag-index + 对象存储 + 审计仓文档入库解析、Embedding 索引、引用片段管理、审计日志同步调用 LLM 生成最终用户可见的回答

一次 RAG 问答的完整调用序：

  1. BFF 校验 JWT 获取用户租户/角色，附带 session_id 转发至 orchestrator
  2. Orchestrator 调 rag-service：混合检索（Milvus ANN + ES BM25 + RRF）→ 得到 top-k 片段与 chunk_id
  3. 拼装 messages（system prompt + 引用块 + user query），经 gateway POST /v1/chat/completions
  4. 流式 SSE chunk 经 BFF 转发到客户端；末包中 usage 写入审计库，按部门归因 Token 消耗

# CorpAssist 分层职责配置 (application-corpassist.yml)

      gateway: llm-gateway

  retrieval:
    top_k: 6
    rerank: true
  ingest:
    queue: corpassist-doc-ingest
    dead_letter: corpassist-doc-dlq

Spring AI VectorStore 选型矩阵

Spring AI 1.0.0-M6 支持 6 种 VectorStore 实现。选型需要综合性能指标、运维成本和现有生态。

实现索引类型P50 延迟 (1K dim)运维成本适用场景
MilvusIVF_FLAT / HNSW12ms中（需 K8s 或 Docker）大规模生产 > 1M 向量
ElasticsearchHNSW + BM2518ms低（复用已有 ES 集群）混合检索 + 已有 ES 生态
PgVectorIVFFlat25ms低（PostgreSQL 插件）小团队快速原型
QdrantHNSW15ms中（Docker 可跑）独立向量库需求
RedisFLAT（暴力）8ms低缓存级 / 小规模
Cosmos DBDiskANN20ms高（Azure 生态）Azure 原生栈

CorpAssist 推荐 Milvus + Elasticsearch 双路：Milvus 负责向量 ANN 检索，ES 负责 BM25 关键词召回 + 元数据过滤。双路结果通过 RRF 融合后由 Reranker 精排。这比单一向量库召回率提升 8.2 个百分点。

java
@Configuration
public class VectorStoreConfig {

    @Bean
    @Primary
    public VectorStore milvusVectorStore(MilvusServiceClient milvusClient,
                                          EmbeddingModel embeddingModel) {
        return new MilvusVectorStore(milvusClient, embeddingModel,
            MilvusVectorStoreConfig.builder()
                .collectionName("corpassist_policies")
                .dimension(1024)
                .metricType(MetricType.COSINE)
                .indexType(IndexType.IVF_FLAT)
                .build());
    }

    @Bean
    public VectorStore elasticsearchVectorStore(
            ElasticsearchClient esClient,
            EmbeddingModel embeddingModel) {
        return new ElasticsearchVectorStore(esClient, embeddingModel,
            ElasticsearchVectorStoreConfig.builder()
                .indexName("corpassist_policies_text")
                .build());
    }

    @Bean
    public HybridSearchService hybridSearchService(
            @Qualifier("milvusVectorStore") VectorStore milvus,
            @Qualifier("elasticsearchVectorStore") VectorStore es,
            EmbeddingModel embeddingModel) {
        return new HybridSearchService(milvus, es, embeddingModel);
    }
}
先想 10 秒：如果你的团队已经有 ES 集群，选型时优先考虑 ES Vector Store 还是额外部署 Milvus？

如果数据量 < 500 万向量且 ES 集群有资源——优先用 ES Vector Store，省去运维 Milvus 的复杂度。当向量规模增大或延迟敏感时再考虑 Milvus。CorpAssist 初期用 ES 单路，v2 引入 Milvus 做双路。

CorpAssist 知识库的数据流设计

数据流分为写入流（文档索引 Pipeline）和读取流（问答检索 Pipeline），两者通过索引存储解耦。

写入流（文档索引 Pipeline）

  1. OA 系统通过 Webhook 通知文档变更 → 发送 MQ 消息到 corpassist-doc-ingest 队列
  2. 索引服务消费消息，调用 TikaDocumentReader 解析文档
  3. TokenTextSplitter 分块（chunk-size: 512, overlap: 64）
  4. MetadataEnricher 提取元数据（作者/部门/版本/权限标签）
  5. 调用 Embedding API 向量化后并行写入 Milvus + ES
  6. 确认双写成功后 ACK 消息；失败进入 DLQ 重试

读取流（问答检索 Pipeline）

  1. 用户输入问题 → BFF 鉴权 → orchestrator 收到 session_id
  2. Orchestrator 并行调用 Milvus（向量）+ ES（BM25）双路检索
  3. 双路结果通过 RRF 融合入候选池（各取 Top-50 融合为 Top-20）
  4. BGE-Reranker 对 Top-20 精排，取 Top-5
  5. Prompt 模板将引用片段拼入 System Message + User Message
  6. ChatClient 调用 LLM 生成回答 + 引用溯源

面试答题框架：架构题的五步回答法

面试被问到「请设计一个知识库 RAG 系统」时，用五步框架回答会显得思路清晰、经验丰富。

步骤话术示例考察点
① 明确需求边界「首先确认场景——是内部制度问答还是客服文档检索？预期 QPS 和文档量级？」需求分析能力，不做过度设计
② 分层架构「我们采用三层——接入层处理鉴权和流式透传，编排层组装 RAG 链路，数据层管理索引」系统设计能力，职责清晰
③ 选型对比「向量库选 Milvus vs ES？看数据规模和现有生态。Milvus 适合大规模纯向量，ES 适合已有搜索集群」技术广度，不唯技术
④ 关键指标「我们的目标是 Recall@5 ≥ 90%、P99 延迟 ≤ 2s、单次查询 Token 成本 ≤ ¥0.2」量化思维，指标驱动
⑤ 权衡与局限「RAG 不擅长数学推理和多跳推理，在这些场景需要结合 Agent 或 Chain-of-Thought」工程务实，知道边界

完整回答示例：「我们采用三层架构——接入层用 Spring Cloud Gateway + LLM Gateway 做鉴权和流式转发，编排层用 Spring AI ChatClient 组装检索与生成，数据层用 Milvus + ES 双路索引 + BGE-Reranker。Recall@5 92.3% ，P99 1.8s，单次查询 Token 成本约 ¥0.18。主要局限是多跳推理场景效果差，正在结合 Agent 方案。」

CorpAssist 面试场景模拟

面试官追问：「你负责的知识库最近检索不准，排查思路是什么？」回答应结构化：① 查检索日志看实际召回了哪些片段及其 RRF 分数；② 检查分块参数是否合理（512/64 还是默认值）；③ 看 Rerank 权重配置是否需要调优；④ 检查文档版本——是否混入废止版本文档；⑤ 将该查询加入黄金评测集做回归。

本章结论

  • 三层架构（接入/编排/数据）与微服务架构一一映射，每层职责清晰不可越界
  • Spring AI 支持 6 种 VectorStore，生产推荐 Milvus + ES 双路，Recall@5 92.3%
  • 数据流分写入（MQ 触发索引 Pipeline）和读取（双路检索 + RRF + Rerank）两个方向
  • 面试架构题采用五步回答法：需求 → 分层 → 选型 → 指标 → 权衡
  • 具体选型关注 P99 延迟 / Recall@K / 运维成本三维度，避免唯向量论

面试可能问

Q1: 三层架构中编排层和接入层的职责边界在哪里？

接入层负责非智能的透传职责：JWT 鉴权、SSE 流式转发、模型路由、API Key 托管、用量落库。编排层负责智能的组装职责：RAG 检索链调用、Prompt 模板版本管理、多轮对话状态维护、工具调用编排与降级。关键约束：编排层不持有厂商 API Key，不暴露公网；接入层不做 RAG 核心逻辑。

Q2: 为什么选 Milvus + ES 双路而不是单一向量库？

纯向量库的缺点：① 关键词精确匹配弱，特殊编号（HR-2024-001）无法精确命中；② 元数据过滤依赖向量库实现，不如 ES 成熟。双路的优势：向量召回语义相似文档 + BM25 精确匹配编号/专有名词 + RRF 融合排序。实际业务中 Recall@5 从 84.1% 提升到 92.3%，代价是额外 15-25ms 延迟。

Q3: 数据写入流被脏文档堵住了怎么办？

生产级方案：① MQ 消费端设置 max-retry: 3 和死信队列 DLQ；② 脏文档进入 DLQ 后通过管理后台人工审核和重试；③ DocumentReader 设置 skip-on-error: true 跳过异常文档；④ 监控仪表板跟踪索引队列积压和失败率；⑤ 定期巡检 DLQ 并实现半自动重放机制。

判断练习

场景：项目初期想只用 ES 做关键词搜索不加向量检索，你怎么评估？

可以接受 v1 快速上线，但需预知局限：① 用户用同义词/近义词提问时无法匹配（如「年假」vs「带薪休假」）；② 长文本语义理解差，无法根据内容相关性排序；③ 不支持模糊语义查询。建议 v1 用 ES + 同义词扩展词典做快速验证；v2 引入向量检索做语义增强。

场景：架构评审中有人建议编排层直接调用 LLM 不通过 Gateway，你怎么反驳？

Gateway 承担三大核心价值：① API Key 统一托管——避免密钥散落在各微服务中，轮换时只需改一处；② 用量审计——每个租户/部门的 Token 消耗可追溯归因；③ 模型路由——可按部门/场景路由到不同模型（HR 用 qwen-plus、技术用 gpt-4o）。去掉 Gateway 意味着这些职责分散到各服务，无法统一治理和审计。
