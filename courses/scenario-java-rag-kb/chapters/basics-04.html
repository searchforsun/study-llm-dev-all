为什么学这章

单一检索方式（纯向量或纯关键词）在企业知识库场景中都有缺陷。混合检索 + Rerank 是提高召回率和回答质量的关键技术。CorpAssist 实验表明双路检索 + Rerank 比纯向量检索 Recall@5 提升 8.2 个百分点。掌握混合检索才能在面试中展示 RAG 系统的深度理解。

能编写 Java 混合检索代码（Milvus + ES + RRF 融合）；能部署和调用 BGE-Reranker HTTP 服务；能解释 Recall/MRR/NDCG 等检索指标；能识别检索失败的典型模式并给出排查策略。

Spring AI VectorStore + Elasticsearch 混合检索

混合检索结合了向量语义检索和 BM25 关键词匹配的优势，是生产级 RAG 系统的标配。

java
@Service
public class HybridRetrievalService {

    private final MilvusVectorStore milvusStore;
    private final ElasticsearchVectorStore esStore;
    private final EmbeddingModel embeddingModel;

    public HybridRetrievalService(
            @Qualifier("milvusVectorStore") MilvusVectorStore milvusStore,
            @Qualifier("elasticsearchVectorStore") ElasticsearchVectorStore esStore,
            EmbeddingModel embeddingModel) {
        this.milvusStore = milvusStore;
        this.esStore = esStore;
        this.embeddingModel = embeddingModel;
    }

    public List hybridSearch(String query, int topK) {
        // 1. 向量检索 (Milvus ANN)
        SearchRequest vectorRequest = SearchRequest.builder()
            .query(query)
            .topK(topK * 2)
            .similarityThreshold(0.5)
            .build();
        List vectorResults = milvusStore.search(vectorRequest);

        // 2. 关键词检索 (ES BM25)
        SearchRequest keywordRequest = SearchRequest.builder()
            .query(query)
            .topK(topK * 2)
            .build();
        List keywordResults = esStore.search(keywordRequest);

        // 3. RRF 融合排序
        return ReciprocalRankFusion.merge(
            vectorResults, keywordResults, topK,
            FusionWeights.builder()
                .vectorWeight(0.7)
                .keywordWeight(0.3)
                .build()
        );
    }
}

RRF（Reciprocal Rank Fusion）的核心公式：score(d) = w_v / (k + rank_v(d)) + w_k / (k + rank_k(d))。其中 k 为融合常量（通常 60），rank_v 和 rank_k 是文档在向量路和关键词路的排名位置。这种方法不需要训练，实现简单且效果稳定。

java
public class ReciprocalRankFusion {
    private static final int K = 60;

    public static List merge(
            List vecResults,
            List keyResults,
            int topK,
            FusionWeights weights) {

        Map scoreMap = new LinkedHashMap<>();
        Map docMap = new LinkedHashMap<>();

        // 计算向量路的 RRF 分数
        for (int i = 0; i < vecResults.size(); i++) {
            Document doc = vecResults.get(i);
            String id = doc.getId();
            double score = weights.vectorWeight() / (K + i + 1);
            scoreMap.merge(id, score, Double::sum);
            docMap.putIfAbsent(id, doc);
        }

        // 计算关键词路的 RRF 分数
        for (int i = 0; i < keyResults.size(); i++) {
            Document doc = keyResults.get(i);
            String id = doc.getId();
            double score = weights.keywordWeight() / (K + i + 1);
            scoreMap.merge(id, score, Double::sum);
            docMap.putIfAbsent(id, doc);
        }

        // 按总分降序排序，取 topK
        return scoreMap.entrySet().stream()
            .sorted(Map.Entry.comparingByValue().reversed())
            .limit(topK)
            .map(e -> docMap.get(e.getKey()))
            .collect(Collectors.toList());
    }
}

@Value
@Builder
public class FusionWeights {
    double vectorWeight;   // 默认 0.7
    double keywordWeight;  // 默认 0.3
}

BGE-Reranker 的 HTTP 调用封装

Reranker 使用交叉编码器架构——query 和每个 document 拼接后输入模型做深度交互打分，比向量相似度（双编码器）精确得多。代价是推理速度较慢。

java
@Service
public class RerankService {

    private final RestClient restClient;

    public RerankService(
            @Value("${corpassist.rerank.url}") String url,
            @Value("${corpassist.rerank.timeout:5000}") int timeout) {
        this.restClient = RestClient.builder()
            .baseUrl(url)
            .defaultHeader("Content-Type", "application/json")
            .requestFactory(new JdkClientHttpRequestFactory(
                HttpClient.newBuilder()
                    .connectTimeout(Duration.ofSeconds(5))
                    .build()))
            .build();
    }

    public List rerank(String query, List candidates) {
        if (candidates.isEmpty()) return candidates;

        RerankRequest request = new RerankRequest(
            query,
            candidates.stream().map(Document::getText).toList()
        );

        try {
            RerankResponse response = restClient.post()
                .uri("/rerank")
                .body(request)
                .retrieve()
                .body(RerankResponse.class);

            // 按相关性分数降序排列，取 Top-5
            List indices = response.indices();
            List scores = response.scores();

            List reranked = new ArrayList<>();
            for (int i = 0; i < Math.min(5, indices.size()); i++) {
                Document doc = candidates.get(indices.get(i));
                doc.getMetadata().put("rerank_score", scores.get(i));
                reranked.add(doc);
            }
            return reranked;
        } catch (Exception e) {
            log.warn("Rerank failed, falling back to original order", e);
            return candidates.subList(0, Math.min(5, candidates.size()));
        }
    }
}

@JsonIgnoreProperties(ignoreUnknown = true)
record RerankRequest(String query, List documents) {}

@JsonIgnoreProperties(ignoreUnknown = true)
record RerankResponse(List indices, List scores) {}

检索失败的典型模式与对策

即使配置完善，检索系统仍可能以特定模式「失败」。理解这些模式是生产排障的基础。

失败模式现象根因对策
语义偏移召回内容相关但无法回答问题分块包含多个主题，检索命中非目标段落缩小 chunk-size 或启用 SemanticSplitter
匹配缺失精确编号查不到（HR-2024-001）BM25 对编号不敏感或向量化丢失精度增加 ES BM25 权重，使用 RRF 融合
旧版污染召回了废止版本元数据未过滤或索引混入旧版FilterExpression 加 version 条件，蓝绿切换
高频噪声常见问题召回极多重复片段高 TF 词主导 BM25 或向量坍缩停用词过滤 / BM25 参数 k1,b 调优
无结果返回检索结果为空文档未索引或 Embedding 模型维度不匹配检查索引状态和模型维度一致性

生产排障的「五步法」：① 查看检索日志——检查实际召回的片段 ID 和 RRF 分数；② 用同样的查询重现检索——确认不是偶发问题；③ 检查元数据——FilterExpression 是否误过滤了有效文档；④ 检查监控指标——Recall/NDCG 是否持续恶化；⑤ 切 10% 流量到新参数做 A/B 验证修复效果。

CorpAssist 排障实践

某银行客户反馈「公司年假政策」查询总是返回过时信息。排查三步：① 检索日志显示召回片段中混有 2019 版和 2024 版；② 检查 FilterExpression 发现 metadata.effective_date 是字符串格式（"2024-01-01"），FilterExpression 做字符串比较而非日期比较；③ 2024 版生效日期字符串排在 2019 版之后。修复：将日期改为 ISO8601 格式 + FilterExpression 加 date_gte: 2024-01-01 条件。

本章结论

  • 混合检索（Milvus ANN + ES BM25 + RRF 融合）是 CorpAssist 标准方案，Recall@5 达 92.3%
  • BGE-Reranker-v2-m3 作为第二阶段精排，50 条候选 P99 延迟 200-400ms，Rerank 失败时降级到 RRF 排序
  • 核心指标：Recall@K >= 90%，NDCG@K >= 0.88，P50 <= 350ms，P99 <= 1.8s
  • 五种检索失败模式各有对应排查策略，排障五步法：日志 -> 重现 -> 元数据 -> 指标 -> A/B
  • RRF 融合权重推荐 0.7（向量）: 0.3（关键词），k 常量 60

面试可能问

Q1: 向量检索和 BM25 各自擅长什么场景？为什么需要两者结合？

向量检索擅长语义相似度匹配——用户说「年假」而文档写「带薪年休假」，向量距离仍然小。BM25 擅长精确关键词匹配——「HR-2024-001」这类编号只有 BM25 能精确命中。企业知识库中既有自由表述（「产假多少天」）又有精确代码（「查阅 HR-2024-001 号制度」），两者缺失任何一个都会造成大量漏召回。双路结合后 Recall@5 从 84.1% 提升到 92.3%。

Q2: Reranker 的延迟开销怎么接受？怎么评估值不值？

Reranker 延迟增量约 200-400ms（P99），但能带来 Recall@5 +8.2%。CorpAssist 把 Reranker 作为第二阶段精排：第一阶段宽召回 50 条用轻量级的 RRF 融合（~20ms），第二阶段只对 50 条做交叉编码器重排（~250ms）。整体链路 P99 约 1.8s，对知识库问答场景是可接受的。如果延迟压力大，可以：① 减少 Rerank 候选数（50 -> 20）；② 用更轻量的 Reranker（如 BGE-Reranker-v2-minicpm）；③ 对高频查询加缓存跳过 Reranker。

Q3: 怎么评估当前的检索召回率是否达标？怎么建立评测体系？

建立黄金评测集的过程：① 从生产日志中采集 200 个真实用户查询；② 为每个查询标注其对应的正确答案文档 ID（人工标注，可众包）；③ 定义一个「相关」标准（精确匹配、语义匹配、部分匹配三级）；④ 对每个查询计算 Recall@K 和 NDCG@K。CorpAssist 标准：Recall@5 >= 90%，NDCG@5 >= 0.88。低于标准触发优化流程——调整分块参数、双路权重或 Reranker 模型。

判断练习

场景：MVP 阶段团队想跳过 Reranker，只用双路 RRF 融合 + Prompt。你觉得可行吗？有什么风险？

MVP 阶段可以接受——简化架构加快上线。但需预知限制：① RRF 融合的排序精度不如交叉编码器，Top-5 中可能混入 1-2 个不相关片段，影响回答质量；② 不相关片段可能误导 LLM 生成错误回答。建议 v1 不做 Reranker，v2 引入。CorpAssist 实践中 Reranker 在 v1.3 引入，Recall@5 从 84.1% -> 92.3%。

场景：线上发现某高频查询（占 10% 流量）的召回率只有 60%，其他查询正常。怎么排查？

第一步：查看该查询的检索日志——实际召回了哪些文档及其 RRF 分数；第二步：分析这些文档的分块——是否被切碎或跨块；第三步：检查元数据——FilterExpression 是否误过滤；第四步：单独对查询做 Embedding 和相似度对比——确认是否是 Embedding 模型在此查询上表现差；第五步：将该查询加入黄金评测集做回归测试。常见原因：该查询包含专有名词或缩写，向量化后语义坍缩，需要加 BM25 权重或做查询改写。
