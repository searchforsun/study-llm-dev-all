面试回答框架（五步法）：

  1. 需求边界：企业内部制度问答场景，5000 用户，日均 400 次问答。文档以 PDF/Word 为主约 5000 份
  2. 分层架构：接入层做鉴权和透传，编排层做 RAG 链路，数据层做存储。每层职责分离
  3. 选型对比：VectorStore 选 Milvus + ES 双路（Recall@5 92.3%），Embedding 用 text-embedding-3-small
  4. 关键指标：Recall@5 >= 90%，P99 <= 2s，单次成本约 ¥0.18
  5. 权衡局限：多跳推理用 RAG 效果差，正在结合 Agent 方案

Q2: 分块怎么选

面试官问：你的知识库分块参数怎么确定的？为什么这样选？

回答结构：

  1. 初始选择：chunk-size 512 + chunk-overlap 64，基于 Spring AI TokenTextSplitter 的 tiktoken 计算
  2. 实验验证：在 CorpAssist 的 846 页 HR 制度文档上做消融实验，对比 5 组参数（256/512/768/1024）。512 在 Recall@5(92.3%) 和回答完整性(4.5/5)上均为最优
  3. 特殊处理：表格和代码块保持完整性——表格问答准确率 +34%，技术文档问答 +28%
  4. 元数据标记：通过 MetadataEnricher 自动标记 content_type/security_level/policy_code，检索时用 FilterExpression 精确过滤
  5. 配置化：分块参数通过配置中心管理，不硬编码，支持运行时调优

加分项：提及「分块参数需要根据文档类型调整——操作手册类可以稍大（768），制度类 512 最优」。这展示了你的实际落地经验。

Q3: 检索不准

面试官问：用户反馈知识库回答不准，你的排查思路是什么？

回答使用「五步排查法」：

步骤具体操作工具
① 查检索日志查看该查询实际召回了哪些文档、RRF 分数、Rerank 分数检索日志 / Elasticsearch
② 查分块检查召回片段是否被切碎，边界处是否语义完整chunk 查看器
③ 查元数据FilterExpression 是否误过滤了有效文档FilterExpression 调试
④ 查模型单独对查询做 Embedding 和相似度对比Python 脚本验证
⑤ 做回归将查询加入黄金评测集，运行回归测试验证修复Testcontainers + JUnit

典型根因举例：「HR-2024-001」查询不准 -> 排查发现 BM25 权重 0.3 太低，编号类查询向量化后精度丢失。调整权重到 0.5:0.5 后解决。

Q4: RAG vs 微调

面试官问：知识库为什么用 RAG 而不是微调？什么场景该用微调？

回答框架：

维度RAG微调
知识更新改文档后重新索引，分钟级生效需要标注->训练->评估->部署，以周计
幻觉可控强制引用原文片段，引用准确率 97.3%模型可能「记住」了但无法追溯来源
审计追溯每个回答附带 citations 列表，逐条可追溯无法追溯
适用场景知识频繁变化/内容多的制度问答格式固定/术语专有的场景（如合同模板生成）
成本每次问答消耗 Token（约 ¥0.18）一次性训练成本（数千元）+ 推理成本

结论：CorpAssist 的制度问答优先 RAG。微调仅在两种场景考虑：① 需要固定输出格式（如合同条款必须按固定模板输出）；② 需要理解深度行业术语（如法律文本解析）。实际落地中，80% 的场景用 RAG，20% 用 RAG + 微调增强。

Q5: 评估指标

面试官问：你怎么评估 RAG 系统的好坏？线上用什么指标监控？

评估分三层：

1. 检索层（自动化）

  • Recall@5：目标 >= 90%，黄金评测集周运行
  • NDCG@5：目标 >= 0.88，考虑排序位置的质量
  • MRR：目标 >= 0.85，第一个相关结果的排名

2. 生成层（自动化 + RAGAS）

  • Faithfulness（忠实度）：回答是否基于检索内容，RAGAS 评分目标 >= 0.85
  • 引用覆盖率：有引用支持的句子比例 >= 95%
  • 拒答准确率：该拒答时正确拒答 >= 98%

3. 体验层（人工）

  • CSAT：用户满意度评分，目标 >= 4.5/5.0
  • 人工抽检：每周随机抽检 100 条回答，标注正确/部分正确/错误
  • 坏例库：积累的 Bad Case 驱动持续优化

面试加分：提到「我们建立了从评估 -> 发现 Bad Case -> 归因 -> 修复 -> 回归的闭环流程，每周迭代一次。坏例库从最初的 200 条降到 30 条。」

本章结论

  • Q1 架构：三层 + 双路 + 蓝绿，用五步法回答
  • Q2 分块：512/64 最优，消融实验验证，表格/代码块特殊处理
  • Q3 检索不准：五步排查法，五种失败模式
  • Q4 RAG vs 微调：80% RAG + 20% RAG+微调增强
  • Q5 评估：三层指标体系（检索/生成/体验）+ 闭环迭代流程

Q6: 你们做 RAG 过程中踩过最大的坑是什么？

最大的坑是脏文档。上线第三周，一个 500MB 的扫描件 PDF 导致 DocumentReader OOM，整个索引 Pipeline 阻塞了 2 小时。之前完全没想到企业文档质量能这么差。之后的改进：① 前置校验（文件大小/页数限制）；② 扫描件 OCR 预处理；③ Pipeline 设置 skip-on-error + DLQ 机制；④ 文档上传时做格式检查和用户提示。这件事教会我们：企业级系统的稳定性取决于最薄弱的那个环节。
