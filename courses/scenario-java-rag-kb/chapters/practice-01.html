Spring AI DocumentReader 链路

企业文档接入是 RAG 系统的起点。Spring AI 通过 DocumentReader 接口以策略模式支持多种文档格式解析。完整的索引 Pipeline 分为四个阶段：Read -> Split -> Transform -> Write。

java
@Configuration
@ConditionalOnProperty(name = "corpassist.ingest.enabled", havingValue = "true")
public class IngestionPipelineConfig {

    @Value("${corpassist.ingest.source-dir:./docs}")
    private String sourceDir;

    @Bean
    public DocumentReader documentReader() {
        // 根据配置选择文档读取器
        return new TikaDocumentReader(new FileSystemResource(sourceDir));
    }

    @Bean
    public TextSplitter splitter(IngestionProperties props) {
        return TokenTextSplitter.builder()
            .withChunkSize(props.getChunking().getChunkSize())
            .withChunkOverlap(props.getChunking().getChunkOverlap())
            .withSeparators(props.getChunking().getSeparators()
                .toArray(new String[0]))
            .build();
    }

    @Bean
    public DocumentTransformer metadataTransformer(AppVersion appVersion) {
        return documents -> documents.stream()
            .peek(doc -> doc.getMetadata().put("ingested_at",
                Instant.now().toString()))
            .peek(doc -> doc.getMetadata().put("ingest_version",
                appVersion.getVersion()))
            .peek(doc -> doc.getMetadata().put("doc_id",
                UUID.randomUUID().toString()))
            .collect(Collectors.toList());
    }

    @Bean
    public IngestionPipeline ingestionPipeline(
            DocumentReader reader,
            TextSplitter splitter,
            List transformers,
            VectorStore vectorStore) {

        return IngestionPipeline.builder()
            .documentReader(reader)
            .textSplitter(splitter)
            .transformers(transformers)
            .vectorStore(vectorStore)
            .build();
    }
}

在 CorpAssist 实践中，索引 Pipeline 通过 MQ 异步触发——OA 系统发 Webhook 通知文档变更，MQ 消息驱动索引服务消费。这样在线问答链路和索引链路完全解耦，索引压力不影响问答延迟。

元数据提取与权限标签

企业文档的元数据不仅用于检索过滤，更用于行级安全控制。以下是 CorpAssist 的元数据模型和自动提取实现。

java
@Component
public class PermissionTagEnricher implements DocumentTransformer {

    @Override
    public List transform(List documents) {
        return documents.stream()
            .map(this::enrich)
            .collect(Collectors.toList());
    }

    private Document enrich(Document doc) {
        String text = doc.getText();
        Map meta = doc.getMetadata();

        // 1. 基于内容自动推断安全等级
        if (text.contains("保密") || text.contains("机密") || text.contains("薪酬")) {
            meta.put("security_level", "confidential");
            meta.put("allowed_roles", List.of("admin", "vp", "hr_manager"));
        } else if (text.contains("内部") || text.contains("仅限")) {
            meta.put("security_level", "internal");
            meta.put("allowed_roles", List.of("admin", "manager", "employee"));
        } else {
            meta.put("security_level", "public");
            meta.put("allowed_roles", List.of("*"));
        }

        // 2. 从文档路径推断所属部门
        String path = (String) meta.get("source_path");
        if (path != null) {
            for (String dept : List.of("HR", "IT", "FIN", "OPS", "LEGAL")) {
                if (path.contains("/" + dept + "/")) {
                    meta.put("department", dept);
                    break;
                }
            }
        }

        // 3. 提取文档标题（首行或 Markdown H1）
        String firstLine = text.lines().findFirst().orElse("");
        if (!firstLine.isEmpty() && firstLine.length() < 100) {
            meta.put("title", firstLine.replace("# ", "").trim());
        }

        // 4. 计算内容哈希用于去重
        meta.put("content_hash",
            DigestUtils.sha256Hex(text.substring(0, Math.min(1000, text.length()))));

        return doc;
    }
}

// 检索时注入用户权限条件
@Component
public class SecureSearchService {

    public SearchRequest buildSecureRequest(String query, UserContext user) {
        FilterExpression.Builder filter = FilterExpression.builder();

        // 角色过滤
        filter.in("allowed_roles", user.getRoles());

        // 部门过滤
        if (user.getDepartment() != null) {
            filter.eq("department", user.getDepartment());
        }

        return SearchRequest.builder()
            .query(query)
            .topK(10)
            .filterExpression(filter.build())
            .build();
    }
}

权限标签的核心价值：检索时通过 FilterExpression 自动注入用户角色和部门条件，确保「用户只能看到其有权访问的文档」。如果在检索层不做过滤而只靠后端丢弃结果，属于安全设计缺陷——信息可能通过检索日志、缓存等侧信道泄露。

生产实践中，自动推断只做初标，标记结果需人工确认后方可生效。安全团队定期抽查标记准确率。

脏文档处理策略

企业文档质量参差不齐——扫描件 OCR 错误、表格格式混乱、图片型 PDF 无法解析。脏文档是最容易被低估的线上问题源头。

脏文档类型现象处理策略
扫描件（图片PDF）Tika 解析结果为空或乱码先用 OCR 服务（PaddleOCR / Tesseract）处理后再进 Pipeline
加密/密码保护读取时抛出 IOException记录到 DLQ，通知管理员解密后重新上传
超大文档(>200页/50MB)解析耗时 >30s，OOM 风险按页分批处理，设置 max-pages: 100，max-file-size: 50MB
损坏文件POI/Tika 解析异常转 PDF 后重试，连续失败 3 次进入人工审核
非中文编码乱码文本自动检测编码（ICU4J），转换失败记录日志并跳过
java
@Service
public class DirtyDocumentHandler {
    private static final int MAX_PAGES = 100;
    private static final long MAX_FILE_SIZE_BYTES = 50 * 1024 * 1024L;
    private static final int MAX_RETRIES = 3;

    private final OcrService ocrService;
    private final DeadLetterSender dlqSender;
    private final RetryTemplate retryTemplate;

    public IngestionResult handleWithRetry(Resource resource) {
        return retryTemplate.execute(ctx -> {
            try {
                return handle(resource);
            } catch (Exception e) {
                log.warn("Attempt {} failed for {}",
                    ctx.getRetryCount(), resource.getFilename(), e);
                if (ctx.getRetryCount() >= MAX_RETRIES - 1) {
                    dlqSender.send(new DirtyDocumentEvent(
                        resource.getFilename(), e.toString()));
                    return IngestionResult.failed(resource.getFilename(),
                        "Max retries exceeded: " + e.getMessage());
                }
                throw e;
            }
        });
    }

    public IngestionResult handle(Resource resource) throws Exception {
        // 前置校验
        validateFileSize(resource);

        // OCR 预处理（图片类 PDF）
        if (isImageBasedPdf(resource)) {
            log.info("OCR preprocessing for: {}", resource.getFilename());
            resource = ocrService.preprocess(resource);
        }

        // 按页分批（超大文档）
        if (isLargeDocument(resource)) {
            return batchProcessByPage(resource);
        }

        return processNormally(resource);
    }

    private void validateFileSize(Resource resource) throws IOException {
        if (resource.contentLength() > MAX_FILE_SIZE_BYTES) {
            throw new IllegalArgumentException(
                "File exceeds max size: " + resource.getFilename());
        }
    }

    private boolean isImageBasedPdf(Resource resource) throws IOException {
        // 通过 PDF 元数据判断是否扫描件
        try (var reader = new com.itextpdf.kernel.pdf.PdfReader(
                resource.getInputStream())) {
            var doc = new com.itextpdf.kernel.pdf.PdfDocument(reader);
            // 如果所有页面都不含文本内容，判断为扫描件
            for (int i = 1; i <= Math.min(doc.getNumberOfPages(), 5); i++) {
                var page = doc.getPage(i);
                var text = com.itextpdf.kernel.pdf.canvas.parser.PdfTextExtractor
                    .getTextFromPage(page);
                if (!text.isBlank()) return false;
            }
            return true;
        }
    }
}

脏文档处理的基本原则：永远不让脏文档阻塞 Pipeline 主链路。CorpAssist 的策略是「跳过 -> 记录 -> 通知 -> 人工复核」，保证 99.9% 的正常文档及时更新。DLQ 中的脏文档通过独立的人工审核流程处理。

本章结论

  • 索引 Pipeline 四阶段：Read（TikaDocumentReader）-> Split（TokenTextSplitter）-> Transform（MetadataEnricher）-> Write（VectorStore），通过 MQ 异步触发
  • 元数据模型包含权限标签（security_level / allowed_roles / department），由自动推断 + 人工确认的结合方式生成，检索时通过 FilterExpression 实现行级安全
  • Java-Python 索引契约通过 OpenAPI 规范定义（POST /api/v2/index/batch），解耦两栈开发
  • 脏文档五大类型各有处理策略，核心原则是跳过 + DLQ + 人工复核，不阻塞主链路
  • 自动权限推断只做初标，最终结果需人工确认，安全团队定期抽查标记准确率

面试可能问

Q1: 索引 Pipeline 遇到大文件解析超时怎么办？

三种策略：① 按页拆分——在 DocumentReader 层设置 max-pages: 100，每次只处理 100 页，分多次 MQ 消息完成；② 异步拆分——PDF 解析后按页面拆成多条 MQ 消息，每条独立消费，并行索引；③ 预检与超时控制——上传时校验文件大小和页数，超限即拒绝并由异步服务处理。CorpAssist 优先采用策略 ②，在上传时不阻塞，由后台 MQ 并行消费。

Q2: 权限标签如果标记错了怎么办（机密文档标成公开）？

属于安全事故等级。多层防御：① 自动推断只做初标，标记结果需要人工确认后方可生效；② 每次索引生成审计事件，安全团队定期抽查（建议每周 5% 样本）；③ 敏感关键词黑名单机制——包含「薪酬」「股权」「审计」等自动标记 confidential；④ 安全设计原则——「降级不降权」，自动推断无法确定时默认为机密而非公开；⑤ 万一发生泄露，通过审计日志追溯责任人。

Q3: Java 和 Python 两套索引服务同时跑，怎么保证索引数据一致？

推荐单写多读模式：生产环境中数据写入由一方负责（Java 侧生产索引），Python 实验只从同样的索引库做只读查询（不会写入）。开发环境两栈各自使用独立的测试索引库，通过统一提交到 Git 的分块参数配置确保一致性。在生产切换前用 Python 的评测工具（RAGAS）对 Java 侧索引结果做一次全量验证。

判断练习

场景：脏文档直接跳过会丢失重要信息。有同事提议「所有文档必须解析成功后才能继续，失败就重试直到成功」，你怎么看？

不推荐。在线索引 Pipeline 因为一个脏文档阻塞，导致所有其他正常文档的索引更新延迟，风险更大。跳过 + DLQ 策略保证 99.9% 的正常文档及时更新。DLQ 中的脏文档通过独立的人工审核流程处理，确保重要信息不会真正丢失——管理员审核后可以修复文档并重新触发索引。

场景：Python 团队修改了分块参数（chunk-size 512 -> 768），怎么保证不影响 Java 生产环境？

标准流程：① Python 在测试索引库上用新参数验证效果（Recall/NDCG）；② 效果确认后提交分块参数变更到配置仓库（Git）；③ CI 自动运行 Java 侧的集成测试（Mock 索引库）；④ 确认测试通过后，通过配置中心（Nacos）灰度下发——先 10% 的文档走新参数索引；⑤ 观察 1-2 天无异常后全量切换。
