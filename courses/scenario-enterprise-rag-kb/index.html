合规边界检查清单

  • 来源合规：哪些文档禁止入库（薪资、未审批、过期）？
  • 角色合规：不同角色能看哪些文档？
  • 场景合规：知识库的答案用于哪些场景（内部问答 vs 客户展示）？
  • 审计合规：谁在什么时间问了什么、看到了什么？
  • 数据驻留：文档数据存储在哪个区域？是否符合 GDPR / 个人信息保护法？

ROI 量化模型

kb roi（知识库投资回报率）是企业 RAG 项目立项的核心依据。CorpAssist 推荐使用以下公式：

ROI 计算公式
text 
年节省人天 = 员工数 × 人均日检索次数 × 每次节省时间(小时) / 8

ROI = (年节省人天 × 人均日薪) / (年系统成本)

其中：
- 年系统成本 = 硬件/云资源 + LLM API 调用费 + 维护人力
- 基准命中率 = 人工检索命中率（约 40%）
- RAG 命中率目标 ≥ 85%
- 节省时间 = 人工检索耗时 × (RAG 命中率 - 基准命中率)

ROI 模型 Python 实现

python 
# roi_model.py — CorpAssist ROI 计算器
def calculate_roi(
    employee_count: int = 1000,
    daily_searches_per_employee: float = 0.5,
    manual_search_minutes: float = 15.0,
    rag_hit_rate: float = 0.85,
    manual_hit_rate: float = 0.40,
    daily_rate_per_person: float = 800.0,  # 人均日薪(元)
    annual_system_cost: float = 300000.0,  # 年系统成本(元)
) -> dict:
    """计算 CorpAssist 知识库 ROI"""
    # 每次检索节省的时间（考虑命中率差异）
    time_saved_per_search = manual_search_minutes * (rag_hit_rate - manual_hit_rate)

    # 年总检索次数
    total_annual_searches = employee_count * daily_searches_per_employee * 250  # 250 工作日

    # 年节省人天
    saved_person_days = (total_annual_searches * time_saved_per_search / 60) / 8

    # 年节省金额
    annual_savings = saved_person_days * daily_rate_per_person

    # ROI
    roi = annual_savings / annual_system_cost

    return {
        "saved_person_days": round(saved_person_days, 1),
        "annual_savings": round(annual_savings, 2),
        "annual_cost": annual_system_cost,
        "roi": round(roi, 2),
        "time_saved_per_search_min": round(time_saved_per_search, 1),
    }

if __name__ == "__main__":
    result = calculate_roi()
    print(f"年节省人天: {result['saved_person_days']}")
    print(f"年节省金额: {result['annual_savings']:.2f} 元")
    print(f"ROI: {result['roi']}x")

常见问题（FAQ）

ROI 模型中的「年系统成本」包含哪些项？
主要包含三部分：(1) 基础设施——向量数据库（如 Milvus/Pinecone）和 GPU 资源年费；(2) LLM API 调用费——按 token 计费，CorpAssist 使用 gpt-4o-mini 时约为 0.15 元/次调用；(3) 维护人力——约 0.5 人年的运维成本。三部分合计约 20-40 万/年（千人规模）。
「合规边界」和「权限控制」有什么区别？
合规边界是策略层——决定哪些文档能进知识库、能被哪些场景使用。权限控制是执行层——在检索时根据用户角色过滤结果。边界在前（文档审核时确定），控制在后（问答执行时落实）。CorpAssist 的两层设计让合规策略与工程实现解耦。详见 权限、审计与多租户。
知识库上线后，如何持续衡量 ROI？
建议建立三个核心指标：(1) 检索命中率——每次问答中 top-5 是否包含正确答案；(2) 员工采纳率——日活员工数 / 全员数；(3) 节省人天——通过问卷估算「之前找多久，现在找多久」。CorpAssist 运营看板自动汇总这三个指标，按月出报表。
CorpAssist 与传统企业搜索（如 Elasticsearch）有何不同？
传统搜索依赖关键词匹配，无法理解「下个月的年假我还没休完怎么办」这类自然语言问题。CorpAssist 的 RAG 架构通过语义向量检索+LLM 推理，理解用户真实意图并给出有引用来源的具体答案，而非返回文档列表。详见 混合检索与 Rerank。

端到端数据流中的关键设计：Ingest-Query Split

ingest query split 是 CorpAssist 的核心设计原则：文档摄入（Ingest）与问答查询（Query）使用不同的技术栈。Python 擅长文档处理（NLP 库多、生态成熟），Java/Spring AI 擅长企业级编排（事务、安全、监控）。二者通过检索层（向量库+ES）解耦。

# index_pipeline.py — CorpAssist 索引管线入口
from unstructured.partition.auto import partition
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Milvus
import json

def ingest_document(file_path: str, metadata: dict):
    """单文档索引流程"""
    # 1. Unstructured 解析
    elements = partition(filename=file_path)
    full_text = "\n\n".join([e.text for e in elements if e.text])

    # 2. 语义切分
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=512, chunk_overlap=128,
        separators=["\n\n", "\n", "。", "！", "？", " ", ""],
    )
    chunks = splitter.create_documents([full_text])

    # 3. 标注元数据（含权限）
    for chunk in chunks:
        chunk.metadata.update(metadata)
        chunk.metadata["doc_id"] = metadata["doc_id"]
        chunk.metadata["version"] = metadata["version"]

    # 4. 向量化 + 写入
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vector_store = Milvus.from_documents(
        documents=chunks,
        embedding=embeddings,
        collection_name="corpassist_kb",
    )
    print(f"Indexed {len(chunks)} chunks for doc {metadata['doc_id']}")

// 选型决策框架 — 面试/答辩可用
public class RAGvsFineTuneDecider {
    enum Approach { RAG, FINE_TUNE, HYBRID, PROMPT_ENG }

    public Approach decide(ScenarioRequest req) {
        boolean needsNewKnowledge = req.hasNewDocuments()
            || req.getDocUpdateFrequency() > 0;
        if (needsNewKnowledge) {
            if (req.requiresStyleAlignment()) {
                return Approach.HYBRID;
            }
            return Approach.RAG;
        }
        boolean needsBehaviorChange = req.hasStyleGuide()
            || req.hasOutputFormatTemplate();
        if (!needsBehaviorChange) return Approach.RAG;
        boolean hasSufficientData = req.getAnnotatedSamples() > 1000;
        if (!hasSufficientData && req.canUsePrompt()) {
            return Approach.PROMPT_ENG;
        }
        return Approach.FINE_TUNE;
    }
}

常见问题（FAQ）

RAG 和微调可以同时用吗？
可以，这是 CorpAssist 推荐的「混合模式」。RAG 负责事实检索，微调负责行为对齐。具体实现：先用 RAG 检索相关文档片段，送入经过微调的 LLM 生成答案。微调不注入新知识，只改变表达方式。
为什么要 Python + Spring AI 双栈，不都用 Python？
Python 在文档处理生态更丰富。Spring AI 在企业级编排方面有成熟方案。双栈的维护成本通过清晰的接口定义（检索层 API 契约）来控制。详见 文档接入与索引流水线 和 Spring AI 问答编排。
在线问答的检索延迟超过 2s 怎么办？
从三个方向优化：(1) 检索层——使用 HNSW 索引降低向量搜索延迟；(2) Rerank 层——控制 Rerank 输入量；(3) 缓存——高频问题用 Redis 缓存答案。详见 效果数据与优化复盘。
微调后的模型还能接 RAG 吗？
可以。微调针对 LLM 参数，不影响 RAG 检索流程。微调后的模型仍然接受 prompt 输入，在 prompt 中拼接检索到的上下文。CorpAssist 的 Advisor 链设计天然支持这种组合。

本章结论

  • 四层架构解耦扩展：接入→处理→检索→编排每层独立演进，Python 索引 + Spring 问答双栈分离是核心设计。
  • 数据流决定工程质量：从上传到检索再到生成，每一步的延迟、权限、一致性控制点都需要明确定义。
  • RAG 优先于微调：企业知识库场景中 RAG 是默认选择，仅当需要行为对齐时考虑微调。决策树帮助快速判断。
  • 答题框架结构化输出：场景→架构→数据流→关键设计→选型→评估，六步框架覆盖面试/答辩所需。

为什么要学本章

分块（chunking）决定了检索质量的上限。再好的检索策略，如果文档切分不合理——把关键技术参数切成两半、把表格拆成碎片——召回率也无法提升。本章通过 CorpAssist 的制度文档、产品表格、代码示例三类典型文档，讲解 fixed chunk、semantic chunk 等策略的选型与参数调优。

学完你能
  • 区分固定分块与语义分块的适用场景并选择合适的 split strategy
  • 为表格和代码块设计特殊的分块策略（table chunk / code block chunk）
  • 通过 chunk experiment 设计实验对比不同参数的召回效果
  • 设置合理的 overlap size 并管理 chunk metadata
本章先记住 3 件事
  • 没有通用最优参数：chunk_size 取决于文档类型，制度类 512-1024，代码类按函数。
  • Overlap 不能替代语义切分：overlap 只是兜底，语义边界切分才是核心。
  • 表格切分需要特殊处理：表格行不能被拆到两个 chunk 里，否则检索语义断裂。

# chunk_strategies.py — 固定分块 vs 语义分块对比
from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    CharacterTextSplitter,
)
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

# 固定分块：按字符数硬切
fixed_splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=128,
    separators=["\n\n", "\n", "。", "！", "？", " ", ""],
)

# 语义分块：基于 embedding 相似度
semantic_splitter = SemanticChunker(
    OpenAIEmbeddings(model="text-embedding-3-small"),
    breakpoint_threshold_type="percentile",  # 按百分位断点
    breakpoint_threshold_amount=95,  # 相似度低于 95% 分位则切分
)

# 对比测试
text = """休假政策：员工每年享有 15 天带薪年假。
年假可按半天为单位申请，需提前 3 个工作日在系统提交。
未休完的年假可顺延至次年 3 月底。超过部分作废。
加班规定：工作日加班按 1.5 倍工资计算。
周末加班可按 2 倍工资或调休处理。
法定节假日加班按 3 倍工资计算。"""

fixed_chunks = fixed_splitter.split_text(text)
semantic_chunks = semantic_splitter.split_text(text)

print(f"固定分块: {len(fixed_chunks)} 块")
for i, c in enumerate(fixed_chunks):
    print(f"  [{i}] {len(c)} chars: {c[:40]}...")

print(f"\n语义分块: {len(semantic_chunks)} 块")
for i, c in enumerate(semantic_chunks):
    print(f"  [{i}] {len(c)} chars: {c[:40]}...")

# custom_chunker.py — 表格和代码块的特殊切分
import re
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.schema import Document

class CorporateChunker:
    """CorpAssist 定制切分器：识别表格和代码块做特殊处理"""

    def __init__(self, chunk_size=512, chunk_overlap=128):
        self.default_splitter = RecursiveCharacterTextSplitter(
            chunk_size=chunk_size,
            chunk_overlap=chunk_overlap,
            separators=["\n\n---\n\n", "\n\n", "\n", "。", " ", ""],
        )
        self.chunk_size = chunk_size

    def split_document(self, doc: Document) -> list[Document]:
        content = doc.page_content
        chunks = []

        # 1. 提取代码块（``` 包裹的片段）
        code_blocks = re.findall(r'```[\s\S]*?```', content)
        remaining = re.sub(r'```[\s\S]*?```', '', content)

        # 代码块按函数切分
        for block in code_blocks:
            funcs = re.split(r'\n(?=def |class |public )', block)
            for func in funcs:
                if func.strip():
                    m = Document(
                        page_content=func.strip(),
                        metadata={**doc.metadata, "type": "code_block"}
                    )
                    chunks.append(m)

        # 2. 提取表格（行中有 | 符号）
        lines = remaining.split("\n")
        table_buffer = []
        non_table_lines = []

        for line in lines:
            if "|" in line and "--" not in line:
                table_buffer.append(line)
            else:
                if table_buffer:
                    # 每行为一个 chunk，保留表头
                    header = table_buffer[0]
                    for row in table_buffer[1:]:
                        chunk_text = f"{header}\n{row}"
                        m = Document(
                            page_content=chunk_text,
                            metadata={**doc.metadata, "type": "table_row"}
                        )
                        chunks.append(m)
                    table_buffer = []
                non_table_lines.append(line)

        # 3. 剩余文本用默认切分器
        default_text = "\n".join(non_table_lines)
        if default_text.strip():
            default_chunks = self.default_splitter.split_documents(
                [Document(page_content=default_text, metadata=doc.metadata)]
            )
            chunks.extend(default_chunks)

        return chunks

# 使用示例
chunker = CorporateChunker(chunk_size=512, chunk_overlap=128)
result = chunker.split_document(Document(
    page_content="## 产品规格表\n| 型号 | 价格 |\n| A100 | ¥99 |\n| A200 | ¥199 |\n\n```python\ndef hello():\n    print('world')\n```",
    metadata={"doc_id": "PROD-001"}
))
for c in result:
    print(f"[{c.metadata['type']}] {c.page_content[:60]}...")

from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from typing import List, Dict

def evaluate_chunk_params(
    docs: List[str],
    test_queries: List[Dict],  # [{"q": "...", "expected_chunk_ids": [...]}]
    chunk_size: int,
    chunk_overlap: int,
) -> dict:
    """评测一组分块参数的检索效果"""
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
    )
    chunks = splitter.create_documents(docs)

    # 建索引
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vector_store = Chroma.from_documents(chunks, embeddings)
    retriever = vector_store.as_retriever(search_kwargs={"k": 5})

    # 评测
    recall_at_5 = 0
    for tq in test_queries:
        results = retriever.invoke(tq["q"])
        retrieved_ids = {r.metadata.get("chunk_id") for r in results}
        expected = set(tq["expected_chunk_ids"])
        if expected & retrieved_ids:
            recall_at_5 += 1 / len(test_queries)

    return {"recall@5": round(recall_at_5, 3)}

常见问题（FAQ）

chunk_size 越大越好吗？
不是。chunk_size 越大，每个 chunk 包含的上下文越多，但检索精度下降——用户问「年假几天」时，一个 2000 字的 chunk 包含 5 个政策条款，LLM 需要从中定位正确答案，容易出错。CorpAssist 经验：制度文档 512-768，技术文档 768-1024。
对于手写扫描件，分块策略有什么不同？
扫描件先过 OCR，OCR 输出通常带有段落标记（如每页底部有页码）。建议在分块前先清洗 OCR 输出：去掉页码、页眉页脚，然后按段落结构做语义分块。如果 OCR 质量差（<80% 准确率），chunk_size 宜减小至 256-384，降低噪音密度。详见 生产坑：数据、权限与索引 的脏 PDF 专题。
分块后 metadata 应该存哪些字段？
CorpAssist 标准元数据：doc_id, version, chunk_index, type(paragraph/table/code), compliance_level, allowed_roles, tenant_id, source_file。这些字段在检索层的 ACL 过滤和引用溯源中都会用到。详见 chunk metadata。
已上线的知识库如何变更分块参数？
分块参数变更需要重新索引（re-index）。CorpAssist 采用「双索引」策略：新参数构建新索引 v2，旧索引 v1 继续服务，验证新索引指标达标后切换。切换过程通过配置中心的开关控制，无需停服。

本章结论

  • 选策略先于调参数：制度文档用递归切分，表格按行保留表头，代码按函数独立。
  • Overlap 是兜底不是银弹：15-25% 的 overlap 可以缓解边界切断问题，但不能替代语义边界的正确切分。
  • 实验驱动参数决策：用测试集做 Recall@5 对比，不同文档类型单独验证。
  • Metadata 是检索层的基础设施：doc_id、version、type、ACL 等元数据必须在索引阶段完整标注。

# hybrid_retrieval.py — CorpAssist 混合检索
from rank_bm25 import BM25Okapi
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from typing import List, Dict
import numpy as np

def rrf_merge(
    vector_results: List[Dict],
    bm25_results: List[Dict],
    k: int = 60,
    top_n: int = 10,
) -> List[Dict]:
    """RRF (Reciprocal Rank Fusion) 融合排序"""
    score_map = {}
    for rank, doc in enumerate(vector_results):
        doc_id = doc.metadata.get("chunk_id", str(rank))
        score_map[doc_id] = score_map.get(doc_id, 0) + 1 / (k + rank + 1)

    for rank, doc in enumerate(bm25_results):
        doc_id = doc.metadata.get("chunk_id", str(rank))
        score_map[doc_id] = score_map.get(doc_id, 0) + 1 / (k + rank + 1)

    # 按 RRF 分数排序
    sorted_ids = sorted(score_map, key=score_map.get, reverse=True)[:top_n]
    return sorted_ids

class HybridRetriever:
    """向量 + BM25 混合检索器"""

    def __init__(self, vector_store, bm25_index, alpha=0.5):
        self.vector_store = vector_store
        self.bm25_index = bm25_index
        self.alpha = alpha  # alpha blend 权重

    def retrieve(self, query: str, top_k: int = 10) -> List:
        # 向量检索
        vec_results = self.vector_store.similarity_search(query, k=top_k * 2)

        # BM25 检索
        tokenized_query = query.split()
        bm25_scores = self.bm25_index.get_scores(tokenized_query)
        top_bm25 = np.argsort(bm25_scores)[::-1][:top_k * 2]

        # RRF 融合
        merged = rrf_merge(vec_results, [], top_n=top_k)
        return merged

先想 10 秒：BM25 的 k 参数从 60 改成 10，会使融合结果有什么变化？

k 值越小，排名靠前的权重越高（对高排名更敏感）。k=10 时前 5 名的权重差异更大，k=60 时各排名权重更平滑。CorpAssist 默认 k=60 追求稳定，在调试时可以用 k=10 放大差异。

# reranker.py — BGE-Reranker 精排
from sentence_transformers import CrossEncoder
from typing import List

class BGEReranker:
    """BGE-Reranker Cross-encoder 精排"""

    def __init__(self, model_name: str = "BAAI/bge-reranker-v2-m3"):
        self.model = CrossEncoder(model_name)

    def rerank(
        self,
        query: str,
        candidates: List[str],
        top_k: int = 5,
    ) -> List[tuple]:
        """对候选文档精排，返回 (text, score) 列表"""
        pairs = [[query, text] for text in candidates]
        scores = self.model.predict(pairs)

        # 排序后取 top_k
        ranked = sorted(
            zip(candidates, scores),
            key=lambda x: x[1],
            reverse=True,
        )
        return ranked[:top_k]

# diagnose_retrieval.py — 检索失败诊断
def diagnose_query(query: str, expected_doc_id: str):
    """诊断单个查询的召回失败原因"""
    # 1. 检查文档是否在索引中
    all_docs = vector_store.get()
    expected_exists = any(
        d.metadata.get("doc_id") == expected_doc_id
        for d in all_docs
    )
    if not expected_exists:
        return "FAIL: 文档未索引或已被删除"

    # 2. 检查向量检索
    vec_results = vector_store.similarity_search(query, k=10)
    vec_hit = any(
        r.metadata.get("doc_id") == expected_doc_id
        for r in vec_results
    )

    # 3. 检查 BM25 检索
    tokenized = query.split()
    bm25_scores = bm25_index.get_scores(tokenized)
    bm25_hit = any(bm25_scores[idx] > 0 for idx, _ in enumerate(all_docs))

    # 4. 检查 ACL
    if vec_hit:
        for r in vec_results:
            if r.metadata.get("doc_id") == expected_doc_id:
                allowed = r.metadata.get("allowed_roles", "")
                if user_role not in allowed:
                    return "FAIL: ACL 过滤阻止了召回"

    if not vec_hit and not bm25_hit:
        return "FAIL: 双路均未命中，考虑 query 扩展或同义词"
    if vec_hit and not bm25_hit:
        return "WARN: 仅向量命中，加入 BM25 可提升"
    if not vec_hit and bm25_hit:
        return "WARN: 仅 BM25 命中，考虑增强向量表征"

    return "PASS: 双路均命中"

常见问题（FAQ）

RRF 融合和加权平均（alpha blend）有什么区别？
RRF 基于排名位置而非分数，天然消除了向量和 BM25 分数尺度不一致的问题。alpha blend（加权平均）需要先将两者分数归一化到同一尺度。CorpAssist 默认用 RRF（k=60），因为更稳定、调参更少。
BGE-Reranker 可以用 GPU 加速吗？
可以。BGE-Reranker 支持 ONNX Runtime 或 PyTorch GPU。CorpAssist 生产环境使用 NVIDIA T4 GPU，30 个候选的精排延迟约 150ms（CPU 约 600ms）。详见 rerank latency。
Rerank 的输入候选集越多越好吗？
不是。候选集越大，Rerank 延迟越高，且 recall 增益递减。CorpAssist 实验显示：top-30 → top-50 的 MRR 提升仅 +0.01，但延迟 +80%。建议设置 top-30 作为默认值。
用户的提问很短（2-3 个字），如何提升检索效果？
使用 Query Rewriting：在检索前让 LLM 将短查询扩展为完整问题。CorpAssist 的 Spring AI Advisor 链中包含 QueryRewriteAdvisor，自动将「年假几天」扩展为「公司规定的年假天数是多少」。详见 混合检索 + Rerank 联调。

本章结论

  • 混合检索是必选项：纯向量检索在企业知识库中会漏掉制度编号、产品型号等精确关键词。
  • 三阶段管线提升命中率：双路检索 → RRF 融合 → Cross-encoder 精排，MRR 可从 0.62 提升至 0.83。
  • Rerank 增益显著：BGE-Reranker 精排可提升首答命中率 10-15 个百分点，是企业场景的标配。
  • 系统化排查召回失败：按「索引存在→向量命中→BM25 命中→ACL 通过」四步诊断，快速定位根因。

# unstructured_parse.py — CorpAssist 文档解析
from unstructured.partition.auto import partition
from unstructured.cleaners.core import clean_extra_whitespace
from typing import List, Dict
import hashlib

def parse_document(file_path: str, ocr_if_needed: bool = True) -> Dict:
    """解析单文档，返回结构化内容"""
    # 自动检测格式并解析
    elements = partition(
        filename=file_path,
        ocr_mode="auto" if ocr_if_needed else "never",
        languages=["chi_sim", "eng"],  # 中英文混合
        include_page_breaks=True,
    )

    # 提取文本，清洗噪声
    pages = []
    full_text_parts = []
    for el in elements:
        if el.text and len(el.text.strip()) > 10:
            cleaned = clean_extra_whitespace(el.text)
            # 过滤常见噪音
            if not any(noise in cleaned for noise in [
                "Microsoft Word", "Created by", "Page ",
            ]):
                full_text_parts.append(cleaned)
                if el.metadata.page_number:
                    pages.append({
                        "page_num": el.metadata.page_number,
                        "text": cleaned,
                    })

    full_text = "\n\n".join(full_text_parts)
    doc_hash = hashlib.md5(full_text.encode()).hexdigest()

    return {
        "content": full_text,
        "pages": pages,
        "char_count": len(full_text),
        "element_count": len(elements),
        "hash": doc_hash,
    }

# 批量解析
def batch_parse(doc_dir: str) -> List[Dict]:

# index_job.py — CorpAssist 批量索引任务
from celery import Celery
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Milvus
from langchain_community.vectorstores import ElasticsearchStore
import logging

logger = logging.getLogger(__name__)
app = Celery("corpassist_index", broker="redis://localhost:6379/0")

embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)

@app.task(bind=True, max_retries=3, default_retry_delay=30)
def index_document_task(self, doc_path: str, metadata: dict):
    """单个文档的索引任务（支持重试）"""
    try:
        # 1. 解析
        parsed = parse_document(doc_path)
        if parsed["char_count"] < 100:
            raise ValueError(f"Low quality parse: {parsed['char_count']} chars")

        # 2. 切分（使用 CorpAssist 定制切分器）
        chunker = CorporateChunker(chunk_size=512, chunk_overlap=128)
        from langchain.schema import Document
        doc = Document(page_content=parsed["content"], metadata=metadata)
        chunks = chunker.split_document(doc)

        # 3. 标注元数据
        for i, chunk in enumerate(chunks):
            chunk.metadata["chunk_index"] = i
            chunk.metadata["chunk_id"] = f"{metadata['doc_id']}-chunk-{i}"
            chunk.metadata["index_version"] = metadata.get("index_version", "v1")

        # 4. 写入向量库
        vector_store = Milvus.from_documents(
            documents=chunks,
            embedding=embeddings,
            collection_name="corpassist_kb",
            index_params={"index_type": "IVF_FLAT", "nlist": 1024},
        )

        # 5. 写入 BM25 索引
        es_store = ElasticsearchStore.from_documents(
            documents=chunks,
            embedding=embeddings,
            index_name="corpassist_bm25",
        )

        logger.info(f"Indexed {len(chunks)} chunks for {metadata['doc_id']}")
        return {"doc_id": metadata["doc_id"], "chunks": len(chunks)}

    except Exception as e:
        logger.error(f"Failed to index {doc_path}: {e}")
        raise self.retry(exc=e)

# 批量触发
def batch_index_all(doc_dir: str):
    """触发目录下所有文档的索引任务"""
    import os
    tasks = []
    for fname in os.listdir(doc_dir):
        path = os.path.join(doc_dir, fname)
        meta = {"doc_id": fname.split(".")[0], "source": fname}
        task = index_document_task.delay(path, meta)
        tasks.append(task)
    return tasks

常见问题（FAQ）

索引管线失败了，整个系统就不能用了吗？
不会。索引管线是离线任务，不影响在线问答服务。但旧索引仍然提供服务，直到新索引验证通过并切换。CorpAssist 采用「两阶段切换」：新索引构建完成后，先在预发环境验证 MRR，达标后通过配置中心开关灰度切换。
如何保证同一份文档不会被重复索引？
通过 doc_id + version 去重。在写入向量库前检查是否已有同 doc_id + version 的 chunk，如果有则跳过（幂等写入）。如果 version 更新，则先删除旧 chunk 再写入新 chunk。CorpAssist 的索引任务管理器维护一个索引状态表。
OCR 识别准确率低怎么办？
三步优化：1) 提升图片质量——扫描时设为 300 DPI 以上；2) 选择更好的 OCR 引擎——PaddleOCR 对中文手写体优于 Tesseract；3) 后处理——对 OCR 结果做拼写检查。CorpAssist 实测 PaddleOCR 对中文合同达到 95%+ 准确率。
一个 200 页的 PDF 索引需要多久？
取决于文档复杂度和硬件配置。以 CorpAssist 生产环境（4 核 CPU、16GB RAM、NVIDIA T4）为例：纯文本 PDF 约 15 秒；含表格 PDF 约 30 秒；扫描件（需 OCR）约 2-3 分钟。建议将大文档拆为单个索引任务，通过 Celery 并行处理。

引用溯源实现

citation source 是 CorpAssist 的生产强制要求。每个答案必须标注信息来源，格式为 [doc_id#chunk_index]。Spring AI 通过自定义 Advisor 和 Prompt 模板实现。

java 
// CustomCitationAdvisor.java — 强制引用溯源
@Component
public class CustomCitationAdvisor implements Advisor {

    @Override
    public AdvisedResponse advise(AdvisedRequest request) {
        // 从上下文中获取检索到的文档
        List documents = request.retrievedDocuments();
        if (documents == null || documents.isEmpty()) {
            return new AdvisedResponse(
                "抱歉，我在知识库中未找到相关信息。",
                request.params()
            );
        }

        // 构建引用标注
        StringBuilder citationBlock = new StringBuilder("\n\n**参考资料：**\n");
        for (int i = 0; i < documents.size(); i++) {
            Document doc = documents.get(i);
            String docId = doc.getMetadata().get("doc_id").toString();
            int chunkIdx = (int) doc.getMetadata().get("chunk_index");
            String source = doc.getMetadata().getOrDefault("source_file", "未知来源").toString();
            citationBlock.append(String.format(
                "[%d] %s#chunk-%d (%s)\n",
                i + 1, docId, chunkIdx, source
            ));
        }

        // 修改 prompt，要求 LLM 在答案中标注引用
        String enhancedPrompt = request.userText() + "\n\n"
            + "注意：对于答案中的每个事实，请在括号中标注来源文档编号，"
            + "例如「根据规定（来源：HR-001#chunk-3），年假为15天」。";

        return new AdvisedResponse(
            enhancedPrompt + citationBlock.toString(),
            request.params()
        );
    }
}

# validate_citations.py — 验证引用完整性
import re
from typing import List

def validate_citation(answer: str, expected_doc_ids: List[str]) -> dict:
    """验证答案中的引用是否完整"""
    # 提取所有引用
    citations = re.findall(r'\[([A-Za-z0-9\-]+)#chunk-(\d+)\]', answer)

    if not citations:
        return {"valid": False, "reason": "答案中无任何引用标注"}

    cited_docs = set(c[0] for c in citations)
    expected_set = set(expected_doc_ids)

    missing_docs = expected_set - cited_docs
    if missing_docs:
        return {
            "valid": False,
            "reason": f"以下文档未被引用: {missing_docs}",
            "citations": citations,
        }

    return {"valid": True, "citations": citations, "count": len(citations)}

Spring AI 流式控制器

java 
// QAController.java — 流式问答端点
@RestController
@RequestMapping("/api/qa")
public class QAController {

    private final ChatClient chatClient;

    public QAController(ChatClient chatClient) {
        this.chatClient = chatClient;
    }

    @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux> streamAnswer(@RequestParam String question) {
        return chatClient.prompt()
            .user(question)
            .advisors(advisor -> advisor
                .param("tenant_id", getCurrentTenant())
                .param("user_role", getCurrentUserRole()))
            .stream()
            .content()
            .map(token -> ServerSentEvent.builder()
                .data(token)
                .event("token")
                .build())
            .concatWithValues(ServerSentEvent.builder()
                .data("[DONE]")
                .event("done")
                .build())
            .doOnError(e -> log.error("Stream error: {}", e.getMessage()));
    }

    @PostMapping("/ask")
    public Mono ask(@RequestBody QuestionRequest req) {
        return Mono.fromFuture(chatClient.prompt()
            .user(req.question())
            .advisors(advisor -> advisor
                .param("tenant_id", req.tenantId())
                .param("user_role", req.userRole()))
            .call()
            .content()
            .thenApply(content -> new AnswerResponse(content)));
    }

    private String getCurrentTenant() {
        // 从 SecurityContext 获取
        return SecurityContextHolder.getContext().getAuthentication()
            .getDetails().toString();
    }
}

常见问题（FAQ）

Spring AI 和 LangChain 可以混用吗？
CorpAssist 架构中二者各司其职：LangChain（Python）做离线索引，Spring AI（Java）做在线问答。不混用在同一个进程中。通过向量库（Milvus）作为共享存储实现数据层面的协作。这是经过生产验证的 dual stack s1 模式。
流式输出时，引用怎么处理？
CorpAssist 采用「检索后置引用」方案：先流式输出答案文本，答案结束后推送一个引用块。这样首 Token 延迟不受影响，同时用户可以看到完整的引用信息。实现方式是在 Advisor 链中增加一个 PostGenerationAdvisor。
FilterExpression 支持复杂条件吗？
支持。Spring AI 的 FilterExpression 支持 AND/OR 组合、IN、CONTAINS、比较操作符。CorpAssist 常用组合：(compliance_level in ['public','internal']) AND (allowed_roles CONTAINS 'employee')。详见 权限、审计与多租户。
如果 LLM 调用失败，Spring AI 有重试机制吗？
有。Spring AI 的 ChatModel 内置重试机制，可通过 spring.ai.retry.max-attempts 和 spring.ai.retry.backoff 配置。CorpAssist 配置为最大 2 次重试，指数退避（初始 1s，最大 10s）。

  • 延迟预算要分阶段管控：检索 + Rerank + LLM 各占多少，超出预期的环节要独立优化。

@Component
public class HybridRetriever {
    private final VectorStore vectorStore;
    private final ElasticsearchVectorStore esStore;

    public List hybridSearch(String query, int topK) {
        // 向量检索
        var vectorResults = vectorStore.similaritySearch(
            SearchRequest.query(query).withTopK(topK * 2));

        // BM25 检索
        var bm25Results = esStore.similaritySearch(
            SearchRequest.query(query).withTopK(topK * 2));

        // RRF 融合
        return rrfMerge(vectorResults, bm25Results, topK, 60);
    }

    private List rrfMerge(List vec, List bm25,
                                     int topK, int k) {
        Map scores = new HashMap<>();
        indexDocs(vec, scores, k);
        indexDocs(bm25, scores, k);

        return scores.entrySet().stream()
            .sorted(Map.Entry.comparingByValue().reversed())
            .limit(topK)
            .map(entry -> findDocById(entry.getKey(), vec, bm25))
            .collect(Collectors.toList());
    }
}

# ablation.py — CorpAssist 消融实验
from dataclasses import dataclass
from typing import List, Callable
import time

@dataclass
class AblationConfig:
    use_vector: bool = True
    use_bm25: bool = True
    use_rrf: bool = True
    use_reranker: bool = True

def run_ablation(
    test_queries: List[dict],  # [{"q": "..", "expected_chunk_ids": [...]}]
    config: AblationConfig,
) -> dict:
    """运行一组 ablation 配置，返回 MRR 和延迟"""
    start = time.time()
    total_mrr = 0.0

    for tq in test_queries:
        candidates = []
        if config.use_vector:
            vec_results = vector_store.similarity_search(tq["q"], k=20)
            candidates.extend(vec_results)
        if config.use_bm25:
            bm25_results = bm25_retrieve(tq["q"], k=20)
            candidates.extend(bm25_results)

        if config.use_rrf:
            candidates = rrf_merge(candidates, [], top_n=10)
        else:
            candidates = candidates[:10]

        if config.use_reranker:
            reranked = reranker.rerank(tq["q"], candidates, top_k=5)
            candidates = [c for c, s in reranked]

        # 计算 MRR
        for rank, doc in enumerate(candidates[:5]):
            if doc.metadata.get("chunk_id") in tq["expected_chunk_ids"]:
                total_mrr += 1.0 / (rank + 1)
                break

    elapsed = time.time() - start
    mrr = total_mrr / len(test_queries)
    return {"mrr": round(mrr, 3), "latency_ms": round(elapsed * 1000)}

# 执行 ablation
configs = [
    ("纯向量", AblationConfig(use_vector=True, use_bm25=False, use_rrf=False, use_reranker=False)),
    ("向量+BM25", AblationConfig(use_vector=True, use_bm25=True, use_rrf=True, use_reranker=False)),
    ("全量(含Rerank)", AblationConfig()),
]
for name, cfg in configs:
    result = run_ablation(test_queries, cfg)

常见问题（FAQ）

Ablation 实验应该在 Python 端还是 Spring 端做？
两端都要做。Python 端做索引参数的 ablation（chunk_size、overlap、embedding 模型），Spring 端做检索策略的 ablation（BM25 on/off、Rerank on/off、RRF k 值）。CorpAssist 的实践是：先 Python 端固定索引，再 Spring 端调检索策略。
Python 和 Spring 两端的 embedding 模型必须一致吗？
必须一致。Python 端用 text-embedding-3-small 向量化，Spring 端查询时也要用同样的模型。否则向量空间的分布不同，检索结果无意义。CorpAssist 在 Spring AI 配置中指定 spring.ai.openai.embedding.model=text-embedding-3-small。
延迟预算中 Rerank 占了 200ms，如何优化？
三种方案：1) 减少 Rerank 候选数（从 top-30 降到 top-20）；2) 使用更快的 Reranker 模型（bge-reranker-v2-m3 比 v1 快 30%）；3) 使用 GPU 推理（NVIDIA T4 约 150ms / 30 候选，CPU 约 600ms）。详见 效果数据与优化复盘。
联调时发现 Python 索引的 Recall@5 是 0.85，但 Spring 端只有 0.76，可能原因？
常见原因：1) Spring 端的 embedding 模型配置与 Python 端不同；2) Spring 端检索的 FilterExpression 额外过滤了部分文档；3) 两端的相似度阈值不一致（Python 端可能没设 threshold 但 Spring 端设了 0.7）。CorpAssist 的 joint debug 工具可以逐层对比两端的检索中间结果。

本章结论

  • 共享索引解耦双栈：Python 写入索引，Spring 读取，通过相同 collection 协作。索引 Schema 一致是前提。
  • Ablation 实验量化每步贡献：BM25 混合 +0.12 MRR，Rerank +0.07 MRR，决策哪些组件必须保留。
  • 延迟预算分阶段控制：向量检索 100ms + BM25 100ms + Rerank 200ms + LLM 500ms，P95 控制在 2s 以内。
  • 联合调优需要两端工具链：Python ablation 框架 + Spring 延迟监控，定位瓶颈时两端日志关联分析。

# refusal_check.py — CorpAssist 拒答判断
class RefusalPolicy:
    """三层拒答策略"""

    SENSITIVE_PATTERNS = [
        r"薪资|工资|salary",
        r"密码|password",
        r"身份证|id_card",
        r"银行账号|bank_account",
    ]

    def __init__(self, score_threshold=0.5, citation_threshold=0.8):
        self.score_threshold = score_threshold
        self.citation_threshold = citation_threshold

    def should_refuse(self, query: str, retrieval_scores: list,
                      citation_count: int, total_sentences: int) -> tuple:
        """返回 (是否拒答, 原因)"""

        # 1. 检索相关度检查
        max_score = max(retrieval_scores) if retrieval_scores else 0
        if max_score < self.score_threshold:
            return True, "NO_RELEVANT_INFO"

        # 2. 引用覆盖率检查
        coverage = citation_count / total_sentences if total_sentences > 0 else 0
        if coverage < self.citation_threshold:
            return True, "INSUFFICIENT_CITATIONS"

        # 3. 合规敏感词检查
        import re
        for pattern in self.SENSITIVE_PATTERNS:
            if re.search(pattern, query, re.IGNORECASE):
                return True, "SENSITIVE_QUERY"

        return False, ""

    def get_refuse_message(self, reason: str) -> str:
        messages = {
            "NO_RELEVANT_INFO": "抱歉，我在知识库中未找到与您问题相关的信息。请尝试换个问法或联系知识库管理员。",
            "INSUFFICIENT_CITATIONS": "我找到了一些可能相关的信息，但不足以形成完整答案。请将问题描述得更具体一些。",
            "SENSITIVE_QUERY": "该问题涉及个人敏感信息，不在 CorpAssist 知识库的服务范围内。",
        }
        return messages.get(reason, "抱歉，暂时无法回答该问题。")

强制引用与置信度评分

citation required 要求每个答案必须包含至少一个引用来源。confidence score 综合检索分数和 LLM 自评，给出答案可信度。

python 
# confidence_scorer.py — 置信度评分
class ConfidenceScorer:
    """综合检索分数和 LLM 自评计算置信度"""

    def __init__(self):
        self.retrieval_weight = 0.6
        self.llm_weight = 0.4

    def score(self, retrieval_scores: list, llm_confidence: float) -> dict:
        """
        计算置信度
        Args:
            retrieval_scores: 各 chunk 的检索相似度得分
            llm_confidence: LLM 自评置信度 (0-1)
        Returns:
            {"score": float, "level": str, "action": str}
        """
        avg_retrieval = sum(retrieval_scores) / len(retrieval_scores) if retrieval_scores else 0
        combined = self.retrieval_weight * avg_retrieval + self.llm_weight * llm_confidence

        if combined >= 0.8:
            return {"score": combined, "level": "high", "action": "direct_answer"}
        elif combined >= 0.6:
            return {"score": combined, "level": "medium", "action": "low_confidence_review"}
        elif combined >= 0.4:
            return {"score": combined, "level": "low", "action": "human_review"}
        else:
            return {"score": combined, "level": "very_low", "action": "refuse"}

from typing import Optional

class ReviewItem:
    def __init__(self, query: str, answer: str,
                 retrieval_scores: list, confidence: float):

        self.query = query
        self.answer = answer

        self.confidence = confidence

        self.reviewer = None
        self.review_note = ""
        self.created_at = datetime.now()

class ReviewQueue:
    """人工复核队列管理"""

    def __init__(self):
        self.items = []

    def enqueue(self, item: ReviewItem):
        self.items.append(item)
        self._notify_admin(item)

    def review(self, item_id: str, reviewer: str,
               action: str, note: str = "") -> bool:
        for item in self.items:
            if item.id == item_id and item.status == "pending":
                item.status = action  # approved/rejected
                item.reviewer = reviewer
                item.review_note = note
                return True
        return False

    def get_pending(self) -> list:
        return [i for i in self.items if i.status == "pending"]

    def _notify_admin(self, item: ReviewItem):
        """发送审核通知（钉钉/邮件）"""
        print(f"[NOTIFY] 新复核项 {item.id}: {item.query[:50]}...")

常见问题（FAQ）

拒答率太高（>30%）怎么办？
拒答率高通常是检索质量或阈值设置问题。排查方向：1) 降低 score_threshold（从 0.5 调至 0.4 观察变化）；2) 检查索引覆盖率——是否有些文档没有被索引；3) 分析拒答原因分布——如果是「NO_RELEVANT_INFO」多，说明检索召回不够；如果是「SENSITIVE_QUERY」多，说明敏感词列表过严。
引用强制是否会影响回答的自然度？
会影响，但这是企业场景必须接受的成本。CorpAssist 的做法是：在答案末尾集中展示引用来源（而非在句中标引用编号），保持正文自然。「后置引用」方式经用户测试满意率从 62% 提升至 89%。
人工复核队列的 SLA 怎么定？
CorpAssist 建议：普通问题 4 小时内审核完毕，紧急问题（打标 urgent）30 分钟内审核。如果队列积压超过 50 条，自动告警知识库管理员。审核后的优质答案（good case）每周汇总补充到评测集和调优数据中。
LLM 自评置信度可靠吗？
不完全可靠。LLM 对自己生成的答案倾向于高置信度（约 0.3 的偏差）。CorpAssist 将 LLM 自评权重设为 0.4，检索分数权重设为 0.6，以检索证据为主。同时增加「反方验证」：让 LLM 从检索文档中找不支持答案的证据，降低自评偏差。

本章结论

  • 三层拒答策略保底线：检索相关度 → 引用覆盖率 → 合规敏感词，任何一层不通过即拒答。
  • 强制引用反幻觉：引用覆盖率达 80% 以上才允许生成答案，答案中必须有可核验的引用。
  • 置信度驱动分级响应：high→直接回答，medium→加警告，low→人工复核，very_low→拒答。
  • 人工复核是兜底机制：审核过的 good case 反馈到评测集和拒答策略优化，形成治理闭环。

# eval_metrics.py — CorpAssist 评测指标
import numpy as np
from typing import List, Dict

def reciprocal_rank(rank: int) -> float:
    """计算单个查询的 RR"""
    return 1.0 / (rank + 1) if rank >= 0 else 0.0

def dcg(relevances: List[float], k: int) -> float:
    """折损累积增益"""
    relevances = relevances[:k]
    return sum(rel / np.log2(i + 2) for i, rel in enumerate(relevances))

def evaluate_retrieval(
    test_queries: List[Dict],
    retriever_callable,
    k: int = 5,
) -> Dict:
    """
    评测检索质量
    test_queries: [{"q": "...", "expected_ids": ["chunk_id1", ...]}]
    """
    total_mrr = 0.0
    total_recall = 0.0
    total_ndcg = 0.0
    n = len(test_queries)

    for tq in test_queries:
        results = retriever_callable(tq["q"], top_k=k)
        retrieved_ids = [r.metadata.get("chunk_id") for r in results]
        expected_set = set(tq["expected_ids"])

        # MRR
        first_rank = -1
        for i, rid in enumerate(retrieved_ids):
            if rid in expected_set:
                first_rank = i
                break
        total_mrr += reciprocal_rank(first_rank)

        # Recall@K
        hits = sum(1 for rid in retrieved_ids if rid in expected_set)
        total_recall += min(hits / len(expected_set), 1.0)

        # NDCG@K
        relevances = [1.0 if rid in expected_set else 0.0 for rid in retrieved_ids]
        ideal_relevances = sorted(relevances, reverse=True)
        actual_dcg = dcg(relevances, k)
        ideal_dcg = dcg(ideal_relevances, k)
        total_ndcg += actual_dcg / ideal_dcg if ideal_dcg > 0 else 0

    return {
        "mrr": round(total_mrr / n, 4),
        "recall": round(total_recall / n, 4),
        "ndcg": round(total_ndcg / n, 4),
        "count": n,
    }

# badcase_attribution.py — 坏例归因
class BadcaseAttributor:
    """四阶段坏例归因"""

    STAGES = ["parse", "chunk", "retrieve", "gen"]

    def attribute(self, query: str, expected_doc_id: str,
                  parsed_text: str, chunks: list,
                  retrieved_ids: list, answer: str) -> str:
        """归因到具体阶段"""

        # 1. Parse 阶段：文档是否被正确解析
        if not parsed_text or len(parsed_text) < 50:
            return "parse"

        # 2. Chunk 阶段：expected doc 是否被正确切分
        expected_chunks = [c for c in chunks
                           if c.metadata.get("doc_id") == expected_doc_id]
        if not expected_chunks:
            return "chunk"

        # 3. Retrieve 阶段：expected chunk 是否被召回
        if expected_doc_id not in retrieved_ids:
            return "retrieve"

        # 4. Gen 阶段：答案是否包含正确信息
        expected_content = expected_chunks[0].page_content[:100]
        if expected_content not in answer:
            return "gen"

        return "pass"

# 归因统计
def attribution_report(attributor, test_cases):
    report = {s: 0 for s in attributor.STAGES}
    report["pass"] = 0
    for case in test_cases:
        stage = attributor.attribute(**case)
        report[stage] += 1
    return report

class RegressionGate:
    """版本发布门禁"""

    def __init__(self, eval_runner):
        self.eval_runner = eval_runner
        self.gates = {
            "mrr": (">=", 0.80),
            "recall": (">=", 0.85),
            "ndcg": (">=", 0.75),
        }

    def check(self, new_index_version: str) -> dict:
        results = self.eval_runner.run(new_index_version)
        report = {"passed": True, "details": {}}
        for metric, (op, threshold) in self.gates.items():
            value = results[metric]
            passed = value >= threshold
            report["details"][metric] = {"value": value, "threshold": threshold, "passed": passed}
            if not passed:
                report["passed"] = False
        return report

常见问题（FAQ）

评测集需要多少条 QA 对？
CorpAssist 经验：至少 100 条，建议 200-300 条。覆盖所有文档类别（制度、产品、技术），并包含边界情况（模糊问题、多义问题）。标注一条 QA 对平均耗时 3-5 分钟，建议由知识库管理员而非研发标注。
增量索引后需要重新跑全量评测吗？
需要。即使只变更了 1 份文档，也建议全量评测——因为向量分布可能变化（新增向量影响 KNN 结果边界）。CorpAssist 的 CI/CD 管线中，每次索引变更自动触发评测，耗时约 2 分钟（200 条 QA）。
门禁没通过（MRR 从 0.82 降到 0.78），可以强制发布吗？
可以，但需要人工审批和灰度策略。CorpAssist 方案：先进行「差异分析」——查看 MRR 下降的具体查询，确认是否属于「可接受回退」（如修复了一个坏例但影响了 3 个好例）。如果下降原因是优化了 A 类场景但影响了 B 类，需要决策优先级。
线上答案错误怎么快速回滚？
通过索引版本切换。CorpAssist 保持最近 3 个版本的索引在线，通过配置中心的 corpassist.index.active_version 开关可以秒级切换到 v1/v2/v3。切换不会影响问答服务，只需重启 Advisor 实例。

# diagnose_pdf.py — 脏 PDF 诊断
import fitz  # PyMuPDF

def diagnose_pdf(file_path: str) -> dict:
    """诊断 PDF 文件状态"""
    doc = fitz.open(file_path)
    report = {
        "file": file_path,
        "pages": len(doc),
        "has_text": False,
        "has_images": False,
        "is_scanned": False,
        "text_length": 0,
        "recommendation": "",
    }

    # 逐页检查
    total_text = ""
    image_count = 0
    for page in doc:
        text = page.get_text()
        total_text += text
        image_count += len(page.get_images())

    report["text_length"] = len(total_text.strip())
    report["has_text"] = len(total_text.strip()) > 50
    report["has_images"] = image_count > 0

    # 判断是否为扫描件
    if not report["has_text"] and report["has_images"]:
        report["is_scanned"] = True
        report["recommendation"] = "扫描件，需要启用 OCR (PaddleOCR)"
    elif not report["has_text"] and not report["has_images"]:
        report["recommendation"] = "加密或损坏 PDF，尝试修复或重新生成"
    elif report["has_text"] and report["text_length"] < 200:
        report["recommendation"] = "文本量少，检查解析完整性"

    return report

# 批量诊断
def batch_diagnose(doc_dir: str):
    import os
    for fname in os.listdir(doc_dir):
        if fname.endswith(".pdf"):
            report = diagnose_pdf(os.path.join(doc_dir, fname))
            if report["is_scanned"]:
                print(f"[SCAN] {fname} — {report['recommendation']}")
            elif report["text_length"] == 0:
                print(f"[FAIL] {fname} — {report['recommendation']}")

ACL 检索层过滤 vs UI 层过滤

permission leak 是知识库最严重的安全事故。ACL 过滤必须在检索层（向量库侧）执行，而非 UI 层（前端侧）。

危险：UI 层过滤
  • 检索返回所有结果（含无权限文档）
  • 前端 JS 过滤掉无权限的结果
  • 攻击者可抓 API 获取完整结果
  • 网络传输中包含敏感数据
安全：检索层过滤
  • FilterExpression 在检索时执行
  • 无权限文档根本不会被检索到
  • API 响应只包含有权限的结果
  • 检索层（向量库/ES）执行过滤

# acl_audit.py — ACL 过滤审计
def audit_acl_filter(vector_store, user_role: str, test_queries: list) -> dict:
    """审计 ACL 过滤是否正确——用户不应看到 forbidden 文档"""
    results = {"pass": True, "violations": []}

    for query in test_queries:
        # 不带 ACL 过滤的检索（管理员模式）
        all_results = vector_store.similarity_search(query, k=20)

        # 带 ACL 过滤的检索（用户模式）
        filtered_results = vector_store.similarity_search(
            query, k=20,
            metadata_filter={"allowed_roles": user_role},
        )

        all_ids = {r.metadata.get("doc_id") for r in all_results}
        filtered_ids = {r.metadata.get("doc_id") for r in filtered_results}
        forbidden_access = all_ids - filtered_ids

        if forbidden_access and any(
            r.metadata.get("compliance_level") == "confidential"
            for r in all_results if r.metadata.get("doc_id") in forbidden_access
        ):
            results["pass"] = False
            results["violations"].append({
                "query": query,
                "forbidden_docs": list(forbidden_access),
            })

    return results

if __name__ == "__main__":
    audit = audit_acl_filter(vector_store, "employee", test_queries)
    if not audit["pass"]:
        print(f"ACL 违规! {len(audit['violations'])} 条查询泄露")
        for v in audit["violations"]:
            print(f"  Query: {v['query']} → 泄露 {v['forbidden_docs']}")

Graph 补召回与 Re-embed 管理

graph recall 解决跨文档关联检索问题——例如用户问「社保缴纳和公积金缴纳有什么关联」，向量检索可能分别召回社保政策和公积金政策，但无法识别二者的关联关系。CorpAssist 通过文档图补充跨文档召回。

python 
# graph_recall.py — 基于文档图的补充召回
class GraphRecallAugmenter:
    """通过文档关联图补充召回"""

    def __init__(self):
        # 文档关联图：doc_id -> [related_doc_ids]
        self.graph = {
            "HR-001": ["HR-002", "FIN-001"],  # 休假政策 → 加班政策
            "HR-002": ["HR-001", "FIN-002"],
            "FIN-001": ["HR-001", "FIN-002"],
        }

    def augment(self, retrieved_docs: list, top_k: int = 3) -> list:
        """如果召回的文档有关联，补充关联文档"""
        retrieved_ids = {d.metadata.get("doc_id") for d in retrieved_docs}
        extra_ids = set()

        for doc_id in retrieved_ids:
            related = self.graph.get(doc_id, [])
            for rid in related:
                if rid not in retrieved_ids:
                    extra_ids.add(rid)

        # 加载关联文档的 chunks（需缓存）
        extra_docs = [self._load_doc(rid) for rid in extra_ids if rid]
        return retrieved_docs + extra_docs[:top_k]

    def _load_doc(self, doc_id: str):
        """从缓存或检索层加载文档"""
        pass

# reembed_manager.py — Re-embed 版本管理
class ReembedManager:
    """Embedding 模型变更管理"""

    def __init__(self, vector_store, eval_runner):
        self.active_versions = {}  # version -> collection_name
        self.vector_store = vector_store
        self.eval_runner = eval_runner

    def build_new_index(self, new_model: str, version: str) -> bool:
        """构建新 embedding 模型的索引"""
        print(f"Building index v{version} with {new_model}...")
        # 新建 collection
        new_store = Milvus.from_documents(
            documents=all_chunks,
            embedding=new_model,
            collection_name=f"corpassist_kb_{version}",
        )
        self.active_versions[version] = f"corpassist_kb_{version}"
        return True

    def compare_versions(self, v1: str, v2: str) -> dict:
        """对比两个版本的 MRR"""
        mrr_v1 = self.eval_runner.run(v1)
        mrr_v2 = self.eval_runner.run(v2)
        diff = mrr_v2["mrr"] - mrr_v1["mrr"]
        return {"v1": mrr_v1, "v2": mrr_v2, "diff": diff}

    def switch_active(self, version: str):
        """切换活跃索引版本"""
        self.vector_store.switch_collection(self.active_versions[version])
        print(f"Switched to index v{version}")

常见问题（FAQ）

扫描 PDF 必须用 OCR 吗？有没有不需要 OCR 的替代方案？
如果 PDF 是纯图片（扫描件），必须 OCR。替代方案：1) 从源头解决——要求文档提供方上传可编辑版本（Word/原生 PDF）；2) 如果无法获得源文件，使用 PaddleOCR（中文最优）或 Azure Document Intelligence（商业方案）。CorpAssist 实测 PaddleOCR 对中文合同达到 95%+ 准确率。
权限泄露事故发生了怎么办？
紧急预案： 1) 立即关闭问答服务或切到维护页面；2) 确认泄露范围（哪些用户看到了哪些文档）；3) 通知合规部门和法务；4) 修复 ACL 过滤（确保 FilterExpression 在检索层生效）；5) 恢复服务前做 ACL 审计测试。事后复盘写入安全运营报告。
Graph 补召回会不会引入噪音？
会。Graph 补召回是基于「关联关系」补充文档，不是基于内容相关度。CorpAssist 的做法是：补召回的结果标记为「graph_expansion」，在 Rerank 阶段给予较低权重（0.7x），避免关联但不相关的文档排在前面。
Re-embed 后 MRR 下降，怎么排查？
三步排查：1) 确认新 embedding 模型的向量维度是否变化（text-embedding-ada-002 是 1536 维，text-embedding-3-small 是 512 维）；2) 确认新旧模型在同一批测试文档上的向量余弦相似度分布；3) 如果分布差异大，考虑在新模型上重新选择 chunk_size（因为语义边界可能变化）。

本章结论

  • 脏 PDF 需在索引前诊断：扫描件必须 OCR，加密 PDF 需解密，解析质量门禁拦截空内容。
  • ACL 过滤必须在检索层：FilterExpression 在向量库和 ES 侧执行，不能依赖 UI 层过滤。
  • Graph 补召回跨文档关联：手动或自动构建文档关联图，补充向量检索无法发现的跨文档关系。
  • Re-embed 必须蓝绿切换：旧版索引保留，新版构建后对比评测，灰度切换，全程可回滚。

行级 ACL 设计

row level acl（行级权限）在 CorpAssist 中实现为每个 chunk 的 metadata 权限标签。检索时根据用户角色和租户信息过滤。

CorpAssist · 行级 ACL 场景 CorpAssist 服务于一家千人企业，需要支持：HR 查看薪资文档、经理查看绩效报告、普通员工只看员工手册。三种角色使用同一个知识库，但看到的文档范围不同。

// AclAdvisor.java — 行级 ACL Advisor
@Component
public class AclAdvisor implements Advisor {

    @Override
    public AdvisedResponse advise(AdvisedRequest request) {
        // 从 SecurityContext 获取用户信息
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        String role = auth.getAuthorities().stream()
            .map(GrantedAuthority::getAuthority)
            .findFirst().orElse("employee");
        String tenantId = (String) auth.getDetails();

        // 构造过滤条件
        String filter = String.format(
            "tenant_id == '%s' AND allowed_roles CONTAINS '%s'",
            tenantId, role
        );

        // 注入到检索请求
        SearchRequest searchRequest = SearchRequest.builder()
            .filterExpression(filter)
            .topK(5)
            .build();

        return request.with(searchRequest);
    }
}

@Component
public class UserContextFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) {
        HttpServletRequest req = (HttpServletRequest) request;
        String authHeader = req.getHeader("Authorization");
        // 解析 JWT，提取 tenant_id 和 role
        Claims claims = Jwts.parser()
            .setSigningKey(secretKey)
            .parseClaimsJws(authHeader.replace("Bearer ", ""))
            .getBody();
        String role = claims.get("role", String.class);
        String tenantId = claims.get("tenant_id", String.class);

        UsernamePasswordAuthenticationToken auth =
            new UsernamePasswordAuthenticationToken(null, null,
                List.of(new SimpleGrantedAuthority(role)));
        auth.setDetails(tenantId);
        SecurityContextHolder.getContext().setAuthentication(auth);
        chain.doFilter(request, response);
    }
}

             acl_pass, answer, confidence, latency_ms, created_at)
            VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, NOW())
            """;

            record.requestId(),
            record.userId(),
            record.tenantId(),
            record.query(),
            String.join(",", record.retrievedChunks()),
            record.aclPass(),
            record.answer(),
            record.confidence(),
            record.latencyMs()

            .answer(event.getAnswer())
            .confidence(event.getConfidence())

多租户隔离方案

multi tenant kb 支持多个企业客户共用同一套 CorpAssist 服务。CorpAssist 推荐逻辑隔离（通过 tenant_id 字段过滤）而非物理隔离（独立实例）。

物理隔离（不推荐）
  • 每租户独立部署一套服务
  • 运维成本线性增长
  • 无法共享基础设施和模型
  • 版本升级需要逐个更新
逻辑隔离（推荐）
  • 所有租户共享同一套服务
  • 通过 tenant_id 字段隔离数据
  • 基础设施和模型共享
  • 版本升级一次生效

逻辑隔离的代价是：一旦 FilterExpression 配置错误（如漏写了 tenant_id），可能导致租户 A 看到租户 B 的文档。CorpAssist 通过以下措施加固：

  • 默认拒绝：FilterExpression 中强制包含 tenant_id，不配置则拒绝查询。
  • 定期 ACL 审计：每周自动运行 acl_audit.py 检测跨租户泄露。
  • 请求溯源：审计日志记录每请求的 tenant_id，便于追溯。

常见问题（FAQ）

行级 ACL 和文档级权限有什么区别？
文档级权限是整个文档要么可见要么不可见。行级 ACL 可以精确到文档内的某个 chunk。例如一份《绩效评估标准》文档，普通经理只能看到评估流程（前 3 个 chunk），HR 总监可以看到全部（含薪酬关联部分）。CorpAssist 通过在索引时标注每个 chunk 的 compliance_level 实现。
审计日志保留多久？
CorpAssist 建议至少保留 6 个月（合规审计要求），推荐 12 个月。超过 12 个月的日志归档到冷存储（如 S3 Glacier）。日志量估算：1000 日活 × 10 次/天 × 1KB/条 ≈ 10MB/天，6 个月约 1.8GB。
多租户场景下，每个租户的 embedding 模型可以不同吗？
技术上可以（不同 tenant_id 使用不同 collection），但运维复杂度高。CorpAssist 推荐所有租户统一 embedding 模型，通过 tenant_id 字段隔离数据。如果租户有特殊需求（如金融行业需要私有化部署 embedding 模型），可以考虑物理隔离。
如何应对合规审计？
CorpAssist 准备了 compliance kb 审计检查表：1) 提供审计日志导出功能（支持 CSV/JSON）；2) 准备 ACL 策略文档和配置截图；3) 提供数据驻留说明（向量库和 ES 所在区域）；4) 提供数据删除流程（租户退租后清理所有数据）。

本章结论

  • 行级 ACL 在索引时标注、检索时执行：每个 chunk 标注 tenant_id/allowed_roles/compliance_level，FilterExpression 在检索层强制过滤。
  • 审计日志记录每次问答：request_id/user_id/query/chunks/answer/confidence/latency，7 个核心字段覆盖合规需求。
  • 逻辑隔离优先于物理隔离：通过 tenant_id 字段实现多租户数据隔离，运维成本远低于独立部署。
  • 安全加固是持续过程：默认拒绝策略 + 定期 ACL 审计 + 请求溯源，三道防线确保数据安全。

    参考答案

    在检索层加硬删除。 除了 FilterExpression 过滤外，在检索结果返回前再执行一道「mask 检查」：如果检索结果中包含 compliance_level=confidential 的 chunk，直接将其从结果集中删除（而非标记「无权限」）。这样 API 响应中完全不包含敏感信息。

    参考答案

    会影响延迟但不影响数据安全。 大租户的索引变更可能导致 Milvus 的 segment 合并，暂时影响同 collection 下所有租户的检索延迟（通常 <100ms 抖动）。数据安全方面不受影响——只要 tenant_id 过滤正确，小租户不会看到大租户的文档。如果抖动不可接受，考虑按租户规模分层：大租户用独立 collection。

# optimization_report.py — 优化报告生成
class OptimizationReport:
    """生成优化旅程报告"""

    def __init__(self):
        self.milestones = []

    def add_milestone(self, name: str, mrr: float, recall: float,
                      latency: int, cost: str, reason: str):
        self.milestones.append({
            "name": name, "mrr": mrr, "recall": recall,
            "latency": latency, "cost": cost, "reason": reason,
        })

    def generate(self) -> str:
        """生成 Markdown 报告"""
        lines = ["# CorpAssist 优化旅程报告\n"]
        lines.append(f"| 阶段 | MRR | Recall@5 | P95 | 成本 | 归因 |")
        lines.append(f"|------|-----|----------|-----|------|------|")

        for m in self.milestones:
            lines.append(
                f"| {m['name']} | {m['mrr']:.2f} | {m['recall']:.2f} | "
                f"{m['latency']}ms | {m['cost']} | {m['reason']} |"
            )

        # 计算总提升
        first = self.milestones[0]
        last = self.milestones[-1]
        lines.append(f"\n## 总提升\n")
        lines.append(f"- MRR: {first['mrr']:.2f} → {last['mrr']:.2f} (+{(last['mrr']-first['mrr'])*100:.0f}pp)")
        lines.append(f"- Recall@5: {first['recall']:.2f} → {last['recall']:.2f}")
        lines.append(f"- P95 延迟: {first['latency']}ms → {last['latency']}ms")
        lines.append(f"- 总成本: 增量约 ¥{self._total_cost()}/月")

        return "\n".join(lines)

    def _total_cost(self) -> int:
        costs = [int(m['cost'].replace('+', '').replace('¥', '').replace('/月', '').split()[0])
                 for m in self.milestones if m['cost'] != '0' and m['cost'] != '—']
        return sum(costs)

// SlaMonitor.java — SLA 监控
@Component
public class SlaMonitor {

    private final MeterRegistry registry;

    @Scheduled(fixedRate = 60000)
    public void checkSla() {
        // P95 延迟检查
        Timer timer = registry.find("rag.total.latency").timer();
        double p95 = timer.percentile(0.95);
        if (p95 > 2000) {
            alertService.sendAlert("SLA_BREACH",
                "P95 latency " + p95 + "ms exceeds 2000ms threshold");
        }

        // MRR 检查
        double mrr = registry.find("rag.mrr").gauge().value();
        if (mrr < 0.80) {
            alertService.sendAlert("SLA_BREACH",
                "MRR " + mrr + " below 0.80 threshold");
        }
    }

    @EventListener
    public void onRequest(RequestCompleteEvent event) {
        registry.counter("rag.requests.total").increment();
        registry.timer("rag.total.latency")
            .record(event.getDuration());
    }
}

常见问题（FAQ）

答辩时，评委问「成本这么高，能再降吗？」怎么回答？
分三条路径回答：1) 基础设施——将 GPU Reranker 改为 CPU 版本（MRR 降约 0.05，成本降 ¥800/月）；2) LLM 调用——用 gpt-4o-mini 替代已是最优，但可加缓存命中高频问题（预计减少 20% 调用）；3) 维护人力——0.5 人年不可压缩，但可通过文档化和自动化减少。核心观点：当前成本对应 MRR=0.85 的最佳性价比。
优化故事中 BM25 贡献最大（+12pp），是不是更应该关注关键词检索？
这说明企业知识库中精确关键词匹配（制度编号、产品型号）占比较高。是的，对于企业场景，关键词检索的贡献往往被低估。但不要忽略向量检索的长尾价值——对于语义匹配问题（「我还没休完的假怎么办」 vs 「年假余额处理」），向量检索不可或缺。两者是互补关系。
SLA 的 P95 延迟包含网络传输吗？
包含，P95 延迟是从用户发起请求到收到完整答案的全链路时间。CorpAssist 在用户侧（浏览器/App）和服务端（Spring AI）分别埋点。服务端 P95 通常比用户侧少 200-500ms（网络传输差异）。SLA 承诺的是服务端 P95。

评估指标：面试回答要点

text 
CorpAssist 用三个指标衡量检索质量：

MRR@K（Mean Reciprocal Rank）：
  第一个正确答案在检索结果中的平均倒数排名。
  例如正确答案排第 2 → 贡献 1/2 = 0.5。
  目标：≥ 0.80。衡量「用户多快看到正确答案」。

Recall@K：
  前 K 个结果中包含正确答案的比例。
  目标：≥ 0.85。衡量「正确答案是否在候选集中」。

NDCG@K（Normalized Discounted Cumulative Gain）：
  考虑排序位置的折损增益。
  目标：≥ 0.75。衡量「排序质量」。

此外我们还跟踪：
- 首 Token 延迟（目标 < 800ms）
- 服务可用性（≥ 99.5%）
- 用户采纳率（日活员工 / 全员数 ≥ 30%）

答辩追问应对

答辩中评委可能会追问以下高频问题，提前准备好了就能从容应对：

追问 1：你提到 Ingest-Query Split，Python 和 Spring 两边如何保证 metadata 一致？
回答： 通过共享的元数据 Schema 定义。我们在项目初期就定义了 corpassist_metadata_schema.json（9 个字段的类型和约束），Python 端索引时按 Schema 写入，Spring 端消费时按 Schema 读取。两端共用同一个 Milvus collection，字段名和类型必须一致。版本变更时先更新 Schema 定义，再更新两端代码。
追问 2：如果数据量从 1000 份文档增长到 10 万份，你的架构怎么扩展？
回答： 四条扩展路径：1) 检索层——Milvus 增加分片数（shard），ES 增加节点；2) 编排层——Spring AI 服务水平扩展实例数，QPS 不够时加实例；3) 索引层——Celery Worker 从 5 个扩展到 20 个并行索引；4) 存储——使用对象存储（S3/MinIO）存储原始文档，Milvus 只存向量。10 万份文档预估检索延迟增加 < 20%，总成本增加约 3-4x。
追问 3：你的系统如何支持多轮对话？
回答： CorpAssist 当前版本通过 Chat History 实现：每轮对话将历史消息拼入 prompt。Spring AI 的 ChatMemory Advisor 自动管理对话上下文。改进方向：使用 Contextual Compression 避免上下文过长，以及通过 Query Rewriting 将「它」等指代替换为具体实体。
追问 4：如果 OpenAI API 不可用，你的系统怎么办？
回答： CorpAssist 设计时就考虑了模型可替换性。Spring AI 抽象了 ChatModel 接口，可以无缝切换为本地部署的模型（如 Qwen2 或 Llama）。我们的生产实践是：默认用 gpt-4o-mini，预配通义千问作为降级方案。当 API 超时或返回错误时自动降级到本地模型，延迟增加约 500ms 但系统不中断。

判断练习

  1. 面试官问「你的知识库能保证 100% 准确吗？」你怎么回答？

    参考答案

    回答： LLM 的生成本质上是概率性的，我们做不到 100% 准确。但我们有三道防线将准确率推到 90%+：1) 检索层——只有高相关度（score>0.5）的文档才进入生成；2) 引用层——每个答案需要至少 80% 的句子有引用来源；3) 审核层——低置信度答案进入人工复核。综合下来，我们的可接受准确率目标是 95%，关键的合规场景要求 99%+。

  2. 全部 13 章学完了，你能用 1 分钟总结 CorpAssist 全貌吗？

    参考答案

    60 秒总结： CorpAssist 是一个企业级 RAG 知识库系统，采用 Python 离线索引 + Spring AI 在线问答的双栈架构。索引管线用 Unstructured 解析多格式文档，CorporateChunker 按类型智能切分，写入 Milvus + ES 混合索引。问答服务用 BM25+向量混合检索、RRF 融合、BGE-Reranker 精排、LLM 生成带引用溯源的答案。通过行级 ACL 和审计日志保障安全，评测飞轮（MRR/Recall/NDCG）驱动持续优化。
