本章先记住 3 件事

BM25 + 向量：为什么需要混合检索

hybrid retrieval 是企业知识库检索的标配。纯向量检索在语义相似场景表现出色，但在精确关键词匹配上可能漏掉——例如用户输入「HR-001 号文件」时，向量检索可能将「HR-001」理解为「人力资源相关的文件」，而不是精确匹配文件编号。

CorpAssist · 检索失败场景 员工问「Q4-2024 季度的报销截止日期是哪天」。纯向量检索返回了「报销制度 v2.1」的 general 内容，但漏掉了标题包含「Q4-2024 报销截止通知」的关键文档。因为「Q4-2024」在向量空间中没有独立语义。

BM25 分支捕获「HR-001」精确匹配，向量分支捕获语义相关，RRF 融合后命中率显著提升。

混合检索 Python 实现

python 
# hybrid_retrieval.py — CorpAssist 混合检索
from rank_bm25 import BM25Okapi
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from typing import List, Dict
import numpy as np

def rrf_merge(
    vector_results: List[Dict],
    bm25_results: List[Dict],
    k: int = 60,
    top_n: int = 10,
) -> List[Dict]:
    """RRF (Reciprocal Rank Fusion) 融合排序"""
    score_map = {}
    for rank, doc in enumerate(vector_results):
        doc_id = doc.metadata.get("chunk_id", str(rank))
        score_map[doc_id] = score_map.get(doc_id, 0) + 1 / (k + rank + 1)

    for rank, doc in enumerate(bm25_results):
        doc_id = doc.metadata.get("chunk_id", str(rank))
        score_map[doc_id] = score_map.get(doc_id, 0) + 1 / (k + rank + 1)

    # 按 RRF 分数排序
    sorted_ids = sorted(score_map, key=score_map.get, reverse=True)[:top_n]
    return sorted_ids

class HybridRetriever:
    """向量 + BM25 混合检索器"""

    def __init__(self, vector_store, bm25_index, alpha=0.5):
        self.vector_store = vector_store
        self.bm25_index = bm25_index
        self.alpha = alpha  # alpha blend 权重

    def retrieve(self, query: str, top_k: int = 10) -> List:
        # 向量检索
        vec_results = self.vector_store.similarity_search(query, k=top_k * 2)

        # BM25 检索
        tokenized_query = query.split()
        bm25_scores = self.bm25_index.get_scores(tokenized_query)
        top_bm25 = np.argsort(bm25_scores)[::-1][:top_k * 2]

        # RRF 融合
        merged = rrf_merge(vec_results, [], top_n=top_k)
        return merged

# 使用示例
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vector_store = Chroma(
    persist_directory="./data/chroma_db",
    embedding_function=embeddings,
)
retriever = HybridRetriever(vector_store, bm25_index, alpha=0.5)
results = retriever.retrieve("HR-001 号文件关于年假的规定")

Spring AI 中的混合检索配置

java 
// Spring AI 自定义混合检索实现
@Component
public class HybridSearchService {
    private final VectorStore vectorStore;
    private final ElasticsearchRestTemplate esTemplate;

    public List hybridSearch(String query, int topK) {
        // 1. 向量检索
        var vectorResults = vectorStore.similaritySearch(
            SearchRequest.query(query).withTopK(topK * 2));

        // 2. BM25 检索（通过 Elasticsearch）
        var bm25Query = QueryBuilders.multiMatchQuery(query, "content", "title");
        var esResults = esTemplate.search(bm25Query, Document.class);

        // 3. RRF 融合
        return rrfMerge(vectorResults, esResults, topK);
    }

    private List rrfMerge(List vec, List bm25, int k) {
        Map scores = new HashMap<>();
        for (int i = 0; i < vec.size(); i++) {
            scores.merge(vec.get(i).getId(), 1.0 / (60 + i), Double::sum);
        }
        for (int i = 0; i < bm25.size(); i++) {
            scores.merge(bm25.get(i).getId(), 1.0 / (60 + i), Double::sum);
        }
        return scores.entrySet().stream()
            .sorted(Map.Entry.comparingByValue().reversed())
            .limit(k)
            .map(e -> /* 从缓存加载 Document */)
            .collect(Collectors.toList());
    }
}
先想 10 秒：BM25 的 k 参数从 60 改成 10，会使融合结果有什么变化？

k 值越小，排名靠前的权重越高（对高排名更敏感）。k=10 时前 5 名的权重差异更大，k=60 时各排名权重更平滑。CorpAssist 默认 k=60 追求稳定，在调试时可以用 k=10 放大差异。

权重调优与 Rerank

weight tuning 解决的是混合检索中向量和 BM25 各自贡献的占比问题。加上 bge reranker 精排后，检索管线分为三个步骤：

检索 + Rerank 三阶段管线
flowchart LR
    Q[用户提问] --> S1[Stage 1: 双路检索]
    S1 --> V[向量检索\ntop-50]
    S1 --> B[BM25 检索\ntop-50]
    V --> F[Stage 2: RRF 融合]
    B --> F
    F --> R[top-30 候选]
    R --> S3[Stage 3: BGE-Reranker\nCross-encoder 精排]
    S3 --> OUT[top-5 最终结果\n送入 LLM]

三阶段管线：双路粗排 → RRF 融合 → Cross-encoder 精排，每级缩小候选集。

BGE-Reranker 集成代码

python 
# reranker.py — BGE-Reranker 精排
from sentence_transformers import CrossEncoder
from typing import List

class BGEReranker:
    """BGE-Reranker Cross-encoder 精排"""

    def __init__(self, model_name: str = "BAAI/bge-reranker-v2-m3"):
        self.model = CrossEncoder(model_name)

    def rerank(
        self,
        query: str,
        candidates: List[str],
        top_k: int = 5,
    ) -> List[tuple]:
        """对候选文档精排，返回 (text, score) 列表"""
        pairs = [[query, text] for text in candidates]
        scores = self.model.predict(pairs)

        # 排序后取 top_k
        ranked = sorted(
            zip(candidates, scores),
            key=lambda x: x[1],
            reverse=True,
        )
        return ranked[:top_k]

# CorpAssist 检索管线完整样例
retriever = HybridRetriever(vector_store, bm25_index)
reranker = BGEReranker()

query = "年假可以累计到下一年吗"
candidates = retriever.retrieve(query, top_k=30)
candidate_texts = [c.page_content for c in candidates]

final_results = reranker.rerank(query, candidate_texts, top_k=5)
for text, score in final_results:
    print(f"score={score:.4f}: {text[:60]}...")
    # score > 0.5 表示强相关，0.3-0.5 弱相关，< 0.3 不相关

@Component
public class RerankerService {
    private final OrtSession session;
    private final OrtEnvironment env;

        this.env = OrtEnvironment.getEnvironment();
        this.session = env.createSession("models/bge-reranker-v2-m3.onnx");

    public List rerank(String query, List candidates, int topK) {

                return new ScoredDocument(doc, score);

            .sorted(Comparator.reverseOrder())
            .limit(topK)
            .collect(Collectors.toList());
    }

}

检索失败模式与排查

recall failure（召回失败）是知识库上线后最常见的运维问题。CorpAssist 总结了四大典型模式：

失败模式现象根因解决
语义漂移 相关文档未被召回 查询与文档用词不同（如「销假」vs「取消休假」） MultiQuery 改写、增加同义词、query 扩展
关键词缺失 制度编号查不到 纯向量忽略精确匹配 加 BM25 混合、提升关键词权重
权限过滤 部分用户查不到 ACL 过滤去掉了用户有权限的文档 检查 metadata ACL 标签正确性
索引过期 最新版文档查不到 索引 v1 还指向旧版文档 增量索引、版本切换

排查用 Python 诊断脚本

python 
# diagnose_retrieval.py — 检索失败诊断
def diagnose_query(query: str, expected_doc_id: str):
    """诊断单个查询的召回失败原因"""
    # 1. 检查文档是否在索引中
    all_docs = vector_store.get()
    expected_exists = any(
        d.metadata.get("doc_id") == expected_doc_id
        for d in all_docs
    )
    if not expected_exists:
        return "FAIL: 文档未索引或已被删除"

    # 2. 检查向量检索
    vec_results = vector_store.similarity_search(query, k=10)
    vec_hit = any(
        r.metadata.get("doc_id") == expected_doc_id
        for r in vec_results
    )

    # 3. 检查 BM25 检索
    tokenized = query.split()
    bm25_scores = bm25_index.get_scores(tokenized)
    bm25_hit = any(bm25_scores[idx] > 0 for idx, _ in enumerate(all_docs))

    # 4. 检查 ACL
    if vec_hit:
        for r in vec_results:
            if r.metadata.get("doc_id") == expected_doc_id:
                allowed = r.metadata.get("allowed_roles", "")
                if user_role not in allowed:
                    return "FAIL: ACL 过滤阻止了召回"

    if not vec_hit and not bm25_hit:
        return "FAIL: 双路均未命中，考虑 query 扩展或同义词"
    if vec_hit and not bm25_hit:
        return "WARN: 仅向量命中，加入 BM25 可提升"
    if not vec_hit and bm25_hit:
        return "WARN: 仅 BM25 命中，考虑增强向量表征"

    return "PASS: 双路均命中"

常见问题（FAQ）

RRF 融合和加权平均（alpha blend）有什么区别？
RRF 基于排名位置而非分数，天然消除了向量和 BM25 分数尺度不一致的问题。alpha blend（加权平均）需要先将两者分数归一化到同一尺度。CorpAssist 默认用 RRF（k=60），因为更稳定、调参更少。
BGE-Reranker 可以用 GPU 加速吗？
可以。BGE-Reranker 支持 ONNX Runtime 或 PyTorch GPU。CorpAssist 生产环境使用 NVIDIA T4 GPU，30 个候选的精排延迟约 150ms（CPU 约 600ms）。详见 rerank latency。
Rerank 的输入候选集越多越好吗？
不是。候选集越大，Rerank 延迟越高，且 recall 增益递减。CorpAssist 实验显示：top-30 → top-50 的 MRR 提升仅 +0.01，但延迟 +80%。建议设置 top-30 作为默认值。
用户的提问很短（2-3 个字），如何提升检索效果？
使用 Query Rewriting：在检索前让 LLM 将短查询扩展为完整问题。CorpAssist 的 Spring AI Advisor 链中包含 QueryRewriteAdvisor，自动将「年假几天」扩展为「公司规定的年假天数是多少」。详见 混合检索 + Rerank 联调。

本章结论

  • 混合检索是必选项：纯向量检索在企业知识库中会漏掉制度编号、产品型号等精确关键词。
  • 三阶段管线提升命中率：双路检索 → RRF 融合 → Cross-encoder 精排，MRR 可从 0.62 提升至 0.83。
  • Rerank 增益显著：BGE-Reranker 精排可提升首答命中率 10-15 个百分点，是企业场景的标配。
  • 系统化排查召回失败：按「索引存在→向量命中→BM25 命中→ACL 通过」四步诊断，快速定位根因。

判断练习

  1. 某次查询中，正确文档在向量 top-2 但 BM25 top-100+，RRF 结果中正确文档排在第 3。RRF 的 k 设为 10 还是 60 能将其提升到第 1？

    参考答案

    k=10 可能不升反降。 因为 RRF 公式中 k 越小排名权重越大，但 BM25 的排名很低（100+），贡献的 RRF 分数几乎为 0。解决方案是增加 BM25 的召回数（BM25 top-200），或直接使用 alpha blend 加权。

  2. 生产环境 MRR 从 0.83 降至 0.71，最可能的原因是什么？

    参考答案

    嵌入模型变更或向量漂移。 常见原因：1) embedding 模型升级但未 re-index；2) 新索引的 chunk 参数与旧索引不同导致向量分布变化；3) Reranker 模型被误替换。参见 生产坑 re-embed。
