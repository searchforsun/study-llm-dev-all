脏 PDF 解析链路

dirty pdf 是 CorpAssist 最常见的数据质量问题。典型脏 PDF 包括：扫描件（图片无 OCR）、加密 PDF、含杂音字体的排版 PDF。

CorpAssist · 脏 PDF 事故 某客户索引了 200 份扫描版 PDF（1990 年代的操作规程），索引管线没有触发 OCR，所有文档都成功「解析」了但提取内容为空。结果线上检索 100% 命中率为 0。排查发现 partition() 的 ocr_mode 默认是 "never"。

脏 PDF 排查树：从「文本是否为空」开始，逐级排查。

python 
# diagnose_pdf.py — 脏 PDF 诊断
import fitz  # PyMuPDF

def diagnose_pdf(file_path: str) -> dict:
    """诊断 PDF 文件状态"""
    doc = fitz.open(file_path)
    report = {
        "file": file_path,
        "pages": len(doc),
        "has_text": False,
        "has_images": False,
        "is_scanned": False,
        "text_length": 0,
        "recommendation": "",
    }

    # 逐页检查
    total_text = ""
    image_count = 0
    for page in doc:
        text = page.get_text()
        total_text += text
        image_count += len(page.get_images())

    report["text_length"] = len(total_text.strip())
    report["has_text"] = len(total_text.strip()) > 50
    report["has_images"] = image_count > 0

    # 判断是否为扫描件
    if not report["has_text"] and report["has_images"]:
        report["is_scanned"] = True
        report["recommendation"] = "扫描件，需要启用 OCR (PaddleOCR)"
    elif not report["has_text"] and not report["has_images"]:
        report["recommendation"] = "加密或损坏 PDF，尝试修复或重新生成"
    elif report["has_text"] and report["text_length"] < 200:
        report["recommendation"] = "文本量少，检查解析完整性"

    return report

# 批量诊断
def batch_diagnose(doc_dir: str):
    import os
    for fname in os.listdir(doc_dir):
        if fname.endswith(".pdf"):
            report = diagnose_pdf(os.path.join(doc_dir, fname))
            if report["is_scanned"]:
                print(f"[SCAN] {fname} — {report['recommendation']}")
            elif report["text_length"] == 0:
                print(f"[FAIL] {fname} — {report['recommendation']}")

ACL 检索层过滤 vs UI 层过滤

permission leak 是知识库最严重的安全事故。ACL 过滤必须在检索层（向量库侧）执行，而非 UI 层（前端侧）。

危险：UI 层过滤
  • 检索返回所有结果（含无权限文档）
  • 前端 JS 过滤掉无权限的结果
  • 攻击者可抓 API 获取完整结果
  • 网络传输中包含敏感数据
安全：检索层过滤
  • FilterExpression 在检索时执行
  • 无权限文档根本不会被检索到
  • API 响应只包含有权限的结果
  • 检索层（向量库/ES）执行过滤

ACL 过滤验证脚本

python 
# acl_audit.py — ACL 过滤审计
def audit_acl_filter(vector_store, user_role: str, test_queries: list) -> dict:
    """审计 ACL 过滤是否正确——用户不应看到 forbidden 文档"""
    results = {"pass": True, "violations": []}

    for query in test_queries:
        # 不带 ACL 过滤的检索（管理员模式）
        all_results = vector_store.similarity_search(query, k=20)

        # 带 ACL 过滤的检索（用户模式）
        filtered_results = vector_store.similarity_search(
            query, k=20,
            metadata_filter={"allowed_roles": user_role},
        )

        all_ids = {r.metadata.get("doc_id") for r in all_results}
        filtered_ids = {r.metadata.get("doc_id") for r in filtered_results}
        forbidden_access = all_ids - filtered_ids

        if forbidden_access and any(
            r.metadata.get("compliance_level") == "confidential"
            for r in all_results if r.metadata.get("doc_id") in forbidden_access
        ):
            results["pass"] = False
            results["violations"].append({
                "query": query,
                "forbidden_docs": list(forbidden_access),
            })

    return results

if __name__ == "__main__":
    audit = audit_acl_filter(vector_store, "employee", test_queries)
    if not audit["pass"]:
        print(f"ACL 违规! {len(audit['violations'])} 条查询泄露")
        for v in audit["violations"]:
            print(f"  Query: {v['query']} → 泄露 {v['forbidden_docs']}")

Graph 补召回与 Re-embed 管理

graph recall 解决跨文档关联检索问题——例如用户问「社保缴纳和公积金缴纳有什么关联」，向量检索可能分别召回社保政策和公积金政策，但无法识别二者的关联关系。CorpAssist 通过文档图补充跨文档召回。

python 
# graph_recall.py — 基于文档图的补充召回
class GraphRecallAugmenter:
    """通过文档关联图补充召回"""

    def __init__(self):
        # 文档关联图：doc_id -> [related_doc_ids]
        self.graph = {
            "HR-001": ["HR-002", "FIN-001"],  # 休假政策 → 加班政策
            "HR-002": ["HR-001", "FIN-002"],
            "FIN-001": ["HR-001", "FIN-002"],
        }

    def augment(self, retrieved_docs: list, top_k: int = 3) -> list:
        """如果召回的文档有关联，补充关联文档"""
        retrieved_ids = {d.metadata.get("doc_id") for d in retrieved_docs}
        extra_ids = set()

        for doc_id in retrieved_ids:
            related = self.graph.get(doc_id, [])
            for rid in related:
                if rid not in retrieved_ids:
                    extra_ids.add(rid)

        # 加载关联文档的 chunks（需缓存）
        extra_docs = [self._load_doc(rid) for rid in extra_ids if rid]
        return retrieved_docs + extra_docs[:top_k]

    def _load_doc(self, doc_id: str):
        """从缓存或检索层加载文档"""
        pass

Re-embed 版本管理

re embed 发生在 embedding 模型升级时（如从 text-embedding-ada-002 升级到 text-embedding-3-small）。新旧模型生成的向量分布不同，不能原地覆盖。

Re-embed 蓝绿切换流程
flowchart TD
    A[旧 embedding 模型] --> B[构建索引 v1]
    C[新 embedding 模型] --> D[构建索引 v2\n双 collection]
    D --> E[评测 v2 MRR]
    E --> F{MRR v2 >= v1?}
    F -->|是| G[切 10% 流量到 v2]
    G --> H[观察 24h]
    H --> I[全量切到 v2]
    F -->|否| J[分析差异原因]
    J --> K[调 chunk 参数\n或放弃升级]

Re-embed 必须经过「蓝绿构建 → 对比评测 → 灰度切换 → 全量切换」四步。

python 
# reembed_manager.py — Re-embed 版本管理
class ReembedManager:
    """Embedding 模型变更管理"""

    def __init__(self, vector_store, eval_runner):
        self.active_versions = {}  # version -> collection_name
        self.vector_store = vector_store
        self.eval_runner = eval_runner

    def build_new_index(self, new_model: str, version: str) -> bool:
        """构建新 embedding 模型的索引"""
        print(f"Building index v{version} with {new_model}...")
        # 新建 collection
        new_store = Milvus.from_documents(
            documents=all_chunks,
            embedding=new_model,
            collection_name=f"corpassist_kb_{version}",
        )
        self.active_versions[version] = f"corpassist_kb_{version}"
        return True

    def compare_versions(self, v1: str, v2: str) -> dict:
        """对比两个版本的 MRR"""
        mrr_v1 = self.eval_runner.run(v1)
        mrr_v2 = self.eval_runner.run(v2)
        diff = mrr_v2["mrr"] - mrr_v1["mrr"]
        return {"v1": mrr_v1, "v2": mrr_v2, "diff": diff}

    def switch_active(self, version: str):
        """切换活跃索引版本"""
        self.vector_store.switch_collection(self.active_versions[version])
        print(f"Switched to index v{version}")

常见问题（FAQ）

扫描 PDF 必须用 OCR 吗？有没有不需要 OCR 的替代方案？
如果 PDF 是纯图片（扫描件），必须 OCR。替代方案：1) 从源头解决——要求文档提供方上传可编辑版本（Word/原生 PDF）；2) 如果无法获得源文件，使用 PaddleOCR（中文最优）或 Azure Document Intelligence（商业方案）。CorpAssist 实测 PaddleOCR 对中文合同达到 95%+ 准确率。
权限泄露事故发生了怎么办？
紧急预案： 1) 立即关闭问答服务或切到维护页面；2) 确认泄露范围（哪些用户看到了哪些文档）；3) 通知合规部门和法务；4) 修复 ACL 过滤（确保 FilterExpression 在检索层生效）；5) 恢复服务前做 ACL 审计测试。事后复盘写入安全运营报告。
Graph 补召回会不会引入噪音？
会。Graph 补召回是基于「关联关系」补充文档，不是基于内容相关度。CorpAssist 的做法是：补召回的结果标记为「graph_expansion」，在 Rerank 阶段给予较低权重（0.7x），避免关联但不相关的文档排在前面。
Re-embed 后 MRR 下降，怎么排查？
三步排查：1) 确认新 embedding 模型的向量维度是否变化（text-embedding-ada-002 是 1536 维，text-embedding-3-small 是 512 维）；2) 确认新旧模型在同一批测试文档上的向量余弦相似度分布；3) 如果分布差异大，考虑在新模型上重新选择 chunk_size（因为语义边界可能变化）。

本章结论

  • 脏 PDF 需在索引前诊断：扫描件必须 OCR，加密 PDF 需解密，解析质量门禁拦截空内容。
  • ACL 过滤必须在检索层：FilterExpression 在向量库和 ES 侧执行，不能依赖 UI 层过滤。
  • Graph 补召回跨文档关联：手动或自动构建文档关联图，补充向量检索无法发现的跨文档关系。
  • Re-embed 必须蓝绿切换：旧版索引保留，新版构建后对比评测，灰度切换，全程可回滚。

判断练习

  1. 用户反馈「搜索 HR 制度时出现了财务报销单」，可能是什么原因？

    参考答案

    三种可能： 1) ACL 未生效——财务报销单的 allowed_roles 包含了该用户的角色；2) embedding 语义相似度误匹配——「报销」和「制度」在语义上接近；3) BM25 关键词匹配——「制度」一词出现在报销单中。排查步骤：看返回文档的 metadata 中的 doc_id 和 doc_category 确认是否属于「财务」类别。

  2. 昨晚执行了 re-embed，今早 MRR 从 0.82 降到 0.65。最快的止损方案是什么？

    参考答案

    立即回滚索引版本。 如果 re-embed 后旧索引 v1 还保留着（蓝绿部署），通过配置中心将 corpassist.index.active_version 切回 v1，无需停机。然后分析 v2 MRR 下降的原因——通常是新 embedding 模型需配合新的 chunk_size。
