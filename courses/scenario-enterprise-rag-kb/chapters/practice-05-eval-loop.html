评测指标：MRR、NDCG、Recall@K

mrr ndcg 是信息检索领域最常用的三个指标。CorpAssist 用它们衡量检索质量。

CorpAssist · 评测场景 CorpAssist 评测团队标注了 200 条 QA 对（每条问题关联 1-3 个 expected chunk）。迭代索引策略后，需要对比新旧版本的 MRR 和 Recall@5 来判断是否通过门禁。
指标含义范围CorpAssist 目标
MRR@K 第一个正确答案在检索结果中的平均倒数排名 0~1 ≥ 0.80
Recall@K 前 K 个结果中包含正确答案的比例 0~1 ≥ 0.85
NDCG@K 考虑排名的归一化折损累积增益 0~1 ≥ 0.75

评测代码

python 
# eval_metrics.py — CorpAssist 评测指标
import numpy as np
from typing import List, Dict

def reciprocal_rank(rank: int) -> float:
    """计算单个查询的 RR"""
    return 1.0 / (rank + 1) if rank >= 0 else 0.0

def dcg(relevances: List[float], k: int) -> float:
    """折损累积增益"""
    relevances = relevances[:k]
    return sum(rel / np.log2(i + 2) for i, rel in enumerate(relevances))

def evaluate_retrieval(
    test_queries: List[Dict],
    retriever_callable,
    k: int = 5,
) -> Dict:
    """
    评测检索质量
    test_queries: [{"q": "...", "expected_ids": ["chunk_id1", ...]}]
    """
    total_mrr = 0.0
    total_recall = 0.0
    total_ndcg = 0.0
    n = len(test_queries)

    for tq in test_queries:
        results = retriever_callable(tq["q"], top_k=k)
        retrieved_ids = [r.metadata.get("chunk_id") for r in results]
        expected_set = set(tq["expected_ids"])

        # MRR
        first_rank = -1
        for i, rid in enumerate(retrieved_ids):
            if rid in expected_set:
                first_rank = i
                break
        total_mrr += reciprocal_rank(first_rank)

        # Recall@K
        hits = sum(1 for rid in retrieved_ids if rid in expected_set)
        total_recall += min(hits / len(expected_set), 1.0)

        # NDCG@K
        relevances = [1.0 if rid in expected_set else 0.0 for rid in retrieved_ids]
        ideal_relevances = sorted(relevances, reverse=True)
        actual_dcg = dcg(relevances, k)
        ideal_dcg = dcg(ideal_relevances, k)
        total_ndcg += actual_dcg / ideal_dcg if ideal_dcg > 0 else 0

    return {
        "mrr": round(total_mrr / n, 4),
        "recall": round(total_recall / n, 4),
        "ndcg": round(total_ndcg / n, 4),
        "count": n,
    }

if __name__ == "__main__":
    # 模拟评测

  • 评测结果关联 changelog
先想 10 秒：MRR 和 Recall@5 哪个对用户体验更重要？

Recall@5 决定「正确答案是否在候选里」，是基础；MRR 决定「正确答案排第几」，直接影响用户看到有用内容的速度。CorpAssist 两个都看，但迭代时优先保 MRR。

坏例归因代码

python 
# badcase_attribution.py — 坏例归因
class BadcaseAttributor:
    """四阶段坏例归因"""

    STAGES = ["parse", "chunk", "retrieve", "gen"]

    def attribute(self, query: str, expected_doc_id: str,
                  parsed_text: str, chunks: list,
                  retrieved_ids: list, answer: str) -> str:
        """归因到具体阶段"""

        # 1. Parse 阶段：文档是否被正确解析
        if not parsed_text or len(parsed_text) < 50:
            return "parse"

        # 2. Chunk 阶段：expected doc 是否被正确切分
        expected_chunks = [c for c in chunks
                           if c.metadata.get("doc_id") == expected_doc_id]
        if not expected_chunks:
            return "chunk"

        # 3. Retrieve 阶段：expected chunk 是否被召回
        if expected_doc_id not in retrieved_ids:
            return "retrieve"

        # 4. Gen 阶段：答案是否包含正确信息
        expected_content = expected_chunks[0].page_content[:100]
        if expected_content not in answer:
            return "gen"

        return "pass"

# 归因统计
def attribution_report(attributor, test_cases):
    report = {s: 0 for s in attributor.STAGES}
    report["pass"] = 0
    for case in test_cases:
        stage = attributor.attribute(**case)
        report[stage] += 1
    return report

增量索引与版本门禁

incremental index 支持文档变更时只重新索引变更部分，而非全量重建。version release 通过蓝绿部署确保平滑切换。

python 
# incremental_index.py — 增量索引
class IncrementalIndexManager:
    """增量索引管理"""

    def __init__(self, vector_store, index_version="v1"):
        self.vector_store = vector_store
        self.index_version = index_version

    def upsert_document(self, doc_id: str, new_version: str,
                        chunks: list) -> bool:
        """增量更新文档：删除旧版 chunk，写入新版"""
        # 1. 删除旧版本 chunk
        delete_filter = {
            "doc_id": doc_id,
            "index_version": self.index_version,
        }
        self.vector_store.delete(metadata_filter=delete_filter)

        # 2. 写入新版 chunk
        for chunk in chunks:
            chunk.metadata["doc_id"] = doc_id
            chunk.metadata["version"] = new_version
            chunk.metadata["index_version"] = self.index_version

        self.vector_store.add_documents(chunks)
        print(f"Upserted {len(chunks)} chunks for {doc_id} v{new_version}")
        return True

    def rollback_to_version(self, doc_id: str, target_version: str):
        """回滚到指定版本"""
        # 从备份存储中恢复
        pass

    def list_current_versions(self, doc_id: str) -> list:
        """列出文档的当前索引版本"""
        results = self.vector_store.similarity_search(
            query="", k=1000,
            metadata_filter={"doc_id": doc_id},
        )
        versions = set(r.metadata.get("version") for r in results)
        return sorted(versions)


# 版本门禁检查
class RegressionGate:
    """版本发布门禁"""

    def __init__(self, eval_runner):
        self.eval_runner = eval_runner
        self.gates = {
            "mrr": (">=", 0.80),
            "recall": (">=", 0.85),
            "ndcg": (">=", 0.75),
        }

    def check(self, new_index_version: str) -> dict:
        results = self.eval_runner.run(new_index_version)
        report = {"passed": True, "details": {}}
        for metric, (op, threshold) in self.gates.items():
            value = results[metric]
            passed = value >= threshold
            report["details"][metric] = {"value": value, "threshold": threshold, "passed": passed}
            if not passed:
                report["passed"] = False
        return report

常见问题（FAQ）

评测集需要多少条 QA 对？
CorpAssist 经验：至少 100 条，建议 200-300 条。覆盖所有文档类别（制度、产品、技术），并包含边界情况（模糊问题、多义问题）。标注一条 QA 对平均耗时 3-5 分钟，建议由知识库管理员而非研发标注。
增量索引后需要重新跑全量评测吗？
需要。即使只变更了 1 份文档，也建议全量评测——因为向量分布可能变化（新增向量影响 KNN 结果边界）。CorpAssist 的 CI/CD 管线中，每次索引变更自动触发评测，耗时约 2 分钟（200 条 QA）。
门禁没通过（MRR 从 0.82 降到 0.78），可以强制发布吗？
可以，但需要人工审批和灰度策略。CorpAssist 方案：先进行「差异分析」——查看 MRR 下降的具体查询，确认是否属于「可接受回退」（如修复了一个坏例但影响了 3 个好例）。如果下降原因是优化了 A 类场景但影响了 B 类，需要决策优先级。
线上答案错误怎么快速回滚？
通过索引版本切换。CorpAssist 保持最近 3 个版本的索引在线，通过配置中心的 corpassist.index.active_version 开关可以秒级切换到 v1/v2/v3。切换不会影响问答服务，只需重启 Advisor 实例。

本章结论

  • 三指标全面衡量检索质量：MRR（排名质量）、Recall@K（覆盖度）、NDCG（排序增益），目标分别为 ≥0.80/≥0.85/≥0.75。
  • 四阶段归因定位根因：parse/chunk/retrieve/gen 四个阶段，每个阶段有对应的优化手段。
  • 评测飞轮驱动持续迭代：标注→评测→归因→优化→发布→收集坏例，形成数据驱动的优化闭环。
  • 版本门禁防止回退：增量索引 + 蓝绿部署 + 自动门禁，确保每一次版本变更质量可控。

判断练习

  1. 200 条评测集中有 50 条归因到「retrieve 阶段」，你会优先尝试哪些优化？

    参考答案

    优先顺序： 1) 检查 BM25 权重是否过低；2) 检查 Reranker 的候选集大小；3) 验证 embedding 模型是否与索引时一致；4) 分析这些坏例是否集中在特定文档类别，针对性地调 chunk_size。CorpAssist 经验：retrieve 阶段问题通常混合权重或 Rerank 配置不当。

  2. 团队说「我们没有资源标注 200 条评测集，先上线再说」，你怎么说服他们？

    参考答案

    方案： 1) 先用 30 条核心 QA 对做基线，承诺 2 周内扩展到 100 条；2) 使用「半自动标注」——先用 LLM 生成 QA 对，人工校核（比从零标注快 3 倍）；3) 标注评测集的成本远低于「无评测上线导致生产事故」的代价。没有评测 = 不知道效果好坏 = 无法迭代。
