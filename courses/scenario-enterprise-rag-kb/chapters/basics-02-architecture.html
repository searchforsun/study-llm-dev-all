本章先记住 3 件事
  • RAG 适合「事实问答」：答案来自企业文档，LLM 只做摘要与改写。ingest query split 是核心设计原则。
  • 微调适合「行为对齐」：要求模型以特定风格回答或执行复杂指令时考虑微调，但无法注入新知识。

CorpAssist 分层架构设计

layered arch（分层架构）将系统按职责分为四层，每层可独立升级、扩缩容。

CorpAssist · 分层架构 某制造业客户部署 CorpAssist 后，需要对接多种文档源（SharePoint、NAS、扫描仪），支持千人并发问答，并在 500ms 内完成检索+生成。分层架构让每层的扩展互不干扰。

接入层处理多端请求 → 编排层做 RAG 编排 → 检索层做混合检索+权限过滤 → 处理层做离线索引构建。

分层职责明细

层次核心职责技术栈扩展方式
接入层 多端接入、身份认证、限流 Spring Cloud Gateway / Kong 水平扩展实例数
编排层 RAG 编排、Advisor 链、流式输出 Spring AI / Spring Boot 按 QPS 扩展
检索层 混合检索、Rerank、权限过滤 Vector Store + Elasticsearch 分片数 + 副本数
处理层 文档解析、切分、向量化、索引 Python / LangChain / Unstructured 任务队列 + Worker
不推荐
  • Python + LangChain 同时做索引和在线问答
  • 检索与处理层共用同一台机器
  • 无 API Gateway，客户端直连问答服务
  • 所有层绑定同一技术栈
推荐
  • Python 离线索引 + Spring AI 在线问答（双栈分离）
  • 检索层独立集群部署
  • API Gateway 统一鉴权与限流
  • 每层选最优技术栈
先想 10 秒：为什么在线问答用 Java Spring AI 而不用 Python FastAPI？

企业场景需要 Spring 生态的成熟特性：事务管理、声明式缓存、AOP 审计、与现有 Java 基础设施集成。Python 更适合离线的数据密集型任务。

索引阶段批量处理，问答阶段实时流式。两个阶段通过共享的检索层（VS + ES）衔接。

端到端数据流中的关键设计：Ingest-Query Split

ingest query split 是 CorpAssist 的核心设计原则：文档摄入（Ingest）与问答查询（Query）使用不同的技术栈。Python 擅长文档处理（NLP 库多、生态成熟），Java/Spring AI 擅长企业级编排（事务、安全、监控）。二者通过检索层（向量库+ES）解耦。

Python 索引管线入口

python 
# index_pipeline.py — CorpAssist 索引管线入口
from unstructured.partition.auto import partition
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Milvus
import json

def ingest_document(file_path: str, metadata: dict):
    """单文档索引流程"""
    # 1. Unstructured 解析
    elements = partition(filename=file_path)
    full_text = "\n\n".join([e.text for e in elements if e.text])

    # 2. 语义切分
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=512, chunk_overlap=128,
        separators=["\n\n", "\n", "。", "！", "？", " ", ""],
    )
    chunks = splitter.create_documents([full_text])

    # 3. 标注元数据（含权限）
    for chunk in chunks:
        chunk.metadata.update(metadata)
        chunk.metadata["doc_id"] = metadata["doc_id"]
        chunk.metadata["version"] = metadata["version"]

    # 4. 向量化 + 写入
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vector_store = Milvus.from_documents(
        documents=chunks,
        embedding=embeddings,
        collection_name="corpassist_kb",
    )
    print(f"Indexed {len(chunks)} chunks for doc {metadata['doc_id']}")

if __name__ == "__main__":
    ingest_document("./data/employee_handbook_v3.pdf", {
        "doc_id": "HR-001", "version": "3.0",
        "compliance_level": "public", "allowed_roles": "employee",
    })

RAG vs 微调：选型决策树

rag vs ft tree 是 CorpAssist 架构设计中最关键的决策之一。很多团队「先做 RAG 再补微调」，但根据场景选择初始方案可以节省 80% 的重复工作。

    Q1 -->|是| RAG[选 RAG]
    Q1 -->|否| Q2[需要改变行为/风格？]

    Q3 -->|是| FT[选微调]

    Q4 -->|是| PE[Prompt 工程]

    RAG --> R1[优势：知识实时更新]
    RAG --> R2[优势：可溯源、可控幻觉]

决策树从「需要注入新知识吗」开始，多数企业知识库场景落在 RAG 分支。

CorpAssist 场景下的选择建议

业务场景推荐方案理由
制度文档问答 RAG 文档频繁更新，RAG 仅需重新索引
客服话术生成 RAG + 微调 RAG 获取产品知识，微调统一回答风格
代码生成辅助 微调 需要特定的代码风格与 API 使用模式
多语言翻译 微调 纯能力提升（行为对齐），无需外挂知识
合规审查问答 RAG + 强引用 必须可溯源，降低幻觉风险

决策实现的伪代码框架

java 
// 选型决策框架 — 面试/答辩可用
public class RAGvsFineTuneDecider {
    enum Approach { RAG, FINE_TUNE, HYBRID, PROMPT_ENG }

    public Approach decide(ScenarioRequest req) {
        boolean needsNewKnowledge = req.hasNewDocuments()
            || req.getDocUpdateFrequency() > 0;
        if (needsNewKnowledge) {
            if (req.requiresStyleAlignment()) {
                return Approach.HYBRID;
            }
            return Approach.RAG;
        }
        boolean needsBehaviorChange = req.hasStyleGuide()
            || req.hasOutputFormatTemplate();
        if (!needsBehaviorChange) return Approach.RAG;
        boolean hasSufficientData = req.getAnnotatedSamples() > 1000;
        if (!hasSufficientData && req.canUsePrompt()) {
            return Approach.PROMPT_ENG;
        }
        return Approach.FINE_TUNE;
    }
}

常见问题（FAQ）

RAG 和微调可以同时用吗？
可以，这是 CorpAssist 推荐的「混合模式」。RAG 负责事实检索，微调负责行为对齐。具体实现：先用 RAG 检索相关文档片段，送入经过微调的 LLM 生成答案。微调不注入新知识，只改变表达方式。
为什么要 Python + Spring AI 双栈，不都用 Python？
Python 在文档处理生态更丰富。Spring AI 在企业级编排方面有成熟方案。双栈的维护成本通过清晰的接口定义（检索层 API 契约）来控制。详见 文档接入与索引流水线 和 Spring AI 问答编排。
在线问答的检索延迟超过 2s 怎么办？
从三个方向优化：(1) 检索层——使用 HNSW 索引降低向量搜索延迟；(2) Rerank 层——控制 Rerank 输入量；(3) 缓存——高频问题用 Redis 缓存答案。详见 效果数据与优化复盘。
微调后的模型还能接 RAG 吗？
可以。微调针对 LLM 参数，不影响 RAG 检索流程。微调后的模型仍然接受 prompt 输入，在 prompt 中拼接检索到的上下文。CorpAssist 的 Advisor 链设计天然支持这种组合。

本章结论

  • 四层架构解耦扩展：接入→处理→检索→编排每层独立演进，Python 索引 + Spring 问答双栈分离是核心设计。
  • 数据流决定工程质量：从上传到检索再到生成，每一步的延迟、权限、一致性控制点都需要明确定义。
  • RAG 优先于微调：企业知识库场景中 RAG 是默认选择，仅当需要行为对齐时考虑微调。决策树帮助快速判断。
  • 答题框架结构化输出：场景→架构→数据流→关键设计→选型→评估，六步框架覆盖面试/答辩所需。

判断练习

  1. 团队说「我们用微调吧，RAG 太复杂了」，场景是公司制度问答（300 份文档，每月更新 10 份）。你如何回应？

    参考答案

    建议选 RAG。 每月 10 份更新的频率下，微调需要重新训练，成本和风险远高于 RAG 的重新索引。而且制度问答需要引用溯源，RAG 天然支持引用标注。

  2. 数据流中，如果不做 Rerank 直接取 top-5 向量结果送入 LLM，会出现什么问题？

    参考答案

    精度下降。 向量检索可能召回「相关但不准确」的结果。Rerank 通过 cross-encoder 精排，可提升 MRR 约 10-15%。CorpAssist 实测显示加 Rerank 后首答准确率从 78% 提升至 87%。
