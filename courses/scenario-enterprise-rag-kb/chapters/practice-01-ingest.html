为什么要学本章

索引管线是 RAG 系统的「制造车间」——文档在这里从原始 PDF 变为可检索的向量索引。如果管线出问题，在线问答质量直接受损。本章用 CorpAssist 的 ingest pipeline 讲解如何使用 unstructured parse 解析多格式文档，用 python index job 构建索引任务，并标注 metadata acl 实现权限过滤。

学完你能
  • 用 Unstructured 解析 PDF/Word/扫描件并控制 parse quality
  • 设计 Python 批量索引任务（batch ingest）
  • 在 chunk 元数据中标注权限标签（doc permission）
  • 配置索引 metadata schema（index metadata）
本章先记住 3 件事
  • 脏 PDF 是头号敌人：扫描件不做 OCR 直接索引 = 检索不到任何内容。
  • 元数据在索引阶段标注：ACL、版本、来源等信息在写入向量库前必须完整。
  • 索引管线是可重入的：断点续传、幂等写入是生产级管线的基本要求。

Unstructured 文档解析

unstructured parse 是 CorpAssist 文档接入的核心组件，支持 PDF、Word、Excel、HTML、扫描件等多种格式。

CorpAssist · 文档接入场景 CorpAssist 的某制造业客户有 5000+ 份文档，涵盖扫描版 PDF（1990 年代的手写操作规程）、Word 版制度文件、Excel 版产品价目表。索引管线需要统一处理这些异构文档。

Unstructured 解析代码

python 
# unstructured_parse.py — CorpAssist 文档解析
from unstructured.partition.auto import partition
from unstructured.cleaners.core import clean_extra_whitespace
from typing import List, Dict
import hashlib

def parse_document(file_path: str, ocr_if_needed: bool = True) -> Dict:
    """解析单文档，返回结构化内容"""
    # 自动检测格式并解析
    elements = partition(
        filename=file_path,
        ocr_mode="auto" if ocr_if_needed else "never",
        languages=["chi_sim", "eng"],  # 中英文混合
        include_page_breaks=True,
    )

    # 提取文本，清洗噪声
    pages = []
    full_text_parts = []
    for el in elements:
        if el.text and len(el.text.strip()) > 10:
            cleaned = clean_extra_whitespace(el.text)
            # 过滤常见噪音
            if not any(noise in cleaned for noise in [
                "Microsoft Word", "Created by", "Page ",
            ]):
                full_text_parts.append(cleaned)
                if el.metadata.page_number:
                    pages.append({
                        "page_num": el.metadata.page_number,
                        "text": cleaned,
                    })

    full_text = "\n\n".join(full_text_parts)
    doc_hash = hashlib.md5(full_text.encode()).hexdigest()

    return {
        "content": full_text,
        "pages": pages,
        "char_count": len(full_text),
        "element_count": len(elements),
        "hash": doc_hash,
    }

# 批量解析
def batch_parse(doc_dir: str) -> List[Dict]:
    """批量解析目录下所有文档"""
    import os
    results = []
    for fname in os.listdir(doc_dir):
        if fname.endswith((".pdf", ".docx", ".xlsx", ".html")):
            result = parse_document(os.path.join(doc_dir, fname))
            result["file_name"] = fname
            results.append(result)
            print(f"Parsed {fname}: {result['char_count']} chars")
    return results

Python 索引任务架构

python index job 是 CorpAssist 的批量索引任务框架。它负责任务调度、状态管理、断点续传。

索引任务调度架构
flowchart TD
    S[Scheduler\nAPScheduler/Celery] --> Q[任务队列\nRedis/RabbitMQ]
    Q --> W1[Worker 1]
    Q --> W2[Worker 2]
    Q --> W3[Worker N]

    subgraph Worker 流程
        W1 --> R[从队列取任务]
        R --> P[Unstructured 解析]
        P --> C[CorporateChunker]
        C --> V[向量化]
        V --> M[写向量库 + ES]
        M --> S2[更新任务状态\ncompleted]
    end

    S2 --> E{解析失败?}
    E -->|是| D[标记错误\n可重试]
    E -->|否| DONE[完成]

Celery 任务队列 + 多 Worker 并行索引，支持断点续传和错误重试。

批量索引管线代码

python 
# index_job.py — CorpAssist 批量索引任务
from celery import Celery
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Milvus
from langchain_community.vectorstores import ElasticsearchStore
import logging

logger = logging.getLogger(__name__)
app = Celery("corpassist_index", broker="redis://localhost:6379/0")

embeddings = OpenAIEmbeddings(model="text-embedding-3-small", dimensions=512)

@app.task(bind=True, max_retries=3, default_retry_delay=30)
def index_document_task(self, doc_path: str, metadata: dict):
    """单个文档的索引任务（支持重试）"""
    try:
        # 1. 解析
        parsed = parse_document(doc_path)
        if parsed["char_count"] < 100:
            raise ValueError(f"Low quality parse: {parsed['char_count']} chars")

        # 2. 切分（使用 CorpAssist 定制切分器）
        chunker = CorporateChunker(chunk_size=512, chunk_overlap=128)
        from langchain.schema import Document
        doc = Document(page_content=parsed["content"], metadata=metadata)
        chunks = chunker.split_document(doc)

        # 3. 标注元数据
        for i, chunk in enumerate(chunks):
            chunk.metadata["chunk_index"] = i
            chunk.metadata["chunk_id"] = f"{metadata['doc_id']}-chunk-{i}"
            chunk.metadata["index_version"] = metadata.get("index_version", "v1")

        # 4. 写入向量库
        vector_store = Milvus.from_documents(
            documents=chunks,
            embedding=embeddings,
            collection_name="corpassist_kb",
            index_params={"index_type": "IVF_FLAT", "nlist": 1024},
        )

        # 5. 写入 BM25 索引
        es_store = ElasticsearchStore.from_documents(
            documents=chunks,
            embedding=embeddings,
            index_name="corpassist_bm25",
        )

        logger.info(f"Indexed {len(chunks)} chunks for {metadata['doc_id']}")
        return {"doc_id": metadata["doc_id"], "chunks": len(chunks)}

    except Exception as e:
        logger.error(f"Failed to index {doc_path}: {e}")
        raise self.retry(exc=e)

# 批量触发
def batch_index_all(doc_dir: str):
    """触发目录下所有文档的索引任务"""
    import os
    tasks = []
    for fname in os.listdir(doc_dir):
        path = os.path.join(doc_dir, fname)
        meta = {"doc_id": fname.split(".")[0], "source": fname}
        task = index_document_task.delay(path, meta)
        tasks.append(task)
    return tasks

Celery 任务支持最大 3 次重试，每次重试间隔 30s，避免瞬时故障导致索引失败。

元数据与权限标签设计

metadata acl 是 CorpAssist 权限体系的基石。每个 chunk 在索引阶段就带上了权限标签，后续检索层直接按标签过滤。

元数据 Schema 设计

字段类型示例用途
doc_idstringHR-001文档唯一标识
versionstringv3.2文档版本，用于版本过滤
tenant_idstringacme-corp多租户隔离
chunk_indexint5chunk 在文档中的序号
compliance_levelstringconfidential合规等级：public/internal/confidential
allowed_rolesstringemployee,manager,hr逗号分隔的允许角色列表
doc_categorystringhr_policy文档分类，用于检索层面过滤
index_versionstringv1索引版本号，用于增量更新
source_filestringhandbook_v3.pdf来源文件名，用于引用溯源
不推荐 · 无元数据
  • chunk 只有 content，无 doc_id/version
  • 无法追溯答案来自哪份文档
  • 无法按租户/角色过滤
  • 索引版本无法管理
推荐 · CorpAssist Schema
  • 9 个标准化元数据字段
  • 引用溯源显示 doc_id + chunk_index
  • 检索层 FilterExpression 按角色过滤
  • 支持蓝绿索引版本切换

// 在 Spring AI 中配置 ACL 过滤
@Bean
public QuestionAnswerAdvisor aclAwareAdvisor(VectorStore vectorStore) {
    return QuestionAnswerAdvisor.builder(vectorStore)
        .searchRequest(SearchRequest.builder()
            .filterExpression("allowed_roles CONTAINS 'employee'")
            .filterExpression("compliance_level in ['public', 'internal']")
            .topK(5)
            .build())
        .build();
}

// 引用溯源 — 从 metadata 中提取来源
@Component
public class CitationFormatter {
    public String formatCitations(List documents) {
        return documents.stream()
            .map(doc -> String.format("[%s#chunk-%d]",
                doc.getMetadata().get("doc_id"),
                doc.getMetadata().get("chunk_index")))
            .collect(Collectors.joining(", "));
    }
}

常见问题（FAQ）

索引管线失败了，整个系统就不能用了吗？
不会。索引管线是离线任务，不影响在线问答服务。但旧索引仍然提供服务，直到新索引验证通过并切换。CorpAssist 采用「两阶段切换」：新索引构建完成后，先在预发环境验证 MRR，达标后通过配置中心开关灰度切换。
如何保证同一份文档不会被重复索引？
通过 doc_id + version 去重。在写入向量库前检查是否已有同 doc_id + version 的 chunk，如果有则跳过（幂等写入）。如果 version 更新，则先删除旧 chunk 再写入新 chunk。CorpAssist 的索引任务管理器维护一个索引状态表。
OCR 识别准确率低怎么办？
三步优化：1) 提升图片质量——扫描时设为 300 DPI 以上；2) 选择更好的 OCR 引擎——PaddleOCR 对中文手写体优于 Tesseract；3) 后处理——对 OCR 结果做拼写检查。CorpAssist 实测 PaddleOCR 对中文合同达到 95%+ 准确率。
一个 200 页的 PDF 索引需要多久？
取决于文档复杂度和硬件配置。以 CorpAssist 生产环境（4 核 CPU、16GB RAM、NVIDIA T4）为例：纯文本 PDF 约 15 秒；含表格 PDF 约 30 秒；扫描件（需 OCR）约 2-3 分钟。建议将大文档拆为单个索引任务，通过 Celery 并行处理。

本章结论

  • Unstructured 统一解析入口：支持 PDF/Word/Excel/扫描件，OCR 自动降级，质量门禁拦截低质量解析。
  • Python 索引任务可扩展：Celery + Worker 架构支持批量并发，任务重试和幂等写入保证数据一致性。
  • Metadata ACL 在索引阶段标注：9 个标准化字段支持租户、角色、版本、分类的精细权限控制。
  • 索引与在线问答解耦：索引管线失败不影响在线服务，蓝绿切换保障版本平滑过渡。

判断练习

  1. 某文档索引后，检索不到任何内容。已知文档是扫描版 PDF。排查步骤是什么？

    参考答案

    步骤： 1) 检查 parse_quality 日志，看 OCR 是否被触发；2) 查看 OCR 置信度是否低于 0.6；3) 查看提取文本中是否包含有意义的字符；4) 检查 quality_gate min_text_length 是否过滤了该文档；5) 如 OCR 未触发，检查 partition 的 ocr_mode 配置。

  2. 索引管线需要处理 5000 份文档，预计耗时多久？如何加速？

    参考答案

    估算： 假设平均每份文档索引耗时 10s（含 OCR），串行需要 ~14 小时。通过 Celery 启动 10 个 Worker 并行，可缩短至 ~1.5 小时。建议还将 PDF 解析和向量化拆分为两个任务，通过任务管道流水线加速。
