行级 ACL 设计

row level acl（行级权限）在 CorpAssist 中实现为每个 chunk 的 metadata 权限标签。检索时根据用户角色和租户信息过滤。

CorpAssist · 行级 ACL 场景 CorpAssist 服务于一家千人企业，需要支持：HR 查看薪资文档、经理查看绩效报告、普通员工只看员工手册。三种角色使用同一个知识库，但看到的文档范围不同。

Spring AI 行级 ACL 实现

java 
// AclAdvisor.java — 行级 ACL Advisor
@Component
public class AclAdvisor implements Advisor {

    @Override
    public AdvisedResponse advise(AdvisedRequest request) {
        // 从 SecurityContext 获取用户信息
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        String role = auth.getAuthorities().stream()
            .map(GrantedAuthority::getAuthority)
            .findFirst().orElse("employee");
        String tenantId = (String) auth.getDetails();

        // 构造过滤条件
        String filter = String.format(
            "tenant_id == '%s' AND allowed_roles CONTAINS '%s'",
            tenantId, role
        );

        // 注入到检索请求
        SearchRequest searchRequest = SearchRequest.builder()
            .filterExpression(filter)
            .topK(5)
            .build();

        return request.with(searchRequest);
    }
}

// 用户身份解析过滤器
@Component
public class UserContextFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) {
        HttpServletRequest req = (HttpServletRequest) request;
        String authHeader = req.getHeader("Authorization");
        // 解析 JWT，提取 tenant_id 和 role
        Claims claims = Jwts.parser()
            .setSigningKey(secretKey)
            .parseClaimsJws(authHeader.replace("Bearer ", ""))
            .getBody();
        String role = claims.get("role", String.class);
        String tenantId = claims.get("tenant_id", String.class);

        UsernamePasswordAuthenticationToken auth =
            new UsernamePasswordAuthenticationToken(null, null,
                List.of(new SimpleGrantedAuthority(role)));
        auth.setDetails(tenantId);
        SecurityContextHolder.getContext().setAuthentication(auth);
        chain.doFilter(request, response);
    }
}

# acl_indexing.py — 索引时标注 ACL
def index_doc_with_acl(doc_path: str, tenant_id: str, allowed_roles: list,
                        compliance_level: str):
    """带 ACL 标签的文档索引"""
    chunks = parse_and_chunk(doc_path)
    for chunk in chunks:
        chunk.metadata["tenant_id"] = tenant_id
        chunk.metadata["allowed_roles"] = ",".join(allowed_roles)
        chunk.metadata["compliance_level"] = compliance_level

    vector_store.add_documents(chunks)
    es_store.add_documents(chunks)

审计日志实现

audit log kb 记录每一次问答交互的完整上下文，用于合规审计和安全追溯。

java 
// AuditLogger.java — 审计日志记录
@Component
public class AuditLogger {

    private final JdbcTemplate jdbcTemplate;

    @Async
    public void logQuery(AuditRecord record) {
        String sql = """
            INSERT INTO kb_audit_log
            (request_id, user_id, tenant_id, query, retrieved_chunks,
             acl_pass, answer, confidence, latency_ms, created_at)
            VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, NOW())
            """;
        jdbcTemplate.update(sql,
            record.requestId(),
            record.userId(),
            record.tenantId(),
            record.query(),
            String.join(",", record.retrievedChunks()),
            record.aclPass(),
            record.answer(),
            record.confidence(),
            record.latencyMs()
        );
    }

    @EventListener
    public void onQueryComplete(QueryCompleteEvent event) {
        AuditRecord record = AuditRecord.builder()
            .requestId(event.getRequestId())
            .userId(event.getUserId())
            .tenantId(event.getTenantId())
            .query(event.getQuery())
            .retrievedChunks(event.getRetrievedChunkIds())
            .aclPass(event.isAclPass())
            .answer(event.getAnswer())
            .confidence(event.getConfidence())
            .latencyMs(event.getLatencyMs())
            .build();
        logQuery(record);
    }
}

// 审计记录模型
record AuditRecord(
    String requestId,
    String userId,
    String tenantId,
    String query,
    List retrievedChunks,
    boolean aclPass,
    String answer,
    double confidence,
    long latencyMs
) {}

多租户隔离方案

multi tenant kb 支持多个企业客户共用同一套 CorpAssist 服务。CorpAssist 推荐逻辑隔离（通过 tenant_id 字段过滤）而非物理隔离（独立实例）。

物理隔离（不推荐）
  • 每租户独立部署一套服务
  • 运维成本线性增长
  • 无法共享基础设施和模型
  • 版本升级需要逐个更新
逻辑隔离（推荐）
  • 所有租户共享同一套服务
  • 通过 tenant_id 字段隔离数据
  • 基础设施和模型共享
  • 版本升级一次生效
java 
// TenantAwareConfig.java — 多租户配置
@Configuration
public class TenantAwareConfig {

    @Bean
    @RequestScope  // 每个请求独立
    public TenantContext tenantContext() {
        return new TenantContext();
    }

    @Bean
    public QuestionAnswerAdvisor tenantAwareAdvisor(
            VectorStore vectorStore, TenantContext tenantContext) {
        return QuestionAnswerAdvisor.builder(vectorStore)
            .searchRequest(SearchRequest.builder()
                .filterExpression(
                    "tenant_id == '" + tenantContext.getTenantId() + "'")
                .topK(5)
                .build())
            .build();
    }
}

// 租户上下文
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class TenantContext {
    private String tenantId;
    private String userId;
    private String role;
    // getters/setters
}

逻辑隔离的代价是：一旦 FilterExpression 配置错误（如漏写了 tenant_id），可能导致租户 A 看到租户 B 的文档。CorpAssist 通过以下措施加固：

  • 默认拒绝：FilterExpression 中强制包含 tenant_id，不配置则拒绝查询。
  • 定期 ACL 审计：每周自动运行 acl_audit.py 检测跨租户泄露。
  • 请求溯源：审计日志记录每请求的 tenant_id，便于追溯。

常见问题（FAQ）

行级 ACL 和文档级权限有什么区别？
文档级权限是整个文档要么可见要么不可见。行级 ACL 可以精确到文档内的某个 chunk。例如一份《绩效评估标准》文档，普通经理只能看到评估流程（前 3 个 chunk），HR 总监可以看到全部（含薪酬关联部分）。CorpAssist 通过在索引时标注每个 chunk 的 compliance_level 实现。
审计日志保留多久？
CorpAssist 建议至少保留 6 个月（合规审计要求），推荐 12 个月。超过 12 个月的日志归档到冷存储（如 S3 Glacier）。日志量估算：1000 日活 × 10 次/天 × 1KB/条 ≈ 10MB/天，6 个月约 1.8GB。
多租户场景下，每个租户的 embedding 模型可以不同吗？
技术上可以（不同 tenant_id 使用不同 collection），但运维复杂度高。CorpAssist 推荐所有租户统一 embedding 模型，通过 tenant_id 字段隔离数据。如果租户有特殊需求（如金融行业需要私有化部署 embedding 模型），可以考虑物理隔离。
如何应对合规审计？
CorpAssist 准备了 compliance kb 审计检查表：1) 提供审计日志导出功能（支持 CSV/JSON）；2) 准备 ACL 策略文档和配置截图；3) 提供数据驻留说明（向量库和 ES 所在区域）；4) 提供数据删除流程（租户退租后清理所有数据）。

本章结论

  • 行级 ACL 在索引时标注、检索时执行：每个 chunk 标注 tenant_id/allowed_roles/compliance_level，FilterExpression 在检索层强制过滤。
  • 审计日志记录每次问答：request_id/user_id/query/chunks/answer/confidence/latency，7 个核心字段覆盖合规需求。
  • 逻辑隔离优先于物理隔离：通过 tenant_id 字段实现多租户数据隔离，运维成本远低于独立部署。
  • 安全加固是持续过程：默认拒绝策略 + 定期 ACL 审计 + 请求溯源，三道防线确保数据安全。

判断练习

  1. 安全团队要求「不允许用户看到任何敏感文档的片段」，即使文档是高置信度检索结果也不行。你怎么实现？

    参考答案

    在检索层加硬删除。 除了 FilterExpression 过滤外，在检索结果返回前再执行一道「mask 检查」：如果检索结果中包含 compliance_level=confidential 的 chunk，直接将其从结果集中删除（而非标记「无权限」）。这样 API 响应中完全不包含敏感信息。

  2. 一个小租户（10 个用户）和一个大租户（10000 个用户）共用逻辑隔离。大租户的索引变更会影响小租户吗？

    参考答案

    会影响延迟但不影响数据安全。 大租户的索引变更可能导致 Milvus 的 segment 合并，暂时影响同 collection 下所有租户的检索延迟（通常 <100ms 抖动）。数据安全方面不受影响——只要 tenant_id 过滤正确，小租户不会看到大租户的文档。如果抖动不可接受，考虑按租户规模分层：大租户用独立 collection。
