参考答案（120 秒版本）

text 
面试官你好。我以 CorpAssist 企业知识库为例来回答。

首先是场景确认：我们需要支持一家千人企业，
文档类型包括制度手册、产品规格和技术文档，
QPS 约 50，要求 P99 延迟 3 秒以内。

架构方面，我采用四层分层设计：
接入层用 API Gateway 做统一鉴权和限流；
处理层用 Python（Unstructured + LangChain）做离线索引；
检索层用 Milvus 向量库 + Elasticsearch BM25 做混合检索；
编排层用 Spring AI 的 QuestionAnswerAdvisor 做 RAG 编排。

关键设计是 Ingest-Query Split：Python 管索引、Spring 管问答，
通过共享的向量库和 ES 索引解耦。

数据流分两阶段：索引阶段走
「上传→Unstructured 解析→切分→向量化→写入 Milvus 和 ES」；
问答阶段走
「鉴权→混合检索→RRF 融合→BGE-Reranker→LLM 生成→引用溯源」。

选型上，企业知识库优先选 RAG 而非微调，
因为文档频繁更新（月均 10+ 份），RAG 只需重新索引。

效果方面，我们上线时 MRR 为 0.60，
经过 BM25 混合、Rerank、分块调优和 Query Rewrite 后达到 0.85，
P95 延迟控制在 610ms 以内，ROI 约 3.2x。

以上就是我的思路。谢谢。

第二问：分块策略

chunk question 面试题通常表述为「如何选择 chunk_size？」或「你的分块策略是什么？」。

面试题回答要点（CorpAssist 实践）
如何选 chunk_size？ 制度文档 512-768，tables 按行保留表头，代码按函数独立。通过 Recall@5 实验确定最优值。
Overlap 多少合适？ chunk_size 的 15-25%（128-200 字符）。overlap 不能替代语义切分，只是兜底。
表格和代码怎么处理？ 自定义切分器 CorporateChunker：检测表格行（含`|`）按行切分保留表头；检测代码块（```）按函数切分。
Metadata 存哪些字段？ doc_id、version、chunk_index、compliance_level、allowed_roles、tenant_id、doc_category、index_version、source_file 共 9 个字段。
python 
# 面试口述版分块代码
def get_chunk_strategy(doc_type: str) -> dict:
    strategies = {
        "policy": {"chunk_size": 512, "overlap": 128, "separator": "paragraph"},
        "table": {"chunk_size": "per_row", "keep_header": True},
        "code": {"chunk_size": "per_function", "language": "auto"},
        "report": {"chunk_size": 768, "overlap": 150, "separator": "semantic"},
    }
    return strategies.get(doc_type, strategies["policy"])

检索失败排查四步法（面试版）
flowchart TD
    Q[用户查不到答案] --> C1{Step 1: 文档\n在索引中吗?}

四步排查法：索引存在性 → 向量命中 → ACL 过滤 → LLM 生成。每步一个 yes/no 判断。

python 
# 面试口述版排查脚本
def diagnose_retrieval_miss(query, expected_doc_id, user_role):
    steps = [
        ("文档是否存在", check_doc_exists(expected_doc_id)),
        ("向量检索命中", vector_search_hits(query, expected_doc_id)),
        ("BM25 命中", bm25_search_hits(query, expected_doc_id)),
        ("ACL 通过", acl_check(expected_doc_id, user_role)),
        ("LLM 生成质量", llm_generation_check(query, expected_doc_id)),
    ]
    for name, passed in steps:
        status = "PASS" if passed else "FAIL"
        print(f"[{status}] {name}")
        if not passed:
            print(f"  -> 建议: {get_remedy(name)}")
            break

第四问：RAG vs 微调 & 第五问：评估指标

rag ft question 和 eval metrics qa 经常一起出现。

RAG vs 微调：面试回答要点

维度RAG微调
知识更新 秒级——重新索引即可 天/周级——需要重新训练
知识来源 外部文档，可溯源 模型参数，不可溯源
适合场景 事实问答、制度查询 行为对齐、风格统一
成本（初期） 低 — 无需训练 高 — 需要标注 + 训练
CorpAssist 选择 ✅ 默认方案 仅当需要统一话术风格时

评估指标：面试回答要点

text 
CorpAssist 用三个指标衡量检索质量：

MRR@K（Mean Reciprocal Rank）：
  第一个正确答案在检索结果中的平均倒数排名。
  例如正确答案排第 2 → 贡献 1/2 = 0.5。
  目标：≥ 0.80。衡量「用户多快看到正确答案」。

Recall@K：
  前 K 个结果中包含正确答案的比例。
  目标：≥ 0.85。衡量「正确答案是否在候选集中」。

NDCG@K（Normalized Discounted Cumulative Gain）：
  考虑排序位置的折损增益。
  目标：≥ 0.75。衡量「排序质量」。

此外我们还跟踪：
- 首 Token 延迟（目标 < 800ms）
- 服务可用性（≥ 99.5%）
- 用户采纳率（日活员工 / 全员数 ≥ 30%）

答辩追问应对

答辩中评委可能会追问以下高频问题，提前准备好了就能从容应对：

追问 1：你提到 Ingest-Query Split，Python 和 Spring 两边如何保证 metadata 一致？
回答： 通过共享的元数据 Schema 定义。我们在项目初期就定义了 corpassist_metadata_schema.json（9 个字段的类型和约束），Python 端索引时按 Schema 写入，Spring 端消费时按 Schema 读取。两端共用同一个 Milvus collection，字段名和类型必须一致。版本变更时先更新 Schema 定义，再更新两端代码。
追问 2：如果数据量从 1000 份文档增长到 10 万份，你的架构怎么扩展？
回答： 四条扩展路径：1) 检索层——Milvus 增加分片数（shard），ES 增加节点；2) 编排层——Spring AI 服务水平扩展实例数，QPS 不够时加实例；3) 索引层——Celery Worker 从 5 个扩展到 20 个并行索引；4) 存储——使用对象存储（S3/MinIO）存储原始文档，Milvus 只存向量。10 万份文档预估检索延迟增加 < 20%，总成本增加约 3-4x。
追问 3：你的系统如何支持多轮对话？
回答： CorpAssist 当前版本通过 Chat History 实现：每轮对话将历史消息拼入 prompt。Spring AI 的 ChatMemory Advisor 自动管理对话上下文。改进方向：使用 Contextual Compression 避免上下文过长，以及通过 Query Rewriting 将「它」等指代替换为具体实体。
追问 4：如果 OpenAI API 不可用，你的系统怎么办？
回答： CorpAssist 设计时就考虑了模型可替换性。Spring AI 抽象了 ChatModel 接口，可以无缝切换为本地部署的模型（如 Qwen2 或 Llama）。我们的生产实践是：默认用 gpt-4o-mini，预配通义千问作为降级方案。当 API 超时或返回错误时自动降级到本地模型，延迟增加约 500ms 但系统不中断。

判断练习

  1. 面试官问「你的知识库能保证 100% 准确吗？」你怎么回答？

    参考答案

    回答： LLM 的生成本质上是概率性的，我们做不到 100% 准确。但我们有三道防线将准确率推到 90%+：1) 检索层——只有高相关度（score>0.5）的文档才进入生成；2) 引用层——每个答案需要至少 80% 的句子有引用来源；3) 审核层——低置信度答案进入人工复核。综合下来，我们的可接受准确率目标是 95%，关键的合规场景要求 99%+。

  2. 全部 13 章学完了，你能用 1 分钟总结 CorpAssist 全貌吗？

    参考答案

    60 秒总结： CorpAssist 是一个企业级 RAG 知识库系统，采用 Python 离线索引 + Spring AI 在线问答的双栈架构。索引管线用 Unstructured 解析多格式文档，CorporateChunker 按类型智能切分，写入 Milvus + ES 混合索引。问答服务用 BM25+向量混合检索、RRF 融合、BGE-Reranker 精排、LLM 生成带引用溯源的答案。通过行级 ACL 和审计日志保障安全，评测飞轮（MRR/Recall/NDCG）驱动持续优化。
