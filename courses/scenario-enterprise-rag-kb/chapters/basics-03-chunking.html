为什么要学本章

分块（chunking）决定了检索质量的上限。再好的检索策略，如果文档切分不合理——把关键技术参数切成两半、把表格拆成碎片——召回率也无法提升。本章通过 CorpAssist 的制度文档、产品表格、代码示例三类典型文档，讲解 fixed chunk、semantic chunk 等策略的选型与参数调优。

学完你能
  • 区分固定分块与语义分块的适用场景并选择合适的 split strategy
  • 为表格和代码块设计特殊的分块策略（table chunk / code block chunk）
  • 通过 chunk experiment 设计实验对比不同参数的召回效果
  • 设置合理的 overlap size 并管理 chunk metadata
本章先记住 3 件事
  • 没有通用最优参数：chunk_size 取决于文档类型，制度类 512-1024，代码类按函数。
  • Overlap 不能替代语义切分：overlap 只是兜底，语义边界切分才是核心。
  • 表格切分需要特殊处理：表格行不能被拆到两个 chunk 里，否则检索语义断裂。

固定分块 vs 语义分块

fixed chunk 按固定字符数切分，速度快但可能切断语义段落。

CorpAssist · 制度文档切分 CorpAssist 需要索引一份《员工手册》PDF，包含「休假政策」「加班规定」「社保公积金」三节。固定分块可能在「休假政策」内容的中间切断，导致检索时只拿到半个规定。

两种策略的代码实现

python 
# chunk_strategies.py — 固定分块 vs 语义分块对比
from langchain.text_splitter import (
    RecursiveCharacterTextSplitter,
    CharacterTextSplitter,
)
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

# 固定分块：按字符数硬切
fixed_splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=128,
    separators=["\n\n", "\n", "。", "！", "？", " ", ""],
)

# 语义分块：基于 embedding 相似度
semantic_splitter = SemanticChunker(
    OpenAIEmbeddings(model="text-embedding-3-small"),
    breakpoint_threshold_type="percentile",  # 按百分位断点
    breakpoint_threshold_amount=95,  # 相似度低于 95% 分位则切分
)

# 对比测试
text = """休假政策：员工每年享有 15 天带薪年假。
年假可按半天为单位申请，需提前 3 个工作日在系统提交。
未休完的年假可顺延至次年 3 月底。超过部分作废。
加班规定：工作日加班按 1.5 倍工资计算。
周末加班可按 2 倍工资或调休处理。
法定节假日加班按 3 倍工资计算。"""

fixed_chunks = fixed_splitter.split_text(text)
semantic_chunks = semantic_splitter.split_text(text)

print(f"固定分块: {len(fixed_chunks)} 块")
for i, c in enumerate(fixed_chunks):
    print(f"  [{i}] {len(c)} chars: {c[:40]}...")

print(f"\n语义分块: {len(semantic_chunks)} 块")
for i, c in enumerate(semantic_chunks):
    print(f"  [{i}] {len(c)} chars: {c[:40]}...")

import org.springframework.ai.document.Document;
import org.springframework.ai.transformer.splitter.TokenTextSplitter;
import java.util.List;

public class ChunkingService {
    private final TokenTextSplitter splitter;

    public ChunkingService() {
        this.splitter = TokenTextSplitter.builder()
            .chunkSize(512)
            .chunkOverlap(128)
            .build();
    }

    public List split(String content) {
        Document doc = new Document(content);
        return splitter.apply(List.of(doc));
    }
}

表格与代码块的特殊处理

table chunk 和 code block chunk 需要特殊的分块策略。CorpAssist 在处理产品规格表和 API 文档时发现，默认的递归字符切分器会破坏表格和代码的结构完整性。

表格行不能跨 chunk 切断，每行需要保留表头信息以确保语义完整。

自定义表格/代码块切分器

python 
# custom_chunker.py — 表格和代码块的特殊切分
import re
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.schema import Document

class CorporateChunker:
    """CorpAssist 定制切分器：识别表格和代码块做特殊处理"""

    def __init__(self, chunk_size=512, chunk_overlap=128):
        self.default_splitter = RecursiveCharacterTextSplitter(
            chunk_size=chunk_size,
            chunk_overlap=chunk_overlap,
            separators=["\n\n---\n\n", "\n\n", "\n", "。", " ", ""],
        )
        self.chunk_size = chunk_size

    def split_document(self, doc: Document) -> list[Document]:
        content = doc.page_content
        chunks = []

        # 1. 提取代码块（``` 包裹的片段）
        code_blocks = re.findall(r'```[\s\S]*?```', content)
        remaining = re.sub(r'```[\s\S]*?```', '', content)

        # 代码块按函数切分
        for block in code_blocks:
            funcs = re.split(r'\n(?=def |class |public )', block)
            for func in funcs:
                if func.strip():
                    m = Document(
                        page_content=func.strip(),
                        metadata={**doc.metadata, "type": "code_block"}
                    )
                    chunks.append(m)

        # 2. 提取表格（行中有 | 符号）
        lines = remaining.split("\n")
        table_buffer = []
        non_table_lines = []

        for line in lines:
            if "|" in line and "--" not in line:
                table_buffer.append(line)
            else:
                if table_buffer:
                    # 每行为一个 chunk，保留表头
                    header = table_buffer[0]
                    for row in table_buffer[1:]:
                        chunk_text = f"{header}\n{row}"
                        m = Document(
                            page_content=chunk_text,
                            metadata={**doc.metadata, "type": "table_row"}
                        )
                        chunks.append(m)
                    table_buffer = []
                non_table_lines.append(line)

        # 3. 剩余文本用默认切分器
        default_text = "\n".join(non_table_lines)
        if default_text.strip():
            default_chunks = self.default_splitter.split_documents(
                [Document(page_content=default_text, metadata=doc.metadata)]
            )
            chunks.extend(default_chunks)

        return chunks

# 使用示例
chunker = CorporateChunker(chunk_size=512, chunk_overlap=128)
result = chunker.split_document(Document(
    page_content="## 产品规格表\n| 型号 | 价格 |\n| A100 | ¥99 |\n| A200 | ¥199 |\n\n```python\ndef hello():\n    print('world')\n```",
    metadata={"doc_id": "PROD-001"}
))
for c in result:
    print(f"[{c.metadata['type']}] {c.page_content[:60]}...")

Spring AI 中通过 DocumentTransformer 实现

java 
// Spring AI DocumentTransformer 实现自定义分块
@Component
public class TableAwareTransformer implements DocumentTransformer {
    @Override
    public List transform(List documents) {
        List result = new ArrayList<>();
        for (Document doc : documents) {
            String text = doc.getContent();
            // 正则检测表格行（含 | 的行）
            Pattern pattern = Pattern.compile("(\\|[^\\n]+\\|)");
            Matcher matcher = pattern.matcher(text);
            // 表格行作为独立 document 保留
            while (matcher.find()) {
                Document tableDoc = new Document(matcher.group(1));
                tableDoc.getMetadata().put("type", "table_row");
                tableDoc.getMetadata().putAll(doc.getMetadata());
                result.add(tableDoc);
            }
        }
        return result;
    }
}

参数实验代码

python 
# chunk_experiment.py — 分块参数对比实验
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
from typing import List, Dict
import numpy as np

def evaluate_chunk_params(
    docs: List[str],
    test_queries: List[Dict],  # [{"q": "...", "expected_chunk_ids": [...]}]
    chunk_size: int,
    chunk_overlap: int,
) -> dict:
    """评测一组分块参数的检索效果"""
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
    )
    chunks = splitter.create_documents(docs)

    # 建索引
    embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
    vector_store = Chroma.from_documents(chunks, embeddings)
    retriever = vector_store.as_retriever(search_kwargs={"k": 5})

    # 评测
    recall_at_5 = 0
    for tq in test_queries:
        results = retriever.invoke(tq["q"])
        retrieved_ids = {r.metadata.get("chunk_id") for r in results}
        expected = set(tq["expected_chunk_ids"])
        if expected & retrieved_ids:
            recall_at_5 += 1 / len(test_queries)

    return {"recall@5": round(recall_at_5, 3)}

# 参数扫描
param_grid = [
    (256, 64), (512, 128), (768, 128),
    (1024, 200), (1536, 256),
]
for size, overlap in param_grid:
    result = evaluate_chunk_params(docs, test_queries, size, overlap)
    print(f"chunk_size={size}, overlap={overlap}: recall@5={result['recall@5']}")

常见问题（FAQ）

chunk_size 越大越好吗？
不是。chunk_size 越大，每个 chunk 包含的上下文越多，但检索精度下降——用户问「年假几天」时，一个 2000 字的 chunk 包含 5 个政策条款，LLM 需要从中定位正确答案，容易出错。CorpAssist 经验：制度文档 512-768，技术文档 768-1024。
对于手写扫描件，分块策略有什么不同？
扫描件先过 OCR，OCR 输出通常带有段落标记（如每页底部有页码）。建议在分块前先清洗 OCR 输出：去掉页码、页眉页脚，然后按段落结构做语义分块。如果 OCR 质量差（<80% 准确率），chunk_size 宜减小至 256-384，降低噪音密度。详见 生产坑：数据、权限与索引 的脏 PDF 专题。
分块后 metadata 应该存哪些字段？
CorpAssist 标准元数据：doc_id, version, chunk_index, type(paragraph/table/code), compliance_level, allowed_roles, tenant_id, source_file。这些字段在检索层的 ACL 过滤和引用溯源中都会用到。详见 chunk metadata。
已上线的知识库如何变更分块参数？
分块参数变更需要重新索引（re-index）。CorpAssist 采用「双索引」策略：新参数构建新索引 v2，旧索引 v1 继续服务，验证新索引指标达标后切换。切换过程通过配置中心的开关控制，无需停服。

本章结论

  • 选策略先于调参数：制度文档用递归切分，表格按行保留表头，代码按函数独立。
  • Overlap 是兜底不是银弹：15-25% 的 overlap 可以缓解边界切断问题，但不能替代语义边界的正确切分。
  • 实验驱动参数决策：用测试集做 Recall@5 对比，不同文档类型单独验证。
  • Metadata 是检索层的基础设施：doc_id、version、type、ACL 等元数据必须在索引阶段完整标注。

判断练习

  1. chunk_size=512, overlap=128 时，一个 600 字的产品介绍段落会被切分吗？

    参考答案

    不会切分。 600 字小于 chunk_size × 2 的触发条件，RecursiveCharacterTextSplitter 会尝试合并到不超过 chunk_size。但如果段落内有明确的换行分隔，可能会在换行处切分。

  2. 生产环境中，团队反馈「某个政策条款总是查不到」，排查步骤是什么？

    参考答案

    步骤： 1) 检查该条款是否被成功索引（确认文档解析正确）；2) 查看该条款在哪个 chunk 中、chunk 内容是否完整；3) 查该 chunk 的 embedding 向量与问题的相似度；4) 检查 ACL 元数据是否阻止了该用户的访问。
