为什么要学本章

企业办公自动化是 LangGraph 工作流 最直接的价值落地场景。CorpAssist 团队调研 50+ 企业后发现，文档处理、数据提取、报告生成三类场景覆盖 70% 的重复性人工操作。本章从真实痛点出发，帮你建立「哪些该自动化、哪些不该、投入产出比多少」的判断框架。

学完你能
  • 用结构化方法识别办公流程中可自动化的环节（至少 3 条判断标准）
  • 画出 CorpAssist 发票处理 Agent 的决策流程图，并说明每个节点的回退策略
  • 为自动化场景撰写 ROI 测算表：人力成本 vs API 调用成本 vs 准确率收益
  • 识别数据合规红线（PII 脱敏、跨境传输、审计留存）并设计初步防护方案
本章先记住 3 件事
  • 高频≠高价值：每天花 2 小时的低风险操作（如格式转换）自动化收益远高于每月 1 次的高风险审批。
  • 合规是上限：Agent 代操作员工邮箱、ERP 系统前，必须先过法务与安全评审。
  • ROI 要算总账：LLM API 调用费 + 向量存储 + 人工审计兜底，减去节省的人力工时，才是真实收益。

发票 Agent 按金额分路由：小额自动审批，大额引入 人工审批 节点。所有操作写入审计日志。

不推荐：全自动无人工兜底
  • Agent 直接操作 ERP 写入，无审批节点
  • 异常发票直接跳过，无人工复核路径
  • 无审计日志，出问题无法追溯
  • OCR 识别失败时静默跳过，造成数据缺失
推荐：分级审批 + 完整审计
  • 按金额/供应商维度设置自动/人工路由规则
  • 置信度低于 0.85 的字段自动标记为待人工确认
  • 每步操作写入审计日志（操作人、时间、输入输出摘要）
  • 提供 Web 界面供财务人员审核异常条目
先想 10 秒：如果发票 Agent 误将 50 万的发票判断为「金额 ≤ 5 万」自动入账，你会如何从架构上防止？

在 LLM 提取阶段之后、金额判断之前，加一道规则校验：如果 LLM 返回的金额字段与 OCR/PDF 原始解析的数值不一致，或关键字含「万」「W」「十万」等，自动降级到人工审核。规则层的优先级高于模型输出。

合规边界：Agent 能做什么、不能做什么

办公自动化的最大风险不是技术失效，而是合规越界。CorpAssist 在多家金融客户部署中总结出三条红线：

红线说明架构应对
PII 数据不出境 员工身份证号、银行账号等个人敏感信息不能发送到境外 LLM API 本地部署小模型（如 Qwen2.5-7B）处理含 PII 的字段；使用 工具沙箱 隔离敏感数据流
操作不绕过审批 Agent 不能直接执行「转账」「删除」「修改权限」等高风险操作 人工审批节点 作为强制 Gateway，所有写操作必须经人确认
审计日志不可篡改 Agent 的每一步决策和工具调用必须可追溯、不可删除 LangGraph 检查点 + 日志 append-only 存储（如 ClickHouse）

合规检测代码示例

python 
# compliance.py — CorpAssist 合规检查器
from pydantic import BaseModel, Field
from typing import Optional
import re

class ComplianceChecker:
    """合规检查：PII 检测 + 操作白名单 + 审计日志写入"""

    SENSITIVE_PATTERNS = [
        r'\d{18}(?:\d|[Xx])',       # 身份证号
        r'\d{16,19}',               # 银行卡号
        r'1[3-9]\d{9}',             # 手机号
    ]

    WRITE_OPERATIONS = {
        "erp_write", "payment_execute", "data_delete",
        "permission_change", "email_send",
    }

    BANNED_OPERATIONS = {
        "payment_execute", "data_delete", "permission_change",
    }

    @classmethod
    def check_pii(cls, text: str) -> list[dict]:
        """检测文本中的敏感信息"""
        found = []
        for pattern in cls.SENSITIVE_PATTERNS:
            matches = re.finditer(pattern, text)
            for m in matches:
                found.append({
                    "type": "PII",
                    "pattern": pattern,
                    "position": m.span(),
                    "preview": text[max(0, m.start()-10):m.end()+10],
                })
        return found

    @classmethod
    def check_operation(cls, tool_name: str) -> dict:
        """检查工具调用是否合规"""
        if tool_name in cls.BANNED_OPERATIONS:
            return {"allowed": False, "reason": f"{tool_name} 是禁止操作，需人工审批"}
        if tool_name in cls.WRITE_OPERATIONS:
            return {"allowed": True, "requires_approval": True}
        return {"allowed": True, "requires_approval": False}

    @classmethod
    def mask_pii(cls, text: str) -> str:
        """脱敏处理"""
        for pattern in cls.SENSITIVE_PATTERNS:
            text = re.sub(pattern, lambda m: m.group()[:4] + "****" + m.group()[-4:], text)
        return text

# 使用示例
if __name__ == "__main__":
    text = "请将款项转入6228480012345678，用户身份证号 110101199001011234"
    pii = ComplianceChecker.check_pii(text)
    print(f"检测到 {len(pii)} 处敏感信息")
    masked = ComplianceChecker.mask_pii(text)
    print(f"脱敏后: {masked}")

    result = ComplianceChecker.check_operation("payment_execute")
    print(f"操作检查: {result}")

ROI 测算：自动化投入与收益量化

部署 Agent 自动化前，必须量化 ROI。CorpAssist 使用以下模型进行测算：

ROI = (节省人力成本 + 错误减少收益) - (LLM API费用 + 向量存储费 + 开发分摊 + 人工审计兜底)

发票处理 Agent ROI 测算实例

项目月均数值说明
人工处理时长 200 份 × 5 分钟 = 16.7 人·天 月均 200 份发票，5 分钟/份
Agent 处理时长 200 份 × 30 秒 = 1.7 人·天 其中 15% 需人工复核（每份 2 分钟）
人力节省 15 人·天 ≈ ¥30,000/月 按日均 ¥2,000 计算
LLM API 费用 200 × 1.5K tokens × ¥0.015 ≈ ¥4,500/月 gpt-4o-mini，含输入输出
向量存储 ¥500/月 Chroma 或 Pinecone
错误减少收益 人工 8% 错误率 → Agent 2% 错误率 每单错误平均损失 ¥200
月净收益 ¥27,000 3 个月回本
ROI 陷阱案例 某团队部署了「全自动合同审核 Agent」，但仅 LLM API 月费 ¥12,000，且法务仍需逐份复核（Agent 置信度仅 70%），实际人力节省为 0。教训：先验证准确率基线，再算 ROI。如果 Agent 达不到 90%+ 的准确率，人工兜底成本会吃掉所有收益。
不推荐：忽略人工兜底成本
  • 只算 LLM API 费用 vs 节省工时，忽略人工复核成本
  • 假设 Agent 准确率 100%，不做兜底预算
  • 一次性开发成本分 1 个月摊销
推荐：全口径 ROI 模型
  • LLM API + 向量存储 + 推理基础设施 + 开发摊销（12 个月）
  • 人工兜底成本 = 处理量 × (1 - 准确率) × 人工处理单价
  • 运维成本（监控、告警、模型更新）预留 20% 余量

常见问题（FAQ）

Agent 自动化和传统的 RPA 有什么区别？
RPA（机器人流程自动化）基于规则和屏幕抓取，适合固定步骤、结构化操作。Agent 自动化基于 LLM 推理，能处理非结构化输入（发票图片、自由文本）和动态决策（异常路由、多步规划）。两者互补：RPA 做确定性操作，Agent 做需要判断的环节。CorpAssist 常将 Agent 作为 RPA 的"大脑"来驱动 RPA 执行。
Agent 出错时谁担责？合规上怎么设计？
企业部署 Agent 必须建立人机协作的责任边界：高风险操作（付款、删除数据）必须经人工审批，Agent 不能独立执行。低风险操作（信息提取、格式转换）Agent 可独立执行但需审计日志。推荐在 SLA 中明确「Agent 辅助决策，人类负最终责任」。
部署 Agent 需要企业 IT 做哪些前置准备？
三件事：1) API 网关暴露内部系统接口（ERP、OA、CRM），支持 OAuth2 或 API Key 鉴权；2) 确定数据合规边界——哪些字段可以发给云端 LLM，哪些必须本地处理；3) 准备测试数据集——至少 100 条真实脱敏数据用于验证 Agent 准确率。
小公司没有 ERP API，能做 Agent 自动化吗？
可以。CorpAssist 的 多工具协作 模式支持通过邮件、Excel、Web 界面作为输入/输出。例如，Agent 从邮件附件提取数据 → 写入 Google Sheets → 发送摘要邮件。不需要 ERP 也能跑通端到端流程。

本章结论

  • 办公自动化四大场景：文档处理、数据搬运、审批催办、报告生成，覆盖 70% 的企业重复操作。
  • 合规三条红线：PII 不出境、操作不绕过审批、审计日志不可篡改。每条红线都有对应的架构应对方案。
  • ROI 要算全口径：LLM API + 基础设施 + 人工兜底 + 开发摊销 vs 人力节省 + 错误减少收益。Agent 准确率不足 90% 时 ROI 可能为负。
  • 发票 Agent 是标杆场景：从收到发票到入账通知的全流程展示了 Agent 决策、条件路由、人工审批的完整模式。
  • 本课路线：从基础认知到 LangGraph 实战再到生产部署，逐步深入 Agent 自动化全栈。

判断练习

  1. 某团队在部署 Agent 时只计算了 LLM API 费用（¥3,000/月）和节省的人力成本（¥15,000/月），得出 ROI 极高的结论。上线后发现人工兜底成本 ¥8,000/月，开发摊销 ¥4,000/月。请问实际 ROI 修正值是多少？

    参考答案

    实际月净收益 = 15,000 - 3,000 - 8,000 - 4,000 = 0。本案例说明忽略人工兜底和开发摊销会严重高估 ROI。Agent 准确率不足 93% 时，人工复核成本会吃掉大部分收益。

  2. 合规检查中，以下哪项操作 Agent 可以独立执行？① 查询员工考勤记录 ② 将 ERP 中订单状态改为「已取消」 ③ 发送周报汇总邮件

    参考答案

    ① 和 ③ 是查询和通知类操作，Agent 可独立执行。② 涉及数据修改，需要人工审批节点授权。写操作一律需要审批，读操作可独立执行但需审计日志。
