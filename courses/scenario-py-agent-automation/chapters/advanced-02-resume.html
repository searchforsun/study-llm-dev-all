本章先记住 3 件事
  • 基线决定成败：没基线就没对照，优化效果无法证明。上线前先跑 100 条测试数据。
  • 成本要透明：Agent 的 Token 消耗直接等于钱。每步多少钱、每任务多少钱必须能算清。
  • SLA 要事前定：准确率 95%、完成率 90%、P99 延迟 5s——上线前和业务方对齐目标。

效果指标体系与基线

CorpAssist 使用以下指标体系衡量 Agent 效果：

维度指标计算方式目标值
准确性 任务完成率 成功完成任务数 / 总请求数 > 90%
效率 平均步数 工具调用总步数 / 任务数 < 8 步
成本 每任务 Token 输入+输出 Token 总数 / 任务数 < 4000 tokens
体验 首 Token 延迟 (TTFT) 用户发送请求到看到第一个 Token 的时间 < 1s
可靠性 P99 完成时间 99% 的任务在多长时间内完成 < 30s

效果基线建立

python 
# baseline_eval.py — 效果基线评估
import json
import time
from typing import List, Dict
from dataclasses import dataclass, asdict

@dataclass
class EvalResult:
    task_id: str
    question: str
    expected_answer: str
    actual_answer: str
    steps: int
    tokens_used: int
    latency_ms: int
    correct: bool
    error: str = ""

class BaselineEval:
    """评估器：建立效果基线"""

    def __init__(self, agent_executor):
        self.agent = agent_executor
        self.results: List[EvalResult] = []

    async def run_eval(self, test_cases: List[Dict]) -> dict:
        """运行评估"""
        for case in test_cases:
            start = time.time()
            try:
                result = await self.agent.ainvoke({"input": case["question"]})
                actual = result["output"]
                steps = len(result.get("intermediate_steps", []))
                correct = self._check_answer(actual, case["expected"])
            except Exception as e:
                actual = ""
                steps = 0
                correct = False

            elapsed = int((time.time() - start) * 1000)
            self.results.append(EvalResult(
                task_id=case["id"],
                question=case["question"],
                expected_answer=case["expected"],
                actual_answer=actual,
                steps=steps,
                tokens_used=0,  # 实际从 LangSmith 获取
                latency_ms=elapsed,
                correct=correct,
                error=str(e) if 'e' in dir() else "",
            ))
        return self.summarize()

    def _check_answer(self, actual: str, expected: str) -> bool:
        """检查答案是否正确（使用 LLM 评估）"""
        from langchain_openai import ChatOpenAI
        llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
        prompt = f"""判断实际答案是否与预期答案一致。
预期：{expected}
实际：{actual}
只需返回 YES 或 NO。"""
        response = llm.invoke(prompt)
        return "YES" in response.content.upper()

    def summarize(self) -> dict:
        """生成摘要报告"""
        total = len(self.results)
        correct = sum(1 for r in self.results if r.correct)
        avg_steps = sum(r.steps for r in self.results) / total if total else 0
        return {
            "total": total,
            "accuracy": round(correct / total * 100, 1) if total else 0,
            "avg_steps": round(avg_steps, 1),
            "avg_latency_ms": int(sum(r.latency_ms for r in self.results) / total),
        }

    def report(self) -> str:
        """生成可阅读的 Markdown 报告"""
        summary = self.summarize()
        report = f"""# Agent 效果评估报告

## 总览
- 测试用例数: {summary['total']}
- 准确率: {summary['accuracy']}%
- 平均步数: {summary['avg_steps']}
- 平均延迟: {summary['avg_latency_ms']}ms

## 失败案例分析
"""
        for r in self.results:
            if not r.correct:
                report += f"\n### {r.task_id}: {r.question}\n"
                report += f"- 预期: {r.expected_answer[:100]}...\n"
                report += f"- 实际: {r.actual_answer[:100]}...\n"
                report += f"- 错误: {r.error}\n"
        return report
不推荐：无基线优化
  • 上线前没跑评估，不知道 Agent 实际准确率
  • 优化靠感觉改 Prompt，无效果验证
  • 出问题后无法量化影响范围
  • 向领导汇报只能说「效果不错」
推荐：基线驱动优化
  • 上线前跑 200 条测试用例建立基线
  • 每次优化后重新评估，对比基线看提升
  • 准确率从 60% → 85% 的叙事有数据支撑
  • 汇报时说「准确率提升 25 个百分点，步数降低 40%」

成本计算模型

Agent 的运营成本 = Token 成本 + 向量存储 + 基础设施 + 人工兜底分摊

python 
class CostCalculator:
    """Agent 运营成本计算"""

    INPUT_TOKEN_PRICE = 0.15 / 1_000_000   # gpt-4o-mini 输入
    OUTPUT_TOKEN_PRICE = 0.60 / 1_000_000  # gpt-4o-mini 输出

    def estimate_monthly(self, tasks_per_month: int, avg_input_tokens: int, avg_output_tokens: int) -> dict:
        token_cost = tasks_per_month * (avg_input_tokens * self.INPUT_TOKEN_PRICE + avg_output_tokens * self.OUTPUT_TOKEN_PRICE)
        vector_store = 500  # Chroma 月费
        infrastructure = 2000  # 服务器
        human_oversight = tasks_per_month * 0.1 * 2 * 0.5  # 10% 需人工复核，每人 2 分钟，每分钟 0.5 元
        return {
            "token_cost": round(token_cost, 2),
            "vector_store": vector_store,
            "infrastructure": infrastructure,
            "human_oversight": round(human_oversight, 2),
            "total_monthly": round(token_cost + vector_store + infrastructure + human_oversight, 2),
        }

常见问题（FAQ）

Agent 效果评估用什么做 ground truth？
三种来源：1) 人工标注——让业务方标注 200 条数据作为标准答案；2) 历史数据——已有的正确的处理结果；3) LLM 自评——用更强模型评估弱模型的结果。CorpAssist 使用人工标注 + LLM 自评双重校验，不一致时人工裁定。
Agent 的准确率和人工的准确率怎么公平对比？
用同一组测试集（至少 200 条），分别让人工和 Agent 处理，对比结果。注意：人工的「准确率」不是 100%——人工也会犯错（CorpAssist 统计人工发票处理的错误率约 8%）。Agent 的目标是超越人工基线，而不是追求 100%。
SLA 定 95% 还是 99%？
取决于成本。从 95% 提升到 99% 可能需要 2x 的成本（更贵的模型、更多的验证步骤、更多的人工复核）。建议先定 90%，稳定运行 1 个月后再评估是否提升到 95%。不要一开始就追求 99%——成本可能无法承受。

本章结论

  • 指标五维度：准确率、步数、Token、TTFT、P99。上线前先建立基线。
  • 基线驱动优化：每次修改后重新评估，对比基线数据看效果。没用基线优化的都是玄学。
  • 成本透明：Token + 向量存储 + 基础设施 + 人工兜底，每月运营成本可精确计算。
  • SLA 事前约定：准确率、延迟、可用性目标在上线前和业务方对齐。
  • 效果叙事：从 X 到 Y 的数据叙事比「效果不错」有说服力 100 倍。
