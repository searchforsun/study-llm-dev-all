第一问：架构设计

面试题：请画出你们 Agent 系统的整体架构图，并说明为什么这么设计。

面试答题：Agent 系统架构
flowchart TD
    subgraph 接入层
        API[FastAPI Gateway]
        Auth[鉴权/限流]
    end
    subgraph 编排层
        LG[LangGraph 工作流]
        Checkpoint[检查点持久化]
        Human[人工审批节点]
    end
    subgraph 能力层
        Agent[ReAct Agent]
        Tools[工具注册表]
        Memory[三层记忆]
    end
    subgraph 基础层
        LLM[LLM 网关]
        Vector[向量库]
        Monitor[LangSmith]
    end

    User --> API --> Auth --> LG
    LG --> Agent --> Tools
    Agent --> Memory
    LG --> Checkpoint
    LG --> Human
    Agent --> LLM --> Vector
    Agent --> Monitor

架构回答四层法：接入层→编排层→能力层→基础层。每层说明选型理由。

回答框架：总-分-总

总：「我们采用四层架构：接入层负责鉴权限流，编排层用 LangGraph 管理工作流状态，能力层是 ReAct Agent 和工具集，基础层提供 LLM 和向量能力。」

分：「接入层用 FastAPI 因为异步性能好...编排层选 LangGraph 因为需要检查点持久化支持人工审批...能力层用 ReAct 模式因为步数较少（<10）...基础层用 gpt-4o-mini 平衡成本和质量...」

总：「这套架构支撑了发票处理、采购审批等 5 个 Agent 场景，月均处理 2000+ 任务，准确率 97%。」

不推荐回答
  • 「我们用了 LangChain」——太笼统
  • 「就是调 API」——暴露缺乏深度
  • 「和网上教程一样」——无差异思考
推荐回答
  • 「四层架构，每层选型有对比分析」
  • 「为什么选 LangGraph 而不是 ReAct」
  • 「踩过的坑和优化经验」

第二问：分块策略

面试题：企业文档做 RAG 时，你怎么选择分块策略？chunk_size 和 overlap 怎么调？

文档类型推荐分块chunk_sizeoverlap理由
政策文档（PDF） 语义分块 1024 200 按自然段落切分，保留完整语义
代码仓库 AST 分块 按函数 0 按函数/类级别切分，保留语法结构
表格数据 固定分块 512 50 表格行间无语义依赖，固定大小即可
对话记录 时间窗口 2048 256 按对话轮次切分，保留上下文

回答框架：「分块没有银弹。我们针对不同文档类型选择策略：政策文档用语义分块（chunk_size=1024, overlap=200），保证段落完整性；代码用 AST 分块，按函数粒度索引。调优方法是跑召回率实验——对比不同参数在 200 条测试集上的 Recall@5，选最优组合。」

第三问：检索不准

面试题：用户反馈 Agent 检索结果不相关，你怎么排查和解决？

检索问题排查决策树
flowchart TD
    Problem[检索结果不相关] --> Check1{检查召回结果}
    Check1 -->|召回结果相关但 LLM 没用| Fix1[优化 Prompt\n加入「严格基于上下文」]
    Check1 -->|召回结果本身不相关| Check2{检查 Embedding}
    Check2 -->|Embedding 区分度低| Fix2[换模型\ntext-embedding-3-large]
    Check2 -->|Embedding 区分度好| Check3{检查分块}
    Check3 -->|chunk 太小| Fix3[增大 chunk_size]
    Check3 -->|chunk 切断了语义| Fix4[语义分块]
    Check3 -->|分块没问题| Check4{尝试 Rerank}
    Check4 --> Fix5[接入 BGE-Reranker\n提升 top-5 精度]

排查步骤：先确认召回结果，再检查 Embedding 质量，最后优化分块和 Rerank。

回答框架：「分四步排查：1) 先看检索结果是否相关——不相关则问题在检索侧；2) 检索相关但 Agent 没用——优化 Prompt 强调必须基于上下文；3) 检索不相关——检查 Embedding 模型和分块策略；4) 最后加 Rerank 提升 top-5 精度。CorpAssist 实践中 60% 的检索不准由分块不合理导致。」

第四问：RAG vs 微调

面试题：你的 Agent 场景，什么时候该用 RAG，什么时候该微调？

维度RAG微调
数据更新频率 高频（文档随时更新，RAG 只需更新索引） 低频（知识相对稳定，微调一次用半年）
知识来源 大量外部文档（政策、合同、说明书） 内部固化知识（产品规格、流程规范）
幻觉容忍度 低（RAG 通过检索约束，幻觉更低） 中（微调模型可能记忆不准）
开发周期 短（几天即可上线） 长（数据准备+训练+评估，2-4 周）
成本 运营成本（每次调用按量付费） 训练成本 + 推理成本（需部署模型）
CorpAssist 选择 发票处理、知识库问答、报告生成 代码补全（FIM）、客服话术模板

回答框架：「我们以 RAG 为主，微调为辅。RAG 适合知识频繁变化的场景（政策文档每周更新），检索即更新索引。微调适合固定知识+特定输出格式（如代码补全的 FIM 格式）。一个实用组合：RAG 提供事实知识，微调提供风格和格式。」

第五问：评估指标

面试题：你怎么评估 Agent 的好坏？准确率怎么算？

层级指标评估方法
检索质量 Recall@5, MRR 人工标注相关文档，计算召回率
工具选择 工具准确率 对比 Agent 选择的工具和正确答案
任务完成 任务完成率 端到端评估：最终结果是否正确
用户评价 用户采纳率, NPS 用户是否接受了 Agent 的建议

回答框架：「四层评估：检索层看 Recall@5，工具层看工具选择准确率，任务层看端到端完成率，用户层看采纳率。每层评估结果指导不同的优化方向——检索不准改分块，工具选错改 Prompt，任务失败改工作流逻辑。」

面试技巧总结

CorpAssist 面试答题模板
  • 先定性再定量：先说「我们分层解决这个问题」，再给出具体数据。
  • 对比显深度：不要只说「我们用了 X」，说「我们对比了 X、Y、Z 后选了 X，因为...」
  • 踩坑赢好感：「我们一开始也犯了...错误，后来通过...解决了」——面试官爱听 real experience。
  • 数字说话：准确率、召回率、延迟——每个回答至少带一个数据点。
  • 连贯叙事：五连问的回答要串起来，展现你从架构到评估的完整思考链。

常见问题（FAQ）

没有大厂 Agent 项目经验，面试怎么说？
用自己的 Demo 项目代替。重要的是展示你的思考深度而非项目规模。比如：自己做了一个发票识别 Agent，在 github 上有完整代码和 README。面试时讲解架构选型、踩过的坑、优化效果——这比「我在大厂负责一个小模块」更有说服力。
面试官问「如果 Agent 准确率只有 80% 怎么办」？
答案是分情况：1) 先确认 80% 是检索问题还是推理问题；2) 检索问题优化分块和 Rerank；3) 推理问题优化 Prompt 和工具描述；4) 如果还不行，加人工兜底——准确率低于 90% 的任务走人工审批。关键是展现出你有系统的优化方法论，而不是期待一次性 100%。
面试官问「对比 LangGraph 和其他框架」？
LangGraph 的核心优势是检查点持久化、人工审批、有状态工作流。对比：CrewAI 更适合多 Agent 编排但状态管理弱；AutoGen 有对话管理但偏研究；Dify 偏低代码平台不适合深度定制。选 LangGraph 因为生产级特性最完整。

本章结论

  • 架构题：四层架构（接入→编排→能力→基础），总-分-总框架回答。
  • 分块题：按文档类型选择策略，调优靠 Recall@5 实验。
  • 检索题：四步排查法——召回结果→Embedding→分块→Rerank。
  • RAG vs 微调：RAG 主事实知识，微调主风格格式，可组合使用。
  • 评估题：四层评估体系——检索→工具→任务→用户，层层递进。
