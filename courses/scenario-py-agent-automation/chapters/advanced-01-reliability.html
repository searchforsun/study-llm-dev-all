本章先记住 3 件事
  • 重试不等于可靠：不设退避的重试会打爆下游。指数退避 + 随机抖动是标准做法。
  • 熔断要快速失败：连续失败 N 次后直接返回错误，不再调用，给下游恢复时间。
  • 最终兜底是人：所有自动化方案都失效时，告警通知人工介入。

重试策略：指数退避与抖动

当工具调用失败时，直接重试往往会让问题更糟——下游系统可能已经过载。正确的做法是指数退避 + 随机抖动。

python 
# retry_strategy.py — 指数退避重试
import asyncio
import random
import time
from functools import wraps
from typing import Callable, Any

def retry_with_backoff(
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    jitter: bool = True,
    retryable_exceptions: tuple = (TimeoutError, ConnectionError),
):
    """指数退避重试装饰器"""
    def decorator(func: Callable) -> Callable:
        @wraps(func)
        async def wrapper(*args, **kwargs) -> Any:
            last_exception = None
            for attempt in range(max_retries + 1):
                try:
                    return await func(*args, **kwargs)
                except retryable_exceptions as e:
                    last_exception = e
                    if attempt == max_retries:
                        raise
                    delay = min(base_delay * (2 ** attempt), max_delay)
                    if jitter:
                        delay = delay * (0.5 + random.random())
                    print(f"[重试] 第 {attempt+1}/{max_retries} 次，等待 {delay:.1f}s")
                    await asyncio.sleep(delay)
            raise last_exception
        return wrapper
    return decorator

# 使用示例
class UnreliableAPIClient:
    @retry_with_backoff(max_retries=3, base_delay=0.5)
    async def call_erp(self, query: str) -> str:
        """不稳定的 ERP 接口"""
        if random.random() < 0.3:
            raise TimeoutError("ERP 超时")
        return f"结果: {query}"

async def test():
    client = UnreliableAPIClient()
    try:
        result = await client.call_erp("SELECT * FROM orders")
        print(f"成功: {result}")
    except TimeoutError:
        print("最终失败")

# 同步版本
def retry_sync(max_retries=3, base_delay=1.0):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if attempt == max_retries:
                        raise
                    delay = base_delay * (2 ** attempt) * (0.5 + random.random())
                    time.sleep(delay)
            return None
        return wrapper
    return decorator

if __name__ == "__main__":
    asyncio.run(test())

熔断器模式

python 
# circuit_breaker.py — 熔断器
import asyncio
from enum import Enum
from datetime import datetime, timedelta

class CircuitState(Enum):
    CLOSED = "closed"       # 正常
    OPEN = "open"           # 熔断
    HALF_OPEN = "half_open" # 半开（尝试恢复）

class CircuitBreaker:
    """熔断器"""

    def __init__(self, failure_threshold: int = 5, recovery_timeout: int = 30):
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.last_failure_time: datetime | None = None

    async def call(self, func, *args, **kwargs):
        if self.state == CircuitState.OPEN:
            if datetime.utcnow() - self.last_failure_time > timedelta(seconds=self.recovery_timeout):
                self.state = CircuitState.HALF_OPEN
                print("[熔断器] 进入半开状态，尝试恢复")
            else:
                raise CircuitBreakerOpenError("熔断器已打开，请求被拒绝")

        try:
            result = await func(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                print("[熔断器] 恢复成功，关闭熔断器")
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = datetime.utcnow()
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
                print(f"[熔断器] 熔断打开（连续失败 {self.failure_count} 次）")
            raise e

class CircuitBreakerOpenError(Exception):
    pass

人工兜底与告警体系

当自动化方案全部失效时，必须有人工兜底通道。CorpAssist 的告警体系：

告警级别触发条件通知方式响应要求
P0 危急 Agent 连续失败 > 10 次 / 影响核心业务 电话 + 短信 + IM 15 分钟内响应
P1 严重 熔断器打开 / 单个工具持续不可用 IM + 邮件 30 分钟内响应
P2 警告 Agent 步数超过阈值 / Token 消耗异常 IM 4 小时内处理
P3 通知 检查点写入延迟 / 偶发重试 日志 下一个工作日
可靠性实践 Checklist
  • 每个工具调用都有超时和重试（指数退避）
  • 熔断器保护关键下游（ERP、CRM）
  • Agent 有 max_iterations 和 max_execution_time
  • 降级路径可用（缓存 → 替代工具 → 人工）
  • 健康检查和自动恢复
  • P0-P3 告警配置完整
  • 人工兜底流程已演练过

常见问题（FAQ）

熔断器打开后，半开状态下如何决定是否恢复？
半开状态允许一个请求通过。如果成功，熔断器关闭；如果失败，熔断器重新打开并重启 recovery_timeout 计时。每次失败都增加 recovery_timeout（指数增长），防止频繁切换。
指数退避 + 抖动后，用户等待时间太长怎么办？
给用户即时反馈：「正在重试（第 2/3 次）」，避免用户以为系统卡死。同时在前端显示进度。如果重试超过 2 次，建议切换到降级路径而非继续等待。
人工兜底的具体流程是什么？
P0 告警触发后：1) 值班人员收到电话/短信通知；2) 查看 LangSmith Trace 定位失败原因；3) 手动执行 Agent 该步骤的操作（如查询数据库、发送邮件）；4) 将结果手动注入工作流（update_state）；5) 工作流恢复执行。全程记录在事后复盘文档中。

本章结论

  • 重试用指数退避：base_delay=1s, max_retries=3, jitter 随机化避免惊群效应。
  • 熔断快速失败：5 次连续失败打开熔断器，30s 后尝试半开恢复。
  • 四级降级：正常→限流降级→功能降级→人工兜底。每级有明确的触发条件。
  • P0-P3 告警：按严重程度分四级，每级有对应的通知方式和响应时间要求。
  • 人工兜底是最后防线：自动方案全失效时，值班人员查看 Trace 手动恢复。
