为什么要学本章

前几章我们用 ReAct Agent 解决简单任务，但真实业务需要有状态、可中断、可恢复的工作流。LangGraph 是 LangChain 推出的图结构 Agent 框架——它把 Agent 流程定义为有向图（节点是处理函数，边是状态流转），支持检查点持久化和人工审批中断。这是生产级 Agent 的核心基础设施。

本章先记住 3 件事
  • 图 = 节点 + 边：节点是处理函数，边是状态转移条件。LangGraph 的核心就是定义一个 StateGraph。
  • 检查点 = 断点续传：工作流每步自动持久化状态，崩溃后可从最后检查点恢复。
  • 人工审批 = 中断等待：在审批节点设置中断，有人确认后才继续执行。

LangGraph 图定义与状态管理

LangGraph 的核心抽象是 StateGraph——一个有向图，其中：

  • State：工作流的全局状态（一个 TypedDict 或 Pydantic BaseModel），在所有节点间传递
  • Node：处理函数，接收 state 返回更新的 state（或部分更新）
  • Edge：节点间的连接，可以是有条件的（condition）
CorpAssist · 报销审批工作流 员工提交报销单后，Agent 需要：① 提取发票关键信息 → ② 验证发票真伪 → ③ 检查预算余额 → ④ 部门经理审批 → ⑤ 财务打款。其中步骤④需要人工审批中断，其他步骤可自动执行。

LangGraph 工作流实现

python 
# expense_workflow.py — LangGraph 报销审批工作流
from typing import TypedDict, Dict, Any, Optional, Literal
from langgraph.graph import StateGraph, END
from langgraph.checkpoint import MemorySaver
from langchain_openai import ChatOpenAI
import json

# 1. 定义状态
class ExpenseState(TypedDict):
    """报销工作流状态"""
    expense_id: str
    employee_id: str
    amount: float
    invoice_text: str          # 原始发票文本
    extracted_info: dict       # LLM 提取的发票信息
    validation_result: dict    # 真伪验证结果
    budget_check: dict         # 预算检查结果
    approval_status: Optional[str]  # None / approved / rejected
    approval_comment: str      # 审批意见
    payment_result: str        # 打款结果
    error: Optional[str]       # 错误信息

# 2. 定义节点函数
def extract_invoice(state: ExpenseState) -> dict:
    """节点1: LLM 提取发票信息"""
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    prompt = f"""从以下发票文本中提取信息，返回 JSON：
发票文本：{state['invoice_text']}
返回字段：invoice_no, date, amount, vendor_name, tax_id"""
    result = llm.invoke(prompt)
    extracted = json.loads(result.content)
    return {"extracted_info": extracted}

def validate_invoice(state: ExpenseState) -> dict:
    """节点2: 验证发票真伪（模拟税务API）"""
    invoice = state["extracted_info"]
    # 真实场景调用国家税务总局 API
    valid = invoice.get("invoice_no", "").startswith("INV")
    return {
        "validation_result": {
            "valid": valid,
            "message": "发票验证通过" if valid else "发票号码异常",
        }
    }

def check_budget(state: ExpenseState) -> dict:
    """节点3: 检查预算余额"""
    # 模拟 ERP 预算检查
    dept_budget_remaining = 50000.0
    if state["amount"] <= dept_budget_remaining:
        return {"budget_check": {"sufficient": True, "remaining": dept_budget_remaining - state["amount"]}}
    return {"budget_check": {"sufficient": False, "remaining": dept_budget_remaining}}

def process_payment(state: ExpenseState) -> dict:
    """节点5: 执行打款"""
    return {"payment_result": f"已向员工 {state['employee_id']} 打款 {state['amount']} 元"}

def process_rejection(state: ExpenseState) -> dict:
    """节点6: 驳回处理"""
    return {"payment_result": f"报销被驳回: {state.get('approval_comment', '无理由')}"}

# 3. 条件边：根据审批状态选择路径
def route_after_approval(state: ExpenseState) -> Literal["process_payment", "process_rejection", "wait"]:
    if state.get("approval_status") == "approved":
        return "process_payment"
    elif state.get("approval_status") == "rejected":
        return "process_rejection"
    return "wait"  # 尚未审批

# 4. 构建图
def build_expense_graph():
    workflow = StateGraph(ExpenseState)

    # 添加节点
    workflow.add_node("extract_invoice", extract_invoice)
    workflow.add_node("validate_invoice", validate_invoice)
    workflow.add_node("check_budget", check_budget)
    workflow.add_node("process_payment", process_payment)
    workflow.add_node("process_rejection", process_rejection)

    # 添加边
    workflow.set_entry_point("extract_invoice")
    workflow.add_edge("extract_invoice", "validate_invoice")
    workflow.add_edge("validate_invoice", "check_budget")
    workflow.add_conditional_edges(
        "check_budget",
        lambda s: "process_payment" if s["budget_check"]["sufficient"] else "process_rejection",
    )
    workflow.add_edge("process_payment", END)
    workflow.add_edge("process_rejection", END)

    # 编译：启用检查点
    checkpointer = MemorySaver()
    app = workflow.compile(checkpointer=checkpointer)
    return app

if __name__ == "__main__":
    app = build_expense_graph()

    # 初始状态
    initial_state = ExpenseState(
        expense_id="EXP-001",
        employee_id="EMP-001",
        amount=3500.00,
        invoice_text="发票号码 INV-2026001，金额 3500 元，开票日期 2026-05-15，销方 某某科技公司",
        extracted_info={},
        validation_result={},
        budget_check={},
        approval_status=None,
        approval_comment="",
        payment_result="",
        error=None,
    )

    # 执行工作流
    for event in app.stream(initial_state):
        for node_name, node_state in event.items():
            print(f"[{node_name}] 状态更新: {node_state}")

检查点持久化与中断恢复

LangGraph 的检查点（Checkpoint）机制在工作流的每个节点执行后自动持久化状态。如果进程崩溃，可以从最后一个检查点恢复执行。CorpAssist 检查点使用场景：

  • 崩溃恢复：Agent 在第 5 步崩溃，重启后从第 5 步继续
  • 人工审批：工作流在审批节点暂停，等待人工回复后继续
  • 审计追踪：每个检查点记录完整的状态快照，用于事后审计
python 
# checkpoint_demo.py — 检查点持久化与恢复
from langgraph.checkpoint import MemorySaver, SqliteSaver
from langgraph.graph import StateGraph, END
from typing import TypedDict
import sqlite3

class SimpleState(TypedDict):
    messages: list
    step: int

def step_processor(state: SimpleState) -> dict:
    """模拟处理步骤"""
    current = state.get("step", 0) + 1
    return {
        "messages": state["messages"] + [f"完成步骤 {current}"],
        "step": current,
    }

# 使用 SQLite 持久化检查点（生产环境）
conn = sqlite3.connect("checkpoints.db", check_same_thread=False)
checkpointer = SqliteSaver(conn)

# 构建工作流
builder = StateGraph(SimpleState)
builder.add_node("process", step_processor)
builder.set_entry_point("process")
builder.add_conditional_edges(
    "process",
    lambda s: END if s["step"] >= 5 else "process",  # 5 步后结束
)

app = builder.compile(checkpointer=checkpointer)

# 第一次运行
thread_id = "thread-001"
config = {"configurable": {"thread_id": thread_id}}
initial = {"messages": ["开始"], "step": 0}

for event in app.stream(initial, config=config):
    print(event)

# 模拟崩溃后恢复：从检查点继续
print("\n[恢复执行] 从检查点继续...")
for event in app.stream(None, config=config):  # None 表示从上次继续
    print(event)

# 获取历史状态
state_history = list(app.get_state_history(config))
print(f"\n检查点数量: {len(state_history)}")
不推荐：无状态 ReAct
  • 每次调用都是独立会话，崩溃后丢失全部上下文
  • 无法在中间步骤暂停等待
  • 没有审计追踪能力
  • 人工审批只能通过外部系统模拟
推荐：LangGraph 有状态工作流
  • 每步自动持久化，崩溃可恢复
  • 原生支持中断/恢复（人工审批节点）
  • 检查点历史作为审计日志
  • 内置检查点 API，无需自建持久化

人工审批节点实现

python 
# human_approval.py — 人工审批节点
from langgraph.graph import StateGraph, END
from typing import TypedDict, Optional

class ApprovalState(TypedDict):
    request_id: str
    amount: float
    status: Optional[str]       # pending / approved / rejected
    reviewer: Optional[str]
    comment: str

def request_approval(state: ApprovalState) -> dict:
    """人工审批节点：暂停等待"""
    # 状态设为 pending，LangGraph 在此中断
    print(f"[审批请求] #{state['request_id']} 金额: {state['amount']} 等待审批")
    return {"status": "pending"}

def after_approval(state: ApprovalState) -> dict:
    """审批后的处理"""
    if state["status"] == "approved":
        print(f"[审批通过] 审批人: {state.get('reviewer')}, 意见: {state.get('comment')}")
    else:
        print(f"[审批驳回] 审批人: {state.get('reviewer')}, 原因: {state.get('comment')}")
    return state

# 在外部恢复执行时设置审批结果
def resume_with_approval(app, thread_id: str, approved: bool, reviewer: str, comment: str = ""):
    """恢复工作流，设置审批结果"""
    config = {"configurable": {"thread_id": thread_id}}
    app.update_state(
        config,
        {"status": "approved" if approved else "rejected", "reviewer": reviewer, "comment": comment},
    )
    # 继续执行
    for event in app.stream(None, config):
        print(f"事件: {event}")

常见问题（FAQ）

LangGraph 和直接写 Python 状态机有什么区别？
LangGraph 提供了状态持久化、中断恢复、流式输出、LangSmith 追踪等基础设施。自己写状态机需要手撸：状态序列化/反序列化、崩溃恢复、并发控制、审计日志——这些 LangGraph 都已内置。而且 LangGraph 与 LangChain 生态（工具、记忆、LLM 调用）无缝集成。
检查点存在哪里？生产环境推荐什么存储？
LangGraph 支持 MemorySaver（内存，开发用）、SqliteSaver（SQLite，单机）、PostgresSaver（PostgreSQL，生产）。生产环境推荐 PostgreSQL + 连接池，支持 ACID 事务和水平扩展。检查点记录了每次 state 变更，注意设置生命周期避免存储膨胀。
一个 LangGraph 工作流可以嵌套另一个工作流吗？
可以。LangGraph 支持节点内调用子图（Subgraph）。你在一个节点函数中调用另一个编译好的 StateGraph，子图有自己的状态空间和检查点。这在多 Agent 协作场景中非常有用——每个 Agent 是一个子图，由编排 Agent 统一调度。

本章结论

  • StateGraph 是核心：用 TypedDict 定义状态，用节点函数处理逻辑，用边定义流转条件。
  • 检查点是生产保障：自动持久化每步状态，支持崩溃恢复和中断恢复。生产用 PostgreSQL 存储。
  • 人工审批节点：工作流在审批点中断，等待外部输入后恢复，通过 update_state 注入审批结果。
  • LangGraph vs ReAct：ReAct 适合简单工具调用链，LangGraph 适合需要状态管理、中断恢复的复杂流程。
  • 审计与可观测：检查点历史天然就是审计日志，配合 LangSmith 实现全流程追踪。
