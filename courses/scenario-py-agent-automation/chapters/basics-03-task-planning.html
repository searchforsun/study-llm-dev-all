为什么要学本章

前两章的 ReAct Agent 可以完成单步工具调用，但真实企业场景往往需要多步规划：比如「生成季度销售报告」需要先查数据、再分析、再排版、再发邮件。如果让 Agent 毫无规划地乱试，步数和 Token 会爆炸。本章教你如何让 Agent 像项目经理一样先规划后执行。

本章先记住 3 件事
  • 先规划后执行：让 Agent 先产出一份任务清单，再逐步执行，比边走边想效率高 2-3 倍。
  • 子任务有依赖：DAG（有向无环图）是任务规划的核心数据结构，决定并行/串行边界。
  • 规划会出错：实际执行结果与规划不符时，Agent 必须有能力重规划。

规划策略：CoT 与 Plan-and-Solve

任务规划的核心问题是：如何将用户的一句话需求拆解为可执行的子任务序列。两种主流策略：

策略思路适用场景步数效率
Chain-of-Thought (CoT) 在 ReAct 的 Thought 中逐步推理，边想边做 步数少（< 10）、步间依赖紧密 中
Plan-and-Solve (P&S) 先一次性产出完整计划，再逐个执行 步数多（10~50）、子任务相对独立 高
ReAct + 重规划 边执行边调整计划，每 N 步重新评估 动态环境、外部输入不可预测 低但鲁棒
CorpAssist · 季度销售报告生成 销售总监要求 Agent 生成 Q2 销售报告：① 从 CRM 拉取 Q2 订单数据 → ② 计算同比/环比增长 → ③ 分析 Top 10 产品和滞销产品 → ④ 生成 PPT 报告草稿 → ⑤ 邮件发送给管理层。共 5 个步骤，其中 ①→②→③ 有依赖关系，③→④ 和 ③→⑤ 可并行。

    Plan --> T3[子任务3: 分析Top 10和滞销品]
    Plan --> T4[子任务4: 生成PPT草稿]

Plan-and-Execute 实现

python 
# plan_execute.py — CorpAssist Plan-and-Execute Agent
from langchain_openai import ChatOpenAI
from langchain.agents import tool
from langchain.chains.plan_and_execute import (
    PlanAndExecute, load_agent_executor, load_chat_planner,
)
from langchain_community.agent_toolkits.load_tools import load_tools
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

@tool
def query_crm(period: str) -> str:
    """从 CRM 系统查询销售数据，period 格式如 '2025-Q2'"""
    # 实际调用 CRM API
    return f'{{"period": "{period}", "total_revenue": 1250000, "orders": 340}}'

@tool
def send_email(recipient: str, subject: str, body: str) -> str:
    """发送邮件"""
    return f"邮件已发送至 {recipient}，主题: {subject}"

@tool
def generate_ppt(title: str, content: str) -> str:
    """生成 PPT 草稿，返回文件路径"""
    return f"/tmp/reports/{title}.pptx"

def create_plan_agent():
    """创建先规划后执行的 Agent"""
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    tools = [query_crm, send_email, generate_ppt]

    planner = load_chat_planner(llm)
    executor = load_agent_executor(llm, tools, verbose=True)

    agent = PlanAndExecute(
        planner=planner,
        executor=executor,
        verbose=True,
    )
    return agent

# 手动实现规划器（更灵活）
def create_custom_planner():
    """自定义规划逻辑：LLM 先生成 JSON 计划"""
    planner_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    planning_prompt = ChatPromptTemplate.from_messages([
        ("system", "你是一个任务规划助手。将用户需求分解为子任务列表，返回 JSON 格式。"
         "每个子任务包含: step_id, description, tool, depends_on(依赖上一步的step_id列表)"),
        ("human", "{input}"),
    ])

    chain = planning_prompt | planner_llm
    return chain

if __name__ == "__main__":
    # 测试计划生成
    planner = create_custom_planner()
    plan = planner.invoke({"input": "查询 Q2 销售数据，生成 PPT，然后邮件发送给 CEO"})
    print(f"生成的计划:\n{plan.content}")
不推荐：无规划直接执行
  • Agent 边走边想，每次调用都损失上下文
  • 无法识别可并行步骤，串行执行浪费时间
  • 中间步骤失败时从头重试，浪费已完成的子任务结果
  • 难以估算总步数和 Token 消耗
推荐：Plan-and-Execute
  • LLM 先输出完整的子任务列表和依赖关系
  • 用 DAG 调度器识别可并行执行的子任务
  • 失败时仅重试失败的子任务及其下游
  • 可预估总 Token 消耗和完成时间

执行-观察-反思循环

执行-观察-反思（Execute-Observe-Reflect） 是提升规划质量的关键机制。Agent 在执行过程中定期暂停并反思：当前进度是否符合预期？是否需要调整计划？

执行-观察-反思循环
flowchart LR
    Plan[初始计划] --> Execute[执行子任务]
    Execute --> Observe[观察结果]
    Observe --> Check{结果符合预期?}
    Check -->|是| Next[执行下一子任务]
    Check -->|否| Reflect[反思原因]
    Reflect --> Adjust[调整计划]
    Adjust --> Execute
    Next --> Done{所有任务完成?}
    Done -->|否| Execute
    Done -->|是| Final[输出最终结果]

    style Plan fill:#7B68EE,color:#fff
    style Reflect fill:#D2691E,color:#fff
    style Final fill:#4A90D9,color:#fff

每执行一个（或一组）子任务，Agent 观察结果并与预期对比，不一致时触发反思和计划调整。

带反思的规划 Agent 实现

python 
# reflect_agent.py — 带反思的规划 Agent
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field
from typing import List, Optional, Literal
import json

class SubTask(BaseModel):
    """子任务定义"""
    step_id: int
    description: str
    tool: str
    depends_on: List[int] = Field(default_factory=list)
    status: Literal["pending", "running", "done", "failed"] = "pending"

class TaskPlan(BaseModel):
    """完整任务计划"""
    goal: str
    subtasks: List[SubTask]

class ReflectAgent:
    """带执行-观察-反思的规划 Agent"""

    def __init__(self, model: str = "gpt-4o-mini"):
        self.llm = ChatOpenAI(model=model, temperature=0)
        self.plan: Optional[TaskPlan] = None
        self.results: dict[int, str] = {}

    def plan_task(self, goal: str) -> TaskPlan:
        """第一步：LLM 生成任务计划"""
        prompt = f"""将以下目标分解为子任务列表，以 JSON 格式返回：
目标：{goal}
要求：
- 每个子任务包含 step_id, description, tool, depends_on
- depends_on 是依赖的上一步 step_id 列表
- 无依赖的任务列在 depends_on: []
- 返回格式：{{"goal": "...", "subtasks": [...]}}"""
        response = self.llm.invoke(prompt)
        data = json.loads(response.content)
        self.plan = TaskPlan(**data)
        return self.plan

    def execute_step(self, step: SubTask) -> str:
        """执行单个子任务（模拟）"""
        print(f"[执行] 步骤 {step.step_id}: {step.description}")
        # 实际工程中调用对应工具
        return f"步骤 {step.step_id} 完成: {step.description}"

    def reflect(self, step: SubTask, result: str) -> Optional[str]:
        """反思步骤：检查结果是否需要调整计划"""
        prompt = f"""你正在执行以下任务计划：
目标：{self.plan.goal}
当前步骤：{step.step_id} - {step.description}
执行结果：{result}

请反思：
1. 结果是否与预期一致？
2. 是否需要调整后续计划？
3. 是否有新的依赖关系出现？
回答格式：{{"consistent": bool, "adjustment": "调整内容或null"}}"""
        response = self.llm.invoke(prompt)
        data = json.loads(response.content)
        return data.get("adjustment")

    def run(self, goal: str) -> dict[int, str]:
        """执行完整规划+执行+反思流程"""
        print(f"[规划] 目标: {goal}")
        self.plan_task(goal)
        print(f"[规划] 共 {len(self.plan.subtasks)} 个子任务")

        for step in self.plan.subtasks:
            result = self.execute_step(step)
            self.results[step.step_id] = result

            adjustment = self.reflect(step, result)
            if adjustment:
                print(f"[反思] 计划调整: {adjustment}")
                # 此处可修改 self.plan.subtasks
        return self.results

if __name__ == "__main__":
    agent = ReflectAgent()
    results = agent.run("查询 Q2 销售数据，生成 PPT 报告并邮件发送")
    for step_id, result in results.items():
        print(f"  步骤 {step_id}: {result[:60]}...")
先想 10 秒：如果子任务 2 执行失败，子任务 3 依赖它的结果，你怎么设计恢复策略？

三种策略：1) 重试：子任务 2 最多重试 3 次，每次间隔指数退避；2) 降级：子任务 3 使用子任务 2 的部分结果或默认值继续；3) 重规划：通知规划器重新调整任务分解。生产环境推荐组合使用：先重试，失败则降级，降级不满足业务需求时触发重规划。

Token 预算与步数控制

规划 Agent 的 Token 预算 管理是生产化的核心问题。每步 Thought/Action/Observation 都会增加 Token 消耗。

策略Token 消耗/步10 步总消耗优缺点
CoT (边想边做) ~500 ~5,000 灵活但消耗大
Plan-and-Solve ~200 ~2,000 高效但不够灵活
ReAct + 重规划 ~350 ~3,500 灵活性和效率折中
CorpAssist 实践建议

对于 5~15 步的中等复杂度任务，推荐 Plan-and-Solve + 每 5 步触发一次反思。这样既能保持效率，又能应对执行中的偏差。设置 max_total_tokens=8000 作为安全上限。

常见问题（FAQ）

Plan-and-Solve 和 LangGraph 是什么关系？
Plan-and-Solve 是策略层概念——它描述 Agent 先规划再执行的行为模式。LangGraph 是实现层框架——它提供了定义有状态工作流的图结构。你可以用 LangGraph 实现 Plan-and-Solve：规划阶段用一个 LLM 节点，执行阶段用一串工具节点。实际上，下一章的 practice-01 正是用 LangGraph 实现可检查点的任务规划工作流。
如果 LLM 生成的计划有 20 步，但实际只需要 5 步，怎么办？
这是「过度分解」问题。解决方法：1) 在规划 Prompt 中给出步数范围（「请分解为 3-8 个子任务」）；2) 在反思阶段检查是否有子任务可以合并；3) 使用动态规划——先分解为粗粒度子任务，执行时再进一步细分。CorpAssist 的实践经验是：粗规划 + 细执行 = 最佳平衡。
Agent 规划错了步骤（比如第三步其实应该放在第一步），怎么处理？
分两种场景：1) 执行前发现——可以用另一轮 LLM 调用做「计划验证」，让第二个 LLM 检查计划正确性；2) 执行后失败才发现——依赖检测机制。当子任务 B 依赖子任务 A 的结果，但 A 还没执行时，B 的 Observation 会报错「缺少依赖数据」，触发重规划。推荐在工具函数中做前置条件检查。

本章结论

  • 任务规划三策略：CoT（边想边做）、Plan-and-Solve（先计划再执行）、ReAct+重规划（动态调整），按场景复杂度选择。
  • DAG 是规划核心：子任务的依赖关系用有向无环图表示，决定串行/并行边界。
  • 执行-观察-反思：定期暂停并检查进度，发现偏差时重规划，提升鲁棒性。
  • Token 预算管理：CoT 最耗 Token，Plan-and-Solve 最省。设置 max_iterations 和 max_total_tokens 防止失控。
  • 过度分解要避免：粗规划 + 细执行，用反思阶段合并可合并的子任务。

判断练习

  1. 如果子任务 A 和 B 互不依赖，但 Agent 仍然串行执行了 A→B。你的优化建议是什么？

    参考答案

    在规划阶段检查 depends_on 字段：如果两个子任务的 depends_on 列表没有交集，且都没有被对方依赖，它们可以并行执行。实现上可以使用 asyncio.gather 或 LangGraph 的 parallel 节点。并行执行可以将总耗时从 T(A)+T(B) 降至 max(T(A), T(B))。
