为什么要学本章

Prompt 改一句线上行为可能巨变。没有 ab experiment 与 eval gate，只能靠「感觉更好」上线。本章把 RAG/Agent/工单三类模板的迭代做成可度量、可回滚的 prompt deploy 流程。

学完你能

  • 选取 prompt metric（faithfulness、拒答率、JSON 通过率等）
  • 配置 release gate 与 significance 判断
  • 执行 rollback prompt 与版本锁定
本章先记住 4 件事
  • 一次只改变量：A/B 对比时仅改 system 或仅改拒答句。
  • 离线先行：golden 集过 eval gate 再开流量。
  • 指标成对：质量指标 + 成本指标（latency、Token）。
  • 回滚预设：feature flag 绑 prompt_version，一键 rollback。

实验设计

CorpAssist · RAG 拒答句 A/B 对照 rag-v2.2（拒答句 A）与 rag-v2.3（拒答句 B + 更严 context-only）。假设：B 降低幻觉率、略升拒答率。流量 variant split 10% / 10%，其余 80% 保持 control。
要素CorpAssist 做法
假设「v2.3 提升 faithfulness ≥ 3pp，拒答率 ≤ +5pp」
分组按 tenant_id hash 或 session_id sticky 分桶
周期至少 1 周或 500+ 有效样本
隔离不与模型版本变更混测
Prompt A/B 实验生命周期
flowchart LR
  H[假设 + 改 diff] --> OFF[离线 golden eval-gate]
  OFF -->|通过| CAN[Canary variant-split 5-10%]
  CAN --> MET[prompt-metric 采集]
  MET --> SIG{significance + 闸门}
  SIG -->|通过| FULL[prompt-deploy 100%]
  SIG -->|失败| RB[rollback-prompt]
  FULL --> MON[持续监控]

指标

prompt metric 因场景而异。S1 RAG：faithfulness、引用覆盖率、拒答率；S2 Agent：工具成功率、平均步数；工单 JSON：parse 通过率、repair 次数。

指标定义闸门示例
faithfulness人工或 LLM-judge：答案是否由 context 支持treatment ≥ control + 0.03
refusal_rateno-answer 句式占比Δ ≤ +0.05
json_pass_rate首次生成即 schema 通过≥ 0.92
p95_latency含 repair-loop 的端到端延迟≤ control × 1.1
坏实践
  • 只看「用户点赞」无对照组
  • 同时改 Prompt + 换模型
  • 样本 50 条就宣布显著
好实践
  • primary + guardrail 指标成对
  • significance 用比例检验 + 最小样本量
  • 审计日志带 prompt_version

发布闸门

release gate = 离线 eval gate + 在线 canary 闸门。离线：golden 集回归无劣化；在线：treatment 相对 control 满足假设且 guardrail 未破。

python 
# 离线 eval-gate（评测脚本）
def eval_gate(control_scores, treatment_scores, cfg) -> bool:
    delta_f = treatment_scores["faithfulness"] - control_scores["faithfulness"]
    delta_r = treatment_scores["refusal_rate"] - control_scores["refusal_rate"]
    return delta_f >= cfg["min_faithfulness_delta"] and delta_r <= cfg["max_refusal_delta"]
java 
// Spring · prompt-deploy 路由
@Service
public class PromptVariantRouter {
  public String resolveRagPromptVersion(String sessionId) {
    Experiment exp = config.get("rag-refusal-v23");
    String bucket = stickyHash(sessionId, exp.variantSplit());
    return bucket.equals("treatment") ? exp.treatment() : exp.control();
  }
}

回滚

rollback prompt：将 variant_split.treatment 置 0 或 feature flag 指回 control；无需重新部署代码，只需配置中心改 prompt_version 指针。事故时保留 treatment 日志用于 postmortem。

回滚决策
flowchart TD
  ALERT[guardrail 告警] --> CHECK{faithfulness 降 > 2pp?}
  CHECK -->|是| RB[rollback-prompt 至 control]
  CHECK -->|否| HOLD[继续观察]
  RB --> AUDIT[记录 incident + diff]
  AUDIT --> FIX[离线修复 + 新 ab-experiment]
关键术语速览

ab experiment · prompt metric · release gate · rollback prompt · variant split · significance · prompt deploy · eval gate

先想 10 秒：离线 golden 全过，能否跳过 5% canary 直接全量？

不建议。离线集无法覆盖生产分布与延迟 guardrail；canary 验证在线 primary/guardrail，破线则 rollback prompt。
