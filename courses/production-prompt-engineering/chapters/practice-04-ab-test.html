为什么要学本章

「改一句 System Prompt 上线」在生产等于未评估的配置变更。CorpAssist 曾将 RAG 拒答阈值从 0.72 调到 0.65，拒答率下降但幻觉投诉上升——缺少 Prompt A/B 实验 与发布闸门就会踩坑。本章把 Prompt 迭代纳入可观测、可回滚的工程流程。

学完你能
  • 设计单变量、流量可分的 Prompt A/B 实验方案
  • 选取 Prompt 指标（质量、成本、延迟、拒答率）
  • 配置评测 + 灰度的 发布闸门
  • 执行版本化 Prompt 回滚 与 badcase 复盘
本章先记住 3 件事
  • 单变量：一次实验只改 Prompt 的一处（如拒答阈值文案），否则无法归因。
  • 离线评测先行：golden set 不过闸门，禁止进 A/B 流量。
  • 回滚要秒级：Prompt 版本存 Git + 配置中心，回滚 = 切版本指针。

实验设计

CorpAssist · RAG 拒答 Prompt v2 vs v3 v2 拒答话术生硬，产品希望 v3 更友好。实验假设：v3 在用户满意度不降的前提下，拒答率变化 < 2pp，且幻觉率不升。

Prompt A/B 实验 需明确：对照组/实验组 Prompt 版本、流量比例、分流键（tenant_id / user_id 哈希）、实验窗口、停止条件。

要素CorpAssist 约定
分流单位tenant_id 级（避免同一企业内体验不一致）
初始流量5% → 20% → 50% 阶梯（每阶 ≥24h）
单变量仅 refusal.templates.low_confidence 文案
停止条件幻觉率 +0.5pp 或 P95 延迟 +200ms 立即停
Prompt A/B 流量阶梯
flowchart LR
  OFF[离线 golden 评测] --> G{闸门通过?}
  G -->|否| FIX[修 Prompt]
  G -->|是| AB5[A/B 5%]
  AB5 --> AB20[20%]
  AB20 --> AB50[50%]
  AB50 --> FULL[100% 全量]

每阶需满足指标 SLO；任一级触发 stop rule 则回滚至上一稳定版本。

实验配置示例（ab-plan.yaml）

yaml 
experiment:
  id: rag-refusal-copy-v3
  control: rag-prompt-v2
  treatment: rag-prompt-v3
  split_key: tenant_id
  stages: [0.05, 0.20, 0.50, 1.0]
  single_variable: refusal.templates.low_confidence
  duration_min_hours_per_stage: 24
常见误区
  • 同时改 Prompt 和模型 — 无法判断提升来自哪一侧。
  • 按请求随机而非 tenant — 同一用户前后体验不一致，引发投诉。

实验文档 ab-plan.yaml 必填项

  • hypothesis：可证伪假设（如「更友好拒答话术不升幻觉」）
  • owner：产品 + 研发双负责人
  • rollback_contact：on-call 轮值
  • success_criteria：量化通过条件

指标

Prompt 指标 分四类：质量（幻觉率、 cite 覆盖率）、体验（CSAT、拒答率）、成本（input/output tokens）、性能（P95 延迟）。CorpAssist 在 Langfuse / 自研看板按 prompt_version 标签聚合。

指标定义闸门阈值（示例）
hallucination_rate人工或 LLM-judge 标为无依据回答占比实验组 ≤ 对照 + 0.5pp
refusal_rate走拒答模板请求占比产品定义区间
cite_coverage含合法 cite 的回答占比≥ 95%
token_p95input+output tokens P95≤ 对照 + 10%
latency_p95端到端 P95 ms≤ 对照 + 200ms

指标查询伪代码（按 prompt_version 分组）

python 
# 离线 golden + 在线 A/B 共用评估函数
def aggregate_metrics(traces: list[dict]) -> dict:
    by_version = {}
    for t in traces:
        v = t["metadata"]["prompt_version"]
        by_version.setdefault(v, []).append(t)
    return {
        v: {
            "hallucination_rate": sum(t["hallucination"] for t in rows) / len(rows),
            "refusal_rate": sum(t["refused"] for t in rows) / len(rows),
            "token_p95": percentile([t["tokens"] for t in rows], 95),
        }
        for v, rows in by_version.items()
    }

不一定——可能是阈值放松导致强行回答，幻觉率会升。须与 hallucination_rate 联看。

发布闸门

发布闸门 是 Prompt 进入流量的强制检查点：离线 golden 评测 → 安全红队抽检 → 小流量 A/B → 全量。任一环节失败则阻塞发布。

Gate 1 · 离线
  • golden set ≥200 条，覆盖 S1/S2/工单
  • 关键指标不低于 baseline
  • diff 审查（Prompt CR）
Gate 2 · 在线
  • 5% 流量 ≥24h
  • stop rule 未触发
  • on-call 确认

闸门检查清单（CI 可集成）

bash 
# demos/practice-04-ab-test-lab/run-gate.sh
python eval_golden.py --prompt rag-prompt-v3 --baseline rag-prompt-v2
# 期望: PASS cite_coverage>=0.95 hallucination_delta<=0.005
python diff_prompt.py rag-prompt-v2 rag-prompt-v3
# 期望: single_variable confirmed

CI 失败则阻塞 merge；紧急 hotfix 须 tech lead + 产品双签 waiver，并在 24h 内补跑离线评测。

用数据说话：展示上次跳过实验导致的 badcase；提供「24h 5% 快速实验」折中；记录 waiver 签字与回滚预案。

离线 golden 与在线 A/B 指标不一致怎么办？

检查 golden 是否覆盖真实 traffic 分布；在线可能有长尾 query。应扩充 golden 而非直接否定 A/B。

回滚

在线指标触发 stop rule 或重大投诉时，执行 Prompt 回滚：配置中心 active_prompt_version 指回上一稳定 tag，无需重新部署服务（热切换）。

回滚 Runbook 要点

  1. 确认异常与 prompt_version 标签关联（非模型或检索变更）
  2. 配置中心切换：active_prompt_version: rag-prompt-v2
  3. 验证 canary tenant 请求 metadata 已变
  4. 15 分钟内指标恢复；否则升级 incident
  5. 48h 内 badcase 复盘 + Prompt CR 记录
yaml 
# config/prompt-routing.yaml
active:
  rag: rag-prompt-v2        # 回滚目标
experiments:
  rag-refusal-copy-v3:
    enabled: false          # 实验立即关闭
    treatment: rag-prompt-v3
rollback:
  last_stable: rag-prompt-v2
  previous: rag-prompt-v1

回滚演练 checklist

  1. 预发环境验证配置中心切换延迟 < 30s
  2. canary tenant 请求携带正确 prompt_version metadata
  3. 实验 enabled: false 后 treatment 流量归零
  4. on-call 收到 PagerDuty 与 runbook 链接

每季度在预发做一次回滚演练，验证从 v3 切回 v2 后指标在 15 分钟内恢复正常，并更新 runbook 中的实际切换耗时。

回滚决策须同步产品、客服负责人，避免实验组用户继续收到已下线话术版本的混淆体验。

Prompt 版本与 Git tag 一一对应（见 变量、版本与配置管理）；回滚后实验组 tenant 自动回到 control 版本。

本章结论

Prompt 迭代 = 实验设计 + 指标驱动 + 闸门 + 可秒级回滚。

  • 实验设计：tenant 级分流、单变量、阶梯流量与明确 stop rule。
  • 指标：质量/体验/成本/延迟四类，按 prompt_version 聚合，拒答与幻觉联看。
  • 发布闸门：离线 golden 先于在线 A/B；CI 可集成 eval 与 diff 脚本。
  • 回滚：配置中心切版本指针 + 关闭实验；Git tag 与复盘闭环。
