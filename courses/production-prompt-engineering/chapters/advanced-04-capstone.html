为什么要学本章

Capstone 把本课散落能力收成可上线资产包：three scenarios 各一套模板 + 评测 + 文档。面试与项目答辩常问「你如何交付企业级 Prompt 套件」——本章给出目录结构与验收清单。

学完你能
  • 组装 S1/S2/S5 三套 template pack
  • 建立 Git version repo 与 golden bind
  • 输出 capstone prompt 交付说明（回滚、指标、负责人）
本章先记住 5 件事
  • S1 来自 RAG 专用 Prompt：引用、拒答、仅依据上下文。
  • S2 来自 Agent 与工具 Prompt：工具描述、计划、失败恢复。
  • S5 来自 内容生成与创作自动化：风格链、长文、安全审核。
  • 三套共享：注入防护（注入攻防）、语调合规（语气合规）、PromptOps（协作流程）。
  • 交付 = 模板 + eval + registry + 发布记录，缺一不可。

版本库

version repo 约定：语义化版本 MAJOR.MINOR.PATCH——破坏兼容升 MAJOR；新增变量升 MINOR；文案微调升 PATCH。生产只引用 tag，不引用 main 浮动分支。

评测绑定

每个模板必须 eval bind 独立 golden set；共享 injection 子集。指标示例：

  • S1：faithfulness、引用准确率、拒答准确率
  • S2：工具选择准确率、任务完成率
  • S5：语调分、禁用语命中率、段落重复率（dedup）

跨场景依赖

判断练习

  1. S1 golden 全过、S2 客服话术未测，能否打包交付？

    参考答案

    否。Capstone 要求三场景均 eval bind；缺 S2 则 delivery pack 不完整，面试无法辩护端到端交付。

  2. shared/injection-guard.yaml 升 MAJOR，三场景模板是否都要升 MAJOR？

    参考答案

    共享合规块 MAJOR 意味着行为契约变化；三场景至少 MINOR 并各跑 golden + injection 子集；是否升 MAJOR 看是否改变拒答/边界语义。
