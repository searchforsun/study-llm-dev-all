本章先记住 4 件事
  • 示例质量 > 数量：过时 few-shot 比 zero-shot 更糟；维护 example bank 并定期回归。
  • CoT 有价：每一步 cot step 都消耗 Token；生产可用「短链 / 隐藏链 / 仅失败时开链」。
  • 分层放置：few-shot 通常进 system 或 user 前缀，不破坏 边界声明 的优先级。
  • 双栈一致：dual template 指同一 YAML/MD 真源，Jinja 与 PromptTemplate 只负责渲染。

示例选择

CorpAssist · S2 意图路由（30 秒） 客服 utterance 需分为「查物流 / 改地址 / 投诉 / 闲聊」。产品给了 50 条历史对话，但含 2022 年已下线流程。错误示例会把模型引向已废弃 API——few shot select 要比「多塞几条」更重要。

Few-shot 通过输入→输出对演示期望行为。生产环境应维护 example bank：带标签、版本、适用场景（S1/S2/S5），并与评测集联动更新。

S2 意图分类 few-shot（Markdown）

markdown 
## 示例（仅格式演示，生产从 example-bank 加载）
输入：我的包裹到哪了？单号 SF123
输出：{"intent":"track_shipment","slots":{"tracking_no":"SF123"}}

输入：你们经理在吗？我要投诉
输出：{"intent":"complaint","slots":{"handoff":true}}

输入：今天天气怎么样
输出：{"intent":"out_of_scope","slots":{"reply":"我是客服助手，无法回答天气问题"}}
不推荐
  • 从生产日志随机抽样，未脱敏、未标注
  • 示例输出与当前 JSON schema 不一致
  • 无版本号，产品改流程后 Prompt 未同步
推荐
  • example bank Git 管理，变更关联评测 diff
  • 按 S1/S2/S5 分目录；选择 策略文档化
  • 坏例回流进 bank（PromptOps，见 advanced 章）
先想 10 秒：S1 制度问答需要 few-shot 吗？什么时候 zero-shot 更好？

引用格式简单、模型跟 system 边界即可时，zero-shot + 强边界 更省 Token。当输出结构复杂（表格比对、多段引用）或错误率高时，再加 2～3 条同格式 few-shot。

思维链

CorpAssist · S1 制度比对（30 秒） 问「异地办公与驻外补贴能否同时享受？」需交叉两篇制度。模型若直接给结论易跳步幻觉。Chain-of-thought 要求先列依据条款，再结论——便于人工审计与自动 faithfulness 检查。

思维链（CoT） 让模型显式写出中间 推理步骤。工程上要决定：步骤是否展示给用户、是否写入日志、失败时是否缩短链。

CoT 三种部署模式
flowchart TB
  Q[用户问题]
  Q --> FULL["完整 CoT · 展示+日志"]
  Q --> SHORT["短链 · 仅关键 2 步"]
  Q --> HIDE["隐藏 CoT · 内部推理不返回"]
  FULL --> ANS[最终答案]
  SHORT --> ANS
  HIDE --> ANS

S5 长文规划常用完整链供编辑审核；S2 高 QPS 场景优先短链或隐藏链以控 Token 预算。

CoT Prompt 片段（S1 比对）

markdown 
请按以下步骤回答（步骤可写入 <reasoning>，最终答案写入 <answer>）：
1. 列出 context 中与问题相关的条款原文（带引用）
2. 说明条款之间是否冲突或互斥
3. 若信息不足，在 <answer> 中拒答
4. 仅在步骤 1–2 充分时在 <answer> 给结论
不推荐
  • 「Let's think step by step」无结构，步骤不可解析
  • 生产全量返回 10 步推理，用户读不完、Token 翻倍
  • CoT 与 few-shot 示例格式矛盾（示例无 reasoning 块）
推荐
  • 固定 XML/JSON 块：<reasoning> / <answer>
  • few-shot 含完整 cot step 示范
  • 监控 CoT 长度，超阈值降级为短链
CoT 一定提高准确率吗？

不一定。简单任务可能更慢更贵且无增益；复杂多跳推理、高错误成本场景（S1 合规、S5 长文结构）收益更大。用评测集对比 zero-shot / few-shot / CoT 再放量。

成本权衡

Prompt 成本 ≈ 输入 Token × 单价 + 输出 Token × 单价。Few-shot 与 CoT 同时开启时，输入与输出都会涨——需按场景算 tradeoff。

策略Token 影响适用场景CorpAssist 建议
zero-shot 最低 强 system + 简单输出 S1 标准 Q&A（引用格式已在 system）
few-shot (3–5) 输入 +30%～80% 格式对齐、意图分类 S2 意图 / S5 大纲结构
full CoT 输出 +50%～200% 多步推理、可审计 S1 交叉制度、低 QPS 分析
动态降级 按请求切换 高 QPS + 长尾复杂题 S2 默认短链，置信度低时再开 CoT

粗算示意（教学用，非账单）

python 
# CorpAssist S2：单次请求 Token 粗算
system = 800          # 人设 + 边界
few_shot = 3 * 120    # 3 shots × ~120 tokens
user = 200            # 用户 utterance
cot_output = 400      # CoT reasoning + JSON

baseline = system + user                    # zero-shot ≈ 1000
with_fs_cot = baseline + few_shot + cot_output  # ≈ 1960 (+96%)

# 决策：日 100 万次 → 优先减 shot 或隐藏 CoT
3 条

条数增加通常提升格式稳定性，但挤占 RAG context 窗口；S1 需为检索 chunks 留预算。

100 万 QPS 的 S2 能否默认 full CoT？

通常不能。应默认短输出 + 隐藏或省略 CoT；仅对低置信度、投诉升级、复杂工单二次调用开链，并记录 token-budget-cot 指标。

双栈模板

双栈模板：同一份 prompts/s2-intent.md（含 few-shot 占位与 CoT 指令），Python 用 Jinja2 渲染，Java 用 Spring PromptTemplate 渲染，保证字段名与示例顺序一致。

Python · Jinja2
python 
from jinja2 import Template

template = Template(open("prompts/s2-intent.md").read())
system = template.render(
    few_shots=load_example_bank("s2-intent", shot_count=3),
    cot_mode="short",  # short | full | hidden
    persona=load_persona("s2_cs"),
)
messages = [{"role": "system", "content": system}, {"role": "user", "content": utterance}]
Java · Spring PromptTemplate
java 
@Value("classpath:prompts/s2-intent.md")
private Resource promptResource;

public Prompt buildS2Prompt(String utterance, List<Example> shots, String cotMode) {
  PromptTemplate template = new PromptTemplate(promptResource);
  Map<String, Object> vars = Map.of(
      "few_shots", formatShots(shots),
      "cot_mode", cotMode,
      "persona", personaService.load("s2_cs")
  );
  return new Prompt(List.of(
      new SystemMessage(template.render(vars)),
      new UserMessage(utterance)
  ));
}

不推荐
  • Python 与 Java 各写一套不同 few-shot 文案
  • 示例写死在代码里，产品无法 PR 评审
  • 改 shot 数量不上评测直接全量
推荐
  • MD/YAML 真源 + dual template 渲染
  • shot_count、cot_mode 配置化，可 A/B
  • 双栈共用 example-bank 路径与 semver tag

本章结论

  • 示例选择：维护 example bank，优先时效与格式一致；3～5 shot 为常见起点，S2 意图 / S5 结构受益最大。
  • 思维链：CoT 用结构化 步骤 换可审计性；S1 复杂比对可用，S2 高 QPS 宜短链或隐藏。
  • 成本权衡：few-shot + CoT 叠加会显著涨 Token；按场景 tradeoff，并为 CoT 预算 设监控与降级。
  • 双栈模板：MD 真源 + Jinja / PromptTemplate 双渲染，避免 Python 与 Java Prompt 分叉。

判断练习

  1. 找错：团队方案 — ① 从生产日志复制 30 条 few-shot ② 全站默认 full CoT ③ example bank 无 Git 版本 ④ Python/Java 各维护不同示例。哪两条最违反本章原则？

    参考答案

    答：③ 与 ④。无版本无法回归与审计；双栈分叉必然 A/B 与指标不可比。① 质量风险高但可脱敏后入库；② 违反 成本权衡，高 QPS 需降级而非默认 full CoT。

  2. 权衡：S5 长文生成准确率不足，产品要求「加 10 条 few-shot + full CoT」。窗口已常满，你如何回应？

    参考答案

    优先减检索/历史或换小模型做规划链；few-shot 保留 2～3 条最贴近结构的示例；CoT 用于规划阶段隐藏执行，最终只返回正文。用评测集对比 Token 与质量后再放量，而非一次性 +10 shot。
