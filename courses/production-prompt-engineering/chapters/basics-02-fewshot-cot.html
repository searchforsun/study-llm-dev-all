示例选择

few-shot 通过示范输入输出形状，降低模型对 CorpAssist 工单分类、拒答话术的「格式猜测」。生产里示例不是越多越好，而要代表边界：正常路径、拒答、缺 Tool、多实体歧义——让模型在未见过的输入上也能对齐 JSON schema。

CorpAssist · 工单分类 员工在 IM 里描述问题，编排层需将自然语言映射为 ticket_type（咨询 / 投诉 / 审批 / HR）。选 3 条 few-shot 分别覆盖「模糊描述」「跨部门扯皮」「明确单号」三类，避免模型只会猜「咨询」。
原则说明CorpAssist 反例
多样性 覆盖主要意图簇与边界分支 5 条全是「查订单 SO-xxx」
新鲜度 字段与当前 OpenAPI schema 同步 示例仍含已废弃 legacy_id
匿名化 脱敏真实 PII，用合成单号 示例带真实手机号或工号
长度 单示例 < 400 tokens，只示范形状 贴整页退换货政策 PDF

example selection 可静态维护在 Git 管理的 YAML，也可按意图向量检索 Top-K 动态示例（与 RAG 事实索引物理隔离，避免示例污染检索层）。动态检索时设上限（如 Top3），防止 token 失控。

CorpAssist 工单分类 few-shot YAML

yaml 
few_shot_examples:
  - input: "客户说发货慢，要投诉，单号 SO-2026-8842"
    output: |
      {"ticket_type":"complaint","order_id":"SO-2026-8842","need_tool":true}
  - input: "帮我看看年假还剩几天"
    output: |
      {"ticket_type":"hr_query","escalate":false,"need_tool":true}
  - input: "没有单号，只说很急，要找人"
    output: |
      {"ticket_type":"unknown","reply":"请提供订单号 SO- 开头或说明具体事项"}
不推荐
  • 从生产日志直接复制含 PII 的对话做示例
  • 示例 output 字段与线上 JSON schema 不一致
  • 示例库与 RAG 知识库共用同一向量索引
推荐
  • 2–3 条精选示例 + 强 System 边界
  • 示例版本与 prompts@tag 一并发布
  • badcase 修复后回流为候选示例，经 PR 审核入库
先想 10 秒：为何示例要与评测 badcase 同源维护？

badcase 反映真实失败分布；把修复后的对话抽成 few-shot，可形成「评测—示例」飞轮。若示例与评测脱节，会出现「实验室 99%、线上仍翻车」——示例教的是理想分布而非生产分布。

思维链

chain of thought（CoT）要求模型先输出推理步骤再给出结论，适合 CorpAssist 多条件判断：是否越权、是否必须调 ERP Tool、是否触发拒答。对外部员工可见回复时，应使用 hidden cot 或结构化 reasoning 字段，仅审计日志留存。

CorpAssist · 审批前风险判断 员工问：「帮我把这笔 8 万的采购直接批了。」模型需先判断：是否有审批权限、金额是否超阈值、是否需二次确认——这些步骤不应原样出现在 IM 气泡里。
CoT 与对外回复分离
flowchart LR
    User[User 问题] --> CoT[内部 reasoning 3 步]
    CoT --> Decision[工具/拒答/升级决策]
    Decision --> Answer[用户可见 answer]
    CoT -.->|仅日志| Audit[审计 trace]

推理链不直接进入客服气泡，除非产品明确要求展示思考过程。

Python：分步推理 Prompt 后缀

python 
COT_SUFFIX = """
在回答前先在 <reasoning> 标签内用恰好 3 步分析：
1) 用户意图与涉及数据域
2) 是否必须调用工具（订单/HR/审批）
3) 合规风险（越权、PII、需升级）
再输出 <answer> 给用户，answer 中不得出现 reasoning 原文。"""

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM + COT_SUFFIX),
    ("human", "{question}"),
])

Java：Spring AI 结构化推理字段

java 
public record TicketDecision(
    @JsonProperty("reasoning") String reasoning,
    @JsonProperty("ticket_type") String ticketType,
    @JsonProperty("need_order_id") boolean needOrderId
) {}

// 网关 ResponseFilter：剥离 reasoning，仅下发 ticketType + reply

隐藏 CoT 在 API 网关层剥离 <reasoning> 或 JSON 中的 reasoning 字段，防止泄露内部规则与风控逻辑；与安全课「输出过滤」形成闭环。步数必须有硬上限（如 3 步），避免模型「想太多」挤占输出预算。

不推荐
  • 对用户气泡展示完整 CoT 与内部风控理由
  • 无步数上限，推理冗长导致延迟与爆窗
  • 用 CoT 替代 Tool 查询事实（如订单状态）
推荐
  • 固定 3 步模板，与评测集断言步数结构
  • reasoning 写入 trace，供排障与合规审计
  • 事实性问题走 Tool，CoT 只做决策分支
检查理解：何时不该上 CoT？

单意图、格式已由强 System + JSON schema 约束的场景（如固定字段抽取）通常 zero-shot 即可。CoT 适合多条件分支；滥用会增加 output tokens 且可能放大幻觉式「推理」。

成本权衡

每增加 1 条 few-shot 或拉长 CoT，都会线性增加 prompt token cost 与 output 延迟。CorpAssist 日调用 50 万次时，示例从 3 条涨到 8 条可能每月多数千美元，P95 延迟也会上升——必须用数据驱动增减。

CorpAssist · 成本复盘会 工单分类场景准确率 94.2% → 94.6%，但 P50 input tokens 从 1.1K 涨到 1.9K。产品问：是否值得为 0.4% 多花 40% token？本章教你用仪表盘回答这类问题。
策略收益典型成本适用场景
Zero-shot + 强 System 最低 tokens 基线 稳定单意图、schema 简单
Few-shot 3 条 JSON 格式稳定 +600~1200 input tokens 分类、结构化抽取
CoT 固定 3 步 复杂决策更准 +200~800 output tokens 审批、越权判断
动态示例检索 Top3 按意图补示例 检索延迟 + 波动 tokens 意图种类 > 20 的长尾

用 cost dashboard 按场景查看 P50/P95 tokens、单次成本与准确率曲线。经验止损线：当准确率边际收益 < 0.5% 且 tokens 增幅 > 15% 时，回滚示例条数或缩短 CoT。完整 A/B 流程见 A/B 与迭代流程。

成本估算速算（Python 伪代码）

python 
def monthly_cost(calls: int, tokens_per_call: int, price_per_1m: float) -> float:
    return calls * 30 * tokens_per_call / 1_000_000 * price_per_1m

# 例：50万/日，+400 tokens/次，input $2/1M → 月增约 $1,200
常见误区
  • 用 CoT 查事实 — 订单状态、库存应走 Tool，CoT 不能替代 API。
  • 示例与线上一致性 — 字段改名未更新示例，模型会「学会错误格式」。
  • 只看准确率不看 tokens — 微小提升可能不值得线性成本。
成本护栏：什么情况下应回滚 few-shot？

当 A/B 显示准确率提升 < 0.5% 且 tokens +15% 以上，或 P95 延迟超 SLO 时，优先减示例条数或改动态 Top-K，而非继续堆示例。

双栈模板

dual template 指同一份 prompts/ticket_classify.yaml 被 LangChain 与 Spring AI 加载，差异仅在占位符语法与 Message 类型适配层。CorpAssist 双轨（Python Agent 与 Java 微服务）必须共享示例库，否则同一工单在两端分类结果不一致。

CorpAssist · 双轨发布 Python 侧客服 Agent 与 Java 侧工单 API 同时消费 ticket_classify.yaml。CI 在 merge 前渲染双栈 message 并比对 hash，不一致则阻断上线。
双栈 YAML 同源加载
flowchart TB
    YAML[prompts/ticket_classify.yaml]
    YAML --> PY[LangChain FewShotTemplate]
    YAML --> JV[Spring AI ResourceLoader]
    PY --> Hash[message hash]
    JV --> Hash
    Hash --> CI{CI 一致?}

适配层可不同，渲染后的语义消息列表必须一致。

python 
from langchain_core.prompts import FewShotChatMessagePromptTemplate

examples = load_yaml("prompts/ticket_classify.yaml")["few_shot_examples"]
few_shot = FewShotChatMessagePromptTemplate(
    example_prompt=example_prompt,
    examples=examples,
)
chain = few_shot | chat_model
java 
@Value("classpath:prompts/ticket_classify.yaml")
Resource promptResource;

List<Message> fewShotMessages = yamlParser
    .toFewShotMessages(promptResource);
Prompt prompt = new Prompt(Stream.concat(
    fewShotMessages.stream(),
    Stream.of(new UserMessage(userText))
).toList());

发布时在请求 trace 中记录 template_id、prompts@tag 与渲染后 message 条数，便于与 dual stack prompt 其他场景对齐排障。

动态示例检索如何与双栈 YAML 共存？
静态 YAML 保底 2–3 条边界示例；动态检索作为增量，检索结果写入独立 examples_dynamic 块。双栈共用同一检索服务与 Top-K 配置，避免 Python 用 5 条、Java 用 3 条。

本章结论

Few-shot 与 CoT 是格式与复杂决策的杠杆，但必须版本化、隐藏推理、并用成本仪表盘约束膨胀。

  • 示例选择：2–3 条覆盖边界，与 badcase 同源，与 RAG 索引隔离。
  • 思维链：固定步数、内部 reasoning，用户只见 answer。
  • 成本权衡：边际收益 < 0.5% 且 tokens +15% 时止损。
  • 双栈模板：YAML 同源 + CI message hash。
