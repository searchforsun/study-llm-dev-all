工具描述规范

CorpAssist · 订单查询 + 工单创建 Agent 持有 query_order 与 create_ticket 两个工具。员工说「订单 8842 没发货，帮我提 IT 工单」——模型须先查订单再建单，且 ticket 的 category 只能取枚举值。

工具描述规范 要求：name 动词短语、description 说明何时调用、parameters 每个字段有 type + description + enum/required。

检查项合格不合格
description 首句「当用户询问订单状态或物流时调用」「查询订单」
参数说明order_id: 数字订单号，不含 # 前缀order_id: string
互斥工具注明「不要用于创建工单」（无边界说明）
副作用create_ticket 标注 writes=true未标注写操作

工具 JSON Schema 示例（query_order）

json 
{
  "name": "query_order",
  "description": "查询 CorpAssist 商城订单状态与物流。用户提到订单号、发货、物流时使用。不要用于退款或创建工单。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "纯数字订单号，例如 8842，不要包含 # 或 ORD- 前缀"
      },
      "tenant_id": {
        "type": "string",
        "description": "从会话上下文读取，调用方注入，模型不要猜测"
      }
    },
    "required": ["order_id", "tenant_id"]
  }
}
常见误区
  • 工具过多不分组 — 超过 12 个工具时应按域拆 Agent 或做 retrieval-over-tools。
  • description 与实现不一致 — Prompt 写「支持模糊搜索」但 API 只接受精确 id，必出 hallucinated 参数。

create_ticket 工具 Schema 补全要点

写操作工具须在 description 中声明需要用户明确授权，且 category 使用 enum 限制，防止模型发明 urgent-it 等非法值。

json
{
  "name": "create_ticket",
  "description": "在 ITSM 创建工单。用户明确要求建单、报障时使用。writes=true。",
  "parameters": {
    "type": "object",
    "properties": {
      "category": { "type": "string", "enum": ["network", "hardware", "software", "access", "other"] },
      "summary": { "type": "string", "description": "10-500 字问题摘要" }
    },
    "required": ["category", "summary"]
  }
}

计划步骤

复杂任务需要 计划步骤 Prompt 约束推理格式。CorpAssist 采用「简要计划 → 逐步执行 → 汇总」三阶段，兼容 ReAct 与 Plan-and-Execute。

CorpAssist Agent 计划循环
flowchart LR
  U[用户目标] --> P[Plan: 2-4 步]
  P --> A1[Action: tool_call]
  A1 --> O1[Observation]
  O1 --> A2[Action / Finish]
  A2 --> R[回复用户 + 审计日志]

每轮须输出 structured thought；Finish 前不得跳过未完成的写操作确认。

Planner System Prompt 核心段落

markdown 
## 计划规则
1. 先输出 PLAN（最多 4 步），再逐步执行
2. 每步格式：Thought / Action / Action Input
3. 收到 Observation 后，决定下一步或 Finish
4. 写操作（create_ticket、cancel_order）前必须向用户确认意图（若会话未明确授权）

## 禁止
- 单步调用超过 3 次相同工具且参数相同
- 编造 Observation；仅使用工具返回 JSON
- 在 PLAN 中包含系统未提供的工具
先想 10 秒：Plan 写 8 步细计划 vs 2 步粗计划，哪个更适合生产 Agent？

生产宜 2-4 步粗计划 + 动态调整；过长 Plan 易与工具实际返回脱节，且浪费 token。

ReAct 单步输出示例

text
Thought: 用户提供了订单号 8842，应先查询物流状态。
Action: query_order
Action Input: {"order_id": "8842", "tenant_id": "corp-a"}

Observation: {"status": "shipped", "carrier": "SF", "tracking": "SF123"}

Thought: 已发货，无需建单，向用户说明物流。
Finish: 您的订单 8842 已发货，顺丰单号 SF123……

CorpAssist 审计日志保存 Thought/Action/Observation 全链，供合规与排障；对用户仅展示 Finish 段。

失败恢复

工具调用失败是常态：网络超时、429 限流、业务 404。失败恢复 Prompt 规定重试次数、退避策略与用户可见话术，避免 Agent 陷入死循环。

错误类型Agent 行为用户话术
TIMEOUT同参数重试 1 次，仍失败则降级订单系统响应慢，请稍后重试或联系值班
404 NOT_FOUND禁止重试；询问用户是否输错单号未找到该订单，请核对订单号
403 FORBIDDEN停止工具链；记录审计您无权执行此操作
429 RATE_LIMIT等待 2s 后重试 1 次系统繁忙，正在重试…

失败恢复 Prompt 片段

yaml 
failure_recovery:
  max_same_tool_retries: 1
  on_persistent_failure: |
    向用户说明已尝试的步骤与失败原因（不含内部 stack trace）。
    提供可行动选项：1) 修正参数重试 2) 转人工 3) 创建工单由后台处理
  forbidden:
    - 无限 loop 调用同一工具
    - 向用户暴露 API key、内部 URL
    - 假装工具已成功（伪造 Observation）
Observation 里应否包含完整 API 响应体？

生产应注入精简 JSON（字段白名单 + 截断长文本），完整响应写入 trace；Prompt 中要求模型仅依据 Observation 摘要推理。

429 限流时 Agent 能否换工具绕过？

禁止——应等待退避后重试同一合法工具，或降级告知用户；换工具绕过会导致业务逻辑错误。

用户可见话术与内部 trace 分离

  • 用户消息：简洁、无错误码、提供下一步
  • trace：完整 HTTP status、request_id、重试次数
  • Prompt 明确：Finish 段不得包含内部 URL、API key、stack trace

双栈

CorpAssist Agent 在 Python（LangGraph / LangChain Tools）与 Java（Spring AI ToolCalling）双栈部署时，双栈 Prompt 须共享同一份 YAML 源：System、工具 description、失败恢复规则，仅 adapter 层不同。

python
from langchain_core.tools import tool

@tool(parse_docstring=True)
def query_order(order_id: str, tenant_id: str) -> dict:
    """查询 CorpAssist 商城订单状态与物流。用户提到订单号、发货、物流时使用。"""
    ...
java
@Tool(description = "查询 CorpAssist 商城订单状态与物流。用户提到订单号、发货、物流时使用。")
public OrderStatus queryOrder(
    @ToolParam(description = "纯数字订单号") String orderId,
    @ToolParam(description = "租户 ID，从会话注入") String tenantId) {
  ...
}
应对齐
  • 工具 name / description 原文
  • Planner System Prompt 版本号
  • 失败恢复 max_retries
可差异
  • trace 字段命名（OpenTelemetry span）
  • Tool 框架特有 annotation
  • 会话存储（Redis vs JDBC）

双栈 parity 测试：同一 golden 对话脚本，比对 tool_call 序列与最终用户可见回复（允许措辞微差，不允许工具名或参数不同）。

本章结论

S2 Agent Prompt 把「工具 API」翻译为「模型可正确调用的语言」，并约束计划与失败行为。

  • 工具描述规范：description 写清何时调用、参数边界与副作用，与 OpenAI Function Schema 对齐。
  • 计划步骤：2-4 步 PLAN + ReAct 循环，写操作前须确认授权。
  • 失败恢复：按错误类型区分重试/降级/拒答，禁止伪造 Observation。
  • 双栈：YAML 单源 Prompt + 双栈 adapter，golden 脚本做 parity 回归。

动手练习

操作步骤

  1. 对照正文工具 Schema 表，在 tool-desc-checklist.md 为 create_ticket 填写 description 与 category enum。
  2. 写出含 PLAN 规则的 System Prompt 片段（至少 4 条禁止项）。
  3. 为 TIMEOUT 与 404 各写一条用户可见话术，写入 failure_recovery 配置。
  4. 运行 python check_tool_desc.py，输出 PASS: 2 tools validated。

判断练习

  1. （找错）下列工具 description 哪条最可能导致错调？「A) 查询订单 B) 处理用户问题 C) 当用户询问订单状态或物流时调用，不要用于工单 D) CRM 工具」

    参考答案

    B — 「处理用户问题」过宽，模型会对几乎所有意图调用该工具；C 才是合格边界描述。
