为什么要学本章

Agent 的「智能」一半在模型，一半在工具契约与 Prompt 编排。模糊的工具描述会导致错调 API；无 plan steps 约束易循环调用；无 failure recovery 会在 ERP 超时后 hallucinate 成功。本章把 agent prompt 写成可版本化的生产模板。

  • 工具描述 = API 文档：参数类型、必填、示例、失败码写清楚。
  • 先规划后行动：复杂任务要求先输出步骤再调工具。

计划步骤

多步任务（查单 → 发邮件 → 建工单）需在 agent prompt 中要求显式计划：plan steps 可以是 ReAct 的 Thought，或单独 Plan: 块。限制 max_steps 并在 system 写明「完成目标后立即 Final Answer」。

Agent 计划与工具循环（ReAct）
flowchart TD
  U[用户目标] --> P[Plan: 分解步骤]
  P --> T[Thought]
  T --> A[Action + tool]
  A --> O[Observation]
  O --> D{目标完成或达 max_steps?}
  D -->|否| T
  D -->|是| F[Final Answer]
  O --> E{工具错误?}
  E -->|是| R[failure-recovery / retry-hint]
markdown 
## plan-steps 片段（写入 agent-system）
复杂任务流程：
1. 用一句话复述用户目标
2. 列出 2～5 步计划（Plan:）
3. 逐步执行；每步一次工具调用为宜
4. 信息足够时输出 Final Answer，勿继续调工具

## react-format 示例
Thought: 需先查 PO 状态再决定是否发邮件
Action: query_erp_order
Action Input: {"po_number": "PO-2024-889"}

失败恢复

CorpAssist · ERP 超时 智能客服 AgentS3行业说法Conversational Agent · Customer Service Bot
多轮对话 + 工具查单/改状态；强调话术与失败降级 query_erp_order 返回 504。客服 Agent 不应回答「您的订单已发货」，而应触发 failure recovery：告知延迟、建议 retry 或转人工。

retry hint 写在 system：429/503 可间隔重试 1 次；404 禁止重试；参数校验失败应修正 JSON 而非换工具。Observation 模板应结构化：{"ok": false, "code": "504", "hint": "..."}。

坏 Prompt
  • 工具失败仍要求「给出最终确认」
  • 无 retry 上限，循环 10 次
  • Observation 纯 HTML 错误页
好 Prompt
  • 按错误码分支：404/504/429 策略不同
  • 超过 2 步失败 → 转人工话术
  • retry-hint 与网关超时配置一致

双栈

dual agent：Python（LangChain agent / 实验）与 Spring（agent-runtime 生产）共享同一份 agent-system-v*.md 与工具描述 YAML。Python 侧快速迭代 react format；Spring 侧强制 max_steps、租户工具白名单与审计。
