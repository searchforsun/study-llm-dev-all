为什么要学本章

高级 LLM 应用岗面试很少让你背「什么是 Few-shot」，更常给一条生产 badcase：cite 丢失、拒答飙升、JSON 破 schema——要求你在 15 分钟内讲清根因、改写方案与如何验证。prompt debug interview 本章把 CorpAssist 真实排障路径压缩成可复述的 interview defense template。

学完你能
  • 用 badcase analysis 框架定位 Prompt / 检索 / 工具三类根因
  • 选择 prompt rewrite 与 ablation strategy 验证假设
  • 用 metric delta 叙事证明改动有效且无副作用
  • 套用 interview defense template 完成 TOP5 场景答辩
本章先记住 4 件事
  • 先现象后假设：贴 trace、用户句、模型输出，再谈 Prompt。
  • 单变量改写：一次只改 System / User / 示例 / 检索 之一。
  • 指标要对照：改前改后 + 回归集 + guard 指标联看。
  • 答不完也要闭环：根因 → 改写 → 验证 → 回滚预案。

坏例分析

CorpAssist · 面试题 S1 面试官：「上线 rag-prompt-v2.3 后 cite 覆盖率从 0.82 降到 0.61，用户投诉『答案没出处』。给你 trace_id，你怎么查？」——期望你展示结构化 badcase analysis，而非猜测「模型变差了」。

坏例分析的第一步是冻结上下文：prompt_version、检索 top-k、chunk 文本、完整 messages 数组、模型输出。CorpAssist 在 flywheel badcase 平台归档上述字段，并与 root cause prompt 分类标签绑定。

根因类别典型信号CorpAssist 检查点非 Prompt 因素
Prompt 指令 模型不输出 [doc_id] 格式 System cite 段是否被截断 —
检索 CONTEXT 空或 chunk 无 metadata retriever score、chunk_id 索引延迟
组装 CONTEXT 在 System 之后被覆盖 message 顺序、token 裁剪 网关限长
模型/参数 全版本同步下降 temperature、模型路由 供应商降级
Badcase 分析决策流
flowchart TD
  Start[收到 badcase] --> Trace[拉 trace + messages]
  Trace --> Q1{CONTEXT 非空?}
  Q1 -->|否| Ret[查检索/索引]
  Q1 -->|是| Q2{cite 指令在 System?}
  Q2 -->|否| Asm[查组装/裁剪]
  Q2 -->|是| Q3{输出格式违规?}

先排除检索与组装，再动 Prompt；避免「盲改 System」。

badcase 归档 JSONL 字段

json 
{
  "case_id": "BC-RAG-0412",
  "prompt_version": "rag-prompt-v2.3",
  "symptom": "cite_coverage_drop",
  "user_query": "年假如何计算？",
  "root_cause_tag": "prompt_instruction",
  "trace_id": "otel-8f3a..."
}

root cause prompt 是内部分析用模板：让 LLM 根据 trace 输出结构化假设列表（仅供工程师参考，不自动改生产 Prompt）。与自动修复不同，分析 Prompt 要求引用 message 序号作为证据。

面试扣分
  • 不谈 trace，直接说「加强 Prompt」
  • 把检索问题全归咎于模型
  • 无法复述用户原句与错误输出
加分
  • 按表格四类根因逐一排除
  • 引用 prompt_version 与 diff
  • 提出可执行的下一步实验
先想 10 秒：cite 下降但 faithfulness 仍高，最可疑哪一层？

模型回答仍「像」有据，但未输出规定 cite token——优先查 Prompt 指令是否被 v2.3 改动或 token 裁剪截断 System 中的 cite 格式段，而非检索召回本身。

改写策略

确认根因落在 Prompt 层后，prompt rewrite 须遵守单变量与可 diff：每次 MR 只改一类元素，并在 prompt-diff.txt 标注假设。

症状改写杠杆示例改动风险
无 citeSystem 格式 + Few-shot增加「每句末必须 [doc_id:idx]」拒答率上升
过度拒答拒答分支条件低置信阈值 0.35→0.28幻觉上升
JSON 破 schemaschema 嵌入 + 修复循环additionalProperties:false延迟
注入泄露边界声明 + 不可信区boundary_instruction 加强误杀
语调漂移style_profile / tone补禁用词与示例多语言不一致

ablation strategy 用于面试与实验：依次去掉 Few-shot、缩短 System、替换检索 chunk，观察哪一步指标恢复，从而证明根因。

  • 对 badcase 集 + 回归集跑 eval
  • 对比 cite_coverage、refusal_rate
  • CI 门禁通过再合入
  • 1% 流量 shadow 新 Prompt
  • 不暴露用户，只记指标
  • 48h 无劣化再放大

cite 修复 diff 示例

diff 
--- rag-prompt-v2.3
+++ rag-prompt-v2.4
@@ System @@
- 请依据 CONTEXT 作答。
+ 请依据 CONTEXT 作答。每一 factual 句末必须附 [doc_id:chunk_idx]。
+ Single variable: cite format only.

改写须联动 A/B 与迭代流程 的 single_variable 规则；禁止在同一 MR 里同时改 cite 与拒答阈值，否则 metric delta 无法归因。

面试中被问「你会加更多 Few-shot 吗？」
先答「取决于 ablation」：若去掉 Few-shot 后 cite 仍差，则非示例问题；若恢复示例即恢复，再讨论示例质量与 token 成本，展示工程化思维而非套路。

指标变化

面试官要听的不是「指标好了」，而是改什么、升什么、降什么、如何解释 trade-off。metric delta 叙事格式：Δmetric = after − before，附样本量与置信说明。

指标v2.3→v2.4 Δ解释联看指标
cite_coverage+0.19 (0.61→0.80)cite 格式句生效faithfulness
refusal_rate+0.02格式严导致边缘拒答hallucination_rate
hallucination_rate−0.01cite 约束抑制编造user_csat
p95_latency+80ms输出略长token_out
指标联看（拒答 vs 幻觉）
flowchart LR
  Change[Prompt 改写] --> Cite[cite_coverage ↑]
  Change --> Ref[refusal_rate ↑]
  Cite --> Good[用户信任 ↑]
  Ref --> Risk[覆盖下降风险]
  Risk --> Hall[须联看 hallucination_rate]
  Hall --> Gate{发布闸门}
  Gate -->|通过| Ship[全量]
  Gate -->|fail| Roll[回滚 v2.3]

单一指标改善不足以上线；与 发布闸门 一致。

答辩时准备一张「before/after 小表」即可，口头补充：回归集 200 条全绿、badcase BC-RAG-0412 等 12 条已修复。引用 flywheel badcase 说明持续迭代，而非一次性补丁。

红线表述（避免）
  • 「准确率 100%」——无评测口径
  • 「用户都说好了」——无 CSAT 样本
  • 只报 cite 升、不提 refusal 边际变化

答题模板

interview defense template 将 15 分钟答辩切成四段，每段 2～4 分钟，确保即使被打断也能独立成块。

段时长内容CorpAssist 话术锚点
1 现象2 min谁投诉、什么指标、哪版 Promptrag-prompt-v2.3、cite 0.61
2 分析4 mintrace 排除检索/组装BC-RAG-0412、root_cause_tag
3 改写4 min单变量 diff + ablationv2.4 cite 句、golden 回归
4 验证3 minmetric delta + 回滚cite +0.19、refusal +0.02、rollback tag

STAR-P 模板（Prompt 版）

markdown 
**S**ituation：CorpAssist S1 上线 v2.3 后 cite_coverage 降至 0.61。
**T**ask：在不升高幻觉的前提下恢复 cite，48h 内 hotfix。
**A**ction：trace 排除检索；System 增加 cite 格式句（单变量）；golden + 12 badcase 回归。
**R**esult：cite 0.80（+0.19），refusal +0.02，已 tag v2.4.1。
**P**rompt：附 prompt-diff 与 eval 报告路径。

TOP5 面试场景（S1～S5）与 NS 能力对齐：prompt debug interview 题库应覆盖 RAG cite、Agent 工具死循环、JSON schema、注入拒答、S5 去重——每题各备一条 STAR-P。

弱答（泛泛）

text 
我们优化了 Prompt，加了示例，效果好了很多。

强答（可验证）

text 
v2.3→v2.4 仅改 System cite 句；golden cite_coverage +0.19，refusal +0.02 在闸门内；BC-RAG-0412 等 12 条 badcase 关闭；回滚点 git tag rag-prompt-v2.3.2。

与 实战：CorpAssist Prompt 套件 的 graduation checklist 衔接：答辩材料 = eval manifest + 一条完整 badcase 闭环故事。

本章结论

  • 坏例分析：冻结 trace，按 Prompt/检索/组装/模型四类排除根因。
  • 改写策略：单变量 diff + ablation，离线 golden 再在线 shadow。
  • 指标变化：metric delta 联看 refusal 与 hallucination，避免片面报喜。
  • 答题模板：STAR-P 四段式，现象→分析→改写→验证可独立成块。
