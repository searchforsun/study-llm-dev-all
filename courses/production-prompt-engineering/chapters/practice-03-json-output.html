本章先记住 4 件事
  • Schema 进 Prompt：字段名、类型、enum 与后端 DTO 一致。
  • 只输出 JSON：禁止 markdown 围栏外的解释（或约定 raw_json 字段）。

  • 工单场景：intent、priority、summary 等字段有业务校验，不只 JSON 语法。

JSON schema

CorpAssist · 对话转工单 客服对话结束需生成 ticket json 写入 ITSM。字段：intent、priority、summary、order_id（可选）。Schema 必须与 ticket-service DTO 一致，否则 400。
json 
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "CorpAssistTicket",
  "type": "object",
  "required": ["intent", "priority", "summary"],
  "properties": {
    "intent": { "type": "string", "enum": ["refund", "shipping", "account", "other"] },
    "priority": { "type": "string", "enum": ["P1", "P2", "P3"] },
    "summary": { "type": "string", "maxLength": 500 },
    "order_id": { "type": "string", "pattern": "^PO-\\d{4}-\\d+$" }
  },
  "additionalProperties": false
}

修复循环

repair loop：parse → validate 失败 → 将错误信息作为 user 消息追加 → 再生成。配合 parse retry 上限（通常 2～3 次），避免无限循环与 Token 爆炸。

结构化输出 repair-loop
flowchart TD
  P[json-schema-prompt] --> G[LLM 生成]
  G --> X{JSON.parse OK?}
  X -->|否| E[提取错误行/列]
  E --> R[repair user: Fix JSON: ...]
  R --> G
  X -->|是| V[JSON Schema / Bean 校验]
  V -->|失败| R
  V -->|通过| OK[写入 ticket-service]
  G --> L{retry > max?}
  L -->|是| F[降级: 人工队列]
python 
def generate_ticket_with_repair(llm, prompt: str, max_retries: int = 2) -> dict:
    messages = [{"role": "user", "content": prompt}]
    for attempt in range(max_retries + 1):
        raw = llm.invoke(messages).content
        try:
            data = json.loads(strip_markdown_fence(raw))
            jsonschema.validate(data, TICKET_SCHEMA)
            return data
        except (json.JSONDecodeError, jsonschema.ValidationError) as e:
            if attempt == max_retries:
                raise
            messages.append({"role": "assistant", "content": raw})
            messages.append({"role": "user", "content": f"Fix JSON. Error: {e}"})
    raise RuntimeError("unreachable")
java 
// Spring · repair-loop 伪代码
for (int attempt = 0; attempt <= MAX_REPAIR; attempt++) {
    String raw = chatClient.prompt(prompt).call().content();
    try {
        CorpAssistTicket ticket = converter.convert(raw);
        businessValidator.validate(ticket); // output-validate 语义层
        return ticket;
    } catch (ConversionException | ValidationException ex) {
        if (attempt == MAX_REPAIR) throw ex;
        prompt = prompt + "\nFix JSON: " + ex.getMessage();
    }
}
throw new TicketParseException("repair exhausted");
CorpAssist · 工单 priority 枚举漂移 模型输出 "priority": "high" 而非 P1/P2/P3。repair user 消息示例：「priority 必须是 enum [P1,P2,P3]；high 无效。」第二次生成通过 schema，parse retry 计 1 次。

校验

output validate 分两层：语法（JSON.parse）、语义（Schema + 业务规则，如 P1 必须含 order_id）。Prompt 层写清 enum；代码层不信任模型，必须校验后再调用 ITSM。

校验项Prompt 侧代码侧
enumschema 列举jsonschema / @NotNull
patternPO 单号格式示例regex 二次验证
长度maxLength in schema截断或 repair
先想 10 秒：只用 OpenAI JSON mode，能否省略 repair-loop？

不能省略校验。JSON mode 提高语法正确率，但 enum/pattern/业务规则仍可能违例；生产必须 output validate + 有上限的 repair loop，失败降级人工。

方式优点局限CorpAssist 建议
Prompt-only schema-inject模型无关、可审计 diff围栏/enum 仍可能错默认路径，配合 repair
原生 response_format语法正确率高enum/业务规则不保证可选叠加，不替代 validate
BeanOutputConverter / with_structured_output与 DTO 绑定紧双栈需同源 schemaJava/Python 共用 ticket-schema.json

# ticket-service 业务规则（output-validate 语义层）
rules:

python 
# strip_markdown_fence · repair-loop 前置
import re
def strip_markdown_fence(raw: str) -> str:
    m = re.search(r"```(?:json)?\s*([\s\S]*?)```", raw)
    return m.group(1).strip() if m else raw.strip()

本章结论

  • JSON schema：schema inject 与 DTO 同源；禁止 additionalProperties 漂移。
  • 修复循环：repair loop + parse retry 上限；失败降级人工。
  • 校验：output validate 语法+语义；不信任裸模型输出。
  • CorpAssist 工单：ticket json 模板版本化，对接 ITSM。

判断练习

  1. repair-loop 设 max_retries=10 以追求 100% 解析成功，合理吗？

    参考答案

    不合理。应设 2～3 次上限并监控延迟与成本；仍失败则降级人工队列，避免无限循环与 Token 爆炸。

  2. Schema 写在 Prompt 里，Java DTO 可以不同步更新吗？

    参考答案

    否。schema inject 应与 OpenAPI/DTO 同源生成，否则模型输出与下游契约漂移。

  3. 模型输出合法 JSON 但 intent 为 refund、summary 仅 5 字，能否直接 POST ticket-service？

    参考答案

    不能。语法通过不等于语义通过；output validate 业务层要求 refund 场景 summary ≥ 20 字。应触发 repair 或降级人工，不可裸 POST。
