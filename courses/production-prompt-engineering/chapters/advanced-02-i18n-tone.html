本章先记住 4 件事
  • 语调是契约：产品定义的 style guide 须落到 Prompt 变量，而非靠模型「自觉」。
  • 合规先于流畅：金融/医疗类场景免责声明不可省略。
  • 多语言要分层：检测 locale → 选模板 → fallback 链，避免混语输出。
  • 高风险走人工：模型输出 + 规则引擎 + 人工复核三级闸。

品牌语调

CorpAssist · 集团统一客服 persona 华东 BU 希望回答「报销流程」时更口语，华北法务要求对外一律「您」且禁止 emoji。若没有 brand voice 矩阵，双栈各写一套 System，A/B 后投诉率上升 18%。

品牌语调不是「写活泼一点」，而是可测试的 Prompt 参数：人称、句式长度、是否使用术语表、禁止词列表。CorpAssist 在 style guide YAML 中维护全局默认值，各 tenant 可覆盖 tone_profile。

维度档位Prompt 变量CorpAssist 示例
正式度formal / neutral / casual{formality}对外客服默认 formal
人称您 / 你{address}B2B 场景强制「您」
长度brief / standard / detailed{verbosity}移动端 brief
术语glossary_strict / explain{term_mode}制度库引用 glossary_strict

tone matrix 把「场景 × 渠道 × 角色」映射到上表组合。例如 channel=wecom_external + intent=refund → formal + brief + 禁用承诺性用语。

CorpAssist 语调选择流程
flowchart TD
  Req[用户请求] --> Detect[locale + tenant 检测]
  Detect --> Matrix[tone_matrix 查表]
  Matrix --> Merge[合并 style_guide 默认值]
  Merge --> Sys[System Prompt 注入 tone 块]
  Sys --> LLM[LLM 生成]
  LLM --> Lint[style_linter 规则]
  Lint -->|pass| Out[输出]
  Lint -->|fail| Retry[降档重写或拒答]

语调在组装 System 前确定；生成后仍要经过 style_linter，防止模型偏离矩阵。

Spring AI：语调 Advisor 注入

java 
@Component
public class ToneMatrixAdvisor implements CallAroundAdvisor {
  private final ToneMatrixService matrix;

  @Override
  public AdvisedResponse aroundCall(AdvisedRequest req, CallAroundAdvisorChain chain) {
    ToneProfile profile = matrix.resolve(req.getContext().getTenantId(), req.getUserText());
    String toneBlock = templateRenderer.render("tone-block.st", profile);
    req = req.mutate()
        .systemText(req.getSystemText() + "\n" + toneBlock)
        .build();
    return chain.nextAroundCall(req);
  }
}

LangChain：Runnable 合并 style_guide

python 
def inject_tone(payload: dict) -> dict:
    profile = tone_matrix.resolve(
        tenant_id=payload["tenant_id"],
        channel=payload.get("channel", "web"),
        intent=payload.get("intent"),
    )
    payload["system"] = (
        payload["system"]
        + render_tone_block(profile, style_guide.load(payload["tenant_id"]))
    )
    return payload

chain = RunnableLambda(inject_tone) | prompt | llm | style_linter
反例
  • 在 User 里写「请用正式语气」——易被后续 User 覆盖
  • 每个 BU 复制整份 System，版本漂移
  • 只靠 post-hoc 人工改稿，无自动化 lint
推荐
  • tone 块作为 System 子段，带 priority=high 标记
  • 矩阵 + 租户 override，单源 style_guide
  • golden set 测 formality_score / banned_word_hit
先想 10 秒：为什么语调要进 System 而不是 User？

User 区在 CorpAssist 被标为不可信；攻击者或误操作可能在 User 中要求「改用随意语气」。System + Advisor 注入的 tone 块位于可信区，与注入防护章节的边界模型一致。

合规话术

合规不仅是法务审文案，而是 Prompt 里的强制片段：compliance phrasing 定义何时必须附加 regulated disclaimer、何时必须拒答或转人工。

风险等级触发条件Prompt 动作输出要求
L0 一般FAQ、操作指引标准回答无额外 disclaimer
L1 提示涉及费用/时效追加「以实际政策为准」句末固定 disclaimer_id
L2 监管投资/医疗/个人数据模板 + 禁止承诺收益regulated_disclaimer 全文
L3 禁答法律意见、未授权产品拒答 + 转人工仅输出 refuse_template

compliance_phrases.yaml 片段

yaml 
regulated_disclaimer:
  finance_v3:
    zh: "以上内容仅供参考，不构成投资建议。具体以持牌机构说明为准。"
    en: "For informational purposes only. Not investment advice."
  health_v2:
    zh: "不能替代专业医疗诊断，请遵医嘱。"

compliance_rules:
  - id: CR-FIN-01
    match_intents: [investment, yield]
    level: L2
    append: finance_v3
    forbid_patterns: ["保证收益", "稳赚"]

与 安全与合规工程security-compliance-engineering 的数据分级对齐：L2/L3 回答须写入审计日志，字段含 disclaimer_id、risk_level。

模型已经「说得像合规」，还要强制 disclaimer 吗？
要。LLM 输出不稳定，监管检查看的是可证明的模板命中，不是主观像不像。CI 应用 regex / structured check 验证 disclaimer_id 是否出现。
多语言 disclaimer 如何与中文 System 共存？
按检测到的 response_locale 选 disclaimer 子键；System 可保留 bilingual prompt 指令：「正文与免责声明使用同一语言」。
复盘 · 漏写 finance_v3 某次 Prompt 精简时删除了「append disclaimer」步骤，模型仍给出看似谨慎的回答，但监管抽检认定缺少持牌声明。修复：输出 schema 增加必填 disclaimer_ids[]，校验失败走修复循环。

多语言

CorpAssist 三语并存时，常见失败模式是：用户用英文提问、检索到中文 chunk、模型输出中英混杂。需要明确的 locale fallback 链与 bilingual prompt 结构。

策略适用Prompt 要点
mirror用户语言与知识库一致「使用与用户相同的语言回答」
translate-at-edgechunk 仅中文检索后翻译 chunk 再进上下文
fallback_en小语种ja → en → zh 顺序尝试
fixed_locale监管区域欧盟租户强制 en-GB 输出

在 MessageChatMemoryAdvisor 前增加 LocaleDetectAdvisor，将 LocaleContextHolder 写入 prompt 变量 {response_locale}。

用 RunnableLambda 调用 fastText / CLD3，结果写入 payload["locale"]，ChatPromptTemplate 按 locale 选 messages 文件。

locale_fallback 配置

yaml 
locale_policy:
  detect: [user_text, accept_language]
  response_locale: mirror_user
  fallback_chain: [ja, en, zh-CN]
  chunk_locale_mismatch: translate_at_edge
  bilingual_system_header: |
    你将在 {response_locale} 作答。引用制度库时保留原文 cite，解释部分翻译。

评测时增加 locale_consistency 指标：回答语言与用户语言一致且 disclaimer 语言匹配。与 RAG 专用 Prompt 的 cite 格式联调，避免翻译时丢失 chunk_id。

注意
  • 勿在 User 指定语言 — 与注入防护冲突，应使用网关解析的 Accept-Language。
  • fallback 要可观测 — span 标记 locale_fallback_used=true 便于排障。

审核

全自动无法满足 L2/L3 场景。CorpAssist 将低置信、合规命中、用户投诉三类流量导入 human review queue，并与 Prompt 版本绑定可追溯。

人机协同审核队列
flowchart LR
  Gen[模型输出] --> Auto[规则 + LLM-judge]
  Auto -->|pass| Ship[直接下发]
  Auto -->|review| Q[human_review_queue]
  Q --> Agent[人工坐席]
  Agent -->|edit| Pub[发布修正稿]
  Agent -->|reject| Train[回流 badcase + Prompt 迭代]
  Pub --> KB[可选写入 FAQ 候选]

审核结论应回写 trace_id 与 prompt_version，供 PromptOps 下一章迭代。

入队条件优先级SLA
compliance_level >= L2P14h
style_linter 失败 ≥2 次P224h
用户 thumbs_down + 高价值 tenantP14h
locale_consistency 失败P348h

审核工单 JSON（Spring 侧 DTO 示意）

java 
public record ReviewTicket(
    String traceId,
    String promptVersion,
    String draftAnswer,
    List<String> missingDisclaimerIds,
    Locale responseLocale,
    ReviewPriority priority
) {}

人工修改后的「金标准」应答应进入评测 golden set，防止下一版 Prompt 回归。详见 LLM 评测与质量llm-evaluation-quality 的 human-in-the-loop 章节。

审核速查
  • 入队必带 trace_id + prompt_version
  • 人工编辑 ≠ 临时补丁，须回流 eval
  • L3 禁答场景禁止人工「帮模型圆场」

本章结论

  • 品牌语调：tone matrix + style guide 参数化，双栈 Advisor/Runnable 注入 System。
  • 合规话术：L0–L3 分级，regulated disclaimer 模板化且可校验。
  • 多语言：locale 检测、fallback 链与 mirror 策略，避免混语与 cite 丢失。
  • 审核：human review queue 与 trace、Prompt 版本绑定，修正稿回流评测。
