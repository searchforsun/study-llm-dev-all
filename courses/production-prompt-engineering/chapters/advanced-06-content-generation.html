风格与结构 Prompt

CorpAssist · 季度全员信 S5 CEO 办公室要「温暖专业、800 字、钉钉渠道」的季度信。BFF 注入 audience=employee、channel=dingtalk；style structure prompt 从 tone 卡继承「您」称谓，结构固定为：开场共情 → 3 条业务要点 → 行动号召。

style structure prompt 明确：受众、渠道（公众号/邮件）、段落结构（钩子-要点-CTA）。继承 品牌语调 的 tone 卡，并增加结构约束。

markdown 
## 任务
根据 outline 撰写 {{channel}} 文稿。

## 风格（不可偏离）
- 语调：{{tone_persona}}
- 结构：标题 → 3 段要点 → 行动号召
- 禁止：绝对化用语、未证实数据

## 输入
outline: {{outline}}
facts: {{verified_facts}}
坏 S5 Prompt
  • 「写得好一点、活泼一些」无结构
  • 未注入 verified_facts，数据靠模型编
  • 与 S1 共用「必须 chunk 引用」导致文案僵硬
好 S5 Prompt
  • tone 卡 + 段落结构 + max_words
  • 数字/政策来自 facts 块
  • 创意句与事实句分区生成

长文规划与去重

长文采用两阶段：longform plan prompt 生成章节大纲 → 按章生成 → 合并。dedup prompt 在合并后要求模型标注重复句并改写，或用 embedding 相似度 > 0.92 自动重写。

S5 长文 Prompt 链
flowchart LR
  Plan[plan prompt] --> Sec1[section 1]
  Plan --> Sec2[section 2]
  Sec1 --> Merge[merge]
  Sec2 --> Merge
  Merge --> Dedup[dedup prompt]
  Dedup --> Out[终稿]
markdown 
## longform-plan-prompt（节选）
根据 topic 与 max_words 输出 JSON outline：
{ "sections": [{"title":"...", "bullets":["..."]}] }
禁止在 plan 阶段写正文。

## dedup-prompt（节选）
标注重复率>0.9 的句子对，改写后者保留语义、换表述。

质量评估流水线

quality pipeline：生成 → review prompt 打分 → 低于阈值则重写（最多 N 次）。Rubric 维度：事实一致、结构完整、语调、可读性。

阶段Prompt 角色输出
生成style + section草稿
评审review promptJSON 分数 + 修改建议
修复repair 仅改扣分段终稿候选
json 
{
  "factuality": 4.2,
  "structure": 4.5,
  "tone": 4.0,
  "readability": 4.3,
  "pass": true,
  "fix_hints": ["第2段缺少CTA"]
}

review prompt 与 结构化输出 共用 schema 约束；低于阈值时只把 fix_hints 对应段落送 repair，避免全文重写浪费 Token。

内容安全审核

content safety prompt 与规则引擎双轨：模型自检敏感主题 + 外部内容安全 API。命中则拒发或转人工，与 输出 filter 共用配置。

风险类Prompt 层后处理
绝对化广告法forbidden phrase 列表规则引擎扫描
未授权优惠仅允许 verified_facts 中的价格人工审核队列
政治/敏感content-safety 自检指令第三方 API block

个性化参数

personalize var 由 BFF 注入，禁止用户随意覆盖：

  • audience：员工 / 客户 / 高管
  • channel：钉钉 / 邮件 / 官网
  • max_words：控制成本与阅读时长
  • locale：联动 i18n 模板

面试题：S5 与 S1 RAG 的 Prompt 最大差异？

S1 强调仅依据检索、必须引用；S5 强调可控创造性（风格/结构/个性化），事实块用 verified_facts 注入而非自由发挥。二者都要安全审核，但 S5 额外要质量流水线与 dedup。

判断练习

  1. S5 能否像 S1 一样要求「每句必须带 chunk 引用」？

    参考答案

    不能一刀切。创意段落允许生成；事实性陈述应绑定 verified_facts 或素材库引用。结构与 tone 用 Prompt 约束，事实用 RAG 块注入。

  2. 找错：团队把 plan/dedup/review 合成一个超长 System Prompt 以省 Token，问题在哪？

    参考答案

    无法分阶段评测与 A/B；失败难定位是规划还是去重还是评审环节。应拆链路与 quality pipeline 对齐，每步可单独回归。
