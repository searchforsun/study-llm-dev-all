约 45 分钟 · 阶段：生产治理 · 能力：为 CorpAssist S5 内容工作室设计风格化长文 Prompt、去重与质量流水线

为什么要学本章

CorpAssist 除客服 RAG 外，还要支撑市场部「一键出稿」：S5 content studio 场景要求同一套 Prompt 能产出白皮书、公众号推文与内训讲义，且风格不漂移、段落不重复、敏感词可拦截。本章把创作自动化从「写一段 clever prompt」升级为可评测、可回滚的 content QA pipeline。

学完你能
  • 编写 style structure prompt 与 content block template
  • 设计 outline planner 与 section deduplication 策略
  • 搭建可读性、事实性与品牌一致性的 content QA pipeline
  • 配置 content safety review 与 personalization params
本章先记住 5 件事
  • 结构先于文采：先锁 outline，再逐段生成，避免长文跑题。
  • 去重要两层：段内 n-gram + 跨文档 embedding 相似度。
  • QA 流水线可阻断：任一 gate 失败即回炉改写，不直接发布。
  • 安全审核写进 Prompt 与后处理：双保险。
  • 个性化是参数化：tenant / 受众 / 渠道走变量，不改 System 主干。

风格与结构 Prompt

CorpAssist · S5 内容工作室 市场部提交 brief：「面向制造业 CIO 的 3000 字 AI 转型白皮书，正式、数据导向、禁止夸张形容词」。若没有 style structure prompt，模型常输出营销腔 + 无层级长段落，无法通过品牌审核。

创作类 Prompt 与问答类 Prompt 的分野在于：输出是制品（artifact），须同时约束体裁、结构与语调。CorpAssist S5 采用「System 锁风格 + User 给 brief + 中间 JSON outline」三段式。

块职责CorpAssist 键失败症状
Style 层 人称、句长、禁用词、引用规范 style_profile 语调漂移、emoji 泛滥
Structure 层 章节深度、每节字数、必备小节 structure_schema 缺 executive summary
Block 层 可复用段落类型（案例框、数据表） content block template 格式不统一
S5 风格与结构 Prompt 组装
flowchart TB
  Brief[User brief] --> Planner[outline planner]
  Style[style_profile YAML] --> Assembler[Prompt 组装器]
  Struct[structure_schema] --> Assembler
  Blocks[content block templates] --> Assembler
  Planner --> Assembler
  Assembler --> LLM[LLM 逐段生成]
  LLM --> Draft[section drafts]

风格与结构在组装器汇合；brief 只驱动内容主题，不覆盖 style_profile。

style_profile 片段（YAML）

yaml 
s5_whitepaper_v2:
  audience: manufacturing_cio
  tone: formal_data_driven
  sentence_max_chars: 120
  banned_phrases:
    - 颠覆性
    - 遥遥领先
  required_blocks:
    - executive_summary
    - methodology
    - case_study

style structure prompt 与 语气、合规与多语言 的 tone matrix 共享 tone_profile 键，避免 S5 与客服域各维护一套品牌口径。

误区
  • 在 User 里写「写得专业一点」代替 style_profile
  • 一次生成 3000 字全文，无中间 outline
  • 结构要求只口头约定，不进评测集
推荐
  • System 嵌入 style + structure，User 仅 brief 与变量
  • outline → 逐 section 生成 + 合并
  • structure_schema 字段进 golden 校验
先想 10 秒：为何创作 Prompt 要把「结构」与「风格」拆成两个 YAML 块？

结构决定可评测的硬约束（必备章节、字数），风格决定软约束（语调、禁用词）；拆分后 A/B 可只改 style 而不动 outline schema，指标归因更清晰。

长文规划与去重

超过 2k token 的制品若单次生成，常见问题是后半段重复前半段论点，或悄悄引入 brief 外的主题。outline planner 先产出可校验的 JSON 大纲，再按 section_id 并行或串行生成。

策略适用优点风险
串行 section强叙事、依赖前文连贯性好延迟高
并行 section模块化白皮书快需 merge 去重
Map-reduce 摘要超 long brief + 多源控 token丢细节

outline JSON 示例

json 
{
  "doc_id": "wp-ai-transform-2026",
  "sections": [
    {"id": "exec", "title": "Executive Summary", "target_words": 400},
    {"id": "s1", "title": "现状与痛点", "target_words": 800},
    {"id": "s2", "title": "CorpAssist 实践", "target_words": 900}
  ]
}

section deduplication 分两层：段内 检测连续重复 n-gram（如 13-gram 重叠 > 40%）；跨段 用 embedding 余弦相似度，阈值通常 0.85～0.92，命中则触发 rewrite 或 merge。

  • 实现快，适合 catch「同句复制粘贴」
  • 对 paraphrase 重复无效
  • 阈值过高误杀引用块
  • 捕捉语义重复段落
  • 需向量服务与 section 边界
  • 与 RAG chunk 去重可共用索引
复盘 · 白皮书 §2 与 §4 语义重复 v1 outline 未分配「案例」专属 section，模型在「实践」与「路线图」两节各写一遍 CorpAssist 案例。修复：structure_schema 增加 case_study 独占 section + 跨段 dedup gate 阈值降至 0.88。

规划 Prompt 要显式写：「已生成 section 摘要」注入后续 User，避免并行 worker 各写一遍相同论点。与 上下文与混合记忆工程context-memory-engineering 的 working memory 模式同源。

质量评估流水线

内容制品不能仅靠「人眼觉得还行」。content QA pipeline 在发布前串联自动 gate + 抽检，任一失败进入 rewrite 队列或人工审核。

CorpAssist S5 质量评估流水线
flowchart LR
  Draft[section drafts] --> G1[structure gate]
  G1 --> G2[readability gate]
  G2 --> G3[brand voice gate]
  G3 --> G4[factuality spot-check]
  G4 --> G5[dedup gate]
  G5 --> Publish[CMS 发布]
  G1 -->|fail| Rewrite[rewrite prompt]
  G2 -->|fail| Rewrite
  G3 -->|fail| Rewrite
  G4 -->|fail| Human[human review]
  G5 -->|fail| Rewrite

structure 与 dedup 失败可自动 rewrite；factuality 失败默认升人工。

Gate检查项工具 / Prompt阈值示例
structure必备块、字数区间JSON schema validator100% 块存在
readability句长、被动语态比规则 + LLM judgeavg_sentence < 120 字
brand voice禁用词、tone 分类classifier + 词表0 命中 banned
factuality可验证声明 vs 来源RAG cite 校验cite_coverage ≥ 0.7
dedup跨 section 相似度embeddingmax_sim < 0.88

LLM-as-judge Prompt 片段

markdown 
你是 CorpAssist 品牌审核员。仅依据 style_profile 评判下文。
输出 JSON：{"pass": bool, "violations": ["..."]}
禁止补充解释。待评文本：
{{section_text}}

各 gate 结果写入 OTEL span：qa_gate、pass、prompt_version、section_id。与 LLM 评测与质量llm-evaluation-quality 的 threshold 报告格式对齐，便于 S5 与 S1 RAG 共用 dashboard。

structure gate 已通过，readability 仍失败，应先改 Prompt 哪一层？
优先改 style_profile 的 sentence_max_chars 与 Few-shot 示例句长；若仅改 User brief 无法稳定约束句法。
factuality gate 没有外部来源时怎么办？
S5 内部案例可绑 source_doc_id；纯观点段标注 non_verifiable 跳过 cite 校验，但须过 compliance 审核。

内容安全审核

市场稿件一旦含敏感政治、歧视或未经披露的客户名，后果比客服误答更严重。content safety review 在 Prompt 层声明禁区，并在生成后跑分类器 + 关键词 + PII 扫描。

层级动作CorpAssist 配置失败动作
L1 PromptSystem 禁区清单safety_policy_block模型自检拒写
L2 后处理敏感词 + 正则content_safety_rules.yaml替换或阻断
L3 分类器多标签风险分moderation API人工队列
L4 发布CMS 双人复核workflow tickethold

与 Prompt 注入攻防 不同，内容安全侧重生成物合规而非攻击者指令；但 RAG 引用的外部素材仍可能间接注入，S5 应对 retrieved 块复用 injection boundary。

误区
  • 仅依赖 CMS 人工发布前扫一眼
  • 把客户未授权案例写进 Few-shot
  • 安全规则与 Prompt 版本不同步
推荐
  • safety_policy 写进 System 且进 regression
  • 案例库 anonymize + 授权标记
  • safety_rules_version 绑定 prompt git tag

content_safety_rules.yaml 片段

yaml 
rules:
  - id: CS-PII-01
    pattern: "\\d{17}[\\dXx]"
    action: block
  - id: CS-COMPETITOR-01
    pattern: "(?:竞品|友商)全称"
    action: flag_review

个性化参数

同一 S5 模板要服务 12 个 BU：受众、渠道、语言与数据披露级别各不相同。personalization params 通过变量注入实现，禁止为每个 BU 复制整份 System Prompt。

参数来源影响块示例
tenant_idJWT / 路由style_profile 覆盖huadong_formal
audiencebrief 表单术语深度、案例行业manufacturing_cio
channel发布目标structure_schemawechat vs pdf
locale用户选择language blockzh-CN
disclosure_level合规档位数据与案例粒度public / internal

面试常问
  • 「3000 字白皮书如何控重复与风格？」→ outline + dedup gate + style_profile
  • 「多 BU 如何不 fork Prompt？」→ personalization params + tenant overlay YAML

本章结论

  • 风格与结构 Prompt：style_profile + structure_schema + block 模板三层锁制品形态。
  • 长文规划与去重：outline planner 先行，段内 n-gram 与跨段 embedding 双去重。
  • 质量评估流水线：structure → readability → brand → factuality → dedup 串联 gate。
  • 内容安全审核：Prompt 禁区 + 后处理规则 + 人工 CMS 四层级。
  • 个性化参数：tenant/audience/channel 变量化，避免 BU 级 Prompt fork。

动手练习

为 CorpAssist S5 编写 style_profile 与 outline-plan.json，配置 QA gate 清单，并在 demo 目录跑验收脚本确认 structure 与 dedup 字段齐全。

操作步骤

  1. 根据「风格与结构 Prompt」表，填写 style_profile 的 banned_phrases 与 required_blocks（见 demo YAML）。
  2. 按「长文规划与去重」JSON 示例，为 3000 字白皮书写 4 段 outline-plan.json（含 target_words）。
  3. 把「质量评估流水线」决策表五类 gate 写入 qa-gates.yaml 的 gates[] 列表。
  4. 在 content_safety_rules.yaml 增加 1 条 PII 规则（参照正文 CS-PII-01）。
  5. 可选：完成 demos/advanced-06-content-generation-lab/ 验收命令。

判断练习

  1. （场景）团队一次 Prompt 生成 3500 字全文，发布后 §2 与 §4 大段重复。最应优先补哪两个机制？

    参考答案

    outline planner + 跨段 dedup gate：先拆 section 再生成，合并时用 embedding 相似度拦截；仅改 User「不要重复」无法稳定生效。

  2. （判断）structure gate 全绿但 brand voice gate 失败，应优先改 style_profile 还是 brief？

    参考答案

    style_profile：结构已满足 schema，失败在语调/禁用词；brief 只驱动主题，不承载 tone 硬约束。
