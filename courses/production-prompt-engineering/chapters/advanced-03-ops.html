约 40 分钟 · 阶段：生产治理 · 能力：建立 CorpAssist 产品-研发 Prompt 协作流、文档规范、评测绑定与工具链

为什么要学本章

Prompt 变更在 CorpAssist 曾像「改个字符串」：产品飞书文档、研发 Git 各一份，上线后才发现评测集未更新。PromptOps 把 Prompt 纳入与 API 同等的变更管理：PRD handoff、prompt spec、eval binding 与 CI prompt gate 缺一不可。

学完你能
  • 设计产品到研发的 PRD handoff 与 Prompt CR 流程
  • 编写可评审的 prompt spec 与 change log
  • 将 Prompt 与 golden set 做 eval binding，启用 semver prompt
  • 用 Langfuse 等实现 Langfuse trace 与 CI 门禁
本章先记住 4 件事
  • Prompt 是配置：变更需 MR、Review、版本号与回滚计划。
  • 文档即契约：prompt spec 列变量、边界、评测指标，评审可签字。
  • 评测跟版本走：每个 semver tag 绑定 eval manifest。
  • 工具链打通：Git + CI + 观测，避免「文档写了、线上没生效」。

产品-研发流程

CorpAssist · 拒答话术 v4 需求 产品希望在保持合规前提下让拒答更「有温度」。若跳过 PromptOps 流程直接改 System，常出现：研发改了 Java 模板却漏改 Python 链、评测仍跑 v3 golden——上线后双栈行为不一致。

PRD handoff 在 LLM 场景需额外字段：影响场景（S1/S2/工单）、风险等级、是否动 compliance、所需评测增量。CorpAssist 使用统一 PROMPT-CR 模板。

handoff 关键在 spec 与 golden 同步，MR 不得绕过 CI。

反模式
  • 飞书文档即源码，Git 滞后一周
  • 产品直接改生产配置中心
  • 无 semver，只说「最新版」
PromptOps
  • Git 为唯一源码，文档从 spec 生成
  • 变更必须 MR + 双人 Review
  • semver prompt：patch=措辞，minor=结构，major=行为契约
先想 10 秒：为何 Prompt 变更也要 semver？

便于 eval binding、回滚与 A/B 对照；「最新」无法关联 trace 与 badcase，排障和审计都会失效。

PROMPT-CR 模板必填项

字段说明CorpAssist 示例

semver_bumppatch/minor/majorminor（新增变量）

CR 合入前，研发须在 MR 描述粘贴 PROMPT-CR 链接；CI 读取 semver_bump 决定跑全量还是增量 golden。

文档化

prompt spec 是评审与实现的桥梁：变量表、System 结构、few-shot 来源、拒答/合规引用、观测字段。每次 MR 必须更新 change log。

prompt-spec.md 最小结构

markdown 
# CorpAssist RAG Prompt v2.4.0
## 变量
| name | type | source | required |
| tenant_id | string | gateway | yes |
| tone_profile | enum | tone_matrix | yes |
## System 结构
1. persona 2. compliance 3. RAG 指令 4. cite 格式
## 行为变更（vs v2.3.0）
- 拒答阈值 0.72 → 0.70（minor）
## 评测
- golden: rag-golden-v14.jsonl
- 阈值: cite_coverage ≥ 0.95
semver典型变更文档要求
patch错别字、语气微调change log 一行
minor新增变量、few-shotspec diff + golden 增量
major输出 schema、拒答策略产品签字 + 全量回归

与 变量、版本与配置管理 的 Git 策略一致：prompt 文件路径 prompts/{scene}/{semver}/，禁止覆盖式提交。

spec 与代码注释重复，还要维护吗？
要。spec 面向产品/合规/QA 可读，代码注释面向研发；CI 可校验 spec 中的变量名与模板占位符一致。
change log 写多细？
patch 一行即可；minor 须列新增变量与 golden 增量文件名；major 须附产品签字 PDF 或工单号，便于审计追溯。

change-log 条目示例

markdown 
## 2.4.0 (2026-06-04) — minor
- 新增 tone_profile 变量，默认 formal
- golden: +15 条拒答边界 case (rag-golden-v14)
- 双栈: Spring ToneAdvisor + LangChain inject_tone 同步
## 2.3.1 (2026-05-20) — patch
- 修正 cite 示例 typo，无行为变更

评测联动

eval binding 指每个 Prompt 版本在 manifest 中声明依赖的 golden set、指标阈值与注入/合规子集。MR 触发 CI prompt gate，不通过则不可合并。

manifest 字段含义示例
prompt_versionsemver tagrag-prompt@2.4.0
golden_sets[]绑定的评测集rag-golden-v14
thresholds指标下限cite_coverage: 0.95
suites[]附加套件injection-regression

eval-manifest.yaml

yaml 
prompt_version: rag-prompt@2.4.0
golden_sets:
  - id: rag-golden-v14
    path: eval/golden/rag-v14.jsonl
thresholds:
  cite_coverage: 0.95
  refusal_rate_max: 0.12
suites:
  - injection-regression
  - compliance-disclaimer

联动 LLM 评测与质量llm-evaluation-quality 与 A/B 与迭代流程：离线 gate 通过后，才进入在线 A/B。

复盘 · golden 仍指向 v13 MR 升级模板到 2.4.0 但未改 manifest，CI 误用旧集通过——线上 cite 回归。修复：gate 校验 manifest.prompt_version 与 Git tag 一致。
CI gate 执行顺序
  1. lint：spec 变量与模板占位符一致
  2. semver：MR label 与 manifest.prompt_version 匹配
  3. eval：跑 golden + suites，产出 HTML 报告 artifact
  4. diff：Prompt CR 双人 approved 状态

工具链

PromptOps 工具链：Git（源码）→ CI（gate）→ 配置中心（运行时指针）→ Langfuse trace（观测）。双栈须上报相同 prompt_version 标签。

Spring AI：ChatClient 观测标签

java 
ChatResponse response = chatClient.prompt()
    .system(systemFromGit)
    .user(query)
    . advisors(a -> a.param("prompt_version", "rag-prompt@2.4.0"))
    .call()
    .chatResponse();
otelSpan.setAttribute("prompt.version", "rag-prompt@2.4.0");

LangChain：Langfuse callback

python 
from langfuse.callback import CallbackHandler

handler = CallbackHandler(
    trace_name="corpassist-rag",
    metadata={"prompt_version": "rag-prompt@2.4.0"},
)
result = chain.invoke(payload, config={"callbacks": [handler]})
CI prompt gate 流水线
flowchart LR
  MR[Git MR] --> Lint[spec/template lint]
  Lint --> Eval[跑 eval-manifest]
  Eval --> Inj[注入/合规子集]
  Inj --> Report[评测报告 artifact]
  Report --> Merge{阈值 OK?}
  Merge -->|yes| Main[合并 main]
  Merge -->|no| Block[阻塞 MR]

与 CI prompt gate 脚本共享阈值定义，避免 Java/Python 各一套标准。

Langfuse / 自研看板按 prompt_version 聚合 latency、token、拒答率；与 可观测与可靠性运维observability-reliability-ops 的 SLO 对齐。

运行时从配置中心读 active_prompt_pointer，回滚 = 指针切回上一 semver，无需重新部署代码。

本章结论

  • 产品-研发流程：PRD handoff → prompt spec → MR → CI，双栈同步。
  • 文档化：prompt spec + semver + change log 是可评审契约。
  • 评测联动：eval binding manifest 与版本一致，gate 阻塞不合格 MR。
  • 工具链：Git/CI/Langfuse trace 统一 prompt_version 标签。

动手练习

为 rag-prompt@2.4.0 编写 prompt-spec 摘要、eval-manifest.yaml 与 change log 条目，并配置 CI 脚本校验 manifest 与 Git tag 一致。

操作步骤

  1. 按「产品-研发流程」表，填写 PROMPT-CR：场景 S1、风险 L1、是否动 compliance=否。
  2. 在 prompt-spec.md 增加变量表与 vs v2.3.0 行为变更（对照 semver minor 规则）。
  3. 编写 eval-manifest.yaml，绑定 rag-golden-v14 与 cite_coverage 阈值 0.95。
  4. 在 Spring 与 Python 样例中设置相同的 prompt_version metadata（见工具链 code-block）。
  5. 可选：完成 demos/advanced-03-ops-lab/ 运行 python check_ops_manifest.py，期望 manifest_ok=true。

判断练习

  1. （排障）MR 合并后线上 trace 仍显示 rag-prompt@2.3.0，最应优先检查哪一环？A) golden set B) 配置中心 active_prompt_pointer C) Langfuse 采样率

    参考答案

    B：Git 已合并但运行时指针未切换最常见；A 影响评测不影响线上 tag；C 只影响采样可见度，不应改 version 字段。
