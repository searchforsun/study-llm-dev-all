为什么要学本章

Prompt 是跨产品、研发、算法的共享资产。没有 pm dev flow 时，会出现「产品私改文案上线」「研发不知道哪版 Prompt 在生产」等问题。本章建立登记册、变更单与评测联动，为 CorpAssist Prompt 套件 交付打底。

学完你能
  • 描述 Prompt 从起草到发布的 review workflow
  • 维护 prompt registry 与 prompt doc
  • 把 golden set 与 A/B 与迭代流程 接入 eval linkage
  • 选型 Langfuse / 自研看板等 toolchain 组件

产品-研发流程

PromptOps 变更流
flowchart LR
  PM[产品起草] --> CR[change request]
  CR --> Dev[研发模板化]
  Dev --> Eval[golden 评测]
  Eval --> Review[双签 Review]
  Review --> AB[A/B 灰度]
  AB --> Prod[全量]

每条 change request 必填：场景 ID（S1/S2/S5）、变更原因、影响指标、回滚版本号。与 变量、版本与配置管理 的 prompt semver 对齐。

角色职责产出
产品业务话术、拒答策略CR 描述 + 示例对话
研发Jinja/Spring 模板、变量PR + 版本 tag
算法/QA评测与门禁eval 报告

文档化

prompt doc 最小字段：prompt_id、semver、owner、变量表、依赖模型、关联 eval 集、上次 A/B 结果。prompt registry 可放在 Git 目录 prompts/registry.yaml 或内部 Wiki，但以 Git 为单一事实来源。

yaml 
- id: corpassist-rag-qa
  version: 3.2.1
  scene: S1
  path: prompts/rag-qa-v3.j2
  eval: eval/rag-golden.jsonl
  owner: team-prompt
  last_ab: { variant: B, faithfulness: 0.91 }

评测联动

eval linkage：CI 在 Prompt PR 上自动跑 golden + injection 子集；指标劣化则阻断合并。发布时再次跑全量，与 eval gate 一致。

  • PR 阶段：快速集（< 200 条），P95 延迟 < 15 分钟
  • 发布前：全量 golden + 红队集
  • 线上：A/B 看业务指标 + 自动采样进评测库
