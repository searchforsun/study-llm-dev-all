为什么要学本章

Prompt 是跨产品、研发、算法的共享资产。没有 pm dev flow 时，会出现「产品私改文案上线」「研发不知道哪版 Prompt 在生产」等问题。本章建立登记册、变更单与评测联动，为 CorpAssist Prompt 套件 交付打底。

学完你能
  • 描述 Prompt 从起草到发布的 review workflow
  • 维护 prompt registry 与 prompt doc
  • 把 golden set 与 A/B 与迭代流程 接入 eval linkage
  • 选型 Langfuse / 自研看板等 toolchain 组件
本章先记住 4 件事
  • Prompt 是代码：走 PR、semver、CODEOWNERS，禁止 Wiki 私改上线。
  • CR 必填场景 ID：S1/S2/S5 变更影响不同 golden 集与指标。
  • 评测是闸门：PR 快速集 + 发版全量集，劣化阻断合并。
  • 工具链可观测：trace 对比 prompt 版本，支撑 排障。

产品-研发流程

CorpAssist · 拒答文案紧急变更 合规要求 24 小时内把 S1 拒答句从「无法回答」改为带工单入口的固定模板。产品想直接改配置中心；研发要求走 CR + golden。本章流程即为此类冲突提供可审计快车道：CR 仍必填，但可开 expedited eval 子集（50 条）+ 双签后 5% A/B。
PromptOps 变更流
flowchart LR
  PM[产品起草] --> CR[change request]
  CR --> Dev[研发模板化]
  Dev --> Eval[golden 评测]
  Eval --> Review[双签 Review]
  Review --> AB[A/B 灰度]
  AB --> Prod[全量]

每条 change request 必填：场景 ID（S1/S2/S5）、变更原因、影响指标、回滚版本号。与 变量、版本与配置管理 的 prompt semver 对齐。

角色职责产出
产品业务话术、拒答策略CR 描述 + 示例对话
研发Jinja/Spring 模板、变量PR + 版本 tag
算法/QA评测与门禁eval 报告

pm dev flow 与 Git 版本 绑定：产品只提交 CR 与示例对话，不直接改 Jinja；研发把文案模板化并升 PATCH/MINOR；算法在 PR 评论贴 eval 摘要。

反模式
  • 产品邮件发 Word 文案，研发手工粘贴上线
  • 无回滚版本号，出问题只能「再改回去」
  • 三场景共用一份 System，改 S2 误伤 S1
CorpAssist 做法
  • CR 模板含 scene、metrics、rollback_tag
  • 按场景独立模板路径与 golden
  • 双签：产品 + 研发 Prompt Owner

文档化

prompt doc 最小字段：prompt_id、semver、owner、变量表、依赖模型、关联 eval 集、上次 A/B 结果。prompt registry 可放在 Git 目录 prompts/registry.yaml 或内部 Wiki，但以 Git 为单一事实来源。

yaml 
- id: corpassist-rag-qa
  version: 3.2.1
  scene: S1
  path: prompts/rag-qa-v3.j2
  eval: eval/rag-golden.jsonl
  owner: team-prompt
  last_ab: { variant: B, faithfulness: 0.91 }

review workflow 在 PR 描述中链接 registry 条目 diff；Reviewer 核对：变量表是否更新、eval 路径是否仍有效、last_ab 是否需重置。

markdown 
# Prompt CR 示例（S1 · 拒答模板）
- scene: S1
- prompt_id: corpassist-rag-qa
- from_version: 3.2.1 → to_version: 3.2.2 (PATCH)
- reason: 合规要求拒答句增加工单链接
- impact_metrics: [refusal_accuracy, faithfulness]
- rollback_tag: prompts/v3.2.1
- golden_subset: eval/rag-golden-smoke-50.jsonl
- approvers: @pm-li @dev-wang
markdown 
# prompt-doc 模板（每模板一份 README 片段）
## corpassist-rag-qa v3.2.1
- **owner**: team-prompt
- **scene**: S1
- **variables**: context_blocks, user_dept, refusal_phrase, cite_format
- **model**: qwen-plus @ temp 0.1
- **eval**: eval/rag-golden.jsonl (min faithfulness 0.88)
- **last_ab**: variant B +3pp faithfulness, 2026-05-20
- **rollback**: prompts/v3.2.0
- **changelog**: PATCH 拒答句增加工单链接

prompt doc 与 registry 条目一一对应；PR 必须更新 doc 中的 changelog，Reviewer 核对变量表是否与 Jinja 占位符一致。

评测联动

eval linkage：CI 在 Prompt PR 上自动跑 golden + injection 子集；指标劣化则阻断合并。发布时再次跑全量，与 eval gate 一致。

  • PR 阶段：快速集（< 200 条），P95 延迟 < 15 分钟
  • 发布前：全量 golden + 红队集
  • 线上：A/B 看业务指标 + 自动采样进评测库
先想 10 秒：Prompt PR 只改标点，要不要跑 golden？

要。标点/措辞可能影响拒答率与 faithfulness；CI 应跑快速 golden 子集。纯文档 CR 可跳过，但模板正文变更必须 eval linkage 门禁。

阶段数据集阻断条件典型耗时
PR 快速集smoke-50 / injection-20任一 primary 指标 < control -1pp< 15 min
合并前场景 golden 子集 200 条guardrail 未过线< 45 min
发版全量全 golden + 红队faithfulness / 注入率劣化2～4 h
线上 A/B5% 流量 + 业务 KPI投诉率 / 完成率异常24～72 h
CorpAssist · eval 阻断合并 研发在 S1 模板中微调拒答句，PR 快速集 faithfulness 从 0.91 跌至 0.87。CI 阻断合并；Prompt Owner 对比 trace 发现新拒答句在边界 case 上过度拒答，回滚 diff 后指标恢复。
yaml 
# .github/workflows/prompt-eval.yml 片段
on:
  pull_request:
    paths: ['prompts/**', 'eval/**']
jobs:
  golden-smoke:
    runs-on: ubuntu-latest
    steps:
      - run: python scripts/run_golden.py --set rag-smoke-50 --fail-if-below 0.85
      - run: python scripts/run_injection.py --subset core-20

发版流水线在 merge 后触发全量 eval；与 release gate 共用阈值配置 eval/thresholds.yaml，避免 PR 与 prod 标准不一致。

事故与 expedited 路径

合规/安全类 CR 可走 expedited：不可省 CR 记录、快速 golden（≥50 条）、双签、5% A/B；可压缩的是全量 golden 等待时间（发版后 24h 内补跑全量）。

级别触发PromptOps 动作
P0 安全注入成功率升 / 合规违规rollback tag + 热修 CR + 2h 内 eval 报告
P1 质量faithfulness 跌 > 5pphold 流量 + diff System + 子集 golden
P2 体验tone 投诉集中正常 CR 周期 + tone audit

与 面试排障 衔接：事故报告须含 prompt_version、metric delta、rollback tag 三要素。

RACI产品研发算法/QA运维
起草 CRRCII

R=负责 A=审批 C=协商 I=知会。Prompt Owner 通常由研发资深工程师担任，对 registry 与 semver 负最终责任。

坏 PromptOps
  • Wiki 改文案、无 CR 记录
  • registry 与生产 tag 不一致
  • 事故后无 rollback tag 文档
好 PromptOps
  • CR → PR → eval → 双签 → A/B 全链路
  • trace 可按 prompt_version 过滤
  • P0 两小时内产出 eval 报告 + rollback 记录

判断练习

  1. 产品直接在 Wiki 改拒答文案并通知研发「今晚上线」，流程哪里错了？

    参考答案

    绕过 CR、Git 版本与 eval 闸门；Wiki 非单一事实来源。应走 CR → 模板 PR → golden → 双签 → A/B。

  2. registry 只存路径、不存 last_ab 结果，有何风险？

    参考答案

    无法快速判断当前生产版本是否经 A/B 验证；回滚与审计困难。registry 应含 semver、eval 集、last_ab 摘要。
