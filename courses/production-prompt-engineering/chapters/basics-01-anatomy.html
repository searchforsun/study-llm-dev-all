为什么要学本章

生产里「Prompt 不好用」往往不是模型差，而是消息角色混写、边界未声明、人设漂移。本章用 CorpAssist 企业助手建立 prompt anatomy 的分层心智：System 定规则、User 承载任务、Tool 回灌事实——再落到双栈模板。

学完你能
  • 区分 System / User / Tool / Assistant 四类消息及其注入时机
  • 为 CorpAssist 编写可审计的 boundary declaration（拒答、权限、数据范围）
  • 落地 CorpAssist persona 人设模板并在 Spring AI / LangChain 复用
  • 诊断「模型不听 system」时的角色与优先级根因
本章先记住 3 件事
  • System 写规则，User 写任务：不要把运营话术、检索结果塞进 System。
  • 边界要可测试：拒答、权限、引用范围写成可回归的条款，而非形容词堆砌。
  • 人设是契约：语调、称谓、 escalation 路径版本化，与模型版本解耦。

System/User/Tool

Chat Completions API 将一次推理输入组织为消息列表。system message 定义助手身份与全局约束；user message 承载本轮用户意图与结构化任务；tool message 回灌工具调用结果，供模型在下一轮推理中引用事实。

CorpAssist · 订单查询对话 员工问：「帮我查订单 SO-2026-8842 的物流状态。」编排层应先注入 System（客服人设+边界），再拼 User（问题+租户 ID），若调用 ERP API 则将 JSON 摘要写入 Tool，最后由模型生成面向用户的 Assistant 回复。
角色典型内容稳定性常见错误
System 人设、输出格式、安全边界、工具使用总则 按场景版本化，变更走 Git 把 RAG 检索片段塞进 System
User 用户原话、任务指令、JSON 工单字段 每轮变化 在 User 里写「你永远要…」类永久规则
Tool API 返回、SQL 结果、检索 metadata 每次工具调用一条 原文日志未截断导致爆窗
Assistant 模型历史回复（多轮时） 累积 把 Assistant 当 System 续写规则
Prompt Anatomy：消息分层
flowchart TB
    subgraph Request[单次 Chat 请求]
        S[System\n人设+边界+格式]
        U[User\n任务+上下文占位]
        T[Tool\nAPI/RAG 事实回灌]
        A[Assistant\n历史回复]

    U --> T

    S -->|最高优先级| Rules[可测试规则]

System 定规则，User 定任务，Tool 定事实；Assistant 仅承载历史语气。

在 dual stack prompt 中，Python 侧用 LangChain ChatPromptTemplate 组装消息；Java 侧用 Spring AI Prompt + Message 类型（SystemMessage、UserMessage、ToolResponseMessage）保持同一语义。

Python：LangChain 消息组装

python 
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.messages import SystemMessage, HumanMessage

SYSTEM = """你是 CorpAssist 企业客服助手。
- 仅依据 Tool 与检索上下文回答订单/工单问题。
- 不确定时回复：我需要查证后再回复。"""

def build_messages(user_text: str, tool_payload: str | None):
    msgs = [SystemMessage(content=SYSTEM), HumanMessage(content=user_text)]
    if tool_payload:
        msgs.append({"role": "tool", "content": tool_payload})
    return msgs

prompt = ChatPromptTemplate.from_messages([
    ("system", SYSTEM),
    ("human", "{question}"),
])

Java：Spring AI 消息类型

java 
import org.springframework.ai.chat.messages.SystemMessage;
import org.springframework.ai.chat.messages.UserMessage;
import org.springframework.ai.chat.messages.ToolResponseMessage;
import org.springframework.ai.chat.prompt.Prompt;

String system = """
你是 CorpAssist 企业客服助手。
仅依据 Tool 与检索上下文回答订单/工单问题。
""";

Prompt prompt = new Prompt(List.of(
    new SystemMessage(system),
    new UserMessage(userQuestion),
    new ToolResponseMessage(toolResults)
));
不推荐
  • 单条 User 消息里混写人设、检索结果、API 返回
  • 每轮重写 System，导致 A/B 无法对比
  • Tool 返回整页 JSON 不截断
推荐
  • System 固定版本化，User 只变任务
  • Tool 只放模型决策所需字段
  • 双栈共享同一份 YAML 消息契约
先想 10 秒：为什么检索到的知识库片段不应写入 System？

System 应稳定且可版本化；检索片段每轮变化，写入 System 会导致无法 diff、无法回归，且挤占角色约束的注意力。检索内容应走 User 占位或独立 context 块，并在 RAG 课中强化引用格式。

边界声明

boundary declaration 是把「能做什么 / 不能做什么」写成可测试条款，而不是「尽量准确」这类无法验收的形容词。CorpAssist 面向企业内部员工，边界通常覆盖：数据域、操作权限、拒答与升级。

边界条款结构

条款类型示例验收方式
数据范围 仅回答当前 tenant_id 下订单与工单 越权 tenant 用例应拒答
操作边界 不得代用户发起退款，仅生成工单草稿 诱导退款指令应拒绝并引导流程
拒答 无 Tool/检索支撑时不编造单号 空检索场景固定话术
升级 涉及薪资、法务转人工队列 关键词触发 escalation

边界声明 YAML 片段

yaml 
corpassist_boundaries:
  version: "1.2.0"
  data_scope:
    - "仅使用请求头 X-Tenant-Id 对应租户数据"
    - "禁止跨租户推断用户身份"
  refusal:
    - trigger: "no_tool_and_no_retrieval"
      reply: "当前没有查到可靠记录，我需要查证后再回复。"
    - trigger: "prompt_injection_attempt"
      reply: "该请求超出客服助手授权范围。"
  escalation:
    keywords: ["劳动合同", "诉讼", "薪资调整"]
    action: "route_to_human_queue"

边界应与评测集绑定：每条 refusal policy 至少 3 条正例 + 3 条对抗例，发布前跑回归。与安全课衔接时，把用户输入净化放在 Gateway，边界声明放在 System，形成双层防御。

边界写进 User 可以吗？
临时调试可以，生产不推荐。User 每轮可变，边界在 User 中会被历史淹没或遭注入覆盖。永久规则应放 System，且版本号与 Git tag 对齐。
Tool 返回里含有 PII 怎么办？
在工具层做字段白名单与掩码，再写入 Tool message。Prompt 层声明「不得复述完整身份证号」是第二道闸，不能替代数据面脱敏。

CorpAssist 人设

CorpAssist persona 是面向员工的统一语气与行为契约：专业、简洁、可执行，避免 C 端客服的过度寒暄。人设不是「性格形容词」，而是可版本化的模板字段：称谓、语言、结构、升级路径。

CorpAssist Persona 字段
flowchart LR
    Persona[CorpAssist Persona v1.3]
    Persona --> Tone[语调: 专业务实]
    Persona --> Lang[语言: 简体中文]
    Persona --> Structure[结构: 理解确认-事实-建议]
    Persona --> Escalation[升级: 敏感词转人工]
    Persona --> Brand[品牌: 禁用竞品贬损]

人设字段与边界、输出格式一并纳入 Git 版本库。

人设 System 模板（节选）

markdown 
# CorpAssist Persona v1.3
你是 CorpAssist，面向 {{company_name}} 员工的企业智能助手。

## 语调
- 使用「您」，句子简短，避免夸张emoji。
- 先确认理解，再给出事实与下一步建议。

## 输出结构
1. 理解确认（1 句）
2. 查询结果（条目或表格）
3. 建议行动（含是否需要二次确认）

## 禁止
- 编造订单号、审批单号、金额。
- 在未调用工具时声称「已为您办理」。

从 prompts/persona_v1.3.md 读取，经 PromptTemplate 注入 company_name；禁止在代码里硬编码长 System 字符串。

使用 ResourcePromptTemplate 或配置中心 key corpassist.persona，与 Python 共用同一 Git 路径，CI 校验 hash 一致。

常见误区
  • 人设过长 — 超过 1.5K tokens 的 System 遵守率下降，细节应拆到场景子模板。
  • 每模型一套人设 — 应维护「逻辑人设 + 模型适配层」，而非 N 份拷贝。
  • 忽略 Assistant 历史语气 — 多轮后旧回复会带偏语调，需结合下一章历史策略。
检查理解：人设与边界为何要分开文件维护？

人设变更频率低、影响全局语气；边界随合规与产品迭代更频繁。拆分便于权限审批（法务改边界、品牌改人设）与独立回归基线。

本章结论

生产 Prompt 的第一性原理是消息角色清晰、边界可测、人设可版本化。

  • System/User/Tool：System 定规则，User 定任务，Tool 定事实；禁止混写。
  • 边界声明：拒答、权限、升级写成可回归条款，绑定评测集。
  • CorpAssist 人设：语调与结构版本化，双栈共享 Git 源文件。
