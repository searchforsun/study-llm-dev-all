为什么要学本章

生产事故里最常见的 Prompt 问题不是「词不够炫」，而是规则放错层：拒答写在 user 里被覆盖、人设与 S1 RAG 指令冲突、Tool 回传未纳入上下文。本章以 CorpAssist S1 知识库 / S2 客服 / S5 内容生成 为参照，建立可版本化、可 A/B 的 Prompt 解剖 基线，再谈 Few-shot 与模板变量。

学完你能
  • 画出一次 Chat 请求中 system / user / assistant / tool 各层职责，并说明 Agent 场景下 tool message 的位置
  • 为 S1/S2/S5 各写一条可审计的边界声明（拒答、禁止编造、数据范围）
  • 输出一份 人设卡片，并与 Spring PromptTemplate / Python 模板对齐字段
  • 衔接下一章 Few-shot 与 CoT 的示例层设计
本章先记住 3 件事
  • 分层：稳定规则进 instruction 层（system）；可变输入进 user；工具结果用 tool/assistant 按协议回灌。
  • 边界前置：边界声明 写清「能做什么 / 不能做什么 / 无依据怎么办」，S1 偏引用与拒答，S2 偏转人工，S5 偏品牌与安全。
  • 人设可治理：角色设计 不是一句「你是助手」，而是带版本号、可评测的 persona card。

System/User/Tool

OpenAI 兼容 Chat API 用 role 区分消息来源。生产 Prompt 的 解剖 本质是：把长期有效与随请求变化的内容分开，避免用户一句「忽略以上规则」就击穿 system。

CorpAssist 单次对话：消息分层
flowchart TB
  subgraph layers["Prompt 解剖"]
    SYS["system · 人设 + 边界 + 输出格式"]
    USR["user · 用户问题 + 检索上下文占位"]
    AST["assistant · 历史回复"]
    TOL["tool · 工具 JSON 结果"]
  end
  SYS --> LLM[Chat Completions]
  USR --> LLM
  AST --> LLM
  TOL --> LLM
  LLM --> OUT[结构化 / 流式输出]

S3 Agent 会在 assistant 与 tool 间多轮往返；S1 RAG 常在 user 中注入 <context>，但拒答与引用规则仍应写在 system。

Role典型内容CorpAssist 场景常见误区
system 人设、合规、输出 schema、拒答阈值 S1：仅依据上下文 + 引用格式；S2：工单字段规范；S5：品牌语调 把业务数据硬编码进 system，导致无法按请求注入
user 用户问题、检索片段、会话变量 S1：{{chunks}}；S2：订单号 + 用户意图；S5：选题与受众 在 user 里重复写一遍 system 规则，维护两份真源
assistant 模型历史输出 S2 多轮客服摘要、S3 计划步骤 未裁剪历史导致 Token 爆炸（见 多轮对话与状态）
tool 函数 / 插件执行结果 S2 查单 API、S3 ERP 回传 工具结果未声明格式，模型误读 JSON 字段

双栈：同契约，不同组装方式

Spring AI · PromptTemplate
java 
Prompt prompt = new Prompt(
    List.of(
        new SystemMessage(systemTemplate.render(vars)),
        new UserMessage(userTemplate.render(vars))
    ),
    ChatOptions.builder().temperature(0.2).build()
);
// Tool 回调由 ChatClient + FunctionCallback 注入
不推荐
  • 把所有指令塞进一条超长 user
  • 只改 user 模板，system 仍是半年前的「通用助手」
  • Tool 返回非 JSON 文本且无字段说明
推荐
  • system 管 角色 与 边界；user 管本次上下文
  • system/user/tool 分层写入 Git，变更走评测
  • 工具结果结构化，必要时在 system 附 tool message 解读规则
先想 10 秒：S1 检索到的 chunks 应放在 system 还是 user？为什么？

通常放 user（或专用 context 段）：chunks 随问题变化；system 保持稳定的拒答与引用格式。若把 chunks 写进 system，每次检索都要重写 system，成本高且难缓存。

边界声明

CorpAssist · S1 制度问答（30 秒） 员工问「异地办公补贴标准」——检索命中 2023 年旧版片段。模型若凭常识补全，合规风险极高。边界声明必须写：无依据拒答、禁止编造政策编号、建议查看最新 HR 门户或转人工。

边界声明 是 instruction 层 的核心：用可测试的句子约束模型行为，而不是模糊的「尽量准确」。

边界类型声明要点S1 / S2 / S5 示例
知识范围 仅依据提供的上下文；禁止用训练记忆补全 S1：仅依据 <context>；S5：仅依据品牌手册与活动 brief
拒答 无依据时的固定话术 + 下一步动作 S1：「根据现有资料无法确定」；S2：「为您转接人工客服」
禁止行为 泄露、越权、违法、冒充人工 S2：不得透露其他用户订单；S5：不得承诺未审批优惠
输出格式 引用、JSON schema、步骤编号 S1：[来源:文件名#段]；S2：工单摘要字段（见 practice 章）

边界声明模板（Markdown system 段）

markdown 
## 边界（CorpAssist S1 · v1.2.0）
1. 仅依据用户消息中 <context> 块回答；不得使用外部知识补全政策细节。
2. 每个事实陈述必须带引用 [来源:文档名#段落]。
3. 若 context 不足以回答：输出固定句「根据现有资料无法确定」，并建议联系 HR 或查看制度门户。
4. 禁止：编造文号、猜测生效日期、讨论其他员工个案。
弱边界
  • 「请认真回答用户问题」
  • 拒答句只在产品 UI 展示，未写入 Prompt
  • S1 与 S5 共用同一段 system，未区分场景
强边界
  • 编号清单 + 可回归测试用例（黄金集）
  • 拒答阈值与引用格式可自动评测
  • 按模块（S1/S2/S5）拆分 system 模板文件
判断：把拒答规则写在 user 末尾「请遵守以上规则」是否足够？

不够。user 内容易被多轮对话稀释，且用户输入可能注入冲突指令。拒答与合规应放在 system，并在评测集中覆盖注入样本（advanced 章展开）。

CorpAssist 人设

CorpAssist · S2 客服 + S5 内容（30 秒） S2 需要耐心、简洁、可转人工；S5 需要品牌调性、结构清晰、可审核。同一条「你是 CorpAssist」无法兼顾——应用 人设卡片 分场景维护，共享底层合规底线。

CorpAssist 人设 = 语气 + 身份 + 能力边界 + escalation 策略。它是 角色设计 的可交付物，而非产品经理口头描述。

人设卡片 → 三场景 system 模板

  CARD["persona-card.yaml"]

  CARD --> S2T["s2-cs-system.md"]
  CARD --> S5T["s5-content-system.md"]
  S1T --> GIT[Git tag v1.x]
  S2T --> GIT

共享字段（合规、语言）在 card 中定义一次；场景差异在子模板覆盖。

字段S1 知识库S2 智能客服S5 内容生成
语气 客观、引用先行 礼貌、短句、共情但不承诺 品牌调性词表、禁用夸张
身份 企业内部知识助手 官方客服助手（非真人） 营销文案协作者
边界 无引用不结论 不处理支付密码；复杂工单转人工 不生成未审核优惠承诺
输出 答案 + 引用列表 摘要 + 建议下一步 标题/正文/CTA 结构

persona-card 片段（YAML + system 预览）

yaml 
# prompts/persona-card.yaml · CorpAssist 共享
id: corpassist-core
version: "1.2.0"
locale: zh-CN
tone:
  base: 专业、简洁、可审计
  forbidden: [口语辱骂, 虚假承诺, 冒充真人客服]
escalation:
  default: 建议联系对应业务负责人或转人工
scenarios:
  s1_rag: { extends: base, cite_required: true }
  s2_cs:  { extends: base, max_sentences: 5, handoff: true }
  s5_content: { extends: base, brand_lexicon: marketing-2026 }
markdown 
你是 CorpAssist 企业智能助手（S2 客服场景）。
语气：礼貌、简洁，单次回复不超过 5 句。
你必须在开头说明「我是智能助手，非人工客服」。
不得确认退款/赔偿；涉及账户安全一律建议转人工。
人设与边界重复了？如何分工？

人设管「像谁、怎么说」；边界管「什么绝对不能做」。二者都在 system，但评测维度不同：人设用 tone audit，边界用拒答率 / 违规检出率。

本章结论

  • System/User/Tool：稳定 instruction 进 system，可变上下文进 user，Agent 工具走 tool；双栈共享同一消息契约。
  • 边界声明：拒答、引用、禁止行为写编号清单并可评测；S1/S2/S5 分场景模板，避免一条 system 打天下。
  • CorpAssist 人设：用 persona card 治理语气与 escalation，与 角色设计 版本同库管理。

判断练习

  1. 找错：同事方案 — ① system 只写「你是助手」② 拒答规则写在 user 末尾 ③ S1 chunks 放 user ④ Tool 结果不格式化。哪条最违反本章分层原则？

    参考答案

    答：②（及 ①）。拒答与合规属于 边界声明，应进 system；① 缺少可版本化人设与边界。③ 符合惯例；④ 应改进 tool 格式与 system 解读规则，但②对安全影响更大。

  2. 权衡：S5 营销文案要求「更有感染力」，产品希望把人设 tone 从「客观」改为「夸张促销」。你如何在 persona card 与合规边界间取舍？

    参考答案

    在 card 中为 S5 单独定义 brand_lexicon 与 forbidden（禁止未审批优惠、绝对化用语）；「感染力」通过结构示例与 Few-shot 提升，而非突破合规边界。变更走版本号 + 内容安全评测（见 advanced 语气章）。
