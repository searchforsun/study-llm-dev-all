为什么要学本章

Few-shot 与 CoT 教会你在提示词里放示例与推理链；上线后 Prompt 会随产品、制度、模型一起变。Prompt 是代码——必须用 模板引擎 抽变量、用 Git 版本 留痕，并在 dev/staging/prod 之间做 配置管理，否则「改一句拒答话术」就会在半夜把客服准确率打穿。

学完你能
  • 用 Jinja2 与 Spring PromptTemplate 渲染 CorpAssist RAG/客服模板，避免硬编码拼接
  • 为 Prompt 建立 语义化版本 与 Git 目录规范（prompts/ + 变更日志）
  • 区分 dev/staging/prod 的 环境差异（模型、top-k、max_context）而不混用配置
  • 绑定 回归基线 数据集，发布前通过 baseline-lock 闸门
本章先记住 4 件事
  • 变量外置：制度名、检索片段、用户工号等走模板变量，禁止在 Java/Python 里用 f-string 拼整段 System。
  • 版本可追溯：每次 Prompt 变更对应 Git commit + prompt-semver（如 rag-qa-v3.2）。
  • 环境隔离：staging 可连评测模型；prod 只读锁定配置，env-diff 写入 manifest 而非改模板正文。
  • 基线闸门：baseline-lock 的 golden 集未通过不得全量发布。

模板引擎

CorpAssist · 制度问答 RAG 同一套 System 要注入：检索到的制度片段、用户部门、当前日期。若三处服务（BFF、Python 评测、Spring AI 网关）各写一份字符串拼接，一周就会出现「staging 能答、prod 拒答」的漂移。

模板引擎 把结构与运行时数据分离：模板文件描述槽位，编排层注入变量。Python 侧常用 Jinja2（.j2）；Java 侧用 Spring AI 的 PromptTemplate，语法不同但槽位名应对齐。

对照：硬编码拼接 vs 模板

不推荐 · 代码内拼 Prompt
  • system = "你是助手..." + chunks + user_id 散落多处
  • 改拒答话术要搜全仓库字符串
  • 评测脚本与生产 BFF 模板不一致
推荐 · prompts/ 模板 + 变量
  • 单文件 rag_qa.system.j2，变量 context_blocks、user_dept
  • 渲染前校验必填变量（缺检索片段则短路拒答）
  • 双栈共享同一份模板仓库（见 Few-shot 与 CoT）

模板源文件：rag_qa.system.j2

jinja2 
{# prompts/rag_qa.system.j2 — CorpAssist S1 制度问答 #}
你是 {{ persona }}。仅依据下列「上下文」作答；无依据时明确拒答。
用户部门：{{ user_dept }}
引用格式：{{ cite_format }}

{% if context_blocks %}
--- 上下文 ---
{% for block in context_blocks %}
{{ block }}
{% endfor %}
--- 上下文结束 ---
{% else %}
{# 编排层应在 render 前短路；若仍渲染则显式拒答 #}
当前无可用检索片段，请回复无法确认。
{% endif %}

双栈渲染：Jinja2 + Spring PromptTemplate

Python · Jinja2
python 
from jinja2 import Environment, FileSystemLoader, StrictUndefined

env = Environment(
    loader=FileSystemLoader("prompts"),
    undefined=StrictUndefined,  # 缺变量直接报错，避免静默空串
    autoescape=False,
)
template = env.get_template("rag_qa.system.j2")

if not retrieved_chunks:
    return short_circuit_refusal()  # 缺 context_blocks 不调用 LLM

system_prompt = template.render(
    persona="CorpAssist 制度助手",
    context_blocks=retrieved_chunks,
    user_dept=session.dept or "未知",
    cite_format="【来源:{doc_id}§{section}】",
)
Java · Spring PromptTemplate
java 
// resources/prompts/rag_qa_system.st — 槽位名与 .j2 对齐
// 你是 {persona}。仅依据下列上下文作答；无依据时拒答。
// 用户部门：{user_dept}
// 引用格式：{cite_format}
// ---
// {context_blocks}

@Bean
PromptTemplate ragQaSystemTemplate(@Value("classpath:prompts/rag_qa_system.st") Resource res) {
  return new PromptTemplate(res);
}

String system = ragQaSystemTemplate.render(Map.of(
    "persona", "CorpAssist 制度助手",
    "context_blocks", String.join("\n---\n", chunks),
    "user_dept", session.getDept(),
    "cite_format", "【来源:{doc_id}§{section}】"
));

双栈可共享 prompts/ 目录：Python 用 .j2，Spring 用 .st，槽位名与语义保持一致。

常见误区
  • 用 f-string 在运行时拼整段 System — 无法 diff、无法 A/B、无法绑定 golden。
  • Jinja 缺变量静默变空串 — 应设 StrictUndefined 或编排层预校验。
  • Python 与 Java 槽位名不一致（context vs context_blocks）— 评测与生产分叉。
先想 10 秒：能否用环境变量直接覆盖整段 System 正文？

仅适合极短开关（如 ENABLE_STRICT_CITE=true）。长 System 应进 Git 模板；否则失去 diff、评审与 回归基线 对齐能力。

Git 版本

Prompt 变更应像 API 契约一样可审计：Git 版本 记录「谁、何时、为何」改了哪一行指令。推荐目录：

  • prompts/ — 模板源文件（.j2 / .st）
  • prompts/CHANGELOG.md — 人类可读的版本说明
  • config/prompt-manifest.yaml — 逻辑 ID → 文件路径 + prompt-semver
  • eval/golden/ — 与版本绑定的评测集（见下一节基线）

语义化版本约定（prompt-semver）

段含义CorpAssist 示例
MAJOR 行为契约变化（拒答策略、输出 JSON 形状） rag-qa-v4.0：强制带引用否则拒答
MINOR 增强但不破坏评测通过率 rag-qa-v3.2：补充 few-shot 示例
PATCH 措辞、错别字、无逻辑变更 rag-qa-v3.2.1：语气微调

流程：Prompt PR → semver 晋升

Git 版本 · Prompt 变更评审
flowchart LR
  PR[PR 改 prompts/]
  DIFF[diff + CHANGELOG]
  SEM[prompt-semver 判断\nMAJOR/MINOR/PATCH]
  REV[产品/合规 Review]
  MERGE[合并 main]
  TAG[打 tag rag-qa-v3.3]
  PR --> DIFF --> SEM --> REV --> MERGE --> TAG

MAJOR 变更需产品签字；MINOR/PATCH 可由 Prompt 负责人 + CI 回归通过合并。

代码：prompt-manifest 片段

yaml 
prompts:
  - id: corpassist-rag-qa
    version: 3.2.0
    template: prompts/rag_qa.system.j2
    owner: platform-ai
    changelog: prompts/CHANGELOG.md#v3.2
    tags: [rag, compliance]
    git_sha: ${CI_COMMIT_SHA}  # 部署时注入

发布流水线应在镜像或配置中心携带 Git SHA + prompt-semver，便于线上事故回溯：「3 月 15 日 02:10 部署的 rag-qa-v3.2 导致拒答率上升」。

常见误区
  • 只在 Confluence 改 Prompt、仓库未更新 — 生产与文档分叉。
  • 无 CHANGELOG 的「悄悄改一句」— 无法解释 A/B 指标波动。
  • 模板与评测集版本未绑定 — 通过评测的其实是旧 Prompt。

环境差异

CorpAssist · 三环境发布 dev 工程师用便宜小模型调试模板；staging 跑完整 golden 回归；prod 锁定商用模型与更严的 max_context。若把 staging 的「宽松拒答」配置误打进 prod，合规风险与幻觉率会同时上升。

环境差异 应体现在配置层（config-mgmt），而非复制三份几乎相同的模板正文。模板定义「说什么规则」；环境 manifest 定义「用哪模型、多少 Token、是否开严格引用」。

环境参数对照表

参数devstagingprod
model 小模型 / 本地 Mock 与 prod 同族商用模型 商用主力模型
rag_top_k 3（省费用） 5（对齐 prod） 5
max_context 4000 8000 8000
strict_cite false（便于调试） true true

代码：环境覆盖（manifest）

yaml 
# config/prompt-env.yaml — 同一 prompt id，不同环境覆盖
prompt_id: corpassist-rag-qa
version: 3.2.0
defaults:
  model: qwen-plus
  rag_top_k: 5
  max_context: 8000
env_overrides:
  dev:
    model: qwen-turbo
    rag_top_k: 3
    max_context: 4000
    strict_cite: false
  staging:
    strict_cite: true
  prod:
    strict_cite: true
    # prod 禁止在运行时改 template 路径，仅切换已发布 semver
不推荐 · 三份模板副本
  • rag_qa_dev.j2 / rag_qa_prod.j2 各维护一份
  • 修 bug 只改 prod 文件，staging 仍用旧拒答句
  • 环境差异写进模板 {% if env == 'prod' %} 分支
推荐 · 单模板 + env manifest
  • 一份 rag_qa.system.j2，差异进 env_overrides
  • CI 校验 staging 覆盖项是 prod 的子集（除 model 降级外）
  • config-mgmt 与模板分仓库或分目录，权限分离
5 片段

top-k 增大提升召回但挤占模板 context_blocks 预算；dev 可降 k 省费用，staging/prod 应对齐并写入 config-mgmt manifest。

dev 能否连 prod 向量库做真实检索？
不推荐。dev 应连隔离索引或 snapshot；避免调试写入污染 prod 检索统计。模板变量可 mock context_blocks。
staging 与 prod 模型必须完全一致吗？
尽量同族同参数；若 staging 降级省费用，需在 golden 上标注「模型差异容忍区间」，且 prod 发布前仍跑同族全量回归。

回归基线

Prompt 发布不是「感觉更好」：回归基线 是一组固定输入（golden）与期望行为（引用、拒答、JSON 字段），每次改模板后跑批对比。通过阈值才允许 baseline-lock 晋升为新基线。

序列图：改 Prompt → 回归 → 发布闸门

Prompt 变更 · 回归基线闸门
sequenceDiagram
  participant Dev as 工程师
  participant Git as prompts/ Git
  participant CI as 评测流水线
  participant Base as golden 基线
  participant Prod as 生产配置
  Dev->>Git: PR 修改 rag_qa v3.2→v3.3
  Git->>CI: 触发 eval/rag-v3.3.jsonl
  CI->>Base: 对比 v3.2 锁定指标
  alt 通过率 ≥ 阈值且无合规回退
    CI-->>Dev: baseline-lock 可晋升
    Dev->>Prod: 发布 prompt-semver 3.3.0
  else 指标回退
    CI-->>Dev: 阻断合并 / 回滚模板
  end

golden 集应覆盖：有依据问答、无依据拒答、注入样本、多语言 tone（与进阶章衔接）。

基线指标示例

指标说明阻断阈值（示意）
cite_rate 有检索时回答带规范引用比例 较基线下降 > 2pp 则失败
refusal_precision 无上下文时正确拒答比例 下降 > 1pp 则失败
json_valid_rate 结构化输出场景 schema 通过率 < 98% 则失败
latency_p95 模板变长导致 Token 膨胀 上升 > 15% 需评审

代码：CI 回归脚本（伪代码）

python 
# eval/run_regression.py — 绑定 prompt-semver 与 baseline-lock
BASELINE = load_metrics("eval/baselines/rag-qa-v3.2.json")
CANDIDATE = run_golden("eval/golden/rag-v3.3.jsonl", prompt_version="3.3.0")

gates = {
    "cite_rate": CANDIDATE.cite_rate >= BASELINE.cite_rate - 0.02,
    "refusal_precision": CANDIDATE.refusal_precision >= BASELINE.refusal_precision - 0.01,
    "json_valid_rate": CANDIDATE.json_valid_rate >= 0.98,
}
if all(gates.values()):
    promote_baseline("rag-qa-v3.3", CANDIDATE)  # baseline-lock
else:
    sys.exit(1)  # 阻断 CI / 合并

与 A/B 与迭代流程 衔接：回归基线保证「不变差」；A/B 在真实流量上验证「是否更好」。二者缺一不可。

PATCH 版本也要跑全量 golden 吗？
建议至少跑冒烟子集（50～100 条）；若触及拒答、引用、JSON 契约，仍跑全量。PATCH 不能成为跳过 CI 的借口。
基线数据集能否放用户真实对话？
需脱敏与合规审批；默认用合成 + 人工标注 golden。真实日志可作为补充，不可替代合规样本。

本章结论

把 Prompt 纳入工程化配置：模板、版本、环境、基线四件套。验收时能说明 CorpAssist 一次 Prompt 发布经过哪些闸门。

  • 模板引擎：用 Jinja2 / Spring PromptTemplate 渲染变量；StrictUndefined 或等价校验避免静默缺槽。
  • Git 版本：prompts/ + prompt-semver + CHANGELOG；部署物携带 Git SHA 与版本号。
  • 环境差异：单模板 + config-mgmt manifest 的 env_overrides；禁止维护三份漂移副本。
  • 回归基线：golden 集与版本绑定；baseline-lock 未通过不得晋升 prod。

动手练习

为 CorpAssist RAG 模板补全版本一行与环境覆盖，并对照回归闸门说明发布顺序（约 25 分钟）。

操作步骤

  1. 打开 demos/basics-03-variables-lab/version-log.md，按正文 semver 规则为「拒答阈值」变更填写 v3.3 一行（含 MAJOR/MINOR 判断一句）。
  2. 在纸上列出 rag_qa.system.j2 的变量：context_blocks、user_dept、cite_format；说明缺 context_blocks 时应短路还是渲染空串。
  3. 复制正文 prompt-env.yaml 片段，为 prod 增加 strict_cite: true 与 staging 对齐，并写出为何不应复制三份模板。
  4. 列举 2 条 golden 样本类型（有依据 / 无依据），说明它们如何绑定 rag-qa-v3.2 基线。
  5. 完成右侧章节测验。

判断练习

  1. 工程师在 prod 通过环境变量注入整段新 System，未走 Git 与回归。最可能的风险是？A 延迟略升 B 版本不可追溯且指标与 golden 脱节 C 仅影响 dev

    参考答案

    选 B。绕过 Git 版本 与 baseline-lock，staging/prod 行为分叉，事故无法回滚到已知 semver。

  2. 将「无依据必须拒答」从可选改为强制，semver 应升？A PATCH v3.2.1 B MINOR v3.3 C MAJOR v4.0

    参考答案

    选 C。拒答策略属于行为契约变化，应升 MAJOR，并跑全量 golden + 产品/合规签字。

  3. v3.2.1 仅修改客服问候语，未改拒答逻辑。是否可跳过一切评测？

    参考答案

    否。至少跑冒烟 golden；若问候语影响多语言 tone 或合规关键词，仍要关注 回归基线 中的 tone/合规子集。
