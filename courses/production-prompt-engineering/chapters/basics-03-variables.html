模板引擎

prompt template engine 将人设、场景子模板与运行时变量解耦。CorpAssist 不在 Java/Python 代码里拼接超长 System 字符串，而是用 {{company_name}}、{{tenant_id}}、{{escalation_queue}} 等占位符，由编排层在请求前渲染。

CorpAssist · 多租户问候 同一套 greeting.v2 模板服务 200+ 租户：渲染时注入 company_name 与当前用户部门，避免为每个客户维护一份完整 System 拷贝。
字段含义示例
template_id 全局唯一标识，写入 trace corpassist.greeting.v2
variables.required 缺失则拒绝渲染 tenant_id
variables.default 缺省值，dev 可覆盖 company_name: "示例科技"
body Markdown / 纯文本模板正文 含 {{var}} 占位

问候模板 YAML 片段

yaml 
template_id: corpassist.greeting.v2
version: "2.1.0"
variables:
  company_name: { required: true }
  tenant_id:    { required: true }
  user_dept:    { required: false, default: "全体员工" }
body: |
  您好，我是 {{company_name}} 的 CorpAssist 企业助手。
  当前服务租户 {{tenant_id}}，面向{{user_dept}}。
  我将依据工具与检索结果回答，不确定时会明确说明。
模板渲染流水线
flowchart LR
    Git[Git prompts/] --> Load[加载 template_id]
    Load --> Vars[合并 session 变量]
    Vars --> Validate[必填校验]
    Validate --> Render[Jinja2 / Spring 渲染]
    Render --> Trace[trace: template_id+tag]
    Render --> API[Chat Completions]

变量来自会话与租户配置；渲染失败应短路，不把未替换占位符发给模型。

双栈渲染示例

python 
from langchain_core.prompts import PromptTemplate

tpl = PromptTemplate.from_file("prompts/greeting.v2.yaml")
system_text = tpl.format(
    company_name=ctx.company_name,
    tenant_id=ctx.tenant_id,
    user_dept=ctx.dept or "全体员工",
)
不推荐
  • 在 Service 类里硬编码 2KB System 字符串
  • 未替换的 {{var}} 直接发给模型
  • 模板 ID 未写入 trace，排障无法对齐版本
推荐
  • Git 管理 YAML/Markdown 源文件
  • 渲染前 schema 校验必填变量
  • 与 dual stack prompt 共用路径
先想 10 秒：为何禁止在代码里拼接超长 System？

代码内字符串无法被非研发角色 Review，难以 diff 与回滚，且双栈容易分叉。模板引擎把「写什么」交给 Git，把「填什么」交给运行时上下文。

Git 版本

prompt git version 要求每次 Prompt 变更走 PR、Review，上线打语义化 tag（如 prompts@1.4.2），并在每条请求 trace 中记录 template_id + tag，实现与模型版本独立的可审计发布。

CorpAssist · 拒答话术升级 法务要求更新薪资相关拒答条款。研发在 prompts/system/boundaries.yaml 提 PR，合并后打 prompts@1.4.2，Canary 5% 流量携带新 tag，出问题可秒级回滚到 prompts@1.4.1。
目录内容变更频率
prompts/system/ 人设、全局边界、输出格式 低
prompts/scenes/ 订单、HR、审批等场景子模板 中
prompts/fewshot/ few-shot 分类与 JSON 示例库 中
prompts/change_log.md 破坏性变更说明 每次 major bump

打 tag 与推送

bash 
# PR 合并后由发布流水线执行
git tag -a prompts@1.4.2 -m "CorpAssist: 薪资拒答条款 + HR 场景边界"
git push origin prompts@1.4.2

# 应用启动时读取 PROMPT_TAG=prompts@1.4.2 或配置中心同步

禁止在 prod 配置中心无审计地粘贴整份 system message 全文。紧急热修也应生成临时 tag 并补 PR，否则无法与日志中的 template_id 对齐归因；发布前用 cost dashboard 观察模板变更是否带来意外 token 增幅。

prompt tag 与模型版本是什么关系？
独立维护。A/B 实验应同时记录 model_id 与 prompts@tag，否则无法判断回归来自 Prompt 还是底座模型切换。
何时 bump major 版本？
JSON schema 字段重命名、拒答策略语义变更、删除必填变量等破坏性变更必须 bump major，并更新 change_log.md 通知下游解析方。

环境差异

prompt env diff 用环境配置文件覆盖变量默认值，而不是复制整份 System。dev 可打开调试开关、指向沙箱 ERP；prod 关闭 debug、使用内网 endpoint——模板正文保持同一份 Git 源。

CorpAssist · dev 联调 研发在 dev 需要看到完整 CoT reasoning 排障，但 prod 必须 hidden cot。通过 allow_debug_cot 变量控制网关是否剥离 reasoning，而非维护两套 System 文件。
变量dev 默认prod 默认说明
erp_base_url https://erp-sandbox https://erp.internal Tool 描述中的 API 根路径
allow_debug_cot true false 网关是否向客户端返回 reasoning
max_fewshot 5 3 动态示例上限，dev 可放宽试验
escalation_queue dev-human prod-human 升级话术中的队列名占位

Spring / Python 读取同一变量名

yaml 
# application-dev.yaml
corpassist:
  prompt:
  vars:
    erp_base_url: https://erp-sandbox
    allow_debug_cot: true
java 
@ConfigurationProperties("corpassist.prompt.vars")
public record PromptVars(String erpBaseUrl, boolean allowDebugCot) {}

// Python 对等：os.environ["CORPASSIST_ERP_BASE_URL"]
常见误区
  • 复制 System 做 dev/prod 分叉 — 一周后两边拒答条款不一致，回归无法覆盖。
  • 环境变量未文档化 — 新同事不知道 prod 关闭了 debug，误以为 CoT 丢失是 bug。
如何发现 dev/prod 变量漂移？

CI 任务导出各环境变量表并与 Git 中的 env-diff.yaml 契约 diff；发布前 checklist 要求 prod 覆盖表已评审签字。

回归基线

regression baseline 是发布前的固定用例集（建议 ≥50 条，CorpAssist 全量基线常维护 100+ 条），覆盖拒答、JSON 合法性、多轮指代、越权与 Tool 缺失等场景。baseline 全绿是 Canary 的前置闸门。

CorpAssist · 发布闸门 工程师修改了 ticket_classify.yaml 中一条 few-shot。合并前 CI 跑 baseline 120 条：若 json_valid 从 98.5% 跌到 96%，流水线自动失败，禁止进入 Canary。
baseline → Canary → 全量
flowchart LR
    PR[Prompt PR] --> BL[回归 baseline]
    BL -->|全绿| Canary[5% Canary]
    BL -->|失败| Block[阻断发布]
    Canary --> AB[A/B 对比]
    AB --> Full[全量 + 新 tag]

无 baseline 的「直接改 prod」无法归因，只能全量回滚。

基线 manifest 片段

json 
{
  "baseline_id": "corpassist-prompt-v1.4.2",
  "prompt_tag": "prompts@1.4.2",
  "cases": 120,
  "categories": ["refusal", "json_output", "multiturn", "escalation", "tool_missing"],
  "thresholds": {
    "json_valid": 0.98,
    "refusal_exact_match": 0.95
  }
}
不推荐
  • 无 baseline 直接改配置中心 System 全文
  • baseline 仅 5 条「开心路径」
  • Prompt 变更后不跑 baseline 直接 A/B
推荐
  • badcase 沉淀为 baseline 用例
  • 场景子集（工单/HR）可分层跑，全量发布跑 120 条
  • baseline 绑定 prompts@tag 与模型版本

完整 A/B 与迭代节奏见 A/B 与迭代流程。与上一章衔接：few-shot 变更必须同步更新 baseline 中依赖该示例的用例期望。

无 baseline 上线会怎样？

线上投诉上升时无法判断是 Prompt、模型还是流量分布变化；回滚只能全量切旧 tag，且不知道旧 tag 是否仍覆盖新场景。baseline 是 PromptOps 可治理性的底线。

本章结论

Prompt 工程化的核心是模板变量化、Git 版本可追溯、环境只改变量、baseline 守发布闸门。

  • 模板引擎：占位符 + 必填校验，禁止代码内硬编码 System。
  • Git 版本：prompts@tag 与 trace 对齐，PR 可审计。
  • 环境差异：覆盖变量默认值，不复制 System 正文。
  • 回归基线：≥50 条、全绿后再 Canary。
