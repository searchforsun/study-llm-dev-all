为什么要学本章

RAG 检索到的 chunk 只是「原料」——模型会不会只依据上下文回答、会不会标注引用、检索为空时会不会拒答，全由 Prompt 决定。CorpAssist HR 知识库上线后，审计方要求「每条结论可追溯到文档片段」；本章把 S1 场景的 Prompt 契约写成可版本化模板。

学完你能
  • 设计统一的 RAG 引用格式（如 [doc_id:chunk_idx]）供前端高亮
  • 编写检索为空 / 置信不足时的 拒答 Prompt 分支
  • 用「仅依据上下文」约束实现 grounded-only 回答
  • 配置 幻觉抑制 指令与后置校验钩子
本章先记住 3 件事
  • 引用格式要机器可读：前端靠结构化 cite token 做跳转，不要只靠「见上文」。
  • 拒答是产品能力：无依据时明确说「不知道」，比编造更保护品牌与合规。
  • Grounding 写进 System：「不得使用上下文外知识」须与检索块注入顺序一起设计。

引用格式

CorpAssist · HR 知识库问答 员工问「年假最多可休几天？」——回答末尾须带 [hr-policy-v3:§2.1] 形式引用，便于审计与前端点击跳转至 PDF 对应段落。

生产 RAG 的引用不是脚注装饰，而是可追溯性契约。RAG 引用格式 需在 Prompt 中固定：每条事实句末附 cite token，token 语法与 metadata 字段一一对应。

字段含义示例
doc_id知识库文档稳定 IDhr-policy-v3
chunk_idx块序号或章节锚点§2.1 或 c0042
page可选 PDF 页码p12

CorpAssist RAG System Prompt 片段（引用规则）

markdown 
## 引用规则（强制）
- 每个包含事实的句子末尾必须附引用，格式：[doc_id:chunk_idx]
- 多来源时并列：[hr-policy-v3:§2.1][leave-faq:c0012]
- 禁止编造 doc_id；仅使用 CONTEXT 块头 metadata 中的 id
- 若一句综合多 chunk，列出全部相关 cite

## 输出结构
1. 直接回答（1-3 句）
2. 要点列表（可选，每条带 cite）
3. 引用汇总行（重复列出用到的 [doc_id:chunk_idx]）
反例：不可审计的引用
text
根据公司规定，年假一般为 10-15 天。（详见员工手册）
推荐：结构化 cite token
text
司龄满 1 年可享 10 天带薪年假，满 10 年为 15 天。[hr-policy-v3:§2.1]
引用：[hr-policy-v3:§2.1]
常见误区
  • 只要求「标注来源」 — 模型会写「根据文档」而无 doc_id，前端无法跳转。
  • cite 与 chunk metadata 不一致 — 检索管道改了 id 规则但 Prompt 未同步，审计全失效。

前端 cite 解析契约

CorpAssist Web 端用正则 \[([^:\]]+):([^\]]+)\] 提取 cite，映射到 GET /kb/docs/{doc_id}#chunk={chunk_idx}。Prompt 须禁止模型输出半角/全角混用或圆括号变体，否则解析失败。

场景Prompt 要求前端行为
单来源回答每事实句 1 个 cite高亮对应 PDF 段落
多来源综合并列多个 cite展示来源列表供用户选择
纯寒暄/澄清可无 cite不显示「来源」栏

拒答

检索 TopK 为空、相似度低于阈值、或用户问题超出知识库范围时，CorpAssist 必须走 拒答 Prompt 分支，而不是让模型「尽力回答」。

触发条件拒答类型用户可见话术
检索 score < 0.72低置信知识库中未找到足够依据，建议联系 HR 值班号
TopK = 0空检索暂无相关政策文档，请提交工单由人工答复
问题含 PII 请求他人数据合规拒答无法查询他人信息，请本人登录后重试
Prompt 注入痕迹安全拒答无法处理该请求（不解释攻击细节）

拒答分支 Prompt 模板

yaml 
# prompts/rag-refusal-v2.yaml
refusal:
  trigger:
    min_score: 0.72
    empty_retrieval: true
  system_override: |
    当前 CONTEXT 为空或置信不足。你必须拒答，不得猜测。
    使用 REFUSAL_TEMPLATE 中的话术，不要补充任何政策细节。
  templates:
    low_confidence: |
      我在企业知识库中未找到足够可靠的依据来回答「{question}」。
      建议：1) 换关键词重试 2) 联系 HR 服务台 {hr_hotline}
    empty: |
      暂无匹配的政策文档。您可通过 CorpAssist 提交 HR 工单。
先想 10 秒：拒答时是否应告诉用户「检索到了 3 条但都不相关」？

生产环境通常不暴露内部分数与条数，避免泄露索引范围；仅给出可行动下一步（换词、工单、人工）。

拒答与正常回答的 Prompt 路由

应用层在组装 Prompt 前判断检索结果：若触发拒答，则不注入 CONTEXT，改用 system_override + 固定模板，避免模型在空 CONTEXT 下仍「编造」政策。

  • 正常路径：System（含 cite 规则）+ CONTEXT + User
  • 拒答路径：System（拒答专用）+ User（含原问题）+ 禁止 CONTEXT
  • 低置信路径：可注入 CONTEXT 但 System 要求「仅当 score≥阈值 才可引用；否则拒答」

仅依据上下文

Grounded-only 约束要求模型把 CONTEXT 块当作唯一事实来源：未出现在 CONTEXT 中的数字、日期、政策名一律不得输出。这是 S1 场景通过合规审查的底线。

RAG Prompt 组装顺序（CorpAssist）
flowchart TB
  SYS[System: 角色 + grounded-only 约束]
  CTX[CONTEXT: 检索 chunk 带 metadata 头]
  Q[USER: 员工问题]
  SYS --> CTX --> Q --> LLM[ChatModel]
  LLM --> OUT[回答 + cite tokens]
  OUT --> VAL[后置: cite 存在于 CONTEXT?]

System 约束在前，CONTEXT 紧挨 User 问题，便于模型「只读块内事实」。

CONTEXT 块注入格式（双栈通用）

text 
<<<CONTEXT doc_id=hr-policy-v3 chunk=§2.1 score=0.89>>>
司龄满一年者享有 10 个工作日带薪年假……
<<<END>>>

<<<CONTEXT doc_id=leave-faq chunk=c0012 score=0.81>>>
试用期员工不享受带薪年假……
<<<END>>>

规则：回答仅可使用上述 CONTEXT 中的信息；若 CONTEXT 不足以回答，触发拒答。
python
def format_context_block(doc_id: str, chunk: str, body: str, score: float) -> str:
    return f"<<<CONTEXT doc_id={doc_id} chunk={chunk} score={score:.2f}>>>\n{body}\n<<<END>>>"
java
String block = """
                <<<CONTEXT doc_id=%s chunk=%s score=%.2f>>>
                %s
                <<<END>>>
                """.formatted(docId, chunkId, score, body);

双栈实现时，LangChain 的 ChatPromptTemplate 与 Spring AI 的 PromptTemplate 均用同一 context_blocks 变量占位符，由检索服务填充，保证 Python/Java 输出字节级一致。

CONTEXT 块顺序是否影响 grounded-only 效果？

建议按 score 降序排列，最相关块靠前；Lost in the Middle 效应下，中间块可能被弱化——CorpAssist 对超过 5 块的情况做合并摘要后再注入。

能否把检索结果放在 System 而非 User？

可以，但 System 动态膨胀会挤占固定指令空间；CorpAssist 将 CONTEXT 放在 User 前、独立分隔，便于监控 CONTEXT token 占比。

幻觉抑制

即使写了 grounded-only，模型仍可能「补全」看似合理的政策细节。幻觉抑制 采用 Prompt 层 + 应用层双层策略：指令禁止臆造、输出后校验 cite 与数字来源。

Prompt 层：显式禁止清单

  • 不得输出 CONTEXT 中未出现的具体天数、金额、百分比
  • 不得虚构 doc_id 或 chunk 引用
  • 不得将不同 chunk 的信息合并成 CONTEXT 中不存在的结论
  • 对比较类问题（「A 和 B 哪个更…」），若 CONTEXT 只覆盖其一，须说明范围限制

应用层：后置校验伪代码

python 
import re

CITE_PATTERN = re.compile(r"\[([^\]:]+):([^\]]+)\]")

def validate_rag_answer(answer: str, context_doc_ids: set[str]) -> list[str]:
    errors = []
    cites = CITE_PATTERN.findall(answer)
    if not cites and len(answer) > 80:
        errors.append("MISSING_CITE")
    for doc_id, _chunk in cites:
        if doc_id not in context_doc_ids:
            errors.append(f"UNKNOWN_DOC_ID:{doc_id}")
    return errors  # 非空则触发重试或拒答
校验失败时重试还是直接拒答？

CorpAssist 默认：1 次带错误提示的重试（「你引用了不存在的 doc_id，请仅使用 CONTEXT 内 id」），仍失败则降级为拒答模板，并记录 badcase 供 Prompt 迭代。

本章结论

S1 RAG Prompt 的核心是把「检索结果」转化为「可审计、可拒答、可 grounding」的生成契约。

  • 引用格式：固定 [doc_id:chunk_idx] 语法，与 chunk metadata 对齐，供前端与审计使用。
  • 拒答：空检索、低置信、合规冲突时走独立模板，禁止模型猜测补全。
  • 仅依据上下文：CONTEXT 块结构化注入 + System 约束，构成 grounded-only 回答边界。
  • 幻觉抑制：Prompt 禁止清单 + cite/数字后置校验，失败可有限重试后降级拒答。
