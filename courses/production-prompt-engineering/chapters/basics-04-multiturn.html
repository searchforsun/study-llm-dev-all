本章先记住 4 件事
  • 历史有预算：system + RAG + 本轮优先；历史按 window-slice 从新到旧填充至预算。
  • 事实要外置：工号、审批状态写入 session-state，不要只靠模型「记住」被裁掉的对话。
  • 框架是钩子：Advisor vs LangChain 负责注入/裁剪消息，策略仍由你定义。
  • 用实验验收：对照实验 对比裁剪前后拒答率、槽位召回与 Token 成本。

历史裁剪入门

CorpAssist · 智能客服第 8 轮 用户前 3 轮已提供工号与请假类型，第 8 轮问「审批到哪了」。若把 8 轮全文塞进 prompt，输入 Token 暴涨且可能挤掉 system 里的合规指令。编排层必须在 POST 前执行 history-trim。

API 无状态：每次调用都要重新发送 messages。多轮对话 的「记忆」= 你组装进 prompt 的内容 + DB 里的 session-state。历史裁剪 在 Token 预算内决定保留哪些轮次，而非假设模型持久记住被裁掉的内容。

流程：多轮消息裁剪（CorpAssist 客服）

多轮对话 · 历史裁剪决策
flowchart TB
  subgraph input [会话输入]
    SYS[system 模板\n始终保留]
    STATE[session-state 槽位\n工号 / 工单号]
    HIST[DB 全量历史]
    CUR[user 本轮]
  end
  subgraph trim [history-trim]
    SLICE[window-slice\n最近 max_turns 轮]
    SUM[turn-summary\n旧对话压一条摘要]
    TOK[Token 计数至预算 70%]
  end
  subgraph out [发往 LLM]
    MSG[messages 数组]
  end
  SYS --> MSG
  STATE --> MSG
  HIST --> SLICE
  SLICE --> TOK
  TOK -->|仍超限| SUM
  SUM --> MSG
  SLICE --> MSG
  TOK --> MSG
  CUR --> MSG

原则：system 与槽位注入优先；历史从新到旧填充；超限先缩 window-slice，再 turn-summary。

策略对照表

策略做法CorpAssist 适用
window-slice 只保留最近 N 轮 user/assistant 默认；max_turns: 6（见 Demo）
turn-summary 第 10 轮后用小模型把更早对话压成一条 system 摘要 长会话需保留「已提交材料清单」
session-state 注入 结构化槽位写入 system 或 tool 上下文 工号、审批单号——防裁剪丢失
Token 硬预算 从新到旧累加至 70% 输入预算 与费用控制、输出 max_tokens 预留联动

代码：裁剪策略配置（YAML 示意）

yaml 
# 与 demos/basics-04-multiturn-lab/trim-policy.md 对齐
multiturn:
  max_turns: 6              # window-slice：最近 6 轮 user/assistant
  summarize_after: 10       # 超过 10 轮触发 turn-summary
  reserve_system_tokens: 1200
  history_token_budget_ratio: 0.70
  slot_inject:              # session-state → system 前缀
    - employee_id
    - ticket_id
    - leave_type
不推荐 · 无界堆历史
  • 每轮发送 30 轮完整对话
  • 触顶后厂商静默截断头部，system 或工号被挤掉
  • 费用随轮次线性增长
推荐 · 裁剪 + 槽位
  • max_turns: 6 + Token 预算双保险
  • 工号写入 session-state 再注入 system
  • 旧轮次用 turn-summary 保留事实而非全文
先想 10 秒：把 max_turns 从 6 调到 20，模型一定更「记得住」吗？

不一定。更长历史带来噪声与费用，且仍受窗口限制；关键事实应进 session-state。是否调参应用 对照实验 看槽位召回率。

Spring Advisor vs LangChain 预览

双栈都提供「在调用模型前改写 messages」的扩展点；策略（保留几轮、是否摘要）仍由应用配置。对比 advisor-vs-langchain 时，关注谁负责持久化与谁负责裁剪。

对照表

能力Spring AILangChain (Python)
会话存储 ChatMemory / JDBC / Redis 实现 ChatMessageHistory + 自定义 backend
调用链注入 MessageChatMemoryAdvisor 等 Advisor RunnableWithMessageHistory、Memory 中间件
裁剪钩子 Advisor 或自定义 ChatMemory 实现 ConversationBufferWindowMemory（k 轮）
与 Prompt 模板 PromptTemplate 渲染 system，Advisor 追加历史 Jinja system + memory 变量

代码：Spring ChatClient + Memory Advisor

java 
// CorpAssist 客服 — Spring AI 预览（策略在 memory 实现中配置 maxMessages）
ChatMemory memory = MessageWindowChatMemory.builder()
    .maxMessages(12)  // 约 6 轮 user+assistant，按消息条数计
    .build();

ChatClient client = ChatClient.builder(chatModel)
    .defaultAdvisors(new MessageChatMemoryAdvisor(memory))
    .build();

String reply = client.prompt()
    .system(systemFromPromptTemplate)  // 含 session-state 槽位
    .user(userUtterance)
    .advisors(a -> a.param(ChatMemory.CONVERSATION_ID, sessionId))
    .call()
    .content();

代码：LangChain 滑动窗口 Memory

python 
from langchain.memory import ConversationBufferWindowMemory
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# k=6 与 trim-policy max_turns 对齐
memory = ConversationBufferWindowMemory(k=6, return_messages=True)

prompt = ChatPromptTemplate.from_messages([
    ("system", "{system_with_slots}"),
    MessagesPlaceholder("history"),
    ("human", "{input}"),
])

# 编排层：先把 employee_id 写入 system_with_slots，再 chain.invoke

无论哪栈，上线前用同一套 trim-policy.md 与 golden 多轮对话用例回归，避免「Java 保留 12 条、Python 保留 6 轮」的隐性 环境差异。

对照实验

CorpAssist · 裁剪 A/B 团队争论 max_turns 用 4 还是 10。用固定 50 条多轮脚本（含第 3 轮报工号、第 9 轮追问审批）做 compare-exp，对比槽位召回、平均输入 Token、拒答率。

实验设计模板

组配置观测指标
A（基线） max_turns: 6，无 summary slot_recall、avg_prompt_tokens、合规拒答率
B max_turns: 10 同上 + 费用增幅
C max_turns: 6 + summarize_after: 10 长会话第 12 轮槽位是否仍正确

验收标准示例：slot_recall ≥ 98% 且 avg_prompt_tokens 不高于基线 15%。若 B 组槽位略升但 Token +40%，产品需明确是否值得买单。

本章结论

多轮 Prompt 工程 = 裁剪策略 + 状态外置 + 双栈一致 + 实验验收。验收时能解释 CorpAssist 第 9 轮为何仍知道工号。

  • 历史裁剪入门：history-trim 用 window-slice 与 turn-summary；system 与槽位优先，禁止无界堆历史。
  • 与《上下文与记忆》课衔接：memory-bridge 划分 Prompt 组装 vs 记忆架构；深化见 姊妹课。
  • Spring Advisor vs LangChain 预览：advisor-vs-langchain 提供注入钩子，max_turns 语义应对齐 trim-policy.md。
  • 对照实验：compare-exp 用 slot_recall 与 Token 成本选优，结论回写版本库。

判断练习

  1. 为「记住工号」，团队把 max_turns 从 6 提到 30，未写 session-state。第 15 轮仍可能丢失工号的最可能原因是？A Token/窗口仍裁掉早期轮次 B temperature 太低 C 模型不支持中文

    参考答案

    选 A。轮次增多仍受 window-slice 与 Token 预算约束；工号应写入 session-state 并注入 system，而非指望堆历史。

  2. Python 用 k=10、Java 用 maxMessages=12，评测脚本用最近 6 轮。上线后双栈行为不一致。首要修复是？A 统一 trim-policy 语义并回归 golden B 换更大模型 C 关闭 Advisor

    参考答案

    选 A。对齐 trim-policy.md 与 compare-exp golden，避免隐性 advisor-vs-langchain 差异。
