历史裁剪入门

多轮对话时，history trim 决定哪些 User/Assistant（及历史 tool message）进入下一轮 Prompt。CorpAssist 默认策略是「最近 6 轮对话 + 超长时滚动摘要」，避免 history 挤占 few-shot、RAG 与 System 边界条款的注意力预算。

CorpAssist · 长会话查单 员工先问订单 SO-2026-8842 物流，又追问退款进度，再切换到另一张单。若把 30 轮全文塞回模型，input tokens 暴涨且旧轮次里的错误指代会污染当前回答。裁剪保留最近 6 轮，更早内容进摘要块。
历史裁剪策略
flowchart TB
    Full[Redis 全量会话] --> Win[滑动窗口 6 轮]
    Win --> Check{超 token 预算?}
    Check -->|否| Inject[注入 messages]
    Check -->|是| Sum[摘要模型压缩]
    Sum --> Inject
    Sys[System 消息] --> Inject
    Inject --> API[Chat Completions]

System 永不裁剪；先窗口后摘要，摘要 prompt 独立版本化。

消息类型是否裁剪说明
System 否 人设与边界必须每轮完整存在
User / Assistant 是（窗口外丢弃或摘要） sliding window 保留最近 N 轮
Tool（历史） 视情况 过期 Tool 结果可摘要为「曾查询：订单已发货」

Python：滑动窗口裁剪

python 
MAX_TURNS = 6  # 6 轮 = 最多 12 条 user/assistant（不含 system）

def trim_history(messages: list[dict], system_msg: dict) -> list[dict]:
    """保留 system + 最近 MAX_TURNS 轮对话。"""
    dialog = [m for m in messages if m["role"] != "system"]
    trimmed = dialog[-MAX_TURNS * 2 :]
    return [system_msg, *trimmed]

触发摘要的条件示例：裁剪后仍超 8K input tokens，或会话轮次 > 12。摘要指令应要求保留：当前关注的订单号、未完成的审批动作、用户明确表达的约束（如「不要自动退款」）。

不推荐
  • 无上限堆叠全部历史，直到 API 413
  • 裁剪时误删 System 或把 System 混入 Assistant 续写
  • 摘要后丢失「禁止编造单号」等关键边界
推荐
  • System 固定首位，窗口只作用于 dialog
  • 摘要 prompt 列出不可丢失字段清单
  • 裁剪参数（MAX_TURNS）版本化并写入 trace
先想 10 秒：为何 System 绝不能被 history 裁剪挤掉？

边界与人设写在 System；若长 history 挤占预算导致 System 被截断或模型「遗忘」拒答条款，会出现长会话后编造单号的典型 badcase。预算分配应优先保障 System。

常见误区
  • Prompt 课实现 Redis — 应定义契约，存储交给上下文课。
  • 摘要与 RAG 混写 — 会话摘要≠知识库检索，引用格式应区分。
两课联调时最先对齐什么？

session_id 生成规则、history 在请求中的 JSON 形状、以及 MAX_TURNS 默认值。三者不一致会导致双栈或多服务间「失忆」。

Spring Advisor vs LangChain 预览

spring advisor 在 Spring AI 调用链前后织入记忆读写；LangChain 用 langchain memory 包装 Runnable。CorpAssist 双栈必须统一 session_id 与窗口大小，否则同一员工在 Python Agent 与 Java API 会看到不一致的多轮上下文。

CorpAssist · 双栈会话 前端 IM 网关生成 session_id=ca-8842-uuid，Python 侧 Agent 与 Java 侧工单服务共用 Redis 中的 message list，遵循 dual stack prompt 会话契约。Advisor/Memory 只负责「读多少、写哪里」，裁剪策略仍由 Prompt 配置驱动。
记忆织入调用链
flowchart LR
    Req[HTTP 请求] --> Adv[Spring Advisor / LC Memory]
    Adv --> Load[从 Redis 加载 history]
    Load --> Trim[按 MAX_TURNS 裁剪]
    Trim --> Prompt[组装 Prompt]
    Prompt --> LLM[模型]
    LLM --> Save[写回 Assistant 到 Redis]

Advisor/Memory 是管道；裁剪参数来自 Prompt 版本配置。

Java：MessageWindowChatMemory

java 
@Bean
ChatMemory chatMemory() {
    return MessageWindowChatMemory.builder()
        .maxMessages(12)  // 6 轮 user+assistant
        .build();
}

// PromptChatMemoryAdvisor 在 call 前注入 memory 到 Prompt

Python：ConversationBufferWindowMemory

python 
from langchain.memory import ConversationBufferWindowMemory

memory = ConversationBufferWindowMemory(
    k=6,
    return_messages=True,
    memory_key="history",
)
chain = RunnableWithMessageHistory(runnable, memory=memory)
不推荐
  • Python k=10、Java maxMessages=6，双栈窗口不一致
  • session_id 由两端各自生成，无法共享会话
  • 把记忆逻辑写进 System 字符串
推荐
  • 网关统一签发 session_id，双栈只读 Redis
  • MAX_TURNS 写入 prompts/multiturn.yaml 版本配置
  • Advisor 顺序：Load → Trim → 再拼 System/User
Advisor 与 Prompt 模板谁决定「保留几轮」？

策略数值（MAX_TURNS、摘要触发阈值）应由 Prompt 版本配置定义并审计；Advisor/Memory 负责执行。避免在 Java @Bean 里写死 magic number 而无 Git 记录。

对照实验

上线裁剪策略前，用同一 CorpAssist 问题集做 trim experiment：对比无裁剪、6 轮窗口、窗口+摘要三种配置，记录 JSON 合法率、指代消解准确率与 tokens P50/P95。

CorpAssist · 裁剪 A/B 选取 50 条含多轮指代的真实对话（如「那笔单」「刚才说的审批」），在 staging 跑三种策略，填实验表后选默认 MAX_TURNS=6 上线。
策略tokens P50指代消解准确率风险
无裁剪 高（>12K） 高 易爆窗、成本高、延迟差
6 轮窗口 中（~4K） 多数场景够用 超长会话早期约束可能丢失
窗口 + 摘要 低（~3K） 长会话更好 摘要 prompt 质量决定下限

长会话 badcase（如摘要后忘记「必须二次确认才能退款」）应沉淀进 回归基线，而非仅调大窗口。完整 A/B 流程见 A/B 与迭代流程。

摘要丢失「必须二次确认」约束怎么办？
摘要 prompt 显式列出不可压缩的约束清单（订单号、待确认动作、用户否决项）；永久规则保留在 System，不依赖摘要传递。
排障：长会话后模型忘记「禁止编造单号」，优先查什么？

先查 history 是否挤占预算导致 System 被截断或摘要覆盖边界；再查 MAX_TURNS 是否过小丢失拒答上下文。盲目加轮次往往恶化 token 成本，应优先加固 System 与摘要约束清单。

本章结论

多轮 Prompt 的核心是 System 优先、先窗口后摘要、双栈 session 一致，并用对照实验验证而非凭直觉加轮次。

  • 历史裁剪：默认 6 轮 + 条件摘要，System 永不裁剪。
  • 课衔接：Prompt 写契约与文案，上下文课做存储与预算。
  • 双栈预览：Advisor/Memory 执行，参数版本化在 Git。
  • 对照实验：记录 tokens 与指代准确率后再定 MAX_TURNS。
